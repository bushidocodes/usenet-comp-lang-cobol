```
I have to parse an input bytestream of COBOL records.  The input bytestream
is in ASCII.

There are several fields of the following type -- S9(9)V9(2).  My
understanding is that a field of this type occupies 15 bytes of storage.
The first 14 bytes will contain values in the range of 0x'30' -- 0x'39', and
they do.  The 15th byte not only contains the digit value, but the sign as
well.  The sign is implicite for positive values.  So for positive values,
the 15th byte will contain values in the range of 0x'30' -- 0x'39'.  But for
negative values, the 15th byte will contain values in the range of 0x'70' -- 
0x'79' (ascii chars 'p' -- 'y').

But, I'm seeing ascii chars }, J, K, L, M, N, O, P, Q and R.  But, these are
the characters that are used to represent a negative sign in EBCDIC!!!  The
hex encoding is different than EBCDIC, but the characters that the hex value
represents, are the same.

Is this bad data?  Or, is this some COBOL thing, that I don't understand.
I've found a MicoFocus COBOL manual that contains a "Sign(EBCDIC)" column in
a table for the Negativly-signed values of the ASCII charset.  So this leads
me to believe that the EBCDIC characters for negative values can be used to
replace the standard ASCII character 'p' -- 'y'.

Can someone kindly explain what is going on here?  What type of system, if
any, would be generating data like this?

The parsing tool that I'm using will only accept 'p' -- 'y' for a negative
sign.  So, I may have to translate the 15th byte of these fields in a
pre-processing step.

Thanks in advance.....
--matt
```

```
Matt,
It seems likely that the data originated on an EBCDIC platform, and was 
transferred in text to the ASCII platform.  The perfectly valid EBCDIC 
characters J thru R were translated to the same ASCII character.

How about the "-0"?  It would have been X'D0' in EBCDIC.  Any idea what 
that would translate to in ASCII?  It would seem that there would be 
about 10% of the negative values which ended n zero.

This is why the most sensible thing to do when moving data is to 
reformat it to SIGN SEPARATE LEADING / TRAILING.  Then, all of the 
numbers are really numbers, easily and unambiguously translated in 
either direction.  And the sign character will contain a "+", a "-", or 
a space.

Happy hunting,
Colin
```

```
IBM mainframe format that has been around since Bill Gates was in
diapers. It is not "just" a COBOL thing.
```
