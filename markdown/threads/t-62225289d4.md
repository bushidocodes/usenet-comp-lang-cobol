And it is even simpler than that - there was an enhancement "late" in the
life of "IBM COBOL for z/OS & VM" (and in Enterprise COBOL).  IF (and only
if) you use the SQL "coprocessor" - it is now supported to have EXEC SQL
statements in COPY members.  See:

 http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/IGY3PG10/3.2

and related sections of the manuals.

To: comp.lang.cobol (with BCC's to some of my "Usually reliable sources")

If you are having problems with the coprocessor (from COBOL - not PL/I) then
you should definitely report this to the COBOL "support" team.  It is true,
that *THEY* may then re-queue it to DB2 (or require DB2 assistance) - but
the FIRST place (Level 1 / Level 2) that you should go is via COBOL.
ESPECIALLY if it is for something that "works" with the pre-processor.

P.S. The same for the CICS integrated compiler support.

You are correct.

In order to compile, the INCLUDES will have to be part of the main
program and not in a COPYBOOK.   However, this phenomenom is shop
specific.  At my last shop, the COPY statements were resolved 'before'
the pre-compile, in which case, the host-variables would have been
recognized and the code would have compiled.
