The ZZ9.99 field is really designed for screen usage and
printer usage only.  The usual way of doing this is to use
that for the user, but convert to 999v99 before validation.

If you use the screen section, then the conversion can be
completely automatic; the screen section picture is ZZ9.99,
but the from/to used for the input data would be 999V99.

You are going to have to have it in a numeric format before
you can use it for anything beyond playing it back to a human
anyway, so you might as well validate in that form ...

Well, what I would do is first change the PIC from ZZ9.99 to 9(3)V99. 
Since the field in question is on the input side I would not use an 
edited field in this situation. 

The code in the procedure division would simply test WS-ORDPRICE as 
follows:

IF (WS-ORDPRICE IS NUMERIC)
THEN
    IF (WS-ORDPRICE >=1 OR WS-ORDPRICE <= 999.99)
    THEN
        valid data.

The logic seems a bit redundant but the first test verifies if numeric 
data is present in the field and then the second test verifies the value
range. It may not be necessary to check for the upper limit since it is 
the highest value that is possible but it can't hurt to have it in place.

Just my 2 cents.

First off, I agree with the others, change WS-ORDPRICE to 9(3)v99.

Your condition then is:
    88 VALID-WS-ORDPRICE VALUES 0.01 THRU 999.99.

But to answer your question with zz9.99:

      $set ans85 flag"ans85"
       identification division.
       program-id. rangeval.
       data division.
       working-storage section.
       01 ws-ordprice pic zz9.99.
         88 valid-ws-ordprice value "  0.01" thru "999.99".
       procedure division.
       begin.
           perform with test after until ws-ordprice = "  0.00"
               accept ws-ordprice
               if valid-ws-ordprice
                   display "Order Price is Valid"
               else
                   display "Order Price is INVALID"
               end-if
           end-perform
           stop run
           .
