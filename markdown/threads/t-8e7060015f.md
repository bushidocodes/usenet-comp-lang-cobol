I'm having a cross language contest at the moment with a file/data
handling problem and I'm wondering if any of the locals can come up
with a better solution than I have for COBOL. It would be nice to show
how valid COBOL really still is for this kind of task. Even with my
quick afternoon hack, it's looking good though.

I have an SQL database with 7.5 million records in the table, some of
which are duplicates. The task is to weed out the duplicates.

In SQL, this can be achieved with a simple SQL statement such as
selecting all rows grouped with count(*) > 1. This however requires
huge amounts of time and resources. More than the server, a dual
Pentium Pro 200 with 1Gb RAM, NT Server and MS SQL Server.

So, more traditional methods are needed.

I've got three solutions, one of which has completed already and I
know works. A Delphi app selects data down in manageable chunks into a
table and looks for duplicates (with SELECT count(*) etc)  before
creating a clean table of new data and putting it back onto the
database in another table. I'll copy the table across once I'm sure
it's correct. This took 29 hours to run.

As this was taking a long time, I wrote a COBOL app to run along side
on a similar machine. The 7.5 million records are comma delimited
lines straight from SQL Servers BCP command. They contain variable
length data.

The COBOL program is a typical conversion program in that it reads in
the line sequential file and then writes to an indexed file. The key
is the whole record as I don't want duplicates remember. Any WRITE
giving an invalid key is written off to a duplicates file for later
investigation. The ok records are then written off to another indexed
sequential file to form a valid BCP input file that I can bulk copy
into the database.

This works fine but I notice the count is getting slower as the index
gets bigger which is as I expected. After I started it off, I thought
of a few other ways of achieving the same goal but I'm not sure if I
they'd be quicker. For instance, I could sort the file first then just
go through it checking for matching lines next to each other. No index
needed. Any other ideas? File handling like this isn't really my
forte.

Program is as below. It's running at the moment and I'd guess from
it's status it'll take approximately 14 hours. Add in 2 hours to BCP
the data back and It'll beat Delphi and SQL by about 12 hours I'd
think ;-)

       ENVIRONMENT DIVISION.
       FILE-CONTROL.
       select In-File   ASSIGN "infile.dat"
           organization is line sequential.

       select Out-File  assign "outfile.dat"
           organization is indexed
           access mode is random
           record key is out-rec
           file status is out-status.

       select in-File-seq  assign "outfile.dat"
           organization is indexed
           access mode is sequential
           record key is ifs-rec
           file status is out-status.

       select dup-file assign "dupfile.dat"
           organization is line sequential.

       select flat-file assign "flatfile.dat"
           organization is line sequential.

       Data Division.
       File Section.

       FD In-File.
       01  In-Rec          pic x(101).

       FD Out-File.
       01  Out-rec         pic x(101).

       FD in-File-seq.
       01  ifs-rec         pic x(101).

       FD Dup-File.
       01  Dup-Rec         pic x(101).

       FD Flat-File.
       01  flat-rec        pic x(101).

       working-storage section.
       01 rec-count        pic 9(8) comp value 0.
       01 dup-count        pic 9(8) comp value 0.
       01 ok-count         pic 9(8) comp value 0.
       01 stat-count       pic 9(4) comp value 0.
       01 end-of-file      pic 9 comp value 0.
           88 eof          value 1 false 0.
       01 out-status.
           03  os1         pic 9.
           03  os2         pic 9.

       procedure division.
           display space upon crt
           display "Duplicate strip" at 0101
           perform find-dups
           perform update-stats
           display "Completed duplicate search" at 0601
           perform create-bcp
           stop run.


       find-dups section.
           open input in-file
           open output out-file
           open output dup-file

           perform until eof
               read in-file
                   at end
                       set eof to true
                   not at end
                       move in-rec to out-rec

                       write out-rec
                           invalid key
                               perform write-dup
                           not invalid key
                               add 1 to ok-count
                       end-write

                       add 1 to rec-count
                       add 1 to stat-count
                       if stat-count = 1000
                           perform update-stats
                           move 0 to stat-count
                       end-if
               end-read
           end-perform
           close in-file
           close out-file
           close dup-file
           .

       write-dup section.
           add 1 to dup-count
           move in-rec to dup-rec
           write dup-rec
           .

       update-stats section.
           display "Records read = " at 0201
           display rec-count at 0216
           display "Records ok   = " at 0301
           display ok-count at 0316
           display "Duplicates   = " at 0401
           display dup-count at 0416
           .

       create-bcp section.
           set eof to false
           move 0 to rec-count
           move 0 to stat-count
           open input in-file-seq
           open output flat-file

           perform until eof
               read in-file-seq
                   at end
                       set eof to true
                   not at end
                       move ifs-rec to flat-rec
                       write flat-rec
                       add 1 to rec-count
                       add 1 to stat-count
                       if stat-count = 1000
                           display "BCP records written = " at 0801
                           display rec-count at 0823
                           move 0 to stat-count
                       end-if
               end-read
           end-perform

           display "BCP records written = " at 0801
           display rec-count at 0823
           close flat-file
           close in-file-seq
           .



The third test I'm running is for those 'use a text processing
language' types. I know someone would say use awk so I'm doing just

I'm running the following on  a Pentium II 300 with 96Mb ram, NT

If you are looking for real speed, there are two fast changes
I would make that I bet will speed it up significantly.

First, the validation file.  Instead of writing the whole record,
write a record consisting of key only.  That is all you need for
a duplicate check.

Then, use the INVALID status for two streams.  One
line sequential output to send back, and one line sequential
output of duplicates. That way you do not have to unload
it at the end.

As in thermodynamics, for every problem there is a minimum amount of work
below which you cannot go.  _Any_ solution to this problem will take a
long time.  The question is, _how_ long (in relation to the other
solutions).  You know without even trying it that AWK is not competitive;
not because AWK is a bad language, but because an interpreted program has
more overhead than a compiled program.  You knew without even trying it
that a SQL solution is not competitive; not because SQL is a bad language
but because a database program has far more overhead than a procedural
program.

Sorts are pretty highly evolved procedural programs.  Your CoBOL program
is essentially using side effects; a lot of the hidden processing is
wasted.  I'm not saying I know where the comparison would come out, but
the sort is definitely what I would try first, because it requires no
programing.  I know I could beat the sort in CoBOL if I had to, but it
would take some time to tune, time that I don't need to spend if the sort
comes out in the ballpark.

There is of course the hardware effect.  A few years back, I had a lovely
benchmark consisting of a day's installment loan transactions from a
midsize retail bank.  It took four and a half hours to sort on a Dell
486.  The bank bought a shiny new RS/6000 from their friendly
neighborhood IBM salesman.  Based on the hardware specs in the glossy
brochure, they were expecting this machine to be many times faster than a
486, and were surprised to find that it took three and a half hours to
sort, though they shouldn't have been.  The same data set took almost
ninety seconds to sort on the development 3090.

I did something like this years ago on a smaller (but not small) file.  I
suggest you sort the file (in place or to another sequential file).  Then make
another pass through the file culling duplicates.  

This sounds clunky BUT... this task will run as fast as I-O to the disk can be
done.  The fewer block  I-O's, the faster the run.  The I-O requirements of
locating a single record in an indexed file will take 3 to 10+ block I-O's.
Because of the huge number of records the index tree will also be huge,
magnifying the amount of physical I-O needed to navigate the tree in order.  

3 to 10+ I-O's on  a sequential file will get more done than 3-10 I-O's on an
indexed file.   The sort is also more efficient than navigating the index tree
of an index file.  I strenuously avoided this conclusion until I was faced with
actually applying it.  My sort-then-cull method was far faster than my initial
indexed file method.  That's why they do it that way on mainframes.

If you must do this in a database environment then I suggest dropping all the
indexes, create an index of just the fields (and the ROWID if possible) that
are to be checked.   When the index is created then fetch it one record at a
time and use the ROWID in the DELETE statement to remove the duplicate records.
 When you're done, recreate the indexes.  The goal again is to reduce I-O to
the minimum level.

Denis Dew

If you are extracting from one table and inserting into another why not let
the SQL tell you if it is a duplicate, then simply move on to another?

Define the output table as having unique keys, read through one record at a
time, inserting them one at a time. If you get a duplicate record status
code back from your SQL server, then drop the record. This is assuming that
if a table is defined with unique keys, that it will not insert a duplicate
record, physically allowing only one record with that particular key in the
output table.

Another solution would be to use a new fanlgled process, called parallel
processing. Have several programs perform what you have already written, but
only for a group of keys. Say program 1 processes keys A-C, PGM2 D-F, PGM3
G-I ... AND SO ON. Easily done by passing each program a parm and using SQL
to select those particular keys. You may want to figure out the distibution
of the keys to better determine the key groupings. Either way it should
reduce processing time. Make sure you use different jobnames for each
grouping, else they will still run in sequence.
