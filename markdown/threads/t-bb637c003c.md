Huib,

A *static* object can be defined as a WORKING-STORAGE data item,
where the corresponding object is created when the WORKING-STORAGE
SECTION is allocated. Moreover, the attributes of a static
object are arranged in memory in the same way as if we would
replace the data item for the static object by the data items
corresponding to its attributes.

The standardization board has considered the
issue, but I don't know if it was part of some version of
the draft or why it was eventually removed.

Raymond Obin's Book on OO COBOL for COBOL programmers gives
some surprising examples for static objects.

Ralf

Let me give another "explanation" of Static vs Dynamic objects (as I
understand them). The following is a "user-friendly" dfinition and is
neither EXACT nor precesily what is being considered/proposed in the
Standard.

The first thing that you need to know is that (in COBOL) they tie in with
recurssion - where multiple "instances" of a program may be going at the
same time.

Do you know how EXTERNAL and non-EXTERNAL work? How the same item is
available to multiple programs (wihtout passing them as parameters in a CALL
statement)? Well STATIC items are similar in that they exist (are available
and their values are "kept") from instance to instance of every occurance of
the same program that is used recursively.

On the other hand, for dynamic items, each occurrence of the item is
"speicifc" to that instance of the recursive program. So recursive program
ABC can have a value of "Y" in field-1 in instance one while it has a value
of "N" in instance two.

Does this help?

P.S. I really have tried to avoid this area of the Standard - so if I have
desribed something that has nothing to do with Static vs Dynamic items, I
apologize profusely in advance - and hope that somene corrects me as soon as
possibl. I know that the distinction that I am talking about exists and I
think it is the difference between static and dynamic - but I won't bet my
life on it.

P.P.S. The current draft talks about "automatic" storage and that may be
either what I am thinking of or related to this topic.

William,

as far as I understand, your explanation of Static versus Dynamic
rather concerns the term of a static variable as for example in C.
A static variable is local to some block, i.e. it is visible in
the corresponding block only, but its lifetime is the entire run
of the program and, indeed, recursion does not allocate a new
instance for the static variable.

Static versus dynamic as I have seen it in Raymond Obin's
book concerns the use of objects versus object references.
Usually, we declare data items of data type USAGE OBJECT
REFERENCES. Thereby, we are forced to use references all the
time. The block for the working-storage data with the data
item contains a reference, only. With a static object, the
working-storage section contains directly the attributes of
the object. There is no need to create the object.

This concept is useful for example in migration of legacy
code: We look for data items, for example the fields
with PIC 9(7) according to a date, which can be encapsulated
in an object. Then each access can be rewritten as method
invocation for the object.

Static objects are useful in general, whenever the lifetime
of a certain object should coincide with the lifetime
of the corresponding working-storage section. For dynamic
objects we have to perform creation; we have to think about
finalization. Other languages, like Ada and C++ permit
static objects. I didn't found the term in the COBOL
committee draft.

Static objects should work in the working-storage section
and in the file section. Object references make no sense
in the file section. Another striking point for static
objects.

Ralf

Ralf and William,

The standardization committee decided to abandon the keywords "static" and
"dynamic" and introduced the local-storage section (lss) for dynamic things.
The good old working-storage (wss) section remains and is meant for static
data. An embedded method with a wss can put data in its wss-fields. On a
next invokation that data is still present. Data in the lss on the other
hand is re-initialized on each entry of the method's proc. division. The lss
is stacked in case of recursive calls.

The way you, Ralf, described static to me, sounded a lot like embedded
object in c++, for example:

class Example1
{
private:
OtherClass anEmbeddedObj;
};

This as opposite to an embbeded object reference, a pointer in C++:

class Example2
{
private:
OtherClass * ptrToAnObj;
};

The construct in example 2 most resembles the COBOL object references in the
coming standard. But no dereferencing is needed as in C/C++.
Static objects in the sense of directly defined instances of a class are IMO
not possible. The only way to do somthing alike is to define in the wss:
01 myStatObj usage object ref.
and in the init (method/section) of the program
invoke "ClassName" "new" returning myStatObj

The Fr_OO_g (OO Frog :) )
