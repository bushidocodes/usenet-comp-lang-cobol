Absolutely on target.  Sure, 'all Windows applications look the same',
but that's the whole point.  One of the greatest contributions that
Windows has made to software has been the imposition of a consistent
user interface.  Unlike the Bad Old DOS days, at least you know how to
close a Windows application without digging through the manual!

The hard part is learning all of the Windows 'standard' conventions.
There are a ton of keyboard conventions (which virtually no one uses)
that you still need to take into account, because it's a certainty
that if you use, say, F10 to trigger some action, someone will come
along and wonder why it doesn't enable the menu k/b interface like
tapping the Alt key does.

Much of this stuff is on the Microsoft site.  The problem is, there
does not seem to be a central location for standard keyboard shortcuts
and other design considerations becaues much of this stuff depends
upon which control is active;   Still, an hour of digging should yield
some good information.  

It's especially troublesome when you're converting an old DOS
application to Windows because so many key combinations that were just
fine in a DOS application, have default, standard Windows actions
assigned to them.  Then, the quandary is, should you retain the old
DOS key combos and pre-empt the Windows standard, or bite the bullet
and make the converted application truly Windows?

Personally, I favor the latter.  It's really nice to know your stuff
behaves like a Windows application 'should'.

Grinder:

Sorry for my slow response, but I was staffing our booth down at CA
World in Orlando and just returned to the officve this morning.

We don't really publish a set of sp2 coding conventions, because the
product is designed to allow you to create user interface screens in
any manner which suits your specific needs.

Although we strongly advocate creating Windows screens according to
the Microsoft design standards, there really isn't any forced
compliance in the product.

Our philosophy is that programmers must have a high degree of control
over the user interface design, because sometimes, the Microsoft
standards really don't fit the needs of a very specialized type of
application.

>(1)  When focussing an input control, the insertion point should
>move to the end of the text.

This is accomplished automatically through the use of right justified
custom fields.

>(2)  For record entry screens, there should be an OK and Cancel
>button.

COBOL sp2 allows you to create "template panels" which contain
standard screen layouts.  These "template panels" can be dragged into
the Format Window from the Treeview Control when creating a new panel.


If you create a standard template panel for record entry screens, you
would simply create the OK and Cancel buttons on the template panel.
When you want to create a new record entry panel, simply drag and drop
the template panel from the treeview control to the Format Window in
the Panel Editor.  Make any cuystom changes to the record entry screen
and then click on the File menu bar option and the Save As pulldown
menu bar option.  Give the panel a new name.  Your template panel
remains intact and you created a panel without having to create all of
the standard controls on the panel because they were already contained
in the template.

>(3)  When closing a record w/ unsaved changes, the user should
>be prompted with "Save changes?  <yes> <no> <cancel>"

We suggest creating a standard "Save changes?" message box routine in
your program and invoke it anytime the end user closes a record entry
panel.  That allows you to have a single routine which is used by all
record entry panels ion your application.

>These are all detail examples, but I'm looking for a larger view
>of how an SP2 application should present itself.  Thanks.

Again, there really isn't any specific manner in which an sp2
application should present itself.  We *strongly* recommend designing
your user interface screens in the "Microsoft design standards" but
certainly don't force you to design your screens in that manner.

I hope that this helps.  If you have any specific design issues, we
would be happy to assist you.  Please contact me at rtwolfe@flexus.com
if you would like to obtain more assistance.  I'll refer you to
someone who can help you with panel design or programming questions.
