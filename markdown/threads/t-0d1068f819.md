For the OPTIONAL files you have to fake out the system.

If you leave off the DD statement, the system will probably crater you with a
Charlie-Four, atleast on OPEN INPUT. I am not really sure what OS/390 LE will
do for a missing DD statement for an OPEN OUTPUT.  (if the dataset meets its
criteria for a sysout class dataset, it could create it dynamically with a
temporary DSN! But lets ignore that for now, you seem to be doing input).

If on the relevant DD you code the DSN with a DISP=SHR, then the system expects
to be able to find the file. (The file not found is coming from the intergrated
catalogue facility, the catalogue search came up empty handed).

So, you need something like DSN=NULLFILE.  I know that this technique allows
opens/closes and even mindless writes. I am not sure about reads.  Probably it
will return EOF upon the first read (that is, AT END in COBOL).

I am not sure of the FILE STATUS at open time.  You may get something other
than 00 at OPEN.  That it probably to your advantage.

I recommend, that you code FILE STATUS for that file and DISPLAY it at open
time.  In the mainframe environment you may get early information that it is a
NULLFILE, which you can choose to work with or not. At anyrate, you probably
should get the EOF you need upon first read.

For your information there are some coding techniques that could snag you here,
and cause time consuming error analysis.  You can avoid coding DCB information
in COBOL programs for an existing file, usually. (This would be RECFM BLKSIZE
and LRECL).
This is the usual approach.

But if you have no such information in the FD you are possibly in trouble if
you do not code it explicitly on the JCL DD statement.  The OPEN macro can
fail, and READ macros can fail in the absence of BLKSIZE or RECFM information
(it depends upon OS and release level).

So as strange as it seems, if you do not have an true exsiting file, no
catalogue entry, (and no VTOC entry, and no DSCB info), you may need to code
some DCB=(...) subparms explicitly in order for the OPEN and READ to pass early
validation checks internally.

In a mainframe environment you almost always do want to leave this out of your
COBOL source. And most JOB streams that really do bring in an existing file
would want to abstain from DCB parameters, lest they become obsolete.  

Yet when you are being tricky with OPTIONAL files, the OS macros do need
certain minimum info for a successful control block deployment, and you have to
code the DCB. It seems weird, but its true.

You can use model DSCBs for that so that you do not proliferate JCL that needs
to be updated when file characteristics evolve.

Best Wishes,

Robert Rayhawk
RKRayhawk@aol.com
