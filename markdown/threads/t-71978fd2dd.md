I can't remember the specifics of 2.4 so what I say next could be somewhat
off, but I believe we 'bound' the .EXE modules so they would work on DOS and
OS/2. At the time, Windows was not well used as 2.4 was out pre Windows 3.0.

The problem with dual mode or bound executables is that applications marked as
OS/2 window compatible, confict with Windows Window compatible. So Windows
thinks it's a Windows program but after the initial start up, it uses OS/2
api's and you get that message. It seems a bit daft but thats the way
Microsoft did it. ;-)

2.4.17 is really old though, you should upgrade.

› It sure is a pain to have to reboot into DOS to compile programs.

Well, it's not supposed to work on Windows if you read the spec. We're now on
v4.0 which is a 32bit product or 3.2.50 which is 16bit or if you're just
doing Windows development and don't need mainframe dialects, there's Visual
Object COBOL. Check out the web site in my sig.
