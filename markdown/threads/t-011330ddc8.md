Over the last few months I have been deeply involved in designing and
building better access to Relational Databases from COBOL.

This has meant dozens of experiments and benchmarks using hand written and
generated SQL, embedded in COBOL programs.

Mostly this has been through ODBC, as required by Fujitsu NetCOBOL (unless
you use the OLE COM interface).

One of the measurement criteria that has emerged from this is the number of
COBOL records that are returned by a process every second.

For example, let's say you have something written in COBOL and running
against an indexed file (VSAM/KSDS or PC ISAM).

For (skip) sequential retrieval against parts of this file you issue a START
followed by a loop of READ NEXT...

How many records per second do you get from your READ NEXT?

Is it slower if you are reading backwards (GET PREVIOUS or START
DESCENDING)?

Usually, SQL code has to be hard coded for a particular table on the RDB
because the elements in the table need to be spelled out to the SQL.
Languages like C# have a facilty called "reflection" which means you don't
have to do this, if you use LINQ instead of SQL. The language uses metadata
about the table in order to determine the columns in it. While this is very
generic and ideal for generating code solutions, it is poor performance
wise.

(A recent test saw reflection drop the number of records per second from a
couple of hundred (assembled from more than one RDB table) down to 20...
Replacing the reflection saw the numbers rise to over 1500 per second.)

The problem is not with LINQ (which can retrieve thousands of "objects" per
second) it is with the fact that COBOL is designed for record processing and
relational databases, generally, aren't. The RDB is designed to work at an
elemental level dealing with the only the columns that need to be
referenced.

To "translate" the above COBOL scenario into SQL means defining and opening
an SQL cursor to retrieve "records" row-by-row. This means trips to the
RDBMS system for every record retrieved. (Most RDBMS handle cursors pretty
well and the results are usually buffered in memory and returned when you
FETCH a row. LINQ retrieves a "result set" as a single RDBMS access and you
can then manipulate this list as an object and do whatever you like with it,
WITHOUT further RDBMS interaction.)

The data retrieved has to be placed in a COBOL buffer so it looks like it
was read from a file and that means each column has to be converted to the
right size string at the right location in that buffer. This is usually
accomplished with Host Variables which push all the casts and conversions
"under the covers". (There is still a problem with handling date types
because COBOL has no equivalent type and if a column on a table is defined
as a date this means specific code in COBOL to deal with it. Using LINQ gets
around this because both C# and the database recognise a Date object.)

Using LINQ in C# there is no need for the Host Variables or for the SQL
cursors and we can update the COBOL buffer directly back in the calling
program. But it needs to be hard coded rather than reflected in order to get
maximum performance. Fortunately, it isn't too hard to write tools that can
generate this access, exactly as we did with SQL. PRIMA will be releasing
LINQ DAL generation later this year.

The current DAL solution generates COBOL code with ESQL in it to manage a
specific set of normalized tables on the RDB.

The new solution will do pretty much the same, but using LINQ for all the
heavy lifting.

We are seeing performance increases of up to 400% by doing this.

Although we have developed and run comprehensive test plans I would be
interested in hearing about anyone using the current ESQL solution with
COBOL, who is not happy with the performance of it.

I'm looking for performance test cases and there will be no charge to you if
you are prepared to let us analyse and use your code and we accept your
example.

After months of work, I am satisfied that LINQ is simply superior to ESQL in
every way and PRIMA is moving to base our COBOL solutions for RDB on this
technology. ESQL will be phased out and replaced by LINQ, generated into C#
COM Assemblies as part of the Data Access Layer and called from COBOL.

There are other benefits in using LINQ as well:

1. LINQ doesn't use ODBC so it doesn't suffer from the connection
limitations (and overall slowness) imposed by this technology.

2. LINQ is pretty much "future proof". LINQ queries can run against data
retrieval technology that hasn't been invented yet, because the queries are
totally object oriented and NOT tied to a specific underlying hardware.
(LINQ can query lists of in-memory objects, XML files, and Relational
Databases, using pretty much the same syntax. It is a generalized object
retrieval and management system.)

3. LINQ queries can be nested (just like SQL) but they are deferred in
execution until the last minute. (You CAN force immediate execution but
generally this isn't wise...)

4. LINQ queries can be treated programmatically as objects and this has
benefits that are way beyond SQL. All of the things that make using objects
useful, are now available with LINQ. (reflection, properties, object
references, etc.)

5. LINQ updates ONLY reference the fields that changed. This is all done for
you using a data context that is "change aware". As a result updates to your
Database require much less bandwidth, impose less overhead, and execute
faster than the COBOL ESQL equivalent. You don't have to build INSERT and
UPDATE statements, LINQ does it all for you, invisibly and as efficiently as
possible.

SQL only and that can be misleading in the "real world".

For example, in C# on one particular test plan, we got 30,000 rps returned.
But that is returning objects, not records. (In effect, it is pointers to
records...) Those same objects have to be expanded into strings that
represent what a COBOL program would see in its data buffer, and that
constructed buffer has to be passed to the COBOL app (or shared with it; we
very soon found out that sharing the buffer was the way to go...)

That is where the time overhead comes in. There are several ways to
translate the object into a COBOL buffer and I wanted the most generic
possible so the first cuts used reflection in C#. This has the advantage
that you can use it for any table set without changing the code, but it is
horribly slow. (In effect, the .NET Framework has to consult metadata at
runtime on every field in the record ). It has currently been replaced by
using the C# StringBuilder Class and this is much better, but it still takes
time. And it means every field has to be "hard-coded" for the StringBuilder.
That isn't a problem because the surrent ESQL solution has to have every
field (column) "hard-coded" into the SQL statements also. As all of this
code is generated and compiled by the Toolset in seconds, it isn't
problematic; if you change a record layout you simply generate a new DAL
module.

Nevertheless, it is quite surprising how the environment slows down the
"pure" SQL Server access.

The original structure for our LINQ solution retained the COBOL DAL layer
with all it's validations and housekeeping. (Doing this also meant that the
interface to applications remained unchanged; you just add some .DLLs (which
are transparent to the applications), in order to support LINQ.) This layer
then called to LINQ to do the heavy lifting.

When we came to performance testing we found that the overhead of going
through this layer dragged things down, so it has been eliminated and the
COBOL apps now call the C# COM .DLLs directly, exactly as they used to call
a COBOL COM .DLL. There is no change to the existing Transformed COBOL; if
it worked with ESQL DAL, it will work with LINQ DAL and the interface
remains identical.

Finally, in performance terms, the LINQ stuff has been coded with
parameterized caching so that if bottlenecks are experienced it is pretty
easy to tune on a specific table set basis.

The final results to date are around 4 times better than the ESQL solution
in terms of performance (and the ESQL solution was no slouch), but they are
not in the same league as you would see from the SQL Profiler.

Work on the final LINQ templates for code generation is ongoing so I might
suggest we try using the SQLProfiler and make sure our generated DBs are as
good as we can make them (we think they are, but it doesn't hurt to look
again), and the actual access is as good as we can get.

Thanks for asking.

It has been the hardest thing I have ever written (and I've been writing
program code for almost 50 years now...)

I tried setting it aside and doing other stuff but it kept gnawing at me
until I went back to it. :-)

I have reviewed the design many times and am satisfied that the architecture
is sound but it is just very hard work.

There are two major components that have proven problematic:

1. The ACCESSOR.

This is the set of actual LINQ constructs that maintain the internal entity
set. It has to support Sequential, Skip Sequential, Random, Insert, Update,
and Delete access for the base table and all attached repeating groups in
the tableset under consideration, and the access has to be fully dynamic
with keys not known until runtime. Writing this code is not so hard, but
writing code to GENERATE this code is a bit more difficult...

2. The TRANSPORTER.

This is a set of methods that transfer data between the entity set and the
COBOL data buffer. It has 2 parts:

1. TransportOUT is concerned with getting result sets fetched by LINQ, from
the entity set into the COBOL record buffer (and thence, back to the calling
program. I have opted to share the COBOL buffer directly with whatever is
calling us, rather than passing addresses to it. This avoids the problems
with stack cleanup and calling conventions in different languages, but it is
"riskier". (or, it would be if I hadn't put things in place to minimize the
risk...)

2. TransportIN is concerned with getting the data in the COBOL record buffer
into the entity set so that LINQ can deal with it, and broker the actual RDB
access.

The structure, type, offset in the COBOL record buffer, and length, of every
field on the database has to be known for the code to be generated, so I had
to write something which analyzes the DDL generated by the Toolset and
creates an XML "data dictionary" of every element stored on the database.
This is then available to the LINQ DAL generator at DAL generation time.
Once the DAL objects are generated they no longer need this data dictionary
at run time.

The original prototype code I wrote used C# reflection to determine the
information for each element. That is now provided by the XML data
dictionary, but this was horrendously slow and 1200 records of around 1500
bytes each, took nearly 30 seconds to process. The new code, generated by
the tool, and NOT using reflection, takes 1.6 seconds to do the same job.
(The first time it ran I couldn't believe the timings and had to carefully
inspect the results before I could accept it had actually done everything it
was supposed to...:-))

The current COBOL/ESQL generated DAL takes around 8 seconds.

LINQ simply rocks...

The main areas that have been troublesome are as follows:

1. It is easy to get confused between what needs to be in the generator and
what needs to be in the generated code. There have been many occasions when
I wrote what I thought was pretty elegant code to do something and then
realized I had written it in the wrong place... :-)

2. COBOL REDEFINES has to be considered. For an optimized database,
REDEFINED fields need not be stored on the RDB, but there is a user option
which allows them to be and it must be catered for. (Similar considerations
apply to group fields.)

3. The use of multidimensional COBOL tables within the record definition is
a nightmare when locating data in the COBOL buffer. If data has been defined
to 3 levels of OCCURS then each level has to be calculated at runtime and
the particular instance of the OCCURS clause has to be derived as an offset
and added into the mix for locating the specific element that is indexed as
(J, K, L). This proved tedious for both TransportIN and TransportOUT and I
tried several different approaches to it. The one that has emerged is robust
and efficient and seems to be working but it took many hours of work and
copious test plans to get it.

As of today, there are no known bugs in the Transporter but there are still
a number of test plans to be run for TransportIN. The total time expended
