COMP-3 is packed decimal.  for your s5(9)v99 field you have 7 digits and a
sign, so it is expecting 4 bytes.  If you were to look at the number in hex
for a value of 12345.67 it would look like 1234567C.  if an even number of
digits are used, the first half-byte is 0.  each 4 bits is an integer value
0-9, except for the sign nybble which is going to be A-D (if I remember
correctly, I always confuse the signs <g>).  Any compare of the COMP-3
field, or a move requiring a numeric conversion, will generate an exception
if the value in the field does not follow the rules.  SPACES, HIGH-VALUES
and LOW-VALUES in the field will cause the exception.  So, either your data
is corrupt, does not match your PIC, or whatever.  You can do a numeric
check on the field and move it to the display field when it is numeric and
move zeroes when not numeric, but you should really validate the data.
Maybe your file has a header record with field names or space records for
display purposes or something.  Figure out what in the data is causing it
and decide how to handle it.


Glenn

Hi, here is a little bit on numeric definition, -use and -hints: .

Definition of COMP-3 fields (IBM EBCDIC rule, though):
COMP-3 means packed decimal representation. Instead of one digit per byte,
you have two, except for the loworder byte, in which the loworder half-byte
contains the sign. This example shows how the number -1234 will be stored
using dislplay format and packed format:

PIC S9(5)               :  5 bytes   F0 F1 F2 F3 D4
Each 9 represents 1 byte. High-order part of loworder byte contains  the
sign. Value for + is hexa F or C, - is hexa D or E

PIC S9(5) COMP-3:  3 bytes  01 23 4D
Each 9 represents 1 half-byte. Sign in the rightmost halfbyte.

USE OF COMP-3 fields:
Like any numeric defined fields, it is the programmers responsibility to
assure numeric content according to the field definition (packed, decimal,
binary etc) before any references are made to it. In general you have no
guarantee that the field contains only numerics according to the format you
have specified.y.
If , by example, you initialize your record by moving spaces to the 01-level
field, the packed defined field will look like
40 40 40, and 0 is not a valid sign code. INITIALIZE would be better. Then
the fields will be set to the specified VALUE, or spaces/zeros according to
the elementary item definition.

HINT:
If you cannot know what the contents of the field really is, you should
perform the IF field NUMERIC test before further referencing the field. IF
NUMERIC is true when the content is numeric according to the field's usage
clause.

SOME OTHER HINTS RELATED TO NUMERIC FIELDS:

1. Very often a field will be used both numeric and alfanumeric. Often you
will see clumsy definitions like
02 ALFAFIELD PIC X(8).
02 NUMERICFIELD REDEFINES ALFAFIELD PIC S9(8).

Why not remove the confusing REDEFINES and specify like this?:
02 FIELD-ALFA.
03 FIELD-NUM   PIC S9(8).
In COBOL, the default USAGE for group items is alphanumeric.

2. Avoid COMP-3 if possible. The reasons for using it is not as valid as
before, and different platforms may treat it differently.

3. Avoid "numeric programming mode".
A good thumb-rule is to stribe after working with alphanumeric fields only.
This rule should be modified, of course, when the field is to be used in
arithmetics or for table indexing or report numeric editing. At least you
will reduce the 0c7 abend risk by doing so, and in maintenance you may have
less "numeric problems" bothering you .

Good liuck
Bjorn
