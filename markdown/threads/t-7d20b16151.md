So true.  In addition, maintenance chores are easier.  For example, if
a value is referenced repeatedly, it is better coded in the DATA
DIVISION rather than in the PROCEDURE DIVISION.  Should the value
change in the future, you will have to make the change in only one
place, rather than every time it is referenced.

The places where initialize can "cost" are (1) large complex group items
where you really only need to initialize some fields, but choose to
initialize all fields because the INITIALIZE statement is "so easy"; (2)
multi-dimensional Group items with occurs-depending-on items with many
subordinate occurs-depending-on elements.  The INITIALIZE statement may
generate the equivalent of a PERFORM loop for every OCCURS DEPENDING
element, so complex multi-level structures can generate a lot of control
code.  If the INITIALIZE is only done once in a batch program, this
probably is not worth worrying about.  If done repeatedly during the
execution of the same batch program, or if it is in an on-line program
executed 1000's of times daily, then there may be better alternatives. 
One common trick is to create a single group item structured like the
outer-most group OCCURS item, initialize that single group item and use
it in a single loop with a group-item move to initialize the larger
structure (a really clever compiler could be smart enough to do this
optimization for you, but unless you know this to be the case for your
compiler, assume a separate move for each elementary field item and a
loop control structure for every OCCURS clause).  Another technique is
to avoid having to initialize all elements of a large array by keeping a
"max index" pointer and designing the program logic so that you only
reference items into which actual data has been stored. If a complex
structure only needs some items initialized, you may also be able to
improve on efficiency by separating the group structure into two
separate structures, separating elements requiring initialization from
those that don't.

If you have a group item containing no OCCURS clauses in itself or any
subordinate fields, and if all fields in the group item really require
initialization, and if the default initial values do the job, then
clearly the simplest and most efficient way is the INITIALIZE
statement.  Other cases may require some thought and analysis if
efficiency is a prime concern.

The initialize verb (spelt wrong because of the yanks) is a very handy
tool if it used where it really needs to be used.  All to often I see
some coders put an initialise (sorry) at the beginning of a paragraph
that fills every field with no if's, but's or maybe's.  Know your code
and don't waste any steps.

Consider the order of your fields.  There is an instruction to load a
single character across a large section of storage by one command, so
put you X fields together (with no fillers) if possible.

If you can guarantee that the numeric fields are filled and the other
fields needs space use 'move space' and then do your numeric moves, but
this is possibly a bad idea from a maintenance point of view.

The use of initialise with fillers is great for date formatting

	dd pic x(02)
	filler pic x(01) value '/'.
	....

Once again if you are storing the year month and day you just don't need
the overhead of an extra useless step.

Initialising group fields can be very confusing to a beginner, we have
all been caught by thinking that we have tracked the modifications of a
field to find that it has a group initialise, sometimes it is clearer to
explicitly clear fields, eg:

	01 ws-counters
	   05 record count
	   05 detail count

when moving between accounts you need to clear these counters, sometimes
(not always) it is better to be explicit and clear using two move zero
statements.  Consider the next person along and not the shortest path to
a coded result.
