If I understand your question, it's the application that responds to
the Alt-N that "%{N}" sends.  It happens to be what the author of the
Dialog box in question decided to accept.  In older versions of
Windows it was usual to indicate the shortcut keys with an underline
(thus the "No" button would have the N underlined, but that was not a
hard relationship - a perverse app could put the underline on the N
and then accept Alt-Q instead), although more recent apps tend to omit
that fairly frequently.

MS has published style guides about all of this stuff:

http://msdn.microsoft.com/en-us/library/aa511258.aspx

http://msdn.microsoft.com/en-us/library/aa511453.aspx

http://msdn.microsoft.com/en-us/library/bb545460.aspx#dialog

And they specify that applications *should* assign a unique shortcut
(access key) to most buttons (some exceptions are OK and cancel, which
have standard assignments - enter and esc), and for "yes" and "no"
buttons they specify:  "For example, the access key for Yes and No
commit buttons should always be "Y" and "N", regardless of the other
controls in the dialog box."

Of course that doesn't prevent the author of an application from doing
something completely different.

The answer to your question is that he knew it because it is standard 
Windows behaviour.

Look carefully. Both "Yes" and "No" have a little underline under the Y and 
the N.

These are "keyboard shortcuts".It means that if you press Y the system will 
activate the "Yes" button, and if you press N it will activate the "No" 
button. The script has opted not to save the document, as it is an example. 
Try changing the script line to: WshShl.SendKeys "%{Y}" and see what 
happens... :-)


When you construct buttons for Windows forms you can specify what the 
keyboard shortcut will be for that button. It doesn't have to be the first 
letter... For example, say I had a button labelled "Magic".  I can specify 
any of those letters to be the keyboard shortcut.  If I chose g, the letter 
g on the button would appear with an underline and if you pressed the g key, 
the system would behave as if you had clicked the "Magic" button.

Sometimes, when you have many buttons on a form, there can be a conflict 
between the keyboard shortcuts. So you choose a different letter.

Why doesn't Cancel have an underline?

Because "Esc" is the standard default for cancel. Any button labelled Cancel 
will be activated by default if you press Esc.

Note that, as a Windows developer, I can assign a keyboard shortcut to 
Cancel, just like any other button and I could use C, for example.

Good Windows form design will have a keyboard shortcut for EVERY button, but 
many developers neglect this or don't know how to do it. (It can be tricky 
as it varies between WPF or Silverlight where XAML is being used to 
construct the form, and standard Windows Forms.You can also specify at 
design time that you want a combination keyboard shortcut and Alt is usually 
chosen. So it is Alt+<whatever>. This is particularly important if you have 
input fields sharing a form with your buttons, as you don't want typed input 
to activate a button on the form..

In the Wordpad dialog box there is no possibility of confusion with anything 
else as there are no entry fields that require typing so it doesn't require 
Alt+<default key>.

Hope this helps.

Pete.
