Anytime you hear so many BUZZ words strung out together you have to look at
the title of the person making the statement. If their is BUSINESS anywhere
in the title you can almost be sure they do not know what they are talking
about.
VB has a RUNTIME MODULE which is called be the VB pseudo code it generates.
All standard VB compilers produce non functional code which to be functional
must call a RUNTIME module which performas all the basic functions of the
language.
COBOL produces functional and independant executable code which does not
require any runtime module to run. With possibly a few exceptions.

To be Y2K compliant, whether it be VB or COBOL all executable programs or
runtime modules called must use 4 digits to represent the year. They also
must never convert that date back to 2 didgits except to output it to a
report, which may still cause problems.

Earlier versions of COBOL are not Y2K compliant simply because they return a
2 digit year from their SYSTEM date. There was no date computations built
into earlier versions of COBOL other than returning the date from the system
clock.

Regardless of what language is used, if the date computations are not using
4 didgits for the year, or an equivalent representation, none of the code
can be considerred Y2K compliant unless it does not use dates in any manner.

What the manufaturer is trying to say is that,"We are not willing to make
our code Y2K compliant. You will have to find a way around our unwillingness
to fix our code. We will however continue to charge you for licensing and
support of our NON-Y2K compliant code."

He also could be saying that the code you guys are responsible for is not
Y2K compliant and they can't be held responsible for Y2K damages until you
guys fix your code.

I have never heard any COBOL executable code referred to as a COBOL COMPILER
RUNTIME. He also could be referring to your COBOL compiler. Whic as I
previously stated may not be Y2K compliant. However this can always call a
Y2K compliant application to retrieve a proper 4 digit year. Then as long as
the 4 digits are mantained throughout the computations and data storage
everything comes up roses.

Program written in high level languages are insulated from a lot of the
operating system communication required to have a program run, i.e. the stuff
an assembler programmer has to explicitly code.

The cobol run time will typically:

   Allocate storage and buffers
   Provide error handling
   Assist with more complicated verbs
   etc.

Now will this cause you trouble? Hard to say.

1. Are you using any of the assists to get date info? 

If so, what format is a Y2000+ date returned, and can you code around it?

2. Do any of the assists require correct date info?

In this case you might be in deep ^@%$^$%$ and you may have no choice but to
go to a different compiler.
