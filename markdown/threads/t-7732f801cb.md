This is an interesting question.  The simple answer is that there is almost
a different answer for every file.

Certainly the most common place is that MOST files that you work with are
actually either input files that already exist (so the definition is already
defined) or output files (print or disk/tape) that someone else tells you
what they need/want.

HOWEVER, somewhere there must be someone who is figuring out what new files
look like.  I can think of three (common but not universal) ways that these
are designed:

1) Print files - the "record size" is usually determined by the printer(s)
to which it will be sent.  The actual layout is usually designed by an
"analyst" in consultation with the end user.  (The analyst MAY be a
programmer/analysts - so it may be you.)

2) Databases (of the relational or hierarchical types) - There are entire
methodologies established on how to design these.  In most (but not all)
cases, the efficiency of the database engine has almost as much to do with
the user/program needs in how these are designed.  The term "DBA" refers to
a database analyst who will usually work with a senior programmer/analyst to
figure out how to set these out.  It is important to understand when
designing these that the same database will (almost certainly) be used by
multiple applications - therefore, the design may include fields that your
program/application will never use - and the design may be other than what
you would expect for your specific program.

3) Traditional COBOL output files (sequential, indexed - and rarely IMHO -
relative) - mostly these go thru the same process as databases listed above.
However, it is more common for these to be specific to a single application
(or even program).  Therefore, it is more common for the individual
programmer/analyst to set out their layout.

Almost right.  IBM cards, the most common, were designed
around the EBCDIC system for representing character data. 
Each column in a card represented a single character (not a
bit).  There are 12 rows on each card.  The top two rows are
called "zone" punches; the remaining 10 rows are "numeric"
punches - each row representing a particular digit. Digits
are encoded by punching the correct digit.  Alphabetic
characters and punctuation marks are punching a hole in a
zone row and one in the numeric row.  One source that
provides more detailed description than this is Austing and
Cassel "File Organization and Access" published in 1988 (see
page 106).  

I have never heard of 128 byte cards - anybody know how
these were formatted?  Smaller holes, bigger card, different
standard?  I do know that other manufacturers used other
different sized cards - though I do not know which
manufacturer, I remember seeing one that used a 3"x3" square
format.  One of the reasons IBM was so successful as a
computer company was that they were using 80-column "IBM
Cards" in their mechanical accounting machines long before
they began making computers.  When a company was ready to
install a computer, they could use the same data they had
always used - i.e., the IBM cards.  No conversion was
necessary to a new format card.

The 128 byte format was also started by IBM.  This is/was the format
used in their first generally available floppy disk entry unit, the
3741.  This was an 8-inch floppy with 78 tracks (1 sided), 26
record/track, and 128 bytes per record.

The 3741 was also used as a primary input unit to the System/3 series of
IBM mid-range systems along with the 3x3 card format that had 3 lines of
32 columns with 6 rows per column.  (The card format died because it
wasn't capable of holding all possible values for a byte.)

IBM JCL (OCL on the System/3) was kept at 80 bytes due to inertia (there
are still companies out there running 1401 emulators), but I think that
the latest version of OS390 allows different sizes, but I've not seen it
otherwise (more inertia).

There is an advantage to using LRECL=80 in any simple MVS business
application.  JCL(Job Control Language) is processed as 80 byte
"cards".  If my input file is defined as LRECL=80, then the following
JCL option may be used to feed in my hand-keyed test data:
	//MYINFILE DD *

This is much more convienent than having to allocate a separate file and
populating it with my test data.  If my data requirements are less than
80, I will always pad it out to 80 for the above reason.

Pete
