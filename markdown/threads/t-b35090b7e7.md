```
The 2002 standard states that debugging lines and DEBUGGING MODE are
now obsolete and are due to be removed from the next (2010?) standard.

I have used it quite a lot myself because I found it very useful.
Although one can use interactive debuggers, which have become much
more reliable over the years, it can be hard to thoroughly check all
relevant work areas during the course of execution when stepping
through interactively, while having a listing (it doesn't have to be
printed) of snapshots taken at judicious points of execution can helo
make such checking more thorough.  My personal preference is to use
both facilities when and as appropriate.  The "D" before each
debugging line does make it clear what is a debugging line, which
conditional compilation doesn't in the same way.  If I ever wrote
programs again, I would probably also use conditional compilation as
well, when it becomes available.  One problem to be wary of with
either debugging lines or conditonal compilation, is ensuring that
final testing of the program with and without the debugging lines is
comprehensive, it can be an easy mistake to have something set in a
debugging line that is required when the debugging facility is turned
off.  Self discipline can mitigate this risk.
```

```
My (vague) memory was that IBM customers that I talked about thought it might be 
"nice" but certainly not a high priority (compared to other things in the '02 
Standard - like 1-byte binary fields, bit fields, REPLACE ALSO, etc).

As far as the "debugging" facility goes (debugging lines), most (certainly not 
all) IBM mainframe people do "interactive debugging" rather than
    D DISPLAY whatever
these days.

Micro Focus first introduced THEIR conditional compilation because they used it 
internally.  The compiler itself (written in COBOL) used it so they could 
"single source" the code for various target platforms.  (Same source code did 
multiple branches of "source code streams" depending upon whether they were 
building a compiler for DOS, OS/2, HP,whomever).  Although you can do this 
checking at "run-time" - it makes (in most cases) for more efficient code if you 
only include the "relevant" source code in what gets compiled.  Using different 
COPY members is another approach used by some people, but again conditional 
compilation makes this less "error prone".

In general, it is a "nice" feature for anyone who wants to maintain a single 
source stream but compile it differently for different run-time (object code) 
variations.

P.S. One thing that really bugged me about how ANSI/ISO implemented it, was that 
they didn't include a
  >>DISPLAY
statement to let you "print" a message in the compile-time listing - so that you 
could figure out which branch of the conditional compilation you were "on".
```

```
Many years ago we (internally) did a preprocessor (in Cobol), to
support conditional compilation.  It did the basic .IF stuff, nested
conditions, and a few other items.  Nothing too fancy, and a bit
rough, but we've been using it internally for years.  Both PC and
mainframe versions.  If people were interested, I could see if there
is interest in OSS'ing it, maybe we could do it as a SourceForge
project.

Anyway, here's some junk source code I slapped together as a sample:

       INPUT-OUTPUT SECTION.

           .PRINT 'DISPLAY COMMENT ON CONSOLE'

           .IF OS2
               SELECT SOURCE-R-FILE ASSIGN TO VARYING DS-SOURCE
                   ORGANIZATION IS SEQUENTIAL
                   ACCESS IS SEQUENTIAL
                   FILE STATUS IS SR-STATUS.
           .ELSE
               .SETTRUE COMPUTEDCONDITION
               SELECT SOURCE-RRR-FILE ASSIGN TO VARYING DS-SOURCE
               .IFNDEF UNIX
                   ORGANIZATION IS SEQUENTIAL
                   ACCESS IS SEQUENTIAL
                   FILE STATUS IS SRRR-STATUS.
               .ELSE
                   ORGANIZATION IS SEQUENTIAL
                   ACCESS IS SEQUENTIAL.
               .ENDIF
           .ENDIF


           05  OUTPUT-RECORD-R-2 REDEFINES OUTPUT-RECORD-TABLE.
               10  FILLER                       PIC X(06).
               10  OTR-COMMENT-1                PIC X(02).
               10  OTR-VAR-NAME                 PIC X(32).
           .IF COMPUTEDCONDITION
               10  OTR-VAR-DEF                  PIC X(12).
               10  FILLER                       PIC X(04).
               10  OTR-VAR-STATUS               PIC X(05).
               10  FILLER                       PIC X(04).
           .ENDIF
               10  OTR-USED-COUNT               PIC X(03).
               10  OTR-COMMENT-2                PIC X(02).
               10  FILLER                       PIC X(186).
```
