Yes, Ronald, that is indeed the rule if you want REDEFINES and OCCURS to
play nicely together...(It isn't quite that rigid and can be broken as long
as you know exactly what you are doing, but it is a VERY GOOD rule for
people new to COBOL to remember...)

So why did you break it with the sample you posted below, where you have
Value/Occurs/Redefines <G>?
(And you haven't done the things you would need to do to make this work...).

It is hardly helpful to offer a rule, then immediately give an example that
violates it, is it?

Try this...

01  tline.
      12 tline-group pic x(79) value space.
      12 varlen-tline redefines tline-group.
           15 vtline-char pic x  occurs 79
                                          depending on sub-5.

It should be pointed out that this is a completely pointless exercise.
Nothing will be gained by using OCCURS...DEPENDING ON (ODO). If you really
want to pick up chunks of tline that vary in size up to 79 bytes, and given
that the required length has been set into sub-5, then use Reference
Modification. This avoids the use of ODO altogether and gives a much better
result...

01  tline      pic x(79) value space.
...
move 12 to sub-5
move tline (1:sub-5) to ...

this will move 12 characters from the start of tline to wherever you want.
Setting any value, up to 79, in sub-5 before the move will do likewise.

If your requirement is to do something with EACH CHARACTER, limited by the
number in sub-5, then:

perform varying
     J from 1
          by  1
        until  J > sub-5
             if tline (J:1) is interesting...
                ...do something with this character...
             end-if
end-perform

As you can see it is not ALWAYS necessary to use OCCURS and REDEFINES (never
mind ODO) at ALL, in order to get at varying data chunks. Sometimes in fact,
it is better NOT to...

Hi Markus,

With regards to your program, there seems to be a bit of problem in your
definition of working storage items.  As far as I know, OCCURs DEPENDING ON
clause are used for defining variable length tables and in your sample
program, redefining a fixed length working storage item with a
variable-length table is wrong or are not allowed by cobol.  Likewise,
OCCURS on 01 level are always ignored or are not allowed by Cobol compilers
(I'm saying this based on my Cobol experience on mainframe).

More proper way of working around with it would be
01  SUB5            PIC 99.
01  WS-FIX-ITEM     PIC X(79).
01  WS-VAR-ITEM.
    05  WS-VAR-ITEM-FIELD  OCCURS 1 TO 79  DEPENDING ON SUB5  PIC X(1).
01  PETER.
    12 PETER-GROUP        PIC X(79)  VALUE  SPACE.
    12 VARLEN-PETER       REDEFINES  PETER-GROUP.
       15 peter-char      PIC X      OCCURS 79.

Collections do not REQUIRE arrays or tables. TUPLEs are instances of a
relationship, which is often implemented as a row in a Relational database,
or an element of a linked list. Usually the tuple consists of a key (showing
its position in an ordered set) and some related data. That's why they are
often implemented as rows (Relations) on an RDB. Tuples are not new, they've
always been a part of the Computer Science terminology for OO Collections.
The sets to which they belong can have an indefinite number of
"occurrences". (And, in my opinion, they should.)

The newly emerging Languages for parallell computing (like LINDA) are doing
interesting things with tuples like allowing asynchronous access to the
Collection.

The concept of using OCCURS is archaic and there is no need to limit systems
because of a COBOL construct. You know as well anyone here, Jimmy, that
collections can serve the same purpose as tables, with the advantage that
they are unlimited and dynamic and can have elements added, deleted, and
changed easily and freely, using the methods inherent in the collection. The
OO Concept of a collection is a more than adequate replacement for the
concept of a Table defined to a specific limit with an OCCURS cause. It is
the constant changing of these table limits that introduces errors into
programs and is the instigator for a large percentage of maintenance. If we
didn't have tables in memory, there would be no need to change them...

While I do not wish to denigrate Eiffel (I managed a project where it was
used to great effect) the fact that it doesn't support variable arrays (as
COBOL doesn't either) has no bearing on the use of collections or tuples.
Only when programmers get it into their heads that they can load a
collection into memory is there a requirement for an array. This is just old
thinking...manage the collection from wherever it is located! The whole
point of the collection is that its physical implementation is irrelevant.


>And yet given that ODOSLIDE

element,

No it isn't. It just seems that way <G>.

If you are determined to have the collection in memory you can get the same
result without ODOSLIDE. In either case you will have to allocate a maximum
occurrence. ODOSLIDE determines the location of
the item following the item that has OCCURS...DEPENDING ...ON in it, (but is
not subordinate to it). This is achieved by relocating the whole structure
at run time. It is an horrific overhead for a very little gain (a few bytes
of memory) but it does let you think that you have actually defined a
variable array... It isn't truly variable because what happened was that the
maximum size was allocated and then "adjusted" to suit. A Collection on the
other hand never  (well, it shouldn't...) allocates space (in any media)
that it doesn't need.

There is a difference between an unlimited number of elements and
"n-dimensional" also...

Markus,

There is a (fairly minor) difference between:

        01  TNUM PIC 9(18).
        01  FILLE1 REDEFINES TNUM.
            05 FILLE PIC 9 OCCURS 18.

and

    01  TNUM.
        05 FILLE  OCCURS 18 DEPENDING ON SUB5 PIC 9
           VALUE ZERO.
        05 FILL1 REDEFINES FILLE              PIC 9(18).

In your "new" coding TNUM is a numeric field and FILL1 is an alphanumeric
item.  In the original version, this was reversed.  If you need/want both an
alphanumeric and a numeric definition of the "whole" 18 characters, then you
should code it as:

  01  FILL1 PIC 9(18).
  01  TNUM REDEFINES FILL1.
        05  FILLE  OCCURS 18 DEPENDING ON SUB5 PIC 9
                          VALUE ZERO.


This should leave all the fields with the same attributes as the original
version of your code (and should compile cleanly)

That's a very good point you raised, Jeff.

It made me think and I came up with the following observations:

1. Linear table searches are usually on small tables. (otherwise we would
load the table sequentially and use a binary chop on it.) Personally, I use
SEARCH to do linear searches and SEARCH ALL to do binary chops.

2. There is a run time overhead in referencing items declared with ODO. This
overhead applies to every entry that is referenced and could therefore be
more than the "saving" that was expected by using ODO to "limit" the search
in the first place.

3. Modern processors, doing a linear search on a small to moderate size
table in memory will show very little gain if the search is limited by an
ODO. So our reason for doing it is really an aesthetic one (it makes us feel
good to know we have optimized something to the limit of
reasonability...<G>) This "limiting" of the search is based on the doubtful
proposition that what we are looking for is NOT there. If we KNOW that what
we are looking for IS there, then nothing at all is gained by limiting the
search. Similarly, if what we are looking for is there say, 90% of the time,
the overheads incurred in limiting the search  to accommodate 10% of the
access, may not be justified.

4. There is nothing to stop us from counting entries as they are loaded into
the table (or defining a table size variable if it is a static table defined
in WS). We can still use it to limit the search without the overhead of ODO.

some examples...

01   my-table.
       12 table-entries   pic x(30)
                               occurs 50
                               indexed by mt-x1.

01   my-table-last    pic s9(4) comp.

01   found-flag        pic x value space.
       88 foundit                 value '1'.
       88 not-found             value '0'.
......

*> the table has been loaded elsewhere and my-table-last was incremented
with
*> each entry. It now contains 23...
*>
      set mt-x1 to 1
      search table-entries
            at end
                    set not-found to TRUE
            when
                    mt-x1 > my-table-last
                    set not-found to TRUE
            when
                    table-entry (mt-x1) = thing-im-looking-for
                    set foundit to TRUE
      end-search
....

While this looks reasonably elegant at the source code level, and it
achieves the goal of NOT searching 27 unused entries in the table if what we
are looking for is not there, it also incurs an overhead... the table index
is checked on each iteration to see if it has reached the limit. With ODO
the WHEN clause that does this would not be necessary. However, with ODO,
every entry would need variable addressing with the concomitant register
loading and arithmetic for indirect addressing, so there is an overhead
there too. The point is that there are run time overheads that should be
balanced, whichever way you do it, IF you CARE about run time overheads!
(Since the invention of the Pentium Gig processor, I have stopped worrying
about it... let the sucker work; that's why it was invented..<G>)

These days I am much more concerned with readability and source elegance
than balancing marginal processor overheads.
(It is interesting how this emphasis has changed over years... in the 60s,
run time efficiency was EVERYTHING. That's why many people resisted COBOL as
it gained popularity. It could never reach the purism that Assembler could.
Bear in mind that many of the processes, procedures and practices which
today are taken for granted had not been formulated then... the last thing
on our minds was ease of maintenance. If you were a programmer you should be
able to understand program code without it being written with maintenance in
mind...By the late 70s, with the advent of serious transaction processing on
mainframes, we were still pretty obsessed with getting every bit of
performance, but it was tempered a bit by the knowledge that we might have
to come back to this code in a few years time so a few comments in it
wouldn't hurt...<G>. Today, I find that being able to maintain the code I
wrote years ago is pretty important. Nowadays I am packaging it as
components and eventually Web Services so, although the actual source code
is not important to other people, it is to me. The march of technology,
driven as always by demand in the Market Place, has brought us processors
where it really DOESN'T matter too much if something is not totally
optimized in source. None of us can tell whether a process takes 100
Nanoseconds or 110... (Please don't flood CLC with mail saying I am
advocating sloppy programming and it is OK to write crap code; I'm not. I'm
merely advocating a realistic balance based on what is available to us.)

For these reasons I shall continue to avoid ODO, but after reading your
post, Jeff, I MIGHT be inclined to limit serial searches, as shown above.
