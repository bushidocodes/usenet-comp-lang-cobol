I'm in the process of designing a program for a client that must run
in a CICS region and will be accessed by several different
applications.  Its purpose will be to take a data stream sent to it,
reformat that data stream into a common transaction layout and then
send the whole thing off to another computer in another place and
await a response and report back to whomever accessed this program.

I am trying to anticipate all the needs of this program before actual
specifications are delivered so that I can head off any technical
problems in the very beginning of the project.

One such requirement I am anticipating is the manner in which the
program will be accessed.  I currently believe the client will need at
least two ways.  1) through MQ Series; 2) via CALL or CICS LINK.
Since the method of retrieving a data stream and returning same would
be different, the program would need to know how it was accessed.

For the life of me, I cannot think of a mechanism to accomplish this
and I'm not even sure it is possible.  If someone out there in this
most diverse group has knowledge of this one way or the other, could
you please share?

Thanks.

You have a few different things to check.

The first, STARTCODE, might tell you some things, but it will apply to 
the entire transaction.  E.G. SD will tell you started as a background, 
with data -- but will not tell you that MQ did it.

I think your best bet might be to do a RECEIVE and see if you get an 
END-OF-CHAIN condition (CALL) or an MQ Trigger Message structure.
