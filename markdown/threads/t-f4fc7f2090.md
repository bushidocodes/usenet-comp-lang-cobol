numeric format has an unused bit; if this bit is one should NUMERIC be
true?

To augment, Mr. Castro's reply, USAGE BINARY, or COMP, is *always*
numeric, and in the above example, it will always have a numeric value
when executed. Depending upon the vendor's implementation, the value of
BIN (and, this may be a RESERVED word in some compilers) may start as
zero or whatever the uninitialized allocated storage contains. Thus,
the
first logical ref to BIN of PROC DIV should be COMPUTE BIN = 0 or
MOVE 0 TO BIN.

USAGE BINARY can also be used to represent PIC s9(9)V99 fields, and
the computation proceeds much faster than COMP-3. It is useful for
cascading accumulators where a group re-initializing can be a simple
MOVE LOW-VALUES TO.... which still beats the INITIALIZE verb.

Just to make the "attached" response clearer.

A) This (allowing an IF NUMERIC test against a USAGE BINARY item) is a
substantive change in the draft Standard - the '85 Standard does *not* allow
an "IF NUMERIC" test on a USAGE BINARY item - but the draft does.

B) As the attached quote indicates, this test does "validate" (actually
VALIDATE is a new verb - so don't confuse this with that) that the value in
the item matches both its USAGE and PICTURE. A BINARY item will (with every
system I have ever heard of) *always* match the USAGE (any old combinations
of bits is a valid binary item) HOWEVER, many bit patterns that would be
valid binary numeric values will NOT pass this test because they do not
match the PICTURE clause. For example, given the question
77 BIN PIC 9 BINARY.

If BIN includes a hex value of X"0A" this would be a valid BINARY numeric
value - but would NOT match the PICTURE clause which requires a one-digit
value (expressed in base 10).

C) The draft Standard does introduce a bunch of new USAGEs that allow for
USAGE only (without any PICTURE clause) for handling all the valid values in
a binary item. For example,

77 Bin-Char Usage Binary-Char.

would pass an IF NUMERIC test if the field included X"0A" - or even X"AA".
(I can't think of any time that a USAGE BINARY-CHAR would ever fail an IF
NUMERIC test.)

I hope this helps explain what is standard today versus what will be allowed
in the (hopefully near) future. There are, obviously, several compilers
that do already allow (as an extension) an IF NUMERIC test on a USAGE BINARY
item - some of them validate against PICTURE and some do not.

Far be it from me to speak on behalf of the developers of COBOL85 on A
Series ...

but ...

as someone who is concerned with ensuring the consistency of COBOL
compilers with their corresponding applicable standards ...

My strong opinion is that a COBOL85 test for NUMERIC on a USAGE BINARY item
should either (A) receive a syntax error, based on the standard (see
below), or (B) produce well-defined and well-documented meaningful and
useful behavior along with a syntax WARNING that the construct is outside
the bounds of the standards.

In strict terms, "IF BIN IS NUMERIC" (or for that matter ALPHABETIC,
ALPHABETIC-UPPER, ALPHABETIC-LOWER, or a class-name defined in
SPECIAL-NAMES) is a syntax error if BIN is described as anything other than
USAGE DISPLAY. This is based on ANSI X3.23-1985 Page VI-57 Paragraph
6.3.1.2 "Class Condition" subparagraph (5) which states that "The usage of
the operand being tested must be described as DISPLAY." There's no
ambiguity in that statement.

I agree that it would be laudable for an extension that provided some
documented useful behavior in this case to do so in a manner that was
consistent with the draft, but as COBOL85 is not obligated to adhere to
standards that have not yet been solidified, the developers actually have a
free hand in this matter.
