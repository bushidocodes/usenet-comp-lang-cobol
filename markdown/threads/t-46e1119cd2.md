Using RAND functions for any sort of testing has not been a profitable
excercise in my experience.  Real world data distribution and workloads are not
even pseudo-random; and using randomly generated values and/or volumes can lead
to invalid test conclusions.

For one example, consider a randomly generated account number for use in volume
testing a credit card or ATM switch.  As a rule, these account numbers need to
pass check-digit routines and range edits.  Should you simply spew randomly
generated account numbers at the switch, then the reject rate will be
artificially high. This could lead to spending a great deal of time tweaking
error routines that will only be called every few thousand transactions in
actual duty.  If you start by pre-filtering your account numbers, then you may
as well simply build an iteration loop to generate the account numbers.

The same problem holds for Sort routines.  Algorithims that beautifully sort
keys of even distribution can be deadly pigs when dealing with clustering keys.
 Random generation can lead you  astray in your performance (and sometimes your
functional) testing.

As for Monte Carlo simulations, I must admit that my only brush with them was
in a  'C'  class taken roughly in the mid Pleistocene.  Still, were I
mistakenly selected to do any serious statiscal probability analysis, (and I
would be the first to call out that I would be in way over my head trying to do
that) I would not depend on the standard library function to deliver
sufficiently sophisticated pseudo-random numbers.

Assuming we are talking about the IBM COBOL for this-and-that compiler(s),
then the answer is NO, there is no way to get the compiler to "optimize for
size" (and given OS/390 architecture - I can't really imagine why you would
want to).

FYI, here is what the COBOL Programming Guide says about the OPTIMIZE
compiler option,

"The COBOL optimizer is activated when you use the OPTIMIZE compiler option.
The purpose of the OPTIMIZE compiler option is to do the following:

Eliminate unnecessary transfers of control or simplify inefficient branches,
including those generated by the compiler that are not evident from looking
at the source program.

Simplify the compiled code for both a PERFORM statement and a CALL statement
to a contained (nested) program. Where possible, the optimizer places the
statements inline, eliminating the need for linkage code. This optimization,
known as procedure integration, is further discussed in "PERFORM Procedure
Integration" in topic 4.6.2.1.1 and "Contained Program Procedure Integration"
in topic 4.6.2.1.2. If procedure integration cannot be done, the optimizer
uses the simplest linkage possible (perhaps as few as two instructions) to
get to and from the called program.

Eliminate duplicate computations (such as subscript computations and repeated
statements) that have no effect on the results of the program.

Eliminate constant computations by performing them when the program is
compiled.

Eliminate constant conditional expressions.

Aggregate moves of contiguous items (such as those that often occur with the
use of MOVE CORRESPONDING) into a single move. Both the source and target
must be contiguous for the moves to be aggregated.

Delete from the program, and identify with a warning message, code that can
never be performed (unreachable code elimination).

The FULL suboption requests that the compiler discard any unreferenced data
items from the DATA DIVISION, and suppress generation of code to initialize
these data items to their VALUE clauses. "

   ***

If you are interesting in IBM's recommendations (which stress "GOOD
algorithms" over compiler options), I suggest that you look at:

 http://www.s390.ibm.com/bookmgr-cgi/bookmgr.cmd/BOOKS/IGYPG204/4%2e6

which is the topic
  "4.6 Chapter 24. Program Tuning"

in
  Title: COBOL for OS/390 & VM Programming Guide
  Document Number: SC26-9049-04
