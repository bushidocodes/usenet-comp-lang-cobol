Hmmm...guess I must've missed that.

 I am currently involved in working with SOAP and am delighted to find that
Fujitsu Version 6 makes it extremely easy.  (It will generate the necessary
XML "wrappers" without knowledge of this technology being required on the
part of the User)

Thane has been using it for some time and advised me to do so ages ago.

He was right (as he usually is...<G>).

For some time now I have advocated the use of components and have built my
own COM and ActiveX components for deployment on both the desktop and the
Web.  The problem here is that nonMicrosoft platforms have difficulty
deploying these objects. (It can be done with CORBA but it is not an
exercise you would not greet with enthusiasm...).

The Simple Object Access Protocol (SOAP) enables these components to be
"wrapped" in a way that makes them more acceptable than the COM wrapping
that is usually served up with ActiveX. (The term "ActiveX" is falling into
disuse as OLE takes over the functions of what used to be DDE (Dynamic Data
Exchange). The rapid expansion of the whole component based architecture
field has led to much confusion in the various terms. This confusion can be
avoided if you remember that COM means "Component Object Model", in other
words an architecture for building components. They are actually implemented
using On-line Linking and Embedding (OLE) so this is the "wrapper" that goes
arund them and enables them to be served from various servers.)  So, SOAP,
(through the artifice of XML) enables "objects" (components, usually) to be
deployed locally and remotely across different platforms (UNIX, Win32, and
mainframe eventually...)

This is a bonanza for the "mining" companies...<G>  Grab the Legacy code,
wrap it in SOAP, and Hey Presto! Web Enabled Legacy, without any Java in
sight! Makes the COBOL -> Java tools obsolete. (Well, at last as far as
justifying them because it is the ONLY way to Web Enable COBOL...there may
be other, less readily apparent, reasons why you MIGHT want to convert your
Legacy COBOL to Java...)

Of course, if you are using Java you might also want to wrap your Beans in
SOAP to ensure a wider distribution for them. (I think the course that is
advertised in Bill's post (see below) is about using Java with SOAP...pity,
it would be more interesting if it was about using COBOL with SOAP, but I
believe only Fujitsu have integrated this capability into their COBOL
product so far. In fact, it is possible to wrap components written in Java,
VB, C++, DELPHI, or Fujitsu COBOL in SOAP currently.(I don't know about MF,
it should be possible but it may not be easy.)

So why is the mainframe late in getting this technology? Why are they only
talking about using it with Java on S390?

I'll tell you.

Because the whole underlying concept of this approach is OBJECT ORIENTED!

And fortress COBOL programmers (the Dilworths of this world) take to OO the
same way short guys take to nude beaches ("all right for some, but you won't
catch me there..."). (Think about poking your nose into everybody's
business...<G>)

If you intend to be a programmer and you haven't started getting to grips
with OO, you might as well be a Pilot without knowing about stalls and
emergency landings. You'll (probably) get from A to B, but you'll never push
the envelope and you better hope it stays fine...

Learn Java or OO COBOL now! (In fact, learn 'em both...<G>) Yes folks, you
heard it here first (around a year ago now...<G>)

Before we get too carried away with SOAP, realise that this is just an
interim measure to prepare you for bringing your components under the .NET
umbrella and promoting them to WEB SERVICES...

If you develop in .NETCobol there is no need for SOAP. Your components are
already integrated into an environment where the source code (and Language)
is irrelevant. The only importance is the use of the OO paradigm and the
exposure of  Properties, Methods, and Events. Web Services will run on any
platform, remote or local, that has .NET on it. (" 'tis a consummation
devoutly to be wished...")

So, just like the COBOL->Java tools I mentioned in passing above, SOAP too
will be rendered redundant by .NET.

Before anyone parts with hard-earned readies to go on the SOAP course, ask
yourself whether you are going to become an OO Programmer... If you're not,
then my advice would be "Save your money".

But then, my advice would be "Learn OO!". See above...

Pete.

(Peter Dashwood is NOT actively engaged in promoting the learning of OO for
any personal gain or remuneration from any training organizations. He is
giving you this advice for the sake of your career and the good of the
industry. NO Java programmers were injured in the preparation of this
statement. One so-called "COBOL Programmer" may have been...hopefully...I
wish...)

Now ignoring the various flavours of  getting into the Web, Activex, XML, SOAP,
Components etc., a question for those who are using the Web.

Firstly - BACKGROUND :

Will Price produced a second book, non-OO,  "Elements of COBOL Web Programming",
which was kindly dished out with V3.1 of Net Express. His book is about the
Forms module in the Net Express IDE to design and 'wizard' generate Web COBOL
source and forms/pages.

Fortunately he kicks off with the buzzwords, Client, Server, Protocols, URLs,
HTMLs, CGIs etc. (No mention of SOAP - don't think it was around when he wrote
his book). Although I have no immediate priority to get into the Web, with it
sat on my bookshelf, thought I should give it a read. Absorbed the first part in
great detail, but as I read on, thought the Forms module did not present a big
challenge, once you got into it (Interestingly, unlike the N/E Dialog Editor,
the Forms module allows me to design arrays. Currently I've overcome the Dialog
Editor deficiency by dynamically creating an array using the Entryfield GUI
class).

Second - THE ACTUAL QUESTION

This is where I need clarification. The impression I formed was (a) Your
generate a request form, which you display at the user; (b) the user fills in
the answers and presses <SUBMIT>. At this stage, the COBOL 'control' program
does a complete validation on what is sent back. For example if requested to
fill in say :-

    Name and address info
    phone and e-mail
    Your age group
    Your salary range

If you fail to complete the Name address info or the remainder, then it is
redisplayed at you with error messages.

Other examples could be where you go on-line to a pizza joint and place your
order, either selecting from dropdown lists, checkboxes or radio buttons. Same
sort of thing with Amazon.com where you place book orders in a shopping basket.

The above are fairly straightforward and a common usage of the Web.

Now to a specific problem. Initially DOS but also the same with OO :-

- for quick data entry, user wants to enter numerics without decimal points,
particularly as over 80% are decimal only values. He enters, I store and
redisplay -
Imperial - pic z.999 blank when zero, or Metric - pic zzz.99 blank when zero..

- some of the coding is Mnemonics - he enters "T" and I redisplay 'Top", or "S"
and redisplay "South"

- agreed it depends upon the application but I do have dialogs where the 'core'
represents say, 60% of the display. Other controls are hidden and disabled
unless the user makes a specific selection.

Now Pete may contend that Components handle this very nicely, (built-in self
validation ?). I'd be curious to know.

Sum my query up by -

(1) Is it viable to use the Web for data entry intensive applications,
particularly those geared to numerics as above ?

(2) Net Express users - have you opted for either COBOL Java classes or direct
invokes of Java in preference to the Forms module ?

Jimmy, Calgary AB
