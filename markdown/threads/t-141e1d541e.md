```

I'm trying to find the BEST way to catch file locking
errors in a process, and really could use some advice 
from you "old hands" out there. 

Once you get beyond INVALID KEY and AT END, is the method
of using a DECLARATIVE section on each file both efficient
and common? It is a rather ugly exception handler to me,
but I honestly don't know of any better way. 

The application is a common case, where many people may 
be viewing the same record, and it is read with ALLOWING ALL,
except in the case where someone wants to edit it, at which 
time I manually reread and lock the record. 
If a second person attempts to edit the record during the 
time the record is locked, they simply get an error message.

I'm worried about this method working cross platform and
in particular, cross-compiler. AcuCobol eats it fine,
VisualAge seems to have some grief with it though... (?)

Thanks
-Paul
```

There is some differences between compilers in syntax and
effect.  I don't know 'ALLOWING ALL', the appropriate MF
is LOCK MODE MANUAL or LOCK MODE AUTOMATIC.  Some compilers
won't provide a locked record (or didn't many years ago
when I looked), MF at least gives a record locked by another
process.

Use WITH NO LOCK when viewing only, WITH LOCK to edit, check
File Status, and loop until it is clear while allowing 
operator to abandon.  When looping put in an OS delay (use
INT2F, AX=0x1680 on DOS/Windows).

> 
> I'm worried about this method working cross platform and
> in particular, cross-compiler. AcuCobol eats it fine,
> VisualAge seems to have some grief with it though... (?)

Because record locking is not Cobol yet (X/Open is a usable
standard in the meantime) there will be differences.
