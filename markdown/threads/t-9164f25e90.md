```
This is a new thread - but a follow-up on the scope terminator discussion.

As someone who is NOT currently supporting production COBOL code, I just
thought that I would throw out a "personal" opinion on two coding techniques
(that are MEDIUM controversial).  I have worked in shops where these were
strongly recommended and others where they are almost never used.

1) *ALWAYS* check for the success of file I/O operations.  In '85 Standard
compilers there are 3 ways of doing this - each has PROs and CONs.
  - Use <not> AT END/INVALID KEY  (not available on OPEN/CLOSE - and may
require further "investigation")
 - Status Key checking after each I/O (remember that only the first digit
being "0" means successful,  02, 05, and the new 0x codes may require
additional checking).  This may be a problem as the Standard and
implementors add new codes
 - "Traditional" I/O Declaratives (Let's you check file status code and
"group" types of I/O - but is certainly not structured)

  With the 2002 Standard, new "common exception handling" declaratives are
also available - and provide for significant new "Standard" error checking.

2) *ALWAYS* use a <not> ON SIZE phrases with arithmetic statements UNLESS
they are nested within IF statements that have already checked for "valid"
possible results (which is NOT a particularly efficient way to do such
things).

    ***

Coming from (long ago and far away) an IBM mainframe environment where it
was relatively common to let a program "ABEND" if there were (QSAM) file
problems or unexpected arithmetic "results" - I know that there is much code
out there without such "condition handling" - but especially as the world is
going more and more to online systems, long running batch programs (with
tight production windows) and 24x7 systems,  I just don't (personally) think
that new (or significantly modified) code should be relying on abnormal
program termination as the best thing to do when the "unexpected" happens.
```
