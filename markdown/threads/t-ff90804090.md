Probably the easiest thing to do would be to mate the "argument" with its
corresponding "answer" in an array and use the SEARCH verb to obtain this
answer.

If the table size remains less than 100 elements, then a sequential SEARCH
would probably be more efficient than a SEARCH ALL.

Regardless, the ARRAY _M U S T_ be kept in "ACCT" order.

03  WS-ACCT-CODE-TBL-REC.
      05  FILLER PIC  X(011) VALUE 'A1010010140'.
      05  FILLER PIC  X(011) VALUE 'A1011030640'.
      05  FILLER PIC  X(011) VALUE 'B2532560650'.
03  WS-ACCT-CODE-TBL REDEFINES WS-ACCT-CODE-TBL-REC
                                                OCCURS 3 TIMES
                                                ASCENDING KEY IS
WS-ACCT-ELEMENT
                                                INDEXED BY X-WS-ACT,

X-WS-ACT-MAX.
      05  WS-ACCT-ELEMENT  PIC  X(06).
      05  WS-CODE-ELEMENT PIC  9(05).
03  WS-SUB                              PIC  9(08) BINARY.
*
DIVIDE LENGTH OF WS-ACCT-CODE-TBL-REC BY
              LENGTH OF WS-ACCT-CODE-TBL (1)
              GIVING WS-SUB.
SET X-WS-ACT-MAX TO WS-SUB.
*
At this point, if index X-WS-ACT-MAX (for example) exceeds 99, then perhaps
a SEARCH ALL will be more efficient than a straight SEARCH.

If you code for both, then as the ARRAY becomes larger, your indexed
threshold logic will dictate to the program which lookup method (SEARCH or
SEARCH ALL) would be best.

HTH....

Bill

This is simpler than an array:

 EVALUATE OH-ACCOUNT
    WHEN 'A10100' MOVE 10140 TO CODE
    WHEN 'A10110' MOVE 30640 TO CODE
    WHEN 'B25325' MOVE 60650 TO CODE
    ...
    WHEN OTHER  MOVE '?????' TO CODE.

This requires one line of code per account/code pair, same as an array. It
eliminates the need for messy loop control or SEARCH statement to find the
entry, nor do you need indexes or subscripts. 

I believe in the MISS approach -- Make It Simple, Stupid (even if it isn't).
