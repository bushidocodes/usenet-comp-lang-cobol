Mostly you use them to handle, under program control, situations which the
operating system normally comandeers and a COBOL program would not be coded
to handle on a file-by-file basis.

Example: hard parity errors, physical boundary violations (in some OSs, the
max size is determined when the file is created and it cannot be expanded
without re-creating the entire file), insufficient disk space, or other
situations where you do not care to code some handler for a non-success FILE
STATUS.

For example, if you work with sequential files, you probably interrogate
file status for 'at end' (status = '10'); with indexed files you might
interrogate 'invalid key' (status='23') . But what if the status is some
weird hardware error?   Depending on your program, it may be possible to
recover from that error with operator intervention. The DECLARATIVES can
give your program a 'hook' to prompt an operator after all the 'reasonable'
file errors have been handled in your program.

Example: insufficient disk space. If you are writing a report file and you
get this error, you could use the DECLARATIVES to:

1. Prompt the operator to remove some files from the disk to make space, (or
even for a "new disk ID")
2. Returning to your program, resume work with the increased space or on the
new pack.

You can use this technique, coded ONCE in the DECLARATIVES, and it will work
for each file in your program specified in the DECLARATIVES paragraph.

I'm of mixed opinion.

When I started with COBOL we coded dummy declaratives - because you
HAD to code either "Invalid Key" clauses or declaratives.  The '85
standard still requires this, although most compilers do not, and the
next standard does not - detecting file errors is entirely up to you.


I do pretty well with file status values.  However, the code at work
uses declaratives extensively - and it works very well as a catch all
for file errors that might otherwise go undetected.  They seem
tedious, but I think they can be a good thing.

On the other side of the coin - when I write a brand new program - I
don't use them.

007800*

008300*
008400*SOURCE-COMPUTER.  IBM-370 WITH DEBUGGING MODE.

   .
   . 
   .

044800 PROCEDURE DIVISION.
044900D
045000D
045100DDECLARATIVES.
045200D

045400D    USE FOR DEBUGGING ALL PROCEDURES.
045500D
