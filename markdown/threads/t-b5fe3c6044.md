Sorry Stephen,

I strongly disagree on this and consider it is very bad advice.

INSPECT will meet his requirements admirably. REXX would do so even better.

What one should NEVER do when dealing with text is assume it will be in a
fixed location.

Having worked on COBOL compilers and knowing a thing or two about scanning
and parsing text, it is my firm opinion that relying on a fixed format input
is the path to ruin.(I accept that this is opinion, not fact, that it is
arguable, but it is informed opinion and I am about to give my arguments for
believing it.)

Having tried it on numerous occasions, I am persuaded that it is really not
much more difficult to write a parser that will find text in ANY location,
than it is to write one that finds it in a fixed location. The difference
with a general solution to this problem is that you do it once and the
problem is solved forever. If you tie it to fixed locations you are buying
maintenance in the future. What happens if a new release of whatever it is
that you are scanning the output from comes out and they have changed the
format?

I would not advise anyone considering any kind of parser or scanner to tie
it to fixed locations, other than starting at 1....

Pete.

COBOL string manipulation gets a bad rap.  Very undeserved.  Consider
this problem and the COBOL solution:

One has a string of data with elements of varying length.  The
elements are delimited by a /.  You want to put each element in a
separate table entry, losing the / and appending a low-value
delimiter.  When you get all done, you want to know how many entries
you have.

The data also has places where a / is required in the output - when
this is true it uses // in the data - that must be accounted for - and
result in // in the output table element.

 Oh - and there's a twist - you don't want to start at the front of
the string - you want to omit the first 20 characters.  (Real problem,
I recently coded for).

(Note - input string always has a trailing / - I check for it in other
code and insert it if it's not there).

  inspect USR-PR-value replacing all "//" by high-values
  move 21 to string-pointer
  move zeros to cntr

  perform with test after until s-field (cntr) not > spaces
     add 1 to cntr
     move zeros to chars
     unstring USR-PR-value
              delimited by "/"
              into s-field (cntr)
              count in chars
              pointer string-pointer
     end-unstring
     if s-field (cntr) > spaces
        add 1 to chars
        move low-values to s-field (cntr) (chars:1)
        inspect s-field (cntr) converting high-values to "/"
     end-if
  end-perform
  subtract 1 from cntr

Cntr has the number of table elements
