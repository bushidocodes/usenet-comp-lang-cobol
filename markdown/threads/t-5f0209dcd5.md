Sounds like your application may be using SIGN IS TRAILING SEPARATE
CHARACTER if the data you described is in a file.

If you are using a debugger to view the data as the program executes,
your debugger may be showing the sign as a separate character even
though the sign is overpunched on the last digit. That's the way
MicroFocus' Animator works.

If it is DISPLAY format without a SIGN SEPARATE clause, it should take 3 bytes.
(That is what you should get if you don't have a USAGE IS clause nor SIGN
SEPARATE on it or one of its parent fields.)

If [USAGE IS] DISPLAY with SIGN SEPARATE (with or without TRAILING), it should
take 4 bytes.

If it is packed decimal (IBM's COMP-3), it should take 2 bytes.

If it is a binary field (IBM's COMP), it should take 2 bytes.

If it is a floating point number, it should take 4 bytes (COMP-1) or 8 bytes
(COMP-2).

If you are using MicroFocus Workbench to watch that variable, or if you are
using a DISPLAY statement, the number is being "cooked" to make it displayable,
so you can see the value of the variable but not really the internal
representation of that variable.

How the sign is represented is up to the compiler implementor. For data
interchange with unlike systems the SIGN SEPARATE clause should always be
used. The compiler documentation should indicate how the various data types
are stored and how the sign is handled if it is not separate. If you are
looking at the output of a debugger rather than the raw data, it may be
converting it to a readable format.
