Good evening, everyone.

I've a small problem. I need to develop a foreign (non-US)
mail sortation program, but have almost no programming experience.
Since this is unit-record work, COBOL is probably the best language
in which to write the program. I have started reading an entry-level
book on the language and have the MicroFocus Visual Object COBOL for
Windows v 1.0 compiler at my disposal.

The largest problem I have is that I have almost no control
over the format of the lists that I process. If they are in a
consistent format, i.e. field data is consistent across records, I have
tools available that can handle the sortation, but when they are not, I
must manipulate the data by hand until it is in a suitable format.
This can take hours and is something that the computer should be able
to do for me, given the proper instructions. Also, many customers wish
for their data to not be altered in any way, a wish I am not able to
comply with at all times and still achieve a legal sort. What I need
to be able to do is parse each record backwards from the end (the lists
are all line-sequential, each record terminated with CRLF, and are
almost all quote-delimited and comma separated--those that are not can
easily be made so. A reverse search should be more efficient since the
country name is almost always near the end of the record), looking for
a country name, appending to the record in a fixed location a value
corresponding to the found country name, then, based on that value,
parse the record again looking for either a city name or postal code
(postal code needed only for Canada), then append to the record a value
corresponding to the found city name/postal code. I know that the
lookup tables will be large, but I plan to keep the city table
manageable by only including the largest cities, since I'm not likely
to have a large number of items for the smaller cities (USPS regs
require that 6 pieces for any one city be strapped together; without
that restriction, I wouldn't need a city sort at all). Anyway, once
the extra data is appended, I can write the file to disk and pull it in
to another program that can sort and print for me, at least until I get
around to coding the additional modules for this program.

My question is (thanks for sticking with this so far!), is what
I've proposed here realistic? If it is, does anyone here have some
good starting tips for a newbie? I'm not even sure if the lookup tables
should be in the code or called from disk (as ASCII files?). All I've
played with so far are some of the snippets of sample code from the
book, and this is a great leap forward from there.

Many thanks,

Troy Smith
```

All right, newbie, you caught me in a good mood. Yes, it's doable. Some
considerations:
- you'll want to become familiar with the UNSTRING command.
- the sort won't be pleasant. Cobol sorts take longer, usually, than
system sorts (JCL, etc.). Since you can't alter the data, create one large
field to hold the entirety of the record and have separate sort fields that
are filled from the UNSTRING. One way to do this is by using input and
output-procedures rather than using and giving clauses in your sort
statement. Here's sample code:

FD Input-file.
01 Input-record PIC X(1272).

SD Sort-file.
01 Sort-record.
03 Sort-field1 PIC X(10).
03 Sort-field2 PIC X(28).
03 Held-data PIC X(1272).

FD Output-file.
01 Output-record PIC X(1272).
...

PROCEDURE DIVISION.
SORTING SECTION.
SORT Sort-file ASCENDING KEY Sort-field2 Sort-field1 (whatever order
you need)
INPUT-PROCEDURE Input-data
OUTPUT-PROCEDURE Output-info.
STOP RUN (or EXIT-PROGRAM).
INPUT-DATA SECTION.
OPENS.
OPEN INPUT Input-file.
READS.
READ Input-file AT END GO TO END-INPUT. (I know, don't use GO TOs,
blah, blah)
UNSTRING Input-record DELIMITED BY ","
INTO Sort-field1 Sort-field2, etc... (you'll need to make sure you
have no embedded commas in you fields for this to work, and you'll have to
strip the quotes, possibly with INSPECT).
MOVE Input-record TO Held-data.
RELEASE Sort-record.
GO TO READS.
END-INPUT. EXIT.
OUPUT-INFO SECTION.
OPEN-OUT.
OPEN OUTPUT Output-file.
RETURNS.
RETURN Sort-file AT END GO TO END-OUTPUT.
MOVE Held-data TO Output-record.
WRITE Output-record.
GO TO RETURNS.
END-OUTPUT. EXIT.
```

What you have proposed is EXACTLY what COBOL was designed to handle. I
would make the city tables indexed files, rather than actual tables, but
other than that, what you want to do is very possible.

Here is how I would proceed. This is a very loose outline.

I would define a universal data structure for use by the end sort/label
print product. This format would be what all input coming in must look
like. Field sizes and types must be sufficient to handle all incoming
conditions.

Then you need a system to format the variable input into this universal
data structure. You indicated most data is quote/comma delimited. The
ORDER of the fields in the data file should be provided for you, or should
be pretty easy to determine. I would create a program that would allow the
input of the field order as it relates to the universal data structure
defined above. The program would then read the input file, parse the
fields, and then move the fields to their corresponding fields in the
universal data structure.

Here is the method I use to parse the quote/comma delimited files. You
must be careful NOT to parse on the , fields alone. Here is why:

"First","Last","Address","City State" would parse fine, but
"First","Last","Address","City, State" would fail miserabley. The embedded
comma will cause problems. Here is what I do:

I scan the record for ",", and change the , in between the "" marks to a
high value character. I then unstring the record into individual fields.
I then replace all " with spaces, and then perform a left justify routine
on each data field. These fields are then moved to their corresponding
fields.

You have bitten off a lot to do for a novice Cobol programmer. The MF
Visual Object Cobol can do it, but is may not be the best tool for this
project. A lot of the learning curve you will be fighting will be in the
user interface design of the MF Visual Object COBOL, and not in your
code/cobol understanding.

Hope I have helped some!
