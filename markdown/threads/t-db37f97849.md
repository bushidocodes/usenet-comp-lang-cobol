For IBM, the "create file on OPEN" processing is really interesting <G>

The CBLQDA(ON) behavior is (generally) hated by IBM z/OS users. Just like 
TRUNC(STD) this is "required for Standard conformance, but almost NEVER actually 
used" feature of Enterprise COBOL. It creates an "unfindable" file and often 
(usually) hides a JCL error.

On the other hand, VM has *always* created files when they weren't "available" 
on OUTPUT and because of VM file naming, this works "just hunky-dory".  This is, 
the "common" behavior on PC and Unix/Linux systems.

My suggestion to IBM (sometime around 1987 or so) was that they have a compiler 
option that allowed you to specify the "default" high-level of the dataset 
name - and that the output would be catalogued when created.

  ***

Now we come to VSAM.  IBM *requires* that an initial IDCAMS be run to 
"pre-allocate" a file to make it pre-available (i.e so it will be created on an 
OPEN OUTPUT of a "not available" file).  This, of course, meant they needed to 
have something "else" for files that ARE available.  Therefore, they defined 
this as not-only allocated (IDCAMS) but also that it has included at least one 
record. See:
   http://publibfp.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igy3pg40/1.10.6

I don't LIKE the way most of this works, but it is pretty well "documented" 
*and* (I think) conforms to the ANSI/ISO Standards

application and/or the compiler/assembler used to create that application.

e.g, under Windows (CreateFile() function) there is no such thing as "open

used to control the behavior of the open based on the "presence" or
"absence" of the named file.

From the reference material for CreateFile()

Files are opened for GENERIC_READ or GENERIC_WRITE; these two flags may be 
combined to result in I-O access.

Behavior on open:

      CREATE_NEW Creates a new file. The function fails if the specified
file already exists.
      CREATE_ALWAYS Creates a new file. If the file exists, the function
overwrites the file, clears the existing attributes, combines the specified
file attributes and flags with FILE_ATTRIBUTE_ARCHIVE, but does not set the
SD specified by the SECURITY_ATTRIBUTES structure.
      OPEN_EXISTING Opens the file. The function fails if the file does not
exist.
      For a discussion of why you should use OPEN_EXISTING for devices, see
Remarks.

      OPEN_ALWAYS Opens the file, if it exists. If the file does not exist,
the function creates the file as if dwCreationDisposition were CREATE_NEW.
      TRUNCATE_EXISTING Opens the file and truncates it so that its size is
zero bytes. The calling process must open the file with the GENERIC_WRITE
access right. The function fails if the file does not exist.

I have heard many programmers echo your sentiments, Howard, but there is 
more to it than meets the eye.

Separation of software into layers is NOT just "needless overhead", even 
though it can seem that way on the surface.

Neither is it about how difficult it is to interface to a given subsystem.

The standard COBOL access module used by your shop "decouples" the data 
access from the program requiring the data.

The "utility" makes VSAM calls but it could just as easily make SQL calls or 
IDMS calls (in your shop) and, it can do so WITHOUT any impact on the 
programs which use it. (They request an action and get/put a data buffer) 
They can have a system independent interface to a data service provider. 
This would NOT be the case if SQL or VSAM or any other data provider was 
embedded directly in the applications. (putting ESQL into applications is 
simply a tragic mistake.)

Time and again, in different places over years, I have been thankful that 
someone provided a separation layer into COBOL code. Performance can be 
tuned and experimented with, WITHOUT severe impact on applications; upgrades 
to the subsystem concerned don't impact the applications or require every 
application program to be recompiled and regression tested.

Separation is a VERY Good Thing. In the object world, separation provides a 
convenient way of collecting "like-minded" objects into layers where they 
can be more easily manipulated. Components are already encapsulated, but a 
tiered achitecture means that groups of components can be isolated (for 
example, a set of data access objects that manage a given relational 
database). This has huge benefits when things go wrong. It is easier and 
faster to identify the offending layer and find the component that is 
failing. It can be repaired/replaced WITHOUT needing to recompile all the 
applications or regression test them, and without any chance of "knock on" 
errors elsewhere in code. The amount of code that needs to be looked at 
comes down to hundreds of lines, rather than thousands.

Your VSAM system gets replaced with SQL based data access; no problem. 
Applications continue getting what they always got, without needing 
change.(Only the "utility" needs to be amended...)

I have just completed building tools that can identify every ISAM access 
request in a COBOL program and change it to invoke an independent data 
access object. This would be a huge manual task (it is non-trivial, even 
using such a tool) but it would NOT be necessary if the legacy system had 
been layered, with separation between applications and the data source.

Tiered system architecture has not traditionally been the "COBOL way". That 
is a contributing factor to the decline of  COBOL. Integrated monolithic 
programs (even well written and well structured ones...) that "do 
everything" (presentation, business logic, data access) are a time bomb 
waiting to explode. (it can be justified for batch processing, but even 
then, it is a sticky wicket...)

I have been preaching the "Objects and Layers" mantra to various shops which 
are migrating to component based systems, for some time now

For COBOL, "objects" can be replaced with "closed or encapsulated modules 
that have a single entry and exit point and simple, consistent interface to 
anything that uses them." but that's a bit unwieldy so lets just loosely say 
"modules"...

Modules and layers. (say it once; it will encourage the men...)

code '97' oversight, as this would involve extensive testing and
signing off of every single program that called it.    In order to
move to multiple use of encapsulated routines, different testing
methods and standards are required. 

So we lived with the error.

The way around such standards tends to be switching to a different
language, such as Java - and getting someone else to create the new
standards.   That has been the case with many technology  changes, in
IS and elsewhere.   Heck, it isn't even limited to technological
changes!


And another thing was with the large copy members that were not
designed with any thought of changing the output from the original
design.    The component is there - but only to do what was easier to
do within the program.   We had another program that printed reports,
after we figured out the copy members and designed the reports to fit
its copy members.   It was not designed so that we could sometimes
write to XML instead of to the printer.    

We didn't gain any of the advantages of encapsulation.   I suppose
they heard that encapsulation was a good idea, but didn't think things
through and make it flexible enough.  

What would be good would be a called program that created XML, and
another one to turn that XML into standard reports.   But this program
was older than XML.

A tool such as encapsulation can be very good indeed.    Which doesn't
mean it can be misused or abused.
