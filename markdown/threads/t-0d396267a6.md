Sure is. I'm not doing this in COBOL but that's irrelevant. I have a
set of spreadsheets that look more like forms with some hidden VB code
to validate and extract the data. You click on a send button and it
emails the data off to a notes server where a program processes the
input and mails you back a spreadsheet full of pie charts and figures.


It was done this way as the prospective end users are accountants and
prefer a spreadsheet to a program (little do they know it's pretty
much a VB app in itself) 

The spreadsheet generated is created by a delphi program with COM
objects and it gets data from an SQL database.

There is no reason either of those two programs couldn't be COBOL of
course. The client side app could be a COBOL app using OLE instead of
the VB. In my situation, VB is better as all the end-user sees is a
single .XLS file. Very thin (or is it thick ;-) ) client. We just mail
them a new .XLS file if we need to change the input program or form.

>
>I can almost envision coverting a screen section into an X*Y
>spreadsheet by a program.  Instant GUI.

And stick a bit of VB in there and you can have a bit of
interactivity. You can get quite funky with excel and VB.
