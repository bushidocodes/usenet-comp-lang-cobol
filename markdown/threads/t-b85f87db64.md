```
This is a simple way of solving the problem, but whether or not it works
depends on the machine on which the program runs. If the machine is
Hitachi's mainframe (maybe compatible with IBM's mainframe), the above
program will work whithout any problem. But for some machines, on whitch
a 2-byte interger area (let's call the left byte A and the right byte B)
stands for the number A + 256*B, the program must be like this:

01 WS-WORK-FIELD.
05 WS-WORK-SIX-BITS PIC X.
05 FILLER PIC X.

And instead of

MOVE LOW-VALUES TO WS-WORK-FIELD.

I would use

MOVE 0 TO WS-WORK-NUMBER.
```

```
Assuming that you are using a standard COBOL 85 compiler, your
native collating sequence is ASCII (I assume so because of your use
of LOW-VALUES - it is not guaranteed to be binary zeros) and you have
no PROGRAM COLLATING SEQUENCE clause, just do

COMPUTE ws-work-number = FUNCTION ORD (ws-pass-field) -1

No need for REDEFINES, strange data types, and so on.
```

```
Tony B. Shepherd (t.··.@iii··i.net) wrote:
: And now for a trickier question: how can one tell if the program is
: executing on an Ascii or Ebcdic machine at execution time?

Under ASCII, "0" has value 48, while "A" has value 65. Under EBCDIC,
"0" has value 240, while "A" has value 193. So just compare:

02 CHARACTER-0 PIC X VALUE "0".

IF CHARACTER-0 < "A"
we are on an ASCII system
ELSE
we are on an EBCDIC system
.
```
