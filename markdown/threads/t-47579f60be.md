```
During the recent J4 meeting, I accepted an assignment to write a paper to
"fix" some of the  wording on what exactly the PICTURE clause symbols really
do (mean?).  I plan on creating a test-case that various people can run with
the ('74 or '85) compiler of your choice, but my question is what do you
think SHOULD happen (today - not with a future Standard) with the following
type of source code:

01 GROUP1.
     05 ALPHA-EDIT   PIC X/X.
01 ELEM1                  PIC XX.
     ...
MOVE "ABC" TO GROUP1
MOVE ALPHA-EDIT TO ELEM1


   ****

 The question is what the 2nd move "should" do.  I can think of (at least) 3
possibilities:

1) The data "AB" stored in the first two bytes of ALPHA-EDIT will be moved to
ELEM1

2) The data "A" stored in the first byte of ALPHA-EDIT will be move to the
first byte of ELEM1 - and the character "/" (which is what the picture SAYS
is in the 2nd byte) will be moved to the 2nd byte of ELEM1

3) The program will ABEND or have other "unpredictable" results because the
data in ELEM1 does not "match" what the PICTURE clause says *should* be in
that 2nd byte.

  ***

OK, folks - what do you think should happen?  (Any comments on what various
compilers do - both when running in their "ANSI-conforming" modes and in any
"full optimization" mode - will be gratefully accepted.)
```

```
I pick #1.

After the first MOVE:

MOVE "ABC" TO GROUP1
ALPHA-EDIT   contains "ABC"
The edit character is ignored because the move was to the GROUP item.

If the code was changed to:

MOVE "ABC" TO ALPHA-EDIT

ALPHA-EDIT would contain "A/B"

After the Second MOVE:

MOVE ALPHA-EDIT TO ELEM1

ELEM1 contains "AB"
The length of the receiving field causes truncation.
The Edited item is treated as PIC XXX when it is used as the sending field
of the move.
```

```
I've used code similar to this for the past 6 years and it works
consistently on all three versions of COBOL we have.  We are currently using
NetExpress 3.0 and it still works the same, so I'd like to see it stay the
same.

When you move "ABC" to GROUP1, just as Dennis Harley pointed out, the value
in GROUP1 and ALPHA-EDIT are both "ABC".  This stands to reason since GROUP1
is in essence a PIC X(3) with no edit mask.  When you move ALPHA-EDIT to
ELEM1, there still isn't any mask so the first 2 bytes get moved and the
result in ELEM1 is "AB".

If, however, you do move "ABC" into ALPHA-EDIT then the mask takes effect
and the value for both GROUP1 and ALPHA-EDIT is "A/B".  When this is moved
to ELEM1 the first 2 bytes are again moved so the value of ELEM1 is now
"A/".

If it was decided that by moving "ABC" to GROUP1 would result in a value in
GROUP1 of "A/B", then what would the following definition do?

01 GROUP1.
     05 ALPHA-EDIT   PIC X/X.
    05  ALPHA-EDIT2 REDEFINES ALPHA-EDIT
                                    PIC XX/.

bill
```

```
No one is thinking of changing the value of "ABC" in ALPHA-EDIT.  It has to
work that way (because a single elementary item under a group item - really
does work just like a "redefines").  I don't think there is ANY chance of
this ever changing or working differently in any other compilers.

The question, however, is whether the compiler may (is permitted) to believe
the PICTURE clause when you do the move "from" ALPHA-EDIT.  Let's look at a
similar (but slightly different) case,  if you have

  01 Group1.
       05  Num-Edit   Pic 9,9.    *> a numeric-edited item

  01 Num     Pic 999.

Move "123" to Group1
Move Num-Edit to Num

The value that will be placed in Num is either:
     13   - because you "de-edit" numeric-edited values when moved to a
numeric elementary item
            or
   "abend" - because num-edit really doesn't have contents that match what
the picture SAYS it is supposed to have.  (When I say "abend" this is similar
to a NetExpress RTS 163 error - when you have +F specified.  "ABEND" is a
typical mainframe term.)

OK,  when moving from a numeric-edited field, it is clear that the COBOL
run-time *must* look at the picture clause - not just the contents of the
field.  (what it does when the contents don't match the picture is up to the
implementor).  On the other hand, there are not the same (or at least not as
clear) rules about what happens when you move an alphanumeric-edited item to
an elementary alphanumeric item.  My guess is that (almost) all compiler do
move the real contents (i.e. the "AB" that you originally reported).
However, I think there are some "optimizing" compilers that believe that the
contents of alpha-edited will match what the picture says they will.
Therefore, the will *at compile-time* move a "/" to the 2nd byte of the
receiving field, and only move the 1st byte of alpha-edited to the 1st byte
of the receiving item (a run-time action that takes more "cycles" then the
move of the literal "/" which can be figured out at compile-time).

Does this help explain my question/issue?
```

```
I think the answer here is simple.  For numeric fields, it is not the
characters themselves that are important, it is the numeric value.
When moving from edited numeric fields, it therefore makes sense to
'de-edit' to get the actual numeric value, when it is being moved
to a numeric receiving field (only).  But for alphanumeric fields,
the characters themselves are the issue.  Just as when moving a
numeric edited field to an alphanumeric field, moving an alphanumeric
edited field to alphanumeric field should do no 'de-editing'.
```

```
COBOL (since the '85 Standard) has supported "local" variables - but they are
local to a nested program - not to a paragraph.  If you want data/procedure
"encapsulation" (of this type), the COBOL technique IS to use nested
programs.

If you are interested in adding data definitions in the middle of the
Procedure Division, you could suggest this to J4 (hopefully for the revision
AFTER this one).  My guess is that it would not receive a lot of support, but
it is up to you to suggest it if you think it is worth considering.  You can
communicate this suggestion (or any suggestion) directly to J4 by sending
email to the J4 chair at:
    doncobol <at> mediaone.net
```
