it was ever legal/supported.

What are you TRYING to do?
  Normally using "floating hyphens" (minus signs) are used in DISPLAY fields,
and "V" is used in internal fields.  Therefore, I would suggest that you
change the "V" to a "." (period or full-stop).  If you really do NOT want any
"display" decimal point in our output when there are digits to the left of
the decimal point, then I am not    quite certain what to do.

(I have not checked the Standard or the IBM documentation on this - but I
don't think that the "display-type" sign symbols can be used with the
"internal-type" decimal point)

Final question (if the previous doesn't help).
  Is the "B" actually in your PICTURE clause - or did you just put it in the
note to let us see where the PIC ended?  Again, I don't think you can use the
"B" (external format) symbol with the "V" symbol.

Not wishing to be accused of doing any of *my* work at the
moment, I decided to look at different ways to accomplish
what seems the intended goal: If the value is zero, show a
single "0" in the 18th position of a 19 position field.

Each of these options was checked in a program with ANS85
flagging active. The original field was unchanged, that is,
    PIC ----------------V--B.

Option 1: In the redefinition, insert a B between the V and
the 9, actually anywhere to the left of the 9.
    PIC ZZZZZZZZZZZZZZZZVB9B.
This could be confusing; but inserting the B is the smallest
change that could be made to the program to have the "0"
display in the correct position.

Option 2: In the redefintion, replace the picture clause with
something different.
    PIC BBBBBBBBBBBBBBBBB9B.
This shows that only one digit is to be presented; but it is
rather difficult to spot the 9.

Option 3: Use a conditional to set the value in the original
field.
    88 FORCE-2-ZERO VALUE "                 0 ".
Then use SET FORCE-2-ZERO TO TRUE in the procedure
division. No redefinition is required; but it is a most unusual
use of a conditional.

Option 4: Redefine the field as alpha-numeric.
    PIC XXXXXXXXXXXXXXXXXXX.
then, in the procedure division, MOVE "                 0 " to the
field. This, at least, clearly shows the intended result.

Reference modification could be used; but was not otherwise
considered.

HTH
------------------
Rick Smith

For readability, it is customary to use a repetition factor. e.g. -(16), 
instead of repeating the character, if more than two or three occur in a row. 
In my experience, most programmers will write PIC XX, PIC XXX, and PIC X(4), 
but never PIC XXXXXXXXXXXXXXXX -- always PIC X(16) instead.

As has been pointed out by at least one other poster, the combination of - and 
V, or Z and V, in a PICture string is improper, and the V should be changed to 
 in each of the above strings. Even so, it's unlikely that you will achieve 
the results you desire -- a value of negative 0.01 will print as <18spaces>-1, 
for example.

I think what you're really after is just one PICture: "PIC -(16).99".
Using that, here's how various values should print:
+1.00	prints as                1.00
-1.00	prints as               -1.00
zero    prints as                 .00
+0.01	prints as                 .01
-0.01   prints as                -.01

One exception I have seen (and done):

05  my-field      pic $,$$$,$$$,$$$.99.
05  my-field-r    pic xxxxxxxxxxxxxxxx  redefines my-field. 

With the visible alignment this can be readable.

It's nice to make pictures similar to make it easy to add up sizes when
appropriate:
05   my-field-1   pic x(01).
05   my-field-2   pic x(20).
05   my-field-3   pic 9(05).

It is easy to add these together.  Can't always do this, but it still is
