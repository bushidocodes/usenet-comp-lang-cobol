With text books there is difficulty separating the author's
preferences from that which is permitted by the language.
Stern & Stern, 8th Edition, 1997, has a similar proscription.
On page 195, they show a table which says the move is not
permissible. On page 194, they write,

"As a rule, do not move alphanumeric fields to numeric
fields. The move will be performed by the computer; but
the results could cause a program interrupt or termnation
of the job if the receiving field is used later on in an
arithmetic operation and it does not contain numeric data."

In other words, writing such a move into a program is
permitted by the language and will not cause a syntax error;
but because the programmer may make a mistake (in not
ensuring the data is numeric), they (the authors) would
prefer that the programmer not write such a statement.

There are similiar, stronger statements proscribing such
moves in Stern & Stern, 1st Edition, 1970. They clearly
don't like it and never have!

As nearly as I can tell, the move of an alphanumeric data
item to a numeric or numeric-edited data item has been
permissible since at least COBOL 68. One need only
consult the LRM for the compiler to see that it is permitted.

For such a move, the COBOL standard requires the
compiler treat the alphanumeric field AS IF it were defined
as an unsigned numeric integer. If the content of the
alphanumeric field is not an integer, the result of the
move is undefined.

Finally, whether to validate the field before using it,
whether to change its data description, or whether to
explicitly redefine the field are matters of programmer
preference. Personally, I see no problem with it, as
written.

Nearly all IBM compile time options can be specified in the source.  The rules are fast
and simple

Place one or mor CBL statements rigt at the beginning ofyour source, there may be no
preceedig empty line or comment.  Add as many compile time options as you need.  If one
line is not enough, add another CBL line.  Do not end the CBL line witha dot.

I prefere to put the CBL line into column 8  (guess why)

Therefore, the bginning of a COBOL program might look like this:


        CBL NOZWB, TRUNC(BIN), SSRANGE
        CBL FASTSRT, APOST, NOOPTIMIZE
        CBL NOSEQ, LIB
        IDENTIFICATION DIVISION.
         PROGRAM-ID. Mypgm.
etc etc
