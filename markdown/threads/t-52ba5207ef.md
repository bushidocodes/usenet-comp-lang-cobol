screen was added because of 'trade discounting' for 
example, where certain customers were 'trade' (such as 
panelbeaters) and got a fixed discount rate.  They
would have a record on disk for their address that would
be permanent and this would also indicate their rate.

You would then want to get this data before the estimate,

> end-routine are the subroutines of A and B (A10000, A20000, A21000, etc
> you get the idea)  is it acceptable to write this way or should the

I dislike the idea that routines are 'fixed' to their 'owners'
by having hierarchical prefixes or names.  All routines
are potentially available (or should be) to all other
routines.

I have seen a recommendation that all routines only ever
be called from one point and thus are immovably fixed on
the hierarchical tree by its prefix.  If the same functionality
is required in anothe part of the program then presumably
they just wrote a new identical block of code with a 
different prefix.

In general terms I would not fix the user interface to a
predefined sequence without good reason.  The customer
may give his name first, last, or in the middle of a 
transaction, or not at all.  The user should be able to 
do the input in any suitable sequence.  He should be
able to go back and change the data at any time.

Enter a few lines of estimate, take the customer name
which then indicates special pricing so the program 
changes the price, halfway through the address the
customer wants to add another item to the estimate so
back to the estimate then finish off the address.

The idea that the program should in parts 'A' and 'B"
and the order is start, A, B, end, will not give a
flexible user interface or happy users.

Separating code into multiple programs is usually for one of these reasons,
in no specific order.

  Program is so big it is unwieldy to compile or edit (many thousand lines).

  To get around memory limitations (not as common these days).

  More than one person will be maintaining code simultaneously.

  Code needs to be CALLed instead of PERFORMed, so must be packaged as a
  separate program.  (In the proposed COBOL standard you can write your own
  functions, and they will have the same 'separate program' requirement as
  called subroutines do now.)

  Code needs to be executed from different programs.

  Logically it makes sense to do so.

If you do not have one of the above reasons to split the program, it is
usually simpler to keep it one program.

I am sure that he has a fairly good idea that those are the 
reasons to or not to.  What he really wants to know is
whether these apply to him or not.  

Obviously if a program gets so big it won't compile or the
editor karks then it needs to be split, but how does one
know ahead of time that this may happen ?

In general with interactive programs I like to have one
program per screen and a clear interface (linkage) into
it so that it can be called from other programs as 
required.

For example another screen being added may be 'list of
existing quotes'.  This may be called from either the
'estimate' screen or the 'customer' screen and may give
different sequences or selections depending which it
was called from.

This will allow for facilities such as: 'make a new quote
that is a copy of <select from list>' or 'find the quote
for customer ,select from list>'.

Obviously for the 'find quote' you need to able to bring
up the customer screen _without_ going through the estimate
one so that, for example, search parameters can be input:
name is smith, or phone is 276something; so that the list
can be usefully shortened and manageable.

Then while scanning the list the user may say 'what is
that one' and want to show more details which would then
call another program.

It may be that the user would want to start with the list
(in say descnding order of date - latest first) and then
pick one to change - the phone number on one yesterday
was entered wrongly.

However, re-entrancy is usually too difficult to design for
and so the number of possible permutations may need to be
constrained - I usually pass a flag through the call
structure to prevent re-entering an active program when
this may seem to be a possibility.

I think it a huge mistake to simply "make a rule" and assume that
a standard approach is correct, or even preferable for all platforms
and for all shops ... even if the language is always Cobol.  The
dynamically loaded subroutine, for example, may indeed be the
best solution to canned logic --- when running on a virtual memory
machine in a professional shop with production runs.

For my installations, it would be an unmitigated disaster.  When
memory is a constraint, it is hugely dangerous.  All it would take
on a DOS machine, for example, would be for someone to install
some software that increased the environment, or lessened memory

the DOS the linker.  Since most large systems consist of some

code, at least as a general rule.

The bottom line is that memory constraints affect the choice of
methodology in a very real way, as do all the other "wrinkles"
imposed by any compiler/linker/OS.  "Best" has to start with
"it's gotta work".

The other point mentioned above is the run circumstances. Very
different methodologies are required for a system that is installed
once by professionals, then tuned, then maintained, as compared
to a system that is delivered on a CD, installed by an amateur, and
expected to work with no maintenance for several years, often
through staff changes and by users that have never seen a computer
before.

that indicates to me that it is a section of code I need not
concern myself with. I know what it does, since I wrote it,
and I also know that it is a routine that has almost nothing
to do with the local problem.  It will be system wide standardized
procedure that may have to change once per century (like
date routines).  It will also be well tested and proven, as those
procedural copy routines are usually done as post-mortems
at the end of a project for use on the next one.  It will also have
a standalone test routine somewhere, if I want to check it out.

I think the think to remember is that "what is best" has to be
taken in context, and changes.

of copy routines to subroutine calls.  If they had not originally
