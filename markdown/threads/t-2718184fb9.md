- **From:** "Darius Cooper" <Darius_cooper@Bigfoot.com>

```
Using MF-COBOL v4.1 (with patch 13.10 applied) on AIX, I cannot open a file
larger than 2GB for input the normal way, but it works if I use the
redirection "< " character in the input-file dd_FNAME.

When I try the " > " output redirection character in the output dd_NAME
however, the program abends just before the file size hits 2GB. So, I
wondered if this was a limitation in MF-COBOL.

I have read up on file-striping and would prefer not to go that route unless
I have no other option.

Any tips will be helpful.

- Darius Cooper
```

not sure about this.

I think the next version of MF Cobol does.

I'm afraid I don't know how TAPE output would work.

If this is truly a sequential file i.e.  'open input, read to end, close' or
'open output, write, close', why not write input and output C routines to
handle the I/O as a temporary solution.  You wouldn't have to change the
logic of your Cobol,  just replace the COBOL I/O commands with calls  to the
relevant  C routine to get or write the next record.

I believe that MF Cobol will handle large single files soon so, if and when
this happens, you can put back your standard Cobol.

I must admit that I don't like the idea of file striping and find it a
clumsy concept.   Unfortunately we might be forced to use it ourselves soon.
Oh well.

Rick

I found the answer to my question on a sequential file.
What one does is set up a dd statement that looks like this:

export dd_OUTFILE="> cat /> filename"

I had tried something similar previously but did not realize that I had to
escape the second ">" (stdout  from cat) with a "/" (the usual UNIX escape
character.

Thanks to everyone for their input.

- Darius
