Did you really think so?

I found it just appalling. I kept reading it because I simply couldn't 
believe that anyone would try and wrap such rubbish in a layer of 
scholasticism.

Just exactly WHO is going to buy this forthcoming book? (The 575,000 
gainfully employed American COBOL programmers?)

Eventually I started skimming and then gave up altogether before I reached 
the end of it.

Fortunately, I no longer care but a few years back I would have been fuming 
over this.

The claims are just patently obviously untrue.

Anyway, I don't intend to get into a pointless argument over it, but here 
are a few observations I would offer to anyone who thinks this reflects 
reality....

1. Function points are THE most widely discredited method of estimating work 
that has ever been devised. Time and again they are seen to be wrong and 
yet, (especially within IBM where they were treated like Holy Writ) people 
required to use this methodolgy fiddle actuality to make the Function Point 
analysis look true... I have NEVER seen a project where FPA correctly 
predicted the work involved, and I've had acknowledged masters of this 
technique do the assessment. To extrapolate the amount of work being done by 
COBOL programmers in the USA, based on Function Point Analysis, would be 
simply hilarious if it weren't so sad.

2. Totally unsupported figures that couldn't even appear on Wikipedia 
without requirement for citation

"As of 2009 there are close to 45,000,000 software application throughout 
the world. " (sic)

Are you sure it isn't 46,000,000?   Why not 55,000,000?  What's a few 
million applications more or less?

How can anyone take this seriously?

3. Even Gartner made an effort to revise their much vaunted and quoted 
figures on COBOL when they realised how far off the beam they were. But this 
paper quotes the "old" figures.

4. How can I give credibility to someone who thinks that XML and ActiveX are 
"programming languages"?

5. "Although software applications run on many computers and under many 
operating systems the most common for large businesses and government 
agencies are applications on mainframe computers running IBM's MVS operating 
system and written in the COBOL programming language."
That was pretty much axiomatic in 1970. Since then it has become 
progressively less true. Just exactly HOW much less true is difficult to 
estimate (and you have to ask whether there is any point in doing so 
anyway...). Certainly, there are government agencies and large businesses 
still running their core business on mainframes in COBOL, but how many of 
them plan on doing that for, say, the next 10 years? How many have ALREADY 
moved off the mainframe, or relegated it to be an expensive server or batch 
processor?

6. For people employed in the "COBOL industry" the only thing that matters 
is how easily they can get another job, and what the prospects are for their 
careers. If you live in the United States try a few trend searches in 
www.indeed.com Click the Trend button then enter various languages into the 
search bar. Separate them with commas... my favourite is:  COBOL, Java, 
Ruby, Python, C#

Guess which one is the ONLY one going down? I was surprised to note that, as 
of mid 2007, Python crossed the COBOL line and there is more demand for 
Python skills than there is for COBOL. In fact, Ruby and COBOL are about 
equal. I suspect that would be different in Europe, where Ruby is enjoying 
an increasing wave of popularity.

BOTTOM LINE:

The best we can do for COBOL is salvage as much of it as possible. That 
means accepting that, for procedural code, the party is largely over. 
Continuing development in procedural COBOL is like going to the river with a 
leaking bucket when everyone else has hot and cold water on demand.

The future belongs to objects and layers.

Pete.

OK, here's one:

1. COBOL was designed for a centralized, single-threaded architecture, with 
full control of all operations. (Yes, I know about privileged instructions, 
Sam, and I know about SVCs and ESTAEs... I'm old school, just like you - at 
least, there's part of me that is...:-) - for the sake of this argument, 
lets say that COBOL called on such facilities when it needed them...)

2. Client/server networks, and, especially, the Network of Networks 
(Internet and Web), are NOT such an architecture. There are many processing 
"cores" available; they are at different locations, the program no longer 
has central control (it is driven by "events" - you cannot decide when a 
user will click a mouse or drag and drop a control within the logic of your 
program... you can if you disable these facilities, but users get a bit 
pissed off with systems that don't work the way everything else does, or 
don't respond when they click a button...)

3.In a straight batch processing environment COBOL is completely at home; in 
a distributed environment , it isn't.

4. As the network developed, so new techniques evolved to deal with it. Some 
of these had already been under consideration, but were so ideally suited to 
networking that they just naturally flourished in that envirobment. The idea 
of "objects" with their own properites and behaviours was one such idea. If 
an object "knows" how to react to a certain event, it no longer needs a 
centralised program to tell it what to do. It can use the nearest available 
core, or it can be transported to a remote place where events are occurring 
that it is interested in (for example). In another variation of the 
scenario, events can occur and be processed by objects that are far removed 
from where the event was raised, or objects can activate behaviours that 
raise events far removed from the activating object. The point is that 
although objects mostly run and process on a local core, they don't HAVE to. 
It is a DIFFERENT paradigm, a DIFFERENT scenario,  a DIFFERENT Type of "data 
processing", not just because it is "event driven" but because the whole 
"behaviour" of the code is different from a procedural batch COBOL program.

5. By the time COBOL was extended to support Object Oriented Programming, it 
was already losing the race to newer languages (like Java) which had been 
designed from day one to support the concept of Classes and Objects. The 
result was: "Yes, you CAN write OO Classes in COBOL, but does it make sense 
to?" The answer is: "It depends..."

The above is not a reason for "getting rid of COBOL"; rather, it is a 
justification of why COBOL doesn't cut it any more. Now, you can reasonably 
argue that if all you are ever going to do is sequential batch processing 
you have no need of other tools, because COBOL can handle that very nicely 
thank you.

But the fact is that it isn't just IT that is changing. The world is 
changing, the marketplaces are changing and the methods of "doing business" 
are changing accordingly. While batch processing is still an important part 
of many businesses operations, there is serious doubt as to how long this 
will remain the case. Ultimately, the goal of a commercial IT system is to 
model the real state of the business in real time. In the real world 
transactions occur. For some time there simply wasn't the processing power 
available to fully process these transactions, so they were partially 
processed and clues and triggers were set up for overnight batch processing 
to complete the rest of the model. This is changing. Distributed systems can 
apply local processing cores to update remote databases (such as central 
corporate repositories), immediately in real time. (Transactions can run 
anywhere on the network and don't need to be collated into batches, sorted, 
and presented for sequential updating of master files or databases.) The 
dream of having the IT model reflect the real world at any given moment 
(almost to the millisecond) is becoming achievable. Better ways of "mining" 
and "warehousing" data are combining to obviate the necessity for batch 
processing to obtain historical data. Background processes can do this, not 
just in "overnight batch" but 24/7.

The bottom line is that it is "no contest" between COBOL and the "languages 
du jour", as far as network solutions are concerned. As networked solutions 
are becoming more and more popular, so COBOL is declining in popularity.

professional programmer will need both types of tool, and knowledge of the 
implications for each.

Yes, cost savings can be illusionary.

Yes, there WILL be a lot of work in any conversion.

Yes, sometimes, given the conditions you carefully noted, it is "just as 
quick" to write something new. (Really it comes down to being "just as 
quick" as doing the searching to find and understand the existing code; it 
can never be quicker to write something new if the existing code does what

But, even given the above, I find Doc's points more compelling:

1. In general, money CAN be saved by salvage and recycling.
2. (Quoting from Doc's earlier post on this:

"I have move from environment to environment and rarely copied a file

calculate when and where to wrap around the alphabet... these I have
copied and used again and again.  I may have a few years on me, true, but

(and the need to check for leap years), March 31 days... and April 32
might need a bit of looking at.  No need to rewrite, just copy."

That is the nub of encapsulating and re-using functionality. It isn't just 
about dates; there are THOUSANDS of functions in any business that could be 
similarly isolated and re-used.

(Next time you have a COBOL listing in front of you, have a look at the 
PERFORMED routines... Why were they PERFORMed? Because the programmer 
realised the same code would need to be executed from more than one place 
and, quite properly, put it in a form where it could be shared. But what 
about all the other programs that use the same functionality, or minor 
variations on it? They all have similar (or even the same ...especially if 
written by the same programmer) code copied into them. It was "too small" to 
warrant putting in a separate module, but it was too frequently used to code 
it inline every time it was needed.)

So we end up with the same functionality propagated and perpetuated 
throughout the system(s), using a different interface in every program, with 
different datanames.

This is partly because of the historic separation between programming and 
analysis. The programmer is responsible for getting debugged working code 
written to a schedule, by following a specification provided by an analyst. 
The analyst simply boilerplates the function into each program spec. so it 
is hardly any wonder it gets propagated at code level.

If the analysis methodology is different (as it would be on an Agile site, 
for example) the results will be different. Joint Application Development 
workshops mean that programmer, analyst, and client ALL look at the kind of 
functions which will be required by the proposed system. The programmer is 
no longer blinkered to a program spec. but understands the business 
requirement. Now it is possible to look at what are likely to be common 
functions OUTSIDE of the context of a specific program. Major Classes 
(Customer, Product, etc.) can be identified and populated with properties 
derived from various use cases which involve them. In the same way, 
functions common to a number of business processes can be identified.

Now, you COULD (and this seems to be what you are advocating, Howard) 
say:"Let's forget about the existing stuff, we can just write the functions 
and classes we have identified, from scratch, and get a completely new (in 
every sense of the word) system.

Then the programmer speaks up and says: "We already have a lot of this 
functionality encoded (s/he knows, because s/he wrote it...), shouldn't we 
try and salvage some of this?" It could save time, effort, and, 
consequentially, money.

Or, it might not, if the factors you identified are in place and it is very 
difficult to identify and retrieve the code.

Even in the scenario I described with what is fundamentally the same 
function being PERFORMed from all over a system, I still think it is worth 
salvaging it. You only need to use ONE copy of it as a template for a new 
class.

Doc made the point that even when moving within the SAME TECHNICAL 
ENVIRONMENT (same programming language, no change in analysis methodology) 
he has found it useful to copy existing code. (So have I, back in "the old 
days", and the OO approach is really just an extension of that experience... 
encapsulate it into an OO COM Class and you have it forever, even if the 
next place you want to use it is NOT a COBOL shop...)

You made a very good point that it is the analysis that takes the time, not 
the programming. But it is the ongoing maintenance that takes even more time 
than the analysis and programming put together.

Encapsulating and re-using functionality can DRAMATICALLY not only reduce 
the time it takes to do maintenance, but it can reduce the amount of 
maintenance required and the amount of regression testing required.

Pete.
