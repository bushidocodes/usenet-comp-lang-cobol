As a COBOL programmer switching to VB I'd say that it depends on the
application. One I just finished had 2 parts, a data entry part and an
extract/report part. The extract/report part was remarkably similar to
COBOL but the data entry part was (and, frustratingly still is) an entirely
different beast.

VB is event driven, but for an extract/report program the only true event
is starting it. (maybe selecting the criteria if it's variable criteria,
but the more variable it is the more the focus changes to the user choosing
the criteria vs. the program generating the report.)

I've never worked with Delphi but I assume it would be similar, i.e. if a
program lends itself to sequential processing then you code it sequentially
and it ends up similar to COBOL. If it takes event-driven processing then
you write it that way and it ends up quite a bit differently.
