DELIMITED BY SPACE means each space is a delimiter; DELIMITED BY ALL SPACE
means that consecutive spaces count as one space for delimiting purposes.

Yes, here it is; plug-and-play.  No loops, no reference modification, no
thrills...

       identification division.
       program-id.  unstr.
       data division.
       working-storage section.
       01  IN-PTR  PIC 999.
       01  PART-NO-SUB-COUNT PIC 999.
       01  WS-PART-1    PIC x(20) JUST.
       01  WS-PART-2    PIC x(20) JUST.
       01  WS-PART-3    PIC x(20) JUST.
       01  WS-PART-NO   PIC X(50)
                        VALUE "     AAAA     BBBBBB CC            ".
       procedure division.
       a.
           MOVE 1 to IN-PTR.
           MOVE 0 to PART-NO-SUB-COUNT.
           INSPECT WS-PART-NO TALLYING IN-PTR FOR LEADING SPACES.
           UNSTRING WS-PART-NO
               DELIMITED BY ALL SPACES
               INTO     WS-PART-1    WS-PART-2    WS-PART-3
               POINTER IN-PTR
               TALLYING PART-NO-SUB-COUNT
           END-UNSTRING
           DISPLAY "sub count=" PART-NO-SUB-COUNT.
           DISPLAY WS-PART-1.
           DISPLAY WS-PART-2.
           DISPLAY WS-PART-3.


Results:
sub count=003
                AAAA
              BBBBBB
                  CC

If the part number really consists of numbers, then you may simply change
the PIC x(20) JUST to PIC 9(18) and that will work too.

Don't forget to put in the error handling suggested in my original post.

I think this tells it.  Your first field has leading spaces, so the UNSTRING
is seeing an immediate delimiter.  Here is a suggestion:

01  IN-PTR  PIC 999.
01  PART-NO-SUB-COUNT PIC 999.
...
MOVE 1 to IN-PTR.
MOVE 0 to PART-NO-SUB-COUNT.
INSPECT WS-PART-NO TALLYING IN-PTR FOR LEADING SPACES.
UNSTRING WS-PART-NO
        DELIMITED BY ALL SPACES
        INTO     WS-PART-1    WS-PART-2    WS-PART 3
        POINTER IN-PTR
        TALLYING PART-NO-SUB-COUNT
    ON OVERFLOW
        <handle malformed part number>
        <caused by blank part number>
        <or too many subfields>
END-UNSTRING
IF PART-NO-SUB-COUNT NOT = 3
    <handle malformed part number>
    <too few subfields>
END-IF

I would suspect a non printing character in one of the "space" positions.
"All space" should do the trick, but will not if it is actually something
like "space tab space".  I often find it usefull to do a few examine
replacing statements before the unstring to handle anything non-standard.
For example:

    EXAMINE WS-PART-NO REPLACING ALL X'09' BY SPACE.
    UNSTRING WS-PART-NO DELIMITED BY ALL SPACE
        INTO ....

YM,
As you describe the data, it contains delimiters (spaces) before the first data,
between each data, and after the last data.  In this case, you'd need another
receiving field for the null item _before_ your first data.  I would expect
WS-PART-1 to be null (have whatever was in it before the UNSTRING), WS-PART-2 to
have the first data, and WS-PART-3 to have the second data.  The UNSTRING would be
"unfinished"; that is, if you had a POINTER, and had set it to 1 before executing
the UNSTRING, it would be at a value less than the total length of WS-PART-NO after
execution (actually, pointing at the start of the third data).
Colin
