I've performed all 4 types of development.

1.  Unix style (VMS/CICS) where the processing and files reside on the
same box, and the interface is "thin client" in it's most basic sense.

2.  Thick client - where files reside on the file server, but all
processing is on the local PC - Indexed based client.

3.  Client/Server - where the client does processing, but access the
data that is presented to the client from a process running on a
server.  (This is what Oracle and SQL server do).

4.  Client/Server thin client.  The database runs on the server, as
does the process accessing the data.

By far the first is the fastest - obviously fewer layers of
processing.

2 can be faster than 3 depending on the load on the database and the
speed of the process running at the workstation.  Even for large
numbers of users.  However, option 2 has the disadvantage of the fact
that a workstation crash will likely corrupt data.  Speed vs 3 depends
on how loaded the database is.  I have had systems that were much
faster as type 2 because of the load on the database serve and/or
organization of the database tables.

Type 4 has the potential to be the fastest and most stable.  However,
you have now put all your eggs in one basket.

Another thing to consider is expense, both is hardware and software.

In general, I agree with what Russell says below.  However, there are
some exceptions.  For example, some PC based COBOL compilers permit
you to use Btrieve as your indexed file handler.  Btrieve is a very
powerful transactional based file system.  I wrote a non-trivial (100k+
LOC) multi-user inventory system in COBOL that uses Btrieve.  It has
been online for about 12 years, and my client has never had to reload
the database because of a corrupted file.  Assuming the same file I/O
engine, compared to record level I/O requests, SQL is very inefficient.

    There have not been any flames (yet - Monday morning), but I feel that I
should qualify my previous message by noteing that I was referring to a pure
native indexed file, that is nothing runs on server but file and print
services, and no 3rd party database services, such as Btrieve, Fileshare
(Microfocus), Oracle, etc.

    I also assumed that each terminal is modestly busy, and that a fair
degree of contention exists for record locking.

    The runtime version also matters.  I have been using MF v3.1 for dos (16
bit).  V2.5 to 3.1 are known to be unnaturally slow when accessing shared
data files (even if open input).  Version 3.2 is faster, but we have had
stability problems with 3.2 (dos 16 bit).
