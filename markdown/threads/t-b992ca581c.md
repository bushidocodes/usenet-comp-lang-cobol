```
I am taking a class on Database Management and Design and have a question
that I am looking for some info on. My professor explained on the first
day of class that the advantages to database processing are as follows:

1.) Economy of scale
2.) Getting more information from the same amount of data
3.) Sharing of data
4.) Balancing conflicting requirements
5.) Enforcement of standards
6.) Controlled redundancy
7.) Consistency
8.) Integrity
9.) Security
10.) Flexibility and responsiveness
11.) Increased programmer productivity
12.) Improved program maintenance
13.) Data Independence

Per my professor, these reasons are why a 4GL Relational Database System
is better than the 3GL Flat File Systems like Cobol and Basic. I am
confused! We (at work) are currently using a Unix operating System
running Cobol programs. This sounds like an economy of scale and sharing
of data. We enforce logins and passwords which force the user directly to
an individual menu --- that sounds pretty secure to me. As to the
integrity, consistency, redundancy, and standards, that is dependent on
how well the file structure is set up and how well the programs maintain
it. I think we've managed to cover most of these pretty well, but I'm new
to all this. What I would really like, is to know what are the *real*
advantages and disadvantages of databases over file-systems. I would love
do my presentation (a big part of my grade) on this, so any input anyone
could provide would be helpfull.

Thanks in advance.
Elizabeth.
```

```
Hello:
Your professor makes good points. But COBOL can access SQL data
thru both dynamic and embedded SQL. So COBOL doesn't only need to
read sequential tapes and indexed files. It can access SQL.

Today I was studying ODBC and SQL. In a COBOL program, typically
to retrieve the data from DB/2 or SQL I would have to write a program
loop to do fetchs (like read statements) to get one record (opps, row)
from the file(opps, table).

Anyway, it takes alot of code and then it is in stone until it is
compiled again.

Microsoft query and other such things allow you to do access using
point and click. Well, 4GL allows you to develop stuff faster, but
COBOL is more flexible/robust.

Sure you can get something up fast using various 4GLs, but COBOL
and other 3GL's such as my new favorite, Visual C++, are more
powerful.

To me, 4GLs are for quick and dirty reports, while COBOL and it's
supporting transaction processors such as CICS are for
data maintenance/addition. As a generalization, 4GL are for
user programmers. Real programmers use 3GLs. Less smoke and mirrors.
3GLs let you bit-twiddle with impunity...

Let 4GLs be used for adhoc queries, and data schmozling, but keep the
data updating and transactions in source-controled environments, so
the programmers don't shoot themselves in the foot. (as quickly...)
```

```
I too am taking a similar class, while working in the real world with
both relational databases and flat files in COBOL. One thing that was
not mentioned in class is how bad design and implementation of a
database can give you the same disadvantages of a flat file system,
while good practices with flat files will give you some of the
mentioned advantages of a database.

There are some advantages to a relational database over flat files
(ignoring the 4GL vs 3GL aspects). One of which is that a relational
database is stored as one centrally defined 'thing' with the ability
to enforce referential constraints, whereas flat files are separate,
and have no capabilities inherent in the file structure to support
those same constraints.

There are other comparisons that address this, but in short I'll just
mention that classes like this tend to take an optimal view of how
databases are used.

and we can compare experiences.
```

```
Elizabeth, congratulations on trusting your own judgement.

First, 4GL does not equal Relational. Many "4GL" products rely
on flat file systems. And it is possible to have SQL (relational)
access to virtually any COBOL file system (see below). COBOL is
often the "back end" output language of CASE tools.

Second, unlike COBOL there exists no ANSI standard 4GL. When a
manager chooses a proprietary 4GL, an act of faith occurs. If a
COBOL vendor ceases to exist, there are other COBOL vendors from
which to choose, and moving an application is comparatively easy.

Focus your efforts on what it takes to solve real business problems.
Check our web page for the various white papers, including Relativity,
which allows SQL (i.e.relational) access to virtually any COBOL flat
file based system (what will your professor make of that?). Check
out The COBOL Foundation (http://www.cobol.org/) for more information
and links to other 3GL (specifically COBOL) tool vendors to find
COBOL based solutions for virtually every item on your professor's list.
```

```
At the risk of being attacked by all the other guys out there, your
prof is quite correct. Whilst Cobol has and always will be, there is a
time and a place when you should (must) consider making use of the
advances in technology going on around you. One of those advances, is the
database - for all the reasons which he expounded. However, one specific
species of RDBMS is the Nested Relational model. This animal associates
very, very closely with the Cobol structures - hierachies, repeating groups
without any flattening or normalisation. It allows the reconstruction of
indexed and sequential files as nested RDBMS tables and transparently re-
directs the Cobol application I/O without any source mods or usage of
dynamic SQLs. Once you're in this position, of course you can access your
data in any number of exciting ways, do anything you want with it, with
virtually any of the GUI desktop tools out there.

BTW, the company that provides this is called Unidata Inc.
```
