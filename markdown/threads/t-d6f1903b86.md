you don't really need cblwin.dll

cblwin.dll is a dirty trick that MicroFocus uses to prevent you from using
their DOS compilers under Windows (in a DOS box).

There are several ways around this. One of the most elegant is to use
the DOS debugger to call the compiler. The Debugger prevents you
(and MicroFocus's compiler) to detect that you are under Windoze.
Other than that the debugged programs run at full speed.

Here is a typical bat file to do the trick (one has to fight dirty tricks
by being even more dirty and devious):

@echo off
if (%1) == () goto help
if exist %1.cob goto compile
echo %1.cob not found
goto end

:compile
echo Compiling %1...
if exist %1.err del %1.err > nul
debug %mf%\bin\cob.exe %1.cob,%1.obj,nul,nul
use"%etkdir%\d.dir"<%etkdir%\q>nul
if errorlevel 1  goto compile_error
if not exist %1.obj goto compile_error

link %2 %1,,,coblib+cobapi,,; > nul
copy %1.map cobol.map > nul
del %1.map > nul
del %1.obj > nul
goto end

:help
echo  COBOL program
echo  Compiles and creates an .EXE module from
echo  a source program file.  If there are any
echo  errors a file is left with extension .ERR
goto end

:compile_error
debug %mf%\bin\cob.exe %1.cob,%1.obj,n,n
use"%etkdir%\d.dir"<%etkdir%\q>%1.err
debug %mf%\bin\cob.exe %1.cob,%1.obj,n,n use"%etkdir%\d.dir"<%etkdir%\q
if exist %1.int del %1.int > nul

:end

------------------------------

here is how it works:

the important line is this one:
debug %mf%\bin\cob.exe %1.cob,%1.obj,nul,nul
use"%etkdir%\d.dir"<%etkdir%\q>nul

I use the %mf% environment variable to point to where the compiler binary
is.
I also rename it to cob.exe (forget if this is needed).
then call 'debug' with the call of the compiler as argument.
My compiler options are stored in the file %etkdir%\d.dir

Now this is very important:
current input is redirected from a little file called %etkdir%\q

in this file are exactly SEVEN characters: g CR LF q CR LF EM
or in hex: 67 0D 0A 71 0D 0A 1A
this tells the debugger to (g)o, then (q)uit.


here is my d.dir file (you probably use other options)
01SHUFFLE
ALIGN"4"
NOALTER
NOBOUND
NOCHECKDIV
FASTLINK
FLAG-CHIP
KEYCOMPRESS"5"
IDXFORMAT"4"
REMOVE"PROMPT" "COMP"
OPT"1"
OPTSIZE
OVERRIDE"COMP-5" = "COMP"
NOQUAL
NOQUERY
NOTRICKLE
NOLIST
NOSMALLDD
NOTRUNC
RTNCODE-SIZE"4"
TARGET"386-16"

--------------------------------------------------

The solution advocated here is a good example of how the
'nasties' out there (e.g. BG and MF) can block each others
natstiness. BG does not want you to debug Windows (so
disables support for Windows - basically INT 2F) and MF
does not want you to use their DOS product under Windows
(so aborts if INT 2F is not available).

There is a facinating story behind the use of INT 2F by
MicroFocus as part of their installation and configuration
control, but this is not the place to tell that.

Leif,
      The executables in the product (COBOL.EXE, RUN.EXE etc) are dual-mode
DOS/OS2 executables ie they have both OS/2 code and DOS code embedded in
them.  So, on OS/2 the operating system would load and execute the OS/2
code, and on DOS (or Win95) the DOS code would be executed. DOS has a 640K
memory limit unless extended/expanded memory is used and that is what XM or
XM-built triggers such as XMCOBOL give access to.

I would only expect to see the 'cannot find cblwin.dll' message if you try
to run a Windows executable such as COBOLW.EXE or RUNW.EXE and you did not
have all of the Windows support for your COBOL system installed.
