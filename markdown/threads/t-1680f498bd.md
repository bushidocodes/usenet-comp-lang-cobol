Easy (but a tad confusing) answer to your question.

1) a *limited* number of LE callable services are available to be called by a
VS COBOL II program.  The ones that are allowed are ONLY the date/time ones.

2) In order to use those few callable services, you *MUST* call them
dynamically.  (This can be done either by CALL identifier format of the CALL
statement *or* by compiling with the DYNAM compiler option).  *HOWEVER*, in
order to make even a single dynamic CALL in a VS COBOL II program, you MUST
use the RES compiler option.  If even one of the programs in your application
is already (or will be) compiled NORES, then this will force your entire
application into "MIXRES" mode - which introduce a WHOLE huge set of "new"
rules when using the LE run-time library.

3) In case it wasn't clear,  your VS COBOL II compiled programs *must* be
link-edited with *and* be run with the LE run-time library and not the VS
COBOL II one (and again, this means that all your existing NORES VS COBOL II
programs will need to be re-linkedit).

   ****

All of this is documented in an IBM APAR.  I don't have the APAR number
handy, but your COBOL "support person" should be able to find it by checking
IBMLINK for "VS COBOL II" and "LE dynamic calls".  (Let me know if you/he/she
has any trouble finding the APAR and I will get the number for you).

I hope this helps - even if it may not be what you really wanted to hear.

Steve,
  I think you are INCORRECT.  If the compiler itself makes a call to an
LE-service (as it does with IBM COBOL for this-or-that), then you don't need
to use RES (of course it isn't even an option with that compiler).  HOWEVER,
if you have a dynamic call to ANY program explicitly coded in your program,
then RES is forced on at compile time.  The compiler does NOT pay any
attention to what the name is of the called program (and as it is dynamic,
theoretically, it could change from execution to execution).

Again, using an explicit dynamic CALL to an LE-service *will* force on RES in
a VS COBOL II program.
