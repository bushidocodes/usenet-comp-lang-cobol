    Is decision tables part of a Cobol standard (I don't think so) or

They are not part of Cobol, but it is relatively easy to generate Cobol
code from decision table using a preprocessor.

>    How many places actually use Decision Tables as opposed to not using
> them?

I used them in between 1970 and 1974 or so, but not since. I have a
couple of books here on how to generate code from DTs and did do a
simple preprocessor, but never got sophisticated enough to be
worthwhile.

>    Is there any advantage to using decision tables over using IF ...
> THEN .... ELSE or EVALUATE?

The EVALUATE statement with its ALSO and ANY can create decision tables
directly, though they are 'sideways' compared to the usual form.

Anecdotal:

In my past 27 professionbal years I only came once about a complex
situation where I was forced to implement a decision table. This
happened in the EACP (European Ada Compiler Project) somewhen
in the Eigthies.

What I did was creating a nightmare of nested logical expressions
which nobody including me could understand. So I got lost when
trying to get the identation right.

Then I had the brilliant idea to give that code to a List-Interpreter.
This program tried to make most sense of it and had an intrinsic
pretty-printing mode. Then I took the code and reshaped some
things from Lisp back to Ada whilst keeping the identation - and
that was the production code.

This piece of code never failed - but I think I was just lucky.
Today, I would try a more table-directed approach - or better:
Is a DT really the right design?

Herwig
