>What's the difference between passing by content and passing by value? 
>Aren't those the same thing?
>

Essentially COBOL passes arguments in a list in one of three ways:

1. By reference; the list contains the address of the actual data item
2. By content; the list contains the address of a copy of the data item
3. By value; the list contains a copy of the data item

From the caller's perspective,

* by content allows one to include literals in the argument list of the call

* by content allows the passed item to be "read only" in the sense that
       any changes made to the argument in the called subroutine
       are not reflected back to the caller

* by value is there to support COBOL/C interoperability better; not all
  data items can be passed by value; you can pass and receive
  arguments by value between two COBOL programs, though as
  one poster noted, by value doesn't really enhance COBOL to
  COBOL communication in any signficant way that I can see



From the subroutine's perspective,

* by content and by reference appear to be the same: an
  address of the data item; the subroutine cannot distinguish
  between these two

* an argument's being passed by value needs to be known in
  advance and the using phrase on the procedure division 
  header needs to include a "by value" modifier

Hope this helps.

> What garbage?


Temporarily allocated memory, such as intermediate work areas that
may be used for COMPUTE, STRING, UNSTRING, OCCURS ...
DEPENDING ON, CALLs to functions and procedures (e.g. pass
by content), etc. This is a serious issue in languages like BASIC and
FORTRAN that permit implicit variable storage allocation on the fly,
but much less of one in COBOL, since virtually all working fields are
not only explicitly defined, but most are mapped as well (i.e. Records).
