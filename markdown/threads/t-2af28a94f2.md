compiled/linked it for several months.  Nothing obvious has changed.

Does anyone have any thoughts on this?  Any suggestions for correcting this?

my problem ... but I would like to know what is really going on.

Thanks!

Forgive me for dashing this quick answer. I've been out of
town at a client's site since Monday and I'm trying to catch
up on several things.

The BUFL and BLKSIZE are related, but not the same. The BUFL
must be => BLKSIZE and all it does is make sure that the
buffer areas acquired are the size specified with BUFL. This
is a very old work-around to problems from way way back in
OS/360 days.

The only problem with BUFL is, if you set it too high, then
you will get x buffers of that size that have to be paged.
However, when I/O is done, the whole buffer has to be page
fixed. Ok, let's look at it this way:  Absolute max buffer
size is 8192 but you specify BUFL=32000. Also assume that
your system will default to 5 buffers. So 5 buffers of 32000
must be obtained (and backed by real page frames!). But you
won't S0C4 on a buffer overflow because you now own the
storage beyond the actual buffer (8192?).

So the advantage is, if you are using NON-cataloged data
sets, then you can make sure that any READ will not cause a
block to be obtained that is longer than any buffer. The
disadvantage is, BUFL is static and not dynamic. So you will
use that buffer length (size) regardless of how big or small
the actual blocksize is.

The light dawneth.  You have described the record as 2468 bytes long. 
If in fact it is less than that and it is the last record in the block
or the effective length of it plus the rest of the records in the block
is less than 2468 bytes, when you move it you are addressing memory
beyond the end of the block.  Depending on the size of the buffer, which
normally is the size of the block including block descriptor word and
probably rounded to a double word and how the GETMAIN was done, you will
be addressing memory not allocated to your program.  

Two solutions, both of which require that you add a RECORD VARYING
DEPENDING ON RECL-IN-WORKING-STORE-UNSIGNED clause where
RECL-IN-WORKING-STORE-UNSIGNED (or your name) is a 5 digit or larger
unsigned field, preferably USAGE BINARY in working storage.  The first
solution is to do a READ file-name INTO working-storage-record.  The
second solution is to always qualify the move of the record with
reference modification (1 : RECL-IN-WORKING-STORE-UNSIGNED).

If you don't like those solutions, your program will always have to
determine the length of the record read and confine its operations to
the area actually read.

You don't say whether or not it processes any records before it falls over. As
far as I can see there are two possibilities:

a) Your READ has failed

You don't seem to have any FILE-STATUS clauses, so how do you know if the READ
worked OK (or for that matter, the OPEN)? Suggest checking the FILE-STATUS
after every file opereation. On all files. If it's anything other than '00'
(OK) or '10' (EOF), then write out a message. It's also good practice to  do
your OPENs one at a time as it's easier to check which ones have worked (or
not).

b) You have read a short record as the last record of the block. Because your
MOVE is a separate statement, COBOL will try to move the full 2500-odd bytes to
working storage. If the record is only 175 bytes, you will get a protection
exception when COBOL tries to access the remaining bytes that are off the end
of the buffer. This explains why the BUFL fix works, as it forces the operating
system to get a bigger buffer than you really need, just so that you can avoid
tripping over the end. The best solution to this is to use READ INTO instead of
READ followed by MOVE. COBOL is smart enough to move only as many bytes as you
actually read from the input record in this instance, and so won't run off the
end of the buffer.

Hope this helps.

Steve
