>I prefer the method that you outlined, avoiding the use
>of duplicate keys.  It makes file updating and locking
>much simpler.  I use duplicate keys quite often, but tend
>not to use them for random access.

In general, I try to avoid duplicate keys, because of the overhead it
can cause (from serial searches for the key) on deletes and rewrites.
If disk space is not a problem, I often append a unique record key to
the alternate key(s).  Takes a bit more space, but makes for faster
updates.  I almost never declare a single, non-unique field as a key.
```

I wrote a large (>100k LOC) PC COBOL multi-user client-server inventory
system for a client in 1987.  I used Btrieve for the database and wrote
all the file I/O stuff in COPY files.  All the status codes are constants
in COPY files.  The application programs COPY in the files for the files
being used, and PERFORM the appropriate routines.  It works well, and is
very modular.  If I have to convert to another database, it shouldn't be
too difficult to simply rewrite the I/O handling code, and redefine the
constants as needed.  One refinement I have considered is to break the
PERFORM COPY files into individual paragraphs, then use a code scanner
to determine which specific routines are used in a given program and
insert the appropriate COPY statements.  It would reduce the compile
times, and make the programs a bit smaller.
```
