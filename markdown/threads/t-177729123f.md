I can't emphasize how important the last paragraph is. On some
machines COMP-3 (PACKED-DECIMAL) has horrible performance. In my
observations, COMP (BINARY) is much better than COMP-3 on almost
any machine that is not IBM big iron. The reason is that very few
have instructions to support packed-decimal. In some cases COMP-3
is better than DISPLAY, but in some it is not.

With regard to IBM S/360-S/390 etc.
It is true that COMP calculations are faster than COMP-3, however,
COMP uses fixed binary arithmetic in the hardware registers which
must be loaded from storage and the results stored again.
Ex. Load-Calculate-Store = 3 instructions vs 1 for COMP-3.
What's worse is COMP is internally converted to COMP-3 as an
intermediate step whenever the value is used for DISPLAY.
In other words, all the conversions that must be done (in
generated code) to handle COMP more than offset any
advantage of the individual calculations. COMP-3 should be
used for all "decimal business usage" while COMP is ideal for
subscripts and complex calculations which aren't often displayed.

OK, if the program won't be compiled exclusively for the IBM S/390,
then you're better off with just COMP.
