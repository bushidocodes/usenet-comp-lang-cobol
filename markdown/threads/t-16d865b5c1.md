This sounds like a problem we had with some DOS based 16-bit applications
when we tried to run them under OS/2 or Windows 95. Examples were
WordPerfect 5.1, and a number of other programs which responded to
individual keystrokes. The problem was that the program was using a
polling loop to check for keystroke inputs (a DOS function), so when it was
in a multi-tasking environment (for which it was not designed), it sucked
down huge amounts of CPU.

The problem was not as bad as it actually seemed, since the operating
system would only let it use 100% when nothing else was going on. However,
it looked absolutely awful, and it did slow down other applications
somewhat.

In your particular case you may be using one of the keyboard routines
available in MF COBOL, and it is generating a similar polling loop.

The clue is that the application will run under regular DOS, and that it
doesn't wait for an Enter or Fkey to do something. Not all such
applications will do this, but a surprising number did.

Right on the button Gary. Quite often older DOS based programs are ran in a
window using old style polling routines. Fine when you were the only
application but not so good in a multitasking environment.

Look for keyboard input routines using runtime calls, especially by-number
calls. Avoid x"B0" but x"AF" is usually OK. Stick some 'sleeps' in any tight
polling loops. These give back CPU cycles to the OS when you're waiting for
keyboard input.
