As a follow-on to earlier threads, I thought that I would add this
information concerning the use of OO COBOL in IBM mainframe shops (OS/390
and/or z/OS).

1) My internal sources indicate that IBM *does* know of some (more than
one - less than "lots") of shops using IBM COBOL's OO features.  None of
these shops, however, have gone thru the procedures to be an "official"
reference shop, so which they are is a "secret".

2) Enough shops are using (or considering using) this that IBM was "forced"
to keep the parts of SOMObjects needed for OO COBOL in LE (Language
Environment) when they removed it from being a "full feature" of OS/390 and
z/OS.

3) The SYNTAX on the Workstation (Windows and AIX) for OO COBOL from IBM is
totally compatible with that on OS/390 (and is INCOMPATIBLE with the current
draft of the next ANSI/ISO Standard).  Therefore, it is "highly likely" that
IBM will be needing to provide SOME migration path for existing IBM OO COBOL
source code.  (Whether this is a "source converter" or a "which OO dialect
switch" - is certainly unclear to ME now - but those internal to IBM are
*probably* thinking ahead to this already).

  ***

The issue of OO COBOL on IBM mainframes is related to the "perception"
(incorrect - but fairly common) that OO and GUI's are "inter-related".  Most
"enterprise" size shops that are looking at OO (any language) first (and for
some ONLY) look at it for "user interface code".  As "user interfaces" in
IBM mainframe environments are USUALLY handled by CICS, IMS, ISPF, etc -
this is a "non-issue" for their application development (and maintenance).
If they are moving to a "workstation front-end" for their applications, then
they do look at a GUI *and* OO programming environment.  HOWEVER, as such
programs (by their nature) are done from "scratch" - it is likely that shops
will look at the OO programming languages with the most "press" rather than
where they already have in-house expertise.  (Furthermore, as these UI
portions of applications PRIMARILY run on the workstation, it doesn't
involve any OO on the mainframe.)

Bottom-Line:
  Until it is commonly "perceived" that OO is beneficial to the
NON-user-interface business logic, I don't see OO COBOL catching on (very
much) in IBM mainframe environments. When/If there are *standard*
(off-the-shelf) OO Class Libraries for *BUSINESS* rules, then I think that
they will be "accepted" and brought into use in IBM mainframes - to a much
greater extent than today.

upon VisualAge COBOL and DB2.  It troubled me that VA COBOL was generating OO

generate "classic" COBOL.  I never got the answer.

Often, the code doesn't even compile, because I'm using a different compiler
than the author.  To me, THIS ISN'T COBOL!  The reason COBOL is the most widely
used computer language is precisely because of the standard, that describes not
only the language syntax, but the results of using that syntax.  COBOL is
supposed to accept the same source code on all platforms, and the source is
supposed to perform the same everywhere.

I find the "early adoption" of OO COBOL to be an unfortunate by product of the
slow adoption of new COBOL standards.  People naturally want to try out "what's
now".  However, those who are using the OO COBOL than any vendor offers right
now are taking a BIG risk.

What do others think?

Whether we like it or not, the future of computer programming lies in Object
Orientation. If COBOL misses this boat it will be dead in the water within
10 years.

The rise of OO languages like Java and C++ is not just down to fashion. It
is also because the Object Orientation makes them efficient and useful.

OO is the springboard that will give us components and that is where the
future is. (I have written at length elsewhere on this. If you want to see
the arguments go to http://www.aboutlegacycoding.com, take the "Archive"

You said in your post that you had trouble with understanding OO.

So did I.

OO COBOL is verbose and difficult when you approach it from a COBOL
background. The concepts are obscure and buried in new jargon. However,
there are shortcuts you can take and they are covered in the article
referenced above. Once you have the concepts, it falls into place pretty
quickly.

The game is coming down to using the best tools for the job. One single tool
won't hack it anymore. COBOL is still the best tool for certain jobs and is
likely to be for the foreseeable future.

But it is isolated by its archaic file system, and its lack of "object
awareness" which prevents it playing nicely with other languages.

OO addresses this and puts COBOL on the same footing as the other languages.
Components developed in OO COBOL behave just like components developed in
any other OO Language. The Source Language is irrelevant and should be the
one best suited to the purpose.

If you want to have a future using COBOL for anything other than Batch
Processing, learn OO and encourage others to do so too.

The future belongs to components, and you need OO COBOL to build them.

The "early implementations" which you perceive as "risky", are giving people
a chance to get with the program and improve their skill sets before it is
too late.

Apart from that, there are huge long term benefits in adopting OO, even if
you never write a single component with it.

years to come.  Furthermore, that MOST of the programs that will be

they could ALMOST be compiled by an ANSI '74 compiler (at least a compiler

programming environments in which it is not fully entrenched today.  The

for COBOL already exists and there are many tasks for which COBOL IS "the
best tool for the job". However, in the absence of an OO Version, it is
tempting for people to use a less appropriate tool because they have to do
SOMETHING...

I already have Client Server applications where VB, C++ and OO COBOL all
co-exist peacefully on the same GUI form. Each component written in the
language best suited to it, and the whole lot completely transparent to the
User. (This stuff has been running in a live environment for over 6 months
now without a single failure by any component).

This violates totally the old COBOL dictum of "One Language for ease of
maintenance". The difference is that these components don't NEED to be
maintained, any more than you would maintain an individual brick in your
house. (You might replace it; you wouldn't try and maintain it).

I am currently working to bring Java and HTML into this same scenario so
that my existing applications can be Web Enabled easily and quickly. Fujitsu
Version 6 (Enterprise Edition) supports COBOL calling Java, but it is via a
limited interface and you cannot call COBOL from Java. As I am not using V6
for reasons which I won't go into here (but you could try reading some of
the posts on this subject in CLC), AND I don't have the Enterprise Edition,
AND I want to be able to call both ways, I am building my own Classes to
enable calling Java  (and, hopefully, vice versa) from Fujitsu Version 5,
Professional Edition. This may be achieved by using JNI or CORBA, I'm not
sure yet as I am still looking at it. It will CERTAINLY require OO COBOL or
PowerCOBOL (which is innately OO).

The point is that there is no point in using a hammer for every kind of job.
Sometimes a chisel is more appropriate.

All of the arguments involved in maintaining Source code or building a
Component Repository are covered in the previously referenced article. I
cannot go into them here. It is a significant attitude adjustment,
especially after a lifetime of working on COBOL Source, to find that within
the foreseeable future this will very probably be replaced because it is
simply not cost justifiable to do it any more, and it is becoming
unnecessary. Off the shelf components, developed in Languages which are
ideally suited for the purpose for which the component is intended, and
selling for the cost of a few programmer hours, will make the maintenance of
large IT development teams unjustifiable.

The keyword here is "reuse". And that is provided automatically by OO.

IT development will be "outsourced" just like the company car pool

(response to comments "below")

1) I agree with Stephen that "cost" of software itself is RARELY the major
stumbling block to "upgrades" in an IBM mainframe environment.  It usually
is SOME consideration (If you don't know about IBM's policy about the
difference in costs between upgrading between releases and versions - you
have a "real treat" ahead for you).  Actually, there is OFTEN a bigger
consideration in LARGE shops when you "add" a CPU or upgrade the "group"
that your main CPU is in - because the price of everything that you already
own MAY go up.

2) My perception of LARGE IBM mainframe shops is that there is always,
ALWAYS a "backlog" of maintenance and "minor enhancements" that need to be
done on existing applications.  This means that there is *never* time to
re-do (fix from scratch) or do totally new development - places where it is
easiest to use new tools, paradigms, and programming approaches.  Although
it is POSSIBLE to add a few "scope-terminators" or EVALUATE into an existing
application, "switching it" to OO (or even nested programming) is simply
"out of the question - as it can't be cost justified (in terms of
resources - training, money, time, etc).

3) I know of several shops that did "pilot" projects with various
application generators, development techniques, testing tools, and other
"new" ways of solving maintenance and enhancement backlogs - during the '80s
and '90s.  Even when such pilot projects went well (which often they did
NOT), moving the "new" approach into existing application groups was often
exceedingly difficult and equally often unsuccessful.

4) "growth in skills and approaches" in such shops is NOT totally hopeless.
Things like Y2K or IBM dropping support for older hardware/software does
SOMETIMES cause shops to "come up to date" in how they do things.
Furthermore, I have often seen that one (or a few) "experienced" programmers
who are well respected can introduce a "new" technique into the way that
they work - and that others will then follow (gradually - VERY GRADUALLY).

  ***

Bottom-Line (IMHO) - IBM mainframe shops are NOT deterred by the cost of
new/different software (well not as much as PC users are).  They *ARE*
however, severely impacted by the "cost" of actually implementing new
techniques and tools so that their (large) programming staffs actually use
them and expand their use into both new and existing applications.

Take the DBI - yes I do it in a particular way in OO COBOL but it can be paralleled

Very briefly, and I don't want to get into whether it is a sub-program, nested,
called or whatever (currently being hotly debated here <G>), but the following is
achievable both OO and non-OO. Bear with me, I'm so far removed from conventional
style programming because I code solely in OO classes, but essentially you need to
do the following to mimic what I do in OO ::-

(a) To have a 'File Program'' as a TSR, i.e. 'memory resident' while in use - in
other words you have got to avoid EXIT PROGRAM, (which will CLOSE the file before
you really want it to be closed), as a result of a call
(b) Have access to the file-status for this file
(c) Pass it the full pathname of the file you are using
(d) And to mimic OO methods use an entry-point for each file VERB
(e) your file record format is a copyfile used in both Business Logic and the DBI
'File Program'.

Unfortunately it would take too long to spell out the detail here.

Given that you resolve (a) through (e) you have a working 'File Program'. Firstly of
course you test the following against a dummy program. Gently taking ONE file at a
time, you now search for those programs that use this file. Hopefully your source is
decently structured and you have specific PARAGRAPHS that you PERFORM for file
accesses. Don't delete the paragraphs but remove the READS/WRITES/REWRITES etc.
substituting 'Call Programfile 'entrypoint' using AAAA returning BBBB (see I don't
even know the syntax). 'Entry Point' is a separate piece of code for each COBOL file
VERB. Using OO syntax it would look as follows :-

*>----------------------
method-id. "readRecord".
*>----------------------
Linkage seciton.
01 lnk-InPrimeKey                pic x(06).
01 lnk-values.
copy "results.cpy" replacing ==(tag)== by ==lnk==.
copy "customerRec.cpy" replacing ==(tag)== by ==lnk==.
Procedure Division using lnk-InPrimeKey returning lnk-values..

    move lnk-InPrimeKey to Customer-PrimeKey
    Read CustomerFile
        key is Customer-PrimeKey
            invalid key
                 set RecNotFound to true ( in lnk-result copyfile)
            not invalid key
                move Customer-record (in Global W/S as a copyfile)
                to  lnk-Cusrecord
    End-Read

You can even use the same technique to access SEPARATE Table Programs which
are the only ones containing SQL code/logic. I mention this because your comment
above seemed to imply SQL was tied into OO - it is just as legit in non-OO.
For SQL, the above method  becomes, (abbreviated) :-

    SELECT DISTINCT WHERE CusPrimeKey = :host-CusPrimeKey etc...

and with the addition of a copy file for host variables, you can still use the
copyfiles above to return to your Business Logic.


The above models would dove-tail into OO, without major change,  if you ever
switched from non-OO.

*>-------------------------
Method-id. "SeeIfTheyQualify".
*>----------------------------
Local-storage section.
01 ls-AccountCode                        pic x(02).
Linkage section.
01 Lnk-CustomerNumber        pic x(12).
01 Lnk-DiscountRate             pic (02)v9(01).
