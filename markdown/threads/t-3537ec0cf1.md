```
I have heard & seen much about code reusability as a benefit of
stuctured programming since the mid-70's when structured coding was the
buzzword of the industry (for mainframers).  But in reality, how much
code is actually suitable for reuse in other programs, and what methods
are available to catalog and document these routines, after all these
years.  Zilch !!

I have seen programs or routines cloned many times, and find that the
extraneous data areas and/or procedure code has not been removed, so
that it obscures to some extent the actual usable portion, and thusly
does not contribute to the maintainability of the new program.  The
reality is that most "production programmers" (as opposed to academics)
simply don't have time to clean up this cloned code, and in fact don't
normally even know of usable code routines other than those they have
either written or  maintained personally, as there is no facility to
search for these routines within an organization's code bank. 

I myself have found, over the years, that by the time I locate a
suitable  candidate segment of code, and clean it up, I might have well
have written it again from scratch, with everything about it tailored to
the task at hand.  It is simply more cost-effective, and that is what IT
as an organizational P&L center is judged on.  (Please let's not digress
into Dilbertism here).

Code reuse is an admirable goal, but the reality is far short of the
goal !


Bob
```

```
The copy-and-modify approach is the lowest form of reuse, and it 
suffers from the kinds of problems you describe.

A better form of reuse is the use of separately compiled subprograms,
because the reuser doesn't (or shouldn't) have to look at the internals
of the subprogram, only at the interface and the documented behaviour
of the subprogram.

However, this approach calls for careful design, identifying chunks of
generic and reusable functionality.  In my experience most COBOL
programmers just don't think that way.  They are used to doing
everything in one huge monolithic top-level module.  At most they make
a few CALLs to some low-level utility modules, often written in 
Assembler because they do things you can't readily do in COBOL.

Documenting and publishing the available routines is another issue.
It's time-consuming and rather tedious, but I don't know why it would
be any worse for COBOL than it is for other languages.  For C, at 
least, there is a long tradition of buying commercial libraries of
routines for various purposes, or downloading free ones off the Net.
I don't know how many shops develop such libraries in house or how
well they reuse them across applications.

For my own application I developed a number of reusable routines and
documented them with web pages on the company intranet.  That's a
pretty good form of documentation because of the usefulness of
hypertext.  However, none of the people in other applications has
shown the slightest interest.  I guess I'm not a very good evangelist.

COBOL has no tradition of freeware or public domain code.  Our code
belongs to our employers and we can't share it, so we continually
rewrite the same code others have already written, feeling no
incentive to make it generally useful because we're the only ones
who will ever use it anyway.
```

```
And we become pretty efficient at rewriting code too.  It doesn't take

There are big advantages to an integrated system which share code and

it exists.  It seems safer to change it for one program, test, &
implement, then change it for another program, test, & implement, etc.

I also don't like having one person maintaining a called program while
another person maintains the calling program.  Or worse replace those
with objects.
```

```
The function types are: Alphanumeric, Boolean, Integer, National, and

data division.
working-storage section.
01 ptr pic s9(4) value 1.
linkage section.
01 last-name pic x(30).
01 first-name pic x(30).

procedure division using last-name, first-name, middle-initial

    move spaces to composite-name
    string first-name delimited by space

        into composite-name with pointer ptr
    end-string
    if middle-initial not = spaces *> prevent additional spaces in output
        string middle-initial delimited by space
            space delimited by size
          into composite-name with pointer ptr
      end-string
    end-if
    string last-name delimited by size *> last name may contain
                                                            *> an embedded
space
        into composite-name with pointer ptr
    end-string
    .    *> the exit must be in a sentence by itself - CD 1.2
    exit function.
end function composite-person-name.

One would define a function-prototype for the function and
reference that prototype in the REPOSITORY paragraph.
The file (? - method is implementor-defined) referenced would
contain the text:
-----------
function-id. composite-person-name is prototype.
data division.
linkage section.
01 last-name pic x(30).
01 first-name pic x(30).
01 middle-initial pic x(2).
01 composite-name pic x(64).
procedure division using last-name, first-name, middle-initial
              returning composite-name.
end function composite-person-name.
-----------
Using the function:

    move
      composite-person-name ( em-name-last, em-name-first, em-name-mi )
        to pr-em-full-name

referenced in the repository paragraph is sufficient to prevent it
from being recognized in other contexts, such as an array.

paragraph allows the translator to verify that the parameters to the
function are of the correct type.
```

```
Thats cause of economics:
Since only a programmer could re-use the code, only a programmer would
get paid to find reused code. But if a programmer gets paid more to
write new code, then cataloging it would mean loss of income.

The other thing, which non-programmers don't understand, is re-use is
a buzz-word. It's one thing to say re-using code is a strategy. It's
another thing to actually go through an existing program, and find
source or data which *matches* the re-use portions, and make certain
the re-use itself doesn't get in the way.

Pretend you re-use a portion of code which is in a normal accounting
program. Now pretend you take that same code, and re-use it in a
device driver. A stupid programmer could simply cut and paste, or
place it into a copy file. A programmer with experience would know the
device drivers *compile* must be *assured* to still have interrupts
off or on, and the code as compact as possible.

The other thing is, re-use is nice, but eventually you get to a point
where a pointer to the re-used code is *harder* to maintain than
duplicating it. An example is porting mainframe code to PCs. A PC
editor might require you to load the source into seperate windows, or
just edit one file at a time. A mainframe editor might have the copy
books virtually pasted and updated. The mainframe code, when ported to
a PC, if it reuses too much would take forever to edit, *because* the
PC programmer would have to open and close for each piece of reused
code. The mainframe programmer would not understand this when
designing the code. And if the PC editor doesn't work like the
mainframe editor, are you going to hire a person to rewrite the
mainframe source code editor as well as porting the code? How many
others would invest in such a change?

Considering tech support gets $40/ call, it would cost $1000 for this
renovation, I would think, at least.

The main catch to code re-use, though, is to *pay* some programmer to
review code.
```

```
writing software. They follow a waterfall methodology to define a number of
processes. Having defined these processes, they then assign a number of
analysts to define specifications which eventually get distributed to
programmers to crank code. Some time later all these independent pieces are
thrown together in a test environment for integration testing.

This kind of thinking is very much like the techniques used before the
industrial revolution, with a corresponding degree of productivity, quality,
etc. If industrial engineering techniques are applied to software
development then dramatic productivity increases can be achieved. Today the
term 'software engineering' is an oxymoron.
```
