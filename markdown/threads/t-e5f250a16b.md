```
The above assumes that both fields are left justified, which is normally 
the case. If that is not the case, remove the leading space with an 
unstring delimited by ALL SPACE.  The ALL will tell it to consider 
strings of space as a single instance of a space. That is not needed 
above, as you do not really care how many trailing spaces are there.

Your problem with the console/screen is that Cobol has both an 
operator's console, and the screen section has a screen window.  They 
are two different things. The DISPLAY verb can be directed to either(see 
the ON modifer) ... but the default is that a simple "DISPLAY 'Hello 
world'" would come up on the operator's console, while a "DISPLAY 'Hello 
World' AT LINE 1 COLUMN 1" would come up on the screen. All screen 
section stuff goes to the screen, not the console. The normal way of 
doing it in Cobol is to use the console for crash information ... stuff like

		DISPLAY "The sky is falling !!!!!!!"
		DISPLAY "Fatal Error - Aborting".
		STOP RUN.
```

```
This looks a lot like homework, and I don't know anything about the Fujitsu
compilers, but a couple of suggestions from "generic COBOL" presuming that
Fujitsu V3 is an '85-compliant compiler:

I'd locate the first trailing blank in each field  using the INSPECT verb to
tally *leading* spaces in the results of the intrinsic function REVERSE
using the field as argument.  You can also do this "longhand" by redefining
the field as an array of single characters, and examine each character
starting with the last looking for something other than spaces, but that's
rather more tedious.

Once you know where the trailing blanks start in each of the fields, the
bulk of the work's done.  It's probably easiest to preinitialize the
destination to spaces.  Then you can position the two names individually
using reference modification on both source and destination.  Or you could
use a single STRING whose source arguments are the first name (suitably
reference modified) delimited by size, a single space delimited by size, and
the second name (also reference modified) delimited by size.

That's the approach I'd take, anyway.  Reference modification was introduced
as a feature in 1985 standard COBOL, and the REVERSE function in the
Intrinsic Function Amendment of 1989, so unless you're using a dialect of
ANSI-74 COBOL, neither is a vendor extension.  .

    -Chuck Stevens
```

```
I mention the following because of those out there who may not understand
the tradeoffs between subscripting and indexing.

An actual reference to a table using an index should (in general) always
be at least as fast as subscripting, usually faster. However, there are some
other considerations. The COBOL standard does not specify how indexes
are stored, it only provides the framework for the compiler vendor to use
whatever efficient method is applicable to the target hardware. The vendor
could simply implement indexes as subscripts, and the two would be the
same. Most implementations I am aware of store indexes as table offsets
{i.e. (subscript - 1) * entry size}. The Burroughs/Unisys V Series COBOL
compilers actually stored both the occurrence number (subscript) and the
offset, IIRC. However,the compiler must generate code to convert between
subscript and offset when a SET or compare is done between integer and
index data items. The actual tradeoff is a complex relation involving the
number of conversions between integer and offset, the number of actual
references and the relative speed of  the conversion. But if the program
averages 100 conversions and compares between indexes and integers for
every indexed table reference, you can be sure that indexing will be slower
overall than subscripting, in that program.

Another interesting situation is when the table elements are one byte/word
(depending on the hardware's addressing schema) long. If the hardware
architecture is such that the address of the beginning of the table is greater
than zero, so that the address of FirstElement - ElementLength >= zero,
indexing and subscripting can be identical in performance. In this case, the
compiler can simply move the subscript to the index register and reference
(FirstElement - 1) + index. Zero overhead.

My point (as ever :) is that indexing will not be faster than subscripting in
every possible situation. In order to evaluate these tradeoffs, a programmer
must understand at least generally how things are done at the machine code
level. I shudder when I see programmers blithely using an exponential
operator in a bit fiddling routine, believing that because it looks elegant
and concise it therefore must also be efficient. Ouch! Give me a break! :-)
```

```
code, when programmers used the COBOL IF verb.

In fact, it ALWAYS generated extra instructions to support an ELSE branch,
whether these were required or not.

As Chief Programmer on a Burroughs site at the time (1969), I endorsed a
local standard requiring people to AVOID the use of IF when writing COBOL
for this machine. We used ALTER instead, as it was extremely efficient and
(unlike the programming wimps of today <G>)  we never had any trouble
maintaining it. (All "switch" type comparisons were implemented with ALTER
and we only used IF when it was a KEY (string) comparison. Arithmetic
comparisons were carried out using subtraction and sign testing... You can
tell whether A >, <, or = B by subtracting A from B and testing the sign of
the result...)

If you find that amazing, consider that it was standard practice at the time
I came on board, to do a COBOL compile, punch an object deck, then patch the
cards to remove the redundant branches from the generated IFs, before
loading and running the program. At least the above was an attempt to avoid
patching object decks.

(Bear in mind that my team was composed exclusively of ex-B500 Assembler
programmers who had no qualms at all about modifying instructions
dynamically. They were just getting used to using COBOL and the body of
standard and best practices which we take for granted today, was not in
place then...)

Anyway, "everybody knew" that IF in COBOL was crap, and avoided it.

Now let's consider our hypothetical programmer moving out of this
environment into the new B3500 environment.

He hasn't had a course on the new machine (Hey, it's COBOL, right? We bought
it because Burroughs told us we could increase productivity and it was
platform independent... And training takes time and costs money...)

So our hypothetical programmer has had it drilled into him that IF is "Best
Avoided" and by now (it is a few years later), ALTER is being "frowned on"
as well.

But wait! There's this EXAMINE verb that can check for something being zero
or not WITHOUT using an IF!

Set the fields with an initial value of zero. Then test if they have
anything in them. No worries about arithmetic exceptions and the new
powerful hardware should cream it. After all, it is only one byte we're
checking...

While it is impossible to establish whether the above is actually what
happened or not, the point I am trying to make here is that, in my
experience, programmers (for the most part) are NOT stupid people. If they
do stupid things it is usually because they are operating on false
information or assumptions.

In 2003 it is easy to snigger at "solutions" from 30 years ago. Before you
do so, consider that someone in 2033 may do the same to that beautiful piece
of code you cut yesterday, of which you are so proud...
```

```
You didn't say specifically whether you are using the 'simple' ANSI
DISPLAY or the F/J Screen Section. Chuck has given you one solution. The
following is  less elegant and not rocket science :-

SurName = 30
FirstName = 20

perform varying n from 30 by -1 until SurName(n:1) <> space
*> the above is a 'play safe' but could be written as
*> perform varying n from 30 by -1
  if Surname(n:1) not = space
     move n to Length1
     EXIT PERFORM
 End-if
End-perform

-- same for FirstName giving Length2

intiliaze ws-Text51
string Surname (1:Length1)     delimited by size
         x'20'                             delimited by size
         FirstName (1:Length2)  delimited by size
         into ws-Text51
End-string

Some might deprecate the following technique, but it's one I use. It ties
back to your initial design. I have many sets of files which have
repetitive descriptions, some 40 characters used over and over again -
sheer waste of disk space. So I 'build' the descriptions from hard coded
lists of descriptions which terminate with a Tilde (~) and STRING the
parts together using the Tilde as the delimiter.

Using it with your Name fields, the file contents would read :-

pic x(31) = Smith~
pic x(21) = Adam~

As I said my lists are hard coded, but if you were using data in a file
then you could adopt the following approach :-

Input LastName pic x(30) - file - pic x(31) allowing for maximum name plus
Tilde
Input FirstName pic x(21) - file - pic x(21) etc.....

You accept the names, do a perform FORWARDS ( using a positive 'n')
checking for the first space and put the Tilde in that position (Reference
Modification) in your file record. Only relevant if you are *constantly*
stringing names/descriptions together. There will be times where you want
to display fields independently - not showing the Tilde - INSPECT MyField
REPLACING ALL "`" by space.

The latter comments are not your immediate concern - but I've included
them as something for you to bear in mind in your future IT career.
```
