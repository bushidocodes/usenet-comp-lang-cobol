Why would they want to maintain the JCL?

Talk to them. What they REALLY want is to have their current batch
processing performed in a client/server environment.

The best approach to this is to analyse the FUNCTIONALITY of their batch
processing. (This requires the input of at least one knowledgeable
human...).

Once the batch functionality has been determined, it is possible to write
windows scripts that make it happen. Windows Script Hosting is excellent for
this and for many years I never used it, considering it was some esoteric MS
technology. It isn't. It is a very useful equivalent to JCL on a PC. (Unix
scripts could do the same in that environment).

Sometimes people think they know what they want,because that is the extent
of their experience. Find out what is really needed and wanted, then
implement that, using the tools of a PC environment.

A JCL "converter" might be a possible approach, but it is not one I would
personally favour.

Get them to realise that this is a huge opportunity to review what happens
in their batch processing and possibly improve it. The action of analysing
the functionality of their current batch processing is a valuable one and
will drive out opportunities for streamlining it in the new evironment.

Emulation depends on the legacy platform.  For example, an HP3000 MPE
emulator exists for HP 9000 unix, but to my knowledge, not for Windows
or something else.  Such products are expensive and typically contain
a layer to interface to the target File System - a license.

I have a facility I wrote in native command language several years ago
for Windows 2000, specifically native DOS.  It requires a standard,
wrapping each step with job-like statements to control sequential
batch flow, typical of some legacy JCLs.  All statement are native
DOS.  This is possible since the enhancements to the command
interpreter in Windows 2000.  It is possible to build stacks with DOS
environment variables which was missing in earlier versions of DOS.

The functionality...

- No compiler or intermediate level parsing, no add-on script
languages - pure native DOS batch language.

- The mainframe batch model can be migrated to Windows.  The support
model can be migrated...the steps and inter-launch (call) structure of
the JCL can be migrated.  Calling can chain or launch parallel batch
jobs, like legacy.  A sequential job number facility was built in.

- A source batch file is protected, never edited by a support person,
never partially executed.  Production batch files can be locked down
in a production environment.  Support documentation mirrors the batch
step structure.  Development follows a standard for implementation.

- Job start up, step execution and job termination is stamped and
logged.  An abnormal termination is externally retained when a batch
fails.  This is job step tracking.

- The internal step flow of batch files is sequential, is NOT a
looping batch model.

- Fully supports nested batch files.  For example, batch files can
call 10 levels deep (or more) and a failed step in the 10th batch file
will exit all the way up through the call structure to the leading
batch file and terminate with an abnormal status, leaving the job
outstanding in JOBSTEP and a clear path for notification to support
and an outstanding restart point for the job.

- Step management allows support to restart the job at the failed step
or at some other step, or to flush it from active job status and
restart the job at the beginning.  The support person controls
interrrupted batch jobs externally.  Everything is logged creating a
pseudo resolved JCL output list.

- The facility is scalable, can be implemented at the server level or
at the application level, can support multiple instances of the same
application on a single platform.

...

There is more...its been over a year since I looked at it.  I am not
offerring a perfect or flawless solution, but one that is simple and
functional.  I wrote and implemented a version of this facility at a
major American corporation, one of the largest corps in the world.  It
was based on a legacy Jobstep facility they had on all their mini
computers.

As a legacy migrator, I initially wrote 45 lines of DOS and saved the
corporation a pile of money.  After I left, I simplified and improved
the facility, added nesting, job number and flexibility...changed the
way it looked in source code to the point that it is a new product.

It is especially good for migrating batch because you can preserve the
batch model AND the support model.  The mainframe support people can
support both the Windows implementation and legacy implementation with
nearly identical support views.  And the legacy style control is
migrated.  The support documentation was virtually THE SAME for both
platforms.

What you have to consider is how to handle flat files.  You should
look for a replacement for the DOS COPY command, one that behaves
exactly the way you want, handles empty files elegantly.  You can
write one in COBOL using the native OS library for byte stream IO. 
Net Express Cobol (Microfocus) provides such an interface.  You just
have to pretend you are writting in C, know how to buffer IO without
assuming a record size.

MF Net Express also provides a fully functional SORT utility that
works just like legacy sorting.  COPY, SORT and Jobstep typically
solve the largest chunk of JCL migration to DOS.  The DOS environment
is one of the simplest interfaces for return codes from compiled
programs.

You can contact me at    star_ledge@yahoo.com

I have since worked as a developer for a migrator looking to write a
converter for JCL to unix/Windows or whathaveyou.  They convinced
themselves that JCL could be converted to Java and wouldn't consider
much else.  Windows script, PERL, COBOL script, JAVA script, Java,
etc...bad idea.  They all create runtime "environment" issues that are
very difficult if not impossible to resolve.  (and they went bankrupt)

A JCL file has to be able to launch another JCL file (similar to a
Cobol CALL), pass parameters, preserve the caller's stack of
variables, receive a return code, decide what to do, log...be
flexible.  The runtime environment is a huge issue.                   
                                                                 `
It is not just a technical issue, it is very much a management issue. 
You have to rewrite the documentation and retrain support, perhaps
support two platforms at the same time.  It costs money.  If the
system is mission critical, an accidental duplicate step can trash a
database...if it is noticed.  Every time something is done, be it copy
a file or update a database, the step MUST be logged.
