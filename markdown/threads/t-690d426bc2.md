I thought the programming looked like a homework assignment myself.

So I assumed the teacher would want to see the "ENDIF". So I mentioned using the
"ENDIF" to indicate where the "IF" ended.  Since it is pseudo-code, you can not
just end it with a period like COBOL.  It looks like a bunch of "IF, THEN"
statements with no clear ending.  I did make a suggestion to use "IF, ELSE IF,
ELSE" however.  

It isn't wrong to use several "IF, THEN" statements, it just causes the program
to check them all.  Normally if you have historical information, it is best also
to use the most popular choice first; however, this would throw the the flow of
the logic off a bit.

I used to prefer that and if the "if test" is easy to understand then
I will go that route.  My first preference is to make the if test easy
to comprehend.  I avoid not logic in the if test like the plague.  I
mostly end up using not logic in the "if test" when I am working with
existing code and not using the not logic would cause a fair amount
more work then it is worth.

Now to the main point of the thread.  I rarely pseudo-code.  I usually
take a stab at it straight away as code and some time debugging.  If
the debugging is not getting anywhere then I will step back
pseudo-code the process and run some thought experiments with the
pseudo-code.  Then I will re-code.

Pseudo-code is a thing that usually defies standardization, it is
primarily meant as an aid to the original coder of code.

While I agree with you re. "nots" in IF stmts, I also feel that they're

to say "Yeah, yeah, but I still like my way." It takes
