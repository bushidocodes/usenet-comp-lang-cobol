```
Assuming the receiving field is alphanumeric, then using X'04' in a VALUE
clause or in a MOVE statement works fine. You can not use Hex littorals in
the value clause of a packed-decimal (COMP-3) item. However, you could
(obviously) use a COMP/BINARY field as the sending item if the receiving
item is numeric. Then you would just give it a value of 4 or 8.
```

```

› Can you tell me the theory associated with the cobol COMP-3 and COMP-4
› clauses.

Comp-3 is packed decimal, Comp-4 is binary.

Packed decimal is NOT a cobol format per se. Packed decimal
originated as an IBM mainframe format and is built into the
hardware of mainframe machines. There are specific machine
language instructions to manipulate packed data. Comp-3 is how
cobol implements the USE of packed decimal.

Comp-4 equivalent to Comp on the mainframe is standard binary.
Stored either as a half-word, full-word or double-word or
2, 4 or 8 bytes respectively.

That's the "theory". Look up the rest in your manual.
```

```
COMP is from the ANSI standard. In theory it represents the most
efficient numerical representation on the system. COMP-n (for various
n) is an extension to the standard. In IBM (IIRC), COMP is binary,
COMP-1 is float, COMP-2 is long float, COMP-3 is packed decimal, and
COMP-4 is binary. COMP-4 is guaranteed to be portable from one IBM
system to another, while COMP might be implemented differently from one
platform to another.

In Burroughs Medium System (later Unisys V Series) the native numeric
format was decimal so COMP was decimal. On Burroughs Large System (now
Unisys A Series) the native numeric format is a 48 bit real, with an
integer stored as a real with a zero exponent, so that was COMP. COMP-3
was packed decimal on both systems for compatibility with IBM. Then
corporate standards came along and decreed that COMP would be packed
decimal in all Cobol-74 compilers. It really killed us on benchmarks
until we implemented the BINARYCOMP compiler option. In older A-Series
compilers floats were COMP-4 and long floats were COMP-5, but now we
call them REAL and DOUBLE.
```

```
I frequently had the need to work with hex constants in OS/VS COBOL and
in ANSI-74 COBOL for an IBM Series/1. This is how we did it:

01 HEX-CONSTANTS.
05 HEX-0003 PIC S9(4) COMP VALUE +3.
05 FILLER REDEFINES HEX-0003.
10 FILLER PIC X(01).
10 HEX-03 PIC X(01).

I had to use the same thing in reverse to convert 1-byte binary date
fields (this was back in 1979) generated by a Series/1 assembler
program. You do have to know how to convert X'DC' (for example) to
numeric value 0f +210 in order to initialize your constants, but you get
the general idea.

This method may not be portable. I did something like this in Realia
COBOL which prefers COMP-5 binary format. IBM iron is big-endian, Intel
is little-endian, so I had to reverse the order of the bytes under the
redefinition.

I used to have to view COBOL source code in PANV that included embedded
hex constants. There was one old program that always crashed PANV
because it had an embedded value that messed up the 3270 data stream.
```
