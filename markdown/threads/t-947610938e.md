Mike,

I develop PC based business systems on the side. I have several that I
have developed over the years. My first were with an ANSI 74 compiler and
are rife with GO TOs. They work. Later ones are Ansi 85 and have fewer GO
TO's but they are still there. After listening in here on the news group
for the last few months, and reading Don Nelson's fine book on ANSI 85
COBOL, I have taken up the GO TO less banner.

Over the years I had reached a certain level of proficiency. I had a
certain level of "bugs" in my code. Always different - I learn from my
mistakes - and never VERY serious - some really busted the code, but were
easy to fix. At any rate, my last 2 systems were developed without using
any go to's. Design had to be re-thought a bit, but the avoidance of the
GO TO has led to a design that has been a JOY to maintain, and one that has
had noticably fewer problems - NONE in one system. Testing was simpler and
easier, the design has proceeded faster and the code is cleaner, more
concise and easier to understand. I'm on the band wagon!

After 10 years of maintenance programming I have come to the
conclusion that 99.5 percent of all bad programming comes from a
single cause: the programmer does not fully understand the problem
before he starts writing code. Whether he uses all GO TO's and no
PERFORM's, or all PERFORM's and no GO TO's, is insignificant. What
does the damage is not understanding things like Variable overhead,
Returned goods allowance, Consigned goods, Bill and hold, Present
value, Labor variance, 401K, Retained earnings, etc.
This failure comes from three sources:
1. Unrealistic deadlines on the part of management;
2. Laziness on the part of the programmer;
3. Fear of looking stupid (User, I know you have explained Standard
Cost to me seven times, but I still don't get it. Would you explain
it to me again?)

OK, here are some examples of EVAULATE vs IF:

DISPLAY "ENTER Y OR N:"
LINE 12 POSITION 01 ERASE.
ACCEPT YN-ANSWER
LINE 12 POSITION 15 UPPER ECHO PROMPT NO BEEP.
IF YN-ANSWER = "Y"
DISPLAY "YOU ENTERED A Y"
ELSE
IF YN-ANSWER = "N"
DISPLAY "YOU ENTERED A N"
ELSE
DISPLAY "WRONG ANSWER! TRY AGAIN"
END-IF
END-IF

same code w/ evaluate:
DISPLAY "ENTER Y OR N:"
LINE 12 POSITION 01 ERASE.
ACCEPT YN-ANSWER
LINE 12 POSITION 15 UPPER ECHO PROMPT NO BEEP.
EVALUATE YN-ANSWER
WHEN "Y"
DISPLAY "YOU ENTERED A Y"
WHEN "N"
DISPLAY "YOU ENTERED A N"
WHEN OTHER
DISPLAY "WRONG ANSWER! TRY AGAIN"
END-EVALUATE.

--or-- (I prefer the previous example to this one)
DISPLAY "ENTER Y OR N:"
LINE 12 POSITION 01 ERASE.
ACCEPT YN-ANSWER
LINE 12 POSITION 15 UPPER ECHO PROMPT NO BEEP.
EVALUATE TRUE
WHEN YN-ANSWER = "Y"
DISPLAY "YOU ENTERED A Y"
WHEN YN-ANSWER = "N"
DISPLAY "YOU ENTERED A N"
WHEN YN-ANSWER NOT = "Y"
WHEN YN-ANSWER NOT = "N"
DISPLAY "WRONG ANSWER! TRY AGAIN"
END-EVALUATE.

You can get pretty deep with this EVALUATE thing. I typically use it when
I have a large IF-ELSE structure. It makes the code much easier to read.
There are times, though when the EVALUATE statement is not efficient from a
coding standpoint, and I use IF. Any ANSI 85-compliant compiler will have
EVALUATE. I hope this helps you out.

ELSE
IF variable = '2' DISPLAY 'variable = TWO'
ELSE
IF ....

Scenario #2 used the 'EVALUATE condition' test, ie:

EVALUATE variable
WHEN '1' DISPLAY 'variable = ONE'
WHEN '2' DISPLAY 'variable = TWO'
WHEN '3' ...
END-EVALUATE.

Scenario #3 used the 'EVALUATE TRUE' test:

EVALUATE TRUE
WHEN variable = '1' DISPLAY 'variable = ONE'
WHEN variable = '2' DISPLAY 'variable = TWO'
WHEN variable = '3' ...
END-EVALUATE.

As you might have guessed, Scenario #2 could be considered the most
efficient because it generated the least amount of assembler
instructions (only 66 in my test pgm). This is because 'variable' only
needs to be prepared for comparison once, rather than once each for each
'IF variable =...' and 'WHEN variable =...' test. The 'nested if' and
'EVALUATE TRUE' scenarios resulted in almost twice as many assembler
instructions (103 and 111, respectively), with the 'EVALUATE TRUE'
seeming slightly less effecient than the nested if.

*will*) vary depending on the number and types of comparisons being
done. The test program I wrote is probably not terribly realistic, but I
believe it does prove a point. Code maintainability slash readability
should also be considered, and nested if's or EVALUATEs may come out the
winner here in different situations.

much easier to control your code vs. the amount of data you will need to

The relationship between "goto-less" programming and "structured"
programming is an old one. It began with a simple observation (Edsgar
Djykstra) "Good programs have fewer GOTO's than bad programs"

What does this mean? It means simply that if you do a good job in
DESIGNING the program well before it is written, you don't need many (or
any) GOTO's; and that if you do a crappy job of designing, or if you code
the program on the fly, then you tend to need a lot of GOTO's. Removing
GOTO's from a bad program gets you a bad program with no GOTO's, not a
good program. You get a good program by designing it well before writing
it. Period.
