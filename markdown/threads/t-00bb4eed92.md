```
I am looking for a way to uniquely identify a load module generated by
an IBM compiler.  The program prolog area is pretty clear in the
assembler listing such that I can easily pull out the WHEN-COMPILED
timestamp.

Can anyone tell me how often this prolog area changes?  Does IBM
guarentee the layout or are we supposed to just pretend it does not
exist?

I would need to support load modules from Cobol for OS/390 v2.2 and
forward.

Any information is greatly appreciated.

Joe Zitzelberger

First of all, load modules are created by the linkage edittor^W^W binder, not
by a compiler. Compilers create object decks.

At any rate, will this load module be in storage or be in a program library?
Do you need to know the state of the first object deck, or all of them?

```
Our programs have the following two lines at the front:

      * LCSVERSION TMPFPF,01A,01/10/92,11.33.12,RICHARD-CHERNIN        
0000001 
      * LCS PROGRAM MAIN MODULE                                        
0000002 

When we start work on a new version the program is copied from production
to development and the version is bumped up and the suffix is set to A. For
each update ( or comiple?) the suffix and date and time are updated. For
each compile the info from the first line is used to update a copy book
which is copied following WORKING-STORAGE SECTION. This is what the filled
in copy book looks like. If you want I can provide what it looks like
before it is updated.

01 core-mark.
    03 filler pic                         pic x(32) value 'working storage
starts here for '.
    03  this-lcs-pgm                  pic x(08) value 'tmpfpf01'.
    03  pgm-field-breakdown redefines this-lcs-pgm.
        05  lcs-pgm-id                 pic x(03).
        05  lcs-pgm-ident             pic x(03).
        05  lcs-pgm-vers              pic x(02).
    03  this-archie-version          pic x(05) value 'none'.
    03  update-number               pic x(05) value '000002'.
    03  date-lcs-updated             pic x(08) value '92/01/10'.
    03  time-lcs-updated             pic x(08)  value '11:33:12'.
    03  date-last-lcs-access        pic x(08) value '92/01/10'.
    03  programmer-lcs-updated pic x(15) value 'richard-chernin'.
    03 librarian-dsn                   pic x(44) value
'dvl.source.libmast'.

Maps and copy books have a different second line and are handled a little
differently.

Parts of this were implemented using Roscoe's RPF language and I think we
may have paid CA to make a modification (or provide an exit?) to Librarian.
These processes should be automated and required otherwise people will
forget to make the required updates. I hope this helps at least to give you
some possible ideas.
