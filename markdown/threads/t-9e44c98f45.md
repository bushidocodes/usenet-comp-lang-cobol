There are always exceptions, such as when the copybook is external and
proprietary, but shop internal copybooks are rarely, if ever, cast in
stone.  But when it does happen, you can still rename your proprietary
copybook, by using your own copybook, and REDEFINE it:

COPY  THEIRCPY
COPY  MYCPY
            REPLACING ==MYLVL01.==
                           BY  ==MYLVL01 REDEFINES THEIR-01.==.

Or if REPLACING isn't available and your copybook's sole reason for
existing is to redefine a specific proprietary copybook, set the 01
level as:  01 MYLVL01 REDEFINES THEIR-01.


And now that you've started, I feel the same way about Level 77s.
They've outlived their usefulness.  In mainframe apps, they were used
to set the field to the next halfword boundary, which helped when
reading Dumps some 15-18yrs ago, before the wide usage of ABEND and
Dump reading tools such as Abend-Aid.

Now, I never have to read Dumps.  Now, I look up the address of the
ABEND, error msg and explanation, value in problem field, which
program, linklib, when compiled, and then scroll through the ABENDed
program's compiled listing, through it's Hexloc of it's MAP, and then
back to the source line that it x-referenced.  Which is 100-fold
better than what it was leafing through an 18" high paper listing of a
Dump, trying to calculate hex offsets.

I can't think of anything that a 77 Level can do, that an 01 Level
can't.

88 Levels should be used in limited quantities and have specific value
naming.  My current pain in my side, is a huge program with well over
2 dozen fields using 88 levels that in turn have anywhere from 3 to 25
(or more) 88 Level value fields.  Trying to keep track of which field
uses which 88 Level can be quite maddening.

Whenever I've used an 88 Level, I always tried to describe the field
value AND the field.  However, most programmers don't do this.

I've been tempted to respond to this subthread several times.

The description herein may be all well and good if the computing universe is
precisely defined as one in which the terms "halfword boundary" and "ABEND"
are applicable to every machine.

Unfortunately, that's not the computing universe as I see it, and it
certainly isn't the particular galaxy in which I find myself..

In my galaxy:

Using level 77 for alphanumeric items does not provide any functional
advantage over an elementary 01.   And there are some cases in which numeric
77-level items enjoy no advantage over elementary 01-level items or, for
that matter, subfields of a record.

But only 77-level numeric (and, for that matter,. 01-level elementary
numeric, which by and large are treated similarly) may be optimized into
on-stack variables.  Where this optimization is appropriate,  access is
direct (avoiding access through descriptors) and in binary form (regardless
of declared usage, thus avoiding decimal-to-binary conversion).  This
provides significant execution-time performance over the unoptimized
equivalents.

On *REAL* machines, anyway!!!     ;-)

The point of all this is that to propose the elimination of a feature
because it doesn't provide any benefit *on the machines one is familiar
with* strikes me as evidentiary of a somewhat provincial attitude if it can
be shown that the feature provides a benefit on other machines.  Unisys A
Series COBOL74 (and for that matter, COBOL(68) which some users continue to
demand) distinguishes carefully between 77/01-elementary and subfields of a
record, and that distinction is of considerable value to users of those
compilers (and those machines).

        -Chuck Stevens

I know I should have kept my mouth shut, but since I didn't here goes:

Traditional Cobol has no local variables. Every data-item is global to
the entire program and can be used by code in every paragraph.

This leads sometimes to nasty and hard-to-find bugs, where one
paragraph uses a certain data-item and also performs other
paragraphs. Due to maintenance, some of these other paragraph
end up also using the that data-item for some other purpose,
with not-so-good results.

You can guard against that to a large extent by defining an 01 level
data block with the 'local' variables for the paragraph, complete with
a suitable prefix to indicate that:

01  customer-validation-vars.
      02  cust-valid-data-item-1     pic x(10).
      02  cust-valid-data-item-1     pic x(1).
....

        perform customer-validation
.....

 customer-validation.
      if cust-valid-data-item-1  ....
            perform zip-code-validation      ....
      .

--------

If you use 77 level data-items you lose the benefit of the
above convention and tend to end up with errorprone
constructions like this;

77  temp1     pic x(10).
77  temp2     pic x(1).

A maintenance programmer in need for a temporary 10-character
variable may look thru the code and say: "gee, there is a handy one
to use (the 77 level temp1)" and use it without being aware of its
use elsewhere or of its "time-to-live". You might counter that no-one
in his right mind would do such a thing, but that is just sticking your
head in the sand.
