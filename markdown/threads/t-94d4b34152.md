You're confusing runtime libraries with platform specific libraries
here. Having ANSI C with platform specifics is roughly equal to having
ANSI COBOL with EXEC SQL... support. Having the ANSI specified
*standard* libraries with ANSI C is roughly equal to having the
necessary file, etc. support that ANSI COBOL specifies.

What comes to clib, ANSI C doesn't standardize that the runtime
environment (say, fopen, for example) be in libraries. You could (but I
doubt you would) write an ANSI C implementation in a single source file
that could be compiled to a single executable. Well, ANSI C requires you
to have a specified set of header files, though, so those should be
written as separate files... :)

Actually, it makes no such requirement.  There is nothing in the
language standard that requires that <stdio.h> (for example) be an
actual, physical file that you can go out and look at with a text
editor.  The only behavior that it requires is that

    #include <stdio.h>

must introduce all of the declarations/macro definitions/etc. that
the standard requires it to.  This being the case, when a C
implementation encounters "#include <stdio.h>", it could simply go
out into its internal symbol tables and insert the appropriate
entries there without actually going out and reading a file.

There are C implementations that actually do this.
