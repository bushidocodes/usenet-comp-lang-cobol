'Twas 22 Apr 1999 03:04:48 GMT, when Ludlow@NewsReader.Com (Ann Marie
Ludlow) illuminated comp.lang.cobol thusly:

Well let me give you a good dose of negative feedback:

This is not sour grapes.  I scored 4.55 and was thereby judged to be "a
Certified Master COBOL II Programmer".  My score is all the more
outstanding when you consider that I have never used nor studied COBOL II.

COBOL II is IBM's name for their implementation of COBOL-85 (ANSI
X3.23-1985, ISO 1989-1985).  If you mean this to be a test of COBOL II
proficiency then you should label it "IBM COBOL II" and include some
questions specific to IBM extensions.  The test as presented should be
labeled a test for COBOL-85.

The user is allowed three minutes (180 seconds) to answer each question.
For most simple multiple choice questions this is ample time, but for
questions where the choices are complex it can take more than that amount
of time to determine the differences.  I timed out on Q.2 and Q.30, and
nearly timed out on Q.21.  Though the instructions specify this three
minute limit, it doesn't tell me this:  If I select one of the radio
buttons and then it times out, do I get credit for my answer?  I suggest a
five minute limit for the questions with lots of code to read.

The following are problems with the specific questions.  I have striven
not to reveal your actual questions and answers.

Q.2:  None of the answers are right.  The PERFORM will continue
indefinitely; the condition will never be met.  (Actually this code should
get a syntax error, because there are two extraneous END-PERFORMs.  It's
possible this is COBOL II specific.  Some compilers allow END-PERFORM on
PERFORM procedure-name, because X3.23-1985 defined this poorly.)

Q.6:  The input data is all upper case, yet the expected output is mixed
case.

Q.13:  The data fields are separated by FILLERs with PIC X(15) yet you
squeeze out all the spaces in the data layout.

Q.15:  This is essentially the same question as Q.4.

Q.16:  Two of the answers are correct.  I chose the one with more detail.

Q.20:  The value of A-COUNT1 cannot be determined, because it was never
initialized.  I assumed it was initialized to zero.

Q.25:  The first definition of the table doesn't match the redefinition.
Each X(30) should be X(20), or you need to add 05 FILLER PIC X(10) to the
end of the table definition.

Q.26:  Your asking for code to produce MM/YY but there's no place for the
/ so you'll only get MMYY.

Q.27:  If the problem is that line X ought to come after line Y, which is
wrong, line X or line Y?

Q.29:  The "correct" answer assumes that the table was initialized to
zeroes, which was not stated.

Q.33:  The task is to change the table definition to do a binary search,
yet none of the answers say put the table in order (and it isn't already).

Q.36:  One of the errors is where indicated, but the other error is in the
given code that is presumably correct.

Q.39:  The "error" in this case is bad style and probably a logic error
(the two go hand in hand) but it is not strictly an error.

Actually, as has been pointed out to the people who offer this test,

1) The name of the IBM product is *NOT* (and never has been) "COBOL II" - It
is "VS COBOL II".

2) This is an OLD (non-strategic) IBM product - that has been replaced by 3
subsequent products (IBM SAA  AD/Cycle COBOL/370 - followed by IBM COBOL for
MVS & VM followed by IBM COBOL for OS/390).    VS COBOL II is an ANSI
High-level '85 compiler - but is NOT a FIPS High-level compiler (because it
doesn't include intrinsic function support - which was added in the later
products).

3) The code in the sample questions was (while still in betas) pointed out
to be INVALID and clearly would NOT compile on any VS COBOL II compiler.
(The example of erroneous END-PERFORM statements for out-of-line PERFORMs
was pointed out about a year ago).  It is pretty clear that the test
examples were never compiled and the answers never verified.

4) Generally, as the people offering this test seem to have NO interest in
offering a correct (or corrected) test - my assumption remains that they are
"trolling" for names and not in determining VS COBOL II - much less current
IBM mainframe COBOL - competency.
