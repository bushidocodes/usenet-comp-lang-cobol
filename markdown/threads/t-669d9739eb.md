A SQLCODE of -10000 is a SQL Server ODBC driver error - if you lookup
sqlcode in the on-line NX docs, you'll see the following info:

-10000
(The first error message returned by the ODBC call SQLError after the
OpenESQL run-time module detected an error)

See your ODBC driver documentation, or other ODBC reference documents for
details of ODBC error messages. SQLSTATE will contain a unique code for each
ODBC error condition.

You'll then to go into the MS SQL Server on-line docs and look for "SQLSTATE
Messages" index entry and use the sqlstate that is returned with the error
to figure out what kind of SQL Server error you are getting.

I ran your question by one of my contacts who is more knowledgeable on SQL
Server issues - his comments:

I think there's some confusion here between transaction isolation and
concurrency. There is no concurrency setting for 'allow dirty read', that's
a transaction isolation setting.
Are they running with MTS? When setting transaction isolation normally you
MUST do a commit or rollback IMMEDIATELY bore the 'set transaction
isolation'. However, with MTS managed transactions you can't use
commit/rollback in OpenESQL, so it hase to be done IMMEDIATELY after
connecting. It's always worth checking SQLCODE after a 'set transaction
isolation' to be sure you got what you wanted.

Now for the busy stuff...

The 'connection busy' happens when a firehose cursor (forward+readonly,
'client cursor' or 'default result set' in MS docs) is open and you attempt
to do something else. If you need to do other things while reading a cursor
with SQL Server, a scrollable cursor (static, keyset, dynamic) will do the
job, but is quite slow. The best choice when you don't need to update via
the cursor is fast forward + readonly. This gets you a 'server cursor'. It's
not as fast as a forward+readonly (great choice of terminology on MS part!),
but it does let you do other things.

A select distinct may well force a sort, so it may result in a non-default
result set, or it may cause the 'feature not implemented', though this is a
bit unlikely.

I'd suggest using putting the concurrency and scroll settings on the
declares to make it more obvious what's happening and use

        exec sql declare x fast forward read only cursor for select ...

if you don't need updates and

        exec sql declare x forward lock cursor for select ...

if you do need updates

You may have problems with lock concurrency on a select distinct, since if
it has done a sort it may be reading result from a temp table rather than
direct from the database.

Hope this helps.

Randy Zimmerman / Micro Focus
