My impression is that the COMPUTE statement (at least in some environments and 
compilers) is OVER-used (i.e. used even when it isn't needed).  If you 
"understand" what each compiler does with intermediate results, then it can be 
pretty powerful.

It is also important to know that with the '89 Standard definition of intrinsic 
functions, you may NOT "MOVE" their results to a field, you MUST use COMPUTE. 
With the 2002 Standard (and as an extension with some - but not all earlier 
compilers) you can use MOVE, but this is another place where COMPUTE is 
valuable.

Personally, I have rarely seen a series of "simple" arithmetic statements, where 
a single (complex) COMPUTE statement could be used.

Simply define a variable (e.g. SEE-NOTE-ABOVE) in Working Storage
as follows:

*  NOTE: The " SEE-NOTE-ABOVE" variable below is used to prevent
*  erroneous truncation of COMPUTEd values that use division or multi-
*  plication.

 01   SEE-NOTE-ABOVE       PIC9(1)V9(17) VALUE ZEROS.

Now add this variable to any COMPUTE expression as follows.

 COMPUTE   F  =  (A / B) * C   +
                 SEE-NOTE-ABOVE

The addition of zero is not going to change the actual value,
but it does nullify the Truncation Menace!

presumption that multiplication is commutative and produce identical object
code for the following:
    COMPUTE F = (A / B) * C.

    COMPUTE F = A / B * C.
    COMPUTE F = C * A / B.

and equivalent object code for
    COMPUTE F = (A * C) / B.
    COMPUTE F = A * C / B.
(which differs only in that the first two operands in the expression are
reversed).

Is there a reason they shouldn't?
