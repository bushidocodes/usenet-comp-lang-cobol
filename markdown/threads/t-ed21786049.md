Dear Clark:
Numbers are stored in different ways on the computer.
In COBOLese these are divided into two groups:
usage display and usage computational.

If you pull out your ASCII or EBCDIC chart(s), you can follow this
discussion more easier...

Lets take a look at the number 3. In ASCII, the code for the character
3 is represented by the Hexidecimal notation 33 in one byte.

In Ebcdic, it is represented by F3. (one byte with one hex digit per
4 bits). This is the display usage as in:
01 MY-DISPLAY-NUMBER PIC 9(01) VALUE 3 USAGE DISPLAY.

Now, the default for cobol numbers is usage display, so we can leave
the usage off...
01 MY-DISPLAY-NUMBER PIC 9(01) VALUE 3.

The C analogy would be a char variable type as in:
char my_display_number = '3';

For COMP, short for computational, the C analogy is an integer.

int comp_nbr = 3;

While in COBOL
01 COMP-NBR PIC S9(4) COMP.

To align this on a double word boundary we use the SYNC phrase:
01 COMP-NBR PIC S9(4) COMP SYNC.

If we look at this in hex, the contents would be 03, or 00000011 in binary.

In comp-3, an IBM Mainframe thing, the Digits are represented in 4 bits, so
the Number 23 could be stored in one byte.

I hope this helps more than it confuses...

COMP-3 on an IBM mainframe and in PC compilers that support IBM
mainframe compatibility is the same as PACKED-DECIMAL. Thus S9(3)
COMP-3 VALUE +123 would be 2 bytes with the hex value 123C, if the
value were -123 then the hex bytes would be 123D and if the field is
unsigned , 9(3) COMP-3 value 123 the 2 bytes would be 123F. This
could be different on other systems since COMP-3 is implementor
defined. COMP SYNC is defined as the computational mode for any
system but depending on the machine that could be PACKED-DECIMAL,
BINARY, or if it is old enough, DISPLAY (granted that this would
probably be only a machine in a museum but I worked with one in the
1960's. On the IBM mainframe (and I would guess most systems) COMP is
the same as USAGE BINARY. SYNC means that the field should be
aligned on a natural boundary so S9(4) COMP sync will align on a
half-word boundary (the address is divisible by 2) and S9(9)
COMP SYNC will align on a full word boundary (address
divisible by 4. Again this is for IBM mainframes and you
would have to check the manual for the compiler on the system you
are using to see what the rules are there.

For the IBM mainframe (S/360 - S/390) the above answer is
accurate and precisely correct. In conclusion:
By using the appropriate Picture caluses, the compiler will not
have to generate extra instructions to perform the conversions
required by the hardware. For maximum efficiency, the Picture
clause should describe the "intention" of the field. COMP (USAGE
BINARY) should be used for internal fixed-binary calculations that
seldom, or never, need to be converted for display. They are ideally
suited for subscripts. COMP-3 is the most efficient format for decimal
calculations where the results are usually moved to DISPLAY fields.

Dear Clark,

COMP-3 and COMP SYNC are two examples of what are referred to as USAGE
clauses associated with a COBOL data item. There are currently about 9
or 10 different "COMP" types that are supported by most COBOL compilers,
although ANSI only identifies "COMP" and "COMP-3", I think. In any case,
they are designed to give the programmer flexibility in how to store data
in memory. The original reason for their development was to conserve
resources (memory) on the computer, a use which has become less important
with the recent availability of cheap hardware. The other main use of
them is for performance: Certain binary types (COMP) are better suited
for certain operations like table indexing than other types, like USAGE
DISPLAY, which is the default USAGE type. How much benefit you get will
depend on the compiler you use.

One problem with these USAGE clauses is that they are not very portable.
It is very often the case that one compiler manufacturer implements
comp-3 in a slightly different fashion from another compiler
manufacturer. This is especially true when it comes to the storage of
signs...

Anyway, to answer your question, COMP-3 is referred to as
"packed-decimal" storage. In this format, two decimal digits are stored
in each byte, with an additional half byte allocated for the sign, even
if the value is unsigned. The actual size of the data item (as opposed
to it's declared size based on the number of 9's in the picture clause)
is based on the number of 9's in the picture clause + 1 for the implied
sign divided by 2 and round fractions up....easy, huh? In practice, this
means that, for example, a PIC 999 field only takes 2 bytes of storage
instead of 3. Internally, the value 123 stored in a pic 999 field would
look like this: 12|3F. (two bytes and the "F" is the value stored for
the sign).

I am not sure what COMP SYNC is really supposed to do since the compiler
I use does not support it (There's that portability issue). I could make
an educated guess though, and state that it probably implies COMP (binary
storage), with a requirement that the data be syncronized on word
boundaries. COBOL has no requirements for syncronization, but when
passing COBOL data to subroutines written in other languages, like "C",
those routines might interface with the hardware/operating system at a
much lower level than COBOL and therefore may require synchronization,
because the hardware/operating system does.

As you talk to other COBOL programmers you will probably find two groups
of people: those that avoid USAGE clauses like the plague and those that
think they are the best thing since sliced bread. Personally, I lean
toward the first group.

Anyone else?
