CoBOL is the original source of many of the ideas that are today subsumed
under the "database" rubric. The ability to copy was added to ComTran so
that you can define data exactly once and copy it in to your program
whenever and wherever you need it.

The best way to create a copybook that would allow to assign a different
01-LEVEL name is to put only the data definition in the copybook!
Although it is not a syntax error, it is a serious misunderstanding of
the copy feature to put an 01-LEVEL in a copybook.

Once you have placed a data definition in a copybook, there are the
obvious ways to use it:

FILE SECTION.

FD IN-FILE
ACCESS MODE IS SEQUENTIAL.

01 IN-DAT.
COPY "DAT.LIB".

WORKING-STORAGE SECTION.

01 WS-DAT.
COPY "DAT.LIB".

LINKAGE SECTION.

01 LS-DAT.
COPY "DAT.LIB".

But where data definition really comes into its own is in co-operation
with that other great CoBOL innovation, the level number. If there's no
01-LEVEL in the copybook, you can copy it in at different levels:

01 IN-REC.

05 IN-HDR.
COPY "HDR14.LIB".

05 IN-DAT.

10 IN-SEG-01.
COPY "DAT59.LIB".

10 IN-SEG-02.
COPY "DAT20.LIB".

01 OT-REC.

05 OT-HDR.
COPY "HDR83.LIB".

05 OT-DAT.

10 OT-SEG-01.
COPY "DAT04.LIB".

10 OT-SEG-02.
COPY "DAT59.LIB".

01 DAT59.
05 WS-DAT59 OCCURS DAT59-MAX INDEXED BY DAT59-NDX.
COPY "DAT59.LIB".


So, if you are fortunate enough to be the one who creates the copybook,
don't put an 01-LEVEL in it. The rule of thumb, for those who haven't
forgotten it, is that 20 should be the lowest LEVEL in a copybook.

However, if you're like most of us, the copybooks are handed to you
already written, and you're not allowed to modify them. Every year I see
more and more copybooks with 01-LEVELs in them. If you're stuck with one
of these, the syntax is:

COPY "copybook"
REPLACING existing-01-level-name
BY desired-01-level-name.

You can see that this defeats the generality of the copy feature because
the existing 01-LEVEL name is now hard-coded into the program.
