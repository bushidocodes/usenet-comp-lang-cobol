```
Hi,

About 2 years ago i realized the power of OO Cobol. Since then i use only
object cobol and  NetExpress. The projects we have developed are based on
windows platform. Recently i 've started the conversion of a GUI application
into a WEB based product. I am not an experienced developer on the fields of
web programming and i am still at an experimental stage. A great difference
i think i have undrestood is that a server side program has no state. This
means that the client (browser) sends a request to the server. The server
side program starts, performs the business logic, sends back the results and
'dies'. The problem is, how can i keep my objects 'alive' between the
invocations of the server program ? Do i have to instantiate them all every
time the client sends a request ? Is there any way to save the instances
(persistency) between invocations ?
```

```
Costas,

Indeed, when you write COBOL CGI-exe's via NetExpress, the state of the application is lost after eacht HTML-out. Merant provides some VERY basic help with theit Server-Side-State-file-Routines (the MF_STATE_* calls). But you have to re-instantiate the objects, indeed. I knew there was some (again VERY basic)) persistancy mechanism provided in earlier OO-COBOL-flavours of Micro Focus (Personal COBOL for instance), bu wether this feature is still present under NetExpres ... don't know.

I have written some web-apps. I encapsulated the ServerSideState routines in a more sophisticated set of sub-routines.

The following is based on what I've read and understood, not on experience:

Perhaps you can switch to ISAPI or NSAPI i.s.o. CGI. In that case your programs are compiled as DLL's, which remain loaded in memory. Objects that have to persist, or better sai, that have to stay alive between interactions, could be in the WORKING-STORAGE of some DLL. (Which is static in NetExpress). But you should dive into this yourself to get it working in written&specified behaviour, not on tricks of me ...

The cObOl Frog
```

```
1. cookies.
2. Hidden variables
```

```
The way I developed it, uses 1/2 + 3: A unique id is assigned on start of a session, this id is in a cookie or hidden field, but the actual state data is NOT passes over the net, but instead held in a file. I prefer cookie above hidden field, being more elegant AND the cookie can persist on the client to allow for later recognition. ("Hello Mr, Frog, nice to see you back ...")

To keep a unique id and to store stat date at the server side is indeed (as suggested higher in this thread) much CICS-like. My CICS and IMS-DC experience did help me top develop my state-preserving mechanisms.

And, it has indeed nothing to do with OO. (Allthough the design and/or implementation could be done either OO or procedural).
```
