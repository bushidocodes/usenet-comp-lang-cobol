I know that it will outrage many readers of this newsgroup, but ANS'74
syntax *is* still quite common in IBM mainframe shops.  This occurs whenever
the older (currently unsupported but still in relatively common use) OS/VS
COBOL compiler is used with the LANGLVL(2) compiler option.  What may
surprise you, is that ANS'68 syntax is STILL in common usage when the
LANGLVL(1) compiler option of OS/VS COBOL is in use.  (There was a thread
about this a few weeks ago in the newsgroup.)

For IBM mainframe shops, the move from OS/VS COBOL to VS COBOL II or
COBOL/370 or COBOL for MVS & VM or COBOL for OS/390 & VM (the *4* compilers
later than the one that your teachers think is common) was a quite painful
migration.  It involved a LOT more than just upgrading a few syntax
features.  Therefore, many shops were (and still are) slow in upgrading.

P.S. Even with the newer compilers, when you compile with the CMPR2 compiler
option, then you are still (basically) limited to ANS'74 syntax.

Just to make one more thing clear.
   I think that there is NO EXCUSE for using a 1985 version of the Micro
Focus product and any of the more recent (currently supported) versions
would allow you to select whether you want ANS'74 or ANS'85 or IBM
compatible syntax or whatever.  You school shouldn't even use cost as an
excuse, because Micro Focus has programs to make it quite affordable for
schools to get and maintain their more current compilers.

If the point of the course is to teach the principles of data
structures, then COBOL '74 is not a good choice of language, and
neither is COBOL '85.

I understand the term "data structures" to include things like linked
lists, stacks, queues, and various species of trees.  It is awkward
to implement these kinds of things in a language which lacks pointers
and dynamic allocation of memory.  You can allocate objects from
static arrays, using array subscripts instead of pointers, and in
some cases I've done exactly that.  However, the resulting machinery
is clumsy enough to obscure the principles which the course is trying
to teach.

In fairness, even COBOL '74 would be better than primitive dialects of 
FORTRAN or BASIC.  Other dialects of COBOL (up to the upcoming standard) 
don't support pointers and dynamic allocation of memory either, except
as vendor extensions.  Even with these extensions the syntax for 
pointers is clumsier and more error-prone than it is in C or Pascal, 
but the extensions at least give you an option you wouldn't otherwise 
have.

For teaching data structures per se I'd rather use something like
C or Pascal, depending on the expected background of the students.
If the point of the course is more specifically to teach COBOL 
implementations of data structures, then COBOL '74 is a defensible
choice because it is approximately a subset of later dialects.  Code
which works in COBOL '74 will work with any later COBOL, possibly with
some minor tweaks.  Still, it would be instructive to use a dialect
where pointers and dynamic memory allocation were available.

Kitty,
  What you are asking about is one of those things that you can ONLY know
for sure with hind-sight.  Certainly, going from DOS/VS COBOL with
LANGLVL(1) on VSE to NOCMPR2 with COBOL for MVS & VM on MVS is doing about 6
migrations - all in one giant step.

The advantage of doing this type of giant-leap migration is that WHEN
EVERYTHING WORKS CORRECTLY, then you only need a single step of conversion
and a single time of "testing" to verify that your applications are all
working as expected.

The disadvantage of this type of migration, is that when ANYTHING GOES
WRONG, then you don't know which of the steps caused the problem.  Was it
the VSE to MVS changes?  Was the LANGLVL(1) to ANS'85 changes that caused
the problems?  Was it something else (JCL? CICS? other?)

Personally, I would be petrified of the specific migration that your shop is
going thru - as it is changing SO MANY things ALL AT ONCE.  On the other
hand, I could imagine (a very EXPENSIVE) set of experts who could be hired
to do the migration in question.

As far as the COBOL itself goes, I would estimate that the "best" LANGLVL(1)
DOS/VS COBOL program and the best NOCMPR2 COBOL for MVS & VM (or even VS
COBOL II) program that were intended to do the same thing would have well
over 80% and maybe as much as  95% identical source code.  COBOL just
hasn't/doesn't change that much.   The problem is that what has/does change
requires an understanding of the changes - and from what you have said
previously, I do NOT think that is true at your shop (or the hired company
that is trying to  do the migration for you).

Does this help? Address your question?
