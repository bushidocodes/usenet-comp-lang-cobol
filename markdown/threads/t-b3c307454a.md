Regulars here will know I am committed to the use of software components
(ActiveX, COM, Java Beans).

It seems that every day I find new components (many of which are shipped as
part of Windows) that I didn't know were available, and are sitting
installed on my system. (The Collaboration Data Objects that enable direct
mailing without a MAPI client are one such example.)

Yesterday I had occasion to produce a view (and optional print) of a  text
file (actually, it is a log file) from within a Windows application.

I see many posts here from people grappling with printers and screen
scrolling from COBOL under Windows.

It puzzles me why anyone would want to re-invent the wheel, but I guess we
all have our reasons... Anyway, I see no point in doing something that is
already carried out by software that exists on my system.

So the decision came down to "Shall I use Notepad, Wordpad, or MS Word 2002
for this...?"

I opted for Notepad. (It is simple, fast, and includes print capabilities.)

Now, unlike MS Word, Notepad is NOT a COM component  so it ISN'T directly
possible to instantiate it and use its Methods and Properties with COM. (I
say, "directly" because at least one very clever programmer has found a way
to make it behave like a COM component, by sending Windows system messages
to it that comply with the COM interface... This is way beyond the scope of
this post, and for us mere mortals a much simpler (and preferably NOT
requiring C++) solution is necessary.)

Olde Tyme MS-DOS and Windows programmers can "kick applications off" by
using a facility called "Shell".  This uses the Windows API to launch
another Windows application. Some programmers use the API themselves,
directly, but the problem with this is that you may need to link support for
the API into your application and mess around with libraries and stuff.
Also, there is a fair chance that the next version of Windows will not
support your API call in exactly the same way as you have used it... Some
people have no problem with dealing with all of this; me, I like the simple
life...

So I need a simple way to use the Shell facility to start Notepad and point
it at my log file.

A quick browse (notice I said "browse", if you do this be careful NOT to
inadvertently update) of the XP system registry reveals that there is an
ActiveX control that will do the shell for me (and for you too, if your
registry contains an entry "Shell.Application" under the HKEY_CLASSES_ROOT
section of the Registry...). You may note in passing what a "goldmine" of
facilities are waiting for the intrepid explorer of things Component, as
identified in your Registry...

Because this IS an ActiveX control, I CAN instantiate it and use it just
like any other COM component. It is pretty simple...

If you have MF or Fujitsu (or any COBOL compiler that supports OO) here's
the code:

For Fujitsu I need a REPOSITORY entry (I'm not sure how MF deals with this,
but those of you have it can look it up).

ENVIRONMENT DIVISION.
REPOSITORY.
       Class WinShell as '*COM'.

 ENVIRONMENT     DIVISION.
 DATA            DIVISION.
 WORKING-STORAGE SECTION.
 01  ShellObj OBJECT REFERENCE WinShell.
 01  ShellProgID     pic x(30) value 'Shell.Application'.
 01  ShellReturn     pic s9(4) comp-5.
 01  parm1           pic x(15) value 'Notepad.exe' & x'00'.
 01  parm2           pic x(15) value 'WBRAgents.log' & x'00'.
 01  parm3           pic x value low-values.
 01  parm4           pic x(4) value 'open'.
 01  parm5           pic x value '1'.
 PROCEDURE       DIVISION.
     invoke WinShell "CREATE-OBJECT" using ShellProgID
            returning ShellObj
     end-invoke
     invoke ShellObj "ShellExecute"
            using Parm1 *> Command to run..
                  Parm2 *> Parms to the Command...
                  Parm3 *> NULL
                  Parm4 *> 'open'
                  Parm5 *> '1'
        returning ShellReturn
     end-invoke
     set ShellObj to NULL

A few quick comments by way of explanation:

1. The "WinShell" entry in the Repository is just a name I made up. You
could use nearly anything... It simply says that I am going to use a Class
called "WinShell" that will inherit all the behaviours and attributes
(Methods and Properties) of the COM Class

2. The "ShellObj" entry defines an Object Reference of type "WinShell".
When the Class is instantiated by the "CREATE-OBJECT" method of COM, a
"pointer" to the Object instance will be placed here.

3. The "ShellProgID" entry tells the CREATE-OBJECT method of COM what system
Registry entry has the details on how to construct the instance.

4. "ShellReturn" accommodates a return code passed back from the Shell
Object. (I don't use it in the code because if the thing fails, it is pretty
obvious to the User (he won't see a document), but I did use it the first
time I ran this to see what it returned. (it was zero...success.)

5. Although some of the parameters would lend themselves to being invoked BY
VALUE, the COM class requires BY REFERENCE for everything, so I had to
prepare the parameters in WORKING-STORAGE. No big deal, but worth
remembering...

I needed to check out the Web to obtain a few details about the Shell
interface but the total time to implement this solution was not more than 20
minutes.

The above code is called when the user clicks on a button to "View/Update
Log File".

It worked the very first time I ran it, with no debugging whatsoever. This
is one of the nice things about components; they do what they do.

It is pretty obvious that you could use this to "kick off" any windows based
application, just by changing the parameter values.
(Wanna provide Help? Change the first two parameters to "WinHelp" and the
name of your Help file...then place the code on a "PFKey 1 pressed"
event...)

While this doesn't have the same detailed level of control that
instantiating MS Word and using its Object Model has, it is a very suitable
solution for quick print/update/view requirements, and it is a long way from
WRITE ...AFTER... ADVANCING...

I hope this is of use, and if anybody actually tries it, and it works for
them, please drop me a line.

Pete.

> Is there any systematic way to locate all such components on a system?

Not that I know of.

You can easily find "controls" because they are often designated as .OCX.

But .DLLs can also be ActiveX controls if they are registered and have the
required interface.

The system Registry of Root Classes is a good hunting ground as most of
these entries will be COM components. But you cannot always determine what
the actual.DLL or .OCX is easily, from the ProgID as this is a kind of
"alias". (You need to search the registry carefully and it takes a bit of
practice.)

What I usually do when I have something I suspect is an ActiveX or COM
control is import it into PowerCOBOL using the "Custom Control" menu.
Unfortunately the class and Object Browsers in PowerCOBOL will only let you
look at stuff that is in a Repository. (I guess this is not unreasonable as
they couldn't cater for every possible Class library.)

So the next step is to try and drop the imported Custom Control onto a form.

If it is COM Compliant, this will display all its properties.

To find out its methods you need to reference it in an event procedure.
Write an "invoke" that references the name of the alleged control. Highlight
the name and right click the mouse. It will give a list of the all the
public Methods of the control.

That is about the extent of my knowledge so far. Iam still working on
building my understanding of these components and identifying all the free
ones that came with Windows.

Another good source is the Web, of course. You can often find articles for
VB and C++ programmers that discuss things component in great detail. It is
a bit of a job, but after a while you can gain enough skill to "translate"
the sample code into working OO COBOL.

The same goes for MSDN. The Object Models of most MicroSoft software is
published there, along with the Methods and Properties of each compnent. It
is a bit like going into a library to look something up and coming out
several hours later with a bunch of really interesting things, having
totally forgotten why you went in in the first place <G>. The main problem
with MSDN is that it often tells you much more than you really want to
know... I am learnng to cut through it now and get to the essential stuff.
It is still annoying to have to translate all the sample code into COBOL ,
but I am getting much beter at doing this with practice.

> Are

Yes and no. It depends what you mean by "key". As for unlocking the
behaviours and attributes, see above.

I find this a quite enjoyable process; it's a bit like gold mining...every
so often you strike a nugget.

instead of being customized for FUJITSU Cobol only.

After all, Power Cobol is NOT a cobol compiler, as most people believe.  It
is source code generator.  That is, it builds a data base of objects needed
for the a project, out of pre-defined objects.  It then generates FUJITSU
Cobol that is compiled using their standard Cobol Compiler.  The hook is
that it is then tied to their sepcifiv run time, which is powercobol
specific and not oop cobol specific.

There is no reason a similar system cannot be built that is generic.  In
other words, it uses it's data base, plus its runtime, to build it's source
code.  Since the source code is generic, it in turn could be re-compiled
under a different
compiler for a differnt runtime.  The system could therefore be bootstrapped
accross all Cobol systems.

Control is a horrible word to use.  So is ActiveX and COM.  Let's just
get down to brass tacks shall we?

All of these things are really and truly automation servers.  Some you
"may" use in your own applications given the properly MS license - and
some you "may not".  Some are "licensed" (want more info on that ask
me), some are not.  Some are In Process servers - some are out of
process servers.  Some are in process servers that require a container
- some do not.  The differences are what them them ActiveX controls vs
COM, vs COM+ etc... Bottom line - they rae all accessed via the OLE2
interface.

Flexus has recently enhanced their ActiveX support to handle more than
just UI controls - they now support in process and out of process,
insertable and non insertable ActiveX controls, COM servers,
Automation servers etc.

PowerCOBOL and Generic Fujitsu COBOL support these as well - I do not
mean to imply that they do not.

MS didn't do anyone any favors playing "Naming Roulette" with any of
this.
