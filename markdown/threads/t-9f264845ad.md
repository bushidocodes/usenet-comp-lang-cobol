```
OK, I am writing a DLL to convert COBOL datatypes to IEEE types. (will be
freeware). The DLL will be callable from BASIC or C/C++ (or COBOL, for that
matter; but that is kind of a weird thing to do, considering your COBOL
program already uses COBOL datatypes).

I would like some comments on two options for what the user passes:

A. Users provides the specs on the PICTURE IS clause, thus:

Given: PICTURE IS S9(04) USAGE COMP-3.

Parms.DataType = Comp3     << comp3 is a constant; other constants for other
datatypes
Parms.Digits      =  4
Parms.Decimals   = 0
Parms.Signed      = -1          << not zero = true
Parms.SignSeparate = 0     << immaterial for type COMP, but needed for
DISPLAY
Parms.SignLeading  = 0     <<  ditto
Parms.DataLength   = 2     << Length in bytes of the data coming in. It
better be right
Parms.DataBuff      =    (two bytes of data)
Parms.DataPtr       =   Address of databuffer

Thenyou call the DLL thusly:
BASIC:  Status = COB2IEEE (Parms)
C/C++:  Status = COB2IEE (&Parms)        << something like that. Pass the
address of the Parms structure

B. Alternately, you can give me the literal for the PICTURE IS :
Parms.Picture = "S9(04) [USAGE IS]] COMP[UTATIONAL]"
Parms.DataPtr       =   Address of databuffer (you allocate)

You call the DLL, and I calculate the datatype, digits, sign, etc about the
PICTURE IS from this (I already have this parsing code). This second method
is a lot easier for the user BUT IT FAILS IF USING COMP or PACKED-DECIMAL or
SIGN IS .. **at the group level**. (Which personally I never liked, but it
exists).

Comments?

2. I am thinking about returning the value in one of two forms:

A.   Mantissa  =  32 bit signed integer (LONG), more or less normalized.
      Exponent  = + values, multiply by 10 this many times; - values, divide
by 10 this many times.
B.  Fill a buffer in Parms with the data (i.e, a long integer is 4 bytes, a
double-precision 8 bytes, etc). The user provides a parameter telling what
kind of data he wants out. (I return error code if out of range, wring
number of characters in the data buffer, stuff like that).

Hell, I could return both if desireable.

Comments on this?

3. Are there any "C" newsgroups I should query re this?

Michael Mattias
Tal Systems
Racine WI USA
michael.mattias@gte.net
```

```
Am I right when I guess that you do not want to support Cobol with your DLL?
The reason I ask is that I see there will be a challenge to use your
functions from Cobol (at least Acucobol).
I would for instance have big problems providing you with an address of a
databuffer (Params.DataPtr).

Another issue is that I expect this would be a batch, converting a lot of
numbers, how about having one function to configure the engine:

CFGCOB2IEEE(DataType, Decimals, Signed, SignSeparate);

Where all of these could be Longint.

Then a single function for converting:

CVTCOB2IEEE(Number);

If Number was a LPTSTR pointer, you wouldnt need to worry about getting the
length of the data, as it is zero terminated. The great score about this, is
that you could use the very same data item to return the converted value. By
returning the converted value as a string, you give back the opportunity to
the developer to decide what format to store it in. The great benefit with
this, is that the developer does not need to worry about which data type to
use. There is quite a number of number formats out there, which tend to vary
depending on the host environment. This could definitely be a clue now that
we have 64bits NT coming up in just a few years. I remember all the hassle
with determining the size of an integer in Borland Pascal in its first
releases for Windows.
```
