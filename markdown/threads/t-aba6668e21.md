```
Greetings,
  We have an IBM mainframe with compiler PL/I for MVS & VM Ver 1 Rel 1
Mod 1. We have the situation below we are unable to resolve and could
use any ideas or clues. This is a batch non-multi tasking process.

Main module A (COBOL) dynamically calls PL/1 modules D and E. Module E
dynamically calls module F (assembler). This flow runs successfully.

A change was made that now main module A dynamically calls modules B
and C which are both COBOL. Module B dynamically calls PL/1 module D
from above and module C dynamically calls module E from above. When
this flow is run we get the following abend on the first call attempt
in module E to module F.

IBM0576S ONCODE=3915  An attempt to use a CALL statement with the
TASK, EVENT, or PRIORITY option was found in a non-tasking
environment.

When the call to module D is removed from module B, this flow runs
fine. It's as if the call to this first PL/1 module (D) is influencing
the call structure in module E.

Again, any ideas, hints, or clues are appreciated.
```

```
You didn't say, but I presume all the COBOL modules (A, B, and C) are all
compiled with some COBOL compiler that is LE-enabled (i.e.: COBOL for MVS & VM
or COBOL for OS/390 & VM).

In the "Writing InterLanguage Communication Programs" there are two points that
may be of interest:

  * If a COBOL program is to call a PL/I program dynamically, the outermost
    proc in the PL/I  program should include FETCHABLE in the OPTIONS list

    [I know that sometimes you can get away without this, but
     it is not supported]

  * When a load module is dynamically called, control passes to an entry point
    identified as FETCHABLE unless the load module also contains a MAIN
    procedure [presumably not the case here], in which case control passes
    to the MAIN procedure

you should also link the dynamically called PL/I routine to have its entry
point to be the same as the module name.

Hope this helps.
```
