Are you publishing to UNIX or NT?

On NT you must have a cgi-bin folder for your web application
and all associated MF files must be included (e.g. yourapp.exe,
cblrtss.dll, cblvios.dll, etc.). The folder must have the proper
IIS permission (e.g. Execute). Permissions seems to be the
biggest problem with setting up a web application.

To prove that it really does work, check out
http://www.oms2000.com/oms2000udpates and try to logon. The
logon cgi script opens an ISAM file to verify the user has
proper authorization. If the Login ID and Password don't agree
an error HTM file is shown.

Rick,

If you deploy on some "anonymous" server of some "ISP", and you got that ISP so far that he allows YOU to run cgi's on THEIR server, you are veeeery lucky. People will want to know that friendly ISP!

When running under ISAPI (Microsoft, Internet Information Server(IIS)) of when runnning under NSAPI (Netscape, forgot their webserber's name) than you have to deliver DLL's that will be put in a dedicated directory on the webserver machine. The webserver machine itself can f.e. be running Windows NT, the webserver software runs under(?)/on(?) the OS. When running as CGI, than you deliver EXE's that will be put in another dedicated directory on the webserver machine, a dir which has the web-share "cgi-bin" applied to it.

Typical setup:

Intel Hardware --> Windows NT server --> IIS      --> ISAPI --> your prog as DLL
or
Intel Hardware --> Windows NT server --> Netscape --> NSAPI --> your prog as DLL
or
Intel Hardware --> Windows NT server --> IIS      --> your prog as EXE
or
Intel Hardware --> Windows NT server --> Netscape --> your prog as EXE

In all these cases (other hardware and/or operating systens are possible), when the executables (DLL or EXE) are derived from COBOL via a NetExpress compiler, there has a so-called COBOL-runtime system to be installed on the webserver machine. This runtime systen (rts) contains the system-modules that f.e. represent the action to read/write files, used builtin functions etc. etc. When you decide not to install the rts, you will have to link all the rts-modules with the EXE's / DLL's, making those boys very big.

<quote from NE docs>
You want to make a standalone executable (one that doesn't need the shared run-time system to work). This isn't recommended, apart from those cases where you are only going to have one or two COBOL CGI programs on a server, and don't want the overhead of installing a COBOL run-time system. Link the program as described in Building a Static Run-time System CGI Application.
</quote>


So install the RTS. Two possibilities:

1) Install them in the same directory as the DLL's/ EXE's. That's nice for a quick solution and possible nice if you work with the aforementioned ISP, but it is NOT smart when installing on a server that has to be managed by yourself/your company: When 2 or more web sites are set-up on the same machine, the rts gets duplicated.

Also rts-upgrades are safer and simplier when the rts is in its own dir.

2) Install them in their own directory.

See the NetExpress Help, topic: "Development Environment". Look especially to the topics "Shipping files with your Application" and "Running Applications".

There is much to read. Documentation of NetExpress around this topic (deployment) is weakly organized by Merant/Micro Focus.

Good Luck

Thanks for your input.

It just happens the ISP we use is upstairs and have worked very
closely with them in setting web sites with cgi. They have a
simple structure with public as the main folder where all webs
are configured on their NT Server. Under public we setup the
folder for the web site and when we published with
FrontPage2000, we created a cgi-bin folder. Then we had the ISP
make sure the cgi-bin folder had the proper permission.

public
--WebSite
---_private
..
---cgi-bin
..
--NextWebSite
..

This has served us well and as long as you can get your ISP to
cooperate, you should have no problems running the COBOL cgi on
any NT Server (haven't had experience with UNIX).
