I use the architecture you describe with Fujitsu PowerCOBOL and COBOL 97
(Version 5).

Different modules handle different cursors. (This is not essential and there
is no reason why a single module can't define and handle multiple cursors; I
just happen to do it this way...)

It is important to ensure that the cursor name is different from EVERY OTHER
name (NOT just other cursor names) in the module, and that the same cursor
name is not used in more than one module.

Given the observance of these simple rules, I have never experienced the
problem you describe.

Pete.

Hello,
Besides what Peter said, the "out of sequence statement" error, comes
when you open a cursor, and then try to perform some other operation
on the database which doesn't involve the said cursor. I don't kown if
this a standard RDB behaviour or just an MS SQLServer characteristic,
but between an "open cursor" and a "close cursor" the only things you
are allowed to do are operations on the opened cursor (fetch, update
where current, delete where current, etc.).
This aplies to one connection only environments. If you define 2 or
more connections within your running unit, then you're free to jump
between them without the above constraints. But be aware that each
opened connection eats up a seat license.

Paulo
