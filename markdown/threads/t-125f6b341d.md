Bryan,

Did you mean to write ISAM (Indexed Sequential Access Mode) COBOL files ? If
that's the case, then you really do need the COBOL source of the indexed
file format. You referred to the 'header record' - Micro Focus ? All this
has is the 'housekeeping' that the run-time needs; it doesn't spell out the
record layout. .

The long way - if you have programs which print from this particular file -
is to do a print report and try and figure it out - but if numeric fields
are held as Comp-3 or other forms of binary, then you really do have a
problem.

If you can establish the file format, output a Sequential
Ascii-Delimited/CSV file, all fields separated by commas and additionally
text fields surrounded by quotes. Then Delphi will be able to read it in.

Jimmy

If you mean ISAM, it is not possible to interpret the structure from either
the header record OR the COBOL source. Each vendor of COBOL has their own
proprietary ISAM format. You will need:

1. The original COBOL compiler and the original COBOL source to write a
COBOL program to reformat the existing file to a flat or text file; or

2. PERHAPS some utility that came with the COBOL compiler package that can
extract the data within the ISAM file; or

3. PERHAPS a third-party utility can do this if you know the COBOL compiler,
version, and platform on which the offending file was created.

Think of an ISAM file as a linked-list on steroids. Imbedded within the file
are data, linkage structures, and positioning pointers. It is a huge chore
to separate the data from the control information.
