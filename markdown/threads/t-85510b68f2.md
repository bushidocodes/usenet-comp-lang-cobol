I think you would have to reference the memory pool using the LINKAGE
SECTION instead of the WORKING-STORAGE SECTION.  Here is a possible
example:

LINKAGE SECTION.

01  POOLADR-POINTER            USAGE IS POINTER.

01  ZZPOOL-MEMORY.
    02 FELD-1       PIC X(1000).
    02 FELD-2       PIC X(1000).

And then somewhere in your procedure division you would have to do
this:

CALL "ZZPOOL" USING POOLADR-POINTER.
SET ADDRESS OF ZZPOOL-MEMORY TO POOLADR-POINTER.
MOVE "ABCDEFGHIJK" TO ZZPOOL-MEMORY.

This assumes that your ZZPOOL assembler program does a GETMAIN or
equivalent to acquire the memory pool, and then returns the address of
this memory pool to the COBOL calling program.  The calling program
would then establish addressability to the memory pool (using the SET
ADDRESS statement).  Only then would the COBOL calling program be able
to read or write to the memory pool.

I hope that helps!

hello Detlef,
may be it is possible for you to switch to compiler COBOL2000 version
V1.0B (or greater): it supports data pointers; the code Arnold has
proposed would need only two little changes: POOLADR-POINTER should be
defined in WORKING-STORAGE section and ZZPOOL-MEMORY should have the
BASED clause (and your assembler program then needs the 'L R1,0(R1)'
proposed by Adreas). Incompatibilities between COBOL85 V2.2 and
COBOL2000 are none to speak of and concern mostly compiler options and
and increased set of reserved words (for details refer to
'Freigabemitteilung' for COBOL85 V2.3A and COBOL2000).
if you need to stay with COBOL85, an other possibility for your COBOL
program to access the reqested memory would be to call the COBOL program
*as a subprogram* after requesting the memeory by the caller and handing
over the address of the memory as a parameter, that is defined as
ZPOOL-MEMORY in the LINKAGE section: you could thus avoid data pointers
in COBOL

hope that helps
Karl
