Before I say anything else, you should be aware that I am a COBOL
programmer for MasterCard International, and I have worked with
ISO-8583 Bit-Mapped messages since about 1990-1991.  

MasterCard began processing ISO-8583 messages natively in 1988, if I
recall correctly, but it wasn't until 1992 that all the older
fixed-length INAS (InterBank National Authorization System) message
formats were made obsolete.  INET (International Network for
Electronic Transfers) still uses fixed-length message formats to this
day, although IPM is coming later this year.  

I am not fully conversent on the differences between ISO-8583 and IPM,
but I believe IPM makes greater use of subelements (subtags) than
ISO-8583.  As I recall, subelements began to creep into ISO-8583
message data elements in the mid-1990's.  MasterCard publishes a
hardcopy manual titled "Customer Interface Specifications" that
describes the ISO-8583 message format.  It's a very big book.  As far
as I know, this book should be available to banks and other financial
institutions that are customers of MasterCard.  They have a department
of Manuals & Publications that may sell or otherwise distribute this
book.  I have a copy because I need to reference it in my work.

WOB (Bill) is quite correct.  Writing an ISO-8583 (or IPM) message
parser or formatter utility is a fairly difficult piece of work.  I've
seen several of these beasts, for batch & CICS, generally as separate
utilities for parsing (breaking out the fields) and formatting
(compressing fields into one message).  Some were written in IBM S/370
Assembler, and some were written in COBOL.  MasterCard may have had
parser/formatter utilities in S/370 assembler for batch that were
distributed to member banks, but I'm not sure if they are still
available.  
I'm aware of parsers & formatters written in C for Unix
implementations, but I've never seen the code for those.  I've never
written one of these beasts, although I have cloned or enhanced a
couple, and I've certainly used them extensively.

Basically the Data Elements are numbered and have some defined
formats.  Some data elements are fixed length, such as DE-014 (Card
Expiration Date), and some are variable-length with a two-digit or
three-digit length prefix.  The initial bitmap tells you how many
bitmaps are present, and the bitmaps tell you which data elements are
present in the compressed message.  Subelements are contained within
defined data elements, and normally consist of a two-digit subelement
identifier (subtag), a two-digit length, and a data value.  The
disadvantage to subelements is that they are another level down.  You
have to parse out the data element before you can parse its
subelements.

There may be commercially available message parsers and formatters for
both ISO-8583 and IPM message formats, although I have no information
on vendors.  But, hey, if you can get a free one from WOB, more power
to you!

It makes no sense re-inventing the wheel, although performance
efficiency is definitely a consideration in parsing/formatting
messages, since the volume of records can be quite large.

STANDARD DISCLAIMER:  I am NOT an official corporate spokesperson for
my benevolent employer, and anything I said was my personal opinion. 
If you want an official corporate spokesperson, I think I know two or
three.

Gents,

I've been writing parser/formatters for ISO-8583 Authorisation
networks since 1985. Visa, MasterCard, EuroPay, JCB, AMEX and most
recently, the Australian Standard AS2805. They've been written in
Assembler, COBOL and PL/I. Of course, Assembler is the fastest method,
but you can write some very efficient re-entrant code in COBOL and
PL/I. The benefit of not writing in Assembler is maintenence. They
ain't making new Assembler people anymore. Reference modification and
the SUBSTR verb in PL/I make it simple and finding people to maintain
it is that much easier. The only catch is dealing with different data
types, of which, AS2805 is the KING. Visa uses EBCDIC, BCD and HEX.
MasterCard/EuroPay are all EBCDIC. JCB is a potpouri of Visa and
MasterCard. AMEX is EBCDIC but uses the 1993 standard of ISO-8583 (as
opposed to the 1987 standard) and AS2805 uses ASCII Display, EBCDIC,
BCD and HEX. The customer wanted this Parser/Formatter written in
Assembler, but I urged them not to. They got their way.

Page ===> http://www.aboutlegacycoding.com/ I have placed four
sub-programs that can be used with the ISO-8583 standard. They are:
BIT2BYTE, BYTE2BIT,CHAR2HEX and HEX2CHAR. I've used these sub-programs
in some of the COBOL Parser/Formatter sub-systems.

I suspect that James could use them in his IPM endevor.

FWIW, all of this work was done on a Mainframe, either in an MVS or
VSE environment.

VisaNet also uses ISO-8583 for their messages.  The main difference between Visa
and MasterCard is that MasterCard uses all EBCDIC (or ASCII??) for their
messages (in other words, a numeric value might be x'F0F0F0F0F5F9F9F5' for
00005995 (implied decimal point)) while Visa uses what they call a 'numeric
packed' field, ie x'00005995'.  This makes Visa's a bit harder, at least for
mainframes, since there's no natural COBOL pic clause for that data type.

At my shop we communicate with both Visa and MasterCard (Cirrus).  We have a
vendor (CV Systems Corp - www.cvsystems.com) for our card transaction processing
system, and they wrote the 'formatter' (in COBOL) for both types of messages.
While it is somewhat confusing to learn at first, once you have a basic
understanding it's not really as difficult as it seems.  I'd post more detail
but since we didn't write the code I'm not sure it would be 'kosher' to describe
how they parse the messages.  But basically you'd have little subroutines for
each possible 'data type'.  One for fixed length alphanumeric, one for variable
length alphabetic, one for fixed length numeric, etc.  You then have a table for
each field that says which data type that field is, and you then perform the
proper subroutine.  (The hardest one, of course, is when the data type itself
has a bitmap followed by subfields.  Yikes!)

I believe MasterCard uses all EBCDIC, even on machines where the
native codeset is ASCII.  Unlike Visanet ISO-8583, a MasterCard
ISO-8583 message only contains character data outside the bitmaps (no
binary, packed-decimal, or BCD fields).  Even credit card magnetic
stripe tracks are converted to EBCDIC, which include field-separators
that translate to relatively uncommon EBCDIC characters.

>

There are several parsing strategies.  The parse I use most frequently
has a COBOL copybook of nearly 1000 lines to describe the uncompressed
message.  I know of one IPM parser that simply generates a table with
one entry per Data Element (including all subelements) that contains
the offset into the compressed message and al length.  Then the data
is actually referenced by a table lookup, and the offset and length
are populated into a reference-modified MOVE statement.  

DocDwarf might be able to convert that to IKFCBLOO, but otherwise that
strategy requires an ANSI-85 compliant COBOL compiler.
