```
I have developed a program which will hang my CICS session when it runs.

The problem occurs on the Open of a cursor. Here is the senario.

There are 2 programs involved.

Program A calls Program B passing data to it using CICS Link.

Program B uses the input for various purposes. Eventually it creates a
Cursor upon Table Z. It then opens that cursor. At this point it is working
correctly, and proceeds to the next step. After some more processing Program
B inserts a new row into Table Z, and ends, returning control back to
Program A.

Program A immediately calls Program B again, and as before passes data using
CICS Link.

Program B proceeds as described above until it tries to open the cursor on
Table Z where it hangs.

If I put a commit between the 2 calls to Program B, the hang does not occur
and all works fine. Unfortunately, Program A is not allowed to perform the
commit until all iterations of B are successful.

I have compiled and bound my programs using an isolation level of 'UR'.

I have developed these programs on Windows NT v4 sp5, using VA Cobol 2.2,
DB2 v5.2, VA CICS EAD v3.1.

Any assistance would be appreciated.

James
```

```
Seems to me like the cursor in program B is still open and the next time
you open it you should get a DB2 error trying to open a cursor that is
already open.  Remember you said you used LINK to get to program B so it
looks like one transaction with no syncpoint (if you connected via XCTL
you would automatically get a syncpoint and therefore a commit when you
use RETURN transid to get back).  Try just FETCHing the next row from
the cursor in B after the first time through.

Joe
```
