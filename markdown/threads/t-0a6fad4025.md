COMP-2 is implementor specific; that is, there is no "COBOL" standard for
it.

What it *generally* means is an 18-significant-digit (mantissa) number like
an IEEE double-precision float. ( or 'C' "double").

BUt... if you are trying to read data created with another compiler, you're
going
to have to figure out what that vendor used for COMP-2.

And one more thing: if the data come from an IBM mainframe, there are two
possible formats: IEEE or HEX (aka NATIVE); this option is either set at IPL
(system startup) or may have been overridden when the program was compiled.

Most important thing for you to do now is cast aspersions on the validity of
the birth of the imbecile who stored data on disk in COMP-2 format when the
possiblity of a port existed.

More info: read http://www.flexus.com/ebd2asc.html

It doesn't surprise me at all that TinyCOBOL does not *yet* support any
floating-point data-type (such as COMP-1 or COMP-2 - that are common in IBM
and IBM compatible compilers) - because the EXISTING ANSI COBOL Standard
doesn't support it.

HOWEVER, the draft of the next COBOL Standard *does* include support for
some NEW
    USAGE FLOAT-xxxx
data-types.

I suggest that you contact the TinyCOBOL people and ask them if they have
any "intention" to support this sooner than later.

ALTERNATIVELY, you can change your C program (if you have control over it)
to support a binary or decimal type field.  These are supported in ALL
ANSI/ISO conforming COBOL compilers.
