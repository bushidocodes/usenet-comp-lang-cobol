```
If you use Fujitsu's PowerForm, it has barcodes built in.

Otherwise, there seem to be two choices:

1. You can use a barcode font (almost any symbology is available free
if you look hard enough), or
2. You can develop a career sending bits representing a single
symbology of a single size to a specific printer.

You may think I'm prejudiced in favor of fonts: You're right. With a
font, you don't have to worry about scaling or printer technology
(laser, ink-jet, dot-matrix).

We found it so much trouble to fuss with hand-crafted barcodes, that
we opted to buy BARCODE PRINTERS that do all that stuff in firmware.
Now, with Windows handling all the code for us, we can configure our
software to print barcodes on almost any printer (usually sheet-fed,
but we do have some users who print on dot-matrix - they need PostNet
barcodes on mailing labels).
```

```
Alister Munro wrote:
> 
> How do you do a barcode with PCL ? Just starting to look into this :)

I do it by drawing lots of tall rectangles in black.  

I based this on a public domain C program that I found on the net and
recoded it to Cobol, changing it so that I could call the routine and
have it emit the appropriate PCL code for various sizes.  It does
code-39.

Basically there is a table with an item for each valid character
containing the pattern in terms of 9 '1's and '0's.  To print the
barcode first grab the position on the page then process each character
in the code and emit rectangles wherever there is a '1' in the pattern,
stepping along by the bar width.

This does, however generate quite a bit of text and may affect whatever
else you want on the line.  I store away the barcode requests as I print
the page, remembering the position, size and code, into a table and then
output from this this table at the end of the page.
```

```
These are the PCL codes I use :)

Pound Sign x'9C'
Hash Sign x'23'
Carriage Return x'0D'
Reset x'1B45'
Landscape x'1B266C314F'
Portrait x'1B266C304F'
Fixed Spacing x'1B28733050'
Courier x'1B28733430393954'
CPI Selection x'1B283132551B28733070' + STR + x'68307330623430393954'
    Replace STR with cpi desired (eg. "10", "12", "17.6")
LPI Selection x'1B266C' + STR + x'44'
    Replace STR with lpi desired (eg. "6", "8", "10")
Bold On x'1B28733342'
Bold Off x'1B28732D3342'
Underline On x'1B26643044'
Underline Off x'1B266440'

All above codes are in hex except for STR which is just an ascii string.

Its a good idea to setup a file containing printer codes and parse print
lines
replacing literals with the actual printer codes. My reports ask for Device.
Printer, Paper, & Margin Size's. Other than standard forms I can the scale
the printed report on a best fit basis for the printer/paper combo. I use a
copy-book containing literals such as 'op-fx-bold-on', 'op-fx-bold-off',
etc.

If it helps, send me an email and ill send you my code for parsing print
lines.
```

```
If you are working with Windows then you need to change the printer
settings so it prints in "RAW" mode.
If you don't do this then all "esc" sequences will be ignored.

The best way to see if a sequence is correct is to create a text file
with "edit" (not wordpad or notepad), and printing it in a DOS box.
Such file would have some sequences to set up different font sizes and
then print a line of text for each one.

Once this is working then the COBOL program should also work
(if everything is correctly configured in the COBOL side).
```
