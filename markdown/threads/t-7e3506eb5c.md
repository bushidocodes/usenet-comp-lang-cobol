 //MYJOB JOB ......
 //S1 EXEC PGM=IEBGENER
 //SYSPRINT DD SYSOUT=*
 //SYSIN DD DUMMY
 //DD1    DD DISP=(NEW,CATLG),DSN=SOME.CONVENIE.NTFILE,
 //          UNIT=SYSDA,SPACE=(TRK,(0))
 //SYSUT1 DD DATA,DLM='><'
 //MYJOB JOB ....  notice identical job name, but that isn't enough
 //S1 EXEC PGM=MVSCMD,PARM='OCITOPE,030'
 //DD1 DD DISP=(OLD,DELETE),DSN=SOME.CONVENIE.NTFILE  notice DISP=OLD
 //SYSUT2 DD SYSOUT=(A,INTRDR)
 ><
 //S2  EXEC PGM=disable_transaction (sounds familiar)
 //DD1 DD  DISP=OLD,DSN=*.S1.DD1
 //S3  EXEC whatsoever
 //DD1 DD  DISP=OLD,DSN=*.S1.DD1
 //

S1 would submit a job with the same jobname.  In most, but not all,
systems this second job would now wait

To make the whole thing bullet-proof, S1 allocates a file and gets an
exclusive lock.  All other steps in the job get the same exclusive
lock to the same file.  Even if the job submitted by s1 gets started,
it will be "waiting for datasets" until the original job ends

If MYJOB fails for any reason, no matter, the submitted job will run
and enable the transaction again

Note:  The jobnames don't even have to be identical in this construct,
but they may be identical
