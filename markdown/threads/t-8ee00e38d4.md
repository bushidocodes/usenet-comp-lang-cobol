```
Using COBOL-2 in an MVS environment, I've recently had the annoying
experience of losing 4 hours of work due to a misspelled DDname on my
output JCL.

Selects which cannot resolve their DDnames in the JCL used to trigger
abends with the JES message "DDNAME NOT FOUND". Now the system simply
and wordlessly routes such output to a temporary buffer which is then
deleted at end of jobstep.

Can anybody tell me in what environment or situation this might
possibly be called a "feature" ?

Thanks.
```

```
This is the documented way output is treated starting with COBOL II R3. It
was IBM's way of making COBOL "cross-platform." (Thing is, in CMS you can
still recover the output; in MVS it's toast).

Many MVS customers were peeved and demanded that the ABEND U1035 be issued
for a missing or misspelled DD as in R2. This can be done using the CMPR2
compile option. This may not be feasible in your shop if your apps must be
NOCMPR2. For COBOL II R3 and R4, IBM supplied a user mod (not a PTF) to
restore the CMPR2 behaviour and issue an ABEND.

If you move to LE, you can customize the run time option CBLQDA to specify
OFF instead of on. I have documented this in
www.travel-net.com/~denisb/mvs.html
```
