As regards the Screen Section, I would think the various compilers are
pretty compatible - with the exception of AcuCobol, which offers the
best of both worlds, a Screen Section which does Text or GUI.

For info - in the M/F DOS version, I had an indexed file containing menu
text lines and a program makemenu.cbl which accessed this file to
create text menus using Screen Section.

(c) GUI - now that's a very different baby. Take NetExpress as an
example :-

(i) 
Their own own Dialog System will create dialogs and menus - I don't use
it. (Their demo example uses RECURSION to produce the menu - but I did
use this demo as the basis for creating (iii) below).

(ii)
Using the same concept as C/C++ they also provide a Micro Focus      
Dialog Editor, (based on M/S Dialog Editor); coupled to this you create
a file mymenu.mnu which is included in your Resource File, compiled to
Resource.dll, and is callable from (1) Structured, (2) calls to Win API
(which Ken Mullins uses) (3) and I can use the same Resource file as Ken
to call from OO.

(iii)
Approach (ii) necessitates - design the menu, generate an h.cpy file for 
M/S linking (#define CUSTOMER_NAME 1001) and a file of 	Level 78s (78
CUSTOMER-NAME value 1001) which is a 'carbon copy' of the h.cpy file, to
be used in your program. I find this a bit tedious, so having drafted
what the menu should look like, using mywindow.cbl I have a table in
Working-Storage and then pass an OO OrderedCollection to makemenu.cbl to
generate the MenuBar and SubMenus. My logic for doing it this way - If I
add a menu item I need only change/recompile two files (a) The
business.cbl to handle the new event and (b) mywindow.cbl to include the
additional item in Working-Storage; makemenu.cbl remains unchanged.

Another feature tied to approach (iii); I currently have 62 items in my
MasterMenu, excluding separators, and five lines which are :-

Edit Files	Data Entry
   Items	 Readings - enter history
		 Readings - with Items (Item data already setup)
		 Readings - without items (Item not yet setup)
		 Readings - enquiries (view records - no editing)

All five access the same program(class); when setting up the menu events
I have the option of using two parameters. For all five I use the first
parameter to access a common method and then use the second parameter to 
pass the option 1 through 5 - and of course, the invoked/called
class/program behaves differently, depending upon the option chosen.

Note the procedures above for Dialog System and Dialog Editor are
specific to the N/E compiler. I don't think it will give you any
assistance with Fujitsu, but if you want to see my source, e-mail me
direct.

Acucobol has a platform independent menu system, proprietary of course, here
is a sample:

Notice that this sample implies a lot of assumptions, it is made in a hurry
to provide a generic understanding of the concept, not to provide a fully
functional demo. For those who want that I would refer to www.acucorp.com
for running samples.

      *Escape key place holder.
       77  KEY-ENTERED                         9(03).

      * IDs for items on the menu
       78  MENU-EXIT                           VALUE 101.
       78  MENU-CHANGE-SIGN                    VALUE 201.
       78  MENU-CLEAR-ENTRY                    VALUE 202.
       78  MENU-CLEAR-ALL                      VALUE 203.

      *Create Menu root.
           CALL    "W$MENU"         USING
                   WMENU-NEW.
      *If the menu creation returns zero, it is either out of memory or
      *something else bad is going on.
           IF      RETURN-CODE      = ZERO
                   GOBACK.

           MOVE    RETURN-CODE      TO MENU-HANDLE.
      *Add top level item "Exit".
           CALL    "W$MENU"         USING
                   WMENU-ADD
                   MENU-HANDLE
                   0
                   0
                   "E&xit"
                   MENU-EXIT.

           CALL    "W$MENU"         USING
                   WMENU-NEW.

           IF      RETURN-CODE      = ZERO
                   GOBACK.

           MOVE    RETURN-CODE      TO SUB-HANDLE-1.
      *Add top level item "Edit"
           CALL    "W$MENU"         USING
                   WMENU-ADD
                   MENU-HANDLE
                   0
                   0
                   "&Edit"
                   0
                   SUB-HANDLE-1.
      *Add sub item "Change sign" to "Edit"
           CALL    "W$MENU"         USING
                   WMENU-ADD
                   SUB-HANDLE-1
                   0
                   0
                   "Change &sign"
                   MENU-CHANGE-SIGN.
      *Make a separator after "Change sign" and before next item
           CALL    "W$MENU"         USING
                   WMENU-ADD
                   SUB-HANDLE-1
                   0
                   W-SEPARATOR.
      *Add sub item Clear entry to "Edit"
           CALL    "W$MENU"         USING
                   WMENU-ADD
                   SUB-HANDLE-1
                   0
                   0
                   "Clear &entry"
                   MENU-CLEAR-ENTRY.
      *Add sub item "Clear all" to "Edit"
           CALL    "W$MENU"         USING
                   WMENU-ADD
                   SUB-HANDLE-1
                   0
                   0
                   "&Clear all"
                   MENU-CLEAR-ALL.

      *Show the menu
           CALL    "W$MENU"         USING
                   WMENU-SHOW
                   MENU-HANDLE.

      *Any accept will provide the menu selection as exception from now on.
      *We may want to disable the menu if we want, and we can hide it.
      *Unless we actually delete it, it will live as long as the runtime
      *executes.

      * Delete a menu

           CALL    "W$MENU"         USING
                   WMENU-DESTROY
                   MENU-HANDLE.

      * Check mark a menu

           CALL    "W$MENU"         USING
                   WMENU-CHECK
                   MENU-HANDLE
                   MENU-CLEAR-ALL.


      *Example of an entry with a menu established in the background;

           PERFORM UNTIL            1 = 2
                   ACCEPT           ONE-CHAR
                                    LINE 1
                                    AUTOTERMINATE
                                    NO-ECHO
                                    ON EXCEPTION
                                       KEY-ENTERED
                                       CONTINUE
                                    END-ACCEPT

      * Handle exception keys here
                   IF               ONE-CHAR = SPACE

                                    EVALUATE KEY-ENTERED
                                      WHEN   MENU-EXIT
                                             STOP RUN
                                      WHEN   MENU-CHANGE-SIGN
      *Do something
                                      WHEN   MENU-CLEAR-ALL
      *Do something
                                    END-EVALUATE

                                    EXIT PERFORM CYCLE

                                    END-IF

      * Handle data keys here
               EVALUATE ONE-CHAR
                   WHEN '0' THRU '9'
      *Do something
                   WHEN DEC-POINT
      *Do something
                   WHEN OTHER
      *Do something
               END-EVALUATE

           END-PERFORM.

The menu provided above will appear like this:

 Exit Edit

Where Edit will have the following submenu:

  Change sign
  -----------
  Clear entry
  Clear all

This menu will appear and the application logic behave equally, no matter
whether the host is gui or character.

Although I am not able to show it here, & notation and underlined characters
are supported.
