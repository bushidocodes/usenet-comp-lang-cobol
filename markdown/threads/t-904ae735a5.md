```
This comment on the standard is absolutely right, however, SQL Server was
not written with COBOL in mind....

I use Fujitsu 4.2 and have found from experience that it is NOT always
necessary to EXACTLY match the database definition when declaring Host
Variables for COBOL. For example, a field defined on the database as 12,2
will load happily into a field defined in COBOL as pic S9(10)v(99). (It will
do so even if I make it usage comp-5).

It seems to me your best bet is to declare your numeric variables as
S9(13)v9(5).  I don't think it is likely you will be hitting the trillions
(unless you happen to work for a large Bank in Italy or Spain...I remember
some problems at Banco Hispano Americano in Madrid, but that's another
story...).

The only other alternative I can see is to use pic x(19), and redefine it.
This would involve writing your own "double length" working routines and is
not an attractive proposition. Furthermore it is likely that ODBC will give
errors on the type mismatch and not transfer the data....(this depends on
the ODBC driver and the implementation of the database definition.)

I think you'll find that defining the fields with one high order digit short
will work OK. Please let us know what happens and how you solve it...(might
return to Banco Hispano Americano someday...)

Pete.
```

```
01 WORK-FIELDS
     05  DOLLARS-9
        10  WHOLE-DOLLARS     PIC 9(14).
        10 PENNIES                    PIC SV9(5).
      05 FILLER REDEFINES DOLLARS-9
         10  DOLLARS-X             PIC X(19).

I don't know how how SQL server stores/selects the data which is 19,5, but
you can play around with this concept. Maybe you can SELECT numbervalue into
whole-dollars (truncation desired) and numbervalue into pennies (truncation
again desired) . Maybe you can treat the 19 characters as a string and
SELECT the whole shebang into dollars-x and play with that. (Watch sign
bytes).

If this is monetary totals, the last three digits after the decimal cannot
be significant; if the five-digit field after the decimal is used for unit
cost/prices, the high-order digits cannot be significant.

So depending on the application data, you may need to move the data to some
in-range PICTURE IS clause to work with it.

Alternately:

1. You can use COMP-2 (external float) as a destination field if Fujitsu
supports it, 18 significant digits is enough and the occasional rounding
error is not a concern.

2. Depending on the application field, SELECT the data into a scaled decimal
PICTURE clause receiving item. i.e., choose either the high-order or
low-order 18 digits for this piece of data.
```
