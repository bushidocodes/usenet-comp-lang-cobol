```
It is a common misconception that C programs are faster
than COBOL programs.

The primary determinent of magnitudes of difference in
performance is the algorithm, not the langauges. Where the
algorithm is the same, a significant difference in performance
may reflect the naive use of one of the languages. Where the
programs have been optimised for both the language and the
platform, the performance will be nearly identical. (Excluding
assembler versus other languages!)

A few years ago, I did such a comparison with one non-trivial
program, expecting the C program to be significantly faster
than COBOL. It wasn't! After optimising for the languages and
the platform, the performance of the two programs was nearly
identical with the COBOL program being slightly faster.
```

```
One would think that it would be difficult to design an optimizing
compiler for C.  Optimizing compilers need to know what you want done. 
Cobol is more like a RISC computer, with larger functions which can each
be optimized fully.  C is CISC functions, which cannot be optimized
individually.

I know that some compilers have looked at benchmark programs and figured
out how to eliminate their loops, but nobody has been very successful in
optimizing complete programs by figuring out what the complete programs
are trying to do.  It works better to look for loops and Cobol sized
instructions.

Now large cataloged procedures can be optimized greatly.  (I bet Java
will soon be near C++ speeds).  

There is a danger in programmer optimization.  I remember using an Atari
basic program which was optimized by having the first line a go-to to
the main program past all of the loops which are listed in order of use
at the start.  I plugged in a third party Basic cartridge and that whole
optimization was lost.  The new Basic took a moment when it started a
program, storing where all GOTOs pointed to, so that it didn't have to
search each time.   C is supposed to be portable.  Some of your
optimizations for one platform won't be efficient on another platform
(or compiler).

At one time, people were taught to use indices instead of subscripts in
COBOL.  (not to mention ALTERed GO-TOs).  These were designed to speed
up execution time.  Nowadays, compilers have changed, making subscripts
just as fast as indices.  PERFORMs have very little overhead, etc.

Sure, there are some obvious efficiencies we can make, but sometimes
even obvious ones can be contra-productive given the right optimizing
compiler!
```
