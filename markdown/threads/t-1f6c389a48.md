When you reference an elementary item like TABLE-ITEM(X Y Z) above, the compiler will be
aware of it's usage and will insert the appropriate type of literal or provide data
conversion code as appropriate so to answer your question: yes.  If you do something to a
table at the group level (except INITIALIZE), the entire table is treated as alphanumeric
regardless of how individual fields are defined.  If you MOVE ZERO TO
01-table-name-that-has-comp-3-or-packed-in-it you won't discover the error until you run
the program.  If all fields in a table are defined as 99 or 9999v99 with no COMP(-?) in
them doing a MOVE ZERO TO group-name would do exactly what you want.

To clarify ALL Standard conforming compilers will set numeric fields to ZERO
and alphanumeric fields to SPACES.  (Of course, in this example, COMP-5 is
not Standard conforming - but no matter WHAT USAGE you use, numeric fields
will be set to zero.)  The INITIALIZE verb is the simplest way to do this
type of thing - but not always the most efficient (varies by compiler,
compiler options, and exact source code).  INITIALIZE would also allow you
to initialize the alphanumeric items to some other value than SPACES (such
as low-values or high-values) or numeric items to some other value (such as
1) - just look at the REPLACING phrase of INITIALIZE in your language
reference.

The simplest method is to use no Procedure Divison code at all:

01.
      02     occurs 10.
                03    occurs 20.
                        04   occurs 30.
                               05  zero-field pic s9(5) comp-3   VALUE ZERO.
                               05 disp-field  pic 999                   VALUE 123.
                               05  alpha-field pic x(2)                 VALUE SPACE.


Or method 2: This method had the advantage of easily reinitializing the table multiple
times during program execution, for the cost of a dummy occurrence.
This will also execute considerably faster than INITIALIZE, or PERFORM loops.

01 table-plus-1-occurrence.
      02  pic s9(1) comp-3   VALUE ZERO.
      02  pic 99                   VALUE 45.
      02 pic x(3)                 VALUE '#!&'.
      02     tab    occurs 10.
                03    occurs 20.
                        04   occurs 30.
                               05  zero-field pic s9(1) comp-3..
                               05 disp-field  pic 99.
                              05  alpha-field pic x(3).

MOVE table-plus-1- occurrence TO tab.

You are correct in that the data will cascade. On MVS, the compiler will generate MVC
instructions. This is not a dangerous assumption. From IBM's POP  (SA22-7201-04) manual:
"For MOVE (MVC), each operand is processed left to right. When the operands overlap, the
result is obtained as if the operands were processed one byte
at a time and each result byte were stored immediately after fetching the necessary operand
byte.

As for guaranteeing the WS is in an intial state you can use:
PROGRAM-ID. xxxxxxxx INITIAL.

If a piece of code is not clear or confusing to you, nothing prevents the addition of
sufficient comments to make clear the intent of the code. In fact, ALL code should be
commented heavily. As comments do not become part of the load module, there is no adverse
inpact to adding as many as you think you need.

As far as performance, try executing your loop to initialize the table 100,000 times vs the
MOVE method. These methods are not reinvented each time they are used - at a cost of hours
of programmer effort.  They can be coded as rapidly, or faster than an INITIALIZE loop.
