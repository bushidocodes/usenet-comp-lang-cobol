MEMMaker is still on Win95, but is flagged for DOS-Mode only.

Not that this is the case, but I did on one occassion run into the situation
where the old programs assumed only 640K and the base  kernel and all
subsequent programs always begin loading at 640K down to around 597K.
In the load process the programs would check for a specific range.
When memory was below OR ABOVE this range, you get the infamous
Insufficient Memory message.

On your new setup, check how much memory is actually available.
Anything above 597K in the base 640K is suspect as a possible cause.
Try creating a LowMemory DOSPrompt, specifically requesting 600K
in the conventional memory section/tab.

Yes. It is running out of base memory - the 640Kb
that DOS programs get to run in.  With MS-DOS 6.2 one
ran FREEMEM to optimise the OS loading to give
maximum DOS free memory in the base memory area - up
to 605Kb.  With Win95 this was hidden as MS tried to
hide the underlying DOS origins of Windows (extra DOS
utilities were on the CD but not loaded).  With Win98
the extra DOS stuff was not issued, with ME
(aledgedly) there is little or no DOS support at all.

> Any suggestions on fixing this problem or others
that have seen it and can
> steer us in the proper direction would be
appreciated.

1) Check all your CANCELs and reorganise the
CALLs/CANCELs to ensure that it is 'Last In - First
Out' to ensure that the maximum free memory area is
kept contiguous and unfragmented.

2) Check out the 'ON OVERFLOW' option of the CALL so
that at least it can close down tidily and without
data file corruption.  This _should_ take the
exception path when memory runs out so that the files
can be closed etc.  When files are closed set the
errorlevel and stop run, have the .bat file that runs
the system then do a restart using the ON ERRORLEVEL
to detect that this is required.

This may be a slow way to recover, but it might work.

3) Dig the FREEMEM set of files from the Win95 CD and
run it to maximise base memory.

4) Convert to MF Cobol and use XM to access several
megabytes of memory (expensive).  But at least MF
supports MS Cobol syntax.

5) Convert to another Cobol and ISAM file structure.
