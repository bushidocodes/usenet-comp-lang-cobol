I also lost interest in the original thread, so don't know if anyone
presented the following.

The original posting asked for comments about internal vs. external sorts.
It was pointed out to me several years ago that internal sorts, where Input
and Output Procedures are used, as opposed to Using/Giving, were frequently
more efficient than using an external sort.  Obviously, if the required
processing can be accomplished via sort parameters, then that's the way to
go.  But, assuming that those conditions don't apply, consider the case
where you have a data source (data base or flat file) from which you need to
extract data, sort it, and produce a report.

Using an external sort, you would have the following steps:

1) extract program:
reads the data source
writes extracted data to an output file

2) sort step:
reads file from step 1
writes to intermediate sort storage
reads from intermediate sort storage
writes to output file

3) report program:
reads output from sort step
writes report file

If those steps were replaced with a single program that included an internal
sort with Input and Output procedures, you would have the following:

1) Input Procedure:
reads the data source
writes extracted data to intermediate sort storage
2) Output Procedure:
reads from intermediate sort storage
writes report file

So the use of an internal sort eliminates two entire read/write passes of
the extracted data.  It's frequently pointed out that the sort utility's I/O
is much faster than that of an application program.  But not having to do it
at all is even faster.

Tom McFarland
