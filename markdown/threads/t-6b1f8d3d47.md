and Level II format indexed files do not require one. The data file
can be read as a sequential or relative file and the index rebuilt by
a simple user program as explained in the manual.

Cobol/2 version 2.1 also, as I had already said, uses an ISAM
structure that is a data file of fixed records that can be read as a
sequential or relative file in the same way as above and the manual
says exactly the same as Level II for rebuilding the index. It refers
to Cobol/2 version 1.2 files, indicating they are as described for
Level II.

CrackerJack box".

* That there is no 'rebuild' is not a shortfall of the product as
rebuilding or processing as a sequential file can be done without such
a product.  Later versions (eg 2.4) had an additional, different
indexed file structure and did require a utility, which was provided.

* Converting to sequential is unrequired and not useful. It does _not_
make the output into 'text'.

way would be with the same compiler that was used to create the program that 
creates the file.  (This would mean that all "length" and deleted record issues 
were handled identically).  However, creating a "one-off" conversion program 
(using the .DAT file as input) should be reasonably easy in other languages - as 
long as the person understands all the COBOL USAGE clauses (e.g. COMP, COMP-3, 
BINARY, COMP-5, etc).

Estimated time of actual programming and testing of such a conversion program 
would probably be less than 2 hours AFTER you decide on programming language and 
set up an environment where this is available.

If, on the other hand, you (your user) can't get a hold of a "competent" COBOL 
programmer, then, yes, converting your output "report" may well be the best way 
to go.
