```
During the last few weeks I have been evaluating Telephony Appplication
Programming Interface (TAPI) components which enable your application
program to talk to a Voice modem, sound card, or digital telephony card.


They range in price and function from some very basic "after school" type
components written in VB, right up to fully abstracted development
environments that let you build an IVR application by dragging and dropping
"functions" (like, "answer the phone", "detect a digit pressed", "play a
sound file"...) around on your screen until you are happy with the flow, and
you never need to cope with the underlying TAPI functions.

There is a certain element of "fun" in doing this as you can use a good
waveform editor (I strongly recommend  AutoEdit Deluxe from Mystik
Media...worth the registration fee many times over) to build your own
outgoing messages and sound files.

I have looked at more than a dozen such components and am not completely
"happy" with any of the ones I can afford.

Many are written in China and the associated Help files are in pidgin
English and do not cover the ground properly.

I've therefore resolved to write my own, using COBOL.

The problem is that the TAPI functions seem to be fairly unstable and leave
your system in a state where it has to be rebooted after failed tests that
use certain functions of the interface. I suspect this is down partly to my
own inexperience in this area and partly to evolving TAPI standards. (Either
way, it can only get better...<G>)

My question is this:

Has anybody in CLC used the TAPI interface to build an answerphone, IVR, or
similar system?

Can you share what was good, bad, etc?

If anyone is interested in this kind of software and would like to
participate, please mail me privately.

Pete.
```
