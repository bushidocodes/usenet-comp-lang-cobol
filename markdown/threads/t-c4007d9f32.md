```
Oh my what is structured programming?

This is a can of worms.

According to my original Business Programming Logic book,

The 4 major structures are:
    Sequence Structures.
    Decision Structures.
    Do-while structures.
    Do-until structures.

The programs are all a collection of paragraphs called Processes.  These are
a kind of pieces of reusable code that do a specific task.  The whole object
is to have specific ways of doing things.

The term procedure is only used in conjuction with the term
Procedure-Division.  In this sense a COBOL program is divided up into 4
separate divisions one of them being the procedure division.  According to
the three books I have the procedure division is made up of Paragraphs or
Routines which none of my books refer to procedures.  I guess you can call
them a procedure if you want.  Sometimes these paragraphs are called a
process.  Usually Prcedure is what the entire procedure division does.

In my experience or lack thereof newer books just invent terms so you have
even more terms to understand.  I just stick to the terms that refer to the
language they were designed for.  For instance in C++ you have functions and
in Java you have methods.  COBOL isnt that mystical.

On the other hand the term Procedure is used in my newer RPG book to refer
to single procedure programs with a single entry point.  According to the
book, "Procedures are IBM's most recent RPG enhancement for developing
modular, reusable code."  (Programming in RPG IV by Judy Yaeger). According
to the book peices of code that are compiled separately and called as needed
are called sub-procedures.  In the past programs were written as a single
procedure.  Thus a source to back up the concept of a program being a
procedure.  Therefore the procedure division is simply the part of the
program where the procedure is performed.

Charles.

p.s.  Reading is a dangerous thing.
```

```
The classical definition that I learned of structured programming (at least
in the COBOL world) is programming that does not make use of the GOTO
statement.  Relying on logical data structures (PERFORM UNTIL, IF...END-IF,
EVALUATE...END-EVALUATE) to process data, rather than branching all over the
place to lots of different paragraph names with lots of different GOTO
statements.  It was developed in order to avoid spaghetti logic.

Paul
```

```
Here, here!

If anyone really understood how "structured programming" with a read of the
same file in forty-eleven different places causes unbelievable maintenance
nightmares, we'd all go back to:

100-read-It.
       read any-file at end go to 100-read-it-exit.
       perform 200-do-this thru 200-do-this-exit.
* no reads in here
       perform 300-do-that thru 300-do-that-exit.
* no reads in here
       go to 100-read-it.
 100-read-it-exit.
       exit.

At least when I get an invalid IO on a read, I only need to look at one
place.  With the "structured code" (at least in what I work with) I could
look at 15 different reads, depending on what other routine was being
performed.

Besides, if you look at the machine code that is assembled, you'll see
plenty of "go to" statements in structured code in the form of assembler
"branch" statements.  To me, six of one and half dozen of another.
```

```
For IBM mainframe shops (especially large applications that are
"time/response sensitive") "tuning" of IF's and EVALUATEs is (and has been
for years) one of the easiest and best things that can be done.

"Way back" in the days of OS/VS COBOL and CAPEX (before CA-Optimizer) there
was an "easy" way to get a listing showing where (what source code line OR
object code sections) you were spending the most "time" (resources) in.
Therefore, it was fairly typical (at least in the shops that I worked in) to
do a systems test (not unit test) where you ran several days worth of "real
world" data thru this reporting feature and "moved" any IF (later evaluate)
statements around so that the "most common" tests were first.

OBVIOUSLY, this didn't make sense if the test was only done a "couple" of
times in a multi-hour batch run.  But if, on the other  hand, you were in a
"tight loop" and the same test was done multi-millions of times and you were
trying to get an 8-hour batch job done in 7-hours, this was certainly ONE
easy way to make some progress.  (Obviously, there were LOTS of others, and
this one rarely did the trick by itself).

If you don't have CA-Optimizer, but do have the IBM "debug tool" this same
functionality is still available with the FREQ command (and can  be used in
a "batch debugging" session).  This only tells how many times a specific
statement is executed, not how much time/resource is spent with it.  For
that, I think people still use STROBE or Omegamon.

Similar tools are available with some (not all) other COBOL products and
environments, but they certainly are still used in the IBM mainframe world.
```

```
I learned programming around 1975...  When I was still a junior
programmer, my employer, Irving Trust Co, in NYC, brought in some
people to conduct a two-week course in structured programming. 
Everyone in the department took the course.  Much of what we used was
IBM-prepared material.
This was the first time I had been exposed to any methodology; and I
realized that I had been writing "garbage that worked" up until then.

Structured Modular programming involves much more than the elimination
of GO TO.  It is actually very much like OO programming --- each
logical process is encapsulated in a module (in structured
programming, we would say that each PERFORMed routine has "cohesion".)
 There are a lot of rules.

The one which causes the elimination of GO TO, is the rule that says
every logical construction must have only one entrance and one exit. 
In COBOL, this is done with the PERFORM or CALL statements.
  
There are only 3 kinds of logical constructions that are used:
PROCESS (Do....)
IF ... THEN (Do....) ELSE (Do....)
DO... WHILE.
The DO .. UNTIL is a special case of DO .. WHILE.

Another rule is that you code from the top down.  This is NOT the same
as "Top Down Programming".  It means you do not code any module unless
you have already coded all of the logical connectors to all of the
logically higher elements in the program.

 Along with a few other rules, I have found that writing programs this
way
1. Makes design easier. The program naturally organizes around the
structure.
2. Makes coding easier.  You do not have to keep track of which GO TO
statements need to be 'fleshed out'... it has been so long since I
coded any other way, I can't remember what I am doing easier!  I DO
know that, the more  I discipline myself to do the code in accordance
witrh the rules, the easier it is... I have less and less code to go
back to and change.  Most of the serious programmers that I know, who
use this methodology, agree with me on that.
3. Makes maintenance easier.  I am proud to say that I have often been
told by people who came after me, that my programs are easy to change
and update.

I wish I could find a good reference book to recommend, but I have not
been paying attention to that sort of book for a long time...


Consider a program organized like this:

PROCEDURE DIVISION.

0000-MAINLINE.

    PERFORM 1000-INITIAL-HOUSEKEEPING.
    PERFORM 2000-READ-AND-PROCESS
        UNTIL DRIVER-FILE-AT-END.
    PERFORM 9000-CLEANUP.

1000-INITIAL-HOUSEKEEPING.

    PERFORM 1100-OPEN-FILES.
    PERFORM 1200-DATE-ROUTINES.
    PERFORM 1200-INITIALIZE-REPORT-HEADERS,
    PERFORM 1400-MAYBE-MORE-STUFF.

1200-DATE-ROUTINES.

    PERFORM 1210-SYSTEM-DATES.
    PERFORM 1220-MASTER-DATE-FILE.
    PERFORM 1230-FORMAT-DATES.

EACH level logically includes everything below it.  When I first wrote
this example, I did date routines first.  Then I realized that the
MASTER-DATE-FILE had to be opened.  It was extremely simple to move
the OPEN routine in front of the DATE routine. (I COULD have done the
OPEN in the DATE routine, but I think this way is easier to maintain
... all of the related elements are together.) Had I been using
straight line code with GO TO, I would have had to insert the OPEN
code in front, and possibly wound up with some spaghetti ... this
early in the program.

(It could be argued that OPENing the DATE file is logically part of
the MASTER-DATE-FILE processing, and that I could have done it there. 
However, OPENing the file is a system/compiler requirement [not all
computer languages require an explicit OPEN statement], and is
physically but not LOGICALLY related to obtaining data from the file. 
I place all of the open statements together because they are all the
same kind of system/compiler-required operations.)
```
