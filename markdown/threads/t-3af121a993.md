What is 'straightfoeward and simple' is entirely the same as 'what one

(or few anyway) paragraph names and the same idiom for several decades
so it is easy for you to recognise at a glance.

Most others don't see 'straightforward and utterly simple' when they
look at your code, they don't 'recognise' it (unless they have been
forced to work on programs written in the 60s). They see intertwined
paragraphs and twisty little passages.  They see that the program is
scattered all over from top to bottom (it starts at the top and the STOP
RUN is the last statement).

They don't 'rewrite it just to eliminate GO TO', they rewrite it to
convert what they see as a twisted jumble of code into something they
can understand.

While you may have used W-Field-1 and -2, X1 and X2, in specific ways
for decades and so <<know>> what these are, noone else has this brain
pattern hard wired in and so they need to refer back to determine what
these are.  They may even have to notate these mentally to translate
them into names that they use regularly.

The way that they try to understand your code is to rearrange it into
something that they do recognise and with it arranged into parts that
define the overall logic and parts that do the detail work instead of
how you wrote it with these intermixed.

Of course this means that you no longer see it the same way and thus you
think that it is more complicated.  As you say, it is utterly foreign to
you.

Most programmers don't do much work with trivial examples so they tend
to use the same idiom for examples and for real programs.  This is
because they recognise the code better if it follows some structure and
also uses meaningful names.

Foodman wrote:
> 
> In the trivial example here, there is nothing wrong with GO TO.

There is never anything wrong with GO TO.  Where the 'wrongness' lies is
at the label where the GO TO goes.

That is, if one is examining a program written elsewhere it is necessary
to start by understanding it in small pieces.  As each piece is
understood these can be assembled into a whole so that eventually the
whole process can be understood.  It can then be changed as required.

If in examining one piece you come across a GO TO then this can be
readily understood.  It is very clear why it is being actioned and how
the program logic flows around it.

However, when one is examining a piece of code of a program that has GO
TOs and PERFORMs and there is a _label_ then it is impossible to know
how the logic flows around that piece of code without examining the
_whole_ program.

That label may be the target of a PERFORM, or a PERFORM may end at that
point, or it may be the target of a GO TO.  The flow may stop at that
label, or it may start at the label or it may flow through the label. 
It cannot be known without understanding the _whole_ program.

Thus the presence of GO TOs means that the program _cannot_ be
understood in pieces (without having prior knowledge of it as you do).

Now the original programmer (you for example) will already know the
whole program, especially if it is the same type of program they have
been writing for several decades.  But using structured programming is
_not_ about writing for the benefit of the writer, it is about the
benefit of the 'next guy'.

By restricting the types of coding used, such as 'no GO TO, no THRU, no
SECTIONs', and ensuring this by checking source code with, say, grep
then it is possible to understand code much more readily.  For example
it is only necessary to check that the first paragraph contains a
reachable STOP RUN to _know_ that the logic flow around each label can
only be that the paragraph above ends at the label (returning to the
PERFORM) and the logic flow starts here for each reference to this
label.

This type of restriction means that every paragraph can be examined
completely in isolation without having to know the complete program.

Other types of restriction can also work, such as 'only PERFORM SECTION
and GO TO Exit', but this does require more complex checking such as
making sure the GO TOs are to the correct exit point and that SECTION
keywords are correctly placed.

You don't seem to notice why others find this important. I can only
assume that you don't have to work on code written by others.  When you
do examine some other code you probably get quite angry and yell 'why
can't this programmer write programs like mine so that I can understand
it'.

While trivial examples can always be understood easily because the
'whole thing' can be grasped at one time, the style does not scale at
all well.
