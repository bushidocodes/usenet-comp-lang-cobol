I'm trying to find out how the COBOL'85 standard handles implicit scope 
termination, and I found the following on the web

> Implicit scope termination occurs:
> 
…[6 more quoted lines elided]…
>        scope of any unterminated contained statement.

 From this I gather that when any COBOL statement may have nested 
statements (like IF, PERFORM, etc.), any following statements are tied 
to the nearest containing statement.

So,

IF FEE < 1000
     MOVE FOO(1) TO FUM
ELSE
     MOVE FEE TO FUM
     MOVE 9 TO FOO.

is the same as

IF FEE < 1000
     MOVE FOO(1) TO FUM
ELSE
     MOVE FEE TO FUM
     MOVE 9 TO FOO
END-IF.

And

IF FOO
   IF FEE
     MOVE 9 TO FUM
   ELSE
     MOVE 7 TO FEE
     MOVE 3 TO FUM.

is

IF FOO
   IF FEE
     MOVE 9 TO FUM
   ELSE
     MOVE 7 TO FEE
     MOVE 3 TO FUM
   END-IF
END-IF.

Is this rule correct in general, or are there any exceptions I should be 
aware of (of course, with respect to COBOL'85) ?

Thanks in advance,

Kris De Schutter

```
An implicit scope terminator *only* exists at the end of a CONDITIONAL
statement - and in such cases "implicitly" terminates the conditional
statement WITHOUT turning it into an imperative statement (while an EXPLICIT
scope terminator *does* turn a conditional statement into an imperative
statement).

In the following statement:

  If A = B
      Add 1 to C
 Else
     Add 1 to D
     End-Add
 End-If

both ADD statements are imperative statements - one with an explicit scope
terminator and one without.

   ***

In the following statement

  If A = B
    Add 1 to C
       On Size Error
           Display  "This has an implicit scope terminator"
  Else
     Add 1 to D
        On Size Error
           Display "This has an explicit scope terminator"
    End-Add
  End-IF

The 1st ADD statement is a CONDITIONAL statement with an implicit scope
terminator - while the 2nd ADD statement is an "explicitly terminated"
conditional statement which is (therefore) turned into an imperative
statement.

   ***

Does this make the differences clear?
```

  That 2nd bullet item was determined to be WRONG (or at least misleading)
in the original '85 Standard. The approved Correction Amendment (an official
part of the "revised third Standard COBOL" definition) was changed. The
following correction was made on page B-6 of X3.23b-1991,

"IV-27     Paragraph 4.4.4 entitled Explicit and Implicit Scope Terminators,
last paragraph, item 2,
    delete the last sentence that reads: "Examples of such phrases are ELSE,
WHEN, NOT AT END, etc."

The reason that this change was made was that you may ONLY have an implicit
scope terminator where a "conditional" statement (rather than an imperative
statement) is allowed - which basically limits it to IF statements.

This means that  the following is conforming

    If A = B
 *   Conditional statement is allowed in IF and follows
        If   C = D
            Perform Para1
       Else
          Display "implicit scope terminator is assumed to follow"
    Else
          Display "this statement is valid"
      .

While the following is NOT conforming (as you may not have a conditional
statement within a READ)

   Read  ABC
     At End
       If C = D
               Perform Para1
       Else
               Display  "implicit scope terminator is NOT allowed to end
this IF statement"
     Not At end
        Display "This statement will NOT compile with a conforming compiler
    End-Read

Your two examples ARE cases of "correct" use of implicit scope terminators
(because the conditional statements are nested within IF statements - where

Also, in case it isn't clear, a "scope terminated" statement is  ALMOST
ALWAYS allowed where either a conditional or imperative statement is
allowed.  The "notable" exception is that you may NOT use an END-IF or
END-SEARCH phrase to delimit an IF or SEARCH statement where the NEXT

conforming source code.  Many implementations do allow even this - as an

If you look at any of these - where they describe the formats of statements,
they should show you where you may nest an IMPERATIVE statement and where
you may nest a CONDITIONAL statement.  In fact, the ONLY place where you may
(in an ANSI/ISO conforming program) nest a conditional statement is within
an IF statement.

Because of this, an ELSE (which is legal within an IF statement) *may*
implicitly terminate a nested conditional statement.

On the other hand, a WHEN, an NOT AT END, or any other similar phrase
(besides ELSE) may only be used with statements that REQUIRE that what is
nested be an imperative statement.  This means that the nested statement
MUST end with an explicit scope terminator or it must already be an
imperative statement (with no conditional phrase of its own).
