The acronym stands for COmmon Business Oriented Language
-- - - -

It was developed as an attempt to create programs, especially business
programs, by using common lanuage as the syntax, e.g:

move field-a-value to field-b-area

instead of

mvc fldb,flda

The result was to generate a program that was self-documenting and that
could be picked up and easily understood by someone else. I make no
judgements on the success of that goal. Since field names are
completely arbitrary, it's possible for COBOL to also have stetements
like:

move black-fish to red-sea

and while it makes interesting reading, cannot be understood without
more in-depth study. The language then becomes only as useful as the
skills or the integrity of the person doing the coding.

Dave Polzin

COBOL is a programming language (but you knew that!). Anyway, it is the
FIRST high level language (or maybe second, maybe FORTRAN was first - its
been a long time since I was in school). It is supposed to be designed to
be "English-like" in that anyone who can read English, should be able to
read and understand COBOL (fat chance, altho it is easier than say C++ or
APL). COBOL stands for COmmon Business Oriented Language. It really is
designed for *business* processing. Basically, to me, that means read and
updating files. It is not designed for heavy duty numeric processing, but I
guess you could do it if you're masochistic. In its simpliest form, the
verbs are meant to be easy to understand and use (ADD, SUBTRACT, MULTIPLY,
DIVIDE, ...). One way that it is business oriented is in the DATA DIVISION.
It is very easy to create a "record" description which consists of multiple
"field" descriptions. Also, it really likes decimal numbers. These are very
difficult in some other languages, such as FORTRAN. It is also designed to
be "structured". You are supposed to divide your PROCEDURE DIVISION (the
actual executable code) into paragraphs. These paragraphs should do a
single, specific function such as READ-THE-NEXT-RECORD. Also, unlike some
other languages, the variable names can be quite long (around 30
characters, I think - I never use ones that long). This allows things to be
more "descriptive" as well. Of course, this all depends on the programming
using these facilities.

Without a better idea what you want, this is all I have to say. Hope it is
of some help to you. Anybody out there can feel free to correct me, I don't
mind because I know that I sometimes make mistakes.

John
