```
found a section in the manual that hints at this - it's SC26-9046 Language
```

```
You are incorrect in thinking that compiling (or link-editing) a program as
RE-ENTRANT in ANY WAY impacts the COBOL definition of "last-used state".  If
you compile (and link-edit) a COBOL program as RENT and have an Assembler
driver that does not CALL the module - but rather "starts it as a separate
task" multiple times, then it is true that each "instance" will get its own
copy of Working-Storage.  HOWEVER,  as long as your COBOL environment
remains "active" (and there are a NUMBER of ways this can be done), each
CALL from an assembler driver to the same program will get it in "last-used
state" - and this does include return-code.

If you want to be guaranteed of getting a "fresh copy" of the COBOL program
each time, then add the "INITIAL" attribute to the Program-ID paragraph -
and I think you will get the results that you are looking for.  (This does
NOT require the RENT compiler option - as that really does NOT have anything
to do with "last-used state")
```

```
The issue is that "RENT" is an MVS (OS/390 - z/OS - whatever) concept and is
NOT a COBOL concept (although there is a COBOL compiler option). For the
RENT compiler option to allow for "multiple" instances of the same program
to be "active" at the same time - with multiple copies of "data" - you need
to have SOME driver that is handling "MVS tasks".  The most obvious example
of such a driver is CICS - which does this for EVERY COBOL program (which is
why RENT is required for CICS - COBOL programs).  If you want your
"home-grown" assembler driver to do this, then you need to get into "task"
initiation and management and NOT use standard "CALL" logic.

INITIAL is a (relatively) new feature in COBOL which "guarantees" that every
time that you enter this program (whether from Assembler, COBOL, or anything
else) that you get its data in "initial state".  The default, in COBOL,
always was (and continues to be) that you get it in "last-used state" on the
2nd thru Nth CALL to the same COBOL program in a run-unit.  There is one
other way to get "initial state" for an "already called" COBOL program - and
that is to use the COBOL CANCEL statement against that module between CALLs
to it (but this is only really available to COBOL drivers).

NOTE: One thing that "last-used state" allows is for you (the programmer) to
initialize only those individual fields that you want reset between CALLs.
Simply provide a "switch" to indicate whether or not a subprogram has ever
been called.  Check it - on the first time in, it will be set to whatever is
in the VALUE clause.  Change it to another value.  Then whenever it is "that
other value" initialize (in your own program logic) and/all fields that you
want to be initialized.

NOTE2:  I do want to be clear about COBOL drivers.  The "default"
(especially with pre-LE or non-LE-enabled Assembler) is to start a new
run-unit EVERY time you call a COBOL program.  The symptoms that you report,
make me think that you are using an LE-enabled Assembler driver (or the
Assembler driver is actually under some OTHER program).

NOTE3: There are also some issues related to CANCEL and last-used state that
apply when you have "statically" linked load modules rather than
"dynamically" called subroutines.  If this is an issue for you, let me know
and I can give you some more "hints and tips" on them.

I hope this makes things a little clearer.  If not, please ask again.
```

```
A few follow-up ideas/thoughts/comments:

1) Use of INITIAL in the program-id *only* impacts Working-Storage items
with VALUE clauses.  It does *not* reset other data items to the value
specified in the STORAGE LE run-time option (in case that was what was
expected/desired).  This is true regardless of RTEREUS or what starts the
program going.

2) Use of INITIAL does *not* mean that the program will always "start a new
run-unit".  It only means that the working-storage items will always be in
"initial state" rather than last-used state (and that, of course, for items
without a VALUE clause means that their contents are "undefined")

3) Use of Local-Storage in a program that is not defined as recursive *may*
give the results desired, but there are some things to think about related
to it as well - especially in a program that is not defined as recursive.
The draft Standard gets into some detail on exactly when you "push and pop"
Local-Storage items - and when they are or are not "still active".  IBM may
or may not have implemented these the way they will (eventually) end up - so
I am not positive that I would recommend this for multi-tasking.

4) I did some more searching in the LE R10 manuals and there is a LOT more
on the implications of RTEREUS with "multiple enclaves".  I would STRONGLY
recommend that you look at these pretty thoroughly before using this option
in the type of application described in this thread.
```

```
called modules. If a program does not initialize it, it can give back a
value that is set by a previous called program.

It is logical that it behaves like this because the return-code is not
a private variable from one program.
```
