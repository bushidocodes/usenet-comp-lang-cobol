```
In COBOL you (normally?) don't use the same variable to determine the number
of occurrences in a table and as a subscript. Furthermore, standard COBOL
doesn't let you nest variable occurrence tables - but many vendors do as an
extension. Let's see if the following (non-Standard - but acceptable with
many compilers) code is what you are looking for

01 Row-Variable Pic 99.
01 Column-Variable Pic 99.
01 NumRows Pic 99.
01 NumColumns Pic 99.
01 Full-Table.
05 Rows-Level Occur 1 to 10 times
Depending on NumRows.
10 Columns-Level Occurs 10 times
Depending on NumColumns.
15 Each-Item Pic X.

Move current-number-of-rows to NumRows
Move current-number-of-columns to NumColumns
Perform
Varying Row-Variable from 1 by 1
Until Row-Variable > NumRows
Perform
Varying Column-Variable from 1 by 1
Until Column-Variable > ColumnRows
Display "Current-Item is:"
Each-Item (Row-Variable Column-Variable
End-Perform
End-Perform
```

```
Not necessarily, this is implementation dependent. For example, in some
compilers, the index is maintained as both an occurrence number and an
offset into the table. The trade off comes with how often the index is
SET using

SET index-name-1 TO data-name1 or index-name-2
or
SET index-name-1 UP/DOWN BY data-name1 or index-name-2

as opposed to how often the index is used. In general, you can

SET index-name-1 UP/DOWN BY integer-1
or
SET index-name-1 TO integer-1 or index-data-name-1

as efficiently as do the same function with a subscript. But when you

SET index-name-1 TO data-name-1 or index-name-2

you will get a multiply (unless the element size is 1), and a divide if
index-name-2 is associated with a table having different sized elements.

In general, every time you use an index to reference a table, you have
gained efficiency over using a subscript. Every time you use

SET index-name-1 TO data-name1 or index-name-2
or
SET index-name-1 UP/DOWN BY data-name1 or index-name-2

it costs you over a subscript. So if you use the index more often than
you use those SET constructs, the index will generally be more efficient.

Efficiency aside, you must use an index with SEARCH. And if you're still
using COBOL 74, indexes permit relative modifiers:

data-name-1(index-name-1 +/- integer-1)

But in COBOL 85 that capability is extended to subscripts as well.

data-name-1(data-name-2 +/- integer-1)
```
