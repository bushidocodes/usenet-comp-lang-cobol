```
I got bitten a few days ago by COPY statements in column 8 for a COBOL 
program.

The Transformation Tool in the Migration Toolset has to process legacy COBOL 
source, and this is where the problem occurred.

This is in very ancient COBOL (some of the programs are dated 1982) but I 
know that many sites use COPY currently.

My Migration Toolset COBOL  parser uses reserved words as part of the 
parsing and doesn't expect to find reserved words (other than COBOL Headers) 
in the A margin.

(This is NOT the parser that is incorporated into the cobdata tool; that one 
doesn't use reserved words. See:
http://primacomputing.co.nz/cobol21/newcobdata.aspx )

On checking the Fujitsu implementation (based on the COBOL '85 standard), I 
found that COPY in the A margin is correct. The Migration Toolset has been 
amended to accommodate it, but it irritated me, for reasons I shall now 
explain...

This, to me seems completely inconsistent.

A COBOL parser is expecting to find COBOL structure. (We are not talking 
free format COBOL here...) The A and B margins have specific purposes and 
COBOL verbs in the A margin are not supposed to happen (as far as I know). 
If you write MOVE in the A margin you get a syntax error. So how come COPY 
is OK?

What is the consensus on this? Sure, I know everyone uses COPY and has done 
for decades, and they write it in the A margin (on many sites), and it works 
fine, but it really is inconsistent, in my opinion.

Does anyone NOT write it in the A margin? (I have worked on sites where I'm 
pretty sure they didn't, but I'm not positive...it was a long time ago.)

Can anyone think of any other COBOL verbs that can legally appear in the A 
margin? (I'm not sure about EJECT...)

To me, COPY is an ancient anachronism that was helpful in the time when 
source was everything. If 6 programs all referenced the same file structure 
then there would be 6 COPYs of it. It was necessary to keep everything 
synchronized and it led to versioning nightmares that led to complex 
versioning software like PANVALET and ENDEVOR.

(If things are organized properly there is no need for this stuff. You have 
a test version; you have a production version, they are stored in separate 
libraries and work in different areas of the system.  (In my case they are 
separate virtual machines))

In component based systems there is only one definition that is needed, and 
that is in the interface. The interface itself is an object and it gets 
instantiated ("implemented" is the technical term...) into anything that 
needs it. There is only ever ONE copy and ONE version.

(I don't personally use COPY, even in COBOL,  as I don't like propagating 
source code and I have no requirement for it anyway; record definitions are 
incorporated into Data Access Layer objects where they occur just once and 
are generated by the Toolset. DAL objects manage table sets on a Relational 
database, but they can also manage flat files if necessary. Standard COBOL 
applications which use the DAL objects will have a record definition in them 
(this is populated and read by the DAL object, and is part of the interface 
to that object) and this is typically provided by COPY in legacy code. Once 
the legacy is converted to OO, there is no more need for it; application 
objects and DAL objects simply implement the SAME interface.)

(I've been telling you for years that OO reduces maintenance... :-))

OK, I know that for most people reading this the idea of NOT using COPY 
would be Heresy and Sacrilege but you'd need to use OO programming to 
understand it doesn't have to be the way it is. If you deal with objects 
(rather than source) the world becomes "different"...

I'd be interested to hear your opinions about this and I'd appreciate it if 
anyone can offer more examples of verbs in the A margin.

Thanks,

Pete.
```

I think it is one of the weaknesses of COBOL, inasmuch as it propagates 
source code, and requires versioning. I KNOW "most companies use it" but 
that, in itself is neither here nor there. Many companies use many things 
that are not necessarily ideal...

The more source code there is, the more maintenance is required.

I wouldn't mind a dollar for every time I've had to fix production COBOL 
because someone changed a COPY book and failed to re-compile EVERYTHING that 
used it.

It isn't even like the program tells you it is using the wrong COPY book... 
you get a S0c7 (or equivalent data exception on other platforms) because the 
data passed is not in the format expected, then you pore through a dump to 
find why the field doesn't contain what it should, then you realise the 
buffer is formatted incorrectly and next day someone pipes up, "Oh, yes, I 
changed that COPY book yesterday and I re-compiled, tested and promoted all 
the programs in the subsystem. It was fine."

"Did you not remember that data from that subsystem is passed to other 
subsystems?"

Embarrassed silence and shuffling of feet...

Systems like ENDEVOR are supposed to prevent this kind of scenario, and, 
generally, they do, but they are not foolproof.

I'd rather write a component that "objectifies" the data definition and let 
anything that wants that definition instantiate the component. There are a 
number of benefits in doing this, apart from the obvious one of only having 
ONE definition that does NOT require re-compilation of everything that uses 
it, every time it gets changed. For example...

If you use a COPY statement, EVERY program that includes that COPY statement 
will receive a definition of that data structure and EVERY field in it, 
whether it uses every field or not. If it is an object, you can reference 
ONLY the properties that are actually used. That is part of the advantage of 
using a Data Access Layer; programs don't have to include file IO; they can 
simply get and put fields to an object (which handles all of the necessary 
IO automatically.) Obviously, it is even better if you have a tool that 
writes these objects for you, and PRIMA customers get exactly that. If you 
move from flat files to databases, it has absolutely NO impact on the 
applications. Move from RDBs to CAS or LINQ or whatever new technology is 
announced next week and you can implement it WITHOUT  the slightest impact 
on your applications. Parallell processing for data? No problem. Functional 
lambdas representing queries? No problem.

COPY just ties you into the past. (However, if you haven't moved to OO you 
are tied into the past anyway, so I guess it doesn't matter... :-))

Pete.

COPY isn't really a COBOL verb.  It is a reserved word, but I think it 
falls into the category of "compiler directing" statements.  There are 
other reserved words that can appear in AREA A, such as EJECT, SKIP, and 
at the very beginning of the program IBM Enterprise COBOL allows CBL and 
PROCESS statements (compile time options) to begin in AREA A.  I believe 
Realia COBOL had keywords that could appear in Area A for turning the 
compile listing on or off.  I suppose that might have saved some paper 
back in the days when we actually printed compile listings and filed 
them.  We got rid of paper listings about 10 years ago.  The compile 
listings are now stored on disk and managed by SAR (SYSOUT Archive and 
Retrieval).

I have on rare occasions used COPY in the procedure division, but it's 
not my preference.  Generally we use COPY for working-storage data 
definitions, and we have rarely used it for FD's in the File Section.

One advantage to using COPY is that it's usually quite simple to find 
all the programs that use a particular copybook, and that saves some 
analysis effort.  One disadvantage is that it becomes difficult to mix 
test and production copybooks in the same program.
