I think Thane's suggestion is the best place to start. If possible, get
the correct source for the map and reassemble it (I'm assuming your BMS
proc does both DSECT and physical map in one run). If you have multiple
map libraries, i.e., the DSECT output from the map assembly, make sure
it's in the SYSLIB concatenation for the COBOL step and recompile your
program.

Something else I'd look at is to browse the loadlib where your physical
map is and turn HEX ON. The date the map was link edited is in (IBM)
Julian format at the end of the 3rd or 4th record in the load module.

Also, make sure you're compiling your program into a loadlib which is
available to the CICS region where you're testing, and insure that you
don't have a stale copy of your program or map in another loadlib in the
DFHRPL concatenation before the loadlib into which you're linking your
program in 1998.

Bill Lynch

PS: If your BMS proc doesn't do both MAP & DSECT passes in one job,
either update it, if you can, or make some noise. This is a maintenance
nightmare.

The types of things that can cause this problem are:

1. Uninitialized logical map (record layout) with garbage sitting in it and the
map is being sent without the MAPONLY. Fix by either doing ERASE MAPONLY (which
means what is on the screen before the SEND MAP is ignored and the record
layout is not used), or moving low values into the record layout for the map
before you start populating the record layout before the SEND MAP.

2. If there is a different map on the screen, you may (probably will) have to
put ERASE as one of the directives on the SEND MAP.

3. Discrepancies between the logical map (DSECT) and the physical map (MAP).
Make sure that you are pointing to the correct library to pull in the current
record layout. If the record layout is hand-coded, you may have to do a BMS
assembly for DSECT and then compare field sizes and atrributes between the two.

Good luck!

This is a student context. Probably there is a lack of full initialization. 
The suggestion of moving LOW-VALUES is an excellent start.  But more generally,
the logical screen (that is the WS area in your program from which you are
going to SEND the screen) needs to be initialized properly before you send.

Very often this can be done by doing a RECEIVE first. That lucky situation
fills up the buffer with something you can immediately SEND back out. But this
engages a chicken and egg issue.

How can you receive a well formatted image of that mapped data if you have not
yet sent it?

Anyway, initialize it before you send it. Learn how to initialize it.

The stated problem includes the phrase, "... The map sends correctly ...". That
is not completely true.  If unprotected fields are protected (that is you can
not enter data into them), then you did not send it correctly. Probably because
you have not initialized it and there is garbage in the attribute fields.

The basic approach for the initial creation of a screen is to 1) allocate the
area for the screen (if that is not done implicitly by inclusion of the
copybook in WS, some programs use GETMAINs for this), 2) initialize it, then 3)
send it.

Once you are cycling through repeated use of a screen layout, you will be
reading the image into an allocated area, and do not need full
re-initialization.  But it can be necessary to do partial re-init (to
de-highlight previous temporary error fields etc., and to get control of the
cursor position).  

However, this cycling does not work too well on the data fields unless the user
has touched each field, or you get the Modified Data Tags (MDTs) under control.
 From your description, you are not at that stage yet.  But you may be.

You have an initial non trash value that you asked CICS for in the user
selection field, and that is contradicted by your description of trash in the
field.  CICS probably did not put trash there, it must be coming from your
program.  If you do not set the modified datatag automatically (by CICS
definition), AND you do not touch that field (and you say you can't), it will
not map back into your program. This is a crucial lesson.  

Even if data starts right and/or is established successfully by your program or
by user entry, upon some subsequent iteration if that field is not touched and
you have an absence of preset MDTs, it does not come into the next scheduling
of your program. If you send that screen back out, the good data on the screen
is replaced with the unmanaged data in the buffer from where you do your SEND.

So on the first iteration it is not a bad idea to initialize the whole buffer
with binary zeroes, but then put some character data into the text fields (such
as SPACES). Generally it is best to preset the MDTs to get the good data that
either you or CICS placed INITially upon the screen. Otherwise you need lots of
extra logic to retrieve from some convoluted place all the data that you are
going to 'remember' that you had put onto the screen.  In other words, use the
screen as your short-term memory.


Also whenever garbage comes into the program,clear it. That is different from
echoing user errors back to them. Real garbage you must clear. User errors are
policty decisions, echo back the error with diagnostics and highlighting, or
clear, and prompt all overagain.  But definitely clear actual garbage.

You are lucky that the screen is coming up.  You will no doubt encouter this
uninitialized situation again.  But as your screens get larger, the
uninitialized buffers become more likely sources of trash that will make the
screen PROG; becasue they have a binary value that looks like a plausible 3270
data stream command, but are illegal.  

If you are going to send a screen, every datafield should have reasonable data
in it before you SEND. The control attributes must have intentionally populated
values.

So initialize upon first construction.

Upon receipt and prep for re-sending, make certain every field has legitimate
values. That is edit everything, no matter what small subset you choose to
diagnose.  Edit every user field. Edit every user field, and make sure you edit
every user field BEFORE you send the screen back out. 

Take-over each user data field BEFORE ever sending out a screen. Every field
must have a value you establish or have reviewed.

Learn what MDTs are. Exploit them. Use the buffer inside of the 3270 (emulated
or not, or any other device) as your memory.  The way to remember is to map it
back in with MDTs.

You have had good suggestions from other posters about sending initially
MAPONLY.  That is very frequently perfect. But, also sometimes you want to send
that first screen from a buffer (initialized and controlled) to get default
values that very per context (such as operator ID, or some previous process
factor), which can not be coded as single initial values in the CICS
definitions.  It is for that reason good to learn how to set up the entire
image, user data and all, from scratch within your program.
