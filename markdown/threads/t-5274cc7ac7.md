Hashing and generating a unique key are not exactly the same operation.
To get a unique key, you first compute a hash value, then divide that by a
prime number somewhat larger than the maximum number of items you
are processing. Then you insert the item and its key in the table. If you
get a collision (you will!), you must calculate another hash key and prope
again. This is standard hashing theory, and I assume that you know all
that.

Now to the hash value calculation. This is surprisingly difficult. The main
problem is to convert the characters of a data item to numbers so you
can compute on them. Cobol does not really have the concept of characters
as bit patterns. There is also the question of character set used (e.g.
EBCDIC
or ASCII or Unicode or ...) and of general portability. Converting
characters
to (bit) numbers could be done by redefinition of binary numbers, by using
a function (if your compiler supports it), and other such (mostly
non-portable)
tricks.

that gives the same result with all compilers on all platforms. It can
easily
be generalized to use more than space, A-Z, and 0-9.
