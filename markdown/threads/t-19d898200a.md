Robert:

A GUI port at the API level is fairly difficult. The functions available in
Windows (or PM) are pretty basic and require considerable support code to do
things such as position cursors, resize and scroll windows, tab from field to
field, and so on. We already had much of this for our DOS screen manager,
however, so we didn't have to reinvent much of this logic.

We are in the thick of porting our ScreenIO COBOL screen manager to the Windows
environment (I don't think we'll bother with PM 'cause there isn't much demand
for OS/2 stuff these days). It took about 4 months, but we now have a solid
alpha version running with CA-Realia (16-bit) and Micro Focus 4.0 (32-bit) in
Windows 3.1, 95, and NT environments. It's working great.

Although the CA-Realia and MF-supplied GUI sample programs were helpful, they
looked as though they gave a C guy a COBOL manual and said "write a COBOL
Windows application" because the stuff translates almost line-for-line to C.
Hard to follow, in other words, and definitely NOT true COBOL.

Our approach preserves the procedural COBOL programming method (existing
ScreenIO applications require only a couple of lines of code changes to run
under Windows). The other COBOL Windows products (Visual Realia, Dialog
Manager, SPII, Fujitsu COBOL) are implemented more along the event-driven C or
VB models and require quite a bit of code modification (or a more-or-less
complete rewrite) to utilize them.

Because of incompatibilities between the old DOS fonts (in Windows, the OEM
fixed font, used by DOS applications, contains characters such as line-draw
characters, which don't exist in the Windows GUI character sets), this version
of ScreenIO is a hybrid. It is a true Windows application but it implements
only GUI elements that are compatible with legacy ScreenIO applications, such as
menus and some message boxes.

We expect to release this upgrade to ScreenIO licensees in late February.

Next step is a true GUI ScreenIO.
Kevin J. Hansen

Kevin:

While I congratulate you on your upcoming Windows Screenio release,
part of your message suggested that sp2 takes an event driven approach
which is not true.

Just to set the record straight, COBOL sp2 is NOT implemented along
the event-driven C or Visual Basic models. All screen handling is
accomplished with ANSI standard COBOL CALL USING statements. All
parameters passed in the CALL are COBOL Working Storage Variables.

All screen control is accomplished in a procedural flow approach. The
COBOL programmer is NOT required to process any Windows events at all.
COBOL sp2 takes care of translating Windows events into values
contained in COBOL Working Storage variables.

In addition, existing character mode sp2 applications can also be
transparently ported to Windows without re-writing the code. We have
many customers who started using our product in character mode (some
back as far as 1986) and now run in native Windows mode, OS/2
Presentation Manager and UNIX Motif by making a couple of minor
modifications to their code. We've supported transparent text mode to
GUI mode ever since we released our first GUI version for Sun Open
Look back in 1989.

I apologize if I've made this sound like a commercial, but it's
important that the COBOL programming community be provided with
accurate information regarding the nature of ALL programming tools
available to them.

Again, congratulations on your upcoming Windows version!!

SP2. I had to add one additional call first to "open" the panel file and a
call at the end to close the panel file. Those were the extent of the
changes necessary to get the programs working under Windows with SP2
instead of DOS only with no mouse, and SCREENIO.

SP2 applications are indistinguishable from any other windows application.
All the normal windows objects are supported. In addition, there are
toolbars and "hints" - when the mouse is over an object a small pop up
description is available.

interact with VBX objects. You can add level 1 VBX objects to your COBOL

Kevin:

I know that Micro Focus Dialog Manager incorporates a scripting language
and CA-Visual Realia incorporates the BASIC language to accomplish the
GUI building process in their Cobol products, but Sp2 from flexus does
not even remotely resemble this approach. The Sp2 product uses a
standard Cobol "call" to the Sp2 runtime using two parameters. In no
way does it require the programmer to handle or even be concerned about
event-programming--Sp2 takes care of that behind the scenes. Sp2 is
very similar to Norcom's ScreenIO in this respect.

The sample programs you refer to, which Micro Focus and CA-Realia
provide, are examples of coding native windows programs. I agree with
you about the difficulties of writing native windows programs and that
you do have to do a great deal of C-to-Cobol translation. That's
exactly why I believe a good tool is necessary and should be used--who
has the time to do it natively--I don't. Anyway, I thought it was
important to distinguish the differece between Sp2 and the other
products.

In my experience, converting ScreenIO applications to Sp2 did not pose
any great difficulties. Converting them to Dialog Manager or CA-Visual
Realia would have been far more time-consuming, not to mention the
learning curve. I chose the Sp2 product because it was designed with
the Cobol programmer in mind. If I wanted to do it some other way, I
would have chosen Visual Basic or something else. But with Sp2, I
don't have to change. I can leverage my
long-time Cobol skills and spend my time providing solutions instead of
reinventing the wheel all of the time. I think that this is the
approach that most Cobol programmers will find the best and is what they
need.
