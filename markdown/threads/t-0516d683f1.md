This is a fascinating article because it demonstrates so well two very
different ways of thinking under two very different environments. In
it, COBOL programmers take a class in Perl, without the expected
experience in C like languages or a Unix environment. Although
experienced, both instructor and students hit some roadblocks. It's a
great example of pointing out what programmers take for granted!

First, Perl has variable length records with explicit end of line
markers (newlines). But on the mainframe, record length is maintained
by the system, not manifested by the data itself.

Second, Perl doesn't have record oriented I/O. The program reads data
and then splits it up into fields, whereas in COBOL, the work of
spltting up the input data is done by the data structure itself, not
by program code. (Something a Pascal programmer *would* be comfortable
with.)

[Slightly O.T., except that file transfer issues are relevant here.]
Point #1 hit me bluntly a few years ago when I used FTP to transfer a
file from an IBM mainframe running MTS (Michigan Terminal System) to
my PC running MS-DOS. I had not specified a text transfer, so I got a
binary image of the file. OK, I thought, since I had a program that
translated EBCDIC to ASCII. To my horror, the output file was all
smashed together with no line breaks at all. Obviously MTS stores
record lengths, but they're not in the data!

I wonder what it would be like for Perl programmers to learn COBOL?
