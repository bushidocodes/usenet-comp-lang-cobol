```
One reason I prefer standard COBOL to the OO work and distributed
processing I have done is that everything is visible in one place.
Stand alone systems not dependent upon anything else means there's no
finger pointing.

A co-worker is working on a distributed system where he was on the
carpet when one customer couldn't get his data FTP'd to our mainframe.
Only when the bigwigs became involved did the customer admit that his
changing their firewall may have had something to do with the problem
(that WAS the problem).  Finger pointing occurs when different people
own different pieces of a system.

One reason OO's use of reusable objects is much smaller than predicted
is because people don't want to be dependent upon code/behavior owned by
others and/or hidden from us.   We don't want finger pointing.  We don't
want to search layers within layers to find out why some system is
behaving the way it does.  We want OUR code to be predictable.

A powerful, reliable centralized system provides the background for
stand-alone jobs and programs with few dependencies on other people's
code.
```

```
Howard offers a lot of valid points. It's funny how as adults we need
complex rules for sharing, something we were supposed to have already
learned as kids.

While centralization has merits, there are legitimate reasons to
decentralize. There are reasons why businesses are divided into distinct
units. Rather than ignoring the divisions, software developers need to
respect the divisions. We have to find ways to make software development
effective in the 'real world' of the corporation rather than in an ideal
world where we have all the answers to all the problems.

For example, I worked on a project where a business unit was responsible for
keeping track of customer information. Clearly the information was useful to
many other departments. However, the rules concerning the presentation and
validation of the data were the responsibility of my department.
Centralization would be inappropriate since it would transfer the
responsibility for the problem domain out of the hands of the experts and
into the hands of some other group that could not (and should not)
understand all the details of customer registration.

The solution to the problem was to create a 'stock exchange' for objects.
The stock exchange is an appropriate model for sharing objects. Exchanges
establish rules for trading and provide enforcement for those rules. Groups
that want to trade on an exchange are required to abide by the rules.

Getting back to my earlier example: any group can 'list' on the exchange,
provided they adhere to the rules of the exchange which means standards for
code,  versioning, documentation and, most important, a service level
agreement which details the mechanism by which defects and design changes
will be addressed. Any group can use a listed object as long as they agree
to follow the rules for reconciling problems. The central group is
responsible for the process rather than the objects themselves.

A side-benefit of the exchange is each object is documented and indexed.
This addresses a big O-O dilemma, finding the appropriate object for the
job. If you can't find it you won't use it.

There are techniques for using somebody else's objects which provide
stability and isolation from change. I suspect that Howard even uses these
techniques. Every time he uses a compiler, system subroutine, or database he
is depending on some other company's code so I'm sure there are procedures
for getting around problems and upgrading.

I agree with Howard when he says that code should be 'predictable'. I submit
that code which is based on shared components is more likely to be stable
than code which is developed from scratch.

-- Karl Wagner
```

```
Certainly, but hidden code is a necessary and basic part of OO.

My PC's tape backup program hasn't worked since I upgraded it a year ago.
Today I got an e-mail from tech support to upgrade the bios on my tape drive
and also to upgrade the bios on my motherboard.  I had used their program to
find the bios on the tape drive, so they knew what to require there, but I
suppose I'll have to find the bios chip on the motherboard.  Last time I
needed to upgrade the bios on a PC was when I got a large hard drive - at
that time, I couldn't find a new bios and bought a new motherboard instead.
See my problem?  These hidden dependencies make assumptions.  An upgrade of
an object may assume that everything else is up-to-date.   This isn't too big
of a problem where we can see everything which is dependent upon that hidden
object - where it's use isn't too widely distributed.

Complex, interactive dependencies mean that bugs are complex and widespread.
Where this is needed in the physical world (say, a bridge), overbuilding is
necessary.  The internet allows multiple paths so that we're not dependent
upon a single path.  I have to admit that I have seen "belts and suspenders
code":
IF A=B GO TO A1
ELSE  GOTO A2.
GOTO A3...
But this isn't the same thing.  I do put in some "never happen code", though:

IF  FUNCTION LENGTH (IABRCBC)
  = FUNCTION LENGTH (BEFORE-RECORD)
     CONTINUE
ELSE PERFORM ABORT-ROUTINE.
Or
LAST-PARAGRAPH.
    Display "Code should never reach this, contact whomever last did
maintenance".
    PERFORM ABORT-ROUTINE.

I wish all COBOL functions provided failure codes.  But still, most serious
bugs are design bugs, not coding bugs.  If the code/object/subroutine doesn't
quite do what it did before, when we used it for our program, it might not
give an error message.  It may simply work partially wrong.  But it isn't
YOUR object, you didn't test the change in that object, since you didn't know
it was changed.

When a compiler change makes a program different, we have to compile it to
make that change.  Then we test it.
```
