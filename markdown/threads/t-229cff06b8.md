While this is not the ONLY way to do it, it is a good way.


Here are the fundamentals:

1. DON'T embed SQL in the application. Instead either CALL (non-OO) or 
INVOKE (an OO object method)  to do the access.

2. Ensure there is a data buffer in which data can be transferred to and 
from the DAL object/module.

3. It is sometimes helpful to have a status and a few other bits and pieces 
returned by the DAL object/module, particularly when things go wrong.

You can therefore have two parameters passed to the DAL object/module or 
combine them into one. The Toolset DAL has a single "parameter block" (8K) 
which serves for control, status, error messaging, and data. (A COBOL 
definition of it is in the manual.)

So, implementing the above allows a separation between your application and 
the data retrieval sub-system. But it hasn't answered your question 
regarding granularity. (Do I write a DAL handler for ALL tables, or for EACH 
Table?)

The aim is to get generic code. So, a general template should be devised and 
then used to generate specific solutions.

If you write a template for ALL tables, it will be specific to that 
database. For a DB with a handful of tables on it that is quite feasible. 
But what about when the DB has 100 tables on it?

The solution I opted for is to have a handler for each TABLE SET. (That's a 
base table and any repeating groups attached to it.) These tables together 
will be used to build a COBOL record, complete with OCCURs and REDEFINES in 
it. The DAL object/module constructs and deconstructs the COBOL record(s) 
supported by the TABLE SET.

(TABLE SETs happen when a database is normalized. If it is representeing a 
COBOL record that had OCCURS clauses in it, there will be attached tables as 
part of normalization to 2NF.)

The templates I use are held internally in the Toolset and used to generate 
a handler for any given TABLE SET.  (You COULD do this manually without 
using the Toolset, but it takes a couple of seconds with the toolset and the 
generated code has already been verified and debugged.)

The SAME template is used for ALL TABLE SETS on ALL Databases. It is truly 
generic and can generate a DAL object for ANY TABLE SET on ANY Relational 
database that has metadata from table generation available. If you don't 
have metadata available (your DB was not generated by the Toolset) you can 
still generate the DAL object, but you would need to manually provide what 
would have been provided by the metadata., (Mostly this is information on 
primary and alternate key structures...) (In response to requests from 
users, the latest version of the Toolset incorporates the capability to 
generate DAL objects from ANY tables on a RDB, but there are some minor 
limitations if they were not generated by the Toolset as there is then no 
metadata available.)

The manual referenced above describes the methods available to the generated 
DAL object/module. (Note that they are applied against the TABLE SET it was 
generated for),

Don't confuse my views on OO in Micro Focus with my views on COM support in 
Micro Focus.

Micro Focus implements OOP using a choice of standards (ISO and MF). 
(Although you HAVE to use the Micro Focus standard if you are writing COM 
components).

In general, OO support in Micro Focus is perfectly adequate. However, it is 
poor, when it comes to COM (in my opinion; it is arguable). I found COM 
development (and I was developing the DAL base template for Micro Focus 
COBOL with it) to be clumsy, confusing, and without batch compile 
facilities. See my previous post here and on the Micro Focus community web 
forum. To be fair, Micro Focus offer both in and out of process COM server 
development (which is good), but I'd rather see more explicit documentation 
on in process server development (which is what most component based systems 
are using), and there should be a CLI batch compile process available for 
COM. (Currently, COM servers developed with MF COBOL MUST have a project 
attached, because the project hides some of the background process used to 
compile and link. I ended up having to write my own COBOL parser and IDL 
generator because of this. If you are batch processing dozens of COM objects 
it is pretty much not an option to have to attach each one to a project...

I was interested in late bound in-process COM server development, using OO 
COBOL. In this specific area, in my opinion, MF was just outclassed by 
Fujitsu, where the whole process is simple and straightforward. (Fujitsu 
provide a special *COM Class that takes care of most of the Housekeeping; in 
Micro Focus you have to do everything yourself...)

I will say that MF provided friendly, useful, and willing support through 
their forum and, as this is free, I was very impressed. Their support 
"culture" was exemplary.
