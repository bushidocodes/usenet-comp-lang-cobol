'Possible' is a big word, but as a rule, the answer is no. You need what is
called the "file description" to understand COBOL-created data. FWIW, the
"file description" should be provided to a licensed user by any software
vendor on request even if you have not licensed the complete source code.
(But not all software vendors will do so).

That's not to say you can't sit down with a hex viewer and figure it out,
but that method (does this qualify as a method?) is at best dependent on
making a number of really good intelligent guesses.   Ed Guy has something
called "parserat" which can help if you choose to go this route.

In the meantime, here's some light reading: a free tutorial on using
COBOL-created data with either 'other operating systems' or 'non-COBOL'
systems:

http://www.talsystems.com/tsihome_html/downloads/C2IEEE.htm

```
Rey wrote:
> it's possible to determine structure from a COBOL data file without
> having COBOL sources??

Um, maybe. If the file is anything other than ISAM, you can dig out the
structure using hex editors. For example, if you find carriage-returns/line
feeds you probably have a line sequential file. Absent those, you probably
have a straight binary file and, by diligently studying random bits, can
probably deduce how many bytes in each record (look for repeated stuff).

Some files/records have inscrutable bytes at the front: these are usually
counters to tell the program the record lengths.

Now you've got the records separated. As to the data inside the record,
you're on your own, just like any other file ("Hmm, a '35.' Is that quantity
on-hand or the employee's age?")


With ISAM files, there is no hope of manual deconstruction.
```
