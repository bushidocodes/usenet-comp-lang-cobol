Your problem sounds a lot like one I recently experienced with a 3rd
party C product, but with a different COBOL compiler. My problem was
caused by the 3rd party C product having all but one of its api calls
use the standard Windows system calling convention, while one of them
used the standard C calling convention. It never crossed my mind that
there would be an inconsistency in the 3rd party use of the calling
conventions. If it had not been for CA-Realia support, I might still be
struggling with the problem.

Try a different calling convention. If it fails with the C convention,
try the Windows system convention, then maybe Pascal. If it fails with
all, contact your 3rd party C function provider.

We will probably need some more information. I presume your application and
the DLL are of the same kind (16bit & 16bit) or (32bit & 32bit), if not,
make it the same.

Are you sure you have loaded the DLL itself successfully? If not, get sure.

The function you call, is it using C or Pascal convention for parameter
passing? (The fact that the DLL and the function itself is written in C does
not imply that the parameter passing is using the C convention!) Check if
the functions convention (this is specified pr. function!) corresponds with
the technique being used by your compiler.

How do you pass the parameters? By address or by value?

I don't know NetExpress, but I use to pass the address for strings (BY
REFERENCE) and the value for numbers (BY VALUE), this is Acucobol. If
passing numbers BY REFERENCE, try changing to BY VALUE. If passing strings
BY VALUE, try changing to BY REFERENCE.

Are you passing strings? If so, do you know if the function you call does
any modification of your strings? If that it the case, do you know if it is
accessing the strings expecting to find a string zero terminated (very
likely). In that case you will have to terminate your string using ascii
zero,
 could for instance be done like this:
    STRING "Hello world!" LOW-VALUES delimited BY SIZE INTO param-1.

Generally speaking, it sounds like it is a parameter passing convention
problem (phew! long sentence) The rules for parameter passing does, in
addition to decide the order of the parameters, also rule who should clean
up afterwards. It makes sense if Netexpress believe he is in charge while
actually the called DLL and its function has taken over the ship.

But, as I said, my suggestions are based on Acucobol and general DLL rules,
I don't know if all of them apply to you. It depends on the implementation
technique being used with NetExpress.

Hope this was of any help.

needed to be set to have the parameters removed by the called program
and I was using the MF default of remove by calling program. The dll

test program at the other end was receiving and processing commands ok,
