Steve,

Eight (8) bytes packed is represented in COBOL as 9(15) COMP-3. Five (5)
bytes packed is 9(09) COMP-3. What is the format of the date, CCYYMMDD?
Remember, all valid packed-decimal data contains a sign, located in the
low-order nibble of the last byte. Also, all packed-decimal data is
right-justified with high-order zeros. I'm not familiar with the nuances of
EZTRIEVE, but the actual data must match the copybook definition of 8-Bytes
Packed. If it doesn't, then unpredictable results may occur (plagiarized IBM
statement). The other definition you included for data-type 'N' is probably
for DISPLAY-NUMERIC data, which is NOT packed.

HTH....

By eight byte to you pic PIC 9(15) COMP-3 OR
PIC 9(8) COMP-3?

Anyway

PAK-VAR   PIC S9(15) COMP-3
equals  PAK-VAR 1  8 P 0 in easytrieve speak. 

1 means it starts at position 1 in the file, goes for a length of 8
((15 + 1) / 2) and has no decimal places (if you want Easytrieve to
handle the value as signed you must give the number of decimal places
even if it's 0 - Easytrieve assumes if you don't specify decimal
places that the value is unsigned)

If it was PAK-VAR PIC S9(13)V99 COMP-3 it would be
PAK-VAR 1 8 P 2 (the length always includes the number of decimal
places).

The "8" is the length in bytes, so it represents 15 digits (not 16 that another
message mentioned--the right half of the rightmost byte is a sign, not a
digit).

If the COBOL COPY book contains PIC S9(8) COMP-3, it is a _five-byte_ field (8
half-bytes containing digits, followed by a sign half-byte, but since only
integral number of bytes may be present in a field, there is a leading
half-byte of zeroes, so that makes 10 half-bytes or 5 bytes). (It is also 5
bytes even if unsigned--the sign in that case would be X'F', at least on IBM
mainframes.)

So the Easytrieve-Plus equivalent of COMP-3 PIC S9(8) is __ 5 P 0 (where the __
should be replaced by the starting location, in decimal).

Mark A. Young
