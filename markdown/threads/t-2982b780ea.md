I shouldn't have to guess, but it seems to be MF Cobol on DOS.

The menu at 345Kb must have been static linked and thus include
the run-time, ADIS and EXTFH. On MF these add up to around
270Kb. Unless the called program is *huge* then it too looks
like it was static linked with its *own* copy of the run-time
linked in.

You probably need to look at using the shared run-time so that
a complete program consists of COBLIB.DLE, ADIS.EXE, EXTFH.EXE
and your programs which should be around 10Kb - 100Kb each.

I usually have a small 'startup' program which holds the
common data area and then simply loops in a CALL-CANCEL loop
until some terminating condition. In the linkage area there
are fields Program-Name, Next-Program and Menu-Program.

Startup:
---------------------------------------------------------

WORKING-STORAGE.
COPY "InterFce.WS".
PROCEDURE DIVISION.
MOVE "MyMenu" TO Program-Name Next-Program Menu-Program
PERFORM Call-Cancel
UNTIL Next-Program = "FINISH"
STOP RUN
.
Call-Cancel.
MOVE Next-Program TO Program-Name
MOVE Menu-Program TO Next-Program
CALL Program-Name USING InterFace
CANCEL Program-Name
.

Menu program:
------------------------------------------------------------
LINKAGE SECTION.
COPY "InterFce.WS".
PROCEDURE DIVISION.
blah blah blah

IF Selection = "X"
MOVE "FINISH" TO Next-Program
ELSE
....
MOVE "ThisProg" TO Next-Program
...

This ensures the maximum free memory available for programs and
allows a great deal of flexability. If a print program is to
follow a processing program the it is only necessary to put
MOVE "printname" TO Next-Program in the latter. If different
menus are required the change Menu-Program. These are usually
a little more sophisticated than indicated here, but you
should get the idea.

Also remember to CANCEL what you no longer require. Using the
shared run-time on DOS the programs are not moved around so
stay where they are first loaded. It is useful to pre-load
any required routines so that they load early and leave the
maximum contiguous area. In the startup I do a DISPLAY AT
and open a dummy ISAM file to force ADIS and EXTFH to load
(I probably should just link them to the start up) and CALL
a utility module that remains loaded and is callable from
any other program.

Now to the problem.......

I had a very similar experience, I think it was with AcuCobol on DOS,
but it could have been RM/Cobol or MicroFocus (We are distributors for
all three).

What actually happens when a COBOL call statement is issued is the
follows.

In program AAAAA you call program BBBBB using the following statement

CALL BBBBB using ZZZZZZZ.

Program AAAAA is resident in Memory (It is the active program until the
control passes to BBBBB).

Program BBBBB gets loaded (You now have both AAAAA and BBBBB in memory).

The passing area ZZZZZZZ gets transferred from AAAAA to BBBBB.

Program AAAAA is removed from memory.

As you can see from the above, you require enough free memory to hold
both AAAAA and BBBBB simultaneously.

As I started by saying, I have seen this before, but can't remember the
full details, but that is what I can recall.

You said later on this is MS-DOS, which gives you your answer to the
problem. 345kb plus 237kb is 582kb, so you need to do one of the
following:

1. Reorganise your DOS memory to give you 582k free at the DOS prompt,
plus a bit over probably. Depending on your environment you may be
stuck with less than that, or you may be able to get up to 630k. With a
suitable bit of trickery to re-use the mono mode video buffer you can
get up to 704kb or even more free - OS/2 is good at this. Still won't
give you enough space to load three decent sized independently linked
modules though.

2. Link in all the programs into one executable together, which will
gain you many kb as you only need one copy of the RTS and so on, like
this:

link menu+prog1+prog2+adis+.....

where the menu calls "prog1" which calls "prog2" and so on.
Disadvantage is that your single executable may need to be entirely
loaded in one shot, so for a large suite of programs you still may not
have enough space. In this case you'll probably need to investigate
segmentation, and maybe overlaying - I don't know how the DOS RTS
handles CALL/CANCEL. My own recent experience is under OS/2 where this
is not a problem, we use dynamic link libraries which are loaded on
demand and have no memory limitations, but maybe someone else can fill
you in on building large apps under DOS.

On DOS (without XM) the MF rts (as in Microsoft COBOL v3 and later)
does implement CALL/CANCEL but uses DOS's memory management, which is
rudimentary at best. DOS memory is easily fragmented and exhausted.

E.g. If A.EXE calls B.EXE then calls C.EXE then cancels B.EXE, the
memory map will show A followed by a hole (the size of B) and then C.
The hole can be used to load another called program, if it fits in the
hole. However, the load of an EXE happens in two parts - the header
is loaded to get the link mapping, then the body is loaded. This can
mean the the hole left after B is canceled is too small to allow B to
load again.

Also, if you are using call/cancel then make sure things like adis are
brought in by the first (main) program and not by subprograms, to
avoid fragmentation.
