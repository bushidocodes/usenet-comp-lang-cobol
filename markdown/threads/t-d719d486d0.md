Let's talk about GLOBAL vs EXTERNAL.

The first question (and I think everyone is talking at cross-purposes on
this - so far in the thread) is do you want the called program to be a
nested program or a separately compiled program?

If the program is a nested program (which of course you will "CALL"), then
you probably want to use the GLOBAL attribute (theoretically, this can be on
the record/01-level under the FD, but for simplicity and better programming,
I will talk about when it is on the FD itself). When this is specified in
the outermost (or actually any "outer") program, you do NOT need to
respecify the Select/Assign or FD in the called (nested) program. You can
do any I/O operation in the called (nested) program while referencing the
names defined in the outermost program.

If the program that you want to call is a separately compiled program, then
you probably want to use the EXTERNAL attribute (again on the FD). When you
do this, you need to define the Select/Assign and FD in each program.
However, you CAN do one I/O operation (say the OPEN) in one program and
another operation (e.g. a READ or WRITE) in another. It does NOT matter
which operation occurs in which program.

The only method in which you actually explicitly PASS the FD name as an
argument in a CALL statement is an extension available with some compilers
and which have implementor specific requirements on what the called
subprogram can and cannot do - and has nothing to do with the original
question in this thread.
