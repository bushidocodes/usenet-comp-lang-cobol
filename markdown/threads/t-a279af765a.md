```
I thought this issue had gone away with the passing of the IBM-370
series of computers and COBOL-F!  However, I recently came across this
when I wrote a file that was being processed by another system and
that other system choked on one of my fields.  Here's the scenario:

In my program I have a file and one of the fields in the file is
defined as PIC S9(9)V99.  We'll call it TRAN-AMT.

Throughout the program,  calculations are done and eventually a final
result is moved to TRAN-AMT.  The intermediary calculation fields are
all defined as PIC S9(11)V99  COMP-3.

The program that processes my file after I've written it is happy with
all of my fields.  The other system, not so much.  When it comes to
TRAN-AMT it says it isn't numeric and moves zeroes to it making it
lifeless as well as useless.  I do not have access to the other
system's source, so I don't know what definitions it is using for my
fields.  My guess is they are PIC X something and they're doing an IF
NUMERIC type test on it.

Some research showed that the last byte of TRAN-AMT carried the
arithmetic sign of C plus whatever number.  For example if the last
byte was equal to 1, then in hex you would see C1.  This is a
perfectly valid numeric field, in most cases.  Sure if you browse the
file and see that field, that byte would display as an A instead of 1,
but nonetheless, it is still a valid numeric (as would D1 be).

I made some changes in field definitions within my program and the end
result was that TRAN-AMT would always carry the F sign in the last
byte.  It is not possible for the field to ever be negative.

My query to the group is, if you were in the same situation I found
myself in, what coding solution would you come up with?  I'm curious
as to how others may have solved this.

This is IBM mainframe, IBM Enterprise COBOL for Z/OS 4.2.0.

Regards,
```

```
B) that your program never produces a negative output.

THEREFORE, a simple solution is:
   Change TRAN-AMT from
        PIC S9(9)V99.
to
        PIC  9(9)V99.

If there is no "S" in a USAGE DISPLAY output numeric field, then Enterpirse 
COBOL will put a X"F" as the sign-nibble on all output operations.

No other changes should be required in your program or the other program. 
For IBM Enterprise COBOL, usage DISPLAY numeric fields, those two PIC 
clauses require the same storage.  You should be aware, however, (and 
comment in your program) that if this field ever - IN THE FUTURE - needs to 
hold negative values, then you will need to make other changes.

If you have a numeric (usage display) field that you plan on "sharing" with 
another program AND the field can be negative, then the use of SIGN SEPARATE 
is often (almost always) a "good solution".

NOTE: Enterprise COBOL almost always accepts a X"F" as a "valid" input sign 
nibble for numeric fields with an "S" in the picture-claue, but will produce 
X"C" on output.  It is possible that the "other program" is using 
NUMPROC(PFD" - if it is even in COBOL - but that is just a guess.

NOTE2:  This is a "common" problem in migrating from IBM mainframes to 
"ASCII" environments.  IBM mainframes distinguish between unsigned and 
positve USAGE DISPLAY numbers, while ASCII environments (by default - 
without any EBCDIC or IBM compatibility switch) treat these two as 
identical.
```
