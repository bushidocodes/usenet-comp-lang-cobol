XLS simulates a specific 'current' date for a specific transaction while all
other transactions remain their real current date unchanged. Works witch DB2
too, but not witch the time, helas. --> Test phase

Xpediter allows you to walk through a running COBOL, Assembler or PL/I
program (main or routine) and to peek and change the variable contents. The
transaction is running while you debug the module, but you can select a
specific terminal- and/or userid so all other transactions that call that
routine run unchanged. --> Debug phase (before and after the test phase)

AbendAid interprets the dumps when an abend occured and removes duplicate
dumps (same abendcode, same PSW). --> All phases from programming to
production

Footnote about Xpediter/BATCH and Xpediter/CICS: In order to use this
tool, the developer must do an additional compile phase for Xpediter,
which results in storing an Xpediter "listing" into a dataset.  This
listing is required for Xpediter to be able to show you the application
source as you are stepping thru the code.  

We have found that this Xpediter compile phase can in some instances
double your compile time!  This is not an issue for us because we have
MIPs to burn, but for some shops, this could be a killer.
