Fulvio,

I hit this same problem a few days ago.

You have an instantiated object that you need to pass down to another form.

Normally, you would expect to be able to make the Object Reference EXTERNAL.

You can't use EXTERNAL if you are using OO COBOL as the scripting language.

Bugger!

I tried everything I could think of (including redefining the object
reference), but nothing worked.

I don't know what the "official" solution is (my support with Fujitsu has
lapsed) but I solved it by doing the following:

1. Create an ActiveX control that has the Second form included in it.

2. Add a custom Property  to the ActiveX control, that is the Object
Reference for the Class you want passed to the form.

3. Drop the new ActiveX onto the Main form.

Now you can instantiate the Class within the Main Form, set the Object
Reference property in the ActiveX control, then invoke a Method of the
ActiveX that displays the second form (or whatever you want to do with it.).

It means thinking outside the square and NOT invoking the second form with
"CallForm" or "OpenForm".

I see no reason why you couldn't have any number of forms included on your
Main form as ActiveX controls in this way.

Hope this helps,

Pete..

OK. I am successful to solve my problem!

My problem:

I need to create a class that manage the DataBase
connection for my applications.
So I need to creat it one time, on program start, and share this
object-instance inside my run unit and between forms.

In this test-class I'm trying to use the MyVbQL.dll to connect to
MySQL data_base with InnoDB that support the row-level locking.

In a previous test where I have used the ESQL, the problem already
had been resolved passing the connection name between forms with
EXTERNAL clause and set it with the "SET CONNECTION", but
unfortunately the FJ don't perform the FETCH PRIOR, FIRST, LAST
on cursors.


Solution:

implement the "Singleton" when create a new object-instance of
the class.

How to:

1. create a new class called "singleton" that inherit FJBASE.
2. put the "singleton" class inside the repository of the class
   that you want instantiate only one time.

That's all.

The "singleton" class:

**************************************************
 IDENTIFICATION  DIVISION.
 CLASS-ID. SINGLETON INHERITS FJBASE.
 ENVIRONMENT DIVISION.
 CONFIGURATION SECTION.
 SPECIAL-NAMES.
 REPOSITORY.
     CLASS FJBASE.
 FACTORY.
 DATA DIVISION.

 WORKING-STORAGE SECTION.
 01 CTRL-OBJ OBJECT REFERENCE SELF.

 PROCEDURE DIVISION.

 METHOD-ID. NEW OVERRIDE.
 DATA DIVISION.

 LINKAGE SECTION.
 01 L-CTRL-OBJ OBJECT REFERENCE SELF.

 PROCEDURE DIVISION RETURNING L-CTRL-OBJ.

     IF CTRL-OBJ = NULL
        SET CTRL-OBJ  L-CTRL-OBJ TO SUPER :: "NEW"
        INVOKE  CTRL-OBJ "INIT"
     ELSE
       SET L-CTRL-OBJ TO CTRL-OBJ.

 END METHOD NEW.
 END FACTORY.
 END CLASS  SINGLETON.
**************************************************

This small class has one variable that is a pointer to itself.

In the implementation of the "new" method, there is a test that checks
if
the variable has been initialized, and if not, creates a new instance.

This way of initialization in the control block, means that the
Singleton instance
is initialized, or created, only on the first invoke to the "new"
method.

In this way I can invoke the "new" method on my own class in the
parent form and
"create" the instance, and invoke the "new" method again in the sub
form (child)
and instantiate the same object.

I hope that it can be useful.

Thanks for posting this solution.

You asked me for my comments, so here goes...

It means invoking "new" in two places when there is no need.

As it is for a database connection, it makes sense for this to be a separate
Class, as you have shown.

You could invoke it once and set the reference obtained as a custom
property. (It would be the "last instantiated instance").

This avoids the overhead of a Method invoke from the subform. (or any other
sub form that needs to use this database connection...)

Causing the new method to ensure that the same instance is passed back on
every subsequent invoke (after the first one) may be suspect and is not
really what the "new" Method is for. (Constructors should construct, not
"construct sometimes <G>")

Still, if it works and you are happy with it, good job!
