Hi Gang,

I have been reading some of the postings related to the topic ELSE-IF 
and I have a couple of thoughts on the subject. 

First, I have to admit that I could not see the purpose for this. It 
seemed to me that the existing IF/THEN and the EVALUATE covers almost 
every possible condition that could come up. And the ELS-IF was just a 
bit redundant, IMO.

Then I thought what the heck, why not add features and if I need them 
later at least they would be there. I may not use it right away but at 
this point Cobol could use more options and this may not be a bad idea.

After some thought I started to wonder what code using the ELSE-IF would
look like, I have not used it so I was curious. So I though let me code
up a few lines and see what it looks like and compare it to the current
methods available and here is the reslut.

Example 1:
Problem:

To make this simple lets say you have four variables A, B, C, and D. 
And only the following conditions could be possible:

If A = B then perform TEST-1
If A = C then perform TEST-2
if A = D then perform TEST-3

The following is code is not using a scope terminator.

Using the ELSE-IF: 		                         
IF A = B 			
THEN
    PERFORM TEST-1
ELSE-IF A = C
    PERFORM TEST-2
ELSE
    PERFORM TEST-3.

Using the IF/THEN:
IF A = B
THEN
    PERFORM TEST-1
ELSE
    IF A = C
    THEN
        PERFORM TEST-2
    ELSE
        PERFORM TEST-3.
			
Comment: 
It looks ok, I guess. I think it is a lot cleaner
than the existing structure and it makes quick work of a 
simple example. In this case the standard IF is a bit large 
and you really don't need to use an EVALUATE, so this works
out real well. 

Now here is my question, how do you apply scope terminator(s)
to the ELSE-IF in the above example assuming that the scope
terminator would be END-ELSE-IF.

Any comments, thoughts???

Jeff
JeffreyFarkas@earthlink.net

http://home.earthlink.net/~jeffreyfarkas/

The problem with this construct, particularly for a long list of conditions, is
that it moves most of the compares down to additional paragraphs, giving the
false impression that the additional conditions are conceptionally subordinate
to the first condition.

If there are multiple conditions that are conceptionally at the same level, it
is better to have them together. This is easily achieved with the EVALUATE
statement or, in other languages, with the ELSE-IF.

The problems with building an ELSE-IF ladder using IF ... / ELSE IF ... is that
the indendation gets quite deep with just a few conditions and there is too big
of a risk that at least one of the IFs wouldn't be properly terminated. (It's
easier to see the end of a bunch of conditions with an END-EVALUATE or, in
other languages, an END-IF, than it is to count END-IF statements to see if the
first IF was terminated, that is, if one doesn't use a period to end all the
IFs.

If I amy add to this, also keep in mind that the ELIF coding construct (or
ELSE-IF or ELSIF) can not be split onto two lines. The two words ELSE IF can be
placed on two lines. Thus ELIF reduces the number of _possible_ indentations.
And the cacophony about it is, per force,  less.

When folks do simple non recursive indentation of ELSE/IF codes they can end up
with a large number of END-IFs that look like the Empire State Building
constructed with END-IFs for each floor.

When folks do recursive indentation they end up with Puffs cousin, END-IF The
Pointless Dragon, that I have commented on elsewhere.

Dragon or skyscraper is hard to modify in real code, especially when embedded
within other conditionals. ELIF is linear in appearance as well as function.

With the absence of the END-IF dragon I would predict that the use of ELIF
would lead people to stop recursive indentation of ELSE/IF constructions,
because it would dangle distractingly.

Two level:
IF A = B
    PERFORM TEST-1

    PERFORM TEST-2
END-IF

Three level:
IF A = B
    PERFORM TEST-1
ELSE-IF A = B
    PERFORM TEST-2
ELSE
    PERFORM TEST-3
END-ELSE-IF
*> would END-IF still be required?

Four level
IF A = B
    PERFORM TEST-1
ELSE-IF A = C
    PERFORM TEST-2
ELSE-IF A = D
    PERFORM TEST-3
ELSE
    PERFORM TEST-4
END-ELSE-IF *> close ELSE-IF for A = D
END-ELSE-IF *> close ELSE-IF for A = C and IF A = B
*> what about here?

Better to have infinite level:
IF A = B
    PERFORM TEST-1
ELSE-IF A = C
    PERFORM TEST-2
ELSE-IF A = D
    PERFORM TEST-3
* more ELSE-IF
ELSE
    PERFORM TEST-n
END-IF

An IF increases the IF-nesting level by 1.
An ELSE-IF closes and maintains the same IF-nesting level as
the previous IF or ELSE-IF.
An ELSE closes and maintains the same IF-nesting level as
the previous IF or ELSE-IF.
An END-IF closes the previous IF, ELSE-IF, or ELSE and reduces
the IF-nesting level by 1.
An end of sentence (period/full stop) resets the nesting level to 0.

These rules still allow ELSE IF to work correctly.
IF A = B *> nesting level 1
    PERFORM TEST-1
ELSE IF A = C *> nesting level 2
    PERFORM TEST-2
ELSE IF A = D *> nesting level 3
    PERFORM TEST-3
ELSE
    PERFORM TEST-4
END-IF *> closes IF A = D nesting level 2
END-IF *> closes IF A = C nesting level 1
END-IF *> closes IF A = B nesting level 0

An end of sentence after PERFORM TEST-4 would also reset
the nesting level to 0 eliminating the requirement for the three
END-IF scope terminators.
