The initial "S" indicates that this numeric field is SIGNED (rather
than unsigned). COMP is an abbreviated form for the clause USAGE IS
COMPUTIONAL or USAGE IS COMP or USAGE COMP. On an IBM mainframe this
means the data item is fixed-point binary, with two's complement for
negative numbers. The fact that there are a total of 6 digits of
precision means it will be allocated as a 32-bit fullword binary
integer, at least on an IBM mainframe. This would not necessarily be
true on a different platform. The "V" is just an indication of where
the logical decimal point would be. The compiler uses it to generate
appropriate code to scale the results of computation and to convert to
edited numeric data items with floating dollar-sign, or suppressed
leading zeros.

In the IBM mainframe world, COMP or COMPUTATIONAL or BINARY data items
are 16 bits if defined with one to four decimal digits, and 32 bits if
defined with five to nine decimal digits of precision.

I'm not sure what the equivalent C declaration would be. My limited
knowledge of C doesn't include "assumed decimal points".

› 05 DEF PIC S9(5)V99 USAGE COMP.
 
› What's the USAGE for?

USAGE is an optional keyword. USAGE DISPLAY would be zoned decimal or
numbers in displayable character format. If a DISPLAY USAGE number is
signed, the sign will be stored in the leftmost 4-bits of the
rightmost digit. USAGE COMP-3 or COMPUTATIONAL-3 or PACKED-DECIMAL
would be IBM packed-decimal, with two BCD digits per byte, and the
rightmost nybble (4 bits) being a sign.

If these definitions are from some platform other than IBM, the
internal representation of the numbers may be different.

In C, anything with a decimal point is of type float or double, is usually
stored in coefficient form and usually in 4 (single precision) or 8 bytes
(double precision). The common floating point representation used by many
microprocessors is given by the IEEE Standard for Binary Floating-Point
Arithmetic (ANSI/IEEE Std 754-1985), but compiler writers are not bound by
that if they want to do it another way.

Depends on the compiler and the compiler directives in effect. Most good
compilers will detail in their documentation how each data type is
represented, with examples. The USAGE is an optional keyword. S is for
"signed" and the way in which this is done, again, depends upon the
compiler and directives in effect. The best doco would be for the compiler
used to actually compile this code for production use, but any COBOL book
will address issues such as signed values, implied decimal points, and
numeric precision. 9(n) is n numeric digits. V99 means two fractional digits
are kept. COMP mean computational format (versus DISPLAY format). COMP
could be equated to COMP-4 COMP-5 COMP-2, hell, even DISPLAY, using
compiler directives. This is for compatibility reasons when moving from one
CPU/OS combination to another. COMP-5 is usually least-significant-byte-first
(Intel, ???) whereas COMP-4 is most-significant-byte-first (IBM large systems).

You NEED your COBOL compiler manual, and the directives in effect at compile
for production use of the code. Should be a pretty easy conversion if this is
the extent of the representations used. Oh, you might need to speak with a
systems person responsible for compiler installations and configurations at your
site to find out about the compiler directives in effect in your shop, if your
shop is a large systems shop (IBM, UNISYS, SPERRY, BURROUGHS, etc etc).

One difficulty you will have is that for COMP and COMP-3 fields there is no
C equivalent. In C here are the basics:

PIC S9(4) COMP-5 - INT
PIC S9(9) COMP-5 - LONG INT
PIC X or PIC 9 CHAR

Floats cannot be trusted in any conversion I have seen to date. The above
assumes you stay on the same platform. Moving from a mini or a mainframe
to a PC may yield erroneous results. Wish I could be of more help, your
