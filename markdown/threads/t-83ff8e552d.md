I use external sorts whenever I have a sort by a field in a file:
Advantages:
     I can change the sort without changing the program
     Several simple steps are easier to maintain than fewer, more
complicated steps
     I can use the same program with different sorted files when sort
order should change
Disadvantages:
     External sorts do not use the field names that we see in COBOL
     I can programmatically select one of multiple sorts in an internal
sort.

I use internal sorts whenever I have a sort by a value which is not in
my file.
This can be a calculated value, it can be a value from looking up
something in a database.  
It can also be in a sorted report which keeps track of pages and/or
columns.

I noticed that Pat as patp@cpinternet.com said...
> thanks, Howard, for the answer to my sort question.

There's a further dimension on mainframes where the sort program is 
actually in three phases; input, sort, output.  There are several exits 
available via which COBOL and other programs can be called, but the 
relevant ones are E15 and E35.

The E15 passes unsorted data records from the input phase while the E35 
passes sorted records from the output phase.  Thus, it is perfectly 
feasible to write a COBOL program that receives sorted records via the 
E35 and produces an informative report for management.  It could even be 
enhanced by a routine dropping irrelevant records via the E15 so that 
only those needed are actually sorted.

There's loads of scenarios that can be fulfilled in this way but, in 
reality, with the sample above, most people don't know about the sort 
exits and simply specify a program that ignores records, sorts what's 
left and reports on the remains.

How you put the spin on this depends on your point of view really 
because the compiler, in doing the latter task, actually does the former 
but with a greater overhead.  So, you could recommend external sorts for 
efficiency, or internal sorts for homogeneity in applications.

I can think of another good reason for external:
In my environment COBOL II OS390, when a program with an internal sort
abends, the Abend-Aid usually ends up in the middle of a very large dump
from SyncSort, sometimes 500,000+ lines.

I just finished writing a daily job with 40 steps.  17 steps are programs, 6
steps are IDCAMS, 1 step is IEFBR14, 2 are IEBGENER and the rest (14) are
simple SyncSort steps.  1 of the programs uses 1 internal sort, another
program uses 2 internal sorts.  Most of the programs are under 500 lines of
code.  Performance is decent, nothing spectacular.  Reading all those files
over and over again is technically reproduction of work which is very bad
from an efficiency standpoint, and from a common-sense standpoint.  I know
this job would be alot faster, at least twice as fast, if I had used VSAM
files.  But the difference in maintainability and simplicity is large enough
for me to have taken the sequential processing, external sorts route with
this job.  It is what THIS PARTICULAR JOB required.

If anybody ever tells you one is better than the other (external vs.
internal), remember it all depends on what tool better gets the job done.  I
would not drive a nail into a wooden board using my head.

The debate has raged periodically over internal (program invoked such as
the COBOL SORT verb) versus external or JCL invoked sorts.  I am
revisiting the debate since the environment has drastically changed over
the past 20 years.  One of the big differences came when we went from
real memory systems to virtual memory systems and an even bigger
difference occurred when large virtual spaces became common.  

One of the main advantages and disadvantages of the external sort is
that it is seen as just a part of the script or JCL and not subject to
the same amount of review when a change is made.  An external sort also
can be far more cryptic - 
SORT FIELDS=(5,17,CH,A,28,4,PD,D) 
INCLUDE COND=(32,1,CH,EQ,C'Y'')
in a mainframe sort for example.  The previous statements sort selected
records from a file on 2 fields.  The field positions have different
meaning to a COBOL program depending on whether the file contains fixed
or variable length records.  On a mainframe, it gets really confusing if
the file is VSAM because the sort (SYNCSORT at any rate) may or may not
impute a record descriptor word for a file with variable length records
depending on circumstances.  A record begins at position zero so far as
the COBOL data division map is concerned, position 1 for fixed length
records and position 5 so far as the sort is concerned.  This is because
the record descriptor word is part of the record so far as the sort is
concerned but not so far as COBOL is concerned.  

The internal sort will tell you the actual fields being sorted and if
copy books are used to describe the record or records being sorted, the
likelihood of getting the fields wrong is very low.  The program has the
option of handling I-O errors on the input and output files rather than
taking the system defaults.  Except for very large numbers of records,
invoking the sort in a program (if the sort is properly set up) should
not cause any additional passes over the external sort.  This means that
an input procedure might take the place of one program while an output
procedure eliminates another depending on the processing needs.  This
means overall reduced I-O if the separate files would be temporary
anyway.  Once you get used to the flow, it becomes simple and easy to
follow.  Structured programming and 85 standard COBOL compilers make it
easier to keep the programming clear for the next person to inherit it.  

The improvements to DF SORT are nice but miss the mark if the SORT can
not use the standard COBOL (or PL/1) copybooks either directly or by a
utility that converts them to descriptions the sort can use.  

As someone who has used the SORT to reformat records and with complex
INCLUDE/OMIT statements, I can guarantee you that those sorts are more
cryptic and prone to error than the corresponding COBOL programs.  Since
many shops (most?) do not require comments and there is no way to
guarantee their accuracy, these sorts can be a joy to the maintenance
staff.  Indeed it is ironic that many shops have less control on these
utility controls cards which may be executing rather critical business
logic.  I am grateful that I was able to keep one shop from using the
SYNCSORT report writer features and the other shop I was at didn't use
them.  Note that I would have considered them safe if I could have used
standard or derived COBOL copybooks and field-names instead of
displacement, size, and data-type for all the fields.

I used to be a bit of a SORT bigot and I have written stuff to take
advantage of many of the more obscure capabilities in it.  Sort has
many exits which permit extensive manipulation the data during the
sort. 

All of the issues you raise can be addressed by using the SORT E15
exit.  It  can be written in COBOL.  As each record is read by SORT,
it's passed to this exit.  Within the exit, the record can be dropped,
reformatted, parsed into multiple records or additional records added.

It's very easy to write since there is no read/write logic for the
sort files. The exit is just a called subroutine with the record
passed in the Linkage Section and a return code. As a result, the
program is access method independent as long as SORT can read the
input. Also, it can use the COBOL copybooks for the record eliminating
those issues. 

On the output phase you have a similar exit, E35, with the same
capabilities. 

Most people don't know about these exits and it's a shame as their use
can result in substantial reductions in the amount of coding needed to
perform these operations.

than 1/2 as fast as with an internal sort.

This is very apparent when using UNIT=TAPE.  I find sorting from tape to
tape fast and reliable in our shop(OS/390).  The processor knows to keep the
tape dataset in memory while it is working with it so it is very fast, and
offloads the cache to tape when enough resources are available.  Many times
a step will fail because not enough DASD is available (at least in our
shop).  With TAPE, this is not an issue for almost all datasets, because the
cache is on WORK volumes. Therefore, very large datasets can be sorted fast
with external sorts, even with tape.

When I first started programming in COBOL/JCL, I used only internal sorts.
Now with 1.5 years and aproaching ~50,000 lines of code in production, I
must say that both are good.  I typically use the external more now.  But
internal has it's merits in SOME situations.  It should be used with
caution, as the GO TO should.  Don't do it just to prove you can.

I still use the internal sort occasionaly.  Excellent examples are for
sorting on a calculated field, or a field which is not available on the
input files (from the database for example).  Sometimes, for reliabilities
sake, it is good to do an internal sort of the data before processing, even
if this is duplicating extra work.  This is good because A) It is hidden
from a maintenance programmer just looking at the job; they don't care or
worry about doing the extra work    and B) Alterations to the sort sequence
in a job's external sorts by an unknowingst programmer will not screw up
your job.  But I don't put all this in new stuff.  I've seen it though.
