```
This harks back to the olden days when memory was REALLY expensive
(like a buck a bit).
Let's say you want to store something that is expressed in hundreds, i.e.
the data is always like 100, 1300, 6500, 3200, etc., then you can use the
P character as you did and only the hundreds, i.e. 01, 13, 65, 32 are
stored and take up memory, but the numbers otherwise behave normally
(i.e. they ARE 100, 1300, etc), so when you compute on them you get
the correct result without having to think about the fact that they are
scaled.

E.g.    02  SCALED-NBR   PIC 99PP.
           02  BIG-NBR             PIC 9(8).

MOVE 3200 to SCALED-NBR
COMPUTE BIG-NBR = 10 * SCALED-NBR + 1

gives BIG-NBR = 32001

I don't think there are many still alive that use this anymore.
```

```
The purpose of the P picture character is to put the decimal point
outside the digits actually declared.  Examples I have seen over the
years:

In a certain banking package, monetary amounts were (almost) always in
cents, so that they wouldn't have decimal points.  Fields were declared
in dollars only for the purpose of moving them to edited output fields.  
 
     05  ACCT-BALANCE         PIC S9(10).
     05  ACCT-BAL-DOLLARS  REDEFINES ACCT-BALANCE
                              PIC S9(8)V99.

But to save space, the credit limit field was in whole dollars.  All
monetary calculations are done in cents, so the credit limit field needs
to be declared in cents

     05  CREDIT-LIMIT-DOLLARS  PIC 9(6).
     05  CREDIT-LIMIT REDEFINES CREDIT-LIMIT-DOLLARS
                               PIC 9(6)PP.

Also, back when savings interest was always under 10%, some systems
stored it as a percentage with just one position in front of the decimal
point.  But to do math with the interest rate you want it in units, not
percent.

     05  ANNUAL-RATE-PCT       PIC 9V999.
     05  ANNUAL-RATE-CALC REDEFINES ANNUAL-RATE-PCT
                               PIC P9999.
```

```
A "P" indicates an assumed decimal scaling position. A "P"
```

```
<ibm>
The P indicates an assumed decimal scaling position, and is used to specify the
location of an assumed decimal point when the point is not within the number
which appears in the data item.  The scaling position character P is not
counted in the size of the data item.  Scaling position characters are counted
in determining the maximum number of digits (18) in numeric edited items or in
items that appear as arithmetic operands.  In any operation converting data
from one form of internal representation to another, if the item being
converted is described with the PICTURE symbol P, each digit postition
described by a P is considered to contain the value zero, and the size of the
item is considered to include these zero digits postitions.
</ibm>

The best I can gather from this is that it is a weird way of saving numbers
like 10000, 20000, 30000 or 0.001, 0.002, 0.003 using less space than normal
picture clause by not storing the digits that would always be zeros.  I think
it acts more as a muliplier when the data gets moved into another picture type.
 I have never seen it used.
```
