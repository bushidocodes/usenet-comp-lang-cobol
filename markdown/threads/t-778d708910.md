Micro Focus does:

 Identification Division header - always optional (per '02 Standard)

Program-ID paragraph optional - if the program name default is accepted

Procedure Division header optional (I can't remember exactly when this is 
allowed and when not)

No paragraph (or section names) required in Procedure Division

   ****

All marked as "extensions" when FLAGSTD or FLAG(ANS) is specified.

This "sample" is often cited as  showing how COBOL doesn't have to be verbose

Which compiler - Micro Focus for as long as I can remember. I know it
wont turn you on but it does me. I only have a viewing copy of Acrobat
so I can't copy and paste from the Standards document.  The following is
just as applicable to 'Procedural' as it is to OO. The Standard has a
number of source examples to illustrate OO, one I looked at occupied a
complete page. Usual stuff , Class-Id., Repository, Environment
Division, Data Division, blah, blah , blah.

Using the concept of  'cut out the red tape' M/F allows you to leave out
the 'superfluous' - e.g., why put in either Data Division or
Working-Storage Section headers if you use neither in a particular
program/class. (I don't use the UPPER-CASE approach below, so
syntactically I may not be accurate).Take a hypothetical method(s) :-

METHOD-ID. doSomething.
LOCAL-STORAGE SECTION.
01 SOMETHING1            PIC 9(03).
LINKAGE SECTION.
01 LINK1                           PIC 9(03).
01 LINK2                          PIC 9(03).
PROCEDURE DIVISION USING LINK1 RETURNING LINK2
MOVE WS-A TO SOMETHING1
COMPUTE LINK2 = LINIK1 * SOMETHING1
EXIT METHOD
END METHOD doSomething.
METHOD-ID. doSomethingAgain.
PROCEDURE DIVISION.
LOCAL-STORAGE SECTION.
01 SOMETHING1            PIC 9(03).
LINKAGE SECTION.
01 LINK1                           PIC 9(03).
01 LINK2                          PIC 9(03).
PROCEDURE DIVISION USING LINK1 RETURNING LINK2
MOVE WS-A TO SOMETHING1
COMPUTE LINK2 = LINIK1 * SOMETHING1
EXIT METHOD
END METHOD doSomethingAgain.

So first I'll go for LOWERCASE - doesn't that tie in with 'not
SHOUTING', and my style - I like comments as separators

*>------------------------------------------------------
Method-id. "doSomething". *> can be quotes/no quotes
*>------------------------------------
01 Something1                  pic 9(03).
Linkage-Section.
01 lnk-1                              pic 9(03).
01 lnk-2                              pic 9(03).
Procedure Division using Lnk-1 returning lnk-2.
move ws-A to Something1
compute lnk-2 =  Something1 * lnk-1
End Method "doSomething".
*>------------------------------------------------------
Method-id. "doSomethingAgain".
*>------------------------------------
01 Something1                  pic 9(03).
Linkage-Section.
01 lnk-1                              pic 9(03).
01 lnk-2                              pic 9(03).
Procedure Division using Lnk-1 returning lnk-2.
move ws-A to Something1
compute lnk-2 =  Something1 * lnk-1
End Method "doSomethingAgain".
*>------------------------------------------------------

Note in above I don't need 'Local-Storage Section" - it is understood.
Similarly EXIT-METHOD is superfluous - only required where exiting from
a conditional just like exiting an in-line PERFORM, or, to signal the 
end of the procedure if your method does use Paragraphs. My preference, 
it allows me to focus on the 'guts' without reading verbiage, and the
latter could be an unnecessary cause of compiler syntax errors. I really
am curious that no other compiler vendor picked up on this feature - so
yes, it is an extension, and you do have portability problems  - but I
don't see that happening in OO COBOL in my lifetime !

Look at this one, which while OO is still the same concept   - user
clicks on a push-button to re-sequence a Listbox :-

    *> ByKey is a Level 88 up in global Working Storage
       *>-------------------------------------------------------------
       Method-id. "PB-SequenceByKey".
       *>-------------------------------------------------------------
          if not ByKey
             move zeroes to ws-CurrentIndex
             set ByKey to true
             invoke self "createMaterialsCollection"
          End-if
       End Method "PB-SequenceByKey".
       *>-------------------------------------------------------------

So yes, and I've done it :-

*>------------------------------
display 'Hello World'
*>------------------------------

Compiles clean and displays - providing your monitor is switched on :-)
