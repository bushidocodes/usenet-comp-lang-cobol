Hi All,

I ran into an interesting problem with an occurs depending on in a COBOL
program that calls an assembler program that gets a trailer record that
has an occurs 1 to 10 depending on some value. The record is returned
from the assembler program to the COBOL program and when the record is
moved into working storage in the COBOL program storage beyond the
intended fields are overwritten on the move. The program this is
happening to has only a 3 line change no where near the code that isn't
working. The program has been compiled through endevor with a vs/cobol
processor group. It had been compiled outside of endevor with a vs/cobol
compiler. The only difference in the compile options is the old compile
had nobatch and the endevor compile has batch. If this is familiar to
anyone I would appreciate some help in determining the cause of this
problem. Thanks in advance.

Rick Smith

Rick,
If I understand correctly, your 1974 ANSI (OS/VS COBOL) program calls an
Assembler program to perform I/O. (If it's 1985 ANSI (VS COBOL II or
later), I don't think it could make any difference.) I assume the CALL
includes the Working-Storage area where the record is to be stored (and
where the description includes the OCCURS 1 TO 10 DEPENDING... clause).

If this is true, I doubt that it is your COBOL program that you need to
worry about. You're letting the Assembler routine have access to your
Working-Storage, starting at the beginning of the record description, so
your program is unprotected against (1) errors in the subroutine, and
(2) changes to the input records which result in more than ten
occurrences.

My guess is that the Assembler routine just reads records and returns
them to its caller, without regard to their actual length versus the
caller's expected maximum length. (Communicating your acceptable
maximum would require an argument containing the "10" that is your max.
Do you pass this to the subroutine?) You've told the subroutine where
it can store the record, so it just does it (possibly by giving the
address to the I/O routine, and letting it move the data). The
subroutine, if given the record area address and the max occurs or max
record size, could "protect" you against bad MOVEs, but it would have to
do the MOVE from the buffer under its control.

Final question: Do you need to use the subroutine? It sounds like a
type of I/O you could do directly with COBOL.
Colin Campbell

Hey Rick, Colin. It's kinda hard to pinpoint the problem without looking
at the actual COBOL code itself but I'll take a guess. I doubt it's the
assembler pgm. It may pass data but your COBOL pgm still controls where
to put the data. Here's my suggestions: 1. display the index/subscript
and depending on field to see the values, 2. and/or, if possible, move
the table to the end of the WORKING-STORAGE right before the PROCEDURE
DIVISION. If and when the pgm accesses beyond what's available, pgm
should abend, S0C4 i think. Then, check the dump for the values of your
index/subscript and depending on field.

I really think it's in the COBOL logic.

Hi Rick,

Most likely you've run into the same problem I've once had.

The variable that the occurs depend on is not initiated !

It may contain a very large binary value which will give you
a very large move which consequently will overwrite any code
within the range of the move.

Hope this helps,
