>why would you like to use ASM? and not C or C++ or even Cobol?
>what is the REAL reason for this question?
>
>

In my case, it would help standardize code between the PC and Mainframe.
We already have an Assembly language utility on the mainframe that we can 
pass parameters to identify the external DD name to read, where to find the 
record key and what data from the record to return.  The assembler routine
loads
the entire file into memory (Dynamically allocated to use only the amount
required to store the file), returns a status to indicate success or failure of
load, receives requests for data and returns matched data or 'ZZZZZZ'.
We have yet to develop an equivalent on the Pc and opted to use implementation
specific COBOL/ISAM file processes created as .DLLs for a short term solution.

I'm translating a trading-program (about 200 sub-programs) from
MF-COBOL 2.0 to FJC. In MF-COBOL I've used many
own written procedures in assembler to set colors and check
function-key's. But the normal ACCEPT in FC will not return
my function-key's as I expectl (example using ALT-E for ending a

but *very* fast...) and even not interprete the description

I can explain it with a short sample (pls don't laught, it was written
in 1987 for RM-COBOL 74 and then MF-Cobol 2.0,
today will know better way's for doing that)

01 S-NR PIC S9(6)V99 VALUE ZERO.
01 W-PIC PIC X(10).
...
MOVE "S9(6)V99" TO W-PIC.
CALL "_ACC" USING S-NR
   C-LI C-POS W-PIC C-ATR C-FKT.

so get in C-FKT the code of the function-key entering
the field and control the color with the attribute C-ATR.

This method made it very easy for making a generator
for whole masks.

But in FJC I can't use this routine, and can't replace it
by a COBOL routine (what I could do with
the WINDOW-routines for making a pop-up window)

I need a way to use this call's,so I've not to replace
the whole mask-coding with PCobol or other
GUI-masks-programms.

Thanks in advanced for your good ideas
