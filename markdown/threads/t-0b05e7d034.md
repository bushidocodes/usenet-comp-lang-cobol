- **Message-ID:** `<8q85ee$5ik$1@news.igs.net>`

```
I just found a bug in one of my systems that came up and grabbed us as a
result of upgrading from Fujitsu 4 to 5.  I though that I would post it in
here and possibly save some people some time.

Fujitsu version 4 will not allow spaces in filenames.  In fact, a space is
considered a delimiter, so assigning the name "C:\XX.DAT YY.DAT" will quite
happily work with the file "C:\XX.DAT".

Fujitsu version 5 will allow spaces in filenames.  The same code running on
the same machine will look for the filename "XX.DAT YY.DAT", including the
space and the two dots.  If opened in output, it will CREATE the file
"XX.DAT".  However, if you then try to open the same file with the same
filename in I/O or input mode, you will get a status 35.  Somewhat
confusing, to say the least.

I got bitten on it through a subroutine call with mismatched parameters.
The main program was seeing an X(40) id, while the subroutine was seeing an
X(80) id, with other data in the second group of 40.  Version 4 did not
care.  Version 5 does.

Donald
```
