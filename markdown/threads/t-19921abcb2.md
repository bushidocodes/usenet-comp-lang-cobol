We used to use InterTest until about 1993, when we converted from CICS 2.1.2
to CICS 3.3. At the time, InterTest had been bought by somebody (probably
Computer Associates) and did not have a version for CICS 3.3. I remember it
as being a CICS-only tool. It was very capable, and we were a little worried
to be changed debuggers. Our CICS environment has virtually no BMS maps, so
a debugger is absolutely essential. InterTest was a very good product and
we also used its FILE and CORE transactions for diagnosing and resolving
production problems. They could be seperately secured in a CICS region.

When we converted to CICS 3.3, we also converted to using Expeditor. It is a
good product, and has the advantage of a slightly more user-friendly
interface -- it can automatically display the contents of variables on the
currently executing statement, while you have to jump to another screen to
do this in InterTest. It also allows you to interactively debug batch
programs from a TSO session. I found the setup for batch testing through
TSO to be a little confusing, and I tend to prefer building target test files
and using DISPLAY's when debugging a batch program. But if you need to
debug a batch program you couldn't do that in InterTest, as far as I ever
knew.

We did have one serious performance problem with Expeditor that persisted
for years. In CICS, we would have extremely slow response time, sometimes
15 to 30 seconds for each single-step through a program. We inadvertently
discovered that the bad response time was caused by a DFHSIT parameter
called MROBATCH (or something like that). The SIT parameter was supposed to
improve MRO performance in CICS by waiting for a certain number of MRO
messages to queue up before transmitting them. Unfortunately, this also
affected VSAM buffering, and Expeditor stores its copy of your source code
in a VSAM file. We had to change this SIT parameter to turn off MRO
batching, in order to resolve a performance problem, and suddenly we started
getting excellent response time in Expeditor.

We are in the process of switching to Viasoft SmartTest. It is also a good
product, but requires TSO all the time, for both CICS and Batch program
debugging. This does free up some DSA storage in your CICS region, but TSO
is a performance problem in our shop, which makes me less favorably inclined
to SmartTest. But it is a good product that has been around for several
years and is regularly updated.

Our criteria for a CICS debugger included the ability to examine and modify
storage, Queues, and files. All three of these debuggers will do that.

Of these three, I prefer Expeditor, now that we have resolved our performance
problem. I prefer it primarily because you don't need TSO for CICS debugging
and because the user interface is a bit nicer than InterTest. I have no
experience with Realia Work Bench or Code 370, so I cannot comment on them.

I am a COBOL mainframe developer of 18 years standing. Over the years I
have used COBTEST, TESTCOB, EXPEDITER & the MicroFocus
debuggers paccked with their character-based COBOL & their Visual Object
COBOL. My general feeling is that the micro based products were easier to
use (less set up, more intuitive, etc.) - but that you were of course not testing in
the environment in which the programs were actually going to run. So for use
during development I favor the PC tools - then migrate to the mainframe. For
emergency debugging stay on the mainframe both for speed (no migrating back
and forth) and just in case the problem is peculiar to the MF environment.

I forget which is the newere product (COBTEST or TESTCOB), but "it" has
some nice features that EXPEDITER lacks (ability to use aliases for
llllllooooonnnnngggg COBOL names, better handling of "CALL" statements)
while EXPEDITER has better animation & data display characteristics.

Good luck in your search!
