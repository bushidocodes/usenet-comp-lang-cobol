1) Circular reasoning. The speaker has decided that well-written, legible
programs use short variable names. Therefore, any program that uses long
variable names is not well-written and legible. Therefore, by definition,
they haven't seen any program that meets their definition. It's meaningless.

2) Ask them to look you straight in the eye and, without blinking or
cracking a smile, tell you that "AP-TRANSACTION-FILE-OPEN" is harder to
remember and more cryptic than "APTRNFOPN". If they are able to do it, be
impressed, but don't ever buy anything from them.

3) See #2.

Of course, a lot of this has to do with what people are used to. Obviously
the format that you are used to working with is the format that you will be
more comfortable with, and therefore more easily able to work with. Trouble
is, some people are unable to distinguish between "I like this better" and
"This is better". The former is a perfectly reasonable opinion. The latter
is a religious statement, unless the parameters and justification for the
statement are A) clearly defined, and B) aren't just more opinions expressed
as emphatic statements.

I had to write a pgm a few years ago that generated another cobol
program. The obvious way to do this wouodl have been do have some
variable and paragraph names be the DB2 key values with a short
prefix. Unfortunately this would have made the names too long so
we had to build a lookup table, allocate random names (eg
N-99999) and then use them. This cost me several days work. It
resulted in output that was harder to understand and debug. 

This was a legitimate application - the savings in CPU costs were
millions of dollars per year, using generated code instead of
table lookups (let alone leaving the tables in DB2!).

IMO there should be no arbitrary limits unless it is really
essential. Someone will always have a need to go past the limit,
and it will be a lot of bother and cost to do so eg the Intel 286
and its infamous 'brain damaged' 64k segment size.

I say recently that IBM has introduced a limit for the size of MQ
Series messages in queues shared across a sysplex. That limit is
64k! Come the revolution, those responsible will be in big
trouble!

Once the program is compiled, the size of the source code datanames is
immaterial, because the datanames no longer exist.

table).

Yes, but why have a limit at all? Sometimes it gets seriously in
the way (as in the example I gave previously). And COBOL has a
large number of different, and more or less arbitrary, limits,
that the programmer is going to run into.

In my compiler, the limit is 'available memory'. I had to add in
a check for 30/31 characters for standards compliance and issue a
warning. This slows down writing the compiler and also the
running of the compiler. There are *lots* of similar limitations,
all for no good reason IMO.

If there is no good reason for a limitation it should not exist.
If it has to exist it should be as large as is feasible. It
should efinitely not be a number that is guaranteed to cause
someone grief.

This is not just a problem in compilers, but in computer systems
generally. Intel decreed that segments woudl be 64k, asking 'who
needs more'. The result: Windows 3.1, and the six different
memory models along with other abominations.
