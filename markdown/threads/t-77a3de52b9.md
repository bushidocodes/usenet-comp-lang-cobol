This thread was brought to my attention, and I would like to provide
some perspective.  I work with Micro Focus, Fujitsu, and COBOL-IT for
mainframe re-hosting projects plus some local workstation
programming.  I am reasonably familiar with the business directions of
each of these companies because I have been involved in potential
partnering opportunities with each one on different projects. I
formerly worked with CA-Realia and with AcuCOBOL before the
acquisition.  I have not worked with OpenCOBOL and know of only one
major project done with it here in the US.

COBOL-IT was forked from OpenCOBOL in August of 2008. I am told that
there were a lot of bugs and some performance issues that needed to be
fixed before the compiler could pass third party certification.
Resolving those issues consumed many months, but the compiler is now
certified by Oracle (Tuxedo), Red Hat, MySQL, HTWC (XFrame), Sun, HP,
and Syncsort. Certification projects are in progress with DB2 (UDB)
and Oracle Pro*COBOL; there are no known problems but official
certifications have not been issued yet so far as I know. COBOL-IT has
been successfully tested with UniKix, but without a certificate being
issued.

OpenCOBOL has benefitted from the work of COBOL-IT, as Roger While and
COBOL-IT are regularly in touch and many COBOL-IT bug fixes have been
incorporated into OpenCOBOL. Users of the free Standard version of
COBOL-IT for Windows have benefitted as Windows API calls (STDCALL)
are now supported, Microsoft C compilers are supported, and fully
compiled and linked Windows binaries are available for free download.

The goal of COBOL-IT is to release and maintain an open source
compiler that can be used for mission critical applications by large
scale, high performance users in Windows as well as Linux and Unix
environments. This is why multi-threading was added. Small users
rarely need it, but it is critical for large complex applications.

The biggest difference between COBOL-IT and OpenCOBOL is not technical
but business.  Large organizations cannot use unsupported products,
and OpenCOBOL is not an option.  COBOL-IT is providing professional
support on the MySQL model - a free product with a support forum, and
a professional product that requires a support agreement.

Because of this support orientation, COBOL-IT's development schedule
is being driven primarily by the requirements of large customers,
though this will probably change when the first US distributors are
announced later this year, one of which focuses on smaller users.  The
support for multi-threading was a lot of development work but the
result was relatively invisible to most users.  There has also been a
lot of attention given to Windows users recently.

This is also why COBOL-IT has added a debugger and an Eclipse plug-in
to both the standard and the enterprise versions of the compiler.  The
debugger is basic in its features, but it is being regularly worked
on. There is also an external SORT module and SQL pre-compiler
available for MySQL, PostgreSQL and ODBC, though this is a proprietary
extension to the product and there is no free version.

On a technical basis, Micro Focus COBOL, as the market leader for
commercial COBOL, has been the standard for comparison of features,
not OpenCOBOL. If there are deficiencies between MF and COBOL-IT, you
should report it on the forum at www.cobol-it.com.  There have been
some surprisingly quick fixes issued, even for the free version.

Bottom line: I will use COBOL-IT for serious projects, just as I would
use Micro Focus or Fujitsu.  It's up to my clients which one to
choose.

COBOL to a  "free" compiler.  In those cases MF compatibility is often most 
important - even for conversions from other compilers).

A) COBOL-IT (and OC) both have a HUGE advantage over MF, in that they are free

not a GCC-junkie (and familiar with C based development).  It can be done, but I

commercial product, I would select COBOL-IT if I had primarily "commercial 
COBOL" programmers - while I would select OC if I had primarily GCC/GNU or C

make Micro Focus look "cheap".  As far as I know, TinyCOBOL is "static" at this

This discussion caught my eye because earlier in the year I had to
research whether it would be possible for our Windows/Web COBOL
product to support OpenCOBOL and COBOL-IT.  At the time we supported
Micro Focus 4.0 and Micro Focus NetExpress and Fujitsu  The folks at
COBOL-IT were anxious to work with us to support the Windows API
calling convention and made considerable changes to support Micro
Focus dialect - including some of the more common compiler directives
that we use.  Their releases for Windows are already built and come
with an installer.  As Bill Klein has pointed out COBOL-IT comes with
an interactive debugger that is not yet as fully featured as the ones
that come with NetExpress and Fujitsu.  It does have the basics and
we've been told it will continue to be improved.  They also provide a
COBOL plug-in for the Eclipse IDE but since our product has its own
IDE interface we did not evaluate it.

It was my experience when researching OpenCOBOL that they have a solid
COBOL compiler but they did not seem to be very interested in the
Windows environment.  Unless something has changed recently you have
to download sources from OpenCOBOL and other places to build the
compiler.  While this is natural in the Linux/UNIX world it is not
that straightforward in the Windows world.  However, the big show
stopper for us was lack of  native support for the Windows' API.

Long story short, we found it easy to work with COBOL-IT to provide
support for their compiler for Windows and Web development.

> Vaclav Snajdr
