Yea, I use internal COBOL sort all the time 'cause I hate
hit-counting. I've seen cases where a one-byte mistake in using the
external sort control statements would cause hours of pain and
suffering in finding where the problem resides. So I just hard-copy
my copybook record layouts in the FILE SECTION SD and change the
data-name prefix (I use a four-letter prefix to describe a record
layout in WORKING-STORAGE and only one copybook layout per production
file. If I have to use that layout in the FILE SECTION, I expand the
prefix to the eight letter implementor-name).

Later, if I have to change the layout that causes the data-name to
shift, I just delete the SD record layout, copy the revised copybook
layout back in, then change the prefix. Simple.

I'll be brief, because I think this subject is going to be thrashed to
death:

1) The performance penalty for using an internal sort on an IBM mainframe
is negligible. This was fixed with the COBOL II compilers.

2) One function should equal one program. Multi-function COBOL programs
rapidly become maintenance nightmares. If portability and readability
are that much of a concern, a COBOL program that does nothing but sort
makes for an acceptable solution.

Designing clean, maintainable systems requires clear function definition. I
honestly believe that the second you can say a function does "a" *and* "b",
you really have two functions. And, to my way of thinking, in a COBOL system
1 function = 1 module = 1 program.

Nobody has mentioned yet that COBOL-IIs internal sorting with the FASTSRT
option is as fast as an external sort with a few caveots: The sort must
use USING/GIVING and not INPUT/OUTPUT PROCEDURES. And that the excess
memory in the region (before sort) is sufficient to allow the sort to
execute at maximum speed as an external sort would. I do not know how
much below the line memory is needed for sort, but both COBOL-II and SORT
use above the line memory.

Certainly INPUT and OUTPUT PROCEDURES in the SORT verb have their place
for pre and post processing. Sort performance is a very complicated
issue and I hate to make any generalized statements. If the file were
small enough I'd use a program invoked sort (the sort verb). My posting
was to point out that if in COBOL-II and USING/GIVING were used instead of
INPUT/OUTPUT
then invoke the compiler option FASTSRT. It is much faster
than the alternative E15/E35 exit processing. When E15/E35 exits are
used (as they are for INPUT/OUTPUT) and if the file is large, then it is
generally accepted that the elapsed time lost due to the E15/E35 (one
record at a time exchange) is greater than the elapsed time lost due to
writing an output file, JCL sorting without E15/E35, and reading the
sorted file into a third program. BUT there are many variables, and the
only true test is to benchmark the two tests.

At this moment, my newsreader shows twelve followups to Glenn Estes'
article, and I doubt that I see even half of the articles posted to
comp.lang.cobol.

Each of these followups uses Mr. Estes' article as a pretext to elaborate
on its author's opinion of the SORT verb. Not one pays even lip service
to Mr. Estes' question about multi-platform portability. So here are two
anecdotes about portability:

1. Company X wrote their product with a SORT verb to improve its
portability. They were more than somewhat surprised to discoved that IBM
did not implement the sort-merge module under VM. This oversight has
since been remedied, but too late for Company X, which went out of
business. They still owe me money. Remember: The sort-merge module is
optional.

2. Company Y was porting their product from MVS/XA to Stratus for
reliability. It happened to use six SORT verbs. I was able to save this
one, but it was expensive. Remember: the SORT verb itself is the least
of it. The program that executes when you use a SORT verb is not part of
CoBOL at all. In fact, it's usually the same program that executes when
you use a separate sort. So the SORT verb ports with CoBOL, but the sort
itself doesn't.
