This probably doesn't help you much, but several years ago I used a Cobol
compiler from a company called Austec (the product was called ACE Cobol,
which was a Data General ICOBOL compatible compiler for Unix). Austec was
later bought by LPI, which was then bought by Ryan McFarland (or vice
versa - I forget because we now use Acucobol).

The standard ACE Cobol compiled Cobol programs into p-code, which was then
executed by a runtime executable. The p-code was portable between systems.

Before they were bought, they released a compiler that compiled a Cobol
program into C source code, which was then compiled to give you an
executable version of the program. It turned out to be not that useful.
The C source that it generated was mostly function calls into the runtime
library, which is understandable. The speed difference was not that great
because of this and the executables were huge (well, at that time they were
huge because memory was expensive - this was back in the mid to late 80's).

If I recall correctly, the entire program was "compiled" into the main()
function. PERFORM's were not implemented as function calls - they were
implemented with goto's within main(). The source was basically not human
readable.

As I said, this probably doesn't help, but might give you some ideas on
implementation.

Incorrect. The output of the compiler is either interpreted (.int) or
native (.gnt) dynamically loaded code. Dynamically loaded by the COBOL
runtime system. Execution of either does NOT require source to be
present (unless you are executing under the debugger, of course).
What you don't get is output as .obj files (and hence, as Bob said,
cannot create a *stand alone* executable).
