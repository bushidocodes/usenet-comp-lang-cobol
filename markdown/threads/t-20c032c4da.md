Because if you don't you have something we in the industry call a "data

be guaranteed that the record he thinks he's updating is the record that

what kind of online system you have. But the basic way to do this is a read

(1) the user cancels his update, you unlock the record

(2) the user requests the record be updated, you write his record back,
    which normally unlocks the record

Yes, I think the OP's objection was that if the record is locked no-one can 
read it. It isn't like VSAM SHAREOPTIONS where you can have any number read 
it but only one update it (or less frequently used options that support both 
optimistic and pessimistic locking schemes.)

It is now around 20 years since I designed VSAM based systems but I don't 
think the problem is with the platform; an idiot can screw up software on 
ANY platform. (Mainframes are NOT immune...)

I honestly can't remember how locking worked on PCs and whether a locked 
record was actually inaccessible or only inaccessible for update. But the OP 
is looking for a better way and, so far, there haven't been many useful 
suggestions within the constraint of using flat files. (Apart from the 
traditional mainframe use of pessimistic locking.)

Personally, I favour optimistic locking where collisions are not highly 
probable and pessimistic where there is high update activity. Reading what 
the OP posted, it looks to me like optimistic would be the answer here.

(Records carry a timestamp or other string which can be stored when they are 
read. Before you update them you read them again and check their timestamp 
or flag data to see if they have been updated by someone else. If they have, 
you notify the user and restart the transaction; if they haven't, you 
replace them with the user's update applied and make sure you have also 
updated the timestamp.  This obviates or minimises the need for locking at 
any time. Obviously, if you use a RDBMS most of this is done automatically 
for you. And the RDBMS has the ability to roll back all uncommitted updates 
in a transaction if there are many of them.

Without a database (which automatically timestamps updates or uses 
clean/dirty flags), optimistic locking is tiresome to implement manually and 
that's why the "usual" scheme for mainframes using non-RDB access is 
pessimistic, as described above. This gives generally better integrity but 
it is a trade off against concurrency).

For networked systems with high concurrency requirements, optimistic locking 
is generally favoured.

But most modern systems use Databases, so it is pretty much irrelevant. You 
simply tell your RDBMS which scheme you want on what databases, or take the 
default.

Programmers shouldn't need to be concerned with this; system software can do 
it much better.

The person who SHOULD be concerned about it is your DBA or Data Resource 
Administrator (DRA)... (It's no wonder they always look worried... :-))

Pete Dashwood wrote:
> 
> (Records carry a timestamp or other string which can be stored when they are 
…[5 more quoted lines elided]…
> any time.

This protocol still requires a lock to ensure consistency if there 
will be concurrent updates. There's a race window begining with 
reading the record the second time (to obtain the timestamp) and 
writing the updated record, and ending when the timestamp has been 
updated. (Depending on the underlying storage mechanism, it may be 
even wider than that.)

The storage mechanism may provide that lock for you, but there's still 
a lock, or you don't have a guarantee of consistency.

 > Obviously, if you use a RDBMS most of this is done automatically
> for you.

Yes, because the DBMS provides ACID transactions - which implies locking.

> Without a database (which automatically timestamps updates or uses 
> clean/dirty flags), optimistic locking is tiresome to implement manually and 
> that's why the "usual" scheme for mainframes using non-RDB access is 
> pessimistic, as described above.

I don't know that optimistic locking is particularly "tiresome to 
implement" - I suppose that's a fairly subjective assessment on the 
part of the developer. Certainly it can be implemented once and 
abstracted away, freeing the application developers from doing it.

A more plausible reason for avoiding optimistic locking, I suspect, is 
the unfortunate user experience: having an operation rejected because 
the data it was based on is no longer valid. In some cases the 
application can handle this automatically, but often the correct 
response depends on information not available to the computer, and the 
best the application can do is apologize and return the user to the 
update stage, retaining as much information from the failed update as 
possible. That's frustrating for users (and many applications don't 
even go this far, so they provide an even worse user experience), who 
are often unfamiliar with issues of data integrity.

Computer systems that govern relatively intense competition for 
relatively scarce resources - as in Doc's flight-seat-assignment 
example - will always suffer from contention and locking issues like 
this. And, as Doc noted, lock lifetime often has to be limited in such 
systems to prevent resource starvation or misallocation. (The airline 
doesn't want a seat going empty because it was sitting in a lock, 
waiting for an update.) In that sort of environment, it makes sense to 
train the users (airline counter agents) to handle update failures 
gracefully, because the software can't: if a seat becomes unavailable 
while someone's trying to select it, the user has to choose a 
different seat; the software can't select it. There you have the 
optimistic-locking user experience in any case, so you might as well 
use optimistic locking for better concurrency.
