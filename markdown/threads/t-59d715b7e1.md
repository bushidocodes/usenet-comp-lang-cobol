On Unix a file can be opened for reading while it is still being
written. If the file is arriving by, say, ftp and your program is
running from a cron job, or otherwise cycling looking for files that
have newly arrived, then it may be trying to process the file before
it is completely written and thus reaching the 'end' at that time.

If this is the case then try doing a sleep() or somehow mark the file
and do it next cycle.

Yes there is. Programs don't write files, they request the operating system 
to do so.

It IS possible, though highly unlikely, that a program can complete and 
another start before the operating system has finished purging its buffers.

As a diagnostic, I'd have the program that creates the file count the 
created records and log this count. The program that reads the records 
should also count and compare its tabulation to that of the creating 
program. When the second program detects a difference, it should fuss like 
the third monkey on Noah's gangplank.

Try reading the file as soon as it's moved to folder B and see what 
happens;  you want to get as close as you can to the root of the 
problem.  If you get partial files in folder B, then go back to program 
A and insert a delay between closing file A and moving it to folder B.

closing it just to see if this triggers whatever magic data flushing is 
necessary.

what's been written to disk and what hasn't.
