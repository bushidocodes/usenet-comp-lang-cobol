I've started using level 78's. Basically, anytime I'm defining a string
constant, like 78 WS-GERMAN-FOR-YES VALUE "JA". The nice thing about them is
that the compiler won't let you change it's value in the procedure section
which is the way a constant should be treated. The bad thing about them is
that they seem to always be defined as alpha-numeric variables, even if you
define it as say 78 WS-LETTERS-IN-ALPHABET VALUE 26.

Reference modifications work nice when moving into a field where truncation
occurs. For example if you are move alpha data, the move will normally
truncates the trailing characters if the receiving field is smaller. You
could have it truncate the beginning characters or characters at both ends
using a reference modification.

I'm not sure how helpful I can be, but I have been using Level 78
extensively in my programs since I saw examples provided by MF as part of
NetExpress. They are particularly useful when you wish to keep track of
things like error codes or when handling Win95 API calls. The examples that
were provided can be found in a sample directory called apidemo and includes
cobol source code and copy statements which use the Level 78 plus some other
intriguing extensions like type definitions. If you ever have to get into
api coding, you'll need stuff like this.

For now, I'm using it for loading constants into integer or comp-5 elements
when I need to pass stuff back to a VB5 program for error processing and
display. My experience has been that the data is treated according to how
it is used within the program. If I load a number into a number field, it's
treated that way, if a constant is loaded into a string, it's treated as a
string. The stuff seems to work for me.

The following is a short illustration of how this is used for an API call
with the Level 78 representing the literal that is defined in a copy
statement. That way it can always be called the same way. This technique,
btw, represents an approach commonly used in C programming with header
information. If you do the following example, you'll also have to use the
CASE compiler directive, otherwise the dll will not be found.

Hope this helps.

Stan L

78 GLOBAL-MEMORY-STATUS VALUE 'GlobalMemoryStatus'.

01 MEM-STRUCTURE.
02 MS-LENGTH PIC 9(8) COMP-5.
02 MS-LOAD PIC 9(8) COMP-5.
02 MS-TOTAL-PHYS PIC 9(8) COMP-5.
02 MS-AVAIL-PHYS PIC 9(8) COMP-5.
02 MS-TOTAL-PAGE-FILE PIC 9(8) COMP-5.
02 MS-AVAIL-PAGE-FILE PIC 9(8) COMP-5.
02 MS-TOTAL-VIRTUAL PIC 9(8) COMP-5.
02 MS-AVAIL-VIRTUAL PIC 9(8) COMP-5.

CALL WINAPI GLOBAL-MEMORY-STATUS USING MEM-STRUCTURE.

Thane, I use 78's to localize constant definition. For example, I
often set up these fields in a report program:

77 WS-LINE-COUNT PIC 9(03) COMP VALUE 99.
78 WS-MAX-LINES VALUE 55.

Then I compare WS-LINE-COUNT to WS-MAX-LINES everywhere in the program,
and if I need to change it I only need to change the one place. If I
port the program to a compiler that doesn't support 78's I just change
it to a 77, no sweat. But the 78 is superior to a 77 for this purpose,
because you cannot inadvertently modify it.

I find them great for defining maximum table sizes, too. You reference
the 78 instead of using a literal for the highest entry, everywhere in
the program, including the OCCURS clause. You can then easily change
the table size by changing the value of the 78. To convert it to a
compiler with no 78's, just make the 78 a 77 and put a literal in the
OCCURS clause. Again, no sweat.

The way to do this in standard CoBOL is with the REPLACE statement. I
include an imaginary "constant section" at the beginning of my programs:

* CONSTANT SECTION.

REPLACE
==LINE-SIZE== BY ==105==
==MAX-LINES== BY ==55==
[etc., etc., etc.]
.

This lets you portably get "inside" the PICTURE clause and the OCCURS
clause:

05 WS-LINE PIC X(LINE-SIZE) OCCURS MAX-LINES.
