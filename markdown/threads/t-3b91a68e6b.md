In my book on "OldBOL to NewBOL: an IBM COBOL Migration Tutorial" I spent an
entire Appendix on APARs that were "fixed" between VS COBOL II R3.2 and R4.0
(the currently supported release of VS COBOL II).  Most of these shouldn't
cause you MAJOR problems, but you never know.

You have said that you are RES - which is certainly a good first step.  You
don't say, however, if all of your programs are compiled with NOCMPR2.
Although the newer COBOLs still support CMPR2, I would say that you would
want to get away from that as soon as possible.

The biggest issues when moving to LE are your run-time (and of course
link-edit SYSLIB) library are:

1) Make certain that you do VOLUME tests as well as just unit tests.  Some of
the performance issues don't hit you until you try running "real" performance
tests (i.e. online multiple users or "batch runs" with real size input.)

2) Do make certain that you use the RECOMMENDED LE run-time options - and not
the ones that are supplied by IBM as defaults.

3) Do make certain that you read BOTH the LE and the COBOL Migration Guides.
They both have lots of useful information that will help with this type of
migration.

Dynamic allocation of missing DD's actually was introduced in COBOL II
(probably R4). It is a COBOL idiosyncracy, based on the ANSI/ISO standard, but
is controlled by a runtime option; under LE, as Bill Klein pointed out, the
option CBLDQA(OFF) will revert to the previously expected behavior. I forget
how you turn this off for COBOL II running outside of LE.

We encountered a variation of this problem in 1998.  A COBOL II program
had been running daily in production for several months with no
problems.  Then we replaced COBOL II runtime libraries with LE/MVS as
the default.  Suddenly this COBOL program abended.

It turned out that an LE/MVS option had been set to abend if there was
no DD for an output dataset.  The option was set that way to mimic OS/VS
COBOL behavior, because many departments had only recently converted
their batch from OS/VS COBOL to COBOL II.

The JCL for this program had been incorrect for months.  With the COBOL
II runtime libraries the output dataset was dynamically allocated and
deleted at end of job, but nobody missed the file.  They ended up
changing the JCL to add the DD statement, since they decided they wanted
to keep the LE/MVS option.

See William Klein's post for the name of the LE/MVS option.

One other "gotcha" I ran into on my last migration - the JCL, either as a
maintenance reduction measure, or just being sloppy ::)), had larger max lrecls
for variable files than the program code.  This causes an abend, and there was
a single line (among many) in the output that identified the cause.  This
appears to be one of those cases where IBM fixed the code to match the
documentation.  I'm sorry I don't remember the exact abend code (3 years ago),
but if you get a strange abend on jobs accessing variable length files, check
to be sure JCL and program code record lengths match.

If you want to know the current settings, run a COBOL main program and
