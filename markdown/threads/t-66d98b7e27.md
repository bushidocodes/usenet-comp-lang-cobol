There's where your problem is.  In COBOL, your data items must be able 
to be uniquely qualified - that is, if the expanded code looks like this...

FD INP-FIL.

01  NCB-REC.
     [more stuff]

FD INP-FIL2.

01  NCB-REC.
     [more stuff]

...you have no way to uniquely qualify the data areas.  To accomplish 
what you're wanting to do, you have a few options...

1)  Use one FD, and use it to process the files one at a time.  If you 
have sorting requirements on your output file, you could build an 
indexed file that's sorted on the required sort criteria; or, if you're 
required to output flat text, just sort the output file when you're done.

2)  Use COPY REPLACING to change the 01-level items in the copybook.  If 
NCB-REC is the data item (I'm guessing at that, as you didn't say what 
was in the copybook), you could do something like...

FD INP-FIL.
COPY BCBSECR REPLACING NCB-REC BY NCB-REC1.

FD INP-FIL2.
COPY BCBSECR REPLACING NCB-REC BY NCB-REC2.

With this, you'll need to qualify your references to the key in the 
select statement, and anywhere in the program where it's referenced.

If I were doing this, I'd probably go with #1.  But, your requirements 
may mandate #2 - if so, it can be done.  One of the advantages I see 
with #1 is that you can code a simple "read and dump" loop, and perform 
it iteratively on the five files.  Of course, if you're just combining 5 
files, MERGE may be a better solution.

My advice would be to look up COPY, qualification, SORT, and MERGE in 
your reference manual.  See what the rules are for each one, and make a 
decision that works for you.

> Help required.

Just a quibble - that should be "requested".  You require help to get 
through the problem, but you're requesting help from us.  :)

You need only one FD for the five input files, not five. Make the file name variable in
the select, then do this:

move 'file 1' to external-file-name
open inp-fil
(copy file 1)
close inp-fil
move 'file 2' to external-file-name
open inp-fil
(copy file 2)
... etc.

If you are MERGING the five input files (you didn't say so), it would seem that you need
five FDs. Not really. You can code one inp-fil FD and launch five instances of the
program, each in its own thread. For good measure, launch a sixth thread for the output
file. Doing to requires making the program thread-safe and managing communication between
main and the six programs running in threads. 

Do that and it will run much faster. You'll get an A+ for sure.

Unless a merge is, for some strange reason, prohibited,
do it anyway because it is the easiest COBOL solution to
implement without "copy-and-paste".

The next easiest would seem to be using SAME RECORD
AREA for all five input files and maybe the extract file if
complete records are extracted. Five "copy-and-paste"
loops, one for each input file. And a common test-write
routine. SAME RECORD AREA permits the test-write
routine to use the same file-name qualification regardless
of which input file is being processed.

After that, all-COBOL solutions seem to become less
appealing and JCL solutions more appealing.
