Chris,

The clue is in your "ASSIGN TO 'TWNSHP.TXT'". You have used an editor to
create a text file which *automatically" contains CR/LF, (x'0d0a')  after
each record. The concept harkens back to ye oldie typewriters - get to the
right of the page, Carriage Return and then lever for Line Feed <G>.

Your definition should be :-

FILE-CONTROL.

select             TOWNSHIP-FILE
assign             'TWNSHP.TXT'
organization    LINE SEQUENTIAL
access            SEQUENTIAL
file status        ws-fileStatus.

and I'm assuming you have in Working-Storage :-

01 TOWNSHIP-REC-LENGTH. pic 9(04).  *> use a 'big' number and you are

playing it safe.

If you have the Animator you can check each READ and you will see the
'actual length' stored in Township-Record-Length.

Note that LINE SEQUENTIAL is the format from text editors. SEQUENTIAL with
NO CR/LF is typically where you are initially WRITING transactions from
WITHIN  a program to a file and then you can subsequently READ that file
sequentially. You can't inter-mix SEQUENTIAL and LINE-SEQUENTIAL

Sometimes the 'obvious' can be difficult to anticipate and describe even
in Help files - part of the learning curve, I'm afraid <G>

Hi Chris,

You can try redefining:
01 TOWNSHIP-RECORD      PIC X(30).

as:
01 TOWNSHIP-RECORD.    
     05  TS-REC-BYTE  OCCURS 30
           DEPENDING ON
           TOWNSHIP-REC-LENGTH            
                                     PIC X(001).

This should clear up the problem. After the READ, TOWNSHIP-REC-LENGTH will
contain the true length of the record just read.

When you want to write a Variable rec move the desired length of the rec to be
written to TOWNSHIP-REC-LENGTH before the WRITE stmt is executed.

Using the same format as Chris, has allowed me to write FOUR programs in OO that
will access ALL files, ISAM, Relatrive, Sequential and Line Sequential. Each of
the classes looks like my FD Data-File above, with one exception - the assumption
is that the PrimeKey for an ISAM is in the first 20 positions; you could make it
50. (The idea came from Donald Tees and Pete Dashwood has a variation). Your
Primekey might be 6 or 12 characters - just fill the rest with spaces. So for ISAM
it reads as follows :-

FD  Data-File
    record varying from 20 to 1500 characters *> compiler tells you it *must* be a
minimum of 20
       depending on ws-RecordSize.
01  Data-record.
        05 Data-PrimeKey          pic x(20).      *> This is what causes the
*must* above
        05                                   pic x(1480).

That's all the File classes need. The *real* record definition is held in the
Business Logic class as a copyfile and is passed to the File Class with current
record length, as you suggest, or it returns the 'generic' Data-record above from
a READ.

Yes I know, above doesn't cover Alternate Keys - haven't got that clever yet <G>.
Besides I'm switching to SQL, so it's hardly a priority for me  !!!

I haven't tried it, but looking at M/F file handling examples I believe the same
technique is available outside of OO - providing you pass the full path/filename
and for fixed-length records you pass the max record length - but you have to
create the files INITIALLY as VARIABLE, (even if they are truly fixed length),
otherwise it wont work. The only snag in non-OO is that given you want to access 3
ISAM files you can't call the same ISAM program - depending upon how many you have
in a Business program, you would have to rename the ISAM, and compile for each,
so that you can call ISAM1, ISAM2, ISAM3 etc.

Compare that to OO where I could have 20 INSTANCES of ISAM Files in one program
but only the ONE Class ISAM File in memory.

Chris,

A tip about learning. Each to his own style of course. You didn't say which Micro Focus
compiler - but if you have Net Express or the University Edition then the following apples
:-

I find the standard Windows help feature irritating, you find yourself hoppity-skippty
jumping all over the place, nothing flows in a logical topic sequence. Both Fujitsu and Net
Express offer some excellent On-line books as well as the standard On-line Help. Net Express
has two specific books - Language Reference Manuals 1 and 2. Bit of a pain but worth
printing out.

I taught myself COBOL from scratch using the printed RM/COBOL manual produced for Radio
Shack. Hadn't a clue, (1) Quickly skimped through it, (2) Then a second slightly slower
read, and (3) a more in depth read of topics which seemed immediately pertinent. Did the
same as I switched to Microsoft and Micro Focus COBOLs.

That's how the legal sharks operate. Shelves and shelves of legal tomes - they haven't
absorbed the contents but they build up a memory index so they can zero in on the right
shelf and the correct book, knowing that the topic is there 'in the second part of the book,
somewhere....'. For their current client they are now armed with precedents from the court
case of A versus B etc.

A Language Referenced Manual is a good starting point, followed up by extras that the vendor
supplies as additional features. Your 'memory index' triggers a reference to the topic,
which you can refresh by reviewing from the book, or immediately jump to the on-line Help
for quick examples.

Bill Klein's COBOL FAQ available at cobolreport.com, makes reference to some sites where you
can get examples. Check out following, some are vendor specific, but still worth a look for
ideas  :-

University of Limerick - http://www.csis.ul.ie/COBOL/examples/
Fujitsu - http://www.adtools.com/student/samples/index.htm
Liant (RM/COBOL) - http://www.liant.com/support/code/
Micro Focus - http://www.syntax.gr/departments/CobolSolutions/files/downloads/demos.htm

Best of luck in your programming.
