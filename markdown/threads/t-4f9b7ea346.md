Todd sometimes some coders use Low-values to fill a field which will not be
displayed, especially in CICS I had this problem with dates which are
normally described as alpha numeric.  I would describe the field as alpha
numeric and have a redefine as numeric.
Use the redefines to test for a numeric condition
If <var> <nemeric one> is numeric move to <numeric field> else move zero to
numeric field.

As a warning you may be encountering incompatable data or data that is
corrupt somehow.  You could actually move the key and the values in question
to a table and print them out as alphanumeric and skip over those records.
Then you could evaluate what exactly is in those fields.  Possibly somehow
someone is entering data that does not make sense. There is a lot to be said
to inspect what exactly is in the field.

Also sometimes data may appear to be alphanumeric in a key or other field
which is actually packed data described as numeric.  Programmers sometimes
use this technique to use the smallest possible amount of space.  For
instance I have seen a SSN that is packed described as an alphanumeric.  It
would occupy 5 positions in an alphanumeric field but actually be a 9 digit
numeric SSN.
To convert you have to redefine as pic 9(09) comp-3, move the comp-3 to a
pic 9(09) normal numeric field.  SSN is normally displayed as alphanumeric
so we would redefine as alphanumeric.  Sometimes in the past we even turned
the ssn around so it would sort better as an alphanumeric field then packed
it and called in alphanumeric.

All of this is done to make the key as small as possible so you can search
through the key-fields as fast as possible.  I don't know if anyone else
does this anymore or not.  Storage costs may have gone down enough so that
it is not necessary.  I would set a trap in the program to either stop the
program and display the bad data, or to make a table of the bad data (with a
table size limit) to print out the data.  Often use counters to count the
number of bad data items so the programmer knows how many times bad data is
actually encountered and skip over it.  It all depends how many there are;
if less than say 100, you could just print out on the job control what is
actually in the field, with the record key, so you know where to go and
correct the data.
