```
Simple answer is "no" there is no way to GUARANTEE that what you want to
happen will happen.

Practical answer -
  If you set a switch to indicate that you want to "get out quick" - Micro
Focus supports the "EXIT PARAGRAPH" statement and you could test that switch
and when it is set, exit each level of the PERFORM structure.

Possible answer -
  If you never use another "go to" to enter one of those paragraphs again
(but instead only enter them thru EXPLICIT PERFORM statements - and never
"fall-thru") then you probably will NOT "notice" that you haven't reset the
PERFORM buffers.  ON THE OTHER HAND, this is exactly what is meant by
"spaghetti code" and will be a nightmare to maintain.
```

```
Binyamin,
  UNFORTUNATELY, COBOL (or at least the ANSI/ISO Standard) has no such
requirement.  There is a requirement that when you EXIT PROGRAM from a called
subprogram, that PERFORM ranges are "re-set" - but there is no such rule for
when you "GO TO" out of a PERFORM RANGE.  Worse yet, there is a significant
amount of (truly UGLY spaghetti) code that actually "relies" on this.  EVEN
WORSE YET, this isn't either required OR prohibited, so the actual run-time
behavior of different compilers MAY be different in such cases (i.e. same
soruce code - different run-time results).
```

```
The EXIT statement _always_ does nothing.

The end of a PERFORM is not controlled by an EXIT, but by the
presence of the next label of the same type (ie section or
paragraph).

A GO TO out of a PERFORM range may, or may not, lead to
complete loss of control of the program or of the next
maintenance programmer.
```

```
If there is an instance for "bailing out" of the perform several levels
deep, I would say that it's time to rethink the logic for doing those
nested performs.  Setting switches to do this is messy and inefficient and
leads to maintnance headaches(especially at 2 a.m.)

1.  Changing the nested PERFORMs to serial PERFORMs.
2.  Change the logic to add a special routine for this new instance.
3.  The program was not designed for this new instance.  Maybe it is time
for a redesign.

I have seen very few programs that have required nesting past 2 or 3 levels
deep.  More time in the design phase can lead to less time in debugging and
maintaining.  Modularity of design also makes things better.  I have
encountered few programmers that understand structured COBOL.  Most of them
were enamored with GOTOs.  To them, Top-down meant you started at the top
and ended at the bottom.

I have been programming in COBOL for over 20 years.
```
