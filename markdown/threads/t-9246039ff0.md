IMHO...

VB.NET (YES) for the same reason that COBOL has survived. Lot's of
lines of code existing in production applications. That includes the
implication that many VB6 applications are currently being converted
and re-written to be VB.Net.

C# (YES) simply because Microsoft has promoted it so heavily as part
of their .NET product introduction. For those that have made the
switch over to C#, I often hear tham comparing it to programming with
Java (but without the J of of course).

COBOL.NET (YES) Starting with Fujitsu (the compiler I used while
writing my book), they have made a major investment into their product
- and it actually works well. In my opinion, to see Micro Focus come
along (afterwards) and produce a COBOL.NET product as well sort of
validates the need for this type of product. In other words, why would
Micro Focus join the "provider of a COBOL.NET compiler" group if
Fujitsu had made a mistake in doing so in the first place?

C++.NET and Delphi.NET - Sorry. I have no opinion for these.

J# - My guess is NO for this one. Other than the on going legal issues
of Microsoft using the letter "J" in any programming lang, there
simply is no real compelling reason to use J# when you have C# (and
Java) as choices.

How could all of these languages coexist? Simply, the programming
world is big-enough to support them all - really.

Once getting into .NET programming (deep enough), it really is just
about the Framework and the Common Language Runtime. What language you
happen to use in order to instantiate one class or the other really
becomes a secondary issue.

Perhaps, a shake-out for the various Programming IDE's may be another
question. In my opinion, Microsoft's Visual Studio.NET offers huge
advantages. So, as other vendors come along with their own IDE's, I
think that we will see more of a fight-for-survival in this arena.
Keep in mind, that several vendor's display of .NET support includes
the featured advantage of integrating their language compiler into
Microsofts's VS.NET IDE. An example of this is Fujitsu's COBOL.Net (or
rather NetCOBOL for .NET product). Yes, Fujitsu has their own IDE.
Additionally, Fujitsu's NetCOBOL for .NET product integrates
seamlessly into Microsoft's VS.NET IDE.

All just opinions of course...
