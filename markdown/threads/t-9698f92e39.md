The BASIC programming language never used line numbers
(or sequence numbers). The 10 and 20 in your example are
labels, the same as in FORTRAN.

Some implementors of BASIC interpreters, most notably
Microsoft, perverted the language by adding a requirement
that each line have a sequence number that also served as
a label. This did not follow the intent, or the implementations,
of the language designers, John G. Kemeny and Thomas E.
Kurtz.

[Microsoft corrected their perverse implementation of the
BASIC language in their compiled QuickBASIC and
interpreted QBASIC products.]

Kemeny and Kurtz founded a company to produce and
distribute True Basic, a product which did follow the intent
of the language designers! See < www.truebasic.com >.
Their early (1983) product only required numbers when used
as labels. Their current product is a multi-platform, compiled,
structured (modular) BASIC.

Rick Smith

This is not necesarily a contradiction. It is a difference in
the understanding of what constitutes BASIC at different
times. I am saying that BASIC never used line numbers;
but BASIC interpreters did. I am also saying that the use
of line numbers was not a requirement of the language;
but was a requirement of the interpreter. I am also saying
that by requiring line numbers, the interpreters were not
*really* implementing the BASIC programming language
as conceived by its designers.

This is similar to disagreeing as to whether implementor
extensions to COBOL are *really* COBOL. The difference
is that the COBOL standard does allow such extensions;
but there was no BASIC standard during the time in
question. The absence of a standard allowed anyone to
create anything as a language and call it BASIC.

I understood that some, if not most, BASIC interpreters,
through the 70s and early-80s, did use line numbers; but
the original BASIC from 1964 was compiled. It is my
understanding the compiled version did not use line
numbers.

I came by my understanding from a reference with a quote
from either Kemeny or Kurtz that, as I recall, specifically
pointed to line numbers as something horrible that was
done to the language.

I was unable to find that reference in a quick search.

I did find the following quote from
< http://www.digitalcentury.com/encyclo/update/BASIC.html >.

"Kemeny and Kurtz had gone on to other things and were not
actively involved in the maintenance of BASIC. In 1983, they
used a PC for the first time and were surprised at "�how
terrible were the crude 'street' versions [of BASIC] that high
school and college students all over the country had to use."
Appalled at what had happened to their language, Kurtz
became chairman of the ANSI Standard BASIC committee,
and in 1984, he and Kemeny introduced their own version,
called True BASIC."

Although this quote does not specifically point to line
numbers, it does give a sense that the originators of
BASIC were very unhappy with versions that did have
line numbers. Again, I recall seeing a comment against
line numbers.

So the problem is, "At the various times in question, what
consituted the definition of BASIC?" There may not be
a satisfactory answer, prior to 1987, since the language
was not protected. As of 1987, the language was defined
by an ANSI standard and later by an ISO standard.

Finally, the language you knew, at Dartmouth, as BASIC
may have been BASIC Sixth or some other such number.
------------------
Rick Smith

For short strings, where the maximum length is known,
C is adequate; but then so is COBOL.

For longer strings, in C, finding the length requires,
typically, one instruction per character. Also using dynamic
memory to work with long strings is independent of the
string handling functions. This means that manipulation
of C type strings can be slower than manipulation of
BASIC type strings.

For BASIC type strings, the length is part of a control
block, so the length can be accessed quickly. Also it
should be understood that the implementation of BASIC
type strings includes the necessary dynamic memory
operations.

Some of the elementary operations for manipulating
BASIC type strings are present in the draft standard and,
that which isn't, could be implemented as user-defined
functions. I just think it would be much more useful if it
were built in to the language.
------------------
Rick Smith
