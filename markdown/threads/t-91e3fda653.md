Richard:

You are absolutely correct.  We advocate designing the user interface
according to the task to be performed.  For example:

If the application is an "inquiry-oriented" type of system in which
there will be minimal or no data entry, mouse-based or keyboard-based
navigation is fine.

BUT....if the application is heavily data entry oriented, we recommend
that the programmer consider disabling the mouse if the application is
to run in a Windows GUI environment.  It a very simple process of
setting a switch from the program and can even be done on a "user id
by user id" basis.

The end user learns quickly that to operate the application, the
fingers must stay on the keyboard.  If the end user is operating the
application constantly during their working hours, they may not need
the mouse.

I'm curious if VB or Delphi provide a mouse switch to turn a mouse on
or off for an application or a specific end user.  Anyone know?

It seems to me that this is a fundamental option to allow the
programmer to create an ineffective interactive business data entry
program or an effective one.

Like you said....for graphics design work, desktop publishing or a
program in which dragging and dropping objects can be utilized, a
mouse can be a very nice tool, but for some business-oriented
programs, it can really get in the way.

Unfortunately there are two criteria that must be met by every packaged
application.  It must, of course, meet the needs of the users who will
actually work with it, but first, in order to get to that stage, it must
be bought (or sold).

The person making the buying decision is often not one of the users that
will work with it.  This means that the application must be targetted at
the buyer.  To make it so that he will buy it it must be instantly
usable, ie so 'easy to learn' that "even an accountant can use it"*.

This usually means GUI, mouse, and 'click-around to see what happens' to
make the buyer feel that he could actually work out what is gouing on
and thus feel talented.

In order to meet the criteria of being sold _and_ doing useful work an
application requires _two_ different user interfaces.  However, most
software houses that rely on sales recognise that the only actual
criteria is getting the product sold.  In fact if the application is not
the most effective in use then additional licences will be required plus
more training and more chargable support.

Just tell me if I am being too cynical.


(* Actual quote from a sales brochure)

I don't think that you are being too cynical.  I agree with you that
the Windows UI is highly overrated when it comes to "ease of use" and
in many cases a poor alternative to a well-designed character mode
user interface.  Too often, the first criteria in the buyers mind has
nothing to do with the quality of the application being purchased.

But, our philosophy is that the user interface should is written so
that mouse clicks on icons/pushbuttons be assigned corresponding
function keys which perform the same control functions.  This makes it
is possible to write a single application which allows an
administrator type person to configure the application to turn the
mouse on or off for an end user.

The marketing guys at the trade show booths can click around to their
hearts content in order to sell the application and the administrator
can shut the mouse off so that the employees can get a reasonable
amount of work accomplished.
