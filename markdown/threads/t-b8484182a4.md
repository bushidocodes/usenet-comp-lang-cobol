I confess I have not read the standard recently, but from a practicality
standpoint I think it highly unlikely such a requirement would ever be
accepted into any standard for the language.  To require commitment to
disk for an individual WRITE statement would disallow the use of
buffering and blocking techniques by the underlying operating system,
and at least for mainframe systems could only be implemented at the cost
of severe performance degradation.  I believe successful completion of a
CLOSE is the only point at which commitment of all output records to the
actual device is guaranteed.  It would be nice if error recovery at
abnormal termination could always resolve things and write out any
pending records, but even in environments which might attempt to do
this, the overwriting of file control information by a berserk program
could make it impossible.

>What does the current standard actually say about the WRITE statement

The execution of the WRITE statement releases a logical record
to the operating system.

It is not possible for the COBOL standard to specify how the operating
system will treat logical records.

>In the implementation I am working with, it is clear that the WRITE
statement
>does not commit the data (to disk) at the time it is executed, which is a
pain
>when trying to follow debugging logs.  Is the implementation following the
>standard?

It is most likely the implementation is following the standard. Even
if there were some minor technical violation, it is not likely that
complaining about it would solve "your" immediate problem.

There may be an option for the COBOL system you are using,
which option will allow unbuffered writes; if unbuffered writes are
allowed by the operating system.

For example, with Micro Focus, the directive is WRITETHROUGH.

As always make sure you understand the effect of using the option.
In this case, forcing unbuffered writes may negatively affect the
performance of the application.

I can't address what is contemplated by the standard committees;
only on what they have published, that I know about. I am also
free to be very wrong in my comments! ;-)

Within the area of file access, the biggest changes will be in the
addition of file and record locking. In the case of OPEN OUTPUT,
the operation will continue to be as you describe since the file
locking will be exclusive. There are some areas of file and record
locking that are under review; hence it is not possible to know,
at this time, what the final standard will be.

Having said that, it would appear that one file connector could
be opened:

OPEN EXTEND SHARING WITH ALL OTHER file-name-1,

while another file connector could be opened:

OPEN INPUT SHARING WILL ALL OTHER file-name-2.

In this case, it may be possible for the second file connector to
read the records written by the first file connector; however, this
may also be platform dependent. That is, it may not be available;
or it may be possible to read, only those records that were available
prior to adding records, with file-name-1; or it may be be that
all records in the file will be available to file-name-2.
