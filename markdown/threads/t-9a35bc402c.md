It is an Insertion characer. Allows whatever follows to be treated as an
insertion to the picture.

> 2) The programs use a verb called TRANSLATE. This verb seems to be
> implemented on ICL only. What does the TRANSLATE verb do?
>
It allows one character set to be mapped against another. The ICL machines
had a 24 bit word mapped as 4 x 6-bit characters. This meant that the 6-bit
characters needed to be mapped against incoming 8 bit EBCDIC, and vice versa
if it was required to write standard EBCDIC data.

Back in the days of the Great Byte Wars (circa late 1960s) it was touch and
go whether the Industry would adopt a 6 bit "byte" or an 8 bit "byte".
Control Data, NCR, and ICL all  used 6 bit characters as part of a  64 bit,
24 bit, and 24 bit "word", respectively.  ICL had a project to convert their
24 bit architecture into EITHER 4 x 6-bit bytes or 3 x 8-bit bytes (pretty
smart, huh? <G>) but it was scrapped as too expensive.

Eight bit bytes won the day (partly because they were backed by the might of
IBM, but probably more because they provided a 256 character set, as opposed
to a 64 characte set that required ESCAPE/CONTROL codes for anything
esoteric.)

TRANSLATE did the mapping.

________________________________________

The TRANSLATE statement (C2 only)

1 Function
The TRANSLATE statement enables data to be translated from one character
set to another.

2 Format
                              -                   -
                             |     {ISO         }  |
TRANSLATE {identifier-1} ... | TO  {EBCDIC      }  |
                             |     {identifier-2}  |
                             |     {literal-1   }  |
                              -                   -

3 Programming rules
  Identifier-1 ...
    Identifier-1 ... must reference an elementary item with implicit or
    Explicit usage DISPLAY or a byte aligned group
    Identifier-1 ... and elementary subfields of identifier-1 ... which
    have implicit or explicit usage DISPLAY may be described by OCCURS
    clauses. If the DEPENDING option of OCCURS is used, the TRANSLATE
    statement will act on the current size of the field in question and
    not on its maximum size
    Identifier-1 ... must not have level-number 66
  Identifier-2
    Identifier-2 must reference an unedited elementary field with
    implicit or explicit usage DISPLAY or a byte aligned group field.
    It must not reference a non-byte aligned group field
    Identifier-2 must not have level-number 66
  Literal-1 must be a nonnumeric literal. It must not be a figurative
    Constant or an ALL literal
  If the TO clause is omitted, the effect is as though TO ISO had been
  specified

4 Execution of the TRANSLATE statement
If the TO phrase is omitted or if the ISO option is used,
identifier-1 ... is assumed to be in EBCDIC. If the EBCDIC option is
used, identifier-1 ... is assumed to be in ISO. In both these cases,
the TRANSLATE statement causes each character in identifier-1 ... to be
replaced by the corresponding character of the other collating sequence.
If the identifier-2 or literal-1 option is used, the TRANSLATE statement
Causes each character in identifier-1 ... to be interpreted as a bit
pattern (value N, say) and then to be replaced by the (N+1)th character
in identifier-2 or literal-1 . If any character in identifier-1 ... has
a value N such that (N+1) is greater than the number of characters in
identifier-2 or literal-1 , the effect is unpredictable.
If identifier-1 ... is a group field, each of its elementary subfields
With implicit or explicit usage DISPLAY will be translated as above
except that the following will be unaffected:
  Elementary items containing a REDEFINES clause
  Elementary items subordinate to a group item containing a REDEFINES
  clause where the group item is subordinate to identifier-1 ...
  Elementary items described as FILLER items
If the storage occupied by identifier-1 ... and identifier-2 overlaps,
The result will be unpredictable.

5 Efficiency considerations
The following recommendations apply when using the identifier-2 or
 literal-1 option:
  The number of characters in the translation table specified by
  identifier-2 or literal-1 should be either 256 or (N+1), where N is
  the highest value of any character that may appear in identifier-1 ...
  For example, if identifier-1 ... can contain any character in the VME
  character set, then the highest value of any character that may appear
  is hexadecimal FF, which is 255 in decimal, and so the translation
  table should be 256 characters long
  The translation table should not contain any uninitialised positions.
  If some positions in the table represent characters which cannot
  validly appear in identifier-1 ..., then these positions should be
  assigned some standard error character (for example, SPACES or
  HIGH-VALUES or ALL "?"), so as to avoid any problems that might
  otherwise arise in the event of identifier-1 ... containing invalid
  data If some of the replacement characters in the translation table
  do not have graphic representations, then identifier-2 can be
  initialised either by reading it in from a file or by describing
  any such positions in the table as 8-bit bit items and initialising
  them with numeric initial values. For example,
01   TRANSLATE-TABLE.
     02 T-1 PIC X(7) VALUE "ABC*%56".
     02 T-2 PIC X(10) VALUE SPACES.
     02 T-3 PIC 1(8) COMP-5 VALUE 49.
     etc.
