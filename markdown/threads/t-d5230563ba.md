Not familiar with "AcuODBC", but one of the greatest sources of slowdowns
i've seen with ODBC software in general is if the software does an
SqlExecDirect for each row instead of doing a PREPARE of the statement,
binding the parameter(s) and then just EXECUTE'ing.  I've also seen software
(e.g., Mercator mapper) which opens and closes the database connection each
time it executes.

These are both performance killers of which Quentin Tarentino would be
proud.

If "AcuODBC" is a generic tool, I'd doubt that eiher of these
situations -especailly not number two) exist, but it's probably worth asking
about.

Also, you may want to see if you can disable transaction support... since it
is a conversion, it probably is not terribly useful to carry the overhead
necessary to be able to ROLLBACK each INSERT, as you can always "start over"
with empty tables. Maybe there is some kind of option you can set for this
in AcuODBC.

A few suggestions:

- If your database tables have any indexes, drop them before loading the
data, then recreate them after it is loaded.  This is a good idea no matter
how you are loading the data.

- Look into using DTS (Data Transformation Services) in SQL Server.  I have
not used it with AcuODBC, but it should work.  You can create a DTS package
that will read data from the Vision files (using AcuODBC) and store the data
in SQL Server.  You can script the whole thing and even schedule it to run
automatically.

- The vendor's suggestion of creating export files is not a bad one.  If you
can get the data into delimited files, ideally that mimic the table layouts
exactly, you can execute the SQL Server "bulk insert" statement or the bcp
utility, which loads data into SQL Server very quickly.  (One tip - with the
"bulk insert" statement, the filename you reference needs to exist on the
system were SQL Server is running.)

It shouldn't matter whether there are 2 or 500 files - if you have a single
program that creates all 500 files, you can set up a single SQL script with
a "bulk insert" statement (or a batch file with "bcp" commands) that will
load all 500 files.

If the speed issue is in AcuODBC, then only Acucorp is going to be able to
answer that.  Try support@acucorp.com - they are usually pretty helpful.

My only other suggestion might be to determine where the speed issue is - is
it really with AcuODBC or is it with the combination of AcuODBC and DTS.
(Note that when you are using DTS, it is not doing a bulk insert/bcp.)

Look on www.hotfiles.com and do a search for "ODBC export".  There are
several utilities there (free and not free) that will allow you to create a
delimited file from a table through ODBC.  Download one and time how long it
takes to export data from the large Acucobol file (through AcuODBC) to a
delimited file.

If that's relatively quick, then you have the beginnings of a possible
solution.  Find an ODBC utility (or write one yourself - see below) that
lets you script your exports and use that to create the export files, then
script the imports with bcp commands or "bulk insert" commands through Query
Analyzer.

There are a variety of ways you could write the export portion yourself.  If
you are handy with C, that would be the fastest way in terms of runtime
speed.  It's not that hard if you know C and the ODBC API (I have utility
that I wrote in C that exports data from ODBC to delimited files, but it's
part of a larger system and wouldn't be of much use if I posted it - and my
employer owns the code, so I probably couldn't anyway).  But there are
several examples on msdn.microsoft.com that use the ODBC API to get data
from a "select" statement - all that is needed is to add code to open a file
and write the data with delimiters between the fields to the file.

The other ways that come to mind are to use Visual Basic or VBScript using
the Windows Scripting Host (wscript or cscript) and ADO to do it (again, you
can find examples that would provide the basis for this on
msdn.microsoft.com).  Or use perl with the DBD:ODBC module if you know perl.

But the key would be to find out if you can get data out of AcuODBC quickly
because you know you can bcp/bulk insert it into the database quickly.
