```
I have a COBOL program which uses very large arrays in memory.  When compiled 
on my laptop with MF Professional and run with a certain volume of data drawn 
from MS Access, the program executes in a reasonable amount of time.

When that same program is compiled on a state-of-the-art NT server with 
NetExpress and is run against an Oracle database, the time taken to run 
increases 10x.   Only by decreasing the array sizes by almost half, was I able 
to get the program to run in an acceptable - though not favourable - amount of 
time.

Decreasing the array size is not a long-term solution I would like to persue.  
Is there something about the NT operating system and how it deals with file 
swapping?  Or is my problem somewhere else?  Any suggestions would be very much 
appreciated.

Lynne
```

```
Maybe the problem is in the difference between how Access(r) and Oracle(r)
handle SQL statements which return multiple rows. ( I assume this is why you
are able to vary the array sizes with impunity).

In  this case, all you can do is "tune" the table size based on  the
specific database  unless the DB vendors provide some performance
guidelines.

As another thought, the network software in case #2 may be the limiting
factor - you may need to tune your SELECTs to keep network traffic and 'data
block' size to what the network software likes. But even here, you get to
'tune' your software, and I know of no real 'rules'.

I would not blame the application software; although here, too, Net Express
may have some buffer sizes you can tune. (I'm not an NE developer).
```

```
I had similar problems when developing for SQL Server, when my support team
came to me and said "This takes far to long to run!", after investigation I
discovered various affecting factors. The first thing to do was to run a
trace to ensure that the commands issued in my COBOL code where reaching the
server and being run and then isolatting the command that took an extended
amount of time.

In my case it so happened that it was the support team at fault.....they had
copied the database and dropped all the keys to the tables, so SQL Server
was scanning the complete table  (seen in the trace in the trace).

Another thing to watch out for is cursors.....when setting cursors in SQL
Server, it reserves memory for all the colums in your cursor statement and
for the length of the complete table.

So I suggest do a trace and work back from there.
```
