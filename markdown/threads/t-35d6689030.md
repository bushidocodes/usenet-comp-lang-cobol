```
Greetings to All,

I'm attempting to write some file management DLLs in Fujitsu Cobol 3.0
(32-bit).  Towards that end, I've been teaching myself and perusing the
online doc and trying different things.

CA-Realizer is Computer Associates' answer to MS Visual Basic.  Unless
something drastic happens at CA, it will never be a strong competitor to
MSVB, but it's what I've got, so I use it.

I've created two different DLLs.  The first one doesn't receive any
parameters, but is called by the Realizer program, DISPLAYs a string, then
exits the program.  This one works fine.

The second one is setup with STDCALL linkage and a 2-byte binary field (PIC
S9(4) COMP-5).  I am trying to call it in the same manner as the first DLL,
passing it a 2-byte integer, and it's causing CA-Realizer to barf all over
the place.  I've tried it with PASCAL linkage (which reverted to STDCALL
linkage) and with C linkage (which compiled OK, but still didn't work).

I'm not using the Windows interface to do the compiling, instead, relying on
a .BAT file to automate the process as much as possible.  I'm an old DOS guy
and find the command line more appropriate to writing programs.  But I may
be leaving something out of the .BAT file that I'm not aware of.  I don't
know what it could be (obviously).  I've copied the example in the online
doc directly and, as I said, the first DLL worked.

I've also taken a quick look at http://www.adtools.com and it simply
directed me here.

An obvious answer is that I'm doing something wrong in the CA-Realizer call.
I've done these calls before, but only with the Win32 API functions and
PowerBASIC DLL code.  I'm continuing to look in that direction, but I don't
think that's necessarily going to be successful.

Has anyone had any experience with this (admittedly marginal) endeavor?  Any
advice to give?  Words of encouragement?  If anyone would like me to post
code, batch files, other information, I will gladly cooperate.

TIA,

Paul
```

```
Thank you both Juergen and Sff5ky...

Juergen, your little note jiggered something in my mind.  The example with
the compiler is for VB and shows

    CALL MYCOBOL( vbInteger, vbString )

In the DECLARE, the vbInteger is passed BYREF (as the default) and the
string BYVAL (explicitly defined).  I began playing with the Realizer CALL
and what I ended up doing was define the EXTERNAL as

    (POINTER(INTEGER), POINTER)

which did what I wanted.  It had been

    (INTEGER, POINTER)

Changing the EXTERNAL from the second to the first, the COBOL DLL
successfully modified the integer (first parameter) and the string (second
parameter).  And my system didn't crash either, which is always a good sign.
I might would have figured it out eventually, but thanks to your shaking the
box, it was a lot sooner.

Thanks again for your help.

Paul
```
