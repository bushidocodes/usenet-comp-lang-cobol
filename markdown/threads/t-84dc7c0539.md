The letter O is EBCDIC for 6 with a negative sign. The
program most likely needs the ASCII equivalent character.

After reading the record and before using the data, execute
the following instruction for each signed (numeric-display) field.

INSPECT field-name CONVERTING "}JKLMNOPQR"
TO "pqrstuvwxy".

This will convert ASCII character, EBCDIC signs, to ASCII
character, ASCII signs.

Micro Focus is sensitive to this.

HTH
------------------
Rick Smith

With READ INTO, the data you are using is in working-storage.
Use the field-names from the record description in working-storage
in the INSPECT statements. Place the INSPECT statements
immediately after the READ INTO.

It is also possible to change the READ INTO to a combination
of READ and MOVE. In that case, the INSPECT statements
would be between the READ and the MOVE and would use
the field-names of the record description for the FD.

Also, as Michael Mattias mentioned, change the right parenthesis,
")", to a right brace, "}".

HTH
------------------
Rick Smith

To summarize previous threads related to this topic;

For IBM Mainframes,
    Thou Shalt NOT Manipulate Data In File-Buffers.
For Other ANS-85 COBOL,
    Thou Shalt NOT Manipulate Data In File-Buffers,
        While The File is Closed.
For Micro Focus COBOL,
    Do Whatever You Want With Data In File-Buffers;
        Just Don't Expect The Whatever To Be Portable.

Are there good and decent reasons for accessing the data
in file buffers directly? Yes!

Given that all computer programs consist of code that
implements concepts and that these concepts are
implemented in different ways depending upon the
language used; then the statement "READ file-name INTO
record-name" implements the concept "Get record into
record-name.", directly. Other concepts might include:
"Get corrected record into record-name.", "Get transformed
record into record-name.", "Get subset of record into
record-name.", etc.; such that the data in record-name is
precisely that which is required for the other concepts that
are implemented in code.

Each of the other concepts may be implemented by
accessing the data in the file buffer, directly. They may also
be implemented by incorporating the additional step of
reading the external record into working-storage and by
transforming or manipulating the data within the internal
records. The only precaution one should take before
accessing an input buffer is that the record was sucessfully
read. (I did not mention this in my response to Pat.)

Since data, successfully read, may be accessed from and
manipulated directly in the file buffer; whether to do so is
a matter of style or preference. I find that using the buffer
directly is clearer than moving the data to working-storage.

The above comments apply equally to WRITE and WRITE
FROM. If file status indicates the record buffer is available
use it directly if it pleases you to do so.

This is an "IBM Thing".  Because (correct me if I am wrong, IBM'ers :)
IBM COBOL compilers don't actually assign a separate record area, they
use pointers into the actual file buffers.  On input, the record area
isn't 'available' until after the file is opened and a record read.
On output, the record area is not available until the file is opened,
and the contents of the record just written is not available in the
record area, because the pointer has been repositioned to the next
record position in the buffer.  I can see only two reasons for their
'only use READ INTO and WRITE FROM' obsession.  One is they don't seem
confident that IBM COBOL programmers can be taught the above rules.
Unless IBM COBOL programmers are morons ;-), this is pointless.  The
second reason is that it may be less efficient to access an indirectly
referenced record area than a directly referenced Working Storage area.
On modern processors, this should be negligible, but I am not familiar
enough with IBM architecture to know for sure.  At any rate, I believe
most of us would agree that such efficiency considerations would be of
less importance than program clarity.

I believe the practice was started a long time ago by IBM (after all,
*they* don't mind if programs use more memory :), rather than teach
simple record area discipline.  This has been brainwashed into the
generations of IBM programmers since.  They are so indoctrinated to it

First of all, starting with the IBM COBOL 1985 compilers, the record
area associated with a file is available between OPEN and CLOSE. 
Further after some PTF level with Language Environment 1.5 (OS390 level
1.3), the CEEDUMP listing included the records for the files for all
COBOL 1985 compilers. This means that the results of a straight read are
easier to find in the file area than they are in working-storage. 
Currently they get messed up with the file names (DDNAME for the file,
not the FD name) in the dump if there is an internal sort and the SD is
not the last descriptor in the FILE SECTION.  Trust the BLF numbers
associated with the file in that case.  Since either the working storage
or the file area contains the record to be written for a WRITE statement
depending on whether the from option is used, write from buys nothing in
the new environment other than you don't have to initialize the record
completely before each write but you can use fields that are still
valid. 

The COBOL standard specifically states that the availability of a record
area associated with a file is undefined before OPEN and after CLOSE. 
With prior IBM compilers, addressability for input records was dropped
at end of file for at least the non-VSAM sequential files so that READ
file AT END MOVE HIGH-VALUE to record would probably cause a protection
exception.  

The main danger with using READ rather than READ INTO comes with
variable length records and probably is the same danger on most
platforms.  If you attempt to access fields that are not in the actual
record, results are unspecified and if you move to a field beyond the
last byte of the current record, you will corrupt a subsequent record in
the buffer.  With VSAM I suspect that you always have a separate record
area but I have not investigated the code.     

Note there are other subtle problems you get with READ INTO that you
don't get with READ when you are dealing with indexed files.  If the
only true description of the record is in working storage, if that
changes when you use READ INTO/WRITE FROM all sorts of debugging hell
can occur if you done change the record lengths correctly.  Despite my
dislike of the COBOL awkward form of qualification, in any new program I
copy the record description in wherever the record is used so that I am
guaranteed that I make ONLY ONE change to change all instances of the
record.  
With the advent of large working storage sections (over 64K) it may take
the loading of a register to provide addressability to the record either
in working storage or in the file buffer so the efficiency of addressing
is the same.  You save a MOVE when you use READ or WRITE instead of READ
INTO or WRITE FROM. 

Because of the changes in technology, they have fairly well stopped

No!  The routine in the generated COBOL code did not get called upon a
READ!  That code only got called when the elementary field that was the
object of "occurs depending on" phrase got altered, that included a
change to any group level to which that elementary field was a part of.

That was why any intelligent programmer would identify which of those
01s in the FD was the proper record just read in and then before
referring to any variable field in that record would "MOVE RCD-CNTR-
(whichever) TO RCD-CNTR-(itself)".  This then caused the compiler to
set the length of THAT 01 to the correct length.

READ INTOs always used max record as the from length and the length of
the to area as the to length.  This could result in either truncation
or padding as will any move of dissimilar length areas.

As a note, to avoid the constant recalculation of variable length
areas, the ODO object was often redefined with a slightly different
name and then could be referenced without the compiler doing a
recalculation.

With all this clamor about READ versus READ INTO of variable length
records, I now am understanding why some interviewers in the past were
so concerned with variable length record I/O skills - while I never
thought much about them....

I learned in a shop that had many variable length KSDS files,
synchronized to each other (SQL was a long ways away).  Many of those
files had multiple ODO phrases and one had four.  To modify there
records did require a READ INTO but mere input only reads did not.

Learning a few simple "rules" that were required to avoid clobbering
storage was all that was needed, and these rules are still valid to
this day.
