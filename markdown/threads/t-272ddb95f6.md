In my web pages I described a systematic procedure for eliminating
ALTER statements from old spaghetti code. Later I found the
following paragraph in the manual for VS COBOL II:

Altered GO TO statements in independent segments are returned
to their initial states when control is transferred to the
independent segment that contains the ALTERED GO TO from
another independent segment with a different priority.

So far as I can figure out, this means: when certain peculiar things
happen, an ALTERED GO TO can be reset without any statement having
to explicitly ALTER it. When this kind of thing happens, the
procedure I describe in my web pages would not accurately preserve
the original behavior of the program.

I suppose this possibility is one of the reasons ALTER has such a bad
reputation. It sounds like a debugging nightmare to me.

What I don't understand is the kind of event which causes this
resetting to occur. I've never encountered a program which used
priorities or segmentation. The manual doesn't define its terms
very well, in fact it nowhere defines "independent segment".

Here are my guesses:

1. A segment consist of all sections having a given priority.

2. An independent segment consists of all the sections with no
explicitly designated priority.

3. The priorities affect the treatment of overlays in some way,
as by specifying who may overlay whom.

4. The priority of a section may be specified as a one- or two-digit
number at the end of a section header, between the key word
SECTION and the terminating period (full stop).

How close am I? Can someone provide a coherent explanation? Is it
likely that anyone would deliberately use this implicit resetting of
an ALTERED GO TO, rather than invoke it by accident?

Please don't tell me "don't use ALTER." That much I know already.
I'd like to understand these pathologies better so that I can
ALTER my web pages accordingly.

Your interpretation of the ALTER/segmentation problem is basically
correct. Using ALTER is a maintenance nightmare anyway, but
segmentation makes it worse. In theory, ALTER is implemented by
changing the object code to point to the new address. If the code
segment is reloaded from disk, it's reloaded in its initial state. But
some compilers never implemented the semantics this way, some stored the
ALTER information in resident memory.

Any program which depends on the state of ALTERed GO TOs in overlayed
segments is a bug waiting to happen.

Your ALTER fix up program should check the SEGMENT LIMIT clause, and the
segment numbers, and warn about any ALTERed paragraph in a non-resident
segment.

Michael:

You have once again posed an interesting and useful question to the
group.

Michael C. Kasten wrote:
› 
› In my web pages I described a systematic procedure for eliminating
…[10 more quoted lines elided]…
› 1. A segment consist of all sections having a given priority.

This is correct. The correct terminology is not 'priority' but rather
'segement-number' (yes, I know that IBM has chosen to call it priority,
but I don't like the other ideas that the word priority conjures up).
The segment-number is found on the SECTION header. Segments with
segment-number in the range 0 through 49 belong to the fixed portion of
the program, while segments with segment-number 50 and above are
independent segments.

The fixed segment may be further subdivided into an nonoverlayable and
overlayable portion by use of the SEGMENT-LIMIT clause in the
OBJECT-COMPUTER paragraph in the ENVIRONMENT DIVISION, CONFIGURATION
SECTION. This subdivision does not affect program behavior (the thing
you are concerned about) in any way other than changing the tradeoff
between run time and memory usage.

The change in behavior that concerns you is that found between the fixed
portion of the program and the independent portion (made up entirely of
independent segments).

› 
› 2. An independent segment consists of all the sections with no
›    explicitly designated [segment-number].
 
› This is not true.  See above.
 
› 
› 3. The [segment-numbers] affect the treatment of overlays in some way,
›    as by specifying who may overlay whom.

All segments of the same segment-number are considered to be gathered
together (by the compiler and/or linker) for the purpose of overlaying
the program. In compilers implementing segmentation at a lower level
(according to FIPS terminology) all segments of the same segment-number
are required to be physically adjacent in the source code; this
requirement is relaxed at the high level of implementation.

› 
› 4. The [segment-number] of a section may be specified as a one- or two-digit
…[5 more quoted lines elided]…
› an ALTERED GO TO, rather than invoke it by accident?

You are very close. Unless you've implemented this nightmare, you
probably never combined all the facts in the same program to generate
all the possible pathologies.

This particular behavior is tied to what the standards writers probably
had in mind for the overlays. If you think in terms of the following,
it may be more easily understood:

Temporary storage associated with the procedure division (such as might
be used for storing paragraph exits) is never overlaid for the fixed
portion of the program. Temnporary storage associated with the
independent segments is overlaid, and is initialized upon every implicit
or explicit transfer of control into an independent segment from a
segment with a different segment number.

The truly bad pathology in all this is not the ALTER (which has been
avoided by many for many years) but the interaction of the PERFORM and
independent segments. Consider a nested PERFORM in segment 51 the
innermost of which performs a paragraph in the fixed nonoverlayable
segment (no segment-number, or a segment number below the SEGMENT-LIMIT)
which contains a PERFORM of a section with a segment-number of 52.
According to the standard, when the PERFORM in the fixed segment exits,
it causes a transfer of control back to segment 51, thereby initializing
its temporaries (which had been used for segment 52), and the PERFORMed
paragraphs in segment 51 are now falling through. Some compiler
implementors have used a stack (rather than storing paragraph exits in
temporaries) which would avoid this behavior, but which also explicity
violates the standard.

I think anybody depending on this behavior probably got there by
accident, or by malice.

Lesson 2: Avoid the following type of construct:

PROCEDURE DIVISION.

PERFORM INITIALIAZATION.

COMMON-STUFF SECTION.
A-COMMON.

B. PERFORM A-COMMON.

EXIT-SECTION.

MAINLINE SECTION 60.
A.

PERFORM B OF MAINLINE.

GO EXIT-SECTION OF MAINLINE.

B. PERFORM A-COMMON.

EXIT-SECTION.
EXIT.

HANDLE-ERROR SECTION 90.
A.


The PERFORM HANDLE-ERROR, while perfectly legitimate from the compiler's
perspective, has the effect at runtime as follows:
1) Causes, by explicit transfer of control, independent segment 90
to be loaded, in its initial state (not a problem so far).

2) When the paragraph A-COMMON exits, it is an implicit transfer of
control back to whichever independent segment performed it, thereby
loading that independent segment in its initial state, which means that
paragraph exits are in fall-through mode. So, paragraph B falls through
to EXIT-SECTION (since paragraph B's exit has been reinitialized), and
EXIT-SECTION falls through to the next SECTION (since EXIT-SECTION's
exit has also been reinitialized).

There exist many wonderful ways of disguising these transfers of
control. Consider the possibilities with DECLARATIVE sections, for
example.

The discipline required is to allow, within the fixed portion of the
program, PERFORMs of independent segments only like those in THE-FIRST
section in the example; all other PERFORMs in the fixed segment should
refer to other procedure-names within the fixed segment.

big to fit into memory, and must be overlayed in smaller pieces.
This process is possibly obsolete now with the current memory

I have had to use segmentation/overlaying for a PC system running
under DOS to fit a complex update into the standard 640Kb memory.

In the above *simple* example the STARTUP, PROCESS1, PROCESS2 and
CLOSEDOWN sections are overlayed and occupy the *same* area of
memory in turn. In a complex program there may well be more each
used sequentially or used depending on the type of processing
required.

The overlayed routines can perform any non-overlayed section or
paragraph (ie. section number less than 50) and can be performed by
any non-overlayed routine.

One overlayed section CANNOT perform another because there will be
nowhere to go back to, ie. the code in memory is the current
overlay NOT the calling routine.

Each time an overlay is called, it is normally read into memory
thus any ALTERs will be at the initial state.

I believe this to be the COBOL standard, but different manufacturers
implementations may vary slightly in actual detail.

On the COBOL that I started on (ICL 1900) you could have up to 5
different overlay areas, all section 5x were overlayed together as
overlay area 1, ie. 50, 51, 52, .... 59 all occupied that same
memory area in turn. The same was true for 6x, 7x, 8x and 9x as
overlay areas 2, 3 4 & 5. This meant that a 5x could perform a 6x
etc., but a 5x could still not perform another 5x, nor a 6x perform
another 6x and so on. This could be useful where memory was
limited, BUT careful program design was required and program could
disappear into the wild blue yonder if you broke the rules.

I hope that this helps to clarify things.

Michael:

You have once again posed an interesting and useful question to the
group.

Michael C. Kasten wrote:
› 
› In my web pages I described a systematic procedure for eliminating
…[10 more quoted lines elided]…
› 1. A segment consist of all sections having a given priority.

This is correct. The correct terminology is not 'priority' but rather
'segement-number' (yes, I know that IBM has chosen to call it priority,
but I don't like the other ideas that the word priority conjures up).
The segment-number is found on the SECTION header. Segments with
segment-number in the range 0 through 49 belong to the fixed portion of
the program, while segments with segment-number 50 and above are
independent segments.

The fixed segment may be further subdivided into an nonoverlayable and
overlayable portion by use of the SEGMENT-LIMIT clause in the
OBJECT-COMPUTER paragraph in the ENVIRONMENT DIVISION, CONFIGURATION
SECTION. This subdivision does not affect program behavior (the thing
you are concerned about) in any way other than changing the tradeoff
between run time and memory usage.

The change in behavior that concerns you is that found between the fixed
portion of the program and the independent portion (made up entirely of
independent segments).

› 
› 2. An independent segment consists of all the sections with no
›    explicitly designated [segment-number].
 
› This is not true.  See above.
 
› 
› 3. The [segment-numbers] affect the treatment of overlays in some way,
›    as by specifying who may overlay whom.

All segments of the same segment-number are considered to be gathered
together (by the compiler and/or linker) for the purpose of overlaying
the program. In compilers implementing segmentation at a lower level
(according to FIPS terminology) all segments of the same segment-number
are required to be physically adjacent in the source code; this
requirement is relaxed at the high level of implementation.

› 
› 4. The [segment-number] of a section may be specified as a one- or two-digit
…[5 more quoted lines elided]…
› an ALTERED GO TO, rather than invoke it by accident?

You are very close. Unless you've implemented this nightmare, you
probably never combined all the facts in the same program to generate
all the possible pathologies.

This particular behavior is tied to what the standards writers probably
had in mind for the overlays. If you think in terms of the following,
it may be more easily understood:

Temporary storage associated with the procedure division (such as might
be used for storing paragraph exits) is never overlaid for the fixed
portion of the program. Temnporary storage associated with the
independent segments is overlaid, and is initialized upon every implicit
or explicit transfer of control into an independent segment from a
segment with a different segment number.

The truly bad pathology in all this is not the ALTER (which has been
avoided by many for many years) but the interaction of the PERFORM and
independent segments. Consider a nested PERFORM in segment 51 the
innermost of which performs a paragraph in the fixed nonoverlayable
segment (no segment-number, or a segment number below the SEGMENT-LIMIT)
which contains a PERFORM of a section with a segment-number of 52.
According to the standard, when the PERFORM in the fixed segment exits,
it causes a transfer of control back to segment 51, thereby initializing
its temporaries (which had been used for segment 52), and the PERFORMed
paragraphs in segment 51 are now falling through. Some compiler
implementors have used a stack (rather than storing paragraph exits in
temporaries) which would avoid this behavior, but which also explicity
violates the standard.

I think anybody depending on this behavior probably got there by
accident, or by malice.
