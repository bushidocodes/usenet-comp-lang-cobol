In our shop the "no Go TO's' rule is one that is never given an
exception. Pretty much all other rules have exemptions occasionally.

Even our error exit handling looks like this:

If Hideous-Error-Condition
MOVE '4536' TO ERR-PARA-NO
MOVE 'Hideous Error' to ERR-MSG
PERFORM 9999-ERROR-CONTROL.
^^^^^^^^^
....
....
....
9999-ERROR-CONTROL.



STOP RUN.


Anyway, it is not any hardship on my part to code this way, and it's
MUCH easier to do maintenance on other folks' code. Granted, it would
all be a little harder if we were not using COBOL II.

Slightly off-topic here, but in your error-control routine you can
use the error number (or status code etc) to refer to a common
table which contains the desired messages and actions. Here's a
rough example:

...
IF ERROR-STATUS NOT ZERO
PERFORM 9999-ERROR-CONTROL
END-IF
...
9999-ERROR-CONTROL.
EVALUATE TABLE-ACTION (ERROR-STATUS)
WHEN ZERO
* ignore it completely
MOVE ZERO TO ERROR-STATUS
WHEN 9
* fatal error
DISPLAY "Aaaargh"
STOP RUN
WHEN OTHER
* keep a record of the error but continue
DISPLAY TABLE-MSG (ERROR-STATUS)
END-EVALUATE.

Well, that works too. But perhaps I oversimplified my example. Part
of our standards are to indicate 1) paragraph error occured in 2)
Description of error 3) image of current input record(s) and if DB2 or
CICS error, 4) tables being referenced when error occured, 5) dump of
all returned status codes. These are all fields set up in the ERR-INFO
record that the 9999 para uses. And we have to test for error
conditions for everything (well, almost.) But ALWAYS after any and
all EXEC SQL's, EXEC CISC's, any kind of external file usage, after
any calc that could over/underflow, overflow of any internal table and
before any divide to check for divide by zero. Also, any program that
abends because of a testable condition has to have an error check
added.

So, does the source code get bigger all the time? Yeah, sure does.
But the programs are not too much less readable; after a while you
learn to recognize the error check stuff and can mentally block it
off. Is it a pain to write new code? Not too; it's all cut and
paste. Hardest decision to make is how necessary an error test is for
a given procedure. Some stuff (see above) automatically gets an error
test.

Execution speed doesn't suffer too much, and is made up for by the
speed in which we can remove problem records and rerun jobs (if
appropriate)

And as nice as ABEND-AID is, I'd rather not have to look thru any
dumps to figure out a program abourt.

Aloha,

I disagree with the GO TO usage here; I don't see it as "more appropriate or
maintainable". I'll give you an example of why. Suppose your code reads:

IF error-condition-1
GO TO 9999-ERROR-PROCEDURE
END-IF
...

9999-ERROR-PROCEDURE.
...
IF error-condition-1
process this error
END-IF
IF error-condition-2
DISPLAY 'SECOND ERROR'
STOP RUN.
..

Do you see the 'real' error? There a missing period (or END-IF) for the
second IF in the error paragraph. This is a bug and hopefully will be found
in testing. But, how will the bug manifest itself? Well, if you use PERFORM,
the program will continue processing as if error-condition-1 was a non-fatal
error (maybe an endless loop or an abend or just bad results) BUT IT WILL
STILL BE FOLLOWING THE NORMAL LOGIC FLOW. If you use GO TO, the program will
fall through to the code below and produce really weird results. Worse, if
it's the last paragraph, the compiler may generate a STOP RUN for you,
'hiding' the bug until you add another section of code after it. Or get a
compiler that does not generate a STOP RUN.

Also, what happens if you find that there is a way to recover from
error-condition-1 and you WANT to return the normal logic flow in some
circumstances? If you've used GO TOs, you have to go back and change to
PERFORMs. If this coded is COPYed or INCLUDEd in multiple programs, this can
be a big task. If you've used PERFORMs, it's not.

If your standards say "9999-ERROR-PROCEDURE" never returns because the it does
a STOP RUN (or some equivalent), then I don't see why a PERFORM is any harder
to conceptualize that a GO TO.

Another reason for using a PERFORM is concerned with "levels" of
error recovery. The innermost level might be unable to recover
from an error but the outermost level wants to continue anyway.

For example, an inner level reads a credit card; the outer level
is preparing a bill with the option to enter the card number
manually. Or the inner level reads a diskette drive and the outer
level is testing archived records for validity and consistency.

In these cases, the inner routine can call the PERFORM to remember
the error (and probably set an error flag) but the outer routine
can choose to accept the error and continue, or retry etc.

Coding GO TO at the innermost level prevents *any* recovery.
Using PERFORM at least allows you to make improvements later.
