Well, It's not an OFFICIAL response, but it's as far as I care to take

----------------------------------------------------------------------

Thane,

This is my personal opinion, not an official interpretation of J4.
If you want to submit an official interpretation request to J4,
let me know.  However, J4 is giving priority to preparation of the
next COBOL standard, so a swift reply should not be expected.

The content of the record area after a READ statement
that results in the AT END condition is analogous to the content of
the record area after a WRITE statement. Both are undefined.  

As you stated, the record area is available after the OPEN statement 
and is no longer available after the CLOSE statement.  There are no
words about the record being unavailable after the AT END condition, 
only words that say "the content of the associated record area is
undefined".

Sincerely,
Don Schricker

control purposes.  At one time, some files that required multi-volume media
had end-of-media records written by the file creation program.  This record
might also contain some "management information".  It even might contain an

final version of this record and possible other information.  First and

Assuming that record storage can be referenced after an EOF sounds like
a risky proposition to me.  The fact that readers of the standard are
uncertain whether the wording "undefined" in this context might also
mean "undefineable" suggests that compiler designers might also feel
justified in assuming the most convenient (for them) interpretation.  I
would argue that it is certainly unclear what should be the proper
semantic interpretation associated with changing "record storage" when
no record exists.

Purely from the standpoint of one familiar with the Assembler QSAM I/O
macros on MVS (used by COBOL under the covers), COBOL is doing the
equivalent of a LOCATE Mode GET to return a pointer to (the address of)
the next record in the actual I/O buffer pool.  When an EOF is detected,
"GET" branches to an EOF address and NO address is returned.  What value
does the COBOL runtime assume for its record pointer in this case?  It
may by default be the last valid record pointer for the previous record,
but this sounds like an implementation dependent happenstance that could
change in the future.  If it pointed instead just after the last record
in the last block read, references to this storage might work most of
the time but occasionally do "strange" things when this happens to hit
on the last block in the buffer pool and changes affect storage outside
the buffer pool.  The program could appear to function OK in testing,
yet exhibit "nondeterministic" behavior in production.

the "runtime" either does, or does not, MOV data from the low-level

the "program" itself OR in those cases where the low-level buffer

You are assumming that Cobol is very simplistic in its use
of this underlying mechanism.  It is usually inappropriate
to simply have the record area point to a position in the
file buffer.  It is (in many cases) valid for the program
to set the record area to any arbitrary values for, say,
doing a START or random READ, or even to REWRITE the
record.  You certainly wouldn't want this to happen in
the file buffer as it may constitute corruption.

In fact with a variable length record it is quite valid
to change it and REWRITE (in general) and if this is
done in the buffer it may overwrite the next record(s)
or cause the system to lose track of where the next record 
is.

If Cobol did use the mechanism you described that it
certainly would have already exhibited non-determinsitic
behaviour.

Perhaps 40 years ago Cobol did use that mechanism and
there were problems.  'Tribal memory' in the form of
site standards were developed and, though Cobol 
specified exactly what should happen, many rely
on ancient taboos and folk medicines rather than
understand the language.
