```
I have to choose a COBOL compiler for an NT platform which will use MS
SQL Server for database access.   The app has no screens - it's a
suite of batch transaction processor programs being ported from an
older platform. 

Microsoft has promoted their server synchronization products to
address what we're doing, however, after reviewing their Replicator,
it simply does not let us scale our business logic without
re-engineering every line of code we have.

 I'm trying to understand if this can be developed and deployed in
this environment by simply rewritting the database interfaces.  The
folks at Merant tend to reply with sales brochures - not much help.  

I require COBOL programs to run on the server and access MS SQL Server
running on the same server, so I'm not clear on the topology, the
requirements, the ability for it to work.

1.  Can it be done ?  Has anyone done this and what did you use?

2.  Must a COBOL runtime package be deployed on the server platform?
Isn't this different from database access originating from a client?

3. I expect the ODBC API will do the translation - yet the client
resides on the server.  Is there anything extra I'm missing here?

4.  We'll be replacing the older database CALLs with SQL.  
Must I embedd SQL or do I have the option to store procedures and
access them with COBOL? 


Thanks.

John.
```
