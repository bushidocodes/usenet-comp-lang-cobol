#### ↳ solution to convert numerics to strings

- **From:** "Torsten Reichert" <Thorsten.Reichert@dresdner-bank.com>
- **Date:** 2002-11-11T10:16:34+01:00
- **Newsgroups:** alt.cobol,comp.lang.cobol,ibm.software.cobol
- **Message-ID:** `<aqnrg0$9te1@news-1.bank.dresdner.net>`
- **References:** `<aqm54h$9ta1@news-1.bank.dresdner.net>`

```
I have to build dynamic SQL-Statements without host-variables (for technical
reasons).
Our compiler setting defines "," as decimal point., whereas informix
database is set to use "."

...
               05 GLAETTUNGSWERT      PIC S9V999999.
               05 S-VORKOMMA       PIC S9(1).
               05 S-NACHKOMMA      PIC 9(5).
               05 S-GLAETTUNGSWERT    PIC X(20).
...
           MOVE GLAETTUNGSWERT TO S-VORKOMMA
           COMPUTE S-NACHKOMMA = ( 10^^5 ) * FUNCTION REM ( GLAETTUNGSWERT ,
1 )
           STRING
                S-VORKOMMA
                "."
                S-NACHKOMMA
            DELIMITED SPACE
            INTO S-GLAETTUNGSWERT
```

```
Frederico (and Torsten)

 I replied to this in comp.lang.cobol - but wanted to make it clear that
doing an INSPECT / REPLACING of the numeric-edited version has a "minor"
problem.  It will leave a "." where the system expects a "," as the decimal
point.  This would either lead to incompatible data or incorrect "scaling".
What you would PROBABLY want to do is to do the INSPECT / REPLACING on an
alphanumeric redefinition of the  numeric-edited item.  Alternatively (as
this is a "fixed format" field), you could simply do a

  05 GLAETTUNGSWERT      PIC S9V999999.
  05 Edited-variable     PIC -9,999999.
  05  Alpha-variable redefines Edited-Variable  Pic x(9).
        ...
  Move  GLAETTUNGSWERT     to Edited-Variable
  Move "." to Edited-Variable (3:1)

but you MUST not then use Edited-Variable as a VALID numeric-edited field.
```
