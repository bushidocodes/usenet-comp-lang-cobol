```
There are a number of things you can do with ISAM which a Relational DB 
doesn't like and won't let you easily do.

For example,

let's say you are using START...  READ NEXT   to sequentially or 
skip-sequentially process a series of records.

You can issue a WRITE, REWRITE, or DELETE against any record you fetched or 
against other records that have nothing to do with what you fetched, and 
READ NEXT faithfully remembers where it is up to. (Not only that, but your 
updates are applied immediately with no waiting on deferred COMMITs. Of 
course you can't roll them back either...)

This functionality can be matched (albeit, primitively...) by opening a 
cursor in ESQL.

Fine and dandy.

Trouble is, if you want to replace any of the records you fetch, or race off 
and update some (un)related records, you can lose your cursor very quickly. 
There are options in SQL to deal with this but they apply to specific 
instances and it is hard to generalise them

I was testing a Data Access Object yesterday and everything was going fine 
until we got to the bit where I started throwing updates, inserts and 
deletes into the same object that was running my cursor. Worked sometimes. 
(I realized later it actually depended on where the cursor was positioned 
and whether what I was accessing was in the cursor range or outside it... 
all very interesting...). Unfortunately, "sometimes" is not an option for 
software that people are paying for, so I had to find a solution to this 
problem.

These objects have generalised methods to deal with sequential, 
skip-sequential, and random access, as well as any maintenance. But it 
really got its knickers in a twist when I required it to run a cursor while 
dealing with everything else. After spending a couple of hours looking for 
non-existent bugs, I hit the books and found that these limitations are 
documented.

Most of the solutions which occurred to me were messy and ugly... recognise 
when an update action was about to occur, save the current cursor position 
and re-open it automatically on the next FETCH...always declare cursors as 
updateable and non-static (but that doesn't help with related records that 
must also be upated)... that kind of nonsense.

Before I tell you how it was solved, with a general solution that works for 
any kind of access and access pattern that ISAM can do,  with immediate 
commits that faithfully emulate the ISAM processing, has anyone here 
experienced this kind of problem when mixing SQL maintenance with a running 
cursor?

Sure, for a specific case you can use positioned updates and play with 
cursor isolation, but isn't it a pain?

I really am coming to the conclusion that cursors are "best avoided"... 
they're so primitive and really only necessary with COBOL (I don't use 
cursors with Java, C#, or VB) because it doesn't easily handle result sets, 
and is grounded in ISAM skip-sequential processing.

Thoughts/opinions/comments/experiences, please.

Pete.
```

```
Because the Data Access Layer is comprised of objects, all that was 
necessary was to instantiate 2 objects, instead of 1.

1. Is a GET stream

2. Is a PUT stream

Identical code without logic changes in both of the objects.

Each object has an instance stamp provided by a factory process. I use this 
instance stamp (which uniquely identifies the object instance) as a DB 
connection identifier. Each object sets the DB connection to its own DB 
connection identifier whenever it carries out data handling actions.

So the 2 objects each have a separate connection to the same DB server. (I 
wasn't sure this would work, but it does...)

The calling (application) program has two separate (identical) interface 
areas, one for the GET stream (all requests that don't require maintenance) 
and one for the PUT stream (which handles all INSERTs, UPDATES, and 
DELETES).

The PUT stream can immediately commit its actions because there is no danger 
of conflict with the cursor(s) or random fetches.

The net result is that the effect of processing is identical to what it was 
with ISAM.

Here's some sample code from a harness program that simulates a series of 
data access requests from an application:

 MAIN SECTION.
 a000.
     perform startup-housekeeping
     perform main-logic until finished
     perform close-down
     .
 a999.
     stop run.
*>-----------------------------------------------------------
 STARTUP-HOUSEKEEPING            section.
     set not-finished to TRUE
*> Instantiate COM Servers...
     invoke COM "CREATE-OBJECT" using MOSTTest-ProgID
                                returning objPUTStream
     end-invoke
     invoke COM "CREATE-OBJECT" using MOSTTest-ProgID
                                returning objGETStream
     end-invoke
     .
 sh999.
     exit.
*>-----------------------------------------------------------
 MAIN-LOGIC                      section.
 ml000.
*>
*>  This processing is intended to simulate a stream of requests
*>  for data access from an application program.
*>
*>  The program will have all of its ISAM access requests changed
*>  to the appropriate MOST invoke by the Migration Tool: MOSTAMD
*>
*>  The MOST COM Server that services these requests is generated
*>  by the Migration Tool: MOSTGEN, using Migration Metadata and
*>  the MOST Base Template. (MOSTGEN and MOSTAMD are written in C#)
*>
*>  This harness program tests the generated MOST Data Access Layer module.
*>
*> Now try the methods...
*>
*>  initialize both instances... returns formatted interface blocks...
     invoke objGETStream "ARPMF0MTInit"
            using MOST-interface-block-GET
     end-invoke
     invoke objPUTStream "ARPMF0MTInit"
            using MOST-interface-block-PUT
     end-invoke
     initialize ARPMF0-REC   *> the COBOL record layout
     set MOST-primary of datablock-GET to TRUE  *> specify which key of 
reference
     set MOST-dynamic of datablock-GET to TRUE *> specify access mode
     set MOST-input   of datablock-GET to TRUE     *> specify ISAM OPEN mode
     move '= ' to MOST-relation of datablock-GET
     set MOST-primary of datablock-PUT to TRUE
     set MOST-dynamic of datablock-PUT to TRUE
     set MOST-I-O     of datablock-PUT to TRUE
     move '= ' to MOST-relation of datablock-PUT
*>
*> Random access... blank key
*>
     move ARPMF0-REC to MOST-data-buffer of datablock-GET
     invoke objGETStream "ARPMF0MTGetRandom"
            using MOST-interface-block-GET
     end-invoke
*> should return '02000'...
*> set key and try again
     move 'AMBER' to ARPMF0-CD
     move ARPMF0-REC to MOST-data-buffer of datablock-GET
     invoke objGETStream "ARPMF0MTGetRandom"
            using MOST-interface-block-GET
     end-invoke
*> should return '00000' and data...
*>
     move MOST-data-buffer of datablock-GET to ARPMF0-REC *> Return base 
record as seen by COBOL

*>    Change the attached record...
     move 200.0 to ARPMF0-BAL (2)
     move 300.0 to ARPMF0-BAL (3)
     move 400.0 to ARPMF0-BAL (4)
     move ARPMF0-REC to MOST-data-buffer of datablock-PUT
     invoke objPUTStream "ARPMF0MTUpdate" *> note direct update, no fetch 
first
            using MOST-interface-block-PUT           *> Fetch was on the 
"other" stream
     end-invoke
*> should return '00000' and data...
     move 'ANACONDA' to ARPMF0-CD
     move ARPMF0-REC to MOST-data-buffer of datablock-GET
     invoke objGETStream "ARPMF0MTGetRandom"
            using MOST-interface-block-GET
     end-invoke
     move MOST-data-buffer of datablock-GET to ARPMF0-REC

*>    Change the attached record...
     move 200.0 to ARPMF0-BAL (2)
     move 300.0 to ARPMF0-BAL (3)
     move 400.0 to ARPMF0-BAL (4)
     move 500.0 to ARPMF0-BAL (5)
     move ARPMF0-REC to MOST-data-buffer of datablock-PUT
     invoke objPUTStream "ARPMF0MTUpdate"
            using MOST-interface-block-PUT
     end-invoke

*> now try sequential access... THIS WILL CREATE AND OPEN DYNAMIC CURSOR
*>
*> This will start from ANACONDA
     move ARPMF0-REC to MOST-data-buffer of datablock-GET
     move '>=' to MOST-relation of datablock-GET
     invoke objGETStream "ARPMF0MTStart"
            using MOST-interface-block-GET
     end-invoke
*> should return SQLSTATE '00000' and data buffer unaltered...
     if MOST-SQLSTATE of datablock-GET = '00000'
*>        get 10 records...
        perform 10 times
           invoke objGETStream "ARPMF0MTGetNext"
                 using MOST-interface-block-GET
           end-invoke
           move MOST-data-buffer of datablock-GET to ARPMF0-REC
        end-perform
*> 10 records on from ANACONDA is BBQ... this should be in the buffer.
*> intersperse random write into sequential stream...

had been carried out against the equivalent ISAM file.

This is not an end result, it is merely a first step. It shows that COBOL 
can be automatically converted away from using ISAM,  to a relational DB as 
the data source, and processing logic does not change. At the same time, 
direct data access calls (whether to ISAM or ESQL) are removed to a separate 
Data Access Layer, which is then invoked from the application code.

This is step 1 of the diagram shown on: 
http://primacomputing.co.nz/cobdata/COBOLPortal.aspx
```
