```
Hi All,

I the past few weeks I have been experimenting with
using a standard Web browser as the interface to
interactive Cobol programs. In these days of "thin clients"
this is the ultimate thin client, as NO piece of the
application need be on the client machine.

You can download a demo of this from http://www.etk.com.
The demo allows you to provide a description of one of
your files as a copybook. From this description, the
Web-enabled Cobol program program is generated
in typically 30 seconds.

Try it and let me know how it can be improved.
All ideas are welcome. Thanks.
```

```
work on any platform and with any (Cobol) compiler.

WebServer and the Cobol program. Here I use
the LRWP protocol for Xitami from iMatix. (www.imatix.com)

The communication piece is isolated in a
single subroutine and must be changed for
other protocols.
```

```
Hi Leif -
  That stuff looks pretty darn good. I'm impressed. 
We have been able to do pretty much exactly what you
have done, and ran into some troubles with different
web servers and compilers. 

  Using COBOL-CGI with Fujitsu drives me slightly nuts;
sometimes it works and sometimes it doesn't. :) It doesn't
like Domnio GO webservers very well, and has trouble
with IIS version 4 (well, we only have Fuji version 3 here,
perhaps versin 4 works better.) 
 
  AcuCOBOL seems to have some trouble linking to STDIN and
STDOUT under Windows NT, and doesn't work at all with the
Xi* webserver. In fact, it doesn't work well at all with
IIS 4, Domnio, or Netscape, probably due to having 
to call the runtime. We loose connectivity to STDIN and
STDOUT. Interstingly enough, with old servers, AcuCOBOL
works flawlessly. 

  We finally settled on IBM Visual AGE COBOL as the development
platform and Domino GO as the server. This was deliberatly 
choosen to allow ports to AS/400 and S/390. It has a few
problems talking to STDIN and STDOUT under Windows NT as well,
but they were fairly easily solved by using a C wrapper in a
cmd script. I chatted with a chap at the Santa Theresa labs
and it those minor troubles may have a relatively easy fix.
Won't know till they have time to check it out. 

  Now given all the above, we found that just putting out
text emulation screens didn't really appeal to users;
besides that would have been TOO easy - we can do that 
from CICS. Without any effort at all.

  But the CICS model is what we are implementing; we 
found that small screens with discrete peices of data
entry on them are what we have to present to the users.
And that means tracking the transaction state between
web pages. Sheebooppa! That isn't EASY. 

  Some of the mechanisms we have explored include sending
hidden fields from page to page, writing partial data,
and writing to a transaction state data file. The last
path is the path we are going to implement. However, 
we haven't been able to sucessfully use LRW yet. On the
other hand, once the IBM runtime loads under NT, we 
get very FAST response from CGI programs. Under simulated
load, a 300 mhz Intel box with 64megs of RAM and plent
of disk space managed to handle 1500 web transactions per
minute with no problems. I didn't push it to the limit,
but I would not be surprised to see that figure 
double with a little optimization. Besides, we can always
cluster the darn things, and CICS will be inserted 
between the web and the database/VSAM files or whatever
data storage is eventually decided upon. It will almost
certianly be a mixture of DB2 and VSAM. 

 The other bugaboos we are battleing with include
"silly" stuff, such as dealing with data validation. 
A small example; on screen or in a GUI I would normally
break up a telephone number into an areacode, an exchange,
and an extension. Three simple autolinked fields on a
screen. On the web, I can't do that because the darn field
sizes tend to change with some whim of browser. One
annoying problem is I often get 15 charcter wide fields
on screen when I want 3 chars. The fields MAXLENGTH
paramter works, and only allows 3 characters in, but
doggon it - it's ugly! And there is no way I know of
in the browser to force numeric entry, so I am 
forced to code login in the COBOL program serving that 
form or part of that form to redisplay it with 
data errors. 

 We also found we had to use templates, if for no
other reason to avoid massive numbers of "display"
statements in the COBOL code. :) Actually, we are 
workign on parameter substitution in the template
files; for example "@username@@@@@@" will be replaced
by the "username" field if embedded in a line, 
and by a series of user information lines if found on
a line by itself. "@HIDDENFIELDS@@@" on a line by itself
inserts all the transactional information we want to
pass along to the next stage. (Usually a key and the transaction
id for the transaction file.) 

 I'm really pleased to see other people workign along these
lines; COBOL is pretty ideally suited to use with the WEB,
and FAR underratted. :) 

Yours,
-Paul
```
