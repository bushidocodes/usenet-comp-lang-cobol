This is a common problem with CICS programming. The CICS pre-processor or
translator runs as a separate step before the COBOL compiler. It does not
expand copybooks, so the translator cannot generate code for any CICS
statements in the copybooks. When you get to the compile step, your CICS
statements are copied in from the copybooks, but they haven't been
translated, so the COBOL compiler rejects them.

I built a copybook much like the one you are trying to build, but I looked
up all the EIBRESP values in a CICS manual and hard-coded them into the
copybook. The DFHRESP(NORMAL) stuff requires the translator. Your copybook
would have to be coded something like this:

01 RESPONSE-CODES PIC S9(8) COMP VALUE ZERO.
88 NORMAL-RESP VALUE +0.
88 NOTFOUND-RESP VALUE +13.

The important thing to remember is that you cannot put EXEC CICS ... END-EXEC
statements or DFHRESP(...) statements into a copybook, because the CICS
pre-processor will never see them.

› 
› 2.  Also, at work most programs use the handle condition, and i am trying
…[10 more quoted lines elided]…
› BFE··.@a··.com

It is a good practice to check the EIBRESP following a LINK or XCTL command.
If an EXEC CICS XCTL command (transfer control) succeeds, you will transfer
to another program without returning. But if the program cannot be found,
you will remain in your current program, and you should do something about
that error condition.

I don't bother to check EIBRESP following an ASKTIME or RETURN command. If
those commands fail, you have serious problems with CICS, not with your
program. If you are doing a RETURN with COMMAREA, there might be some
benefit in checking the EIBRESP code if the RETURN fails, but I doubt it.

There is generally no need to check the EIBRESP code following an EXEC CICS
ADDRESS TWA command.

Also, you should always use EIBRESP rather than the older EIBRCODE, because
EIBRCODE is much harder to translate and place into an error message.

There are one or two unusual cases where you might actually need to use a
HANDLE condition, but normally you want to avoid them because they generate
an implied 'GO TO' and you cannot return from it.

I hope that helps!

This is not correct. You CAN put CICS statements into copybooks, but
you must run them thru the pre-processor. We run our CICS copybooks
thru the translator and then catalog the generated code into a library
that our CICS programs then "COPY" from.

We have very little EXEC CICS code in our online programs - source code
that is actually coded in the program itself. Most of our CICS code is
referenced via copybooks. It works great - all programs use the very
same CICS code this way.

Great stuff, guys! I'll try to add to this instead of just saying
'me too'. The original post referred to the DFHRESP facility of the
preprocessor. A major disadvantage of using it --- regardless of HOW
-- is that the resulting COBOL source program and listing show only
the derived literal -- the meaningful definition is gone. For
example, whereas the original source code may have had an IF statement
for DFHRESP(NOTFND), the resulting source code will show only the
literal value giving no clue to a junior programmer on the problem
being tested. The copybook with 88 levels for each works best here.

My recommendation is to set up a copybook as defined previously and
use the RESP keyword in all CICS commands, causing the response code
to be placed into a working-storage field. As mentioned previously,
there should be no need for GOTO's or HANDLE conditions. I encourage
all programmers to test conditions of all commands -- better
documentation, better knowledge, and better performance.

On my system, the translator/pre-processor takes care of this for you.

IF CICS-RESPONSE = DFHRESP(NORMAL)
NEXT SENTENCE
ELSE
IF CICS-RESPONSE = DFHRESP(NOTOPEN)
:
:

* DFHRESP(NORMAL) = 0 INSERTED BY TRANSLATOR
IF CICS-RESPONSE = 0

While this may be an acceptable 'get-around', I'm not sure I'd condone
such a practice unconditionally. What if the IBM-supplied copybook
changes, for instance, with a new release of CICS? You'd have to
re-create the copybook, which is in itself no big deal, but then you'd
have to keep track of which version you want (especially if you're in a
bigger shop or a service provider that needs to support/maintain
multiple regions that might be at different releases). I'm perhaps lucky
because years ago, someone at my shop recognized this catch-22 and wrote
an assembler routine to tackle this: all our source code runs through
this 'pre-preprocessor' which expands the COBOL copybooks *before* the
code runs through the CICS preprocessor. Neat trick, eh? Unfortunately
it doesn't handle nested copybooks, but these are rare.
One other thing: take a look at the NOHANDLE option, which can be used
on any EXEC CICS statement. It does exactly what it implies - it
temporarily overrides HANDLE CONDITION's without getting into POP's and
PUSHes (which are naaaasty). Then you can check EIBRESP directly without
requiring a separate WS field used with RESP(...), e.g. IF EIBRESP =
DFHRESP(DISABLED). This is IMHO a decent technique as well.
Just my $.02.
