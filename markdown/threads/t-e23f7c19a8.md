```
In a PC environment using REALIA-4/5 COBOL under WIN95, WIN98, DOS 5, 6.0,
6.22.

Consider the following:

SELECT (120Mb ISAM file on a CD-ROM)


OPEN INPUT
READ (a bunch)
CLOSE

The CLOSE operation takes 5-15 seconds on a P-II 233MHz with
a 24X CD-ROM. Can anyone imagine what a CLOSE operation
would be doing to a read-only file opened as input? (Sequential
text files don't exhibit this behaviour.) The time taken to close
seems dependent on how big the "bunch" is (how many records
have been read). In other words, it takes far, far longer to close
the friggin' file than to look something up in it.

Guesses, even, are welcome.
```

```
› The CLOSE operation takes 5-15 seconds on a P-II 233MHz with
› a 24X CD-ROM. Can anyone imagine what a CLOSE operation
```

```
I was going to suggest a buffering problem, same as Thane, but then
realized the file id READ ONLY. Flushing the buffers could easily take
that long on a file that large ... but a read only file has no buffers to
flush.
Two hours later, a flash came. This is a wild assed guess, but does your
compiler update the last time accessed for read only files? Some do. If so,
then the problem might be that it is a CD-ROM. In effect, the effort at a
write causes a timeout, at which point the program recovers. I think the
idea of copying the file to the hard disk, an testing there is a good idea.
If the delay goes away on the hard disk, then I think you will find that
something (maybe the directory date) is being updated even though it
is a read only file.
```
