The essentials are consistancy with other programs and quality. Desirable is
beauty. Cosmetic is, what?, personal idiosyncrasies, which by definition are
non-standard. 

I dislike your model because it contains a veiled threat -- Essential is
non-negotiable, Desirable would make management happy, Cosmetic are your
boundaries of self-expression. 

My advice is:

.. Make the standard positive rather than negative. Make it a list of mostly DOs
rather than DON'Ts. In other words, provide leadership rather than repression.

.. Emphasize structure rather than grammar. Structure is where the most
expensive programming errors are made. Nit-picking grammar rules DO NOT correct
structural flaws.

.. Write top-down -- general principles first, technical details at the end. If
you organize it like a COBOL manual, you're doing it wrong, your thinking is
bottom-up. 

.. Provide lots of examples. A structure point cannot be made in a 5-10 line
code snippet. One precept may require 100 lines of non-trivial code. Writing an
adequate standard is not a weekend project.

.. Write the standard so it encourages growth rather than retarding it. 

.. Make it a consensus of programmers' style rather than tablets handed down
from the clouds. If you do this, it will be self-enforced by the programming
staff, as described below. If you impose it on them, you'll foster justified
resentment and disinclination to cooperate.

We discussed that at length in May, in a thread titled "A Style Question." At
the time I wrote:

I have contracted to shops where programming standards were nearly all about
grammar. They were followed rigorously. But the structure of those
shops' programs ranged from poor to horrid. Grammar controls do not produce
quality programs. I dislike them for two reasons:

1. They give the illusion of quality control without the substance.

2. They stifle growth. We should be empowering programmers, not
disabling them.

How to achieve reasonably consistent style across a shop? During the
25 years I managed programming shops, I made it very clear that nobody
'owned' a program. If you write a program today and it needs
maintenance three months hence, the task will NOT be assigned to you.
You will be working on someone else's code. Code is all community property;
we're all team players.

In that environment, a consensus style is developed by the programmers
themselves, out of self-interest. They'd be challenged if programs
looked radically different from each other .. in structure OR grammar.
When someone wants to depart from the norm, he discusses it with other team
members informally. Not at a conference table where egos are at risk.
They each have standing to voice an opinion because they will be
maintaining his code next month.

I, as a team member, voice my opinion, which carries a 'little' more
weight, but is not decisive. In the rare event of deadlock ... that's
where leadership comes into play.

In that environment, style can grow in a evolutionary fashion. When
grammar is formalized, it grows much more slowly .. too slowly to keep up with
changing technology and BUSINESS NEEDS that its champions claim to be concerned
about.

There is an example of what the first page might look like:

* Write a procedure like a story -- with a beginning, middle and end.
In the beginning you clear counters, acquire memory, construct
objects, open files, etc. In the middle, you loop through whatever
you're doing. In the end you print out totals, release memory, destroy
objects, close files, etc. Initialization means 'done at the
beginning'. Programs that 'clean up' at the end of a procedure rather
than the beginning are poorly designed. As a formatting convention,
leave a blank line between the beginning, middle and end.

* Design from the top down; program from the bottom up. When you are
writing a program, part of the time you will be designing and part of
the time you will be programming. Learn to tell the difference so you
can employ the correct mental process. Mediocre programmers do
everything bottom-up.

* Use the company as a design model for your program or system. High
level code should make high level decisions; low level code should do
the grunt work and refer exceptions to higher levels for handling. If
the first code in your program opens files, you have the model
upside-down, you have a file clerk running the company. If low level
code can abort the run due to an exception, that's like giving a file
clerk authority to declare the company bankrupt. For real-time
applications, use the network model, in which modules pass messages to
each other in response to events. Think of each module as a company
department, within which there can still be a hierarchy.

* Write function-like programs of reasonable size and simplicity. Glue
them together to form 'programs'. A reasonable size is 25-100 lines of
procedure. File IO is an especially good candidate for a callable
program.

* Use descriptive data and procedure names that anyone can understand,
not abbreviations or jargon. Run a spell checker on your program. A
program is written once; it is read hundreds of times. If the purpose 
of some code or data is not obvious, explain it with a comment written
in complete sentences which avoid jargon.

* Pass data between programs in memory, not by writing it to a
temporary file. Files are for transport through time, not between
programs. Top down design makes this easy; bottom up design creates
temporary files.

* Handle errors. Do not ignore return codes. Do not say you'll get
around to it later.

* Check limits on arrays, pointers for null, loops for progress. This
is called Defensive Programming. Your programs should never abend nor
go into an endless loop.

* Write one statement per line. Indent a consistent number of spaces
under data structures, if, evaluate and perform.

*Do not duplicate (cut and paste) the same procedure in more than one
place. If you feel the need to do so, your design is poorly
structured. Redesign so the code appears once. If that is not
feasible, due to scope of project, at least make it a callable
program.

For files and screens, I would be inclined to use file-ids and
screen-ids as the prefixes for data-names, for example ACCT-FILE
datanames would all have a prefix of ACCT.

In working-storage, I would be inclined to have several group items,
one each for switches, counters, subscripts, file-statuses, etc, the
group levels being something like WA-Switches, WB-counters,
WC-subscripts, WD-file-statuses, etc.  The subordinate constituent
data items would have names that are all prefixed by the group name
prefix, e.g. WA-acct-status-sw, WB-acctfile-rec-counter,
WC-acct-table-sub, WD-acct-file-status.  Prefixes up to about WI-

constructs.  You might prefer to use indexes instead of subscripts,
but sometimes subscripts are more helpful, particularly when used to
refer to more than one table.  Conditional variable names would be of
the form WD-acct-file-OK, WD-acct-file-at-end,
WD-acct-file-key-not-found, etc, where the file name follows the
prefix and the individual condition description follows the file name.
 Sometimes it is better to associate subscripts more closely to the
tables for which they are used.

In the linkage section, I would be inclined to group parameters into
as few group items as possible consistent with ease of use, e.g. one
group for control parameters followed by data parameters, or to put

Internal COBOL sorts are neither necessarily more or less efficient
than external ones.  Depending on the compiler, internal sorts
normally use the same sort program as external sorts.  Using external
sorts permits greater flexibility when the sort sequence is to be be
varied between applications, or the same sorted file is to be used by
several other programs.  Using internal sorts can reduce the amount of
I/O that the system has to do, this applies when using either an input
procedure or an output procedure or both.  SYNCSORT and DFSORT both
provide powerful additional processing facilities for record selection
and manipulation, sometimes to the point where another program is
unnecessary.  These additional facilities can also be used when these
sort programs are used in internal sorts.
