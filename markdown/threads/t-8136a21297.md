 string the message and set flags for the messagbebox types
 invoke MyErrorMessages "new" using ws-parameters returning ls-response
*> MyErrorMessages is sub-classed to MessageBox to display the actual
*> messagebox

That's not OO, that's old wine in new bottles. One way to do it OO would be to
overload the errorMessages method. That's not easy in COBOL because program-
pointers cannot be used for methods (unlike C++) and method names cannot be
variables (unlike dynamic calls). If you think about it, overloading is not what
you want anyway. The method is the same for all languages; the only thing
different is T1 -- the message values. Why not create a class named Messages,
create one instance named CurrentMessages filled with the appropriate text by
New, then say simply

 move CurrentMessages::GetErrorMessage to MSG

That looks to me like a truly OO approach.

The cobbler's children have no shoes. OO classes cannot be applied to the
compiler nor Workbench. If you mean messages on the target machine, sure, there
are several choices -- ISO-8859 and ISO-10646 aka Unicode. Between the two, 8859
is the better choice because it doesn't require 'double byte' support from the
compiler. Because the heavy lifting is done by device drivers rather than OO
classes, Workbench will show the correct graphics.

handle the problem if only the compiler offered table sort. If it's not MF and
you can't wait a few years, use the CONVERTING option of the INSPECT verb for
single-byte replacements. There is no COBOL solution for double-byte, as in the
above example; you have to do it yourself.
