I have an older DOS/OS2 set of Micro Focus Cobol manuals.   The COBOL
User Guide says the following about the COMPUTE statement...

*Do not use the COMPUTE statements except for performing calculations
involving floating-point data*

Obviously COMPUTE works nicely with other data types in many
situations, however, when considering the behavior of level 78s in
general, which are an MF extension, always avoid them in arithmetic
expressions - especially compound ones.   I seem to recall them
causing trouble elsewhere too.  

I don't see this behavior as a mistake, rather I see it as an
implementation issue.  I have found other languages to have unusual or
unexpected behavior with constants as well, depending on just where
they are defined and how they are used.   What makes sense in one
language may not make work in another.  One compiler I saw, which had
no statement terminators for constants would interpret any trailing
spaces following a constant definition as part of the literal value -
including numeric literals which of course became strings.
