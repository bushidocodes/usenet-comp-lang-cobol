You will need to take that list of file-actions and substitute
equivalent 'op codes' for EXTFH - which I believe are pic x(4) comp-5.
Same goes for file-status - substitute the error codes EXTFH will give
you. Plus you will need to change the syntax for each file verb :-

	Call EXTFH using x, y, z 
		   returning ........


One file - One Class :-

Now the reasoning behind my approach. First, I took what Will Price did
and used that as the basis - developing my DBIMethods to receive message
to pass on to all file classes and return messages with results to the
classes which invoked the DBI. This has worked out very well, after
experimentation.

Quite simply the reason for a SEPARATE class for each file was
simplicity - use a template and delete those file access verbs that
don't apply to this specific file - for example no point in having an
access to read a RELATIVE in an indexed file etc. This too has worked
out fine - with one exception - I did create a class ASCII File used for
writing and reading ascii-delimiteds. I got in one hell of a mess with
this until I realized I had to tell the file class the size of the
record I was processing for AsciiRecord-A (e.g. 1,200 chars) and then
AsciiRecord-B (750 chars). To do that I have messages passing record
sizes for each of the ascii record sizes.

Now, using one Fileclass for ALL files - yes I'm sure that is possible,
and I think I'm correct in saying that is the approach Thane Hubbell is
taking, experimenting with Fujitsu OO.

Given that you have one FileClass and 5 physical files that means you
are now holding 5 instances, each with their own object references. But
it does present problems as I see it :-

(a) You will have to hold handles in OBECT STORAGE SECTION to indicate
which file you are handling  with the CURRENT method being invoked 

(b) You can hold these handles as a ordered or sorted collection of
object references.

(c) You would have to hold the equivalent of the file-status code for
EACH file object in the collection. 

(d) Also I suspect, you may need to keep the copy files clients1.cpy and
clients2.cpy in GLOBAL Working Storage or Object Storage, rather than in
an access method. The alternative would be an access method for each
file, (which I think is what you want to avoid), which aligns to the
Object Storage Section copy of the record (os-DataRecord)

I'm sure it is possible - but without a model it is only achievable by
experimentation.

Hope the above helps.

Jimmy
