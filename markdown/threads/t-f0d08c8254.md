Using a recent version of MF Cobol (3.1.27, I think) I encountered a
very odd little bug today- to flag a line for quick finding, because
there was a very long dataname coded there, I entered **** in col 1-6 of
the line. The program would no longer compile! The compiler insisted
that the data-name on that line was not defined anywhere. Removed the
**** and it compiled clean. !!!

No bug but a feature would you believe. An asterisk in column 1 is ignored
by the compiler to allow a listing file with errors to be edited and
then resubmitted as the source. Any lines added by the compiler like
headings, etc. have an asterisk in col 1 and get ignored.

Tho it seems pretty pointless now, I suppose it must have been useful
15 or so years ago when it first appeared. Trouble is, take it out now and
and you bet there'd be lots of folks relying on it one way or another.
```

No, standard COBOL says columns 1-6 is the sequence number area that
can contain anything the user wants. Column 7 is the indicator area
where an asterisk (or /) signals a comment line.

Some implementations (not MF) have a special non-standard format where
you can miss out the sequence number area, perhaps you were thinking
of this?

Incidentally, the draft of the next standard (and MF) includes free
format source in which there is no sequence number area. A comment
is introduced by "*>" but this can be used for an inline comment as
well as a comment line, which I think is rather handy.
```
