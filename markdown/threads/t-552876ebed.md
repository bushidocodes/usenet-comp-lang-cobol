Dear Fellow Cobol Programmers,

This is not Cobol specific, but I think that there are many more z/OS
programmers in here than on the ibm-main list...

I'm going to stir the pot again, as I seem to often do.  What can I say;
it's important to me.

One of my big nitpicks with z/OS versus VSE is in VSE we have, for 30 years,
specified "system level" definitions for what libraries are to be searched
(in a particular LPAR/VM guest image) to locate both application executables
and system executables.  Here is an example of such a system level
definition:

LIBDEF PHASE,SEARCH=(THIRD.TCPIP,PRD1.BASE,PRD2.SCEEBASE,PRD2.DB2750,  X
               PRD2.DLI1A0,THIRD.SORT,PRD2.PSF,PRD2.PROD,PRD2.AFP,     X
               USER.PROD),PERM  

This is executed at IPL, specifying the "permanent search chain" for all
jobs that don't otherwise somehow override it.

Things of specific interest are PRD2.DB2750 which is DB2/VSE V9.5,
PRD2.DLI1A0 which is DL/I VSE V1.10, and USER.PROD which contains all
"application load modules".

This makes a simple production JCL look something like this:

// JOB MYJOB
// EXEC PGM=MYPGM
/&

MYPGM is an executable in USER.PROD.  It may call SUBR1 and SUBR2, which are
both executables in USER.PROD.  Perhaps it also uses DB2.  It gets those
modules from PRD2.DB2750.  All is good.

Let's say I want to change MYPGM and I'd like to test it (always a good
idea).  I compile/link it in to USER.FJS.  Now here is my "test" JCL:

// JOB MYJOB
// LIBDEF PHASE,SEARCH=(USER.FJS)
// EXEC PGM=MYPGM
/&

The libraries in the LIBDEF specified in the job are searched first, and if
a module is not found there the operating system will then search through
the "PERM" libraries.  So in this case the version of MYPGM in USER.FJS is
executed.  SUBR1 and SUBR2 are in USER.PROD and not in USER.FJS so the prod
version are executed.  Again, all is good!

So let's go to z/OS.  Let's say we are like (apparently) most z/OS shops
that don't have application loadlibs in LNKLST.  So here is production JCL:

//MYJOB    JOB
//JOBLIB   DISP=SHR,DSN=PROD.APPL.LOAD
//STEP01   PGM=MYPGM
//

Hmm, that's OK.  Don't love it, but it's OK.  Now we'd like to test changes
as above.  So now we have:
//MYJOB    JOB
//JOBLIB   DISP=SHR,DSN=FJS.APPL.LOAD
//         DISP=SHR,DSN=PROD.APPL.LOAD
//STEP01   PGM=MYPGM
//

Well now I'm getting irritated.  I not only had to add "//JOBLIB  
DISP=SHR,DSN=FJS.APPL.LOAD", which I have no problem with, but I also had to
keep the existing JOBLIB of PROD.APPL.LOAD *except* I had to remove the
JOBLIB name from it so that it's concatenated to the end of the new JOBLIB. 
Yes, yes, petty, but it just shouldn't be necessary!

On the other hand, if PROD.APPL.LOAD is in the LNKLST all is good:

Production JCL:
//MYJOB    JOB
//STEP01   PGM=MYPGM
//

Test JCL:
//MYJOB    JOB
//JOBLIB   DISP=SHR,DSN=FJS.APPL.LOAD
//STEP01   PGM=MYPGM
//

Much nicer!  When I want to execute a "test" module I add a JOBLIB.  When I
don't I have nothing (no JOBLIB or STEPLIB).

In our new z/OS LPAR where we are testing our migration we currently do
have, after a bit of prodding of systems, our application loadlibs in
LNKLST.  There were two additional considerations we had to deal with.  We
had to change our IEASYSxx system parms module from having LNKAUTH=LNKLST to
instead have LNKAUTH=APFTAB.  This is because if you have the former than
anything in the LNKLST because by default an "authorized library" when
accessed via the LNKLST concatenation.  With APFTAB this does not occur.

The other thing we did is we added an LLA REMOVE command to CSVLLAxx so that
our application libraries are *not* in the LLA.  Specifically (member
CSVLLAAP):
REMOVE(EMER.APPL.LOAD,PROD.APPL.LOAD)
We could have chosen to leave them in the LLA, but that would mean doing a
refresh of the LLA each time we make a change to this libraries, and that
didn't seem to have a good cost/benefit return.

Do any of your z/OS shops out there utilize LNKLST for applications
libraries?  I realize many of you are consultants and really have no say how
a particular shop is set up.  Then again, you also probably have experience
with many different ways.  What do you prefer?  What heartaches do certain
methods cause you?

Don't bother telling me it's a petty concern.  I've been told that enough
times already.  :-)

Anyway, I was pretty happy with going down this path.  I then asked a
question on the IMS listserv and someone responded that not only should we
not have application libraries in LNKLST, but we shouldn't even have IMS
system libraries in it!  So now production JCL might look like this:

//MYJOB    JOB
//JOBLIB   DISP=SHR,DSN=PROD.APPL.LOAD
//         DISP=SHR,DSN=IMS.V10.SDFSRESL
//STEP01   PGM=MYPGM
//

His point is that we might have a case where we'd, in one LPAR, want to have
some applications to point to one version of IMS and others to point to
another.  Knowing my shop as well as I do (13+ years in applications
development) I don't think we will ever want to support two version of IMS
(or DB2) in the same LPAR.  That being said I am curious from an
applications point of view what kind of JCL is used to support this
possibility.

In the end, assuming I give up my LNKLST dream, I am thinking I would have
something like this:

//MYJOB    JOB
//         INCLUDE APPL1
//         INCLUDE APPLIBS
//         INCLUDE SYSLIBS
//STEP01   PGM=MYPGM
//

And another job might be a member of "application 2" and look like this:
//APP2JOB  JOB
//         INCLUDE APPL2
//         INCLUDE APPLIBS
//         INCLUDE SYSLIBS
//STEP01   PGM=APP2JOB
//

Here are the "include" members

APPL1 member
//         SET DFSVER=V10
//         SET DSNVER=V910

APPL2 member
//         SET DFSVER=V10
//         SET DSNVER=V910

JOBLIBS member
//JOBLIB   DD DISP=SHR,DSN=PROD.APPL.LOAD

SYSLIBS member
//         DD DISP=SHR,DSN=PROD.APPL.DFSRESL
//         DD DISP=SHR,DSN=IMS.&DFSVER..SDFSRESL
//         DD DISP=SHR,DSN=DSN.&DSNVER..SDSNEXIT
//         DD DISP=SHR,DSN=DSN.&DSNVER..SDSNLINK
//         DD DISP=SHR,DSN=DSN.&DSNVER..SDSNLOD2
//         DD DISP=SHR,DSN=DSN.&DSNVER..SDSNLOAD

For a test job I could then have
//MYJOB    JOB
//         JCLLIB ORDER=(FJS.APPL.PROC)
//         INCLUDE APPL1
//         INCLUDE APPLIBS
//         INCLUDE SYSLIBS
//STEP01   PGM=MYPGM
//

And in FJS.APPL.PROC I would then have a JOBLIBS member:
//JOBLIB   DD DISP=SHR,DSN=FJS.APPL.LOAD
//         DD DISP=SHR,DSN=PROD.APPL.LOAD

Does all of this seem reasonable?  As much as I've argued for use of LNKLST,
this might be something I can live with.  But I'd love to see examples of
how other shops deal with this.

Thanks!
Frank

For a number of reasons, my shop is encouraging the use of symbolics in 
JCL, and I dislike them.  It does make it easier to test production JCL 
in our test environments, but it also makes production JCL harder to 
read when trying to figure out why a batch job went down at 3 AM.

In my shop, the jobs look something like this:

//MYJOB   JOB (whatever goes in the jobcard)
//SYSAFF  CPU=(whatever)
//JOBLIB   DD DSN=HLQ.PROD.PGMLIB,DISP=SHR
//PROC01  EXEC PROC01,P1=HLQ

A typical batch job where I work only has one library in the JOBLIB DD. 
  But I have seen as many as five DSN's under the JOBLIB DD, for DB2 or 
certain vendor products or extremely specialized applications.

We have three test environments before PROD (don't ask why), and they 
are called DTF (Development Test Facility), ITF (Integrated Test 
Facility), and CERT.  So if I need to build JCL for the DTF, the JOBLIB 
DD looks like this:

//JOBLIB   DD DSN=DTF.PROD.PGMLIB,DISP=SHR
//         DD DSN=ITF.PROD.PGMLIB,DISP=SHR
//         DD DSN=CERT.PROD.PGMLIB,DISP=SHR
//         DD DSN=HLQ.PROD.PGMLIB,DISP=SHR
//PROC01  EXEC PROC01,P1=HLQ

If the names seem funny, that's because they aren't exactly the real 
names, and believe it or not, in each test environment we have TEST and 
PROD libraries for that environment.  So there is both a DTF.TEST.PGMLIB 
and a DTF.PROD.PGMLIB.  By convention, we use PGMLIB as the final node 
of our application load library names instead of LOAD or LOADLIB.  That 
was probably an arbitrary choice made nearly 30 years ago.

I remember during the runup to Y2K, I fixed and tested a date routine 
that was a dynamically called assembler subprogram.  Then I tried to get 
it installed into production.  To my surprise I found it did not live in 
HLQ.PROD.PGMLIB, but in a LNKLST library owned by the sysprogs, named 
something like SYS1.LINKLIB3, so my install required an LLA refresh.  I 
hardly knew what an LLA refresh was at the time, because Application 
programmers shouldn't have to think about it.  So installing this one 
little subprogram became almost as complicated as scheduling a POR and 
IPL.  I think they had to stop all production batch processing to do the 
LLA refresh, and that was quite a scheduling problem.

If it had been in the HLQ.PROD.PGMLIB with all the other applications 
programs, it would have been a snap.

I don't mind the typing, and there are a number of advantages to using 
STEPLIB.  For example, you want to test a new version of a dynamic 
subprogram in one step of a job but not all the steps.  I would add a 
STEPLIB DD to that step only.  Basically, you use STEPLIB if you want a 
different library search order for a single step in a multi-step job.

We typically do not use STEPLIB in production jobs, because it's rarely 
needed.  But I don't think we have any standards to discourage us from 
using STEPLIB in production jobs.

Some ideas (and samples).

1) If you leave JOBLIB pointing to your "production" versions and ADD a STEP LIB 
statement for each step that needs to access the test module, the search order 
will do what you want (I think).

2) I have ocasssionally, seen JCL PROCs with something like

//JOBLIB DD DSN=&LIB1,DISP=SHR
                 DD DSN=&LIB2,DISP=SHR

where the PROC defaults BOTH the LIB1 and LIB2 symbolics to the same production 
load library.  This allows for an override to a test library for LIB1 while 
still keeping LIB2 pointing to the production library.  The only time this would 
end up with "extra" performance issues would be if you were going to get a S806 
ABEND because the module wasn't in the production library any way.

   * * * * * *

For HIGHLY modularized applications with LOTS of dynamic CALLS, there are 
"performance" issues with the "call search order".  I have (as I recall) seen 
performance improvements by moving a STEPLIB added to make lots of dynamically 
called programs "accessible" early in the search path.  In such cases, however, 
use of some "preload" facility probably works better.

NOTE:
  If you actually will be running anything under ISPF (batch or interactive) be 
aware that ISPLLIB does funny things to the "normal" CALL search order.
