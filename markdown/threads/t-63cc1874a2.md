FYI, I have never written a line of COBOL in my life although I admit that,

If you take the trouble to check, you will see that until now I have not 
posted any messages to this thread.  When it first appeared, I took a look at 
the IBM page cited and concluded that the exercise was ill conceived and 
decided not to get involved.

Here are some of my reasons for this statement:

According to the specifications, the timing of the "benchmark" is started at a 
certain point and stopped at another point.  In other words it is elapsed real 
time between these two points in the program's execution. Now, if you take a 
program that implements this benchmark and run it several times the elapsed 
time will vary from one execution to the next due to chance, because even if 
no other "applications" are running a typical system (even if it is a single 
user system) has numerous demons, monitors, and the like running which will be 
causing program interruptions every few milliseconds.  This is true for a 
program doing pure computation and all the more true if the program does a 
substantial amount of i/o.  In short the benchmark execution time is a random 
variable.  Therefore, simple comparison of the outcome of two or more runs is 
essentially meaningless.

Now the benchmark specification calls for a comparison of the time with full 
calculation versus the time for i/o only.  It is silent on topic of how this 
comparison is to be done, and the postings I have seen seem to have been based 
on a single run of each type.  This is utter nonsense.  The only way to do 
such a comparison is by running numerous replications of each type of run 
followed by a proper statistical analysis.

Moreover there are other difficulties:  Modern operating systems provide io 
caching with readers and lazy writers.  Medium to high end hardware also 
provides efficient DMA modes that permit overlapping of processing and i/o 
transfer.  Therefore, simply subtracting the average i/o time from the average 
total time gives an estimate of the average non overlapped computation time, 
not the average total computation time.

Any attempt to measure total computation time using similar techniques is 
fraught with difficulty.  Modern processors have onboard cache for data from 
main memory.  Simply performing the computations in a loop with constant input 
data or data generated from a formula with negligible computation time will 
not do.  The "real" program is essentially a loop consisting of: read input; 
compute result; write output.  Thus between any two instance of executing the 
computation instructions, reasonably extensive i/o calls are executed which 
will alter the cache contents so that the number of cache misses will be 
different from the number that would have occurred in a simple loop.  This 
will have a drastic effect on the execution time of the next instance of the 
computations.

A computer system used in telephone billing is almost certain to be a multi 
user system and the benchmark specification is silent on how much additional 
activity is going on in the system while the billing program is running or how 
this is to be dealt with.

I could go on to give other reasons, but this should suffice to show why I 
contend that it is difficult to say what, if anything, the benchmark measures 
as it stands, and why I maintain that considerable additional effort must be 
expended to make it into anything useful.
