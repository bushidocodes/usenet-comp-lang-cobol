To my knowledge, as of 5.0.0 Acucobol does not support report section. I
must also admit that I have never really missed it. I belong in the camp
that does not see the big benefit with the report section. It automates a
few things yes, but in general, it is as easy as to do the report manually.

> Additionally, if you want to use something like Crystal reports,

DI Systemer AS in Norway has a Windows report tool which is unique in the
sense that they provide you with a DLL that functions as a printer engine,
leaving to you as a developer to determine the level of control you want,
ranging from complete (the user won't see the engine at all) to virtually
non existing (all conditions are handled by the engine). The scoop with this
utility is that it totally removes the tedious task of designing the report
from the Cobol developer, so he/she can focus on what counts, namely the
program logic. The report may be designed using a wysiwyg designer, putting
graphics and data where you want. The report may then be designed and
modified by anyone, be it the support department, a value adding reseller
and/or the customer himself.

A generic demonstration:

    load DLL
    load preferred form
    open print
    pump data
    request status
    close print
    unload DLL

The data are transferred like: "FIELD_ID;DATA"

You may have multiple reports and whatever redesign that is done to the
reports there are no need to modify the Cobol source. The idea is that the
Cobol application transmits whatever data it has to the printer engine, the
printer engine then picks what the report says it needs. The beauty of this
is obvious compared to report utilities like Crystal Report and other ODBC
dependent report utilities, you keep control and use of the original logic
in your Cobol application.

There are support graphical basics and you may even do a preview of the
report to the screen.

The engine was originally developed for DI Systemer AS own customers in
Norwegian, but is now translated to English and they are considering to let
other software companies use it.

The software may of course be used with other languages than Acucobol. The
demand is that the hosting application must be 32 bit Windows capable of
using a DLL.

Cobol rules.

Cheesle
