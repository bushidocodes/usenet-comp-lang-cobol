Sure, it is possible to come up with an unusual case that would be
enhanced by parallel processing. However, the meaningful term is
"unusual". The normal COBOL program does not do any operations other
than i/o or sorting that would normally benefit. For that reason, I
don't know of any COBOL compilers that attempt to optimize such things
(the payoff is not worth the effort involved - maybe a small amount of
performance in one of a few hundred programs).

In real COBOL applications most studies show that about 25% of the cpu
time is spent in the COBOL generated code. The rest is spent in the
operating system doing i/o and the like. That is one of the reasons
that few, if any, COBOL compilers do global or parallel optimizations.

At VERB level of thinking it may be difficult to come up with
examples of significant saveings, but what about the case of the
on-line world, where users are waiting for the transaction to finish
before they can go on to the next task?

Even if the transaction processing can be done in parallel there is no
standard way to have COBOL do this for you.

We have implimented this in the networked PC environment by spinning
off long running tasks (reports, updates of multiple records etc) to other
PCs on the network so as to allow the work flow at the keyboard
to resume as quickly as possible.

We could have done this by spawning another task in a multiprocessing
environment (such as Windows), but at the time (pre-windows days) there
were no such OS capabilities, but plenty of networked PCs sitting idle to
which work could be queued.

Now if/when COBOL has a way to lanch a subprogram and continue execution at
the next statement it will be easier to make use
of multitasking OS, or multiprocessor machines.

Actually, almost all implementations have a method of doing this.
With Tandem you call a library routine (called CREATEPROCESS strangely
enough). And, by using READ and WRITE you can communicate with the
process. In the standard you can use MCS (message control system) to
do this (MCS is now optional and it is being declared obsolete in the
next standard). Because so many systems have so many different ways
to do this, the standards committee could not come up with a standard
way to do it. Therefore, other than improving the CALL mechanism to
allow calling nearly any API in nearly any language, nothing was done.
I was sort of chartered to come up with a replacement for MCS that
does such things and is more modern. This would not be until the
standard after the next one (2010?).
