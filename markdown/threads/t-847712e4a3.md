Well, here is one way to do it. I doubt it is the best way, but we did
what we could with what we had. On my project we built a Powerbuilder
front end for a CICS application running on a mainframe. The
Powerbuilder front end sat on top of some C code that handled all the
TCP/IP socket calls. TCP/IP connectivity was also added to the CICS
region. As data came across the socket to the CICS side it would kick
off the appropriate transaction, which generally sent information back
into Powerbuilder (which eventually populated the user's Excel
spreadsheets). This front end was only for a small part of the CICS
application. It works pretty well, and is quite fast.

I'm sure there is middleware available that can do the same kind of
thing. I think you can even do stuff like this using ODBC calls to
connect to DB2, as long as you have a connection to the mainframe. But,
I'm not very familiar with either of these methods so I could be wrong.

Mark

Dave:

We (at KPMG Peat Marwick) have a c/s procurement system developed in
PowerBuilder. We had a client that needed to "interface" their legacy
accounting system to our c/s system. The accounting system was a CICS,
COBOL, MVS, VSAM application. We used a middleware product called
netGateway, from SYBASE, to develop the "bridge" between mainframe and
"pc".

On the mainframe side, we wrote "transactions" handlers in CICS and used
the native SQL-VSAM capabilities from SYBASE to be called by the
client. The Powerbuilder programs accessed the mainframe by "procedure
calls" to the netGateway.

It worked out to be a very efficient method to access the mainframe.

and translates EBCDIC to ASCII, etc. It usually runs on a UNIX machine,

There are *innumerable* choices -- middleware, database gateways, etc.
Which one is right depends a lot on your application(s). For instance,
OLTP or Decision Support?

For OLTP, my favorite is IBM's CICS External Call Interface. It is a
very
lightweight DLL that does not put much of a strain on the PC. If you go
to
our website you can pick up links to IBM's pages on it.

We use this interface in our Application Generator when we generate
PB/CICS
applications. Our customers report that it is *very* fast. It has also
been
trouble-free, IBM has done a very good job.

Your PB/VB program sends a message to a CICS/6000 COBOL program over
TCP/IP.
The CICS COBOL program gets the message in the COMMAREA and replies the
same
way.

Only problem with the ECI is that it is C-oriented making it a little
hard
to use from PB or VB. We wrote a DLL to make it easy (not a real big
deal).
