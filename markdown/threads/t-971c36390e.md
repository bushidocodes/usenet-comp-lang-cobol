I am trying to combine a couple of programs to reduce I/O. Both
programs have sort procedures. Upon the return of the data from the
first sort I write the record for another application.
But I would like to further parse the returned record
and resort for further processing within the same application.
Can I have another sort procedure in the output section?

file XC I cannot release a record to file XC.
Also the input section would also have to be the current
output section.

The coding would be something like:

Return Record <-------------

Parse/Manipulate .... |
Release Record For sort |
Return Another record Loop-----------

YMMV but my COBOL manual (DEC COBOL) says that the output procedure of
a SORT can't have a SORT verb in it.

Depending on operating system you may be able to get by the
restriction such as using VMS mailboxes to pass the output of the
first sort to the input procedure of another sorting program that
parses/manipulates the record and releases it into the next sort.
Probably UNIX pipes, too.

Of course then you will have 2 programs running, and may need to
duplicate a lot of code or have problems keeping things synchronized
with other data.

Keeping it in one program brings you back to writing a temporary file
and reading it back. Skipping over the horror of being bound by a
restriction you hadn't counted on, will the program really take that
much longer to run?

Just my thoughts.

Not sure I quite understood your process Stan, but I think your
trying to avoid re-reading a record after it is in the same order
the 2nd program to process it.
If so, I will make a couple of suggestions that may help.

Let me state first, I don't know Unisys, so some may not be possible.

I gather you have a sort that drops about 900,000 records and the
remaining 100,000 is read by 2 programs with no re-sort of those
same records.

To combine the 2 programs and only read each record ONCE would require
you to make ProgA do the reading and reporting for ProgA and ProgB.
That shouldn't be a problem if you print to 2 report files.

If for some reason you can only have 1 report in a program on Unisys
then ProgA Reads and does ReptA then Calls ProgB passing the record
to it, ProgB (first time in Opens ReptB) and does ReptB. Of course,
ProgB has the Select etc.. for ReptB only, and needs an entry point
for ProgA to Call it on EOF, or pass High-Values in the record to it,
so it can Close ReptB.

If your after something else, let me know, maybe you need to create
an ISAM file with alternate keys, one for ProgA and one for ProgB,
and drop the sorts?
