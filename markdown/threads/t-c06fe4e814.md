2. It is doubtful whether anyone cares.

    PERFORM VARYING INDX FROM 1 BY 1 UNTIL INDX > 10000
        MOVE 'A' TO A-ARRAY(INDX)        [Line 'y']
           END-PERFORM 

Your SETting INDX UP BY 1 
DOUBLES the overhead, because in your version, J is being incremented
as well!

I think the whole discussion depends (used to depend) on the compiler AND
the machine architecture.

The old WANG VS 74 (and I believe 85) compiler used to be smart on
incrementing indexes with literals.  If indexing an array of 500 byte
elements, the WANG would compile SET INDX UP BY 2 as ADD 1000 TO INDX.   So
when using literal values, any multiplication was done at compile time not
run time.   Of course if you set an index to a variable value it had to
generate an inline multiply.   In those days (the 80s) the run time saving
using indexes was generally good enough for us to use indexes as a standard
to cut out the multiplication of subscripts on each use.

The AS400 OPM COBOL compiler (back in 1990) wasn't that smart.  It would
generate the any SET INDX to use a multiply.   In fact, if indexing through
an array of 1 byte elements it would still generate a multplication by 1.

However this didn't really matter.  Because of the way the AS400 describes
its data, subscripts are used raw.  It doesn't have to multiply them by the
element size to get to the correct element.  So on the AS400 subscripting is
(was?) always faster than indexing.

So on the WANG indexing was better and on the AS400 subscripting was better.
Nowadays, for commercial procesing, I don't think it really matters at all.

Rick
