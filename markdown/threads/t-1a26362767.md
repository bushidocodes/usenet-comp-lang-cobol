```
Someone (not me) can give the internal representation in MF-COBOL (as long as
this is on only one platform), but not for ANSI COBOL.

The reason is, that internal representation is up to the implementer - it isn't
part of the COBOL standard.   So what someone sees on an IBM mainframe, Univac
mainframe, PC, or MacIntosh can be completely different.
```

```
Homework, eh? :-)

Off the top of my head, there are these types:

Alphanumeric                    PIC X
Alphanumeric Edited         PIC XBX
Alphabetic                         PIC A
Numeric                             PIC S99v99
Numeric Edited                  PIC $$$.99-
Index                                  OCCURS ...  INDEXED BY ...
Index Data Name               USAGE IS INDEX

Numeric fields can also be USAGE DISPLAY (decimal, full byte per
digit) or USAGE COMPUTATIONAL[-?] which is implementation
dependent, but usually includes binary and packed BCD. Numeric
fields may contain up to 18 digits plus sign. COBOL is probably more
flexible in directly supported data types than almost any language.
Micro Focus COBOL even supports C type null delimited strings.
Some versions of COBOL support procedure and/or data pointers.
Standard COBOL also supports various literal types, and most
compilers support others, such as hex or binary.
```

```
Both USAGE BINARY and USAGE PACKED-DECIMAL are specified by ANSI X3.23-1985.
The format of each is implementation dependent, as is that of COMPUTATIONAL.

However, that standard does require of PACKED-DECIMAL that "a radix of 10 is
used to represent a numeric item in the storage of the computer.
Furthermore, this clause specifies that each digit position must occupy the
minimum possible configuration in computer storage ...".  Thus, it doesn't
say in so many words that a packed-decimal digit *must* occupy exactly four
bits, but three bits is too few and five bits is more than the minimum
required ...
```
