I use both ENTRY and nested programs. They have separate
uses and using the wrong one can cause problems.

Nested programs work well when a procedure is unique to
the containing program and where the procedure can be
understood independently of the containing program. The
contained program has its own working storage and
access to the containing program's working storage, as
required. When I did a mass file conversion for Y2K, I used
a nested program with 12 contained programs. Each of the
contained programs converted one file. In this case, the
procedures were unique to the containing program, which
converted a system. The procedures could be understood
independently of the containing program. It was a good fit!

Entry programs work well when the procedures are common
to many programs and especially when the entry points share
common routines and working storage within the program.
As part of Y2K remediation, I wrote a program with 38 entry
points and 9 common routines. Each entry point accepted
one format for a date or time and returned another format
using the common routines, as necessary. Only one copy of
the program is loaded into memory and that copy is used by
about 45 separate programs. It was a good fit!

Had the entry point program been written as nested and copied
into the 45 separate programs, the routines and common data
would have been loaded into memory as many as four times.

Had the nested program been written as a program with 12
entry points ... well, that never would have happened!

The bottom line is that the reasons for using entry point programs
and nested programs are fundamentally different.
------------------
Rick Smith

The reason ENTRY will not become part of the COBOL
standard is because it is considered "unstructured."
This does not mean ENTRY is not the best way to solve
some problems.

There is an alternative to ENTRY that I believe could be
added to the standard.

MODULE-ID. module-name.
environment-division
data-division
PROCEDURE DIVISION.
    program-entry ...
    function-entry ...
END-MODULE.

That is, a module consists of one or more programs or
functions were elements of the environment division and
data division are common to all programs and functions
within the module.

This is similar to a class definition where object data is
available to all methods within the class. It is different from
objects because only one instance of the data is present
in the run unit.

Availability of the module is controlled by calling and
canceling the module. After the module is called and until
canceled, the program and function entries are made
available to other elements in the run unit.

This, generally, meets the unwritten requirement for
"structured" while providing nearly the same capabilities
as a program using ENTRY. In fact, I could convert the
program, I discussed in my earlier response, from
ENTRY to MODULE with no changes to the source code
in the 45 programs that use the ENTRY program. The
module program would not be as "clean"; but it would
work.
------------------
Rick Smith

You are correct that within the same run-unit you may NOT have both an
external program and a nested program with the same program-id.

From you other post, it sounds (now) like you want a single program to do
all the I/O - but for this to NOT be your main program.  In this case, what
you probably want to do is

1) Define in your main program a WS-area that is a copy of your "record
layout" (probably a copy statement). Also define a field which contains a
"flag" indicating what I/O operation you want performed (e.g. OPEN, READ,
WRITE, CLOSE).

2) Call your subprogram using the record-layout and the i/o-flag.

3) In your subprogram, define your complete file (Select/Assign, FD, etc)
and include the COPY record-layout in the LINKAGE Section.  (Make certain
that your Procedure Division USING statement references all the 01-levels
from your Linkage Section.)

4) Then your subprogram does whatever I/O you want.  Use READ INTO
Linkage-section-layout and WRITE FROM linkage-section-layout statements.

5) Make certain that your main program calls this subprogram "individually"
for each read and write that you want performed - and that you have some way
of determining (in the main program) when you have reached end-of-file (if
you are looping thru reads).

   ***

Now for the big question ...
  WHY do you want to separate the I/O from the main program?  This can be
done, but I don't understand WHY you want to do it.  This (normally?) only
is useful if you have multiple applications using I/O to the same file and
you want a "common" file handler routine.  Is it possible that all you
really wanted (in the beginning) was to use PERFORM statements and isolate
the I/O into separate paragraphs and not separate programs?  Repeatedly
CALLing subprograms has some overhead (depending on your system potentially
SIGNIFICANT overhead).

NOTE:
  If you are using nested programs, look at the GLOBAL attribute and see if
that is of any use to you.

Place something like the following In a separate source
file named sub1a.cbl.

copy sub1 replacing
    sub1 by sub1a
    "sub1" by "sub1a".

This creates a copy of sub1 using the same code as
sub1 but named sub1a. It does so by replacing the
identifier (program-name) in the program-id paragraph.
If the program has its name in the source code, as a
literal, the literal replacement will change that.

In the main program, call "sub1". In sub2, call "sub1a".

Each program has its own flags, file position indicator,
etc.; but, with the exception of the copy statement, they
share the same source code.

I tried the copy statement, above. It seems to work fine.
