Dear reader,

We are currently re-hosting a big COBOL app (from Bull DPS7 mainframe)
to a Wintel platform. NT server with Win95 clients (WinNT4.0 in the
future), developing with NetExpress. While re-developing an
authorization and security mechanism to run on this c/s, the following
are some technical problems:

At application start-up, we want to show the user a dialog box with both
a user-id and a password entry field. In the user-id field we would like
to fill the user-id with which the user did login in the NT-domain
earlier. That way he only has to re-enter his password.

Which method could we use to get the current user-id, in Win95 now and
in WinNT 4 still working? So far I stumbled into the following C code.
Can I use this, after tranlating it 1 to 1 in COBOL, or is there a
better way?

void ShowUserDomain(void)
{
HANDLE hProcess, hAccessToken;
UCHAR InfoBuffer[1000],szAccountName[200], szDomainName[200];
PTOKEN_USER pTokenUser = (PTOKEN_USER)InfoBuffer;
DWORD dwInfoBufferSize,dwAccountSize = 200, dwDomainSize = 200;
SID_NAME_USE snu;

hProcess = GetCurrentProcess();

OpenProcessToken(hProcess,TOKEN_READ,&hAccessToken);

GetTokenInformation(hAccessToken,TokenUser,InfoBuffer,
1000, &dwInfoBufferSize);

LookupAccountSid(NULL, pTokenUser->User.Sid, szAccountName,
&dwAccountSize,szDomainName, &dwDomainSize, &snu);

printf("%s\\%s\n",szDomainName,szAccountName);
}

Second problem is to check that entered user-id and password are
correct. Will the following call do, or is there some better method
(this also in C, to be translated to COBOL of course)?:

NET_API_STATUS NetUserChangePassword(
LPWSTR domainname,
LPWSTR username,
LPWSTR oldpassword, // as entered
LPWSTR newpassword // same as old
);

We intend to check status values, something like:
status x = OK = good user+password combination (no real change of pw)
status y = not authorized, meaning user+oldpassword is wrong
but perhaps we get:
status y = same new as old. Now we wil not know if this system check has
been done AFTER its security check or that we still may NOT conclude
that the change is allowed so the combination is allowed.

To check user+password I found some other alternative, but am not sure
if this works:

First call:
BOOL LogonUser(
LPTSTR lpszUsername, // string that specifies the user name
LPTSTR lpszDomain, // string that specifies the domain or server
LPTSTR lpszPassword, // string that specifies the password
DWORD dwLogonType, // specifies the type of logon operation
DWORD dwLogonProvider, // specifies the logon provider
PHANDLE phToken // pointer to variable to receive token handle
);
LogonType option: LOGON32_LOGON_NETWORK
LogonProvider option: LOGON32_PROVIDER_DEFAULT
Second call:
BOOL CloseHandle( HANDLE hObject // handle to object to close (i.e:
phToken from the LogonUser)
);

I am aware that all these pieces of code contain API functions that do
not run on Win95 but do run at the server side (according to the Win SDK
help file). We could get over that by calling from the client some user
defined functions at/in the server and returning. But in that case, does
NetExpress run under the same userid at the server side?

Please help!
Huib

You may find the easiest thing here to write a service in C++ (provided you
have a compiler) which has a DLL with published functions that you can call
from COBOL. Or write them in COBOL if you do not mind doing the conversion.
Checking passwords is a lot harder than it seems because of access
privileges required for doing so. At least with C++ you have examples and
the Win32 SDK has all the headers etc declared.

The GetUserName API function will easily get the ID of the currently logged
on user, and the LogonUser API function will test that a ID/Password/Domain
passed to it are valid or not.

To call the LogonUser function, the logged on user must hold the
"SeTcbPrivilege" privilege - it is called "Act As Part Of The Operating
System" in user manager advanced privileges. You may also need
SE_CHANGE_NOTIFY_NAME which is called
"Bypass Traverse Checking" in advanced options under User Manager.

This is basically the approach we took, but we were using NT4 Clients not
Win95 so I do not know what problems you may encounter on 95.

Good luck.
Scott
