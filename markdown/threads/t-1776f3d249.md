```
I'm reading a flat file containing display data.  I need to issue SQL queries
comparing some of these fields to database values which are packed.  Therefore,
before making the SQL statement, I must MOVE my display values to packed
fields.

Can't I do this?

Under my FD do this
01  REQ-REC.
      05  JOBCODE-REC  PIC S9(3)V.
      05  REQNO-REC      PIC X(6).
      05  MINSAL-REC      PIC PIC S9(6)V99.

In Working-Storage do this
01  REQ-REC-WS.
      05  JOBCODE-WS  PIC S9(3)V             USAGE IS COMP-3.
      05  REQNO-WS      PIC X(6).
      05  MINSAL-WS      PIC PIC S9(6)V99   USAGE IS COMP-E.

When I READ do this
READ REQ-FILE
    AT END
        SET EOF TO TRUE
    NOT AT END
        MOVE JOBCODE-REC TO JOBCODE-WS
        MOVE REQNO-RED TO REQNO-WS
        MOVE MINSAL-REC TO MINSAL-WS
END-READ

I usually read INTO my working storage, and I understand that it is dangerous
to reference a buffer, but doesn't a read INTO do almost exactly what I've done
here?  Doesn't it read to the buffer and then perform a move?  It seems to me
the only thing I've done is break up the group move of a READ INTO, making
several elementary moves instead.  This allows me to convert the usage right
within my READ STATEMENT.  It seems nicer than reading into working storage and
then performing another move to yet another group of variables in working
storage.

This method works fine in my DB2 project, but my teacher advises me not to get
in the habit of doing this.  So why not?  It's not as if I'm coming back later
to reference the buffer.  I'm doing it right within the READ statement, the way
COBOL does it anyway, no?

RoadRaat
```

If the COBOL environment you have is designed properly (and with IBM
mainframes that fluctuated until finally they got it right with the
version of Language Environment with OS390 2.5 and possibly with PTF's
with that available with 1.3), you can easily find your input record.
With that version of LE all you have to do is find the area associated
with the BLF number in your dump (note if there are SD's in the program
before the last FD the associated DDNAME's are wrong starting with the
BLF associated with the SD).  

Of course a READ file-name INTO record-area will give you bad data in
record-area if bad data is on the file.  Unless you use the same
copybook for both the record area and for the description of the record
on the file, you can get certain other subtle errors and not so subtle
errors when the record description changes in size or key locations.

The idea of reading into a working storage area was to identify the last
record read/processed for the file. This meant that you did not need to
calculate displacements from the start of a buffer in order to identify
the last record read. However, good dump facilities can identify the
last record (and in the case of Abend-aid, the previous record), so it
should no longer be necessary to READ INTO. However, if you are happy
with your method carry on.
