For Raul, here are some advantages of OO classes over called modules, in a
mainframe  (or any other) environment:

1. Classes can be instantiated as often as required and therefore cope with
recursive processing much better than called modules. Yes, with the latest
compilers you can use local storage in a called module to handle recursion,
but it will never be as elegant as an object instance. (Electronic Data
processing on commercial installations rarely requires recursive processing
so I admit this one is weak...<G>)

2. You can easily extend a class without having to regression test
everything that uses it. Only objects that use the new or overridden method
need be tested.  (If you modify a called module EVERYTHING that uses it
needs to be reviewed.)

3. You can add a new interface to a class without having to re-compile every
module that calls it. Only modules that use the new interface need be
re-compiled and regression tested.(If you change the interface to a called
module by one byte, EVERYTHING has to be recompiled... )

In cases 2 and 3 above everything that DOESN'T use the new facilities
continues to function exactly as before and does NOT require to be
re-compiled. (Irrespective of whether the system uses static or dynamic
linking).

OO and Classes allow you to thoroughly test aspects of your system then
"extend" them without having to go back to square one and rewrite. The
system can be developed on a "trial and error" basis, with successful
functionality being "trickled out"to the Users rather than  them having to
wait 9 moths to get EVERYTHING, and then find it is something like what was
wanted 9 months ago, but is no longer relevant because the Business has
moved on in the face of a dynamic market place...
This approach lends itself to RAD much more than the Waterfall, and that is
why most mainframe installations (that have had the Waterfall ingrained in
them over the last 40 years) don't understand the advantages of it.

4. Using non-OO structured code has some inherent safety flaws. Because the
data structure is separate from the code that accesses it, maintenance
programmers can bypass an installation standard routine and simply write
their own code instead. (Sometimes this happens by accident, because the new
bloke simply didn't know there was an installation standard routine for
calculating pricing plans (for example) and did a whole series of complex
calculations that took several hours (of his time, by the time he ran around
and gathered the information he needed to do it...) and duplicated existing
code, and was not entirely correct because there were exceptions that the
analyst he spoke to about it wasn't aware of, and 9 months later in the
middle of the night... <many reading this have been there...>. And the poor
sod who was called out expected the installation standard routine to have
been used and spent an hour looking for a non-existent bug in the standard
routine, then finally realised the calling program did its own calculations
and for the past 9 months the wrong pricing plans have been used... <G>)

Can't happen with OO because the data and the methods are combined and
encapsulated. The Pricing plan object contains it's own code to manipulate
its own data structure. Rather than having to know that there is an
installation standard module, if you want a pricing plan you instantiate the
pricing plan object and invoke one of its methods. (This is second nature to
OO programmers and is consistent across all OO systems.)

OK, it isn't the end of the world if it is a wrong pricing plan, but suppose
it had been a data structure that was controlling heavy machinery and people
were working on it or near it...?

I have no intention of re-running another Religious war in CLC.

I really don't care whether you use OO or not...I am not on commission here
and I have no axe to grind. The reason I am posting is because it gets

OOP is not a silver bullet or panacea for bad programming, but I like it and

started programming in 1965), it is fair to say that when it comes to
programming, I know my way around.

(My experience covers 7 different brands of mainframe  (IBM, ICL, and the
BUNCH), countless PC platforms, including n-tier networks and client/server
and I am fluent in 4 different Assemblers, 3 High level languages, 9
different Operating systems (excluding versions), and a plethora of
application products and tools. As far as experience goes, my full CV is
over 30 pages and my career covers 37 years all over the World...)

I don't believe I have been "conned" into using OO...<G>  And there is no
way now that I would go back to structured programming.

The final and, for me at least, MOST significant advantage of OO...

5. OO enables the construction of components. And THAT is the Keys to the
Kingdom...

This is perhaps one of those touchy areas that depends upon
who one quotes. Parnas in "On the Criteria to be Used in
Decomposing Systems into Modules," 1972, provides a
description of modular programming from Gouthier and Pont
which states, in part, "A well-defined segmentation of the
project effort ensures system modularity. Each task forms
a separate, distinct program module. At implementation
time each module and its inputs and outputs are well-defined."

Armstrong in "Modular Programming in COBOL," 1973, states,
"We have identified, then, three types of modularity:
"1. Applicational--involved with the business at hand, ...
"2. Procedural--concering the basic computing tasks, ...
"3. Structural--the separation of data, code and control."
Of structural modularity, Armstrong writes, "A given module may
be conceived of as consisting of three basic parts: data,
code (or process), and control. Data enter the module as
input, are operated upon by the code, and leave as output."

Both of these early references show that concepts of modular
programming were simple data flow transformations or
process isolation which allowed for extensive testing.

During the 1980s a change in usage ocurred in conjunction
with new languages such as C, Modula, Ada, etc. McConnell
in "Code Complete," 1993, says, "A module is a collection
of data and the routines that act on that data. A module might
also be a collection of routines that provides a cohesive set
of services even if no common data is involved."

Stroustrup in "The C++ Programming Language," 1997,
writes, "Over the years, the emphasis in the design of programs
has shifted from the design of procedures and toward the
organization of data. Among other things, this reflects an
increase in program size. A set of related procedures with
the data they manipulate is often called a module."

Hence, the significance lies in how one defines a module,
old or new. A "called routine" is modular programming
in the old sense. My reference is to modular programming
in the new sense. And in this new sense modular
programming is similar to simple uses of object-oriented
programming.

OO, object-orientation, covers analysis, OOA; design, OOD;
and programming, OOP. The areas of analysis and design
include formal notions of these processes including several
different methods for presenting designs graphically.

RAD is, basically, an informal means for providing solutions
by using ideas from any source, such as different software
development methodologies. Hence, use of RAD could
include ideas from SA and OOA, SD and OOD, and these
may be implemented using ideas from PP, MP, OOP, etc.

OO is formal. RAD is informal. It is not possible to accomplish
the formal by informal means; therefore, one cannot use RAD
to do OO!
