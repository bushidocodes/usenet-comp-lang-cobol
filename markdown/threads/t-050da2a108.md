```
Not a good choice.

When you convert from EBCDIC to ASCII it does all sorts of damage to the
data and this conversion is on a record level not field.

My recommendation is write a simple cobol program that converts the same
layout into clear text and downloading that.  Remember to use sign
separate though.

If you still want to do it from the original source then each character
field will need to be ebcdic/ascii converted and then each COMP3 or
number field.  Good luck this is major fun for a programmer to solve!!!
```

```
You are correct.  In fact, if you have any kind of binary data, then you
cannot convert EBCDIC to ASCII or vice-versa and recover all the data.
There is not a 1:1 mapping between the two character sets, and some of
the binary data will inevitably get zapped.  I once had to send binary
graphics data from a COBOL program on an EBCDIC mainframe to a laser
printer attached to an ASCII PC.  I broke down the stream of binary data
into 6 bit chunks which I translated into EBCDIC characters that would
translate into ASCII characters that, when bit shifted to the right one
bit on the PC, resulted in the original 6 bit binary chunks, which were
reassembled into 8 bit binary bytes for the printer.  I wrote an assembly
language TSR to intercept the printer interrupt and do the translation
and reassembly of the data on the PC, to make the process transparent to
the application software.  As you said, it was major fun! :-)
```

```
If you have ftp,  just convert the character fields to ascii and then do a
binary transfer.  Everything should be OK.

We move data frequently between a VS (ascii), an AS400 (EBCDIC) and
rs6000/hp (ascii) but I must admit we convert binary/packed decimal fields
to display on the sender and back again on the recipient.

Which raises the perpetual moan, why doesn't MOVE CORRESPONDING
automatically work with occurring fields/structures?  We tend to take the
record layout structure and remove binary/packed decimal usage on the output
and use MOVE CORRESPONDING.  This is a no brainer when all the fields are at
the same level but unfortunately that is not normally the case.
```

```
You can use the Cobol statement  INSPECT ... CONVERTING ... to handle the non
comp-3 fields to convert them to ascii.  For the comp-3 fields move them at
the field level, removing the comp-3 usage clause on the target fields if you
want to convert them.
```
