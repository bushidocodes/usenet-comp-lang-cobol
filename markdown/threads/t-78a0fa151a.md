I think if you move '0' to C-C, you will get a blank line between print
lines (a double space); if you move '-' to C-C, you will get two blank lines
(a triple space); if you move a blank to C-C, you get the default
single-spacing; if you move a '+' to C-C, you get no spacing, you print on
top of the previous line.  Anything else, from '1' to 'C' corresponds to the
channel code.  Normally '1' is top of page, 'C' is bottom of page, and
everything in between is evenly spaced in between ( a channel '6' should be
about halfway down the page).  But of course, this is a shop standard that
could  vary from installation to installation, depending on how they built
the print forms control buffers..  or carriage control tapes if they have
old 1403 printers....

I'm not sure if that AFP-5A thing is exactly correct, but I'd guess

I don't think that ADV/NOADV have any affect on being allowed to use
the first position as the control character.  The only difference is
that using NOADV allows you to explicitly define the byte that will
hold the control character, whereas ADV has the compiler do it for
you.  What I mean by this is just because you use NOADV doesn't mean
that if you define LINE-1 as

01  LINE-1.
      05  L1-CC          PIC X VALUE '1'
      05  L1-DATA   PIC X(132) VALUE 'THIS IS A TEST'.

and then say "WRITE OUTPUT-REC FROM LINE-1 AFTER ADVANCING 2" you
won't actually advance to channel 1.  All that will happen is the
control character to advance 2 lines will overwrite the '1' in the
first position (well, I think it would actually do that in the FD
buffer area (OUTPUT-REC), not in the working-storage area).

Keep in mind I'm not at work right now, so I don't have access to
programs and manuals to verify all this!  Also keep in mind that this
all pertains to VSE.  If MVS or OS/390 is different, well, I don't
know.

The ADV/NOADV option determines how an output print line is treated.  If
the ADV option is specified, the compiler prefixes every record with 1
byte into which the control character generated by the WRITE record-name
AFTER ADVANCING ... is stored.  Thus a 132 byte record in the program
source becomes a 133 byte record when written to disk or spool.  If the
NOADV option is specified, the compiler will place the generated control
character into the first byte of the area described by the record
description.  This gets to be more confusing if you inadvertently mix
WRITE record-name with WRITE record-name AFTER ADVANCING ...  (you
definitely do not want to mix WRITE record-name BEFORE ADVANCING ...
with WRITE record-name AFTER ADVANCING ...).  

In a historical digression, natively IBM impact printers were designed
as WRITE record-name BEFORE ADVANCING devices (i.e. the native machine
control codes were designed to print and then advance either 0 - 3 lines
or to a line determined by a punch in a carriage control tape).  The
COBOL compilers prior to COBOL 74 also had an extension WRITE
record-name AFTER POSITIONING control-character where the legal values
were space - advance 1 line, zero - advance 2 lines, hyphen - advance 3
lines, plus sign - advance 3 lines, 1 through 9 - advance to carriage
control tape positions 1 through 9, A through C - advance to carriage
control tape positions 10 through 12.  These values were labeled
American Standards Association values (now American National Standards
Institute).  By convention "1" was top of form and "12" was bottom of
page.     

It gets to be very interesting if you have packages from multiple
vendors and they differ on this option.
