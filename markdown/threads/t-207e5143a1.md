Hey Tony,

I think the rate of change has accelerated to the point where we're all
thinking about tossing in the towel. Relational databases are definitely a
major part of new systems, including but not limited to MS Access.
Volumetrics, dollars, and whatever article some PHM (pointy-haired manager)
last read, pay a big role in deciding which relational database to use

Flexibility is key for new systems. It has to be possible to unbolt a user
interface and replace it with another, or possibly a non-human interface.
There are already many examples where the same back-end process is handling
HTML, GUI, green-screen, touch-tone phones, EDI, and other clients.

As for batch, I'd say it has a diminished role in modern systems. The need
for batch processing is largely driven by performance limitations. Hardware
advances make it possible to deal with data NOW as opposed to waiting until
off-peak hours. Batch processing is being reduced to queuing transactions to
be handled by the primary (on-line?) system which is available 24/7.

I think most large-scale systems are based around transaction processing,
which is nothing new but is finding broader acceptance.  A transaction
represents a well defined unit of work so it can be created and processed in
a multitude of ways.  A well-designed transaction will take care of all the
little details that are related either directly or by invoking other
transactions, so many of the batch jobs that existed to synchronize data are
not required any longer.

Just my $.02.

Karl Wagner
