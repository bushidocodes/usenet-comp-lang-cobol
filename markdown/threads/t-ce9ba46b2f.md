I don't see anything wrong with COBOL, but I don't see what's
wrong with these progressions as well.  You talk as if change
is intrinsically bad.

Many of the things you've mentioned aren't programming
languages:

XML -- universal data format standard.  This is a really good
thing, as there's never been one before.  If it pans out,
though, is yet to be seen.

J++/J# are development environment for a trendy newcomer, Java.
In terms of COBOL, these things are closer to PowerCOBOL and
VisualAge COBOL -- not new languages, but rather
implementations of them.

Delphi is also a development environment that utilises Object
Pascal.  Pascal has been around since 1970, and object pascal
since about 1993.  Object Pascal has grown, but in a controlled
and upwardly compatible manner.  (How many months does it have
to be around before it's no longer the "flavor of the month?")

FoxPro is still around and in use -- it wasn't replaced by VB,
but probably lost some market share to developers wanting a
general development tool, rather than one embedded in a RDBMS.

.NET is not a language, it's a runtime system.  Compilers have
been written for existing languages: VB, C++, Java, Delphi's
object pascal, and other as well, such that those existing
syntaxes may be used with the runtime.

C# does qualify as a language, but in my opinion, only
marginally so.  It's an obvious attempt to rip-off Java, but
also doubles as a well-tuned syntax for the .NET runtime.  I
like it, and will most likely use it, but it has not replaced
C++.

Rewrites are always risky; extracting the business rules is NOT a trivial
thing, even for small programs.

Which is why you almost always find a compromise, where new systems are
being layered over legacy systems, and the legacy systems are being phased
our over a long period. It is not unusual for that period to encompass a decade
or longer.

Which of course simply fuels the fires of controversy- COBOL has been around for
decades with only what amounts to evolutionary change. A COBOL programmer from
1962 would be able to quikcly come up to speed with COBOL in 2002; he just wouldn't
have a grip on the support tools such as editors and so forth. A programmer from 1982
would be perfectly comfortable in the COBOL environment of today in very short order.

Now, take the average BASIC progammer from 1982, it would take that programmer
a signifigantly longer period to simply realize that something like Visual Basic *was* basic.
The paradigms have changed far more radically than for COBOL.

The same, in some measure, is true for a C programmer from 1982, a Pascal Programmer from
1982, or even a Java programmer from 1995. In short, the modern languages that do confer
so many benefits evolve much faster than large systems can handle.

In the decade it takes to move a large system to a "modern" language, that modern language will have changed more than COBOL did in
its entire history. The ROI for that is dramatically LOWER than
most people would assume.

Given that COBOL has been evolving as well, and can now do a lot of what the modern languages
can do, well... other than predjudice in the minds of inexpereinced managers, there is little reason
to move off it. It is certainly more efficient than any Java implementation, more understandable than
any C or C++ implementation, and it still does the job.

Some areas are natural points for migration away from COBOL; we services for example, are much better implemented in Java than
COBOL. However, the backend logic that is implemented by web services is
often (though not always) much better implemented in COBOL.

What does this say to me?

We ought to be writing interfaces that allow COBOL to be used by "modern" products as easily as
those same products currently use a database. And those interfaces are pretty much going to need to
be open-source, because no compiler manufactuer is going to be able to keep up with the requirements,
as well as most COBOL manufactuers are reluctant to give up a potential profit center.

That's just some of the thoughts on the subject I've been developing by the way - please don't anyone
take that as scholarly dissertation!!

One big problem with estimates is that too many people don't know the precision
of the basic assumptions behind the figures.  Maybe they need to practice simple
slide rule practice.

species will use broadband radio and keep using it.  After making a bunch of
WAGs, we can plug in precise numbers and get lots of precision in an answer that
is still a WAG.

assumed.  And they need to consider strategies for what happens when the
assumptions are wrong.   A 20% chance of your CEO telling you to go with his

Not so. Earlier in the thread, one of you referenced HTML as being a "form" of SGML as well.

HTML is a DTD (Document Type Description).  People think of it like a language in and of itself, but it is not. There are many DTD's
out there, and the job they perform is to describe the content structure of a document. A DTD is not a language, nor is it a "form"
of SGML, it is simply the set of rules and definitions of how a document is formed, ordered, what elements it may contain, and what
the possible attributes of those elements are.

The fancy graphics and such that people tend to associate with HTML actually have not a single thing to do with the HTML; they are
how the presentation software chooses to present the content of the HTML document. That may seem like a trivial difference, but if
you spend a little time thinking hard about it, you will realize that it is a really tremendous amount of difference.

XML, on the other hand, is an --> extension <-- of SGML, and actually includes new fucntionality. SCHEMA's replacing DTDs (which, by
the way, XML still supports) are the major innovation. They pretty much provide the reason for the 'eXtensionable' part of the XML
name.  Like SGML, XML has documents that each have a specific and well defined document type. Unlike SGML, that document type may be
defined in a DTD *or* in a Schema.


> >XML has a defined standard, just as Cobol language has a defined
> >standard.  To be useful Cobol has to be written into a program.  For
…[10 more quoted lines elided]…
>

No - a standard is a set of rules that are provided by an committe or other organization with the authority to set standards for a
particular subject matter.

In the sense that you refer to the organization and content of an XML document being 'standard' - there is no standard. The DTD or
Schema defines that.

In the sense that any XML document must be able to be parsed and validated against it's DTD, there are a good solid set of
standards. Again, XML is a superset of SGML and acts and behaves as such.

I edited out the rest of the "discussion" below, but let me point out the difference between traditional EDI solutions and XML.

XML, by definition, carries with itself the description of it's document. In a way, it is a self defining document.

That does not mean that it describes the meaning of each element in the document or the meaning of the content of an element; just
where the element may be, how many times it may appear, what the attributes for that element are, and other similar information.

The parties exchanging information don't need to agree on the structure of the document, since they can always determine the
structure of the document on the fly.

The parties exchanging information DO need to have a  pre-existing agreement on the meaning represented
by each element in the DTD or schema. In other words, they have to agree on a standard as to exactly what  a particular element or
attribute of an element means before they can exchange any meaningful communications. They simply do NOT have to have pre-existing
knowledge the structure of the document exchanged (though it would rare to find someone exchanging XML documents that does *not*)
since the document, by definition, carries that information along with it.
