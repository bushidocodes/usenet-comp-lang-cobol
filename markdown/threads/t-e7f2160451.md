About a year ago, I used Fujitsu COBOL 3.0 to compile a callable COBOL
program, originally running on our IBM mainframe, as a dll.  The COBOL
program has no I/O except through the LINKAGE SECTION.  The resultant
dll is called from VB6.  That dll, and all the Fujitsu 3.0 run-time
dll's, are kept in the same folder with the VB6-produced .exe when
running production, and in the same folder as the VB6 .vbp when doing
development work.

This past week I modified the COBOL source and recompiled the dll.
The new dll works fine on all tested PC's when running the VB6 .exe.
And, on my PC, it's fine from the .vbp.  However, other developers are
getting "file not found" VB errors when calling my dll.  They can get
past these errors by putting the full path name in the Declare, such
as "\\Server1\Test\MyID\MyApp\new.dll", rather than just "new.dll".
However, we didn't need the full path name previously.  How can I
insure that Windows 95 developer PC's automatically look for the dll in
the folder where the program is being executed from?

By the way, I tried adding:
     CurDir = App.Path
to the VB code but that didn't help.  Registering the dll with regsvr32
also did not help.

Thanks for any ideas.
