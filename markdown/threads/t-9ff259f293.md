Accessing the data using 'normal' odbc will surely increase network-
traffic because all comparings will be done on client site AFTER
retrieving all datas (hopefully retrieved by index-optimization).

But I can't see any data integrity problems if your updates are using
the right strategy. Will say: Your update/delete clause has to be set to
'key and updateable columns ' or (depends on your requirements) 'key
and modified columns'. But never use 'key columns only'.
Since we're using the ParkWay Server (avaible for Unix only as far as I
know...) our PB-Applications are running at least 10-15 times faster
than accessing the files using direct network access. In fact, users
can't see any difference accessing cobol ISAM-files or accessing tables
on a Sybase Server.
Now it makes a lot of fun to replace existing (very old) cobol-progs
thru PB-applications....

One more problem: Be sure that your ODBC-Driver and your Cobol programs
are both using the SAME file-handlers. If not, index corruption will be
a big problem.

HTH - Stefan Meyer
