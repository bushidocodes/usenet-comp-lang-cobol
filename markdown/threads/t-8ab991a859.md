```
Can someone answer an IBM MVS type question regarding files with Generation
Data Groups (GDGs)?

I create a GDG base, then write a file named A.B.INFILE(+1) containing Rec1:
AAA; Rec2: BBB; Rec3: CCC.  This creates file A.B.INFILE.G0001V00.  I then
write another file A.B.INFILE(+1) containing Rec1: DDD; Rec2: EEE; Rec3:
FFF.  This creates file A.B.INFILE.G0002V00. I then run an IEBGENER to copy
A.B.INFILE without any generation specified (to select all generations) to
file A.B.PROCFILE.  File A.B.PROCFILE now contains 6 records: R1: DDD; R2:
EEE; R3: FFF; R4: AAA; R5: BBB; R6: CCC.  When I now process file
A.B.PROCFILE I have records out of order, processing newer records before
older records.

Does anyone know a way to copy all generations of A.B.INFILE **in order,
oldest first** to one file, without knowing how many generations of
A.B.INFILE actually exist? And/or, does anyone know a better newsgroup on
which to ask this question?
```

Interesting question - I haven't got a clue !

There are plenty of work arounds including adding timestamps etc, for
sorting, or using intermediate datasets, to stage into the generations in
"reverse order" so they copy in the correct sequence.

My MVS book is at work - I'll check for an option on Monday. If you can use
the idea of a sort, why not scrap the IEBGENER completely, and just use
Syncsort, to cut out an extra step?

I hope someone here will post a command option, 'cos I love to know!

Run an IEFBR14 step to pre-allocation a sufficient number of generations,
with DISP of MOD.  Then hard code the generations in reverse order in the
concatenation listing. This may not be the cutest solution, but with 2 or 3
comments, should be understood at 3:30am, when it crashes, unlike an obscure
MVS GDG option.
