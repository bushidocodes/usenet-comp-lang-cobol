```
We are having the following problem with some Telon OS/VS COBOL IMS program:

" We have experienced some working storage parameters not being refreshed
each time a new transaction is being scheduled in acceptance environment.
We have done some investigations and found out that if we manipulated the
linkage parameter, different effects will be produced.
If we don't specify REUS in the JCL for compile and link, the reusability
will become UNSPECIFIED and the working storage variables will get refreshed
If we specified REUS in the JCL, the reusability will become SERIAL and
the results are unexpected -- i.e. working storage variables not being
refreshed, each time we command line to the screen, the old values remains
in the working storage and we will pick up whatever left in the working
storage. We use the SERIAL option and it works only in our development environment
but neither in BTS nor in our user acceptance environment.
The UNSPECIFIED option works in all cases."


Although we can get around this problem by changing the linkage parameter, we don't
have a comfortable feeling as to why this is occuring. Can anyone enlighten us as to
when we should use the different REUS options.
```

```
Well, don'tknwo about Telon but in our PL/I - IMS environment we always
use REUS and EXPECT variabeles not te be initialised. The reason is that
a normal IMS program looks like:

GET MESSAGE

While any-available

process message
send reply

GET another MESSAGE

End

If your system is busy a program like this will process multiple messages
from various users so you can't rely on variables being initialised.

REUS will make sure that an already loaded program will be used again
on the next schedule, so it doesn't have to be reloaded from disk.
```

```
OS/VS COBOL programs are not naturally reusable. Such programs should
always be front-end initilized. Marking them REUSEable by the linkage
editor can cause trouble such as you have found. This is especially true
if the program stategically mis-positions its initialization.

Two cases can occur. The first is that you get a second message from IMS
to a single scheduling of the program. The second is that IMS re-uses a
"completed" program due to the REUS attribute.

For the first situation, here is one way you can test.

Check the PROCLIM parameter (in the IMS SYSGEN) for the transaction in
question. If it has a value (of the first sub-parameter) of 2 or greater
(IMS 3.1 or earlier), or 1 or greater (IMS 4.1 or later), the
transaction's program can be serially fed multiple messages in a single
scheduling of the program.

Set the transaction for the program to a normally unused IMS class - on my
primary site I use class 39 for such experiments - check with your IMS
support staff for help with this.

Make sure no message region is active for this class.

Enter the transaction multiple times. If the transaction is generated as
"RESPONSE MODE" (likely for Telon types), you wil have to use multiple
terminals.

Temporarily set a message region to process the "class 39" (or your
choice) transactions. By only having a single message region (active for
the class), and multiple messages queued, all of the messages should be
processed in the single region. The program should only be invoked a
single time, loaded once, and the trans passed one at a time to the
program.

If this fails, your program is not refreshing the working storage each
time it loops back to read a message from the IMS queue.

The second case is more difficult. Simple answer if the above works yet
it fails in production (or under load), IMS is reusing the module after it
has returned to IMS.

The best cure here would be to move the application to VS COBOL II with
the RENT compiler option. Usually easily done for TELON applications.
With this option, the load module is made truely re-entrant (and of course
re-usable) even if the program logic is not re-entrant. The details take
more typing than I can hack out tonight, but e-mail if you are still
confused.

The specification of RENT and REUSE are just as dangerous as the
overriding of AMODE and RMODE for a MVS load module.

Quoting "Planning COBOL Migration" Topic 4 (Edge Information Group, via
Carl Gehr at 734··.@com··e.com)

Those who lie to the linkage editor and force AMODE and RMODE
fall into two camps: the very clever who know exactly what they are
doing
everyone else who does not understand
that you should
not lie to the great MVS!
```
