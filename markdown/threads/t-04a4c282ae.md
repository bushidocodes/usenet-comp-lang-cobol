Hi Arno,

You should always indicate your operating-system and the compiler-version you
are using!

I suppose you use a big IBM because BINARY is an IBM-extention to
COBOL!

In IBM VS COBOL II, the terms 'COMP', 'BINARY' and 'COMP-4' are synonyms and
it is allowed to use them as 999V99. Anyway you should not use binaries with
decimals, because they should represent signed or unsigned integers!

It is not very easy to use COMP-4 because it depends on the compiler-option
'TRUNC(BIN)' or 'TRUNC(STD)' weather you can store a binary halfword (1 to
32K) in a 'PIC 9(4) COMP-4' or just the range from 1 to 9999, even if such a
field always uses 2 bytes. PIC 9(1) to PIC 9(4) uses 2 bytes, PIC 9(5) to PIC
9(9) uses 4 bytes.
If you use TRUNC(STD) and you get (as parameter or from a database) the value
10,000, you just compute zero (the high order number is truncated!). And
TRUNC(BIN) is much more CPU-time consuming than TRUNC(STD).

So you'd better use 'COMP-3' (equals 'PACKED-DECIMAL') if you compute and
'USAGE IS DISPLAY' for your output and parameters!

Cheers

Daniel

Not copying all the stuff that has come before, but just answering "what is
an extension" issue:

USAGE BINARY is a part of the current ANSI Standard (as is USAGE
PACKED-DECIMAL). USAGE COMP (or COMPUTATIONAL) has been a part of the
Standard for quite a while.

USAGE COMP-1, COMP-2, COMP-3, .... COMP-n are *not* a part of the Standard
(but were "sort of" a part of the old CODASYL JOD).

Whether COMP = BINARY is not an extension but is "implementor defined"
which means that on some compilers it is true and some it is not.

P.S. (In reply to another part of this thread) If you ARE using IBM
compilers (on either S/390 or the Workstation) then the TRUNC(STD)
TRUNC(BIN), etc compiler option applies equally to USAGE BINARY, USAGE COMP,
and USAGE COMP-4. Almost all other compilers have some comparable option
concerning TRUNCation to the PICTURE clause for "binary" items - but how and
when this works is implementor specific.
