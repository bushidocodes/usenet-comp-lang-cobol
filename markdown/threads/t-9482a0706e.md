        01 FILE-ID-AREA             PICTURE X(50).
        01 BADFILE-AREA             PICTURE X(50).
        01 MESSAGE-AREA.
            02 DISPLAY-PART         PICTURE X(60).
            02 FILLER               PICTURE X(512).
        01 CURRENT-LINE             PICTURE 99 VALUE 2.



        FILE-RECOVERY-ROUTINE.
            MOVE FILE-ID-AREA TO BADFILE-AREA.
            INSPECT BADFILE-AREA CONVERTING
               "abcdefghijklmnopqrstuvwxyz" TO
               "ABCDEFGHIJKLMNOPQRSTUVWXYZ".
            INSPECT BADFILE-AREA REPLACING ALL ".IT" BY ".ER".
            MOVE SPACE TO MESSAGE-AREA.
            DISPLAY "Recovering: " LINE CURRENT-LINE COLUMN 1.
            DISPLAY FILE-ID-AREA LINE CURRENT-LINE COLUMN 13.
            DISPLAY BADFILE-AREA LINE CURRENT-LINE COLUMN 30.
            ADD 1 TO CURRENT-LINE.
            CALL "CFURCOV" USING BY REFERENCE FILE-ID-AREA
                                 BY REFERENCE BADFILE-AREA
                                 BY REFERENCE MESSAGE-AREA.
            INSPECT MESSAGE-AREA REPLACING ALL X'00' BY " ".
            DISPLAY MESSAGE-AREA LINE CURRENT-LINE COLUMN 1.
            ADD 2 TO CURRENT-LINE.

The above expects an ISAM file with the extension ".IT".  place the 
fully qualified name in BADFILE-AREA.  It will leave a new file under 
the original name, and rejected records in "samename.ER". Note the fudge 
allowance for lots of data back ... it is a pretty dumb routine, and 
will return a null terminated string in message area. Only the first 
50-60 characters are of any interest, but it will mess up a whole lot of 
stuff if you do not allow enough space. The 512 byte allowance seemed to 
prevent "problems".

Donald

When testing a file status you should not test both bytes for "00",
you should only test the first byte for indicating success.  When the
first byte is '0' then the operation has been successful but the
second byte may contain additional information.

For example "02" indicates a successful read or write with additional
information that there is an allowed duplicate alternate key.

> What should we do to detect corrupted registers in the file?

When you say 'it doesn't return an error' are you saying that the
file-status is "00" and the program continues, or is it that the
program completely abends before it gets to the IF ?

Of course this is Fujitsu so the DISPLAY will occur on the 'console'. 
Has the console been suppresed or lost so that you are just not seeing
the 'ERROR' ?

1. Without DECLARATIVES, the above code will generate an on-screen error
message UNLESS you've put an entry in the COBOL85.CBR file to suppress them.
In other words, the compiler puts scary stuff on the screen BEFORE it
processes your file-status code.

2. You can check the internals of a file via:

   CALL 'CFURCOV'
    USING BY REFERENCE WSIDISAM
    BY VALUE WSINDX
    BY REFERENCE WSMESSAGE-AREA
    RETURNING WSRETURN-VALUE.
