The ALTER statement is one of the most insideous COBOL statements as it is
used to modify *code* at run-time. In the following example:

PARA-A.
GO TO PARA-B.
PARA-B.
ALTER PARA-A TO PROCEED TO PARA-C.
. . .
PARA-C.
. . .
the statement "ALTER PARA-A TO PROCEED TO PARA-C" will modify the contents
of PARA-A to be effectively "GO TO PARA-C" instead of "GO TO PARA-B".

It is now "obsolete," but you still occasionally run across it in programs
which have not been compiled in a long time. It was most often used for
bypassing some initialization logic as in this example where PARA-B is only
to be executed the first time through. And, of course, it is only used in
concert with GO TO statements which most of us no longer recommend [barring
event driven programs for which there is often no acceptable alternative].
If you need to recompile these old programs, the below will do exactly the
same.
01 ALTER-SWITCH PIC X VALUE "B".
. . .
PARA-A.
IF ALTER-SWITCH = "B" GO TO PARA-B
ELSE GO TO PARA-C.
PARA-B.
MOVE "C" TO ALTER-SWITCH.
. . .
PARA-C.
. . .
Of course, I'd use better paragraph and data-names than the above generic
example, but this should illustrate it idea.

ALTER comes from the days of punched cards, paper tape, and magnetic tape.
It might take 2 or 3 days to compile a program, and certainly you could not
compile a program without the bringing down the business system itself.
Furthermore, there were virtually no resources available for testing. It
was desirable to have a program that would run both on the initial startup
of a system and also on all subsequent day-to-day runs of the system.
If you have a new master file to create, the first time you run the program
you might want to open it OUTPUT. For all subsequent runs, you would open
it INPUT or I-O. The ALTER changes an unconditional branch within the
object code (executable) so that the instructions following the ALTER are
only done once during the life of the compiled code. All goes well until
some unsuspecting soul comes alongs, adds a few minor code changes, and
then *recompiles* the program. The post-ALTER instructions are now back in
the object code, and the next time the program runs, it reinitializes the
master file. And probably nobody knows why.

If you haven't gotten the idea already, ALTER is a dangerous verb and most
sites have patched their compilers to remove it.

Depending on the original intentions of the ALTER writer, you might add
File Residency checks; or you might write a one-shot program to initialize
the master; or you might bypass altogether the instructions which would be
invoked by a fresh ALTER.

Wrong!!!!!!

The ALTER verb does NOT change the object code itself, only the copy of the
object code which is in memory executing. After all, in many cases (I was
there) the object code was contained on punched cards. And a running
program could not retrieve the object deck from the output bin of the
reader and repunch a few columns in one of the cards.

The ALTER verb only changes the copy in memory so the GOTO goes somewhere
else. It could be reALTERed later if desired as well. See my earlier
posting on this subject.

Actually, I always liked ALTER - as you see from the other response, primarily it was
used to save a few machine cycles - rather than testing a switch each time into a
paragraph, you could avoid entering the paragraph altogether by altering the flow of
control of the program. The CA-Realia compiler (which is written in COBOL) has no ALTER
statements - but I have been tempted at several times to use them in time-critical
portions of the processing to speed things up a bit (in fact they did speed the compiler
up by about 5% when I tried them in the input parser, but merely a 5% gain was not worth
the maintenance headaches of introducing them).
If you are doing something where every microsecond counts - use assembler language.
COBOL applications should be maintainable.

OK. This is not a generally approved of verb, nowadays. Behind each
paragraph label in COBOL (I can only speak for IBM mainframe versions
with any authority) was a NOP or BC 0,xx assembler instruction. The
effect of using ALTER is to change the NOP to an unconditional branch.

Thus, you could have (and it was a common use of ALTER at one time)

READ-FILE-A SECTION.
RFAS-P1.
ALTER RFAS-P1 TO PROCEED TO RFAS=P2.
OPEN INPUT FILE-A.
MOVE 'R' TO FILE-A-STATUS.
RFAS-P2.
READ FILE-A INTO WS-FILE-A-REC
AT END
GO TO RFAS-P3.
GO TO RFAS-EXIT.
RFAS-P3.
CLOSE FILE-A.
MOVE 'C' TO FILE-A-STATUS.
(next line only if you want to really confuse)
ALTER RFAS-P1 TO PROCEED TO RFAS-EXIT.
RFAS-EXIT.
EXIT.

This is, however, self-modifying code and, as such, does not provide you
with re-entrant code. It is not necessary to have re-entrant code for
batch processes, though.

You'll probably get the "do not ever use this verb, on pain of death
argument" until you're sick of it, but I believe that was not the
purpose of your question. Ignore the hysteria, listen for the truth.

It is not recommended that you use ALTER too freely because you do run
the risk of confusing anyone else who comes along after you. It is
usually recommended that you never use it, though I can still see times
when it can be useful. But then, I've never been keen on fascism of any
sort.
