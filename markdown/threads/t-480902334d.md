for a Video rental company, and I'm having a few problems. Here is my 
customer file:

       FD  CUSTOMER-FILE
	       LABEL RECORDS ARE STANDARD.
	   01  CUSTOMER-RECORD.
	   05  CUSTOMER-NO         PIC 9(6)    VALUE ZERO.

	   05  CUSTOMER-FIRST      PIC X(30)   VALUE SPACE.
	   05  CUSTOMER-ADDRESS    PIC X(30)   VALUE SPACE.
	   05  NUMBER-VIDEO        PIC 9       VALUE 0.

Whenever a customer rents a video the NUMBER-VIDEO should increment.
However, it won't let me do an ADD 1 TO NUMBER-VIDEO. It will compile,
but then crash at that point.

What I *can* do is:
               ADD 1 TO WS-NUMBER-VIDEO.
               MOVE WS-NUMBER-VIDEO TO NUMBER-VIDEO.

However, WS-NUMBER-VIDEO is initalised each time, and therefore only the 
number 1 will be moved to NUMBER-VIDEO, even if the customer rents 2 or 3

I then tried:
               MOVE NUMBER-VIDEO TO WS-NUMBER-VIDEO.
               ADD 1 TO WS-NUMBER-VIDEO.
               MOVE WS-NUMBER-VIDEO TO NUMBER-VIDEO.

and the same thing happens, compiles, but then crashes at that point. 
Both times, the error was:

Execution error : file 'ass2a.int'
error code: 163, pc=B4E, call=1, seg=0
163     Illegal character in numeric field 

It seems to me that I can't manipulate the fields of a particular record 
directly, it has to be done via a variable. Nor can I read a field *into*
a variable. How do I get around this ?

BTW, I'm sure I'm using the READ CUSTOMER-FILE commands correctly, 
because I *can* do a DISPLAY NUMBER-VIDEO, I just can't ADD TO, or MOVE 
from the field.

Please help. BTW I'm using Micro Focus COBOL for UNIX V3.1 revision 049.

Thanks in advance.
```

In a FD, VALUE clauses are treated as comments. NUMBER-VIDEO
likely contains a SPACE. A SPACE is an "Illegal character in (a) numeric
field".

After opening the file, do one of the following:
    INITIALIZE CUSTOMER-RECORD
or
    MOVE ZERO TO NUMBER-VIDEO.
Either of these instructions will ensure that NUMBER-VIDEO contains
a legal value.

On another point, you have triggered a pet peeve of mine by using
CUSTOMER-NO and NUMBER-VIDEO. I avoid the use of "number"
in any form, I would suggest CUSTOMER-ID and
VIDEOS-RENTED-COUNT as replacements. See the ng thread
"WS prefixes" posts by Judson McClendon and myself.

HTH

Your problem is that the VIDEO-NUMBER in the record
was not set to zero when you created the record. It
probably contains a NULL instead of the ASCII ZERO.

For the arithmetic to work, it *must* contain a valid
numeric value --- IE: one of the digits 0-9.  The correct
way to create the record (and this should go into the routine
where you originally create that record) is to initialize
each field before you write it.  The alpha fields should
be set to spaces (Or a name like "new csutomer") and the
numeric fields should be set to zero, or some other legal
value.

Some Cobols automatically initialize working storage, even
though that is not to spec.  Either that is the case, or you
set the WS value to zero with a value statement, which is
why it works in WS. The move of a WS value will aslo work, as
that is how you initialize it.

There have been several posts about trying to do arithmetic on fields in
your FD, but I think that either they - or I - have missed the point of your
application.  Am I correct that each record is suppose to have a higher
number than the previous record?  If so, then you need to define your
"counter" in Working-Storage any way and move it to your FD.  You can do
this explicitly - or you could do it by doing a WRITE FROM.  In this case,
your whole record description would be in working-storage; you would
initialize all the fields before ANY record was written; and you would build
each record in working-storage - including an ADD 1 to your counter.

Just a little offtopic, but why do people prefer to use ADD, SUBTRACT,
etc., rather than COMPUTE? 

The assembly code for COMPUTE usually comes out the same or even a
little more efficient than the standalones, and the infix notation 
is more common, albeit, not 'English.'

I occasionally do use FD data items for computational purposes,
usually when incrementing counters. 

   READ MASTER-CONTORL-FILE ...
   COMPUTE THIS-INDEX = THIS-INDEX + 1
   MOVE THIS-INDEX to WS-STORAGE-SOMEWHERE
   WRITE MASTER-CONTROL-FILE-RECORD...

(With appropriate error checking on the reads and such.) 

Common wisdom has this as being a BAD thing, but most modern compilers
have no problem with it. I do note that people sometimes expect the 
FD storage to be valid *after* the write, and *that* isn't so.

Unless there is something I don't know about (possible! :) FD space
is nothing special, just an allocated memory area like any other. 
I don't see a lot of advantage to moving a value to WS, computing
it there, then moving it back to FD storage. (?) 

-Paul

COBOL was specified as a common business oriented language.  Early on, the
math educated  designers were sure that ment that the users were
only trained to add, subtract, etc.  They felt that FORTRAN took care of the

not the early designers. COBOL was and continues to be a work in progress.

Unless I have a performance issue, I use whichever construct fits best
with the function of what I'm doing.   If I am incrementing, I use ADD
(or more likely use varying or even set).  Complicated arithmetic is
done with a COMPUTE.

Sometimes there will be a difference in the following code:

MOVE ZERO TO MY-FIELD.
COMPUTE MY-FIELD = ZERO.

This is dependent on your environment, and the difference (if it exists)
has to do with internal representation of signs.

paulr wrote:
>Just a little offtopic, but why do people prefer to use ADD, SUBTRACT,
>etc., rather than COMPUTE?

I know of three reasons why people may avoid COMPUTE.

 1. On some compilers COMPUTE is not very efficient.

 2. There are times when you need extra control over intermediate
    results, such as rounding.

 3. If the particular COBOL compiler's COMPUTE logic isn't great,
    you can get intermediate underflow/overflow on chained multiply
    or divide operations unless you are careful.  For example, it is
    a good idea to code B * C / D rather than B / D * C, especially
    if you are carrying the result to several decimal places.

Personally, I use whichever construct seems most appropriate for the
specific calculation.  If the calculation is a single simple operation
I usually use the operation directly rather than COMPUTE, because it
seems to make the logic more obvious.  For more complex computations,
particularly when the expression makes it clearer, I use usually use
COMPUTE, except when the reasons above apply.
```

For simple calculations (such as the increment of the video rental count
in the problem we are considering), it is both simpler and clearer to
use ADD instead of COMPUTE.

Consider:

   add 1 to rental-cnt

   vs

   compute rental-cnt = rental-cnt + 1

So superior is the COBOL approach that most languages invented since
have been careful to follow its example:

   rental-cnt += 1

   or   

   rental-cnt++
