A recent thread in this forum requested help with setting Environment 
Variables (EVs) from COBOL.

(Please see my response to Lee Unterreiner in the "Newbie - can Fujitsu 
COBOL issue a DOS SET command?" thread if any of the following confuses 
you...)

The trouble with using ACCEPT and DISPLAY is that the EVs do not persist. 
They are lost as soon as you issue a stop run.

Below is a Windows Script File (.wsf) which takes 2 parameters. Put this 
file either in the current directory or somewhere on the system path.
(The description below is from a viewpoint of Win XP, but the script works 
fine in Win 7 and Vista also, as long as you have the necessary Registry 
write priveleges.)

The parameters are a value pair representing the name of an EV and the value 
you want set in it.

So, if you wanted to set an EV called "MyTestEV" to a value of "My Test 
Value" you would type the following on a Command line:

setEV  MyTestEV "My Test Value"

If you now type a SET command you will NOT see the new EV. However, if you 
close the command box, re-open, and type SET again, you can see that the EV 
has indeed been created/updated, and it has persisted beyond the process 
that created it.

Had you used the Windows GUI to check the EVs you would have seen it 
instantly. (Start>Control panel> system> advanced tab: Environment 
Variables)

OK, here's the script code (don't worry if you don't understand what it 
does; all you have to do is execute it at a command prompt...):

' This script sets an Environment Variable on Windows systems
' On entry the first parameter contains the variable name and second 
contains the required value.
' The "USER" EVs are set permanently by this and will persist after the 
process completes.

' Author: Peter E. C. Dashwood  Date: July, 2011
'
<package>
<job>
<script language="VBScript">
dim EVName, EVValue
Set oArgs = WScript.Arguments
EVName=oArgs(0)
EVValue = oArgs(1)
Set WshShell = WScript.CreateObject("WScript.Shell")
Set objEnv = WshShell.Environment("User")
On Error Resume Next
objEnv(EVName) = EVValue
</script>
</job>
</package>

Cut and paste the above into a text file and call it: setEV.wsf  You have 
now created yourself a new command line command called "setEV"

Once you are satisfied that it works, the next step is to call it from 
COBOL.

There are a number of ways to activate things from COBOL. Sadly most of the 
examples posted use the Win API directly and document for VB or C++. The 
example below uses the COM interface which is availabe in both Fujitsu and 
Micro Focus COBOLs. (The code is necessarily a little different between the 
two COBOLs, as Micro Focus does not use an intermediate COM handling Class 
and Fujitsu does (*COM). Nevertheless, it is a lot easier to create a COM 
object and invoke methods on it in either of these COBOLs, than it is to 
interface to the "StartProcessA" API function with the complex structures it 
requires.

Here is the Fujitsu version of the code:
--------------------------------------------------------------------------------
 @OPTIONS ALPHAL(WORD)
 IDENTIFICATION DIVISION.
 PROGRAM-ID. SetEVs.
*
*  This program demonstrates the use of the Windows Shell
*  component, which is a much easier way to
*  execute applications from COBOL than using the StartProcessA
*  Windows API function. You can start ANY executable using
*  this component, including Batch files, Scripts, and other
*  applications.
*
*  In this example, a Windows Script File is being used to
*  set a persistent Environment Variable. You can dynamically
*  decide what EV you want and what value you want it to have.
*  When the COBOL program completes, the EV persists and can be
*  referenced by any other processes.
*
*  Author: Peter E. C. Dashwood, July, 2011.

 ENVIRONMENT     DIVISION.
 CONFIGURATION SECTION.
 REPOSITORY.
     Class WinShell AS '*COM'.
*------------------------  DATA   DIVISION  -----------------
 DATA DIVISION.
 WORKING-STORAGE SECTION.
 01  ShellObj OBJECT REFERENCE WinShell.
 01  ShellProgID     pic x(30) value 'Shell.Application'.
 01  ShellReturn     pic s9(4) comp-5.
 01  parm1           pic x(15) value 'cscript.exe' & x'00'.
*> parm1 has the name of the executable you want.
 01  parm2           pic x(50) value spaces.
*> parm2 has the parameters you want passed to it.
 01  parm3           pic x value low-values.
 01  parm4           pic x(4) value 'open'.
 01  parm5           pic x value '1'.
 01  EVName          pic x(30).
 01  EVValue         pic x(50).
 PROCEDURE       DIVISION.
 Main Section.
 p000.
*
*> Set up the value pair (Name and value of the required EV), as two
*> parameters to the "setEV" Windows Batch file.
*>
*
     move "TestEV" to EVName *> these are an example; make it anything you 
like
     move "Value set from COBOL" to EVValue *> Note that the string may 
contain
                                            *> spaces.
*>
*> Build the command line as the second parameter to the Windows shell
*>
     move spaces to parm2
     string
          "setEV.wsf "          *> this will be the name of your Windows 
script
                                *> batch file
             delimited by size
          EVName
             delimited by space
          space
             delimited by size
          quote
             delimited by size
          EVValue
             delimited by "   " *> used 3 spaces as COBOL won't take ALL 
here
          quote
     delimited by size
          x'00'
             delimited by size  *> terminator for the value string
                into parm2
     end-string
*>
*> Ready to go... get a shell object from Windows...
*>

     invoke WinShell "CREATE-OBJECT" using ShellProgID
            returning ShellObj
     end-invoke
     invoke ShellObj "ShellExecute"
            using Parm1 *> Command to run..
                  Parm2 *> Parms to the Command...
                  Parm3 *> NULL
                  Parm4 *> 'open'
                  Parm5 *> '1'
        returning ShellReturn
     end-invoke
     set ShellObj to NULL
*>
*> Your EV is set into the USER EV area and saved in the Windows registry.
*>  It is there until you delete it and is available to any other processes.
*>
        .
 p999.
     exit program.
---------------------------------------------------------------------------
Clearly, if you change parm1 (The executable name) to be "cmd.exe" you can 
run DOS Batch commands (remember to put "/c" at the start of parm2). 
Similarly, you can put the name of ANY executable there and Windows will run 
it when you invoke the ShellExecute method of the COM component.

If you run an application that returns a completion code, Windows will put 
it into ShellReturn.

I hope this clarifies what can be a bewildering exercise in COBOL sometimes.

Pete.

It works. :-) (I have tested it. Remember to make sure the "setEV.wsf" file 
is on an executable path or in the same directory.)

The main points I'd like to make about this are as follows:

1. Windows has hundreds of useful components in it. You get them 
automatically when you buy the OS, so why not use them? It is really very 
easy to create an instance of the Windows Shell COM component and invoke the 
"ShellExecute" method of it. (See above). It is simple, clean, and elegant. 
Unlike the Sample provided by Fujitsu (I didn't check the MF ones), which 
uses the Win API and requires a dreadful interface. (To be fair, Fujitsu do 
provide sample code for the interface.)

2. It pays to find out more about things you want to do than is immediately 
obvious in COBOL. I never knew there were separate areas for EVs because 
COBOL only lets you use one of them (the local user area which is 
temporary). After using C# and becoming aware that setting and reading 
Registry values is a doddle (3 lines of code, including creating a registry 
object) it seemed reasonable to me that it should be possible to do these 
things with COBOL. It isn't ,because the COBOL runtime is static and uses 
its environment for storing EVs. (They are not pushed to the Registry for 
persistence).

3. Windows Script Hosting (the 21st century Batch file equivalent), gives 
you access to just about anything in Windows, including the Registry and the 
File System. You get an object instance of what you need and invoke whatever 
methods you want to use. The methods are well documented on Google and MSDN. 
You can add, change, delete directories and files before during and after 
running your COBOL, save application settings in the Registry, and much 
more. In fact, most of the stuff you CAN'T do with COBOL you can do FROM 
COBOL using WSH. (Personally, I find it easier to just give up COBOL 
altogether and use a language that lets me do everything, but YMMV...:-))

OK, so now you think you want to have a go at some of this? How can you find 
out what components are available? You need an OBJECT BROWSER. This will 
show you everything registered on your system as a COM (Component Object 
Model) component (including Automation components like Word, Excel, etc...) 
If you have MS Office installed there is an Object Browser that comes with 
Word Basic. Otherwise, there are free downloads. Failing that, you can 
browse the registry directly looking for COM registrations. I use something 
called COMExplorer but it is no longer available. If I run it on Windows XP 
SP3 it reports around 800 components. Some of them are extremely useful and 
can save you hours of work.

If you use either Micro Focus or Fujitsu COBOL to develop Windows 
applications, I would urge you to get into using components; it makes life 
much easier and it's actually fun :-)

Pete.

"WinShell" is a name I gave to the Windows COM component that has a ProgID 
of "Shell.Application". It is a user name for a Class. ALL COM components 
are Classes. In your programs you can give them any name you like,.just as 
you do with a file or a storage area.

"ShellObj" is also a user name as you can see by the fact that it is an 
object reference to an object  instance of Class WinShell. (I could have 
called it "Fred" or anything I liked, but COBOL encourages the use of 
meaningful names :-).)

(All of this is normal OO COBOL programming.)

I think maybe the question you are really asking is: "How did I know about 
this Class, as it isn't documented anywhere in the Fujitsu docs?"

I covered how you can find these things in another post.

There are HUNDREDS of useful COM components provided with Windows and 
THOUSANDS of useful .DLLs. But if people don't bother with OO programming 
they will never find them.

The real trick here is in using WSH to set a permanent EV instead of the 
temporary one set by COBOL or DOS BATCH (CMD.EXE). You CAN'T do it with the 
DOS SET command because that doesn't persist. (You can if you use Lee's 
convoluted generation of a DOS batch file from your app, but that is hardly 
an elegant solution).  When I learned C# and noticed how easy it was to 
access the Registry and set EVs, I suddenly realised that most (if not all) 
of the .Net facilities are available through Windows Script Hosting. All I 
had to do was break away from DOS the same way I broke away from COBOL.

(In 2011 it is high time all Windows Users broke away from DOS... :-)  It is 
really difficult... instead of using CMD you use CSCRIPT to execute. 
Suddenly you have everything you had in DOS, plus a whole lot more besides. 
BUT, you really need to understand Objects and Layers because the OS is 
built that way.) Compare the way the .wsf file creates an object instance of 
the Windows Shell with the way the COBOL program does it. Are they so 
dissimilar? Is there anything mysterious, inscrutable or difficult about it? 
It is tempting when you look at the 2 objects to think: Why not eliminate 
the script altogether and just set the User environment from OO COBOL in the 
COBOL program? You can certainly do that in C#. (You probably could in COBOL 
too, but I haven't investigated the Environment properties of the Shell 
Object created by COBOL. There are some advantages in having a separate 
script file because you can use it in other places, outside COBOL.)

By the way, nobody taught me these things, I just investigated them myself. 
Not something which is beyond most people.

The nice thng is that you don't have to understand the samples posted here 
in order to use them. They are simple and use less code than interfacing via 
the Win API. (I promise you will have less trouble interfacing to a COM 
component than you will in translating VB variables into COBOL. This is 
because the COM definitions are standardised and automatically "translated" 
by the COBOL compiler. For example if you define something as a string (PIC 
X(whatever)...) It gets passed to the COM Dispatch as type BSTRING,  PIC 
S9(5) COMP or BINARY or COMP-4 will be passed as type I2, etc. COBOL 
progammers don't have to know the COM data types because the COMPILER does 
the conversions under the covers. It doesn't do that for the Windows API.

There are around half a dozen lines in the COBOL program which are directly 
OO COBOL. You don't have to be deeply immersed in OO COBOL to be able to 
manipulate COM components with it, and anyone who has used PowerCOBOL to any 
extent has probably done exactly that (whether they realised it or not...)

Hope this helps. If it doesn't, try reading the other posts on it and if it 
is still unclear let me know and I'll do something on the COBOL 21 site by 
way of a step by step explanation.

Pete.
