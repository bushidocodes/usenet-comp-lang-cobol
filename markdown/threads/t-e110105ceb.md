Barticus@att.spam.net (Randall Bart) writes:
> In Medium Systems COBOLV(68) we used = or _
> in phone numbers because - wasn't allowed.  Then the insertion rule was
> expanded to allow - as an insertion character

That goes much farther back on A-Series, probably to the
early 1970s. I can't remember it ever *not* being available
there, though my memory of things 25 years ago is rather
less than perfect.

> I can't submit FTRs anymore, those were standardized out of existance many
> years ago.  I did file a UCF.

Every UCF is subcategorized as either an FTR or an NFS,
though it's true that in common lingo UCF usually means
FTR. I was involved in the CUBE/UNITE NFS handling for
several years, and I think this confusion is one reason the
number of NFSs from the field has declined dramatically. To
submit a New Feature Suggestion you must file a UCF, and
because people think a UCF is an FTR, they don't know where
to file an NFS. Now I'll get my face out fo the alphabet
soup.

> Certainly they don't need to give
> two releases of warning before they fix a bug?

If it's been in existence so long that users are using it,
counting on the existing behavior, then yes they do need to
give two releases of warnings. It doesn't matter that the
documentation says it works a different way. We judge and
debug our program output by the actual behavior of our
programs, not by the documented behavior.

In contrast, they can change the documentation any time
without warning. As long as it's only a content change, no
one will complain. (Note the specification of content
change. When they made documentation unavailable on
character-mode terminals without warning, there were some
very loud complaints.)

> >  New COBOL85 treats the ZZ as insert characters when the
> >  source is non-zero, but inserts ** when the source is zero!
…[4 more quoted lines elided]…
> ANSI behavior.

Thanks for the explanation.

However, my point is that in 44.2 COBOL85, moving 0 to a
PIC ***.ZZ field results in output of ***.**. This is NOT
fixed insertion behavior for the ZZ. Thus in a ***.ZZ
picture, the ZZ is now treated as a fixed insertion
character (as Chuck claims) when the source is non-zero,
but treated as a digit when the source is zero (contrary to
Chuck's claim that it is now a fixed insertion character).

Edward Reid

I don't have a copy of ANSI X3.23-1985, but I do have a copy of the
current ISO draft.  

P 326, 13.17.38.4 (8):  "If the value is zero and the suppression symbol
is '*', the entire data item, including insertion editing symbols except
the actual decimal point, will be '*'.  In this case the actual decimal
point will appear in the item."

That *should* be the same as Cobol-85.  IIRC, the rule has been all stars
but the decimal point since ANSI X3.23-1974.

BTW, the compiler served up a humorus warning today.  I was scanning thru
a program with over 100 truncation warnings, and I found one that I won't
blame on the programmer.  One of a long string of datanames was flagged
for a trunction warning.  I scanned up to the begining of the statement;
it was MOVE ZERO!

It seems the field was PIC V9(4), which isn't enough digits to hold ZERO.
