The final note in this response is certainly NOT true in IBM mainframe
environments - or any other environment that I know of.  Packed-Decimal (not
binary) "digit" nibbles *must* be 0-9 and NOT "A-F".

Also, (again in IBM environments - and in reply to another note), the
sign-nibble *may* be A-F.  It is true that C or D are "preferred" but "F" is
tolerated (even in signed-numbers) in many (not all) COBOL cases - and is
often produced by non-COBOL applications.  A, B, and E are *never* produced
by COBOL, but are valid in IBM S/390 (S/360 or S/370) architecture - and
certain non-Assembler applications do produce them. All IBM mainframe
compilers from OS/VS COBOL on (I don't know about earlier) have options for
handling A, B, and E

This is the hardware implementation on the IBM side. The following is

A.1.2 Decimal Integers

Decimal integers consist of one or more decimal digits and a sign. Each 
digit and the sign are represented by a 4-bit code. The decimal digits 
are in binary-coded decimal (BCD) form, with the values 0-9 encoded as 
0000-1001. The sign is usually represented as 1100 (C hex) for plus and 
1101 (D hex) for minus. These are the preferred sign codes, which are 
generated by the machine for the results of decimal-arithmetic 
operations. There are also several alternate sign codes (1010, 1110, and 
1111 for plus; 1011 for minus). The alternate sign codes are accepted by 
the machine as valid in source operands but are not generated for results.

Decimal integers may have different lengths, from one to 16 bytes. There 
are two decimal formats: packed and zoned. In the packed format, each 
byte contains two decimal digits, except for the rightmost byte, which 
contains the sign code in the right half. For decimal arithmetic, the 
number of decimal digits in the packed format can vary from one to 31. 
Because decimal integers must consist of whole bytes and there must be a 
sign code on the right, the number of decimal digits is always odd. If 
an even number of significant digits is desired, a leading zero must be 
inserted on the left.

Numbers are just symbols that were arbitrarily set up on this planet.

Hexadecimal is base 16: 10 in HEX is really 16 in decimal format
Binary is base 2
Decimal is base 10 (humans have 10 fingers - go figure !)

In order for civilization to survive, there must be order in everything.
That is why humans have set up standards for measuring anything to avoid
any kind of miscommunication between humans.

Even though all data is binary (bits are just 2-state switches: on or
off) as far as the computer is concerned, human beings find it easier to
represent it in HEX or hexadecimal.

A byte consists of 8 bits or 2 HEX digits (each HEX digit has 4 bits).
For example: 123 in decimal format is F1 F2 F3 in HEX format. This is
actually 3 bytes. Remember that each HEX digit is 4 bits, so F1 in
binary is 1111 0001. F2 is 1111 0010. F3 1111 0011. Notice that each bit
is a power of 2 (from 2 to the zero which is 1 in deciaml up to 2 to the
3 which is 8 in decimal). The example I just showed is really a COBOL
PIC 9(3) field or a display numeric field.

On the other hand, a COBOL S9(9)V99 COMP-3 is a packed decimal field and
this is 2 bytes. Let me explain why. A packed decimal field is
represented as HEX digits followed by a C or D as the last HEX digit for
the sign. In this case, S9 means it is a signed numeric field. The
(9)V99 means that this number is composed of 1 integral position
followed by 2 decimal places. The V in the V99 is implies that there are
decimal places to follow. For example: If the number is +3.14, it is
represented as HEX 31 4C which is 2 bytes. If this was -3.14, it is now
HEX 31 4D. The C means positive (it could also be an F) while D is
negative. Keep in mind that the decimal point is implied and does not
take up space. A COMP-3 field is used for computations, so any numeric
field in COBOL always gets converted to a COMP-3 field (if it was a PIC
9(9)V99 display field) no matter what. That is why COBOL programs should
always use a signed COMP-3 field when computing efficiently. The
programs will run faster if you practice this.

In ASSEMBLER code, the instruction to do this is a PACK instruction.
What this does is convert a display numeric field into a PACKED DECIMAL
field or COMP-3 in COBOL before doing any numeric comparisons or
computations.

There is a numeric PIC S9(9) COMP field or binary field, but I will not
go into this. Suffice to say that this is useful for subscripting
purposes (addressability) when dealing with tables or arrays.

1.The number of bytes required for a packed-decimal item containing n
decimal digits is INT((n+2)/2). E.g. n=2-3 needs 2 byte, n=10-11 needs 6
bytes.
(from C. Baroudi "Mastering Cobol")

2. For a binary data item the most efficient code is generated if the item
has a sign and has 8 digits or lower.
When the data item is larger then 8 items or is also used with display, the
PACHED-DECIMAL is a good choice.
The most efficient code for PACKED-DECIMAL is generated if the item has a
sign and an odd number of digits
(http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/library, COBOL for
OS/390 & VM: Programming guide; 4.6.1.2 Data Usage)
