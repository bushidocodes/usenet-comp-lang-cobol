Cobol will be around for a very long time.  One measure of this is that a
lot of businesses are keeping their business logic in Cobol even after
spending a lot of money fixing year 2000 problems.  It would seem to some
that these companies should have made a break for another language since
they were going to spend a lot of money on Y2K anyway.  Several factors play
a role in why this has not been the case.  Because of these factors,
companies will most likely have to take the slow evolutionary approach with
their Cobol.

Synkronix (www.synkronix.com) has a product called PERCobol that started out
as a Cobol-to-Java conversion tool.  It is now a full Cobol compiler that
outputs code that runs in the Java environment.
What is the difference?  Business logic belongs in COBOL.
    Why should businesses convert to Java, then work with unfamiliar,
converted code when their business logic and programmers are COBOL oriented?

You may ask:  "So how does one get the same benefit from PERCobol, without
messing with the Java that is output?"

Answer:  We provide interface mechansims that allow interfacing COBOL with
Java.  This is the same as messing with the output code, but from the COBOL
side rather than the Java side.  The resulting code will all be Java in the
end.

Do you see the potential for this product?

PERCobol (PERC as in coffee, or a good thing.. :-) ) is at the mercy of the
Java Virtual Machine for the particular platform.  We can, however, improve
our overhead.  One issue we face is how to measure the overhead-to-JVM
ratio.  With Symantec's JVM on NT, the sort routine that Leif posted ran in
646,000 milliseconds for a size of 8000 on our Pentium 266.  We have
recently improved that speed to 48,440 milliseconds by optimizing our
runtimes, and we are in the process of improving the time even more.  This
is still considerably slower than natively compiled COBOL code which
executes at 10's to 100's of milliseconds.  However, if you truly want to
measure interpreted code vs. native code, you must natively compile the Java
(try natively compiling Acucobol!).

The question really comes down to whether or not the benefits of using a
tool outweigh the drawbacks for a particular need.

If you can improve the run time by a factor of 13, you can hardly blame the
JVM for the first 87% of the run time.  But, you are on the right track.
The usual overhead by using the JVM is a factor of 25.

There is a similar argument on IBM's AS/400 system. Programs were
compiled to something called MI (for Machine Interface), then the MI
was converted by 'just-in-time' process to native code. This makes the
AS/400 about 10 times slower than a traditional machine with the same
clock rate. But the AS.400 is an extremely useful system because the
MI defines a very large (64-bit) address space as well as a single-level
store where there is no difference between 'RAM' and 'DISK'.

So being a bit slow is OK, as long as you are not excessively slow.
Keep working on the improvements. PERCobol is an intriguing
concept. I'm in the process of testing it for my applications.

Troll or not, homework or not, the question is interesting. COBOL has
been strong because it does certain things very well, and these things
tend to outweigh other posible negative characteristics. Competing

advantages of doing so not at all clear.

royalty licencing is very annoying for small to mid-sized application
developers, and puts them at a very distinct price disadvantage to their
competitors.

realistic chances of expansion in other areas.
