No, they were OO related and more precisely a combination of OO and I-O.  We
developed I-O modules for all of our files and then we found out that the
data in these modules is contained at the class so no matter how many
instances of the object we had, it all pointed back to the same instance of
the data.  The result was we could only open a file once, if we read data
sequentially and then right in the middle of this we did a random read we
lost currency, we still can't inherit the data (we have a file with 96
record types and we have a separate I-O module for each record type.  We put
all of the common I-O routines in one program and had all of the other
programs inherit this one, but MicroFocus doesn't support that at this
time).  The problem I mentioned with the two apps running on separate COBOL
versions is they don't recognize each others record locks so two users could
lock the same record -- not usually a good feature, but we patched the
4.1.10 COBOL and added a runtime tunable and all works fine now.  I guess
our main problem is that we've been using MF COBOL for many years (we still
have machines running 1.3 COBOL and others running 3.2) and that worked
pretty well.  Now we all feel like we are running on the bleeding edge and
when a bug occurs that is difficult to reproduce or happens intermittently
we don't know whether the bug is in our code or in the COBOL.  I'll be the
first to admit that I am no expert in OO COBOL and have often gone down the
wrong path trying to solve a problem many times, but we have encountered our
fair share of problems that weren't our fault.  I think MicroFocus is doing
a good job and they have been extremely helpful (frankly, we've been a royal
pain in the ass at times) and we plan on sticking with them.  In a few years
everything will have settled down and all of this bitching will be "the good
ol' days).

Bill
