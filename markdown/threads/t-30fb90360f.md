```
This question pertains to getting procedure/symbol addresses
(GetProcAddress) for current program enclave--main and subprograms--to
pass onto c++/c# program or to use as "callback" for another WinAPI
function (so far the alternative is unbearable--monitoring message
queues and passing control to paragraph or subprogram).

program gets module handle of current process/thread, but on call to
GetProcAddress, no address is returned.

Using a null module handle GetProcAddress also returns null, unlike
GetModuleFileNameA.

What does work well, so far, is using "GetModuleFileNameA" to return
the name and path of file creating the process.

     call 'GetModuleFileNameA'
          with stdcall linkage
          using by value 0, by reference module-file-name, by value
256
          returning mod-file-len
     display 'mod file len = ' mod-file-len
     display 'mod file name: ' module-file-name (1: mod-file-len)
```

GetModuleHandle () returns a handle to the MODULE, not the process or 
thread. To get handles to the process or thread you would use 
GetCurrentProcess (pseudo-handle valid only within that process) or 
GetCurrentThread (also a pseudohandle, valid only within the context of that 
thread).(But the pseuodhandles may be duplicated for use elsewhere. To do 
that, you use the deviously-named "DuplicateHandle" function).

Or, you can use OpenProcess() or OpenThread() if you have the process or 
thread identifier (and for OpenThread(), Windows/ME or later).

That is, a module is not a process is not a thread.

If no address (null) is returned on a call to GetProcAddress() , 
GetLastError() will tell you why. Except in this case I think the only 
possible returns are "file not found" and "invalid parameter" where the 
invalid parameter is almost certainly the required module handle.

As far as GetModuleFileNameA() returning a name when passed a null module 
handle, but GetProcAddress() not doing so, that is something those of us who 
work extensively with the Windows API cleverly call  "documented behavior."
