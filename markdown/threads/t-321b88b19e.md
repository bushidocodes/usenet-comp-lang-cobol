You seem to have some strange ideas about what happens in a Cobol
run-unit.  While the .dll or .so contains the code and the initial
values for W-S the actual W-S and L-S is created in the run-unit's
address space.  If it abends then the data areas disappear too.

If two separate run units on the same machine both call the same .dll
or .so each run-unit has a new freshly created W-S (and L-S) etc in
its own address space.  If one abends (call it 'A') then the other
won't get A's data, nor will any new run-unit.

Now there is a problem with dlls that when a new version is compiled
or installed the old one may still be in memory and so calls will not
load the new one.  _That_ is why Windows may be rebooted.  It has
nothing to do with Cobol Working-Storage as you claim.

> memory.

It is not particularly relevant what you do.  You made a suggestion
about what should be done by others.  I was pointing out that this was
useful only in circumstances where there were restricted usage types,
as you have in your programs.

Specifically the 'solution' may be just as bad as the 'problem'
depending entirely on what types are used.

> Pic 1 was an extension before 2002, so it means whatever the

You seem to fail to notice a difference between 'instances', which are
_within_ a run-unit, and run-units, which are completely indepenent.

In the case of an 'abend' the whole run-unit disappears and there is
nothing that can 'get your data' as you claimed.
 
> If a new W-S is created for each instance, as you claim, why was it necessary > to invent L-S? 

You are confused.  Within a single run-unit two instances of a program
will share one W-S but will have a separate L-S (ie reentrant(1) or
lack of reentrant(n)).  The new 'reentrant(2)' (as I understand it)
means that two instances within one run-unit will not share W-S but
will each get a new one - in which case L-S may be redundant for those
programs.

Two separate run-units (tasks) will have W-S that is _not_ shared with
any other run-unit regardless of any 'reentrant(n)', nor if one
run-unit abends will the other get access to the data of the one that
abended (abent?).

Ummm ... Really?

ANSI X3.23-1974 explicitly defines "IPC" as an abbreviation for
"Inter-Program Communication" (not "Inter-Process Communication", with or
without the hyphen) on Page I-8 in 1.8, SHORTHAND NOTATION.

ANSI X3.23-1985 does the same on page I-4 in 1.4, HOW TO USE THE STANDARD.

ISO/IEC 1989:2002 uses the expanded term "inter-program communication"
throughout, avoiding the abbreviation; since that standard provides neither
modules nor permissible levels of implementation within those modules, there
is no need for an abbreviation for the module name; the inter-program
communication capabilities are simply part of the overall standard.  I see
neither the term nor the concept "inter-process communication" addressed in
the 2002 COBOL standard, with the possible exception of the mention of its
possibility in "A run unit is an independent entity that may be executed
without communicating with, or being coordinated with, any other run unit
except that it may process files and messages or set and test switches that
were written to or will be read by other run units." on page 385 in 14.5.1,
Run unit organization, first paragraph.
