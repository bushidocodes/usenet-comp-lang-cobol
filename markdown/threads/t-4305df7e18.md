Virtual Memory was invented in the late 1950s in Manchester (England)
under the name One-level store (OLS).  IBM worked on virtual memory and
virtual machine concepts in the lab in 1964 (perhaps even before).  The
first IBM computer using virtual storage outside of an IBM lab was a

1966.  Then the 360/67 and CP/67 came along and the rest is history . . .

the team at the time:

>which was compiled manually,

Not quite.  Two compilers were written at the same time.  The "father of
them all" was written in a language called OSIL which produced B5000 object
code on a B220 (first generation) system.  The process of building the very
first B5000-resident ALGOL compiler was a long and arduous one; after the
first "assembly" of the object code, which required twelve straight hours of
machine time and a whole lot of manual tweaks in the process, the
participants agreed that they wouldn't do *that* again, and from that point
all revisions to the *original* compiler were through patches to the object
code.  There was only one assembly/bind of that original compiler.

>and went thru weeks of debugging and patching before it self-compiled.

Yes, both the original version and its updated-in-parallel ALGOL-language
version were tweaked until the original version could produce an ALGOL
compiler from the ALGOL source that would in turn compile a working ALGOL
compiler from the same source.    That process began on a B5000 simulator
running on a Philco 2000 system, because the hardware was not quite ready
for testing at that point.  But the developers had not actually gotten the
ALGOL-symbol version to compile and run on the simulator by the time the
B5000 hardware was ready for their use, so the original "handcrafted"
compiler did indeed run for a time on the B5000 system itself.

Once the "handcrafted" compiler successfully compiled a new ALGOL compiler
using the ALGOL symbol file, and *that* compiler in turn compiled a *second*
ALGOL compiler that seemed to work, the developers thought they were done,
but unfortunately, that *third* compiler produced from the same symbol file
failed because of an obscure problem in the original handcrafted compiler.

But once the ALGOL-language version of the ALGOL compiler successfully
"triple-compiled",  the original compiler produced on the B220 was rendered
obsolete.   I have been told that, from the very beginning, the code
produced by the ALGOL-language ALGOL compiler was always much more
efficient, both in processor and memory utilization, than its "handcrafted"
predecessor.

From that day to this, updated ALGOL compilers shipped to Unisys customers
are three generations removed from their previous versions; that is, the
patched ALGOL compiler is always compiled with a patched ALGOL compiler that
itself was compiled with a patched ALGOL compiler.   Having a compiler that
will compile itself proved not to be quite enough.

    -Chuck Stevens

I would like to explain at length the reason for a triple compile,
what it demonstrates, and why it doesn't quite demonstrate that.

We start with an old object, and a new source.  We compile the new
source with the old object producing the first new object.  This isn't
the same as the old object (of course, it's from a new source), so the
first (and last) run of the first new object is to compile the new
source producing the second new object.

The first and second new objects should be functionally the same
because they reflect the same source, but since they were produced by
different compilers they may contain different object code to
accomplish the the same function.  So the first (and not necessarily
last, see below) run of the second new object is to compile the new
new source for the third time producing the third new object.  The
second and third new objects were produced from the same source and
functionally identical compilers, so they are compared.  If the second
and third objects are not identical, or if the third object is not
created (due to syntax errors, MCP crashes, etc) the new source has
failed it's triple compile. 

>From that day to this, updated ALGOL compilers shipped to Unisys customers
>are three generations removed from their previous versions;

I was told by someone in Release Control (Pam Whittington's boss, at
the time) that it's a popular misconception that we ship the third
object.  He said we ship the result of the second compile, not that it
makes any diference since the second and third objects are identical.

Triple compile only tells you that a compiler compiles itself.  That
is far from a complete test.  Lex Borger told me we once shipped a
NEWP compiler with a bug in it, which successfully triple-compiled
itself, yet which couldn't triple compile the predeessor source
because that source hit the bug.
