### Loading Cobol .DLL's in and out of memory

Hi there.

We are writing a rules based system for a life insurance company.
We are developing the Product rules system on ADW (Case tool from
Sterling), which
runs under OS/2. A set of templates is produced by our systems
architecture team. The Product rules developers use this templates to
write the programs.
The Case tool generate COBOL code. The COBOL files are compiled for the
mainframe and Windows. On Windows it is compiled into .DLL's. This .DLL's
is single entry point files. One function per .DLL. Each COBOL .DLL is
linked with enough run-time system routines to satisfy its run-time
requirements.

For each table on the database there is one IO function which handles all
the IO actions on that table.
For each rule there is a Process function. This process calls a IO
program to do something on the database.
The complete system will consists of about 200 .DLL's that will be on one
subdirectory.
One example is where we want to determine CLIENT BENEFIT AVAILABILITY.
To be able to get a answer, 26 different functions must be called. Each
function calls one or more .DLL which perform an IO function on a
database.
The visual front-end we are using is KEY:Assemble from Sterling
(OBJECTVIEW).
A memory area is built up in Key:Assemble that consists of all the
datafields and output variables.
That data is sent to a translator .DLL which translate all the C like
datatypes (integer, long integer, floating points to COBOL specific data
types) The output from the translator is a 31 000 bytes (31 K) buffer
which is sent to the main driver (a COBOL .DLL). The translator is also a
COBOL .DLL.
All the following .DLL's are Case generated COBOL files which are
precompiled, compiled and linked to .DLL's.
Key:Assemble calls a control module (MIC020.DLL) with the 31 K buffer.
This control .DLL calls another COBOL .DLL (LNA999.DLL). This driver
calls a IO COBOL .DLL (SPR0350.DLL) which determine which functions to
execute. Each function (a separate COBOL .DLL each time) is then called.
This one Query of the system will call 26 different COBOL .DLL's. Each
one of these .DLL's calls one or more .DLL's to perform actions on the
database.
Each one of the 26 .DLL's will be loaded into memory and released from
memory afterwards. The most COBOL .DLL's that will be in memory at the
same time, will be 5.
They are the translator .DLL (LNA999X.DLL), the control .DLL
(MIC020.DLL), the main driver .DLL (LNA999), a Process .DLL (LNA700) and
a IO .DLL (SPR0270.DLL). All of these is Case generated COBOL .DLL's.
Each .DLL contains data in the working storage of about 100 K. The
average size of the .DLL is 120 000 Bytes on disk.

The idea is that the system must run on Windows 3.11 as well on the
Mainframe.
The source code must be exactly the same on both platforms. That is the
reason for having single entry point .DLL's.

Our client environment will be as follow:
The PC will have the following configuration :
16 MB of RAM
1 GIG
PENTIUM 90
Windows 3.11
The COBOL .DLL's is build with MicroFocus COBOL 3.2.5. We are using the
SQLBase precompiler for COBOL to precompile the SQL statements.
database: SQLBase 5.2 runtime
MQSERIES Client.
KEY:Assemble. The visual front-end.

By calling and cancelling dynamically linked subprograms, it is possible
to have free areas of memory scattered among blocks of memory allocated
to active subprograms. With free memory fragmented into several small
blocks, there might not be enough contiguous free memory to load a
dynamically linked subprogram.

Because our system will continuosly load and unload .DLL's, it will
result in fragmented memory.
I just want confirmation that this can maybe cause problems.

I want to know if the above method is feasible on Windows 3.11.
Can you give us your opinion of this exercise.


I want any comments on this topic please.

Christo Stemmet
sda··.@met··o.za
Cape Town
South Africa
