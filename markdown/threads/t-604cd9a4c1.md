### COBOL vs. Proprietary 4GL

- **From:** "brian..." <ua-author-44027@usenetarchives.gap>
- **Date:** 1996-06-27T20:00:01+00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<4qvlmm$e64@newsbf02.news.aol.com>`

```
Help! I'm drowning in proprietary "4th-generation"
languages! Especially the ones that claim
"no programming required"!

When I started programming at my present employer
10 years ago, we had COBOL on a mainframe. A few
years later, I snuck Turbo C in the back door, so as to
have some access to the data that had begun passing
through PC's.

Now, COBOL is a forbidden language. Our business
logic is officially enshrined in ABAP/4. Of course, since
ABAP/4 lacks the flexibility and universality to do certain
tasks, we now fill in the gaps with C, Visual Basic,
JetForms, Mercator and who knows what they'll buy
next month. But don't even *mention* COBOL, the
Evil Language of the Old Regime!

I've got a report to write that has at the heart of its logic
a three-dimensional array. Now ABAP/4 just doesn't
do 3D arrays, no way. I got around the problem last
time by putting the 3D array logic into a C program,
which then spit out about 10,000 lines of ABAP/4 code
which faked its way around the problem. Amazingly,
it worked pretty well, and I felt way proud of my
ingenuity.

But it was really in the realm of Stupid Computer Tricks.
So this time I was thinking of just dumping raw data out
using the obligatory ABAP/4, then doing the analysis
in C or maybe VB.

I know C. I like C. But for business logic, C just isn't
the right tool for the job. I'm also pretty good at VB,
but I've never been able to get VB to move data between
a raw input line and a data structure -- which I think I'd
rather be able to do before I tackle this program.

So I downloaded my 30-day trial version of MF Visual
Object COBOL today. It compiles my old Sperry 1100
code with only trivial modification, and I'm getting psyched
to try my hand at COBOL again after all these years.

But COBOL is still The Forbidden Language. "Fourth
Generation Languages make programming so much
easier!" they say. "COBOL is a dinosaur, and if you get
identified with it, so will you be!"

I've noticed about our 4GL's that they do simple things
very easily, but get drastically more difficult when the
task is non-trivial. The salespeople demonstrate to the
Executives that they can produce a simple little report
in 5 lines of code or so, where it would take 50 lines of
COBOL. The Execs then get all excited -- an order of
magnitude improvement in productivity! Let's buy this
thing and make them use it for everything!

Problem comes when you get a problem that would take
maybe 2,000 lines of COBOL. The 4GL turns into a
nightmare! Or so it seems to me.

Does anyone have any solid information about the
relative productivity of 4GL's versus COBOL (or other
standard procedural languages) when dealing with
complex tasks? I'd like to have some indication of
whether my gut feeling is right or wrong before I
take up advocacy, even surreptitiously, of a
dangerously heretical language.

Brian McCarthy
```

So much for an officially enshrined (and enlightened) decision. There is
really only one answer for the 4GL question--they are a dime-a-dozen.
There are so many you could never standardize. From what you mention
in your post, your company prefers a briar patch. It sounds like you
have a typical "manage by magazine" management. You'd probably be
**safer** to go with the flow there, or find an employer you would
be happier working for, at least where the job duties are concerned.

Some times total failure is the best solution--at least it forces you
to re-think your position and start over.

I work for a finance company in New Zealand which has invested
in a tailor made system developed in COBOL and they wouldn't dream
about throwing that out for something 4GLish. We are talking about a
million lines of fine tuned COBOL code. This company has wisely chosen
for a smooth conversion from all that code from a mainframe to UNIX
using MF Cobol with maybe later when things are stable GUI interfaces
can be created. This is just one example of a company which has
invested heaps of money in a system and are not going to throw that
away. COBOL does not mean less produtivity, not less performance, not
less anything. COBOL can be a wise choice when you have a established
system and need to convert to a faster hardware platform.

I would however not recommend COBOL as a choice for complete new
development because then 4GL is a far better option in terms of easy
development and fast results with very limited development time.
