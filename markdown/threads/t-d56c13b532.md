I would like to know your basis for saying this, Carlos.

I have used DAO with Version 5, invoking it through *OLE, and I see no
reason why I couldn't do the same with ADO (although I haven't tried it).

Have you tried using ADO with Version 5 or has some salesman told you it can
only be done with V6.1?

In fact, there is very little you can do with version 6.1 that CAN'T be done
with version 5 (the Java interface is the only thing I can think of
immediately, and this is not so great in V6.1. It does not allow Java to
Call COBOL, only COBOL calling Java. As a consequence I am currently writing
my own Java interface for V5. )

Although I have bought and paid for V6.1 and have the CD available, I will
not install it for the following reasons:

1. It provides no major benefit over V5 (Unless you move to the Enterprise
Edition, and I'm not prepared to do that...)

2. The license does not support easy uninstall or re-install and, if you
live outside the United States, it is not viable to simply call Fujitsu when
you need to re-install. (Apart from the cost, there are time differences to
be considered. I have also found in the past, that, while phone support from
Fujitsu is very good, there are often problems in reaching them via their
mail server.)

3. The resources required to support V6.1 are more than I am currently
prepared to devote to it, especially when the "additional features" are nil.

The above reasons mean that I am unlikely to install V6.1 in the near
future. V5 meets my requirements adequately and will very likely be the last
product I install from Fujitsu. (I am reserving judgement on the .NET
compiler to see if the same insane licensing policy applies to it... If it
does, I will either move to another vendor or move away from COBOL
altogether.)

There is a tendency to believe that whatever is "latest" must be best.

In the case of Fujitsu COBOL Professional Edition, this is just NOT true.

Pete.

Sorry, you may know that, but I certainly don't <G>. There is a specific
*OLE EXCEPTION Class which DOES trap errors at the OLE server and returns
full details of them to the invoker.

>May be if we could reference the .tlb files, we could achieve early
> binding and the PowerCobol pre-processor could provide a wrapper class to
> handle the OLE Server events (as it is done in C++).

OO COBOL (and PowerCOBOL) does this better than C++ (in my opinion), because
there is no need to deal with the details of .tlb, C type Header, or any of
the ActiveX wrapping... you just instantiate the Exception class and point
it at the OLE server you want exceptions notified from. (I would stress that
I only used it once to check it out; most of the time if I have a server
failure, I abort the transaction).

Obviously, it is "horses for courses", however I do a lot of tricky direct
RDB
analysis and maintenance using DAO and it works fine (even bypassing the
*OLE class and referencing the Database Objects and Methods directly), hence
I feel ADO should be no problem.

Pete.

I'm not convinced it can't be done, PROVIDED the OLE Client makes its events
visible. ActiveX controls (which are really just OLE servers "wrapped" in a
standard way) can (and in some cases do) reveal their Methods, Properties,
and Events.

Have you checked the Excel documentation to see what Methods, Properties,
and Events are available to OLE? Maybe ADO itself supports certain event
detection?

understand what you're saying...

If you really are so desperate to look for this level of event (and there
may well be cases where you are) then why not invoke the Windows API and
check the message loop? (Talk to Windows directly rather than OLE).
