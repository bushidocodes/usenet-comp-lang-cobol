I started out coding OS/2 GUI apps about 7 or 8 years ago...Started with a
simple windowed application and kept expanding...One of the first apps I
developed under OS/2 was the CNN Headline News SportsTicker (at the bottom
of the TV screen)...All native OS/2 API code in mf cobol...It is still being
used today...CNN Sports Illustrated also uses the same app...

I looked at the dialog system, but it is / was lacking a lot of the required
functionality for a true GUI application...It didn't process a lot of the
standard windows messages that you have access too with native API coding...

I also looked at using panels...Way too slow...And missing a lot of
functionality...

I don't do much OS/2 coding now, but I do a lot of Windows development...All
native APIs, except for 3rd party DLL's like "tab pro" and "list
pro"...Actually using these 3rd party controls is basically the same as
calling a MS API...

One thing I have found over the years, is that an application uses only a
very small portion of the available API calls...You create a window,
enable/disable a window, add an item to a listbox, process push buttons,
etc...Once you learn how to do these things, it's relatively simple to code
a new program...

I use the dialog editor (when it's not screwing up a resource file...:-)) to
create dialogs which become the clientarea of my application (modeless
dialogs)...I usually have a separate .cbl member per dialog and one for the
main window...It is even possible to code up an entire application without
even a messageloop...

Initially, the most confusing part to learning windows development is
getting away from batch oriented thinking to message oriented and reentrant
code thinking...The application program is no longer start here and end
there processing...the user is totally in control...The app must allow the
user to change his mind, etc...

Also, it's kind of difficult getting use to the fact that you can do a
windows function that will result in your code being reentered...The
'SendMessage' API for example...Or an API that results in other windows
messages being generated while you are still processing a message...

I'm not into OO, so I can't really compare native coding to windowing in
OO...I will say though that the OO examples I have seen to date are probably
more complex to me than calling API's...I have used my own developed
vocabulary for windows functions...for example

cmEnableWindow handle is hWindow

This verb issues a "EnableWindow" API call...Probably easier to follow and
code than the API itself...The problem I have with the vocab feature is the
slow compile times...Mostly because of the large windows copybooks
required...For that reason, I no longer use the vocab feature...

James -
After about a zillion years of COBOL batch and batch-oriented screen
work,
we made the switch to Fujitsu. The OO part is mostly invisible to us
as
FJ fences that off quite nicely. To our old, tired way of thinking,
we've had
two areas of interest to force to the front of our brains:

1. Keep as few balls in the air as possible; and
2. Remember the user can make some really crazy choices
under Windows (like clicking the 'Close Program' button in the middle
of entering a name).

The hardest parts - honestly - was learning the eccentricities of a
new
compiler and how to move stuff back and forth between a screen and
a program, at the right time, in the right place, with the right
values.

There are fun parts:
1. Designing a screen with a painter and toolkit is way better than
columns and rows.
2. Neat, ready-made stuff like progress indicators, graphs, text and
list boxes that can hold any amount of data (with the scroll bars
built in), loadable graphic images, complete control over fonts,
their sizes, and renditions, sound effects (I found a really great
bagpipe web site).
3. Thought-free linkage editting and really smart debugging.

So, what we did, was start with a basic form. As the programming
progressed we would reach a point where some feature we'd seen
in a Windows application would be useful and we'd figure out how to
implement that feature in our code. The experience with other
Windows programs told us what was doable - all we needed, then,
was to figure out how we could do it.

Our outstanding mysteries are:
1. User types something in a field and, under certain - but not rare -
conditions we don't get a chance to check it.
2. How to control the application-installation procedure.

Well, you ask a loaded question.

I have a split work personality.  

Where I worked full time the tool of choice was MicroFocus COBOL.  
Where I worked at home, my tool of choice was CA-Realia COBOL.  Where 
I worked the history was - RM/COBOL  and CA-Realia concurrently, them 
move to MF totally.

At any rate, I have experience with 3 of these - API, Dialog System, 
and Sp2.

At Home:

When Realia released their 4.0 product, they also showed examples of 
GUI programs using the Windows API and OS/2 Presentation Manager.  I 
wanted to create GUI programs, but the API method was very very 
tedious.  It took quite some time to understand it.  The whole message
queue thing makes porting a DOS based text mode application a 
nightmare.  I had been developing text based applications using 
SCREENIO (From Realia, actually created by Norcom).  I had my user 
interface largely separated from my code - a big plus in the 
conversion game.  However, my main complaint was that I could easily 
get bogged down in User Interface development and my productivity went
way way down.  No reasonable solution in site.  (1992-1996).

At Work: 

Finally we got the impetus to move to Windows.  I had played with 1.0 
Dialog System (henceforth called DS) - and was encouraged by the fact 
that one could create text or gui applications with the same code - in
fact it would run on OS/2, Windows or Unix.  Pretty nice -- I thought.
 The tool was patently unstable.  It was virtually unusable as far as 
cross platform development.  We initially shelved it.
(1992-1996).  

When MF released 2.0 and rapidly followed with 2.5 DS we decided to 
try it again.  The boss was leaning, reluctantly, toward learning 
Visual C++, but did not like the prospects of a rewrite and struggled 
with OO.  At this same time, our West Coast parent company (from which
I had trasnfered) was developing a "replacement" for the primary 
Mainframe COBOL application.  This was in an attempt to re-license the
system to NT network users instead of Mainframe users (as we had done 
previously), and to eliminate the mainframe.  Initially the project 
was subcontracted to MicroSoft - who's team of consultants (mostly 
H1-B Visa holders) used Visual Basic.  They were fired after a year 
and a new team was hired.  The new team  used OO and MFC.  The staff 
was made up of EXPERTS in the fields.  2 of the main people had PHD's 
in OO design.  The project eventually collapsed under it's own weight 
- no one working on it under stood it.
The mainframe still runs the company.  

With these failures in sight, the decision was made to stick with 
COBOL - but what to use?  I was given the task of converting one of 
our systems from Text mode to Dialog System GUI - then with 2.5 DS.  I
did so and learned several things.

1 - I hate Dialog System.
   a.  Alien Scripting language.
   b.  Restrictive rules about order of processing.
   c.  Still unstable.

2 - Any conversion is a rewrite.

Our code was field by field accepts.  I had taught them to use the 
Screen Section, but it was only used in new systems, and not in 
existing.  It would not have helped with DS anyway.  

I learned to hate event driven programming with DS.  It's a nightmare 
of dependancies.  It's a step up from the API development method, but 
not easy.  Bottom line:  My boss asked me if it was possible to 
convert our systems using this.  My prototype showed it was, but I 
said deployment would be a nightmare, and stability was just not 
there.  (The thing never worked on 2 different machines the same 
way!).

Back at home:  

I had some demand to get my text mode applications to GUI.  I don't 
remember how I heard about Flexus, but I fould them and downloaded the
evaluation version of sp2.  What a revelation.  A GUI solution that 
worked - and worked well.  It was easy to use.  Porting my existing 
DOS mode applications was a breeze.  Having used SCREENIO, all I had 
to do was convert my screenio screens to Sp2 panels, replace the 
calls, and off I went.  SIMPLE.  I was thrilled.  Things that took me 
days to accomplish at work with DS, took minutes at home with sp2. I 
bought it.   I was very excited to tell my boss about sp2 and how well
it worked.

At work, the applications would still need to be rewritten, however.  
Field by field accepts are the problem, no matter what your solution. 
It would STILL be loads faster than DS and more stable.  But they 
turned their back on Sp2 for one simple and terrible reason - it was 
not from MicroFocus.  

I fought for another year with DS before leaving the company for 
another position.  They have yet to have the system fully converted, 
and deployment has been a nightmare, even though they have moved to 
NetExpress and the current release of DS.

Back at home, I have been happily developing Windows applications with
sp2.  I have also been able to easily switch from 16 bit Realia to 32 
bit Fujitsu because of using sp2!  Conversion is easy.

Now some general comments about Windows development.  

It's still slower than my DOS text mode development by about 1/3.  
Printing was a bear until I switched to Flexus Form Print - now it's a
breeze.
The details of the GUI are not daunting, because sp2 handles the 
tedium for me, letting me concentrate on the code.  Sp2 does not limit
my flexibility.  It's a nice balance.  

I'm getting faster developing Windows applications now, but often 
Windows gets in the way.  I remember things happening like: Windows 
automatically searches the path for data files when the file is not in
the current directory.   Drove me crazy for a while. 

One final thought.  There is one more thing you get from using sp2 
that you do not get from the API or DS approach.  Transparant 
conversion to a Thin Client application.  This is where the program 
runs on a server, and the UI exists on the clients desktop.  This is 
very cool.

Our first step was to move to the graphical environment, in that sense the
application behaved like a native windows application and not being run in a

This phase included core functionality as printer support and keyboard

The further road has pretty much been built as we were walking. graphical
elements been taken into the application as it felt natural more than
anything else. The message box for instance, had been there for years,
although character based, because it was one common source, it was easily
replaced with a gui one.

the applications.

But your question isn't really about how we did it, but with what we did it.
I don't know MF or Fujitsu or any of the others too good. They may support
it the same way that Acucobol does, I get the impression however, that both
Fujitsu and MF has moved away from Cobol, toward the structure of C++ and
pascal. And i ask myself, for what good?

You saw my earlier example code of how to do a message box in Acucobol,
DISPLAY MESSAGE BOX
compare this to the examples that there has been of how to do a message box
in MF. What is most Cobol alike?

Whean dealing with the API on the level of which is provided in the sample I
have seen of MF here in this NG, obviously, you have to do a lot of
housekeeping yourself. Freedom of course, anyone would claim freedom, as
would I, to choose to do whatever I want. Well. The price of this, is that
you yourself have to deal with the baby, when it is born, raised and at its
death.

You would probably try to avoid implementing too much new stuff at one time.
That is one of the benefits of the Acucobol approach, and which we have
used. Because the code is portable, we can move it around and apply changes
as we go. We may, but we don't have to; have concerns for messages, handling
callbacks and whatever.

I enjoy using Acucobol because I still have the feeling that I am
programming in Cobol, although I show fancy listboxes and grids and you name
it. DISPLAY and ACCEPT is what I used yesterday, and I still use it. The
most serious impact the gui stuff has had to applications that I have done
is that the exception procedures has been growing, to deal with the events,
which I of own choice has wanted to deal with.

At the same time, if I want access to the API, I can do it, like the file
functions I sent in as a demo app to the NG a while ago. From my humble
point of view, that is what Cobol should do, it should protect me from the
low end API and provide me with the tools to accomplish my everyday tasks
without forcing me into complicated construction work for even the simplest
task.

It should be no secret that I consider OO Cobol to be somewhat of a paradox,
I must admit though, that I have not been playing around with OO Cobol so

of view. For myself it was more important not to be bugged with the lowlevel

from C++, Delphi and Acucobol. The choice is Acucobol because of the
simplicity. When I need to do advanced stuff, I got the tool for that as
well. Now, someone will point out the benefit of being able to do it all in
one language, and they definitely got a point. That is why I am happy to say
that there has been two things I have had to do externally, that is serial
communication and of all things, clipboard communication. Considering the
simplicity of the tasks that is the real reason we chose Cobol in the first
place, I feel that it is to live with.

Oh, I am talking way too much.

Listen, James, I know these things are difficult to express writing,
however, why don't you give it a try, and I will see if there is anything I
can explain. Although I don't know MF or Fujitsu too good, I feel that I
have a good idea of what Windows is all about.

I was involved in the development of some COBOL GUI development tools so I
had to learn the native APIs. The trickiest part was learning enough C to be
able to translate a call. Once I had that skill, the rest is an exercise of
struggling through the various references for samples that come close to
what you're trying to do, until you build up a repertoire.

A book that was a great help, and dates me, is 'The COBOL Presentation
Manager Programming Guide' by David M. Dill published by Van Nostrand
Reinhold. I suspect there's a Windows 3.x version but I doubt if there's
many other references that cover the Win32 API for COBOL programmers.

The biggest weakness of COBOL GUI programming is the lack of available
reference material. You can choose to use a third party development tool
like NetExpress, Power COBOL, Flexus, et al but they require you to learn a
particular vendor's solution to the problem, not the native API. You're also
limited to whatever they produce in the way of documentation. More
sophisticated interfaces, fancy controls like calendar functions, and
state-of-the-art stuff is usually written and documented for C++ or Visual
Basic programmers.

While I like COBOL for a lot of things and can make it do just about
anything with the Win32 API, it would not be my first choice for developing
a Win32 application. The reasons are many, but the most critical reason is
the pool of people that could maintain it in the future is quite small.
Consider a multi-tier architecture using COBOL for the business and I/O and
develop the UI using whatever tools get the job done. It's easy to call
COBOL from C++, VB or Java. A COBOL CGI front-end is also pretty
straightforward.

Jimmy,

Since you asked...

GUI ScreenIO is basically a screen-at-a-time interface, with options
that allow you to get control at the field or character level if you
really want to.  You call our routine to display and manage the
window(s) and it returns when an event of interest to you takes place;
a button press, tabbing to a different field, menu selection, and do
on.  It is a true Windows product and supports many of the popular
Windows features; you can't tell by looking at it that it's a COBOL
application.

SP2 is, of course, our primary competition (look out, Bob) and I would
be inclined to say that our product is both more powerful and far, far
easier to use.  Unlike SP2, however, GUI ScreenIO is strictly a
Windows product - but shucks, that's a pretty big market.

In general terms, you design your windows in our WYSIWYG editor which
produces a panel copybook that contains all of the runtime info needed
to operate the panel.  They are quite compact.  

One of the more interesting things about our GUI ScreenIO, is that all
types of windows are treated (from the application programmer's point
of view) identically. 

Property sheets, popup dialogs, wizards, and regular dialog boxes are
just another panel to your program.  If you find that a dialog becomes
too complicated, you can convert it to a property sheet and use the
property sheet with perhaps one line of changed code in your program.

If you require multiple language support, you can translate all of
your strings using the panel editor, and the runtime will (soon) allow
an end user to switch languages on-the-fly.  Again, all of this is
stored in the panel copybook so you don't need any external files.

We presently support a good handfull of controls, and have tons of
features in the design.  You can use edit controls, listboxes,
comboboxes, treeviews, pushbuttons, and - soon - listview controls.
We provide tooltip support.   We're also providing a simple interface
to the Windows Common Dialogs so you can use stuff like the standard
File/Open dialog.  The list of features is quite extensive.

We expect to release a stable and very functional (though not
complete) product later this month.   

I'll email you the beta information.  We would be most interested in
your impression of the product.

As for OO, I'm not terribly impressed with "strict"  OO re
inheritance, polymorphism, etc. but if you mean you just want to call
other languages from your program, that's something else entirely.
GUI ScreenIO's runtime dll is CA-Realia COBOL, which is callable from
any 32-bit language.  Older versions of Micro Focus runtimes were
somewhat problematic - they did strange things - but I believe that
NetExpress works fine.  We have called Micro Focus dlls from
CA-Realia, and I suppose that it would be possible to do the same with
Fujitsu, but our expertise lies with CA-Realia so I can't give any
substantive advice re the other products.
