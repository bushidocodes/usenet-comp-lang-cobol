If they supported OO it wouldn't be TOO hard to write some COM support
classes (Fujitsu do this and their COM support is outstanding; far better
than Micro Focus, although Micro Focus also supports the model.)

Although it is a Microsoft model, COM components are brokered by CORBA to
run in the LAMP world as well as the MS one.

It is a standard, consistent, interface for components to interact with each
other, irrespective of platform. Despite a number of attempts to phase it
out or replace it, it is just too useful for people to let it go (a bit like
SOAP in that regard). Ever since I opted for component based design (around
20 years now) all of my own software has been built around COM. It really is
seamless and amazing in the way it allows components written in different
languages (and even from different platforms) to work together. BUT, it is
based on Object Orientation... (COM objects have properties, accessible by
GET and SET methods, and methods (encapsulated behaviours). COBOL COM
components call (and are called) by other COM components written in other
languages which support the Component Object Model, and all of it is
seamless and painless. The entire codebase for PRIMA Computing is written
using this model and many of these components are reused in different
software packages and solutions. COM components that were once written in OO
COBOL are gradually being replaced by C# and I can see a time when there
will be no more COBOL in our inventory. This is not because of a failing in
Fujitsu COBOL, rather it is because it is just easier and quicker to write
object classes in a language designed to support them.

To work on the "Open COBOL" project would require facility with C and C++
and , for me at least, that is just hard work. Even if I had the time (which
I don't) I'm not facile enough with these languages to be able to contribute
easily. I made a foray into C some years ago (around the time I learned
Java) and decided it was just too hard (for both development and
maintenance) for the benefits. Then C# came along and that gives me
everything and is a delight to write. Over the last few months I have been
coding intensively in it every day, my knowledge has expanded, and I enjoy
writing it every bit as much as I used to enjoy COBOL. (It is less work than
COBOL because it is more powerful (so, less writing) and the tools are
generally better...)

If "Open COBOL" is to achieve commercial credibility it will need OO
(definitely) and COM support (ideally).

That's just silly. The implementation language of the compiler has
darn little influence on what the compiler supports. Consider GCC,
for example, which has until recently been pure C (or at least the GCC
dialect of C).

Of course OO has a number of features that make writing a compiler
easier, and I would certainly pick C++ over C as a development
language for a new compiler*, but that applies whether or not the
target language is OO or not.

There may be some confusion here about the OpenCobol implementation
though. That compiler *produces* C (IIRC) code as output, which is
then compiled by a C compiler. I agree that it would probably be a
good idea to change the *output* of the compiler to produce C++ code
when supporting Cobol OO, since many of the Cobol OO constructs will
map neatly onto C++ OO constructs, which would otherwise have to my
simulated on top of C constructs. But that has basically nothing to
do with the language in which the compiler is written.


*Assuming other consideration did not prevent that. And actually C++
wouldn't be my first choice, except that my first choice (Haskell) is
not likely to meet some of the more mundane tool-chain support
requirements. But the implementation language for the compiler should
be a language that's convenient for implementing the compiler (where
"convenience" includes the mundane practical requirements).
