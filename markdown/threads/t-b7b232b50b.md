Kevin is absolutely correct when he says that just because you can
incorporate your own personal design preferences into your GUI screens
doesn't mean that you SHOULD do it.

Although many of the "Windows default settings" don't necessarily
enhance data entry productivity, your design consideration should be:

1.  Who is using the system? and
2.  What is the purpose of the system?

General rule of thumb:

If the system is designed for a group of employees in which there is a
high degree of turnover, stick closely to the Windows defaults to help
minimize the learning curve.

If the system is designed for high volume data entry, turn the mouse
off and force keyboard navigation or field autotabbing, if possible.
Nothing kills productivity more than the end user typing, then moving
their hand to the mouse to move to the next field.

If the system is designed for minimal data entry and heavy browsing,
the mouse can be very useful, for example, an executive information
system.

As if things weren't complicated enough, now we have user interface
design considerations for web browser based user interface systems.
This really throws a monkey wrench into the works, because web browser
interfaces are being designed in every manner imaginable.

In addition, unless you use Javascript or a Java applet, forget about
the menu bar in your web browser interface.  Windows doesn't allow
child windows to support menu bars and pull down menus.

Sometimes I wonder whether we change the way we do things because it
is better or if we change the way we do things just because it is
different.

The API calls can be encapsulated into OO objects...OR...They can be
contained within subroutines or DLLs...Personally I use a "Common Routines"
module for most API functions...For example to enable a window I use

Call WinAPI crEnableWindow
   using by value hWndWindow.

or

Invoke crEnableWindow
   WindowHandle is hWndWindow.

The code for entry point 'crEnableWindow' is a call or perhaps many calls to
the operating system API(s). I have a common routines module for Windows and
one for OS/2...The actual user app code does not change...I also reduce the
complication of dealing with the API(s) by having routines in the common
routines module that performs one, several or even many actual API
calls...For example opening and initializing a COM port...This might require
5 different API calls...But from my app, I simplify that to a single call :

Call WinAPI crComOpen
   using by reference z'COM1'
         by reference z'19200,N,8,1'
returning wsPortHandle.

Many call(s) get made to the actual operating system APIs, but my app is
coded using a very simple call to the crComOpen entry point. Once these
common routines are developed (and it will take some time to develop) the UI
or interface to OS functions like reading and writing to com ports, etc is
greatly simplified.

Window procedures and callbacks again are nothing more than entry points
that Windows calls with associated parameters passed...These can be
encapsulated into separate modules as well...Recursion and messaging is a
fact of life in a windows environment...An understanding of how these things
work is a requirement that cannot be avoided no matter how you handle the
user interface...

I suppose in summary, someone who is inquiring about what direction to take
in their future development should explore ALL the various options that are
available to them...API programming is one of those options available and
works well...They should not be intimidated by vendors of UI products
telling them how complicated API programming is, when in reality, it is not
really that complicated, ONCE the learning curve is completed and messaging,
recursion and multiple entry points is understood...Programming in windows
