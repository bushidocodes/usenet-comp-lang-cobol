> Does it make sense to develop COBOL applications

You've had some very sensible answers, from experienced people who know what 
they're talking about.

The question indicates that you don't really understand the PC environment.

Unlike a mainframe, a PC is almost pointless, unless it is connected to 
other PCs.

So applications are developed for the Network, not for a a single 
workstation or desktop. There WAS a time (a few years back), when it made 
sense to develop Windows applications for the desktop, (even for a single 
desktop) and do so in COBOL. Today that is more the realm of hobby 
programming than for serious commercial enterprise.

(Despite brave attempts by certain Companies with a vested interest to talk 
it up, the world isn't buying it and has voted with its feet. COBOL has less 
than 1%  (actually .4% 
http://www.tiobe.com/index.php/content/paperinfo/tpci/index.html  ) of the 
commercial development going on in the world today...it is now the 30th 
language for commercial development and has been dropped from reporting by a 
number of web sites which analyse language use and trends. Java has a 
seemingly unassailable lead as #1 with around 18% of the world's code 
currently being written in it. (C# around 6%))

COBOL does not really lend itself to Network programming (at least, not as 
well as some other languages), and this, combined with the  higher initial 
compiler costs, and run time fees if you distribute it, along with the 
higher cost of more labour intensive maintenance, has led to its decline as 
the Network programming language of choice.

Network programming is about objects and layers; programs are interrupt 
driven by events which occur and must be immediately responded to. This is a 
different paradigm from the Procedural COBOL language where the program 
decides when it will do a read or a write etc.

You CAN write COBOL programs for a PC and they will run just as if it is a 
"mini mainframe", but the question is increasingly becoming one of whether 
you would want to or not.

>
> Do enterprise companies use/prefer COBOL apps that run on a PC
> or a PC server, or the mainframes are here to dominate
> the enterprise market ?

Mainframes will increasingly become just another Network server (maybe with 
a load levelling priority to handle batch and Data Warehousing, UNLESS the 
TCO on them comes down. (It has considerably in the last 10 years, but there 
is still a gap between the most powerful Network servers and a top end 
commercial mainframe, in price if not in performance.) Of course, if the TCO 
achieves parity then there will be no real difference between a mainframe 
hooked into the Network and a Network server hooked into the Network. The 
fact that the architectures may be completely different is not a problem as 
far as the Network is concerned.
>
> I ask it because I do not own a mainframe I have to develop COBOL apps
> on a PC .

Hmmm... slavery has been abolished in most of the world for at least a 
century now.  WHAT or WHO is compelling you to write COBOL for a PC?

Could it be that you perceive computer programming as only being about COBOL 
so that MUST be the way to do it?

If that is the case and you want to do some "home programming" you can 
certainly do it in COBOL. But don't kid yourself it will be commercially 
viable unless you have a specific niche market and they really need your 
specialised  industry knowledge and skill. (As opposed ot programming 
knowledge and skill...)

I make a living at the moment largely based around helping people move OFF 
COBOL for their Network solutions. They don't make this brave Migration, 
with the consequent significant upheaval, so I can make a living. They do it 
because they are increasingly finding that maintaining COBOL for a Network 
environment is non-viable. Unanimously (in our exprience), people have found 
that their costs go down and their productivity goes up when they get off 
COBOL. (Once they have navigated the learning curve into a newer 
language/toolset.) My Company helps them do this without having to throw 
everything away and start over, and the tools we provide automatically 
generate a layered, object oriented solution from their existing  (Standard 
COBOL) applications.

If COBOL is what you know and love and you want to do some programming in 
it, and all you have available is a PC, you would do well to look at at 
products like Open COBOL and TinyCOBOL (as Alistair already advised you). 
They are free, but they have limitations.

For a Networked solution you will need an Object Oriented compiler and these 
are NOT free. (There are student versions available but again, these are 
limited.)

If you want to start a Home business writing commecrial software for PCs (or 
mobile devices like phones and PDAs - a vast emerging market...) look at the 
free downloads of C# and Visual Studio Express. There are free instructional 
videos available that will teach you the basics of both these products and 
the rest is up to you...

Errr...  Using C as an intermediate target is *hugely* popular, and
for very good reasons.  Many high level structures can be mapped well
to C, C compilers are ubiquitous, and with a bit of effort the
generated C code can be highly portable, eliminating the need to write
complex back ends for every target you'd like to compile code for.
All things which do not happen if you generate assembler or machine
code directly.

Writing back ends is *hard*.  Writing *good* back ends (IOW, ones that
generate good code) is *very* hard.  Even reusing preexisting back
ends in a (reasonably) well defined structure (like in GCC) is pretty
hard (and if your language requires things that the existing back ends
don’t support well, you stuck either patching around it, or trying to
convince a bunch of people to add support for something in a multitude
of back ends) – which is not to say it doesn’t happen, look at all the
front ends in GCC (although the OpenCOBOL folks have clearly decided
not to tie themselves that closely to GCC)..

In the distant past Fortran was a popular target, although mainly
because of its (then) ubiquity (C is a better target in almost all
respects, much of what is valuable about Fortran has to be worked
around in that role).

These days you also see a bit of C++, too, and a smattering of other
languages, but C is by far the most common.

And *nobody* intends for that intermediate C code to ever really be
visible outside the innards of the compiler (although you often see
remnants in the debugger).  The intermediate C code tends to be
really, really ugly, and almost unintelligible.  In fact it often
stresses C compilers in severe and unusual ways (is your C compiler
choking on  a switch statement with 5000 cases, or a function with
30,000 lines?  You can bet that code came from a code generator).  But
this intermediate C code is not like the code generated by
translators, which are intended to allow the programmers to switch to
the new language (although most of those don't generate particularly
good or readable code either).

FWIW, the first C++ was a front end translator to C.

The clear purpose of OpenCOBOL is to compile Cobol programs.  That a C
compiler is needed as part of the infrastructure is immaterial.
