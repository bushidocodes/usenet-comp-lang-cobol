I think it was Cheesle who suggested 'formalizing' the rules, so here
they are.


The Great Programming Contest.

Initially, the idea was to develop an OO program and an old-fashioned
program at the same time so as the throng could learn what the heck
OO is all about and to compare different methods.

It grew from that to become a Free-For-All.

You are invited to enter the competition. You may use any language
or method you want - anything at all but it must run on the PC under
Windows because everyone has access to Windows.

I think everyone would like to see a structured entrant and
at least one OO entrant - after all it was OO that started all
this.

The contest is to design, program and distribute a simple Time-Clock
application. The application must do the following:

a. Allow the creation and subsequent maintenance (change, delete)
   of employee records in a file.

b. For each employee, the initials (prime key), name, address, social-
   security number and telephone must be provided for. All fields are
   required except the second line of address. State must be validated.

c. Employees must be able to punch-in and punch-out.

d. The application must provide the means for the user to make a
   backup of the employee-file.

e. You must design the screen(s), files, etc. You are developing
   a complete, though tiny, application.


Of course, a real-world program would do more. Like print an
hours-worked report. The objective of the contest is to compare
programming methods so we need to keep it simple so additional
things like that should be left out. Points are not awarded for
additional features, in fact, there are no points to be won.

Here are the rules:

You can use anything you want and do it any way you want.
Since this is intended to be educational and to allow people
with different opinions to evaluate different methods, please
document all but the most blatantly obvious things.

People got vociferous when I suggested that the contest be
conducted in three parts which reflected the approach I would
take. Some people do things backwards, it seems, perhaps me?

So, all you have to do is email your executable and all source
code in ZIP format to Foodman123@aol.com. I will make entries
available via the Internet. Remember this is not a race.

Would someone like to come up with some criteria for
evaluating the entrants?

Thanks

Tony Dilworth

It is a good rule of thumb.  The hard part for most programmers is to
stop wasting time with something that is unimportant.

For example you may benchmark you suite and find that the people can
leave an hour earlier if you simply move reports they do not need
until tomorrow after the interactive stuff.

Example 2:  You may find the reaction time on transaction X is not
acceptable to people.  You look at the process and find that it does
some updating that can be pulled out of the response cycle and run as
a background task.  Assuming you cannot do this, you are only looking
at a single transaction, not the whole system.

The issue is that performance is typically not an issue.  First get
the system going and then find out where your problems are, then fix
them.
