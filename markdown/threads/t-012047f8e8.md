```
Resolve the above post, using the technique below;

1. type the REGEDIT from run command prompt
2. find the strings under
   HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\Current Version
\Policies\Explorer
3. create new DWORD key, named it as "NoDrives"
4. modify the value data to 128 (Decimal) for drive H
5. reboot the client PC (Windows XP)

So initially when boot-up, the drive H is now hidden from the My
Computer.... though it is still accessible using the Cobol application
(it can still map it). However, using the "My Network Places" the \
\server\sharefolder can still be seen on the client.
```

```
I'd like to add my endorsement to your statements on this Robert.

There are far too many bad systems that assume users are idiots who should 
not be allowed to see stuff that they might be tempted to meddle with. The 
correct approach (as you have suggested here) is to make it visible, just 
like everything else, but protected from unauthorised access. This can save 
hours of sweat when it actually does need to be repaired by people who know 
what they are doing.

C# applications often use an "app.config" file to store sensitive settings 
and such. I expected that such a file would be editable by people wishing to 
change the configuration (it defaults to XML format), and therefore took 
steps to protect it with permissions. I needn't have worried... when the 
application is deployed it is placed into an obscure directory that is very 
difficult to find. I was a bit dismayed by this as I felt it SHOULD be 
visible and editable by appropriate people. Instead, I provided an interface 
via a GUI to it, so that the "right" people can change the entries that I 
would want to allow them to change, and then put appropriate permissions on 
that form. Obviously, MS have realised that most users run their Windows 
applications as Admins and so they have tucked this stuff away, but I still 
think the right approach is the one you have described.
```

```
There are a wide variety of permissions that can be granted by user and role, on
directories and on individual files. Using Explorer, right click on a directory,
Properties, Security, Advanced, (select a user), View/Edit.  See all the permissions that
can be granted and denied?

Security is achieved by granting PERMISSIONS to users, not by denying them access to the
disk. 

>The network platform is simple and I do not use DFS (distributed file
>system) in windows server. Did not use it because I only have one PC
>server.

You are using SMB and Domain authentication.

>How can I connect to the \\server drive "without" exposing the mapped
>network drive? Can I probably use autoexec.bat or win.ini file to
>connect to \\server without exposing (naming it drive H) it as mapped
>network drive? The client PC uses Windows OS.

Drive mappings can be set to automatically reconnect. You shouldn't put them in
autoexec.bat or win.ini. You're not running MS-DOS. You don't need mappings unless you
want the shared directories to appear in Explorer. 

If you want the application to have different permissions from users (why?) start them
using runas:
runas /profile /user:(server)\(user) "(yourprogram)"

Your users will need to know the password for (user), so they could run the above with
(yourprogram) = cmd, or do their own drive mapping.
```
