The G01, G02 etc are overlays caused by the SEG command, that is
the memory will hold ProgramB.GNT sections plus _either_
the .G01 sections or the .G02 sections, they will overwrite
each other.  If they are completely unrelated (such as G01
being initialisation and G02 being processing) then this is OK
otherwise, if say it alternates between a G01 routine and a
G02 routine, it may become somewaht slow.

Are these really large programs that require to be overlayed to
fit ?

: I package the files together in a LBR call PROGRAMA.LBR

: The system will then "Chain" from back and forth between the programs depending
: on the function the user is doing. 

I dislike the Chaining of programs.  Though I have not had problems
with it in the past I prefer having a small 'control' program
that CALL/CANCELs as required.  This usually holds an Interface
block in working storage that is passed to each called program.
There is a menu program that is the default 'next program' so
if a program exits without having set the next program to go to
then the menu is displayed.

Something like:

            WORKING-STORAGE SECTION.
            01  Interface.
                03  Program-Name             PIC X(8).
                03  Next-Program             PIC X(8).
                03  Menu-Program             PIC X(8)).
                03  Company-Parameters
                    ......
                    common data as required

             PROCEDURE DIVISION.
                ...

                MOVE "SYSLOGON"       TO Program-Name
                MOVE "SYSMENUA"       TO Next-Program
                MOVE "SYSMENUA"       TO Menu-Program

                PERFORM Call-Cancel
                    UNTIL Next-Program = "FINISH"

                STOP RUN.

             Call-Cancel.

                 CALL Program-Name USING Interface
                 CANCEL Program-Name

                 MOVE Next-Program   TO Program-Name
                 MOVE Menu-Program   TO Next-Program
                 .

Each program can then just EXIT PROGRAM and the menu will
display or can set Next-Program if required.

Under DOS you also have to worry about memory fragmentation
when programs are called and cancelled or file opens cause
buffer blocks to become lodged.  The RTS tries to resolve
this but it is best to avoid the problem by loading all
common routines first.

My start-ups (as above) also open an indexed file to load
EXTFH and do some displays to load ADIS and also do dummy
CALLs on system routines (such as your SCREEN.GNT) to make
them load into lower memory.  This then allows maximum
contiguous memory to CALL/CANCEL the changing programs.

If you call programa which then calls screen, screen will
be loaded above programa.  f screen is not cancelled then
it will break memory into two parts (though the RTS may be
able to resolve this - I don't use the RTS I use dynamic
run-time COBLIB).
