upload orders etc, now using EMail.  Because of the volumes I mark
records with a block of flags when they are updated, one for each

branch.  But then a product may be updated 20 or 100 times in one day,
but extracted once.

        Central server
        Client resort
        Owners/Members

Are you keeping all data on the central server and making payment

were you wanting to have the records maintained on the resort and the
EMail is to the Owner/Membrers machine to keep a set of records there
?  Or were you EMailing from the resort to keep the central server
updated ?

selectsoftware@knology.net (Keith) wrote

Sounds reasonable.
 
> We will have a web server, initially with an ISP 

Ah, now there's the rub.  Many ISPs will only allow certain types of
CGI processing.  Typically they will allow .ASP, or Perl/Python/PHP,
perhaps SQL Server or MySQL.

The chances of you doing it in Cobol on their server may be quite
small.

Some ISPs seem to allow hosting of user servers.  ie you can put your
machine will your software in their room for a fee.  This allows you
to run anything you like but management becomes an issue.

> and then there will be the
> property computer (client). Both computers will have a member data file,
> payment data file and reservations data file.

This is where the _real_ problems arise.  Due to latency between ISP
based server and client machine there _will_ be cases where one
customer reserves via Web at the same time as another has phoned the
resort directly for the same reservation.
 
> When a member logs into the server they will be able to check their payment
> history and/or check the availability of units/sites for rent. If they
> reserve a site/unit we want an e-mail sent to the client computer. 

Easy to do (or should be).

> When the
> client computer receives the e-mail, a program will read it and update the
> reservations file on the client. 

Should be quite easy to do.  A program needs to run continuously in
background looking for incoming EMails and sleeping if none found.  I
do this with a batch file, but there needs to be an EMail server that
will put new EMails out as separate files and distributed by user.  I
have used Mercury for this (see www.PMail.com).  The batch file uses
MIME64.EXE (http://www.ibiblio.org/pub/micro/pc-stuff/freedos/files/net/
) to extract the attachment to a file - in this case a .ZIP which is
unzipped.  Then the processing Cobol program runs on the unzipped
text.  You won't need the zipping.

> If the property (client) makes or changes a
> reservation the client will create an e-mail to the server and when received
> a program will read it and update the server reservations file.

If you can get the ISP to do this it may be OK, but I would probably
do a html connect from the client system and send data in through the
CGI program.  ie send it a CGI Query with the data items attached as
if it was coming from an HTML <form>.  This way there is only one
program to worry about and and it can be 'instant'.  The CGI would
'know' that it was the resort client computer sending and could reply
with a much simplified page that is easy to parse.  Or even use SOAP
or similar.

Because it would quick this could be part of the confirmation process.
 basically the ISP's database becomes the reservation 'master' and the
resort's
machine must bounce reservations off that before confirming.

> The same thing will happen, mainly from the client, when new
> charges/payments are added to a member's file. The client will create an
> e-mail and send it to the server and the server will update the member data
> file.

Or send a CGI query to update.

> The e-mails would have to include the property number, member number, type
> of transaction (reservation addition/change or payment history update), the
…[4 more quoted lines elided]…
> Thanks again. Keith

Once the EMail attachment has been received by Mercury (or some other)
and broken out by MIME64 (and PKUNZIP) then it is just a text that can
easily be read as a LINE SEQUENTIAL.

To send EMails from Windows I use BLAT (see
http://www.interlog.com/~tcharron/blat.html ) it will add a file as an
attachment and bang it out.

If you can do a CALL "system" or similar to execute a batch file you
can do this directly, otherwise just write out a file to a specific
place and have a batch file cycling look for files there to be BLATted
out. This too needs to have a real sleep for a minute or two betwen
activations, or could be another part of the incoming process.

Of course if the Web Servers were in-house at the resort the CGI
program could directly interract with the live data files.  For that I
use Xitami server (Xitami Pro does https secure serving) (see
http://www.imatix.com ) on Windows or Apache on Linux and just did my
own CGI code - it isn't particularly hard if you can get the program
to use stdin/stdout and read environment variables.  The code is 99%
the same for indows+Xitami /OS/2+IBMServer /Linux+Apache
/UnixWare+NetScape.
