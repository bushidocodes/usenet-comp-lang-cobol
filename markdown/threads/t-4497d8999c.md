There are a number of ways you can choose to process.

The 2 main ones are using either a Command object or the Resultset object.

(As this is quick free help, I won't cover the command object; 
recordset/resultset processing is the most popular anyway...)

If you intend to use the recordset/resultset processing you will need to 
invoke methods on the  RS object.

This can be a bit tricky as there are quite a few possibilities. To get you 
started, here's an example...


       *> set up your query....
                      move "SELECT model, colour FROM showroom WHERE marque 
= 'Yugo'"
                            to SQL-string
                      set adOpenDynamic to TRUE
                      set adLockOptimistic to TRUE

       *>  get the result set...
                      invoke OBJ-RS "Open"
                            using  SQL-string,
                                      ADO-OBJ,
                                      ADO-RS-cursor-type,
                                      ADO-RS-lock-type
                      end-invoke
 Process it using the various commands available until it returns EOF...

I strongly recommend you take a look at the ADO command set and do a quick 
tutorial on ADO processing.

W3C schools are excellent.

http://www.w3schools.com/

SOME programs DO have the cnnection string(s) hard coded in them.(These tend 
to be amateurish "first attempts"and as experience is gained, coding 
techniques improve...)

For Fujitsu (using ODBC) They are usually stored in an external file. 
Richard posted a sample here a little while back. As you can see, the 
pasword is encrypted.

In ASP.NET programs the connection strings are stored in a Web Config file. 
(They don't HAVE to be, but there are obvious advantages in doing so, and it 
is accepted practice for C#). Again you can encrypt them if you want.

Finally, connection strings CAN be embedded in code with user and pasword 
passed to them as runtime parameters (or not...) depending on programmer 
preference.

For dynamic connections with varying users and passwords, connection strings 
are built at runtime (usually by simple concatenation), although a 
"template" of the connection string may be obtained from an external source 
first, and then the variable parts slotted into it by a Regular Expression 
or static concatenation. For different access methods and RDBMS, the strings 
are often similar, but not identical. There are thousands of posts on the 
Net from people needing help to decide exactly WHICH format is correct for 
the flavour of their data source.

Fujitsu Cobol's SQL is built in and only does what the compiler
defines:

   CONNECT TO <servername> [AS connectionname]
        USER <username/password>
or
   CONNECT TO DEFAULT

The servername is required to be set up in the ODBC.INF file with the
details. The .INF file is just a text file so your program could write
it before trying the connect.

Alternately you could use a pre-processor like other cobols do and not
use the Fujitsu built-in syntax, or use a CALL/INVOKE interface to
something else.

Years ago there was a large distinction. MySQL was easy to set up and
very fast on searching but did not have transactions. It was regarded
as being a lightweight server mainly for web servers (such as Google)
where there was little updating compared to searching.

PostgreSQL was the full featured database, slower but more reliable.

Now they are much closer with PostgreSQL improving its speed while
MySQL added transactions and other features (such as Oracle
compatibility). But this was done by adding complexity such as having
to choose the database type.

I've always preferred PostgreSQL for production.
