You don't say which COBOL product you are using, but with Micro Focus
COBOL there is a hight level common communications interface (CCI) which
is independent of environment and transport. Ie. using it you can have
a COBOL program that is portable (source code, not objects of course)
between environments that use TCP/IP. It will also work with other
network protocols like novell IPX, LU6.2, LU2.0, DDE, Named pipes,
NETBIOS and Async comms.

Low level coding for TCP/IP will be a nightmare in comparison I'd guess.

See if you can get hold of a copy of the standard manual Micro Focus
Object COBOL, Programmer's Guide to Communications.

I think we are currently using V5 of MVS/ESA. V2 or V3 of TCP/IP for
MVS. And the Interlink product I used, CPT/Tools, was V1. I have no idea
what version of TCPaccess they had installed. At my shop, the systems
people purchase this stuff and install it, so I'm not 100% sure on the
versions. And because of that I can't give you a price comparison
either, between Interlink and IBM.

Performance wise they seem about the same really. No noticable
differences. I did no benchmark testing to compare them though. The IBM
product seemed easier to set up. But that is from an application
development standpoint. I can't tell you what it takes to install this
stuff into a CICS region or anything. The APIs are equally easy to work
with. The Interlink package has these 'tools' that you can set up so
that you don't have to deal with the API at all. You simply set up some
TDQs and write data to them to send data, and read data from them when
data is received. This makes the CICS program very easy to write, but it
limits you in what you can do. With these tools, your socket connection
is only one way. But if that is all you need these tools are real simple
to use. Other than that, I've noticed no dramatic difference between the
two in ease of use.

Mark
