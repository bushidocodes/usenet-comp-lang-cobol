I feel for you as I've done exactly what you are about to do.  I've
since gone back to programming!

But here a few things I can pass on:

1)  Read a book by Neal Whitten called "Managing Software Development
Projects".  You can learn many good things from this book that will
help you throughout the life of your project;

2)  Lear how to manage by "deliverables".  For example, a programmer
has a program to write that has been estimated to take 16 hours.
After 8 hours the programmer says he's 50% done, or on schedule.  But
what do you have to show the 50% done and the 50% incomplete?  In
other words, the task should be broken down from 1 task taking 16
hours to maybe 4 tasks of varying lengths with something to show to
back it up. (print outs, test results, data maps etc. etc.).

3)  Ensure you have a change management process in place that is
agreed to by everyone including senior management.  The change
management process would kick in whenever someone requests a change to
the project plan that would cause a task to be lengthened by some
agreed upon amount.  I've used anything from 8-20 hours on various
projects.

4)  Be sure your project has a senior manager sponsor.  This manager
would be the final decision maker on any disputes and would be
ultimately responsible for the success or failure of the project.

There are many more things involved, but hopefully these will get you
started.
