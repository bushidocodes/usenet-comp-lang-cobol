If in doubt, run AMBLIST on the subprogram, or just open up the
library in View mode in TSO/ISPF and you should be able to see the
AMODE and RMODE for each program.

> 
> So my remit is to find out what can be done.  To this end what is the
> answer to the question
> Can programs compiled with AMODE 31 statically be linked to older
> programs that are compiled almost certainly with AMODE 24?  

Yes you can statically link AMODE(31) programs with AMODE(24)
programs.  AMODE is Adressing Mode, meaning that the object code is
generated to support 31-bit addresses, AMODE(31), or 24-bit addresses,
AMODE(24).  I have seen COBOL II load modules that were marked
AMODE(ANY),RMODE(24).  Your older programs are almost certainly
AMODE(24),RMODE(24).

> Does the
> RMODE matter?

Yes it does.  RMODE is residency mode, meaning where the program must
reside in memory, either below the 24-bit address boundary, RMODE(24),
or anywhere above or below the 24-bit address boundary, RMODE(ANY). 
The 24-bit address boundary is also known as the 16-megabyte line,
hence "above or below the line".  Some IBM services still require
control blocks below the line, so you may see some programs built as
AMODE(31),RMODE(24).  It is possible to have an AMODE(31),RMODE(ANY)
program reside above the line and dynamically call an
AMODE(31),RMODE(24) program, which will be loaded below the line at
runtime.

I am not aware of any cases where an AMODE(24) program can be anything
but RMODE(24).  If you statically bind any number of object modules
into a load module, the presence of even one RMODE(24) program should
result in the linkage editor downgrading the entire Load module to
RMODE(24).  If you generally have COBOL main programs calling non-LE
assembler programs built many years ago, this should work.  You can
also override the linkage editor to explicitly build the load module
as RMODE(24).   

>

…[4 more quoted lines elided]…

In my shop they've converted every LPAR to have the Language
Environment option ALL31(ON) in order to improve runtime performance. 
In that environment any LE-enabled program that calls an AMODE(24)
program will abend immediately.  There is a runtime parameter that can
be added to those programs, something like this:

//STEP010 EXEC PGM=MYPROG,PARM='/ALL31(OFF),,STACK=BELOW' 

If your program already has a PARM statement, then the existing
parameter data is placed before the "/" character.  Everything after
the slash character is passed to LE.  The "ALL31(OFF)" override
parameter should allow the program to run, but with some slight
performance penalty compared to an AMODE(31) program.  

The PARM won't work for a non-LE main program calling an LE-enabled
subprogram, because the PARM doesn't get passed through to LE.  In
that case you can statically link in an LE option override module for
that program.  You might still have issues with vendor supplied code.

If you are using the LE ALL31(ON) option, you can still run
AMODE(24),RMODE(24) non-LE programs (Assembler, Easytrieve) without
any problems.

I have been told that an older assembler program may not be able to
call an LE-enabled COBOL subprogram, because the old prolog code
tyically creates 24-bit addresses in the register savearea, and LE
requires all savearea chains to support 31-bit addresses.  You may
need assembler expertise for some of those older programs.  

I hope that helps!

Imagine a S/370 assembler subprogram with the following prolog code:


90ECD00C = STM   14,12,12(13)        SAVE THE REGISTERS 
18CF     = LR    12,15               COPY OUR BASE ADDR TO R12
1821     = LR    2,1                 COPY PARMLIST ADDR TO R2
4510C074 = BAL   1,116(12)           BRANCH AROUND SAVEAREA &
EYECATCHER
           DC    18F'0'
           DC    CL32'** MYPROGRM &SYSDATE &SYSTIME **'  
50D10004   ST    13,4(1,0)           STORE CALLER'S SAVEAREA ADDR
501D0008   ST    1,(8,13)            STORE LOCAL SAVEAREA ADDR
18D1       LR    13,1                POINT TO LOCAL SAVEARA
1812       LR    1,2                 RESTORE PARMLIST ADDR 
05C0       BALR  12,0                MAKE R12 AS OUR BASE ADDR
           USING *,12

If this program receives control in 24-bit addressing mode, the BAL
(Branch And Link) instruction in the fourth line loads the 24-bit
address of the local savearea into R1.  The BALR (Branch And Link
Register) instruction in the penultimate line loads the 24-bit base
address of the subprogram into R12.  If this subprogram calls a COBOL
program (and that COBOL program has been recompiled in an LE-enabled
COBOL compiler) LE will abend your program, because the savearea chain
address is invalid.  It's a 24-bit address when LE requires a 31-bit
address.

The way it was explained to me was that you had to use LA (Load
Address) or LR (Load Register) rather than BAL or BALR, in order to
preserve the addresses as 31-bit. 

Converting 24-bit assembler to 31-bit is tricky.  In AMODE(24), there
are bit switches and masks stored in the high-order byte.  Sometimes
clever programmers stored their own byte or bit flags in the
high-order byte, since the computer wasn't "using" it.

Our oldest assembler routines came from DOS/VSE.  At any rate, I
didn't see them using SAVE macros.  We had our own ENTER macro,
although that was not universally used.  The sysprogs changed the
ENTER macro to remove any use of BAL or BALR.

All savearea chains have always had room to save the entire register of 32
bits.  The addressing mode bit in the PSW affects how some instructions
work.  In amode=24 LA  only stores a 24 bit address into the register and
set the leftmost bits to zero.  Some programmers took advantage of this to
pass a parameter in those high order bits.  In amode=31 LA places a 31 bit
address into the register and sets the leftmost bit to zero.

The real problem with an amode=24  program calling an amode=31 mode program
is that the program receiving control expects to receive it in 31 bit
addressing mode BALR cannot do this, hence the newer instructions BSM and
BASSM instructions.  See Advanced Assembler Language and MVS Interfaces by
Carmine Cannatello Chapter 14 for a more detailed explanation.

loaded below the line and the AMODE(ANY) permits both 24 & 31 bit 
addresses to be passed (the mode depends on the calling program).

One part of the equation not discussed in other responses is the ability in
VS COBOL II and later programs to compile your programs with DATA(24) - but
link-edit it with RMODE(ANY) and AMODE(31).

With DATA(24), you *may* dynamically call AMODE(24) programs and "pass"
parameters to them (below the line) while still keeping this program "above"
the line.  This can be QUITE useful if you have "vendor provided" AMODE(24)
programs that you need/want to interface with.

P.S.  There is also a "new" term that you may run into and that is "above
and below the BAR".  This refers to 64-bit addressing.  So far COBOL doesn't
"play" in that issue, so you shouldn't have to worry about it.  However, you
may run across it in some "current" information.
