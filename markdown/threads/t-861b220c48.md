a start. They are both obsolete.  The block statement really only applies to

larger than the record description in the copy.  You are probably forgetting
the record delinator at the end when you counted the characters.  If you get
rid of the record contains statement, and let the compiler count the
characters, then the problem will probably correct itself.

your programs to run well you need to add non-standard code.

I'll disagree with Randall on this one.

I view "BLOCK CONTAINS 0" to be as valid an extension as "GoBack".  The
Standard tells you what block contains 1 thru N means - it does not tell you
anything about what Block Contains 0 means (or that it is any more "illegal"
than "transform" as a verb).

The fact that most PCs actually treat "Block contains 2" as "documentary"
and do NOT do blocking *might* be considered non-Standard - but even that is
in the "hardware dependent" area that I would consider totally acceptable.

In fact, the only real "bug" in compilers (which almost all PC and UNIX
compilers have) is when you create a file defined with "Block Contains 2
records" and then read it (OPEN INPUT) with "Block contains 3 records" and
do NOT get a file-status of 39 (which the IBM mainframe would give).  This
is certainly AGAINST the "intent" of the '85 Standard (although a later
interpretation "sort-of" allows it).
