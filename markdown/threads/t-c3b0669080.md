```
I strongly recommend use of this register on programs. What it does is
place the time and date of compilation in a field where you can display
it or print in on the I/O total log. In case of run problem, this is
very helpful in back tracking, as this will show whether the program was
recompiled or not when it should (or should not) have been.
```

```
The first few bytes of WS in *my* programs are always devoted to
program-id literal
version #
when compiled
when run
[01-level format not shown here.]

In addition, every batch program has an audit/stat file that shows
the above information plus date/time completed, appropriate stats
(e.g., records in), and -- if needed {shudder} -- error/debug info.

And every report includes the program-id of the MAIN program that
produced it.

This file is operations-documented for the Key User [if s/he is
computer literate] or the maintenance programmer ONLY.

Makes life a lot easier if you can say
The reason that your 7 April run of BZD071T didn't give
you any report lines is that there were no new transactions.
[Of course, in this case, one _should_ classify this as AT LEAST
a warning-level error and print it on the end user's report.]

My philosophy when developing error messages, reports, and docs is
Empower the end user,
Eliminate support calls to the maintenance programmer
This takes more time&effort to begin with (tho' it's "automatic"
with me now), but it not only saves loads of FUTURE time&effort
for the developer, it also makes for much happier end users.
And happy end users are both more willing to authorize new development
(which is what *I* would rather do than answer HelpDesk questions) and
more understanding when that dreaded "oops!" happens.
```
