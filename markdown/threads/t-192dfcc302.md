My client is planning to use a called date routine in both CICS and batch.
This routine in in COBOL 85 and compiles cleanly and executes in both COBOL
II and COBOL for MVS and VM. In batch it will be a dynamically called
routine. The choices for CICS are either use it as a nested program or as a
dynamically called routine. (Static linking has some problems in an
evironment with both COBOL II and COBOL for MVS and VM that we have not felt
worth resolving.) If it is to be a dynamically called routine, it would be
identified to CICS as a resident program. The concern is about the freeing
of the WORKING-STORAGE obtained by the date routine upon initial entry.

1. Is the storage automatically freed when the caller executes either EXEC
CICS XCTL or EXEC CICS RETURN?

2. Is the storage automatically freed when the caller abends?

3. If it isn't freed in both of the above circumstances, would it be freed
if the caller executed the COBOL CANCEL verb after each call?

4. Would a better alternative to 3 be defining the CICS version of the
routine as an INITIAL program on the PROGRAM-ID statement of the CICS
version of the routine. The assumption is that the WORKING-STORAGE would
be obtained on every entry and freed on every exit.

Also implicit in the above questions is that any other storage obtained would
be freed at the same time the WORKING-STORAGE is freed.

(note that the source for the separately compiled program is a copy statement
and that all of the code for the program is in the copybook)

We use dynamically called programs in CICS (ESA ver. 4.1.0) for Cobol II
pgms a lot. All we've done is:

provide a PPT entry for the CALLed program,
code 'CALL data-name USING parm1 parm2...' to force the dynamic call
(NODYNAM is the compile option for CICS programs).

We have achived what appears to the developer to be the same subroutine
for both Batch and CICS, by setting up the subroutine as an assembler
front end. The assembler front end program can determine what
environment
you are running under at execution time, and dynamically call the
correct
module for that environment.
PRO: Developers only learn one program name, and one set of
parameters.
CON: You have to maintain 3 components.
We don't use this coding trick anymore. Our current preferred method is
to make two copies of the subroutine for the two environments.

On dynamically calling subroutines under CICS, we became aware that the
the issue of being re-entrant rears its ugly head. That is two tasks
calling
the same subroutine could get each others subroutine results, because
the
subroutine only has one set of workingstorage in memory. To overcome
this, our
CICS subroutines are usually passed a large commarea with lots of FILLER
from the calling program, and the subroutine then uses that space
for its "working storage" area. This assures that two different
concurrent
CICS tasks that call the same subroutine can not get each others
subroutine
results.

I agree - as I understand it, in IMS/DC you need to re-initialize your
W-S variables if they might have been changed by a previous invocation
of the program. In CICS each task gets a fresh copy from the original
W-S, for each W-S referenced by the TGT's. So your variables come fresh
with the values you declared when they were coded. I learned this the
hard way when I began work at an IMS shop after 10+ years of CICS. I
dimly remember an old version (70's) of CICS where W-S was shared, but
that may have been MACRO level.

Here's another solution to the original problem, modeled after what some
vendors provide. In order to make their code portable, they keep the
CICS logic in a separate sub-program, and the processing logic in a
separate sub-program, and the DB2 logic in a separate sub-program, and
the flatfile I/O,..., ..., etc. Then you mold various conglomerate
programs according to your needs via INCLUDEs in the Linkage Editor. The
ones I worked on were via a Static link, but right off the top of my
graying head, I can't picture why they couldn't be Dynamic. Each
sub-program has some expectations of Input and Output passing, and must
be compileable by itself.

There are considerations (problems?) with this. One biggie is
remembering where each sub-module is used. You need to recompile all the
callers if Static links are used and you change a sub-routine. Another
is debugging thru the various routes. Some of the packaged debuggers
don't like traveling through a road littered with programs. This is not
a simple solution to hand to the new guy from Bill Gates' University.
