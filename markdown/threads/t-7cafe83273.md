```
Starting a new thread (as a follow-up on another thread),

Some comments on WHY (personally) I would suggest a style with minimal
periods and Next Sentence phrases:

1) "Next Sentences" is in the "archaic" category in the 2002 COBOL Standard.
Although it won't be SOON, it is certainly possible that COBOL code that you
write today *may* still be in use at some time in the future when "next
sentence" is *not* a portable feature to all compilers.

2) "Next Sentence" with "END-IF" is an extension today (and has been since
'85 when "END-IF" was first introduced).  Therefore, the use of  "next
sentence" in some (not all) constructs is already NOT portable to all
conforming ('85 much less 2002 Standard) compilers.  This may well be a
"dropped" extension in some future compiler provided by vendors who support
it today.

3) Whether working with printed or "viewed on a screen" source code, a
period (aka "full stop") is relatively easy to miss.  This (traditionally)
was/is one of the biggest problems in maintaining code that requires KNOWING
where the "next period is".

4) There are STILL many programmers who *think* that a "Next Sentence" *is*
the same thing as a "continue" with an End-IF (or End-Search).  "Fixing"
this is "only" a matter of training, but is a constant training issue - as
long as there are new COBOL programmers or new programmers to a "shop" that
uses "Next Sentence".

5) Periods in the middle of paragraphs (or sections - if that is your
preference) *do* change logic flow of Next Sentence phrases.  Such "next
sentence" phrases *may* be far away from both the original and any "new"
periods - in my experience this is a maintenance problem.  (it certainly
WOULD be a problem if you had a precompiler inserting either "Next Sentence"
phrases or "periods" where the maintenance programmer doesn't see them -
either before or after added new code.)

    ***

None of the reasons listed above are sufficient (IMHO) to do a "major"
rewrite of an existing/functioning program or application.  On the other
hand, I *do* view them as sufficient reasons to ENCOURAGE (not require) a
change to a "single period per paragraph style of coding with NO "next
sentences" included in new programs and new "major" insertions of code/logic
in existing programs.
```

It looks to me as if you recognize a reason NOT to use periods, but want to keep
using them anyway.  I'm part of the "don't code 'em if you don't need 'em" camp,
and since you never need 'em except at the end of a procedure (paragraph or
section), I recommend not using periods anywhere except there.

I seriously doubt if there is a COBOL programmer anywhere who hasn't seen some
of the following:
1.  A period that caused an logic error.
2.  A missing period that caused a logic error.
3.  A set of imperative statements which all ended with periods that had to be
made subordinate to a conditional (IF, EVALUATE, PERFORM ... UNTIL, etc.)
statement, meaning that all of the periods had to be removed.
...and I'm sure there are more.

The standards writers gave us a way out of this set of potential messes by
providing the scope terminators.  In addition, they gave us more freedom to code
what the program requires, without artifical solutions such as a PERFORM that
moves code out of line, or writing the same statement twice, etc.

Not to take full advantage of the improved language is just like throwing small
change on the ground, because it is too much trouble to carry it around.
Eventually, you've cost yourself some real money.

    I do use scope terminators.  You can get into trouble either way,
whether you use one period only per paragraph or if you jam one in
anywhere you can make one fit.

    My point was that I do not understand why I should consider
removing all periods but one as an end in itself. I have personally
experienced trouble with go to's and perform thru's.  Debugging problems
here can be especially hard when the go to in question is used only after an
error.  But I can see pro's and con's for both one period and many period
styles.

    Simple example - If's nested six deep, but only five END-IF'S.  If you
put a period after the fifth end-if, there is no error.  Without the period,
all statements up to the end of the paragraph are erroneously part of the
if.

    Oddly enough, there can be an advantage to using both scope terminators
and periods.  Extra periods can result in compiler warnings.  The compiler
is
much better at spotting that sort of thing than any person.

    Maybe the best bet would be a compromise.  Cut back on routine
periods, but continue to use them after the final END in a sentence.

I don't see it as an end in itself. It's part of a package of teaching 
programmers to use scope terminators consistently rather than depending 
on periods to end scope.

>     Simple example - If's nested six deep, but only five END-IF'S.  If you
> put a period after the fifth end-if, there is no error.

I strongly disagree with your conclusion. "There is no error" ONLY if 
the missing END-IF is the LAST one. But if I omit the last END-IF, it 
will stand out because of the prominent gap in the code, and it's very 
likely that I'll catch it by eye, and so adding the period didn't buy 
me anything. (Of course, I have all the code indented properly. I also 
leave a blank line after an outer level scope terminator, making its 
absence even more prominent.)

OTOH if I omit an inner END-IF, then it's far less obvious to the eye. 
There's generally code following the missing END-IF at the same 
indentation level, and code farther down to the left. I'm a lot less 
likely to see the missing END-IF. This is not just theoretical; I know 
this from experience.

And if I omit an inner END-IF and then later close it with a period, 
then I have almost certainly introduced an error. The period didn't fix 
the error, but rather helped to hide it (by limiting the scope of the 
bug and thus making it less likely to be detected on any given test).

Now again, I agree that omitting the period doesn't solve this problem. 
Only using a tool which requires matching scope terminators for all 
conditionals solves it. But programmers who are accustomed to using 
scope terminators consistently are much more likely to want, use, ask 
for, insist on, a tool to do this. And are also more likely to catch a 
missing END-IF by eye. Abjuring excess periods is just a part of the 
whole package.

The more I write about this, the more I want all my programs 
automatically checked every time I compile ...

Compilers cannot warn about missing END-IF in general because COBOL85 
never requires you to use END-IF. This feature can only be an option

restricted for safety. But if the user has requested to be warned when 
exceeding this subset, then the compiler certainly can warn for every 
END-IF that is missing according to the subset definition.

Safety-restricted subsets have a long history, and can be valuable 
tools.

On Thu, 2 Jan 2003 10:36:07 -0500, Howard Brazee wrote
> Would you want it for all commands that can have an END?  Or would 
> it be better to only flag missing END IF's? 

It should flag the missing scope terminator for all *conditionals*.

I place the scope terminators in four categories:

1) PERFORM

2) primary conditionals: IF, EVALUATE, SEARCH

3a) statements with secondary conditional(s), for example READ AT END, 
STRING ON OVERFLOW, etc.

3b) statements which can have secondary conditionals but as written do 
not -- for example, READ with neither AT END nor NOT AT END.

PERFORM (1) is not an issue, as there is never an optional END-PERFORM.

Primary conditionals (2) must always include the scope terminator in my 
safer subset.

Statements with secondary conditionals (3a) must always include the 
scope terminator in my safer subset.

Statements without conditionals (including 3b) should not include the 
scope terminator. However, they may include the scope terminator, due 
to the odd cases which require either a scope terminator or some other 
syntactic de-fuzz to avoid ambiguities. (I'm tempted to say never 
include the scope terminator, and use a trivial PERFORM/END-PERFORM to 
disambiguate when necessary. But I haven't studied this cranny 
carefully enough to state that this is always a viable alternative.)

The distinction between 3a and 3b allows us to make maximum use of 
scope terminators without absurdities like including END-ADD for every 
simple ADD, a practice which would make most programmer revolt.
