That may be so, but I haven't looked at a memory dump for a PL/I program for
some 30 years.  Haven't needed to, because the facilities provided by
PL/I for locating errors are excellent.  These facilities include
printing the source statement number when an error occurs, and the subsequent
printing of the values of variables under the control of the programmer.

>If you compile your COBOL  module with TEST(NONE,SYM) your load-module is

I can do that in PL/I -- WITHOUT A MEMORY DUMP -- simply by specifying
the GOSTMT option (or in one of IBM's compilers that doesn't support it,
by specifying OFFSET to provide an offset table).

The values of variables can be printed after an error occurs, in decimal
or character -- that is, in a readable form (see later).

> This is impossible for PL/I (read the manual section CEE3DMP that states
> that the 'variable' option is valid for all languages except PL/I). PL/I LE
> dumps display only the statement number but not directly the content of the
> variables (compilation option GOSTMT)

As I said, the GOSTMT option option will give all that to
you in PL/I without the need for a memory dump at all.
Furthermore, after an error has occurred, a PL/I program can print the contents
of variables, decimal variables in decimal, character variables in character
form, and so on.
A memory dump is not a good place to begin looking for the values of variables.

> About the debugger:
> The debugger is common for all the language, so not advantages here

One can use the debugger, but PL/I has its own built-in debugging facilities.

> - about error checking
> You can activate option that check the same things as PL/I (either clause ON
> ERROR in the code or compilation option to check arrays out-of-bound
> conditions or overflows)

If it's a compilation option, it applies to the whole compilation unit.  But
that's a facility that's not part of the language.
When it's an ON ERROR clause, it applies to a statement.

PL/I has the means to enable the subscript check on a particular statement,
or to suppress it on a particular statement.

The check can be enabled for an entire procedure or BEGIN block, or disabled
for a whole block.  Individual statements within that block may 
enable or disable it.

These facilites are part of the language.

>In PL/I
> - about the size of the source
> It is much easier to create smaller modules in COBOL and dynamically call
> them (even under CICS).

One can write simple procedues in PL/I.  They can be as small as one
statement (apart from the PROCEDURE and END statements)
They are simpler to write than COBOL.

> Even if PL/I allowsto dynamically fetch modules,

IBM mainframe PL/I does.

> there are really annoying limits due to the language itself that block you
> in doing this (the famous fetch limitation, that should be alleviated in the

A standard facility?

> however nobody uses this in COBOL because there is
> no such limit in COBOL than in PL/I about the size of stuff that the
> compiler can optimize (for example the PL/I compiler optimizes only the
> first 255 variables...)

There is no such limit in any of the PL/I compilers I have used.
Such a limit is NOT a language feature, it's an implementation restriction
completely unrelated to the PL/I language.

> - about the easiness of PUT DATA:
> With DISPLAY in COBOL you can do exactly the same thing by simply writing
> DISPLAY <var>,

PUT DATA; is a complete statement that writes out the NAMES of the variables
AS WELL as their VALUES.
PUT DATA (K,X); writes out the names K and X as well as their values, in
the form K = 12345, X = 9.8765;
Does DISPLAY do this?
DISPLAY would, however, be equivalent to PUT LIST ('V=', V); which tends to
become clumsy for structures and long names.
