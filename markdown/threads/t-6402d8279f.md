Hi David,

Our company has three COBOL tools we market directly. In addition, we
are the developers of the user interface tool sold by the RM/COBOL folks
as RM/Panels. Our web site is http://www.netins.net/showcase/etsinc.

The products we market directly are:

Cobol-WOW (Windows Object Workshop)

This is a Windows application development tool for COBOL. It provides a
Visual Basic style development environment that integrates visual
interface design, application coding, compilation, execution, and
debugging, and a runtime. We have just released the 16-bit RM/COBOL
version and will release the 16-bit Acucobol version very soon. Later
this year we will release the 32-bit version for MF, RM, Acucobol, and
possibly others. The 16-bit version has support for VBX controls and the
32-bit version will support OCX and ActiveX controls.

What makes Cobol-WOW special is that it makes it easy to write business
applications in COBOL at the Windows level. It takes care of the
mechanics of event handling and makes the Windows API easy to use. In
addition, it provides two simple calls that can set and retrieve any
property of all windows, standard controls, and VBX controls,
eliminating the need to rely on the Windows API for all GUI operations.

We do not have any information on Cobol-WOW available on our web site
yet, but we will soon.

Cobol-RPC (Remote Program Call)

This is a distributed logic tool for COBOL. Cobol-RPC allows client
programs written in COBOL, VB, Delphi, and 'C' to call and execute COBOL
programs on a Windows or Unix server. When COBOL is being used on both
the client and server Cobol-RPC works by extending the capabilities of
the COBOL CALL statement. When used with other client languages
Cobol-RPC provides a simple CallCobol function.

With Cobol-RPC normal COBOL data is passed between client and server.
COBOL programs on the server simply declare the data to be received in
their LINKAGE SECTION.

Cobol-RPC has been generally available for about 1 year and the
installed base is growing very quickly. It is available for Acucobol,
RM, MF, and MF Visual COBOL (old mbp cobol).

Cobol-CGI (Common Gateway Interface)

This product makes it easy to interface COBOL programs with web pages on
Windows and Unix web servers. It provides two functions; CGIREAD and
CGIWRITE, that make communicating with the client simple. Cobol-CGI has
just been released for Acucobol, RM, and Fujitsu.

Thanks for the opportunity to present our products.

Sincerely,

Bob England

Jim,

We too ran into the same problem, and the solution was to specify
compiler option TRUNC(OPT), which IBM recommends if you used TRUNC with
OS/VS COBOL. We've been doing this for years with minimal headaches.
Running under CICS is a little different--IBM makes the same
recommendation as for COBOL/370 and COBOL for MVS, but we went with
TRUNC(BIN) for COBOL II (we having migrated to LE yet).

The following is an excerpt from the 'COBOL/370 and COBOL for MVS & VM
Compiler and Run-Time Migration Guide', Chapter 10. Compiling Converted
OS/VS COBOL Programs:

TRUNC controls the way arithmetic fields are
truncated into binary receiving fields during
MOVE and arithmetic operations.

Use TRUNC(STD) if your shop used TRUNC as the
default with OS/VS COBOL.

Use TRUNC(OPT) if your shop used NOTRUNC as
the default with OS/VS COBOL (except for
select program that require guaranteed
non-truncation of binary data). For programs
that require non-truncation of binary data,
use TRUNC(BIN)--especially if there is a
possibility that data being moved into binary
data items can have a value larger than that
defined by the PICTURE clause for the binary
data item.

Note: On COBOL for MVS & VM, programs
compiled with TRUNC(OPT) can give different
results than OS/VS COBOL programs compiled
with NOTRUNC. Mainly, programs can lose
non-zero high-order digits. For statements
where loss of high-order digits might take
place, COBOL for MVS & VM issues a diagnostic
message indicating that you should ensure that
either:

The sending items will not contain large
numbers, or
The receiving items are defined with
enough digits in the PICTURE clause to
handle the largest sending data items.

I thought this was common knowledge in the COBOL2 mainframe world
especially to the conversion people. COBOL2 has a bug when using COMPUTE
ROUNDED. COBOL2 rounds off intermediate results using the number of
decimal places of the resultant. COMPUTE PERCENT ROUNDED = 2 / 8 * 100
will give you zero if PERCENT is PIC 999 but will give you the correct
answer of 25 if PIC 999V99. What happens is COBOL2 looks at 2 / 8,
rounds off the result, multiplies the intermediate result by 100, then
rounds off the result again. It will also give you the right answer if
you do this: 100 * 2 / 8 with PIC 999.
