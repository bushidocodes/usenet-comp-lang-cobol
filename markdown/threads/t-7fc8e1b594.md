Hi:

When you look at some of the samples posted by people taking
COBOL courses, it becomes apparent, to me at least, that these
people are not being taught how to program but how to code.

Using the unfortunate Deek as an example (this is NOT to
criticize Deek but the people who are 'teaching' him). If
he had a decent teacher who knew what they were teaching, it
would not be necessary for Deek to resort to seeking help
from the forum. (Does Deek resort to the forum because his 
teacher is hopeless, or what?)

The latest sample posted by Deek appears to be some sort
of report program since it has heading and detail lines
in WORKING-STORAGE. Presumably written in approved 
structured style, the first part of the PROCEDURE DIVISION
says:

MAINLINE.
PERFORM 100-INITIALIZATION.
PERFORM 200-PROCESSING.
PERFORM 300-TERMINATION.
STOP RUN.

Isn't this an awful oversimplification of the logic required by 
Deek's recently-posted 'Project 5'?

At present, Deek's program is far from complete. The likelihood of
him ever getting a thoroughly professional job done on this program
is remote because he hasn't the faintest idea of what the program
NEEDS TO DO and I don't see anybody showing him what he needs to 
know. Again this is not to criticize Deek but his teachers. 

Put yourself in his position. You know NOTHING about computers,
COBOL or business application design. How the heck are you
supposed to know about little things like breaking control,
page headings, checking for invalid conditions, etc. if
you have NO experience. How do you know that minor totals
have to be printed before major totals, etc.? How do you know
how to design records which lend themselves to efficient coding?
You don't.

You can see that the teachers show their students how the 
various divisions are coded and the syntax, etc. They are
probably taught the approved structured method of coding
using PERFORMs, and the extremely important business of
having only one period per paragraph which always goes on
a separate line.

But, do they teach them how to PROGRAM?

In the olden days, there used to be programmers and coders.
The programmer would design the program and the coder would
write the code from the flow chart prepared by the programmer.

Certainly, before you sit down to write a program you have
to decide what it is going to actually do. Does the absence
of a page-counter and line-counter in Deek's W-S mean
he doesn't know he will need them? Does he really know
all the things his program must consider? Unlikely.

Deek's problem is that he has never been taught the things
which he needs to be taught. He will learn the hard way -
like most programmers - either that or he will give up.
The typical cycle of program completion for students 
like Deek is to write something, find out that it is
lacking, stick something in to fix the lack, find another
one, stick in more code, find another one, stick in 
more code and end up with a kludge. If the student 
knew in advance about all these crucial details, they would 
be incorporated in the first version, somehow. But how?

When I used to hire programmers, the first thing I would
ask of them is to draw me a flow chart for a simple report
with two levels of control. If you were to ask Deek or
any other student to do this, what would you get? For sure,
you would get umpteen variations ranging from the ridiculous
but not to the sublime.

Another thing which Deek won't be taught is that most of
the programs he may ever write will be all basically the same
without regard to application or the nature of the enterprise.
All report programs have page counts and line counts and print
headings, break control, and so on but at Deek's stage of 
development he probably does not realize this.

If Deek perseveres with 'Project 5', he will learn about
all the little details as his work progresses. Unfortunately,
again, this will require sticking in code to fix all the shortcomings 
which should have been recognized before he started. 

It seems a shame, but these days people don't seem to consider
flow charting part of being a COBOL programmer. Structured
programming seems to have eliminated flow-charting. As someone
said in some other thread when asked about flow-charting, 
'no, we just sit down and start typing'. That may be fine
for someone who has lots of experience but it is not the way
for Deek or any other student.

Anyway, I have prepared a Web-page which may be of help
to Deek and any one else who wants to be a business COBOL
programmer.  Just remember, this is not about coding, it is
about programming.

To see it, go to: foodman123.com/flow2.htm

Thanks

Tony Dilworth

Having seen some of your previous discussions on your programming
mechanisms it doesn't surprise me that you think that this is the case. 
Certainly it seems that you mostly rewrite the same programs over and
over, in fact you have said this to be the case.

However, I haven't written any code connected directly with "page counts
and line counts and print headings, break control" for many years. I
would hope that at the appropriate time he is taught that he should
_not_ be writing programs 'all basically the same', that is a complete
waste of expensive resources.

I got very tired of rewriting the same report and maintenance programs
early on in my career in fact when I moved from operations to
programming, 30 years ago, the very first production program that I
wrote from scratch was an invoice print that I developed in two distinct
parts: a reusable framework that handled all the file access and form
handling logic and a set of specific modules for page headings, product
item lines, page totals, invoice totals, etc.

20 years ago I started development of my 'synthetic system' which
includes a 'report writer' to handle the majority of enquiries and
reports for my systems.  This reads a simple report script and uses
dictionaries to determine where the data comes from.  In many cases the
users can construct their own reports, mostly it takes me a few minutes
to write the script to suit their requirements, occationally I add code
into the program to cope with specific cases.  There is a GUI front end
that allows 'click'ing a new new report from selection lists.

The 'synthetic system' also includes a table driven maintenance program
so that the basics of a new system can be built entirely by building
some dictionaries, screen layout tables, report scripts and other system
control text items.  The procesing programs are then added.

While you seem to prefer the rewriting of the same set of programs I
prefer to use one set of programs in each new environment.  This also
has the benefit that old systems get updated when I add in new features
for a new system.  

Another approach is to write a 'program generator', which I have also
done.  This allows repetitive programs to be expressed as text scripts,
such as print-line layouts, dictionaries, screen layouts and such and
then to generate the code from templates.
