```
With VSE if you want to include a separately compiled 'static' module with
your program the static module must compiled into an object module and then
cataloged to a library.

This also works with z/OS.  Yet it appears that you can also link a "fully
linked" load module and it appears to function just as well.

Please don't focus too much on my very like erroneous terminology, but feel
free to correct it.

Let me give an example...  We have a sub-routine called SUBR (yes, it
actually is called SUBR!).  We can do a compile followed by a link and place
the resulting module in to a load library.

A separate program can do a dynamic call of SUBR (either by using CALL
literal or by using the DYNAM compile option).

What surprised me is I can also do a static call of the routine, and during
the link-edit (bind) step, as long as the library that contains the SUBR
load module is in my SYSLIN concatenation the SUBR routine is automatically
linked in to my final load module.  As far as I know this is not possible
under VSE.  You can only link object modules together, not load modules
('phases' in VSE).

Is this something that has always been available for MVS and its successors,
or is this something fairly "new" (where new might mean within the last 20
years!).

I just want to make sure this isn't "too good to be true".  All my testing
indicates that it works, but I don't know if there are cases where it
won't.

Don't get me wrong, I am happy it works the way it does.  If we ever wanted
to do something like this on VSE we had to have both the OBJ and the PHASE
versions of the modules available.  This way we need only one.

Oh, and don't take this as saying we do a lot of static calls.  We prefer
dynamic generally, especially for those modules that actually change.  But
the SUBR module hasn't changed in ten years, other than to add a few new
subroutines.  Now we can leave them as static calls and still have the
ability to change them later to dynamic calls, while requiring only having
the single compiled module.

I should note that SUBR is written in assembler.  I haven't tried it yet
with a Cobol subroutine.  I'm assuming it will work as well, but who knows.

Frank
```

```
Frank,
  Yes, under "MVS" (aka z/OS) you can "link-edit" (or more recently "bind") 
either object modules or load modules.  If you are using a mixture of object 
modules and load modules, you need to be "careful' about which version of 
identically named object code you bring into the final module. 
(Link-edit/binder statements control this).

I couldn't tell from your statement if you were trying to do "dynamic" calls to 
a program that was STATICALLY link-edited (bound) into the same load module. 
This is a VERY dangerous thing to do and can lead to "unpredictable" results. 
Similar unpredictable results happen if you "mix" static and dynamic calls to 
the same subprogram in an application, i.e

Compile with NODYNAM and
  call "literal-1"
    and
   call identifier-1  *>  with "literal-1" as the value of identifier-1

There used to be some warnings about this somewhere in the COBOL programming 
guide.  If you need me to, I can try and find it.

P.S.  In z/OS shops, it is actually VERY unusual to "archive" object modules; 
usually, only load modules are maintained/retained - because there is not need 
to retain object modules for "future use".  Some shops keep object modules, but 
certainly not most that I know of.
```

```
Yes, it does work exactly the same way for COBOL.  You can dynamically 
call a COBOL subprogram from a load library, or you can statically link 
the same COBOL subprogram into a main program.

There's another useful trick you can do with the IBM linkage editor 
(both the old version or the new Binder, as far as I know).  If you have 
an old main program with a statically bound subprogram, and a new 
version of the subprogram, you can run the Linkage Editor (IEWL?) with 
commands to tell it to REMOVE the statically bound subprogram from the 
main program and REPLACE it with the new version of that subprogram. 
You don't need to recompile the main program in order to link in the new 
subprogram.

You do need to be careful, I remember a case where we had a main program 
that called subprograms B, C, and D (all static calls), but D also 
called B.  A careless programmer managed to get a different version of B 
linked to D, from B linked to the main program.  So depending on whether 
B was called by the main program or by the D subprogram, you got 
different results.

That was 20 years ago.  Now our standard is all dynamic calls and the 
problem does not arise.
```

```
Unix does not allow static linking of code packaged for dynamic loading .. on purpose. 
If it did, people would abuse deployment by statically linking third party libraries. I've
seen quite a few failed attempts. One well-known global company, which thinks static
linking is a religion, routinely deploys executables sized at 150-200 meg. About  half is
dead code that can never be executed, linked because they include whole libraries to get
one function. Unix comes with a 'strip' tool that removes dead code. It shrinks their
executables to 75-100 meg. They refuse to use it; I guess it's against their religion. 

It used to be semi-common practice to statically link the Micro Focus runtime, which adds
1 meg. You can't do that any longer. 

Defenders of static linking usually give excuses 1. or 2. below; they refuse to admit the
real reason, 3. below.

1. Faster calls. 
    At program start time, Unix loads ALL dynamic libraries and puts entry point addresses
into the symbol table, in the same place where static goes. Calls are exactly the same
speed. The only way they might be slower is when the function name is a variable that must
be looked up. Cobol is the only language (I know of) that encourages, but does not
require, dynamic calls through a variable (which, in fact, doesn't change). The solution
is to use linker options to make Cobol calls dynamic, not by calling through a variable. 

2. Security
  The're afraid a bad guy will put a function containing malware ahead of the real one in
the execution time search path. The Unix linker has many ways of preventing this from
happening, including remembering the directory where the function was at link time,
remembering the version number and remembering the function's hash. Windows has even
better safeguards available (although they're seldom used). 

3. Poor control over deployment and execution environment
  They've been frustrated by jobs failing because the wrong library was loaded at
execution time. Rather than deal with the actual problem, they statically link everything
they can, including functions in sister systems called by the executable, which in turn
call second cousin systems. Companies that do this are also unable to keep dependencies
straight, so wind up doing weekly 'full builds' taking 15 hours.
```

```
Robert wrote:
> 
> Unix does not allow static linking of code packaged for dynamic loading

Some Unix variants do. The Unix specification (SUSv3) doesn't cover
linking, except for specifying the interface for the dlfcn API, the
actual behavior of which is left to the implementation.

On any Unix platform, there's nothing to prevent creating additional
executable formats, which could support both static and dynamic
linking of a single module.

> Unix comes with a 'strip' tool that removes dead code.

strip removes unneeded symbol entries. I'm not aware of any
implementation that removes dead code.

SUSv3 leaves it up to the implementation what strip removes, but says
that its "effect ... shall be similar to the -s option to c99". In
historical and mainstream contemporary compilers, -s removes unneeded
symbol information, such as debug symbols.

The GNU strip only removes symbols.

> At program start time, Unix loads ALL dynamic libraries

Except for the ones that it doesn't. Various Unix dynamic loaders
support various kinds of deferred loading. And pretty much all of them
support explicit loading as well, which obviously doesn't happen at
program start time.

> and puts entry point addresses into the symbol table, in the same
> place where static goes. Calls are exactly the same
> speed.

This is not true for ELF, which is probably the dominant Unix
dynamic-linked binary format. See for example the discussion in:

	http://vx.netlux.org/lib/vsc06.html

In ELF, each shared object used by one or more active processes will
have a shared Procedure Linkage Table, and each process has a private
Global Offset Table.

By default, the first call to an entry point in a shared object will
invoke the runtime dynamic linker to resolve the symbol and put its
address in the GOT, using information from the PLT. The PLT can't be
changed to point directly to the entry point, because it's shared by
other processes, which may have loaded the shared object at different
addresses.

On subsequent calls, the entry point can be called through the address
in the GOT. Note, however, that this is still an indirect call, unlike
a call to a statically-linked entry point.

There's an environmental option to force early binding of GOT entries,
but it's rarely used. Lazy binding is the default.

Most other Unix dynamic linking implementations are similar. AIX's
XCOFF model, for example, uses indirect calls through the TOC (Table
Of Contents) section.
```

```
2000 project I was on, I found that there was a repeatable and
measurable difference between a dynamic call and a call to a nested

dynamic link.  Note that the overhead was not enough in any shop that

compiles must be NODYNAM unless the rules have changed.
```
