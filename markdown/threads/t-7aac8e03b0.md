Here's one man's opinion:
--------------------------

The future of OO COBOL is doubtful.

Not because there is anything wrong with it... in fact, it is a superb
product.

The reason it is doubtful is because the mainframe arena has strongly
resisted it (rightly or wrongly... the debate has raged endlessly...) and
this has meant withdrawal of support for it by IBM.

(Bill Klein will correct me if I'm wrong here... I've given up trying to
follow what is happening in "Fortress COBOL" ...)

By the time they realise that this is an essential part of distributed
systems (which the Fortress COBOL people are not too keen on anyway) it will
be too late, and COBOL will be relegated to a Batch processing development
Language  for Big Iron. (yes, I know everyone is thinking that CICS will
keep it alive... don't hold your breath. There are developments currently in
the pipeline that will render even CICS pointless. I predict that within 15
years NO mainframes will be doing teleprocessing. It just isn't economically
sensible when machines with equal or better capacity can be obtained for a
third of the price. The WWW and Java will take over the on-line processing.
There is one possible reprieve; that the price of mainframes and the
infrastructure necessary to keep them running, falls dramatically. I believe
this is unlikely, but I could be wrong.)

OO COBOL on Client/Server is excellent and, fortunately, interest in "modern
COBOL" appears to be increasing. We are seeing more youngsters doing COBOL
courses. (Unfortunately, many of these courses are just teaching traditional
Batch COBOL which leaves the students kind of bewildered, and with a
completely wrong impression of the current capabilities of the COBOL
Language. They learn about OOP in the Java part of their course, but never
get to see that the same facilities are availabe through OO COBOL)

I have been using OO COBOL for just over two years now and I find it an
excellent complement to Java. I use it to build Components that can be
deployed on servers and in desktop programming. I use it to link to OLE
servers and so combine all the "standard desktop software" into my COBOL
applications. I am sold. It works for me. However, I am also realistic
enough to realise that unless it is embraced by Fortress COBOL it's days
will be numbered.

The real question we have to ask is: "Is there a place for OO COBOL in the
application development of the Future?".

The answer appears to be: "It depends..."

It depends on what application development you are looking at. For the stuff
I am doing, it is brilliant.

For mainframe programming, the answer appears to be "No". (Personally, I
reject this. There are demonstrable benefits to be obtained from OOP in ANY
environment... OO COBOL is the route to these benefits in the mainframe
environment, but there is such fierce resistance to change in this
environment, (and there is no denying that these are new concepts to the
average mainframe COBOL programmer, so a learning curve is involved), that
the result is the rejection of this Methodology.)

I believe that by the time the COBOL groundswell wakes up to the need for
it, it will be too late. Other OO Languages will have claimed the ground
that was once held by COBOL. (Ths ground is shrinking anyway, see next
paragraph...)

COBOL is axiomatically the best language for developing commercial computer
applications. But the lines are getting blurred. The days when we needed to
maintain source code (for which COBOL is the perfect language) are drawing
to a close. We are entering a future where software development will NOT be
undertaken by EVERY corporation. It will be too expensive to maintain
development on-site, especially when the Company is NOT in the IT business.
Component based solutions will reduce the cost of "tailored packages" to the
point where it would be insane NOT to use them. In this environment the
Source code will be irrelevant. You will simply purchase and "plug in" the
functionality you want at the point where you want it.... Eventually "Smart
Software" ("Wizards", if you like) will work with non-technical end users to
do this.

OO COBOL is a perfect tool for developing commercial Components.

So, there you have it. If you are planning a career as a specialist IT
Component builder, employed by an IT provider, then go for it. If you expect
to use this skill while working in an existing IT department (part of a
non-IT oriented corporation) there is little point in learning it. (At least
at the moment... it MAY change, but, based on the observed reactions and the
evidence over the last couple of years,  I am not optimistic.)

Pete.

Let's turn the discussion inside out -- instead of talking about languages, 
talk about what they're being used for...

There are two type of programming environments -- procedural-driven and 
event-driven.  A procedural environment is yer typical batch process -- the 
code drives the procedure; you code the logic to identify what happens 
next.  An event-driven environment is yer typical online process -- the 
operator (or an external interrupt) drives the procedure; the program code 
is an "object" which reacts to the "event" that has just occured.  

With a procedural environment (batch) the code drives the process; with an 
event-driven environment (interactive) the code reacts to the event (a 
monitor follows the events, and loads the reacting "objects").

An example of how they differ: try to flowchart the logic involved in 
programming an elevator to respond to buttons pushed, floors reached, etc.  
With a "procedural" environment, you'll go around in circles until your 
head hurts.  With an "event-driven" environment, it's relatively simple.  
We're talking about a "shift in paradigm" as far as the comprehension is 
concerned.

COBOL (traditionally) is the king of procedural processes.  There is no 
language out there which is better suited for running batch code on a host 
(let's forget about PL/I... a great language that should have had better 
acceptance didn't). <This will prob'ly raise a bunch of rebuttals; start a 
new thread if you want to rebut>

Do not underestimate the importance of batch processing... it is *much* 
more efficient to stockpile the transactions and process them in one fell 
swoop... for example, there isn't a major credit card company in the world 
which does it's account posting online... they're all stored for the off-
hours batch update.  That's why your payment isn't reflected until tomorrow 
(with exceptions where the transactions are made available for online 
inquiry to make it look like they've already been processed).

COBOL is the only efficient language (remaining) on the big hosts.  

However, we do have a need for interactive processing.  Enter CICS...  a 
bunch of procedures that allow for screen input/output and testing what key 
the operator pushed.  Older than the hills; introduced long before personal 
computers were common...  Prob'ly designed by a bunch of batch guys who 
just wanted to be able to handle screen processing...

The bugaboo is that as soon as you get into interactive processing, you no 
longer want to be using a procedure-driven environment, you need to get 
into an event-driven environment.  CICS doesn't support the event-driven 
model; the programmer still has to code the logic to figure out what key 
the operator pushed and to control the sequence of logic depending on this.

Object-oriented programming works hand-in-glove with event-driven 
processes.  Let Win98 keep track of what key was pushed; simply execute the  
code that was put together to handle that situation.

When you code a program interacting with CICS (whether it be COBOL, PL/I, 
whatever), the majority of the logic of the program is figuring out what 
the hell button was just pushed, and which one was pushed before, yada yada 
yada.  The application programmer ends up having to re-invent the wheel -- 
the equivalent of the Win9x OpSys that simply loads the correct "object" to 
react to the "event"that has just happened.

If CICS were a truly event-driven model, the online mainframe programming 
environment would be a lot different.

Interactive PC-based applications depend on the model driving them -- say 
Win98.  There are a bunch of events that are available (e.g. "button 
pushed").  When you code the logic to handle that little event (an 
"object") it doesn't matter a lot whether you're talking about Visual 
BASIC, Fujitsu PowerCobol or even MS Access... you just have to identify 
what has to be done.  Used them all, you "just" have to pass back the 
correct values in the correct variables.  A "move" statement in BASIC does 
the same as one in COBOL, it just looks a bit different.

So, to summarize (it's about bleedin' time, you prob'ly say!)...

* The most effective use of COBOL will be for batch processing.  There is, 
simply, no other mainframe language better.  You will *NEVER* see a 25,000 
line COBOL program translated into C++.  Nor would you ever try to write 
(or maintain!) such a program in any other language.  PC-weenies, don't 
laugh... there are a lot of programs out there with > 10K lines of source 
code.

* For green-screen online mainframe processing, the only game in town is 
CICS or IMS/DC.  This is not an event-driven (i.e. Object-Oriented) model 
and the application programmer will still have to fudge the event-driven 
model using a procedural language.  Don't bet that IBM (or any other 
mainframe company) will ever offer a replacement.

* OO-COBOL is a nice idea.  However, until there is an event-driven 
processor available to call the objects (i.e. a CICS replacement) we're 
looking at something which doesn't have a lot of merit in the real world.

Just my $0.02Cdn worth...

Lots of good stuff Gerry but there is one fundamental flaw in your argument. OO
is NOT SOLELY about GUIs and events. Both Simula and Smalltalk designers were
interested in trying to tackle the huge problem of programming in a systematic
way, to overcome application complexity, and logic errors causing programs to
go belly up, and at the same time (1970's) face the problem of restricted
memory.

The concept of objects was introduced together with small granular procedures
(methods) which could constantly be re-used. Coupled with this was the concept
of inheriting from super (superior) classes for 'common' methods. Objects are
NOT just GUI images, but can also be things that we don't display, such as file
systems, or a program performing calculations for interest charges.

It would be perfectly viable to produce an OO application without screens !

This was the path followed by the Xerox PARC team. At the same time they
recognized that having 'pictures' made the whole thing more comprehensible to
end users, rather than playing it 'blind' as happened on mainframes with a set
of JCL cards - and hope the operators used the correct input, (tapes or disks),
and that you got the right output (dear old GIGO).

As regards the 'pictures', unbeknownst to her bosses, Adele Goldberg with a
felonious compatriot slipped some computers out of the front door of Xerox PARC
and tried them on some kids at a local school. The kids reacted favourably to
the pictures - Bingo ! The first GUIs were born. Stevie from Apple got in on
the act, giving us Lisa and leading to Billy G and Windows.

Regrettably much of the activity associated with OO is concentrated on GUIs or
components, but that is only one segment of the story. You don't need events to
trigger either the above file handling or interest calculations. (As an example
look at my recent message showing a class for a generic Line Sequential file -
doesn't it make sense to prove something once and just introduce instances of
it, regardless of how many different file formats you are using ?)

How best to summarize this ? Mine is one approach which could be one of many.
You don't need to be told that using copyfiles for repetitive routines or
calling sub-programs represents the sensible way go in non-OO. We all know that
there are things we find ourselves typing, over and over again. "Hey ! I
remember doing this before, but with just a slight twist to the code" - so can
we get some mileage out of it, (reusability) ? Yes if we can call that piece of
code with different parameters depending upon which program  is doing the
calling. Non-OO or OO - if we can establish that particular piece of code and
prove it, it can constantly be re-used without additional testing. Incorporate
the code into an existing Business Logic program, and as Howard points out, (his
bugaboo about maintenance), now you have to test each program affected.

Let's stick with COBOL files for the moment, (some of this can be 'slightly'
paralleled if you are using SQL). Non-OO I would say to myself, "Damn, I've got
to include copyfiles for FileX and when it comes to accessing the file, "What's
the syntax again for that particular VERB,and what did I call the different
elements of the PrimeKey ?". Similarly, if I now include perform PARAGRAPHS for
each of the file verbs they again become an integral part of  'this' program
which has to be tested for accuracy - and each time I have code or a paragraph
for a file verb - there's the possibility of either syntax or logic errors,
demanding each entry be tested thoroughly.

So taking just an ISAM file - the verbs are :-

        Open Input
        Open Output
        Open I-O
        Start
        Read using PrimeKey
        Read Next
        Read Previous (?) - I think that one is non-standard
        Delete
        Write
        Rewrite
        Close

(I think I've got them all <G>).

So whether or not I'm talking Customer File or Product File the above are common
routines to both. So now I could, (a) non-OO set up a separate called program
per file or, (b) with OO a separate File Class. The problem with non-OO is
keeping a handle on the file-status in a called program - not so, with OO. I've
got to test access to the Customer and Product Files as each has a separate
program for access. At this stage both file/class programs are unique and can
contain the specific copyfile information.

Each time a large installation goes this route, thorough testing is required for
EACH  new file format introduced.

Now apart from Customer and Product files holding separate contents, the means
of accessing them is identical. Can we get additional mileage by making the
access generic. In OO - Yes. Do you recall my trying to illustrate INVOKE 'NEW'
?

My solution (and by no means the final word), is to create a class GenericISAM.
This contains methods, (liken it to paragraphs or the eventual User Defined
Functions) for each of the file verbs above, with each method having its
appropriate parameters and results. How do we handle Customer and Product
records - the first may be 150 characters and the second only 56. One Solution
:-

FD  Data-File
    record varying from 1 to 1500 characters
       depending on ws-RecordSize.
01  Data-record.
      Data-PrimeKey               pic x(20).   *> Assumes you can get all
PrimeKeys
                                                                *> in first 20
characters
      Data-RestOfRecord         pic x(1480).

The 1500 would be an arbitrary figure picked by an installation - even if it
subsequently had to be upped to 1700, the only program requiring change would be
this ISAMGeneric. Using standard COBOL syntax we control the 'true' size by
ws-RecordSize.

Back to the "NEW" bit. We have a Business class program that needs access to
both Customers and Products - so :-

    invoke ISAMGeneric "new" returning os-CustomerFile
    invoke ISAMGeneric "new" returning os-ProductFile

This is done in the FACTORY (or construction area) of ISAMGeneric.
We have handles to the two files (object references) from os-Customer and
os-Products. We "called" ISAMGeneric twice, but only ONE copy is in memory with
TWO INSTANCES. Part of the initialization is to pass the full filepathname plus
the particular record-size.

In the OBJECT SECTION I have the following information, a separate 'copy' for
each of the above :-

File-status                    ??                                ??
Filename                        c:\customers.dat        c:\products.dat
ws-RecordSize                150                        56

The remainder of the OBJECT SECTION contains methods to use the individual file
verbs. So now I can invoke from the Business program :-

    invoke os-CustomerFile "readPrimeKey" using ws-PrimeKey
                returning ws-CustomerInfo

    invoke os-ProductFile "readNext" returning ws-ProductInfo

Both the above invokes 'hit' the one program in memory - ISAMGeneric.
Off hand I've no idea what the maximum instances of one class can be, (but you
can bet there is a limitation, probably more in terms of memory available ).
Nevertheless I could create 50 or 100 instances, say, each giving back a handle
like Customer and Products. (God forbid, if somebody wants 100 files in one
program, I think I'll snooze into my dotage).

ISAMGeneric is written ONCE and proven ONCE - and can be used for umpteen
different sized ISAM files. (The objective is to put Howard and the rest of the
Maintenance team out of work. Now I *know" why he's such a hard sell on OO <G>)

What I've been describing above looks like this :-

    [CustomerOrders]----------->[Customers - ISAMGeneric(1)]
    [CustomerOrders]----------->[Products - ISAMGeneric(2)]

and what I was originally trying to clarify in my correction was :-

    [CustomerOrders]----->[CustomerDBI]-------->[ISAMGeneric (1)]

The concept of the above ISAMGeneric doesn't work (well) with IBM 
compilers... completely impractical because of the underlining differences 
between the physical file storage methods and access methods.

1. There is a difference in the manner in which variable- and fixed-length 
files are physically stored.  For a variable-blocked file (VB) there is a 
pointer at the beginning of the physical block which identifies the length 
of the entire block.  Immediately following, there is a pointer which 
identifies the length of first record.  These "record length" pointers are 
repeated for each record within the physical block.  The record-access 
routines provided by IBM resolve these automatically -- when you define the 
file as being a VB file (which is (normally) captured from the RECORDING 
MODE clause on the FD which created the file).  If you try to access a 
fixed-blocked (FB) file using an FD referring to a VB file (or vice-versa a 
VB file using an FD referring to an FB file) you will end up with a system-
level abend or failed FILE-STATUS.  No way around this, except as defining 
*all* files (including fixed-length ones) as variable-length.  This adds 
processing overhead which would be unacceptable in the real world.

Going a step further, VSAM is an order of magitude more "advanced" than the 
QSAM VB/FB example above.  Not even close to how VB and FB files are stored 
or the IBM-supplied access methods.

2.  Accessing a file requires a SELECT clause, an FD (File Definition) 
clause, and the various I/O routines such as OPEN INPUT etc.  In the IBM 
world, the linking of a logical file within a program and the physical file 
itself is done thru the SELECT clause

 SELECT YADDA-FILE ASSIGN TO 'YADDADD'

where the YADADD is required in the JCL:

//YADDADD DD DSN=YADDA.WHATEVER,other parameters.

The YADDA-FILE and YADDADD portion is not a variable.  If you wanted to 
make this invocation a generic (OO) routine, you would need to code a 
SELECT statement for every potential file.

This would add some additional issues... if you wanted to use YADDA as both 
an input and output files (e.g. you are reading it, updating, and writing 
as a new version) you would require two separate versions... one pointing 
to the existing (input) version and one pointing to the new (output) 
version.  These would again require separate SELECT clauses, as well as 
separate FD definitions (each with unique internal and external filenames).  
Again, the OO implelentation of COBOL does not allow these to be changed as 
parameters, they would have to be uniquely defined.

To summarize: If you have 500 files in a system (having both input and 
output versions) you would have to specify 1000 separate SELECT and FD 
entries, and include run-time logic to choose which one.  Imagine the 
overhead (and don't think that 500 unique files for an application is 
large, because it isn't).

The suggested ISAMGeneric function would have to incorporate every one of 
these 1000 combinations, as well as logic to choose which pair of these 
1000 combinations to use.  I (personally) wouldn't allow this level of 
overhead in a production environment.

3. The OO approach has a lot of merit.  What a lot of advanced packages 
contain are CALLed routines for each file.  As an example, suppose a file 
is called "AMNA" -- in this case, the name of a file in a commonly-used 
card processing system which contains Name&Address info for the card-
holder.  There would be a routine called AMNAAB (used to access the AMNA 
file in an "access batch" environment).  This routine would contain 
numerous SELECT clauses, the FD statement, and all of the I/O access 
methods previously identified... OPEN INPUT... thru READ whatever thru 
CLOSE).  When an application program CALLs the routine, it specifies the 
access mode to be used (OPEN INPUT, OUTPUT, I/O or DYNAMIC) as well as the 
operation desired (READ NEXT etc.).  This access method routine would be 
coded once, and called by any programming requiring access to that file.

Based on the access-method parameter specified (I/P, I/O, O/P or DYNamic) 
the routine chooses the FD at initiation time.  The choice is stored, and 
subsequent operations are validated against it (i.e. you can't do a WRITE 
against a file you opened as I/P).  For an I/P file, AMNA could reference a 
DD statement called AMNAIP, for the O/P file a separate DD statment called 
AMNAOP, for I/O one called AMNAIO and so on. Each would point to a separate 
physical dataset.

Once the first CALLed access method is put together and debugged, it is 
then cloned for subsequent files.  The only thing that has to be changed on 
the new object is the filename (e.g. change AMNAAB to (for example) AMBSAB 
all the way thru the routine -- done with a generic "change" command in the 
editor), the record length in the FD portion, and the definition of the key 
portion in the I/O for direct-access operations.  This typically takes less 
than a half-hour and is considered to be trivial overhead.

This headspace has been around for decades, long before objects were 
conceived.

1. The most typical change would be (for example) interest rate tables, or 
perhaps parameters.  Key word here is "table" -- this is a copycode change.  
Library management packages (e.g. ENDEVOR, CHANGEMAN) know which programs

Typical existing code could be:

    	WHEN AEROGOLD-VISA

routine to make sure the intermediate results are what are expected.

CICS and LE are not in the least related.  CICS is an IBM-supplied ($$$$) 
routine to handle screen & transaction & event processing.  LE is a "new" 
run-time library of routines (I/O, date-handling, etc.).  Used by CICS, 
batch, common routines used by virtually all languages supported by IBM.

CICS is a teleprocessing monitor.  When you are referencing files, the 
files are assigned to the actual CICS region the application is running in, 
not to the actual application program.  The CICS region looks like a batch 
job, when executed it "brings up a CICS region".  The batch job which 
brings up the region has a JCL "DD" statement allocating physical files to 
it.  Once the region is running, no other job can update the physical file 
(depending on options, some can read them).  For that reason, you cannot 
have a CICS application program that directly accesses the physical file 
(because you could/would have simultaneous updates of a record).  Instead, 
the file accessing of a file assigned to this online region cannot use 
existing COBOL syntax commands.  Instead, you issue a CALL to CICS 
requesting the access, and CICS himself resolves the file locking etc. 
needed to prevent simultaneous updating, etc.

That's why CICS file access doesn't look like COBOL.

Not certain what you mean by the above, "fixed attributes can't be
changed"; but the VSAM controls blocks can be tailored to whatever the
use wants dynamically with the GENCB, SHOWCB & MODCB macros. The
following GENCB generates an RPL with the parameters determined
dynamically:

*---------------------------------------------------------------------*
*        Time to generate the RPL for this file.                      *
*---------------------------------------------------------------------*
	GENCB	BLK=RPL,                                               x
		AM=VSAM,                                               X
		ACB=(ACBREG),                                          X
		LOC=ANY,                                               X
		AREA=(AREAREG),                                        X
		ARG=KEY,                                               X
		AREALEN=(LENREG),                                      X
		RECLEN=(LENREG)         

	STM	R15,R0,SAVEAREA         * SAVE FOR DEBUGGING
	LTR	R15,R15                 * GENCB work?
	BNZ	GENERR                  * NO


This will generate an ACB for a DDNAME supplied by the caller:

	GENCB	BLK=ACB,                * generate ACB                  X
		AM=VSAM,                *                               X
		EXLST=EXLST,            *                               X
		MAREA=MAREA,            *                               X
		MLEN=L'MAREA,           *                               X
		LOC=BELOW,              * keep ACB's under 16-Meg line  X
		RMODE31=ALL,            * gen control blocks & buffers  X
		DDNAME=(*,FCBEXTRN)     * above 16-Meg line

	LTR	R15,R15                 * GENCB work?
	BNZ	GENERR                  * nope


The following MODCB primes the RPL for a Get with Update:

GETU	DS	0H                        
	MODCB	RPL=(RPLREG),           *                               X
		AREA=(AREAREG),         *                               X
		OPTCD=(KEY,DIR,ARD,UPD,KEQ)

	LTR	R15,R15                 
	BNZ	GENERR                  


DCB's are trivial. The DCBD macro maps the DCB and the correct values
can be moved to it.

While it was not my intention to "write off" CICS in my post (I agree it
still has a useful life, and may yet transform itself successfully to the
Client/Server environment as a competitor for MTS), my post was intended to
show that CICS, in and of itself, will not keep COBOL alive. The use of the
Web is obviating the need for transaction processors altogether. Transaction
processing is implicit in the use of Web Forms, whether they are processed
by CGI code, Java Servlets, or ISAPI on the Server. It comes down to an
argument about server performance, not architecture.

Having just arrived back in the U.K. and looking at opportunities (euphemism
for "looking for a job" <G>), I did a search yesterday of a major European
IT job-finding site on the Web. Out of 4600 jobs posted, there were 8 for
COBOL. Of those 8, 2 were contract.

10 years ago, the same search on the same site would have revealed around
14,000 jobs of which at least 2000 would have been COBOL. Around 800 of
these would have been contract. These days I don't work as a COBOL
programmer and it is most likely that I'll get employed through the network
of previous contacts I have established, but I ran a COBOL filter just for
old times sake...

We have discussed at length the death of COBOL in this group; now we are
seeing the reality.

In Europe, my assessment is as follows:

COBOL is fast becoming a mainframe Batch programming language which will be
used by permanent employees who can be kept to an average salary of around
�25000. (Reporting this gives me no pleasure at all and I won't say "I told
you so" even though I did ...<G>)

The more astute COBOL people have already moved to other skills (I have been
contacted socially since my return by several who I either worked with or
trained in the past, and none of them are using COBOL as their primary means
to make a living any more.) I am also in the same boat, not having made a
living from COBOL for over 10 years now.

So, why is it?

Well, my preliminary analysis does NOT reveal that COBOL has fallen victim
to the whims of fashion, as most of us might expect.

The fact is that the MARKETPLACE (which is always the ultimate driver of
technology) has changed.

More and more Corporations are questioning why they should even attempt to
maintain expensive IT development in house. More and more IT departments are
limiting their activities to the rollout of desktop technology, the
selection of packages and third party suppliers (outsourcing, if you like),
and the Web Enablement of existing systems (read "Converting to Java").
Development in the sense of "computer programming" is becoming non-existent.
Quick build tools like VB are still in use although even this is on the
retreat.

The modern Corporation is saying: "We are NOT an IT Company. We can buy the
service just like we buy any other service."

 In the past this wasn't possible. There wasn't the plethora of "Business
Solutions" which are available today (SAP, Siebel, etc.), they KNEW they had
to have IT technology to remain competitive, and the only way to ensure it
was to do it in-house.

The advent of Client/server brought component based solutions (databases
like ORACLE, INGRES, SQL Server, Server-based shared re-usable services that
can be instantly deployed wherever needed), and, of course, the promotion of
the hardware-based LAN into a Corporate IntraNet with portals to the
Internet (Global connectivity and distribution of Corporate systems).

The main reason for running a mainframe was the power of it. Now the same
throughput can be achieved with hardware at a third of the price and without
the need for an expensive technocracy of system programmers and support
infrastructure. (There is still a need for infrastructure, but it is much
less expensive.) The main reason for running a mainframe NOW, is that most
of the Company's systems are in the form of Legacy code, and HAVE to be
mainframe based! As time goes by these systems will be gradually replaced by
more easily distributed and deployed Client/Server based solutions.

There is a strong market for Datawarehousing and Legacy-leveraging tools at
the moment. However, I predict this is a limited life market (say 10 years)
which contains the seeds of its own destruction (once you have "converted"
there is no "repeat business"...).

So, for many COBOL programmers, coping with the daily problems of file
handling, return codes, printing, and all the things which make "computer
programming" the fascinating art that it is, it is easy to lose sight of the
"big picture" and see what is shaping the future. By the time we become
aware that jobs are evaporating, it is too late.

Once again, I would strongly urge ALL COBOL people to invest some time in
acquiring new skills. Right now Java is good (it also introduces exposure to
Object Orientation and, if you are determined to make a career as a
"programmer", you WILL need this knowledge) and so is some Web
familiarisation (any COBOL programmer worth their salt can teach themselves
HTML in an afternoon...it is just Word Processing).

DON'T Rely on COBOL being enough to feed and clothe you and yours in the
years to come. If you are an experienced Programmer/Analyst try and get on a
SAP course. (Whatever we may think of SAP from a technical point of view, it
is enjoying a rising wave of popularity and "Management" (who are nowhere
near as concerned with technicalities as we are) love it. Besides, as it
succeeds, many of the worst technical flaws in it will get resolved...)

EVERYBODY needs to learn RDB (SQL) and move away from the COBOL file system.
(This factor is one of the most potent in the decline of COBOL - it isolates
the language and applications developed in it.)

All of the above is one man's opinion. Many of you will disagree.

Time will tell.
