Anybody got any useful techniques for doing this.

We have an output file written by program A, which is read in by program
B. Program A defines the file as:
FD SSDEBCDC DATA RECORD IS SSDEBCDC-RECORD
LABEL RECORDS ARE STANDARD
RECORD CONTAINS 1 TO 176 CHARACTERS
RECORDING MODE IS V
BLOCK CONTAINS 0 RECORDS.

01 O-PKG-RECORD.
03 O-PKG-CONTENT-CHARACTERS PIC X OCCURS 176
DEPENDING ON WS-DISPLAY-LENGTH.


Obviously, WS-DISPLAY-LENGTH is calculated at runtime, and results in
records of (unsurprisingly) varying length being written out.

Note that none of the data content of O-PKG-RECORD is anything to do
with the length of the data record itself (although that would be
nice). for the purposes of this example, assume that record#1 contains
'ENGLAND' and record#2 contains 'USA'

So....MVS stores record#1 as:
ENGLAND
0000CDCDCDC
0B005573154

..and record#2 as
USA
0000EEC
0700421

in both cases the first two bytes indicate the length of the entire
records, including the 4-byte record descriptor area.

Currently, to access the length information when reading these records
as input into program B, we define the input as:
01 I-PKG-RECORD.
03 WS-CONTROL-BYTES PIC XX OCCURS 2.
03 FILLER PIC X(172).

Then, we code reference to WS-CONTROL-BYTES(-1) (yes, that's negative
one), enabling us to address the control bytes of the record descriptor
area which contain the record length.

this works fine, but is very inelegant. Also, we are required to code
the compiler option NOSSRANGE to prevent the compiler detecting a
'subscript out of range' error. It *would* be far more elegant to code
an attribute of the data record as a record-length value (e.g.
07ENGLAND) so that is directly available to the program, but we are
prevented from doing that by other downstream processes

So, given the datarecord layout which we use, has any one got any better
ideas ???
Other things we have tried are:-
1- LENGTH OF I-PKG-RECORD, always returns 176
2- Making the record O.D.O, doesn't work because you cannot know what
length to calculate the DEPENDING ON value to

If you are using VS COBOL II or COBOL for MVS and VM (any IBM ANSI-85
compliant COBOL compiler), have you tried:

FD SSDEBCDC
RECORD IS VARYING IN SIZE FROM 1 TO 176 CHARACTERS
DEPENDING ON WS-SSDEBCDC-LRECL
BLOCK CONTAINS 0 RECORDS.

01 O-PKG-RECORD PIC X(176).

You would have to define WS-SSDEBCDC-LRECL in working-storage as
unsigned numeric integer (display, binary, or packed), and the actual
length is returned upon read.

You can also use this ANSI-85 feature to write variable length records
by setting the record length in WS-SSDEBCDC-LRECL before issuing the
WRITE command.

In MVS your JCL will have to specify the LRECL as 4 bytes greater than
the maximum record length seen inside the COBOL program.

I've used this very successfully for both reading and writing variable
length records.
