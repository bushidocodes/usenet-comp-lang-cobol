If you literally want to pass the *contents* of a file, you have to
load it into memory, such as in WORKING-STORAGE, and then pass the
place where you stored it.

However, I don't think that's what you want to do.  I think you want
to open a file in one module, and read it (or write to it) in another.

One approach is to declare the file as EXTERNAL.  Then you can access
it from any module which declares it similarly.

I prefer to confine the file to a single access module, and provide 
multiple entry points (or a single entry point with multiple action 
codes) to perform various operations.  This approach provides better
encapsulation -- any changes to the file structure, or any special
processing which must be applied to it, are confined to a single
source file.

Both approaches are limited in that you can't switch files from one
call to the next.  You can't, for example:

	CALL "PRINTHDR" USING FILE-A

...in one paragraph and 

	CALL "PRINTHDR" USING FILE-B

in another, where A and B are any arbitrary files.  For this kind
of flexibility you would have to write your access module in some
other language such as C or Assembler, unless your flavor of COBOL
provides some kind of vendor extension (which is what BY DESCRIPTOR
looks like).
