```
As others have pointed out, limits on table sizes in either the FILE SECTION
or WORKING-STORAGE SECTION are a function of the compiler, not the COBOL
language per se.

Depending on your OS and your compiler, there are 'tricks' you can use to
simulate tables larger than the maximum permitted by the compiler; or system
services which allow you to allocate and access more memory.

You say you are using AccuCobol as a compiler; in what environment are you
running?

And, is this table part of an FD in the FILE SECTION? If so, you may be able
to the file differently - that is, in smaller pieces - and avoid compiler
limitations that way.
```

```
Large arrays are often _not_ required in Cobol systems.  Many systems
require data to be shared with other users.  Locking data up in an
internal array does not allow it to be shared.  Consider an ISAM file or
adding an alternate key to an existing file to give access to the data
items that you need while allowing others to access the same data.

I think that this says more about the limitations of your experience
than it does about any limitations in actual Cobol.

Cobol 74 was limited to 3 dimensions but had no restriction on any
sizing.  Implementations may have limited the number of _occurances_ to
32768, but a size limit of 32k bytes was certainly not in 'most'.
```

```
The limits for various parts of COBOL *all* depend on the compiler and
operating system.

IBM's OS/VS COBOL compiler *did* have a 32k limit for tables.  However, that
hasn't been their strategic compiler for a couple of decades now.  There
are, however, a number of compilers that have "options" to emulate it.

One problem that people often (occasionally?) have is to try and use a
"table" where they should be use a file (with separate records) - often an
indexed file.  Or even a SQL database.

In general, when posting to CLC, it helps if you mention the compiler,
release/version, and operating system.  Also, if you are having a
compile-time problem, it helps to show the source code and the actual
message from the compiler.  There MIGHT be a way around your current problem
(without going to another compiler) - but with the information posted so
far, I don't know what it would be.
```
