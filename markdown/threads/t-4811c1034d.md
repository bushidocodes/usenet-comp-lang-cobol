(Original post quoted in toto infra).

Many database products require some kind of reorg to reclaim space occupied by
'deleted' records.  (Sometimes copying a file does this, sometimes a reorg
utility is needed).  Dont' feel bad about the product you are using. The
situation is very common.

The motivation is that just posting a record as logically deleted is fast,
typically requiring just a single byte or even a single bit flip. 

Physically removing a record in a complex file access methodology could require
fixups to the previous record's forward pointer, and fixups to the following
record's backwards pointers. Also index pointers to the specific record in
question would need to be sponged up too - THAT could require multiple index
record fixups if the record in question was the lowest or highest record in a
group.

In short, physical deletes can create serious performance bogs. Generally, a
dbs intended for 'online' use (and most PC work is actually 'online') will
avoid physical delete, and just go with logical delete, tricking any subsequent
access by making the deleted record invisible.

File reorganization is usually done offline at a time when performance is less
critical. The culture that is out there is to do reorg at roughly the same time
as you do file backups, so 'copying' the file for reorg purposes is a trivial
additional step.

Some access methods do dynamic space retrieval; but this introduces performance
issues as outlined above, and introduces reliability issues, and makes forward
recovery a much more dubious adventure.

Recoveing the valuable disk space is important. More frequent reorgs are the
usual prescription. But like anything else in this biz it is a trade-off. There
is certainly a subset of applications that involve very frequent record
deletion, in which cases a less typical access method strategy involving
dynamic space recovery could be indicated.

Some extremely advanced access methods organize your single db intro numerous
physical files, any one of which can be varied offline and reorganized or
recovered independently (even while the rest of the db stays online).  But you
need a megabucks application to justfy the extra cost of dealing with data that
way.

Just copying the file will not remove the "Deleted" records.  To
actually remove them, write a program that:

   Writes the good records to a temp file
   Open the production file as output
   Put back into the production file the records in the temp file.

The accounting software I work with comes with pr-written programs
that do just this.  They provide an ascii export for each file, and
also a corrosponding import. Thus, to reorg, we export the file, then
re-import it. 

Hope this helps,

Let me add one point to Tom's excellent post:

Not only does standard COBOL support *real* databases, it supports real
*relational* databases.

Let's not forget that the term "relational database" refers to one where
each relationship in a set of data is expressed in a simple indexed
table. Being relational doesn't require either SQL or a client-server
engine. The major requirement is to follow Codd's "normal forms" when
designing your data files.

A relational table can be fully implemented as a COBOL
indexed-sequential file. A relational database can be fully implemented
as a set of such files. Moreover, in most cases, such an implementation
will be far more efficient than one using a SQL-based database engine.
(It's also cheaper and usually easier to deliver to an end-user.) 

I will grant you that a SQL engine may bring additional flexibility
(e.g. the ability to do on the fly SQL queries) and data compatibility. 

I will further grant you that when a full-blown client-server system is
required, a more elaborate calculus applies which *may* argue for an
off-the-shelf engine. (Or, more likely, the ODBC approach that Liant's
Relativity uses.)

--John
