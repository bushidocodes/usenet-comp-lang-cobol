I used TELON for many years in an IBM mainframe environment.
The expression that most describes TELON is the one used by Pansophic,
the TELON Development Facility, TDF for short.
It's purpose is to provide a very rigidly structured,
fill-in-the-blanks-in-the-proper-place-in-the-prescribed-manner-or-else environment,
for the development of online and batch COBOL programs, IMS, CICS or even conventional.
You import your DBDs (database definitions) into the facility up front,
to allow developers to rapidly integrate each required database and all or selected database
segment definitions into the program with a few keystrokes in the development phase.
Developers can checkout an existing module from a production program library such as Panvalet,
import it into the TDF, make the required changes, and compile it straight from the TDF.
When tested, the module can then be exported from the TDF and staged for a production move.

Issues include
* Delegation of responsibility for maintenance of the DBDs in the TDF.
   The DBAs may or may not consider it their duty to update the TDF
   whenever a database definition change occurs.
* Multiplicity.  It may be necessary to create multiple TDFs, for any number of reasons.
   Security, for instance, to make it possible to lock anyone
   but Payroll devlopment group personnel out of the PAY TDF,
   or to restrict the MKT TDF to Sales & Marketing.
* Concurrency.  It may be necessary to permit concurrent development projects
   which require that Group A have stable access to SubRoutine X Version 6,
   while Group B has unfettered access to SubRoutine X to make whatever changes they please.
   There are a number of ways this can be addressed, usually by allocating specific TDFs
   and separate development test environments to each group.

Here's a very simple TELON module to give you an idea of its content and structure,
although the cut-and-paste didn't do the formatting any favors - sorry:

./ ADD NAME=SCRNDEF
***********************************************************************
* CUS063.BD EXPORTED ON: 02/21/95 20:07:18 BY: PRGHNC *
***********************************************************************
TELON LANGLVL=2.0,LANG=COBOL,HEADER=CUS,ID=063
***********************************************************************
* BATCH DESCRIPTION ~~ SCAN UMSCRD1 FOR UMSCRD1 CLEANUP *
***********************************************************************
      BATCH 063,HEADER=CUS,SIZE=(058,133), C
          DESC='SCAN UMSCRD1 FOR UMSCRD1 CLEANUP',WKAREA=WKAREA, C
          LANG=COBOL,PRCTRAN=PRCTRAN,TERM=TERM
***********************************************************************
* D A T A A C C E S S *
***********************************************************************
      TPPCB NAME=ALT
*
DATASET NAME=INPUT,ACCESS=SEQUENTIAL,OPEN=INPUT,LRECL=00016
INPUT RECORD DEFINE,COPY=NONE,COPYLV1=Y,COPYLBL=INPUT-UMSCR-DATA, C
COBDIV=WS
AUTOEXEC TRANSACT INPUT
*
DATASET NAME=OUTPUT,ACCESS=SEQUENTIAL,OPEN=OUTPUT,LRECL=00018
OUTPUT RECORD DEFINE,COPY=NONE,COPYLV1=Y,COPYLBL=INPUT-UMSCR-DATA, C
COBDIV=WS
CREATE OUTPUT
BATCHPGM DLIWGHT=Y,TRACE=N,GENPCBS=Y,LNKCOPY=LNKCOPY, C
USGCOPY=USGCOPY
DLIPSB
END
./ ADD NAME=LNKCOPY

./ ADD NAME=PRCTRAN
*_______________________________________________________________
* PURPOSE: ENSURE THAT A TRANSFER IS TO A VALID TRANSACTION_
*_______________________________________________________________

CALL 'CBLTDLI' USING CHNG-FUNC
ALT-PCB
INPUT-HDR-KEY-TRAN.
IF ALT-PCB-STATUS-CODE = ' '
NEXT SENTENCE
ELSE
ADD 1 TO WS-NUM-UMSCRHDR-ROTE
MOVE
ALT-PCB-STATUS-CODE TO INPUT-HDR-KEY-STAT
PERFORM U-100-CREATE-OUTPUT.

./ ADD NAME=REMARKS

****************************************************************
* READ THE INPUT FILE *
* *
* ISSUE A CHANGE CALL *
* *
* WRITE THE RECORD IF ALT-PCB STATUS = A1 *
****************************************************************

./ ADD NAME=TERM

DISPLAY ' '.
EXHIBIT NAMED BWA-TRANSACTION-COUNT.
EXHIBIT NAMED WS-NUM-UMSCRHDR-ROTE.
DISPLAY ' '.
DISPLAY '*************************************************'.
DISPLAY '************* END OF PROGRAM EXIT **************'.
DISPLAY '*************************************************'.

./ ADD NAME=USGCOPY

****************************************************************
* THIS IS THE PROCEDURE DIVISION USING *
****************************************************************

ALT-PCB.

./ ADD NAME=WKAREA

01 WS-NUM-UMSCRHDR-ROTE PIC 9(09) VALUE 0.

01 INPUT-UMSCR-DATA.
05 INPUT-HDR-KEY-USER PIC X(08).
05 INPUT-HDR-KEY-TRAN PIC X(08).
05 INPUT-HDR-KEY-STAT PIC X(02).
-------------------------------------------------------------------

Thsi statement AUTOEXEC TRANSACT INPUT is expanded by TELON
into the full-blown dataset access logic required to read the INPUT dataset.

You can define IMS databases, segments and call statements in similar shorthand notation,
which the TELON code generator will expand into standard COBOL for input
to your regular COBOL compiler.

Checkpoint/restart field definitions are handled in similar fashion.  For example:
CHKP     READ IOPCB,TPPARMS=(CX-LONGEST-IOA-LEN,CX-CHKP-CALL-ID,
               BWA-AREA-LTH,BATCH-WORK-AREA,CX-SAVAREA-LEN2,                       C
               CX-SAVAREA2,CX-SAVAREA-LEN3,CX-SAVAREA3,CX-SAVAREA-LEN4,C
               CX-SAVAREA4),FUNC=CHKP
XRST     READ IOPCB,TPPARMS=(CX-LONGEST-IOA-LEN,CX-XRST-CALL-ID,       C
               BWA-AREA-LTH,BATCH-WORK-AREA,CX-SAVAREA-LEN2,                       C
               CX-SAVAREA2,CX-SAVAREA-LEN3,CX-SAVAREA3,CX-SAVAREA-LEN4,C
               CX-SAVAREA4),FUNC=XRST

There's no way I can adequately cover the product in a short(ish) post,
but I trust that you now have a basic glimmer as to what TELON is.

to be executed at various predefined points on the output processing side.

Ah, one of my favourite topics.

The SPA area can be much larger than 4K.  It depends on how you define the
underlying data base.

IMO, the best way to organize the telon work spa is to define a copycode
that resembles:

   03 XFER-SPA-AREA                                    PIC X(4000).
   03 XFER-FILLER    REDEFINES XFER-SPA-AREA
       05 XFER-COMMON-AREA                     PIC X(2000).
       05 FILLER REDEFINES XFER-COMMON-AREA.
           10 XFER-CURR-DATE                            PIC X(10).
            ....

        05 XFER-PROGRAM-AREA                      PIC X(2000)..

This way, if you add too many fields to XFER-FILLER, the compiler will tell
you that XFER-FILLER is larger than XFER-SPA-AREA.

Each screen should redefine XFER-PROGRAM-AREA to suit its own needs.  The
fields in XFER-PROGRAM-AREA should *only* be used for the screen to talk to
itself, and should *never* be used to communicate between screens.  That is
what XFER-COMMON-AREA is for.  Adhere to this principle, and you'll have
fewer maintenance problems down the road.
