Starting a new thread (and hopefully, I have read all the notes on the
existing one),  I wanted to talk (a little) about designs, trade-offs, and
working in and with IBM "architecture".

Rule 1 (IMHO - and I won't add this to all future "rules") - there are NO
rules that apply to every application, program, operating system, etc.  Any
"design decision" (or "shop/programming standard") *must* be subject to
modification based on specific application "needs".

Rule 2 - Among other things, what makes COBOL, COBOL - is the fact that it
is "highly" self-documenting.  Programmers who write code that is
intentionally (or even unintentionally) obscure, deserve what they get.
HOWEVER, what is "self-documenting" really DOES depend on who will be doing
maintenance on a program (short, medium, AND long-term).  If you are a "one
person shop" - then no matter how odd (to me) your "style is" - as long as
you are consistent, it PROBABLY will be self-documenting to you (even 5 or
10 years from now); just don't "die" or stop programming while you have
"active" users.  SIMILARLY, if your shop has 100's of programmers who ALL
use COBOL Report-Writer, then "any one" should be able to pick up a RW
program and maintain it - while if you are the only one in your shop who
knows/understands it, it will be a NIGHTMARE to anyone else who must do
"fixes" or maintenance. (Same for anything from complicated UNSTRINGs to OO
to 3rd party products)

Rule 3 - if run-time performance is your GOAL, then (almost?) everyone
agrees that the place to start, is the underlying programming logic.  As IBM
says - at:
  http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igy3pg00/7.1?

"Examine the underlying algorithms for your program. For top performance, a
sound algorithm is essential."

Rule 4 - It is extremely rare (but not impossible) that "tricky tuning" of
some part of your program that isn't executed "frequently" will result in
real-world performance enhancements.  On the other hand, a "simple change"
(even moving a data description from DISPLAY to Packed-Decimal before
entering a loop) in a "tight loop" of a routine that is performed millions
of times, MAY out-weigh any (potential?) maintenance/portability problems.
Testing should ALWAYS include "system tests" as well as unit tests (or
examining "generated code") to see if "performance enhancements" are WORTH
the "trouble" (current and future).

Rule 5 - "portability" of both source code and output data (files,
databases, whatever) is a WONDERFUL thing to have.  9 time out of 10, it
will make some FUTURE port or maintenance of an existing program much, MUCH
easier.  On the other hand, 1 time out of 10, making something "portable"
will cause sufficient "degradation" of existing requirements, that it should
be "avoided like the plague".  It does NOT mater how fast/powerful existing
processors are, if you need to finish a "nightly batch run" in 3 hours and
your "enhancement" makes it take 3 1/2 hours, something has to GIVE.
Similarly, if you are supporting multi-thousands of simultaneous online
users, changing from 1.8 to 2.2 second response time, may loose you (and
your boss) a job!

Rule 6 - "Isolating" compiler-specific, OS-specific, or site-specific
programming "stuff" may well solve trade-offs of performance vs
maintainability vs portability.  Certainly, moving user-interaction (in an
online environment) and data base (flat, relational, whatever) AWAY from
"business logic" is a great idea.  Use of COBOL CALL statements, OO
facilities, even COPY statements, can do this for you.  Unfortunately
(again - on the other hand), you might find that "highly modularized" or OO
applications do NOT meet performance requirements; only systems testing in
YOUR environment will answer this question. (If you DO find a problem, you
might want to contact your "vendor of choice" to see if they will ENHANCE
their product's performance in such environments)

Rule 7 - If you have a  common subroutine (used by lots and LOTS of
applications) or a file (database, whatever) that is common to MANY, many
programs - all on your "single" platform *and* if that routine is relatively
"stable" (as such files and routines often are), then the chances are that
it SHOULD BE optimized for the environment where it runs/is used.

Rule 8 - If you have a file (database, whatever) that is often (even
primarily) used for "communicating" information from  platform to platform;
or environment to environment, then the chances are that it should be
optimized for PORTABILITY.

   ***

So, back to the "subject" of this post.

If you work in an IBM mainframe environment (or - it should be added an
OS/400 environment), use of:
   Packed-Decimal
   VSAM
   EBCDIC
   even CICS (or IMS) is many (most?) cases

should and CAN be assumed
  - that all/most programmers know
  - that all other same-platform "tools" (programming languages, etc) will
understand
  - that both the compiler and all supporting "things" will be OPTIMIZED for
run-time performance

It can even (probably) be assumed that IF you use these "proprietary"
features *and* need to interface occasionally with other environments where
they are NOT supported, that there will be PLENTY of tools for making the
"transition" easy - and probably even pretty efficient.

This does not mean (IMHO - again) that you should AUTOMATICALLY use these
OS-specific "things" when creating a new application or a new file, but it
does mean that you are SAFE in doing so - AS LONG AS your *primary* target
is an IBM mainframe. What's more, if you are REQUIRED to interface with
existing programs or data, the chances are that this is the BEST way to
work.

If you think that your "new" (or even modified) application will be
interfacing with non-IBM environments (or even IBM workstation ones), then
you really SHOULD look seriously at:

  - Relational databases (for almost all data - except "report files" and/or
"logs")
  - USAGE DISPLAY / SIGN IS SEPARATE numeric fields (for "external" - not
internal data)
  - ASCII (or better yet Unicode/10646) character data
  - *ALL* OS specific "code" in isolated modules (subprograms and/or
methods/objects)
  - Compiling programs (at least once) with ANSI/FIPS flagging turned on so
you are aware of what implementor extensions your code does use

Hear, hear. Clarity, not cleverness. Or should I say, not cleverness at the expense
of clarity. I really shot myself in the foot doing this once. Way back when VS COBOL II
first arrived and you could suddenly have more than 3 dimensions in a table, I had this
big report program to write. Having a lot of time to just screw around I loaded all the data,
headings, detail, totals, subtotals and what not, into a humongous 6D table and then unloaded
it all back to print with a bunch of "perform varying, after, after,...." statements. It was the
most confusing thing but worked pretty good until a couple of years later when it to be
changed. The changes just would not quite fit into the tables. One of the occurs had to change
for the new data which messed up the logic for the existing data, which messed up the performs,
which messed up the printing etc etc. After a few more years and several go-rounds of this,
I finally had to spend a weekend rewriting it all - extract program, jcl sort, print program with
standard control breaks like it should have been in the first place. Learned a valuable lesson -
don't be cute, it'll bite you in the butt later. Be clear, be straightforward, don't be
unnecessarily
obscure and if something must be a bit unusual, use that "*" in column 7 and say what in the
heck you're doing. You might be the one that has to fix it some day.
