```

EVALUATE field_name
WHEN "value1" PERFORM PARA-A
WHEN "value2" PERFORM PARA-B
WHEN "value3" PERFORM PARA-C
WHEN "value4" PERFORM PARA-D
WHEN OTHER PERFORM PARA-OTHER
END-EVALUATE

The OTHER clause allows you to handle values that aren't in the WHEN list, it is optional. If no OTHER clause is coded, no operation takes place if the value in field_name isn't in the WHEN list.
```

› EVALUATE field_name
›   WHEN "value1" PERFORM PARA-A
…[4 more quoted lines elided]…
› END-EVALUATE
 
› The OTHER clause allows you to handle values that aren't in the WHEN list, it is optional. If no OTHER clause is coded, no operation takes place if the value in field_name isn't in the WHEN list.

You can also do this:

EVALUATE TRUE
WHEN condition-name-1 PERFORM PARA-A
WHEN condition-name-2 PERFORM PARA-B
WHEN condition-name-3 PERFORM PARA-C
WHEN conditional-statement PERFORM PARA-D
WHEN OTHER PERFORM PARA-OTHER
END-EVALUATE

Where condition-name-# is an 88-level conditional name. You can also
put entire conditional statements (ex: ws-letter-grade = "c") in the
WHEN clauses while using TRUE.

Or:

EVALUATE identifier-1 ALSO identifier-2
WHEN literal-1 ALSO literal-2 PERFORM PARA-A
WHEN literal-3 ALSO literal-4 PERFORM PARA-B
WHEN TRUE ALSO TRUE PERFORM PARA-C
WHEN TRUE ALSO FALSE PERFORM PARA-D
WHEN OTHER PERFORM PARA-OTHER
END-EVALUATE

The ALSO serves as an AND. Great for patterning a decision table
directly into code. When identifier-# is an expression and your
compiler supports FALSE, one can really make EVALUATE into a table.

Or even:

EVALUATE identifier-1
WHEN literal-1
literal-2 PERFORM PARA-A
WHEN literal-3
literal-4 THRU literal-5 PERFORM PARA-B
WHEN OTHER PERFORM PARA-OTHER
END-EVALUATE

literal-1 and literal-2 serves as an OR to identifier-1. literal-4
THRU literal-5 serves as ranging.

Well, you get the idea. One can get quite creative using EVALUATE.

So experiment.

EVALUATE something
WHEN something

WHEN something-else

WHEN OTHER
do other imperative statements
END-EVALUATE

I prefer it simply because it leaves more room on the line for longer names,
and because EVALUATE may be indented under an IF statement. I have seen a
lot of programmers put a blank line before each WHEN clause, but we don't
