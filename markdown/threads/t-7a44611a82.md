There are several approaches to such bridging. The correct solution
depends on the problem to be solved. See below...

›
› I have an adress for COBOL Access+ (www.rldt.fr)
› but I want more application in order to compare.

1. This product, available for RM/COBOL (its original version marketed
as Rm/plusDB) and also Micro Focus COBOL, essentially traps the indexed
I/O statements (i.e. READ, WRITE, REWRITE, DELETE, etc) for one or more
indexed files and translates these record-oriented statements into
database operations. For example, an indexed file READ statement might
be translated into one (or more) SELECT statements. This approach has
the benefit of using a relational database for storing data that
traditionally has been stored in VSAM, RMS, C-ISAM, or other indexed
file implementations, without changing the COBOL application in any way,
saving the cost of conversion. The drawback, however, is that some very
normal I/O sequences (from the COBOL standpoint) have extremely bad
performance on a relational database. Mr. Dambrain at RL-DT has done a
good job of minimizing these performance bottlenecks, but be aware that
they do exist.

2. Contrast the Access+ technique with a more traditional approach of
embedded SQL. Embedded SQL would preserve most of the structure of your
COBOL application, but the work of replacing indexed file I/O statements
with embedded SQL would be expensive, and would face some of the very
same performance issues of the Access+ approach.

3. A different approach, which allows the COBOL application to be
unmodified, is to continue to use the COBOL indexed files (VSAM, RMS,
C-ISAM, etc) for the COBOL application and use a product such as
Relational DataBridge to allow ODBC SQL access to the data, enabling the
use of ODBC-enabled tools such as Crystal Reports; Power Builder; MS
Visual Basic, Access, Excel, and Word; Cognos Impromptu; or many, many
others.

This approach is typically more cost-effective due to: a) the cost
difference between indexed file systems vs. relational databases, and b)
the fact that absolutely no modifications are necessary to the COBOL
application. See
http://www.liant.com/products/retooling/databridge/index.html for more
information.

Good point Tom.

Essentially in my experience there are three reasons why people want to make
use of a database under their Cobol applications.

The first reason would be to improve the accessability their data by using
Windows based report writers and spreadsheets in conjunction with their
application. Your relational databridge product obviously solves this need
very easily.

Unfortunately it ignores the other reasons that users want this kind of
facility.

1. - Data security (Commit and rollback transactions).

2. - Sharing information real-time between two systems. I have a user with a
Cobol based hand-written system for their account and financials and an
INFORMIX based package for their despatch system. What they want to do is
move some of their client details into INFORMIX so that they can share this
data between the two and when information gets changed it updates both
systems.

Relational Databridge won't solve either of these needs.

As usual there are many ways to skin a cat, and we have to decide which one
fits best and solves the most needs.

Mike.

customer wish list yet...far from it.

We DO offer security in terms of grank/revoke style security on the

› 
› 2. - Sharing information real-time between two systems. I have a user with a
…[6 more quoted lines elided]…
› Relational Databridge won't solve either of these needs.

Not quite true. Relational DataBridge supports UPDATE and INSERT, so it
clearly is possible to update both systems. How you do this is a
problem of integration between the systems. Do you write a layer, say
in VB or MS-Access, which updates both the INFORMIX and Relational
DataBridge tables? Or perhaps you modify the INFORMIX package (presumed
to be more flexible than the COBOL system) to use an ODBC data source
for its client details? Or if INFORMIX package is really not that
flexible, perhaps you modify the COBOL to use embedded SQL.

We are seeing many of our customers using Relational DataBridge as a
means of integrating two, formerly disparate, systems as those customers
begin to appreciate the UPDATE/INSERT capability. Of course, the
integration task is not as easy as the initial "accessability" for
report writer/spreadsheets, but it is happening; this is the nature of
the integration task. A case of evolutionary vs. revolutionary, I
think.
