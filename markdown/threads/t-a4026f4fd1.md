OK boys and girls...  I am doing a bit(!) of a rewrite on an existing
'maintenance' program.  It's currently a single CICS program that can be
used to:

Add an incoming wire record.
Add an outgoing wire record.
Browse the wire record file (can browse all or can chose some selection
criteria).
View an incoming wire record and possibly change it.
View an outgoing wire record and possibly change it.
Print wire notices from the wire records (some or all).
View 'wire totals'.

I am wanting to break up this rather large and disorganized program into two
separate programs.  One would be the presentation logic (still CICS at this
point) and the other would be business logic.

My basic questions is what do I allow for with regards to errors in my
COMMAREA.  We don't currently have any programs broken up this way, so I
don't have any examples to look at.  (This one will hopefully be a good
example for further development!)

Currently when we determine that a user has entered invalid input into a
field we do the following:
- Generate an error message to be displayed on the screen.
- Set the cursor position to go to that field.
- Use SEND MAP to display the screen.

With the business logic separate from the presentation logic I will not, of
course, be able to say "MOVE -1 TO FIELD1L".  I will have to set some field
in the COMMAREA to something such that the presentation logic program will
be able to evaluate that field to know where to place the cursor.  Show I
just use a numeric field and set it pretty much at random as long as each
number corresponds to only one field?  IE

**business logic**
  PERFORM LOOKUP-REF-NBR
  IF REF-NBR-ALREADY-USED
      MOVE 'REF NBR ALREADY USED' TO CA-ERROR-MESSSAGE
      MOVE +1 TO CA-ERROR-FIELD
      GO TO EXIT-PARA.

  PERFORM VALIDATE-ABA
  IF ABA-NBR-IS-VALID
      CONTINUE
  ELSE
      MOVE 'INVALID ABA NUMBER' TO CA-ERROR-MESSSAGE
      MOVE +2 TO CA-ERROR-FIELD
      GO TO EXIT-PARA.

**presentation logic**
  IF CA-ERROR-FIELD > ZERO
      PERFORM ERROR-LOGIC
  ELSE
     PERFORM COMMIT-LOGIC
  END-IF

ERROR-LOGIC.
  MOVE CA-ERROR-MSG TO WIRB-ERROR-MSGO
  EVALUATE CA-ERROR-FIELD
    WHEN +1
        MOVE -1 TO WIRB-REF-NBRL
    WHEN +2
        MOVE -1 TO WIRB-ABA-NBRL
  END-EVALUATE
  EXEC CICS SEND MAP(WIRB) END-EXEC
  EXEC CICS RETURN END-EXEC

This seems decent enough I guess.  It just forces me to figure out which
error-field value correponds with which field, but that's no big deal.  I
guess I was just wondering if there's a better way.

Also brings up the issue of, do I validate *all* of the fields rather than
short-circuiting on the first error.  If so, do I have an error table with,
say, 10 occurances where each occurance has both an error message (text) and
an error field value.

Of course the whole point to this is moving the presentation logic off of
CICS.  After this current project is complete we'll be starting on allowing
business customers to create wires from an Internet site.  Breaking this up
would allow the Internet server to "link" to the mainframe business logic
fairly seemlessly.

Anyway, would love to hear things from someone who's been there and done
that!

Thanks,


--- 
Frank Swarbrick
Senior Developer/Analyst - Mainframe Applications Development
FirstBank Data Corporation
