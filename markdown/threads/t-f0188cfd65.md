Tony,

There have been several reports of 30,xx or 98,xx errors being returned
by the RM/COBOL for Windows runtime (version 6.51 and greater) when used
with Windows 95, 98, or NT clients connected to Windows NT or Novell
Servers.  The following information, although written to apply to
version of RM/COBOL for 32-bit version of Windows (version 6.5 and
greater), may very well apply to any version of RM/COBOL running 32-bit
Windows.  Keep in mind that RM/COBOL v5.36 has not bee tested with
32-bit Windows.

Generally, the types of status codes that are returned to the COBOL
application are 30 and 98 errors.  Systems experiencing any unusual
frequency of 30 errors, or any 98 errors, should consider making the
changes suggested below.

Liant Software has found that these changes almost always eliminate the
30 or 98 errors.  I strongly recommend that the changes below be applied
to *all* workstations and servers accessing your RM/COBOL programs and
data, even just one workstation or server without these changes can
cause 98,xx and 30,xx errors that can effect all workstations and
servers.

1. VREDIR.VXD
This section applies only to Microsoft Windows 95.
When the Microsoft Client for Microsoft Networks is used, files that
reside on a server (such as a Microsoft Windows NT Server) may become
damaged or may contain invalid data if multiple workstations access the
file at the same time.

This occurs because the network redirector (VREDIR.VXD) caches data
locally for files it accesses on the server. If the last modified time
or file size does not change within a two-second interval, the
redirector reads file data from the local cache rather than from the
actual file on the server.

To resolve this issue, Liant recommends that you ensure that each
Windows 95 machine, has version 4.00.1112 (dated 2/11/97) or later of
the file VREDIR.VXD.

This update can be downloaded from the Microsoft web site:
   http://support.microsoft.com/download/support/mslfiles/vrdrupd.exe

For more information, see the Microsoft Knowledge Base
Article:
   http://support.microsoft.com/support/kb/articles/q148/3/67.asp


2. Network Redirector File Caching
This section applies only to Windows NT Server and Workstation.
By default, when the Windows NT redirector opens a file for read or
read/write access, the redirector using the Windows NT system cache. As
a result, when data is written to the file, it is written to the cache
and not immediately flushed to the redirector. The cache manager flushes
the data at a later time. If an unrecoverable network error occurs while
the data is being transferred to the remote server, it may cause the
write request to fail.

NOTE: Microsoft warns that this change will slow down network I/O.  In
addition, improper registry changes can disable your network server.
Liant Software recommends that only a qualified technician make these
changes to your system.

To disable network redirector file caching, perform the following steps:

A. Start Registry Editor (Regedt32.exe) and go to the following subkey:
                   HKEY_LOCAL_MACHINE
                     SYSTEM
                       CurrentControlSet
                         Services
                           Rdr
                             Parameters

B. On the Edit menu, click Add Value.

C. Enter the following:
   Value Name: UseWriteBehind ( This entry enables or
                                disables the write-behind
                                optimization for WRITE ONLY
                                files. )
   Data Type:  REG_DWORD
   Data:       0  (1=True, 0=False. The default is 1.)

D. Go to the following subkey:
                   HKEY_LOCAL_MACHINE
                     SYSTEM
                       CurrentControlSet
                         Services
                           Lanmanworkstation
                             parameters

E. On the Edit menu, click Add Value.

F. Enter the following:
   Value Name: UtilizeNTCaching  ( This entry indicates
                                   whether the redirector
                                   uses the cache manager
                                   to cache the contents
                                   of files. Disable this
                                   parameter to guarantee
                                   that all data is flushed
                                   to the server
                                   immediately after it is
                                   written by the
                                   application. )
   Data Type:  REG_DWORD
   Data:       0  (1=True, 0=False. The default is 1.)

G. Shut down and restart your computer.

For more information, see the Microsoft Knowledge Base Article:
   http://support.microsoft.com/support/kb/articles/q163/4/01.asp


3. Opportunistic Locking on Windows NT
This section applies only to Windows NT Server.

With Opportunistic Locking, if a file is opened in a non-exclusive mode,
the network redirector requests an opportunistic lock of the entire
file. As long as no other process has the file open, the server will
grant this oplock, giving the redirector exclusive access to the
specified file. This will allow the redirector to perform read-ahead,
write-behind, and lock caching, as long as no other process tries to
open the file.

Liant Software recommends that this Windows NT feature be disabled.

You can disable Opportunistic Locking by performing the following steps:

A. Start Registry Editor (Regedt32.exe) and go to the following subkey:
                   HKEY_LOCAL_MACHINE
                     SYSTEM
                       CurrentControlSet
                         Services
                           LanmanServer
                             Parameters

B. On the Edit menu, click Add Value.

C. Enter the following:
   Value Name: EnableOplocks
   Data Type:  REG_DWORD
   Data:       0  (1=True, 0=False. The default is 1.)

D. Shut down and restart your computer.

For more information, see the Microsoft Knowledge Base Article:
   http://support.microsoft.com/support/kb/articles/q129/2/02.asp


4. Virus Protection Software
This section applies to all Windows and Novell environments.
There have been a few reports of certain virus protection programs
interfering with RM/COBOL data files. This can result in file corruption
or invalid error messages.  If you are experiencing either of these
problems, and have virus protection software either enabled on the
client or the server, Liant Software recommends adjusting the parameters
that control the behavior of the virus protection software.

If possible, you should configure the virus protection software so that
it does not scan the COBOL data files after every modification.  If
necessary, completely disable scanning of COBOL data files.


5. Novell Client32 for Windows 95 Version 2.11 Required
The following paragraph applies to those users who choose to use
Novell's Client32 package for access to NetWare rather than Microsoft's
Client for NetWare.
NOTE: The RM/InfoExpress SPX client, rmspx32.dll, requires the use of
Novell's Client32.
RM/COBOL v6.5 or higher for Windows 95 requires version 2.11 (or later)
of NetWare Client32 for Windows 95 (dated 8/21/96, file size 461,359
bytes).  Record locking does not work properly and file corruption may
occur with earlier versions of Client32.  Additionally there may be open
errors (94,xx) or other permanent errors (30,xx) with older versions of
Client32.  Receiving error 30, MS Windows error 1 (30,12,00001 from
C$RERR) may indicate a defective version of Client32.

Other errors that have been reported are: 30,05 (Access Denied) or
"Error invoking unauthorized copy of runtime on network" error messages.

There have also been reports of NetWare's Client32 for Windows 95
causing file corruption.  Novell recommends turning off "Opportunistic
Locking" and "Packet Burst".  These settings can be adjusted from within
the Control Panel/Network/Client32 Properties/Advanced Settings.

Contact your Novell support representative for further information about
the Client32 software.

I hope this information is helpful to you.
