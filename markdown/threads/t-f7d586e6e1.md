### Avoiding the "wrong" Go To XYZ-EXIT problem

- **From:** "William M. Klein" <wmklein@nospam.ix.netcom.com>
- **Date:** 2002-12-16T14:06:30-06:00

```
This is my usual "suggestion" (follow-up on another thread) on how to
"guarantee" that you never cut-and-paste or otherwise cause a "wrong" Go To
XXX-EXIT code problem.  (This assumes that you have the new "EXIT"
enhancements from the 2002 COBOL Standard - available as extensions in some
'85 Standard compilers)

1) NEVER code a "GO TO" any where in your program

2) Never code a "PERFORM ... THRU" in your program

3) Always use ONLY paragraphs or only sections (no combinations of the two)

4) Use EXIT PARAGRAPH (or EXIT SECTION - depending on what you select for
item 3) to "escape" from within a procedure.  (Although it is true that this
is an "implicit Go To" - it never uses a Label and can never lead to "fall
thru" logic")

    ***

If you follow those 4 rules (obviously easier in NEW code than in
maintenance), then you will NEVER run into a problem with incorrect
"exit/escape" logic labels - but will still be able to "escape" from within
a procedure without deeply nesting IF's or using switches.
```

```
The "traditional" example of a (semi-?)good use of a Go To is to "escape"
from within a procedure to its exit - without the need of deeply nested IF's
or switches - or more nested performs.

For example (in semi-pseudo-code)

 Mainline.
      Perform  Para1 thru Para1-exit
          Until   Something
     Stop Run.
Para1.
    Do some-logic
    If  Field-1 = X
       Do Something-Else
       If Field-2 = Y
            Go To Para1-Exit
       Else
             Do Some-other-thing
       End-If
    Else
      Do Still-Something-else
   End-If
  Do All-but-both-X-and-Y
     .
Para1-Exit.
   Exit.

****

It is POSSIBLE to get around the "GO TO" statement - but it isn't easy - and
probably (depending on what you are used to) is more error prone.

If you try and "restructure" this code to avoid the "GO TO" - you will see
how it MAY actually end-up more "complex".  (FYI, this situation is the
rationale behind the "EXIT PARAGRAPH" feature of the 2002 Standard).
```

```
A lot also depends on coding standards in your shop.  For example, our shop
standard is to use a period at the end of a paragraph only.  It is not used
otherwise.  In an if construct if you need to exit the paragraph you state
NEXT SENTENCE which finds the next period and continues from there.
Otherwise you use CONTINUE which simply drops you to the end-if you are
currently in and continues from there.  You thus avoid having to write GOTO
some-exit-point.

Thus:
Para1.
     Do some-logic
     If  Field-1 = X
        Do Something-Else
        If Field-2 = Y
             Next Sentence
        Else
              Do Some-other-thing
        End-If
     Else
       Do Still-Something-else
    End-If
   Do All-but-both-X-and-Y
    .

Since the only period is at the end of the paragraph, the next sentence will
cause the paragraph to be exited.
```

```
PROHIBITS the use of NEXT SENTENCE in the same IF that has an END-IF with

There are a NUMBER of compilers that allow this (as an extension), but it is
NOT portable to a "strictly" ANSI/ISO conforming compiler.

  the "NEXT SENTENCE" construct has been labeled "archaic" (not yet
OBSOLETE) in the 2002 COBOL Standard.
```

```
Peter,

The IDMS coding you wrote 20 years ago would not look much different today.
IDMS calls do not take the form of EXEC/END-EXEC, but use their own set of
verbs that the pre-compiler recognizes, and converts to CALLs to IDMS.  It's
also possible to store code in the IDMS data dictionary that can be copied
anywhere in your program by use of 'COPY IDMS entity-name' statements.

I think that the situations that Howard is referring to come about due to
the use of a protocol option called AUTOSTATUS which will generate code
after every IDMS DML statement in order to insure that the return code from
the DML statement is checked.  The default code that is always generated is
simply a PERFORM of a section or paragraph named IDMS-STATUS.  The source
code for IDMS-STATUS is stored in the data dictionary and is included in the
source code via a COPY IDMS statement.  The code in IDMS-STATUS is protocol
dependent, i.e. different code is used for batch vs on-line programs.  It
can also be shop customized.  But it's primary function is to abort the
program on any non-zero return code from the IDMS call.

Obviously, there are some error status conditions that are expected and
acceptable, like 'not on file' or 'end of set', so its possible to allow
processing to continue in those circumstances by coding an IDMS 'ON'
statement.  The most common acceptable status codes have level 88 condition
names associated with them in the IDMS control block which must be copied
into every IDMS program.  It's those level 88 names that must be used in the
IDMS 'ON' statement.

With the AUTOSTATUS option, if you code:
MOVE '12345' to CUSTOMER-NUMBER
OBTAIN CALC CUSTOMER-RECORD

The precompiler will generate, from the OBTAIN statement:
* OBTAIN CALC CUSTOMER-RECORD   ---comment your DML
    CALL 'IDMS' USING etc         ---generated (also some MOVE statements
preceding the CALL)
    PERFORM IDMS-STATUS     ---generated

If you want to allow for the 'record not found' condition, you would code:
MOVE '12345' to CUSTOMER-NUMBER
OBTAIN CALC CUSTOMER-RECORD
ON DB-REC-NOT-FOUND
     your code to handle the condition
END-IF

and you will get:
* OBTAIN CALC CUSTOMER-RECORD    ---commented
* ON DB-REC-NOT-FOUND                     ---commented
   CALL 'IDMS' USING etc                ----generated
    IF NOT DB-REC-NOT-FOUND   ----generated
          PERFORM IDMS-STATUS     ---generated
    ELSE                                               ---generated
         your code to handle the condition    ---your code
    END-IF                                              ---your code

If you were to replace the END-IF in your code with a period, then the
resultant IF-ELSE will end with
a period.
```

```
One of the differences between 'heavyweight' and 'lightweight' styles
is the perceive cost of creating a new procedure.  In a 'heavyweight'
style the code must be enclosed in section header, exit paragraph,
given a number and put in the source in some place suitable for its
level.  Thus 'heavyweight' programmers tend to go to great lengths to
avoid this (such as using GO TO for loops, or fiddling nested loops).

'Lightweight' style just breaks a paragraph in two where it is and
sticks in a paragraph label and a PERFORM.  This means that they don't
need duplicated code (just two performs) and don't need to convolute
code to avoid this.
```
