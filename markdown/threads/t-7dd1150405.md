You already agreed this is "messy code" but I don't think it is really THAT
bad. The major problem with it is that you need to read the XML file as Text
AS WELL as XML in order to apply it...

The real problem here is that the OP has fallen into the trap of using Micro
Focus XML extensions so it can all be "COBOL like", rather than processing
XML with an external processor designed for that purpose, either by calling
it or invoking it. (I won't digress here into how you CAN have XML
integrated with your language if you use C# (LINQ) or Java (JOOQ), but NOT
with COBOL. (Oops, I forgot... you CAN have LINQ with COBOL if you use our
tools... mind you, it still isn't integrated...) For COBOL, the best options
with XML are CALL or INVOKE.

I believe his confusion has arisen because the OP read about COUNT IN as one
of the constructs used to extend the Micro Focus COBOL definitions, and he
has a feeling there is some kind of count you can apply to both OCCURS and
non-OCCURS fields and it will maybe return the actual length if it is
non-OCCURS. He is right, to the extent that there is such a counter, but it
won't do what he wants it to do.

The advice which Richard gave was sound in that, if you are processing XML
you really shouldn't be concerned about the actual length between tags
(especially not in COBOL). But we all come across things that are a bit out
of the ordinary occasionally and make us stretch to find a solution.

I see two solutions here (apart from doing it properly in the first place
(i.e. don't do it with a COBOL extension to a vendor's language
implementation... :-)))

1. You could use a regular expression to match on the and
tags. The match groups can return the start position of each match and you
can deduce the length between from those two numbers by simple arthmetic.
However, COBOL (even Micro Focus COBOL...) does not lend itself easily to
regular expressions. I found some example code here:
http://community.microfocus.com/microfocus/cobol/extend_and_acucobol/w/knowledge_base/18170.c-regexp-sample-program.aspx

It is pretty ghastly and has been based on the C++ regex libraries, much as
it would be in a mainframe environment.

And if you even did this, you would STILL have to read the XML as text to
get the tags.... (not really viable; same problem as in Kerry's example
above...)

2. Post process the "name" field after it has been loaded by the XML file
READ. If it is numeric, count the leading zeros, and if it is alphanumeric,
count the trailing spaces. You can then deduce the original actual length
between the tags in the XML file. BUT it is a very inelegant solution, it
isn't really 100%, and it doesn't seem worth the trouble.

COBOL is defined by its standard, not by any vendor's extensions
at a particular point in time!

The Technical Report for "Native COBOL Syntax for XML Support"
was released in 2007.

"This Technical Report provides extensions so that COBOL can
process XML documents as easily as it can read files. The new
syntax to process XML documents:
a. Is based on the familiar approach used with COBOL I/O support
b. Provides Document Object Model (DOM) style parsing
c. Handles multiple input sources to handle XML in an extremely
flexible manner
d. Reads, updates, and writes XML documents
e. Checks that XML documents are well-formed
f. Provides an optional validity check of an XML document against
a schema or Document Type Definition (DTD).

The decision was made to publish the specification in a Type 2
Technical Report so that the specification can be available for
implementation as soon as possible and so that implementations
can be undertaken on an experimental basis."

The report also states, in "Unresolved technical issues",

"In this Technical Report, the IDENTIFIED clause maps XML data
to a COBOL data item. There is no way to determine the number
of characters of XML data that are transferred into the data item.
Is there a need to know this length and whether the data was
truncated?

This restriction is temporary because when this TR is included
in a future edition of ISO/IEC 1989, the ANY LENGTH clause
can be specified on the data items that receive the XML data
and then there will be no limit to the size and the size can be
determined."

COBOL 2014 is the "future edition of ISO/IEC 1989" noted in the report,
and the changes to the ANY LENGTH clause means that standard COBOL can do
what the the OP requested, "obtain real field length between XML tags"
-- at least, in theory!

Unfortunately, it takes time for updates to standards to make
their way into implementations. It's not just COBOL, my understanding,
possibly mistaken, is that Microsoft is still updating their C++
to conform to C++ 11.
