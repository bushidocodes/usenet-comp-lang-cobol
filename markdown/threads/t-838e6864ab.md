```
Another follow-up on another thread.  Partially a comment - partially a
question.

Comment:
  The 2002 Standard "cleared" up the issue of the period after a COPY
statement versus a period at the end of a sentence.  The period for COPY
(and REPLACE) has now been "added" to the general format and the concept of
a
    compiler-directing statement
        versus
    compiler-directing sentence
has been removed.  This means that the separator period at the end of a COPY
statement (whether in the procedure division or elsewhere) is "just" a part
of the syntax and has no relevance to NEXT SENTENCE, delimiting procedural
statements, or anything similar.
 (This change bothered me a bit when first discussed, but I think I have
gotten to where I agree with it.)

Question:
   Joe indicated in a post (as I understood it) that using COPY statements
(in the Procedure Division) was a good reason for NOT using periods in the
middle of paragraphs. (I assume he meant in the "main" source code).  Howard
came back with why he thought that they actually helped to insure that the
"unseen" code would have the proper scope (avoiding unexpected logic
problems).

My question to both of them (or others) is
  - If you disallow NEXT SENTENCE *and* you use SCOPE terminators to end all
conditional statements (turn them into imperatives) in *both* your "main"
and COPY code, can someone help me understand what difference the use of
periods in the middle of paragraphs (other than style) would make?  I don't
(personally) think that COPY makes any difference to the "basic" discussion
(which is a matter of style - as far as I can see).  However, I would be
"happy" to see counter-examples if they exist.

I do see that using periods in the COPY source itself, limits its use to NOT
being where a "nested" imperative statement would appear in the "main"
source code.  This may have been Joe's original point.  I think that any
COPY member procedural source code should be one of the following:

 A) an entire paragraph (including paragraph header and "final" period at
the end of procedural statements)
        or
 B) one or more "scope terminated" imperative statement (allowing it to be
copied into any other procedural code)

Anything "in between" is certainly "error-prone" and limits the usefulness
of the copy member's code.
```

```
In the '85 Standard - (assuming that the COPY member did NOT end with a
period) you could (already) have either of the following (depending on what
you wanted to do):

  If A = B
      copy xyz.
  Else
     Display "this is still valid"
  end-If

        or

   If A = B
      Next Sentence
   Else
       Copy xyz. .

(The period at the end of a COPY - or REPLACE - statement has NEVER ended a
"procedural" sentence and has always been required in standard-conforming
code. It is just that the 2002 Standard "clarifies" this.)
```

```
I've watched this bewilderment. <G>

Can anybody clarify for me why you would WANT to COPY into the Procedure
Division?

Presumably, it is because you have code that you use many times over in many
different places, and it has been debugged so you can just pop it into your
source code and forget about it?

This is like negative evolution...(I think they call it "retarded" in the
trade...<G>).

Forty years ago we did this because we HAD to. (We even maintained card
decks of source code that could be duplicated and physically inserted into
other card decks...).

But then we found that the ability to CALL functions (Modules) precluded the
need for this. Instead of a COPY statement in the Procedure Division you
could use a CALL, and there was no need to ensure that you had all the Data
Names for the included code, defined in Working-Storage (with another COPY,
right? <G>), just an interface.

The reuse and encapsulation of code has shown a steady forward progression
from COPY statements, through Modular programming, into Object Oriented
encapsulated Classes, and, currently, to Components, which are not concerned
with Source Code at all.

(I covered this in an article on components I wrote recently. (see
"COMPONENTS101" on this month's edition of www.aboutlegacycoding.com)).

Elsewhere, in another thread here, concern was expressed  (I think it was by
Jimmy) that COPY statements might need to change as a result of OO
programming using Properties...

It is a good point. I decided about 18 months ago  (having tried various OO
experiments doing it both ways...) that the best way to manipulate
components is by setting their properties, THEN invoking their methods,
rather than increasing the parameter list passed through INVOKE.  This
approach also has the advantage that Properties can be Public or Private and
some very subtle behaviour can be obtained by careful use of this.

Anyway, If anybody is still using COPY in the Procedure Division of their
COBOL code could they please give some examples of WHY they do this?

I'd like to see if I have missed something here.

Thanks,

Pete.
```

```
Pete,
  I don't know what experiences others have had (or still have), but the use
of COPY members for DECLARATIVE SECTION "exception handling" where you want
to have the entire program's Working-Storage (and/or Linkage and/or
Local-Storage) available - but want to have a "shop-wide" set of actions is
one place that I have used Procedure Division COPY statements.  If you
replace these with CALL statements (or INVOKE) you run into "losing" access
tot he program that has encountered the problem.

Similarly, one can use a COPY member to handle various file status codes
"consistently" across programs - or SQL return codes or similar returned
"values".

Finally, it is possible to put an entire Nested Program in a COPY member.  I
don't know if you consider this to be a procedure division COPY statement or
not - but it is a technique that I am familiar with.

HOWEVER, I don't PERSONALLY think any of these are particularly common or
don't have other solutions, they are simply valid (IMHO) reasons for doing
this - in answer to your question.
```

```
What you failed to understand is that there is a differene between 'Quality 
Control' and 'Quality Assurance'.

If there was QC in the factory they would have a weighing device in the 
production line and any under weight would be discovered and rejected.  The 
rejects would be used for promotion or recycled.

As this QC is mechanistic there is no requirement to involve any 
statistical work, so you, or indeed any computer people, would not be 
involved at all, and thus would not know it occurred.

With QA, as you say, the under weights are not discarded but are 
distributed.  Samples are taken and analysed, the calculations are used to 
determine how many should have been rejected, and more importantly, the 
probability of anyone noticing and complaining about it.

Both mechanisms have costs.  The QC has the direct cost of the rejects, the 
QA has the direct cost of the sampling and the indirect costs of the 
complaints, possible government fines, and customer dissatisfaction.

With QC there may still need to be some sampling as proof that the QC is 
working.

Both are _valid_ means of obtaining quality (of the labeling),  the 
_choice_ of which to use depends on the manufacturer and which costs he is 
prepared to bear.

In the case of the response times, the customer also has a choice of having 
the system give some approximation of QA figures, or of having a quality 
control mechanism.
```

```
"Peter E.C. Dashwood" <dashwood@enternet.co.nz> >

> Can anybody clarify for me why you would WANT to COPY into the Procedure
> Division?
>

While I use to use three copy statements for each file (select, fd, and all
the procedures), I no longer do.  A few years back, this was the easiest way
to ensure correct methodology accross an entire system, when such things as
file/record locking were required.  Now, of course, I use OOP objects
instead, and the "copy" is done at the object level instead by creating a
new instance of the file handling instance.

However, I still use copy statements, even in OOP, and in one particular
case for procedures.

The SP2 screen system generates copy items for all of the screen parameters
used in any particular screen. Normally, and in the supplied samples, they
are working storage copy elements only.  They are, however, generated by a
scripting language that is designed to produce Cobol code.

I have modified that to also create standard methods for opening and closing
screens, setting the title, updating the screen with an immediate exeit
(write only) and entering the screen, waiting for input, and exiting with a
status code.  These are complete methods, from the method-id statement to
the end-method statement, so the question of periods is irrelevant.

My standard screen handling object simply has some common methods at the
top, and two copy statements per screen.  The first is a working storage
copy that sets up data fields for the screen as Cobol properties.  The
second is a procedural copy that includes the basic methods for each screen.

Here is some code that uses such an object.  Note that the cobol for the
ERROR-OBJECT is copletely generated, and can be moved from system to system
unchanged.  By modifying the code generator, I should be able to generate
code for any cobol on the market that has the ability to compile OOP
comliant code, recompile the error-object, and leave the below code
completely unchanged.  Once eveyone is up to the new standard, I will be
able to leave the code generator alone as well.

The only last thing required, is to make all the Cobol compilers link
compatable.  Then I should be able to put my error-handling OOP DLL on a
system as an application, and use it from within any cobol compiler.  That
is, if the language was specified right in that new standard .... can we do
that, and if not, why not?

       BAD-SEARCH.
           INVOKE LEDDLL1 "NEW" RETURNING ERROR-OBJECT.
           INVOKE ERROR-OBJECT "YESNO-OPEN-THE-SCREEN".
           MOVE SHORT-TITLE TO LOCAL-TITLE OF ERROR-OBJECT.
           MOVE SPACE TO YESNO-MESSAGE-A OF ERROR-OBJECT.
           MOVE "Search Failed !" TO YESNO-MESSAGE-B OF ERROR-OBJECT.
           MOVE "Search again?" TO YESNO-MESSAGE-C OF ERROR-OBJECT.
           INVOKE ERROR-OBJECT "YESNO-SHOW-THE-DATA".
           INVOKE ERROR-OBJECT "YESNO-GET-THE-DATA".
           MOVE SCREEN-KEYSTROKE OF ERROR-OBJECT TO LOCAL-KEYSTROKE.
           IF LOCAL-KEYSTROKE IS EQUAL TO YES-HIT
               MOVE "N" TO END-FLAG
               MOVE "Y" TO SEARCH-END-FLAG
           ELSE
           IF LOCAL-KEYSTROKE IS EQUAL TO NO-HIT
               MOVE "Y" TO END-FLAG
               MOVE "Y" TO SEARCH-END-FLAG
           ELSE
               MOVE "N" TO END-FLAG
               MOVE "Y" TO SEARCH-END-FLAG.
           INVOKE ERROR-OBJECT "YESNO-CLOSE-THE-SCREEN".
           SET ERROR-OBJECT TO NULL.
```
