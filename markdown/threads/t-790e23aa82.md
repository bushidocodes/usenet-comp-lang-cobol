Thank you for all your comments  and suggestions.  I got the problem
resolved today by writing disk files, then examining them to see what
happened using various code methods and compiler directives.  Then used
other programs to sucessfully read and process the data items.  A lot of
what I uncovered is what I found in my e-mail and groups posts when I got
home a few minutes ago. ( I cannot access newsgroups or home e-mail from my
college office).  What I don't yet understand is    (a) Why I never had this
problem in previous years of teaching COBOL, mostly on EBCDIC mainframes,
but with some coverage using microcomputers,  and,    (b)  Why is there not
any coverage of this subject in the textbooks that use microcomputer cobol.
I find I have a new lecture subject to bring out in my classes.

In summary, I found that only lowercase letters would work, p thru y for
last digits of 0 thru 9, when the field is negative.  The clause 'sign is
trailing' is unnecessary.  No runtime compiler directives ($SET commands)
affected my outcomes.

Again, I thank you for your support.  I will certainly be aware of you
contributions in assistlng and broadening my approaches to microcomputer
cobol.  As Net Express arrives on the scene, I can envision that cobol will

friends.

Per your comments:
Volker Bandke  --  your detailed e-mail was most appreciated - wish I could

Tony Dilworth, Thane Hubbell  --  I wasn't worried about file status in this
test code.  It is certainly used appropriately in commercial code for most
I/O.

Michael Mattias  --  A Read is actually treated as a group operation in
COBOL.  The actual run time error occurs when you attempt to use the
'corrupt' data item...  say, when my test code or student programs attempt
to move or add an item.  Using Read Into wouldn't alleviate the problem, but
can allow more ways to validate an incoming data value.

William Klein  --  ASCII is the default of MF Cobol, which I have verified
many times over the past years.  The $SET compiler directive SIGN"ASCII" did
not affect results.  It is the default if not overturned.  The CHARSET was
returned as invalid the one attempt to use - did not pursue as I started
getting good responses from the disk writing trials by then.

Ray Smith,  Pat  --  The code conversion (EBCDIC to ASCII) idea is good, and
