There are four paramters. First is usually the program name. Second is the
request area. It contains the command, the key name, the key value, the
return code etc. The third parm is the record layout. The last parm is the
element. Most of the time there is only one element e.g. for the FGA table
it would be FGA01. We put filler at the end of the record to allow for
expansion but sometimes we run out. Then you can create a second element
e.g. FGA02. Think of this as one logical record with two physical pieces.
You can retrieve them together if you want or separately. Anyway that is a
pretty rare occurence. Most commands require all four parms but not all of
them do. I have a sheet that lists the commands and another that lists the
return codes. I have some documentation. If you are interested in it I can
make a copy and mail it to you. Send me your address off newsgroup.

The orginal Datacom was more navigational. Read key greater than (REDKG),
read next (REDNX), read key exact for update (RDUKX) etc. Then they had
something called set at a time processing. We never used that much and I
was told it was because of poor performance, but I don't know if that was
true. Now days if your tables are defined correctly i.e. according to some
restrictions, like no 88 levels (condition names) then there is a SQL
interface that you can use. Again I have not used that. Some of our tables
have numbers of records in the hundreds of millions and we get good
performance using primarily the navigational approach.
