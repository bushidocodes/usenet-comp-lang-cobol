While you might be able to read the DATA portion in a meaningful way, the
INDEX portion is usually proprietary and contains pointers, pointers to
indexes, pointers to data tracks, pointers to overflow areas, pointers to
available records, and badgers.

I don't hold out much hope for deciphering the index portion of the ISAM
file.

Could you contact the compiler vendor with the symptom?

My bet would be that the index file marks the index 
references of changed or deleted records, and adds new 
pointers for the new location of the changed or added 
record.  This probably causes the increase in size - check 
update and add activity.  The rebuild process probably 
removes the now unused pointers, and reorganizes the file 
back into the correct indexed state.  At the same time it 
probably actually deletes the records from the file and 
possibly reorganizes (sorts) it back into the index order to 
increase access time.  Just a guess as I don't know what 
kind of files these are - ISAM, VSAM, etc.
