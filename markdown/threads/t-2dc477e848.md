In my company we use Refernce Modification with some restraint, no more than 5% of programs.
Generally, it is better to handle these problems through the use of redefined data items that can be
given 88-level names. If it is used too liberally, it is easy to lose track of what data is actually being
manipulated. The feature eliminates COBOL's inherent self-documenting attribute. Carried to an
extreme, one could write a program that never names any record's data items, only refering to them
as absolute positions within the record. That sort of program would be as hard to maintain as an
Assembler program.

There are some places where it can be VERY useful, such as when a data item must be redefined
on-the-fly.
