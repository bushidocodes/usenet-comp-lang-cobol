```
I am translating legacy Cobol ('74) batch programs into pro*Cobol.
When the older I/O verbs become embedded SQL, there is a
need to add host variables to the DATA DIVISION. Basically, any
table column that gets accessed by the Cobol program, may need
a 'host variable' to act as a common denominator between Cobol's
way of defining fields and Oracle's way of defining columns.

A simple way of adjusting to the need for
'host variables' is to add one for _each_ of the existing Cobol fields.

The advantage of this approach is to leave the old procedural code untouched.
The disadvantage is possibly doubling the size of the DATA DIVISION.

Is anyone familiar with this dilemma?

Is it worth the trouble to re-formulate the DATA DIVISION,
replacing all the incompatible COBOL data fields with 'host variables'?
Any suggested naming conventions for the host variables? For example,
the host variable for emp_ssn might be called host_emp_ssn.
```
