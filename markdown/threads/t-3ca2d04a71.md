And just to make things even more interesting, the DATA(24/31) option doesn't
always have meaning.  It is affected by (at least) the RENT and RMODE compiler
options and the HEAP(BELOW / ANYWHERE) run time option.

You should read the relevant COBOL compiler Programming Guide completely for
all of the compiler options, and the relevant Language Environment Programming
Guide for STACK and HEAP Storage information.

I had thought that I could have "fast" 31-bit programs by compiling with
DATA(31), NORENT, RMODE(ANY), but later found out that the DATA setting was
only meaningful if RENT was chosen.  Since RENT causes COBOL to allocate
Working-Storage and FD records areas dynamically (using HEAP storage), and then
there is code executed to set initial values, it seems that RENT programs would
have to execute a little slower than NORENT.  Of course, the load module is
smaller, so they can load faster, but I'm sure the operating system is pretty
efficient at program fetch.

Interestingly enough - from the Enterprise COBOL V3R1 "Performance Tuning

"RENT or NORENT
Using the RENT compiler option causes the compiler to generate some
additional code to ensure that the program is reentrant. Reentrant programs
can be placed in shared storage like the Link Pack Area (LPA) or the
Extended Link Pack Area (ELPA). Also, the RENT option will allow the program
to run above the 16 MB line. Producing reentrant code may increase the
execution time path length slightly.

Note: The RMODE(ANY) option can be used to run NORENT programs above the 16
MB line.

Performance considerations using RENT:

    | On the average, RENT was equivalent to NORENT."
