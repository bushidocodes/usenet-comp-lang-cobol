```
We are writing a calculation system that uses a central accumulation
routine with a large I/O.

This is made up of 2 or 3 dimensional arrays - 12 indicies are uses throughout
the 106 arrays, and to cater for different types of contract, the arrays are 
allocated dynamically.   This is done by defining the indicies and deriving the
values for a particular case, then using an Assembler program to calculate the 
size of each array and allocate the storage accordingly.  The addresses of the 
arrays (01 levels) are stored in pointer variables, then whhen the COBOL wants
to access the storage, SET ADDRESS commands associate particular storage with 
the corresponding array.

The arrays are defined like :

01  CI-CHG-ARRAY.                                   
    05 FILLER                 OCCURS 1440 DEPENDING ON    
                                          MAX-MTH.        
       10 FILLER              OCCURS 250  DEPENDING ON    
                                          MAX-FUND.     
          15 CI-CHG                       COMP-1.


For most cases, the values of the indicies will be fairly low, but we have
to allow for the maximum.  The idea was that the system will size the arrays
and compare to a pre-defined limit on a table - if the size is less than the 
limit then the case is deemed to be small enough to run through the 'fast' 
transaction, if the size of the case exceeeds the limit, then the 'slow' 
transaction is used, with more memory available, hence the need for dyncamic
arrays.

However, when we're testing the prototype we're finding that using 'OCCURS 
DEPENDING ON' increases performance by about 2-3 times, which is unacceptable.

Does anyone know why the ODO would affect performance so much?

Are there any other ways to achieve what we want i.e. a system that can dynamically
define the mask for a piece of storage?


Thanks,
```

```
My GUESS is that this is an IBM mainframe environment.  If not, please tell
us which compiler and environment you are talking about.

Nested ODO's is an IBM extension - and is (as you have found out) a fairly
"complex" construct for the compiler to handle.  I assume you are using the
latest IBM compiler (Enterprise COBOL) and are using the OPT compiler
option.  There have been some optimization improvements for ODO's from
release to release, so this MIGHT improve your performance.

My suggestion (at first glance - and there may be better ways of doing this)
would be to

 A) make the lowest level a fixed format OCCURS
 B) either keep the top level an ODO - or (possibly even better) make it a
fixed OCCURS also - but never "access" beyond the limit used by your
assembler routine for acquiring storage.  As the table must be in Linkage
Section (as you use a SET ADDRESS OF to get its addressability), then you
shouldn't be in any problem if you define the table as "too large" but do
not try to ACCESS any storage beyond what your Assembler obtained.

Again, there may well be some "better design" approaches for this
situation - but without knowing all the details, it is hard to come up with
something that will meet your run-time performance requirements.
```

```
I do it by making the second dimension another table, like this:

01  CI-CHG-ARRAY.                                   
    05 FILLER                 OCCURS 1440 DEPENDING ON    
                                          MAX-MTH.
         10  link-to-funds    pointer.
 01  FUND-ARRAY.        
       10 FILLER              OCCURS 250  DEPENDING ON    
                                          MAX-FUND.     
          15 CI-CHG                       COMP-1.

Then change:

move CI-CHG (MTH, FUND) to ...

to:

set address of FUND-ARRAY to link-to-funds (MTH)
move CI-CHG (FUND) to ..

This technique provides an infinite number of dimensions. You're not limited to
the COBOL max and your speed problem will vanish.
```

```
Just as a caution:  "OCCURS <integer-1> DEPENDING ON data-name-1" is an
implementation-specific  extension.  In any OCCURS format in which DEPENDING
appears, TO must appear also, in '68, '74, '85 or '02 standard COBOL, and
any implementation that relaxes that rule does so at the expense of
portability.  See for example ISO/IEC 1989:2002 Page 306, 13.16.36 OCCURS
clause, Format 2 general format (note the absence of square brackets in the
salient area of this format), and Page 307-308, syntax rule 21, which
applies to Format 3 and explains the need for square brackets in that
format.

    -Chuck Stevens
```

```
(1000, 1) requires the code to go through outer-dimension (1) through
outer-dimension (999), adding their sizes to determine where outer-dimension
(1000) starts. The table of pointers I suggested eliminates that calculation.
```
