The DDNAMEs are defined in the using PROGRAMs and changing it requires the
program be recompiled; JCL just tells the system "when a program wants this
DDNAME, use this DATASETNAME (DSN).

If you want to amend your question to ask, "can I validate that all the
DSN's in all these JCL members are valid?" then the answer is, "I doubt it."

You can read the JCL as a sequantial file; when you come to a DD statement,
parse out the DSN from the line. Then attempt in your program to OPEN that
file, and interrogate the file status. I've seen code here and I'm pretty
sure its in the manual how to create the control structure needed to open a
dynamically-named file from your program.

BUT...

1. Many JCL members will use SYMBOLICs in the DSN (e.g
DSN=&QUAL..&NODE.MY.DATASET); in this case you are in essence, "SOL" as the
SYMBOLIC expansion occurs at EXEC time.
2. JCL will contain datasets which are created by a program... meaning that
dataset will not exist, which is a Good Thing.
3. JCL will refer to datasets created in a previous job step, meaning those
datasets will not exist until run time (also a good thing).
4. DSNs will sometimes be identified with a referback (e.g,
DSN=*.stepname.ddname), which will immeasureably make your parsing task more
difficult.
5. DDs and the corresponding DSNs may not even be specified in the job step:
they may be specified as an override in the EXEC statement which actually
starts the job; or they might be specified in the job step, but overriden
anyway!

All this assumes, of course, that what you changed were the DSNs, not the
DDs. If you changed the DDs, you had to recompile anyway and presumably you
tested. (If you changed DDs but did not change the programs, please restore
from backup and start over).

Bottom line, I think you are probably going to wait on "run time JCL Error"
to discover any mistakes. If nothing else, you should keep track of each JCL
as it is executed, so at some point you will know that all that JCL has been
tested (tested by fire, that is!).

MCM

There are a few things you can try,
if this is an IBM mainframe, (MVS) OS/390 , Z/OS.

First you don't mention if you are using PROC, or JCL execution.

If you are using PROCs, you can use a TYPRUN=SCAN on a single JOB that
Executes all the PROCS.
That will give you one hell of a listing, but everything is in one listing.
If you are on SDSF, you can use the PRINT D command to create a file of the
expanded JCL.

The tricky part is finding the DSN= in the lines that say SUBSTITUTION JCL.
If you are up to it, you can get the DSNs out of the listing.

You can also do a LISTCAT (ENTRIES or LEVEL) to check the names of the files
on your system.
You'll have to test to see the return code you get if the file isn't there.

1. Get the DSNs out of the listing
2. sort the names, and eliminate the duplicates
3. write a program to use the names to generate a LISTCAT  statement, and
run them through IDCAMS.
4. Check the return Codes from the LISTCAT.

This is a bit vague, and you'll have to look at the listings to see what's
there.
This is NOT a simple method, but writing the program could be fun.
I am assuming that you are using PROCs, and they have symbolic
substitutions.
A DSN=& is something you don't want.
The other DSN= statements should give you the file name.

Discuss this with your co-workers before trying to do it.
