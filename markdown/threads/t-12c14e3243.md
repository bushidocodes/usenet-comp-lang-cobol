As far as I know, ++INCLUDE never had any feature that would support
copy replacing. It does, however, support nested ++INCLUDE statements.

I really haven't used COPY REPLACING enough to be facile with it. I've
seen it in an old OS/VS COBOL program, and I believe it doesn't work in
the same user-friendly way in COBOL II, which would mean it was an IBM
extension that was replaced by ANSI standard COPY REPLACING.

Back when we still used Panvalet, most compile JCL was set up to
++INCLUDE the entire source program into the submit JCL. We also had
++INCLUDE statements embedded in the source code, but these caused
tremendous problems when we converted to Endevor for source code
management. Basically, we had to convert all the ++INCLUDE's into COPY
statements and move the copybooks into true copybook libraries.

It would be an interesting exercise to write your own pre-compiler to do
your own version of copy replacing.

A couple bits of info.

1) Panvalet CAN be run as a separate job step BEFORE the compiler step. In
such cases, any COPY or COPY/REPLACING statement which is any level of
++INCLUDE will look like a "normal" COPY or COPY/REPLACING statement by the
time the compiler sees it.

2) PANVALET has a method of being invoked "dynamically" during a compile
step. Similarly, VS COBOL II and all the COBOL for this-and-that compilers
support EXIT compiler options which can invoke PANVALET (or any other
program) during source code input (and COPY) processing. If this is the
technique you are using, then I would suggest NOT trying to have standard
COBOL COPY or COPY/REPLACING statements intermixed with ++INCLUDE
statements. There might be a way to get this to work, but I think it would
be more trouble than it would be worth.

3) The differences between OS/VS COBOL and the current COBOLs for
COPY/REPLACING processing is minimal if any. (There are some problems if
you are moving from OS/VS COBOL LANGLVL(1) - but even these can be
automatically fixed by products like CCCA.)

4) When looking at these issues, do NOT forget our friends CICS and DB2.
The interaction between EXEC SQL INCLUDE and COPY *or* ++INCLUDE can get
tricky - and, for CICS, you need to remember that you can only COPY or
++INCLUDE source code that either has no EXEC CICS in it or which has
already gone thru the precompiler.

Note: For readers with Librarian, almost all the rules for ++INCLUDE also
apply to -INC.
