```
I've seen questions on this topic posted here a couple of times.
Recently I found I had a need to create a socket client under Windows.


So how to do it.... I found the Sockets API on the internet and
downloaded it, and I downloaded the documentation.  I've interfaced
with the 16 bit Windows API, but never anything as complicated as the
Windows API.  The sockets were not a new concept either - recently I
wrote a program on the OS/390 mainframe (Actually I ADAPTED a program
that someone else here wrote) that acts as a Server.  We are
interfacing to it from the DEC/VMS system.  At any rate, I was
impressed by it.  It's similar to what I wrote a while back (Async
streaming communications), and I figured it all out pretty fast.  

But this Windows API sometimes seems purposely obtuse.  I scanned
DejaNews and found a post by Dave Maxwell, who indicated he had some
source code for MicroFocus COBOL that accessed the Sockets API.  Dave
was kind enough to share that code with me, and I ported it to work
under the Fujitsu compiler.  He gave me permission to make his code
available on the internet, so... if you want some source code to
inteface with the sockets API, MicroFocus and Fujitsu versions are
available from my web site. 

Go here: http://www.geocities.com/Eureka/2006

and press the downloads button - You will have to scroll down to the
bottom.

This was my mini project I was talking about earlier.  It really helps
to understand C, and especially Pointers when interfacing with the
Windows API.
```

```
The comments and the Symbolic constants are the only non 85 standard
stuff.  Several EXTENSIONS that are Fujitsu specific are used to
interface with the Windows Sockets API.  The syntax, however, is very
similar to the MicroFocus syntax.  

The Evaluate below is standard.

The copy that was used in the MicroFocus example was generated (I
beleive) via a MicroFocus tool that will take C header files and
convert them to COBOL - but it used several MicroFocus specific
featuers that made it eaiser for MF to interface with the API - these
include the 78 levels (which Fujitsu will support in the next release)
and dynamic field typing.  ie:  New usages were created by these
headers.  They were implemented then as external variables in nested
programs.  Since Fujitsu didn't support the 78 levels or Dynamic
typing - AND I AM NOT SUGGESTING THAT THEY SHOULD!  I find #DEFINE
after #DEFINE after #DEFINE to be confusing and unnecessary.  For
example in the API Windows redefines and Unsigned Integer to WORD -
WHY?  Why not just call it an unsigned integer! <G>.  Anyway, I just
tried to make it as clear as possible, while understanding it and
porting it.
```
