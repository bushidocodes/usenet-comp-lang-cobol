this trying to avoid a S0C7 ABEND. When INFIELD wasn't
numeric, it still got the S0C7 (in the middle of the "IF").
So, in an attempt to avoid this problem, basically they did
the following:

05 INFIELD....
05 INFIELD-X REDFINES INFIELD PIC X(4).
   .
   .
   .
  IF INFIELD-X IS NUMERIC
     PERFORM A
  ELSE 
     PERFORM B.

Can you guess which paragraph was performed no matter what
the test data was?

(how it got mangled was another story).

The IF NUMERIC test is part of what's called a CLASS test (NOT "Class" in
the OO sense; this was formulated 40 years ago when there was no such thing
as Object Orientation...)

Before PICTURE was available, the CLASS of data was very important and we
used to describe data by its CLASS, SIZE, and USAGE (SIZE 5, CLASS NUMERIC,
USAGE DISPLAY would be pic 9(5), for example.)

Data could be one of 3 CLASSES:

1. NUMERIC.
2. ALPHABETIC
3. ALPHANUMERIC

Even after PICTURE was admitted (sometime in the early 60s if I remember
correctly) the CLASS tests were retained because it was sometimes useful to
see if something in a pic x string was actually NUMERIC, or ALPHABETIC, or
ALPHANUMERIC.

However, as the fancy new PICTURE clause already implied certain classes,
there was no point in applying a CLASS test to something you already knew
the class of.

F'rinstance, if you have something defined as pic 999, it MUST be NUMERIC
because the NUMERIC Class is implicit by the character 9 in the picture. Any
attempt to store a non-numeric character here will result in a compiler
diagnostic...
Because the compiler knows the CLASS of the field from your Data Division
entry, it can check what you are moving to it and rap you over the knuckles
of you take liberties...

The early standards therefore said that CLASS tests could ONLY be applied to
data held in ALPHANUMERIC strings... So you could check if something that
was technically ALPHANUMERIC (could hold literally ANYTHING) was in fact one
of the 3 CLASSES mentioned above.

It is the same logic as if you defined something like this...

01 alpha-field  pic A(12).

It is nonsense to code:    if alpha-field is NUMERIC.... it cannot be. It
has been defined to hold letters of the Alphabet and space. (It's CLASS is
ALPHABETIC). Similarly, it would be nonsense to code: if alpha-field is
ALPHANUMERIC... for the same reasons.

The same logic applies to coding: if NUMERIC.. for a field which is defined
with 9s in the picture and is therefore implicitly of CLASS NUMERIC... It
HAS to be NUMERIC...

Now of course, in reality, there is a "hole" in this reasoning in that
something which is NOT NUMERIC COULD be moved to a CLASS NUMERIC field if it
is done at the group level... or read in as part of a record moved at the
group level.

Non-IBM mainframes would simply process whatever was in the field and give
wrong results (it was common practice to make ALL input fields ALPHANUMERIC
and check them with the CLASS test before moving the numeric fields to
numeric pictures so they could be used in arithmetic...); IBM gave the
infamous S0c7 and dumped. As dumps are fairly tiresome, there was a lot of
grizzling and the IBM implementation was Extended to allow people to check
even non-ALPHANUMERIC fields for being NUMERIC. So, you can be forgiven for
believing that ANYTHING in COBOL can be checked for NUMERIC.(Especially if
you have only been exposed to IBM COBOL...)

But realize that this is a VENDOR EXTENSION to make life easier on a
particular platform. It is NOT "Standard" (whatever the Hell that is these
days..<G>) COBOL.

Whether we agree or not with the above reasoning, in the light of current
practice, that's how we got to where we are...

Pete.
