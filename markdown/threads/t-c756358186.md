```
The following short program doesn't work. I can create temporary table
(sqlcode = 0), but if I try to access it after that sql server allways
returns error code 'Table doesn't exist'. It looks like after I create a
temporary table program loses connection to database and table is
dropped. Why is this happening? I am using NetExpress 3.0 and MS SQL
server 7.0 on win 2000 server.

-----------------------------------------------

      $set sql(dbman=odbc)
       WORKING-STORAGE SECTION.
       exec sql include sqlca end-exec
       procedure division.

           exec sql
               connect to 'database' user 'sa'
           end-exec

           exec sql
               create table #table1 (
                   numb1   int     not null,
                   name1   char(10) not null)
           end-exec
           display sqlcode  *> Everything ok here
           exec sql
               commit
           end-exec

           exec sql
               insert into #table1
               (numb1, name1)
               values
               (1, 'adsdfas')
           end-exec
           display sqlcode  *> Here's the problem: 
           exec sql         *> Table doesn't exist
               commit
           end-exec

           exec sql
               drop table #table1
           end-exec
           display sqlcode  *> Same thing here
           exec sql
               commit
           end-exec

           exec sql
               disconnect current
           end-exec

           stop run
           .
```

Your problem may have something to do with the table name you picked.  Some
SQL engines may not like the "#" in the table name or will only accept it if
you enclose the table name with the identifier quote character.  You can use
the SQL Assistant tool in Net Express to display that info after connecting
to the data source and right mouse clicking on the data source and selecting
"data source properties".

I checked with one of our developers who know more about sql server and what
happens under the covers and his reply is:

This is a known featurette of SQLServer 7.0. Essentially, OpenESQL
 prepares and executes most statements. SQL Server creates temp stored
 procs for prepared statements. When you execute you execute a stored proc
 which creates a temp table, at exit SQLServer decides the table is now out
 of scope and drops it. With 6.5 you could tell SQL Server not to create
 temp stored procs, but 7.0 doesn't give you the choice. Hence the upgrade
 from 6.5 to 7.0 has broken some apps.

 The solutions are:

 - use global temp tables (i.e. ##temptable rather than #temptable). This is
  likely to be unsuitable in mutli-user situations, but could be OK for
 batch jobs.

- do the create table via an exec sql execute immediate ... end-exec. In
 this case OpenESQL does SQLExecDirect rather than prepare + execute to
 create the table, so SQL Server doesn't use a temp stored proc and the
 table is available to other statements (whether execute immediate or
 regular 'static' or dynamic SQL).
