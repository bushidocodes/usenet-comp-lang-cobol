I basically perform as a technical support liason for the COBOL
developers here at my current place of employment. I ran into a system
much the same as you have described here. When I asked the development
team why they did not compile to .GNT files for production execution, I
was told that they had tried but were unable to get the programs to
compile in that mode. After much hair-pulling I found that many of
these programs were very large. And that many of those very large
programs called other programs which were also very large and utilized
Linkage Sections which were not large, but friggin' HUGE. After much
bit-twiddling, counting and (God help me) cursing, I was finally able to
come up with the methodology to get these programs to compile to .GNT's.

A number of issues were encountered, but these were addressed by using
the following $SET commands inserted on the first line of code. (Oh BTW,
there is a cockroach in v3.2 that screws up your SET commands if they
are not all on the same line - can't continue to 2nd line):

SEGSIZE "65536" 64KPARA 64KSECT (and the real killer) SEGCROSS"1"

You will need to look all of these up to determine which one(s) are
applicable to your case. In the case of SEGCROSS you will need to
identify which unit of yout linkage section crosses over a 64k boundary
(if only executing in 32 bit mode). If you are executing in 16 bit
mode, the manuals are incorrect in identifying 64k boundaries. The 16
bit world wigs out when you cross a 32k boundary.

Your applications when functioning as .INT files have a number of things
going against them:
1) .INT is a "development environment" file which, when utilized by the
runtime, is given far more latitude memory wise than is generally
required (makes for memory pig programs).
2) .INT files, as I have found in benchmark testing, are as much as 50%
slower than .GNT files.
3) The memory utilized by .INT files is **NOT** released back to the OS
by the runtime. Subsequent operation (calling) of a previously called
.INT program will yield "tainted" memory (memory which is not
re-initialized from the previous execution of the program) - regardless
of any "CANCEL pgmname" commands.
4) Calling of subprograms written in other languages is flakey.
5) ISAM file handling is very very shakey as the runtime thinks you are
running in a "development" mode and will allow you to get away with
murder (one of our files (2,000,000 records) was created with over
400,000 duplicate alternate keys with actual duplicates having been
created - REBUILD.exe puked whenever it tried to re-org the darned thing
- turns out there is a 64,000 duplicate keys limit within a given file
structure)

Bottom line -
.INT files do *NOT* behave appropriately in all cases when executing out
of the development environment.

I hope this helps.
