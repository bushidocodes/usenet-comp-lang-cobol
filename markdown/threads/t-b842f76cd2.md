There are really two different problems one has to solve with direct
access storage devices to run 24x7: (1)dealing with hardware problems
which could potentially affect data availability; and (2)dealing with
planned changes caused by lease terminations, upgrades to new
technology, or growth. Historically, we have found that planned changes
have consumed much more of our time.

First of all, in a mainframe environment the DASD (Direct Access Storage
Device) Subsystem encompasses much more than the physical drives. If
there is any single-point-of-failure which could interrupt the
successful flow of data between the mainframe and the physical media,
then you potentially lose your 24x7 goal. Mainframes have been evolving
over 30+ years to address the interrelated issues of availability and
reliability, which is both a hardware and software issue.

Typical mainframe DASD subsystems today involve 100's of drives (in some
cases 1000's) which are organized physically into "strings", each string
connected by multiple paths to at least two controllers, and each
controller connected to the mainframe via at least two independent
channel paths. During normal operation of a string, all string paths,
controllers, and channel paths are used concurrently to increase the
available bandwidth to the drives in a string, but in the event of a
detected failure (and in the mainframe environment undetected failures
are so rare they can be safely ignored), failed operations are retried
and persistent problems with a path or controller will cause the failing
component to be logically disabled. To achieve redundancy at the
physical drive level, either data mirroring or other RAID designs are
used so that failure of single physical drives does not disable the
ability to read/write data to a logical volume. The entire DASD
Subsystem is designed with logical and physical power boundaries so that
all but the most unusual failures can be repaired without disrupting
access to data (performance may suffer because of reduced bandwith if
this occurs at a peak usage period). The software and hardware have
evolved and been closely integrated to insure that failures are detected
and diagnosed; that failed hardware components are isolated; and that
failed operations are then retried. Any physical repair can then be
made as convenient, depending on the exposure involved.

Planned changes in the DASD Subsystem generally involve much more work
because physical data movement must be deliberately initiated, but
generally you have several months to perform the task. The only way I
know to do these kinds of changes for the most part non-disruptively
requires that replacement DASD strings and controllers (which can be
added non-disruptively to new channel paths) be available in advance of
the cutover date to migrate files from out-going volumes. There are
generally ways to disable the departing volumes for new file allocations
and to enable equivalent new space on the new volumes. Normal
application maintenance functions will typically cause most files to
move off the old volumes over several months. However, there are some
files which require a reinitialization of the operating system to move,
and inevitably some of these are on out-going strings and some down time
will be required to move these (the MVS concept of "sysplex" can address
many of these problems). We normally run 24x7 -1 just to allow a small
weekly window to introduce such "changes", but some shops can't afford
that much down time.

Locating a file in MVS is generally a 2-stage process: identifying the
volume(s) containing the file (from examining system datasets called
"catalogs"); then looking at the Volume Table of Contents (VTOC) on that
volume to find the physical location of the dataset. This is different
from things in PC land, where there is no concept of identifying a file
just by name - you must know the volume/drive in order to start the
process - a reasonable requirement on systems where the average number
of drives is around 1; a very unworkable scheme when the average number
is over 100. On MVS, entire volumes can be copied/moved
nondisruptively, but to use this to free up a drive that is to be
de-installed requires that none of the datasets on the volume be OPEN by
any application--usually not obtainable if any applications are active,
and that's usually the case if we're talking about 24x7 operation.

Moving individual applications is usually irrelevant to above
considerations as there is rarely any one-to-one correspondence in the
mainframe world between DASD volumes and applications. Applications
running on mainframes tend to be "large" both in total volume of data
and transaction rate. Related program libraries and data files are
typically spread over many different volumes for reasons of both
performance and size.

Mainframe, I/O controllers, DASD units are all individually powered in
mainframe world and each typically has multiple internal power
boundaries which can be used to isolate subsections of the unit during a
repair process.

The most current IBM drive architecture is 3390, which comes in several
model flavors. The latest implementations of this architecture emulate
real 3390's using RAID-5 with IBM 4.5GB or 9GB 3.5in drives under the
covers (RAMAC 2, RAMAC 3, RAMAC RVA), which makes it very hard to
describe performance except in context of specific applications.

For the older native 3390 Mod 3 drives:
56664 max bytes per track (depends on choice of physical block size)
15 tracks/cylinder
2.838 GB/logical device (or volume), GB=10**9
32 logical devices/string (181.6 GB @ 7.5GB/sq-ft floor space)

avg seek time: 15.0 ms
avg latency: 7.1 ms
full rotation: 14.1 ms
data transfer rate: 4.2 MB/sec (MB=10**6)
concurrent string-to-controller data transfers: 4
This means that a minimal random physical I/O operation to a device
would be on the order of 15.0 + 7.1 + data transfer, say on the order of
25 msec for less than 32K of data, but depending on the "channel
program" used to drive the device (operating system and application
dependent), that data transfer could involve up to a full cylinder of
data. Access time as perceived by the mainframe is better than this
because the controller has considerable (64M to 256M or larger) read
cache and nonvolatile write cache which allows many I/O requests to be
satisfied without waiting on physical I/O (the 4 paths between
controller and mainframe are 10-17 MB/sec paths).

The later RAMAC implementations have cache at the controller, "drawer"
(a RAID-5 array emulating 4 3390-3 units), and physical drive levels.
The cache is "sized" to give statistically better performance than
native 3390 DASD, but one lacks the physical details to predict expected
response for individual I/O requests.
Typical configurations for RAMAC-2 would be:
60 logical devices/string (170 GB @ approx 20 GB/sq-ft)
In our environment, one RAMAC-2 string and associated controller is able
to sustain I/O rates of 400-600 I/O requests/sec.

› if you have a mainframe that must be up 24 hours a day, how do you go
› about changing the hard drives?

We aren't a 24-hour shop.

When we had 3380s, if we had a disk crash, we would be down until the drive got
replaced and restored from backup tapes. Some shops have disk controllers that
do mirroring, so when a disk crashed, they could switch over to the mirror.

Now we have a RAID II box (RAID 5 technology), which writes strips fo data
across three drives, along with "parity" information, so that if any one of the
three drives goes bad, all the data can be recreated from the other two.
Furthermore, we have a hot swap drawer so that if a problem is detected with a
drive, the information in that drawer is copied to the hot spare drawer and
that becomes addressed by the built-in controller instead of the drawer with
the damaged disk. The IBM Customer Engineer can come in, change the disk,
format it, run diagnostics, and then tell the box to copy the data back from
the hot spare drawer to the drawer where the data normally resides--zero down
time!

I had a demonstration of that while in the initial process of installing
VSE/ESA 2.2.2, and I continued the install process while our IBM C.E. did
repairs on the drawer where the system was being installed, and I continued the
process while the data was being copied back from the hot spare drawer back to
the original data drawer, and I did not notice one bit of degregation during
the entire process.
