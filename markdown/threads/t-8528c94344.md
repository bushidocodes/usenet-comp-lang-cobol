```
The new 3-5-6 version of CobolTransformer tools and 
converters is released. It has the following
new features and improvements:

* data2cr: Crystal Reports Reader for Cobol Data Files.
After you install the data2cr software, your 
Seagate Software Crystal Reports report generator
will be able to read Fujitsu and Micro Focus 
Cobol data files directly.

Just imagine: you will be generating elegant and informative 
reports from your Cobol data files directly, using 
powerful graphical user interface of Crystal Reports, 
instead of tediously programming report generation
directly in Cobol language.

For more info look at http://www.siber.com/sct/datafile/

* Data Reader for other Cobol data formats.

Feel free to send us your Cobol data file, so that we can 
add reader for your format to the Data Reader library.
We are interested in seeing data files of Ryan McFarland Cobol,
AcuCobol, and IBM mainframe Cobols -- we want DataReader
to read these formats too.

* You can use repeaters such as X(9) or 9(8) in picture strings 
in data item descriptors in RDD files now. Originally RDD files
were intended to be used only as intermediate format generated 
by cbl2fdd program from the Cobol source. But several users
who did not have the program source and yet need to read the data,
chose to compose RDD files themselves. When you do this, having
repeaters and not just unrolled picture string is convenient.


Future versions
---------------

This improvements are planed for version to be released
in the next 2-3 weeks. They are being implemented and 
tested as you read this message.

* Date and Time recognition in record items.
You will be able to specify special DateTime picture
string that will tell DataReader that certain record 
item should be interpreted as date/time and will specify
format of date/time.

Once data item is interpreted as date/time, you will be
able to reformat it to your liking using data output
facilities such as Crystal Reports generator.

* EBCDIC <-> ASCII translation.
Cobol Data File reader will be able to perform
EBCDIC <-> ASCII translation, so that mainframe
Cobol data files can be read.

Best regards,
Vadim Maslov
```

```
If you are using Microsoft Outlook Express, that happens because of the
"character set" used on the message. Vadim's post includes this option in
the message header (open/properties): charset=koi8-r.

Go to "Tools/Options/Read/Fonts" to set your viewing font based on the
character set in the message header. If this charset is not in the list
(it's not in mine), I think Outlook Express just switches to this charset
and forgets to switch back unless it gets a message to read with a "charset"
option which IS in its list.

The real solution is to ask eveyone who posts to only use a "charset" option
which is supported by Outlook Express.
```
