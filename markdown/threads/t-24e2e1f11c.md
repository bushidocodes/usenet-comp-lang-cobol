> I am new to cobol . I just wanted some information how to convert
> cobol output to text file. I will appretiate if anyone can help me

> Thanks

This smells a bit like homework. So a very broad outline - check your
Reference Manuals or on-line help for the appropriate VERBS and then go
figure.

You already have your input file specified; perhaps some of the fields
are stored as comp-3 etc. Take same format and expand it in Working
Storage. Put commas between each field :-

01 MyInputRecord.
  05 AccountNumber        pic x(05).
  05 Name                        pic x(30).
  05 etc....
  05 Balance                      pic s9(06)v9(02) comp-3.

01 MyWorkRecord.
  05 AccountNumber        pic x(05).
  05 pic x                         value ",".
  05 Name                        pic x(30)
  05 pic x                         value ",".
  05 etc....
  05 pic x                         value ",".
  05 Balance                     pic -999999.99.
  05 pic x                         value ",".

Your compiler might reject :-

  05 pic x                         value ",".

so, substitute :-

  05 Filler pic x                         value ",".

Note that the field names are identical for both files - so that MOVE
CORRESPONDING could be used. If you find that you have to MOVE
field-by-field, you could write :-

MOVE AccountNumber of MyInputRecord to AccountNumber of MyOutputRecord

- but that is a real pain in the butt. Better to have :-

MOVE I-AccountNumber to O-AccountNumber.
MOVE I-.......................... to O-............. etc.

I'm assuming you are using a compiler which allows for Line Sequential -
repeat the above format used for MyWorkRecord for MyOutputRecord. The
commas in the output file are not legit - but it wont matter because
they are put there by moving the W/S record to the O/P record.

Check Level 88's which I'm using below for More and NoMore Records.

set MoreRecords to true
perform until NoMoreRecords
 read inputfile next record
    at end set NoMoreRecords to true
    not at end
            MOVE CORRESPONDING (can be written MOVE CORR)
                MyInputRecord to MyWorkRecord
            write MyOutPutRecord from MyWorkRecord
end-read
end-perform

Still a fair bit of work for you to do - depending upon your input file
and how binary fields (comp-3's etc) appear, you may have to move some
packed fields to a usage display (pic -999999.99), in Working Storage.
In which case, you MOVE field by field. Check out MOVE CORRESPONDING
rules so that you get the 'desired' interpretation of binaries when
MOVE(d) to a usage display..

That's the guts, and of course there are other ways of doing this - now
go to it !.

Jimmy

Yes, I remember.  And some would argue Mr. Prakken was correct.

On the evening of Friday, December 31st, 1999, I was working from 8:00
PM to 4:00 AM.  I actually got to go home at 3:00 AM because I had a
terrible cold and the "crisis" was well contained.  But we did have an
actual Y2K problem that occurred at 5:00 PM.  It was a weekly job with
a compiled EasyTrieve program that compared the date of the last
update tape (99359) to the date for the current week's update tape
(00001) and rejected the new tape as already processed.  

We missed the EasyTrieve program because we only scanned COBOL
programs for Y2K.  As I recall, we overrode the job to execute a test
version of the Easytrieve program that omitted the date check
completely.  In 2002 we had to recompile the EZT program again for
compatibilty with LE/MVS "ALL31(ON)" and the new Easytrieve runtime
library.  I guess it might break again in 2099, if it's still running.

If it had been written in Cobalt, I would have fixed it permanently.

Here's one of my favorite COBOL Y2K bugs:

Given:

01  MY-DATE.
    05  MY-DATE-YY   PIC 99.
    05  MY-DATE-MM   PIC 99.
    05  MY-DATE-DD   PIC 99.

And...

MOVE 991231 TO MY-DATE.

And, later...

IF MY-DATE-YY + 1 EQUAL ZERO
    PERFORM SPECIAL-1999-LOGIC
ELSE
    PERFORM NORMAL-DATE-LOGIC.

It turns out that SPECIAL-1999-LOGIC can never be executed, because
"IF MY-DATE-YY + 1 EQUAL ZERO" is *always* false, for a very subtle
reason.  Easy to fix, but extremely difficult to diagnose.

Another favorite bug got caught in unit testing, when a junior
programmer coded something like the following:

IF MY-DATE-YY IS LESS THAN 50
    MOVE 19 TO MY-DATE-CC
ELSE
    MOVE 20 TO MY-DATE-CC.

In his unit testing, all the transaction log records were date-stamped
with the year as 2098!

> What was the reason for complicating up that IF clause?

That's a very good question!  Unfortunately, I don't have the answer. 
It could have been that there was a desire to preserve the contents of
MY-DATE-YY, or it could have been that it was inserted into existing
logic and it actually read IF MY-DATE-YY + 1 EQUAL LITERAL-YEAR-00.

In any event, the problem was that, while MY-DATE-YY had a picture
clause of 99, the expression "MY-DATE-YY + 1" generated an
intermediate work area with more digits, which was then compared to
Zero.  So 99 + 1 equals 100, which is certainly NOT equal to zero.

I was never taught that arithmetic expressions could be used in IF
statements, so the first time I saw a similar example I wondered if it
was valid COBOL.

Back to your original question.  I don't see any good reason for
complicating up the IF clause.  It would have been much simpler to say
"IF MY-DATE-YY EQUAL 99".  Another alternative would have been to
compute, say, WORK-YY as MY-DATE-YY + 1, which could coerce a
two-digit result, and then compare that to zero.
