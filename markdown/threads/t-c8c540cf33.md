Hi folks

I've just returned to mainframe (IBM) programming after a number of years
absence and I have a COBOL problem that is probably very simple but is
driving me crazy.

A third party supplier transmits a file to us every day.  The file is
sequential but has varying length records.  Not a problem - even for me,
however, the header record contains a number of totals fields for the number
of each record type following in the file. Total-records, total-type-a,
total-type-b, etc.

I have a program that strips out certain records and I keep up-to-date
totals for these and totals for records excluded.  I then have to tag the
other totals to the end of the header record - increasing the length of the
header record (by over 100 bytes).

To do this I have to process the file until eof (excluding records along the
way), close the file, open it again, read the header, update (and add)
totals, REwrite the file, and close.  I can't sort the file first (it would
destroy the record relationships).

Everything up to and including reading the header is OK, but when I rewrite
the file I get an error code '44' - I've looked up the COBOL manual and
whilst I understand what it's telling me, I can't figure out a way around
it.  Plus the program seems to be writing a spurious record after the header
(it's hex'00') for the whole record - I'm assuming this is something to do
with my other difficulty.

Any help from a seasoned COBOL head would be greatly appreciated - and make
tomorrow easier.

TIA
Fergus

Next time I'm taking a PL/1 contract!!

Description of 44:

A boundary violation exists because an attempt was
made to rewrite a record to a file and the record 
was not the same size as the record being         
replaced, or an attempt was made to write or      
rewrite a record that was larger than the largest 
or smaller than the smallest record allowed by the
RECORD IS VARYING clause of the associated        
file-name.                                       

You have several choices:

1. Have the third party pad the first record so you can fit your data in.

2. Find some data in the first record that you are not using and place your
statistics there.

3. Create a new file.

There are several different ways you could handle this.  The 44 error on the
REWRITE makes an update in place untenable.

The first option is to simply write another file, with the same DCB as the
original file, but opened for OUTPUT instead of I-O.  If you're using GDGs
this may be the easiest way.

Sorting the file is not a great difficulty either.  The simplest way would
be to stick a sequential number on the front of each record, do your
processing, and then sort on the sequential number.  The data would come
back the same way it was originally read.

Pseudocode time....

(a) process file, stripping out the un-needed records and maintaining your
counts.

(b) create your sequential number for your sort key as each record is
processed.

(c) at EOF, update your header, and release it to the sort with the updated
counts

(d) return the sort file

(e) strip off the sort key

(f) write the record to the output file

Fergus,

As you are an old hand I don't want to teach grandmother to suck eggs - but just
in case you missed the subtlety here - the business of different record sizes.
It's a problem I came up against a couple of months back, and both Bill Klein and
Pete Dashwood helped me through it.

In OO I wanted a generic class to handle Line Sequential (but same concept as
Sequential). So I have this 'common' program which says "record varying in size
from a to b depending upon c" All my Line Sequentials are in fact FIXED length
but to use that common program I have to tell it is getting "a varying depending
on" - and I pass the size factor to the common program before I start processing
for EACH Line Sequential I'm using.

Here's an interesting one - I feed Ascii-delimiteds(CSVs)  to it and they are
FIXED length, say 100 chars, 248 chars etc. Didn't work ! Because the compiler
picks up on the fact that the Line Sequentials are FIXED length and doesn't 'buy'
my occurs varying. So initially to get CSVs into the system had to have SEPARATE
programs to accept the FIXED length CSVs. Once into the application then my 'own'
files can use the occurs varying.

Take your input and output - Let's assume your header record is 100 chars, but
then you are adding stuff to it for output - is this a fixed amount or varying ?.
(Can you specify the maximum size it will grow to). You can't use the same syntax
:-

    Input file - varying from a to b depending on c( c = 100)
    Output file - varying from a to b depending on c( c = 100)

You must have optimum  size for output file; so the second entry above becomes :-

    Output file - varying from a to b depending on c ( where c = 150,  2,000,
                                                        2,500, 3,000  etc.)

Above may be off track - but just in case.....
