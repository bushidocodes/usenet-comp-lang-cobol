There is / are 2 issues here?

1- most important I bet, the file is improperly defined with respect to
CA/CI.
    If you are getting that many splits, it should be defined  with a 20 -
40 percent
HIGHER CA/CI level.  Check the IDCAMS manual on how to establish this
better.

2 - the ALTERNATE INDEXES are probaly set to 'UPGRADE' which means
that the DMS updates the indexes after every operation that affects them.

What the 'MASS INSERT' in DB2 does is causes DB2 to NOT update its indexes
UNTIL you end that Operation.  This saves time as it can 'batch' the Index
updates.
COBOL has no such function / language capability.

  If a large percentage of the file (> 5%) is being replaced or
updated,then Jeff's suggestion of using a sorted sequential file,
merging the updates, and rebuilding the VSAM file may be best.
  If this is an initial load of the VSAM file, sorting the records in
key order and loading the file sequentially rather than randomly is the
only efficient way to go.
  If you are talking about other cases where random updates make sense,
then the most crucial thing (other than tuning mentioned below) is to
minimize CA-splits, which you can do by specifying a significant
percentage of CA Freespace when you define the file with IDCAMS.  If the
updates tend to be uniformly distributed throughout the file, you may
also benefit from CI Freespace as well, but if inserts are clustered at
only a few points, CI Freespace will consume much space throughout the
file with only minimal benefit.  If the file is undergoing such update
activity on a regular basis, it also needs to be reorganized on a
regular basis. 
  Unless you are using extended format files with with system determined
buffering (available at DFSMS 1.4(?) and above), the single greatest
thing you can do for batch VSAM performance is buffer tuning by
specifying BUFNI and BUFND parameters on the DD statement.  If you are
taking the default values you are GUARANTEED to be doing at least 3
times the amount of physical I/O you should be when doing random access
to a KSDS VSAM file.  The absolute minimum you should use for BUFNI for
random I/O is the number of index levels for the KSDS, which is at least
2 for any nontrivial file with more than 1 CA (system default for BUFNI
is 1).  If you can afford the buffer space, you may even want to go as
high as BUFNI = total number of records in the KDSD INDEX component.  If
you are talking about a batch run which still performs > 100,000 EXCPs
on the file after BUFNI/BUFND tuning, you may want to consider using
Batch LSR (BLSR), which though JCL changes allows for efficient
management of 100's of index and data buffers, deferred writes, etc.
which can also greatly improve random update activity.

There are 2 other possible solutions, in addition to the ones already
posted.

1) If you can, process your input in descending key sequence this may reduce
your CA splits.
                                      -- OR --
2) If you are processing randomly in ascending key sequence,
a) change access to sequential
b) open file in I-O mode (DO NOT use OUTPUT, the REUSE option may wipe out
your existing data if you do).

I think I will suggest that you listen to Jeff. You are saying 'handling
high #'. Please study the nature of VSAM files and especially the impact of
sizing CA/CI when you define the VSAM files but go for Jeff's suggestion.

Now a story about how a bad design can destroy a VSAM file.

Many years ago, when I was young(er) I made a smashing design. Users in a
CICS system could order batch-jobs and that was rather unusual then. In
fact, they could make dayly, monthly orders and, of course, they could
choose a certain run-date in the future. The great idea was to replace the
month and year by all '9' when the order was monthly. Orders for dayly batch
were 9999.99.99.

Every day we took orders for today and sent the job ordered to the internal
reader. No operators needed no job-flow setup; everything was in the
ordering system. The design had just a minor disadvantage - it dumped
frequently, until we changed the primary key.

The dayly batch was 95% of all the orderes and of course VSAM tried to put
all the 9999.99.99 into the same CA.

The story shows, that if the keys are very similar, you will never get good
performance. You will get the best performance out of VSAM if the keys are
very different. I am a lazy person, so I simply took the time and placed in
front of the run date to spread the orders into diferent CA's

Hope that it helps to laugh at one of the oldies.

Regards
Ib
