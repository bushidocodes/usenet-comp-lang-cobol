Before I retired in August of 2016 our development shop converted from
Z/OS COBOL 4.2 to 5.1, which also required storing load modules in
PDSE's rather than PDS's. This was quite controversial across the IBM
customer base, with some companies refusing to make the change due to
the perceived weaknesses in PDSE support and difficulty converting.

Without going into too much detail, upgrading from PDS to PDSE for
storing load modules was critical for supporting new features in the
compiler and debugger, and not upgrading meant running an unsupported
COBOL compiler. Conversion is tricky and there are quite a few
technical issues. For example, PDSE support doesn't come up until a
little later in the IPL sequence. It's harder to share PDSE's across
CPU's. Copying a COBOL load module from PDS to PDSE requires converting
the load module file format, which happens automatically with certain
utility software but not with others. Load modules cannot be copied
from PDSE to PDS, they must instead be compiled with the out of date
compiler.

The applications programmers provided some input, but this work was
entirely driven by Tech Services/system programmers. So I completely
agree with you about requiring five years IBM Mainframe experience.

Obviously, if you are using COBOL the source code will include an FD and
01 level description of the data (as seen by the program) and it is easy
to create a Database table based on this (if you don't have the actual
DB table any longer...)

If you are using C# (for example) you will access a database through an
interface or an ORM (Object Relational Mapping). This describes the data
just as the COBOL definition does and, similarly, you can deduce or
re-create the actual database table structure using it.

(In fact, for C# you can define whatever data structure you want as a
collection with an IEnumerable property, and then use SQL through LINQ
to access it.)

There is nothing special about having the definition of the data
included in your program and, in fact, there are very good arguments for
NOT having it in your program, but accessed through an interface...

As far as self-documentation is concerned, this is a fable constructed
by IBM salesmen in the 60s and 70s who were trying to persuade bean
counters that they could "read" the computer code just like English...

The famous example which appeared in Datamation back in the 60s was
along the lines of:

02 flt-ctr REDEFINES zx PIC s9(13) comp-3 OCCURS 27 INDEXED by zog.

Show this to an English-speaking bean counter and see what he makes of
it... (or a programmer NOT trained in COBOL for that matter...)

ANY language that is properly commented (and some languages make this
easier than COBOL) can be maintained just as easily as COBOL and without
the verbosity.

None of the above is a reflection on COBOL. Rather, I am seeking to show
that some of the believed advantages of COBOL are simply urban myths.

In fifteen years time we don't know how important manually written code
will be... Advances in AI may have specialized programming applications
writing and maintaining it (there are already moves in this direction...).

For myself, I'll be unlikely to see it, but the point is that there are
much more serious things to worry about than whether languages other
than COBOL can be used into the future.

Pete.
