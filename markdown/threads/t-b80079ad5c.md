The problem is that because of certain COBOL semantics (like, returning locked
record contents along with the locked record status) the COBOL RTS cannot use
OS locking on the physical parts of the file containing the records. Therefore
we must use a different strategy in our modified C-ISAM and it is this
differing locking stragegy that will give problems when the same file is being
accessed simultaneously with a third party C-ISAM.
This is why we allow a third party C-ISAM to be linked into our RTS - you
simply lose certain COBOL semantics (and also certain enhancements both to
the feature set and to the limits imposed by CISAM). A full list is documented
in the Micro Focus COBOL System Reference.

›› but it would be great if I can make COBOL add a .dat extension or make
›› C-ISAM not check for the .dat extension.

The problem then is that (I would imagine) opening "x.dat" will give you
"x.dat/X.dat.idx" from COBOL and "x.dat.dat/x.dat.idx" from C-ISAM, unless
they BOTH see the .dat and are intelligent about it.

I think that the way to do it will *probably* be the dd_ environment variable
method of mapping data/index names (the dd_ variable contents start with an
ampersand, I believe). I`m afraid this isn`t something I'm overly familiar
with, so you'd have to check the docs for the full description.
