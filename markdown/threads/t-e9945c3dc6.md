The problem is that of writing vs. maintaining the programs.  It's easy to write
them that way but the very devil to maintain.  Given that proper documentation
is never prepared, it becomes a marathon of frustration and wasted effort having
to look at another program, then another, then another - just to find that
you're off down the wrong tree.  Then you have to remember where you started and
begin again.  At least if all the modules are within the same program - i.e.,
performed - you don't have to go as far back to start over.

There are definitely cases where programs should be in the form of stand-alone
called modules - but they are all of the utility or frequently-used function
type.  And there are cases where one program becomes too "huge" for varying
reasons.  But IMnot so HO, you have to decide which is more important in the
long run - ease of writing or ease of maintenance.

I didn't see the beginning of this thread - but some "thoughts" (no
DEFINITIVE pro or con)

1) The "theory" behind nested programs has to do with "localizing" data to
the procedural code that uses it.  (Similar - but not quite the same as
"encapsulation" with OO).  There is certainly SOME evidence that having data
associated with the procedures that act upon it - is a "good thing"
(especially for maintenance and security).

2) Lots of people who USE nested programs disagree with this, but I am a
STRONG believer in using "COPYBOOKS" (copy members) to define nested
programs.  This "insures" that all "main" programs that use the same nested
"routine" get the same version.  (Of course you need to do a RECOMPILE -
unlike separately compiled programs - to "bring in" the revised version of
the nested program.  But this is (IMHO) easier than trying to keep source
code "in sync")

3) Run-time performance of nested programs versus PERFORMED procedures
versus CALLs to separately compiled programs is TOTALLY dependent upon the
compiler (and run-time).  IBM, for example explicitly recommends nested
programs over separately compiled programs *IF* run-time performance is of
highest concern.  On the other hand, Micro Focus has recommended
"nonestcall" (If I recall correctly) for compile-time (and I think run-time)
performance.

4) It is important that programmers using (AND maintaining) applications
with nested programs are adequately trained.  This DOES include
understanding both EXTERNAL and GLOBAL data definitions - and the
differences between them.  Similarly, you need to learn about COMMON
attribute on the program-id paragraph.

5) Check for 3rd party (and compiler) tools - and their support for nested
programs.  In an IBM mainframe environment (for example) there USED to be
restrictions on DB2, CICS, and debuggers such as Xpediter.  I think that
MOST of these have been removed (by now) - but you should check if "YOUR"
release supports them.

6) Nested programs have been available in ANSI/ISO conforming compilers for
over 15 years now.  If your site is NOT using them, then the chances are
that you aren't using some/many of the OTHER enhancements that have been
available for a decade and a half (and the next Standard is due out
"relatively soon").  You need to SERIOUSLY look at your use of COBOL.  If
you are using it ONLY to maintain existing applications - with a
"diminishing" staff, then you are probably "safe" in sticking to what worked
20 years ago.  If, however, you are doing new development (or even major
enhancements), then I think you need to have "management" look at keeping
your COBOL skills current.

Bottom-Line:
  I know of NOTHING that can be done with nested programs that can't be done
some other way.  HOWEVER, whether nested programs will enhance both your
application's performance AND its maintainability (with adequately trained
staff) is something that should be seriously looked at - and might well
depend on both your programming environment and use of COBOL.

P.S.  If you tell us your "environment" (compiler, operating system, etc) -
we might have some other comments.

The question of whether nested programs (not separately compiled programs)
are or are not "more difficult" to maintain than PERFORMed "procedures"
(paragraphs or sections) is DEBATABLE.  If you use "identical" nested
programs in multiple "main" programs, then they do cause difficulty in
maintenance - as you need to "figure out" which main programs use the
routine and how they will be impacted by any changes.  Futhermore, unlike
separately compiled programs (especially those that are "dynamically
called" - in an OS/390 environment), you MUST recompile every program that
that "uses" the same nested program.  (Comments can help - tools such as
SCLM or Endevor can also help - but nothing is "fool proof")

The theory of having "data near procedures" is certainly one that is "touted
as beneficial" by many programming language theorists.  I have no evidence
that it isn't helpful - but I also have no evidence that it IS helpful.
There are a NUMBER of 3rd party products (again in an OS/390 environment)
that can HAPPILY trace data thru PERFORM statements; there are significantly
less (possibly none?) that can do the same thru nested programs.  Therefore
"program/logic understanding" of an UNFAMILIAR program *may* suffer when
using nested programs rather than PERFORMed routines.  On the other hand, if
you "goal" is to have a black box "routine" (handled by such things as COPY
SUPRESSING) then a nested program might well be better than using a
PERFORMed routine.

One thing that I do (still) think is critical - is that you MUST have
adequate training to use nested programs (or any other feature that is "new"
in the last 15 years).  Furthermore, it is important that all your "add on"
tools work with whichever feature you are "introducing" to your environment.

No, it's not the same thing. 

In COBOL, nested programs are subroutines where the source is in the same text
file for the callers and the callees. In earlier COBOL, and many other
languages, subroutines are separately compiled and managed. This is why the
whole maintenance issue raises its head.

In PL/I, procedures may be external or internal; if they are internal, they are
like nested programs. But PL/I does not have the analogous construct of
performed paragraphs and sections.

For a traditional COBOL programmer, the need for the nested program concept
seems strange indeed. You have internal routines that you PERFORM (paragraphs
or sections) and you have external subroutines that you CALL.

The PERFORM statement cannot pass parameters, but the CALL can. PERFORMed
routines must be in the same source file; CALL'ed routines (before the advent
of nested programs) must be in a separately compiled source file. Nested
programs are a strange amalgam of the two. 

But it's not a big deal, per se. Good programmers can write good code with our
without nested programs; bad programmers will create headaches whatever they
choose.

Identification Division - header

your nested program.

program' is pretty USELESS - but it is "conforming".  (Possibly of SOME use
as a "stub" program before developing the "intended" code for an existing
CALL statement.

in a main program as well.  However, in that case the "assumed Exit Program"
doesn't work as PROBABLY desired;  such a program may, however, be compiled
as a "separately" compiled program that could be called by an "external"

Now finally that's the gist of it. Use separately compiled subroutines
whenever reasonable, especially when other programs might find them
useful. Use internal nested local routines only for stuff unique to the
program, to emulate a Perform Using, so that it's not as cumbersome as a
regular Perform and it can have its own local variables and accept
parameters. Use performs for the simple internal calls that don't need
parameter passing or local variables.

Generally, only data definitions e.g. record layouts, not executable code,
should be in copybooks. Exceptions might be standardized interfaces, used
macro-style, e.g. with copy replacing, and even then the code in the
copybook should be as little as possible and if it needs to do anything
major then it should call a subroutine. If an internal nested routine
becomes popular enough to tempt anyone to put it in a copybook, or
otherwise to have it called/included by more than one program, it should
be converted into an externally compiled subroutine. If it badly needs to
access global variables, change it to a parameter so it can be a separate
subroutine. Use a copybook to simplify the interface as above if needed.

to establish addressability for pointer manipulation.  For instance, if you
have a program that wants to SET ADDRESS OF something TO something-else,

subroutine, then you can easily make it one by coding a very simple outer
'shell' program that does nothing but call your program, passing "something"
as a parameter.   You don't *have* to code the 'shell' program as an outer
program, but in this case it makes sense because the 'shell' and your
