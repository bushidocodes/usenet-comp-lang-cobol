(Different people will probably answer different parts of this note)

A) If at all possible, when posting to comp.lang.cobol - use "Plain Text"
and NOT HTML (even if you can't get "pretty" red lines that way).

B) My guess is that you are running on an IBM mainframe.  The following
assumes that is true - if it isn't, then tell us what compiler and operating
system (and printers) you are using.

If you use the ADV compile option, then it is correct that you ONLY define
132 characters in your FD - and COBOL "puts" in the other byte.  You still
need to define your DCB for your output file as LRECL=133 *and* RECFM=FBA
(or FA or FBM or FM) - something to put that "control" character into.

If you use the NOADV compiler option, then you need to define 133 characters
under your FD and you have to "fill in" that 1st character yourself (and not
use the AFTER ADVANCING stuff).

C) Check your programming guide and/or LRM for a full list of file status
codes.  There are "lots" of useful information that you might want to get
from them.  (Again, if you are on an IBM mainframe, there is even a 2nd
status code field for VSAM files with even MORE information in it.)

Some programmers check each possible file status code - others use "AT END"
or "INVALID KEY" phrases still others (fairly rare) use DECLARATIVES.  There
are PROs and CONs for each (and a "religious war" can occur when you try to
convince someone to switch from one to the other).  However, I think that
most people will agree that you should NOT "mix" the various techniques.
OTOH, some people use a "phrase" to catch something other than "00" and then
do detail checking of various serious and semi-serious problems reflected by
other values.

D) Lots (most?) programmers do try to "centralise" their file I/O - and then
perform such routines (either a PARAGRAPH or a SECTION - but that is another
"religious war").  In older compilers this (sometimes) had a negative
performance impact - but often made program maintenance much easier and less
error-prone.

Not quite true...

If you do not use advancing at all on a file then the ADV command
has no effect, if you define 133 then it writes 133.  The effect
only kicks in when you use the advancing commands (and it knows
about it).

If you use NOADV then you are (effectively) required to define 133
and the first byte is reserved.  You can put it in but if you
combine with advancing then the values you put in are lost. 
Probably good advice by Bill but not required as implied.

Adding to the discussion:

1  = skip to channel 1 (always top of page except sometimes)
+  = don't skip any lines (overstrike, bold, underscore)
' '= skip a single line (normal single line feed)
0  = skip two lines then print
-  = skip three lines then print.

When you look at it don't be surprised that your blank lines just
do not appear.

Ken
