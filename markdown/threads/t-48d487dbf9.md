```
I am working my way through Thane Hubbell's TY COBOL book in
preparation for using it in the classroom in a couple of
weeks.  Chapter 14 raised a question (for me anyway) that I
do not know the answer to:  When working with indexed files
using both a primary and a secondary key, neither of which
allows duplicates, is there anyway to determine which key
generated a "duplicate key" error?  

It would seem useful to have separate error handling
procedures depending on which key generated the error (e.g.
one might print an error message to a log file, while the
other might ask the user to re-enter the data) but this
could only be done if one can detect the key causing the
error. 

Thanks for any feedback.

Floyd Johnson
```

```
File Status "22" . Thought about it a little more after supper, and
here's what I am actually doing. Principle applies to PCs/Mainframes :-

Inter-active Windowing system :

- select record from ListBox(get Primekey) or user enters
  Primekey
- set RecordNotFound and read Primekey
- RecordFound display/change; NotFound enter data/change
- OK update :
- Read using AlternateKey - if AltKeyFound display "Record already
  exists with this name......" - back to screen.
- AltKeyNotFound - RecordFound rewrite; RecordNotFound write new record

"Playing it blind" - input from cards/tape/OCR/OMR or an ascii-delimited
from a database etc. :-

- read next InRecord until FileFinished 
- read i-o DataFile using AltKey - AltKeyRecordFound writer error-line
  ignore InRecord
- read i-o DataFile using PrimeKey - RecordFound write error-line
  ignore InRecord else write record
```

```
There have been lots of ideas and thoughts related to this thread - but I
thought I would had a couple of "inputs":

1) For those who commented on the fact that you can only have duplicates for
alternate keys - not for primary keys - that is true (according to the
Standard) *but*
  A) Some compilers have extensions that provide for duplicate primary keys
  B) It really is irrelevant to this question - as the problem is when
NEITHER the primary nor the alternate allow duplicates
  C) It is true that you DO get a FS=22 when EITHER the primary or the
alternate would cause a duplicate (and when duplicates are not allowed for
the alternate).

2) (I could be wrong on this) - but I think that whoever thought that you
"always know" which key you are dealing with is not right.  Lots of I/O
verbs do require this - but WRITE isn't one of them.

3)  OK - now for my suggestion to "solve" the problem:

If you open the INDEXED file
    a) as I-O and *not* as OUTPUT
       *and*
    b) you are in ACCESS RANDOM or DYNAMIC
then you can always do a READ on either key (or both keys) after the FS=22
and find out which one already exists.  (You probably would need to try
BOTH, because a FS=22 could happen if BOTH the primary and the alternate are
duplicates - and your logic should handle this too)

Having said that, I know of a number of compilers and file systems where
"loading" a file is MUCH more efficient when you OPEN OUTPUT and have ACCESS
SEQUENTIAL.   So taking this approach, might be a performance issue in a
"real" application.

4) Finally, I have passed this "question" on to J4, because I think that it
would be a "nice" enhancement in a future Standard (not the one that is
currently in draft) if you COULD figure out which key (or keys) get the
FS=22 when you do open a file OUTPUT.
```

```
I can't think of any system where several unique alternate keys
where they aren't unique deliberately (and thus will never
duplicate) would be useful.

Primary keys are unique because they are the customer number or
the product code or similar and their uniqueness is inherent.

If there is a genuine alterante unique way of referring to the
data item that the record represents then an alternate key without
duplicates is valid, however this would be unusual.  I would look
to the design if I saw this situation.

Where there is a need for an alternate key it is usually that
some specific ordering is needed below the 'obvious' sequencing.
In some cases I have repeated the primary key as part of the 
secondary and this does guarantee non-duplication with no
question as to the meaning of a '22'.

But to test which key(s) caused a '22' it is only necessary
to START on each key and check the result (and READ NEXT to
give the user a chance to know why it is a duplication).
```

```
I've  a method of of file design (that I describe in COBOL Unleashed)
where the primary key of a file is a "handle" that is unique and never
changes.  What appears as a unique prime key to a user (Name for
instance), is really a unique alternate key.  If they want to change
the name, I let them, then rewrite (which use prime key) and I get a
22 if that person's name is already in the file with another handle.  

Now, suppose you added Drivers license for instance as a second
(really third key).  This is just one example of useful implementation
with multiple unique alternate keys.  It's easy to handle, and really
makes life simple down the road.  

For example, let's say you want to had a whole lot of detail
information about the individual in a different file.  Simple key by
the mail files "handle" - then "name changes" wihich are apparantly
primary key changes (but really aren't) won't lose the associated
detail information.

I've now used this design with several systems and I am very pleased
with the outcome.
```
