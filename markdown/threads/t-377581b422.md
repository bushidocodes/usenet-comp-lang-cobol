```
One warning about passing tables (often known as "arrays" in other
programming languages).  If you use the INDEXED BY phrase in your CALLing
and CALLed programs, there is no ANSI/ISO requirement that these "reference"
the same index.  Therefore, do NOT set the index in one program and expect
it to be available in the other.  If you DO need to have references to an
"index value" in both the calling and called program, then create a USAGE
INDEX data item (quite different from an INDEXED BY item) and pass it as
well as the table.
```

```
USUALLY* when you pass a parameter what is actually communicated is
the starting memory address of the parameter. Therefore whether the
item is an array or a variable is irrelevant. Also the variable names
used in the two modules is immaterial. ONLY INTERNAL ADDRESSES ARE
PASSED. For example:

Calling program:
01  MYARRAY.
   02  MYSTUFF OCCURS 10  PIC X(10).
...
   CALL 'MYSUB' USING MYARRAY

Called program:
LINKAGE SECTION.
01  PASSED-THINGY        PIC X(100).
PROCEDURE DIVISION USING PASSED-THINGY.

will compile, link, and execute with no problem.

In other words, it makes no difference how a passed item is defined as
far as the compiler/linker is concerned. Of course it makes a world of
difference if you get confused.
The common rule is that the passed parameter be defined exactly the
same in the calling and the called programs.

A common mistake is:

01  MYPARAM  PIC X(10).
CALL 'SUBPROG' USING MYPARAM.

LINKAGE SECTION.
01  MYPARAM  PIC X(1000).
PROCEDURE DIVISION USING MYPARAM.
   MOVE SPACE TO MYPARAM.

Which wipes out 990 bytes of (presumably) good data back in the
calling program, causing all manner of confusing symptoms (such as
"Divide By Zero," "Invalid Function Call," and "Danger, Will
Robinson.")


[*There are ways to pass actual values (instead of addresses), but
don't go down that road yet.]

Question 2: There is no practical limit to the number of parameters
you CAN pass. Anything more than 3 to 6 (depending on the compiler)
take a performance hit. For LOTS of parameters needed by a subprogram,
it is more efficient to collect these disparate parameters into one
group item and pass (the address of) that single group item. For
example:

01 PARAM-LIST.
  02  PARAM-NAME   PIC X(30).
  02  PARAM-DATE  PIC X(8).
  02  PARAM-COLOR  PIC X(4).
  02  PARAM-DOG-BREED  PIC X(45).
  02  PARAM-PRIME-NUMBER  PIC 9(7).
  02  PARAM-QUARTS-IN-A-DECILITER PIC 9(5)V9(5).
  02  PARAM-ATOMIC-WEIGHT  PIC 9(9)V9(9).
...

CALL 'SUBPROG' USING PARAM-LIST.

Remember, how you define the parameters in your program is immaterial
to the passing process (but not irrelevant to your logical process).
```

```
Actually, if all that happens is you wipe out 990 bytes of data, that's the
best that can happen here. (or is it the worst?).

The called program does not necessarily have access to the 990 bytes
following the ten bytes allocated in "caller". Trying to access memory you
don't own gets you a S0C2 (mainframe) or GPF (Windows) real quick.

Compilers do not allocate memory for Linkage section items; they expect the
calling routine to allocate and own it.
```
