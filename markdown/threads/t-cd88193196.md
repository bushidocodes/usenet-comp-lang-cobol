I first encountered this in Java, where Rogers Cadenhead defines it as "a 
group of related classes that serve a common purpose".

Java is built almost entirely around "packages" which contain all the 
classes to do a specific kind of activity. For example, one form of 
graphical interface in Java uses the Graphics class. But this class exists 
inside a "package" of other related classes called the "Abstract Window 
Toolkit". So, if you need to have a graphic display in your Java program you 
would start by importing the class you need from the package:

import java.awt.Graphics

This is fine for simple graphics but if you want something much sexier you 
would probably use Java Foundation Classes and these are "packaged" into the 
java.awt.swing package.

import java.awt.swing

"Swing" as it is commonly referred to, has a number of advantages over the 
standard graphics package, but it is HUGE...(in Java terms). It has 8 
subordinate packages, more than 20 interfaces and over 75 classes. (I like 
Swing because it is comprised entirely of components...) Still, I guess size 
is relative; the latest incarnation of the .NET framework has around 70,000 
classes...it is no wonder that the download is .6 GB compressed.

So, you can see that a "package" IS a convenient way of grouping related 
classes together. Very often, the "package" is implemented as a dynamic link 
library (not usually in Java, but there are other systems that use 
packages).  The difference between a .DLL and a package is more of a 
conceptual one. (A package could contain several .DLLs)

Taking a leaf out of Robert Persig's book ("Zen and the art of motorcycle 
maintenance"), think of it hierarchically, like this:

Package
   Class... (there can be many within a Package)
        Method (there can be many within a Class)
              Function (there can be many within a Method)

As long as you are happy with the understanding of a Function (most 
programmers are...) then this should make sense, even without an OO 
background.

The conceptual bit concerns the implicit relationships between the above 
elements.

A Method defines behaviour and can have disparate unrelated functions in it 
(although often the functions in a method ARE related. Certainly, they are 
all part of achieving the goals of the method. For example, a method might 
contain functions to calculate a square root and to find the length of a 
string . Not obviously related when viewed separately... :-) Put the 
"Method" umbrella over them and suddenly it becomes clear what they are 
trying to achieve.

The Class establishes a higher level set of goals and acts as an "umbrella" 
for the methods, in very much the same way as the Method acted for the 
functions. (The Methods actually define and implement "behaviours" of the 
Class.)

(At this point some of you are thinking "Methods are just procedures; I know 
all about that" and that is where it all goes pear shaped. You are applying 
ITSA when you should be applying ITSLIKE (if you REALLY must apply 
anything... sometimes it is best to cear your mind and pretend you've not 
seen anything before. This is why Newbies pick up OO better than old 
hands... It is the CONCEPTS that make all the difference, although that is 
not immediately obvious when you come to it first up. Most of us try to 
apply and relate things we already know to things we are trying to learn. 
There are times when it is best NOT to do that.)

So, we have a Class that behaves in certain ways which we control. And we 
understand that its behaviours are designed to achieve the goals of the 
Class.

Now, we take an even higher level goal and apply a group of Classes to it. 
They are related in the same way that the methods relate to the Class, and 
the functions relate to each method:  That's a PACKAGE.

Having had a shot at putting the requisite concepts in place, I would offer 
the following as a definition of a PACKAGE:

"A package implicitly defines, and explicitly implements, the highest level 
goals to be achieved by a given process. It is an umbrella for the 
sub-processes (Classes) contained within it and provides a convenient means 
of transporting them across platforms."

have words meaning different things within the discipline of
programming.

different people.   "Package" means specific different things to, say

speaking the same language when the words are the same, and the
dialects are close.

often miss important or even vital parts of our process.

Even if we used made up words (or if we didn't know English),
technical terms used in our business would still quickly acquire
baggage that could mislead us.
