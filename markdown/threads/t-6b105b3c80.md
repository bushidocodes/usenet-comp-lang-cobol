Help Please,

I have a Cobol program dynamically called in CICS  by many users/tasks.

The Program contains a 50k Table (of static parameters)  in
Working-Storage.

It appears to me (& I am not a cics programmer)  that each task/user is
getting a fresh copy of the Working-Storage for this Sub-program on each

call.

The table is clearly in the LOADLIB(loadmodule) and CICS is only doing
one program-fetch for the whole day.
But each/every task is using a lot more memory now.

OBJECTIVE:  What I want is a single central copy of the Sub-program AND
it's working Storage,
i.e. someway to bind the working storage to the program.
So that each/every user isn't using more storage,  all would go thru the

single/central/constant copy.

Q1) I THINK that COBOL may separate the Procedure-Divn  from the
Working-Storage to allow:
one central copy of the Procedure-instructions but
each user to have their own copy of Working-Storage.
Perhaps someone can confirm this is probably whats happening.

Q2) What should I do to acheive the "objective" above.

I suspect the answer may lie in Compile & link options  such as
RENT    REUSE    RES   NOADATA   DYNAM   TERM    etc but this a little
beyond my normal knowledge (& all others I have asked so-far).

Functionally the program is working 100% as wanted (except each/every
user uses more storage!).

Can anybody help please.
Regards
Phil Davey

====================
Other Supporting Information
====================

Current COMPILE Options:  NOADATA,  ADV, APOST, DATA(24), NOOPTIMIZE,
RENT, RMODE(AUTO) NOTERM,  NODYNAM, FLAG(I,W)
(there is no mention of RES,  and NoDynam is apparently not now needed
for dynamicly called routines).

Current LINK Options: LIST, XREF, SIZE=(2048K,512K), RMODE=ANY, AMODE=31

We are using IBM  COBOL-for-MVS.

The Sub-Program is not really a CICS program (we don't do a cics
precompile) - it makes no CICS calls and no other CALLS or I/O.

All our programs are running "below-the-line" - we are looking
separately at this with the hope that we can operate above the line -
but I don't know if this is going to change things a lot ???

---o0o---

Phil,

    You can't do much about the Working-Storage. That's just the way it
works. What you're attempting to gain is 100 percent re-entrant code. COBOL
uses the terminology serially-reusable which just about does the same thing.
I think you're OK with that.

    Have you considered a Static-Assembler Table that gets addressed by your
COBOL/CICS Program via the LOAD command. Basically, you'd be addressing
storage that is EXTERNAL to your program. Whenever a change occurs to the
Table, just re-assemble it and off you go. It would certainly reduce task
storage significantly.

    You'll need to define a PPT Entry for your table. You may also want to
discuss with your Sys Prog about the possibility of defining it as
Core-Resident. This will ensure that it won't get paged-out, as it will be
placed in a non-pageable area of DSA or EDSA, depending on your AMODE.
However, if you feel that it's usage will be significant enough that CICS
won't page it out during his periodic compressions, then Core-Residency may
not be required.

    You'll need to define your DSECT (COBOL copybook) in the LINKAGE SECTION
of your program after DFHCOMMAREA. You can then establish addressability to
it via the LOAD command. Discuss with your Sys Prog about the Core-Resident
definition and possibility of loading the address of the table into a 4-byte
fullword in the CWA. By doing this, your program issues an ADDRESS CWA
command and you obtain the table address from the CWA. This method would
require a 3rd-Stage PLT/PI Program that would establish table addressability
and populate the CWA fullword at CICS Startup.

    Now, after having said all this, you may have to re-consider this design
if you are planning on migrating to a SYSPLEX (minimum CICS Version/Release
is 4.1.0 for MVS) environment, where CWA's and Assembler Tables are NOT in
the cards, in which case, a CICS User-Data Table (a VSAM File) may be your
answer. I won't go in to it too much, but let me say that SYSPLEX has
serious ramifications for users of TWA, CWA, Core-Resident Programs, GETMAIN
SHARED, the passing of addresses and several other CICS areas and commands
used by thousands of current systems. IBM has espoused COMMAREAS as the
means to pass data around and DPL (Distributed Program Link) to centralize
high-usage modules for all regions. Also, please remember that IBM sells
HARDWARE, something that you will definitely need to increase in the
CICSPLEX/SYSPLEX environment.

    BTW, when you say DYNAMIC, it really isn't. DYNAMIC issues an MVS FETCH
(such as a dynamically called batch program). But, in CICS, NODYNAM is
required and CICS locates the PPT address and establishes addressability to
the load module, much like an ASSEMBLER BALR 14,15. So much for particulars.

    Also, there are a number of PTF's (IBM fixes) that may need to be
applied to your COBOL/MVS environment that could be causing SOS
(Short-on-Storage) conditions. Verify with your Sys Prog that they have been
applied.

    The remaining issues you outline, such as below and above the line,
should be reviewed and discussed with your Sys Prog as certain parameters at
the PCT and PPT level must be considered.

    I know that this is a mountain of information, but hopefully it will
assist you. You may reply to me privately at wobconsult@sprynet.com if you
wish further assitance off-line....

Cheers,

WOB,
Atlanta

That is correct and will occur either if you invoke the  program as a
subprogram via COBOL CALL verb as well as if you invoke it with an EXEC CICS
LINK.

Both techniques make the table subprogarm part of the drivers overall
structure.  The COBOL and CICS facilities are designed to keep that structure
distinct from any other program structure within the system. 

The use of a common procedure division is simply an efficiency step.  The data,
constant or varible, must be kept distinct (according to the basic philosophy
of the tools you are using). 

This is a matter of structure, brought on by the manner of expression, a CALL
statement ( or also a LINK command).

What you may be after is a matter of scheduling.

Your ask, 
>>
Q2) What should I do to acheive the "objective" above.

... (single/central/constant copy.). <<

You may wish to schedule your program as a transaction oriented task.  This
will allow you to restrict the proliferation of the large table image (although
schedulable transactions are by no means restricted to just one copy, the
number of copies can be  controlled by CICS systems programmers).
Then a single copy (or relativley few copies) can be scheduled to handle the
talble activities in sequence.

That would require some redesign work.  The various invokers would have to be
prepared to go away and then come back when the table processor is scheduled
and then ends. The DFHCOMMAREA is intended to preseve the necessary information
while this round-robin sequence proceeds. Such a redesign may be too large in
scope.

Another approach is to place the table on a Temporary Storage Queue, and read
it into programs only when needed.
Potentially you could break the table into slightly smaller pieces and assign
them different ITEM numbers.

Yet another way is to reduce the table to some external file  (VSAM, of course)
and read only the records you need, .. and by  key.  VSAM can provide
surprisingly good performance if the table-like-file is basically locked into
memory because of constant hits.

I bet that you knew of most of these ideas already.  The Constant Table
Function is a fairly classic challenge in CICS.  Structuring the function as
part of your executable does proliferate the image into all tasks that invoke
the function.
The advantage is speed.  To trade out the huge storage penalty will introduce
complexity and cost you execution time.

The only way to avoid the penalties is to examine the table usage pattern.   If
you could divide the table into functional areas, then you could stick with the
lighting fast solution you have, get samller tables proliferated, and pay only
a slight penalty in program component management.

The internal design of CICS requires that evfery CICS task gets a
separate copy of the programs working storage.  If you want a single
copy to be available to many programs, you have several possibilities:

1)  Load the data into a KSDS VSAM file and load it as a Data Table,
or a shared data table.  That makes the data available throgh EXEC
CICS commands to all applications in this CICS (or the CICSPLEX)

2) Use Temporary storage instead of VSAM

3) use DB2 instead of VSAM (ever heard of the guy that took a 38 ton
truck to buy a pack of cigarettes?)

4)  A little bit more involved, but doesn't need VSAM calls.  COBOL
programs can still you  indexing and/or subscripting to access the
table:

Create an Assembler program containing the static data:  This would
look similar to this:

MYTABLE  CSECT
	DC	CLxx'Data for first entry'
	DC	CLxx'Data for second entry'
	DC	CLxx'you get the idea...'
	...
	END

Assemble that program and link it into a module, MYTABLE, or any name
you choose.

The COBOL program needs the following to access the table


	WORKING-STORAGE SECTION.
	LINKAGE SECTION.
	01	MYTABLE.
		05  TABENTRY	 PIC X(xx) OCCURS nnn TIMES.
	PROCEDURE DIVISION.
		...
		EXEC CICS LOAD 
		    PROGRAM('MYTABLE') 
		    SET(ADDRESS OF MYTABLE
		    NOHANDLE
		END-EXEC	                	
		IF EIBRESP = DFHRESP(NORMAL)
		    now you can process TABENTRY(subscript)
		    ...

Sure, if you're willing to CALL an ASM module (our batch system uses
this a lot). Post if you want a sample.

Bill Lynch

PS: The ASM's pretty basic (no pun intended), e.g.:

LOADER	CSECT
LOADER	AMODE 31
LOADER	RMODE ANY
	SAVE  (14,12)
	LR    BASER,R15
	USING LOADER,BASER
	LM    R2,R3,0(R1)          <<=== caller's parms
*                                I'm assuming R2 = A(table name)
*	                                      R3 = target area for table
(presumably in caller's working storage)
	LOAD  EP=(R2) 
	LTR   R15,R15
	BNZ   load_error

	STM   R0,R1,TABLE_ADDR     <<=== R0 = A(table)
*	                                 R1 = length(table) in doublewords
	SLL   R1,3                 <<=== multiply by 8 (to get the byte count)
	LR    R15,R1
	LR    R14,R3
	MVCL  R14,R0               <<=== move table to caller's area 
*	                      a programming palindrome, MVCL R14,R0 assembles
as x'0EE0' (fun for the feeble minded) <g>

	RETURN (14,12)



TABLE_ADDR  DC   2F'0'

This is right off the top - it's not reentrant and does not
provide/chain an RSA (which I would do in the real world). I'm assuming
you have register equates and that BASER is neither R2 nor R3. My
apologies if I remembered the parm for LOAD wrong. I'm sure there'll be
some suggestions/corrections, I never get these "off the top" coding
examples right. If you don't need to save the address of the table (so
you don't load it twice), then the code's reentrant. I've patterned this
after our main table loader, which is used for multiple tables.
