Initially, when I started my course in COBOL (because I "had to") I didn't
think it was a good idea either. I had the preconception, probably from
hearing other people talking, that COBOL was a "dead language", "popular in
the 70's", etc. I must say however, having completed the course (my final
exam is in two weeks) my opinion of the language and its usefulness has
changed entirely.

A very interesting point my lecturer brought up in the final lecture was
that many languages come and go, hundreds are created but only a half a
dozen or so have lasted as commercially or academically viable systems
languages. COBOL being one of them. It is of particular relevance in the
commercial arena. Many legacy and mainframe systems utilise COBOL's robust
and strong index file system to maintain complex databases, and hundreds and
thousands of records. COBOL is one of those languages that develop as
technology develops, unlike many languages that don't have much support when
it comes to new technology.

A question I think a lot of people need to ask themselves:
If it's such a useless language, why has it lasted so long and why do people
continue working on trying to improve it?

As for job prospects, I just did a search on an employment site using the
keyword 'Cobol' and the search returned at least 15 positions that required
some knowledge of Cobol programming. Does that answer your question about
jobs out there?

Cobol and databases go well. Did you know that SQL server uses multiple
index tables (the concept that was popular in Cobol long before anybody even
heard of SQL) and that SQL and Cobol can communicate and share data
effectively?

I can also tell you that I've programmed in Java and C++ and (from what I've
seen) neither languages provide an emphasis on database development, they're
more likely to be used for application development.

Of course, what it comes down to is what you want to do and where you want
to work. Perhaps you could search around newspapers, employment agencies,
and employment websites or maybe even ask the course
instructors/co-ordinators what they would recommend based on where you want
to go?

Hope this helps,

eRRaTiK

As I understand the "generation" terminology: 
1st:  machine code
2nd: direct conversion to machine code
3rd:  direct conversion to the 2nd level
4th:  direct conversion to the 3rd level

As I understand the languages, 
Assembler and C --> machine    therefore 2nd
COBOL --> Assembler    therefore 3rd
Java      --> C                 therefore 3rd
C++      --> C                 therefore 3rd

that defines a "random access file" as one that requires you
to count the numer of bytes--remembering that characters 
count as 2 bytes, integers count as 4 bytes, doubles count
as 8 bytes--multiply that by the number of records involved, 
and presto! you go to the record, with a probability of having 
calculated correctly that is greater than zero.  (Barely.)

Hi, everyone

I'm just going through this very topic in my text book (Discovering
Computers, Shelly Cashman Series, 2001 edition), and here's how the book
classifies the programming languages:

"1GL - Machine Language

2GL - Assembly Languages

3GL - High-level languages, are machine-independent, use English-like words;
require the program to tell the computer not only WHAT to accomplish, but
also HOW, therefore often called procedural languages; use a compiler or
interpreter to translate the source code into machine language

4GL - use English-like statements; are NON-PROCEDURAL languages. The
programmer only specifies WHAT the program accomplish without explaining
HOW. Many 4GLs work in combination with a database. Example: SQL

5GL, or Natural Languages - do not have to follow a specific set of rules
and syntax, like 4GLs. Are often associated with expert systems and AI.
Popular in the medical field, but are not widely used in business
applications."

By the way, the book is being very nice to Cobol, and it states that
..."Cobol is one of the more widely used procedural programming languages
for business applications. Although Cobol programs are often lengthy, their
English-like statements make them easy to read, write, and maintain... The
most popular personal computer Cobol program is Micro Focus Personal Cobol,
which supports both procedural and object-oriented programming..."

How many languages the code goes through is a function of what tools were
available.   Java produces byte code (not C), I have seen an applications
generator which produced International Basic on a mainframe.  I had a
macro assembler which I could plug in Atari Basic code...

What we care about is how well the language models the computer vs how
well the language models the business/gaming/whatever needs.  Early on,
this was fairly easy to see (machine language, assembler, compiler, apps
generator).   But the direction of change hasn't been direct from machine
to application.  C++ requires memory management, which is not a business
need.

Application generators haven't worked as well as hoped, while certain
applications have required a tighter interface with the hardware and
operating system.
