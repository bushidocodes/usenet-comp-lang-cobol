No problem Mark.  Simply put your two fields in the table under a group
item, and define a similar group in Working Storage, then be sure your
table is in sequence by that group item.  When you want to search, move
the two values you are looking for into the Working Storage fields and
SEARCH ALL for EQUAL TO the group item.  Assuming your compiler generates
a binary search for SEARCH ALL (almost certain), you're not going to get
much faster than that.  If it doesn't, then code a binary search. :-)

Just to clarify the issue of using the same index for multiple tables.

1) If we are talking about index-names (the thing defined in the INDEXED BY
phrase) and not an index-data-item (the thing defined with USAGE INDEX),
then
  A) The ANSI/ISO standard explicitly says that you may NOT use the index
defined in one table for another table.
  B)  Many (most but not all) compilers *DO* allow you to use such indices
across same-length tables.  In fact, some don't give you a compile error if
you use them across different length tables - but the results are often
"funny".

2) If we are actually talking about index data-items, then they can be used
for any and all tables - but they are never used to directly access a table
entry - only for "storing" the value from one table's index-names.
```

I also haven't checked my manual, but I think you could be a "tiny" more
efficient (in a strictly ANSI conforming compiler) by doing.

Set index-associated-with-table-2 to index-associated-with-table-1

Move Elem-from-Table-2 (Index-associated-with-table-2) to wherever.

In other words, you can set an index (from one table) to the value in
another table (whether they are the same length or not).  The compiler (or
run-time) will then need to convert the offset (if that is how the index is
stored - which it usually is) to a "value" that then gets converted to a new
offset.  Certainly not wildly efficient - but possible.
```

Mark,

If you are really concerned about performance, before selecting a lookup method,
look at your data first. Your search keys are integers, in which case both
serial and binary searches are extremely inefficient compared to key-indexed
search and/or hashing, and no compiler optimization would compensate for a slow
search algorithm. If your keys, in addition to being integers, fall into a
narrow range, then use key-indexing; if they can assume any integer value, use
hashing. Your particular situation seems to belong to the first category.
Suppose that none of your table entries exceeds 999. Then one can proceed as
follows: 

1. Declare another numeric ('key-indexed') table , say, KEY-X, with 1000
occurrences, and initialize its elements to -1 (or anything
PART-ENG-BITS-TBL-3801() cannot possibly be).
2. Loop through CUST-CD-ENG-BITS-TBL-3801() and PART-ENG-BITS-TBL-3801() once,
and move each
PART-ENG-BITS-TBL-3801(ENG-INDEX) to KEY-X(CUST-CD-ENG-BITS-TBL-3801
(ENG-INDEX)). 
3. Given a key KEY to be searched for (in your particular case, it is 1), simply
examine the location KEY-X(KEY). If it is equal to KEY the search has been
successful, otherwise it has failed. 

Just how fast is key-indexed search? Regardless of the number of lookup keys N,
it requires only 1 comparison and no sorting whatsoever, duplicate keys being
eliminated automatically. Loading the key-indexed table apparently runs in O(N)
time, and is a one-shot operation anyway. Contrasting that, binary search
requires logN or logN+1 comparisons depending on success or failure, plus the
search table must be sorted beforehand. So, if you have, say, 512 keys to look
up, it will take 9 to 10 comparisons (plus computing 9 midpoints) for binary
search to find the key or establish that the search has failed. Key-indexed
search will do both by merely examining 1 table bucket (plus computing offset).
It is easy to conjecture that on the average, key-indexed search always runs
logN times faster than binary search and N/2 times faster than serial search. By
the way, loading your arrays first is not necessary at all, and key-indexed
table can be loaded straight from the file (or whatever you are loading the
tables from).

What if the keys do not fall into a narrow range? For instance, they may be
6-digit figures, and so to facilitate key-indexed search a 1000000-element
key-indexed table would be necessary, plus most of the space thus allocated
would be wasted. The remedy is to allocate a 'hash' table only slightly larger
than the number of lookup keys, uniformly map the keys onto that table's
addresses using a hash function (in its simplest form it is just the key modulo
hash table size), and introduce some policy to resolve collisions between keys
hashing to the same location. Pursuing the topic further will probably lead too
far, so here is a good reference instead:

Donald E. Knuth. The Art of Computer Programming.
Volume 3. Sorting and Searching. Second Edition.
Section 6.4. Hashing. p.513.    

Look for 'coalescing list hashing' algorithm, and also for 'open addressing with
double hashing'. Both are trivial to implement in COBOL given the level of
description given by Knuth and the richness of COBOL's data representation
structure. Note also that hashing is not limited to integers but can deal with
deliberate strings as keys, effectively allowing to create structures indexed by
the values of character keys. In SAS, such structures are known as formats
(although the underlying algorithm is not hashing!). In Perl, they are known as
associative arrays or simply hashes and are intrinsic part of the language. It
would not be bad to have them in COBOL, too!

Just because the data distribution happened to work in favor of serial search.
For successful search, best case for both methods is 1 comparison: For serial
lookup - if the matching key is the first, for binary - if it is in the middle,
whilst worst case is N for serial and logN - for binary. For failure, binary
search never makes more than logN + 1 comparisons, and serial always makes N. On
the average, the point where linear and binary searches are equal in performance
is given by logN = N/2 (N < 7) for success and by logN + 1 = N (N < 4) for
failure. Adding some computational and logical overhead associated with binary
search, it will never run slower than serial search for N > 10, on the average. 

Presorting a file with several dozen records is hardly resource intensive. For
really large files it is, but we are talking about sorting a file whose keys and
satellites can fit in a table in memory, and SyncSort swallows much bigger meals
without choking in a matter of a CPU second. On the other hand, for a lookup
table exceeding mere 1000 entries, serial search is out of question, so sorting
is more than justified. In addition, if keys are distributed evenly throughout
the range, interpolation search (which is almost as trivial to code as binary
search) reduces the average number of comparisons for both success and failure
to loglogN. The latter reaches 5 at N = 4.3 billion; practically it will never
exceed 4. 

If sorting, nonetheless, is not desirable, hashing should be used. It does
require some programming and realistically takes about 50% more memory than
binary/linear search, but has the unique property of searching in constant time.
Because of the nature of the algorithm, it never makes more than 1.5
comparisons, on the average, for either success or failure regardless of the
number of lookup keys. That is, a hash table with 1,000,000 keys is searched
exactly as fast as one with 100 keys, provided that both are equally sparse.

> Wouldn't work if the key were alpha would it?

   Yes it would, although not _exactly_ the same way. The extension of the idea
of key-indexing called hashing will work with keys of any type. (Even though
digital strings - now of any length - are still hashed easier and more
efficiently than a deliberate alpha.) This is precisely how a COBOL compiler
looks up entries in the symbol table: It hashes identifiers you use in your
program to find their associated addresses. As you can tell from compilation
speed, this is a very fast process. Properties of hashing are similar to those
of key-indexing: a) Search time is independent of the number of lookup keys N,
i.e. it occurs in constant, O(1), time (rather than O(logN) for binary or
O(log(logN)) for interpolation search). Instead, it depends on the sparsity of
the hash table (dictated by you and available memory); b) No sort is needed, and
duplicates are weeded out automatically.
   Choosing a good hash function X, which is perfectly doable in COBOL, and a
half-full hash table will guarantee that any search key will be found or
rejected in a matter of less than 1.5 key comparisons (again, regardless of the
number of lookup keys). Working somewhat slower than key-indexing because of
extra logic and data movement, hashing is still light years ahead of binary
search the more, the more keys are to be searched. At only N=1024, binary search
requires 10-11 comparisons, hashing - 1.5, at N=1048576, it is 20 and still 1.5,
and so on.
