This is a good problem for demonstrating some simple techniques that you
need for enumerating large combinatorial spaces.

First, notice that this problem is zero-symmetric because of the
squaring.  Furthermore, there's no point in testing zero because 65 is
not a square:

      PERFORM VARYING A FROM 1 BY 1 UNTIL A > 8
          PERFORM VARYING B FROM 1 BY 1 UNTIL B > 8
              IF A*A + B*B = 65
                  DISPLAY " " A "  " B
                  DISPLAY " " A " -" B
                  DISPLAY "-" A "  " B
                  DISPLAY "-" A " -" B
              END-IF
          END-PERFORM
      END-PERFORM.

This snippet performs only 64 tests, as opposed to the 289 tests
performed by the original snippet.  Furthermore, examination of the code
emitted by this particular compiler (VAX CoBOL) shows that each of the 64
tests is almost twice as fast as each of the 289 tests in the other
snippet (primarily due to changing A and B from signed to unsigned).  Of
course, this is not something you need to care about for most programs,
but combinatorial spaces have an unpleasant habit of exploding.

Second, notice that this problem is mirror-symmetric.  We find both {1 8}
and {8 1}.  The following snippet performs only 36 tests.  Furthermore,
if you look at it carefully, you can discover for yourself why the next
optimization would be to hoist the A*A term:

      PERFORM VARYING A FROM 1 BY 1 UNTIL A > 8
          PERFORM VARYING B FROM A BY 1 UNTIL B > 8
              IF A*A + B*B = 65
                  DISPLAY " " A "  " B
                  DISPLAY " " A " -" B
                  DISPLAY "-" A "  " B
                  DISPLAY "-" A " -" B
                  DISPLAY " " B "  " A
                  DISPLAY " " B " -" A
                  DISPLAY "-" B "  " A
                  DISPLAY "-" B " -" A
              END-IF
          END-PERFORM
      END-PERFORM.

Generally, though, displaying the mirror pairs like this is considered
bad form.  It is enough to display all the unique pairs.
```

Very clear and succinct.  Just for fun, permit me to suggest a few
'improvements'.  One potentially big one is to change A**2 and B**2
to A*A and B*B respectively.  If the compiler is not clever enough
to recognize an integer power and use a different algorithm, it may
generate code to calculate the logarithms of A and B, multiply them
by 2, then calculate the antilogarithms.  Ugh!  Hundreds of lines
of assembly, and *very* slow.  The second is to calculate A**2 (A*A)
in your outer loop, a minor speed improvement.  In some compilers
the compute statement is less efficient than an individual multiply.
The third is to only iterate B using values >= A.  This excludes
duplicate solutions such as 8,1 and 1,8.  The fourth one is subtle,
and most compilers it wouldn't matter.  On some architectures making
the IF condition 'B*B + ASQ' instead of 'ASQ + B*B' can simplify the
generated code like this:

    ASQ + B*B -> move ASQ to temp1
                 multiply B by B giving temp2
                 add temp2 to temp 1

    B*B + ASQ -> multiply B by B giving temp
                 add ASQ to temp

And finally, display a space between A and B. :-)

           PERFORM VARYING A FROM -8 BY 1 UNTIL A > 8
               MULTIPLY A BY A GIVING ASQ
               PERFORM VARYING B FROM A BY 1 UNTIL B > 8
                   IF B*B + ASQ = 65
                       DISPLAY A " " B
                   END-IF
               END-PERFORM
           END-PERFORM.

    -8 -1
    -8 +1
    -7 -4
    -7 +4
    -4 +7
    -1 +8
    +1 +8
    +4 +7

Not that this application needs it, but depending on the compiler and
system architecture, the above code could run as much as hundreds of
times faster than the original.  There are times when it is useful to
know such things. :-)
```
