```
At last week's SHARE (IBM user group) meeting,  I learned something (well one
among many things) that I thought that I would  share with comp.lang.cobol,
IBM-MAIN, some other newsgroups/fora - and will probably do an RCF on.

Non-IBM mainframe people may "ignore" this note as you will.

Bottom-Line: (starting from the conclusion)

In (new) COBOL programs that use the Local-Storage Section (in addition to - or
instead of Working-Storage)
  - Be VERY careful that you (the application programmer) initialize fields - if
you require them to have specific "initial" values.
 - do NOT think that LE STORAGE(00, ...) run-time option will set Local-Storage
items to binary zeroes (as it does for WS items)
 - do NOT specify LE STORAGE(xx,xx,00,...) run-time option to initialize
Local-Storage items - unless you want your programs to "walk" rather than run.

    ***

Background:
  IBM's (ancient) OS/VS COBOL compiler (and/or its run-time) was "real nice" and
initialized WS items to binary zeroes - when they didn't have any VALUE clause
specified.  Especially under CICS, many application RELIED on this "feature" -
even though IBM *clearly* documented that "results were unpredictable" when no
VALUE clause was specified.
   When IBM introduced VS COBOL II, many applications DIED a horrible death -
because they ASSUMED WS items with no VALUE clause would have binary zeroes.
Therefore, IBM, in a relatively early release introduced the WSCLEAR run-time
option.  UNFORTUNATELY, rather than "fix" programs relying on uninitialized
values, programmers continued (with WSCLEAR support) to "assume" binary zeroes.
  With LE and LE conforming compilers (SAA AD/Cycle COBOL/370 and later), IBM
provided the first suboption of the STORAGE run-time option to "fix" not only
COBOL, but PL/I and any other program "assuming" uninitialized data items would
have binary zeroes (or some other hex value specified by the installation - not
the application programmer).

  ...

Today (and in the future),  For several releases, IBM has supported the ISO 2002
Standard COBOL feature of "Local-Storage" (particularly useful for recursive
programming and OO - but available to ALL new programs).  Unfortunately, LE
"gets" the storage for Local-Storage NOT from the "heap" but from the "stack".
This means that the first parameter of the STORAGE run-time option has NO EFFECT
on the initial value of Local-Storage items.  Rather, it is the 3rd parameter
that can be set to "initialize" data items with no VALUE clause upon initial
access to Local-Storage.  See the "dsa-alloc-value" described at:
   http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/CEEA3140/2.3.52

What is NOT clear (or at least doesn't "hit you over the head") in that
documentation, is the fact that setting the 3rd parameter of the STORAGE
run-time option has DRAMATIC (bad) performance implications for ALL applications
running with it on. (Using the initial parameter has *some* impact, but not
significantly bad).

As the Local-Storage Section should NOT be in "common usage" in "legacy"
application, and as it is NEVER been a good habit to rely on the internal values
of data items with no VALUE clause, no NEW program should require this
performance degrading option.  On the other hand, this is a fact that many
application programmers (used to WSCLEAR and
STORAGE(00)) simply may not think about.

I suggest that you "spread the word" among programmers who use new COBOL
features - and make certain that setting that 3rd parameter of the STORAGE
run-time option (other than to NONE) be something that SysProgs resist with all
their effort.
```
