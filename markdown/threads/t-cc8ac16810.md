I assume that your problem is that you have a file that has some "character"
data in it (that Standard "text" downloads will handle) and other fields
that are either COMP or COMP-3 that should NOT be converted in this way.

Whether you can get a free utility may depend on what you are going to use
the data with on the PC. For example, if you own Micro Focus Workbench, you
will get a free "intelligent" converter with the rest of the product. I
wouldn't be surprised if CA-Realia's product didn't have the same (although
I am not certain of that).

If you are not going to work with the files with COBOL on the PC, then you
are probably going to have to write your own conversion routine. (Although
File/AID PC might work for you - but it is a purchased product also) Your
utility will need to convert the character (PIC X) data from EBCDIC to
ASCII. It will need to convert the COMP/BINARY data from big-endian to
little-endian. And it will need to convert the sign nibble in both Display
and Packed-Decimal fields from mainframe conventions to PC conventions. All
of this will require that you know what the record layout looks like - so
you can do a field by field conversion. (Watch out particularly for files
that have multiple record layouts - depending on the value in some specific
field.)

P.S. If you happen to have any DBCS data, you will need to call special
software for doing that conversion. Also watch out for Shift-Out/Shift-In
byte loses.

This is a common problem (we are dealing with again this month).

As Bill said, MicroFocus does have support for EBCDIC handling on PC, and
some utilities. The utilities are of limited value, as they are difficult
to set up for some types of files (mixed record formats, occurs depending
on, redefines, etc.). They also do not seem to exist on the UNIX version
of the product, instead there is a much more limited set of utilities. The
EBCDIC option also has limits, as it is set on or off for the whole
program, not file by file. However, due to an idiosyncrasy of MF COBOL,
you can write a simple READ...MOVE CORRESPONDING...WRITE program, setting
CHARSET"EBCDIC", and setting the organization on the output file to LINE
SEQUENTIAL. The line sequential files always come out ASCII, regardless of
the charset used for the program compile. Of course, line sequential get
trailing spaces trimmed, and record delimiters put on the ends, but it is
generally useful.

Fujitsu COBOL does not yet have an EBCDIC option. We have had some
indications it is coming in the next release, but we have not been able to
get a rumored beta version to try it. For them you have to put in
individual INSPECT REPLACINGs for each field. It helps, if you can control
the output record format, to put all the character under one group level,
and convert that just before write.

Lastly, Realia had, at one time, the best approach of all (although I don't
have the current release, so I cannot confirm it is still there). In
Realia you could specify at run time, on a file by file basis, what
character set you wanted to use. This meant you could write a single
program, compile it once, and use it for any combination of ASCII/EBCDIC
inputs and outputs. This was really great.

File-Aid PC will also do an intelligent conversion, but you need both the
PC and mainframe products, and that is somewhat expensive. If you have an
extra $20,000 or so to blow, you might get in the door.

Oh, BTW, EBCDIC to ASCII is not your only problem if you have variable
length files (say, from a tape) on the mainframe. Most file transfer
methods will drop the RDW words, so your target system cannot determine
record lengths. If you DO transfer them, the target system doesn't know
what they are, or what to do with them, so you end up writing a program to
handle that (write me for an example, if you need it). MicroFocus also
gives a sample program called VRECGEN to tag the length onto the record,
download it, and then feed it to their conversion routine on the PC (SPF/PC
can also read VRECGEN files), but getting it to run on your mainframe files
usually involves customizing a version for each file layout, and you have
to pass the files two extra times (once on mainframe, once on PC) to
convert them, so large files can eat a significant amount of extra time.

Good luck, and remember...it's not that you don't know what you are doing,
but that this is ALWAYS more trouble than it should be.
