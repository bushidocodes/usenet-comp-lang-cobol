This is actually one question in two.

As far as concerns moving your character mode application to run as an
windows application that should not be a problem, at my former employer we
moved from Microsoft Cobol to Acucobol in roughly one week, only a few
syntaxical changes were necessary. The application looked and behaved as
before, as the Acucobol runtime were simulating the character screen for our
application, while the runtime itself were a native windows executable. The
upside of this being that the application could continue to run on any
character mode system be it unix, dos or OS/2 console. The downside was that
there still was no windows features.

The other question then, do you get GUI? No, your interface is still the
same, which makes sense. Although computer software improves every day in
terms of automate tasks, something really is difficult to accomplish. Hence,
making a GUI is still up to you.

I know, however without having tried it myself, that the integrated
development environment which ships with Acucobol, does have an opportunity
to "read" screen layout from old programs, and provide you the lay out as a
template to work on.

I also know that you may have both gui and character screen layouts in your
Acucobol code without much overhead.

In Acucobol GUI is implemented as an extension to the language rather than
depending on external forms, this is a matter of taste, personally I prefer
doing:

    DISPLAY ENTRY-FIELD  MULTILINE AT 1001 HANDLE entry-fld.
    ACCEPT entry-fld UPDATE ON EXCEPTION ...

rather than:

    CALL "SomeForm" USING SHOW-CMD.
    CALL "SomeForm" USING ACCEPT-CMD GIVING return-code.
    EVALUATE return-code
        WHEN ...
   END-EVALUATE.
