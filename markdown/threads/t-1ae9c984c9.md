Bob,

I just do the following: Use embeded SQL including a EXEC SQL CONNECT ...
END-EXEC somewhere at the start of the app. The CONNECT uses a ODBC data
source that is setup via ODBC (control panel). No linking with a linker (obj
modules) is needed. Of course MS SQL Server has to be installed on some
server and the client install has to be done on each client PC (Win NT or
'95 / '98).

The Frog

Ken,

It is possible to keep the standard I/O statements (read/[re]write/open/close/etc) when using SQL. It could however be wise to use SQL to access the flat files, indexed or not. In the latter case one needs ODBC drivers to access those files as if they were DB-tables. These ODBC drivers need to be provided by the vendor of the file system, Merant/Micro Focus in this case.

Perhaps there are independant vendors that sell this kind of ODBC.

> Have you posted this on answerlab? they should be able to point you
> in the right direction...

? Whom do you ask this? (I just answered some q. of Bob).
In case you refer to the question "what is involved with linking COBOL to MS SQL Server" of Bob, the answer is:
0) *I* did not ask answerlab.
1) Install SQL server on server(s)
2) Install ODBC on clients
3) Use SQL in programs.
and of course:
either 4a) keep flat files, keep using reads/writes fro them
or     4b) convert flat files to table, switch to SQL i.s.o. standard COBOL i/o.
