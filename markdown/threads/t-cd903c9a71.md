'COMP', which is also spelled 'USAGE IS COMPUTATIONAL', specifies that
the data field is computational, i.e., numeric in nature.  There are
several COMP-n types:

    Type       Description
    -------    ------------
    COMP       binary
    COMP-1     single-precision float
    COMP-2     double-precision float
    COMP-3     packed decimal, a.k.a. BCD
    COMP-4     same as COMP
    COMP-5     alternate form of COMP, usually native endian
    COMP-X     binary, but treatable as character

For COMP-3 (packed decimal), you get two BCD digits per byte, most
significant digit first, with a sign nybble in the low 4 bits of
the last byte.  The field varies in length from 1 byte (1 digit)
up to 10 bytes (19 digits), or maybe even up to 16 bytes (31 digits).
In other words, B bytes yields B*2-1 digits, while N digits need N/2+1
bytes.

The sign nybble is one of these values:

    Bits   Sign
    ----   ----
     A      +
     B      -
     C      +  (preferred)
     D      -  (preferred)
     E      +
     F     unsigned (positive)
    other  invalid (treat as positive)

For example, a 7-digit packed decimal field with the value +1234567
would be represented by the following bytes:

    0x12 0x34 0x56 0x7C

Converting packed decimal values into integer (binary) values is
fairly straightforward.  Fortunately, the representation doesn't
vary much from system to system.  I've done extensive work in this
area, so I recommend a lookup table for each byte.  For example,
the first byte above (0x12), when used as an index into the table
would yield the value in decimal form (12).  Simply shift by two
digits per byte (i.e., multiply by 100), except for the last byte
which has only one digit.  Then adjust for the sign.

But be prepared to handle overflow.  A 32-bit integer can only
hold 9 full decimal digits.  Packed values with more than 9 digits
need a bigger binary target type, perhaps 'long long'.

Converting packed decimal into characters (strings) is easier,
and follows pretty much the same algorithm above.  You don't have
to worry about overflow unless your target character field is too
short.

Also be aware than COMP fields aren't necessarily integers; they're
fixed-point.  Which means you have to take into account the
placement of the decimal point during your conversions (unless
you're certain that you're only dealing with integer fields).
