### Paragraph Numbering and Program Structure

```
I was taught as a beginning programmer to use ascending numeric prefixes
on all my paragraph names.  Over the years, I have kept this practice, and
have embellished it in a number of ways.  I like to use the paragraph number
to reflect a relative structure hierarchy.  Over the years, I have also
learned to appreciate ease of copying code from one program into another,
so I have standardized on a six digit prefix.  Below is a stylized example
of how a program might be structured, and how the paragraph numbers would
be used to reflect the hierarchy.  Note the way the paragraph numbering
reflects the actual structural hierarchy of the program.  By reflecting
the structural hierarchy of the code in the paragraph numbering, it makes
it very easy to recognize the program structure.  A paragraph does not
only PERFORM a paragraph of a subordinate level, of course.  The 'global
utility' paragraphs are for routines which are not necessarily subordinate
to the major functions of the program, but can be used by them.  Examples
might be a date edit, or field convert of some kind.  For independent
routines of one paragraph, I use 00nn00, and for routines with several
subordinate paragraphs, I use 00n000 for the top level paragraph.  In
general, paragraph 120000 will have every paragraph 12nnnn subordinate to
it (12300 -> 123nnn, 123400 -> 1234nn, 123450 -> 12345n).  As an aside,
when we used to use sections to segment our code, paragraph 000000 would
start a section, and each different nn0000 paragraph would start another
section.

Many of you may use a similar method.  But I have found this to be a very
flexible and valuable tool over the years, and I like using it very much.
Only rarely do I need to move a group of paragraphs to a different 'level'.
It has contributed a great deal to my productivity, and the effect is
cumulative.  The more code I collect which is similarly structured and
easy to use, the less code I need to write.  The advantages I have reaped
from this are too many to enumerate.  It is easy to find any paragraph in
the program, you do not need to think up unique paragraph names for two
very similar routines in different parts of the program (e.g. doing the
same thing to different tables), it is easy to see the hierarchical program
structure, etc.  That last one is extremely valuable.

If you like this method, then I'm happy to share it with you.  If so, or
if not, please post your comments.  If this example is not clear, let me
know and I will try to be clearer.

      *
      *  ** TOP LEVEL CONTROL STRUCTURE (000000) **
      *
       000000-CONTROL.
           PERFORM 000100-INITIALIZE
           PERFORM 000200-PROCESS
               UNTIL (end of data).
           PERFORM 000300-TERMINATE.
           STOP RUN.

      *
      *  ** SECOND LEVEL CONTROL STRUCTURE (000n00) **
      *
       000100-INITIALIZE.
           Get today's date, open files, read first input record, etc.

       000200-PROCESS.
           PERFORM 010000-major-function-1.
           PERFORM 020000-major-function-2.
           ...
           PERFORM 990000-major-function-99.

       000300-TERMINATE..
           Print ending totals, close files, etc.

      *
      *  ** GLOBAL UTILITY ROUTINES (00nn00) **
      *
       001000-utility-routine.
       001100-utility-routine.
       001200-utility-routine.
           ...

      *
      *  ** MAJOR FUNCTIONS (nn0000) **
      *
       010000-major-function-1.
           PERFORM 011000-intermediate-function-1-1.
           PERFORM 012000-intermediate-function-1-2.
           ...

      *
      *  ** INTERMEDIATE FUNCTIONS (nnx000) **
      *
       011000-intermediate-function-1-1.
           PERFORM 011100-minor-function-1-1-1.
           PERFORM 011200-minor-function-1-1-2.
           ...

      *
      *  ** MINOR FUNCTIONS (nnnx00) **
      *
       011100-minor-function-1-1-1.
           PERFORM 011110-sub-function-1-1-1-1.
           PERFORM 011120-sub-function-1-1-1-2.
           ...

      *
      *  ** SUB FUNCTIONS (nnnnx0) **
      *
       011110-sub-function-1-1-1-1.
           PERFORM 011111-low-function-1-1-1-1-1.
           PERFORM 011112-low-function-1-1-1-1-2.
           ...

      *
      *  ** LOW FUNCTIONS (nnnnnx) **
      *
       011111-low-function-1-1-1-1-1.
           ...

       011111-low-function-1-1-1-1-2.
           ...


       011120-sub-function-1-1-1-2.
           ...


       011200-minor-function-1-1-2.
           ...

       012000-intermediate-function-1-2.
           PERFORM 012100-minor-function-1-2-1.
           PERFORM 012200-minor-function-1-2-2.
           ...

       012100-minor-function-1-2-1.
           ...

       012200-minor-function-1-2-2.
           ...


       020000-major-function-2.
           PERFORM 021000-intermediate-function-2-1.
           PERFORM 022000-intermediate-function-2-2.
           ...

       021000-intermediate-function-2-1.
           PERFORM 022100-minor-function-2-1-1.
           PERFORM 022200-minor-function-2-1-2.
           ...

       021100-minor-function-2-1-1.
           ...

       021200-minor-function-2-1-2.
           ...

       022000-intermediate-function-2-2.
           PERFORM 022100-minor-function-2-2-1.
           PERFORM 022200-minor-function-2-2-2.
           ...

       022100-minor-function-2-2-1.
           ...

       022200-minor-function-2-2-2.
           ...
```

A serious problem with this scheme is that if the structure changes during
maintenance (remember the program may live for decades) you have to
rename the paragraphs as well. This is tedious and errorprone. Also,
numbering paragraphs makes re-use more difficult as the program
into which you import the paragraphs to be re-used may have a different
structure.

In the old days where you would pore over a 4-inch thick listing it did make
some sense to number stuff, but today it does not. A respectable editor
should assist you, by e.g. letting you click or right-click on a paragraph
name and then simply show you that paragraph. Same thing with data-items.
There are also many tools that will show the structure of a program in
various
graphical ways.

A very efficient way of writing non-trivial programs is to
treat them as Finite State Machines (FSMs). An FMS
operates with a (Finite) set of States. In each state a
number of Events can happen. When an event happens
a list of Actions is carried out, and a New State is selected.

The Actions could simply be paragraphs to be performed.

Now, the FMS is described externally to the program, e.g.
as a simple text file. From this description and from a list
of Actions (in any order), the final Cobol program can be
readily generated. The Actions as written by the programmer
have no structure. So, an FMS is kinda the opposite of
a "structured" program. The structure is supplied by the
FMS description, not the Cobol fragments written by the
programmer.

This way of programming is inherently event-driven and
thus very well suited for interactive programs, but works
equally well for batch programs. In batch, the events are
the arrival (reading) of transactions.

If the program has no structure, paragraph numbering
is superfluous.

One of the many advantages of "structureless" programs
is that there is no structure to be destroyed during
maintenance. Traditional structured program tends to
become less structured with time.

After reading Judsons original post and all those answers I'll put in my own

The usage of external general functions or procedures in a procedural
environment or the usage of methods defined in classes in an OO environment is
completely different from the usage of internal subroutines in a program. The
former routines / methods are intended to be re-used in other (future) contexts
and as such should be clearly defined and named to make them reusable. The
latter ones, internal subroutines, are building blocks of a component and not
meant to be exposed outside, so naming is less important, at least the name will
be seen by less people.

But in a more conventional environment one sees the following kind of programs
lots of times:

Batch programs with multilevel structures, based on processing sequential piles
of data, sorted on one or more key fields which unavoidably shows up in the
program's structure. Of course OO techniques could be applied against such type
of data processing but would IMO not give some extra strength: development will
not be much quicker in time, re-use of classes tends to be low (every separate
data set to proces is a new class). The "standard "approach" using known methods
and techniques, like JSP, IDO, etc (all based on Bï¿½hm/Jacopini concepts,
possibly I am a little benulux oriented here) is reliable, fast and proven.

like CICS, IMS-DC, IDMS-DC and other look-alike tp-monitors, have a lot of
similar subroutines, all coming from (and thus unavoidably programmed in
reaction on) the standardized calls/statements/structures that are enforced by
the pseudo-conversational tp-monitor. Lots of the stuff in this world is more of
the same. software products look a lot the same, hardware is a lot the same.
Everybody mimiced IBM in those days ...

In all those situations, the internal program structure could very well be
cleared up (or made some clearer) by using a numbering/naming scheme like
Judson's.

Very different will be the components and their public API's that are build to
compose systems that have to run in environments where modern gadgets rule the
world: OLE, clipboard, ActiveX, ORB's etc. etc. These systems tend to look like
the environment they run in (just like the CICS programs): open-enden API's,
event-driven. Lots of the parts of such systems will be accessible from outside
and as such require a name for usage i.s.o. a number.

internals to the outside world. An object has his well-defined interface to get
from state to state and uses names for that interface, not (containing) numbers
that would show some of the hidden implementation.

object/program/function/component could very well number those to reflect some
things, internal structure for example. Just he and perhaps one or two
maintenance programmers are the people that have to understand or change those
numbers. Those 2 or 3 people are a lot less than the audience that will use the
object/prog/func/compo by it's public interface.

Yes it does.  I love my PC based text editors, and they significantly
improve my productivity.  However, it is quicker to just look and see
what the structure is than it is to have to search for each paragraph
you're interested in, trying to discern what the structure is.  You
write it only once, you may read it many times.  The #1 rule I follow
in formatting source code is this:

  "The program listing serves but a single purpose: to convey information
  about the program through the programmer�s eyes to the programmer�s
  brain as efficiently as possible"

The better it serves this purpose, the more maintainable the program is.
This issue is quite separate from program structure, which is also very
important, of course.  Once the structure is determined, how best should
the code be formatted to achieve the above result?  The answer is tied
closely to the way the eye and brain processes visual information, and
is not nearly as subjective as many think.  One can adapt to any number
of personal preferences, but many of these preferences cannot be defended
from an objective standpoint.  I've spent a good portion of the last 25
years thinking about, studying and experimenting with these issues.  The
result is a series of programming standards which I have made available
and taught to my clients.  The results in improved programming accuracy
and productivity have been so impressive that many MIS shops in this
area have adopted some or all of my standards on their own, distributed
solely through word of mouth.  It is unfortunate that some programmers
exhibit a knee-jerk opposition to methods which do not fit their P.C.
thinking.  The question is, did they come to this P.C. thinking through
objective test and study, or are they merely parroting what they have
been taught by others who are also parroting what they have been taught?
This is a question each person should ask themselves.  Personally, I
always try to remain open to techniques and methods which might improve
my productivity.  I have found this approach to be extremely beneficial.
