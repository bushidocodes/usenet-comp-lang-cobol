I read through to-day's messages in 10 seconds :-)

So somebody asks me can he use the M/F Menu and Pop-Up Menu classes to 
create Context Menus. First thing I did was to google to find out what 
Context Menus were ! (Good decision on my part to quit programming - I 
don't want to saturate my mind with all this additional jargon).

 From the initial screenshots I perused looks like dear old Microsloth 
has thrown another one at developers to keep them on their toes. The 
screenshots for the most part conveyed traditional pop-ups, plus of 
course individual selections pop-up a sub-dialog, (child window), either 
a short message or perhaps with three fields or so, one of which might 
be a URL reference.

I told my enquirer that I currently have MyMenu.cbl which is a set of 
literals with accelerator keys which I pass to MakeMenu.cbl.
It's Makemenu which creates an Ordered Collection and with indicators 
(from the passed parameters) I inform it to make sub-collections 
representing a sub-menu. When used of course the click gives me the 
index position which takes me to the relevant program number (class) in 
my Master Menu (or main window/parent window).

Having looked at MSDN a while back I read the general MS approach to get 
at keys in collections/dictionaries is to have two parallel collections 
- click on say a Menu and the index position can be used to access the 
second collection.

I don't know all the implications of Context Menus but my approach might 
be something like the following. Extend the literals I send to MakeMenu 
  to have two features (a) the Menu Item and Accelerator key and (b) the 
second part any additional info required, e.g. like URL references. 
MakeMenu - using two collections, uses (a) to generate the displayable 
Menu entries and (b) is stored in the second collection.

I offered him my source if it will help. Yes I know he could buy a 
component, but if my source is close to what he wants to do then he has 
complete control in one language. Who knows, he may well go the 
component route via dotNet if he is using Net Express V 4.0. I haven't 
yet had his reaction to my reply.

I'm not going to delve deeply into this, but seems there are two 
possibilities for use, a Context Menu in his COBOL application or 
perhaps  he wants to trigger it while browsing a site on the web.

Any thoughts. For the artist not the paintbrush - Michael - how would 
you handle this one in Basic. Richard - I betcha you might suggest 
scripting languages which I saw reference to on this topic when 
googling. Any other takers ?

(Presumably, if you were trying to introduce your own Context Menu into 
something like the Mozilla Firefox Browser there has to be some wake-up 
trigger ?).

Any other takers in addition to Michael and Richard ?

Jimmy

A point that may have been missed here is that context menus are a standard 
part of the Windows environment. You can create them easily by modifying a 
few Registry entries. No code is required, although most people would 
probably install them in the same script that starts the app. (if it starts 
with a script...).

For windows applications like IE and Word, you can certainly  modify or add 
your own context menus to the standard context menus, using Registry mods, 
and you can probably do the same with any given window (I haven't tried 
it...)

IF I was going to write a component which did context menus (and I'm not 
:-)) I would simply call the API to modify the appropriate Registry entries 
for the application window. There would be a Method in the component that 
let the User define what was wanted....the tree required and the events that 
trigger that tree, bearing in mind that the tree selections depend on the 
context in which they are activated.  Once the component had the 
requirements, it would make the necessary Registry amendments. It is really 
pretty simple.

a very valuable learning tool to understanding the Registry and how keys 
work. In other words, to USE the Registry, rather than MESS with it...:-)

I may not have been clear in my earlier post that the Registry settings for 
context menus are a one-time thing. I would NOT change the Registry every 
time the application that required context meus was activated. Instead, an 
entry would be made under the necessary headings in the Registry for that 
specific application, ONCE only.

Therefore, to implement context menus would come down to amending the 
Registry in either of two ways:

1. If you know what you are doing, manually make the necessary entries (one 
time) then run your application. I submit that this is NOT "messing" with 
the Registry...

2. If you feel that the Registry is a place you don't want to go, and really 
believe that some of the effects Jerry has described will be magicked into 
existence by mysterious Registry Guardians, then use an interface that 
simply lets you describe what you want to happen, and makes the Registry 
entries for you. (This is what I described if I was going to make a 
component for dealing with context menus.)

As always, fear of the unknown results in superstitious belief. :-)

The best way to deal with it is to acquire knowledge. There is nothing 
mysterious about using the Registry in a windows environment. I have been 
doing it for years and I have never crashed, frozen, or destroyed my system. 
Certainly there are risks, but all of them can be mitigated by taking 
sensible precautions, and the Registry actually provides simple exports and 
imports so you can export a hive or set of value pairs before "messing" with 
them. Then, if you totally screw up, you simply restore what was there when 
you started. (I have never had to do this, but I have the exports if I ever 
did need to...) I also use a Registry tool  (in my case, Registry First 
Aid - I recommend it) to keep the Registry in reasonable condition and 
re-organize it every couple of weeks. It will allow you to see what is 
simply occupying space, old, no longer used keys from software you removed 
but which didn't clean up after itself properly, and so on.

We had Business processes decomposed into Use Cases where each Case provided 
certain deliverables. It's certainly possible, by JAD review (including the 
Business) of each of the possible paths through these Use Cases, to ensure 
that the code works and provides the deliverables. Not only that, but we can 
be sure that what is being delivered is what the Busineess needs NOW, (not 
what they thought they needed during Requirements Gathering all those months 
ago...) so that the developers and our customers (the Business) are all

for the Business to change the process  (functionally and/or technically) 
during these reviews if they have priorities that are more important than 
what is being delivered. There are no sign offs, and nothing is carved in

No slippage, no extensions. Do it once (in the sense that this Use Case is 
completed ONCE, although it may require a number of iterations), do it 
right.

development (as opposed to production) system, running against data that is 
subsets of the real data, but the process is ITERATIVE (up to the limit

(More complex products, contact types, larger volumes, etc.)

Finally, the whole system is parallel run before going live and this uses 
real data, real volumes, real network, servers, and infrastructure. Every 
Use Case is ticked off in the 'psuedo-live' environment  by the 'shop floor' 
before the system is declared live.

live which reveal hitherto unexplained flaws.

I would respectfully submit that everything that could reasonably be done to 
preclude this, was done, and would therefore expect this to be minimal.

production, running 24/7, there were 2 flaws identified and each of them was

flaws and I never stop trying to improve this figure... :-) I wish I could

I am keeping the posting below for background for this posting.  The
use of the registry in Windows as the repository for all sorts of
application information leads to a more vulnerable system.  OS2 has
the same vulnerability (OS2 still exists but only barely).  I don't
know whether Unix and Linux have a registry equivalent.  What we have
is a single entity crucial to the system manipulated by applications
which come and go.  The uninstall process may or may not clean up the
registry entries.  There may or may not be utilities adequate for safe
updating of the registry.  I have tried a few of them, all of which
required paying close attention and understanding a fair amount about
the system.  While the parameter library entries used by z/OS can be
cryptic and confusing, they are in plain text and those libraries used
by the system are normally only affected by the system related entries
or such things as databases or systems utilities and enhancements.
Normal applications normally don't touch them and all information
controlling the applications is normally kept in libraries/files
unique to the application or application suite.  In short I find that
the use of the registry by applications is a design with opportunities
for foulup and malice.

In article <utf1g21ger22q0c4n0cq8nghqrpdqrtapr@4ax.com>, Clark F Morris <cfmpublic@ns.sympatico.ca> writes:
> I am keeping the posting below for background for this posting.  The
> use of the registry in Windows as the repository for all sorts of
> application information leads to a more vulnerable system.

True.  It's a classic OS-design mistake.

The Windows Registry is what's sometimes called a "compound file";
in essence, it's a filesystem implemented in a file.  (In the case
of "the" Registry, there are actually multiple files, which is even
worse, because you have multiple virtual filesystems that have to
be "mounted" at the proper points to create the single hierarchical
view.)

While compound files are useful for some applications (zip archives,
for example), they require implementing most of a filesystem, and
have all of the problems of a filesystem - doubled, because there's
the filesystem that actually hosts the compound file.  Look at the
Registry: it has hierarchical relationships and paths and ACLs and
content types, all the information that NTFS *already provides*.
If each Registry entry were a small file under a well-known location,
MS could have saved all that duplicated effort.[1]

Worse, since the compound file isn't an actual filesystem, all the
normal file-processing tools and utilities won't work on it.
Instead you have this black box and a handful of tools which have
to be invented specifically for the compound file.

There are Windows utilities (such as U/Win, an AT&T Unix-on-Windows
product) which actually provide a filesystem view of the Registry,
so you can use file-based tools on it.  Unfortunately, it's painfully
slow, as the Registry is poorly implemented.

Alas, Microsoft's love affair with compound files shows no sign of
ending.  They also use them for the PDB debug-symbol file format,
for Outlook email archives[2], and so on.

> I don't know whether Unix and Linux have a registry equivalent.

Standard Unix (SUSv3) does not.  OS configuration is almost entirely
kept in well-known human-readable formatted-text files, which is one
of the things that makes Unix so much easier to administer than
Windows, for the experienced user.  (For one thing, you can search
through thousands of plain-text files in half the time it takes to
locate one Registry entry with a tool like regedit.)  Most Unix
applications follow suit.

Some Unix flavors have experimented with non-text configuration.
IBM's AIX has a small built-in DBMS called the Object Data Manager
(ODM) which is used to hold some system configuration information.
AIX 3 (the first modern release, for the original RS/6000 line) used
it for some of the standard Unix system configuration stuff.  That
was not popular with users, and with AIX 4 most of it was moved back
to plain-text files.


[1] Some people might object to this design on the grounds that the
filesystem's cluster size means significant space would be wasted
by all those small files.  That problem was solved years ago by
BSD's fragmenting filesystem feature, and there's no reason why MS,
with all of its resources, couldn't have implemented something
similar.

[2] Unfortunately, it seems this particular disease - storing email
messages in compound files - is pandemic on Windows.  Even Mozilla
Thunderbird, which has now reached the stage of being almost
competent, continues to commit this blindingly stupid error.  Look,
folks, BSD Mail got this right thirty years ago: PUT EACH MESSAGE
IN ITS OWN FILE.  It's faster, it's safer, it's more robust, it's
easier to manage, it's easier for the MUA developer.  There is *no*
good reason to use a compound file for email archives.
