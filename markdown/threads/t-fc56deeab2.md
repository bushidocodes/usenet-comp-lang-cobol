A little while ago I promised to show how I solved the problem of writing
XML easily from COBOL.

I REALLY wanted to use LINQ-to-XML but it isn't easy from COBOL. (Once you
have a connection to it, ceating and manipulating XML with LINQ is every bit
as easy as creating and manipulating databases with LINQ-to-SQL or even
in-memory arrays with LINQ-to-Objects...)

It seemed to me it should be fairly easy to find the libraries that need to
be referenced and attached and plug them into COBOL. (Maybe someone with an
inquisitve spirit and some time to experiment might have a go at this?)

It isn't. (At least, not with the native code generating COBOL compilers
which most people have; if you have a newer IL generating compiler (...
COBOL for .Net) then you CAN simply attach references and update the
repository in your COBOL program...but I don't have such a compiler and have
no intention of getting one.) As I am pressed for time at the moment, I
stopped trying, and cheated by creating a C# COM component to deal with the
XML. (It was much quicker for me, as I have standard templates I wrote that
can be used to create different forms of C#, including COM components.)

A long time ago I DID write XML directly from COBOL using the MS library
that is part of Windows, and the COBOL 2002 standard specified a XML
extension for the language (although I odn't know of any vendors who have
implemented it...), but both of those approaches mean using native XML, and
I'd rather use LINQ...

Currently, as part of the effort to help people migrate away from
PowerCOBOL, I am writing code that analyzes PowerCOBOL screens and gets them
to describe themselves in a way that other software can use, to duplicate
the screen in different environments. (XAML for the Web, and XML for the
Desktop, currently.)

Briefly, there is a PowerCOBOL component I wrote that gets added to the
existing PowerCOBOL project, and this code builds a couple of tables with
information about the Form and the controls on it. We then need to export
those COBOL tables (all fixed length fields) to anything else that might be
interested in the PowerCOBOL form. (I could write them to a sequential file
but XML, is becoming the lingua franca for cross platform communication. So,
I knocked up my C# COM component (which has access to the LINQ libraries)
and simply invoked it from the COBOL, passing it the two tables from COBOL.

That's the solution. (It is all working very well...)

Here are the details:

COBOL code:

CLOSE-DOWN section.
cd000.
invoke COM "CREATE-OBJECT"
using XPCFormCreateXML
returning objXPCFCX *> get a reference to C# program for
LINQ 2 XML
end-invoke
*> iterate the rows and create XML for them...
perform
varying efd-x1
from 1
by 1
until efd-x1 > controls-count
move "$" to ctlCOMEndMarker (efd-x1)
move controlDescriptions (efd-x1) to XPCFCXParm1
if efd-x1 NOT = 1
set efd-x3 to 1
search suppControlDescriptions
at end
move "NULL" to XPCFCXParm2
when
suppName (efd-x3) = ctlName (efd-x1)
*> there is supplementary data for this control...
move "$" to suppCOMEndMarker (efd-x3)
move suppControlDescriptions (efd-x3) to XPCFCXParm2
end-search
else
*> we are doing the Form description...
*> set Project Name into Supplementary
move ctlName (1) to suppName (1)
move passedProjectName to suppText1 (1)
move "$" to suppCOMEndMarker (1)
move suppControlDescriptions (1) to XPCFCXParm2
end-if
*> Call C# to write the XML...
invoke objXPCFCX "XPCFCX"
using XPCFCXParm1
XPCFCXParm2
end-invoke
end-perform
move "END" to XPCFCXParm2 *> close off the XML
invoke objXPCFCX "XPCFCX"
using XPCFCXParm1
XPCFCXParm2
end-invoke

Here's the LINQ code that does it...

private void createXML()
{
if (firstTime)
{
firstTime = false;
// create the document declaration and comment, and the Form
description...
var xml = new XDocument(new XDeclaration("1.0", "utf8",
"yes"),
new XComment("PowerCOBOL Form description
generated by PRIMA Tool XPCForm."),
new XElement("Project",
new XAttribute("ProjName", suppText1),
new XElement("Form",
new XAttribute("FormName", ctlName.Trim()),
new XElement("Text", ctlText.Trim()),
new XElement("Dimensions",
new XElement("TopY", ctlTop),
new XElement("TopX", ctlLeft),
new XElement("Height", ctlHeight),
new XElement("Width", ctlWidth)),
new XElement("Colors",
new XElement("Forecolor", ctlForeClr),
new XElement("Backcolor", ctlBackClr)),
new XElement("Controls"))));
publicXML = xml;
}
else
// we are processing controls...
{
var xml = new XElement("Control",
new XAttribute("CtlName", ctlName.Trim()),
new XElement("Type", ctlType),
new XElement("Text", ctlText),
new XElement("Parent", ctlParentName.Trim()),
new XElement("Dimensions",
new XElement("TopY", ctlTop),
new XElement("TopX", ctlLeft),
new XElement("Height", ctlHeight),
new XElement("Width", ctlWidth)),
new XElement("Colors",
new XElement("Forecolor", ctlForeClr),
new XElement("Backcolor", ctlBackClr)));
doSettings(xml);
publicXML.Root.Element("Form").Element("Controls").Add(xml);
}
}

Obviously, all of the above looks much better in colour and formatted
properly, but you can get the general idea.

Nether the COBOL nor the C# is excessive, although some other supporting
routines in both programs are not shown.

The XML above can be easily "tweaked" into XAML and fed to tools like
Expression Blend to create WPF screens which have vectored rendering with
proper gradients and look really smart, or the XML itself can be fed to
other tools (currently being written) to create new Visual Studio Windows
Forms projects with the identical screen from PowerCOBOL.

The major effort for this project will be in transforming the current
PowerCOBOL scriptlets which "drive" the screen, into a standard NetCOBOL
component that will become code-behind for the Windows Form. Fortunately we
(PRIMA) already have tools and components that can provide a solid base for
this effort.

Moving PowerCOBOL to .Net (Windows Forms or WPF) provides a much richer
enviornment in terms of available controls and features than the PowerCOBOL
IDE provides, and it means you can write your code-behind in COBOL OR other
languages.

XML from COBOL? Pretty easy with an IL generating compiler (you can use
LINQ...), but that is an expensive option and many people are not looking
for further heavy investment into COBOL. Get C# for free, learn the basics
from online videos (you don't need "advanced C#" to do this job...), and
call that...

Use the right tools for the job.

That's a bit pointless if it won't compile. The case in point (trying to use
LINQ directly from COBOL) requires specific constructs that need repository
data that is Micro Focus compiler specific (Or Fujitsu compiler specific if
you use that) . So you go to all the trouble of getting it to work then
find:

a. You are not allowed to distribute it.
b. The source won't compile on a different COBOL compiler.

Either way (Fujitsu or Micro Focus) you have to buy an expensive COBOL
compiler (which may also incur run fees on applications you write with it)
and it is not compatible with other COBOL if you use anything .Net related.

Unfortunately, it would be of no use to me at all, given the limitations
described.

Much quicker (and certainly cheaper) to download Visual C#, watch a few
basic videos, then get cracking on all the .Net development you want
(including use of LINQ) with no worries about paying license fees, and at
the same time, expanding your skill set. (Support for newbies in C# is
astounding and there are many millions of people who are happy to help with
both the language and the IDE (Visual Studio). As I showed in the post, you
can mix and match COBOL and C# without any problem or complexity, provided
you use OO COBOL. (It doesn't HAVE to be .NET COBOL; the COBOL sample code I
posted is unmanaged FujitsuNetCOBOL; it interacts seamlessly with C#
components and the reverse is also true.)

I think if I was a "COBOL hobbyist" or somebody looking "to give students
and professional developers the opportunity to add COBOL to the skillset.",
I'd be looking to GNU COBOL. At least you can distribute the applications
you write with it.

I've also "moved away" from Micro Focus COBOL in general, for the following
reasons:

1. Run Time license fees. (Why would you pay these when there are
alternatives that don't require it?)

2. Lack of proper COM support. Support for COM is ESSENTIAL for me and my
Company. There are no Micro Focus support classes provided (as there are
with Fujitsu) and (at least with Net Express) it is impossible to batch
compile COM componens. You have to build each one manually and interactively
with the provided IDE...and it gets tiresome having to set up COM at the
level required by Micro Focus for each component.

3. Everything seems to be at a lower level with Micro Focus and it's just
more complex. It's like trying to emulate C++ in COBOL.

I would stress that the above are just personal opinions (but based on
experience) and the Micro Focus products are of good quality generally. (I
also consider them to be a reputable Company who will stand by their
products; a consideration for people persuaded to go the "Open" route...) I
haven't tried their "Visual COBOL" CIL generating compiler for .Net because
it is too expensive (I haven't tried the Fujitsu one either, for much the
same reason...), and because I have no need of either of them. C# is free
and does everything these compilers do, and then some.

It's the 21st century; there is a choice.

We no longer need to do everything in COBOL and some of the alternatives are
WAY better...
