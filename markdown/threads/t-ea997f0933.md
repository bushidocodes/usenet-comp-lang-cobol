I just discovered a production code problem which fails when run on a 
Microfocus platform, but is running without error on an IBM platform. Here's 
the edited code:

VAR-1   PIC X(08).
VAR-2   PIC 9(09) COMP.
...
MOVE '123456bb' TO VAR-1  (bb is two spaces)
MOVE VAR-1 to VAR-2

Under microfocus, this gives a runtime error of  something like "Invalid 
character in numeric field."
Under the IBM, it moves 12345600 to VAR-2. As it turns out, this is exactly 
the data we wanted, so I guess no one noticed.

Is there a complier option which would account for this difference? We tried 
this on the IBM with InterTest, and forced 12345678XX into VAR-1, and it 
moved 12345677, In EBCDIC, an X is hex 'E7' so it appears to be just 
stripping the zone.

The PC and mainframe are two different architectures.

On the mainframe to convert DISPLAY format data to COMP data, the DISPLAY 
format data is first PACKed into packed decimal/COMP-3 format.  That PACKing 
changed the X'F1F2F3F4F5F0F0' to X'1234500F'.  All of the zone digits are 
removed except the rightmost zone.  That one is used as the sign and an F 
sign is considered positive.  After all that the PACKed data is converted to 
COMP/BINARY via a CVB (Convert to Binary) instruction.

On the PC there is no packed decimal format.  A different process is used 
and in this case the result is a different format from the mainframe.  There 
may be compiler options that can change this, if so I am sure someone will 
respond with them.  I am not that familiar with PC COBOL as I am not willing 
to spend thousands of dollars for a COBOL compiler. I hope this helps.

When compiling under Micro Focus, what compiler option do you have set for the
  DIALECT
compiler option?

If you are using IBM Enterprise COBOL on the mainframe, then
  DIALECT(ENTCOBOL)
with Micro Focus should give you mainframe compatible results.

If you are using Enterprise COBOL on the mainframe, what are you using?
If you do not have a current (supported) version of the Micro Focus compiler, 
then you may need to look at the "WB" directive instead of the "dialect" 
directive.

If
  DIALECT(ENTCOBOL)
doesn't give you mainframe compatible results *and* you are running a currently 
supported Micro Focus compiler, then you should contact "support" and this may 
be treated as a "bug".
