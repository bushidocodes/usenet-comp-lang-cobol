Where last we left our Intrepid Coder he was... lost. The difficulty was
in finding a replacement for:

INCLUDE COND=(184,4,CH,GT,C'9212',AND,184,4,CH,LT,C'9302') or
INCLUDE COND=(184,4,ZD,GT,9212,AND,184,4,ZD,LT,9302)

using any of the Y2q parms for which SyncSort's new versions allow. Mr
Reda, of Syncsort, suggested:

INCLUDE COND=((184,2,Y2C,GT,92,OR,
(184,2,Y2C,EQ,92,AND,186,2,ZD,GT,12),AND,
(184,2,Y2C,LT,93,OR,
(184,2,Y2C,EQ,93,AND,186,2,ZD,LE,2))))

... and this Did Not Work; for a given input file the original INCLUDE
created an output of 2197 recs while this produced 13170 recs.

Over the weekend Mr Bart asserted:

› The correct answer to your problem is
›
› INCLUDE COND=(184,4,Y2C,GT,C'9212',AND,184,4,Y2C,LT,C'9302')
›
› If this doesn't work, there is a *BUG* in Syncsort.

... and, despite this assertion, such an implementation generates:

INCLUDE COND=(184,4,Y2C,GT,C'9212',AND,184,4,Y2C,LT,C'9302')
*
WER161B ALTERNATE PARM USED
WER253A INCLUDE/OMIT FORMATS INCOMPATIBLE

... so, when it is polished up a bit to:

INCLUDE COND=(184,4,Y2C,GT,9212,AND,184,4,Y2C,LT,9302)

... one then gets:

WER251A INCL INVALID LENGTH

This, according to Mr Bart, is a '*BUG*'.

Finally, someone (my apologies, your name got lost in my forwarding of the
post to myself) suggested that Mr Reda's alternative be modified, viz.


--begin quote
Again the increased record count is due to an expanded selection criteria.
[ IF ( (pos184:2 > 92) or ( pos184:2 = 92 and pos186:2 > 12 )) and
( (pos184:2 < 93) or ( pos184:2 = 93 and pos186:2 <= 2 )) ]
this select should get 9301 and 9302 dates if this field is truly a date
field. Primary difference is the ending value for the range due to the LE
test.
--end quote

This I implemented by changing the last LE in Mr Reda's example to an
LT... and the result was the same 13170 recs out (vs the original 2197).

Any other suggestions out there? As should be obvious I'll try anything
at least once and report back with the results.

DD
