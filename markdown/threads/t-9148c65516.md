Personally I avoid negative IF statments like the plague.  But to sorta answer your question:  Using an 'equal to' condition on something like this can mess you up if you don't have exactly the right number of lines on a particular page - like your double spacing - where you add 2 to the line-counter instead of 1 and go 1 past MAX-LINES.  Unless you know that each page will be EXACTLY alike don't do it.

Instead of this code:
PRINT-ENTRY.
    IF (LINE-COUNT NOT < MAX-LINES)
        PERFORM PAGE-HEADING
    END-IF.

I would code:
PRINT-ENTRY.
  IF LINE-COUNT > MAX-LINES
        PERFORM PAGE-HEADING
  END-IF.

Notice that 1) I use the 'greater than'  and 2) I don't have the parens as I only use parens for compound IF statements

IF (LINE-COUNT > MAX-LINES) AND (CONTROL-BREAK = 'N')
   PERFORM PAGE-HEADING
END-IF.

Lousy code - but an example.

When working with 'sloppy' counters (thinks like line-counters, subscripts) 'equal to' can be dangerous because you risk not hitting the equal condition and going off to who knows where.

Just my thoughts.

Eileen

Eileen Preston wrote:
>Personally I avoid negative IF statments like the plague.

Why?  A programmer should be able to understand any logic statement
that is clearly and simply written, negative or no.  The essence of
computer programming is logic.  My suggestion is, if you have problems
using and understanding simple 'not' logic, then study and practice
your logic rules until you can use and understand them well.  To do
anything less is an acceptance of a real deficiency in your programming
ability.  Would you trust a doctor who said "I avoid using some very
important diagnostic tools, because they are too difficult for me to
understand?"  You would probably respond "Get away from me, and go back
to medical school!"  When it comes to using logic correctly in computer
programming, if you can't stand the heat, get out of the kitchen. :-)

>Instead of this code:
>PRINT-ENTRY.
…[8 more quoted lines elided]…
>  END-IF.

The above two statements are not equivalent.  LINE-COUNT = MAX-LINES
would be found true on the first, but fall through the second.  Also
you need to consider that other factors are involved.  For example, if
you want to print lines to 50, then page break, it is most logical to
set MAX-LINES = 50.  For (LINE-COUNT > MAX-LINES) to work properly,
then LINE-COUNT must be maintained to be 1 greater than the actual
number of lines printed, or you must set MAX-LINES to 1 less than the
actual limit.  Either of these violates the Rule of Least Astonishment.
However, using the simple 'NOT' violates no proper design rule.  It is
the simplest and most straight forward expression for this particular
purpose. :-)

reflect the nature of what the programmer is looking for, as to make it
'simple'.  More is going on in a programmer's mind when examining a
conditional test than in simply understanding the specific logic.  The
programmer is initially scanning the code to understand what the logic
of the program is trying to do.  For this purpose, the code should be
rapidly assimilable, as to intent.  Only after a given conditional is
targeted for close scrutiny is the detailed logic fully analyzed.  I
seek to structure conditionals to most clearly expresses the intuitive
sense of the purpose of the conditional test, as well as to be easy to
read.  For example, if the condition '(A > B) AND (C > D)' was commonly
used in the program to test for a specific condition, and this code was
to test for that condition being false, then I would likely use the
former structure.  To use your form in this case would make the specific

If I might comment on writing complex boolean expressions:

Usually these do not stand alone, but as part of a larger 
decision tree. One has code along these lines:

if <complex condition #1> then <action #1>
else
  if <complex condition #2> then <action #2>
  else...

In my cobol writing days, I found that an old-fashioned 
decision table was very helpful in setting up such code. Not 
only did it help you verify that you hadn't overlooked some 
odd combination of circumstances (completeness), but also made 
excellent documentation when written into the code as comments.

By modelling the nesting of the if's on the structure of the 
table, you made future maintenance a piece of cake. And these 
complex decision trees often require modification.[1] Heaven 
help us all though, when some rookie modifies the code without 
modifying the documentary table!

I generally preferred nested if's to complex parenthesization.

It really didn't matter if lots of the terminal actions were 
"next sentence".

[1] I suspect I am preaching to the choir.
