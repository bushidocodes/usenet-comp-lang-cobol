Hi,

I was wondering if anyone has had success bringing a large amount of
cobol legacy code to a web interface.

I have the task of trying to figure out the best way to do this and I
have been looking into the following options: -

 - Using Micro Focus Net Express to create server side programs with
cgi to let the web interface talk to the existing but modified cobol
programs.
 - Using Micro Focus Net Express to create either COM Objects or Java
Beans for EJB from the existing business logic code that exists in our
cobol code. THen use ASP or JSP for the dynamic interface.
 - Using IBM's VisualAge for Java to convert the cobol to Java Beans
and running something like websphere to run the full system on EJB.

Can anyone give me some advantages/disadvantages or advice on what
you've found is the best way..

 - CGI will end up messy and a maintenance effort once it takes off
and the system grows.
 - COM is nice and easy with ASP, could base the whole thing on .NET
 - Personally I prefer java, but then we would need to buy websphere,
and possibly VisualAge.
 - Complete rewrite would be nice but probably not viable.

The cobol code is pretty old and some of it is very messy and poorly
written - in a lot of ways I would like to completely ditch it and
re-write something from scratch. But at the same time, there's no
point doing that if its going to take 3 times as long as re-using
what's there in the form of modern objects.

Just on Net Express - I have been using it for the past couple of days

There have been several errors.  One is the code 114 alignment execution error
on the Sun Solaris described in the attached Oracle note.  Unfortunately not all
occurrences have been corrected using the -C compiler option and adding sync to
the working storage declaration.  We also could  reorder working storage
elements to make sure that S(9)'s fall on 2 byte boundaries but that would be
very burdensome (800 applications, 3 million lines of code) but we may have to
do that for some of the programs.

The other problem was with a report that listed transactions that had unprinted
documents associated.  In the materials management side there are transactions
that would create a transfer order, issue order, stowage order, etc and these
documents would be saved on the database.  A print daemon would print the
document on the proper printer ( a user in Georgia could have an order print out
to a warehouse clerk in California) base on the transaction data in the data
base record.  The daily exception report would contain orders that didn't print
due to printer being down and so on.  

The document  contained page breaks and line feeds and defined as PIC X(2000)
and initialized to spaces, and some transactions would create a transaction
record with only one document and some would create two.  
.

The logic was something like this (names changed to protect the innocent)

IF WS-ORDER-DOC-2 > SPACES
	MOVE WS-ORDER-DOC-2 TO  REPORT-BODY-1
	ADD 1 TO WS-REPORT-CTR-2
ELSE
	MOVE WS-NO-DOC-2-FOR-TRANS-MESSAGE  REPORT-BODY-1
	ADD 1 TO WS-NO-DOC-2-CTR
END-IF.

When animating or executing the .int using NetExpress,   if  WS-ORDER-DOC-2
contained a document it did move WS-ORDER-DOC-2 to the report as you would
expect.  

However, on the Sun server animating with ServerExpress, even though
WS-ORDER-DOC-2 was NOT filled with spaces and contained a valid document
begriming with a page break, the else statement was executed and the no second
document message was moved to the report for every occurance.

I  changed the logic to the following which worked correctly on both platforms

IF WS-ORDER-DOC-2 EQUAL SPACES
	MOVE WS-NO-DOC-2-FOR-TRANS-MESSAGE  REPORT-BODY-1
	ADD 1 TO WS-NO-DOC-2-CTR
ELSE
	MOVE WS-ORDER-DOC-2 TO  REPORT-BODY-1
	ADD 1 TO WS-REPORT-CTR-2
END-IF.

Hope this helps.  YMMV

James Johnson

can be more efficient and is viable if you are SURE you will not be
transporting the code to platforms other than MicroSoft.).

There is no dobut in my mind that for serious development, components
are the way to go. In a multi-tiered application it really doesn't
matter whether you use ASP or HTML for the presentation layer or use
XML to transport the data from the component back to the presentation
layer.

We had a requirement recently to look at all the compay's legacy
systems to obtain Customer data (and update it), from a single front
end, running under Siebel CRM. (In other words, customers who have
beenn migrated to our "New World" solution (Siebel) would be obtained
from the ORACLE databases supporting Siebel, but customers who are in
Legacy systems that have not yet been migrated were a problem...

The initial solution was to do a quick "Cut down" migration, migrating
customer "stubs" to Siebel then running extracts from Legacy every
night to update them. This would ensure that Customer data could be
accessed under Siebel. It would also require customer data in two
places for the same customer, and all the horrors that batch
synchronization entails...

I vetoed it.

Instead we produced a Legacy Portal Component (it took half a day,
given that we had most of the requisite code existing in Legacy and
simply had to wrap it as a component) and will plug this component
into Siebel next week.

This will enable all our Legacy systems to continue running while we
properly address the full migration issues and, at the same time ALL
of the Customer base can be accessed transparently from a Siebel
terminal (which is what the Business wanted.) No Batch sync runs, no
"half-arsed" migration, no impact on existing Business Processes, and
no impact on either Siebel or Legacy. The customer data is simply
obtained from wherever it resides (Siebel or Legacy)in real-time.

This to me, is the best pay-off I have yet seen from using Component
technology. It saved 6 man-months of work...

The COM Objects should reside on the server and their methods can be
invoked remotely from the clients using DCOM, or directly on the
server using ASP or whatever you are scripting with.

As I said, I am not an expert in the UNIX environment. Traditionally,
UNIX has tended towards using CORBA and Java, but I understand the
latest version of CORBA includes support for the MS world of ActiveX,
COM, DCOM, and OLE. If you develop with SOAP the objects become Web
Services and can then be deployed in any environment where .NET is
present.

If it is going to continue to run on a UNIX box you might consider
wrapping the COBOL code with Java and building Java Beans which can
then be deployed with CORBA in a similar way to COM in the MS
environment.

That's IMHO, the best hope for a CoBOL future - although as far as business is
concerned, we could have a replacement for CoBOL to run its functions and
business rules - plugged into a system that uses other languages.

And that's the only reason I see that has a chance of selling OO CoBOL for
mainframe, enterprise work.  Encapsulate some CoBOL business rules in an object
and use them to talk with Java and other tools.

Multiple use of an object appears to have more costs than benefits where
companies require testing of everything that touches a changed module.  (and
CoBOL's biggest strength is in its maintainability).   But no tool is the best
for all purposes.   We can have an isolated CoBOL object used for one function
only - that the user accesses through an XML/Java interface, we only have the
one thing to test - but can pick the best tools for the various portions of the
product.
