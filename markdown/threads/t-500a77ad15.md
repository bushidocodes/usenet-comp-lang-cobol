### Executing OS/VS COBOL & VS COBOL II on OS/390

```
Year 2000 considerations are going to require nearly every IBM MVS
system in the world to upgrade to the OS/390 operating system. Many
(most?) of the installations are still running COBOL programs compiled
using the above compilers. I've even heard of a company running
COBOL(E) - check your insurance policies!

IBM no longer supports these compilers, nor the use of their run time
libraries. The executable COBOL program can be supported IF the run
time library used is Language Environment (LE for MVS). IBM states that
programs compiled with the NORES option (which causes the run time
routines to be link edited with the program), must be relinked to
replace the old run time routines with LE versions.

Thus far, I have seen one posting on the Y2K e-mail group which said
that a shop had encountered a load module which wouldn't execute under
OS/390. It was at Howard University, and was an OS/VS COBOL program,
compiled NORES, that did some VSAM I/O. Relinking the load module
solved the problem.

My questions:
Have other shops begun to run on OS/390 yet? (At mine, we're testing
OS/390 now, and the upgrade is a couple of months away.)
Have other shops encountered the COBOL run time problem yet?
Do other shops have a plan for dealing with the COBOL run time problem?
Do other shops have plans to migrate all of the COBOL to the current IBM
compiler?
If so, will it be a mass migration, or a gradual one, or will it follow
some other strategy?

Thanks for your input,
Colin Campbell
```

```
My understanding is that OS/VS COBOL has been off support for several years,

VS COBOL II will not longer be marketed as of June, 1997, but I've been

OS/VS COBOL is particularly hard to get rid of, because we don't pay any
license fees for it anymore. We are planning to install LE/370 and COBOL for
MVS & VM and in September, and at some point after that we will cancel
our COBOL II license because we don't want to pay two license fees.

We are upgrading from MVS 4.3 to MVS 5.2 in August. I'm not sure why they
don't want to upgrade to OS/390, but it may have something to do with the
fact that we run Amdahl mainframes instead of true blue IBM iron. In any
event, MVS 5.2 is a Y2K compliant release, which is what we need. My
applications group has already done some initial testing with MVS 5.2 and
there have not been any problems.

› Have other shops encountered the COBOL run time problem yet?

NO, but I'll be watching for run time problems and NORES. Thanks for the
tip! The applications in my department are almost exclusively COBOL II,
so I don't expect too many problems. We have a lot of development groups.
It wouldn't surprise me if some of them have a few NORES programs.

› Do other shops have a plan for dealing with the COBOL run time problem?

We still support some CICS 2.1.2 regions, which cannot use LE/370. We're
also upgrading to CICS 4.1 beginning this fall. I've been told that any
regions which have problems can be easily set up to use COBOL II runtime
libraries (we run our OS/VS COBOL using COBOL II runtime libraries). We
tried a quick test of a CICS 3.3 region with COBOL II & LE/370 at the end of
our MVS 5.2 test. The region wouldn't come up, but I am pretty sure it was
because we haven't done all the right preparation yet. Our tech services
people tell us that CICS 3.3 regions can run with LE/370, so we have to do a
little more tweaking to get there.

We plan to be upgrade all CICS regions to release 4.1 by mid-1998, in order
to be on the Y2K compliant release. My regions are all CICS 3.3 and COBOL
II, so I expect that to be relatively painless. The departments still
running CICS 2.1.2 will have some catching up to do.

› Do other shops have plans to migrate all of the COBOL to the current IBM
› compiler?

I think we will have some resistance. Our plans are to install LE/370 as the
default runtime environment for batch programs. CICS regions will be handled
on an exception basis. Based on my initial LE/370 testing, I don't expect to
recompile any COBOL II programs for LE/370. But I am eager to start using
the new date intrinsic function.

› If so, will it be a mass migration, or a gradual one, or will it follow
› some other strategy?

I expect we will follow a gradual migration strategy, because different
departments seem to have their own priorities. I volunteered to test
COBOL/MVS. I took a 7300-line COBOL II program and recompiled it with no
source changes (not even to embedded compile options in PROCESS/CBL lines).
It produced identical results. I also executed the COBOL II version using
the LE/370 run time library and had no problems. All our COBOL II programs
were compiled under release 3 or higher and use NOCMPR2.

I would really like to see us stop using OS/VS COBOL, but we have a few
departments that won't use anything else. Some of them are still using
BDAM and macro-level CICS.

About two months ago I attended the project kick-off meeting for the LE/370
conversion. I represent an applications group, and there were several other
applications people in the meeting. Some of them were shocked when our Tech
Services guy mentioned cancelling the license for COBOL II -- they're just
now considering converting to it, and we're getting ready to junk it!
```
