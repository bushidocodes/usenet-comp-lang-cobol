```
I have only done what you suggest once in 4 years, and even then it was in
my training and was not used in production.

A pointer is just a large library number, or as you suggest a field of type
POINTER. you give it a value by using the SET statement. You reference a
fields pointer by using the ADDRESS OF clause. There is no built in way to
allocate memory from the HEAP in COBOL. The method I used called a
subprogram called GETMAIN and FREEMAIN to allocate space for each NODE in
the LINKED-LIST/TREE/STACK. Then using the SET statement with the ADDRESS OF
phrase to traverse the LINKED-LIST I implemented. Such as:

4000-INIT-NODE.
    CALL 'GETMAIN' USING NEW-NODE
    .
4100-APPEND-NEW-NODE.
    SET    ADDRESS OF NEXT-NODE OF CURRENT-NODE
        TO ADDRESS OF NEW-NODE
    SET    ADDRESS OF NEXT-NODE OF NEW-NODE
        TO LOW-VALUES
    .
4200-MOVE-TO-NEXT.
    IF NEXT-NODE OF CURRENT-NODE = LOW-VALUES THEN
       SET END-OF-LIST TO TRUE
    ELSE
       SET    CURRENT-POINTER
           TO ADDRESS OF NEXT-NODE OF CURRENT-NODE
       SET    ADDRESS OF CURRENT-NODE
           TO CURRENT-POINTER
    END-IF
    .


Good luck on finding documentation for GETMAIN and FREEMAIN. I assume they
are assembler routines.
```

```
A further qualification would be that I have done this on an IBM MVS

Although dynamic memory allocation would be good, using a table-implemented
linked-list or possibly implementing it in a RANDOM/INDEXED file would
probably be more practical. Pointers in the TABLE would simply be the index
of the row of the table the node is contained in. Whereas in a file it would
be the record key where the node is located.

The whole idea behind DATA STRUCTURES is for them to be platform
independent. Whether you implement it using dynamic memory, tables/arrays,
or files only the low level IO routines should change. This will be true if
the routines are properly modularized. Once done transporting them from
system to system should be relatively painless.

One thing I did in COBOL to overcome a memory limitation was to do something
like:

01 table-data-1.
   05 filler pic x(143000).
01 table-data-2.
   05 filler pic x(143000).
01 table-data-3.
   05 filler pic x(143000).   <--- some number representing
                                   the maximum size of a field.
66 linked-list redefines table-data-1 thru table-data-3.
   05 first-node      pic 9(18) usage comp.
   05 curr-node       pic 9(18) usage comp.
   05 last-node       pic 9(18) usage comp.
      88 full-list    value some-rediculously-large-number.
   05 max-node-index  pic 9(18) usage comp
                      value some-rediculously-large-number.
   05 node-count      pic 9(18) usage comp.
      88 empty-list             value zero.

   05 node occurs 1 to some-rediculously-large-number.
      10 data-field   .......
      10 next-node    pic 9(18) usage comp.
         88 no-next-node        value zero.
      10 prev-node    pic 9(18) usage comp.
         88 no-prev-node        value zero.


Then you simply use first-, curr-, last- nodes as pointers. Your logic
should then be easily adapted to COBOL. Mind you the 143000 number was off
the top of my head. That number will definitely differ from system to system
depending on the maximum field size which the compiler allows.
```

```
Some COBOL compilers support pointers, perhaps some support dynamic
memory allocation and deallocation.  But I suggest you loosen up your
thinking a bit. :-)  Many younger programmers who have finished school
since C has become popular have been taught a somewhat limited view of
what 'linked list' means.  Programmers were using linked lists long
before dynamic memory allocation and pointers were available other
than in assembly language.  While dynamic memory allocation, pointers
and linked lists work well together, like ham and eggs, pointers are
not necessary to implement a linked list.  Many linked lists are in
data structures on disk, and the 'pointers' are record numbers.  In
memory, just create an array ('table' in COBOLspeak) and use subscripts
instead of pointers.  Works just as well, is far less error-prone, the
only real disadvantage for most uses is that the size is not dynamic.
But as abundant is RAM is these days, that's usually no big deal. :-)
```

```
Done it in simple Cobol (listen up Bill cuase this is probably

What is a linked list:  It is the ability to locate a random entry
into a list from the preceding element. The technique is simple
but must be on like elements (which they mostly are).

01 element-free      PIC 9(04) COMP VALUE 1.
01 element-first     pic 9(04) COMP VLAUE 0.

01 element occurs X times
   05  element-details.
     10 .....
   05  element-next  PIC 9(04) COMP.

Element next points to the next element but is really a simple
subscript.   Initialise the array with each element-next =
subscript + 1 and there is your free list.

The element list starts at element-first and finishes when it hits
a zero (initialy empty).  As you add to the list you can either
add to the end or figure out an insertion for element-next.

Pretty simple huh!

Yes I know it is not a 'real' linked list but it performs the same
functions.  I use it in the cobol parser to link the subordinate
working storage to it's group, which in turn points to it's group,
the '01' level has a zero.

Note I use 9(04) COMP because it is the most efficient subscript
for MVS.  Results may vary (as the advertisement says).

Ken
```
