What is the source of your data? Is it ASCII? You'll never bomb moving BINARY
(COMP) to DISPLAY-NUMERIC, regardless whether it was created as ASCII or
EBCDIC. Your BINARY-DATA will be loaded into a register, an instruction
converts it to COMP-3 and it is then Unpacked. I suspect the culprit is the
PACKED-DECIMAL (COMP-3) Data, in particular, the sign-nibble in the last byte.
Having said that, what puzzles me is that moving PACKED-DECIMAL to DISPLAY-
NUMERIC should not cause a S0C7 (this can't happen with an UNPK instruction),
unless your sending or receiving picture clause specifies an implied decimal-
point. In which case, a decimal shift instruction could very well bomb if the
data is not a valid PACKED-DECIMAL value.

Could you post the basics of your code so we can have look see and maybe a
snapshot of some of the data?

As pointed out in another post, you can't get INVALID data in a COMP field.
(You can get data "out of range" - but that isn't the same thing and doesn't
cause S0C7's).  If your PIC S9(04) COMP field has "spaces" in it, then
X'4040' will be evaluated as a number and treated accrodingly.  Similarly,
if you have high-values X'FFFF" will give you a numeric value.  Finally, you
can't even get an invalid sign in a binary field (IBM mainframe - other
systems might do this - but I doubt it).  The high-order bit is going to be
on or it is going to be off - and both are valid signs.  (Everything stated
above is NOT true if the field is really Packed-Decimal or COMP-3 - on IBM -
but it is true for Binary or COMP)

NOTE (for those following the draft of the next Standard),  they have
defined the extended "IF NUMERIC" test so that binary fields with values
outside the range of the PICTURE clause *will fail* the IF NUMERIC test (for
the traditional binary data types - not the new C-like "true binary" data
types).
