It is indeed unusual to find them both on the same site...

Still, there is no reason why they shouldn't be used together and the fact
that they are usually not is probably more historical than practical.

PL/1 was originally PL/I (Programming Language International) but IBM
revised it when they realised it was not enjoying the universal uptake they
had hoped it would. Later versions of the documentation show PL/1 indicating
it was the first attempt at an International Programming Language. (Some of
us are still waiting for PL/2...<G>)

As people are starting to realise that it makes sense to use the best
Language for a particular purpose, and as PL/1 is a "programmer's language"
designed around the principles of structured programming, it will not be so
surprising to find mainframe sites using PL/1, in conjunction with COBOL,
for tasks that might previously have been done in Assembler.

I have only used PL/1 with mainframes and only for subprograms, but I found
it effective and useful. The main "advantage" it has over COBOL is that it
can implement dynamic arrays. In my day this was done without resorting to
GETMAIN at the programmer level (the dynamic run time calls for memory were
indeed implemented by calls to GETMAIN, but no assembler knowledge was
required and it was transparent to the programmer). Nowadays of course, with
the LE environment, God knows how they implement it, but it apparently
works....

(Of course, COBOL has many other features which make it useful and in which
PL/1 is not so strong...they are both useful.)

As for things to avoid in PL/1 programming, avoid the same things you would
avoid in ANY programming... <G>

Thanks for letting us know at least one site has this environment.

Pete.

PL/I was released in 1964 and found little uptake outside of IBM. COBOL 61
was well established and had built on the success of COBOL 59.
Most commercial shops were happy to use COBOL. In those days it was a very
brave EDP manager who would go up against the dictates of IBM so some shops
did try the new language. Those that didn't, usually pleaded an existing
investment in COBOL that made it non-viable.

IBM decided to re-market the product and re-badged it as PL/1. (And yes,
they did say that the "I" was a Roman numeral "1". This saved re-printing a
lot of existing material. However, if you look at the manuals released when
the product was standardised in 1976 (ANS), you will see that the "I" has
become a "1". (A capital "I" is not normally used to denote a Roman "1"
anyway; it should be a small "i")

I don't doubt that if you were on an internal IBM course in 1968 you would
have been given the official "party line" and to this day that is what many
people "remember". Certainly, that is how IBM "remember" it...I worked for
them at Hursley Park for a while and there was only one other guy there who
knew what I was talking about, when the subject came up over lunch one day.

BY 1972 when PL/M was released, a number of shops were using PL/1 and by
1976 when it attained ANS standard there was a fair penetration of the
available market.  (PL/M disappeared without trace, as the market could not
really sustain TWO contenders for COBOL's crown.) However, it always had to
compete with COBOL and that is why there was a "division" between shops that
used COBOL and shops that used PL/1. It really had little to do with the
relative merits of either language; more to do with IBM pressure and
marketing.

I think it had everything to do with the fact that mixing PL/1 and
COBOL is a real pain in the neck. I've used and like both languages,
but in the early days, even into the ninties, setting up and
maintaining both a PL/1 and COBOL run-time environment was confusing
and special steps had to be taken to make sure that they didn't step
on each other. Debugging a PL/1 subroutine abend in a COBOL run unit
was difficult because of the way PL/1 used the registers.  Internally
it did not use the standard IBM linkage. Storage management in PL/1
was beyond strange.  Special care had to be taken to make sure the
correct compiler options were used.  There was a whole chapter in the
PL/1 programmers guide on how to pass and receive arguments from
COBOL.  Tables were stored differently.  PL/1 supported data types
that Cobol didn't and didn't do a lot of type checking. Rounding and
truncation and alignment issues between modules written in different
languages were deadly and difficult to debug. 

Other than that it was a lot of fun. <G>

Obvious mistakes? There are a couple of problems, some of them might be:

- STATIC and AUTOMATIC variables behave different when entering a procedure.
- Pl/I stuctures are ALIGNED by default while COBOL structures are not SYNC. The
two languages insert their padding bytes in a defferent manner.
- You may CALL a COBOL program from PL/I without any problem, but not
vice-versa. You have to read carefully the ILC manual if  mix both languages.
- Be aware of the scope of names in PL/I.
- There are many defaults in PL/I. You may even use variables which have no
declaration and therefore they are declared implicitly by the compiler
accordeing to the first letter of their name.
- There are no reserved keywords in PL/I as are in COBOL. Whether a word in the
source is a variable or a program control atatement is determined by the
compiler. IF IF = THEN THEN THEN = IF * 2; is a perfectly valid statement.
- There are no 'verbs' in PL/I like 'MOVE', 'COMPUTE' or 'STRING'. There are
statements which end with a semicolon and blocks which end with an 'END;'
statement.
- Addresses are much more appreciated in PL/I than in COBOL. You'll find PTR and
BASED everywhere in PL/I sources
- Arithmetic operations in PL/I often end with FOFL conditions while COBOL
handles most of this type of error.

Conclusion: You might need to RTFM and/or need assistance by an experienced PL/I
programmer.
