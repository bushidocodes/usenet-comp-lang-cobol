I'd like to know if it is possible to use internal floating point numbers
that comply to IEEE 754 format
with CICS Cobol/2 on a IBM mainframe.

I know there are PIC COMP-1 and PIC COMP-2 which are supposed to be single
and double
floating point numbers, but the description I found wasn't compatible with
IEEE 754.

On VAX, when you compile a program, you can add a parameter to tell the
compiler which
format it should use : VAX or IEEE 754.

So, can IBM Cobol use IEEE 754 format ?

IEEE 754 format for a single (32 bit floating point number) is :
1 bit (32-32) : Sign / 8 bits (31-24) : exponent / 23 bits
(23-1) : fraction.

You can find a full guide on IEEE 754 here :
http://www.sns.ias.edu/Main/computing/compilers_html/common-tools/numerical_
comp_guide/index.html

IBM 360 floating-point arithmetic was designed in 1960, long before IEEE
754, and IBM can't or won't change it. Even at the time, other
manufacturers had superior floating-point arithmetic, but they're all
gone, and the 360 endures, for technical superiority doth not mean
jackshit in the market.

It's not just CoBOL, either. Any compiler emits the instructions of the
target machine, even ForTran or C.


› On VAX, when you compile a program, you can add a parameter to tell the

The VAX hardware has both, so VAX ForTran, VAX C, and VAX CoBOL can
compile into either.


› So, can IBM Cobol use IEEE 754 format ?

Since an IBM mainframe does not have IEEE 754 floating-point arithmetic
in the hardware, it would have to be done in software. If it means a lot
to you, you could write an emulator. Personally, when I have to write
floating-point for the mainframe, I don't bother with an emulator, I just
write it for the mainframe and put "DON'T PORT" in the comments.

S/360-S/390 floating point is sign-magnitude, base-16, excess X'40'
exponent (representing a range of -64 to +63), with a mantissa of 6
significant hex digits in single precision, 14 significant hex digits in
double precision. The implied hex-point is to the left of the
most-significant digit. Usual convention is to work with normalized
values, so the most significant digit of the mantissa would be nonzero
unless a numeric value of zero is represented.
E.g., a value of +18.5 (base 10) = 12.8 (base 16) = .128000 (base 16) *
16^2, so the the excess X'40' exponent would be X'42', making the single
precision hex representation X'42128000', while double precision would
be X'4212800000000000'. If the value were negative, the only difference
would be that the leading bit would be '1' making the leading hex digit
change from '4' to 'C' in this example. The usual representation for
0.0 is X'00000000' (positive, smallest possible exponent, zero
mantissa), although larger exponents are used in special cases where
deliberate truncation on addition is desired. There is also an extended
floating format, which I believe looks basically like two double
floating values.
