Here's my take.  It's based on the way I know to do control break
logic, so if you know another reliable way, please take this in good
grace.

First, the key between the two files is the vendor account.  In the
vendor file, the vendor account is defined as VEND-ACCT, and has a
single element under defined as VEND-ACCT-9, which is PIC 9(5).  This
is where I have some confusion.  Why does VEND-ACCT have a
sub-element?  Two different names to refer to the same storage
defined, effectively, the same way.  Why?

In 010-BOJ: the following important things happen
1:READ the payables file.
2:READ the vendor file.
3:ZEROS are MOVEd to PREV-PAY-VEND.
4:VEND-ACCT-9 is MOVEd the working storage field PREV-VEND-ACCT.

In 020-MAIN-PROCESS:

1-We determine whether (PREV-VEND-ACCT = VEND-ACCT) and (VEND-ACCT =
PAY-ACCT)

Right here, we have a problem.
There is an attempt to match the vendor-file against the payables
file, but there is also an attempt to match the vendor-files against
itself, probably an attempt at controlling the break when the a new
vendor is found.  However, at least as far as I know, without actually
reading the next record from the vendor file, and literally having two
records on hand at once, one an "active" record, and one "the next
record", this isn't easy (I've tried the way presented here, but could
never make it work).

In control break logic, a working storage field is used to "hold" the
value of the "active" record, or whatever elements of that record that
need to be passed on to other parts of the program or output files.
Why?
Because of the need to READ another record.  That way, the buffer
represented by the 01 record layout under the FD statement holds "the
next record".  We must know what is in the second record to proceed.
Once you have this, the program can then check in the main loop
whether the "active" record in working storage is different from "the
next record", and whether to do the break logic.

If the active record is different from the next record, do the break
logic, and print/write info on the active record, then move the next
record into the hold area, where it becomes the "active" record, then
another record is read and it becomes "the next record" against which
the "active" record will be checked.

If both are the same, accumulate info on the "active" record, move
"the next record" into the hold area, where it becomes the "active"
record, then another record is read and it becomes "the next record"
against which the "active" record will be checked.

The way this program is set up, the hold field is pre-set to be equal
with the key of the active record.  This assumes that the actual
physical 1st and 2nd records in the vendor file are the same, even
though what it's really trying to do, though it can't, is attempting
to set up a special first-time-through condition.  This is not
possible . . .

The active record is in the FD buffer, and info on "the next record"
is (no offense) "faked" in working storage for the first pass.  But
the reality is that there is no way for the program to really know
that the actual physical second record in the vendors file is
different or the same; therefore there is no correct way to set the
working storage hold field (AFAIK).

Try:

READ vendor-file
READ payables-file
MOVE vendor-file's ACCT to ws-vendor-rec (whole copy of record in
working storage)
READ vendor-file



IF ws-vendor-acct = payables-vend-acct
    accumulate
    READ payables-file
    any other needed stuff
ELSE
    control break logic
    any other needed stuff
END-IF


IF ws-vendor-acct < payable-acct
    MOVE vendor-file to ws-vendor-rec ("the next record" becomes the
"active" record)
    READ vendor-file                  (a new record from the
vendor-file becomes "the next record")
    any other needed stuff
END-IF

IF ws-vendor-acct > payables-vend-acct
    READ payables-file
    any other needed stuff
END-IF

Where "any other needed stuff" is code for other stuff that may being
going on that happens where it needs to (before or after the other
code I've put in).

Try this and see if things work out.  Come back and tell us what
happened, or what you did to solve it on your own.
