       - en 31 en el 1.

Lo siento que no puedo contestarle en Espanol, pero si Usted puede entender
Ingles, aqui es un possibilidad:

I am assuming you have an input string with two characters for each
hexadecimal digit, and want to produce a string of the ASCII or EBCDIC
equivalent.  So, the character string '3031' ASCII should result in an
output string of '01', and 'F0F1' EBCDIC should result in '01'.

You could construct a character string of all possible hex digits,
'0001020304....', and REDEFINE the string as an array OCCURS 256 INDEXED
I-X PIC X(02).  Then, use SEARCH ALL to locate the entry that matches your
first two input characters.  Define another array that contains the
Character equivalent and REDEFINE it as OCCURS 256 INDEXED O-X PIC X(01).
SET O-X to I-X.  Then, MOVE the output character from the second array to
your output string.

You could use Reference Modification to access each pair of input
characters and to access each output character.

Another variation would be to define a single array that has the result
character immediately after the two hex characters.  SEARCH ALL would still
work, and you would not need to SET one index to the other.  For example:
'...300311322333344...', and REDEFINE it as
05  ARRAY-ENTRY OCCURS 256 INDEXED I-X.
10  ARRAY-IN        PIC X(02).
10  ARRAY-OUT    PIC X(01).

Estoy seguro que otras soluciones existen.  !Buena Suerte!

Otra solucion, puede ser un poco mas rapido:

Using reference modification, move two characters at a time into a
PIC X(2) field which is redefined as a 16-bit binary integer.  Use
that integer as a subscript into an array of 4096 (= 256 * 256)
characters.

For ASCII conversion, initialize that array so that:

    A[hex 3330] = '0'  ('30')
    A[hex 3331] = '1'  ('31')
    ...

For EBCDIC conversion, it depends on whether you're running on an
ASCII machine or an EBCDIC machine.  If you're running on an ASCII
machine:

    A[hex 4f30] = '0'  ('F0')
    A[hex 6f30] = '0'  ('f0')
    ...

On an EBCDIC machine:

    A[hex c6f0] = '0'  ('F0')
    A[hex 86f0] = '0'  ('f0')
    ...

This would use more memory, but it might be faster.

You could have your translation program initialize the arrays, or
you could have a separate program create a copybook.

If your input is clean, you might be able to use one array for
ASCII and EBCDIC, since the codes don't overlap.  I'd recommend
having two arrays and initializing unused cells to low values (or
whatever) to catch bad data.
