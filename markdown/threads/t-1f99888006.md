```
I recently had a conversation with a programmer in another department,
and he tells me they are using dynamic calls with procedure pointers. 
This is using COBOL for MVS & VM, under 0S/390 release 2.8.

I've never seen this before, but apparently they also use EXTERNAL
working-storage items.  All programs that use exactly the same 01
EXTERNAL record definitions have them available in their working-storage
section without naming them in the call statement or procedure division
using clause.  The EXTERNAL 01 records cannot have VALUE clauses in
them, and they apparently reside in working-storage for both the caller
and callee.

The call is set up something like this:

01  SHARED-DATA-RECORD   EXTERNAL.
    05  WHATEVER-ELSE     ...

01  PTR-SUBPROG         USAGE IS PROCEDURE-POINTER.

SET PTR-SUBPROG TO ENTRY "PROGNAME"
CALL PTR-SUBPROG

My colleague tells me they've done extensive performance testing, and
this form of dynamic subprogram call with procedure pointers and
EXTERNAL data items is significantly faster than a standard dynamic call
(CALL MY-SUBPROGRAM USING PARM-1, PARM-2, et cetera).

Has anyone else used this technique?  Any comments on it?  
```

On the other hand, GLOBAL can only be used either on the program that
defines the variable as GLOBAL, or in a program contained directly or
indirectly in the program that defines the variable.

ex.
Prog-1.
WS.
01 var-1 pic x GLOBAL.

MOVE "2" to  var-1.

prog-2
proced.
display var-1.

prog-3.
proced.
move "3" to var-1.
end-program prog-3

end-program prog-2

end-progran prog-1

Note that prog-2 and prog-3 can only be called by a program contained
directly or indirectly in prog-1 or by prog-1.



So EXTERNAL is to use variables across programs that define the same
variable exactly with the same PIC.
GLOBAL is to define a variable in a "TOP" program, so that the
variable is available to all the programs contained inside that
program.
