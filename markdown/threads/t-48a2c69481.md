```
Parts of this thread seem to be "out of order" - but to test a single data
item for two values, there are two "valid" (conforming" ways of doing this,

Evaluate Data-Item
  When "Value1"
  When "value2"
      Perform Either-1-or-2
 End-Evaluate

   *OR*

Evaluate True
  When Data-Item = "Value1" or "Value2"
    Perform either-1-or-2
End-Evaluate

the one thing you may NOT do  is

Evaluate Data-Item
  When "Value1" or "Value2"
    Perform either-1-or-2
End-Evaluate

   ***

Actually, there is also a 3rd technique which is using 88-levels, eg.

01 Data-Item Pic X(10)
    88  Good-Value  "Value1" "Value2".

Evaluate True
   When Good-Value
      Perform either-1-or-2
End-Evaluate

With only "two values" to be tested, which you use is pretty much a matter
of choice.  With more complex situations, you really can find one "clearer"
than the other.
```

```
------------------------------------------------------------
PROGRAM-ID. Evaluate1.
WORKING-STORAGE SECTION.
01 char                         pic x.
   88 Afield                    value "A".
   88 BField                    value "B".
   PROCEDURE DIVISION.
move "A" to char
perform CHECK-IT-1
perform CHECK-IT-2
perform CHECK-IT-3
perform CHECK-IT-4
move "B" to char
perform CHECK-IT-1
perform CHECK-IT-2
perform CHECK-IT-3
perform CHECK-IT-4
STOP RUN.

CHECK-IT-1.
Evaluate true
    when char = "A" or char = "B"
        display "Check1 - I've got A or B"
    when other
        display "Check1 - Test failed"
End-evaluate
.
CHECK-IT-2.
Evaluate true
    when char = "A"
    when char = "B"
        display "Check2 - I've got A or B"
    when other
        display "Check2 - Test failed"
End-evaluate
.
CHECK-IT-3.
Evaluate true
  when Afield
  when BField
        display "Check3 - I've got A or B"
    when other
        display "Check3 - Test failed"
End-evaluate
.
CHECK-IT-4.
Evaluate true also true
    when char = "A" also char = "B"
        display "Check4 - I've got A or B"
    when other
        display "Check4 - Test failed"
End-evaluate
```

```
Just to clarify, the COBOL85 compiler on the Unisys NX5600 generates, for
    EVALUATE REPORT-CODE
    WHEN "A" PERFORM REPORT-A-CODE
    WHEN "B" PERFORM REPORT-B-CODE ...
the same object code as for
    EVALUATE TRUE
    WHEN REPORT-CODE = "A" PERFORM REPORT-A-CODE
    WHEN REPORT-CODE = "B" PERFORM REPORT-B-CODE ...
which for this simple case is the same code as for
    IF REPORT-CODE = "A" PERFORM REPORT-A-CODE
    ELSE IF REPORT-CODE = "B" PERFORM REPORT-B-CODE ...

All three of these avoid the "retest" that
    IF REPORT-CODE = "A" PERFORM REPORT-A-CODE.
    IF REPORT-CODE = "B" PERFORM REPORT-B-CODE.
entails.

And, by the way, there is no assembler on the Unisys NX-5600, or any of its
ancestors, nor is there a BNE operator among the mnemonics for the hardware
instructions.  The mnemonics for the code generated for testing the result
of these single-character comparisons are EQUL, BRFL.
```

```
The draft standard *does* allow (but not require) compilers to "flag"
(reject? other?) when a program violates a "semantic" rule (not a syntax
rule), e.g.

01  Field-0  Pic 9 Value 0.
  ..
Compute SomeNumeric = 10 / (field-0 - field-0)

(i.e. divide by zero)

However, there is not (and I hope there never is) a REQUIREMENT that
compilers have to do "full logic" checking.  Taking this example, would you
want it to flag

01  Field-A  Pic X Value "a'.
01  Another-A Pic X Value "a".

Evaluate True Also True
  When somefield = Field-A also somefield < Another-A or > another-A

Must it check to see if any of these field are ever modified?  How about if
a CALL statement *might* give addressability to where these fields are
defined?

Bottom-Line:
  Compilers check SYNTAX (and *may* give you "helpful hints" when something
is EASY for them to do - given their particular compiler design);
  Compilers do NOT check that you know what you are coding or that you
understand run-time logic.
```

```
1. Compiler writers should do everything possible to assist programmers and
```

```
My reading of this issue is that the syntax can be generalized to be
    EVALUATE TRUE ALSO TRUE
    WHEN <some completely arbitrary conditional expression>
        ALSO <some other completely arbitrary conditional expression> ...

While it might be possible for some compilers to detect an impossible
relationship between the two boolean expressions in this example, I do not
feel it is the obligation of any to do so.  Precisely because the
conditional expressions are arbitrary, and more particularly there is no
requirement that there be, or not be, any commonality of operands between
them, I would suggest that a compiler that issued a syntax error in this
case would be guilty of precluding something that the standard allows!
Whether the two arbitrary conditional expressions are mutually exclusive is
none of the compiler's business; that's a programmer issue.

I would personally not be in favor of making the conditional expressions in
this case any less arbitrary than they now are,  in either syntax or general
rules.  I wouldn't object if some vendors issued an "oh, by the way ..."
warning, but it's perfectly legal for the user to code a combination that
can never actually exist, just as legal as coding a terminating condition
for a PERFORM that can't ever be "set" from within the PERFORMed logic.

    -Chuck Stevens
```

```
Subject:	"Partial expressions" in EVALUATE

Author:	Stephen J Spiro
	Wizard Systems


References:

1. Base Document:  FCD

PROPOSED REVISION:

Make "=" optional when it is the leftmost operator in a partial
expression.
 
JUSTIFICATION:

In a discussion in comp.lang.cobol of the new features in EVALUATE,
several otherwise knowledgeable participants (blush!) made the mistake
of including examples which omitted the equal sign in partial
expressions. For instance,
		EVALUATE	something	ALSO	something-else
		    WHEN     	A or B		ALSO	Field-C and Field-D
				Do....

Mr Klein and others pointed out that, since the rule is
"5) A selection object is a partial-expression if the leftmost portion
of the selection object is a relational operator,
a class condition without the identifier, a sign condition without the
identifier, or a sign condition without the
arithmetic expression.",

the current syntax would require
		EVALUATE	something	ALSO	something-else
		    WHEN    	= A or B	ALSO	= Field-C and Field-D
				Do....

The omission of partial expressions for EVALUATE in the last Standard
has been the cause of much foul language in programming organizations,
and this correction of the omission was solidly in place before I
joined the committee.

Requiring the physical expression of the EQUAL operator will continue
the foul language; there is no need for it, and it "reads funny".

There has never been a requirement for the equal sign in a case such
as
		EVALUATE	something	ALSO	something-else
		        WHEN	E		ALSO	Field-F
				Do....
But, meta-syntactically, "E" is the same as "A or B", and "Field-F" is
the same as "Field-C and Field-D".

Let's fix it now, before the complaints flood in as the new standard
is implemented.

COMPATIBILITY:

This feature is compatible with ANSI X3.23-1985 as amended by ANSI
X3.23a-1989 and ANSI X3.23b-1993.

"Partial condition" is not in the index. I believe the term is not
used anywhere in the standard except under the EVALUATE instruction.

PROPOSED CHANGES TO BASE DOCUMENT:

1. Page 449, 14.10.12.2, Add at the end of General Rule 5.  "If the
leftmost operator in a partial expression represents equality, that
operator may be implied; that is, it need not be coded explicitly."


RESOLVED ISSUES: 
This will handle the equal sign, EQUALS, IS EQUAL TO, and all the word
permutations.  Note that if the operator is negated, the "NOT EQUAL,
etc" would still be required, because that operator is INequality.


OPEN ISSUES:	None
```

```
The problem here is not with EVALUATE, it's with the choice of construct.

Use IF.

if  status-field = 'A'
            OR
    status-field = 'B'
         perform A-or-B-process
end-if

other statuses that "map" to the same or different processes? (EVALUATE is
ideally suited to such a scenario...)

Loadsa options...

1. Change the "end-if" to an "else" and add further "if"s...

2. Let the next poor sod who maintains it deal with it...

3. Convert each status-field value into a binary number then use GO
TO...DEPENDING ON...

you want it to be processed by, then set a pointer with the status field and
branch on it. (Very elegant...really like this one...)

conditionals in EVALUATE will be supported, and you can say WHEN 'A' OR 'B'
(Good option...by the time this happens you won't be working there...in fact
```
