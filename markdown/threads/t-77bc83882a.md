This discussion is from an IBM mainframe perspective, i.e, MVS & VSE
implementations.

In case I mix "block" and "buffer", a "buffer" is an area of memory that (in
this context) will hold a record (unblocked files and unit-record files such as
card readers, card punches, and line printers) or a "buffer" will hold the
equivalent of a disk block of data (for blocked files, typical of disk and
tape). For example, if a file is defined with a record size of 80 and a block
size of 23440 (MVS terminology: DCB=(LRECL=80,BLKSIZE=23440,RECFM=FB)), a
buffer of 23440 bytes will be used (double-buffering: two buffers of 23440
bytes each), and the data in the buffer, when written, will write a disk block
of 23440 bytes, or the equivalent of slighly half of half of a 3380 track. So
one could say that the buffer "holds a block of data".

Typically, records defined in the FD are actually pointers into a buffer, and
the buffer is accessed in locate mode, meaning that the 01 in the FD is set to
the address of the logical record being processed.

For unit record devices (card reader, line printer), if double buffering is
used, the FD's 01 alternates between the two buffers, e.g., when the file is
opened, two buffers are allocated (if not already reserved) and the 01 is set
to point to the first buffer. The values are moved to it, a WRITE causes a
Start I/O operation (or EXCP) and the 01 is moved to the other buffer. (In
COBOL terms, a WRITE FD_rec_name FROM ws_rec_name is equivalent to a MOVE
ws_rec_name to FD_rec_name followed by a WRITE FD_rec_name.) Data is moved to
the buffer that the 01 now points to and when the second WRITE is issued, the
job waits until the first SIO completes, then the 01 is moved to the first
buffer and a SIO is issued against the second buffer. In this way (using
double-buffering), one of the buffers can be in the process of being filled
while the other buffer is tied up with I/O, and the two buffers alternate
roles. The theory here is that the program can be doing serious work and an I/O
can be taking place at the same time, producing a net throughput improvement
over a single-buffering situation.

The practical side is that after moving values into an 01, after a WRITE is
issued, the 01 no longer points to the same buffer, so one cannot look at the
01 to see what the value was that was just written.

CLOSE guarantees that the SIO operation will be allowed to complete before the
program is deallocated and memory (including the buffers) are deallocated.

Single buffering seems more reasonable because the 01 for a unit-record device
would point to just one buffer. But in this case, once values are moved to the
01 and a WRITE is issued, the program is suspended while the SIO is taking
place and the program is not allowed to continue until the I/O operation is
posted as completed. If dealing with real devices (instead of a spooler
system), there would be no overlap between processing and I/O (the processing
would be suspended until I/O completes), so the overall runtime of the job
would be increased.

SAM files on disk or tape, unblocked, single or double buffered, would run
typically as described above.

For SAM files (disk or tape), it is common for the file to be blocked. At OPEN
time, one, two, or (MVS) more buffers are set aside for I/O, and the FD's 01 is
set to point to the start of the first block. Let's consider two buffers. When
data is moved into the 01 it is actually being moved into the first area of the
buffer. (If the 01 record is 80 bytes long, data would be moved into the first
80 bytes of the buffer.) When a logical WRITE is issued, the 01 is moved to the
next area in the buffer. (E.g., the 01 is moved 80 bytes into thebuffer.) When
there have been enough logical WRITEs that the entire buffer has been filled,
the 01 is moved to the start of the second buffer of memory that was set aside
for I/O, and a Start I/O operation (SIO or EXCP) is started to physically write
the first buffer to disk or tape. As additional logical writes take place, the
second buffer is finally filled, at which time (for double buffering) the
program is suspended until the first buffer had finished being written to disk
or tape. Once the SIO for the first block has been posted as completed, an SIO
is started to write the second buffer of data, the 01 is updated to point to
the start of the first buffer of data, and the program is allowed to resume.

By blocking data (having multiple records saved into a buffer so that when an
SIO starts, the block of data it writes contains multiple logical records),
fewer SIO operations (VSE terminology: EXCPs) are required for writing the same
amount of data, so less operating system overhead is required, fewer tape
start-stop operations are required, fewer inter-block gaps are used (both disk
and tape), with the result that data density (disk and tape) is generally
significantly better and the amount of time spent doing physical I/O and
supervisor overhead for I/O is significantly decreased.

However, that means when a COBOL program does a WRITE, the value just written
will not be available in the 01 again until several or even hundreds of WRITE
operations later.

A CLOSE operation will check to see if the buffer the 01 currently points to
has had one or more logical WRITEs issued to it since the 01 was readdressed to
it and, if so, will write the buffer out to disk (typically a short block,
i.e., the block written will be the size of the buffer that the I/O routines
think were actually used), as well as guarantee that the most recent SIO
completes before control goes back to the program.

I haven't covered more-than-2 buffers (buffers can be filled further in advance
of the SIOs but still an SIO will have to be completed before a buffer can be
reused), nor have I covered VSAM (read-only non-spanned: 01 points to the
buffer holding the data CI and the 01's pointer is updated for each sequential
READ; for spanned records or output: a secondary buffer the size of the largest
possible record as defined in the VSAM catalog is allocated and the 01 points
to that buffer, but it is not recommended to rely on this particular behavior).

On input files, the 01 may point to an "indeterminate location" once an
end-of-file is reached, so it is a bad practice to move anything to the FD/01
of an input file, especially once EOF (sequential) or INVALID KEY (random read)
has been reached.

The story is completely different for WORKING-STORAGE. The behavior of
WORKING-STORAGE is very deterministic: the location does not change for the
duration of the program. If one does a WRITE ... FROM ws_rec_name, the value of
ws_rec_name will not be changed by the WRITE (unlike what the value of the 01
because the address of what the 01 points to can be modified on every WRITE).
That means that after the WRITE ... FROM ws_rec_name, you can still access the
values just written by examining ws_rec_name. Likewise, when doing a READ ...
INTO ws_rec_name, you are allowed to make subsequent changes to values in
ws_rec_name, even when AT END (i.e., end-of-file) has been encountered. (A
common trick for merges is to move high values into the key of the
WORKING-STORAGE layout for the file whose eof one just encountered--you cannot
do that to the FD/01, but you can to the WS/01.)

Another benefit of having output record layouts be in working storage is that
one can use VALUE clauses or do once-only MOVEs of initial values to that
record layout, and it will stay there. (FD/01 VALUE clauses, except for 88s,
are treated as comments or warnings.)

My preference is to have record descriptions in working storage. In olden days
when one got hex dumps, it was a lot easier to see what record one was
currently working on when it was in WORKING-STORAGE because it would be in a
specific location, but one would have to decode control blocks if one had to
look at the file buffers to see what record the program was working on at abend
time. This probably isn't so important today when using COBOL/VSE or COBOL/MVS
and the Language Environment, and the code had been compiled with CBL
TEST(,SYM) because then the Language Environment dump routines would (should?)
figure out which record in the file buffers was the currently active record
(i.e., which record the 01 was really pointing at). Oh yes, I remember coding
all kinds of eye catchers, e.g.,

  01 FILLER PIC X(32) VALUE 'WORKING STORAGE STARTS HERE:'.
  ...
  01 FILLER PIC X(16) VALUE 'INPUT RECORD:'.
  01 WS-IN-REC.
       ....
but those eye-catchers aren't needed with TEST(NONE,SYM).

Even so, for debugging purpose, it is critical that one is able to look at the
raw, unadulterated input record currently being processed, so if _any_
modification of the input record takes place, it is better (for debugging) if
the record is moved into WORKING-STORAGE (e.g., READ ... INTO) and the
WORKING-STORAGE version be modified, while letting the FD/01 remain as the
virgin record image.

Mark A. Young

You probably got into the habit of working with data in working storage
because some wise, experienced programmer most likely told you, way back
when you were starting out as a rookie programmer, that you should "NEVER
work in the buffer". This was (is!) good advice because data that was in a
record buffer before a WRITE was not necessarily there after a WRITE.  Many
COBOL compiler manuals have stated that the contents of the FD 01 were to be
considered "undefined" after a WRITE operation.
