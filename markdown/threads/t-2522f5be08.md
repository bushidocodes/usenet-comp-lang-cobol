The reason that there is no open source COBOL compiler is because it is
such a powerful language.  It is very difficult to write a compiler for
it.  The progress made on open COBOL compilers is really making good
progress.  It just doesn't appear to be.

The C language has only about 40 odd function points.  It is a
primitive language.  COBOL has about 300 or more function points.  Some
of those function points are very complicated.  In fact, the COBOL
standards committee keeps bringing up "function overload" as a
problem.  The INVOKE function point was created in order to not
overload the CALL function point.  My rough estimate is that COBOL is
20 to 50 times more powerful that C, C++, C# or any homologs like
Java.  Maybe 100 times more powerful.
Hence it takes 50 times more effort to produce a COBOL compiler.

People like Bork remain ignorant of this fact.  Businesses are
only now beginning to realize that COBOL is a powerful tool that
increases developer productivity, especially in business areas.  Over
the long term COBOL centered businesses are more competitive that C
centered businesses.  Bork focuses on academia.  Business focuses on
practical productivity.

Now let's be fair to academia. Their primary job is to teach people how to
learn, not to give them a specific skill. Nor is the curriculum based on
'fun', it is based on 'fundamentals'. As others have said, if you want to
learn a trade go to a trade school.

My first computer-related course, more years ago than I care to admit, was
called 'Numerical Methods'. Assignments were to be done in Fortran. No class
time was spent on the language. We were expected to muddle through that on
our own, with some help from the overworked teaching assistant. The
following semester there was a 'Digital Systems' course which involved
writing algorithms in M6800 assembler, but learning the language, or the
VM/CMS operating system so we could use the cross-assembler,  was not part
of the curriculum either. I had to teach myself another 3 or 4 languages and
as many operating systems before graduation. Anybody else coded DART
applications for a Zilog Z80?

I also take exception to the term 'powerful' as an adjective to COBOL. I
would support the use of 'effective' or 'verbose'. COBOL relies on verbs.
Most modern languages rely on functions. COBOL waits on the whim of compiler
vendors, and a standards committee that moves slowly to adopt change.

Pragmatic organizations will continue to use COBOL regardless of what some
confrontational news article says. However, as organizations adopt packaged
solutions, or migrate to newer database technologies, the relevance of COBOL
will diminish. In the meantime, COBOL still has a long life ahead.

I intentionally use the adjective "powerful" as it is appropriate.
COBOL does decimal arithmetic.  C does not.  In C, one has to write
their own subroutine.  Try explaining to a CFO why the numbers don't
come out exact on his statements because C uses floating point
processing.

To sort a file in COBOL, simply put:
SORT FILE-SORT
    INPUT PROCEDURE BRING-IN-FILE-A
    OUTPUT PROCEDURE WRITE-OUT-FILE-A.
Notice that this allows one to dink around with the records as they are
coming and going.  What can one do in C besides calling an operating
system command to sort the file?  Can one dink with the records?
Write a sort subroutine in C and it could cost one weeks of development
time.  In COBOL one does it in minutes.  One also has the advantage in
COBOL that most all COBOL compiler manufacturers use the most
sophisticated sort algorithms known to man.  Does every C developer
know these when they write their own sort routines?

COBOL allows one to put keys into records and retrieve them
dynamically.  In C one calls a relative database package to do that
sort of thing.

COBOL allows one to search a table by typing
SEARCH ALL TABLE-A
    AT END DISPLAY "NOT FOUND"
    WHEN TABLE-A-KEY(INDEXER) = "ABCD"
        DISPLAY "FOUND IT".
When COBOL programmers see this they instantly know a table is being
searched and how it is being searched.  In C, one would see a call to a
subroutine that a developer wrote.  One would have to check the
subroutine to see what is going on.

COBOL has built in numeric formatting features.  In C, one must write
their own.

I could go on and on.  In the interest of bandwidth I'll stop.

COBOL does an abundance of work for the developer.
C just gives one the frame of a development environment.

COBOL is much more powerful than C.  Always has been.

The introduction of OO into C to make C++ was primarily to overcome the
limitations of the C language.  For example if you want a 'package' of
complex numbers in C you have to do it with functions.  

With built in types one can write expressions, ( a + b ) * c, where a, b
and c are of integer or float types, long or short.  If one creates a
'type' (usually a struct) for a complex number, or long decimal, then
these cannot be used in expressions and you have to use functions:

          cmplxaddint(complex1, integer);
          cmplxmultflt(complex1, float);
          ....

for _every_ combination.  This soon gets very tedious.

With C++ the complex or long decimal can be made an 'object' and the
expression operators can be overloaded so that these can be used in
expressions.

In COBOL a MOVE will work with every type that can be described.  All
numeric types can be used in any COMPUTE: packed, float, fixed decimal,
in any combination.

COBOL never had the limitations that C++ was developed to overcome in C.

Now it is true that OO has more general application such as information
hiding and inheretence.  These can be done just as well without OO
language support, whereas overloading cannot.  

The problem with OO is that C++ becomes a tool with which a programmer
can develop their own computer language.  They can define what the
operators are to mean, it is then that the language manuals will not be
able to tell you what a piece of code will do, it will be necessary to
refer to the documentation, or actual code, of several objects.  While
_superficially_ and in small projects it appears that C++ can simplify
the code, underlying this is all the complexity of the classes which may
be unique to each site or system.

I prefer to write in a deterministic language, such as Cobol.

I have practical experience in maintenance of poorly written
programs done in both COBOL and C. I can tell you, from
that experience, that ease of maintenance is determined
much more by the choice of developer than the choice of
language!

A good analyst will make fewer errors of omission when
writing the requirements. Sometimes omissions are
corrected during the maintenance phase of the life cycle.
There is nothing easier to maintain than a system that
requires no changes.

A good system designer will partition a system to prevent
interdependencies. Fewer interdependencies means
easier maintenance.

A good program designer/coder will reduce coupling,
increase cohesion, use meaningful names, test, test, and
test again to assure conformance with specifications and
to remove "bugs"; all of which eases maintenance.

Choosing a language that is expressive in the application
area eases understanding which eases maintenance.

In conclusion, if everything prior to the maintenance phase
is done well, the choice of language is almost irrelevant.
If everything prior to the maintenance phase is done poorly
then choice of language may be important. A business
application that was done correctly and well in C, is
easier to maintain than a business application that was
done incorrectly and poorly in COBOL.
------------------
Rick Smith
