```
It seems that most of our biggest and most expensive problems in our
industry are when big, integrated systems fail.  Since everybody KNOWS
that big, integrated systems are good, we spend lots of effort
developing better ways of making it more likely that those systems won't
continue to fail the way they do.

I don't see much talk about analyzing how to eliminate dependencies, to
analyze risks in having such dependencies, or even to isolating various
systems so that a failure in one is not a show stopper.

Data marts are useful here.  If you need data from disparate systems,
these can be used as the go around.  But this cannot do everything.  If
we want to make sure a student doesn't get his diploma or refund until
we check each campus to see if he has any outstanding problems, we need
to have some way for different campus's to talk to each other in
transaction time.  So businesses demand immediate connectivity, and we
give them constant program dependencies.  Which means constant bug
dependencies.

OO seems designed to make these dependencies less dangerous.  But
somewhere in our analysis we should take time to decide which
dependencies can be avoided, which dependencies can be isolated (TCP/IP
like or via data marts), and which we just have to live with.  

Our psychology tends to be - we can solve these problems, let's be
direct and allow the dependencies.  As long as there aren't bugs (not in
MY design), it should work fine.
```

2. DONT over design or program, DONT solve tomorrow's problem today,

5. Refractor. This means find common code and put it into subroutines, or

   It worked for this universe!

6.  The XP process starts with CRC cards.  The design is complete when
you begin but just very light weight.  The finer detail is worked out
with the user that must be on the spot in an XP team.

Extensibility argument:

a.  Because the system has such a heavy duty testing regime you can
safely totally reengineer the system with a guarantee that it will
work correctly in production.  The major cost of redesign is the
testing.

b.  The regular rework of the smaller details to make the system more
usable means that you should not have to make such drastic
modifications to the whole infrastructure.  It is slowly evolved to
the new infrastructure and methods that would not have occurred to you
at the outset just appear using Darwin's theory of evolution.

c.  You often make a mistake about what is going to need it.  As
outlined the flexible error messages that never get changed.  I
personally wrote a totally parameter driven formatting engine.  It was
never changed in it's entire life which terminated recently because
the upstream system was not up to scratch.  That exercise cost at
least $30K more to build and it was wasted.  Wait until you need it
and then build it.

d.  We often only know about reuse after we have developed the
solution a few times. I have a report pagination routine now, I have
literally written 100's of reports before I designed this.  If I had
refactored after the first 100 I would have saved a lot of time in the
long run.

e.  There is no firm rule that says you cannot design a better
solution in XP.  You must convince your partner and this stops some of
those side trips that programmers love to take.

As per usual, my thoughts greatly exceeded my hands ability to type.
Whole books are written on refactoring.
But, basically, when refactoring, you are doing one of two things:
1. Creating common code that may just be a different method in the current
   class.
2. Creating common code that is being placed into a new method in a class
   higher up the class hierarchy. This in fact may require a class to be 
   inserted immediately before your class to contain the common code.
   This code can then be tailored by your class.

Of course, where you read CLASS you can also use OBJECT, for the current
class, with slightly different ways of achieving the same effect.
I.e. object imbeds object.
