Do you know the LRECL of your file? (preferably both the smallest and
largest) Then assuming the smallest (withOUT the RDW 4 bytes) is "n" and
the largest (again withOUT the RDW) is "m". If you code an FD with

Record contains n to m characters

or if you code

Record varying in size from n to m
depending on WS-field-to-hold-current-record-size

You shouldn't get any fs=39. My guess is for the problem is one of the
following
A) your "record contains" clause is a lie (OS/VS COBOL was VERY forgiving
about this)
B) you have an 01-level under the FD that is outside the REAL size your
file can ever have
C) you are including the RDW in your calculations on what to have in the
FD.
D) your FD is assuming an unblocked rather than blocked record. (I
think - but have NOT checked - that BLOCK CONTAINS 0 RECORDS should solve
this)

COBOL II is a lot pickier about these things than '74.

If your file (for example) is VB,lrecl 1004 then you must have a record
defined as 1000 bytes under your fd. You should be receiving a message
telling you the size record it is expecting. Something like ' An Attribute
mismatch detected. The file input-file-name has a length of xxx bytes and the
file from the assign clause is yyyy bytes in length.

You can always define a dummy record under the fd:

FD your-input-file.

01 dummy-record pic x(1000).

01 the-real-record.
05 field-1 ......

Good luck,
