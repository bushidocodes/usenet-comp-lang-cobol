The basic issue is one of how to pass information efficiently between
two separate but related transactions. One could of course in all
cases, including the simple one described above, pass the information
via a file. Such an approach will work as long as you have a low enough
transaction rate, but this simplistic approach scales upward poorly.
CICS subsystems can support very high transaction rates if they are not
throttled by avoidable file activity, so it makes sense to teach how to
use the features of CICS that avoid I/O. Even with adequate buffering,
all writes to the file are still sent to the I/O subsystem. At some
point the drive containing your "communication" file is hopelessly
saturated and response time goes off the chart. Long before that point
is reached, you will have been forced to effectively dedicate an entire
DASD volume to that communication file, because no other application can
tolerate the bad performance caused by contention with that file.

The rewrite to the file, as stated below, is probably the best solution,
since it will even survive after the CICS region goes down, and is brought
back up. There are very few things other than a file which survive the
machine being turned off.

However, if you are looking for something ingeneous other than a file,
check out using a temporary storage queue. This is a sort-of scratch pad
area maintained by CICS (sometimes in core, sometimes on disk) which can be
addressed by name. You can write an entry, read it back, and even rewrite
it. It is also shared region wide, so multiple users can use the same
counter. The Command level EXECs are very simple, and should be documented
in your text.

An even more esoteric place to hide a counter is on the screen of the
terminal. Define a field on your map (presuming BMS), with the attributes
autoskip, protected, modified data tag, and dark. You will get an area on
the screen in which you can place a text item which teh user will not be
able to see or modify. Because you have set the MDT bit, when you receive
the screen (or the data buffer), you will get the dat you wrote back.
While this sounds a little weird, it actually works (we used to hide the
security profile for the user like this in a system I worked on at Farm
Credit Services). In theory you can use the screen buffer to hold a whole
1919 characters (if you don't want to display anything useful, of course).
