Options are as follows

1 - Use the NVL function to convert the retrieved NULL to another
value, such as zero or blank. This is the simplest solution.

2 - Use an indicator to record the presence of the NULL. You probably
should use this option when you want a specific action to be taken
when a NULL arises.

3 - Revise the cursor definition so that no columns possibly
containing NULL values are retrieved.

1 or 2 are the only options and they will probably imply less code
changes.

You will _still_ be using Pro*COBOL because that is the pre-processor
supplied by Oracle. Micro Focus and Oracle do testing to make sure thier
products work properly together. Look in the section "Database Access" in
the MF documentation. Also see posts on this forum by Simon Tobias or
"wiggy" for answers to specific questions that have come up here in the
past.

In short - the difference between CONTINUE and NEXT SENTENCE is this:

CONTINUE tells the current structure (IF, EVALUATE etc) to stop what it's
doing and processing continues at the next statement regardless of any "Full
Stops" (periods to our American friends).

NEXT SENTENCE is similar with the important exception that processing
contunues AFTER the next full stop/period.

Some people put full stops/periods after exvery statement. CONTINUE and NEXT
SENTENCE would appear identical to them.

On the other hand, some people only put full stops/periods at the end of a
paragraph. In this situation the two statements would behave very
differently and would make a choice as to which statement to use for the
required effect.

Some people are inconsistent with thier full stops/periods (*cough*) and
need to be very careful which they use.

Hope this helps.

Just a clarification.

Compilers that allow a NEXT SENTENCE with an END-IF are *not*
"non-compliant" - as long as they document such as an extension *and*
provide a "flagging mechanism" for detecting this (and other) extensions.

As I have stated before/elsewhere, it is my impression that there are more
current COBOL compilers that *do* allow NEXT SENTENCE with END-IF (as an
extension) than those that don't.  I even believe that MOST (not all) of
those with this extension process it correctly and distinguish (as Howard
describes below) the behavior of CONTINUE from that of NEXT SENTENCE in such
cases.

Hello Tom

I agree with what the others have said, but would add the following
comments:

1)

If you do end up having to test individual SQLCODES, it would best be
done with 88 levels, SQLCODES are largely if not entirely implementor
defined, which is why SQLSTATE was later introduced with the intention
that it should become the preferred means of testing results.  The SQL
standard provides an explicit list of SQLSTATE values for the results
of executing SQL statements and though implementors may have added
some of their own (I am not sure of this), they are largely common
between platforms thereby considerably improving portability.

Having said that, some implementors may have compatibility modes for
making their products look like other products in respect of SQLCODES.

The use of a installation standard copybook with common 88 levels for
SQLCODES is the next best thing to using SQLSTATE, since generally
only the copybook needs to be changed (and all programs recompiled) to
handle the different SQLCODES when porting from one product to
another.  There may be complications when using 88 levels to specify
ranges, since the ranges may differ in nature between implementors.

2)

The EVALUATE statement often provides a clearer way of handling
multiple conditions arising from an SQL statement, for example

EVALUATE TRUE
   WHEN SQL-OK
      CONTINUE                *> execution will resume after
END-EVALUATE
   WHEN SQL-NO-ROW-FOUND
      take appropriate action
   WHEN OTHER
      take appropriate other action
END-EVALUATE.

3)

While the NVL function (of which I have no experience, is it a
precompiler feature to automatically convert all nulls to some other
value when returning values to host variables?) may well do what is
required for your platform, the standard SQL COALESCE function is a
better (and more portable) way of handling individual nulls in
statements, when you just want to substitute a blank or other such
value for a NULL.

It is used in the following fashion

SELECT col1, COALESCE(col2,' ')
  FROM tablex
  INTO :hostvar1, :hostvar2

The COALESCE in this example will put the value from col2 into
hostvar2, except that when col2 is null, it will put the ' ' into
hostvar2 instead.  Read the documentation for further details of how
to use COALESCE in more complex situations.
