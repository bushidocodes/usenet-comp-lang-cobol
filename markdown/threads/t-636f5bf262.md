```

›› In this case, it is a basic fact of computer hardware that all fields
›› roll over, like the odometer in your car.  A two-byte two's-complement
›› binary field rolls over from 32767 to -32768, a two-byte packed-decimal
›› field rolls over from 999 to 000, etc.  If you don't know this, you will
›› get into trouble in any language, not just in CoBOL.

Wrong. On some systems you get an arithmetic fault and your program
terminates. One should never never count on "rolling over" because it
might work today but may not work tomorrow. On RISC machines it is
even worse.

Don Nelson
```

Wrong. If you load a two-byte two's-complement binary field with a value
of 32767 into a 32-bit register and add one you get 32768, not an integer
overflow trap. When this number is stored back into the field, it
becomes -32768. On a RISC machine with 64-bit registers, it is even
better.

Seriously, CoBOL runs with integer overflow disabled because of the ON
SIZE ERROR phrase.
