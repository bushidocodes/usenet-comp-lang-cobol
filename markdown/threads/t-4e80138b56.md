I cannot speak from experience of these routines themselves (but I've
used other memeotry handlers...)

Looking at the specifications it seems to me that what you should be
doing is:

1. get address of memory that you want to make visible to others:
either by ADDRESS OF or CBL_ALLOC_MEM, you just need the POINTER to it

2. use PUT_SHMEM_PTR to associate the POINTER from (1) with a name
that is then visible by other programs.

3. the other program(s) use GET_SHMEM_PTR to recover the POINTER
associated with the name. Of course these "other" program have to
"know" tha names and what the named node refers to, i.e. its structure
etc. so that it can use SET to map the POINTER to some linkage section
memory (as with other POINTERS)

So: if program A has two blocks of memory it wishes to share; it calls
PUT_SHMEM_PTR to tell the world about "nodes" FRED and MARGARET
passing the name and the address of the area with each call. The
program that wants to access FRED refers to it in a GET_SHMEM_PTR, but
that only retrieves the POINTER and it has to be mapped to a correctly
structed memory area (a good case for COPYfiles, of course). If it
doesn't want MARGARET it doesn't need to access it at all. And so on.

I don't have any experience of VFILEs...

I have followed this thread with some interest, as data sharing under
Windows/32 is one of my "hot buttons." You can read my cleverly-titled
article, "Data Sharing in 32-bit Windows" if you obtain a copy of the
October, 1999 issue of BASICally Speaking magazine. Reprints available from
the publisher www.infoms.com, follow links. (Reprints also available from
me, since I own the copyright).

But I digress...

If you are working under 32-bit Windows and your brand of COBOL can call a
standard Dynamic Link Library (the whole Windows API is a bunch of DLLs),
Windows itself contains a vast number of functions to allow *cooperating*
processes to share memory: DDE, memory mapped files and atoms among others.

I emphasize *cooperating* processes in that most of these methods require
the creation of a named object of some sort; the cooperating applications
would each need to know the name used. (via COPYLIB, or even reading a disk
file to get the current name).

There is no reason these functions could not be incorporated into a single
DLL, and it could be written so that the COBOL programmer's total set of
"things to do" would be:

1. WS entry for name of object.
2. WS table of action/return/error codes and literal values.
3. These "call" rules (subject to design refinement; also your syntax may
vary for WinAPI calls)
     CALL 'foo'  USING
          ACTION,
          Dll-name
          Object-name,
          Pointer-to-data,
          length-of-data
     RETURNING Result

Action = Action-initialize >>  gives the program access to the
library/shared data
          = Action-get        >>   returns pointer-to-data and data-length
(fills a buffer)
          = Action-store    >>    pass pointer-to-data (buffer) and
data-length
         = Action-terminate >>  close off this program's access to the
shared data

Result would be based on the value of action; i.e, success, failure,
something like that.  Gets and Stores would have mutual exclusion to
guarantee data integity.

Using this kind of design, you don't even have to worry about linking the
code (although you 'd no doubt have to make some entry somewhere to indicate
an external DLL is used by this application), and you'd have to insure the
DLL was available at run time. (There is a way around that, too, but that
would require a direct call to Windows from your code).

To make this useful across multiple-source-code-language programs you
currently have installed, the DLL used should not rely on the COBOL or C or
whatever run-times are required to operate the individual programs - it
should use only "pure" Windows calls. (The same DLL could be used by
programs written in different languages as long as they all have the ability
to call a WIndows DLL).

If you wanted, this DLL could simultaneously support multiple buffers; this
might add a parameter to the CALL, but would not otherwise present any
significant technical challenge.

Data limit? 2GB  (more if you can accept the use of Windows objects
requiring hard-disk space).(And have such disk space available!).

If you (or anyone else) are are interested in obtaining such a DLL, drop me
a note. I could create one to this spec for a pretty modest price (agreeing
on the specs will probably take longer than coding and testing).

(Sorry, no: I could not create this DLL with a Component Object Model
interface).

In general Cobol can do anything that C can do.  It is just that C does
almost everything by calling library routines, so it hard to tell the
difference between what is 'part of C' and what is add on libraries.  In
Cobol all that is required is to call those same library routines.  Of
course it is not always easy to work out what is actually going on and
emulate it exactly.

> I did not try the EXTERNAL command (although it sounds like it does

EXTERNAL is like C's external in allowing several programs within one
run unit (one set of programs that call each other) to mutually share
blocks of memory (or files).  The easiest way to do this is to include a
COPY book in each program where this contains:

         01  Shared-Block           EXTERNAL.
             03  some-data          PIC ...
             ....

Each program can then use this one block of global data, even
dynamically loaded programs.

If the program is Windows API or is multi-threading then it will need a
protocol to read or write to the block (or blocks) as several parts of
the run-unit may be active at any time.

EXTERNAL files are much the same, they can be opened in one program and
used in others.  This may mean that if one program READs a record from a
file and then CALLs another program the record area (of which there is
only one in the run-unit) may now have a quite different record in it
(or none).
