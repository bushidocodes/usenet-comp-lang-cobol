A few months back I posted here asking if anyone had some simple code to
batch conpile Fujitsu Projects.

The Fujitsu docs have a sample (EXTREMELY primitive) DOS batch file which
can do it for PowerCOBOL and HeyBub posted a better example, based on the
Fujitsu one. (They don't provide anything I could find for NetCOBOL)

Recently, Jimmy Gavan raised the point about building releases from projects
for Micro Focus, and observed, very correctly, that organizing this  is part
of the wider system design, or architecture.

In my "spare time" over the past few months  I have been looking at this for
some Fujitsu clients of PRIMA, but I think the same applies to any vendor
where the IDE uses "Projects".

Here's a brief summary of the situation:

1. People developing software for delivery to PC platforms often choose to
build related functions into .DLLs.

2. A "project" will deal with source (COBOL in the cases we are talking
about, but it doesn't have to be...), and it will produce executables in the
form of .DLL or .EXE files. For a simple application (like "Bottles of
Beer") a single project is enough, but consider an application that deals
with say, hospitality. The system has to deal with bookings and
reservations, accounting, restaurant(s), front office functions, back office
functions, tours, groups, ski hire,  phone calls, movie hire, and so on.
Obviously, it would be foolish to build all of this under one project.
Instead, the core is placed into a single project in a "base" directory, and
the other functions (sub-systems) go into subdirectories as separate
projects.

Typically, a system like this would have around 70 projects which produce
around 300 .DLLs and .EXEs.

Some of the projects will involve GUI applications, and for Fujitsu
customers this often means PowerCOBOL. Other projects deal with batch
processing or providing common functions and these are built in Net COBOL.
So, in order to deliver a new release of the entire system, a number of
PowerCOBOL and NetCOBOL projects need to be built. The IDE enables projects
to be built with a single mouse click, but it is pretty tedious if you have
70 projects, to have to open up each one and click on the "build" icon (then
check the results).

3. Then there are the COPY and INCLUDE Libraries. Each project can indicate
where the COPYbooks it uses are located. It can be done by a separate
configuration file that is included in the project, or there are options
within the IDE to specify paths as compiler options, or you can use an
Environment Variable, and any of these is much better than hard-coding COPY
library paths in the source code. Often people will place the COPY libraries
onto a single server and the workstations map to it. The point is, that
unless the projects can find the libraries, there will be no executables
produced.

4. A couple of my clients were interested in something to deal with this
release build problem. One of them found that to build a new release of
their package needed hours to run and if it was left overnight, it could
fail, and even if it didn't, it was a nightmare to go through all the
listings and see what failed and what succeeded, bearing in mind these
listings could be distributed across a directory hierarchy.

As a result, I started working on a "Projects Manager" tool.

I finished it yesterday :-)

(At the moment, it deals only with Fujitsu Projects (PowerCOBOL and
NetCOBOL), although I'd consider doing a Micro Focus version if people were
interested.)

During the last few weeks I have had some people who have been kind enough
to test what I had, and their feedback has been incorporated into the
finished product. As this is a programming forum, I'll comment on the
programming...

My early efforts were pretty stumbling and there was no multithreading, so
while the batch processes ran, the GUI was frozen and looked like it had
hung :-). Mouse clicks were ignored and so on...

I was surprised to find that win XP does not (of itself)
"remember" the last size and position of an application window, so this had
to be coded into the tool, and the whole development was an adjunct to my
self-education in  more advanced C#. The tool accesses the registry to
detect the presence and version of COBOL in use, it kicks off and monitors
each compile as a task WITHOUT using DOS shell, and knows immediately if a
compile succeeded or failed (no waiting until everything is finished).
Progressive information is posted to the Screen in real time. It also
gathers, collates, and wraps the reporting from disparate directories  where
projects are located and provides a single consolidated summary, or separate
reports for each project, all provided in the base directory.

It has confirmed my previous opinion that .Net simply rocks. (C# is really
just a vehicle for the underlying power of the Framework) and, as my
confidence in it grows, C# is the most enjoyable language to write that I
have ever used. You can do really cool things really quickly :-)

I loved using COBOL, and it has a simplicity and elegance that makes it
attractive and easy to write, but it just doesn't have the succinctness or
(more importantly) the sheer power that I get with C#. Tasks that would take
a lot of time in COBOL (first to find out how to do it and what API calls
were necessary, then to get the interface to the API working correctly) are
provided by Intellisense in Visual Studio, with method signatures
documented, exactly when and where you need them.

When I found out the problem with window resizing not being remembered, I
had no idea where to start. Straight to GOOGLE and in 10 minutes I had cut,
pasted, and modified to my exact requirements, sample C# code. I couldn't do
that in COBOL. The sample base just isn't there, if you want to do anything
a little less ordinary. (This is why I used to get really annoyed with
programmers who said: "Anything I want to do, I can do in COBOL".... it
shows such limited imagination :-).)

The final release of the tool incorporates multithreading  (accomplished by
asynchronous delegates in C# (although there are other ways to do it as
well) - very interesting and kept me fully occupied for several days,
researchng and testing until I got the hang of it :-). Unlike a cut and
paste of some sample code, I wanted the full background and conceptual
understanding on this, so it meant hitting the books...), and multi tasking.

The tool can run multiple instances of itself so large package builds can be
multi-processed with several subsytems being built concurrently. It can
recurse directories to find the subsystem projects, and it can operate
across a LAN with projects and libraries on different drives.

I received the following email from a lead programmer:

"Hi Pete,

Congratulations - The tool is fantastic!
As <name suppressed> said we ran the batch compile on the slowest PC & it
actually took only about one hour to process the 75 projects selected (have
both Powercobol & Netcobol). It successfully compiled about 296 dll/exe, it
would take me at least half a day to do the same job manually. "

I was really pleased by this, as this girl is one of the most capable IT
people I have ever seen (and one of the best COBOL Programmers), and half a
day of her time would be a day and a half of most people's... :-)

I have other emails (all unsolicited), all very impressed and saying it
would make life much easier (which was the main reason I wrote it), but
enough self indulgence... :-)

The tool has too many features to enumerate here but if you'd like to have a
look at it or find out more, here's a link:

http://primacomputing.co.nz/PRIMAWebSite/BatchCompile.aspx

(If you run the embedded video on the page linked to above, you will get to 
spend 15 minutes with me, in my world. (Remember, surreal experiences are 
good for personal growth... :-))

Getting back to the point...

Deciding how to organize your projects and deciding what will BE in a
project, is a major consideration, and all-to-often, overlooked.

Too many projects and you have a very tedious task to build a release; too
few, and you have too many eggs in each basket...

It isn't the fault of the IDE. It doesn't matter whether you use Fujitsu's
IDEs for PowerCOBOL and NetCOBOL, Eclipse, Micro Focus, Visual Studio, or
anything else, the fact is that careful thought should be given to the
overall packaging and architecture of your application right from day one.

If you let projects just "grow like Topsy" it isn't long before you can't
find stuff , or there are COPY conflicts or  you miss the fact that one
build out of the many didn't succeed, you deploy your package and have a
very disgruntled customer with a problem that is not easy to run down.

Using "Projects" is a step up from building everything at a command line,
but the projects themselves are also components in your overall architecture
and deserve the same consderation you would give components at the 
programming
level.

Pete.

That's a very good question, Clark and I thought about this at length when I 
first got my head around VS and C#.

The new product from Micro Focus DOES have these advantages for COBOL (as 
does Fujitsu NetCOBOL for .Net) and you can run both of these COBOLs in 
either Visual Studio or Eclipse.

At first, I kind of wished that the same environment I was enjoying in C# 
was available in COBOL. (Now, it is, at least for people who fork out for a 
CLR generating compiler.) But, as time goes by, here's what happens...(all 
comments based on personal experience; YMMV...)

1.You start to realise that even if you DID have the same facilities in 
COBOL there is still a lot of writing required...
2. There are facilities in the other languages that simply AREN'T in COBOL. 
Even with a version of COBOL which opens up the whole .Net framework, there 
are still things that COBL simply cannot do. It wasn't designed for an OO 
environment. You can push it into one, using the amazing OO COBOL features 
provided by both Micro Focus and Fujitsu, but it is still Jake the Peg (with 
his extra leg... http://www.youtube.com/watch?v=KJleJbn9G6Y ).
3. As time goes by and you find yourself becoming more facile with the new 
language you realise there is SO much that COBOL doesn't even look at.  Some 
Examples?

1. For me, I was getting tired of embedded SQL which is tiresome and 
primitive in today's world. C# opens up LINQ (Language integrated queries) 
which is a whole new ball game for data access. (Note: this is not a .Net 
Framework facility; it is a LANGUAGE facility and represents a completely 
new way  (largely driven by Anders Haljsberg) of looking at how we query 
information.) Currently, it supports (and generates) SQL under the covers, 
but it uses the concepts of functional programming to only hit the disk at 
the last minute (JIT) and you can manipulate queries without ever actually 
doing them... :-) With the arrival of new data storage devices and 
subsystems, LINQ is poised  to lever the advantages of new facilities in 
hardware without having to keep everything tied back to SQL for 
compatibility.

2. Taking the current example of the Projects management tool. If I had 
written it in PowerCOBOL, I would have been stuck with a frozen GUI as the 
background batch processes executed. (Or I would have been off into another 
excursion into the Windows API for thread management) C# offers delegation 
seamlessly, and in a single line of code I can create delegate methods that 
can be used for all kinds of purposes (not just multithreading). I like to 
write code that is generalised so I have a fighting chance of being able to 
reuse it. With a delegate method I can set up a generic signature or 
interface and  run multiple objects through the same "controller". I can't 
do that in COBOL.

3. I needed access to the registry (to check what levels of COBOL are 
installed locally. 3 lines of C#. In COBOL, an API call and all the fiddling 
with getting the right definitions in the interface . (Certainly, CLR 
generating versions of COBOL can use the .Net Framework to do this just as 
easily as C#, but I don't have, want, or need an $11,000 compiler when C# is 
free.)

Over the last couple of years I have been moved to the inexorable conclusion 
that there are free tools out there which are streets ahead of COBOL for the 
Networked environment I am developing for and using myself. The support is 
better (and free), and I don't have an arrogant vendor who treats me as a 
"locked in" prisoner who will take what he's given and be thankful for it.

As a small Company, PRIMA has to consider what it spends on tools. Even when 
I WAS prepared (and had the funds allocated) to buy into NetCOBOL for .Net, 
it was not a decision that caused me joy. Thankfully, events transpired to 
change my mind and the money I saved was far better spent elsewhere. (I also 
came to the realisation that such compilers are actually pointless, but that 
is another story...)

I can't afford to invest in stuff unless I know it will pay for itself. (I 
replace my notebook every 3 years or so, but I check that the one I am using 
has paid for itself, before doing so... :-) )  A CLR generating compiler 
could never compete with a free language like C# or VB.Net, on price, never 
mind functionality.

All that is required for membership in this club is the desire to join it, 
and the determination to allocate some time for re-education. (Everything I 
know about the environment I am using came from books, videos, analysing 
sample code, asking questions in forums, and simply trying stuff out.  I 
have had no formal training in C#, but I manage to produce applications 
OK...(To be fair, I DID have a basic background in OOP, gained using OO 
COBOL to write Component Classes over a number of years.) Fortunately, it is 
an ongoing process; you don't have to "learn C#" overnight.  I started with 
some basic free videos on VS and progressed from there (at my own pace). 
Within a few days I was writing simple applications and actually chuckling 
at how much fun it was :-)

Never mind. I have reached a time now where I see no need to evangelize and 
whatever people use is fine by me. :-)

I have documented my experiences here and elsewhere just so people can know 
they have a choice; I really odn't care whether they exercise it or not. (I 
am glad to help anyone who does, though :-))
