```
As a follow-up to another thread and NOT expecting consensus of opinion, I 
thought I would ask this question:

When is it a "good" thing to test only the first character for "0" versus 
testing for specific file status codes?

(This IGNORES the entire question of any "9x" "successful" status codes like 
IBM's "97")

   ****

The question that I would ask is when writing a program isn't it 
"program-specific" whether you would want to handle NEW successful status codes 
the same as old ones?

In other words, suppose you have two approaches for "successful" status code 
checking today.

Approach 1:
   If FS-1 = "0"   *> checks first byte (character) for "0"
        versus
Approach 2:
    If FS-both = "00" or
       FS-both = "02" or
       FS-both = "04" or
       FS-both = "05" or
       FS-both = "07"

Now either of these could be replaced by 88-level tests (and I am not asking 
about that)

The question is (then) what happens if the implementation (as an extension, new 
implementor defined "0x" or the Standard) introduces a new "0?" successful 
status code.

For example,  (and this is QUITE contrived and I don't think it will/would ever 
happen),

If the Standard introduced a "09" status code to indicate that an OPEN was 
successful but the file was on a non-mass storage device.   (This might be 
useful information for some applications).

If seems to me that an application programmer more often than not (but NOT 
universally) should check for the specific status codes that are "known" at the 
time the application is created.  In that way, any NEW "successful" status code 
will "fall thru" (probably to error handling).  It may be that the application 
will NOT need changing to handle the newly available information, but it is ALSO 
possible that the newly detected situation would require "new program logic".

Therefore, if I were to make a suggestion for new program development, I would 
probably suggest checking BOTH bytes (but NOT just for "00") for successful I/O 
status - and to make certain that "currently unknown values returned" cause 
error process to be activated.  This certainly would not be necessary for all 
applications but MIGHT be useful for some/many.
```

```
One issue which hit me is the difference in how we handle multiple-use
programs.     We have a program which a vendor wrote for a package,
which is used by lots and lots of programs.   It doesn't check for
'97', but instead returns its own 'bad status' to its calling program
when it doesn't get a '0x' status.

So periodically we have aborts.    Trouble is, we have old standards
about changing programs, where every single job that touches a change
has to be thoroughly tested and approved by users.    This type of
standard obviously wasn't even considered for the OO part of the shop,
but our standards are older.    It would only take a couple of minutes
to correct that program, but the cost of getting user approval for
every job that called that program was prohibitive, so we have lived
with the problem.   (Over the years, the number of programs using it
has decreased as we stopped using VSAM files).

Moving applications from the mainframe to Java servers has had some
interesting impacts on standards.    On one hand, there's a lot of
confusion where standards don't yet exist in an organization, but on
the other hand, we can accomplish stuff that old standards wouldn't
allow us.    Hopefully, we will have new standards designed around the
new information system that will work.     The only way for the old
standards to go away is to replace the system they are tied to.
```

```
I have been required to devise and implement standards in three 
organisations during the course of my career.

It was always done by consultation and discussion and on the clear 
understanding that we were implementing standards in order to facilitate 
working; not to limit, restrict, make life difficult, or stifle innovation 
or imagination by individuals. And there was ALWAYS provision to override 
the standard, provided the case for doing was properly documented and 
approved by the lead programmer. Standards were also jointly reviewed and, 
if necessary, revised, every 4 years. (That was a standard... :-))

The thing about standards is that they are the tail, not the dog. You use 
them as a tool to increase productivity and encourage 'best practice', not 
so that some little Hitler of a programming manager can enforce his 
stylistic will on the poor buggers who do the actual work, or so that some 
born again evangelistic lead programmer can bring everybody to salvation by 
enforcing what he read in Computer Weekly on the train coming in to work...

Any standards that seek to enforce are questionable. The idea is to 
encourage and to facilitate, not to threaten and coerce. People will 
willingly adhere to standards that make sense; they won't if those standards 
are enforced on them or rammed down their throats as the 'only way' to 
programming Heaven.
```
