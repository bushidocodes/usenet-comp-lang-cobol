You may be looking for a problem in the wrong place.

Could you compile the COBOL program as a Class and let each thread 
instantiate its own instance of it?

That way the threading is not a problem.

Instead of CALLing it, the thread creates an object instance and INVOKEs the 
Main method.

This might mean some wrapping in the COBOL program (The main logic becomes 
an object Method, for example).

If the problem is with ORACLE connections, a separate instance should 
establish a new connection with an isolated view, so that threads can run 
concurrently without problem, as you required.

Regardless of whether your COBOL routines create threads, they must be
compiled for multithreading if they will be called in a multithreaded
program. That means compiling with the -t option, and setting the
appropriate compiler directives - probably including one of the
REENTRANT directives. I can't tell you what to use specifically
without knowing quite a lot more about your application; I suggest you
refer to the documentation.

Are you following all the instructions in the documentation for
calling COBOL from a non-COBOL program? See the chapter "Interfacing
with C Programs". In particular, are you calling cobinit(), cobtidy(),
and cobthreadtidy() in the appropriate places? Are you using cobcall()
to invoke your COBOL routine?

This type of problem can also arise if the C++ program wasn't
correctly compiled and linked for multithreading.
