Yeah, I know, if we all wrote perfect code we wouldn't need it...

I just spent two days (not entirely full on, other things as well) trying to 
account for and change some "odd" behaviour in a program running remotely 
thousands of miles away...

It made me think about how much we take for granted these days.

Even the lowliest COBOL system has SOME kind of debugger included in it and 
the better ones have some excellent debugging tools.

But that wasn't always the case.

I remember in the early days you made displays of where the code was and the 
data in certain fields and that was how you debugged. Then we got "READY 
TRACE" which was good for tracking flow but didn't say much about the data, 
then we learned to read core dumps and so on...

Today, it is really pretty easy. Tools like Animator (for both Micro Focus 
and Fujitsu) make tracing through code pretty simple and we take it as read 
to be able to reset the position, inspect fields in scope,set break and 
watch points, and run to the cursor etc. Even the mainframe has some proper 
tools for debugging now...

But I'm making this post because I realised that whether we have tools or 
not we can ALWAYS debug, so there is no need for despair when traces are not 
available.

Here's the story... some of you will be amused, some of you will think I'm 
an idiot; both points of view are valid... :-):

1. I write and use a number of components and web services using C#.

The Integrated Development Environment (Microsft Visual Studio 2008/2010) is 
superlative and it has debugging services that are simply a dream.

2. The end point is deployment to the PRIMA server which is in a server farm 
in the USA. (When you access COBOL 21 or PRIMA sites they are being served 
to you from that server). Obviously, security, both physical and logical, is 
a major concern of the Service Provider.

3. As I can't usually remotely debug, I develop using IIS on a local machine 
(local server) and when I'm satisfied everything works locally, I then 
transfer the whole lot to the remote server. This means I have an exact copy 
of SQL Server tables on the local database that are the tables used in 
production on the remote server. I can also access SQL Server on the remote 
system exactly as if it was local so I can compare local and remote tables 
in the same SQL Server Management System console.

4. So I develop a certain web service, check it out locally, am satisfied it 
works, deploy it to the remote system, test it, and it fails.

5. I run dozens of tests trying different methods in it and find it mostly 
works but occasionally fails in certain methods under certain conditions. 
The failure involves an invalid cast from a field that is "DBnull" so I know 
it is table related...

6. I cannot debug remotely (although I understand it can be arranged, Visual 
Studio was denied access permissions even though it is my server, because of 
the shared nature of the service) but I really shouldn't have to. I know 
conditions under which it fails so I should be able to duplicate them 
locally, fix it with the tools I have and redeploy. Except that when I do 
this, it works perfectly locally and I am unable to duplicate the error.... 
Ever been there?. I spent considerable time comparing the tables locally and 
remotely and found no differences.

7. The web service is around 4000 lines of C# and has about 5 methods in it. 
Three methods are used by the path that fails so I know the problem has to 
be in one of those. It only takes a moment to deploy the web service so it 
is no problem to rebuild and deploy it. I start thinking about writing a log 
from it (obviously, as it is running on a server, displays or user 
interfacing are not an option...), then I remember that it gets passed an 
interface block that has a message area in it so I start inserting a few 
lines of code to place a message in this buffer and return the block. In 
effect, I let it connect to the database, do some SQL, then stop it.  If it 
hasn't failed, I know the problem is further on in the sequence. Gradually, 
I narrowed it down to an area of around 20 lines of code... (But it was not 
a debugging method I would recommend to anyone...)

I don't know how many tests I ran but it was dozens. I inspected the 20 
suspect lines very carefully and compared to the local copy, which works. 
Identical. It HAS to be the data. But the data is identical too. It seems a 
fundamental principle of computer programming that if an identical program 
runs against identical data the results must be the same. UNLESS there is a 
configuration difference... I spent more time checking that out and although 
there were differences between the server and my local IIS, none of them 
were significant.

By now I was getting pretty pissed off with the whole thing and would have 
given my eye teeth for a chance to trace the remote service with a proper 
tool.

As that wasn't an option, I persevered with the trial and error (I'd like to 
call it the "empirical" aproach because that sounds less primitive and 
stupid...) Eventually I got it to three lines of code and that pointed me at 
the access it was doing. I found that one obscure column in an obscure 
table, in a template row (I had only checked data rows), was set to null 
when it needed to be zero. Remotely changed the table column to be zero, 
re-ran, everything worked.

So what would have been about 20 minutes work, had modern tools been 
available, turned into 2 days of swearing and muttering... and being at my 
wit's end. (It's times like that you start thinking "maybe I'm getting too 
old for this shit..." but then it works and the labour pains are forgotten 
in the joy of seeing it run :-))

The bottom line is: Don't ever give up. Keep at it and eventually you'll 
crack it.

But getting some decent tools is not a bad idea...

Pete.
-- 
"I used to write COBOL...now I can do anything."

Both Java and C# are powerful and useful OO languages. (I still think COBOL 
is the best of all the non-OO (procedural) languages.)

I've never really thought about this until you asked, Charlie, but here are 
some things which were contributing factors:

1. At the time the Fujitsu (Alchemy) CIL generating compiler (NetCOBOL for 
.Net) was released I already knew Java, having learned it to assist me 
understanding OO COBOL. I had also been writing OO COBOL for some years and 
was very keen to move this experience into .Net. So, I was pretty excited 
about the new compiler offering. Unfortunately when I tried to buy it I was 
treated with rudeness and arrogance. It was like: "We are the only game in 
town so you can have this if we are pleased to offer it to you..." although, 
not in those words of course. As I didn't see that paying several thousand 
dollars and begging to be allowed to purchase their software was a viable 
option, I looked for alternatives. I could simply move to Java but my client 
base is Microsoft and it would be a bit clumsy (though quite feasible) 
writing Java applications for .Net.

Then I realised that C# had recently become available and after watching the 
channel 9 video with Anders Hejlsberg discussing how he came to invent the 
language (sadly, this doesn't seem to be available any more - wish I had 
downloaded it) and the influences he based it on, it looked like it might be 
worth investigating. By this time I was totally persuaded about component 
based development, and C# lends itself to that better than anything I know.

2. C# was free and easily available. (zero risk... if it turned out to be 
unsuitable all that was lost was time.)

3. C# is less verbose than Java (not that Java is "verbose" but you do less 
writing in C#)

4. I was committed to the .Net platform and C# was developed for that 
platform. (Although it runs on just about anything, like Java)

5. Support and help while learning it are available for free with excellent 
videos and a user community of millions. Sample code, background 
explanations, video walkthroughs, and even "unofficial" support from 
Microsoft employees all instantly available from the Internet. (I had some 
problems understanding and using Interop Services when I first started. I 
needed this to be able to reuse my existing COBOL components with the new 
language. I posted to a forum and a couple of Microsoft guys were very 
interested to see COBOL as a candidate for Interop and very quickly showed 
me where my problem was (It was in the COBOL compile options not in Interop 
Services...) These days my legacy components in COBOL play nicely with 
components in C# and I don't need a CIL generating compiler to achieve this. 
(Sometimes a bad experience can have a good outcome...)

These days I really enjoy writing C# and it meets my needs perfectly. 
Learning it has forced me to widen my horizons and I now understand things I 
never did while using COBOL.

So, for me at least, it is a good result.

blocks do what they are supposed to)  things just work a whole lot better.

if there is a problem and regression testing is something I no longer do. If
