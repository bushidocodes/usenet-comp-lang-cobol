Marty, don't assume that everyone sending mail from cs.nyu.edu is a student
there. It is unlikely that I will graduate from NYU ever, since I got my
PhD in 1968 and am now past chairman of the computer science department
and associate director of the institute. I also know a fair amount about
COBOL and business applications, having written the entire backend and
runtime of what is now the Computer Associates Realia compiler. I have 
also been involved with large scale COBOL development (by the way the
Realia compiler is itself written in COBOL).

So, you say to yourself, how can someone with some small measure of 
experience still say nice things about the dreaded GO [TO] verb?

Of course we all know that programs can be constructed without using
GOTO and that  a very small set of primitives is sufficient. FOr example
recursion along with conditionals suffices, and as some interesting work
in using Martin Lof logic as a programming language shows, even the
restriction to primitive recursion together with higher order functions
is in practice sufficient for most puroses.

Of course since COBOL lacks both recursion and higher order functions, more
traditional low level primitives like loops are used, but of course you
don't *need* the GO in COBOL.

The use of GOTO is quite limited in well written COBOL. I realy know of
two uses, one for exceptions (the lack of an exception mechanism, such
as that defined in Ada and C++ is VERY unfortunate), and one for finite
state machines.

To take the first. Suppose you have a deeply nested functional 
decomposition (I do hope that in the enthusiasm for eliminating gotos,
people have not forgotten that the *key* to structured programming rests
in proper functional decomposition, and it is quite appropriate to use
the program unit structure of COBOL to achieve such a decomposition),  you
find an exception that requires termination of the run.

In Ada you would just raise the exception Terminate_Program, and it would
be caught at the outer level in a clean manner.

In COBOL you have two choices, either you return a flag which is tested at
each level of the decomposition, and eventually results in skipping
the remaining processing. That can get awfully messy.

The other appropriate approach in COBOL is to execute a STOP RUN statement.
But once you have a STOP RUN, it is not only acceptable, but preferable
to transfer control to the para containing STOP RUN using GO TO rather than
PERFORM.

The other appropriate place for using gotos (even in a language with
exceptions like Ada) is in mapping a finite state machine with arbitrary
transition diagram. Mapping the states to the program counter is a neat
and efficient method of encoding such a machine.

If you disagree with the latter, please tell me how, given an arbitrary
finite state machine, you would encode it into an efficient easily
modified program.

If you graduated from NYU learning that gotos were always bad, you were
unfortunately given a false impression (hopefully you did not graduate
from the CS department :-) Usually such impressions get modified to a more
reasonable position with age.

I have had the misfortune to read hundreds of thousands of lines of 
extremely badly written, badly structured COBOL code. A large percentage
of this code was written completely free of GO TO's. I am not saying
that using GO TO's would have improved the situation, but it is tiresome
to run into COBOL programmers who think that structured programming is
all about not using GOTO statements, and that if they succeed in that, they
have, ipso facto, structured their program.

Robert Dewar

OK Robert, here goes..

+++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
> [1] .. deeply nested ..
> .. you find an exception that requires termination of the run.

The innermost (deeply nested) logic cannot decide to terminate 
the run, since that is the prerogative of the outermost logic.

Hence the 'error' *must* be passed outwards until some level
decides to retry or abandon or otherwise dealt with the exception.
That is the correct logic, and it doesn't have to be messy.

If it is not dealt with by the time it reaches the outermost
level then the program would terminate (or, better, ask the 
runtime system which might ask the job control and so on).

Examples:
(a) nuclear power: divide by zero while updating a display
    might choose to set a minor alarm with a double beep.
(b) banking: divide by zero when calculating interest on a
    dormant deposit account could be flagged for later action
    so as not to hold up daily processing.

+++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
> [2] .. given an arbitrary finite state machine, [say how] you
> would encode it into an efficient easily modified program.

Number the possible machine operations,
  eg op 1 = move to the next state
     op 2 = display current state
     op 3 = open valve (or whatever)
Write procedures for each operation.

Use a state counter PIC 99 (the size is easily modified).
Set up a description table X(20) OCCURS 99 for the states.
Read in the text descriptions for the states.
Set up a transition table PIC 99 OCCURS 99.
Read in the state transitions
  eg state 12 => state 15 means MOVE 15 TO table (12).
Set up an indexed file/table of operation sequences by state
  eg given state 12, do: op 2, op 3, op 1, end.

Read current state.
Set state counter.
Loop until end-state
   Set starting position in the indexed file/table
   Loop until end or change of state
     Pick up next op code
     Evaluate op code
     Case 1: PERFORM OP1 [move transition (state) to state]
     Case 2: PERFORM OP2 [display description (state)]
     etc..
     End evaluate
   End loop
End loop.
STOP RUN.  [if this is allowed :-)]
+++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++

Robert Dewar (dewar@cs.nyu.edu) wrote:
: The other appropriate place for using gotos (even in a language with
: exceptions like Ada) is in mapping a finite state machine with arbitrary
: transition diagram. Mapping the states to the program counter is a neat
: and efficient method of encoding such a machine.
But not the only way, nor the best way.
: If you disagree with the latter, please tell me how, given an arbitrary
: finite state machine, you would encode it into an efficient easily
: modified program.

Easy, actually (though I take no credit - this technique is not mine).
You encode the FSM as dual tables, one with state transitions, and one
with actions per transition.  In our system you can attach lists of
actions to each transition.  Then, you drive though the tables using
a generic, stable 'motor'.  Your own program consists of the FSM itself, 
which is easily encoded as a nice diagram, plus the action modules and
data.  Maintenance means adding, removing, or changing the action
modules.  Net result: you can build *enormously* complex programs that
are readable (ie. you read the FSM) and maintainable.  This technique
works well for batch programs, and very well for interactive programs,
since all user interaction is handled as events in the FSM.  Robust
and flexible.

My company sells a Cobol toolkit that uses such techniques seriously.
We regularly take programs that are 5-15 years old, and maintain them 
with none of the usual pain associated with old code.

Since at least 1985, I began measuring 'efficiency' in terms of my time,
not that of the CPU.  I really do not care if an implementation of any
solution takes 100 times longer to run if it saves me 50% of my work.
This is especially true since 99% of my work is highly portable and has
a lifespan of a decade or more.  So, any 'loss' of efficiency by not
programming the state machine into the hardware microcode is more than
offset (IMHO) by the gain to me in productivity.

Furthermore, my FSM programs basically drive a Cobol program which is
almost always i/o limited.  Each step through the FSM executes a block
of serious code.  If I reduced the FSM overhead to zero, my program
would speed by by perhaps 0.001% to 1%.

Of course, if I'm writing something like a serial port driver, I'll do
the whole thing in assembler.  Conclusion: don't use terms like
'efficiency' in the wrong context.
