```
Hi, 

I'm wondering about the performance I am to expect from Fileshare in
Netexpress. For our application we've always been using the normal
file handling routines, and when in a network we've just mapped the
server direcotry with our data files to the clients as a network
drive. This way we leave it to the clients to do record locking and
caching which really should be a problem, and it is when we're using
NT as a fileserver rather than Netware. We suffer, in an NT
environment, from poor performance, corrupt indexfiles and duplicate
key problems. It's as if "cache coherency" is something NT doesn't
really care that much about. (We turned off opportunistic locking,
which led to a substantial increase in performance on random access
operations, but it didn't really decrease the number of corruption
problems.)

So, I thought, perhaps using Fileshare would solve our problems. Then
the server could take care of the file handling itself which should
mean -- as the manual says -- increased performance and security.

So, I set up a Fileshare server on one of our NT-stations, and
compiled some programs with Fileshare to test the thing. It works
OK. It's just that it is completely unusably slow. I am aware that
sequential reads is the type of action that should benefit the least,
but the fact that a certain program takes less than five seconds to
execute when not using Fileshare (ie when the data files are accessed
through a mapped network drive) and more than 90 seconds with
Fileshare bothers me. 

So, is this the type of performance to be expected from Fileshare, or
am I doing something wrong? The configuration options seems pretty
limited.


Kristofer Andersson
```

```
We use Fileshare on a Point of Sale system which is Unix based.  We found
that Fileshare can really 'eat' the processor on the Fileshare Server.  We
use a dedicated server for Fileshare and another application server for the
thin client based applications.  Since Merant Micro Focus is supported on
Linux and most flavors of Unix this might be a good solution for the File
Server without affecting the client environment.  We get very respectable
i-o speeds on applications based on random access reads & writes. Sequential
reading is slower than Micro Focus file i-o on a single Unix box but faster
than NFS.  Incendenly back when we were running DOS apps the Netware
Fileshare server was quite a bit faster than NT.  We have found that the
most important condition for network performance is using a dedicated file
server box.

Joshua Seltzer
```
