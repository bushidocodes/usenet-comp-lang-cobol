The way that I handle names/name lookup is to always have a separate index
for the purpose, and then include that in the data entry. There are really
no algorithms to handle it in a completely standard way, due to differences
in human language conventions.

You can easily devise a method to create that index that will handle 95% of
all names, but stuff like "ABC, a Division of 1234 Ontario" is always going
to require hand tweaking to get it to fit in where you want it.

You are making an assumption on the file layout that first name and second
name are fixed length fields.  That is not usually true.  A description
like:

01 CUSTOMER-RECORD.
        05 name                              PIC X( 40).
        05 PHONE-NUMBER        PIC X(10).

is much more common.  There is also the problem of time.  Very few telephone
books have only two names.  Estimate how long it would take to look up a
number in the average telephone book if you use the method "start at a, and
read every name until you come to it".  I do not think I would give you much
of a mark if that is the best you can come up with.

Reding all the different views upon this matter I remember a system I made
many years ago. We had discussions with elements similar to many of the
elements in this thread. The solution was a standard for entering names and
a standard for displaying of names.

We didn't split the name into different fields but whenever we stored a
name, it was in the format: <Last-name', 'other-names>.  When we displayed a
name, we turned the format around (the comma and space told us where to find
the name(s) to put in front)

In that standard, your name-examples would be our 'display format' and the
names we stored, would look like this:

Howard, Howard E.
O'Haris, Frank
Peters, John            **** Contains comma - stored asis (displayed correct
ever after)
Smith,  John Jacob Jinglheimer
Junior, JJ
III. M. James Smith
Gonzales, Margarita Maria Lopez y

Except for 'M. James Smith III'  the store format is fine for searching and
handling that exception is up to the designer. If the users know the
difference between the second and third Smith you can leave it, but if it is
normal to search for Smith, you must add logic to handle roman numeral, JR
and all that stuff that primarily is a part of US naming conventions.

Another thing we did was to convert the stored name to upper case and put
the result to another name-field, and the name index was built upon that
field. Every search field entered was converted to upper case before the
search.

Perhaps the ideas can be useful.

Regards
Ib
