file and then moves it to a DETAIL-LINE to later be printed. Is there

state? (Such as "State NA").

blank, not an alternate value.

The only way that I can think of is to test the value yourself in the
PROCEDURE DIVISION, something like:

IF STATE = SPACE THEN
MOVE "NA" TO PRINT-STATE
ELSE
MOVE STATE TO PRINT-STATE
END-IF

In this case, I assume that PRINT-STATE is defined as PIC XX. If it's
larger, you could put more descriptive information in the field.
Hope this helps some.

If you set up your record definition with an 88-level something like
this:

15 ADDR-STATE PIC X(13).
88 ADDR-STATE-NOT-FOUND VALUE " *unknown* ".

then in the PROCEDURE DIVISION you can say something like this:

IF IN-STATE = SPACES
THEN SET ADDR-STATE-NOT-FOUND TO TRUE
ELSE MOVE IN-STATE TO ADDR-STATE
END-IF

The advantage of this over a pure PROCEDURE DIVISION solution is that
every program that copies the record definition gets the same value. To
change the value, you change the copybook, instead of having to change
each program individually.

I was following this thread, and I don't think Chris Westbury wrote any of the above, but he did
respond to the original question. I think Bill is subsequently asking if REDEFINES can designate an
alternate value in some cases.

The answer is that REDEFINES provides an alternate layout of the organization of variables
in some part of working storage, or in an Input/Output buffer; but REDEFINES cannot provide
alternate values, since a byte in memory can hold only a single value. In fact if data A
REDEFINES data B, then B data definitions can use VALUE clauses but not A's.

To change a memory location during program execution, you must normally use an instruction in
the PROCEDURE DIVISION -- like MOVE. Even BLANK WHEN ZERO is just designating an additional edit
on the field after each move...

Another point is that if DETAIL-LINE is a record definition and not in WORKING-STORAGE, then
DETAIL-LINE is really just a map superimposed on an output buffer. And that output buffer is
probably different after each WRITE statement. Which is why VALUE clauses aren't allowed
on record definitions.
