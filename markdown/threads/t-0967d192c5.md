Donald,

I cannot imagine an implementation that does not allow for nested COPY
statements, though the 1985 standard is silent about this.  I don't recall
whether the Cobol Compiler Validation Suite attempts nested COPY statements.

RM/COBOL allows for nesting to 5 levels, with a relaxation to 9 levels if
one uses a "tail COPY" technique wherein the last statement of a COPY file
is a COPY statement (the limit of 9 is imposed by the listing format that
has one print position for COPY nesting level).

These limits have been in place since RM was the OEM compiler for many
midrange computers.  I am unaware of anyone ever constrained by these
limits.

So, follow your nesting instinct.  :)

Tom Morrison
Liant Software Corporation

Nested copies *are* prohibited in the '85 Standard.  (Send me private email
if you need/want an explanation of this.)  HOWEVER, they are a very common
extension in many implementations today.

They will be EXPLICITLY allowed (as a substantive change) in the next
Standard.

There is a "place" where you *may* run into problems - and that is if you use
REPLACING.  I believe that the draft Standard explicitly disallows any
REPLACING phrase in nested COPIES.  Some existing extensions allow it in the
"bottom" level of nested copies.   If you have a compiler that allows it at
other than the bottom level, I think you will find NON-portable results.

The problem with REPLACING is when it gets turned on and off - and what it
impacts.  For example:

Copy level1  replacing ==level2== by ==levela==.

where level1 has

    Copy level2.

Does this or does this not change the name of what is copied? (the rules of
when replacing takes place would USUALLY say no).

Also, how about

Copy level1 replacing ==text1== by ==text2==.

Level1 has

copy Level2 replacing ==text1== by ==text3==.

Does the original REPLACING change the replacing in the 2nd COPY or not?
(arguments pro and con both exist).

finally, consider

Copy level1 replacing ==text1== by ==text2==.

Level1 has

 Copy level2.

and Level2 has

   05  Text1  Pic X.

Is the original "replacing" still in effect or not?  Probably it would be -
but the "owner" of Level1 that THINKS they are doing a COPY without replacing
would be surprised by this.

Generally, rules COULD be written for all of these, but they would be more
complex than most people would want and the committees didn't think the
benefits would outweigh the problems.

FYI,
  You *may* use the (newly enhanced) REPLACE statement in these situations to
do what you (probably) want - so the functionality is there, just not the
specific syntax.
