Hi,

We have an existing Micro Focus COBOL applications with VT100
Character UI hosted on MF Server for COBOL.

What options are available for "non-intrusive" Web enabling them or
wrapping them as Web Services?

The only option I can think of is "Screen scraping using VT100
terminal emulators".

For IBM mainframe green screen applications, programmtic communication
with 3270 screens is possible via LINK3270 bridge which intercepts the
Screen I/O commands programmtically. Is there anything similar in
Microfocus?

For "intrusive" Web enablement I can think of 3 options
1. Modify the existing VT100 apps to use MF COBOL CGI/NSAPI features
to interact with the web UI directly
2. Modify existing MF COBOL modules to wrap them as EJBs and deploy on
a J2EE server
3. Modify existing MF COBOL modules to use Web services features and
deploy them directly on MF Server for SOA.
4. Anything else?

How easy it is to make the above changes - are there any tools for
automating this?

With 3270 or 5250 protocol you've the screen and field info, so with a
little "magic" with some protocol API you can wrap a session with a web
service that expose a screen abstraction (this is NOT a theory : I work
on a host to web product and we make an exact 3270 / 5250 XML-HTML
workspace replicator).

With VT100, as far as I know, you don't have the necessary screen /
field info to build a screen scraper so, as we do in Windows and Linux
platforms with MF Cobol, you must build a map interceptor and XML
converter that expose every map inut / output in XML. In our ERP the
resulting "XML channel" in then conveyed on the 5250 / 3270 post
processor for the graphic rendering.

It's intrusive as much as an IF to detect the environment and a call to
the map interceptor, and our customers cannot tell if under the browser
ther's a Linux, Windows, AS/400 or mainframe COBOL application (or a
native host one, or a Java app using the same presentation framework).

If you're interested (or maybe only curios) I can give you our web
address.
