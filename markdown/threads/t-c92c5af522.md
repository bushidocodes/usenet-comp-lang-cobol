```
In all  examples of  code below,  we assume  that the data
being sorted is defined as follows:

01  TABLE-TO-SORT.
    02  TABLE-SIZE              PIC S9(5)  COMP.

The size  of the key and data simply reflect the data used

01  LARGE-TABLE-ITEM1.

1.                    Bubble Sort

As the  name suggests,  bubble sort moves items up a table
like bubbles  in a  tube.  The algorithm can be explained as
follows: pass  over the data, comparing and exchanging items
so that  the largest  item ends  up at the end of the table.
Repeat for  the remaining  items until  the table is sorted,
that is, no exchanges were made during the latest pass.

The following code does a bubble sort:

01  VARIOUS-INDICES.
    02  ITEM-NBR                PIC S9(5)  COMP.
    02  SWAP-NBR                PIC S9(5)  COMP.
    02  JUMP-SIZE               PIC S9(5)  COMP.
    02  UPPER-LIMIT             PIC S9(5)  COMP.
01  VARIOUS-VALUES.
    02  SWAP-ITEM               PIC X(20).
    02  SWAP-INDICATOR          PIC X(1).
       88  NO-MORE-SWAPS           VALUE IS SPACE.

BUBBLE-SORT-THE-ARRAY.
    MOVE "SWAP" TO SWAP-INDICATOR
    MOVE   1    TO JUMP-SIZE
    PERFORM BUBBLE-SORT
         UNTIL NO-MORE-SWAPS
    .

BUBBLE-SORT.
    MOVE SPACE TO SWAP-INDICATOR
    COMPUTE UPPER-LIMIT = TABLE-SIZE - JUMP-SIZE
    PERFORM COMPARE-AND-SWAP-KEYS
         VARYING ITEM-NBR FROM 1 BY 1
            UNTIL ITEM-NBR > UPPER-LIMIT
    .

COMPARE-AND-SWAP-KEYS.
    COMPUTE SWAP-NBR = ITEM-NBR + JUMP-SIZE
    IF TABLE-KEY (ITEM-NBR) > TABLE-KEY (SWAP-NBR)
          MOVE TABLE-ITEM (ITEM-NBR) TO SWAP-ITEM
          MOVE TABLE-ITEM (SWAP-NBR) TO TABLE-ITEM (ITEM-NBR)
          MOVE  SWAP-ITEM TO TABLE-ITEM (SWAP-NBR)

     .
```

```
In my opinion, the example you found has some problems.  In addition to
omitting the part of the code that loads the table, as another post
mentions, it omits the DEPENDING, ASCENDING / DESCENDING (KEY), and
INDEXED clauses, so SEARCH ALL could not be used, and all 1600 items
will participate in a search, because COBOL cannot tell how many actual
items are in the table.  In such a case, it is necessary to set unused
items to HIGH-VALUE, so that the compares work, and this increases run
time.  It mentions limits that existed in IBM's OS/VS COBOL compiler,
but have been relaxed (by a factor of 128) in newer compilers.  I don't
know too much about other vendors' compiler limits.  Finally, it does a
bubble sort, which is space efficient but time inefficient, and to crown
my complaints, it uses subscripts in place of indexes (probably not too
bad in the bubble sort case, if the compiler has a good optimizer).

Why not just insert the table items in the proper sequence?  This is
faster than waiting until the table is loaded to sort the entire set of
items (it is essentially an insertion sort, if I remember my sorting
definitions), and if the data is received in the proper sequence, it is
as fast as a load that doesn't check sequence, except for the one IF
statement.

Here's an example:
First, here is an example of how I like to define a table:
01  SOURCE-CODE-LIMIT PIC S9(4) BINARY VALUE +4096.
01  SOURCE-CODE-SIZE  PIC S9(4) BINARY VALUE ZERO.
01  SOURCE-CODE-TABLE.
 05 SOURCE-CODE-ENTRY
               OCCURS 1 TO 4096
            DEPENDING SOURCE-CODE-SIZE
            ASCENDING SOURCE-CODE
              INDEXED SC-X.
 10 SOURCE-CODE       PIC X(06).
This definition tells COBOL all it needs to do an efficient binary
search of the table, and provides the necessary variables to allow the
programmer to test for overflow while loading.

Here is generic code for adding an item to a table in the proper
sequence:
      ADD 1            TO 'object'
      IF  'object'     >  'table-limit'
          PERFORM TABLE-OVERFLOW
      END-IF
      SET 'index'      TO 'object'
      SET 'index' DOWN BY 1
      IF  'object'     >  1
          PERFORM MAKE-ROOM-IN-TABLE
              VARYING 'index' FROM 'index' BY -1
              UNTIL   'index'   <  1
                 OR   'new-entry-key' >  'entry-key(index)'
      END-IF
      MOVE 'new-entry' TO 'entry(index + 1)'
      .
MAKE-ROOM-IN-TABLE procedure reads:
  MAKE-ROOM-IN-TABLE                             SECTION.
      MOVE 'entry(index)' TO 'entry(index + 1)'
      .

The TABLE-OVERFLOW procedure would do something useful such as reading
the rest of the input file and adding to the counter, so that it could
display a message that says the table capacity is "n", and the number of
input records is "m".  Then, the programmer wouldn't have to do much
investigation to fix the problem.

Obviously, the MAKE-ROOM-IN-TABLE procedure could be eliminated in favor
of using an in line PERFORM, but I wrote this before our shop was fully
off of 1974 ANSI COBOL compilers.
```
