I mentioned in another thread that the next big challenge for me is 
providing tools that "decompose" existing COBOL programs and identify and 
isolate blocks of code that can be encapsulated. The goal is to salvage 
existing COBOL functionality by encapsulating it and wrapping it as an OO 
component.

Obviously, PERFORMED routines, CALLed modules, and coherent inline code are 
all candidates for this.

I can't even start to do this at the moment but before I do, I'd be 
interested to hear any thoughts or comments on the subject. In particular, 
if anyone feels like writing something that is a step in this direction, I'd 
be interested to see it and if it gets used by PRIMA we'll pay you for it. 
(This is not intended to be a solicitation for commercial software, it is 
just about ideas and approaches. Technically, this is "recreation", not 
"solicitation" :-))

I see a process that scans COBOL source, identifies code that could be 
"self-contained" and spins it out to a new module. The data fields 
referenced in the code must have their definitions picked up and written 
into the new module's data division.

I would want the new modules to be wrapped as COM components. That is a 
technicality that can be met further down the track. In effect each module 
would be a component with a single method (although I don't mind if they 
have more than one method as long as each method is discrete and has clear 
inputs and outputs (interface).)

The idea would be to identify the functions provided by the program at 
whatever level you can identify those functions. The criteria are:

1. Clear interface.
2. Clearly identified method(s)
3. No dependency on anything outside the module. (This is a tricky one and 
it may require some deeper thought...)

The language you write this solution in is not important  (provided it can 
run under the .NET framework 3.5) although COBOL (any dialect) or C# would 
be preferred. COBOL solutions will need to be wrapped in a process very 
similar to the one we used for Robert Wagner's COBDATA tool (see the cobol21 
web site - http://primacomputing.co.nz/cobol21 for details). As Fujitsu 
NetCobol is very good for this and is the tool employed by PRIMA for dealing 
with COBOL, Fujitsu is a good choice for your solution, but Micro Focus or 
other source can be wrapped too.

This process is step 3 of a process for migrating existing COBOL to .NET. To 
see where it fits, please see the diagram on this link:
http://primacomputing.co.nz/cobol21/ShowMe.aspx?ptitle=Process%20flow%20diagram...&pcontent=images/MigrationProcessOverview.jpg

Even if you don't feel inclined to write any code, I'd still be interested 
to hear your comments.

Pete.

Any thoughts? Ok, you asked...

This is terrific hobby idea: you are converting a one-off program to OO 
syntax ....because you can.

However, in the non-hobby environment, why bother?  If it ain't broke, don't 
fix it.

Better I think that if you want to build reuseable components, you think of 
common routines and build those from scratch, and build them right. Then 
applications developers can create applications using these building blocks.

For many years I have said that converting  or porting source code from one 
language to another on a verb-for-verb, function-for-function basis is a 
stone loser. You get zero benefit of any new technologies or features of the 
target language/environment, and have done little more than add one 
'maintenance cycle' to software which has a finite number of same... for no 
tangible benefit. This sounds like but a miror variation on that theme.

application code is standard COBOL and not an existing "OO architecture"),

horrific ways people use REDEFINES and OCCURS, the 7 different date formats 
in popular use, the fact that COBOL doesn't even HAVE a date type, the fact 
that Relational storage is different from indexed file storage, the fact 
that a number of organizations have spent millions of dollars trying to do 
it and failed, you'd really have to be an idiot to even attempt it... Yet, I 
have a growing number of enquiries about using the tools I wrote which do 
it.

It is interesting to me that the people who are not versed in OO see it as

much brighter future than code that can't do that.

It's all very well to say that the system should be rebuilt from scratch.

in) can afford that luxury. For most people in the SME category, salvaging 
as much as possible from the COBOL legacy is not just desirable, it is 
essential.

Processing standard procedural COBOL into Object Oriented components is not 
so difficult; take a look at the COBDATA tool. Robert's code runs fine as a 
COBOL batch application that reads a file and produces a listing. But once 
it becomes an OO component, suddenly it can grab stuff off the system 
clipboard, it doesn't need a full file for input, it can have a really nice 
GUI interface, with menu options, and can be easily embedded into other 
applications. The simple act of making it a component means it has access to 
a whole new world of stuff, and it can be reused over and over. People are 
finding new uses for it which neither Robert nor myself could have 
anticipated.

Regarding your "big if"... It is a very fair one. If it were really 
difficult to embed components into existng procedural code, it probably 
wouldn't be cost effective. Fortunately, it isn't difficult. I can't do it 
fully automatically (yet...I can with DB access...) but I can assist the 
manual process to the point where it becomes viable. Components are very 
flexible, and adding a dozen lines to a COBOL program so it can interact 
with OO components, written in any other language, that implement the 
Component Object Model, really is opening up a new world for COBOL.

processes that are implemented as large monolithic blocks of code. Small is 
beautiful. It costs less to maintain and it runs more efficiently in online 
environments. ("Big" is good for batch...)

Certainly, there will be cases where the cost of refactoring is not less 
than the cost of rewriting, but there are many more cases where it is much 
less to refactor and salvage. (Especially if there are tools available to 
automate some or all of it...). If you can salvage say, 60% of the 
functionality, even (sometimes you can get considerably more than that),, 
then that 60% that you didn't have to rewrite can get you new enhancements 
for free.

I'm not suggesting that refactoring should be applied unconditionallly or 
exclusively; all I have ever said is that it can be an option and can go 
into the mix with other options.

I have one client who simply COULDN'T afford to rewrite his application from 
scratch. He'd go broke. The time and cost of rewriting would be more than 
the business can support. And even if he did, he'd just have a new COBOL 
system which really gets him no further ahead. By refactoring to objects he 
can use those objects with ANY language he decides to implement (including 
COBOL). His migration can be phased at a pace he can support. There is time 
for training and coming to grips with a new environment.At the end of it he 
has a component based system that can be easily reconfigured and enhanced.

For smaller businesses the option to redevelop everything simply may not be 
viable. Migrating to a component based approach keeps options open and 
salvaging exisiting code into components makes sense for that environment 
(provided it doesn't cost a fortune to do it; that is why I am looking at 
automating it or at least assisting the process with tools...)

Pete,all

You may find the updated version of open source zcobol and z390 coming
out this week as z390 v1.5.00e to be useful.  This latest update
includes zcobol translator (zc390.java) which now successfully
translates all 459 ANSI COBOL 1985 test suite into HLASM compatible
macro assembler source (CBL to MLC).  The MLC macro calls for each
verb in COBOL can then be expanded into executable source programs in
HLASM assembler, C, Java, or Intel MASM using the included zcobol
macro libraries for selected target language.

There are examples of all 4 target languages which can be compiled
from original COBOL source to executable module and run on Windows or
Linux.  Only the HLASM target language is being fully developed
currently and at present 179 of the 459 NIST COBOL programs compile to
HLASM load module with only MNOTE warnings versus any more serious
assembly errors.  I hope to significantly increase this number over
the next few z390 updates.

The data divison fields are mapped into macro calls to WS.MAC macro
with positional parameters for all attributes. The WS macro loads all
data fields into global macro arrays which are then accessed by all
the COBOL verb macros to fetch required information to generate target
language source code for statements in the procedure division.  There
is also a GEN_WS macro which generates the targer language data field
definitions (DS, DC, and ORG statements for REDEFINES in the case of
HLASM target language).

The global macro data arrays containing all the data definitions and
procedure and section names could be used to generate any type of
analysis or code generation such as the OO modules described.
