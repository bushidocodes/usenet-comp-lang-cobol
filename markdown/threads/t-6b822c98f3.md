I can see the error (and also why it probably isn't one).

The original suggestion was to have an 01-level for the file status key and
then an 88-level for "00" (a good read).  The perform was done until it was
no longer "good" (00).

The (theoretical) problem with this was that you did not initialize the
status-key field before the loop.  Therefore, it would (theoretically) be
possible to skip the entire loop because of "bad" (non-00) data that was
already in the field.

Now, the reason that this (shouldn't) be a problem in this case (although we
didn't really see the full program) was that there SHOULD have been a
successful OPEN before the loop.  This should have already initialized the
status-key - so the loop would be performed if the OPEN was successful - but
not if the open had failed.

Having said, that - there is still the (again theoretical) problem of what
would happen if the OPEN was never performed before getting to the loop.
Clearly this would be a "logic error" - but if the filed "happened" to have
00's in it already, you could be in trouble on your first read.

Bottom-Line:
  Generally, if you are performing a loop until an 88-level becomes false, it
is a "good idea" to make certain that it is true before the first test (or
vice versa).
