OK for my first step basic concept:

The parser will contain a basic syntax parser  It will find every
division statement and have a parser for each.  If the parser des not
understand the code then it gives up, it assumes that the program
compiles so does not need to be smart, and you have the code and can
easily fix it.

release 1 will ignore anything between EXEC and END-EXEC.  Special case
later we might make it a bit smarter for SQL especially.

The parser for identification is simple: program id only (MVS extension
is handling the with debug and with recursion).  All else is obsolete.

The next is the environment division:

the data division has three sections

configuration section, object computer:   obsolete

Special-names:  ??

Input-output section:

File control paragraph:  File details marking obsolete bits and
recording others.

I-O-CONTROL.   ??? never user it ???  first pass of code ignores
entirely.

Data division:  has file section, working storage section, linkage
section (and obsolete communication where does this go)

The file section contains:  FD (with obsolete bits) and data definition.

Data definition is number, fieldname, PIC, + keywords= COMP-3, COMP,
BINARY, VALUES, etc.  (Oh look my cobol for MVS manual has 'justified
right' as a keyword - cool).

That's far enough for now:

In order to accurately parse the cobol code we use a directed parser. 
It looks at the first word and then assumes that it is that sort of
statement. (This gets harder in procedure with no dot to separate it.)

The parser will accept a list of keywords that are assigned a magic
number e.g.:  ZERO, ZEROS, ZEROES are all value zero, PIC and PICTURE
are value 1, etc.   The numeric value is defined by the cobol source
code but the load able table makes it slightly more extensible.

When we parse a variable name (defined by definition context) then the
variable text is stored internally to the program and defined as a
number.  When a variable is read in code it is parsed to VARIABLE,
VARIABLE-NUMBER.  This make the parser a bit easier to break out.  If a
variable is defined as the same value as a reserved word then it will
never be located in code as the reserved word look up is first.

The variable definition will have a fixed number of fields in it:

Internal reference number (implicit by index)

Level number.

text Pic x(32)  (any advances...)

pic descriptive.

Comp type  11=display, 12=comp, 13=Binary, 14=packed, 15=comp-3. 
(values defined from the reserved words table but again specific to the
parser program.  This means (again) that computational-3 and COMP-3 have
the same value.  Note that 88 get a special value of boolean.

Blank-when-zero clause (blank token assumes it is followed by when token
and then zero token, if it does not get it it dies with a message).

Value clause:  (String parsing is an issue here, reformatting it back
out in a pretty printer program will be interesting as well).

Justification clause:

Occurs clause, index, order by, depending:  Note that the depending
clause may 'predefine' a variable so there is a special case to handle a
later definition of this.

Redefines:

Sign clause:  trailing/leading, separate/not defined

Synchronized:  obsolete.

.....

Over to you, comment, rip apart, make better, or just bag it.

In earlier years, when I wordked with MF's work bench, I indeed used SPF/PC.
It's editor could be integrated in the workbench (replacing the standard
WrkBnch editor)) by replacing the standard EDIT.LBR with the specially written
EDIT.LBR provided by the builders of SPF/PC.

The other way around was a bit more tricky but possible: From the command line
of SPF/PC the MF-compiler was callable, compiling the source being edited. The
embodying of the resulting messages had to be done by issuing the command-line
command MSGFILE xxx.MSG, if memory serves me well. That was the tricky part:
one had to setup some compile option (directive) to produce a .MSG file and
after each compile one should thus tell the editor to read it.

Now-a-days I use NetExpress. Allthough it has still lots of workbench-behaviour
in it (those guys heavily re-used their software: the compiler directives, the
possible settings using environment variables, lots of other tricks and
features), NE is certainly not the same as the workbench. ... Now I come to
think of it, I'll look if there still is and "EDIT.LBR".
(looking around on my C: ...)
Nope. No EDIT.LBR

Besides, the NE source code editor has functionality enough for me so I no
longer feel a strong need to use others. And, what has SPF/PC more than source
code colourization and some elementary column handling? (I know the product up
to version 3). NetExpress uses the edit window also while debugging and has put
a lot of specific COBOL-oriented and own-product-oriented features in it. I
doubt if SPF/PC has that as a replacement-editor or makes it worthwile to use
their editor outside the NE IDE.

Anybody out there who knows if it is still possible to plug in another editor
in NetExpress?

The Frog
