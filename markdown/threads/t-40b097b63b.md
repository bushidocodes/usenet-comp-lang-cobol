Hello: I want to warn S. Comstock about the Block contains clause.
A few years ago, I developed a program on the IBM PC and forgot to
add the block contains on MVS when I transfered the source code.

At that time, the Block contains "told" cobol that the data set
was blocked, as opposed to unblocked. The program overides the JCL,
which overrides the dataset. Thus, if the real dataset is blocked,
as was the case in my program, the DCB will not match what the
cobol load module expects, and an abend will occur, since the
cobol expects an unblocked dataset versus the blocked dataset it
is given.

Here is some sample JCL:
//step01 exec pgm=test1
//input dd dsn=theinput,disp=shr,
// dcb=(recfm=fb,lrecl=80,blksize=8000)

Notice that the Recording format (recfm) is fixed blocked. This won't
match what the cobol program expects without the block contains clause!

In the cobol program:
fd input-file
(no block contains)
01 input-record pic x(80).

open input input-file.
(abend or bad return code on file status)

So the answer is Block contains is required for MVS, but on ibm pc,
it isn't really necessary since MS-DOS stores data outside of
the cobol programmers control as far as blocking factors....
