Quite to the contrary, Bill, it will work with any Cobol
I have ever seen. The records are line sequential,
and the array does not need to be variable length.
The unused array space will be blank filled.

When the data is processed, it is a simple matter
to do a perform varying the subscript from 1 by 1
until greater than the number of courses. You
process the data using the actual size of the data,
but you read a line sequential data file with the
buffer size equal to the largest record.

Using an array length in an FD record that varies
depending on a number contained in the record
being read, however, I have my doubts.  That is
a construct I would never use, and just makes the
code more difficult to fathom, at least in my opinion.

Even if it does work, the processing should not
vary an iota, (though it must vary depending on)<S>.

I prefer not having field names, beyond what is absolutely required in
my FD 01 description.  I either READ INTO or do a READ then a MOVE.  

The simplest way to do your work is to figure out what the maximum value
can be for NUM-COURSE-IN.  (The size of your input record may determine
this).

First check to see that your value for NUM-COURSE-IN is in the accepted
range - this check is ABSOLUTELY necessary in COBOL because COBOL does
not check to make sure you're processing within your table.

Say the max = 12.
Then also make
05 COURSE-INFO-IN        OCCURS 12 TIMES. 

Create a subscript variable      COURSE-IDX   PIC 9(03)

Now I can't really tell you what to do next, it depends on what you want
to do with your data!  But here's an example:

    PERFORM 0100-PROCESS-COURSES
       VARYING COURSE-IDX FROM 1 BY 1
         UNTIL COURSE-IDX > NUM-COURSE-IN.

0100-PROCESS-COURSES.
    DISPLAY "SECTION = ' SECTION1-NUM-IN (COURSE-IDX)
            "  GRADE = ' FINAL1-MARK-IN  (COURSE-IDX).
