I've developed an application using MicroFocus Cobol/2 (c 1990 - old I

The same application (without any modifications) has run successfully under
DOS, Windows 3.1, Windows 95 and Windows 98 without any printing problems.

fine except printing. When I print a report, nothing prints on the printer
UNTIL I exit the application when the report prints OK. It appears as though
all printer output is buffered until the application exits.

The code looks like this:


Under the Environment Division:

      Select PrintFile Assign "LST:"
      Organisation Line Sequential.


In the Data Division:

FD PrintFile.
   01 PrintLine.
      03 PrintRecord PIC X(132).


When printing:

Move "alskdfsadh" to PrintLine.
Write PrintLine.



Any suggestions would be greatly appreciated. Hopefully there is an NT
configuration/printer configuration option which can be modified (printer is
a dot-matrix NEC).

It can think of a few things...

1) If it's a 16-bit application these notes from MSDN apply

--- start of MSDN excerpt ---

MS-DOS Applications Running Locally
If you print from an MS-DOS application that is running locally, or from a
Windows NT command line utility that sends data to a printer port, NTVDM
receives the print job. NTVDM queries the redirector to determine if the
redirector is managing the target port (you run NET USE LPT1:
\\SERVER\SHARE) or if the spooler is managing the port.

If the redirector is managing the target port, the redirector takes control
of the print job, and none of the spooler options are used. If the
redirector is not managing the port, but a printer defined in Print Manager
is printing to that port, the print job goes to the printer, and the
printer's default datatype and spooling options are used. If neither the
redirector nor any printers are managing the port, then the job goes to the
port device driver, unaltered.

For example, one printer in Print Manager prints to COM1, none of the
printers in Print Manager print to LPT1, and you issue a NET USE command to
redirect output from LPT2 to a network print share. If an MS-DOS-based
application prints to LPT2, the print job goes to the network print share.
If the application prints to COM1, NTVDM submits the print job to the
printer that prints to COM1, and used that printer's default datatype. If
the application prints to LPT1, NTVDM submits the job directly to the
parallel port device driver; no alteration occurs.

--- End of MSDN excerpt

2) If you have not closed the print file then it may not be handed to the
spooler, so it waits until the program exit closes the file before printing.

3) You may need to disable the spooler for that printer if it used primarily
by your 16-bit application.

4) Visit www.microsoft.com and check the KnowledgeBase and troubleshooters

-- Karl Wagner
