Just for the "record" the following source code will compile "cleanly" and
is semi-valid with IBM mainframe compilers (discussion afterwards), (please
assume correct "columns" if they don't show up in your browser)

        Identification Division.
          Program-Id.  TestPgm.
        Data Division.
         Working-Storage Section.
        01  Test-Field   Pic X  Value "Y".
       Procedure Division.
        Main-Para.
            If Test-Field = "N"
               Next Sentence
            Else
              Display "Always"
            End-If
            Display "In same sentence"
       *       No period - full-stop here
        New-Para.
             Display "New Paragraph *and* Sentence"
        Fall-Thru.
              Stop Run.

****

Reasons -

1) IBM has a documented extension that allows NEXT SENTENCE with END-IF

2) IBM has an "undocumented" extension (which gets at least a W-level
message) that allows the "final" period before a paragraph-name (starting in
the A-margin) to be omitted.  (This "extension" *will* go away with the next
COBOL Standard - when the A-/B-margin distinction goes away)

Again, I do *not* recommend the use of either of these, but did want to
comment on whether or not this was "valid" source code (and the fact that
this depends on compiler used and acceptance of error/warning/informational
messages).

Just a comment on the introduction of the enhanced EXIT statements and
"structured programming".

As has been clear in recent discussions, there is a LOT of different
definitions of WHAT exactly Structured Programming (not Structured
methodology) means.  At least one definition says that you need to have a
"single entrance/single exit" for each procedure.

The new EXIT statements (unlike the NEXT SENTENCE phrase and the GO TO
statement) actually ENFORCE the behavior.  Because the "exit" spot is an
un-named point (end of PERFORM, PARAGRAPH, or SECTION) and there is (by
definition) ONLY ONE per procedure, this is viewed by SOME as enforcing
"structured programming".

For those shops that (today) "enforce" the GO TO xxxx-EXIT paragraph
structure and this is the ONLY time that a GO TO is allowed and there are no
paragraphs/sections between the GO TO and the EXIT paragraphs, this new
"format" works the same way and is just as structured (or unstructured -
depending on your definition) as the older technique.  The advantage that it
has, is that it is IMPOSSIBLE to create spaghetti code with the new features
because you are unable to put a "new" exit or go-to in the middle of the
existing procedure.

  ***

From a "real world" perspective, I really like the new features as it allows
a (structured) method of doing an "immediate get out" of a procedure - when
a specific error (or other) situation is detected.  This gets rid of the
need of either GO TO or deeply nested (or repeated) IF statements.

Come ON Stephen,

 The definition of the EXIT PARAGRAPH is *very* carefully written so that it
DOES go to an "implicit continue" immediately after the last statement in
the paragraph.  This means that there is NO way to cause "confusion" in
maintenance.  There is nothing that you can add or subtract that is going to
make this have "crossing" logic paths.  This is WHY having a "single exit"

Logically, the above sequences are inconsistent with the idea that
sections are containers for paragraphs (FCD page 389, 14.5.1,
"A section consists of ... paragraphs."). The logical sequence is:

1. Implicit continue from EXIT PARAGRAPH.
2. Return from perform of paragraph BZ.
3. Implicit continue from EXIT SECTION.
4. Return from perform of section B.

The defect may be repaired by making the following changes
to the FCD.

Page 389, 14.5.2 Paragraphs, change the last sentence to read,
"A paragraph ends immediately before the next paragraph-name,
the end of the containing section, or the end of the procedure division."

[The text referring to declaratives is removed since no paragraph
in the declaratives portion of the procedure division would share
a boundary with the keywords END DECLARATIVES.]

Page 455, 14.10.13.3 EXIT GR 10, change in part to read,
"... an implicit CONTINUE at the end of the current paragraph."

Page 455, 14.10.13.3 EXIT GR 11, change in part to read,
"... an implicit CONTINUE at the end of the current section."

Page 496, 14.10.27.3 PERFORM GR 3, change in part to read,
"a) ... return is immediately after the end of procedure-name-1.
 b) ... return is immediately after the end of procedure-name-2."

The above changes also eliminate the conflict with the implicit
continue of EXIT PERFORM when PERFORM ... END-PERFORM
is the last statement of a paragraph.

Over a long period most programs will grow as new requirements are added
to meet changing business needs.  The growth may be in specific modules
and this may make the modules unwieldy and difficult to maintain.  The
solution may be, as you say, to break the module down into more easily
handled parts in a way not envisioned in the original design.

There are mechanisms which make breaking a module up (or down) difficult
and costly in terms of programmer time and, especially, testing and
debugging time.  One such mechanism is the 'perform section (or thru)
and go to exit'.  On the one hand it appears simple to just 'jump to
exit' to avoid the 'complications' of nesting the code within and IF ...
END-IF, on the other there is the very high cost of reconstructing the
code as two or more modules.

These costs may involve having to (by some convention) assign new
hierarchical level numbers for the labels, moving the code to new places
in the source determined by the number, creating the bracketing for the
code (headers, exits, etc), and most importantly rewriting all the logic
to cater for different exits, with flags being passed back so that a 'go
to exit' can take place when necessary.

This high cost, and the resulting messy code, is often blamed on the
'fetish' of trying to remove GO TOs.  In fact attempts to remove GO TOs
_without_ breaking down the module often leads to less desirable code,
such as 500 line IF ... END-IF. I can see that many 'high overhead'
modules will think that the EXIT paragraph/section is a saviour that
will allow removing the GO TO by replacing it with an EXIT SECTION.  

However, as you say, the EXIT SECTION is no better than a GO TO ~Exit,
except for the tiny assistance that you cannot specify the wrong ~Exit
and have the program rage out of control.

These do not, however, help at all in making the program more modular,
and certainly are just as much an impediment to breaking modules up as
GO TO is.  Thus modules will continue to grow as code is added during
maintenance and restructing becomes more costly and less likely to
occur.  The programs become less modular as the same number of modules
takes in more code and more requirements.

Programmers in high module overhead environments will not understand
your 'breaking into two or more paragraphs' at all.  They do not see
this as a solution to any problem because it is simply too expensive for
them to even think of doing.  

Of course I break down modules all the time to simplify them, but I do
not use any mechanism that gets in the way of doing this.  By using a
low overhead module construction I can rip out a nested IF or break a
module in half in a trice, all it takes it to write a new label name,
copy line and add PERFORM then block move the code a few lines, or leave
it where it is depending on what follows.

Because I do not use any high overhead mechanism (such as section, thru,
go to, exit paragraph, etc) I am assured that breaking down modules
(paragraphs) in this way will never lead to loss of control or unwanted
logic flow.  This means that I never need check, or even remember, when
to do fix-ups.

The difference between low overhead module and high overhead module
programming is so great that practitioners of the latter will not
understand how the former could possibly work, so they will not
understand why you object to EXIT section/paragraph, after all it does
allow them to replace the GO TO (by an implied GO TO) so it must make it
