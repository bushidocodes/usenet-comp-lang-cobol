It will come in on the command line, exactly as if you had named the 
file at a prompt. For example, clicking on file ABC in C:\temp is 
identical to typing "MYPROG.EXE C:\TEMP\ABC". The default area is *NOT* 
set, so it is not quite the same as a PIF start.  You will have to know 
where your program is to open a file on *your program's* area, or parse 
the file name to open a file in the same area as the file. The default 
area will likely be your windows desktop at execution time, so be 
careful when opening files, or they will end up on your desktop. That 
includes temporary files that are opened without a qualified name.

This is not a "precise" answer because I've never tried it. You would do 
better to join M/F's Answer Exchange and ask the question under Net Express.

However, one possibility. I'm assuming the file(s) you want to select 
from are probably in one directory (folder) ? Whether using  (a) Dialog 
Editor and GUI Classes, or (b) Dialog System - look at the documentation 
on Common Dialogs - File Dialog, (again haven't tried it) but I *think* 
you can set the 'Open Dialog' to the folder/pathname that you want. 
Actually there are so many ways you could do this - check the Answer 
Exchange.

As one example - I had a need to convert RM/COBOL files to M/F format. 
Initially I create RM ASCII files (CSV), then from there generate input 
M/F CSVs, (system design changes - since originally written in RM). The 
new CSVs may contain errors - missing info required for new application. 
(Have a look at the OO classes - Class 'Filename', which somewhat 
parallels the CBL_FILE routines). I know which folder these text files 
are in so I can use a wildcard to get an OrderedCollectiion and then 
display them in a Listbox. User can then click and edit each 
independently. (Note - as a default in the application - the user sets 
his 'favourite' editor which is then loaded and opens the selected file).

Possibly take a look at the X"??" routines - might find features there 
that you can intermingle with the above.

You probably *could* do it via Windows Explorer - but I think that's a 
tough furrow to plough. If you go to Answer Exchange check, (SEARCH 
feature) - Treeviews - David Sands for an example of generating your own 
'Explorer' Treeview.

However - where are you going ? Are you going to migrate to Fujitisu 
doNet or upgrade to M/F dotNet ? Both are different and there are also 
*minor* subtle differences, (for OO), between  N/E 3.1 and N/E V 4.0 
dotNet. (If it's F/J - check their documentation on Common Dialogs).

>>Should I just start with Fujitsu's Cobol?  Ultimately it would give me >>better distribution opportunities". 

The run-time fees ! To be honest, why bugger around - if you are going F/J dotNet -  why not get your teeth stuck into that product, rather than a two-stage process which will ultimately result in rewrites ?
