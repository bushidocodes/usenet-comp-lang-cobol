Theo,

Actually, your SAS code is too wordy... Using TRIM function is redundant since
LENGTH function returns the length of a string without trailing blanks, anyway.
Hence,

FIELDLEN = LENGTH(FIELD);

will suffice. Here is some extra couple of items that would make the lives of
fellow COBOL guys a lot easier if they were implemented in COBOL:

1. Length of a string including LEADING and TRAILING blanks:

FIELDLEN = LENGTH(RIGHT(FIELD));
  
2. Length of a string without LEADING blanks:

FIELDLEN = LENGTH(LEFT(FIELD));

3. Length of a string with all blanks eliminated:

FIELDLEN = LENGTH(COMPRESS(FIELD));

4. Length of a string with multiple blanks squeezed into a single blank:

FIELDLEN = LENGTH(COMPBL(FIELD)); 

5. Number of embedded blanks in a string:

NUMBER = LENGTH(LEFT(FIELD)) - LENGTH(COMPRESS(FIELD));

6. Number of times the character '*' appears in a string:

NUMBER = LENGTH(COMPRESS(FIELD)) - LENGTH(COMPRESS(FIELD,"*,' '"));

7. Number of words in a string delimited by blank(s):

NWORDS = (COMPRESS(FIELD) NE ' ') * 
         (LENGTH(LEFT(COMPBL(FIELD)))-LENGTH(COMPRESS(FIELD))+1);

8. A string with leading blanks removed:

FIELD = REVERSE(TRIM(REVERSE(FIELD)));

9. A string shifted 1 character to the right:

FIELD = ' ' || FIELD;

10. A string shifted N characters to the right:

FIELD = REPEAT(' ', N - 1) || FIELD;

11. A string shifted N characters to the left:

FIELD = SUBSTR(FIELD, N + 1);

12. The FIRST nonblank character of a string:

FIRST = SUBSTR(LEFT(FIELD),1,1);

13. The LAST nonblank character of a string:

LAST = SUBSTR(LEFT(REVERSE(FIELD)),1,1);

or

LAST = SUBSTR(REVERSE(RIGHT(STRING)),1,1);

14. The digits of a phone number with slashes, dashes, parentheses removed:

PHONE = COMPRESS(PHONE, '/-()');

15. A phone nnumber with letters converted to digits:

PHONE = TRANSLATE(PHONE,'22233344455566677778889990',
                        'ABCDEFGHIJKLMNOPQRSTUVWXYZ');

16. A left-aligned string:

FIELD = LEFT(FIELD);

17. A right-aligned string:

FIELD = RIGHT(FIELD);

18. A centered string:

FIELD = SUBSTR(RIGHT(FIELD), CEIL(.5 * (LENGTH(RIGHT(FIELD))
                            - LENGTH(LEFT(FIELD)))) + 1;

And so on... Of course, it is just a tiny tip of the iceberg. 

Regards, Paul.

Brian,

First, it is I/O handling and not string manipulations where COBOL's performance
superiority versus SAS actually resides. Second, it seems to me that you have
missed Theo's point. It was simply asserted that it would be a lot more
convenient and self-explanatory to be able to write something like

MOVE LENGTH(FIELD) TO FIELDLEN

rather than having the FIELDLEN as a byproduct of UNSTRING statement with the
extra expense of a junk variable, to say nothing of the more verbose COBOL
example posted by Theo. Another big benefit of having functions instead of
corresponding statements is the ability of the functions to be nested. That
allows to avoid using auxiliary variables performing no duty but holding
intermediate results. Whilst it is silly to replace a self-documenting statement
by a cryptic function for the sole purpose of being concise, it is hardly
possible to invent anything more self-explanatory, efficient and syntactically
simple than 

FIELDLEN = LENGTH(FIELD)

(Theo's usage of TRIM function is redundant), regardless of whether it is 'MOVE'
from left to right or '=' from right to left.

Paul.

I don't want to "defend or attack" any parts of this conversation so far,
but did want to make a couple of points.

One of the examples of how things could be done in COBOL used the CALL verb.
This was simply trying to demonstrate that a STANDARD program could call a
simple (user-defined routine) to do whatever you want with a string.  With
the draft of the next Standard (and by the way - you will note that there is
no such thing as "Standard SAS" - because SAS is a proprietary language),
you will also be able to have user-defined functions as well as subroutines.
This means that in the next Standard (and those implementations that have it
already as an extension), the CALL statement could be changed to a function
reference - just like the SAS example.

As far as adding things to the Standard, COBOL for various reasons has had
pretty poor string handling facilities.  (STRING, UNSTRING, and INSPECT -
along with some '85 Standard intrinsic functions can do most things - but
are pretty awkward and/or inefficient in many cases).  When the Standards
committee began planning the scope of the current revision two topics that
were discussed but were rejected (and I cannot tell you why) were:

  - improved string handling functions
  - enhancements to report writer to make it compete with "word processing"
programs

In both cases, it would seem to me that the features fit well in the arena
of a "business oriented language" but in both cases, little work has been
done in the current draft Standard.  My guess is that they think that, for
these facilities, there ARE better products than COBOL to do some tasks.  If
I were you and I were interested in COBOL improving its facilities in these
areas, I would send a note to Don Schricker (the chair of J4) at
das@microfocus.com and suggest that the items be added to the (already
existing) list of possible enhancements for a future Standard.

Just FYI,  in my opinion intrinsic functions could be provided for
  - stripping leading/trailing characters
  - finding the location of one string within another string
  - padding (leading/trailing) a string
  - several items existing in languages like SAS, REXX, etc

Note: Most of these CAN be done using INSPECT, STRING, UNSTRING, etc - but
NOT in what I would call an "intuitively obvious" syntax.
