There are three types of data which I have needed in order in on-line
processing.

1.  Large amounts of data - which need to be sorted before hand, either
with an indexed file or an ordered database.

2.  SQL queries which get sorted via my SQL calls (I am currently
working with IDMS, so that doesn't apply).

3.  Very small amounts of data - in which case I code the sort by hand,
figuring that the overhead of calling an external sort is larger than
the inefficiency of using COBOL procedural code.

Again, a difference in working environments.  As a Var,
my product is usually going into a standalone computer,
and most of the data originates at my end.  So I get
completely random events happening, and I funnel them
into a format for a shop like yours.  I normally, for example,
would build an Isam file of them during the day, then
purge into a sequential/ordered flat ASCII file
forwarded to head office.  The Isam file is then
closed, opened output, closed, and reopened in I/O mode
for the next day.

I never see sequential input files, unless it is a conversion,
then they are dumped in order.
