I'll take your word for it that the code is not ANSI-conforming.
However I don't understand your point about "C" being an inline
PERFORM.  Suppose I make the example a little less schematic:

 PERFORM
    IF AAA = 3
	PERFORM PARA-A3
    ELSE IF BBB = 7
	PERFORM PARA-B3
    ELSE IF CCC = 5
        PERFORM UNTIL COUNTER > 10
           PERFORM PARA-C5
           ADD 1 TO COUNTER
        END-PERFORM
 END-PERFORM

If that's the sort of thing you had in mind, then I don't see the
problem.  The first END-PERFORM would terminate the inner inline
PERFORM, and the last END-PERFORM would terminate the outer one.
With only one END-PERFORM there would be a syntax error because the
outer PERFORM would be unterminated.

Were you thinking of something else?  Off the top of my head I can't
think of a case which would lead to ambiguity.  Since others have
indicated that some compilers accept this syntax, it must not be too 
difficult to implement.

I'm beginning to like this proposed construct.  Legalizing it would
disrupt the language less than adding OR-IF (or one of its other 
spellings).  Unlike OR-IF, it would enable us to eliminate excessive 
END-IFs in cases where EVALUATE wouldn't help.
