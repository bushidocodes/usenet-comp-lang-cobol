I don't know about DFSORT, but for SYNCSORT, one needs the
input exit (15?) to process multiple VSAM input files. You
can't concatenate VSAM for input as you can SAM (QSAM
typically).

The only other reason for the exits is where you need to
do/handle things that SORT can't necessarily deal with, such
as short records where the selection criteria occurs after
the record ends -- I typically run into this with SMF data.
Some records happen to have special cases where a normal
item you need to check is beyond the length of a special
"short" record.

We have a weekly batch reporting job that reads in a multi-volume tape
file, typically 150 million records.  The file is sorted by several
fields with customer number being the high-order portion of the key. 
A few records required special handling where the original customer
number was replaced by a different customer number, depending on
certain business rules.

Originally, we used SYNCSORT to sort the data and split it into two
files.  A separate COBOL program modified the customer number for the
special cases (which required a separate file lookup) and both files
were merged together (again using SYNCSORT).  

Since the unmodified records were usually 99% of the file, and that
file was usually more than 15 tape volumes, we were able to process
the big file only once by using an E15 SORT exit to modify the keys
and sort all the records in one pass.
