Bill,

  There must be something wrong with our books, because you can never call a
C subprogram if it has been marked as MAIN.  With C a program is predestined
to be either a SUB or a MAIN, but it can't be both.  I am a little confused
about 'calling a function directly'...does he feel that that is different
from 'making a dynamic (non-DLL) call from a
COBOL module to a C module'?

  ILC works great, and many customers have done it successfully. In fact,
with the newer COBOLs you can make the COBOL program look like a C program
for '#pragma-less ILC'.  Since Colin did not mention #pragma, nor whether he
is using C or C++ (different rules, same compiler) I wonder if he is aware
of the C/C++ differences.  The old way to have a C program mixed with COBOL
(and it still works) is to use '#pragma linkage
COBOL' in the C program, and force the C program to follow 'normal' linkage
rules.
Nowadays there is another choice of using new COBOL language features
(USING BY VALUE, RETURNING, etc) and avoiding the #pragma statements in
the C programs.  Now for C++, (which uses EXTERN statements rather than
#pragma) he could use EXTERN COBOL or the new COBOL features.  In all cases
he has to designate the C or C++ program as a MAIN or SUB, and you can't
call a MAIN
from another program, because then it is a SUB!

  So, the questions are:
  1) C or C++?
  2) #pragma-less or EXTERN-less?
  3) Can Colin get a C program to call a C program successfully?
  4) Does he know about how to get his C function as non-DLL,
      or is he maybe getting DLL behavior accidentally?
  5) If C++, does he know about SOM C++ versus non-SOM C++?
     (Only SOM C++ can work with COBOL)
