College student writing a sequential file update program.  Master File +
validated Transaction file gives Audit Report (printer) and New Master file.
I am wondering if I can avoid using so many MOVE statements. About 7 fields
in Master file, same 7 in Transaction file with a Transaction code at the
beginning of each record.  Output is to printer and Master file (that is, of
course) has same fields as the original Master file.

Is there a way to move one record to another without field to field moves
for every field?  If I write the MASTER-RECORD to the NEW-MASTER-RECORD,
since they are the same, I can WRITE
NEW-MASTER-RECORD FROM MASTER-RECORD. That's easy enough.

Other moves are going into records which contain the same data items (same
PIC's and all) but in different order, and sometimes with an added blank PIC
X field.

Tks to any ideas,
George

As Bill Klein has pointed out, you can use MOVE CORRESPONDING.  But I
believe that is a bad idea, and not a good way to address this issue,
especially with today's programmer's editors.  Here's my approach:

1. Always use a consistent data naming scheme.  This is an *extremely*
   important thing.  I have found it expedient and profitable to follow
   these rules:

   A. Always append a short (2-5 character) prefix to each data name.
      This prefix is unique to each 01 record within the program.  An
      acronym derived from the 01 data name works well.  e.g. If you
      have an 01 named DATE-WORK-AREA, all the subordinate data names
      should start with 'DW-' or 'DWA-'. (I use 'WS-' for all 77's)

   B. The remainder of a given data name is consistent throughout the
      entire application, not just a single program.  The only exception
      is when a given data field actually changes it's essence when moved
      from one place to another.  For example:
      MOVE DW-CURRENT-DATE TO DW-PREVIOUS-DATE.

   C. Use consistent abbreviations.  e.g. Don't abbreviate 'NUMBER' as
      'NBR' in some data names and 'NO' in other data names.  When writing
      computer systems, I often create an 'abbreviation dictionary' for
      consistency.  One primary difference between the amateur and the real
      pro in any field is consistency.  The amateur can produce good ones
      on occasion, but the real pro turns out good ones consistently, like
      a cookie cutter.

2. Using your nifty-gee-whiz programmer's editor create a line with just
   'MOVE' in the appropriate column, then duplicate it at least as many times
   as you have fields to be inserted (see next step).  The result will look
   like this:

           MOVE
           MOVE
           MOVE
           ...
           MOVE

3. Use your editor's 'column copy' command to copy the names of the 'from'
   fields from the record description (01) immediately to the right of the
   MOVE's.  After you insert all the field names, you may need to delete any
   names which aren't needed.  The result will look like this:

           MOVE TRAN-FIELD-ONE
           MOVE TRAN-FIELD-TWO
           MOVE TRAN-FIELD-THREE
           ...
           MOVE TRAN-FIELD-NNN

4. Use a 'column insert' or macro to place 'TO XXXX-' after the field names,
   where 'XXXX-' is the prefix of the receiving fields.  You now have this:

           MOVE TRAN-FIELD-ONE   TO MAST-
           MOVE TRAN-FIELD-TWO   TO MAST-
           MOVE TRAN-FIELD-THREE TO MAST-
           ...
           MOVE TRAN-FIELD-NNN   TO MAST-

5. Now use your editor's 'column copy' to copy the part of the data names
   after the prefix from the left to the right.  The result should look like
   this:

           MOVE TRAN-FIELD-ONE   TO MAST-FIELD-ONE
           MOVE TRAN-FIELD-TWO   TO MAST-FIELD-TWO
           MOVE TRAN-FIELD-THREE TO MAST-FIELD-THREE
           ...
           MOVE TRAN-FIELD-NNN   TO MAST-FIELD-NNN

6. When I want periods after each move, I make sure to copy a space at the
   end of each data name, so each line will end with at least one space.
   I now do a search and replace on those lines, replacing all ending spaces
   with a period.  In Brief that sequence would be: replace ' +>' with '.'.
   You now have:

           MOVE TRAN-FIELD-ONE   TO MAST-FIELD-ONE.
           MOVE TRAN-FIELD-TWO   TO MAST-FIELD-TWO.
           MOVE TRAN-FIELD-THREE TO MAST-FIELD-THREE.
           ...
           MOVE TRAN-FIELD-NNN   TO MAST-FIELD-NNN.

Though this took a while to describe, with practice you can do it in just a
few seconds.  The above sequence on 50 moves would take me perhaps five to
ten seconds.  Plus, because you use the editor to copy the data names from
the record itself, the result is unlikely to contain typing errors.

If you don't have a programmer's editor that can do the above kind of thing,
then get one! :-)

So... I tend to use MOVE CORRESPONDING a lot, especially with dates. 
My 'template' always has something like this in it:

        01 WS-DATE                     PIC 9(8).
        01 WS-DATE-FIELDS REDEFINES WS-DATE.
           15  THE-YEAR.               Pic 9(4).            
           15  THE-MONTH               PIC 9(2).
           15  THE-DAY                 PIC 9(2).


 Almost all the 01 and other fields that have dates are then:

        01 WS-DISPLAY-DATE.
           15 THE-MONTH                PIC 9(2).
           15 FILLER                   PIC X VALUE '/'.
           15 THE-DAY                  PIC 9. 
           15 FILLER                   PIC X VALUE '/'.
           15 THE-YEAR                 PIC 9.
  
TO display the date, for example, them becomes something like
     ....
     .....
            accept ws-date from date YYYYMMDD
            move correspong ws-date to ws-display-date
            display ws-display-date
     .....
     ....

It works just as well when you split up the year in CC and YY,
and allows you to always store dates in the system native fashion. ;) 
It does trip up people because they can not do a MOVE CORRESPONDING
from WS-DISPLAY-DATE to ws-date, at least in most compilers. 
The filler fields cause some kind of consternation in the compiler.


-Paul

Remember, you can have multiple levels in a record.

01  my-record.
    05  my-field1
    05  my-field2
    05  my-field3
    05  my-field4

If you can't move my-record because my-field4 has a different format, do
the following:

01  my-record.
    05   my-move-fields 
      10  my-field1
      10  my-field2
      10  my-field3
    05  my-field4

Now, you can move my-move-fields and move my-field4.  See if this
strategy makes sense for your data.

If you have 2 *identical* records, like this:

01  record-1.
    05  fld-a  pic x(5).
    05  fld-b  pic 9(5).
    05  fld-c  pic s9(7) comp-3.

01  record-2.
    05  fld-1  pic x(5).
    05  fld-2  pic 9(5).
    05  fld-3  pic s9(7) comp-3.

You can MOVE RECORD-1 TO RECORD-2.  Notice that field names do not have to be
the same, but field descriptions do.


If you have 2 dissimilar records, but with identical field names, like this:

01  record-1.
    05  fld-1  pic x(5).
    05  fld-2  pic 9(5).
    05  fld-3  pic s9(7) comp-3.

01  record-2.
    05  fld-2  pic 9(5).
    05  fld-x  pic x.
    05  fld-3  pic s9(7) comp-3.
    05  filler pic x.
    05  fld-1  pic x(5).

then you can MOVE CORRESPONDING RECORD-1 TO RECORD-2.  A lot of people,
myself included, really don't like MOVE CORRESPONDING, for a variety of
reasons.
