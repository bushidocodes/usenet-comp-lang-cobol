True, but in this case the usage is obviously DISPLAY, that is, it is
what's known as a 'zoned decimal'. The '{' is an EBCDIC 'C0'x and the
'H' is an EBCDIC 'C8'x, which means that it's positive (the 'S' in
'S9(7)V99' signifies that the number is signed). If the 'S' were
omitted, and the data element was the target of an elementary move (as
opposed to a group move), the 'C8'x would most likely become an 'F8'x,
and the value would look more like '000030218' instead of '00003021H'.
Oh, one more thing: the 'V' is an *implied* decimal. The compiler knows
where it is when it comes time to do decimal alignment and calculations.

Douglas,
This is NOT floating point representation. What you have here is
numeric with an implied decimal point and overpunch sign on the last
byte. 00000000{ is zero with a hex C0 as the last byte
(F0 F0 F0 F0 F0 F0 F0 F0 C0 is how the hex representation would look in
a core dump). The implied decimal would equate the value to
0000000.00+.

00003021H would equate to 0000302.18+ (F0 F0 F0 F0 F3 F0 F2 F1 C8 in
hex)
Remember you are working with EBCDIC values here! Get yourself a
System/370 reference summary quick reference card to interpret EDCDIC to
ASCII values but the general idea with trailing overpunched signs is as
follows:

value Hex EBCDIC representation
----- --- ---------------------
0+ = C0 {
1+ = C1 A
2+ = C2 B
.
.
.
9+ = C9 I
0- = D0 }
1- = D1 J
2- = D2 K
.
.
.
9- = D9 R

I hope this helps you solve you problem. Just remember for usage in C
that the sign bit is combined on the last 2-bit byte of the word.

For starters, PIC S9(7)V99 is *fixed* point, not floating point. The V
indicates the position of the implied decimal point. S means it's signed,
9(7) means 7 numeric digits, 99 means two numeric digits. So S9(7)V99
describes a signed numeric field, nine digits in length, with a decimal
point implied between the 7th and 8th digits.

The last character is a numeric digit with an embedded sign:
{ = 0+, A = 1+, B = 2+, ..., H = 8+, I = 9+
} = 0-, J = 1-, K = 2-, ..., Q = 8-, R = 9-

Thus, in your examples,
00000000{ = 0.00+
00003021H = 302.18+

and
00004321K = 432.12-

Hope this helps.

Once everyone else has squared you away. I would like to offer a
viewpoint that might help (instead of words like zoned decimal, etc).
Think of the number as an INTEGER (long or short, whatever fits the
number of significant digits in the PIC clause (eg PIC 9(02)V99 is 4
digits, PIC 9(07)V99 is 9 digits)). The V symbol is to tell you where
the decimal place goes so you can scale your operations properly.

So that if you have a PIC 9(09)V99 field with a value of 12345.67, the
represented as 1234567 (once you strip off the trailing useless stuff)
and you try to multiply by a number defined as PIC 9(01)V99999 with a
values of 0.01234, represented as 001234; what you are really doing is
represented by the following formula:
((1234567 * 10^-2) * (1234 * 10^-5))
which can be restated as:
((1234567 * 1234) * (10^-2 * 10^-5))
which can be simplified to:
((1234567 * 1234) * 10^-7)

Any way, I hope this makes sense and is helpful.

This could be done better, but I'm trying to illustrate... and be brief,
and, of course, terse

Ascii or ebcdic:

char *n; /* pointer to ebcdic number */
int i; /* typically cryptic c style index */
long value; /* resultant value */
double real; /* another form */

value = 0;
for (i = 0; i < 9; i++) /* 9 significant digits in number */
{ value = value * 10; /* multiply current value by 10 */
value = value + (*n++ & 15); /* add low-order nibble */
}
n--; /* point back to the sign byte */
if (*n & 0xd0 == 0xd0) /* negative sign ? */
value = -value; /* make the result negative */
real = (double)value; /* convert to real first */
real /= 100.0; /* then insert the decimal point */

As stated, the "V" implies a decimal point
The high-order 4 bits of the low-order byte contain the sign:
0xc0 = positive
0xf0 = unsigned
0xd0 = negative

› real = (double)value; /* convert to real first */
› real /= 100.0; /* then insert the decimal point */

The above lines are a potential source for rounding errors. Since COBOL
works with fixed-point calculations, it does not have a rounding-error
problem, but C, with its floating points, does. Fixed-point represents the
number exactly...floating point, no matter how precise it is,
double-precision, triple- what have you, is just an approximation. After a
whole lot of additions, subractions, mults, etc. you'll eventually get
penny-off errors with C, which will throw off balances and cause
comparisons like

if (x == y)
doSomething();

to fail in C when the equivalent program in COBOL will succeed. This is
because the rounding errors in C have made x and y a tiny bit different,
even though they should be exactly the same, so the "if" will fail, since
they're not "equal".

There are solutions to this problem in C, but it involves using a
"fixed-point math" library (either 3rd party or "roll your own"). It's a
bit more complicated than just dividing by 100 and converting to "double".
As I said earlier, in COBOL, the problem just does not come up.
