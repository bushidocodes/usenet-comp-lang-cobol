```
I have an application that uses a security key dongle. The key requires that
it be polled every 90 seconds to ensure that the application is still
active.

I am trying to determine the best method of accomplishing this without using
too many system resources.

I have thought about creating a thread that would check the time and after
90 seconds,  poll the key.
I also understand there is a Win API called "SLEEP" that would create a
pause that I could use in the thread that checks the key.

Has anyone ever done anything like this?
Does anyone know of the best way to do this without expending too many
resources?
Any ideas would be appreciated.

Thanks,
Kevin
```

```
I responded in the Flexus list, Kevin, but am including it here
as it may be of interest to a few not using SP2 as well.

I am just setting the timer in the screens for polling.  That will force a
screen
exit with a "timer" keycode.  Since I am coding my screens as objects, the
routine to service the "interupt" (even though it is really a poll) is
buried in
the object.  The structure is as follows.

The screen is an object of class "x", and is generated by SP2. A second
class (Y) inherits all the screens, and manages the common elements.
Screens are opened by invoking them by name from class Y, which sets
screen position, and handles the "help key", as well as any timed events.
Since all screens are of that class, it is common for all screens in the
system.  Note that this means *every* screen in the system must have
the timer set, but that can be done in class Y, as part of the open.

It works very well for most stuff, provided you have nice big buffers
for any serial input.  I am polling three RS-232 devices plus the keyboard
for all my screens.  Input can be from any of the four devices.  One
RS-232 looks like a second keyboard, and actually "fakes" SP2 return
keys quite invisibly.  The other two RS-232 are a card reader and a scale.
The scale is just read, but the card reader fills in a screen field, then
"presses" the "APPLY" button on a screen.  The high level Cobol code
does not even know that it was not operator input, which makes it nice
and clean to code.
```
