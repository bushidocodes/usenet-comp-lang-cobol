Hi Frank,
Gael meant remove the WINAPI from the CALL statement for simple C calls. 
No call convention would be necessary.

In your case, if the C functions have the same convention as WIN32 APIs 
(as the code below says), *do* use it.

>extern "C" int WINAPI cibmerge(int argc, char *argv[]);
>extern "C" int WINAPI CibMergePFile(LPSTR szParamFile);
>extern "C" int WINAPI CibMergeVB(LPSTR strParameters);

"cibmerge" takes two parameters, whereas your COBOL example passes lots!
>               CALL WINAPI 'CIBMERGE' USING
>                               BY VALUE ANZ-PARAM
…[5 more quoted lines elided]…
>                            RETURNING RC
 
The second parameter of cibmerge is a by reference parameter to an array 
of pointers to strings, so you have to set up that array just as you did 
in Visual Basic. 
That is:
 01 argument pic x(256) occurs anz-param.
 01 argv pointer occurs anz-param.
..
 move "-L/CIBMERGE.LOG" & x"00" to argument(1)
 move "-I/BA.RTF" & x"00" to argument(2)
 etc.
 set argv(1) to address of argument(1)
 set argv(2) to address of argument(2)
 etc.
 call winapi "cibmerge" using by value anz-param 
        by reference argv(1) 
        returning rc

Haven't tried it, but I think this should do it.

Steve.
