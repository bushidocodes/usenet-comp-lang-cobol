Three cheers for NIU! They are absolutely correct, and we so few universities
who bother to actually check the marketplace these days.

Yes, there is a demand for Java and C/C++ programmers today. And that demand
will probably grow over the coming years.

But there is also a large market for COBOL programmers today and for the
foreseeable future. The reason is twofold. First, there is something like one
trillion lines of COBOL running production in the world today. Thousands of
companies have bet the business that programs written in this language will get
the job done, whether it be inventory control, order entry, billing,
accounting, even database analysis. These programs will need to be maintained,
enhanced, and even added to.

The second reason there is a future demand for COBOL is that it is a living
language. The coming standard embraces CORBA compliant object technology, for
example. At least two vendors I'm aware of already have the object features in
the compilers they have out there now (IBM and MicroFocus). You can create
classes, methods, and instances using COBOL. You can even create classes that
can then be referenced by C++ programs. Or, you can use COBOL to access C++
defined classes.

So COBOL is poised to use object technology when and where it is appropriate
for the task at hand. And it can still be used to create quality programs when
the task at hand doesn't need the faciliities of objects.

And for all the flak COBOL takes about being "verbose", it is easier to create
readable, maintainable code in COBOL than virtually any other programming
language out there today. Code that can be read and largely understood even by
non-programmers (this makes, say, auditorss, happy).

You won't go wrong by adding COBOL to your tool chest.

need to stay with AcuCOBOL.

differences, but making those changes in 2001 or whenever a new Standard
