Ignore all prior posts except Mr Cleveland's; they're wrong. The
CICS reference manual clearly states that:
1. Calls made via a call (rather than an EXEC XCTL or LINK) <MUST> be
dynamic.
2. Where a working-storage area is used to contain the called program's
name it <MUST> have a length of PIC X(08).
3. If the CALLed routine is to access any CICS resources than the first
two parameters in the CALL list must be DFHEIB and DFHCOMMAREA.
I've mixed CALLs, XCTL and LINK since the v3.x days -- no problems.
Do, however, check the the Programmer's Guide for the compiler options
that CICS programs require.

>Regarding dynamic links if you read the manually you will find that the
>DYNAM option is how dynamic calls are made and not whether they are
>made.  For example call 'fred' is static, call fred is a dynamic call.

No. DYNAM controls whether or not CALL 'FRED' is a static call or dynamic call.
If the compiler option is DYNAM, CALL 'FRED' will be a dynamic call and a load
module (VSE: FRED.PHASE) named FRED will be loaded at execution time. If the
compiler option is NODYNAM, CALL 'FRED' will be a static call and object module
FRED (VSE: FRED.OBJ) will be linked with the program at linkedit time.

CALL varname (where varname is a PIC X(8) field) will always be a dynamic call
and the load module / phase loaded at execute time will be the name contained
in that variable.
