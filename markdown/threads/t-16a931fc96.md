```
A while back someone asked to see a standard COBOL program and an OOCOBOL
program that solved the same problem.

Mike Murach's Structured COBOL book chapter 17 has an inquiry program,
Inq1000.cbl, written in MF COBOL using MF extensions for processing screen
I-O. I will post the listing as a reply to this posting.

The same book chapter 22 has an OOCOBOL solution to the same problem. It
consists of 5 classes: Bookinq.cbl, Bookmng.cbl, Bookfrm.cbl, Bookui.cbl,
and Bookinv.cbl. I will post them in another reply to this posting.

Quoting from th book page 740:

"After reviewing the oo program presented you may be wondering whether oo
programming is worth the trouble. The 4 page program presented in chapter
17 took nine pages of code when converted to OO code. Keep in mind that if
you design your classes with resuability in mind, you should be able to use
the same classes and methods in other applications.

Also keep in mind that oo COBOL in not an all or nothing proposition. You
can use it when it makes sense and use standard COBOL when it does not. 
.....

Two situations where oo programming almost always makes sense were
illustrated. First a file manager class could be used to perform file i/o
for any file. .....

Another situation is illustrated by the book inventory class. A class like
this can hold the data for a single record from any file.  Then its methods
can perform any required operations on that data, ......"

I hope this gives a sample flavor of the differences.  Of course the
explanations in the book would be of great help but typing in 2 whole
chapters is too much for me at this time. You could always get the book
from the library if you don't want to buy it.
```

           PERFORM 130-ACCEPT-ITEM-NUMBER.

           IF NOT VALID-ENTRY

           ELSE
               PERFORM 170-DISPLAY-BLANK-LINE.

                       TO ERROR-MESSAGE.

               INVALID KEY

       160-DISPLAY-ERROR-MESSAGE.

           DISPLAY ERROR-MESSAGE AT LINE 23 COLUMN 1
               WITH FOREGROUND-COLOR BRIGHT-WHITE BEEP.

       170-DISPLAY-BLANK-LINE.

           DISPLAY SPACES AT LINE 23 COLUMN 1.

       180-DISPLAY-ITEM-DATA.

           MOVE IM-UNIT-COST TO UNIT-COST.
           MOVE IM-UNIT-PRICE TO UNIT-PRICE.
           MOVE IM-REORDER-POINT TO REORDER-POINT.
           MOVE IM-ON-HAND TO ON-HAND.

           DISPLAY IM-ITEM-DESC     LINE 7  COLUMN 22.

           DISPLAY UNIT-COST        LINE 8  COLUMN 22.
           DISPLAY "Unit price:"    LINE 9  COLUMN 1.
           DISPLAY UNIT-PRICE       LINE 9  COLUMN 22.
           DISPLAY "Reorder point:" LINE 10 COLUMN 1.
           DISPLAY REORDER-POINT    LINE 10 COLUMN 22.
           DISPLAY "On hand:"       LINE 11 COLUMN 1.
           DISPLAY ON-HAND          LINE 11 COLUMN 22.
           DISPLAY "On order:"      LINE 12 COLUMN 1.
           DISPLAY ON-ORDER         LINE 12 COLUMN 22.

       190-GET-NEXT-ITEM.

           MOVE "Y" TO VALID-ENTRY-SWITCH.
           PERFORM 200-ACCEPT-NEXT-ITEM.
           PERFORM 210-EDIT-NEXT-ITEM.

               PERFORM 160-DISPLAY-ERROR-MESSAGE

               PERFORM 170-DISPLAY-BLANK-LINE.

> What is the deal with three digit numbers prefixing paragraph names?

Enumeration facilitates the grouping of functionally similar paragraphs,
giving a structured nature and overall cohesiveness to the program.

I use an alphanumeric prefix scheme, A-0000- thru Z-9999.
This allows me to easily group similar paragraphs, and also helps me 
to quickly determine if I have already coded a specific piece of performed code,
and also helps me, or anyone else, to quickly locate the particular section
which contains this logic, and, of equal if not more importance, all similar logic.

I can use a specific letter to preface all of my database call logic,
another for checkpoint/restart, another for exception processing, etc.
I can restrict inline logic to paragraphs A through whatever, 
and use the remaining characters for common performed routines,
such as database calls, exception processing, etc.

Coupled with consistent paragraph-naming practices, this makes it much
easier for me to control the construction of a program.  I rarely have to wonder 
what Z-0250-ORDERSLI-MISS is supposed to do, or Z-0680-CREDITCK-FAIL, 
or where in the program they should be/are located.

> Wouldn't they mean the same without the prefixes?

It's a matter of choice.  I could have multiple associated paragraphs in my program, 
A-1260-CREDITCK-FAIL, B-1320-CREDITCK-FAIL, C-1140-CREDITCK-FAIL etc,
and Z-0680-CREDITCK-FAIL, and instantly recognize that the A-1260/B-1320/C-11140 routines
are in-line processing paragraphs, each executed for one of several input transaction types, 
while the associated Z-0680 routine is my common exception processing for all transaction types,
performed from each of the aforementioned inline paragraphs.

It's a fair question.

The 3 digits are useful for a human to locate a PERFORMed paragraph. They
generally follow the "flow" of the program and you can be reasonably sure
that  400-whatever will appear in the source code AFTER 200-something. Some
people extend this to encompass all sorts of extra stuff (like a special
"group" for "mainline" paragraphs/sections, another "group" for
"subroutines" that are activated from "above", and so on.)

The main reason for using them is that if you are called in on the late
shift (or have given birth 20 minutes previously <G>) you don't have to
think too hard about where a given paragraph/section is located in source.

Despite JerryMouse's laconic assertion to the contrary, the fact is that
these prefixes make no difference to the actual "meaning" of the name, as
interpreted by the computer.

Bookinq.cbl:

       Identification Division.

       Program-id. Bookinq.

       Environment Division.

       Object Section.

       Class-Control.
          BookMng is class "bookmng".

       Data Division.

       Working-Storage Section.

       01  handles.
           05  bookMngObj       object reference.

       Procedure Division.

           Invoke BookMng "New" returning bookMngObj.
           Invoke bookMngObj "process-inquiries".
           Stop Run.

Bookmng.cbl:

       Identification Division.
       Class-id.  BookMng
                  inherits from Base.

       Environment Division.
       Object Section.
       Class-Control.
           BookMng     is class "bookmng"
           BookUI      is class "bookui"
           BookFM      is class "bookfm"
           BookInv     is class "bookinv"
           Base        is class "Base".

       Object.
           Data Division.
           Working-Storage Section.

           01  handles          object reference.
               05  bookUIObj.
               05  bookFMObj.
               05  bookInvObj.

           01  switches.
               05  end-of-inquiries-switch     pic x(01)  value "N".
                   88  end-of-inquiries                   value "Y".

           01  error-message                   pic x(50).

           01  book-data.
               05  bk-item-number              pic x(05).
               05  bk-descriptive-data.
                   10  bk-item-desc            pic x(40).
                   10  bk-unit-cost            pic 9(03)v99.
                   10  bk-unit-price           pic 9(03)v99.
               05  bk-inventory-data.
                   10  bk-reorder-point        pic s9(05).
                   10  bk-on-hand              pic s9(05).
                   10  bk-on-order             pic s9(05).

           Procedure Division.
      **********************************************************
           Method-ID. "process-inquiries".
           Procedure Division.

               invoke BookUI "New" returning bookUIObj
               invoke BookFM "New" returning bookFMObj

               invoke bookFMObj "open-file"
               invoke bookUIObj "display-heading"

               perform until end-of-inquiries
                  invoke bookUIObj "get-item-number"
                      returning bk-item-number
                  if bk-item-number = "99999"
                      set end-of-inquiries to true
                  else
                      invoke bookFMObj "create-book"
                          using bk-item-number
                          returning bookInvObj
                      if bookInvObj = null
                          move "Inventory record not found."
                              to error-message
                          invoke bookUIObj "display-error-message"
                              using error-message
                      else
                          invoke bookUIObj "display-blank-line"
                          invoke bookInvObj "get-desc-info"
                              returning bk-descriptive-data
                          invoke bookUIObj "display-desc-info"
                              using bk-descriptive-data
                          invoke bookInvObj "get-inv-info"
                              returning bk-inventory-data
                          invoke bookUIObj "display-inv-info"
                              using bk-inventory-data
                          invoke bookUIObj "get-next-item"
                              returning end-of-inquiries-switch
                          if not end-of-inquiries
                              invoke bookUIObj "display-heading"
                          end-if
                      end-if
                  end-if
               end-perform

               invoke bookFMObj "close-file"

           End Method "process-inquiries".

      ***********************************************************
       End Object.
       End Class BookMng.

Many years ago (back in the days when I still used MicroFocus COBOL (Version
3, I think...)), I wrote a system that had many screens. I used a separate
module for each screen and the interface was via a buffer, as described
above. Each screen module was specific to that particular screen and
contained all the fields, defined in a Screen Section. The interface buffer
was exactly the same as an IMS/DC buffer with a two byte attribute preceding
each field and the same values used for these attributes as were used on the
mainframe. So you could set an attribute to NUMERIC-HIGH-BRIGHTNESS (for
example...) and the field would appear on the screen intensified with
numeric data. You could also insert the cursor through the attributes and
you could set colours. All of the code to do this was provided by a single
skeleton of the called module. The "screen specific" parts were generated by
a program I wrote called ACEPAINT. This let you "draw" the screen as you
wanted it to look, with coded symbols for ""start fixed" and "start
variable". When you were happy with it you generated it.

The generation phase generated a Screen section, an IMS/DC buffer (like a
combined MID/MOD, but for use on the PC), complete with attributes, a CICS
BMS buffer with attributes (this was really just as a Sales point and
because it was easy to do it...we only had a very small investment in CICS
and the application this was done for did not use it...), and the necessary
DIF/DOF and MID/MOD code to run the screen under IMS/DC on a mainframe. (My
plan was to use the PC to build and debug mainframe code, at a time when
this was still a very novel idea...<G>).  The generated bits were then
merged with the skeleton code and the resulting module compiled. (It could
then be CALLed from the application when required).I had a single process
that could do the whole lot from painting the screen right through to the
compiled screen support module. You could design a screen and generate
support for it in minutes... I thought it was pretty cool...(So did the
people I was working for..)

It all worked beautifully and it was kind of spooky to see a PC sitting
alongside a Mainframe 3270 with the two of them showing identical screen
formats, as the same COBOL application code ran on both machines. (In those
days they were not connected, and were each running the application as a
standalone app. using their own files (In one case, VSAM/KSDS and the other
ISAM...<G>) The only difference was that the mainframe didn't call the
screen support module, it used IMS/DC and DIF/DOF/MID/MOD instead, but the
application code and the interface to the screens was identical.

Today (just like Screen Section in COBOL) it is obsolete.

Hopefully above clarifies it a little. Now to your 'Why more than one object

I would have to carefully think out my DBI approach versus the BookRecord
class, but you could have :-

Object-storage.
01 os-BookDBI occurs 10 object reference value null.

move 6 to n

- obviously you would have to carefully track which "n" object you were
currently accessing. That's why the Object Properties approach, with a
separate object per book  is probably safer.

Another aspect of multi-objects. No good using books - let's take ladies'
fashions.
Spring Styles identified to a product group, sub-group etc. The central
buyer/merchandiser wants to view the performance of similar styles before
re-ordering  I would tend to give him a listbox of records to choose from -
hence my reference to collections above. He's curious how Style A from
Manufacturer X is going compared to Style B from manufacturer Z - bearing in
mind seasonal merchandise has a very short life. The DressRecord (equivalent
of the BookInv record) may contain a picture of the Style to remind him of
what it looks like, and summarized at corporate level, quantity, colours,
sizes. Additionally what is the performance at each retail branch. Now I'm not
suggesting this is ALL stored in the DressRecord but you can also have
sub-classes of DressRecord for different properties.

So he selects Styles A and B and C from the Listbox creating separate
DressRecord objects. Now he/she can hop, skip and jump between the various
style objects to make buying decisions.
