Certainly.  ACCEPT data-name FROM DATE is the old format and returns YYMMDD.
Some compilers can have that statement return YYYYMMDD as an option or the
syntax ACCEPT data-name FROM CENTURY-DATE might work.  If neither of these
work with your compiler, a subprogram or copied code can look at your two
digit year and insert 19 or 20 accordingly.  HTH

Yes it can.

MOVE FUNCTION current-date TO <receiving field>

The format of the receiving field is  ccyymmddhhsshh+oo where:

yyyy - four digit year
mm - month
dd - day
hh - hour past midnight
mm - minutes past midnight
ss - seconds past the minute
hh - hundredths of second 
+  = + or - for offset from GMT
oo - gmt time offset hours

My explanation is a little sketchy and the GMT offset stuff is a little
more involved, nevertheless, the intrinsic function "current date" is
well documented (and simple).

Hi Andy,

Of course, you can make a system call to OS to get current date and time. You
have to contact the Cobol supply if you are not sure how to do it.

I think the key of Y2K is not how to get a date, but is how to store a date
and how to operate on it.

I am using a VOS Cobol, it is running on the VOS which is an OS like UNIX. In
my Y2K solution, I use a comp-5(integer) variable to hold the julian day
number of a date. I create a subroutin to transform any input forms to integer
and transform integer to any output forms.
