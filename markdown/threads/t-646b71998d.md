> Can you give me an example of orthogonality in cobol, please?

A component, written in COBOL, which has a stable interface. There is an 
example on the new Cobdata web site which will be published shortly. (The 
site is almost ready but my ISP decided to upgrade to Server 2008 and the 
result is custard... it should be sorted within a week.)

> Or cobol
> isnt orthogonal?

Procedural COBOL CAN be orthogonal (modular programming is a step in the 
right direction, but it doesn't guarantee orthogonality), although in 
practice (certainly from the procedural code I've seen) it usually isn't. 
This leads to much wailing and gnashing of teeth on many COBOL sites when a 
few lines of code get changed, and something in a subsystem far away falls 
apart.

Object Oriented COBOL, when written using component based techniques, is 
DEFINITELY orthogonal. Like other OO languages the concept of 
"Encapsulation" ensures that it is.

>
> I think orthogonality is a complex concept...

Like most complex things, they can be conceptualized in ways that make them 
less complex. Imagine an isolation ward in a hospital. All access to the 
patients is carefully controlled and the staff cannot remove items from the 
ward and use them elsewhere.

The mathematical, and lower level computer language meanings of 
orthogonality are a bit more confusing (until you actually study them (then 
they're not)) but the above would serve as a general example.
>
> Thanks.
HTH,

Pete.

I'm talking more about how free are you to combine sets of different
things?

Look at DD's example. He says "Any instruction can use any register in
any addressing mode." For some instruction sets that is probably true.
On the 8088 you can not use AX as an index register. So the 8088's
instruction set is not orthogonal in that respect.

Likewise there is an interaction between a file's organization and how
it can be accessed. Perhaps this is tautological? However there is
SOME freedom to handle "random access" files sequentially as well. So
you have SOME flexibility.

But COBOL is not completely rigid. Look at MOVE. Move is quite
flexible. It's actually polymorphic. Arithmetic operations also work
with a variety of data types. Constrast that with languages in which
you have to invoke a method on an object to get it to convert itself
to a different data type.....

Then there are languages like "C" in which you can combine almost
anything with anything else. It will almost certainly compile, it
might even do something that you want, or it might start WWIII. In
that sense "C" is more orthogonal. You have more expressive power with
which you can shoot yourself in the foot!

I hope this makes SOME sense. [It did when I posted it. :-)]

If you want to combine COMPONENTS almost at will, as you describe,
then they are orthogonal too!
