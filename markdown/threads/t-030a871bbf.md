```
For y'alls enjoyment, I'll relate this little story. :) 

I'm working on a "block mode" terminal emulator project right now and quickly coded it for proof of concept using COBOL. (Well, once you guys pointed out a sample of how to use sockets from COBOL! Thanks again...! :) 

Well, it was determined that a very small part of this project needed to run inside a browser. Okay, this is a simple talk un Java, right? Just feed a Java program a byte stream, and give it some simple behavior rules, and voila! I should have a simple, easy, display coming up inside a browser. 

Hah! 

First off, Java Strings are not strings of bytes, they are "Unicode" strings. Just *try* and find what a Hex value of say, x'084' is in Unicode!  Testing turned out to be very difficult, since it is about as impossible as it can be to insert a HEX value into a string in Java. 

Ahah, says I - I will simply use an array of Bytes, since they have those things in Java. Since it is an 
array, I can look for or insert values I want in the array and then use the toString() thing to display it. 
Turns out bytes (or Bytes, which are *different* things!) are ALWAYS signed, which means, yep - you guessed it. You have to do signed conversions back and both to figure out what they heck you are looking for. (Negative values.) 

Well, after working through that nonsense, I ran into displaying the data on screen in a reasonable manner. There are (I counted them! ) over 110 (!) different classes that can or do have some effect on just opening a Window and displaying some text. The coding turns out to be simple, but figuring out WHAT to code is worse even than doing it in Smalltalk. (Until now, SmallTalk held my top place for the most capable language no-one can figure out how to operate... :) 

The long and the short of if all this is, even though I do have 3 years of working with Java, in Servlets and integrating stuff to Mainframes and under AIX/Unix, something that is a perfectly simple straightforward task in a procedural language is far far harder to do in Java. With all those objects, you wouldn't think so would you? 

To top it all off, I posted some questions in comp.lang.java.programmers and comp.lang.java.gui asking about particular points. In each group I got one answer of roughly the same quality *this* group puts out as a matter of course. The rest were, at least in some level, insulting and pretty much non-informative. (You should have seen the ones that came directly to my mailbox!) 

God save COBOL, because there certainly is a vast gulf between the expertise and professionalism of people here and people in most other groups. 
```

```
-Paul
```
```

A past experience of object-oriented design that has assisted,
though not really "solved," the task of report writing may be
in order.

I worked on an application that was new development in the
500-700 k$ range, the was back-ended by an RDBMS.  There were
perhaps 150-200 principal tables, and a similar number of
windows/screens.  I wrote most of the reporting logic, though
the layouts for those reports were done by an associate.

Purely out of laziness on my part, I wrote a set of generic
reporting objects that were more-or-less SQL knock-offs. They
were taught rules about set operation, data validation, and how
to collect schema from the database, as well as the syntactic
basics.  Even though these objects would start with database
schema, I didn't go so far as to extend all of this to the
user's interface.

Instead, it served as a platform form me to code details in a
shorthand fashion.  I could for instance say, for reports on
Customers:

1)  these parameters must be provided to generate this report
2)  these parameters may be provided
3)  here's the list of columns that can be included
4)  here's the list of columns that must be included
5)  here's the list of columns that can be sorted upon
6)  here's the list of columns that can be grouped upon
7)  here's a layout template to use
8)  here's the list of available reports
9)  these columns should be totalled, averaged, min or maxed
10)  this can only be run by Executive or Accounting users
(this was a service industry application)
11) ...

This and more could be detailed for a report (or more
accurately a group of reports considering the parameters to
it) -- all in about the amount of code as text I've used to
describe it.  The user would get some options, but within
constraints imposed by the report's definition.

There are two big benefits to this "wrapper" approach:

1)  Controlled parameterization of the reports allows users
(perhaps selectively) to change format of the report without
having to know the database design.

2)  Encapsulation kept generic byte pushing seperate from
business logic.  99% of post-release development was to the
business logic -- which was concise and easy to maintain.  If,
however, a general enhancement was needed -- say going one to
multiple levels of grouping -- the generic logic could be
extended to the potential benefit of any of the specific report
objects


In the last month of the project, we scored major contract
extensions by being able to add:

1)  Report persistence -- users can save the details of a
query, only the keys, or (of course) the report's data.  (This
was not a decision left up to the user, but rather a
distinction made by the application.)

2)  Report memos -- a user with more flexible reporting
priveleges could "design" and send a report to someone else to
say "take a look at these figures."  Or, if they're convinced
of their genius (and have rights to do so,) they could make it
available to the application.

3)  Report batching -- reports may be generated on a schedule.
If a user goes after the report, the system can either provide
the last generated report or re-run it.  (A little bit of
report annotation was put in as well - a notes field was added
to the persisted report data.)

4)  Report layout editor -- not really a feature of the report
objects, but back-ended by them.  I was not really a fan of
this feature, but editor was fun to write.
```
