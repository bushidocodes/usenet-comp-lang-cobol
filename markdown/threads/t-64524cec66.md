COBOL supports 'Indexed Files'.  The term 'indexed-sequential' refers
to a specific method of organizing index tables internally, which is
incidental and completely dependent on the compiler vendor.  Some
compilers generate code that store Indexed Files using indexed-sequential,
but some use b-tree or numerous other methods.  On computer systems that
support built-in database capability, their compilers often use the
built-in database support to store Indexed Files.  Some compilers permit
the use of third party database software (e.g. Btrieve) to store Indexed
Files.

You probably want LINE-PRODUCT OCCURS 10 TIMES.

This means there are 10 items called LINE-PRODUCT, numbered
1 thru 10.  Accessing them requires the use of a "sub-script".
Example:  

    MOVE WS-LINE-PRODUCT TO LINE-PRODUCT (1).

However if you are using some variable for 1 thru 10 to indicate
which LINE-PRODUCT it is like a counter or some numeric value
in a field you can use a variable instead.
Example: 

    MOVE WS-LINE-PRODUCT TO LINE-PRODUCT (SUB).

This works best if you make a structure exacly like LINE-PRODUCT
and call it WS-LINE-PRODUCT without the occurance so that when you
accept the values it goes into a structure the same size as
your LINE-PRODUCT.  Then you move the group as a group move instead
of moving each individual field. It is a common practice to use the
prefix "WS" for temporary (Working-Storage) storage fields.
EXAMPLE:

>fd product.
>    	01 line-product occurs 10 times.
>    	    	05 product-id    	pic 9(3).
>    	    	05 product-price    	pic 99V99.
>    	    	05 product-name    	pic x(12).

        01 ws-counters.
                05 sub                  pic 9(02) value 1.

   	01 ws-line-product.
    	    	05 ws-product-id    	pic 9(3).
    	    	05 ws-product-price    	pic 99V99.
    	    	05 ws-product-name    	pic x(12).

accept ws-product-id.
accept ws-product-price.
accept ws-product-name.
MOVE WS-LINE-PRODUCT TO LINE-PRODUCT (sub).   (this is a group move)
compute sub = (sub + 1).

Something like that.

A Sub-Script must be numeric.
A Sub-Script must be larger than 1.
A Sub-Script must not be larger than the occurs clause.
Otherwise you will receive a sub-script error.

This is just one way to do it.  It might work fine using a subscript
in the accept line, but I have never tried that.

Can't speak to Merant, but can speak to the old Unisys A Series COBOL(68)
(only recently retired) which allowed OCCURS at Level 01 ...

The purpose here was to specify a "true" two-dimensional array  -- an array
of descriptors (associated with the first subscript) pointing to individual
memory spaces, instead of a single descriptor pointing to a single memory
space as for "normal" 01-level records.

Everything was pretty much OK if the descriptors pointed to arrays that
themselves had OCCURS clauses, but results got very strange when the only
OCCURS in the record was at the 01 level, particularly when there was no
subordinate elementary item with a PICTURE clause.

Ultimately (in 1989), the COBOL(68) compiler was modified to issue a syntax
error if a two-dimensional array was declared with fewer than two dimensions
...

Subsequent COBOL compilers for the A Series (COBOL74, COBOL85) eschewed this
particular extension as not providing enough benefit to outweigh the risks,
pitfalls and "gotchas" inherent in the design.

Best I can tell the standards have consistently disallowed OCCURS at Level
01.

        -Chuck Stevens
