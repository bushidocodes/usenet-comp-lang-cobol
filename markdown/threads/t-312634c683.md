```
I thought I had figured out a way to do this about 10 years ago; however, I've
forgotten how I did it; Maybe I did it in (shudder) assembler; Here's the problem:
I'm trying to split up some big tape files into a series of smaller chunks, based on
the number of records in each big file. I need to save the beginning and ending keys
of each smaller file as an index to each.

Here's what I thought I could do:
When you execute a cobol read, logically you get one record, the length you told the
compiler must match the length of the record on the file label. Physically, more
records get read into the buffer than what you know about, for more efficient
sequential record processing. What I want to do is override the compiler: read an
entire buffer's worth of data into my program at one time, into a table data
structure, and from there do some parsing etc. on chunks of the data. Then (ideally)
I would write out the buffer's worth of the data, retaining on the label the correct
length of the logical record.

So far, I can't remember how I did this, and I darned sure can't get it to work for
me. This is what happens when you start messin' with newfangled technology like
relational databases - you forget how to do things the hard way!

Any suggestions would be appreciated.
Environment = IBM 3090 ESA, MVS OS, COBOL II
```

```
You will probably find this technique will not work in native VS COBOL II.

It did work (to some extent) in previous COBOL's by overriding RECFM, and
then perhaps fishing in control blocks for the real physical block length.

VS COBOL II does extensive checking of record length and record format
across program, file label and JCL, and will introduce you to file status
39 if there are mismatches.

Yes, you could write assembler I/O to do this, but why bother?

Other factors such as system determined blocksize and system managed
storage effects ( reblocking ) could cause "interesting results". Not to
mention on the output side - how to make labels on the file correctly
reflect what you want the file to look like.
```
