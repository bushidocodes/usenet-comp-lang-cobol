```
This is a generic reply to all the people who've tried to help me.  I've
tried all of the following SELECT statements for SELECT INPUT-FILE-1:
         ASSIGN TO SYS005-UR-2540R-S-TFIL.
         ASSIGN TO SYS005-S-TFIL.
         ASSIGN TO SYS005-TFIL.
         ASSIGN TO SYS005.

Again, recall that I have // ASSGN SYS005,SYSIPT in my JCL.  My goal is
to read the RDR cards after the JCL statement.

The following statements are true:

The first assignment works when the program is compiled with DOS/VS
COBOL and run with the COBOL II runtime library.

The first assignment works when the program is compiled with DOS/VS
COBOL and run with the LE/VSE runtime library.

All assignments work when the program is compiled with VS COBOL II and
run with the COBOL II runtime library.

*None* of the assigments work when the program is compiled with VS COBOL
II and run with the LE/VSE runtime library.

I haven't tried one person's suggestion of "SYS005-S-SYS005", but I have
no reason to believe this will work either.

I *can* do "ACCEPT DATA-REC" instead of treating it as a file, and it
works with no problem.  The problem with that is that we have dozens of
programs out there using it as a sequential file.  No way we want to
make changes to all those programs.

I do have a couple other observations:

With the COBOL II run time if I use the following SELECT assignment:
ASSIGN TO SYS005
and the following JCL
// ASSGN SYS005,SYSIPT
// DLBL SYS005,'FJS.TEST',,VSAM

It will read from file FJS.TEST.

If I remove the DLBL it will read the cards from the RDR.  This is why I
don't think SYS005-S-SYS005 will make any difference.

It's also interesting to note that if I replace the DLBL above with
// DLBL SYS005

I will get the same error (90) with both runtime libraries, and it's the
same error I'm getting with no DLBL and the LE/VSE runtime.  It seems to
me that the LE runtime library is interpretting them in the same
manner...a DLBL to a non-existant file (or something like that).

Anyway...one of my systems programmers is going to check IBMLINK.  From
there, who knows...

Thanks all.
```
