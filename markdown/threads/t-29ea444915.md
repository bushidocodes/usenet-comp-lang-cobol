```
1. INSPECT field-name TALLYING byte-counter

STRING

  '"'               <== that's a double-quote inside a single-quote

   '"'

2. COMPUTE byte-counter = FUNCTION STORED-CHAR-LENGTH(field-name)
```

```
For each character field move it to a working-storage area and replace
all trailing spaces by nulls (x"00").  You could do this using FUNCTION
REVERSE and INSPECT or by writing a routine that does:

              PERFORM
                  VARYING I FROM Max-Length BY -1
                    UNTIL I < 1 
                       OR Work-Char(I) NOT = SPACE
      
                  MOVE x"00"       TO Work-Char(I)
              END-PERFORM

Each field could be moved to Work-Text (made up of Work-Char OCCURS 100)
and its length to Max-Length then perform this and move it back.

Then output the CSV with:

              STRING '"'     DELIMITED BY SIZE
                     Field-1 DELIMITED BY x"00"
                     '","'   DELIMITED BY SIZE
                     Field-2 DELIMITED BY x"00"
                     .... etc

Putting quotes around the test fields ensures that commas within the
field aren't a problem.

Numeric fields shouldn't be enclosed in quotes and can be moved to an
edited numeric --------9.99, left justified then null terminated for
stringing.
```

```
       SELECT WARRANTYCSV
	 ASSIGN TO "C:\WARRANTY\EXPORTCSV.TXT"
	 ACCESS MODE IS SEQUENTIAL
	 ORGANIZATION IS LINE SEQUENTIAL.


 DATA DIVISION.
 FILE SECTION.

 FD    WARRANTY.
```

```
Just so you know in advance, "sliding ODO's" (where there are either nested
OCCURS DEPENDING ON - or fixed data FOLLOWING an ODO):

 - is not in the ANSI COBOL Standard, therefore
 - not every compiler supports it, and furthermore,
 - those that allow it at compile-time do NOT all treat it the same at
run-time

This doesn't (necessarily) mean you shouldn't use them - but you should be
aware of the "portability" problems in doing so.
```
