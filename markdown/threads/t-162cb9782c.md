July  24, 1999

As I have been "advertising," I am working on creating a "Convert COBOL
datatypes to IEEE datatypes" function I could put into a standard 32-bit
Windows DLL to be called from any language which supports them. (Freeware
for non-commercial use).

Yes, I realize doing the documentation before doing the code is a strange
concept, but it works for me.

After initial comments and some really sophisticated design ideas which I
pitched because I felt them too difficult to deploy, I think this will be
best. (PS thanks to Cheesle for the starting point).

The proposed DLL will provide two distinct functions:

1. The PARSE function.  This will take a series of literals and return
information needed in the conversion phase. Programmers already familiar
with COBOL datatypes, or if there are special alignment or SYNC
considerations, may elect to skip this phase and just table up the data for
the calls to the conversion phase.

The user supplies literals as shown in the ParseParmsType structure below,
and is returned a Status code, ErrorMessage if applicable, and a
DataInfoType structure. The information in the DataInfoType may be used by
the programmer to determine the location (offset) and size of the data
buffer to retrieve from the input and pass to the conversion function.

2. The Convert Function.

This will actually convert the data to a numeric value, or simply effect a
character set conversion from EBCDIC to ASCII. The user supplies a
DataInfoType structure, an input buffer and an EBCDIC flag.
The function returns decimal mantissa and exponent for numeric data types,
or the output text buffer if text (COBOL picture X(nn) or A(nn)).

The function will perform EBCDIC to ASCII conversions if the flag is set and
the datatype qualifies for conversion (e.g., will not convert binary or BCD
numerics).

"Plain old" EBCDIC-ASCII conversions can be effected by passing data as
character type, desired length and setting the EBCDIC flag.

The user is responsible for casting the numeric values returned into the
desired output and must handle the multiply or divide by ten "exponent"
times.

One last thought I had (see comments below re usage from a COBOL program):
might it be better to pass and return numeric parameters in character
(DISPLAY) format? BASIC users could get numeric values using VAL, C/C++
users could get values using scanf, COBOL, of course, could just use USAGE
DISPLAY to read the values. (Just a thought)

Comments?

Michael Mattias
Tal Systems

NOTES:
- ASCIIZ is a null-terminated string
- STRING * whatever is a fixed-size buffer. (Well, so is ASCIIZ, but we
can't very well use ASCIIZ if there is binary data, now, can we?)
- The BASIC "TYPE" is a C/C++ "struct" or COBOL "group item."
- Input and output buffers are only valid up to BufferLength characters.
- If calling from a COBOL program, the user is responsible for supplying
any required numeric parameters in IEEE format. (Although why you would call
this from a COBOL program eludes me at the moment).
- I am not planning on supporting "Edited Numeric" datatypes.

TYPE ParseParmsType
  Picture      ASCII Z   * 32     '"S9(04)V9(2)"  "X(15)", 999999V9999"
  Usage        AS ASCIIZ * 256   ' space/null (display), "DISPLAY"
                                 ' "COMP" "COMP-3","COMP-5" etc.
  SignIs       AS ASCIIZ * 256  '"LEADING" "SEPARATE" or both;
' space/null to use default
END TYPE

TYPE DataInfoType
   DataSize     LONG    ' may be used to figure out how much of the
                        ' input data to retrieve and pass to conversion
   DataType     LONG
   Digits       LONG
   Decimals     LONG
   IsSigned     LONG   'yes/no 0 = no, else yes
   SignLeading  LONG   'yes/no
   SignSeparate LONG   'Yes/No
END TYPE

TYPE ConvertParmsType
  EBCDIC       AS   LONG     ' yes/no
  BufferLength AS   LONG
  BufferIn     AS   STRING * 1024
  BufferOut    AS   STRING * 1024
  Mantissa     AS   LONG
  Exponent     AS   LONG
END TYPE


TYPE ParmsType
  Action       AS LONG            ' 1 = parse, 2 = convert
  DataInfo     AS DataInfoType    ' supply for convert; returned on
  Convert      AS ConvertParmsType
  Status       AS LONG             ' 0 = OK, not 0 = error
  ErrorMessage AS ASCIIZ * 256     ' when non-zero status
END TYPE


Error Examples:

Invalid input data (e.g., x'0A' found in DISPLAY NUMERIC field)
Overflow/Underflow
One or more characters of EBCDIC input buffer do not have ASCII equivalent
Unsupported Picture
Unsupported Function

** ENDS ***

The embedded binary 0 was my specific concern when I wrote this. You will
note that all the non-ASCIIZ buffers have a separate, explicit field to say
how much of the data are valid. (Not unlike an "occurs depending on"
clause).

As far as reading the file, you would probably do something like build a
table containing the data picture, usage and sign info with space for the
other fields returned by the parse function (size, sign flags, etc).

Call the "parse" function of the DLL for each field; save the returned data
sizes, types, etc  in your table.

Now to convert:

FOR each record
   READ Input-Buffer
   FOR each field
      Calculate offset in record (using accumulated sizes of previous
fields)
      Move portion of buffer for this field, and the data statistics
gathered during the parse to the   Datainfo portion of the parameter block
     call the dll
     convert numeric output to desired data type, build/write output buffer.

COBOL: Reference Modification is the way you could move the data to this
buffer:

01  THE-RECORD             PIC X(1000).

MOVE  Field-Length  Parms-BufferLen
MOVE  The-Record (startpos:Field-length) to Buffer-In
CALL "the dll" using Parms-Block.

In BASIC, you use the MID$ function (a lot like reference modifier in
COBOL).

In C, I think you use a character array (table) and do a memory copy from
the address of the source element to the address of the destination.

I guess this means I should write a sample conversion program, one in BASIC,
one in COBOL, and find someone to write one in "C".

FWIW, I have added another parameter: alignment. You will be able to specify
byte, word or dword alignment when parsing and I'll use that in the size and
value calculation later. (Also to mask off any slack bytes resulting from
ALIGN or SYNC).

I have the parse part working except for some extra editing I want to do,
and the data conversion is a snap, so I may actually be looking for
volunteer testers next week.
