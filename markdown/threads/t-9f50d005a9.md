### A case for multiple inheritence.

I posted a message indicating I would provide an example of why I like
Multiple Inheritence in COBOL.  Others have questioned it's utility,
and the language du jour doesn't support it.

While it can be abused, making a morass of inheriting classes, it can
also be a boon.  Let's talk theory and I'll try to throw in the
example.

Consider Airplanes.  An Airplane class might grant inheritence to
several subclasses.  The different types of airplanes might inherit
the general characteristics of the Airplane class.  For example:

Airplane
  - Bomber

Airplane
  - Transport

Airplane
  - Fighter

Now, each of these aircraft might be of a different type.

Airplane 
  - Bomber
    - Stealth
      - Stealth Fighter

Airplane
  - Bomber
    - Stealth
      - Stealth Bomber

So far so good.

Now, let's consider the ordinance.  Each of these has different
requirements.
Let's say that we want to specify the bombs.  We could do this:

Airplane -
  - Bomber (Method BombType)
      - Stealth
        - Stealth Fighter


But.. what about fighters?  F16's are fighter bombers and they might
use the same bombs as stealth fighters.  Associating bombs with
bombers won't allow us to have an F16 that inherits from the Fighter
class and still get the BombType method.

One solution, using single inheritence is to place the BombType under
the Aircraft class and then override them in the indivudual subclasses
so that they don't have bombs.  This is silly, and extra overhead for
all the inheriting subclasses that don't have bombs.  It also results
in excessiving "overrides".

Another is just to code the BombType method over and over at the
lowest level where it is necessary - adding extra methods to
subclasses.  This is wasteful and results in excessive maintenance
should something about bombs change.

Instead, we could put the BombType method in a separate class that
handled multiple types of bombs.

Then you simply inherit where necessary and avold all the extra
overhead where it is not necessary.

IE:

Airplane
  - Bomber, - Bombs
    - Stealth
      - Stealth Fighter

Airplane
  - Transport
    - C-130

Airplane 
  - Fighter
    - Fighter-Bomber - Bombs
      - F16

See?  This is what Multiple Inheritence is good for!
```

Before I comment, I'll come clean as to my aesthetic appreciation of
multiple inheritance.

I'm greedy enough to want it.  Even if I don't use it very often, it does
not complicate the language if you take a single inheritance approach.  I'll
think about it some more, but I can't think of an instance where you gain
anything by knowing that an object cannot inherit from more than one base.
(A single inheritance language compiler can probably do it's job faster than
one that support multiple inheritance, but I know of no examples we could
time to determine this.  Object Pascal compilers beat the pants off of C++
compilers, but C++ has some ambiguities that would likely make this
so--without the complexities of multiple inheritance.)

That said, I can't make a forceful argument against being able to multiply
inherit.

An example of multiple inheritance that I've used before is thus:

Consider a business object.  It has a series of scalar attributes or
properties that define the object.  (It's a lot like a single row of a
recordset if you're familiar with that pattern.)  Values may abstractly be
retrieved from and put to the object, with that object's implementation
being in charge of validation and other computational rules.  It's basically
a "smart array" of named values.

Another object, a business list, is more like the recordset mentioned
parenthetically above.  It has named attributes, but also rows of them.
(Other patterns by name: a grid, spreadsheet, etc.)  In addition, it has
aggregate functions that operate to sum or average a column of values.

Each of these objects have fundamental applications in data management and
may be used in their own right.  Consider a more specific example, though, a
Bill object.  (I mean a financial statement, not William Klein.)

The Bill certainly has a use for object-level attributes: customer, bill
date, disposition, etc.  Contained within that bill is a list of
charges--much like the business list.  In this scenario, you could rapidly
construct a Bill object by inheriting from the business object AND the
business list.

You could certainly construct a base class that implements both of these
schemes, but using multiple inheritance offers quick extensibility.  Perhaps
you need to maintain a list of payors for the bill--inherit another business
list into a new namespace and you're ready to mash.
```

I won't swear to it, but I think that the "classic" example of multiple
inheritance that is often used is a bank "environment".

Let's say that there is a type of object that is a CD (certificate of
Deposit) account - now this "inherits" (gets) some of its attributes from a
"normal" savings account - and it gets other attributes from "treasury
bills".

Now treasury bills are DEFINITELY not a type of savings account
 Neither are savings accounts a type of treasury bill.

Therefore, "how" do you define your CD account so that it can gets "some
attributes" (and/or properties) from savings accounts and others from
treasury bills?

Answer:
  Multiple inheritance !!!
   A CD account gets some of its "methods" from savings accounts and others
from treasury bills.

  ***

Note well:
  This may well NOT be how you would actually create a "full function
real-world" banking application, but I think that is show the "thinking
behind" multiple inheritance.

Note2:
  If you language does *not* support multiple inheritance, then you could
create a "dummy" (interface?) object for *all* types of accounts.  This
would have all the methods *names* for savings accounts, treasury bills, and
CD accounts.  However, in the actual savings account object definition, you
would override (and dummy out) the inherited methods that do NOT apply to
this type of account - and similarly you would dummy out the methods in the
treasury bill account object definition that ONLY apply to savings accounts.
While your CD account would have all the methods that it needs - without
multiple inheritance.

  ***

Warren - is this clear enough?

OO people - have I mis-stated this "simplistic" example?

If I invoke F117 I am going to get everything that's in

If an engineer designs a plane called the F117Passive which

    copy "Airplane" working-storage code

    copy "Fighter" working-storage code

    if value-airplane
        perform Airplane
    if value-fighter

    if value-fighter-bomber

    if value-F117

Methods library
    method Fly

    method BeStealthy

Classes library
    class Airplane
