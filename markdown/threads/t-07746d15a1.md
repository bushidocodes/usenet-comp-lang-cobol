```
I am running various MF 3.1 COBOL apps (DOS) under Windows95. I have a
dot matrix printer and a laser printer, both of which are connected to
LPT1. I use an electronic switch to control which printer is active. The
switch accepts commands from within my COBOL programs using the CALL X"91"
FUNCTION 35 routine.

My COBOL apps all use the dot matrix printer. My Windows apps all use the
laser printer.

When I enter a COBOL app, I use a CALL to set the switch to the dot matrix
printer. When I exit this app, I use a second CALL to reset the switch to
the laser printer, so that it will be ready for any subsequent Windows
apps.

The difficulty I am encountering deals with the Windows print spooler
(which I do not want to disable). When the second CALL is issued, the
command is immediately transmitted to the switch, irregardless of any
tasks still pending in the spooler. The result is that the first portion
of the report is sent to the dot matrix, and the last portion is sent to
the laser printer.

Is there any way to delay the transmission of the second CALL until all
data has been printed? Any suggestions will be appreciated.
```

```
You might try testing to see if the printer is still busy by using the
MF
call-by-name function PC_TEST_PRINTER. If the printer is still busy,
wait a little bit and test it again before issuing the CALL to switch
printers.

I'm not sure how the PC_TEST_PRINTER function works with laser
printers, but in my experience it works well with the dot matrix
variety.
```
