ignoring the problem depending on what the programmer has indicated.

The following is probably much LONGER than some people will want, but it is
the "concepts" section's discussion of the VALIDATE  facility from the draft
of the next COBOL Standard.

"C: Concepts 717
C.17 Validate facility
The validate facility provides a major procedural statement VALIDATE and
several associated data division clauses that enable data to be checked for
various errors and inconsistencies in a high-level and comprehensive manner.

C.17.1 General description
The VALIDATE statement can be used to perform validation on any data item
defined in the file, working-storage, or linkage section. The item is
checked for conformance to its data description and messages or flags of the
programmer's choice may be issued in response. Data items can also be stored
automatically in target locations.

The detailed processing undertaken by the VALIDATE statement is established
entirely by the description of the referenced data item rather than by code
in the procedure division. As well as general data division clauses, such as
PICTURE, additional clauses may be included that control the action of the
VALIDATE statement but have no effect
otherwise.

The data item may be a group or elementary item of any length and
complexity. The operation of the VALIDATE statement is divided into several
stages. Each stage is completed for the entire data item, including all its
subordinate items, before the next stage begins. If an elementary data item
fails one stage of validation it cannot be rejected at any further stage. If
a check fails, there is no interruption to processing: instead, errors are
indicated by the storing of messages or indicators defined by the
programmer. The stages are as follows:

C.17.2 Format validation
This stage uses the PICTURE clauses, possibly modified in their meaning by
the SIGN and USAGE clauses, to check that each data item has the expected
data format. Elementary items that fail this check or (if usage display)
contain all spaces are assigned a default value that is used in all
subsequent references to the item. A group item may also have a DEFAULT
clause.

The clauses relevant to this stage are: PICTURE, DEFAULT, USAGE, SIGN.

C.17.3 Input distribution
This stage activates any DESTINATION clauses that have been defined for the
data item by storing items in their target locations where indicated. By
virtue of the default values, target locations always receive valid data,
unless DEFAULT NONE is specified when they are unchanged.

The clause relevant to this stage is: DESTINATION.

C.17.4 Content validation
This stage checks that each data item lies in the expected set of
permissible values. 88-level entries may be used to check the values of the
data item as a whole and the CLASS clause to check each of its characters.
The range of values specified in 88-level entries can depend on specified
conditions.

The clauses relevant to this stage are: CLASS, VALUE.

C.17.5 Relation validation
This stage checks that data items have appropriate values in relation to any
other data items. Other terms for this stage are "inter-field checks" or
"cross-field validation". The data item may be allowed to contain certain
values, regarded as a "special case" which are always regarded as valid
whatever other checks may be specified. For example, the clause ALLOW ALL
SPACES for a numeric item means that it can be skipped over as input and
left "blank".

The clauses relevant to this stage are: ALLOW, INVALID.
718 C: Concepts

C.17.6 Error indication
This final stage operates on any ERROR clauses that are defined and uses
them to set up messages or indicators, specified by the programmer, to
identify which items, if any, have been rejected. It is also possible to
distinguish between the three reasons for rejection. The ERROR clauses are
not placed within the data item being validated but elsewhere in the
program's data division.

The clause relevant to this stage is: ERROR.

C.17.7 Validation of more complex formats
The data description can contain subordinate entries that have an OCCURS
clause. Each repetition of the data item can then be checked independently,
can have an independent DESTINATION, and can be assigned an independent
error message or indicator. If the data description has alternate formats,
these can be described using the REDEFINES clause and the appropriate
description can be selected using the PRESENT WHEN clause.

The clauses relevant to these formats are: PRESENT WHEN, REDEFINES, OCCURS,
VARYING.

I have used it once.  I think the situation was that I needed to write out
a flat file with delimiters between all the fields.  The info was coming
from a Honeywell DMIV record, which could be similar to a working-storage
group level, but it's a little different.  Anyway, here's the way we
layed out the outgoing file description (as best as I can remember):
  01  OUT-REC.
    03  REC-FLD1           PIC X(5).
    03  DEL-1              PIC A.
    03  REC-FLD2           PIC 999.
    03  DEL-2              PIC A.
    03  REC-FLD3           PIC XXX.
    03  DEL-3              PIC A.
    ...

And here's the kind of code we used (as best as I can remember):
    INITIALIZE OUT-REC.
    INITIALIZE OUT-REC REPLACING ALPHABETIC BY ":".
    MOVE CORRESPONDING MY-RECORD TO OUT-REC.
    WRITE OUT-REC.

What this did was put colons in all of the PIC A fields and spaces or
zero's everywhere else.

It's been a few years since we did it, so I wasn't able to find any real
examples, but I tested this with Micro Focus and it did what it was
supposed to.  I realize that move corresponding is usually shunned, but
for our needs, it was the best solution.  Best as I can recall, the reason
we didn't simply move all ":" to the record was because this resulted in
some sort of illegal character abort.

Ken Row
