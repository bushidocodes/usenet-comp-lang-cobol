Something like this is what you could use:

FD  Input-File.
01  Hdr-Record.
    05  Hdr-Id         pic x(01).
    05  Hdr-OrderId    pic x(06).
    05  Hdr-CustID     pic x(04).
    05  Hdr-OrderDate  pic x(06).

01  Dtl-Record.
    05  Dtl-Id         pic x(01).
    05  Dtl-StkNo      pic x(06).
    05  Dtl-Amount     pic 9(04).
    05  Dtl-Price      pic 9(04)v99.

Then read each record and make a decision based on the
ID field.  Having two 01 levels allows you to redefine
the record two ways.

if Hdr-Id = 'H'
    perform Header-Logic-Using-Hdr-Fields
else
    if Hdr-Id = 'I'
        perform Item-Logic-Using-Dtl-Fields
    else
        perform Error-Logic
    end-if
end-if.

Not elegant, but effective.

HTH
Terry

This is a very common file organization.  One solution is to redefine
the working storage where you READ INTO.  Another solution is to have
two working storage spots.  Read a record, and pick which 01 to move the
data two depending on your record type.  This second lets you still see
your header data while processing detail records.
