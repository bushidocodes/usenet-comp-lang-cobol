In my travels as a Migrant Programmer a.k.a. consultant, I have found an
almost universal standard for the naming of Paragraphs, and restrictions of
GO TOs in a program.

1) The paragraph has a high level qualifier which must be used for all
Paragraph names from the entry point
to the EXIT.
.i.e   0300-BEGIN-PARAGRAPH.
        0300-something-in-the-middle.
        0300-exit.             exit.

2)  Any GO TO within the Paragraph is confined to the scope of the
Paragraph, unless the GO TO is to an ABEND routine. (My fanaticism, sorry).

There are a variety of solutions to the enforcement of rule 2.

1) A COBOL program can be written to report any violations of the rule.

2) Edit macros can be used to either:
  a) report violations of the rule
  b) present the source program in a manner which simplifies visual
inspection of the code.

The Source Editors I am familiar with (ISPF Edit, XEDIT, KEDIT, and SPF/SE)
all have Edit Macro languages.
ISPF Edit, XEDIT, and KEDIT use REXX, and SPF/SE has a C Interpreter.

They all support the suppression (hiding) of source lines. (EXCLUDE).

You may be able to use the following method to determine the scope of the GO
TOs within your program without the need of 1 or 2a.

1) EXCLUDE ALL of the source.
2) Find ALL occurrences of the word GO.
3) Exclude COMMENTS ("*" in col 7)
4) You should now have only the lines with "GO TO" showing. (provided the GO
TO is on one line)
5) Find ALL non-blanks in col 8 thru 11. (this will show you the Paragraphs,
and "01" "FD" etc.)
6) Locate the Start of the PROCEDURE DIVISION.
7) Page through the program. It should be easy to spot GO TO Violations.

The Text Editor is the most powerful tool we have. The Macro Languages
extend the basic instruction set of the editor.
