do not return to the mainline.

I agree and I disagree.

It is true that ANY program that does not have an Exit Program, Stop Run (or
the extension GoBack) has a logic flow problem. It is also true that in
most cases, this means that the program will fall thru from the "mainline"
to the next paragraph (section) - and exhibit unexpected results.

HOWEVER, there was a change between the '85 and '74 Standards concerning
what happens when you do not have an Exit Program in a subprogram (which is
how I read the original question). The difference is that the program in
'85 must behave as if an Exit Program WERE coded at the end of the source
code (physical end - not logical end). This was not true in '74 COBOL.
Therefore, you are correct that the chances of you actually falling thru to
the last paragraph are slip (and then falling thru to the implicit Exit
Program) - this is how the Standard is written.

Now to get to the posters REAL question. I think DOS/VS COBOL works like
OS/VS COBOL did (I have only worked with OS/VS COBOL - so this could be
wrong). In OS/VS COBOL, if you do get to your final source code paragraph
(with no return to the mainline) and the program "falls thru" - what happens
is that it will ABEND. On the other hand if you are using VS COBOL II or
COBOL/VSE (with NOCMPR2 - so it is '85 Standard conforming), and again you
get to that last source code paragraph and have no explicit return (of
perform or GoBack) - then it will work AS IF an Exit Program were coded.

P.S. For all you CICS programmers out there - MVS, VSE, or otherwise - I am
well aware that you rarely (never?) actually execute a GoBack, Stop Run or
Exit Program. Your programs usually get out of an application with an EXEC
CICS something-or-other. The rules described above apply to "normal" (ANSI
conforming - usually batch) applications.
