If you're still talking about ISAM files, N.B., no two compilers
handle ISAM files the same way - ISAM files are NOT portable between
compilers (other physical formats may not be portable either).

While the source is portable, the data the compiled program
uses/generates may not be (and probably isn't) portable.

I agree with the point that Jerry was making - but I am not certain whether
or not it is "relevant".

If you find things that the Fujitsu compiler does with indexed files that
are not "conforming" to the ANSI/ISO Standard as far as the standard OPEN,
CLOSE, READ, WRITE, etc statements, then you have found a BUG in that
compiler (as they do claim to be Standard conforming).  If this is the case,
I would be interested in the "symptoms" that you find (and particularly if
you have compiled your program with Fujitsu's - or Micro Focus - standard
"flagging" option - to make certain you really ARE using Standard features
and NOT extensions such as split keys or READ PREVIOUS).

If you are talking about having the "object code" produced by one compiler
be able to READ/process the data files created by another compiler, then you
MAY WELL be out of luck (especially for indexed files).  I know that Micro
Focus (used to?) handle some "external" file formats such as C-ISAM and
BTRIEVE.  If Fujitsu also handles either of those "file types" then you
MIGHT be able to share ISAM files across applications created by different
compilers.

NOTE: (IBM) Mainframe programmers - this is a NON-issue for you - where the
"indexed file system" belongs to the Operating System and *not* to the
compiler vendor.  However, in the "workstation" environment, this can turn
into a REAL nightmare.

It seems that you want to have the printer set to 'compressed' or
landscape or similar rather than this being a problem with Cobol.

You could send an initialisation string to the printer, but this depends
entirely on the make and model of the printer and may also be affected
by setting in the compile or run-time.

Find the manual for the printer and work out the codes required.  You
can write hex values into your program using x"1B" for Escape etc.

Check for 'zero insertion' in your Cobol implementation MF).  There is a
run-time option that cause control characters to be preceeded by x"00",
you don't want this.
