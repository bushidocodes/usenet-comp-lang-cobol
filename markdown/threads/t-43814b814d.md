You don't need to worry about binary and COMP-3 data from the
 mainframe.

I'm in a similar situation - I'm doing mainframe (EBCDIC) to UNIX
(ASCII) and back to mainframe.

We've written our own codeset translator as a called / linked-to
module that can handle several different record definitions.  All we
do is pass and a unique identifier for each record, the input &
 output
codesets, which translation to perform and a couple of areas large
enough to hold the inbound and outbound buffers.

The translator then looks at the "NULL" record buffer (the values
loaded into a record type with the INITIALIZE) verb and if a
particular byte is either a space (ASCII X'20') or zero (ASCII
 X'30'),
we replace the character in the inbound buffer with it's equivalent
character in the outbound buffer.  If the it's not one of those 2
characters, just move the byte from inbound to outbound.

WARNING - I don't know what compiler you're using, but under
MicroFocus v3.? (and probably others) you must use caution with
 signed
numeric usage display fields - PIC 9(05).  MF does NOT initialize any
bytes with a sign unless you use a SET directive.

This is what we've done and it seems to work.  Hope this helps.
