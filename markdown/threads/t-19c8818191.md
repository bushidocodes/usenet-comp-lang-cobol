We developed an IBM mainframe CICS system in the three tier architecture a
few years ago. This meant the following.

Each transaction is really divided into...
1). The main (screen?) program.
2). A business validation / execution layer
3). All of the data modules.

All of the data was used via linkages set up specifically for each module.

So a customer maintenance transaction would...

LAYER 1.
1a. Receive all of the screen data. This data was captured in a "view"
linkage.
1b. use an "Evaluate" statement to check the actual function to be performed
(ie add, change etc), and call a central control module (a scheduler?) with
the "view" linkage and the function to be performed.
1c. Evaluate the return code. If any errors occured display error.
That  is all the top layer would do.


LAYER 2.
2a. The control module (scheduler?) would then just call the appropriate
validation module. The secret here is to have a validation routine for
"each" item to be validated. If a module is coded that does more than one
validation, then you are really asking for trouble (believe me I know. We've
done this and it cost us undue hardship).
2B. Each of the validation modules performs some kind of validation, and
returns some kind of status whether it was successful or not. If it wasn't,
a status is returned from the validation module to the Control module and
from there to LAYER1.
2C. If all was fine the control module does the commit, or returns any data
collected via another linkage (ie the "collected" linkage back up to LAYER
one.


LAYER 3,
For each of the files used, a module is written to perform all of the
relevant actions.

To fully understand this,  the linkage of the CONTROL system ( in our case
anyway), is a constant redefinition of a 4k buffer into the various linkages
we use. The control module (the scheduler) is called many times throughout
the transaction life.

Generally this approach is OK, but one must stick to it like glue. any
deviation from this could have catastrophic results. We have a particular
validation module which is function specific (ie we validate all of the data
required for RECEIPTS). This module consists of code which is is as complex
as you get. The simpler approach would have been to have validation modules
for each item to be validated. Yes that would mean more programs but hey,
simpler to manage, and maintain.


As a final note, if we need to turn some of our VSAM files into DB2 tables,
all we need to do is rewrite the module in LAYER3. etc etc....

I hope this makes sense.
