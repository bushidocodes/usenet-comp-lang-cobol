Careful, David. Even what you "know" is not necessarily so.

An existing file on tape or disk, say, is not necessarily a table. Nor is
it necessarily an input file: it is just a file; it may be used as input,
but it could also be used for update; or it could be extended (have
records added to the end) without being an input file.

Get a few concepts down solidly first:

A file is a collection of logically related records. For example,
information about a student would be a "student record"; the collection of
all the student records for your college would be a file. In itself a file
is neither input nor output: it is just a file, it exists in a magnetic
form on a media, it is "data" (or, if you prefer, "information").

A program that works with a file may use the file as input, or output, or
update, etc. We classify a file as input, output, update, etc. simply
based on how the particular program uses the file: it's a function of the
program usage, not the file itself.


To talk about

>simple source code of an external file that has been written and then
loaded

is to murder the language. A file does not have source code, simple or
otherwise. I think careful use of English (or whatever your native
language is) helps clarify concepts and thinking enormously.

I think what you want to see is "source code for a simple program that
processes a sequential file". But it is not clear because the context of
your post centers on tables. If you understand that files are not tables,
you can break a bad habit of thought.

Let's look at a simple program that processes a sequential file:

Identification division.
Program-id. ISDF2F.

Environment division.
Input-output section.
File-control.
Select INFILE assign to INDD.
Select OUTFILE assign to OUTDD.

Data division.
File section.

FD INFILE
Block contains 0 records.
01 INREC pic x(128).

FD OUTFILE.
01 OUTREC pic x(128).

Working-storage section.
01 Record-work-area pic x(128).

Procedure division.
Initialization.
Open input INFILE output OUTFILE.

Copyfile.
Read INFILE into record-work-area at end go to termination.
Write outrec from record-work-area.
Go to copyfile.

Termination.
Close INFILE, OUTFILE.
Stop run.


This program reads in records from a sequential file and writes them out
to another sequential file, unchanged. Now this does not help with your
request for tables, but at this point I am concerned with your
understanding of files, not tables.

Part II: A brief look at tables.

Think of a table as a list of items that all have the same attributes.

A list of country names, say, or cities
A list of sales totals from various stores or outlets
A list of stolen credit card numbers

To describe such a list in COBOL, specify how many items there are in the
list and then describe the attributes of the typical item. For example:

01 Countries.
05 Country-name occurs 30 times pic x(25).

This says I have a list of country names, each 25 characters long. There
are 30 entries in the list. The list as a whole is called "Countries".

In the Procedure division, we specify the name of a typical item followed
by an indication of _which_ element in the list we mean:

if country-name(5) = 'France' then ...


This indication of which item we are referring to in a list is called an
occurrence number. The number in parentheses is called a _subscript_ and
it is used to indicate the occurrence number we are referring to.

A subscript may appear as an integer greater than 0 but not greater than
the maximum number of elements in the table (list); or it may appear as a
data name that has been defined as an integer item:

if country (5) = 'Britain' then ...

if country(c-sub) not equal to trans-country then ...

You do not reference records in a file using subscripts, because you can
only process the current record in memory and there is only one record in
memory at a time.


Often, the elements in a table are more than elementary items: they may be
group items with subordinate elementary fields:

01 Countries.
02 Country occurs 30 times.
03 Country-name pic x(25).
03 Population pic s9(11).
03 Currency-name pic x(15).

Now, references to any of the subordinate items requires a subscript.
Population of which country, for example. Well,

Population(co-num)
or Population(5)

Note that an OCCURS clause may not be specified at the 01 level.

There's a lot more to be said, but this post is getting excessive. I hope
this can get you started. Read. Think. Work on precision of language.

Good luck.

Dear David:
There are different types of files, but let's look at the classic
old card reader input loading a table of Names and Phone Numbers.

Suppose some ancient COBOL system was written to read in some cards
and put the data from them in an internal array (table).

Suppose that the cards are 80 columns wide. Suppose the card contains
the persons first name in the first 20 columns and last name in the
second 20 columns. So far this adds up to 40 bytes. Then suppose
the area code is 3 characters. Then suppose the phone number to be
7 characters.

So Key punch would type in the following:

Bob Smith 4084321234
12345678901234567890123456789012345678901234567890
Columns-->

So here is the COBOL record layout:
ENVIRONMENT DIVISION.
..
SELECT INPUT-FILE
* WS-FILE-STATUS GIVES I/O RETURN CODES...
FILE STATUS IS WS-FILE-STATUS
ASSIGN TO UT-S-INPUT.
DATA DIVISION.
FILE SECTION.
FD INPUT-FILE
RECORD CONTAINS 80 CHARACTERS
DATA RECORD IS CARDIN-RECORD.
01 CARDIN-RECORD.
05 CARD-FIRST-NAME PIC X(20).
05 CARD-LAST-NAME PICTURE X(20).
05 CARD-AREA-CODE PIC X(03).
05 CARD-PHONE-NUMBER PIC X(07).
05 FILLER PIC X(30).

You see, 20 + 20 + 3 + 7 + 30 add up to 80 columns (characters)


The card has 80 columns as it is read into the card reader.
The card is like one record on a sequential file such as a tape.

The COBOL program in question will open the input, read each
card one at a time and move the data to the table, while adjusting
the table (array) index to file the table.

A table is like a storage area that can contain more than one
card, and each card is accessed using the subscript.


Now here is the working storage definition of a name and phone table.

We don't need the filler, just the name and phone. So we define it
as so:

01 WS-PHONE-TABLE.
05 PHONE-ENTRY OCCURS 300 TIMES INDEXED BY IMA-INDEX.
10 TBL-FIRST-NAME PIC X(20).
10 TBL-LAST-NAME PIC X(20).
10 TBL-AREA-CODE PIC X(03).
10 TBL-PHONE-NBR PIC X(07).


And so we open the input file of cards, either on a tape, in the card
reader, or copied via a utility to a sequential disk file as input:


OPEN INPUT INPUT-FILE.
READ INPUT-FILE
PERFORM VARYING IMA-INDEX FROM 1 BY 1 UNTIL IMA-INDEX GREATER
TBL-MAX OR
WS-FILE-STATUS GREATER ZEROES
MOVE CARD-FIRST-NAME TO TBL-FIRST-NAME (IMA-INDEX)
MOVE CARD-LAST-NAME TO TBL-LAST-NAME (IMA-INDEX)
MOVE CARD-PHONE-NUMBER TO TBL-PHONE-NBR (IMA-INDEX)
READ INPUT-FILE
END-PERFORM
CLOSE INPUT-FILE.
At the end of this loop all our card up to 300 (value of tbl-max)
will be in that array (table). And the first one we read will
be having the subscript value of 1. And the second one will have
the value of 2, etc...

Now we can use the search verb to find somebodys phone number in the
table...

set ima-index to 1.
search phone-entry
when tbl-last-name (ima-index) equal ws-look-for-last-name
display 'phone number = ' tbl-phone-nbr
end-search.
