```
The word TERMINAL is a reserved word in VS COBOL II Release 4.0 - the
compiler which I assume you are refering to.

In the Language reference manual for that product / release, TERMINAL is
marked as "Non-COBOL" in the reserved word appendix (I don't write this
stuff, just looking it up). The definition of "Non-COBOL" is:

"Words marked under Non-COBOL are COBOL 85 Standard Reserved Words that
are NOT implemented in VS COBOL II Release 4 and are flagged with an
S-LEVEL message if used as user-defined names."

In the manual for VS COBOL II Release 3.2, the description of the reserved
word status of TERMINAL was different, TERMINAL was marked as "85 ANSI
Standard", which was defined as:

" Words marked under 85 ANSI Standard are NOT implemented in VS COBOL II
Release 3 and are flagged with an S-LEVEL message."

One can build a user reserved word table - it is described in the
Installation and Customization manual for the product, but I have never
done it to kill a normally reserved word and make it available as a user
word.

Biggest question, is why is the compiler seeing the word TERMINAL at all.
I would beleive that only the CICS precompiler should see this and by the
time the compiler sees it, the word should have been changed to some CICS
mumbo jumbo, and the only thing the compiler would see is a CALL to the
CICS interface.

If the EXEC CICS is buried in a COPY book, then you get into the problem
of "what order which process sees the source", but if in open code, the
precompiler should parse the EXEC CICS stuff, and leave only native COBOL
for the compiler to deal with.

Hope this sheads some light on the problem. I would concentrate on why
the compiler is seeing the word TERMINAL at all, since I beleive the
compiler should only see CALL DFHxxx with some parameters.
```
