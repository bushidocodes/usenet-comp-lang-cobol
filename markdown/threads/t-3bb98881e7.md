However, I do not think that the problem.  The problem is, as Thane said,
that a LINE SEQUENTIAL file is one of those funny little cobol things that
really means "whatever your system regards as text" or maybe "source code
format".  It is system dependent, and on a PC usually has tabs expanded to
spaces at an invisible level.  There may be a switch that can be set to stop
that, I do not know.

There are several ways you can take care of the problem. I would create a
file object that mimics the behaviour of the file type I wanted
wanted(unexpanded tabs), and use the "UNSTRING DELIMITED BY ALL x'09'."
method, just as the code is written. The actual code in the object could
read the file one byte at a time, sequentially.  That way you will see
everything, character by character, including whatever your system uses for
an end of line (PC <CR><LF>). While it would be system dependent, you could
easily put in a platform PROPERTY that allowed you to set the local file
type.  That could be stored in the local setup object.

For that matter, the whole LINE-SEQUENTIAL file structure could be packed
into a standard FILE-OBJECT that had PROPERTIES settable at runtime instead
of OPTIONS set at compile time. That is the sort of thing that object cobol
standards should set.  I'd be willing to bet money that the runtime
libraries already contain code to do both, and check a flag at runtime
anyway. But that is a whole other topic. (or is it?)

Donald.

> If  mystring(1:22) =  x"09"

No, that is not the problem at all.

When a LINE SEQUENTIAL file is read the TAB characters in the record are
exapnded to a number of spaces by the READ statement.  This means that
the TAB characters are not there to be tested against x"09", or any
other value.

There is no mechanism to avoid this TAB expansion on reading with a LINE
SEQUENTIAL file.

The best solution is to get the extract file in comma-separated format. 
If this is not possible then you may need to read the file as RECORD
SEQUENTIAL (ie SEQUENTIAL).  This imposes fixed length and ignores the
CR/LF terminators, but doesn't expand TABs.

Option 1:

   Make the record size 1 byte and read until CR/LF or EOF building the
line in memory.

   This is very slow.  

Option 2:     

    Make the record size 128 (or so) and extract records into W-S, each
time the end of the record is reached read another.

    You will get a 9/18 error on the last record but you can extract
from it anyway.  As a precaution move spaces to the record area before
each read.
