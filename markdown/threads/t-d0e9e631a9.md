```
We ran into this Y2K defect in Mainframe COBOL just this week.
(My apologies to those that have already seen this problem.)

   02  DATE1YY PIC 99.
   02  DATE2YY PIC 99.
   ...
   IF DATE1YY = DATE2YY + 1
      ... do something ...
   END-IF

We have discovered that the cobol compiler computes the expression 
"DATE2YY +1" and stores the result in a field that has more precision 
than "PIC 99", so the compare-value ends up being "100" when we expected 
the compare-value to be "00".  The value of 100 will cause this IF to 
always fail.

We have already corrected this piece of code.  I'm just posting it here 
so someone else might be able to spot this sort of defect in their 
systems.

Pete
```

You are absolutely correct. It will not impact the computation of the
arithmetic expression. The reason is that while the arithmetic
expression is being calculated, it is done to a greater precision, so,
assuming that DATE1YY is '00' and DATE2YY is '99', the evaluation of
the IF statement will prove to be false, because 0 is not equal to 100.
This is quite clearly explained in the section on "Intermediate
Results" in the "Programming Guide" for your platform (I'm looking at
the one for COBOL for OS/390 & VM, V2R1).

However, if, given the values above, you were to execute the statements

    ADD 1 TO DATE2YY.
    IF DATE1YY = DATE2YY
      .... do something
    END-IF

you will find that the proper (and expected) truncation rule is
followed when storing the value of DATE2YY, and the value will be '00',
causing the conditional to succeed.

This is most emphatically NOT a Y2K bug.

Jerry

I think Jerry P meant to say that 1999 was in the 20th century,

I think it's reasonable to call the two high-order digits of a
four-digit year "CC" in COBOL, simply because it allows you to work with
both 2-digit years and 4-digit years.  How else would you name the
high-order digits?

About two years ago there was a clever piece of COBOL code posted in
comp.software.year-2000 that used a 4-digit year split into CC and YY to
make a more efficient test for leap year.

01  SOME-DATE.
    05  YYYY.
        10  CC      PIC 99.
        10  YY      PIC 99.
    05  MM          PIC 99.
    05  DD          PIC 99.

IF YY = 00 THEN
    DIVIDE CC BY 4 GIVING USELESS-JUNK REMAINDER LEAP-INDICATOR
ELSE
    DIVIDE YY BY 4 GIVING USELESS-JUNK REMAINDER LEAP-INDICATOR
END-IF
IF LEAP-INDICATOR = 0 THEN
    it's a leap year, so do your leap year stuff here
ELSE
    it's a normal year
END-IF

So, 1900 is not a leap year, because 19 mod 4 does not equal zero.  But
2000 is a leap year because 20 mod 4 does equal zero.  1999 is not a
leap year because 99 mod 4 does not equal zero.  But 1996 is a leap year
because 96 mod 4 does equal zero.  I was impressed by the simplicity,
since it allows you to apply the 4-year, 100-year, and 400-year rules
while executing only one DIVIDE statement instead of three.

The person who thought of this should get some credit for creative

Historical note.

The IBM MVS macro TIME was defined as returning the date in format x'00YYDDDS'
in general register 1, where YY was the last two digits of the year, DDD was
the ordinal date number within the year and S was the positive sign. When IBM
became concerned about making the macro 4 digit year they ran into a problem.
If they changed it to x'YYYYDDDS' the results wouldn't be downwards compatible
and date arithmetic would return incorrect results. The compromise was to use
x'CCYYDDDS' where CC is defines as the century number where 19=00, 20=01, etc.
Thus the IBM code is downwards compatible.

Note that the TIME macro has new operands to get the YYYYMMDD value as well as
other forms. But the program last compiled 20 years ago will still work with
the old forms.
