Richard Smol wrote in message <7gmopn$n3m$1@pale-rider.INS.CWRU.Edu>...

>C is more like a loaded gun that you have to handle carefully. VB is more
>like a hand grenade with a pin that may drop out any moment ;)  I have seen
>some big VB projects which caused major problems, because VB doesn't give
>programmers any real insight about what they are doing.

Why are you blaming VB for that?  If you mean that conceals the details of
GUI programming (the event loop, the Windows API, etc.) then I agree, but
this is a good thing most of the time.  VB does allow people without any
experience in system analysis to put together a program that looks good on
the outside but is horribly designed.  However no environment forces you to
ignore good analysis and coding techniques.  Even with DCL (Vax's command
language), which has no control structures other than GOTO, doesn't force
you to write spaghetti code - if you understand structured programming
techniques.


>You just have forms
>that you can extend at will, leading to horrid cases of the Blob
>AntiPattern.


It's a different method.  Someone who was formally trained in classic system
analysis techniques (like I was) sometimes frowns on the idea of rapid
application development and iterative development (put the screens together,
write just enough functionality for the user to get the idea of how it will
work, use suggestions to improve the interface, and start the process over).
But then you hit the Real World, where specs change and users don't know
exactly what they want.  Again its not the fault of VB that some programmers
don't know about standard debugging and testing techniques.

From what you are complaining about, C++ Builder would lead to the same
problems.

>
>>I don't think its too much to assume that you've never done a real project
…[5 more quoted lines elided]…
>
You've never had this problem with C? At least with VB you usually don't
have to worry about a statement in one line causing a side affect that won't
be apparent for 20 or 30 lines down.  For instance I just had an error in a
C program where writing to an invalid array element that had nothing to do
with the file or the file pointer caused fclose() not to work with an errno
of "File name not found".  Yes I'm well aware of the tradeoffs involved in
the lack of bounds checking in C, but because VB won't let you get away with
this or a myriad of other things that C will, it allows for any programmer
with the same skill level to write *more* robust code than with C.  Also VB
handles the intricacies of GUI development which allows you to get the work
done faster.  It's another side of the standard "make vs. buy" decision that
any IT professional has to make.

If the programmers are writing unmaintanable code you need to train/replace
the programmers not the environment.

>>So now a language needs to be object oriented to be a "real" language,  I
>>guess the C, and Cobol programmers who are reading this aren't using
"real"
>>languages either.
>
