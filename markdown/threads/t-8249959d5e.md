No, but it's relatively easy.

01  HASH-NUMBER-X.
   02  HASH-NUMBER  PIC 9.
01  HASH-SUM   PIC 9(6).


MOVE FILE-RECORD-X(I) TO HASH-NUMBER-X.
ADD HASH-NUMBER TO HASH-SUM.

Some compilers or run-times may fuss about the above construct for any of 
several reasons, but it should be a simple matter to fool the system.

If you're concerned with security, another way is to store the date the file 
was modified as part of the file itself. The date modified is part of the 
file's directory entry and is moderately difficult to hack. When you access 
the file, you can compare it's stored modification date with the one 
recorded by the system.

First, of course, you can't have a function in standard COBOL that
does anything to a file, because files aren't first-class objects in
COBOL. A COBOL function (or procedure) could operate on the *data*
from a file.

Second, "hash-code" means almost nothing without further
qualification. A hash is just the result of applying some function to
the input; usually it has the property that the range is smaller than
the domain, so multiple inputs give the same output (and so the
function is a one-way, aka lossy, compression function), but that's
not strictly required.

Adding the values of the bytes in the file, as HeyBub suggested, is a
valid hash function. Whether it meets your requirements we can't say,
since you didn't tell us anything about them.

You also didn't tell us what platform you're using. Many have a
variety of hash functions that can be called from COBOL, either as
part of the operating system or freely available.

So incorporate the notion of a check-digit to catch transposition problems. 
When creating the sum, multiply each character by the successive digits 
0,1,2,...9,0,1,2...

Let A=1, B=2, C=3

Then the hash for the file containing "ABC" becomes (1*0 + 2*1 + 3*2) = 8

While the file containing "CBA" becomes (3*0 + 2*1 + 1*2) = 4

If you limit the check-digit to one digit, you decrease the chance of an 
undetected error by 90%. Using two digits, you get 99% reliability. And so 
on.

Or (easier) just leave it up to the customer to catch the problem.

mario wrote:
> 
> I need a unique hash value of a file, so that i can control if a file
> is unique,

It's impossible to have a unique hash value for any file, unless the
hash output is at least as long as the file, by the pigeonhole principle.

If you have other guarantees about the nature of the input files, you
can create a unique hash value for each file (a "perfect hash"), but
usually the best you can do is use a hash that has a very low
probability of duplicate values for different files ("collisions").

> the problem in the methode of summing the chars is that the file
> considering   "adam loves eva" brings the same hash value as "eva
> loves adam" even if these are to very different files!

Yes, that's a collision. A simple checksum isn't very
collision-resistant, because addition is commutative.

> my program imports files, and i need to ensure that one file will not
> be importet twice,  for example 2008_09_data,  2008_10_data, but it
…[5 more quoted lines elided]…
> pretty save but not to 100%

You probably can't have 100%. You can reduce the probability of a
collision, though. An MD5 hash, for example, has an extremely low
probability of accidental collision, and even creating a deliberate
collision (a "chosen-preimage collision") is difficult.[1]

There are probably MD5 implementations available for your platform,
which you can call from your COBOL program. But you still haven't told
us what your platform is, or what COBOL implementation you're using,
so we can only guess.

It's possible, though non-trivial, to implement MD5 yourself in at
least some versions of COBOL. I wrote an implementation for MF COBOL
that's freely available; I posted it to the MF Forums.[2] If you're
not using MF COBOL, though, porting it to another COBOL dialect is
probably more trouble than it's worth.

There are a great many hash algorithms between simple modular
checksums and cryptographic hashes like MD5. Arnold Trembley mentioned
CRCs, which are more resistant to collisions than checksums. A good
CRC-32 (eg, the IEEE polynomial) should have close to probability
N*2**-16 of an accidental collision, where N is the number of files;
so if you test 256 files, for example, you should have about a 99.6%
chance that they'll have different hash values. Even with 2048 files,
the chance is about 96.9% that they'll have different hashes.

Calculating a CRC in COBOL won't be much fun, but it's doable.

Adler-32 is less collision-resistant than a good CRC-32, but it's
easier and faster to calculate. You should be able to implement it
pretty easily in COBOL, especially since Part A of the Adler checksum
is the sum (mod 2**16) of the bytes in the data, which is similar to
your existing code.

See eg http://en.wikipedia.org/wiki/Adler-32.

Note that Adler-32 would be a poor choice if the files are very short.
It works best if the typical length of the files is at least about
5600 bytes.


[1] Not as difficult as it should be, which is why MD5 is no longer
recommended for cryptographic applications, but hard enough for most
purposes.

[2] Go to http://forum.microfocus.com:8088/~Micro_Focus_Products/login
and log in, or create an account if you don't have one. Go to the
"Micro Focus Products" board and search for MD5.
