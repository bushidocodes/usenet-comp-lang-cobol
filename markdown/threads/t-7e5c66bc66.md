```
Welcome to the "wonderful world of File Status 39"

This has got to be one of the biggest inhibitors to migrating from OldBOL to
NewBOL.

Bottom-Line:
  With the new  IBM compilers
     You must (absolutely positively) make certain that your FD and its
records match your DCB attributes EXACTLY.

A few solutions (all of which are documented in the IBM migration guide)

A) Make certain that all programs that reference the file (create and read)
have EXACTLY the same FD and all the same records.

B) Use the *new* RECORD VARYING IN SIZE option in the FD (both create and
read) with the minimum and maximum record sizes (minus the 4 byte RDW - as
always)

C) Make certain that you have at least ONE 01-level (record) under the FD in
each program that is the maximum size (again 4 bytes smaller than your JCL
LRECL) and still specify RECORDING MODE V in the FD.

  ****

There are probably a few other possibilities, but one or all of these should
get you thru.  (Compiling with the CMPR2 compiler option also MIGHT work -
but I would really consider both temporary and "last resort")

In general, you are talking about your program "aborting".  I do suggest
that you get used to file status code checking - in all programs for all
files - not just for VSAM (which used to be the "rule of thumb" with older
IBM compilers)
```

```
I recommend C. I always recommend specifying as little as possible in the FD
statement. That way if you need to change the maximum record length you will
always be changing it in the same place, at the 01 level. Changing it in the
FD statement could possibly cause a compile error. I try to avoid these at
all cost, you never know what your response time for job submissions is
going to be.

You could also try specifying the LRECL in the JCL for the input file:

//INFILE  DD DSN=INFILE.WHATEVER,DISP=SHR,DCB=(LRECL=198)    REMEMBER 194+4
```

```
Unless you have a record with OCCURS m TO n DEPENDING ON variable_earlier_in_FD,
it is difficult to ask COBOL to read variable length files.  Try reading the file
one byte at a time.  After four reads you will know the block length (1st 2 bytes
of the Block Descriptor Word).  The next four bytes will give you the Record
Descriptor Word (same format).  The record length - 4 will tell you the number of
consecutives reads you will need to fill out a record.  You will also have to
keep up with the total number of bytes read in each block to know when to read
the next block.

HTH
```

```
Just a "tid bit" of information.  According to the ANSI Standard (as
interpreted for the ANSI'85 Standard), if you have an FD with

      RECORD CONTAINS   n TO M

it is IMPLEMENTOR DEFINED whether the file is "variable format" or "fixed
format"  (I am NOT kidding - there were reasons for this interpretation -
but it sure surprises some).  If you want to GUARANTEE that a file is
"variable format" *and* you are using an '85 compiler,
use the following (new with '85 COBOL) format:

    Record Varying in Size

(For our IBM-limited friends, the "RECORD FORMAT V" phrase is an IBM
extension.)
```

```
Bill Klein's recommendations are right on the money.

As far a reading a variable record one byte at a time to get the BDW* or the
RDW* in COBOL, faggetaboutit. COBOL does not give you the option of reading a
block of records and deblocking them yourself. You can do this in assembler if
you really are masochistic.  The simplest way to fix your problem is to align
the 01 length with the actual file.

*For the mainframe challenged:
BDW: block descriptor word
RDW: record descriptor word
```
