```
You must create a new cobol program. Declare your old file (.dat) for input
and the new text file for output, so rewrite the new from the old. An
example:

program-id. newTextFile.
       environment division.
       configuration section.
       special-names.
    decimal-point is comma
    cursor is cursore
    copy 'tasto.sna '.
       input-output section.
       file-control.

    select olfFile assign to "oldFile.dat"
    organization is indexed.

    select newFile assign to "newFile.txt"
    organization is line sequential.

       data division.
       file section.

    fd  oldFile label record standard.
       01  rec-oldFile.
        02 cod                              pic 9(9).
        02 name                            pic x(35).

       fd  newFile label record omitted.
       01  rec-newFile                              pic x(44).

       working-storage section.

 procedure division.

    open input oldFile
    open output newfile.

readOldFile.
    read OldFile next
      at end
      go stop run.

write rec-newFile from rec-oldFile
go read OldFile
```

```
Although the main logic should be as stated, do not forget that the
COBOL file may have some numeric date in a format that should be
converted before writting to the new file.

so the correct thing to do unless the final system understands the
COBOL numeric format, or if the file in question only has alphnumeric
or numeric edited fields is as follows.

    fd  oldFile label record standard.
       01  rec-oldFile.
        02 cod                              pic 9(9) COMP-3.
        02 name                             pic x(35).
        02 maxcredit                        pic S9(9) COMP-3.

    fd  newFile label record omitted.
       01  rec-newFile                              pic x(60).

       working-storage section.

  01 new-record.
        02 w-cod                              pic 9(9).
        02 w-name                             pic x(35).
        02 w-maxcredit                        pic -9(9).
....

     move cod to w-cod
     move name to w-name
     move maxcredit to w-maxcredit
     write rec-newFile from new-record.
     go read OldFile
```

```
My experience (PC and mainframe - legacy and new - systems) is that
CORRESPONDING is almost as "religious" an issue as SECTION versus
PARAGRAPH - and with as little "conclusive" evidence on either side.

It CLEARLY allows for "maintenance" errors - if you don't use COPY members
for the defined items.  All you need to do is add a field or a level to "get
in trouble".  On the other hand (ESPECIALLY for "quick and dirty" programs)
it can be a REAL time saver.  When used with COPY statements (with or
without REPLACING), it is medium-easily maintained.

There was a (false) rumor many years ago about it being "removed" from the
Standard (due to its removal - and subsequent re-appearing) in the CODASYL
JOD.  However, there is NO current ANSI/ISO J4/WG4 interest in even
"denigrating" it.

One thought (and I will admit that I haven't ever REALLY done this) is to
put a COMMENT in the record-description of the sending/receiving fields
making it CLEAR that they are used in CORR statements.  This might alert a
future maintenance programmer.
```

```
I used to use it and it was useful. IBM also implemented ADD...CORRESPONDING
which was also useful...

Like so many things in COBOL, there is often more than one way to achieve
your goal and the specifc method chosen depends on factors at the time you
are coding (pressure to get it out, maintenance, efficiency if you still
care, and so on...

There is ALWAYS an alternative to either ADD or MOVE... CORRESPONDING (do
individual MOVES or ADDS) but sometimes they can save time.  (Especially
where LARGE record layouts with many fields are involved).

In these days of on-line editors when we have the ability to quickly cut and
paste ("clone") source code it is maybe less important than in the days when
every line of code had to be carved into a cardboard card...

Why did I stop using them?

Well, I found the necessity to qualify fields in the remainder of my code,
more than offset the time I saved with the MOVE or ADD... CORRESPONDING in
the first place!

Remember, you can use duplicate names in COBOL (and you MUST if you intend
to use the CORRESPONDING options...<G>) but then you must qualify every
subsequent reference to EITHER of the Duplicated names....

So you end up coding "field1 of input-rec", "field1 of output-rec"  and so
on for every one of the fields...tedious. As I have observed elsewhere,
patience does not come naturally to me (I have to work at it) so I pretty
soon got fed up with doing this...

Hence, today I clone the moves and adds and use unique names.
```

```
This is, and has been, a feature of the standards since *at least* ANSI
X3.23-1974 (albeit at Nucleus Level 2).  Any compiler claiming *full*
conformance with that standard (or with the '85 standard) at the highest
level of specification supports ADD .. CORRESPONDING, as well as SUBTRACT
... CORRESPONDING.   See ANSI X3.23-1974 pages II-55 and II-89, and ANSI
X3.23-1985 pages VI-73 and VI-134.   I don't have a copy of the '68 standard
to check, but secondary evidence (the lack of any indication that this
documented feature is an implementor extension in the documentation for
COBOL '68 from two different vendors) leads me to believe these variants
were part of that standard as well.

ADD ... CORRESPONDING is not an IBM extension, it's been standard COBOL for
a very long time.
```

```
sometimes cosmetic changes can lead to greater use of a function. In
this case it also matches the syntax used by SQL, DYL280 (I think I've
seen it in Easytrieve) and C that I know of.  The current qualification
syntax is considered awkward and hard to read by many programmers who
have English as their first language.  Obviously I am one of thos who
dislikes it.  I think that it makes the subscipt or reference
modification clearer.  

Similarly cosmetics of a different sort may have doomed the MOVE and
ADD/SUBTRACT CORRESPONDING verbs.  By not having a compile option (no
standards involvement needed), that would cause the individual moves
(adds etc.) generated to be listed as comments after the CORRESPONDING
statement, any errors became difficult to find.  Another nice option
would have been a diagnostic that listed the mis-matches between the
sending and receiving structures.  If these were available you might not
see the antipathy toward the use of CORRESPONDING.

In conclusion, cosmetics can be important and influence the use of a
function.
```
