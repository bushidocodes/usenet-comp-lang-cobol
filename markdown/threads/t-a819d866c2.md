tlmfru wrote:
> 
> In Intel-style assembler, calls are implemented through the stack.  In
> IBM-style assembler, they are implemented by a BAL (branch-and-link)
> instruction, and the first step in the called routine is to save the return
> address which the BAL provides.

Yes. Most modern processor architectures follow one of these two
models: a contiguous call stack, or (possibly non-contiguous)
activation records. The majority use stacks.

> - there must be a similar concept in Intel - access to the "current address"
> that the program is executing at

Sure. The EIP (Extended Instruction Pointer) register is not directly
available to the operations that read the general-purpose registers,
but you can read the previous value (actually, the return address, so
the previous value plus the size of a call instruction) off the stack.
So by calling a minimal subroutine that copies its return address from
the stack to a general-purpose register, you can get the EIP value of
the instruction after the call.

(There are other, more arcane ways to do it; for example, use the
single-step flag and have the debug handler copy the execution address
into a register. But the call method is the simplest I know of offhand.)

> - if so, it could be used to implement calls without using the stack

Yes.

> - if so, is there any reason why it shouldn't be used?

Several:

- The Intel x86 ABI says that you use the stack. Do anything else, and
you're incompatible with every other piece of native code written for
that processor.

- Getting the EIP value on x86 is expensive, because you have to use a
call, as explained above.

- You have to allocate your activation frames from somewhere. If you
allocate them from the stack, you're using the stack, so you haven't
achieved anything different from the normal stack-based call
mechanism. If you allocate them from somewhere else (eg a heap), you
need to implement some kind of memory management. That's a bunch of
work, and expensive at runtime.

- x86 is optimized for stack-based calling. The push and pop
instructions are fast. The "leave" instruction is fast, and the
"enter" and "ret" instructions are relatively fast.

- Primitive contiguous stacks are inherently fast, which is why
they're popular for microprocessors, which is where the x86 family
comes from. Allocation and deallocation are just a matter of updating
a single register. Access to on-stack data is just a matter of using a
base-register-plus-offset memory-access mode, or if that's not
available, a register store followed by an immediate-mode add and a
memory load.

There are various advantages to activation records, because they're
more flexible than stacks, but they're more expensive, particularly
when stacks are easy to optimize in the processor.

Two cases come immediately to mind, both from lessons painfully learned as a 
"maintainer", and both involve situations which required stripping extraneous 
punctuation from code.

modified to be sandwiched within conditionals, in cases where it would be confusing 
or dangerous to relocate the entire chunk of code to some other paragraph.

	PERFORM PARA-ONE.
	PERFORM PARA-TWO.
	PERFORM PARA-THREE.

becomes:
	IF CONDITION
		PERFORM PARA-ONE
		PERFORM PARA-TWO
		PERFORM PARA-THREE.

... requires removing most of the totally unnecessary periods from each line.

with a comma. When reading thru the code, a comma can easily appear to be a period, 
and more than a few times I've been burned when a procedure appeared to be 
terminated by a period that in fact (after much troubleshooting) turned out instead 
to be a comma.

	PERFORM PARA-ONE,
	PERFORM PARA-TWO,
	PERFORM PARA-THREE.

It seems obvious in this brief example above, but in a large program, when EVERY 
line of code has a dot at the end, they all tend to look alike.
