Numbers in CoBOL are strongly typed - but CoBOL does treat I/O somewhat
differently than is common in other languages:

A field described with a PIC ZZZZZZZ or 999.99 is not really a number -
numbers are moved here to be displayed - but the this field is not directly
usable as a number.   Think of it as directions to map the number onto a
printer.

So we have numbers that we do arithmetic with.   These may look like
9(9)V99.   The "V" is an implied decimal point that the compiler is aware
of, but isn't really in working storage.   Do whatever arithmetic you want
with this number, and then move that number to the display field (the
computer will map the number onto the correct format).

I don't see from your post how you are doing input.  But to figure out the
code, do the following:

Move 5 to WS-UEP.
ADD 2 to WS-UEP.
Move WS-UEP to DISP-PREM.
Display DISP-PREM.

(I don't know the command you will se for your display).   You should see  "
        7".

I currently don't use screen input with CoBOL, but after you have tried
this, you may wish to ask for help with your input.  Please state your
environment and compiler.  Someone who is more familiar than I am with your
compiler will surely assist.

01  wrk-tprem             pic x(9).
01  num-tprem             redefines

01  ned-tprem             pic -(6)9.99.

01  wrk-eprem             pic x(9).
01  num-eprem             redefines
wrk-eprem           pic s9(7)v99.
01  ned-eprem             pic -(6)9.99.

01  wrk-uprem             pic -(6)9.99.



move pol-tprem    to wrk-tprem 
move pol-eprem    to wrk-eprem 
compute ned-uprem  = num-tprem - num-eprem
move num-tprem    to ned-tprem
move num-eprem    to ned-eprem

display  �pol-tprem = � ned-tprem
display  �pol-eprem = � ned-eprem
display  �pol-uprem = � ned-uprem

Comments:

Since the pol- flds are in a copybook it�s difficult to redefine them without
moving them to work fields.
The num- redefines create the cents positions and develops a sign in the event
you have a negative value in either fld.

The ned- flds create the edited version of the fields for display. This displays
a period (.) before the cents positions and a minus sign (-) if negative #s are
displayed. Leading zeros are also suppressed.

Hi HD,

Now that we see the fields come from a CICS map, it opens the problem into othe
areas.

I don't think your error routine is robust enough to handle all variations of
the numeric fields you're accepting, e.g. a 999.9999 entry would result in a 
99999V99 used in the computation; not good. A 4o99.99 entry would result in a
499V99 used in the computation; ditto.

You could first remove the spaces (leave the period (.)), check for the period
in the proper position, then remove period, then do "if numeric" test. If either
test is unsuccessful, reject the field. Pseudo code follows:

Move each non-space byte to a work field defined or redefined as below:

01 wk-fld-1.
05 wk-dollars    pic x(007).
05 wk-assumed-dp pic x(001).
05 wk-cents      pic x(002).

if wk-assumed-dp not = '.'
perf error-rtn
exit etc.
end-if
move wk-fld-1 data to wk-fld-2 using "move corr" as defined below:

01 wk-fld-2.
05 wk-dollars    pic x(007).
05 wk-cents      pic x(002).
01 wk-fld-2-num     redefines
wk-fld-2              pic 9(007)V99.   

if wk-fld-2 is not numeric
perf error-rtn
exit etc.
end-if
