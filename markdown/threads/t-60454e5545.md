There are several fallacies in this assumption:

1.It isn't HOW a language is implemented that is important, it is WHAT it
can do that matters. COBOL is a brilliant response to the problem of
implementing Batch Programming in a commercial environment. It was extended
into On-line processing by the addition of teleprocessing monitors like CICS
and IMS/DC and that is all fine. But today, we have objects running on a
Network with the need for distribution of the objects and remote invocation
of Methods. While OO COBOL is up to doing this, standard COBOL isn't.

breed" of programmers who have never been exposed to "English-like" syntax
in programming, simply find it verbose and cumbersome when they encounter
it. These people will implement Business Rules according to an algorithm,
and the shorter the code the better. They will argue that they can maintain

consider this code obscure and obtuse, does not make it so...)

containing encapsulated functionality will gradually replace standard
programming or not (and I agree it is arguable to what extent this will

functions as a matter of course. There is no need to maintain 10,000 lines
of source code so it doesn't matter whether it is English-like or not.

"3. As it stands, the Language is very well suited to what it was originally

maintenance for batch programming of commercial applications.) The  systems
environment has moved on way past this and better languages, more suited to
the Network environment are cheaply and easily obtainable. This is not a
problem with COBOL. It is simply a reality."

makes it that, it is COBOL programmers who are not prepared to expand their
skill set. "Everything I want to do I can do in COBOL". Fine. This says more
about limited aspirations than it does about a programming language.

and fingers. But I will use other languages and tools as well. I don't
consider that a "betrayal" of COBOL.

Some scholars speak Latin and Greek. They may even love the cultures and

"langue du jour" when they go to buy groceries at the corner store or
converse with friends...

COBOL doesn't need re-designing any more than Latin does.

If we did, it wouldn't be COBOL. The extensions to enable it to take a
meaningful place in the modern world have been dutifully implemented by the
vendors (and even the ANSI Standards people) and rejected by the User Base.

I reckon that has sealed its fate.
