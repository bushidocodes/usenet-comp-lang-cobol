The original IBM 360 architecture used 24-bit addressing (16 Mbytes). Although
the architecture has since been extended to 31-bit addressing, various
inconveniences still occur when crossing the boundary between 24-bit addresses
and 31-bit addresses ("the line").

There is no intrinsic reason why a compiler could not emit code to compensate
for this, but, as a practical matter, in the absence of any demand whatsoever
for tables larger than 16MB, CoBOLII doesn't.

More interesting is why you would want to do this in the first place. I suspect
it would be easier to solve your problem with a smaller table than it would be
to get your table compiled under CoBOLII.

Sounds like you're really trying to create a file in memory! I believe
the problem you describe is a limitation of the IBM COBOL compiler.
Address spaces can be up to 2GBytes in size, but for some reason, IBM
limited tables to 16MBytes (24-bit addressing).

Workarounds include splitting the table into two or more parallel tables
(i.e. put one or more data items in the first table, others in a second
or third table and reference them using the same subscript or index
value). You might also consider using data spaces which can be up to
2GBytes. That requires someone write assembler stubs to
insert/retrieve/search the table. As a final suggestion, you can
externalize the table as a VSAM file.

Good luck.

I sent an earlier response to this item, but beleive that the "AOL send
to newsgroup" address garbler trashed it. So I will try to write it again
and repost.

The entire IBM VS COBOL II and successors to date family of compilers (VS
COBOL II R 1.0, 1.1, 2.0, 3.0, 3.1, 3.2, 4.0, COBOL/370 1.0 thru 1.1, and
COBOL for MVS and VM) has the same restriction on working storage element
size. Sorry, 16MB per 01 is it. Not sure why - some reasons exist for
cases where a load module with embedded data must be generated. Both
CSECT and Load Modules have a 16 MB inherent limit - dating to the 1960's.
For RENT option compilations the data is external to the generated load
module, and in theory could range up to about 2000 MB in the 31 bit
address space. Some problems would no doubt occur if one were to specify
a VALUE clause for such a monster, even it the actual data could be
accomodated, the initializing code and data might turn out to be the
limiting factor.

While the designers may have thought that the increase in table size from
128K to 16 MB would keep the users quiet for a while, I personally
implemented a program about 10 years ago (using VS COBOL II) which ran
into this wall. The case in point was a program where for performance
reasons it was desirable to read what had been an entire VSAM file into
storage, and then drive transactions against a table instead of a file.
The end result was a 100 times improvement in run-time performance, taking
a multi hour job with millions of VSAM I/O's to a 3 minute job (albiet
with a Large REGION specification).

The technique used was to partition the data into 4 separate tables. The
first partition was not necissitated by the 16 MB limit, but was to
provide reduced paging and beter locality of reference for critical data.
The actual view of the table was about 32,000 1000 byte entries. This ran
into the 16 MB limit. The first partition was to extract the 8 byte keys
from each table entry and place them in a pair of key only tables. This
resulted in 2 tables 8 bytes x 16000 entries. The idea was that the keys
would be scanned a lot, and dense data would eliminate the paging. Why 2
tables instead of one? Idea here was since the big data tables would have
to be partitioned into two chuncks, the key tables would work better
partitioned the same way. Finally the actual data portion of the table
was partitioned into 2 1000 byte x 16000 entry tables. In the particular
application a very obvious key element (domestic or international
location) allowed an easy selection of which of the pairs of key and data
tables to use.

Why bother with all of this rot when one could cheat??? Everyone knows
the olde cheat (from early COBOL's on MVS) of placing a dummy table after
the real one, and just overflowing into the second (or third, or ....)
table's space. This cheat will eliminate the ability to use the compilers
SSRANGE debug facility - a great loss on any program with significant use
of subscripts. Additionally, with COBOL/370 1.1.1 and later a new
optimization option (OPT(FULL)) will cause the compiler to NOT ALLOCATE
any run time storage to unreferenced data elements (such as the overflow
dummy table). In addition, no guarantee exists that adjacent 01's will
receive adjacent run-time storage.

Builder of software archeology tools and other strange programs to help
survive in a legacy based world.

you could code a storage-allocate program that allocates storage via working storage,
say 10 meg, then call your processing program, passing the allocated ws storage.
Your processing program could then address the linkage section as well as it's own
working storage. using this method you can beet the 16 meg compile prob.

ex
*****storage-allocate prog.
working storage section.
01 ws-big-area-10-meg.

procedure division.
call processing-prog passing ws-big-area-10-meg.
goback.


******processing-prog.
working storage section.
01 ws-big-area-10-meg.

linkage section.
01
05 filler pic xx.
05 ls-big-area-10-meg.


procedure division using ls_big_area.

you can now reference ws-big-area-10-meg and
ls-big-area-10-meg thus addressing 20-meg of data with
no compile probs.

Also, reading my Janossy "VS COBOL II: Highlights and Techniques"

both hard-coded and runtime-loaded tables. That's better than
COBOL-74 limits of 131,070 bytes hard-coded and 32,767 bytes
runtime-loaded.

BTW, runtime-loaded means you define the table in your source but it's
empty until you load it with data (usally from a file) during the run.
The table size is set dynamically with the "OCCURS DEPENDING ON"
clause. Also, bytes means characters in COBOL.
