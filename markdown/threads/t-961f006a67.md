Environment variables, properly speaking, are the "exported" ones.
All Unix processes have an environment, which is a list of name=value
pairs.  This environment is inherited by all child processes (ie all
processes that the process creates using fork), though the child
process can then alter its own environment (using putenv or, if it's
exec'ing a new image, execve).

In addition, most (all?) Unix shells keep their own list of
name=value variables, separate from the process environment.  Bourne-
derived shells pretend that the two lists are the same, and that some
variables are marked as "exported" and so inherited by child processes.
(csh, notably, does not employ this fiction - it has two separate
commands for setting variables, "set" and "setenv".  Whether this is
an advantage is debatable.)

I don't know if Bourne-type shells generally actually move variables
from their private list to their own environment when you use the
"export" command, or if they simply mark them as "exported" and then
build a new environment for execve when loading new images in child
processes; it's irrelevant to this question.  (A quick check of the
source would probably reveal the answer, if anyone's dying of
curiosity.  Actually, since the order of setting the variable and
exporting it doesn't matter, it's probably the latter, as it'd be more
work to implement the former and still have variable assignment and
exporting work as they do.)

The main point, though, is that if you're able to retrieve a value
from the environment, then it's in the environment - in other words,
it's been "exported".

> I know, via testing, that if I use the following:
> 
…[4 more quoted lines elided]…
> value works properly.

More properly, MYENVVAR is in the environment.  You can't create an
"unexported environment variable", since there is no such thing.

It's easy to test this, by the way.  Write a trivial program that
just gets the value of a variable from the environment and then
displays it.  Use a name that's not already set in your environment.
Run the program; it should display nothing.  Set the variable without
exporting it and run the program; it should display nothing.  Export
the variable and run the program; it should display the value you
set it to.

In fact, I just did such a test, and those were the results I got.
