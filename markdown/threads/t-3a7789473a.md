I used to renumber source code a few years ago when small disk
drives and slow disk writes precluded me from producing full
compile listings.  The ERRLIST would give an error message
but it could be very hard to find where this actually was in
the source because the position was the number after it had
added the copy files.  Having a numbered source made it very
easy to find saving quite some time.

I started doing this when using 8085 machines with 2x 1MByte
8" floppys and CIS Cobol.  One of the first things I developed
was a compile system that would run the compiles unattended
overnight.

Line number were still useful on overfull 20Mbyte hard disks
where I couldn't afford all compiles to have COPYLIST.  

Certainly for the last few years I haven't bothered as getting
a full listing, even if a recompile is required to get it
out, is not a problem in either disk space or time.

In answer to several comments regarding prettycbl.

The reason I was doing the capitalization the way I described is that 
I was working on a Unix system using the VI editor. Unix and VI are 
case sensitive so if you capitalized the contents of a quoted string 
and that string happened to be a file name the program would no 
longer be able to find the file. Also, searches in VI are case 
sensitive so, without prettycbl, we always had to look for things 
twice, once in upper case and once in lower to find all instances 
and, of course, there's always someone who spells cat "Cat").

The reason for putting in line numbers is that in the VI editor you
copy or delete blocks of code by a line count, i.e. to delete ten
lines you position on the first line and enter "10dd". So this means
that it's handy to be able to compute the number of lines between to
two places in the program by just subtracting.
