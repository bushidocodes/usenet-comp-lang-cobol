3 + 5
    3 = permanent error condition
    5 = "The OPEN statement in INPUT, I-O, or EXTEND mode was
        executed for a file without the OPTIONAL clause specification
        The file has yet to be created, however"

In other words, you attempted to open a file in anticipation of
reading it and there was no file there.

One source for these IO status codes (assuming they don't
exist in your textbook) is www.adtools.com. Other vendors
of compilers no doubt have the messages on their web sites.

I don't know of any 'sample' code sites for this. Even if you found
one,
you would go nuts trying to figure it out. Here's how.

Create Relative File:
    OPEN INPUT SEQUENTIAL-FILE
    OPEN OUTPUT RANDOM-FILE.
    PERFORM VARYING RANDOM-RECNO FROM 1 BY 1 UNTIL
        INPUT-FILE-STATUS NOT = '00'
        READ INPUT-FILE NOT AT END
            WRITE RANDOM-REC FROM INPUT-REC
            END-READ
    END-PERFORM.
    CLOSE INPUT-FILE.
    CLOSE RANDOM-FILE.

To update a relative file:

OPEN I-O RANDOM-FILE.


MOVE record-number TO RANDOM-RECNO.
READ RANDOM-FILE.
do some processing
REWRITE RANDOM-REC.

Hope this helps.

Just another general FYI,

I've been writing code using Realia for years.  I've written new
programs using Fujitsu COBOL - but until recently I had not converted
any code.

My Realia programs use Lock Mode Exclusive.

Talk about meaning two different things with the two compilers!  (I'm
not complaining, just a heads up).

With Realia, a file status 9A was a successful open on a file with
exclusive access specified.  Multiple programs can open a file input,
but only one for IO or Output.  

A status of 9L meant the file was locked.

I would look when encountering 9L's until the file was unlocked, or I
gave up.

With Fujitsu, if you leave the Lock Mode Exclusive specified, it's
REALLY EXCLUSIVE - only one program can open the file at a time.  A
lock is indicated by file status 93.  There is no 9A equivalent (nor
need there be).

To get the same behavior as the Realia programs with Fujitsu one must
NOT specify ANY lock mode.  It was easy to find out how to handle it
from reading the user manual, but my first attempt was to test the
programs running side by side reading the same file.  Boy was I
suprised to see the 93 when both programs opened the file INPUT.

No on to more happy conversion......

We keep getting confused by this one "I'm using COBOL 85" it's the last
Standard issued not the 'name' of the compiler you are using.

First suggestion, as you are university-based, any of the good COBOL
text books in your library will cover the following in much greater
detail.

Your Error 35 - (unless it's something compiler specific), if it is a
file-error -  I get Open/Input/External non-optional - check the
text-books on file-status errors. 

You need, again from the text books, to get a handle on the difference
between sequential and relative.

Sequential - unordered :-

If I enter invoices into an A/P system, and assuming they are entered as 
follows :-

	Green Giant
	Badcombe & Widdock
	Heinz
	Cadburys etc....

That's the sequence they are in your 'unordered' sequential file.
As you'll appreciate the file can be sorted to make them ascending on
descending on certain fields. Primarily you are using sequentials for 
temporary/intermim transactions.

Now to relative. I first queried merits of 'relative' back on the old
Compuserve MicroFocus Forum. Michael Mattias here and Pete Dashwood
responded. Pete pointed out that relative had its day in early 60's
before indexed(ISAM) came into play. Michael favours using Relatives as
temporary working files.

With Relatives you don't have a key, but use an external key, e.g
ws-Relative-Key Pic 9(03) or whatever, in Working Storage. Your records 
are stored using that ws-Key pointing to a 'physical' position in the
file. The disadvantage of relatives can be that if you write your first
record giving it key = 100 then you finish up with a chunk of space
where records 1 through 99 would fit.

I don't know under what circumstances you would want to do what you are
asking about, but if it is a test, here's the pseudo code.

  ws-RelKey	pic (03).

  first make sure your Sequential and Relative BOTH have file-names
  initialize ws-RelKey
  Open sequential as INPUT
  Open relative as OUTPUT
  perform read sequential until end
  move sequential-info to relative-record
  add 1 to ws-RelKey
  write relative-record
  AND close both files before STOP RUN

Now here's the problem - there is no index or key system for the
Relative-File - if you want to get at Record-22 you've got to know it is
Record-22 -

move 22 to ws-RelKey 
Read relative record,
change it then rewrite it.

Now take it from a topic with which you will be familiar. Your Registrar
has a file of students. Assuming there are only ever 100 then he
allocates the next number. Somebody leaves, he deletes their info and
re-uses the number. But that's no good, because he may need to check up
on Student 22 = Was Joe Blow and is now 22 = Mickey Mantle.

Doesn't work. And in your case let's assume 4,000 on campus at one time,
Registrar allocates next number and keeps on going but he does it for an
Indexed File. If he did it with Relative the file gets HUGEeeee... and
even if he deletes records from a Relative the space is still there.
Don't know how Registrars do it but suggest they probably have a file
covering current students and probably hive off the dead-ones (those no
longer at the university, not just the ones in the cemetery), and if he
gets a query, refers to the old 'condensed' history file.

As I said - check the text books - but hope above is a reasonable intro.

Jimmy, Calgary AB
