### Libraries (and/vs) Native Language support

- **From:** "William M. Klein" <wmklein@ix.netcom.com>
- **Date:** 1998-08-26T00:00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<6s1u0s$6vp@dfw-ixnews8.ix.netcom.com>`

```
In both the Perl and C discussions comparing them to COBOL, there have been
some side comments about "libraries" vs native language support for certain
features.  One post explicitly said something about how "stupid" it would be
to put some things in the native language that could/should be in callable
libraries.

This is an "interesting" debate and one that I think is "emotional" with
minimal chance of ever convincing a believer in one that he/she should
change to the other.

For example (and there are MANY other examples too), one of the "strengths"
of COBOL is its native language support for "Indexed files" - and this
support is required for any COBOL compiler and run-time (that claims to be
Standard) - regardless of what products or operating system support there is
for "indexed files" outside of COBOL.  Furthermore, that language is truly
portable and MUST work the same regardless of what product (COBOL, operating
system, or separate file product) is providing the underlying support. To
most COBOL advocates this is a real advantage and is much better than
requiring a separate library for such file handling (much less a separate
library for accessing each different type of indexed file).

On the other hand, today (and in the next Standard), Standard COBOL relies
on a call interface to provide GUI/Windowing screen interfaces.  Some
implementers have added native language for this (most notably AcuCOBOL) but
most have not.  One of the places that COBOL is changing between '85 and
2000 is in providing Standard support for all the features needed to access
such API's - no matter what system they were originally designed for.  (This
includes adding standard pointers, procedure-pointers, return by value,
recursion, etc).  In this way it is hoped/expected that COBOL applications
will be able to take advantage of any library of callable routines that can
be used by other languages.

There is, however, still a question about when new native language should be
added vs when a call interface is acceptable.  In the history and design of
COBOL, the general (with some notable exceptions) guiding principal has been
to ADD native language when it is expected that the same "functionality"
will be desirable in a wide variety of applications and platforms and to
rely on outside API's when the interfaces are to operating specific or
platform specific features.

Bottom-Line: If you think that having minimal native language is best and
using callable libraries (APIs)  is the way to go, that is a reasonable
view - for a language that has grown up that way.  However, the goal of
COBOL is to provide the most portable source code that separates the
application logic from the environmentally specific implementations of
universally available functionality.

P.S. I have not mentioned the arguments of what you need to do when your
"interface software" changes, for example from one ISAM product to another.
Do you recompile? re-link-edit? re-bind? re-something else? run with a
different run-time library?  Again, I don't think I will ever convince
people who are used to one to that the other is better?
```
