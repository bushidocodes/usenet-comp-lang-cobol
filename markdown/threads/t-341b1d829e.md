Peter E. C. Dashwood wrote, concerning a least-recently-used algorithm:

> The algorithm maintains a count for each buffer and increments it when the
> buffer is used. When a flush is required, the buffer with the lowest count
> is used (in the case above, it's address is returned with an indicator so
> the caller simply loads into it. The internal count is set to 1 (not zero as
> you might expect) in the expectation that the buffer will be used.

Maybe I'm missing something, but it looks to me like the algorithm above
does *not* pick the least recently used buffer -- it merely picks the 
*least used* buffer.

Any buffer which has been used enough times becomes effectively immune
to flushing, even if it is never used again, just because its counter 
got big.

Likewise you might have just used a buffer a few nanoseconds ago, but
since it hasn't been around very long it hasn't accumulated much of a 
count.  In fact the buffer to be flushed is likely to be the one *most* 
recently used.

To me the most obvious way to implement an LRU algorithm is to use a
linked list of buffers (probably, but not necessarily, a doubly linked 
list).  Taking the example Mr. Foskey brought up, where he caches some 
number of customers:

1. When you search the cache and find a hit, transfer the customer to
the tail of the list.

2. When you search the cache and don't get a hit, discard the customer
at the head of the list, fill that buffer with data for the new 
customer, and move it to the tail of the list.

With this approach, the buffer at the head of the list is always the one
least recently used (apart from the initial loading of the list) -- it
got there by working its way through the ranks without being used.

Variations are possible in which you would might remove a buffer from
the list altogether, at least temporarily, in order to protect it from 
flushing. In fact that's how Unix manages its system file buffers.

Linked lists are more typical of C, but they're entirely possible in
COBOL.  If you're comfortable with pointers and dynamic allocation of 
memory, and you're not worried about the portability problems these 
techniques probably entail, by all means use them.

Otherwise you can implement a linked list within a conventional COBOL
table.  Declare each table element to have one or two members used as 
subscripts or indices referring to other table elements.

For more details, see the Data Structures section of my website.

Ah.  Yes, that would work.  Whenever you need to pick a buffer, you
scan the buffer pool and take the one with the lowest counter.

>    This is not as elegant as a linked list, but works well enough.

...and it would be easier for a novice to code or maintain.  For a 
smallish cache of a half-dozen buffers, this approach is probably the 
way to go.

For a larger cache, a linked list has an advantage in performance 
because it operates in constant time.  The buffer to pick is always
right there at the head of the list, and all you have to do is juggle
a few pointers.  Using a counter, by contrast, you have to scan the
entire buffer pool to look for the lowest counter -- unless you code
something more complicated than a linked list would have been in
the first place.
