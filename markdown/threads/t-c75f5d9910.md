MF have kept their _Unix_ prices for Linux, presumably because they
think of Linux as a form of Unix.

However, most Linux users see this OS as a desktop system that can
run server facilities rather than as a multiuser box.

Typically a Unix system has a central server with client machines
that run from dumb 80x25 terminals through to Windows machines.
The Unix box is seen as if it were a small 'mainframe'.  The cost
kept it in that position while DOS and Windows were seen as
cheap.

Now that MS is pushing Windows 2000 as servers and clients (the
Edsel of Operating Systems) this will become the expensive 
option as MS try to hang onto their 'brand loyalty' pricing
model (when brand loyalty has now been broken) and pushes up
the costs through draconian licencing and extra licencing
requirements.

The Unix/Linux market will become the lower cost option as 
lightweight single-function servers on cheap hardware reduce
the TCO by almost completely eliminate administration costs.
eg see Cobalt.

With hardware costs falling the cost of MS software exceeds the
cost of everything else.  Thus hardware vendors see their
profits eliminated while MS rakes in most of their revenue.
The solution for these vendors to compete is to build their
own solution based on Linux for everything from PDAs to
mainframes.

Ths will bring Linux to the corporate desktop either through
server based terminals or as desktops.

On the current desktop systems: DOS, Windows, OS/2 I can distribute
licence fee free using MS Cobol (old), MF Cobol or Fujitsu.
Accu provide added value by catering for multiple distribution
targets from a single compilation system, thus though there
are licence fees, they provide advantages to the developer
by widening his target market to hundreds of platforms that
are somewhat captive.  ie users that cannot or will not buy
shrink wrapped software.

With MF on Linux it seems that I would need to buy an expensive
10 user runtime for each machine.  Where I would sell software
for a small single user site, or SOHO, MF now wants _more_
for the apps server than I would charge for a simple system.

Of course I could put the DOS system under DOSEMU, or even a
Windows system under Wine, but MF does not yet 'get' Linux.

With your pricing structure MF will simply miss out on the
Linux market.  Your sales figures will show that 'Linux is
not selling' even as they take significant market share.

If Fujitsu (or some other) comes out with a pricing model
that matches their Windows pricing model then they will
take the complete market.  (unless one of the free ones
comes through with a usable system).  Alternately the
Cobol programmers will move to Java, or Python, or ..
and the Cobol market will die in the next iteration of
desktop and user machines (where this has progressed from
CP/M to DOS to Windows 3.x to Win95/98 and has staggered
around OS/2 then NT and will next move to Linux rather
than the Edsel of 2000).

If you recall the Edsel, it was built on the basis that it
incorporated _everything_ that anyone had said they wanted
in a car (just like Windows 2000).  Where it failed was that
noone wanted it all in _one_ car (as is W2000), they wanted
some features in a car for commuting and different features
in a car for the wife to take kids to school/shopping, and
different features in a sports car.

One thing that I want to _not_ have is a Windows 2000 PDA
that has multi-processor support and data centre features.
Another thing that I want to _not_ have is a departmental
server that can runs games.

With Linux I can have the same system that is cut down or
built up to suit the application and the users.  This makes
the scope of Linux from PDAs to server farms with simple
desktop NCs and workstations in between.  MF can't see this
yet.  Linux is the new DOS and Windows as well as the new
Unix.
