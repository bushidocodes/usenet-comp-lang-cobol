```
Two weeks ago I attended a meeting where the user wanted to load some
information from purchase orders into a tracking system.  Several months ago I
had coded up the programs that load data from purchase orders for my division
into the same system.  It turns out that the purchase orders for this division
are also on the file I use I just ignore them by checking for the code that
distinguishes my division from the others.  It would therefore be logical that
I would remove the check from my programs and load all the data into the
tracking system.

Needless to say that's not the way things came out in the meeting.  No matter
how many times I emphasized that the process was already in place it was
ignored.  Things like 'mainframe is going away'  'we can do it faster' (from
the head EDI guy - more on him later) were bantied about until I just figured
I'd shut up and let them hang themselves.

The other day a frantic e-mail from the user came across my screen - hey - is
anybody doing anything?!  Nope - not here - ask the head EDI guy (who is gone
until Monday and hasn't returned any other e-mails from the user).

I wandered over to the desk of the EDI programmer this afternoon and asked if
his boss had asked that he work on the extraction of the data to load into the
tracking system - hey if he could do it for that division he could do it for
mine - same file after all, and the mainframe is going away, and yada yada yada
. He said yeah but he couldn't give them all the data they were asking for -
figured I was pulling some things off other files.  Nope - I'm not - all the
data is coming from the purchase order files (850's and 860's for you EDI
people).  I said it's your file I'm using (he formats the EDI data into a flat
file I use on the mainframe for several systems).  Well he says - he can't
extract it, the data just looks like a jumble to him. Cobol can parse it out
the way the user needs it.

So we peons sat down and determined that Cobol could do it better and I believe
this should be done in the next day or so.  Seems that the other division's
purchase orders have a few 'quirks' in them that I need to code for.  Nothing
Cobol can't handle :).
```
