I have been using lower-case for all reserved words, and upper-case
for all identifiers which I have declared. I find mixed case far
easier to read than all-CAPS -- I think our natural pattern-recognition
mechanisms have more to work with in lower case (words have more
distinctive shapes and thus are more easily recognized at a glance).

One simple rule is that you should try to follow _some_ consistent rule
so that the case is not sending confusing messages.

To me using lowercase makes a program easier to read, but that may be
because I program in other languages also (Clipper, Perl etc) and they
all allow lowercase.

To hard-core COBOL programmers using lowercase may seem strange and
perhaps more difficult ot read, but I think this is just case of
getting used to.

I think it is more important that you set clear rules of how to write
the source so that programmers can find there way easily because they
recognize the parts of the program and know the purpose of certain
fields. Offcourse these rule can include using lowercase. For example
you could use uppercase for the reserved words and lowercase for the
variables.

I used to type the program out in any old case I happened to use so that
generally meant all lower case. I then stuck it through a text filter program
to tidy it up so that it was initially caps. ie.

Move Data-Item To Other-Data-Item

The filter was easily modified for whatever style you wanted though. I
excluded some reserved words such as 'to'.

That worked ok for my code from a blank sheet but if I was modifying someone
elses I tended to type all in lower case and not touch other peoples style
straight off. Most people I found used lower case or initial caps style with
only older mainframe programmers sticking to upper case.
