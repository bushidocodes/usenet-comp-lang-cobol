Were using CA-Realia COBOL beta 5.0, and have come across a perplexing
question. Our application runs in protected mode. However, we need to access
some network routines from a library file (Banyan Vines) that only support
REALMODE. Since the library talks directly to the NIC card in the workstation,
we have no way of making it (network library) protected mode safe.

Now, we could launch a REALMODE application to do our network calls, but we
would rather try to find a way to have a REALMODE DLL running in a protected
mode environment, and allow us to do standard dynamic calls to the DLL.

Anyone have any experience or suggestions on this?

First your idea of putting the real mode code into a dll to run in
realmode within the context of a Dos extended app. This is not
possible with the Tenberry (Rational) DOS/16M extender that CA-Realia
Cobol uses. Some 16 bit dos extenders such as Phar Lap's support
realmode dlls within protected mode apps but this is mute as CA-Realia
is tied to Tenberry (OS/2 emulation and other Realia runtime issues).
I am not knowledgable about the particulars of Banyan but if this is
similar to Netware the libary is a C based wrapper to call functional
via interrupts. This brings up a couple of possibilities you could
contact the Tenberry people and see if there is a port of this libary
that will run in the DOS/16M environment (unlikely) . You could
contact the Banyan people and see if they have such a port (unlikely).
You could license the source to the libary and port it (good luck).
You could see if Banyan low level interrupt based interfaced is
documented. I know that Ralf Browns interrupt list contains interrupt
information on Banyan. If Banyan publishes there interrupt based API's
you are still not out of the woods as you will be doing a lot of
DOS_PM_POINTER, DOS_RM_POINTER, and DOS_RM_INTERRUPT
and you will probably find that many of the high level libary calls
translate to a great many low level interrupt calls. Hope fully you
will not require any libary routines that are totally implement in the
libary.

Good luck

Yes it depends on the version of CA-Realia COBOL you are using in
4.2xx versions you can use DPMI (int 31h) calls. But these are really
akward from C and you really gain little over using the DOS_xx calls
via a COBOL stub with a C linkage. As you know with inline assembly
from C you can call CA-Realia runtime calls by building the Realia
linkage manually. If you are using Realia 5.0 as you stated in your
first message you could call the C based Tenberry (the company was
known as Rational) D16 libary calls. In the Realia 5.0 version the use
of the D16 dll is technically possible as the new loader stub used by
Realia is the one from Tenberry so the private DOS/16M kernal areas
required by the D16 libary can be resolved. If you do not own the
Tenberry DOS/16M SDK you may not be aware of the D16 calls which in
short is a libary for using the extender. The D16 libary calls are
mainly documented and targeted toward C development although the
libary is really language independent. You will find that most
software libarys (communication etc) that you buy on the open market
that claim compatibility with DOS/16M require the calls in the D16
libary. This means that the D16 libary is sometimes usefull to a
CA-Realia COBOL developer. I have tested minimally the D16 libary in
the Realia 5.0 environment and it appears to work although I did not
test the calls you would be using.

If you don't already own the DOS/16M SDK I cannot really recommend
buying it just for this purpose. At the time we purchased the SDK it
ran $5000 per seat and I think we pay $540 a quarter for maintenance
(off the top of my head). I understand that Tenberry has a cheaper
version of the SDK that costs less than what we pay for maintenance
but I do not know what it includes. Further as you are probably aware
Tenberry has a aggressive policy on royalties. If you use any part of
their SDK such as the D16 libary dll you will have to negotiate a
redistribution license with Tenberry. This libary is not covered under
your redistribution agreement from Realia. This means that if you ship
a commercial application you will have to give a percentage of your
gross (on each product) to Tenberry or negotiate a large one time
payment that covers you indefinitely on your product (every product
you ship has to have its own redistribution agreement). I would
suggest using COBOL stub routines...
