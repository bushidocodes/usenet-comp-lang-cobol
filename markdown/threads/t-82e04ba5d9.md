Sure.  The advantage of having reusable code is much smaller than advertised if
you only use your own objects though.

Still, it can be a pain with any layered technology, to drill down to where a
problem exists and fix it.  I discovered this with Paradox for Windows and C
before I ever tried OO.  Sure, COBOL doesn't eliminate this problem, but
generally, one can do a search in a COBOL source code module for a variable
string and have a high degree of confidentiality that he finds what he's looking
for.

On the other hand, the days of simple solutions are gone.  Even with a mainframe
computer with tight controls over the data of record, we have people at remote
sites entering data on PCs, cash registers, sensors, web servers, remote
databases, etc..   Monday a COBOL programmer lost a day's work here because the
remote site personnel claimed that there was no way that their new firewall made
their cash machines stop being able to communicate with the mainframe.  They
were wrong.  We don't have our simple integrated world anymore, so we have to
encapsulate procedures and data as they go back and forth.

OO is becoming more and more necessary even though it is a lot more work than
what we had before our world was so complex.  So we have to learn it.  But that
doesn't mean we should implement it for simple solutions.

At first blush, yes it does seem like a lot of hard work. Judson touched
on development saying OO is currently at 1.5. I would suggest I started
at -2.0 and worked my way up from there !  As I use it more and more I
become daily more comfortable.

The real issue is in the planning - and that's not particularly
different from structured :-

- (a) what to store, where - DB system ( in my case flat files )
- (b) what to do with it - editing/processing programs
- (c) how to display/get data - screen-section/windowing etc. (And must 
I say it again - OO is NOT about Windows !)

The planning is of course (a) and (c) - with (b) giving the reason for
the other two to exist.

DATABASE :-

As I've indicated previously, I have a common template for flat file
handling :-

- 1. the Calling program(class) - Edit, calculate, print etc.

- 2. A DB interface, (primarily has methods to create
collections/dictionaries used for listboxes/droplists),  and,

- 3. The specific file-handler, (all the file VERBS). 

I am now able to generate a new file and all its handling in some 30
minutes (and that's assuming that the file structure is right and
holding the data I need to access).

PROCESSING :

Editing/Processing - each is unique to a specific process in an
application, but can also call (invoke) reusable code like date
checking/formatting, imperial/metric conversion etc. from pre-canned
classes.

WINDOWING :

Screening/Windowing - and this is where I started at -2.0 but I'm
getting very quickly to Judson's +1.9. Having struggled with GUI classes
and invariably doing a 'window' in a different fashion as each one came
along, I came to realize that there was much in common. (At the current
time I don't use all the gizmos but restrict myself primarily to -
entryfields, the odd text label, pushbuttons, radiobuttons, listboxes
and droplists). Thinking on it, (Ken Mullins is doing a similar thing
with Windows APIs, and some prodding from Ken Foskey), I now have a
common template (Mydialog) which generates dialogs by passing a series
of parameters from the invoking class. One routine, like file-handling ,
to cover all dialogs. The next stage is to subclass MyDialog, (which
does the bread-and-butter stuff), to handle two-dimensional entryfield
tables.

The real gain in OO COBOL will come from common classes, (and as Donald
stresses), with the ability to pass objects between say, Fujitsu and
NetExpress, or any other OO compiler. As Bill has indicated J4 has
'common classes' as a priority after the WG4/J4 get-together in Newbury
in May. This common class format is PARAMOUNT to success of OO within
COBOL. Simple example - if I can write the following in NetExpress :-

	*>--------------------------------------------------
	Method-id. "display-hello".
	*>--------------------------------------------------
	  Display "Hello World"
	End Method "display-hello".
	*>-------------------------------------------------- 

	OR :-

	*>--------------------------------------------------
	Method-id. "get-date21".
	*>--------------------------------------------------
	Linkage section.
	01 lnk-date21			pic x(21).
	Procedure Division returning lnk-date21.
           move function current-date to lnk-Date21 
	End Method "get-date21".
	*>-------------------------------------------------- 
	
then the above limited syntax, ( no 'red tape'), for portability, MUST
work in other COBOL compilers using OO.

We are very much at the beginning with OO COBOL - but as more folks get
involved, common distributed classes, (with the source), will allow us
to make speedy progression to Judson's Stage 3.

Jimmy, Calgary AB
