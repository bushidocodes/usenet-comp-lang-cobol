I just love to dispel such folklore issues of how this is
better or that is slower.  A few months ago I wrote a series
of articles under the title "Application Tuning" for the
Wang e-zine, HotVS.  If you want to read them go to www.hotvs.com,
hit the [archives] button, go back to the start of the series in
November.  The main jist of the series is how much savings can be
achieve using different techniques.  One comparison is index vs.
subscripts.  The overall theme was code lookups from a file and
by pre-loading a table and doing indexed searches it was possible
to lose 99.6% of the fat.

It excites me that you examined the difference in index operation
on a VS and the AS/400.  The method of implementation on the
Wang VS is the norm (using offsets) but then the AS/400 is a
strange bird.

That was just the example used to get people thinking
of ways to make their apps run faster.  The first was
to avoid I/O by doing lookups instead.  Then to look
at how to make the lookups more efficient with the
objective being to get them to use SEARCH ALL where
appropriate.

The examples read the entire file into Working Storage
and worked from there.  In a later series I'll address
caching lookups.  Search WS first, if not there read
from disk and stow it away for future use.  I've been
able to get some pretty good results this way.

Unfortunately the responses I get from some programmers
is, "Oh, but then I have to think too much."

Seriously, that's a quote.

Anytime I can chop 95+ percent from the runtime by just
shifting from record lookups to tabling and SEARCH ALL 
I'll do it.  I doesn't take much thinking and not much
typing either.


But the thread here was "are indexes faster than subscripts".
Yes they are.  Are they worth the tiny bit of effort?  Yes.

In a situation where data must be looked up thousands of times more
frequently than being changed, sure, use a sort procedure to input the data
and build it into a sorted table to support a Search All. Or if it's a lot
more data, sort it and load it into an indexed file via JCL and just use
indexed read access in your program. Neither has anything to do with
indexes vs. subscripts except that the Search All uses indexes
automatically. For truly random access in a program, subscripting is
probably more efficient than indexing and is a lot easier to read.
