a project directly from the command line without having to use cobol and
cbllink, using all the build settings defined in the project.

example: projmake -build myproj.app

The advantage of being able to build from a command line is that these
command lines can be included in a makefile as well as the dependancies
that will indicate when it is required to be actioned.

Quite often some shared code is enhanced while working in one program
which may, for example, change the way it interfaces.  Developing by
using whatever is best for the programmer in the way of file managers
and editors (rather than what is mandated by the IDE) and then using
make, in another task, ensures (or should do) that all affected code is
recompiled without the programmer having to sit and watch while clicking
around the system in the IDE.

Set up Projects A, B, C and D in four separate directories(folders). Real simple

each program - just two values I don't use.

ProjectA - ProjectA1.cbl and ProjectA2.cbl
ProjectB - ProjectB1.cbl and ProjectB2.cbl

Realizing that copyfiles could be required by any 'sub-project',  Project C is
just a list of copyfiles - means in projects above, specific path required, e.g.

    copy "\cobol\examples\projC\copy1.cpy"

Did  a Rebuild 'ALL' on Projects A and B.

Set up Project D - first program is ProjectD0.cbl which calls above four
programs

the four files got physically copied into the Project D directory. Created an
additional  ProjectA3.cbl  - which shows "via absolute/relative path" - i.e. it

    F7 - gives you items 1 and 2 below
    Ctrl + F7 - Compile this program - *** that's easy enough to remember

    Alt + Ctrl + F8 = Rebuild ALL

To my mind, the biggest weakness of IDE's (the ones I have used) is that
they are not iterative.  If I write a batch file, I can call it in another
batch file.  If I write cobol code, I can put it in a library, and call it
from cobol.  If I design a project, I should be able to use it in my next
project as a single word.  I cannot.  I have to copy it, and if it changes,
then I have to change the copy.  If I re-use it 1000 times, then I have to
change it 1000 times.

Hell, even the Cobol copy statement can be nested, and it has been arround
for 35 years.  My biggest "rebuild all" is about 40 minutes on a 1 gig
machine.  141 dependencies would take care of the first level.

The second biggest problem is their reliance on shitty editors.  I want an
editor that can do a global data search, in hex, while I am editing a
specific source file, in ascii. Being able to efficiently search at least
two or three thousand source files for a data name in a minute or two is a
minimum requirement.  Just using the Fujitsu editor would cost me a couple
of hours a day.

Donald

In all the cases where I have worked on commercial software the number
of files involved in the complete package is quite large and
invariably some batch process is used to compile and link them. In
these cases the IDE could be used, I suppose, but if you want to
perform the build as part of a process (backup sources, build, move to
release preparation area etc.) some sort of command line still seems
to be a preferred option (it may not be the *only* way but it's easy
and it works).

In addition with a couple of hundred files the time to "rebuild all"
may not be excessive but it's longer than I would choose to do staring
at a NetExpress IDE. Most developers that I've known will have a
Project(s) that reflects what they are currently working on and let
some separate task run the full system builds. I also have suspicion
that there are practical limits on Projects after having tried to get
dependencies set up for a few hundred source files a year or so ago
and giving up when it was still analysing dependencies the next
morning!

But, ultimately, it's all down to personal taste, I suspect.
