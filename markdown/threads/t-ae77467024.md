And very thought provoking comments too. The whole question of the value of
4GLs hinges around whether or not they really make things "easier".

It is also true that we (as programmers) may need to adjust our mindset in
order to get the best from them. We are used to having control and doing
everything; along comes a "tool" which says: "Sit back and let me do that.
Look I can increase your productivity 3000%..." When we do so, we find that
the tool's solution is far removed from what we would have done and the
general reaction is one of shock...

Hence, I suggest that the evaluation of such tools comes down to the
following:

1. Do I have to maintain what it generates? (Responsibility for the
generated code rests with the tool in my book...)

2. Does what it generates work correctly? (without requiring "tweaks" from
me).

3. Does it actually save me time and effort? (setting aside any personal
bias, when I do actual quantitative analysis of time spent, do I come out
ahead?)

4. Do I feel comfortable using it? (If my mindset is such that I cannot
trust it and will never feel comfortable with it, then it is academic
whether it is a "good" tool or not, I'll just never be happy with it.)

I submit that any tool which passes the above tests is OK. Otherwise, I
stick to what I do...

The reason that there has not been a lot of clamouring for 'conditional
compilation' in the way that, eg C, has it is that there is no (or
little) need for it.

In C its portability was endowed primarily through conditional
compilation.  That is a C program of a few years ago was 'portable'
because the differences in the compilers, environments and operating
systems was catered for by having several sets of different code in the
source file, the appropriate set being chosen at compile time by the
conditionals.

With Cobol the portability is endowed primarily by not needing any
differences in the source code.

Where there are differences it is usually possible to cater for this at
run time.  For example I have source code that accesses system functions
by performing or calling routines that are in COPY books.  The programs
have flags available to them that indicate the compiler used and the
operating system and these are used to execute the particular routines
that are appropriate.

Rather than conditional compilation, I do much the same by conditional
execution at run-time.
 
For example to get a list of files I perform 'GetFile' with a 'first' or
'next' flag.  This will actually CALL 'UGETFILE' or 'MGETFILE' or
'CGETFILE' dynamically depending on which operating system is being run
on.

C doesn't (generally) have uniform data items, or dynamic calling, or
(until recently) standards and so it needs mechanisms to resolve
differences at compile time, hence it needs conditional compilation
where Cobol does not.

While I agree with the rest of your post, Nathan, I take mild issue with the
above.

Many years ago, before PCs were invented, it was considered almost
impossible to port code between mainframes. Furthermore, every few years a
new release of database software, screen handling, or Operating System meant
that all existing applications had to be converted.

This was costly and tedious and one day (for a bet, originally) I suggested
that if we layered our systems and separated the Application code from the
System dependent code, we could probably convert not only to a new OS
release but even to a completely different mainframe. A bet is a bet, and in
the next 2 years in my spare time I wrote such a system. All system
dependent calls were to a standard interface called "ACE" (Application
Common Environment). Instead of embedding IMS-DC or CICS calls in the COBOL
code you simply called ACE. Instead of coding DL/1 calls you called ACE.
Instead of handling ISAM files, you called ACE. You could also get ACE to
store stuff across dequeue boundaries, do some basic Task Management, obtain
memory for you and terminal dependent buffers, check security and logons and
even handle abends in a consistent and standard way. The interface between
ACE and the Application was consistent and simple. Programmers with no
knowledge of IMS (for instance) could access and maintain IMS databases
without ever using a DL/1 call.

Obviously, the ACE calls were implemented by "system  dependent modules"
which were transparent to the Application Programmers. However, many of
these modules were also written in standard COBOL and could be easily
re-compiled for the target environment. We developed several successful
medium size projects using ACE, then ported one of these from the IBM 360
series to an ICL 1902. The ACE "ScreenMan" (Screen Management) subsystem was
converted from CICS to Driver beads for the ICL system, the Indexed files
handled by "DataMan"  (Flat file Data Management) were simply re-compiled
and everything else ("TaskMan", "StoreMan" and "PoliceMan" (Task, Storage,
and Security Management, respectively),  worked perfectly. NO Application
code required modification!

The system proved itself some years later when it was used for a Major
on-line implementation in Spain.

I was paid for the original bet, and also received a substantial sum for the
software (eventually). It was evaluated by British Aerospace whose technical
people loved it, but the bean counters couldn't see the necessity for it and
wouldn't authorise the expenditure....

Finally, when PCs came along I converted it to the PC, and to this day I
could develop applications on the PC which  would be immediately runnable on
any system which has an ACE environment (currently IBM mainframes and ICL
1900s...don't see too many of those around these days <G>). This makes it
perfectly feasible to design, build, and debug mainframe Applications
totally on the PC. Once deliverable they are simply re-compiled on the
target mainframe and run in the provided ACE environment.

I still have a screen painter I wrote which generates MID, MOD, DIF and DOF
for IMS DC, CICS MAP for CICS, or Screen Section in COBOL for PC, and all of
them can be handled by the same common interface (ACE ScreenMan),
communicating with the Application by means of an ACE interface block which
contains the Message buffer, an action code, a reply code and a few other
modifiers used on various calls.

Now, the point about all this is that portability need not be a myth. It is
achievable by consistent, layered, interfaces which separate Application
(Business logic) from System dependent code. The modern equivalent of this
is, of course, Component technology, which encapsulates methods and data,
and provides a completely consistent interface (COM or CORBA).

ACE is sadly (well, to me anyway...) overtaken by events and is no longer
pertinent to the PC environment, where it has been replaced by OO and
Components. It will take a little while before this occurs in the mainframe
world, but I believe it is inevitable.

I console myself with the thought that the fundamental idea I had 25 years
ago has come to fruition and we DO have completely portable code in the form
of Components. It is achieved by encapsulating  (separating) and
interfacing, almost exactly as I envisaged. I never once imagined that it
would be Object code, rather than Source code, though....<G>
