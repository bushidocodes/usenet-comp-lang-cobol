Some Background..
Where I work we use ANSI Cobol 75 on a Unisys V series mainframe. We
have some software for tranferring prints via a gateway PC to a Novell
LAN to be printed on the Line Printers. The prints are VERY large >30000
records and take ages to dump.

What I was thinking of is a program to compress (ZIP ?) the print,
dowload it, and then de-compress it on the LAN. This means the program
must be compiled for the Mainframe and PC.

There is a hardware sollution (which use ZIP format) but this costs
$40000 for the Mainframe card. !

Question. How good a ratio could you get using cobol (if it can be done)?
Is there PD scource available?

Appreciate any responses..
Gary de Beer
South Africa

Gary,
We did something similar on an NCR Criterion Mainframe. We wrote an RLE
compression routine, but in assembler (NCR's NEATVS) on the mainframe and
the network (TASM). The logic should translate to Cobol but I would worry
about the speed. Although RLE is simple the results on our reports was up
to 4 to 1, thats a lot of whitespace.

Our basic routine was to read a page from the spoolfile into an indexed
workarea than outputting by character with a lookahead routine. All
occurences of 2 or more characters are coded as a 2 byte binary pair, the
hi-order byte is the characters hex value + 128 (sets the 8 bit) and the
lo-order is the number of repetitions. As we processed each page we also
created a page index and up to 8 field indexes. The indexes were appended
to the front of the file before transmitting and stripped off at the
network. We're sending about 750,000 pages a month through this system
now, it's saved big bucks and the users love the index feature.

Good Luck -- Mike

: Could you explain how a specific language has anything to do with the
: achievable compression?

Very simple. If the language cannot do certain mathematical operations
such as XOR or bit toggling, certain compression algorithms which use bit
processing or which use XOR, AND or other binary bit-level tests, cannot
be performed.

Some of these may be emulatable using testing and addition, subtraction,
and/or division, and other mathematical formula, but then you have
additional overhead.

There is a version of a Squeeze format compressor for ZIP files which was
written in Pascal. In theory, it should be convertable to COBOL. Might
be worth trying.
