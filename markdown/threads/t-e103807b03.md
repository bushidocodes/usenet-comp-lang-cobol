On Thu, 2 May 2002 15:25:42 -0400, Michael Mattias wrote
> The biggest 'catch' is poor software design.

I most emphatically agree. Code tuning typically achieves a few percent 
improvement, and the improvement is linear. The major improvements are 
almost always the result of improving the algorithm. To get better than 
linear improvements you must improve the algorithm. The most general 
aspects of the design are the most important, because those are the 
most difficult to change later.

If you ever have to write your own sort, read Knuth first. I don't know 
how many times I've seen programs where someone had written a bubble 
sort -- because it's intuitively appealing. (For the uninitiated, 
bubble sorts are appealing but appalling -- not the worst sort 
possible, but close.) Once many years ago I slipped a copy of Knuth to 
a programmer who was running a program with a bubble sort -- I knew 
because of the console displays. He put in the code for a Quicksort, 
and it ran so much faster (5 seconds vs 15 minutes) that he added code 
to check the results ... which of couse he should have done to begin 
with, even with the bubble sort.

That said, I recently helped with a program optimization where the 
problem turned out to be that the code was initializing a 325,000 byte 
working storage area for every transaction, even though most of it 
wasn't used for any given transaction. Regained about 90% of the CPU 
time in a program which had been using I think 3 CPUs for a couple of 
hours every night for many years. Ironically, this program contained a 
bubble sort, which I immediately suspected, but it was NOT the problem.

Most COBOL programmers just aren't oriented to the kinds of tools 
needed to track down something like this. Most also aren't oriented to 
thinking about algorithmic complexity, a good reason to have analysts 
with some knowledge of algorithms review the program and system design. 
Most COBOL systems escape the worst problems only because in the 
applications for which COBOL is typically used, the most obvious 
algorithms are usually OK.

Edward Reid

I'm glad you added that "at first look" qualifier, because there is no
general rule which says there is any relationship between number of source
code statements and the size/efficiency of the machine code created by the
compiler.

Compilers exist to make it easy for the programmer to execute many machine
instructions  with few programmer instructions.

Look at your own code: what is more 'efficient?'

CALL BIG-ROUTINE
or
<the 200 lines of source code which make up BIG-ROUTINE placed in-line>

The 200 lines of source code, executed in-line, is more efficient - from the
machine's perspective. Same thing applies to COBOL verbs like READ,
INITIALIZE and ADD: the compiler expands each verb into some number of
machine instructions. How many? Depends on the compiler.

To continue the discussion, I think that optimisation is very
important, but that the original enquiry was too narrow in its
approach.  As others have mentioned good system design is the major
factor and is therefore arguably the starting point of optimisation
for specific tasks.  Having said that it is often the case that
optimisation is found to be necessary rather later in the development
process, or often even later, when the program or system is found to
be underperforming in the circumstances in which it is to be used.

I think optimisation is about getting the job done effectively at a
reasonable cost in the particular circumstances of the specific task,
for which the subsidiary factors to which optimisation is applicable
are as follows.

Time it takes to train staff to do their jobs reasonably effectively.

Time and cost to develop a software product.

Time a software product takes to run.

Other resources a software product needs for execution, e.g. disk
space, softward libraries, etc.

Time it takes to debug a program/system.

Comprehensibility of a program/system to those who design,
maintain/enhance and use it, the harder something is to understand,
the longer it takes to work upon it.  This is often as or more
important than the minimisation of the time it takes to execute.

Maintenance of staff morale in the provision of continuing career
development by improving their design and programming ability and
style, this may improve the retention of staff in an organisation,
i.e. those who know how the organisation's systems work and can
therefore debug problems, apply enhancements and develop associated
new facilities rapidly.

implement an algorithm which our customer had "invented" so solve a problem.

subtractions!  But my manager insisted that since the customer wanted it

algorithmically poor code our there.  And in general that is probably much
worse than a poor coding style.

But there is another side of the issue too: for typical business
applications, maintenace costs are the major expense, and the need to
quickly implement changes a important requirement.  Again, way back when in
my programming career I wrote a simple piece of code to replace a month
number by its name (I had an array of month names defined in working
storage) and I coded something like:

move month-name(month-number) to result.

IF month-number is equal to 01

    MOVE "February" to result.
etc..

would always be done (I could not even convince him to add the ELSEs to at

inefficient it was but my manager's response was:  "Is there any doubt in
your mind what that piece of code is doing?"  Of course the answer is NO -
so he said: "Then it is the better code!".

I would hope most programming managers don't take maintenance requirements
to quite that extreme - but I learned to agree that maintenance Must be
considered when writing a piece of code - and complexity of the code
dramatically affects maintenance costs.  So before you optimize the heck out
of what you are working on, think of the poor schnook who will have to work
on it next.

Just MHO

(IMHO) any such enhancement even less "required" - CONSTANTs and "arithmetic

simply have an occurs of

  05  SubscriptValue  Pic 99999.
        88 ValidSubscipts  Value 10000 thru 99999.
  05 MyTable occurs 89,999 times.
      10 EachElem  Pic X.
        ...
Move EachElem (SubscriptValue - 9999) to SomeOtherField.

With the next Standard, that "9999" can be defined as a constant (to make
maintenance easier) or can actually be another data item - if the value is
"determined" at run-time.

any training in programming -- which especially means those whose 
training has been entirely OJT -- often don't understand tables (which 
of course are called arrays in almost every other programming language) 
at all. A very simple static table subscripted by constants, maybe ... 
but table-driven code based on a static table will be even more foreign 
to them than a simple table with dynamic values.

Not too long ago, someone mentioned that some COBOL code I had written 
for them was rather complex. My answer was that complex problems often 
require complex code. They had contracted with us because their staff 
wasn't comfortable doing the task themselves. That's no reflection on 
their programmers, who are generally quite good -- they simply aren't 
trained in programming techniques. Of course, I explained the code in
