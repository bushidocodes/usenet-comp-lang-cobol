Kevin, to answer the implicit question in the first paragraph, COMP
storage is not standardized because the semantics of COMP is left to the
implementor. It need not even be binary, even though that is the case in
most compilers.

If this is part of your ICOBOL to RM/COBOL conversion, please explain the
format of COMP data in ICOBOL. Perhaps if it is the same as an RM/COBOL
binary type, you might be able to get by without converting the data by
using what RM/COBOL calls a binary sequential file (i.e. a sequential file
that does not presume that the contents are to be printed but rather one
that provides data transparency). If ICOBOL's COMP is the same as IBM's
binary format (big-endian; 2-, 4- or 8-byte allocation depending on
picture length) then this is equivalent to RM/COBOL's BINARY type (BINARY
is a 1985 standard numeric type, but it too is subject to the variations
imposed by the various COBOL implementors).

WRT your last paragraph: Why do you need to have the records the same
length?

I have been lurking on this thread while Uwe has attempted to help, but
please provide a better explanation of the problem you are attempting to
solve so that we can give specific advice that will work the first time.

No, it's not standardized. I've learned never to assume that something
was standardized unless I've actually read it in the standard for myself!
In the case of COMP, the ANSI standard explicitly says that it's not
standardized:

"The USAGE IS COMPUTATIONAL clause specifies that a radix and format
specified by the implementor is used to represent a numeric item in
the storage of the computer. Each implementor specifies the precise
effect of the USAGE IS COMPUTATIONAL clause upon the alignment and
representation of the data item in the storage of the computer,
including the representation of any algebraic sign, and upon the
range of values that the data item can hold."


: * THIS IS THE LAYOUT OF ACCOUNTS RECEIVABLE

[snip]

Because you have COMPUTATIONAL data items in a record, you
have another problem to be aware of: FILLER that your compiler
may be automatically adding in order to increase object-code
efficiency. Quoting again from the 1985 standard:

"Each implementator who provides for special types of alignment
will specify the precise interpretations which are to be made.
The use of such items within a group may affect the results
of statements in which the group is used as an operand. Each
implementor who provides for these special types of alignment
will describe the effect of the implicit FILLER and the
semantics of any statement referencing these groups."

So if you want portable files, you can't make any assumptions about
the size, format, or alignment of COMPUTATIONAL items in a record.

: If I go to display the redefines isn't going to work! (unless I counted
: wrong) and I prefer the data in two's complement rather than worrying
: about where the sign is!

COMPUTATIONAL will not guarantee that your data will be stored in
two's-complement format. If you want your data files to be portable,
you should stick with usage DISPLAY. Other formats (COMPUTATIONAL,
BINARY, PACKED-DECIMAL, etc.) are not standardized. Also, for
DISPLAY items, the compiler is not allowed to insert automatic FILLER
in a record.

As for worrying about where the sign is, not to worry! Take control.
Just include the SEPARATE CHARACTER phrase and specify whether you
want the sign to be LEADING or TRAILING.

In summary, if you want your data files to be portable, don't specify
any usage other than DISPLAY, and use a SIGN clause with SEPARATE
CHARACTER for any signed numeric items. That's about as portable
as you can get (assuming you don't have to worry about the character
set).

: Help anyone?

If you don't have control over the format of your data records,
you need to consult the vendor's manuals for the compilers you are
using. There may be some compiler options you can use to control
the alignment of the COMPUTATIONAL items. Also, there may be an
option to print a variable map from the compilation, which may show
the exact layout of the records in question. I can't help you
there unless you happen to be on an HP-3000 using COBOL II.

Good luck!
