The "COBOL Unleashed" authors are the best ones to ask about this.

But if I recall correctly, data in plain old numeric format (ex: S9(03))
uses only the last four bits for the 0-9 digit.  The leftmost four can be
used for other things (like a sign).  So, for example, if you write -68 to a
file as pic S99 and look at it in Fujitsu's file editor you'll see two 8 bit
numbers: 36, 58 (00110110, 01011000).   The last two nibbles are obvious (6
and 8), but notice the sign is in the last byte as a 5 (0101).  If it was
+68 it would be 36, 38...

In IBM I remember looking at hex Fs in the leftmost nibble (1111) for
positive and something else (C or D?) for negative.  Either way, that gives
you some idea of what's going on.

However, I think you have a different problem if you're trying to get the
sign from an ASCII number in VB after the fact.  ASCII uses the full 8 bits
to signify a digit and therefore the sign is separate.  If the conversion
process didn't put it in there somewhere, you're SOL.  If you're somehow
reading EBCDIC or something similar, then my explanation above should give
you some clue.  Just look for a known negative number and examine the first
4 bits in the last byte.

Each byte has 8 bits but in unsigned data, the value of the digit is only in the
last 4 bits, called nybbles.  With origins in the punched card era, the first 4
bits are the zone with the last 4 being the digit.  In signed numeric (unpacked)
data, the sign will be in the zone of the low order byte.  On platforms that
must pack the data (comp-3) to work with it, the zone and digit of the low order
byte are reversed while the zones of other bytes in the field are ignored.

Usually x'F' means unsigned, x'C' is positive and x'D' is negative.  E.g.:
+123 = 12C
-123 = 12L  In both cases, the byte containing the sign appears as an alphabetic
character.
