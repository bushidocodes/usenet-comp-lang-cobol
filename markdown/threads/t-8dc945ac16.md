9/041 is the _detection_ of an index that had been previously
corrupted. The corruption may occur when a write has been done but it
is usually only later when the record is read back that the corruption
is detected. Or it may be that the reading of some other record
entirely may detect the corruption.

- I take a dual approach to COBOL file handling. I'm using OO but that 
is not dissimilar to having a CALL to a program which specifically 
handles one file. Unless there is something oddball going on, in theory 
at least, the only two things that should cause a corruption are a 
REWRITE or a WRITE.

Firstly a copyfile which returns a result to my CALLing program :-

        *>----------sqlresult.cpy-------------------------------------

           (tag)-SqlResult                        pic 9(03).

            88  ResultOK                    value 0.
            88  FileResultOK                value 0.

            88  RecNotFound                 value 1.

            88  NoMoreRows                  value 2.
            88  FileFinis                   value 2.

            88  DuplicateKey                value 3.

            88  DataError                   value 97.
            88  CursorError                 value 98.

            88  FileError                   value 99.
            88  TableError                  value 97 thru 99.

        *> This copyfile is multi-purpose with Level 88s used for either
        *> COBOL files or (R)DBMS Tables using SQLERROR or SQLSTATE

        *>-------------------------------------------------------------

So my coding for REWRITE and REWRITE :-
------------------------------------------------------------------------
        Set ResultOK to true (this gets passed back to the CALLer)

        REWRITE Data-Record                         *> R-E-W-R-I-T-E
              from lnk-record (1:lnk-RecordSize)

        if   ws-fileStatus <> "00"
             set FileError to true
             invoke super "showFileError" using ws-ErrorParams
        End-if
-------------------------------------------------------------------------
	Set ResultOK to true (gets passed back to CALLer)


        WRITE Data-Record                         *> W-R-I-T-E
              from lnk-record (1:lnk-RecordSize)

        if   ws-fileStatus <> "00"
             set FileError to true
             invoke super "showFileError" using ws-ErrorParams
        End-if
----------------------------------------------------------------------------

If I have a File with Alternate Keys, then of course I have to indicate 
either <blank> or 'with duplicates'. Given I allow duplicates, then in 
the ws-fileStatus checks above :-

        REWRITE Data-Record *> ( OR WRITE )
              from lnk-record (1:lnk-RecordSize)

          if   ws-fileStatus = "00" or "02"  *> '02' = Dups allowed
               continue

          else set FileError to true
               invoke super "showFileError" using ws-ErrorParams
          End-if

----------------------------------------------------------------------------------

But note I have two checks on the transactions and ALWAYS check 
file-status unless I indicate an exception.

```
> the only two things that should cause a corruption are a REWRITE or a WRITE.

Or a DELETE.

But the problem is (and I have had problems with 9/41 errors) that the
write, rewrite and delete give a perfectly good zero status (in byte 1)
even when the update of the index has corrupted it. It is only later
when some record is read that the 9/41 occurs. In fact it may not even
be in the same area of the file.  I have had systems where, say, the
invoice number always increased so writing was always at the physical
and logical end of the file yet I would get a 9/41 when rereading a
record that was months old. Eventually we managed to get a different
version of MF Cobol which didn't have the problem.

The Index does get rotated from time to time to keep it balanced and
thus all parts of the index may be updated.

It is most likely nothing to do with the cobol program but is a
difference between the way files are handled between Win98 and XP SP2.
It may even be a 'bug' in MF Cobol run-time that simply did not cause a
problem on DOS based systems (such as Windows98). But NT/XP is not DOS,
even the command line is not DOS (as it was on '98) but is an emulator,
and the file system is not DOS.
```
