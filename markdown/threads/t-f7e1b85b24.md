Whether an OPEN EXTEND does or does not "create" a NEW file - if it doesn't
already exists - depends on whether you do or do NOT have the "optional"
keyword in your Select/Assign statement. Check this out in your Language
Reference Manual.


P.S.  For those of you who missed one critical note in this thread - the
people "with" the Tiny COBOL project have already admitted they have a "BUG"
in their run-time system and that the original reported problem is with
TinyCOBOL itself.

This does NOT change the fact that
 A) you shouldn't "Play with" the FD area while a file is closed
 B) the original program could/should have checked the file status after the
WRITEs as well as the OPENs.

However, even if both of these "changes in source program" had been done,
TinyCOBOL still wouldn't have worked correctly.

Just to correct something in my last post.

It is possible to get an "05" on an OPEN statement - but not on the OPEN
statement in the original example.  An "05" is described as follows (which
was NOT the case in the original code sample)

"I-O status = 05. An OPEN statement is successfully executed but the file is
described as optional and the physical file is not present at the time the
OPEN statement is executed. If the open mode is I-O or extend, the physical
file has been created."
