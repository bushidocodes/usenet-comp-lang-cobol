There are a ton of posts on this already, but here is my take (MVS
COBOL II only).

When a dataset is created on either DASD or Tape, it gets a dataset
label.  The operating system gives a DASD dataset a label
automatically.  I can't imagine why you would create a disk dataset
without a label, but there might be a reason.

However, for tapes in MVS, each tape can either have a "tape label"
written to it, or it can go without.  A tape label is a set of records
on a tape.  A "standard" IBM MVS tape label starts with a 6 byte
volume serial, followed by an 80 byte Header-1 record, and an 80 byte
Header-2 record (there are also "tape marks" and End of Volume and End
of Tape records, and a lot of rules about datasets spanning tapes, but
I'll skip that discussion) .  These records contained bottom-line
information about what was on the tape (dataset name, lrecl, recfm,
blksize, etc.).  Most of the time, having this data directly on the
tape facilitates tape management software, and is a good last resort
to look at via a tape dump utility called DITTO if you've got no clue
what is going on with a particular tape (label ripped off, from
offsite, etc.).  In some ways, it was middlin good for security (if
the sysadmin cared to control the ways around using labels).   Going
without was good for portability.  By specifying NL in the JCL
(non-labeled processing), an OS wouldn't even check for a label and
would just start reading bytes, which prevented the OS from choking on
some foreign label system that it couldn't read.

By specifying LABEL RECORDS ARE STANDARD, you are telling your program
to expect the dataset you are using to have a "standard" dataset
label.  For DASD this is pretty much pointless (or at least it is in
my shop).  All DASD datasets have standard labels (I've never seen
someone try to force a disk dataset to go NL).  For tape, however,
it's a real point.  Specifying LABEL RECORDS ARE OMITTED forces the
JCL to require NL on the LABEL parm.  Specifying LABEL RECORDS ARE
STANDARD forces the TAPE to have a "standard" label, if it doesn't
have such a label, your program probably will take some sort of open
macro error; and the JCL hadn't better try to say LABEL=(1,NL) when
the program is looking for something else.

The bottom line is, the information in this line can be specified in
JCL, which is easier to change than the program (usually).  Avoiding
the LABEL RECORDS clause in your programs can help avoid problems, and
can make change management easier.


In no way would either OMITTED or STANDARD affect file format.  These
labels precede the data and identify information about the data to the
OS.

Ken Foskey,

Technically, you are correct, however:

From Expert MVS/ESA JCL: A Guide to Advanced Techniques, page 11.

"The VTOC contains entries which are known as DSCBs (Data Set Control
Blocks).  There are several different types of DSCBs, the most common
of which is a Format 1 DSCB.  For each non-VSAM data set on a volume
there exists such a DSCB in the VTOC of the volume describing it.  It
is in essence the label of a disk data set."

I was thinking of these DSCB entries, but not actually describing them
to avoid getting into something that is pretty much in the hands of
the access methods, when I was talking about a DASD dataset label.
DSCBs come in various formats and do more things than tape labels,
like noting where on the volume a dataset's extents are.

>> DASD datasets do not have a label.  They have a catalog entry that
>tells

Wrong. All DASD data sets have a DSCB, which _is_ the label for a disk data
set. A DASD data set does not need to have a catalog entry (you can have
uncataloged DASD data sets, though I don't know why one would do that today).
If a DASD data set is cataloged, then it also has a catalog record (entry). The
catalog entry does _not_ "tell you all about them". The catalog entry contains
the data set name, device type (UNIT), and volume serial(s) (VOL). It is the
DSCB that has information like extents (where the data are found on the disk),
DCB information, creation date / expiration date, etc.
