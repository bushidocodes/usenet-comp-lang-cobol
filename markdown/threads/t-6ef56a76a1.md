If I remember correctly,  most machines (in my experience) store long
integers as bytes 1,2,3,4 but on pcs (and others) they store them as
3,4,1,2.  If you are transferring data without converting then long integers
will be mis interpreted.  I.e.  X'00FF' would be interpretted as X'FF00'.

I'm going to limit my response to only one suggestion because this topic can
go on and on and on . . .  Aside from stability, one of the strengths of
COBOL is that you have programs 20+ years old still maintainable.  This is
because there is a well defined standard and compelling reasons for compiler
vendors to comply with the standard.  Some major changes in requirements
have occurred over the years, GUI being one of them.  GUI has been around
for awhile so the requirements for a suitable interface is getting pretty
well defined.  If I could code a suitable GUI interface within the
strictures of ANSI COBOL I could probably rest assured this code would be
serviceable for a decade or more and I would get a good return on my
investment.  Somewhere, somehow, it usually comes back to $$$.  That's my 2
cents worth.

All of these techniques give results which are dependent on the
character set.  In ASCII, A-Z is nothing but upper case letters.  In
EBCDIC, such a range includes miscellaneous bits of trash.  Other
character sets may, in principle, be even more perverse -- what if
the alphabet is encoded in reverse order, or otherwise scrambled?

Offhand the only simple (but tedious) way to code such a range test
portably is to enumerate all the eligible characters, whether in an
88-level, an EVALUATE, or whatever.

I think the original wish was for some mechanism which was both
succinct and portable.

```
Given the recent discussions about the EBCDIC alphabetic characters not
comprising a discrete (?) series of hex values, I thought that I would let
the newsgroup know about an interesting discussion going on (not yet
resolved as far as I know) at J4 and WG4.

The question is whether or not the next Standard should require, allow, or
be indifferent to the question of whether the alphanumeric character set (or
as it is now known "repertoire") is a complete set of ALL the hex values
that fit into a specific "size" of data (usually a byte - but not required
to be).

In other words, can an implementor say that X'00' thru  X'10' and X'12' thru
X'FF' comprise the alphanumeric character set - but that X'11' is NOT a
valid alphanumeric character in their implementation?  Obviously (as
discussed in previous threads), the current "official" (little used) ASCII
character set is from X'00' thru X'7F' - but does *not* include X'80' thru
X'FF' - and J4 isn't about to disallow that.  On the other hand, no known
(or at least known to me) current character set has "holes" in the middle.

One reason that this becomes important is that the next Standard (like many
implementations) is supporting hex literals.  However, the question is
whether X'02'  represents a specific bit-pattern (as most of us would
expect) or whether it really represents the 3rd (X'00' is 1st) valid
alphanumeric character.  The problem with using the former interpretation is
that the Standard has never before talked about bytes or bits; the next
Standard is going to include "bytes" (without specifying how many bits there
are in a byte) - but so far (I think) has avoided any mention of bits.

Some of you wonder WHY it takes so long to get a "correct" Standard.  It is
this type of problem that takes so long to resolve in a truly portable way.
It is easy for any specific vendor to define hex literals - because they
know the byte/bit definitions of their operating systems - but when the
Standard tries to create a "generic" definition, this type of thing is
remarkably hard to do.
```

There is great value in knowing that what you code today will compile tomorrow and
the year after regardless of the operating system.  When you have to turn to 3rd
party tools to accomplish a goal you loose this value.  If you were able to do a

the xBASE languages would be non-standard.  When you use the non-standard
language, *OR* a 3rd party tool,  you are at the mercy of the compiler vendor who
has an economic interest in continuing sales.  Simply put, you can expect your
code to become obsolete as the vendor encourages you to upgrade.  They more or
less have to do this to stay in business, there's nothing evil about that.
However, as a developer I am primarily interested in getting the greatest return

Hmm. Well, without looking at the code, I am not sure what to say.
The first thing is, a pic variable on one mainframes compiler may be
different than a pic variable on a PC.

Thus,

01 somevar pic s99.

might have different byte orders. The next problem is it may be stored
in packed decimal format on the mainframe, and display or binary
format on the PC.

I'd start by looking through all the code, and replacing unknown
variable types with the default mainframe-compiler type.

thus,

somevar pic s999.

would become

somevar pic s999 usage packed decimal.

or

somevar pic s999 usage comp-6.

or comp-5 or comp-4 or whatever.

I don't really know what comp-number refers to what machine, but after
a few tries, you should find the default data-storage type.
