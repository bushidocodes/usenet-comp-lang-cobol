Howdy all!

If I wish to pass an item of data through a number of tests where a fail
in any of those tests is fatal for that item of data AND I wish to react
differently to each type of fail, is it better to have all the nested
loops in one routine or to parcel some of them out in called routines?

Here is a pseudocode example:

Read dessert-suggestion TILL no more suggestions
AT END
    MOVE true to no-more-suggestions
    SELECT dessert-suggestion from list-of-yummy-things
NOT AT END
   IF dessert-suggestion has ice-cream
       think about it...
      IF dessert-suggestion has chocolate topping
         think about it hard...
         IF dessert-suggestion is for two people
             think about it really hard...
            IF dessert-suggestion has no bananas
               Imagine what it tastes like...
               ADD dessert-suggestion to list-of-yummy-things
            ELSE
               SAY ï¿½This dessert has bananas!ï¿½
            END-IF
         ELSE
            SAY ï¿½I want something for two people!ï¿½
         END-IF
      ELSE
         SAY ï¿½I want chocolate topping!ï¿½
      END-IF
   ELSE
      SAY ï¿½I want ice cream!ï¿½
END-READ


Each of those tests might have one or two extra lines of processing in
either the THEN or ELSE section.

a) IF I keep the tests all in one routine I have potentially hard to
read code

b) IF I parcel some of those tests into called routines and use boolean
flags to signify whether a test has been passed, my code might be more
readable but I also increase the length of my code and incur the extra
processing costs of those extra method calls and boolean flag handling
techniqes, particularly if I am dealing with thousands of data items
(thatï¿½s LOT of desserts!)..


Is efficiency of greater importance than readability here?

*smile*

Robert.

Sometimes sacrificing some efficiency for functionality is better. An
example might be to check for as many errors as possible on a data entry
screen so that operators can correct as many as possible before pressing
enter again.


Loop
  Error-count = 0
  If error-cond-1
    queue error-msg-1, highlight bad field, etc.
    add 1 to error-count
  Endif
  If error-cond-2
    queue error-msg-2, highlight bad field, etc.
    add 1 to error-count
  Endif
  If error-cond-3
    queue error-msg-3, highlight bad field, etc.
    add 1 to error-count
  Endif
Exit if error-count = 0
  Beep and display accumulated error messages and error-count at bottom
  Reset highlighting and read screen again
Endloop


This is inefficient in that every condition is tested, but that's because
that's what you want to do in this case. It does avoid excess nesting, and
you still get a nice summary go/nogo condition at the end.

There are probably better examples, such as a compiler which should
diagnose as many errors as possible, since another compilation might be
more wasteful of system resources (and more importantly of highly paid
human time) than getting only one error message each time you press enter
on a data entry screen with lots of bad fields.

This example could be turned around, in case you want to quit testing and
fail immediately at the first error, by adding "and error-count = 0" to
each if. Inefficient in repeating that test, but in some cases it might be
better than more nesting. However in this kind of case it's usually much
better to use a case structure like Evaluate. As others have pointed out,
that's the best way to avoid excessively nested IFs. However, this is not
always possible if testing for some conditions requires more complicated
code and can only be done safely if other conditions have already been
ruled out, then some nesting is unavoidable. However, this can usually be
broken down into groups, e.g.

Evaluate True
  When safe-to-test-now-error-cond-1
    Response-1
  When safe-to-test-now-error-cond-2
    Response-2
  When safe-to-test-now-error-cond-3
    Response-3
  When Other
    code needed to set up tests for more complicated conditions
    (where this code might have been invalid in the other cases)
    Evaluate True
      When cond-4
        Response-4
      When cond-5
        Response-5
      When cond-6
        Response-6
      When Other
        etc.
    End-Evaluate
End-Evaluate


Each Other case might be suitable for a Perform as appropriate.

I can sympathize with this stance to some extent but many clients seem to
carry it to extremes and standards become a way of enforcing the lowest
common denominator of program quality by stressing conformance and
consistency to the exclusion of good coding practices. This only propagates
the continuing degradation of code with each change, resulting in
unmaintainable code that should be thrown away.

The correct purpose and application of standards is to raise the quality of
the code by encouraging good coding practices, and any resulting similarity
of code is a useful but secondary byproduct. As better coding techniques
become available they should be phased into use in some reasonable manner
at every opportunity, even as changes in old programs. This will tend to
leave each program in better shape, rather than worse, with each change.
