I would use GOBACK in all cases for the following reasons:
1.  If a called program ever becomes a calling program, the EXIT PROGRAM
statement functions like a CONTINUE statement (not good).
2.  If a calling program ever becomes a called program, the STOP RUN
statement will not return to the calling program, but will end the run unit.

The distinction is between CALLed program and MAIN programs.  Some systems 
make no distinction at compile time, only at run time as whether they have 
been 'CALLed' from another COBOL program or from the JCL (or equivalent).  

Other systems make a firm distinction such that a program compiled as a 
'MAIN' _cannot_ be CALLed and a non-MAIN can _only_ be CALLed and cannot be 
directly activated.

In such systems the programmer is _never_ confused as to which of EXIT 
PROGRAM or STOP RUN to use.

solution should be obvious, the behaviour is designed so that the two can 
be used as:

              program code
              ...
              EXIT PROGRAM
              STOP RUN
              .

If the EXIT PROGRAM does not work (because the code was not CALLed) then 
the STOP RUN will work.

> the EXIT PROGRAM
> statement functions like a CONTINUE statement (not good).

No.  This behaviour is _good_, it is designed that way, it allows the above.

Unlike GOBACK, it also allows code to be between the EXIT PROGRAM and STOP 
RUN that will only be executed if the program was not CALLed.

4.167 run unit: One or more runtime modules that interact with one another and

That's pretty vague (not to mention, optimistic). A better definition is found

"C.4.1.2 Runtime level organization
At the runtime level the most inclusive unit of organization is the run unit. A
run unit is a complete problem solution
consisting of a runtime module or of several inter-communicating runtime
modules. A runtime module results
from compiling a compilation unit. A run unit is an independent entity that may
be executed without
communicating with, or being coordinated with, any other run unit except that it
may process files and messages
or set and test switches that were written or will be read by other run units.
A runtime element results from the compilation of each function, method, or
program."

Suppose I fork five tasks and wait for them to complete. Are they five "run
units?" The standard appears to say they are not because they are certainly
"coordinated with" my task. Yet it wants each of them to terminate with STOP
RUN, in which case they _are_ "run units." Make up your mind. 

[helpful example]

"From each of the 3 compilation units, a separate runtime module is created by

C.4.5.1
"When control is transferred to a called program, execution proceeds from
statement to statement beginning with
the first nondeclarative statement of the called program. If control reaches a
STOP RUN statement, this signals the
logical end of the run unit. If control reaches an EXIT PROGRAM statement, this
signals the logical end of the called
program only, and control then reverts to the next executable statement
following the CALL statement in the calling
runtime element. If control reaches a GOBACK statement and the program has been
called, controls continues as
for the EXIT PROGRAM statement; otherwise, control continues as for the STOP
statement. Thus the EXIT
PROGRAM statement terminates only the execution of the program in which it
occurs, the STOP RUN statement
terminates the execution of a run unit, and the GOBACK statement returns from
whence it came, whether it was
another program or the operating system."

Here the standard explicitly says use GOBACK to get a generic return. That seals

Richard is correct and I almost said it myself that the difference is
between CALLed programs and MAIN programs.  A CALLing program may be either
a MAIN program or a CALLed program (or with the 2002 Standard either a
user-defined function or a method may CALL another program).

Speaking of the 2002 Standard, the use of GoBack becomes *more* "useful"
(and/or more ambiguous) in that it is valid to "replace"

   Exit Program (as IBM et al has always allowed it)
   Stop Run (again as commonly implemented today)
  Exit Method  (as IBM and Micro Focus document it today - and Fujitsu MAY
allow it)
        and
  Exit Function

As I stated in another thread, this means that if you have common "procedure
division" source code, you can code a GoBack and not worry about what type
of "compilation entity" uses it.  However, as others have pointed out this
is pretty unusual.

I have YET to see any good reason for NOT using GoBack.  I have seen several
reasons why it may not be NEEDED in some environments (where you MUST know
at compile time if you have a MAIN or CALLed program).  In the case where
you WANT to do additional "things" between a MAIN program's exit and a
CALLed program's exit (something I have never seen - but which is certainly
POSSIBLE), you can (using 2002 Standard code - where EXIT Program need not
be in a sentence by itself)

    Exit Program/Method/Function
        Perform  In-Main-Program
    GoBack
     .

And in this case one MUST use the "EXIT" statement.  As I have stated
before, I have worked primarily in IBM and IBM-emulation environments and
this simply is NOT something that I have had a functional need to do.  I
accept, however, that others may.

    ***

Concerning the Fujitsu (and potentially other vendor) requirement to KNOW at
compile-time (or at least linking time) whether a program (without a Linkage
Section) is intended to be used as a MAIN program or a CALLed program (and
to so mark it - so it can't be used as the other),  I have always felt that
this was NOT conforming to the ANSI/ISO Standard.  Others may know of such a
"get-out" - but as far as I have ever seen, the ANSI/ISO Standard *requires*
that a program (without a Linkage Section) *must* be usable as either a MAIN
program or a CALLed program (without parameters passed to it) - without
recompiling.

Therefore (IMHO), Fujitsu is simply non-conforming in this requirement.
That is NOT to say that there is anything "wrong" with a programmer knowing
at compile-time which type of program they are creating - but it is to say
that the compiler vendor REQUIRING this decision at compile-time is
NON-conforming (as far as I have ever seen).  I would be interested if
others can find some place in the ANSI/ISO Standard ('85 or 2002) that
allows this requirement.

I am not certain how to address this because we are talking
about something that applies to older products; but consider
the following.

As a developer using MF COBOL/2 3.2, writing for Windows 3.0,
I might include the following in COBOL.DIR.

OMF"OBJ"
TARGET"286"
DEFFILE
DEFFILETYPE"WIN"

Then in any main program I include the directive:

      $SET NODLL

NODLL means EXE. The default directive is DLL.

Thus when the code is compiled, a Windows definition file (.def)
will be created that is marked for either .exe or .dll. When the
object files are linked, the appropriate file, executable or library,
is created.

This arrangement, imposed by a developer, is no different than
a directive imposed by a development environment. Both are
required for creating COBOL applications for Windows.
Neither arrangement, necessarily makes the source code
non-standard.

'compiled' to .int intermediate code or to .gnt generated code.  When this 
is done the 'main' program is the run-time cobrun.exe or application server 
or somesuch (cobrun on Unix/Linux too).  

Another way is to produce .EXEs and .DLLs and use the static or the shared 
run time code.  For DOS these can all be .EXEs even when CALLed.  For OS/2 
and Windows the 'main' program is a .EXE and the CALLed programs are .DLLs.

If a 'real' Windows API program is to be written then the 'main' program 
may need to have an actual Windows message loop in it, so there is no 
chance of it not being known to be 'main'.

So, with MF it may be possible to compile without knowing, but it may not 
be possible to link without knowing, and this depends on how the 
application is built and what run-time will be used.

If the MS LINK is used then either compiler options are required or the 
linker must be told which 'statup' object to link in.
