PS. On the subject - or off - these are large VSAM file loads.  They
are never updated, only loaded and read.  I'm tight on space.  So I
tried some tricks.  There are two related files - an ESDS data file -
containing the detail information - large - and a KSDS data file
containing the key information for a variable range of records in the
ESDS data file - referenced by RBA.  Large system.  I'm not going to
re-engineer it, I just need to move it.  Anyway, I decided to play
with the VSAM file definitions to make it more efficient.  CI SIZE was
4096 - the records in the KSDS are 48 bytes - 40 bytes are key
information.  The records in the ESDS are variable length from 87 to
350.  Cool.  So I decided I'd used SPANNED.  Then I changed the CI
Size to 32K.  Figured I would waste the least amount of space that
way.  Then I loaded a test file.  It went from 250 Cyl on the ESDS to
255 - from 82 Cyl on the KSDS to 84.  Fine idea that was.  So  I
decided to just leave out the CISIZE definition and let VSAM do what
it will.  The physical records on the ESDS ended up around 16K - a
little less - but the utilization went down to 220 cyls on the ESDS
and 67 on the KSDS.  Interesting.  I left it spanned.  Now all I have
to do is test to see if the programs (ASM) can read the data still.
If so, I think I'll just let the system do what it wants from now on.
OS/390 2.5 and VSAM seem to be smarter than I am these days...

Thane,

I'm not quite understanding your post.  While I can vaguely see why it
doesn't have anything to do with the post it appears to be responding to,
I've got a couple of things I'd like to say.

One is: a 32K block size doesn't necessarily get you the best space savings.
On our aging 3390 mod 2s, the optimum blocksize in vsam is 18432 bytes, for
a total track usage of 98.2%.  4K only gets 87.2%, and 32K only gets 90.9%.
But these numbers, or so I have been led to believe, is pretty much device
dependent.
Modern storage arrays use virtual storage techniques that allow the array to
"appear" as any device or number of devices to the mainframe that the system
programmer wants, but actually stores the information where ever it feels
like it, so I don't know how this would impact things.
For the best results, check your device manual or with your friendly
neighborhood systems programmer.
