```
Personally, I think that the INSPECT with FUNCTION REVERSE is the most (to
me) self-documenting.  But I believe that in most systems where CLC people
have run tests, a PERFORM VARYING *performs* better. (Also if you are
running with a pre-Intrinsic Function compiler, you will need to use PERFORM
VARYING). We have had this discussion several times in the past and you
might want to check out the archives (google?) for performance reports.

As far as the other note asking about whether it is "faster" to PERFORM
VARYING from the beginning or the end of the string, I would *guess* that
would depend on whether the majority of strings "searched" had more than
half or less than half of the "field" filled.  OTOH, if you are searching
from the beginning you need to "compare" the entire rest of the string
(reference modification) for spaces not to fall into an "intermediate" space
causing you to end the PERFORM while if you start at the end, you just need
to stop when you FIRST hit a single space character.  Depending on the
compiler (generated machine code) and operating system, that might be faster
even if most of the strings are less than half filled.

To "recapitulate" (and I am typing these in by hand - so compiler errors -
or even logic errors - might occur)

NOTE WELL:
   Others in CLC *may* have "tweaks" to the following to optimize each of
the approaches.  I haven't tested them, so there might be some better coding
techniques - especially depending on what optimizations your compiler can
do.

For:

 01  Variable-Length-String
        05  EachByte     Occurs 1 to 300 times
                                   Depending on SomeThing
                                   Pic X.
*> if your string  is smaller than 255, your results MIGHT be better.
 01  End-Results    Pic 9(03).        *> using a sign or USAGE
BINARY/COMP-5/whatever might help
 01  Temp-Num1    Pic 9(03).        *> using a sign or USAGE
BINARY/COMP-5/whatever might help

Option-1 (probably a medium-poor performer - but easy to understand)

    Move Zero to Temp-Num1
    Inspect function Reverse (String-to-Test)
         Tallying Temp-Num1 for leading spaces.
     Compute End-Result = (Function Length (Variable-Length-String)) -
Temp-Num1.
* You could use SomeThing instead of "Function Length
(Variable-Length-String)
*   If that really DOES include the current string length

Option-2  (possibly the BEST performing on many systems)

    Move Zero to End-Result
    Perform
       Varying Temp-Num1
          From Something by -1
          Until Temp-Num1 < 1
                If String-to-Test ((Temp-Num1 - 1) : 1) Not = Space
                    Move Temp-Num1 to End-Result
                    Move 1 to Temp-Num1
               End-If
    End-Perform
    If End-Result = Zero
        Move SomeThing to End-Result
    End-If

Option-3 (Might perform best if the majority of strings are less than half
used)

    Move Zero to End-Result
    Perform
       Varying Temp-Num1
          From by +1 by +1
          Until Temp-Num1 >= Something
                If String-to-Test (Temp-Num1 : )  = Space
                    Subtract 1 from Temp-Num1 Giving End-Result
                    Move SomeThing  to Temp-Num1
               End-If
    End-Perform
    If End-Result = Zero
        Move SomeThing to End-Result
    End-If
```

```
There are a number of ways to do this with perform varying but
I find this to be the most straightforward, well to me at least.
At the end of the perform data-name dl contains the length of
the data string. Everything is done in the perform, no additional
if's.


05 dl  pic 99.  (dl = data length)
05 field-1 pic x(20) value 'abcde'.

perform varying dl from length of field-1 by -1
 until field-1 (dl:1) not = space or dl = zero
   continue
end-perform.
```

```
William M. Klein opined:

<<But I believe that in most systems where CLC people have run tests, a
PERFORM VARYING *performs* better.>>

Arguably in some implementations, maybe even most, but definitely not in
all.   It depends on the underlying platform and the language
implementation.  And I would contend that this is an area where both
hardware support and smarter code generation might be useful.

On Unisys MCP/AS systems, I find that the performance of the "hard-coded"
PERFORM variant varuies dramatically with the number of trailing blanks
encountered in the field at execution time, and the fewer the trailing
blanks, the faster the results are obtained; the more, the slower.   Without
some pretty hairy compile-time optimizations on the part of the compiler
implementor, I'd expect that to be a general truism regardless of compiler
or platform.

The implementor of INSPECT as well as FUNCTION REVERSE may have a greater
repertoire of resources on a given system than would be reflected in the
"hard-coded" approach -- while ONE way to implement INSPECT FUNCTION REVERSE
would be to do a PERFORM VARYING BY -1 and compare each character, it's not
necessarily the BEST way, or the ONLY way.

For example, INSPECT FUNCTION REVERSE approach in MCP/AS COBOL85 takes
advantage of hardware support for character scans, and thus for each such
INSPECT a MAXIMUM of two compares against spaces -- one single-character
compare, one scan if the first character of the FUNCTION REVERSE result is
blank -- is done.   The INSPECT FUNCTION REVERSE takes almost exactly the
same amount of time whether a 2000-character field contains spaces, a single
leading nonblank, all nonblanks, or a single trailing blank on that system,
and it's actually the FUNCTION REVERSE that eats the time.

On the particular system on which I do my work, if say only the first 10
characters of that 2000-character field are non-blank, the INSPECT will be
nearly twice as efficient as the hard-coded PERFORM approach at determining
how many trailing blanks there are.  Even for a 30-character field with 20
trailing blanks, the INSPECT tells me noticeably more quickly how many
trailing blanks the field cointains than the "hard-coded" approach does.

   -Chuck Stevens
```
