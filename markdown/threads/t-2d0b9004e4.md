Chris,

your best bet is the EXCI - External CICS Interface.  This allows a (COBOL) batch program
to run a CICS application program by using 

	EXEC CICS LINK PROGRAM('CICSPGM')
                       COMMAREA(My-commarea)
                       NETNAME('PRODCICS')
                       SYNCONRETURN
       END-EXEC

Look at SC33-1944 CICS External Interfaces Guide

No one as yet addressed the 'modify the CICS app to send output back to
batch rather than
Terminal' part.

Be aware that the EXCI (EXternal Cics Interface) is ONLY initiated from
OUTSIDE CICS.
ALSO, it is a SUBSET of DPL:  you can NOT have any Terminal I-O and most of
the
ASSIGN ... are NOT available.  There is NO way within CICS to communicate
with Batch
UNLESS Batch is the requestor.  Now, many sites have Console Log exits that
exmaine
all the CICS Write To Operator / Log for specific strings and then can 'kick
off' an MVS job
for you, AND that job can contain a program that is an EXCI CLIENT.

Also read the IBM Manual:  CICS/ESA V4R1 (or later) External CICS Interface
(Document Number SC33-1181-01) AND  Intercommunication Guide  (search the
book on 'EXCI')


P.S., the example code given in IBM manuals is written to be very brief so
you can paste it
into the editor and create a working program very quickly.

Your Systems person(s) will need to ensure that you have an EXCI connection
(VTAM, etc)
as well as probably modifying the startup, because the default does not
allow EXCI.

Please be sure that you batch code checks the return code AND sub-codes.
When IRC is disable OR CICS is down, you get the same erorr.

Good Luck
