› My COBOL program must read a VSAM file created in a PL1 layout and check a
› one-bit flag. If the record layout had been created in COBOL using bytes,
› this would be a snap. Has anyone used COBOL to read data on the bit level?

Some versions of COBOL did support bit manipulation but that was not
an ANSI standard item. I used to do it as follows, assuming that the
PL/1 field was at least one characater in length and is defined as one
character in the input record you're reading.

05 binary-item pic s9(04) comp.
05 character-item redefines binary-item.
10 dummy-field pic x(03).
10 pl1-field pic x(01).

This defines a half word binary field and redefines it as character so
that you can move data in with no conversion. Then the code looked
something like this assuming that the low order bit holds the value
you want to check.....

Move low-values to dummy-field.
Move input-field to pl1-field.
If binary-item =1 or binary-item = 3 or binary item = 5 ....
binary-item = 255 then
do something

The test in this case is for all possible values with the low order
bit on, i.e. odd numbers. Obviously the test could be done with 88
levels or any other way you want. If the item is not in the low
position, you'll simply have to define the numbers between 0 and 256
that match whatever bit you are searching for.

You can 'cheat' and process a byte or bytes numerically to determine the
bit settings.
Consider the following example:
Say your file has a one-byte field, which represents up to 8 bit
settings. If all bits were 'on' ('FF'x), the unsigned decimal value
would be 255. Two-bytes ('FFFF'x) would be 65535. To make life easier,
you code a subroutine or a generic subprogram that processes one byte at
a time. The code would look like this (I just made this up--it was not
taken from a working program):

WORKING-STORAGE SECTION.

01 FLAG-BYTE PIC X.
01 BYTE-VALUE PIC 9(4) COMP.
01 FILLER REDEFINES BYTE-VALUE.
05 HI-BYTE PIC X.
05 LO-BYTE PIC X.
01 BIT-FLAGS.
05 FILLER PIC X OCCURS 8 TIMES.
88 BIT-ON VALUE '1'.
88 BIT-OFF VALUE '0'.

01 your-record.
05 some-fields
05 RECORD-FLAG-BYTE PIC X.
05 some-more-fields

PROCEDURE DIVISION...

MOVE RECORD-FLAG-BYTE TO WS-FLAG-BYTE.
PERFORM SET-BITS.
IF BIT-ON (1) .........

SET-BITS.
MOVE ZERO TO BYTE-VALUE.
MOVE ZEROES TO BIT-FLAGS.
MOVE WS-FLAG-BYTE TO LO-BYTE.
IF BYTE-VALUE > 127
SUBTRACT 128 FROM BYTE-VALUE
SET BIT-ON (1) TO TRUE.
IF BYTE-VALUE > 63
SUBTRACT 64 FROM BYTE-VALUE
SET BIT-ON (2) TO TRUE.
IF BYTE-VALUE > 31
SUBTRACT 32 FROM BYTE-VALUE
SET BIT-ON (3) TO TRUE.
IF BYTE-VALUE > 15
SUBTRACT 16 FROM BYTE-VALUE
SET BIT-ON (4) TO TRUE.
IF BYTE-VALUE > 7
SUBTRACT 8 FROM BYTE-VALUE
SET BIT-ON (5) TO TRUE.
IF BYTE-VALUE > 3
SUBTRACT 4 FROM BYTE-VALUE
SET BIT-ON (6) TO TRUE.
IF BYTE-VALUE > 1
SUBTRACT 2 FROM BYTE-VALUE
SET BIT-ON (7) TO TRUE.
IF BYTE-VALUE > 0
SUBTRACT 1 FROM BYTE-VALUE
SET BIT-ON (8) TO TRUE.

For testing purposes, check BYTE-VALUE at this point--it should always
be ZERO!
There may be other more elegant ways to do this, but this is a quick and
dirty.
Hope this helps :-)

There is indeed. The LE callable service for testing bit positions is
CEESITST. The syntax is
CEESITST(parm1, parm2, fc, result)

parm1(input) - a 32 bit integer which is to be tested
parm2(input) - a 32 bit integer in the range 0 - 31; the bit position
to be tested
fc (output) - a 12 byte feedback code indicating the success of the
call
result (output) - a 32 bit integer which contains:
0 if the bit position is 0
1 if the bit position is 1

bits are counted from the right

Other bit manipulation services in LE are:
CEESICLR sets a selective bit to 0
CEESISET sets a selective bit to 1
CEESISHT shifts bits right or left a specified number of
positions

Here is a COBOL example of how to use the bit test service:

Picking up on Christopher Westbury's idea, you could try this as a generic
routine (untested).

WORKING -STORAGE SECTION.
01 BIT-TWIDDLE-AREA.
05 BIT-TESTER-BINARY PIC S9(04) COMP.
05 BIT-TESTER-ALPHA.
10 BIT-TESTER-HIGH PIC X(01).
10 BIT-TESTER-LOW PIC X(01).
05 BIT-TESTER-RESULT-TABLE.
10 BIT-TESTER-RESULT-CHAR PIC X(01)
OCCURS 8 TIMES.
PROCEDURE DIVISION.
MOVE BIT-FLAG-BYTE TO BIT-TESTER-LOW.
MOVE SAPCES TO BIT-TESTER-RESULT-TABLE.
PERFORM VARYING TALLY FROM 1 TO 8 BY +1
MOVE LOW-VALUES TO BIT-TESTER-HIGH
MULTIPLY BIT-TESTER-BINARY BY 2
IF BIT-TESTER-BINARY > 127
MOVE 'Y' TO BIT-TESTER-RESULT-CHAR (TALLY)
END-IF
END-PERFORM.

The original poster asked about a one-bit flag, and that's the question I
answered. If you need to test one bit, just test the damned bit! There
is no need to explode the byte.

But if you need a generic routine, here's one (tested, big-endian):

05 WS-BIT PIC S9(4) COMP
OCCURS 8 INDEXED BY WS-BIT-NDX.

05 WS-COMP PIC S9(4) COMP VALUE ZERO.
05 FILLER REDEFINES WS-COMP.
10 FILLER PIC X(1).
10 WS-BYTE PIC X(1).

MOVE whatever TO WS-BYTE.
PERFORM VARYING WS-BIT-NDX FROM 8 BY -1 UNTIL WS-BIT-NDX < 1
DIVIDE WS-COMP BY 2 GIVING WS-COMP REMAINDER WS-BIT(WS-BIT-NDX)
END-PERFORM.


Then to test, say, bit 4:

IF WS-BIT(4) NOT ZERO
THEN DISPLAY "TRUE"
ELSE DISPLAY "FALSE"
END-IF.
