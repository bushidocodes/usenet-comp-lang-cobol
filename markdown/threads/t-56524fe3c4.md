I have become tired of manually converting MS-ACCESS table definitions into
COBOL, so they can be DECLARED for access in SQL via ODBC.

Mainframe COBOL provides a facility called "DECLGEN" (Declaration Generator)
which accesses the structure of a DB2 database and builds COBOL definitions
for the tables in it.

I have written the same software for the PC and am prepared to give it away
to members of this forum.

Basically, you design and build your database in MS-ACCESS (using ONLY the
database (RDB) facilities - no macros or ACCESS specific code - pure
DEFINITIONS of tables, relationships, indexes, etc.). The principle is not
to PROGRAM in ACCESS; just use it as the excellent PC RDB repository it is,
and do all the programming in COBOL. I have found this leads to excellent
performance on the network and, despite horror stories you may have heard
about ACCESS, it actually performs very well if it is left to just managing
the RDB.

Using ACCESS in this way also provides a ready growth path for your clients.
When the volumes outgrow ACCESS, you simply define exactly the same table
structures to INGRES or ORACLE (or DB2 or UDB on the mainframe), do a data
load and away you go, with no re-programming thanks to ODBC.

You will need to have the MS-ACCESS Documenter facility installed with
ACCESS. You run the Documenter, which produces a standard PC file with all
the ACCESS definitions in it (the settings you require to do this are
documented with DECLGEN - it takes about 2 minutes to set it up the first
time you do it...). You then run DECLGEN, pointing it at the Documenter
output. It produces a correct COBOL definition for every table you have
documented.  Two COBOL definitions are produced:

1. The standard COBOL record structure at an 01 level with the various
fields defined beneath it. (Useful if you want to separate and isolate your
SQL access to specific modules of code which are not "visible" to the rest
of the system. You communicate with them by passing a standard COBOL record
structure and the action you want carried out.) This definition would be
used in WORKING-STORAGE and LINKAGE sections.

2. The "host variable definition" form of the table. This definition can
then be included into the  SQL DECLARE section of your COBOL program. In
this version ALL the "columns" of the original table are defined as 01
levels within BEGIN and END DECLARE statements, and entries are also
provided (optionally) for declarations of SQLCODE and SQLMSG, so that
replies from ODBC can be passed back to the programs.

In my company we have found that doing this gives us a platform independent
data repository which can easily be grown as required. We no longer use
COBOL specific access methods (Indexed and Relative files) and have no
problems with having to convert data across platforms or new releases of
COBOL.

The other major spin off from this is that we can use Crystal Reports (this
fantastic piece of software is also independent across RDB platforms) to
generate all the reports and train the Users to generate their own. (We put
an ActiveX control which links to the Crystal build functions on some of our
application screens, and another similar control allows access to the
Crystal run-time. A table of available reports is maintained easily by the
users and as they develop new reports they simply update this table. Our
applications allow them to select the report they want and invoke the
Crystal run time via the ActiveX control, using the report name and location
from the table. They can view reports on screen, transmit them via e-mail or
fax, or print them to hard copy, all selectable when they click the Crystal
control. COBOL listings using Report Writer or WRITE BEFORE/AFTER ADVANCING
are obsolete for us.

I have written an Access Basic routine that does what Peter described.  To
use this routine, create a command button on a form.  Use the supplied code
as the "On Click" method for the command button.  When you open the form and
click on the button, Access will generate COBOL descriptions for all the
tables in the Access database.

Notes

1.  This is rather rough and ready and I have only defined some field types.
I would welcome any suggestions to make the code better or more complete.
2. I am not positive of the COBOL format of the field types and I have not
tested the generated code.
3. The code produces two files for each table - <table name>1.cbl contains a
record
layout and <table name>2.cbl is the list of 01 level fields.
4. The code produces code for all tables except those beginning with "MSys"
(Access internal tables) .
5. Access creates the files in the last directory that you accessed through
Access.

Ed Milne

-------The Code-------
On Error GoTo declgen_Error

  Dim db As Database
  Dim con As Container
  Dim tdf As TableDef
  Dim i As Integer
  Dim fld As Field
  Dim outfile1, outfile2 As String
  Static prefix00 As String * 8
  Static prefix01 As String * 11
  Static prefix05 As String * 14

  prefix00 = "      * "
  prefix01 = "       01  "
  prefix05 = "           05 "
  Set db = DBEngine(0)(0)
  For Each tdf In db.TableDefs
    With tdf
      If Left(.Name, 4) <> "MSys" Then
        outfile1 = .Name & "1.cbl"
        outfile2 = .Name & "2.cbl"
        Open outfile1 For Output As #1
        Open outfile2 For Output As #2
        Print #1, prefix01; .Name; "."
        For Each fld In .Fields
          Select Case fld.Type
          Case dbLong
            Print #1, prefix05; fld.Name; Tab(52); " COMP   PIC S9(9)."
            Print #2, prefix01; fld.Name; Tab(52); " COMP   PIC S9(9)."
          Case dbCurrency
            Print #1, prefix05; fld.Name; Tab(52); " PIC 9(9)V99."
            Print #2, prefix01; fld.Name; Tab(52); " PIC 9(9)V99."
          Case dbText 'text
            Print #1, prefix05; fld.Name; Tab(59); " PIC X(";
Trim$(Str$(fld.Size)); ")."
            Print #2, prefix01; fld.Name; Tab(59); " PIC X(";
Trim$(Str$(fld.Size)); ")."
          Case dbMemo 'memo
            Print #1, prefix05; fld.Name; Tab(59); " PIC X(256)."
            Print #2, prefix01; fld.Name; Tab(59); " PIC X(256)."
          Case dbDate
            Print #1, prefix05; fld.Name; Tab(59); " PIC 9(8)."
            Print #2, prefix01; fld.Name; Tab(59); " PIC 9(8)."
          Case Else
            Print #1, prefix00; fld.Name; " undefined data type"
            Print #2, prefix00; fld.Name; " undefined data type"
          End Select
        Next fld
        Close #1
        Close #2
      End If
    End With
  Next tdf

Exit_declgen:
  Exit Sub

declgen_Error:
  MsgBox Error$
  MsgBox "Error #" & Err & " " & Error & "in procedure declgen."
  Resume Next
