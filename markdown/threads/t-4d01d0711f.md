```
easy? you could have a program which intercepts open calls to the ISAM
handler, and gives them a hold error message, and a program which
during every *close* to the file, attempts an *exclusive* lock on the
file.

to do this, you would need a pointer to the function which opens and
closes the ISAM file. that, and the parameters taken.
you would also have the problem of whether the open function is
currently in use AND if so, how a bad call to it would be handled (a
stack based program counter, or a failed return which crashes the
entire machine?)

These are the most possible Points of Failure I can see, from a
computer scientists point of view.

the simplest option, however, is to constantly attempt an exclusive
lock on the ENTIRE ISAM file. sure, polling will take up alot of CPU
cycles, but most CPU cycles are not used anyways. 

on the other hand, you may not have to lock the entire ISAM file. just
lock 1 part at a time.
```

```
You can specify LOCK MODE EXCLUSIVE in the SELECT and then
when you try to OPEN it you will get a File-Locked status
(9A from memory).  Once you have sucessfully opened it others
will get that status when they try to access it.

If you need to signal other programs that they need to close
it you will have to be able to change them to check for a
flag that you set somewhere.  They would need to periodically
look for this.

Alternately it could be done by record locks.  Have a control
record in the file, with a key of, say, low-values, and only
allow updating of the file when a successful KEPT LOCK has
been obtained on this record.  This will allow readers of
the file to continue but updaters can be excluded (as long
as they all follow the rule).
```
