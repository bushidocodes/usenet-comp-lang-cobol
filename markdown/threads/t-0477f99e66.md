From "Effective Java" 1st edition by Joshua Bloch, item 14:

"Inheritance is a powerful way to achieve code reuse, but it is not always 
the best tool for the job.  Used inappropriayely, it leads to fragile 
software.  It is safe to use inheritance within a package, where the 
subclass and the superclass implementation are under the control of the same 
programmers.  It is also safe to use inheritance when extending classes 
specifically designed and documented for extension.  Inheriting from 
ordinary concrete classes across package boundaries, however, is dangerous. 
As a reminder this book uses the word "inheritance" to mean implementation 
inheritance (when one class extends another).  The problems discussed in 
this item do not apply to interface inheritance (when a class implements an 
interface or where one interface extends another).

Unlike method invocation, inheritance breaks encapsulation [Snyder86].  In 
other words, a subclass depends on the implementation details of its 
superclass for its proper function.  The superclass's implementation may 
change from release to release, and if it does, the subclass may break, even 
though its code has not been touched.  As a consequence a subcalss must 
evolve in tandem with its superclass, unless the superclass'e authors have 
designed and documented it specifically for the purpose of being extended."

The reference is (presumably) to Alan Snyder's talk "Encapsulation and
Inheritance in Object-Oriented Programming Languages" at OOPSLA '86.
While Snyder's thesis is well-known (or at least widely cited) and
important, OO language theory has had more than 20 years to reconsider
it; and there seems to be a widespread (though not universal)
conclusion that while it might be a useful starting point, it's
overstated.

And probably more importantly, in the very next year (1987), Barbara
Liskov (who just received the Turing Award from the ACM) published the
Liskov Substitution Principle, which formalizes how type hierarchies
should be constructed to avoid breaking encapsulation. Arguably, that
alone rendered Snyder's thesis moot, or at least incomplete.

> In

> superclass for its proper function.

Since every component of a computer system depends on the correct
functioning of other components, this is a tautology.

While the coupling introduced by implementation inheritance is not
academic, it's also not nearly as simple as offhand references to
Snyder, like the one you quoted here, make it sound. Implementation
inheritance (inheriting from types that are not purely abstract) is an
important option for solving many problems.

One other useful thing N/E has, and again non-Standard, is the

inclusion or exclusion), otherwise it takes too long to write and read, 
but you can have :-

Class A inherits from Base
Class B inherits from Class A WITH DATA
Class C inherits from Class B WITH DATA

Global Working storage used in either Factory or Object/Instance can 
generate copyfiles. You require both, even if one area is empty. To 
achieve this the classes must be compiled separately, top to bottom.

Using the correct syntax when Class A is compiled, there are two copy 
files generated, even if one is empty :-

ClassA.CLS (CLS = Class or Factory)
ClassA.INS (INS = Instance or Object)

If Class B is passing on Global data to C, when B is compiled, then you 
get a repeat :-

ClassB.CLS (CLS = Class or Factory)
ClassB.INS (INS = Instance or Object)

Assuming the original 'Global' in Class A had 10 variables and Class B 
adds its own 5 variables, then the cumulative effect when compiling 
ClassC C with the inclusion files, is that it now has access to the 15 
variables.

Whether a Dialog, COBOL record or row from a DB table, that you are 
editing, then you are ONLY changing the 'record' in the first class that 
specifies it. The inclusion copyfiles are solely there so that the 
sub-classes well compile and wont hiccup when they see a line "Add 10 to

It's an interesting topic and one we covered back in softwaresimple.com 
when roughly a dozen of us were discussing OO COBOL - 'Invoking Up and 
Down". Given a set of classes with inheritance, and this philosophy I'm 
sure must apply to all other OO languages :-

a) To point upwards - from the class you are in, say Class K,

		invoke SUPER "doThis"

and the search is upwards through J to A for the method - and you sure 
get an error message if you made a typo on the method name.

b) to point downwards - from the class you are in and it is Classs K,

		invoke SELF "doThis"

which means take a method from within Class K; however, assuming your 
inheritance hierarchy was down to Class P, that invoke would go to the 
lowest, Class P, starting a search and eventually find the method it 
wants in Class K. (Initial reaction it looks klutzy but it is a valid 
approach and works).  Why does it go downwards initially - assume you 
had a method "doThis-2" in Class M. The invoke is a SELF invoke :-

		invoke self "doThis-2"

Down to the bottom Class P and search upwards now looking for "doThis2" 
which it finds in Class M.

"If that's your sole, or main, criterion for evaluating a programming

COBOL implementations can generally call external libraries. If the
functionality needed to solve a particular problem isn't built into
the language, it's probably available in an external library. If it
isn't, *then* you'd have some grounds for evaluating your language
choice on the basis of available functionality.

Far better would be criteria like "how well do I understand this
language?", "how elegantly can I solve my problem in this language?",
"how difficult will it be to maintain a solution written in this
language?".

The question is even more ridiculous in the .Net case, since .Net
COBOL implementations can use nearly everything in the .Net Framework.

(Incidentally, Javadoc is not a "library". It's a documentation
system. The standard Java classes reside in the Java Runtime Environment.)
