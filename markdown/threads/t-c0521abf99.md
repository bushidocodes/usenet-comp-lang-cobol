Assuming you have a real business reason for converting (VSE/ESA
can almost do everything that MVS does these days) the JCL and
COBOL conversion is not really that difficult...Consulting firms
tend to make it sound far more complicated than it really
is...your biggest problem will be any assembler code you have that
exploit VSE specific functions, other than that, COBOL is pretty
much the same except for the SELECT clauses for non-VSAM
sequential files, and the JCL is actually very flexible once you
get the hang of it (note: no chance of correction at the console
so you need to get it right)...Bottom line is, take a deep breath,
role up your sleeves, and just go at it...Good luck

It's been a long time, but training in jcl was time consuming, but not
difficult. The batch cobol programs (vsam and sequential files) seemed to
only need the select statement changed for vsam. Sequential files also
needed select changed and we required that the 'block contains' be added or
changed to 'block contains 0' in the FD so jcl blksize could be used. Now
that can be set to 0 also so system will determine block size based on device
the file is allocated on (mvs/esa). As long as you have to look at all the
programs anyway, might want to look at mvs cobol (used to be cobol/370 or
le/cobol) and fix any turn-of-the-century problem.
