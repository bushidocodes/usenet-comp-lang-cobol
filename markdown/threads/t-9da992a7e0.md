One approach is to define an Extrapartition Destination (see CICS Resource
Definition) in the Destination Control Tabble (DCT). This defines what is
basically a sequential dataset to which you can write with the READQ TD
(Read Queue Transient Data) command.

After the queue is defined in the DCT you put a DDcard in the CICS startup
deck (or, your CICS Systems Programmer puts a DD...), and point it to the
JES internal reader. Then, anything you write out from your CICS program
goes directly into JES.

For instructions on the WRITEQ TD, see the Transient Data Control section
of the CICS Application Programmer's Reference.

I've heard there are other ways to do this, but this one is very simple to
implement.

This is a tried & true technique and has worked for years across
multiple releases of CICS. Some shops constantly open & close the
extrapartition TDQ - lots of overhead (SVCs) and no value. End your set
of JCL with a "/*EOF " statement. You definitely need assistance from
the CICS group (or whoever codes the procs you use to bring up your
regions), because this approach requires a DD statement to route the
output (from CICS) back into JES. It looks something like: "//INTRDR
DD SYSOUT=(A,,INTRDR)".

There's a newer technique, using EXEC CICS SPOOL commands. I was warned
away from them (CICS/ESA 3.3, I think) and have no experience with them.

You are correct about the ENQ/DEQ. It should be emphasized to use the CICS
ENQ/DEQ commands, not the system level macros. Unfortunately everywhere I
have been the systems programmers do not trust mere aplication programmers
to use ENQ/DEQ (for good reason, considering how many do not know how to
use them). It is best if you *do* use these, but try to make sure that
your code is not going to lock your resource by enqueuing, and then not
releasing it.

While it is somewhat more complicated, you can use interval control to
ensure that your task does not lock up your transient data queue. The main
submit program should:

a) Write the JCL images into a personal temporary storage queue (or some
equivalent resource)
b) Enqueue the transient data queue for the submit.
b) Post an event to indicate completion of the JCL submit. Give this a
very long interval.
c) Post a second event on which it will wait a specified interval. Give
this a somewhat shorter interval (how long you actually expect it to take
to get the JCL written, maximum, under normal conditions).
d) Dispatch a task with a secondary submit program to write the JCL from
the temp queue to the transient data queue (START command), passing the
addresses of the posted blocks (sometimes called ECBs, or Event Control
Blocks), and the TS queue name.
e) Issue a wait event for the second posted ECB. This will suspend the
main task either until the JCL has been submitted, or for a reasonable wait
time.
f) When the wait returns to the main program (when the second event has
expired) examine the first posted event to see whether the submit has
completed.

If the first event has not completed:

1) Cancel the started JCL submit task.
2) Do whatever error recovery neeeds to be done (the submit has failed).

g) Dequeue the TD queue resource.

The dispatched task submit program needs to:

a) Accept the TS queue name, and copy the entires in to the TD queue.
b) When complete, clear both posted ECBs by writing x'40' to the first
byte, and x'80' to the third byte.
c) Return to CICS.

The idea of all this is that you will have an automatic way of cancelling a
failed submit attemt so that it will not lock up your queue.

Your main program can do an EXEC LINK to the main submit program, passing
it a TS queue name. In this way you have a neatly packaged general submit
routine any program can use from within CICS, which will recover itself
from errors. And, needless to say, there are a number of different
approaches, which are every bit as good. This is just a suggestion.

Be aware that unless the JOB card of the submitted job specifies a USER
parameter, these jobs will run with the RACF security of the submitting
CICS region, which may or may not be OK depending on your environment
and what the job is doing. Because in our environment this is not
always reasonable, we require application code to pass the JCL to an
intermediate program which enforces installation standards on the JOB
card and then writes the JCL, rather than allow applications to directly
write to the internal reader destination. To the extent possible, we
also limit the complexity of these jobs (since there is no ready
mechanism for detecting job failure or handling job restarts), so that
in some cases these jobs may just initiate a demand to the job scheduler
(CA-7) to initiate another job under CA-7 control.

Since generation of JCL is under the control of the CICS transaction,
any JCL modification, display or user edit of JCL prior to submission
must be built into the transaction application itself (there is no
special CICS support for JCL per se).
