Is anybody aware of a tool that can analyze a library or a job and
scan for which modules within that library or job are using the most
memory? We have a private storage area located below the line that
holds 10000 K. Currently, some of our jobs are defined with a region
of 9500 K and will sometimes run above that, which will inevitably
push against and sometimes past the line, which of course causes the
system to freeze. I'd like to look at how a job is running and which
modules are using the most amount of memory or have a large Working
Storage section.

Note this response assumes an IBM mainframe with either an OS390 or z/OS
operating system.

I would be very surprised if there is such a tool.  Various IDR listing
runs will tell you which modules are linkedited to run below the line
(RMODE 24) and there are programs on the CBT tape that would allow you
to find COBOL modules compiled with the DATA(24) option.  Neither will
tell you how much storage is obtained by the module.  Since storage
obtained by the module is normally far greater than the storage
represented by the load module size itself this may be of limited value.

The SMF 30 step termination records (sub-type 4 I think) will contain
information about memory usage.  These probably are being produced every
time a job is run.  See your technical staff (systems programming,
systems support, etc.) for details.

Let me add to Clark's assumption, and assume you are running with Language
Environment as your run time library.  If so, turn RPTOPTS and RPTSTG.
RPTSTG is really the one that could tell you the most.  You may see large
amounts of Stack or Heap storage being used.

If this doesn't tell you all you need to know, there are lots more ways to
figure out what's going on, so get back to us.

Also, why even try to run large apps "below the line"?  The operating system
and COBOL have supported 31-bit addressing and Residence Mode ANY for
decades already.  It's time to take advantage!
