Now you know, that's a most interesting observation.  As a longtime COBOL
programmer, recently transfered to a Web/PC team with C++ and Java, that
dovetails nicely with one of my own observations.

After many years of coding COBOL, I had reached the point where I could
pretty much "code through my fingers".   The general technique, verbs,
procedures, etc. would just flow from my brain through my fingers into the
computer.  And the code was in general very good, easy to work with, and
easy for others to understand.

When I began Java and C++ programming one of the first things that struck me
was the dichotomy.  The languages themselves are simple, very simple.  The
primary keywords and the language syntax are simple to master.  But the
extensions!  Ah, now those are where things get wild, particularly in Java.
Going to Sun's web site and perusing a Java help file with all the possible
CLASSes can be an exercise in humility.  There is so much to learn, so much
that can be done, that "coding through my fingers" seems to be a forlorn
hope at present.  Add the new paradigm of object orientation and the keen
reliance of Java on the OO paradigm and the task grows even more difficult
(at least for us old time COBOLers).  Another programmer on the team, one
who has been doing this far longer, tells me that it can't be done, that
there is simply to much to know.  And from what I've seen so far, she may be
right.

To answer your question, I think both.  Java/OO will be around in 30 years.
And more importantly, yes, it takes a lot of doc to understand all the
aspects of Java.  Something that may end up happening is that
sub-specialists of Java will come about.  Some specializing in database
applications, others in socket code, others in GUI applications, etc.  It's
hard to tell.  As rich and widespread as the language is, possibly the
biggest problem will simply be getting a controlling handle on the thing.

Paul
