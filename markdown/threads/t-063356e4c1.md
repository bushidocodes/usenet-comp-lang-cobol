OK, I took some time out from Web Development and looked at Excel... :-)

I'm running Excel 2007, so I can't guarantee the result on other versions.

The most difficult thing to decide is exactly WHICH properties need to be 
manipulated.

I opened the VB object browser in Excel and there are hundreds (thousands?) 
of classes and obects.

I ran some searches on things like "percentage" and "number format" etc... 
and found some things that looked promising, then I spent an entertaining 
hour trying to get them to work. Having no idea what values they will accept 
didn't help and I could see it was going nowhere.

Then it suddenly dawned on me...

"All ya gotta do is..."  record a macro that does what you want and the 
system will write the code for you!

I did this and, sure, enough, it wrote the VB code I needed.

Here's how you can see exactly what's going on...

Open a new blank Excel spreadsheet. Select the "Developer" tab and click 
"Visual Basic".  Paste the following into the right hand pane...


Sub Pete()

Dim objA1
Set objA1 = Sheet1.[A1]
objA1.Select
Selection.Style = "Percent"
Selection.NumberFormat = "0.00%"
objA1.Value = 0.1234

End Sub

Now press F8 to step through the code.

Here's the blow-by-blow:

Dim objA1......... allocates storage for what will be an object reference.

Set objA1 = Sheet1.[A1]..... creates an object instance of cell A1 in 
Sheet1. This gives us access to all the things the cell "knows"... its 
properties and methods.

objA1.Select ......invokes the cell's "select" behaviour. Cell A1 is now 
selected. This also creates the "Selection" object which we can reference, 
instead of objA1. You can think of the selection object as a "proxy" for the 
cell object. They are really one and the same in this instance.

Selection.Style = "Percent".....sets the "Style" property of the selection 
object to have the value "Percent" . Any number placed in the cell will be 
interpreted as a percentage.

Selection.NumberFormat = "0.00%".... This tells the selection object (via 
its "NumberFormat" property) how many decimals to shift the result by, and 
to put a percent sign on the right of it.

objA1.Value = 0.1234...... this puts the value "0.1234" as you used in your 
example, into the cell.

Now switch back to your Excel spreadsheet which should still be on the 
screen behind the VB stuff (there's a little Excel icon in the toolbar on 
the VB screen that will swap it back if it is minimised or you've lost it).

Look at cell A1.    It should be showing "12.34%".  Notice that the cell 
actually contains "0.1234".

Try entering other decimal values into the same cell and you will see that 
it does what you want.

So the two magic properties (for any cell you want to have this behaviour) 
are:

.Style  (has to be set to "Percentage"
.NumberFormat   has to be set to "0.00%"

NOW, you need to translate this into COBOL and amend the program that reads 
the .CSV accordingly. You said it has a number of invokes in it. From those, 
you should be able to see what objects are being used and what names have 
been given to them.

For example, you will need to see how the program references a given cell. 
As it is using Automation to "talk" to Excel it would be something like:


W-S
...
01  objSheet1   OBJECT REFERENCE  COM.
01  objA1         OBJECT REFERENCE  COM.

01  cell-A1    pic xx value "A1".
01  percent    pic x( 10) value "Percentage".
01  cellFormat  pic x(5) value "0.00%".
...

PD
...

invoke objSheet1  "cells"
           using  cell-A1
           returning objA1 *> gets an object reference to A1...
end-invoke

invoke objA1 "SET-Style"
           using  percent
end-invoke


invoke objA1 "SET-NumberFormat"
           using  cellFormat
end-invoke

I'm sorry these are Fujitsu examples using the Fujitsu *COM Class for 
Automation. This is a layer that simplifies using COM Automation. It will 
work differently under MF. You need to read the MF docs and find out how 
they use COM Automation.

Then, when you have it working, write up what you did and send me the code; 
I need some MF examples of COM interfacing for the cobol21 website... :-)

If this all looks terribly intimidating to you, visit the website and read 
about the Component Object Model  (COM) in the section on components. (It 
will be a day or two yet before I finish writing it).  It really isn't that 
hard.

http://primacomputing.co.nz/cobol21

Pete.

Then you have a problem, Paul.

CSV files recognise 2 types: Numeric and Alpha.  Your Spreadsheet recognises 
a number of types, including percentage and date (as you discovered).

What, to the CSV file is a string,  say, "10/05/09"  on the spreadsheet is a 
string only if it is stored in a cell of type string. If you store it in a 
cell of type Date then it will display as a date using whichever date format 
is selected for the display of dates (it could just as well display on the 
spreadsheet as: "Saturday, 10th May, 2009")

Percentages are NOT a special case; they are a normal part of life in

No, there are no "self-identifying" fields. What identifies the type of a 
field is the type of the cell it is being loaded into.

For example, if I have on the CSV file a field which is .1234. If I load it 
into the spread sheet the sheet will assume it is a numeric field and it 
will be loaded to a cell suitable for holding numbers with fractions. If I 
display the contents of the cell it will show as "0.1234" or whatever format 
string I have applied to it for zero suppression or whatever. Now, if I tell 
the spreadsheet that the SAME CELL is a percentage, and provide a percentage 
format string, it will display as "12.34%".  The contents of the actual cell 
are still .1234 and they never changed. (This is a little different form 
COBOL where we would move the filed to an editied picture... on the 
spreadsheet you can think of the spreadsheet moving the field to a work area 
where it is edited. Correctly, it "applies" the format string to the data, 
but does not change the data.

Again, if I have a string on the CSV file that is "10/05/09". This might be 
a part number, an account sales territory, or a myriad of things... it 
doesn't HAVE to be a date... You aply it to the spreadsheet and (unless it 
has been told differently) the spreadsheet cell stores it as a general 
string. It displays as "10/05/09". Now we tell the spreadsheet it is a date, 
and we select "long date" as the format for it. Suddenly it appears on the 
spreadsheet as "Saturday, 10th May, 2009". but the cell still contains 
"10/05/09". Change the format on the same cell to be "short date" and it 
displays as "10/05/2009", but the content is still "10/05/09". (That isn't 
strictly true because the actual content is a floating point number; but 
"conceptually" it retains the original string)

So you can see that the cell type and the format properties of the cell 
decide how the data renders on the spreadsheet.

It is also important to note that the spreadsheet validates data before 
storing it. If you try and put 29/02/2009 into a Date cell it will reject it 
(or change the cell type to be a general string so it can accommodate it, 
depending on how things are configured.) But there is no problem with 
loading this into a cell defined to hold a general string.

BOTTOM LINE: The target cell type determines how data will be displayed on 
the spreadsheet. As long as you put numbers into numeric cells, strings into 
string cells, percentages into percentage cells and dates into date cells, 
your spreadsheet will contain valid data. Usually, if the spreadsheet is 
blank it will try and accommodate whatever it encounters as either numeric 
or string. (You can hardly expect it to know psychically that a certain 
number is a percentage or a certain string is a date...:-))

You can if you precede each field in the CSV file with a numeric code 
representing a data type, but there are some flaws with this idea...

1. It is unwieldy and inelegant.
2. Even if you KNOW the type of the input field, the spreadsheet will 
interpret it as the type of the TARGET cell, and completely ignore your 
input type.
3. It complicates the processing of the CSV file which is no longer 1:1 with 
the spreadsheet.

So, it doesn't help you to know the type of the source.

You could use a "Type mapping" for every spreadsheet. Again not an 
attractive proposition.

WHAT TO DO...?

So maybe you need to apply some "intelligence" to the CSV fields.

You can already identify NUMERIC or STRING pretty easily. Why not add some 
further refinement?

if it is numeric, check if it is a percentage.

if it is a string, check if it is a date...

You should get back from this something telling you: number, string, 
percent, date. Set the target cell properties appropriately, THEN move the 
data to it.

Personally, I'd write a component that takes a CSV string as input and 
returns the type as above. (It might be useful in other places...)

Up to you how you decide what is what, but half the battle is knowing what 
you are trying to do... :-)
