As far as I know, an INITIALIZE results in the same object code as a MOVE.
(How else can a compiler generate a move-constant?) The real benefits are
readability when an INITIALIZE is applied to a group item, and the
elimination of uninitialized fields when items are added later to the group.

Note that an INITIALIZE will not update FILLER or unnamed fields within a
group.  Also, the items of an INITIALIZEd group will not show up in the
cross-reference for that statement (at least, not in IBM Cobol II).

An example I recently encountered where INITIALIZE worked better than MOVE
SPACES TO at a group level:

Using UNSTRING  to convert  ASCII comma-delimited records from an uploaded
PC file, there were dates that were unstrung into their elements (DD, MM,
CCYY). There were FILLER elements (PIC X VALUE '/') in between the date
elements to make dates looking like 03/29/2001.  The records were variable
length so I had to initialize the "unstring-area"--if not just out of good
practice.  If I moved Spaces to the unstring-area prior to performing the
unstring I ended up with dates that looked like 03 29 2001.  As was said,
using Initialize didn't update Filler, so in this case Initialize was
preferable to Move Spaces To because it didn't make the slashes dissappear
and the routine made dates as intended.  Otherwise, it's probably just a
style (or perhaps shop standard) issue.

Of course you would. Poor record design (just kidding). In our shop,
we try to put alphanumerics at the start of the record and DISPLAY
numerics at the end. So, a typical record layout looks like:

01  MYCUST.
   02  MYCUST-ALPHA
      05  MYCUST-NAME
      05  MYCUST-ADDR1
      05  MYCUST-ADDR2
(etc. followed by)
      05  MCUST-SPARE1 [note 1]
   02  MYCUST-NUMERICS.
      05  MYCUST-CURRENT-BAL     PIC S9(7)V99.
      05  MYCUST-CREDIT-LIM           PIC S9(7)V99.
      05  MYCUST-LAST-PAYMENT
(etc. followed by)
      05  MYCUST-SPARE2 [note 1]

[Note 1] This notation "X-SPAREx' takes the place, in our shop, of the
traditional 'FILLER' used to reserve room in the record for future
expansion. This notation also gives the INITIALIZE verb something
tangible to chew on.

We've considered the possible downsides to this policy. The two
biggest disadvantages, efficiency and economy of disk space, just
don't seem to overcome the resultant ease of use/programming and
elimination of confusion (as you saw in your example). In debugging,
for instance, a human can actually READ the data files instead of
having to decode the elements. In programming, many controls can slurp
up a FILE and display or print the results with no programming or
conversion code on our part.

Maybe our goofy-meter is in the red. But I've seen programmers use a
COMP-3 field for a ZIP code because five-digits packed-decimal uses
two bytes less than DISPLAY and one byte less than BINARY. That, in my
view, seems not only extreme, but silly.

there is anything "guaranteed" to do this. The problem is (as other have
stated) that if you have a field defined with NO "VALUE" clause, then its
value at program initialization is UNDEFINED.  This means that you may or
may NOT get the same values at the beginning of every execution of that same
program (same compiler, environment, etc).

On IBM mainframes (for example) there is a "fairly unusual" situation in
which "uninitialized" fields get WHATEVER is left in storage from the last
time that was "used".

You may use the "new and improved" initialized statement to "re-set" all
fields (including FILLER fields) to whatever VALUE clause was specified in
the program source code (including the "now allowed" uses in Linkage Section
and File Section) - but there is still no way to get the "same garbage" as
those fields had when the program first started - if they had no VALUE
clause for them.
