In another thread, Pete talked about being surprised (I think that was his word) 
that you couldn't use FUNCTION REVERSE with INSPECT REPLACING (but could with 
INSPECT TALLYING).

As a (relatively) "mono-lingual" programmer, I am curious, do other programming 
languages allow for functions to be RECEIVING items (i.e. to be changed)?  COBOL 
certainly does not allow this for any <intrinsic> function.

COBOL ('02 Standard) does allow a "object reference" to be a receiving item if 
it explicitly or implicitly has a SetProperty method - and that makes sense to 
me.  However, I see a "fundamental" difference between a "function" and a 
"subprogram" being that a function ONLY "returns" a value and doesn't 
(hopefully) have "side-effects" that can change values.

William M. Klein wrote:
> In another thread, Pete talked about being surprised (I think that was his word) 
> that you couldn't use FUNCTION REVERSE with INSPECT REPLACING (but could with

> certainly does not allow this for any <intrinsic> function.

There the function isn't a receiving item; the application of the
function (the value returned by the function) is a receiving item.

There are languages where functions are first-class objects, which
means you can treat them as any other kind of value. In a language
where functions are first-class and side effects are allowed, it would
be possible to change a function. Usually, though, functions are
immutable objects even if they're first-class.

What you can do, in languages where functions are first-class, is
create new functions, and in many (but not all) such languages you can
assign a new function to the identifier of an existing function. So
you're not "changing a function", but you're changing the function
that an identifier refers to.

In Common LISP, for example:

	(defun foo (x) (+ x 1))
	(foo 1)
	2

	(defun foo (x) (+ x 2))
	(foo 1)
	3

defines and then redefines function "foo".


In many languages, you can operate in certain ways on a function
application. In C, for example, a function application cannot be an
lvalue (the C equivalent to a receiving item in COBOL), but it can be,
say, a pointer that can be dereferenced to an lvalue:

	char test[] = "hello";
	char *FirstChar(char *s) { return s; }
	*FirstChar(test) = 'H';
	puts(test);

will print "Hello". FirstChar(test) returns a pointer to the first
character of the string, which can be dereferenced and assigned to.

Languages with functional constructs usually have mostly immutable
objects with non-destructive functions and garbage collection. That
means functions that "change" their parameters really copy them, alter
the copy, and return the altered copy. If the input isn't used
subsequently it'll eventually be garbage-collected.

In such a language you can operate on the results of such functions
because you're really operating on temporary copies.

> COBOL ('02 Standard) does allow a "object reference" to be a receiving item if 
> it explicitly or implicitly has a SetProperty method - and that makes sense to 
> me.  However, I see a "fundamental" difference between a "function" and a 
> "subprogram" being that a function ONLY "returns" a value and doesn't 
> (hopefully) have "side-effects" that can change values.

In computer-science terms, "pure functions" don't have side effects,
"impure functions" have side effects but don't alter their parameters
(eg a function that writes a value to a file), and "destructive
functions" alter their parameters.

But this doesn't really have any bearing on the case you originally
mentioned, where INSPECT REPLACING can't use a function application as
its receiving item. COBOL forbids that because INSPECT TALLYING is, in
CS terms, a destructive function, and the COBOL syntax provides no way
to get at the results if they're not already bound to an identifier.

FUNCTION REVERSE creates a temporary data item with no name. You can
use that as a sending item in the same statement, because in that case
it doesn't need a name - its position in the statement identifies it.
For example, in:

	DISPLAY FUNCTION REVERSE(foo)

DISPLAY knows to operate on the temporary anonymous data item created
by FUNCTION REVERSE. But in:

	INSPECT FUNCTION REVERSE (foo) REPLACING ...

INSPECT knows what to operate on (the temporary anonymous data item),
but after the INSPECT you'd have no way to access that value. INSPECT
wouldn't be altering the data item named by "foo" - it'd be altering
the anonymous temporary. But at the end of the statement (in effect)
that temporary would be thrown away.

COBOL doesn't have any way to associate an identifier with the
temporary item created by a function. You can only operate on the
contents of that temporary item. (That lets the language
implementation do very simple reclamation of temporaries - eg by
putting them on the call stack, if there is one, and letting them get
scrubbed when the current frame is discarded - rather than doing
full-blown garbage collection.)

We could imagine some functional COBOL extension that did let you do
something like:

	let bar be function reverse(foo)
	inspect bar replacing ...

where "bar" would be a linkage item; this would associate the
identifier "bar" with the temporary item created by reverse(). Or even
maybe something like:

	let bar be result of inspect function reverse(foo) replacing...

(where probably "result of" would be optional syntax that could be
omitted - unless we also wanted this form to be able to construct new
functions...).

Howard Brazee wrote:
> 
> If I had designed (with my current knowledge) the way CoBOL handled

> errors.   It's not obvious how to do that with the CoBOL design where

This is one of the great problems of C as well - and it's a lot worse
there, since combining lots of function calls in a single statement is
idiomatic in C.

C++ tries to solve this with exceptions. Of course it wasn't the first
language to provide exception handling; it inherited that from various
languages such as Smalltalk, which in turn had taken the idea from
earlier attempts to manage exceptional conditions, such as PL/I's
condition facility.

C++ exceptions are in some ways more primitive than Smalltalk
exceptions or PL/I conditions. In particular, the latter support
restartable exceptions, which let you correct the condition (if
necessary) and keep going. With C++ exceptions, the stack has been
unwound and you can't go back. On the other hand, C++ offers
more-powerful exception typing than PL/I does, I believe.

Many subsequent languages, such as Java and the .NET languages, have
basically followed the C++ model. So does the Structured Exception
Handling (SEH) facility in the Windows OS, more or less.

(There are various other interesting ideas in this area. Classic Unix
has signals, which are primitive but relatively easy to implement,
understand, and use. The VAX has some very nice exception handling,
particularly for assembly programming, plus a generalized mechanism
for asynchronous programming with Asynchronous System Traps. And so on.)

The basic idea with exceptions is that you take a block of code and
say, "OK, if anything goes wrong here, stop what you're doing and go
to this other block of code, where I'll deal with it. Oh, and
regardless of whether things work out, here's some cleanup code that
always has to be executed."

That spares you from having to check the result of every function for
an error, and so lets you write tighter and more-readable code. The
inner "try" block looks like the bare algorithm without any error
handling - it's the simple, optimistic version of the code. The
cleanup stuff is all in the "catch" and "finally" blocks.

MF .NET COBOL has full support for exceptions, including
try/catch/finally keywords. I don't remember if that's true of ISO OO
COBOL as well.

That's true in many other languages as well - it's certainly not
confined to C. And while "=" is the assignment operator in C, per ISO
9899, that line is also a statement. (Specifically, it's an
expression-statement, ISO 9899-1999 6.8.3.)

I take your point that some languages allow assignment to the results
of some expressions, while others only permit assignment to
identifiers. (And others, such as Erlang, don't allow assignment at
all.) But "operator" versus "statement" seems to me just an accident
of nomenclature; an "assignment statement" is just a statement using
an assignment operator, even if the latter has more restrictions than
it does in C.

> Because the equals is an operator, expression evaluation is allowed to
> the left of it.

Yes, though only some expression evaluation. The LHS of the "="
operator must be a unary-expression that yields a modifiable lvalue.
(6.5.16) The result of applying the "*" indirection operator to a
valid operand can be a modifiable lvalue, but the result of, for
example, applying the "&" address operator cannot.

In fact, there are relatively few C expressions that yield lvalues.
This is a frequent source of questions on comp.lang.c, particularly
from people who have used C-like languages that are more permissive
(such as Gnu C in non-standard mode, which is not C).
