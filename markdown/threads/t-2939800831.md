Thanks Pete

As it is Obj COB V4 there is no support available from MF, or at least WE don't have a contract for it.

I am able to compile it successfully using Netexpress 5.1 using this:

COBOL %1.CBL,%1.OBJ,,LASTONE.LST DB2(DBMAN=DB2) DB2(BIND=%1.BND) DB2(DB=HMS1800) DB2(NOINIT) DB2(NOPRE) DB2(ISOLATION=UR) REF STICKY-LINKAGE OOCTRL(-G+N-P+Q-W) /CONSTANT VN (1800) /CONSTANT GOGO (1);

As you can see, there are no logon credentials needed for this.

Ian

I understand what you are saying and if the parameters above work, it is
puzzling.

Nevertheless, the error code from DB2 is quite clear: an incorrect length
parameter has been passed to the login routines of the DB2 .DLL being used.

The code SQL4903N is a standard "missing or incorrect length parameter" and
is used for many called routines.

I just looked at your post again...

"I get a SQL4903N error. The length of parameter "runtime_pid" of function
"sqlastrt_trusted" is not valid."

Looking at this again, it looks like the parameter may be a process ID or
maybe a passed account ID (just speculating here...). It is a startup
routine that it is being passed to, and it looks as though a trusted
connection may have been specified (which would not necessarily require
login credentials, provided the target database is able to be connected and
will accept a trusted connection.) I wonder if it passes the process ID of
the compile process as a trusted login id? (Again, just speculating...)

Can you think of anything regarding logins for Windows or DB2 that could be
checked? Is there a list of DB connection strings stored somewhere by MF?

Beyond trial and error (given there is no official support) I can't suggest
much more.

One more thing though... Have a look at the documentation you have for the
SQL parameter and see if it CAN take login credentials, then mess with that.
You will be no worse off... :-)

Sorry,
