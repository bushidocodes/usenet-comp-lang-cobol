I once worked on a telephone billing system that handled several hundred
million records per month. The records came from the telephone switchgear,
via some unix boxes, to the mainframe. The records arrived sorted
chronologically (i.e. in the order in which they were created). The result
was your telephone bill.

If you seriously believe that creating indexed files prior to processing is
the correct way to handle that kind of volume then you have a lot to learn
about large scale systems development.

I concer with Karl.  The amount of data that I have to work with would preclude the use of VSAM (which are indexed files - ISAM on mainframes died years ago) as well.  And yes Tony - a flat file to a mainframer is a sequential file in no particular order etc aka QSAM to some.

Anyway - to take my data entry files and create a VSAM I'd have to write a program (COBOL of course :D) to write the records to the file.  Since these may not be in key sequence I have to make sure the size of the VSAM file when defined allowed for all the nasty splits such an operation could produce as the file is being written to.  I have no idea if an ISAM file on a PC has such things as CI or CS splits but VSAM on mainframes do and they cause loss of space for the file.  Anyway because of all these splits processing is slowed down.

Now - sorting the flat file into key sequence and using a utility to load the file makes a nice clean file and takes about 20% of the time.  Of course once the file is sorted it may not even be necessary to create a VSAM file because probably all you're doing is a report and now that the file is in order by the control breaks the program for the report is COBOL 101 (for most students - we've seen examples of others here).  

VSAM is great for master files or when it is necessary to access the file randomly - otherwise give me the good ol' sort.

Transaction files can be created in time-date sequence by adding a new
record at the end of the file.  This is probably the quickest and
easiest way to get that record stored outside of some database tricks. 

The second quickest is using a relative file, but often that requires
two I-Os unless you are a single user.  If you are a single user, it is
probably the quickest.

Anyway, there is no updating of keys, slowing things down.  On the other
hand, accessing data is slow, as you have to do a linear search for what
you are looking for.  But often you don't care.  That file gets sorted
and fed into another program which processes further.

I have even seen where a nightly job takes an existing keyed file and
copies it to a flat file.  This gets sorted with the daily transactions,
and then the keyed file gets recreated from this sorted combo.  This is
done to make the searches more efficient, at the cost of not having
today's data available.
