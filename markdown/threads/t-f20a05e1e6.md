Clayton, 

There is a very easy way, assuming your data is always terminated with
a space:

STRING VARIABLE-1 
	VARIABLE-2 DELIMITED SPACE INTO VARIABLE-3.

If your data might be terminated with other characters or might
contain an embedded space, it gets more complex; you might look into
the UNSTRING verb for more possibilities.

STRING and UNSTRING are incredibly useful for text manipulation...

I am surprised that no one suggested the NUMVAL function. That is what I am
using for all numeric data now, and it seems to be more reliable than
unstring.  It will handle a number delimited by NULL, quite nicely, for
example.  That would break the unstring method, and is common with data
coming from C.

COMPUTE proper-cobol-numeric =
    FUNCTION NUMVAL( screwy-input-data-string).

To do the same with unstring, you really need:

UNSTRING screwy-input-data-string
    DELIMITED BY (NOT NUMERIC)
    INTO proper-cobol-numeric.

Which, I believe, is syntactically incorrect, though it would be nice to be
able to do.

of these verbs inefficient?  Parenthetically, I was pleasantly surprised
at the code generated for reference modification when I wrote a parsing
program to find all uses of CALL and COPY in a program.  

On the other hand I have found these areas of inefficiency:
1.  FUNCTION INTEGER-OF-DAY and FUNCTION INTEGER-OF-DATE (generates a
lengthy preamble, calls a subroutine that returns the integer in
binary).  I have written routines in COBOL that are substantially
faster.

2. ADD FIELD-A TO FIELD-B where both fields are PACKED-DECIMAL generate
add packed, zero add packed.  I think the ZAP is to take force a
potential negative zero obtained by an addition that also causes an
ignored overflow such that the remaining digits are zero.

3.  ADD FIELD-A TO FIELD-B where both fields S9(9) BINARY generates
double precision binary arithmetic.  I believe this is because the
compiler doesn't differentiate between this and ADD FIELD-A FIELD-B ...
TO FIELD-N where you have to take into account intermediate overflow.

4.  DIVIDE A INTO B GIVING C REMAINDER D where all operands have the
same number of decimal places doesn't take advantage of the fact that
the divide operation provides a remainder.  The code generated is DIVIDE
A INTO B GIVING C, COMPUTE D = B - (A * C).

couple of cases, they were my programs where I spent the time to
eliminate ALTER statements and make the programs structured.  When

reduction in CPU time.  I also have sometimes rewritten code because it

advance.  After all they were paying the bill.

span of the code, and the amount of testing that would be needed anyway
all influenced the amount of change I made.

'EXIT PARAGRAPH' is neutral in terms of complexity or proneness to
bugs.  GO TO xxxx-EXIT is not neutral.  Actually it is not the GO TO
that is the problem, it is the fact that it needs a label to be the
target for the GO TO.  The EXIT PARAGRAPH does not require such a
label.  

The addition of the xxxx-EXIT label into the program to be the target
for the GO TO introduces several requirements, each of which can be
error prone.  

For example: 

* The 'paragraph' must now be PERFORMed THRU, or must be a SECTION
(which requires that it be terminated by the next 'paragraph' also being
made a SECTION).  

* There must not be any code inserted between the 'paragraph' and its
EXIT label (wheras when there is no GO TO and only paragraphs are
performed there is no restriction on where new paragraphs can be
added).  

* Every GO TO and every THRU must be checked that the correct EXIT
paragraph is being referenced.

None of these problems occur with EXIT PARAGRAPH.

None of these problems occur if the only PERFORMs allowed are of
paragraph labels only, no THRU, no SECTIONs.

It seems that those who use GO TO don't realise that it is not the GO TO
that is the problem but the paragraph label that is the target.

CD 1.8, page 447, 14.10.13.3, GR 10, control passes "to an
implicit CONTINUE statement immediately following the last
statement in the current paragraph." In other words, you EXIT
the PARAGRAPH.

What happens after that depends upon how control passed to
the paragraph. If control passed to the paragraph as the result
of "PERFORM paragraph-name", then it depends upon how
the implementor interpreted the following:

CD 1.8, page 487, 14.10.27.3, GR 3a, "... return is after the
last statement [of the paragraph]."

Since the rules for both EXIT and PERFORM refer to the same
point, "the last statement in/of the paragraph", it is ambiguous
whether control should return prior to the implicit CONTINUE
statement or after the implicit CONTINUE statement. If it is
interpeted as "prior to" then Exit Paragraph means fall-thru to
the next paragraph. If it is interpreted as "after" then Exit
Paragraph means return control to the end of the PERFORM
statement.

The question now is, does J4 think it ambiguous?
