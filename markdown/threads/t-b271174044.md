A couple of points about this example:

1) "USAGE COMP" declares a binary field which will be stored
in COBOL order (highest byte first).
A "C" int is stored in machine native order.
In Micro Focus COBOL, at least, a "USAGE COMP-5" is defined
to mean "native machine order". This is perfect for passing
to other languages, but has it's own drawbacks (for instance,
because you can`t know what byte order the field is stored in,
REDEFINES have to be handled very carefully (preferably
avoided altogether)).

2) The default type of parameter passing is BY REFERENCE (ie, pass
the address of an item). This means that in the C routine you're
calling, it must be received as a pointer. In the above example,
that means yet_another_sub(char [], int *i). In which case, you
have to be very careful of another thing - alignment. COBOL will
let you misalign data in records all you want and cope with it
(albeit with a performance penalty). C will not. If the host
system has a CPU which does not allow misaligned data access, then
accessing the "int" via the pointer may not work.
To be sure of things, only pass 01 level items BY REFERENCE to C
(I know you did in the example, but I'm just underlining that it's
more of a requirement than an arbitrary choice).

If you did want to pass the "int" on the stack, use BY VALUE in
the COBOL call (ie, CALL "xxx" BY VALUE ima-int). In this case, you
don't need to worry about alignment (a copy is passed) or byte
ordering (the COBOL system will swap the byte order of the copy for
you).
