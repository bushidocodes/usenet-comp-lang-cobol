I am writing a program that needs to process a VSAM file both sequentially
and dynamically.  The main processing loop will process the file
sequentially, retrieving all records where the alternate index contains a
value of spaces.  However, for each record within the main loop, I need to
call a sub-process to access the file dynamically using the same alternate
index (with different values).  My question is, how do I set the pointer in
the main loop to get the next sequential record (where the alternate index
contains a value of spaces) after returning from the sub-process?  If I do a
START where the alternate index = spaces, I'll just go back to the
beginning.  Will doing a START where the primary key > the last sequential
read key, followed by a READ where the alternate index = spaces work? (I
suspect this is a not allowed)  I'm a little new to VSAM, so any help would
be appreciated.  The following are some additional supporting details:

Basically, the VSAM file has the following structure (well, there's more,
but this is all that matters):

Acct #
Superceded by Acct #

I need to be able to build Acct Chains (ie., Acct # A was superceded by Acct
# B was superceded by Acct # C) using this file.  A chain starts with the
Acct # having a blank Superceded by Acct #.  So, to build the chain, I'm
using the following "logic":

READ VSAM where Superceded by Acct # = spaces
WHILE not EOF-Main DO
   WRITE Acct #
   READ VSAM where Superceded by Acct # = Acct #
   WHILE not EOF-Sub DO
      Write Acct #
      READ VSAM where Superceded by Acct # = Acct #
   END DO
   READ next VSAM where Superceded by Acct # = spaces
END DO

I'd like to avoid creating a "control file" of all records where Superceded
by Acct # = spaces to use in place of my main loop, if possible.

Thanks,

Sue
```

>The easiest, dirtiest way to do this is to define *two* identical files
>and have them refer to the same DSN in their DDnames.


More properly stated, have 2 file ID's, 2 SELECT statements which refer to
the same file/DDNAME. Define the file identifiers with different access
modes. Each file identifier will work as an independent pointer to the same
file.

Another solution would be to run SynSort/DFSort selecting off the Alternate
indexes containing SPACES. This in turn would generate What I call a TRIGGER
file to use as a control file.

You also can do what you suggest. Have 2 keys defined in WORKING-STORAGE one
for sequential access and one for random. Define the file identifier as
DYNAMIC access. You can then read the primary key sequentially, testing the
alternate key for spaces. When one is found, save off the primary key in
WORKING-STORAGE. Do your random access thing, then START the file pointer
back where the key in WORKING-STORAGE was pointing, then just READ NEXT.
There may be some minor problems to insure you get to the next key, but you
should be able to work that out.

I recommend using the first method with 2 file identifiers pointing to the
same file. I have had problems in the past doing this with sequential files,
but you should be alright with VSAM.
```

Hi Sue - the best way to do this is probably to think out of the box. :) 

Use ACCESS MODE IS DYNAMIC in the SELECT statement, and have the first
program process the file sequentially and CALL a second program that open 
the file and processes it dynamically.

You would of course, CALL the second program USING the appropriate key 
value, which you just happen to have available. ;) 

The second program can run around all it wants in the file and
it will not disturb the first program. (Well, within limits.) 

-Paul
