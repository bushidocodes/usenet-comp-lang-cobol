For what it is worth (possibly nothing <G>)

The current draft revion of the COBOL Standard introduces two new features:

 - Dynamic Capacity Tables
 - Dynamic length elementary items

The former actually provides for ODO's where only the "currently needed" 
storage is allocated (and this amount can either grwo or shrink, based on 
explicit or implicit needs.  (These tables may be nested in other dynamic 
capacity or "traditional" tables)

the latter is (sort of) like C null-terminated strings, i.e. it is an 
elementary itme that can either be prefixed or suffixed in such a way to 
specify its current length - again this can change during execution

The specifications in the draft revision have gone thru many variations and 
I think they are now relatively reasonable - from a programmers point of 
view.

HOWEVER, as has been stated in the other thread, from what I have heard 
run-time "cost" of such features many turn out to be prohibitive.  Also, as 
stated elesewhere, the influence of the Standrd is minimal.  Both of these 
make me wonder if either or both of these features will ever make it into 
any generally available COBOL compilers.  Nevertheless, these are features 
that also demonstrate an ATTEMPT in the Standard process to provicde 
"well-defined" fetures that are in response to actual COBOL programmer 
"needs" (well they were needed in the '70s, 80s, etc).

Anyway, I think both of these could be quite useful.  Especially for the code I wrote recently.  I know some(!) hate ODO, but I've found it to be of great value, if somewhat awkward.  Basically, I've used it to "simulate" the "dynamic length elementary items".  Or really the "any length" data items; I'm not sure that these are the same...

The simplest example is this: a routine can be called with a text string that varies in length.  So the call might look like this:

PERFORM WITH TEST AFTER UNTIL DONE-SW = 'Y'
    COMPUTE MY-CHUNK-LEN = CONTENT FUNCTION LENGTH(MY-CHUNK)
    CALL MY-SUBR USING MY-STRING, CONTENT FUNCTION LENGTH(MY-STRING)
                       MY-CHUNK, MY-CHUNK-LEN
                 RETURNING DONE-SW
    DISPLAY '(' MY-CHUNK (1:MY-CHUNK-LEN) ')'
END-PERFORM

(I actually use "LENGTH OF MY-STRING" instead of "FUNCTION LENGTH(MY-STRING)", so I'm not 100% sure the latter is supported, but I believe LENGTH OF is an IBM extension).

The subroutine has code like this:

WORKING-STORAGE SECTION.
77  PS-POS            PIC S9(9) COMP-5 VALUE 1.
77  DONE-SW           PIC X VALUE 'N'.
    88  DONE                VALUE 'Y'.
77  CHUNK-MAX         PIC S9(9) COMP-5 VALUE ZERO.

LINKAGE SECTION.
01  PASSED-STRING.
    05  PIC X OCCURS 0 TO 9876 DEPENDING ON PASSED-STR-LEN.
77  PASSED-STR-LEN    PIC S9(9) COMP-5.
01  CHUNK.
    05  PIC X OCCURS 0 TO 500 DEPENDING ON CHUNK-LEN.
77  CHUNK-LEN         PIC S9(9) COMP-5.
77  IS-DONE           PIC X.
PROCEDURE DIVISION USING PASSED-STRING, PASSED-STR-LEN
                         CHUNK, CHUNK-LEN
                   RETURNING IS-DONE.
    IF CHUNK-MAX = ZERO
        MOVE CHUNK-LEN TO CHUNK-MAX
    END-IF
    IF NOT DONE
        PERFORM PARSE-STRING
    END-IF
    MOVE DONE-SW TO IS-DONE
    GOBACK.

PARSE-STRING.
    MOVE CHUNK-MAX TO CHUNK-LEN
    MOVE SPACES TO CHUNK
    UNSTRING PASSED-STRING DELIMITED BY X'19'
             INTO CHUNK CNT CHUNK-LEN
             WITH POINTER PS-POS
    NOT ON OVERFLOW
        SET DONE TO TRUE
    END-UNSTRING
    PERFORM HANDLE-CHUNK
    .

HANDLE-CHUNK.
    DISPLAY QUOTE CHUNK QUOTE
    .

I'm out of time at the moment, but I do want to point out the usefulness in this case of ODO.  Both the UNSTRING of PASSED-STRING and and the DISPLAY of CHUNK work beautifully with it; very readable.  If not for it I'd have to explicitly use reference modification, which while useful, would not be nearly as elegant in this case.

Tomorrow(?) I will post how I imagine this would look with recent enhancements to COBOL in the standard.
