```
Somewhat a "follow-up" on several recent and ongoing threads, I thought I
would say what I think I would do if I were to create a "best practices"
document (for new COBOL programmers -and for shops considering new or
revision to existing in-house standards).

1) Start by stating which compilers and environments I was most familiar
with - and what others I know about "slightly". Finally list other compilers
and O/S that I do not have personal experience with - so my comments may be
INCORRECT for such environments.

2) Say that the most important thing that should be done in creating ANY new
COBOL program or doing "significant" modification to existing code is to
find out what your goals and requirements are.  Some things to find out:

 A) Is your program intended for "online" (or interactive) use - or "batch"
processing.  If the former, what type of response times are
expected/required - if the latter, what is your "window" for completion?

 B) Related to the above, what is your expected "volume" (transactions or
"records") and what type of "hardware" power will you have?  Is this
expected to change (grow) over time and if so how rapidly?

 C) Is it currently expected that the source code will need to be ported to
other O/S environments?  other compilers on the same environment?  (How many
times in the past has your "shop" - or customers - claimed it wouldn't need
to be ported, but then requested it? How many times have they asked for
"portable" code that actually remains on the same O/S with the same compiler
for its full "life-time"?)

D) Is either the input or output must be shared across platforms (operating
systems) or across programming languages (including to/from other COBOL
compilers)  Are you expected to interface with some Database system?  If so,
is that system "portable" and/or its access available across systems.

3) Indicate that when doing "minor" maintenance on an existing program, it
is almost always best to use the STYLE and practices of that program and NOT
try and throw "new" techniques into an existing and functioning program.

4) Does your shop use a "proprietary" Operating System (primarily?
exclusively?) if so, is there a "proprietary" COBOL product set for that
environment sold by the same vendor?   If not, is there a "de facto"
compiler for the O/S or environment?

5)  Do you expect to sell this application to other sites or to customers?
If so,
 A) Is it safe to "assume" that the purchasing sites already own the
"run-time" for your compiler (e.g. IBM and LE) or .NET with "new" Windows
environments - or must each customer machine get a "new" copy of the
run-time?
 C) Will you be distributing your application as source code or object code?
If source code, are there any government requirements (such as *must* be
ANSI conforming)

6) In general, what are the "performance" versus "ease of maintenance"
trade-offs for this application?  For the site in general?  (Is it likely
that you "personally" will ALWAYS be doing the maintenance on this
application? Alternatively, will all maintenance go thru a "tight" audit
system for conformance to "shop standards")

7) If the "new" program is supposed to be a "quick&dirty" program for a
single use, what is the track record of your shop eventually moving such
programs "into regular production and maintenance"?

  * * * *

There are probably BUNCHES (no "pun intended") of other questions that I
think impact "best practices".  However, once these questions have been
answered, I would probably include such statements as:

 "-  Use of INDEXES versus SUBSCRIPTS ...
   Whether you use a subscript (data-item) or an index (INDEXED BY phrase)
to access a specific element may depend on what your shop (and maintenance
programmers) are most used to as well as what your performance requirements
are.  In a case where the specification of which element is to be accessed
relates "naturally" to some other data item that you are already
manipulating, then probably it is easiest to use this data item directly as
a subscript - rather than setting an index to that item's value.  On the
other hand, if you are "moving thru" the table elements in some algorithmic
way, the use of an INDEX may well be most self documenting.  When using a
subscript in a "tight loop" or heavily used procedure, it is usually
important to optimize the USAGE and PICTURE of the data item.   In many
systems, using a "full word binary" item performs the best for such a use.
However, you should check the performance information for your specific
compiler to determine what is the best definition for it - and what the
overhead is in "moving" a value to such an item.  For infrequent references
to a table element (within an "actual run of a program), there is probably
minimal performance difference between indexes and subscripts.  In such
cases, using the "most  well known and understood" technique in your shop
will probably be your best choice.

  NOTE:   Read your language reference material for further information on:
  - Index data-items versus Index-names
 - Relative subscripting and indexing
 - Search versus Search ALL and index requirements"
```

```
There is an HISTORICAL (not current with the '85 Standard) reason for this -
that is also unknown to IBM mainframe programmers from their '74 compiler.

In the '74 Standard, one wasn't able to "transfer control" outside an Input
or Output procedure.  Therefore, as the Input or Output Procedure itself was
executed "only once" - the only '74 Standard way of "looping" thru the
procedure (which, of course you probably wanted to do) was to GO TO the
start of the input/output procedure from the "end" if there were more
records to release/return.  ALTERNATIVELY, you could code a PERFORM THRU
within the Input/Output procedure, but in that case you had to "GO TO" some
sort of exit after the last record to avoid "falling thru" to the performed
procedure after the "mainline" of your procedure was completed.

Many programmers never "noticed" that this restriction was removed with the
'85 Standard (and '85 Standard compilers) and you may now PERFORM THRU and
"outside procedure" from within your Input/Output procedure without ever
doing a GO TO.
```

```
This relaxation has an interesting side-effect.  It's no longer reasonable
to determine for certain *at compile time* whether a RELEASE or RETURN is
being executed from within such a procedure (as the SYNTAX rules for these
verbs require).  The 2002 standard changes this to a GENERAL rule to correct
this; in any event, if it's to be unconditionally prevented either the
compiler or the operating environment has to take specific steps to do so at
execution time.

It is also not reasonable to determine for certain *at compile time* whether
a SORT or MERGE is executed from within such a procedure (illegal), but this
is correctly a GENERAL rule and need not be enforced at compile time;
likewise, if "nested SORTs" are a problem, "somebody" must actively prevent
them at execution time.

(Note that I originally had "possible" where I now have "reasonable".  It is
*possible*, at least in theory, to detect these violations at compile time,
but in complex programs it can be done only by predicting all the possible
execution-time paths through the program -- just as it is *possible* to make
sure that all execution-time paths that read a file are preceded by an
execution-time path that opens the file appropriately -- but I for one think
doing such analyses is far beyond what one should reasonably expect from a
compiler!).
```

```
That used to be at the top of my list, as well. Twenty years ago I would have
agreed, perhaps even ten years ago. In the old days, our system designs were
heavily influenced by hardware performance issues. The tail was wagging the dog.
Nowadays, hardware is fast enough that we can usualy design for application
requirements while ignoring machine constraints. An exception is database
access, were some features such as unions are still too slow.

If performance is an issue, write prototypes to measure speed of alternative
approaches. Real-world data is more valuable than projections based on
experience which may be obsolete.

I concur. There is no way of knowing whether code must be portable. The

Again, nobody knows what will happen in the future. The safest course is make

The original author sets the de-facto standard. As you said above, maintenance
should follow his style. The 'style police' are usually called in for
maintenance rather than original development. When there is a conflict between
his style and the 'shop standard', they will accede to following his style. The
alternative is a complete rewrite, which they lack the power to authorize. 

Hopefully the new style will become a precedent to change the shop standard. Not
because it is better, solely because it is 'in production'.

Happens all the time. Version control is one way to draw the distinction
between production and one-time. A better way is execution logging. When I see a
program run ten times in a month, that's not a 'one time' program. It needs to
be ramped up to production quality, documented and designated production.
```
