YukonMama,
Our shop has had STROBE for over two years.  Actually, we had it some years before
as well, but dropped the license.  We have documented improvements in applications
that have resulted in machine cost reductions of over $600,000.  We continue to
study programs, and find candidates for improvements.  Most of the biggest savings
have come from either improving table handling* or introducing "smart" alternatives
to mindless* data base access.

I have found that the most important factor in using a performance analysis tool
like STROBE is knowing where to look.  That is, you can't save much money unless
you are looking at the jobs that cost the most.  Oftentimes, programmers may think
they know where to look, but programs may slip thru the cracks just because they
have been running for years (maybe decades) without any attention from anyone.

If you really want to do your shop a favor, create a report of the most costly
batch job steps.  (I'm assuming you'd know if an on line transaction was seriously
out of whack, because response time would be so bad that users would complain.)
Choosing the "top jobs" from that report, run STROBE against those big resource
users.

If you are lacking such a report, and don't know how to get one, I suggest
selecting System Management Facility (SMF) Type 30 records from your production
system (maybe one or more LPARs).  Here are SORT statements I use to select what I
want:
  SORT     FIELDS=(11,4,PD,D,7,4,BI,D)
  INCLUDE  COND=(15,4,CH,NE,C'SA90',AND,       SYSTEM ID  =PROD LPARS
                 15,4,CH,NE,C'SD90',AND,
                (06,1,CH,EQ,X'1E',AND,         RECORD TYPE=30
                 245,1,CH,EQ,C'2',AND,         JOB CLASS  =2
                (23,2,BI,EQ,X'0004',OR,        SUBTYPE    =4 OR 1
                 23,2,BI,EQ,X'0001')))
I eliminated two LPARS, SA90 and SD90, chose only a specific Job Class, and certain
"subtypes".  I would have to go look again to see what is significant about the
subtypes.  In a reporting program, I drop records with less than a threshold CPU
time.  That selection could be added to the INCLUDE above.  We have a relatively
fast CPU, and I drop steps of less than 30 CPU seconds.  We have another shop that
runs a CPU of about 1/3 the speed, and I also look at their "big hitters".

You could get a decent report using SORT facilities.  I wrote COBOL programs to do
the job.  If your shop uses the infamous MICS tool to do billing, be aware that it
is SAS based, and is itself likely to be one of the costliest things you run.  I
prefer to read the SMF data directly, and then prove to the billing people how much
money they're wasting with MICS.

You'll need to read SMF documentation (for type 30 record contents), which can be
found in BookManager on your system, in the OS/390 or z/OS Bookshelf, or on the
Web.

*Improving table handling often entails changing sequential lookups to binary
searches.  Sometimes, table contents are in the same sequence as the records they
are being matched against, so saving the last occurrence number can provide the
starting point for the next search.  Several different "solutions" are possible;
this takes analysis and thought.

*Mindless means doing a data base access to validate some data element in millions
of records, when there are only a few valid values for the data element.  After the
data base information is returned, and the validity is established, it can be
tabled so that subsequent validations are a table lookup.  In one case 10 millions
DB2 calls were reduced to 95.  You can imagine how much CPU time was reduced, as
well as elapsed time.  Once, we found that an IMS data base was dumped before the
program that was doing "lookups" against it -- we changed the program to a two file
match and saved over 95% of CPU time.

This is a little out of date for your question but here is one example
on a case statement in DB2

SELECT
    CASE CDL.AD_VAL_SVC_ENB_IR
       WHEN ' ' THEN 'SPACE'
       WHEN '0' THEN '0'
       WHEN '1' THEN '1'
       WHEN '3' THEN '3'
    END                                    AS "AD_VAL_IR"
   ,COUNT(*)                            AS "UOW3.5.15"

FROM
        D168AA.TCMSCUS  AS  CUS
   JOIN D168AA.TCDLTBL  AS  CDL
     ON CUS.CDL_NR = CDL.CDL_NR

This will subsitute the word 'space' for a char of ' '.
