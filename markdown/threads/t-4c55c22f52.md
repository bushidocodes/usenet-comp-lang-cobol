```
On PC's there is no such thing as record length, because there are no such
things as discrete physical records: all files are simply streams of bytes.
It is up to the application to decide if there are logical records of a
fixed or variable length within the byte stream.

As for your specific example, if "DOC" refers to a Microsoft Word (r)
document, you've hit the triple jackpot, because DLL, DOC and XLS files are
all built from a series of variable-length records in which there are either
"assumed" or explicit (embedded) logical record lengths.
```

```
A hex editor (for example) stops when told by the OS it

way to tell the length of a RECORD. There isn't one.

>

> that is not possible in COBOL?

There is NO general purpose "End-of-Record" marker for ANY
language, let alone COBOL. IF you know, in advance, the file
is of a certain type (say an ordinary text file), then you may make
use of the conventions of that file type to detect when you encounter
the end of a record (in the case of a text file, the convention is
Carriage-Return or Carriage-Return/Line-Feed).

The operating system informs your program when the end of the
file is reached. The OS gets the length of the file from the file's
directory (you can too, but not directly with COBOL - you must
invoke either and OS API or a vendor-supplied subprogram).
```

```
There have been LOTS of replies to this note - and I won't go over their
information in my reply.  However, I did want to mention one thing that is a
BIG shock to those moving from (IBM - probably most) Mainframe environments
to PC environments.

On (IBM) mainframes, there is an entire "product" (or 2) that defines
"FILES" and these are available to all programming languages and software
packages (e.g. QSAM, VSAM - older ones like ISAM and BDAM - and even ones
NOT supported "well" in COBOL such as BPAM and the lowest level of EXCP).

Every product (whether from IBM or a 3rd party) uses THESE file formats -
and they all understand each others files.  (Well actually, some vendors -
including IBM use "funny" records occasionally to create "proprietary" file
formats - but even those are based on the "basic" ones that come with MVS,
OS/390, z/OS, VM, etc)

  ***

Now in the PC (and UNIX) worlds, there is NOTHING like this.  Usually
(possibly always) the Operating System "knows" about

   byte-stream I/O (read a file, executable, or  even "junk" one byte at a
time)
    AND
   line sequential files (a file where CR/LF - carriage-return/line-feed -
or just LF) indicates a "break" between records

*ALL* other "file types" need to be provided by the specific software
vendor.  This is why a Micro Focus "variable length sequential file" canNOT
be read by a Fujitsu program (normally) - and an indexed file created by
(almost) any COBOL compiler is TOTALLY useless to the "native" I/O of any
other programming language.  There are some 3rd party product (such as
C-ISAM and Btrieve) that are specifically INTENDED to provide
"multi-language" support for files, but they cannot be assumed as being
present - much less used in creating a COBOL file.

When you start talking about .DOC, .XLS, etc files you are talking about
INCREDIBLY "proprietary" file formats.   You *can* parse them in COBOL - but
there is no way to do so without
  A) using that vendor's documentation on the file formats
  B) knowing what type of file it is BEFORE you try and read it.

 ***

The best example, I can think of for an IBM mainframer is to ask you to
imagine the task (in a COBOL program) of figuring out the actual UNDERLYING
VSAM, OSAM, whatever record size for a SQL or IMS record.   It could be
done - but certainly not very easily.
```
