Robin,
Not to be too picky, but COBOL is an acronym, and should be capitalized.

My first reaction to your post is wondering why you'd want to do such a
thing. Maybe it is your "fix" for Year 2000. If so, please give it
more thought, then avoid it like the plague!! Do date expansion for the
"final" solution, or some type of windowing as a second choice. Don't
mess up your data for now and forever! Remember, the "good data" you
have now will be disappearing from your files and data bases over time,
and someday all you'll have left is the junk you seem to be creating for
years beyond 1999.

Date expansion usually is thought of in terms of larger records, but
using the technique does not mean that data records HAVE to get larger.
Character dates can be packed (putting 8 digits into a 6 character field
with room to spare) or converted to hex (requiring a subroutine to
operate on them arithmetically, but not to compare or sort them), years
can be converted to BINARY, etc. None of these options is totally
effort free, of course.

Also, remember that IBM's DFSORT and Syncsort have added facilities for
"Y2K sorting" of years, so maybe you're planning to mess up your data
for no good reason. The SORT extensions cannot, so far as I know, be
used in a sort operation invoked from a COBOL program. In such
circumstances, a modification like this to the data would be necessary.

Now, maybe I'm guessing wrong. If so, here's what I'd do, if I were
programming on an IBM 1985 ANSI COBOL compiler (VS COBOL II, COBOL for
MVS & VM, VisualAge for COBOL for OS/2, etc.):

IBM has added an extension to their COBOL language for the defining of
non-numeric literals as hexadecimal values. This allows you to define a
value in hex:
01 HEX-VAL-A00 PIC X(02) VALUE X"0A00".
(A lower case 'a' thru 'f' works, too.)
01 HEX-A00 REDEFINES
HEX-VAL-A00 PIC S9(4) BINARY.
Then, just redefine this item as USAGE BINARY (or COMP, but BINARY is
part of the COBOL language, while COMP is implementor dependent),
redefine the other item (which is probably now defined as character) as
BINARY, and add/subtract the two.

Other vendors may have the same extension.

Remember that this won't properly take care of all cases. Carry out of
a hex 'nibble' can cause overflows, etc. To do that, write a subroutine
and CALL it, rather than putting a lot of code into each program.

There is a quick and dirty way around.
Hexadecimal values are known in Cobol as COMPUTATIONAL-1, COMP-1, BINARY
and so on depending upon the Cobol you are using.
The trick here is to Re-Define the Display Variable as Comp-1, now make
whatever aritmatical operation you wish on the 2 variables and that's
it!
In other words, you will have someting like this

01 VARIABLES
05 DISP-VAR PIC X(02) VALUE "00".
05 HEXA-VAR REDEFINES DISP-VAR PIC S9(04) COMP-1.
05 HEXA-VALUE PIC S9(04) VALUE X'F00'

PROCEDURE DIVISION
ADD X'F0' TO HEXA-VAR.
ADD HEXA-VALUE TO HEXA-VAR.

Now DISP-VAR would contain x'FF0', wich you will see in your
screen/printer provided it is a printable char.
