```
Hello all, I am having difficulty passing tables of varying sizes to a
sub-program which I expect to design to process them. As I want to keep the
calling mechanism as simple as possible ie.

CALL "SUBPROG" USING WS-TAB.

the problem gets slightly trickier. What I would like to know is how to specify
the linkage in 'SUBPROG'. My solution would force the calling program to contain
details of the tables dimensions before it was called:

01 WS-TAB.
03 WS-ROWS PIC 9999 COMP VALUE 2400.
03 WS-COLS PIC 9999 COMP VALUE 55.
03 WS-ROW OCCURS 2400.
05 WS-ITEM PIC X(55).
.....
.....

CALL "SUBPROG" USING WS-TAB.

Therefore the linkage section definition would look something like :

LINKAGE SECTION.
01 LS-TAB.
03 LS-ROWS PIC 9999 COMP.
03 LS-COLS PIC 9999 COMP.
03 LS-ROW OCCURS 1 TO 9999 TIMES
DEPENDING ON LS-ROWS.
05 WS-ITEM PIC X OCCURS 1 TO 9999 TIMES
DEPENDING ON LS-COLS.

Has anyone any experience of dealing with large tables such as this ? or anyone
got any other solutions. I would like a degree of device independence - so
assembler is out.

TIA

Nigel Benson
```

```
Nigel,

Is your problem that the table is too big or are you having trouble with
addressability?

You could try changing the ODO objects in your linkage section to working
storage fields, initialized at the maximum occurs via VALUE clauses, then move
LS-ROWS to WS-ROWS, etc. As long as you don't index beyond the acquired
storage, I think it might work.

If that doesn't work, try defining an elementary item in your linkage section
the size of the largest possible area passed and then redefining it with your
variable table. (I ask myself, can you use REDEFINES in the LS? Maybe you
can't...).

If that doesn't work, maybe you can define the variable table in working
storage, use the LS-ROWS and LS-COLS to set WS ODO objects, then do a move from
a single linkage section elementary item (defined after the LS-ROWS and LS-COLS)
to the working storage table.

NOTE: Your real problem *may* be related to the use of COMP fields and boundary
alignment (slack bytes). Try changing them to COMP-3 and see if it makes a
difference.

Anyway, if none of the above work, well, maybe someone else has a suggestion.
It's getting late and I'm afraid I might be babbling ;-)
```

```
Your proposed code looks basically good. I would, however, strongly
suggest making your WS-ROWS, WS-COLS, LS-ROWS and LS-COLS signed
variables, e.g. PIC S9(04) COMP.

As an example of the popularity of this approach, it is the same as IBM
COBOL OS/VS, COBOL II, etc. use to pass parameters from JCL to the COBOL
program. They are passed as a single dimension table of varying length,
with single character rows. Example:

LINKAGE SECTION.
01 PARM-AREA.
05 PARM-LEN PIC S9(04) COMP.
05 PARM-DATA.
10 PARM-CHAR OCCURS 0 TO 100 TIMES DEPENDING ON PARM-LEN.
PROCEDURE DIVISION USING PARM-AREA.
```

```
How about the following, where Program A calls Program B?
This assumes that there is a finite set of table layouts that could be
processed in Program A. I.e., you don't have to handle all tables from
(1,1) to (999999,999999), just some identifiable subset that can be
uniquely depicted.

The key concept here, as pointed out by others, the data (in the tables)
referenced by program B is actually located in Program A's storage.
There are not two copies of each table when 'passed' from A to B.
Nothing but addresses are being transferred into program B's storage
areas. (In a dump for the data assigned to the Linkage areas in B, one
can see their addresses are actually in A.) So it's simplest just to
uuse the same layouts in the two programs.

So:
1. In Program A, define each potential table as an 01 with 03 ROW OCCURS
and 05 WS-ITEM, etc.
2. Call Program B using TABLE-TO-WORK-ON, TABLE-A, TABLE-B, ... (i.e.,
all the defined tables)
3. In program B, LINKAGE contains the same 01's as defined in Program A.
4. Program B: PROCEDURE DIVISION USING INCOMING-TABLE-NAME, TABLE-A,
TABLE-B, ... (all of them)

IF INCOMING-TABLE-NAME = TABLE-A
PERFORM TABLE-A-ROUTINE
IF INCOMING-TABLE-NAME = TABLE-B
PERFORM TABLE-B-ROUTINE
...

As an enhancement:
If (while in Program A) you can decide that only a sequential group of
rows (beginning and ending somewhere) of TABLE X are filled:

CALL PROGRAMB USING INCOMING-TABLE-NAME, START-ROW, END-ROW, TABLE-A,
TABLE-B, ... (all of them)
Then in TABLE-N-ROUTINE, PERFORM HANDLE-TABLE-N VARYING ROW-INDEX-N FROM
START-ROW UNTIL ROW-INDEX-N GREATER THAN END-ROW.

All the above assumes you don't get beyond the size limits, etc. that
your compiler and COBOL version can handle. I don't know how any things
can be in a PROCEDURE DIVISION USING... statement but I would guess at
least 255.

It's possible I'm depicting a simpler situation than the one you're
faced with.
```
