1) Every batch program run under MVS, OS/390, or z/OS has the ability to return
an integer, called the Return Code or Condition Code to the JCL stream; the
value must be in the range 0-4095 decimal; the convention is (although you are
not obligated to follow it) that a return code of zero says the program ran
successfully and that non-zero return codes indicate some kind of error, with
larger values implying more sever errors

2) The JCL for a step in a job can test the return code value from any previous
step in the same job, to determine if the current step should run or not; if
you don't do return code checking, the current step runs, by default, unless a
prior step abends. So you might have steps that you only want to run under
certain conditions.

A common example is a job to do a compile and link. All the compilers (and the
Assembler) put out return codes of  0, 4, 8, 12, or 16 (and maybe some do 20),
where '4' generally implies warning level compile errors. So you could set up
the link step to test the return code from the compile step, and if the RC is
greater than 4, do not run the link step because there were serious compile
errors. You can do the same thing for your own application batch steps.

The way to set RC in a program varies depending on the language you are coding
in.

There are two ways to test RC values in JCL. 1) use the COND parameter (not a
good idea any more); 2) use the IF/THEN/ELSE/ENDIF JCL statements (the
recommended way).

Just one other "warning" about RC (Return-Code) and COBOL.

As has been discussed in this NG before, there are some products (such as
DB2 - and even infrequently CICS) that are not "nice" about clearing R15 all
the time.  What this means is that if you have a COBOL program that CALLs
(uses) these facilities (and they don't all have to be in-house or IBM
products/applications) and then your COBOL program "exits", the SYSTEM will
use whatever is "left" in R15 (register 15) as the Return-Code that is
"reported" back to the system.

This *may* cause SEVER problems for subsequent programs in your job stream -
and even if this is a "stand-alone" program, will be QUITE misleading.

When/If this happens, then you should add a "Move Zero to Return-Code"
statement in your application program just before doing a Stop Run or final
GoBack.  I do *not* recommend doing this "in general" - but only as a
"solution" for applications with "misbehaving" subprograms. (On the other
hand, I can't think of any "problem" in adding it in programs that don't
need it - when you are POSITIVE that "all is ending well")

============
If you have a batch job that consists of many programs, the return
code is very useful, because it allows you to run or suppress
subsequent job steps (programs) depending on whether a program ran
successfully.

In the IBM mainframe world, you can set the return code within your
program, and then write JCL that evaluates that return code.  You can
then take different actions for different return codes.

In addition, the system (IIRC) will set a unique recturn code if your
job terminates abnormally.  Normally, this causes the system to flush
the balance of your job.  (It abended, right?)

HOWEVER, in our mainframe batch job days, my partner and I discovered
that you could code your JCL so that it would run a program (job step)
even if a program abended.

What was that good for?  Suppose you have a job that consists of 60 or
80 programs that takes hours to complete.  Suppose, also, that there
are points along the way where the files are updated in such a way
that they are valid.

Now:  Suppose that, at those points, you take a quick backup of your
disk files before continuing with your job.  Then, if a program
following your backup terminates abnormally (leaving the files in an
inconsistent state of being updated), you can code your JCL to restore
them from the latest tape backup so that the online portion of the
system will come up properly the next day.

--> This also means that you don't need to get up and go to work at
2AM to restore the files yourself. <--

No one at the State of Alaska data center knew you could even do this,
and the first couple of times it happened, we STILL got calls from the
operators wondering why the job that just bombed was requesting a
tape.  "Just load the tape and let it do its restore," we told 'em,
"and don't call me next time!"

Worked like a champ.  We'd fix the bug the next morning, knowing that
the online system would be up and running the next day in spite of the
abend.

Amazing what you can learn if you CAREFULLY read the JCL manual!

EXECUTES, gets a RC, either from the program or the system.

Please note, here 'curious',

When the Program Does Not Have Control

For the system to act on the COND parameter, the step must abnormally
terminate while the program has  control. If a step abnormally terminates
during scheduling, due to failures such as JCL errors or the inability to
allocate space, the system bypasses the remaining steps, no matter what the
COND parameter requests.

steps can be executed/bypassed based on the RC of previous steps.
Communication in the opposite direction (from JCL to program) can be done
via the PARM option of the execute statement.  See the JCL manuals at the
www.mvshelp.com site for more details about RC.
