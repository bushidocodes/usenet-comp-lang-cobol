It's some time since I used DB2 when I don't think it was then possible
to use host variable in LIKE predicates, you now can in version 8.  I
don't know what version you are on or in which version it became
possible.  The SQL reference manual indicates that the host variable
must be a character string variable, preferably a varying length string
variable as the lengths for fixed length host variable indicate that
the lengths must match, and then refers to the application programmer's
guide, which doesn't seem particularly helpful, e.g. it doesn't seem to
provide examples of such use.

You could also try subscribing to the DB2-L listserve group which is
purely email, presumably to reduce spamming risks.  You can find
instructions to subscribe on http://www.idugdb2-l.org/

I've never worked either IBM COBOL nor DB2.  But, if your "LIKE" clause 
is put together like "1235%", you can use the "BETWEEN" clause instead. 
  In some RDBMS's and cases, BETWEEN uses indexes where LIKE will not. 
The format is

SELECT [something]
   FROM [somewhere]
  WHERE [something-else] BETWEEN '1234' AND '1234~'

Maybe the (pre)compiler will like that better.  Of course, if you're 
using a clause like '%456%', or specifying particular characters, then 
BETWEEN wouldn't be a workable solution.

> VALID

It is an _operand_ that is invalid (not the statement or the value).
The operand is WS-WihHolder-Like-1.  This should be declared within an
EXEC SQL BEGIN DECLARE SECTION END-EXEC with its corresponding END
DECLARE. Then it can be used as a host variable.

There are ways with some pre-processors for making the DECLARE SECTION
implicit so that all variables in W-S etc can be used as host variables
but the standard is that all host variables must be in an explicit
DECLARE SECTION.

It works with non-numeric values, but I don't think it's going to be 
able to replace that statement.  Basically, if the EIN is 123456789, 
this is going to find records with fields like...

123456789ab
12345678998
12345678909

...but would *exclude* records like...

123456789abc
1234567898765432

The between clause would also select these last two.  I saw earlier that 
someone got around a limitation by using Dynamic SQL - this may let you 
get around this, because the statement would be evaluated at run time, 
not compile time.  (The downside to this is that you lose the validation 
of the tables that (may) be happening at compile time - at least that's 
how ours works.)
