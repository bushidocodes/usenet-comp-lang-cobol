Depends on your purpose. I'm professionally still into Cobol (now IBM 6.2) but besides all the rinkydink things like Visual
somethings, Java, Javacript, PHP I see Python on the rise and I see a future for that.

The problem with most languages in my opinion is longevity. To many new and incompatible versions over a short period, too many
securityleaks, too many dialects and: many of them are tied to a specific company.

Python has it's quirks but seems well positioned to become quite big as it is a preferred language on SBC's like Raspberry Pi. I'm
using it myself as the backend for MQ (Mosquitto) and DB interaction (MariaDB) with more and more IoT (Arduino/ESP8266/ESP32)
clients. Those IoT devices can be programmed in Micropython which would mean a single language for all platforms (using i.e.
Jetbrains PyCharm as an IDE) but I'm using the Arduino IDE with a C like language. Reason: Library support. Support for what you
want to do would want to do should be a main consideration. Almost any current language has a lot of online support with snippets
available.

Finally: If you're a good programmer (better: software developer!) the language does not matter. In all languages it comes down to
being able to decompose your problem. The language is just a tool. Now, if you intend to make money in a new language different
consideration come into play....

You don't say what applications you want to write or what tradeoffs or
attributes you value the most in a programming language. For example one
tradeoff is dynamic typing vs static typing. On the average the first choice
offers more flexible and quicker development at the cost of slower execution
(I do emphasise , on the average). So without knowing more information I will
simply mention 2 languages that I like and which are not well known :

1. TXR at http://nongnu.org/txr

It's a Lisp dialect which tries to combine the best aspects of Common Lisp
and Scheme. There's abundant documentation so you can easily learn more.

2. oforth at http://oforth.com

It is a Forth dialect but less close to the metal than standard Forth. It
supports dynamic typing , garbage collection and has a built-in parallel
processing model with "tasks". These are oforth objects which model
parallel execution. You can map them to different operating system threads
but you can also have them all under the same operating system thread and
then the oforth system is responsible for switching execution between
tasks. You can also pass immutable objects between tasks.
