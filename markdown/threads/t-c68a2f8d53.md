```
*            CORRESPONDS TO FILE ASSOCIATED WITH DD SORTOF01
```

```
Normally I would say: "Read the Manual" or something like that. However, I
am in a good mood, and your problem is very general to many new students.
Your question can be answered in many ways, but here is a simple model,
everyone can use for writing reports:

You define a line-counter and a page-counter. Initial value of line-counter
is GT max. no of lines per page and initial value for page-counter is zero.

You define the headings including Disp-Page-Count (pic zzz9.).

Whenever the line-count is GT max. no of lines per page , perform your
heading (write after advancing 1 page). The first statement in your heading
routine should be 'Add 1 to Page-Count'. After that, 'Move Page-Count to
Disp-Page-Count. The last statement in your heading should be 'Move Zero to
Line-Count'

Whenever the line-count is LT max. no of lines per page + 1 (you can put an
'Else' upon GT  above) you should write a detail line - remember to 'Add 1
to Line-Count'. Write after advancing 1 line.

It is very easy to play the violin, when you know how.
```

```
I have separate code for throwing a page and printing headings.
The code that looks after all this is buried in copy books so
I never have to worry about it, but the first headings are
printed without throwing a page and a page throw is done at
the end of the program - even if the report is interrupted
and abandoned.

This always leaves the printer with the last printed page 
available to be torn off and pages are not wasted.

If the page is not thrown at the end of a report then the
user has to wind it through to make it available and then
the next report will throw a blank page.  Having the user
adjust the paper in _any_ way genearlly leads to lineup
problems.

With back fed or sheet feed (laser) printers it is possible
to design paper handling to avoid waste and avoid user
interaction.  I always set up the printers using micro-feed
adjustments so that the user has to just press the 'load'
button and the paper is perfectly aligned.

In one case there was wastage of invoices occurring when
cash sales were made because one form was wasted after
each print-run.  With cash sales a print run was 1 form.
(I didn't write the system originally). By replacing the
printer with a back-fed one and changing the code 
slightly the savings in stationary paid for the new
printer in 6 months.
```

```
The following example uses a header with three lines.  Use
as many header lines as is appropriate for your application.
You define the maximum number of lines per page in Working
Storage so it is easy to modify later.  If your compiler
has syntax for constants (e.g. 78 level in MF COBOL), that
is even better.

WORKING-STORAGE SECTION.
    77  PAGE-COUNT          PIC  9(04).
    77  LINE-COUNT          PIC  9(02) VALUE 99.
    77  MAX-LINES           PIC  9(02) VALUE 5.
...
    01  PAGE-HEADER-1.
        ...
        03  PH1-PAGE-COUNT  PIC  Z(04).
        ...
    01  PAGE-HEADER-2.
        ...
    01  PAGE-HEADER-3.
        ...

PROCEDURE DIVISION.
...
    PERFORM PRINT-ENTRY <until no more entries>.
...

PRINT-ENTRY.
    IF (LINE-COUNT NOT < MAX-LINES)
        PERFORM PAGE-HEADING
    END-IF.

    <move data to print record>

    WRITE PRINT-RECORD BEFORE ? .
    MOVE SPACES TO PRINT-RECORD.
    ADD 1 TO LINE-COUNT.
...

PAGE-HEADING.
    IF (LINE-COUNT < 99)
        WRITE PRINT-RECORD BEFORE PAGE
    END-IF.
    ADD 1 TO PAGE-COUNT.
    MOVE PAGE-COUNT TO PH1-PAGE-COUNT.
    WRITE PRINT-RECORD FROM PAGE-HEADER-1 BEFORE ? .
    WRITE PRINT-RECORD FROM PAGE-HEADER-2 BEFORE ? .
    WRITE PRINT-RECORD FROM PAGE-HEADER-3 BEFORE ? .
    MOVE SPACES TO PRINT-RECORD.
    MOVE 0 TO LINE-COUNT.
```

```
I dislike condition names (88 levels).  Though they appear to
make the code more readable when the code is written they make
it difficult to manage large projects later.

Given a typical system with dozens to hundreds of source files
it is usual (for me) to locate programs that use particular data
items by greping for them.  In this particular case I may do a
'grep Line-Count *.CBL *.PD >linec.grp'  then pick up the
output file in the editor.  This contains the file names so
I can jump from there to each file.

When 88s are used I also have to grep for each of these to
find all places where the data item is used.

I find it preferable to have a variable such as 'line-limit'
which is set to the default.  These items are in a copybook
'printer.ws' along with other items related to report printing.

In particular cases the 'line-limit' can be reset by the program
dynamically which an 88 level cannot.  This gives more
flexability in the common print routines.

For example I can set line-limit to zero to suppress page throws
(setting heading-count to zero suppresses headings).  It may
aslo be that if a landscape printer is selected I need to
change line-limit to around 52, while it can be 64 for the
same device in portrait mode.

I can also grep for where line-limit is changed with
grep " to line-limit" *.pd *.cbl.
```

```
You are correct.  I avoid ending a loop by testing the loop variable
being equal to some value, for two reasons.  First, as you point out,
you can 'miss' the value if the loop variable is incremented in an
unexpected way by some subsequent change to the program.  Second,
to my mind it is usually clearer, and expresses the essence of the
termination condition better, to use a relative test.  You stop when
you *pass* the boundary conditions.  Using equal does not express that
as clearly.  Always try to use a test that is clearest and makes the
most sense.  For example, if you are looping through values 1-5, it
makes more sense to test for > 5 than for not < 6, since 6 is outside
the range.  If you test against the highest or lowest values in the
range, then it is easier to find the references later.  Even better,
use a constant so you can change it in one place.

Of course, it is perfectly permissible to end a loop by testing for
equal, when you are looking for a specific condition other than the
value of a loop variable.  For example, when looping through a table
looking for some value (e.g. exiting when Table-Entry(sub) = <value>
is perfectly fine). :-)
```
