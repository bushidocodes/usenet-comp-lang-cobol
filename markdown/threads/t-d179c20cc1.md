```
A MicroFocus Cobol program runs fine on the PC but gets SQLCODE=0100
on all subsequent queries.

I have a program that performs a main query to database A. Here is
some Pseudo Code that represents the overall flow.

Do While more rows in main query

   Deletes row from table 1,2,3,4 in database B (Key from main query)
   COMMIT
   Insert row in table 1 in database B
   COMMIT
   Opens query 2 in database A SQLCODE = 0
   Fetches from query 2 (always gets SQLCODE = 100)

   Opens query 3 in database A SQLCODE = 0
   Fetches from query 3 (always gets SQLCODE = 100)

   Opens query 4 in database A SQLCODE = 0
   Fetches from query 4 (always gets SQLCODE = 100)

   Fetch row from main query
end 

On the pc the fetches for Queries 1,2, and 3 have rows, SQLCODE=0.
I have tried different MODE=, hold_cursor=yes, WITH HOLD on cursors
from DBase A.

No cursor is from a prepared statment. 
MicroFocus Server Express 2.2 on UNIX and Net Express 3.1 on the PC.
Oracle 9i client setups to 9i database (A) and 8i database (B).


Any help would be appreciated. Thanks.
```

```
Paul,

Try running it without the commits (until end of run).

In some systems COMMIT resets the cursor, even if you specify WITH HOLD at
the time you declare it.

If it works without the commits, it will confirm that that is the problem.

You might also consider positioned update/delete if you are running from a
cursor.

Finally (a bit obvious, but sometimes overlooked...) ensure that the data is
exactly the same on both systems...

Pete.
```

```
Paul,

I'm assuming that you're running the same version of the Oracle client on
both Win32 and UNIX (which platform is it, by the way?) ? I'm assuming that
you're using Pro*COBOL here, and that you've precompiled locally on the

Following on from Peter's suggestion, have you tried the CLOSE_ON_COMMIT

Simon.
WITH HOLD Clause in DECLARE CURSOR Statements

Any cursor that has been declared with the clause WITH HOLD after the word
CURSOR remains open after a COMMIT or a ROLLBACK. The following example
shows how to use this clause:

EXEC SQL

DECLARE C1 CURSOR WITH HOLD

FOR SELECT ENAME FROM EMP

WHERE EMPNO BETWEEN 7600 AND 7700

END-EXEC.

The cursor must not be declared for UPDATE. The WITH HOLD clause is used in
DB2 to override the default, which is to close all cursors on commit.
Pro*COBOL provides this clause in order to ease migrations of applications
from DB2 to Oracle. When MODE=ANSI, Oracle uses the DB2 default, but all
host variables must be declared in a Declare Section. To avoid having a
Declare Section, use the precompiler option CLOSE_ON_COMMIT described next.
See "DECLARE CURSOR (Embedded SQL Directive)".

CLOSE_ON_COMMIT Precompiler Option

The precompiler option CLOSE_ON_COMMIT is available to override the default
behavior of MODE=ANSI (if you specify MODE=ANSI on the command line, any
cursors not declared with the WITH HOLD clause are closed on commit):

CLOSE_ON_COMMIT = {YES | NO}

The default is NO. This option must be entered only on the command line or
in a configuration file.

Note: Use this option carefully; applications may be slowed if cursors are
opened and closed many times because of the need to re-parse for each OPEN
statement. See "CLOSE_ON_COMMIT".
```
