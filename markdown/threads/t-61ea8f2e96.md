Hi Carlos
Working with cobol (chiefly RM/Liant tools-Unix+Dos+Windows environment) for
the last ten years and, although not being flooded by such an amount of
clients as yourself, I myself have been dealing with the same problems,
which amount to two:
1.GUI, for several years now I've given up all the accept/display stuff
thanks to a pannelling procedural tool/extension which allows you to design
controls (pulldown menus, radio&push-buttons, list boxes and the like, which
work alike in a character or graphical interface) and retrieve their events
for appropriate prodedures - but this is by no means a true Windows
solution (however it is a true event driven solution) , yet I like it very
much. Lately, cobol compiler producers have been releasing VB-like
development tools, which one tends to look at (at least I do) with some
suspicion. Other suppliers (like the previous posters) are also offering
tools for the development of true GUI, which one can 'paint' cobol apps
with, but if you choose to steer towards VB (at least for some purposes, as
I'm doing), then the second question arises...
2.ODBC, yes one can easily have cobol apps running and maintaining their
files and at the same time have VB client programs (or Access or Whatever)
accessing (and, why not ?, maintaining those files). In a mixed environment,
with different kind of needs on the various levels of users/managers having
hands on the sistem, I stand by such a mixture approach and am, right now in
the process of installing ODBC acess
to cobol files (they happen to be RM files but could as well be Microfocus'
or any other dialects'): the tool that allows this is Relativity. I already
tested it and so far it works wonderfully. Further development is on the
way, but as for now I'm happy with it.

As a conclusion on the topic and - maybe - lighting up some fire, I would
say that programmers usualy blame languages for their own inefficiencies and
the quest for a graphically assisted language is sometimes driven by the
need of hiding those weaknesses.

PSCruz
