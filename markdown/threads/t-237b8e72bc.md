<<Can the COBOL internal SORT facility be used to SORT non-file storage?

"Table sort" is a feature of 2002 COBOL.  See ISO/IEC 1989:2002, page 524,
14.8.36 SORT Statement Format 2 (table).

Syntax example and summary is

    SORT data-name-1 ON <ascending/descending>... KEY data-name-2 ...

where data-name has to have an OCCURS clause, and data-name-2 must either be
the same as or subordinate to data-name-1.

I don't know which, if any, implementation provides this feature; I would be
surprised, however, were there none.

    -Chuck Stevens

I would not bother with the SORT statement.

Simply define another table of the same size, and copy the entries one by
one, putting them in the correct sequence.  This takes about 10 lines of
code, and if the entries happened to be in sequence, it is roughly equal
to a sequence check in time expended.

Basically, you code the following:
Working-Storage:
01  SOURCE-CODE-LIMIT PIC S9(4) BINARY VALUE +4096.
01  SOURCE-CODE-SIZE  PIC S9(4) BINARY VALUE ZERO.
01  SOURCE-CODE-TABLE.
 05 SOURCE-CODE-ENTRY
               OCCURS 1 TO 4096
            DEPENDING SOURCE-CODE-SIZE
            ASCENDING SOURCE-CODE
              INDEXED SC-X.
 10 SOURCE-CODE       PIC X(06).

Procedure Division:
Setup code:
To  guard against mistakes when table sizes get changed,
code the following statement in the 'housekeeping'  area
for each table that has a 'table-limit' item:
        COMPUTE 'table-limit' =  LENGTH OF 'table-name' /
                                 LENGTH OF 'table-entry'(1)
(In  the  sample  above,  'table-limit'  corresponds  to
SOURCE-CODE-LIMIT,    'table-name'    corresponds     to
SOURCE-CODE-TABLE,   and  'table-entry'  corresponds  to
SOURCE-CODE-ENTRY.)
The COMPUTE verb will set the 'table-limit' item to  the
correct  value,  even if the programmer forgot to change
it!  However, every effort should be made  to  keep  the
OCCURS  clause  and 'table-limit' in synch, because this
will aid maintainability of the program.

The sorting code:
Adding an item in sequence to a table:
The method is to move the last item in the table  to
the next occurrence, repeating this move until there
is  room  for  the  new entry at the proper point to
maintain the table's sequence.

          ADD 1            TO 'object'
          IF  'object'     >  'table-limit'
              PERFORM TABLE-OVERFLOW
          END-IF
          SET 'index'      TO 'object'
          SET 'index' DOWN BY 1
          IF  'object'     >  1
              PERFORM MAKE-ROOM-IN-TABLE
                  VARYING 'index' FROM 'index' BY -1
                  UNTIL   'index'   <  1
                     OR   'new-entry-key' >  'entry-key(index)'
          END-IF
          MOVE 'new-entry' TO 'entry(index + 1)'
          .
The MAKE-ROOM-IN-TABLE procedure reads:
      MAKE-ROOM-IN-TABLE                             SECTION.
          MOVE 'entry(index)' TO 'entry(index + 1)'
          .

Also possibly of interest:
Deleting an item from a table:
The method is to move the next  item  in  the  table
over the item being deleted, and repeating this move
until  the  end of the table is reached.  The result
is a table one item shorter than before, even to the
point of becoming 'empty' (zero occurrences).

The item to be deleted is pointed to by 'index', and
the size of the table is contained in 'object'.  The
value of 'object' must be  greater  than  zero,  and
'index'  must  be in the range of 1 to 'object', inclusive.

          IF 'object'  >  1
              PERFORM TAKE-ITEM-OUT-OF-TABLE
                  VARYING 'index' FROM 'index' BY 1
                  UNTIL   'index'   =  'object'
          END-IF
          SUBTRACT 1 FROM 'object'
          .
The TAKE-ITEM-OUT-OF-TABLE procedure reads:
      TAKE-ITEM-OUT-OF-TABLE                         SECTION.
          MOVE 'entry(index + 1)' TO 'entry(index)'
          .

Happy sorting!

JerryMouse,
I thought you were being humorous in another post where you mentioned being
against using the COBOL standard index with arrays.

Maybe I could recode this example.  But why?  Your compiler and mine both
support everything in it.  The objective is to get the work done, and this will
do it.

Of course, I have personal preferences, too.  They run more to forbidding use
of ALTER and GO, using only one period per procedure, etc.

I think your argument was that indexes do not provide a significant performance
improvement, so why use them?  I could argue that likewise, subscripts do not
provide a significant performance improvement (in fact, I do not know of any
case where they offer any), they must be "correctly" defined as to usage and
the like, and they must be separately defined, and therefore could end up
getting used for multiple tables (which could be dangerous), so why use them?

There are some COBOL language elements (SEARCH / SEARCH ALL comes immediately
to mind) that require indexes.  Unless your programming environment has an
equally convenient method for table searching, I assume you grit your teeth and
use indexes some of the time.

The example I posted is torn from a larger document that discusses good table
handling techniques.  There, I extol the virtues of OCCURS ... DEPENDING
(because one never needs to initialize a table), limit checking when loading a
table and what to do when table overflow occurs (read the rest of the table
input, so a programmer knows how much to increase the table size), use of
SEARCH / SEARCH ALL, etc.  In the context of that document, the use of indexes
is integral.

Hmm.  I would caution that ODO entails more range checking for reliability.
Not only must you (or the compiler) ensure as for "normal" tables that the
index/subscript is in range, you (or the compiler) must ensure that the ODO
dynamic limit hasn't exceeded the physical array bounds.  Actually, we do it
the other way 'round; we check the ODO limit against the table upper limit,
and then if that passes we check the subscript against the ODO limit.  ODO
may not be suitable for performance-critical logic because of the overhead
of the bounds checking (user-supplied or compiler-supplied).

I don't see any difference in the checking needed.  You can set the index to 1
(or move 1 to the subscript).  Then, you load a table from some external source,
and check as you load that the index or subscript doesn't exceed the table
limit.  The index (or subscript) gets incremented by 1.  When you have finished
loading, the index (or subscript) is one higher than the number of entries in
the table.  You set the index down by 1 (or subtract 1 from the index), and use
that to put a value in the object of the ODO.  From then on, you are set to use
the table, until such time as you may want to add more entries or delete an
entry.

There is a bit of run time checking of the ODO object, but I think in an IBM
COBOL implementation, this is done when the object is changed (so that isn't
often at all).  The object's value probably gets passed to run time subroutines
for SEARCH and the like.

I just know it works a heck of a lot better than filling a table with HIGH-VALUE
and then loading over 10% of that table, then doing a SEARCH ALL where the first
two or three looks are at "fake" entries.
