I'm not surprised: the number of combinations of n things taken r at a time is 
given by C(n, r) = n! / ((n - r)! * r!). C(45, 6) = 8,145,060. That's a lot of 
combinations to evaluate. And due to some bugs in the program, you're 
generating a few more than that, while failing to generate others that should 
be.

The innermost loop of your nested PERFORMs is being executed some nine million 
times -- and that innermost loop is performing a SORT, *and* reading from a 
file. Soooooo.... you've written a program that's performing nine million 
SORTs, eight million WRITEs, and nine million READs; is it any wonder that it 
appears to stop responding? I'm sure it's still running, but it's gonna take 
some time to finish. 

How much time? Well, let's try some rough estimates. Due to the nature of the 
program, most of the reads will be for records that are not present in the 
file, so you won't get much benefit from disk caching. Figure the 17 million 
logical I/Os to result in half as many (8.5 million) physical I/Os. And 
they'll all be *random* I/Os, too, very few sequential track-to-track disk 
seeks here. Assuming a 10ms seek time and 6ms latency, you're looking at about 
38 hours to do all that physical I/O.

And then there are the nine million SORTs, too. If each one takes only 10 
milliseconds, that adds another 90,000 seconds, or 25 hours.

Add in the PERFORMs, and the logical I/Os, and just off the top of my head, 
I'd guess you're looking at a minimum of three solid days for this program to 
complete.

There is a *much* simpler algorithm for doing this (it should be much faster 
too) without the SORTs or the READs. As I suspect this is a homework problem, 
though, I won't describe the algorithm, but rather give you some hints toward 
discovering it.

1. My assertion that an algorithm exists, in which the sort is unnecessary, is 
itself a huge hint toward discovering it. Think about how you could rewrite 
your PERFORM statements so that the numbers are guaranteed to be always in 
ascending sequence.

2. My assertion that an algorithm exists, in which reading an indexed file is 
unnecessary, likewise is itself a huge hint toward discovering it. The logical 
consequence of this assertion is that an algorithm exists which will generate 
each combination exactly once, removing the need to check each generated 
combination to see if has been generated before. Your program unnecessarily 
evaluates both (45, 44, 39, 42, 41, 35) and (45, 44, 35, 42, 41, 39), for 
example. An algorithm which generates only one of these permutations is 
superior to one which generates both, and then eliminates the duplicate.

3. I referred to some bugs your program contains. An understanding of these 
bug may help guide you to the discovery of a better algorithm: Although the 
purpose of the program is clearly to generate lottery numbers, it produces 
some permutations which could not possibly be lottery numbers, for example 
(39, 39, 39, 39, 39, 39) or (45, 44, 39, 40, 39, 40). The structure of your 
PERFORM statements shows that you have attempted to avoid this situation, but 
a closer look will show clearly that this code misses the mark. Specifically, 
the six loops will generate these sets of numbers:
45, 39, 33, 27, 21, 15, 9, 3
44, 39, 34, 29, 24, 19, 14, 9, 5
43, 39, 35, 31, 27, 23, 19, 15, 11, 7, 3
42, 39, 36, 33, 30, 27, 24, 21, 18, 15, 12, 9, 6, 3
41, 39, 37, 35, 33, 31, 29, 27, 25, 23, 21, 19, 17, 15, 13, 11, 9, 7, 5, 3, 1
40, 39, 38, 37, 36, 35, 34, 33, 32, ...
There is considerable overlap between these sets. Look for a better method of 
avoiding overlap.

4. Another bug resulting from the incorrect structuring of the PERFORM 
statements is the program's failure to generate certain correct combinations. 
For example, since 40 and 38 each appear *only* in the sixth series, the 
program cannot generate any combination containing both 40 and 38. Likewise, 
since 42, 40, and 36 are contained only in two sequences, the fourth and 
sixth, no combination containing all three will ever be generated. Look for an 
algorithm that guarantees that every possible combination will be generated.

If the range of possible numbers is 1..45, then there are only 40 series of 
the form (k, k+1, k+2, ...): (1,2,3...) thru (40,41,42,43,44,45). Similarly, 
there are only 17 series of the form (2k, 2k+2, 2k+4, ...): (2,4,6...) thru 
(34,36,38,40,42,44).

How many other series there might be following some particular patterns would 
of course depend on what patterns you choose. If you're only interested in 
arithmetic progressions, it's not difficult to calculate how many there are. 
Given that all possible progressions will be of the form (n, n+k, n+2k, n+3k, 
n+4k, n+5k) where k and n >= 1 and n+5k <= 45, all you need do is figure out 
how many different values of k there can be for each value of n:

If 1 <= n <= 5, then 1 <= k <= 8
If 6 <= n <= 10, then 1<= k <= 7
If 11 <= n <= 15, then 1 <= k <= 6
..
If 31 <= n <=35, then 1 <= k <= 2
If 36 <= n <= 40, then k = 1

This means (8 + 7 + 6 + ... + 1) * 5 = 180 possible arithmetic progressions.

In the small range of 1..45, the only possible geometric progression is the 
powers of 2 (1,2,4,8,16,32).

>
>and how would I best be able to remove these sets?

You could check to see if (u - v) = (v - w) = (w - x) = (x - y) = (y - z), but 
it's probably simpler to remove them after the fact.

If you really want to make millions of sets of numbers to be
printed/written to a file, try setting up collumns of numbers and see
how many accross you want to go.  If you are writing them to a file one
write per number will really slow things down.  If you computed say
5,000-30,000 numbers at a time in a table and wrote them in blocks of
the largest number of bytes that is possible That might really speed
things up.  For instance if you are writing really short record and the
size was say 5 2-digit,lottery numbers side by side that is Ten
Characters, but if you make that a packed number it could even be
smaller.  As a packed number, pic "9(11) comp 3" only takes up 6
character positions! If you are using a 512 character block you can
write about 80 records or so in a block.  This would really speed up
the writing process.  I don't know if you can do that on a PC or not,
but I am guessing that you can.

What type of MODE you write in will make the file smaller. If all of
the numbers are end to end with no control characters for end of record
then you can save space.

It takes one extra half bite to get an odd number of packed digits in a
packed number because one half byte is used for the sign. If you
redefine the packed number as a 6 character alphanumeric field it will
sort even faster.  That is a little harder and a pain to code but it
would speed up a sort.

Another thing is what if some of the winning combinations are repeats?
If these are lottery balls then each number can only appear once.  For
instance you can not have "01 01 01 01 01".
If you had an array of numbers to pick from with no duplicates and you
chose one number the first time by using a random number for the
subscript, then you would want to move a valid number from the last
array element into that position (Unless it is the last), and then use
a random number of one less to choose the second number.  Therefore the
possible number combinations are reduced by one for each succeeding
ball picked.
Once the array size is less than the total number of balls picked you
are done, which will also limit the number of possiblities.

Since the lottery people always display the balls in numeric order,
once you choose all the balls combinations with a one in it you can
remove the number one from the group of 60 or 50 or how ever many. This
will also reduce the number of combinations quite a bit.
