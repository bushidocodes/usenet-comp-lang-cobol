Dear people,

Greetings to everyone from Buenos Aires,

After of a lot of years of developing I'm here as a COBOL QA Analyst.

The Architecture and Infrastructure Managers are looking for programs that scan COBOL/400 source programs in search of metrics, definition errors or inconsistencies like as:

Possible improvements in performances, (Eg use COMP-3 for arithmetic operations)
duplicate code,
coding standards (Eg use meaningful names for fields, align the IF / ELSE / END-ID / MOVE)
logical coverage, (eg with EVALUATE WHEN OTHER ...)
degree of complexity (Cyclomatic Complexity = Number of loops, selections and bifurcations)
truncation errors or exception data (Eg move from one variable to another shorter or an alphanumeric to numeric)
best practices,
comments. (Required to know how the program works quickly)

We are evaluating SonarQube and Kiuwuan, Have any of you used or use these tools?

Many Thanks,
Marcelo Shulman.

Mr Shulman, this looks like a fascinating area of study. I do not have
any experience with those tools or any others to accomplish the tasks you
mention.

Some things - dead code, truncation - are detected by certain compilers;
it is most likely that a site which is concerned with optimising COBOL/400
will not implement one of those.

Improvements in performance should be addressed application by
application.

Relying on comments to learn how the program works can result in
disappointment.

A good first question to ask might be 'what are the measurements being
used which give the conclusion that the current system has unsatisfactory
performance?'

(If someone is expected to listen to the answer to this question it might
be a very good idea to decdide on a rate, or range of rates, for this sort
of expert professional advice.)

Hello, Marcelo...

I worked in the same position you describe for a fortune 100 company for quite a while. Of course, maintainability of code is almost always more important than performance. Nonetheless, as others have posted, often the compiler can help quite a bit. Also, relative to performance, I/O will almost always be the biggest performance issue. As I determined various best practices, I used awk to do a lot of source parsing in order to identify deviations. Be sure to include the developers in this whole effort. It is good to remember that ultimately programming is human activity. Make sure that successes and failures improve the whole process; but if you can get the developers to feel like you are listening to them, accrediting them, etc. they will participate with much more alacrity. If you find some useful looking tools, get some demos of them on some of your worst and best code.

There are a lot of interesting things that you can do. Keep studying, talking, learning, and being involved.

Sorry, I didn't answer your question directly. I thought a few aphorisms might be generally helpful.

Respectfully, Doc
