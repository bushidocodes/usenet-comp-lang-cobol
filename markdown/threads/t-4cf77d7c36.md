Here's a working example with some in-house stuff just to give you the
flavor. Personally I use external sorts unless there is no alternative.
But this is good classroom experience - you'll run into them in COBOL
land.
There are other ways to set this up. A common mistake is not to have
something stop the program after the 'SORT SORT-FILE' statement. What
happens is that it finishes all its sorting, etc., then drops into the
B100 paragraph again, eventually falling out of the bottom of the
program with an addressing exception (SOC4 in IBM) or a data exception
(SOC7 in IBM).

Just to help clarify, I put some comments with //* in parens that are
not in the program I stole this from.

Also note that you do not OPEN/CLOSE SORT-FILE like you would for a data
file.

Good luck
John


INPUT-OUTPUT SECTION.
FILE-CONTROL.

SELECT SORT-FILE ASSIGN TO SORTFILE.

DATA DIVISION.
FILE SECTION.

SD SORT-FILE
DATA RECORD IS SORT-RCD.

01 SORT-RCD.
03 SORT-MATERIAL-PROPERTY-FLAG PIC X.
03 SORT-SITE-CODE PIC X(06).
03 SORT-CONTRACT-NUM PIC X(09).
03 SORT-IDENT-ITEM-NUM.
05 SORT-PROPERTY-NUM PIC X(07).
05 SORT-MATERIAL-IDENT.
10 SORT-OFF-SITE-SHIDOC PIC X(6).
10 FILLER PIC X.
10 SORT-PAF-ITEM-NUM PIC X(03).
03 SORT-DOLLAR-VALUE PIC 9(11)V99.

PROCEDURE DIVISION.


SORT SORT-FILE
ASCENDING KEY SORT-SITE-CODE
SORT-MATERIAL-PROPERTY-FLAG
INPUT PROCEDURE B100-INPUT-CONTROL
OUTPUT PROCEDURE C100-OUTPUT-CONTROL.

STOP RUN.

B100-INPUT-CONTROL.

PERFORM X110-OPEN-MATLCSR (//* Opens data source - DB2
table)
PERFORM X120-FETCH-MATLCSR (//* Gets data from DB2
table)
PERFORM UNTIL SQLCODE = 100 (//* End of table
records)
PERFORM B110-SETUP-MATERIAL-RCD (//* fills in SORT-RCD)
RELEASE SORT-RCD (//* sends record to
SORT-FILE)
PERFORM X120-FETCH-MATLCSR (//* gets more input
data)
END-PERFORM
PERFORM X130-CLOSE-MATLCSR. (//* closes data source - all
records now on SORT-FILE)
(//* the SORT SORT-FILE statement above will now sort
them)

C100-OUTPUT-CONTROL.
RETURN SORT-FILE
AT END MOVE "YES" TO PROPERTY-EOF.

(//* records are now sorted - do more stuff)

Oh, let's see if I can remember this... I'll start with the Procedure
Division statements, if you need more just holler.

sort sortfil ascending key sort-rec-key
input procedure inpt-proc-para
output procedure otpt-proc-para.

inpt-proc-para.
read input-files at end (do whatever).
(edit rtns.)
release sortrec.
(note: been so long that my initial impulse is to control inpt-proc-para
flow with GO TOs... that will not earn you a decent grade)

otpt-proc-para.
return sortrec into ws-rec at end (another imperative).
(process returns).

I believe that when the first RETURN is encountered is when the actual
SORT occurs; successive RETURNS act as READs on the sortfil.
