Well, it has ben requested that I pollute comp.lang.cobol with some ASM...
so here goes.

The question was the efficiency of the following two alternatives:

READ INOUT-RECORD
AT END SET WS-END-OF-FILE TO TRUE
END-READ.
IF NOT WS-END-OF-FILE
ADD 1 TO WS-TOTAL-RECORDS-READ-ACCUM
END-IF.

... and ...

READ INPUT-RECORD
AT END
SET WS-END-OF-FILE TO TRUE
NOT AT END
ADD 1 TO WS-TOTAL-RECORDS-READ-ACCUM
END-READ.

First a word to the Olde Tymers who posted such code... shame, shame,
*shame* on you! There are Newbies here and you are teaching them Bad
Habits, now repeat after me:

Files are opened, closed and read.
Records are written.

So, in the abovegiven example, do a mental global replace on INPUT-RECORD
with INFILE.

(note: WS-END-OF-FILE was defined as an 88 on a PIC X so as to keep the
SET and the IF to MVI/CLI... and I, personally, would have called such a
flag NO-MORE-INPUT... but that's a matter of taste)

So, then, what's the difference? Best I can tell... not very much. Both
are equally readable; the second method saves a single CLI per record.
This sort of savings can be important if transaction volume is
sufficiently large to merit it (say, tax-record processing for large
populations over long periods of time... I recall hearing about a monthly
process run by the American Internal Revenue Service which takes three
weeks to run) but for the overwhelming majority of applications I would
say it makes little or no difference...

... and now, of course, since I have stated that, to me, in most
processes, an additional CLI per record read makes no difference I am sure
to generate Great Discussion and Disagreement... but hey, what's the
UseNet for?
