One way factories are used is in handling input streams (parsing). Suppose
you're reading a set of name/value pairs from an input stream, a pretty
common task if you're coding a CGI, or EDI application. You can use a
factory to create objects based upon the contents of the stream, kind of
like UNSTRING on steroids. Having created the object, all the methods are
now available for that object; such as isValid(), toStorageFormat() or
anything else you dream up.

In effect yes. You are telling the system to build up an array. The
structured equivalent of :-

01 myTable.
   05 myRecord occurs
	"how many do you want ?" 	pic x(???).

I'm a little vague if you don't set an initial size, (assuming you
aren't adding records right now), but the compiler must set an initial
value). And you can control the size of collections "growto" to a
smaller or larger number. There are a whole daffy of collection types,
bags, orderedCollections, SortedCollections and Dictionaries - and you
have the ability to turn one into another, or extract single or 'blocks'
of records into another collection. And they can contain intrinsics,
objects and bitmaps. The essence of how they work is that the collection
has an object reference (say, I dunno I'm not into hex - OA OB 00 00)
when you look at it in Animator. Then each element in the collection
also has its own object reference. Obvious caution here that your
collection doesn't get too large, otherwise you are going to blow
memory. Some of the collections have an index, others don't.

Example of dictionary - from RM data files I read in old 'Systems
records" having built a dictionary of what I've already got in N/E I can
then test the dictionary to compare descriptions - then allocate records
in the N/E system as appropriate.

No maximum guidelines - but I have created a collection with 950 entries
so - 951 object references + 950 x 40 characters. It worked fine, but I
got nervous about this, goes back to only having 64K with the 'Shack
Model II - I've done away with that collection by generating the
descriptions from a set of mnemonics. My prime use is for
Listboxes/DropLists.

You should hear young Simon Hart in Norfolk enthuse. He uses them for
all sorts of things; I know Michael M. likes using relative files as a
temporary - Simon builds up a collection in memory. And of course the
buzz-word I've read in CLC is 'populate' the collection, but I suppose

Thane Hubbell wrote:

> 1.  Can you explain the SELF and SUPER object references when used
> instead of object Handles?

When you first define your application you define the interface not
how it does it.  If you want to store and retrieve data do that, today
you might buffer a sequential file into memory, tomorrow Oracle
database.

When you inherit from a parent you get a set of methods for that
parent automatically.  You can make the choice to use all these
methods except one.  When you use and instance of a child object you
call the method on the child, it may well be serviced by the parent
method.

Now if I define my basic trade and set up a method that does something
(can't think of an example).  You want the child to perform additional
steps in that method but also the original parent code.  You define
SUPER method in the child objects method and do the parent one as
well.

Note:  there is a big distinction between inheritance and using an
object for reuse.
