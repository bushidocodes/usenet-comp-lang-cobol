ROUTINE that doesn't take century into account will break down.  Not just
mainframe COBOL programs.  I'm talking spreadsheets, DOS batch files, Unix

COBOL programs will probably be affected more than those of langauges like
C because C (for example) has become widely used much more recently than
COBOL which goes back to the early 60's. I started programming in COBOL in
the mid-late 70's and nobody on our staff even considered that our
programs might still be running 35+ years hence. To a bunch of 20-somethings,
that was a lifetime and a half away. This was the dawn of the mini-computer 
era, and new systems were leaping off the keyboards all over the place.
We figured if our programs lived 3 years we had created a legend.

I started heavy C coding in the mid-eighties, was almost 10 years older,
and much more aware of the passage of time - and how soon the turn-of-the- 
century was coming up.  I wrote a lot of YYMMDD financial COBOL code in 
the late 70's and early 80's which *will* blow up if it is still running 
by 2000!

C's date/time is measured in seconds since January 1, 1970 and is an 
ANSI C standard. In the 70's, each vendor started their time-zero from
some different date. Even the IBM-PC (ca 1981) starts at 1/1/80. I
like using C's timestamp since it is the only one (that I know of) which
is defined by the language standard and works across platforms. It
should be good as-is till about 2100 A.D. Even at that, the C language
standard defines a changable type definition for the time (currently a
32 bit integer) which *could* be changed someday.

This is not an advertisement for C. Just an example of how more recently
developed languages have dealt with the problem.

-- A contributor to doomsday
