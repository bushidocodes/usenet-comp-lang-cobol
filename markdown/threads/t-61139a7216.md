1. The checklist
----------------
1. Code compiles before walk through
2. Code has passed automated quality checks (none implemented yet)
3. Code complies with coding standards
4. Program does what it is supposed to do
5. Variables and procedure names comply with naming standards
6. Names used are meaningful
7. Names used do what they imply.
8. Comments are meaningful, do not restate what the code does.
9. Program comments - does the program definition fit the program.
10. Are variable sizes appropriate, do the calculations have any
errors?
11. Code is maintainable.

1.1	Inputs required
-----------------------
ï¿½ Time - an effective review can not be achieved without proper
preparation.  It is not to be achieved with a quick skim of the
code.
ï¿½  A listing for each participant.  They should be given the
listing 24 hours before the actual meeting to allow time to go
through the program before hand.
ï¿½ Requirement documents available for review by all participants.
ï¿½ Any output generated from a preceding walk though.

1.2	Outcomes required
-------------------------
One of the following: Acceptance without modification, Acceptance
with minor modifications, walk through again.
If the program is accepted with minor modifications then a second
review will be done.  This review is by a person other than the
change author after the changes have been completed.
If the program is walked through again then all other outputs go
into the next walk through.
The review meeting should produce a list of the changes required. 
This can be in the form of an annoted program listing and a sheet
outlining any further details required.  These comments may in
fact highlight limitations with the design of the application
rather than the programmer's implementation.

2.	What is a walk through
------------------------------
A walk through is a process of eliminating problems before they
become major problems or even take time debugging.

3.	Definitions
--------------------
3.1	Code compiles before walk through
----
The program must be complete before a walk through.  If the code
is incomplete you are giving advice and assistance not providing a
detailed working through of the code. While there is a probability
that you will go over some design issues while you are looking at
the code it is unlikely that you will step through the program
from beginning to end.

3.1.1	The code should compile without warnings.
-----
If the programmer leaves warnings in the code then there are
potential errors missed.  Some warnings produced by the compiler
can be safely ignored (eg move corresponding with filler items)
however it is often easy to work around these warnings giving a
clean compile.

3.2	Code has passed automated quality checks (none implemented
yet)
----
At a later date there will be some basic checking commands
developed to ensure simple questions are handled without human
interaction.  This will be advised as it becomes available.

3.3	Code complies with coding standards
----
Please refer to **your standards** for details:

3.4	Program does what it is supposed to do
----
This requires an understanding of the complete user requirements
and also the complete system that the program integrates.
It is very common for a one-line program modification to be
accepted and create serious problems with the system in which it
resides due to lack of consideration for the total picture. 
Impacts on other programs must be considered when modifying a
system.
The program must be reviewed with the user requirements in mind. 
It is possible that the author of the change has a different
concept than the designer.  If the code is superficially looked at
then this point is commonly missed.  It is also possible that the
author has the correct interpretation and the design document is
incorrect.  It is possible also that both the design and code
changes are incorrect.
This is difficult to pick up but the rewards of finding these
errors far outweigh trivial formatting errors in code.

3.5	Variables and procedure names comply with naming standards
----
Please refer to ** your standrds** naming conventions.  This
document outlines some abbreviations that are in use.

These are to be adhered to for; DB2 names, working storage
variables and file variables.

Some examples of what to check for:
- NO is not allowed.  It should be NBR (number) or NOT.
- NMBR is not allowed, correct abbreviation is NBR
- NUMBER is not allowed, it should be abbreviated to the standard
NBR

3.6	Names used are meaningful
----
Variables such as WS-SUB, WS-X and X are not to be used.
Paragraph names such as B100-READ-NEXT are not to be used, use
B100-READ-NEXT-CUSTOMER.

3.7	Names used do what they imply
----
Paragraphs must do what they imply.
ï¿½ READ-EMPLOYEE-FILE does not read the warehouse file, nor does it
calculate pay rates.  It should only read records.
ï¿½ Paragraphs with a name ending in -EXIT must only contain one
statement 'EXIT'.
Variables are used to store the values that the name describes.
ï¿½ WS-CUSTOMER-NBR is not used to store a temporary value in a
calculation.

3.8	Comments are meaningful, do not restate what the code does
---
This is probably the most difficult to judge.  It is often very
difficult to divorce the actual code from the comments provided.

* THIS CURSOR IS USED TO EXTRACT ALL THE ROWS FROM TM002H00     *
* ACCORDING TO THE SELECTION CRITERIA                           *

              EXEC SQL
                DECLARE MAIN_CSR1 CURSOR WITH HOLD FOR
                      SELECT REQUEST_NBR
                            ,CUST_NBR
                            ,ACTION_CD
                            ,ACTUAL_COMPLTD_DT
                            ,ACTN_RSN_CD
                            ,LM101_GRP_CUST_NBR
                        FROM TM002H00
                            ,LM101T00
                       WHERE ACTUAL_COMPLTD_DT = '01/01/0001'
                         AND ACTION_CD IN ('B', 'D', 'E', 'H',
'K', 'L')
                         AND PROC_STAT_CD = 'I'
                         AND CUST_NBR          = LM101_CUST_NBR
                       ORDER BY 1, 2
                WITH UR
              END-EXEC.

The above comment might have been better stated as 'select action
codes that require a down load file for processing'.

Included in this point is that if a piece of code is discussed for
a reasonable period of time without modification then that piece
of code needs a descriptive comment.

3.9	Program comments - does the program definition fit the program
----
Program comments are the comments at the top of the program. 
These should succinctly describe the function of the program. 
Below this is a log of the changes applied to this program in date
order (most recent change first).

Often programs are modified and small changes made to the internal
logic but the comments at the top are not altered to reflect the
current function.  This can also occur when one program is used
for a skeleton to another program.

For example a simple change may remove a restriction on a SQL
select and the comments in the immediate vicinity are changed (say
the paragraph comment) but the comment at the top of the program
still says that it is restricted.  When analysis on another change
comes along the program may be skipped for analysis because of
this comment.

3.10	Are variable sizes appropriate, do the calculations have any
errors
-----
It is possible that the variables storing numbers will be to small
to hold the potential values.

There are sometimes problems with the number of places in a
calculation.  For example 'PIC9V99 = PIC99 / 100' may give
different (and incorrect results) than 'PIC9V99 = PIC99 /
100.00'.  This will vary as the compiler changes but those errors
must be known and validated.

3.11	Code is maintainable
-----
This is a personal issue and must be done with care.  The
structure should follow the classic three-step process in the main
body:
	Perform a000-initialise
	Perform b000-process-records
		until end-of-records
	Perform c000-terminate
The code should not make it difficult to determine the underlying
business logic.

Extreme care must be taken with this item.  If the program is not
'what you would have done' consider whether the program meets the
objectives of the business and is readable.  DO NOT use 'it works'
as a judgement on this issue, while this is important the code
must be understandable, over the life of a program twice the cost
of writing it will be spent on maintenance, badly written programs
always take more.

Is the code extensible without modification?  
ï¿½ Is the selection criterion really static or can it be
parameterised for greater flexibility in the future.
