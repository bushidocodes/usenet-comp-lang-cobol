```
All righty... here's the situation.  There's an input file, flat,
containing a bunch of records from different locations, indicated by a
four-character numeric.  I've been asked to write a program to split this
file into two files, one containing a set of specified locations, the
other containing the rest.

(As per Standard Practise Here... the specified location numbers are keyed 
into a PDS member which is loaded into a working-storage table in the 
program; each input record's location number is then matched against the 
table for appropriate processing.)

Anyhow... the order of the records in the output files is the reverse of 
that of the input file.  I asked if this would have any effect on future 
processing and got a resouding 'That's A Goooooood Question... why don't 
you just put them back into original order.'

Fortunately both output file formats have a large-enough chunk of filler 
that I can put in a sequencing-number during processing; since current 
processing has shown an average file to be about 20,000 recs I have put in 
an 8-digit display numeric counter in the last 8 bytes of the outrecs... 
read an inrec, look up the location in the table, format it according to 
SEARCH result, add 1 to the sequence, move the sequence to the end of the 
record, write the appropriate output, etc.

Next, of course, comes a SORT on those last 8 characters DESCENDING... 
hey, presto, everything is in order.

Next comes a FileAID step to turn the sequence numbers back into spaces... 
gotta be neat about it, don'cha know.

That was all set-up... here's the question.

The FileAID control-card will allow for two ways to do the same thing.  I 
can code it as:

$$DD01 UPDATE  REPL=(493,C'        ')

... which specifies REPLacing whatever is at position 493 with the 
following Character string, in this case 8 spaces... or I can code it as:

$$DD01 UPDATE  REPL=(493,X'4040404040404040')

... and the difference (for those who speak EBCDIC) is obvious.

Being of the Olde Schoole, taught to mark a space with a slashed 'b', I am 
tending towards the latter example; leaving a bunch of... nothing in a 
control-card, to me, screams out 'HERE!  HERE!  SCREW ME UP *HERE*!'

On the other hand, of course, is... why specify hex when you can use 
character?  Trying to be fancy and 'dirty', perhaps... and in JCL, too!... 
well, what can you expect from a COBOL-jockey, anyhow...

Anyhow, that's what I have.  Comments are invited, fire away... and 
thanks much for your time.

DD
```

```
If you're doing a sort on the sequence numbers anyway, why not blank 
them out in the same pass using the OUTREC statement instead of doing a 
separate pass with FileAid?  With DFSORT, you have lots of  choices for 
how you code the blanks, such as:

  OUTREC FIELDS=(1,492,8X)
  OUTREC FIELDS=(1,492,C'        ')
  OUTREC FIELDS=(1,492,8X'40')
  OUTREC FIELDS=(1,492,8C' ')
  OUTREC FIELDS=(1,492,X'4040404040404040')
  OUTREC FIELDS=(1,492,500:X)

Note that you can put the column number before the blanks if you like as 
well, e.g.

   OUTREC FIELDS=(1,492,493:8X)

Just a suggestion.
```
