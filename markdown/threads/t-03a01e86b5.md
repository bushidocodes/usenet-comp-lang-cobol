You did not specify which compiler you are using.  This would have some
impact on the suggested code.

The quickest, assuming you have intrinsic functions, is to use FUNCTION
REVERSE.  Then you may INSPECT tallying leading spaces.

If you do not have FUNCTION REVERSE, or if it causes too much overhead, try
something like the following:

PERFORM VARYING I FROM 1 BY 25 UNTIL I > 1000
    IF THE-STRING (I:) = SPACES
        SUBTRACT 1 FROM I
        PERFORM VARYING I FROM I BY -1
                UNTIL I = 0 OR THE-STRING (I:1) NOT = SPACE
            CONTINUE
    END-IF
END-PERFORM
* At this point I is the string length = the displacement of the last
non-space character
* Note that the order of the tests on the UNTIL clause on the second PERFORM
is quite important

This code fragment assumes that most of your strings are actually much
shorter than 1000.  If this is not the situation, then a 'binary chop' would
actually do fewer tests.  If most of your strings are quite long, then a
variant of the code fragment which tests from the right would be faster.

In 1968 on an NCR 315 running a Tape based COBOL Compiler I first saw the
PERFORM construct used as you describe, Stephen.

Modification, so a rather ingenious programmer had got round the problem by
doing the following:

Perform dummy
   varying n
       from MaxLen
          by -1
        until char (n) <> space.
move n to TrueLen.

(Yes, we put full stops after every statement (well, sentence,
actually...<G>) as there were no scope delimiters...)

[Somewhere else in the program...]
dummy.
    exit.

EXIT Statement, and yet it was so simple and elegant and worst of all...it
worked!

that no other work can proceed until it is formally signed off! In fact, the
design process is also iterated, with a better design emerging from each
round.

This is a dynamic, flexible approach to EVOLVING a system, rather than just
building it.

The Spiral Model actually incorporates several development
methods, including RAD and SDLC. It does so by assessing
risk. That which is to be untertaken next will depend upon the
risk.

If all risks are low, "code and fix" may be selected. This
avoids the cost of creating a formal specification, though
an informal specification may apply.

If user interface risks are high and other risks relatively low,
RAD may be selected.

If other risks are relatively high, the more formal SDLC
may be selected.

Any job offering that mentions or implies SDLC and uses terms,
such as, risk assessment/analysis/management, prototype,
benchmark, etc., may be Spiral. This is because Spiral uses
those terms and SDLC does not.

The best way to do this is to ensure that the specification is written
in an appropriate way that can be matched by the program code.  This is
much easier if the program code is written first.  In fact once the code
has been written then the specification is almost redundant and what is
actually required is a user guide.

The way I deal with this is that as I write the code I add comments
within marked blocks eg:

      *<html>
      *<hr /> <!-- --------------------------------------- -->
      *<h4><a name="pricecode">PRICE CODE</a></h4>
      *
      * The price code indicates which price is given to a 
      *     customer and it should be specified as:
      *<ul>
      *<li> R = Retail price is to be used.
      *<li> C = Cost price plus markup is to be used, 
      *         the markup is given in the Group record
      *         for the product.
      *<li> n, where n is 1,2 or 3 = Trade price for category n
      *         will apply.
      *</ul>
      * The price codes need to be setup in the 
      * <a href="azdecode.htm"> Decode File Maintenance</a>
      * under the 
      * <a href="azdecode.htm#PRICING">PRICING field</a>.
      *</html>

           [code dealing with price code here]

Comments that simply relate what the code is (supposedly) doing are a
waste of space.  These html comments explain _why_ the code is doing it
and can be easily extracted into web pages that together make up the
system documentation.

Having the specification (actually as a User Guide) within the code make
it more likely that it will be updated as the code is changed and using
make to reconstruct the web page documentation means that the latest
version should always be available.  It can also be linked together ( <a
href=...> ... </a> ) so that users can navigate through a subject.

If the system has Help pages (eg using SP2) that can be called up then
these too can be extracted from the marked blocks in a somewhat
different way.

These days I do it all from being given a half page 'requirement' by the
client to installing the hardware and laying the cabling.  In previous
incarnations I have from time to time been a 'Project Leader' supposedly
co-ordinating the workers and reporting progress.  After doing a
half-hour 'project plan' and Critical Path the Management Schedule is
discarded and the project is reorganised to put all resources onto
critical path activities.

started, indeed before the customer had been met. Of course the client
wasn't told this, nor the management.  Usually I wound up writing the
User Guide (this was 20 years ago) and had the customer sign this off
while the programmers also worked from it and the trainer used it.

I have always found that it is best to start with the end products in
any system (report layouts, web page customer enquiry pages, extract
data to other systems) and work back through the processing to the
input.  For whole systems also start at the end product (User Guide,
support desk(?)) and work back from there.  If there is to be a formal
spec then do it last, that way you may not need to do it at all.

"w" was vv or vice versa.  Having an EXIT SECTION within a performed
paragraph is obviously bad, but unlikely to be detected by the compiler
or the run-time leading to program out of control.

An EXIT PARAGRAPH within a performed section has the property that it
doesn't exit the perform but merely goes to a point immediately before
the next label.  It is an unnamed GO TO.  Some programmers have
indicated that they add paragraph labels into sections as commentary
separators (a practice that is very bad in my opinion in any case).  The
point that the EXIT PARAGRAPH does its GO TO can be changed by adding or
commenting out these commentary names.

In the case of real GO TOs the label names can be searched for, or if
the label is commented out the compiler will complain.  With EXIT
PARAGRAPH the action can be silently changed by modifying code
_at_some_other_point_.

A similar problem occurs with combinations of NEXT SENTENCE and END-IF. 
One contributor here contrived a mechanism that defeated the intent of
the standard and allowed this combination.  Again with this the NEXT
SENTENCE acted as a GO TO to the unnamed point that was the next full
stop.  Some one else working in the area of this stop may remove it or
add another of their own because this stop is otherwise unrelated to
logic flow and there is no indication at that point that it is a target
for a jump.

If you are going to use a GO TO then make it explicit so that no one
misses it.  Using unnamed GO TOs such as these may lead to problems when
changes are made somewhere else.

In the real world of "free form" input data, there is NO "safe" number of
spaces to check for.  Especially as there are (at least) three"absolutely"
safe ways to check for this discussed elsewhere in this thread:

1) Do an INSPECT of the REVERSE of the field looking for first non-space
character

2) loop backwards thru the field finding the first non-space character

3) Loop forward with reference modification until you find the first *ALL*
spaces ending section of the code.

  ***

If this is something that you really do LOTS in your application and
performance/efficiency is essential, then PROBABLY (depending on field size,
compiler, OS, etc) #2 is most likely best.  My *guess* is that for many
programmers #3 is "best-self-documenting" (as it does what a "verbal"
description of the problem is asking for.  As far as the "simplest" (fewest
number of COBOL statements goes), then I would go with #1 - but that's
because I am "extremely" comfortable with intrinsic functions.  The only
thing that I would NEVER do is "assume" that some specific number of spaces
means that ALL the remaining characters are spaces (unless, of course, some
other part of the application has already done a compress multiple to single
space - for some other application reason).
