Gazaloo:

Your questions are good ones...

The last major release for sp2 was version 4.0 which was released in
March of 2000.  Since then, we have had a few maintenance releases to
add some neat new enhancements and correct some problems.  The current
production release is version 4.0.31.

Since we first released the sp2 product back in 1986, we have always
provided regular major updates to the product on an annual (or even
more frequent) basis.  Our first Windows version was released in 1990,
so we've had a ton of experience in this environment.

>2. can you "import" Micro Focus Dialog System screen sets?

Sorry, the answer to this one is no.  But there is absolutely no
reason why one couldn't be written....even by one of our customers.
You would have to have some way of importing the screen set into a
COBOL program.  If Dialog has some sort of export tool which exports
the screen set definition to an external file, then it shouldn't be a
problem.

We can import a lot of different types of screen definitions, because
sp2 includes special Panel Maintenance Functions which can be invoked
directly from a COBOL program.  This allows the COBOL programmer to
completely build a screen definition "on the fly" from their COBOL
application.

These functions are in the standard off-the-shelf product and most of
the conversion programs available have actually been written by our
customers.

>3. is the connection for thin/web client secure?

Yep.  We provide a 128 bit data encryption routine (even the keys are
encrypted) which can be turned off or turned on by the COBOL
programmer.

In addition, we're adding authentication routines to the server side
processing right now which will help verify the identity of the client
machine when Thin Client first launches an instance of the COBOL
program.

>4. does thin/web client require the cobol application to use SP2 for screen
>i/o?

Yes.  The Thin Client is tailored to sp2, because the COBOL program
passes a very small parameter block to sp2 in order to display a
screen (or print a form remotely if you want to use our FormPrint
product with Thin Client).  Because the parameter block is so small,
the speed of Thin Client across a TCP/IP network is very fast.  It
would make less sense to create a generic Thin Client, because
Microsoft already sells one....Terminal Server.

In fact, sp2 and FormPrint both work with Terminal Server.  There
really isn't any need for our Thin Client if you use Microsoft
Terminal Server.  

But there are some disadvantages with Terminal Server.  One
disadvantage is that you cannot run an Active X Control on the client
machine and perform local client machine tasks.

With the Flexus Thin Client and COBOL sp2, you can actually utilize an
Active X Control to perform functions at the remote client site.  This
could include such tasks as com port I/O for point of sale terminals,
reading scales or other client side I/O tasks.

The other issue is that there is a serious limit to the number of
concurrent clients which can be supported using Microsoft Terminal
Server.  Terminal Server has an upper limit of a relatively low number
of concurrent users...Thin Client can support hundreds of concurrent
users in an NT environment.

The overhead associated with Terminal Server is huge.  Thin Client's
overhead is actually very small.

Another advantage of Thin Client is the fact that we support both the
UNIX and Windows 32 bit Server platforms.  Many of our customers
prefer using a UNIX server with a Windows 32 bit client as their
client/server platform of choice.  Terminal Server only works with
Windows.
