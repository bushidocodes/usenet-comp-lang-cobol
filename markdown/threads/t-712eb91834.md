> 
> 
> Couldn't agree more!
> But - what's the purpose of the "high-value trick" at all?
> I've never done this in any of my programs.


In the particular example given the whole file was being read
and so using file status to end the processing is reasonable.

However, most of the time only part of a file is being 
processed and so it is slightly more complex.  For example,
all the lines of an order, all the transactions for a
customer, etc:

        MOVE Order-Number TO OrderLine-Key
        START OrderLine-File
            KEY NOT < OrderLine-Key
             INVALID KEY
                 MOVE HIGH-VALUES TO OrderLine-Key
        END-START
        PERFORM
            UNTIL OrderLine-Order NOT = Order-Number

             READ OrderLine-File
                 AT END
                      MOVE HIGH-VALUES TO OrderLine-Key
                 NOT AT END
                 IF OrderLine-Order = Order-Number
                      ......

This caters for the order being the highest number on file
and having no lines (will give invalid key on start) and
for last having lines and any embedded order.

It is not possible to use just the file status because
it needs to stop before end-of-file, and because there
may be, say, a record lock (you do allow for file
sharing don't you ?).  Also because START and READ will
not change the record when the fail (INVALID KEY or
AT END) it is not enough to look for change in key by
itself.

Now it would work if you used NOT INVALID KEY on the
START to avoid processing when that fails, and then
doing nothing in the AT END and using file status to
terminate the perform, but then it will read to
physical end of file each time.

This may not be noticed with test data (short file) or
when order numbers are being added at the end as there
will only be a small overhead each time because only
a small numbers of orders have been added.  But if 
the number is ever reset the processed time will 
suddenly blow out.

Indeed the area is static after AT END but only until CLOSE! My experience is that a lot
further in time / in the program again some statement would like to test for EOF and thus

I think/recall in particular programs processing in one run a mster file together with
one or more files containing updates to the master. (balance-line mechanism using key
compares). In this kind of programs it is quite common to define a working-storage copy
of the key (sorted)fields together with the EOF flag. Like (3 key fields):

01  currentSituationForFileMaster.
    03  csm-upto-field3.
        05  csm-upto-field2.
            07  csm-upto-field1.
                09  csm-eof      pic x  value zero.
                    88  eof-FileXXX     value "1".
                09  csm-field1   pic x(..).
            07  csm-field2       pic x(..).
        05  csm-field3           pic x(..).

01  previousSituationForFileMaster.
    03  psm-upto-field3.
        05  psm-upto-field2.
            07  psm-upto-field1.
                09  psm-eof      pic x  value zero.
                09  psm-field1   pic x(..).
            07  psm-field2       pic x(..).
        05  psm-field3           pic x(..).

Such a set is there for each update file too. O, you probably all know the trick.

Anyway: at all and every time you can without any harm reference each file-situation.

The read-routine of each sequentially accessed file with any key/sort order in any real
world program I always wrote/write is as such:

ReadFileXXX.
    Move currentSituationForFileXXX to previousSituationForFileXXX
    Read FileXXX next record
    At end
        Set eof-FileXXX to true
        Close FileXXX
    Not at end
        Add 1 to recCountXXX
        Move inrec-field1 to csX-field1
        Move inrec-field2 to csX-field2
        Move inrec-field3 to csX-field3
        If (currentSituationForFileXXX <  previousSituationForFileXXX)
*                                      <= if duplicates are not allowed
        Then
            Display "Sequence error on file ... etc at record # " recCountXXX
            Stop run/Call/Perform error handling
        End-if
    End-read

With //.. DD ..,FREE=CLOSE in JCL, this is some routine that always gave me more than it
cost.

At some IBM location (the IBM centre in Uithoorn, The Netherlands) this was THE standard
routine, in COBOL and in PL/1. For good reasons.

>I beleive that the record area for a file is static only while it is
>open.

On IBM mainframes without any additional specifications (viz, without the SAME
AREA clause), the record area may be a pointer into a buffer (buffer large
enough to hold a block of data), and there may be two buffers. The result is
that after each READ or WRITE the record area points to the next record in the
buffer or, if it was already pointing to the last record of a buffer and two or
more buffers were allocated, the record area pointer points to the first record
of the next block.

What happens at an AT END time is not defined, but the record area pointer may
end up being a "wild pointer".

For keyed files, the record pointer may also be a "wild pointer" after a random
operation that failed to locate a record (e.g., a START that fails, or a KEYed
READ with the record not found).

These are indeterminate record areas in addition to the "record area" being
indeterminate before OPEN or after CLOSE.

If one needs access to the same record area before OPEN, after CLOSE, at AT END
time, or after random I/O operations that fail to locate a record, the only
safe way is to use a WORKING-STORAGE area to hold the record image.

Now if one is dealing with spanned records (i.e., records that may span the
block or span a CI for VSAM files or FBA disks), a record area may be allocated
and remain static. However, I find it risky to rely on charactersistics
external to a program to guarantee proper operation of the program. (As a
Systems programmer, one of the changes I generally look at is  whether or not
it is feasable to redefine a VSAM cluster so that the records are NOT defined
as spanned. I have broken Easytrieve-Plus programs that way.)
