I've just been approached by an editor from IEEE Software to write an article
on converting legacy COBOL code to new COBOL code, specifically for the IBM
COBOL compilers.

This would include 
   * reasons for converting
     - supported compilers
     - access to new features
     - interoperability with other compilers
   * sample code fragments, showing how using new features produces
     better code
   * discussion of conversion techniques, including the IBM products
   * discussion of COBOL in the WebSphere environment

and little about the conversion products. The editor suggested we work with the
COBOL language newsgroup, to draw on their expertise and insights. I liked the
idea so this post is to announce this project to see if we can get some real
world perspectives.

March, 2000 issue of IEEE Software. Anyone who contributes material that is
used will be recognized as a co-author or contributor. (There is no
renumeration for me or anyone else, which is not all bad.)

So let me know, send me your ideas. You can post them on the newsgroup for
group discussion or you can email me. (Probably best not to post long pieces on
the newsgroup).

Thanks for your help, let's see what we can do.

Your list doesn't include the inadvertent problems that occur on
coversions and the costs they entail. It sounds like your article is
entended to sell the concept of upgrading to the latest COBOL regardless
of the cost.

To show the other side of the story consider the following list of
conversion traps T made up when converting IBM VS/COBOL (ANSI 74) to
COBOL II (ANSI 84).


COMP Fields

Problem

VS/COBOL ignores the picture for COMP fields except for determining the
size of the field. For example, a field defined as
"COMP PIC S9" is identical to "COMP PIC S9(4)". However, COBOL II
truncates the results of any calculation to a COMP
field to match the picture.

For example, suppose a program has a field defined as "COMP PIC S9" and
the field contains a value of 9. If you increment
the field in VS/COBOL, you get 10 (9 + 1 = 10). If you increment the
field in COBOL II, you get 0 (9 + 1 = 10 truncated to 1
digit gives zero).

Since programs often use COMP fields as subscripts and zero can be a
valid subscript value, this difference can produce interesting and
exciting results.

Solution

Identify all COMP fields in working storage and give them a picture of
at least S9(4). This is their effective picture under
VS/COBOL.


COMP-3 Fields

Problem

The version of VS/COBOL used before August 1996 used the full physical
size of a COMP-3 field regardless of the picture.
If a picture only defined an even number of digits, an extra digit was
automatically used. For example, the compiler treated
"COMP-3 PIC S9(4)" as if it were "COMP-3 PIC S9(5)". However, COBOL II
truncates any values moved to such fields to
the number of digits actually defined.

For example, suppose a program contains a field defined as "COMP-3 PIC
S9(4)" and moves 12345 to that field. Under the old VS/COBOL compiler
the result is 12345. Under the new VS/COBOL compiler and COBOL II, the
result is 2345.

Programs recompiled since August 1996 should have been changed to
correct this problem. However, any reengineering project may recompile
programs that have not been changed in many years.

Solution

Change the pictures for all COMP-3 fields so they define an odd number
of digits unless they should only contain an even number of digits. Most
fields should have an odd number of digits to take full advantage of the
physical size of the space used. However, some fields, such as dates,
should only have an even number of digits.


NEXT SENTENCE

Problem

After converting a program to COBOL II, it is natural to use the COBOL
II structures for ending statements rather than
periods for new code. This mix of old and new code can cause problems
with the NEXT SENTENCE clause in existing
statements.

For example, in the following code

    IF <test> THEN
      NEXT SENTENCE
    ELSE
      <move statement>
    END-IF
    <add statement>.
    <subtract statement>

if the test is true, the next statement executed is the <subtract
statement> rather than the expected <add statement>. This happens
because a "NEXT SENTENCE" clause continues following the next period
rather than with the next statement.

Solution

Make a mass change of all occurrences of "NEXT SENTENCE" to "CONTINUE".
The new CONTINUE clause does nothing. It just continues with the next
statement, which is the real intent of the "NEXT SENTENCE" clause.

I have compared the generated code from three compilers:

IBM OS/VS COBOL RELEASE 2.4

1. For the problem with COMP field the outcome depends on the TRUNC option.
With TRUNC(STD) (TRUNC for OS/VS COBOL) the effective outcome is identical
for all three compilers: the result is truncated so that only the last
digit (when interpreted as decimal) is kept.
With TRUNC(OPT) (NOTRUNC for OS/VS COBOL) the generated code is identical
for all three compilers, the variable is treated as a halfword, no
truncation takes place.
With TRUNC(BIN) (no equivalent for OS/VS COBOL) the newer two compilers
truncates the result at the capacity of a halfword, that is: no truncation
for the the value used.

2. For the problem with COMP-3 field the outcome is identical for all three
compilers: 2345.  The constant generated in the literal pool is X'02345C'
in all cases.  I don't have access to a very old OS/VS compiler, so you may
well be right there.

3. For the problem with NEXT SENTENCE the effective outcome is identical
for all three compilers: NEXT SENTENCE is interpreted as after next period
(by definition).  I don't see what the problem is, how should OS/VS COBOL
have behaved different?
