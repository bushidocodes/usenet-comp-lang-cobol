Another answer to the wrong question. It really is interesting that even in
response to a question that made VERY clear that he was asking people about
whether something was legal in COBOL, people still generally interprete
this as a question of whether something will work. As I said before, this
distinction is a vital one, and yet it is one that many programmers simply
don't register.

Recursion is illegal in COBOL because recursion is illegal in COBOL because
it says so in the COBOL standard. Whether or not recursion "wipes out return
points" has no bearing on the answer (although of course such implementation
concerns did play a part in devising the rules in the standard)

```
Mayer asks if recursive performs are allowed in COBOL. The definitive
answer is NO, they are definitely NOT allowed. Some compilers may permit
them, but if you write such a program, it is not a COBOL program.

The distinction between what is correct and what works confuses many
programmers (who simply don't know the difference). The question here
was asked in a very clear manner. The question was not "does this work"?
[the answer to that question is sometimes], but "is this correct COBOL"?
and the answer to that question is NO.

In my experience COBOL is remarkably portable, but ONLY if programmers
really understand what is and what is not legal COBOL.
```

"In short, I suppose there isn't a definitive answer. It obviously depends on

No, that's exactly wrong, the answer does NOT depend on the compiler. THe
answer to whether it works may depend on the compiler, but the questioner
asked if this was valid COBOL. The answer to that question can be found only
in the COBOL standard, not elsewhere.

P.S. You would not necessarily expect a compiler to detect this violation,
even if it caused chaos, because it is obviously undecidable (in the
computability sense) whether a given program results in a recursive
perform. Yes, simple cases could be detected, but it is not possible
to detect all cases, so many compilers may not bother to detect ANY
cases.

If I remember my college course on business data structures right,
COBOL, like BASIC, is hard to write recussively because the languange
cannot maintain a stack automatically. You must provide a stack
array with stack push and pop routines as you enter and exit each
level of recurssion. As the fixed stack array fills up, you may
either lose the data for the level of recurrsion or crash the
program, which ever causes you the greater embarrassment. Of course
debugging such programs is a certified nightmare since it's hard to
find whether the stack, logic, or basic assumption is at fault.

If at all possible, try a non-recurssive solution. The code may be
bigger, but it will save you many a sleepless night debugging the
logic.

Confusions abound here! It is easy to write recursively.
It is not the language that makes recursion difficult, it is the
(assumptions about the) underlying operating system.

And creating an iterative solution to a recursive problem (eg a
tree structure) costs you more in design/debugging time.

compiler. It is quite feasible (and has been done) to have a
dynamic stack maintained by the OS with a voluntary maximum size.

if the legacy remains evident in some languages/implementations.
