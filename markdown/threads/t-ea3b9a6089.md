Anyhow, the mechanism is (to me, that is) obvious, in NewBOL; the guts
of it would be reference-modification comparisons of characters, starting
at both ends of the field:

PERFORM VARYING SUB1 FROM 1 BY 1 UNTIL SUB1 > LENGTH OF PAL-FLD
    IF PAL-FLD(SUB1:1) NOT = PAL-FLD((LENGTH OF PAL-FLD + 1) - SUB1:1)
        SET IS-NOT-PALINDROME TO TRUE
    END-IF
END-PERFORM.

... or something close to that.

be slow on many compilers.  I propose the following fairly robust
solution, which does not use FUNCTION REVERSE.  It recognizes
"Madam, I'm Adam." and 1234321 as palindromes.

       IDENTIFICATION DIVISION.
       PROGRAM-ID.    PALINDROME.
       ENVIRONMENT DIVISION.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  REVERSE-AREA.
           03  RA-SUB                  PIC  9(04) COMP.
           03  RA-SUB1                 PIC  9(04) COMP.
           03  RA-SUB2                 PIC  9(04) COMP.
           03  RA-POS2                 PIC  9(04) COMP.
           03  RA-SIZE                 PIC  9(04) COMP.
           03  RA-TEXT                 PIC  X(50).
           03  RA-TEMP1                PIC  X(50).
           03  RA-TEMP2                PIC  X(50).
       PROCEDURE DIVISION.
       PALINDROME-TEST.
           DISPLAY "Palindrome test, enter text".
           ACCEPT RA-TEXT.
           MOVE 1                        TO RA-SUB1.
           MOVE FUNCTION LENGTH(RA-TEXT) TO RA-SUB2.
           MOVE 0                        TO RA-SIZE.
           PERFORM VARYING RA-SUB FROM 1 BY 1
                   UNTIL (RA-SUB > FUNCTION LENGTH(RA-TEXT) )
               IF (RA-TEXT(RA-SUB:1) NOT = SPACE)
                   AND
                  ( (RA-TEXT(RA-SUB:1) ALPHABETIC)
                     OR
                    (RA-TEXT(RA-SUB:1) NUMERIC) )
                   MOVE FUNCTION UPPER-CASE(RA-TEXT(RA-SUB:1) )
                                TO RA-TEMP1(RA-SUB1:1)
                                   RA-TEMP2(RA-SUB2:1)
                   ADD 1 TO RA-SUB1 RA-SIZE
                   SUBTRACT 1 FROM RA-SUB2
               END-IF
           END-PERFORM.
           IF (RA-SIZE ZERO)
               DISPLAY "No text"
           ELSE
               ADD 1 TO RA-SUB2
               IF (RA-TEMP1(1:RA-SIZE) = RA-TEMP2(RA-SUB2:RA-SIZE) )

               ELSE
                   DISPLAY "Text is not a palindrome"

test-begin.
  if pal-fld = spaces
     go to test-empty.

  move 1 to p1.
  move !length(pal-fld) to p2.
test-loop.
  perform with test before varying p1 by 1 until pal-fld(p1:1) not = space.
  perform with test before varying p2 by -1 until pal-fld(p2:1) not = space.
  if p1 >= p2
     go to test-true.

  if pal-fld(p1:1) not = pal-fld(p2:1)
     go to test-false.

  add 1 to p1.
  subtract 1 from p2.
  go to test-loop.

test-empty.
test-true.
test-false.

My own stab at it (we don't have functions on our compiler yet) is as follows:
"First accept the text into a character array specifying either 'upper' or
'lower'
so all characters would be converted to the same case.  In a perform loop from
1 to the
array size I test each character for being alphabetic and, if it was, increment

a second subscript and move it to a second character array.  That would remove
all spaces and punctuation as well as provide a count of the number of
characters.
A second subscript would be initialized to that count while half that count
would
be the upper value for another loop.  In the last perform loop, the character
in the
2nd array referenced by the primary subscript would be compared with the
character in that array referenced by the 2nd subscript.  If they matched, I
would
subtract 1 from the 2nd subscript for the next pass, if not I would set a flag
for a
'don't match' condition and exit the loop."
