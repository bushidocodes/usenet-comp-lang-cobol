There are slightly different answers for ANSI and ISO to these question.  I
noticed that your email ID is ".uk" - so I'll give both answers (to as much
as I can).

1) The last full revision was in 1985 (ANSI = X3.23-1985; ISO = 1989:1985)

2) There have been two (not just one) Addenda since then;
 the Intrinsic Function Addendum - ANSI X3.23a-1989; ISO 1989 amd/1 1993
    (there was a bureaucratic delay in getting the ISO version out)
 the Correction Amendment (Addendum) - ANSI X3.23b-1993; ISO 1989 amd/2
1994)
    (the "1989" in all of the ISO numbers identifies it as "COBOL" and is
NOT a year)

3) The current "full revision" is presently out for both ANSI and ISO Public
Reviews.  It is still THEORETICALLY possible to ask for things to be added
or deleted - but unlikely to get either.  If you check the comp.lang.cobol
archives or www.cobolreport.com you will find an "article" that I did on how
to get the draft and submit reviews (either in or outside the US).

4) The ENTRY statement has never been part of any ANSI or ISO Standard,
although it is a "relatively" common extension.  The same functionality (but
certainly NOT the same syntax) can be accomplished with nested programs
(among other approaches).  J4 (the ANSI COBOL group maintains a list of
"candidates for a future revision") If there is something that you would
like "added", then you should send your suggestion (with as much detail and
supporting information as possible) to the J4 chair at:
    doncobol <at> mediaone.net

5) Besides documenting (in their "vendor documentation") what are extensions
and what is standard, every ANSI/ISO conforming compiler is REQUIRED to
provide a mechanism for flagging the use of extensions at compile-time.  If
you are using a compiler that supports ENTRY statements, my GUESS is that
you should look for a "FLAGSTD" compiler option or directive.  If your COBOL
compiler doesn't have that, tell us which compiler you are using - and we
can tell you how to get compile-time warnings of your use of extensions.  (S
ome of which you MIGHT want to suggest get added to a future COBOL
Standard.)

6) "Real COBOL coders" are encouraged to "participate" in the COBOL
standards development at all stages - suggestions for new features, review
of draft documents, etc.  HOWEVER, the responses are *not* "speedy".

     ***

I hope this helps.

Alister Munro wrote:

> Was the last Cobol standard set back in 1989 ?

The last language standard was 1985, the 1989 addendum added the
intrinsic functions. The next standard has been scheduled since about
1997 but probably won't be this year either. Did you want new standards
more frequently ?  Did you want a language like VB where each release
requires programs to be rewritten (as will occur again with the next
release).

> I need to know what standard is available within each compiler
> and what defines that standard.

Every major compiler should be implementing the full and correct 85/89
standard.  I suspect that you really want to know what _non-standard_ is
in each compiler.  If you look at the MF manual you will see that the
extensions are marked 'MF', 'OS/VS', 'X/Open', etc.

For other versions read the manuals.

>    'The reason ENTRY will not become part of the COBOL
>    standard is because it is considered "unstructured."
…[5 more quoted lines elided]…
> an overhead on the both the cpu and my fingers. 

You confuse size of source code (ie fingers') with executable size (ie
cpu).  Even though ENTRY may be less source code than, say, nested
routines, does not mean that there is less executable code.

In fact the way that Cobol works with dynamic loading of routines there
is a process that must be executed for each call and that is to search a
table of loaded routines.  If the routine name is not found then the
disk is searched and the program loaded.  All routine names are then
loaded into the table so that later CALLs can find them.  If you use
ENTRY extensively then this table will get larger and there will be a
larger search overhead on every call.

> Who decides
> what goes in and what comes out ? Can we as Cobol coders
> contribute ?

Yes, you can submit to the standards committee, but they have closed off
for the next standard (2002?).  Next one after that may be another
decade or more later.
