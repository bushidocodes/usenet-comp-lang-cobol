I have seen copybooks used well.  For example in a personnel
application called CHRIS that I worked on it was non platform specific
and hid the CICS stuff in standard copybooks and subroutines.  The
main driver was copied in and as Volker said overridden:

   COPY STDPROC  replacing ==standard-verification== with
			 ==special-verification==

This is effectively macro processing.   OK but i would prefer easier
macro methods.


I have seen the result of putting copy book code into place and then
changing from sections to paragraph (for standards) causing very
interesting results.

In most cases it should be subroutines.  You can get better
reusability from subroutines than performs e.g.:

	Call 'chknum' using number1 num1rslt.
	call 'chknum' using number2 num2rslt.

instead of;
	move number1 to called-number
	perform ..
	move number2....

The only problem I have with calls in cobol is that you cannot put a
definition into place on them, as the above sample:

	call 'chknum'  using number3.

will probably bomb with a protection error rather than the compiler
picking it up.

The advantages of calls over copybooks is that you can save time
recompiling if you use make on PC or Unix and SCLM (which has a make
built in) on MVS.

Ken

One of the prime tenets of good structured design is that procedures
should be self contained with minimal linkage.  By their very nature,
Procedure Division COPY files are conceptually a type of 'procedure',
whether the resultant code is actually structured as such.  To include
straight line code in a COPY statement increases at least the implied,
or potential, linkage, and strike me as a bad idea.  I prefer to make
all COPY files to be free standing paragraphs, or sets of paragraphs,
placing necessary Working Storage fields in an associated COPY, with
well defined input, output, and function.  Of course, making the COPY
a callable subprogram would also be fine.  My $.02  :-)

COBOL simply requires more programmer discipline to maintain proper data
segregation.  The data naming method I use has served me well for several
decades.  I use carefully named 01 entries for each COPY procedure, then
create a 3-6 character acronym from that name.  Then I append the acronym
to each data name under the 01.

For example, for a COPY paragraph named "DATE-EDIT", the associated 01
entry might be "DATE-EDIT-WORK", and all the subordinate entries would be
prefixed with "DEW-" or "DATWRK-".  When I develop systems, I maintain a
data name dictionary and abbreviation dictionary for that system so every
field is named exactly the same thing everywhere it is defined or used,
except for the unique prefix of the particular 01 record it is in (COPY
file or no).  Each file, of course, has its own unique prefix.  Print
file, temporary work file, and Working Storage 01 record prefixes need
only be unique within the program.  Using "WS-" as a prefix for all
Working Storage entries is pointless, because some programs have hundreds,
perhaps thousands, of such entries.  Unique prefixes tell you which 01
record is involved at every reference.  Plus, because the prefixes are
unique, the rest of the data name can be consistent throughout the whole
system.

There are 2 considerations for me in using COPY in Procedure Division.
Development:
When I develop modifications for the system the COPY statement provides ease
of implementation. I can change one copybook and it would eventually apply
to many load modules in many libraries. I can make last minute change and
start mass recompile.
Debugging:
When I work on production support and get calls in the middle of the night
because system is down/abend, the COPY statements caused me allot of
trouble. I have to constantly switch from 'source' library to 'copy' library
just to read code. This is the time when I HATE COPY statement.

Called program would be preferred method for me in both of the above
instances.

P.S. I work in mainframe environment with many production libraries and
often provide support for programs I have never  seen.
I read this group periodically and find this group very helpful - first time
posting.

A someone who originally worked with assembler where the use of
macros is rampant and essential I think this is a good idea in
principle. Macros are used in C a lot as well.

Basically they reduce the amount of code you have to write. One
program I was working on we replaced multiple invocations of
various routines by copy verbs and reduced line of code by 50%.
At no loss on meaningfulness (if the copy expansion is concerning
you, have a look at the cmpilationm listing - you do keep the
compilation listings for your live code, don't you?). We also
fixed a lot of bugs. Previously, the programmmer had to do
something like

Move '12' to parm1

move 'a' to option-setting-a

move infile-data to parm-3

set parm-orig to true.

(etc)

call 'mangeleit' using parm1 option-setting-a parm-3 did-it-work

(The programmers were supposed to use standard parms and a
separate paragraph to call the subroutine, but that was too
boring for some of them).


This turned into 

copy "mangleitpd" replacing
	==function== by ==update==
        ==option== by ==speed==
        ==data== by ==infile-data==.        


If you forgot a parameter the copy was set up to generate a
syntax error.

Of course you can overuse them, use them in tricky ways, and use
meaningless names and thereby turn a good think into a bad idea.


The copy verb lacks a lot of power (eg nested copy replace) and
the inability to have data local to a paragraph does create
problems. But used with discipline it is very useful.


Tim Josling
