Morgan, check your SORT verb syntax for more help - BUT - From
NetExpress on-line help :-

SET ANS85

select ifile assign to ipf
   organization is line sequential
   file status is ipstat.

select sfile assign to "temp.dat".

Select ofile assign to opf
   organization is line sequential.

fd ifile.
01 irec           pic x(80).
fd ofile.
01 orec           pic x(80).
sd sfile.
01 srec           pic x(80).

working-storage section.
01 ipstat.
  03 iskey1       pic x.
  03 iskey2       pic x.
01 ipf            pic x(20).
01 opf            pic x(20).
01 ext            pic x(20).

01 clin           pic x(132).
01 len            pic 9(2)   comp-x.
01 a              pic 9(2)   comp-x value 0.
procedure division.
    accept clin from command-line
    unstring clin delimited by space into ipf, opf, ext
    if ext not = space
        display "too many arguements
    end if

sort sfile on ascending srec
    input procedure sortin
    output procedure sortout

stop run.

sortin section.
    open input ifile
    read ifile
    perform until ipstat not = "00"

        move irec to srec
        release srec
        read ifile
    end-perform
    close ifile.

sortout section.
    return sfile at end go to sortout-exit
    display srec
    go to sortout.
sortout-exit.
    display "Done".


Jimmy, Calgary AB

That is to be expected.  Indexed files are a wonderful invention which
allow us to do certain tasks (random access) quite effectively.  But
they weren't designed to replace the far more efficient SORT MATCH/MERGE
processes.  

Keeping output in a multiply indexed file ready for an efficient report
can occasionally be useful, but administration is much easier if we
simply extract the data from the input database, and recreate the report
as needed, using temporary sort files which go away when the job is
done.

Adding on an index every time you want your data in a different order,
is a process guaranteed to slow down updates, to create tremendous
overhead, and to complicate the system.   On the other hand, it is
slightly less work for the initial programmer, who might not care about
his customers and the administrators.

There are many good reasons for adding this overhead.  Multiple indices
can make random data retrieval effective.  But sorting a report is not
one of these good reasons.

Having done VSAM tuning in the not that distant past, I can guarentee
that it can make a tremendous difference.  I have seen it cut an hour
off a run where the program was randomly accessing elements of a table
file.  Because the entries covered more than one data control interval
(CI) normal random-access VSAM caused a LARGE amount of physical I-O as
the control intervals were overlaid and re-overlaid.  (Normal VSAM will
only keep one data control interval or buffer in memory plus use a
second for splitting the information between two buffers during an
update if needed).  Since batch LSR will keep more than one data control
interval in memory as well as multiple index control intervals including
sequence set intervals, almost all of the physical reads were
eliminated.  This was almost as effective as building an in-core table
which I had done as the first attempt at optimization.  Since batch LSR
was only a JCL change and didn't require user approval, the in-core
table solution was scrapped even though it further improved
performance.

I have only used batch LSR (local shared resource) for KSDS (key
sequenced data set) files.  Since there is only one component to an RRDS
, I would assume (perhaps dangerously) that local shared resources bring
no benefit to the optimization.  Similarly I doubt that ESDS (entry
sequenced data sets) will benefit from LSR pooling.  

While this discussion is oriented to MVS/OS390 systems, some principle
may apply to other environments.  I used SMF (systems management
facility) information from type 30 (hex 1E) subtype 4 - step termination
and type 64 (hex 40)  - VSAM close records to determine the areas where
use of BLSR would help.  These records had been extracted into DB2
tables by the database group and formatted for easier use.  I used
DYL280 jobs to target job steps by execution time and others to find
files with a large amount of physical I-O in relation to the logical
read-write activity on either the data component or the index
component.  In the shop in question, I found that there were cases where
BLSR gave drastic improvements and others where it actually caused worse
performance.  It was difficult to determine the type of access in many
cases because the access was DYNAMIC for the major programs.  This was
because the package used one I-O subroutine for each file that had all
types of logical access (random, sequential or dynamic).  The shop had a
standard of coding the AMP parameter on the DD to improve the buffering
over the default for 2 data control intervals and 1 index control
interval.  I also improved VSAM performance in sorts by eliminating the
AMP parameters and allowing the SORT program to do the optimization
(SYNCSORT from SYNCSORT, Inc.).  I assume that DF/SORT and CA-SORT have
similar capabilities.

Note that in MVS the indexed file and relative record file systems are
components of the operating system.  There is one OEM replacement that
is used by many shops although at none I have been at.  On other
platforms, it may be worth gathering performance statistics and tuning
file access.  One advantage of this type of tuning on the mainframe is
that no program changes are required and the change control and
verification requirement are relatively low in many shops.  I would be
interested in seeing any postings on the I-O tuning done on other
platforms.

The following comments are aimed primarily at the COBOL community for
IBM's MVS or OS390 operating systems but the methodology and principles
should be common to many platforms.  

While most programmers rarely even check generated code to see if they
are doing things efficiently (and doing things efficiently is becoming
less important), there are times when an applications group is asked to
investigate problems with performance.  In general you should be
checking with your systems programming/administration/support groups to
find out what tools are available.  The database and transaction monitor
(CICS etc.) people also should be consulted.  This way you may be able
to use the tools they already have to get the information you need to do
the job.  In the IBM world, you can find out the record layouts for all
of the standard IBM system measurement facility records online because
all of the manuals are online.  You can also read the library normally
called SYS1.MACLIB for the assembler record descriptions (they begin
with IFA as I recall) if you have read access.  However you may well not
have read access to the files where these are stored and you shouldn't
be reading the online collection files to which you probably shouldn't
have access.  Similar comments apply to the other environments.  

In specific, I have used the SMF 14/15 (hex '0E' and '0F') - non-VSAM
data set read and write records primarily for usage information (what
job or tso-id) is reading or writing the data set or library member.  I
have also used the information on these records to find badly blocked
files and small tape files that could be put on disk.  The SMF 30 (hex
'1E') sub-type 4 step termination and subtype 5 (job termination or
started task termination) have be used to measure overall cpu and
run-time plus physical input/output operations for job steps, as well as
submitter and owner of jobs.  The SMF 64 (hex '40') VSAM close records
have a wealth of information on I-O activity, both physical and logical
(now if only they kept track of read not found and write duplicate key
conditions).  CICS cuts a type 110 (hex '6E') record with various
sub-types that may contain useful information.  DB2 also writes various
records.  Many years ago I used the GTF trace load, link and attach
records to find heavily used dynamically called vendor and in-house
subroutines.  I was surprised at the low overhead of this trace.  I have
written COBOL programs that used the types 14, 15, 30 and 64 records. 
There may be records being recorded and saved by various monitoring
packages that can be used.  

The use of some of the records can be interesting, especially the SMF 30
record where the job and step identification information is located in a
part of the record with a variable offset from the beginning of the
record.  COBOL 85 handles this record easily but it has to be
reformatted for general use such as sorting.  In one shop I was on the
systems programming staff when I was using the information.  In another
shop I was on the application maintenance team and Year 2000 project
where I used the systems information for both application tuning and
obtaining program and data set usage.  In this posting I have focused on
the records with information that I believe is useful in tuning at the
application level and finer (transaction, job, etc.). Note that any
improvements that can be made without changing code are far easier to
sell and implement.  The information I was looking at helped find the
areas of opportunity.

This is essentially how the program was fixed.

The search of the 10,000 entry table was especially inefficient.  The
table was searched serially with PERFORM UNTIL for each input record (10
million to 17 million input records).  The table was always searched
until the last entry was checked, even if the table entry's customer
number was greater than the customer number of the current record.  That
added an extremely long instruction path to the program.

The big table was loaded from a file that was in sequence by customer
number, but code was added to verify its sequence and abend if out of
order.  It was relatively simple to build the index table while loading
the primary table.  The original PERFORM UNTIL was replaced by a SEARCH
ALL on the index table to obtain the subscript into the primary table. 
The primary table was then processed sequentially until a control break
on customer number.

The end result was approximately a 93% reduction in CPU time and a 75%
reduction in wall clock run time.  No I/O optimization was attempted. 
The program was moved into production without any JCL changes being
required.  The original program tied up three tape drives for the entire
duration of the job, so any performance improvement frees up quite a lot
of system resources.

Most of you picked up immediately on table searching as a primary
suspect.  The big clue for me was the excessively high CPU time.  A
typical batch COBOL program is I/O bound, and should consume only a few
seconds or a few minutes of CPU time for each hour of wall clock run
time.

Yes, it IS VSAM. But it is NOT VSAM Key Sequenced Data Sets (KSDS). Many
people reading this and unfamiliar with mainframe environments will assume
that VSAM is the equivalent of ISAM on PCs, and will mistakenly assume that
mainframe DB2 uses indexed datasets like ISAM.

For non-mainframers:

VSAM (the "Very Silly Access Method", a bit less silly than ISAM (see
below), or "Virtual Storage Access Method" if you like, is comprised of
three parts:

ESDS = Entry Sequenced Data Sets - unordered data sets randomly accessible
on Relative Byte Address (RBA)

RRDS = Relative Record Data Sets - accessible on relative record number
(direct hashing)

KSDS = Key Sequenced Data Sets - accesible on sequential key just like ISAM
on the PC.

DB2 is implemented using VSAM ESDS and is NOT based on a Key Sequenced
method.

To clear this point and hopefully resolve confusion:

Mainframe ISAM (the "Incredibly Silly Access Method" or "Indexed Sequential
Access Method" if you like...) came first and was a system which used
Indexes on sequential files to provide random access via a key.

It suffered because it's maintenance methods were poor, with records to be
inserted placed in "overflow" areas with pointers to them. This meant that
manual housekeeping had to be performed at regular intervals to reclaim and
re-organize the underlying files. This was such a pain that ISAM in the
mainframe world is now almost (I nearly typed "virtually" there...<grin>)
extinct.

VSAM (see above) was the next step and was infinitely superior because it
was "self re-organizing" . No need for housekeeping. All of the "flavours"
of VSAM detailed above are self re-organizing.

This is achieved by binary splitting of control areas and control intervals
to distribute free space throughout the file.  (Housekeeping happens
automatically as you use the file and you just have to cop the overhead.
Actually it is very good because it only does it when it HAS to, and it only
does what is necessary (unlike ISAM, which always required the ENTIRE FILE
to be re-organized.)

The nearest thing I can think of to compare it to in the PC world would be
B-TRIEVE. Unlike ISAM, VSAM is alive and well in the mainframe world and is
so good it has been around for 20 years and, as the underlying access method
for the major RDB offerings in Mainframe Land, will probably be around for
the next 20.

So when we talk about ISAM in the PC world we are talking VSAM KSDS  as the
equivalent in the mainframe world (PC ISAM files today, are also self
re-organizing, based on the B-TRIEVE model.)

Relational Data Bases on the mainframe (DB2, UDB) do NOT use Indexed
Sequential access methods. (These are archaic methods which mainly exist
today as part of the COBOL file system [yeah, I know, you all love it and
think its wonderful, but it is NOT an industry standard and isolates any
systems you write which use it.])

The reason I advocate using ODBC and embedded SQL in COBOL programs is
because you can then attach your system to ANY relational DB and get all the
benefits of the other tools which now exist (like, why write reports in
COBOL when your users can call off WHATEVER they want quickly and easily
using tools like Crystal Reports?)

Database changes? No problem. Generate the new COBOL record definition using
DECLGEN and recompile.(no maintenance of existing code required).

Volumes increase and consequent performance degradation? No problem. Move
the users to another database (say from MS-ACCESS to ORACLE or INGRES or
DB2 or UDB or whatever. Duplicate the table structure on the new database
and reset ODBC to point at it. Done! Not even a re-compile!

Multi user system? No problem. Locking and scheduling of multiple threads is
automatically handled by the database software. So is recovery and
transaction integrity. Why sweat it with manual locking when the system can
do it for you? Not only that, but you are GUARANTEED integrity.

And please don't insult my intelligence by telling me horror stories of
terrible performance using RDB.

That was then, this is NOW.

If your RDB doesn't perform, there are a plethora of tools to analyse what
the problem is and get it fixed. Any DBA group worth its salt will solve the
problem within a day, and most of them will prevent it happening in the
first place by making sure the corporate data resource (whether distributed
around the organization or centralised) is within agreed design guidelines.

These days, when I have to handle data I ask myself the following:

Need random access to this? Yes; RDB (ODBC and embedded SQL). No; LINE
SEQUENTIAL. End of story (at least for me...I respect your right to get it
wrong...<grin>)

COBOL file system is NOT an option (for me) for any new development.

DB2 (and others) and speed:  DB2 buffer files in memory in a common
address space.  If you have multiple programs hitting the same data at
the same time it is in memory and very quick.  It does chew a lot of
memory as a result of course.

DB2 has transaction control.  I can do a complete update and then
something goes wrong and I can roll back to the start of the entire
process.  Great for program crashes, just rerun it after correcting
the problem.

DB2 has SQL.  Joins across tables then you simply get a set of values
back in the order you specify ready to process.  This virtually
eliminates extract sort report logic.

DB2 and Indexing:  You can add and remove indexes without changing any
code.  

DB2 and Indexing 2: Often the path you think is efficient given a set
of indexes is not the fastest way to the data.  For example you might
start at customer, locate orders, and then locate order items and
check for outstanding flag.  DB2 collects statistics (runstats) that
tell it to reverse the traversal because it narrows the search
faster.  It also uses parts of the index without returning data to
subset searches, you simply cannot do this with VSAM files.

DB2 and binds:  They are a pain but they make the access very
efficient.  If you do not run 'runstats' it can be incredibly dumb as
well.  With the good comes the bad.

I developed a timesheet entry system for a mid-size company -- about 600
employees -- so that the managers could get manhour reports quickly.  
The reports, of course, needed to be sorted and summed in various ways.  

The environment was CICS, and in CICS you're not supposed to use the
SORT verb.  (At least you weren't back then; I don't know about now.)  
To get around that problem, I used a VSAM file as a temporary sort 
space.  I built a key depending on the kind of report and wrote the 
records to the VSAM file.  Then I read them back sequentially,
prepared the report, and deleted the records.  By including a task id
in the sort key I could prepare multiple reports concurrently without 
getting them mixed up.

It was all terribly clever and it worked, but it was an absolute dog. 
Some reports took hours to run and cost hundreds of dollars in bills 
from the service bureau which ran the computer for us.  We wound up 
writing JCL to an internal reader, including the records as instream 
data, and preparing the reports with ordinary batch jobs submitted from 
the CICS region.  That way the reports showed up on people's printers
within minutes.

The lesson I learned was: VSAM is a terrible sort utility.

Mr. Dilworth might argue that we should have built a VSAM file with
multiple alternate keys, instead of using VSAM as a temporary sort 
space.  We didn't try that.  Perhaps it would have worked.

Others have noted, correctly, that large volumes of data may call for a
sort rather than for indexed files.  However the volumes in my example 
weren't terribly large -- maybe tens of thousands of records, or 
hundreds of thousands if a report covered a long time period.  (That 
was before I started working for a telephone company.)

The point raised by my example is the timing of a sort.  If you need to
do a bunch of sorting all at once, a good sort utility will beat an 
indexed file every time.  It will probably beat a database, too.  On the 
other hand if you rely on indexes to do your sorting for you, then you 
distribute the sorting over time, as you add records (or update their 
key fields).  If you add records a few at a time, as in a typical online
environment, the overhead of updating the indexes probably won't be
noticed.  If you add large numbers of records in a batch process, the 
overhead may well be intolerable.

Part of the discussion on sorting depends on volumes.  In another
context I looked at a mainframe job that was grossly inefficient but the
total CPU and elapsed times were so small that it made no difference.  

In the context of the files that Tony is dealing with there are several
differences between the PC world and the mainframe world that I am
familiar with. 

1.  The Indexed Access methods (B-trieve, etc.) may well be more
efficient than moderately tuned VSAM.  AIX's (alternate index files) are
especially hard to tune.  I am guessing that sufficient read-write
buffers are retained for both the data areas and each of the indexes so
that if there is sufficient locality of reference the buffers stay in
memory.

2.  My guess is that there is relatively low concurrent usage of files
by multiple users in each of Tony's systems.

3.  The systems have relatively low insert/delete volume on the files
because this is where the time is eaten and depending on the underlying
ISAM architecture intriguing file growth can occur.

4.  In keeping with my earlier statement, the use of sort/extract
instead of multiple keys and other designs alluded to earlier might
bring vast but unseen and unneeded efficiencies to Tony's systems. 
Given the nature of the PC and the relatively low amount of concurrent
processing and the interactive nature of most processes, the response
time perceived by the person at the keyboard may be the only important
metric.  

5.  Definitely on the mainframe and probably in the PC environment,
relational databases can offer some distinct advantages over indexed
files.  If each table is considered the logical equivalent of a file,
the enforcement of relationships (you can't delete the customer until
all orders for that customer go away, etc.) and potentially new ways of
indexing can both improve performance and reliability.      

6.  On modern systems with large amounts of both real and virtual memory
available, both PC and mainframe, internal sorts should be about as fast
as external sorts and you can tell what fields you are sorting on.  In
Tony's environment, they might best be used for the oddball request
where the frequency of use doesn't warrant setting up another key
(index).  In Tony's case an external sort might be dangerous because the
customer might attempt to change it and the amount of documentation
necessary for the customer to do so safely could be interesting.

A file is far more useful only if you use it.  Keeping it is a cost, if
you don't use it , it is a waste.

You can avoid using a sort, or you can avoid using an ISAM file.  Don't
avoid either one, but instead be familiar enough with the various
methods that YOU can pick the best solution for your situation.

When I have needed sorted data, I have found each of the following has
advantages in different situations:
1.  Create an ISAM file.
2.  Tell my database to provide me my results sorted.
3.  Use an external sort and a temporary file.
4.  Use an internal sort.
5.  Sort by hand.
6.  Create an index for an already existent ISAM file.

Over the years, my use of ISAM files has dropped considerably, while my
use of databases has increased.  My use of sorting has remained pretty
constant.  Administration costs generally say, don't create files -
instead manage data.

Your Mileage May Vary.  Certainly, you don't want to assume that either
Tony's environment nor mine will apply to all situations.
