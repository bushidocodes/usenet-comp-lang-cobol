I've done this in the past. When dealing with small tables, a bubble sort
seemed not worth the time to code. When dealing with large files, a common
way today is to create an indexed file with the key whatever your sorting on
and moving one file, or its keys, into the sort file. Then use the indexed
sort file as your sort. In the past few years I've found little use for
this, excepting doing an extract and dump program. Instead, I usually add
another key to the file (note this is not an endall but depends somewhat
on the situation).

Anyone ever heard of a COMB sort? It was published in BYTE magazine
about four or five years ago. I ended up coding a COBOL equivalent and
found it to be INCREDIBLY fast given the relative simplicity of the sort.
Basically, you pass through the list with an ever decreasing "comb".
(i.e. In a list of 100, you'd compare 1 and 50 and make sure 1 is lower
than 50. Then you'd compare 51 and 100 and make sure 51 is lower than 100.
At this stage, 50 is the size of the comb. You continue to decrease the
size of the comb until you reach 1 and you pass through the list with
no swaps. NOTE: When the comb size is 1, it's just like the first pass
of a common bubble sort.)

I ended up taking a colleague's Turbo Pascal program, which compared
QuickSort, Insertion sort and Bubble sort, and added the COMB sort as
an option. The program compared sort times and I was quite amazed that
the COMB sort was just barely slower than the QuickSort but handily
beat the other sorts by several orders of magnitude.

If anyone would like the code, I can post it. It's very short and I've
used it a number of times when I have an in-memory table that I need
to sort. Given how simple it is, I've replaced using BUBBLE sort
techniques with COMB sort techniques. It also works very well on
large lists. Also, I don't think it slows down if the list is already
sorted. (In fact, it should speed up since no swaps would be required.)
