```
I would start simple to build confidence, then gradually add more functions,
therefore complexity. For instance, start by creating an input file with 1
record consisting of just 3 fields; a, b, and c.  Write a program to open
the input file and an output file, read the input file, add a to b giving c,
move the record to an output record area, write the record, close the files
and end the program.  Look at your output file to see if you got what you
expected.  If not, use displays in your program to debug where you went
wrong.  After success, add more logic like using perform statements,
evaluate statements, checking your input fields for valid values, or valid
ranges, etc.  Keep adding COBOL verbs until eventually, you've used ALL of
them.  As you add each new statement, study it's syntax in the programming
reference manual to fully understand what it does.  The key is to start
simple and gradually add complexity.
```

```
Hi:

Try doing something you will have to do in the real world like a
simple report program. "Hello world" is not the sort of program
you would get paid to write. Besides, I think you are beyond that
stage.

Here are the specs.

You have a sequential file (which you can create with your editor)
in sequence by employee number PIC 9(03) in cols 1-3. Columns 4-8
contain the hours-worked PIC 999v99. (I would not suggest
creating a file with an editor in a production system, for your
purposes you can).

The program has to read the file and print totals by employee. There
can be many records per employee. This will introduce you to the
classic control break problem.  Total lines contain employee number
and the total hours. Grand total at the end. Page number on the
reports - skip to new page after 55 lines. You will have to break
control on employee number. You might as well sequence check the
file to make sure it is in sequence - put this in the control break
part of the program. This is real world stuff.

I also recommend drawing a flow chart before you start to code. I
don't know if they teach you flowcharting in school or not. I am
inclined to believe it is a dead art - try it to see what you might
learn.

Things to consider are: a file with no data in it; a file with only
one record in it; no file in existence, etc. These things will
happen in the real world and you should prepare for it.

Be sure to include file-stats checks, you never know what will
happen next in the real world.

If you use this, feel free to ask further questions.
```
