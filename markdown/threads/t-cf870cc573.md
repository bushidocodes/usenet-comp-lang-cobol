```
If you use Java, your apps will be binary portable (as are many Cobol
systems).  C++ has a CPU speed advantage, but this will be neglible for type of
I/O bound processing you describe.  A C++ standard is only now being
standardized, and there are very few conforming compilers (most C++ compilers
try to track the evolving standard, but are behind).  Java 1.1 is standard and
stable.

> How would the Java or C++ program look to say do the equivalent in Cobol of
> opening a file reading each record, writing processed records.  How well
> will it perform on say a 40 mil record flat file.  Any tricks to deal with
> any 2 gig operating system limitations.

Neither C++ or Java include record I/O as part of a standard library.  Java 1.1
includes an SQL API (JDBC) that is close to what you want.  If you want ISAM,
rather than SQL style access, you'll have to use or write a proprietary library
for either Java or C++.  We use an XOPEN access library for Java that we wrote
ourselves.  IBM has also written an ISAM access library.  It was needed for the
AS/400, but is 100% pure Java and useful anywhere ISAM style access is
desired.  It is on their AlphaWorks site.

Performance will be about the same as an interpreted Cobol (one of the binary
portable Cobols, e.g. AccuCobol), and is mostly determined by the ISAM you are
using.  You can use the same ISAM your Cobol uses if the interface is
documented.
```
