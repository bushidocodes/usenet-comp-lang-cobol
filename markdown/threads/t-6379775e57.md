вЂє As an aside, what COBOL compilers can handle multi-threading?

Micro Focus does. Has done since at least the late 80's, under DOS even, though
I suspect it's not your or rather Microsofts idea of multi-threading. More like
concurrent tasks in the the style of Deskview.

It's a bit tricky to use without knowing the run-time and XM really well but it
works. I still have an application which uses 18 concurrent tasks with a
scheduler written in COBOL. Device drivers in COBOL, Networking (NETBIOS) in
COBOL. Runs faster than the Windows version (about 6 times faster) and only
needs about 2 meg of RAM as opposed to 16+

The newer products, VisOC, NetExpress onwards have a whole lot more runtime
calls for handling threads, semaphores, mutexs etc. You had to handle the
semaphores yourself in the older product hence writing your own scheduler is a
good idea.

Acucobol 3.2 and higher support multithreading via some extensions to the 
CALL & PERFORM verbs:

77 Thread-A  			HANDLE OF THREAD.

PERFORM THREAD Paragraph-A 
  HANDLE IN Thread-A

or 

CALL THREAD "CBL-PROG" HANDLE IN Thread-A.


You can also send arbitrary data items to other threads using the SEND and 
RECEIVE verbs:

SEND My-WS-Item TO ALL THREADS.
RECEIVE Another-Item FROM ANY THREAD.
IF Another-Item(1:2) = "RD" THEN
  SEND "Redraw" TO LAST THREAD
END-IF.

Finally, there are synchronization functions:

WAIT FOR THREAD Thread-B.
or
WAIT FOR LAST THREAD.

There is one caveat to this: currently the runtime does not use operating 
system threads, as there are significant portability issues, particularly 
since many of the platforms we support do not support threading. Support 
for OS threading on platforms that support it is being investigated, so 
this could change in the future.
