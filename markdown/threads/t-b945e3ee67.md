### OSVS Cobol vs COBOL II under MVS

- **From:** "Tom Eisenman" <Tom_Eisenman@NOJUNKbigfoot.com>
- **Date:** 1999-01-13T00:00:00

```
We have a COBOL program that is CALLed by a program written in another
language.  The COBOL program reads a long file and builds a table in working
storage. It then does a search to find the item that the calling program
needs.

This program has always been compiled with IBM OSVS COBOL.

Recently our amount of data grew to the point that the table size exceeded
the maximum size allowable under OSVS COBOL.  We recompiled the program
under COBOL II.  The program works but we now have a severe performance
problem.

Under OSVS COBOL the program would read the long file and build the table
the first time it was called; but on subsequent calls during the same
execution it would only have to consult the table it had already built.  The
contents of the table would remain available from one call to the next.
This is a difference of reading 10,000 records from a disk once per run and
reading 10,000 records several thousand times!

We know that we can probably set compiler parameters in COBOL II to make the
program behave as it did under OSVS COBOL but nobody here can figure out
just how to set them!

We don't have any of the newer COBOL compilers available.  We are working on
an IBM 3090 with MVS/ESA.
```

NORENT is the option you need.  What is happening is that upon each entry into the
COBOL II program, you are starting the program in an initial state. The OS/VS
COBOL's Working Storage is maintained between calls.  Checking for a first time
flag maintained in the calling program will fail, as your table does not contain
valid data.

To paraphrase IBM:

COBOL programs that are compiled RENT behave differently
under the VS COBOL II run time.

For VS COBOL II and IBM COBOL programs that are run under Language Environment
Release 5 or later
 the same copy of WORKING-STORAGE is used for each call. And, the program
 is entered in its last-used state, unless there is an intervening CANCEL.

Under VS COBOL II, different copies of WORKING-STORAGE is used for each...

There's really two questions here.  If you want to retain the contents
of working-storage in the COBOL subprogram across CALL statements, then
it looks like you need NORENT (non-re-entrant).  The RES/NORES option
just determines whether the program requires a runtime library (RES) or
will have the COBOL runtime routines statically linked into it.  But you
no longer have the choice of RES/NORES with COBOL for MVS and VM, you
have to use the runtime library.

Generally RES,NORENT is fine for batch COBOL.  We had a bad experience
with an OS/VS COBOL subprogram that was converted to COBOL II.  As a
COBOL II program, we compiled it with the RES option, so it required
access to the COBOL II runtime library.  But the old OS/VS COBOL program
was compiled NORES, and it was called dynamically by an assembler main
program in another department.  The first time that program called the
COBOL II version, it abended because it had no COBOL runtime
environment.

I have my favorite compile time options, so here they are (some will
disagree with my choices):

X(FULL)    The Cross-reference will include non-referenced items
OFFSET     Offset list, so you can relate an offset addres to a COBOL
line number
TRUNC(OPT) For my money, the closest approximation to the OS/VS
"NOTRUNC" option
NUMPROC(NOPFD)  Maximum sign fixup, slightly inefficient, but prevents
more S0C7 abends            than NUMPROC(MIG) does.
RES	   COBOL library routines are invoked at runtime
NORENT     Generate non-re-entrant code

For CICS programming, there are some differences:
LIST       Generate a full disassembly, rather than an OFFSET list
RENT	   Generate re-entrant code (required for CICS)
AMODE(31) AND RMODE(ANY) for virtual storage constraint relief.

There's been some debate here in the past on TRUNC(OPT) and
NUMPROC(NOPFD) but I use them for both batch and CICS without any
problems.

Folks,  several rather critical corrections to this thread  (unless stated
otherwise, this is for VS COBOL II  - LE and COBOL for this-and-that have
slightly different rules and regulations)

1) MOST CRITICAL
   "last used state" has NOTHING to do with RES, RENT, or related options.
It has to do with "Is this program a main program?  Has it been canceled?
Does it have the 'IS INITIAL' attribute'"  - All of these have to do with
application (not just individual source code) issues.

2) RENT requires RES; there is no corresponding "requirement" for NORES if
you are NORENT (i.e. NORENT, RES, is perfectly valid).

   NOTE WELL: RENT was most frequently used with VS COBOL II (non-CICS) in
order to get your application above the 16M line.  If (as indicated in the
original question), you are increasing the size of a COBOL table, check out
whether or not you have enough "private" space below the 16M line - if you
want to use NORES and/or NORENT.

3) If you are NORES, not only are you due for serious problems going to LE,
you also may NOT use any dynamic CALLs (either via the DYNAM compiler option
or by the CALL identifier syntax).

4) In order to get a COBOL subprogram to "maintain" data (in Working-Storage
Section) from invocation to invocation (under a non-COBOL driver) you MUST
use one of the techniques that identify the COBOL program as a subprogram
rather than a main program. - e.g.
     ILBOSTP0 (called by driver)
     IGZERRE  (called by driver)
     COBOL stub (calling the non-COBOL driver)

5) If you have a FORTRAN driver (rather than Assembler, PL/I, etc), please
see the topic
  "Calls between VS COBOL II and FORTRAN" (viewable on the web at

http://ppdbooks.pok.ibm.com/cgi-bin/bookmgr/bookmgr.cmd/BOOKS/IGYA1101/3.9.2
..3
before trying to handle this

6)  IBM has *NEVER* even hinted that you can recompile an OS/VS COBOL
program with VS COBOL II and "blindly" expect it to work the same.  There
are lots AND LOTS of issues involved with this.  IBM has migration guides
(and there are non-IBM books - like mine) dealing with this.  Please do not
try this and expect a "simple" process.

7) NORES and CMPR2 are compiler options that do solve "some" minor migration
options - but NORES has already disappeared in the current strategic COBOL
products (COBOL for this-and-that under LE) and CMPR2 is not strategic (and
is dropped from VisualAge for COBOL).  Please do NOT try and use these as
anything but very temporary "circumvention's" (not fixes) to individual
problems.  Allocate the time/resources as soon as possible to FIX the
underlying problem - if/when you use one of these.

8) Other compiler options to look at include (when doing migrations)
    NUMPROC(MIG)
    TRUNC(???)  - nothing works exactly like OS/VS COBOL (watch out for
performance ramifications too)
    WSCLEAR (VS COBOL II) and STORAGE (LE) - run-time, not compile-time
options
    OPT  - (you can lose literals from working-storage)

9) Static link-edits (as opposed to NCAL or dynamic COBOL calls) *can*
impact the retention of data across CALLs - but they still don't make a
subprogram into a main program.

   ***

Good luck!

  I still think the best (possible) circumvention for the originally
reported problem, is to create a VS COBOL II stub to call the FORTRAN
driver.  This should avoid any source code changes and SHOULD solve the
"last used state" problem.  However, I have never tried this with a FORTRAN
driver, so some problems may still remain.
