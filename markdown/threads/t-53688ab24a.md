According to a table in Appendix B of the Fujitsu 5 "Cobol Language
Reference - Volume 2" the maximum size of an indexed file is
1,073,741,824 bytes. The maximum record size is 32,760 bytes.

There is not an express limit on the 'number' of records. As long as
your file is composed of records less than 32,761 bytes and the total
data size of all records is less than 1,073,741,825 bytes you should
be ok.

I think there is an optional DLL that can be copied from the
installation CD that increases the file capacity to nearly 4 GB.
The maximum record size remains the same, though.

HTH,

That won't work on an indexed file because the file size is the sum of
(the number of records times the record length) plus all the internal
positioning information the ISAM file system needs. In some cases, it
is possible for the latter to exceed the former. After a period of
use, most ISAM files contain either significant wasted space or
significant key-chasing. I don't think anyone has yet devised an ISAM
file system whose efficiency couldn't benefit from a periodic
reorganization.

VSAM Key Sequenced Datasets (which replaced ISAM to a large extent in the
IBM Mainframe environment ) were developed in response to the need for a
self reorganizing Indexed file structure.

They use binary fission to acquire space at two separate logical levels:
within a "block" (control interval) and within a sub-file space (control
area). Free space is administered by a series of chained pointers at both
levels. "Overflow" (in the sense understood when referring to ISAM files)
can NEVER occur (as long as disk space is available, obviously), and the
datasets NEVER need to be off-loaded and re-loaded or re-organized.
(Although sometimes this used to be done on small systems to  reclaim unused
free space).

It is an ingenious and highly efficient file organization.
