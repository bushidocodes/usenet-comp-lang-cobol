Well, much of this is correct. You can decipher COMP-3 if you
remember to keep numbers in 2 4-bit segments, especially if working on
a mainframe. For example, 0 through 9 is F0 thrun F9 unsigned, C0
thru C9 signed positive, D0 thru D9 signed negative. A display field
of 3 digits would be F1F2F3 for 123, a signed display field would be
F1F2C3 for +123 (would appear as 12C because C3 also represents the
letter C). In a packed field, the leftmost 4 bits are stripped for all
except the rightmost byte. So, 123 would be 0001 0010 0011 1100 (the
data component and the sign component are flipped for packed data.)
This would appear in hex as 123C (two bytes). to fully grasp, you
need more than a few posts from us folks with good intent.... :-)

The way COMP-3 numbers are usually stored is implementation dependent. If we
are talking about IBM Mainframe COBOL and EBCDIC, with picture S9(4) COMP-3
it will look like this:

+1234 in hexadecimal 01 23 4C
-1234 in hexadecimal 01 23 4D
1234 in hexadecimal 01 23 4F (unsigned positive, PIC 9(4) COMP-3)

Note that every digit and every sign is stored in a nybble (4 bits). Note
also that IBM COMP-3 is a variation of BCD, and will be more efficient with
an odd number of numeric digits. In this example, whether you use PIC S9(4)
or PIC S9(5) the number is stored in a three-byte field. So coding an even
number of digits wastes one nybble in the COMP-3 format.

If you were talking about DISPLAY usage, it's a little different:

+1234 in hexadecimal F1 F2 F3 C4 ('123D' in character mode)
-1234 in hexadecimal F1 F2 F3 D4 ('123M' in character mode)
1234 in hexadecimal F1 F2 F3 F4 ('1234' in character mode, unsigned)

Again, the sign is stored in a nybble (4 bits). Whether the picture is
signed or not, the number is stored in four bytes, and the sign takes up no
additional space because it is coded in the high-order nybble
of the low-order byte.

In IBM assembler the high-order nybble is called the ZONE and the
low-order nybble is called the DIGIT or DECIMAL. So IBM display
usage numbers are called ZONED DECIMAL.

SEPARATE sign, LEADING or TRAILING may be available, but I've never seen it
used in the IBM mainframe world. Those options might be useful for unloading
the data for conversion to another platform.

3.3.1 Sign Representation

There are six valid sign representations for external decimal (USAGE
DISPLAY without the SIGN IS SEPARATE clause) and internal decimal (USAGE
PACKED-DECIMAL) COBOL data. These are:

Positive: X'C', X'A', X'E', and X'F'

Negative: X'D' and X'B'

The VS COBOL II compiler option NUMPROC affects sign processing. When you
use NUMPROC(PFD), the compiler assumes that the sign in your program data
is one of three "preferred signs." The preferred signs in COBOL are:

Signed positive or 0: X'C'

Signed negative: X'D'

Unsigned or alphanumeric: X'F'

Based on this assumption, the compiler uses whatever sign it is given to
process data, even if the sign is incompatible with the data definition.
The preferred sign is generated only where necessary (such as when signed
data is moved to an unsigned or alphanumeric data item). Using the
NUMPROC(PFD) option can save a great deal of processing time, but you must
be sure you use preferred signs with your data for correct processing.

When the NUMPROC(NOPFD) compiler option is in effect, the compiler accepts
any valid sign configuration. When processing is done with DISPLAY or
PACKED-DECIMAL data, the sign in the sending item is converted to the
preferred sign before the operation is performed. The preferred sign is
also generated in the receiver. NUMPROC(NOPFD) is less efficient than
NUMPROC(PFD), but it should be used whenever data that does not use
preferred signs may exist.

Note: In the case where an unsigned, external decimal sender is moved to
an alphanumeric receiver, the sign is unchanged (even with
NUMPROC(NOPFD)).

When NUMPROC(MIG) is in effect, the compiler generates code that is
similar to that produced by OS/VS COBOL. This option can be especially
useful as a tool when migrating OS/VS COBOL programs to VS COBOL II.

NUMPROC has no effect on binary data or floating-point data.
3.3.2 Incompatible Data and Numeric Class Test

The compiler assumes that the values you supply for a data item are valid
for the item's PICTURE and USAGE clause and assigns the value you specify
without checking for validity. When an item is given a value that is not
compatible with its data description, then references to that item in the
Procedure Division will be undefined and your results will be
unpredictable. (For details on defining numeric data items in your
program, see Figure 18 in topic 3.3.3 and the USAGE clause description in
VS COBOL II Application Programming Language Reference.)

Frequently, values are passed into your program and assigned to items that
have incompatible data descriptions for those values. For example,
nonnumeric data may be moved or passed into a field in your program that
is defined as a numeric item. Or, perhaps a signed number is passed into
a field in your program that is defined as an unsigned number. In either
case, these fields contain invalid data. It is your responsibility to
ensure that the contents of a data item conform to its PICTURE and USAGE
clauses before using the data item in any further processing steps.

You can use the numeric class test (IF data-item IS NUMERIC...) to
perform this data validation. For example:

LINKAGE SECTION.
01 COUNT-X PIC 999.

The numeric class test checks the contents of a data item against a set of
values that are valid for the particular PICTURE and USAGE of the data
item. For example, a packed decimal item would be checked for hexadecimal
values X'0' through X'9' in the digit positions and for a valid sign in
the sign position, as described under "Sign Representation" in
topic 3.3.1.

Execution of the numeric class test is affected by the NUMPROC compiler
option and the NUMCLS installation option. Figure 16 and Figure 17 show
how the combination of the various settings of these two options affect
the sign representations recognized by numeric class testing. Only
external decimal, external floating-point, and packed decimal items are
checked for sign, because binary and internal floating-point items use an
"operational" sign which is always valid.

Figure 16 shows the sign representation when the installation option
NUMCLS is set to NUMCLS(PRIM). Figure 17 shows the representation when
the option is set to NUMCLS(ALT).

Note: NUMCLS is an option set at installation time. To determine the
NUMCLS setting used at your installation, consult your systems programmer.
For information on the NUMPROC compiler option, refer to its description
under "NUMPROC" in topic 5.2.3.25.

3.3.3 Data Format Conversions

When the code in your program involves the interaction of numeric data
items with different data formats, the compiler converts the items:

 Temporarily, for comparisons and arithmetic operations

 Permanently, for assignment to the receiver in a MOVE or COMPUTE
statement.

The conversion generally requires additional storage and processing time
because data is moved to an internal work area and converted before the
operation is performed. The results may have to be moved back into a work
area and converted again. A conversion is actually a move of a value from
one data item to another. The compiler performs any conversions that are
required during the execution of arithmetic and comparisons with the same
rules that are used for MOVE and COMPUTE statements. The rules for moves
are defined in VS COBOL II Application Programming Language Reference.
When possible, the compiler performs the move to preserve the numeric
value as opposed to a direct digit-for-digit move. (For specific
information on truncation and predicting the loss of significant digits,
refer to Appendix B, "Intermediate Results" in topic APPENDIX1.2.)

Conversions between fixed-point data formats (external decimal, packed
decimal, and binary) are completed without loss of precision.

The more significant conversions, where a loss of precision is possible,
are those between fixed-point data formats and floating-point data formats
(short floating-point, long floating-point, and external floating-point).
For details on the conversions that occur during arithmetic evaluations
which have a mixture of both fixed-point and floating-point operands, see
"Floating-Point (COMP-1, COMP-2, and DISPLAY) Items" in topic 3.3.4.3.
