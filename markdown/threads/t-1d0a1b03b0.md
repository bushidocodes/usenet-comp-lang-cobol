### When to use GO TO

- **From:** "10025..." <ua-author-17071258@usenetarchives.gap>
- **Date:** 1997-06-15T20:00:01+00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<5o4chj$ltb$1@mhafc.production.compuserve.com>`

```

The verb "GO TO" transfers control. Transfers of control
abound in target code. They are produced as direct
translations of GO TO statements and in countless other
circumstances, such as when an ELSE is processed by the
compiler.

When an algorithm consists only of the constructs of sequence,
iteration and selection, the required transfers of control can
always be effected in ANS85 Cobol without the use of GO TO,
except where the algorithm is a co-routine, as one would use to
resolve certain classes of what Jackson calls a "structure
clash". The same is not true of earlier ANS standards.

There is a finite set of commonplace constructs which feature in
nearly all algorithms. I have mentioned sequence, iteration and
selection. Jackson's "posit ... quit ... admit" construct is
another. Not even ANS85 can enable this structured programming
model to be applied without the use of GO TO or something far
worse in every case.

If and when an exotic algorithm of some kind is the natural
choice, a GO TO can be avoided, as Bohm and Jacopini proved
mathematically, by using a stack of booleans. If code littered
with settings and testing of such "flags" is not to high proce to
pay, by all means ban the use of GO TO at your site.
```
