Shame on you, Charlie! :-)

Trying to start a war where there is no contest.

I have spoken with Veryant and their product set is not doing what I do with 
the Migration Toolset. Even if it was, I don't think there would be a war... 
(They are very nice people at Veryant... :-))

What this brings home to me is that I need to be much clearer about what 
exactly the process is that I am advocating. I'll be addressing this with a 
new PRIMA web site as soon as I can get round to building it.

In the meantime, here are the summarised planks in my philosophical 
platform:

I believe...

1. COBOL has no long term future. It isn't about COBOL as a language, it is 
about a different paradigm, for which COBOL is not as well suited as other 
languages which were developed specifically for that paradigm. (object 
oriented programming).

2. The future is in objects and layers. Platforms like .NET and Mono support 
this extremely well so migration to one of these platform environments is a 
desirable goal for development. (Languages like C# and VB.NET are ideally 
suited to working in this environment and, for myself, I chose C#, but I 
have at least one client who uses VB.NET. Objects generated by the Toolset 
work fine with either, as well as with Legacy COBOL.)

3. Separation of systems into layers (tiers) is highly desirable.  Data 
access should be separate from business logic, and so should presentation 
and input. (Business code should be isolated through an interface to the 
data it needs. Objects are ideal for this.)

So, at least for me, any solution that seeks to keep COBOL in place is not 
going to work long term. Middleware solutions that trap existing COBOL calls 
and translate them to Database or Screen calls simply lock you in to using 
that middleware, so that you can keep using COBOL. It doesn't move you 
forward. Migration away from COBOL is a case of "when" not  "if"... sooner 
or later the cuddle blanket has to be discarded and the new world embraced. 
I advocate sooner because the longer you leave it, the more it will cost.

The only regret I have about moving off COBOL as my standard development 
language, is that I didn't do it sooner. It took quite some time for me to 
realise that the stuff I was used to just wasn't good in the new world. Once 
I grasped that and tried out new stuff, it became easier.

For me, Migration must achieve the following:

1. It must not lock me in to any middleware solution. (I have reviewed 
several migration systems that do this and some of them are very good. But 
it means continuing wth COBOL and it means you are dependent on the 
middleware layer.)
2. It must not cost an arm and a leg. (Even a finger is too much :-)) My 
solution costs less than a new .NET compiler.
3. It should not throw the baby out with the bath water. I want to be able 
to retain existing investment if I need to. But I want it in a form where I 
can use it in the New environment.
4. It should enable me to "open up" my data resource to other packages and 
tools. (Usually, this means converting indexed files into Relational 
Database table sets, and the Migration Toolset does this automatically, 
achieving at least Second Normal Form (2NF) and, very often, 3NF. At the 
same time, the Toolset generates Data Access Layer objects to manage the new 
table set. These can be invoked from standard or OO COBOL, or ANY OTHER 
language that supports the Component Object Model (so new development, in 
the new language, can use these objects just like the current COBOL 
codebase...). DAL objects can be invoked from Web pages or desktop code so 
they provide a universal Data Access Layer, that can be used by Legacy AND 
New development.

As far as I know, the PRIMA Toolset is the only Migration system that offers 
such a layer, generated 100% automatically and instantly. The Toolset also 
offers a tool (Transformation) that changes all your existing application 
codebase to use the new DAL objects, instead of the old indexed files. 
Instant, automatic, and error free.

The goal is to ease the transition to a new environment and new development 
that offers much more powerful facilities, it is NOT to retain COBOL as a 
development language by bolting stuff on to it or recompiling it for .NET.

This is where the PRIMA approach differs from many other approaches which 
are designed to let you think that you can keep using COBOL indefinitely.

(Obviously, for COBOL people this looks like an attractive proposition, but 
it is flawed long term. Any kind of migration has to be AWAY from dependency 
on COBOL. Why? Because COBOL is no longer the best tool for the job of 
handling networks and the Web, or for getting the most from platforms like 
.NET. You CAN use it, just as you CAN ride a horse to get somewhere, or you 
CAN drill holes with a brace and bit, or you CAN start a fire by rubbing 
sticks together...)

Summarising:

The PRIMA Toolset offers tools that:

1. Generate a normalized RDB from your existing COBOL definitions.
2.  Generate modules to load the new database with your existing indexed 
file data.
3.  Generate a Data Access Layer comprised of objects that manage the new 
RDB.
4.  Transform your existing codebase to use the DAL objects from 3 above.

(There are some ancillary tools as well, like a DECLGEN facility that will 
generate Host variables for COBOL to interface to the RDB directly, or will 
generate a COBOL record layout from an existing DB table.)

But all of this is just a stage in the overall migration to a new tech 
environment.

We are currently working on refactoring COBOL into objects that can be used 
in the new environment. So far, this is a largely manual process that needs 
COBOL programmers to do it, but gradually paths and procedures are emerging 
which I believe will enable at least some of it to be automated. Again this 
is about getting away from dependency on COBOL by isolating functionality in 
a form that allows it to be used alongside new development. (Once it is 
objects, your legacy code can be utilised by C# or VB.NET or whatever...)

The whole process is graphically described on: 
http://primacomputing.co.nz/cobol21/mig.aspx

Having read and absorbed the above outline can you see why it is no contest 
between Veryant and PRIMA? Their products are Java based (which provides 
good platform independence) and I believe the quality is excellent, BUT, 
they are still committed to keeping COBOL and I am not.

I don't think they are "wrong" any more than Alchemy or Micro Focus are 
"wrong" for offering solutions that perpetuate COBOL by recompiling it for 
.NET. Many people will be very happy to have these solutions, and they  can 
get their solution from established companies with resources I can only 
dream of.

PRIMA consists largely of myself with occasional help from others, and some 
local programming support on a part time basis. And yet, we have enough 
credibility for people to take us seriously. Clients say very glowing things 
about the Toolset and there is no shortage of references available.

The PRIMA endpoint is non-dependency on COBOL (or any other specific 
language, really - objects can be written in whatever is appropriate and the 
soruce code is generaly not important...), a platform based (.NET, although 
our approach could theoretically work for Mono as well) environment, with 
applications, packages, and scripting all using objects and layers.

I would be the first to admit that this will not be attractive to people who 
only know COBOL and resist change. However, I am getting increasing 
enquiries from people who are interested in our approach and recognise that 
the time has come to move off COBOL as a primary development platform, yet 
retain existing investment where it makes sense to do so.

I realise this is not the place to promote products and services and I do 
try to avoid doing so, but on this occasion you kind of invited it :-)

In future, I'll confine this type of comment to the PRIMA web site (as soon 
as it is built... :-)) and refer people to that.

Pete.

Obviously, there are a number of companies that offer migration solutions, 
and so there is a choice for people there as well. I am keenly aware that 
people undertaking migration DO have a choice and for that reason I have 
tried to ensure we have a solid and achievable strategy, not just for 
migration, but beyond it.

These days, it isn't about just offering a product or service, it is about 
full support and commitment to seeing the success of clients using your 
products and services.

I know Veryant shares this philosophy, and that was what prompted my comment 
to Charlie, that even if we WERE in the same market, it would not be 
hostile. Business is business, sure, but, as least as far as I'm concerned, 
any company that treats their staff and clients ethically, is likely to 
compete in the marketplace using the same ethos, and I have absolutely no 
problem with that :-)

We've seen quite a bit here recently about the dreadful practices some 
companies employ against their own staff; it makes me wonder how they treat 
their clients and competitors... I can't imagine such companies lasting very 
long and I think the only reason some do is because they have an entrenched 
position in a huge marketplace. (Even then, that is no excuse for some of 
what goes on...).

I believe that competition doesn't have to be a war. (If you compete in a 
running race, for example, do you need to feel hatred for the other 
competitors, before you can be successful? Interviews with Olympic athletes 
show they have respect and, often,  regard, for others in their event)

I'm more than happy to make a case to anybody for using our products and 
services, and I respect their right to look at similar cases from competing 
companies. If we lose the contest, I would be disappointed, but if they go 
with a company that will see them right, then that's a good result. It is 
when you get unscrupulous companies entering a market place, who are solely 
interested in ripping it off, and who have no empathy or concern for their 
clients, that things can go custard. My own take on that would be to blow 
the whistle on what they are doing and why itis wrong. (But I'd do that
