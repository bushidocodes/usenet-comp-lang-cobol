Well, I programmed in both on a Wang VS computer.
Natural programs should be short. I tried to write a natural program
a la Cobol, and my teacher almost killed me. I commented to much (!).
Better skip documentation in the program itself. There is a seperate
library for that stuff. Programs should be short in Natural. You better
write several short subroutines which can be called.
Depending on the system you're working, there is no debugging facility.
You have to re-process the whole database if one variable get's changed.
Although natural is platform independent (that's what Software AG claims),
you probably run into troubl when applications are moved from one
platform to another, because version differences. Anyway, we encountered
that problem when applications were shipped from an IBM system to
VAX systems, or to our Wang system. Pretty bad.
It has a nice screenpainter though, and when using Predict, you're forced
to think about your design upfront before writing the applications.
Screens are split from code, and verification rules can be placed in
the databse, no need to put that in the code.
Natural is a 4 GL (one of the oldest) and nowadays code created on
a mainframe can be run on a pc windows environment without changing the code.

However, I still prefer COBOL. I think COBOL is more flexible. In COBOL
I only have to open 1 program, and everything is there and at hand.
No need to retrieve code from multiple resources/dictionaties.
Our COBOL environment has a pretty good debugger to say the least.
In a sense COBOL is platform independent too, but alas, most computer
companies have "extended" this language with their own "extensions"...
If you can choose, choose COBOL.
If you're stuck to Adabas/Natural, there are probably worse languages
out there, although I don't know which :D

Bo.
