 - Oracle

with, this is actually POSSIBLE for any (well almost any?) COBOL compiler on
any environment with any database - but there are NONE that are totally
portable to all environments and all compilers and all databases.

jean-michel -

I believe the confusion you keep running into is partly due to your mis-use of
the term 'native-cobol.'

There are many products out there that require some sort of pre-compiler to
translate their code into "native cobol."  The ones I am most familiar with
are CICS, DB2, and IMS.  I strongly suspect that other database products,
such as IDMS, work the same way.  That is their non-cobol code -- be it a SQL
statement, a CICS command, whatever -- are enclosed in a "EXEC product ....
END- EXEC" delimiter.  The precompiler scans the source code for "EXEC"
statements that it recognizes, parses it, comments it out, and replaces it
with "native cobol."  These are invariably MOVE and CALL statements because,
ultimately, the only way for a cobol pgm to access the services of these
subsystems is by CALLing a routine that is supplied by the sub-system.	MOVE
and CALL are as "native-cobol" as you can get.

Now, as I understand it, what you are looking for is something to recognize
the EXEC SQL ... END-EXEC statements and, instead of replacing them with
calls to the realational database, replace them with standard cobol file
access methods.

Frankly, I don't think it can be done.	There are too many things in SQL that
simply don't map to standard cobol file access.  More certainly, there are
requirements of standard cobol file access that don't map to SQL.  Why would
you want to?  You mentioned "portability" but cobol is already the original
"portable" language.  I have taken several cobol programs across platforms,
and have never had to make any changes beyond the environment section.

Ed Stevens
