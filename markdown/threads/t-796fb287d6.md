Michael, I'm surprised at you!  *Every* bug is caused by programmers.
The question is: "Which techniques can be helpful decreasing programmer
errors, and which ones are harmful and increase programmer errors?"  Do
you use GO TO?  Do you use ALTER?  Why not?  Because those constructs
have been demonstrated beyond question to increase programmer errors and
make programs using them more difficult to safely modify and debug.  There
are two reasons why those constructs are harmful: they are "open ended"
and their function precludes much compiler support in catching errors.  On
the other hand, a procedure CALL (PERFORM) is *not* open ended, and the
compiler can easily generate code to detect errant subscripts.  But pointers
are the most "open ended" things there are, and generating code to test for
errant pointers is almost never practical.  Even if it were, that would more
than negate any performance gained by using them.

> Pointer variables are IMNSHO an EXCELLENT way to enhance performance.

Driving to work in a jet powered dragster gives you more performance too,
but such vehicles don't belong on the street.  Same with pointers, particularly
procedure pointers.  This is why there are so many debugging tools for C, a
language *requiring* pointers for any non-trivial task.

The pointers themselves are cleaned up, or is it the other way
around?

It's my understanding that OO-COBOL will have garbage
collection.  When the last reference to an object goes out of
scope (or is explicitly cleared,) the object becomes eligible
for collection.  In this scenario, you don't have to worry
about dangling object references, as you would in a scheme that
employes explicit creation _and_ destruction of objects.

Certainly the memory for those references would ultimately be
de-allocated by the end of the run unit, but hopefully there's
no need for logic that hunts down and clears (nils, nulls, ...)
the actual pointers.

I can't see a condition that would require you to force a
finalization outside of the reference counting (or trace
mapping) that's normally used to expire an object, but there
are those that try.  Even so, having the runtime clip dangling
references would appear to be a logicstics nightmare.

___________________

This area of system design is interesting from a developmental
point of view, and also promises to play a key part in the
debates over secure computing.  The open-ended pointers that
Judson refers to are even closer to the heart of it.  Even if
they aren't strictly bad form (also suggested by Judson,) they
are powerful tools in scrutinizing a system.  As such, they
present a danger to secure computing.

references left, the object may (and will) be destroyed.  In
your example, an object is created, and a referenced is passed

the calling program.  When your sub-program terminates, any
references you've made will be released, so the overall number
of references will be the same as it was before the call to the
sub-program.

Defects in the implementation of the gc can be a problem, but
the goal is to destroy objects when there are no longer any

no longer accessible because the run has ended.

There has been some significant work done in the standard to prevent
this "invalid pointer" situation for object references.  I'm sure
there's some way to mess it up, but the aim was to prevent it from
happening.  Object references can have "types" (bad term) that allow
conformance checking.  You can specify that an Object Reference data
item can only reference an object of a certain class or interface - or
you can make it a "Universal" Object reference allowing it to contain
any type of Object reference.  Doing so requires more runtime
conformance checking and might be slower depending on the
implementation.  I think this is what is meant by "Open Ended" object
references.
