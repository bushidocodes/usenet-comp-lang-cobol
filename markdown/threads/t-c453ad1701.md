You cannot access the DLL in Fujitsu NetCOBOL OR PowerCOBOL, "using the code
below".

There are too many Micro Focus specific lines of code and directives.

You CAN access the .DLL (or any other .DLL that is not OO/COM/OCX) using the
Fujitsu mechanism provided.

This involves making the called .DLL known in the COBOL85.CBR environment
file and specifying the entry points you want. It is pretty straightforward
and doesn't require you to call into the WINAPI. That is done under the
covers.

Although you are using PowerCOBOL, you need to read chapter 10 of the
NetCOBOL Users Guide which explains everything you want to know, and
decribes the various Fujitsu linkage models.

Calling C++ or C or VB .dlls from Fujitsu is not complex and it doesn't need
the low-level approach you have used with Micro Focus.

PowerCOBOL is an OO environment and it is VERY easy to load and use COM/OCX
components from third parties or written by yourself., but to use non-OO
.DLLs, you need to use the standard mechanisms provided for NetCOBOL.

You probably COULD do this using the WINAPI but it is really tiresome to
have to work at that level when a much higher, easier, level is provided.

Set up a simple COBOL85.CBR file and make the .DLL and entry points known to
it.

zero in return-value and the WREF string unchanged. If so, then you should

CALL CALLWD
USING WREF
RETURNING return-value

I am very impressed you are doing this conversion.

Unfortunately, as you found, there are some bugs in Robert's original code
which I have only recently discovered. It simply doesn't work correctly if
you have multi-level COBOL tables defined.

Robert seemed to be under the illusion that nested tables are stored as
blocks contiguously when, in fact they are not.

Example (for simplicity, and to improve the clarity I have not coded the
INDEXED BY clause I always use with OCCURS...):

...
12 COBOL-NESTED-TABLE. *> 3 DIMENSIONAL TABLE.
15 LEVEL-J OCCURS 2.
18 AFLD-J PIC XXX.
18 LEVEL-K OCCURS 2.
21 AFLD-K PIC XXX.
21 LEVEL-L OCCURS 2.
24 AFLD-L1 PIC XXX.
24 AFLD-L2 PIC XXX.

Robert's COBSTRUCT code behaves as if this data was stored thus:

AFLD-J;AFLD-J;AFLD-K;AFLD-K;AFLD-L1;AFLD-L2;AFLD-L1;AFLD-L2

...when it is really stored thus:

AFLD-J;AFLD-K;AFLD-L1;AFLD-L2;AFLD-L1;AFLD-L2;AFLD-K;AFLD-L1;AFLD-L2;AFLD-L1;AFLD-L2;AFLD-J;AFLD-K;AFLD-L1;AFLD-L2;AFLD-L1;AFLD-L2;AFLD-K;AFLD-L1;AFLD-L2;AFLD-L1;AFLD-L2

› 004995 move word-text (1) (function STORED-CHAR-LENGTH

STORED-CHAR-LENGTH is a very useful Fujitsu extension to the COBOL
intrinsics. It returns the number of characters actually stored in a string.
So, if you have a string defined as: PIC X(30) and move "this is 21
characters" to it, it will return the number 21. (The intrinsic LENGTH
function would return 30). (STORED-CHAR-LENGTH is much more efficient than
using INSPECT... TALLYING LEADING SPACES on the reversed string, although
you could do that to remove the dependency on Fujitsu.)

The code above is pulling out a COBOL level number (2 characters) from the
first word in an array of words. Without looking at it I suspect that
"word-text-alpha" REDEFINES or is a subfield of "word-text".

Here's a clearer interpretation of the code (ignore that word-text is part
of an array as we are only looking at one entry in it):

MOVE WORD-TEXT (length of word-text stored characters - 1: 2) TO
temp-level.

(It is a pretty standard use of refmodding when you break it down...)
