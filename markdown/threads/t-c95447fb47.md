A subscript is just a method of referencing something in a list.  For
example, instead of having a single name referenced by a variable name, I
declare the name as occurring a certain number of times. I then refer to it
by a  number in a subscript.  So NAME-VALUE(1) would refer to the first
value in the list, NAME-VALUE(2) would refer to the second value in the
list, and so on.

Data items are referenced by using their data name, such as STREET-ADDRESS,
CUSTOMER-NAME, etc.  You can think of a table as a mechanism for having a
number of data items, all with the same name.  This collection of data items,
all with the same name, is called a 'table'.  You can create a table in COBOL
using the OCCURS clause of a data declaration.

    03  COUNT    PIC  9(06)  OCCURS 20 TIMES.

This creates a table of 20 data items, all named COUNT.  In COBOL 85 the first
element in a table is always 1, and the last element is defined in the OCCURS.
Sometimes that maximum number is determined at runtime by the contents of
another numeric data item specified in the OCCURS.  For example:

    03  MAX      PIC  9(02)  VALUE 15.
    03  COUNT    PIC  9(06)  OCCURS 1 TO 20 TIMES DEPENDING ON MAX.

In this example, as the value of MAX is changed in the program, it will
be as if the OCCURS clause was simultaneously changed to the same value.

Because each data item has the same data name, each time you reference the
table, you must somehow indicate which of the several data items you are
referring to.  To do this we use what is called a 'subscript', as in
mathematical notation.  Since most programming languages use straight text,
and do not support the tiny fonts used in printed subscripts, the subscript
is typically enclosed in parenthesis () following the data name.

    COUNT(1)     Data name COUNT, #1
    COUNT(2)     Data name COUNT, #2
    COUNT(12)    Data name COUNT, #12

In some programming languages (e.g. C/C++), subscripts are enclosed in square
brackets [] instead of parenthesis (), but they do the same thing.

Tables and subscripting, along with loops, are some of the most powerful
features of programming languages.  They allow you to write one set of
instructions which can be applied over and over to different data.  This is
accomplished by using a numeric data item instead of a numeric literal as a
subscript into the table, like this:

    COUNT(SUB)   Data name COUNT, # contained in SUB

Obviously, SUB must be a numeric data item, and the value must be an integer
within the valid subscript range.  To add 5 to each data item COUNT in the
table, you could code a simple loop like this:

    PERFORM VARYING SUB FROM 1 BY 1 UNTIL SUB > 5
        ADD 1 TO COUNT(SUB).

The data item in the table may be any normal COBOL data item, even a group
item.  For example:

    03  FRIEND-TABLE        OCCURS 100 TIMES.
        05  FRIEND-NAME         PIC  X(20).
        05  FRIEND-STREET       PIC  X(20).
        05  FRIEND-CITY-ST      PIC  X(20).
        05  FRIEND-ZIP          PIC  9(05).

The OCCURS clause may not be used on a level 01, 66, 77 or 88 item, but only
on levels 02 through 49.  You may not REDEFINE an item containing an OCCURS
clause, but you may REDEFINE an item subordinate to an item containing an
OCCURS clause.

Tables with one subscript are very useful, but sometimes there are situations
when we need more.  For example, say you have divided your city into a 10x20
grid, and you want to store the population of each grid square into a table.
You could do this:

    03  CITY-GRID-X      OCCURS 10 TIMES.
        05  CITY-GRID-Y      OCCURS 20 TIMES.
            07  CITY-POP         PIC  9(06).

This creates a table with 200 data items named CITY-POP, but now when you
reference it, you must specify two subscripts, one for each dimension:

    CITY-POP(1, 1)              CITY-POP grid at X=1, Y=1
    CITY-POP(3, 5)              CITY-POP grid at X=3, Y=5
    CITY-POP(10, 20)            CITY-POP grid at X=10, Y=20
    CITY-POP(SUBX, SUBY)        CITY-POP grid at SUBX, SUBY

If you wanted to store the population of each grid square for ten cities
you could do this:

    03  CITY-CITY        OCCURS 10 TIMES.
        05  CITY-GRID-X      OCCURS 10 TIMES.
            07  CITY-GRID-Y      OCCURS 20 TIMES.
                09  CITY-POP         PIC  9(06).

This creates a table with 2000 data items named CITY-POP, but when you
reference it, you must specify three subscripts, one for the city and one
for each dimension.  To sum the population of all the cities, you could
code this:

    03  TOTAL-POP        PIC  9(08).
    03  CITY             PIC  9(02).
    03  SUBX             PIC  9(02).
    03  SUBY             PIC  9(02).

    MOVE ZERO TO TOTAL-POP.
    PERFORM VARYING CITY FROM 1 BY 1 UNTIL CITY > 10
        AFTER VARYING SUBX FROM 1 BY 1 UNTIL SUBX > 10
            AFTER VARYING SUBY FROM 1 BY 1 UNTIL SUBY > 20
                ADD CITY-POP(CITY, SUBX, SUBY) TO TOTAL-POP.

As a matter of form, I prefer to indent my PIC and other clauses four more
spaces with each level of OCCURS.  This provides a clear visual cue when
you are scanning a record definition.

In addition to subscripting, COBOL provides a feature called 'indexing'.
Indexing works very similarly to subscripting, but there are differences.
First, an 'index' is defined as part of the table with which it is
associated, by using the INDEXED BY phrase of the OCCURS clause.  The city
table above could have been defined like this:

    03  TEMP             PIC  9(02).
    03  CITY-CITY        OCCURS 10 TIMES INDEXED BY CITY.
        05  CITY-GRID-X      OCCURS 10 TIMES INDEXED BY SUBX.
            07  CITY-GRID-Y      OCCURS 20 TIMES INDEXED BY SUBY.
                09  CITY-POP         PIC  9(06).

The above PERFORM would work on this table in the same way.  But if you want
to change the value of an index you normally must use the SET verb.  For
example:

    SET SUBX TO 4.
    SET SUBX UP BY 1.
    SET SUBX DOWN BY 2.
    SET TEMP TO SUBX.
    SET SUBY TO TEMP.
    SET SUBX TO SUBY.

Using indexes instead of subscripts can have advantages.  One is that you can
often write faster code using indexes, and the other is that you can use the
COBOL SEARCH verb on tables which have indexes declared.  But those are beyond
the scope of this brief introduction.

My guess is that you're confused between subscripts and indices.  They are
slightly different ways of doing the same thing, and look the same in many cases.

An index is set up when you create the occurs, and used to be slightly more
efficient than subscripts.  It is still required for SEARCH statements, but I
don't see it used much anymore outside of this.  It requires the SET command to
change, and cannot contain a calculated value.

A subscript is not declared around the OCCURS clause and works just as you would
expect a subscript to work.

If you think this is the source of your confusion, please reply and we will give
you a more detailed explanation, although it may be that what I said will enable
you to read the text with better understanding.

There are a few points to this that I haven't seen others mention.

One is that COBOL is supposed to be 'English like'.  "SUB=1 TO 10" is
not English like, therefore violates the original premise of COBOL.  I
can't give you specific, hard data here, but I will say that one of the
things that I have 'intuitively' learned over the years is that it is
very important to have a theme, paradigm, strategy, whatever you want
to call it, and to stay with it, unless and until it is proven to be
unsatisfactory.  It provides a coherence that is not present otherwise.
And creative, productive construction is, in essence, the production
of coherence and order out of chaos, or other, less desirable or useful
order.  There is also a feature of human nature that we are creatures
of habit.  Habits are at the core of our ability to deal with a complex
environment.  Once something becomes so familiar it is a habit, then it
can be 'internalized' by our brain, then assigned to some subconscious
part of our mind, where it can be carried out with little or no effort
at the conscious level.  We employed this process over and over as we
learned to crawl, walk, tie our shoes, etc.  Though created from the
bottom up, the result is a top down, stepwise refinement of common
tasks, to permit the enormous details to be managed at levels far below
the conscious level, where we deal with highly abstracted forms.  Because
we are constructed this way, we naturally tend to use habits in all our
activities, to a level most people hardly realize, and in things we
don't usually consider in this light.  Even when we are not aware of it,
our minds are searching and analyzing patterns, then using them.  We pick
up the body language and speech patterns of our friends, usually without
even realizing it.  We do the same thing with more abstract activities,
such as programming.  When we study a programming language, as we learn
the syntax an semantics of that language, our pattern recognition ability
automatically begins to internalize the 'theme' of the language, just as
it does in the other common tasks we do.  If the 'theme' is coherent, it
greatly facilitates this process, in the same way that learning a series
of interrelated facts is easier than learning a series of disjoint facts.
In other words, when a programming language has a clear and coherent
theme, and stays consistent to that theme, it makes mental processing of
that language quicker, easier and more reliable.

All of us have built an enormous body of patterns involving the syntax
and semantics of our native language(s).  COBOL is built upon English,
and when we learn COBOL, because the semantics are also built upon the
same base as English, we can use this body of pattern recognition that,
for native English speakers, is embedded and entrenched far deeper than
any later learned programming language can be.  The result is that we
have a built-in, habitual 'reflex' that automatically interprets and
checks syntax against semantics.  In other words, the 'style' of COBOL,
because it is based on our (English speakers) own natural language, is
reinforced by the body of English natural language pattern recognition
already learned.

I'm not sure it would be possible to quantify the above benefits, but
I am completely convinced of their validity.  I think they contribute
significantly to COBOL being one of the easier languages to learn and
master, at least by English speakers.  There are programmers who do
productive jobs as COBOL programmers, but who would never master C/C++.
I believe this to be an advantage of COBOL.

But a further benefit of COBOL being 'English like' is that it can be
largely read and understood by non programmers.  I have, on several
occasions, gone over code with accountants or other non programming
users, to make sure we were understanding each other.  I doubt if
anybody ever does this with C/C++.  (In APL you don't even do it with
another APL programmer! ;-)  Personally, I think this is a valuable
thing, and in this context,

    PERFORM SUB=1 TO 10

is certainly less readable to a non programmer than

    PERFORM VARYING SUB FROM 1 BY 1 UNTIL SUB > 10

which most English speakers would understand straight away. :-)

Not quite.  While it is true that, when we are dealing with abstractions,
the details are not necessarily involved at that level, it is also true that
they are involved at the point where they are processed, even if not
consciously.  My point is that using a deeply ingrained, long established,
pattern structure consistently leads almost certainly to greater efficiency
at that level.  And if you take poll of people who last studied math ten
or more years ago, you would be surprised how few people think in
those terms. :-)  Anyone involved with child development will tell you
that young children have a far higher capacity to learn language than
they do later, even during childhood.  Language is a tool for the
expression of ideas and concepts, and a computer language is only
different in that it is far more narrowly focused, highly formalized and
(hopefully :-) completely unambiguous.  Also, nothing academically
learned after a person reaches the age of six (actually about age four),
will ever be so internalized as to become 'instinctual' to the degree of
those things learned at a very early age.  Severe physical or emotional
trauma might approach it, but certainly nothing academic.

One common expression of what I am trying to say is called the
"Rule of Least Astonishment."  In design, generally, the principle is to
so design the human interface such that, if the user does not know
how to do a certain thing using the interface, that the interface would
work so as to 'cause the least astonishment'.  In other words, work
most closely to what the user would expect.  The reason for this rule
is that it was learned long ago, long before computers, that human
interfaces that match what is as close as to what would be normally
expected by an untrained person are easier to learn, faster to use,
and produce lower error rates.  Such is what one would expect, is
it not? :-)  Natural English (or any other natural language) is quite
different from mathematical notation.  COBOL has the feature of
following natural English as closely as a programming language
might, and still be effective and non-ambiguous.  Mixing paradigms
midstream for no apparent reason would be, by definition, a violation
of the Rule of Least Astonishment.  Actually, these principles can be
easily verified by testing, and ergonomics labs live and die by them.
Being completely unaware of the effects does not invalidate them. :-)

We're not discussing "can work" or "can't work" here, we're discussing
whether mixing paradigms simply for the sake of some brevity would
enhance or detract from a programming language. I contend that the
body of evidence and established knowledge on human nature and
mental processing shows that you pay a penalty for every violation
of the Rule of Least Astonishment.  The more violations, the greater
the cumulative effect on the mental time and energy needed to
process the rules, each reducing efficiency to some extent.  Personally,
I don't think gaining a small amount of brevity makes a good trade. :-)
