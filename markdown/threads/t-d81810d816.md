I assume this was a joke.   In reality, there are still a lot of people successfully
developing software with COBOL.  Moreover,   the current COBOL standard,
and its planned successor, are both quite easy for applications developers to
learn and use.  One of COBOL's virtues has been that, in large corporations,
people who understand the problem domain have been able to transition to
the role of programmer and create solutions.   Typically, these people have
little training in computer science.  They are able to learn on the job.

I have seen a lot of excellent COBOL programmers who have made the
leap from clerical jobs to programming jobs and who have done really
good work.   Also, COBOL is not going away anytime soon, contrary
to popular opinion.    And it is still better for business data processing
applications than C++, any day of the week.   I would have to say it is
better for most of these kinds of applications than Perl or Scheme, Smalltalk
or even Java.    Ada might be the exception because it has a built-in decimal
type and a well-defined Information Systems Annex.    However, I don't
see Ada replacing COBOL anytime soon any more than I see any of those
other languages replacing it.   COBOL continues to evolve and improve,
but few in the programming community keep up-to-date about those
changes.  So many of them wrote it off long ago that they don't know
how it has improved over what it once was.    It is much like those who
still criticize Ada based on the old Ada 83 standard or Smalltalk based
on an outdated view of that language.   Hmmmm. Maybe Ruby will do. :-)

Jimmy, I wish you were right.  However what Marie is saying for her
environment mirrors what is happening in a lot of places for good
reasons.  It is expensive to write, maintain and document a system
regardless of language so packages are replacing in-house systems in
many organizations.  People are even getting smart and minimizing
changes so that they can go to the next release.  For code on the client
side where the user interface is more important, C/C++ may be the way to
go although I think that Java may be more appropriate.  The COBOL
community hasn't come to grips with HTML and its delimited text
organization (as opposed to the fixed field that I know and
understand).  COBOL has nothing native for dealing with databases
(although I would like to see a view treated as a file and think that
there could be tremendous productivity gains from doing so in COBOL). 
Thus, they are not going to be making as many changes as when they
wrote, owned and maintained the system and if they did the job right,
the maintenance should be relatively easy regardless of language.  

The one caveat to my statements above is that someone had better make
certain that the package(s) is(are) doing the arithmetic correctly, the
rounding and the handling of intermediate results.  Binary floating
point can give different results from fixed point decimal (more
accurated in some cases but that is another question).

orders placed and still outstanding), and that in turn increases the Open-to-buy based

possibly different percentages per project).

they both have vendor files, outstanding invoices and have to print cheques. Perhaps you
can clone the two from a base system - perhaps you can't. (No doubt SAP would claim you
can).

something more specific to the task of communication.

into the browser, the HTML tag developers stopped creating new tags for

have better tools to do the job. I would have prefered that these
"Programming Domain" classes be made native to the language. That is as
discrete language elements or verbs and not as a generic interface to be
"invoked". Then the OO stuff could be used for the "problem domain"

In short, the proliferation of complex tools is not helping the issue of
