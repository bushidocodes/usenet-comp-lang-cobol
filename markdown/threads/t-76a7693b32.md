  goble@gtech wrote:

What happens when you get two people named John Smith or something like
that?

Won't you have a duplicate Key???

If the name and a member number were together that would make it unique.
We have one file that has a codex key.  A kind of 5 letter phonetic key
to search the last names that sound the same.  Even that is tied
together with an identification number we just switch the order around
for the alternate key.  The US Census uses this.

I would use one specific key.  Dont describe an alternate key unless
you are using it.  Otherwise, most compilers think you want to use the
alternate key.

I was looking at a write statement from one of our programs it reads
like this:

(first we move new data to a group name in the working storage.)
write record from data-storage-record
    invalid key        perform err-routine(your err routine)

normally if we use the first key we dont mention the alt key.  If you
do the compiler expects you to use the alternate key.  I wrote a
program a while back and I described the alternate key and it wouldn'
allow me to use the primary key.  Your working storage area hould
contain the field with the key.

How  PC record index works I dont have a clue.  I tried a program once
on a pc that used a key and it makes the record index seperate from the
record.  Maybe you have to have a statement saying what the key is when
you write or something.  I just make the key/keys part of the record
(described in working storage) and write the record from that.

Example

---------------------------------
 SELECT CLASS-FILE
   ACCESS IS INDEXED
   RECORD KEY IS CLASS-RK
   ALTERNATE RECORD KEY IS CLASS-AK
   FILE STATUS IS FILE-STATUS.
-------------------------------------
WORKING-STORAGE SECTION.
 01  CL-CLASS-FILE.
     05  CL-KEY        PIC 9(09).
*   ALT KEY
     05  CL-ALT-KEY    PIC X(18).
*   remainder of non-key record
     05  CL-NON-KEY-PART.
         10  CL-FLD-1      PIC 9(05).
         10  CL-FLD-2      PIC X(30).
---------------------------------------
     MOVE WS-SURNAME TO CL-ALT-KEY.
     WRITE CLASS-FILE FROM CL-CLASS-FILE
        INVALID KEY        (your code).
----------------------------------------

A compiler may want something in "both" the alternate and primary key
to write a record.

I don't do many programs with an add to the file, usually the online
programmers do that.  Usually just a rewrite statement to reset
switches.  The IBM book says it just writes the record on the end of
the file.  The key holds the location of the file.

The answer is also simple, David.

There has to be a logic error. (Or maybe a definition changed between the
version you are running and the version which created the file...)

The fact that the primary key has spaces in it has no bearing on it. You
have correctly defined the Alternate key to be the "Key of Reference".

This apparently exists on the file, but the system is sure that it
doesn't...

Check that the single record on the test file has the correct information in
the Alternate key field. It is not one byte out, or something similar. You
will get an INVALID KEY on READ if the record is not found. It won't be
found if the key on the file doesn't match the key in the program.

This is one reason why it is often better to use a status code where you can
actually see EXACTLY what happened rather than an INVALID KEY "catch-all"

In the snippet you posted you also define "key" and "first" fields. These
are reserved words and will cause unpredictable effects if they get past the
compiler.

If you check the above and it still doesn't work, please mail me the full
source and your test file.
