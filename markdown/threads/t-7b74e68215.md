There was a good reason for this methology, why I hear you ask?

On some of the earlier compilers going back to IBM 1401, 360/30 (ish) etc:
1.  If the file was closed for whatever reason the data within the FD would
be junk, you would not get a regular exception error (suprise, suprise).
2.  Some compilers subject to what the compiler configuration settings
where you could even get an Abend (for those that don't, it was very
fatal).
3.  Also on 360's code generated was longer both in length and time to
execute using FD's as they used a buffer system that was in high memory.
4.  WS used low memory on both (and including ICL 1500,1900,2900) so
programs where a lot more efficient and did not break so much outside of a
programmers control.

There were also other procedures to adopt to keep the code as tight as
possible including avoiding Compute verbs where at all poss, eg, stick to
the regular four, add, subtract, multiply and divide. Also Perform's across
bounderies and a ton of other stuff.

I recall creating programs that went through all of the coding
structures and looking at the assembler produced (compiler can be told to
list it immediately after the Cobol statement) to find best practice for a
given version of a compiler.

Heck, have not thought about that in years but.. still do slightly similar
with compilers for PC's but not so important these days as Ram
availability is somewhat larger than 16kw - 256kw.


Don't mind me, just an almost 65 year old remembering back to the early
1960's  :)


Vince

I found it was perfectly OK to manipulate data in IO buffers (provided the 
file was open and addressibility to said buffer was established.) Doing so 
saved storage which was very expensive at that time and we had little of it.

Perhaps I can entertain Doc as he suggested... a scenario where READ INTO 
and WRITE FROM could be less than advantageous, even harmful.

Consider a file with millions of records on it.

Every time you read one of these records you are moving it to somewhere 
else, and every time you write it you are moving it from somewhere else.

Unnecessarily. You don't HAVE to do it.

I seem to recall someone explaining that the new INSPECT statement could
