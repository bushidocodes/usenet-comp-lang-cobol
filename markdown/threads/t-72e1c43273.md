The ON verb allows the command (i.e. display, add...etc.) to be executed
based on the ON-verb's parameters. For example: ON 1 DISPLAY "XYZ",
displays XYZ the first time this statement is reached; ON 2 AND EVERY 1
DISPLAY "XYZ" displays XYZ the second time and every time thereafter
that this verb is reached. The same effect can be achieved using a
counter and an IF-statement.

I've never seen "ON 2 AND EVERY 1", but I well remember "ON 1". As far as I
can tell, the ON verb was intended to be used for testing/debugging. You
could code something like "ON 50 GO TO EOJ-ROUTINE" and test your program
with a large production file. Your program would stop on the 50th record.

In actual practice, "ON 1" was almost universally used for any first-time
switch. Since it seemed to be an IBM extension, it was not portable. That's
probably why it was dropped in COBOL II. Another argument I have heard
against using it was that it could be difficult to determine the state of
the "ON 1" first-time switch in a dump.

The ON verb is essentially a trigger action that gets executed on the Nth
occurrence of its "execution" ...

ON 1 DISPLAY 'XYZ' would display the string 'XYZ' on the system logical
output device upon its first execution and subsequently be a "NOP"
instruction

ON 2 AND EVERY 1 would do nothing the first time it was executed and
would begin operating on the second execution and each thereafter.

I sort of like this one ....

ON 1000 AND EVERY 1000 DISPLAY 'RECORD ' FILE-REC-KEY ' NOW PROCESSING'
UPON CONSOLE. this is sort of a progress counter that I've seen used.

The ON verb is a part of the DEBUG facility and should be in your manuals.

ON 1 executes the following verb the first time thru the path.

ON 2 executes the following verb the second time thru the path.

ON 2 EVERY 1 executes the following verb the second, and every subsequent
pass thru construct.

BTW, the counter used in OS/VS COBOL is a 24 bit integer, so strange
results occur after 16*1024*1024 passes.

i.e. ON 1000000 EVERY 1000000 will act strangely beyond 16,000,000
passes.

This construct was dropped in VS COBOL II and subsequent compilers (IBM
offerings). It can be easily converted to more conventional code.
