```
You'd be better off with an EVALUATE here, designed for multiple
conditions. In your example you need two END-IF's (paired with the two
IF's). I take it you took the single END-IF out but left the period
(full stop)? Lose the period & add a 2nd END-IF. I'll bet your compiler
loves that.

Bill Lynch

sample code for your example:

EVALUATE	TRUE
WHEN		CONDITION-1
	statement
WHEN		CONDITION-2
	statement
WHEN		OTHER
	yet another statement
END-EVALUATE

BTW, if you want to take the same action for multiple conditions, you
can code:

EVALUATE	TRUE
WHEN		CONDITION-3
WHEN		CONDITION-4
	statement(s)
.
.
END-EVALUATE
```

```
I would really like to see the original code with all the "bits and pieces"
just before and after it.  I would also want to see the exact error message
that the compiler produced.  Like Russell, with the limited amount of
information that we have, it seems to me that there should have been no
compiler error message (although the code would have been better with a 2nd
END-IF and better indentation - but that is "style" only).

I can think of a couple of "things" that might have caused an error message
with the original code.

1) If one of those "statements" were actually a "NEXT SENTENCE" and not a
"real" imperative statement.  (Some compilers allow it, but a "NEXT SENTENCE"
phrase is not allowed "directly" in an IF that is terminated by an END-IF -
if ANSI conforming compilers).

2) If this code were actually "nested" in another conditional statement (such
as READ AT END - before the first "IF").  In that case, the period would end
both pending "IF" statements, but the original statement (whatever came
before the 1st IF) might have been a conditional that could NOT have a
"nested" conditional - even if the IF were terminated by a period.

  ***

Bottom-line: Show us (me) the full original code and error message, and I may
be able to explain it.  However, it is always "better" to make certain that
you have the same number of "END-IF" phrases as you have original "IF"
statements.  Better yet, do look at the EVALUATE statement, which sounds to
me like the real "logic" statement that you are trying to code (in this
case).  See other posts to show how this can/should be done.
```

```
older than structured programming and still has some baggage left
over from before standard control structures were added, just like
COBOL does.  Actually, you can write very well structured code, even
in QBasic.  Modern BASIC has SELECT CASE, DO WHILE ..., DO UNTIL ...,

declarations and parameters, data typing, user defined data types,
local and global variables, and lots of other goodies.  Like COBOL,
BASIC is a good language if you use it properly, and stay away from
old non-structured features, left in for compatibility reasons.  I'm
not sure you could call VB or VBA 'BASIC' in the traditional sense,
because they have been altered so drastically.  After all, BASIC is
an acronym for Beginner's All-purpose Symbolic Instruction Code. :-)
```
