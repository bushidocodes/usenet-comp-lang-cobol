I imagine the reason for this is some text from the application is being
output back to the web server before the proper headers which should be
produced by a CGI application.

If you are using the MERANT Microfocus extentions for CGI check you are not
displaying any html code or other text before the first DISPLAY
external-form in your application  or the first EXEC HTML statement which
produces output. As it is when the first of these which when encountered
will automatically produce the appropiate headers.

A method I sometimes use to debug this kind of thing is to place a

  display "Content-type: text/plain" & x"0a" & x"0a"
or
  display "Content-type: text/html" & x"0a" & x"0a"

as the first line in my program. That way I can see everything which is
being passed to the web server and spot any problem areas. This will work if
you are doing all the CGI stuff yourself or using the features provided by
the product.

So if you use the text/plain version and you see something like

Some Text Here from my application
Content-type: text/html

<html>
<head>
...

Then it is the 'Some Text Here from my application' which is upseting the
web server and the other one ignoring it.

If you do the above and don't understand what the problem is, perhaps you
could send me a copy and I'll see if I can help.

Hi Carlos,

Thanks. Works with SOLO, goes now hill with another web server. Well that
helped me remember another possible senario which can cause the problems
you have. That is if the CGI executable is producing a run time error.

One common reason for this is because it can't find part of the COBOL
system.
On your local machine the executable (if linked to use the dynamically bound
run time) can find the  runtime system via the registery system in the
Windows
operating system. On another system the runtime system won't be avaliable.

Check to see how your application is linked together. It wants to use
the static runtime linking (in the IDE check out build settings and the
link tab). Or use cbllink -BSF ... Try this first. If that does not work..

Another debug method, because I'm sure the reason for the http error
is because something invalid is being passed back to the server from
the application, is to catch this using a stub program.

mybat.cmd:

@echo off
@echo Content-type: text/plain
@echo.
@echo Test batch file.
@e:\...\myapplication.exe

So you form tag will need to say
<form ... action=".....\mybat.cmd" ... METHOD="GET">

Using the Microsoft Management Console to look at the properties of
my scripts directory (right click on scripts, select properties, then
clicking on configuration), under the 'application mappings' I see I
have the line

.cmd    c:\winnt\system32\cmd.exe /a /c %s

Now I can't remember if I added this myself or is there automatically
but one needs this present if you want to execute .cmd files in the
<form> tag from your browser. Best to select in the permissions of
.exe and scripts for now.

Hopefully one now can see the text which is causing the webserver
problems.

Note using this stub method the CGI info is not passed to the application
so your application may not output correct results. I only use it to
see if the application is producing unexpected run time errors.

A slight aside, but you may need this depending on the complexity of
your application.

Another alternative to static linking, which I don't know too much about
is the 'application  server' which I think has all the run time files
needed for  distributing your  application. I've heard in theory when
installed it updates the registory so the cobol runtime system can be
found again.

A third alternative is when installing the application server is
updating the system environment variable PATH to point to the
run time system as PATH is also searched when looking for .dll files.

I hope the documentation will explain the application server side more.

Let me know how you get on.
