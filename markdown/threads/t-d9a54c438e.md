>>The thorny issue of what to do when an apparent error is outside
>> the scope of the present project . . .
In addition to the issues already raised, 
please also consider that perhaps nobody has complained since 
COMPENSATORY errors 
have been introduced elsewhere.  

There have been times when working over legacy code that
I've found an error in one place  that is "undone" by another
error elsewhere -- fix either one alone and you've introduced
a bug.  

My usual response on finding an error that is out of scope for
the present project is one of two:
1) major errors - try to get scope widened to include it
2) minor errors, and major where scope is not widened - insert comment
explaining that a) it looks like a bug, b) because (reasons), c) date/time/signature
ending with explanation that d) per instructions from "on high"
it has not been examined/fixed

The comment doesn't change program logic, it is quick, and it seems
a whole lot better than doing nothing.  (And it has the nice side effect
of making clear just who is responsible for leaving this bug in the code . . .)
