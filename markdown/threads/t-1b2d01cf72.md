> PERFORM UNTIL account-check = "Y"

The scope of this perform includes ALL of the code below. Before obtaining
transactions this condition is set TRUE (see the NOT INVALID KEY code...).
On completion of the first transaction the END-PERFORM is encountered and so
no further action can occur until you get another Account. It will only
process the first transaction. After that it reaches the end of this perform
scope and the condition is TRUE, so it "stops"...

Review the scope of this perform, and decide exactly when it should be set
to TRUE.

Try indenting the source so you can easily see what ranges are involved.

Pete.

ritchie_s01@yahoo.com (ritchie) wrote in message 

> I have two files.
> File 1 holds account info.
…[6 more quoted lines elided]…
> not all the transactions.

Are you sure that 'all the transactions' are actually in the file ?

> 		SELECT transaction-file ASSIGN TO "File 2"
> 			ORGANIZATION IS Indexed
> 			ACCESS MODE IS DYNAMIC
> 			RECORD KEY IS transaction-ac-no.

You seem to have the unique key as just the account number, you can
have only one record for each different account number.  If you have
been writing records to the file they have probably failed to write
and gave a file status error code of '22' = Duplicate record key.

First thing wrong is that if you want multiple records per account you
need to have a key that is more than just the account, it might
include a sequence number, or a type and reference, for example a
unique invoice number or journal number or payment reference.

2nd thing is add a file status to every file and _check_it_.  Without
this you may never know what is happening.

> *READ TRANSACTIONS for THIS account
>  MOVE ac-in TO transaction-ac-no
>  START transaction-file KEY = transaction-ac-no

This will only start with an exact match.  You would change this to >=
or NOT < when you add additional fields to the transaction key.

> ...
>    READ transaction-file  NEXT RECORD 

This reads the first matching record

>     PERFORM UNTIL user-answer NOT = "Y" OR end-of-file-flag 
>            OR ac-in NOT = transaction-ac-no
>           READ transaction-file NEXT RECORD AT END SET end-of-file-flag TO RUE

This reads the _SECOND_ matching record, but there isn't one.

I would suggest using something like:

        SELECT Transaction-File
             ...           
            RECORD KEY Transaction-Key
        ...     

        FD  Transaction-File.
        01  Transaction-Record.
            03  Transaction-Key.
             05 Transaction-Ac-no         PIC X(6).
             05 Transaction-Ref           PIC X(6).
             05 Transaction-Seq           PIC 999.
            ....

        MOVE SPACES    TO Transaction-Key
        MOVE Ac-In     TO Transaction-Ac-No
        START Transaction-File
            KEY NOT < Transaction-Key
            INVALID KEY
                 MOVE HIGH-VALUES  TO Transaction-Key
        END-START
        MOVE "Y"      TO User-Answer
        PERFORM 
            UNTIL Transaction-Ac-No NOT = Ac-In
               OR User-Answer NOT = "Y"

            READ Transaction-File
                AT END
                    MOVE HIGH-VALUES TO Transaction-Key
            END-READ
            IF ( Transaction-Ac-No = Ac-In )
                process record

                DISPLAY ...
                ACCEPT ...
            END-IF
        END-PERFORM
