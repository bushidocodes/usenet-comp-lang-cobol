OO, you may not have declared an Object Reference. Alternatively can be

Linkage section.
01 Lnk-Item		pic x.

			 lnk-Item2
		   returning lnk-return.

-  check your call/invoke passes all parameters
-  your Linkage shows all using/returning parameters
-  your Procedure Division shows all using/returning parameters
-  your calling/invoking program receives back correct parameters
-  ensure parameters are listed in same sequence for the above - a tip

Hi Enric.

While there are many potential causes of this error, as you're using
NetExpress 3 there is a way to get more information on the cause - get
the people running the program to search their hard drive for the file
'mfdebug.log' - this will be created in the current working directory
of the COBOL process when the error occurred. It details the specific
type of trap, the register set, and the COBOL call stack etc. At the
very least, it's a useful file to send to Technical Support with your
bug report, should you decide that it's not an error in your program.

The most likely ways of causing this error from COBOL code are:

Bad linkage access
Wild reference modification parameters
Bad parameters passed to CBL_ calls, Windows APIs or other non-COBOL
libraries.

Cheers, Kev.

I forgot to mention:

If it's a problem with referencing linkage items which have not been
passed, this will generate an error '203' in .INT code, but by default
is not checked so may generate a '114' error in GNT/OBJ/DLL. However,
if you use the LINKCHECK code-generator directive, GNT/OBJ/DLL code
will also check the linkage parameters and produce a '203'.

I don't *think* this will be your problem, as something as fundamental
as this would most likely show up on your machine also, but it's worth
trying if you don't get anywhere with other investigations.

Cheers, Kev.
