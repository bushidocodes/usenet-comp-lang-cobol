Sometime ago I was looking for some help with a routine to find out
which user had set up a record lock and then gone on a two week
vacation.

We talked about lots of stuff. Logical application locking vs OS &
COBOL locking. We talked about process ids vs user ids and other stuff
but we didn't get a solution. I left the dicussion convinced that the
OS knew which process had the lock and also knew which user had the
process but that getting the data out of there was going to be a black
art.

Anyway, we have a solution, mostly down to the support team at
MicroFocus (Merant) in the UK. They pointed us to Vic Abell @Purdue
University Computing Center
ftp://vic.cc.purdue.edu  Look in pub/tools/unix/lslk.

Here we found a great tool that reports process ids for locked files.

Because the designer of our ancient application had the foresight to
handle all of his file IO through a standard library. A few changes to
the library and we have lock messages that report the user id of the
offending user. We are pleased with this and so are our customers.
Something that has been a minor problem for them for 10 years or more
has now been killed.

We won't make any money out of this as the application has been
replaced but the sysops who have to look after this system will now
have an easier life.

Regards

Mark

Doug,

You found the solution, but I think the most significant thing in your
message was "Because the designer .... had the foresight to handle all
his file IO through a standard library....".

Once the problem was found you could resolve it, but without that
foresight touch, somewhat more difficult ....?

I'm wondering just how many people, mainframe or otherwise have adapted
to this approach. A certain New Zealander, yes you Pete if you are
looking in again, mentioned back on the old Micro Focus Compuserve
Forum, 'three-tier system design' - a new term to me, but I easily
understood what he was getting at. Certainly many of us some 20 years
ago conceived structural design as say, File Section, Screen Section
followed by procedural code.

Recall there weren't the tools then, so we took a simpler all-embracing
approach. As I recall under DOS/MSDOS my editor was a crummy thing
called EDLIN - but we got there.

When I go back to my old RM code I see, within individual programs,
where I wrote in Program A, the handling routine for file X. In Program
"B" my wording is changed for handling file X - but I am attempting to
do the same - and gave myself a dual testing problem - when a common
routine would have meant only one test.

The 'three-tier' is certainly the approach I have followed with OO
design and as yet there isn't one instance where my separated
file-handling has let me down. Sure, I've had 'blips' but they have
always been related to logic or code errors in the calling programs,
never the file-handling modules. That's not to suggest my file-handling
techniques are unique or are the ultimate - far from it; I'm sure they
can be vastly improved, but in their current state they do give me
something with surety that I can hang my hat on.

I'm just as convinced the same technique in structured programming gives
the same benefits whether you are using flat files or databases. And you
can expand this three-tier approach to all aspects of Screen/GUI
handling.

If you are sceptical, may I suggest to others you review your existing
programs and consider what the design impact would be to achieve this. 
You might find this enlightening - and if you should come to the
conclusion to give three-tier a shot, I'll guarantee you will never look
back.

Jimmy,

In building our product, which runs on Win98/NT, we took the approach
of a user presentation/interface layer, an application/business logic
layer and a data access layer. Is this what you mean?

We wanted to be able to write something once and use it infinitely. We
also wanted to be able to "unplug" one database and "plug in" another.
This software design approach achieved both objectives for us.

Even though we have standard templates for various types of user
presentation/interface (table browse, row update, etc.), we have been
creating a form for each instance. we are now trying to think more
along the line of reusable forms. We use Flexus SP2 and VB6 for this.

For the application/business logic layer, we build DLLs that contain
processes that are used throughout the system. When we need a slightly
different version of an existing process, we modify the DLL to support
the requirement rather than create a new DLL. We use CA-Realia COBOL
for this.

The data access layer is another set of DLLs. It then is a simple
matter of distributing the system with the right set of data access
DLLs. We use the language/tool to create these DLLs that is most
appropriate for the database we want to "plug in".

We haven't gotten into OO yet, but hopefully we will in the future.

John

I agree 100%. In fact the design goes quite a bit further than that.

All screens are created in a text editor, then using a COBOL program,
the system generates its own SET DEFAULTS, GET data from data layer,
DISPLAY, ACCEPT, VALIDATE, PUT data to data layer and PRINT to print
layer modules for each screen.

All the business logic is handled in code modules (of which there are
standard building block modules for common tasks, e.g. a table
maintenance model, a report model etc) calling the the generated code
for user interface. One the business module is happy with the code is
calls the database layer which has open, start, read, write, delete,
read next etc etc close methods. This design has allowed us to move
from CP/M to DOS to UNIX to Windows. The database is still flat C-ISAM,
there doesn't seem to be a reason to go to a relational database. If we
need to reformat a file there's a generator to create the new layout
transfer the old to the new, create the indexes etc and execute this at
install time on user systems.

Admittedly the functionality of all of this has improved with time but
the original design had all the building blocks, three tier, generated
code, database independence and user specific database extensions (the
business objects pass them around as a memory block, only the programs
that need to know the content will manipulate the data).

All of this started in the late 70s. The bloke's a genius (and very
wealthy now). He made some mistakes - the file lock handler that we've
debated, a transaction processing system that's missing a start,commit
design but on the whole it's great a system work with.
