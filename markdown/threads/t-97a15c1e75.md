I don't agree with the 'it depends...' crowd. A subprocess should never
presume to know how a parent process will deal with a problem, regardless of
the level of severity. The parent process should always have control of the
situation. Sure, the parent process may decide to ABEND, but it may not. The
parent may have resources tied up that will be improperly cleaned up if it
does not get back control. Examples include failing to release locks,
failing to respond to a client process, leaving it waiting indefinitely,
possibly forever, failing to deallocate memory, leaving the end-user facing
a screen of death.

I agree completely with Karl (or Oscar, as the case may be), for all
the same reasons.  There might be some special exceptions, but I've
never seen one.

A more difficult question is: how do you issue the error messages?
We are on the horns of a dilemma:

On the one hand, the module that detects the error is the one that
can best describe it.

On the other hand, the error may be detected by a low-level module
which, in principle if not in practice, might be called from any
of various kinds of environments with quite different ways of
issuing messages.  It might be a batch program, a CICS transaction, an
IMS transaction, a CGI program driven by an HTTP server, a GUI, etc..
If you let the low-level module issue the message directly,
you confine that module (which might otherwise be completely
generic) to a very specific environment.

Here's how I've generally approached the problem, but I'm not
convinced that it's the best solution:

1. The module that detects the error constructs a suitable message
of a standard length (50 characters seems about right).  If the error
is so serious as to render further processing meaningless or 
impossible, the module sets a fatal-error-switch in WORKING-STORAGE; on
future calls it refuses to do its normal job.  Finally the module
returns an error code.

2. The calling program notices the error status and calls a separate
entry point in the lower-level module in order to fetch the 
corresponding message.  It stores the message in its own WORKING-
STORAGE, sets a fatal-error-switch if appropriate, and returns an
error message to its caller.

[Note how I have slipped in a subtle advertisement for multiple
entry points, the subject of another thread.]

3. The next level of program notices the error status, calls a 
separate entry point to get the message, and so forth, until...

4. Eventually the error condition percolates to a level where the 
means of issuing messages is pretty clear -- a level that is definitely
a CICS program (for instance), and which would never be confused with
(for example) a batch program.  This level is usually somewhere in
the middle.

5. Now that we know what environment we're in: issue the message.

It's easy to imagine scenarios where this approach breaks down, but
in practice it has worked reasonably well for me, though a bit
clumsily at times.

The biggest drawback is that every module relies on its caller to
check for error codes.  Checking for error codes all the time is
extremely tedious.  More importantly, it's easy to omit or forget
this check.  A single lapse of discipline, especially at an upper
level, can make your program ignore whole categories of errors,
possibly with bewildering results that will be difficult to decipher
later -- if you even recognize that there *is* a problem.

This drawback is probably the best argument for aborting the program
at the first sign of trouble.

It is also an argument for using exceptions, where the environment
supports them.  With error codes you have to go to extra trouble to
notice a problem, but with exceptions you have to go to extra trouble
not to.

How have the rest of you approached these issues?

We take a similar approach; except that we generally provide a
standard calling structure that includes both the return code AND the
error text; that way, you don't need to make another call to obtain
the error text.  

The calling program then takes the appropriate action to notify the
user of the error, shutdown, etc.

We also make it a point to incorporate a standard event-logging
mechanism in critical code.  It can be enabled/disabled by a simple
switch and when enabled, calls an error routine of ours that decodes
and formats the message, then logs it to a file.

It's often easier to figure out what went on after the fact if you
have a log of events leading up to the failure.  You might include
keys, return codes, actions taken, whatever, in the log.

It was particularly helpful when working with our GUI product because
the introduction of a debugger into Windows code can modify its
behavior, making it impossible to debug.  The event logger does not
affect the execution path of the module.

It was exactly this approach which drove me to devise something better.

When I first joined a certain application group, they used a cycle
checking subprogram which, if it encountered an error, returned one of
a long series of error codes (and no message).  Almost every load 
module in the application included this subprogram.  Program after 
program contained hundreds of lines of tedious, repetitious code to 
translate error codes into the corresponding messages.  Sometimes this
code took up more space than all the rest of the PROCEDURE
DIVISION.

With a few exceptions it didn't really matter which error code turned 
up -- any kind of error required that we abort the job -- except that 
they needed different messages.

After I rewrote the subprogram to provide its own error messages, I
could quickly eliminate thousands of lines of code in the calling
programs.  The new messages were also more informative because they
could include details which were otherwise internal to the subprogram.

Here's another consideration: what happens if you change the internal
implementation of the subprogram?  For example, maybe you start using
a database instead of indexed files.  You're likely to find yourself 
with a different collection of error conditions -- but the calling 
programs only understand the old error conditions, and their messages 
are obsolete or worse.

My rule is: the module which detects an error should be the one to 
define the error message, because that's the module that knows what 
the error is.  How to deliver the message is another question, which 
different people answer in different ways.

Usually my subprograms provide different error codes for different
error conditions, just in case the calling program needs to treat 
different conditions differently.  In my experience, however, it is a 
rare program that cares which kind of error it gets, as long as it can 
deliver an appropriate message and shut down.  (That may not be true of 
interactive programs such as a GUI.  My experience is mostly in the 
batch world.)

With the approach I described earlier, the called program provides
the text of an error message on request.  With your approach it
provides an error code on request.  These dependencies look identical 
to me, except that your approach adds a dependency on an external file
to provide the message text.

> REMOVES ALL THAT CODE FROM MEMORY UNTIL NEEDED, then
> ONLY a small program is loaded.

This approach has both advantages and disadvantages.

One advantage, as you have pointed out, is that the memory requirements
are potentially reduced.  I say "potentially" because the code needed
to fetch the message from an indexed file may itself occupy a
non-trivial amount of memory.  Depending on the COBOL dialect, of
course, you may be loading much of the IO subsystem anyway even if
you don't use it.  Alternatively you could call the message program 
dynamically, so that (again depending on the dialect) it wouldn't be
loaded until it was needed.

Another advantage is that you can change the messages by changing the
message file, without a recompile.  As Mr. Morris has pointed out in a 
related posting, this approach is especially useful for 
internationalization.

Now for the disadvantages:


1. Messages built or defined within a program serve as documentation,
describing the error being reported.  Once you separate the error
message from the program which detects the error, it may not be 
obvious just what the error is.  For clarity you would have to add
comments which would otherwise be redundant.

2. Hard-coding a message into a program is quick and easy.  Providing
a reference to a message stored elsewhere is not.  Not only do you 
have to add an entry to an external file somewhere, you also have to
assign an error code for it, and make sure that the error code is
unique across the system (or maybe find a message which has previously
been defined, and reuse its error code).  The sheer tedium of
maintaining messages in this fashion is likely to discourage 
programmers from providing good messages.

3. If you accidentally return the wrong error code, or if the message 
file becomes out of sync with the program, you're unlikely to notice, 
because you never see the text of the messages unless you look for it.

4. A new release of the program will likely involve a new release of
the message file as well, creating additional opportunities for
mistakes.

5. Often a generic error message that you could easily store in a file 
will not be as useful as a more specific message that incorporates 
information from the specific incident.  For example, a complaint
about an invalid date format will be much more useful if it includes
the invalid date.  If you build the error message locally you can
easily insert this information.  You can't do so with externally
stored messages unless you make the messaging system rather complicated.


My own COBOL experience has been mostly in the mainframe world, on
programs for use only within the company.  Neither memory constraints
nor internationalization has been an issue.  In this context the
use of a message file has little to offer.  One can imagine other
situations where it would make more sense.
