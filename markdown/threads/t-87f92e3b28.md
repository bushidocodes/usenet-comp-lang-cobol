Kellie,

What you've described above is not your elegantly named "flasher" <G> - I'll come back to
that last of all. I don't want to write a treatise, but a a paraphrase to give you some
background.

Firstly, because I don't think I've mentioned it to you previously - you can find all the
M/F support classes and methods available in N/E. I've got a shortcut Icon set up using
the following :-

"C:\Program Files\MERANT\Net Express\Base\Bin\MFNXCL31.CHM"

Secondly all those support classes contained above - you can also access the source code
if you go down through the directories :-

\ProgramFiles\Merant\Net Express\Base\Source\Basecl
\ProgramFiles\Merant\Net Express\Base\Source\GUIcl

So if you are doing something with OO or GUIs in the Animator you can set options so that
your program steps through the OO classes as well as your own program. It's rather a
futile exercise to do this - you can go dizzy looking at the iterations - and in "run
mode" it does all this referencing in a flash.

Quick summary - SIMULA(Simulation) from Norway, then Smalltalk from Xerox PARC (Palo Alto
Research Center), 'copied' by Steve Jobs for his Lisa Apple machine, then "borrowed" from
Steve by Bill Gates giving us Windows.

When OO COBOL started it meant re-inventing the wheel. Certainly both Micro Focus and
Hitachi based their OO support classes on Smalltalk. Not sure if Fujitsu went a different
route. Then of course there's IBM with their old Visual Age product - not sure what that
followed.

Even using Smalltalk as a design base, the COBOL vendors  probably (a guess) took at least
a year doing the grand design before coding classes, because they wanted the concept of
hierarchies. That reference to classes above - take simple statements like :-

    invoke thisObject "hide" OR invoke thisObject "show"

They are shown under the respective GUI classes that you are using, but via the hierarchy
concept, they are accessed from  a "super" class above - haven't checked but "hide" or
"show" maybe some three or four levels above..

One of your early questions here was, "Is OO any good ?". My own personal opinion,
REUSABILITY is paramount to OO, achieved through an hiercarhial structure to achieve those
"hide" and "show" methods above. You are well aware that the methods written in COBOL are
intend to be user friendly (to us the developers), and disguise the fact that the "super"
method calls on support from the operating system - whether it's file/DB handling or using
GUIs..

So now back to your "flasher" - The words obviously have other meanings, but in a
Programming GUI context :-

MODAL DIALOG   - a dialog box that "must" be acknowledged by user response from  either a
dropdown or popup menu, check-boxes, radio-buttons or pushbuttons, and possibly some
others.

MODELESS DIALOG - does "not" require user response and the user can focus on other dialogs
displayed..

An example of MODELESS - you have it in that Infobox source I sent you, the very first
lines of which read :-

CLASS-ID.       InformationBox
                          inherits from Modeless.

While my InfoBox displays info being processed, it requires no acknowledgement from the
user.

An Example of MODAL - that's what you are doing :-

".....the message will be flashed on the screen if the end-user select an option
from the main menu that requires sorting the master indexed file, and then
process his request which is populating a listbox to view a complete listing
of entry items, where he or she can make another selection for more details.".

Subject to selection from the Menu -

If somehow you have some flag in your application that tells you the Index File does need
sorting THEN you could display a child MODELESS telling the user that is what is currently
going on. However if you want to prompt the User with "Do you want to sort" - then that's
MODAL.

Does that clarify it a bit - hopefully so ?

My initial reaction was to clarify Modal/Modelss, but now viewing your objective above - I
see that we can ignore your "flasher" entirely. If you are using Sorted Collections it
doesn't matter a tuppeny-cuss what sequence your ISAM or DB Table is in. As you create the
collection you determine the sequence by using the DefaultSort.which is left to right,
(ascending). So given a file record :-

(1) A/C No  (2) City    (3) Address Line1   (4)  Name (5) etc.....

you firstly select the fields you want and put those into a record added as an element to
a collection - which could be :-

(4) Name (2) City (3) Address Line 1 (1) A/c No.

Now Judson will be pleased - you haven't thrown a "flasher" at him - just produce and
display the Listbox and he can select an entry !

Jimmy, Calgary AB
