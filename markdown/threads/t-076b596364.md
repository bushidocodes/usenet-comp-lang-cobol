```
Is this file very static, or is it updated more than seldom?  I would hesitate
to use standard MF Indexed Files for that much data unless it is not often
updated, or it isn't critical.  I would use something with a robust rollback
and recovery capability.  MF COBOL and Net Express can be configured
to use Btrieve as an Indexed File handler.  Then you can use transaction
processing, and Btrieve (Provantage SQL 2000 now, but still provides the
underlying Btrieve API to the engine) is bulletproof.  I wrote a substantial
(>100k LOC) multi-user client server inventory control system in 1986-87
using MF COBOL and Btrieve (DOS then, Win32 now) for a client.  This
system has more indexes per file than any I have ever done, typically 4 to
6, some more than 10 (with 1 to 8 fields per index)  The database is well
over 500MB, and we have never had to restore the database because of
data corruption, except when a drive bit the dust.

This was the last system I wrote (begun in 1986) using 2 digit years, so
I had to expand the dates for Y2K.  I wrote programs to copy the old data
files to flat files, then reload the flat files into the new date files.  Total
conversion time (several million records) was only about 2 hours on a
Dell Pentium II 550, 256MB.  Considering the number of indexes that had
to be populated, I thought Btrieve's performance was very good.
```
