```
Our users are having a problem printing in reports in a landscape format
from one of our applications which has been ported to MicroFocus COBOL.
This problem only occurs on certain printers.  It's been observed that the
printers associated with this problem are those which are Postscript
compatible, but I can't say for certain that that�s true.  This problem
happens with a Xerox Doc Center 220/230 PS2 printer.  It also happens with
some HP printers, e.g. with an HP LaserJet 4Plus.  The program works fine
with many other printers, and serves to allow the application user to print
reports from the application.

We are not sure when the problem started.  The initial version of the
application was a 16-bit version fielded at a few of our sites, and seemed
to work fine.  Then the 32-bit version was developed and fielded at a few
more sites and seemed to work fine.  As it was fielded at more and more
sites, we began to receive reports about
this problem.

The error is:
"Execution error:  file 'PRTTEST.gnt'
 error code: 114, pc=0, call=24, seg=0
 114  Attempt to access item beyond bounds of memory (signal 11)"
Press Enter to continue.  After pressing enter, the program aborts.

The program in question uses calls to: "CBL_OPEN_FILE",
"PC_WIN_OPEN_PRINTER_EXT", "PC_WIN_SET_PFONT", "PC_WIN_WRITE_PRINTER", and
so on.

The program is compiled with MicroFocus COBOL:
Workbench v 4.0.32
Advanced Organizer Initial Release, version DETV 1.1.25
Animator V2 ANM 8.b.cc
Dialog 2.5.64

We did not develop this system, but do have access to the source code.  The
developer is overloaded with work and so now we have begun to look into
this.

Any ideas or suggestions will be appreciated!

Mike
```

I just discovered an interesting windows bug that might be the culprit. It
seems that windows will handle file names like "LPT1:" only when *all*
file names in the run unit adhere to 8.3 style names, EVEN FOR DIRECTORIES.

Example.  Go into a dos window, and copy a test file to a captured printer
port.  IE:  Copy abc lpt1:

This will work from a directory called "temp", but not from a directory
called "temp.extension".  As soon as the file being copied does not
adhere to the 8.3 rule throughout (even though it is not mentioned in
the command line), then the device "LPT1" no longer seems to get
recognized.

I just had a print routine fail, and it failed because I was running the
program from an area with an extended name.  As soon as I executed
it from an area with a DOS compatible name, the problem goes away.

Bloody silly, but then who the hell knows what that damned OS is doing?
