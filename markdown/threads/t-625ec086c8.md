Neither C nor C++ have graphics as part of their languages.

Graphics are very device or platform dependent. Input
devices are even more platform dependent.

If I had any desire to write a game for, let's say, Windows, I
would probably get the Microsoft DirectX SDK and write to the
API. Basically, this means that, if a COBOL implementation
for Windows can call a C language API that is implemented
as a DLL, then COBOL can be used to write games.

It can, therefore it can!

I am sure you had a particular system in mind; but I will ask if C++
comes with graphics in all UNIX implementations. Probably not.

As for Windows and any C++ environment which uses MFC, a quick
check under the hood will show that the GDIObjects in MFC maintain
some information about the objects but the actual graphics work is
done by calling Windows.

>  When

If the COBOL environment is for Windows, it has (proverbially) all the
graphics capabilities of C++. It is just that it may be somewhat more
difficult to use; or it might just be far less familiar to COBOL
programmers.

WRT older MS-DOS implementations, MS C did have some
presentation graphics capabilities that were extensions to the
standard library and I seem to recall Micro Focus having an
add-on package, CO-Graph. for presentation graphics. IMO,
these graphics capabilities were probably not suitable for games
programming.

COBOL itself has some means to do character screen handling via ACCEPT and
DISPLAY. This is enough to write elementary games. The ones with high
resolution stuff could also be made using object oriented features (to be
included in the next language standard) or vendor-specific extensions.
Today, sound and graphics are quit easy possible from within COBOL, however
the language is not primarily designed for such things, so it will always be
better to do game programming in other languages.
