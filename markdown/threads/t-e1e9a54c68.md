There have been discussions in the Standards committees about "phasing out"
77-levels - but that is not currently on any plan for ANSI or ISO Standard
COBOL. This feature is not in either the OBSOLETE (definitely to be removed
in the next Standard) or the ARCHAIC (you shouldn't use it - but it will
stay around until programmers stop using it) lists in the draft for the next
Standard.

(I cannot remember if the JOD actually removed or deprecated them. This is
something that Don Nelson, who watches this newsgroup, could confirm or
deny).

Concerning the original question about any actual software difference
between using 01s and 77s - this is an implementation specific issue. What
is efficient (speed, size, other) for one compiler may or may not be true on
another. This is certainly true for whether 01 data-items are or are not
allocated on doubleword boundaries. As the original question concerned
Micro Focus COBOL, you should check out the ALIGN directive which does
impact where these items are allocated - but which applies equally to
01-levels and 77-levels.

Yes - 01's are aligned on double-word (at least in IBM). 77's are not -
they just snuggle up to each other.
You can demo this by writing some code and then using a PMAP (or is it
DMAP?) on the compile to see the offsets.
In the old days, when every byte was precious, we used 77's. However, an
01 (such as 'WORKING-FIELDS') followed by 05's accomplished the same
thing.
Thus (I was told many years ago), some upcoming version of COBOL would
not allow 77's. So we went about changing to 05's and not making new
77's.

Today, with storage not so much a worry, we often see 01's used where we
would have seen 77's years ago. However, I personally feel that we
should use 01's like 'WORKING-FIELDS' followed by 05's anyhow, just for
good programming practice.

On a related note - I was told a long time ago that math calculations
are done on a full-word boundary (hence 01 COUNTER PIC S9(4) COMP). So
05's embedded in an 01 (or for that matter snuggled 77's) would only be
a repository for the math end result - the math would be performed in
some other work full-word portion of core. This was a performance
consideration in the 70's but may not be now. Also, maybe new compilers
don't act like this anymore.

But based on this perception, I try to create an '01 COUNTERS' where
each 05 under it is S9(4) COMP or S(9) COMP. Other newsgroupers (more
knowledgeable than me on this) may have better words to say on this sort
of thing. I could be completely wrong, for example ;).

John

When I first looked at the storage layout for COBOL, on an IBM OS/MVT
(yeah, that old), as I recall storage was assigned in the sequence that
it was defined in. Padding occurred where necessary to force alignment.
All COMP fields were full word aligned. All COMP-2 fields were double
word aligned. Programmers took advantage of this by coding the first
variable as a PIC(X) containing:

'WORKING STORAGE STARTS HERE'

Of course, the last variable defined was another PIC(X) containing:

'WORKING STORAGE ENDS HERE'


That way when they looked in the dump they could find the working
storage. Darned lazy programmers. If you are going to shoot a dump, then
shoot the dump.

The last time I looked (two years ago) the IBM COBOL compiler moved all
the variables around to optimise storage usage. So these two strings
were no longer marking the beginning and end of working storage.

So if you still see any code doing this you might as well take it out
because it no longer serves a useful purpose.
