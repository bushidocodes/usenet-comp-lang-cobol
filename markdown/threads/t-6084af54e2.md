Tricky Thingy, huh? LOL

Michael Bruyninckx ?

I'm not entirely sure I understand what you're saying, but, I think what you
want is to read in data from one file, and create a sorted, purely sequential
version, based on a specific field. In addition, the sort field has to ignore
everything that isn't alphanumeric (A-Z,0-9), and translate all lower case
characters to upper case. Is this correct?

If so, then this can't be any simpler - trust me!

Set up your SELECT/ASSIGN and FD areas for your input and output files, making
both access modes sequential. Set up a 3rd SELECT/ASSIGN with an SD for the
sort file. When laying out the record for your sort file, duplicate the key
field so you can see that the translation is progressing as it should.

For example...

FD INPUT-FILE.
01 INPUT-RECORD.
05 INP-KEY-FIELD-1 PIC X(05).
05 INP-KEY-FIELD-2 PIC X(10).
05 INP-DATA-FLD-01 PIC S9(07)V9999 COMP-3.
05 INP-DATA-FLD-02 PIC S9(07)V9999 COMP-3.
05 INP-DATA-FLD-03 PIC X(30).
05 INP-DATA-FLD-04 PIC X(10).
05 INP-DATA-FLD-05 PIC S9(05)V99 COMP-3.
05 INP-DATA-FLD-06 PIC S9(09) COMP-3.
SD SORT-FILE.
01 SORT-RECORD.
05 SRT-DATA-FLD-04 PIC X(10).
05 SRT-KEY-FIELD-1 PIC X(05).
05 SRT-KEY-FIELD-2 PIC X(10).
05 SRT-DATA-FLD-01 PIC S9(07)V9999 COMP-3.
05 SRT-DATA-FLD-02 PIC S9(07)V9999 COMP-3.
05 SRT-DATA-FLD-03 PIC X(30).
05 SRT-DATA-FLD-04 PIC X(10).
05 SRT-DATA-FLD-05 PIC S9(05)V99 COMP-3.
05 SRT-DATA-FLD-06 PIC S9(09) COMP-3.
FD OUTPUT-FILE.
01 OUTPUT-RECORD.
05 OUT-KEY-FIELD-1 PIC X(05).
05 OUT-KEY-FIELD-2 PIC X(10).
05 OUT-DATA-FLD-01 PIC S9(07)V9999 COMP-3.
05 OUT-DATA-FLD-02 PIC S9(07)V9999 COMP-3.
05 OUT-DATA-FLD-03 PIC X(30).
05 OUT-DATA-FLD-04 PIC X(10).
05 OUT-DATA-FLD-05 PIC S9(05)V99 COMP-3.
05 OUT-DATA-FLD-06 PIC S9(09) COMP-3.

This example assumes that you want to translate and sort on INP-DATA-FLD-04.
Note that the INPUT-FILE and the OUTPUT-FILE are laid out exactly the same, so
you could put the definition in a single copybook. You could use the same
copybook for the SORT-FILE as well, just adding the additional field.

FD INPUT-FILE.
01 INPUT-RECORD.
COPY FILE-LAYOUT.
SD SORT-FILE.
01 SORT-RECORD.
05 SRT-DATA-FLD-04 PIC X(10).
COPY FILE-LAYOUT.
FD OUTPUT-FILE.
01 OUTPUT-RECORD.
COPY FILE-LAYOUT.

However you decide to do it doesn't matter since you're still in school. In a
work environment, however, there >>will<< be strict rules governing how your
programs are coded.

Next, you'll need two tables in your WORKING-STORAGE to translate all lower
case to upper case, and to select out the characters you want to include in the
SRT-DATA-FLD-04 field.

05 WS-XLATE.
10 FILLER PIC X(26) VALUE
'aAbBcCdDeEfFgGhHiIjJkKlLmM'.
10 FILLER PIC X(26) VALUE
'nNoOpPqQrRsStTuUvVwWxXyYzZ'.
05 FILLER REDEFINES WS-XLATE.
10 WS-XLATE-ENTRY OCCURS 26 TIMES
INDEXED BY XLATE-INDEX.
15 WS-X-LOWER PIC X.
15 WS-X-UPPER PIC X.

Quick side note here, Michael Bruyninckx . Depending on the
version of COBOL that you're using, you may not need to have a lower/upper case
translation table -- there may be an intrinsic function available to do this
for you. Check in the LRM for more details, or get with your professor.

05 WS-SELECT PIC X(36) VALUE
'ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789'.
05 FILLER REDEFINES WS-SELECT.
10 WS-SELECT-ENTRY OCCURS 36 TIMES
INDEXED BY SELECT-INDEX.
15 WS-SELECT-X PIC X.

You're also gonna need a couple of work fields...

05 WS-I-INDEX PIC S999 COMP-3 VALUE ZEROS.
05 WS-O-INDEX PIC S999 COMP-3 VALUE ZEROS.
05 FILLER PIC X VALUE SPACES.
88 CAN-CONTINUE VALUE SPACES.
88 TOLD-TO-STOP VALUE 'S'.
05 WS-KEY-FIELD PIC X(10) VALUE SPACES.
05 WS-BYTE PIC X VALUE SPACES.

Now that you have your definitions, your tables, and your work areas in place,
the rest is easy. Use the SORT verb to populate the OUTPUT-FILE from the
INPUT-FILE.

During the input procedure, for each INPUT-FILE entry, use reference
modification to scan through the INP-DATA-FLD-04, byte by byte, from left to
right, first translating any lower case characters to upper case (you can use
the SEARCH verb to do this), then exclude any characters not in the WS-SELECT
string (again, use the SEARCH verb). Put the resulting string in the
SRT-DATA-FLD-04 field, again, using reference modification. Note that the input
index will not always be the same as the output index, so don't attempt to use
the same variable.

SET CAN-CONTINUE TO TRUE.
MOVE ZEROS TO WS-O-INDEX.
MOVE SPACES TO WS-KEY-FIELD.
PERFORM
VARYING WS-I-INDEX FROM +1 BY +1
UNTIL TOLD-TO-STOP
IF (WS-I-INDEX IS GREATER THAN +10)
SET TOLD-TO-STOP TO TRUE
ELSE
MOVE INP-DATA-FLD-04 (WS-I-INDEX:1)
TO WS-BYTE
* Check with your professor to see if there's an intrinsic function available
to do this
* lower/upper case translate before bothering to copy this search...
SEARCH WS-XLATE-ENTRY
WHEN (WS-X-LOWER (XLATE-INDEX) IS EQUAL TO WS-BYTE)
MOVE WS-X-UPPER (XLATE-INDEX)
TO WS-BYTE
END-SEARCH
ADD +1 TO WS-O-INDEX
MOVE WS-BYTE TO WS-KEY-FIELD (WS-O-INDEX:1)
END-IF
END-PERFORM.
SET CAN-CONTINUE TO TRUE.
MOVE ZEROS TO WS-O-INDEX.
MOVE SPACES TO SRT-DATA-FLD-04.
PERFORM
VARYING WS-I-INDEX FROM +1 BY +1
UNTIL TOLD-TO-STOP
IF (WS-I-INDEX IS GREATER THAN +10)
SET TOLD-TO-STOP TO TRUE
ELSE
MOVE WS-KEY-FIELD (WS-I-INDEX:1)
TO WS-BYTE
SEARCH WS-SELECT-ENTRY
WHEN (WS-BYTE EQUAL TO WS-SELECT-X (SELECT-INDEX))
ADD +1 TO WS-O-INDEX
MOVE WS-BYTE TO SRT-DATA-FLD-04 (WS-O-INDEX:1)
END-SEARCH
END-IF
END-PERFORM.

Once the SORT-RECORD is populated, RELEASE the sucker. During the output
procedure, for each SORT-FILE entry returned, write out a new OUTPUT-RECORD
(matching up the fields, of course).

A couple of quick notes here.

The sort file definition depends on the OS and hardware that you're working on.
Under VMS and OS/400, the pointer (ASSIGN) isn't important since the OS takes
care of it for you. But under VSE, you do have to make sure the sort work area
is properly defined. I have no idea how you'd do this under MVS. I'd suggest
getting with your professor on this.

Depending on the OS, and version of COBOL that you're using, you may have to
open the OUTPUT-FILE in one of two ways: OPEN OUTPUT or OPEN EXTEND. Under VMS,
I'd open OUTPUT and let the OS create the file for me. Under OS/400, however,
you'd use CL to create the physical file (using QCMDEXC) and then open EXTEND.
Same thing under VSE. Again, I have no idea how you'd do this using MVS.

Hmmm... in fact, Michael Bruyninckx , I'm not even sure whether
you'd even be able to do any of this using MVS... :o But (!), under VMS,
OS/400, and VSE, there'd be no problem... :D :D :D

Hope this has answered your questions!
