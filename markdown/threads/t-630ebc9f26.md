```
The explanation is actually in the Designer help file; start with the topic
"Defining Record Types".

Essentially, one uses COBOL 88 level(s) (which may be added with the
Designer if they do not exist in the original source) to associate the
particular values that identify the record type(s).  Complex expressions
involving several level 88 items may be used, as well as the possibility of
using several synthesized elements (e.g. whether a specific data item is a
valid numeric).  Usually, however, the record type is simply determined by
one or two values in the record and all this is quite straightforward.

After one has defined record types, the record type appropriate for a table
is selected, and then the dta elements for that table are dragged/dropped on
the table definition.

This is a short explanation, of a process that is documented in the help
file in all of its (possible) complexity.  I will be happy to answer
specific questions;  seeing the FD in question is usually helpful.
```

```
As a general guide, you can handle multiple record types in a file by using
REDEFINES - either explicit if using working-storage, or implicit if using
multiple '01' level record descriptions in the file section FD.

To use implicit redefinition in the FILE SECTION, you might code...

FD  Multi-record-file.
01   Header-record        PIC X(100).
01   Detail-record          PIC X(100).

Each READ of the file fills both header-record and detail-record; it is your
task to figure out which it is. For this it is pretty common to use an '88'
level entry. If the first character of the record contains 'H' or 'D' to
identify header or detail record types, a revised file section entry might
now look like...

FD  Multi-record-file.
01   Header-record        PIC X(100).
01   Detail-record          PIC X(100).
01   FILLER                 PIC X(01).
        88 IS-HEADER-RECORD  VALUE IS 'H'.
        88 IS-DETAIL-RECORD     VALUE IS 'D'.

If the records are different sizes, all you really gotta watch out for is
that you list the largest record size first.

Usually I am more comfortable using a single buffer (FD), and reading the
data into working storage, but the data structures look about the same. That
may be more style than substance - although in "header" and "detail"
scenarios, you often need to keep the header record around while processing
multiple details, and if you use the implicit redefinition shown above, as
soon as you read a detail, the header record is no longer available to you.
```
