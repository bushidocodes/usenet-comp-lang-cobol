What I always do is this:

01 I PIC S9(4) COMP SYNC.
01 W-STRING.
05 W-STRING-X OCCURS 20.
10 W-STRING-9 PIC 9.
01 W-NUMERIC PIC S9(15) COMP-3.
01 W-S PIC S9(1) COMP-3.
01 W-F PIC S9(4) COMP-3.
* COMP-3 stands for packed decimal number.
01 W-DP-FLG PIC X.

MOVE WS-ALPHA TO W-STRING.
PERFORM STRING-TO-NUMERIC.
COMPUTE WS-COMP-2 = W-S * (W-NUMERIC / W-F).

STRING-TO-NUMERIC SECTION.
MOVE 1 TO W-S W-F.
MOVE "0" TO W-DP-FLG.
MOVE 0 TO W-NUMEIC.
MOVE 1 TO I.
PERFORM UNTIL I > 20 OR W-STRING-X (I) NOT = " "
ADD 1 TO I
END-PERFORM.
IF I > 20 THEN GO TO STRING-TO-NUMERIC-EXIT.
IF W-STRING-X (I) = "-" THEN
MOVE -1 TO W-SIGN
ADD 1 TO I
END-IF.
PERFORM UNTIL I > 20 OR W-STRING-X (I) = " "
IF W-STRING-X (I) IS NUMERIC THEN
COMPUTE W-NUMERIC = W-NUMERIC * 10
ADD W-STRING-9 (I) TO W-NUMERIC
IF W-DP-FLG = "1" THEN
COMPUTE W-F = W-F * 10
END-IF
ELSE
IF W-STRING-X (I) = "." AND W-DP-FLG = "0" THEN
MOVE "1" TO W-DP-FLG
ELSE
do whatever error procedure you like
END-IF
END-IF
ADD 1 TO I
END-PERFORM.
STRING-TO-NUMERIC-EXIT. EXIT.

I hope this can be of some help.

Solutions in this group are called static because the format of WS-ALPHA

I am afraid this is not a practical way of handling the problem.

The answer is "No" in usual cases. Consider when we need such a conversion
from an alpha-numeric item to a numeric item. The alpha-numeric item will
be entered from a keyboard or read from a text file. So in many cases,
WS-ALPHA is *not* in the format of 99.9999 and sometimes it may have an
invalid value that can not be converted to numeric.

I found some bugs in this coding but would not point them out because
they would be found easily.

And my coding follows.

01 STRING-TO-NUMERIC-WORK-AREA.
05 S2N-I PIC S9(4) COMP SYNC.
05 S2N-STRING.
10 S2N-STRING-X PIC X OCCURS 20.
05 S2N-W-X
10 S2N-W-9 PIC 9.
05 S2N-A PIC S9(15) COMP-3.
05 S2N-N PIC S9(15) COMP-3.
05 FILLER PIC X.
88 S2N-DECPNT-APPEARED VALUE "1".
88 NOT-S2N-DECPNT-APPEARED VALUE "0".
05 S2N-SIGN PIC X.

MOVE WS-ALPHA TO S2N-STRING.
PERFORM STRING-TO-NUMERIC.
COMPUTE WS-COMP-2 = S2N-A / S2N-N.

STRING-TO-NUMERIC SECTION.
MOVE 0 TO S2N-A.
MOVE 1 TO S2N-N.
SET NOT-S2N-DECPNT-APPEARED TO TRUE.
MOVE "+" TO S2N-SIGN.
MOVE 1 TO S2N-I.
INSPECT S2N-STRING TALLYING S2N-I FOR LEADING " ".
IF S2N-I > 20 THEN GO TO STRING-TO-NUMERIC-EXIT.
IF S2N-STRING-X (S2N-I) = "-" OR "+" THEN
MOVE S2N-STRING-X (S2N-I) TO S2N-SIGN
ADD 1 TO S2N-I
END-IF.
PERFORM UNTIL S2N-I > 20
MOVE S2N-STRING-X (S2N-I) TO S2N-W-X
EVALUATE TRUE
WHEN S2N-W-X IS NUMERIC
COMPUTE S2N-A = S2N-A * 10
ADD S2N-W-9 TO S2N-A
IF S2N-DECPNT-APPEARED THEN
COMPUTE S2N-N = S2N-N * 1O
END-IF
WHEN S2N-W-X = "." AND NOT-S2N-DECPNT-APPEARED THEN
SET S2N-DECPNT-APPEAED TO TRUE
WHEN S2N-W-X = " "
MOVE 20 TO S2N-I
WHEN OTHER
do some error handling procedure
END-EVALUATE
ADD 1 TO S2N-I
END-PERFORM.
IF S2N-SIGN = "-" THEN COMPUTE S2N-A = - S2N-A END-IF.
STRING-TO-NUMERIC-EXIT. EXIT.

If you save the source codes of STRING-TO-NUMERIC-WORK-AREA and
STRING-TO-NUMERIC SECTION in a library, you can include them in
your program by using COPY statement.

(The question was about converting an alpha field into numeric.)

Here's a simple solution where you don't have to even know where the
decimal point is. I wrote it online, and it's been over a year since the
last time I used cobol, but I have really used something like this:

01 ALPHA-FIELD PIC X(9).
01 A-CHARACTERS REDEFINES ALPHA-FIELD.
03 A-CHAR OCCURS 9.
03 A-NUM PIC 9.

01 NUMBER-FIELD PIC 9(5)V9999.
01 NUM-REDEF REDEFINES NUMBER-FIELD.
03 NUM-DEC PIC X OCCURS 9.
01 INDEX PIC 99.
01 IND2 PIC 99.

...
MOVE 0 TO NUMBER-FIELD.
MOVE 1 TO INDEX.
LOOP.
IF A-CHAR (INDEX) NUMERIC
COMPUTE NUMBER-FIELD = NUMBER-FIELD * 10 + A-NUM(INDEX)
IF INDEX < 9
ADD 1 TO INDEX
GO TO LOOP.

IF A-CHAR (INDEX) NOT = "."
GO TO DONE.
MOVE 5 TO IND2.
* THE LAST POSITION BEFORE THE DECIMAL POINT IN NUMBER-FIELD
LOOP2.
ADD 1 TO INDEX IND2.
IF INDEX NOT > 9
IF A-CHAR (INDEX) NUMERIC
MOVE A-CHAR (INDEX) TO NUM-DEC (IND2)
GO TO LOOP2.
DONE.

Sorry, that won't work. You will probably get a data fault because
there is a non-digit in the string (the decimal point). Also, the
single quotes are illegal in some COBOLs (they are not standard).
The following is legal and works in any COBOL 85 conforming
implementation.

01 ws-area.
05 ws-alpha PIC X(07) VALUE "67.2503".
04 ws-alpha-9 REDEFINES ws-alpha PIC ZZ.9999.

Then, the MOVE works (by using de-editing). However, the easiest way
if you have an up-to-date COBOL 85 is to use the intrinsic function
NUMVAL as in MOVE FUNCTION NUMVAL (ws-alpha) TO ws-comp-2.
