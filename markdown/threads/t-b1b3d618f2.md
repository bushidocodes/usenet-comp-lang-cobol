In our development of ScreenIO, and maintenance of dozens of 
systems for clients, we have had to work this way for the last
20 years.

I have at least 5 different compilers installed here.

There is usually no problem in installation, I simply point the
installation to different directories, (which most of the 
installers do by default anyway).  We tend to store all
software packages on the network, but not always - such as
when the license won't allow it.

Tricks:  Some arrogant compilers presume to set environmentals in
your autoexec.bat as if they were the sole reason you purchased
a PC.  Occasionally then mess around with your path.  (Love to
get my hands on the programmer who decided that was a good idea,
Fujitsu, this means you...)

Occasionally they set an environmental variable that conflicts
with another compiler (rare, but I seem to remember it).

Extract any of these path setting or environment settings
and put them in a separate bat.  Name the bat for the compiler.

Launch a dos box and invoke said bat. Then from
the command line invoke any compiler work benches (if
you insist on using the work bench - which I always avoid
if possible).  Anything invoked from said dos box
will inherit the environment from which it is launched.

We tend to have bat files witch do all the
drive mapping, environment setting, etc for each 
software system we work on.  That bat knows everything
needed by that system.  We set up our compilers to put
everything in the proper place: separate directories
for OBJs, LSTs, Copybooks, source, etc.  So often
Workbenches will, out of the box, put everything in
one huge hodgepodge directory. What a mess.

We've done this on a single windows box, or a network
(Novell, MS-Workgroups, Linux, etc.). Works very
well.

You talk about the windows environment, I guess.

With contemporary Disk-Size, the solution of choice is quite a bunch
of operating systems rather than a bunch of compilers.

1) Install a Windows and a Linux. The Linux Installation should
   have a few dozen GigaBytes available. Also install
   the Linux LoadManager (LILO) which gives you the choice of
   booting either Windows or Linux.

2) Boot Linux. Save your whole Windows-Partition (using dd ...)
   as a file somewhere into the Linux-system. That is the virgin
   Windows-Installation which you might need now and then.

Now, for all Cobol-compilers your want to installate:

3) Boot Linux.

4) dd the virgin Windows-Installation-file to the Windows-partition

5) Boot Windows.

6) Install Cobol-Compiler (or whatever).

7) Boot Linux

8) dd the Windows-partition to a new Windows-Installation-file.

Repeat that as often as desired.

One of the advantages is that you can avoid windows-installation-orgies
if you bomb the window-installation - which may happen now and then in
SW-Development. dd-ing the windows back is much faster than a new
windows re-installation.

LILO allows for more than Windows-Installation - so, for trying a few
Cobol-Compilers in parallel, a re-boot is all you need. It's just
a matter of careful Disk-Layout-Design.

But you said that you not even wish to reboot, which is understandable.
So the next step would be to try VMWARE - this should allow running
a few Windows installations at the same time - but I myself never
tried VMWARE so I cannot say anything for sure. Maybe someone else
is familiar with VMWARE.

Further advantages of this approach: If you destroy a
windows-installation,
you can go Linux, mount the damaged Windows-partition as well as the
disk-image of that windows (Linux can do such tricks), and make a
recursive
diff. That may be a tool to find out what went wrong.

Excuse all typos and English Language Errors.
