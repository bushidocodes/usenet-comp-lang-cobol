In CICS a task has a numeric identifier. I suppose a CICS TASK is most like
a thread, but I don't have enough UNIX experience to make that claim with
confidence. CICS runs multiple tasks concurrently, but it is like Windows
3.1 in that it is very close to cooperative multitasking. A task will
always have a Transaction ID associated with it and will usually have a
terminal-ID.

Most of the time I think of a transaction as a single message processed in
CICS, but IBM has a very specific definition of a CICS transaction. A
transaction is a 4-character identifier, associated with a task. In the PCT
(program control table) a transaction-id is always defined with an initial
program. During processing of the transaction, control will frequently be
transferred to a different program for the same Task/Transaction. CICS
allows multiple transactions with the same transaction-id to execute
concurrently, but each has to be associated with a different terminal-id and
a different task number. A CICS transaction can be invoked by typing its
4-character identifier on a blank screen. You can also start an asynchronous
transaction from any currently executing CICS program.

CICS programs are quasi-reentrant. When you write a CICS program, you are
really writing a subprogram for CICS. CICS manages transactions, and you
make CICS API calls from your program to obtain CICS services. There will be
only one copy of your CICS program in memory, even if multiple tasks are
executing that program concurrently.

If I've been too loose with these definitions, maybe someone who really
understands CICS internals can correct me here.

At the risk of being either too brief or too windy, here goes:

Program: a physical load module. It can be comprised of several object modules,
written in different languages. It is normally defined to CICS in the PPT table
or RDO definition as being written in the language of it's primary entry-point.

Transaction: Actually a "lay" term often used to represent the execution of one
or more programs to accomplish a unit of work from a "users" point ot of view.
eg. an inquiry transaction within an inventory application. A typical CICS transaction
executes one or more programs from start of message to it's output display using
a single task.

Task: A unit of work from the CICS point of view.
A task may be started in several ways, such as; a Transient data trigger level, a Start
command from another running task, a Time initiated event, or most commonly; by a
user "Transaction ID" (sometimes with data) entered from a CICS terminal.
When a task is started, a specific program is given control (as defined in the PCT or RDO).
The initial program may call, link (dynamic-call) or transfer control to other programs in
sequence while maintaining the task's initial common storage structures. Specifically, the
Task Control Area (TCA) and the various things chained from it.
Normal task termination occurs when the last (or only) program in sequence, issues a
CICS RETURN, typically after writing a response to the terminal that initiated the task.

A task is an executing instance of a transaction. all CICS task are
identified with a TASKID which is the key in a CICS table to a program
name. Each program in a CICS task is a seperate "Run Unit". There can
be many programs in a task. When a program/run unit is started it runs
until it ends by a RETURN, XCTL or CICS shoots it. At that time the
run unit (thread) dies. the next time the program is invoked a FRESH
copy starts up and runs etc.

CICS is a process, each task is a new thread and any programs LINKed
from a program are new subthreads.

I hope this is not too confusing.
