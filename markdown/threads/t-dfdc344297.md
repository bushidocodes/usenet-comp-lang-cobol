```
7/28/99

As you know, I am working on a conversion tool to convert COBOL datatypes to
IEEE datatypes so BASIC and C/C++ programmers can read COBOL data files.
(Free for personal use tool).

I have got the COMP (binary) datatypes working, and have run into a problem;
a problem which I will also have on the BCD and DISPLAY types.

Returning a long (32-bit) integer mantissa (the significant decimal digits),
we get nine (sometimes ten) significant digits. My test data includes data
up to 15 significant digits, and I think COBOL can handle sixteen or
seventeen.

A. Is this 'significant digit deficiency syndrome' going to be a problem in
the real world?

B. If so, what is the best solution?
  1. A 64-bit mantissa (can Visual BASIC handle 64-bit integers? How about C
compilers?)
  2. 64-bits the hard way: two thirty-two bit data items.
  3. A display (character) output, to be used with BASIC's "VAL" or C's
'scanf' functions?
  4. If oversize, use an IEEE double-precision real (float) for the value
(16 significant digits)
  5. Some combination of above? Fielder's choice?

Also, for the COBOL-enabled:
I have pretty-well exhausted my test data and would like some "real world"
data to test with. Yes, it can be "fabricated", but I've always found "live"
data includes some of those items which make our jobs so much fun: illegal
sign nibbles in BCD data, unexpected binary zero in the middle of character
data (i.e., leftover MOVE LOW-VALUES TO GROUP-ITEM). PICTURE S9(6)
containing "DOGGY ", etc.

If you can contribute a data file (does not need to be too big, please), the
matching COBOL FD, and some kind of file or printout showing me what the
values should be, I will be very appreciative.
(Also let me know if the data are EBCDIC or ASCII. This tool will also
convert the appropriate data from EBCDIC to ASCII if you tell it to).

Yes, you, too, get a credit. (No cash, no merchandise, which is exactly what
I get, too!). (Unless someone wants to do something commercial. Then it
ain't free).

Michael Mattias
Tal Systems
Racine WI USA
michael.mattias@gte.net
```

```
What is important is "more than ten siginificant digits."

As long as it's "more than ten" it really makes no difference if it's 18 or
31 digits .... Well, 31 could be a challenge, as even a sixty-four bit
integer only has 18 (sometimes 19) significant digits and I'd have to dream
up some way to do the math.

Now, how should I return these values? Display? Since a 64-bit float
(COMP-2) only has 19 digit accuracy (and more importantly, is subject to
implementor variations) that does not seem the best way. Inventing a 96-bit
binary kind of defeats the purpose of the exercise, as the user will end up
re-doing all the work the software does. (BTW, I know have the BCD and the
binary working and all I have left is the display input). (Except for
expanding the output digit count).

Well, inspiration should come....
```
