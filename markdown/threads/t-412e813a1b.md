```
>         01  100-TRUE                        PIC X(01) VALUE "T".
>         01  100-FALSE                       PIC X(01) VALUE "F".
>         01  100-END-OF-FILE-FLAG            PIC X(01).
>             88  100-END-OF-FILE             VALUE "T".

I always hated this construct.  While newer COBOLs allow one to set
100-END-OF-FILE to true making this even worse, but I never cared for
it. 

When you do maintenance, MOVE "T" to 100-END-OF-FILE-FLAG is just as
clear, and we don't have to double check the value of "T".

I can see an argument for using 100-TRUE.  If you try to move 100-YES to
the value, you get a syntax error.   But using the SET command is more
clear and we don't have to even look at working storage when we see the
code.

I just wish we could SET 100-END-OF-FILE TO FALSE before performing
until 100-END-OF-FILE.  But moving space or "F" to 100-END-OF-FILE-FLAG
is clear - I wouldn't double check their values while reading the code!
```

```
Just a couple of suggestions.  I generally use numbers as above to distinguish
paragraphs.  I  tend to prefix my working storage fields with WS (EX:

paragraphs in the code.

>        01  100-TRUE                        PIC X(01) VALUE "T".

>        01  100-END-OF-FILE-FLAG            PIC X(01).

If you initialize 100-END-OF-FILE-FLAG in working storage, then you will not
have to move 100-FALSE to it as below . 100-END-OF-FILE-FLAG 's value will not
change again until end of file is reached, so to 
move 100-FALSE to it on every read is redundant.
      01  100-END-OF-FILE-FLAG            PIC X(01)              VALUE "F".

I also agree with Howard Brazee it is much easier to read the code when at end
you move "T" instead of true.  When programs become quite lengthy, you don't
want to have to go thru the code over and over to identify what each field is
set to.  This isn't as important in code you write and maintain, because you
will be aware of you own practices.  But in the real world other people will
pick up your code to read and modify and you really don't want to piss off your
fellow workers.  .  Even if you are the only one to look at your code, think of
looking at it 6 months after you wrote and you have no idea what it's supposed
to do.  One simple rule to remember when you are writing code is to make it as
straight forward as possible.

Well so much for my 2 cents.  

As far as you write problem.  You stated that the key is defined as 9(3) in the
file.  In you FD you define it as X(3).  If the field is _10 or 10_ , the
spaces is not numeric.   The file is probably looking for 010.  Check your to
see if key is numeric.   You may have to do some redefines if this is the case.
```
