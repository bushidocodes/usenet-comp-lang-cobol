PERCobol:    2,945,000  miiliseconds    (hundredths of seconds on ACCEPT
always zero)
Realia v3.105:          170 milliseconds    ( 17,000 times faster)
Fujitsu v3.10:               50 milliseconds

On 32-bit systems you can increase the maximum size for ARRAY-SIZE.
I did that with Fujitsu to 9,999,999. This produced a DLL of size 80,037,376
bytes. I'm impressed that it complied and even more that Micro$oft's linker
could handle a DLL of that size. Even more impressive is that the program
actually runs. Here are the times in *milliseconds* to sort various array
sizes:
size 10,000            60 msecs
size 100,000     1,160
size 1,000,000  14,830
size 5,000,000  84,480

It is amazing that it sorts 5 million items in less than 1.5 minute.

This program with such a large table would not run in the java VM,
which is effectively a 16-bit system for data areas.

Now, one could argue that this test is a very atypical Cobol program,
that *real* Cobol programs do I/O and simple moves only. ACUCobol
makes such a claim and PERCobol will undoubtly too. One hears
the advise: don't use Cobol for non I/O bound applications,
use C++ or some other real programming language, but I just can't by
that.

Looking at the generated java code, it appears that the Cobol program
is NOT translated into jave, but that rather is interpreted by a generated
java program, which in turn is interpreted by the VM. I'm not sure if that
is the way to go.  The argument *for* a java solution is (apart from its
obvious fad-appeal) that the code will run anywhere (somewhat doubtful
as this has not been demonstrated - it is largely an unfulfilled promise).
A better way of making the code run anywhere would seem to insist
that all the vendors produce standard-compliant compilers without
extensions and funny stuff. (sigh)

Actually, PERCobol uses a runtime library much as any COBOL compiler
would use a runtime library.  We keep the code in the Java side as
close as possible to the original COBOL code in order to help ensure
accuracy.  Also, that allows us to optimize the runtime library on the
known basis of COBOL-like statements rather than generalized Java
statements.  (This form of optimization has only begun for the next
release.)  Some of the standard Java API's are _markedly_ slower
than the equivalent code written for more special-purpose applications.
If Java code not dependent upon our COBOL runtimes were produced,
the generated Java code would forever be dependent upon the
slower API's.  Currently, our COBOL runtimes use the slower runtimes,
but we are transitioning to our own special-purpose API's which are
_much_ faster; this transition is invisible to the user as it is hidden
behind a layer of abstraction, allowing code produced now to run
faster in the future with only a runtime swapout.
