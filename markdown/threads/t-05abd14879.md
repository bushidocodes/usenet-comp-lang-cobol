The format has probably been clobbered by posting here, and the Upper Case 
(shouting) looks unattractive to modern eyes, but the code itself should do 
the job. If it is going to represent COBOL in a public forum then it should 
look "attractive" :-) This is very subjective and certainly arguable, but 
I'll try and show what I mean. The idea is to make it easily readable 
(formatting helps) and to try and eliminate "ugly" constructs and redundant 
code.

*>  displays the count of bottles currently on the wall...
 100-do-count-lines section.
 100-dcl-000.
       move spaces to display-line
       perform 110-get-number
       string
             NUMBER-NAME
                   delimited by space
             BOB (1: boblen)
                   delimited by size
             OTW

 100-dcl-000.
        exit.

*>  displays the count of bottles remaining after one is removed
 100-do-remaining-lines section.
 100-drl-000.
       move spaces to display-line

       perform 110-get-number
       string
             NUMBER-NAME

             BOB (1:bob-len)
                   delimited by size
             OTW
                    delimited by size
                        into display-line
       end-string

       display display-line
       compute COUNTER = COUNTER + 1
       .
  100-drl-999.
        exit.
*>------------------------------------------------------------------------------
  110-get-number section.
  110-gn-000.
        if tens NOT = zero
            set atens-x1 to tens
        end-if
        evaluate TRUE
            when COUNTER < 1
                  move spaces to NUMBER-NAME
                  move "NO MORE BOTTLES OF BEER " to BOB
                  move "GO GET SOME MORE BOTTLES OF BEER..."  to TODAPIA
            when COUNTER < 2
                  move spaces to NUMBER-NAME
                  move BOB-1 to BOB
                  move 24 to bob-len
                 move "TAKE IT DOWN AND PASS IT AROUND" to TODAPIA
            when COUNTER < 20
                  set adigits-x1 to counter
                  move ADIGITS (adigits-x1) to NUMBER-NAME
            when OTHER
                  move ATENS (atens-x1) to NUMBER-NAME
                  if digits not = 0
                     set adigits-x1 to digits
                     string
                                  NUMBER-NAME
                                        delimited by space
                                  "-"
                                        delimited by size
                                   ADIGITS (adigits-x1)
                                        delimited by space
                                             into NUMBER-NAME
                      end-string
                   end-if
        end-evaluate.
        .
   110-gn-999.
         exit.

readability, it removes the need for the TRIM function and the multiple 
DISPLAYS with and without ADVANCING.

(perhaps someone could let me know?) but it DOES compile and run on Fujitsu 
NetCOBOL Version 9. Although it compiled (and ran) OK, the name COUNTER is a 
reserved word in some COBOLs so I changed it to ITERATION-COUNT.

this sounds like someone tried to redirect something to the device NULL. (It 
should be NUL:)

If the file that won't go away is called "nul" then something seriously 
wrong happened :-). Redirection to "null" or "null:" creates a file called 
"null" which is easily removed.

On WinXP redirection to "nul" or "nul:" does NOT cause a problem and it 
doesn't create a file, it just "burns" the output. (I just tried it to 
confirm this...)

I ran exactly the same tests on Win 7 and it does exactly the same.

So, if you have a file called "null" or "nul" (the latter appears to be 
impossible; I tried everything I could think of in Win 7 to create one 
without success) and the standard DEL (NOT "Delete") does not remove it it 
has to be because the file is protected.

Try this:

(On Win 7)

1. Open the command prompt and CD to the path where your file is. (Let's say 
the file is called "null"...)
2. Type the following at the prompt: ATTRIB NULL -R -A -S -H
3. press enter.
4. type the following: DEL nul*.*   (there is a risk here, but you said it 
was the only file in the directory... if that's right there is no risk.)
5. Press enter.

It should be gone. Try: DIR nul*.*  to see.

If it is still there it is because it has been locked by some long forgotten 
process (or a running system process) and you will need a tool like 
"unlocker" to remove it.

If the file is actually called "nul", amend the lines above as follows:

2. Type the following at the prompt: ATTRIB NUL -R -A -S -H
4. type the following: DEL nu*.*   (there is a risk here, but you said it 
was the only file in the directory... if that's right there is no risk.)

If you pipe it to MORE, it will fill a screen (page) and stop. (I think it 
says"Press any key to continue..." butI can't remember. Anyway, if you press 
Enter (or any other key) it will fill the screen again with the next "page", 
and so on.) This is NOT writing the output to a file; it is sending the 
output from the program directly to a screen utility called MORE, which 
places it (one screenful at a time) on the screen for you. (That's why you 
can't page back...)

If you take the scond option and redirect the output (this is NOT the same 
as piping it as we did with MORE; note that redirection involves a chevron 
(>) while piping involves a vertical bar ( |   pipe?)) the output is 
redirected to the file you have specified. You can then look at the file it 
created with any software that can handle text. The effect of holding down 
Enter will depend on the software you are using but I shouldn't think it 
would be useful with most programs :-).

Note that if you are using a program (like Notepad) which places the file in 
a window, you can then smoothly scroll through the file, (UP and DOWN) using 
the Window scroll bar on the right.
