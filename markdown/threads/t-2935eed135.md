Probably the first thing you need is a UNIQUE key to the record.  You'll likely find 
that you have more than one "John Smith" and maybe more than one "John Smith" with a 
wife called "Mary".

Trying to make an existing field act as the primary key in these circumstances might get 
you more grief.  Most database software allows you to have a primary key which is 
assigned automatically (autoincremented) when the record is created and is used only as 
a unique identifier.  All of the other indexing is done as secondary indices which use 
the primary index to find the record (or records) which match the value in field you are 
searching on.

> COBOL.

There's almost always a simple solution in COBOL. If ALL you want is to
combine fields for ISAM, the trivial way is:

01  Account-Holder.
   03  Account-Owner-and-Spouse
      05  Account-Owner
      05  Account-Spouse
   03  Account-Address
....
Where the SELECT statement would read:
   SELECT MYISAM-FILE ASSIGN TO (something)
     RECORD KEY IS Account-Owner-and-Spouse
     etc

To get all the "SMITH" people:

START MYISAM KEY NOT < "SMITH"
READ MYISAM-FILE NEXT.
PERFORM UNTIL Account-Owner NOT = "SMITH"
   IF Account-Owner = "SMITH"
      (process record)
      READ MYISAM-FILE NEXT
      END-IF
END-PERFORM.

Of course this fails miserably in the real world because:
1. One family may have multiple addresses,
2. Two or more families may have the same names, or
3. There may be multiple renditions of the same family names.
SMITH, JOHN            MARY
SMITH, J                    MARY
SMITH, JOHN            MARY LOUISE
SMITH, JOHN            M.L.
SMITH, JOHN            MIRY
etc.

Other observations:
1. A single field for street address is insufficient. Many people need two
or more lines (Apt 123, Suite A, Accounts Payable Department, etc.).
2. Your "State" and "ZIP" fields won't handle other country's postal codes.
You don't have a field for Country.
3. There is no need to include the dash ('-') (or any other constant) when
storing data.
4. "ZIP" is an acronym and should be in all caps. These can be inserted at
execution time, thereby saving storage on external media.
5. You defined your ZIP code storage as "X" (alphanumeric). This is
Good(tm). A sensible rule to follow is: If you can't do arithmetic on a
field, the field should almost never be defined as numeric (like account
codes, social security numbers, area codes, etc.).
6. ISAM files must have at least one unique key; you are permitted ALTERNATE
KEY WITH DUPLICATES, but the main key must be unique. We use fingerprints.

Floyd,

You've already had several answers to this, but let's get back to basics of
design, not programming. Back in ye olden dayes you used card indexes to do
the above, and these also developed into systems where you could poke a
knitting needle through pre punched holes at the top of cards to select those
that met your criteria.

What you are attempting to do wouldn't work with those cards, and as
illustrated by others you run into all sorts of duplication on names. Further,
I would question can you do it satisfactorily even with a DB.

As a practical example, I can think of a local charity, folks taking their
cats and dogs into hospitals/nursing homes (Pet Assisted Therapy). Given some
400 volunteers, approx. some 15 - 30% could be family members. They want to
distribute a quarterly Newsletter to all members, but given 'families',
restrict the mailing to one copy per family. I'm not aware of current details,
but there is an annual membership fee of $10, and using record linking, this
fee is reduced for family members.

The DB, (originally Paradox, but now Clipper I believe), has a record per
volunteer with ID cross-references to other family members. Only one record
has an active flag 'Send Newsletter'. (Sure they can goof and set the Active
flag for other family members, but this is minor).

The only safe way, COBOL files or DBs, is to have a unique record ID,
cross-referencing records where appropriate. I doubt the application you are
thinking of warrants it, but another alternative would be :-

- A Person record (e.g. name info, DOB, SSN etc.)
- An Address Record

with the first containing a 'key' reference to the second.

Jimmy, Calgary AB

COBOL does not allow more than one field to be a key for an ISAM file.
However some compilers allow split keys.

From MF help: The normal syntax allows a single data name following RECORD
KEY IS.
  RECORD KEY IS data-name-5
The extension to the language:
   RECORD KEY IS split-key-name-1 = data-name-6 [data-name-7 ] ...

RULES:
9. Split-key-name is a concatenation of one or more data items within a
record associated with that file-name. It can be referenced only in START
and READ statements.
10. The data items referenced by data-name-5 and data-name-8 and any
data-names referenced by split-key-name-1 and split-key-name-2 must be
defined within a record description entry associated with the file-name.

These data items can be defined as any category. However, the key is treated
as an alphanumeric data item by input and output statements for the file
connector referenced by file-name.
