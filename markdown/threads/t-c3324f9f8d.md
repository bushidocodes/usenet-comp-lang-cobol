the B(lank) so this is a case where going around the problem is probably
the shortest path. Redefine the compare key as pic x(04), occurs 5. Make
the display key a five element table of x(04) and x(02) and load it with a
small loop. It will probably execute more quickly than allowing CoBOL to
do it AND it will improve portability.

I've been down similar roads.

As you've stated the problem, it ought to work fine.

If you write MOVE KEY-FOR-COMPARE TO KEY-FOR-DISPLAY it will do what
you are expecting. BUT if KEY-FOR-COMPARE is actually a group item, it
won't!

identification division.
program-id. spreadout.
* Joseph's problem in Microfocus or DEC COBOL
data division.
working-storage section.
1 this-wont-work.
3 this-will-work pic x(20) value "abcdefghijklmnopqrst".
1 move-to-me pic xxxxbxxxxbxxxxbxxxxbxxxx.
procedure division.
begin.
move this-wont-work to move-to-me
display move-to-me
move this-will-work to move-to-me
display move-to-me.

The move "compare" to "display" is working correctly!

You made a logic error!

You need to rethink what the move does versus what you want it to do

You are asking move to put only 20 bytes into a 28 byte field

Move will put each byte in 1 at a time starting at the 1st to 20th byte

Answer: use an 05

Break down the 20 compare bytes into 5 groups of 4 bytes

Break down the 28 display bytes into 5 groups of 4 bytes each separated
by two blanks


01 KEY-FOR-COMPARE PIC X(20).
05 ONE-C PIC X(04).
05 TWO-C PIC X(04).
05 THREE-C PIC X(04).
05 FOUR-C PIC X(04).
05 FIVE-C PIC X(04).
01 KEY-FOR-DISPLAY PIC XXXXBBXXXXBBXXXXBBXXXXBBXXXX.
05 ONE-D PIC X(04).
05
05 TWO-D PIC X(04).
05
05 THREE-D PIC X(04).
05
05 FOUR-D PIC X(04).
05
05 FIVE-D PIC X(04).
