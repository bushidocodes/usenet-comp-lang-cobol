I'm no JAVA expert, but it's my understanding that JAVA is an interpreted language.
That's to say it's only when the source code is presented to the interpreter of
whichever environment under which the program is run that the source is converted
into a machine readable format.

This means that if you're running a JAVA program on a 64 bit machine, using a 64
bit interpreter then you end up with a 64 bit program. Use a 64 bit machine and a
32 bit interpreter then you'll end up with 32 bit object code on a 64 bit machine.

Presumably there are compiled versions of JAVA out there and conceivably you could
have a little fun trying to run 64 bit code on a 32 bit machine?

Mind you I could be wrong.

Java is an interpreted language, as is, for instance Acucobol. This means
that it is the particular runtime that defines whether the application is 8,
16, 32 or 64 bit.

You certainly have noticed the programming contest in this NG. Once I have
developed that one, it may be run using Acucobol 32bit Windows runtime, and
I can claim it to be 32 bit. I could also choose to execute it using
Acucobol 16bit Windows runtime, then it would be 16 bit. I don't know for
sure, but I suspect there might be a 64 bit Acucobol runtime on some Unix.
If I then executed this application there, it would be a 64 bit application.

But if your competitor is executing on Windows 98, their application will be
32bit as much as yours, because the Java interpreter on that computer may
not be more than 32bit capable.

When all this is said, I am most positive your application will outperform
the Java competitor, no matter whether they claim to be 64bit or not. Java
is well known for poor performance. So, so much for the benefit of 64bit
there. When I now have compared Acucobol and Java in this context, I would
also like to say that it is my experience that Acucobol outperformes Java as
well.

From my personal experience, I wouldn't even dream of using Java as a point
of sale application. POS systems demands a high uptime, fast execution and
the claim for portability is nonsense as a POS system has dependency on
hardware that makes it hard to port anyway.
