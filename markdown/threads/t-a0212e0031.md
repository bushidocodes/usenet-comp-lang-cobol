You will occasionally see differences between OS/VS COBOL (a both 68 and
74 compiler) and the various VS COBOL II / COBOL/370 / COBOL for MVS and
VM versions.

These differences exist only when fields involved will not pass the IF
NUMERIC class test. This includes blanks in display numeric fields, and
signed values in unsigned fields.

VS COBOL II R3 and later products introduced a trinary option (yup 3 way)
that affects the behavior of numeric display fields, and comp-3 fields.
This option has no effect on valid data, but will cause differences when
the data is not valid. The particular option is called NUMPROC, and has
values PFD (Preferred Signs), NOPFD (No preferred signs), and MIG. All of
these options attempt to force valid signs in output fields (matching
picture). The difference in behavior is principally what is done in
processing fields before the operation (move or arithmetic) is commenced.

A detailed description of the various eventualities takes about 1 hour in
"Planning COBOL MIGRATION" (a stand up class that I wrote and conduct),
and even that description is not 100% complete.

The important thing is that results are predictable and correct across all
of the compilers under discussion so long as the data will pass "IF
NUMERIC"

In addition a long existing compiler option Zero When Blank (ZWB) can
also affect results involving display numeric fields.

For code being moved forward from OS/VS COBOL the best results seem to
result from using ZWB, and NUMPROC(MIG).

Look at the code patterns from both compilers and you will see what is
going on. BTW, you patterns will vary, the new compilers are very clever,
but the description should hold close to the facts.

I think Harvey Bookman documented this pretty well in his book "COBOL
370". We encountered problems related to this where I work.

There is a new compile option called NUMPROC (numeric processing). It
can have three values: NUMPROC(PFD), NUMPROC(NOPFD), or NUMPROC(MIG).

NUMPROC(PFD) means numeric processing with preferred signs. The compiler
generates code assuming that the programmer will always manage signs
correctly. This includes generated CLC opcodes (Compare Logical
Character) for comparing packed-decimal numeric fields, because CLC is
faster than CP opcodes (Compare Packed). Of course, if your signs are
not compatible, you will not get the results you expect! For example, if
you compare a signed zero field to an unsigned zero field with
NUMPROC(PFD), COBOL will tell you that signed zero is less than unsigned
zero. They are not equal.

NUMPROC(NOPFD) means numeric processing with NO preferred signs. This
means the compiler will do the maximum amount of sign fixup on your data
before generating CP opcodes for packed-decimal numeric compares. This
code will execute a little more slowly, but will prevent a great many
S0C7 abends for non-numeric data. The results you get for numeric
comparisons and arithmetic calculations will probably be more like what
you expect.

NUMPROC(MIG) means numeric processing in a migration mode. This is
supposed to be similar to the amount of sign fixup that IBM OS/VS COBOL
does, but Harvey Bookman says it is not exactly equivalent to old COBOL.

Generally, your tech services department will supply you with compile
procs that have certain defaults built in. If they use NUMPROC(PFD) I
think you will have problems. You should probably override all your
compiles with NUMPROC(NOPFD) or NUMPROC(MIG), or better yet, use the
PROCESS or CBL option to embed compile-time options in your source code.

I believe there is also an installation option with COBOL II to tell the
compiler whether it should accept A, B, and E as packed-decimal or zoned
decimal signs. Normally, only C (positive), D (negative), and F
(unsigned positive) are used.

In any event, COBOL II will ALWAYS generate the correct sign in the
target field of a MOVE statement or calculation, no matter what NUMPROC
option you use. But if you cannot guarantee what sign will be present in
the source field of a MOVE or computation, or in the input data, then you
should seriously consider using the NUMPROC(NOPFD) compile option. I
think it will save you a lot of grief.
