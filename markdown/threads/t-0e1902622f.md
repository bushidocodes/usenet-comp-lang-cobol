A number would tell you for sure, but it's more likely to have spaces.
Programmers usually don't use numbers. It's also possible for the
first line to contain a compiler option such as one of the following
starting in column 7, usually, or 1. If you see one of these, you know
the format. $SET is Micro Focus; @OPTIONS is Fujitsu. Other compilers
will have similar options. IBM doesn't support free format.

$SET SOURCEFORMAT"FREE" 
@OPTIONS SRF(FREE,FREE)
$SET SOURCEFORMAT"FIXED"'
@OPTIONS SRF(FIX,FIX)

>Does anyone know a trick to distinguish a fixed format cobol program
>from a variable format one ?

Two passes.

>For example, if the column 1-6 only contains spaces, then it is
>variable format ?

It's probably fixed format, not not necessarily.

>Or a way to determin the dialect (assuming each dialect uses only one
>format)?
>Or is it impossible, and the user must specify the format manually ?

Compilers require the user to specify, either in the compile script or
at the top of the program. The default is fixed.

A few quibbles:

The right-hand end of the COBOL source area for "*fixed" was by convention
usually in Column 72, but is actually defined by the individual implementor,
as early as the '74 standard.  Your "*fixed" and "*var* seem to me to be one
and the same, as far as the COBOL standards go.

The distinction between Column 8 ("margin A") and Column 12 ("margin B"),
which you don't mention, was significant for '74 and '85 COBOL, but has been
erased in the '02 standard.
.
Your "*free" is not standard COBOL.  Personally I don't know of any
implementations that use this form.

What you describe as "*fsc-free" and unique to Fujitsu is actually what is
mandated as "free-form source format in ISO/IEC 1989:2002, the 2002 standard
for COBOL.   Note that the "indicators" are also floating in that format -- 
*> for comment indicator, >> for compiler directive. "- or '- for literal
continuation indicator, and >>D for debugging indicator.

The default format in 2002 standard COBOL is still fixed-form reference
format:  sequence (which need not be a number) in columns 1-6, indicator (if
fixed) in column 7, program-text area starting at Margin A in Column 8 and
extending (as for '74 and '85) however far the implementor defines.

There's also a Unisys MCP (descendant of the Burroughs Large System)
extension that they call "free"; let's call it "*mcp-free":
    columns 1-6 :  sequence number area
    columns 7-72:  program text area (debug indicator not allowed)
    columns 73-80: the traditional "identification" area.

Thus:


>    - if the first 6 columns of the file contains a sequence number,
> assume it is fixed

Needn't be a number in '85 COBOL or in fixed-form reference format in '02
COBOL.  The '74 standard indicates that it contains digits.

standard COBOL is to ensure that your COBOL is as specified in the standard,
and that includes the source formats!    Since free-form reference format
wasn't approved as standard COBOL until 12/1/2002, it's not a surprise that
a number of compilers implemented according to previous standards don't
include it!

Your "*fixed" and "*var", as described above, both fall within the
description of the "reference format" for the '74 and '85 standard and for
"fixed-form reference format" in the '02 standard, with "*var" having the
more accurate description, I think.

Your "*fsc-free" is virtually identical to the "free-form reference format"
defined in the '02 standard.   To get there you've got to have something
like ">> SOURCE FORMAT FREE" as the first line of the compilation group.
That line may be in either fixed form or free form reference format, by the
way.

But "*free", as you have described it, is a vendor extension and is not
standard COBOL.

    -Chuck Stevens
