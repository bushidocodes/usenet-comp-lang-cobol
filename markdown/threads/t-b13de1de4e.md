```
On June 20th, 2018 a decision was taken that web servers running TLS 1.0
(Transport Layer Security) would have their pages marked as "Not Secure"
when accessed with a Browser.

This has currently affected over 850,000 servers around the world. To be
fair, they had two years to fix it but many of them simply can't.

Nothing much happened because the major Browsers didn't implement it
until a week ago, on the 2 year anniversary.

The result is that the Web Site I carefully built over many years and
provided certification for, now displays "Not Secure" even on pages that
are transported as HTTPS: and certified.

There's nothing I can do about it (maybe move the site to another host,
but the current one has been so good over more than a decade and even
implemented COBOL COM components for me, that I just wouldn't do that.)

Today he has offered to move the site to another type of (more
expensive) server that doesn't have the problem, and will do it at no
charge to me. But it is a fair bit of work and I think the least I can
do is to ensure that the footprint on the new server will be as small as
possible.

So, over the coming couple of weeks, I will be looking to completely
refurbish the PRIMA Domain. Some of the historical stuff for COBOL will
be ripped out, some of the free components will disappear, and some of
the current pages will be changed.

There are pages there from the original site in 2009 that will now
disappear. It's a bit emotional for me...

I considered just removing the lot, but we are still doing some
migration business (very little... most people migrated from COBOL some
time ago or found other solutions. However, currently, we are the only
game in town that can offer a solution for moving PowerCOBOL into
Windows and there is still the occasional site looking to do that.)

The site has also provided some places and things of interest to people
with a COBOL background.

I have used it to describe my own philosophy of component based design
using COBOL and also to promote and explain OO COBOL where I can.

For Network development I still believe that things need to be based
around Objects and Layers and COBOL can be used for that, although the
modern OO languages are more facile.

Get the free S2N component while you still can.
https://primacomputing.co.nz/PRIMAMetro/S2NTestServer.aspx

The PRIMA site has had over 65000 hits since it was implemented in 2009
but I believe its time is coming to a close (as indeed, is mine... :-))

The free COBDATA tool will still be available on the "new" site and I
will fix the obscure known bug in it (to do with deep levels of
REDEFINES) as soon as I get a chance. The engine is running correctly in
some of our other tools, so the bug is a low priority.

I added some facilities to the site like a lounge and a chat room but
they have not been utilized so I'll remove them unless it is really easy
to transfer them.

I have noticed the decline in access to this newsgroup which was also
once a lively place, but the facts are that people don't have COBOL
questions because it is not being widely used for production systems as
it once was.

COBOL has really become a "hobby programming" language.

Never mind. It was a big part of my career and I have no regrets about it.

I am sorry to see the web pages go (I sweated blood to build them), but
everything has its time and I had a great deal of joy out of it.

I'll let you all know when the "new" streamlined site is running and
I'll reset all the counts. How long I keep it running will depend on the
levels of access to it.

For me, this is the end of an era.

Pete.
```

```
Hi Federico,

good to see you are still going... :-)

If your component is written in COBOL there is no problem with making it
RESTful; you can use a temporary attached local file to save state
values. This is easier than trying to use the Web facilities like
storing variables in headers or temporary cookies, but remember that
they only last until the component is instanced again. (In effect, every
time you create an instance of it, it is like starting a new thread...)

There is absolutely no problem in invoking your COBOL COM component from
C#, or vice versa if you have COM components written in C# that you want
to use from COBOL.

I used to use Fujitsu NetCOBOL for Windows (Native code compiler) to
write COM components in COBOL, because it supports OO COBOL and you need
that if you want your component to be COM visible. (Fujitsu support COM
MUCH better than Micro Focus because they provide a COM/OLE Class that
takes care of the tedious housekeeping for you. However, you CAN write
COBOL COM components in Micro Focus, it is just a lot more work.)

Nowadays I write COM components in C# and they target the .Net Framework
(CLI instead of Native code, but it can be 32 or 64 bit and it is FREE!)

Mail me privately if you get stuck.

Cheers,

Pete.
```
