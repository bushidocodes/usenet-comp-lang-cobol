Since you told pstat_getproc() that you didn't have room for any
structures in your array, it didn't fill any in.

If what you get back still looks wrong, you might have alignment
issues.  I'd put a "$set align'4'" at the top of the program.

right size.  A quick test on an HP-UX 11.11i machine here seems to
show that it should be 768 bytes in 32-bit mode and 1176 bytes in
64-bit mode.

I suspect the main problem is that C leaves the size of various data
types up to the implementation, whereas COBOL specifies just how
large a data item with a picture clause like "pic x(4) comp-5" is.

Thus when a C program that uses struct pst_status is compiled in
32-bit mode, most of the integer fields are 4 bytes; but when it's
compiled in 64-bit mode, most of them are 8 bytes.  When that program
is linked, it's linked to either the 32-bit or 64-bit version of the
library that provides the pstat_getproc function, and that library
will have the matching structure definition.

When h2cpy processed the C header file, I bet it assumed that
various integer fields were 32-bit, so it created COBOL data items
with "pic x(4) comp"; but since you're compiling in 64-bit mode,
you'll use the 64-bit COBOL runtime, which is linked to the 64-bit
system library that implements pstat_getproc, which expects the
64-bit version of that structure.

The fix is likely to be going through the generated copybook and
changing a lot of "pic x(4) comp" clauses to "pic x(8) comp".

Also, it seems that h2cpy missed a field somewhere, because you're
getting 764 for the length, but C reports a length of 768 in 32-bit
mode.  Probably that's a bit of padding somewhere.  In fact, it's
very likely at the end, generated by the C compiler to satisfy the
"pad for alignment" rule for C structures.

I can't spend any time investigating this further today and tomorrow,
unfortunately, but I might be able to get back to it next week.
Creating the right COBOL copybook should just be a matter of tweaking
this and that until it matches the C definition.  (There are some
tricks like binary-searching to find out where field offsets no
longer match, that can speed that up.)

SOLUTION:


When using h2cpy you need to include the -D_PSTAT64 option on the
command line to force h2cpy to build a 64-bit version of the pstat
libraries.

h2cpy /usr/incluse/sys/pstat.h -I /usr/include -D_PSTAT64

Once this is done, everything works as expected.


Thanks to the folks at MF SupportLine for helping me to figure this
out.
