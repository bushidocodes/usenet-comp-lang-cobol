```
The ZZ9 field is not a number.  You can move a number *to* it, and
then it will contain the edited (formatted) text associated with that
number (for example with the leading zeros supressed), and you can use
that to display or output the formatted value, but not as input to
arithmetic.

Usually you keep the number you're working with in a plain numeric
field (for example the 999 field), and then move that to an edited
field when you're ready to output the value.


   05  Qty pic 999.
   05  Qty-Out pic z99.
   ...
  add 1 to Qty.
   ...
  move Qty to Qty-Out.
  Display Qty-Out.
```

```
I should add that validation of the contents of a numeric-edited data
item is problematic, the NUMERIC class test can't be used for this
purpose.  Hopefully, one would get a data exception if the MOVE was
unsuccessful, but if the contents were partly valid, the invalid parts
might just be ignored with an inappropriate result.  The intrinsic
functions NUMVAL and NUMVAL-C also do de-editing, but again have no
standard specified facilities to indicate validity.  The 2002 standard
introduced TEST-NUMVAL and TEST-NUMVAL-C, which do validate the
contents, but may not have been provided with your compiler.

Best not to do this and follow the previous advice and only use
numeric-edited numbers for screen and report output.
```
