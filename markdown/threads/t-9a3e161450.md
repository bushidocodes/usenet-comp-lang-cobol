You may be able to do something like:

    CALL "system" USING "SET MY_ENV=ZALECS_PC" & LOW-VALUE

However, issuing this command is unlikely to actually be what you want
to do, in that it probably won't do what you think you want.

The COBOL way of changing the environment variables is:

     DISPLAY "MY_ENV"    UPON ENVIRONMENT-NAME
     DISPLAY "ZALECS_PC" UPON ENVIRONMENT-VALUE

You may need to have SPECIAL-NAMES entries to cater for this.

But, due to the way that environment variables work, you may find that
this is not what you want either. In particular if you did:

    C:> SETZALEC
    C:> SET

The setting that 'SETZALEC' did will not show in the 'SET' listing.
When you run a program it is given a copy of the current environment
and any changes made are discarded when the program ends.

Changing the environment won't work from within a Fujitsu COBOL
program.

The runtime creates a copy of the existing environment when it starts
up and creates a new process.  That environment goes away when a STOP
RUN is encountered and the process is terminated.

The way to do it is to to start the cobol app from a batch file and
then have the batch file execute a second batch file (which you create
in the cobol app) after the cobol app finishes.

Lee

> Changing the environment won't work from within a Fujitsu COBOL
> program.
>

It depends on how you go about it.


> The runtime creates a copy of the existing environment when it starts
> up and creates a new process.  That environment goes away when a STOP
> RUN is encountered and the process is terminated.

That is absolutely true, if you use the local EV space in the inherited 
environment to store your EV.

However, the Windows OS has a number of different spaces for EVs. There is 
the temporary local one which is part of the environment inherited and used 
by the COBOL run time, but there are also USER and SYSTEM spaces which have 
more permanency.

If you want the EV to persist, you need to store it in one of these spaces. 
One way (and it is the way I use) to accomplish that is by using a Windows 
Script File. (For people who haven't cottoned to it yet, Windows scripting 
has replaced the DOS Batch facility, is infinitely more powerful than the 
old Batch files, and kind of does what JCL Procedures do on mainframes.) 
Windows scripts are contained in .wsf or .wsh files and these are executable 
just like.bat files. In effect, these are the Batch files of today.)
>
> The way to do it is to to start the cobol app from a batch file and
> then have the batch file execute a second batch file (which you create
> in the cobol app) after the cobol app finishes.

That is certainly one way, although it is a little convoluted.

Until I moved off COBOL and thereby gained more detailed understanding of 
what CAN be done, I probably would have done it like that too.

Nowadays, I don't do it that way as it seems unnecessary to constrain apps 
into starting from a batch file and then spawn another, not to mention the 
internal logic that must be added to your COBOL app so it can write the 
batch file which will be started after it completes.

Better by far just to write the EV into a persistent area. No generation of 
batch files required, no requirement for your app to be started from a batch 
file (in fact the technique works fine for PowerCOBOL as well, where there 
is a GUI interface and starting from a batch file is not usual.)

I checked the Fujitsu docs and samples to see how they recommend doing this 
and it is pretty awful.

Fujitsu Sample32 uses the hideous "StartProcessA" API interface to execute 
the batch file and they give no example that people could use to get around 
the need to generate a DOS batch file for setting EVs. (You also need two 
different versions of the interface depending on whether you are passing 
parameters to the app or not - it is just silly, really.)

I am a fan of component based programming and it just so happens that the 
Windows OS comes with hundreds of components already programmed into it. 
There is a COM component for doing a ShellExecute and it is very much easier 
than trying to pretend you are a VB program and interface directly through 
the WinAPI to StartProcessA. The advantage of a COM component is that it can 
be easily invoked from both Micro Focus and Fujitsu COBOL

I have posted a new thread describing all of the above with sample code, for 
people who can use it.

Pete.
