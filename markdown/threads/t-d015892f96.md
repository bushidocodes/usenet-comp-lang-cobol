```
max Boettner wrote:
> 
> How can a cobol program pass a variable to JCL, so that JCL can use the
> content?

The simple answer is, you can't. 

JCL is designed to pass information to the JOB steps. Unlike
VSE where you could use the "Communications Region" to pass
info to the next or subsequent executing program(s), this is
not available natively in "MVS". 

But let us suppose that you really need to do this, and
information that is currently part of the now executing JOB
is not needed. What you can do now is, write to a DD coded
as follows:

//SUBMIT  DD  SYSOUT=(A,INTRDR)

You will need a SELECT/ASSIGN for the DDNAME of SUBMIT and
an FD that will be writing RECFM=F/FB LRECL=80 records.

What will happen here is (assuming you aren't barred from
this), you will now submit a whole new JOB that will inherit
the current JOB's security profile (assuming you don't have
a SUBMIT exit in your shop that does other things). This way
you can pass all the parameters in JCL that you want by
having your COBOL program "generate" the whole JOB stream
(or however much of it you want while "hard-coding" the
rest).
```

```
Are you sure you don't mean the other way around - pass a variable in JCL
to a program? Nevertheless, if you mean it the way you have written, this
is not possible. This is not a cobol restriction, it is not possible from
any language. You can set a return code and execute (or not) JCL based on
it. A program can create new JCL to be executed as a separate job. You can
pass data through a file. That's about it. What is it you are wanting to
do?
```

```
You want COBOL program A in Step1 of Job JCL1
to pass a variable to Step2 of Job JCL1?

Method #1
Have COBOL program A in Step1 of Job JCL1
create a sequential dataset DATASET.STEP1OUT.

Have Step2 of Job JCL1 read in DATASET.STEP1OUT.

Method #2
Have COBOL program A in Step1 of Job JCL1
create a Parmlib member PARMLIB(STEP1OUT).

Have Step2 of Job JCL1 read in PARMLIB(STEP1OUT).

Method #3
Have COBOL program A in Step1 of Job JCL1
update a Proclib member PROCLIB(STEP2IN).

Have Step2 of Job JCL1 execute PROCLIB(STEP2IN).

Normally, I would try these theories before posting,
to resolve any technical issues, but I cannot do that right now,
so you'll have to work out the bugs yourself.
```
