>mechanisms.

You should also know that it makes very little difference. Version 4
supports structured COBOL event coding just as Version 3 does.

I migrated everything from Version 3 to Version 4 without incident when
Version 4 was released. The Object Oriented extensions in Version 4 are
OPTIONAL and it is certainly misleading to suggest that a "completely
different" way of coding is REQUIRED. (Some of the Objects and Methods
provided had their names changed in Version 4, but with a global edit the
impact is minimal.)

Now in Version 5, I DO use the OO extensions for building COM components. (I
am also experimenting with OO COBOL in COBOL97, as I believe this will be
the future direction of COBOL).

Building ActiveX (COM) components is the ONLY place in PowerCOBOL where OO
COBOL is REQUIRED. Even then, you can avoid using OO if you simply "wrap"
structured code in a PowerCOBOL COM Wrapper.

(I am not suggesting that you SHOULD "avoid OO COBOL", merely that you CAN.)

Pete.
