Possibilities come to mind:

Given definitions of:

01  WS-VARIABLES.
    05  AAA  PIC 9(5) VALUE ZEROES.
    05  AAB  PIC 9(5) VALUE ZEROES.
    05  AAC  PIC X(5) VALUE SPACES.

01  WS-CONSTANTS.
    05  LITERAL-A  PIC X  VALUE 'A'.
    05  LITERAL-B  PIC X  VALUE 'B'.
...
    05  NUMERIC-HIGH-VALUES-5 PIC 9(5) VALUE 99999.
then:

1)  MOVE NUMERIC-HIGH-VALUES-5 TO AAA will give the equivalent of MOVE 
99999 TO AAA and fill the field with a value greater than which the field cannot 
hold (the functional definition of HIGH-VALUES); this can be tested in the 
code later with IF AAA = NUMERIC-HIGH-VALUES-5.

2)  MOVE HIGH-VALUES TO WS-VARIABLES will populate AAA with 
x'FFFFFFFFFF'... and everything else in WS-VARIABLES, as well; your 
compiler will not, probably, allow the statement of IF AAA = HIGH-VALUES.

One can also REDEFINE the AAA field:

05  AAA PIC 9(5) VALUE ZEROES.
05  AAA-ALPHA REDEFINES AAA PIC X(5).

... and then MOVE HIGH-VALUES TO AAA-ALPHA.  Again, there might be a 
problem with the compiler accepting IF AAA = HIGH-VALUES but a work-around 
would be:

IF AAA NOT NUMERIC
    IF AAA-ALPHA = HIGH-VALUES
        PERFORM EOJ-RITUAL THRU EOJ-EX.

DD
```

Others have responded and indicated that you need to have an alphanumeric
"variation" of the field to move high-values to.  However, if you really DO
want to get the highest (or lowest) algebraic value in a field, you might be
interested in the new

   HIGHEST-ALGEBRAIC
    and
   LOWEST-ALGEBRAIC

intrinsic functions that are introduced in the draft COBOL Standard.  If you
think these would be useful to your development environment, you might want
to contact Liant about their adding these features to their product
(when/before they add support for the full revision).
```
