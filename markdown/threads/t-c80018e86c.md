As an OO COBOL user for nearly 20 years now, and seeing some fairly direct 
criticism in the "Phase 1 - Rosetta bottles of beer" thread, I thought it 
might be interesting to see how people feel about OO COBOL, and why they 
feel that way (particularly if they have never used it, although not having 
used something doesn't preclude people having an opinion about it :-))

Jimmy Gavan posted an OO COBOL example of the well-known "Bottles of beer" 
problem.

To an "ininitiated" (read "unfamiliar with OO COBOL...") eye this code looks 
like a severe departure from Standard procedural COBOL. It provokes a strong 
reaction from some people.

I don't intend to criticise Jimmy's code in any way, here, but the following 
factors should be kept in mind by people used to standard COBOL, when they 
review this code.

1. It is written using Micro Focus language extensions which will be 
unfamiliar to many people.
2. It is just one interpretation of an OO COBOL solution to the problem.
3. It was not intended to be a textbook example of "how to write OO COBOL"
4. It DOES demonstrate at least one useful facility of OO COBOL (recursion).

History has shown that ANYONE posting code here will be subject to criticism 
(not necessarily a bad thing... as long as you are not a sensitive flower or 
easily discouraged) and it is the nature of programming that programmers 
will differ in the way they do things.

However, on this occasion some of the objections don't seem to be about the 
code, but rather the whole concept of Object Oriented Programming in COBOL.

As someone who has walked that path I'd like to offer the following:

1.Most people who write OO COBOL are NOT doing so because they want to 
emulate what is available in other OO languages.
2. Some of us who seized OO COBOL when it first became readily available 
(and Jimmy and myself are 2 survivors from that camp) saw it as bringing the 
benefits of Object Orientation without having to pick up another language. 
(For myself, I can only say I was mistaken about that; I never really 
grasped OO COBOL until I taught myself Java.)
3. Depending on what you want to do, OO COBOL can be an excellent way to do 
it.

I used OO COBOL to write components for many years. I'd probably still be 
doing it if events hadn't transpired to make me think again about using 
COBOL as a development language, at all.

Those events DID transpire and led to me moving away from COBOL. So, at 
least for me, the question posed by the topic should have the "OO" removed 
from it.

For batch processing there is no doubt that COBOL is an excellent choice, 
now, and for at least the immediate future. (The question THAT raises is 
whether batch processing is an approach that will proceed far into the 
future or whether it will be subsumed by online systems and data 
warehousing, running on more powerful and distributed processors.)

For network processing, the future is in Objects and Layers. SOME kind of OO 
processing needs to be adoted to move into this camp.

The success of languages like Java, C++, C# etc. is NOT because they are 
good for batch processing...

Howard posted an excellent response in the thread referenced above, where he 
pointed out that OO COBOL can be used to leverage existing COBOL into the 
world of Objects and Layers.

He is absolutely right. Using OO COBOL to wrap existing procedural code can 
mean that much of a company's existing critical code base can be brought 
into the world of Objects and doesn't need to be re-developed. (At least, 
not right away. Using OO COBOL can buy you time.)

If nothing else, at least it provides a strategic option; you MAY choose to 
re-develop in a new llanguage or using a new language, but you don't HAVE 
to.

I have always advocated not throwing the baby out with the bath water, and 
spent several years developing tools to assist migration from Procedural 
COBOL into the world of Objects and Layers using OO COBOL, so that COBOL 
Objects can interact with Objects in other languages as full citizens in the 
Brave New World of the Network.

So, the first part of my personal opinion on the topic would be: "Yes, OO 
COBOL is not only relevant, but very useful, for extending the life of  an 
existing COBOL code base."

Wrapping existing code with the dozen or so lines of OO COBOL needed to 
create Classes in COBOL provides advantages far beyond the approach of 
simply re-compiling for .Net.  (Succesful migration is not just about 
platforms, it is about architecture...)

The second part of my response is more problematic.

It has to do with code, and tools, and development.

We've seen the reponses to the OO code Jimmy posted.

It DOES look "strange" and "non-COBOL like" when you first encounter it.

And there is a plethora of jargon surrounding OOP, just as there is in other 
aspects of technology. But it is NOT "unknowable" or "mystic" in a 
scriptural sense.

OO COBOL (although it varies slightly in different implementations; I've 
always thought the Micro Focus implementation is a bit "closer to the metal" 
than the Fujitsu one, for example, but that is just a personal opinion and 
arguable...) DOES allow you to implement the key concepts (like Classes, 
Inheritance, Interfacing, Polymorphism, etc.) of an Object Oriented 
Language.

The question this raises is why would you WANT to? (That one is answered 
above) Given that you WANT to, why use OO COBOL when there are other OO 
languages which were designed for that purpose?

This is a much stickier wicket.

I guess the immediate response would be: "Because I can get the OO leverage 
I want for the Network, without having to learn a new language."

This is probably true for some people; it wasn't for me. That particular 
promise was not fulfilled by OO COBOL.

These days I use a language designed for OO (C#) and find it much more 
powerful than COBOL ever was, so I could say: "No, for OO development, OO 
COBOL is NOT relevant today. There are much better tools available for this 
job."

BOTTOM LINE:

As you have seen from the discussion above, I find myself in a "yes"/"no" 
situation and the answer is not totally clear.

If OO COBOL disappeared tomorrow, would I miss it?  Yes, definitely. But not 
for new development.

Do I still build components in OO COBOL? No. I still use component based 
design and development, but not in COBOL.

Am I sorry I moved away from COBOL as a development language? No. I wish I 
had grasped the nettle and done it sooner.

This post has been made in order to explore some of the implications of 
using OO COBOL a bit more than superficially, and to hopefully show that 
using it is not about trying to write C++ in COBOL.

I'd be interested to hear what other people think.

Pete.

Hey Pete -
I like your thinking, and decided to go ahead and take a few minutes 
and write up a response to it. This is, of course, my opinion, not 
stone cold facts.
First, Objects. Objects are great. For Design and depending upon the 
language and implementation, not too bad for implementation. But when 
you take them to extremes, you wind up in some “interesting” situations.
SmallTalk is probably the most object-oriented language around. You can 
design correct SmallTalk programs with object layer and object layer, 
and get reliable behavior from them. Methods and data are wonderfully 
encapsulated, and you just don’t have problems with buffer overflows 
and such.
All the IBM “VisualAge” products were based upon a SmallTalk front end, 
from VisualAge COBOL to VisualAge Generator. (There was one exception I 
know of - VisualAge RPG.) You could design some nice GUI’s with no real 
programming, and events were encapsulated in methods that were easy to 
code with program snippets.
But- when you get down to the non-visual program logic, essentially, 
read a record, do something with it, and write it back out, or doing 
intense calculations, SmallTalk became a serious nuisance. You can not  
just read a record for instance. You wind up asking a method to 
populate a collection, or some other abstraction that really is more of 
a interference than a help.
So what all that means to me is that Objects are great for design - but 
like normalized databases, must usually be “denormalized” 
(de-objectified?) to actually do the work efficiently. I think most 
people agree with that point of view, at least people who have done 
large projects (especially real time projects) tend to feel that way. 
It really works, and you find that U.S. Government Defense projects are 
almost always specified to use something along these lines these days. 
For example, the High Level Design Document might be specified to be 
compilable.
Languages like Ada really show this mixture very very well. Highly 
Object oriented, both in design and implementation at the top level 
design and implementation. Very procedural at the low levels when you 
need it to be. In a controlled and usable manner.
Of course, at this point, most people ask what the heck these “Objects” 
are anyway… as they appear to be mysterious things out there that 
refuse to reveal their innards to a normal person’s inspection.
Let me work up to this a bit - in the bad old days - say the 1950’s - 
programming was a wild and wooly adventure. Assembler Languages were 
all the rage, and for good reason - compilers usually had to load 
stages from tape. Assemblers tended to fit into memory!   A typical 
COBOL compile job could take a long time to run - hours even depending 
upon how many passes were needed.
Programmers learned to do things like use registers well and write 
inline code with well defined branch operations. Remember testing the 
negative options first? It really helped on some machines.
But anyway, things like good memory management, how to optimize code, 
good branch management (how to use GOTO, B, BAL, BAS, JSCI, and other 
processor dependent techniques were learned, and more importantly 
documented. We also learned how to write codes in modules, and the 
basics of encapsulation. (JSR/JSCI instructions on some machines would 
save all the state information, jump to and execute a subroutine, 
return to the calling routine and restore the state. State usually 
being registers, but sometimes more.)
In the next stage of evolution, we learned how to do the same thing 
with high level languages like COBOL. This built of course, on the 
stuff the Assembler people had already learned. It was very common for 
instance, to have the COBOL compiler generate assembler code, which was 
then studied intensively to figure out the best ways to do things. Sure 
a lot of this was driven by hardware limitations of day, but the end 
result of this was “Structured Programming” - structured flow charts 
and so forth. In other words, we learned how to apply the good things 
we learned in the assembler days to high level code, and we learned how 
to design code in ways that helped ensure the  implementation of that 
design was accurate.
That of course, inevitably led to better programs, not only in terms of 
efficiency, but the programs were more correct. Hardware of course, 
progressed to the point where most development was online, and as 
hardware got better, the programs running on that hardware pushed more 
and more into new capabilities.
Minicomputers entered at around this stage, and things really took off. 
Vaxen, for example, while having good “batch” facilities, were 
primarily an online system, as were TOPS-10 “mainframes”, Wang VS, Data 
General, and a slew of others. Minicomputers took naturally to high 
level languages - mostly because of the cost. It wasn’t any more 
expensive to run an online program than it was to run a batch program, 
or at least, not much. Besides, online systems were fun.
So the greatest number of us “old farts” really got to use 
Minicomputers as our primary leaning platforms - the mainframes were 
just too darn expensive for people to learn on.
And then the MicroComputer hit, and everyone and their Aunts and Uncles 
were learning to program- usually in Basic. People got results. A whole 
generation or two of today’s IT experts grew up with Applesoft  Basic, 
or TRS80 Basic, or Coco Basic, or IBM Basic. Those basics require one 
to learn how to code subroutines and well mannered GOTO’s - if one 
wants to have a program that runs.
Of course, COBOL as quickly available on micro computers too - though 
it was so darn expensive that only large companies could afford it.
And while this was happening, another player snuck into the game - Unix 
was being used by more and more people every day - on centralized 
systems like Vaxen, or on Sun Workstations. Unix became the lingua 
franca of IT, and with Unix, one must inevitable learn C.
Arguably, C is a high level implementation of PDP-11 assembler - at 
least I think so and I now C and PDP-11 Macro assembly very well. :) 
The point being however, that all the good things we learned in 
assembler are pretty much available in C. And a LOT of the good things 
we learned in high level languages are also available in C. C lends 
itself well to structured design.  Just like COBOL.
During the late 1980s and early 1990s, we went through a dramatic stage 
of software thinking, during which time you could easily hear five 
different development and design philosophies just while eating lunch. 
Data Flow Diagrams emerged, along with UML, RAD, Top Down, Bottom Up, 
Layered Virtual Machines, and more.
Okay, I will get to the point - what we call objects today can trace 
their ancestry back to those subroutines in assembler, and paragraphs 
in COBOL, and functions in C, and data section records, and Pascal 
functions and subroutines, and so on.
In other words - they are, in general, the embodiment of just about 
every good practice and technique we IT people have learned over the 
past 60 years or so. Some implementations are better than other - C++ 
is awful in my opinion, while Ada is near perfect.

But that is really they are - fancy subroutines that embody good practices.

COBOL can make great use of methods and object oriented programming, 
but mostly when it is dealing with outside connections, GUIS, or when 
accessing some system services.
CURRENT-TIME for example, is in my opinion an example of a well used 
COBOL object.

I know, that is kind of a radical point of view, but it is what it is. :)

-Paul

When people say OO programming is not good for "x", I think it mostly means 
they just don't like OO programming for whatever reason. What makes OO COBOL 
not as good for batch programming as regular COBOL?  Isn't regular COBOL a 
subset of OO COBOL?  Just because the OO features are there does not mean 
that you have to use them, or at least you can minimize their use when the 
situation calls for it..

Like Pete I never really got OO programming until I learned Java.  I still 
did not grasp why it was so popular until I read a book on design patterns. 
The design patterns book showed me clever ways of combining objects (think 
functionality) to solve a wide range of problems.  Prior to this I primarily 
thought of  each PERFORMed paragraph or Called module as performing a single 
function.  I think structured programming influenced my thinking on this. 
I have to admit that most of the business programs that I wrote did not need 
the extra power of the design patterns. Almost all of the business programs 
that I have written have involved serially invoking one modular 
function/procedure/module after another, and this worked very well.

Objects can be more versatile (some might say more complicated) than serial 
use of functionality.  Objects can have many methods and objects can be 
combined using composition or inheritance.  This involves combining 
functionality in ways that I had not previously considered, yielding 
increased power of expression.  You can get some of the OO benefits via 
manual coding in that style but an OO programming language makes it much 
easier.  In the past most COBOL applications did not need any of these extra 
capabilities, but as applications have moved to the internet,  to gui 
interfaces, to components etc. these extra capabilities start to come in 
handy.

So is OO the end all and be all?  Of course not.  If you look at the Java 
library to see how some of it is implemented, at the bottommost layer you 
will still see native routines written in C++.  Is OO design and OO 
programming the answer to everthing?  No, and it can be hard to get it 
right.  The Java I/O features are a good example of this, as the supposedly 
highly skilled programmers at Sun have redone I/O more than once.  Without a 
knowledge of the design patterns some of I/O implementation does not seem to 
make sense when you first see it.

Back to the question, is OO COBOL relevant? Yes it might be, if you want to 
be able to use some of your existing COBOL in the network environment.  But 
the cost of the compilers is high and the learning curve is steep.  One 
criticism of OO is the new terminology and OO COBOL only adds to this with 
terms like "Repository" and "Factory" which other OO languages do not use. 
If you are like me and Pete you might have to learn Java or C++ or C# first

Exactly, the criteria is 'depending'.

Most applications that were developed in COBOL were based on the
principle of serial reuse. To process a sales order you read through
the order lines one at a time dealing with each one in turn. This
reuses the record area. There is no need to create a 'record object'
for each order line and it is usually incorrect to 'process them all
at once'.

(I recall Robert Wagner once recommended that for process a sequential
file it could be done by creating an object for each record and then
telling them all to read.)

Most applications that lend themselves to OO are not written in COBOL.
Granted that GUIs tend to be OO, or at least pseudo-OO, and one can
write COBOL GUIs.

(I wrote Windows fully graphics COBOL programs for Windows 3.0. Not
just GUI but graphs and 3d views).

As it happens an OO structure can be created using dynamically called
programs. Methods can be emulated by calling with a 'function'
parameter that is used, for example, to dispatch to nested programs

Instead of INVOKE 'dosomething' of Customer ...
it can be CALL Customer USING 'dosomething' ...

The main constraint of this is that there can be many occurrences of
objects while there can be only one module of each name.

The question then is 'do we need more than one of these at a time?'.
