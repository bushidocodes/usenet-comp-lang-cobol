It's all about functionality. If a join can get you the information you need,
then the DB2 bind optimize process will almost certainly give you better
perfomance than you can do by programmatically simulating a join. But if you
were doing a lot of joins where simply adding a column to a table puts the info
you need into a single table, then single table access will be faster. That's a
database design issue.

Joins can be expensive, and if a naive programmer sets up an unneccessary join,
or a poorly structured join, then you can pay a high price in performance. So
it makes sense to visit those programs that use joins to see if the join: 1) is
necessary at all; 2) could be avoided by a small database redesign; 3) could be
coded more effectively. But to unilaterally say "Let's get rid of all joins"
seems a bit extreme.

standard "wisdom" has been that VSAM -> DB2 in best case scenario
is 50% decrease in raw throughput (seen this number on DB2 forum).  
worst case can be a lot worse.  what a DataBase buys is flexibility,
not raw speed.  go faster:  get closer to the iron.  TPF is faster than
VSAM is faster than DB2.  on the other hand, rewriting the code (COBOL
i suppose) to dump the hand built concurrency and transaction and data
management (those pesky joins) will get you close to a wash.  leave that
code in, and get the data from DB2 in its previous VSAM form, you win 
the worst of both worlds:  DB2 protects the data, and so do your COBOL
routines.  low level tuning, so the gurus say, can get one back to
near VSAM throughput.  it's also worth remembering, at least with DB2,
that the physical files are still (somewhat altered) VSAM.  what seems
to be most signficant to me is:  lots of physical files, indexing has
to fit the VSAM paradigm.  other kinds of DB's that don't map tables to
physical files can store joined rows together.  don't think DB2 can.  

not to be catty, but lots of DBA's (alas) who grew up on file system
systems tend not to fully understand the power of keys and relational
structures in DataBase engines.  this seems more true in the MainFrame
world than Unix.

yes, gathering data from two physical files can be more work than reading
a record from contiguous space on one physical file.  now what do you
do with it?  if it's data from a file system file, you have to manage
concurrency in your code, or let some transaction processing monitor
do it.  when you write it out, the code has to make sure that nothing
mucky has happened.  if you want to delete data, same thing.  

anyway, it comes down to philosophy:  do you want a data-centric system
where an intelligent database engine does most, if not all, of the 
management of the data; or a code-centric system where the data is just
some lump on the file system which is managed by code (COBOL, for 
example).  with a data-centric system, any application can use the data
without mucking it up.  some code shops are threatened by this, and
contend that the "business rules" embedded in those millions of lines of 
code could never be replaced by primary keys, foreign keys, referential
integrity, triggers, and stored procedures.  "and all this code works
just fine on flat files, why do we have to change?"

Peter Dashwood (either on this forum, or DB2) has pointed out that the
file system is the Achilles heel of COBOL.  i suppose that's what this 
missive says, too.

A few years back I worked for a gas/electric utility and were having to
deregulate the business and give customers the choice of different suppliers.
One program I wrote was to pass new customer info on to the appropriate supplier
and drop notifications back to us.  

The first production run with a pilot of just 50,000 accounts it took over 35
minutes to complete.  The performance monitor showed almost 90% of execution
time was in one paragraph that was just a select with a join.  Before the job
had completed I had drafted a change to get rid of the join and just do two
seperate selects.  

Made the change in time for the next nights run and it completed in less that 5
minutes.  Joins are nice but I learned that they can be quite resource
intensive.
