> As long as Windows 2000 is binary compatible with Windows NT, then
> I would expect that the Workbench product will work as expected."

It is not quite as simple as that. In general terms I would consider most
Cobol that is functional with Windows NT 4.x also to be functional in Win2K
environments, there is however a couple of pitfalls;

- Installation is one issue. As Win2K introduces multiuser capability
natively to the OS, there are
  various actions taken, and various actions that should be taken into
consideration. These are too
  numerous to be listed here, and those interested will find extensive
information about it on the
  Microsoft web site.

- Version information is extended (pretty obvious), given your current
implemented version check
   is done in that particular way, your application is likely to receive
erronous information.
   For instance, to detect a that a Windows NT 4.x is a server, one used to
look up the registry,
   this part of the registry are now protected due to security issues, and
will fail using the 4.x
   approach.

- The DLL & registry entry policy is changed. For both is protection heavily
increased, the aim here
   is obviously to avoid an application to enter an invalid condition
because of user modifications like
   deleting registry entries and deleting files.

- Finally the worst stuff. The multi user environment. This also affects the
installation and not to forget
  network capability. Obviously, Microsoft has attempted to force the
expected feedback from the
  API to provide expected results as if the application were run locally,
but if absolute file paths,
  registry look ups, low level network api is used. Error conditions might
happen, depending on the
  context. A simple example here is if an application installs an ActiveX
object into the local user
  space and not the machine space. This would effectively block other users
from accessing the
  ActiveX.

Just my 2 cents,

Cheesle
