At the Dept. of Veterans' Affairs in Philadelphia, we have several programs
that were translated to COBOL--verbatim, almost--from 7080 Autocoder by a
contractor. The result was as incomprehensible as the Autocoder--completely
unstructured and nothing but go-tos. We used two mainframe products--one,
PATHVU, to help us understand what the code was doing, and RETROFIT, to try to
put the programs in some sort of structure.

The RETROFIT tool produced only marginal results (to us)--probably because we
just didn't have the time or resources to really learn about the tool or the
program being restructured. The paragraph names were the same as the ones in
the old code--RETROFIT just put prefixes on them. Also, some code was
duplicated by the tool, while in other sections, the output code would never
be executed.

We only restructured two or three large programs with RETROFIT, and gave up on
the rest. (However, the paragraph-prefixing system of RETROFIT *did* give us
ideas on refine our paragraph-naming standards!)

As a result, I tend to distrust restructuring tools. For one thing, the tool
is just a program that blindly takes code and changes it according to set
parameters. Unless the user is able to take the time to thoroughly understand
the code he is restructuring, what it's doing, the business rules imbedded in
the code, etc., and can tweak the restructuring tool to give the code most
suited to the application, the "restructured" program will make no more sense
to him than the old program.
