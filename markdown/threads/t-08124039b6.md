Michael ValeCruz wrote in message <7pu984$h0q$1@nntp2.atl.mindspring.net>...
>If the routine returns a pointer to the allocated memory block then how can
>the block be relocatable?

The block is not relocatable! It will remain at the same
location until CBL_FREE_MEM is called using the same
pointer (value) that was returned by CBL_ALLOC_MEM.
After CBL_FREE_MEM, the block is invalid.

>Would the pointer be void at the point the memory is relocated.

The value of the pointer will be set by CBL_ALLOC_MEM.
The only (reasonable) means for changing the value of the
pointer are: SET pointer-name TO NULL, after freeing the
block, and CBL_ALLOC_MEM, for another block after
freeing the original block.

I wonder if you are confusing the phrase "dynamically
allocated memory" with "relocatable memory".

Since you mention void pointers; in C language terms,
'realloc()' may relocate a memory block; but it does not
apply to this case. The subprograms CBL_ALLOC_MEM
and CBL_FREE_MEM behave the same as the C functions
'malloc()' and 'free()'.

In (draft) COBOL terms, similar behavior is described in
the new statements, ALLOCATE and FREE; though
ALLOCATE has more functionality than CBL_ALLOC_MEM
or 'malloc()'.

HTH

Rick Smith

No need to be sorry about your disagreement. I was wrong!

MF WB 3.2 allows for relocatable with either shared or
nonshared. It also allows fixed and nonshared for DOS.
This is what I have used in the past so the relocatable
option wasn't foremost in my mind.

I think the best answer for Michael is, don't worry about it
UNLESS you are using dynamic allocation in a way that
will cause severe fragmentation. If only a few small blocks
are allocated and they are all freed or freed in the reverse
order of their allocation, memory will not become severly
fragmented.

I ran some tests using various block sizes to see if I could
better understand the underlying operations. There
appears to be separate relocatable and fixed areas. Using
small blocks, all memory seems to be allocated from
the relocatable area. With larger blocks, it appears to
use the fixed area even when flagged as relocatable.

Finally, I set up a table of 256 pointers and ran the program
several times using block sizes of from 16 to 32768 bytes.
With the smaller blocks, all memory was allocated from
the relocatable area. As I increased the block size through
256 bytes, some memory was allocated from relocatable
and some from fixed. Around 4096 bytes, all memory was
from the fixed area. Even with 256 blocks of 32768 bytes,
or 8 megabytes, I could find no problem allocating or using
the blocks.

The next test would be to force the fragmenting of memory.
I chose not to do that test.

I think that the only time the MF COBOL runtime would
relocate a block of memory is during CBL_ALLOC_MEM.
It would do this as part of a memory compaction process
to create a larger block for the new allocation. During
the compaction process, it would move through chains
of pointers that contain the locations of pointers that were
set to the memory address of the blocks being moved.

As long as external code, such as a C module, does
not save the address of the relocatable block, for later
reuse, there is no problem. If each CALL passes the
address of the block, the external code will never know
the block was moved since the last CALL.

There is a potential problem with passing relocatable
blocks to other MF COBOL programs that CALL
CBL_ALLOC_MEM. Prog-1 allocates block-1 and
calls prog-2 using block-1, then prog-2 allocates
block-2. If block-1 is moved when block-2 is allocated,
the contents of block-1 could become inaccessable
in prog-2!

One other potential problem is with relocatable blocks
that contain pointers to other relocatable blocks.
If block-1 contains a pointer to block-2, then if block-1
is moved, memory management would need to adjust
its chain of pointers to pointers to block-2 to reflect
that the location of the pointer in block-1 was changed!

Fortunately, designers tend to take such potential
problems into account.
