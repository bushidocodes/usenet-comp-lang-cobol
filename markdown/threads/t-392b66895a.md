There are bunches and BUNCHES of threads on this topic if you check
DejaNews - but to "summarize"

1) When using NOCMPR2 (ANS'85 semantics) and VS COBOL II, COBOL/370, COBOL
for MVS & VM, COBOL for OS/390 & VM (and the VSE versions) *ALL* require
that your FD with its record descriptions REALLY match the DCB of your input
variable length file. If they do not, you will get a file status of 39
during the OPEN. (How this shows up will depend on whether you have a
DECLARATIVE, what your ABETERMENC or other run-time options are, or if you
check your SYSLOG). For OUTPUT files, the same is true, but it is less
likely that you will be creating new files with the wrong combination of FDs
and DCBs.

Note: Using the CMPR2 compiler option *may* get your program thru
avoiding some FS=39 situations. However, this means that you lose ALL the
benefits of ANS'85 COBOL. It also means that you can't develop your
programs with IBM's workstation programs.

2) To be the safest (and there are other ways to handle this), you really
need to know what the smallest and largest legal (not just existing) records
are in the physical file (ignoring the 4-bytes for the RDW). Let's say the
smallest possible record is 10 bytes and the largest is 100 bytes, then the
BEST way to define your file is:

FD MyFile
Record varying in Size
from 10 to 100 characters
Depending on WS-Size-Counter.
01 Any-Old-Record.
05 Each-byte occurs 10 to 100 times
Depending on WS-Size-Counter.
10 aSingleByte Pic X.

If your file actually only contains two types of records, the 10-byte and
the 100-byte variations (and no 11, 12, ... 99-byte records), then you could
replace the "01 Any-Old-Record" with the following:

01 Small-Rec Pic X(10).
01 Big-Rec Pic X(100).

The WS-Size-Counter needs to be defined in Working-Storage and NOT in the
File Section. After each READ (assuming this is an input file) this field
will contain the size of the record that was read. (On a WRITE for an
output field, you need to fill it in before the WRITE to tell the system how
big a record to create.)

3) If you want to make less changes to your OS/VS COBOL source code, you
can use the following (which should work to avoid the FS=39 - but doesn't
take advantage of the new facility to know how big a record you are
reading).

FD MyFile
* the following 2 lines are IBM extensions - don't use it if you want
* to create portable source code
Recording Mode is V
Record Contains 0 Characters.
01 Small-Rec Pic X(10).
01 Large-Rec Pic X(100).

If your actual record structure DOES include a length field in the first
(fixed) part of each AND EVERY record, then you can replace the two 01's
with something like.

01 Every-Rec.
05 I-Know-the-Length Pic S9(04) Comp.
05 Different-for-Each-Rec
* remember that the length counter took 2 bytes on big-blue
occurs 8 to 98 types
depending on I-Know-the-Length.
10 aSingleByte Pic X.

**************

The biggest problems that most IBM MVS applications have in migrating
variable length files to the new compilers are:

A) They don't KNOW the real minimum and maximum record lengths.
B) They try and use the same FD for files that REALLY do not have the
same LRECLs.
C) The variable length records don't have any real field that includes
the length (just some type of record-indicator that tells what type of
record they have, e.g. "A" in byte-1 means this is an ADD record and it is
10 bytes long.)

**********
