could be worth money.)

decide that analysis is for you or not, I strongly advise you to make
an effort to get to grips with "new technology".

(The trick is NOT to approach it as if it were a programming language

programming.

with decent in-house applications experience (maybe gained as a COBOL
programmer), and a couple of projects as a Team Leader or PM, could
all add up to a more worthwhile package than simply "COBOL
Programmer". This is a good one for those who are a bit older as well,

GUI needs OO. (All the GUI controls are Objects with their own methods and

used it so can't be sure.) It is this object orientation that allows GUI
screens to be built quickly with drag and drop, re-using buttons,
tree-lists, menus, drop downs, and so on... and then incorporated into
applications as components. Otherwise it is necessary to utilise GUI Classes
and instantiate the objects as you want to use them. (This is a much more
long-winded process.)

to COBOL many years after the language was invented so it is not seamless.

not ONE set of GUI Classes included in it, but SEVERAL. (Paul mentioned the

finish called "metal" and both of these are in addition to the "Standard"
presentation class for Applets, Servlets and Beans...) As far as I know,

6.1 Class libraries and I could be wrong.

GUI classes available (or not) is fairly unimportant. You can reference
standard Windows objects and use the standard Windows FORMS objects,
referencing them as components. Or you can use the Java Classes if you
package them as beans or components.

So the answer to your question, Ken is that Java is well served when it
comes to GUI and COBOL isn't.

only tool I know that lets you get into GUI without OO knowledge is
PowerCOBOL. It simply hides the OO stuff from you. (I guess M/F DIALOG does
a similar function...)

Neither of them is much chop when it comes to complex relational DB
processing... (So we wrap them around SQL, or we set up RDB procedures that
can be triggered by them). And so it goes on...

The best tools for the job...
