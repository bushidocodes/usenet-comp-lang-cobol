The current name for LINC is Enterprise Application Environment, or EAE, it
is alive and well, has proven a very popular product, and it has indeed
evolved significantly in the last thirteen years!

The current version (EAE 3.1, AKA LINC 17) is available on a number of
platforms (NT/2000, MCP/AS, OS2200, UNIX that I know of).

The rest of this discussion is limited to the MCP/AS environment, which is
the one I am most familiar with, and my familiarity with LINC is limited to
its interaction with the various COBOL compilers over the last
decade-and-a-half.  .

In general, I find generated COBOL code hard to follow, and I think that's
as true of the code generated by LINC in the past as for any other COBOL
source generator from non-COBOL input.  Another "wrinkle" is that older
versions of LINC were designed to generate monolithic programs, and for
large LINC environments these single-source programs sometimes got HUGE
(exceeding half-a-million lines).

Although previous versions of EAE/LINC generated code compatible with the
COBOL74 compiler (the latest ones also allowed use of COBOL85), the COBOL
code that EAE 3.1 generates is COBOL85 source code exclusively, makes use of
many of the features of that language, and tends to avoid generating immense
monolithic COBOL programs in favor of a more modular approach, binding the
modules as subprograms in a larger ALGOL host file.

Yes, generated source can be difficult to follow sometimes.  But in general
I believe the EAE/LINC approach is to use COBOL as an "internal tool";
corrections and changes are made at that level rather than at the COBOL
source level; users are not encouraged to maintain multiple sets of changes
to the software (one to the LINC specifications, one to the COBOL source
programs, one to each of the associated data bases, etc.).   We believe the
application development tools available with EAE make that sort of
after-generation "tuning" unnecessary.   The fundamental idea is that the
EAE/LINC developer need not be aware of the fact that the environment is
generating COBOL code at all, much less be concerned with its intricacies.
The user interface has been vastly improved over LINC's lifetime, making
this a much more practical approach to application system design than once
it may have been.

I found the EAE Info Center at  http://www.unisys.es/sw/app_dev/eae/  .
Hope the information there proves useful; there are some other links that
also might be useful via a search on "eae" from www.unisys.com .

    -Chuck Stevens

Er, yes... superficially.

The action of simply indexing does not ensure inversion, at least not in the
sense that a DBA would understand it. (It is more properly called "binary
inversion" but convention allows us to drop the "binary"... it is fully
explained below.)

Inverted indexes (and consequently, the "inverted files" they refer to)
first made their appearance in the late sixties as disk based technology
took hold. There was still a substantial "residue" of Assembler Programming
around and "everybody knew" that Binary was best. (The advent of High Level,
byte oriented Languages, like COBOL, slowly changed this perception until it
disappeared and today's Programmers are happy to deal with bytes, rather
than bits.)

Consider the following (Clear your mind of the beloved Indexed Sequential
model, and try to think in terms of RAW DATA, with no structure, sequence or
organization. Long ago, at the dawn of cybertime, when the first disk
actuators began clicking across the ferric oxide, this was what we dealt
with <G>...)

A file....each record has 3 fields. We'll call them "ID", "SIZE", and
"COLOUR". Note that the ID is unique, but the other fields are not. In fact,
it doesn't matter whether ANY key is unique or not in an inverted file (it
is MUCH more efficient if it ISN'T unique!), as we shall see in a
moment...(I did it this way because this is what most COBOL programmers are
familiar with.)

Record     ID     SIZE     COLOUR
(position)
1             7a1        6            blue
2             3b5        8            blue
3             7a3        8            red
4             7a4        10          blue
5             7a2        8            blue
6             7a6        10          red
7             7a5        6            red
8             3b3        10          blue

There are a few things we can immediately note about this data:

1.  "7a" objects are represented in SIZE 6, 8, and 10, and COLOURs red and
blue.
2.  "3b" objects are represented in SIZE 8 and 10, and COLOUR blue.
3.  It is NOT possible to derive the SIZE or COLOUR from the ID by a
consistent mathematical process.
4.  The IDs are unique.
4.  There is NO intended order or sequence to any of the data.

Because the IDs are unique we may decide to build an index on them and call
it a "primary key".  It (the index) looks like this:

Record     ID
(position)
1             7a1
2             3b5
3             7a3
4             7a4
5             7a2
6             7a6
7             7a5
8             3b3

We can sort it to provide the records in ID sequence. This is NOT an INDEXED
SEQUENTIAL organisation...(actually, Burroughs (the half of Unisys which was
NOT Sperry <G>), used to refer to it as INDEXED RANDOM which is quite a good
name for it...)

But what about the data? Well, we want to be able to search VERY quickly
without reading physical records (we will ensure that any indexes we create
are small enough to be in memory) so the best technique for this is called
FILE INVERSION. Here's how it works...

Let's build an inverted index for the SIZE. Start with SIZE 6.

There are 8 records on the file so we need a bit string 8 bits long. It
looks like this:

10000010

What this tells us is that record 1 and record 7 on the base file have a
SIZE attribute equal to "6".

Here's SIZE 8...

01101000

and SIZE 10...

00010101

Now we can do a check that all records have been accounted for by ORing the
3 indexes together, like this...
10000010
01101000
00010101
-----------
11111111

If the result is x'FF' (HIGH-VALUES, if you prefer...) then we have indeed
accounted for every record and our system has integrity.

Building COLOUR indexes is carried out the same way...

BLUE index
11011001

RED index
00100110

Note that in this example, we have organised ALL of the information (apart
from the ID), in ALL of the base records, in a total of 5 bytes!

The information in the base file has been "inverted" in the sense that we
can now search for any value and get back to a record, instead of the normal
approach of reading records to find a value.

So I want to find a "red" size 10 object...   I can AND the RED index and
the SIZE 10 index.

RED         00100110
SIZE 10   00010101
                ------------
                00000100

Immediately, I find there is only one, and it is in record 6.

Remember that to do this I used 2 bytes and one of the fastest instructions
in any machine's instruction set. (Logical instructions on bits are
typically MANY times faster than equivalent operations on bytes, if you care
about that sort of thing...<G>).

Databases which implement inverted files are typically blazingly fast. The
best one I ever saw was called "system 2000". Sadly, it has fallen victim to
the unstoppable onset of the Relational Model and I don't know of anyone who
is using it today. It was a Networked DB (like IDMS), with all sets
inverted. I saw it search a database of Ford motor parts in the
early 70s. It searched over 2,000,000 components in under 5 seconds and, at
the time, that was just awesome. An IMS database to do the same job would
have taken several minutes.

 (Willem noted elsewhere in this thread that he believed ADABAS was inverted
and NOT Networked. It was common practice for databases at the time to
establish inverted indexes to the underlying data, but the mechanism for
storing the underlying data was based on either the Network or Hierarchic
Database Models.)

So when Warren says a file with a primary index is a "single key inverted
file", he is not exactly wrong, because the key data HAS been "inverted",
inasmuch as we can get back to the base record from it. However, most DBAs
will consider an "inverted" file to be one which is referenced by inverted
indexes of the type described above. (i.e. binary inverted indexes)

By now you will have realised that if a key is UNIQUE, it is NOT  a good
candidate for binary inversion. It would mean having an index with a single
bit set, for every key value. However, back in the 60s there were systems
that did that. Where the file size is, say, not more than 4096 records
(index requires 512 bytes; easily stored in memory without requirement for
paging, even on a small system) it was considered feasible to index unique
values in this way. Today we wouldn't (and don't <G>) do it; we just opt for
a non binary unique index, or (preferably, in my opinion) move to RDB.

> A file with records indexed on ALL fields (keys) is  a fully inverted
file.
> As popularily said in these parts, you pick the inversion to fit the
> problem.
>
Yes, that's right, but it's not just the keys. EVERY field is a KEY if it is
represented by a binary inverted index. The criteria you use are:

1. Are there a number of unique fields? (Bad candidate for binary inversion;
may be good for Indexed Sequential with Alternate indexes (remember,
alternate indexes don't HAVE to be non-unique...)

2. How important is incredibly (as opposed to just fairly...<G>) fast
access, and what about available storage?

3. How big are the files? (you need 1 bit of storage in the index for every
record on the base file, and an index for each unique value of a data
field...)

I hope the above may stimulate some of the more adventurous to try some of
this as a departure from the Relational model <G> (this is not an implied
criticism of the Relational Model, which has achieved its place because it
is the best "all round" model for a data repository... but programmers need
to have some fun occasionally, and building indexed random, binary inverted
databases, for small (non critical) systems is not only fun, it is immensely
satisfying...<G>)

Pete.
