```
Hello group,

As a first project in OO COBOL, I've been porting Jon Skeet's C#
snippet compiler, Snippy, to Micro Focus Object COBOL. Once ported, I
plan to add support for compiling COBOL snippets. After that comes the
Reflector add-in. :)

MainWindow.xaml.cbl and Commands.cbl are done. ParagraphWriter.cbl is
also nearly complete, but for the life of me, I can't figure out how
to port C# delegates and .NET Actions to COBOL.

MF Object COBOL clearly does support the delegate-id paragraph, and
I've even seen a delegate example in this group's archive. So this
should be easy, but I'm still missing something.

Would anyone be willing to show how a COBOL delegate would be used
port this C# snippet?

  using System;
  // ...
  using System.Windows.Documents;
  using System.Windows.Threading;

  namespace Snippy
  {
    class ParagraphWriter : TextWriter
    {
      /// <summary>
      /// Equivalent of Action in .NET 3.5
      /// </summary>
      delegate void VoidAction();

      Paragraph paragraph;

      // ...

      public override void Write(string text)
      {
        VoidAction action = () => paragraph.Inlines.Add(new Run
(text));
        paragraph.Dispatcher.BeginInvoke(DispatcherPriority.Normal,
action);
      }

    }
  }

-----

Skeet's XML comment above suggests using the .NET 3.5 Action type. So
the Write function could be rewritten in C# as follows -- and I'd also
like to know how to do this in COBOL:

  public override void Write(string text)
  {
    Action<string> deed = AddRun;
    paragraph.Dispatcher.BeginInvoke(DispatcherPriority.Normal, deed,
text); // <---
  }

  public void AddRun(string text)
  {
    paragraph.Inlines.Add(new Run(text));
  }
```

```
There's nothing wrong with COBOL. It is an excellent language. It was 
conceived in generosity, with a spirit that transcended individual interests 
of the parties concerned. But it belongs to a different age.

The problem is not with the language but with the paradigm. The future 
belongs to objects and layers. COBOL was extended (by some amazingly 
foresighted software engineers) to encompass objects, but nobody wanted to 
know. Now it is too late. Other languages implement the object paradigm 
better than COBOL and that's all there is to it. It isn't about what C# or 
Java can do and whether it can be done in COBOL; it is about how easily and 
quickly it can be done in a language best suited to the task.

COBOL will remain very useful for things like Batch processing (as long as 
that remains necessary... I believe it (batch processing) will be extinct 
within 10 - 15 years as more powerful processors with multi cores come on 
line and can do everything in real time. When you have Business models which 
are true analogues of a business, in real time, and are capable of 
re-constructing any events that have occurred at any time in the history of 
the business, batch processing is unnecessary. The way in which we do 
business will change also, just as it changed from the Victorian model to 
our current models.)

COBOL, (largely due to its simple construction and English-like syntax) has 
appeal for many people, but it is not a Programmer's language, in the 
computer science sense of the word.

Yes, you can write programs in it, but they are procedural programs for a 
former time, when things were different. The changes coming in this century 
will make COBOL even more anachronistic than it is already.

Having said all of that, it is likely to be around for a while yet. I 
predicted it would not be in use  by 2015, in 1997. I now think that may be 
arguable. Certainly it will never have the ascendancy it once enjoyed as 
"the only game in Town" back in the late 70s and early 80s, but I don't 
think it will completely disappear. OO COBOL has a future as a means of 
moving Legacy systems to newer environments and as a means of enabling 
existing COBOL programmers to transition into OOP,but, as I mentioned 
elsewhere, once the OO paradigm is grasped, the limitations in OO COBOL 
become apparent. It isn't that you CAN'T do stuff in OO COBOL (you mostly 
can...) it is just that it is more labour intensive than doing it in other 
languages which are now available. As a major development language, I think 
my 2015 prediction will not be far off the mark, but there is almost 
certainly going to be continuing work on legacy, even then.

Most people here will think I am anti-COBOL. I'm not. (I'm happy to use it 
for jobs where it is ideally suited, and I do. I also have a fairly large 
chest of OO COBOL components which I developed over years and won't be 
replacing any time soon...). But I do believe it is not relevant to the 
environments in which I wish to work.

1. The skill base is being eroded by age, perception, and lack of training.

3. Legacy and history are driving most of the COBOL sites still in

4. It costs too much to develop and deploy. Only large companies can afford
```

```
There's the nub of it.

In a simplistic centralised environment there is no need to change the way 
we think. Processes on the computer are sequential Von-Neumann type 
processes which emulate the human approach to the same problem.

In the real world somebody does a series of steps, takes a decision, does 
another series of steps, and iterates the process until a result is achieved 
(Dyjkstra had the insight to realise this and formalise it mathematically). 
Computer processes in the early days modelled what people did. Just as a 
person has one brain the computer had one core.

But as soon as you start networking there are are many cores available. They 
can work synchronously and/or asynchronously, can use their own data or 
share data with every core on the network, and it isn't just local it is 
world-wide. We are coming to a time where not only will there be many nodes 
on the network (each one with a processing core), but there will be many 
cores available to each node. This has implications which are not 
immediately obvious.

Objects allow easy "packaging" of information (which knows how to process 
itself and has various methods which define its behaviour) to be flashed 
around the network and present the same interface to every core it 
encounters. Layers allow selection and isolation of objects from and to 
various cores. Hence my statement that the "future belongs to objects and 
layers". Even from this very brief simplistic description, it is immediately 
obvious that this represents a fundamental change in the way we approach 
problem solving on a computer. In other words, there is a paradigm shift, 
and what was OK in the 1960s with a single processing core is way less than 
optimum in the 21st century with multiple cores available.

But the way we think about things (except for a very few of us) HASN'T 
changed. If you look at any specific part of the network model described 
above, you see data being processed by a core, and that seems right, 
familiar and comfortable, so there is apparently no reason to change the way 
things have traditionally been done. And you CAN achieve results that way, 
even in a networked environment.

BUT, if you make the  mental switch, understand the whole business with 
encapsulated objects (components) and layers, latch on to the idea of data 
which has behaviours built into it, that can process itself any way, 
anywhere, anytime there is a core available, then you can really make this 
paradigm sing and the results can be quite spectacular.

I've been talking about this to a team of COBOL people on a site which is 
migrating away from COBOL and dragging themselves into the 21st century 
(first step is migrating from ISAM to RDB...) They are already planning and 
looking forward to refactoring their existing COBOL into objects. I have 
given them tools which automatically convert their existing data access into 
an RDB with an Object Oriented Data Access Layer (various objects handle 
various table sets on their newly created RDBs),  so one layer in their 
triple tier target has been automatically achieved. The Business Logic and 
Presentation Layers will need manual intervention, but they are enthusiastic 
and fired up about doing it. I showed them some demos of what can be 
achieved with objects and layers and why this is important. When I left, 
they were a COBOL team with fire in their eyes and the talk was all of 
objects and layers. The Management of this site are wise, and are happy to 
invest in C# and Visual Studio training for the existing team. They have a 
long term plan which will give them a system running in .NET, totally Object

salvaged around 80% of their existing COBOL investment.
```

```
I cobbled something together based on MF's DelegatesDemo and CobolDoKu
examples. The code compiles, which is progress.

With apologies for the bad COBOL style, here are the delegate
definition and the ParagraphWriter's Write(string) method:

   delegate-id. VoidAction as "AddOneSnippy.VoidAction".
   data division.
   linkage section.
   01  textRun  type "System.Windows.Documents.Run".
   procedure division using by value textRun.
   *> The method is just a prototype and should contain no
implementation.
   end delegate VoidAction.

   ---------------------------

   method-id. "Write" is public.
   data division.
   working-storage section.
   01  deed     type "AddOneSnippy.VoidAction".
   procedure division using s as string.
       *> Create a new delegate
       set deed to new
"AddOneSnippy.VoidAction"(outputParagraph::"Inlines"::"Add"(
           new "System.Windows.Documents.Run"(s)))
       *> Pass the delegate to a dispatcher
       invoke outputParagraph::"Dispatcher"::"BeginInvoke"(
           type
"System.Windows.Threading.DispatcherPriority"::"Normal", deed)
       .
   end method "Write".
```
