>

PERFORMing SECTIONs and any PERFORM THRU leads to situations where 
it is necessary to have a GO TO or it simply drops down through
paragraphs.  (except for the trivial case where there is no or one
paragraph in the section in which case there is no point in 
performing the section).

> at the end of *every* SECTION in your program.  This will allow
> you to copy most any routine with out having to fret about ugly
> GO TOs used for exiting the code.   Please note that the *real*
> exit point of a PERFORMed SECTION is just before the *next*
> SECTION definition -or- end of source.

This is one of the reasons why performing sections leads to 
errors.  Eliminate the problem and only perform paragraphs.  It
may take a little more thought to structure the code as you can't
just use a no-brain go to the exit, but the benefits are a cleaner
design and fewer bugs (when it is done right).

>
> This also alleviates the need for those not so pretty 
> PERFORM ... THRU ... statements.  And, of course, you can
> still PERFORM specific paragraphs within the section.

Arggghhh! performing the whole section and (later) paragrahs within it.
This is the stuff of disaster code.  The ONLY point of using perform
of sections is to allow no-brain go tos to the exit point.  If you
then allow individual paragraphs within this to be performed then
what happens if a later coder decides to bail out of that paragraph
with a go to because he is only considering the logic related to
the section perform ?  

Functional units should have ONE entry and ONE exit.  This excludes
overlapping performs entirely from sensible code.

```
 
: Arggghhh! performing the whole section and (later) paragrahs 
: within it. This is the stuff of disaster code.  
 
  > No! "Disaster code" comes from *any* language, style, verbs
  > etc. when the programmer does not take the time and effort
  > to properly design the code *before* writing the program.
 
Certainly there may be other ways of forming "Disaster code", but I 
suggest that Performing sections and (as implied by the original 
message) later performing individual paragraphs within that from 
outside shows bad style and bad design. 
 
  > Actually, I thoroughly agree that PERFORM ... THRUs are only
  > slightly better than ALTER ... GO TOs... ;-)
 
My view is that the PERFORM of a section is equivalent to a PERFORM 
[first paragraph] THRU [last paragraph] and is as bad as a PERFORM ..
THRU ..
 
  > But performing sections and performing paragraphs within a 
  > section is a nice way to modularize COBOL code.  
 
Modularising related code by grouping it into a section is a good 
idea that I practice.  I started doing this decades ago as it made 
efficient overlaying easier if it became necessary.  It became 
automatic to organise the sections by descending frequency of 
execution so that changing the SEGMENT-LIMIT was all that was 
required to adjust the run size.
 
But having related code in a section does not make it necessary to 
perform the section.  Have the first paragraph as the one that is 
performed from outside the section, and this can perform the others 
within it as required.
 
  > If you still have problem with the idea, think of the SECTION as 
  > a subprogram and each PERFORMed paragraph as a sub-subprogram 
  > only *intended* to be used by that particular SECTION/subprogram.
 
But PERFORMing a SECTION results in at least one of three situations:
 
 1.  There is only one paragraph, in which case just perform that.
 2.  There are GO TOs (bad)
 3.  The paragraph labels are dropped thru (evil)
 
Section performs also mean that AT ALL TIMES the very next thing 
after the section MUST be another section.  This leaves traps for 
the unweary.  Add a COPY to the end of a program and it has not a 
section header, or add an ENTRY, or a simple paragraph and strange 
things happen.
 
Actually I've just changed my mind: the PERFORM of a section is 
WORSE than a PERFORM [first paragraph] THRU [last paragraph], as the 
latter will survive the [accidental] adding of a paragraph after the 
'exit' paragraph.
 
Perform paragraphs ONLY (with some exceptions) and there will be 
less problems to check for and less complexity.
 
  > Even the *best* program can be destroyed by "brain-dead"
  > updates....
 
A good design (IMHO) can cater for later "brain-dead" updates by 
ensuring that traps are not laid.  Eliminating performed sections 
and perform ..thru.. removes some of these potential traps.
 
This is important to me as I have to maintain all my own code, and 
will have to into the foreseeable future, I can't afford to employ 
someone to do it for me.  I want to be able to make "brain-dead" 
updates on code that I haven't looked at for years and have the code 
work, so I design to allow this.
 
The only point of performing a section is so that a go to can be 
used to exit (in fact it usually winds up as being compulsory). Use 
a better design and not only can the no-brain go to be elimininated 
but later "brain-dead" updaters won't be tempted to emulate this 
but will be forced to actually design the change.
 
cheers
```
