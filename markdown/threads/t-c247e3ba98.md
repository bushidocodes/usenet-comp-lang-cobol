According to the standard, a dot does not end an inline PERFORM. All
four PERFORM formats end with

[ stment END-PERFORM ]

Since both the statement and the END-PERFORM are in the same set of
brackets, you must have both an inline statement and an END-PERFORM, or
neither an inline statement nor an END-PERFORM.

The compiler in question is therefore correct in diagnosing an inline
statement without an END-PERFORM, and it will doubtless correctly
diagnose an END-PERFORM without an inline statement.

(I haven't fought my way thru this entire thread, but I think I can answer
the thrust of this one note.)

A) Micro Focus knows that it has an "extension" which allows the END-PERFORM
phrase to be omitted at the end of an inline PERFORM - and if you use their
FLAG(ANS85) you should get a message telling you this. (True not only in
NetExpress but every MF compiler that I have known of for years.)

B) The draft of the next COBOL Standard has "cleaned up" the syntax diagrams
for PERFORM so that it is clearer that END-PERFORM is *required* for inline
performs and prohibited for out-of-line PERFORMs. This was always
documented in the rules, but others besides Huib have had problems finding
these rules in the Standard.

C) The use of a period as an "implicit/explicit scope terminator" does have
some problems and issues. However, the intent is that it only applies to
"conditional statements" - and an inline PERFORM statement (even with
UNTIL - but certainly without UNTIL) is not considered a conditional
statement - hence the underlying philosophy is consistent in NOT using a
period to terminate an inline PERFORM - (or so I was told when I raised this
same question at the J4 committee many years ago).

I haven't seen the new syntax diagram, but I know what was wrong in
Cobol-85. Cobol-74 (which had only PERFORM procedure-name) had had four
syntax formats: Simple, TIMES, UNTIL, and VARYING. In Cobol-85 they
added PERFORM in-line by grafting it onto the existing four formats.
This arrangement is inside out.

Better would be two have two formats:

FORMAT 1 (PERFORM procedure-name)

PERFORM procedure-name-1 [THRU procedure-name-2] iteration-clause.

FORMAT 2 (PERFORM in-line)

PERFORM iteration-clause statement ... END-PERFORM.

In format 1, the perform range is from the first statement of
procedure-name-1 to the last statement of procedure-name-2 (if not
specified, procedure-name-1). In format 2 the perform range is the
statements between PERFORM and END-PERFORM. The iteration clause
defines how many times the perform range is executed.

(Documenting the iteration-clause is left as an exercise to the reader.
There are four options: Null, TIMES, UNTIL, and VARYING. Null can be
documented as a degenerate case of TIMES, ie, the default is 1 TIMES.)

I know this oversimplifies it, but to find bugs more easily, it is
good tokeep in mind how the compiler works:


It is constantly predicting what it should find next. Quick example:
After a period, the compiler expects a Cobol verb. If it does not
find one - it predicts that what it has now found, must be a paragraph
or section name. Immediately it predicts then that a period or the
word SECTION. will follow the name, depending on whether it is a
section or paragraph name.

In your case you had a PERFORM and the compiler then expected to find
either a paragraph or section name or a condition and full statements
to be performed. It found a condition and then a statement and so it
knew that for this configuration of a PERFORM, the end of the whole
perform must be indicated by an END-PERFORM. And so it waits for the
END-PERFORM.

But in stead, after a statement, it found a paragraph name, which is
not allowed inside an in-line PERFORM, and so it was only at this
point that it realized that there was an error and it was only
reported at this stage: according to the compiler the error is that
there is now a paragraph in the middle of this PERFORM in stead of the
predicted statement or END-PERFORM.

The error message is a good one to give you a tip as to what to search
for, as long as you realize the error could have been way up in your
coding, but the compiler only "realizes" it now.
