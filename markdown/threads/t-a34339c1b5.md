To activate FRED, I would compile this program to PBSTR.DLL, using the Microfocus
product. I. the calling program, first (1) activate to dll, then (2) call the
entry point FRED.
(1) is only once needed.
* in COBOL:
- 03  someDllPointerVariable usage procedure-pointer.
- ...
- Set someDllPointerVariable to entry "PBSTR"
- (Some people code ... to entry "PBSTR.DLL", but the .dll extension is not
needed)
* in C/C++:
- LoadLibrary API function (which COBOL does behind the scenes)
* in Powerbuilder do whatever is needed to get the Library Loaded
(2) as many times as wanted:
* in COBOL:
- call "FRED" using ...
* in C/C++:
- activation of function: int FRED(char * lpszParmA, char * lpszParmB);
* in Powerbuilder do whatever is needed to "call" a subroutine named FRED.

Note the case (in?)sensitiveness of entrypoint names in COBOL/Powerbuilder.

To acitvate the DLL and to call the entry point are two separate actions. As you
wrote:

  "I can see by using my WINAPP utility that the PBSTR.DLL  is loaded into memory;
   but my entry point does not execute.  The system hangs."

When you want a dll with this one entry point, you need no entry statement. Instead
the program name (program-id) is the main entry point. Remove the exit program
statement and the entry statement.
When you want a dll with more than one entry point (which is more like what dll's are
meant for), then you need one or more entry statements. Still there is an extra entry
point named after the program-id, due to the nature of COBOL. If not needed you could
indeed code an exit program or even throw an error right after the procedure header
(as in your original code), because that's the code executed when the main entry is
called.
BTW: after the procedure division header, at least a paragraph header or even a
section header AND paragraph header should follow; this as standard COBOL (allthough
some vendors allow you to leave these out) and as such more portable.
