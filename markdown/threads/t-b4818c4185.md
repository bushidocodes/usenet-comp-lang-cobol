```
Just as a follow up on a current (and past) threads.

The COBOL (draft and past) Standard is written as a "formal definition."  It
just isn't the same TYPE of definition that any language (computer, natural,
before, or since) has ever used. <G>

Furthermore, (and this has been a recent re-occurring theme), it is QUITE
INTENTIONAL that some of the "ambiguities" that have been there (and grown
over the years) are kept in.

It is the view of the Standards committees (and I think this is correct -
from my experience) that most (not all) production COBOL programmers and
sites want the SAME results over CORRECT results.

What this means to the Standards developers is that "upward compatibility"
(even of ambiguities and errors) OFTEN is of the highest priority. (Some
corrections are made and included in the "potentially impacting" appendix -
but the Standard writers TRY to keep these to a minimum - and only do them
AFTER most compilers are already working the same way.)

Now, I know that there are some individuals (and even shops) that like
getting "better" and/or more portable results.  But even when it comes to
portability, it is frequently in terms of "please make everyone ELSE's
environment work like mine". Furthermore, some people want the "new stuff" to
be more rigorous, even if the old parts have problems.  UNFORTUNATELY, as any
of us who have ever tried to write (even the smallest) proposal against the
'85 Standard know, it is SO full of problems, that if you "touch" one
paragraph, you are liable to fall into multiple existing problems.

There is, BTW, an ANSI/ISO "formal language definition" language which is
approved (and as far as I know used by almost no one).  I think (fear? hope?
expect?) that you are more likely to see ALL operating systems written in
COBOL before you see the COBOL Standard using any of the "known" formal
language definition languages.
```
