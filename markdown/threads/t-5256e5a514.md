Much as I like PL/I, I would recommend COBOL for your garden-variety
business application, unless you're already a PL/I shop.

PL/I really shines in other types of applications that require a lot of
bit-twiddling, floating-point math, character string manipulation,
dynamic storage allocation, etc.  Processing records with fixed fields
and fixed-point numbers is what COBOL was designed for.

I use PL/I as a substitute for assembler in applications where ease of
coding, readability, and maintainability is more important than
squeezing out the last few machine cycles. I.E system programming, image
manipulation, Printer-dependent print datastreams, text processing,
etc.  If I was oing to do a payroll, I'd start with COBOL. (well, *I'd*
use PL/I, but that's a subjective choice).

The following is the reply that I got from one of my "usually reliable
sources" (who has a slight COBOL slant),

"> The main criteria are:
> - Readability of the sources

COBOL is much more English like, so if the programmers know English
than COBOL would be more readable.  (If the programmers do not know
English, then PL/I might be more readable)  You can't "hide" tricks
in COBOL like you could in PL/I.  PL/I has more flexibility to do
things like use the same name for a variable as for a function, which
is hard to read and maintain.

> - Maintenance of the source

COBOL is harder to be "tricky" with, so it can be easier to maintain.
See also the readability answer above.  In addition, there are many
more 3rd party vendor tools available to work with COBOL than PL/I

> - Difficulty to find specialized developers

There are more COBOL programmers in the world than for any other language,
including Java and C/C++.  Of course, that means more than PL/I programmers
too.

> - Quality of support from IBM

COBOL and PL/I are equal here

> - Language advantages/drawbacks/flaws of the language (built-in
> functions...)

Too hard to answer.  COBOL has built-in functions now, but PL/I might have
more.  This is a very subjective question.  What I might see as a flaw in
one language, another might see as an advantage.  Is the language to be
used for business applications or system-type code?  COBOL is the best
language
in the world for implementing business rules, while PL/I is better for
system
programming.

> - Limits in the management of  memory (Limits in working-storage size,
size
> of arrays...)

COBOL and PL/I are equal here

> - Arithmetic limits & precisions (integers, floating points,
packed-decimal
> variables...)

PL/I has the advantage here, though both languages support all of the most
commonly used data types.

> - Performance at execution (Quality of the compiler & optimizer)

The COBOL optimizer has many patents and is well known as very good, but
resulting code depends on the skill of the programmer, and I would say that
COBOL and PL/I are probably equal here.

> - ILC with COBOL, PL/I, Assembler (For example PL/I fetch limitations).

COBOL does not have the fetch from fetch restriction, and any COBOL program
can
be MAIN or SUB at any time.  COBOL can be made to look like C/C++, and can
be shared easily as common code with dynamic CALLs, common linkage
conventions, automatic start up of run-time library, 'normal' naming of
entry points (Entry point of program SUB1 is SUB1, not CEESTART like PL/I
and C)

While not PL/I vs COBOL, there's a real good PL/I vs C comparison that might
help you out as far as understanding what PL/I has to offer.  

It's called "Power vs. Adventure - PL/I and C" and can be found at:
 http://www.uni-muenster.de/ZIV/Mitarbeiter/EberhardSturm/PL1andC.html

While (IMHO) COBOL can't hold a candle to PL/I, PL/I did make big mistakes by
omitting built-in SORT and SEARCH verbs (SORT's are done via CALL's and
SEARCHE's are done via DO loops). 

PL/I supplies a much more robust programming language with features COBOL
should not be lacking at this late date - a "real" MACRO preprocessor,
user-definable functions, useful POINTERs, bit manipulation, STREAM and STRING
I-O, dynamic memory allocation, array functions, local variables, true
INITIALIZATION capability, default processing (e.g., files automatically OPENed
and CLOSEd in not done explicitly), etc.

BTW, the latest implementation eliminates the FETCH limitation noted above and
also provides a way for the more anal-retentive to reimpose COBOL-like
"features" such as reserved keywords.
