› Does anyone know how I can get a 9(08) date from the system

If you're running on a system that has Language Envrionment for MVS
running, you can make a LE/MVS call and get the complete date (I
assume there's no problem with a Cobol II program calling a LE/MVS
service). Check the LE/MVS programming reference manual if you have
LE/MVS running. I'm not aware of a Cobol II way to get the 8 digit
date, though. Course, if your shop is solely a VS Cobol II shop, you
probably won't have LE/MVS running.

William M. Klein wrote:
›
› If you don't have the LE run-time, but you are "current on maintenance" then
› look for a call to IGZE4DT (I think that it is its name, but I haven't
› looked this up - it might be IGZEDT4 - or something else like this)

Here's a quick example of how to use IGZEDT4:

01 MY-YYYYMMDD-DATE PIC 9(8) USAGE IS DISPLAY.

CALL 'IGZEDT4' USING MY-YYYYMMDD-DATE.

For the IBM VS COBOL II runtime library the APAR number is PN76666 and
the PTF number is UN83685.

And for LE/MVS the APAR number is PN84080 and the PTF number is UN92085.

APAR = Authorized Program Analysis Report
PTF = Program Temporary Fix

STL Time Machine Reports are available at:
http://home.att.net/~arnold.trembley

› Does anyone know how I can get a 9(08) date from the system
› using VS COBOL II?

› 

If your compiler is using intrinsic functions it's:

MOVE FUNCTION CURRENT-DATE(1:8) TO YYYYMMDD.
(where YYYYMMDD is defined item-level PIC 9(8))

if not then the hard way:

WORKING-STORAGE.
01 YYYYMMDD-AN.
05 CC PIC 9(2).
05 YYMMDD.
10 YY PIC 9(2).
10 MMDD PIC X(4).
01 YYYYMMDD REDEFINES YYYYMMDD-AN PIC 9(8).

PROCEDURE DIVISION.
ACCEPT YYMMDD FROM DATE.
IF YY IS LESS THAN 60
(or some number you know your company won't use for some
time in the next century or you'll be gone through
retirement/fast_getaway before it trips with errors)
THEN
MOVE 00 TO CC
ELSE
MOVE 19 TO CC
END-IF

Hope this helps,

Boyce G. Williams, Jr.
