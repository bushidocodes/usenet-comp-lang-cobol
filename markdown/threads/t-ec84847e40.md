The only thing I have ever used FUNCTION MOD is to check to see whether
or not it is zero.

The following are examples of other values (from an IBM on-line manual):

The returned value is argument-1 modulo argument-2. The returned value
is defined as:

argument-1 - (argument-2 * FUNCTION INTEGER (argument-1 / argument-2))

The following table illustrates the expected results for some values of
argument-1 and argument-2.

Why would someone want a return value of 4 or -4 above?   They had to
design that function to work that way for a purpose, I just don't know

05  day-of-the-week    pic 9.

    compute day-of-the-week =
      function mod (function integer-of-date (yyyymmdd), 7)

I have a trust account progamme in which my clients are given numbers which are 
modulo 11. It was originally programmed in Softbol, a variant of DEC's CoBOL.

The idea is that when you enter a number, it does a modulo 11 validation check, 
and if you've miskeyed the number, it should (!) get rejected.

This is why they are used for IRD (IRS) numbers etc. I believe some 'mission 
critical' applications may even use both modulo 10 and modulo 11 checks.

I found a lot of info from the IBM manuals, as all I had at first was the data, 
and had to work out the record formats etc, but thanfully the users manual gave 
some hints and referred to modulo 11 numbers.

Using MOD this self check digit system is easy to implement.

There has been a lot of mathematical analysis of check-digit schemes
in recent decades in an attempt to discover the best formula for
detecting common transcription errors. The most common errors are, of
course, mistaking one number for another ("7234" instead of "1234")
and transposition errors ("1324" instead of "1234").

I believe one of the most common techniques is (abcd... are digits in
the number):

CheckDigit = MOD { (Sum [a + 3xb + c + 3xd...]) 10}

That is,
Sum the odd-numbered digits
Sum the even-numbered digits and multiply that sum by 3
Add the two sums
Take the modulo-10 of the total of the two sums
Attach the result to the end of the original number.

i.e.:
1234 = 1 + 2x3 + 3 + 4x3 = 22
Mod (22 10) = 2
New number = 12342

Test:
7234 check digit is 0 which differs from 2  ***FAILURE***

1324 check digit is 4 which differs from 2  ***FAILURE***

Modulo-11 may catch more errors than modulo-10 -BUT- Modulo-11 creates
another problem. The Modulo-11 answer may sometimes be TEN (10) (in
the case of books, represented by "X" ) which, in turn, complicates
data entry, barcodes, and all that sort of machine stuff.

I assume I am one of "they".  We did it that way because that is the
mathematically correct way to do it.  Also, the mod functions in other
languages and in the ISO mathematical function guidelines do it this

it has to be done in the correct fashion.  Just because it doesn't seem
right for business does not mean it is not right.  I would assume
business would want mathematically correct answers as opposed to
something incorrect.
