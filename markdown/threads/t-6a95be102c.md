gvw··.@ix.··m.com (G Moore) wrote:
›
› anyone know what a pic usage index means, and how to manipulate it?

Recall that an index is defined within the definition of the table in the
DATA DIVISION, for example:

05 FILLER OCCURS 100 INDEXED BY WS-NDX.
10 FIELD-1 PIC ...

Notice that there is no PICTURE clause for WS-NDX. An index is an
"opaque type", that is, the compiler writer is allowed to do anything he
wants. Generally, this allows the compiler writer to generate more
efficient code for indices than for subscripts, although there are a few
machines with hardware subscripting where subscripting is as efficient as
indexing. Indexing is never less efficient than subscripting.

A subscript is usually an ordinary elementary item somewhere in the DATA
DIVISION, for example:

05 WS-SUB PIC S9(9) COMP.

You manipulate subscripts just like other variables, for example:

MOVE 1 TO WS-SUB.
MOVE FIELD-1(WS-SUB) TO ...

However, the MOVE verb does not work for indices. The SET verb is there
to remind you that an index is not a subscript. For example:

SET WS-NDX TO 1
MOVE FIELD-1(WS-NDX) TO ...

Now, from time to time you may wish to remember a subscript. Perhaps you
found an entry in a table that you are going to use later. With
subscripts, it's pretty obvious:

05 WS-SUB-SAVE PIC S9(9) COMP.

MOVE WS-SUB TO WS-SUB-SAVE.

Then, later on, when you want to use it again,

MOVE FIELD-1(WS-SUB-SAVE) TO ...

But what if you want to save an index? This is where USAGE IS INDEX
comes in. You set aside a save area, for example:

05 WS-NDX-SAVE USAGE INDEX.

Notice again that there is no PICTURE clause. In order that the compiler
writer be able to take advantage of the features of your platform, you
are not allowed to know the internal representation of an index -- even
the number of bytes in a USAGE IS INDEX varies from platform to platform!

Now you can say

SET WS-NDX-SAVE TO WS-NDX.

and later on you can say

SET WS-NDX TO WS-NDX-SAVE.
MOVE FIELD-1(WS-NDX) TO ...

Obviously, this is way oversimplified, but it might give you a start.
For example, see if you can figure out how to pass an index as an
argument and use it in the called program.

in a museum, because you can't know what software engineering really
means unless you've sat behind the throttle of the express master file
update run with a full head of steam in the boiler, a full load of data
in the punch cars, and seven clear tracks ahead.

Chris Westbury wrote:
›
› Indexing is never less efficient than subscripting.


Oops! Any time the ratio of the number of times that the index is
converted to/from, or compared with, numeric data items, is sufficiently
high relative to the number of times the table is referenced using the
index, you will get a crossover point beyond which indexing is *less*
efficient than subscripting. The exact point is compiler and CPU
dependent, of course.

I will agree that any *single table reference* using an index *should* be
at least as efficient as subscripting. If not, the compiler is flawed.
