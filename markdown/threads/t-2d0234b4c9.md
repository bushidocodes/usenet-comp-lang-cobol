```
I guess there are two different definitions of "runtime".

I knew some applications need a runtime environment to run programs.  This
changes the cost structure.  I wasn't aware that there were CoBOL compilers
in this category.

The other definition concerns itself with called subprograms.  One option
when compiling a program is to pull in the sub program and compile it to
make a real efficient program.  If you change the subprogram, the calling
program doesn't change - this is a compile time call.

A different option is for the calling program to call the program at run
time.  If the sub program has been changed, that is the version called in.
```

```
Some compilers compile to native code and others compile to psuedo-code
(p-code or byte code).  Some can do both.

With a p-code compiler, it compiles to a file that is interpreted by a
runtime interpreter, similar to Java.  Advantages are that the p-code object
file is usually much smaller than an executable file and can be portable to
any number of platforms.  For example, we use Acucobol and compile to
p-code.  We compile on a IBM AIX system, but our end users have a variety of
systems - Windows, HP-UX, Sun, SCO, Linux, DG, etc.  We don't need to own or
have access to any of those types of systems to do a port - we don't even
need to recompile.  We just order the proper runtime from Acucobol and our
p-code object files work on the desired platform.  (Acucobol does now have
an option to compile to native code on some platforms.)

Even if a compiler compiles to native code, it still may require a runtime
(for example, DLL files under Windows).  This would be similar to Visual
Basic - it compiles to native code, but still requires the VB runtime to be
installed on an end user system (a series of DLL files).  MS doesn't charge
for VB runtimes - don't know if any Cobol vendors who generate native code
charge for runtimes.
```

```
Very, very, very true - it would be completely reasonable to consider
WINDOWS itself as a "runtime" for a whole lot of applications...

But on a more serious note..."runtime" packages are usually more
straightforward to work with insofar as the developer is more isolated
from the operating system than if they need to produce full .EXEs. As
mentioned elsewhere, the developer can just produce the intermediate
code modules, smack in the requisite "runtime" and evrything will just
work fine!!! (if only...).  With a .EXE there are often problems with
linking (which libraries to use, where are they etc.) and the
different operating systems and versions thereof often have different
linkage requirements. In theory the compiler writers do this work for
the developer when they create the appropriate runtime packages.

Probably, because COBOL has tended to be a cross-platform product,
most COBOL compilers has gone this way. On the other hand other
laguages, e.g. C/C++, are often (perhaps thinking of Microsoft here!)
produced with a specific target operating system in mind; in that case
they can rely on that environment to provide the "runtime" environment
without ever calling it a "runtime". Another well-known cross-platform
language has already been mentioned as having an intermediate code
type structure and a run-time: Java.
```

```
A few "clarifications" on LE (Language Environment)

1) Originally, it was a "separate" product, it is now "bundled" into the
operating system. If you have a current release of OS/390, z/OS, or VM fully
installed on your machine, then you also have LE installed.

2) As far as the Microsoft world goes, it is QUITE similar to the .NET
environment (i.e. it provides a "language independent" operating
environment)

3) There are two VERY different types of "modules" in LE
  A) Those that are well documented and explicitly CALLable by "application
programs" (these do things like date manipulation,  etc) - These routines
may be used by ANY supported LE programming language.
  B) those that ARE used by the compiler generated code, are named but not
documented, and provide functionality that COULD be included in the
generated code - but for various reasons are not.  These are specific to one
(or in some cases a few) programming languages.

4) There are "flavors" of LE for OS/390, z/OS, VM, VSE, and OS/400.  (the
status of OS/2 and/or Windows takes more time that I want to include in this
note).  This does provide IBM platform independence.

5) If you use ANY of the currently supported IBM COBOL, PL/I, C, or C++
compilers, you *must* have LE available at run-time or your object code will
"die a horrible death".

6) As far as COBOL goes, in "ye goode olde dayse" - there was a feature
"NORES" that allowed your object code to be "bundled" with its required
run-time support and then to run on a machine without any "run-time" support
programs.  That feature is REMOVED from the LE-enabled COBOL products.
```

```
Which doesn't make it a 'cross-platform product', nor does it stop it
producing hard code for the CPU.

All high level language systems use libraries to provide generalised
routines for such areas as I-O and mathematical routines.  These
libraries can either be linked into the program statically or can be
accessed dynamically at 'runtime'.  The LE is a set of shared dynamic
libraries.

The 'runtime environment' of products such as RM, Accu, MF Cobols are
more like the Java VM in that a compiled program has no hard coded CPU
specific code in it and it can run on any platform (including an IBM
mainframe) that has a runtime environment (VM).  Other systems that have
been doing this for a quarter of a century or so have been: UCSD
(Pascal, Modula/2, Fortran), BOS (MicroCobol), Pick (Basic), more
recently Java, and now (or indeed any year 'soon') MS's .NET.
```

```
Assuming that your program is platform independent (e.g. doesn't make
any OS calls, etc.) then you can compile a program to dynamically link
to the runtime environment or compile it to statically link to the
runtime environment. (Basically this is compiling to intermediate code
or to a standalone EXE.) The main tradeoff is portability vs. size. If
you compile it to dynamically link to the runtime environment then you
can move the compiled program to another platform, install the
appropriate runtime environment for that platform, and your program will
still run. But if all your programs run on the same platform then you're
installing a lot more than you need (for one program) when you install
the runtime environment.

Using an older version (4) of MicroFocus COBOL Workbench as an example,
a program statically linked to the Windows runtime environment is
approximately 500KB larger than that program compiled for dynamic
linking to the runtime environment. I don't recall the minimum size of
the runtime environment, but 50MB sticks in my mind. Assuming that is
correct, it's more efficient (in terms of space usage) to statically
link your programs if you have fewer than 100 of them.
```
