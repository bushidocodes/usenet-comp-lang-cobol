Yes application starts with a 'trigger' program (procedural COBOL), which gets
the MyApplication and MyMenu as objects and passes handles between each.

    CALL 'MATERIALSEDIT' Using etc...

but in OO I have :-

    INVOKE MATERIALSEDIT using etc....

Note that because OO syntax is 90% plus Procedural COBOL, you can both
CALL(procedural program) and INVOKE(a class) in an OO configured application.
Primarily for consistency of style I invoke classes - but from Pascal, you will
be aware that there are other advantages.

From the menu I could also invoke PrintMaterials etc...., which in turn would

> 2)  Declarations of objects are made with an "object reference" type.  Is
> there object typing available, or must all of these references be made with
> a generic object type?

Yes - an object class/dialog/collection etc. is referenced using an OBJECT
REFERENCE. There are properties in the COBOL 2002 syntax which allow different
features on invoking - but as yet not part of M/F. Just as I'm sure you can do
in Pascal, I can pass the reference/handle for os-ObjectA via Linkage Section to
other classes where it becomes ObjectB, C, D etc... all unique handle references
but all 'pointing' to the one object. Destroy Object C and it destroys the rest.
Back to the duality in COBOL

    01 Flag                    pic 9.
        88 ResultOk        value 0.
        88 RecNotFound value 1.
        88 etc...

either MOVE ZERO TO FLAG or SET RESULTOK TO TRUE

OO does the additional :-

Linkage section.
01 lnk-ObjectA                         object reference.

SET OBJECTB TO LNK-OBJECTA ('receiving' object is FIRST)

Again early version - J4 have now produced a TR(Technical Report) which can be
added to the Standard. I thought they had finished, but apparently what I saw
was not so - they were discussing this in their current May meeting. One of the
beefs about COBOL - it takes too bloody long to crystallize features - this one
has been going on for over two years.

M/F offers the interim (a) "finalize" a single, (b) 'deepfinalize" a collection
and (c) "deepFinalizeWithDepth" an n-dimensional collection. The purpose of the
J4 paper is to introduce the spec for auto garbage collection.

Copyfiles are a big feature of COBOL - I have 150 class source files and 141
accompanying copyfiles. You can get real nifty in COBOL using the syntax COPY
REPLACE. Take the one you have hit on :-

------------------dilgretn.cpy ------------------------------

*> "Dialog return" -
*> Parameters returned from MyDialog - indicating receiving
*> Method Name plus Table Line and Column positions when
*> appropriate

01 (tag)-ReturnValues.
   05 (tag)-Line                    pic 9(03).
   05 (tag)-Column                  pic 9(03).
   05 (tag)-length                  pic x(4) comp-5.
   05 (tag)-text.
      10 occurs 1 to TextLength
         depending on (tag)-length  pic x(01).

----------------------------------------------------------------
"Sending" CLASS - MYDIALOG
*>--------------------------------------------------------------
Method-id. "returnEntryObject".
*>--------------------------------------------------------------
Local-storage section.
copy "dilgretn.cpy" replacing ==(tag)== by ==ls==.

*> Using "replace etc... - each field prefixed "(tag)" becomes "ls"

01 n                             pic x(4) comp-5.
01 ls-object                     object reference.
01 ls-string                     object reference.

Procedure Division.

perform varying n from 1 by 1 until n > MaxEntryFields

    invoke os-Collection(1) "at"
           using n returning ls-object

    if  os-CurrentObject = ls-object
        invoke os-CurrentObject "getText" returning ls-string
        invoke ls-string "trimblanks" returning ls-string
        invoke os-CurrentObject "Getlength" returning ls-length
        invoke ls-string "getValueWithSize"
             using ls-Length returning ls-text
        invoke ls-string "finalize" returning ls-string

        invoke os-Collection(2) "at"
             using n returning ws-MethodRec
        move ws-Line   to ls-Line
        move ws-Column to ls-Column
        invoke os-Caller ws-MethodName using ls-ReturnValues
        EXIT METHOD
    End-if
End-perform

End Method "returnEntryObject".
*>--------------------------------------------------------------

*>---------------------------------------------------------------
