I do not see a cross post listed for your current inquiry, so I encourage you
to take note of 
  comp.unix.programmer
and other related newsgroups which will get you connnected to AIX experts. 

Your questions are definitely welcome here, the interdisciplinary aspects of

A few meager comments is all I can offer.  The SET commands, if I understand
you, are setting environment variables. I may be going too far to say that
environment varibales started in UNIX, but for sure Mr. Bill copied them from
UNIX into DOS (and its descendants, WindowsOn&On).

UNIX makes very heavy use of environment variables. You won't have any trouble
here, but there can be a few things to learn.

First let me comment on the .BAT file concept. That kind of thing is present in
nearly all operating systems.  In UNIX the concept is known as scripts. But
first you need to know that the operating system will execute from the command
line either a command directly entered or a shell that is to read and interpret
a script named on the command line.

A script file usually contains an identifier of the shell it needs encoded as a
comment on its first line. (In the absence of such a comment, there is a
default shell).

You will come to know that there are three basic shells the Bourne Shell, the
Korn Shell and the C Shell.  Generally, they are similar.  They process a
syntax very much like a simplified programming language. And the environment
variables are the variables the shell script syntax uses.

However, you will find that UNIX shells are more like an execution of MAKE in
DOS.

DOS .BAT files have an nasty way of exiting stage left with no return should
you name another .BAT file as a command. UNIX shells just 'shell' to the next
shell script with the intention of returning control to the original shell.  

And these shell invocations fully establish entirely new executable
environments, which environmens are then POPped, as it were, upon return to the
invoker.

And exactly what happens when the environment is established varies from Bourne
to Korn to C shell. This you need to get a handle on right away.  In some cases
your exisiting envrionment variables are not simply instantiated automatically
into the next lower shell, but are exported to it. There is a difference.

The syntax of the UNIX shells are somewhat the same. They are very advanced
compared to DOS .BAT files.  You will be able to convert, but you need to
really understand the syntax; the effort will be largely manual.

Your best bet is to simplify wherever you can, to reduce the amount you must
learn to get this thing up in the new environment.

Even if you have "no UNIX experience", don't worry, there is plenty of help
around. The AIX systems are excellent. Usually there are mountains of tutorial
matterial online. You should have Infoseek (pardon me if that is not the exact
name).  It is useful, and has very powerful search capability.  The man pages
are also very complete in AIX.

Get help early to make sure you have all necessary packages installed and 
files systems mounted for the man pages and Infoseek.

I think you know this, but I'll say it for completeness, AIX is very standard
UNIX so nearly any of the books on UNIX will be useful to you as a beginner.
(It does not have to be AIX specific, you will find all the specifics you need
right on the machine). 

You also asked about parms.  Again DOS is modeled on UNIX for parm processing.
But really, the whole thing is up to your runtime environment. The compiler
documentation should clarify how the parms are passed into your COBOL program.
But you will need to describe the actual situation with the BASIC front-end, if
that survives, to get comments on its combination with COBOL.

There is a possibility that your DLL implementation strategy is dead, but I'm
going beyond my knowledge here.

Also FYI, Microsoft Visual BASIC, is not an IBM Visual Age product. Exactly
what BASIC do you plan to use on the AIX box?

Beware of the use of quotes (single or double) in parms, that is a learning
curve unto itself. You could get variance in behavior across the UNIX shells,
and maybe none of it would match what you did in DOS .BAT files with quotes on
parms.

The standard for parms in DOS is to encode a slash in front, as in /parm, in
UNIX the standard is a dash infront, as in -parm.  You will find that many DOS
programs accept -parms (UNIX format) as well as /parms (DOS format).  But not
all. This was to help bridge the work place to UNIX programmers. (This is
especially true with developer programs like compilers, linkers, debuggers).
UNIX  systems defintely do not do this, because the slash looks like a
directory name, /dir/subdir/file.nam.  So if your parm expressions survive,
they will have lots of little fixes.

There is a large amount of UNIX tutorial information on the net.   If you need
URLs ask here or on the UNIX specific newsgroups.

UNIX has a steep learning curve. You and your manager need to understand it
will take time because UNIX is vast! Yet the part of UNIX you need is probably
small. So keep; asking yourself about that when you get bogged down; what is
the minimum part need to make progress?

How much use does your application make of Windows95 itself? Does Micro Focus
offer adapters, or converters to the X Windows system?

Generally you can do anything from the script that you can do from the command
line. Just like a btach file.  So you can invoke a command. You can set an
environment variable.  You can invoke a command (program) passing myriad
parameters; some file names, some switches, and the script shell will even
resolve command line textt $ENVYOUNAMEIT to what ever you set,as well as do all
the filename globbing that DOS copied from UNIX.


You can do the exact equvalent of 
" SET TFPGMLOG=C:\XFRMRV51\MFORGW\TFPEDIT\TFPGMLOG.TXT "
in UNIX.

How you are hooking TFPGMLOG in as the DDNAME is not clear to me (that may be
the Compiler that does that magic). But it is not difficult to believe that a
similar hook is going to be available to you with the UNIX compiler. That is
actually a _convention_ of your compiler, not an OS feature.

You have long file names, which for complex programs, in itself tends to lead
away from explicit entry of all the file names on command lines (whether
entered directly or from within a script). Although UNIX is fully capable of
passing file names in as parameters.

list of file names. A portion of your posts suggested to me that a small part
of your system does this, but even if not done yet you could consider it. You

things with the file names. But really that is what shells are for. So my point
is, that if you had this type of functionality in your BASIC front end, or even
in some of your COBOL application initialization logic, then convert it to
shell scripts.

In a small way what BASIC on DOS machines does (only in part now) is that it
provides the text substitution logic of a UNIX shell, the text scanning and
replacement function of UNIX commands grep and sed and the popular program perl
(which can do sockets like some DOS BASICs).

Any serious orientation effort in UNIX will get you to grep and sed.  If you
hace anything like a control file that survives in your restructured system,
you may not need much more than that.

But actually I would be radical.  Simplify, big time. This is to reduce the
amount of learning you must do to get to the end. Focus on one shell, as your
compiler vendor seems to be leading you to.  Try to do every configuration and
setup with shell scripts (this so you don't have to learn too many tools).

Generally, these are just notions, not a golden rules: if something comes out
of the old BASIC front-end code, put it into a script, do not place it into
COBOL.  

Try to avoid changes to the COBOL code. Atleast do not add environment
management to the COBOL programs (although extract it from COBOL source

You will find that shells are like computer languages in that people can be
very emotional about them. Don't get too drawn into the preference war. There
are real differences. Your plan, IMHO, should be to stay simple. Try to use one
only for no other reason than to limit the number of learning curves.

But a thing you need to grasp is that shell scripts invoke other shell scripts
(with eventual return back up the stack). These invokations just look like
commands in the scripts, but they are major junctions.

Each time a script starts it can use a different shell, at that exact point the
conventions for passing environment variables can be somewhat different from
Bourne to Korn to C shell (this is just a caveat). Basically UNIX systems spend
all day bouncing around from shell to shell, it is all wonderful, lots of fun,
but just be aware that the environment variables can be tricky, both on the way
in and the way out.

It is important to know for example that environments can be copied upon
entering a shell.  You could then proceed to manipulate the heck out of your
environment variables for your COBOL system run.  Adding and changing variable
that represent file names. Now everything is done, we are ready, ... script
ends, ... shell has no more work, ... shell ends, ... environment is popped!,
... no lasting effect, ... you bound back to the higher script under another
shell (or just a different instantiation of the 'same' shell), ... all your
vars are gone.

(This goes on within make in DOS, but not really in .BAT files, where the
system just careens forward, never popping back, and never losing the effects
accumulated in the environment..)

So you end up inexorably considering configuration files (perhaps slightly more
frequently known as control files in COBOL realms outside of UNIX), which have
files names and possibly parm text.  These files, when editted (actually piped
into temporary work files), will 'permanently' store your 
changes for the current run.

This leads to grep and sed.

Life is magnificently fluid in UNIX, resource control file (*.rc) and .config
files are basically the answer to the popping environment phenomenon (that is
just a genralization for the current discussion).  Note that you can 'edit'
(with grep and sed)  script files in any number of ways just before executing
them.

But keep in mind the notion of simplification.  Just because UNIX can pop and
environment, does not mean that you have to do that.  Just get it all clear
before it burns you. Truth is that you could create a cascade of scripts to
mimick the brain dead simplicity of DOS .BAT files, ... just keep descending
into deeper scripts, passing the environment in the manner you are comfortable
with.  Design your script so that there is nothing to return to after the
invokation of the lower script (that is no more commands in the higher script).

You don't have to limit yourself to this technique. But you are doing a
conversion, you need to get this thing up: a cascade down scripting sequence
can help you get started. You can become more of a virtuoso later.

Anyway, I know I have said too much already, and it is dinner time so that's
all you get.  Hope it helps some. Please continue to post and let us know what
dragons show up.

Yes, you can do this in Unix shell. The Bourne Shell for this is

TFPMLOG=/path/path/path/path/file.ext
export TFPMLOG

(the "export" command says that sub-processes will see this variable -
without

In Korn Shell, you can just say:

export TFPMLOG=/path/path/path/path/file.ext

Yes. Note that a "shell script" is really little more than a
non-interactive
command line shell, so you can try all of this stuff yourself
interactively on
the command line:

$ sh
(creates a new shell)
$ XYZ="abc"
$ sh
(creates another new shell)
$ echo $XYZ
(nothing echoed XYZ is a local variable in our parent shell)
$ exit
(return to previous shell)
$ echo $XYZ
("abc" echoed)
$ export XYZ
(declare XYZ as an "environment variable")
$ sh
(creates new shell)
$ echo $XYZ
("abc" echoed - it was inherited this time)
$ XYZ="def"
$ exit
(return to previous shell)
$ echo $XYZ
("abc" echoed - environment variables are only inherited "forwards").
$ exit
(return to login shell)

> I hope the DLL strategy isn't dead !  I've yet to get the Micro Focus software

You don't (as far as I can see) say what version of AIX you are using.
This has
a bearing. The standard for this sort of thing on Unix is "SVR4 Shared
Objects"
(which usually have the extension ".so"). However, AIX 4.1 only supports
a
proprietary "shared library" mechanism which has simiarities with DLLs.
From AIX 4.2 onwards, SVR4 shared objects are supported (as well as the
AIX-only
"shared libraries").

With regards to the support for these files in Micro Focus COBOL, the
product
you have will support the AIX "shared libraries", but not directly - you
need
to drive the system linker yourself to build them (use "man ld" to see
the 
manual page for the system linker). It is not straightforward.

The next release of Micro Focus COBOL (currently in Beta and due out on
certain
platforms next year) provides full support for SVR4 shared objects and
makes
creating them incredibly simple. I cannot say whether AIX is a supported
platform.

It sounds like you've got your work cut out just learning the Unix side
of
things and putting together the front end etc. A reasonable way to
proceed
might be to compile your COBOL code to .GNT format until you get things
together, *then* worry about packaging to shared library format - you
might be
in a position to take advantage of the newer Micro Focus release by
then, if
it is released on AIX. At the very least, you'll be more familiar with
Unix
and will not have such a problem fathoming out the AIX "shared library"
format.

Going with shared objects with the current compiler
is not for the faint of heart, although if you are
only going to be targeting AIX vis-a-vis other UNIX
flavors, then it's much eaiser, but you will have a
learning curve.  See the $COBDIR/docs/dynlink.1 text

much easier to go with dynamic linking:

   cob -i myprog.cbl

runs the checker that builds an intermediate file,
myprog.int that can be interpretively executed by any MF
COBOL Run Time System (RTS) on any platform (SLOWLY).

   cob -u myprog.int

runs the Native Code Generator (NCG) that builds a
generated file, myprog.gnt, a processor specific
version of the program that can be native mode
executed by any MF RTS on the same processor; e.g.,
Windows NT or SCO UNIX on a Pentium box.

Both .int and .gnt files are executed by the cobrun
command that fires up the RTS and scans the directories
defined in your COBPATH environment variable to find
the .int or .gnt file; e.g.,

   cobrun myprog

Shell scripts execute in a child shell instance, with an
environment inherited from the parent shell.  The child
cannot change the parent's environment by design.  However,
there is a shell command, . (a period, pronounced "dot") in
Bourne, Korn, and POSIX shells, and source in C-shell, that
treats the file of commands defined in its argument as
though they were input to the shell processing the source
command.  The /etc/profile, $HOME/.login, and other
shell/session initialization files are all processed by
this mechanism.  Shell source files do not need to be
executable, and probably shouldn't be.

Yes, when "running a script" you are really invoking a "non-interactive
command shell" - that is, a command shell which is taking it's commands
from a file. It's no different to typing "sh" or "ksh" on the command
line and getting a new level of *interactive* shell.

Bourne and Korn shells have the "." command to say to the current
shell "execute the named file in the current process (ie, as if each
line had been typed at the prompt)".
So if you have the script "setvars" in the current directory, the
shell command ". ./setvars" will execute "setvars" in the *current*
shell, then return to interactive mode. Note that this command also
works for non-interactive mode, where it will resume execution of
the script after the "." (rather than going interactive). This
effectively lets you include common "library" code into a script.

Again, in both Bourne and Korn shell: The command line parameters can
be accessed with the "numeric variables" $0 thru $9. $0 is the name
of the script itself, $1 is the first real parameter. $# tells you how
many parameters have been passed. So you can say something like:

cobrun TFPEDIT $1

then run the script with

scriptname FORM==SPGDUPLY

or, try:

cobrun TFPEDIT "FORM==$1"

and run it with:

scriptname SPGDUPLY
