This is getting silly.  There is probably not a construct in the entire
language that cannot be used to save memory, and the idea that redefines
CANNOT be used to save memory is obviously wrong. That was never the claim
that I made with the original post.  The original post was in response to
the following:

> > > This memory sharing or saving feature can cause much more trouble than
> > > saving several bytes of memory.

My response was that REDEFINES is not "a memory saving feature", and that I
had never seen it used simply to save memory.  It is a way of refering to
the same data using two different descriptions.  The clause "RESERVE NO
ALTERNATE AREA" was put into the language to save memory.  The clause
REDEFINES was put into the language so that the same data could be described
in several different ways, NOT so that the same memory could be used for
several different purposes.

The exact same verbs in Cobol use data quite differently, depending on how
it is described.  If there was no way to described the same data twice, in
different ways, then it would be VERY awkward to USE the same data in
different ways.  That is the purpose of redefines, to allow that. It is not
a construct that was designed as a "memory saving feature" ... it's purpose
is quite different.

The correct use of just about any language construct, from the ADD verb to
the logical OR, can be used to save memory.  That is simply the nature of
programming.  To get back to the original statement, I will respond again.

> > > This memory sharing or saving feature can cause much more trouble than
> > > saving several bytes of memory.

So can *any* language feature, BUT that is not the purpose of the REDEFINES.
The correct purpose of the redefines is to redefine the same data several
different ways, so that you can use it several different ways.  If you think
it just a way to save memory, and that is the way you are using it, then
yes, such a usage would probably cause more problems than it solves.

Donald

How does that save memory?  It is simply describing the data buffer several
different ways. The memory used still has to be the amount used by the
largest record.

There is nothing to stop you from wasting memory by declaring four or five
buffers for one file, then only using one at a time. I can assure you,
though, that READ INTO A, B, OR C, depending on WHAT YOU ARE GOING TO READ,
does not work.  You have to read it into a buffer first, then decide to move
it somewhere else.  A single file has to have a single buffer. Period. What
you do with the data after the read is another thing altogether.

The redefines is used to describe a single area in several ways.  If you
want to create multiple copies of the data to do the same thing, then there
is nothing to stop you, but it is the making two copies of it that *wastes*
memory, not the other way around.  You are using the REDIFINE to avoid
having to make three copies of it, having already wasted one buffer by
implimenting a programming practice founded on a specific platform.

Donald

I wouldn't agree, especially on the READ INTO.  READ INTO implies a WS Field
is being employed, which means that for every different type of record, you
would need another matching field of matching size to move the record to.
(ie READ INTO REC-A, IF REC-A IS TYPE-1 MOVE REC-A TO REC-1 ELSE etc.).  As
a result, you end up needing X-size-of-record * Y-potential-types bytes,
rather than just the original record.  Case in point:

I work for an insurance company that keeps policy information in up to 16
different types of 300-byte records.  Not redefining would require that I
set aside 5100 bytes in working storage on the odd chance that I encounter 1
of each type ( (16 + original) * 300).  To make matters worse, some of the
main types are further broken down into sub-types of the main type, which
means potentially more records if I don't redefine the main types.  But by
redefining the original READ INTO Record, I use only 300 bytes and can
process accordingly.
