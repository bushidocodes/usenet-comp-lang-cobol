### Data on disk but not in cache (MF on netware 4.11)

- **From:** Scott Dickson <scottdickson@ibm.net>
- **Date:** 1998-11-04T00:00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<3640741A.8517C97F@ibm.net>`

```
We are seeing a record returned with all NULLs in a successful read
after successfully writing the record.  The data is being read from
cache, no disk io takes place (as far as we can tell).  The problem
persists from more than one workstation (going after the same record).
Upon rebooting the server, the read now returns the correct data.

The server is a new Dell Poweredge 4200 running dual Pentium II 333,
Netware 4.11 SMP (with smp installed).  Support pack 5b is installed.

The client is running a program developed with Micro Focus Cobol using
there filehandler.  The program has been running for years without this
NULL record problem occuring.  The problem first appeared 4 days after
the new server went online and has occurred 5 times (in a two week
period).

If we had to take a stab, our guess (which may be dead wrong) is that
somehow the cache is not updated even though the disk is.  Upon reading
the record, the OS is attempting to return the record from cache and is
therfore returning the wrong data (NULLs).  Because of a lack of
understanding on what is in control, we have no basis for how this could
happen (or even if it is at all possible).

Anyone have any idea where to begin?

Scott Dickson
```

Scott - I had a similar problem using the MicroFocus product itself - 
never found a solution.

Running a Novell 4.1 server - Windows 95 client, using Novell Client 
32 drivers.

We could compile a program, and if SOME OTHER workstation looked at it
it was fine, but if the workstation that created it, copied it to 
another drive, an "incomplete" version was copied.

Here's how we found out.   When you comile with MF there's a header in
the .INT - type as in TYPE XXXXX.INT and you will see what I mean.  If
a compile is not successful this header is not updated with the 
information you see when you type the file.  

This was happening.  The compile was complete, the header was updated 
- on the Novell disk - other workstations could see the updated header
- but the machine creating the data could not.

It was fun finding this  - started out as corrupt INT files being sent
to users of the software - but looking at them later on the disk, they
were fine.  Just when they were copied quickly to the backup library -
the copy was wrong the orig was OK.

Now, with a DOS client the problem did not exist.

Is any of this similar to what you are seeing?
