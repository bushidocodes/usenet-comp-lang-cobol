(all original stuff snipped - but I think I can "help" with a couple of
issues)

1) Let's look at what the REAL coding problem was (and why the program - as
it executed under OS/VS COBOL had a "bug")

If you have

   Sort XYZ
      Usign ABC
      Output Procedure   LMN
        .       <<<<< notice this period or full-stop
    Do something else

What this means is *supposed* to be the same in the old COBOL and the newer
COBOLs - however, (as pointed out before) in ANS'74 COBOL (e.g. OS/VS COBOL)
"LMN" was supposed to be a SECTION while in newer COBOLs (VS COBOL II, or
IBM COBOL for-this-and-that) it can be EITHER a section or a PARAGRAPH.

But the meaning is the same; it says -
   "Perform LMN" exactly once (no more, no loop, no until, no nothing else)"
Then come back and "do something else".

If LMN is coded as:

LMN Section.
   some code with or without a paragraph header
LMN-exit.
    Exit.
 Another-paragraph-header.
       More code
 ...  end of program

Then, the SORT Output Procedure has been told to PERFORM *everything* under
the LMN Section header.  Remember that LMN-Exit is a paragraph-header and is
NOT a section header that would "terminate" the LMN Section.

Therefore, if you change
   LMN-exit to
   LMN-Exit Section

       *OR*
you change
   LMN Section to
   LMN (without the word SECTION)

Your SORT output procedure will *only* perform the one paragraph or section
that you want it to.

  Is this clear what the original problem was - and WHY adding the word
"SECTION" solved the problem?

        ****

2) Sort ERROR MESSAGES -  where do they go and why can't I always find them?

  - As I indicated previously, there WAS a bug that IBM "fixed" (but that
your site may not have applied) that was stopping IBM from "putting" out
SORT messages (good, bad, or whatever) in certain cases.  This has an EASY
solution - get your shop to apply the current maintenance to your run-time
and you will CORRECTLY get the messages in some output file - for your
debugging purposes.

 - COBOL (VS COBOL II and later) have a compiler option called "OUTDD" which
allows you to specify which DDNAME you want COBOL "DISPLAY" output to go to.
If you use this compiler-option and set it to something LIKE

   OUTDD(MYMSGS)

then, you will get COBOL messages in "MYMSGS" and get OTHER products'
messages in the "traditional" SYSOUT DDNAME.  Some sites find this useful -
and others hate it.

Alternatively, LE has a run-time option (MSGFILE) which can ALSO allow you
to specify the name where LE messages go.

Finally, SyncSort (and possibly DFSORT) allow you to specify in a $ORTPARM
statement, a DDNAME for SORT messages.

Bottom-line:
   Everyone WANTS to tell you what is happening - but they all "default" to
something like SYSOUT while allowing you to "confuse yourself and your
operators" with various alternative DDNAMEs.  I *highly* recommend that your
shop establish a "standard" for who gets which DDNAME and try to enforce it
as much as possible.

    ***

Yes, DFSORT does allow this.  (DFSORT's equivalent of
$ORTPARM is DFSPARM, but you can change to using
$ORTPARM instead of DFSPARM with the PARMDDN=$ORTPARM
installation parameter.)

You can specify something like:

//DFSPARM DD *
   MSGDDN=MYOUT
/*
//MYOUT DD SYSOUT=*

In this case, your DFSORT messages will go to MYOUT.

With the code you posted,it is probable that the output procedure has been
executing twice for each sort output record since the begining.

You posted the comment 
<<...because the s1-exit statement in some
senses is a NOP. It doesn't terminate the section.
>>

More particularly the EXIT verb in that paragraph is a NOP, always has been,
always will.  A thru clause as in a PERFORM THRU is an ALTER GO TO device,
sufficiently transparent to make EXIT look to many programmers like they do
something; never have, never do, never will.

In your new environment, your old duplicious program was caught .... it did not
start misbehaving recently, it has been strange all along.  

A talented (intentionally or accidentally) programmer can actually turn a tail
recursion into a loop, the sort RETURN function would not know the better of
it. The OUTPUT PROCEDURE can be a perform loop, a go to magnificence, or a
two-up staccato bumbling fallthru extravaganza. Whatever... you get one back at
a time from each  RETURN verb executed.

Unless, ...

Unless you munge up the registers that point to something important (like
service module vectors!). Then whatever your loop-dee-loop, it will stumble.

When doing two at a time via the time proven fallthru to optimistic re-RETURN,
there is a chance that code generated by new compilers did not leave registers
in the same condition as the old compiler generated code.

The instruction
    l 15,496(0,2)     v(igzesmg)
works on the normal iteration, but the second half fallthru probably has
strange value (by the time of the fallthru) in register 2. If you look real
hard and can not find a visible stomping of R2, then it is safe to assume that
igzesmg does it for you. And although igzesmg may be successfully instantiated
at run environment intialization, it is 496 off of the correct base value,
which is nolonger in register 2 upon fallthru.

Concerning your many important comments and questions about methods for
debugging this.  The answer is, and I mean this respectfully, all of the above.
By definition in a Charlie-Four situation, you do not even know where you are. 
All of the things that point to where things ought to be are suspect of having
been stepped on.

In an 0C4 situation, question everything, especially the source code. 

Many people think EXIT does something, never has, does not, never will.  A THRU
clasue alters a goto which invisibly resides after a paragraph (be it an 'EXIT'
paragraph or not.  The THRU clause is an ALTER GO TO. Dr. Grace Hopper
perfected these kind of techniques long ago because they did not have linkers
back then, and they did not work with a stack model of the machine. The
executable source code of COBOL is a trafic cop.

When you get an 0C4 in a new environment with an old program ... look at the
subscripts and look at,,,, you asked for it ... the THRU clauses or the absence
there of.  Old strangeness may make a difference in a new environment.

In the revealed code, register 2 is being used to manage the vector of  service
module addresses. It is sensitive, and it got clobbered. So don't clobber it.

As has been commented occasionally in this newsgroup, it can be believed that
mixing sections with paragraphs is inherently dangerous.  A section is kind of
like a pervasive THRU clause.  A way out is to make everything a section or to
make everything a paragraph. The implicit THRU-ness (pardon my koineage) of a
section is foreshortened by abutting it most agencently with another section.

So in da charly four things, check subscripts, check THRU clauses or lack there
of, and check for those fun mixtures of sections and paragraphs.

An 0C4 will teach you to question everything, most especially the integrity of
the register content.
