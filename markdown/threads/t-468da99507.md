Hi Oliver,
Dynamic (late) bound method calls across process boundaries are always 
going to incur some overhead no matter what the language. There's always 
going to be a balance between the number of calls you have to make verses 
the number of parameters used on each call, too.

However, NetExpress 3.0 supports dual interfaces for COBOL OLE objects. 
This means that your VB program can early-bind to your COBOL object, and 
this will improve the call speed.

To do this, you need to create a type library which includes your 
COBOL object (marked as dual), register it with your object and import it 
into VB. There are various tools to help you to create this library (which 
are too technical to go into here, but are fully documented in the 
product).
