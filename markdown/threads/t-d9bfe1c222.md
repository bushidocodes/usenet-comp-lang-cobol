DllHandle := LoadLibrary('mydll.dll');

If you are able to call the DLL from another Borland C program, then it is
reasonably safe to figure that this is in LIBRARY.DLL with an entry-point
of _insert_string.

I have some concerns for how you might be able to compile a COBOL program
with the '_' as part of the DLL call. Skipping last this.

If you call the DLL like 
       CALL "_insert_string" USING ......
This is NOT a DYNAMIC call and the LIBRARY.LIB file must be included
in the LINK process.   I am not sure about how to get a C .LIB file into the
LINK unless it is entered on the C-Library line.   When working with COBOL
.DLL files, I include the .LIB of  the DLLs with the OBJ of the Calling
program.

If you call the DLL like 
      CALL MYDLLNAME USING .....
where MYDLLNAME is in Working-Storage and contains a value of '_insert_string',
This would be DYNAMIC and requires the DLOAD compile option to be set to YES.
In this case the .LIB file from the .DLL is not required during the LINK
process.

In either case , since you are using multiple entry-points, You must setup an
[ENTRY] cross reference area to identify the 'entry point'='dllname'.

There may be additional syntax to tell the COBOL compiler use C-calling syntax
versus pascal or system.
   CALL..... WITH [C|PASCAL|STDCALL] LINKAGE

I particularly think of the dll call convention and the zero termination. I
do not know if this applies to Fujitsu, but I successfully called your dll
from Acucobol.

       IDENTIFICATION DIVISION.
       PROGRAM-ID. CALLDLL.

       ENVIRONMENT DIVISION.
       CONFIGURATION SECTION.
       SPECIAL-NAMES.
           DECIMAL-POINT   IS          COMMA.

       DATA DIVISION.
       WORKING-STORAGE SECTION.

       77 WS77-WORK           PIC X(070) VALUE SPACES.
       77 WS77-WORK2          PIC X(070) VALUE SPACES.
       77 WS77-POSITION       USAGE SIGNED-INT.
       77 WSERROR             USAGE SIGNED-INT.

       PROCEDURE DIVISION.
       P1.

           MOVE "Nilo" TO WS77-WORK.
           STRING  "My name is " LOW-VALUES DELIMITED BY SIZE INTO
            WS77-WORK2.
           MOVE 0 TO WS77-POSITION.
      *Acucobol does not auto NULL terminate strings, I have to do that
      * manually.
    MOVE X"00" TO WS77-WORK(70:01).
           INITIALIZE WSERROR.
      *Specify pascal (aka standard) call convention, this may be your
problem
           SET ENVIRONMENT "DLL-CONVENTION" TO 1.

           CALL    "LIBRARY.DLL"
                    ON EXCEPTION GO TO LOAD-ERR.

           CALL    "_insert_string"
                   USING   BY REFERENCE WS77-WORK
                           BY REFERENCE WS77-WORK2
      *The host function is expecting a value not a pointer
                           BY VALUE WS77-POSITION
                           GIVING WSERROR
                           ON EXCEPTION GO TO EXEC-ERR.

           DISPLAY WS77-WORK AT 0201.
    ACCEPT OMITTED.
           STOP    RUN.

       EXEC-ERR.
           DISPLAY "Error during call of _insert_string" AT 0101.
           STOP    RUN.
       LOAD-ERR.

           DISPLAY "Error during call of library.dll" AT 0101.
           STOP    RUN.

Hope this helps, at least there is nothing wrong with the DLL.

Dear Cheesle,

A lot of things you told me was part of the solution.

1. You said that Acucobol does not auto NULL terminate strings. Well,
neither Fujitsu does.

2. Pascal call convention is valid only in Fujitsu 16 bits compiler, not in
32 bits, that is the situation. When used CALL xxxx WITH STDCALL LINKAGE
(tip from Sff5ky) it works.

3. The command SET ENVIRONMENT ...  is not valid under Fujitsu Cobol V3.
Item 2 replaces it (so and so).

4. You are completely right about the use of BY VALUE and BY REFERENCE. If
the C function expects an *int* and not a pointer, i must pass it BY VALUE.

5. USAGE SIGNED-INT does not work in Fujitsu. The correct in this case is
PIC S9(4) COMP-5.

6. There's no need of CALL "LIBRARY.DLL" before calling functions inside th
dll. The trick in Fujitsu is made in a file called COBOL85.CBR, that makes
the association from entry points to the file that contains the dll. Here's
the content of the file:

[CALLDLL.ENTRY]
_insert_string=library.dll
_edit_line=library.dll
_str_index=library.dll
_str_len=library.dll

....and so on....

work, including you, my friend.
