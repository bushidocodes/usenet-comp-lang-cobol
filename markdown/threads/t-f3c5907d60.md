As an exercise only, one could use Newton's method of successive
approximations.  I'm working off the top of my head, but let's say you
want to find the square root of 100.

1.  Guess 50  test (by squaring it), that is too large
2.  Guess 2   test, that is too small
3.  Guess 25 and 4.
4.  Guess 12 and 8
7.  Guess 10.

The difficulty in creating this program is in picking the correct
precision for the task and testing where your answer is not perfect.

This difficulty applies when using irrational numbers anyway.  People
expect that if we divide $10 by 3 and then multiply it by 3, we will
have $10.  Occasionally we will need to make sure that happens in a
COBOL program.

Except for cases important enough to use a polynomial approximation
with pre-computed coefficients, or a table lookup, Newton's Method is
still about the fastest way to calculate a square root.  The key to
making Newton's Method sing is in choosing a good first approximation.
There are a zillion ways to do it, but here is an interesting one of
my own devising.  Take a look at this chart I prepared years ago when
writing a high precision floating point math library.  'Exp' is the
exponent which would be used if the number was expressed as a
normalized decimal floating point number.

                                Square Roots

Note that the exponent of the square root will tend to be about half
the exponent of the number.  For floating point numbers, that means
you can have a reasonably good approximation using any mantissa with
an exponent of half the number.  Or expressed another way, a square
root will tend to have about half as many digits between the decimal
point and the high order digit as the number.  This last observation
provides a way to make a good first approximation in COBOL.  For real
numbers, you can only obtain the square root of a non-negative number,
so create a number field which is DISPLAY and has no sign.  Example:

  01  ROOT-WORK.
      03  RW-DIGIT      PIC S9(02)       COMP     VALUE 0.
      03  RW-NUMBER     PIC  9(09)V9(09) DISPLAY.
      03  RW-SQRT       PIC  9(09)V9(09) DISPLAY  VALUE 0.
      03  RW-APPROX     REDEFINES SQRT
                           PIC  X(18).

Now calculate how many digits from the decimal point is the high order
digit of the number (assuming a non-zero RW-NUMBER.  (Note: there are
significant efficiency issues here which I will address below.)

  INSPECT RW-NUMBER TALLYING RW-DIGIT FOR LEADING ZEROS.
  COMPUTE RW-DIGIT = 9 - RW-DIGIT.

To calculate the approximate number of digits from the decimal point
which the square root will have:

  DIVIDE 2 INTO RW-DIGIT.

To calculate the high order digit position for a first approximation
in RW-SQRT above:

  COMPUTE RW-DIGIT = 10 - RW-DIGIT.

In summary, to create a 'good' first approximation using the above:

  INSPECT RW-NUMBER TALLYING RW-DIGIT FOR LEADING ZEROS.
  COMPUTE RW-DIGIT = 10 - ((9 - RW-DIGIT) / 2).
  MOVE "5" TO RW-APPROX(RW-DIGIT:1).

Then loop through Newton's Approximation a half dozen times or so.

  PERFORM 6 TIMES
      COMPUTE RW-SQRT = (RW-NUMBER / RW-SQRT + RW-SQRT) * .5
  END-PERFORM.

For a process like this, it is easy to eat up more machine time in
calculating the approximation than you save by having a better
approximation.  Some machines have efficient scanning instructions
which the COBOL compiler uses to implement constructs like the
INSPECT above.  The Unisys Medium Systems, for example, would take
only two inline instructions to do that INSPECT.

I'm not recommending this method as a general approach.  Just more
food for thought. :-)  The following program will run using MF COBOL
3.2, but you may have to change the ACCEPT for other compilers which
do not automatically align the decimal point.

       IDENTIFICATION DIVISION.
       PROGRAM-ID.    SQRT.
       ENVIRONMENT DIVISION.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
