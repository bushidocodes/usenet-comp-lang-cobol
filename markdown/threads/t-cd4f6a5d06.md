```
I had recently posted that fujitsu made a visual cobol which was free.  What
I meant by this is that the product itself is 'visual' as in 'visual' c++ or
'visual' basic.
  Fujitsu actually called their version power cobol but it is a 'visual'
product.  This caused some of the older programmers to become upset with me,
because they used a product years ago called 'visual cobol' which is no
longer sold on the market.
  Power Cobol is graphical, just like visual c++, or visual basic. You
simply click and drag buttons, edit boxes, etc, and then write the code
(procedure) underneath.
  Since the product does develop applications for the pc, hpux, solaris, (
and the copy i had also said AIX ), and the product claimed you could write
a visual product on the pc and transfer it to a unix variant as a gui app.
  I have not tried that, I have only made executable programs on the pc, and
since it is a free compiler with a lot of functionality and ease of use, I
thought a lot of other people might enjoy using it.
  PowerCobol was actually given away on a cd in the back of the  book 'cobol
for dummies', and several other good books on cobol.
  I know that there are some gui enabled forms of unix, by which i mean,
rather than the command line type prompt, you can actually run a 'windows
like' environment on your unix mainframe and run gui apps.
  At any rate I apologize for the confusion to all the old-timers who were
rattled at my inappropriate misnaming of an old no longer marketed product.
```

```
John Hastings wrote:

> yes Richard, I use 'visual' interchangeably with gui like development

To you it may be, but then there is so much that you seem to get wrong, 
probably from a lack of experience and understanding. 

There are is fact _actual_ 'visual languages' where the 'programming' is 
done using graphical representations of building blocks and the structure 
and logic of the program is created by drawing links between them.  That 
is, the 'source code' is only seen visually and not texturally.

With 'Visual C++' and 'PowerCobol' there is still textuaral source code to 
be written and it is only the user interface that is designed 'visually' 
(actually graphically).  These may be 'Visual development environments', 
but they are far from being 'visual languages'.  

In fact the 'visual' part has little to do with the language itself.  With 
Windows 2.x and 3.x the primary way of developing graphical user interfaces 
was 'visually' using the Dialog Editor. This is completely independent of 
the lanuage used.  Later some code generators would take the resource file 
and would generate various source code for different languages to interact 
with that dialog box.  The Dialog editors were enhanced and could accept 
more details about each element which was then used in the code generation.

'Visual C++' is just a visual deveopment environment (ie Dialog Editor) 
that has a code generator, code management and compiler for C++.  In fact 
the code tools could equally be for other languages (and are also) without 
changing the 'visual' parts.  It doesn't make C++ into a 'visual language'.

At least IBM got the naming of their products correct with 'Visual Age 
_FOR_ Cobol' and 'Visual Age _FOR_ Smalltalk' (though they are often 
written wrongly - without the for).

Now 'Visual C++' and 'Visual Cobol' are trade names referring to specific 
products.  A generic term would be 'visual development for <language>'. 
PowerCobol is not 'visual Cobol' but is 'Visual development for Cobol'.

Of course, clueless newbies wouldn't know the difference.

> Not the old way of designing console applications.  

I have done 'console applications' 'visually' for more than two decades. 
MF's Forms2 was out in the late 70s, as was SourceWriter and many others. 
Draw on the screen what you want the user interface to look like and add in 
the code to do the processing.  Currently I do my Cobol console apps with 
SP2 designer console version.  Of course this will also run as graphical in 
suitable environments.

Even 'Visual BASIC' (trade name) was a 'console application' originally.  
VB for DOS was used to draw the user interface visually and the user added 
source code.

So, contrary to your assertion, it _is_ the old way of designing console 
applications, just enhanced and extended.

I noticed that you didn't address any of the other issues.
```

```
Robert: I think what you are referring to is modelling language-based
software tools which, from model diagrams, can generate code. For
example UML (Universal Modelling Language) tools particularly (but not
solely) suited for OO-based design - e.g. the toolset from Rational
(now being taken over by IBM I think), Popkin. Most of these tools can
generate C++ and/or Java. The only one I know of which could generate
COBOL is a tool from MicroGold software (can't remember the name of
the tool.)

There is a free UML-tool at http://argouml.tigris.org written in Java
although I don't think it can yet generate code of any kind.
```
