I've been attempting to post this to the NG but without much luck. I'm
sending it to you privately. I remember the original post indicated it was
for ANSI 74 so I don't have anything like Hex Values and/or Reference

++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++

03 WS-HALFWORD PIC S9(04) COMP.
03 WS-HALFWORD-X REDEFINES WS-HALFWORD.
05 WS-HALFWORD-HIGH-ORDER
PIC X(01).
05 WS-HALFWORD-LOW-ORDER
PIC X(01).
03 WS-DISPLAY-BITS.
05 WS-DISPLAY-BITS-TBL
OCCURS 8 TIMES
INDEXED BY X-WS-DBT
PIC X(01).
03 WS-BIT-VALUE-TBL OCCURS 8 TIMES
INDEXED BY X-WS-BVT
PIC S9(04) COMP.

MOVE ZERO TO WS-DISPLAY-BITS.
MOVE LOW-VALUE TO WS-HALFWORD-HIGH-ORDER.
MOVE 'A' TO WS-HALFWORD-LOW-ORDER.

PERFORM EXTRACT-BITS THRU EXTRACT-BITS-EXIT
VARYING X-WS-DBT FROM 1 BY 1
UNTIL X-WS-DBT IS GREATER THAN 8.

EXTRACT-BITS.
*
IF WS-HALFWORD IS GREATER THAN +127
MOVE '1' TO WS-DISPLAY-BITS-TBL (X-WS-DBT).
*
COMPUTE WS-HALFWORD = (WS-HALFWORD * 2).
MOVE LOW-VALUE TO WS-HALFWORD-HIGH-ORDER.
*
EXTRACT-BITS-EXIT.
EXIT.

After completion of EXTRACT-BITS, the field WS-DISPLAY-BITS will equal
'11000001', because a value of X'C1' (B'11000001') was loaded into the
low-order of the halfword. By multiplying the halfword by 2, I'm simulating
an ASSEMBLER 'SLL' (Shift Left Logical) Instruction but of course without a
register. If the test of the halfword is true for a value greater than +127
(X'7F', B'01111111') then the bit that I am testing is 'ON' (not less than
X'80', B'10000000'). However, every time I Shift-Left 1-Bit (Multiply by
2), I must ensure that the high-order byte of the halfword is X'00'.
Otherwise, my testing of the halfword for greater than +127 would be bogus.

Converting WS-DISPLAY-BITS back to a single 8-Bit (Single Byte) value is
just the opposite:

MOVE +1 TO WS-BIT-VALUE-TBL (8).
MOVE +2 TO WS-BIT-VALUE-TBL (7).
MOVE +4 TO WS-BIT-VALUE-TBL (6).
MOVE +8 TO WS-BIT-VALUE-TBL (5).
MOVE +16 TO WS-BIT-VALUE-TBL (4).
MOVE +32 TO WS-BIT-VALUE-TBL (3).
MOVE +64 TO WS-BIT-VALUE-TBL (2).
MOVE +128 TO WS-BIT-VALUE-TBL (1).
MOVE ZERO TO WS-HALFWORD.
SET X-WS-BVT TO 1.

PERFORM EXTRACT-BYTES THRU EXTRACT-BYTES-EXIT
VARYING X-WS-DBT FROM 1 BY 1
UNTIL X-WS-DBT IS GREATER THAN 8

EXTRACT-BYTES.
*
IF WS-DISPLAY-BIT-TBL (X-WS-DBT) IS EQUAL TO '1'
ADD WS-BIT-VALUE-TBL (X-WS-BVT)
TO WS-HALFWORD.
*
SET X-WS-BVT UP BY 1.
*
EXTRACT-BYTES-EXIT.
EXIT.

After completion of EXTRACT-BYTES, the field WS-HALFWORD-LOW-ORDER will
equal X'C1' or WS-HALFWORD will equal +193.

The remarks about bit fiddling here reminded me of what a pain I was to
vendor members of the COBOL committee in my time. MOSTLY, I felt each
time one of them would use the remark that such and such did not belong
in COBOL, I thought "who appointed them GOD?" I usually said out loud,
"Quit helping me." Many things done under the umbrella of business data
processing are not in everyone's definition of bdp.

The main consideration might be expressed in my version of the computer
design commandments as:

Keep it simple.
Keep it self documenting.
Keep it as universal as possible. Portability is a goal, but not a
destination.
Keep it able to use all of the resources, AND if need be,
create the need to engineer new abilities that must be built as
subroutines, or hardware. The computer is a problem solving tool, and
makers of a general purpose tool should realize that in it's design the
ability to solve real problems is first and foremost.

It's very bad to have bit ability on the system, but no ability to apply
to the problem solution. The same thing seems to be true today for
things like networks, HTML, and advanced methods of data storage and
retrieval.

Warren Simmons

When Dave expanded aByte-Binary to 2 bytes he changed the algorithm. The
results he shows are correct as far as they go. However, the original
bit shifting would have worked as follows:

64 = 0100 0000

The algorithm multiplies by 2 and compares the result to 127 or 0111
1111 so:

1st pass: 0100 0000 < 0111 1111 ... "0"
2nd pass: 1000 0000 > 0111 1111 ... "1"
3rd pass: 0000 0000 < 0111 1111 ... "0"
4th pass: 0000 0000 < 0111 1111 ... "0"
5th pass: 0000 0000 < 0111 1111 ... "0"
6th pass: 0000 0000 < 0111 1111 ... "0"
7th pass: 0000 0000 < 0111 1111 ... "0"
8th pass: 0000 0000 < 0111 1111 ... "0"

Output is "01000000" as expected.

The change to two byte integer breaks the algorithm because the bit that
would have been discarded isn't, leading to the results Dave shows. The
algorithm needs to be changed to zero the high-order byte so you get:

01 aByte-Grp.
05 aByte-Hi pic x.
05 aByte pic x.
01 aByte-Binary redefines aByte-Grp pic 9(4) COMP.

Perform varying I from 1 by 1 until I > 8
If aByte-Binary > 127
Move "1" to aByte-Unpacked (I)
Else
Move "0" to aByte-Unpacked (I)
End-If
Multiply 2 by aByte-Binary
Move low-value to aByte-Hi
End-Perform
