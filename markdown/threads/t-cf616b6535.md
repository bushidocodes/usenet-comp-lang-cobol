containing an object, written in any object oriented language including
Cobol, how can I invoke it from a standard Cobol program? How do I set up
the linkage, etc., so that it finds the appropriate code?

Well, yes and no.

There was an interpretation request against the '85 Standard - concerning
what happens in "'85 Standard COBOL if a COBOL program compiled with
compiler-xyz calls a COBOL program compiled with compiler-abc - do the rules
of Standard COBOL apply - or do the rules of 'inter-language' CALLs apply".
The bottom-line (according to J4) was that the rules of inter-language CALLs
apply.  For example, EXTERNAL data defined in one program need NOT be
"recognized" by the other COBOL program.

One of the goals of ALL "OO" theory (as I understand it) is to "hide" the
internals of an object - but to use only "externalized interfaces" to invoke
methods on such objects.  Therefore, such invocations SHOULD work from COBOL
program to COBOL program to COBOL object to COBOL object - regardless of
which compiler is used (or even if it is a COBOL compiler) as long as the
"appropriate" external interface is used - and you don't rely on any
"internals" being the same in the multiple objects (such as sharing EXTERNAL
data, files with the same formats, DISPLAY statements going to the same
devices, etc, etc).

Does this help explain what I think you will and will NOT be able to count
on - in such cases (assuming the draft ever becomes approved)?

So far, I haven't heard a summary much less seen the minutes of the J4
meeting that just took place.  To be fair,  it is my impression that the
current voting members of J4 really do believe that they will be delivering
an "acceptable" draft from J4 to WG4 in January 2000.  Then assuming all
goes well, there really, REALLY may be an "approved" Standard at the end of
2000 or beginning of 2001.

I have personal doubts about each and every step of this schedule - but
those who believe it SHOULD know better than I do.

As far as implementations go, you might want to look at the extensions (OO
and non-OO) in the latest release of Micro Focus (MERANT) NetExpress.  It is
my understanding that they have already implemented a "good-sized chunk" of
the draft Standard. Similarly, both IBM and Fujitsu most definitely have
produced OO products that match at least some variation of the draft
Standard.  In theory, many of the COBOL vendors have said to J4 that they
have been looking at the spec and "estimating" the work involved in
implementing it.  Again, I have personal doubts - but implementations may
well be sooner than some might think/fear.

ON THE OTHER HAND, my biggest worry is that NO vendor will EVER implement
this next Standard - even after it is "fully" approved.  In the US, the fact
that there are no longer people maintaining (much less enhancing) the NIST
certification test; the fact that there is significant evidence that there
will never be another FIPS Standard for COBOL (historically required when
selling compilers to the government); and the fact that there seems to be a
diminishing interest in COBOL in general among the major software/hardware
vendors - all make me question whether there ever will be a "conforming"
implementation of this next Standard - again assuming there is a next
Standard.

I do need to re-iterate that I am well known as the "extreme pessimist"
among the COBOL Standards committees, so anything that I say needs to be
taken with a LARGE grain of salt. But on the other hand, so far I really
think that my Cassandra-like pessimism has proven to be depressingly
accurate.

Well, if what you mean is "is there no single standard for OO languages
to call each others objects?" then I don't believe that there is. Each
language has their own default way of exporting objects and methods
"natively" - for Java it is JNI, for C++, mangled signatures etc.

The solution, as Ken suggested, is Microsoft COM or CORBA ORBs, which
were designed to be suitable for this job (and include remote object
connections as a bonus). These match up quite well with the OO
definition for the new COBOL standard (which, as Willian Klein said,
was designed towards this), and I'm sure that several companies have
products that already exploit this (including my own, MERANT, which has
COM/OLE/ActiveX and CORBA support) or will do so.
