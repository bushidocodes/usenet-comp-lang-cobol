```
We seem to be unable to contact Fujitsu via their e-mail address for the
last two days ("recipient name not recognized"). Can anyone help with the
following please?

1. When data is entered into a numeric field in the Screen Section the keyed
numbers are accepted into the LEFT end of the field instead of the right.
eg. if the value 23 is entered into a 9(4) field then the result is 2300
instead of 23 which is what all our users are accustomed to. It seems the
only way to force the value to be accepted as 23 is to enter 23. (ie 23
followed by a full stop).

2. How do we stop the ENTER key from acting as a Terminator key when
accepting data into fields in a Screen Section? ie. our users expect the
Enter to move to the next field (like Tab) and NOT to terminate the ACCEPT.
We have already used a keyboard definition file to remap the Function Keys
to return values which suit our code, but there does not seem to be any way
to remap the ENTER key in this manner.

Regards, Robert.
```

```
You can't.  In MicroFocus you could REDEFINE Enter to be Tab, but I 
never found a way to do that with Fujitsu.  You can't do it on the DEC
either - or any of the compilers that comply with the X-Open screen 
section.  But there IS a solution.

You can DETECT the enter key after the accept, using the CURSOR 
special name, find the cursor position, then set it at the position of
the next field and reissue the accept.
```

```
I have generally used accepts of individual fields inside
a performed paragraph so that I can process each field
in turn rather than leave validation to the end of screen.

          MOVE 1      TO Field-Number
          PERFORM Get-Fields
              UNTIL Field-Number > Field-Count
          ....

       Get-Fields.

          IF Field-Number < 1
               MOVE 1    TO Field-Number
          END-IF

           IF Field-Number = 1
               ACCEPT First-Field   AT Field-Pos(1)
                PERFORM Check-Function-Key
               IF Key-Special NOT = "Y"
                   validate field
               .


Withing Check-Function-Key the 'Exit' key will set Field-Number
to, say, 99, the 'Back' key will subtract 1, a valid field
will add 1.  If 'Help' or 'List values' key is pressed this
is processed and Field-Number is left at the same value.

I have done forms processing systems in which there is a 
full screen accept but specific trigger points where
fields may be entered and validated.  For example a 
name and address may just be entered on the form or, at
the name field start a customer number may be keyed and
a function will then convert this into a name and
address.  This allows for forms for customers or non-customers.

In this case the CUSOR IS Cursor-Position is used to find
where the function key was used and to set the next position.

In the case of just a screen accept a simple table of fields
(built by processing the SCREEN SECTION entries to generate
a copy book perhaps) could be used to set next-previous
position.

As a sophistication for, say, file mantenance, it can be
useful to allow the user to set a 'start position'.  Often
a lot of records need to be changed in one simple way, with
just one field actually changed.  When the first is done
the user presses a function key to mark that field, then
on subsequent records the program puts the cursor there
for the user.

Easy to do using Field-Number technique above or using
Cursor-Position.

Also helped if there is an 'end' key without having to
tab to the end of all fields.
```
