As a follow-up on another thread, I thought that I would point out what may
turn out to be a MAJOR inhibitor for some shops (with some compilers) in
upgrading from an '85 Standard compiler (with an extension) to a requirement
of the new 2002 Standard.

The issue arises for 2 or more dimensional tables.

With all the compilers that I know about (IBM, Micro Focus, Fujitsu, Realia)
when their "extension" (or more accurately "debugging tool") boundary
checking is turned on, then one ONLY gets a "boundary violation"
notification for a 2-thru-N dimensional table if the COMBINATION of
subscripts (or indexes) references storage totally OUTSIDE the storage of
the "full table".

For example,

  01  Full-Table
        05  Level-1 occurs 10 times indexed by Ind1.
              10  Level-2 occurs 100 times indexed by Ind2.
                    15  AnElem   Pic X.
  ....
   Move AnElem (1 101) to Other-Field.

will *not* receive any "boundary violation" message (with the products that
I know of).

ON THE OTHER HAND,  the 2002 Standard "requires" that an

    EC-BOUND-SUBSCRIPT

exception condition be set.  See page 87, rule 2 that says (in part),

"The highest permissible occurrence number for any given dimension of the
table is the maximum number of occurrences of the item as specified in the
associated OCCURS clause. If the value of the subscript is not a positive
integer or is less than one or greater than the highest permissible
occurrence number, the EC-BOUND-SUBSCRIPT exception condition is set to
exist."

I can EASILY imagine that this difference - when actually introduced and
implemented by some vendors - may come as a fairly UNWANTED surprise to many
programmers and shops.  Obviously, vendors will be able to continue to
support the "older" behavior as a NON-conforming option - but when run in
"strictly conforming" mode, the new behavior will be required.

Not true for Unisys MCP/AS COBOL85.  If bounds checking is not explicitly
disabled, on access to the table element each subscript or index is checked,
and a fault ("assertion failure on range test") occurs if any subscript is
out of range.

For MCP/AS COBOL74, you don't (and can't) get bounds checking for indices
anyway; the hardware prevents access outside the 01-level record containing
the table, but otherwise that record's fair game.   Subscripted access is as
for COBOL85 unless range checking is explicitly disabled (fault text is
"invalid index").

I used a four-dimensional table for my minimal test.

I am pretty certain that Docdwarf was "joking" in his original question -
and that Chuck responded courteously (and knew it was a joke) and that DD
continued "politely".

Having said that, I did want to revisit "ancient" history a bit.  I suspect
many of those reading this newsgroup remember the days when the "data
processing" world was divided into IBM and everyone else (aka "BUNCH" - an
acronym of the various "other" vendors' names - including both of the
companies that EVENTUALLY became "Unisys").

In "ye goode olde dayse", it was medium common for IBM to do what they
wanted (conforming or not conforming to "the Standard" - COBOL or otherwise)
and have everyone copy them (because that is what SOME - not all - customers
thought "must" be done.)

Furthermore, when the "PC" world became more popular, MOST (again not all)
of the COBOL implementations had as a partial (often MAJOR) customer base
those who were doing IBM mainframe COBOL development.

Given these two facts ("follow-the-leader" for SOME features by "BUNCH"
vendors *and* PC "IBM mainframe emulation")  it is not too surprising that
some IBM mainframe COBOL compiler extensions (or even violations of the
Standard) became "de facto" Standards for some (still not all) COBOL
development.  The following are just a few "things" that many programmers
think are almost universal - but which are definitely NOT "Standard" (at
least thru the '85 Standard).

 - "single quote" (aka APOST) for alphanumeric literal delimitation (and/or
figurative constant QUOTE)
 - Comp-3 meaning Packed-Decimal
 - GOBACK being equivalent to EXIT PROGRAM if appropriate, STOP RUN
otherwise
 - REDEFINES of other than the "1st" entry in a series of REDEFINES
 - ENTRY statement (not to be confused with the '85 Standard OBSOLETE
"ENTER" statement)
 - Between column 72 and 73 being the R-margin (but allowing comments in
columns 73-80)

Chuck and others from non-IBM and non-IBM "emulation" environments, please
excuse me if some of these really DID get introduced by some other vendor
than IBM before IBM did them.

But all of this "boils down" to a semi-serious answer to DD's originally
(humorous) question

  "Mr Stevens, so much seems to be not true for/inapplicable to/shown to be
an exception by this Unisys MPC/AS  compiler that I feel it might be wise to
ask...

    ... are you sure it's COBOL?"

(Obviously, this also applies to Karl and Rosa's comments on the Siemens
implementation.)

The answer is that many (originally IBM) extensions "seem" to be a universal
part of COBOL - but only to those using features from "early" days of IBM
domination or from compilers explicitly targeting IBM COBOL development.

Yes, I definitely remember BUNCH.  I joined the Burroughs field support
organization while all five companies were still active in the marketplace.

Bill wrote:

> The following are just a few "things" that many programmers think are
almost universal -
> but which are definitely NOT "Standard" (at least thru the '85 Standard).
>  ...
> - Between column 72 and 73 being the R-margin (but allowing comments in

This is the only one of your list that I see our implementations allowing.
I see *some* indications that this might have been a feature of First
Standard COBOL, based on the fact that I have usually found the IBM DOS and
Burroughs Large Systems COBOL (68) reference manuals taken in combination to
be pretty reliable indicators of what's standard and what's not.  Both show
the location of Margin R as fixed at Column 72, don't indicate that it's an
extension, and indicate either that the area following is used for program
identification (IBM) or that it may be used for anything the user desires
(Burroughs).

This leads me to believe that allowing the source code to "leak over" beyond
Column 72 was something people didn't expect to do, and that in earlier
COBOL (either by convention or by fiat) compilable code stopped there.

The last printed version of what started life as the Burroughs Large System
COBOL manual states, under the general introduction to the discussion of the
coding form "The format of the COBOL coding form has been defined by
CODASYL, ANSI, and common usage.  The A Series COBOL compiler accepts this
standard format.", which does indicate some possibility that this particular
margin, and the conventional use of what followed it, were "common practice"
rather than mandated by the standard.

I suspect these columns have been used for "patch marks" indicating program
change history since the very earliest days of external COBOL source file
management (merging of source changes into disk or tape master files by
sequence number) on Burroughs systems (Medium Systems "SOLT" and "SOLD"
files, and probably later, Large System SYSTEM/PATCH), but I can't say that
IBM's expressed use of these fields for program identification predates or
postdates that use.

The other features Bill mentions are not things our users have requested,
for IBM compatibility or otherwise, of Unisys, or of Burroughs before the
merger.

    -Chuck Stevens

While you folks are wandering down Memory Lane - which is absolutely
fascinating, care to comment on what we now call ISAM.

Purely from memory, so my figures and descriptions might be inaccurate. RCA (and
I note they don't appear in BUNCH above - or do they ?), produced their MCF
(Magnetic Card File) around the same time as NCR's CRAM - this just before the
introduction of the first IBM hard disk drives.

MCF was a big beastie say about 3 feet x 10 feet, standing at table level. One
unit had 8 magazines each containing 256 magnetic cards. Each card held 166,400
characters (I hope that's correct). So 166,400 x 256 x 8 - not bad storage for
mid sixties.

Programmatically, (and originally using either Assembler or perhaps some
proprietary RCA tool), cards were allocated for a particular indexed file. Each
card could be designated in 'buckets' (what I suppose we now call Blocks) in
sizes of 128, 256, 512 or 1024 per file. So given that you knew the size of the
records you were going to store, you picked the 'best' bucket size, and
allocated a set of cards to hold your file and anticipate expansion.. (Note -
records could not spill over into the next bucket).

In addition a separate area on the cards was allocated for the index. Depending
upon on your PrimeKey size, again you picked the appropriate bucket size. (Can't
be sure, but doubt there were Alternate Keys - we probably did it with Sorts).

Within the Index you had a bucket containing a group of PrimeKeys. I think it
functioned as follows. If there was space in the bucket then the new PrimeKey
was positioned in ascending sequence, shifting higher keys to the right.
(Obviously the key entry contained a reference to where the data record was on
another card). If there was insufficient room in the PrimeKey bucket then a
'Tag" record was allocated referencing the key, perhaps on an entirely new card.

So far as I can remember, actual records were added 'sequentially' in the data
area buckets.

There was a considerable amount of housekeeping done to get the indices and
records back into shape - plus the nightmare of failure - which made backups
ever present.

Now I know you can get books, and books, and..... on Indexing, Merging and
Sorting techniques - which is a very specialized area to the 'devoted'.

Question - just curious whether Indexing has moved dramatically from the above
model  ?

Jimmy, Calgary AB

Ummm... I think that's a bit of an oversimplification.

There's no identifier varied in SEARCH ALL because there's no VARYING clause
in that format; the varying is done "behind the user's back" in the INDEXED
BY item.

For SEARCH, if you have a VARYING clause, it can reference an index data
item or a numeric integer (but not an index-name).  Use of the index-name
associated with the INDEXED BY clause in the table being SEARCHed is
accomplished (arguably most efficiently) by simply dropping the VARYING
clause.

INDEXED BY clause at all, and that most definitely would be an extension to
standard COBOL.

For SEARCH ALL, though, I don't see offhand where the compiler would be
expected to indicate where the "match" occurred because there's no VARYING
clause to begin with.  There'd have to be some "extra syntax" over what the
standards specify in the Format 2 SEARCH to indicate where to put that
location to accomplish that.
