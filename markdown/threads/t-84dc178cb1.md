Yes.

First, you've paraphrased the announcement. Corrected, it reads:

"...Micro Focus International Ltd., which last week announced the first
product enabling Cobol code to be deployed directly as a Web service without
requiring third-party software."

The key part of this statement is "deployed directly", not "without
requiring third-party software". Most SOAP toolkits generate "wrapper" code
(typically in Java/C#) to handle the SOAP request and to make the call to
the "interfaced" code (i.e. your COBOL application). This approach does not
scale well.

With Net Express V4 (and Server Express later this year), that extra layer
is no longer required. Execution of the Web service is handled directly by
COBOL and its transactional (scaleable) execution environment.

Note: Since March 2002 Net Express V3.1 has also supported the "indirect"
approach; notably through a partnership with Cape Clear software.

Peter and all,
I rarely post to comp.lang.cobol and certainly don't want to get involved in
any of the Fujitsu/Micro Focus rivalry that is going on here - however I
feel the need to clarify some points here.

To state the obvious and set a background: when you talk about Web Services,
strictly speaking, you are talking about services made available over the
Web that are invoked by a sending a Soap encoded message either directly or
indirectly (through Soap intermediatories) to an ultimate Soap receiver.
The key things here are that the messages are encoded in Soap and, for 'Web
Services', the protocol and transport for transmission are http over
tcp/ip - of course the Soap encoding can and is used with lots of other
protocols and transports.

Of course, a Soap message is just a message, the actual invocation and
implementation of the Web Service must be written in some programming
language (whether it be C# or C++ or COBOL).   In order to realize a binding
to implementation, there is a technology/software stack on both sides of the
equation (ie. on the sending side and on the receiving side).  These
software stacks takes care of conversion of the data types and data
representations from/to the source(target) language (C#, C++ or COBOL)
to/from the data encoding used in the Soap message.

The point of the Micro Focus announcement is that on both the
sending/receiving side the software stack required to handle the creation of
a Soap message or the reception/execution of a Soap message is 1 level deep.
The Micro Focus Soap handler directly converts from Soap data types and
representations to/from  COBOL data types and representations and then
directly invokes the COBOL implemented Web Service.  There is no
intermediate language layer (such as C# or Java) and no third-party software
is required (ie. no further Soap toolkits, no further Soap servers and no
further infrastructure).

So you see that marketting statement was correct technically.

Barry Rosetti,
Sr. Director of Development, Product/Technology Architecutre
Micro Focus
