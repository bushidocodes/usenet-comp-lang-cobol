When I was working for Micro Focus, I created a "test" program to see exactly
what COBOL statements would cause S0C7 ABENDs - depending upon which types of
input values (i.e. spaces in numeric fields vs low-values vs "garbage") - AND
depending upon which specific settings of compiler options.  This ended up
with about 1000+ separate paragraphs - each with a single (possible S0C7
statement)  and with various "initialization" of the sending fields.

What I then did was at the beginning of each paragraph, check what the LAST
paragraph did and write it out to a file as "successful" (no S0C7) - if it
worked OK.  Then I would write to a file that I was starting this paragraph -
then I would try the COBOL statement.  If, it didn't cause a S0C7, it would
indicate that this statement worked OK and the program would go on to the
next paragraph.  If, on the other hand, the program ABENDED, I would then
restart the program and it would check (my temp file) to see which paragraph
had started OK, but not finished - and write this out as a combination that
caused a S0C7.

Talk about a "strange" application design <G> !!!  (and thank goodness, no
real application would ever have to be designed this way)
