More notes/feedback:

1: yr mod 4 was meant to be "y mod 4". Most figured this out.

2. If you have some static memory lying around, you can replace the
code of:

   INT((y - 1924) * 1461 \ 4)

with an array lookup (in FORTRAN or other HL languages) or a
based-index pointer or XLAT function in low-levels, thereby dropping
the MULT and the IDIV!

   such as ARRAY(Y) + ar(m) + d. You'd need to precalculate the array
values and store them as static vals as DATA:

DIM ARRAY(1274) AS INTEGER {1923*0, v1, v2...vn} STATIC
COMMON INTEGER*4 ARRAY(1274) /1923*0, v1, v2...vn/
  and so on depending on the language used.

3. Element Zero: Some compiler models choose array base "0" and "1"
depending upon configuration. Use DIMENSION (1 to nnnn) in Fortran or
(OPTION BASE 1) in basic-compatibles.

Or define the (ar) array subscript as:

0,33,61,92,122,153,183,214,245,275,306,336

with ar[0] being zero (in cases such as C++)

By direct test it can be unambiguously stated that it
*doesn't* handle these things correctly.

First: assume that \4 really means divide by 4 (I've
never seen \ used as divide before in any language)
and that yr was really supposed to be y.

If you give it today's date: Dec. 20, 2001 the program
gives 28480.  Now give it a date 4 centuries from now:
Dec. 20, 2401.  The program gives 174580.  The difference
between these two numbers should be the number of days
between the two dates.  The difference is 146100.  But
the number of days between those two dates is actually
146097.

This is because the program assumes that all years
divisible by 4 are leap years, but the years 2100,
2200, and 2300 are *not* leap years.

Further tests reveal additional problems.

Dec. 31, 2001 gives day number 28491.  Jan. 01, 2002
gives day number 28490.  They are consecutive days,
but your program claims they are in reverse order!

Jan. 01, 1924 gives day number 3.  Jan. 01, 1925 gives
367.  But, the two days are 366 days apart (1924 was a leap
year).  Your program claims they were 364 days apart.
Where are the two other days?

Take a specific, very simple, example to analyze more
carefully:  Feb. 29, 1924 and Mar. 01, 1924.  Since
the year is 1924, the first term in line 10 is zero, and
the year mod 4 is also zero which means that line 20
does nothing.  So, your day numbers are simply
ar[m] + d.  Well, for Feb. 29, ar(m) is 33 and d is 29,
so the day number is 62.  For Mar. 01, ar(m) is 61 and
d is 1, so the day number is 62.  So, your program claims
they are the same day.  But, 1924 was a leap year, so Feb. 29
was a legitimate date.

By direct observation of the answers your program
generates we can all see that it has serious problems.
Before cross-posting to 5 different newsgroups that
all the people that have responded to your program
are wrong, why not actually test it?

A Fortran version that's similar to yours in concept,
but should give correct answers is:

      integer function dateval(m, d, y)
      integer, intent(in):: m, d, y
      integer, parameter:: ar(12) = (/365, 396, 59, 90, 120, 151, 181, &
         212, 243, 273, 304, 334/)
      integer :: yr

      yr = y
      if (m < 3) yr = y-1
      dateval = yr*365 + yr/4 - yr/100 + yr/400 +ar(m) + d + 1721060

      return
      end

Remember that in Fortran, the quotient of two integers
is an integer, so truncation with an INT function is not
necessary here.  With so many language's newsgroups
cross-posted, it's best to point these things out.

This function obviously doesn't test for the validity
of the input, so you can feed it nonsense (like
15/45/-2002 or something).  A month that's out
of range though might trigger an error depending
on whether the program is compiled with array
bounds checking turned on.

The answer is correct for any date in the Gregorian calendar
(so, dates after Oct. 15, 1582 are the only valid ones).
Obviously this is only true if the calculation doesn't
overflow your integers.  The return value is the Julian
Date (which begins at noon), and is based on the
epoch of Jan. 01, 4713 BC as day 1.

You can count from any epoch you like by changing that
large constant.  For example, to use Jan. 01, 1924 as day
one, replace 1721060 with -702725.

There are several ways to do this same calculation.  Many
are very clever and dispense with the need for an auxilliary
array and/or use fewer operations.  But, with memory so cheap
these days (and CPU cycles cheap too), there seems to be
no reason to adopt anything other than a clear statement
of the actual rules of the calendar.  At least it's easy to
verify that it's right.
