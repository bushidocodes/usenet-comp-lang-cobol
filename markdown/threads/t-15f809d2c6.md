I missed the original post, so I'm piggybacking here. The following
summary is based on the IBM manual for VS COBOL II. Other dialects
may differ -- I don't know what the 85 standard says.

By default, a COBOL variable is accessible only within the program
in which it resides, and *not* within any other programs nested in
the same source file. Within that scope, any paragraph or section
in the PROCEDURE DIVISION can access it.

With the GLOBAL clause, a variable is accessible not only within its
own program, but also to any nested program contained therein (as
long as the nested program does not declare its own variable with
the same name).

With the EXTERNAL clause, the variable is accessible to every program
in the load module. Each program which needs it must declare it with
the same name, and must include the EXTERNAL clause. Such
declarations need not be identical, but they must allocate the same
number of bytes. You may not initialize the variable with a VALUE
clause.

What you cannot have is a variable local to a section or paragraph in
the PROCEDURE DIVISION. If you want that degree of locality, you
have three main choices (apart from using another language):

1. Be careful. Don't use that variable outside the right paragraph.
A suitable naming convention may be helpful.

2. Convert your section or paragraph into a nested program or a
separately compiled subprogram.

3. If you're really ambitious, invent a preprocessor which translates
a COBOL-like language (which allows local variables within sections or
paragraphs) into real COBOL. It would enforce locality of references
to the local variables, and then move the variables into
WORKING-STORAGE, with mangled names to keep them unique.

While you're at it, you could have the same preprocessor enforce
your preferences concerning the use of END-IF, PERFORM THRU, and
other constructs. (I'm not trying to touch off another war about
those topics.)

Thane Hubbell wrote:

[snip]

› That reminds me to ask. Does anyone actually use Nested programs?

I only used them once. I wanted to use a CALL instead of a PERFORM
so that I could pass different objects to the same code at different
times. In this case the usual trick of MOVE whatever TO PSEUDO-PARM
would have been ugly, awkward, and slow. On the other hand I didn't
want to compile the routines separately because they were so
specialized. There would have no conceivable use for them in any
other context. So I nested them.

While nested programs were an appropriate solution in that case,
I've never felt tempted to use them more widely:

1. The syntax is cumbersome and verbose because you have to include
all four DIVISIONs in each program. You also have to be careful
about who is nested in whom.

2. My usual habits of editing don't work as well. If I want to go
to the top of the PROCEDURE DIVISION, I can't just enter "f PROC 8
first" on the command line in the SPF editor, because there's more
than one PROCEDURE DIVISION. Likewise there's more than one
WORKING-STORAGE, and possibly duplicate paragraph names if I'm not
careful.

3. A nested program cannot have multiple entry points (at least not
in VS COBOL II).

4. In a shop like mine, where archaic practices are the norm, nested
programs are likely to leave a maintenance programmer completely
baffled.

In fairness, nested programs do offer some significant advantages
over sections and paragraphs:

1. They can receive or return parameters.

2. They can have their own local variables.

3. They help chop up the code into smaller, more manageable pieces.

4. You can't possibly enter one via a GO TO (sorry, I just had to
throw that in).

However, separately compiled subprograms offer all the same
advantages, plus you can CALL them from multiple programs, plus you
can give them multiple entry points if you think it is useful to
do so. Their main disadvantage is that they make you keep track of
more source files and object modules. Big deal.

I could get used to nested programs if they were the prevailing
practice, but I don't see a good reason to promote them.

Now let's hear from someone who likes them. I'm willing to change
my mind.
