```
If your .NET application runs on a PC (or Linux) the ONLY places that I know
of where .NET is currently implemented, then something is odd in your
request.  VSAM is an IBM *mainframe* implementation of an "indexed
sequential access method" (aka ISAM).  What are you REALLY trying to do?

 - Do "remote" file access from a .NET environment to mainframe situated
files?

 - Access files that were created in a COBOL (or other) application that was
PORTED from a mainframe environment to the PC (or Linux)?

 - Access COBOL "ISAM" files on the PC (or Linux) created by some other
COBOL compiler?

Any (all) of these CAN be done (with existing software) - but we need to
know exactly WHAT you want to do - before helping more.

P.S. There was (well maybe still is) a PC version of something
"sort-of-like" VSAM - part of the "SmartData Utilities" products.  I may be
mistaken on this, but I don't know of anyone actually USING it (or using it
much).  If you want more information on it, you might want to start at:
  http://publibfp.boulder.ibm.com/epubs/pdf/dd0wvmas.pdf

or any of the other "SMARTData" books at:
 http://www-3.ibm.com/software/ad/cobol/va/library/
```

```
If you have an existing COBOL CICS application that has been ported to the
PC - is it STILL a "COBOL CICS" application on the PC?  (This is possible
with a VARIETY of COBOL products and a few CICS products).  If so, that
should give us (me) a clue as to what the VSAM files "are".

Some PC COBOL CICS applications use C-ISAM, some use IBM-proprietary ISAM
files, some use Micro Focus or Fujitsu proprietary ISAM files.

Again, if the application that is CREATING this files runs on the PC, you do
*not* have "VSAM" files.  If you are trying to create a program on the PC to
*replace* a mainframe COBOL CICS program, that can be done too.

Will the PC application still be in COBOL?  If so, use a "native" PC COBOL
compiler and you will "auto-magically" get a VSAM compatible file system
(but not the same from compiler vendor to vendor).

If you are just trying to create a NEW (replacement) application that will
access "indexed files" - but not from COBOL, then I would suggest that you
look at C-ISAM or BTRIEVE, both of which have "full" VSAM compatible
support.  (I don't know if either C-ISAM or BTRIEVE have announced ".NET"
versions - but I would check with those vendors.)

Mainframe COBOL CICS application programs do *not* get into the "structure"
of VSAM, therefore, I strongly recommend that you don't either.
```
