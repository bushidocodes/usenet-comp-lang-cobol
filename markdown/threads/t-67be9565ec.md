As part of a school project I am designing some office system improvements
based on a real office I have selected.

This office presently receives printer output over a dedicated line from a
mainframe several states away, (printer is always on, waiting to receive)
but the local office needs that information to print to a disk as well.

The solution I hope will work is to unplug the cable from the back of the
printer, then plug that cable into a splitter, and feed one end of the
splitter to the printer and one end to a computer (but where on the
computer?).

If that can be achieved, the next step will be to get _cobol_ to recognize
the 'printer signal' and process it into an ascii diskette file.

But can a printer signal be converted to an ascii file?

And, can cobol handle that sort of communication?

Any help would be greatly appreciated.

Thanks,

B.Yates

The sollution is the following.
I assume that you have a serial line.
Connect the cable to the com1 of the computer

make this program.

select file-in assign to "com1"
organization line sequential.

......

fd file-in.

01 file-desc pic x(200).

....
procedure division.

p1.
open input file-in.
p2.
read file-in at end go end-prog.
write record to printer or hard-disk or both.
go p2.
end-prog.

If the system is a unix this program can run in the background.

Best regards

Frederico Fonseca
close file-in.
stop run.

This sort of thing I have done before and though I am a regular supporter
and user of COBOL, this probably should be done in C. Plug the cable into
a serial port, write a program to buffer the port input into blocks and
write the blocks to a file. It may make more sense to plug both the input
line and printer line into serial ports on the computer and have the program
also send the input to the printer port. Using a splitter has the potential
of signal loss and creating handshaking conflicts.

The resulting file, even if written by a C program can be read as a line
sequential file into any COBOL program if you wish to extract the data.
