Hello.

COMP-3 is a data format which stores one decimal digit in each
half-byte of the field. When you want to calculate the number of bytes
occupied by such a field;
* add all the digits (in your example 17)
* add one for the sign ALWAYS (even for unsigned fields)
* add one if the sum, thusfar is odd
* divide the final sum by two.

Thus; (digits + sign + odd) / 2
17 + 1 + 0 / 2 {NOT ODD, DON'T ADD ONE}
= 9 BYTES
-------------------

Ted,

Comp-3 is hexadecimal and each numeric value is stored in a half-word. If you looked at the file (or
whatever the PIC is getting its' data) and saw, for example, the value '123456.78999'; values '12' would be
stored in the first byte, '34' in the second byte, etc. So that PIC S9(12)V9(05) COMP-3 takes up 9 bytes in
storage ([12 + 5 + 1 for the sign] / 2).
I am confused, though, why they are redefining from a PIC 9(09). I would expect it to be PIC X(09) or
PIC S9(12)V9(05).

-------------------------------------

The comp-3 item requires 9 bytes (17 digits + 1 sign). Each 2 digits
is stored in 1 byte except for the right end which contains 1 digit and
a sign. When numeric data is moved to the comp-3 item, it is right-
justified.

The redefinition is useless however, since if numeric data is moved to
the comp-3 name, the data would not be numeric if referenced by the
display name. In addition, if numeric data is moved to the display
name, it would not be numeric if referenced by the comp-3 name.

The only use that I can think of for this is to save space.

COMP-3: Hexidecimal...also considered BPD...or Binary Packed
Decimal... the V is an implied decimal point....

# of digits + sign + 1 (if odd) / 2 = # of bytes...

For example, in the s9(12)v9(5) field, if
-123456789012.34567 was stored in it, it would be...

12 34 56 78 90 12 34 56 7D

the end half byte holds the sign....

C if postive
D if negative
F if no sign.
