A Quick suggestion would be that you define the file as line sequential as opposed to sequential
when reading the record in you will just get the contents of a single line and no further

making the asumption you are creating the file with VI or some other unix editor each line
will have a line feed character at the end of it, when reading the file sequential using a straight
sequential file you would have to add a filler bytr to jump over the line feed character

My tendency for handeling config files is as follows

Open file line sequential and use the following code fragment


perform while true

read file

if at-end
exit perform
end-if

if recname = spaces or recnam(1:1) = "#"
exit perform cycle
end-if

move 1 to ptr
unstring recname delimited space or ":" into fieldname1 pointer ptr
unstring recname delimited space or ":" into fieldname2 pointer ptr
etc...

end-perform

I know unstring is unefficient but there again when reading a config file who cares
using this method a standard text editor created unix file can be parsed easily I
always use the convention of using a colon as a separator and always use the unstring simply
to allow the user to enter numbers as he likes such as

# Text file format number1:number2
123:1224
01223:2233

hope this solves your problem
