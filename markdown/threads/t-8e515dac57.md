Hello,

In a COBOL batch and CICS environment where many data fields are all
digits but never used for arithmetic, I'd be interested in knowing if
there exists any study that shows how installations have defined such
fields, e.g. student, part, vendor, policy, bank account numbers, etc.

Using a six digit field, I can think of:
PIC X(6).
PIC S9(6).
PIC 9(6).
PIC S9(4) COMP-3.
PIC S9(4) COMP.
Is/was/wiill there (be) a prevailing method?

I can think back many years when I often defined a field as one of the
first three above simply because it 'DISPLAY'ed nicely during testing
despite DASD constraints.
Thanks,
Graham Hobbs
```

It is irritating when someone used some datum such as an ID number as
packed decimal just because it is numeric and mandatory.

Things change.   When we are mandated to not use Social Security
Numbers as the Personal ID, the option to use alphanumeric characters
is more expensive now.

At least packed decimal had the advantage of saving space back when.
But now it means we have to translate to decimal before FTPing to an
ANSI machine, and the space savings is not valuable anymore.

But even with PIC 999999999 USAGE DISPLAY, we can have data exceptions
if we decide to make it alphanumeric.    If we're not doing arithmetic
with the ID, make it alphanumeric.   We can do IF NUMERIC for editing.
```

Back in the olden days, COMP-3 or COMP-5 was often used to conserve space.

Today, in my view, if you can't do arithmetic on it, it's not a number and 
should not be defined as a number. There are a couple of reasons:

* Your example of easy display
* The definition of the field may change. In the U.S., postal codes are 
numeric, but if you have somebody who lives in Canada, where they're not, 
you're a goner.

In fact, I'd expand the definition of using DISPLAY to numeric fields that 
are not totally internal to a program (such as subscripts and indexes). That 
is, master and intermediate files should not render numerics in anything but 
S9(n) or similar.
