Not so typical. ENTRY statements aren't that common in COBOL programs, I don't

>
>What is the differece between the ENTRY statement  and CALL statement in
>COBOL ?

An ENTRY statement defines a callable point in a program other than the main
procedure. A CALL statement can name the program-id (to enter at the procedure
division), or it can name the name identified in the ENTRY statement.

So CALL goes out to a program point. That point is the program-id or an
alternate ENTRY point.


>
>What is the right way to link the programs having the CALL as well as the
>ENTRY statements ? 

Several ways. Basically, if the module with the ENTRY point you want is brought
in through the linkage editor's automatic call resolution (i.e.: look in
SYSLIB) , the ENTRY point will be found automatically. Otherwise, you will have
to implicitly INCLUDE the module.

The entry statement defines where in the procedure code of the called
program is to start executing. The CALL would be to that entry point,
using the same name to match up.

The only times I've ever used this delibrately was in a i/o subroutine
(which I didn't have any say in designing) where we called the  main
entry point to open the file, and other entry points for read,
rewrite, write and close. Probably the most common use of entry
statements is for specialized evironments like IMS, DB2 or CICS where
you get something like 

PROCEDURE DIVISION. 
	ENTRY DLITCBL USING xxxxxx. 
PARAGRAPH1.

and then a little program stub is link edited to the program to handle
some environmental issues. 

In your example, assuming static links (NODYNAM) the only reason for
an entry statement would be to define the point where the program was
supposed to start executing when the load module is executed.  All the
other calls should be automatically resolved by the linkage editor
matching the calls to the corresponding entry points in the called
program..

You should be able to dump the need for an entry statement simply by
changing the load order or including a linkage editor ORDER statement
so that the primary module is first in the load module and assuming
that you want to start executing at the beginning of the procedure
division.
