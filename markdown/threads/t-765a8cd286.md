I'm in the middle of writing a conversion program on a IBM mainframe using
COBOL OS/390, and am needing to deal with a large number of unsigned packed
numeric fields.

This is how I'm handling them.... as an example:

Source field defined as:
15  LARAMST-00-ACCOUNT-NUMBER PIC X(8).

Working storage fields I'm using to access this field:
08  WS-LARAMST-00-ACCOUNT-NUMBER     PIC 9(16) COMP-3.
08  FILLER REDEFINES WS-LARAMST-00-ACCOUNT-NUMBER.
  12  WSR-LARAMST-00-ACCOUNT-NUMBER  PIC X(8).
  12  FILLER                         PIC X(1).

Procedure division statements so I can access this field:
MOVE ZERO TO WS-LARAMST-00-ACCOUNT-NUMBER.
MOVE LARAMST-00-ACCOUNT-NUMBER TO
                           WSR-LARAMST-00-ACCOUNT-NUMBER.
DIVIDE 10 INTO WS-LARAMST-00-ACCOUNT-NUMBER.

I've been dealing with unsigned packs like this for years, and hey, it
works, but might there be a "better way" of doing this?

I've just migrated from COBOL II to OS/390 lately, so perhaps there's a
feature in COBOL that I've overlooked just for these?

Thanks in advance...

Gonzo

Without running a test compile, my guess would be that this would generate a
move with offset, rather than a decimal divide.  Try compiling with the LIST
and OPT options.  If you see an MVO instruction, you're OK.  If you see a DP
instruction, you're burning a few extra cycles and you should append a zero
to your original number, move it to a packed field then lose the last byte.
Thus:  X'F1F2F3F4F5F6F7F8' -> X'F1F2F3F4F5F6F7F8F0' -> X'123456780C' ->
X'12345678'

You can still use the same technique, with a minor modification.  Append a
packed zero to the field, move to a character field, then move n-1 bytes to
a comp-3 field:
X'0000000012345678' -> X'00000000123456780C' ->
X'F0F0F0F0F0F0F0F0F1F2F3F4F5F6F7F8F0' ->X'00000000012345678C'
Did you try running a compile with LIST option?  Again, if it's using MVO,
it's probably faster than the UNPK and PACK that this technique would use.
Divide instructions are usually implemented by iterative subtracts and
shifts, which tends to make them inefficient.
BTW, you can change your "12 FILLER PIC X(1)" to "12 APPEND-ZERO PIC S9(1)
COMP-3" which will save a little on the move of zero.
D de J

If you don't mind losing the hi-ord nybble in the source field you could work
from there. E.g., redefine it to pic 9(016) comp, redefine it again to 
pic 9(015) comp-3. Then:

compute src-fld-comp = (src-fld-comp * 16) + 15

If I didn't mix my modulos, you should wind up with 000000000001234F in 
src-fld-comp-3.

If you can't lose the hi-ord nybble, you'll have to move the source fld to a
larger work fld, but the same principle applies.

Gonzo,

        If you move your 8-byte BCD value to the first eight-bytes of a
redefined PIC 9(16)V9 COMP-3 field (ensure a low-order X'0F'), move the PIC
9(16)V9 COMP-3 field to a PIC 9(16) DISPLAY NUMERIC field. An MVO is
performed in the interim ASSEMBLER code, followed by an UNPACK into the
DISPLAY NUMERIC field. No decimal instructions that can get you into
trouble.

        I've been doing it this way for a long time now, in MVS and VSE
environments, without any problems.

        Even better yet, try PIC 9(16)V9 COMP-3 to PIC 9(16)V9 DISPLAY.
Byte-09 of the COMP-3 field doesn't even require initialization, as the
compiler automatically inserts a X'0F'.

HTH....

Bill

Okay, folks... this is how I've decided to handle this:

Working storage is now:
08  WS-LARAMST-00-ACCOUNT-NUMBER     PIC 9(16) COMP-3.
08  WSQ-LARAMST-00-ACCOUNT-NUMBER.
  12  WSR-LARAMST-00-ACCOUNT-NUMBER  PIC X(8).
  12  FILLER                         PIC X(1) VALUE X'0F'.
08  WSS-LARAMST-00-ACCOUNT-NUMBER REDEFINES
    WSQ-LARAMST-00-ACCOUNT-NUMBER    PIC 9(15)V9 COMP-3.

Procedure division is:
MOVE LARAMST-00-ACCOUNT-NUMBER TO
                           WSR-LARAMST-00-ACCOUNT-NUMBER.
MOVE WSS-LARAMST-00-ACCOUNT-NUMBER TO
                           WS-LARAMST-00-ACCOUNT-NUMBER.

The original code generated the following: MVC, ZAP, DP, MVC, an OI.

The new code generates: MVC, MVI, MVO.

I figure a good improvement has been made here. (I tried the OPT, but it
took over 30 minutes to compile, so I went back to NOOPT).
