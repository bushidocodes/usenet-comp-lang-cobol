```
Same idea as Jerry's :-

*>--------------------------------------------------------------
Method-id. "openInputFile".
*>--------------------------------------------------------------
Linkage section.
copy "sqlreslt.cpy" replacing ==(tag)== by ==01 lnk==.

Procedure Division returning lnk-SqlResult.

  set ResultOK to true
  open input Data-file

  if   ws-fileStatus = "00"
       set FileOpened to true

  else set FileError to true
       invoke super "showFileError" using ws-ErrorParams
  End-if

End Method "openInputFile".
*>--------------------------------------------------------------
Method-id. "closeFile".
*>--------------------------------------------------------------
Linkage section.
copy "sqlreslt.cpy" replacing ==(tag)== by ==01 lnk==.

Procedure Division returning lnk-SqlResult.

  set ResultOK to true

  if FileOpened
     close Data-file

     if    ws-fileStatus = "00"
           set FileClosed to true

     else  set FileError to true
           invoke super "showFileError" using ws-ErrorParams
     End-if
  End-if

End Method "closeFile".
*>--------------------------------------------------------------
```

```
jacksleight@hotmail.com (Jack Sleight) wrote

> If the pgm uses FILE-STATUS for the file why not rewrite the IF stmt as:
> 
> IF  WS-FILE-STAT NOT = ('00' AND '97')
>     CLOSE DISKFILE WITH PURGE.

Because there will be other file operations between the OPEN and the
CLOSE perhaps, and these will change the content of the File Status
variable.

Note that when checking the file status for successful completion only
the first byte should be checked as this being a '0' indicates success
and the 2nd byte may supply additional information and may have a
value that is not '0'.

While it may be that you have only seen a result of '00' from your
current compiler and system (or '97' only for IBM mainframe stuff),
some new version, or even porting the code to a different system (as
in this thread), may start to give other success status values which
are '0x' and not '00'.
```
