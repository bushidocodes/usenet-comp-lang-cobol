### Mail and Collaboration Data Objects CDO

```
Inspired by Howard Hinnman's recent sample of .NET code to use SMTPMail, I
started thinking about what those of us who DON'T have .NET can do about
handling e-mail from COBOL programs.

Howard's sample utilises a .NET Class and it is pretty straightforward to
simply invoke methods of this Class (as Howard showed so elegantly.)

However, I am currently doing some stuff where I need to e-mail a
confirmation to a customer who has made a Web Booking. As this code will be
running from a portal, there are a number of different Sellers who my code
must represent.

I need to do the following:

1. Ask the customer whether he will accept HTML or requires Plain Text.

2. If he will accept HTML, send him a nicely formatted document with the
appropriate seller's letterhead, containing the details of his transaction
with us. This should look just like a letter he might receive through the
mail.It will be produced  on-the-fly by a mail-merge in Word or a similar
process (I have a "home made" component that I use for building dynamic HTML
that would do this job with a lot less overhead; I'll decide after running
some tests...)

If he opts for plain text, dynamically generate a suitable message and send
it to him.

I started getting into how mail works and discovered a few things of
interest...

1. There are some limitations on the SMTPMail Class, both technical and
functional.  (It is intended for very quick low-level mail, using a local
SMTP Mail server and a MAPI client.)

2. There have been a number of implementations by MS of an evolving COM
technology that started off as an OLE Messaging service and is today an
advanced COM+ component, callable from all languages that support this
interface, whether you have .NET or not.... (It really saddens me that MS
pay lip service to COBOL support - both Fujitsu and MicroFocus have
partnership arrangements with MS, and MS is on record as saying it sees
COBOL as being an important contributor to .NET, yet I have never seen an
MSDN "How To" that gave sample COBOL code... Someone here was saying
recently that they would like more "How To" stuff with code samples
(presumably, in COBOL <G>)).

Anyway, this component is called "Collaboration Data Objects" (CDO). It's
previous incarnation was CDONTS. (CDO for NT Server), but, as implemented in
Win 2000 and XP it is called CDOSYS. I spent a few hours yesterday reading
about this stuff and came to the conclusion that it SHOULD be available from
COBOL (given that COBOL supports COM+ interfacing.)

I knocked up a Project and tested it. It immediately posted mail to me! I
was very surprised (I never expect things to work without hassle on the
first attempt <G>), so decided to push the envelope a bit... I spent a bit
more time finding out the events, properties, and methods of the CDO
component, and then got stuck into firing off Mail, in HTML and plain text,
with attachments, directly from my test project, via a  local SMTP server on
IIS, AND a remote SMTP server (my local ISP's) and flooding my inbox with
test mail... Great stuff!

All of this was achieved without .NET. Turns out the .NET Class simply
repackages CDO anyway.

If you think you might need this kind of facility, check your
..\Windows\System32\ directory for a .DLL called "CDOsys.DLL".

I won't rob you of the pleasure of discovering for yourself how to use it,
but if you get stuck, mail me privately.

For the less adventurous, I now have a comprehensive e-mail component that
can be used from the Windows desktop, or the Web, and can send AND receive
e-mails from within an application, as well as access the address book and
diary in Outlook (if you HAVE Outlook), but, unlike the SMTPMail Class, does
NOT require a MAPI client (like having Outlook or Exchange server installed)
in order to run.

Never would have got there if Howard hadn't posted his code sample. (Thanks
Howard...)

CLC definitely has value for some of us...<G>

Pete.
```
