The properties you describe are part of the receptacle, not the contents.

You create the contents and these are loaded into a default container.

Try setting up an Excel spreadsheet with the attributes you want (line/col 
width, bold, etc) then save it as a template.

This should give you the .XLS you are looking for.

Alternatively, you can use Automation (Excel as a COM componenet) to access 
Excel directly from your program and set the attributes you want , then load 
the data directly (without needing the .CSV) or run a macro to load it, all 
under program control. There are many ways to do what you are trying to do.

If you want the users to initiate the Excel stuff, and you want it to look 
as you intended, then provide a template for them to use.

HTH,

Pete.

This is a post from a guy who wants to move OFF Excel to Open Office. He 
already has a number of  programs using Automation to Excel and is wondering 
if there is an API in Open Office that can support the same Classes, 
Methods, and Properties. He's looking foir a COM interface to Open Office.

I have no idea whether there is one, but I'd be surprised if it supports the 
same method calls as Excel.

Anyway, this is way too difficult a route for you to try if you are not into 
OO COBOL (although I admire your courage and attitude... :-))

Fortunately, for what you need to do you only need a template. You can 
provide this to the users and tell them to use it when they import your 
.CSV.

That will get the job done.

HOWEVER...

If you DO want to get into program control and use COM Automation (which 
allows you to connect to and control most of the Office Suite (I'm not sure 
about Power Point... never done it), including MS Access and Excel you 
should find  a list of the properties and methods on MSDN. Of course a very 
easy way is to use an Object Browser to look at Excel. You already have one 
on your system to support VBA macros... It's a bit hard to explain without 
posting screenshots, but try this (Description is for Excel 2007; tabs and 
menus may be different in other versions...):

1. Open Excel.

2. Select the "Developer" tab.

3. Click the icon to open Visual Basic. The Visual Basic Editor opens and 
displays a toolbar with small icons on it. There should be one that looks 
like an open cardboard box with 3 shapes floating over it. This is the 
Object Browser. (The toolbar icons have tooltips so floating the mouse over 
it should confirm if you have the right icon.) Alternatively try pressing F2 
while on the Editor screen., that should open the Object Browser

4. Change the drop down that says <All Libraries> to be Excel. Immediately 
you can see all the classes that Excel supports in the left pane, and all 
the Methods, Properties and Events for the selected class in the right pane. 
Note that properties have a hand holding a page icon, methods have what 
looks like a green eraser, and events have a little lightning flash. To get 
more information about any one of these, select it, then click the question 
mark icon near the top of the screen, or click  Help.

I wrote a number of components in Fujitsu COBOL that interfaced to EXCEL and 
ACCESS through COM Automation and found it very powerful.The experience 
gained was useful because now I do it from C#. The only source of 
information I used was the VB Object Browser, as described above.

The number of methods and properties available is indeed overwhelming, but

Paul,

Softly, softly, catchee monkee :-). There's no easy way when tackling 
something new - you'll have to plug away through the documentation 
available from MicroFocus, (and of course MS).

Some questions first :

(1) Is it just ONE file that you want transferred to Excel or several 
with different record formats
(2) Given Excel Spreadsheet 'A' - presumably your users will want to 
keep different sheets per month, year etc., or is it cumulative for the 
year.
(3) If separate per month, then you need an EMPTY Master Spreadsheet 'A' 
that you can carbon-copy for each month etc., and within OLE programming 
save under different filenames.

Think on it - if I have an MS Access Table for which I want to transfer 
data TO a COBOL file, I could write a CSV from MS Access to update the 
COBOL file or even update the COBOL file directly. Ahh... but if I 
haven't got the FD for the COBOL file, then I'm SOL. Reverse applies - 
before writing from a COBOL file to MS Excel (or Access) I've got to 
know the Spreadsheet/Table format !

Specify your Excel format first giving you a blank Spreadsheet 'A'.  The 
fonts in Excel - that's something you can specify when setting up blank 
spreadsheets - it can of course be done through OLE programming.

Some sources you should be looking at :-

1) MicroFocus Forum - 
http://www.cobolportal.com/microfocusforum/index.asp?bhcp=1

Join and going to Net Express search (in the toolbar) for "XLS", "EXCEL" 
"OLE" etc.

While in the Forum do a search on 'Young Kim' by his name - the message 
is "Calling MS SQL Server APIs", dated 2002 Oct 23 - His code covers ADO 
and uses COBOL files to update EXCEL and ACCESS. If you like what you 
see you could try and contact him for the missing copyfiles from his 
source - Young Kim ysk@kimsoft.com. (He was teaching OO COBOL at a 
university in Ohio).

2) MicroFocus KnowledgeBase (new format) -
http://kb.microfocus.com/display/1n/searchDirect/?searchString=21852**
&searchType=allwords&searchby=keywords&r=0.474005520343781

(I've split the above URL by the two * (asterisks) before 
'&searchType......')

 From the Treeview - go to Net Express

3) M/F On-Line examples -
http://supportline.microfocus.com/examplesandutilities/nesamp.asp

Look at 'SQL' - I know it's SQL but the difference between accessing 
Excel or Accesss is not that great

4) Last and certainly not least - On your own hard disk go to N/E under 
Program Files and from 'Merant' sub-folder to ---> Net Express----> 
Base---->Demos----->OLEDEMOS......> several sub-folders, including Excel

Can't recall which of the above uses it but there's a copyfile 
OLETypeLibrary.cpy. This is an extract of methods applicable for Excel
