Mohan,

thanks for the clarification.

Your set theory examples are a "longhand" form of Boolean Algebra.

It "translates" as follows:

U = union = OR = +
^ =  intersection = AND = .
'  =  negation = NOT = '

What I believe you are looking for is an algorithm for Boolean
Simplification of a complex expression.

As far as I know, there is NO general algorithm to do this. However there
ARE some basic rules of Boole's Algebra and these can be stated as follows:

1. It is commutative for addition and, consequently, multiplication. (note
that AND represents multiplication and OR represents addition..)

A.B = B.A    A+B = B+A

2. It is associative for these operations also.

A.(B.C) = (A.B).C   A+(B+C) = (A+B)+C

3. It has a multiplicative identity. (1 and 0 represent TRUE and FALSE)

A.1 = A    (literally, "A AND TRUE = A")

4. It has an additive identity.

A+0 = A  (literally, "A OR FALSE = A")

5. It has a NEGATION operation.

A'  (this leads to an interesting effect... A.A'=0  A+A'=1... A AND NOT A is
ALWAYS FALSE, A OR NOT A is ALWAYS TRUE)

6. There are 2 further Laws which were formulated by the French
Mathematician, De Morgan, and are therefore known as "De Morgan's Laws"...

De Morgan 1:

(A.B)' = A' + B'    (A AND B all negated, is equivalent to NOT A OR NOT B)

Many computer programmers recognise this as "if you negate an AND it becomes
an OR" and vice versa.

De Morgan 2:

(A+B)' = A'.B'     (A OR B  all negated, is equivalent to NOT A AND NOT B)

Given the above simple, basic, rules it is possible to simplify some very
convoluted and complex logic.
(There are some additional rules regarding implication, but these are more
advanced and push the Algebra into something called Propositional Calculus.
Not necessary for what we are considering here...)

Some time ago in this forum "Yukon Mama" (Eileen) posted some COBOL code
that was pretty hideous and taken from a program she was maintaining which
had obviously been over maintained and grown like Topsy.  She asked for
opinions about re-structuring the code and  many of us responded with
various solutions.

Normally, I would have a copy of my post available but it is still in the
process of being recovered so you will need to search the Google archive for
this Group.

It took about 20 minutes to apply the above rules and come up with something
mathematically equivalent (same truth value) but less convoluted.

Now let's have a look at what you are REALLY looking for...

Please ignore the code I posted in response to your request; it doesn't
address this at all and simply shows that I misunderstood your question.

I'll start with your own example (translated to Boolean notation as
described above...sorry, I just feel more comfortable with this form):

 (AB').C)+ (AB) [note that A AND B can be written as "A.B" or simply "AB"]

1. Regrouping by the associative law...

A.B'.C + A.B = A.((B+B').C + B)

2. From Rule 5 above we see that B+B' is always TRUE, so we can write...

A. ((1.C) + B) or simply... A.(C + B)

which translates into "English" as "A AND (B OR C)"

As you noted, this is fairly trivial, but the difference is that each step
of the above is based on the Rules described.

It is the RULES which allow the Simplification, NOT a general Algorithm.

Having said that, you COULD describe a set of operations, based on the
rules, which could be applied to the simplification of any complex Boolean
statement, and with ANY number of variables...(the actual numberof variables
is irrelevant; it is the procedure you follow which will derive the
simplified result).

This would be a bit like what you would do in normal Algebra when required
to solve a complex equation...start by collecting like terms, multiply
through by various factors, etc (forgive me, Maths is not my strong point...
but you understand what I am saying...) It is the PROCEDURE, based on the
rules, which obtains the result.

You have already learned about the use of Truth Tables to check Boolean
simplifications but you are quite correct that these can become unwieldy
when many variables are involved. That's why the use of the rules above,
under program control, can provide a general solution for any number of
variables.

Incidentally, around the time of Eileen's problem I considered writing such
a "Logic Simplifier" to be used as a tool for re-factoring Legacy Code. It's
been done... apparently there are several such products already in the
marketplace.

How would you do this in COBOL?

The same way you would in any other language. Decide the steps for your
simplification procedure and program them.

ANY language that supports AND, OR, and NOT (Boolean) functions could be
used to do these simplifications. (As these operations are fundamental to
the working of a modern computer, it is extremely unlikely that any TRUE
computer programming language would not include them. (I don't include stuff
like HTML as a "computer programming" language, although there are those who
would argue this...<G>).

Once, many years ago, I wrote a computer program which ONLY used the Boolean
Assembler operations. It was for a bet, and the people I wagered with could
not believe that the machine was reading punch cards and printing data
without any standard I/O instructions, ONLY AND, OR, and NOT  and Load and
Store registers. It took me two months in my spare time and the finished
program ran in 4K...(Ah! happy times...but I digress...<G>))

Summarising:

1. I know of no "general algorithm" for simplifying Boolean expressions. (If
you find one, please let me know... I'd be interested to check it out)

2. A general procedure for the simplification of such expressions could be
formulated, based on the rules of the Algebra. This could be the "general
algorithm" you are looking for but I would argue that it isn't. My argument
goes that different people will formulate this procedure differently. (In
the same way that no two programmers will write the same code to solve a
given problem).

3. Your best bet is to study and understand the rules, and annotate your
simplifications showing which rule was used at each stage. Doing this will
help you to see if any patterns emerge in your own approach to the problem
solution.

Hope this helps...

Pete.
