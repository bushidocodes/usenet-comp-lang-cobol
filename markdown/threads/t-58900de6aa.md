I'm writing an API in C that needs to call a COBOL program by reference
instead of by name. It occurs to me that, since any COBOL implementation
that supports a level 2 Inter-Program Communication Module allows writing
something like:

01 identifier-1 PIC X(8).
.
.
.
MOVE 'whatever' TO identifier-1.
CALL identifier-1.

such an implementation must have a way at run time to convert a character
string into the address of executable code (assuming, of course, that what
I've moved to identifier-1 really is the name of some code in the current
run unit). Furthermore, since I find no requirement in ANSI X3.23-1985
that the contents of identifier-1 name a contained program, it would seem
that the mechanism for doing this conversion must be externally available
somehow (either by way of a system call or, more likely, in some table
produced by the linker). What I'm wondering is how I might access this
mechanism from either C or assembler.

I imagine that this is really an operating system question, not a COBOL
question; but it occurred to me that readers of this newsgroup might
have come across this problem before. I need a solution for both IBM
mainframe MVS and VAX VMS; and it would be nice if I could get it to
work with Micro Focus COBOL on Windows NT as well, although that's
not a current requirement, just one that I'm anticipating.

Thanks,

--Bill Seymour (wse··.@ema··s.gov)

I would like to add, that the advantages of this mode of calling, i.e.
dynamically, instead of branching to a statically linked subprogram in
the same load module, are many: 1) you do not need to relink any
modules that call the dynamic module when recompiling the dynamic
module, 2) you keep load modules (executables) smaller, when common
subroutines are called dynamically, 3) you acquire working storage only
once, even when using this in an online environment (f.ex. CICS).

Your example is called dynamic linking in IBM mainframe COBOL. Nothing
special is required to set it up. The subprogram must be compiled and
link-edited, and then stored in a library that is on the search chain when
the calling program is executed. Then MVS reads directories until it finds
the program. You get an 806 abend if MVS cannot find the program, otherwise
MVS loads the program into the caller's address space and links to it. It
should stay resident until the calling program terminates or CANCEL's the
subprogram.

I have no experience with VAX VMS, so I don't know what it takes to set up
dynamic program linking in that environment.

In character-mode DOS and OS/2, I have done dynamic linking with CA-Realia
COBOL V4.2, but it was harder to set up. You had to do special things with
the link-edit. Fortunately, they gave you .BAT or .CMD files to build it,
but it required you to link-edit the subprogram FIRST as a .DLL file, then
you had to link-edit any calling program afterwards with a special BAT/CMD
file in order to read the exported definitions created for the DLL. I tried
to explain it once to a Windows C programmer who was skeptical that a DOS
program could call a DLL. But I realized I didn't fully understand the
linkage requirements.

One other thing you should be aware of (perhaps you already are), is that in
most operating systems COBOL requires some kind of external runtime library,
even in IBM mainframe MVS. Therefore, to call a COBOL subprogram from a
non-COBOL main program usually requires some tedious setup to ensure that the
COBOL runtime is initiated. Mainframers sometimes don't discover this until
they try to call a COBOL subprogram from an Assembler Main program. That's
why IBM is pushing their customers toward LE/370. If all language
translators use the same runtime library then it's a lot easier to mix
components from Assembler, C, PL/1, COBOL, et cetera, without regard to
which language the main program is in.

Good luck!

Bill,
There are two flavors to the COBOL call command and the results of the
actions are quite different dispite the similarities in the syntax of the
statements. In the first form, or static linking, the command
CALL 'XYZPGM' is actually a directive to the compiler to leave the
address of XYZPGM unresolved and let the linker program join the load
module for XYZPGM with the calling program and covert the CALL statement
to a BALR command. In C this is the same as writing XYZPGM as a separate
source file and using CC -o Mainpgm Mainpgm.o XYZPGM.o. The static
call acts as a fucntion call.

In the second form of the Call statement :
move 'XYZPGM' to ws-called-pgm
CALL ws-called-pgm

the compiler marks the call as a weak unresolved external; the linker does
not join the load module XYZPGM into the calling program and code is
generated to invoke the MVS Call Attach facility to start a second process
with the following protocol :

R15 = address of called module
R14 = return address of calling module
R1 = address of parm list

Under MVS using C/370 you could invoke the Call Attach facility directly (
I believe there are LE/370 library functions to do this, BookManger has
much information in it on this subject if it's availabe) but in a Vax or
other Unix environment it is equivalent to doing a fork of the main
program and an execl() or exec() in the child process to start the
desired subtask.


The terms by reference or by value actually apply to parameters passed to
the called module, not the call itself.

InstJCC
