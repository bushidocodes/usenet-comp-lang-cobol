If memory serves me well the schedule PSB call will have two parameters, the
function and the PSBname.

I am not certain you actually get a PSB address per se. Instead, you need to be
trafficking in one or morePCB addresses.  Generally, those addresses would have
been resolved upon entry (ENTRY 'DLITCBL', or procedure division USING clause.

The ancient tradition of program interfaces in the mainframe environment was to
pass a pointer in register 1 to a list of parm addresses.  So for an IMS
application program at entry, register one pointed to a list of PCB addresses. 


The tradition further held that such an address list could be variable in
length, so we needed a way to terminate the list. The high order bit of the
address is turned on to so designate the last (or only) PCB on the list as the
end of the list.  In a 24-bit address achitecture, the entire upper eight bits
of addresses used to be zero, so a end of list address would look like it had
x'80' in it.

This is not a negative number.  The bit is a semaphore.

This style of address management causes problems in 31 bit architectures. 
There  are a lot of scrub-up mechanism to get rid of the x'80' where necessary.
 But you have to inform the compiler and linker of the kind of combinations you
intend so that the right service module bindings will be brought in.

The other responders are right-on in guiding you to the link parms, load libs
and LE settings (if any).  But you may have to resolve whether or not you have
a viable plan.

To use the native DLI calls (if they introduce DATA 24 link requirements), may
not be compatible with what you really want to be doing with CICS. Which would
seem to raise the gruesome prospect of a conversion to EXEC DLI, which I would
bet increases the project size intolerably.

Do you have assurance from the vendor that native DLI calls are viable in the
CICS and OS environment that you plan to deploy?
