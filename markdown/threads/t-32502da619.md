I believe there is still a good case for using sections in COBOL.

With the advent of "Go To" less programming, I see more and more students
trying to put EVERYTHING in line (No, I am NOT advocating that we all go
back to using Go To...<G>) with a plethora of scope delimiters at the bottom
of the code.

Time and again students send me code, unable to find an obscure logic bug
that becomes visible as soon as the program is structured properly into
functions.

Most of them are unaware of the availability of SECTION and so they perform
paragraphs. While you can get away with this for relatively simple batch
processing, it complicates things when this code is maintained (new code is
added) and/or there are time constraints on its execution (as there are in
an on-line environment).

Using SECTIONs and thinking in terms of SECTIONing code, I believe makes the
program much easier to maintain.

I encourage people to make a SECTION for each box on their structure or
Block diagram. In this way the thinking is "compartmented" into functions
and it is very quick and easy to know where to go when problems arise. It
also helps to remove duplicated logic which is another cause of errors in
in-line code.

For programs which are time critical, I DO advocate using a Go To to get out
of the SECTION via a common EXIT.

Here's my case: (This is an extract from the advice I give people who ask
for it).

=================== begin quote ===============================

You are probably wondering why I have emphasised the use of SECTIONs, when
there seems to be little difference between a single section and a single
paragraph.

A SECTION can contain multiple paragraphs and this has implications.

Consider the following:
*----------------------------------------------------------
Xample section.
X000.
    Some codeï¿½
     If <some condition> [we want to bypass all the remaining code and get
out of the section.]
         Go to x999
     End-if
     More codeï¿½
     .
X999.
     Exit.
*-----------------------------------------------------------
Nextbit  section.
N000.
And so onï¿½

I am sure you are being taught NOT to use GO TO, but the above is one of the
cases where its use is really justified. The important thing is to make sure
that your GO TO does NOT branch outside the section it is defined in.
(ï¿½GO TO N000ï¿½ in the above would be disastrousï¿½)
In most cases there is no need to exit the current section, but in some
time-critical processes, it is important NOT to execute any more
instructions than is absolutely necessary, so this GO TO gets us off the
hookï¿½

When you execute the Xample section you do so as follows:

Perform Xample  [Note that we perform the SECTION name. This will execute
all paragraphs within the section]

Whether you exited prematurely via the GO TO, or you went right through to
the exit, control will return to the statement following your Perform in
exactly the same way, just as you expect it to.

You could not achieve this effect by using a single paragraph rather than
using a section.

[ASIDE:
 You COULD achieve the same effect by using PERFORMï¿½THRU. For example ï¿½

Perform X000 thru x999

The problem with this (apart from it being ugly <G>) is that there are
implications for maintenance.

Suppose another paragraph is added AFTER x999? Maybe a common process that
MUST be done, whether we took the GO TO or not. We must now remember to
amend the PERFORM to change its range. If we had used a section, this would
be one thing less to have to think about.

It may sound a bit trivial but in the real world it might be necessary to
add several paragraphs at different points in the process flow and the logic
might be complex.

PERFORMï¿½THRU is best avoided. Use sections.

END ASIDE]

===================== end quote ================

I realise that many here will disagree, and I also respect the rights of
individual programmers to use whatever is comfortable for them.

My experience in working with students over the last few years has shown me
that SECTIONing code, drastically cuts the error rate.

Pete.

I wrote an informal proposal to the standards committe that would add
label qualifiers to section and paragraph labels.  These would set the
valid uses that the label could be referrenced and thus provide warnings
and/or compiler errors to avoid the potential bugs that exist with
performed sections.

(see, I am not an avid anti-perform-section advocate, or even an
anti-goto advocate, I am only an anti-bug advocate. With label
qualifiers these areas exposed by perform section could be eliminated).

In Cobol there is only one label type (well two if sections are counted
as different, but they are the same 'type' of label).  These labels may
be referenced in various ways: PERFORM, THRU, GO TO, drop through,
SORT/etc procedures, ..

In other languages different _types_ of labels are available, each for a
specific type of reference.  In C there are procedures that can only be
called, labels that can only be goto'ed.

The flexability of Cobol (or inability to specify the type of label)
leads to many classes of potential bugs that do not (or cannot) occur in
other languages.  One way to eliminate these bugs is to restrict labels
so that they can _only_ be used in specific ways.  One _provable_ way of
doing this is to eliminate GO TO (hence gotoless programming) and to
eliminate sections in PD (actually it would be equally valid to
eliminate paragraphs and only have sections but that is far harder to
get right and to check for).

Another way is to add restrictions to each label, by way of qualifiers,
such that their misuse is detected by the compiler.

The Qualifiers were (from memory):

    LOCAL -reference from outside current section invalid
    PERFORMED - may only be performed, not dropped into or go to'ed
    FROM - may only be subject of GO TO
    INTO - may be dropped into.

These may be used in combination.
    
one might add NULL for a commentary or structure label that may not be
referenced.  This could be useful to add section headers for overlaying
that may not be performed.

This means that the code might be:

    Process-something PERFORMED SECTION.
    PS-Start NULL.                         (or LOCAL)

        IF (condition)
            GO TO PS-exit
            .
        more code
        .
     PS-exit LOCAL FROM.
        EXIT.

     Another-pargraph PERFORMED.
     ....

We can see here that several bugs can now be detected by the compiler. 
For example a GO TO wrong-exit would be detected because it would be
non-local.  A perform of PS-Start (mistakenly done instead of the
section) would give an error.  It should find that both PS-exit and
Another-paragraph will be dropped into (a GOTO PS-exit would be
_required_ at the end of the previous paragraph).  Note that EXIT does
not exit, many newbies make the mistake of thinking that EXIT is like
C's return leading to nasty bugs.

These changes would help to eliminate the potential for bugs that
performed sections inherently have.  With these changes to the language,
performed sections could be as safe as performed single paragraphs and I
would have no objection to this being taught.  Without these changes the
students are being misled into bad practices (IMHO).

Note that the problem with programs that have GO TO's is _not_ at the GO
TO, the problem is at the label (I have another thesis on this).

One thing that still "surprises" me is the GENERAL (not true in ALL cases)
division between US and Europe on Paragraphs vs Sections (in the Procedure
Division).

 - It is unusual (not totally unheard of) to find SECTIONS in "new" code in
the US
 - It is "common" (but not universal) to find SECTIONS in "new" code in
Europe

(or at least that was my experience with both Micro Focus and IBM mainframe
users).

Personally, given the introduction (in some compilers now - and in the next
Standard) of EXIT PARAGRAPH,  I still "recommend" (for maintainability):

1) Never use Sections (in the Procedure Division <G>)
2) Never use PERFORM para1 THRU para2 (even if para2 is "just" an EXIT)
3) Never use GO TO

There is plenty of evidence that ALL programming "algorithms" can be coded
this way.  Obviously, if you are "maintaining" a program that uses one or
more of these constructs, it might be useful to "maintain" the current
style.  However, for "new" code, I just don't see where/why using any of
these really "helps" - and they CAN all lead to programming errors in
maintenance.

I think it only fair to include the information from Micro Focus
as given in their COBOL User Guide, 3.2, April 1994,
page 4-12 - 4-13.

# Perform sections, not paragraphs. Put EXIT PROGRAM
   and STOP RUN statements at the end of the first (main)
   section.
# Only use GO TO to paragraphs within the same section.
# Do not use PERFORM ... THRU statements.

I like the above because it allows the occasional GO TO
in a controlled manner.

Steve McConnell, in "Code Complete," page 780, wrote:
"Maintenance programmers spend 50 to 60 percent of their
time trying to understand the code they have to maintain ... ."

Anything, and I do mean anything, that tells the maintainer,
clearly and unmistakably, what the code does, is better
than adherence to dogma. My biggest complaint is that
many flags are lies and flags that hide GO TO are big lies.

Using structuring methods that are too restrictive means
the programmer will need, occasionally, to lie to the
maintainer. When that happens the code will be more
difficult to maintain. When the methods are too loose,
the programmer may fail to create maintainable code.

The Micro Focus list seems about right, to me.

Example where using SECTION and GO TO was faster
than paragraph.

Initial code was in the form:

para.
    *> setup a table of 32 conditions
    perform varying xi2 from 1 by 1 until xi2 > 32
        if condition (xi2)
            *> body
        end-if
    end-perform
    *> save results

Faster code was in the form:

sect section.
    *> setup a table of 32 conditions
    move 0 to xi2.
head-of-loop.
    add 1 to xi2
    if xi2 > 32
        go to end-of-loop
    end-if
    if condition (xi2)
        *> body
    end-if
    go to head-of-loop.
end-of-loop.
    *> save results

Notice that the only difference is the replacement of the
PERFORM VARYING with the equivalent lower-level
code. After this change, the program ran 30% faster.
The improvement occurred because the lower-level
loop control was slightly faster than the PERFORM.
The magnitude of the improvement occurred because
the body was executed less than 10% of the time. Loop

I need to address these items, separately. Item 4 will take
some time to answer, so I will answer only item 5 at this time.

The primary effect on optimization, of GO TO, is "in-lining";
that is inserting instructions in-line, or replacing a PERFORM
with the instructions performed to eliminate call / return
overhead.

When a program consists entirely of paragraphs, a single
GO TO statement must prevent in-line optimization because
it is not possible for the compiler to determine flow of control.

When a program consists of sections, a single GO TO
to a paragraph name within a section, need not prevent
in-line optimization. This is because the compiler can
determine flow of control. Therefore, the perform of a section
containing a GO TO with a destination within the section may
be "in-lined". A GO TO outside the section must inhibit
in-line optimization.

One of the examples I provided showed a 30% increase
in performance. The compiler directive OPTSPEED was
used both before and after the change. However, I believe
this compiler did not have in-line optimization.

Sometimes the programmer can do better than the compiler
when optimization is required. We do this often when we
select a faster algorithm. We sometimes make an algorithm
faster with a well-placed GO TO.
