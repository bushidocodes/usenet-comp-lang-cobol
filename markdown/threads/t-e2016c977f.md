```
Can anyone point me to a simple text-compression routine implemented
in generic COBOL 74?

I'm building large files to be passed to another system, and the output is
really eating up the disk space. We're already dropping trailing spaces.
The data is all printable, mostly streams of digits, and we could go with
packed decimal, but there are too many different types of records being
transferred to allow this format-sensitive approach.

So I'd like something that would just take a buffer of text and do some
relatively clever squishing, plus the corresponding unpack-code to
unsquish it at the other end. I'm not expecting pkzip quality, but a 3:1
compression would be really great.

I can't justify much effort on this, but if anything is out there for
the taking it will save us some hassles.

Thanks,

Vic Odhner
```

```
Experiments here on typical data have yeilded about 50% compression with a
Run Length Encoding technique. So far implementation has been in a
variety of languages - usually non-COBOL due to the difficulty in easily
generating a one byte binary number.

The basic algorithm is:

scan forward thru data, if the next symbol is not the same as the current
symbol then emit the current symbol, and move the pointer to the next
symbol.

If the next symbol is the same as the current symbol, then accumulate the
occurances of the same symbol.

When the same symbol string is broken, or EOF is reached, emit the escape
symbol, a count (typically one byte), and the repeated symbol.

If the repeat count is 2 or 3, then just emit that number of the repeated
character.

An exception must be handled if the escape character is encountered in the
input stream. Then the output must alway be escape, count, escape. Thus
the escape character is expanded by 3. Moral of this exception - choose a
good escape character.

For fancier algorithms, unfortunately all in c-speak - a very good book is
available called "the data compression book". Sorry, no ISBN number, but
it was easily obtained from a computer specialist book seller.

Advantage of the RLE type algorithm, all symbols stay as 8 bit characters
(as opposed to Huffman type coding where the symbols become variable
length). Also no fancy data structures. Nothing but a one byte store,
and a one byte counter is kept. One can get fancy, and scan for a good
escape character, then always use it as the first symbol in the file, etc,
etc, but it is really a simple approach that will likely eliminate the
need to chop trailing blanks

survive in a legacy based world.
```
