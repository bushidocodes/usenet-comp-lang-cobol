Subject to compiler yes you can - and others will comment.

However here's something that will be less than apparent. You can READ in
varying records PROVIDING  the file you are reading in was created in the first
instance with varying specified; just changing the  format specification of an
existing file to say it is now varying, wont get you very far. (I know, I
recently found out the hard way <G>).

If you currently already have a large file on which you want to do this, you'll
have to write an intermediary program -  Old file = fixed, New file = varying,
then write Old records to New records. Now you can use the latter as input or
output with varying.

Jimmy, Calgary AB

I don't think you can do directly what you want. Without the file being
written by Fujitsu as a variable-length record file, the file itself
contains no usable information for the OS to deconstruct the individual
records (most variable-length routines actually write the record length at
the start of each record). Obviously the program that wrote the original
file had a similar technique, but the original technique is incompatible
with Fujitsu's (FJ uses 8 bytes at the beginning of each record to designate
the length and you have only four).

If you can get your vendor to build the file as:
Record Length - 4 bytes
Filler 4 bytes with binary zeros
(rest of record the same)

Your problem goes away as the file now conforms to Fujitsu's method.

Fujitsu does provide some external routines (CBL_xxx) to access files.
Here's how we made their CBL_READ_FILE routine work for non-conforming
variable length files:

StartByte = 0
LOOP:
BytesToRead = 32000
Read
Extract record from beginning of block,
Add this record's length to StartByte (don't forget the 4 bytes at the
beginning)
GOTO LOOP

Here's how you might configure the code for your case:

StartByte = 0
LOOP:
BytesToRead = 4
Read into BufLen
Add 4 to StartByte
Set BytesToRead to BufLen
Read into data area
(process)
Add BufLen to StartByte
GOTO LOOP


An alternate way is to read the file one byte at a time, building each
record manually.

This, of course, is a HUGE performance hit.

Hi Mark,

In a previous thread you showed us a record fragment in hex: 

6CBF0000 00500000 00000000 00000000

This looks like the typical VB record format for the 1st record in the block.
The 1st 4 bytes (27,839) being the block size and the 2nd 4 bytes being the
record length (80).

If this is true, then you can read the file using the following in your COBOL
pgm:

     FILE SECTION.
         FD  YOUR-VB-FILE
             RECORDING MODE IS V
             BLOCK CONTAINS 0 RECORDS
             RECORD IS VARYING FROM 1 TO xxxx
             DEPENDING ON BYTES-IN-REC.
         01  YOUR-VAR-REC.
             05  FIL OCCURS xxxx DEPENDING ON
                 BYTES-IN-REC           PIC  9(008).

  where xxxx is the length of the largest record defined by the external vendor
*excluding* the 4 byte segment length.

In the JCL DD stmt the BLKSIZE s/b the largest blocksize expected, including
the 4 byte BDW and the 4 byte RDWs.

When you do it this way you won't see the hex stuff, just the user data.

I forgot the most important part! The "user data" length of the record
(excluding the seg length field length) just read will be in BYTES-IN-REC.

I'm not sure what you want to do after you get your hands on the data, but I
think this should get it for you.

HTH, Jack
