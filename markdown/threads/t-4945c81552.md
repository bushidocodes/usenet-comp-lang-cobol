Foodman wrote:
> 
> > Some influences off the top of my head (various tools trade off the
…[3 more quoted lines elided]…
> ---- As much as you need whatever connectivity is.

By connectivity, I mean the way your application interfaces with other
applications.  Do I need to communicate with a bank, with the web, with
Wal-Mart, with the IRS, using TCP/IP, a LAN, etc.

"As much as you need" isn't sufficient.  Until I analyze the particular
project I have no idea what my needs are.  And it isn't a linear scale -
the best tool for connectivity with my bank may not be the same as the
best tool for connectivity with a supplier.
 
> > How much security is desired
> ----How does this affect the choice? All systems should be designed to
> be secure.

Nothing is 100% secure, but a PC with no connectivity comes close.  Some
tools make it easier to make my internet connection secure, others make
it easier to make sure only the correct employees log on, others have
built in compliance to my bank's standards.  Some are easier to have
off-site backups.  Some are easier to lock up behind guarded doors.  
 
> > How much data integrity is desired
> ----How does this affect the choice? Data integrity is assumed. WOuld
> you design a system without data integrity?

Sure (to some degree).  Every time you download data to a user's PC,
data integrity is risked.  There is no way to guarantee that his copy of
the data stays in sync with the master copy.  If your web application is
cached in a remote site, data integrity is risked.  Shared update of
data is more risky than protected update, but protected update doesn't
allow any other users to use those data.

We warehouse denormalized data - which means a degree of data integrity
is lost.

You can't have everything, but have to pick and choose between
conflicting values.
 
> > Do I want the tool to have an nice GUI?
> ----Of course, why should it be ugly? this does affect your choice.

Thanks, a choice.   But there are other choices between "nice GUI" and
"Ugly GUI".  Most of my applications have "No GUI".
 
> > Do I need to connect to cash boxes or other hardware?
> ----In the survey, no, nothing unusual or special like that.

Most real applications have something unusual or special.  Otherwise, we
would plug in a turnkey application.
 
> > Will my applications be used simultaneously by lots of people?

How about...

* expected life expectancy of the system. How I'd write a throwaway
conversion utility is different from how I'd write a system that's expected
to last forever.

* stability of requirements which impact ongoing maintenance. If I'm
anticipating lots of changes then I would attempt to isolate those
anticipated changes so as to create a single point of maintenance.

* whether or not the requirements can be met by an existing solution.
There's no way I'm going to write a spreadsheet program when I can buy one
for less than $100.

* need to scale. It's fine to write for the PC but what happens if I get a
bigger sale, or the customer's business grows. Suddenly I need an enterprise
solution with no way to get there. Conversely, developing an enterprise
version for a single user is overkill.

* uptime requirements. Hardware and software choices change if I need
reliability.

* who's likely to be stuck maintaining it and their skill set. I don't want
to be stuck maintaining this beast forever so there had better be a large
pool of talent available to take on the task.

* what does the end user wants to see? There's no point in developing a
state of the art GUI system when the end user isn't a human being.
Similarly, there's no reason to limit the design to a simple GUI if a
virtual reality is required. How's about an arcade-style gaming nterface to
your payroll system? Imagine the fun HR could have 'terminating' employees.

* is the application for business users, consumers, military, internet,
machine-to-machine, real-time, etc. They all impose different requirements
which influence the choice of development tools and hardware.

business logic. That gives me lots of reliability and data integrity. The
user interface can be anything from punch cards and paper tape to virtual
reality engines and voice recognition.
