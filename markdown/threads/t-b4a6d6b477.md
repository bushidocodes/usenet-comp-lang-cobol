I've noticed some postings of examples using somewhat antiquated code.
I think it would be in the best interest of all if the latest COBOL
syntax were used rather than older stuff (actually, the latest is 10
years old). One post said to do a simple loop this way:

PERFORM abc VARYING ...

and "abc" was:

abc.
EXIT.

Instead, use

PERFORM VARYING ...
CONTINUE
END-PERFORM

Another in comparing PERFORM against other loop constructs said the
test is always at the front. It can be at either end:

PERFORM [para] WITH TEST AFTER

or

PERFORM [para] WITH TEST BEFORE

You can specify if you want the test after the loop (AFTER) or before
the loop (BEFORE). Much more straightforward that other languages.

I hope nobody thinks this is a flame. I just think it is best to use
the best tools. I realize that lots of you are still stuck with
compilers that do not accept COBOL 85 syntax (after all, it has only
been available in some compilers for 10 years), so you can't use the
newer syntax. I do feel the majority can use COBOL 85 syntax,
however.

not warping young minds to do nonstandard things.

Your point about perform until with test after/before was good. One
thing I wonder about is if there will be a perform while loop. This
would ease many programmers coming from Pascal and C backgrounds.
Perform until is done UNTIL a condition is true, While a perform
while would be done WHILE a condition is true.

Although I am so used to perform until that I don't have to think about
the double negative logic with it, I think the language would be
served by having a perform while option...

Where I work we have decided to go with the COBOL85 standard for new
development. From what I am hearing from the developers they really like the new
features of the language. They are even taking old crusty COBOL74 programs and
migrating them to COBOL85. When I ask why, they reply "It is easier to maintain
and reuse code in COBOL85." They are even going to the point of using the nested
program feature to write reusable procedures that they can use in other
programs. They have discovered the use of EVALUATE and even the INSPECT
CONVERTING. The developers also seem to have changed their analysis for
developement. They now take into consideration the use of nested programs and
code libraries (on Unisys A Series) to make shared code libraries that can be
used by any application. It is just great.

We do not have any figures yet on how much we are saving in maintenance costs
but our development time seems to be reduced by a conservative 20%. We are still
trying to get these numbers.

The use of the in-line PERFORM alone will save save your developers much time.
You really have to try it to get a feeling for COBOL85.
