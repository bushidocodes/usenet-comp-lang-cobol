We have been having a lot of problems with this, both for Indexed and
sequential. MF supports two different methods for this in their current
product (Object COBOL 4.1). You *may* be able to get them to work in your
environment.

The first is their 'file striping' method, which divides the file into 2
gig pieces, and then uses a control file to tie them together logically.
This is OK if you only want to write out your own files, but it is not
going to solve the problem of large input files. Some of the documentation
indicates this only applies to indexed files, but their tech support
assures me that it will also work with sequential files.

Secondly, some of the routines are available to run larger files if you are
on a platform which supports 64-bit processing. The standard callable I/O
routines, which the standard CALLFH calls, use a 4-byte position register
to determine the offset for each read. Obviously this limits the size of
the file read to 2 gig, regardless of your operating platform.

MF has a new set of callable routines, and CALLFH for their UNIX versions.
On our version (Solaris UNIX) we have been told that this is an 'Early
Release' function, and that we must relink the runtime system to use this
feature. This in turn requires a C Development system license for your
Solaris, as the COBOL runtime apparently calls shared object (e.g.
subroutine) libraries that only come with the C compilers. This
development system is no longer bundled into the Solaris OS, so you get to
buy yet another product. Worse, the low level (about $1,000) version does
not seem to have all the required libraries. We are working our way up
that list (next stop the $1,500 version...)

Interestingly enough Fujitsu sent us a set of 64-bit I/O routines
compatible with the MicroFocus UNIX product. We do not have the C
Development system (the purchase order is still in accounting.....), so I
haven't been able to link them into the MF RTS, so I cannot tell you
whether they work better.

You might ask why Fujitsu would supply us with code for MF's UNIX product
when we currently only own licenses for the Fujitsu product on NT. This is
because they really ARE willing to go above and beyond to help their
customers. This is the sort of thing I think COBOL people should keep in
mind when they are selecting whose compiler to purchase.

All of this is amusing if you consider that the IBM 370 architecture is
only 32-bit, but MVS has no problems at all with files much larger than 2
gig. My recollection is that our old 24-bit 360/40 could also read
multi-volume tape files greater than 2-gig. Does anyone else know for
sure?
