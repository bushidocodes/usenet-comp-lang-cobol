I don't use Realia indexed files, I use BTRIEVE files in my productions
systems. I am including information from the current Realia 3.1 workbench
documentation.

Record Locking

The CA-Realia file system provides transactional File Sharing does file
locking, not record locking.  However, because the COBOL compiler's file
system is so inherently fast, record locking can be effectively accomplished
while offering end users access times that rival or surpass pure record
locking file systems.  The Workbench compiler offers four suggestions for
implementing record locking.  Of these the first two are somewhat more
elegant than the latter, however each has its place.  Only you as the
designer can determine which is best suited for your purpose.  The following
suggestions will help resolve potential conflict:

1. Embed an 'in use' flag in the record.  By including the 'in use' flag,
the record can be made available for browse purposes, but eliminates the
possibility of two users simultaneously changing the same record.
 Sample coding follows:

READ file NEXT RECORD
IF IN-USE-BYTE IS NOT = 'Y'
  MOVE 'Y' TO IN-USE-BYTE
  REWRITE record-name
ELSE
  MOVE 'RECORD IN USE FOR UPDATE' TO USER-ERROR-MESSAGE
END-IF
UNLOCK

Note:  Although the record may be marked 'IN USE', other users may access
the record for browsing.

2. Use a shared intermediate file which contains all the records currently
'in use' for update.  The procedure would be to check the intermediate file
first.  If the record is not in the intermediate file, write the file and
record number out to the intermediate file before releasing the intermediate
file.
3. Date and time stamp the record.  Then, before rewriting the record,
compare the date to when the record was accessed.
4. Perform a record to record comparison.  This method involves matching a
before image of the record to the current image of the record prior to
rewriting your changes.

Any of these methods provides simulated record locking under the COBOL
compiler, while still allowing you to use pure COBOL code.

1996, Computer Associates International, Inc.
