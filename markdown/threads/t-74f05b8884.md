```
The question is:  Do you care how that numeric field is stored?   If you're
doing I/O with that COMP field, you do.  If you are just storing it within
the program and/or doing arithmetic with it, then you don't.   COMP, COMP-3,
BINARY, PACKED-DECIMAL, etc. just describe how the machine sees the number.

Unless you have to read it or write it in that format, just look at the size
of the number and whether it is signed or not.  Then write your new code the
same way.
```

```
COMP-3 (PACKED-DECIMAL) is common in COBOL, but that is not an IEEE type.

COBOL COMP data (COMP or BINARY COBOL data are implementor defined) may come
in sizes of multiples of 2, 4 or 8 bytes, which would be non-IEEE.

COBOL COMP-1 and COMP-2 data, while usually IEEE single and double floats,
it ain't necessarily so. (Also implementor-defined).

COBOL numeric USAGE DISPLAY data I do not believe is an IEEE type, though it
might as well be.

COMP-X is not an IEEE type either. (Although you might call it a 'character'
type).

Also, although not an IEEE issue, COMP data is always stored with the MSB at
the lowest address, which is not what Intel-based processors and compilers
use.
```
