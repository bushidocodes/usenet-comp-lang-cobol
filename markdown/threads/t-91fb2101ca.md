This was "fixed" (enhanced?) in the '02 Standard.  Fujitsu conforms to the '85 
Standard where you were NOT allowed to use AFTER for inline Perform.  The 
"solution" in the '85 Standard was to NEST inline performs.  The following 
SHOULD compile (and do what you want )with Fujitsu

  if K is positive *> K has the number of table entries in it...
    perform
          varying totn-x1
              from 1
                by 1
             until totn-x1 > K
       Perform
         Varying  L
                 from totn-x1 + 1
                   by 1
                until L > K
                   if totn (L) < totn (totn-x1)
                   *> swap them (tournament sort...)
                      move totn (totn-x1) to temp
                      move totn (L) to totn (totn-x1)
                      move temp to totn (L)
                   end-if
      end-Perform
    end-perform
  End-If

the general "consensus" was that this was a STUPID restriction (and it was 
removed in the '02 Standard) but if you have a "strictly conforming" (for this 
syntax) '85 Standard compiler, then you need to use the "work-around".

Bill, I don't intend to pick on you, but despite what IBM says, if I saw
code where someone used a COBOL file SORT verb to sort an in memory table,
I would consider the programmer to be clueless. Even an operating system
call to do an in memory sort would be a lot more overhead than a Bubble
Sort of a table with dozens of entries. Operating systems calls are
usually several orders of magnitude slower than a few hundred inline CPU
instructions, and using SORT to sort an in memory table could require many
operating system calls (e.g. RELEASE/RETURN/SORT). Table sorts just aren't
that esoteric.

Quicksort is too data sensitive to be a good candidate for this purpose
in COBOL, but Shell Sort is almost as good, much more stable and simpler.
There is a COBOL Shell Sort routine on my website, if anyone needsa sort
better than Bubble Sort. But for a table with only a few items, I would
still use Bubble Sort or Insertion Sort, because they're simple and easy
to code and understand. Programmers are *supposed* to know how to do this
kind of stuff. :-)

P.S.- My COBOL Shell Sort routine was written for COBOL 74, which is
why there aren't inline PERFORMs, and I used GOTO for efficiency. Need
to update that. :-)
