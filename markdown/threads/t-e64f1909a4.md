Having been away for awhile, I just read several interesting threads and
hope to comment on all in one message.  My comments are from the teaching
world and the business users world.

In teaching I never allow GOTO to be used in COBOL code unless the
individual can prove there is not a more structured method that will work.
In over twenty years, this has never happened.  With the unconditional GOTO
branch, the code always goes to the same location.  That usually means that
the code should be the next physical statements coded, eliminating the need
to go somewhere else.  Many years ago when RAM was only a very few kilobytes
and cards were used for both source and object code storage, very cute
coding methods using GOTOs were often necessary to get larger programs to
work.  This has long passed.  In my opinion, the GOTO is dead in COBOL.

Flowcharts --  In spite that most LOGIC texts advocate the flowchart as a
PLANNING tool, I find in real life it is used as an after-the-fact
documentation tool.  Only if the "boss" insists a flowchart be included in
the file when a project is put to bed are flowcharts typically produced.
Then the flowcharts are quickly constructed from the now proven code.
Pseudocode is usually much more efficient for planning logic.  A quick
flowchart or a decision tree diagram is often useful for working out the
logic of a brief but complex set of (say) IFs, but flowcharts are too
tedious to be practical for most events or the entire program.  I really do
use decision trees a lot for scratching out branching options (especially in
RPG), but I can count the number of flowcharts I have done willingly in my
lifetime on one hand.

COBOL data files with C --  Packed numeric data fields are a pain when used
with other than COBOL and RPG, and I (and many other businesses I work with)
essentially no longer use packed numerics in disk storage.  By recording
data fields in DISPLAY format, the files can be used by many languages,
spreadsheets, and database programs - this makes the files much more useful
across the company.  Disk storage is very cheap - much cheaper than the time
programmers cost in working out all types of data conversions.  The
important base line is cost   -  disk storage is cheap  - programming costs
and bad conversions are not.

Again, just my opinions and practices, not necessarily shared by all.
David C.

I particularly agree with you regarding the use of packed fields.

However.

If we are going to talk about teaching, however, I have a real pet peeve,
one that is being shown in this very echo to be ubiquitous.

That is, students seem to think that "structure" has to do with specific
coding methods.  Typically, in Cobol, that if you do NOT use a "GO TO" ,
then somehow the program becomes structured, and if you DO use a "GO TO",
then some how it magically becomes unstructured.

I have written thousands upon thousands of lines of assembler that was
structured, and it was larded with go to's. The use of GO TO has to do with
the language, not with the absence or presence of structure.  The "go to"
rule is a simplistic way to get the idea across to beginners.  When the
entire idea of structure is simplified to that point, however, it starts to
get in the way of real understanding.

Top down structure is a simple concept.  You take a complex algorithm, and
you divide it into several simpler portions, each of which is as isolated
from the others as possible. That is repeated in an iterative fashion until
you have a tree of very simple algorithms rather than one huge complex
algorithm. You then code a number of very simple algorithms, and it is done.

By insisting that a student not use GO TO, one can *force* a programmer to
isolate the modules, or at least isolate the code portion.  Rules like "GO
TO allowed, but only to an EXIT statement" do the exact same thing in
another fashion.  Outlawing the "GO TO" is an enforcement method; it has
nothing to do with how structure works.

  Amen!
  The original pedantic concept of "structured programming" was that you
eliminated all GO TO's and forced everything into using just the basic
minimal structures of sequence, conditional, and loop.  Most people who
work with programming in the real world realize that although this
limited set of structures may be provably sufficient, that allowing
additional structure variants (case structures, pre-test loops,
post-test loops, mid-test loops, etc.) can in many cases improve clarity
and maintainability of a program.
  If your programming language of the moment does not have any direct
support for some structured construct that best fits the needs of your
algorithm, your best choice may be to implement that structure using a
GO TO statement.  I would argue that GO TO's should be allowed, but
their use should always be restricted to implementation of structure
constructs that lack direct language support in the source language.
  This is admittedly a more difficult concept to teach to neophyte
programmers, and I suspect to some teachers as well, than a dogmatic ban
on GO TO statements.

I agree that packed fields are not as important as they once were, but I'm
not certain that there is no longer a place for them.  If compilers still
generate the type of assembler instructions that they used to, all
arithmetic operations are performed on numbers only AFTER they've been
converted to either packed decimal or binary.  Even adding 2 display fields
and putting the result in a display field will cause the compiler to pack
both fields, add them, then unpack them.  Programs that do a relatively
small amount of computing might not benefit much from this, but I've always
been told that any number (with few exceptions) used in calculations should
be defined as either packed decimal or binary for performance reasons.  As
CPUs get more powerful, this becomes less and less of an issue.

When someone asserts that something cannot be coded without a GO TO,
it's generally because they haven't tried very hard.  I know of only 
a few cases where a GO TO is difficult or impossible to avoid:

1. A section invoked by a DECLARATIVE.

2. In older dialects of COBOL, an input or output section for a sort.

3. Older versions of the CICS preprocessor, which inserted GO TOs
for you and forced you to code accordingly (this was a peculiarity
of CICS and not the fault of the language).

In the case of your example it's not hard at all to eliminate the
GO TOs, even with your evident prohibition on flags:

 MLE-ENTRY.
     PERFORM UNTIL 1 = 0
         READ INPUT-FILE
             AT END 
                 NEXT SENTENCE
             NOT AT END
                 PERFORM RECORD-PROCESS
         END READ
     END PERFORM.
     CONTINUE.
 MLE-EXIT.

(I'm not sure if the weird "UNTIL 1 = 0" clause is legal; you might have
to devise some artificial always-true condition which the compiler would 
not object to.)

I left the MLE-EXIT label in, on the assumption that you invoke
MLE-ENTRY with a THRU MLE-EXIT clause.  If you eliminate the THRU
clause you can eliminate MLE-EXIT.

Another approach is to terminate the loop on the basis of a file
status variable.  However this approach doesn't yield quite the same
results if the READ is unsuccessful for reasons other than end-of-file.

Mind you, I wouldn't code this way myself, for several reasons:

1. The UNTIL condition is at least moderately barbaric.  One must
examine the body of the loop to determine what the termination
condition really is.

2. The NEXT SENTENCE is likely to be misunderstood as the equivalent
of a CONTINUE.

3. The use of NEXT SENTENCE requires that the END PERFORM be followed
by a period in order for the logic to work right.  This requirement
may be difficult to satisfy in other contexts.

My point is simply that the GO TOs are not necessary, contrary to
your assertions.  Whether they are desirable, or better than the
alternatives, is another matter.

> It can
> be transformed into an arrangement of other structures by introducing
…[3 more quoted lines elided]…
> make a COBOL program "more structured"?)

Certainly the standard structured constructs can be implemented by the 
use of GO TO.  I don't think anyone's disputing that fact.  The more
relevant question is: why implement them with GO TO when you can use
syntax designed for the purpose?

The problem with GO TO is that you can GO TO just about anywhere in
the program.  The compiler does not restrict you.  As programmer you
bear the burden of using them wisely, if you use them at all.

More importantly, whoever *reads* the program is burdened with figuring
out where all the GO TOs go to.  Until you analyze *every* GO TO and 
*every* PERFORM THRU clause, you can't safely make any non-trivial
changes, because you may disrupt the flow of control inadvertently.
I really do go through this kind of analysis when the occasion arises.
When there are dozens or hundreds of GO TOs this process is quite
time-consuming.  

In the absence of GO TO and PERFORM THRU, it's much easier to find
your way around an unfamiliar program.  You can safely rearrange the 
paragraphs, or add new ones, without disrupting any fall-through logic. 
You can concentrate on the paragraphs you need to know about, and ignore 
the rest.

> For those who insist on introducing a flag to eliminate the GO TOs,
> look up the word "pedantic".

In practice I would use a flag, without hesitation.  Flags can be
abused, especially by being set or tested in multiple locations.
However a flag that is set in one place, and tested in one or two
other places (especially in the same paragraph), is not likely to 
create any confusion.

Using GO TOs to avoid a simple flag is as pedantic as anything I can 
think of.

The requirements were imposed so that a particular structure

standard COBOL without using GO TO. This was within a larger

------------------ Beginning of quote
It has turned out to be less than I had hoped.

Dijkstra found that any program that is comprised of three
structures: 'sequence', 'decision', and 'loop until', can be proven
correct. This was the birth of structured programming.

Later, two additional structures were added: 'loop while' and
'case'. Both of these structures can be viewed as a composite
of the three original structures However, both can also be
discussed in their own right; that is, without having to discuss
their possible composite.

Two of these five structures are loops: 'loop until' and 'loop while'.
The first can [be] described as having a test for termination at the
bottom of the loop. The second as having the test for termination
at the top of the loop.

The obvious missing loop is one where the test for termination
is in the middle of the loop. However, this loop is not a composite,
as is 'loop while'; hence, the structure is irreducible. Irreducible
in the same sense that 'sequence', 'decision', and 'loop until' are
irreducible.

Here is an example illustrating some of the points.

MOVE ZERO TO RECORDS-PROCESSED
SET INPUT-ERROR TO FALSE
SET DONE TO FALSE
PERFORM UNITL DONE
    READ INPUT-FILE
        AT END
            EXIT PERFORM
    END-READ
    EVALUATE INPUT-TYPE
        WHEN "H"
        WHEN "T"
            EXIT PERFORM CYCLE
        WHEN "S"
            PERFORM SPECIAL-PROCESS
        WHEN "D"
            PERFORM DETAIL-PROCESS
        WHEN OTHER
            SET INPUT-ERROR TO TRUE
            EXIT PERFORM
    END-EVALUATE
    ADD 1 TO RECORDS-PROCSSED
END-PERFORM

The two occurrences of EXIT PERFORM will cause transfer
of control to an implicit CONTINUE after the END-PERFORM,
thus exiting the loop.

The occurrence of EXIT PERFORM CYCLE will cause transfer
of control to an implicit continue before the END-PERFORM,
thus causing the loop to repeat.

The flag INPUT-ERROR serves the purpose of indicating
that a later process is to be perform, such as displaying
an error message.

As shown above, the flag DONE prevents the termination of
the loop and would be unnecessary if a different construct
were available.

The following is a rewrite and may be considered typical.

MOVE ZERO TO RECORDS-PROCESSED
SET INPUT-ERROR TO FALSE
SET INPUT-EOF TO FALSE
PERFORM UNITL INPUT-EOF OR INPUT-ERROR
    READ INPUT-FILE
        AT END
            SET INPUT-EOF TO TRUE
    END-READ
    IF NOT INPUT-EOF
        EVALUATE INPUT-TYPE
            WHEN "H"
            WHEN "T"
                CONTINUE
            WHEN "S"
                PERFORM SPECIAL-PROCESS
                ADD 1 TO RECORDS-PROCSSED
            WHEN "D"
                PERFORM DETAIL-PROCESS
                ADD 1 TO RECORDS-PROCSSED
            WHEN OTHER
                SET INPUT-ERROR TO TRUE
        END-EVALUATE
    END-IF
END-PERFORM

(An alternative for the above is to place the EVALUATE within
a NOT AT END for the READ statement.)

The high-level process described is, simply, "for each record,
process it". This is clearly stated as: 1. read a record, 2. process
the record.

The test  NOT INPUT-EOF is considered objectionable because
its use obscures the process being described. It might be considered
less objectionable to place the evaluate within the NOT AT END;
but it the still obscures the process.

Since these examples are small, it may be difficult to see that the
process is obscured. We can see the difference by comparing
the two examples.

The first example places the READ and EVALUATE at the same
level, thus showing the correct relationship between the two steps
required by the process. The end of file condition is treated as an
exception to the normal process.

The second example and its alternative, make the processing
of the record subordinate to the exception not having occurred;
thus, the second step is described at the wrong level.

However, the second example clearly states the conditions for
terminating the loop be placing those conditions in the PERFORM.
The first example doesn't.

Finally, if one can accept that the presence of flags, in the second
example, obscures the process, then it should be apparent that the
use of flags to control execution in a highly nested structure will also
obscure the process, only more so.

Rick Smith
