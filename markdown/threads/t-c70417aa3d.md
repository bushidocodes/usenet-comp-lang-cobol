Let's make two assumptions:
1. You don't have a "modern" compiler which supports bit manipulation.
2. You are unwilling (or unable) to call a subroutine written in another
language.

Then you must do it the hard way (read that VERY hard).

Move the desired byte to a Pic X field which is the low order
byte of a redefined binary (comp) field.
To turn a bit on (assuming you know that it is already off),
simply add the appropriate power of two to the binary field.
Example to turn on X'04', add 4. For X'40' add 64. etc..
To turn a bit off, subtract the appropriate power of two amount.
Then move the redefined picture X byte to the original location.

Sounds to easy? It is, only if you can be absolutely sure of the
original state of the byte. Since this is rarely the case, you must
go through a harrowing set of comparisons to determine if the
desired bit is already on or off. There are more ways to do this
than I can imagine. You can do a b-tree type of high-low branching,
a computational range of GOTO DEPENDING ON's, etc. But, whatever
you do, it's going to be messy to handle every bit position for every
desired operation.

Here is a little program that I wrote once, when I needed to do this.
It could probably be optimized a bit (no pun intended....).

Some information about Acucobol-85: a type of "pic x(n) comp-x" is
considered an n-byte binary type. (You need to have binary types
in order to care about bits being set. You can replace the references
to those types with the types your compiler supports.)

This program also only works on two-byte values. But I'm sure you know
how to extend it....

Hope this helps. Feel free to e-mail me with any questions about this you
may have.

identification division.
program-id. testset.
author. Randy Zack, acucobol inc.
remarks. This program sets or tests a bit in a value.

data division.
working-storage section.

01 testset-save.
03 save-number pic xxxx comp-x.
88 save-set values 32768 thru 65535.
03 temp-number pic xxxx comp-x.
03 save-bit pic 99.

linkage section.
01 testset-info.
03 testset-number pic xx comp-x. | input
03 testset-bit pic 99. | input
03 testset-test-or-set pic x. | input
88 set-bit value 's'.
88 test-bit value 't'.
03 testset-result pic x. | input if set
88 bit-on value 't'. | output if test
88 bit-off value 'f'.

procedure division using testset-info.
main-proecedure.
if testset-bit > 15
exit program
end-if.
move testset-number to save-number.
move testset-bit to save-bit.
perform until save-bit = 0
subtract 1 from save-bit
multiply 2 by save-number
perform until save-number < 65536
subtract 65536 from save-number
end-perform
end-perform.
evaluate true
when set-bit
move 32768 to temp-number
perform until save-bit = testset-bit
add 1 to save-bit
divide temp-number by 2 giving temp-number
end-perform
if not save-set and bit-on
add temp-number to testset-number
end-if
if save-set and bit-off
subtract temp-number from testset-number
end-if
when test-bit
if save-set
set bit-on to true
else
set bit-off to true
end-if
end-evaluate.
exit program.

We would usually avoid the problem by defining a copy member that

Well using the Unisys A Series COBOL85 compiler it is done this way:

MOVE val TO word-val [0 : 0 : 1].

The values in square brakets are 1st value, starting bit, 2nd value, destination
bit, 3rd value, how many bits. The A Series numbers bits from 0 to 47. This move
statement must use an A Series word as the destination (pic 9(11) binary).
