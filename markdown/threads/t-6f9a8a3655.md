There are several ways:
1. Get a book on Windows APIs and figure out how to invoke them from COBOL
(all 3000 of them).
2. Use VB for the screens and link your COBOL and VB together.
3. Get a COBOL compiler that is Windows aware and can communicate directly
with the operating system (Fujitsu V4+ for example).
4. Get an add-on product such as SP2 to do the windowing work for you.

My advice, forget about making your program stylistically swell. There is no
end of people who can make an application look pretty and too few that can
make the application actually work.

I did just that a year or so ago...I found it unusable for true windows
development...(At least for the type of development that I do)...Mostly
because the developer did not have a way to access the standard windows
message loop using ScreenIO...There are many standard windows development
functions that require access to the message loop...Timers,
sending/receiving messages to/from other programs, Async I-O to com ports,
sockets, pipes, etc...etc...It is not possible to use these functions in
ScreenIO programs, because ScreenIO hides the message loop from the
developer...

Here is a direct quote from one of the folks at ScreenIO -----   "We believe
that GUI ScreenIO does an excellent job of providing functionality while
presenting a simple and easy-to-use programming interface.  If you need to
inspect the Windows message queue and respond to low-level Windows events,
use pipes or deal with tcp sockets at a low level, GUI ScreenIO is probably
not for you."   ...Personally, to date I have not developed a true windows
application that has not had a need to access the windows or OS/2 message
loop/queue...Windows applications have to typically do more than present
screens to the user...If someone needs to develop an application that simply
displays screens to the user, then go for it...ScreenIO might very well do
the job for that person...

Another problem with the product was that it did not support auto-scaling of
windows and controls...If a window is sized by the user, the associated
controls and fonts used within the window should also be scaled and changed
to match the new window size. Hopefully this feature has been added to their
product...

I personally did not like the approach taken for say listbox
controls/combobox controls...Having to pre-define the number of possible
entries...Nor did I like being forced to use a working storage table to hold
the entries in memory...This is redundant and consumes valuable machine
resources...i.e. memory...Windows itself has to keep memory allocated for
entries in a listbox control...Using ScreenIO, the memory requirements for a
listbox are now double what would be required using the standard APIs...I
believe also, that there are maximum size limits for the data in a listbox
or combobox with ScreenIO...When I looked at the product the limit was a max
of 64K worth of data in a listbox...I remember also being presented with 3
or 4 option dialogs to define a listbox that had all options choices
disabled...Why present a developer with a screen to select an option when
all options are disabled? Again, this may have been fixed, but I found it a
waste of my time as a developer to deal with dialogs that should have never
be displayed...

I agree that there is a long learning curve for API development, but once
you have learned to create a window or a button or a listbox, the next time
is simple cut-and-paste...Calling an API to do something is no different
than calling some cobol routine to do something...I have been using API's
for development for many years, and still have not learned them all...Using
native API's to develop true windows applications is very do-able though,
and should not be over looked as a choice for development...

API calls can be encapsulated into OO objects...OR...They can be contained
within subroutines or DLLs...Personally I use a "Common Routines" module for
most API functions...For example to enable a window I use

Call WinAPI crEnableWindow
   using by value hWndWindow.

or

Invoke crEnableWindow
   WindowHandle is hWndWindow.

The code for entry point 'crEnableWindow' is a call or perhaps many calls to
the operating system API(s). I have a common routines module for Windows and
one for OS/2...The actual user app code does not change...I also reduce the
complication of dealing with the API(s) by having routines in the common
routines module that performs one, several or even many actual API
calls...For example opening and initializing a COM port...This might require
5 different API calls...But from my app, I simplify that to a single call :

Call WinAPI crComOpen
   using by reference z'COM1'
         by reference z'19200,N,8,1'
returning wsPortHandle.

Many call(s) get made to the actual operating system APIs, but my app is
coded using a very simple call to the crComOpen entry point. Once these
common routines are developed (and it will take some time to develop) the UI
or interface to OS functions like reading and writing to com ports, etc is
greatly simplified.

Window procedures and callbacks are nothing more than entry points that
Windows calls with associated parameters passed...These can be encapsulated
into separate modules as well...Recursion and messaging is a fact of life in
a windows environment...An understanding of how these things work is a
requirement that cannot be avoided no matter how you handle the user
interface...

Someone who is inquiring about what direction to take in their future
development should explore ALL the various options that are available to
them before they make a decision...API programming is one of those options
available and it works very well...Developers should not be intimidated by
vendors of UI products telling them how complicated API programming is, when
in reality, it is not really that complicated to provide a GUI interface,
ONCE the learning curve is completed and messaging, recursion and multiple
entry points is understood...Programming in windows or OS/2 is going to
require a learning curve regardless of the UI method employed...Be it API
calls directly, OO invokes and property settings, Activex, or ScreenIO or
SP2...There is still a learning curve to be dealt with...The original poster
will need to decide which method he/she wants to use...His decision needs to
be based on fact finding and asking questions and getting responses...

What is a programmer going to do when he encounters something that cannot be
done by a 3rd party UI product? He is going to need to understand how
windows or OS/2 works and that there are APIs to do what the 3rd party
product(s) cannot do...Does ScreenIO deal with reading and writing com
ports, semaphores, mutexes, pipes, TCP sockets, file sharing, networking,
shared memory between modules, threads, displaying graphics, automatic
scaling of windows and dialogs for different screen resolutions or maximized
vs a normal window, owner drawn controls, compression and expansion of files
or data, etc? These are common windows or OS/2 functions that will
eventually be needed in a lot of business applications...May as well get
started with the learning curve now...

BTW, I am not opposed to using 3rd party products by any means...I use the
NE dialog editor, ListPro, InputPro, TabPro, SocketTools, the Victor Graphic
libraries, etc on a daily basis to make my life easier...The nice thing is
all these 3rd party products work just like calling the normal windows
operating system APIs and they conform to the normal windows programming
methods (message loops, recursion, sync or async processing)...I am not
restricted because my windows program is based entirely on a specific 3rd
party UI product...There is NO single 3rd party product that can accomplish
what all of the various tools and operating system API's can accomplish...

I will be happy to send anyone interested, a copy of my evaluation of
ScreenIO which includes their responses to the problems I found with the
product...The product has most likely changed since my evaluation, but my
initial impressions and experiences (including several abends) were bad
enough to keep me from looking at it again...

Personally, I do not think that product vendors should be posting comments
like

<<< That's a fairly significant undertaking do it the level you seem to be
suggesting. There is a huge investment in infrastructure just to get to the
place you can put a button out.">>>

This is nothing more than an attempt to make programming via API's sound too
difficult to do and to sell their product...You can "put a button out" in a
single simple call to the windows API...Folks from this same company have in
the past also made commits such as "you would have to be crazy to use native
windows API's"...If they want to sell and support their product here,
great...I'm all for that...I'm sure there are potential customers here...But
they should not be making absurd commits in an attempt to intimidate someone
away from alternative ways of development, simply to push their own 3rd
party product...

Of course, this is simply my opinion...You and others are certainly free to
voice your opinions as well...Maybe together, we can answer the original
posters question...

As a Windows developer since Win 3.1, I read this with great interest.

Ken's sincerity cannot be doubted and he makes some very good points about
the use of Win API. (And the smokescreens that various interested parties
may put up to persuade you NOT to use them...<G>)

I would agree that once you code a few API calls the process loses much of
its mystique and terror.

Ken is absolutely right in his examples of simple window and control
building and it IS simple. (But so is using a third party tool to do it...)

However, there ARE pitfalls in the use of  more complex APIs which are
tiresome in the extreme; the main ones I have  experienced are with getting
data in the right format from COBOL, to be recognized by the API. There are
no diagnostics or clues, it simply doesn't work if you get it wrong. At
worst it can hang your system. I have spent many hours of trial and error
before getting some calls to work properly and I have been glad of
assistance from programmers who are more experienced in this particular area
to get certain complex calls to work. Once it works, of course, you forget
all about the sweat...

Now, if a third party tool can shield you from this, then, personally, I
would be in favour of using that tool (and I do...). However, in most cases,
as Ken points out, the third party tools don't adddress these areas and you
HAVE to use the API.

For quick building of Windows applications I use Fujitsu PowerCOBOL. It is
OO based, easy to learn, and gets you to a GUI in no time flat. Yes, I could
develop a library of Objects as Ken suggests, to provide the GUI controls,
but Fujitsu have already done it and it works fine. None of the specific
shortcomings he mentions with combo boxes, resizing, etc., are manifested by
the PowerCOBOL objects.(I was shocked at the idea of having to predefine the
number of items to be displayed by a ListBox, Ken <G>...)

On the contrary, the controls are packaged complete with Events, Methods,
and Properties which make using them a doddle, and provide in many cases
enhanced functionality beyond what the raw API call would provide. (No, I am
NOT on commission for Fujitsu...in fact, I won't be using their products in
future (reserving judgement on .Net COBOL until we get to see it), but that
is more about support, licensing, and responsiveness than inherent failures
in the product.)

I think one point that was missed in Ken's post is that using a third party
tool or using the Win API are NOT mutually exclusive. I have Windows
applications which utilise both and they work very well.

BOTTOM LINE:

1. What tools you will need depends very much on what you want to do.

The kind of development Ken is doing demands much more than simply
interacting through a window. (I wouldn't mind betting that your
applications are pretty to watch, Ken...<G>)

2. There is an argument that says investing a large amount of time in
learning the details of a low level interface may be unjustifiable, given
that technology moves at the rate it does. Anyone who has been in this game
more than 15 years can probably tell you about knowledge that was painfully
gained, then rendered redundant within a few years by the march of
technology. (Fortunately, sometimes it comes back...Look at CICS <G>).

This is particularly pertinent when the knowledge applies to a specific
product... (Do you start over if you move to UNIX, Ken...? Nope, just don't
move, right?)

3. It is irresponsible for Vendors to promote their products with sweeping
statements that are not balanced or fair. (But we will NEVER stop them doing
it <G>; all we can do is exercise our judgement and measure their claims
against experience. Of course, the guys who couch their pitch in simple
plain examples and statements of fact, will acquire much more credibility
than the guys who don't...<G> )

Pete.

For most 3rd party tools (at least the ones I am familiar with), this
statement is 100% correct...But in the case of ScreenIO, the approach taken
actually hides the windows message loop from the developer...You could not
get to certain windows messages and notifications...(Unless this has changed
in the last year)...

To me, this is a very serious flaw in their product design...They have
attempted to over simplify true windows development...Without access to the
windows message loop, there are indeed many APIs and even standard windows
functions that could never be used in a ScreenIO application...As a result,
using ScreenIO and "some" native APIs IS indeed mutually exclusive...

A really simple example...

When I looked at ScreenIO, you could not even use a simple windows timer
API...Lets say you wanted to display the clock in the corner of your GUI
screen...The only way to do that was to code a cpu loop...Not something I
would want to do...Windows provides simple APIs to initiate timers...You
could set a timer to expire every second and update your screen clock...But
without access to the windows message queue, you would never know the one
second timer expired...The API works by notifying you that it is time to
update your clock via a simple message sent to you via the application's
message loop...If you can't access the message loop to process the message,
you can't use the API function...

This was just one of many examples of APIs and/or standard windows functions
that could never be used within a ScreenIO application...Windows is a
message and event based operating system...That will never change...If you
do not have access to the application's message loop, you are restricted to
using only the functions and event notifications that ScreenIO passes to the
application...

Windows applications usually are much more than just a pretty user
interface...No 3rd party user interface product can provide all of the
functionality of the windows APIs...

Again, my comments are based on an evaluation of the ScreenIO product one
year ago...Maybe things have changed since then...
