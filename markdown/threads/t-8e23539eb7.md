```
Hi folks,
    on a UNIX/Linux and MF-Cobol4.1 I find the following situation:

I am calling cobol-programs out of an extended rts using the c-function
"cobcall".
After returning from cobcall, the program is logically cancelled by
"cobcancel".
Works fine, no problem.
For technical reasons I have to link certain programs to another name, e.g.
ln -s CVFILE01.int CVFILE01R.int

Problem:
1:   Program CVFILE01.CBL containing a PROGRAM-ID: CVFILE01.
If I _do_not_ "cobcancel" CVFILE01 and the next call is to to the linked
CVFILE01R, I get the error
119 Name is not unique (Symbol 'CVFILE01')

2:   Program CVFILE01.CBL _does_not_ contain a PROGRAM-ID: CVFILE01.
If I _do_not_ "cobcancel" CVFILE01 and the next call is to to the linked
CVFILE01R, it works OK.

Apparently it's ok to skip the PROGRAM-ID, I just wonder if there are
side-effects I possibly have to consider.

Thanks for reading and comments
Wolf
```

```
I've got an idea that MF Cobol (for historical reasons) only stores up to 8
characters internally as program name.  I don't know if that can be
increased.

So although it recognises the file name as different when loading, it uses
the first 8 characters to check if  something of the same name has already
been loaded.
```
