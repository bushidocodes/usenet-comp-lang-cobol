When you are having trouble with a program, you need to learn to patiently
desk-check it.
That means going through a couple of passes of the program line-by-line,
asking yourself "what does this do" as you analyze each statement.
Write down each field content change as it occurs.  Analyze that closely.

Bugs will rear their ugly little heads much faster if you adopt this
training/testing aid.

Hello again

In addition to the changes proposed by Richard and Topcoder, you also
need to have separate 01 levels in the FD for the EMP-FILE to separate
the header from the details as follows:

01  EMP-HEADER.
    05  COMPANY-NAME            PIC X(24).

01  EMP-RECORD.
    05  EMP-CODE.
    etc as before

These two 01 levels being subordinate to the same FD statement are
implicit redefinitions, so you do not have to use the REDEFINES clause
to get the desired effect.  You subsequently need to ensure that the
first READ of EMP-FILE is done outside of the table building
procedure, e.g. in 200-START-UP or 300-INIT-ITEMS, to save the
company-name somewhere else.

Your PERFORM VARYING in 210-PROCESS-DETAIL-RECORD should also test for
end of file so should take the following form

PERFORM VARYING EMP-TABLE-ROW-CTR FROM 1 BY 1
          UNTIL END-FILE = "YES"
             OR EMP-TABLE-ROW-CTR>10
 etc as before
END-PERFORM.

This latter, while not strictly necessary in this case, is generally
good practice as it ensures that you catch the cases when someone
tries to use your program with some missing data.

Making progress, I hope

Robert
