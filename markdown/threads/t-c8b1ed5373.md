an Isam file for Fujitsu Cobol?

Example:

MF -
SELECT DTRK-FILE ASSIGN TO DISK
ORGANIZATION IS INDEXED
ACCESS IS DYNAMIC
FILE STATUS IS DTRK-STATUS
RECORD KEY IS DTRK-KEY
ALTERNATE KEY IS DTRK-INTO-THE-YARD =
DTRK-SHOW, DTRK-TARE-DATE
DTRK-IN, DTRK-KEY
ALTERNATE RECORD KEY IS DTRK-DISPATCH-KEY
WITH DUPLICATES.

alternate key is field1 field2 field3 etc., the compile accepts

that leaves me with no symbolic name for the concatenated key.

I cannot find anything in the docs about split keys.

Thanks, Bill. Thanks to Thane, I was able to get a working
methodology (outlined below), then I went back and tried the
standard (as outlined above), and could not get it to work. The
above crapped out on the fact that the record key was not
defined. I then tried to define it in working storage as a simple
byte record, equal in size to the concatenated source fields. That
produced the error message that it had to be a data name in the
FD. Since the data in the record is non-contiguous, there
appears to be no way to do it. (Though I just thought of defining it
as an 01 record, maybe I will try that later).

However, as I said above, Thane searched out some docs that
pointed in the right direction, and I tested it this morning. It appears
to work as follows.

In the SELECT clause, list the fields like so:

SELECT DTRK-FILE ASSIGN TO DTRK-IDENT
ORGANIZATION IS INDEXED
ACCESS IS DYNAMIC
FILE STATUS IS DTRK-STATUS
RECORD KEY IS DTRK-KEY
ALTERNATE KEY IS
DTRK-SHOW, DTRK-TARE-DATE
DTRK-IN, DTRK-KEY
ALTERNATE RECORD KEY IS DTRK-DISPATCH-KEY
WITH DUPLICATES.

Then in the start, or read, list the keys in the same manner,
like so:

START DTRK-FILE KEY
GREATER THAN
DTRK-SHOW, DTRK-TARE-DATE
DTRK-IN, DTRK-KEY
INVALID KEY
MOVE "N" TO READ-WRITE-FLAG.

Not very elegant, but it does appear to work.

The really correct way to convert split keys
is to replace split key with a list of its components
everywhere where its encountered.

All Fujitsu statements that might potentially take
a split key (READ and START) take the list of keys,
not just one key. I think this is a standard extension
by Fujitsu specifically designed to handle "unsplit" split
keys.

Example:

You have C-KEY = KEY-1 KEY-2 in your file definition
READ ... KEY IS C-KEY

Replace "C-KEY = KEY-1 KEY-2" with "KEY-1 KEY-2" in the file
definition
and replace "C-KEY" with "KEY-1 KEY-2" in the READ
statement.

By the way, mf2fsc can do this work for you automatically.

You can download evaluation copy from
http://www.siber.com/sct/tools/mf2fsc/

Vadim Maslov
