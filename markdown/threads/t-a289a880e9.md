I have also used multiple conditions specified in multiple WHEN clauses
(each of which can be a compound condition, as Bill noted, above), but I
can't remember whether this was valid on the mainframe platform (try it and
see...).

Like this...

set my-table-index to 1
SEARCH  my-table
     AT END
           set not-found to TRUE
     WHEN
           some-condition OR some-other-condition
           set found-it to TRUE
      WHEN
           some-lower-priority-condition OR
some-condition-indirectly-related
           set found-it to TRUE
END-SEARCH
if found-it
   perform process-as-found
else
   perform process-as-not-found
end-if
...

AT END will only be taken if ALL of the conditions in ALL of the WHEN
clauses, after ALL the increments of the table index, are found NOT to be
true. In this case the table index is technically undefined, but in
practice, usually points to the last entry + 1.

 If ONE condition in ANY WHEN clause is found to be true for ANY iteration,
then the Search stops and the action in the WHEN Clause is taken (Set
found-it to TRUE  in the above, but it could be anything you like.) In this
case the table index will be pointing at the entry which was being processed
when the condition was found to be true. Note that, while it is usual for
conditions to pertain to the data in the table, they don't HAVE to... I'm
not going into this here as it is a bit esoteric but take my word for it,
SEARCH can be used to do some VERY sexy things...(WARNING: One man's "sexy"
is another man's (read "maintenance programmer") nightmare...<G>)

As you can see, SEARCH is a very powerful and useful COBOL feature and
avoids the need to increment and check indexes (it is done for you "under
the covers"). A spin off is that f you change the definition of the table by
adding more entries to it you DON'T have to change the SEARCH verb; if you
searched the table manually with your own increment and test (say,
PERFORM... VARYING...) you would need to amend the boundary condition if
more entries were added to the table. (This was a frequent cause of error in
the days before SEARCH was available, and I for one, was very glad when we
could use SEARCH instead.)
