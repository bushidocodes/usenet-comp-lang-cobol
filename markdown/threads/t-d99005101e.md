Using the COBOL SEARCH ALL syntax requires that the table be ordered by the
search key field in ascending order, no duplicates, and the table has to be
completely filled or defined as a variable length with a pointer to the number
of entries in the table.   These are the only criteria I know of for
successfully getting a binary search using the standard COBOL syntax without
having to write your own routine.
As I recall, doing your own routine still involves the above criteria and doing
checks against a LOW, MID, and HIGH point for spltting the table further.
Usually getting an answer within 7 loops.

Not everyone wanted to write a binary search routine so the COBOL
committee made a compromise.  They created SEARCH ALL, and declared it to
have all the necessary restrictions so that it could be a binary search so
that an implementor had a way to offer the efficiency of a binary search.
But they also allowed for an implementor not to implement a binary search.
This way a program which used a binary search could be recompiled on a
system without a binary search and still run correctly.

Pragmatically, the A Series Cobol74 compiler used a sequenctial search for
many years.  Someone had once written a routine for a binary search, but
didn't realize he couldn't use the normal condition parsing routine.  This
never worked.  Rather than fix that routine immediately, the routine was
bypassed by just ignoring the ALL after SEARCH.  This meant we were
allowing any condition in the WHEN clause and allowing multiple WHEN
clauses.  So then we put in deimplementation warning for all those invalid
SEARCH ALLs, and then a couple years after that a binary SEARCH ALL was
implemented.

A sorted table has the same structure as a simple table, with the addition
of some extra indices. These are used when searching and changing the table.
We use a flag (item-found) to improve the search algorithm.

01  VARIOUS-INDICES.
    02  ITEM-NBR                PIC S9(3)  COMP.
    02  COPY-PTR                PIC S9(3)  COMP.
    02  LOWER-LIMIT             PIC S9(3)  COMP.
    02  UPPER-LIMIT             PIC S9(3)  COMP.
    02  MOVE-DIRECTION          PIC S9(3)  COMP.

01  VARIOUS-VALUES.
    02  ITEM-FOUND              PIC X.
        88  ITEM-WAS-FOUND                 VALUE IS "Y".

01  SORTED-TABLE.
    02  SORTED-SIZE             PIC S9(3)  VALUE ZERO COMP.
    02  SORTED-MAX              PIC S9(3)  VALUE +100 COMP.
    02  SORTED-ITEM                        OCCURS 100 TIMES.
        03  SORTED-ITEM-KEY     PIC X(...).
        03  SORTED-ITEM-DATA    PIC X(...).

The Binary Search Procedure

Before anything else, you need a way to search the table. To add an item,
for example, you must first find the place where it should go. You could use
an item-by-item scan, as above for a simple table. Alternatively, you can
take advantage of the fact that the table is sorted, and use a binary search
technique, which is much faster for more than about ten items. For a table
of ten thousand items, we would need on average five thousand comparisons
using a simple scan, but at most 14 comparisons using a binary search.

The binary search is an interesting technique; easy to explain, but much
harder to write correctly because of the variety of end-cases to handle.

Take the entire table, and look at the middle item. If higher than the test
item, look only at the lower part of the table; if less, look only at the
top part of the table; if equal, you have found the item. Repeat this
process, chopping the area of interest into smaller and smaller pieces

Give this problem to any twenty people who say they can write a correct
program, and nineteen will get it wrong the first time. They will forget to
consider what happens if the table is empty, or if the item is not found.
The following code does it correctly:

SCAN-SORTED-TABLE-FOR-ITEM.
    MOVE    "NO"     TO ITEM-FOUND
    MOVE      1      TO LOWER-LIMIT
    MOVE SORTED-SIZE TO UPPER-LIMIT

    PERFORM BINARY-SEARCH-OF-TABLE
      UNTIL ITEM-WAS-FOUND
         OR LOWER-LIMIT > UPPER-LIMIT
    .

BINARY-SEARCH-OF-TABLE.
    COMPUTE ITEM-NBR = (LOWER-LIMIT + UPPER-LIMIT) / 2
    IF SORTED-ITEM-KEY (ITEM-NBR) > ITEM-KEY
        COMPUTE UPPER-LIMIT = ITEM-NBR - 1
    ELSE
    IF SORTED-ITEM-KEY (ITEM-NBR) < ITEM-KEY
        COMPUTE LOWER-LIMIT = ITEM-NBR + 1
    ELSE
        MOVE "YES" TO ITEM-FOUND
    .

After the binary search, item-found is "Y" or "N". If "Y", item-nbr points
to the found item. If "N", lower-limit points to the next higher item. This
is also the place where you insert a new item. The compute statement used in
binary-search-of-table is costly on some systems, and you could use an
alternative:

    ADD LOWER-LIMIT, UPPER-LIMIT GIVING ITEM-NBR
    DIVIDE ITEM-NBR BY 2


On the other hand, do not do this except as an optimization technique for
true inner loops - it is a compromise which sacrifices the readability of
the program for efficiency.

If the table is organized as a heap even the division can be dispensed with
and replaced by an addition. We have here another example of the critical
nature of the right choice of data structure. As far as the primitive
operations you might want to perform on the table (search, add, remove, ...)
its internal structure is really not of interest.

The above code makes up to two comparisons each time through the loop. A
significantly faster method that only makes one comparison is:

SCAN-SORTED-TABLE-FOR-ITEM.
    MOVE "NO" TO ITEM-FOUND
    MOVE ZERO TO LOWER-LIMIT
    COMPUTE UPPER-LIMIT = SORTED-SIZE + 1

    PERFORM BINARY-SEARCH-OF-TABLE
      UNTIL UPPER-LIMIT - LOWER-LIMIT = 1

    IF (UPPER-LIMIT NOT > SORTED-SIZE
    AND SORTED-ITEM-KEY (UPPER-LIMIT) = ITEM-KEY)
        MOVE UPPER-LIMIT TO ITEM-NBR
        MOVE    "YES"    TO ITEM-FOUND
    .

BINARY-SEARCH-OF-TABLE.
    COMPUTE ITEM-NBR = (LOWER-LIMIT + UPPER-LIMIT) / 2
    IF SORTED-ITEM-KEY (ITEM-NBR) < ITEM-KEY
        MOVE ITEM-NBR TO LOWER-LIMIT
    ELSE
        MOVE ITEM-NBR TO UPPER-LIMIT
   .

With this code, upper-limit points to the next higher item, which is also
more natural than having lower-limit pointing to it. The code has a careful
test after the loop, but makes only one comparison per prope, has only one
loop termination test, and does not increment/decrement inside the loop. If
the key is of some length, the improved code is about twice as fast.

For additional speed, the division by two can be replaced by a table-lookup
using the sum of the two limits to index into a table containing half their
value. This table should then be initialized at the beginning of the
program.
