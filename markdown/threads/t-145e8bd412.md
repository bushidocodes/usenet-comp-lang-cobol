```
I have a program that needs to print two types of notices that have some
of the same qualities, but two different messages. My intention was to make
two tables as follows:

(BTW, I'm using IBM VS COBOL II.)

01 TYPE-1-MESSAGE-TABLE.
05 PIC X(79) VALUE 'TYPE 1 LINE 1'.
05 PIC X(79) VALUE 'TYPE 1 LINE 2'.
(...etc...)

01 TYPE-2-MESSAGE-TABLE.
05 PIC X(79) VALUE 'TYPE 2 LINE 1'.
05 PIC X(79) VALUE 'TYPE 2 LINE 2'.
(...etc...)

I would then have a linkage section as follows:
LINKAGE SECTION.
01 MESSAGE-TABLE.
05 MT-LINE OCCURS 20 TIMES
INDEXED BY MT-IDX
PIC X(79).

In the procedure division I would code the following:

IF THE-TYPE = 1
SET ADDRESS OF MESSAGE-TABLE TO ADDRESS OF TYPE-1-MESSAGE-TABLE
ELSE
SET ADDRESS OF MESSAGE-TABLE TO ADDRESS OF TYPE-2-MESSAGE-TABLE
END-IF

and from there I could reference MESSAGE-TABLE (MT-IDX).

Well lo and behold, it's not allowed! Apparently you can only use the
"ADDRESS OF" function to reference fields within the LINKAGE SECTION, but
not within WORKING-STORAGE. What on earth is the justification for this?
C lets you do it all the time.

In any case I simply ended putting MESSAGE-TABLE into WORKING-STORAGE and
moving the value of one or the other of the other tables into it (depending
on the value of THE-TYPE), but this seems to me to be an absolute waste of
memory and computing power.

Don't get me wrong, I like COBOL a lot, but once in a while it really ticks me
off...

Frank
```

```
Please bear in mind that pointers are not (yet) part of the CoBOL
language, so don't blame CoBOL for this.

The vendor extension that you are using was designed many years ago, and
it works this way simply because it was cheaper to write a way for the
programer to access the existing LINKAGE SECTION mechanisms than it was
to write new, more general mechanisms.

The relatively recent IBM AIX compiler, which started out with a clean
sheet and which runs on simpler hardware, will take the ADDRESS OF a
WORKING-STORAGE item. However, even the latest CoBOL for MVS and VM,
which did not start out with a clean sheet, still uses the old LINKAGE
SECTION mechanisms.

The key here is that pointers are rarely useful in CoBOL, except when you
are interfacing with something else. If you think you need a pointer in
CoBOL, you are probably trying to translate a C idiom literally, when
what you should be doing is using the CoBOL idiom that does the same
thing.

There is a close relationship between subscripts and pointers.
Specifically, x[i] is C shorthand for *(x+i). Working backwards, if you
are thinking in terms of a C pointer, you can often (and in this case you
can definitely) use a CoBOL subscript, index, or reference modification
instead.
```

```
There was a thread about something like this a couple of months ago. I
think one possible solution is to create a nested subprogram within your
main program. It would have two parameters when calling it, a pointer
field (to return the pointer) and a generic data field. You need only
call it twice during your program initialization.

CALL 'GETADDR' USING MY-POINTER, TYPE-1-MESSAGE-TABLE
MOVE MY-POINTER TO TYPE-1-TABLE-POINTER.
CALL 'GETADDR' USING MY-POINTER, TYPE-2-MESSAGE-TABLE.
MOVE MY-POINTER TO TYPE-2-TABLE-POINTER.

Of course, you would have to define two pointers somewhere in your main
program working-storage to support this.

Then you could say:

IF THE-TYPE = 1
SET ADDRESS OF MESSAGE-TABLE TO TYPE-1-TABLE-POINTER
ELSE
SET ADDRESS OF MESSAGE-TABLE TO TYPE-2-TABLE-POINTER
END-IF

I haven't actually tried anything like this (other than nested
subprograms), but it should work. It would be nice if IBM allowed you
to take a pointer of a working-storage item directly, but I believe you
can use a subprogram to trick the compiler this way.

On the other hand, wouldn't it be possible to build the messages into a
two-dimensional table? The table initialization would be a little
tricky, but there would be no need for Linkage section or pointers
then. You could just say:

MOVE MT-LINE (THE-TYPE, MT-IDX) TO MESSAGE.
```

```
That's because you're "thinking C" Frank, not "thinking COBOL". If a pointer
to either of the two tables would suffice, then the tables must be identical
in structure. So, instead of making two tables, make one table with an extra
layer of subscript:

01 MESSAGE-TABLE.
03 TYPE-1-MESSAGES.
05 PIC X(79) VALUE 'TYPE 1 LINE 1'.
05 PIC X(79) VALUE 'TYPE 1 LINE 2'.
(...etc...)
03 TYPE-2-MESSAGES.
05 PIC X(79) VALUE 'TYPE 2 LINE 1'.
05 PIC X(79) VALUE 'TYPE 2 LINE 2'.
(...etc...)
01 MESSAGE-TABLE-R REDEFINES MESSAGE-TABLE.
03 MT-TYPE OCCURS 2 TIMES
INDEXED BY MT-TYPEX.
05 MT-LINE OCCURS 20 TIMES
INDEXED BY MT-IDX
PIC X(79).

Then just set MT-TYPEX to 1 or 2, and use MT-LINE(MT-TYPEX, MT-IDX) to
reference the messages. Straight COBOL, straight simple, no sweat, no
runaway pointers. If MT-TYPEX is not initialized properly, you get an abort,
and you know exactly what the problem is, though it may take you a bit to
find out why the pointer isn't initialized.

In my opinion, adding pointers to COBOL is a BIG mistake. One of COBOL's
strengths is "safety through simplicity". Adding pointers just opens COBOL
up to the same kind of hair-pulling problems for which C is infamous.
```
