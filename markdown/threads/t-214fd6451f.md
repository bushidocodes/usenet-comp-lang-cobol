It will depend on your operating system environment. In order for a program
to be re-entrant, it cannot modify any of its own storage. If your COBOL
program is a re-entrant subprogram, it is safe to modify data in the linkage
section, because that storage is owned by the calling program, and presumably
it is isolated to one task. Your COBOL subprogram, however, would need to
dynically allocate and initialize its own working-storage section in order
for it to be re-entrant, because multiple tasks could be executing a single
copy of the program. Each task has to have its own isolated copy of
working-storage.

The compiler has to generate different code to remain re-entrant.

Probably what has to happen is at initialization the re-entrant program must
do one or more GETMAIN's or MALLOC's to acquire space equal to the total size
of its working-storage section, and then initialize it with all the VALUE
clauses you've defined. Then the working-storage section is addressed by
reference for the life of the task.

Any linkage section areas cannot be initialized, because they belong to the
calling program. But the re-entrant program's entire working-storage section
must be initialized at run time.

If your compiler has an option to LIST the generated machine code, try
compiling with and without the RENT option. You should see some obvious
differences in the generated code. I suspect there will be a huge amount
of code to initialize working-storage when the RENT program starts up,
while the NORENT program just loads the pre-initialized working-storage
from the disk along with the executable instructions.
