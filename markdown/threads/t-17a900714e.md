The VALUE clause must be on the first 01, and you cannot redefine an
item containing an OCCURS clause. :-)  I prefer this form:

01  MONTH-TABLE        VALUE
     'JAN01FEB02MAR03APR04MAY05JUN06JUL07AUG08SEP09OCT10NOV11DEC12'.
    03  MONTH-ENTRY    OCCURS 12 TIMES.
        05  MONTH-NAME    PIC X(3).
        05  MONTH-NUM     PIC 9(02).

But I do not use an 01 for tables like yours, for a very good reason
which you may agree with.  After correcting your example:

01  MONTH-VALUES      PIC X(60) VALUE
     'JAN01FEB02MAR03APR04MAY05JUN06JUL07AUG08SEP09OCT10NOV11DEC12'.
01  MONTH-TABLE       REDEFINES MONTH-VALUES.
    03  MONTH-ENTRY   OCCURS 12.
        05  MONTH-NAME    PIC X(3).
        05  MONTH-NUM     PIC 9(02).

Please note that if you make a mistake and the two 01 items are not
declared the same size, the compiler won't complain, because it is
legal to do so at the 01 level.  I never do it myself, but have been
zapped by this in other people's code.  But if you declare it at a
subordinate level as in the following, the compiler should flag it
if the redefining item is larger than the redefined item.  I would
prefer that the compiler flag it any time the size is different.

01  MONTH-TABLE.
    03  MONTH-VALUES  PIC X(60) VALUE
       'JAN01FEB02MAR03APR04MAY05JUN06JUL07AUG08SEP09OCT10NOV11DEC12'.
    03  MONTH-TABLE   REDEFINES MONTH-VALUES  OCCURS 12.
        05  MONTH-NAME  PIC X(3).
        05  MONTH-NUM   PIC 9(02).

The discussion so far has covered most of the subject, with many good comments.
One of the few aspects that remains are performance and the robustness of
initialization

The discussion so far has dealt with syntax of VALUE clauses and the INITIALIZE
verb, and also useful interaction about style issues.

Another way to initialize the DATA-RECORD is to set up separate definition that
looks exactly the same, and has detailed value clauses, either the repeating
values mentioned or the vatiations, say of months of the year.

This new record would be static. Infact, every field except the 01 level could
be a FILLER (utterly unnamed in some compilers, by the way).

When you are ready to re-init just move DATA-RECORD-STATIC to DATA-RECORD. 
This would generate something like an MVC on mainframes or a REP STOSB on PCs.
(It is not inconcevable that some optimizing compilers could generate REP STOSW
or REP STOSD in some cases, but I don't know that personally).

This relates to the original posters question, perhaps, by virtue of the fact
that re-initialization is considered at all. For re-inits inside of inner
loops, it would make a difference if you could reduce it to a move that
executes at the speed of an individual instruction.  

The COBOL INITIALIZE verb is a syntactic short cut for a long list of moves.
The list of moves gets coded for you by the compiler, as it were. And in the
case of numerics, sundry extra instructions to patch signs can actually be
emitted.

Programmer time is the most precious resource, usually. But where speed of the
application counts, a group move can help.

An occurs 20 by two byte work area, is going to initialize quickly no matter
what you do, unless there is a truely large number of executions. Yet sometimes
posters do us the service of reducing examples for our focused analysis.  In
the real world awesome resources are used to init 01s with the INITIALIZE verb.

One other consideration here, and I know this glides away from the main focus
of the syntactical interaction between INITIALIZE and data definitions.  But
semantically INITIALIZE does not mean anything at all for the FILLER and
unnamed data items.

INITIALIZE leaves FILLER and unnamed data items un-re-initialized.
If such items have no original VALUE clause and were not effected by group
moves, then troublesome garbage remains in the data area.

This is a bit of a problem because practical applications have oodles and
oddles of filler to allow for 'expansion' with out increasing the size of data
records.

To get around this problem some apps will initialize by first moving spaces, or
low values to either the entire 01, or numerous subsection group levels.
AND THEN INITIALIZE THE 01, Definitely engaging some duplicate effort by the
faithful CPU.  It is not that the programmers are lame, INITIALIZE actually
misses the target partially.

Moving  something like DATA-RECORD-STATIC to DATA-RECORD can address the
portion of the data record not accessible via INITIALIZE. This is a solution in
a limitted set of conditions.

Also, not to pick on any one comment, but it seemed implied in some of the
discussion that INITIALIZE picks up VALUE clause values.  This seemd to be the
notion in the discussion of the original values if there were not repeating.

For example, if all the PIC 9s items were not to be zero, but say values
ranging from 0 to 9 (cycling), then a redefined area with such values could be
the target of an INITIALIZE verb, and this perhaps to some seems like a way to
get the table to snap back.

I don't think so.  (and perhaps I draw too much of a conclusion about implied
meaning).  The point is INITIALIZE (AFAIK) does things by class
alphabetic/numeric and assigns a single value to each item per class.
You are not stuck with merely zeros and spaces.  But what ever you pick goes
into all the items of that class.

The original value clauses are not referenced by the INITIALIZE verb. And that,
I understand as the language standard.  Let me know if anyone sees it
differently, or has a compiler with extensions that go beyond this.

So, I emphasize, that move of a DATA-RECORD-STATIC to DATA-RECORD has limitted
utility.
But in addition to performance advantages, it permits initialization in a
NON-NEANDERTHAL manner, and it reaches portions of a group item that are
utterly transparent to the INITIALIZE verb.

So I advocate a new verb for the language: NON-NEANDERTHALIZE. And whatever way
we definte it, it would be a prerequisite that it happen quickly, even on
Sierra Class machines.
