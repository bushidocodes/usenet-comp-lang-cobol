```
Here's my troll for today...

In a prior thread, we talked about CONTINUE, NEXT SENTENCE and the use of
one period per paragraph.  We have talked about programmers who code GO TOs
to an exit paragraph, like this:

PERFORM ENTRY-PARA THRU ENTRY-PARA-EXIT.
...........................
ENTRY-PARA.
    DO THIS.
    DO THAT.
    IF SOME-CONDITION
        GO TO ENTRY-PARA-EXIT.
    DO THE-OTHER.
    DO SOMETHING-ELSE.

ENTRY-PARA-EXIT.
    EXIT.

Now, assuming you use one period per paragraph, couldn't you do this?

PERFORM ENTRY-PARA.
...........................
ENTRY-PARA.
    DO THIS
    DO THAT
    IF SOME-CONDITION
        NEXT SENTENCE
    END-IF
    DO THE-OTHER
    DO SOMETHING-ELSE
    .  <---(period)

NEXT SENTENCE moves you past the period, essentially exiting you from the
procedure.  Cute, but strange.

While this is not my cup of tea, it also points out the difference between
NEXT SENTENCE and CONTINUE when eliminating periods within a paragraph. 
Newbies beware!
```

```
Actually, in an ANSI conforming program this is NOT legal code.  There is an
explicit rule that says that "thou shalt not" use a NEXT SENTENCE in an IF
statement that is terminated by an END-IF.  As I recall, you (Cliff) use an
IBM mainframe compiler.  They, like several others, do have an extension to
allow this - but it is NOT "conforming" source code and is not necessarily
portable.
```

```
Hmm, that seems a bit dangerous; one misplaced full stop could bollix
the whole thing. I would be inclined to code it thusly:


    PERFORM ENTRY-PARA THRU ENTRY-PARA-EXIT.

    .

    .


    ENTRY-PARA.

        DO THIS.

        DO THAT.

        IF NOT SOME-CONDITION
            DO THE-OTHER
            DO SOMETHING-ELSE
        END-IF

    ENTRY-PARA-EXIT.  EXIT.


People often cringe at the NOT operator, but I find the above much more
straightforward than using NEXT SENTENCE or CONTINUE.  I have often seen
long chains of:

    IF CONDITION-A
        NEXT SENTENCE
    ELSE
        PERFORM PROCESS-X.

    IF CONDITION-B
        NEXT SENTENCE
    ELSE
        PERFORM PROCESS-Y.

I cannot believe that people would code such simply to avoid the NOT
operator.  Is the above really clearer than the following?

    IF NOT CONDITION-A
        PERFORM PROCESS-X.

    IF NOT CONDITION-B
        PERFORM PROCESS-Y.

Or, to eliminate the full stops:

    IF NOT CONDITION-A
        PERFORM PROCESS-X
    END-IF

    IF NOT CONDITION-B
        PERFORM PROCESS-Y
    END-IF

The last example is my preference.
```

```
maintenance jockey might, at 3:AM have problems with

IF NOT CONDITION-A AND NOT CONDITION-B

is prohibited by a 'No NOT... AND' injunction while a simple negative
conditional might still be clear.

(the condition being checked was for something like a valid department
number; the code was something like

READ-N-EDIT.

    READ TRANSFIL
        AT END MOVE 'Y' TO WS-TRANSFIL-SW
               GO TO RNE-EXIT.

    IF DEPTNO = '0144' OR '3421' OR '3895' OR '3829'...
    (several lines of ORs later)
        NEXT SENTENCE
    ELSE
        PERFORM INVALID-DEPTNO-RITUAL THRU IDR-EX
        GO TO RNE-EXIT.

(the paragraph was invoked, of course by PERFORM READ-N-EDIT THRU RNE-EXIT
UNTIL NO-MORE-TRANSFIL)

I argued that in absence of a lookup-table or VSAM read it would be best
to code a copybook containing:

05  WS-DEPTNO PIC X(4).
    88  VALID-DEPTNO VALUES '0144', '3421', '3895'. '3829'... etc.

... and change the code to something like

    MOVE DEPTNO TO WS-DEPTNO.
    IF NOT VALID-DEPTNO
        PERFORM INVALID-DEPTNO-RITUAL THRU IDR-EX
        GO TO RNE-EXIT.

After the squawking about the negative conditional there came the
interchange:

'But... when we get a new department we'll have to change the copybook and
compile all the programs that use it!'

'That's right... what do you do now?'

'Well... we change all the programs themselves and recompile them.'

'So what's better... changing one source module, a copybook, and
recompiling... or changing a bunch of source modules, the programs, and
recompiling them?'

'Uhhhhhh..... that's a Good Question, I'll have to get back to you on
that.')
```
