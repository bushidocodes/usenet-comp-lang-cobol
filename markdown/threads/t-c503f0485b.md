AFAIK, the COBOL language does not (currently? Wasn't this one of the things
under consideration for future COBOL standards?) have any native "language"
support for dynamic memory allocation. However, I know  COBOL/390 (IBM
mainframe) and any COBOL/Windows which can call the WInAPI can access system
services to do this.

However, when doing this there is no ability to use "regular" file verbs
like READ, WRITE, etc on the handle obtained from "fopen", unless the
operating system treats a handle is a handle is a handle and the COBOL
system can be configured/set to use the system handle rather than its own
internal handle. (Under windows, the API provides read/write/position calls
which allow files to be treated transparently regardless of "physical"
storage).

You need to describe the compiler and platform.  Memory I/O is
platform specific.  It is not part of the COBOL language standard.  In
C, memory allocation appears standard but it really depends on the
platform - Unix, Windows, VAX, HP, etc.  You CALL a memory routine
just like you do in C language.   Neither language has a  given
'command'.

With Micro Focus COBOL on a Windows/DOS platform, the compiler is
shipped with a set of routines that include memory allocation.  For
example, you can allocate a chunk of memory and access it with the
MOVE statement using reference modifiers.   

If I'm not mistaken, it is virtually the same routine that C language
uses with an interface suitable for COBOL, just like the Stream IO
library for COBOL  - sound familiar?

It has been possible to use files in memory on IBM mainframes for any
programming language since at least the mid 70's.  This is done by
specifying UNIT=VIO in the JCL for the execution of the program, The
TSO ALLOC command can be used similarly.  Some installations might
disallow this or give it different names depending on their standards.
 In these cases the files can have seqential, indexed or relative
organisation and can be QSAM or VSAM.  The normal file commands OPEN,
READ, WRITE, REWRITE, START, DELETE and CLOSE can all be used with
such files, their use is even transparent to the configuration
section.

Hi,

Thinking about your original question some more, I think that the file
a.b(mbr) probably already exists and is permanent, i.e. resides on
disk as it appears to be a PDS.  (I don't know whether PDSes can use
VIO, but can't see much point as they are normally permanent files.) 
In this case UNIT=VIO can not be used unless the member is copied to a
temporary file for which UNIT=VIO can be used. Obviously a file in
memory isn't permanent, so if you want to keep it and use memory, then
it must be loaded up into memory to do the processing, then unloaded
to disk to keep it.  However, since PDS members are usually not very
large, I doubt whether there is much point, the overheads of the
temporary location in memory may outweigh the benefits.  Another way
of getting a file into memory is specify the buffering in the JCL such
that the entire file is in the file buffers, this is more likely to be
the solution to your problem.  Another solution would be to read the
entire file into a table and use normal table processing for the
accesses, then, if necessary, rewrite the amended table back to the
file at the end of the job.

I am not very familiar with C so don't know what the FILE command you
used as an example does.

Hope this helps, Robert

datasets that are created and deleted during one and the same job.
For these files all the normal commands, like OPEN, CLOSE, READ, WRITE

Now I know just enough about C to cause serious harm.  What does the
memory file I/O operation really do?  Without more info I can only
guess what could be of help to you.  One thing that comes to mind is
DIV (Data In Virtual) which allows you to associate a huge in memory
array (in virtual storage) to a VSAM file.  You then do not use any
file access commands, but just access the table/memory location you
are interested in.  Only the data that you access will we read into
memory (automatically), and the updated records will be written back
at task/program/process termination

You should also look up Data Windowing Services, which allow a high
level language nearly the same functionality as DIV

Hello Shyam

just use "write output-record from input-record"
where output-record is the record description in the output file FD
and input-record is the record description in the input file FD.

However, for this task as you have explained it so far, I would tend
to use file buffering to hold the records in memory from the file you
are reading and not bother copying the file to a temporary file in
VIO, you can specify the BUFNO=n parameter in the DCB to do this,
where n is the number of buffers required.

e.g. //inputdd    dd dsn=infile,disp=shr,
     //              dcb=(bufno=n)

To work out the number of buffers you want to use, divide the number
of records to be read by the number of records in a block.  There is
probably a limit to the number of buffers that can be used, so a large
blocking factor would be desirable.

for effective use of VIO, I would be inclined to use a block size of
4000 and allocate the space in units of 4000, since the file is being
held in memory and 4000 is the highest multiple of 200 that will fit
in 4096 bytes which is the size of a page in memory.  Larger multiples
of 200 could also be used, such as 8000 which would fit in two pages,
I think the lowest multiple using no wasted space in pages is 102,400
for 25 pages at a time.  Whether or not the operating system would
accept such a block size, you would have to try out yourself. 
However, there wouldn't be any point using such a large block size if
the file doesn't contain enough records to use it all.  I don't know
what the operating system think the size of a track is in VIO, it may
be that it just uses an optimal multiple, on the other hand it might
not.
