Isaac,

There is no such thing as a "cobol database".
There are several kinds of Cobol files (sequential,
relative, and indexed files). You are probably
talking about one or more indexed files.

A Cobol programmer can build indexed files in
any format (s)he wants to. The format is often
described in what is called a COPY BOOK.

If the copy book is available, any Cobol programmer
can write a program to "offload" the file to a
file for excel or some such in about an hour.
But (s)he needs to use the same compiler as the
program that built the original file.

If you have not played with this technique, Leif, then you should.
Basically,
you start the spreadsheet, and it becomes a subroutine of the Cobol
program.  Because it is a "container" you can place data directly into
cells just as if it was a Cobol two dimensioned array. It runs in a separate
window, and you can watch the two programs pass data back and forth
on the screen. The data is automatically converted from format to format
just like any other Cobol move.

It is an excellent way of giving the user the ability to do specialized data
analysis from a Cobol data base.  You simply give them a basic template
for a simple report, and invoke it.  You then dynamically load the data
into one row or column, and let them format/manipulate it to their
hearts content. It gets rid of just about any report under twenty or
thirty pages, and comes complete with all the fonts, graphs,export/import
to word processing documents, etc., etc.

I think it one of the most exciting things about the new compilers ...
it makes the report writer look primitive. (though to tell the truth I've
always considered it primitive) ;<)

Sorry to have taken so long to respond to this, Thane, but it took
me about five minutes to write an example, then half a day to
figure out how to get it out of the IDE and into a text file so that
I could post it<s>.  The wonders of modern tools.  This is for
Fujitsu Version 4.  Version 3 works a bit differently.

First of all, place an excel connection device on your form.  Then
you open a spreadsheet.  The rows and columns of the spreadsheet
become properties of the connection, and data is simply moved
to or from them with standard Cobol moves.

The following is a code fragment that executes on the "click" of
a start button. For it to work, the following conditions must be met.
Excel.exe must be on the current path, as the invoke statement
starts it.  The spreadsheet "Q:\TEMP1\TEST.XLS" must exist,
though there are other invoke methods that could create it. The
excel connection on the form must be called "EXCELL".  That
is not the default name, but I have gotten into the habit of naming
controls as I place them onto a form.

The tested program that I lifted it from had only two controls,
a button to click on and trigger the code below, and the excel
control.  It starts excel, and places four numbers in the upper
left hand four cells of the spreadsheet. It then places the word
"Sample" into column 1 row 3.  You are left with two windows
open ... the program and excel.  You can, of course, move
data in the opposite direction as well.  The help system gives
the format of the Cobol data, I should have grabbed it for this
example but did not.

 ENVIRONMENT     DIVISION.
 DATA            DIVISION.
 WORKING-STORAGE SECTION.
 01 SPREAD-SHEET-NAME PICTURE X(45).
 01 SHEET-NUMBER        PICTURE S9(5) COMP-5.
 01 RETURNVALUE  PICTURE S9(5) COMP-5.
 PROCEDURE       DIVISION.
 MAINLINE.
     MOVE "Q:\TEMP1\TEST.XLS" TO SPREAD-SHEET-NAME.
     INVOKE EXCELL "OpenBook"
      USING SPREAD-SHEET-NAME
            SHEET-NUMBER
      RETURNING
            RETURNVALUE.
     MOVE 1.0 TO "Numeric"(1 1) OF EXCELL.
     MOVE 123.0 TO "Numeric"(2 1) OF EXCELL.
     MOVE 1.5 TO "Numeric"(1 2) OF EXCELL.
     MOVE 2.55 TO "Numeric"(2 2) OF EXCELL.
     MOVE "Sample" TO "Text"(3 1) OF EXCELL.
