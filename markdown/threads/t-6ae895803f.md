```
The COBOL 85 standard specifies a minimum of 3 levels, but many compilers
support more than that.  Unisys A Series, for example, supports up to 48
levels (OCCURS on every level number 02-49).  This is an extravagant limit,
for if each of 48 OCCURS were 2, then 2**48 = 281,474,976,710,656.  That's
a lot of RAM! :-)
```

```
I think we are confusing minimum/maximum "usages" here.  (Judson, was just
wrong about "3" - which was the '74 value.)

What the '85 Standard  says is that EVERY compiler must support tables which
include 7 levels.  It is "allowable" for an implementation to support more
than 7 levels, it is NOT conforming to support less than 7 levels - but
obviously, when you support 7 level tables, you can still have individual
conforming programs that use any number of OCCURS from 1 thru 7.
```

```
I love it when the syntax limit is way larger than the practical limit. 
The compiler isn't that much more complicated than it would be with 5 or
6 levels, and it doesn't make sense to try to guess what is practical
and try to hit it - when things change so rapidly.  In this case, 48
wasn't even a "way large" selection, it was basically no limit at all
inside of the COBOL limit on level numbers.  Good programming design by
the compiler writers.
```
