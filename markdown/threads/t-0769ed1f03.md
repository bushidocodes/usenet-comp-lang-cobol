```
I have the following code in MicroFocus that I want to port to IBM COBOL.
I can not find a graceful way to do it.

                  START SomeFileHandle
                     KEY IS = SomeFileKey
                     SIZE PartialKeyLength
                  END-START

The reason is that IBM does not support the SIZE keyword, (at least
not on win2k were I am testing).

Here is what I have tried:
1)
                  MOVE LOWVALUES TO SomeFileKey
                   (PartialKeyLength + 1:FullKeyLength - PartialKeyLength)
                  START SomeFileHandle
                     KEY IS >= SomeFileKey
                  END-START

   But that has a problem that will return a record that is past the record
   that i care about rather than a not found condition.

2)
                  START SomeFileHandle
                     KEY IS = SomeFileKey(1:PartialKeyLength)
                  END-START
    That did not even compile.

Does anyone have any bright ideas before I resort to reading the
record after the START in 1)?  I will have to read the record to
determine if it is past the record that i am actually interested
in.
```

```
Did you try
    KEY = SomeFileKeyLeftPart ?

where SomeFileKeyLeftPart is "alphanumeric whose leftmost
character ... corresponds to the leftmost position of a record key
... and whose length is not greater than the length of the record
key."   (X3.23-1985)

It's what you were trying to do with your try #2.

Should work for an '85 COBOL.
```

```
I think you already have this solution, sort of...but this is a little
different in that the first record it will return will be the first record
with the matching 'first n' characters...

Select the-file
  organization is indexed access is dynamic record key is the-key.

FD the-file.
01  the-record.
   05 the-key   pic x(20).
   05 the-rest   pic x(9000)

MOVE low-value to the-key
move  SomeFileKey(1:PartialKeyLength)  to the-key
start the-file
       key is not less than the-key
  invalid key
       set no-records-matching-key to true
 .....


FWIW, you are not going to be able to get around testing the actual returned
key value under any condition in which you are not searching for an 'exact'
(including size of significant characters) hit. But you "could" make your
life easier by putting the Start/ReadNext/Compare into a nested program and
just calling it with one line of code from your main...as in...

PROGRAM-ID. MAIN.
..
01 FILLER
     05 HIT-FLAG    PIC X(01).
        88 HIT-YES   VALUE IS 'Y'.
...

CALL AnyHits using SearchKey Hit_flag, Data-Record
IF Hit-Yes THEN
* the first qualfied record is in data-record....
ELSE
   DISPLAY 'No records found matching search criteria'
END-IF
...
END PROGRAM.
PROGRAM-ID. AnyHits.
...
LINKAGE SECTION.
 01  Search-key   PIC X(20).
 01 Hit-flag          PIC X(01)
 01 Data-Record.
    Copy 'above'.
  ...

PROCEDURE DIVISION USING Search-key, Hit-flag, data-record.

(Same logic as above, except, add READ NEXT, comparison test, set Hit-flag,
move returned record to Data-Record if HIT=true)

...
END PROGRAM.


You can use the EXTERNAL clause in the FDs to keep the file pointer in synch
between both programs....of course, lots of this is application dependent
and "programmer-comfort-with-nested-and/or-call-and/or-EXTERNAL-clause'
```
