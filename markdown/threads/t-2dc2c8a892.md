There are two reasons I can think of about this point.

First, COBOL treats every group item as X field. That is if you
declare the following data structure at working storage.

01 WS-GROUP.
05 WS-SUB-GROUP-1.
10 WS-ITEM-1 PIC 9.
10 WS-ITEM-2 PIC 9.

Even all detail items of WS-GROUP or WS-SUB-GROUP-1 are numeric
fields. It is still legal to move anything into WS-GROUP and
WS-SUB-GROUP-1.

For example, MOVE 'XY' TO WS-GROUP.

In the case of reading an input file within the COBOL program. The whole
record will be read in (as a group field). The program at this point
cannot tell whether there is any discrepancy of data type. So the test
for numeric/alphanumeric is taking place.

Second, there is something called REDEFINES at the COBOL languages.

01 WS-GROUP.
05 WS-ITEM-NUM PIC 9(4).
05 WS-ITEM-NAME PIC X(20).
05 WS-ITEM-CODE REDEFINES WS-ITEM-NAME.
10 WS-ITEM-ID PIC 9(8).
10 WS-ITEM-DESC PIC X(12).

In the above example, both WS-ITEM-NAME and WS-ITEM-CODE share the
same physical position of the variable. ie. 5th bytes to 24th bytes
of the WS-GROUP but it can represent different types of data.

The simplest answer I know of has to with reading information from other
vendors. You may not know (or trust) the data format, so you read it in
as alphanumeric. Then, if it passes a numeric test you can move it to a
numeric field for processing, or output it into an error file still in
alphanumeric format.

Also, I have seen some debuggers break when trying to examine a numeric
field which contains non-numeric data. It seems that the debuggers
weren't smart enough to do a numeric test first.

Hope this helps.

(snip)

› numeric, why would you possibly test it for alphanumeric as an example. 

If it's declared as numeric, it would indeed be unusual to test for
alphanumeric. But the reverse is not uncommon.

If the data comes from a screen and, for example, there is a numeric
'edit mask' imposed on the screen's input field, and everyone's terminal
emulator recognizes that imposition, then there is no reason to check
it. However, there may be situations where this is not guaranteed and
you need to check (rather unusual, you would hope).
Likewise, from an input file, you'd hope that the file layout is correct
and that you wouldn't need to check it either.
I think you can conclude (in most cases) that after the initial
'bug-killing', you'd expect the data to be correctly formatted and not
worry about checking it for numeric, etc. You might find testing for
numeric to be handy during debugging, of course.

But here's some examples where you might want always to check it for
numeric, whether from a screen or an input file.

- If there is some significance to numeric vs. non-numeric data in the
field. For example - Managers have employee-ID's beginning with 'M' and
employees simply have numbers. Then you'd ask about numeric to decide
what logic to perform.
- If there is a table of numeric data. Sometimes the entire table is set
to high-values and then individual PIC 9 elements are filled with
numbers. So you could use code to check if the field is numeric before
processing. Of course, you could also check for high-values (with the
new compilers), but I seem to remember that the old ones wouldn't let
you ask the high-values question if the field was PIC 9.
- If there are requirements for display unique to your application. For
example, suppose Department can be 123, 123A, etc. as stored by the
Personnel System's programs. But your system wants to see 123A displayed
as 123-A. You could do a numeric check on the bytes to find if a dash is
needed (and find the place for the dash).

There are surely other methods to do the above, but they are real
examples that I've seen here and there.

Let me answer your second question first. When you are writing an
output
file to disk you can use the FROM clause just as if you were writing to
a
printer. The type of device doesn't matter. You can always use the
FROM
with WRITE.
WRITE PRINTER-RECORD FROM DEL-RECORD
is just like
MOVE DEL-RECORD TO PRINTER-RECORD
WRITE PRINTER-RECORD

Your first question will take a little more explaination. Every PICTURE
may
tell a story but in COBOL it doesn't tell the whole story. You also
need a
USAGE clause to determine how a variable will be stored in a COBOL
program.
If you don't explictly define a variable's usage the compiler will use
USAGE
IS DISPLAY. Other popular usages are BINARY and PACKED-DECIMAL. (COMP
and
COMP-3 to us dinasaur's). BINARY is like Pascal's "integer". DISPLAY
is
more like Pascal's "string" or "array of char". Just like a Pascal
string
can have characters that aren't just digits, any USAGE IS DISPLAY
variable
could have characters that aren't just digits. When you try to MOVE the
contents of a DISPLAY variable to a field that has usage is BINARY and
that
variable has only digits in it the program can translate from DISPLAY to
BINARY. When you try to MOVE the contents of a DISPLAY variable to a
field
that has usage is BINARY and that variable has something other than
digits
in it the program goes "kaboom!". So we test for a NUMERIC condition
before
the MOVE.

I think I can predict your third question: "How can non-NUMERIC data get
in
there in the first place?". COBOL has group-items which are composed of
elementary-items (insert Sherlock Holmes joke here). Group-items don't
have
PICTURE clauses and elementary-items do have PICTURE clauses (well, most
of
them). You can MOVE things around at the group level and that is where
things can get mucked up. Group level MOVEs are alway treated as DISPLAY
to
DISPLAY with no translation. You know about WRITE/FROM. There is also
READ/INTO. That is the most popular way to get non-numeric data into a
numeric field.
