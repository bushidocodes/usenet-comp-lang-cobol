I don't disagree with your method, but to me, a 'merge' and an 'update'
are two separate things.  A merge simply combines the contents of two
or more files in key sequence.  If two records have the same key value,
they are considered equivalent, for merge purposes.  An update, on the
other hand, would consider one file as the 'master', then apply one or
more files against it as 'transactions'.  In this case, sequence becomes
very important, because you need to know if two or more transactions in
different files have the same key value, in which order should they be
processed.  This is a very different situation from a simple 'merge',
and I agree that the logic would need to be different.  I would likely
approach this situation by merging all the transaction files into one
file (if necessary), using a sequence indicator (time?) to determine
processing order.  Then bump the one transaction file against the master
file, applying each transaction in proper sequence.

Yes, your method of testing each file for equality on each pass could
be more CPU efficient, as long as there were enough duplicate records
to more than compensate for the fact that you evaluate the compare for
every key every time, where my code only evaluates the compare until
an equal is found, averaging 50%.  If there are no, or few, duplicate
keys in the several files, your method would be less efficient. :-)

LK-END-FLAG will never be = "X", unless you force another pass through
the loop, when no more records are present.  You would need to code an
additional test to skip the write code when LK-END-FLAG = "X".

I generally prefer 0/1 myself over space/X, and started to use it. :-)

I tend to avoid 88 levels, because you cannot now use them in SET.  If

you think to go look at what the 88 level value is, and what data-name it
is associated with.  However, if you see "MOVE 123 TO data-name" then
later "IF data-name = 123", that is instantly clear.  You can even search
for "123" and find both of them.  When I use a data-name to store codes,
I always put the possible code values and meanings as comments right
