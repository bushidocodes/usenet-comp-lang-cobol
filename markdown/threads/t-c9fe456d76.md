Your most likely problem is Access itself.

Access has its own database engine, and it thereby has its own 'thoughts' on
how to do queries.  If a table has a unique index, Access tends to get a
keyset for the entire table before it actually does a query.  The only
mechanism I have found to defeat this is the use of Passthrough queries.

If you have have MS-Query installed (e.g. for use by Excel or Word), try
using it to build and test your queries.  MS-Query is not nearly as
'intelligent' as Access and can be a lot less frustrating.  When you have a
query behaving as you wish, cut the SQL from the SQL window in MS-Query and
paste it into the SQL window in a new Access query, mark the query as
passthrough, and Access should behave better.

Of course, it _is_ possible to construct queries that cannot use the indices
available, but I think this is not your current problem.

(By the way, I am actively working on a new query for our accounting
department that is exhibiting this symptom.  The query even uses our Order
Header table!)

>
> We tried to install both the mapped drive version
…[5 more quoted lines elided]…
> run ODBC when users are still online.

This makes sense.  The SQL engine is running on the server in C/S, but on
the workstation in the 'mapped drive' version.  The workstation version will
cause a lot more network traffic (thereby four times slower) due to the fact
that it is moving individual disk I/O requests across the network.  On the
C/S version, only the result set is moved over the network.

>

Actually, unless your query is updating tables, there is not any locking
going on.  Through private email, please forward the name of the application
vendor, so that we may do some proactive support.
