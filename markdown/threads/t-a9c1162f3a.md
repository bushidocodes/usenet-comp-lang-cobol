I'm trying to round up some rather arcane information about the
representation of packed and zoned decimal numbers. I used to program in
COBOL many years ago (on a Xerox Sigma VI mainframe), but it's been a long
time and I've forgotten some details that I now want to know. If you can
answer some or all of the questions below, I'd greatly appreciate it.

I recall that the term "packed-decimal" is usually hyphenated. Is that
correct? What about "zoned decimal"?

With a zoned decimal number, the sign is typically overpunched on the
rightmost digit. That digit should then show up as a letter or some other
odd character. How is that usually done? How would I see the value -12345
given the following declaration:

05 my_value  PIC S9(5).

If the platform matters, assume that I'm on an IBM mainframe. I just need a
relatively common example of how this is done.

Packed-decimal numbers have their sign (usually) in the right-most nibble. I
recall that X'C' was used for positive numbers. Thus 12345 would be
represented as X'12345C'. What is the typical value used for negative
numbers? I think a X'D', but I'm not sure.

On my system, admittedly from the stone-age of computing, we used COMP-3 to
declare packed decimal numbers. For example:

05 my_packed_decimal_value PIC S9(9)V99 COMP-3.

Is this still true today? I recently saw a FAQ that leads me to believe that
the following is more likely to be used:

05 my_packed_decimal_value PIC S9(9)V99 PACKED_DECIMAL.

Is that FAQ correct?

One last question. Was COBOL designed in the '50s or the '60s?

I will very much appreciate any help I can get on these questions. I'll
check back on this newsgroup from time-to-time, but it would help me out if
you also copied me by email on any replies.

Thank you.

Jonathan Gennick
Associate Editor
O'Reilly & Associates
jgennick@oreilly.com

storage: Early mainframes were the tools of the accounting department.

numbers. For example, the machine instructions Add Packed, Subtract
Packed, etc.

floating-point representations with its attendant loss of precision. I

Depends on the declaration. If you define my_value, you'll get a
compiler error ;-). If you define 05 my-value pic s9(5) sign is leading
separate, -12345 will be printed -12345.
OK, assuming you have 05 my-value pic s9(5), your field will contain
1234N. The sign-nibble is in the rightmost byte of the field, it will
contain 'D5'x in your case. A positive 5 would be 'F5'x. The character
representation of the negative zoned decimals are (assuming EBCDIC):

-1=J  -2=K  -3=L  -4=M  -5=N  -6=O  -7=P  -8=Q  -9=R  -0=}

the positive. You are correct, the IBM mainframe COBOL compilers sets a
'C' for the positive sign. That's not a COBOL issue, but rather an
ASSEMBLER (or, more accurate a CPU and microcode) issue. After every

packed-decimal data.

hex:      CCCCCCCCCC             DDDDDDDDDD

FYI,
  The revision (next COBOL Standard) introduces the underscore as a "valid"
character.  Yes, there are some implementations that support it today - and
some that don't.  The original question mentioned IBM mainframes when
talking about "characters" that a byte with a sign overpunch would "display"
as (which is HIGHLY significant because of EBCDIC/ASCII issues) - so saying
that underscore isn't valid is reasonable - as IBM mainframes do not
currently allow this in user-defined words.

Similarly, COMP-3 is not (never was - never will be) "portable".  Many
(most?) systems do accept it to be equivalent to USAGE PACKED-DECIMAL (which
was introduced in the '85 Standard) - but this is certainly NOT something
that should be used without/recommended without mentioning that it is
"compiler specific".
