Sure.

Don't do it.

That is, don't move alphanumeric fields to numeric fields. (What do you
suppose happens if you move "APPLE" to a numeric field?)

There is no primitive command to do what you want. You have to right-justify
the string, fill with leading zeros, and test for numeric before you do the
move. Here's one way:

MOVE FUNCTION REVERSE(MYFIELD) TO REV-FIELD.
INSPECT REV-FIELD REPLACING ALL LEADING ' ' BY '0'.
MOVE FUNCTION REVERSE(REV-FIELD) TO MYFIELD.
IF MYFIELD NOT NUMERIC
    you're screwed
ELSE
   MOVE MYFIELD TO NUMERIC-FIELD.

I take "bit map" to be "BMS map". There's no reason to define all your 
BMS map fields as character. BMS permits the use of PICIN & PICOUT if 
you're generating a map for a COBOL program and I would urge you to use 
them. Saves a lot of monkey motion editing fields in WS then moving them 
to your output map field.

If "multiple input" means you have a mapset containing more than one 
map, you have to keep track of which one you're using to specify the 
correct mapname in the RECEIVE command.

Many people mistakenly believe all BMS fields must be character and the 
programmer is responsible for all editing, but that's not the case with 
current versions and hasn't been for more than 10 years, or so.

TEST-NumVal-? function that *does* validate that the input meets the
"requirements" for the function.  (It does NOT return the numeric or
numeric-edited version of the argument - it just returns a "good" or "bad"
