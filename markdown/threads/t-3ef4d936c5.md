I know this is a COBOL newsgroup and to ask an IMS question is breaking the
rules, but I'm stuck and I can't find an IMS newsgroup to address this to.
If anyone can shed any light on this then I will be extremely grateful.
I am setting up a test online system using COBOL accessing IMS test
databases. When doing a get unique call to one database I keep getting an
IMS user abend 0853. I've looked the message up and it states as follows

For a HIDAM or HDAM database, in attempting to locate a root segment, the
segment returned either did not have a segment code of 1, or (for HIDAM)
had a different key value than the key value in the index pointer segment.

Our IMS technical guys have restored the databases and indices a couple of
times, but we still get this. The database is defined as HIDAM. No one
knows why this is happenning and I'm starting to wish my Mummy (or Mommy)
was here. Thanks for any help, advice, suggestions offered and apologies
for the lapse in netiquette.

Thanks to everyone who offered help and advice. One of our IMS tech people
tracked the problem down and it is a bit of a gotcha.
The production DBD source looks like this

DBD NAME=XXXXXXXXX, X
ACCESS=(HIDAM,VSAM)
XXXX DATASET DD1=XXXXXXXXX, X
DEVICE=3380
SEGM NAME=XXXXXXXXXX, X
PARENT=0, X
BYTES=(300,131) X
PTR=TB

The test DBD source looks like this

DBD NAME=XXXXXXXXX, X
ACCESS=(HIDAM,VSAM)
XXXX DATASET DD1=XXXXXXXXX, X
DEVICE=3380
SEGM NAME=XXXXXXXXXX, X
PARENT=0, X
BYTES=(300,131),
PTR=TB

Apparently a previous version of IMS had allowed the production DBD source
to be genned and had taken PTR=TB to be a comment, because BYTES=(300,131)
is not followed by a comma and the continuation character (X) is ignored.
When an attempt was made to gen the DBD in test, I guess it failed on
syntax. When a comma was put after BYTES=(300,131) and the continuation
character removed, the DBD gen worked. However the database was now four
bytes out of sync with the index database due to the inclusion of the twin
backward pointer (PTR=TB). This cause the 0853 error message to appear
when a GU call was attempted.
By commenting out PTR=TB and removing the preceding comma and then
regenning the DBD the problem was solved.
BTW, any takers for an IMS newsgroup or does anyone know of one?
Regards
Alex Chaplin

It could be use of an incorrect DBD, or that the image copies (backups) of the
database were made after the error occured. This is the classic "broken
database" abend -- almost any integrity problem could have caused it.

You mention that it's a test environment -- have you run batch jobs that
trapped without backing out changes or restoring from a before-image? Maybe an
application programmer or an operator restarted a job after a database update
step ran out of space or was canceled. You should also check the logs of the
test on-line IMS, looking for dynamic backout failures or a cold start of IMS
after it (or MVS) crashed. Is it possible that the VSAM share options allowed
someone to run an update (maybe from another CPU) while IMS thought it owned
the datasets?

The reorg idea is a sugestion worth trying, but the db unload will probably
hit the same error (unless the pointer to bad data is in the secondary index).
Ideally you'll have a means of recreating the test database --- make sure you
take an image copy afterwards (2 copies). If the data is very
important, you could have a good DBA zap the bad pointers to zero. You'll lose
some data but not all.

And tell your IMS guys to figure out why their restore files are no good! What
if this was production data! (My best guess of a cause --- they restoring the
primary index with the data image copy and restoring the data with the primary
index image copy...check the datasets yourself).
