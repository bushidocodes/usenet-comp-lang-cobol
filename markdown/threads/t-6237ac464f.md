of a packed field.
Does anyone know of a easy and efficient way to do this?

*Input 8th byte is for the leading sign, space if pos - if neg.

01 In-Num PIC X(8).

*Output 8th byte is for the leading sign, space if pos - if neg.

01 Out-Num PIC X(8).

*Work number

01 Work-Num PIC S9(7)V Packed-Decimal.

bbbbb-12 neg
bbbbbb12 pos
-1234567 neg
b1234567 pos

In CoBOL, each elementary item belongs to one of these categories, according to
its PICTURE clause:

Numeric
Numeric Edited
Alphabetic
Alphanumeric
Alphanumeric Edited

Your question involves the numeric class, which includes the numeric category
and the numeric-edited category. These classes and categories do not exist to
vex you; they are useful abstractions about the way people actually use numbers.
Basically, the numeric category is for internal representations of numbers and
the numeric-edited category is for external representations of numbers.

Consider the following elementary items:

05 WS-NUMERIC PIC S9(5)V9(2).
05 WS-NUMERIC-EDITED PIC $$$,$$9.99.

If WS-NUMERIC has the value "1000000", and you MOVE WS-NUMERIC TO
WS-NUMERIC-EDITED, then WS-NUMERIC-EDITED will have the value "$10,000.00".
There are two things to note here. First, the dollar sign, the comma, and the
decimal point are not physically present in WS-NUMERIC, otherwise it would be
hard for the machine to do arithmetic. Second, these characters are physically
present in WS-NUMERIC-EDITED, so the external representation is three bytes
longer than the internal representation in this case.

You can now see how to convert your internal packed decimal number into the
external format you desire. In fact, it would be worth your while simply to
read through the entire description of the PICTURE clause to get an idea of the
things it can do for you. Your specific question:

05 WS-COMP-3 PIC S9(7) COMP-3.
05 WS-DISPLAY PIC -(7)9.

MOVE WS-COMP-3 TO WS-DISPLAY.

This is about as efficient as you can get in CoBOL. If your hardware has an ED
instruction, it is hard to beat except in special cases. If your hardware does
not have an ED instruction, it can be beaten, but it's a lot of trouble.

Now, you say you want to read numeric-edited data into a numeric field, and you
say you are going to be doing this on large amounts of data. The first question
to ask yourself, very seriously, is why? Disks are much slower than CPUs. No
matter how inefficient your code is, it will not be anywhere near as inefficient
as wasting 12.5% of your disk to store unnecessary sign bytes! Also, each
8-byte input number takes 14.3% more transfer time than the corresponding 7-byte
input number! When you write a file that is going into another computer
program, it is only common sense to use an internal format.

OK, I know: you didn't design it; you're stuck with it; it comes from a C
program; it comes from a spreadsheet; it's a homework assignment; whatever. As
long as you understand that you shouldn't be doing it if you have a choice.
Moving a numeric-edited item to a numeric item is called de-editing, and it
basically does not fit into the CoBOL philosophy, which assumes you have a
choice. The need to de-edit comes up often enough that I wish the compiler
would just do it, and, guess what, after all these years, some compilers finally
offer a de-editing feature! What you should do is MOVE WS-DISPLAY to WS-COMP-3.
If your compiler will compile it, you are home free. If your compiler offers
de-editing, it will be about as efficient as you can get in CoBOL, and it
certainly doesn't take a lot of code.

If you don't have a compiler that offers de-editing, the real fun begins. There
are hundreds of ways to program this, and they all take a lot of code. The best
you can do is use a method where the code is compile-time code, not run-time
code. There is no one way that will be superior on all compilers and all
machines. In order to find the most efficient method for your compiler and your
machine, you will have to experiment. I myself am looking forward to reading
the clever and bizarre methods that others will post. For now, however, let's
look at a simple method starting, as you did, with the INSPECT statement:

05 WS-INPUT-X PIC X(8).

05 WS-COUNTER PIC S9(4) COMP.

MOVE ZERO TO WS-COUNTER.
INSPECT WS-INPUT-X
TALLYING WS-COUNTER
FOR ALL "-"
REPLACING ALL "-"
BY SPACE.

Now that we have removed the minus sign, if any, we redefine WS-INPUT-X to skip
the extra byte, and we pack the number:

05 WS-INPUT-X PIC X(8).
05 FILLER REDEFINES WS-INPUT-X.
10 FILLER PIC X(1).
10 WS-INPUT-9 PIC 9(7).

05 WS-COMP-3-U PIC 9(7) COMP-3.

MOVE WS-INPUT-9 TO WS-COMP-3-U.

By making the picture of WS-COMP-3-U unsigned, the same as the picture of
WS-INPUT-9, we enable the compiler to emit a single PACK instruction. Even on
machines that simulate packed decimal, this avoids wasting instructions to
adjust the sign. Don't forget to make sure all of your COMP-3 fields have an
odd number of digits!

Packed decimal is a beautifully consistent system in EBCDIC, the result of years
of real-world experience plus a lot of deep thought. You may have noticed that
I did not translate the leading spaces into leading zeros before packing. This
is because numbers in the real world sometimes have leading spaces and sometimes
have leading zeros. Packed decimal was designed to handle either kind of number
without wasting run-time cycles on testing, translating, etc. In CoBOL, on the
other hand, a number with leading zeros will pass the NUMERIC class test, but a
number with leading spaces will fail the NUMERIC class test. If you have input
that is supposed to have leading zeros, you test like this:

IF WS-INPUT-9 IS NUMERIC
THEN MOVE WS-INPUT-9 TO WS-COMP-3-U
ELSE [error]
END-IF.

In this case, by the definition of packed decimal, if WS-INPUT-9 is NUMERIC,
WS-COMP-3-U must be also, so it is not necessary to check WS-COMP-3-U. On the
other hand, if you expect the field to have leading spaces, or if you expect the
field sometimes to have leading zeros and other times to have leading spaces,
you test like this:

MOVE WS-INPUT-9 TO WS-COMP-3-U.
IF WS-COMP-3-U IS NOT NUMERIC
THEN [error]
END-IF.

The second example is more common than the first since it is more
efficient (4 bytes versus 7 bytes). This aspect of packed decimal has many
applications in hashing, password generation, random numbers, etc.

When you look at packed decimal in ASCII instead of in EBCDIC, the rationale for
some of the choices made is no longer obvious, and well-meaning people can make
mistakes in their implementations. I have encountered only one incorrect
implementation of packed decimal, but unfortunately that one is on the DEC VAX,
a rather popular machine. Apparently, DEC looked at the PACK instruction and
thought IBM had left out error checking by mistake, not realizing that the PACK
instruction was deliberately designed to defer error checking. As a result, in
VAX CoBOL it is impossible to read a leading-space number without spending extra
run-time cycles, much less a field that sometimes has leading spaces and
sometimes has leading zeros. In order to test whether your compiler has a
problem, do the following, which is a valid packed decimal operation:

MOVE SPACES TO WS-INPUT-X.
MOVE WS-INPUT-9 TO WS-COMP-3-U.

In the unlikely case that the implementation you are using is incorrect, it will
give you a decimal exception on the second move. If that happens, you will have
to convert spaces to zeros, otherwise it is a waste of run time:

INSPECT WS-INPUT-9 CONVERTING SPACE TO ZERO.

Now that we have the absolute value of the input field in unsigned form, we
have to set the sign if it is negative. The most efficient method varies
widely; the following is for illustration only:

05 WS-COMP-3 PIC S9(7) COMP-3.
05 WS-COMP-3-U REDEFINES WS-COMP-3
PIC 9(7) COMP-3.

IF WS-COUNTER IS POSITIVE
THEN COMPUTE WS-COMP-3 = - WS-COMP-3
END-IF.

If you've been following this, you will note that at this point WS-COMP-3 will
still be unsigned if the input was positive. However, packed decimal was
deliberately designed to treat unsigned numbers as positive rather than to waste
run-time cycles adjusting the sign.
