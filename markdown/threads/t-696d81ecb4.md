>I've worked with DB2, and IDMS.
>Can someone give me a quick overview of accessing IMS from COBOL?


Most shops are slightly different....but here  is some of what I know. I am
a bit rusty, havent had to do this stuff in over a year.

IMS is a database arraged in what I see as a tree structure. Where each
segment can have child segments under it. Where there are typically many
different types of segements and hopefully there is a utility to describe
there relationships. To retrieve a particular segment you would need the
keys to the parent segments of that particular segment, or you would have to
cycle through all segments of that type. Typically you would retrieve
segments by specifying the function (GU- get unique, GN get next, GNP get
next within parent) and the SSA which specifies the qualifications of the
segment you wish to retrieve. You specify the SSA as follows:
|Segment name | | Field name | Relational Operator | Field Vlalue | |
|8 bytes      |(| 8 bytes    | 2 bytes             | variable     |)|

A call would look something like :

CALL 'CBLTDLI' USING GU PCB WS-SEGMENT-GROUP SEG-A-SSA

Where:
GU is typically a WS- field which holds a 4 byte function to be used, in
this case 'GU  '
PCB is a kind of status area where error messages will be stored
WS-SEGMENT-GROUP is the working storage area where the requested segment
will be placed
SEG-A-SSA is the specification of wht to bring back

The PCB is sort of a pointer storing where you are in the database. Thus
using different SSA's with the same PCB traverses through the database. If a
database had the following segments and relationships:
PERSON segments containing personal id info such as SSN name birth date and
such
ADDRESS segments which would contain the many addresses a person lived at

To cycle through all of the addresses a particular person lived at you would
do something like:

01  ADDRESS-USSA.
    05 SEGMENT-NAME                   PIC X(09) VALUE 'ADDRESS   '.

01  PERSON-QSSA.
    05 SEGMENT-NAME                   PIC X(08) VALUE 'PERSON  '.
    05 FILLER                         PIC X(01) VALUE '('.
    05 PERSON-KEY                     PIC X(08) VALUE 'SSN     '.
    05 PERSON-RO                      PIC X(02) VALUE 'EQ'.
    05 PERSON-KEY-VALUE               PIC X(09) VALUE '000000000'.
    05 FILLER                         PIC X(01) VALUE ')'.
.
.
.
MOVE '000000001' TO PERSON-KEY-VALUE
CALL 'CBLTDLI' USING GU PCB1 PERSON-DATA PERSON-QSSA
PERFORM 8000-CHECK-PCB1-STATUS
CALL 'CBLTDLI' USING GNP PCB1 ADDRESS-DATA ADDRESS-USSA
PERFORM 8000-CHECK-PCB1-STATUS
PERFORM UNTIL NO-MORE-ADDRESS-SEGMENTS
    CALL CALL 'CBLTDLI' USING GNP PCB1 ADDRESS-DATA ADDRESS-USSA
    PERFORM 8000-CHECK-PCB1-STATUS
    PERFORM 3000-PROCESS-ADDRESS
END-PERFORM
.
.
.
GOBACK.



This may seem like a long explanation, but there is still much more to it,
but this gives you an idea of what is involved in IMS. Obviously you can do
inserts, deletes and replaces. As well as check points, for restarting the
program in the middle of processing after an abend or some other error. You
can lock segments by doing a GHU get hold unique, or GHN get hold next, or
GHNP get hold next within parent. Error checking is everything. Checking the
status code not only tells you about missing segments and other errors as
well as when the level of the PCB changes. Such as if you were to do the
above example with a GN instead of GNP which would allow you to jump to a
particular person then process the addresses of everyone following. The
status code would change when you jump from ADDRESS to PERSON and back down
to the next ADDRESS segment. Inside of the PCB is also what is called a
CONCATENATED KEY which has the keys of all parent segments and the current
segment in one long string. You can also retrieve multiple segments with
each call such as retrieving a PERSON segment as well as the ADDRESS segment
with one call......

but those can be explained better by someone with more knowledge than I.

There have been a couple of good responses - and I think they probably meet
what you are looking for, BUT I did want to add something that is rather
important to know.  There are actually two "flavors" of IMS  - what has been
talked about so far is what used to be called "IMS/DB" - or the database
flavor.  (And from the fact that you talk about DB2 and IDMS, this is
probably what you are interested in).  There is another flavor - which used
to be called IMS/DC (or data communication).  It is quite comparable to CICS
(and if you have done ANY online transaction programming with DB2 or IDMS -
if you don't know IMS/DC already, then you probably know CICS.  If you just
know "batch" then this is irrelevant anyway).

If you do know CICS, then you are probably already familiar with EXEC CICS
statements in a COBOL program.  Although
    Call "CBLTDLI" using ...
is probably the most common way of accessing IMS databases, there is another
way available in CICS environments and that is using
   EXEC DLI
statements,  The two are similar (but not identical) in function.  Depending
on what you want to learn for, you might want to look at both.

As with other posts, this is quite superficial, but might help in pointing
you in the right direction.

The is a Conversational feature within IMS/DC, but it is rarely used, as
the Scratch Pad Area (SPA) and the processing involved consumes
mega-resources.  Most programmers will instead, for example in multiple
screen data entry apps, try to hide data entered in the first screen on
the second screen.  Another technique is to use a work-in-progress data
base to store the data between screens.  Both approaches supposedly more
efficient than conversational transactions.

In a conversational transaction, the first 'Get-Unique' using the IO-PCB
will retrieve the input message (screen), and the second GU will
retrieve the SPA.

The SPA is a lasting work area which is separate per terminal and

> Long ago I just developed the habit of creating a work area database with

That'f functionally the same as what the SPA is.

>  That way, I could always keep  what  user A was
> doing separate from user B.

That's functionally the same reason as for using the SPA.

And then something about terminology:

pseudo-conversational programming means breaking up a (logivally designed)
user dialog in separate programs, each starting with reading in the data from
a screen (CICS: receive / IMS: Get), doing some internal processing and then
writing the next (or same) screen (CICS: send / IMS : Insert). This ensures
that no program is active (not consuming main memory space nor CPU cycles)
during the time that the user is "working with the terminal" for seconds long
(which could very well mean "drinking coffee", "scratching his back", "being
in a meeting").

In IMS, what is called "conversational programming" and using the SPA as
mentioned above is effectively the same as what the CICS manuals call
"pseudo-conversational" and using a TWA. (So far for the logic and
consistency of IBM terms in different manuals :-((  ).

There is a performance goal fos using the SPA: The file in which IMS keeps
those areas is much better accessed by IMS if the variation in length of the
SPA's is minimal, preferably no more than 4k f.e. I know a lot of sites that
standardize the SPA length to 4096 or some other 2power of kBytes.

The Frog
