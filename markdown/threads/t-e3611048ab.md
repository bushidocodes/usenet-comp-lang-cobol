```
Let me start this new (but follow-on) thread with my PERSONAL conclusion -
that there isn't any such thing as a "single mainstream" of COBOL usage
today.

I think that without much doubt the VAST majority of existing COBOL source
code that produces "production" object code in today's business and/or
personal use is on mainframes.  Furthermore, I would *guess* that there is
probably more IBM mainframe COBOL in production today (and being maintained
and enhanced) than all the other environments put together.  However, there
is *significant* COBOL production code running on HP, DEC, Unisys, Tandem
and a variety of "non-workstation" computers.

I believe that there are ANSI/ISO conforming ('85 Standard - not yet 2002)
compilers available for every "production" operating system (except the Mac)
in use today - this includes all releases of Windows,  all variations of
Unix and Linux, proprietary systems from IBM z/OS and OS/400 to "both sides"
of Unisys to HP to a variety of other systems.

Furthermore, I think that COBOL is a "miniscule" player in the "workstation"
(Windows and Unix) "production" environments.  *HOWEVER*, there is
production code available for a variety of applications on such platforms.
(Some using OO methodology - most not.  Some using add-on GUI products -
some not.  Some still in "character mode" - some not.)

For the "average" Western European or American, they may get their paycheck
created by COBOL, their insurance bills sent out and resolved by COBOL, and
even their online brokerage trades done in COBOL .

I believe that there is a MAJOR difference between
 online "mainframe" applications
     versus
  "mainframe" batch applications
     versus
  workstation user-interfaces
    versus
  networked transaction processing

Although I know that "GUI user-interfaces" *can* be done in COBOL (a variety
of dialects, software products, compilers, etc), I do *not* think that is
where *most* day-to-day COBOL programming is being done today (any more than
it was 10 years ago).  On the other hand, I think there IS significant COBOL
maintenance, development, and enhancements being done for all 3 of the other
types of application listed above.

My guess is that there is not 1 in 10,000 production programs that is
compiled with "ANSI/ISO conforming flagging" turned on.  If it were turned
on, the majority of such programs would show the use of vendor extensions -
some in medium-common usage (e.g. Comp-3 for "packed-decimal) some in
"isolated to single OS environments" (e.g. Line Sequential) - and some that
only exist for a single vendor (e.g. AcuCOBOL's GUI screen extensions).

I also don't think that this newsgroup or any "internet" or "web-based"
forum is very representative of the "majority" of those who make their
livings "doing" COBOL.   Therefore, I doubt that this thread will be any
more "definitive" than any of the previous rants-and-raves.

However, MAYBE this thread can be used to get those who are interested in
sharing their experiences and opinions to "qualify" them more accurately as
to their assumptions and target audiences.

Alternatively, if "you" want to address a wide spectrum of COBOL programmers
(new, old, and students), then restrict your discussion to ANSI/ISO
conforming source code.
```
