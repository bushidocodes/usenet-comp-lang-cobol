Another approach is the "de-editing MOVE".  The numeric value is consistent
with one that's been developed by an edited-numeric PICTURE string of the
form Z(6).99 or Z(6).ZZ, and any COBOL compiler compliant with the '85
standard should be able to handle a MOVE from such an edited-numeric item to
a numeric one.  This is particularly useful if you know *for certain* that
the value was developed using a COBOL PICTURE clause in the first place, and
what that PICTURE clause contained.

You started off saying you had a "file." It was poor design practice on the
part of the creator of the file to give you an amount with non-numerics.

Having said that, you are now faced with a poorly-designed input file and
must presume it contains raw, uneditted, and unchecked data.

That opens up a whole new avenue of investigation.

Does the field in question contain only digits (or can be made to do so), is
the value right-justified in the field and is the field left-filled with
zeros?

As a learning exercise, you can probably get away with assuming the field is
right-justified and contains only digits and spaces. So now your task
becomes one of left zero-filling.

The easiest way is:
   INSPECT amt REPLACING ALL SPACES BY '0'.

There are other techniques:
1.   COMPUTE amt-9 = FUNCTION NUMVAL(amt)

2. PERFORM VARYING MY-INDX FROM 1 BY 1 UNTIL MY-INDX > 9
         IF amt(MY-INDX:1) NOT NUMERIC AND amt(MY-INDX:1) NOT '.'
               MOVE '0' TO amt(MY-INDX:1)
              END-IF
      END-PERFORM

CoBOL compiler.   Either use the function NUMVAL, or move around characters

No.  Aligned De-editing moves are allowed in ANS85.  The only problem
with the original code is that the edited numeric doesn't cater for
leading spaces or floating negative.

If the picture of nAmt was -(6).99 or similar then as long as the
