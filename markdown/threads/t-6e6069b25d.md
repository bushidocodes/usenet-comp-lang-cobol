Variable length records on disk only take up the space defined by the size
required to hold the actual data, they save space.  However, in memory the size
of the FD area or Working-storage doesn't (can't) actually change and must be
as large as required for the largest record.  So there would be no 'saving' in
having the WS area defined as variable and it would also be more difficult to
manage programatically because the 'Number-Ordered' would have to be set
correctly before accessing the IBSNs.

> Why try to define ISBN twice??  

It isn't.  What is defined is an area (ISBN-INFO) which will contain all the
info on each book ordered, and there may be up to 6 different books.  Within
this area is the ISBN (ISBN1).  You have ommitted the '05' on the table:

Note that this order is for 1 copy each of up to 6 different books.  If two
copies of the same book is required then it would have to be done by
duplicating the ISBN to give 2 books.

I think that you are having problems with "tables" rather than with
"variable length files".

What is described in your record layout is a "table" with 6 *identical*
entries for six different specific books.

Have you worked with tables (OCCURS) outside of files yet?  If so, I think
that all you need to do is think of this as a "single" table including 6
entries.  If you haven't worked with other tables yet, then you might want
to look at that in your text book or course material before trying to work
with this "file" example.

It isn't defined 'twice', it is defined 'number-ordered' times, the first
book's ISBN would be in ISBN-INFO(1) in character positions 10-19, the second
book in the order would be in ISBN-INFO(2) in characters 20-29, the third book
ordered would be in ISBN-INFO(3) in characters 30-39 and so on.

Image going to the sales counter with a pile of books and they will create one
variable record with the number of books in 'Number-ordered' and each book in
the array of ISBN-INFO with its ISBN in ISBN1(n).
