```
OK but I haven't got N/E up and running. In M/F you specify your 
Alternate Keys and you can also specify Split Keys ( Split Key = D, G 
and A) - see the syntax.

Having specified your 'other keys' and there's some limit I expect, but 
not exhaustible in most situations, you get *I think* :-

Read Customer-File
Key is PrimeKey
Invalid Key..... etc

OR :-

Read Customer File
Key is Alternate-Key1   *> OR :- Split-Key-1
Invalid Key etc......

Don't go nuts on the Alternate keys - in some instances they will be 
numeric and you don't have much choice; however, if they are Descriptive 
fields, Customer-Type pic x(40), Credit-Rating pic x(10) etc., try and 
use mnemonics and do a separate look-up against a Descriptor file using 
the mnemonic when you actually need to print/display a description of 
what Customer-Type or Credit-Rating are. Obviously if you use the 
Customer-Name as an Alternate Key - not much you can do about that other 
than using the complete field. But if accessing on Province/State/County 
or Country use one of the standard mnemonics. Depending upon how 
'widespread' your application is you could even use your own  set of 
mnemonics to represent towns/cities with a Descriptor File look-up.

Of course there's a penalty to using Alternates - as you observed you 
finish up with a lot of cross indexing; I can't recall if it is in one 
file or one for each Alternate/Split-Key. Don't worry about that -  the 
M/F file handling controls that for you.

I kid you not - a really charming man, and some many years at Net 
Express had a problem with files used with a POS System. He had to 
expand the system for a client adding new merchandise departments or 
categories. His solution was to split the application data files - and 
had great problems with volumes splitting it. Again, I kid you not - 
assuming the data file was 1 Gig his index was 3 Gig ! From the size of 
the index he had to be using 'descriptive titles' to access on rather 
than mnemonics.

Pete's solution is still valid - if you want to pull out the records for 
say Customer-Type - I don't think it will necessarily give you Customer 
Type and PrimeKey in ascending order; you'll definitely get the 
Customer-Type in order but not necessarily the PrimeKey. Now if, and 
it's one more addition to the index, you have :-

	Split-Key-2 = Customer-Type + PrimeKey (A/c #)

You will get them in true ascending sequence.

With a READ NEXT If you are being selective on Customer-Type then you 
can conditionally action each record you receive, or ignore it; do the 
READ NEXT and repeat until EOF.
```

```
In the case of alternate keys with duplicates, then what do you do?

Option 1:
  If you know your records and when you do and don't have duplicates, it is 
POSSIBLE that the information you want from records with alternate duplicate 
keys is always the same in all such records.  In such cases, then you can get it 
from whichever record you have retried from the first READ ... KEYS IS 
statement.

Option 2:
  If the information you want from records with duplicate alternates keys is NOT 
the same in all such records, then the question arises as to HOW "you" know 
which is the correct record.  With an '02 conforming compiler or with many (not 
all) '85 Standard compilers - as an extension - you can use a "split key". If 
some combination of keys is guaranteed to get you the correct record, then that 
is why you should do (if split keys are available).

Option 3:
  If records with duplicate alternate keys do not all contain the desired 
information and the way to tell which record you want is NOT also a key, then 
you should:
 A) do a READ ... KEY IS statement for the alternate key in question.  This is 
GUARANTEED (in a standard conforming implementation) to get the "first" record 
with such a match.  If will (as mentioned elsewhere) give you a status code 
indicating that there ARE other records with that key.
 B) You should then do READ NEXTs against the file to obtain (in order) all the 
remaining records with that same alternate key.  You will then need to "look 
for" the data that tells your program that you have retried the correct record.

Option 4:
  If you have no way to determine which of multiple records with the same 
alternate record key actually contains the data that you are looking for, then 
you need to either redefine your specification or your file.  Results are 
unpredictable and highly likely to yield bad data.
```

```
OK but I haven't got N/E up and running. In M/F you specify your
Alternate Keys and you can also specify Split Keys ( Split Key = D, G
and A) - see the syntax.

Having specified your 'other keys' and there's some limit I expect, but
not exhaustible in most situations, you get *I think* :-

Read Customer-File
Key is PrimeKey
Invalid Key..... etc

OR :-

Read Customer File
Key is Alternate-Key1   *> OR :- Split-Key-1
Invalid Key etc......

Don't go nuts on the Alternate keys - in some instances they will be
numeric and you don't have much choice; however, if they are Descriptive
fields, Customer-Type pic x(40), Credit-Rating pic x(10) etc., try and
use mnemonics and do a separate look-up against a Descriptor file using
the mnemonic when you actually need to print/display a description of
what Customer-Type or Credit-Rating are. Obviously if you use the
Customer-Name as an Alternate Key - not much you can do about that other
than using the complete field. But if accessing on Province/State/County
or Country use one of the standard mnemonics. Depending upon how
'widespread' your application is you could even use your own  set of
mnemonics to represent towns/cities with a Descriptor File look-up.

Of course there's a penalty to using Alternates - as you observed you
finish up with a lot of cross indexing; I can't recall if it is in one
file or one for each Alternate/Split-Key. Don't worry about that -  the
M/F file handling controls that for you.

I kid you not - a really charming man, and some many years at Net
Express had a problem with files used with a POS System. He had to
expand the system for a client adding new merchandise departments or
categories. His solution was to split the application data files - and
had great problems with volumes splitting it. Again, I kid you not -
assuming the data file was 1 Gig his index was 3 Gig ! From the size of
the index he had to be using 'descriptive titles' to access on rather
than mnemonics.

Pete's solution is still valid - if you want to pull out the records for
say Customer-Type - I don't think it will necessarily give you Customer
Type and PrimeKey in ascending order; you'll definitely get the
Customer-Type in order but not necessarily the PrimeKey. Now if, and
it's one more addition to the index, you have :-

Split-Key-2 = Customer-Type + PrimeKey (A/c #)

You will get them in true ascending sequence.

With a READ NEXT If you are being selective on Customer-Type then you
can conditionally action each record you receive, or ignore it; do the
READ NEXT and repeat until EOF.
```
