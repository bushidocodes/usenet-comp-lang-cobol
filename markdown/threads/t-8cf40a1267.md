MCM:

Sir;

PIC[TURE] [IS] S9(5)V9(2) IS size 7. The sign is overpunched in the LSB (or
in the MSB, if SIGN LEADING is used).

PIC[TURE] [IS] S9(5)V9(2) SIGN [IS] SEPARATE [LEADING]  would be size 8.



USER:
This is true if LEADING or TRAILING sign is specified.  If it is not
specified where the sign should go then the sign will take up a single
character at the beginning of the data item.  This is certainly true for
Acucobol.

for example

pic s9(05)v9(02) would take up 8 characters.  It would only take up 7 if
LEADING or TRAILING sign was specified.

MCM:
... this behavior is not true for any ANSI-standard compiler.
Perhaps your compiler has some kind of compile-time switch which causes
a default behavior of "DISPLAY NUMERIC IS SIGN LEADING SEPARATE" ....


USER COLLEAGUE:

I understand that a colleague of mine has been emailing you about COBFD.
Specifically about the sign character within a PIC clause.  You were quite
correct, we are using a compiler switch which forces the sign to be
separate.  The switch we use is the -Cr  compiler command.  This is maintain
compatibility with the original code which was written over 10 years ago.
Rather than converting all our files to have the sign overpunched, it was
decided that it would be easier to just add the -Cr switch to the compile
script.

The -Cr Switch forces the code to RM/COBOL compatibility mode.  Maybe this
could be a future enhancement for the COBFD program.  I am sure that we are
not the only company that uses this switch.

=====

I had never heard of this kind of switch before.  As I said, you learn something every day.

AcuCobol has so many options that it is dizzying. During my brief
experience
with the language I found it to be so flexible - so readily tailored to
mimic the peculiarities of each specific implementation - that it is
probably
the most ideal solution for *quickly* porting existing code from one
platform
to another... certainly moreso than any other compiler I've encountered.

The downside is that it does tend to perpetuate any quirks that may
exist
in the codebase due to some previous implementation.  That certainly
seems
to be the case in this instance.  Working with it over time, one tends
to
forget what standard code might have looked like originally, and often
being faced with developing ways to "work-around the work-arounds."

Just a note about 'RM/COBOL compatibility mode':

As far as I am aware, all of our competitors offer an 'RM/COBOL
compatibility mode' which forces this, and other, behaviors.  This was the
behavior of the ANSI 1974-based RM/COBOL, available on many minicomputers in
the 70s and on PCs in the early 80s.  Liant Software Corporation no longer
offers this version of RM/COBOL.

The current version of RM/COBOL was previously known as RM/COBOL-85, to
distinguish it from its ancestor.  Interestingly enough, even RM/COBOL has
an 'RM/COBOL compatibility mode' achieved by a particular set of compiler
switches, one of which forces separate sign.
