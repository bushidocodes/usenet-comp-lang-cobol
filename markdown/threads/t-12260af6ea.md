Others have commented, but you may want some background.

COBOL started in the days of punch cards, hence the 80-column standard
(that's where we get 80-column screens also) (And the original size of
the punch card was based on the size of U.S. currency - in 1890 -
because all kinds of equipment existed to count and stack and
transport currency) (And the holes in punch cards were based on the
size of card punching equipment used in the textile industry governing
the weaving of designs. So, card-size/whole-size = 80 columns.)

If you ever drop a source deck of 3000 punch cards, you'll know
immediately why the cards should be sequenced, hence the sequence
numbers in columns 1-6.

Early assembly language (also on punch cards) had its sequence numbers
in 73-80, and COBOL had, at one time, the construct to suddenly shift
to assembly ("BEGIN ASSEMBLY" or some-such). So, apparently, columns
73-80 were reserved for that.

Word-wrapping editors, terminals, and newer languages with more robust
parsing compilers, without the above constraints, yielded more
'free-form' coding techniques.

Somebody thought free-form would be a good idea for COBOL.

They forgot to tell you.

Here's the deal: Skip 1-6 and 73-80. Use 7 for comments (an asterisk).
Put Division and Paragraph names (and a few other things like FD and
01) in column 8. Put everything else in column 12+. Indent by 4 (or 2)
columns for successive outline clarity.

Your code will compile no matter what the compiler settings or
standards of your shop.

P.S.,
There is never any need for a "Continuation Mark (dash)" in column 7.
Nor is there any real use for a "Debug Mark ("D") in a PC environment
because of debuggers built into the compilers.

OK, OK, I know what you want:

 "D" allowed you to salt your code with all kinds of diagnostics -
like printouts and program halts - and have that imbedded
investigatory material ride along with the code. The compiler treated
debug statements as comments: until the program was re-compiled with
DEBUG=ON, then you'd get all the author's original debug code to
assist in solving a current problem.

And another plug for "does your compiler vendor already support
'enhancements' from the draft Standard.

In the next COBOL Standard, you will be able to get rid of the "-" in column
7 for continuing alphanumeric literals.

Instead of coding

    Move        "abc...column 72
-      "continued stuff" to large-field

You will now be able to code

    Move        "abc...column 72"-
      "continued stuff" to large-field

You will also be able to use ">>D" in any column instead of "D" in column 7.

You will also be able to use ">>PAGE" instead of "/" in column 7

You will also be able to use ">*" instead of "*" in column 7

So *all* of the "column 7" (indicator stuff) will no longer be REQUIRED in
any program (whether it uses fixed-form reference format or free-form
reference format).  The "old" techniques will continue to be supported for
'fixed-form reference format' - but will NOT be included in 'free-form
reference format'

Check with your "vendor of choice" to see if they already support any/all of
these enhancements - and if not, when they will.

I am sort-of surprised that no one (so far) has mentioned ANOTHER way to get
around the "long alphanumeric literal" problem - in the '85 Standard.

Let's say that you currently have:

   Move           "ABC Column 72
-   " Some more stuff" to Long-Field

(where the '2' in '72' is in column 72 - no spaces following it)

then TODAY you can code:

   Move           "ABC Column 7"
           to Long-Field
   Move           "2  Some more stuff"
           to Long-Field (13:)

I am NOT saying that this is (necessarily) a "better" way of doing this than
defining "sub-fields" to "Long-Field" - but it does mean that if you have a
variety of "print-lines" with no COMMON LOGICAL breaking points, you don't
need to "create" artificial sub-fields.

(You can even use reference modification in conditional statements - if you
break up your "long literal" and do combined conditionals - such as
   If ((Long-Field (1:12) = "ABC Column 7")
           and
        (Long-Field (13:) = "2 Some more stuff"))  then ...
