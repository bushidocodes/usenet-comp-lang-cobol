There are three basic models of database:

In order of "evolution"...

1. Hierarchic   (Original implementations of IMS, for example)
2. Network     (Original implementations of ADABAS and IDMS, for example)
3. Relational    (Almost ALL modern databases, and the above mentioned have
been updated to support this model.)

"Network" does NOT mean running on the Network (LAN/Intranet/Internet)
because all of them can do that, it means the model uses Owner/Member sets
in a networked collection that is not necessarily Hierarchic (although it
often is, in practice).

So... RDBMS =  Relational Database Management System, NDBMS = Network
Database Management System.

Given that all business data interrelates in a way that is a network
then when designing for a hierarchic database there had to be
mechanisms designed into the _programs_ to overcome the 'single owner'
implementation limitation of these. This meant that, eg, an order
system may be designed as customer->order->orderline but the program
then had to collect the product code from the line and go to the
product->stock->.. hierarchy.

Whereas in a network model there would already be a database link to
the product 'owner' of the line (or should be).
 
Certainly a network model must degrade to several hierarchic models
and these would have 'soft' links between them.  But what particular
'attributes' would be lost in doing that ?

> Disagree again. That is an implementation-specific consideration (and it is
> true, for the most part). The "real difference" is that the underlying
> mathematical models are different.

Given that all business data has a 'network' design to it then a
network database schema and a relational database schema would be the
same. Well actually, in practice there may be issues with many-to-many
relationships but I have always found they invariably resolve
themselves meaningfully.

With a hierarchic db some of the relationships can be hard pointers
(and thus fast) and some have to be 'soft' and therefore programmed
around.

With a network db all relationships can be hard.

With a relational db all relationships are soft, but the db system
provides mechanisms which emulates hard pointers, such as by using
constraints.

In terms of the data models there is no need for there to be any
difference.  However the relational db _imposes_ restrictions that
would otherwise be sensible anyway, so that it can make assumptions
about how data can be retrieved.

Given a fully normalised data model the only difference between the 3
types of database _is_ the implementation.

Of course the relational db is the only one that imposes this, so
typical hierarchical dbs may have a data model that won't map to a
relational db cleanly.

Seven years of trying to get a data base feature in COBOL had 
to consider the implementation method at the time. There were 
then, as now, constraints on resources that demanded methods 
that provided the means to describe the relationships between 
and among file/records, pages, blobs. Those features may not 
be widely used, but they supported the needs of the systems 
and hardware of the time in a way I called list processing 
until Wm Ollie introduced me to the more general term, Data 
Base. C. Bachman's iinvention on how to do these things with 
the resources available at the time, got him an ACM award, and 
put the subject in everyone's face. Prior to that time 
versions of Indexed sequential were the most common methods 
outside of sequential except in simulation software. GE, 
allowing him to
offer this development into the public domain, started the 
work to prove the IBM method based upon a publication. IDS was
the first way a Relational Database was implemented even prior
to the IBM system. IBM at the time offered a book on how to
program your own Bill of Materials type application. At the 
time, as I recall, the 650 disk system was the main mass 
storage device in commercial use. Systems being sold in the 
marketplace then, were just beginning to have disk packs, and
their primary use was not the storage of files.

I'm not that familiar with the current methods used to create a
data base in various software products. However, it's just a
mater of the degree of indexing used wither that be links 
embedded into records, or as separate records.

Todays state of the art allows you to carry a lot of data in 
you pocket, and we are still not really to really backup for 
long periods of time. In my view, perhaps todays methods are 
the Tin Lizzy's of file management.  Some of you may see much
more advanced methods. I believe in KISS. None of this is SIMPLE.

Warren Simmons
