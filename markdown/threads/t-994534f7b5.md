Slight confusion of terms - NT doesn't support real VSAM, and OS/2
VSAM is a completely different animal than Mainframe VSAM. (VSAM,
real VSAM, on a PC is always remote. Local indexed files are often 
reffered to by careless people like me as 'VSAM', but they are not.)
CICS under NT either uses DB2, which has its own mechanism to handle
EBCDIC informtion, or uses Encina SFS, which again, has it's own
method to handle EBCDIC.

However, from the COBOL point of view, the compiler handles local
files in EBCDIC format quite easily, especially with CICS. 

-Paul

Yes, consider it confirmed. To obtain EBCDIC collating sequence use in 
your programs, use the compiler directive

     PROGRAM COLLATING SEQUENCE IS EBCDIC 

or

     COLLSEQ(EBCDIC)       

as a compiler option.

PC's of course, want to hold integers in a format that is byte-reversed
from the mainframe. (Mainframes are BIG-ENDIAN).  For programs that
will use Mainframe binary data and rely on a mainframe internal
representation of integer values, use the BINARY(S390) compiler
option. (Note: Don't use COMP-5 in this case! You will get   
a native binary format, whatever that may be at the time...) 

Finally, PC's represent floating point numbers in IEEE format,
while mainframes use Hexadecimal. Use the FLOAT(S390) compiler
option to ensure floating point numbers are in mainframe
format. 

File status values can be a bit of a problem, since values in
