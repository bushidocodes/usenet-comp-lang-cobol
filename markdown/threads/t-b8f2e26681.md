Christie,

There is NO "CASE" statement in COBOL.

The equivalent is "EVALUATE" as follows:
[stuff in square brackets are comments and mustn't be coded...]
...
Working-Storage Section.
        01  Variable-in      pic 999.
               88 Gets-nothing       value 1 thru 100.
               88 Gets-10-PC        value 101 thru 500.
...
Procedure Division.
...
            evaluate TRUE
                when gets-nothing
                     move zero to discount
                when gets-10-PC
                     move 10 to discount

[If Variable-in was only allowed to have values in the range 1 - 500, you
could validate this at the same time by using the WHEN...OTHER clause of
EVALUATE. Like this...

                when other
               [carry out exception notification]
]
            end-evaluate

[As the range of values you are interested in is suitable, and COBOL is
intended to be a self-documenting language, it makes sense to use 88 levels
as described above. If the ranges were not so convenient you could test
Variable-in directly with a WHEN clause (when Variable-in NOT > 100...etc.).

You are probably wondering why Variable-in is tested with WHEN instead of
with EVALUATE. Why not write "Evaluate Variable-in"...?

The answer is because, if you Evaluate a variable (dataname in COBOLspeak),
you cannot provide a relational operator (like >, = , etc.). You are only
allowed to provide tests for specific values of the variable. Let's say
Variable-in actually contains a "10" for 10%, a "20" for 20%, a zero if
there is to be no discount, and all other values are invalid. Now we can
evaluate the variable directly, like this...

           evaluate Variable-in
                 when zero
                         move zero to discount
                 when 10
                         move 10 to discount
                 when  20
                         move 20 to discount
                 when other
                 [Do the exception for invalid input...]
           end-evaluate

Summary:

Use "EVALUATE TRUE" when you want to test relations with a variable.
Use "EVALUATE dataname" when you want to test specific instances of the
variable.

Once you feel comfortable with all this, read the manual for the use of ALSO
which allows several variables to be included in the EVALUATE.

Pete.

What Peter's discussing here is that COBOL insists that you evaluate &
test (WHEN) items of the same class. An 88-level is basically a
TRUE/FALSE situation, whereas ZERO, +989, etc., are not. So, if you want
to evaluate a data field you must code values for the field in the same
class. If you want to use 88-levels, start with EVALUATE TRUE... Note
that if you EVALUATE a data name, that's the only thing (for lack of a
better word) considered, whereas, if you code EVALUATE TRUE you can code
tests of various fields, 88-levels of different fields, etc.

I think Peter's mistaken when he states "You are only allowed to provide
tests for specific values of the variable". We may code WHEN's with the
THRU option, e.g.:

EVALUATE  VARIABLE-IN
WHEN  +10  THRU  +99
	imperative statement(s)
WHEN  ZERO
	something else
WHEN  +100
	...
WHEN  OTHER
	...
END-EVALUATE

Bill Lynch

A) The draft of the next Standard does explicitly allow you to put the "to
be tested" variable in the EVALUATE line, and the rest of the conditional

B) Check YOUR specific compiler.  Several compiler vendors have already
introduced this (oft requested) feature as an extension to their '85
Standard compilers.
