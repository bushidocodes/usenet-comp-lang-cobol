1- If your compiler vendor has implemented the new standard (partially
or fully) then you have a "exit paragraph" instruction which does

"If an EXIT statement with the PARAGRAPH phrase is executed, the EXIT
statement causes control to be passed to an implicit CONTINUE
statement immediately following the last statement in the current
paragraph."


2- Use if's

  if not-error
     call xx
     if ret-code = "bad" set erro-found to true.
  end-if
  if not-error
     call yy
  end-if

3- Use perform thru with GOTO paragraph exit
E.G.
perform aa thru aa-exit.

aa.

  call xx
  if ret-code = "bad" go to aa-exit.
  call yy
aa-exit.
  exit.

"jce" <defaultuser@hotmail.com> wrote

> In my time I have met people who HATE HATE HATE GOTO and make it a goal to
> remove them all in COBOL....then will go into a C/Java program and routinely
> type return write in the middle of something.
> I've never understood why it is "acceptable" in one and not the other....

Because it isn't the 'go to' that is the problem.  A 'go to' or a
'return()' at the point they are written is no problem at all. They
clearly indicate what will happen.

The problem occurs because the go to must have a label as the target.

When inspecting a program it must be viewed in small pieces at a time.
 A properly structured program aims to have the essential parts of the
program correctly understandable by looking at the minimum amount of
code, thus being maintainable in the minimum of time.

A label can have several types of logic flow around it:  It can be the
target of a PERFORM, it can terminate a PERFORM or ~THRU, it can be
dropped through or it can be the target of a GO TO.  Because labels
can be paragraph names or section names there is an added complexity
if both types occur.

When inspecting a program and we want to understand a piece of code
that contains a label, we must determine all the ways that the logic
flows through that label.  In the case where the program contains GO
TOs or THRU then we must examine the _whole_ program to determine how
the logic flows around each label.

In a program which has no GO TO, no THRU, no SECTION labels and only
does PERFORM paragraph-label, then it is only necessary to ensure that
the initial paragraph does an EXIT PROGRAM/STOP RUN/GOBACK and then
_every_ label in the program can do just one thing.  It cannot be the
target of a GO TO, it cannot be a dropthru, it can only stop the
perform of the paragraph above and be a PERFORM of the code below
until the very next label.

Given those conditions then inspecting small parts of the program is
sufficient because each part is isolated.

Other languages achieve this by having different types of labels which
can only be used in a single way.  You cannot 'drop into' or 'drop
out' of a C function, or 'goto' it.  You cannot perform or call a C
label, it cannot act as a silent return().

Getting rid of the GO TO isn't about the GO TO verb, it is about the
labels and how they are used.

"The purpose of the archaic language element designation is to discourage
the use in new programs of some features that are unreliable, poor
programming practice, or ill-defined -- where better programming techniques
are available in standard COBOL. These elements are classified as archaic
rather than obsolete because their use in existing programs might be too
extensive to warrant removal in the next revision of standard COBOL.

Archaic language elements are intended to remain in the next revision of
standard COBOL. There is no plan for the removal of archaic elements;
however, should their use in program inventories become insignificant, they
may be considered for designation as obsolete by future architects of
standard COBOL.
   ...

3) NEXT SENTENCE phrase in the IF and SEARCH statements. This phrase can be
confusing, especially when specified in a delimited-scope statement. It is a
common belief among users that control is transferred to a position after
the scope delimiter rather than to a separator period that follows it
somewhere. In addition, it is a common source of errors, especially for
maintenance programmers who inadvertently insert a period somewhere before
the actual terminating separator period. The CONTINUE statement and scope
delimiters can be used to accomplish the same functionality and such
