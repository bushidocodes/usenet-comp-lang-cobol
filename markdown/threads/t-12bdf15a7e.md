The following comments are based on COBOL for MVS and VM and COBOL for
OS390 (IBM 370/390/z series computers).  I did a fair amount of
testing to see what the results of operations were using TRUNC(OPT).
In general if all operations involved literals and USAGE COMP/BINARY,
picture determined whether half word, full word or double word fields
were used and truncation was at the word boundary.  Thus if a S9(8)
BINARY field contained a value of 1.23 billion and it was moved to
either a S9(9) BINARY  field or a S9(10) BINARY field, the result
field would contain 1.23 billion.  If any of the fields was implicitly
or explicitly character or packed decimal (including the result of the
DISPLAY verb), picture would be honored for the operation so that the
same move to either S9(9) PACKED-DECIMAL or S9(10) PACKED-DECIMAL (or
a display of the S9(8) BINARY field) would give a result of 30
million, not 230 million for the S9(9) field or 1.23 billion for the
S9(10) field.  Thus to display the contents of a field S9(8) BINARY,
first move it to a S9(10) BINARY field and then display that field.  I
have been given to understand that code generation for ENTERPRISE
COBOL hasn't changed but it is probably worth doing some
experimentation.  By using some care and understanding the data, it

The current IBM recommendation is to use TRUNC(OPT) and *if* you have any 
specific fields that MAY get values larger than the PICTURE clause, then use 
COMP-5 for them.

As far as the "rationale" behind the (AWFUL) code generated by IBM for 
TRUNC(BIN) (rather than "just" doing normal BINARY assembler) is probably not 
something that you will find anywhere.  They originally (as I recall) *tried* to 
get it to as "close to" ANSI/ISO conforming as possible.  This always seemed 
spurious to me - as TRUNC(BIN) is, by definition, non-Standard.
