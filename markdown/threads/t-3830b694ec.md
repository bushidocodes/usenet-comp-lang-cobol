This is more of a system level issue. 

In your COBOL programs that must be linked AMODE24, use a
CBL statement to force the issue. Do not use AMODE/RMODE
with the Linkage Editor (BINDER). This way, it will not
override the requirements of source that has to exist below
the line.

Now why would the code require AMODE 24? Would it be because
of subroutines that are called that are written to only
handle 24 bit addresses?

At anyrate, with regards to the COBOL and BATCH issues - you
are better off by having your programs not care if they are
RMODE 24/31, but to be RMODE ANY. This will allow them to
load above or below the line. Next is, if you are using VSAM
data sets, then not being fixed to below the line storage
(AMODE 24), allows you to make use of above the line
buffering (AMODE 31), which can improve your elapsed times
on your batch JOBS.

Other than the PL/1 question, I have just done such a
migration from VSE to z/OS 1.1 wherein some of the BATCH
programs required AMODE24/RMODE24. And this was specifically
because the COBOL code was poorly generated by an RPG to
COBOL translator. As a result, a special subroutine had to
be written that the COBOL program called (it contained ESTAE
to catch and fix S0C7 ABEND problems that RPG automatically
resolved in VSE).

More than 90% of the BATCH COBOL code did not care where it
ran, and as a result, the use of HYPERBUF (CA buffer control
product) allowed for better handling of HUGE VSAM datasets)
was enhanced as it did not have to move buffers below the
line (24 bit addressed storage).

I hope this helps you.

Your points where there needs to be clarification:

2) The ALC subroutines need to be looked at for more than
BAS. BAS/R and BAL/R both work the same in 31 bit mode
(PSW:32 = 1). However, you are correct in your idea of using
BASSM to force yourself into 31 bit mode. By doing this,
your code can be RMODE ANY and AMODE 31. Therefore, once you
have fixed these, placing an RMODE ANY statement close to
the START or CSECT statement (which should begin the
subroutine) will tell BINDER what it needs to know during
link edit, and/or LOADer should the subroutine be
dynamically invoked.

3) As long as your PL/1 code and COBOL code are not compiled
with a below-the-line residency requirement, then the code
generated should handle any type of addressing (24 or 31
bit).

4/5) Again the subroutines must be passed parameter lists
that do not contain addresses (call by reference) where the
high-order byte is corrupted by flags. They must be passed
pure addresses. If this is true, then the low level routines
can easily be made RMODE ANY.

Also, any subroutine that is doing I/O, other than VSAM,
must be modified to GETMAIN the area for the DCB below the
line (24 bit storage). OPEN/GET/PUT/CLOSE can all exist
above the line, but the DCB itself must exist below the
line. 

VSAM I/O can reside anywhere. So subroutines that are only
doing VSAM can easily be fixed for RMODE ANY, AMODE 31.

For your ALC questions in this area, you should probably put
them on comp.lnag.asm370. There you will get very serious
answers and sometimes code examples. Accordingly, this post
has had comp.lang.asm370 added to it.

Jose Luis,
My suggestions are as follows:
1.  Locate all of your subroutines.  If you have Assembler Language subroutines,
either upgrade them 31-bit addressability, or convert them to COBOL, so that you
can use 31-bit addressing as a compile / program bind (link edit) option.  Many
things we used to need to do in Assembler can now be done in COBOL using POINTER
data items, etc.  If you have subroutines in other languages besides COBOL, then
compiling them with an LE-compatible compiler should simplify interlanguage
communication.  (My shop didn't have this issue.)  The COBOL subroutines should
be migrated from VS COBOL II to COBOL for OS/390 & VM or Enterprise COBOL, so
that you'll be able to make them 31-bit addressable when you want to or need to.

2.  Locate the subroutines which do not do any CALLs to other subroutines.
These are "lowest level" subroutines, and they can all unconditionally (I think)
be made 31-bit addressable, because anyone can call a 31-bit routine (but only
24-bit code can call 24-bit routines).  Once all of these subroutines have been
upgraded, go through the rest of the subroutines to locate the next lowest level
(the ones which call only the subs you've alreay upgraded).  Bring those up to
31-bit addressability.  When you are finished, you can compile and program bind
any main program as 31-bit, without fear that it will fail because of mixed
address modes.

3.  Start on your main programs, using the same strategy, and soon (maybe not so
soon) you'll be all 31-bit.  This will let you take advantage of some LE
performance enhancements.

You must go to the IBM COBOL Web Site and get the Migration Guide for COBOL and
the one for PL/1.  These manuals will explain how you can get to Language
Environment.  The use of the RESIDENT / NORESIDENT compiler option for VS COBOL
II is an issue in accomplishing the migration.  My shop had been using RES for
both OS/VS COBOL and VS COBOL II for over a decade, so we had few load modules
that needed re-binding (relinkediting).

If you are using VS COBOL II as your current COBOL run time, go ahead and test
replacing that run time with Language Environment, to see what problems you
have.  They may be so small that you can do a "big bang" move to LE; they were
for my shop.  I do not know how easy or hard PL/1 is, because we have
practically no PL/1.

There is more to consider, but time is up for me today.  Your best chance of
success is to get busy as soon as possible!

Buena Suerte!

3)  Look into the Edge Portfolio Analyzer (or other products if you can find

 http://edge-information.com/epa-00.htm

4) Make certain that you verify that all your "development tools"
(debuggers, dump analyzers, etc) are at the appropriate levels to support

true for any "outside" provided applications run in your shop.

 ... After this preparatory work - I have a few suggestions:

A) *Never* override the link-edit attributes for COBOL programs.   If you
set the compiler options correctly (for what you want), then "take" the
linkage editor (or binder) options that are produced.

B) Move to an environment where LE is the *only* run-time library in use
(Linklist, Steplib, etc)

C) *After* you have killed off all older run-time libraries, then make
certain that you move to LE as the only link-edit library in use.

D) *AFTER* you have moved to LE as the only run-time and link-edit library,
then use RACF or similar facility to insure that only the "newer" compilers
are used for maintenance and development.
