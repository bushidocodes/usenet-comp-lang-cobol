```
Preface,
   I am NOT advocating the use of NEXT SENTENCE in new code. However, given
several recent threads, I thought I would pass this along.

It is absolutely true (as pointed out in a previous thread) that the current
(and draft) Standard explicitly prohibit the use of NEXT SENTENCE  in an IF
statement that is concluded by END-IF (also true for SEARCH and END-SEARCH).
This means that the following code (although allowed as an extension in many
compilers) is "non-conforming" source code.

If whatever > something
     Perform aRoutine
Else
     Next Sentence
End-If
Perform anotherRoutine
  .

However, as I pointed out to J4 in an interpretation request (which they
confirmed was true), the following is fully conforming source code (with an
explanation following it)

If whatever > something
     Perform aRoutine
Else
     Copy NextSentence.
End-If
Perform anotherRoutine
  .

Where the NextSentence copy-member includes the following code (which is
ALWAYS logically equivalent to a "NEXT SENTENCE" statement)

Move 1 to Always-one
If Always-one < 2
    If Always-one = 1
         Next Sentence
    Else
         Continue
Else
    Continue
End-If

OK, you may ask WHY is this "legal" (ANSI conforming) source code. The
reason is that the Standard states that an "ELSE" always terminates an
"internal" conditional statement (IF or otherwise).  The Standard also says
that you can't use an END-IF to terminate an IF with a NEXT SENTENCE but it
does NOT state that you can't use an END-IF to end an IF that has a nested
IF with a NEXT SENTENCE.

FYI, I came up with this technique (and interpretation request) when I was
trying to convince ANSI to remove the restriction against END-IF with NEXT
SENTENCE in the next Standard.  I think it is silly that the "logic" can be
included in a conforming source program when the code cannot.  However, J4
(and/or WG4) disagreed and the restriction is still included in the draft of
the next Standard although many (most?) compilers don't enforce it.
```

No, it is actually required (and can easily cause confusion).  A "correct"
COPY statement *must* end with a period (full-stop) - but the entire COPY
statement (and any REPLACING - along with REPLACE statements - which also
need periods) are fully "resolved" before any syntax analysis is done.  I
know that this looks AWFUL to have a period in the middle of a statement -
but 'tis the way it works!
