```
Chuck Stevens argues at length that, because the COBOL68
and COBOL74 compilers treated the ZZ in ***.ZZ as fixed
insertion characters, that it was proper to change the
COBOL85 behavior to match as an FTR correction, without
warning.

I see several flaws in Chuck's arguments:

  1) In fact COBOL74 did not act as Chuck implies.

  2) In fact the COBOL74 compiler still (as of 44.2) does
     not act as Chuck claims.

  3) The 44.2 COBOL85 compiler does not act as Chuck claims.

  4) The change altered existing compiler behavior suddenly
     without warning to users, with the result of altering
     existing user program output to be incorrect but
     without a syntax error or program fault.

It would be much easier to agree with some of Chuck's
points if his premises were correct. In particular, there
are good reasons, based on long asserted user desires, to
keep features from the old compiler in the new one, even
when those features are less than ideal. Users don't want
to do a conversion just to stay with the A-Series systems.

Below, I'll detail the differing behaviors. I'll address
point 4) first, as it's the critical one in my view.

The COBOL85 compiler is no longer a new product. It's been
out for at least five years, more I think. Many new
programs have been written using only COBOL85, and many
programs converted from COBOL68 and COBOL74 have been
extensively modified using COBOL85. It's no longer accurate
to think of COBOL85 as being used mainly to compile
converted programs -- it's a primary compiler in its own
right.

Therefore it's no longer acceptable to make sudden changes
to COBOL85 just to conform to COBOL74. Such changes are
major anathemas to users, because they require sudden and
unplanned expenditures of manpower to alter user programs.
This is the reason that Unisys, for the A-Series, has for
many years guaranteed two releases of warnings for
deimplementations, including major changes of behavior.
This change came with no warning, not even as part of the
release which implements the change. That is, the compiler
does not flag the syntax in question with a "warning:
meaning of insert characters is changing on 44.2, see note
blahblah".

Worse, it altered existing program behavior with no
indication. The altered compiler produces no warning on the
item whose behavior has changed. The program which no
longer works correctly gets no syntax error, no fault,
nothing to indicate a change except incorrect output. While
we would complain about a sudden syntax error or fault,
neither is as bad as suddenly and silently receiving
incorrect output.

Based on Chuck's attitude, it's obvious that large parts of
software engineering still do not understand how damaging
such changes can be in a production environment..

OK, now for the technical issues. Chuck said these changes
were made "for 44.2". Randall Bart says "this behavior was
introduced in patches late in 1998". In fact, the 43.2 and
44.2 COBOL74 compilers act exactly the same in this
respect, but the 44.2 COBOL85 compiler exhibits the new
behavior. I don't know whether COBOL74 patches have been
released post-44.2 to change its behavior.

Consider this program fragment:
     02 EDITED-VAL  PIC ***,***.ZZ.
     02 DATA-VAL    PIC 9(6)V99.

     MOVE DATA-VAL TO EDITED-VAL.

Let's compare the behaviors of old (43.2) COBOL74 and
COBOL85 with the new (44.2) behaviors.

  Old COBOL74 treats the ZZ as though it were 99, except that
  the entire field is blank if the source is zero. Chuck
  implies, but does not directly state, that COBOL74
  previously treated the ZZ as fixed insertion characters.
  What he actually says is that this behavior "was carried
  intact into COBOL74" from COBOL(68), implying that it's
  always been that way. I did not try to investigate farther
  back than 43.2, but it clearly has not always been that
  way.

  New COBOL74 acts exactly like old COBOL74: it still does not
  treat the ZZ as fixed insertion characters.

  Old COBOL85 I did not test, since everyone seems to agree
  that it treated the ZZ as though it were 99.

  New COBOL85 treats the ZZ as insert characters when the
  source is non-zero, but inserts ** when the source is zero!
  This flatly contradicts Chuck's assertion that the ZZ is
  now treated as simple insertion characters.

Thus it appears that the fixed insertion behavior has not
existed in any supported software for several years. Since
existing behavior is the premise of Chuck's arguments, I
can't accept any of his conclusions.

Here's the program I used to test:

  IDENTIFICATION DIVISION.
  ENVIRONMENT DIVISION.
  INPUT-OUTPUT SECTION.
  FILE-CONTROL.
      SELECT RM ASSIGN REMOTE.
  DATA DIVISION.
  FILE SECTION.
  FD RM.
  01 RM-REC.
     02 RM-SOURCE  PIC 9(6).99.
     02 RM-SPACE   PIC XX.
     02 RM-EDITED  PIC ***,***.ZZ.
  WORKING-STORAGE SECTION.
  01 WS-SOURCE     PIC 9(6)V99.
  PROCEDURE DIVISION.
  BEGIN.
      OPEN OUTPUT RM.
      MOVE SPACES TO RM-SPACE.
      MOVE 12345.67 TO WS-SOURCE.
      MOVE WS-SOURCE TO RM-SOURCE RM-EDITED.
      WRITE RM-REC.
      MOVE 0 TO WS-SOURCE.
      MOVE WS-SOURCE TO RM-SOURCE RM-EDITED.
      WRITE RM-REC.
      MOVE 0.01 TO WS-SOURCE.
      MOVE WS-SOURCE TO RM-SOURCE RM-EDITED.
      WRITE RM-REC.
      CLOSE RM.
      STOP RUN.

Edward Reid
```

```
expanded to allow - as an insertion character if it wasn't a leading or
trailing sign, and in adding -, + and other things were added.  But I
don't believe Z, 9, $, *, X, or A were ever insertion characters.
```
