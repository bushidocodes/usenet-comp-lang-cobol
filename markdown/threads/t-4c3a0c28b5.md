*Actually* one of the major (in my opinion) changes in the 2002 Standard is
"C-ification" i.e. the ability to create COBOL code working with all (most?)
of the documented API's that "assume" a C-type environment.  Although many
(most?) vendors provide some or all of the following extensions to their '85
Standard compilers, I view the following enhancements in the 2002 Standard
as FINALLY allowing ANSI/ISO conforming COBOL to "play" in common "modern"
API environments:

 - CALL by VALUE (as well as CONTENT and REFERENCE)
 - RETURNING phrase (on CALL or Procedure Division)
 - Pointer and procedure pointer data types
 - non-intrinsic function "calls"
 - recursion
 - "true" binary fields (i.e. fields not truncated to mod-10/decimal values)

Without these (and some other)  "language" features, I disagree with Michael
that '85 Standard (or '89 Standard) COBOL can "easily" be used to implement
many/most of the API's that exist for Windows or other API-documented
interfaces.

The Windosw API was originally developed using the Pascal calling 
convention rather than the C calling convention. The main difference is 
whether the caller or the called routine does stack tidying.

MS first worked on putting applications onto the Macintosh, this had been 
developed for Pascal.  At the time that MS started writing Windows 1 it had 
Microsoft Pascal, but no C, so both these led MS to make the API as Pascal.

Later, MS bought in a version of Lattice C which they called MSC 4.0.

I have "C Programmers Guide to MS Windows 2.0" 1988:

"The following are required features of a Windows C compiler:
  
  * Must support the NEAR and FAR keywords.

  * Must support the use of the PASCAL parameter passing convention:

     void PASCAL MenuProc( HWND, WORD );

  This keyword indicates the arguments to a called routine are passed
  in reverse sequence to the function, and the function should "pop"
  the stack.
  This is the revesrse of the normal C sequence, where the arguments
  are passed in the same sequence and the caller pops the stack."
