```
Someone CORRECTLY indicated that the problem was the period after column 72.
Moving just the period to a new line would fix it - or you could add an
END-SEARCH statement followed by a period.

The reason that the older compiler handled this was that the next line was a
paragraph-header and the OLDER compiler often (not quite always) assumed a
period before paragraph headers.

Without the period (and without an END-SEARCH), the SEARCH statement is a
conditional statement- and is nested in an "AT END" phrase where an
imperative statement is required.
```

```
The use of NEXT SENTENCE has been made archaic
in the draft COBOL standard (CD 1.10, Appendix E.1, item 3).

"3) NEXT SENTENCE phrase in the IF and SEARCH statements.
This phrase can be confusing, especially when specified in a
delimited-scope statement. It is a common belief among users
that control is transferred to a position after the scope delimiter
rather than to a separator period that follows it somewhere. In
addition, it is a common source of errors, especially for
maintenance programmers who inadvertently insert a period
somewhere before the actual terminating separator period.
The CONTINUE statement and scope delimiters can be used
to accomplish the same functionality and such constructs are
clearer and less prone to error."

NEXT SENTENCE may become obsolete in the following
standard, 2006(?) or later, and may no longer be part of the
COBOL language in any subsequent standard, 2011(?) or later.
------------------
Rick Smith
```

```
The program I rewrote, about 5000 physical lines in the procedure division
before precompiling had scores of NEXT SENTENCE and 76 GO TOs (yes, I counted
them). It seems to me that NEXT SENTENCE  is simply a variation on GO TO.  But
beyond that NEXT SENTENCE works with periods and CONTINUE works with
scope-terminators.  

Taking one of our famous page long (multi-IF imbedded) IF statements all
terminated by a single period... A NEXT SENTENCE would pass execution to the
next executible statement immediately after that single period.  OTOH, if this
page-long IF were written with scope-terminators and CONTINUEs, a CONTINUE
would pass execution to the next executible statement immediately following the
scope-terminator of the imbedded IF that the CONTINUE is inside of; NOT to the
end of the entire page-long IF.

Finally, and I cannot emphasize this enough, the scope-terminator construct
SIGNIFICANTLY enhances program maintenance.  Adding, changing, and deleting
code with scope-terminators is magnitudes easier than the old sentence
paradigm.
```

```
Just so you see some of the confusion (for "real-world programmers") what
you say about CONTINUE is *not* exactly correct.  In practice, it is how it
is used (usually) but the following is perfectly valid code and the CONTINUE
does *not*  "pass execution to the next executible statement immediately
following the scope-terminator ..."

If A = B
   continue
   Read ABC
      At End
           continue
           Display "This will be executed"
    End-Read
    Display "This too will be executed when a = b"
Else
    Continue
    Dipslay "and this will be executed when A does not equal B"
End-if
Display "Nothing will pass execution directly here"
  .

All a CONTINUE statement does is act as a "no-op" and control passes
wherever it would go if there were no statement there.  There is actually a
time when code like this is semi-reasonable, e.g.

  Add 1 to A
     On size Error
         Subtract 2 from B
         Continue
      Not on Size error
         Display "This is part of the ADD not the Subtract"
   End-Add

It is possible to code an END-SUBTRACT statement where the CONTINUE is - but
some people and shops do not like using them with "unconditional"
```
