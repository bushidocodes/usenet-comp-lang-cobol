that was developed during the late Sixties and early Seventies.  The
super strategy is to break down this backbone system, converting it into
a variety of flavours of client/server packages on a fancy network of

that client/server is best and misunderstands what can be achieved by
mainframe systems.  That company is spending a fortune on an unnecessary
exercise and it is Joe Public who ends up paying in (in this case)

i took a course in assembler and one in C. i could get the
programs in C to almost work, and my programs in asm would
NEVER work. but the thing i noted most was that I could
port my C code easily to other machines. assembler would
require u to learn a new asm set, a new declaration, ect.

an example:

in openedition C, you could write:

register 
decimal (dollarsandcents,twodecimalplaces = 2) somenumber;

in asm, you would simply use registers 14 and 15 or something.

but if you wanted to port your code, then

in C, you simply change it to a float.

in asm, you would have to change your declaration, and
any code thereafter which references the packed data format.

lets pretend u have to keep your computer up 24/7...
(which is an ok assumption with micros)

in C, you could use TCP/IP easily, and abstract 1 library
from the 2 seperate machines.

but in asm, you would have to use two different libraries
and two different assemblers.

in asm, u are locked into the assembler-maker vendor. period.

in C, you have the option of changing compilers, and
abstracting the source code based on

#ifdef vendor_compiler
interruptfunction()
{
/*written one way*/
}
#enddef
#ifdef other_vendor
interruptfunction()
{
/*written another way*/
}
#enddef

can u do this in assembler? nope.

in C, the low level memory techniques can be abstracted,

by pointing to a pointer which is register class
for indirect addressing, pointing to a normal variable
for direct addressing, and pointing to the address of
an offset to a variable (or using an array) for 
other indexed addressing (or whatever the word is).

in asm, u have to rewrite *everything*.

before u port architectures, port your asm to C.

the one thing which is missing in C that asm has is
instruction cycle timing, but some C products even
handle that today.
