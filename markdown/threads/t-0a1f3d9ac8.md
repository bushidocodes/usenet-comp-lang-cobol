I think you are overstating your point here. Languages can and do
constrain the ability of the compiler writer to generate
efficient code. 

A case in point is my earlier example about the mandatory decimal
truncation in cobol. Another is the stricter aliasing rules in
Fortran which allows stronger assumptions to be made by the
optimiser - generating significantly better code in many cases
than C. Runtime type and array checking in Java as another
example. The rule that packed data items must be stored in a
constrained manner in COBOL.

The language can have another more indirect effect. If there is
less competition, then there is less pressure to produce better
code.

Tim Josling

Most compilers have licence conditions that prevent the
publication of benchmark results. I can tell you that 

1. COBOL compilers tend to be much (several times) slower than C
compilers for equivalent programs. This becomes a serious
problems for > 2000 line programs, depending on the speed of your
machine. This seems to be simply a result of a less competitive
market.

2. Code generated tend to be somewhat slower for COBOL than C,
due to language rules. For example, if you have a pic 9(8) comp
item, the compiler will truncate any results to 8 digits, whereas
in C, the equivalent 'int' will not require the extra code.

3. Packed decimal arithmetic is 20 times slower than binary
except on the mainframe where it is about as fast as binary.

4. Some cobols are interpretive, or use 'byte codes' like Java.
Acucobol for example. I would not advise intense numeric
calculations in a such a case.

What the heck, I'll give it try!

Interfaces are used to check the conformance of method
implementations. The use of interfaces provides for error
checking at compile time rather than run time.

An interface is similar to a class in that an interface may inherit
from other interfaces and a class may inherit from other classes.

It is different from class in that it only defines method interfaces
while the class defines factory and object data in addition to
complete methods.

One writes an interface by including one or more empty methods
in an interface program; that is, the method-name, linkage data,
and using phase.

One uses the interface in a class by including the interface-name
in the repository paragraph; then, after the factory or object
paragraph header, one references the interface-name in an
implements clause.

Finally, one writes the code that implements the methods that
were defined in the interface.

If there is any difference between the implementation and the
definition, the compiler will report the error. If no interface is used,
the error MAY be reported at run time.

Those who do high-reliability programming will use interfaces
and program and function prototypes extensively, as will those
who work with very large systems.

This, at least, is my understanding from reading the draft with
a few opinions added! ;-)

All this discussion on performance may seem interesting, but I think it
misses the point.  As Gunnar points out, there are other factors which are
just as important such as the skills of the programmer.  Others have pointed
out that the choice of the programmer to use or not use particular features
of a specific language also have a major impact.  When I was doing my
doctoral dissertation (see my home page for a link to it) on how we learn
what are sometimes the wrong ways to do things in COBOL, one performance
issue I specifically investigated was the maxim that one should use blank
comment lines (* in column 7) instead of blank lines because the compiler
uses less CPU time when it can stop scanning after column 7.  I ran some
tests of the same program with 10,000 blank lines vs. 10,000 blank comment
lines (easy enough to do with a good editor).  Well, the maxim is true, the
program with blank comment lines cost $.05 less to compile.  But then one
asks the question, "how much time does it take to tab over to column 7 and
type an asterisk 10,000 times?"  If it takes only 1 second, then the
programmer spends nearly 3 hours doing this task.  Even presuming that he
never makes a mistake and accidently types the asterisk in column 6 or 8,
then at $50/hour (including fringes), these asterisks cost $150.00.  Unless
we plan on compiling the program more than 3000 times during its life (an
average of once every workday for 12 years), we will never make up for the
few nanoseconds of time that the compiler runs quicker/cheaper.

Why are we so concerned about nanoseconds?  The true speed of a program is
usually not limited by CPU speed, rather we should concentrate our efforts
where it matters on such things as (1) readability of programs, since they
are usually read more often than they are written (an area where COBOL can
excel if done properly), (2) efficiency of I/O, using indexes instead of
sequential searches,etc., (3) ease of use, saving the time of both learning
and using by the user of the program, etc.  Nanoseconds might make sense in
calculation intensive applications like nuclear physics.  And COBOL might
not be the appropriate language to use in these problem domains either.

My rules of thumb are (1) choose a language that has features appropriate to
the problem domain, and (2) make efficient and effective use of the most
costly resources.  If I do those things right, then the nanoseconds usually
take care of themselves.

Look at the syntax between two languages.  Some things are just a
pain in the rear with a particular language.  What brought this to
mind is that I needed to test a display field for only alphas and
numerals.  (IBM VS/COBOL II R4.0 compiler).

The IBM hardware con do this in one instruction.

COBOL has a class test, but you can only check for ALPHABETIC
(A-Z), or NUMERIC (0-9 plus '-').  No ALPHANUMERIC class.  After
much through and discussion on coffee break, the best our group
of programmers could come up with is
    MOVE TEST-FIELD TO HOLD_FIELD
    INSPECT HOLD-FIELD
      CONVERTING 'ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789'
      TO HIGH-VALUES
    IF HOLD-FIELD IS EQUAL TO HIGH-VALUES THEN
        ....
    END-IF

In PL/I there is a ALPHANUMERIC class.
In C, there is strspn()
In BASIC, there is VERIFY$
In Rexx, there is DATATYPE() or VERIFY()
In Fortran, you must perform a character loop scan
IN S/390 Assembler, use the TRT instruction.

Now the question again: Which language will perform better?

S/390 Assembler and PL/I tie with 1 instruction.
C, in an optimal environment, takes 3 S/390 instructions
COBOL II, in an optimal environment, takes 3 instructions, but
those three will take longer to execute than those that C will
use.

Moral?  It's not just the platform, or environment, but also the
language structure/syntax.
