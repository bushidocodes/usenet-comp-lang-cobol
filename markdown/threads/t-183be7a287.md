If the file is an ordinary text file, read it like any other.

If the file has a companion .IDX file, the file is a MicroFocus COBOL
Indexed file. All indexed file formats are proprietary to the compiler
vendor and cannot be accessed directly. You must either write a COBOL
program to reformat the file in question to a text file or obtain a
third-party conversion tool (sorry, I don't keep up with these tools but I'm
sure someone else will help out here).

I believe that the MicroFocus's external file handler can be called from
another language to access MF datasets...EXTFH...If you have an MF compiler,
there should be documentation for using EXTFH...Basicly you would call EXTFH
passing an op-code and an FCD (File Control Description) area...

I've pasted some of the available op-codes for demo purposes...

Opcode  Operation                     File Types
------  ----------------------------  ----------
00      OPEN INPUT                    L  S  R  I
01      OPEN OUTPUT                   L  S  R  I
02      OPEN I-O                      L  S  R  I
03      OPEN EXTEND                   L  S  R  I
04      OPEN INPUT WITH NO REWIND     L  S
05      OPEN OUTPUT WITH NO REWIND    L  S
08      OPEN INPUT REVERSED           L  S

You can obtain samples of cobol programs calling EXTFH to access files from
a cobol program instead of the normal open, get, write, close, etc...Should
provide enough info for developing a module in another language to do the
same types of things...See the mf support web site for samples...

If you wish to actually use the index then you're probably out of luck
(anyone else here know of an add-in that reads isam?).  But converting
the data should be relatively simple.  

The .DAT file is probably fixed-length records with a 2-byte prefix on
each record.  The .DAT file itself has a 128-byte header.  I frequently
use Opt-tech sort to do this.

If the file is variable length records, there is a 4-byte header which
contains each individual record length.

I have used CObject control from Ratrix (http://www.ratrix.com/) to
read MicroFocus data files in a VB program.  It maps the fields and
creates some code for reading and writing records.  You have to have
the Cobol DLL's which you probably do if you have the files.

Another option would be an ODBC product for the files.  I tried
something from MicroFocus called "Correlate" a long time ago and if it
is still around it will work, but it was expensive (at least for me).
