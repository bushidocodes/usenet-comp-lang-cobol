Hi,

I have a NetExpress 3.1 server that uses WinSock calls so that
it acts as a server.   There are several 'control' threads
that handle GUI (panels2) as well as the TCP communications.
Mostly, though, there are 'worker threads' to do file i-o 
against MF indexed files.   There are a total of 50 files
and there's 1 file io dll per data file.

My problem is that it appears (very much so) that the CBL_THREAD
routine(s) have a problem that causes (I think) context switching
from one thread to another to be very resource intensive when
there are many threads loaded.

To test my 'problem' I have a client program that sends disk file
write requests to the server repeatedly as fast as it can...
My 'server' is a Dell dual 733 P3 machine with 500M ram, the 
'client' machine is a Dell 733 (1 cpu).

Here are a few examples of some oddities I've seen:

When running 1 client 'test' program I can achieve 400 i/o's per
second at the server and the server's at 40% cpu (dual cpu mode).

While this test is running, if I then start the remaining file-io
dll's the cpu usage goes up to 75% and the file-io's go down
to around 200.

The *problem* here is that the 'remaining file io' programs that 
I loaded all immediately do a CBL_THREAD_SUSPEND.

I've tested this every which way and everything points to the CBL
thread routines using significantly MORE cpu to context switch
when there are more threads loaded (even though 99% of them 
are suspended)...

Using Win API calls instead of the CBL routines for multithreading
appears, though, to work - there is only about a 5% reduction
in throughput with these routines vs CBL's - however - the use
of these routines appears to uncover some possible serialization
errors in the runtime.   I say this because when using the Win API's 
I get an RTS114 error in a CBL_WRITE_FILE call while no other file
io work was going on in the system.

Any ideas or clues as to possible compile / link / something options 
would be VERY much appreciated.


Thanks !


Ethan Schultz
