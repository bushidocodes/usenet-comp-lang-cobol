Stuart,

If this is caused by what I think it's caused by, it's addressed in
later Micro Focus versions. However, a technique I have had success
with is to write a small shell script called "ld" which is found on
$PATH before the real system "ld".

What the script needs to do is to use the arguments passed to it in
order to determine that it is being called by "cob" on the second link
stage (there are usually two) and to call the C++ compiler to do the
final link. If it's not the final link stage for "cob", then the system
"ld" should just be called instead, with the arguments unchanged.

To work out how to determine the 2nd link stage, write an "ld" script
which just prints it's arguments before calling the real "ld". You
should see arguments which both identify a COBOL system link and also
the second stage. Once your script can identify the second "cob" link
stage, it needs to translate the "ld" flags to the equivalent C++
compiler flags before calling it.

The problem is that both "cob" and the C++ compiler need to add some
information to the executable file prior to the final link, so you need
to give both systems a chance to do that.

Cheers,
Kev.

to

extern "C" {
int cobcall()
}

main()
{
cobcall();
}

that'll turn off the C++ name mangling.

You may get a leading underscore or trailing underscore
added (many C compilers do that) and "cobcall" will
remain lower case so if your Cobol compiler changes
linkage items to upper case then you need to use upper
case on the C++ side:

extern "C" {
int COBCALL()
}

main()
{
COBCALL();
}

Remember that at this stage it may still be changing it to
"_COBCALL" or "COBCALL_"
in either case you will have to account for the underscore
somewhere -- C++ compiler pragma's are your best bet


----------
Now once the names match, you have calling conventions to
worry about. Most C compilers (and C++ when using "C" linkage)
require caller to pop the stack (for stack based calling) while most
other languages have the callee pop the stack. C pushes arguments
onto the stack in one order, most other compilers the other way
around. C passes by value (default), with other languages it can
be by value, by reference, by descriptor (for strings and/or arrays).

Getting the external names to match so the linker can resolve all
references is most likely just the beginning of your Odyssey.

Mixed language programming can be a real walk on the wild side . . .

Hi. This confused me at first. Of course, you're quite right, but
your choice of COBOL name is unfortunate - "cobcall()" is an entry
point in the Micro Focus COBOL RTS which allows a *dynamic* call to
a COBOL routine (much like "CALL data-name" in COBOL).

As a general rule, I'd avoid writing programs with entry point names
starting either "cob" or "CBL_".
