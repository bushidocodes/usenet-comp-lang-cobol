Do a 1 instruction step at a time.  When the instruction(s) that
alters your variable is about to execute, go to Working Storage and
look at its value.  Then back to Procedure, let the instruction run
then back to Working Storage to see what the value of your variable
is.

Actually I confused CEDF with CA-Intertest.  Intertest can do 1
instruction at a time or more depending on how you set up the PF Keys.
CEDF, however, only goes from CICS instruction to CICS instruction.
However, you can still do a good job of debugging with CEDF.  The
trick is to put EXEC CICS ASKTIME instructions at points in the
program where you want to look at certain variables.  

Hope that's a little better help.

>Does anybody know how I can trace variable (within cobol statement)
>for COBOL/CICS programs which ONLY use CEDF utility, not any other
>third party software package ? 

Since CEDF only stops when a CICS command is about to be executed it is very
hard to trace a variable within a program. However if you code a command like
ASKTIME or some such that doesn't effect the main processing of the program
before and after the spot where you want to see a particular variable it is
possible to trace one through a process, tedious but possible.

Do me a favour.  Don't use EXEC CICS ASKTIME as an (artificial) break
point for CEDF and CEDX.  The command _does_ something, namely updates
the EIBDATE and EIBTIME fields, and does cost some CPU.  Use  EXEC
CICS SUSPEND .  This command does nothing  (it definitely doesn't
SUSPEND anyone/anything).  The reason:  I have seen way too many EXEC
CICS ASKTIME commands in production version of programs.  SUSPEN
doesn't hurt, ASKTIME may
