Suzie, the REDEFINES has nothing to do with it.  Data item MONEY-TEMP
is inherently alphanumeric because it is a group item.  The same goes
for data item MONEY.  Redefining a data item does not change that data
item in *any* way.   Redefining a data item provides a way to create
one or more *other* data items which describe the same memory location
in a possibly different way.  Above, when you refer to data items
MONEY-TEMP or MONEY, they are essentially identical.  If you refer to
data item CAL-MONEY, you refer to the exact same memory location, but
the description of how the item is treated (the PICTURE) is different.
When you refer to MONEY-DOLLARS you refer to the first four characters
of the same area as a 4 position numeric field.  When you refer to
data item MONEY-CENTS, you refer to the last two characters of the same
area, as a 2 position numeric field.

Usage specifiers at the group level specify the usage for elementary items
subordinate to the group item.  They do not change the usage of the group
item itself, or subordinate group items.  Group items are always inherently
alphanumeric.  If you specify a usage on an elementary item subordinate to
a group item with a usage, they must be the same.

You have received good comments already with which it is hard to disagree.
However, ...

I would like to add.

In modern COBOL X(6) fields (elementary OR group items) are naturally compared
with 9(6) fields.
This is 1998, you know.  Modern compilers will convert the ! X(n) ! field to
numeric by signing it (within a temporary area).

If you do an X(n) elementary field compare to an X(n) or a group level, then
the details under the group (REDEFINES or whatever) do not bring about a
conversion. A character compare is  permitted by the standard.

But if one of the comparands is numeric elementary, then the character field is
converted! and a numeric test performed.  This is 1998.

The X(n) field gets moved to a temp area and a sign (if necessary) is
constructed to match the presence or absence of the sign on the numeric
elementary item that determines the context of the compare.

As far as the other question that entered this thread. Specifying
PACKED-DECIMAL (or for ye olde folkes, COMP-3) at the group level is just
syntactic sugar. 

A group item is alphanumeric in type; 
end of story.

The syntax shorthand is specified at the group level, but applies to the
elementary items. But that is a very good question (Gold Star for Art).

Back to the real issue, ...
the redefinition of a 9(n) field as a S9(n) is not only confusing, it is
dangerous. It can engage many very esoteric issues depending upon settings of
the prefered sign compile option.

But more to the point of the original posters enquiry. There is a real strong
possibility of mis-compares.  The sense of the original enquiry was that
possibly the 9(n) versus X(n) distinction will cause a problem. Actually,
modern COBOL should cover that, unless you get a negative number in there, and
then only in that sense X(n) can never equal S9(n) where the S is negative.
COBOL will promote types to sign, it will not demote the types by stripping
signs.

The problem will be that in the situation where you have one defintion signed
and the other unsigned, the field clearly COULD get populated three different
ways: one unsigned, one signed positive, and one signed negative.  You are in
trouble with that if you compare X(n) group to X(n) group.

Unsigned will not match signed positive if you manage to trigger an X to X
compare. Unsigned will also naturally not match signed negative on an X to X
compare.

Focusing on the final digit of the fields:
  x'Fn' is not equal to x'Cn'
and
  x'Fn' is not equal to x'Dn'

The compiler will do what it can for you if you compare at the elementary data
item (it can not ignore the fact that negative is not equal positive).
But if you truely compare at the group level, you are definitely on your own,
as far as the results of moving to fields variously with their original
defintion, or with their alternate definition.

In short, the advice you have received so far, is good, although I am not sure
that each poster is aware that modern COBOL promotes data types to signed.  And
more to the point, you are into a world of unnecessary mysteries if you
redefine unsigned fields as signed.

You have not posted enough code for anyone to make a conclusion in this
specific case, but once you get used to all these shell games you become
somewhat jaded. Any definition of unsiged by signed is potentially suspect. The
remarkable fact is the the NOPFD (no prefered sign) actual scrubs the mistake
out more times than not. 

You can get into trouble with such an unsigned/signed redefintion under a (PFD)
prefered sign compile assumption even at the elementary level, and frequently
so when you do compares at group level that trigger X to X compare
instructions.

Best of luck with this, it is a valuable technical area to master thoroughly.

Sorry, but you've got this backward.

When a nonnumeric elementary data item or a nonnumeric literal is
compared to a numeric, the _numeric_ operand is treated as if it had been
MOVEd to a temporary elementary alphanumeric data item of the same size.

123456 is equal to 123456, and "123456" is equal to 123456.

0123456 is equal to 1232456, but "0123456" is not equal to 123456.

"123456 ", "123456  ", and so on, are all equal to 123456.

Furthermore, the sign is stripped from the numeric operand (not added to
the nonnumeric operand) by the standard CoBOL MOVE rules:

    $ copy sys$output demo.cob
    IDENTIFICATION DIVISION.
    PROGRAM-ID.  DEMO.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  FILLER.
        05  WS-X-6   PIC  X(6)  VALUE "123456".
        05  WS-S9-6  PIC S9(6)  VALUE -123456.
    PROCEDURE DIVISION.
    0000-MAIN.
        IF   WS-X-6 = WS-S9-6
        THEN DISPLAY "EQUAL"
        ELSE DISPLAY "NOT EQUAL"
        END-IF.
    9999-EXIT.
        STOP RUN.
    ^Z
    $ cobol demo.cob
    $ link demo.obj
    $ run demo.exe
    EQUAL
    $

When a group item is compared to a numeric, the numeric operand is
treated as if it had been MOVEd to a temporary group item of the same
size.  In this case, however, the sign is _not_ stripped by the MOVE
rules:

    $ copy sys$input demo.cob
    IDENTIFICATION DIVISION.
    PROGRAM-ID.  DEMO.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  FILLER.
        05  WS-GROUP-X.
            10  FILLER  PIC  X(6)  VALUE "12345O".
        05  WS-S9-6     PIC S9(6)  VALUE -123456.
    PROCEDURE DIVISION.
    0000-MAIN.
        IF   WS-GROUP-X = WS-S9-6
        THEN DISPLAY "EQUAL"
        ELSE DISPLAY "NOT EQUAL"
        END-IF.
    9999-EXIT.
        STOP RUN.
    ^Z
    $ cobol demo.cob
    $ link demo.obj
    $ run demo.exe
    EQUAL

OK, you've probably got this part, but it never hurts to begin at the
beginning.  All three of these fields are USAGE DISPLAY; MONEY-TEMP is a
group item, class alphanumeric.  The other two are elementary items,
class numeric.

There are two kinds of comparisons.  When both operands are numeric,
their algebraic values are compared.  However, when one or both of the
operands is non-numeric, each operand is considered a string of
alphanumeric characters and is compared according to the program's
collating sequence.  In other words, a PIC 9(6) can be equal to a PIC
X(6) in a non-numeric comparison.  For example, try this:

    $ copy sys$input demo.cob
    IDENTIFICATION DIVISION.
    PROGRAM-ID.  DEMO.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  FILLER.
        05  WS-PIC-9-6  PIC  9(6).
        05  WS-PIC-X-6  PIC  X(6).
    PROCEDURE DIVISION.
    0000-MAIN.
        MOVE  123456  TO WS-PIC-9-6.
        MOVE "123456" TO WS-PIC-X-6.
        IF   WS-PIC-9-6 = WS-PIC-X-6
        THEN DISPLAY "EQUAL"
        ELSE DISPLAY "NOT EQUAL"
        END-IF.
    9999-EXIT.
        STOP RUN.
    ^Z
    $ cobol demo.cob
    $ link demo.obj
    $ run demo.exe
    EQUAL
    $

Now, why are you doing this?  Obviously, to extract the dollars and cents
from the field:

    $ copy sys$input demo.cob
    IDENTIFICATION DIVISION.
    PROGRAM-ID.  DEMO.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  FILLER.
        05  MONEY-TEMP.
            10  MONEY-DOLLARS  PIC  9(4).
            10  MONEY-CENTS    PIC  9(2).
    PROCEDURE DIVISION.
    0000-MAIN.
        MOVE    "123456" TO MONEY-TEMP.
        DISPLAY MONEY-DOLLARS " DOLLARS AND " MONEY-CENTS " CENTS".
    9999-EXIT.
        STOP RUN.
    ^Z
    $ cobol demo.cob
    $ link demo.obj
    $ run demo.exe
    1234 DOLLARS AND 56 CENTS

> 01  MONEY  REDEFINES MONEY-TEMP.
>     04  CAL-MONEY         PIC S9999V99.
>
> so, does the act of redefining money-temp change it??

The act of redefining does not change the attributes or the content of
MONEY-TEMP.  However, MONEY now occupies the same space as MONEY-TEMP, so
if you MOVE something to MONEY at run time, you will change the contents
of MONEY-TEMP, and vice versa.  In order for a non-numeric comparison to
work, you have to take the sign into account:

    $ copy sys$input demo.cob
    IDENTIFICATION DIVISION.
    PROGRAM-ID.  DEMO.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  FILLER.
        05  MONEY-TEMP.
            10  MONEY-DOLLARS  PIC  9(4).
            10  MONEY-CENTS    PIC  9(2).
	05  MONEY   REDEFINES  MONEY-TEMP.
            10  CAL-MONEY      PIC S9(4)V9(2).
    PROCEDURE DIVISION.
    0000-MAIN.
        MOVE +1234.56 TO CAL-MONEY.
        IF   MONEY-TEMP = "12345F"
        THEN DISPLAY "EQUAL"
        ELSE DISPLAY "NOT EQUAL"
        END-IF.
    9999-EXIT.
        STOP RUN.
    ^Z
    $ cobol demo.cob
    $ link demo.obj
    $ run demo.exe
    EQUAL
    $

is used only when _both_ operands are class numeric.

First, SPACES is a figurative constant, not a nonnumeric literal, though
in this particular case the result would be the same.

Second, when a figurative constant other than ZERO is compared to an
external decimal integer elementary item, a _nonnumeric_ comparison is
used. 

In a nonnumeric comparison, the _numeric_ value that was input by the
ACCEPT is treated as if it were MOVEd to an alphanumeric elementary item
of the same size.  If this value had a sign, the sign would be stripped,
by the ordinary CoBOL MOVE rules.  The result is then compared directly
to SPACES.

Every time a compiler writer responds to programer pressure to remove a
"fussy" error message in the interests of making that programer's job
seem easier, there is a corresponding loss of clarity that makes some
other programer's job seem harder.

Think of the PICTURE clause as a promise you make to the compiler, not a
promise the compiler makes to you.  It's not the compiler's job to
enforce the PICTURE, and, as you know, if you break your promise, you may
get an abend.

In this example, it is actually no more difficult to write standard
CoBOL:

       ACCEPT CARD-DATA.
       IF CARD-DATA IS EQUAL TO SPACES
           MOVE CURRENT-YEAR TO SELECT-YEAR
           GO TO 100-DATA-OK.

Of course, there are usually more fields in the card, and before IBM
unilaterally relaxed the rule against numeric/space comparison, we had to
write it like this:

    01  IN-REC.
        05  IN-FLD001.
            10  IN-FLD001-9-4  PIC 9(4).
        05  IN-FLD002.
            [etc.]

        IF   IN-FLD001     = SPACES     MOVE IN-DEF001 TO IN-FLD001.
        IF   IN-FLD001-9-4 NOT NUMERIC  GO TO IN-ERR001.
        MOVE IN-FLD001-9-4 TO ...
