Sorry, but I have to disagree with you. The original question is actually
quite simple to solve. It all comes down to the very first basic function of
information processing and that is "editing". In most systems developed
using Cobol, the front-end processing programs are usually doing the editing
so that data flows correctly downstream.
In this case, the file structure would be:    (Field sizes are for examples
only). Cobol II code.

01 employee-record-input.
    05   emp-name      pic x(30) .
    05   emp-gender   pic x(01).
    05   emp-salarly    pic s9(06)v99 packed-decimal.

01 employee-record-rpt.
    05 emp-nm-rpt    pic x(30).
    05 emp-gen-rpt   pic x(01).
    05 emp-sal-rpt    pic z,zzz,zzz.99-.
    05 emp-sal-stars  redefines emp-sal-rpt pic x(13).

Procedure Division.

     If  emp-name = spaces
         move all '*' to emp-nm-rpt
    else
         move emp-name to emp-nm-rpt
    End-if

    If emp-gender = 'm' or 'f''
       move emp-gender to emp-gen-rpt
    else
        move all '*'  to emp-nm-rpt
     end-if

    If  emp-salary numeric
       move emp-salary to emp-sal-rpt
   else
      move all '*' to emp-sal-stars
    end-if

-- (Write the employee report record.)

There, I helped with somebody's homework

FP

However, your validation checks don't cover it all. We all know about
'Eternal Loops' but what about 'User Generated Eternal Loops". Let me
illustrate when it was ADP and Noah was still loading the Ark. We were
brand new to the game, some of our bright kids were using machine code
and Assembler and we were designing.

It was a system to invoice dairy products. We were feeling our way, and
this was also a brand new game for the people in Sales Ledger/Accounts
Receivable. We designed a Customer input record, with of course an
account number. CustomerX had the option of buying himself, buying
through a Grocers' Buying Association, (that's two accounts), or if it
was a Co-operative could buy as Cardiff Co-op or South Wales Co-op or UK
Co-operative Society (now three accounts) and so on.... All these
groupings were so that retailers could get the current best deal on
prices.

We realized that the A/R people might screw-up when setting-up accounts,
so we designed the card :-

	A/c 1234	Joe Blow	New A/c 6789

They could 'correct' the system by adding that 6789 - which the program
used as the new reference point.

Being real high-tech we had an RCA Magnetic Card File (index system
before IBM disks - 3 Billion chars ?), which whisked 'plastic mag cards'
in a figure-of-eight around the read-write-head then sucked them back
into the appropriate magazine. 

Not unusual on the old clunkers for a process to take 2 hours for one
program doing look-ups. The Ops Manager was used to this. Though he did
find it curious when it took 3, then 4, then 5 hours...... He pulled the
plug.

One little sweetheart in A/R had coded two cards as follows :-

	A/c 1234	Joe Blow 1	New A/c 6789	
	A/c 6789	Joe Blow 2	New A/c 1234

You can't catch 'em all.

Generalizations are generally bad.

Input to a system should be validated.  For example an interface
between the general ledger and foreign exchange should check the input
but between input program and posting program there should not be.

To much code is invalidated if a structure changes,  validation logic
is in too many places and it slows down processing.

How much validation should you apply, as much as is reasonable.  Use
your personal judgement but if you get it wrong you can bet on a huge
amount of time cleaning up the mess.
