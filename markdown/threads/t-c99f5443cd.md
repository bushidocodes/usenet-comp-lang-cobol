improvements.

* ANSI intrinsic functions
   (includes use of ALL as a

    tables for some functions)
* Local-storage section
* Recursive programs
* pass arguments by reference, 
     by content, or by value
* receive parameters by reference
    or by value
* null-terminated strings
* object oriented COBOL
* create DLLs for use by other COBOL
    programs or by C programs

I second everything that Steve says, and would like to share some more
stuff.

I have heard more than a few people say "but you have to upgrade your
COBOL to COBOL/LE to be Y2K compliant!".  THIS IS AN INCORRECT
STATEMENT.  A VS-COBOL application can be Y2K compliant.

My datacenter has upgraded 100% of its MVS COBOL source to the LE level
compiler.  This conversion project went something like this:

1) Implemented the COBOL-II compiler as soon as it became available
(several years ago).
2) Every time a developer checked out a COBOL module for maintenance, it
automatically went through a home-grown code converter to upgrade the
source to COBOL-II.  This converter only accomplished an 80% job.  The
other 20% came out during testing. (Yes, our developers had to be VERY
complete in their testing!)
3) When the LE compiler came along, we discovered that COBOL-II source
can compile just fine under LE without any changes.  LE simply added
more features that COBOL-II did not have.  So we replaced the COBOL-II
compiler with the LE compiler.
4) In the last 12 months, I identified only about 150 programs that were
not yet upgraded.  I upgraded and tested them, thus completing the
conversion!

A major benefit of upgrading, is your software support staff will have a
very difficult time supporting vendor products that are out-of-favor
with that vendor.  For a short while our shop was supporting three
different flavors of COBOL running on the machine, which impacts your
ability to de-bug and support your applications in a quick and effective
manner.

And repeating what Steve said, there are better error handling
routines.  If an LE-COBOL module fails while executing in the LE
run-time environment, you have a better chance that there will be a
report that points you to the real cause of the failure.

Pete

You could write some elaborate edit macro, but if you have a lot of source,
you'd never quite be sure that it made the changes correctly because macro
edits
work on text.

What you really want is syntax-directed transformations that say, in
essence,
if you see *this* language structure, then replace it by *that*.

The DMS Reengineering toolkit can do this.   See
http://www.semdesigns.com/Products/DMSToolkit.html.
It can handle "edits" written in the syntax of COBOL.  Here's a simple
sample:

     default base domain Cobol.

    pattern
conditional-if-then-else(c:conditional-expression,s1:non-if-then-statement,s
2:non-if-then-statement):conditional-if-then-else-statement = "
123456     IF
123456        \c
123456     THEN
123456       \s1
123456     ELSE
123456       \s2
".

    pattern
imperative-if-then-else(c:conditional-expression,s1:non-if-then-statement,s2
:non-if-then-statement):imperative-if-statement = "
123456     IF
123456        \c
123456     THEN
123456       \s1
123456     ELSE
123456       \s2
123456     END-IF
".

rule if-then-else-imperative-to-conditional(c,s1,s2) =
