```
A coworker spent several hours debugging a program today which had
multi-valued switches set and checked as part of its logic flow.

I'm not sure it is a good idea to have two-valued switches - I don't
like forcing people to check WORKING-STORAGE to follow the logic flow. 
This will change when we can SET MONTH-SWITCH TO FALSE.

Three way logic flow switches take up too much of the unfamiliar
maintenance programmer's time, and are more likely to be in error at
some time.

A valid-value switch is a different animal:

Saying IF BOULDER or DENVER or LONGMONT is nice.
```

I think that you missed what he was doing...

He had the end of file switch as a separate switch (away from the file
status).  As in:

01 WS-SWITCHES.
      05 INPUT-EOF-SW                PIC X   VALUE 'N'.
           88 EOF-INPUT                                VALUE 'Y'.
           88 NOT-EOF-INPUT                      VALUE 'N'.

01 WS-FILE-STATUS.
     05 INPUT-STATUS                  PIC X(2).

...

OPEN INPUT INPUT-FILE.

PERFORM UNTIL EOF-INPUT
   READ INPUT-FILE INTO WS-INPUT-RECORD
        AT END SET-EOF-INPUT TO TRUE
        NOT AT END
            IF INPUT-STATUS NOT = '00'
                  abend - or what ever
            END-IF
    END-READ
    do something else...
END-PERFORM.

CLOSE INPUT-FILE.

SET NOT-EOF-INPUT TO TRUE.

OPEN INPUT INPUT-FILE.

PERFORM UNTIL EOF-INPUT
   READ INPUT-FILE INTO WS-INPUT-RECORD
        AT END SET-EOF-INPUT TO TRUE
        NOT AT END
            IF INPUT-STATUS NOT = '00'
                  abend - or what ever
            END-IF
    END-READ
    do something else, but probably not the same thing you did last time  :)
END-PERFORM.

CLOSE INPUT-FILE.

I have used this type of assignment before, but can only see a purpose for
the NOT-EOF switch if you are going to reopen the file for some reason.  I
have drifted toward using this instead...

01 WS-FILE-STATUS.
     05 INPUT-STATUS             PIC X(2)  VALUE '00'.
          88 EOF-INPUT                                VALUE '10'.
          88 NOT-EOF-INPUT                      VALUE '00'.

Where the eof switch is set under the file status, and, again, the only use
for the NOT-EOF switch is to reset the status because I am using the file
again (I do a lot of dynamic allocation to the same DD and have a tendency
to name my file status switches by the DD name).  I also do all of my file
status checking (except EOF) in the same paragraph (so I don't have to keep
repeating the same code and messages).  Anyway...

READ XXXX AT END MOVE 'Y' TO EOF-SW

file status allows you a nice, clean IF following the read..

Richard,

I definitely agree that 88s can make life difficult, and we don't
always use 'em, it depends on the circumstances.

One of the really nice things about an 88-level is that it's
self-documenting when used properly.

A case in point:  

Our old character-mode screen manager didn't use 88-level items for
anything.  As a result, you needed to refer to the manual to figure
out what a given returned value meant, or to know what value to move
in to a field to cause desired behavior.

We fielded lots of phone calls from people who wanted to know how to
do certain things - to the extent that "it's on page 3-23" was the
number one answer to the most common tech support questions...

We resolved we'd figure a way to eliminate this sort of question with
our new screen manager, so GUI ScreenIO generates 88-level items
beneath (virtually) all control fields.  Now, instead of knowing that
you need to move 'X' to panel-whatever to cause something to happen,
you just set the desired plain-English 88-level item to TRUE.  You
don't need to know the value.  The names are self-explanatory so you
don't really need the manual anymore.

Also, since in GUI ScreenIO also predefine all of the events that
could cause a return to your program (and assign your own names to
them), we can also generate 88-level items for those - so all of the
stuff you need to handle in your program is clearly documented in each
panel's copybook.  It REALLY made it easier.

The "shouldn't-need-to-call-for-tech-support-on-this-subject" tech
support calls went way down.  Which is how everyone likes it.

-------
Incidentally, the other nice thing about an 88 level item is that no
code is generated for it unless it's referenced in a program.  So, a
big collection of codes defined as 88's is quite efficient in terms of
storage utilization.
