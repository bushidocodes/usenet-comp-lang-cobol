==>start of cobolcgi source
identification division.

working-storage section.
01 input-page is external-form.
   03  name-field pic x(30) identified by "Name".
   03  phone-no   pic x(30) identified by "Phone".
   03  email      pic x(30) identified by "Email".

01 output-page is external-form identified by "output".
   03  name-field pic x(30) identified by "Name".
   03  phone-no   pic x(30) identified by "Phone".
   03  email      pic x(30) identified by "Email".

procedure division.

    accept input-page
    move corresponding input-page to output-page
    display output-page
    .
==>end of cobolcgi source

==>start of input.htm
<HTML>
<HEAD>
<TITLE>Sample Form</TITLE>
</HEAD>
<BODY>

<P>Input Form

<FORM METHOD="POST" ACTION="/cgi-bin/cobolcgi.exe"></P>
<PRE>
Name     <INPUT TYPE="TEXT" NAME="Name" VALUE="" MAXLENGTH="30" >
Email id <INPUT TYPE="TEXT "NAME="Email" VALUE="" MAXLENGTH="30" >
Phone    <INPUT NAME="Phone" VALUE="" MAXLENGTH="12">
<INPUT TYPE="SUBMIT" VALUE="Send Form"><INPUT TYPE="RESET">
</PRE>
</FORM>

</BODY>
</HTML>
==>end of input.htm

==>start of output.htm
<HTML>

<P>*** SAMPLE OUTPUT FORM ***</P>
<P>Dear :name-field</P>
<P>blah blah blah</P>

</HTML>
==>end of output.htm

The accept is from STDIN and the CGI handling in the web server pipes 
the form data to the program on that file handle. The program has to 
parse the xxxx=value stuff from the form and construct the HTML for 
the output, which is done with the display verb that ends up on STDOUT
where the web server grabs it and sends it off to the client.

POST gives input in stdin.  One problem is that some languages or
versions, such as Cobol, will not complete the ACCEPT until the variable
is full or an enter key which may have it hang on input. Note that
switch S5 is required to be on in MF Cobol for it to use stdin/stdout.

Using a GET requires that the environment variable QUERY_STRING is read.

> > In the past, Richard Plinston has posted something similar, but I believe
> > that Richard has investigated the lower levels of the CGI interface and
> > knows which ports to accept and display to in his environment. (Please
> > correct me if this wrong, Richard) I note that Richard has posted to this
> > thread qualifying your statement also.

Windows is the problem, or in fact, several problems.  Some Web Servers
use 'WinCGI' which avoids stdin/stdout and provides a file interface. 
With Windows it is necessary (in my experience) to set up a message loop
for the CGI program, though it doesn't do anything with it, in other
words it needs to be a Windows API program (YMMV).

Fortunately I did it all some time ago and it sorts itself out at run
time deciding how it takes its input and sends its output.  The
Windows/Xitami version has to link in a different main module for the
message loop.

wanting to send a URL and get a page back in return.  I have done this

easy.
