There are a couple of ways to get there with Fujitsu COBOL.  You can 
use their POWER COBOL offering - which generates OO code, or you can 
use a tool called CASEGEN - written specifically for the Fujitsu 
platform, or you can use my personal favorite COBOL sp2 
(http://www.flexus.com).  There are pluses and minuses to each.  I've 
bored the group here numerous times with comparisons, you can search 
Dejanews for my name and sp2 in the message.

> 2)  Does FUJITSU support its own internal ISAM file system as well as
> handle ODBC database calls?
> 

Yes.  It has the second fastest file handling I have worked with.  
Second only to Realia COBOL.

> 3)  How easy is it to learn and write in their OO COBOL code?  I have
> seen some OO Cobol systems that are worse than standard Cobol!
> 

This is a stylistic thing, and not directly related to Fujitsu because
other COBOL vendors support OO also.  Fujitsu DOES, however, generate 
OO code with POWER COBOL - although this is mostly hidden from the 
programmer.

> 4)  How stabled or bug-free is the FUJITSU compiler and resulting
> code?
> 

The compiler is the cleanest I have worked with (save the COBOL for 
This and That on the OS/390 mainframe).   It is a compiler 
specifically targeted for Windows, and it has the fastest executing 
code of any I have used.  

The resulting code, however, is only as bug free as you make it!  Same
for every compiler! <G>

But if you mean runtime bugs - I haven't seen any.

1. It is quite easy to write GUI code, but there is a big
learning curve.  Fujitsu's documentation is nearly worthless.
Plan on spending a lot of frustrated time, but when you get
it working, it works well.    It certainly beats learning a new
language, for someone like me with 20 yrs of nothing but
cobol!

2. Internal support for ISAM is fine.  Works no differently
than any other cobol ISAM I've used.  Very efficient and
fast file access

3. See #1 above - go slooow and learn.  Its worth the effort.

4. I have seen no stability or bug problems.

OO stuff into an existing COBOL package.

on this one.
