```
>I am trying to get dynamic calls approved as a standard in our shop.  I am
>looking for evidence supporting the use of dynamic calls as opposed to
>static.  We develop COBOL/390 on OS/390 under IMS/DC, CICS, batch, etc.
>accessing IMS, DB2, VSAM, flat files, etc.


I worked at a shop that had them Specifically, building the called
subprogram based on a field input from a screen in CICS. It was horrible to
trace down. If you don't know the valid fields, than you don't know what
subprogram will be called. You would have to document all of the valid
choices. Validate all of the choices. And hope no one changes your logic in
the future.

Programs would be much simpler if these were not used. However if the logic
is properly documented and the standard of documenting the logic is
enforced. Then I don't see a big problem with it.

I seem to recall something about some type of overhead in CICS to maintain
dynamic calls. You may want to check into this before moving on. If this is
true, once your standard begins to proliferate it will begin to bog down
CICs.
```

```
I think this is an argument against such a technique, not against
dynamic CALLs. Do you object to CALLing programs in batch? Following the
program logic concerning CALLs shouldn't be any more difficult than
following the paragraph names you PERFORM. A CALL is simply an external
subroutine (or paragraph, if you prefer). Other languages, e.g., REXX,
don't have separate ways of invoking internal/external routines. You
CALL everything, if it's not defined in your program it must be
external.

I recently found an LE/370 paper (thanks to a URL posted by Bill Klein)
in which the current recommendation is to use dynamic CALLs if you're
concerned about the overhead of CICS LINK commands (search DejaNews if
you're interested). Go figure:-)
```
