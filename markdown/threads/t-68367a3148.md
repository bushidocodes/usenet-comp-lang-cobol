> We would like to have our objects in ONE directory, but this is

Personally we always separate the sources, INTs and GNTs into separate 
directories but to each his own...

As far as I know you cannot change the order, indeed it is mentioned as 
*the* search order (somewhere in the manuals) However why don't you make 
the CALL call a variable

        01  fred.                       PIC X(12)  (...say)
                

        CALL fred USING ...
        
you can construct 'fred' dynamically in your program (CBL_JOIN_FILENAME). 
Have a flag which you can set when you're running via animator (perhaps 
even by setting it manually via ANIIMATOR) and add INT as the extension 
when you are in this 'animator' mode. Thus when in 'animator' mode 'fred' 
would be "XYZ.INT"  and the specific file would be loaded; while in normal 
use it would just be "XYZ" and the files would be searched in the order 
you've mentioned.

In brief - why not use variables to call rather than literals:
        CALL fred USING ...
        
you can then set up 'fred' dynamically using CBL_JOIN_FILENAME, for 
example. To get over your particular problem you could have a flag, 
perhaps read from a file, a command line argument, or perhaps set manually 
when you run animator. This flag is then used by the routine that creates 
the filename: if the flag is set, construct a filename "fred.int"; if not 
set then construct "fred". Say have one routine that constructs the 
name(s) and depending on the structure of your system either set up all 
the names in advance of before each CALL statement.
