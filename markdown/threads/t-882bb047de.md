completely and utterly non portable between compilers at present.

COBOL, it has been slow to be adopted.  Cautious optimism abounds.
However, you will find many here who have adopted it within the
confines of their particular compiler of choice, who are using it for
production code.

Does your project plan recognize the object dependencies in developing a
program? One of my pet peeves is shops that continue to plan in a 'job shop'
mentality when they really need to think about 'software manufacturing'
including critical path analysis. Often planning consists of identifying
individual programs and then assigning them to the worker bees to be
written. If management doesn't know how to manage a project with a high
degree of interdependency then you cannot blame the worker bees for doing
whatever is necessary to deliver their piece.

Once the delivery date starts slipping there is pressure to move that
completion estimate to 100%. Try explaining to some middle manager why 90%
of the programs are at 95% complete pending completion of a single object
that somebody else is writing. They do not want to hear it. Management needs
to be taught a new way to plan deliverables.

The next issue is indexing all those objects. If a programmer perceives that
it would take them longer to find an object that suits their purpose than
writing a new object, they will write a new object. Trying to figure out
which object to use, or even if an object can be used in a given situation
is a lot like playing 'Jeopardy!'. You've got an answer but you have to come
up with the question, or context, in which that answer applies. It would
help if documentation was extensible so that every time an object is used,
the user provided a brief description of the context in which it was used.
That way you could index the question rather than the answer. If you can't
find it, you won't use it.

So, you've found the (almost) perfect object. In the real world that object
is only perfect for the set of conditions that it was developed to handle.
Making it generic takes effort. Designing the 'perfect' object is an
exercise in diminishing returns. Assume that you're going to start with an
imperfect solution and have to adapt as development proceeds. So now you've
got a programmer trying to complete an assignment using unstable components.
That programmer needs to know that any problems that are identified will be
resolved in a timely manner. Any lack of confidence or unacceptible delay
will result in the developer solving their own problem by growing a new
object. A successful project needs a respected technical architect.

Review programmer's work early and often. It's time-consuming but it's a
valuable learning tool for everybody. The architecture is improved by the
feedback, the programmers learn about how the objects were intended to be
used. Object programming is a team exercise. Everybody depends on everybody
else. One project I worked on involved one-on-one startup sessions between
the keepers of the objects and the programmer for each program. They
identified the most logical starting point, objects to handle particular
data domains, common business processes, etc.

Plan to maximize reuse. Leverage commonality whereever possible. If the
analysts aren't working from the same set of program archetypes then the
programmer's will be left to sort out the mess. The best object library in
the world is of little use if it cannot be applied to the problem, as
defined by the analysts. Develop program archetypes, data domains, and
common processes before ever partitioning the work between analysts.
Establish confidence that the archetypes and object library satisfy the end
users, analysts, and data gods by implementing a small part of the system
using them. Make the analysts stick to the plan. If they are allowed to
introduce their own variations then the programmers may be forced to
introduce new objects to deal with those variations.

All of our new code is done like this.  We don't use OO per se, but we
build callable modules.  For example, we have module that we can call
to come back with a single product code or a range of product codes.
That module is then responsible for all lookups, validations, adds (if
allowed by a flag sent in with the call), ranges or single picks
(again, a flag) etc.

However, like you say, it's hard to keep track of what modules you've
built and how to use them.  To this end, we have gone to great efforts
to document the calls, what to send in, what comes out, etc.  They've
proven to be very very useful as we build the new system.  Having one
module to change/correct/maintain has eased the transition.  The added
benefit is the consistancy to our end user.

spend most of your work time working on "old fashioned" COBOL, since that is
the current inventory.  Contrary to what I was taught 15 years ago, most
program life-cycles are counted in decades, not years.  It is simply not
economic for most business users to re-write the entire portfolio every 7 to 10
years.  Thus, you will, if my experience is any guide, find yourself working on
code that was written in  the 80's or even the 70's.

However, if your future employer is going to use the new technologies, like
tying the "legacy applications" to the Internet, or make extensive use of EDI
in most of the current implementations, you will need to know some OO, simply
to write the interfaces, and maintain the connections.  Press your instructors
next semester to spend some time on it - or, go out and find it yourself (what
most of us in the profession have to do on a ongoing basis).  There are lots of
good resources out there, I like Digital Think (www.digitalthink.com) but there
