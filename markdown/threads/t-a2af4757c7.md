```
I want to open up a discussion of the possibilities and problems of
Open Source (OS) COBOL.  Although I have found some public domain COBOL
on the Internet, I have not yet found any OS COBOL and I would like to
know why.

1. Open Source

For the purpose of this discussion "open source" means the formal
definition (http://www.opensource.org/osd.html) given by the Open
Source Organization.

3. License and Copyright

OS is not the same as public domain.  OS is copyrighted and licensed.
The most popular OS license is the GNU General Public License or GPL
(http://www.opensource.org/licenses/gpl-license.html) although there
are other possibilities (http://www.opensource.org/licenses/).

4. Making Money from OS

This has been a much-discussed subject.  I would recommend a look at
http://www.opensource.org/for-hackers.html.

5. Modifications

One of the problems with the GPL is that section 2 (a) requires you to
make public any modifications that you make to the original code.  This
works for system software or general utilities where any confidential
information is contained in data files.  However, COBOL programming is
usually a process of encoding a company's policies and procedures so a
computer can carry them out.  These policies and procedures are
sensitive data and I don't know of a company that would want them made
public.

How would this condition of the GPL have to be changed to make OS
practical for COBOL?  I feel that the principle is important because it
provides a major incentive for programmers to create OS in the first
place.  You get back all the enhancements to your original work and a
lot of knowledge about the possibilities of your code.

One possibility would be to allow the private use of modified code
within a single organization but require publication if it moves
outside that organization.  However, defining "organization" becomes an
interesting problem when many of us work for huge international
companies with many different entities.

6. Fair Use

If you were the programmer who had invested hundreds or thousands of
your valuable hours in creating OS software, what would you consider to
be the fair use of your software?

Having a company make money by selling the results of your effort is
obviously unacceptable and is forbidden by the GPL.  But to what extent
would you accept an organization keeping their enhancements of your
software private?  In fact, what constitutes an enhancement to the
original product compared to a modification to make the product work
for a specific company?

For example, suppose you create a point-of-sale system as OS.  A
company modifies your software to make it web based.  Is this a private
modification for that company because it happens to do business over
the web or a general enhancement that should be made public?

7. Standards

I don't see anything about the use of standard forms of a programming
language.  However, I would assume that the spirit of OS would
encourage the production of the most portable code.  This would mean
sticking with the published standards and avoiding any extension
provided by a specific compiler.

In particular, this would mean, for the moment, giving up OO and GUI
based user interfaces.  However, should you also give up ODBC or CICS
because Microsoft and IBM control them?

The report writer and screen section are a standard part of COBOL but
they are not available on IBM mainframes that are still the most common
environment for COBOL.  Should these also be avoided for proper OS?
```

is the current need ?

can give it a try", and haven't taken into account 'COBOL-97' standards. 
(Also in grand isolation are mainframe WATCOB from U. of Waterloo, not
supported, and the COBOL650). What is the point of just a compiler, even
if it provides 'all'; we now have many of the popular compilers wrapped
into an IDE with source editing, screen-designing, compiling, linking
etc. Open Source that offers less than we have come to expect from
Fujitsu, Merant, AcuCobol, Liant etc. is a non-sequitur.

For Open Source to work it has to be a real team-effort, and I would
suggest, with lots of input from people in this NewsGroup who are
'code-oriented'. The only name currently that springs to mind is Tim 
Josling (Australia) who is involved with Tiny COBOL I believe.
Unfortunately, Tim 'hates' all the new Reserved Words, which only add to
his problems of compiler designing. To be blunt, the current progress
appears to me to be in the category 'Amateur Hour'. (No, don't ask me to
contribute, I'm not a 'coder').

Just to throw a spanner/wrench in the works, I would define Open Source
slightly differently :-

(a) Open Source compiler - which fits your definition above
(b) Open Source examples - a library of source code showing how to do
    things in COBOL, like Bill has included for 'Dates' in the FAQ.

I don't think the concept of an Open Source compiler violates corporate
confidentially as you posed - the compiler needs to contain code to
handle all COBOL syntax, in general terms - and not related to the needs
of a particular user corporation. You cited a POS system as an example -
this is a specific application, and I don't see it as a part of Open
Source.
