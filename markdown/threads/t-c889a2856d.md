I can't speak to the Unisys 2200 side of the fence, or to whether MicroFocus
products have this extension.

But I can say that the ability to pass a file as a parameter to a Unisys
MCP/AS library written in COBOL (which is distinguished from a COBOL "called
program" under ANSI rules only by the capabilities of Unisys extensions) is
indeed a Unisys MCP/AS extension to ANSI COBOL.  It dates back to at least
the COBOL(68) implementation on the Burroughs Large System.

Yes, the "open state" of the file is retained across the CALL.  Remember
that COBOL ANSI IPC calls, as well as COBOL library calls, are
"single-threaded"; while a "called" program is executing, the program that
called it is waiting for the called program to execute an EXIT PROGRAM.  So
long as everybody using the file is sensitive to the possibility that others
may also be using it, and thus accounting for verification of the file's
current state on each call, the answer to the question "how does it work?"
is "Just fine, thank you.  Been working fine for some decades now."

    -Chuck Stevens

Passing a file to a "subprogram" of some sorts is a medium common extension
to PRE-'85 Standard compilers.  (Once the EXTERNAL attribute was introduced,
I think that few - if any - NEW vendors added this.  Others, such as Unisys
may well have kept it for upward compatibility).

In the IBM mainframe world, the OS/VS COBOL compiler allowed one to pass
either a sequential (QSAM) or Indexed (VSAM) file to a subprogram.  What was
ACTUALLY passed was a pointer to the DCB or FCB, so only an Assembler
subprogram could do anything "useful" with this.  In VS COBOL II and later,
you could pass a QSAM/DCB - but not a VSAM/ACB.  (Using USAGE POINTER data
and a LOT of "systems internal" information, you can even process this in a
semi-useful way.)

In Micro Focus, you should look at the FCDREG compiler directive to see what
it "gives" you.  I don't think you will be able to "port" a Unisys
application (from the information Chuck provided below) DIRECTLY using this.
However, I think (but am not positive) that you should be able to emulate
much of the Unisys FUNCTIONALITY with this Micro Focus feature.

Oops, the EXTERNAL goes on the FD, I should have checked my programs 
instead of glancing at the manual.

The EXTERNAL on the SELECT ASSIGN makes the actual name of the file come 
from a source external to the program, ie from an environment variable.

EXTERNAL on the FD is what is required to make the one file accessible in 
several programs in the same run unit.

      FD   filename EXTERNAL.

      01   ...

I don't use EXTERNAL on the record areas, and don't know if the same record 
areas are used for the file in the two programs, I guess not.  It is 
important that the file attributes be identical in the programs that use 
this.  I was confused when I was getting a run time error in Fujitsu when 
the programs worked fine in MF.  Turns out that Fujitsu requires the file 
name data name to be defined:

      SELECT file ASSIGN data-name ...

In MF the 'data-name' is implicitly defined if it has not been explicitly 
defined, this is how MF has been since CIS Cobol.  Fujitsu requires 
data-name to be explicitly defined and gives a compiler error when it finds 
MOVE something TO data-name.

Of course I had declared the data-name in the program that set up the name 
and OPENed the file, but I had not done so in the program that merely READ 
the file and had no use for the data-name item there.

Fujitsu failed at run-time because the file wasn't OPEN.  Actually it 
treated the two instances of the file, both declared EXTERNAL, as if they 
were two different files instead of the one being shared. It worked fine in 
MF, In the program that OPENed the file it could READ, in the CALLed 
program it failed 'file not open'.

It was very annoying because I had another example where it worked 
perfectly in both systems.  Eventually I tried adding an OPEN into the 
failing program, noticed the missing data-name when setting up the name of 
the file and added this.  _Then_ it complained the file was already open.  
Adding the data-name fixed the problem because now all the attributes were 
the same.
