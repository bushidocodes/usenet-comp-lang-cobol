The table is based upon branch number.  If that branch number has to be a

01   MY-INDEX           PIC S9(5) BINARY.
01   MY-MAX              PIC S9(5)  VALUE ZERO.
01   MY-LIMIT             PIC S9(5)  VALUE 100.
01   MY-TABLE
      05   MY-RECORD   OCCURS 100.
             10   MY-BRANCH     PIC XX.
             10   MY-QUANTITY    PIC 9(5)
             10   MY-   put whatever you will need to do your calculations.
....


INIT-ROUTINE
      move all '0'    to my-table.
....


READ-ROUTINE.
      PERFORM VARYING MY-INDEX FROM 1 BY 1
                              UNTIL   MY-INDEX > MY-MAX
                                      OR MY-BRANCH (MY-INDEX) = BRANCH-NO
      END-PERFORM.
      IF MY-INDEX > MY-MAX
            ADD  1        TO MY-MAX
            MOVE BRANCH-NO     TO MY-BRANCH (MY-INDEX)
            IF MY-MAX > MY-LIMIT
                  DISPLAY ' MY-TABLE OVERFLOW'
                  PERFORM ERROR-ROUTINE
            END-IF
      END-IF

      ADD QUANTITY-SOLD TO MY-QUANTITY (MY-INDEX).
      Do other calculations you need now into the table.



When you are finished, do a perform, walking the table and getting its
output to print.

Ways to improve the program:
1.   Sort the file before you read it.   This will help immensely.  You
won't have to search at all.
2.   Use an index instead of a subscript to use the SEARCH command.
3.   If you have numeric subscripts between 01 and 99, don't do a search at
all - but go directly to the RECORD (BRANCH-NO).   BRANCH-NO needs to be pic
99, not XX, and you need to do an IF ALPHA check to make sure it doesn't

Try these all and tell your instructor why you picked the choice you prefer.

Your first step should be to remove the value statements from your record
description.  They are illegal, and will not compile.  You are going to
*read* this data, not set it up in memory with the compiler.  After you read
it from disk, you may wish to check the validity of the data you read, by
checking to see if the asterisks are there.  To do that, you will have to
give names to the fields, rather than making them filler.

As a hint on where to start, you define the file in the file section, then
you describe the table in the working-storage section.  Once that is done,
you write a procedure that reads the data from the file, and places it in
the table. After loading the table, you can manipulate it as much as you
want with further procedures.  Start by writing a program that reads the
data, and simply displays it on the screen. That should be about a five line
procedure.  Once you have that working, then play with putting it in a
table, and get that working.  When that is complete and tested, then write
the analysis stuff, and jazz up your display.

FYI -

A) a "number" of compilers currently accept (as an extension) VALUE clauses
(other than 88-levels) in both File Section and Linkage Section.  They are
(needless to say) "ignored" in most cases.

B) the draft Standard makes it explicit that you MAY code a VALUE clause
(other than 88-levels) in both File Section and Linkage Section.

C) the new "INITIALIZE TO VALUE" variation of initialize *will* pay
attention to these VALUE clauses.

  ***

General Rules 2 and 3 for the VALUE clause in the FCD are:

"2) In the file section, VALUE clauses take effect only during the execution
of an INITIALIZE statement. The initial value of the data items in the file
section is undefined.

3) In the linkage section, VALUE clauses take effect only during the
execution of an INITIALIZE statement. The initial value of the data items in
the linkage section is specified in 13.6, Linkage section."
