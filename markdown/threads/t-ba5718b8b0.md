I don't use list boxes or combo boxes for Customers or Products,
there tends to be thousands of them - far too many to load
reasonably.  I use a separate panel with a repeat group into
which I read 10 (or so) at a time and allow paging forwards and
backwards.

In the customer code field they could input the correct code
and have the customer appear, or they could input (for example,
depending on system) a part code or a part alpha key and press
a function key or button.  The program passes this to the
listing program which opens a panel and starts displaying at
that point using the file key as appropriate.  10 are displayed
and the user can move up and down these, as they do so the
full customer details are read and displayed below the
select list.  PageUp/Down gives forward/back.  Function key
used in the select list gives access to the maintenance 
function which opens a new panel on top.  When maintenance is
finished the select panel reloads starting at the key it
last started so new details (even if created elsewhere)
are loaded.

By having a modular design the customer (or product or whatever)
maintenance screen can call the select list, or after calling
the select list from, say, orders, it can call the maintenance
screen.

A mechanism is then required to prevent recursion.  I use a
EXTERNAL COPYed data area with flags indicating which panels
are active so that the flag is checked before calling.

This is done using SP2, mainly in text mode (but works 
identically in Windows).
