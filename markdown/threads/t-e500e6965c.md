According to the glossary in ANSI X3.23-1974, "Word" is defined on page I-71
as "A character string of not more than 30 characters which forms a
user-defined word, a system-name, or a reserved word."

The same size description appears on Page IV-5 of ANSI X3.23-1985 for "COBOL
Words" in section 4.2.2.1:  "A COBOL word is a character-string of not more
than 30 characters ...".

Note that both of these are *upper* limits.

What word size were you using?  What is the new limitation that you
encountered unexpectedly?

    -Chuck Stevens

It had to do with "compatibility" with some other ISO Standard.  My memory
is foggy about this (because we also expanded the numeric digit size from 18
to 31 - for compatibility with another standard).  I think one of these was
for compatibility with the C Standard and one with the SQL Standard - but
that is only a "vague" memory.

FYI - the reason that COBOL word length is *nice* to have the same as
another language, is that it is nice to have EXTERNAL data with the same
names across products. The draft Standard has other ways of dealing with
this, but it was part of the reason that the word size was increased to 31
characters.

It's unlikely, but if a compiler just throws away the end of a long name,
it can introduce subtle bugs.  Eg:

 MY-FIRST SECTION.
 PAR1.
     PERFORM THIS-PARAGRAPH-NAME-IS-TOO-LONG-B
 ....
 THIS-PARAGRAPH-NAME-IS-TOO-LONG-A.
 ....
 MY-SECOND SECTION.
 THIS-PARAGRAPH-NAME-IS-TOO-LONG-B.

This will compile clean, but it will PERFORM
THIS-PARAGRAPH-NAME-IS-TOO-LONG-A not THIS-PARAGRAPH-NAME-IS-TOO-LONG-B.
The two paragraph names cannot be distinguished, so
THIS-PARAGRAPH-NAME-IS-TOO-LON is implicitly qualified by the current
section.
