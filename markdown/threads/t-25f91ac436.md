As you have realized, it is necessary to use dynamic SQL. This can have 
serious performance implications so there are other steps necessary to 
counteract this.

Rather than a "DB Class" apply your class to a given set of related tables, 
on a specific database. (Relational DBs are excellent for this... :-))

If you try and do it for each individual table you will have update synch 
problems for related tables, (even using cascaded updates, this gets very 
dodgy very quickly...) and if you try and do it for the whole database, it 
becomes too unwieldy, and the resultant object is too large.

> What do you think about it ?
>
I thought it was a very good idea when I first considered it, about 7 years 
ago. It followed naturally from the VSAM systems we devised in the 60s, 70s, 
and 80s where we used "access modules" that could do every possible action 
against a given dataset, but, of course, none of this was object oriented.

However, using separate modules for data access DID provide a number of 
advantages. In the ensuing years these advantages have been realised and 
formalized and may well have provided the basis for the objects and layers 
that modern software is designed upon.

If you can reduce the functionality to a set of simple operations, you can 
then generate the code easily. The obvious solution is to use a template.

I undertook this initially as a labour of love and largely because everyone 
I spoke to said it couldn't be done. Far too difficult to provide data 
access for all of the infinite ways programmers devise to access data. They 
were nearly right... it has taken me several years to get a solution that 
CAN do everything, but it was worth the effort. (It has also been extremely 
challenging and interesting, with new unforeseen problems popping out of the 
woodwork all the way along...)

I decided to incorporate everything I had learned into a commercial package 
about 3 years ago and it was completed a few months ago. The first customer 
to implement it has volunteered to act as a reference site and is very 
pleased with what they got.

My software generates OO COBOL code for an object that handles a given "set" 
of tables on an RDB. (This is exactly what you proposed above, but instead 
of writing it by hand, I generate it from a toolset written in C#)

The collection of these objects constitutes a Data Access Layer (DAL). A 
"table set" in this context is a set of normalized tables in 2NF. These are 
generated from COBOL COPY books and the OCCURs repeating groups are 
separated out to tables which are linked to the base table in accordance 
with first normal form normalization. All constraints, keys and cascades are 
handled by the toolset.

(I note in passing that if you designed the db table set by hand, the 
Toolset could still generate the maintenance object for it... however, I 
would need to write a module that analysed your DB...)

Separation, even for existing COBOL applications, is automatically generated 
from the COBOL COPY Books. The idea is to break existing integrated, 
monolithic solutions into objects and layers that can be accessed from other 
languages as well as COBOL. The generated objects are currently compiled to 
run in the .NET framework, but they don't HAVE to be. Each object is 
generated from a base template that has all the functions you require to 
maintain data (a Maintenance Template). Each object is actually a COM 
server, so the base templates are referred to as "MOST" (Maintenance Object 
Server Templates). The tool which generates a MOST object to handle a given 
Table Set is called MOSTGEN.

The instantiation, setting of object references, DB connection code, 
analysis and generation of multiple DAL instances for separate threads, 
exception handling, DECLGEN and inclusion of host variables is all 
considered to be MOST "housekeeping"; this is done by another MOST tool. 
There is no need for programmers to have to understand OO in order to use 
MOST; the code is generated and inserted into the existing (or new) 
application code, automatically.

Another tool changes all the existing ISAM references in a program into DAL 
object invokes. The end result is to switch the data source for the 
application from ISAM to RDB, WITHOUT changing the business logic. Data 
access is simply generated into a separate layer of access objects.

 The interface is initially via the existing record layouts and the DAL 
objects take these and apply them to the RDB, or access the RDB and pass a 
populated record layout back to the COBOL application.

However, this is just a starting point. RDB views can also be easily 
incorporated into the DAL so that frequently used elements can have their 
own view and it is not necessary to populate a full COBOL record for them. 
Repeating groups are handled automatically by the DAL and tables embedded in 
the COBOL layout are automatically populated from DAL RDB access. I spent 
many weeks optimizing the dynamic SQL used by the DAL objects and received 
very helpful suggestions and advice from Frederico Fonseca and Robert 
Wagner, which I took on board and incorporated into the final release. 
Combined with the capabability to multithread DAL objects, this ensures that 
performance is excellent. Live tests have been impessive.

The main point is to separate data access from business logic. Applications 
no longer contain SQL or ISAM code. Instead, they invoke equivalent 
functions in an object layer. Performance optimization, changes to data 
requirements, new views of data for specific application code, can all be 
implemented without impact on the business logic. Object encapsulation and 
process isolation ensure that "knock on" errors in maintenance are virtually 
eliminated and regression testing is minimised.

There is not enough space here to describe everything the Toolset does (and 
it is not appropriate for me to promote my own software in this forum...I've 
offered the descriptions here to show that your idea has some very 
interesting ramifications...).

However, I do plan on attaching a Toolset portal to the cobdata web site, 
which will show walkthroughs and demos, along with the actual code 
generated, for people who may be interested in getting their existing COBOL 
into the 21st century.

Generalized classes to handle relational data are definitely viable, but 
there is far more to this than you might think on first inspection.

Objects and layers are where the future lies, small is beautiful...

Pete.

An alternative is to make database access an aspect of classes that
represent your data abstractions. Thus you might have a Payment class
with a "commit" method that performs the necessary inserts or updates;
a (static) "locate" method that performs the necessary selects and
returns a new Payment object; and so forth.

This lets you use the Implementation pattern within your application
classes, so the application logic doesn't need to know anything about
the database - or even that it's going to a database. RDBMS, indexed
file, whatever; it's transparent to the application.

Layers and objects.

It's kind of funny, but back in the days when Computer Science (at least 
commercially) was non-existent, we learned by trying stuff and making it 
better the next time we did it. (You could say it was an empirical, 
iterative approach, but unkinder people would say it was fools stumbling in 
the dark and getting there by trial and error, with a modicum of sheer 
luck....)

Today's programmers are mostly professionals and have the benefits of at 
least some formal training.

In the time I'm speaking of it was normal for bright kids in the business to 
volunteer for the "computer department" (later it became "EDP", then "IT") 
get sent on a two week COBOL course, and come back to start implementing 
business rules. Sometimes they would be aptitude tested, but very often all 
that was required was enthusiasm for the "coming technology", knowledge of 
the business, and ability to learn fast.

Given this, it is hardly unexpected that some very poor computer systems 
were written...(there were some very good ones too, but system development 
was very much a hit and miss affair when judged by today's standards. Formal 
project management was unknown, PERT was still being invented to support the 
Polaris project, and the Waterfall evolved out of IT being burned with 
continually changing requirements.)

"Best practices" evolved for many of us out of experience, and not from 
concepts of programming theory.

But the interesting thing is that very often the practices we embraced were 
the same ones that Acadaemia formalized and taught some time later. (I'm not 
suggesting there weren't papers that had been written, or that NO 
universities were teachng programming and Computer Science; I'm saying that 
formalizing programming was not the common approach on most sites, 
especially COBOL ones, and the majority of COBOL programmers at the time had 
no formal programming training, other than that provided by courses run by 
their vendor.)

The idea of separation and patterns WAS being implemented on commercial 
sites but the people doing it did not realise that that was what they were 
doing.  I mentioned in another post in this thread that we built access 
modules for large VSAM-based systems. In fact, I remember leading a team in 
the mid-1970s that built a pan-European database for a major international 
corporation in England. It was entirely VSAM based and templated access 
modules were used. (It was so effective that the users and the management 
resisted violently some years later, when it was decided that the company 
should move to IMS.) We decided to use separation between the the database 
and the applications because the system had to work in 14 countries and some 
of their requirements were specific to that country. We wanted generalised 
business logic with hooks for country specific functions and we felt that if 
everything was accessing the database from everywhere, it would just be a 
jungle.

Because of the architecture we chose, it was not a major trauma to replace 
the VSAM base with IMS, and later this was replaced with DB2.

Similar experience was applied with user interfaces; it really didn't matter 
where the data came from (punched cards, IMS/DC, or CICS BMS) as long as the 
business function received a buffer with data in it that was meaningful for 
that process.

By the mid-80s PCs had arrived, but the concepts of networking were still 
new to many of us. We were used to a centralized processor driving 
everything and the idea of having different "cores" working asynchronously 
was foreign. (In fact, to some, it was anathema...)

I remember the sense of wonder I experienced when I first realised that 
suitably wrapped components could run ANYWHERE on the network, and didn't 
need to be queued up waiting for a processor... With the advent of the 
Internet it meant that encapsulated functionality could be flashed around 
the world and run wherever it was needed. A long way from COBOL on a 360-30, 
driving a 2260 in Foreground 1, in 16K..:-)

For many years now (since I first got into object oriented programming, in 
fact)  I have been sold on the idea of separation and encapsulation. The 
more systems are tiered and encapsulated, the lower the risk of change 
becomes. Smaller granularity presents a smaller risk profile when 
maintenance occurs.

A few years back I came across some formal stuff on patterns (the kind of 
thing that is taught today in most college CS courses). It made me smile 
when I realised that much of this was simply telling me in academic language 
what I already knew. But I'm glad this distilled experience is now available 
to the rising generation.

Today's programmers need to know much more than we did in the early days, 
and it is good to see that need is being filled by formal training.

Computer systems are observably getting better, in general. We trust our 
lives to them daily when we fly or drive or undergo surgery.

Web based systems are providing functionality wherever it is needed, not 
just within the store, but around the world.

A whole generation of "computer literate" people who are not intimidated by 
the technology and expect more from systems than we ever did, are arriving 
or in the pipeline. Technologies we could not have dreamed of 40 years ago 
are emerging and being eagerly embraced.

Programming technology is evolving to support it.

And the key ingredients in this evolving technology are objects and layers.

Pete.
