WRITE PRINT-REC AFTER ?????

On input of a record, the system doesn't know that the file is a print
format file.  It's just data.  The carriage control will print as part
of the record the way you have your statement.

Change to 'WRITE PRINT-DATA AFTER ???'.

It looks like your dealing with ASA carriage controls based on your
sample code, but hete is the complete list (all controls are print
after carriage move):
	space = space 1
	zero  = space 2
	dash  = space 3
	'1'   = skip to channel 1
	'2'   = skip to channel 2
        .
	.
	.
	'A'   = skip to channel 10
	'B'   = skip to channel 11
	'C'   = skip to channel 12
There is a possibility that the CC characters might be machine-code
which is an entirely different set of controls.  Let me know if you
need any information on them.

One thing that I found out (the hard way) was that VS/COBOL accepts
anything in the carriage control position with the POSITIONING clause
while COBOL II and above rigorously checks for valid carriage control
characters if the DCB specifies FA or FBA.

In my case, I was working with a legacy system that stacked several
reports on the same file and used a hex FF in the carriage control
position to indicate the beginning of a new report.  VS/COBOL with
WRITE POSITIONING had no trouble with this but in upgrading to COBOL II
I had to redefine the files as FB and use a straight WRITE statement
without the ADVANCING clause.

This mechanism was a leftover from ancient times when a spooler was a
separate computer with a tape drive and printer.  Stacking several
reports on the same reel of tape cut down on the number of manual tape
mounts required.  The report separator told the spooling computer to
pause so the operator could change paper.

I have read a few of the replies.  I *think* that if you just want to
"copy" the formatting of the input file to the output file - then the
way to do this is

Use the NOADV compiler option
Specify RECFM=FBA (or FBM) for both input and output file
Just copy the input carriage control byte to the output (also defined
with 133 - not 132 bytes)

****

If what you are REALLY trying to do is not copy a print input file to a
print output file - but instead CREATE a print output file with the same
formatting logic as the old program, then look in the COBOL Migration
Guide (IBM publication) which tells you explicitly how to convert WRITE
AFTER POSITIONG to ADVANCING logic.  (Or use the CCCA product - which
will do it for you automatically)
