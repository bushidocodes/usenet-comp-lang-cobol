Hi.
You don't specify how the undefined symbols relate to your program (are they
functions in the C++ code, completely unrecognisable, or something else ?), but
I can tell you that there *are* certain things you should be aware of when
intermixing COBOL and C++.

First: Calling C++ directly from COBOL is (currently) very limited.
C++ function names get mangled by the compiler (for function overloading) and
COBOL will know nothing of this (so you would need to explicitly call the
*mangled* name depending on the type of the arguments).
Also, the internal format of C++ classes are unknown to COBOL, so even if you
managed to get an object reference into a COBOL program as a pointer, you
really couldn't do anything more than simply pass it on to a C++ function (which
as above, is very difficult to achieve).

Basically, due to the fact that COBOL to C to COBOL interlanguage calls have
been made seamless, you will have all the trouble of calling C++ from C when you
try calling C++ from COBOL.
This also tells you that as C++ has a mechanism for calling C (the extern "C"
construct), this is the mechanism you would also use for calling COBOL from C++.

So, if you call "CPPFUNC" from COBOL and your C++ function is called CPPFUNC(),
I can almost guarantee that you'll see CPPFUNC as an undefined symbol (because
the resulting symbol name in the C++ object file will be a mangled version).

I *think* (but don't quote me on this - I'm sure someone who knows for definite
will follow this up) that Object COBOL will (or does?) allow communication with
C++, but not directly (using some sort of message-based system ?). You really
shouldn't take my word for this one, but it might be something you could
follow up with the Micro Focus sales/marketing department.

Second: Linking (I suspect that you aren't trying to do the above, so this is
your only problem).
To link a new RTS containing your own C code for use either by the COBOL program
or as the main program which calls COBOL subprograms, you must use the 'cob'
command. 'cob' is to COBOL what 'cc' is to C - a program which can bring
together sources (in the case of 'cob', C sources as well), object files,
assembler files, system libraries and language support libraries into an
executable whole. It can do this because it knows how system executables
are built by "cc".
With the C++ version of "cc" (I'll call it "CC"), things are done differently.
CC has to ensure that C++ libraries are made available when it calls the system
linker, ensure that C++ startup modules are linked in (eg to ensure that the
constructors for non-automatic objects are called to initialise those objects)
and ensure that the startup modules have certain information available to them
(like, a "list" of non-automatic objects that need initialising).
So, the problem is that although 'cob' knows what 'cc' does to build an
executable (and is thus, in a way, a superset of 'cc'), it does not know what
'CC' does to build an executable, so, for instance, the C++ library code is not
made available to the linker and the linker will fail when it cannot find the
called functions.

I have managed to temporarily get around the second problem for another customer
by making 'cob' call "CC" to do the final link rather than using the system
linker (ld) directly. Therefore, CC gets a chance to set up all that nice stuff
that the C++ object files need. If that *is* your problem, and you want more
information, please email me. The longterm solution is to raise an enhancement
request with Micro Focus technical support.

If you have any other questions, please let me know (via this newsgroup is fine)
Cheers, Kev.
