```
This is one of those situations where you may have done something in
violation of the rules, but it happened to work because of the way the
system was implemented.
If you have a (AMODE 31) program that acquires above the line storage, and
you pass control to one that is executing in AMODE 24, it will crash when it
tries to access a passed parameter that is physically above the line.
Previously, there was a good chance that the "anywhere" storage was
physically below the line, even though it had a 31-bit address (with the
high-order bits zeroes), so the 24-bit program could access it with no
problem.  This will not be the case in the future.
D de J
```

```
Lots of people have replied - and the answers have missed what the REAL
problem is (and when you will see the symptoms).

The problem occurs when you have a "new" COBOL program with RENT,DATA(31)
(compiler-options - and default linkage editor options) that does a DYNAMIC
call to a program that is linkedited as AMODE(24) - and the "above the line"
program passes data to the "below the line" program.

Up thru LE R2.9 in some (many? most?) cases, the storage for the DATA(31)
program was ACTUALLY obtained below-the line.  (This meet the
documentation - which only said that storage MIGHT be obtained above the
line - not that it always would be).  Therefore, when the "above the line
program" passed data (CALL ... BY REFERENCE) to the "below the line"
program, everything "worked fine" - even though you were running in an
"unpredictable" situation.

With LE R2.9 and later "preference" to DATA(31) storage was to ACTUALLY
obtain it above the line.  This meant that when you had the EXACT same
dynamic call situation, the "below the line - amode(24) - program" would NOW
be receiving data that really did require 31-bit addresses.  This would
result in "same application - no recompile - getting S0C4 - or worse" - just
because LE release was changed.

The solution to the problem is:

A) recompile the "above the line" program with the DATA(24) option to force
ALL of its storage below the line (so it can be passed to the amode(24)
program).
   or
B) "fix" the below the line program, so it can be CORRECTLY link-edited
AMODE(31).
```
