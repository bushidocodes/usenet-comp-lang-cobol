We have interactive debuggers (which I don't use).  They can be useful
in development, but when your program aborts in the middle of the night
using transient production data, sometimes all you have to work with is
a snapshot or less.  Also, turning on a debugger can change how a
program works (not often, but it does happen).

I remember a story about a program which didn't work correctly, except
with debugging statements in it.  After lots of work they discovered a
HARDWARE error!  Slowing down the program by putting in debug statements
was enough to change some of the races to make it work.  

Sometimes I see a recommendation to change a compiler option to solve a
problem.  But too many environments need to have a migration procedure
which uses the same compiler options for all programs.  That makes
debugging production programs easier.

Firstly, I have come across two 'bugs' in the cobol compiler for MVS
over the last ten plus years.  The first is the handling of intrinsic
functions (compile time COMPUTE only), the second is that you cannot
recursively perform (documentation will be changed).

The ONLY time I have seen a development compile work and not
production it was still an application bug.  It was hard to track down
but SSRANGE fixed that (checks for memory overrun).

I firmly believe in turning on every option you can to check that your
code is correct in development but turning it off in production (it
slows it down to much).  Therefore I cannot agree with your last
statement.  SSRANGE is the perfect example, slows it down by a lot but
it picks up that elusive bug very easily.

Regarding interactive debugging,  use it.  Use it to test your
programs and force them into the error conditions.  I am surprised how
many production programs actually have not had their error handling
tested.  In one job the error handling actually put the program into a
loop.
