This is copied from NE online Help...Don't see any mention of the fact that
it is only used for .int or .gnt...



Creates a synchronous or asynchronous run-unit.

call "CBL_EXEC_RUN_UNIT" using        command-line

                         by value     command-line-len
                         by reference run-unit-id
                         by value     stack-size
                                      flags
                         by reference tty-cmd
                         by value     tty-cmd-len
                         returning    status-code

Parameters:

command-line pic x(n).

command-line-len pic x(4) comp-5.
run-unit-id pic x(8) comp-5. Must be level 01.
stack-size pic x(4) comp-5.
flags pic x(4) comp-5.
tty-cmd pic x(n)
tty-cmd-len pic x(4) comp-5.
status-code See Key

On Entry:

command-line The command passed to the new run-unit. This should be the
program-name followed by any parameters.

line-length The length of the command line.
stack-size This parameter is ignored.
flags A 32-bit word indicating how the new run-unit is created as follows:

Bit Value Meaning
0 0 CBL_EXEC_RUN_UNIT returns to the caller immediately after creating the
child process.
1 CBL_EXEC_RUN_UNIT waits for the new run-unit to complete before returning
to the caller.
1 0 The newly created run-unit inherits all COBOL and run-time switches,
environment variables and open libraries as they exist in the caller at the
time of creation. See notes.
1 The newly created run-unit only inherits all environment variables as they
exist in the caller at the time of creation. See notes.
2 0 The newly created run-unit will inherit the caller's console for screen
I/O. Only one member of the coru should access this console at a time,
otherwise unpredictable results may be encountered.
1 The newly created run-unit has a new console for terminal I/O allocated to
it. This console is independent from the parent and can be used for screen
I/O.If this flag is specified then the tty-cmd and tty-cmd-len parameters
must also be specified.
3-31 0 Reserved - must be set to 0.
tty-cmd This parameter is needed only if bit 2 of flags is set. tty-cmd
identifies the console to allocate to the new run-unit or provides a command
to create a console for that new run-unit. This parameter should be a dummy
value, as this extra specification is not required by the operating system.
tty-cmd-len The length of the tty-cmd buffer. The length should always be
set to 0, as this extra specification is not required by Windows 95.

On Exit:

run-unit-id The unique handle identifying the new run-unit.

status-code Status of operation:
0  Success
157 Out of memory
181 Invalid parameter
200 Internal logic error

Comments:

The run-unit that makes the call is known as the parent, while the run-unit
created is known as the child. When the child is created it inherits certain
attributes from its parent. The attributes inherited by the child are copies
of the attributes in the parent at the exact time of the call; any changes
made to the attributes of the parent after the call are not reflected in the
child's attributes. Similarly, any changes made to the child's attributes
are not reflected in the parent's attributes.

Bit 1 flag controls which attributes of the parent are inherited by the
child.

The set of run-units containing the initial run-unit and all run-units
created from it is known as the coru.

You should be aware of the following when using run-units:

All COBOL programs in the child are in their initial state.
 Data is not shared between run-units unless explicitly specified using
CBL_GET_SHMEM_PTR, CBL_PUT_SHMEM_PTR or CBL_ALLOC_MEM.
 Open files and heaps are never inherited from the parent.

First, some background...

Although MicroFocus COBOL specifically allows the contents of a dataname to
be passed in the "By Value" clause, this is not how "By Value" is usually
implemented.
If you say "By Value" you should specify a literal. Like:

By Value    2
By Value    'A'

In other words you are passing the actual value to the called program.(This
value will be placed on the stack instead of a pointer)

When you say "By Reference" you must specify a dataname.

By Reference MY-PARAMETERS [Note that this is the default and can be
omitted]

This is because the calling program pushes the ADDRESS of the parameter onto
the stack, from whence it is popped by the called program. That's why you
have different "Calling Conventions" In MF COBOL; the conventions decide
(among other things) whether the stack is cleaned up by the calling or
called program, and the sequence in which the parameters are pushed on it.
"By Reference" means what it says; a "reference" to the parameter is being
passed.

In the NetExpress documentation where it says BY VALUE, try substituting the
actual value as a literal, NOT the dataname. So you need:

>            CALL "CBL_EXEC_RUN_UNIT" USING WS-COMMAND-LINE
>                         BY VALUE     80
…[5 more quoted lines elided]…
>                         RETURNING    WS-RUN-STATUS.

field BEFORE stringing into it (unless you are ABSOLUTELY certain that the
strung values will ALWAYS fill it EXACTLY). I know in this case you will
create an ASCIIZ  string (x'00' on the end) so it doesn't really matter, but

"BY VALUE identifier" is XOPEN, not Micro Focus specific.

>If you say "By Value" you should specify a literal. Like:
>
>By Value    2
>By Value    'A'

"BY VALUE integer" is a Micro Focus extension. As far
as I can determine, "BY VALUE (character) literal" is not
valid with MicroFocus.

Even in languages or implementations that permit values
to be passed directly, an identifier should be used to
provide meaning to the value. Generally, "BY VALUE 2"
has less meaning than "BY VALUE a-well-chosen-name".

The exception occurs when a procedure (function, program,
etc.) name or interface name is so specific that the meaning
of the parameter value may be correctly inferred from the context.
Many C functions have that characteristic. Most APIs do not.
