In my vast quantities of free time, I've been working on a Y2K CASE
tool. The tool reads IBM mainframe flat files displays the EBCDIC and
hex, analyzes the data and regenerates program source from clues
contained in the data. You can select the output language, PL/I, C/C++,
ASM370, or COBOL. We are PL/I, C/C++, ASM370, and Rexx crankers; we
don't know COBOL. I will put the program on the Internet at the end of
April but I'd like some help with the COBOL code generator.

We'll very likely make a base version of the program shareware or freeware for
non-commercial users but put a high price on it for well-heeled corporations.
The idea is that the small guy, the starving lone-wolf consultant, can
experiment with the tool and if he can apply it to a consulting job, we'll want
him to buy a discounted license from us and charge his client full list.

Hexdata.cmd is an OS/2 Rexx that creates a mainframe file containing two data
records. The file is dcb=(recfm=f,lrecl=48,blksize=48).

The record starts with four bytes of character data, the next field is five
digits of packed decimal, which is followed by five characters. The CASE tool
recognizes this pattern and generates a COBOL program.

The program contains a Y2K compliant FD and a non-Y2K compliant FD. It includes
statements to read a file and convert the file from one form to another. At
least that's the goal.

Since we don't know COBOL, we're not sure about a lot of details. Can we read
a file into an 01 associated with another FD? Do we need to generate a comp-?
for the packed decimal field and exactly what is the syntax?

And here's the hard one, what does the COBOL CALL verb use as parameters? The

You can't read one file into another file's FD. Read the record into
WORKING-STORAGE, and then write from the WORKING-STORAGE area.

In COBOL, packed decimal is designated using COMP-3. For example...

05 FIELD-1 PIC S9(05) COMP-3.
...occupies 3 bytes of storage.

05 FIELD-2 PIC S9(05)V99 COMP-3.
...occupies 4 bytes of storage, with two decimal places implied.

The COBOL CALL can call by name (if the called program is coded as a literal),
or call using a variable. The USING clause lists the parameters being passed
(always by reference). The GIVING clause (rarely used) indicates a variable
being returned. For example...

CALL 'MYPROG' USING FIELD-1 FIELD-2.
...calls a module named 'MYPROG' and sends/receives two variables. On an
AS/400, a dynamic search is made for the MYPROG pgm object. Under VSE, the
linker hunts for MYPROG in the relocatable library. On a VAX, the linker
searches for MYPROG in the object library.

CALL MYPROG USING FIELD-1 FIELD-2.
...calls a module identified by variable MYPROG (resolved at runtime), again,
sending/receiving two variables. On a VAX, you have to force the linker to
include all the possible programs. On an AS/400, calls are dynamic, so this
isn't a problem.

Depending on the implementation (VAX, AS/400, MVS, etc), there can be
extensions that permit calling by value.

The reason to avoid C is that C is not universally available on
S/390. In fact, the only universal language on S/390 is S/390
assembly language. Everything else is an extra cost add-on.

The output from REDWOOD is selectable, in a COBOL shop, you'd
click on the COBOL pulldown and generate COBOL source. Our
choices are:

1. produce the date conversion routines as inline code. The
disadvantages of this are, we'll need 4 different versions of
the date adjusting routines; each inline LOC requires about two
lines in the code generator to produce it.

2. distribute the date conversion routines as subroutines in the
native language, a COBOL set, a PL/I set, etc. This has some
advantage over the inline approach but we still end up coding and
debugging 4 different languages.

3. produce one date conversion set and call it from all four
languages. If we picked COBOL as the language for the library,
the PL/I shops would have a hissy fit. If we picked C/C++, the
COBOL shops would have to buy a C/C++ compiler to use our tool.
Everyone has assembler, so that's the obvious choice. Anyway,
S/390 assembler is more readable and maintainable than C/C++.
