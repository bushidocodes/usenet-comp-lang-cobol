What you're calling a 'misconception' here seems to be what I was taught, 
e'er-so-long ago, to call 'proper error-checking'.  Given that Data Are 
Sacred and processing them involves ascertaining their ascribing to the 
Current Rules of Sanctity - a set of rules that changes as per management 
fiat, user request, legal requirement or any one of a host of Really Good 
Reasons - the One True Way of insuring data's Sanctity is by rigorous (and 
often negative) testing.

Consider the classic example of sex.

(A brief pause will be permitted for those who feel the need to giggle 
when sex is mentioned.  This is the *only* such pause as will be allowed 
for this behavior and the rest of the example will be written at normal 
pace.)

Away back when there were two possibilities for this: M(ale) or F(emale).  
If one adheres to the dictum that 'The Source of Data is Always 
Unreliable' then it would not be uncommon to see code along similar to:

(the following example is written in the style of Away Back When)

01  WK-EMP-REC.
...
    05  WK-EMPREC-SEX  PIC X(01).
        88  VALID-SEX  VALUES 'M' 'F'.
        88  MALE-SEX   VALUE 'M'.
        88  FEMALE-SEX VALUE 'F'.
...
    READ A10752-EMPLOYEE-MASTER-EXTRACT
     INTO WK-EMP-REC
      AT END MOVE 'Y' TO NO-MORE-INPUT
       GO TO 5000-MAIN-PROCESS-EX.
...(various edit routines)
    IF NOT VALID-SEX
        PERFORM 8172-SEX-ERROR THRU 8172-EX
        GO TO   5000-MAIN-PROCESS-EX.
    IF FEMALE-SEX
        PERFORM 5348-PROCESS-FEMALE-EMPL THRU 5348-EX
    ELSE
        PERFORM 5297-PROCESS-MALE-EMPL   THRU 5297-EX.
...(et and cetera)

Notice the first thing tested is the NOT VALID-SEX condition.  The reason 
for this is that CPU time Costs Money and if there's a reason not to burn 
cycles checking for nonexistent conditions; if an employee has a sex-code 
of 'R' then the condition gets flagged and one moves on to the next.

Some shops had rules against negative condition checking; coding around 
that would be achieved by using NEXT SENTENCE... which got kind of broken 
in 1985 but that's only a couple of decades and change ago.

[snip]

familiar.  Making code do non-Standard stuff has a curious place in 
Business (CO*B*OL) environments and might be discussed in another 
thread... but the language wasn't really designed to do such things.

As for the issue of whether it should be IF NOT 'M' or 88 levels or an

data items, such as Male/Female, Branch codes, Tax rates, etc should
all be dumped into a 'Decode File' using the field name as the major
key with the code value as the minor key. The client can then decide
which codes were valid and what description was shown by adding or
deleting records in that file using a single maintenance program.

function key in the field or listed to be chosen from. A common piece
of code does the validation for the field. Given the field name, if
the record exists then the field value is valid.

records and the maintenance program uses a control file to specify,
for each decode 'field name', the headings, and other details, for the
input along with help text to guide the usage. Thus one program can
maintain all these small data items.

Then I have a single major file maintenance program which will
maintain most large data files (customer, supplier, product, stock,
etc) and dynamically builds its screen layout based on control file
information specifying the field names, field heading, location in the
record, type, validation requirement, etc. For 'small data' items
validated using the decode file it is only necessary to specify
whether an existing record is required or optional, the description is
shown or is 'NOT FOUND' and the items can be stepped through.

Over the decades I have found this enables me to put together a system
in very short time. Some systems (e.g. order entry, invoicing,
debtors, etc) went from request to running live in 6 weeks.

Interesting "semantic" question:

When talking about "programming languages in general *AND* COBOL"
  does "reserved word" mean "Key words"?

There are reserved words in COBOl which are NOT (in COBOL usage) "Key 
Words", i.e. they are never signficant. For example, "is" is ALWAYS an 
optional word in syntax (which doesn't mean that it can be placed wherever 
you want it).  There are also "English language equivalents" that add to the 
number of reserved words, e.g. "zero" and "zeros" and "zeroes"; not to 
mention "is" and "are".

Also, since 1989, there have been new words "reserved in context".  In 1989, 
the word "function" was introduced as such a word; in the 2002 Standard, 
there are lots more like this.

None of these changes the fact that there ARE more than 50 "key words" much 
less "reserved words".
