It is very unlikely that you would want to try achieving this
by reading the file and rewriting it, even if you could get
all the required specs for a Word 6 documnet.

The best approach may be looking at OLE2. This would require that
you write an OLE server (as a Windows API program), plus, probably,
a simpler program that fired up Word 6 and 'drove' it with
a bit of DDE.

If I were to quote for this work then I would want a month full-time
to learn all that is needed for DDE and OLE2, plus, say, a month
to actually implement the code.

On the otherhand it is reasonably easy to write a comma-separted
file which can be used as a mailmerge file by Word 6. Just
write the first line as the field names, second line as values:

FIELD1,FIELD2,FIELD3
122.75,14.6,293

Then it only needs the document to contain these field names at
the appropriate places, kick off a merge print and ....

Perhaps it would be better if you had Acucobol build the Word document.
We currently have an Acucobol application build an RTF document which is
compatible with most word processors and keeps most document
attributes. Some of the most advanced document attributes may not be
supported in RTF, but most are.

We started by setting up one document in Word and then saving it as an
RTF document. We then read the file as a binary sequential file and
replace the variable text as we go along. Saving a Word document to an
RTF includes a lot of stuff in the RTF document that is not required,
but this is probably the best place to start.
