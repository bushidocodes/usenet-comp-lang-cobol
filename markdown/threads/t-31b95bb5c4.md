Sterling's Gentran brand EDI translator/mapper has been around for about
fifteen years. It's available for MVS, AS/400, Unix, and PC. (The products
are NOT the same across platforms.) I've worked with Gentran/MVS and
Gentran/400 (which *are* quite similar) and found them excellent products
with good support.

Gentran is a general-purpose EDI translator/mapper and therefore is a lot of
horsepower (and expense) if you are only going to do claims (the 837) and
remittances (the 835). However, if you want to be ramped up for the whole
range of documents used in the health care industry and defined in the
implementing regulations for the 1996 HIPAA, it's not a bad choice.

Let me make a technical correction to something:
> entire insurance industry has to convert there [sic] present computer
systems to be HIPAA compliant.

It's not just insurers (payers), it's providers, DME vendors and
laboratories, too. And, a computer system is not HIPAA-compliant: only
'covered entities' are HIPAA-compliant. For details see CFR 45 parts 160 and
162. (Or engage a consultant to get you started). (Someone like, say, well,
me).

My latest project is related to the HIPAA-mandated ANSI EDI electronic
remittance advice . You can view info about the product at
www.providerpaymentpartner.com

Yes, I have a clue.

It means that the file you are accessing has become corrupted. Maybe there
was a power failure or the machine was switched off while it was being
accessed, or you haven't re-organised it for a 1000 years. (Or any other
reason why...<G>)

Your file consists of two parts:  A series of index entries and a series of
data entries. These parts are sometimes physically separate (.DAT and .IDX)
and sometimes combined into one physical file. (It depends on the particular
vendor). In EITHER case Indexed files require re-organisation every so
often. This is so that unused space can be reclaimed, the indexes can be
reorganised for better efficiency, and the user can feel a bit more secure
about his data ...<G>

You must run the MicroFocus REBUILD utility, (supplied with your MF COBOL
system), against the file giving the error.

Read (carefully) the chapter on using it in the MF Manuals. If you still
have a problem, send me the file (both components of it, if it has two...I
can't remember the format for old MF) and I have an old machine here that
has MF COBOL on it and will rebuild it for you then send it back.

> It compiles ok in microfocus.

You will have realised, after reading the above, that this is because the
problem is not with the PROGRAM, it is with the DATA.
