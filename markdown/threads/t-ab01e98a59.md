I hope that the below is applicable to Visual COBOL.  Whether it is or not
let me know if you have the time to respond back.  I use UDB 6.1 NT with
database stored on a Server with MicroFocus COBOL MFE (Mainframe Express).
You may be encountering the basic restriction of UDB that you may not be
aware of regarding BInding of "Packages" (or Collections).  In UDB the first
person that binds a specific program becomes the owner of the package and is
subsequently the only userid that has the access rights to re-bind (the
pgm's Package) and is the only userid that has the access to even execute
the pgm (the pgm's package).  Therefore, the procedure to follow is that
when a new cobol UDB program is created and compiled for the first time, the
first person that compile-binds it must be the person (userid) to issue the
below (2) commands in Control Center:

                        GRANT Package-Name  BIND PUBLIC
                        GRANT Package-Name  EXECUTE PUBLIC

    Note - In above commands "Package-Name" is composed of
"CollectionName.ProgramName".
                For Example:

                        GRANT ABCSYSTEM.PGM1  BIND PUBLIC
                        GRANT ABCSYSTEM.PGM1  EXECUTE PUBLIC

Note: Only the first person that complie-Binds the program has the access to
grant access to other users for re-binding or even executing the program.
The exception of course is that the DBA who has ADMIN access rights can also
issue the above UDB commands to grant re-binding and execute access of the
Package to other userids (a.k.a via access to "PUBLIC").

Note: You should specify a Package "Collection-Name" when you bind your
programs (I do not know how you do this in VisualAge, since I don't use this
package.  If you don't specify a Collection Name when binding, I believe UDB
will subsitute the Userid of person binding as the actual Collection Name to
be used for the program (this is then even more complicated to keep track
of).  You should come up with a Package "Collection Name (a.k.a. Collection
Schema) which is unique to your system and if you do not authority to
"Create Implicit Schema Name" then the DBA will have to create the
Collection "Schema" Name you decide on.  Then you should enter the
"Collection Name" for your application system into Visual Age for Binding
your programs after compilation (I do not know Visual Age, so I am just
assuming it has an option to specify a Collection name.  If your binding by
issuing a command to UDB in command center or caomand window there is
definately a way to specify it, but I could not tell you how since I enter
my collection name in MicroFocus COBOL MFE and it sends to UDB when
interafcing with UDB for binding operation).

I received the following from one of my "usually reliable sources" at IBM.
I don't know how helpful it is - but I pass it on anyway.

"They have to make the 'integrated precompiler' part of  DB2 available to
the user who is compiling, I am not sure if those services can be shared or
not.  They could have a workaround by using the precompiler as a separate
step of compiling, but it probably has similar restrictions.  If it does
work for remote users to use the precompiler, then there is either a setup
problem for the integrated precompiler or maybe each user is trying to use a
different userid or
something.

I would recommend more experiments to narrow the problems down, by comparing
the 2 different methods of compiling programs with SQL statements.    I
would read the
DB2 manuals and also see if the remote users can RUN the programs that
access
the DB2 Data Base.  It may be a general remote access to DB2 problem rather
than
a remote precompiler/integrated precompiler problem."
