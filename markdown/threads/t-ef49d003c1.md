Here is a link to my comments on the draft standard.

http://cobolforgcc.sourceforge.net/cobol2002comments.html

My main gripes are

- too many features of dubious value are mandatory (screen,
report writer)
- OO should be put in a new language "Object COBOL" rather than
being mandatory in COBOL2002
- too many ridiculous restrictions eg cannot redefine a pointer.
- too many magic numbers eg line length must be less than 255,
literals must be less than 160 bytes, PIC must be less than 50
etc etc etc
- editorially, there is too much use of undefined terms - the
list of definitions and index need major work
- the tightening of copyright

fixed width font. I will upload html to my web page fairly soon
and post a link.

I am not late submitting this becasue different deadlines apply
outside the USA. However I realise that not many of these
comments will be accepted, apart from the typos.

> Dubious value?  Report writer, at least, has a lot of value for shops

Many shops actually ban the use of report writer. IBM has not
even implemented it in their mainstream compiler (though they
offer a preprocessor that you can use). Some colleagues did some
tests and found it took longer and took more lines of code to
write various reports with report writer than without it. The
Tiny COBOL project elected not to implement it, and instead
provide a facility that allows you to define fonts etc. I think
it is fair to say that the benefits of report writer are dubious
(=able to be doubted)

I don't have a problem with people using report writer, or with
people implementing it. But I disagree that it should be a
mandatory core part of the language.

> 
> >- too many ridiculous restrictions eg cannot redefine a pointer.
…[6 more quoted lines elided]…
> 

Agree. I suggested to the committee that they adopt the following
principles

- implementors are encouraged to make 'available memory' the
limit where possible
- they must document any limits
- where they have fixed limits they must meet minimum standards
- the minimum standards should not be a large set of random
numbers (like 31, 50, 150, 255) but should be a small set of easy
to remember numbers eg 10, 100.

The old report writer was hard to use because it didn't give any good visible
clues as to why programs abended while execution occurred during the report
writing routines.

The preprocessor is functional for the current version of Cobol, but it's not
maintainable. It is not readable and it goes against the concept of structured
programming where maintainability and modification should be accepted practices
in good program development. In fact, it seems to be much easier and more
productive to rewrite the whole program and coding your own reporting routines.

Someone mentioned DYL-280 about its facility in report writing. If Cobol can
imitate the power and ease of DYL-280, that would be an added engine for Cobol
programmers without having to learn new languages. I know that NATURAL has
incorporated these facilities for report writer.

As for screen development, the current interface to CICS should be sufficient.
The flow between these two domains need somewhat to be improved. From my point
of view, the problem around here is more cultural than technical. I say this
because I've seen NATURAL can do either way, screen development as good as CICS
or reporting writing as good as DYL-280, without making a fuss.

But then NATURAL is proprietary while Cobol is a common language that is being
attempted for compromise by different vendors according to specific standards
of agreement.

Chim
