I think "Structured COBOL Programming" by Stein & Stein is quite good.  It's
designed as a text book - careful and complete explanations and lots of
exercises.  It does not cover the complete COBOL language but there's enought
in there anyway.

One thing I really liked about the book is how the authors contrasted COBOL 74
and COBOL 85 style of programming.  COBOL 85 gave us lots of changes that
fundamentally (allows you to) change(s) the style of programming.

"COBOL Unleashed" is a complete presentation of the COBOL language.

When I was re-learning COBOL after not having written a line of it in 20 years,
I found the Stein & Stein book easy to read and quick to learn from.  The
Unleashed book then became an invaluable reference once on the job.

So what did he expect?

A bounds violation on a table can give you at least four different types of
errors/abends:

If you overrun the table into your own code segment, good chance for a
(S0C1?) "Invalid opcode" abend.

If you overrun your table into memory your application does not own, count
on a (S0C4?) "protection exception" ("unavailable storage") abend.

If you overrun a table but stay in memory your application does own, you can
corrupt other program data and get the aforementioned S0C7 "data exception"
abend.

And, you can overrun your table into memory owned by your application,
corrupt your data, not suffer a S0C7 data exception, but gee whiz can you
get some weird output!

... Which is why every mainframe COBOL programmer should learn how to
compile a program which uses tables using the SSRANGE compiler option when
the program needs debugging. (And how to compile it back for production with
NOSSRANGE, and why you don't use SSRANGE on production code).
