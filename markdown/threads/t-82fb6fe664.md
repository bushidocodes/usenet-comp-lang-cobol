```
Okay, I know I am opening up a small can of worms with this, but I really could use some advice. I find myself in the really weird
position of not being able to decide what compiler / platform is best for this job.

The beastly program will either have to be accessible from the Web, easily, reliably, and very efficiently (i.e. - cannot use
WebSphere for this, we are talking about a plain jane text mode application) *or* it is will have to be easily and quickly
distributable, preferably with no runtimes, on PC's and AIX machines.

I'm in quite a funk on this. My initial call is to just write a CICS application and let the users login to it over the internet.
The downside to this is the expense of doing it on the mainframe. For a while, I was looking at Flex-ES to do it, but the commercial
licensing is still running around the $100K mark - 5 times more than the budget for this project will allow.

Then I thought about using CICS under Windows NT, OS/2, or AIX. AIX looked great, but CICS has some issues - I'm not in Rag's class
and  I could not get the damn thing stable, even using DB/2 as the backend. On top of that, it was slow indeed... On a side trip
from that, I though - ah hah! I'll just call C programs and use curses to do the user interface from IBM Cobol directly under AIX.
This sounded great, except the doggon COMPILER would not successfully open files with more than one index on them. (i.e. Alternate
Index) I don't know why, and have NOT figured out why either.

So then I thought, ah hah! I have an old version of MicroFocus COBOL for AIX, I wonder if it would run... and yes it does. But it is
version 3.2, and that is pretty old I think. Also, I am not sure if I can legally move it from the very old and slow AIX machine I
have it on to a faster and more efficient one. It has a few problems with termlib/termcap too, but those are probably fixable.

Moving along, I remembered I have a developer's version of Cobol ReSource (a Wang VS emulated COBOL) on the AIX machine, and it is a
very fast compiler. Licensing for it is not terribly expensive, and I know the guy who supports it down in Australia. This looked
good, but has two small issues - since it want to pretend it is a Wang, it insists on using 32 function keys, which means I would
need to distribute a copy of PuTTY that Tom Brennan modified for me to use file based configuration files and allow really good key
remapping. This may be a wee bit of a hassle for the user, but it was not a major problem. However, the compiler is not up to snuff;
it is supposed to be a 100% compatible Wang VS COBOL-85 compatible product, but it doesn't have a lot of the extensions in it-
Intrinsic Functions, Address data types, etc. The people using it commercially usually don't care, since they are still running and
supporting COBOL 74 programs for the most part.

Having exhausted the my AIX inventory, I briefly glanced at the Tiny COBOL and KOBOL sets, but was not terribly impressed. So I
moved into the PC commercial stuff I have around here.

IBM COBOL is beautiful, as close as I can get to the mainframe. I can run a copy of CICS under OS/2, use IBM COBOL, and allow up to
20 simultaneous users to connect via TN3270, or just about any number of people to connect using a CICS client via TCP/IP.  This is
essentially a zero cost option for me, since I have OS/2 CICS and the IBM compilers. But I don't have a workstation setup with OS/2
to develop and run it on, and I am unwilling to give up one of the high speed workstations/servers to do so unless I absolutely have
to. It does give pretty good performance though, and it is very compatible with the mainframe. I thought about using VA Cobol and
the PC version of the CICS development environment under WIndowsXP, but the CICS development version cannot run under WIndowsXP.
Guess what fool upgraded all the workstations here to WindowsXP? Stopped crashing, but a few pieces of software simply will not run.
(*sigh*)

Netexpress and AcuCOBOL under Windows run nicely, but the redistribution is a bit of pain. Besides, it feels like going to a small
system deployment, and my customers won't like that. They won't really like anything on a PC I think, unless it is running under
Linux or some other form of UNIX. Same kinda applies to Fuji COBOL, which I like a whole lot.

My last point of endeavor was on a AS/400, which is quite doable, but adds a slight extra cost per user to deploy, and it means I
have to use RPG like screen definitions. Still it is doable.

So all this comes down to several options that are doable, but one of those, or something I simply have not thought of at all, is
going to be better than the others. Please feel free to post your opinions, even if your opinion is that I am nut. :) I am usually
not this wishy-washy, but this is a "showplace" application I need to write, which will hopefully lead to significantly more
business.

Thanks in advance guys. Please tear that thinking apart!

-Paul
```

```
Well, I want to say THANK YOU to everyone here for all the help that has came my way on makin the difficult descision for this one.
The info below is purely my personal opinion, and should not be taken as review for any of the products mentioned. :)

I finally gave up, loaded OS/2 on a Gigahertz computer with 80 gigs of mirrored drive space, and decided to use a slightly older
version of OS/2 CICS (one that is almost 100% compatible with CICS ESA 4.1,. It is labeled as Transaction Server V4.1 for OS/2.).
This provides a nice development environment very close to the Mainframe world without the excessive Mainframe cost. It also allows
me to easily migrate this to a mainframe at a later time should the (hoped for!) need arise.  This uses a backlevel IBM COBOL
compiler (Version 2.2 Enterprise) and runs fine.

I had some trouble getting OS/2 Warp Server for E-Business to deal with an 80 gig drive space, but it eventually gave in.  The end
result for uses is a simple TN3270 environment and/or a CICS Client environment that works at a nice peppy speed over the internet.
It also is not difficult at all to interface to a Web Server with setup, and I have the advantage of having done so before. :)

The top contenders, in order, were:

Fujitsu COBOL for Linux
IBM COBOL Set for AIX
Liant COBOL for Linux
MicroFocus COBOL under AIX
AS/400 COBOL
COBOL Resource under AIX.

Fujitsu appears to have a beautiful product, no runtimes, etc. The cost per developer is $1500, which is a good bargain. I had a
little difficulty getting to someone who understood what I meant by a 'Green Screen' application, though I eventually did. The
reason I choose not to go this path was just some nagging uncertainty in using so new a compiler, and a little bit of concern that
the compiler may not be able to handle terminal screens as wells as I need. I also have a generic worry that a UNIX TERMIO based
solution would give me 'jerky' screens under heavy processor or network load.
```
