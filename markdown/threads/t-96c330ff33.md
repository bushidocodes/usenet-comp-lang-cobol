"tdsgh.hrdat.workfile(jcltorun)" and WRITE it out. in your JCL, you use
SYSOUT=(*,INTRDR) in the DD statement. this writes the JCL to the

would work if MSGCLASS is a hold class but you can change it to any class
your system provides you.

If you're running under ISPF, there's a whole 3GL API that makes all dialog
management services available (e.g., ISPEXEC SELECT CMD(...), file tailoring,
etc.) - consult the ISPF dialog manager manual for COBOL examples.

If you're running in batch or native TSO and just want to submit a job, you
could write to a DDNAME that's allocated to the internal reader (e.g.,
//ddname DD SYSOUT=(A,,INTRDR)).

I could be wrong but native COBOL is not designed as a systems
language. It is a business application language.

You could acheive your purpose by using a add-on feature, if you want
to learn it, known as ISPF dialog for COBOL. Your mainframe may have
it since it's a popular product. But you need to have your program
totally within the ISPF environment to acheive your purpose. That is:
panels (to interact with the program for variables and parameters),
skeletons (to combine the variables and parameters with the JCL),
tables (to store the variables and parameters from one session to
another to avoid retyping), and dialogs (to make it all happen). I've
learned it on my own some years ago much to the hostility from my
computer center (who thinks ISPF is only a systems interface) and will
be happy to post some samples if you're interested.

The book that helped me to learn how to use COBOL with ISPF is "Using
IBM's ISPF dialog manager under MVS, VM & VSE" by Howard Fosdick. It's
likely out of print but check your college Library. It's stack number
QA76.9 I58 F67 1987. Expect to spend time learning and experimenting.

Hope this helps,
