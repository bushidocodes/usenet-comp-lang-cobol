We've used the IBM OS/2 and NT products, both the older and the newer
versions. We have also used the IBM OS/2 with MicroFocus and other third
party development tools. Lastly, we have used the MocroFocus Transaction
System with CICS Option (what I think is the current title for MF's
product). All of these are supposed to be real transaction monitors for
production use, rather than the emulators which MF and CA used to have. I
cannot tell you whether CA's current product is also a transaction monitor,
or just an emulator, as I do not have a current copy of it. I also do not
have MicroSoft's current entry, although I have read that they have a
product which they are not marketing as such, but which does many CICS like
things.

There is a TREMENDOUS difference between these various products, in what
they do, how well they do it, and how you set them up to do it. For this
reason it is important to ask are you using this just to learn again, or do
you intend to actually run code for production?

It also is importatn which COBOL you plan to use. For example, I find that
the MF COBOL 3.x workbench works far better with the IBM CICS-OS/2 Option,
which MF was selling as an alternative to their own product at one time,
than with the MF Transaction system. If you go to the MF 4.x platform then
the reverse is true. This is largely because of the way MF integrates
their workbench in the two different versions. If you use IBM Visual Age
for COBOL, then go with IBM Communications Server for NT or OS/2 (of which
there was a free evaluation not too long ago..check the IBM site.) Realia,
RM, Liant, or AccuCOBOL...I don't know.

When you get one of these you really need two parts, the translator (to
precompile your code), and the actual running environment (which you need
to test and/or run production). If you are trying, for example, to compare
legacy translated code form libraries on a mainframe, you need to be aware
that the translate sets up calling parameters which may be different
depending on what translator you use, and what runtime it is translating
to.

Some of the key problems you will run into with the runtime environments
are slight variations between the mainframe CICS you are trying to learn
about and the PC product. These can be little things like "Which EXEC CICS
calls are supported? All of them, or just a subset?"; "How does storage
initialize on a getmain - low-values, garbage, or ???"; "Does the COMMAREA
retain it's contents between transactions (it shouldn't, but does on some
PC products..or maybe with some compile options)?"; "Can you test from an
external 3270 terminal and/or emulation program, or only from within the
workbench environment?"; "Do the internal tables (which are the soul of
CICS) have the same formats? If you write code to access them (which is
not all that rare) will it run under the PC version?"; etc., etc., etc....

The major problem for our testing of PC developed systems to be run on a
mainframe (not client/server, just copied over at source code level and
recompiled) was getting test data onto the PC platform. This is a major
nightmare in most cases of PC based development for a mainframe target,
CICS, IMS, or whatever-flavor-you-use. And, the answer to which to use can
be dfferent depending on whether you are using just VSAM (they all support
it), or whether you want IMS (probably better go MF), or whether you want
SQL (XDB ==> MF, DB/2-OS/2, or Universal DB==> IBM Communications Server).

Then, there is the problem of trying to tie Fujitsu COBOL to a CICS
emulator. I'm working on it in my spare time....

The bottom line is that if you just want a reference manual and a simple
translator to check your syntax on the commonly used calls, any of the
products which include a translator will work fine. If you want to develop
some limited function systems on the PC and test them, find the one that
works with your COBOL platform, and for which you can generate test data.
If you want to use a Fileshare server, MF. And if you plan to run real
production, either PC based, or in inter-region communication mode, you
probably want the IBM products (OS/2 or NT based), as they come much closer
to the real thing.

Good luck...
