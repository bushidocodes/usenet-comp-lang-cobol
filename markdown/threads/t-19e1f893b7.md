Here's the scoop.

9/018's on rebuilds are BAD,  

If you do: 

REBUILD BADFILE.DAT

and get a 9/018, then do a 

REBUILD BADFILE.DAT, NEWFILE.DAT 

You will get a successful reubuild - but you will have lost records.

What we did in this situation is as follows:

If you get a FILE ERROR 9/018  IMMEDIATELY do a:

REBUILD Oldfile.dat, Newfile.dat -D

The -D switch will ignore the 9/018 and go on.

However, if you JUST rebuild without the in and out and get the 9/018
on that rebuild you are toast. Restore from backup.

Realia COBOL but that doesnt' really make a difference.

1.  Files open INPUT only unless ACTUALLY being updated.
2.  For update Open, Update, Close, re-open Input if a subsequent read
is necessary.
3.  Weekly file reorganization that does the equivalent of a Rebuild
-D is required.

I have in, 7 years only had a single file corruption and this was when
we moved to an NT server from a Lantastic Peer to Peer setup.
