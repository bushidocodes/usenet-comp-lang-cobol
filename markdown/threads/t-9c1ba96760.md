```
  The reason for that (as has been stated in a few replies here)
is that any program can access *only* it's own copy of the master
environment.  When your program terminates, its *copy* of all the
environment variables disappears.

  Note that DOS's command.com is also such a program.  You can
see how this works very easily.  From a DOS prompt, type COMMAND.
This will start another copy of command.com.  Type SET.  This will
list all of the variables in effect for this copy of command.com.

 Now change a few variables using SET commands.  Type SET again to 
to see the list of *new* values for the environment variables.  
They should be just as you had set them.  Finally, type EXIT, then 
type SET again, and you will see that the original values have 
been restored, just as though you had never changed them.
```

```
You must be doing something very strange with your 'calls and
cancels'.

But whay are you using the environment varaibles anyway ?  If these
are Cobol programs then it is much better to use a block of memory
that is passed to each program.

Generally my systems have a small core routing that holds the
interface data and calls/cancels every other program that is required.
 The menu is just one of these.

Main rouitine:

        WORKING-STORAGE.
            COPY "interfce.ws".
       *    includes fields Current-Program, Next-Program,
Menu-Program
       *    plus other intercaing data items    
        PROCEDURE DIVISION
            INITIALIZE Interface.
            MOVE "mymenu"       TO Next-Program
                                   Menu-Program
            PERFORM
                UNTIL Next-Program = "FINISH"
                MOVE Next-Program   TO Current-Program
                MOVE Menu-Program   TO Next-Program
                CALL Current-Program
                    USING Interface
                    ON EXCEPTION ...
                END-CALL
                CANCEL Current-Program
            END-PERFORM
            STOP RUN.
        
mymenu Menu program

        WORKING-STORAGE SECTION.
             ...
        LINKAGE SECTION.
            COPY "interfce.ws".
        PROCEDURE DIVISION USING Interface.            blah blah blah
                 MOVE "myprog"    TO Next-Program
                 EXIT PROGRAM.

Note that if 'Next-Program' is not set by any program then the
mechanism reverts to the menu program but any program can set what is
to be called 'next'.

Any data that needs to be remembered between programs is put into the
Interface data.  If the size changes then the main program _must_ be
recompiled.
```

```
Lybush wrote:

> I'm still using Dos Cobol for a living and am stumpted in a program. I
> need to set an Environment variable while in a program and can't get
> it done. I've tried running a batch file from the program and

There is not just one 'environment', when you execute a new program a
copy of the environment is made for that program and any changes that
happen there are lost when that program exits.

           DISPLAY nameofvariable     UPON ENVIRONMENT-NAME
           DISPLAY valuerequired      UPON ENVIRONMENT-VALUE

The other problem that you may hit is that the environment is created
only large enough to hold the current values.  This means that trying to
set a new variable will fail because there is no room for it.  You will
need to force room in the environment by either having the variable that
you need set to some dummy value in AUTOEXEC.BAT or by setting some
dummy variables that you can set to blank to free up space.

In Autoexec I usually have:

SET ZZZ1=ZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZ
SET ZZZ2=ZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZZ

Then I can:

           DISPLAY "ZZZ1"             UPON ENVIRONMENT-NAME
           DISPLAY " "                UPON ENVIRONMENT-VALUE
           DISPLAY "NEWVAR"           UPON ENVIRONMENT-NAME
           DISPLAY "VARVALUE"         UPON ENVIRONMENT-VALUE
```
