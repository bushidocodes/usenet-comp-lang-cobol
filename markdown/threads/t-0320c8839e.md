typedef struct
{
unsigned nibble:4[16]; /* array of 16 half-bytes 4 bits each, 16 different
nibbles */
} comp_3_of_15_plus_2;

or

typedef struct
{
char bite[8]; /* array of 8 characters 1 byte each, 256 different chars */
} comp_3_of_15_plus_2;


pic s9(13)v99 comp-3 field, which value is e.g. 1234567890123,45+ is something
like this in hex:

12 34 56 78 90 12 34 5X where X is almost anything depending on environment.
(C or D (or F) in MVS)
or
5X 34 12 90 78 56 34 12 depending on some (compiler?)switch.
(in this case you have to define 2-level array)

If you have a cobol compiler you can test it:

identification division.
program-id. test0001.
*author. veli-matti truhponen.
environment division.
select file1 assign to 'file1.dat'.
data division.
fd file1 organization is sequential
access mode is sequential
data record is rec1.
01 rec1 pic x(8).
working-storage section.
01 field1 pic s9(13)v99 comp-3.
procedure division.
begin-here section.
open output file1.
move +1234567890123,45 to field1.
perform just-do-it.
move -1234567890123,45 to field1.
perform just-do-it.
move 0 to field1.
perform just-do-it.
close file1.
stop-here section.
stop run.

just-do-it section.
move field1 to rec1.
write rec1.
just-do-it-ex section.
exit.

and then you can use any hex-editor to see what kind of halfbytes (nibbles)
there are.

there should be enough hints for you

rgds, Vellu
Ps. there may be easier ways to solve this, haven't thought about it much

You need to identify the platform and which implementation of COBOL to
predict the format with complete confidence. But you might try writing
a little C program which takes a string-pointer as its argument, and
gives you a hex dump of n bytes up or down the stack. Especially if you
know somehow what values should be passed, and/or can control the input
to the COBOL, then you should be able to find the values in the dump and
determine how you should delcare it in C. (If you want a nice little
C hex dump routine I'll send it along, but a simple printf should do it.)
