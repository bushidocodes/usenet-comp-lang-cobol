For the past 27 years I have worked as a Freelance in Europe. (UK, Spain,
and Germany).

I worked when I wanted and stopped when I didn't.

This gave me a brilliant lifestyle and I have no regrets whatsoever about
it.

In all that time I was never unemployed for more than 10 days when I didn't
want to be.

Until last year.

The last contract took 3 months to find (to be fair, I turned down several
possibilities because they didn't appeal) but, nevertheless, I was starting
to get decidedly anxious by the time the "right" job came along.

I am convinced that the "boom" is over (well...Duhhhh <G>) and we will never
make the money we made in the 80s again.

However, there are many ways to skin a cat and writing COBOL is not the only
satisfying way to make a living.

Certainly the skills and business knowledge acquired from being a long term
COBOL programmer can be leveraged into other areas. These days I find there
is a lot more money in management than there is in programming and, as a
result, I haven't made a living from COBOL for close on 15 years now.

Others who are younger may find it advantageous to acquire more technical
skills. It is like ANY business; find out what is needed and wanted, then
provide it.

(Now where did I leave my macrame...<G>?)

Pete.

As you mention, this sort of problem has been solved with .NET. 

One of the "requirements" that a "language" should meet in order to
really call itself a .NET langauge is that it be Common Language
Specification (CLS) compliant. That is really a big benefit to us
developers.

So, here are the VB->COBOL and COBOL->VB scenarios that I've explored:

Managed object to Managed Object:
Yes, I've been able to call "COBOL from VB". For example, using the
latest of the two (Fujitsu's NetCOBOL for .NET and Microsoft's
VB.NET), the two work very nicely together. I've tried this with
in-process (DLLs) in a COM+/Enterprise Services configuration as well
as with out-of-process objects(EXEs) - no problem.

By the way, the COM+ configuration does bring into the picture a few
additional steps: (i.e. cleanly exporting a typelib for the COM
object, merging the entry into the windows registry, etc.). But, it is
certainly doable. COBOL to VB and VB to COBOL.

Mixed (managed and non-managed):
If you happen to be mixing legacy/non-managed objects (COM and
in-process) of either lang with a .NET (managed) object of the other,
then the "com-interop" feature of .NET comes to the rescue. Wether you
are using a runtime-callable-wrapper (RCW) - a COM object living in
the .NET world, or a com-callable-wrapper (or CCW) - a .NET object
living in the COM world, the language that you happen to use is
secondary.

However...

In this com-interop scenario, there is one big gotcha, you may have
some issues with the type conversions - for some data types. I've
found that explicit type conversions to be helpful in these cases (on
the VB receiving side). Otherwise, using simple data types decreased
the chances for complexity.

Other...

Beyond that (and perhaps...beyond your original question), I've found
that exploring the object serialization feature of .NET to be
fantastic. In other words, .NET makes it really easy to create an
object, (decorated with a serializable "attribute"), serialize (deep
or shallow) the object and store/transport/pass the object. Then, you
can later deserialize the object for use. A good friend of mine refers
to this as the Star-Trek Transport approach.
