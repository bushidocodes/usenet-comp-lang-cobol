I have written something like that in C, but it could be done in COBOL just
as well.  The algorithm works something like this:

If you have "n" checks outstanding, you have 2**n possibilities of including
or not including each check in a compare sum.  Of course the amount of each
check would be in an array, as would an array of multipliers (PIC 9 will do
nicely, they can only contain a zero or 1).

varying x from 1 to (2**n - 1)
    convert x to binary (divide x by 2 "n" times, save remainder in
multiplier array)
    zero a compare sum area
    varying y from 1 to n, add check-armount(y) * multiplier(y) to sum area
    if the sum area equals the compare amount (constant entered at
beginning) or
      (optional) the difference is less than any previous difference
         display all check amounts with a 1 as their multiplier

We leave the coding of this as homework.

Gary,

I hope you realize what you are running into. There is more than one issue.

1. The number of unique combinations among N elements if the order of
elements in a combination matters not (evidently, your case) is 2**N. As you
have N=100, it translates into approximately 1.27E30. You have got to have
an awfully fast parallel machine to be able to even test the sum of each
combination for equality to 34.22, to say nothing of creating the
combinations.

2. Down to the nitty-gritty level, let's see what kind of code one has to
concoct to do that. Let's start with very modest N=4:

PERFORM VARYING Q1 FROM Q0+1 BY 1 UNTIL Q1 > 4-(1-1)
   COMPUTE SUM = Q1
   IF SUM = 34.22 DISPLAY Q1
END-PERFORM

PERFORM VARYING Q1 FROM Q0+1 BY 1 UNTIL Q1 > 4-(2-1)
PERFORM VARYING Q2 FROM Q1+1 BY 1 UNTIL Q2 > 4-(2-2)
   COMPUTE SUM = Q1 + Q2
   IF SUM = 34.22 DISPLAY Q1 Q2
END-PERFORM
END-PERFORM

PERFORM VARYING Q1 FROM Q0+1 BY 1 UNTIL Q1 > 4-(3-1)
PERFORM VARYING Q2 FROM Q1+1 BY 1 UNTIL Q2 > 4-(3-2)
PERFORM VARYING Q3 FROM Q2+1 BY 1 UNTIL Q3 > 4-(3-3)
   COMPUTE SUM = Q1 + Q2 + Q3
   IF SUM = 34.22 DISPLAY Q1 Q2 Q3
END-PERFORM
END-PERFORM
END-PERFORM

PERFORM VARYING Q1 FROM Q0+1 BY 1 UNTIL Q1 > 4-(4-1)
PERFORM VARYING Q2 FROM Q1+1 BY 1 UNTIL Q2 > 4-(4-2)
PERFORM VARYING Q3 FROM Q2+1 BY 1 UNTIL Q3 > 4-(4-3)
PERFORM VARYING Q4 FROM Q3+1 BY 1 UNTIL Q4 > 4-(4-4)
   COMPUTE SUM = Q1 + Q2 + Q3 + Q4
   IF SUM = 34.22 DISPLAY Q1 Q2 Q3 Q4
END-PERFORM
END-PERFORM
END-PERFORM
END-PERFORM  

I've adopted the Q-notation intentionally to make the induction pattern
clear. To go while nine yards to N=100, you will need 100 blocks whose
number of nested loops increases from 1 for the combinations containing 1
element to 100 for 100 elements. Of course, the latter will loop only once
(there is only 1 unique combination of 100 items out of 100), but you will
not be able to compile it. Heuristically, you can replace the last loop with
a straight test, but how about 50 out of 100? I doubt that COBOL permits 50
nested loops, anyway.

3. Even all the above points had nothing to do with reality, hand-writing
code like this for mere N=20 is, to put it mildly, tedious. You realize, of
course, that I did not do it - I let SAS Macro Language assemble it for me
with the absence of which, you will have to write a COBOL program writing a
code similar to the one above to a file, then compile and execute it. 

4. With that in mind, I would recommend doing it in SAS. There is no limit
as to the number of nested loops, and it does things like that rapidly
enough if you limit yourself to reasonable N=25. If you decide to test only
K combinations out of N, then the number of nested loops is determined by K
< N. For instance, with K=2, N=4, all such combinations will be given by the
2nd block of the code above. However, as you know, the number of the
possible unique combinations in this case is N!/[K!*(N-K)!], so you will
have to assess it first to decide if it makes sense to run the thing at all.
If you are interested in SAS code doing what you need (once again, provided
N is not large!) here it is (ready-to go; just change N accordingly and
populate the array T with your values):

%LET N  = 10;
%MACRO COMBINE;
   Q0 = 0;
   %DO I=1 %TO &N;
      %DO J=1 %TO &I;
         DO Q&J = Q%EVAL(&J-1)+1 TO &N-(&I-&J);
      %END;
      IF Q1
      %DO X=2 %TO &I; + Q&X %END; EQ 34.22
      THEN PUT
      %DO X=1 %TO &I; Q&X T(Q&X) %END;
      ;
      %DO J=1 %TO &I; END; %END;
   %END;
%MEND COMBINE;

DATA _NULL_;
   ARRAY T (&N) _TEMPORARY_
    (1.87 4.76 2.17 9.44 2.13 9.13 23.10 0.47 7.39 62.5)
   ;
   %COMBINE
RUN;

This is actually a problem to which our company has a solution.  The
situation is that we need to determine programatically which invoices a
customer meant to pay when he has several outstanding invoices and sends us
a check for amount X.  So the problem is to find all combinations of N
invoices, limited to M items in the set, which match amount X.  Currently N
is limited to 100 and M is limited to 10.  What makes this easier to solve
is to order to N items in ascending order and stop whenever the sum of the M
items is +AD4-X (i.e. a check for +ACQ-100 is obviously not meant to pay two
invoices of +ACQ-60 and +ACQ-70, nor will adding more invoices to that list (which
will increase the total) arrive at a match either.

The actual code is a little more complicated than that, since it takes into
account credits as well.  We did not code it in COBOL since we wanted very
tight code so it's written in assembler, but it could have been done that
way if we were not as concerned with efficiency.  As it is, the amount of
CPU time consumed is fairly minimal, despite the fact that we process
hundreds of payments every evening.

The code is proprietary so I can't share it here, but the basic outline I
have given should suffice.

FYI, you can improve the 'nested loop' algorithm by pre-sorting the list by
amount. You can ignore any cheques which are larger than 34.22. For the
nested loops you can stop processing when the sums exceed 34.22.

Here;s a little rexx which handles a list of numbers. In this case the
number is also the index in order to simplify the example. It handles
combinations up to 3.  Recursion would simplify the algorithm...

    do i = 1 to nChecks
    if c.i = target then say c.i'='target
    if c.i > target then leave
    do j = i + 1 to nChecks
     if (c.i + c.j) = target then say c.i'+'c.j'='target
     if c.i + c.j > target then leave
     do k= j + 1 to nChecks
       if (c.i + c.j + c.k = target) then say c.i'+'c.j'+'c.k'='target
       if c.i + c.j + c.k > target then leave
     end
   end
 end
