My shop does not have COBOL 370 yet, and some die-hards are still using OS/VS
COBOL. From what I have read the only differences between VS COBOL II and
COBOL 370 is the use of LE/370 runtime libraries and the availability of
intrinsic functions. A COBOL II program should compile under COBOL 370 with
no changes, not even to embedded PROCESS or CBL compile-time options. The
behavior of the generated code should be the same.

Therefore, the effort to convert a program from OS/VS COBOL to COBOL II or
to COBOL 370 is identical (no COBOL 74 program will be using intrinsic
functions).

COBOL II is not ANSI 85 compliant, it's about 90 to 95 percent so the
differences are few and involve mostly the intrinsic functions. COBOL II
was released prior to the completion of the COBOL 85 specs since IBM
needed a compiler that would utilitize the 31 bit addressablity of it's XA
architecture machines and they couldn't wait - COBOL II is not ANSI 85,
COBOL/370 and the newer (or renamed version) COBOL for VM/MVS is.

The major changes between COBOL/370 and OS/VS COBOL are:
Sections are no longer required or supported.
exhibit and exhibit named are gone
examine is inspect
scope terminators are included, e.g., end-if, end-read, end-evaluate
set can be used to set conditions to true or false, instead of moving a
'y' to eof-sw you can say
set end-of-file to true and not worry about the actual literal value that
corresponds to "TRUE".
File status can return extended VSAM codes.
Unloaded or empty ESDS files cannot be opened I-O - this generates a
return code of "35" and must be coded for in the open logic.
Tables can have up to 7 indexed levels and be up to 2Gig in length
removing the earlier restriction of 132K bytes total table length.
[warning COBOL II will let you mix and match indexes for different
tables, I haven't tried this on COBOL/370 yet]

The write after positioning clause is no longer supported only write
before/after advancing.
The new COBOL also includes an initialize verb - if used on large record
definitions it is a CPU hog and is a good verb to forget, quickly.

BLL cells are no longer supported, for addressability use the set
to address of feature.

This is just a quick overview of the changes, a good reference book will
cover all the differences.

I don't use procedure division sections, and I don't have access (yet!) to a
COBOL/370 compiler. But according to Harvey Bookman's "COBOL/370" procedure
division sections with priority numbers are still supported.

Prioritized sections no longer generate storage overlays, but they can affect
optimization. If you place paragraphs in different sections with different
priority numbers, the IBM COBOL/370 optimizer will NOT convert performed
paragraphs into in-line code for speed efficiency. I suppose it is possible
someone might actually use this feature for selective optimization of speed
versus load module size.

I assume that eliminating procedure division sections entirely would break
too many older programs.
