Well, if your PC is connected to the same network as the mainframe, and both
platforms have FTP deployed, use FTP.

Microsoft has their SNA Server software, which should pretty much hookup
with any mainframe also running SNA (of course, it has to be set up on both
sides).

There are other methods, of course.  A variety of vendors (Barr comes to
mind right away, but all my Enterprise Systems Journals are off in storage,
so that's all I can think of off hand) sell software and/or hardware either
for the PC or mainframes that allow communications between the two.

Try doing some web searches to find out more exact information.

Two options: SNA or TCP/IP.

SNA requires a gateway server on the PC. Microsoft, IBM and Novell provide
SNA gateway software. Other names to know are IRMA and RHUMBA.

TCP/IP requires TCP/IP running on the mainframe. Once you've got that
installed then there are a bunch of ways to make it happen. If 'twere me I
would strongly consider IBM's MQ series to manage the PC/mainframe
communications. It'll be responsible for guaranteeing that all messages get
where they're going.

My corporate intranet is run on a Domino Web Server on OS/390.  The
front end could easily be built using either CGI programming or by
Javascript in your browser and SSL (encrypt).  The HTTP protocol that
Browsers communicate with the Web server is dead simple.  You can
simply put in a call to a CGI program passing and receiving the
appropriate data.

For more tied applications you might want to investigate LU6.2, about
all I know is that it was designed for smart terminal traffic and can
hook up with an application on both ends.  There are various tools
that support this, I cant name any though I was never close enough to
the action.

For simple data transfer, as noted elsewhere, simple FTP is the best
method.  The mainframe can push data to your PC system which can
monitor it's arrival and kick off a process and vice versa.  I do this
quite a lot and it works well.

Ok,
1. Forget SNA and APPC/LU6.2.
   You would be tying yourself into a protocol that may not have a future.
2. Use TCP/IP for your transport.
   IBM mainframes have it built in (OS/390) along with a UNIX layer.
3. Use DB2 on the mainframe to hold your data.
4. Use CICS on the mainframe to run your transactions.
5. Use MQ Series to communicate over TCP/IP from your PC to CICS.
   You can go direct to CICS via either Sockets or CICS Web Server.

You did not say how many transactions per second you wish to run.
If the number is high, then you have to break the transaction up into
validation and execution phases.

You have to validate your incoming transaction. If it is allowed to
proceed, you send your response back to the PC allowing the transaction,
you queue the validated transaction to one of many work queues, logging the
transaction details in a journal. The actual work of updating accounts happens
some time later when the queued transaction is processed.

This allows for a fast turn around of the users transaction, and the steady
update of users accounts as the update processes work their way through the
validated transaction queues.
