Yes, Bill I am very familiar with CICS pseudo-conversational programming. I
also understand the serially reusable implementation of the IMS/DC
conversational model.

However, event driven  programming on a PC is a different kettle of fish to
both of these.

There is no terminal dependent buffering requirement in the PC environment.
The Windows Message loop provides the event stream as a chain to any
application that wants to look at it (it is a bit like hooking interrupt
vectors in the old days; when you had finished with the interrupt, you
placed it back in the chain so downstream processes could hook it too if
they wanted to...)

I  would be suspicious of any system that contrived to make the Windows
processing look like it was under the control of the application programmer.

I don't wish to discuss SP2 specifically as I have no experience of it.

In principle, I would rather teach programmers the concepts of event driven
programming, than strive to make it "easy" for them by simulating an
environment they are familiar with. (Besides, the CICS paradigm isn't such a
great one that we should perpetuate it into Client Server... Ironically,
that is exactly what I have just done with some Web programming, but I had
no choice <G>).

That is just my opinion; I accept that many will disagree with it and I have
no problem with that <G>.

And I reiterate again that my comments are not intended to be derogatory on
SP2; I simply have no opinion about that.

Pete.

          WHEN SP2-KEY-Close
