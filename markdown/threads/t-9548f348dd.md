```
Forgive me if this topic is old hat to all of you, but I have not found a
clear, lucid explanation of the link options of AMODE and RMODE and their
relationship to the DATA(31) or DATA(24) compile options.

Specifically, if a program is compiled DATA(24) and linked as AMODE=31 does
that mean a DATA(31) program cannot safely call it dynamically?  It would
seem that if the DATA(24) called program is linked AMODE=31 then it could
access the storage area passed from the DATA(31) calling program.  If that
is not the case (as I have been warned for years), then why would a DATA(24)
be ever linked as AMODE=31?

Can anyone recommend a clear, lucid reference that addresses this?

Thanks for consideration.
```

```
to some dialect of COBOL on an IBM MVS or OS/390, you could consult the
following URL:

http://www.s390.ibm.com:80/bookmgr-cgi/bookmgr.cmd/BOOKS/IGYPG204/2%2e3%2e3%2e10
?SHELF=IGYSH206

It refers to the chapter '2.3.3.10 DATA' of the manual 'COBOL for OS/390 & VM
Programming Guide'. The DATA option is the same for elder versions of COBOL
like VS COBOL II.

The important thing about DATA is the fact that it only affects the place
where the WORKING STORAGE of the program is stored. If you compile DATA(24),
the WORKING STORAGE is always stored below the 16 MB, even when you link it
with RMODE(ANY). In this case, the program itself resides above, its storage
resides below. Now this program can call a AMODE(24) program dynamically
because the latter can access the parameters in its linkage section without
problems.

Assume you have two programs: A is the main, it calls B dynamically (static
call are of no interest).

A has RMODE(24): works with every AMODE/RMODE of B and every DATA of A.

A has DATA(24): works with every AMODE/RMODE of B and every AMODE/RMODE of A.

A has DATA(31) and RMODE(ANY): blows if B has AMODE(24).
```

```
I have read Daniel's reply to this note - and it is quite accurate ...
HOWEVER, I think that this a complex area (that I was hoping someone else
might respond to).  Having seen no one else, let me try to explain some of
the issues that I see.  (Non-IBM mainframe people can happily delete this -
unless you have an obscure interest.)

   ***

The first think that you need to answer is what COBOL product you are using.
There are (slightly) different answers for VS COBOL II as opposed to COBOL
for this-and-that.

If (and only if) you are using VS COBOL II, then you need to know that the
RES and RENT compiler options (don't confuse this RENT with the
linkage-editor RENT option) impact AMODE/RMODE.

The default is
   RES yields AMODE(31) - NORES yield AMODE(24)
   RENT yield RMODE(31/ANY) - NORENT yields RMODE(24)
   To use RENT, you must also use RES
   The DATA compiler option is IGNORED unless you are RES *and* RENT
   The DATA compiler option ONLY impacts where working-storage is located
   If you "statically" call any program with a "lower-level" (i.e. AMODE(24)
or RMODE(24)) then the "lower" value takes precedence.
  It is always "safe" to over-ride the default linkage editor options to a
LOWER value - but is never safe to "raise" them.  (I.e. if it defaults to
RMODE(24) - don't try and over-ride it to RMODE(ANY))
  The "normal" reason that you would want to use the DATA compiler option is
if:
    A) You have a RES,RENT COBOL program that needs to DYNAMICALLY call an
AMODE(24) and/or RMODE(24) program (in this case DATA(24) makes your
working-storage available to the subprogram)
    B) If you have a RES,RENT COBOL program that is dynamically CALLed by an
AMODE(24) and/or RMODE(24) program, then you do NOT need to use the DATA(24)
compiler option.  HOWEVER, you must make certain that the address of the
parameters that were passed to this program have their high-order byte
"cleared" to correctly read the parameters in the called program's linkage
section.

   ***

If you are using COBOL for this-and-that (aka IBM COBOL for OS/390 & VM - or
IBM COBOL for MVS & VM - either with or without MLE), then all of your
programs
   A) do not (cannot) have a RES compiler option
   B) are automatically AMODE(31)

Otherwise, most of the defaults and reasoning listed above still apply ...
HOWEVER, there is a new compiler option Called "RMODE" (not to be confused
with the RMODE linkage editor option - which it impacts)

If you use RMODE(AUTO) - then everything works pretty much exactly as I
listed above for VS COBOL II.

If you use the RMODE(ANY) compiler option, then EVEN IF YOU HAVE USED THE
NORENT compiler option, your program will default to the linkage-editor
option RMODE(ANY) -

Similarly, if you use the RMODE(24) compiler option, your object code will
default to the RMODE(24) linkage editor option - even if it is RENT.

  ****

You may also (besides the URL that Daniel posted) want to look at the
Migration Guide discussion of this - which can be viewed at:
  http://www.s390.ibm.com/bookmgr-cgi/bookmgr.cmd/BOOKS/IGYMG201/5%2e1%2e4

I hope that this helps - and is relatively clear (given how complex the
issues are).
```
