My recent message about an "inline perform" without an UNTIL clause got me
thinking.  Wouldn't it be nice to have a READ that *has* an UNTIL?

Current read loop:
PERFORM UNTIL END-PURGE            
    READ TXFRHST-FILE NEXT RECORD  
    AT END                         
        SET END-PURGE TO TRUE      
    NOT AT END                     
        PERFORM 0510-DELETE-IF-OLD 
    END-READ                       
END-PERFORM                        

How about this instead?

READ TXFRHST-FILE NEXT RECORD 
     UNTIL AT END
NOT AT END                    
    PERFORM 0510-DELETE-IF-OLD
END-READ                      

Notice the addition of "UNTIL AT END".  So it would read each record,
performing the code in the "NOT AT END" clause, and then quit once we've
read and processed the last record.

No new keywords or, really, even new concepts.  Just makes the code more
compact and perhaps more obvious, once you get used to it.  No checking for
"at end" essentially twice.  And yes, I do realize you can have END-PURGE
defined as the 88 level name for the FILE STATUS for the file and do this
instead.

PERFORM UNTIL END-PURGE            
    READ TXFRHST-FILE NEXT RECORD  
    NOT AT END                     
        PERFORM 0510-DELETE-IF-OLD 
    END-READ                       
END-PERFORM                        

That doesn't even work for a RETURN from SORT statement, though, because the
FILE STATUS clause is not allowed.

RETURN TXFRSRT-FILE INTO TXFRHST-REC    
       UNTIL AT END                              
NOT AT END                              
    WRITE TXFRHST-REC                   
    INVALID KEY                         
        PERFORM 20XX-TXFRHST-INVALID-KEY
    END-WRITE                           
END-RETURN                              

Some inspiration(?) came from the Ruby programming language, which has
something like this:

txfrHstFile.each do |txfrHstRec|
    deleteIfOld(txfrHstRec)
end

Anyway, just felt like starting a new argument.  I have no expectation that
this would ever become part of any COBOL standard.  Just wasting time on a
Friday evening...

Frank

No, it wouldn't.

You are expecting it to behave how you WANT it to behave, and not do what it 
actually says.

If there was such a construct, and it did what it says, it would read until 
end of file. (That means it would simply get every record, and your NOT AT 
END would never be raised...)

Interestingly, there is such a construct in C# which reads ALL records in a 
file into a result set that is a single string. Sometimes this can be very 
useful.

(For a line sequential text file, for instance, you can load the entire file 
to memory with a single "read" and then search it much more quickly than if 
you read and individually searched each record. I put "read" in quotes 
because it is done with a stream reader and it isn't a READ as understood in 
COBOL.)

The READ construct currently implemented in COBOL pauses after each record 
(thereby enabling AT END and NOT AT END to be evaluated) because it has only 
been told to get a single record.

If the construct you are proposing were to be impleneted, it would be 
impossible to tell syntactically which behaviour was required, the one you 
describe, or the one I describe.

Perhaps that's why it hasn't been implemented?
