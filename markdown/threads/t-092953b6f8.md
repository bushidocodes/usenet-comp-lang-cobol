Tom,

I don't use NE but I do use the ACCESS ODBC Driver frequently. The ACCESS
error code 3010 (as returned by VB) has nothing to do with cursors (it
indicates that you are trying to create a table which already exists.

I use SQLSTATE rather than SQLCODE and (under FujitsuCOBOL) the ACCESS ODBC
driver returns an SQLSTATE of S1010 when a cursor sequence error occurs.

(Probably addressed the wrong cursor name in an OPEN-FETCH-CLOSE sequence.
Can be caused by ï¿½cloningï¿½ cursor code.)

As this is the closest code I can find to the one you describe, and as it is
pertinent to cursor error, hopefully it might be some help.

The ODBC driver is much more sensitive to syntax than say, Mainframe DB2, so
you need to check this carefully also. If you can't solve it, send me the
code and the Access DB.

Gladly.

It is quite OK to use either (both).

I use SQLSTATE because it offers more situations and is more consistent
across platforms in my experience, especially where ODBC is being used. It
is just as easy to check for '00000' = OK, or '02000' = NOT FOUND  as it is
to check for 0 or 100. SQLCODEs seem to be vendor dependant and, in most
cases they try and make them the same as IBM implemented for DB2. (-100 =
NOT FOUND, for example.)

I have no strong feelings either way but when things go wrong and you want
more information, I reckon SQLSTATE is the best bet. I guess it's analogous
to using "File Status" instead of "AT END"/"NOT AT END" when dealing with
flat files. They all work fine and they are all useful. It just comes down
to preference and experience.

Michael, there is something very important that you are missing here...

This is a COBOL forum.

The calls you are making are not available through standard COBOL as part of
Embedded SQL.

They probably work fine in Basic or C++ but they are not provided for in
Embedded SQL as used in COBOL.

All of the diagnostic fields SQLSTATE, SQLCODE and SQLMSG are provided as
"known" Host Variables in COBOL. There is no need to even use the Linkage
blocks provided for DB2 (SQLCA and SQLDA); these variables are "known" to
COBOL.

Standard Embedded SQL (ESQL) is transportable across platforms; the
low-level calls you are describing are not.

While I take no issue with people using whatever works, it is simpler for
most programmers (especially people making the transition to RDB for the
first time), to simply learn SQL rather than all the ODBC API calls. (COBOL
compiler pre-processors will implement the API for you transparently,
exactly as they do with the Windows API when you use screens...)

With your level of experience, this is not a problem; for many beginners it
is. And there is no need for ANYONE to go to this level if they use COBOL.
Using simple standard ESQL in COBOL, with the RDB treated purely as a data
repository (no DB Procedures, Macros, or other platform dependant code AND
no direct ODBC API calls....<G>) it is possible to write code that can
access RDBs on PCs, mid-range, and Mainframe and the code can be re-compiled
to run on any of these platforms. (Of course it will always run from a PC to
access data on any of these platforms, but sometimes it is nice to use the
PC as a test bed for systems targeted  for the Mainframe, for example.)

I strongly advocate the simplest possible approach to RDB. That means no
low-level calls, just simple, transportable, SQL.

more below...

None of this is standard ESQL. The constants are not part of the COBOL SQL
environment  (and COBOL does NOT support underline characters) and the Calls
are low-level API calls that really have no place in a COBOL program. It is
quite possible to set up the constants but you would need to declare 88
levels on the SQLSTATE Host Variable. Here is ONE POSSIBLE equivalent in
COBOL...(NOT pseudocode, Basic, VB, or C++....COBOL...remember, the "dying
language" ?<G>)

(You know I almost NEVER post code to public forums but this one warrants
it, so here goes...<G>)

Data Division.
Working-Storage Section.
...
        EXEC SQL BEGIN DECLARE SECTION END-EXEC
*> Declare Host Variables....
 01   SQLSTATE pic x(5).
         88 SQL-success    value '00000'.
         88 SQL-not-found  value '02000'.
 01   SQLMSG     pic x(128).
        EXEC SQL END DECLARE SECTION END-EXEC
...
 Procedure Division.
...
         EXEC SQL
               SELECT something
                   FROM  sometable
                WHERE  somecondition
         END-EXEC
         *>  Check what happened....
         if NOT SQL-success
            if NOT SQL-not-found
               *> we could simply out the SQLSTATE and the corresponding
error message
               *> which have been placed into the Host Variables for us by
the SQL call,
               *> but let's pretend we want to do it thoroughly....<G>...

               call 'badsql' using SQLSTATE    *> analyse and report the
nature of the error
               set abort to TRUE
            else
                  call 'setmsgbx' using
                                            by value 'Sorry, no data was
found...'
                                            *> notify user we found
nothing...
                  end-call
             end-if
          else
                *> process the retrieved data...
          end-if

Obviously the above can be tailored to cater for other expected results and
I have made no attempt to show all possibilities. The point is that COBOL
AUTOMATICALLY returns SQLSTATE and the appropriate error message, just as
Michael described,  (and SQLCODE if you define it), into the Host Variables
allocated for them. There is NO NEED to make a low-level API Call. You can
do it in standard (transportable) COBOL!

*    Build the SQL statement :
      STRING  "INSERT INTO " delimited by size

                          [value-2 delimited by size, value-n delimited by

EXEC SQL
    INSERT INTO :table-name VALUES (:value-1, :value-2, ...:value-n)
END-EXEC

of SQL (or even that there can be variations between the same elements in
different drivers), but for the most part the connection to ODBC can be set
for any of the major PC COBOL compilers.

It is only on the mainframe that "Language" support is necessary. On the PC,
given an appropriate ODBC driver, it is possible to interface to any RDB
through the COBOL ESQL.

I have accessed ORACLE, and Mainframe DB2 from a PC in this way, despite
there being no SPECIFIC support in Fujitsu COBOL for these RDBs. (The DB2
ODBC driver was provided by IBM; I believe there is also an UDB driver
available...)

ODBC is exactly that; OPEN Database Connectivity. It works and it works with
ESQL.
