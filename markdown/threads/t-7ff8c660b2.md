```
cobol version :
rm,/cobol-85 version 6.0 on a NCR unix 4.3

Problem:
We are working with cobolprograms from a software house. I make very simple
programs to read the cobol files (generated by cobolprograms of the
softwarehouse) to present the data in an onther way. But I have a problem in
the new millennium. The software house changed the date field and I can not
read the date field for dates in the new millennium. What's happen:
This is a part of the "old" file info. The file name is DATES

004500 BLOCK CONTAINS 4 RECORDS
004600 RECORD CONTAINS 128 CHARACTERS
004700 LABEL RECORDS ARE STANDARD.
004800
004900 01 OPENSTAANDEKLACHT.
005000 03 OPSTKL-KEY.
005100 05 OP-KEY PIC S9(7) COMP-3.
005200
005300 03 NEW-DATE PIC 9(6) COMP-6.
l

When you dump the file DATES, you will see (for example) a date of 990101 in
the field New-Date. That represents a date of 1 january 1999. In this
situation my program works fine.
But when I use the program of the softwarehouse to fill a date in the new
millennium say may 24 2000, the dump of the file is :
9A0524

When I use my program, the field New-Date will be respresented as 99 05 24.
The makers of the program use for dates in the new era a hexidecimal 9A. So
9A is 2000, 9B is 2001. My program cannot read the hexidecimal value. Al
hexidecimal values will be represented as a 9. So when I read 9A, my program
think that it is 99.

Does somebody know how I should change the field New-Date so I can read a
hexidecimal value (with Comp-6)

Thanks,

Steve Bakker
```

Steve,

        When you add X'1F660000' to your example value X'9A0524', the result
is X'20000524'. If you always add this value to your date-value, you'll wind
up with a date-format of X'CCYYMMDD' as a 4-byte Hex-Value. Seems to work
fine when the CCYY is X'9B', X'9C', etc. I'm not NCR literate, but I'm sure
there's a way to do this.

HTH....

Cheers,

WOB
