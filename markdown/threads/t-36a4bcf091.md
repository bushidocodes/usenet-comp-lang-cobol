OK I've had a look at your SQL and agree it is OK... (sorry about delayed
response, I've been very busy just lately).

I'm sure you realise that searching text is always inefficient, and Thane
has already correctly pointed out that LIKE is slow...

As a suggestion, you might consider holding certain frequently used  words
(like "drill" or "bit" as codes or even as text in separate columns on the
database. These could then be separately indexed. (difficult to maintain and
clumsy, but possible...)

Another possibility would be to set up a hierarchy of tools like "Drills",
"Bits", "Spanners" etc. and run the cursor on this. As each record is
retrieved, search the description for your keywords in memory. (Normally, it
is MUCH better to let SQL filter and search the data, but sometimes it is
better to do it yourself...could be in this case.)

Finally, you could build a Keyword in Context (KWIC) index for each record.
Then you can avoid using LIKE altogether.

Given what you have...

Set the Design View Properties of LongDesc_Description to: "indexed=yes"
"Unicode compress=no"


(Although you can't index Memo fields, you CAN index text). Let ACCESS build
the necessary index for it and check that it is recognised as an index in
ACCESS 2000.

ACCESS Query is running faster than your SQL because it is probably building
an index the first time you run it, and saving it for the duration of the
session.

If you fail to see any improvement, then you must tune the ODBC driver.
Consult your Driver documentation for Help on index and data buffer sizes.
Sometimes simply changing from the default 512 bytes to 4096 can improve
performance out of sight (especially where sequential access via a cursor is
being done...). You could also look at the advanced options in ODBC32 to do
this, but make sure you note any existing settings before changing them, so
you can reset them if you have to, and ALWAYS read the Help before messing
with this...

Pete.

Sorry, that may have been misleading. It is NOT a standard option in ACCESS
or anything else that I know of. It is a technique for organising text data
into a form that allows it to be searched in a meaningful way.

The technique was invented by IBM and goes back to the days before disks
were invented and we held everything on tape. You can understand that
searching text under these circumstances was very difficult.

What KWIC did was to enable you to input a "keyword" (like "drill") and it
would then return every record that had this keyword, and show you the
context in which it appeared (usually 10 words from the text on either side
of the keyword, but this was user definable), so you would get "...and there
is no need to sharpen this DRILL because it is self..." or " ....building
morale. DRILL is essential for new recruits..."

From the contexts returned you could tell which source you were interested
in.

I have some routines which I wrote for windows to do this, and you simply
click on the context you are interested in and it brings up the appropriate
source document.

The point is that in your case, the technique could be adapted to give you a
better search on text.
(The wildcards are eliminated because the search has to be on Keyword. Once
you have built the keyword list you could then do a secondary generic
search.)

As you are not familiar with this technique, is not likely to be viable and
I shouldn't have muddied the water describing it...sorry.

I believe that if you index the text field (even though I hate doing this)
you will improve performance, mainly because the index will be memory
resident (paged).

The real question you need to ask yourself is: "Can I structure this data so
it can be searched more efficiently?" Hence my suggestion that you use some
generic hierarchic classifications to quickly identify subsets of the data,
for further detailed processing.

Let us know how you get on.

Pete.
