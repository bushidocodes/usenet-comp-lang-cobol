.......here is the excerpt from the article regarding COBOL:

"Programming languages on the rise: Cobol

It may not be fair to call Cobol a niche language as it was once the dominant language in the enterprise. Grace Murray Hopper, famous for finding the first bug in the early mainframes, helped create the language in 1959 and it's been enhanced hundreds of times since. Cobol jockeys today get to play with object-oriented extensions, self-modifying code, and practically every other gimmick.

That never earned it much respect in some circles. Or as famous academic Edsger Dijkstra put it: "The use of Cobol cripples the mind; its teaching should, therefore, be regarded as a criminal offense." The folks in mainframe shops everywhere ignored this note and soldiered on. IBM calls one of the latest releases "Enterprise Cobol 4.2," but it could as easily be numbered 147.2 or maybe even 588.3. Cobol programmers like the syntax that's more like a natural language with actual nouns and verbs that form clauses and sentences -- a technique that might call Ruby to mind.

While fewer schools are teaching new programmers Cobol, the language is far from dying, with many corporations continuing to invest in their Cobol stacks. A recent search of Dice.com showed 580 jobs mentioning Cobol and 1,070 mentioning Ruby. The bulk of the jobs seemed to involve counting money ("asset management") and counting doctor's visits ("Health IT"). While these are some of the same areas that first adopted computers for back-office processing, the work still needs to be done.

Versions of the languages run on JVMs and .Net virtual machines making it possible to migrate code stacks away from mainframes to Linux boxes. Programmers who want to use a more modern IDE can search for plug-ins to Eclipse, a project that is gaining new support."

Eclipse? New? COBOL has been running on it for at least 7 years.

The fairest conclusion that can be drawn is that COBOL is rising in 
interest, but it still has a long way to go before it is worth investing in.

The "good stuff" in COBOL requires Object support, just as it does for any 
other modern programming language.

That isn't a problem and IBM (mid-range; I don't know about their current 
mainframe compiler...), Micro Focus, and Fujitsu all offer Object support in 
their COBOL compilers. Both Fujitsu and Micro Focus made a lot of noise 
about producing an IL generating compiler for .Net which is fully Object 
Oriented. (And these compilers are very good, largely because they 
effectively harness the power of the .Net framework through their respective 
IDEs (Eclipse and Visual Studio).  Unfortunately, they are also expensive 
and the Micro Focus one requires run time fees...)

There are two problems with this solution, though.

1. You don't need to generate IL code to run OO COBOL on .Net. (If you have 
an existing Fujitsu NetCOBOL or MicroFocus Net Express compiler, that's all 
you need. The .Net Framework provides a special service to run this code 
alongside .Net code. It actually runs faster than .Net IL/CLR code. 
Interaction between .Net objects and Native objects is seamless and works 
very well. I've been doing it for a number of years now and find no trouble 
at all with it. C# Classes talk to COBOL and COBOL talks to C#; neither of 
them cares what language the object they are talking to is written in. This 
is what I mean when I say that source code is much less relevant in 
non-COBOL environments.)

2. If you REALLY want IL code there are FREE compilers for .Net languages 
that can do that and they come with a free copy of Eclipse or Visual Studio.

I see the following adoption pattern (this is just one of many possible 
scenarios... I'm not saying it is definitive, but most end-points do not 
include COBOL):

1. Existing COBOL shop decides .Net is the place to be. Contacts their 
current vendor who sells them an IL generating compiler and a few ancilliary 
tools. Kerch-ching!

2. COBOL guys haven't had a bone for a very long time so they are excited to 
be able to use .Net objects and Classes. The new COBOL is pretty sexy and 
works very well.

3. A year or so later the programmers have got a pretty good idea of how 
objects work and they are starting to realise that thousands of lines of 
code that need maintenance is not a good thing for the company or for them. 
Some of them start looking at C# or VB.Net...(And COBOL still does not look 
good on a CV...)

4. Five years on the COBOL has fallen by the wayside and been largely 
replaced with the free compilers and IDEs.

The point is that COBOL has no long term future and it isn't just because of 
fashion. (However, you can continue using it just as you could keep digging 
with a stick after shovels were invented.)

It is interesting to me that the recent enquiries to PRIMA are NOT coming 
from COBOL people. Certainly last year it was mainly COBOL people who knew 
COBOL, who were talking to us. That seems to have changed. Now it is people 
who are either advising companies how to reduce dependence on COBOL or it is 
IT guys who know all about objects and layers (but not about COBOL) and 
heard we have a solution for COBOL legacy that implements that and allows 
Legacy AND New technology to co-exist on the same platform, accessing the 
same database, without needing middleware solutions that lock you in.

       program-id. bits.

       1 chr global pic x.

        2 occurs 8.
         3 bit-value pic 9.
       1 saved-values.
        2 occurs 5.
         3 saved-value pic x(8).
       procedure division.
       begin.
           display quote "chr2bit" quote
           perform test-chr2bit
           display space
           display quote "bit2chr" quote
           perform test-bit2chr
           stop run
           .

       test-chr2bit.
           move "F" to chr
           call "chr2bit"
           move bit-array to saved-value (1)
           display chr space bit-array
           move "r" to chr
           call "chr2bit"
           move bit-array to saved-value (2)
           display chr space bit-array
           move "a" to chr
           call "chr2bit"

           call "chr2bit"
           move bit-array to saved-value (4)
           display chr space bit-array
           move "k" to chr
           call "chr2bit"
           move bit-array to saved-value (5)
           display chr space bit-array
           .

       test-bit2chr.
           move saved-value (1) to bit-array
           call "bit2chr"
           display bit-array space chr
           move saved-value (2) to bit-array
           call "bit2chr"
           display bit-array space chr
           move saved-value (3) to bit-array
           call "bit2chr"
           display bit-array space chr
           move saved-value (4) to bit-array

       identification division.
       program-id. chr2bit.
       data division.
       working-storage section.
       77 int binary pic 9(4).
       procedure division.
       begin.
           compute int = function ord (chr) - 1
           compute bit-value (1) = function mod
               (function integer (int / 128) 2)
           compute bit-value (2) = function mod
               (function integer (int / 64) 2)
           compute bit-value (3) = function mod
               (function integer (int / 32) 2)
           compute bit-value (4) = function mod

               + bit-value (7) * 2
               + bit-value (8)

           .
       end program bit2chr.
