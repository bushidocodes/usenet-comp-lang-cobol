A little over a year ago (April '01) , there was a discussion here and with
several individuals re the [in]ability to make calls to the Windows API when
using several specific brand-name COBOL compilers....and/or limitations of
"which" calls or "which type" of calls the various brand/model compilers
supported at that time.

Here are a couple of notes I saved...

> >
> > They [COBOL compiler Vendor name lost] have NO support for any of the
Registry API calls because they
> do not provide Import Libraries for the advapi32.dll. In fact, the ONLY
API .dlls  which are supported are the Kernel and User. ... Which I found
out after (too) much experimentation.

> You said the magic word.  Procedure pointers are not yet supported in
> Fujitsu COBOL - and that was the point...

Is there still a need for something which would make calling the Windows API
possible from language products which do not "natively" support this? Or
have all the vendors now enhanced their products to the point where this is
no longer a real problem?

What I had suggested at the time was creating a Windows DLL which one
*could* call from COBOL with something aking to the syntax....

CALL 'CallWin32Api'
     using  functionname,
              libraryname,
             [parameters]
    [returning result]

.. and this DLL would handle all the crap work.

Either that or  it was something like setting it up to use the RUNDLL.EXE
program. The main thing was you could use all 'regular' COBOL data items as
parameters, and would not have to screw around with null-terminated strings,
callback addresses, etc, etc. etc..

If there is still any interest in this, post here or drop me a note; if
enough interest, I'll take a fresh look at it.

What I've taken to doing is wrapping specific API calls in an ActiveX
control.  I have a couple of reasons for doing this - when running in
a Thin Client environment using Flexus tools, I need to use API calls
to do remote file I/O and some other functions.  These need to run on
the client PC - so I use an ActiveX control I wrote to make this
happen.  Additionally - most vendors document their OLE interface ---
however, few document the API in general.  I have found that the real
problem for COBOL programmers is understanding the API and how to
interface with it -- the docs for the API are not COBOL friendly.
