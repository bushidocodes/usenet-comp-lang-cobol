You don't tell us what compiler, etc. but here is a wild guess..

Since you have ACCESS MODE IS SEQUENTIAL, the IO subsystem should reject,
with an appropriate error status ("21"), any record you attempt to write the
key value of which is not greater than the previously written key value.  An
example:

PART-NO-OUT       Successful?
000023                    Yes
000021                    No
000025                    Yes
000029                    Yes
000028                    No

In this example, even though you "wrote" 5 records, only 3 would exist in
the file.

This might help.  Your program does not anticipate any IO error; it is not
quite ready for production :)

Regards,

Try to find FILE STATUS in your reference manual or book. All I/O statements
should be followed by a thorough examination of file status (whether it is I/O
through standard COBOL facilities or through advanced database or telecom
facilities ... if you do I/O then do status checking right then and there).


Also notice,( and this is not yet your current problem), that you will
inevitably do atleast one extra and erroneous WRITE, because of poor
positioning of the WRITE statement.


You posted with the polite and very exceptable note "...  (don't mind the lack
of structure) ...". You will benefit by structuring your code at an early stage
and always structuring your code.

The structure of nearly every COBOL program is management of I/O exceptions. 
The business or scientific information you process is secondary, tertiary or
even quarternary.

So, your structure is something like ..
IF the user says quit then quit
ELSE _!!!!!_PERFORM_!!!!_ do-something.  Use PERFORM and paragraphs. Structure
from the start, that is the fundamental activity of programming - organize. 
Movement of data and calculation, however impressive, is minor activity.

The details really are minor, organize, organize early, keep it organized.

In do-something you might have,
  take in the user data
  if that went well and the data is cool
  _!!!!!_PERFORM_!!!!!_ write-to-file


In write-to-file you should have structure, not later, _NOW_
   WRITE blah blah blah
      ON ERROR
          la ti dah and tell someone about it
or
   WRITE
   EVALUATE  Specific-FILE-SATUS
      WHEN okay
          bump count, tell the world
      WHEN particular anticipatable error
          log problem, perhaps attempt alternative
    * note sometimes errors are exceptable
      WHEN OTHER
          log log log, and then bomb
   END-EVALUATE.

The idea here is not to write your code, the idea is to show you that you do
need to structure, you need lots of structure. Structuring the program is the
essence of the work. The records, fields and contents thereof are very minor
concerns.

It is you that should mind the lack of structure.  You have to start with
structure, it is not an after thought.  Most of your time programming will be
spent trying to perceive and properly represent the structure of the process.

You can benefit by just listing the things a program does or might do.  Code
them as paragraph labels. And then try to figure which paragraph performs the
other paragraph and under what conditions.

Start at the list of things to do level. Try to conceive of how these actions
are related. Don't start inside the IFs. Under no cicumstance should you be
hesitant to create a new paragraph and _!!!!!_PERFORM_!!!!!_ it. Don't be
afraid to have lots of paragraphs.

One other small note, ALL I/O needs status checking, even the OPEN statement. 
This is an extremely valuable lesson to learn. In large applications even just
connecting to SQL services constitutes an I/O action that can fail, and
therefore needs to be checked.

The relevance to your current code, is that an unconditional CLOSE is
potentially harmful. You actually should only close a file that was in fact
successfully opened. Most COBOLs fail softly on this problem, but not all.  

The dificulty you get into is that if the OPEN failure is not detected and not
diagnosed with DISPLAY messages or some other kind of diagnostic then, your
program could proceed to the CLOSE. At which point you could have a very
confusing failure. You could inadvertantly send yourself off on a long and
useless analysis of the CLOSE failure. In a professional environment that could
be time spent in a production down situation when there is much at stake.

The place I have seen this mistake cost the most money is actually the READs of
a file, but it can happen with the CLOSE. Programmers neglect to check the
status of the connect to database services or the status of the open, and then
blithely sail into the READ routines which blow sky-high.  They spend a lot of
time examining the irrelevant READ paragraph and the paragraphs which control
it, searching and arduously finding the keys and control variables, and having
lots of talk about this and that might have happened.  Eventually, after
eliminating all other possibilities they percolate back up the program
structure and discover possibility that the program did not even start
correctly, it can cost alot money to eliminate all other possibilities by
process of elimination.

It is much cheaper, and a lot more fun, to check file status (or database
status) upon open (or upon connect). Screach to a halt if the file does not
open. That is do no WRITEs, no READS, and usually you should not do a CLOSE.

You have a good start with your posted code, let us know how you progress.
Remember to check the file status after every I/O operation.

Best Wishes,
