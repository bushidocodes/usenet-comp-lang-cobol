First, the amendments:  For purposes of this discussion, they're a red
herring; Unisys treats the amendments as part of the '85 standard, and
includes support for both the '89 Intrinsic Function Amendment and the '93
Corrections Amendment.

As to COBOL74 vs. COBOL85 usage:  I've got some informal familiarity with
this issue.  I believe there are far more COBOL74 programs in MCP/AS sites
than COBOL85 programs, and far more COBOL74 customers than COBOL85 ones, as
of this writing.  Last I heard the difference was quite marked.

But that's not the whole story.

There are currently NO plans to add new features to the COBOL74 compiler.

Most COBOL85 users today find generated COBOL85 object code at least as
efficient as, and quite frequently more efficient than, COBOL74 object code.

The architecture of COBOL74's library interface was limited at the design
stage by the decision to make it an extension of the ANSI-mandated IPC
mechanism; since all IPC parameters were "by reference", so are all COBOL74
library parameters.  This has been recognized for decades as a serious
limitation of COBOL74 in the MCP/AS environment.

COBOL85 added an "explicit library declaration interface" which allows the
explicit declaration of a library template, which in turn allows just about
any ALGOL or NEWP library to be called directly from COBOL85 programs.
While in the past, Unisys has carefully ensured that interfaces of general
interest have been compatible with the COBOL74 restrictions, I believe that
there is increasing resistance to continuing to place this requirement on
future mechanisms given the availability of the more-flexible interface in
COBOL85.

There are also a number of limitations imposed by the architecture of the
COBOL74 compiler (for example, on the sizes of 01-level records in
WORKING-STORAGE) that simply do not exist in COBOL85.

For current users of COBOL74 (and even COBOL(68) ), Unisys continues to
provide, to support, and to enhance the COBOL Migration Tool (AKA CMT).
This software product, as far as I know, provided at no additional cost with
the COBOL85 compiler, to encourage users of our older COBOL compilers to
migrate to what is now the "flagship" dialect of that language.

Don't get me wrong here.  I am aware of no plans to discontinue support for
the COBOL74 compiler; in fact, I am aware that there is some reluctance on
the part of Unisys even to initiate such a process, given the painful
experience of decommissioning the COBOL(68) compiler (the process officially
began somewhere around 1988, I believe, and was completed at the end of
1999).

Note that I have a "vested interest" in Unisys continuing to support the
MCP/AS COBOL74 compiler, since I've been working with it for not far short
of two decades now.    If there's anybody on this planet that would be most
likely to encourage a new user to stick with the old tried-and-true MCP/AS
COBOL74 compiler, it'd be me.  And I expect to be supporting it for some
years to come.  COBOL74 is a *stable* product.  It works, and it works well.
But one of the reasons it's stable is that, by and large, the only
enhancements that are being made to it are those that are necessary to allow
the running of existing programs on the latest hardware implementations and
software versions, and to provide upward compatibility to COBOL85.

Over the last several years, COBOL85 has matured, and even blossomed, while
COBOL74 has changed little.  Today I must recommend quite strongly that any
new implementations be done using COBOL85.  Ultimately, the limitations of
COBOL74 will prove crippling to the user, and COBOL85 continues to benefit
from enhancements (in capacity as well as feature content) that I believe
will not even be considered for COBOL74.

MCP/AS COBOL74 is indeed a safe choice.  It compiles programs really fast,
and it's super stable.  But there's a lot of really useful stuff
(standards-mandated stuff as well as Unisys extension stuff) you can do in,
and with, MCP/AS COBOL85 that just aren't possible with MCP/AS COBOL74, and
I think any new user, or existing user considering a new COBOL application
on MCP/AS architecture, would be well-advised to use the newer dialect for
new implementations.  Users considering which MCP/AS COBOL compiler to use
for new applications or significant enhancements to old ones should note
that the COBOL compiler that Unisys is treating as its flagship COBOL
product (from a technical as well as a marketing standpoint) these days is
COBOL85, not COBOL74.

    -Chuck Stevens
