I got some really positive response to the promise of a challenging
problem. Hopefully there is someone out there for whom this is not
challenging. I'll give you a synopsis of the big picture first, then
the small piece I need some help with.

It may be that I can't explain it here well enough for you to
understand. If that is the case, I have some diagrams I will scan and
place on the WWW for us to look at to make things clearer.

What I am doing is calculating something called inbreeding coefficient
for a pedigree. This is not a homework assignment - this is for a
product I am developing. I have done my research on this and have
actually talked to several genetics professors. Several competive
software products calculate this - some correctly some incorrectly,
and none are in COBOL. The calculation comes by analyzing an
individuals parents for common ancestors. From this the "steps" from
one parent to another are calculated, counting up the ancestral tree
from one parent to the common ancestor, then down the tree on the
other. A formula is applied using this number of steps. For each
common ancestor, that ancestor's inbreeding cooeficient is also
computed. Here is the problem I wish to discuss and determine how
best, and fastest to solve it using COBOL.

I have indexed files keyed by individual. Each record contains the
parents, if known for each individual.

I need a method to count each unique path between an individuals first
generation ancestor to the other, via any common ancestor - if one
exists.

Here's a stick figure example. Set your font to something fixed pitch
in order to view properly.

The individual we are examining is A, it's parents are B and C. I
need to calculate the number of unique paths betwen B and C via any
and all common ancestors. In this example there is are two common
ancestors - X.and J Visually this is easy. Now how do I codify it?
Questions? Comments? Possible solutions? Thanks!

H X I J K X L J
D E F G
B C
A


As you can see this one is fairly simple. 2 paths.

B->D->X->F->C

B->E->J->G->C

This can, however, get very complicated. Threre can be multiple uses
of an ancestor on both sides, There may be only one path UP but 3 or
four paths DOWN. How can I make sure that I show each unique path,
and only include it once, and find ALL of them?

Let the questions/comments/solutions fly?

PS. I'm not prejudiced here. I have not attempted yet to tackle this
problem. All solutions will be carefully considered and I am sure
fully debated! Thanks again everyone!

My first thought on this is just to walk each path, starting with A, load
it in some kind of internal structure or table, and then check for the

Table1: B-D-H, B-D-X, B-E-I, B-E-J (4 paths, 3 generations)
Table2: C-F-K, C-F-X, C-G-L, C-G-J

So you could then tie B-D-X TO C-F-X, and also B-E-J to C-G-J. If you use
two tables, you can see every entry in table1 begins with B, and every entry

probably make each table two-dimensional. Each row is one path, each column

I left out the actual formula, as it is simple once the number of
"steps" is determined. Here it is though:

It is the SUM of the results of this formula applied to each path from
one parent to the other via the common ancestors.

For each path:

(1/2)exponent number of steps * 1/2

So a path of A-B-C would be (1/2) to the third power * 1/2.

If that was the only path, that would be the solution. This formula
is used for each unique path, and the sum of the results is the
inbreeding coefficient.

[I've already read the posts from Trembley and Blakemore, so I can't
claim complete originality, but here goes...]

In broad strokes, perhaps we can break the problem into a few
subproblems, and consider them in turn:

1. Traverse the family tree.

2. Make a list of all the ancestors encountered, incorporating
whatever information you need for each ancestor.

3. Store whatever information you need for each possible path
to each ancestor.

TRAVERSING THE FAMILY TREE

In the absence of inbreeding, traversing the family tree would
be a standard programming exercise. We would encounter each
ancestor exactly once.

With inbreeding, we may encounter the same ancestor more than once,
in which case we would encounter all of HIS ancestors more than
once as well. Depending on how your rules work, maybe we should
traverse every subtree once for every path that reaches it.
Alternatively, when we encounter an ancestor whom we have
encountered before, we could discard all of his ancestors as no
longer relevant. I think the latter approach fits your rules
better but I'll let you worry about that. Either way we're still
traversing a binary tree, and the possibility of inbreeding
introduces only minor variations.

In other languages like C or Pascal, the knee-jerk response would
be to use recursion. Most dialects of Cobol, however, don't
support recursion directly. You would have to simulate recursion
by building your own stack, pushing and popping as needed.

Alternatively, you could use a queue instead of a stack. You
could even keep a temporary list in alphabetical order. It
doesn't much matter because the order in which you traverse
the tree doesn't matter for your purposes.

The point is that you start at the starting person (the youngest
one) and repeat the following loop until no known ancestors
remain:

Create an entry in the Ancestor List, or if such an entry
already exists, update it to reflect an additional path
of inheritance. (Of course the starting person can't be
his own ancestor, but it won't hurt to include him, and
it may be awkward not to.)

Add the father, if known, to a Pending Ancestor list. This
list (stack, queue, whatever) contains ancestors who have
been identified but who have not yet been entered in the Ancestor
list described above.

Add the mother, if known, to the Pending Ancestor list.

IF the Pending Ancestor list is empty, THEN
Stop. You're done.
ELSE
Remove an entry from the Pending Ancestor list.
Just pick one; it doesn't matter which one.
GO TO the first step. (No, don't really use a
GO TO, use PERFORM UNTIL. We're talking
pseudocode here.)

I suspect that a stack will be the most convenient way to do
this, because it's so easy to implement a stack with an array.
In addition, the depth of the stack is directly related to the
number of generations at every point. Queues are a little
trickier and offer no obvious advantage.

If you haven't done this kind of thing before it's a little boggling
to contemplate. If you ponder it long enough, though, it becomes
clear that this procedure will eventually encounter every known
ancestor exactly once for each possible path. With a little
tweaking you can control the depth of the search.

MAKING AN ANCESTOR LIST

The Ancestor list is a random-access data store which is keyed
by to the identifier of each ancestor. I assume you have some
kind of PERSON-ID for each ancestor.

This list could be a VSAM file, a database table, a data
structure in memory -- your choice.

I'm not sure what should be stored as data for each entry in the
Ancestor list. You're a better judge of that. You may need some
kind of pointer into the Path list described below.

KEEPING A PATH LIST

One way or another, you need to store information separately for
each path by which you reach each ancestor. You can't store it
directly in the Ancestor list because there may be multiple
paths for a given ancestor. (In principle you might use a 2D
array but it would probably have to be huge and mostly unused.)

Suppose you use a VSAM file. The key would consist of (1) the
PERSON-ID of the associated ancestor (2) a sequence number. The
data would probably consist of the number of generations in the
path. I don't think you need to represent the path itself -- i.e.
the PERSON-IDs of all the people in the path. If you do need to
store the path then things get more complicated but you could no
doubt do it somehow, perhaps in a separate VASM file.

If you want to do everything in memory, then you could keep the
Ancestor list as (for example) a linked list. Attached to each
Ancestor node would be a linked list of Paths.

FOR MORE INFO

Cobol is neither well-suited nor widely used for representing
complex data structures in memory. With some ingenuity, however,
Cobol can do almost anything C can do. My Web pages discuss some
of these techniques.

After further reflection I think we can simplify steps 2 and 3
considerably from what I first envisioned.

As you traverse the tree: for each ancestor encountered, write a
record to a flat file. This record contains:

1. the ID of the ancestor.
2. the IDs of the ancestor's parents, if known. These
aren't necessary since you can look them up, but it
may save IO later to have them here.
3. the number of generations removed from the starting
person, along the path by which you reached the
ancestor at this point in the traversal. In the case
of inbreeding you will wind up writing multiple records
for the common ancestor and for all his ancestors.
4. (if useful) a series of IDs representing the ancestors
along the path. For full generality these should be on
a separate record type, but it would probably suffice
to carry a variable-length array of up to, say, 20 IDs.
5. Whatever else would be useful.

The next step is to sort this file by ID.

Next, read the sorted file and apply simple control break logic to
count the paths by which you reached each ancestor. Discard the
ancestors with only one path, which is presumably most of them.
Keep those with multiple paths for further analysis.

One point confuses me a bit since I don't fully understand the
rules. Suppose you find a common ancestor. All of his ancestors
are also common ancestors but probably shouldn't affect the final
score...UNLESS one of them is also a common ancestor by some
third path. You may not want to throw away a common ancestor's
ancestors until you have ruled out this possibility.

As you identify the common ancestors, insert each one into a
reduced family tree, which you can presumably hold in memory
(at least for humans; laboratory mice are another story).

For each common ancestor by two paths: traverse the subtree of
his ancestors, looking for ancestors who are common by three
or more paths. If you find one, mark the intermediate ones
as intermediate so that you won't count them inappropriately.
If you find none, discard all of this ancestor's ancestors.

For each common ancestor by three path: search his ancestors
for ancestors by four paths. And so forth.

Beyond the sort and selection my design gets a little hazy.
I won't try to figure it out any further without a deeper
understanding of the rules. However, I think something like

After looking at Thane's diagram on his Web page, and reading
some recent posts, I think I have a better idea of what's going
on.

It's not enough to know the number of distinct paths to every
ancestor. He needs the number of distinct round trips, from
the descendant to the common ancestor and back again by a
different path. These round trips are subject to the constraint
that the two paths may not have any members in common apart from
the earliest ancestor at one end and the latest descendant on
the other.

(You did explain this early on, Thane. I just wasn't paying
enough attention.)

As discussed in earlier postings, you can use a recursive
algorithm (though you don't need and probably don't want true
recursion) to traverse the family tree. Each time you encounter
an ancestor you write a record containing the ID of the ancestor
and the IDs of all the other ancestors between him and the
last descendant. These IDs will be available in the stack that
you use to implement the recursive algorithm.

If there is more than one path to reach a given ancestor, then
you will encounter that ancestor multiple times in the course of
the traversal, and write exactly one record for each distinct
path.

Now you sort the file of paths by the ID of the ancestor to which
each path leads. The sort enables you to find those ancestors
with more than one path.

So far I've been summarizing. Now comes the new idea. New to me
anyway.

Each round trip -- from descendant to ancestor and back again --
consists of a *pair* of the paths which we have found as described
above.

Suppose there are 4 paths to a given ancestor. There are 6
distinct pairs of them. Some of them may not count because they
have ancestors in common, somewhere along the way. We need a
procedure to identify the ones which count and the ones which
don't.

Once you have collected all 4 paths into some kind of array
(or whatever), go through a couple of nested loops to examine
every possible pair of paths. For each pair, compare every
point along each path to every point along the other, in order
to see if the two paths pass through the same ancestor anywhere.
If so, ignore this pair of paths. If not, plug it into your
formula.

The stack needs to be only as deep as the tree. You need enough
additional memory to store as many paths as you will ever find
for a single ancestor. For any plausible family tree these
memory requirements should be modest.

Will this work, Thane? Or is there some subtlety that I
overlooked?

Here's what I plan to do at this point.

This method uses all file IO. Some of this can be in memory.

Given an individual, create 2 files.

These files are the "paths" up the pedigree. The first is for the
Sire, the second for the Dam. The individuals I have are keyed by 8
byte ID's. The key on these 2 path files will be a path ID,
numerically assigned in series of creation.

I'll make 2 files, one for either side of the pedigree.

Here's the method of creation.

Start by making a record containing the sire. Record 1.

Start over in that file.
For that record, add it's sire to the path. (Path is FINITE - MAX 16
generations say). REWRITE that record. Get it's dam, if existing.
Create a NEW record with the dam in the path. If not, mark the record
terminated without an ancestor.

Start over in that file and repeat this loop.

When you have reached the generation limit, or have traversed the file
from start to finish without adding any new records (all terminated),
you are done. Termination will be marked with a flag.

While creating this file, write a record to a file keyed by
individual. Any individual in that pedigree gets written to this
file.

Repeat for the Dam side. Only exception is that any individual that
EXISTS in the ancestor file just mentioned above will be marked as a
common ancestor.

Examine the ancestor file for common ancestors. If none exist, we are
done - no inbreeding is present. For any that exist. Make a 4th
file. This file contains that ID and an inbreeding cooeficient (not
yet computed).

Save the data files just created for this individual.

Read the coeeficient file and repeat the above for that individual.

Track all these data file names.

When an ancestor is encountered that has no further common ancestors
for which we can compute the coeeficient, then we compute it's
cooeficient.

We do this by First eliminating any records in the "Sire" path file
that do not contain any of the common ancestors.

Next we eliminate any records in the "dam" side not containing any
common ancestors.

Then we start with each PATH and create a new file. This file
contains the path Up and DOWN for each common ancestor from the sire
side to the dam. Before writing the record, it is checked for
duplicate individuals. If each path on the sire side is considered in
series, for each common ancestor, there is no chance of duplication.
The key could be the ENTIRE path, to eliminate any posibility of
duplication.

After all are evaluated, the steps in the paths are used in the
computation and that number is stored in the common file containing
individual IDs and the coeeficient. Once all cooeficients are
computed down to the initial individual, we are done!

There are some areas here, I know that are a little gray, but it's
pretty straight forward this way, fairly easy to understand, very much
an IO pig, and probably not really fast - depending on the depth of
the pedigree.
