The simple answer is both.

When you decide to go OO then it really 'should' be an all or nothing
solution.  The screen would derive from a standard screen object and
contain the edit boxes, combo, ???? which and in turn objects derived
from generic solutions.

Surprisingly the code sounds like it is bloated but in reality it is
not.  Most objects just need the simple derivation of the parent and
do not contain their own code.  You might override the on-exit method
to do some validation but the rest is just what the parents give you.

An example of this is an FTP object in Delphi.  I can establish a ftp
client sign on and transmit a few files and it all takes about 10
lines of code.  This is the way it should be done though.

A good OO system has many smaller objects that do reasonably simple
tasks that combine to more functional (but still small) objects which
combine to major functional objects which combine to an application
object.  I have rules of thumb that a method should be 5 lines of code
and there should not be more than 10 methods to a class.  This may be
overkill but it gives you an idea of just how tiered the proper stuff
is.
