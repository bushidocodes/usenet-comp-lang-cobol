> The example contains three classes - date, time and date-and-time.

A class is a combination of the data and the methods.  These are
treated as combined.  You cannot manipulate the underlying data
without using a defined method.
 
> Method
> 
…[5 more quoted lines elided]…
> two components together as the name of the entry point.

A method is an operation that changes the state of the object.  This
is represented internally as data.

For example:  If you want to add 1 minute to a time value then do it,
you should not be concerned with how the data is represented.

I may represent the date time (ala unix) as the number of seconds from
1970. or classical yyyy, mm, dd, hours, minutes, seconds.  Your
application only cares about the external representation and should
leave it to the class to do this.


> Encapsulation
> 
…[9 more quoted lines elided]…
> handle variable length data.

Encapsulation is the protection of data from the outside influence. 
This goes back to the difference between changing state of an object
and changing data.

Encapsulation ensures that you can easily replace a data
representation with another without affecting any class except the
altered class.  This is achieved by careful design of the interfaces.

This is a requirement of OO but not restricted to OO.  I do it
regularly using standard Cobol.

Simple cobol example:  I have a routine that receives a four digit
postcode and returns a postal centre.  Internally I read a VSAM file
for every look up, then I decided to just read the file once and store
the lookup table in memory.  The external interface was unaffected,
the solution was encapsulated.

> Polymorphism
> 
…[5 more quoted lines elided]…
> each processes the data differently.

A bit more complex than that, sorry.

I have a class of people,  they have certain features, DOB, name, etc.
I have a class of customers, who are people.
I can simply use customers as if they were people due to inheritance.

Different objects will respond differently to different situations. 
For example you may not be able to remove a DOB from an employee but
you would from a customer.  Employees are sometimes customers and must
act as such but they have a specific discount set on them,
shareholders may have another, tradesmen another, and etc.  The
discount is encapsulated in the customer.   Shareholders may retrieve
their discount from a standard look up, tradesmen may be given their
discount figure individually.  The till application does not care or
worry about how that value is derived.

> Inheritance
> 
…[5 more quoted lines elided]…
> inherits its properties from its two parents,

A class can inherit an interface from it's parent and implement a new
method for that solution.

The date class is used for report titles and is initialized to now().
The requirements change and I am required to print time.  The existing
code is unchanged however I instantiate the original date with a new
type (date-time) that returns a different output to the standard calls
of date time.

My feeling is that you really are missing something about OO.  Yes you
can emulated it to some small degree.  I have heard it said that in
order to learn OO learn smalltalk or a pure OO language.  I to agree
