Hardly surprising Carlos. You will hit yourself on the head when you

Are those four names reg01 field01, field02, field03 supposed to be
describing a record ? If so, I'm surprised it compiled.

It's the LEVELS screwing you up :-

01  reg01.
      05  field01   pic  9(06)
      05  field02  pic  x(90)
      05  field03  pic  x(30).

As you previously had it, it conveyed that 02 field02 is a sub of
field01, similarly 03 field03 is a sub of 02 field02.

Now try your moves and see what happens. You have to get a handle on
these Levels - and it really isn't complicated. To make it readable
*always* indent on the Level Numbers :-

01 CustomerRecord.
    05 CustomerNumber.
           10 CustomerNumberPart1    pic x(06).
            10 CustomerNumberPart2    pic 9(06).
    05 CustomerFullName.
            10 CustomerFirstName        pic x(20).
            10 CustomerLastName        pic x(30).

You don't have to increment the Levels in 'fives' as I've done, they can
be 01, 02, 03, 04 etc., but using 'five' increments allows you later to
add further intermediary Levels.

Just from above you can now move/display data at different Levels :-

    Display CustomerRecord = 62 characters
    display CustomerNumber = 12 chars.
    display CustomerNumberPart1 = 6 chars
    display CustomerNumberPart2 = 6 chars
    display CustomerFullName = 50 chars
    display CustomerFirstName = 20 chars
    display CustomerLastName = 30 chars.

Check your text book/on line help on record design and Level Numbers.

OK to move LOW VALUES, but if you move x'00' which is a TERMINATOR, it's
telling the program that the record finishes at the character BEFORE the
first x'00' - so wouldn't use that If I  were you, until you have some
proficiency under your belt.
Unless you have a particular reason for using Low Values, you can code :-

    INITIALIIZE  Reg01

PS: Are you aware of the difference between Sequential and Line
Sequential ?
Given a fixed record size of 126, a Sequential Record = 126; a Line
Sequential Record also = 126 but is automatically followed by the two
characters CR/LF(Carriage Return/Line Feed) = x"0d0a". So if you view a
Line Sequential, (text file) on a screen, the CR/LF after each record
makes the screen jump to the next line for displaying.  You CANNOT mix
Sequential and Line Sequential.

I'm assuming the given field level numbers are mistakenly 01, 02 &
03...in answer to your questions as to why all A's works, and all
nulls doesn't.  Have you considered that null values side by side are
(as in x00 followed by x00) an end of record/file mark on some
systems?

What may be happening is, the system is expanding the String of nulls,
with a filler byte, to avoid writing to a file the null sequence that
signals an end of file.

The difference (156 - 126) is 30...the value after doubling the length
of the last field to insert these masking bytes.

If you need the "filler" for this field to be nulls within your
program, make it nulls after you read the record and see the mask
characters in the field.  In other words, write A's to the file, then
when you read it and see all A's, move nulls to the field in your
program.

Well, one problem you will have results from moving low-values to a line
sequential record. NEVER put ANY binary values (i.e., non-printable)
characters in a line sequential file. The file read/write logic may/can/will
interpret these non-printable characters as control characters.

For example, if you define a variable as PIC 9(4) COMP-5 and the variable
contains 10,the result looks like "0A00" (in hex). The line sequential file
routines decode this as a carriage return and you monitor will hop right off
the desk.

Anyway, to fix your current problem, and prevent future ones, move SPACE to
FIELD03 instead of low-values.

Or did you just key it in wrong ?

The first number on each line is a _level_ number and not a sequence.

It should be:

   01  Reg01.
       03  field01     PIC 9(6).
       03  field02     PIC X(90).
       03  field03     PIC X(30).

The 01 starts a record, each 03 start a field within the record.  Each
line (data statement) must be terminated by a stop '.'.
