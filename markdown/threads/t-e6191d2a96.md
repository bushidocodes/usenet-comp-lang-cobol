I just thought I would start a new thread - to follow up on a side
discussion in another thread.  (Basically, I disagree with Bob Wolfe - for a
change - but just thought I would raise some issues in this new thread.)

It seems to me that there are a few types of "support" for features that can
be in a programming language.  I will "assume" COBOL here, but much (not
all) applies to other languages as well.

At a minimum, there are (some overlapping with others):

 - Portable (Standard conforming) "native" language support
 - Non-Portable (extensions) "native" language support
 - "Alien" language support (e.g. EXEC SQL or COBOL ENTER statement - not
ENTRY statement)
 - Callable API support (related to "Library" support)
 - Interpreted vs compiled vs object vs machine level support

***

I (personally) do not consider the difference between "interpreted" and
"compiled" code to be relevant to discussions of what a specific language or
compiler supports.  I consider "compiled" and "interpreted" JAVA to be the
same - and I consider "compiled" and "interpreted" Micro Focus COBOL to be
the same.

I (personally) consider that AcuCOBOL's "GUI" extensions to be "native" (but
not portable) COBOL language support.  While CICS' "EXEC CICS SEND/RECEIVE"
to be "alien" language support (the EXEC to END-EXEC statement "escapes"
from the compiler).  While COBOL systems that require "Call WinAPIxxx"
statements are using external APIs.

It is interested where there are also "overlaps" between "native" language
support and some of the others - via "translators" or "pre-compilers".  The
SPC Report-Writer product allows the application programmer to code their
source code as if they were using native COBOL language (ANSI defined)
report-writer syntax - but actually "translates" this into other statements
(but not MOVEs followed by CALL statements - rather standard COBOL I/O).
Similarly, all EXEC SQL and EXEC CICS systems use some form of "translation"
that may or may not look transparent to the programmer (but do end-up being
a MOVE followed by a CALL).

Other fine lines are systems like Micro Focus' C-ISAM support which
"translates" native COBOL file syntax into run-time calls to a 3rd party
product.  (Similar to IBM mainframe applications that use SyncSort or DFSort
to "implement" the native COBOL SORT/MERGE syntax).  On the other hand, IBM
mainframers who use "CALL CBLTDLI" or PC coders who make explicit calls to
BTRIEVE modules are clearly using "API interfaces".

  ***

As the saying goes, all of this is (eventually) translated into on/off (1s
and 0s) - not even HEX much less Assembler.  It seems to me that the
"significant" question is what the application programmer codes.  If we are
talking about COBOL, then the minute that they need to put a "CALL some-api"
or EXEC statement in, then it is not native language support. If they can
access the functionality via a "COBOL verb," then it is native language
support.  If the verb (and all the operands, semantics, and syntax) are as
defined in the ANSI/ISO Standard, then it is "portable native language
support".

   ***

It is hard for me to think of any of this as "controversial" - but maybe it
is.

I wanted to add a few "thoughts" to my original post - which Bob's note
reminded me of (and which may not be clear from my original post).

1) The differences between compiled and interpreted code may WELL impact
"performance."  Obviously, this needs to be considered in producing
production code. On the other hand, the difference between interpreted code
and compiled code  almost certainly will impact WHAT you port - if not where
you can port it to.

2) Even for "interpreted" code, there are differences between systems.  For
example the (I don't know if it is still current) requirement for AcuCOBOL
applications to "rebuild" their run-time system for each application that
calls C subroutines - while this is not true for Micro Focus ".INT" code.

3) There are portable applications and then there are portable systems.  The
use of GoBack or Comp-3 (as packed-decimal) are clearly non-ANSI code.
However, given IBM's dominance in the 60's and 70's in the COBOL-mainframe
worlds, these are *almost* universally supported and portable.  On the other
hand, the use of the PADDING CHARACTER phrase in the FD is ANSI-conforming,
but unlikely to give "portable" results - if you expect it to do anything
useful with it.

4) As discussed in another thread,  it is often (usually?) a good idea to
"isolate" your vendor-specific or operating system specific code in called
sub-routines - if you think your application will be "ported" to other
vendors or environments.  (Alternatively, - for example in the case of the
SELECT/ASSIGN statement - a COPY statement may do the trick.)  ON THE OTHER
HAND, if you are developing a response-time critical application, using
every "low-level" interface and extension - without a CALL statement, may be
the best way to go.

5) In my original note,  I hope it was clear that I was trying to describe
some "classifications" and NOT trying to say one was "good" while another
was "bad".  We have already all (I think) said that learning to use the
right tool for the right task is clearly the way to go.  On the other hand,
when you are still trying to decide which tool to use, it is NICE if you can
use terms that mean the same thing to all involved in the discussion and
decision making process.

In the middle of last night, it suddenly dawned on my why Bob had originally
talked about "native" versus "interpreted" support - and that vendors had
used this distinction.

Going back to my Micro Focus days, there is a concept of "NCG" or "Native
Code Generator" which is a "part" (well sort of) of the compiler.  This is
the "thing" that converts ".int" (or interpreted code - or p-code as it is
called in some environments) into "machine" or "object" code - which is
referred to as "native code" - and which is certainly DIFFERENT from
interpreted code - and which differs on each machine platform (operating
system).

When I (we?) talk about "native language" support for a feature, we are
talking about SOURCE code.  When vendors talk about "native code" support at
run-time, they are talking about machine code.

I had never thought of "confusing" these two concepts, but I certainly can
see how it could happen.

Sorry, Bob.  You are right (when talking about THAT type of "native" code)
that it is quite different from "interpreted" code.
