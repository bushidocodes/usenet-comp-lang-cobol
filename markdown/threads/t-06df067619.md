```
I am running MF Object Cobol 4.1.40 on an HP-UX 11i machine.

There is a directory on the UNIX machine which is an NFS mount to one
of the PC's on our network.

Using the CBL_* routines I can copy/move files TO the NFS directory
and also check if a file exists, but if I try to I can copy/move files
FROM the NFS directory, the run unit hangs.

Example:

These work:

CALL "CBL_CHECK_FILE_EXIST" USING NFS-FILENAME RETURNING STATUS-CODE
CALL "CBL_RENAME_FILE" USING LOCAL-FILENAME, NFS-FILENAME RETURNING
STATUS-CODE
CALL "CBL_COPY_FILE" USING LOCAL-FILENAME, NFS-FILENAME RETURNING
STATUS-CODE

These fail:

CALL "CBL_RENAME_FILE" USING NFS-FILENAME, LOCAL-FILENAME RETURNING
STATUS-CODE
CALL "CBL_COPY_FILE" USING NFS-FILENAME, LOCAL-FILENAME RETURNING
STATUS-CODE

From the UNIX prompt I can execute all of these operations without any
issue, so I don't think this is something at the UNIX level
(permissions, etc.).

Any suggestions?

Thanks in advance.

Chris
```

```
I am not sure if my problem is the same as yours except for the error code
14605.

What I did was place a " around each file name.

STRING """           DELIMITED BY SIZE
       WS-INPUT-PATH DELIMITED BY SPACE
       WS-FILE-NAME  DELIMITED BY "   "
       """           DELIMITED BY SIZE
       INTO WS-CBL-FILE-NAME-1

STRING """           DELIMITED BY SIZE
       WS-OUTPUT-PATH DELIMITED BY SPACE
       WS-FILE-NAME  DELIMITED BY "   "
       """           DELIMITED BY SIZE
       INTO WS-CBL-FILE-NAME-2

CALL "CBL_COPY_FILE" USING WS-CBL-FILE-NAME-1
                           WS-CBL-FILE-NAME-2
                     RETURNING WS-CBL-RETURN-STATUS.


Hope this helps.

-Jason
```

```
I don't know if this will help or not, but at:
  http://supportline.microfocus.com/supportline/documentation/books/sx40sp1/sx40indx.htm

under the topic

"Interpreting the return code as a file status code"

it says,

"If any of the routines

CBL_CHECK_FILE_EXIST
CBL_COPY_FILE
CBL_CREATE_FILE
CBL_DELETE_FILE
CBL_OPEN_FILE
CBL_READ_FILE
CBL_RENAME_FILE
CBL_WRITE_FILE
CBL_CHANGE_DIR
CBL_CREATE_DIR
CBL_DELETE_DIR
CBL_READ_DIR

fails, the RETURN-CODE register contains a file status value indicating the 
failure. This file status is always the standard ANSI'74 file status value. If 
no ANSI'74 file status is defined for the error, an extended file status is 
returned (9/nnn where nnn is the run-time system error number).

You should, therefore, use RETURN-CODE and not a RETURNING clause. If 
RETURN-CODE is non-zero after calling one of these routines, you must process it 
as a file status, for example:
 01 file-status      pic xx comp-x.
 01 redefines file-status.
     03 fs-byte-1  pic x.
     03 fs-byte-2  cblt-x1-compx
   . . .
     call "CBL_xxx_xxx" using parameters
     if return-code not = 0
        move return-code to file-status
           . . .
At this point fs-byte-1 contains "9" and fs-byte-2 contains the run-time system 
error number."
```
