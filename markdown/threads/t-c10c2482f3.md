```
In another post the subject of renumbering paragraphs came up.
It does cause extra work because the renumbered paragraphs have to be moved
to keep the paragraph numbers in sequence.

We ARE programmers, it seems that this is a situation where we should let
the machine do the work for us.
You can sort anything, if you build the appropriate sort key, and append the
original data to it.

A stand alone batch program, or edit macro could make the resequencing a
non-issue.
Just change the paragraph number, and the statements that invoke it, and let
the machine do the rest.
```

Howard,

Yup, you're correct.  We wiykdn't dare pound the copybook with new
paragraph numbers because it would break all the other programs that
use it. We won't break a program that refers to paragraphs, though;
read on.

If your program refers to a paragraph in a copybook, COBClean won't
modify the reference in, say, a PERFORM statement because it can't
find the target (in the copybook).  No problem there.

If your copybook refers to a paragraph in the host program, well,
you'll have a problem because COBClean doesn't know about that
reference.  Of course, it would be rather risky to have code in
copybooks referring to paragraphs outside of the copybook so I'd doubt
this would be much of a problem.

If you want to reformat DATA DIVISION copybooks, it's a simple matter
to wrap a little stub program around them to get the level numbers and
indenting straightened up.

In actual practice, the product works extremely well.
