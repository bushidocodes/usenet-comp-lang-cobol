I work for a large corp with complicated data structure in huge
database. Theoretically, we can frozen the environment, work with
the database team and ask them to rollback the database to the
last backup version. Base on the audit trail system logs, re-run
all those online transactions and re-run any batch jobs until it
reaches the point where the job bombs. Fix the job and re-run
everything after it. Sound OK right?

In real terms it means to shut down all the user connections and
wait for the database team and the systems team fixing the problem.
It takes at least 30 working days and required more depends on
when the DB backup was.

If the company cannot wait for that period, then patching the
data is the only solution. Usually, it takes 3 to 5 days for the
patching including coding and testing.

Well, that is true for sure, but on the other hand changing a working
program to handle one piece of bad data is just as dangerous.  I have
found bad data many times that is ... an anomaly? ... perhaps a bad track,
perhaps a power failure, perhaps a bug somewhere else.  But picture
a nice regular flat ASCII file with a big block of garbage in the middle.

If there is backup, great.  If there is not, then as you say, you have
no choice.  Also, going to backup in some circumstances creates
more problems than it solves.  Often, changing a program can
create greater havoc than fixing the data.  If you fix the data wrong,
there is probably a bad invoice out there. If you fix the program
wrong, there could be several thousand of them.

Mind you, I have an advantage.  Rather than working in a large Corp,
I work for myself, and the systems that I work on I wrote, so I know
what the data should look like and what effects it will have. My typical
installation is a dozen PC's in a network.

Interesting seeing the comments here.  I would bet that the risk
of changing data goes up in direct proportion to the system
complexity, and so the answer depends largely on what people
work with.
