As you've seen from Howard's reply, 'veterans can code quickly, so why
bother....'.
Possibly a sentiment others share, and yet amongst the mainframe crowd (the
majority), they must consistently be calling common routines, or even nesting
programs etc., which is on the same tack as your question.

OK - so I'm into OO - which I think offers far more possibilities. But first
let's back-track to non-OO. In previous messages Richard Plinston has said he
wasn't happy with the syntax available for Screen Section. So he wrote his own
routines, which I believe to be common to all his screen handling.

Initially starting on OO (Net Express) I really floundered with no comprehension
of what OO was about and how these damned GUIs fitted into the whole.
Arranga/Coyle firstly, and then Wilson Price 'Elements of OO COBOL' gave me the
breakthrough as to OO's simplicity once you get beyond the concepts - the $10
words, encapsulation, inheritance, polymorphism etc.

When I first coded, each item selected from the Master Menu was perceived as a
separate entity with its own (business) logic, plus screen and file/DB handling.
As I got more comfortable with OO I latched on to the fact I was continuously
repeating myself - particularly with GUIs.

Stop coding. Put the brakes on and give it a 'real think'. Right back to design.
Quite naturally design is a very personal thing - and there is NO RIGHT way,
there are alternative approaches, just as successful as yours.. In the case of
large installations, even allowing for 'individual stylists' it makes sense,
even in non-OO if you can come up with an installation standard - master
template, that you can repeatedly use for say Editing a Masterfile, (Customer
Names, Chart of Accounts, General Ledger etc.).

Harkening to what Richard wrote, I realized I had to back-track to the beginning
on GUIs, as they inter-play with the rest of the N/E product. My visual design
tool is Micro Focus Dialog Editor - which follows very closely the Microsoft
Dialog Editor. You create your screen images visually then allocate IDs for the
controls. (Note that the Fujitsu POWERFORM and M/F's FORMS modules do a better
job setting properties). Nevertheless the Dialog Editor is not bad plus when
coupled with dynamically 'painted' tables (you try and do a 10 x 24 table, with
a 'static designer <G>).

So rather than pay lip service to the Dialog Editor I read the file and extract
a set of parameters for the controls. Can't do fancy diagrams here, but looks
like following :-

[A : EditProgram]----->[B : DialogParameters]-------->[C :Instances of Dialog]

"A" invokes "B" to pass the controls needed to "C". That being done  "B"  can be
dropped (finalized/destroyed) and the inter-activity from Windows events is
between "A" and "B". While t"A" and "B" are unique to a particular business
problem, "C" is common to all dialogs I create.

Same goes for COBOL file handling :-

[X : EditProgram]----->[Y : DBI]-------->[Z :Instances of file classes]

A copy file (e.g. CustomerRecord) is used in both X and Y (Database Interface)
and "Z" is a 'generic' file class ISAM, Sequential, Linesequential, Relative
achieved by those classes having a record :-

01 DataRecord.
    05 occurs 1 to n depending on ws-Recordsize    pic x.

Most messages bypass "Y" above which is primarily used for creating (a)
collections/dictionaries - prime requirement being Listboxes and (b) accessing
records in "Z" and selectively stripping off fields in "Y" to be passed back to
"X".
(Same sort of technique used with DBs and SQL - except - "Z" is not an instance
in the general meaning, but relates to a specific DB table).

We aren't quite there yet - but hopefully some seeds are starting to germinate.
So back to what Richard does with Screen Section. I read the Dialog Resource
File and generate the parameters for "B". I don't need to worry about "C" as it
will handle all dialogs (at least to date <G>). File handling "X", "Y" and "Z"
apply and for SQL "Z" is applicable to only one table.

I wrote "generate the parameters for "B"...". But coming up with my own design
style for editing, using an 'Edit Template' - I can generate 80% plus of the
code for "A". The model is fairly crude at the moment and I use 'search/replace'
to change copyfile references.

As crude as it may be, and obviously it can be refined to the nth degree, it
sure is going to save me a heck of an amount of time in the future. Mine is a
little guy's version of what those very expensive tools like SAP will do for
you. The only problem with tools like SAP is that they are heavily overloaded
with features to cover every eventuality and to appeal to as many potential
customers as possible.

Even without OO and its own particular techniques, I believe the above
short-cuts can pay off in non-OO development.

PS : I'm no expert but can claim to be a veteran if you take age into account
<G>

Jimmy, Calgary AB

Not a bad objective - but we aren't there yet, but it does lean towards Pete
Dashwood's concepts of using components. Thane mentioned quite a while back, (he'll
have to confirm - it may have been Java), that as you are coding there are some (?)
languages which prompt you to select the right method - not totally what you are
after, but close.

I was curious that there currently didn't appear to be mainframe tools but both
Thane and Chuck made reference to that. The snag is generators tend to be
overwhelming to cover everything - just like SAP does. Needless to say the latter
comes at a hell of a price. I'm now aware of three SAP users in Calgary but they are
all 'big league' - two are oil companies and the third is into agriculture
fertilizers, primarily using natural gas to produce their end products..

The other message that came across with the 'generators' was - 'No touchee source
code otherwise you can finish up in trouble'. That's the sort of message which will
make the bravest of us back off <G>. Even given a tool like SAP - get your initial
design screwed up and you can make it crawl instead of run.

It's a snap to do the SQL stuff in Net Express because of ESQL Assistant doing the
COBOL source coding for queries. The PC world, I would suggest, tends to lend itself
to generators, because there aren't too many programs that don't include a Dialog.
Where you are using JCL on mainframes the PC equivalent for printing a report, very
likely contains a dialog for checkboxes, "print this, in this sequence..', 'Don't
print this one...' etc.

A Window and its controls are finite. Even for the N/E Entryfield Class - there are
some 46 methods, (and I'm assuming these match the API equivalents) - but it's
likely the average programmer possibly wont use more than some 20%. That being said
it becomes possible to set the focus on the controls being used and ensure that when
generating a business logic program, it at least contains the skeleton methods for
those controls.

Depending upon your design some are very generic - take my use of the Pushbutton OK
event returned to business logic from the generic Dialog handler :-
*>-------------------------------------------------------------
Method-id. "PB-OK".
*>-------------------------------------------------------------
Local-storage section.
01 n                             pic x(4) comp-5.
01 n2                            pic x(4) comp-5.
01 ls-response                   pic x(4) comp-5.
Linkage section.
01 lnk-flag                      pic x(4) comp-5.

Procedure Division using lnk-flag.

move lnk-Flag to RecordChangedFlag

*>******* here we start customized coding **********

if   (Probe-ProbeID = spaces) OR
     (RecordNotChanged)
     invoke self "displayEmptyScreen"
     EXIT METHOD
End-if

set ValidationOK to true
perform varying n from 2 by 1 until n > ws-MaxEntryFields
      invoke os-Dialog "returnEntryObject" using n

*>.Above is returning the value of each specific control to the appropriate skeleton

*> method in this program - where the 'customized' validation occurs

      if ValidationError
         EXIT METHOD
      End-if
End-perform

set Field-RB-DUAL-ELEMENT to true
add 1 to Dialog-Field giving n2
invoke os-Dialog "returnRadioObject"
       using Dialog-Field, n2

if not ValidElement
       set Error-Element to true
       invoke self "messagebox" returning ls-Response
       set Field-RB-DUAL-ELEMENT to true
       invoke os-Dialog "setNextFocusClear"
              using Dialog-Params
End-If

*> ******* customized coding ends here ******

if ValidationOK and RecordChanged

       if     RecordFound
              invoke self "rewriteRecord"

       else   invoke self "writeRecord"
       End-if
End-if

*> Write/Rewrite above - same terminology for Files or DB Tables - except
*> within the methods for the latter the SQL syntax is INSERT or UPDATE

if ValidationOK AND ContinueProgram
   invoke self "displayEmptyScreen"
End-if

End Method "PB-OK".
*>-------------------------------------------------------------

procedure division code.  I have used this for a number of program,

work that was generated for me.

preferable and more interesting than many small programs and wrote
script (text file) driven file maintenence program and report writer.

These don't generate code, they just use the control information to
specifiy the file dictionary, screen layouts for file maintenence,
report layouts and selection/sort criteria.

These allow me to get a new type of system up and running very quickly
to the point where data is being setup while I develop the processing
modules and specialised input for it.

These were pulled apart and new SP/2 front ends stuck on for TUI/GUI.
I mostly use SP/2 in text mode.

implemented on Burroughs systems in the days before the merger,
implementations are available on a number of platforms, including (but not
necessarily limited to) Wintel and Unix as I understand it, in addition to
Unisys MCP/AS systems.

Although it is possible to modify the generated COBOL source code, tools are
provided for debugging at the application generator level, and it is not
recommended that users get directly involved with the generated COBOL
source, concentrating instead on the specifications for the application and
debugging at the higher level.  The EAE / LINC developer need have no
knowledge of COBOL at all.

Many large corporations have significant EAE / LINC applications; it has
been a very successful product.

It isn't, strictly speaking, a program that generates COBOL fragments; it's
a 4th-generation system that generates applications, and the procedural code
that it generates to support those applications happens to be in COBOL.

1) First and foremost, if your program has *no* "NEXT SENTENCE" phrases (or

  A) the programmer THINKS (incorrectly) that a "Next Sentence" is always

  B) when the source code is "modified" and a new period is added or an old

  C) when "nesting" an existing piece of code into a new conditional

 Read XYZ
   At End
      Call "ABC"
      End-Call
   Not At End

 Read XYZ

        On Exception
             Display "Program 'ABC' wasn't found or loaded"
      End-Call
   Not At End
      Perform big-loop
End-Read

"unexpected" matching of conditional situations

 If  A = "A"
   Add 1 to NumField
      On Size Error
                 Add 3 to NumField2
Else
   Display "A isn't 'A'"
End-If

 If  A = "A"

there must have been enough belief and agreement between the compiler
manufactures, the standards body and compiler users that a more
granular scope terminator was necessary.  They certainly did not
up-end the entire language for a cosmetic change.  There had to be
some substance behind the move.  Otherwise they would have happily
continued using full-stop periods and the Cobol world would never have
heard of END-Anything.

the equal of explicit scope terminators.  That the choice between the
two is a mere preference.  This idea is directly at odds with whatever
reasons the Cobol community had 2 decades ago for enhancing scope
termination options in the first place.

It isn't (as I am often accused) a case of 'old == bad; new == good'. 
It is a case of 'old == ok; new == improved'.  That we did not see a
move for a change in scope termination in the 2002 standard seems to
indicate that the Cobol-85 standard fixed whatever problems existed
with scope termination before.

The fact that the Cobol community at large decided they needed
enhanced scope termination is evidence enough that periods are not as
good as explicit scope terminators.

The only contexts in which standard COBOL allows NEXT SENTENCE are in IF and
SEARCH statements, and ISO/IEC 1989:2002 designates its use in those
contexts (which, in standard COBOL, is everywhere, though some
implementations extend its use beyond these two verbs) as an "archaic
language element".

From Annex G of that document:  "The purpose of the archaic language element
designation is to discourage the use in new programs of some features ...
These elements are classified as archaic rather than obsolescent because
their use in existing programs might be too extensive to warrant removal in
the next revision of standard COBOL ... There is no plan for the removal of
archaic elements; however, should their use in program inventories become
insignificant, they may be considered for designation as obsolete by future
architects of standard COBOL."

I for one am not really comfortable with demanding that all programs
containing NEXT SENTENCE suddenly become uncompilable on that basis alone.
While the contract programmers might love the opportunity to make lots and
lots of money reengineering existing code to avoid the constructs, I am not
convinced that requiring the companies to whom that code belongs to expend
those funds fixing code that would otherwise work just fine is defensible on
economic grounds.
