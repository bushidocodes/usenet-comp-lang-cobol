```
efficiency) if comp-5 and platform independent (MSB first) if comp-4

check the specific implementation.
```

```
It has been a while since I have had to reply to this totally false (but
persistent) rumor.  THERE IS NOTHING in the COBOL language definition that
says that COMP is  supposed to be

   "native format for efficiency"

The only thing that the COBOL language definition requires is that COMP be
numeric and that the implementor defines what it means and how it is
implemented.

If you check DejaNews, you will find various thread about all the different
ways that COMP can be stored and used - on different compilers and different
platforms.

The COMP-n USAGEs are not even implementor defined (according to the
Standard) but are actually extension.   There is certainly no "requirement"
that COMP = COMP-4 (and as we often point out in this NG, COMP= COMP-4 on
IBM mainframes but COMP= COMP-3 on IBM OS/400s).

You might be interested in using the ANSI/ISO Standard USAGE BINARY and
USAGE PACKED-DECIMAL if you are interested in source code portability - but
even these are "implementor defined" when you get down to the details.
```

```
Dig a little deeper into NetExpress.  What you are looking for is
called WFL, or File Converter.  You can feed it an FD (Actually an
.IDY from a compile) of a COBOL program and convert the data that way.
It works right every time.  Where I used to work we had a product in
the field where users would dial up the mainframe, dload the EBCDIC
data file (packed fields and all) and then the program called WFL to
reformat the data to ASCII.

MAKE SURE you are using a BINARY download from the mainframe, or else
you data is "pre converted" and the packed fields will CERTAINLY be
messed up.
```
