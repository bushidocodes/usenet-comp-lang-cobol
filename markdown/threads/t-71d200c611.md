I happen to believe that I can optimize COBOL performance by techniques 
in source code.  It's certainly possible to get bad performance with 
poor COBOL coding techniques.  But I would have to agree with you that a 
modern optimizing COBOL compiler can write much better Assembler code 
than the best human programmers.

A few years ago I assigned some Indian programmers to convert three 
assembler binary search programs into COBOL.  This included both batch 
and CICS assembler programs.  I believe we were then using IBM COBOL for 
OS/390.  The current compiler would be IBM Enterprise COBOL for z/OS 4.2.

We could not use the COBOL SEARCH ALL verb because the key to each row 
in the table was a from-through range rather than an elementary item. 
So we had to use a hand coded binary search algorithm.

The point of the assignment was to reduce our dependence on assembler 
programmers and potentially make the code more portable to a different 
machine architecture.  We didn't try to code for maximum performance, 
but rather to guarantee that the COBOL binary search produced the 
correct answers.  And to keep this on topic, the COBOL programs did not 
use any GO TO statements.

After they got the code working I had them do some performance testing 
and it was discovered that the COBOL programs were consistently faster 
than the assembler programs.  That was a pleasant surprise.

This thread was cross-posted to bit.listserv.ibm-main, but not all the 
comments there appeared here in comp.lang.cobol.  I believe it was Tom 
Ross of the IBM z/OS COBOL development team who pointed out that a 
number of improvements were made several years ago, beginning with IBM 
VS COBOL II, that made PERFORM much more efficient.  It fact, using GO 
TO is probably worse than using PERFORM now, because it prevents the 
optimizer from generating the most efficient assembler code.
