event).

In OO - using "Reverse" giving you "descending"  - I must confess that I don't
see the use for that. - but I bet there's somebody out there...............

Note: above is N/E - not the future Standard.

As regards Table Sorting, assuming you were doing a sequential read on an ISAM,
by default that's going to give you an ascending by PrimeKey, where you have
used PrimeKey as your default READ NEXT . To get a different sort  it is
basically :-

        sort table ascending xxxx
                        descending yyy
                        ascending zzzz etc....

Then depending upon your sort sequences you are back to your Search techniques.

I don't think I have, but in case I've left you puzzled on "ascending" for a
Table Sort - I have a Dialog where user for certain sub-record fields can
create up to 4 sub-lines. I don't restrict users to entering them in
"ascending" - just as they see fit. When they've finished,  then for the four
lines I do a sort ascending, to store them in a logical sequence when
writing/re-writing the record.

Another example of Table Sorting "ascending", (although it's most likely done
with OO Lists - depending upon which language Amazon uses)  - you traipse
around amazon.com randomly selecting some thirty books putting them into your
'shopping basket'. When finished, Amazon takes that list and resequences them
say :-

- sequenced for picking from the warehouse and order/delivery documentation
- perhaps a book category sequence for the actual invoice.

 Using the simple Table Sort syntax this particular rocket scientist doesn't
have to bother with the 'ins' and 'outs' of sorting <G>. BUT - my major

Something like:

    input procedure release-table

For large volumes I would guess Unix systems come pretty close to mainframes in

In the PC-world, bearing in mind that applications tend to be tightly knitted
together, with no pause in the operation, then what Rick has illustrated can be

Even what Rick has given us can be paraphrased. I'm accepting old RM/COBOL 74
data from ISAM files to be used as input to DB Tables. I read those files
SEQUENTIALLY, i.e. as though they had been specified as ACCESS SEQUENTIAL,
totally ignoring the indices. That doesn't give me the ascending sequencing I
want for the intermediary Line Sequentials. I'm doing it with OO, but the
method shown could just as easily be a small called non-OO program :-

There are seven inputs so for ease I just created seven methods like the
following.
A quick one-off operation, I didn't see the point in trying to make one method
fit all seven files.
*>--------------------------------------------------------------
Method-id. "sortItems".
*>--------------------------------------------------------------
FILE-CONTROL.

     Select               Data-File
     assign               Data-filename
     organization         LINE SEQUENTIAL.

     Select               Sort-File
     assign               Sort-Filename.

FILE SECTION.

FD  Data-File.
copy "rmitemsA.cpy" replacing ==(tag1)== by ==01 Data-record==,
                               ==(tag)==  by ==Data==.
SD Sort-File.
copy "rmitemsA.cpy" replacing ==(tag1)== by ==01 Sort-record==,
                               ==(tag)==  by ==Sort==.
Procedure Division.
  Sort Sort-File on ascending Sort-Record
       using     Data-File
       giving    Data-File

End Method "sortItems".
*>--------------------------------------------------------------

Max volumes probably around 9,000 records for the largest file of the seven  -
so no big deal volume wise. It's integrated into the application and works in
about under one minute for all seven files.

Rather than boning up on Sort syntax I 'lifted' that straight from the examples
in Thane's 'TYCI24H' - merci bien <G>.

Syntax like "sort table-name ascending key table-field" is part of the 2002
standard (and has been since very early drafts), but not the 1985 standard.

I'm not an OS2200 expert, but I have access to some internal material for
that system.  Based on a very limited search it appears to me that it was
NOT included in the non-OO features pulled from an earlier draft of that
standard when OOCOBOL was built for that platform, either, which means it's
unlikely to have been in UCOB, which I believe predates OOCOBOL.

But there is at least one other alternative besides sorting the table "by
hand":  use an INPUT procedure to pass the table elements to the sort
mechanism using RELEASE, and an OUTPUT procedure to retrieve them using
RETURN.  The way I read the rules of COBOL sorts, the input is completely
processed before output begins, so it strikes me as safe to RETURN the
sorted table entries into the table from whence they came.

    -Chuck Stevens
