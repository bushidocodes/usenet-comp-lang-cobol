This may not be exactly what you're after, but in the latest update
for V4.0 (and in a forthcoming update to the 16bit product line), the
early release GUI Data File Editor allows you to open you data files
in shared mode. This means you can look at the file while it's in use
in your application and see which records are being locked.
Unfortunately you can't see who actually has the lock, but if you're
using Fileshare, the FSView tool will probably help.

We've used these tools internally to debug some of out own apps that
seemed to be hogging locks with great success.

This would be environment specific, and completely different between
Unix and DR-Multiuser-DOS, probably different between Unicies as
well.

And then there may be NT or OS/2 acting as application servers.

But why are you wanting to do this ? In any reasonable system the
programs will record lock only what is required and for the
shortest possible time in some sequence that prevents deadlocks.

As the locks will be changing frequently a snapshot may not be
particularly significant, unless you have a programming problem.
