there's a number of different options for compiling Java. briefly (and with
limited knowledge ;-):

* INTERMEDIATE compilation of source->byte-code where the byte-code
represents a virtual (or abstract) machine. this code requires an
interpreter (an instantiation of the virtual machine) to execute it. the
byte-code is very portable -- just requires a resident interpreter, but
generally doesn't run as fast as native code. [MERANT COBOL supports this
model].

* DYNAMIC compilation of byte-code->native-code (more commonly known as a
Just-In-Time [JIT] compiler) -- as the application executes, code ahead of
the execution queue is analyzed and selectively compiled into
machine-instructions. since this compilation is a "feature" of the
interpreter the byte-code remains portable but has the potential to run
faster. there is however, an overhead to executing apps this way, so it
doesn't scale very well. [MERANT COBOL does NOT support this model].

* STATIC compilation of byte-code->native-code (more commonly known as an
Ahead-Of-Time [AOT] compiler) -- code is compiled to machine-instructions
ahead of execution. generally, the fastest and most scaleable way of
deploying an application. obviously, native-code is generally not portable
between different platforms and processors, but for execution most Java and
COBOL applications simply need a recompilation of the application source
code on the new target platform. [MERANT COBOL supports this model].

* ADAPTIVE compilation of byte-code->machine instructions (a la Sun's
HotSpot Java RT/compiler). somewhat similar to JIT, except that the end-goal
of an adaptive compiler is to execute native-code. it does this by
iteratively compiling the executing application. the idea being that over
multiple executions the application will run faster and faster until the
application is n% saturated as native-code. a more aggressive approach than
JIT and difficult to manage in an environment that needs to scale. [MERANT
COBOL does NOT support this model].

Steve,

Right time for me to jump in and support Thane here.  The JVM does not
"interpret" the code Java Byte Code (not in the way most people would
think).  This is a multi-layer process (and is VERY similar to the way
Micro Focus implemented its own INT code and virtual machine many,
many years ago) and I will attempt to show the layers.

A programmer will write a program and store it in a source file.  This
could be Java source, COBOL, C etc.

The compiler then converts the source (syntax checking etc.) in to
Java Byte Code - yes, you could have a COBOL compiler that
generated Java Byte Code - there's an idea <g>.

The JVM should then be thought of as a "translator".  At runtime it
translates the Java Byte Code in native code (e.g. Intel) and this is
executed with the help of runtime support modules.

I hope this makes sense - I have tried to simplify what is a very
complex topic.

For more information see:

http://java.sun.com/docs/books/vmspec/2nd-edition/html/VMSpecTOC.doc.html

Hope this helps.
