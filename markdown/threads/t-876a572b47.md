I'm in a large Unisys 2200 shop (ClearPath IX8400 with 8 processors
partitioned 4 and 4) and we have no problems with COBOL performance. I know
I have a lot more horsepower than an 2200-600 but depending on your
application inventory porting to a new platform could be quite expensive.

Based on the code base that we have, a lot of batch programs use
Unisys-specific features, some of which are accessed via assembler routines.
Some stuff could be converted fairly easily but some I think would be more
troublesome. TIP transactions are potentially your worst problem. Since your
screen I/O handling is probably unlike anything you would be moving to, and
the TIP environment is rather unique, it could be prohibitive to do a port.
You might be closer to a rewrite. What about database access? If you use
DMS2200 you're pretty much in rewrite territory again. If using RDMS you
might have a shot at something.

There is a company in Ontario, Canada, (I don't remember the name) which
claims to have a product(s) which would allow you bring much of your code to
NT. I know they support a TIP environment and I think they even do something
for the DMS code. Looked interesting.

You also don't mention whether you are using ACOB or UCOB COBOL. From the
version you mention I'm guessing ACOB. We saw a decrease in execution times,
sometimes significant (25-50%) by moving applications to UCOB. I think your
machine supports UCOB and extended mode but I'm not sure. The amount of pain
involved in that switch is also dependent upon features used in your current
applications.

Douglas Gallant
