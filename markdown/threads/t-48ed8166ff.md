All -

I (and, I suspect, many others) have been following the prefixing/large
programs/linked suites of small programs debate with a feeling that the
more vocal participants lack a certain sense of proportion. Any - ANY -
coding style can be used to produce coal or diamonds, depending on the
programmer. I've been using COBOL for 17+ years, 14 of those as a
consultant; I've seen lots of code. NO technique can be guaranteed to
produce what I'd see as quality code, unfortunately. Which brings me to
my point: none of these things counts worth a damn when combined with (a)
idiotic naming, (b) poor program layout, (c) badly-expressed or incorrect
comment lines. But I'm more interested in what makes a _good_ program.
Here's my 2 cents worth:

COBOL lives on because it can, in the right hands, be self-documenting.
Even bad COBOL requires less "interpretation" than other languages
because the verbs themselves are a valuable aid to understanding the
intent of the original programmer. COBOL is a viable and very useful
communication medium quite independently of it's use as a programming
language. But what is it that we are trying to communicate? Most
programmers - let's face it - do not think in terms of algorithms or
procedure blocks or entry points or anything of that sort. They think in
terms of getting the job done. So how do the good ones differ from the
bad ones?

My own answer to that question is that a good programmer can break the
overall program down into a series of steps which make conceptual sense
and then write the code in such a way as to maintain a distinction
between those steps. Think of every lousy program you've ever seen: the
read section in a report program (say) contains MOVEs to set up break
indicators (which have nothing per se to do with I/O). The termination
section - which should close the files, display control totals and damn
little else - contains calls to report routines to print total lines
which were too tricky to put into the main "logic". And so on, and so on.
Further, the bad program has little whitespace to allow the code to
breathe. The thing ends up being completely messy because the coder cares
not one whit about the look of the code (on paper, Robert, because some
routines are easier to understand that way). Good programmers are like
good journalists, in a way: they use the language and the layout together
to get the point across.

Whether a particular coder uses prefixes or not is of no import. (I like
them, but that's long habit, and a liking for paper listings, speaking).
The amount of care he or she takes in "expressing" each task as neatly
and crisply as possible most certainly does. Does anyone else feel - as I
do - that not nearly enough attention is paid to this area? Many
programmers have been given a rudimentary vocabulary, and set to work;
what a pity that no-one thought to provide them with a decent grounding
in grammar, and a quick lesson in how to lay a page out attractively, to
go with that (COBOL) vocabulary.

Any more for any more?


Pete
