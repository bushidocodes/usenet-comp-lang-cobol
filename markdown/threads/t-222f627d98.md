```
Hi all,

It's my first post to the list so I hope that you all might be able to help me.
I've spent a couple of days trying to resolve this but have run out of ideas.

We currently have a couple of COBOL programs that call C modules to calculate
SHA and MD5 hash values for external files. The guy who wrote the COBOL and C
programs is long gone and now we are trying to make some changes to the hash
algorithms. I'm not a C programmer but I have been able to get the C modules to
compile cleanly but I can't get them to link properly. When I link the two
object modules together the linker can't resolve the symbolic name of the
called C module.

As an example, the COBOL program is named CALC_HASHES.COB and the C routines
are CALC_SHA.C and CALC_MD5.C. 

Within the CALC_HASHES.COB I have the following call:

   CALL "CALC_SHA"
         USING FILE_NAME, 
                    HASH_VALUE
         GIVING STATUS_CODE.

When I link the two modules together the linker tells me that it can't find
CALC_SHA.

I am working on Digital/Compac/HP OpenVMS running on an Alpha Server.

As a test I wrote a stripped down COBOL program that does nothing but call a C
routine. I wrote a C routine that simply writes the standard "Hello World" to
the screen. I can't get those two to work together either. This leads me to
believe that I'm not getting them linked together properly. I guess there could
also be a C compile parameter that I'm missing too.

Any thoughts?

Thanks in advance,
    Joanne
```

Thanks for the responses. 

My lack of any C training became obvious today. As I was working on the problem
I realized that I was compiling the modules with a C++ compiler. The compiler
does something called "name mangling" that wouldn't allow my COBOL program to
"see" the function names in the C object module. When I used the regular C
compiler I was able to link the modules together just fine.
