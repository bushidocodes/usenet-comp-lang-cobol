```
G'day everyone,
David here from Sydney Australia.
This is the first time I even came close to a Newsgroup, let alone publish
a message. I have to confess I haven't read a lot of your messages, thus I
don't even know if anyone else has mentioned what I am just about to tell
you.

Whilst checking my programs for any Year 2000 problems, I came across a
piece of code that, if left alone, will cause some problems in the
beginning of next century.

A small number of programs do an accept of a Date field from DATE, and the
format of this field is YYMMDD. If the program uses that field to compare
other dates against, and if those dates are before 2000, the results will
be wrong.

Here's an example:

ACCEPT WS-TODAYS-DATE FROM DATE


IF START-DATE = WS-TODAYS-DATE
(START-DATE is a parameter entered by the User)

(this would not cause a problem)

DISPLAY ""
DISPLAY "**" BOLD
DISPLAY "******* This is today's date" BOLD
DISPLAY "******* Extract files not yet available" BOLD
DISPLAY "******* Please try again" BOLD
DISPLAY "**" BOLD
DISPLAY ""
MOVE SPACES TO WS-START-DATE-X
ELSE
IF START-DATE > WS-TODAYS-DATE

(this would always be true if WS-TODAYS-DATE was back before the turn of
the
century)

DISPLAY ""
DISPLAY "**" BOLD
DISPLAY "******* This is a date in the future" BOLD
DISPLAY "******* Please try again" BOLD
DISPLAY "**" BOLD
DISPLAY ""
MOVE SPACES TO WS-START-DATE-X
END-IF
END-IF

I am sure this problem will be fixed before the end of this millenium, the
DATE will, in a near future, show a PIC of 9(8) (YYYYMMDD) instead of the
actual PIC 9(6) (YYMMDD). However, if the programs are not changed to
define the receiving field as Pic 9(8) instead of the current Pic 9(6), the
result will still be the same due to truncation.

Well guys, I'll see you all here in Sydney for the Olympics.
See you later,
David
```

```
IBM OS/VS COBOL is no longer marketed or supported by IBM. It had an
IBM extension that offered a "special register" named CURRENT-DATE, with
the format MM/DD/YY. It was typically used as "MOVE CURRENT-DATE TO
REPORT-HEADER".

I have no experience with Microfocus COBOL, but I presume they supported
the IBM extension for compatibility if you developed COBOL programs on a
PC.

IBM COBOL for MVS and VM includes the intrinsic FUNCTION CURRENT-DATE,
which returns the date in YYYYMMDD format, plus the time in HHMMSSth
format, plus the offset from UTC with leading sign, for a total of 21
bytes. I believe there is also a FUNCTION WHEN-COMPILED that returns
the compilation date/time in the same format.

IBM also offers "IGZEDT4" as a callable subprogram that returns the
current date in PIC 9(8) YYYYMMDD format (DISPLAY USAGE). The APAR
numbers are:

PN76666 - PTF UN83685 to add IGZEDT4 to the IBM VS COBOL II runtime
library
PN84080 - PTF UN92085 to add IGZEDT4 to the LE/MVS runtime library.

In CICS, there is a way to get a four-digit YEAR using the EXEC CICS
ASKTIME and FORMATTIME commands. These are language independent CICS
services. I believe this has been available since CICS release 2.x.

The following example was posted in bit.listserv.cics-l by David A.
Kindness in January of 1997:

01 YR2000-VARIABLES.
03 YR2000-ABSTIME PIC S9(15) COMP-3.
03 YR2000-YEAR PIC S9(8) COMP.
03 YR2000-CCYYDDD PIC 9(7).
03 FILLER REDEFINES YR2000-CCYYDDD.
05 YR2000-CCYY PIC 9(4).
05 YR2000-DDD PIC 9(3).
03 FILLER REDEFINES YR2000-CCYYDDD.
05 YR2000-CC PIC 9(2).
05 YR2000-YYDDD PIC 9(5).
03 YR2000-ODATE REDEFINES YR2000-CCYYDDD PIC 9(7).


EXEC CICS ASKTIME
ABSTIME(YR2000-ABSTIME)
END-EXEC
EXEC CICS FORMATTIME
ABSTIME(YR2000-ABSTIME)
YEAR(YR2000-YEAR)
END-EXEC
MOVE EIBDATE TO YR2000-YYDDD
MOVE YR2000-YEAR TO YR2000-CCYY.

As far as I know, the only standard way in COBOL to obtain current date
with four-digit year is to use the intrinsic FUNCTION CURRENT-DATE. If
your COBOL does not include the standardized intrinsic function calls,
it might be worthwhile to develop a library routine similar to IGZEDT4.
```
