Then you might not need a regular encryption algorithm.  You might just
use a combination of the following ideas I am pulling off the top of my
head:

1.  Have the password file 80 columns, filled with random characters,
with the password characters mixed up in with those characters.
2.  Rot-13 one character of the password, Rot-14 another, etc.  (don't
let anybody look for his own password to figure out your algorithm)
3.  Mix up where the characters of the password are - have a key show
where to start.
4.  If your RANDOM routine will always give the same value with a given
seed, incorporate this "feature".
5.  Each time a password is entered or changed, change the random filler
characters.

Sure these are breakable, but they are cheap, fast, and adequate for
your needs.

You'll also see "One-way hash", "cryptographic hash" and "message digest". This
is how Unix passwords are handled. A secure hash of the password is saved. When
the user attempts to login, their attempted password is hashed; if the hashes
match, it's successful.

The converting option does do an inspect for each character.
The full statement looks like:

                 INSPECT ENCRYPT-DATA CONVERTING
                      " !"#$%&'-./01234567890 <all ASCII characters> ..."
                      BY "zaQbYELmblkjhgfdsa8765432190 <all ASCII
characters> ...".

Each character in the first line is replaced by the
corresponding character in the second line. Each printable
character has to be in each quote once and only once. To
write the decrypt program, you just use cut and paste to switch
the two strings.

I really like the idea of getting EBCDIC involved on an ASCII platform.
Definately confuse the VB and "power user" crowd.  Another twist I used in my
routine was to use the last digit of system time to control the shifting and
scrambling, then hiding that value in the encrypted string so that the
decryption would know how to handle it -- so even a given string gets
encrypted differently every time. In our app, it is used for storing a
password that applications use to log on "under the covers" rather than
confirming a users entered value, so there is no problem with someone
entering a pswd of "AAAA" and analyzing the results.
