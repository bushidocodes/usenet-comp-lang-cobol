be better for programmers productivity than the supposed JAVA write once

solutions can now be automated. That is tools may be created to assist 
the programmer in the task of building the solution. For example no 
circuit designer today designs the layout of the tracks on a circuit 
board. The logic diagram is input and along with certain parameters, the 
circuit board layout programs design the layout by themselves. This 
means that this burdomsome task has been removed and the designer can 
concentrate on the logic and performance of the design - not the 
mechanics.

I agree that this is the wave of the future, but I think that for the

I just thought that I would "re-start" discussion of some of the parts of
the previous thread - indicating that I think some posters are speaking at
"cross purposes" (i.e. not NECESSARILY comparing apples to oranges).

It seems to me that there are a "variety" of different (but DEFINITELY
related) issues:

1) What can/should happen with the current draft Standard (FCD)?

2) What will happen with the COBOL language when (if?) this draft Standard
becomes approved (i.e. will vendors implement all or part of it - will
programmers use the "new" features that the implementors provide them)

3) What is the "best" way to keep the COBOL language "vibrant" (viable?) in
the future (what Standard?  what features?  what programming paradigms, etc)

4) Has there been a problem getting to "where we are" - and if so, is it
being addressed for the future?  (If not, that is "fine" too - but not
necessarily the common perception).

5) What will be the "general" data processing (information system, computer
programming, whatever) "paradigm" in the future (regardless of COBOL) - will
it differ in "mainframe" environments from "workstation environments"? Will
some languages (notably COBOL) be relegated to "maintenance of legacy
systems" while other languages (systems? facilities?) provide the "tools" of
the future?

  ***

I have some FAIRLY strong opinions on each of these - and do see that some
people have already commented on "much" of this.  However, I do think that
it helps "discussion" if we all know what each person it "talking about" and
not assume that
 - the person who is talking about which structures will be used in
maintaining a legacy mainframe COBOL business application
 - should get an answer talking about how to do new development in a
"real-time" networked environment (where there is little or no existing user
expectation)
