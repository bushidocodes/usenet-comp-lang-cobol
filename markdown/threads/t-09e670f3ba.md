With Fujitsu it is necessary to specify BINARY(BYTE) to get binary

Good show. Compile this separately, include it in the link as an OBJ and
you're set. (We put all these subprograms in one SUBS.LIB file and include
the SUBS.LIB file in the project.) But, if you think about it, REC-MIN and
REC-MAX can't be right. You've defined:

REC-MAX   PIC 99 COMP.

This is a 2-byte COMP-5 field. In my test case for 1000 records the field
contains 02 BC which is 1000 decimal if considered as COMP-4. The 02 BC if
viewed as COMP-5 would be 48,000+.

You probably have the default for COMP set to COMP-4.

But it doesn't matter, because a two-byte field can only contain up to
32,000 (64,000 if you ignore negatives). Surely an ISAM file can contain
more records than that. Therefore REC-MAX cannot be the maximum number of
records in the file.

Try writing a file of 200,000 records and see what's there. And, as another
poster suggested, after writing the 200,000, delete 100,000 to discover the
new values.

N.B.: ERRO should be defined as PIC X(2). The IO system CAN return
alphabetic characters ('9E' for example).
