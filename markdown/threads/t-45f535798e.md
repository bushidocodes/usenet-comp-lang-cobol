There have been a bunch of notes on this topic and I wanted to comment on,
reply to, and confirm several issues. Sorry if some of this duplicates
information that has already been provided by others.

1) The current draft of the next COBOL Standard provides a method of
defining user defined functions. (This expands the ability provided in an
Amendment to the '85 Standard that provided "Intrinsic Functions").

A) The most optimistic estimate is that this draft will become an
official Standard in the next millenium (2000 or later).
B) Although this facility is well defined in the current draft, like the
rest of the draft it might well change between now and being approved
(including the REMOTE possibility of being removed).
C) Unrelated to being user-defined or intrinsic, the current draft
provides syntax that allows you to drop the currently required word FUNCTION
when using functions - assuming that you declare this in advance and never
use the same name as a user-defined word.

2) The normal difference between the terms "procedure" and "function" was
pointed out by one responder when they indicated that a function "returns a
value" (like the intrinsic functions do in the current COBOL Standard) while
a procedure passes values to and from in parameters. (Functions can
normally get input in parameters - but they don't return results in them).
UNFORTUNATELY (in my opinion - but not in the opinion of the current COBOL
technical committees), the present design for the next COBOL Standard allows
"unexpected side effects" from using user-defined functions, i.e. a
parameter passed to a user-defined function may end up changed by the
function's logic - as well as getting a resultant value returned from the
function. (The Standard does support the creation of user-defined functions
that require parameters passed by CONTENT or by VALUE - but this is not
required).

3) It is true that Micro Focus currently has extensions that allow the
creation of user-defined functions. It is also true that IBM COBOL for
OS/390 and VM does *not* include syntax for defining user-defined functions.
However, both compilers also provide methods for defining parameters as
being passed OR received "BY VALUE" as well as by content. Depending on why
you want to create a user-defined function, that might give you the
capability that you are looking for. (and if you don't know the difference
between BY CONTENT and BY VALUE - then this probably isn't what you are
looking for).

Obviously, this entire note is passed on the fact that the current Standard
(with all of its current ammendments) doe NOT support the creation of
user-defined functions - which was the original request./question. However,
in my opinion, 99.99% of what you might want to do with a user-defined
functiuon can be done with a currently standard conforming CALL statement
followed by a MOVE into a temprary data item - that is then used where you
want to use the function.

Judson McClendon wrote:
› 
› William M. Klein  wrote:
…[4 more quoted lines elided]…
› Please explain the difference between 'BY CONTENT' and 'BY VALUE'.

BY CONTENT: the calling program makes an unnamed copy of the
parameter and passes its address to the called program. The called
program doesn't know or care whether it was passed BY REFERENCE
(the default) or BY CONTENT. If it changes the value of a parameter
passed BY CONTENT, it changes only the value of the copy, not the
value of the original. Thus the calling program may choose to pass
BY CONTENT in order to protect the value of the original.

BY VALUE: the calling program passes a copy of the parameter, not
just the address, and not just the address of a copy. The practical
effect is similar to a call BY CONTENT except that the called
program has to have BY VALUE in the USING clause, because the
compiled code may have to access the parameter differently. The
detailed mechanisms are up to the compiler writers, and might be
identical for BY CONTENT and BY VALUE.

In a sense, BY VALUE is the reverse of a RETURNING clause. Each
allows information to flow in only one direction. BY VALUE lets
it flow down, and RETURNING lets it flow back up.

I've never used BY VALUE myself because VS COBOL II doesn't
support it, so I may have garbled the details a bit, but I think
that's the general picture.

In article <01bd1845$fe1a7a00$f2d945cf@juddesk>,
"Judson McClendon" wrote:
›
› Please explain the difference between 'BY CONTENT' and 'BY VALUE'.

For historical reasons, a CoBOL program _receives_ all of its parameters
BY REFERENCE. That makes the calling sequence a simple list of
addresses, and in fact the CoBOL CALL statement was abstracted from the
OS CALL macro which uses R1 to point to a list of addresses. Each
address takes up the same amount of space, whether the argument is a
single byte or a humongous 01-level.

When you pass an argument BY REFERENCE, the called routine can change the
value of the argument. You pass an argument BY CONTENT to protect your
data from the called program. The compiler makes a hidden copy of the
parameter and passes the _address_ of the hidden copy to the subroutine.

Note that the subroutine can not tell whether a parameter has been passed
to it BY REFERENCE or BY CONTENT. Or, looking at it from another angle,
the subroutine does not need to know, and therefore you don't have to
specify it in the USING.

On the other hand, passing a parameter BY VALUE means the parameter is
passed _instead_ of the address. Obviously, the called program had
better know which parameters are addresses and which parameters are
values, especially when a parameter is larger than an address. Some
vendors, e.g. Micro Focus, do let you receive BY VALUE when the parameter
is not larger than an address, but this is not portable.

In 1985 standard COBOL the concept of a user defined function does not exist.
The Intrinsic Function addendum to the 1985 standard defined 40-odd builtins,
but not a user defined function in the FORTRAN or C sense.

capability, but these will vary by vendor and are not portable - thus likely
not to be in "standard COBOL" textbooks.
