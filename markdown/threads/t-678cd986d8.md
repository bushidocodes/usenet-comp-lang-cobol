Consider:

01  MYTABLE.
     05  MYLIST OCCURS 50 (table stuff goes here).
          10  MYABBR     PIC X(2).
          10  MYNAME     PIC X(25).

That you've loaded with data.

The code that you might use to find a name associated with an
abbreviation:

PERFORM VARYING CTR FROM 1 BY 1 UNTIL CTR > 50
     IF TEST = MYABBR(CTR)
         MOVE MYNAME(CTR) TO MYRESULT
         END-IF
     END-PERFORM

Now, if you add "INDEXED BY CTR" in the area above marked (table stuff
goes here) you can replace the above code with:

SEARCH MYTABLE VARYING CTR
     WHEN TEST = MYABBR(CTR)
          MOVE MYNAME(CTR) TO MYRESULT
          END-SEARCH

and the above code sequentially examines every entry in the table,
just like the longhand way above.

Now here's the really neat part: If you can guarantee the table is in
order, replace (table stuff goes here) with "ASCENDING KEY IS MYABBR
INDEXED BY CTR"

Then you can use:

SEARCH ALL MYTABLE
     WHEN TEST = MYABBR(CTR)
     etc.

But the search is a binary search instead of sequential and is, for
large tables, very much faster.

Be sure to populate the array with HIGH-VALUES before loading your
data.

It is left as an exercise for the reader to fill in the exotic stuff
(AT END, AND, index item, DESCENDING, etc.)

Hi Deek,

Maybe this will help:

Suppose you have a bookcase with 5 shelves each capable of holding 20
books of equal size. Each book contains 35 chapters.

If I asked you to read chapter 23 of the 15th book on the 3rd shelf,
how would you go about it?

After 3 weeks of this you complained that the requests were too
wordy, taking too much time. So we defined the bookcase as follows:

                BKCASE
                   SHELVES        OCCURS  5
                      BOOKS       OCCURS 20
                        CHAPTERS  OCCURS 35

And we came up with this notation for me to communicate to you what
chapter in a book I wanted:

         BKCASE(3,15,23)

We just made a "table" out of a bookcase and designed a way to get at
any element in the table/bookcase.

You can test your understanding of the concept by answering these
questions:

What if I had 3 bookcases?
What would I have to do to get at a specific word in a chapter?

Please note that in COBOL all the elements within an occurs group must
be the same # of bytes, whereas the words on a line of a book or the
line on a page do not.

Hope this helps you get started anyway. Keep on truckin. Jack

Sorry to keep bringing up the Olden Days, but, in the Olden Days,
IBM and the other manufacturers provided their customers with
things like record-layout and memory-layout forms. These were
nothing more than large pages with little boxes each of which
represented a single memory location.

The reason I mention this is because it may make things easier for you
to be able to visualize the contents of working-storage and record
layouts.

If you were to use one of these forms to lay out a state table,
it would look like this (without the boxes which I cannot reproduce
here)ALAKAZARASCACZCOCTDEDCFLGAGUHI. . . Where each state is a
two-character field and the whole is redefined as a 52(?)-field
table. It might also contain the abbreviations for places like
Guam and the Canadian provinces.

To understand a table think of rows and colums.

Draw a diagram showing the rows and columns and it will be easier to
understand.  The problem is that columns are inside of rows when you
set up the table.  In actuality all the data is one byte after another.
You are defining a table by saying inside of a certain variable, there
is a field which occurs a certain number of times, and if there is a
second level inside of that there is a field that occurs a certain
nunber of times inside that field. Example:

row1(col1,col2), row2(col1,col2)

or:

r1c1,r1c2,r1c3,r2c1,r2c2,r2c3

one way to test this is to build a table and put a default value in
each element and that way you know if you got it right.
