Yes, the Standard allows for USE statements on specific files.  In the 2002
Standard, there can even be separate declaratives for different I-O statuses.

There is (as far as I know) really almost nothing that can be done in an ERROR
declarative that can't be done in a file-status checking routine.  One possible
OBSCURE exception is that an Error Declarative may "catch" an I/O error in files
used in a SORT/MERGE USING/GIVING statement - while there is no real way to do
this with a file status check (unless you CHANGE the code to have INPUT/OUTPUT
procedures).  On the other hand, some compilers don't (always) do this correctly
anyway.  (IBM, for example, doesn't "go to" the USE declaratives when the
FASTSRT compiler option is in effect - which is why they document that compiler
option as "non-Standard")

That's the problem with DECLARATIVES  - it was perceived to be an 'error 
catcher' but is somewhat clunky. Bill is a great enthusiast for 
DECLARATIVES  :-)

You've indicated you already have a lot of code using DECLARATIVES - so 
you probably don't want to move away from that concept because it means 
a lot of re-coding. Years ago I was using the now non-supported RM/COBOL 
V 2 in DOS.  I had that problem with identifying files in the 
DECLARATIVES. Here's a snippet using the
DECLARATIVES approach. - this is a copyfile included in each program 
accessing files.

000130

000180

000220                                                             
000230 I-O-START.                                                  
000240

000260                                                             
000270 INPUT-ERROR SECTION.                                        
000280

000300

000320

000340

000380

000510                                                             
000520*-------------------------- END NDERROR/CBL ------------------

Now above does require some coding in the Business Logic program - each 
time you are going to use a file VERB you have to place the appropriate 
file-number in the variable F.  The called NDE000  has a list of 
RM/COBOL file-status error messages and used an old DOS screen painter 
to display the equivalent of a Windows messagebox, using WS-SUNDRY-1 to 
display the filename from a table. I might add, back then, if I hit an 
error, the application came 'ungracefully' to a halt with a STOP RUN !

Now there is an alternative, still using DECLARATIVES. - but again 
involves you in re-coding.  Assume files CUSTOMER-FILE (ISAM), 
CUSTOMER-TRANSACTIONS (Sequential), etc... You create a program for each 
separate file, from the appropriate filetype template,  which is CALLed 
by your Business Logic program as required. That would allow you to have 
a generalized copyfile in each program which contains a standard set of 
DECLARATIVES. - your errors are automatically identified to the specific 
file. The above concept is using the big-time word from OO - R-E-U-S-E.

As background I'm using OO and have five file classes ISAM (with a fixed 
PrimeKey Length), ISAMAlt1, (with a fixed  PrimeKey Length and a fixed 
Alternate Key1 Length), Relative, Sequential and Line Sequential. Each 
of these is a Template to handle any file - and is controlled by passing 
the External filename and record length for the appropriate file.

Now off the top of my head you could emulate that approach with 
Procedural programs. You create five file templates as above. You want 
to use an ISAM for CUSTOMER-FILE; take a copy of your ISAM template and 
call it "Customer-File"-
include as hard-coded the filename and  record-length with the following :-

       FILE-CONTROL.                                
                                                    
            Select               Data-File          
            assign               Data-filename      
            organization         indexed            
            access               dynamic            
            record key           Data-PrimeKey      
            file status          ws-FileStatus.     
                                                    
       FILE SECTION.                                  
                                                      
       FD  Data-File                                  
           record varying from 20 to 1500 characters  
              depending on ws-RecordSize.             
       01  Data-record.                               
           05 Data-PrimeKey       pic x(20).          
           05 Data-info           pic x(1480).        

To give you some ideas about file-handling, albeit this is OO, go to 
cobolreport.com and look at Gene Webb's OO examples. He has one covering 
files, does appear a little complex, but nevertheless is very comprehensive.

If you should wish to pursue this 'separate file approach' e-mail me and 
I can send you the layout of my file classes - which will give you more 
specific background. Meanwhile, I should add, I've switched to MS Access 
and SQL - so file status codes ain't no bloody good :-)
