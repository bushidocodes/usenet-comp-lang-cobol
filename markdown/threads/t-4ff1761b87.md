```
The question: does the DEPENDING restriction on INITIALIZE apply only 
to unsubscripted references or to all references? (In ANSI85 COBOL.)

Example:

 identification division.
 program-id. init-sub.
 data division.
 working-storage section.
 1 the-sub     pic 99.
 1 the-table.
   2 the-item pic x occurs 0 to 30 depending on the-sub.
 procedure division.
 begin-prog.
     initialize the-item of the-table (the-sub)
     initialize the-item
     .

Now, I understand that the second INITIALIZE is invalid, per the 
standard at 6.17.3(4): "The description of the data item referenced by 
identifier-1 or any items subordinate to identifier-1 may not contain 
the DEPENDING phrase of the OCCRUS clause."

My question is about the first INITIALIZE. The Unisys NX COBOL85 
compiler rejects it, but they have had a whole nest of bugs in 
INITIALIZE (some of which I reported, some others reported), so I don't 
trust that as an authority.

The problem is in the part that says "the data item referenced by 
identifier-1" when identifier-1 is not a simple data item reference. 
When I go back to section 4.3.8.4, the definition of an identifier, an 
identifier includes a leading data-name, optional qualification, 
optional subscript(s), and optional reference modification. This can 
involve numerous data-names. Which is "the data-name referenced by 
identifier-1"?

And if indeed this is prohibited, why? I understand why the DEPENDING 
restriction, to make compilation simpler. (I don't see any ambiguity 
that results from DEPENDING, just complications.) But this reason would 
not seem to apply when the data item with DEPENDING is fully 
subscripted. Is this just an oversight, or is there a good reason?

Edward Reid
```

```
As I read the standard, the first "initialize" could be disallowed for the
very reason you cite -- VI.6.17.3, THE INITIALIZE STATEMENT, SR4, which
disallows "initialize" against a data item that either has, or has
subordinate to it, OCCURS ... DEPENDING ON (hereinafter ODO).

However, coding the data description as
 1  the-table.
   2  filler occurs 0 to 30 depending on the-sub.
      3  the-item pic x.

avoids this ANSI-85 restriction against "initialize the-item of the-table
(the-sub)".

The fact that the circumvention of this restriction is so trivial and so
arbitrary leads me to suspect that the restriction may be unnecessary.
Since this restriction has been explicitly removed from the international
standard now in effect (ISO/IEC 1989:2002), I suspect the authors of that
new standard agreed.  (By the way, the new standard also allows "initialize
the-table").  Because the new standard allows it, and because I can't think
of a way in which the execution-time behavior would differ from the above
substitution in the data declaration, I can't seem to get terribly excited
about a compiler's failure to issue a syntax error on this subject.

However, I would not expect "initialize the-item." (without subscripts) to
be accepted as standard ANSI syntax.

I think you've misunderstood the intent of the standard as to the
optionality of subscripts when you paraphrase IV.4.3.8.4, Identifier, as "an
identifier includes a leading data-name, optional qualification, optional
subscript(s), and optional reference modification. This can involve numerous
data-names. ... ".  What it actually *says* is "An identifier is a term used
to reflect a data name that, if not unique in a program, must be followed by
a syntactically correct combination of qualifiers, subscripts, or reference
modifiers necessary for uniqueness of reference."

Unless the context in which an identifier referencing a table entry is used
explicitly allows the reference to such a table entry without explicit
subscripts, they are required.  And since "initialize" doesn't appear in the
list of such exceptions in IV.4.3.8.2, Subscripting, I would surmise that
when a table element is the target of "initialize" that table element must
be subscripted.

By the way, I can't reproduce your assertion that MCP/AS COBOL85 rejects the
first "initialize".   I get the same results as I get with my source-level
change, and that appears to match what Jerry Mouse reported for Fujitsu 6
for the first "initialize".  But I'm pretty sure your second statement, an
unsubscripted "initialize the-item",  would have to be an extension (flagged
or not) to ANSI COBOL, as I see it, even applying the 2002 standard.

    -Chuck Stevens
```

```
You seem to have missed the main point of Chuck's note.  In the '85 Standard
(but *not* the 2002 Standard) it is "illegal" to do an INITIALIZE against a
data-item defined with an OCCURS ... DEPENDING ON clause.  In your example
listed below you try and do an INITIALIZE against the field

  "the-item"

which is defined with an OCCURS DEPENDING clause.  The correction (according
to what Chuck indicated) is to change your sample code to:

 000100 identification division.
 000200 program-id. init-sub.
 000300 data division.
 000400 working-storage section.
 000600 1 the-sub pic 99.
 000700 1 the-table.
 000800   2 the-item occurs 0 to 30 depending on the-sub.
             *  the next line is new
                  3 Item-with-no-ODO-clause  Pic X.
 000900 procedure division.
 001000 begin-prog.
             * you probably want to set the-sub to some value here to vaoid
bad run-time results
 001100     initialize the-item of Item-with-no-ODO-clause  (the-sub)
 001200     .

Although this "functionally" does what your sample did, it does it in a way
that conforms to the '85 Standard rule.
```

```
No, I don't think that's true.  You should be able to INITIALIZE an item
that's *subsidiary* to an item with an OCCURS clause whether or not it's an
ODO OCCURS (my suggested revision does exactly that, in fact, in what both
Bill and I have stated is a standards-conforming manner).  But if the target
of INITIALIZE is subsidiary to one with an OCCURS, that target's going to
need subscripts.

According to the standard, you can also INITIALIZE a GROUP that contains an
item that has OCCURS; it is however illegal to INITIALIZE a GROUP that
contains an item that has ODO.  COBOL2002 lifts this latter restriction.
```
