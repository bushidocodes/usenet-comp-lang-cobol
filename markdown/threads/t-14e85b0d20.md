To do this right, then, do not used "close system" COBOL. If you use a current
(LE-enabled) COBOL compiler, such as COBOL for MVS & VM, COBOL for OS/390 & VM,
or Enterprise COBOL for z/OS & OS/390, you can code pragmaless calls between
COBOL and C.

In the COBOL program...

procedure division.

       .
       .
       .
       call 'subrtn' using arg_list

NOTE: depending on what the C progam is expecting,  you may need to pass some
or all of the arguments with a BY VALUE phrase



In the C program (which must not be a main, right?)

#include (leawi.h)
.
.
.
void SUBRTN (arg_list)
{
.
.  code for subroutine
.
}



As far as "scripts" (the UNIX way of talking, not particuarly common term in
OS/390), a standard C compile and link job and a standard COBOL compile and
link job should be sufficient.


In the OS/390 world, we don't generally keep object code (talking about "c.obj"
is unheard of); instead, we almost always compile and link into an executable
(load module or program object), since executables can always be linked
together statically or dynamically in a variety of ways. So make sure you are
doing a compile _and link_ of your C program. The link step will put the
executable into the PDS pointed at by the DD statement named SYSLMOD (that is,
//SYSLMOD  DD  DISP=SHR,DSN=executables_library).


I do not know what is in your IGYWCL JCL, but the link step (probably //LKED
EXEC PGM=IEWL or HEWL) needs to include a DD statement for SYSLIB; this DD
statement probably has several data sets concatenated under it, including your
LE linkedit library (a name something like CEE.SCEELKED or SYS1.SCEELKED) and
the library into which you compiled and linked your C subroutine.


Hope this helps.

OK, here is some JCL to do a compile and link of your C routine (make changes
wherever you see lowercase letters and / or the little arrows '-->'):

//------- valid JOB statement here -----------
//C       EXEC PGM=CBCDRVR
//        PARM='FL(I),SO,LO,LANG(EXTENDED),NOLONGNAME,XREF'
//STEPLIB  DD  DISP=SHR,DSN=library_where_compiler_is  --> can omit if in
LINKIST
//               DD  DISP=SHR,DSN=LE_run_time_library           --> often
SYS1.SCEERUN
//SYSLIB   DD  DISP=SHR,DSN=HEADER_FILES                --> OFTEN SYS1.SCEEH.H
//SYSPRINT DD  SYSOUT=*
//SYSCPRT  DD  SYSOUT=*
//SYSUT1    DD  UNIT=SYSALLDA,SPACE=(CYL,(1,1))
//SYSUT4    DD  UNIT=SYSALLDA,SPACE=(CYL,(1,1))
//SYSUT7    DD  UNIT=SYSALLDA,SPACE=(CYL,(1,1))
//SYSUT8    DD  UNIT=SYSALLDA,SPACE=(CYL,(1,1))
//SYSUT9    DD  UNIT=SYSALLDA,SPACE=(CYL,(1,1))
//SYSUT10  DD  UNIT=SYSALLDA,SPACE=(CYL,(1,1))
//SYSLIN     DD  DSN=&&SYSLIN,UNIT=SYSALLDA,DISP=(MOD,PASS),
//             SPACE=(TRK,(3,3))
//SYSIN      DD  DISP=SHR,
//         DSN=your_source_library(SUBRTN)                          --> program
to compile
//*
//         IF  C.RC < 5 THEN
//LKED    EXEC PGM=HEWL,PARM='MAP,LIST'
//SYSLIN    DD  DSN=&&SYSLIN,DISP=(OLD,DELETE)

you need to
//                DD  DISP=SHR,DSN=subroutine_library         --> pick up your
subroutines
//                DD  DISP=SHR,DSN=LE_link_library             --> often
SYS1.SCEELKED
//                DD  DISP=SHR,DSN=LE_link_library             --> often

Shyam,

Well, there are still a lot of unknowns for me. First, regarding your question:

>Now is there
>any option for forcefully guide the control to COBOL program similar
>to ENTRY ....... for c programs?
>I doubt because c entry point is decided at compile time whereas cobol

I don't understand the question at all.

The entry point for a C program is determined at compile and and bind time. The
entry point for a COBOL program is also determined at compile and bind time.

So let's look at the structure of your programs. I don't have the original post
here, but as I recall, you want to have a COBOL main call a C subroutine.

First, the C subroutine must not have a 'main' procedure in it, of course. The
general structure will be this:

#include statements

a prototype for the subroutine itself, to describe the format of parameters
being passed, maybe something like this:

void SUBRTN1 (char in_char[],
                        short *binary_half,
                        .
                        .
                        .
                                           )

then the body of the function...

{
 .
 .
 .
}

This program should be compiled and bound using JCL as I suggested.


Second, the COBOL main will be like any COBOL program, and it invokes the C
subroutine with a CALL, something like this:

      call 'SUBRTN1' USING char-string, 
                                        binary-half,
                                        .
                                        .
                                        .


This program should be compiled and bound using the other JCL I suggested.

If you are still having problems, perhaps you could post the important excerpts
from your programs, the JCL you are using, and any binder control statements.
If this is very long, consider emailing it to me off-list (see my signature for
the preferred email address).
