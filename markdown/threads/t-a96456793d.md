I would like to add the ISBN numbers to a file where 350 titles of books are
stored. To write to a new file you have to use to open output file along
with the write record instruction. To read a file you have to use the open
input file along with the read file next record, but what about to add the

> I would like to add the ISBN numbers to a file where 350 titles of books are

> but where can I write the ISBN number to and how.

I think what your are telling us is the following :-

    01 MyBookRecord.
        05 BookName            pic x(...)
        05 BookAuthor           pic x(...
        05 BookPublisher       pic x(...
        05 Book etc......

Now you've decided you want to add the ISBN. There really are only two
possibilities :-

(a) Does your BookFile have a filler in the record currently not used that could
fit
    the ISBN ?

    if yes - make a copy of your existing record, and taking above :-

    01 MyBookRecord.
        05 BookName            pic x(...)
        05 BookAuthor           pic x(...
        05 BookkPublisher      pic x(...
        05 Book etc......
        05 filler                        pic x(30).

You now take the filler field and split it for your new file record :-

        05 Book-ISBN           pic x(15).    *> the 'fifteen" is a guess
        05 filler                        pic x(15).

Note : we are NOT changing the size of the record.

As an ABSOLUTE minimum you  Open I/O using your new record format and INITIALIZE
the field Book-ISBN to spaces. Read Next record, Initialize Book-ISBN, REWRITE
record. Subsequently you can update the file with ISBNs when they become
available. (Possibly easiest way to do this would be to generate an
Ascii-Delimited/CSV that your users can edit manually to add ISBNs. Then you
take their finished list and re-input to create a new OUTPUT file containing the
ISBNs).

(b) If you haven't got a filler area - you are screwed. So still take a copy of
your original Book- record1 and create a copy file with Book-record2 which has
the additional field Book-ISBN. (Here we have INCREASED the size of the record).
Your conversion program - Open INPUT Book-record1 file, and use Read Next
record. Open as OUTPUT the file with Book-record2. On each Read Next move the
data from Book-record1 to Book-Record2 - ensure you initialize that Book-ISBN
field, now WRITE to file book-record2.

Having got your NEW file with Book-records2 you ignore the Book-Record1 file
altogether. Again suggest you create a CSV to get ISBNs into your new file.

Some may suggest slight changes to the above routines, but it will get you
there.

Jimmy, Calgary AB

If you're working on an IBM mainframe, you can only open a file for I-O and
rewrite a record if the file is indexed (VSAM).  If it's a QSAM file, you'll
have to open the original file for input, read the record into working storage,
build your new record, and write the new record out to a new file.

I think the same would be true in MicroFocus CoBOL (but don't quote me.)

Hope this helps.

I cannot believe how common this MIS-conception is among IBM mainframe COBOL
programmers.  Ignoring for the moment ESDS files (which ARE "sequential"),
it is ENTIRELY possible to OPEN I-O QSAM files.  Obviously, those who think
this is not possible have not read the manuals and/or ever tried this.
(Instead they "think" about what should be logical).  It is REQUIRED that
any ANSI/ISO conforming implementation support OPEN I-O of ACCESS and
ORGANIZATION SEQUENTIAL files (if they support "direct access" type storage
devices).  And the IBM mainframe programmers are conforming in this respect.
If you "doubt" that this is well documented, see,

 http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igypg205/1.8.2?

which says in part (and pretty clearly),

"You can also open QSAM files on direct access storage devices as I-O."
