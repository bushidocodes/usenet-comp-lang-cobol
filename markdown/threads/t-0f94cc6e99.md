I think we have this resolved, although I haven't yet heard back from
Charlie.  A large percentage of our users (dating back to 1999) run
Micro Focus compilers, so it's just a getting-started problem.

Just for some background, the only compiler-specific issue involved
with GUI ScreenIO is the CALL interface to our runtime DLL.  We
created it so that it can be called using the standard Windows system
conventions.

Rather than provide a separate object library for every COBOL
compiler, we chose to provide the source code for an intermediate
COBOL module that is coded so as to call our runtime DLL using the
conventions it requires.  This amounts to all of five or six lines of
code.  You just compile the version for your own compiler, and you're
off to the races.  

The usual problem is that a new user made an error decommenting the
correct code for his compiler in the interface module, which is
generally found/fixed in short order.  

The other common problem is that the integrated development needs to
be told about our object library so that it does the LINK correctly
(just as would be required for any other package), and it is sometimes
difficult to find where these settings must be made.  IDEs are great
once they're set up, but I occasionally feel they hide a lot of stuff
that developers really should be aware of...

And while we're at it, I seem to recall that you took a look at GS
when we first released it in '99, have you looked at the latest?

Charlie,

Your LINK is fine, as is the GS.DLL that you have created.

GS.LIB is just an import library that exposes the entry points you
need in our runtime, GS32.DLL.  If the LINK had failed, you would have
gotten a linker error complaining of unresolved externals.  So,
there's nothing wrong there.

If the Animator is complaining that it can't find GS32.DLL, your
GS.DLL has been built correctly, since the system is indeed attempting
to load GS32.DLL, as it should. 

Therefore:

1.  Verify that GS32.DLL is in the same subdirectory as your
executable (or is in the PATH).  

2.  GUI ScreenIO also requires the Microsoft module GDIPlus.DLL, which
we distribute with the product.  In some cases, we've seen Windows
complain that it couldn't load GS32.DLL when, in fact, it was
GDIPlus.DLL that was missing.  So, verify that GDIPlus.DLL is also in
the same subdirectory as your executable.

LITLINK just tells Micro Focus to generate the calls to the GUI
ScreenIO runtime entriypoints GS32 and GSWINAPI as static, instead of
dynamic calls.  We discuss the advantages and disadvantages of static
and dynamic calls in the "Technical Tips" portion of the manual.

Guess I was a bit opaque there...  ;-)

LITLINK causes the compiler to generate the code as though the CALLed
module is linked statically.

The library GS32.LIB that you include with your LINK command isn't a
regular object library, it's an import library.  Import libraries
allow you (yes, anyone can do this) to expose multiple entry points in
a DLL.  This means that you can distribute a single DLL instead of a
whole bunch of them, which keeps things simple.  

What _really_ happens is that even though your generated code thinks
it's calling GS32 and GSWINAPI statically, in fact the import library
will link to GS32.DLL dynamically when you run your application.
That's how most of Windows actually works.   

The magic of the import library is that it will cause the system to
know, at runtime, that GSWINAPI is found in the module GS32.DLL, and
if it's not loaded, will load it.  If you simply tried to call
GSWINAPI dynamically, the compiler's runtime will go looking for
GSWINAPI.DLL (which it won't find), instead of finding the alternate
entry point in GS32.DLL.

Hope this helps a bit more..
