Hi,

As Steve correctly points out the issue is OS/390 or z/OS Allocation and
the tie with the DDNAME. By the time the program starts the Operating
System has allocated the DDNAME and Volume combination ready for OPEN,
unless the DEFER option is specified on the UNIT parameter, which delays
allocation until OPEN time.

Before going down the Dynamic Allocation route, it might be worth trying
the DEFER option with the existing CLOSE REEL options in the program.

I have a concern about dynamic allocation and retaining the same DDNAME,
as in standard JCL if two DD statements have the same name one is
ignored. I know the requirement was to have the same DDNAME, but to
ensure dynamic allocation works it might be better to change it for each
file. (Being dynamic the DDNAME will be transparent to the JCL.)

The topic of dynamic allocation from COBOL programs has been the subject
of recent discussion in the IBM-MAIN list, so if you have access a trawl
of the archives should provide some help. The alternative to call an
assembler Rouitine.

Hope all the above helps in some way - Terry

Hi Mircea,

Interesting problem.  I wouldn't call this a solution. It's more a
path to a possible solution.
Since you're using the same DSN and DD, matching the data with the
tape for logging is problematic. You may resolve that by reading the
tape label after (or before) the data is written, then creating a
report listing the tape serial #s and the associated data description.

You may be able to use RETPD=2 (or whatever) to keep the operator from
mistakenly remounting the previous tape

The JCL for the data and the label would look something like this:

//DATADD    DD  DSN=TEST.DATA,UNIT=(TAPE,,DEFER),DISP=(,KEEP,KEEP),
//                    VOL=(,,RETAIN),DCB=(.....),LABEL=(1,SL,RETPD=n)
//LABELDD  DD  DSN=TEST.LABEL,UNIT=AFF,DISP=(OLD,KEEP,KEEP),
//                    VOL=REF=DATADD,DCB=(LRECL=80.....),LABEL=(1,BLP)

You'd need  an FD in your pgm for each of the DDs above. The way I've
written the JCL, you'd have to process the DATADD 1st, CLOSE it, then
OPEN and READ the LABELDD looking for the serial# in the label recs,
then CLOSE that and write a report line. Then loop back and do it all
again until you've processed all your input.

If you want to process the label before the data, some mods to the JCL
are required.
    
You should check with your system guys to verify  what you need to do
to process an SL tape as unlabelled and to verify whether or not
they're using 3rd party tape mgmt software and what impact that may
have on your plans.

It's a little kludgey, but it might work.

Regards, Jack.  
P.S. JIC, check the JCL syntax. I coded it from memory. :-)
