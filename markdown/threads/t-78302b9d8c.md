JAVA definitions to clarify the earlier analogy:
An Interface is a special class that specifies a set of behaviors
  these behaviors are like a list of methods 
   each method with a specific signature or parameter list
  methods in the interface have no source code or guts
  the guts of each method comes when you write an "implementation" 
  any implementation must contain each method listed in 
   the original interface

Java , as a good OO often uses objects in the P-List of methods

example: 
 0001 method caller ( using  Implementation_of_the_Interface IotI )

 here, i meant that the arg="IotI" in the signature specifies that:
    the argument must be an Implementation of a particular interface
    that it must implement 3 methods (M1, M2, and M3) the signatures 
      of which are specified by the orig. interface
    So, "Implementation_of_the_Interface" in the signature above
       means "an Object goes here"     

 Callbacks come into play when the original code for the method
"caller"
   which has an Interface class as an argument, then follows with code
   in the body of the method that mentions or uses methods from 
   the list of methods specified by the interface. 

> ref1 = aNewExtension.newInstance();  
> caller(ref1);  // this invokes the method above at "0001 method
  
and the body of caller has:

   IotI.M1()  |
   IotI.M2()  |  any new implementation of these methods  
   IotI.M3()  |  is what gets called back when you use an instance of 
                 the new class in place of 'ref1' as in :
                       caller(ref1)  above

The solution is largely dependent upon what class support IBM provides - no details but I get the impression it is fairly *lean". Prototyping or whatever, you do require basic building bricks, just like Leggo. As it stands
the J4 Draft Standard hasn't yet addressed these 'building bricks' - so Fujitsu and  Micro Focus have currently provided their own. (Probably same applies with Hitachi who based their 'Standard Classes' on Smalltalk, just
like Micro Focus).

As an example, I can build a Dictionary (Key and Data) - a fancy name for an ISAM list, which I think Java calls a MAP. Currently the Dictionary can be made up of :-

Key        Data
----    --------
String    String                        Note : String = "ofValues" in example below
String    Object Reference
O/Ref    String
O/Ref    Object Reference

An example of constructing a Dictionary :-

*>---------------------------------------------------------------
Method-id. "initialize-dictionary".
*>------------------------------------------------------------
Local-storage section.
01 ls-AssocTemplate                  object reference.
01 ls-DataTemplate                   object reference.
01 ls-KeyTemplate                    object reference.
01 ls-KeyLength                      pic x(4) comp-5.

Procedure division.

 if    ClientRecType
       move 3 to ls-KeyLength

 else  move 2 to ls-KeyLength
 End-if

 invoke CobolPicx "newclass"   using     ls-KeyLength
                               returning ls-KeyTemplate
 invoke CobolPicx "newclass"   using     ws-NameLength
                               returning ls-DataTemplate
 invoke Association "newclass" using     ls-KeyTemplate
                                         ls-DataTemplate
                               returning ls-AssocTemplate
 invoke Dictionary "ofValues"  using     ls-AssocTemplate
                                         ws-NameLength
                               returning
                                 os-Collection(ws-CurrentRectype)
End Method "initialize-dictionary".
*>------------------------------------------------------------

The important thing to note is that I'm using support classes CobolPicX and Association, (basic building bricks) to create the structure of the Dictionary. Those two 'building classes' are further assisted by additional
Micro Focus support classes.

You could take two approaches to the above, sub-class, or given the Net Express source, rewrite your own CobolPicX2 and Association2. Certainly, there's nothing to stop you writing classes with 'empty methods'.
