```
We're using generators to produce cobol source code and are now in a 
process of switching from an old to a new generator.

Because the generators will work very different internally it will be
nearly impossible to produce literally the same cobol code with the
two generators. That's why we need methods or even better tools to compare

Any good tips and hints how to do it?
```

```
We're programming for a big car company a large batch
system running on OS/390 V1R3. Some time ago the people
of this company have written a generator to expand some
common so called "macros". When our project started our 
customer (this big car company) has forced us to use their
generator.

But now we've convinced them that this generator is
terrible (it *is* terrible) and to rewrite some of *our*
in other projects existing generators for them.

Our Generators are running on Linux (later on NT) and
generate from a "higher level" language with type
concept, subroutines etc. (the old generator has nothing
like this) Cobol code. This Cobol code is sent to the mainframe 
for compilation.

Because our higher level language is quite different from
the old macro language used before we can't generate 
literally the same code. That's why we need a tool to
compare the generated code and to prove that the sources
are functionally equivalent. The tools must compare a lot
of modules so they have to run in batch modus (platform is
not important).

Some things that will be different in generated sources:
whitespace in sources, order of sections, order of
variables declarations, partly level of variables. The
hierarchy of variables must be preserved of course.

Does it help to understand the problem?

        Burkhard
```

```
Hi Burkhard,

    While I do not have any tool that will help you out with your none to
small
problem I did work with a tool that is similar to what you are describing.

   The product was IEF and it is a PC based tool that would take
pseudo code "macro" instructions and generate either PL/I or Cobol
source code to be used on a mainframe environment. Most of the time
we used this tool the way new versions were tested was by taking the
PC code transfer it to the Mainframe and re-generate the source,
compile, link and bind the programs. At this point we would test the
load modules and determine if they were performing the same
function(s).

  We seldom used or compared the source from one version to
another. First because the code was nasty to use and second
because we never really cared what the source looked like we
just focused in on the results the load module created.

 I would suggest that comparing the results created by the
new load modules will be a better test then attempting to
compare the source code.

My best wishes.
```
