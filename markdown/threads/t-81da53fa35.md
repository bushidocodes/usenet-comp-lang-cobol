Report on April 15-20, 1995 ANSI X3J4 COBOL Committee meeting,
Islandia, New York, USA. When a vote is shown, the numbers are for,
against, and abstain (f-a-a).

The primary topics at this meeting were the proposals on file sharing
and record locking and recursive programs. The other main thrust was
to get the document ready for the CD (committee draft) review at ISO.
It was felt that one more meeting is needed to accomplish this. There
were several important proposals under consideration that passed.

The major topics/proposals we discussed were:

1. A proposal to provide dynamic cultural conventions (locale), based
on the Posix specifications was discussed. The proposal allows
dynamic choosing of attributes like collating sequences, currency
signs, alphabets, etc. The SET statement is used (like SET LOCALE
LC-COLLATE ...). Lots of changes were suggested and it was sent back
for rewrite.

2. A proposal to clarify multiple-octet characters and the report
writer was discussed. The proposal passed and will be added to the
base document.

3. A proposal to tie the standard to the ISO 10646 character-set
standard was discussed, passed, and will be added to the base
document.

4. A proposal for more cultural convention set processing that ties
in with the item 1 above was discussed. It provides sorting by
ISO-14651, an alphabet-name LOCALE for using local sets, a method of
specifying use of cultural convention sets for characters, currency
sign, and number format (DECIMAL-POINT IS COMMA) at runtime. These
can be changed dynamically by the locale methods provided by the
proposal in item 1 above. Also, it provides some functions for
getting various forms of dates and times and doing comparisons. The
proposal was modified significantly and sent back for rewrite.

5. A proposal on pointers (both program and data) was discussed.
There was a great deal of controversy over restrictions, what syntax
to use, and similar matters. It went from a simple facility to a
rather complex one. The paper was sent back for rewrite.

6. A paper on the deletion of parameterized classes was discussed.
The OO group has said they want to delete them in order to simplify
the OO facility. A former OO member says to not do this because you
will lock out generalized collection classes. Since I don't
understand any of this I am not quite sure what it all means. It was
decided to hold a special OO ad-hoc meeting in a month or so in the
San Francisco Bay Area in June to resolve this and other OO problems.

7. Several other papers and proposals on OO were discussed. Most of
this was deferred to the OO ad-hoc. We did decide to replace the
current OO example in the specification with a cleaned-up version. We
also felt that it should be changed in a few ways, such as not using
parameterized classes, no using persistent objects, and so on. A big
paper adding a large class library (some from Eiffel, some from
Smalltalk, some from C++, and so on) was received favorably. It will
be developed more.

8. A proposal on adding functions to convert a numeric value to a
boolean string and a boolean string to a number was discussed,
modified, and passed to be added to the base document. The main
change was to change it from bit strings only to boolean (which can be
either bits or characters) and to remove a function that tested bit
patterns (this can be done with the boolean operators).

9. A proposal to provide a boolean condition and to prohibit moves
and compares between non-boolean and boolean was discussed, modified,
and passed to be added. The main change this proposal makes is the
boolean condition. You can now say "IF boolean-expression THEN". If
boolean-expression (which can be one boolean item) evaluates to all
zeros, it is false. Formerly, you had to say "IF boolean-expression =
ZERO" or some such.

10. The third iteration of the proposal on adding floating-point and
true binary data types was discussed. Usages of BINARY-SHORT,
BINARY-CHAR, BINARY-LONG, FLOAT-SHORT, etc. provide this. The
optional words SIGNED or UNSIGNED follow. Lots of changes were made
and the proposal was passed to be added.

11. The fourth revision of my proposal on recursive programs was
discussed. Several changes were made (most of them to reflect
user-defined functions) and the proposal was modified, passed, and
will be added.

12. A proposal to provide functions to get memory (GET-STORAGE) and
return it (FREE-STORAGE) was discussed and modified. It was returned
for rewrite because it needed more work and depends on the pointer
proposal, which is also being rewritten.

13. A proposal to simplify dynamic tables was discussed. It changes
the syntax somewhat and makes them easier to use. The paper was
changed significantly and sent back for rewrite.

14. The proposal on file sharing and record locking was discussed and
significantly modified. The vote to approve passed by 3-2, but it was
not a 2/3 majority. The proposal will be rewritten and will be
discussed again at the next meeting.

15. Lots of editorial and minor change papers were passed and will be
added to the base document. Nothing very important was done in these.

The next X3J4 meeting will be in Post Falls, Idaho, USA, on June 24-28
1996, with an editing ad hoc meeting on June 22 and 23. The main
topic will be file sharing and record locking as well as some OO
changes. There will also be an ISO/IEC JTC1/SC22/WG4 meeting in
Zaandam, The Netherlands on 4 to 6 June.

Don Nelson
COBOL Development, Tandem Computers, Inc.
Member, ANSI X3J4 and ISO/IEC JTC1/SC22 WG4 COBOL Committees
nel··.@tan··m.com

Don,
A co-worker of mine gave me some copied handouts from recent SHARE
conference which referred to COBOL standards. I noted with interest that
ALTER and GO would be dropped from the language.
I certainly won't miss the ALTER verb, but there are quite a few in
my shop who use GO TO EXIT and will miss it. I understand there are some
extensions to the EXIT verb, i.e. EXIT PROGRAM, EXIT SECTION, EXIT
PARAGRAPH, and EXIT PERFORM (CYCLE). Does this mean that it will no
longer be possible to code PERFORM paragraph-1 THRU paragraph-n?
We have an amusing religious war regarding PERFORM THRU. One
department requires all PERFORMS to be PERFORM THRU, but corporate COBOL
standards prohibit it.
I also noticed that one change listed as "desireable" was to allow
MOVE arithmetic-expression TO elementary-item. I think this should be
easy to implement because it just inverts the order of a COMPUTE verb,
and I was wondering if it had been approved to be put into the new
standards.

Thanks in advance!

Arnold Trembley

No GO statement?????????????

Hey, could you hold off on that one until the industry has had a chance to
cope with the whole Year 2000 crisis?!?!

One of the major factors in this "crisis" is the overwhelming amount of
old code still around that still use a 2 digit year in date fields. If
there is that much old code around that it could precipitate a crisis in
the industry, wouldn't you expect that a lot of this old code would be
rife with GO TO statements as well??

One crisis at a time please!!!! Many sites have decided to just re-code
systems to deal with the date problem rather than redevelop their systems
from scratch. If faced with another issue of this consequence
(eliminating GO TO will force radical logic structure changes) at this
time, they just might decide to ditch the Cobol and go with plan B
(redevelopment ... in another language)

Think about it!!!!!
