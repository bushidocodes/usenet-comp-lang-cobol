To the newsgroup,
I promise to get the answer to this in the FAQ pretty soon. For anyone
who needs it sooner, try DejaNews - but the answers are:

1) Intrinsic Function, Current-Date (Standard if you have intrinsic
functions)

2) ACCEPT *extension* with YYYYMMDD or YYYYDDD keywords(available from some
vendors and taken from draft of next Standard)

3) Call IGZEDT4 if you are an IBM customer using VS COBOL II but not COBOL
for this-and-that. CICS customers should look at EIBDATE.

4) If you are using an old '74 Standard compiler or an '85 compiler without
the intrinsic functions and your vendor provides no extensions for accessing
the 4-digit year in a date, then it is suggested that you look at the
comp.software.year-2000 newsgroup and review the material on survivalists
and moving to rural areas before Dec 31, 1999 :-)

Chuck,

The format returned from ACCEPT FROM DATE is as defined in the ANSI
standard. This has recently been 'enhanced' to allow for YYYYMMDD in the
latest COBOL standard proposal. Micro Focus has implemented this in the
latest versions of its products.

If you are using an older version of our product you can always use the
GET-CURRENT-DATE intrinsic function which will return a four digit year as
well.

Hope this helps.

the nature of the file coming in. Usually mainframe files don't use that
convention. I've only used this once but at lest in the old days), it
amounts to another byte. I.e., +1234 stored as PIC S9(4) is four bytes
with the sign 'hidden' in the upper half of the last byte, but with SIGN
IS LEADING it is 5 bytes (a separate byte for the sign). This could be
the problem also, because the input is not where th program thinks it
is, so a non-numeric value is sent to the table.
