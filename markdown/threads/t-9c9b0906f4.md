```
>(Don't even trust the data if its from your own program, that is, if it's
>passed from one routine to another.)


I disagree with this.  In fact, a basic principle of decent top down
structure is:  "work on the module you are working on, and trust
yourself to take care of other details in the modules responsible
for them".   If every module has to repeat all the work of every
other module because you write in such a way that no module
can be trusted, then you are just going to have a huge mess with
all sorts of distractions.

I would agree that you should be very careful with incoming data,
and always test it.  But once that you have tested it and ensured
that it conforms, then you *have* to trust it, or rewrite your test.
```

```
I'd recommend using a conditional include for debug builds if your compiler
supports it. Have each module check that its data in received in the format it
expects and complain loudly if not[1]. I've found this allows you to very
quickly find where bad data is entering the system while developing the various
modules.  It also allows you to avoid the overhead of the exhaustive checks in
released builds.

the data formats, which is handy for including "Expected x got y". As usual,
```

```
Well it's true, I seldom work on programs where the source is bigger
than 2 or 3 megabytes, and the copy books include more than 2 or 3
hundred modules.  In fact, I think the biggest program that I have ever
written has been less than 350,000 lines.  It's also true that I seldom
have to deal with programmers that don't check side-effects when they
change things, but mainly because I either re-train them or fire them

nature does occur, I also insist that they go back and fix it, rather than
slap a patch in somewhere else in the code.

Maybe that is why the code that was written twenty years ago is still
being sold at a profit, and integrated into new systems.

The place to check for data integrity is when the data enters the
system, not each and every time it is used.  That is a basic
principle of programming, in my opinion.  It is also one of the
basic principles of Cobol, and one of the reasons that the language
is so highly typed.  Business applications *need* that kind of
data integrity, and the larger the project the more important
rigor in managing the programs becomes, not the other way about.
```

```
Albert Ratzlaff wrote in message <3642D992.EFF37B75@infonet.com.py>...

>I'm sorry I left the impression that double-checking the data obliviates

<S>I think perhaps we should declare a truce on this, and start over.
The original statement was "NEVER trust the data", with an addendum
of even if it is your own program.  Since the thread is on technique, I
jumped on that.  Perhaps I went in the wrong direction though, and should
have said "Never make blanket statements" about technique.

One of the things that I see beginners do that invariably gets them into
trouble on large projects, is not chunking things down small enough. I have
found, at least for myself, that the key to making a large project simple
is twofold.  Firstly, you have to make sure that you break it down into
small enough portions that each is easily understood.  Secondly, you
have to isolate the data/parameters for each of those portions so changing
one portion does not affect another portion.

It is that latter that causes the problems.  Obviously, to some extent,
the modules are related, and equally obviously, they are going to share
data.  The problem then, as I see it, is to design the modules in such
a way that each module is responsible for the data integrity of its
own module.  Other modules should, and need to be able to trust
that data or you are going to get into trouble.

An example would be a simple master file.  The master file update
module should ensure that all data in the record is edited correctly.
That might be a file copy or a subroutine, but it is fairly crucial that
all updating of that file, as well as all data checking, should be kept
in that module.  Outside it, one should be only using the data, not
modifying it.

Now, in fact, we often find that purity of approach impossible. The
update module may become too large to fit into a program that
has to change a master file field, for example.  One is also faced
with time constraints, and sometimes a patch is forced on the
programmer, for all sorts of day to day reasons.

Nevertheless, as one gets away from that isolation, the system
suffers for it.  We have all dealt with patchy programs, and I think
we would agree that the more scattered the patches become,
the more difficult the system gets to maintain and extend.

Your example with an index is actually an excellent one.  Yes,
I would agree, that in code of that sort, one should reset the index
and not assume.  However, that is precisely why I never use
the "indexed by" construct.  I know it is more efficient.  But an
array does not need it; a subscript can be used instead. If I had
to write a patch, I would introduce a *NEW* subscript.  That way,
I am absolutely assured that the act of setting it does not create
a problem somewhere else, AND I maintain the principal of
isolation ... in effect my patch becomes an integrated unit that
does not affect any other units.

I also avoid USAGE statements like the plaque for similar reasons.
Those copy books might very well be running on different platforms.
A slight increase in efficiency is not worth a programming problem
in 99.9% of all applications, and I expect every bit of code I write
to live through at least three or four migrations.  In fact, DOS lasting
as long as it did made me as sloppy as hell, and I am re-learning
a lot of those basics.    ;<(

I trust that makes what I am trying to express a bit clearer?
```

```
I have to disagree here (in a collegial, non-flaming, way). The reason
COBOL programmers coded UNTIL blob > some-other-blob is because COBOL,
up to but not including the '85 standard, closed the loop at the top.
Meaning, COBOL increments your loop variable, then tests for the limit.
In the example above, ... > +60, you loop 60 times, then bump the
variable to +61, test it and stop, because it's > +60. In many cases
this caused problems, because we couldn't rely on the contents of the
subscript after the PERFORM was finished. COBOL II (completely IBM MVS
oriented) changed all that with the "WITH TEST AFTER" phrase. Here you
code until something = the-limit. Of course, in COBOL II we can also
code UNTIL blob >=  the-limit.
```

```
I would have to agree with Donald on this one.

If you are talking about raw data then yes you need to validate.  If you
are talking about data generated from a previous program that you
control then no.  For example it is insane to test whether the month is
in the range 1-12 when a previous program calculates the date, if it is
wrong then the previous program should be fixed.  Do not clutter a
program with all sorts of checks.

simple.

evaluate month
when 1,2,....
otherwise
	error
end-evaluate.
```
