> I would like to be able to assign attributes to screen section items

I am not sure that one can with SCREEN SECTION.  I always did
field-by-field processing just using ACCEPT/DISPLAY AT and changed
colour by routines at appropriate places.  Typically I would have:

IF ( Field-Number = 4 )
PERFORM Screen-In
DISPLAY field4     AT field4pos
ACCEPT field4     AT field4pos
PERFORM Screen-FG
DISPLAY field4    AT field4pos
if exit-key move 99 to field-number
else if back-key subtract1 from field-number
else
check field4 ......
if ok add 1 to field-number
end-if end-if
END-IF

This was suitable for being code generated from a text layout file.
The Screen-In, Screen-FG, Screen-BG, etc would change the colour being
used by a DISPLAY or ACCEPT to whatever the user had chosen for his
settings for that system.  It is still very useful for mult-company
sites because the users set up different colours for each company.

This is implemented different ways depending on the version of Cobol,
terminal types etc, and goes back to when I was using serial terminals
on Concurrent-DOS (where the 'colours' may be bold, half-bright,
inverted etc) so the termiinal type still includes ANSI, VT52. I think
it uses CALL x"A7" for IBM screens with MF, for others it DISPLAYs a
control string.

> and to be able to determine
> which field in the screen has control on accepts and displays.

If you are accepting several fields at once then with MF you could use
CURSOR IS Cursor-Position in SPECIAL-NAMES and this can be set to the
cursor position required at the start of an ACCEPT and it reports where
the cursor was at the end of the ACCEPT.  It may be available with RM.

This doesn't prevent other fields being entered, but it does put the
cursor where required.

Your post made me think back to the days under DOS and Win 3.1 when I used

I've decided to make this code public domain because I believe anyone still
using Screen Sections needs all the help they can get <G>.

FUNDAMENTAL PRINCIPLES:

1. Remove screen processing from the Business logic. (Multi-tier design).
This means that application code should simply see a buffer containing the
variables from the screen and it should manipulate those then put them back
(code provided below does this)

2. GENERATE the screen formats by "painting" them. I have a tool I wrote
called ACEPAINT. You draw the fields you want, indicating what is protected
and what is variable, and the colours you want. ACEPAINT scans the screen
and generates definitions as follows:

1. MFS type 1 buffer for IBM mainframe.
2. CICS Map for IBM mainframe.
3. Screen Section for the PC.

These principles then allow the same screen to run on both mainframe and PC.
It is also a fairly simple matter to add other hardware support.

DETAILS (this is long, but it may be of use to somebody...It is taken from a
document I sent to someone in 1994, who had requested very much the same
thing you have, but for a HP environment):

Hopefully, the following will be of some help...please excuse the "old"
COBOL; I wrote this stuff 7 years ago and blush when I look at it now.
Nevertheless, it is fully debugged and working and does the job.

The scenario is as follows:

1. For each screen that the system can display, we will create an
encapsulated module consisting of a screen section which defines the screen,
a linkage section which allows us to pass a corresponding message, and a
procedure division which processes either input or output and calls a module
to deal with dynamic attributes (in the PC case, only cursor placement. MF
ADIS cannot handle true dynamic attributes). These screen handling modules
will be identically coded; the only difference is the screen and linkage
sections.
A copy of the skeleton code I use is included below. Although I have a
generator which I wrote myself (ACEPAINT) to do this for me, it is a simple
matter to do it manually. (If you are likely to be doing  a lot of this,
I'll rebuild ACEPAINT and you can have it for $50)

We'll start with the Calling program's view. There's a message buffer with
all the variable fields used in the Screen Section of the Screen Handling
module for the screen we are concerned with. Each variable field in the
buffer  is preceded by a 2 byte field which may contain a dynamic attribute.
These are "control bytes" which control the way things are displayed on a
screen. The ones described below pertain to the mainframe IBM 3270 and are
accessed by a system called "MFS" (message format services).  As I like my
code to run on both PCs and mainframes, I use these attributes. You may have
different ones for HP.
MicroFocus COBOL only allows cursor insertion as it cannot dynamically
modify attributes. I have organised things so that the same attribute which
inserts
the cursor on the mainframe, also does it on the PC...

Note that the above code uses a table of offsets to the variable fields.This
is produced automatically by my generator (ACEPAINT). It would be more
 tedious to have to do it manually, but it is the only way to get cursor
placement correct in the MF environment. (MicroFocus requires the line
and column to place the cursor, so it is therefore necessary to equate each
variable field with a line/column).

The AVMOVE routine can be replaced by reference modification. It is an
Assembler routine I coded for both the mainframe and the PC, which
allows a totally flexible COBOL MOVE facility. It has been superceded by the
arrival of Reference Modification.
AVMOVE has the following parameters:

1. The source field
2. The offset into the source field (starting at zero)
3. the target field
4. The offset into the target field (starting at zero)
5. the length

Now let's have a look at the screen handling module skeleton:

There will be one of these for each screen you intend to display. Each
module will have a corresponding message buffer to allow input/output of the
fields
defined in the screen section as variable. In my system each of these fields
is preceded by a 2 byte attribute field to support cursor insertion as
you saw above, but if you are happy to always insert the cursor on the first
variable field on the screen (MF default), you could eliminate this. (What
about
when errors are detected and it is necessary to place the cursor on the
offending field?)

As you can see there are 3 "tags" in the above program which tell the
generator where to insert the message buffer (%MSG.TMP), the table which
equates variable
fields to a LINE/COL for MF COBOL (%VAR.TMP), and the physical screen
structure defined in the screen section (%FMT.TMP). The AID (Attention
Identifier)allows us to handle PFKeys (Program Function Keys). For example
if the user presses PFK1, the AID byte returned in the message header will
contain a 1.
Again, most of this has been coded for compatibility with IBM Mainframes. It
is important to remember that even if we are running solely on a PC, these
functions
are still a necessary part of the processing, and the above code has been
tried and tested for many years in a production environment without failing.

This module would be called from a main program which would pass the initial
setup for the screen in the message buffer (if required) and would receive
back a message buffer filled with the user-entered data and an indication of
the PFKey which was pressed (if any). The same screen format would run on an
IBM mainframe and use the same message buffer to get and put the same
information.

You can relace the call to LCTUC with a call to the equivalent MF function.
(I use LCTUC because I wrote it myself in COBOL and it is therefore
transportable to any environment)

environment. If your HP system already has a screen handling module, maybe
you could write something which would convert the existing HP screens into
MF screen section format. Wrap a skeleton around it (something like the
above) and replace the HP calls to the screen Handler with a call to your
own control module which would call the appropriate screen format and return
the message buffer.
