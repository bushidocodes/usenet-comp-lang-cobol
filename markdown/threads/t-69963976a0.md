```
Here is some of my opinions, and I would also like some other opinions on the
following scenario:

Using the call statement 
ProgA -> Calls -> ProgB
If ProgB can return 2 fields
Field1 is 10 bytes
Field2 is 16k bytes
The linkage section of ProgB will contain both fields, plus a parm to tell what
it needs to return, but in 90% of the cases, it will only return the 10 bytes. 
Many programs may call ProgB, as ProgB is a driver program to return either 10
bytes or 16,010 bytes depending upon the parm it receives.

These calls are pass by reference, and I'm curious to know the best performance
if this is getting called continuously by many different programs.  I assume
that performance would not be affected by having the additional 16k in the
linkage section even if 90% of the calls won't use it, as it is just a pointer
to the calling program and won't be using/passing any information if Prog A
doesn't pass it any fields other the a parm and the 10 bytes.  (opinions?)
I have thought about making 2 different drivers (i.e. 2 diff progb's) as one
could contain just 10 bytes, the other 16k, and the calling program could
choose what they need.  This may help performance, but I don't think it would
because I don't think performance is affected because the calling program isn't
passing the additional 16k parm if it isn't needed (as the prev paragraph
states).
I would like to see if anyone has any opinions/experience with this scenario
and if you agree or disagree. (?).   
I am also curious to know if it would help to do a "cancel' from each calling
program after calling progb and getting data back, as cancel will free the
memory since the progb is only needed once in each program that calls it.
Would pass by content instead of reference have any performance advantages?
I don't believe it would, but I have always used pass by reference on the
calls.
Opinions anyone?

                  Thanks.
```

```
All the parameters are passed as addresses.  10 bytes or a zillion 
bytes.  It makes no difference. The only performance hit is that it 
has to pass an extra address, but I doubt if anyone could ever measure 
any difference.

> I have thought about making 2 different drivers (i.e. 2 diff progb's) as one

You would probably take more of a performance hit in figuring out 
which one to call than that for the extra parameter.  Remember, the 
16k is not passed.  Its address is.

> I would like to see if anyone has any opinions/experience with this scenario
> and if you agree or disagree. (?).
> I am also curious to know if it would help to do a "cancel' from each calling
> program after calling progb and getting data back, as cancel will free the
> memory since the progb is only needed once in each program that calls it.

No memory is taken up by the parameters.  There is normally lots of 
overhead in CANCEL (it all depends on the implementation).  There is 
no reason to do this.

> Would pass by content instead of reference have any performance advantages?

Lots of disadvantages because the compiler has to allocate space for 
the parameter, move the data to it, and then pass the address of the 
area.  And, you cannot pass data back to the caller in a parameter 
passed by content.

> I don't believe it would, but I have always used pass by reference on the
> calls.

Which is what you should do in this case.
```

```
There is actually an overhead of having large passareas in
IBM COBOL. It generates a base address per 4k of size in each
linkage area. So for a 4mb area, you get 1024 address resolutions
(at about three instructions per resolution). This is because of
the 4k offset limit in the 390 instruction set.

But even 16k would only generate 4 base registers which should
not be a problem.
```

```
I'm not aware of any useful performance benefits from using
CANCEL. 

It can save memory: in a dynamic environment it can result in the
program being unloaded, thus saving memory if it is not used
again.

On the other hand if it is used again, you are up for a very
expensive reload operation.

In a non-dynamic environment, it has to reinitialize the working
storage, which is expensive.

On the mainframe, a CALL is itself a very expensive operation,
around 100-200 instructions plus about 3 for each paramater, plus
another 3 per 4k of size of the declared linkage area. Dynam
calls are worse again, maybe another 50% on the call itself.

The overhead is much reduced, I am told, if you use nested
procedures - that way it does not have to go through all the
language environment slop every time.
```
