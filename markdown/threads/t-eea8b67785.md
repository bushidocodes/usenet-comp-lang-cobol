I haven't seen any reply to my earlier note, so I wanted to re-post with
slightly different wording.

Consider 3 sets of code:

Sample 1:

Evaluate aField
  When "A"
      Perform A-rtn
   When "B"
      Perform B-rtn
    When "C"
       Perform C-rtn
 End-Evaluate

    ***

Sample 2:

If aField = "A"
      Perform A-rtn
 Else If aField = "B"
      Perform B-rtn
 Else If aField = "C"
       Perform C-rtn
   .

    ***

Sample 3:

If aField = "A"
      Perform A-rtn.
If aField = "B"
      Perform B-rtn.
If aField = "C"
       Perform C-rtn
   .

***

Now the "question" is which two of these pieces of code are LOGICALLY
equivalent and WHY is the 3rd sample "logically different"?

When you "know" (accept) the answer, you will see that NESTED "IF's" can be
coded to be logically equivalent to EVALUATE statements - but consecutive
ones cannot be coded to GUARANTEE logical equivalence.

Then the question becomes "which is more easily maintainable" nested IF's or
EVALUATE.  I certainly would go with the latter - but I do understand that
the former MAY be more maintainable - to the person who uses it regularly and
does not always use '85 Standard compilers with EVALUATE support.
```

COBOL *does* make the "WHEN OTHER" feature optional.  I wasn't part of the
Standard process when this was "decided".  However, it does seem at least
some what reasonable to me - as it does mean that there are "equivalent"
EVALUATE and IF statements, e.g.

 Evaluate aField
    When "A"
        Perform A-rtn
     When "B"
         Perform B-rtn
      When other
          Perform Other-Rtn
   End-Evaluate

is logically equivalent to

 If aField = "A"
        Perform A-rtn
  Else
     If aField = "B"
         Perform B-rtn
    Else
          Perform Other-Rtn
    End-If
  End-If

   ****
While

Evaluate aField
    When "A"
        Perform A-rtn
     When "B"
         Perform B-rtn
  End-Evaluate

is logically equivalent to

 If aField = "A"
        Perform A-rtn
  Else
     If aField = "B"
         Perform B-rtn
     End-If
  End-IF

In other words, either nested IFs or EVALUATE *can* be coded to handle
"OTHER" situations - but it is also true that either can be coded to "ignore"
such.   Of course, the last two examples are ALSO equivalent to:

Evaluate aField
    When "A"
        Perform A-rtn
     When "B"
         Perform B-rtn
      When other
          Continue
   End-Evaluate

is logically equivalent to

 If aField = "A"
        Perform A-rtn
  Else
     If aField = "B"
         Perform B-rtn
    Else
          Continue
    End-If
  End-If

  ***

Once again, which you code may well depend on what you are used to and/or the
complexity of the actual logic and/or the expected maintenance/enhancements
expected for the "life" of the program.
```
