Yes, no problem <G>...

Despite protestations to the contrary, the question is worded clearly.

You got it right, the only bit you missed was the fact that 50 records need
to be stored and each record is "grouped" as two fields.

Here's the code:

01 ws-table.
     12 the-record-level occurs 50
                                    indexed by trl-x1.
          15 the-record-group occurs 15
                                    indexed by trg-x1.
               18 part-no   pic 999.
               18 quantity  pic 999.

You can see how close you were...<G>.

Now, if you want to reference a whole input record (all 15 groups) you can
say:

set trl-x1 to <whatever number between 1 and 50 represents the record you
want>
move the-record-level (trl-x1) to ...<wherever you want to put it>

OR

you can use a fixed subscript (which gets resolved at compile time, instead
of at run time, so it is more efficient) thus...

move the-record-level (14) to ...<whatever>

(Obviously, you can only use a fixed subscript if it will always have that
value...efficient, but not very flexible...)

If you want the part number and quantity together, of the 13th group in the
23rd record, you can say:

move the-record-group (23, 13) to... <whatever>

(or set the two indexes like this...)
set trl-x1 to 23
set trg-x1 to 13
move the-record-group (trl-x1, trg-x1) to...<whatever>

The real advantage of being able to reference your table with variable
indexes comes when you have to "put stuff away"...

HINT:(some code has been omitted for clarity, and to ensure that you do your
own homework <G>)
...
working-storage.
01  EOF-flag    pic x value space.
      88 no-more-input   value '1'.
      88 theres-still-input value '0'.
...
procedure division.
...
perform
     varying trl-x1
         from 1
            by 1
        until trl-x1 > 50 or no-more-input
            read in-file
                 at end
                          set no-more-input to TRUE
                          exit perform
            end-read
            move in-record to the-record-level (trl-x1)
 end-perform
...

Your compiler MAY not support "exit perform"... If it doesn't you will need
to use the "not at end" clause as well as "at end" and you will need to move
the "move in-record to the-record-level (trl-x1)" statement into the "not at
end" scope.

I'm sure you will realise that you can reference the individual part number
or quantity by providing a pointer to the two levels ofthe table, exactly as
we just did for the "Group"...

move part-no (trl-x1, trg-x1) to ...
move part-no (39, trg-x1) to...
move part-no  (47, 1) to...
move quantity (trl-x1, 3) to ...

This is manipulation of a two dimensional table. There is nothing mysterious
about it, once you realise the "cascading" nature of its hierarchic
structure.

Remember only to put the PICTURES at the "bottom", after all the levels of
the hierarchy have been defined...(see above.)

You should read about defining tables with OCCURS, and using indexes and
SEARCH to access them.

Also, realise this is a contrived exercise purely for yur education. In real
life it is VERY unlikely that you would read input records into a table.

Why do you think that is?

HTH,

Pete.

```
With some compilers (certainly not all) on some operating systems (certainly
not all) indexes are still SUBSTANTIALLY faster/better performers than
subscripts (no matter how defined).

Long before OO came along, it seemed to me (and I know that others feel
exactly the opposite) that an "index" more closely "modeled" the real world
of tables than a subscript did/does.  On the other hand, when the "entry
into a table element" is a "real world object itself" - then I expect it to
be a subscript (rather than using lots of SET statements).

I may be wrong, but it seems to me that it NEVER hurts to define an index
for a table (when first creating it).  You may or may not ever actually
reference it in your procedure division.
```

higher level application code ... that person does not need to know how it
works.  They place the data into the object, and execute the
"calculate-prices" method, then check an error-flag.  On errors, they use
the error-number property or the error-message property, or both.  Defaults
are defined as the first of the six codes for each tax, and the actual codes

methods.  Stuff like where the tax table is stored, and how it is loaded,

a contract) is entered and edited within the tax object.

simply fires up the cash register, enters some dollar figures, and hits the

transfers (presuming the table is stored globally on a server).

While most people teach OOP as being a tool to be used within a programming
conundrum, it is also a tool that can be used within a business conumdrum.
Just as a pulldown box can be an isolated object on a screen that has
nothing to do with the application beyond being used, certain things in
bussiness apps can be viewed as isolated objects that have nothing to do
with the application system beyond being used.

used in a cash register vs an accounts receivable system is totally
irrelevant.  By making it a complete standalone system in itself, and simply
using that object in all systems accross a company, one ensures two things

that any bugs in sales tax calculations are isolated into a single place.

system, complete with its own screens, its own methods, and its own source
code.  The registers within that caculator can then be made available to the
higher level programmer by declaring them to be properties.  The programmer
can then move stuff in and out of the registers with simple move statements,
and the programmer can push the buttons on the calculator by invoking
methods.  NEW registers and methods can be introduced at any time, and do
not affect existing systems. Cobol understands the object at the compiler
level, and will give you a compile time error if you try to move a PIC X(10)
field into a PIC 9(5)v9(3) register within the object.

It is really quite neat.
