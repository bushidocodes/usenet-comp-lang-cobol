The "ascending key" describes how the data is STORED - not how it is
accessed.  Therefore, if your table isn't populated with the data in
ascending order, using that phrase will not "force" it into that order.

I don't know which compiler you are using, but some COBOL products have an
extension to the current Standard (which will be part of the next Standard)
to allow you to SORT a table - using the SORT verb.

As indicated in another note, your problem is (most likely) a "homework
assignment" - in which case, I suggest you look at the technique that your
class is expected to use, however, if you are using a COBOL compiler that
supports the "sort aTable" feature, you might find this a useful thing to
learn about.

Assuming that the dates you are talking about are contained in a file,
you have, essentially, three alternatives:

1) Use the sort utility provided with your OS to sort the file by the
field representing the dates, and then, in your COBOL program, populate
the table from that file.
2) Sort the file via USING-GIVING in COBOL (INPUT/ OUTPUT procedures
appear to be an overkill in this case), and then populate the table.
3) First, populate the table from the unsorted file and code a few extra
lines of COBOL to sort the table in place.

From the computer standpoint, the variant 3 appears to be most sensible.
Sorting algorithms work fastest when an entire file is in memory - and
this is exactly what a table is. If your table has a couple of dozen
occurrences, any algorithm, even as poor as bubble or insertion sort,
will do the job rapidly enough. With about 10,000 entries, a more
efficient scheme, like Shell sort or so-called CombSort (that improves
bubble sort much in the same way Shell improves insertion sort) should be
used. With over 100,000+ keys, you cannot go very much wrong using
QuickSort, which is the best comparison-based all-round sorting method.
All the algorithms above are amply described in the literature. With the
exception of maybe QuickSort, they are no-brainers to code. Besides,
coding things like these is an excellent programming logic exercise.

Lastly, you can take advantage of the fact that your keys are dates, and
therefore they can be very easily mapped in one-to-one manner to a set of
natural numbers ranging no wider than 0 to 36500 for an entire century.
It means that, after transforming the keys to such integers via a simple
rapid computation, the table can be sorted in several lines of code using
key-indexed (distribution counting) sort - the fastest existing sorting
method. Distribution counting can also be found in any book on
algorithms, for instance, D.Knuth, R.Sedgewick, or
T.Standish.
