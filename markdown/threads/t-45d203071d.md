Okay. Since you asked, this is where the whole sordid story comes out!

I was being lazy. I was asked to write a generic subroutine that could
be called by any "COBOL" program. And it can. However, when I originally
came to test it I decided I wanted to play with the test data (two
records) and some standing lookup data and call the program (via clist)
all instream in my JCL, and without having to write another test module
to do the call.

That's when I found that the subroutine was doing the initialisation
every time. So I thought (dangerous that sometimes - thinking I mean) it
was obviously something to do with the environment: "What I need is a
cobol stub to setup a cobol environment". I then trawled through the
TSO/E Programming Services manual to learn how to call a clist from a
cobol program. Hit upon the TSO Services Facility and lifted the sample
program which calls IKJTSOEV to initialise a TSO/E environment (hang on,
didn't I want to create a cobol environment) and then calls TSOLNK
(alias for IKJEFTSR) passing %MYCLIST as a parameter to invoke the clist
which in turn calls my cobol subroutine. Hmmmn!

Well it all worked up to a point. I got my cobol stub to run a test
clist, and I was able to interogate the return-code from that. Myclist
was able to call the cobol subroutine, and I could tell from the journal
that the program ran once ok. However, an abend resulted from the return
of control back to the clist, and the whole house of cards collapsed.

The abend was S30A ERROR DURING FREEMAIN and
Reason-code - 14 - subpool supplied is not for private area storage
or private area storage not in indicated subpool.

I guess that's back to environments again! I have been programming for
about 8 years now (mainly standard batch stuff), but this is at the top
end of my system knowledge and I can't take it any further. So much for
being lazy eh?

The thing is I am pursuing this mainly for my personal benefit, and in
the interests of extending the boundaries of knowledge. I don't really
need it for my work. However, any comments or suggestions would be
gratefully received (such as: "suggest you quit while you're ahead").
Many thanks for the interest shown.

The issue in this case is if your clist is using the TSO CALL command, the
target program will be invoked under a subtask of the original program and not
part of the current task. Any storage allocated by the subtask in a non-shared
subpool will be automatically freed when the subtask ends. 

The 30A would make sense if the calling program is trying to do some cleanup
of the subroutine.

There is no way to automatically share the subpools - it would have to be at
the point of the attach.

You might be able to avoid the subtask problem by using PLATPGM but that would
require the assistance of your friendly systems programmer.

Another approach is to write an IKJCT44B (refer to TSO/E Customization) and
have is issue a LINK to your subprogram. I don't recall if IKJCT44B has to be
in STEPLIB or if you could put it in your COBOL load library.

Go for it.
