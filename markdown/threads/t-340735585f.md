Robert,

I have been doing this kind of thing for many years now and have acquired a
large amount of experience, expertise and tools for the exercise.

I wrote DECLGEN for the PC so that PC COBOL programmers could generate Host
Variables and record layouts from RDB tables.  This tool is useful once you
actually have a Relational Database.

NEGLCED was a name I proposed for a function that does the "other side" of
DECLGEN. That is to say it actually builds a 3NF Relational database from
your existing ISAM files. I have now written this tool and it has proven
itself in a major software conversion a few months back.

However I didn't call it NEGLCED. It is called ISAM2RDB.

Both DECLGEN and ISAM2RDB run as Windows applications in a Win32
Environment. But, as I'm sure you are aware, there is much more to
converting files than just running a couple of tools.

It looks as though the people you are working for have not given any thought
(or understood) the concept of the Relational Model and have simply
converted the existing files as a decree of fashion... Many Corporates are
being driven to do this because their data is locked into the COBOL file
system  and they figure if they can get it to an RDB at least it is open and
available to standard tools and software. They are right, but it can be a
horrendous mess if they just convert the existing files willy nilly.

Now to your problem...

If the data has already been converted and contains columns that  reference
other tables, you will need a normalization process to remove the references
and join existing key dependent data directly to the referenced tables.

It is a major effort. The fact is that the data is there, but it is
unnormalized. You need to determine which of these converted tables share
the same key dependencies and restructure the data to a single table, based
on that key dependency. Sorry, it is hard to explain this in simple terms in
a few minutes...

If you have a table (Table A) which has a column (A3) that references Table
B, then the key of Table A is probably the same as Table B. Normalization
requires you to remove key dependencies between tables by combining tables
which depend on the same key structure.

The good news is that it isn't too hard to write something that can do this.
The bad news is that the existing code then also has to be amended.

The process of converting Flat Files to Databases involves three steps:

1. Create the Relational structures to hold the existing data. (ISAM2RDB
automates this and provides a 3NF Relational database).

2. Load the RDB with data currently held on the flat files. (I have a
semi-automatic process that does this and cuts out much of the "donkey
work". It will be fully automatic by the end of this year.)

3. Convert the existing application code to access the new RDB. This has to
be manual, but I have tools that can help, and there are a large number of
"rules" derived from experience at doing it that can save much heartache.
The best architecture is a multi tier one, and some of the code at some of
the levels can be generated. Other code can have minimal maintenance by
using calls to generated routines for updating.

I am very much in favour of automating as much of the above work as
possible, not just because it saves work and time, but because it eliminates
human error. Humans tend to make errors when they have to do repetitive
tasks. Some of these errors can be very hard to find and will trip you up
later.

In your case (and it is sadly often the case) they are so obsessed with
getting the data into RDB that they just don't really consider the other
things that must be done also. They figure they will minimise the impact on
existing code by converting directly on a 1 for 1 basis to RDB. It is  a
false hope and it provides none of the benefits of the Relational Model (you
noted the inability to join such data as just one shortcoming...there are
many others.).

I really hope that anyone considering this exercise will write me. I have a
large amount of useful material and tools and there is not always a cost
involved.

Pete.

It is good practice to normalize the system and then build redundancy back
in to improve performance, IF REQUIRED.

I don't personally adhere to the dictum that only a "pure" database  (3NF
guarantees non-redundant non-key data) is acceptable. In the real world
there are databases that are less than perfect being accessed in ways that
are less than perfect. Sometimes a small amount of redundancy can improve
access pathways and ease bottlenecks, but it needs to be carefully
considered and tested.

This is what your DBA did and I commend him for it.

This kind of "tuning" is much more frequent on Databases that are not based
on the Relational Model.

IDMS is a Network Model based database.
