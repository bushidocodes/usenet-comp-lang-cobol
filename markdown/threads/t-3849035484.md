I appreciate this point, tending to view the situation rather as good than bad
from the standpoint of efficiency. In languages where an internal recursion
mechanism is available, many common algorithms are often unnecessarily
implemented this way. In those languages, including COBOL, whose compilers do
not support recursive calls, iteration is the only way to go, and it is always
more efficient. Note that it does not imply that COBOL is somehow better than
other tongues because of lieu of a certain feature. Nobody makes a C programmer
use recursion instead of iteration, either. There are also situations in SAS
when programming an algorithm explicitly from scratch is much more efficient
(sometimes even the only feasible solution) than using an existing prewritten
routine coded in the underlying software. 

Maybe owing to the very nature of business to which COBOL is most often applied,
few programmers attempt writing core algorithms in COBOL. This is unfortunate,
all the more that COBOL compilers, as a matter of rule, are excellently
optimized and, therefore, ideally suited for high-performance programming.
Taking binary search as an example, it takes several lines of code to write one
explicitly, thus making it practically always faster than linear search -
instead of arguing fruitlessly about SEARCH ALL being more efficient than SEARCH
only for a number of array items exceeding N=100 or so due to the overhead of
the binary search call. Data structures available in COBOL naturally lend
themselves to implementations of internal hashing (whose algorithms, granted,
require thorough understanding) instead of binary search. Even though the former
is, on the average, log(N) times faster than the latter, not to mention that it
requires no sorting, it is almost never attempted. You can add to the
list.                 
          
> >>It is powerful because it has the capability to simplify tasks that would
> be tedious and complex in other >languages.
…[11 more quoted lines elided]…
> has these features too.

As much as it is fun to program from ground up, I see the absence of routines
internally implementing some of widely used computing chores as a major omission
from COBOL. For instance, after sorting, file match is the most common operation
in business data processing. Yet it is not there, even in /370. It is a typical
example of a routine that is not hard but tedious, and hence error-prone, to
code. Same pertains to control break processing. I see no reason why these
features should be available in EZ and SAS but absent from COBOL. They would not
detract a bit from a COBOL programmers' ingenuity; instead, it would leave them
more time for creative work. Another example that readily comes to mind is macro
processing.     

As far as 'definition war' and such goes, no language is a sufficient ground for
religion, even if it helps one to make a buck: It is but a tool. COBOL bigotry
is as funny as that with respect to any other programming language.
Unfortunately, here its smell gets too thick more often than a good dose of
humor would dictate.
