```
The problem here is that there are 256 characters in a collating
sequence.

The order of appearance of literals in the ALPHABET clause specifies each
character's ordinal in ascending sequence. Any unspecified characters in
the native collating sequence have higher positions in the new collating
sequence than all specified characters. The relative order of the
unspecified characters is the same as in the native collating sequence.

So, you can easily change the LOW-VALUE character, but, in order to
change the HIGH-VALUE character, you have to count carefully out to 256.

For example, I added the following ALPHABET to my previously-posted
program:

ALPHABET ALPHABET-EXAMPLE
"A"
1 THRU "@"
"B" THRU "Y"
"[" THRU 256
"Z".

I also modified the program to display LOW-VALUES and HIGH-VALUES instead
of LOW-VALUES and SPACES, and this is what I got:

$ cobol demo.cob
$ link demo.obj
$ run demo.exe
"AAAAAAAAAA"
"ZZZZZZZZZZ"
$
```

```
The purpose of the high and low values statements is to provide the largest
and smallest (or first and last in the collating sequence) values that a
specific
data type can have. The actual values depend upon the picture. For example
low values for pic 99 is "00", while low values for pic XX is " "(space).

Often, one wishes to initialize a value to one extreme or the other. For
example,
to read an entire ISAM file, one would initialize the key to low values, and
then
use the start verb. By using low values rather than 'all "0"' or 'all
space', one gets
around the problems of various character sets, or perhaps a key changing
from numeric to alpha and causing a program bug.
```

```
Close. HIGH-VALUES and LOW-VALUES are defined in terms of the collating
sequence, and you can change the collating sequence in the ENVIRONMENT
DIVISION, but the "purpose" of HIGH-VALUES and LOW-VALUES is to provide a
value that is larger or smaller than any user data that can appear in the
field.

It is "out-of-band" signaling, used for sentinels, sorting, searching,
distinguishing "null" from "zero", &c.


› values that a specific data type can have. The actual values depend
› upon the picture. For example low values for pic 99 is "00", while low
› values for pic XX is " "(space).

No, the actual values depend on the collating sequence, not the PICTURE
clause, and unless you've changed the collating sequence in the
ENVIRONMENT DIVISION, they are X'00' and X'FF'.

Note that HIGH-VALUES, for example, represents one or more occurrences of
the character with the highest ordinal position in the program collating
sequence. Therefore, the CoBOL MOVE rules don't allow HIGH-VALUES or
LOW-VALUES (or SPACES or QUOTES, for that matter) to be MOVEd to a PIC 99
(or any numeric or numeric-edited field, for that matter).

In order to obtain an out-of-band signal on a PIC 99, you have to move
LOW-VALUES to a group containing it, and it will then have the value
X'0000', null, which is easily distinguishable from X'F0F0', zero.
LOW-VALUES for PIC XX is also X'0000'.
```

```
I just started learning cobol a few weeks ago and today we got in touch
with high/low values.
You need them for merging files, for example you want to list all the
elemnts of 3 input files in one output file sorted ascending. You read
the first record of each file and compare them until youÂ´ve found the
lowest which you write to the output file. Then you read the next record
of that file, that contained the lowest. Normally these input files do
not have the same size, so you reach one fileÂ´s end while the others
still contain data. In that case you keep the last input record of that
file in the compareroutine and set its value to HIGH-VALUE (according to
the field length). EXAMPLE: If you have a 5-byte field the highest value
for an arithmetic expression will be 99999, which is F9 F9 F9 F9 F9 in
EBCDIC-code, HIGH-VALUE is encoded as FF FF FF FF FF, which is higher
than every 5-byte word. If you now continue comparing input records
errors will be avoided, because every field to be compared will be lower
than HIGH-VALUE. For descending merging you use LOW VALUE.
Please excuse my english, IÂ´m just learning all these technical terms :)
```
