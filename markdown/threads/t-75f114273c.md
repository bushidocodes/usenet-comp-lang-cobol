Actually, last night I was in the bookstore again, looking at all the Java
books. I'm really tempted to buy one of them, but I just can't seem
to bring myself to do it. Of course, all the stuff I hear about Java
is great. Sure it's maybe 10 or 15 times slower than C, or even COBOL,
.exe files, but it's so amazingly portable. In the end, five years from
now, maybe even sooner, it will run on top of any OS, and any hardware
platform.

But I still have those Visual C++ books around, that I bought last summer.
Sure, I took a close look at a few of them, even wrote a few basic Windows
applications... but I really don't feel I did everything with them that I
could of, or should of...

And now there's these Java books... despite how fabulous it looks, I'm really
having some trouble going up to the counter with one of them in my hands.
Any other COBOL programmers out there having these same types of troubles?

I guess I am going to wait until next week, or maybe the week after that.

As a C++ programmer working in a room full of C++ programmers, there was
great interest in Java when it first came out. I got the Jave in a
Nutshell book ... and have since given it to someone else. In many
respects, Java is a better C++ -- no operator overloading or pointer
arithmetic. What has turned some of us off is the performance of the
I/O library. When you hear 10-15 times worse than C++ I think that is
an average. A computer genius I work with who was thinking about
forming his own company to do Java was getting I/O performance a couple
of orders of magnitude slower when he tried to incorporate database
access into a CGI library. Even if performance is improved, the fatal
flaw in Java is that it is the Sun Microsystems attempt to take over the
internet. Very simply, Microsoft will not allow this and will crush it.
People are already jumping on the ActiveX bandwagon. Microsoft does not
have anything against PC COBOL because it knows that these applications
will increase usage of OLE and will be wrapped inside GUI and Internet
languages from other vendors, i.e. Micorosoft Visual Basic or ActiveX.
If you look at an index for PC Magazine this year, you will find an
article by John Dvorak detailing why Java won't happen.

As for portability, stick to an ANSI standard, isolate system
dependecies in independent, well documented modules and you've got
little reason to worry. I once ported 30,000 lines of FORTRAN77 from
VAX VMS to HP UNIX in two hours -- all I had to do was change the
delimiters in the INCLUDE statement, which I could do in two minutes
now that I know Unix tools like AWK.

Another reason not to be impressed by Java's portability is that in five
years, everyone who is not on a mainframe will be running NT, and those
who are on mainframes will still be using COBOL.

In conclusion, learn Java if you want to make some big money quickly,
but don't plan on a career using it.
