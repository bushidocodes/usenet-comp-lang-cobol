```
Are you using NT 3.51?

I ask this as it's a known bug in NT 3.51 and has been raised with Microsoft.

A DOS application that sets up an interrupt handler for the Control+Break
interrupt, INT 1bh, never receives control when you press Control+Break. This
works ok on Windows 3.1, 95 and NT3.50.

As far as I know, Microsoft haven't fixed this yet.

Of course, you could always try using the Windows triggers instead of the DOS
versions. eg COBOLW.EXE instead of COBOL.EXE. I have to ask why you're using
the DOS product still on NT rather than using the 32bit windows product. No
XM, no 16bit limitations....
```

```
This is probably because of the support for OS/2 applications in Windows NT.

In OS/2 mode, Ctrl+Break detection does not work.

To force your application to run in DOS mode, use either the FORCEDOS or
XM commands, e.g.

forcedos wb

or

xm wb
```
