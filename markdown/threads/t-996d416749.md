```
As I read the "statement of direction" at:

     http://www.ibm.com/isource/cgi-bin/goto?it=usa_annred&on=203-266

a "future" release of z/OS and z/OS.e (and hence LE) will DROP support for
"real" ISAM files.  This means (as I read it) that even existing OS/VS COBOL
programs that use "true" ISAM files (rather than ISAM language to access
VSAM files) will no longer WORK (or certainly will NOT be supported).

This may (finally) be a justification to "management" that is continuing to
use the OS/VS COBOL compiler (with or without the LE run-time) to do a
migration to a supported compiler.  Of course, I don't know how many shops
actually do still have ISAM files.  (My guess is not TOO many, or IBM
wouldn't have announced this "drop").

For sites with such OS/VS COBOL source code, I suggest you look at:


http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3mg10/4.1.4.1.1

However, I would guess that the following text will be CHANGED in the next
release of this manual,

"You can then run your application consisting of both OS/VS COBOL programs
and Enterprise COBOL programs under Language Environment. For details on
running existing programs under Language Environment, see:  ...


Note: This method is presented only as a short-term migration path. Because
OS/VS COBOL is no longer in service, you should rewrite your programs to
eliminate the dependency on OS/VS COBOL and ISAM. (If you do not mind using
an unsupported compiler, this method can still work for you.) "

***

Even if your site does NOT have any "true ISAM" files still in use, you
might want to point out this "change" to your management as an indication
that you really, REALLY need to have a migration plan to get off the OS/VS
COBOL compiler completely!!! (and probably the "VS COBOL II" and "IBM COBOL
for MVS and VM" and "IBM COBOL for OS/390 & VM" compilers as well).
```
