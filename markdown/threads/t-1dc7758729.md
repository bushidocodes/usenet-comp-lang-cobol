Copy and typedef are not equivalent styles, merely similar
within the context of a single program. They are a world
apart (at least in my solar system <g>) when used in the
context of an application consisting of many programs.

Application- or system-wide typedefs may be copied into
every program and, because they are only definitions, not
use one character of storage in the data division (2002,
TYPEDEF clause, GR(2)). The same can not be said of
an approach using nested COPY statements.

Typedefs defining group items are aligned the same as
record descriptions (2002, TYPE clause, GR(2d)). The
same can not be said of COPY statements, because the
COPY statement does not require alignment (2002,
COPY statement, GR(6)).

Because typedefs are not copy statements, there is an
opportunity to think differently about programming. One
can treat a typedef as something similar to an intrinsic type
by defining programs and functions to access and modify
the elements of the typedef. This promotes reuse of code;
this is not an idea that leaps into one's head with copy
statements, perhaps because copy statements, particularly
for files, sometimes contain several items that could be
typedefs.
