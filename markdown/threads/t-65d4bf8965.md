Your web site does not say that PCYACC can do lookaheads of more than 1.
And it is known that Cobol cannot be parsed correctly with lookahed of
only 1
(unless tricks and hacks are used).

Example:

PERFORM A OF B TIMES COMPUTE C = D + E END-PERFORM.

PERFORM A OF B COMPUTE C = D + E.

In the first line A OF B is data-item with qualified name.,
and this is block PERFORM statement.
In the second line A OF B is procedure-name (also qualified)
and this is a regular PERFORM procedure statement.

Parsing: when your parser reads tokens A, OF, B, it still too early to
tell
which PERFORM you are dealing with. Only when you see TIMES (or not
TIMES),
you can determine the type of statement.

How PCYACC Cobol grammar does distinguish between these two cases?
I guess, Cobol symbol table is not implemented (this is a grammar only,
right?), so symbol table hack will not apply here, right?

In my original posting I said that we are talking LALR(1) parsers here
and that we do not expect symbol table feedback. My guess is that
it would be somewhat unusual for PCYACC Cobol grammar to implement Cobol
symbol table -- that goes far beyond grammar per se. What is important,
if we are talking about Cobol grammar (grammar does not include symbol table)
then this grammar cannot be expressed with LALR(1).

And this is only one example -- not the most dramatic one.
There is a better example where even symbol table hack does not help:

Format 1: ADD A B TO C D
Format 2: ADD A B TO C GIVING D

Here formats 1 and 2 of ADD statement look the same
until we get to token D or GIVING. But conflict happens much earlier
-- after A B. So if we selected Format 1 branch after A B, then
if we receive GIVING token, we need to unparse tokens C and TO
and to reparse everything according to Format 2. LALR(1) parsers
cannot do that -- they cannot rollback, they need to make a parsing
decision based on lookahead of not more than 1 token.

Best regards,
Vadim Maslov

There is nothing that says you have to make your grammar productions
match the ANSI X3.23 format rules. A bit of grammar refactoring (or
defactoring) fixes this problem.

On the other hand it is certainly true that COBOL is neither regular,
context-free, nor LR(k) for any k. A more valid example of the latter is
the READ statement, whose syntax depends on the organization of the file
being read. This affects whether AT END or INVALID KEY form part of the
statement or part of an enclosing statement. PICTURE strings and the
lexical elements of the COPY statement are examples of the first 2.
