Hi Gary,

As you will learn, or probably have already learned, there's usually more than
one solution to every programming problem. So, here goes.

You can also code your input/output data descriptions in WS rather than the FD.

But when you read/write the rec you use into/from:

read ip-file into ws-ip-rec
write op-rec  from ws-op-rec

In this case, when you define the rec in the FD you usually code it:

01  fd-ip-rec         pic x(nnn).

There's no need for detail because you've provided it in the WS rec description.


There are advantages/disadvantages to this approach:

You can access the data in WS before the file is opened and after it's closed or
reaches EOF; in certain stuations this can be helpful.

A move to the FD rec to/from the WS rec is generated by the compiler, making the
I/O less efficient.   

Now, aren't you sorry you asked? ;-) Good luck.

Jack

This answer really bothers me.  Your buffers (or other file I/O areas) should be in the dump.  Having trouble finding the last record read or written is NOT a professional excuse. The draft has been fixed.

30 years ago, using the ICL 1900 XE13 compiler (or possibly XE20 or
XEKB) I discovered a _very_ interesting result when updating UDAS ISAM
files.  Normally when a WRITE fails one would expect that the record in
the record area could be output to some other recovery file and this
merged back in later.  However, due to the way the overflow system
worked, it was actually possible that the record being written _did_ get
put into the file and that the failure occurred as it was moving some
other completely different record into overflow, a record that may have
been in the file for some time, and was being held in the record buffer
while the block buffers were swapped (this was a 16Kword multitasking
machine).

(description of UDAS ISAM files with first level and second level
overflow and how this problem could occur available on request, please
apply in writing on the back of a signed blank cheque).

It was only when the write failed by attempting to displace an existing
record into an already full overflow that this was actually a problem,
but after a successful or failed WRITE the record area may have had
_any_ value: some other record, or overflow tags.

Other than that, and that it had less grunt than my watch, it was a nice
system.

If you're writing a file, it doesn't make any difference anymore, although lots
of shops still have a standard of using working storage, and that's what I use.
If you're writing a report, almost everybody uses working storage.   I don't use
Report Writer, so here's an example of creating output to a report.

01   R1-HEADER1.
      05   FILLER        PIC X(10)    VALUE SPACES.
      05   FILLER        PIC X(12)    VALUE '    GROSS'.
      05   FILLER        PIC X(05)    VALUE SPACES.
      05   FILLER        PIC X(12)     VALUE '     TAX'.
      05   FILLER        PIC X(10)     VALUE SPACES.

01   R1-HEADER2.
      05   FILLER        PIC X(10)    VALUE SPACES.
      05   FILLER        PIC X(12)    VALUE '     PAY '.
      05   FILLER        PIC X(05)    VALUE SPACES.
      05   FILLER        PIC X(12)     VALUE '   AMOUNT'.
      05   FILLER        PIC X(10)     VALUE SPACES.

01   R1-DETAIL.
      05   FILLER        PIC X(10)    VALUE SPACES.
      05   R1-GROSS-PAY    PIC   $,$$$,$$$.99.
      05   FILLER        PIC X(05)    VALUE SPACES.
      05   R1-TAX-AMOUNT  PIC $,$$$,$$$.99.
      05   FILLER        PIC X(10)     VALUE SPACES.

01  LINE-COUNT      PIC 999        VALUE 100.

...

MOVE GROSS-PAY      TO R1-GROSS-PAY.
MOVE TAX-AMOUNT TO R1-TAX-AMOUNT.

IF LINE-COUNT > 66
      WRITE OUTPUT-RECORD FROM R1-HEADING1 AFTER ADVANCING PAGE
      WRITE OUTPUT-RECORD FROM R1-HEADING2 AFTER ADVANCING 1
      MOVE SPACES TO OUTPUT-RECORD
      WRITE OUTPUT-RECORD AFTER ADVANCING 1
      MOVE 3        TO LINE-COUNT
END-IF

ADD 1        TO LINE-COUNT.
WRITE OUTPUT-RECORD FROM R1-DETAIL AFTER ADVANCING 1.


Note, if you only have one format for the output line, the 05 levels could just
as easily have been in the File description.  (At one time this caused
troubles).
