COBOL wouldn't care.  When CALL is exectued, program control goes to the
called program.

In a OS390 environment, there are many ways to CALL an external program.
US7VC and REMCMND come to mind.  Even an FTPed file can cause an external
program to 'tap off' from the scheduler software.

So it kinda depends on what you are talking about exactly.

Well, the short answer is that in COBOL "CALL" is used to load (if
necessary) and execute pre-compiled code which was *designed* to be CALLed
rather than executed 'standalone.'

That is, a 'regular' program designed to be executed from the command-line
is likely not a suitable object for a COBOL 'CALL' verb.

Most COBOL systems have a separate mechanism to "Execute an external
'regular' program and return".

In either case - CALLed or 'executed' - the (sub-)program can be written in
any language.
