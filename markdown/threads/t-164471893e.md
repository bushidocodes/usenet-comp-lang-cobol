```
Laura,

you are on the right track.

There are two things you need to find out:

1. How to interface to the ActiveX/COM object using ACUCobol.

 I do this all the time with Fujitsu, but that is an Object Oriented COBOL
compiler and it supports COM and ActiveX by means of a special Class (*COM),
which makes everything very easy. I don't know whether ACUCobol supports OO
or not. I'm sure you'll find out pretty quickly.... I'd be interested to see
how they deal with COM objects if they DON'T support OO...

2. You need to find out something about the Excel Object Model. Basically,
it's Methods, Properties, and Events. This is downloadable from MSDN or the
Knowledge Base. This will tell you how to create a graph, read the .CSV or
whatever activity you want Excel to do.

Equipped with both of the above, it is pretty easy to create a new instance
of the ActiveX object and invoke Excel's Methods with it. You need the COBOL
"INVOKE" verb and I believe it is only available for OO.

Please let us know how you get on with solving this and post the code here
if you get stuck.

Pete.
```

```
Here is a small example, provided you have used AxDefgen.exe to generate the
copy book.

Note, this does not fully complete your request. Have to leave some work for
you as well.

Also, if I were you, I would skip the idea of a temporary storage to a cvs
file all together and rather transfer the date directly to Excel using
SAFEARRAY.

       IDENTIFICATION DIVISION.
       PROGRAM-ID. ExcelSimple.
      *====================
      *
      * Copyright (c) 1996-2001 by Acucorp, Inc.  Users of ACUCOBOL
      * may freely modify and redistribute this program.
      *
      * The purpose of this application is to show the absolute minimum
      * required to access a cell in a Microsoft Excel spreadsheet.
      *
       ENVIRONMENT DIVISION.
       CONFIGURATION                SECTION.
       SPECIAL-NAMES.
           COPY    "EXCEL.def".
                   .
       DATA        DIVISION.
       WORKING-STORAGE              SECTION.
       77  olExcel                  HANDLE OF APPLICATION.
       77  olWrkBk                  HANDLE OF WORKBOOK.
       77  olWrkSh                  HANDLE OF WORKSHEET.
       77  MY-STRING                PIC X(80).

       PROCEDURE DIVISION.
       Main.
           CREATE  Application      OF Excel
                   HANDLE           IN olExcel.
      *This is for training purposes only. If you don't want to see
      *Excel, remove this line.
           MODIFY  olExcel          @Visible = 1.
           MODIFY  olExcel          Workbooks::Add()
                   GIVING           olWrkBk.
           INQUIRE olWrkBk          Worksheets::Item(1) IN
                   olWrkSh.
      *Set the value of a cell.
           MODIFY  olWrkSh          Range("A1")::Value = "Last Name".
      *Get the value of a cell.
           INQUIRE olWrkSh          Range("A1")::Value IN MY-STRING.
           DESTROY olWrkSh.
      *Enforce a close without save, or Excel will prompt you.
           MODIFY  olWrkBk          @Close(BY NAME SaveChanges 0).
           DESTROY olWrkBk.
           MODIFY  olExcel          Quit().
      *This next line is very important, otherwise Excel will stick in
      *memory.
           DESTROY olExcel.
           GOBACK.
```

```
> It would appear that ACUCorp have extended COBOL to provide these
> facilities?

Yes, we have gone beyond the standard to some degree, but we feel that we
have managed to accomplish so in a way that both maintains the traditional
COBOL objectives and at the same time leaves the language dynamics pretty
close up to other languages like VB, Delphi, Java.

> Does AxDefgen generate an interface from any COM compliant software?

If it is an ActiveX/COM object that can be implemented using VB or Delphi,
we support it with a few constraints. These relates to event techniques (no
support for VFTBL, the object must support event sink) and late binding
(like for instance Open Office).

> MODIFY, INQUIRE, DESTROY look like COBOL extensions to provide equivalent
> functionality as that provided by the  Fujitsu *COM GET, SET, and  "set
> pointer to NULL" functions.

Probably, I am not familiar with Fujitsy, though, DESTROY in ACUCOBOL-GT
decrements the counter with the COM interface, the user himself must
actually set the pointer to 0. Having said that, I believe this behavior is
also what one experiences with VB and their use of SET object = 0, decrement
interface pointer that is.

    How is Fujitsu referencing named parameters to methods?
        Acu: MODIFY MyObject MyMethod(BY NAME parameter1 "parametervalue")
    Can you enforce a variant datatype with Fujitsu?
        Acu: MODIFY MyObject MyData = coboldata AS VT_BSTR
```
