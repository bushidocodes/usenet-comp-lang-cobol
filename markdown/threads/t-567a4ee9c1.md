```
I have a rather unusual situation that perhaps someone here has a solution
for.  The platform is Windows, and the compiler is Fujitsu. The machine is
dedicated ... that is, it is running a Cobol system that starts on boot, and
the user never sees the OS.

I am receiving a file that is print ready. I have no idea how it is formed,
or where it came from.  It arrives over the net, and all I know is that it
is HP4 laser compatible, and that the default printer on my computer is also
HP compatible.  I do not know where that printer is, or what it is called.
I only know that the default printer exists. Currently, I print to it using
Flexus FormPrint.  Now I have to print this foreign file as an appendum to
the existing reports.

If I knew, for example, that the default printer was mapped to a port, or I
knew the net name of the default printer, I could do a simple file copy.  I
know neither. If I select a file and assign to "PRINTER", it will default to
the default printer, but it also changes the file type to line sequential,
which will not work.  The incoming file is binary, and contains graphics.
The insertion of CR's and LF's will create havoc ... to do it in Cobol I
have to do it byte by byte.

So:  1. any suggestions?
or
2.  Is there an API call that will give me the name of the default printer
that I can use for a copy?

If anybody has a solution, I would appreciate it.  It will save me a day or
two of experimentation.
```

```
The printer is commonly mapped to LPT1 even if it is a network printer
so you simply copy the file to LPT1:.  This does not always work with
NT because network support do not always set it up this way.

If you are NT and network printers you can copy to the actual printer:
	COPY print.file \\server\printername

As noted the /b might be required, I have never used it though.  Now
if you combine this with Oscars T's suggestion to find the device name
then you might be away.

You should be able to use '\\server\printer' as an output filename as
well but it will rely on a recent and supporting compiler (which I am
sure you have).
```

```
Fujitsu has a solution to that problem.  It is a program called
"printit.exe" that you link and call from within the Fujitsu program
(either cobol or Powercobol).

Works fine with networked printers, as long as the desired printer
is set as the "default" printer on that workstation.

The intended use is for the cobol program to write to a temp file,
and printit.exe copies that file directly to the printer port.  There is
no reason it couldn't be used to copy the file you have, which is
already print ready.

We use it to print forms in a dot-matrix printer, in a car dealership.
Registration applications, finance contracts, etc.

The alternative was to use Fujitsu PowerForm, which works well
with laser printers, but not well with dot-matrix (which was required,
because the final result has carbons).  PowerForm insists on printing
at the dot-level.  We wanted to print at the character-level, which
of course is much faster on dot-matrix printers.

Fujitsu can email you the printit.exe package, but I caution the
setup is not easy to figure out the first time.  Once you have it,
however, it is easy to customize.

don ferrario
http://www.ferrario.com/don
```
