When C programmers complain about a lack of pointers they
usually mean the ability to use and manipulate low level
memory addresses directly.

That is they have been taught that, for example, it is much
more efficient to step along a character array by incrementing
a pointer than to use an index.  As they learnt this in the
'Advanced C' course they think that is an advanced language
feature (and thus Cobol is not an 'advanced' language), when
in fact it is the programmer that must be 'advanced' enough
to cope with low-level grovelling in the bits.

Modern high-level languages (which eliminates C on two counts)
generally avoid pointers because they make it difficult to
have reliable working programs.

Though object references are memory addresses (they need not be)
they are not able to be manipulated in any way and thus are
not pointers as in C.

High level languages leave the details to the compiler.  Thus people who
know the environment can write the compiler to optimize without having
to guess too much what exactly the programmer intended.

It is usually safe to optimize a single command as much as we want.  In
Cobol that single command can be a COMPUTE.  In C it is more likely to
be an increment.  A COMPUTE is complex enough that there is plenty of
optimizing available.  You can't optimize an increment.

Sure a top programmer will be able to optimize better than the compiler
- knowing precisely the specifics of his needs.  But shops are full of
average programmers whose job is to make workable and maintainable code
without obscure tricks.  So put the optimization in the background.

Java is cheaper to maintain than C, partially because memory leaks are
hard to debug.  Let someone else design a way to clean up memory - sure
it will cost some in run time efficiency - but which is more expensive,
run time costs, or maintenance costs?

The problem is not pointers, it is dynamic memory allocation.

COBOL uses pointers for parameters to CALLed programs.
We just don't call them pointers. COBOL works reliably with
these pointers because the memory is statically allocated in
the CALLing program.

C programs could be written to behave as COBOL programs
by using static allocation; but it is far more common, in C, to
use dynamic allocation.

In any case, it is possible to have very reliable programs with
dynamic memory allocation, if one uses accessor functions
to handle the allocating and freeing of memory and to return
pointers, of the appropriate type, to allocated data.

by OOCobol.
