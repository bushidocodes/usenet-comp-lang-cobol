```
Peter Newman <newmanpj@nospamibm.net> wrote:
: I've heard said that one should use PERFORM THRU for one or more paragraphs
: instead one or more PERFORM section-name(s). Why? Isn't it merely matter of
: convention or standards? I would have thought that PERFORM THRU was more
: error prone. The compiler we're using is COBOL for MVS & VM. Sincerely
: appreciate any direction on this.

Both of these can be prone to a certain type of error which is
complete loss of control.  This is because both PERFORM section
and PERFORM THRU are done primarily to allow a paragraph name or
names within the performed scope to allow a GO TO to jump over
code or to get to an EXIT paragraph.

That is, if there is no such GO TO then there is no need to
PERFORM THRU or to PERFORM a section, PERFORMing a paragraph
works well.

Where there is a GO TO in the performed scope it is potentially
that the GO TO may be incorrectly coded, or the Exit-label may
be wrong.  This, of course, can be checked by processing the
source with a special program as Judson will tell you, but it
is not detected by the compiler and can give exceptionally
subtle bugs because the program may do strange things including
_almost_ working correctly.

PERFORM section also has a potential problem in that it is
terminated by the presence of the next section header.  If
this header is missed (ie the label is just a paragraph)
then the perform does not stop where expected.

It is best to program as per the site standards as this is
what the next programmer expects.  However, I have found
that just performing paragraphs (no thru, no section) and
thus no GO TO, is a much more reliable coding structure.
Others may find different results, especially if they have
invested in special checking programs that eliminate the
potential problems.
```

```
You are correct that it would return - but ONLY in the case that
"Some-Condition" is false (in other words, the IF fails - so no GO TO is
performed).  If, HOWEVER, "Some Condition" is true - and the GO TO is
performed, then you will definitely NOT return from the PERFORM.  Try it and
you will see that it does work this way.

Now, there is a separate issue of whether the NEXT time you come thru the end
of the PERFORMed paragraph you do or do not "return" - no matter how you get
there.  This is where "stack-based" versus "perform-bucket" implementation is
used.  Therefore, if you had a "Go To BBB" at the end of the CCC paragraph -
some but NOT ALL - implementations would return back to the MAINLINE at the
end of the BBB paragraph.
```

```
This is an area of "religious wars" that pop up every few months in
comp.lang.cobol.  People who are used to (like? advocate?) one approach often
can't even see the possibility of using other techniques.  In general,
however, I think that most "posters" do NOT like the use of PERFORM anything
THRU something-else - unless the "something-else" is an EXIT paragraph for
"anything" and there are no paragraph (or section) names in-between.  If you
have multiple paragraphs that you want performed in sequence - you do NOT
need to use the THRU option.  Instead, you can

A) perform a single paragraph - that in turn PERFORMs the individual
paragraphs (in sequence)

B) just put the PERFORMs of the individual paragraphs in sequence - where you
"thought about" using the PERFORM THRU.

  ***

Either of these techniques are significantly less "error-prone" than using
PERFORM/THRU - especially if you (or some future maintenance person) happens
to have a "GO TO" in the middle of the PERFORMed paragraphs.

For those who do advocate the use of SECTIONs instead of paragraphs (to be
performed), it is common (although not universal) to recommend AGAINST
putting multiple paragraphs (other than an EXIT paragraph) in each SECTION.
```
