### Parsing a Common Log Record Without Unstring

- **From:** psychedelic-harry@mindless.com (Joe Zitzelberger)
- **Date:** 2003-04-05T08:07:41-08:00

- **Message-ID:** `<16e2f010.0304050807.28606261@posting.google.com>`

```
Just for fun I thought I'd hand code a parser to extract http common
log records into a usable form.  When considering the Common Log
Format, it becomes a bit unwieldy, If not downright hard to deal with
it using UNSTRING.  I thought I'd approach it in a slightly dIfferent
way.

I've only included a skeletal working storage/linkage description and
error checking is fully fleshed out, but it might be a useful example
for someone wanting to avoid multiple nested UNSTRINGS.

To do this easily I start with a simple grammer.  This lets each part
then be coded in isolation and makes it easy to quickly build a parser
for a complex grammer.

The format of the common http log file line has the following fields
separated by a space:

remotehost: 151.99.190.27
rfc931 remote login name (or - If unavailable): -
authenticated user name (or - If unavailable): -
[date:time]: [01/Jan/1997:13:07:21-0600]
"request line as exactly from client": "GET /~bacuslab/celsheet.html
HTTP/1.0"
return status: 200
bytes transfered: 13276

This gives us the following productions:

space => ' '+

string  =>  nonspace-char*

quoted-string  =>  '"' char* '"'

int  =>  [0-9]+

ip  =>  int{1-3}.int{1-3}.int{1-3}.int{1-3}

datetime  =>  '[' int{2} '/' month '/' int{4} (':'int{2}){3} sign
int{4} ']'


and finally we can complete our grammer with:

rhost   => ip
rlogin  => string
authusr => string
request => quoted-string
status  => int
xfered  => int
 
common-log => rhost space rlogin space authusr space datetime space
request space status space xfered


--- Working Storage Section ---
01 Log-Data.
	05 Log-Rhost           Binary Pic S9(8) Value 0.
	05 Log-Status          Binary Pic S9(8) Value 0.
	05 Log-Xfered         Binary Pic S9(8) Value 0.
	05 Log-Rlogin		                   Pic X(32) Value spaces.
	05 Log-Authusr                   Pic X(32) Value spaces.
	05 Log-Request-LL Binary Pic S9(4) Value 0.
	05 Log-Request                  Pic X(32767) Value spaces.

--- Linkage Section ---
01 Next-Char-Area.
	05 Next-String                     Pic X(lots).
	05 Next-Char redefines Next-String Pic X.
	05 Next-Char-9 redefines Next-Char Pic 9.

--- Procedure Division ---

Parse-Common-Log.
	Parse-rhost
	Parse-space
	Parse-rlogin 
	Parse-space
	Parse-authusr
	Parse-space
	Parse-datetime
	Parse-space
	Parse-request
	Parse-space
	Parse-status
	Parse-space
	Parse-xfered
	Exit.

Parse-space.
	Perform until next-char not = space
	   Perform Get-Next-Char
	End-Perform
	Exit.

Parse-rhost.
	Perform Parse-ip
	Move wk-ip to log-rhost
	Exit.

Parse-rlogin.
	Perform Parse-string
	Move wk-string to log-rlogin
	Exit.

Parse-authusr.
	Perform Parse-string
	Move wk-string to log-authusr
	Exit.

Parse-datetime.
	Perform Get-Next-Char
	If next-char = '['
	   Perform Get-Next-Char
	Else
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Parse-int
	If wk-int > 0 and wk-int < 32
	   Move wk-int to log-dttm-day
	Else
	   Move 'Invalid Day' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	
	Perform Get-Next-Char
	If next-char = '/'
	   Perform Get-Next-Char
	Else 
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Evaluate next-string (1:3)
	   when 'JAN'
	      Move 1 to log-dttm-month
	   when 'FEB'
	      Move 2 to log-dttm-month
	   when 'MAR'
	   ... snip to save space, you get the idea...
	   when other
	      Perform 123456789-fatal-error
	End-Evaluate
	Perform Get-Next-Char 3 times

	Perform Get-Next-Char
	If next-char = '/'
	   Perform Get-Next-Char
	Else 
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Parse-int
	If wk-int > 1989 and wk-int < 9999
	   Move wk-int to log-dttm-year
	Else
	   Move 'Invalid Year' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	
	Perform Get-Next-Char
	If next-char = ':'
	   Perform Get-Next-Char
	Else 
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	Perform Parse-int
	If wk-int < 25
	   Move wk-int to log-dttm-hour
	Else
	   Move 'Invalid Hours' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	
	Perform Get-Next-Char
	If next-char = ':'
	   Perform Get-Next-Char
	Else 
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	Perform Parse-int
	If wk-int < 60
	   Move wk-int to log-dttm-minute
	Else
	   Move 'Invalid Minutes' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Get-Next-Char
	If next-char = ':'
	   Perform Get-Next-Char
	Else 
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	Perform Parse-int
	If wk-int < 60
	   Move wk-int to log-dttm-second
	Else
	   Move 'Invalid Second' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Get-Next-Char
	Evaluate next-char
	   when '-'
	      Set log-dttm-off-neg to true
	   when '+'
	      Set log-dttm-off-pos to true
	   when other
	      Perform 123456789-fatal-error
	End-Evaluate

	Perform Parse-int
	If wk-int > 0 and wk-int < 2500
	   Move wk-int to log-dttm-day
	Else
	   Move 'Invalid Day' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Get-Next-Char
	If next-char = ']'
	   Perform Get-Next-Char
	Else
	   Move 'Invalid Date' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Exit.

Parse-request.
	Perform Parse-Quoted-String
	Move wk-quoted-string to log-request
	Move wk-quoted-string-ll to log-request-ll
	Exit.

Parse-status.
	Perform Parse-Int
	Move wk-int to log-status
	Exit.

Parse-xfered.
	Perform Parse-Int
	Move wk-int to log-xfered
	Exit.

Parse-string.
	Perform varying I from 1 by 1 
	  until next-char not nonspacechar
	   Move next-char to wk-string (I:1)
	   Perform Get-Next-Char
	End-Perform
	Exit.

Parse-IP.
	Perform Parse-int
	If wk-int > 256
	   Move wk-int to log-IP-1st-byte
	Else
	   Move 'Invalid 1st Byte' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Parse-int
	If wk-int > 256
	   Move wk-int to log-IP-2nd-byte
	Else
	   Move 'Invalid 2nd Byte' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Parse-int
	If wk-int > 256
	   Move wk-int to log-IP-3rd-byte
	Else
	   Move 'Invalid 3rd Byte' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If

	Perform Parse-int
	If wk-int > 256
	   Move wk-int to log-IP-4th-byte
	Else
	   Move 'Invalid 4th Byte' to fatal-error-string
	   Perform 123456789-fatal-error
	End-If
	Exit.

Parse-quoted-string.
	If next-char = '"'
	   Perform Get-Next-Char
	   Perform varying I from 1 by 1 
	     until next-char = '"'
	      Move next-char to wk-quoted-string (I:1)
	      Perform Get-Next-Char
	   End-Perform
	   Perform Get-Next-Char
	   Move I to wk-quoted-string-ll
	End-If
	Exit.

Parse-Int.
	Move 0 to wk-int
	Perform until next-char not integerdigit
	   Compute wk-int = (wk-int * 10) + next-char-9
	   Perform Get-Next-Char
	End-Perform
	Exit.

Get-Next-Char.
	Set address of next-char-area to next-str (2:1)
	...some range checking would be nice...
	Exit.

Get-First-Char.
	Read LOGFILE 
	   at end Set end-of-log-file to true
	   not at end Set address of next-char-area to address of
logfile-buffer
	End-Read
	Exit.

123456789-fatal-error.
	Display "Invalid Common Log Format, error is ("
	   fatal-error-string
	   ")."
	...do whatever is needed to recover...
	Exit.
```

If you're trying to mimic Corporate COBOL, you'll have to make a few changes. 

First, every Corporate Coder knows that EXIT PROGRAM should be the last line in
the program. Put it at the bottom and perform it. For belt-and-suspenders
security, put a STOP RUN after it. 

Second, all those data names should be prefixed with ws- or ls-. How is the
reader supposed to know which section they're in? Alternatively, use Hungarian
Notation prefixes so the reader can tell at a glance what the picture is. For
best clarity, use both. Example: ws-is-wk-int.

Third, don't be so verbose. Are you writing a book? For example,
fatal-error-string could be abbreviated ws-as-fes. I assume it's in
working-storage. Surely you're going to return it to the caller. 

Fourth, all those EXITs won't compile. Change them to CONTINUE. While you're at
it, put the period on a line by itself. Your boss will be happy when he measures
your productivity by line count. 

Fifth, you might consider 'inlining' those re-usable paragraphs such as
get-next-char. Copy-and-paste is a wonderful tool in cases like this. Why? Two
reasons: your program will run faster and, if someone changes the paragraph, you
won't have to re-test the entire program.

I like the way you avoided reusable code in most of the program. Simple,
straight line, reads from top to bottom. That's good, clean style.
