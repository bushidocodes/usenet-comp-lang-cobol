Okay, here is the skinny, and why it worked.

The data that you are really writing is a stream of bytes consisting of four
types of byte.  You have printable bytes (letters spaces, etc.), plus three
distinct control characters ... CR, LF, and FF.

A carriage return <CR>(0Ah) causes the printer to start a line at the left
edge, but does NOT advance to a new line.  Used by itself at the end of a
line, it will cause the printer to overprint the next line. In Cobol you can
generally get it by using the "PRINT AFTER ZERO" statement. It can be used,
for example, for underlining or hashing out a password.

The <LF>(0Dh) will cause to printer to advance to a new line, but NOT
restart at the rightmost edge. The next line will start at the column that
the previous one ended at.

In general, a line sequential file printed without ADVANCING clauses will
terminate each line with a <CR,LF> pair.  This tends to be the same format
followed by standard programming editors, which makes the file format a nice
one to use when you want to see your output on the screen, or check it with
an editor.

To get to the top of a new page, however, you have to insert a <FF>
character. The <FF> character causes a printer to go to the top of the next
page, or with a laser, causes the page to be printed and ejected. (You can,
BTW, <HOME> the cursor on most lasers and overprint the entire page, but
that is a separate topic <G>).

Where do you insert it?  If you insert it AFTER the last line, then the last
line of the previous page will not have a <CR,LF> at the end, and this often
causes problems with a printer. For example, you might end up with your
header starting in column 80, because that is where the print head was after
the last line of the previous page was printed.

Printing it BEFORE printing the header leaves the header with no <CR,LF> at
its end.  Now you have problems with the second line of the header being
concatenated with the first line.

The solution which you found works quite well. Put the <FF> on a line by
itself.
