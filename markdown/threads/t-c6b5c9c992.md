James,

You certainly have hit the nail on the head with the long note (6
printed pages impressive).

1.  Sorting:  Internal sorts are fine in MVS.  We generally choose to
separate them because they are easier to unit test and separating
large programs into smaller ones makes them easier to write.  If
portability were more important then I would consider internal sorts.

2.  Method names.  Cobol will need to create it's own Standard
Template Library like HP did for C++ as public domain.  The same
problem was occurring for C++ but after STL it quickly drew into
line.  The multiple visual interface libraries is a huge problem still
in C++, this would be our chance to one up them.  (Bob I don't expect
to much support for the visual library ;->)

Judson,

I agree with your comments about the browser interface.  I knew there
was something wrong with the picture but could not accurately describe
the problem, thanks.

... general ...

I have been following a thread about MVC (Model View and Controller)
on comp.object.  This solution has some flaws and I am trying to fully
understand the issues but we must nut this one out.

Basically MVC is how you put the business logic with a screen and
attach that to an input device.  I am getting head aches from the
descriptions I am getting.  One tutorial was one paragraph and
simplistic the other described how to do it in smalltalk without
actually describing what it all means.

Who is volunteering for the common STL for cobol?
Ken
