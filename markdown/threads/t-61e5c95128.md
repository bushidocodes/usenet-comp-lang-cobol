Well, in the IBM mainframe world, if your picture is S9(11)V99 and your
value is +12345678901.23, then you would have a 7-byte field (at least
if the USAGE is COMP-3 or PACKED-DECIMAL).

Those seven bytes would contain the following hexadecimal values:
12 34 56 78 90 12 3C

IBM mainframes are "big-endian". The 'C' in the rightmost nybble is the
normal value for a positive sign. 'F' would be positive but unsigned,
and 'D' would be signed negative.

If, on the other hand, the USAGE of the field is DISPLAY, then you would
have a 14-byte field, and the EBCDIC hexadecimal contents of the field
would be:

F1 F2 F3 F4 F5 F6 F7 F8 F9 F0 F1 F2 C3

This time the left nybble of the rightmost byte contains the sign.
Again, the sign nybbles are C for positive, F for unsigned positive, and
D for negative.

I hope this helps.

This is, unfortunately, very true. You can prevent this kind of behavior
by compiling the program with either the NUMPROC(MIG) or NUMPROC(NOPFD)
options, which will force the compiler to generate CP (Compare Packed)
instructions instead of CLC (Compare Logical Character) instructions.
Then you will get an equal condition when comparing an "unsigned" zero to
a "positive" zero.

IBM, of course, recommends the NUMPROC(PFD) option because CLC
instuctions are "faster". But I don't want the answer faster if the
answer is going to be wrong!
