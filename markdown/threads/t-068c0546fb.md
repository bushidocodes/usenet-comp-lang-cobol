In a MicroFocus environment, NUMERIC testing can only be done on an
alphanumeric field since the runtime tests a numeric field on numeric
contents before accessing it (runtime error 163).

This test can be disabled with the -F switch, but as a nonnumeric
value in a numeric field almost always implies a logical error in the
program, I don't use this switch in my developming environment.
Therefore I test for the NUMERIC condition on alphanumeric fields
only.

I do it this way to avoid a crash.

01 GROUP-ITEM.
     05 NUM-ITEM  PIC 9(8).

MOVE "ABCDEFGH" TO GROUP-ITEM
IF NUM-ITEM NOT NUMERIC
    DISPLAY "Bingo"
END-IF

With numeric checking in the runtime system enabled (+F) the program
will crash in the IF-sentence with error 163 (non-numeric data in
numeric field).

That's why I code 

IF GROUP-ITEM NOT NUMERIC 

or

IF NUM-ITEM(1:8) NOT NUMERIC

This way, I can do explicit numeric checking on items that are more
likely to be non-numeric (for example incorrectly filled incoming
interface files) and handle these errors and can leave the runtime
checking enabled so I can see if I have miscoded something.

Sven Brueggemann wrote:

> In a MicroFocus environment, NUMERIC testing can only be done on an
> alphanumeric field since the runtime tests a numeric field on numeric
> contents before accessing it (runtime error 163).

This is something which should be fixed in the CoBOL standards.  All
numeric testing should be done against the PIC 9 fields.  The compiler
should be smart enough to use the picture to determine whether the data are
correct.   The following code is ugly:

10   BILL-TOTAL-ALPHA   PIC X(8).
10   BILL-TOTAL       REDEFINES BILL-TOTAL-ALPHA PIC S9(7)V99.

...
IF BILL-TOTAL-ALPHA (1:7) IS NUMERIC

Signed numeric, packed, floating, etc.. numbers can all redefine the same
space, we can't guess correctly which redefines you are really checking for
by checking an alphanumeric field for IF NUMERIC, when what you really want
is to avoid a data exception when you do arithmetic.  So the standard
SHOULD be for a way to test the numeric field for validity.   Some
compilers do this, others do not.
