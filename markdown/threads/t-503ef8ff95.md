The main application for PROCEDURE-POINTER in COBOL is to support Language
Environment (LE) condition handlers.

Subsequently, you can also use it, as some have suggested, to simply hold the
address of a routine to invoke (perhaps because you need to pass this as a
parameter). You can also use a procedure-pointer as the object of a CALL
statement, so that you could be passed a parameter containing the address of a
subroutine to invoke. This means the routine you are calling doesn't even need
to be a CSECT or ENTRY point, if you are very careful.

If you are interested in more information about LE, check out our website
(http://www.trainersfriend.com), or even call us in for a class.

I know of one problem that AFAIK can only solved with PROCEDURE-POINTER:
Call a DLL from a non-DLL program. There is a sample in chapter '4.2.2.1
CALLs to DLLs from non-DLLs' in the 'Programming Guide' of the current
'IBM COBOL for OS/390 & VM'. The non-DLL COBOL program (99.999% of all
COBOLs I guess...) calls a C which loads the DLL (written in C, C++,
PL/I or COBOL) and returns a pointer to the entry within the DLL. The
COBOL program then does:

     SET DLL-PROGRAM-PTR TO DLL-PROGRAM-HANDLE
*
*    Call the program in the DLL.
     CALL DLL-PROGRAM-PTR

Sounds very exotic, but I am afraid it isn't. The new 'VA PL/I for
OS/390 & VM' generates DLLs by default (cannot be changed), the C/C++
and the COBOL compiler can generate DLL if you set 'DLL' as a parm. At
least the C programmers will use it certainly.

So we will soon have to migrate... with PROCEDURE-POINTER...
