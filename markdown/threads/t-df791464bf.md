If you're still on OS/VS COBOL (ANSI-74) the effort to convert your
source programs to COBOL II or COBOL for MVS and VM is almost
identical. If you're fortunate enough to have a working COBOL II
program, it will compile clean under COBOL/MVS with no source code
changes.

The special register CURRENT-DATE is gone, and the COBOL/MVS FUNCTION
CURRENT-DATE is completely different (and Y2K compliant). ACCEPT ...
FROM DATE works identically in both COBOL II and COBOL/MVS but only
gives you YYMMDD. IGZEDT4 works the same in both COBOL II and COBOL/MVS
and gives you the system date in YYYYMMDD format.

The special register TIME-OF-DAY is also gone. ACCEPT ... FROM TIME
works the same in both COBOL II and COBOL/MVS.

EXAMINE must be converted to INSPECT. The syntax is slightly different
on some of the options. TRANSFORM is gone forever, but INSPECT can
cover most its functions.

The NOTE verb goes away completely. It has some odd behaviors that make
automatic conversion somewhat difficult.

There's some real differences in the FILE SECTION. Too many for me to
remember and describe here. Most of the weird old stuff here is either
a syntax error or treated as comments. You can omit your LABEL RECORDS
clause.

If you code CICS programs and use the linkage section, get used to SET
ADDRESS OF linkage-section-record TO pointer-name, because it replaces
all your SERVICE RELOAD's. And don't forget to move you CICS BLL cell
pointers from your Linkage section to working-storage.

There's some differences in the behavior of PERFORM ... VARYING VARYING
(if you're looping through multi-dimensional tables). Be careful here.

Replace every occurrence of NEXT SENTENCE with CONTINUE. It will save
you a lot of grief when you start using the explicit scope terminators
(END-IF, END-READ, END-WRITE, END-COMPUTE, END-SEARCH, END-EVALUATE, et
cetera).

Learn how to use Reference Modification. It can sure come in handy.
Just don't overdo it. Be on alert for problems with the compiler
options NUMPROC and TRUNC, they can be very confusing. I recommend
NUMPROC(NOPFD) and TRUNC(OPT). Not everyone will agree with me, but I
think you'll have fewer problems with "unexpected results".

EVALUATE is an extremely powerful addition to the standard. Start out
with simple case structures. You can always add the ALSO verb later.

Start using END-IF. There are a lot of ways it can simplify your
conditional logic.

Don't panic. ANSI-85 (COBOL II and COBOL/MVS) is far superior to
ANSI-74. Once you get comfortable with it, you'll never want to go
back.

Please do NOT automatically convert NEXT SENTENCE to CONTINUE - without
knowing the differences between the two. NEXT SENTENCE *always* goes to the
verb that (logically or physically) follows the next period. CONTINUE
causes flow of control to go to the next logical verb (which may well be
before the next period).

The clasic example of the difference is:

If A = B
If C = D
Display "Here"
Else
Continue vs Next Sentence
End-if
Display "Will be done with Continue"
End-If
* The next line ends with a period
Display "Still will be done with Continue".
Display "Next Sentence goes here".

(The ANSI/ISO Standard does *not* allow a NEXT SENTENCE before an END-IF,
but most of the compilers that I know of allow this - as an extension. The
difference between NEXT SENTENCE and CONTINUE still occurs whether or not
you have this extension.)

he meant: when you convert from OS/VS COBOL to VS COBOL II, convert
every NEXT SENTENCE to CONTINUE, *before* you start adding things
like END-IF.

So far as I can figure, NEXT SENTENCE and CONTINUE have exactly the
same effect if you don't use END-IF or similar scope terminators,
which OS/VS COBOL didn't support. That's why people got into the
habit of thinking that NEXT SENTENCE meant "do nothing."

Can anyone come up with a counter-example? I.e. a construct such
that:

1. It compiles either in OS/VS COBOL or in VS COBOL II;

2. If you mechanically change NEXT SENTENCE to CONTINUE and
recompile in COBOL II, the program behaves differently.

I suppose a trivial example might be:

DISPLAY "Hello"
NEXT SENTENCE
DISPLAY "World".

...but I don't know if the old compiler would allow this code,
since the second DISPLAY would be unreachable. In any case it's
not the sort of code many people would be likely to write.

I phrased the challenge in terms of IBM compilers, but I don't
mean to be parochial. I just don't know what compilers may have
supported scope terminators prior to the '85 standard.

My favorite change from COBOL VS TO COBOL II is the OCCURS DEPENDING ON
clause. If the COBOL VS said:
05 BYTE-FIELD PIC X OCCURS 500 DEPENDING ON X-NDX.
It means that you are making a variable length field of *zero* to 500
bytes long, depending on the value of X-NDX. In COBOL II you *must*
change it to:
05 BYTE-FIELD PIC X OCCURS 0 TO 500 DEPENDING ON X-NDX.
That is, you must add the "0 TO" if the index is ever set to zero in the
program. This becomes narly if the field is in a FD record
description. This is a variable length record. You must change the
RECORD CONTAINS clause if you do not add the '0 to' or remove the RECORD
CONTAINS clause. (you can in COBOL II). Another tidbit: the LRECL in
the JCL must be equal to the largest possible record length plus 4 (
COBOL II ). In VS, the compiler allowed you to put a smaller length,
and it worked, as long as the program never really wrote a longer
record.

You probably won't have to make a lot of changes. Some changes you
won't have to make but will eventually want to. Such as replacing
some massive IF Stmts with the Evaluate verb, or initialization
routines with the Initialize verb.

Heres the list of dropped verbs:

Identification Division
Remarks (use * or / in column 7)

Environment Division
Processing mode clause

Procedure Division
READY TRACE
NOTE
EXHIBIT (use DISPLAY)
EXAMINE (use INSPECT)
CURRENT-DATE (use ACCEPT WS-DATE FROM DATE)
TIME-OF-DAY (use ACCEPT WS-TIME FROM TIME)
TRANSFORM (use INSPECT)
ON statement (use IF...ELSE or EVALUATE)
Write after/before positioning (use just 'WRITE' , drop
positioning.

Additions in Cobol II

Scope terminators for verbs:
END-ADD, END-CALL, END-COMPUTE, END-DIVIDE, END-EVALUATE,
END-IF, END-MULTIPLY, END-PERFORM, END-READ, END-SEARCH

CONTINUE verb use instead of NEXT SENTENCE
NEXT SENTENCE means go to next period
CONTINUE means go to the next scope terminator

INITIALIZE
Will whole copybooks and fields with cascading levels to
spaces or numerics depending on PIC value.

EVALUATE
Uses case structure to replace the use of massive
IF...ELSE...IF....ELSE statements.

There is more, and variations of the above additions and deletions,
along with a few feature changes, and a few inconsistencies between
Cobol and Cobol II, but this will get you going.

The best explanation I ever heard about the difference between NEXT
SENTENCE and CONTINUE is that NEXT SENTENCE is an implied GO TO to the
next imperative statement following the next period, while CONTINUE is a
true No-op (for those of you with some assembler experience). Also,
CONTINUE can be used anywhere in the procedure division while NEXT
SENTENCE was more restricted in its use.

I learned something interesting from the IBM manuals recently. The EXIT
statement in COBOL/MVS behaves exactly the same as a CONTINUE
statement. That surprised me. You can put EXIT in the middle of a
series of imperative statements and it does nothing at all.

›
› INITIALIZE
› Will whole copybooks and fields with cascading levels to
› spaces or numerics depending on PIC value.

Normally, INITIALIZE will set numeric fields to zero and alphanumeric
fields to spaces. But it will not modify FILLER fields. So you can
safely use INITIALIZE on a print record with literals in the FILLER and
it will not wipe out the literals.
