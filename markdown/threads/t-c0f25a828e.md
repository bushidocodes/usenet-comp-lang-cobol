```
Hi all,

I'm running a Cobol-programm trying to process the result of the
following cursor:

SELECT CUSTNO, COUNT(*)
FROM PURCHASE
GROUP BY CUSTNO
FOR FETCH ONLY                 

I'm fetching each result line into the following host-variables:
FETCH <cursorname> INTO
 :RESULT-CUSTNO , 
 :RESULT-COUNT

When I'm running the program, I get the following error:
DSNT408I SQLCODE = -303, ERROR:  A VALUE CANNOT BE ASSIGNED TO OUTPUT
         HOST VARIABLE NUMBER 002 BECAUSE THE DATA TYPES ARE NOT     
         COMPARABLE                                                  
DSNT418I SQLSTATE   = 42806 SQLSTATE RETURN CODE                     

I've tried different definitions of RESULT-COUNT (e.g. S9(8) COMP,
S9(16) COMP, also COMP-3). It's definetely NO overflow (the maximum
count is 800.000) what causes the error.
What format must RESULT-COUNT do have to avoid this error?
Do anyone have some hints for me?

Thank you in advance,
Robert

mailto:kroessr@spardat.at
```

```
Robert,

Your query is flawed.

You need to qualify the SELECT with either HAVING (as you are using GROUP
BY) or WHERE, if you drop the GROUP BY.

At the moment the count will be all rows on the PURCHASE table, and it will
be this EVERY TIME you FETCH the cursor!

If you REALLY want an unqualified scan of the PURCHASE Table I would use the
following:

EXEC SQL
      SELECT COUNT(*)
            INTO  :RESULT-COUNT
         FROM  PURCHASE
END-EXEC
EXEC SQL
      DECLARE cur1 CURSOR FOR
         SELECT  CUSTNO
             FROM   PURCHASE
    ORDER BY   CUSTNO
END-EXEC
EXEC SQL
      FETCH cur1
          INTO :RESULT-CUSTNO
END-EXEC

Pete.
```

```
Hi all again,
thank you very much for your ideas and hint concerning this topic.
In the meanwhile, I figured out, what my real problem was.

I was also fetching an A08-field from DB2, which I read into the
following structure:

01 FIELD-A.
   05 FIELD-A-1  PIC X(4).
   05 FIELD-A-2  PIC X(4).

When DB2 was talking about "HOST VARIABLE 002" it wasn't meaning my
counter field; it meant the second X(4)-field. I replaced the
structure with an A08-field and the error vanished.

I wasn't aware that seperating character fields by using such
structures isn't allowed in DB2.

Robert
```
