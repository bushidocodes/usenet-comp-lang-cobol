```
These standards apply to all new Cobol code.

.. Fixed format.

.. A period at the end of every non-nested statement.

.. No blank data names. 
    You must code FILLER.
   If all operations are via 88 level, don't use FILLER, name it.

.. No inline comments.

.. No REDEFINES.

.. No qualification.
   
.. No more than one statement per line.
   Don't put COPY on the same line with 01.

.. File descriptions in copybooks
   One each for Select, FD and record (in working-storage). If a
   file contains multiple record types, for example header and
   trailer, there must be a copybook for each format.
   If two files contain the same format, there must be separate
record copybooks for each file.

.. No REPLACING.
   If there are six files, there must be six select copybooks,
   six FD copybooks and at least six record copybooks.

.. No common key areas.
   When key fields in the record are not contiguous, you will assemble
   keys in working-storage for two purposes: to sequence check and to
   phase files. There must be three areas per file: current sort key,
   previous sort key and match key. If match key is identical to sort
   key, a separate area is still required. Field names are prefixed
   with the file name and either curr, prev or mtch. Using a single
   key definition with OCCURS is not permitted.

.. No shared File Status code.
   Code a separate pic xx field for each file.
   This implies no common error handling; each file has its own.

.. All working-storage names must be prefixed with ws-, including
   record areas. 

.. Counters must have unique names.
   Do not use OCCURS. If there are 120 counters, write 120 names.
   Do not use OCCURS on report lines with columns of numbers.

Procedure Division

.. Paragraph names must begin with a five digit number, which are
   in sequence and indicate logical hierarchy.

.. All PERFORMs must be THRU nnnnn-name-EXIT, not nnnnn-EXIT.

.. No backward PERFORMs.

.. Main paragraphs must be named -init, -process and -wrap. Don't
   get creative with names like beginning, middle and end.

.. Control checking logic must be procedural rather than arithmetic.
   For example, when the spec says 'abend when the input is empty',
   you must check for end of file after the first read. Do not count
   the records and test whether the counter is zero.

.. For input files, control adds and sequence checking goes in I/O
   paragraph(s). For output files, it goes in business logic.

.. Follow the spec closely.
   For example, if the spec says:
       evaluate true
           when A < B
             ...
           when A = B
             ...
           when A > B
             ...
   write it that way. Do NOT write:
       evaluate A
           when < B
             ...
           when B
             ...
           when > B
             ...

   If the spec has nested IFs, do not change them to EVALUATE.

.. In conditionals, the literal goes on the right
   Wrong: if zero = ws-imbalance-1 and ws-imbalance-2
   Right: if ws-imbalance-1 = zero and
             ws-imbalance-2 = zero

.. ZERO is numeric.
   When checking file status, use '00' rather than ZERO.
```

Without going into style wars or rationalizations, I will just point 
out the differences between these shop standards and the ones in my 
mainframe shop.  I didn't write those shop standards, but I am 
comfortable using them.

Robert Wagner wrote:

> These standards apply to all new Cobol code.
> 
> .. Fixed format.
> 
> .. A period at the end of every non-nested statement.

No requirement for this in my shop.  Some programmers are using the 
single period per paragraph style now.

> 
> .. No blank data names. 
>     You must code FILLER.
>    If all operations are via 88 level, don't use FILLER, name it.

Not such restrictions in my shop standards.  I've seen blank instead 
of filler and FILLER for flag fields that have condition names 
defined.  Due to our data naming standards, a proper condition name 
would look like 88-100-ALL-DATA-PROCESSED.  The original justification 
was so that all condition names would appear next to each other in the 
cross reference listing, but still include the numeric prefix that 
identified the 01 record where they resided, for example, in the 
working storage 100-PROGRAM-SWITCHES record.

> 
> .. No inline comments.

If you mean comments within a paragraph, as opposed to a flower box 
between the end of the last paragraph and before the paragraph name, 
then inline comments are not restricted.

> 
> .. No REDEFINES.

No restrictions againt using REDEFINES.  RENAMES is somewhat 
discouraged, and I have never it used.

> 
> .. No qualification.

Qualification is not restricted, but our naming conventions require 
numbered datanames, with each 01 record having a unique prefix.  That 
pretty much rules out qualification, since there should be no 
duplicated data names to qualify.  But we don't enforce our naming 
conventions strictly.

>    
> .. No more than one statement per line.

Discouraged, but not prohibited.  Rarely seen.

>    Don't put COPY on the same line with 01.

No restrictions, but I've never seen it done in my shop.

> 
> .. File descriptions in copybooks
…[4 more quoted lines elided]…
> record copybooks for each file.

I have seen copybooks for FD's, but it's extremely rare and it is not 
required.  I've never seen copybooks for SELECT statements in my shop. 
  If a file has multiple record layouts, generally we put all the 
record layouts in one copybook.  Record layout copybooks are preferred 
in working-storage, rather than under the FD, but there is no 
restriction enforced.

> 
> .. No REPLACING.

No restrictions on replacing, but I haven't seen it used much, due to 
our naming conventions for data.

>    If there are six files, there must be six select copybooks,
>    six FD copybooks and at least six record copybooks.

Copybooks for record layouts are encouraged, but not for SELECT or FD.

> 
> .. No common key areas.
…[6 more quoted lines elided]…
>    key definition with OCCURS is not permitted.

No such requirement in our standards.  It's left up to the programmer.

> 
> .. No shared File Status code.
>    Code a separate pic xx field for each file.
>    This implies no common error handling; each file has its own.

Separate file status for each file is preferred, but not required.

> 
> .. All working-storage names must be prefixed with ws-, including
>    record areas. 

Each 01 record in Working-storage has a three-digit numeric prefix 
common to all fields in that 01 record, and unique to other 01 
records.  Copybook record layouts may have an alphanumeric dataname 
prefix.  Exceptions are not punished.

> 
> .. Counters must have unique names.
>    Do not use OCCURS. If there are 120 counters, write 120 names.
>    Do not use OCCURS on report lines with columns of numbers.

No such restriction.

> 
> Procedure Division
> 
> .. Paragraph names must begin with a five digit number, which are
>    in sequence and indicate logical hierarchy.

Our naming conventions are four-digit prefixes, but I have seen 
five-digit prefixes used.  Older programs (20+ years old) had some 
very weird prefix conventions, six character alphanumeric prefixes 
that no one understands anymore.

> 
> .. All PERFORMs must be THRU nnnnn-name-EXIT, not nnnnn-EXIT.

No GO TO, no THRU, No exits, No ALTER.  But our standards are not 
strictly enforced, and I have seen a few PERFROM THRU's and GO TO's in 
new production code.

> 
> .. No backward PERFORMs.
> 
> .. Main paragraphs must be named -init, -process and -wrap. Don't
>    get creative with names like beginning, middle and end.

The are no standardized paragraph names, but our style checker 
requires all paragraph names to begin with a numeric prefix, followed 
by an action verb (from a table of 206 action verbs), followed by a 
direct object.

I have seen older programs with an initilization paragraph named 
1000-HOUSEKEEPING.  Most newer programs have something like 
1000-INITIALIZE-THE-PROGRAM.

> 
> .. Control checking logic must be procedural rather than arithmetic.
>    For example, when the spec says 'abend when the input is empty',
>    you must check for end of file after the first read. Do not count
>    the records and test whether the counter is zero.

The exact method is left to the programmer's discretion.

> 
> .. For input files, control adds and sequence checking goes in I/O
>    paragraph(s). For output files, it goes in business logic.

The exact method is left to the programmer's discretion.

> 
> .. Follow the spec closely.
…[17 more quoted lines elided]…
>    If the spec has nested IFs, do not change them to EVALUATE.

Again, all these are left up to the programmer.

> 
> .. In conditionals, the literal goes on the right
>    Wrong: if zero = ws-imbalance-1 and ws-imbalance-2
>    Right: if ws-imbalance-1 = zero and
>              ws-imbalance-2 = zero

Left up to the programmer.

> 
> .. ZERO is numeric.
>    When checking file status, use '00' rather than ZERO.
> 

ZERO, ZEROS, and ZEROES are figurative constants.  They can be either 
alphanumeric or numeric depending on context.  Our style checking 
program warns for "data type mismatch" (comparing packed decimal to 
PIC X, for example), but the ZERO figurative constant will never 
generate any such warning.

I'm not saying the standards in my shop are better than some other 
standards.  I'm just pointing out the differences.

One other thing is that our shop standards specifically allow the 
project leader to approve any variance.  So if you are on good terms 
with your project leader, you have quite a bit of latitude.

With kindest regards,

worrying about locations in the main memory of the machine which is

In the time you are talking about, (pre-1983), there WAS no alternative to
the OS file system, OTHER than network or hierarchic database
implementations that incurred overheads serious enough to make them suitable
only for major applications.

ISAM worked fine to provide both random and sequential access for smaller
files where there were few relationships and ordering requirements, so
alternate keys were minimal. The only real downside was the necessity to
periodically reorganise ISAM files due to overflow (and second level
overflow on some systems).

When VSAM was released it quickly replaced ISAM because of its
self-reorganizing capability. (I still believe VSAM  (IDCAMS if we consider
all VSAM services) is the finest data handling access method that has ever
been developed. I like it better than BTRIEVE, although there is not much
between them...) It is no surprise to me that even today mainframe database
implementations  utilize VSAM as the underlying stroage handling mechanism.

Yet, even then, the Corporate solution was a database. I remember building a
totally VSAM based database which worked very efficiently and effectively
for a major corporation in london in the late 70s. It was replaced within 3
years (to the chagrin of my Management, who had backed it to the hilt) by
IMS which was less efficient, but more acceptable to the 14 European
countries where the system had to run. (I wasn't bothered. I got well paid
for it and did a lot of tripping around Europe...)

Acessibility from non-COBOL didn't matter because there was no other
"standard" software, such as we take for granted today. The whole gamut of
PC software which has revolutionised the way people use computers was not
invented yet.

As time went by, PCs came along, newer and different ways of doing things
evolved, Codd and Date released the Relational model (early eighties)  and
the efficiency of DBMS (including RDBMS) improved many thousands of times.

Today, I wouldn't consider using ISAM on a PC (or VSAM on a mainframe) for
serious application development. The RDBMS implementations offer me so many
facilities it would be silly not to use them. I can design and build
applications using ACCESS or MySQL and know that they will scale perfectly
to whatever platform I want to run them on. (In fact, in a mainframe
environment, while I was waiting for the DBA group to approve the use and
creation and allocation of test databases, I'd already have them running and
under test on my PC, shared with my colleagues. It is a good way to parallel
process the politics <G>) Not only that, but standard software such as
spreadsheets can get to the data without any effort on my part. I can know
that I don't have to build interfaces to all my other systems; the databases
integrate easily and seamlessly simply by adding linked tables.

I believe your thinking may be coloured, Bill, by the fact that IBM have
always provided excellent support for IDCAMS and it is one of the best
things in the IBM arsenal. DBMS use in the mainframe environment is hindered
by the politics of getting DBA approval to do anything that touches
Corporate databases, and the difficulty this imposes on application
developers makes them look for other ways. It is easier for developers to
simply use IDCAMS and handle the data that way. If you were free of those
encumbrances (as people developing in a PC environment are) you would
probably see things differently.

It is gratifying to me to see that when I advocated people should move off
the COBOL file system in the last century, I received vitriolic attack and
abuse, but now people say: "Well of course. " (Apart from people like
yourself, of course, and that is why we are having this discussion... <G> )

Bottom line:

Application developers who are using Client/Server are most likely to use
RDBMS because it is simple, efficient, and brings with it a number of useful
facilities that would otherwise need to be written.

Application developers working in mainframe environments have been raised on
IDCAMS, know it well, don't usually need permission to use it, whereas DBMS
involves political manoeuvering with interface through a DBA group that can
be tiresome and time consuming. They are therefore less likely to use DBMS.

Both groups are being driven together by the corporate need for free data
flow and interfacing to all corporate data systems. This factor weights the
dice in favour of DBMS in the long term.

I should mention in closing that the new breed of users and programmers who
have grown up with PCs wouldn't know what ISAM was... they know SQL and when
they need to store data they knock out a database in minutes.

VSAM ESDS (VSAM sequential files) are used very commonly in CICS for 
writing transaction logs.  That's because using QSAM files in CICS is 
a bit...difficult.  A VSAM ESDS file can easily be open for both input 
and output in CICS.  Not so for QSAM which requires CICS DCT entries 
(Destination Control Table).

In CICS you can specify in the FCT entry that a VSAM file is a data 
table.  You must specify the maximum number of records.  The entire 
file is cached in memory, which means access is much faster, but the 
programs need to be recompiled.  There are two types of VSAM data 
tables - CICS maintained, where CICS periodically flushes buffers to 
the disk, and User maintained, where the file is never copied back to 
disk.

I support applications with both CICS maintained and User Maintained 
VSAM data tables.  For User Maintained tables we define an extra FCT 
entry that points to the same file, without defining it as a data 
table.  We have "sweeper" programs that run at timed intervals to read 
from the UMT and write to the disk version of the file.

It's still a good idea to reorg VSAM KSDS files periodically, 
especially if there's a high volume of insert and delete activity, but 
  you can go for quite a while before performance degrades.

I have never needed to use a VSAM RRDS file, and I'm not sure if RRDS 
supports variable length records.  Obviously, variable length records 
are well supported for VSAM KSDS and ESDS files.
