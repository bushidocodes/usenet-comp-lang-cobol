The motivation for the creation of such "systems" is usually a desire
on the part of the users to avoid the time consuming one-at-a-time
data entry procedure necessitated by the use of a mainframe
online update screen.  They consider it to be a "time saver"
if they can just give IT a flat file and have them upload en masse.

It never works.  You end up saddling one or more expensive
systems analyst/programmers with the job of shepherding
this task to and from the mainframe.

You have 2 choices.  Either develop a new screen which allows
the entry of multiple lines of data simultaneously, or escalate
the issue to management with a recommendation that the users
use the regular online update procedure.  Have the cost figures
available to back this up, and inform the users that they will be
charged for the additional expense incurred if they insist
on using this kludge.

One of the things I did for one of the companies I worked with was a
set of VBA code that would
Phase 1
1- Download a file from a server (AS400 on this case, could be any
FTP) and place it directly on an Excel worksheet
User would then be able to change the data.

Phase 2.
When the user clicks a Macro button the following would be done
1- Data validation.
  Each cell would be validated to see if it contained wrong data (E.G
letters on a numeric column, or non dates on a date field).
  If found those would be highlighted (RED).

2- Once the validation was completed the following would be done.
2.1 - Save the  the file in a specific format (.csv on our case was
fine, it might not be in others)
2.2 - Upload it to the server
2.3 - Run a server program to process the file.


From the above the Upload/Download would never fail by user mistake.

Point 2.3 was done automatically by the macro as the AS400 FTP allows
us to issue a QUOTE command to run the programs. Not all FTP
implementations allow this, so you may need to submit the JOB on the
mainframe side.
