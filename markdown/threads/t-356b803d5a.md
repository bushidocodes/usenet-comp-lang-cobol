It is interesting to look at Tucker's comparison of a data processing
application in COBOL and PL/I ["Programming Languages", McGraw-Hill, 1977].

As a language, PL/I rated better than COBOL in:
	* programming features
	* overall writing
	  * compactness
	  * uniformity
	  * locality (rated equal with COBOL)
	  * segmentation
	* Overall reading
	  * Structural clarity
	* execution diagnostics
	* other debugging facilities
COBOL rated better than PL/I in:
	  * Local clarity.
	* data processing functional support.

In other words, PL/I rated superior in nine out of twelve points, and was rated
equal on one point, while COBOL rated superior on only two points out of twelve.
Note: run times for the PL/I compilers, particularly the Optimizing Compiler are
not valid because the REORDER keyword for the main procedure has not been
specified -- the compiler cannot perform full optimization.  The REORDER keyword
must be specified in the PROCEDURE statement in order to obtain full
optimization.

Note: The PL/I program for master file maintenance required two pages and
9 lines of code.

The equivalent COBOL program required six pages of code.

Also note that the copyright date on that book is 1977.  This is 8 years
before the '85 ANSI COBOL Standard and more than 10 years before the '89
Amendment.  Also it was few years before the time that ANSI/ISO "gave up" on
PL/I - and said that it was "functionally" frozen as a programming language
that they wouldn't be providing any future ANSI/ISO PL/I Standards (while
COBOL continues to be supported in this "portable" ANSI and ISO environments.
Similarly, COBOL is included in the UNIX X/Open Programming Specification -
and PL/I is not.)

Therefore, the comparison is significantly out of date - and ignores
portability - but other than that it is a "nice" historical document!!!

Robin,

I do not agree with many points you defend:
- About debugging:
In the context of OS/390 and Language environment, it is much easier to read
LE dumps for COBOL programs than for PL/I:
If you compile your COBOL  module with TEST(NONE,SYM) your load-module is
slightly bigger  but the performances are the same (We've measured it), you
directly have in the dump the statement number where the problem occur and
you can directly read in the le dump (card CEEMSG) the content of all the
variables of all the modules stacked in your enclave at the time of the
abend (variable name + content) if you dump with the correct option.
This is impossible for PL/I (read the manual section CEE3DMP that states
that the 'variable' option is valid for all languages except PL/I). PL/I LE
dumps display only the statement number but not directly the content of the
variables (compilation option GOSTMT)

About the debugger:
The debugger is common for all the language, so not advantages here

- about error checking
You can activate option that check the same things as PL/I (either clause ON
ERROR in the code or compilation option to check arrays out-of-bound
conditions or overflows)

- about the size of the source
It is much easier to create smaller modules in COBOL and dynamically call
them (even under CICS). Even if PL/I allowsto dynamically fetch modules,
there are really annoying limits due to the language itself that block you
in doing this (the famous fetch limitation, that should be alleviated in the
coming release)

- about optimization
In COBOL also you can request local optimization by specifying a number
after the section name. however nobody uses this in COBOL because there is
no such limit in COBOL than in PL/I about the size of stuff that the
compiler can optimize (for example the PL/I compiler optimizes only the
first 255 variables...)

- about the easiness of PUT DATA:
 With DISPLAY in COBOL you can do exactly the same thing by simply writing
DISPLAY <var>, even under CICS, the output goes in card CEEMSG by default.
