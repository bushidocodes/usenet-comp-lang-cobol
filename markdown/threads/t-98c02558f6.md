```
Hello,

According to the COBOL standard, the contents of a record description
for a file are only supposed to be available to a program when the
file has been opened, the record has been read and the file has not
been closed.

Consequently your DISPLAY statement should be between the READ and the
CLOSE statement.
```

```
Slight correction to what Robert said below (that may or MAY NOT impact this
specific issue).  He said,

"According to the COBOL standard, the contents of a record description for a
file are only supposed to be available to a program when the file has been
opened,  ..."

I believe that the current (past and future) COBOL standards say,

"contents of a record description for a file are only REQUIRED to be
available to a program when the file has been (successfully) opened - and
before it is closed. At other times they are undefined."

In fact MANY (but not all) compilers and run-times leave the "record
descriptor" available after CLOSE of the file.  Some (but not all) allow you
to move data to (and from) the "record descriptor" before an OPEN of the
file.
```

```
DOS has a very poor comm port handling and the data is transitory at
best.  The chances of the data being in the COM port at the time that
you execute the READ is very low.  Using MODE to change the port
settings will reset any data there at the time, the READ won't wait
until data arrives.

If you actually need to read serial port data you will probably need
to use a buffered mechanism that can get the data from the serial port
using interrupts and then store it until your program asks for it -
and it handles the DTR/DSR signals.  Windows will do this (but it
requires writing an actual Windows program with callbacks and such). 
Linux/Unix will do this all by itself so reading /dev/ttyn will give
whatever data has already arrived.

For DOS programs the best approach is to use a FOSSIL driver (or
FOSSIL emulator under Windows).
Serial ports aren't like disk files, you can't just READ and expect
that the data is actually there.
```
