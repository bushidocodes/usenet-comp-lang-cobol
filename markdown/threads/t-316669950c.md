We like it a LOT. It seems to be only COBOL compiler out there
decently priced that is really a compiler, and not a cheap way for
the compiler company to make $$ off runtime licenses. (No insult
to the rest of you compiler companies, but run times are simply
a rip off to anyone from the UNIX world - like me. Especially when
the "compiler" itself costs as much as a datbase development system.)

The PowerCobol stuff is a little clumsy to me, as event handlers don't
seem natural to COBOL, but that is probably personal predjudice on
my part. The COBOL85 system seems to be very complete and well
documented.

We have found it is not as easy as copying files to the Sun servers
and recompiling them from WinNT sources; there are some changes that
have to be made- often dealing with case in the COPY files. The Sun
compiler will often complain about incorrect formats in COPY files
that are acutally correct, but have a case difference. (Okay, I
guess that makes them incorrect, but they work under NT. :)

The access to SQL is not as nice as say, AccuCobol, which can
actually use READ's and WRITES, but works cleanly and is fast.
It works with Oracle, SQL Server, SyBase, DB/2 V5, Informix,
Empress, and AdaBase to my knowledge, and Fujitsu claims it works
with any ODBC compliant database. I have no reason to doubt this
since it has worked for me each time.

The technical support from Fujitsu is enthusiastic and accuracte,
as well as dedicated to their product line. I think they are a fine
example of a tech support shop. I understand they even have pagers
set up to warn them when someone sends mail to the support email
address. I like that a lot. They also will follow up on a problem,
and have a good sense of humor. That is important given the stress
of any tech support popsition.

However, their order processing side is somewhat lacking. We have sent
TWO checks to them for the COBOL profressional system (i.e.- we
were impressed enough in the system to pay for it, which given that
the base system is free, is pretty impressive! :) but they haven't
cashed either check and I haven't gotten any shipments from
them, or notice that they have received the order. This is a little
nerwracking since I want to put out a product with it, at least to
a limited set of clients.

Lastly, they have the NetCOBOL compiler, which generates Java Bytecode.
I haven't tried this one out yet, but I suspect it will be something
I fall in love with.

-Paul

We are trying Fujitsu Cobol out but there are a few features we don't like.
The editor doesn't seem to do searches on any non alphanumeric characters
including spaces! There were also no links on the debug list but it did
include the line number of the error and the editor DOES display this. On
the other hand, we were able to compile and link 150,000 lines of code and

commands or syntax used by our current servers. We used a Cobol program to
accomplish this! I think with a little more cleaning up that this product
can't be beat. We also tried MicroFocus and would have spent months
getting that to work. I would really like to see what they improve in
version 4. They already provide quite a few useful utilities for

The product (3.0) is marketable. They have some work to do in the area of
easing support for the end product (A corrupt file reorganization that can
be called/passed parms). The compiler appears sound. The runtime is of
medium size. Smaller than Microfocus but much larger than Ca-Realia.

Speed? Compile speed fair, execution speed good. Not so good as Ca-Realia
for Indexed file access, but faster than MicroFocus. Optimization of code?
Supreme. Best I have seen in actual comutational execution. It screams.


PowerCobol - in flux. Once they settle down on it, I might review it
again. Better than MF Dialog by a LONG shot - not so good as COBOL Sp2 by
flexus. One thing I have seen with the COBOL GUI tools. Once you start
using the one provided by the vendor of the COBOL you are kind of stuck.
Use SP2 instead - compiler independant. I currently use CA-Realia COBOL
for 16 Bit, but will move to Fujitsu for 32 bit. I won't have to change
the code much. But if I was using Visual Realia, or moving from Micro
Focus (that we use here at work) to POWER COBOL, it would be a rewrite.

Direct conversion issues - good support of Micro Focus source. Not so good
on Realia, but changes are minor.

In all I'd give it a B+.

They have kind of cooled recently in their advertising etc. Support -
Better than Micro Focus - but not so fast. They seem to want to TALK to
you a lot, rather than support Q and A in e-mail. If you like that you
will be happy with the support.

Documentation - Fair. There's a lot in there that is kind of hard to
search out. Many things I had questions on were just not well documented
at all.

I ordered mine and received it in 36 hours. I spread the website through
internal e-mail, and several people at our company ordered and received it
within a week. We have installed from the CD onto both NT and HP-UX. We
will be installing the Solaris version next week. All three are on the CD.


The product works well. We were able to take COBOL II programs, and MF
COBOL programs over, and recompile with almost no difference. The biggest
problem was figuring out how to point at the copylib directories which our
existing MF COBOL was using, and renaming some copybook files. A few
source files with extra long trailing spaces in the lines had to be pulled
up in the editor and resaved to strip trailing blanks. In other words,
nothing serious.

The debugger environment has some features MF did not have, but is lacking
in displaying data structures. There seems to be no way to pick a group
level (e.g, an 01 level structure), and have it explode out all of the
elementary level variables. You have to request each elementary variable
separately to see it formatted. This may not seem like a big deal, but it
is causing us headaches.

We are still working on some benchmarks to compare throughput, but the
initial short tests look OK. We are going to be running some fairly large
datasets (about 70 million records) into conversion routines to break IMS
like records into segments for load into a relational database. This
should output about 400 million records per run. I will report back on our
timings after we run with the big datasets.

As a side note, my company is ordering subscriptions for the NT and Solaris
versions, rather than paying the 'platform conversion' fee for our existing
MF COBOL licenses under OS/2. MF, are you listening?
