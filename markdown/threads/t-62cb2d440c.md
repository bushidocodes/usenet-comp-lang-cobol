In article <6jasr0$qnh$1.··.@nnr··s.com>, sel··.@sow··a.net writes:

› are COBOL coding forms still used?

In the past 15 years the only usees I had for a coding form were as a record
layout for data being exchanged between programs and a make-shift printer
spacing chart.

Coding programs on coding forms dates back to where many fewer people knew how
to type, and it was faster and cheaper for a programmer to print text on a
coding form and have the data entry service punch cards from the coding forms,
than it was to have programmers hunt-and-peck on a keypunch machine.

Now many programmers type faster than they print and editors (even "primitive"
editors like ICCF on VSE mainframes) make corrections a whole lot faster, so it
doesn't make as much sense to use coding forms as it used to.

What do I do when I am away from a screen and I want to lay some code? I use
notebook paper, or I use a sheet of scrap paper with black lined paper behind
it, and I code somewhat freehand because I know that I will be using an editor
that clearly shows the columns when I enter the code.

We still use make-shift printer spacing charts from time to time to visually
see how much data we can put on a 132-column report.

I remember when our instructors insisted on punching sequence numbers on cards.
Fine--having been on both sides of the service counter at a community college
data center, I know how easy it can be for cards to be dropped, particularly
when one has students learning to be operators handling decks of students
learning to be programmers. However, instructors still insisted on sequence
numbers after the programs were being entered on the terminal. (We taught our
students the RENUM command to satsify that requirement.) The instructors
finally learned that those sequence numbers aren't really so necessary after
all.

Sounds to me like your instructor is an idiot that is at least twenty years
out
of date.

First of all, flowcharts. While they may be useful for figuring out short
algorithms,
they are next to useless for complete programs. A properly constructed
top-down
program is a tree structure, always. That means that every single flowchart
is
identical, differing only in the number of nodes and levels. The main
reason
for a flowchart is to help document "go to"s. Modern methodology discourages
their use.

Secondly, the purpose of a coding sheet is as input to the keypunch
department.
They pre-date word processors. The modern method is to edit the program
on screen. Even if a card reader and key punch department still existed
somewhere
in the world, I would bet money that the programmers would prepare the
coding
sheets on-screen, using a PC. The last time I used a coding sheet was 1974.

BTW, if you want some useless knowledge, do you know what to do with
all your card source code and printed output when the machine site is
flooded
and everything is soaking wet?

in my editor, I can re-arrange paragraphs for readability and style, as well
as tidying up indentation as the logic evolves.

The last thing in the world that you want is to do a compile of a 1000 line
program for the first time. I write a stub, then compile, correct, and
tidy. Then

complete.
I seldom write more code than I can hold in my head before debugging. I

work

with

write a short essay on how the code could have been written better for
the next student ... i.e. teach readability.

In the real world, it depends on your employer. The thing you need to
prepare for once you get there is be ready to a *lot* of grunt work:
reading alien code, making the miminum changes needed to enhance or
bug-fix a program (even though you think you can rewrite the whole
program better), dump reading, pride swallowing, etc. It's rare for
someone just out of college to go directly to development work in a
corporation. They want to see if you can be trusted to follow orders
and produce reliable code. Only after a couple years experience will
they put you on development, or you find a shop just starting at the
ground floor where all code is new. In the meantime, take advantage
to study everything: current practices of your shop, best practices
of your industry, the business behind the code, etc. Also take
advantage of you college career by offering to do "free" consulting
work for your classmates. Don't do their homework for them but help
them resolve the really confusing bugs in their code. Best practice
in reading alien code I can think of.

Anyway, the forms I use most often is either high-level HIPO, to
describe to the user how a program or system works, ER-diagrams when
building databases VTOC diagrams to understand the relationship
between a program's modules decisions tables on how to set up complex
conditions and what actions go with what choice. If I'm dealing with
a interactive program or system, I'll throw in an event/response chart
to clarify what a program is to do under what user actions. The
closest to coding with a diagram is when I have to clarify a complex
piece of logic with a Warier-Orr diagram with the aim to simplify the
structure as much as possible.

After a while of a writing a lot of code one tends have a library
containing a pre-defined set of programs because the user would become
rather predictable. If your user wants data from a certain set of
files or tables, you'll find in your personal library of "throw-away"
programs the code that comes closes to what the user wants, modify it
to meet the requirements; run, test and present the results to the
user; then you save the program under a new name to add to your
library. You might be doing something simular to meet your
assignments.

Just my two cents worth adjusted for inflation,
