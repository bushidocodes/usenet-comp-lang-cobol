The good point about doing what everyone else is doing is that it usually 
makes the person who has the most say over signing checks for the systems 
to smile.

Now I know you're 'just asking'... but here's a freebie: if copybooks are 
all kept in a single PDS to which programmers have only read access then 
the likelihood of someone mashing a good version with a bad one decreases.  
If a copybook needs changing then at least four eyes - the coder's and the 
person with update access to the PDS - might give the work a going-over.

(Books have been written and courses offered about the arcane art of 
Version Control... and they usually get forgotten because, as we all know,

Frank,
  As I know that you are "setting up" a new z/OS environment, what you "should" 
do may differ from shops that have inherited (for decades) how they do things.

It has been a LONG time since I actually worked in a production z/OS (MVS-ish) 
environment. Therefore, I will happily yield to Arnold and Doc who have more 
current experience that I do.  However, when I did, it seemed that most (large?) 
shops had 3rd product source code control systems (originally, Panvalet and 
Librarian, then Endeavor, then others) - or at LEAST used SCLM.  With the 3rd 
party products, there were significant advantages in that you could (relatively 
easily) get back to earlier versions of source - when a "mistake" was 
introduced.

If you ever see COBOL source code with
  ++INCLUDE
or
 -INC

These were "equivalent" (sort-of) to COPY statements.  However, with later 
versions of those products you could "process" a member via a COPY statement and 
the product had vendor-supplied "access methods" to make their proprietary 
libraries look like "normal" PDS's.

All of this is to say, that (if what I remember is still current practice at 
many shops), your original assumption that source code was in PDS's is where the 
"error" is.  When you use the source control systems, "naming" and where you 
store things becomes a different question.

Anyone remember the 1980s buzz-phrase of 'we get it right the first time, 
we can't afford to make mistakes'?  The primary advantage of source-code 
configuration systems - be they system-based or a crotechty old coot who 
'signs out' code to you is that they encourage efficient co-ordination of 
maintenance and decrease the probability that two programmers, unbeknownst 
to each other, are assigned to work on the same module.

The primary disadvantage is, of course, they Cost Money and folks still 
manage to find work-arounds that can cause grave difficulties.  My current 
client's site uses CA-LIBRARIAN and two Librarian source libraries, 
HIGHNODE.DEVMAINT.SOURCE and HIGHNODE.PRODUCTN.SOURCE, with only Ops 
having write access to the latter.  Both libraries contain source and 
copybooks... but when in Rome one toddles toga-clad.

When I am told a module needs maintaining/enhancing the first thing I do 
is make copies of both the DEVMAINT and PRODUCTN versions to my own PDS 
(USERID.LIB.COBOL) and run a SuperC compare against them; if all is well 
and good the modules are identical.  I then do what needs to be done, 
compile with my own JCL to my own USERID.TEST.LOADLIB and test... and 
curse... and re-mod, compile, test, curse, etc... until I am satisfied 
with the results.

I then copy what I believe to be my properly-changed code into the 
DEVMAINT library, compile it there using standardised JCL, test again and 
then notify Whoever Needs To Be Told that I'm ready to have someone else 
break my code.  Sometimes they need to do minimal testing - 'the group 
that uses the code was re-assigned and the Department Name needs to be 
changed, leave everything else as is' - and other times it can get a bit 
scary, along the lines of 'Sixteen counties in four states need to have 
taxes re-calculated BUT ONLY IF the employee has an even number of 
children with a total number of teeth which, when divided by three, leaves 
behind no remainder'... at which point someone will ask me for the names 
of the files I cobbled together to do my own testing and grab copies of 
them... but never mind that.

Once the DEVMAINT testing is done the module is copied to the PRODUCTN 
library and recompiled into .PRODUCTN.LOAD ... and life runs on.

The z/OS shop where I work uses Endevor to manage source code and 
executable objects.  Endevor can be configured in many ways, but we have 
six stages from Test to Prod.  Production COBOL source code is stored in 
  PDS libraries that are exclusively managed by Endevor.  When we 
"retrieve" a COBOL program (with or without "sign-out") we store it in a 
PDS where we can play with it.  The PDS would be named something like 
"HIGHNODE.TEST.PDSLIB" so any programmer could use it or "USERID.COBOL" 
so only I can read it.

We used to have Panvalet, but replaced it with Endevor over 10 years 
ago.  Our Endevor is configured to manage source files as "elements" by 
"element type", which we have named as follows:

COB	COBOL source code
ASM	Assembly language
EZT	Easytrieve
CPY	Copybooks (for COB, ASM, or EZT)
JCL	Job Control Language
PRC	JCL Procedures called by JCL
CRD	Control Card files called by JCL and PRC
DOC	Operational documentation for JCL jobs
LNK	Link Edit control statements for specific builds
PKG	DB2 Package control statements

Our Endevor is configured so that ADDing a program to the first Test 
stage requires a compile, and the generated executable is moved to the 
next higher stage whenever the source code is promoted.  Endevor could 
be configured to automatically recompile every time the source is 
promoted to a higher stage, but our management decided that we only 
wanted to promote tested load modules.  Recompiling at every stage might 
make it a little easier to manage copybook changes in sync with program 
changes.

The generated result of an Endevor-controlled COBOL compile is an 
executable load module, in stage 6 that would be one of two PDS 
libaries, either HIGHNODE.PROD.LOADLIB or HIGHNODE.CICS.PROD.LOADLIB. 
But for intermediate stages we have different names such as TEST or CERT 
or PROD in the PDS library name.  But copybooks are generated to 
HIGHNODE.TEST.MACLIB or HIGHNODE.PROD.MACLIB as needed, so they can be 
available for compiles.  Similarly, control card modules are generated 
eventually to HIGHNODE.PROD.CARDLIB, procs to HIGHNODE.PROD.PROCLIB, JCL 
to HIGHNODE.PROD.JCLLIB, and operational documentation to 
HIGHNODE.PROD.DOCLIB.

A DOC file has the same name as a JCL source file, and contains 
scheduling and restart instructions for the operators.

You must specify a "Processor" name when compiling a COBOL program into 
Endevor, because the processor script must be different for batch or 
CICS, for different COBOL compilers, for DB2 or non-DB2, and for 
Debugger support or no debugger support.  And CICS programs must be 
generated into HIGHNODE.CICS.PROD.LOADLIB while batch COBOL programs are 
generated into HIGHNODE.PROD.LOADLIB.

All this might or might not be helpful to Frank Swarbrick.  A friend of 
mine who worked for a retail chain converted their small IBM mainframe 
from DOS/VSE to z/OS and found the use of JCL versus PROCs confusing. 
It was very different from the way they managed DOS/VSE JCL.  They also 
had very few standard naming conventions for source files or data files, 
which made it harder to manage their security rules.

Endevor also supports recovery of earlier versions of source code, and 
even automatic backout of executable programs.  SCLM is from IBM, and I 
believe it is included with TSO/ISPF on z/OS, but I don't know if it 
manages executables in addition to source code.

Endevor is a big, complicated tool and I haven't even addressed the 
problem of naming conventions for programs, JCL, procs, and copybooks, 
or other issues like nested procs in production JCL, or Security/audit 
control procedures for installing objects into production.  Object 
naming conventions can have a large effect on your RACF/ACF2/TOP Secret 
security administration rules.  You can guess that I work in a fairly 
big shop that has to comply with GLBA, SOX, and PCI-DSS.  While I am 
used to the way we do things there is absolutely no guarantee there 
isn't a better way to organize it, or better naming standards to use.

If all I had to do was code and test COBOL programs my job would be much 
easier.

With kindest regards,
