```
The draft of the next Standard (and a number of existing vendors as an
extension) already support what is called (in COBOL) "in-line comments". For
example,

   Move A to B *> B is that field
                     C *> C is that other field

All COBOLs already support "multi-line" comments, just put an asterisk into
column 7.  There USED to be another way by calling something a "NOTE" - but I
think that was dropped in either the '68 or '74 Standard.  (I know it was
gone by '74).  If you use the asterisk in column 7 approach, that should work
in all current conforming compilers.

FYI,
  In the next Standard, there will be both fixed-form and free-form reference
format.  For fixed-form reference format, the asterisk in column 7 approach
will continue to work.  For free-form reference format, you just start each
line with the "in-line comment indicator", e.g.

   *> This is the first line of comments
   *>  and this is a second line.
```

```
UNFORTUNATELY, semi-colon is in the current Standard as (virtually) as a
no-op, i.e. it (a semi-colon followed by a space) has the same meaning as a
comma followed by a space (which is that it means NOTHING more than a single
space).  Although the draft does add the "_" (underscore) character for the
use in COBOL words and the "&" (ampersand) as a concatenation operator, there
was a "goal" not to add new characters to the COBOL character set (now called
the "COBOL Character Repertoire").  For those who went thru changes to
COPY/REPLACING between '74 and '85 for the colon, adding new "special"
characters really can have unforeseen and UNDESIRABLE side-effects - which is
why they (we?) ended up using a combination of characters for the inline
comment indicator.

I hope this helps.
```

```
Lots of programs have semicolons in them.  They are punctuation similar
to commas, but less likely to be perceived as periods.  They don't do
anything, but can indicate that there's more to follow.

e.g.
MOVE X   TO FIELD1; 
            FIELD2;
            FIELD3.
```
