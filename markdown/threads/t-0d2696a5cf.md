Howard,
   This is an "age old" difference between IBM (and I think - but am not
positive MOST/all? mainframe environments).

In IBM (that I know of) mainframe environments, the "file system" is part of
the operating system software - this includes, not only "sequential" files
but also files that "map" on to COBOL's definition of ORGANIZATION INDEXED
and RELATIVE.  (Actually, IBM occasionally has had "some" problems with
this.  Look at the SIMVRD "history" to see such an example, that eventually
got "fixed".)

In NON-mainframe environments (Windows, OS/2, Linux, Unix - possible
others), the operating system usually (always?) provides SOME form of
"sequential" file support (which may or may not relate to the COBOL
requirements - see differences between "record" and "line" sequential
files) - but almost never has "built-in" support for the COBOL definition of
indexed and relative files.    Often their are 3rd party products (not
provided by the O/S vendor or the COBOL vendor) such as C-Isam and BTRIEVE,
but almost always the COBOL vendors themselves provide "file" software to
provide these file types.  Originally, many of these vendors didn't even
provide "external" API's to get at the files - outside of their
compiler-environment.  Now, most of them to provide some sort of "callable"
external file handler - but even in these cases, the "files" are considered
"COBOL files" - not O/S files.

    ***

Does this help?

LX-i wrote:
 >
 > Does anyone know if, in the OO COBOL world, there is an XML DOM or

I know of one COBOL implementation, commercially available in the near
future, that includes support for parsing and generating XML documents -
OO or not as you please.  It's probably not the only one.

 > Or is that one of those things that's better handled by Java or C or
 > VB or <name another lanugage>?

I've parsed XML documents in Java, C, and C++, using the Xerces parser
(which comes with Java and C++ APIs but not a C one), not to mention a
little hand-crafted parsing in C (only a little because I had no
compelling reason to reinvent the wheel in this case).  I'm not sure I'd
argue that any of them are particularly suited to parsing XML.  XML is
in some ways a problem area that OO languages map well to, but as usual
the decision to use OO is probably best guided by the developer's
familiarity with it and the language at hand.

If I had to write my own parser, ignoring any other factors such as
performance, I'd probably pick either a powerful functional language
like Scheme or a language with good built-in lexical analysis features
like Perl (though I'm not fond of Perl syntax; might be a good time to
learn Ruby or Python instead).  I could do it in COBOL, but it wouldn't
be my first choice - largely because I'm not as proficient in COBOL and
I haven't developed a COBOL style I'm entirely happy with.

But since the Xerces parser (optionally plus the IBM ICU extensions for
handling lots more encodings) is freely available, that's unlikely to
happen.  Using Xerces from COBOL might be a tad complicated (even
writing a C binding for Xerces' DOM implementation wasn't trivial), but
fortunately native COBOL XML support is available.
