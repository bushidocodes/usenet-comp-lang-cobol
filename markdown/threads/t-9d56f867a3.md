Actually we did get quite a high level of support from IBM for LE migration.
The issue seemed to be that the risk of application error handler exits not
getting it correct weren't understood. 

At one point one of the LE developers showed up for a site visit. He and I
were the only men there wearing sandals. In my case that was because I had
come in on short notice for the meeting from a "vacation" day I'd booked to
paint the exterior of my house and only had time to take off my coverall
and slip into sandals before heading to the meeting. Ms. Brown, his manager,
told us that he was actually looking particularly presentable that day,
having touched base with his wife to get him tidied up. She showed up for
site visits several other times as well.

I mentioned to the developer an example of an Application PL/I ON ERROR
block which began with PUT DATA; instead of ON ERROR SYSTEM;

When invoked by a data exception it looped until cancelled.

Most of the issues seem to be a misconception about error handling in
environments such as IMS, CICS and DB2.

The promoters of LE made a big PR point of saying that something to the
effect that 
   now in the event of an error you can get control, fully resolve it
   and continue processing

Perhaps some people are clever enough to code a one size fits all error
handling routine that is 100% guaranteed to handle all possible errors,
without exception. I know I'm not that smart. 

For example, failing to enable language features to check for references
beyond the bounds of an array or string can corrupt executable code. At
that point all bets are off. The program may also have corrupted values
needed to issue an IMS rollback call, for example, after a data exception
caused by these types of errors.

The LE refusal to honour NOSTAE,NOSPIE in PLXOPT external variables, for
example, actually made it incompatible with the IMS requirement that
application code never get control in such cases.
 
That is, TRAP(ON) is similar to running STAE, but the IMS manuals said
that PL/I had to run NOSTAE to ensure that IMS could backout to the last
point of consistency if an error occurred. Getting everything coordinated
took a while.

The issue was that if application code gets control and doesn't resolve
problems correctly you can end up with an IMS DB or DB2 table which is
not consistent with your business rules. 

My advice is to have IMS or DB2 programs abend with errors in application
code, rather than having application code try to fix things up.

http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/ceeu1007/5.4.8?DT=19950913203351

It took a while to get the concept across and for IBM to provide a way to
disable application error handlers in production application code while 
allowing normal operating system or DBMS abend or interrupt action outside
application code.

It is sometimes useful for application error handlers to get control in
testing or debugging situations but they are dangerous for production
IMS or DB2 applications.
