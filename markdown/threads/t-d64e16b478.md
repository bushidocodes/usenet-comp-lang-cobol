If you have a command line program in Windows NT that does not execute
(or does not execute correctly), you enter FORCEDOS on the command
line to put it into a more DOS-like environment.

For example, if you have a program named FOOBAR with command line
option /X, you normally enter "FOOBAR /X" on the command line to run
it.  To run it with FORCEDOS, you enter "FORCEDOS FOOBAR /X" on the
command line.

Depending on what exactly FOOBAR does, using FORCEDOS may allow it to
run (if it doesn't run at all), or may allow it run with fewer errors
than without FORCEDOS, or FORCEDOS may be no help at all.  But it's
certainly easy enough to try it out.

> DOS only versions of RUN.EXE.  It had another extension, DLE if I remember

You are probably thinking of COBLIB.DLL and COBLIB.DLE which are the
shared runtime DLLs for OS/2 and DOS respectively.

The problem is actually the compiler: COBOL.EXE is a 'family'
application, as are several other .EXEs.  When it starts in OS/2 mode
it _needs_ to find the OS/2 COBLIB.DLL.  Just substituting the .DLE
won't work.
 
>     This assumes that you are using the INT/GNT runtime, not EXE's.

MS Cobol doesn't supply RUN.EXE to execute INT/GNT, it only produces
.OBJs which are linked to .EXEs for DOS, or .EXE and .DLLs for OS/2. 
These, however, may be 'int code' by specifying OPT"0".

You may note that with MF Workbench Cobol the RUN.EXE run-time for
INT/GNT is part of the OSX and requires a run-time licence to
distribute.  MS Cobol has no mechanism to collect run-time licences as
whatever can be built can be distributed freely.  This means that
[mostly] it can't build anything that MF would require a run-time
licence for.
```

The compiler is a 'family' application that can execute under DOS or
OS/2 1.x, but requires different DLLs depending on which it chooses. 
FORCEDOS hides the OS/2 facility in NT/W2K and thus makes the compiler
'choose' to run as a DOS program.

LINK is not a family application, it is DOS only.  The linker for OS/2
is a different program that is in the OS/2 SDK.

XM is also a DOS only program and thus using this to run your program
would ensure it ran as DOS even if you had managed to build it as a
family app (which is unlikely to be done accidentally).

There are a few other utilities in MS/MF Cobol that are family apps. 
Earlier versions got confused under some protected mode DOS
environments, such as Multiuser-DOS.
```
