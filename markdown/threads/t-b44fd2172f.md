      b)   takes least programmer time

For some reasons, my bosses over the years would prefer a poorly optimized
program that works over a highly optimized program that doesn't work.

I try to encourage the programmers I work with to use numeric data types where
practical, especially for subscripts and accumulators (i.e., fields for
accumulating various totals), no matter what the language. Just changing from
USAGE IS DISPLAY to USAGE IS COMP (for subscripts) or COMP-3 (for accumulators)
can have a big impact on speeding up program execution with minimal to no
impact on code maintainability.

As far as unrolling loops go, I figure that is the responsibility of the
compiler, not the programmer--let the compiler unroll PERFORMs and loops if it
wants to, but keep the source easily maintainable!

In a business shop (as opposed to scientific/statistic), one has to balance out
the amount of effort and time ($) to tune a program against the anticipated
savings for the life of the program, including maintainability of the code.

In some cases, I have gotten better performance by tuning buffers (VSAM KSDS:
BUFNI= and BUFND=; Datacom/DB VSAM/T: tuning VITs; SAM and VSAM ESDS: adjusting
blocksizes or CISIZEs) than I would ever be able to achieve by adjusting the
code.

In a scientific/statistics setting where programs could run for hours doing
number crunching (as opposed to being I/O intensive), adjusting code may have
big pay-backs.
