printers have a predefined character set which may not be compatible
with your mainframe's defined character set. In the past this has
required either a new printer chain (did I get that right?), a new font

In what character set (codepage) have you encoded the special characters?
Big-5? GB2312? Unicode?

The first step in this process is understanding character sets. Just like
we all know that in ASCII a hex 41 represents an "A", we have to have some
assumption about what a hex C0 represents, for instance. There are various
sets of these, called codepages. For Far East characters, there are
multi-byte codepages, because it takes more than one byte to represent all
the possible characters. Then there's Unicode, that can represent all of
these characters (Cyrillic, Japanese, Greek, etc.) in one scheme.

A printer operates in a particular character set. You can often change its
current character set with an escape sequence. But if you're trying to
print a character that isn't built into the printer, then the printer
needs some help -- some software needs to either preload a soft font into
the printer, or print those characters as bitmaps. This is what the
printer driver in Windows does.

One way to solve your problem is to use the capabilities in Windows, by
routing your print job through a Windows PC. We have a couple of products
that can do this. AnzioWin, our Windows telnet client, has passthrough
print that supports various encodings. Our freestanding Print Wizard
product can also handle special characters. Both do automatic font
selection. More information is available on our web site below, or you can
contact me privately.
