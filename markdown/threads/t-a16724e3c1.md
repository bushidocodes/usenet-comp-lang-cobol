```
Define a field as PIC  X(04) then, redefine this field as POINTER.

03  WS-POINTER-X PIC  X(04).
03  WS-POINTER REDEFINES WS-POINTER-X POINTER.
*
MOVE X'09870102' TO WS-POINTER-X.
*
If you're COBOL Version/Release is OS/390 2.2 or greater, you can SET the
ADDRESS OF Working-Storage items via a POINTER.

For any Version/Release less than OS390/390 2.2, you'd have to write a
sub-program that gets Called, passing the name of the WS item and a WS
work-pointer. The sub-program's job would be to set the passed WS
work-pointer to the address of the passed WS item and return to the Caller.
The coding required for such an endeavour is minimal at best.

As far as support for other compilers, perhaps others might have this
information.

Bill

Bill's example should be correct for IBM mainframe COBOL environments,
but I am curious why you would want to address your own working
storage with a pointer.  You already have addressability to it by
name, which is how you would also "pass" working storage items to
called programs.

If you want to address your entire working storage section with one
pointer, there may be problems with uninitialized working storage
being allocated at run time, and not located within the
working-storage layout as it exists in the COBOL source code file. 
Your working-storage section is not necessarily one contiguous block
of memory in the load module.

The problem with addressibility by name is that it is bound at compile 
time.  To get any form of late binding you need to use pointers.  A 
common example might be processing a blob of CR/LF delimited data of 
unknown size.

Outside of IBM environments, this can be fixed with a nested program 
that accepts two parameters and sets one to the address of the other.

>Bill's example should be correct for IBM mainframe COBOL environments,
>but I am curious why you would want to address your own working
>storage with a pointer.  You already have addressability to it by
>name, which is how you would also "pass" working storage items to

Here is an example of why you might want to address working-storage with a
pointer. Suppose you call the OS to get an environment variable. If it doesn't
exist you want to use a default in working-storage.

01  mypath-pointer  pointer.
01  default-mypath  value './*.txt' & x'00'  pic x(8).

call 'cbl-getparm' using 'mypath' & x'00', mypath-pointer
if  mypath-pointer equal to null
    set mypath-pointer to address of default-mypath
end-if

When would you want to set a pointer to an absolute address? When you have a
memory-mapped device, such as the DOS-mode display adaptor at B800:0.
```

I do NOT think that it is correct that current releases of COBOL allow you
to set the address of a WS item to a pointer.  It is true that COBOL for
OS/390 & VM V2R2 *added* support for setting a pointer to the address of a
WS item - but I don't think that it works the other direction.  I think that
you may ONLY set the address of a Linkage Section item to a pointer.

I have *not* checked the manual on this, so I could be mistaken.
