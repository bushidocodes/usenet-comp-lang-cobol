I am attempting to work with a COBOL II program which is not structured
very well, and I need some recommendations.  Unfortunately, I cannot take
the time to rewrite a substantial portion of the code.  So, here is how
the problem area is structured:  

   PERFORM 100-SORT-ROUTINE.     

100-SORT-ROUTINE SECTION. 

   SORT SORT-FILE 
       USING TEMP-FILE  
       OUTPUT PROCEDURE IS 200-SORT-RECS.

200-SORT-RECS SECTION.  

   PERFORM 210-RELEASE-SORT-REC. 
   IF  END-OF-SORT-FILE
       NEXT SENTENCE
   ELSE
       PERFORM 300-BATCH-HEADER. 
200-END.  EXIT.   

210-RELEASE-SORT-REC SECTION.       
   RELEASE SORT-REC
      AT END MOVE 'X' TO WS-SORT-END-OF-FILE.
210-END.  EXIT.

300-MATCH-HEADER SECTION.

   (ALOT OF IF LOGIC CHECKING FOR SOME SPECIFIC CONDITIONS.  NOTHING OUT OF THE
    ORDINARY)

   MOVE SORT-REC TO WS-HOLD-REC.
   PERFORM 200-SORT-RECS.
300-END.  EXIT.

Yuk!  Yes, the PERFORM 200-SORT-RECS at the end of the 300-MATCH-HEADER
routine is of course "piss poor" structure.  My dilemma is that I have
been abending with "Unsucessful Sort 322 S", and then the usual S322
abend.  Through displays, I discovered that after the last Sort Record is
processed, right after the PERFORM 300-BATCH-HEADER statement in the
200-SORT-RECS routine, I displayed the last Sort Record processed (a
trailer record), along with the variable
WS-SORT-END-OF-FILE, and it had a value of 'X".  These Displays were
subsequently repeated, repeated, repeated, etc.  That was the place which
resulted in the S322.

Does anyone know a simple solution to this awful code?  I would really
appreciate it.

Thanks,
Arthur M. Kent

```
To begin with your OUTPUT PROCEDURE needs to be doing a RETURN not a RELEASE. 
(RELEASE is for the input into the SORT, RETURN is fot the output back from the
SORT).

Also the reinvocation of the output procedure is not merely poor it is a
formula for disaster.  There is some chance that you will never achieve
    END-OF-SORT-FILE
because it will only happen on RETURNs, and you don't appear to be doing any. 


Replacing RELEASE with RETURN will help, but you should use a PERFORM UNTIL of
the returning paragraph in the basic OUTPUT procedure.  Do not recurse. In this
case the recursion is

200-SORT-RECS SECTION  
PERFORMs 300-BATCH-HEADER 

300-MATCH-HEADER SECTION
PERFORMs 200-SORT-RECS

On a CPU with stack architecture you could blow stack space this way when
sorting large files. The mainframe 322 is a variation of a time-out (It is a
tribute to the compiler runtime and sort routine that it can get that far. It
actual illustrates the advantage of non-stack architecture).

The recursion does not cause the time-out. The lack of an END condition does.
But you are at substatial risk doing recursion in a SORT context on a
mainframe. You could consume gigantic resources.

Here you timeout because you are succeeding too much in senselessly RELEASing
records.
You actually probably get an error on those release statements, since the
releases phase is over as of commencement of the OUTPUT PROCEDURE.  

But interestingly enough the AT END on the erroroneous RELEASE statement is
blind to errors.  An object lesson.  Errors are NOT AT END!  Within NOT AT END
you can interrogate file statuses, should you choose to, on regular files.  I
am not sure such syntax is available to you in COBOL II on SORT RETURNs.

But you will be much better off if you do RETURNs.

Other than the inappropriate recursion, your code does not look that badly
structured.  You have the rudiments of file status checking, which you might
want to expand. (With a check of SORT-RETURN at the top when done.)

Best Wishes,

Robert Rayhawk
RKRayhawk@aol.com
```
