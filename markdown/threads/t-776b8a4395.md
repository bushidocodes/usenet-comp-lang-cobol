quite well known for its "native language" support for GUI environments.

character based to GUI - without "re-thinking" the interface, is likely to

There are numerous methods to accomplish this. Both using ACUCOBOL-GT and
via third party software.

I don't know third party software too good, but ACUCOBOL-GT provides various
paths to migrate from character to gui. We also provide guides in how to
accomplish this as simple as possible, as well as there is built in
mechanisms in the runtime and AcuBench to semi automate the task of
transferring a character application to GUI. Best of all, is that you can
choose to merge at your own pace, screen by screen.

I have myself been part of a migration of 40 Mb source from character to GUI
some years ago, the move did not introduce many problems.

> I would like to know what are the problems that I should look for !!

The most obvious question is whether you are a non English or not. For non
English applications one have to take into consideration ASCII versus ANSI
issues.

Regardless of approach you go for, I would strongly recommend you to use
much time on prototyping before you start actual implementation. Although
gui seem simple with modern design tools, it really is a challenge to make
an efficient graphical user interface.

You should also consider the fact that gui is event driven, you may pretty
much wrap up the event handling into traditional display/accept statements
in Acucobol, however, it does pay off to think it over a couple of times.
This again, must be seen in the context of what one wishes to accomplish.

> I think the conversion from Vision to ORACLE will be easy to do, but
> the conversion from plain text screen to windows screen, that's the
> part that's scares me the most.

Fear not, how about this:

    Character:
        ACCEPT MY-VAR LINE 10 COLUMN 20.

    Gui:
        ACCEPT ENTRY-FIELD USING MY-VAR LINE 10 COLUMN 20.

Or how about a message box?

    DISPLAY MESSAGE BOX "Hello world".

This is however the simplest approach utilizing the sequential display
accept method. I would recommend using SCREEN SECTION, which will greatly
pay off in later maintenance and enhancement. Actually, if your application
alread is in SCREEN SECTIONs, you're very close to have a running gui app.

> I would like to have some hints on the matter !!

It is such a vast area, I think you should start looking into it, and just
ship specific issues as you experience them, that would be much easier to
handle. There are several developers using Acucobol-GT that monitors this
NG, and I am pretty sure you will get a lot of good advises on your way.
