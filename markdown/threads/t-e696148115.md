Assuming what you are addressing is migration of older IBM mainframe
COBOL's to IBM VS COBOL II, probably for operation on a MVS/ESA platform.

The following might be usefull steps:

1. Determine what COBOL each application component was last implemented
with.

In many situations, I have seen COBOL(F), ANSI Version 2, ANSI
Version 3,
ANSI Version 4, OS/VS COBOL R1, and OS/VS COBOL R2.

2. For OS/VS COBOL R2, determine what setting of the LANGLVL compile
option was used at the last implementation.

3. Isolate any application components ( CALLed or CALLer ) which were not
written in COBOL.

4. Determine the spread of subroutines which are used in multiple
application programs.

5. Isolate any dynamic CALL constructs in the application.

6. Determine any subsystems which are involved in these applications-
i..e. CICS, IDMS, IMS, etc

7. Obtain necessary education on the differences between the various
compiler's, and appropriate migration strategies.

8. Obtain source converter tool(s) if needed.

9. Convert, recompile, and verify each application component.

10. Deploy the converted application.

Steps 1 thru 6 of this exercise can be easily automated. In a matter of
hours a complete inventory of even the largest scale application shops can
be constructed using the Edge Portfolio Analyzer. Information on this
program from Carl Gehr, Edge Information Group, Inc. 513 948 8906 e-mail
734··.@com··e.com

Knowing precisely what you have to start with is a critical success
factor.

Once you know what you have to deal with, combined with appropriate
education, the actual conversion is typically easy. The deployment issues
are usually far greater than the code issues - i.e. testing, common
modules, subsystem considerations, RES / NORES issues, etc., etc.

Suitable education "Planning COBOL Migration" 3 - 4 days instructor led
course. Also from Edge Information Group.

A frequent big issue is finding the actual source code for a given
application - especially in these days of mergers, layoffs, contractors,
etc. Does this source match what we are actually running today???

TESTING COSTS WILL EXCEED ALL OTHER COSTS. Testing costs can exceed all
other costs COMBINED.

Moral of the story - consider migration to current COBOL's ( VS COBOL II
R4, COBOL/370 or COBOL for MVS and VM) as part of routine or scheduled
maintenance - i.e. while doing year 2000 upgrades.

Professional help - with lots of scar tissue in the hindquarters - is
available. Consider bringing in "spot" expertise, even if your own staff
will perform the migration. Why learn the hard way.

I think it would depend on how many programs you need to migrate.

If you only have a handful (less than 50 programs, maybe), just have the
programmers manually convert them. Changing the source code manually
might cost one or two hours of labor per program. Of course, you really
should re-test them before putting them into production. And you should
carefully examine the consequences of your new COBOL II compile time
options -- especially NUMPROC.

If you have a lot of programs, then there are a couple of options you can
consider. There are service bureaus which will perform automated
conversions for a fee. I've never used them, so I don't know what
pricing is like. You used to see them advertised in Enterprise Systems
Journal (formerly the Mainframe Journal).

If you have thousands of programs, then it might be cheaper to write your
own conversion program, or lease/purchase a product like IBM's CCCA (I
think it stood for CICS COBOL Conversion Aid). I haven't seen these
products advertised too many places, so it might be hard to shop around
for them. And the price might be pretty steep.

I wrote one for my company, but it doesn't convert the TRANSFORM verb and
it just does a character substition of the reserved word INSPECT for the
obsolete EXAMINE verb. It did handle CICS SERVICE RELOAD statements
pretty well. If you're interested, send me an e-mail.

I hope this helps.
