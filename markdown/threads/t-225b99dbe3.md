There is no platform specific reason for this kind of behavior, however, NT
is a lot more robust than Windows 95, and threats the execution
significantly different. The build itself should not cause any "downsizing"
issues, what is common however, is that an application may behave excellent
on NT while it blows on 95.

There are basically two reasons for this:

Windows 95 still have a limited heap, while it is way more than the
ridiculous size in Win 3.x it is still limited, if your application are
using resources like a monstertruck would use gas, it would keep alive for a
significant time under NT, while blowing shortly on 95.

Any application on NT has its own protected layer, separating it from other
processes, this means that a "wild" pointer address in an application under
NT is less likely to blow the application than if executed on 95 which is
not half as good as in protecting the applications / OSs.

My general believe is then that you may check out your program and/or add on
products you use. The most efficient way of doing this is to use a utility
like Boundschecker or Purify as host environment, those products will
efficiently track down and tell you which module that is behaving badly.

There is also a minor chance that your binaries has been somewhat damaged,
but in that case your computer shouldn't be stable either.
