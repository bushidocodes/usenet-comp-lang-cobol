Huh?

First, "COPY1" is meaningless at run time. "COPY1" is used only by the
compiler. COPY1 is the name of something used at compile time (a copy book)
and the name COPY1 does not survive the compile process.

Linkage Sections are used to map data from the calling program. Don't use a
Linkage Section in a calling program - only use Linkage Section in a called
program.

You see, a Linkage Section does not allocate memory, it only passes the
location of memory already defined by the calling program. A Linkage Section
uses no memory itself, nor are values stored there. The Linkage Section only
defines the layout of the data in the calling program.

Memory is assigned by Working Storage in program "A".

The calling program, "A" calls program "B" via CALL 'B' USING MYPARAM. But
what is actually passed is the address of MYPARM. Now when "B" gets control,
it uses its Linkage Section to define the layout of stuff beginning at the
passed address (the address of MYPARAM). Specifically,

"A" (calling program)
WORKING-STORAGE SECTION.

COPY COPY1.
   (01  MYPARAM.
       02  MYPARAM-1   PIC X(50).
        02  MYPARAM-2  PIC 9(5).     )

PROCEDURE DIVISION.
...
    CALL 'B' USING MYPARM.    *>not COPY1


"B" (called program)

WORKING-STORAGE SECTION.
01  FROG   PIC XXX.
01  PEANUT  PIC 9(3).
(other items as needed)

LINKAGE SECTION.
COPY COPY1

PROCEDURE DIVISION USING MYPARM. *> not COPY1
...
MOVE 'HELLO WORLD' TO MYPARM-1
...
GOBACK.




>
> In b.cbl, to refer the variable, I executed
> set CD-INIT-COMPOSE IN CD to true  (Is this way to refer the variable
> alright?)

No. You've defined CD-INIT-COMPOSE as alphanumeric. Alphanumeric variables
cannot take on the values "TRUE" or "FALSE," nor can they be SET (as in SET
MY-INDX UP BY 1). You should
    MOVE 'INIT-COMPOSE' TO CD-INIT-COMPOSE.
