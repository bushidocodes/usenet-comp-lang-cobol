```
If indeed the compiler properly implements the switch to check compatibility
with the standard, then the information you get from test runs is indeed
more interesting, but be warned, this switch is (unfortunately) not often
used, not tested by validation as far as I know, and often not implemented
very accurately.

But still .... you will NOT get accurate information this way. There is
NO SUBSTITUTE FOR READING THE STANDARD, I don't know if Andy has a copy
of the standard, probably not, he seems intent on not needing to read it,
but that attitude is to me unacceptable.

Any language standard includes

o Non-deterministic features, where more than one behavior is permitted
o Implementation dependent features

You ABSOLUTELY CANNOT get information in these cases as to what the standard
says from your compiler, which will unavoidably provide an over-specified
interpretation.

For example, suppose you want to figure out what the exact standard semantics
of COMPUTE are in COBOL. You will get totally misleading information by
running a program on your compiler, and you may totally fail to realize the
importance of avoiding most uses of COMPUTE in portable programs (this is
a case where the semantics of COBOL is to a considerable degree
implementation dependent).

The trouble is that if you don't know the standard, you can't know when you
are running across either of these two cases.

For example, suppose in the case under discussion the standard said:

All compilers must support a minimum of 30 character names, but may
accept longer names. The maximum length of names supported must be
documented in the compiler documentation.

It DOESN'T say this, but you can see that if it did, you would not be able
to get information even on a simple syntactic matter like this.

There are many examples of implementation dependent behavior in COBOL,
another familiar example is the representation of embedded signs, and
the native collating sequence.
```

"importance of avoiding COMPUTE"

the reason is simple, the result you get from a complex compute is
implementation defined, and you try to avoid implementation defined
semantics if you are trying to write portable programs:

multiply a by b giving c
divide d into c giving e

is completely well defined

compute e = (a * b) / d

assigns an implementation defined result to e
