I'm looking for opinions regarding the best method of designing and
implementing a character based user interface. Here is some background
info:

OS       - HP UX 11i
COBOL    - MF Object Cobol 4.2
DATABASE - Oracle 9i


I am working with an extremely large existing application (which I
inherited responsibility for) an I am unhappy with how this
application handles the user interfaces, so I'd like to solicit
opinions on possible better approaches.

Here is a snip of psuedo-code for one of our very basic
add/change/delete application on the existing system:

LOOP

  GATHER DATA FOR DISPLAY

  DISPLAY SCREEN-1

  DISPLAY SCREEN-1B

  ACCEPT SCREEN-1C

  EVALUATE USER KEY PRESS

  ACT APPROPRIATELY

END-LOOP

SCREEN-1 is a 1920 byte 01 level data item in the working storage
section, and SCREEN-1B and SCREEN-1C simply redefine SCREEN-1. The
logic behind this is so that different elements of the screen can be
display in varying colors.

This is very cumbersome, as EVERY program that requires user
interaction now needs to carry the display logic, as well as the user
keypress evaluations, etc.

I'd like to separate things into a more "tiered" approach, but this
being my first real chance to venture into this area, I'm not sure the
best way to start out.

Any suggestions/recommendations/pointers would be greatly appreciated.

I've looked into MF's Dialog and Screens utilities, but they are
extremely cumbersome to deal with - I'd like to avoid them if
possible.

Thanks in advance!

Chris

1. Not sure on this - but Bob Wolfe at Flexus will have to confirm - I
*think* his SP2 software allows you to go character based. The Screen biz
is "tiered" - you call SP2 from your COBOL programs. (???)

2. Judson McClendon is a Screen Section guru using M/F - suggestions
Judson ?

3. From the way you've described it, looks like you've got your Screen
Section in the 'Business Logic' program. Hive off Screen Section as
separate programs that you can call with entry points - to get different
layouts for the same screen. In a sort of hierachial way, have a
Screen1.cbl which could in turn call ScreenAttributes.cbl to set say
colours. To get full advantage of the Mouse in character based - you would
need to take a look at Admouse.cbl (If you haven't got it, the text for
that is in Programming -> Character Based Interface for N/E V 4.0 - see
http://supportline.microfocus.com/supportline/documentation/books/nx40/nx40indx.htm).

Using Dialog System in GUI mode of course means a learning curve, and you
have got to get into Window event handling. But I note the above
referenced books refer to Character Based use of Dialog system - do you
have that in your M/F O V 4.2 ?

One thought - did the original designers go 'colour mad' ? Might simplify
things if you reduce colour, using it very selectively to highlight -
against that of course the users might moan, "Where are all our pretty
colours ?"

I don't think Screen Section is quite as cumbersome as you suggest - years
ago I had menu bars and drop-down menus designed in Screen Section - using
as the basis the M/F program Logper.cbl. (The text for the menus was held
in an ISAM file). Going back even further - to Radio Shack Model II days -
where you only had white on black - I was using box characters with
RM/COBOL to pretty-up a screen so that the display looked like a form.

Regardless of tools - the real success will come from a three-tiered
system approach - Business Logic, UI(User Interface), DBI(Database
Interface  - COBOL files or DBMS).

DISPLAY SCREEN-1
  FOREGROUND-COLOR 7
  BACKGROUND-COLOR 1

DISPLAY SCREEN-1B

  BACKGROUND-COLOR 1
  HIGHLIGHT.

ACCEPT SCREEN-1C

One immediate comment on your 01 Level Working Storage 'screen entries'. *IF* currently a
particular screen is being used in several programs - then make it a copy file. Now as to
the colouring - check the Screen syntax - your colour can be a  :-

Background Color - General Format

 BACKGROUND-COLOUR  IS   integer-1
 BACKGROUND-COLOUR  IS   identifier-1 .

So you can specify it as you have - or use an indentifier to move. I haven't tried it yet,
but something like :-

--------------
WORKING-STORAGE SECTION.

copy "mycolors.cpy".
*>------------- mycolors.cpy----------------
01 MyBackground            pic 9(02).
01 MyForeground            pic 9(02).

01 MyBlack                     value 6. (I *think*)
01 MyRed                        value ?
01 MyWhite                     value ?
*>---------------------------------------------------
Following syntax is not correct, but you'll get the idea :-

SCREEN SECTION.
copy "ascreen.cpy".
*>------------------ ascreen.cpy-------------------
01 AScreen foreground-color MyForeground
                   background-color MyBackground.
     05 etc........
*>--------------------------------------------------

Procedure Division.

move MyRed to MyForeground
move MyWhite to MyBackground
display Ascreen

*> with a three-tier approach this becomes something like -
*> assuming you already have the Screen program loaded :-

CALL 'DisplayIt'  using MyForeground, MyBackground
--------------------------------------------------

http://www.syntax.gr/departments/CobolSolutions/files/downloads/demos.htm

Hopefully Net Expres, using the same syntax you have,  is an M/F enhancement of what is
available in your OO COBOL. Rather than using entry points to CALL you could invoke classes
for Screen Programs and SQL Programs - nothing fancy,  just use handles (object references)

This works because the elementary _named_ items only are displayed and
not the FILLERs.  Presumably Screen-1 is PIC X(1920) and the others
are redefinitions with named fields where appropriate.

I have used this mechanism since the late 70s with CIS Cobol, though I
had to devise my own colouration mechanism.  Over the years I
developed code generators to create the screens and to generate blocks
of code to do the display and accepts and data moves.  The input is a
text file that has the screen layout with variable fields tagged with
'%' characters.  A list of field descriptors in the file is matched
sequentially with the tags and each field is thus given a name, its
origin, a type, and a prompt (displayed in line 25) plus some other
stuff.
