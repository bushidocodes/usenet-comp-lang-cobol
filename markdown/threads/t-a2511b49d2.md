I don't think you'll get much sympathy from this group on this
point, but I'm willing to speculate.  I'm not a long-term Cobol
users, but have sufficient experience with VB and Java.

I've not found any tools that purport to do this, but I suspect
they would be severely limited by the variety of coding styles
in use.

Some general thoughts, though:

Any cross-platform conversion of code will not be easy.  You
will have to inspect each line of source code by the end of the
project's conversion.

That said, however, the design of Cobol provides some benefits
in this area.

Namely, there little concept of "scope" in the language, so the
encapsulation of data structures will be relatively easy to
accomplish.  Some features of the language, such as unions
or redefines will present problems, but are resolvable.

Also, since there are no subroutines, the procedural code can
be "canned" fairly easily within the application.  A lot of code
aimed at moving data across program boundaries will no
longer be needed, but any attempt at automatically converting
this will be difficult.

A larger issues is the dynamics of the program.  Presumably,
your application follows a DOS-style of navigation.  Even if
you're using SP2 or another GUI designer, the underlying
code is procedural, not event-driven.  This is a significant
challenge.

A less ambitious approach may prove to be the best.  Cook
out the business rules, and use them as a specification for
your re-work.  Rather than carry vestigial presumptions
about presentation, it's better (in my opinion) to cut the
wheat from the chaff.

AcuCorp makes a reverse engineering tool to is said to be
amazing--at least by their sales force.  Flexus has an
excellent links page for those engaged in this enterprise.

Good luck.

Well, you certainly CAN write to the Windows API in COBOL; that's what
we did.  All you're doing is calling Windows functions from COBOL
using "C" calling conventions.  All of the mainstream compilers will
let you do this.

As for writing windows procedures, you can do that in COBOL, too,
provided that your compiler supports recursion.  I know that CA-Realia
and Merant do, and I believe the high-end Fujitsu compiler also does.
I'm not so sure about AcuCOBOL.  However, while it's easy to write a
window procedure (in COBOL or whatever) that doesn't abend, it's
another thing to actually get it to do the stuff you want it to do.
Lots of the Windows stuff is pretty subtle and finicky.  

IMHO, the subtleties of Windows API programming are just as obscure in
C as they are in COBOL.  There's also the little matter of figuring
out how to communicate between your window procedures and your
mainline application.  You can do it in C using global variables, but
you need to work out other strategies in COBOL.  It requires a solid
background in how software actually works.  If you don't understand
how arguments are passed and  what it means to obtain addressability
to an area of memory, you won't get very far, again, whether you use C
or COBOL.

The REAL key to low-level Windows programming is having a thorough
conceptual grasp of how it actually works - and that takes a lot of
time.  Sure, we got our first (crude) Windows apps up-and-running in
COBOL within 6 months of opening the Windows SDK documentation, but
they were, well, not too well done, and  I didn't really feel that I
had an intuitive grasp of Windows development until I'd been at it for
two or three years.  Today, after more than five years of 100% Windows
API programming - in COBOL - I can generally add new features or use
new Windows facilities within an hour or two of reading about them...
but I still feel I'm  learning new stuff about Windows every day...

Use a third party tool for your Windows UI to avoid this (painful)
learning process and so that you can concentrate on the application
instead of wasting all your time on the interface.  

As a rough indication of the difficulty of the Windows API, we rolled
out our first character-mode UI in 1985 after about 6 months/1,000
hours of development.  It changed little over its 15 year life.  It
was quite complete and solid.

Our GUI screen manager represents an investment of over 4 years/12,000
hours to date.  Sure, it does tremendously more than the old DOS
character mode product, but much of that time was spent in designing a
rational programming interface and actually implementing the Windows
stuff.  It was a complex and extremely challenging software project.
I wouldn't want to go through it again.

We did it in COBOL, by the way, and I'm convinced it would have taken
just as long (or longer) to develop it in any other language.  The
tricky stuff was grokking Windows, not actually implementing the
product.  Windows programming is hard.

So, don't underestimate the difficulty of implementing your UI at the
API level.  The whole point of a UI tool like ours is to hide all that
complexity from you and let you get on with your application.

If you need to call the API directly to step beyond what the UI tool
does, there's generally no reason why you can't do it.  It's
conceptually the same as calling an assembler routine to, say, perform
bitwise operations instead of doing 'em in COBOL.

If all you're after is picking up keystrokes, you could probably do it
at the Windows message-loop level of your program and not worry about
any but the most simple window procedure.

If you're going to do much more, though, such as create and use
standard Windows controls like edit controls, radiobuttons,
checkboxes, and such, you are going to find your life will rapidly
become complicated.  Anything but the simplest and most rudimentary
Windows application is going to require handling stuff in a window
procedure, and the more functionality you want, the more code you'll
have.  If you DON'T use the standard Windows controls, you're going to
need to deal with Windows WM_PAINT messages to redraw your window
contents on demand.  Either way, it takes some work on your part.


Now that I've told you how difficult it is <G>, I'll give you some
pointers:

If you haven't read it yet, a good place to start would be Charles
Petzold's "Programming Windows 95".  Don't take it as gospel, because
it's not really geared towards developing a general-purpose UI like
you want; it just touches the basics.  If you're like me, you'll be
completely befuddled the first time you wade through the first couple
of chapters, but after reading it a couple of times it begins to make
some sort of sense.  

Believe me, Windows programming is like nothing you've ever
encountered before!

Once you get the gist of what Petzold has to say, you'll want to
subscribe to the Microsoft Developer's Network.  That will get you all
of the documentation and libraries that you need.  (And a lot of stuff
you don't.)

getting your arms around THAT mess, and then you can start actually
developing some Windows code.

We also did a lot of work with the Micro Focus compiler around 5 years

Hi, Jimmy,

A minor, but significant, correction - GUI ScreenIO doesn't bother you
with Windows events, it returns "application" events -  stuff that's
of interest to you as the developer.  

What's an "application" event?  Events that you define when you design
your panel; things like menu selections, button presses, whether a
field should cause a return to your program when the user enters or
leaves it, that sort of thing.  You define these up front when you
create your panel in the editor.  You, therefore, define all of the
possible reasons that GUI ScreenIO will return to your program in
advance.  If you think about it, this is a logical way of addressing
the issue of user interactoins with a given screen.

Then, when you display your panel at runtime, the runtime won't return
to your program until one of your predefined events occurs; the user
selects a menu item, presses a button, or whatever.  (You can have it
return in other situations, of course, but you usually wouldn't need
to bother.)  

This is conceptually the same as any other transaction processing
system; where you send a screen to a terminal and wait for a user
action.

You literally are not aware that you're running under Windows because
we do ALL of the low-level stuff.  That, in my opinion, is the whole
point of a UI tool.  Developers can then concentrate on their
application instead of getting all tangled up in the details of the
UI.
