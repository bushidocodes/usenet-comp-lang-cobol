Here's an example of some code that we use.  Our application requires the
user enter a temporary directory.  Let's say they choose "c:\windows\Program
Files\Local Apps\Our App\Temp Directory" as the temporary directory.  I need
to know if this directory exists and if not I need to create it, but the
problem is you can only create one level deep at a time and since this
directory has multiple parts each part must be checked until I get to the
first directory in the path or I find a directory that exists.  So, the
solution is to pass into the routine a directory structure.  The routine
determines if the directory exists.  If so it returns with a good status,
otherwise it calls itself and continues to do so until it gets to the first
directory or finds one that exists.  If it has called itself, when it
returns it creates the directory and exits.  The result is I only write one
piece of code to create the direcotry and one piece to check for the
existence of a directory, and I don't care how many subdirectories the user
enters because the routine just keeps calling itself.  You could manage this
by parsing the original directory and counting the pieces and performing a
loop x number of times, but the recursion is more elegant and more
straightforward.

bill

: Is there a way to do recursion in cobol as it is done in the c language

The 'Programmer's Guide to Writing Programs' in MicroFocus Object
Cobol for Unix shows how to do recursion - ie it does support
this.  This, however, is supplier dependent.  Different versions
may or may not support recursion.

Anything that can be done recursively can be done iteritively,
usually faster.  The advantage of a recursive routine is that
it 'looks after' the storing of each recursion's local data
in the stack frame.  To convert to iteritive it is necessary
to build and look after your own stack and move up and down
this with each iteration.
