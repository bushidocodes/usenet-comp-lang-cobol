MVS/XA, COBOL II, AMODE=31, RMODE=ANY

One of our larger COBOL modules is abending with a S878 - region size problem. However, this
is occuring in the final called sub-module that produces some statistical reports and we are of
the opinion that the problem has been caused by a previous sub-module corrupting the load
module.

When we had our large tables defined as EXTERNAL we ran to completion, but were advised to
convert them to Linkage as it can be allocated above the line. This appears to have caused the
problem.

We have spent a lot of time checking all the linkages to ensure that they all line up and have
tried to analyse the call chain, but without finding the problem.

Does anyone know if there is a limit to the amount of linkage a load module can use and if this
would be less than that available to EXTERNAL storage below the line? Are there any
suggestions as to how we can track down the offending module? We have managed to limit our
input to 3 records that, when run in sequence, result in the abend.
```

The most common cause of S878 type abends is having a program attempt to
allocate storage in the 24 bit space (below the line) when you intend to
allocate the storage above the line.

In order for VS COBOL II, COBOL/370 and COBOL for MVS and VM (all mostly
the same technology) to allocate working storage above the line.

(with some less restrictive cases available in COBOL/370 and later)

Your code must be operating in AMODE(31) VS COBOL II code is normally
marked AMODE(31) capable by the compiler, but the attribute can be lost at
either linkage edit or execution time.

The code (which allocates the data) must be compiled with the RENT
(Reentrant) compiler options (also requires the RES compiler option).
This attribute makes the working storage external to the compiled code,
and obtained at execution time via (eventually) the MVS Getmain method.

The code (which contains the data) must be compiled with the DATA(31)
option.

There is a limit (as I recall) of 16 MB per 01 level data item in working
storage, but I am not aware of any practical limit - short of the ESA
limit of about 2,000 MB for the summation of all working storage.

S878 abends are MVS related - generally a below the line getmain was
issued for a larger amount than was available (max below the line is
typically only about 6 MB).

In complex applications, multiple module, dynamic calls, mixed technology,
etc, etc, it is easy to go wrong and come up with an application
configuration that is not what you expect. One way (subtle commercial #1)
to see what is going on is to examine the generated composite load modules
of the application. Various methods are available to do this, one of
which is the use of a product called the Edge Portfolio Analyzer (MVS
mainframe utility). It essentially dynamically interprets the load
modules, and tracks back to compiler options, attributes, etc.

A thorough discussion of this technology is far beyond what I can pound
into this board, it takes (subtle commercial #2) about 1 day's lecture to
explain most of the COBOL II / COBOL/370 environmental considerations in a
class (Planning COBOL Migration) also offered by Edge Information Group
(information from Carl Gehr 513 948 8906 / 734··.@com··e.com).

As for the specific problem, the critical issues are the CALL chain,
specifically what technology the members are, what compile time options
are contained in the chain, and the connectivity method used.

Storage passed as linkage can be either above or below the 16 MB line.
All linkage storage must originate in Working Storage (or perhaps File)
section. Once all other considerations have been satisfied (again with a
footnote or 2) the location of Working Storage is determined by the
earlier mentioned DATA parameter.

Long, windy answer, just touches the surface contact me directly via
e-mail if I can provide further details.
```

There are limits -- based usually on JCL and the MVS install.

If you haven't done anything, then you're limited to what's avail below the line and 32MB
above. This is the default. Most shops install an exit to increase this, which puts it
into the control of the JCL (check with the MVS systems programmer).

You have a REGION= which can be on the JOB card and/or the EXEC card.

Oh yeah -- if you have a REGION=6M on the job card, the step card is apparently ignored!

Region=xM, where

x < 16, means the standard (32M above the line) and whatever you
ask for below the line. Usually, you can't get more than
7 or 8.

x = 0, means give me everything, both above and below.

x >= 16 means give me everything available below the line and
what I asked for above the line.

My problem has usually been a REGION=6M on the JOB card, which overrides
the REGION=48M on the EXEC card.
