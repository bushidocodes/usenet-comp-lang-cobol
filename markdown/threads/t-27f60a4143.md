```
                 02 PIC X                    VALUE SPACES.

                 02 PIC X(5)                 VALUE SPACES.

                 02                      PIC X(3) VALUE SPACES.
                 02 O-UNIT-COST          PIC $ZZZ.99.
                 02                      PIC X(4) VALUE SPACES.
                 02 O-EX-COST            PIC $ZZ,ZZZ.99.

              01 AVERAGE-LINE.
                 02                     PIC X(7) VALUE "AVERAGE".
                 02 O-ITEM-COUNTER             PIC 999.
                 02                     PIC XX VALUE SPACES.
                 02 O-AVERAGE           PIC $ZZZ,ZZZ.99.

              01 TOTAL-LINE-1.
                 02                      PIC X(6) VALUE "TOTALS".
                 02                      PIC X(5) VALUE SPACES.

                 02                      PIC X(3) VALUE SPACES.

                 02                      PIC X(3) VALUE SPACES.
                 02 O-TOTAL-EX-COST      PIC $ZZZ,ZZZ,ZZZ.99.

       1000-MAIN-CONTROL.

       PERFORM INIT-PARA
       PERFORM UNTIL EOF-FLAG = "YES"
                 IF IN-STOCK-NUM NOT = HOLD-STOCK-NUM
                         PERFORM DESCRIPTION-BREAK-PARA
                 END-IF
                 PERFORM PROCESS-PARA
                 PERFORM READ-PARA
       end-perform
       perform description-break-para
       perform END-PARA
                 STOP RUN.

       INIT-PARA.
                  OPEN INPUT STOCK-FILE
                  OPEN OUTPUT REPORT-FILE
                  PERFORM READ-PARA
                  WRITE REPORT-RECORD FROM PAGE-HEADING
                  AFTER ADVANCING 2 LINES.
                  WRITE REPORT-RECORD FROM HEAD-LINE
                  AFTER ADVANCING 2 LINES.
                  MOVE IN-STOCK-NUM TO HOLD-STOCK-NUM.

       PROCESS-PARA.
                 MOVE IN-STOCK-NUM TO O-STOCK-NUM
                 MOVE IN-DESCRIPTION TO O-DESCRIPTION
                 MOVE IN-QUANTITY TO O-QUANTITY
                 MOVE IN-UNIT-COST TO O-UNIT-COST
                 COMPUTE QUANTITY-TIMES-UNIT-COST = IN-QUANTITY
                 *IN-UNIT-COST
                 MOVE QUANTITY-TIMES-UNIT-COST TO O-EX-COST
                 ADD IN-QUANTITY TO TOTAL-QUANTITY
                 ADD IN-UNIT-COST TO TOTAL-UNIT-COST
                 ADD QUANTITY-TIMES-UNIT-COST TO TOTAL-EX-COST
                 ADD 1 TO ITEM-COUNTER
                 WRITE REPORT-RECORD FROM DETAIL-LINE
                 AFTER ADVANCING 2 LINES.

       DESCRIPTION-BREAK-PARA.
                 MOVE ITEM-COUNTER TO O-ITEM-COUNTER
                 MOVE TOTAL-QUANTITY TO O-TOTAL-QUANTITY
                 MOVE TOTAL-UNIT-COST TO O-TOTAL-UNIT-COST
                 MOVE TOTAL-EX-COST   TO O-TOTAL-EX-COST
                 COMPUTE AVERAGE=TOTAL-EX-COST / ITEM-COUNTER
                 MOVE AVERAGE TO O-AVERAGE
                 WRITE REPORT-RECORD FROM TOTAL-LINE-1 AFTER
                 ADVANCING 2 LINES.
                 WRITE REPORT-RECORD FROM AVERAGE-LINE AFTER
                 ADVANCING 2 LINES.

                 MOVE 0 TO AVERAGE
                 MOVE 0 TO ITEM-COUNTER
                 MOVE 0 TO TOTAL-QUANTITY
                 MOVE 0 TO TOTAL-UNIT-COST
                 MOVE 0 TO TOTAL-EX-COST
                 MOVE IN-STOCK-NUM TO HOLD-STOCK-NUM.


       READ-PARA.
                READ STOCK-FILE
                      AT END MOVE "YES" TO EOF-FLAG
                END-READ.

       END-PARA.

                CLOSE STOCK-FILE
                CLOSE REPORT-FILE
```

```
Just glancing through it quickly, (and figuring formatting looks a bit
odd on the web), I say it looks more professional than half of what I
see in the industry.  (I don't know about "proffessional").  Your style
is simple, structured, and likely to be correct.  I did wonder though
why you had one paragraph name with a number prefix, but not the rest. 
The main reason we use number prefixes is to make it easy to find
paragraphs quickly.  (I like being able to look for a numeric character
in column 8 besides being able to guess where 9000-READ-MYFILE is).

My own stylistic choices also don't care for paragraph names such as
READ-PARA.  We know it's a paragraph, instead I prefer
READ-ACCOUNT-FILE.    When you have multiple input files, that's more
useful.

I have never seen the closes in a paragraph by itself.

I like having a comment in the last line saying ***** CH8EX1 procedure
division ends here ******.   This is a practice which was common in the
days of punched cards, but is still useful when you are editing the
compile listing and want to get to the last line of the source code.

Shop standards also usually have something such as this where REMARKS
used to be:
000500*****************************************************************
000600**                                                             **
```

```
>I have never seen the closes in a paragraph by itself.
>

I do this so that file closings are easily performed for both normal and (user
conrolled) abnormal terminations. The paragraph that opens the files sets a
switch to
true. At normal termination, the close paragraph is performed. The abnormal
termination paragraph checks the switch and performs the close paragraph when
necessary before calling for an ABEND. I've heard this referred to as a
"graceful exit." It also performs paragraphs that display I/O counts & current
date/time. This is useful
info for tracking down the problem and differentiating between messages issued
by multiple occurrences of the same ABEND in the same production cycle. In
cases
of large file/long running programs, I also
set up the output files as CATLG,CATLG. Sometimes the contents of output files
can help determine the cause of the problem.
When you're figuring out an ABEND that's holding up the rest of the daily
production cycle at 2:00 in the morning, there's no such thing as too much
information!
```

```
Breaking up paragraph numbering can be taken to extremes. Mostly I only

0000-   main processing
```

```
This is almost as "religious" a topic as what editor to use or whether to
use Paragraphs or Sections.  You will get highly animated replies related to
this suggestions both PRO and CON).

On IBM mainframes (for example), it is almost (but not quite) universal to
include AND CHECK file status codes for all indexed (VSAM) files - but is
highly unusual (and often frowned upon) to include a file status field for
sequential (QSAM) files.  There are (valid?) reasons for doing this - having
to do with how the (proprietary) COBOL system works with the (proprietary)
operating system.

The bottom-line is that *IF* you reference a file-status field in your
Select/Assign clause - then you *MUST* check it after every I/O operation -
and be aware of the following:

1) Not all "successful" I/O results in an '00' file status; other 0x values
are successful - but meaningful - if you care.

2) All file status codes in the 9x range are implementor defined and you
need to understand how to decipher and handle them. If you are attempting to
create "portable" code, these may well be a problem.

3) *NEW* file status codes *can* be added when maintain occurs to your
compiler or run-time (because of ANSI/ISO Standard changes or new 9x values
by the implementor).  If you use file status code checking, try to put it in
"common  code" (called routine or COPY statement) that can EASILY be
enhanced to handle new values.
```
