In general, C/C++ is very poor for handling money, because it has no
native scaled decimal math.  True, objects could be written to do
this, but even then it would be clumsy, probably inefficient, and
may not be reliable, compared to the built in and well tested native
compiler routines.

It is my opinion that people who want to add features like explicit
pointers to COBOL misapprehend one of COBOL's primary strengths:
its simplicity.  Using pointers can be very subtle and fraught with
danger.  I suspect there are probably more bugs in C/C++ code related
to pointers than all other things put together.  While subscripts and
indexes aren't as widely flexible as pointers, they are far safer to
use, and much less subtle.  I've yet to write an application in COBOL
(and I have written quite a few widely varied COBOL applications,
including a compiler and p-code interpreter) where I felt the need for
explicit pointers.  Simplicity = fewer bugs = less cost + better
reliability.  Windows is written mostly in C/C++, and many heavy duty
Windows applications are written in C/C++.  What is your own personal
experience with the reliability of those, versus applications written
in COBOL?  How many 25 year old C applications do you know of that are
still running? ;-)
