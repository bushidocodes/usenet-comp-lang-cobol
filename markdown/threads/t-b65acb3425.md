Are you trying to display the hex value of 13000 as 32C8? If that's the
case, you need to put it in a PIC X(4), not a PIC X(2). There's probably a
function to do this, but I don't know what it is. You could write a module
to make sure the input is 65535 or below. Then divide by 4096, then divide
the remainder by 256, then divide the remainder by 16. These values, and the
final remainder, will all be values from 0 thru 15. Convert 10 - 15 to A -
F, then assemble using reference modification. (If the output is to be
signed, make sure the input is 32767 or below.)

If, instead, you want 13000 to be converted interally to hex, just move it
to a PIC 9(5) COMP. However, I believe this is compiler dependant. On a
mainframe, this will actually only take up 2 storage bytes, even though it
is defined as (5). BINARY (in stead of COMP) may work for all compilers.
```

Assuming an Intel platform:

01  MYHEX-GROUP.
  02  FILLER            PIC X(2).
  02  MYHEX-NUMBER  PIC X(2).
01  MYRAW-NUMBER REDEFINES MYHEX-GROUP PIC S9(7) COMP-4.

MOVE 13000 TO MYRAW-NUMBER
   (MYHEX-GROUP now contains 00 00 32 68)
MOVE MYHEX-NUMBER TO anywhere-you-want as a 2-byte field.

Note the COMP-4 definition above may be compiler-dependent. You want a
representation that is straight binary.
```

01  mynumeric pic 99999 value 13000.
01  myinteger pic 9(4) comp.
01  myhex redefines myinteger pic xx.


compute myinteger = mynumeric.
or 
add 0 mynumeric giving myinteger
or
move mynumeric to myinteger.

you'll have to find how to flip them back. The largest 5 digit number
you can use is 64KB.

some compilers may have to be forced to NOT truncate at 4 digits.

Divide by 16 taking the remainder, use the remainder+1 as an index into
a table with '0123456789ABCDEF'. Each divide give the next low-order
digit so decrement the output index.

eg:

        PERFORM 
            VARYING Hex-Digit FROM 4 BY -1
              UNTIL Hex-Digit < 1

            DIVIDE Numeric-Value BY 16
                      GIVING Numeric-Value
                   REMAINDER Digit-Index
            ADD 1           TO Digit-Index
            Move Hex-Char(Digit-Index) TO Hex-Value(Hex-Digit)
         END-PERFORM

or similar.
```

This may be made more flexible with a few changes.

Define the maximum possible unsigned integer as input,
an alphanumeric capable of holding the binary character
representation as output, and a radix value for control.

1 numeric-value comp pic 9(18). *> assumes 64-bits
1 alpha-value pic x(64).
1 radix comp pic 99.

1 digit-index comp pic 99.
1 digit-value comp pic 99.
1 char-value pic x(16) value "0123456789ABCDEF".

convert-number-to-character.
    move all "0" to alpha-value
    perform varying digit-index from 1 by 1
      until digit-index > 64
            or numeric-value = 0
        divide numeric-value by radix
          giving numeric-value
          remainder digit-value
        add 1 to digit value
        move char-value (digit-value:1)
          to alpha-value (digit-index:1)
    end-perform.

The result, in alpha-value, is reversed from normal representation,
so the user reverses the part needed.

    move 13000 to numeric-value
    move 16 to radix
    perform convert-number-to-character
    move function reverse (alpha-value (1:4))
      to my-hex-value

my-hex-value has a value of "32C8".

Radix values of more than 16 may be used by extending
char-value to include additional values. Including the
remaining capital letters allows for radix 36.

Furthermore, adding formating routines that use alpha-value
as input allows more flexibility for display. For example,
a routine to format hexidecimal by inserting spaces after
pairs from alpha-value provides a familiar 'dump' pattern.

1 formatted-hex-display pic x(24).
1 format-index comp pic 99.

format-hex.
    move 1 to format-index
    perform varying digit-index from 1 by 2
      until digit-index > 16
        move alpha-value (digit-index:2)
          to formatted-hex-display (format-index:3)
        add 3 to format-index
    end-perform.

Then, after the perform of convert-number-to-character;

    perform format-hex
    move function reverse (formatted-hex-display(1:11))
        to my-hex-display

shows "00 00 32 C8".

Some formatting problems may be solved more easily by
reversing the values before formatting. Translation of
internal numeric forms to external forms is only one of
the problems where this seems to work well.

Note: The above code snippets were not tested.

```
Here's a solution for an IBM Mainframe that will convert/display a numeric
value up to 2147483647 (X'7FFFFFFF') as a 2-8 byte EBCDIC value. The COBOL
Compile option TRUNC(BIN) should be specified to avoid high-order truncation
of binary data.

03  WS-PACKED PIC 9(08)V9 PACKED-DECIMAL.
03  FILLER REDEFINES WS-PACKED.
      05  WS-FWORD PIC 9(09) BINARY.
      05  FILLER PIC X(01).
03  WS-DISPLAY PIC 9(08)V9.
03  WS-DISPLAY-X REDEFINES WS-DISPLAY
                               PIC X(09).
03  WS-NUMERIC PIC 9(05) VALUE 13000.

MOVE ZERO TO WS-PACKED.
MOVE WS-NUMERIC TO WS-FWORD.
* WS-PACKED EQUALS X'000032C80F'
MOVE WS-PACKED TO WS-DISPLAY.
* UNPACKS AS X'F0F0F0F0F3F2FCF8F0'

INSPECT WS-DISPLAY-X CONVERTING
X'FAFBFCFDFEFF' TO 'ABCDEF'.

At this point, WS-DISPLAY-X (5:4) is ===> '32C8'

Hope this works in your environment.

Bill
