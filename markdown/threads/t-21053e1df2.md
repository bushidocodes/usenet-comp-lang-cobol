Hi:

The use of GO TO is universally despised these days. Instead,
people are forced to use the arbitrary 'structured' style.

There is absolutely nothing wrong with GO TO which I use
constantly. (I do not use structured style) At the risk of infuriating
everyone, I would say that structured programming was an invention of
people who didn't know how to program in the first place, it was
not because of anything wrong with GO TO.

People moan about spaghetti code riddled with GO TOs.

However, the author and subsequent maintainers of that spaghetti
code would probably have written equally incomprehensible code
using a structured approach. Structured methods arose becaue
people didn't know how to program; they started a program, they found
out about the things they didn't know when they started to test
and the code just got worse and worse as they attempted to fix each new
bug they encountered.  If the program had been properly designed in
the first place, there would have been no problem.

I get yelled at all the time because I make the distinction between
programming and coding. Programming, to me, is the design of a
solution to a problem, independent of any language, which recognizes
EVERY eventuality and deals with it properly. I guess that COBOL
courses these days deal with coding and syntax and stuff but never
address the design of the logical solution.

I would go further to say that the structured programming style
creates a new set of problems in place of the problems people
had with non-structured programming. It all boils down to whether
one knows how to program or not.

Earlier this year, in another thread, I wondered if anyone drew
flow-charts any more. Someone said, 'No, we just sit down and
start typing'.  If you are confronted with a problem which you
have never faced before, try drawing a flow-chart to solve the
problem and, when you are finished, and you think that it is OK,
code it.

Drawing flow-charts will teach you how to program, once you know
how to program, you don't need structured style.

With apologies in advance. . .

Thanks

TOny Dilworth
