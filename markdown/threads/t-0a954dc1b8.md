Well, then you get to engage in that time-honored practice of "looking
through the bytes and trying to figure out the FD."

Let me save you some research time: If you do not have the COBOL FD, the
only way you are going to figure out your file is as above. Period. There
are no tools, no compilers, no software, no silver bullet, so don't even
ask.

This is the same as me mailing you this..

"935@29nsdfns9u13mnkwe;sndvks"

... then giving you a quiz on it.

If you have a least a clue about what the file contains, someone with a lot
of experience may be able to "intuit" some of the data - by digging through
it byte-by-byte.

If there is no FD, then there is no way to tell what's in a file.
Cobol data files do not store record descriptions in them.

You can try to reengineer the FD by looking at the bytes
and imagining what they might mean. But the resulting FD
will be more a product of your imagination which
may be different from imagination of programmer
who wrote the program that wrote this data file.

One way to make FD guessing easier is to use our
data2flat http://www.siber.com/sct/datafile/ converter. 
You can specify what the record layout is 
to this converter using RDD file.
RDD file is a text file that desribes format of Cobol record.
The RDD file is automatically generated from the Cobol source,
or you can create it manually in a text editor.

So in your case you can just try different record layouts,
put them into RDD file and use data2flat to view the results.

Evaluation version of data2flat is free and it can read first
200 records of your file (commercial version can read the rest
of the records). So it would be a cheap way to try various layouts.

Regards,
Vadim Maslov
