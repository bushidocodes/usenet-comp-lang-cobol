OK, to answer the question:

1. There is no way I am aware of to dynamically declare tabes in COBOL. 
(Various systms allow dynamic allocation of memory by calling low level 
functions, but even having acquired the memory, you strill can't dynamicaly 
declare the tables...)

table at run time with a CALL... USING.

This will allow you to have REFERENCE tables immediately available, but you 
cannot grow them. Still, you might be able to adapt this idea to what you DO 
need...).

virtual memory (RAM) and secondary memory ( disk or other media).

The OS is doing a lot down in the weeds that is invisible to the application 
program.

For very large tables, it is often faster just to fetch the damn record than 
to find it in the table.

This is because the operating system often has the required record cached in 
its buffers and doesn't actually have to read the file.

You can test this:
1. Get a record.
2. Start a timer
3. Get the same record 100,000 times
4. Stop the timer
5. Compute the elapsed time.

I'll wager the elapsed time is insignificant.

Of course you can simulate this in a table look-up also:

IF (requested record) = (last record requested)
   process record
ELSE
   do table-lookup.

The technique chosen often depends on the nature of the data. In one 
hospital billing program I worked on, it was obvious EVERYBODY had 
"Semi-private room" and "Telephone access" and a few other things every day! 
These were semi-hard-coded in the program and, by so doing, decreased the 
processing time dramatically.
