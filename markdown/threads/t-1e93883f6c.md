You cannot "BROWSE" a Transient Data Queue.
You can read the data off - which destroys the queue - and then put it 
back.

Check out transaction CEBR.

You run CEBR without parameters. This assigns you a Temporary Storage 
Queue. Then issue the GET command to read the TD q into the TS q and 
display it. If you want to keep the data in the TD q, do a PUT command 
to write the TS q back to the TD q.

The concept of "destructive reads" of Transient Data is a popular
misconception.  Transient Data is a catch-all for a number of data
structures/devices, including Sequential Data Sets (QSAM).  The CICS
control program reads the file, and just as with any other program,
cannot re-read a record once it has been passed.  Therefore, the
Current Record Pointer is kept by the CICS control program, and
multiple CICS application programs reading the same file ("queue")
start where the last one left off.  It IS possible, using commands
usually kept secret by Systems Programmers <G>, to close and re-open
the file, and read it again.

In alex's case, you can't read it WHILE it is still open for the
task(s) which is (are) writing to it, since it was opened in CICS as
either OUTPUT or EXTEND.
 
The file can be read in batch programs, or by an assortment of on-line
utilities such as File-Aid or SPF.

Well, this is partially true, as far as QSAM files are concerned.  In
CICS we call the Extrapartition Destinations.  They can either be
opened for Input, or for output, and not both at the same time, once
you read a record, you cannot read it again until you close and reopen
the queue.  So far, so good

A lot of CICS TQ queues are Intrapartition Destinations, though.
These can be read from one task, and written by another task _at the
same time_.  The reads are truely destructive, as there is no way to
read the same record again later.  Space of read record (better CIs)
can be reused for records of completely different queues.  The rules
change a little bit when recovery comes into the picture - Records are
retained until the task completes successfully

>Of course Queues as such are nothing new to programming, covered by introductory courses on Data Structures.

Correct.  That is one of the problems there is with CICS Transient
Data Queues.  They come in several flavours, the QSAM variety (the one
Stephen spoke about), which is a QUEUE (in the sense of FIFO
processing).  Closing and Reopening for reading makes the contents
available again

The second flavour, though, is a special type of QUEUE which, in a
UN*X context, is called a PIPE

The use of such a PIPE in CICS is traditionally like this:

Many applications/end users (like people ordering some article)  drop
some information into one end of the pipe (via WRITEQ).  Those could
be termed the PIPE clients. on the other end of the PIPE there is a
single task (the PIPE server, then) which
a) get's automatically started by CICS as soon as data comes its way
b) Removes the data from the PIPE (via READQ) and does is processing
(like printing shipping papers, invoices, whatever).  This part is
repeated over and over until there are no more data, then the server
process terminates

From the application point of view, only one record is available at
any one time:  You can only write to the end of the PIPE, and you can
only read from the beginning of the pipe

Linked Lists are a different solution to a different problem.  They
allow to access a specific element out of a set without having to know
its explicit position (no index, no direct pointer) by following the
set elements via the link pointers

Technically, a CICS INTRAPARTITION TRANSIENT DATA DESTINATION (that is
the full term for the thing I called PIPE above) is maintained as a
LINKED LIST on disk (in a VSAM ESDS)

I only use LINKED LIST when following control block chains, not
usually for end user application data storage
