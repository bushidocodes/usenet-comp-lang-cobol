```
I have used MFCOBOL IMS option. I was able to test MFS screens
and MPP programs. Also, used OS/2 database manager to test
DB2. Only "gotcha" is ebcdic vs. ascii. Be sure to test using
EBCDIC to match the mainframe environment. Host compatibility
option for OS/2 help ebcdic for SQL. But Microsoft SQL server
doesn't seem to have ebcdic option...

source level debugger beats the socks off of locking for displays
in the tmp* regions....

The other gotcha is the lightness in your wallet. But hey,
you get what you pay for...
```

```
› Has anyone done any offloading of IMS applications for
› maintenance using MF COBOL or CA-Realia? Does it work?

We've done that with the MF product. You must consider at least two
things. First you are going to need a useable test environment on the
PC -- there is no way of getting around that. Second, if you have
some in-house callable utilities (date routines, account validation
routines), you will have to do something about them if they are
written in assembler. We were able to rewrite most of them in COBOL
for the PC use and were able to get by with simple stubs for the rest.

Also you shouldn't expect the PC environment to behave identically to
your mainframe. For instance if your programmers are used to using
ISPF and you get the license for SPF/PC, SPF/PC will behave slightly
differently (in some ways better and in some ways worse) than its
mainframe counterpart.

Also note that you should use OS/2 or WIN-NT on well powered machines
if you want to do something like that. Our experince has been that
DOS has too many limitations and WIN 3.1 is just a little too
unstable for this sort of thing.
```
