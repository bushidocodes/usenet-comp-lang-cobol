I presume you are referring to MF here.

The .INT are intermediate pseudo code files that are interpreted
by the run-time. They are not executables but are data bytes that
control the execution of the run-time. The .GNT are generated
object code that can be called from the run time. They are native
object code.

The .INT has the procedure division about one third the size (the
data areas are the same) of the .GNT and about one third the speed.

The main advantage of the .INT is the theroetical portability that
allows it to be moved to different machines and operating systems
and just run. Thus development can be done on one machine and
distributed to a range of machines.

Another [obsolete] advantage of .INT is that it can be made to fit
in smaller machines. This was a significant issue when I was
developing for 1 MByte machines with several users/tasks (I still
have some multi-user clients with 1 and 2 MByte machines). It
is also an advantage on DOS based systems when code bloat hits.

The speed disadvantage is rarely an issue on accounting systems
where disk access is the major constraint, how much CPU does it
take to calculate an invoice ?
