You can think of them as separate source code that may be reused by many
programs, although they are typically used to store program variables or
file layouts. They are brought into the source by the COBOL compiler
(this may vary depending on which source code management system is in
use) and then compiled like the regular source code.

We use copybooks for both file layouts and program logic, and it works
quite well for us.

File layouts are "copied" into the program as follows:

01 CUSTOMER-MASTER-FILE. COPY CBCMFDD.

All file layout are named CBxxxFD or CBxxxDD. CB for copybook, xxx for
file name acronym, FD for file description entry (very limited
definition of the file layout, mainly to define the key structure), and
DD for working-storage entry (we use read ... into .../write ... from
... logic).

The DD copybook looks like:

*01 CUSTOMER-MASTER-FILE.
05 CMF-RECORD-KEY.
10 CMF-....... PIC X(..).
10 CMF-....... PIC X(..).
05 CMF-.......... PIC X(..).

The 01 levels are commented in all file layout copybooks because we use
the 01 reference within the program for the group name.

As far as copybooks for PROCEDURE DIVISION code goes, we follow a
similar standard. The program is coded as follows:

PERFORM 9000-TERMS-EDIT THRU 9000-TERMS-EDIT-XIT.

At the bottom of the program we include all references to copybooks as
follows:

9000-TERMS-EDIT. COPY CBTERMED.

The copybook looks like:

*9000-TERMS-EDIT.
:
:
program logic here...
:
:
9000-TERMS-EDIT-XIT.
EXIT.

Again, the paragraph name in the copybook is commented because we use
the name within the program itself.

Our standard is that a PROCEDURE DIVISION copybook has one and only one
entry point like described above (although we even have 1 or 2 that
violate this standard). By using copybooks in this manner, we know what
copybooks to look in when performing a paragraph.

With this implementation, maintenance for us is not a problem.
