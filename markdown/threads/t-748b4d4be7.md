I thought this debate was over years ago, but I guess not. Yes, you can write a program that does not have a GOTO. I have written many myself. But there is no question that you can write simpler, leaner, and more efficient code in some instances if you utilize GOTO to go to the exit, or to go to a label "below" the GOTO in the same paragraph. Those are the only 2 instances where I use it, and I've not had anyone accuse me of writing non-maintainable code. It's just like any other tool - use it correctly......

Interestingly, the "debate" ended for me about 20 years ago when I discovered
I could write simpler, cleaner, more easily maintained code in ALL
circumstances without even THINKING about GO TO's. By eliminating GO TO,
SECTIONS, and PERFORM ... THRU coding, you eliminate any practical possibility
of "fall thru" errors. One less possible cause of errors; the ability to
rearrange, add, or delete paragraphs without worrying about changing the
program logic; a program XREF that lists every place that a paragraph is
"used" - these are 3 reasons why I say "simpler, cleaner, more easily
maintainable".

Nick, you probably do write maintainable programs; I have programs in the
system I maintain that have GO TOs and they are maintainable. But every time
I have found a "good" GO TO, I've seen a way to eliminate it without making
the code any worse. Every "bad" GO TO (ie spaghetti code) has been improved
by replacing it with good structured design/programming techniques using
PERFORMs only.

Hope all of your GO TOs go where they're suppose to!
