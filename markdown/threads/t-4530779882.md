Sounds like the library definition of the alias is wrong. When the program
is linked, I'll bet the compiler looks in its own definition file for the
alias; that, or it automatically is doing "C" style "alias decoration."  (I
checked my Kernel32.dll and the fi\untion is defnitely exported as "Sleep"
(case sensitive).

If you can't modify whatever file specifies the import alias to statically
link, you can call the function dynamically by..

GetModuleHandle to KERNEL32.DLL   (loaded by all Windows processes)
GetProcAddress of "Sleep"
Call-by-address   the address you get from getprocaddress

If you really get stuck on this, let me know and I'll write you a DLL you
can call from your program which won't have the problem.  (I assume you can
create the necessary import specifications for the compiler if you have a
DLL name and the exported procedure name and parameter count/type).

Free. (Take me 15 mins tops).

Alternately, try changing your CALL to make the delay time a dataname with
correct USAGE instead of a numeric literal.
(CALL "Sleep" WITH STDCALL  USING BY VALUE ws-sleep-milliseconds). Maybe the
compiler really really wants a dataname since you did not specify a SIZE in
your BY VALUE clause.

The actual name of a procedure "as it is to be called" is found in the
export table of KERNEL32.DLL (or any executable, EXE or DLL).  The name
which the calling program is going to use is in that calling program's
import table.  I'm not really too confident about how the FJ linker works,
but my best guess is that the "definition"  files used by the compiler
convert a "user-friendly API function name" into a DLL name and Procedure
name which are then stored in the compiled module's import table.

Even if the program links, if the calling program cannot find the library
DLLS (e.g., KERNEL32) and the referenced procedure names AS THEY ARE SPELLED
IN THE EXPORT TABLE OF THE LIBRARY DLL, Windows will give the now-famous
"missing a component" message and not even complete loading a program.

 When you see an underscore prepended ("_myfunction") or the stack usage
appended ("sleep4") or the function name uppercased or other decoration done
(_SLEEP@4) it is because the programmer who created the DLL told his/her
compiler to create it that way. Prepending underscores, uppercasing, adding
an "@",suffixing the stack usage and other ''name decoration" is common with
'C' compilers and programmers (who often take the 'default' options).

For the Sleep, and just about all WinAPI calls, the function names have no
underscore, no"@" sign and no stack usage integer. (Although many have
either an "A" or "W" suffixed).

Pete had asked about the tool I use to read the import table. It is a
product called "PEBrowse" which I got off a shareware BBS somewhere. When I
tried to send in my $25.00 to register it, the envelope came back (twice!)
with "party not found" and the phone number is 'not in service.'  (I think
this is an indication of less-than-ideal support availability).

If you can find it, it also can show you all the resources in an executable
(EXE or DLL), the size of the PE sections, etc and a whole bunch of other
geeky stuff. Me, I just use it to get the 'actual' names of exports AND

Related but not quite the same: I recently produced a freeware piece of
software which launches a GUI program which will show you all loaded modules
and the path from which they are loaded. You run the program by doing a
createprocess for this EXE program, passing the process ID as a parameter.

This is really handy for  A) creating an 'install' package and making sure
you have all the files you need and B) when testing and your program seems
to be loading a DLL form the wrong place. (You'd be amazed how often
developers and/or Windows do strange and wondrous things with "path,"
resulting in DLLs loading from an unexpected place).

You can get a full package WRITTEN FOR USE WITH THE BASIC LANGUAGE at:

While you could use this with COBOL-based programs, the porting might be
beyond the reach of those not terribly familiar with WinAPI calls - i.e.,
the very audience for whom this might be most useful. (You need to do a
CreateProcess either explictily or implicitly, converting the calling
process ID to a USAGE IS DISPLAY variable, but you'll see that in the source
code).
