No, I don't really think so. We aren't hiding the data at all, just
the structure of the file. In addition, Acucobol does NOT charge for
the File System Interface documentation. We simply ask that the user
sign the non-discolure statement because the structure of the file IS
propriatary and part of our base income. We wouldn't give out the
structure of the object files or meanings of the op-codes either. Try
getting file or data structures from say Oracle, or Informix. We may
not be that complex but we're not alone. If he likes he could get
ahold of a utility named vutil (vision utility) which whold dump the
data out into a flat finary-sequential file which, of course, is easy
to read. Another problem is, since he doesn't have the source code,
how is he going to know what the data elements are. Even the Vision
file system doesn't maintain that information. It's like being given
a record, without the layout, which may or may not contain binary
information and being asked to generate a report from it. I doubt if
that's doable. The Acucobol file system interface simply provides "C"
interfaces which are capable of reading (i.e. retrieveing the RECORDS)
from any Vision file without knowing the number of keys, record size,
etc. It will NOT, however, tell him what's in the record.

BTW, do you know the exact structure, or could you get it of Micro
Focus or RM Index Sequential files. One of them uses ISAM, I know but
I don't believe that it's that easy to get the exact layout.

One other comment: the file system interface also documents some low
level Cobol routines which are capable of reading any Vision File
without knowing the layout. We'll give him every ounce of help we can
to aid his getting at the data. But, like all other companies, we
won't give away the internals of the produces which are our
livelyhood.

Pete Withey
Acucobol Technical Support
