```
I have a client who is running 64 bit processors and wants to use the PRIMA 
Migration Toolset.

Most of the Toolset is written in C#, but there are a number of components 
written in 32 bit Fujitsu NetCOBOL.

The installer is complex as it makes sure all the components COBOL and C#) 
are present and registered before even downloading the Toolset application. 
The Installer is really a series of WSH scripts and MSI installs, glued 
together with Inno Installer. I wrote all of it so I am pretty familiar with 
what it does and how it does it.

In a 32 bit environment everything installs and runs perfectly, but in a 64 
bit environment, it is a disaster. :-)

I am currently waiting on the arrival of a new 64 bit laptop so I can fix 
it. I would hope to have everything resolved by the end of next week, but it 
is a lot of work, partly because I shot myself in the foot when developing 
some of it. Hopefully, others can benefit from this experience...

Most of last weekend was spent combing MSDN and various forums to understand 
what happens when a 64 bit win 7 system installs and runs a 32 bit 
application.

It is all pretty clever and most people won't have any problems, but, as 
usual, COBOL is a bit of a poor relation...

The fundamental problem is that a 64 bit CPU can't load and run 32 bit 
executables without an emulator.

(Win 7 64 bit provides a built in 32 bit emulator called "Windows on Windows 
64" (WOW64))

As many of the system functions we take for granted in development 
(Regsvr32, RegAsm, Registry paths, system paths, etc.), are also used on a 
64 bit platform, windows has to keep the 64 bit (native) version separate 
from the 32 bit version, and it has to do the same with the executables.

It loads 32 bit executables which you might expect to find in "Program 
Files" or "Windows\System32" into specially named versions of these paths 
("Program Files (x86)" and "Windows\SYSWOW64\System32".)

BUT, the original names ("Program Files" or "Windows\System32") ALSO EXIST 
on the system! These now contain 64 bit executables so you can imagine what 
happens if you specifcally code the path to something you want to load as, 
say, "c:\Windows\System32\myConfig.xml". If it was loaded with the install 
it won't be there (It is actually in "c:\Windows\SYSWOW64\system32", because 
it was automatically redirected as soon as the system recognised a 32 bit 
install...) If you are loading a specific executable it will try to load the 
64 bit version and anything can happen - it might even work :-). You get the 
picture.

It does these redirections seamlessly and transparently so that the built in 
32 bit emulator (WOW64) knows where to find 32 bit stuff and everything will 
install and run as long as it is all straightforward.

A C# application, compiled on a 32 bit system with the "Any CPU" option set, 
will automatically adjust itself to the target platform as long as it is 
being run in Version 2 or better of the .Net platform.  It knows what 
directories it has to load itself into and does so.

Pretty smart.

If it CAN use 64 bit it will do that and if it can't it will use the 32 bit 
subsystem emulator. (Remember that the generated code is CIL which is "bit 
size agnostic", so it isn't too hard for it to do this, irrespective of the 
architecture it is actually compiled on.)

However, a COBOL application, compiled on a 32 bit platform, will have 
system run times and system .dlls linked to it that are definitely 32 bit.

After installing on a 64 bit platform, these may not be located where you 
think they are. In particular, user written .DLLs may not be stored where 
you expected. As long as you are not refencing the paths to them it is no 
problem; you don't care where they are as long as the system can find them 
and the system automatically amends all 32 bit references to be redirected 
to the proper directories.

The various non-native CIL generating COBOL compilers (Fujitsu Net COBOL 
for .Net, Micro Focus Visual COBOL, etc.) won't have this problem, any more 
than C# or VB.Net do. (At least, they theoretically shouldn't...It depends 
on what the vendors have implemented. I don't use either of them so I can't 
say for sure...)

It won't be a problem as long as you observe the following when coding 
COBOL: (All of these are good practise, anyway...)

1. Don't hard code directory paths and/or reference them within hard code. 
(You may need to amend your application to detect which environment it is 
running in and change the path reference accordingly.)
2. If you are referencing Registry entries, be aware they may not be filed 
under the structure you think they are. You may need to code separate 
Registry lookups for 32 and 64 bit Registry access.
3. If your install is registering components for COM with Regsvr32 you need 
to make sure it runs the 32 bit Regsvr32 and not the 64 bit one. (If your 
install is marked as 32 bit or at least does not support 64 bit - there are 
options in INNO to do this - the system will store the stuff in the right 
place and reference it there, so your 32 bit app will install and run 
without problem on the 64 bit platform, and without any intervention from 
you.)
4. If you use ADDRESS OF in COBOL there may be implications. (8 byte 
references as opposed to 4 byte references.) I'm not sure about this as I 
haven't tried it yet; it might be OK or it might not. If the executable is 
installed in the 32 bit directory it should be fine...)
5. If you use Windows Script Hosting (WSH) there are functions to tell you 
what you are running on.
6. .Net Version 4 provides C# functions to do the same so I imagine the .Net 
COBOLs will have it too. (Again, depends on the vendors...)

If I find any more once I get into hands on testing of the 64 bit version 
and install, I'll post here.

Pete.
```

```
> The fundamental problem is that a 64 bit CPU can't load and run 32 bit
> executables without an emulator.

That is simply not true. It is not a limitation of the CPU. AMD and
Intel x86-64 CPUs can switch between 64bit and 32bit quickly and
easily.

The problem you have is that programs need dynamic libraries with the
same bitness and Windows cannot have two sets of DLLs active at one
time. Thus it has to have a separate and distinct environments, such
as an emulator, for each set.

More advanced operating systems have no problem mixing 32 bit and 64
bit programs in the one environment and manage the dynamic libraries
for both without problems.
```

```
Most people using 32 bit COBOL compilers will have no problems on Win 7 or 
Win 8 64 bit platforms, provided they avoid hardcoding specifc directory 
paths as noted previously. Their code is ALL 32 bit and it should run 
seamlessly.

My problems arose because I have an application that is mostly C# targeted 
to .Net 3.5 or better and this runs, by default in a 64 bit process space on 
a 64 bit machine. As soon as it was required to load a 32 bit in-process 
COBOL COM server, it fell over. (It can't load a 32 bit process into a 64 
bit process space). The solution was to use the Configuration Manager in 
Visual Studio (or the Corflags Utility which is part of the .Net SDK) and 
set the flags in the Assembly header to force it to always run as a 32 bit 
process (x86).
```
