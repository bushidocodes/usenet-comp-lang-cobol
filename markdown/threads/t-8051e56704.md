I use ELSE-IF quite a bit in Easytrieve-Plus. There is no need for yet another
scope terminator because the ELSE-IF keeps the same conditional depth as the
IF/ELSE/END-IF,  e.g.,

IF A = B
   ...
ELSE-IF C = D
   ...
ELSE-IF E = F
   ...
ELSE
   ...
END-IF

The ELSE statements would be executed if neither the IF nor any of the ELSE-IFs
were true.

COBOL'85 EVALUATE statement serves a similar function, only the syntax would
be:

EVALUATE TRUE
   WHEN A = B
      ...
   WHEN C = D
      ...
   WHEN E = F
      ...
   WHEN OTHER
      ...
END-EVALUATE

I have a question, though. Would the introduction of an ELSE-IF in COBOL buy us
anything that the EVALUATE doesn't?

EVALUATE does a fair job of defining decision tables.

IF/ELSE does a fair job of defining binary decision trees.

IF/ELSE-IF/ELSE would be better than IF/ELSE in defining
non-binary decision trees since the non-binary portions remain
at the same nesting level. For example,

IF TAX-STATUS = TS-MARRIED
    IF FILING-STATUS = FS-SEPARATE
        PERFORM ...
    ELSE-IF FILING-STATUS = FS-JOINT
        PERFORM ...
    ELSE
        PERFORM FILING-STATUS-ERROR
    END-IF
ELSE-IF TAX-STATUS = TS-SINGLE
    PERFORM ...
ELSE-IF TAX-STATUS = TS-HEAD-OF-HOUSEHOLD
    PERFORM ...
ELSE
    PERFORM TAX-STATUS-ERROR
END-IF

The four possibilities for TAX-STATUS occur at the same level
and the three possibilities for FILING-STATUS occur at the same level.

IF TAX-STATUS = TS-MARRIED
    IF FILING-STATUS = FS-SEPARATE
        PERFORM ...
    ELSE
        IF FILING-STATUS = FS-JOINT
            PERFORM ...
        ELSE
            PERFORM FILING-STATUS-ERROR
        END-IF
    END-IF
ELSE
    IF TAX-STATUS = TS-SINGLE
        PERFORM ...
    ELSE
        IF TAX-STATUS = TS-HEAD-OF-HOUSEHOLD
            PERFORM ...
        ELSE
            PERFORM TAX-STATUS-ERROR
        END-IF
    END-IF
END-IF

The full evaluation of TAX-STATUS and FILING-STATUS occur
at different levels.

EVALUATE TAX-STATUS ALSO FILING-STATUS
    WHEN TS-MARRIED ALSO FS-SEPARATE
        PERFORM ...
    WHEN TS-MARRIED ALSO FS-JOINT
        PERFORM ...
    WHEN TS-SINGLE ALSO ANY
        PERFORM ...
    WHEN TS-HEAD-OF-HOUSEHOLD ALSO ANY
        PERFORM ....
    WHEN TS-MARRIED ALSO ANY
        PERFORM FILING-STATUS-ERROR
    WHEN OTHER
        PERFORM TAX-STATUS-ERROR
END-EVALUATE

In the EVALUATE, it is not entirely clear that evaluation of
FILING-STATUS is entirely dependent upon
TAX-STATUS = TS-MARRIED. One may interpret that any
FILING-STATUS is Ok for single or head of household rather than the
intended "FILING-STATUS is not to be evaluated."

The effects of ignoring or evaluating data become more important
with the introduction of FUNCTIONS. Evaluation using a FUNCTION
with invalid data could cause a program failure or the unnecessary
execution of an error routine, if the evaluation causes an exception.
Hence, with FUNCTIONS, "any value is Ok" is not the same as
"do not evaluate."

There are a number of good reasons for ELIF (or ELSE-IF or ELSIF).

Indenting IF/ELSE clauses can get excessive in situation that an ELIF construct
would apply to. And the indentation is not only needless it can be confusing
when you have code that has matured over the years and has a number of
conditionals nested within one another.

With current IF/ELSE coding techniques, it is comparatively hard to insert an
IF in the middle.  In many shops that will require indenting all subsequent
'unchanged' code, placing those functions in needless jeopardy. ELIF construct
coding is sequential just like a sequence of imperative statements - insert or
remove without major side effects on the surrounding code (indentation).

When applied to an ELIF situation the IF/ELSE coding construct frequently
necessitates a long, and basically pointless sequence of END-IFs.  

I call this the END-IF dragon, because it looks like a saw-toothed tail, and is
a monster that eats vertical realestate, a monster that causes indentation
maintence costs, and removes meaningful information from the screen.

IF/ELSE are great, and EVALUATE is great.  The ELIF proposal relates to a
certain subset of what can be handled by EVALUATE, and handled better by ELIF.

EVALUATE has the slight disadvantage of indentation for the WHEN clauses, that
puts the body of each when stanza atleast one indentation further than it would
be within an ELIF construct expression.

Many EVALUATES are not really case structures at all (that is a compliment to
the folks who wrote the standards, they stretched its syntax elegantly).  But
when we code 
  EVALUATE TRUE
we have wasted one increment of the vertical dimension.  On a twenty-four line
screen (and there are plenty) with headers and pull down menu bars, there are
perhaps 20 or 21 lines of code (less if you are in a split screen or windowed
mode).  One-twentieth of the vertical realestate _IS_ worth fighting for,
because something meaningful could be on that screen and it is bumped off by
the EVALUATE TRUE, or as in a _small_ set of cases, 
   EVALUATE TRUE
       ALSO TRUE
       ALSO TRUE
      .....
for complex conditionals, wherein sometimes the WHEN clauses do similar skip
line patterning.

The basic point of an ELIF construct is that it relates to a situation in which
we will make a sequence of tests, and the tests are mutually exclusive.

Sometimes we have only a handful of simple test to do before an homogenous case
structure: for example in file I/O; did we OPEN successfully, did we get an
initial position from a START successfully, then AT END / error status / normal
read.

If we attempt to put the special process into the EVALUATE, we break it from
the simple case syntax; and must resort to placing all of the details in WHEN
clauses anyway. If we code the intial portion as IF/ELSE we end-up with an
EVALUATE sometimes severely indented within the last ELSE (making it very
difficult to indent any conditionals within the WHEN stanzas, ((especially for
adherents of the PERFORM THRU philosophy)) ). ;-).

EVALUATE is extemely good when the conditional involves a repeating subject
that we can represent efficiently in the EVALUATE phrase, and avoid repeating
in the WHEN clauses. But whenever we are motivated to go ahead and put the
entire conditional expression in the WHEN clauses, we are right then close to
an even trade off with the proposed ELIF construct.

The thing aout many mutually exclusive test sequences is that they are not the
same complexity at each point. 

And the presence of ALSO does not help because it requires extra code in the
WHEN clauses that don't need it and the ALSO ANY phrase.

(Don't take me wrong I am defintiely not against EVALUATE.  I even favor it
strongly to implement sparsely populated table logic, and find the ALSO ANY
mechanism self documenting and powerful. But it is rare that you can find an
opportunity to apply it to purely tabular data.)

The EVALUATE becomes misleading (in _some_ case) when ALSO or ALSO ANY phrases
are used to implement mutually exclusive test against conditionals that really
do not have similar topology.  Besides AND is just as good.

Overall ELSE-IF sequences are similar in nature to any CASE coding construct,
including COBOLs successful and powerful EVALUATE. Like EVALUTE ELSE-IF
sequences could have a 'default' case, coded as ELSE in ELIF constructions, and
OTHER in EVALUATE.  And like other case constructs it requires a scope
terminator, and END-IF could be a perfectly adequate scope terminator for ELIF
sequences.

One weakness of EVALUATE, and this is small but every bit helps when you are
talking about the definition of the language itself, is that it seems to say
this is a CASE OF construct involving a mere variation of a simple field test.

In a select subset of applications ELIF would clarify test sequences as
essentially mutual exclusion expressions and not CASE OF constructs. 

In maintenance that would _occassionaly_ pay off as a worker may realize they
need to examine the expression more carefully than they might if they saw an
EVALUATE expression. In other words, people sometimes get the wrong idea from
the leading portions of a EVALUATE sequence of WHENs. Although it is just as
erroneous to cut short a review of an IF/ELSE casecade, I am saying that the
pedagogical notion that EVALUATE is a CASE structure leads programmers latter
to pay less attention to the fact that it might not be a CASE situation at all.
I think ELIF can help clarify those cases.

Availability of an ELIF (or ELSE-IF or ELSIF) construct in COBOL would preserve
considerable vertical realestate.  It would preserve some horizontal
realestate. With it we could eliminate the irksome END-IF dragon, sometimes.
EVALUATE syntax is somewhat misleading in cases where the test are divers; ELIF
can be more emphatic of the eclectic nature of some test collections.

1) OR-IF is a Nice proposal, I think it has a lot strength.  I notice that
Michael C. Kasten is frequently a creator of ideas around here. I look forward
to comments from all as to other earlier proposals relating to OR-IF or 'OR
IF'. And I am grateful to Michael for introducing it in this thead.

The OR-IF construct would _be_ the ELSE-IF or ELIF or ELSIF under recent
discussion. (We would not code ELSE OR-IF).

The idea is that OR-IF would replace ELSE IF in compound IF statements that
are case structures, or mutually exclusive test sequences. For example

IF    file-not-OPEN-successfully
   PERFORM e100-diagnose-miss-file
OR-IF file-stat-from-START-bad
   PERFORM e200-first-rec-not-found
OR-IF file-stat-oaky
   PERFORM p1000-process-it
OR-IF file-stat-not-found
   PERFORM s1000-summation-logic
ELSE
   PERFORM z999-this-aint-kansas
END-IF.


2) AND-IF cousin by implication
   
We might also consider an AND-IF cousin.  The same author, Michael C. Kasten  ,

deserves credit for that too, since mere symmetry implies it.


3) UNIX C Shell analogue and nomenclature

The suggestion of an OR-IF encoding for the ELSE IF construct immediately
brings to mind the conditional operators of UNIX command shell syntax.
Permit me to briefly quote an excellent book on this subject:

"UNIX Unleashed", SAMS publishing
1994 First Edition
ISBN 0-672-30402-3

The book has numerous authors, the part I quote is from 
"Chapter 13, C Shell", written by Rick Rummel and John Valley. 

In a section titled "Executing Commands Conditionally", subtitled
"Conditional Execution on Failure - || (Or)" the authors state

"The or operator is used to join two commands:
  command1 || command2
It causes the shell to execute command2 only if command1 failed ..."

Their first example is

  grep '#include' *.c || echo No files | pr

and they offer further comments ...

"..either grep succeeds, or else the words  No files   are piped to
the pr command ..."

"..use the || operator to provide an alternative action. .."

Another example is given that we might call a make or break encoding,
(meaning it makes or breaks a script, it is an all or nothing requirement)

   mkdir $tmpfile || exit

This or operator (||) works like a case construct, somewhat analogous 
to COBOL EVALUATE and the ELSE IF cascades discussed under the topic the 
ELIF proposal.  (Although there is no OTHER or final ELSE syntax for this
particular construct in UNIX C Shell, UNIX it is rich in other related
constucts).


But note also
In a section titled "Executing Commands Conditionally", subtitled
"Conditional Execution on Success - && (And)" the authors state


"The double ampersand operator (read and) is used to join two commands:
  command1 && command2.
It causes the shell to execute command2 only if command1 is sucessful ..."

The first example 

 grep '#include' *.c | pr && echo OK

My description: you get the Okay if everything works out; and no news is
bad news.

Another example
  cp file1.c file1.bak && rm file.c
with the comment from the authors
"...The rm command will delete file1.c only if it is first successfuly 
copied to file1.bak."

I hope I have done justice to the authors. With so many contributors to 
this fine book it is hard to get complete attribution into a usenet posting.

The UNIX C Shell operators described are an excellent model for discussion
of OR-IF and AND-IF, and the SAMS Publishing book, "UNIX Unleasehed", is 
a very useful book.


4) difference is the COBOL Stork

In these C Shell expressions the command or program execution is the essence of
the
'conditional'. It is not an IF conditional that will permit or disallow 
some other command per se. It _is_ the command, and the operator (&& or ||)
just makes the next command in the cascade dependent upon the outcome of 
preceeding commands.

A COBOL OR-IF would have, not only a conditional, but a bundle of subjuncted
commands, much like a stork. (this would be the 'THEN' clause of the initial 
IF and each succeeding OR-IF, even though we nolonger code THEN).
A case statement expressed as an OR-IF cascade would 
then be a flock of storks. When a stork arrives it may or may not deliver
its bundle. The OR-IF and AND-IF connectives would determine the cascade of 
tests (detemine the test sequence continuation). But that would be distinct
from the delivery!  

Any true test delivers (the baby is dropped), the conjuctions determine how 
the cascade will progress. Deliveries or not. I think it is really useful 
to be able to think of conditionals on two different levels like that.  It 
is actually done in COBOL already, all the time by means of ELSE IF cascades
and EVALUATE statements. But the linguistic particles ELSE IF and WHEN do not
bring out the subjunctivity of the sequence. And there is nothing like 
AND-IF in COBOL just yet.

AND-IFs would have zero, one or multiple deliveries. OR-IFs would have zero 
or one delivery. 

OR-IF would be the classic CASE or ELIF sequence.  Its ELSE clause (call it 
the default: or the OTHER clause), is fairly easy to conjure up.

The ELSE clause of an AND-IF sequence is something more engaging isn't it.


5) more on mathematical people and linguistic people

Use of a construct like the And / Or operators in UNIX C Shell might appeal
more to those I would describe as mathematical individuals. There is a 
chance that some other folks would shun  AND-IF / OR-IF constructs should 
they be added to COBOL,  because, as I would describe it, they are 
perhaps more linguistic. 

I believe that the COBOL world is somewhat segregated, having relatively
few mathematical people. I think that the COBOL world would be better off 
if we had more mathematical people.

And with reference to the OR-IF construct. I think that the different 
expression modality that it could bring forth, would make all users of the
language occassionaly think more mathematically and less linguistically,
which could be beneficial.


6) precedence and operator interaction

In C Shell || has lower precedence than both && and |.
Lower precedence means happens later.  So in a long expression,
two commands on either side of a pipe (|) glue together right quickly,
and things on either side of And (&&) glue together before Or (||).

This comment would get very long if I try to take on examples of combining
OR-IFs and AND-IFs. 
But the issue of permitting combinations and the precedence deserves
discussion. If you consider it, please keep in mind that simple AND and OR 
operators are much hotter and resolve inside individual conditional 
expressions (not stretching across IFs and not across the OR-IFs or AND-IFs
under consideration).

An important alternative is to just disallow combinations of OR-IF and 
AND-IF within a single conditional statement.  Combining this stuff in OS 
commands is remote to say the least, and it is less than clear what a 
coder's intent might be in a procedural language context.

It is certainly sure that the ELSE clause gets weird if we were to
allow combinations of OR-IF and AND-IF.

*) Concluding remarks, 

But abstraction or practicallity, OR-IF is just as compact as ELSIF and just
as distinct as ELIF,  and it does not have the potential for coding fault
that ELSE-IF has ("ELSE IF" visually close to "ELSE-IF)'

OR_IF has the elegant aspect that it could be used by linguistic 
coders as a brute force case constuct just like current ELSE IF 
expression, and not bring the END-IF dragon to life, and not 
engage excessive indentation. The surface of such source code would
be flat, mirroring the topology of the semantics.

An AND-IF constuct would really be a very different kind of thing than was 
originally under discussion for an ELIF construct. Because the stork would
keep dropping _additional_ bundles upon successful links in an AND-IF 
cascade, such a construct is really a progressive conditional, where as 
OR-IF is a mutually exclusive conditional; with reference to the the
delivery of imperative functionality bundles.  

Although both have sequence as the the essence of the subjunctive 
semantics. And that is germane to the original discussion, wherein 
I maintain that case expressing ELSE IF sequences should not 
be indented at all, since there is no more hierarchical depth to 
them than to a simple sequence of imperative statements.
The indentation silliness and the pesky END-IF dragon are much more 
assailable when considered in the light of the flat topology 
conditionals of the UNIX C Shell.

I think the Storks of COBOL OR-IFs and AND-IFs could bring forth some 
badly needed renewal.

You are right and you are wrong. Current day nested IF statements are
progressive conditionals, wherein the flock of storks deliver more and more
individual functional bundles upon successful completion of a sequence of
tests.  

However, with existing available code constructs, the nested IFs are followed
by a sequence of distinct ELSE clauses.  An AND-IF would be slightly different
in that regard.  

Just as an OR-IF sequence of conditionals could be terminated by at most one
ELSE clause, the AND-IF sequence of conditionals could be terminated by a most
one ELSE clause (as an aside, and only academically speaking, these would be
nor-else, and nand-else clauses: lower case non-construct teaching names).
(('nor-else' has a ring to it, doesn't it?))

The ELSE clause of a normal nested IF reduces back to the most recent IF, and
relates to none higher.

In an AND-IF coding construct, the ELSE clause is a global exit upon failure.
The nand-else gets executed if any one test fails. This is part of the feature
that the conjunction (the AND-, and dash; the OR-, or dash) is occuring at a
structural level above the low level AND/OR of simplistic conditionals. The
subjuncts are conjoined, in a manner comparable to but distinct from the
conjunction of portions of the specific conditionals.

For example,

IF this-is-a-building
   PERFORM a100-needs-foundation
AND-IF this-is-a-residence
   PERFORM a200-needs-bedrooms
AND-IF city-inspector-approved
   PERFORM a300-schedule-full-crews
ELSE
   PERFORM z999-warn-incomplete-build
END-IF. 

As we build records, our construction process may only get partially completed.
 At which point we could prevent it from being written or place it on a
suspense file.

The ELSE of an AND-IF is a 'process on _any_ exception' where we want to make
as much _progress_ as possible (for optional partial or full commit), but need
to know in the end whether we failed to complete the whole process.

posts exceptions to the final condition)).

it is a small subset) there is a needless END-IF dragon, cousin to the END-IF

Perhaps our nor-else and nand-else teaching models would be something like
this.

1) OR-IF and the nor-else:

In an OR-IF sequence the ELSE clause gets executed only if _all_ tests fail.
The OR-IF construct is intended for mutually exclusive test sequences.  Any
successful test (mutually) excludes the ELSE as well.

Classical syntax -
  IF condition-a
    PERFORM a100-aaa
  ELSE
    IF condition-b
      PERFORM b100-bbb
    ELSE
      IF condition-c
         PERFORM c100-ccc
      ELSE
         PERFORM n999-nor-else-routine
      END-IF
    END-IF
  END-IF.

Revisionist syntax -
  IF condition-a
    PERFORM a100-aaa
  OR-IF condition-b
      PERFORM b100-bbb
  OR-IF condition-c
      PERFORM c100-ccc
  ELSE
      PERFORM n999-nor-else-routine
  END-IF.

OR-IF is like the EVALUATE statement in that it implements a case structure.
OR-IF may be more appropriate in situations where the complexity of the tests
at each stage in the sequence varies, EVALUATE may be more appropriate where
the testing is relatively more homogeneous. EVALUATE is definitely more
appropriate in its ideal application where a subject can be announced in the
EVALUATE header, and then the WHEN clauses are much less verbose than IF
clauses of any kind.

Use of OR-IF offers a much more efficient use of the human-machine interface
than do conventional nested IF statements, entailing less indentation, no line
break between IF and ELSE, and no END-IF dragon.


2) AND-IF and the nand-else:

In an AND-IF sequence the ELSE clause gets executed if _any_ test fails. The
AND-IF construct is intended for progressive test sequences. The coder indends
to achieve an accumulation affect from tests that succeed, and desires
notification if the sequence did not complete. 

The final AND-IF is the location to detect all test successful, the ELSE is the
place to detect any departure at all from the sequence of tests.
The ELSE indicates that you did not progress all 
the way to completion.

Classical syntax -
 IF this-is-a-building
   PERFORM a100-needs-foundation
   IF this-is-a-residence
     PERFORM a200-needs-bedrooms 
     IF city-inspector-approved
       PERFORM a300-schedule-full-crews
     ELSE
*   **not comparable to nand-else    
*   **throw a switch
     END-IF
   ELSE
***not comparable to nand-else    
***throw a switch
   END-IF
 ELSE
*   not comparable to nand-else   
*   **throw a switch
 END-IF. 

 IF switch-thrown
   PERFORM z999-warn-incomplete-build
 END-IF. 


Revisionist syntax -

IF this-is-a-building
   PERFORM a100-needs-foundation
AND-IF this-is-a-residence
   PERFORM a200-needs-bedrooms
AND-IF city-inspector-approved
   PERFORM a300-schedule-full-crews
ELSE
   PERFORM z999-warn-incomplete-build
END-IF. 

The AND-IF construct is not comparable to the EVALUATE.  The AND-IF conserves
human-machine interface resources, just like OR-IF.

3) application:

OR-IF and AND-IF apply to sequences of tests wherein there is no individual
ELSE clause for each segment in the sequence (Just like EVALUATE).  If you need
an else that is not generic (that is it is not a nor-else in the OR-IF
sequence, and it is not a nand-else in the AND-IF sequence), then the topology
of such commplex requirement does not match the flat surface capability of the
more simplistic OR-IF and AND-IF constructs.

OR-IF and AND-IF are intended for simple situations and provide a means of
making the code look as simple as the requirments: avoiding misleading
hierarchy implying indentation.


4) combinations:

OR-IFs and AND-IFs would be nestable in any manner with other conditional
expressions.


5) subjunction and conjunction:

An IF clause, an AND-IF, an OR-IF and a WHEN clause are subjunctive expressions
indicating the conditions underwhich code is to be executed.  The portions of
the conditional expressions can be conjoined with the logical connectors AND
and OR.

Subjunctive expressions can themselves be joined by means of AND-IF and OR-IF
expressions, as well as EVALUATE/WHEN and classical nesting of IF/(THEN)/ELSE
expressions.
