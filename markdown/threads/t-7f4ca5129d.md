You should be able to control the way the code is generated with the
OPT command. IIRC:

OPT(0) == .int
OPT(1) == .gnt
OPT(n) == various stages of additional optimisation.

When I thought it mattered (2MHz 8085 to 12MHz 386s) the procedure
division of .INTs was about 1/3rd the size of .GNT (Data was same
size) but ran about 1/3rd the speed.

The main advantage of .INT was that the run-time system was shared
code on MP/M and Concurrent-DOS so was only loaded once for all users
and the .INT was smaller so a 1Mbyte RAM 8086 or 80286 machine could
run 4 users.

But that was a quarter century ago.

.INT is 'intermediate' code which is like USCD Pascal or Java byte
code. It is machine independent and executed via a run-time
interpreter or VM.

.GNT is 'generated' which is machine code and thus different for each
machine type.

With old Level II Cobol there were two different programs that did
these formats. The compiler produced .INT and then this was fed into a
'generator' which created the .GNT.

On some systems (MS-DOS, OS/2, Windows) the code can be bundled with
appropriate library and run-time code into .EXE executables and .DLL
shared libraries (actually these are .EXE too on MS-DOS). The code
segments within these can still be byte code or machine code by
specifying the OPT compiler command.
