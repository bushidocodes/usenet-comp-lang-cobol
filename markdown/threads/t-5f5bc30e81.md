I used IBM CSF about 8 years ago, and even put several restructured
programs into production.  It's an interesting tool.  At that time I
had essentially no experience with TSO/ISPF, and it was set up to use
TSO panels to enter your configuration/restructuring parameters and to
submit a job to read old source code and generate new source code.  A
lot of the configuration options affected the appearance of the output
program (for example indent 3 columns or 4 columns on IF statements). 
The output program could be generated with only paragraphs or with all
sections (popular in Europe).  For really ugly Perform-thru's with
multiple entry points and exit points it generated working-storage
switches that controlled branching in the output program.

I never had an output program that wouldn't compile.  Most of the
programs I restructured had dead code that could not be generated to
the output program.  There's an option to include old paragraph names
as comments in the generated code, and to generate "called-by"
comments in each paragraph.  I found that restructured programs had
fewer paragraph-names than the original unstructured version.  Every
program I tried out appeared to work correctly after restructuring,
although I did not do full-blown regression tests.

IBM's COBOL Structuring facility was very good at eliminating GO TO's,
ALTER's, and PERFROM-THRU's, but I found the results a little
disappointing.  If the data-names and paragraph names were unclear in
the original program, they were still unclear in the generated
program.  I still found myself fixing up names, especially paragraph
names, in the generated program to make it more understandable.

They had the product for a few years, then dropped in a budget-cutting
move because I was the only applications programmer who ever learned
how to use it.  I still find old production programs that could
benefit from CSF.  If we still had the tool I would definitely use it
again, but the need doesn't arise very often.
```

Please do look at when/if you are going to be migrating to a supported
compiler.  Are you at least using the LE run-time library for your VS COBOL
II programs?

Now, back to restructuring.

From my experience, moving from the "generated" COBOL code from application
generators to "native" COBOL code almost ALWAYS requires a real programmer
and not an "automated tool".

I know of such "migrations" that have been done from Telon, APS, and
Pacbase.  As far as I know, the ONLY successful migrations have been
"programmer done rewrites" - and not any type of "automated tool".  The
reason is simple,

ALL of these application generators are specifically created so that their
COBOL codes is *NOT* maintainable at the COBOL level but REQUIRE you to
maintain the application at the "higher level language" of the generator.
It is not a matter of (just) paragraph and data names, but it is the entire
way they are "structured" to follow a "template" programming module that is
NOT intended to be directly maintained by application programmers.

I would STRONGLY recommend that you get a gradual "migrate/upgrade/rewrite
when doing normal maintenance" strategy and NOT try and do any mass
migration.
```
