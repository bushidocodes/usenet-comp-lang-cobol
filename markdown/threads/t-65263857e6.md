```
Do you have DUPLICATES specified for the PRIME key or not?  If not, the 22
is EXACTLY the status that you should get.

Micro Focus has had a LONG-time extension of allowing DUPLICATES for PRIME
keys. (definitely NOT standard).  My guess is that Fujitsu also provides
such an extension "for compatibility" - whether they document it or not.

As far as how duplicate prime keys work (at least in Micro Focus) the answer
is BADLY.  They can cause real problems with the semantics of a number of
"standard" COBOL statements.

If your program actually DOES specify "DUPLICATES" for the PRIME key and you
get a 22, then I think there is some sort-of BUG in the Fujitsu product.
```

```
If you write a record that is a duplicate in a key that allows
duplicates you will get a file-status of '02'.  The first character of
'0' indicates that the operation completed successfully, the 2nd
character or '2' indicates a duplicate key.  This status also occurs
on a read where there are multiples of one of the keys.

Ensure that you only check the first character for '0' as success and
not both characters as '00'.

If the write failed because a duplicate key was not allowed the status
will be '22'.

The User Guide has not been edited very well and it seems they
cut-and-paste from the '02' to the '22' without taking off the
'however'.


> 2.  Also, I thought duplicates were not allowed for the primary key.  Is 
>     this an extension in Fujitsu COBOL?  I see where it is allowed for 
>     relative files, but not indexed files.  How does this work?

Duplicate primary keys on an indexed file are possible as an
extension, but would be a real pain.

For a relative file the 'primary key' is the record number, I can't
see how duplicates could be anything other than impossible.
```

```
The Microfocus file handler allows for duplicate primary keys and it 
works.

Normally you would use it to group a set of records in an indexed file
that you would process as a set. Do a read on the primary key and if 
you get a "0/2" file status you do a read next until you see a 
different key.

That way you can access a single key file that provides for a group of
related records.
```
