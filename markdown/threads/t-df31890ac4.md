Are people still doing that?! We thought it was smart in 1966 when we wrote
programs in 16K and the Operating System ran in 8K...but nowadays...???

Anyway to answer your question: No. There is no direct equivalent in COBOL.
(Thank Heaven, I would say...<G>)

You could achieve a similar result by writing a COBOL program with tables
defined in the WORKING-STORAGE SECTION, complete with their static values,
in the usual way. You could then have a LINKAGE SECTION which had an area
large enough to hold the table defined in it (this actually occupies no real
storage; it's equivalent to an assembler DUMMY
Section   (DSECT)). The PROCEDURE DIVISION of this pointless hypothetical
subprogram would contain a USING clause to establish addressability to the
LINKAGE, and the only statements it would execute would  be a MOVE (table
from working-storage to linkage), followed by an EXIT PROGRAM.

Anyone calling the program would receive the table into their own storage.
It seems a long way round but there are cases I guess where you might want
to share a table amongst a number of programs and the above has the
advantage that you only have to maintain it in ONE place. You might wonder
why you can't just define the table in the LINKAGE SECTION in the first
place. The reason is because this section doesn't really exist and COBOL
therefore forbids you to put VALUE clauses on the definitions in it. (It is
therefore impossible to define your table in it.)

The reason I say this is not EXACTLY equivalent to the Assembler solution is
because the above subprogram must be compiled and linked with the COBOL
runtime modules. Even if you use DYNAM there will still be COBOL "baggage"
attached, which would not be the case in assembler. (Having said that, the
overheads in this case are minimal, and the object module  (before linking)
should be approximately the same size as the assembler equivalent. (COBOL
places EXTRNs and weak EXTRNs to the run time, depending on the compile
options.))

Hope this helps,

Pete.

With the '85 COBOL Standard - there actually is a "funny" way of doing this.
(I know this in theory - but have never seen any program actually use it -
in production).

Believe it or not, the PROCEDURE DIVISION is *optional* in the '85 Standard.
Therefore, if you coded a COBOL program with an Identification Division
(name of the table) and a DATA DIVISION with a table defined with "IS
EXTERNAL" - and *nothing* else in the program, this would "load" the table.
(There is an "implied EXIT PROGRAM" in any sub-program without a Procedure
Division.)

Obviously, to access the "table" you would need to DYNAMICALLY (IBM
mainframe usage) call a subprogram that ALSO had the  table in it.  (The
main program wouldn't have the table "loaded" but the subprogram would.)  ON
THE OTHER HAND, I don't know what this separate - no procedure division -
program would gain you - as some subprogram WITH the table would still need
to initialize values for you.
