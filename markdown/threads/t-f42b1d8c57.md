I've just finished reading a white paper put together by Wayne Rippin, formerly
M/F but now an independent consultant. (I kid you not, I was reading it only
five minutes prior to accessing your message) :-

    http://www.microfocus.com/files/whitepapers/wpautomation.pdf

The beauty of the whole thing, (Wayne's paper), and I can't enthuse enough, is
that he gives an intro to OLE using COBOLese wording. In essence his article is
invoking the various M/S Office suite products with OLE - in my particular case,
I got a glint in the eye when it came to using Word as a print substitute for
PC_PRINT.

For me, another added bonus, and something ticking away in the back of my mind
for years, graphics, for which I foresaw I would eventually have to buy an
add-on - not so using OLE techniques I can invoke Excel to produce its various
forms of charts.

Now I mentioned the OO word "invoke" which wont turn you on if you are not the
least bit familiar with or not currently using OO. However, subject to a
compiler supporting OO calls (currently limited to F/J or M/F - apologies if
there are others that support this), you can write standard procedural programs
which just invoke the OO classes/methods as necessary, providing you have the
various data types required with AUTOMATION.

Now as to a 'thinking aloud' example, and currently I don't write any software
associated with dollars and cents, ( mine is tied into oil/gas plants), but we
will stick with that QuckBooks Balance Sheet. Note - when I say 'do this' below,
I'm not suggesting manually, but via a COBOL program :-

- Ages ago I tried some simple template/merging in WordPerfect so I know Word
definitely has the same features

- You could create yourself a template file Balance Sheet specifying layout,
fonts etc.
Copy your blank template to a new Word document.

- Assuming a procedural program, (my particular preference would be turn Wayne's
examples into a supporting class and send parameters), either from COBOL
files/DB Tables you can invoke Word passing parameters as an array of values to
be displayed. (I haven't tired it, but you have the option of 'watching' it or
doing it in background mode based on a SendDispatch signal, at least within Net
Express).

- You display Balance Sheet, depending upon number of account entries, user may
have to scroll through 1 to 3 pages.- this is a standard Word scroll. Using the
Quickbooks technique, user wants to view a different reporting period. Merely
copy the master template to a fresh Word document and iterate through above
routine.

How effective - I don't know, but I'm guessing probably 'pretty damned good'. It
*is* worth the research.

Let me stress again, the article is specifically M/F based, but generalized
enough to describe the technique. Any compiler with OLE support must have
parallel features. If it turns out you are using POWERCOBOL, then you can
probably anticipate some extra guidance from other F/J users. Again assuming
your were using F/J, then in all probability they have some decent on-line help
material as well.

Nope, it's a question of whether or not your report information is
pre-paginated.  Consider two grids: one with fixed row heights, and another
with variable row heights.  In the fixed example I can easily tell you what
page row 10000 is on, by dividing the number of antecedant rows by the
number of rows that fit on a page.

For the variable row height case, I would have to scan through each row
(looking at its height) to determine what page the 10000th row will be on.
This example does not suggest a great difference in performance, but as you
add complexities: word wrap, embedded images, widow control, etc, it gets
more and more profound.
