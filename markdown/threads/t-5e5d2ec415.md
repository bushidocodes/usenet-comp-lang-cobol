Calling a routine in a DLL that is not written in the language that is
calling the routine depends on the calling convention used when the 
routines in the DLL were created and the ability of the COBOL compiler
to allow the user to specify the appropriate calling convention.

For example (using MF COBOL) you usually declare a "procedure-pointer"
and use the SET verb to obtain initialize the "procedure-pointer" with
the init routine for the DLL Once the DLL is loaded for the program 
you can access any of the routines with a call statement. Some 
compilers will require that you include the "xxxxxx.lib" file in the 
LINK command when you build the application.

If you have a DLL that contains several routines that were written 
using the PASCAL calling convention (this is the normal one for OS/2 
and Windows API calls)

This example is from a program I wrote to provide access to the list 
of local and network printers available from a workstation running 
OS/2

environment division.
configuration section.
special-names.
  call-convention 3 is os2api

data division.
* this is the procedure pointer used for DLL initialization
01  this-dll  usage procedure pointer.

linkage section
procedure division

* initialize the spooler api calls DLL
set this-dll to entry "pmspl.dll"

call os2api "DosPrintDestEnum" using
                   by reference szServerName
                   by value sLevel
                   by reference prdinfo3-array
                   by value usbuflen
                   by reference pusEntReturned
                   by reference pusEntAvail
                   giving return-status

I left out most of the data definitions for brevity. The actual 
arguments passed to the API routine are, of course, dependant on the 
particular API call. You note that there is a mix of "by reference" 
(passing an pointer) and "by value" (passing a value directly)

Normally SENDMAIL is an EXE program that takes command line 
parameters. You can look up the  comand line syntax for your OS 
implementation. This is sometimes changed between implementations on 
different operating systems.

I would use 

SENDMAIL -a mail-file-name -f me@mydomain.com 
theotherguy@destdomain.com

to send a file named "mail-file-name" to theotherguy@destdomain.com 
showing it is from me@mydomain.com

SENDMAIL uses a configuration file SENDMAIL.CF that must be properly 
set up before SENDMAIL will work....
