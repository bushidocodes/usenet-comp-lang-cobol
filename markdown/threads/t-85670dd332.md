Tell the student it's only easier to understand in >>small<< programs. Ask
how it would be possible to understand the flow in a 4,000 to 20,000 line
program where the coder persistently used IF.. GO TO IF .... GO TO.
After the 3rd or 4th GO TO, you lose any flow of continuity.

See if you can get a copy of Dykstra's paper of structured programming and
make reading it a class assignment. The only "complaint" I ever hear about
structured programming is "You have to think about the entire program to
write any code." Duh - that's what you're supposed to do. One programming
group supervisor felt writing program skeletons, and top-down code was a
waste of time. His attitude was "Just sit down and write the code." His
programs littered with GO TOs, as a group, were the worst collection of
incompetent trash I ever saw.

Frequent use of GO TOs are an absolute sign of poor design; the coder had
no idea of the logical structure needed to solve the problem. I've written
C and PL/1 programs of 10,000 to 20,000 lines of code with no GO TOs which
were easy to understand.

COBOL doesn't yet have the ability to write "correct" programs as in C and
PL/1, but it can if the equivalent form of the C verb "break" is
incorporated. END-PERFORM is a start.

Until then. a few, and only a very few, GO TOs PROPERLY used can be
implemented, such as GO TO paragraph-name-EXIT and only in forward
direction.

Generally, I avoid them like the plague. I've been in this business 40
years and have seen good code and have seen absolute garbage. The worst
were peppered with GO TOs. I've seen a lot of spaghetti code, and one was
so bad, a colleague referred to it as "a frog in a Waring blender."

Tom McFarland wrote:
› 
› --
…[18 more quoted lines elided]…
› does not determine which verbs to use.)

Correct. I don't think I wrote that it did. My point is that more
programs these days seem to run to thousands of lines of source code
rather than just a few hundred.

When you look at one of these efforts, it can take some time to weed
through the chaff to find the grain of the processing described by the
source program. How many of us actually have full program documentation
easily to hand when we need it? Come to that, how programs in the world
actually have _any_ helpful documentation?

When you look at a short program, it is possible to recognise its entire
structure and purpose in a very short space of time. Any errors stand
out like the balls on a greyhound.

When you limit the function of an individual program, you can also write
it, test it and prove it quite quickly. Once that is done, you can set
it aside with the confidence that it does the job asked and can be near
100% reliable.

Of course, when you write in a modular style with short programs, you
also have the examples to show those people who moan about what they see
as the shortcomings of COBOL: no local variables, poor re-use of code,
etc. With modular techniques, each module has its own local variables -
indeed the only public data is that passed in linkage.

It also has the benefit of getting rid of the mind-set that says an
in-line nested PERFORM of 425 lines (I had to debug one last week) is a
good thing. (Cue my next railing into academics who think this sort of
thing is smart!)

I have to agree as a 10 year + COBOL programmer (and a strong believer
in its future), GO TOs are not good practise but have uses. The main
use I have found is in going to an 'EXIT' paragraph of a section where
a major error has occured. Otherwise, a section would be required
around each statement where an error could occur, thus increasing the
complexity of the program.
