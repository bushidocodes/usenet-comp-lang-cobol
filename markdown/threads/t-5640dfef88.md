### Input pipe error in MF COBOL (HP-UX)

- **From:** "Michael C. Kasten" <mck9@swbell.net>
- **Date:** 2000-07-08T00:00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<39671A65.6FBED367@swbell.net>`

```
MF COBOL provides a way to read, not directly from a file, but from
another process via a Unix pipe.  This feature lets you slip in any
arbitrary preprocessing or massaging without writing an intermediate
file.  Within the COBOL runtime the pipe is opened via the standard
Unix function popen().

Since many of our files are on the large side (tens or hundreds of
megabytes), we've been using input pipes to reduce the IO overhead.
However, we've encountered a problem.

When the process in the pipe encounters an error condition, it stops
and returns a non-zero return code.  However, the COBOL program
appears to be blissfully unaware of the error.  It just sees 
end-of-file.  

In our case there's no reliable way for the COBOL program to detect the 
problem by checking the contents of the input stream, e.g. by noticing 
the absence of a trailer record.

(Before you suggest that we start using trailer records or some
similar trick, let me say that such an approach would be far more 
trouble than it would be worth, due to peculiarities of the application.
We'd resort to a separate step first, with an intermediate file.)

My question is: can the COBOL program examine the return code of the
input pipe, or otherwise detect an error in the pipe?  I know that the 
pipe's return code is available to the COBOL runtime via the pclose()
function, but I don't know if it's available to the COBOL program
itself.  If it's available at all, it's probably available only 
after the CLOSE.

The same question (and probably the same answer) applies to output 
pipes as well, but so far those haven't been an issue for us.

Michael C. Kasten	mck9@swbell.net
http://home.swbell.net/mck9/cobol/cobol.html
```
