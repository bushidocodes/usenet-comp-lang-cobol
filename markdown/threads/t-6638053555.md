› Does anyone know what the main differences are between Cobol vsII and

Since VS COBOL II has been "functionally stabilized", there have been the
following versions / releases of a new COBOL compiler for the mainframe
from IBM:

COBOL/370 (Version 1, Release 1) - added support for intrinsic functions
and two new reserved words ("function" and "procedure-pointer"); this
compiler (and all the subsequent releases and versions listed below)
requires the Language Environment (LE) product for a run-time library; LE
is included automatically with OS/390 but must be purchased separately
under MVS.

COBOL for MVS & VM (Version 1, Release 2) - added a lot of new
functionality: object oriented techniques and support; recursive programs;
null-terminated strings; ability to pass back a return value, like a C
function; ability to receive a returned value; ability to use single
quotes (apostrophes) and double quotes for bounding literal strings in the
same program (as long as the opening and closing characters are the same,
that is, either both double quotes or both apostrophes, for each string);
support for program names longer than 8 characters and for program names
that are case-sensitive; local-storage, for holding variables that are
alloceated on entry to the program and deleted on exit; ability to pass
and receive arguments BY VALUE.

COBOL for OS/390 & VM (Version 2, Release 1) - general availability was
just May 23, 1997; adds support to create and use DLLs (Dynamic Link
Library) and updates support for some OO features (details hazy just yet:
I've only seen the announcement, not the documentation).

Of course, we have courses available on the details of all these versions
and the supporting LE environment.

=============================================

Regarding your recent reply to my earlier note, mainframe languages in the
States are COBOL (by a wide margin), then PL/I, C, and Assembler, and some
FORTRAN; this is just observation regarding the clients I've been working
with.
Most shops are not doing new development in Assembler, and some shops have
tried to outlaw / replace all Assembler application programs but, as I
sometimes hum to myself, "every shop has some Assembler somewhere" (to the
tune of "Everybody Loves Somebody Sometime").

PL/I was grabbed onto by a number of large clients in the early years and
they have a large inventory of programs. However, it is hard to find PL/I
trained programmers, and most 3rd party developers / code generators use
COBOL, so I see some shops that had PL/I moving towards replacing it with
COBOL or possibly C.

C is fairly new on the mainframe front, but because most of the new MIS
grads only know C, the new kids on the block are having some success in
getting C accepted in MVS shops. Also, with MVS OE and OS/390 supporting
POSIX / Unix, using C on the mainframe becomes a little more like "home"
for folks who grew up only knowing C and Unix. (And a little less familiar
for those of us who grew up with DOS, MFT, MVT, etc.)

Of course, there's a smattering of other languages for specific
environments and applications, including REXX, CLIST, SAS, Natural, etc.
