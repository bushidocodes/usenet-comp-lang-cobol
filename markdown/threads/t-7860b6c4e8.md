AFAIK it does,

However there could be problems depending on what you want to do with the
records and why they need to look as though they were generated on an OS390.

Don't forget that files will be created as ASCII.  If the records are being
sent to an OS390 or read from an OS390 then they will get converted to
EBCDIC.  This is fine for any display fields, but disastrous for packed
decimal or binary fields.

If you want to send records between ASCII and EBCDIC machines it is better
if all fields are display.

BTW I thought COMP in IBM Cobols means packed decimal.  It does on AS400,
but I can't remember back to my 360/370 days.   If this is so then these
need to be changed to COMP-3 because COMP in MF means binary.

Rick

The point is to generate a file (on the win95 box) that is not all ascii.  This
can be done with VA COBOL by using the CHAR(390) compiler option.  This changes
all ascii strings to ebcdic, which is fine for the transformation engine that
I'm using.  More importantly, it converts PIC 9 and PIC S9 numeric display types
to IBM types (e.g.  +1234 which is written out as 0x31 31 33 44 by default is
converted to 0xf1 f2 f3 c4).  The BINARY(390) option converts all binary data
types to big-endian.  There is also a compiler option to convert floating point
data as well.

I'm not sure what transport you're speaking about.  Data can be move between
machines where no conversion occurs.  ftp in binary mode is a common one.  Most
middleware products allow the same and also provide the capability to perform
complex conversions.  This is a non-issue for the transformation engine and
middleware that I'm using.

The IBM manuals state that COMP = BINARY and COMP-3 = PACKED-DECIMAL.  They are
nice enough to provide pdf manuals on-line.  So is Fujitsu.  MF on the other
hand doesn't provide this information to the web.  I'm thankful for this news
group.

--matt

Just one other additional bit of information on this thread.

*BOTH* Micro Focus (MERANT) and IBM provide support for creating "EBCDIC"
files on the PC.  Both fully support all the BINARY options and datatypes
that are compatible on OS/390 and PC.

HOWEVER, only IBM provides an option for COMP-1 and COMP-2 to be stored in
OS/390 style floating-point format.  MERANT supports COMP-1 and COMP-2 - but
uses IEEE format (and provides a utility WFL for converting between the two
formats).  IBM allows you to 'select' which format you use on the PC.

NOTE WELL: Very few IBM COBOL applications (that I know of) actually use
COMP-1 or COMP-2 - but if you need it, this is an issue in choosing between
the 2 compilers.
