1    In your COMPUTE statement you have referenced the dataname FEETYPE1
instead of FEE-AMT, which is the dataname that has the OCCURS clause.
Only data items that have an OCCURS clause or are subordinate to data
items that have an OCCURS clause can be subscripted or indexed.

2    You have used the variable ARRAY-INDX as a subscript in the COMPUTE
statement, that would be OK if ARRAY-INDX is an integer data item, but
unsafe if it is a real COBOL index that is associated with another
table, because COBOL indexes are specific to the tables for which they
are defined and can only be used for the original table or other tables
that have identically sized occurring elements.  You haven't told us how
ARRAY-INDX is defined.

3    The term subscript is normally used for data items defined as
integers that are used as subscripts, while the term index is normally
used for indexes defined as being explicitly associated with specific
tables via the OCCURS clause.

4    The statement MOVE ZEROS TO ARRAY-INDX is unnecessary, because the
PERFORM statement specifies the initial value required in the BY phrase.

5    While not essential for the correct operation of your program, your
nomenclature for the data items use to construct your table of fee
amounts is potentially confusing as all group items have a prefix of
FEETYPE, whereas "FEETYPE" as a dataname refers to an independent
variable for which you have not provided a definition.  I would also
prefer to use a name such as FEE-INDX rather than ARRAY-INDX to show its
relation to the table being processed and even better use the name
FEE-SUBSCRIPT or FEE-SUB if it really is a subscript and not an index.
Futhermore it would be even better to specify ARRAY-INDX as the index
for the table containing FEE-AMT and then give the index the name
FEE-INDX. This could be done by using the following statement for the
definition of FEE-AMT:

05  FEE-AMT PIC 9(6) OCCURS 6 TIMES INDEXED BY FEE-INDX.

provide an equivalent to DSNTIAR, its use is not portable across platforms.

are many more than 90 and to be incorporated as a subroutine for general use
by all programs within an installation.  Because the table would be quite

A Binary search (in IBM Asembly Language) has more overhead than a

divided by 2. Then the index is applied to the compare instruction).

the table to compensate for the machine time overhead; fewer than
that, and the overhead for the binary search takes longer than the
additional accesses used in a linear search. Obviously, you could get
a hit on the first read either way, but this number was also said to
take into account the "odds" of a match, using either search.

anything, despite there being sometimes significant differences between ODBC
drivers and their error codes.

Your point about SEARCH...ALL is correct, for cases where large (many
entries) tables are involved, but again, Jimmy's solution will work, and for
90 entries, searching on a 5 character table key (in a worst case scenario
where the entry is the last in the table), it would be under 1 millisecond
on a Pentium...not the kind of interval a human is likely to notice. It
should also be remembered that this code is executed on extreme exception;
if the system is closing down or the process is abending anyway, it really
doesn't matter if it takes a second or two...

SEARCH ALL (as I'm sure you know) requires the table to be sorted in Key
sequence, so that a binary chop can be made on it. There is an argument
(particularly in on-line mainframe environments like CICS and IMS-DC) that
it can be more efficient (not that we should be too obsessed about
efficiency when we use today's platforms...) to have your table ordered in
statistical frequency and NOT use SEARCH...ALL. (Organise it so that the
most likely hits are at the "top" of the table...) Like everything, the
whole environment needs to be considered and there may well be exceptions to
the general rules, based on specific requirements.

I would stress that I am not taking issue with the points you raised, when
applied to a mainframe environment (it was an excellent post). However,
there are many small, independent developers using COBOL who are less
concerned with "installation standards" and who need to take a broader view
than is necessary in a mainframe environment.

Your points about dynamic linkage and setting the table up as a Load Module
are as true today as they were in 1967 when I first did this...(It just
doesn't matter as much today...<G>), and the dynamic loading is well handled
by Object Oriented COBOL on the PC

I liked Jimmy's solution. And he did say it was his first attempt to use
SEARCH...(a significant move forward for many COBOLlers...).
