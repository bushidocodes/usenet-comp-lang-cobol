Seriously, I use a similar method for anything stronger
than read-a-record print-a-line.  I just write out the
functions I want the program to perform, then
keep that on my desk in front of the keyboard while
I'm coding.  Each function becomes a subroutine.

After the program is in production, that basic structure
goes into a file folder as documentation and it really
helps if (no... make that when) I have to go back to make
a mod.  Along with a source listing, of course :)

That's also helpful if the program is going to write a temporary
file or two.  Jot down the data names for filename, record name,
etc. and it sure saves a lot of flipping back up to the data divison.

>subroutine to do the action.  are there guidelines to go by?

A guideline which is frequently quoted is "subroutines should not exceed
a page of code."  However, I have found that artificial and not overly
helpful.  First and foremost, a subroutine should "embody one clear and
separate purpose".  This is important for logical abstraction.  Secondly,
you don't want to put too much code in a subroutine, because it becomes
more difficult to mentally assimilate.  This is where the 'one page'
idea makes some sense, though probably too restrictive.  For in some
programs, such as many online programs, you may have long series of field
validations, each a simple sentence.  There is often no logical place
to break that code up, so you can end up with one very long routine.
But such code is usually easy to recognize and understand.  On the other
hand, breaking up your code into a multitude of subroutines, many of
which have no "clear and separate purpose", also makes the code difficult
to understand.  Good code strikes a balance between these priorities.  In
the final analysis, it becomes more of a skill and an art, rather than an
exact science.
