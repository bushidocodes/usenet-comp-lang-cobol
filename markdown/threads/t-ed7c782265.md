ASP.NET, HTML5, CSS3, javascript and C#, I have come to the conclusion that

everything. The most frustrating thing for me is that it doesn't implement

Firefox is "OK",  (reluctance to upgrade by its user base is holding it 
back - see the link) but Chrome really rocks and does everything I have 
asked of it. I'm still not quite ready to make it my default, but I may do 
so if they don't fix IE 10.

link:

Cobol components compiled as DLLs... manually installed using COM+ (no
wizard deployment). So I could use it directly (anytime) in my ASP and/
or ASP.Net web pages (using jQuery, AJAX).

Say in my ASP page, calling my cobPogram.DLL;

<%
Dim stat, usrname, usrpasswd, usrIPadd
Application("appIPadd") = request.ServerVariables("REMOTE_ADDR")
usrIPadd = trim(Application("appIPadd")
usrname = trim(request.Form("qName"))
usrpasswd = trim(request.Form("qPass"))
'then calling my Cobol application to check the user login
stat = myCobObj.checkUser(usrname, usrpasswd, usrIPadd)
if stat = "00" then
  response.Write("SUCCESS!")
else
  response.Write("FAILED USER LOGIN, TRY AGAIN!")
end if
%>


the "myCobObj" is executed beforehand by default using "global.asa"
for classic ASP as;

<object runat="server" progid="cobProgram" id="myCobObj"
scope="session"></object>

Yes, it has been for some time. :-)

These days, I wire the ASP.NET pages to a C# code-behind and there are no 
worries about creating objects. The code-behind is event driven and it just 
executes C# which creates any COM instances it needs in exactly the same way 
as a program would.

Looking at what you have above, it seems likely to me that COBOL could be 
wired up to a code-behind page in exactly the same way. (Although I have 
never tried it. It would have to be precompiled; in C# it can be precompiled 
or JIT compiled.) The advantage in doing this is that the pages of the web 
site become encapsulated page pairs; you have a layout page in ASP and the 
"smarts" it needs are automatically attached to it. It makes maintaining 
things fairly straightforward. Of course, there is also provision for common 
code that can be activated from any page, but using the wired code-behind 
each control on the layout page can be made "smart".

I have run COBOL COM objects from the code-behind without problem (they must 
be registered on the server, of course), but they have only been called from 
C#, I haven't tried doing it with native COBOL.

Thanks for posting your samples. I found it interesting.
