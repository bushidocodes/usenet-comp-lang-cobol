Au contraire.

The INCLUDE mentioned is to do with the SQL pre-processor. This runs
(Cobol 101) as part of the compilation process to take EXEC...END-EXEC
code constructs (used for CICS and DB2 on IBM Cobol) and convert them
in to Cobol which the compiler can understand. If you use INCLUDE in a
program which is pre-processed to convert DB2 SQL statements in an IBM
Cobol program to Cobol code which can be understood by the compile
step then you will get an error message about the pre-processor being
unable to find the include member and the compilation will stop at
that point.

If you are not using SQL and do not do any pre-processing and are not
using Panvalet (or Librarian) then a simple check of the Cobol manual
will reveal whether INCLUDE is a reserved word in the dialect of Cobol
that you are using. If it is not a reserved word and bearing in mind
all the crap above about pre-processing then it is safe to use it as a
data-item name. I wouldn't but them Im not brave.

From time to time when writing COBOL this issue rears its head.

As there are some 650 reserved words in COBOL on average (varies with 
different dialects, of course) it would be tedious to try and memorise them 
all, especially when some of them are seldom used or not immediately 
obvious.

The wise old grey badger who taught me COBOL when I was just a bright-eyed 
whipper-snapper advised as follows:

"If you want to use a dataname and suspect it might be a reserved word, put 
a hyphen in it."

As that was in 1967 and COBOL has expanded a bit since then, his advice 
might not be sound today, but it can be modified so it would be. For example 
if you add a prefix or suffix to the name that includes a hyphen AND 
something not run-of-the-mill, you can be pretty certain that what you have 
will cause no problems.  (XX-INCLUDE)

I just checked the list of reserved words I use when parsing COBOL source 
and found INCLUDE is in it. However, that may not mean anything for the 
version you are using.
