```
The following are the rules that relate to the

    SET data-pointer UP/DOWN BY arithmetic-expression-1

statement in CD 1.8

"17) If arithmetic-expression-1 does not evaluate to an integer, the
EC-SIZE-ADDRESS exception condition is set to exist, the execution of the SET
statement is unsuccessful, and the content of identifier-10 is unchanged.

18) The address contained in identifier-10 is incremented, if UP is
specified, or decremented, if DOWN is specified, by the number of bytes
specified by arithmetic-expression-1. If this new address is outside the
range of values allowed by the implementor for a data-pointer data item, the
EC-RANGE-PTR exception condition is set to exist and the value of the data
item referenced by identifier-10 is unchanged."

   ***

I think that this "resolves" your concerns.

(FYI, an arithmetic expression can be a single numeric item - so your
Packed-Decimal, Display, or even COMP values would all be treated as the same
number - as would a numeric literal.)

FYI-2,
  The size of a "byte" is totally "implementor defined," so if you are
talking about a 7-bit, 8-bit, or 16-bit architecture, the result of the SET
statement is STILL to increment/decrement to the next "byte" - as defined for
that architecture.
```
