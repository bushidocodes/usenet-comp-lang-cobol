>                ARRAY-FULL
I believe this is the problem here. You're not filling the array at this point 
-- it's already been loaded, and now you're scanning it -- so there is no need 
to test for ARRAY-FULL. Since the array has already been populated, presumably 
ARRAY-FULL is already true, and the program is doing exactly as instructed: it 
PERFORMs the instructions inside the loop once, then tests for the end-of-loop 
condition, sees that ARRAY-FULL is true, and terminates the loop after one 
iteration.

Instead, what you need to test for is AR-INDEX > MAX-ARRAY-INDEX (or whatever 
name you have given the variable that holds the number of array elements). 
Alternatively, you could add one more element at the end of the array, and set 
the date in that entry to HIGH-VALUES; then you need only to test for 
AR-DATE(AR-INDEX) > AR-END-DATE.

>          ADD 1 TO AR-NUMBER-OF-DAYS
>          ADD AR-HIGH-TEMP(AR-INDEX) TO AR-HIGH-ACCUM
…[8 more quoted lines elided]…
>          ADD 1 TO WA-NUMBER-OF-INDEXES
Not sure what this statement does, but I'm guessing that ARRAY-FULL is a 
conditional (88-level item) under WA-NUMBER-OF-INDEXES. This statement appears 
unnecessary here. Again, remember that you're now merely scanning an 
already-populated array.The count of array elements won't change any more, so 
you should not alter that variable.
>   END-PERFORM
>..

Again, your array is already filled. This serves no purpose.

>then I get a good result.  I just want to know how to properly use the TEST

You're using TEST AFTER properly. You just don't have the termination 
condition quite right yet. You don't need to check if the array is full, you 
need to check to see if you have scanned past the end of it.

>AFTER function.  Or any way to do a test at the bottom of the loop, run the
>statements then do the test.  Which I think is a do until, right??

PERFORM WITH TEST AFTER is the Cobol equivalent of DO UNTIL in other 
languages. Perhaps it would help if you imagined how you might code the same 
problem in Pascal or C.
