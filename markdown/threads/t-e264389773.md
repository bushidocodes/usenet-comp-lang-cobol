TRUNC/NOTRUNC. The programs should be written not to
depend on this option.

like this:

MOVE 9000 TO THE-VAR

MOVE THE-VAR TO THE-VAR <=== this is the tricky one

Always specify compiler and version and platform.
It looks like you are using MicroFocus. MF is
notoriously buggy when it comes to COMP variables.
The fact is that you take your chances here.
Over the years I have collected a number of
MF compiler errors all related to COMP. There is not much
you can do about it.

My MF Cobol manual says: TRUNC specifies whether data
stored into a USAGE COMP item is to be truncated.

This seems to include COMP-3

In general, a program should not be written so that it
depends on the setting of the TRUNC/NOTRUNC switch
(actually ANY compiler option, but sometimes you have
little choice).

hey Robby, the reason is simple:

The number of 9's in a COMP-3 field must be odd!! ('cause the
Sign-nibble add the
last half-byte to the number, that then will be divided by 2). So a
field defined
like S9(6) will store the same information as a field like S9(7) (both
are 4 Bytes long with zero-filling from the left!).
The TRUNC directive cut the stored number to the
right number of *defined* digits, rather then the NOTRUNC directive,
wich does
nothing like this and store as much as he can at runtime and the
variable-boudings
are not exceed by the storing.

So your output can be easily explained:
you always use even number of digits instead of odd-numbers, so a change
to

S(9)V9(8)
S(8)V9(7)
S(7)V9(6) ...

will be work fine (just test it..)

Okay, here's what my book (COBOL II, Harvey Bookman) says on TRUNC:

1) The TRUNC option is one of the most difficult and confusing of the
compiler options to understand.

2) It affects binary data fields.

What is binary and what isn't is compiler dependent when you use COMP,
COMP-3, COMP-4.
On the IBM platform, COMP and COMP-4 produce binary data items, and COMP-3
packed-decimal. But this is not the same on every platform. On some
platforms, COMP-3 produces binary data items.

Try out using BINARY (and PACKED-DECIMAL) instead of COMP. This way, you
know what data item you're creating regardless of the compiler. I know
typing out PACKED-DECIMAL is a pain, but I use cut and paste anyway, so I
rarely type it out more than once.

3) The TRUNC option assures that after arithmetic is performed on a binary
field or when data is moved to a binary field, if the resulting field
exceeds the maximum value that the field is defined as, the result will be
truncated to the PICTURE size. This is how packed decimal and display
numeric fields are handled. Extra machine code is required to truncate
binary fields to their PICTURE size, . . .

There is further discussion, but it begins to deal more with the IBM
implementation. What you really need is a description of the SUN compiler
implementation, plus good descriptions of how the various USAGEs are
implemented on the SUN platform. Contact SUN technical support, or the
compiler's technical support (that is, if either option is available to you;
I have seen a lot of interesing messages on this board about Microfocus'
tech support).
