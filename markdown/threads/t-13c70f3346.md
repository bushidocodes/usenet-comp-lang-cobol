```
As far as I know, there is no intrinsic function or LE callable service to
cause a program to "wait".  However, there is a COBOL "callable" program
(available since OS/VS COBOL - if not earlier) that will do this.

If you do a CALL to "ILBOWAT0"  (first of  those is the letter "O" then the
number "0")

Pass it a full word binary field - Pic S9(8) Binary - with the number of
seconds to wait.

You should make certain that the COBOL program that is calling this
subroutine is compiled with DATA(24).  If you have a LARGE COBOL program
that you don't want to use DATA(24) with, then make an "intermediate"
program that does the actual call to ILBOWAT0.
```

```
Please, PLEASE do not do a dynamic CALL passing a 31-bit "piece" of storage
to ILBOWAT0.  It is simply NOT designed to get such - no matter HOW you call
it.   ILBOWAT0 (and many other "older" programs) really, REALLY expect to
only be passed data "below" the 16M line.  This is the reason that the
DATA(24) compiler option exists.

OBVIOUSLY, even if you compile with DATA(31) you MIGHT get storage below the
line, so a "single test" won't prove anything.
```

```
For example, a batch job (something like MacKinney batch CEMT) is used
to submit a CEMT SET FILE(fctname) CLO command into a CICS region.  Or
perhaps you want to switch the CICS dump data sets, or start an online
transaction.  In any event, you want the batch job to wait a bit

CICS action.  So you put a step in your batch job that waits 30

immediately.  So the COBOL program waits 5 seconds before looking for
new data.
```

```
I won't try and explain VSAM, ESDS, CEMT, etc (as I think that was a
semi-rhetorical question).  The answer to WHY you might want a "run forever"
program to sometimes "wait" (stop taking CPU time), the answer is VAGUELY
similar to "time slices" in a PC environment. (The following is HIGHLY
simplistic and all MVS, OS/390, and z/OS gurus can have as much a laugh as
they want.)

On a large IBM mainframe it is likely that you have 10's of programs (or
more?) "running simultaneously".  This is done by the CPU swapping various
programs in and out. There are a VARIETY of ways that MVS (and later)
operating systems have for "prioritizing" shops that are "virtually" running
at the same time.  The oldest simply prioritizes which program STARTS when.
Online applications (often IMS or CICS "transaction processing" regions)
keep going forever and do their own prioritization of jobs (and other
resources).  Relatively new (actually not that new) IBM introduced "WLM" -
which can do a lot of "prioritizing" of shops based on whatever "criteria"
you feed it.

HOWEVER, it is still possible to have a job "looping" thru infinite
processing (almost a "message driven paradigm" <G>).   They may even have
"ownership" of some resources (besides the CPU) that other programs will
want.

The BOTTOM-LINE is that sometimes (less and less as time passes and as new
"prioritization" options becomes available) it is important for an
application to say -
    Please, CPU, don't waste any of your "power" on me - but go and do your
thing with other programs and then come back to me "in a little while".
```

```
Hi Man,

If your pgm is a CICS pgm (maybe it s/b since the IMS data provider sounds like
it is), you could have the IMS xaction create a TDQ rec for each DB2 row it
creates. The TDQ would have a trigger threshold of 1 (or any other quant that
makes sense), then your pgm can be STARTed. Of course your code would be
modified so that your pgm would START itself upon exit to propagate the endless
loop. Talk to your tech guys to get the pgm into the CICS automatic startup
library (I forgot its name).

The advantage of this approach is that the pgm executes only when there is data
in the TDQ. Also when your pgm reads the rec, it's immediately deleted. You may
want to rethink the 1 rec threshold to reduce the scheduling overhead.

Regards, Jack
```

```
Hi Man,

If your pgm is a CICS pgm (maybe it s/b since the IMS data provider sounds like
it is), you could have the IMS xaction create a TDQ rec for each DB2 row it
creates. The TDQ would have a trigger threshold of 1 (or any other quant that
makes sense), then your pgm can be STARTed. Of course your code would be
modified so that your pgm would START itself upon exit to propagate the endless
loop. Talk to your tech guys to get the pgm into the CICS automatic startup
library (I forgot its name).

The advantage of this approach is that the pgm executes only when there is data
in the TDQ. Also when your pgm reads the rec, it's immediately deleted. You may
want to rethink the 1 rec threshold to reduce the scheduling overhead.
```
