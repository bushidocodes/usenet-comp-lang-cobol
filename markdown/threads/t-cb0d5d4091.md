QSAM and BSAM (along with BDAM) are members of the SAM -
Sequential Access Method family.  BDAM - Basic Direct Access
Method - has been replaced by VSAM, as has BISAM and QISAM
(Indexed Sequential Access Method family.)

At the access method level, BSAM is READ and WRITE of physical
blocks.  QSAM is GET and PUT of logical records.  Calls to QSAM
wait until a logical record is available before returning.  BSAM
returns immediately and requires that a CHECK or WAIT be done
before assuming the I/O is complete (i.e. looking at a just read
block, or reusing a just written buffer.)  BSAM also allows
positioning in the file by POINTing to the next block you want to
read or write.

The performance for both is quite good.  If you do several READS
at the start of the program, all buffers will be filled in one
I/O.  If you can process the data fast enough and get a buffer
returned in time, you can keep the I/O going.  QSAM will try hard
to keep buffers filled so that each GET you do will always be able
to return a record without waiting for actual I/O.

In the old days (like before MVS), you could make a big impact
doing I/O by increasing the number of buffers, and using BSAM to
keep buffers scheduled for I/O.  When SAM/E appeared, QSAM took a
big performance gain, catching up to BSAM.  To get better
performance then either access method, you have to write your own
channel programs and manage real storage yourself (EXCP/VR).

VSAM uses a different interface to the raw I/O system (called
media manager) and can out perform either.  Compare a striped data
set of 1 stripe (which is managed by media manager) with a regular
SAM data set.  1 stripe will win almost all the time, and tie when
it doesn't win.

Here's some info from my vague memory many beers (oops - years) ago.
This is what we did - at the time I remember thinking it was always
necessary. Perhaps there were other options.
With BDAM, a programmer declared the number of blocks he was going to
use for the file. Then he wrote an assembler randomizing routine, based
on some key in the records. Using the key, the rr calculated the
precise block the record should be in. I can't remember the math
anymore, but it seems we used some sort of modulo arithmetic using a
prime number. I.e., divide the key by 13.
You couldn't simply divide by the number of blocks, because then (if
you assigned keys sequentially (e.g., Purchase Order Number), the
records would all be in the 'early' blocks - you needed to spread them
around (see below).
When you read/wrote a record you called the randomizing routine to
access the block. Somehow the precise record (of all those in the
block) we wanted came back to us - I don't remember the tech details on
that.
 The theory was to write a randomizing routine that would spread the
records around the blocks. I.e., if there were many records for Chicago
and few for Hick City, you wanted to avoid a 'lump' of records on the
disk in the blocks calculated for Chicago. So the rr would try to
spread Chicago's records all over the place. Similarly, key 6788 should
go to a block far from key 6789. There was a utility that showed the
demographics of the population of the blocks, as I recall.
Whenever you needed more blocks you had to modify the calculation of
the randomizing routine.

Sorry for the wishy-washy on details - it was too long ago.

BSAM and QSAM are access methods, not file types. EXCP is also a valid access
method for flat files. The difference between these three access methods is
the amount of control and complexity of the code.

EXCP will allow complete control over the file. You can read specific blocks,
specific bytes within blocks, in any order, at any time. The tradeoff is that
you must write the channel programs, i.e., the orders issued to the device to
perform the functions. Most control, most complexity.

BSAM will allow repositioning to a specific block, ability to read/write keys,
process multiple PDS members and the ability to perform the I/O asynchronously
at your convenience. The trade off is that you will have to block/unblock the
records and provide buffer management. A lot of control, somewhat complex.

QSAM will allow you to easily read/write sequentially thru the file. Records
will be blocked/unblocked and buffers will be handled automatically. The trade
off is that you cannot reposition the file or choose when to schedule the I/O.
Limited control, relatively simple.

In general QSAM will be used as it is the easiest access method to code.
