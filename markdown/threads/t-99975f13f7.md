 Hi:

I have always felt that most software is written by the 'Wrong People'.
For example, a 'computer-scientist' is the last person who would be
qualified to design and program a payroll application; they would
approach the application from the wrong perspective, and, in all
likelihood, using the wrong tools like C++, Java, VB and their
odious ilk.

Pete Dashwood recently provided fuel for my fire by sending an
article titled 'Building user interfaces for object-oriented
systems, Part 1' by Allen Holub (AH). (You can read the
article at www.holub.com.) Mr. Dashwood describes AH as
a 'World Authority' and 'lecturer at UCBerkeley' on OO.

AH is an apparently distinguished programmer whose experience
is largely C++ and Java. COBOL is not mentioned in the article.
He appears to be a 'computer-scientist' but I seriously wonder if
he has ever designed, programmed, implemented and supported a
SIGNIFICANT business application. For sure, if he had, the
article would, I think, reflect that, but it doesn't.

AH makes some observations about system design and uses them to
argue for an OO approach. To me, the arguments lack validity, are
somewhat contrived, and therefore can not be used to support the
argument. Furthermore, anyone who would suggest using Java for
business applications must be suspect. . .

Since AH appears to be on the forefront of OO development (albeit
in Java), it is not unreasonable to assume that he and others
of his type, are establishing YOUR OO future. Yet, I think that
his background is not suitable to the task when the task is
business applications. If one's perceptions of the problem are
wrong then the solutions will be wrong. (If you need to be convinced
of the utter absurdity of Java vis-a-vis COBOL, read the second in the
series of articles by AH.)

After writing this stuff, I thought about the hostile reception
it would probably receive in the CLC and that AH might take a dim
view of his article being savaged. I decided to post it anyway
with my advance apologies to AH and those who do not share my
anti-OO views.

I have extracted bits of the article and made some comments about
them below.
                   *           *           *

"Bjarne Stroustrup, the inventor of C++, once characterized object-
oriented
programming as 'buzzword-oriented programming' and certainly one
of the most abused (or at least misunderstood) buzzwords in the pack is
object itself."

---Sorry, but I couldn't resist to point out that the article refers to
   a veritable cornucopia of buzzwords, viz: Unicode, SQL, Model View
   Controller, Visual Basic, MFC, Visual Cafe, Visual J++, C++, Visual
   Age, Jbuilder, JavaBeans, Abstract Windowing Toolkit, Remote
   Invocation Method (Pete threw in Active-X and Corba) as well as
   all the usual OO buzzwords and perhaps others I missed.

                   *           *           *

"here are some rules of thumb that you can apply to see if you're
really looking at an object-oriented system:

1. All data is private. Period. (This rule applies to all
implementation details, not just the data.)"

---Private to whom or what? And, if it is private how do you
   share data?

2.get and set functions are evil. (They're just elaborate ways to
make the data public.)

3. Never ask an object for the information you need to do something;
rather, ask the object that has the information to do the work for
you."

---Isn't this like a sub-routine?

4. It must be possible to make any change to the way an object is
implemented, no matter how significant that change may be, by modifying
the single class that defines that object.

5. All objects must provide their own UI."

---I assume that the UI is presented on the screen. How can the object
   know about the current state of the screen if all objects provide
   their own UI? Do they consult each other?

                   *           *           *

"Don't confuse ease of maintenance with lack of complexity, by
the way. Object-oriented systems are usually more complex than
procedural
systems but are easier to maintain."

---Sorry, but that doesn't sound good to me.

"The idea is to organize the inevitable complexity inherent in real
computer
programs, not to eliminate it. Object-oriented designers, as a class,
consider the elimination of complexity to be an impossible goal."

---There are two types of complexity. The first is the logic required,
   for example, to process a file sequentially with many levels of
   control. The second type is the application complexity such
   as calculating taxes, etc.

---The former can be eliminated by providing a universal model for
   that class of applications. The latter you are stuck with.

---Although the elimination of one type of complexity is
   possible, the propOOnents make no attempt to remove it because
   they do not recognize the possibility.

---(If the object-oriented designers can't eliminate complexity,
   perhaps we should eliminate the object-oriented designers.)

                   *           *           *

"If anything, good object-oriented systems are more complex than
procedural
ones, but in such systems the program is better organized and thus
easier
to maintain."

---How does 'better-organized' become 'more complex'? I would like to
   see a real-world example of this. Perhaps, AH could be persuaded
   to enter the Great Programming Contest?

                   *           *           *

"Consider a system designed to get names from users.  You might be
tempted to use a TextField from which you extract a String, but that
just won't work in a robust application. What if the system needs
to run in China?  (Unicode comes nowhere near representing all the
idiographs (sic) that comprise written Chinese.) What if a user wants
to enter a name using a pen (or speech recognition) rather
than a keyboard?"

---What if the user wants to use his A/R system for A/P?
   This is just another fatuous argument.

---A business (COBOL) programmer would never "use a textfield
   from which you extract a string" to get a name.

                   *           *           *

"In the simple example above, you're tasked with adding an employee ID
to
every name in every screen that displays employee names."

---Sorry, but this is an unrealistic example. Whoever designed the
   system which would ultimately require this modification would
   need a good talking-to.

                   *           *           *

"I'll use an ATM machine for this example (as do many books), not
because
any of us will be implementing ATMs, but because an ATM is a good
analog
for both object-oriented and client/server architectures.  Look at the
central bank computer as a server object and the ATM as a client object.

A procedural solution

Most procedural database programmers would look at the server as a
repository of data and the client as a requester of that data. Such a
programmer might approach the problem of an ATM transaction as
follows:

1. The user walks up to a machine, inserts the card, and punches in his
   or her PIN.

2. The ATM formulates a query of the form "give me the PIN associated
with
   this card," sends the query to the database, and verifies that the
   returned value matches the one provided by the user. The ATM sends
   the PIN to the server as a string -- part of the SQL query -- but
   the returned number is stored in a 16-bit int to make the
   comparison easier.

---A business programmer would never store the PIN as a '16-bit
   integer'.

3. The user requests a withdrawal.

4. The ATM formulates another query, this time: "give me the account
   balance." It stores the returned balance in a 32-bit float.

---A business programmer would never store the balance as a
   '32-bit float'. Just another example which proves Java is
   NG for business.

5. If the balance is large enough, the machine dispenses the cash, and
   then posts an "update the balance for this user" to the server."

---Later on, AH 'improves' this 5-step solution with a 9-step solution.
   (The 5-step solution does not meet with his approval. He did say
   that OO was 'more complex', though.)

---I get the impression that, when people talk about client-server,
   that they look upon the program on the client as being completely
   separate and different from the program on the server and that the
two
   just send messages back and forth. This is implicit in AH's example
here.

---Using COBOL and SP2 the SAME program can run on a stand-alone
   PC, be executed by stations on a LAN directly from the server,
   and, be executed as a thin client. The SAME physical program.

---So, my ATM solution has a SINGLE COBOL program.

   1. Accepts card number and PIN from customer.

   2. Looks in ISAM file to verify that PIN is correct for card number.

   3. Customer asks for money which is dispensed and ISAM record
      rewritten to reflect reduced balance appropriately.

---Apart from the mechanical considerations of dispensing the money,
   the ATM application is much simpler than the Time Clock example
   in the Great Programming Contest.  Perhaps, we should have a separate
   ATM contest for the OO fraternity to put their theories into
   practice?

                   *           *           *

"So what's wrong with this picture? Let's start with the returned
balance.
What happens when Bill Gates walks into our bank wanting to open a
non-interest-bearing checking account and put all his money in it?
We really don't want to send him away, but the last time we looked,
Bill was worth about 87.94 gigabucks. Unfortunately, the 32-bit float
we're using for the account balance can represent at most 20 megabucks
(4 gigabucks, divided by 2 for the sign bit, divided by 100 for the
cents)."

---It is silly to suggest that Gates would put all of his money into
   a single checking account. I would hope that people who design
   ATM systems for banks pay attention to the rules and regulations
   of the bank governing the amount of money that can be in an
   account and how much may be withdrawn with a single transaction.
   Surely, the bank would determine the maximum amount and the
   designer would have made the appropriate fields as big as
   required. Why not have some REALISTIC EXAMPLES to reinforce
   the OO theory?

"Similarly, the 16-bit int used for the PIN can hold at most four
decimal digits. What if Bill wants to use "GATES" (five digits)
for his PIN"

---"GATES" is not 'five digits' it is five characters. And,
   the COBOL programmer would not use a '16-bit int'.
   This is another silly example of an artificial requirement
   that wouldn't happen in the real world. A bank is not going
   to let people arbitrarily change the format of their PINs
   and why should they?

"The procedural solution to all these problems is to change the ROMs in
every ATM in the world (since there's no telling which one Bill will
use)
to use 64-bit doubles instead of 32-bit floats to hold account balances
and to 32-bit longs to hold five-digit PINs. That's an enormous
maintenance
problem, of course. "

---I don't know much about the innards of ATMs but would hope that
   they use PCs rather than ROMs. If people are designing ATMs using
   ROMs rather than PCs, they need a good talking-to.

                   *           *           *

"Stepping into the real world for a moment, the cost of software
deployment
is one of the largest line-items on an IT department's budget."

---When VB, C++, Java. etc., software takes up millions and millions of
   bytes and can only realistically be distributed by CD, then
   software deployment would be a large line-item.  However, I
   would guess that a COBOL ATM EXE would be less than 200K -
   NOT 200M as would probably be the case for Java.

---If ATMs used PCs and a thin-client COBOL program, they could easily
   be updated while the city sleeps at no cost because of the
   teeny (relatively speaking) size of the EXE.

---Of course, if the ATM was a node on a network, running a COBOL
   program, it wouldn't be necessary to update each ATM at all.

                   *           *           *

"One of the appeals of Java (and the NC architecture, which is still
alive and well) in an enterprise-level company is that, by eliminating
deployment costs, you can save hundreds of millions of dollars every
year. The client/server equivalent of "swapping all the ROMs" --
deploying new versions of the client-side applications -- is a big
deal.

---But it is only a big deal because of the horrific overhead inherent
   in languages other than COBOL (see above).

                   *           *           *

"Similar maintenance problems exist inside most procedural programs,
even
those that don't use databases. Change definitions of a few central
data
types or global variables (that is, the program's equivalent of the
data
dictionary), and virtually every subroutine in the program might have
to
be rewritten. "

---Rewritten?!?!? Recompiled, perhaps, and that can be avoided with
   foresight.

                   *           *           *

"It's exactly this sort of maintenance nightmare that object-oriented
design hopes to solve."

---But, most of those 'nightmares' can be avoided by design.
   Object-oriented design will create different nightmares.

                   *           *           *

Based on what AH wrote, I conclude that he is 'Wrong People' when
it comes to helping to improve design methods for business
applications. This is not to say that AH is not highly-qualified in
some areas other than business applications.

Based on what I have read about OO, I conclude that it is
totally unnecessary, and will only end up making things worse
than they already are.

The Great Programming Contest came into being in order to
let people prove their points. I dearly hope that someone
will enter using an OO approach to prove that it has merit.

Anyway, that's what I think.

Thanks

Tony Dilworth

Yes, I agree.

However, it is a major jump from there to the idea that the entire
methodology of OO is invalid, or even that it is unsuitable for business
use.

If you look at PowerCobol, for example, you will see all the same problems
that you get with trying to write business apps in Basic. The reason is
simple.  They implemented PowerCobol using null terminated strings for data,
or single sized arguments that effectively disable all the data checking
that Oops uses. In fact, the code that PowerCobol creates is not OOP Cobol
(!!!). It attempts to write pseudo OOP code using standard subroutines, and
fails rather badly ... in my opinion because it attempts to write C++ in
standard Cobol rather than generating OOP Cobol.

I think that if you play around with a true OOP Cobol, and ignore all the
crap coming from the academic world, you will find that it not only has a
viable place in our methodology, but solves a rather large number of
maintenance problems.

More complex when the rest of the transactuion is processed: The trans has

runs out of twenties) Sometimes the ATM only knows something went wrong, and
returns a "best guess" at what really happened.

Only after everything is good does the ATM tell the main database to debit
the money from the account. And even then, it's still only a "memo post"
until the big batch run .... but thats a whole nother tale:)

But the point is that while a few newer ATMs have Client-like capabilities
(and Yup there are PCs inside them) for the most part the logic is still
really "dumb terminal" stuff for the most part. Most of the data only flows
*from* the ATM. What is returned is mostly just commands to tell the ATM
what to do. About all that gets sent back as data is transaction ID,
transaction amount, and, if used, account balances.
