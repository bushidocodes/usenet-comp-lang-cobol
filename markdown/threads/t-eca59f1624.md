When you have

id division.
program-id. someName.
...
procedure division ...
ENTRY "entry1" ...
...
ENTRY "entry2" ...
...
end program someName.

then you can
CALLl "someName"
statically IF you have linked the object of someName together with the object of
the calling program.
This same CALL can be dynamic, if someName was linked separately and when the load
module has the same name.
When you
CALL "entry1" (or other entries"
and you linked together as said before, the linker can resolve the entrypoint.
But when you do that call dynamic and did NOT link someName to the caller, but
separately, than the operating system searches for a separately linked load module
with the name "Entry1". That it does NOT find of course.!

Hope this helps?
The Cobol Frog

The entry points have to be externally defined so they can be
referenced by the loader. Haven't done this in a while, but you used
to have to establish alternate entry points via the linkage editor.

What you probably need to do is add ALIAS control cards to the linkage
editor SYSIN.

ALIAS ENTRY1
ALIAS ENTRY2
ALIAS ENTRY3
NAME PROG1(R)


There also used to be utilites like the TSO PDS command that could
build alias entries on the fly once the module had been link edited.

hope this helps

Scott Peterson

Hi Bruce,

You need code an 'ALIAS'-Statement for every
alternate entry point of your module when you
linkedit it. You'll find then entries for every
name in the directory of your loadlibrary, but
every entry points to the same 'TTR' and has the
same size (it is pysically the same module!).

But pay attention when you copy the load-module
into another library (e.g. production system!).
Don't forget to copy all of the 'ALIAS' too.

Cheers

Daniel
