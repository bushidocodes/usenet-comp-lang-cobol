The answer to this question may depend on the compiler implementation.

A direct move of ZEROS to a signed PIC could result in either an _unsigned_
zero value or a _positive_ zero value (presumably so that a test of the
resultant field for NEGATIVE would not yield a TRUE value.)

I've experienced COBOL compilers where adding a positive quantity to a
negative quantity with an identical absolute value would result in a
_negative_ signed zero amount (where a test for NEGATIVE would yield a TRUE
value even though the absolute value was zero.)

There can also be difficulties encountered in moving ZEROS to a group item
which contains subordinate signed numeric PICs - I have experienced this in
Tandem COBOL compilers.

Dennis J. Minette

The COBOL standard defines the sign condition as

                         {POSITIVE}
   expression-1 IS [NOT] {ZERO    }
                         {NEGATIVE}

which should give it's result depending on the numeral value, either > 0, = 0
or < 0, independant from the stored sign-bit-pattern when value = 0. So this
condition tests for *absolute* place on the "number line" (below, on or above
the origin).

The standard defines part of the relation condition as

   expression-1 IS [NOT] {EQUAL} expression-2 (expr-2 may be ZERO)
                         {=    }

which should give result of comparing TWO values. So this condition test for
*relative* place on the "number line".

Rï¿½sumï¿½:
IF VAR = ZERO
IF VAR IS = ZERO
IF VAR IS EQUAL TO ZERO
are all relational tests
IF VAR ZERO
IF VAR IS ZERO
are both sign tests

All five (should) return the same result.

I wonder if: When someone sees IF VAR IS ZERO i.s.o. IF VAR = ZERO wheter that
person realises these are syntactically and semantically different types of
conditions.

COBOL.FROG
