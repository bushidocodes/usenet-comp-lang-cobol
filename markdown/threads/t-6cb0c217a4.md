```
Others are replying to parts of this post, but I thought that I would give
some "general guidance" related to data-validation of numeric input fields.
(This assumes that the input is usage DISPLAY and the field is defined EITHER
as "numeric" - with V's and S's or numeric-edited - with periods, commas, and
+/-. If the input is coming in from a terminal or similar input device, I
hope - for your sake - that it is numeric-edited, but you never know!)

There are (at least) two approaches to working with such fields during "data
validation" phases (where you can't really BELIEVE that they will only have
numeric data in them).

If the input comes in from same place (like a terminal with a number lock)
that only allows numbers - but MIGHT have a field with all spaces, then look
at using the BLANK WHEN ZERO phrase on your field definition.  This means
that you won't get any "invalid" data messages - and that your program can
handle such fields as "zero" when moving FROM that field.

On the other hand, (and I am afraid that this is more common in "real-world"
cases of data validation phases),  if your input field MIGHT have a space in
the middle of a bunch of numbers - or worse "alphanumeric" and/or unprintable
characters in it, then it is best to have a redefinition of the field as
alphanumeric (PIC (X)) and do an "if numeric" test first, then an "if spaces"
test, and only after these pass should you reference the numeric definition
of the field (and do any range or sign tests).

It is also important (IMHO) that you NOT combine these tests into a single
conditional expression (lots of ANDs and ORs) - but instead do the tests one
at a time - either with nested IFs (or evaluates) or with a "OK-til-now"
switch and sequential IFs.

Clearly, "in the real world," you will find different "reliable" and
"unreliable" types of input data.  What type of data validation you must do
and/or CAN do - will depend on your actual environment.  The only "rule of
thumb" that I think is almost universal is do NOT believe the user when they
say "Well, I never SHOULD enter data like that".

P.S.  For the more "advanced" users, you might want to check out the VALIDATE
verb/facility that is being added in the draft of the next Standard.  This
does lots and LOTS of this "uninteresting" stuff in a very nice (IMHO) way.
```

```
To declare the variable as PIC Xs and
REDEFINE it as PIC 9s.
Some COBOL compiler read PIC 9s with spaces as ZERO.
In the validation process if the value is NUMERIC,
PERFROM the NORMAL-RTN, ELSE PERFORM the ERR-RTN.
In the NORMAL-RTN, use the variable name with PIC 9s.
In the ERR-RTN, use the variable name with PIC Xs.
```
