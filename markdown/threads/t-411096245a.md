The only place Case has an effect is in literals ... for
example: MOVE "hello" TO DATA-NAME.  Within the
code itself, Cobol considers upper and lower case identical.
I enter my code in uppercase, but that is more a matter
of me being a traditionalist than anything.

No, it does not.  However, Fujitsu (and this is compiler specific) is
particularly fussy about the column 8 and column 12 margins.  If
you use a tab in the first character of the line to get over to column 8,
rather than 8 spaces, Fujitsu Cobol can get rather excited about it.
It you see messages refering to "column A" and "column B", then
that is probably your problem.

>
> Ok, kidding aside, is 'project' similar to 'build all' in C? Example,
>in C, I can just build all instead of compiling, and linking. Does
>project in Cobol do the same thing?
>

Yes, the Fujitsu "build all" is a recompile and re-link of everything
in the project.  A project, though, can contain more than one
program.  Think of it as the make file for a C project with multiple
run modules.  In fact, it does exactly that behind the scenes. First
it deletes all the "exe", "obj", and "lib" files, then it creates a make
file (on your disk as "001", "002", etc.), then it runs MS make.

CICS is an entire system under which COBOL (and other languages can
run).  There is no direct interface to the user or data within the
programs that run under CICS - these are handled using CALLS to CICS
specific modules that handle the management of these functions.  In
order to make the calls understandable - the programmer codes "Command
Level CICS" statements in their programs - these are converted via the
translator program to "Call" statements and the translated program is
compiled and linked.  Any language that runs on an IBM mainframe can
make use of CICS command level statements - these include C, Pascal,
BAL, COBOL, PL/1 and perhaps (don't quote me) RPG.

As a CICS SYSPROG of ten years experience I would appreciate the chance
to offer a few comments.

CICS is a mini operating system. My experience suggests that there are
subtle changes in the approach required to COBOL coding for CICS that
are not always initially appreciated by an experienced batch programmer.
That a CICS system is always attempting to spread machine resources as
wide as possible made dabbling in Visual Basic (for example) easier once
I grasped that the fundamental concepts beneath each system were very
similar. Batch programming (and I don't want to appear to be negatively
critical here - I've done both and neither is 'better' than the other)
doesn't require the programmer to be too concerned about other users
with whom they are sharing the machine.

JCL, the mechanism by which work is presented to the machine, is... (I
tried here to think of a suitable word here, and failed) very difficult
to learn. That said, an application programmer who can use JCL is one
who makes my life (and ultimately their own) easier because they're not
waiting for me to build them some compile JCL or a file copy (You can
never know too many languages - the more you know the easier it is to
learn another).

Incidentally. Is JCL a langauge? I reckon that strictly speaking it
isn't, but I'm not sure why. Can anyone confirm my prejudice?

It would be a pretty sad excuse for a scripting language that *didn't* let
you execute programs and utilities.  The question is how much control you
have over those programs.  If JCL isn't wimp, educate me by showing how
you would write the job below.

Specifications:

1)  This job has one parameter:  The name of a company.
2)  There are up to three input files.  The first node of each file is
INA, INB, or INC, the second node of the file name is the company name,
and the third node is today's date.
3)  If INA/company/date is present run PROGA which produces
OUTA/company/date.  Likewise INB->PROGB->OUTB and INC->PROGC->OUTC.
4)  Run the three programs concurrently.
5)  If more than one output file is produced, run PROGM to
create MERGED/company/date.  This program will correctly detect if only
two or all three files are present, so it is run only one way.
6)  If exactly one output file was produced, run PROGX with that file for
input.  If more than one output file is produced, run PROGX agaisnt the
MERGED file.
7)  Feel free to use "." instead of "/" in filenames.

Unisys WFL:

BEGIN JOB UP/TO/3/FILES (STRING COMPANY);
STRING COMPDATE, MFILE;
INTEGER I;

BEGIN SUBROUTINE PASS1;
    IF  FILE INA/#COMPDATE IS RESIDENT THEN
        PROCESS RUN PROGA;
            FILE IN  = INA/#COMPDATE;
            FILE OUT = OUTA/#COMPDATE;
    IF  FILE INB/#COMPDATE IS RESIDENT THEN
        PROCESS RUN PROGB;
            FILE IN  = INB/#COMPDATE;
            FILE OUT = OUTB/#COMPDATE;
    IF  FILE INC/#COMPDATE IS RESIDENT THEN
        PROCESS RUN PROGC;
            FILE IN  = INC/#COMPDATE;
            FILE OUT = OUTC/#COMPDATE;
END PASS1;

COMPDATE := COMPANY/TODAYSDATE(YYYYMMDD);
PASS1;
I:=0
IF  FILE OUTA/#COMPDATE IS RESIDENT THEN 
    BEGIN
    I:=I+1;
    MFILE:= OUTA/#COMPDATE;
    END;
IF  FILE OUTB/#COMPDATE IS RESIDENT THEN 
    BEGIN
    I:=I+1;
    MFILE:= OUTB/#COMPDATE;
    END;
IF  FILE OUTC/#COMPDATE IS RESIDENT THEN 
    BEGIN
    I:=I+1;
    MFILE:= OUTC/#COMPDATE;
    END;
IF I > 1 THEN 
    BEGIN
    MFILE:= MERGED/#COMPDATE;
    RUN PROGM;
        FILE INA  = OUTA/#COMPDATE;
        FILE INB  = OUTB/#COMPDATE;
        FILE INC  = OUTC/#COMPDATE;
        FILE OUT  = #MFILE;

    END;
IF I > 0 THEN
    RUN PROGX
        FILE IN = #MFILE;
END JOB;        

I just wrote that off the top of my head.  There are probably a couple
syntax errors, but the functionality is all available.  RUN causes WFL to
wait for it the program to complete.  PROCESS RUN causes the program to
run simultaneous with WFL.  I'm using the subroutine PASS1, because a
subroutine never exits until all the programs it started have completed.
If you want a more complex multi tasking job, you can assign task
variables to all the tasks, and test their status.

I have read of some MVS installations that actually do use batch REXX
for controlling file allocation and program sequencing logic in their
batch environment.  However, there are some potential drawbacks of
replacing JCL logic with REXX logic:
(1) Allocation of files in REXX (which uses SVC99 for dynamic allocation
under the covers) fails if the file is enqueued by some other process
rather than waiting until the file is available, as happens with a JCL
reference.  This may cause a REXX implementation to fail requiring
complex restart scenariors while a JCL implementation might be only be
delayed for a time before completing correctly.
(2) SVC99 does not support UNIT=AFF allocations, so there is no easy way
to allocate a dataset which is a concatenation of tape datasets without
requiring a physical tape drive for each of the datasets, even though
there is no logical reason they could not share a tape drive.  Also
distinct tape files required during a single program exeuction cannot
share a physical tape drive, even if at most one will be OPEN at the
same time.
(3) Batch job streams will occasionally fail for all sorts of reasons,
some of which may be difficult to predict in advance.  In JCL each
execution of a main program constitutes a jobstep, which is a "natural"
restart point for a jobs that fails somewhere in midstream.  JCL
structure is sufficiently "simple" that a number of products are around
to support typical step restart situations (CA-11, ZEBB, etc.).  The
additional flexibility of REXX also means that if you want to support
restart in a complex sequence of executions implemented as a REXX
routine, you must anticipate and custom design that restart support into
the REXX exec itself.

It should certainly be technically feasible to enhance SVC99 support to
counter deficiencies (1) and (2), provided some consensus is reached as
to a reasonable convention, but these limitations have been around for a
long time.  I don't see any "easy" way to address (3), and for the kinds
of nontrivial batch processes that abound in large main frame shops, you
really need ability to restart as close to the point of failure as
possible, both to avoid wasting resources and to keep job execution
within what may be a narrow batch time window.

The operating system should be smart enough?  Mainframes are capable of
dynamic allocation (SVC99), it's just rarely used, and you would have to do
it from a program written in assembler (or use COBOL options or commands
I've never heard of).  The only utility on our mainframe that uses SVC99
regularly is plain vanilla FTP.  Running that utility in the same job with
other programs that use MVS's normal file allocation is a major pain in the
butt (it has to do with generation data sets; running an FTP step against a
+1 generation, you have to ask for the +0 generation; restarts are worse).

The main difference between dynamic and requested allocation is that in
dynamic allocation, if the system doesn't have enough space, it will write
and write data until it does run out of space, and then blow up.

As an application programmer, it's possible to have a pretty good idea of
the size of the data being working with.  In requested allocation, you enter
this amount, and then provide a second amount to allow for future expansion
(that's simplifying matters).
It is true that under this system you can blow up at both ends (not having
enough space to answer an allocation request; or not having an allocation
that is big enough), at least when you don't have enough space to answer the
allocation, it blows up right away; instead of very likely running for hours
and then blowing up when it runs out of disk space.

The requested allocation system is not intuitive, and actually can be a pain
in and of itself.  But our Unix system servers experience space problems
just like our mainframe.

The real answer is a good storage administrator with good tools and enough
authority to dictate what happens on the harddrives (not always the case).
