It's not easy.  For the DAT files something like ParseRat (http://www.parserat.com/) 
will help in the trial and error process (use the "binary" settings) - you can usually 
see patterns which help you establish record lengths and header size (if any).  Then it 
will help you identify field sizes the same way and show you what each field would 
contain in various formats (e.g. packed, binary, etc).  Once you've got that done, it 
will extract the data to any one of a variety of "importable" formats for other 
packages.

You'll need SOME idea of what  are valid values for numeric data.

IDX files will be indices and you can re-generate them once you have the data extracted 
into some other system.

Mike,

I'll throw this out as a "give it a try" option only.  If I can recall
correctly, the MicroFocus Isam .dat file had the first 128 bytes as the Isam
info record.  As to what the layout of the fields in that 128 bytes is, I'm
not really sure, but open the .dat file in Wordpad or Notepad and see if you
see something that looks like a date (yymmdd) starting in position 9 of the
first record.  I seem to recall that this was somewhat standard in Micro
Focus Isam files, but I haven't used Micro Focus for a number of years now.
If so, then look at position 129 for a couple of bytes of weird looking
characters, i.e. "B�",  "E?", or whatever.  Then continue scanning down the
file, counting each byte until you see the same characters repeat
themselves.  This will give you the actual data "record" length, including
the "weird" characters.  Assuming you have a compiler that can read byte
stream files, i.e. 128 bytes first, then <whatever length counted above>
next, you can possibly create a new file that is more user friendly to use.
In this case, the .idx file doesn't really matter, since it's just pointing
you to the data records in the .dat file anyhow.  However, you still won't
know the contents of each field in the record.  But I have found if the old
application still runs and you can produce the output from it, you can
usually figure out what data is where, give or take a couple of bad guesses.
However, if the data is in comp/binary format, the job becomes, shall we
say, a bit more challenging, but still doable!

Good luck,

Denny

that
they allow you to ship with an application, so you should be able to find
a copy.
