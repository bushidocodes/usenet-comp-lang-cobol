Abend-aid is a formatted dump listing with diagnostics. The blurb at the
front of the dump will inform you of the locations within the program of
the instruction causing the abend and of the two items of data involved.
You take the address of the abending instruction and look up the table
of verbs and hexlocations in the compile listing (make sure it is the
correct version) for the instruction with the highest hexlocation less
than that provided by abend-aid. This will give you a line number to the
instruction in the listing. Look up and note the instruction and the two
data items involved. Now find the data items in the dump and note their
contents. That is how to find what data and instruction is causing the
problem but as to what you do to fix it and how it got to be full of
duff data is something else.

>  Do I need to compile the program using certain
>options? 

No.

> Is there a book/manual on ABEND-AID that I can check out?

Probably, but I've never seen it. If your site has bookmanager then you
may have the manual online. BTW, I learned all about dumps the hard way
with S/370 assembler, so abend-aid was a doddle to pick up. I guess that
to a newbie even a formatted dump is daunting but your seniors should be
able to provide assistance and guidance. Good luck.

I think the previous replies may have been slightly misleading depending
on the Abend-Aid set-up in your shop. On any of the sites I have worked
in, I have only resorted to reading a dump by offsets when there were
problems with Abend-Aid.

At compile time, you need an extra compile step to add an entry to a
DDIO file. This will contain the source details that Abend-Aid requires
along with the offsets. AFAIK, this is also used by the Expediter tool.
There is obviously an overhead to having this additional step in
compiles, so many shops do not have this as a default compile option.

When your program abends, the dump needs to be sent to Abend-Aid. From
what I've seen, if the dump appears in your job, then Abend-Aid isn't
picking it up. A job is generally run at IPL time to tell Abend-Aid
which abends (by job name) to intercept, although this job can be run at
any time.

After the abend has occurred, go into Abend-aid from ISPF. In the entry
panel, the report dataset should refer to the dump dataset, and the
source(s) should point to the DDIO(s) from compile time. When you select
the failed job, Abend-aid will match the dump to the source (if
available).

From the 'output selection menu', I choose option 4 which usually isn't
listed as an option, and scroll to see the line of COBOL source, and the
field in error along with the bad data it contains (if an S0C7). Other
options let you check-out the contents of working storage fields etc.

If Abend-Aid complains that it can't find the executed source in the
DDIO file, it may ask whether it should use the most recent source or
let you select from a list, but this can give spurious results. If it
can't find a match, then you need to start looking at offsets as per the
previous posts.

If you have a system application support area in your shop, they will
probably have more details on how they've set things up.

Mike.

At all the mainframe shops I've worked at, as long as you have if DD
SYSUDUMP in your job step that's where abend-aid's report goes (I just use
SYSOUT=*); but I do believe one or more of the CPU high priests and/or
acolytes need to set something up that says, "when a program abends, start
up Abend-Aid".

As far as compiling options: none are required, but....with your S0c7 (data
exception error), Abend Aid will report the BLW= and the offset of the data
and this data is available only if you get a compiler output list. (Which I
believe everyone does anyway).

You will also want to compile with "XREF(FULL)" so you get the
cross-reference of source code line number to code location; Abend-aid
reports the next sequential instruction at the time of the abend, so if you
have this listing you will be able to tell on which source code line the
program abended. There is some compiler option that will cause the O/S to
report this, too, but IIRC there's a real bad resource penalty for compiling
programs like this.

One other thing: check out www.mvshelp.com if you are doing mainframe work.
Very nice site.
