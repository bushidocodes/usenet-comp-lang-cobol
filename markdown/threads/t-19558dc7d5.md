Based upon what I remember from Bill's earlier posts
on the question, the standard does not specify to which
of two WRITE verbs a single END-WRITE should be
associated.

Most, if not all compilers, use the nearest association;
but the question deserves consideration because the
indentation shows the programmer's intent to associate
the END-WRITE with the first WRITE verb. It is because
the compiler used the nearest association, that the
syntax error occurred.

As to the nested WRITEs working with any other compiler,
one cannot say that with certainty unless ANS 85 flagging
is active. The statements work with Micro Focus also;
but with ANS 85 flagging turned on, the compiler reports
the syntax error.

Although it may sound confusing, the syntax error with
the Fujitsu compiler and messages from ANS 85
flagging can be solved by adding another END-WRITE
scope delimiter. This is because the compiler associates
each END-WRITE to the nearest WRITE statement causing
that nearest WRITE statement, which is imperative, to be
treated as a delimited-scope statement, which is implicity
imperative. This causes the outer WRITE statement to be
treated as conditional, eventhough the programmer
intended the outer WRITE statement to be treated as a
delimited-scope statement and therefore imperative.
Adding a second END-WRITE causes both WRITE
statements to be treated as delimited-scope and implicitly
imperative, which satisfies the requirements for the
enclosing in-line PERFORM which requires an imperative
statement.

In other words, if the compiler reports that an imperative
statement is needed, add an appropriate scope delimiter
or move code as Ed Milne suggested.

The exception is that, if there are no statements, add a
CONTINUE statement. For example, an inline PERFORM,
INVALID KEY, AT END, etc. with no statements still
requires an imperative statement under ANS 85 rules and
that imperative statement may be CONTINUE.
------------------
Rick Smith

Such that scope terminators are guaranteed to terminate
any outstanding, unterminated conditional statements of the
same name. This most frequently reflects (and may always
reflect) the programmer's intent and prevents the introduction
of syntax errors when removing or replacing unrelated
imperative statements.

Consider the following (draft statndard) code snippet.

build-indexed-file.
    set no-more-records to false
    perform until no-more-records
        read input-file
          at end
            set no-more-records to true
          not at end
            perform build-indexed-record
            write indexed-record
              invalid key
                *> format error record
                write error-record
                initialize error-record
            end-write
            *> possible additional code
        end-read
    end-perform
    .
Moving 'initialize error-record' to the beginning of
the code for the 'invalid key' option is a reasonable
change (and likely should have been placed there
initially). Doing so introduces a syntax error at the
'end-read' because the 'end-write', under the current
draft, terminates the 'write error-record' instead of
terminating the 'write indexed-record' as originally
written. If the standard guaranteed that scope
terminators would terminate outstanding, unterminated
conditional statements of the same name, the change
in the code would not introduce a syntax error.

Furthermore, the presence of 'possible additional code'
introduces an element of surprise since the syntax
error cannot be reported until the 'end-read'. This
surprise is that, given 5 lines of additional code, the
syntax error occurred because of a code change some
7 lines before that point at which the syntax error is
reported and the 'initialize' was not part of the 'write'.

To make matters worse, many implementors, as an
extension, treat unterminated delimited scope
statements as if the statements were terminated at
the point where the syntax error would occur under the
standard. In the example, above, with the code change
and the presence of additional code after the 'end-write',
the logic of the program will change and the programmer
may not be aware of the change until testing. The logic
change occurs because the compiler, essentially, moves
the additional statements into the 'invalid key' option of
the 'write indexed-record' when the programmer and
original logic placed the statements within the 'not at end'
option for the 'read'.

I can think of no way that introducing this change into
the new COBOL standard will 'break" any existing code
that has been checked with ANS 85 flagging active.

It could "break" existing code that was not checked
with ANS 85 flagging.
------------------
Rick Smith
