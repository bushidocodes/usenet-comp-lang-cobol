 For this message, open source has the meaning defined by the Open Source
Initiative (http://www.opensource.org/osd.html).

Everyone seems to take Linux as the model for an open source project but this
may not be appropriate for the COBOL community.  Linux is governed by the GNU
General Public License (GPL http://www.opensource.org/gpl-license.html).  GPL
requires that people modifying the code must release their modifications under
the same conditions as the original code.  The problem is that the typical
COBOL project includes an organization's policies and practices.  Few
organizations would want to make this type of information public.

However, we build COBOL programs from components that could easily be made
public.  Items that we could share include:
1) templates - model programs for various situations.  It is already standard
practice to look for a similar program that we can modify to create a new
program.
2) common routines that we can call.  Why should we be wasting our time
reinventing wheels?
3) concepts - non-code items like style and technique


Should we

I know that some people have a philosophical aversion to the idea of open
source software.  It smacks of communism or reduces our opportunities to make
a living.

I do not accept any of this.  For me, it is a chance to learn from others,
reduce the time spent doing grunge work and increase the time doing the more
interesting work.  I do not think that it would reduce the money spent on
software development.  It would just increase the quality of the software.


Could we

The Internet provides a natural way to distribute open source software.
However, many issues would have to be resolved to make a library of open
source software practical.  Here are a few of the issues that I have thought
of.

1. Who's in charge?

Do we need anyone to control such an effort?  If so, how do we select them and
what are their powers?

The library would need volunteers to create and maintain it.  The spirit of
open source would allow multiple parallel development of a library.  If one
implementation proves to be much more popular and successful then the others,
it becomes the standard.  So it's not necessary to have someone making
decisions.  It's more a matter of the community reaching a consensus by its
use of the facility (voting with our feet as it were).

Similarly, I feel that the library should accept any submission with the sole
limitation that the software code works under all conditions.  Anything that
doesn't meet this condition gets pulled from the library.

2. Quality

If I were using such a library, I would want to be able to select the best
example of any type of item.  How could the library identify the "best of
breed"?

Although this is highly desirable, I don't think that it could be done for
two basic reasons: - It's very difficult to get even two programmers to agree
on exactly what constitutes the best way of doing anything.  Getting the
entire community to agree would be impossible. - One of open source's great
strength is the speed at which development takes place.  If someone points
out a flaw in some code, it could be corrected by return e-mail.  Any quality
classification of the library entries would always be out of date.  Trying to
enforce such a classification would only slow things down. Unfortunately,
this looks like one feature that we would have to do without.

3. Language

In my opinion, an open source library should be equally available to anyone
on planet Earth.  This is a special problem for COBOL because its syntax is
based on the English language rather than mathematics.	How could we create a
library that a Japanese speaking programmer could just as easily contribute
to or use as an anglophone?  Should we care?


Would we

Would you want to contribute to such a library?

Stroking their ego seems to be the basic motivation of programmers who
contribute to other projects such as Linux.  I don't see why COBOL programmers
should be any different.  Lots of us think of ourselves as world-class
programmers.  Having the world acknowledge the fact by adopting your code as a
standard would feel very, very nice.


Personally, I feel that this is an idea whose time has come.  Your comments
are invited. -- Ed Milne
