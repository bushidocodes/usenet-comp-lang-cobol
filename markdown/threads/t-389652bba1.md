```
By "VS COBOL" I assume that you mean either the OS/VS COBOL compiler or the
DOS/VS COBOL compiler.  The former has been out of support for several years
and the other one is dying next year.

If by "porting" you mean to Fujitsu, Micro Focus, or some other PC compiler
that provides compatibility with the IBM (dead) compiler language, then you
could do this, but is seems pretty short-sited to me.

I should say, however, that if you have mainframe code that you are "porting
to the PC" for production work and the current version is written in OS/VS
COBOL or DOS/VS COBOL, then, YES it makes sense to do the porting first and
then do the language conversion 2nd.  In other words, do one change at a
time.  Get your mainframe programming working completely on the PC (server
or workstation) using as much of the existing language (even if it is OS/VS
COBOL) and THEN convert it to a more modern dialect.
```

```
would be difficult IF you are using any of the built-in AcuCOBOL extensions.

not certain.
```
