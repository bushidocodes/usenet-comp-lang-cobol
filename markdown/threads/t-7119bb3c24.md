For my twenty cents worth: This is probably due to the fact that you
are reading a member that does not exist at the level you are accessing
it. With SCLM you may have written a process that handles the migration
from Development to test to production. This process involves multiple
libraries, when migrating you do not see the fact that you are really
dealing with not one unified library but a concatenation of three. Have
you actually chosen the right level to access?

If the source is at production and you change a copybook the program
will still compile and create a development load module. If you are
relying on the source existing at development for every load module then
this is not the case, it hasn't changed so it still sits in Production
(look at the library level in edit or browse).

That is my guess to a vague question. Good luck....

File Status 35 is returned when an empty file is OPENed INPUT, I-O, or
EXTEND. We had this occur just the other day on my system (VSE/ESA,
COBOL II, VSAM). You can code this file as "optional" by way of the
select statement: "SELECT OPTIONAL filename". With your program coded
this way, a KSDS or RRDS file will return a file status of 05, while an
ESDS will still return a file status of 00. Look in you IBM COBOL II
manuals for OPTIONAL FILES for more information.
