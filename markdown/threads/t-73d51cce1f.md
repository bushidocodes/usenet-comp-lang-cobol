Well I don't know much detail on STL.  From the C/C++ Users Journal I know
it adds very powerful capabilities to C++ but sometimes at the expense of
readability. Yeah I know it had to believe you can make C++ ever harder to
read.

STL is a set of template classes that implement many  algorithms and data
structures. It lets you write functions that work with any data type. It
implements iterators, algorithms, deques, vectors, lists, maps, and
multimaps, sets, and multisets; searching and sorting algorithms, function
objects etc. yet another book/topic I have not yet got around to reading
about, but I can excerpt from the table of contents :-).

It is related to the concept of reuseable software components. It
originated in 1979 with Alexander Stepanov and later with David Musser. A
lot of the work was done at Hewlett-Packard.

Someday I hope to take topics like this and other interesting books and
translate the ideas into COBOL and perhaps make the programs available on a
web site. All it takes is time. It might not be kosher to write my own book
on the same topic only using COBOL, although I have seen data structure
books that seemed to follow Knuth's books to a great extent. Taking the
same topics and making them attractive to programmers in another language
would probably not be objectionable as it can only increase the possible
readership range for those books. This is something I could do after
retirement and would be a possible justification for buying a PC based
COBOL compiler. It would be fun to program what I want and am interested in
for a change, instead of what others want.

The STL is the only way to do some things, but it takes a hit
(as does all templating) from it being a pre-compile operation.
Handy as it is, compilers can only report on error produced
after the template is applied, thus obfuscating any defects in
the template itself.

The theoretical improvement on this idea is the "generic
class."  Where templates use text replacement, generic classes
will use inheritance or polymorphism to achieve the same
result -- all while maintaining a higher degree of type
enforcement.  I can't say that I've ever used a development
environment that has a sufficiently powerful generic class
mechanism to make STL obsolete, though.

There is an ongoing debate about introducing generic classes
into dotnet.  If this happens, it's entirely likely that there
will be a COBOL that supports them.

OK Charles, let's pick up from there, but it's not quite specific enough for me
at the moment.

Take as a starting point Smalltalk, (If you want to be particularly correct, go
back to Simula). The WHOLE language is based on objects - what is it in C :-

        printf (string) or something......

Same applies with all the other OO languages, so they have to provide
classes/methods to handle any eventuality. There is nothing equivalent to our
pic zzz,zzz.99. You get there by finagling. using the various string support
classes.

Ignoring GUIs/Web etc... for the moment, we already have a RICH complement in
standard COBOL to do both numeric and text handling, (STRING, UNSTRING,  MOVE,
Reference Modification. etc.).

Donald and Pete will chip in if they disagree - but quite a slice of OO COBOL
can be done using conventional pic 9 or pic x - so for the most part, we don't
require OO string handling techniques. Even in preparing lists
(collecitons/dictionaries) the elements in the collections can be expressed as
intrinsics (pic 9 or pic x).

Not one tackled as yet in the proposed OO COBOL Standard, but you *do* require
to get intrinsics into objects for Windows etc. displaying.  F/J and Hitachi -
I'm not aware of what they use, but rather like Smalltalk, Net Express does
provide a set of GUI classes, which spring off from the main branch at a class
called Dependent; this acts as the immediate "super" class for anything to do
with GUIs.

(Recently a suggestion was made by someone, about incorporating GUIs, the J4
folks gave it a rating of  4 out of 60 - so it's as dead as a dodo. OK Dashwood
- no laughter - It *wasn't* ME <G>).

So if you are looking for C functions that *might* be useful with COBOL,
exclude any where you see an obvious 'cousin' already present in our pic 9 or
pic x handling.

To get a bit of background on class structures and how they compare to what you
are looking at with those STLs in the C language, two possibilities.

- Apparently there is a University Edition of Net Express V 4.0, just released.
I don't have V 4.0 - (money !) - so I'm not sure what has been added. Trouble
is with the PC compilers, so much has been added over the years as extensions,
that when we come to look at something "new" based on COBOL 2002 - we don't
recognize it as new because we've been using it for years ! Check with Micro
Focus - you *may* qualify as an interested party to get a copy of the UE. The
old UE version sold for about $100 - don't know what their price would be for V
4. The UE is a learning tool, comprehensive but with a 'lock' so it can't be
used to make a buck on the side.

- The other alternative is to do some class hierarchy checking, if you can
borrow some decent books from an academic tech library. I was going to give you
a reference to some diagrams on class hierarchies at

        http://www.cobolportal.com/j4/index.asp.

but unfortunately when I check the J4 site, doesn't look like 2001 documents
are still on-line.

Back to structuring of class libraries, there is a huge emphasis on
REUSABILITY, sub-classes making constant reference to super support classes. I
*could* change the program to reflect intrinsics, but I'm not going to <G>. The
following extract, based on something I recently posted for Thane is creating
elements as objects in a collection so they can be displayed in a Listbox.

*>---------------------------------------------------------
Method-id. "makeListRecord".
*>----------------------------------------------------------
Local-storage section.
01 ls-length                     pic x(4) comp-5.
Linkage section.
01 lnk-string                    object reference.

Procedure division returning lnk-string.

   if TypeTabbed
       move x'09' to Tab1 of ws-MatListRecord
                     Tab2 of ws-MatListRecord
                     Tab3 of ws-MatListRecord
   End-if

   move length of Name-record to ls-length

   Evaluate true
     when ByName
       move CORR ws-MatListRecord to Name-record
       invoke CharacterArray "withLengthValue" using
          ls-length, Name-record returning lnk-string
     when ByKey
       move CORR ws-MatListRecord to Key-record
       invoke CharacterArray "withLengthValue" using
          ls-length, Key-record returning lnk-string
     when other
       move CORR ws-MatListRecord to Other-record
       invoke CharacterArray "withLengthValue" using
          ls-length, Other-record returning lnk-string
   End-evaluate

   if    Mat-ID of ws-MatListRecord > ws-LastNumber
         move Mat-Id of ws-MatListRecord to ws-LastNumber
   end-if

End Method "makeListRecord".
*>----------------------------------------------------------

No big deal about the above method, just traditional COBOL syntax wrapped
inside a method. It only gets different when we get to the lines :-

       invoke CharacterArray "withLengthValue" using
          ls-length, { Name/Key/Other-record } returning lnk-string

To see what's happening I've set the Animator to pause on "invoke
CharacterArray", so I can see which support classes are called. Not something I
have found a need for but there is a TRACE list that gives me an audit trail of
cllasses/methods invoked. As I've only used TRACE very rarely, I've stepped
through the following manually :-

Class            method
-----            ------------
Charray            withLengthValue
Charray             new
Behavior            new
Collectn            initialize
Base                 malloc
Charray            setValue

The above sequence occurs for each element being added to the collection. It
can get much more complicated than that, iterating through some of the support
classes repetitively, until you go dizzy ! Given the power of modern computers
the above of course occurs in a flash.

As you can see from the table above it does emphasize the concept of
ENCAPSULATION. As Pete has stressed you have 'got to take components on faith',
i.e., you have to accept that the third parties that wrote the programs have
done their homework and thoroughly tested them. This is largely achieved by
keeping individual methods minimal in source size so that the function they
perform is simple and easily proven.

You don't need to see the internal combustion engine - and in the above example
all we are interested in  :-

(a) doing the initial invoke with the correct parameters
(b) lots of 'magic' going on between whiles
(c) we get back the return object from our invoke

J4 are working on Standard classes at the moment. I'm not "in the know" so I
don't know any more than you. If you want to check out, go to the URL and
searach out Thane's name along with the topic 'Standard Classes". At this stage
it would appear they are concentrating on the lowest level in the pyramid,
collections. Having straightened that away, then they will have to look at
"super" classes above, bringing together common routines, building the pyramid
with a series of layers where commonality occurs.

"STL is a set of template classes that implement many  algorithms == ??? and
data
structures.

It lets you write functions that work with any data type == will have to take
account of data types and COBOL 2002 already does..

It implements iterators == N/E certainly allows callbacks with a "do" method

algorithms== ??  deques == ?? vectors== ??

lists, maps, and multimaps, sets, and multisets == depends upon the 'prime'
classes that J4 determines. Lists = collections and your reference to maps,; is
that the same as Maps in Java ? These are called Dictionaries in Smalltalk/Net
Express, and tentatively referred to by J4 as 'Keyed Collections"

searching = Keyed Collections*should* allow for this and another approach is
iterating a callback with a "do" method

sorting algorithms == a default sort, ascending, and currently N/E allows you
to invoke your own SortMethod, the simples being "reverse" gives you descending

function objects = ????

Each language tends to have its own terminology, hence my question marks
against some of your items.Now if you want to zero in and get more specific,
such as "C++ has this - do we have it in OO COBOL etc. .....?"

Suppose you wanted to write a sort routine. In PASCAL the typing is very
strong and includes the array size as part of the type. You could sort an
array of 100 integers but to sort a different size array or an array
containing a different type you would have to code another version of the
sort. C/C++ relaxes this somewhat as the array size is not part of the
type. Still your sort would expect a pointer to an array of one particular
type. STL would let you specify a generic type at compile time and at run
time your sort could work on int, char, short, double, or user defined
types etc.. This is one of the basic ideas behind the STL.

iterators - a generalization of an object pointer. You can declare new
classes and override many of the operators with new meanings. You use an
iterator to access the elements of an ordered sequence, Access means either
storing or obtaining. There are output iterators, input iterators, forward
iterators, bidirectional iterators and random access iterators.

memory allocators - deal with unusual ways to allocate storage for elements
of a container

algorithms - numerous template functions. Most closely resemble traditional
function library. Not supplied as linkable library but typically defined as
STL headers that you can specialize in numerous ways to make generic
program components.

numeric - template functions that perform simple arithmetic operations on
the elements of a sequence. Not constrained to operations on arithmetic
types. Given suitable overloads you can perform very general operations in
the guise of simple arithmetic.

functional - greatly enhance the power of algorithms. Each one encapsulates
a critical predicate or other calculation, keeping it separate from the
template function that implements the algorithm. You can write your own
function objects.

containers -  a container is a class that manages a sequence. Member
functions let you insert new elements, delete exisitinf elements, and
locate elements. The functions return iterators to designate elements so
you can apply various algorithms.
   vector - an array of N or more contiguous elements
    list  - a bidirectional linked list of nodes, each contains an element
   deque  - an array of N or more contiguous pointers to separately
allocated             elements
    set   - a red/black tree of nodes, each contining an element, ordered
by             some predicate applied to pairs of elements, with no two
elements             having equivalent ordering
    multset - a set that also permits pairs of elements to have equivalent 
             ordering
    map   - a set of key/value pairs, ordered by some predicate applied to 
           pairs of keys
    multimap - a map that permits pairs of keys having equivalent ordering
    stack  - LIFO queue
    queue  - FIFO queue
    priority_queue - queue ordered by a predicate on pairs of stored values
                     so higest priority element is deliverd first

Seems to me you're doing pretty well so far.

Another way to think about templates is as fancy macros for creating
"helper" classes that implement common operations for your
application-specific data classes.

You have an app that deals with, say, cars, so you create a "car" class
that has all the information you need to know about a car, and methods
for doing the things you need to do with your car data.  So far so good.
Now you need to keep a bunch of cars in the application - a container.
You'll want to do certain things with this collection.  Maybe searching
is important.  Maybe sequential access is, but only in one direction -
your app never needs to back up.  Maybe it's important that insertion
and deletion are fast, or maybe they aren't used very often and it isn't.

Obviously different data types have different performance and feature
trade-offs, which are generally well-established; and good algorithms
for implementing those features are generally well-known; and if you're
lucky, someone's created a good implementation of those algorithms for
your platform.

The idea with templates is that you can say, give me a container for my
"car" class which has features X, Y, and Z.  The author of the template
doesn't know anything about your car class - it's generic.  It just
contains the logic to encapsulate your data class and perform various
useful common operations on it.

The compiler takes the template and plugs your class into it,
customizing it for your data, in effect.

Charles' list of what's in the STL provides various examples.  The STL
also requires that various types of containers have various performance
characteristics - a given container might be required to offer O(1)
insertion-time behavior, for example.  That gives application
programmers some guarantee that the behavior of their template-using
code will have predictable performance; if you choose to use a list,
it'll act pretty much the same on any implementation.

I haven't used OO COBOL, but certainly generic programming features are
useful in OO languages in general (and often come in handy in non-OO
languages as well, though they can be trickier there).  And the STL's
collection is a good place to start a conversation about what kind of
generics would be good for COBOL.

Peter E.C. Dashwood wrote:
 > I guess I must be stupid, but I don't really understand what the
 > objection is to top posting.

LX-i has already mentioned one reason - the short top-posted response
that makes you have to scroll through a bunch of included text just to
discover there's nothing else new.

Other reasons:

- It keeps material in chronological order.  Older text comes before
newer text.

- Interlaced replies, where a relevant passage is quoted and then the
new post's response appears, provide more specific context for each
piece of the response.  The response can be less verbose and less
material needs to be quoted.  Since trimming quoted material and
otherwise keeping posts shorter when possible is generally considered a
Good Thing on Usenet (to conserve storage and network bandwidth),
interlaced replies are traditionally the preferred form.

- Many news reading agents, dating back to rn, have a feature to skip
past quoted material in a post, by skipping lines until they come to one
that begins with a different character.  If posts use the traditional
">" at the beginning of each quoted line, a reader using one of those
agents can quickly find the new material - if it's not top-posted.  This
feature has reinforced the bottom-posting convention.

(By "traditional" or "traditionally" I'm referring to conventions that
date back as long as I've used Usenet - since 1990, roughly - and
probably date back at least as far as the Great Renaming, which created
Usenet as we know it today.  These traditions are generally acknowledged
by long-time Usenet users and documented in various Usenet style guides.)

The closest thing to a standard for Usenet messages is Henry Spencer's
"Son of 1036", a draft for a proposed RFC (IETF Request For Comments
document) to update RFC 1036, which defined the original Usenet message
format.  While Son of 1036 has not yet made it into the RFC standards
track, and may never do so, it's widely considered a good draft
treatment of current best practices and majority opinion among old
Usenet hands.  Son of 1036 does not specifically recommend against
top-posting, but its recommendations for quoting practice (4.3.2. "Body
Conventions") imply that reply text follow quoted text.

Son of 1036 is available at
<http://www.chemie.fu-berlin.de/outerspace/netnews/son-of-1036.html>,
if anyone's interested.

Of course, bottom-posting and interlaced replies are stylistic
conventions, no more.  I am in general in favor of letting authors
choose their own style; I dislike style guidelines for journals, for
example, since style is clearly part of expression.  In the case of
Usenet, though, those conventions are useful - particularly when people
are trying to read messages in high-volume groups like comp.lang.cobol.
That's why I try to ask people politely to trim quoted material and in
some cases not to top-post.  There are some Usenet readers who are
rather more militant, as you've probably noticed.
