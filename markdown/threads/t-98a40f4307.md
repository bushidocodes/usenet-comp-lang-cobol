CICS (Customer Information Control System) is a general-purpose data communication
system that can support a network of many hundreds of terminals. You may find it
helpful to think of CICS as an operating system within your own operating system
(although this definition might offend purists). In these terms, CICS is a specialized
operating system whose job is to provide an environment for the execution of your online
application programs, including interfaces to files and database products.

The online end users within a network can make all sorts of demands on many different
sets of data. The things they want to do individually are usually short. Often they
are interrelated and share the same programs and data. Furthermore, the response times
they get should be as short as possible. For all these reasons, the users' transactions
are done more efficiently within a single operating system job, rather than as separate
jobs.

If all the transactions are to be handled within the same job, a controller is needed to
look after them, in much the same way that an operating system is needed within a
computer to control the jobs. CICS carries out this controlling function within a DB/DC
job.

CICS provides the communications control and service functions necessary for users to
create their own, customized DB/DC system. This cuts down the total amount of
programming needed. You can customize CICS to the needs of practically any online
application, and it can support networks consisting of a wide variety of terminals and
subsystems.

CICS provides:

The functions required by application programs for communication with remote and local
terminals and subsystems

Control of concurrently running programs serving many online users

Facilities for accessing databases and files, in conjunction with the various IBM
database products and data access methods that are available

The ability to communicate with other CICS systems and database systems, both in the
same computer and in connected computer systems.

CICS stands for Customer Information Control System.

It is the online/realtime query and update facility for mainframe
applications. Most CICS programs are written in COBOL, then probably
Assembler. This is the facility that tellers, operators, and other
business computer users use to update and query the database during
business hours. After business hours are over, CICS is shutdown and
batch processing begins. This batch processing is the COBOL that you
are now learning. The CICS COBOL is very different from batch.

I just read recently that IBM will be coming out with a CICS version
for UNIX to run on Solaris OS.

There is a lot more to CICS programming than just knowing the Syntax of the
EXEC CICS statements. CICS programming is really a separate issue from
COBOL. CICS command level programs can be written in RPG and Assembler
also, they each have minor variations on the EXEC CICS statements.

One contemporary way to look at CICS is that it is the API to the
mainframe. Other issues involved are the design of the display maps (BMS
macros), and the methodology involved with your application processing.

Properly written CICS programs have a characteristic that is called "Pseudo
Conversational". This means that to the user it appears the transaction is
always running, when in reality, between screens the program goes away.
The program must be written to be "re-entrant", in that CICS only keeps one
copy of the program in memory and shares it between users.

All of this is oversimplification. I've been doing CICS programming for 10
years, if there is a particular aspect I can help you with please let me
know.

John Lewis Giddings wrote:

› I am wondering what CICS is? I am a beginning COBOL student, teaching
› myself at home, and a lot of times when I see COBOL advertised, CICS is
› also advertised. Is this another language?

CICS stands for Customer Information Control System. It is an
operating system extension, almost an operating system in its own
right. IBM has supported CICS on its mainframes since the mid-1970's
(or perhaps even earlier). IBM's traditional mainframe operating
systems (DOS/VSE, MVS) are very much oriented toward batch processing.
CICS was created to support online, realtine transaction processing
with multiple terminals. A CICS region (on MVS) or partition (on VSE)
runs as a single batch job that owns multiple terminals and files. A
large MVS system can also run multiple concurrent CICS regions which
also communicate with each other.

CICS programs execute as subprograms of CICS itself. CICS services
are obtained by making CICS API (Application Programming Interface)
calls. Older CICS releases supported macro-level calls, but CICS
release 3.3 and 4.1 only support command-level calls. Command-level
CICS API calls have the format EXEC CICS ... END-EXEC. There are
numerous functions which can be obtained from CICS, timer functions,
queue management, task management, file I/O, screen I/O, and
communications. You can also EXEC CICS START ... END-EXEC to start
asynchronous tasks within CICS. You can also do event-driven
programming in CICS.

Most CICS programs are probably written in COBOL. You must first pass
your COBOL source code thru a translater, or pre-compiler, to
translate the EXEC CICS calls into standard COBOL CALL statements
(with very odd-looking parameter lists). Then you program is compiled
by the normal COBOL compiler and linked with some CICS stub routines
to make it usable in CICS. CICS programs can also be written in IBM
370 assembler (sometimes called BAL, Basic Assembler Language), PL/1,
possibly Fortran, and certainly C and C++.

CICS is available for platforms other than System 370 mainframes.
They include OS/2, RS/6000 AIX, and non-IBM Unix platforms. I believe
CICS is available for certain Hewlett Packard Unix machines.
