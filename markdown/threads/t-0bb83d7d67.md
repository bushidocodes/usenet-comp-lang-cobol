    .NET is a platform, like Java. When you compile a so-called ".NET 
program", you don't actually compile to native x86 executables, but rather 
you're compiling to an intermediate language called MSIL (Microsoft 
Intermediate Language). Windows users then download the ".NET Runtime" which 
basically acts as a interpreter to run these MSIL files. Microsoft 
distributes the C# compiler and .NET runtime for free; if you pay for Visual 
Studio, you're mainly paying for an IDE.

    Mono.NET is primarily a .NET runtime for Linux, to allow Linux users to 
run .NET programs. Last time I played with Mono, which was several years 
ago, there were severe limitations with what you could do with it. For 
example, Microsoft's official class library includes GUI stuff for making 
windows, with textbox and buttons and stuff. Mono didn't support that at the 
time.

    I've the "screenshots" page of the Mono project recently, and I see they 
do support GUI, but using something called the "GTK#" library. I'm not sure 
what this means, but I suspect if you take a C# program written for Windows, 
you may have to make some changes in the source code to make it run on Linux 
(i.e. change all reference to Microsoft's GUI library to the GTK# GUI 
library instead). Probably though, if you write a C# program for Linux, all 
you need to do to get it to run on Windows is install the GTK# library on 
your Windows machine.

    - Oliver

A small correction, not that it much matters in the context of the
discussion, but MSIL is never interpreted. It is JITted (just-in-time
compiled) to machine code on first use for each run.

You are right that Mono has limitations... they're likely not keen to
reproduce the entire Windows environment on Linux... I can't blame
them.

In that sense, C# is not like Java: in order to do anything really
useful in C# you likely have to dip into functionality that is unique
to Mono or unique to Microsoft's .NET runtime. While in theory .NET
code could be multi-platform, in practice if you want multi-platform
you're better off sticking to Java, IMHO.
