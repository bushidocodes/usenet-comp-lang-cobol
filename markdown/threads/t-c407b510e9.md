I want to start a new thread here, partially in response to some of the 
bad feeling that has been evolving, and partially in response to the 
many interesting issues that the thread has raised.  I think that every 
veteran member of this newsgroup will agree that styles of cobol have 
evolved, and that this is a good thing. Being concious of style, and 
willing to spend time learning to code with style is the mark of a good 
programmer.

On the other hand, it has spawned many a bitter battle, and the 
definition of "good style", is perhaps a problem that Grace herself 
would have delegated to a committe dooomed to eternal bickering.

I would propose that style is not a matter of following rules, and that 
if reaches that point, the code is probably doomed to perdition anyway. 
I think many different styles serve the purpose of making code readable 
and maintainable, the most important aspects of source code.

Having said all that, I think that we need to re-look at style 
frequently.  With the addition of OOP, stylistict methods of the past 
may nor be as effective, and new methods may have to evolve.

I am going to post some code ... after some pride induced reflection, 
unretouched, as I put in in my library a week ago.  It is a version 1.00 
of a new system, and is a method within a ledger object.  It does a 
search of ledger items within a ledger object, based on date, account, 
reference number, etc.  It starts off by creating a screen object, then
well ... I'll comment in the code.

Donald

For those not use to OOP, the following exists within an object, with 
it's own working storage section.  This working storage starts fresh 
each time the method is invoked, however the data in the other portion 
remains in effect from the creation of the object to the completion.

This is invoked when the user presses a "search" menu item or button. At 
thet time, it inherits the object it is in, which is a ledger.  The 
ledger exists as an array at the next level up.  That array is what is

This routine creates a screen, and sets the starting search criteria. 
They are set from the main object ... that is, they retain their value 
while th current ledger is open.  That allows one search to start where 
the last one left off.  You'll notice that I use periods, but also use 
indentation to draw the eye to lines with abnormal termination. I also 
use all uppercase .. a matter of personal choice.  My editor 
colour-codes all that stuff, BTW.

            MOVE "N" TO END-FLAG.

SCREEN-OBJECT.

The main loop runs a screen.  It contains four radio buttons, which 
determine the search type ... by account, by date, by reference, and by 
account number that is not in the chart of accounts.

As each radio button is pressed, it greys/enables the fields associated 
with the button.  For example, if the search is by date, the from date 
and the two date are enabled, and the from/to account are greyed out.

Editing and auditing of the fields is done in generated code (Generated 
by the SP2 screen system, in Cobol)

The buttons/menu items on the screen are search forward, backward, from 
current, from the start, home, jump to end etc.

            ELSE
            IF LOCAL-KEYSTROKE IS EQUAL TO 27
                MOVE "Y" TO END-FLAG
            ELSE

ACCOUNT-NUMBER 					(SUB2)

                    MOVE "Y" TO SEARCH-END-FLAG
                    .

                PERFORM BAD-SEARCH

            IF SUB1 IS GREATER THAN DEBIT-CREDIT-COUNT

And some comments on my own style.

You will notice that I write very old fashioned code, with narry an 
endif in sight.  The main reason for that is that I consider the 
architecture of the system far more important stylisticly than the 
actual code.  With the actual code, I tend to prefer the lowest common 
demoninator, and rely on paragraph naming and fairly small modules to 
make reading it simple.

That also includes screen loading, to an extent.  The simple fact is 
that all those extra words, particualrly when you put them on a line by 
themselves, take up screen real estate.  By using OOP to do the main 
chunking, each module is reasonably well isolated from the others, and 
local vs global data is kept under control.

You'll notice I like redundent brackets when entering complex logic. I 
think that they do make the logic much more easily read by the human, 
even if they make no difference post compilation.

Still on this OO topic - who do you and I ask ? There are as yet no OO COBOL gurus
that I'm aware of, and I don't think you can really look to other languages for a
principle.

As a very broad brush statement :-

A COBOL method is a piece of self functioning code, (mini program), with the
ability to be invoked or to invoke other methods. If the method contains 'sub
functions' - consider writing these as 'PRIVATE' methods if there is the
possibility that they can be used by more than one PUBLIC method.

PS: For non-OO-ers, There's no such animal as PRIVATE or PUBLIC in OO COBOL. The
PUBLIC methods are the ones you invoke like a non-OO CALL. The 'PRIVATES' are only
invoked from within the current class, e.g. :-

    invoke self "doThisSubFunction" using x return y.

That would be any COBOL compiler compliant with ANSI X3.23-1974, and the
procedure could also (with THRU section-name-2) be a group of CONTIGUOUS
sections.  Allowing paragraphs or groups of contiguous paragraphs was a
reasonably common extension.

ANSI X3.23-1985 relaxed that restriction, with some side-effects (I've
mentioned that elsewhere in this forum).

More to the point, there's no restriction in '85 COBOL on branching out of
the paragraph to anywhere else in the program, so long as you come back to
the terminus of the procedure.  In '74 and prior, an INPUT or OUTPUT
PROCEDURE was defined by a block of code at compile time; in '85 and
subsequent, the INPUT or OUTPUT PROCEDURE is defined by whatever path the
program actually takes at execution time.

    MOVE PrimeKey TO KeyField OF OS-CUSTOMER-FILE.

For an especially complicated statement, then X might well be 1 or 2
lines.  For a simple series of MOVEs then X could well exceed hundreds
of lines.  The decision of what X is is a heuristic judgement on the
part of the author.  Some programmers have firm rules like "no more
than one screen" or "no more than 50 lines", while some approach the
problem with a 'gut-feeling' of "this routine is doing too much".

The general idea is to keep a separately executable section of code
(paragraph/procedure/function/subroutine/whatever) small enough to be
understandable at a quick glance.  When I see a paragraph called
"MUNGE-WIDGETS", if it is small enough, and clear enough that I can
grasp what it does in a glance, then I can forget about that details
of that section of code and treat "MUNGE-WIDGETS" as an atomic verb. 
This approach supports the 'brick-laying' metaphor of programming. 
Once a brick is built and tested it can be attached to other bricks to
build completely different structures without changes.  Those
structures can then be treated as single bricks and assembled to build
even more complex structures.

This is why the paragraph (function/procedure/subroutine/whatever)
should be named with exactly what the function does.  Not only does it
help clarify for any future reading, but it discourages the original
author from putting unrelated statements in the paragraph (sometimes
called "side effects").  A paragraph named
"A123456789-CENTRAL-LOOP-PARA" can contain ANYTHING and will require
very detailed reading to grasp its purpose.  A paragraph named
"B123456789-READ-FILE-X" usually gives a clue to the reader that it is
going to read file X and nothing more.  A quick glance at READ-FILE-X
will verify if the author kept up this convention or if he did lots of
unrelated things in the paragraph.

As for comments, well, they are usually worse than useless, sometimes
very helpful, and often out of date.  I cannot count the times I have
seen entire programs filled with comments like:

   *   Add amount  outstanding to total outstanding
       ADD XYZ-AMOUNT-OUTSTANDING TO ABC-TOTAL-OUTSTANDING

Such comments are worse than useless as they just clutter up the code
with a nonstandard restatements of the Cobol.

Where I do find a great deal of value is narratives for each
paragraph/section/nested program that states what the author is trying
to accomplish, what they expect in, what they do with it and how they
put it out.  While comments of this style are often great and helpful,
it is easy for them to get out of date as maintenance changes things. 
So unless the program is unchanged from the original implementation I
usually don't trust them without careful verification.

Just my humble $0.02...
