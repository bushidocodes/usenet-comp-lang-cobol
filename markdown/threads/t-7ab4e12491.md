I've always been handed embedded SQL, specifically Ingres embedded in
VMS COBOL.  From what little I know of the others, I think the
differences are small - just different interfaces to stuff your SQL
from here to there and suck the data from there to here.

A bigger change is how you handle the data.  I've done battle with
code that extracts a bunch of data from the database, stores it in
memory or a file, steps thru the records one at a time, sending the
data back to the database as part of a query to get more data, etc.. 
Uses lots of resources: database, network, memory, disk, CPU, time.

I've rewritten these pigs to create temporary tables in the database,
populate them from the other tables, then retrieve the data from the
database for final processing into a report or file.  Compared to the
original program, I may use more database resources, but I use a whole
lot less of the other resources.  I can also execute multiple user
requests for a report at one time, even if their parameters are
different.

So I say don't sweat the interface to the database, try to learn how
to make the SQL do most of the work for you.

Well one's done at compile-time and the other is done at run-time. But then

I believe "Whether of not to use Dynamic SQL" to be a seperate question. I
was merely pointing out and apples and apples alternative to the usual
pre-compiler "exec-sql . . .blah. . . end-exec" approach. I'll include a
brief example below.

With Rdb/VMS dynamic SQL is supported with either the pre-compiler or
the SQL Module Language interfaces. Personally I would only *ever* use
dynamic SQL when statement syntax is dependant on run-time discovery
of environmental variables or user input. Which in my expeience is very
very infrequently.

Some people don't seen to have a problem with interpreters (and let's face
it, at least in Rdb's case, compiled SQL is only Binary Language
Representation and a long way from machine code) but, where possible,
I like to shift CPU utilization and memory allocation to the compiler rather
than the run-time. But then I love full stops, minimise the use of
END-verbs, think INITIALIZE is the dog's bollocks (especially for
arrays), perform paragraphs in sections and then go to fini, and reserve
the use of upper-case to literals that need it - so what do I know?

As far as ODBC goes, it is easy to use (well if you can't say anything nice
:-)

Regards Richard Maher.

Example Module language:-

module    dist_sql
language  cobol
parameter colons

declare pers alias filename mf_personnel

procedure declare_connection_name
        sqlcode,
        :db_spec        char(50);

        attach :db_spec;

procedure set_trans_rw
        sqlcode;

        set transaction read write
                reserving pers.departments for shared write;

procedure update_dept
        sqlcode
        :dept_name    char(30);

        update pers.departments
                set department_name = :dept_name
                where department_code= 'PHRN';

procedure disconnect_db
        sqlcode;

        disconnect default;

Well, you are getting fairly compiler/OS/DBMS-specific here, but my point
was that both these sequences
are (E)mbedded (SQL):

== sequence one ===
EXEC SQL
     PREPARE stmt
END-EXEC

EXEC SQL
   EXECUTE stmt
END -EXEC
================

=== sequence two=====
EXEC SQL
    CALL storedprocedure
END-EXEC
==================

"At the end of the day"  these get to the same place, they just get there
using different routes.

In sequence one, the statment is parsed and a plan prepared at run-time  In
sequence two, the statement is parsed and a plan prepared when
'storedprocedure' is created.

It's been a while since I worked with the IBM-brand mainframe ESQL product;
it's possible, I guess, that if 'stmt' is sufficiently simple the
precompiler might be able to prepare 'stmt' at compile time, but that would
IMO qualify as a brand-specific optimization.

PREPARE is only used when you build an SQL statement within your program.
Stored procedures are executed simply by calling them with any parameters
needed; they are already PREPAREd.

The correct sequence should be something like...

MOVE        "01.01.2003" to VAR1
MOVE        "31.01.2003" to VAR2
EXEC SQL
   CALL "MB_PAY_PREDPR" (:var1, :var2 )
END-EXEC

Var1 and var2 must be SQL variables, usually defined with ...

EXEC SQL
   BEGIN DECLARE SECTION
01  FILLER
      05 VAR1        PIC whatever
      05 var2          PIC whatever
END- EXEC

stored procedures"
