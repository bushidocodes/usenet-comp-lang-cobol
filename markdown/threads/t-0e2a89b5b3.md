```

Packed format in EBCDIC is defined as 1-n bytes (I don't recall offhand
the maximum n...it is probably 9) in which the first n-1 bytes contain
numeric data, and the last, or n-th byte contains a nibble of numeric, and
a sign. The data is structured such that each decimal digit is placed as
a binary value in one half (4 bits) of a byte. The last byte has the last
(least siginificant) digit in the high order nibble, and the sign in the
low-order nibble. The sign can take two valid values, either 0C
hexadecimal (1100 binary) to indicate a positive value, or 0D hexadecimal
(1101 binary) to indicate a negative value. You will also sometimes find
a value of 0F hexadecimal (1111 binary) in the lsign positsign position,
which is 'no sign', but which can safely be treated as 'positive' in most
cases.

An example of all this would be the following:

Value +12345 would be packed 12 34 5C hexadecimal (3 bytes)

Value -12345 would be 12 34 5D hexadecimal (3 bytes)

Value -0123456 would be 01 23 45 6D hexadecimal (4 bytes)

An interesting consequence of this is that any packed decimal item
implicitly has an odd number of siginificant digits.

You may sometimes find a variation on this format known as 'stripped
decimal'. In stripped decimal, a packed decimal value is modified to
remove the sign and store an even number of digits in n/2 bytes. For this
format the packed decimal value is shifted four bits to the right,
dropping the sign nibble off the end. This is especially popular for
storing dates in gregorian format, which makes a typical 011895 fit in
three, rather than four, bytes.

The purpose behind packed decimal is not to 'compress' data, but is
actually the format in which the decimal arithmetic instructions for a 360
or later class IBM mainframe expect the data in the registers. There are
various reasons why, which are best explained in the 360/370 Principles of
Operation manual (if are interested).
```
