Replying to a couple of posts:

In the IBM Mainframe world, external sorts are terribly inefficient, harder to maintain (usually because no-one puts comments in the control cards), and not really easier to code.

If you have an INPUT PROCEDURE and/or and OUTPUT PROCEDURE, the external sort requires that you pass the file an extra time -- both as input and output. Yes, I know there are control cards for selection and editing, but they are hardly ever used, and COBOL is easier to understand and maintain.

If you use the FASTSRT compiler option, the compiler will use the house standard (DFSORT, SyncSort) sort instead of the COBOL Compiler's sort. This, of course, applies only to USING and GIVING, but is much more efficient than the COBOL compiler's own sort. This was not introduced until about 1989 by IBM.

Since external sorts require both control cards and extra JCL, I don't really see them as easier to code than a COBOL program.

I have hardly ever used a USING or a GIVING in quite a number of years.  The file usually has to be manipulated before sorting, and then used somewhere else.  INPUT and OUTPUT PROCEDUREs.

1. To those who believe external sorts are faster, I say, "show me your
benchmarks".
2. Sorts are almost invariably done as input to a program that will do some
sort of processing on the sorted data, and frequently as part of an
Extract/Sort/Report process.  Why do six lots of EXCP's when you only have
to do two?
3. Why add an extra layer of maintenance (coding changes to the sort control
cards as well as the COBOL copybook)?

In the high and far-off times, when memory was expensive, it was very
inefficient to code an internal sort - the sort was cramped for work-space
in memory.  Today, a large percentage of sorts are actually done in main
memory - just check your stats - because the typical region size is 32MB.
