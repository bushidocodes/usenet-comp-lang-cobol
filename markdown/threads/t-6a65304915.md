I would like to see the following minor items added to the COBOL standard
either in this revision or a future revision.

1. A declarative for initial processing. This would eliminate the
awkward ways we currently have for doing first time processing in
called subroutines. I know of one shop that uses the ALTER verb to
avoid having to set up a special field in working storage. This is
done also so that they can have a common copy module. i.e.
BEGIN-SW.
GO TO INITIAL-PROCESSING.
INITIAL-PROCESSING.
ALTER BEGIN-SW TO PROCEED TO MAINLINE-PROCESSING.
|
some code
|
MAINLINE-PROCESSING.

The declarative would enable the compiler to "bundle" the code in the
declarative with the other initialization code required by the
module. USE ON INITIAL ENTRY could be one wording that uses existing
or previous reserved words. This code would not be executed when the
routine is to be entered in its last called state.

2. Add the colon ":" to the allowable picture characters for editing
fields if possible. It would be useful for editing the time.

3. Strange as it may seem, allowing a program with just WORKING-STORAGE
could fill a niche need. The shop I am at has several programs in
assembler that are just data areas loaded at execution time into
the linkage section. It would be nice to be able to do the same
thing in COBOL. The program is treated as an 01 level and the
beginning of the program is the beginning of the data area.

4. Make ZEROES and ZEROS a synonym of ZERO in all instances so that
you could write IF data-name-1 ZEROES ... . If this had been
available I would have save some time in converting some IBM COBOL
74 Standard programs to COBOL 85.

5. Add a with access clause to the OPEN statement so that OPEN INPUT
file-name-1 ACCESS DYNAMIC would be legal and would override the
SELECT statement. I would have found this useful in modifying some
common I-O sub-routines. Many vendors do much of their file I-O
for indexed files in common routines so that they can be replaced
by routines doing data base access without changing the code in
the calling program.

If these have already been thought of and are in the process for the
current standard I would be happy to see them but not at the cost of any
delay to the release of the new standard.
