First of all you said you were checking your logic for correctness. It
seems strange to have a pic 9(8) field when the three values you are
checking for are two digits or less.  Also if the program is correct and
runs quickly enough then you do not have a performance problem.

According to the IBM COBOL Language reference: 1-4 digits in PIC generates
2 bytes, 5-9 digits in PIC generates 4 bytes, and 10-18 in PIC generates 8
bytes.  I believe the compiler "knows" that WA-ACTIVITY-ID is a fullword
because it uses a Load  instruction to put it register 7.  So why is it
creating a double word item for comparision? The only reason I can think of
is that the compiler does "not know" the maximum size numeric literal that
you may want to test at some later point in your code.  What if you decide
to code "IF WA-ACTIVITY-CODE IS LESS THAN 9999999999"? That is ten 9's
which is hex '2540BE3FF' which is larger than a fullword.  I think the
short answer is you cannot expect the compiler to beat human coding in all
cases, as it is sometimes generating code to handle the general case.

So why does it generate the extra  L/SLR/STM instructions?  I can only
guess that it generates code for the second and third tests without
"remembering" what it did for the first test.  Only the compiler writer can
really answer your questions, and as many compiler options "interact" even
he/she would need more information.  From my reading of the manual and from
your tests the TRUNC option you select seems to be a key item.

A) IBM has (slightly) modified its recommendation concering TRUNC(BIN) and
"other" languages/products.  Please read the "recommendation" at:
  http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igy3pg00/2.4.52

B) As also indicated in that same section, *IF* you are using the latest
compiler, you can use TRUNC(OPT) for your entire program and use COMP-5 for
any *specific* data-items that might (realistically) have values outside the
picture clause.

  ***

IBM is quite clear that even with their latest compiler changes, using
TRUNC(BIN) is *always* the poorest performer (for programs that reference a
lot of binary items).

Have you read the section in the programming guide on performance options
at:
  http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igy3pg00/7.1.5.1

and/or the section on "7.1 Tuning your program"

at:
 http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igy3pg00/7.1

Bottom-Line:
  Don't use BAD coding or features (such as TRUNC(BIN)) and expect the OPT
compiler option to do as much as "good coding" will.
