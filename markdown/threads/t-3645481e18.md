Bob,

COBLIB.DLE is indeed the Cobol run-time library, if you are using
dynamic linking, and ISTR that it loads at the top of available
memory.

How is your program structured? If you are using CALL/CANCEL to
access different modules, then the modules will vary in size. Also
buffers for files being opened/closed may have an effect.

It is possible to create a .EXE, if it is a simple program. and
link in the STATIC runtime library from which only used routines
should be incorporated and ther runtime library is not required.

I use M$ Cobol 4.5 which is badged MF Cobol 2.4(?) and build
applications with CALL/CANCEL, normally linking to .EXE using
the dynamic link library which is more flexible, but occasionally
have had to build a simple program using the static link library.

This can be solved rather easily by using XM. Either use XB to link XM
into your main EXE (it kinda takes over) or type XM myprog. This will
load the runtime high, and in the normal MF runtime environment, use
about 250K of base memory leaving the rest free for your DOS shell.
