left on the bookshelf - so many (well there's only about 6 or 8
altogether) generalize about OO COBOL and don't lead you anywhere.
So let me recap on my two "favourites" :-

- SIGS ISBN 1-884842-34-8/Prentic Hall ISBN 0-13-261140-6 - Arranga and
Coyle 'OO COBOL"  $CDN 52

Wilson (Will) Price "Elements of OO COBOL" $?? (can't remember). You can
get this one pronto from objectz.com if you write Ed Arranga - 
74651.2630@compuserve.com. Perhaps Ed has a loft full of the other book
too ;-) . Might be able to get you that. 

They both discuss the topic and then give you examples in 'real' source
code. Arranga/Coyle demonstrate an in-house Library system - ANSI
display and working-storage contains the list of library books. Will
Price's book takes you through it step by step building up source from
structured to OO. Then Will has an addendum dealing with flat file
methods.

Put the two books together, the Arranga/Coyle Library system and
superimpose Will's file handling - and you have a working OO system.

Having got through these two books - then contact me and I'll happily
give you adapted source code based on the Will Price file handling
methodology.

you with OO ? Same goes for any of you other OO people.

Having learned OO via Analysis and Design courses, with uses of the books "Object Oriented Analasys" and "Object Oriented Design" from Coad/Yourdon, I started programming in Smalltalk and in C++, after courses in those languages too.

Meanwhile (some 4 years) I stayed in the business of programming in COBOL for about 10% of the time. Other days were filled with teaching COBOL/PL/1, C++, OO design, Smalltalk, and some ... about 50% procedural, 50% OO. That's practice, not books!

Than I ran into OO-COBOL by promotion speak of Micro Focus, interest etc. 1997 was in the near future, OO-Cobol and COBOL '97 were synonimous those days. I followed some preliminary version of a COBOL course and later-on developed a definitive version. We used the OO-extension of the MF workbench and later Personal COBOL. Then I started using it (OO-COBOL) in real projects, allbeit small-scaled.

I prefer "Object Oriented software Construction" from Bertrand Meyer as *the* book on OO.

Good programming is good design followed by good coding. Good design is possible either OO or traditional. Coding can be procedural in both cases. But a non-OO design (however well-crafted) is almost impossible to code in OO. 

(I highly prefer to code OO if the design is OO, and to code non-OO if the design is non-OO).

In any case, ignoring your snide reply, I'll make some comments on the
book.  (Looks like no one else has heard of it.)  There are three
parts: I) BASIC COBOL, II) ADVANCED COBOL, and III) BEYOND COBOL.  So
far I've only read part I.  It is, of course, not much use to someone
who already knows COBOL.  But I wanted to see how the writer
approached it.

I honestly don't think it would be much good for just learning COBOL.
Various facts are presented, but it really doesn't go in to *why* you
would want to use a certain feature.  Such as REDEFINES.  I mean, it
says it's to save space by using the same data area, but there's no
real hard examples as to how one would do that in a real life
situation.

He also makes several mistakes.  One is just totally wrong.  He states
that PERFORM WITH TEST AFTER UNTIL ... is the default if you just say
PERFORM UNTIL ... We all know that it defaults to WITH TEST BEFORE,
and in fact WITH TEST AFTER was not even part of the 1974 standard.
(Well, nor was WITH TEST BEFORE, for that matter.)

Then, as an example of "Trust the code and not the external
documentation" he uses the following:
*  Determine if Last-Yr is a LEAP YEAR.  If so, Leap-Flag
*  contains ZERO.
      DIVIDE Last-Yr BY 4 GIVING A-Num REMAINDER Leap-Flag

Now if this were an example of *bad* code that would be fine...
But...

Some of his coding style gets to me, but of course that's just a pet
peave.  He likes to use "THEN" in his IF statements:

IF condition-1
    THEN DISPLAY 'THIS IS TRUE'
                 SET Data-Name TO TRUE
    ELSE DISPLAY 'THIS IS NOT TRUE'
END-IF

I do like his statement not using periods except to end the paragraph,
though.

Anyway, I think I'm going off on a tangent here.  Skimming through the
next section (ADVANCED COBOL) I can see some other problems.  He
mentions the "SELECT OPTIONAL"' clause for files that may not exist.
He says you can test the file status by coding the FILE STATUS clause
as described in Chapter 17.  Chapter 17 then goes on to say that FILE
STATUS is rarely used!  Really?  We use it all the time at our shop,
and I've always heard it's the best way to go.

There is mention of the availability of CALL BY CONTENT and CALL BY
VALUE (as well as the default CALL BY REFERENCE).  Here is his
explaination of CALL BY VALUE:
"BY VALUE is similar to BY CONTENT, except it uses linkage that allows
calls to non-COBOL languages, such as C/C++.  The arguments can be
binary, floating-point, pointer, or a single-byte alphanumeric
character."

Now, knowing C/C++, I realize that this means that the actual value is
passed, rather than a reference (pointer) to a value.  But honestly,
if I didn't already know it I'm not sure that I would have figured it
out from this,  (The COBOL I use does not even offer "BY VALUE".)

As for OO, there's only one chapter on it; about 35 pages.

All in all I'm not terribly impressed.  As I said in a previous
message, there is some mention about most every feature of COBOL.  But
I don't think they're all explained very well.  Another more detailed
reference is definately necessary in many cases.
