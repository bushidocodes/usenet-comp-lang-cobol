```
I'm new to Net Express.  I recently moved an old application to NE
3.0.  Everything works fine, except that one user's (the boss :) PC
gets an error message (9/013: file not found) when it tries to print.
In an attempt to solve this, I am testing using the run-time tunable:

    set printer_redirection=TRUE

which sends the printer output through the Windows printing subsystem
instead of directly to the printer.  This is working, because it now
prints to the Windows default printer instead of the local one.  I
think this may clear up my client's problem.  However, the printed
output, while 10 cpi, is not exactly 6 lpi, making the data overflow
the end of the page onto the next sheet.  One report is specifically
designed to fit (barely) onto two pages, so adjusting page length is
not a good solution.

Can anybody tell me an (easy?) way to tell the printer to print at
6 lpi?  I know how to do this in VB.  Other options, fonts, etc.,
would be nice to know, but now I only need to set lines per inch.
Thanks!
```

I'll check them out.  I assume you are referring to the PC_PRINTER_xxx
routines?  Apparently they are not compatible with an FD print file,
because to use them you must work with a numeric 'printer handle', and
I can't see how to get that from an FD, or vice versa.  It seems they
are two completely different methods for using the printer, and can't
be used together on one print file.  Anybody please correct me if I am
wrong here.

Net Express is impressive in a lot of ways, but I am frustrated that
there does not seem to be a smooth, balanced approach, to printing at
least.  Did nobody ever think that since we're now working in 32 bit
environments, it might be nice to simply set the font and print
margins, in addition to opening, printing and closing?  Why force us
to use a completely different approach to printing, just to do that?
Of course, certain kinds of printing (multiple fonts, bold, etc.)
would probably require a new approach.  But anybody might want to
use compressed or large print for a report.  You could implement that
using the VALUE OF clause under the FD without leaving standard COBOL.

The Net Express documentation gives you lots of big pictures and tiny
details, but can be sketchy about how the tiny details relate and
function with each other to make the big picture.  It's as if you can
find the microscopic details of the bark, and maps of the forest, but
the intermediate details, like how the bark or limbs fit onto the
trees, can be missing, or difficult to find.  There seems to be the
implicit assumption that if you explain the little details and the big
pictures, the reader will automatically know the middle level details
without being told.  For me, at least, this is not always obvious.
Just a paragraph explaining how the PC_PRINT routines relate to FD's
would be great.  The routines are explained, but I can find no context
description.  Instead, I must make assumptions, or deduce the context
for myself.  I have written enough documentation to know it is not
easy to always be clear.  No matter how clearly you write, there will
always be some moron who will misunderstand what you say.  But you
must at least say it. :-)  The first assumption you should make in
writing documentation is that the reader doesn't know anything about
what you're documenting except what you tell them, including its
purpose.  How many employees have been handed a new software package
and told to "learn this" with little or no idea about what it does?
More than a few, I'll bet. :-)
