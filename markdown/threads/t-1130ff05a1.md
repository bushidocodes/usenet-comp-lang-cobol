Currently, we are converting Cobol programs (and data) from a HoneyWell
Bull computer to IBM MVS Cobol II.

One of the problems we face is that on the Bull computer a COMP-3 field can
be unsigned.

So a PIC 9(4) COMP-3 field occupies 2 bytes as in MVS we will have an extra
half byte for the sign which means an extra full byte as MVS can not handle
half bytes. This can give a lot of problems in the following situation:

on Bull

03 DATE PIC 9(6) COMP-3. (this occupies 3 bytes)
03 REDEF_DATE REDEFINES DATE.
05 MONTH PIC 9(2) COMP-3. (1 byte)
05 DAY PIC 9(2) COMP-3. (1 byte)
05 YEAR PIC 9(2) COMP-3. (1 byte)

In MVS, MONTH, DAY and YEAR will occupy 2 bytes each . However, DATE will
occupy 4 bytes, so the REDEFINES is not longer correct.

Please don't tell me to put all fields in DISPLAY. I know this will work
but the customer will not accept this.

Anybody an idea for this problem??
Can the new Cobol for Language Environment handle unsigned packed fields???

Well, it looks like you are trying to preserve dates with two digit years.
Isn't this a good time to update the application to support dates with
four digit years?

Suppose you write a program (in Honeywell COBOL) to create a temporary
version of the file containing your dates in character format, and then
wirte a program (in IBM COBOL for MVS & VM [the latest version of
COBOL/370]) using LE services to store dates in Lilian format; in this
way, you would end up with dates stored in four bytes and it would include
support for dates with four digit years. [Ahem; training is available from
yours truly.]

COBOL for the LE environment (COBOL for MVS & VM) cannot handle unsigned
packed fields as unsigned packed fields. I don't see a way to convert and
end up with a three-byte structure containing month-day-year in IBM COBOL
(any compiler).

COBOL's definition of usage for numeric items is that they can of the
types DISPLAY or COMPUTATIONAL and leaves the definition of what is
computational up to the implementor of the compiler. Usage types like
comp-1 thru comp-5 are machine specific extentions to the language and no
compatiblity is guarenteed from one compiler to another as you have
already seen. I encountered the same problem you have while converting
data from a Unisys system to IBM and I spend a lot of time explaining to
many confused COBOL II programmers that in Unisys 9(4) comp and S9(4)
comp were not the same size. The only compatible form is DISPLAY and if
the final data is not stored in this form it must be at least passed in an
intermediary file this way with the sign as a separate character if it is
not to be lost.

This is not exactly a situation where "the customer will not accept
this" applies. You are employing 'implementor specific' portions of
COBOL in the Bull compiler, and now, you have to pay the price for doing
so in order to migrate to COBOL on a different platform.

PACKED-DECIMAL became a standard data type in 1985 ANSI COBOL, and
IBM defines COMP-3 as a synonym for PACKED-DECIMAL. Obviously, BULL
didn't see things the same way. Many vendors have supported COMP-3 as
the same data type as IBM S/390, since there is so much IBM mainframe
COBOL; surely, this must be why PACKED-DECIMAL was adopted as a
standard.

Your basic choices seem to be:
(1) Reformat your data as you move it to the IBM and change the data
descriptions. (Later on, you'll go back and make Y2K fixes.)
(2) Write a subroutine which accepts your BULL format data, and
converts it to some useful data type for IBM - this is similar to the type
of work required to do Year 2000 repair when a shop decides not to
expand dates. (Later on, you'll go back and make Y2K fixes.)
(3) Go all the way, as another post suggested. Fix your data for
Year 2000 (expanding to 4-digit years), change your record descriptions,
and you're set for a long, happy computing future.

Above all, remember the possible cost of using a vendor's language
extensions, whether it is BULL, IBM, Micro$oft, or anyone else. The
extensions may save you costs now, but they also tend to tie you to the
platform (to the vendor's benefit).
