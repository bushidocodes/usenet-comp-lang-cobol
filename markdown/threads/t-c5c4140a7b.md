Hello all. Here's my question:

I have a Stand Alone application, let's call it TEST1, that opens a
Line Sequential Data file for output. I call another application,
TEST2, and I want this second application (TEST2) to be able to access
the data file opened in TEST1 without having to "re-open" the data
file in TEST2. I'm really not sure that this is plausible, but I
would really like to do this. Thus far, I have tried:

Passing the literal TEST-FILE, which I have defined in TEST1 as an
actual data file, in the Using Clause of the Call Statement, which it
accepts, but I cannot find a way to Link into that data area in TEST2
so I have access to it.

Just defining TEST-FILE in the TEST2 application without coding
Open/Closes but attempting to Write to the data file.

As above, but also passed the FD in the Linkage section of TEST2.

We are using Micro Focus Cobol 3.1.35 (after years of using 2.4.38).
and both "applications" are written under this version of the
compiler. TEST1 is the Main Stand Alone application and calls the
Shared Run-Time Application TEST2. Why do I want to do this? Well,
let's just say, I want ONE program to handle all my File Opens and
Closes, while allowing other shared run-time applications to access
those ALREADY open data files.

If you have any ideas, or if you know the answer, please let me know.
If you need more information or details into what I am trying to
accomplish, please let me know.
Thanks!
D.
Also, thanks for taking the time to read this.

The obvious answer, which is sure to be transportable is to put the
file and all logic to read/write to it in a dynamically called
subroutine, so that a single instance of this subroutine can
service all applications running concurrently.

Pass the stuff to be written/read via an argument structure, and
open the file input/output.

You may have to maintain a flag in this routine to keep track
of whether the file is open or not.

See if MF will automatically close this file properly when
the last run-unit useing your subroutine terminates. If not
you will have to plan a smooth reliable file close mechanism
into your subroutine.

I have the answer for you!

Both programs need Selects and FD's. In your FD put "IS EXTERNAL" right
after the FD line (before the record contains). This must be in both
programs, the main and the called program. Perform all opens via the main
program and the called program can read and write to the file with no
opens. Look up "External" in the language reference for more information.

I don't know of any way to share an FD between programs. I'm not
saying there isn't a way, I just don't know of one.

I would suggest you write a subprogram to be a file server. It would
have the only SELECT statement and FD for the file. Call it with a
command parm (open, close, write), and a buffer parm (the record to
write). Have it return a return-code parameter (the results of the
I/O request). Then you can open your file from TEST1 and write to it
from TEST2. You could also have it do error-handling for severe
errors. You could also have it filter out 'illogic' requests
(attempting to open a file that's already open, close a file that's
already closed, et cetera).

I know this method works. One disadvantage is that it is a little
easier to forget to close a file at end-of-job, since you have to
request the subprogram to close the file for you.

I hope that helps.
