I could think of two possible meanings of "run-time configuration" files:

1. A "control" file or control card(s) read by the program, and fields on that
card identify different parts of the program logic to execute or designate
different subsets of data to operate on. For example, a label-writing program
may read a "control card" to tell if labels should be generated for
current-term students, applicants for next term, or for everyone on file
(student or not).

2. Certain compiler options may affect runtime interpretation of certain
commands. Some compilers may take a CBL or *PROCESS card, some compilers may
read options from a disk file. Some options may be retrieved at execute time,
either from a file or a parameter on the EXEC statement or RUN statement for
the program.

Without more specific information, all we can do is guess what your instructor
meant by a "run-time configuration" file.

Andrew,

First possibility: Your professor is referring to a feature of the Cobol
product your using.
I know of several Cobol products that use runtime configuration files.
They are used to affect the "environment" that your Cobol program runs in.
RC files basically allow you to set your own defaults for the resources used
in your Cobol programs while they are running. Thus, they are called
"runtime" configuration files.
For instance, RC files can be used to affect the way data is stored in
files, video settings, keyboard settings, file naming, graphics, memory
usage, standard error messages etc, etc.
The RC settings in the RC file are basically stored in environment
variables that the Cobol programs automatically load into memory and then
reference as needed. For example, if I were to hypothetically set a runtime
configuration setting of FILE-SUFFIX = DAT, and run my Cobol program, it
would use DAT as the file extension for any new data file I created. For
instance, if my Cobol program creates a new indexed file called VENDORS,
then it would actually be stored as VENDORS.DAT
Unfortunately there is no ANSI standard that I'm aware of for runtime
configuration files. Therefore, each Cobol manufacturers version of a
runtime configuration file has it's own specific syntax and functionality.
So, find out the manufacturer of your Cobol compiler and contact them to
get their runtime configuration variable settings documentation.

Second Possibility: Your professor is teaching you the "runtime
configuration" methodology.
He may be showing you how to create your very own "runtime configuration"
files that you the programmer use in your code. In other words, he doesn't
want you to use the Cobol manufacturers version of a "runtime configuration"
file, but instead he wants the students to use the concept of a "runtime
configuration" file to help you understand how to vary the default behavior
of your programs.
For instance, using the FILE-SUFFIX example above, it is possible for
you to use the same concept to come up with your very own configuration file
that perhaps uses FILE-EXTENSION = DAT. In your Cobol program you would
open up your configuration file, find the FILE-EXTENSION setting and then
string the DAT extension and the file name together. You could actually
write a program whose only job is to load the configuration settings into
EXTERNAL or GLOBAL memory for reference by the rest of running programs.
Thus, each program does not have to read and decipher the configuration
file, but instead just reference the configuration variables pre-loaded in
memory.
The "runtime configuration" methodology is very popular in Cobol
applications because it allows programs to use customizable defaults and it
can save alot of extra work for the programmer. You must be careful not to
"over configure" your programs or they may become hazardous and riddled with
quirky glitches.

I hope this helps you Andrew. Stick with it. Hard work always pays off.
God bless.
