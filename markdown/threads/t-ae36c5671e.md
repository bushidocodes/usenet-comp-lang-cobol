I found the following phrasing in the IBM Enterprise COBOL language
reference:

The data description entry for identifier-1 should contain an OCCURS
clause with the INDEXED BY phrase, but a table can be searched using

I couldn't find any example describing how this is supposed to work?

Sure, you can have any number of tables of the same "shape", and only ONE of 
them needs to be indexed. (I would strongly discourage you from doing this, 
it isn't recommended practice, and just because you CAN do something does 
not mean that you SHOULD... :-))

01 Table1.
      12 tab1Entries occurs 30
           indexed by t1-x1.
           15 t1-part1   pic x(10).
           15 t1-part2    pic x(20).

01 Table2.
      12 tab2Entries occurs 30.
           15 t2-part1   pic x(10).
           15 t2-part2    pic x(20).


01 Table3.
      12 tab3Entries occurs 30.
           15 t3-part1   pic x(10).
           15 t3-part2    pic x(20).


Then you can say...

      TAB1-search.
              set t1-x1 to 1
              search tab1entries
                   AT END
                          some action...
                   WHEN
                          some action...
              end-search

Note the search makes no reference to the index...

Now search Table2 (which has no index defined for it...)

      TAB2-search.
              set t1-x1 to 1
              search tab2entries
                         varying t1-x1
                   AT END
                          some action...
                   WHEN
                           t2-part1 (t1-x1) = ws-search-key
                                move t2-part2 (t1-x1) to t3-part2 (t1-x1)
              end-search

Note the same Table1 index can reference Table2 AND Table3, but, in my 
opinion it is foolish to do it.

While this is legal, it is not pretty and it very quickly becomes confusing. 
For the sake of an "indexed by" clause, make sure that all your tables have 
their own indexes... it really is much easier.

Pete.
