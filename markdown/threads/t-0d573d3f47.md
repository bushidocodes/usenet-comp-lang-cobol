>I have a basic question : Is it possible to create procedures in Db2

Yes, they are called Stored Procedures, same as Oracle, et al. They might or
might not speed things up. 

>We are using Neon ODBC driver to acess DB2 from Asp pages, but we
>are expiriencing some performance problems with dynamic sql's
>(select xx from table ...). Is it possible to transform this queries
>in compiled queries ?

To do this, you must have installed the gateway Client Application Enabler
(CAE). You allocate a statement handle, do a SQL Prepare to compile the
procedure, a SQLBindParameter, then a SQL Execute for each execution.

You can get the same result by avoiding Stored Procedures and Prepare-ing your
own SQL source. Even if you don't, DB2 is _supposed_ to cache previously
executed SQL (called Dynamic Statement Caching) across a Commit. Don't start
with a solution until you know what the problem is. Are KeepDynamic and CacheDyn
set to Yes? Is MaxKeepD big enough? Have you looked at a DSC monitor? It may
have nothing to do with SQL compilation. Are you selecting a non-indexed column?
Are you doing awkward sub-selects? Have you looked at an EXPLAIN PLAN?

You thought the slow performance was caused by compiling SQL (on every access?)
and the problem could be solved by making it a Stored Procedure, which you
thought would be pre-compiled. 

Stored procedures are no longer pre-compiled. All SQL is compiled (actually,
tokenized) when you issue a PREPARE or when you use it in a SELECT or EXECUTE.
PREPARE is a nasty thing that's best avoided. It complicates your client program
logic and makes the SQL cache inefficient. It's almost always unnecessary.

Compilation usually takes less than a millisecond. It's not a performance issue
unless the SQL is recompiled every time it's used. To prevent this from
happening, the server normally caches the compiled SQL. If you use the exact
same SQL while it's still in the cache, the server does not compile it again, it
uses the cached code. 

To find out how much re-use you are getting, ask the DBA to run a DSC (Dynamic
SQL Cache) monitor on your server task, or give you permission to run it from
your client.  Re-use percentage should be >99%. If it is, your problem is not
compilation; you'll have to look elsewhere using classical techniques such as
EXPLAIN PLAN. It is the DBA's job to help with this.  If re-use is low, have the
DBA see whether KeepDynamic and CacheDyn are turned on for your task or region.
The first tells it to do caching at all (if off, recompiles will be 100%); the
second tells it to retain cached SQL across COMMITs. If both are on and
recompilations are still high, that means the cache is too small (or you're
using truly dynamic SQL). This is unlikely. MaxKeepD was the old way of setting
cache size. Default was 300 SQL commands, which is plenty. With DB2 version 6,
they moved the cache out of your local server memory to some global pool. The
DBA will know how to tune caches.
