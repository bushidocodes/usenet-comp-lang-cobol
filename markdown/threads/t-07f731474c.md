Sathaporn asked me to post SMF Type 30 record descriptions in COBOL.
Here is the basic definition:
The COBOL program contains this statement:
     COPY SMFTYP30.
This COPY member contains the following:
*    THE FOLLOWING DATA ITEMS ARE REFERENCED IN THE COPY MEMBER,
*    BUT CANNOT BE CODED INSIDE IT:
 01  SMF30ACL-LONG.
  05                PIC X(01).
  05 SMF30ACL-LO    PIC X(01).
 01  SMF30ACL-N     REDEFINES
     SMF30ACL-LONG  PIC S9(4) BINARY.
*    THIS IS A DESCRIPTION OF SMF RECORD TYPE 30 WHICH IS
*    SUITABLE FOR USE IN A PROGRAM'S WORKING-STORAGE.
*    IT INCLUDES THE DATA THAT CAN BE ADDRESSED IN THE "FD" AND
*    THE RECORD HEADER THAT COBOL NORMALLY HIDES FROM THE
*    PROGRAM.
 01  SMF-TYPE-30.
*    THE "VARIABLE LENGTH RECORD HEADER" IS INCLUDED IN THIS
*    DESCRIPTION BECAUSE THERE ARE NUMEROUS OFFSETS IN THE
*    RECORD THAT ARE RELATIVE TO THE TRUE START OF THE RECORD,
*    AND IT WILL BE EASIER TO ADDRESS THE DIFFERENT SEGMENTS
*    IF THE OFFSET VALUES DON'T HAVE TO BE MANIPULATED.

  10 SMF30SEG                   PIC S9(4).
  05 SMF-30-RECORD.

I used an Assembler Language program to read the records, so that I
could get the variable length record "control word".  COBOL would hide
the record control word from the program.

This program was called with the level 01 data-name and a FILE STATUS
item.  It emulates the way COBOL updates file status items, setting
values 35 for an unsuccessful open, 10 for end of file, or 00 for
successful operations.
Here is the Assembler program:
GET      TITLE 'SMF READER ROUTINE CALLED BY "SMF98" COBOL PROGRAM'
SMF98GET CSECT
         PRINT NOGEN
SMF98GET AMODE 31
SMF98GET RMODE 24
GET      CSAVE R12,RETCODE
         L     R6,0(R1)        RECORD AREA ADDRESS
         L     R3,4(R1)        FILE STATUS ADDRESS
         USING FILESTAT,R3
         MVC   FILEST,=C'00'   ASSUME FILE WILL PROCESS WITHOUT ERRORS
         SPACE
         LA    R9,SMFINPUT     DCB ADDRESS
         USING IHADCB,R9
         TM    DCBOFLGS,DCBOFOPN FIRST TIME?
         BO    READREC
         OPEN  SMFINPUT
         TM    DCBOFLGS,DCBOFOPN OPEN SUCCESSFUL?
         BO    READREC
         MVC   FILEST,=C'35'   PASS BACK AN ERROR FILE STATUS
         LA    R15,12
         ST    R15,RETCODE     SET RETURN CODE = ERROR
         B     EXIT
         SPACE
READREC  BAL   R11,READRECS
         SR    R15,R15
         ST    R15,RETCODE     SET RETURN CODE = NORMAL
EXIT     EXIT  GET
         SPACE
READRECS GET   SMFINPUT        LOCATE THE NEXT LOGICAL RECORD
         LR    R8,R1
         LH    R9,0(R8)
         LR    R7,R9
         MVCL  R6,R8           RETURN RECORD TO CALLER'S AREA
         BR    R11
EOF      MVC   FILEST,=C'10'   RETURN END OF FILE STATUS,
         CLOSE SMFINPUT        CLOSE SMF FILE
         FREEPOOL SMFINPUT
         BR    R11
         DS    0D
RETCODE  DC    F'0'
         SPACE
         DS    0D
SMFINPUT DCB   DDNAME=SMFINPUT,DSORG=PS,MACRF=GL,RECFM=VB,EODAD=EOF
         DCBD  DSORG=PS,DEVD=DA
FILESTAT DSECT
FILEST   DS    CL2
         END
The record control word is not required in my program to process the
Type 30 records.  Therefore, the COBOL program could read the records
directly.  If this is done, FUNCTION LENGTH or the special register
LENGTH OF would be useful for determining the current record's actual
length, so that it could be moved from the FD to Working-Storage.  The
record would be moved to the Level 05 field named SMF-30-RECORD.

My program is used to obtain CPU time and EXCP counts, in order to
identify job steps that consume lots of resources.
Here is the relevant code for processing a type 30 record (subtype 4,
which is a "total" record):
 PROCESS-TYPE-30-RECS.
     IF       SMF30RTY                     =  X'1E'
       ADD    1                            TO TYPE30-COUNT
       IF     SMF30STP                     =  4
              ADD      1                   TO T30ST4-COUNT
              PERFORM  CREATE-CPU-TIME-RECORD
       END-IF
     END-IF
     PERFORM  READ-SMF
     .
 CREATE-CPU-TIME-RECORD.
     COMPUTE  CPU-SEGADDR                  =  SMF30COF + 1
     MOVE     SMF-TYPE-30(CPU-SEGADDR:SMF30CLN)
       TO     SEPARATED-PROCESSOR-SECTION
     IF       CPU-MINIMUM                  >= SMF30CPT
              ADD     1                    TO CPUMIN-COUNT
     ELSE
      MOVE    1                            TO SMF-TIME-COUNT
      COMPUTE SMF-EXCP-VALUE(1)            =  SMF30TEP(1)
      COMPUTE SMF-TIME-VALUE(1)            =  SMF30CPT
      MOVE    SMF-TIME-VALUE(1)            TO SMF-MIN-TIME
                                              SMF-MAX-TIME
      MOVE    SMF30SID                     TO SMF-SYSTEMID
      MOVE    SMF30JBN                     TO SMF-JOBNAME
      MOVE    SMF30STN                     TO SMF-STEP-NUMBER
      MOVE    SMF30PGM                     TO SMF-PROGNAME
      COMPUTE SMF-DATE                     =  SMF30DTE + 1900000
      MOVE    SMF30TME                     TO SMF-TIME
      IF      SMF30PSN                     =  SPACE
        MOVE  SPACE                        TO SMF-STEPNAME-IN-PROC
        MOVE  SMF30STM                     TO SMF-STEPNAME
      ELSE
        MOVE  SMF30STM                     TO SMF-STEPNAME-IN-PROC
        MOVE  SMF30PSN                     TO SMF-STEPNAME
      END-IF
      PERFORM WRITE-CPU-TIME-REC
     END-IF
     .
Note the use of Reference Modification to get the portion of the record
that contains the CPU data into a place where it can be addressed.  The
starting address is computed, and the length is taken from one of the
fields in the fixed part of a type 30 record.  The
SEPARATED-PROCESSOR-SECTION looks like this:
 01  SEPARATED-PROCESSOR-SECTION.
     COPY SMF30C.
The field names and their contents are explained in IBM manual "MVS
System Management Facilities", which is in the "MVS Unlicensed
Bookshelf" for OS/390 or z/OS.

Happy coding!
