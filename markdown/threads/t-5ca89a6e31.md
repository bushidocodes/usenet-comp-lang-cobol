A few years back (well, it was 7 to be precise, but time does fly when
you're having fun...) I was struggling to persuade people here that there
were benefits in Object Orientation for COBOL and that it really wasn't too
hard to convert existing standard COBOL into OO COBOL. The general consensus
in this community was that it was unnecessary and just modular programming
re-invented (which it isn't), and it would all pass as fashions do and
everyone would go back to doing it all in COBOL like they always did...

It seemed to me it would be a good idea to create an example application and
show the process of bringing that application into the modern world. Robert
Wagner wrote a very nice and useful COBOL engine which allowed you to get
offsets and lengths for fields in a COBOL buffer (similar to the report
produced by certain passes of certain compilers, but it isn't always
convenient to have to compile a whole program when all you want is some
information about a data buffer in it.

Anyway, we took Robert's application and I converted it to OO COBOL where
it became a COM Component. (COBSTRUCT). I then took the COM component and
wrote a C# GUI executable to interface it to the user and the end result
(COBDATA) was placed on the PRIMA web site, along with all the sources
(COBOL and C#, and a description of the process involved in creating it. It
was a free download intended to show that it is really pretty easy to
convert standard COBOL to OO COBOL, and just one of the benefits of doing
this is that your COBOL application can play nicely with other languages
(there are many other benefits in producing objects, but too much to go into
here and we've kind of thrashed it to death here over years anyway.)

Several hundred people downloaded the Tool from the PRIMA web site.

Unfortunately, while I was doing other development for PRIMA I found that,
despite our having tested this component fairly rigourously, it was flawed.
You can imagine how it feels to be in the middle of a critical development
and realize that one of the lynch pins for that development is broken...

I patched Robert's code to get my project completed but resolved to
investigate and properly fix it as soon as I could do so. This component is
used internally for a number of PRIMA products and it needs to be right...

I'm pleased to say that this exercise has now been completed and there is a
new COBSTRUCT engine. It is still a COM component that can be called from
any language (that supports the Component Object Model; most do...) but it
is now written in C#. Apart from the advantages of being a component, this
means that it can be deployed without needing a COBOL run time system. I
took the opportunity to enhance the original COBDATA wrapper I wrote for the
engine while I was at it, and the new tool has a number of features that
the old one didn't have. For non-COBOL programmers required to deal with
COBOL records, this tool can provide easy access to fields in a data buffer;
for COBOL guys it is handy to be able to cut or copy all or part of a record
layout in COBOL, activate COBDATA and instantly get the "counts" for the
snippet you copied. (I keep a minimized copy of COBDATA on the taskbar if I
am editing COBOL).

I have been quite amazed to see what some people have done with COBSTRUCT
(the engine from the tool, running as a separate component in their own
applications). One client migrated their indexed files to RDB using our
Toolset, then wrote a VB.NET GUI front end that called the DAL to get a
COBOL record, then used COBSTRUCT to get offsets and lengths for each field
in the record so the fields of interest could be placed on GUI screens. In
effect, a generalised data display facility.

The LINQ DAL facility we have just completed uses COBSTRUCT to provide a
"Data Dictionary" which includes everything about every field in the tables
of a given table set on the RDB, as an XML file. This file is used by the
DAL generator to provide hard coded references when transporting data to and
from the RDB. Once this C# code is built to become the LINQ DAL object, it
is very much faster than using reflection to do the job.

I wrote to everyone who registered on the PRIMA web site and advised them
they can have a free replacement of their existing COBDATA tool.

I'm making the same offer to people here.

For more details about the tool and to get the FREE download:
http://primacomputing.co.nz/primametro/newcobdata.aspx

Obviously, if anyone finds any bugs in this, I'd be grateful to have them
reported to me. The new version can be automatically updated (the old one
couldn't) so it should be simple to deploy fixes and enhancements.

Pete.
"I used to write COBOL...now I can do anything."
