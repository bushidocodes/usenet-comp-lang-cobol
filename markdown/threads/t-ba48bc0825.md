```
As part of the current overhaul of DAL code I am looking at sequential 
reversed processing.

My understanding is that for ISAM files there are 2 ways you can read them 
backwards:

1. START... REVERSED  (where this is supported) (A subsequent READ NEXT gets 
the previous record)

2. READ PREVIOUS (or PREV) as an equivalent of  READ NEXT. (START in the 
usual way (without REVERSED) then READ PREV to go backwards from the START 
position.

As the management of SQL cursors across threads tend to be tedious I don't 
really want to have to use one for reading backwards (I don't have much 
choice for reading forwards...). I can get the previous record in sequence 
with a singleton select.

The following COBOL extract from the generated DAL code shows how:

*> Key index is OK so set host variable from key field...
*> "KOR" means "Key of Reference"...
     move KOR-offset (KOR-x1) to K
     move KOR-length (KOR-x1) to L
     move MOST-data-buffer (K:L) to HV-DynamicParm
     move spaces to SQLBuffer
     string
        'SELECT MAX(['
            delimited by size
        keyname (KOR-x1)
            delimited by space
        ']) FROM [ANYCO1-REC]  WHERE ['
            delimited by size
        keyname (KOR-x1)
            delimited by space
        '] < ? '
            delimited by size
              into SQLBuffer
     end-string
     EXEC SQL
        PREPARE  tmpHandle from :SQLBuffer
     END-EXEC
     EXEC SQL
        EXECUTE tmpHandle  INTO :HV-KEY-WORK USING :HV-DynamicParm
     END-EXEC
     *> zero SQLSTATE just means the EXECUTE was OK... it does NOT mean
     *> that data was actually retrieved.
     if SQLSTATE  NOT = '00000'
        move SQLSTATE to MOST-SQLSTATE
        move SQLMSG   to MOST-SQLMSG
        invoke self 'TranslateFileStatus'
        end-invoke
        move MOST-interface-block to datablock
        exit method
     end-if
     if HV-KEY-WORK (1:L) = low-values
        move "No data found or EOF. " to MOST-SQLMSG
        move '02000' to MOST-SQLSTATE
        invoke self 'TranslateFileStatus'
        end-invoke
        move MOST-interface-block to datablock
        exit method
     end-if
     move HV-KEY-WORK (1:L) to ANYCO1-REC (K:L)
                               MOST-rel-data
     move ANYCO1-REC to  MOST-data-buffer
     move MOST-interface-block to datablock
     INVOKE SELF "ANYCO1MTGetRandom"
            using datablock
     END-INVOKE
     .
 p001-end.
     exit method.

The above code uses the key of the record currently in the data buffer to 
dynamically generate a query that gets the key of the previous record. The 
last three lines then use this key to fetch the record randomly. (It 
actually works very well in practice but it is predicated on the assumption 
that there won't be much demand for reversed processing.)

As Fujitsu supports START... REVERSED, Alistair has used it in his AnyVu 
program which is a kind of  DAL exerciser. It is quite elegant; he can flip 
direction forwards or backwards using a START, and all the coded READ NEXTs 
continue to work without change.

Unfortunately, as the Toolset only implements reversed processing using READ 
PREV the Werewolf version of AnyVu simply doesn't work for reversed 
processing.

I had never seen reversed processing implemented in this way until I saw 
Alistair's code. I like it very much and I'll probably implement support for 
it into the Toolset, but before I do I wanted to get any thoughts and 
observations you may have about reversed sequential processing.

Have you ever implemented reversed sequential processing for indexed files 
in COBOL? (even if you haven't please feel free to comment...)

Why was it necessary?

Do you think there is widespread demand for it? For example, would you NOT 
buy a COBOL compiler that DIDN'T have START... REVERSED? Even if it 
implemented READ... PREV?

Are there other ways of reversed processing you have seen used that are not 
mentioned here?

I'm looking for general feelings, experiences, and opinions, as well as any 
specific instances people care to mention.

Cheers,

Pete.
```

```
I used CICS for many years (and even got to work at Hursley Park, where it 
was invented... :-)) and there is a GET PREVIOUS facility in it. You would 
issue a CICS STARTBR using the key of reference, just as you might issue a 
START for COBOL. Follow it with a READPREV instead of a READNEXT and the key 
of the record retrieved is returned in the RIDFLD option of the Command.

The Toolset in its current incarnation does not run on mainframes so it does 
not handle CICS. (It probably isn't fair to say it doesn't handle CICS 
because the presence of CICS (either Assembler level or Command level) will 
not cause it to crash. Those statements just remain untransformed and so 
will execute exactly as they have always done.)

(It would be quite feasible to use it to create a normalised, optimised, DB2 
database from the existing COBOL copybooks, and generate load programs to 
read the VSAM/KSDSs and write the new database. The generated COBOL for the 
load programs would compile and run on the mainframe without problem. 
However, as the mainframe doesn't support Objects in COBOL, as far as I 
know, there would be no possibility to use any created DAL object layer, and 
I don't think the DAL objects would compile successfully on a mainframe. The 
Load process works OK because the generated code is standard procedural 
COBOL; the DAL is object COBOL.)

Pete.
```

```
I have used skip sequential processing on indexed files but never backwards

Fortunately modern RDBMS are VERY good at doing random access and the above 
code executes pretty much as quickly as a Cursor moving forward. (Actually, 
they are both so fast it isn't a problem...In some tests I ran a few years 
ago the cursor was returning 900 records a second and the singleton around 
820. The latest version of SQL Server may have improved on this.) My 
objection to using a cursor for reversed reading is that the DAL object is 
maintaining separate threads and it may already have a forward cursor open 
when someone decides to start a reversed one. Add to this random updates and 
accesses on GET and PUT threads and the whole thing gets a bit messy. (It is 
far from impossible; I just don't want to do it. It seems reasonable to me 
to offer ONE direction of sequential access at any given moment although it 
would be quite feasible to implement both directions and even multiple 
cursors in each direction, all connected to the same application program.)

I have a feeling that reversed processing is not used a lot. I have never 
worked on a site where it was used and I have only one client who uses it. 
(And they don't use it extensively; it is to manage time dependent data 
where they need to show data chronologically and for various reasons they 
don't want to sort it.) But that's why I'm trying to get a feeling from this 
community.

Pete.
```

```
Consider a stripped-down version of a Customer Order File which looked 
something like this:

01  C03452-CUST-ORDFIL.
    05  C03452-KEY.
        10  C03452-CUSTNO PIC X(10) VALUE SPACES.
        10  C03452-ORDDT  PIC 9(06) VALUE ZEROES.
        10  C03452-ORDTM  PIC 9(06) VALUE ZEROES.

... and assuming that a given customer didn't enter more than an order a 
second (and given the systems of the time... they didn't) what would 
result is a dataset containing, for any given customer, the most recent 
order first.

Now... if someone wanted historical data in descending date-time sequence 
there'd be a few ways to do it.  One would be an extractive sort with some 
fields specified ASCENDING and others DESCENDING, eg 

(INCLUDE COND=1,10,CH,EQ,'NYC0010033'),
(SORT FIELDS=(1,10,CH,A,11,12,CH,D)

... to give an historical record of that customer's purchases.  The other 
method - the one that still give shivvers - is to include as part of the 
processing the inelegant kludge of

SUBTRACT WK-C03452-ORDDT FROM 999999 GIVING WK-C03452-ORDDT
SUBTRACT WK-C03452-ORDTM FROM 999999 GIVING WK-C03452-ORDTM

... and write the results out to a temp datatset for processing.

Good thing nobody know what went on back then, eh?
```
