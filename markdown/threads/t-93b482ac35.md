```
We are migrating to Oracle 9.2 from 7.3 and the COBOL programs are not
behaving the same.   The old programs (.exe) that accessed 7.3 can
access 9.2 and produce identical results.  But, when the code is run
through the 9.2 precompiler, the new code is not retrieving as much
data as the old exe's.  Also, the reports out of the new process are
not in the same sequence as the old program...one is in ASCII sequence,
the other is in EBCDIC sequence.  I've tried manipulating the compiler
option "NATIVE" but to no avail.  We were getting 1405's and turned on
the UNSAFE_NULL option.  That got rid of the 1405, but we still can't
match the output of the old code. These programs were written back in
the late '80s and the COBOL expertise has been long departed. I think
the older programs were precomp'ed with DBMS=V6....which is no longer
supported, I think.  So....if anyone has any sugestions for us to try,
it would be greatly appreciated...

Thanks
```

```
Do you still have the "old system" (compiler and precompiler) available?  If so, 
I would do two compiles with the SETTINGS directive turned on (If this is Micro 
Focus compiler).  Then compare what directives are being used in the two 
versions of the compiler.  My *guess* is that the newer system is "forcing on" 
some directive that you want to override.  (Again, assuming Micro Focus - 
CHARSET might be one of them)
```

```
You don't need Cobol expertise, you need SQL expertise. You need to
stop experimenting with  precompiler options and start dealing with
the actual SQL statements. There are probably one or two differences
affecting all programs. Once you identify them, you can fix the source
with a script, text editor, precompiler option or table definition.

I'd start by turning logging on (if off) and comparing old vs. new
SQL.
```
