There is also a product called Ratrix Controls which allows access to
MF-Cobol Files via EXTFH.
They create an object called CobolFile which you drop into your Visual Basic
form. This object has all of the necessary properties to describe the file,
i.e. path/filename, filetype etc. The object also has methods to perform all
I/O operations, OPEN,CLOSE,START (all starts), READ NEXT,PREVIOUS,WRITE etc.
(everything)
It works with all filetypes, Indexed,Relative,Sequential.
They also provide a code generator which reads the file's FD and generates
the VB functions to convert Cobol data-types to VB data-types and
vice-versa. When you execute a method, START for example, it makes a call
to EXTFH, returns the record to VB and converts the data from Cobol to VB
types.
Advantages are :
1. Very fast, it access the files from VB via the same function that Cobol
would use, no ODBC overhead
2.You process the files with the same paradigm, same kind of logic you would
use in Cobol, not through DAO, recordsets, etc.
Disadvantage: (as of about a year ago)
1 You are limited to VB. Does not work with Access, Powerbuilder etc..
However, you might be able to compile VB function to a DLL and call from
other languages.
2. Cannot bind data to controls, but they said they were working on that.
Might be done by now.

I experimented with it, and found that I works as advertised, tech support
was pretty good.
If anybody wants the contact info, let me know and I will look it up.
Regards
Walter Levine

I just converted a 50,000 line system from DOS MF to
windows Liant. I first wrote programs in MF to dump all
the files to flat ASCII, and rebuilt the Isam in Liant.

I converted all the code, and replaced all the menus
with toolbars. As well, I revised the screen sections
so that the subsections ran in separate windows that
"opened and closed" windows fashion. That was
reasonably simple, as I was able to automate it.

I left all the basic screen I/O as part of a standard
screen section, so the code should run just about
anywhere and be convertible to just about any Cobol.

The project went quite smoothly, took about 5 weeks
