Steve, first "normalize" the number using Format$() to get
a string representation of the number which has a consistent
number of digits and the radix (decimal) point in a known
position.

It is then a fairly simple matter of processing the string,
two characters (digits) at a time, using the Chr$() function
to assemble a string representation of the COMP-3 number.


Take care to test for error conditions; double precision is
not a great fit for COMP-3. If you have a choice, the Currency
type is much better.

First of all, I notice that your input involves some numbers
that have more precision than is available in the Currency
VB data type (the COBOL picture equivalent for which is
S9(15)v9(4)). Using Double should not be a problem for
the digit counts you describe (Double cannot represent 18
digits of accuracy).

I would suggest the following metacode example in VB (I'm
going to leave out at least the Dim statements):

Function ConvertToCOMP3 (TheValue as Double, Precision as Integer,
RightOfDecimal as Integer, IsSigned as Integer) as String

; Let VB do some of the work for you...
sFormat = "0"
if RightOfDecimal > 0 then sFormat = sFormat & "." & String$(RightOfDecimal, "0")
sValue = Format$(TheValue, sFormat & "+;" & sFormat & "-")
; sValue now contains TheValue as a string in the form ddd.dddd+
if RightOfDecimal > 0 then
sInteger = Left$(sValue, Len(sValue) - (RightOfDecimal + 2))
sInteger = sInteger & Mid$(sValue, Len(sInteger) + 1, RightOfDecimal)
else
sInteger = Left$(sValue, Len(sValue) - 1)
endif
If Precision > Len(sInteger) then sInteger = String$(Precision - Len(sInteger) & sInteger
If (Len(sInteger) Mod 2) <> 1 then sInteger = "0" & sInteger
; sInteger now contains the desired number with leading zeros
; while the sign is hanging around at the right end of sValue
; convert to COMP-3
s = ""
for i = 2 to len sInteger step 2
HiNib = Asc(Mid$(sInteger, i - 1, 1)) - 48
LoNib = Asc(Mid$(sInteger, i, 1)) - 48
s = s & Chr$((HiNib * 16) + LoNib)
next
; now put the low order digit an sign on it
HiNib = Asc(Right$(sInteger, 1) - 48
LoNib = 15
If IsSigned then
If Right$(sValue, 1) = "+" then LoNib = 12 Else LoNib = 13
EndIf
s = s & Chr$((HiNib * 16) + LoNib)
ConvertToCOMP3 = s
End Function

WARNING: THIS IS CODED OFF THE TOP OF MY HEAD AND HAS NOT BEEN TESTED!
But it should get you close to what you need.

Good luck!

Hmm, You're moving a file from a PC to the mainframe and need to
change a floating-point numeric to packed-decimal. If your PC app can
handle it, why not change the floating-point to character with the
sign external and port the result to a waiting intermediate mainframe
app that'll read the number as unzoned-numeric with sign external and
pack-it for use in the final app?

If that's not practical, you could find out whether the PC app is
using single- or double-precision floating-point fields, purchase a PC
COBOL compiler, write a program to read the file with floating-point
fields and move the fields to a file with packed-decimal fields. Since
you said the mainframe is IBM-compatable, the IBM COBOL extensions for
single- and double-precision fields are:

01 single-precision-4-bytes comp-1.
01 double-precision-8-bytes comp-2.

Note there are no PICs with these fields. Odds are the PC COBOL
compiler extensions will be similar since they try to stay compatable
with the mainframe version. If you're worried whether the
packed-decimal fields gets scrambled in the file transfer, I once
wrote a PC COBOL app that read packed-decimal fields downloaded from
my mainframe and had no problem. If that doesn't work, you always can
convert to a zoned-decimal format with the PC COBOL app then port to
the mainframe for final conversion to packed-decimal.
