A new thread with "truth" vs "rumor" (or memory)

1) All releases of COBOL since VS COBOL II, R1 (c. 1984) have FULLY
supported both dynamic and static CALL statements in COBOL programs run
under CICS.

2)Before that, the documentation indicated that some CALLs were
unsupported - and there was no documented way to CALL a subprogram that
issued EXEC CICS statements.

3) With VS COBOL II (mid-80's - almost 20 years ago), IBM introduced a
"separate" run-time system for some (not all) COBOL functions.  This
continues thru today with LE.

4) For at least the last 10 years, IBM has documented the advantages (in
SOME situations) of using CALL statements over EXEC CICS LINK statements.
On the other hand, EXEC CICS LINK statements are *still* required for some
ILC (Inter-language Communication) situations under CICS.

5) With OS/VS COBOL (and earlier), there were several COBOL statements that
were required O/S versus CICS managed "getmains"  (e.g. INSPECT).  With VS
COBOL II (again mid-80's) IBM provided run-time routines that used
CICS-appropriate run-time support for these COBOL statements - and they have
been fully supported for almost 20 years now.

6)  No release of "mainframe" (MVS, VSE, OS/390, z/OS) CICS has supported
"traditional" COBOL access to files, e.g. Select/Assign, OPEN, CLOSE, READ,
WRITE, REWRITE.  In stead, there are CICS provided facilities that *MUST* be
used to perform such operations on any type of mainframe file (as opposed to
database - Use of EXEC SQL, EXEC DLI, CALL "CBLTDLI", etc provide database
access)

7) As a *minor* exception to the statement above, IBM provided (since the
late 80's) an interface to the COBOL *SORT* statement.  IBM, itself, does
*not* provide any support for SORT under CICS - but does document the
run-time interface that may be provided by 3rd party vendors or in-house.
Also, for about the last 3-5 years (I don't remember the exact release), IBM
added support for the COBOL DISPLAY statement - which uses CICS services to
send output to an appropriate CICS resource.

    ***

From an historical point of view, I know of several shops that violated the
DOCUMENTED restrictions on using CALL statements or O/S-getmain-related
statements with OS/VS COBOL or DOS/VS COBOL and CICS.  *ALL* such sites
(that I know of) had SERIOUS migration problems when moving to either the VS
COBOL II or LE run-time environment where such violations of the documented
restrictions lead to run-time problems or other serious side effects.

And finally, as far as I know (and I certainly could be mistaken on this),
CICS is *still* one of the best performing and most heavily used
"transaction processing" systems in use today.  With its current interfaces
to the web, ATM and similar terminals, and a variety of other I/O systems,
its power and robustness is not questioned by either its supporters or
detractors.

I have a few observations on CICS. (I feel qualified to comment, having used
it (only on mainframes) for around 10 years.)

I also worked for IBM (on contract) at Hursley park (where CICS was
invented) and had coffee with one of the original team that wrote it...does
that count <G>?

1. CICS is a VERY stable Transaction Processor.

2. It is NOT the best transaction processor that has ever been available,
but it is certainly one of the oldest still available.

3. IMS/DC is/was a better transaction processor but it lost popularity due
to its unfortunate association with IBM's hierarchical database offering.
(Sadly, IMS was also a perfectly good database system but the superiority of
Relational over Hierarchic has gradually established itself...there are
still many fierce debates in pubs about this, though; I was in one recently
<G>)

The fundamental requirement for ANY transaction processor is the ability to
store "terminal dependent " data across dequeue boundaries. In other words,
the system can carry on processing other threads while a user is thinking or
entering data at his terminal (which is said to be "dequeued"). When he hits
Enter, the thread "reawakens" and obtains the previously stored data from
that particular terminal. In this way, the "conversation" continues as a
logical chronistic stream. (The difference between the way that CICS
achieves this and IMS/DC achieves it, is the basis for my statement that I
prefer IMS/DC above. It is also the basis for the terms
"pseudo-conversational" and "conversational" when it comes to discussing
these things, but as these same terms have different meaning to an IMS or a
CICS programmer, it causes more arguments than it resolves...<G>)

Anyway, for around 20 years I worked in both these environments and came to
know and understand them fairly well.

Then I moved to Client Server and am currently involved in constructing a
Web Portal that also has to manage multiple threads across multiple
"terminals" (actually, individual Browsers on individual clients). I was
pleased to find that, as expected in the Object Oriented world of modern
programming, there is something called a Session Object that I can use to
store "terminal dependent" data in, just like you would with CICS DFHCOMM
area or IMS/DC's SPA. (I am using ASP (amongst other things...JavaScript,
HTML, and COBOL for the CGI/ASAPI code) and this object is available in this
environment).

I was pretty excited, as to me, this represented "familiarity" with what I
would expect. I gratefully started stashing various variables in the Session
Object.

To my horror, I found that this object was unstable! As soon as I ran a
COBOL CGI program the Session Object was reset, as the system considered it
was now in another session...<G>! I frantically phoned a few people who know
a lot more about this environment than I do (and are not blinkered by having
worked for decades on mainframes <G>) and they made noises like the
mechanics do when you take your car in for what you hope will be a standard
service, but there is much groaning and chin rubbing and a general feeling
of despondency... Well...yes, we've had problems with the stability of the
Session Object (even though we're not using much CGI/ASAPI and certainly not
COBOL)...it's a bit of a problem sometimes...MicroSoft have some advice,
here's a URL...and a NewsGroup... (I checked out the newsgroup and found
many posts complaining about "lost" Session Objects. MS solution was to use
cookies...(not everyone accepts cookies so this is not an ideal
solution...)).

By now I was devastated, but resolved to solve the problem myself. So I
wrote a version of "CICS" that runs on the WEB in a client server
environment and is stable...(Well, of course, I didn't REALLY write CICS,
but I built a mechanism that works the same way CICS does and appears to be
stable. It certainly ensures that threads running on the target Web Server
do NOT lose data across dequeue boundaries...<G>) And it DOESN'T require the
use of client cookies (which is one possible solution to the problem), or
storing stuff directly in TCP/IP Headers (which is another solution to the
problem).

Then a couple of days ago I made an architecture change and needed to
regenerate some nested CGI programs with the Fujitsu Web Software. Not being
particularly stressed, I took my time and noticed some check boxes I had
never used before... One of them said "Clear session object..." It was the
default...<G>

The Moral of this story is that sometimes under pressure it is easy to miss
something that may save you a HUGE amount of heartache later. When you are
using complex tools, it is worth investing some time in getting to know them
THOROUGHLY before rushing to war with them...<G>

The NEXT Web Portal I do might use the Session Object; this one will use my
"CICS simulator". It hasn't been tested under fire yet, but in simulation it
behaves very well.

So, there is a place for CICS knowledge even in this day and age...<G>

I agree that CICS will be with us for some time yet, Bill.

Pete.

Each CICS region runs under it own task control block (TCB) and CICS
provides multitasking for the transaction ids being executed.  The reason
some COBOL facilities were not available under CICS is because they issued
a supervisor call (SVC).  This SVC was a direct request to the operating
system for services, bypassing CICS, and in fact it would cause the entire
CICS region to go into a wait state until the SVC request was complete.
This was extremely NOT GOOD!  Well at least it used to be that way, today
the preceding would be considered somewhat of a simlpification.  In time
some of these restrictions have relaxed, and now we have terminal owing
regions (TORs), appication owning regions (AORs), function shipping etc.

There are very many problems with forms and CGI.  With Cookies at least the 
latest is sent, when an updated Cookie is sent to the client you can be 
sures that an old one will not reappear, they also expire.  Putting hidden 
fields in the form is also a partial solution but what happens when the 
user does a few 'back's and resends an old form ?

I use both Cookies and hidden fields and generate a new hidden code for 
each new form (where appropriate) and store these in a session database. 
When a hidden code is 'used up' by the form returning and the data being 
processed correctly, I delete that hidden code so that a resending of the 
same form can be rejected as 'already processed'.  This saves duplication.

I suspect CICS wouldn't have this problem as the screens get overwritten 
and a user cannot resend an old one.

Modern browers cater for having lists of sites where Cookies are allowed or 
disallowed.  This can be built up by having an 'ask' flag set.  My own 
browsing is done with Cookies off but certain sites are set to allow them.
