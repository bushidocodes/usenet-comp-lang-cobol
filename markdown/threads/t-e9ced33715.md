```
I know next to nothing about Cobol, and I've always felt like it was
the most important language missing from my language bag.  I'd like to
fix that.

First: What's a good book to read to learn the language?  I've been
programming for about twenty years in many, many languages, so I don't
need a book about how to program; a language reference manual might be
the right thing if it were clear and well-organized.

Second: I was talking to a friend of mine about Cobol, and I asked him
to show me typical Cobol code to turn an input like this

	Columbus      Ohio
	Frankfort     Kentucky
        Cleveland     Ohio
        Trenton       New Jersey
        Louisville    Kentucky
        Cincinnati    Ohio

into an output like this:

	Ohio          Columbus, Cleveland, Cincinnati
	Kentucky      Frankfort, Louisville
        New Jersey    Trenton

But my friend didn't know enough to show me how to do it.

It seemed to me that seeing a Cobol program to perform this task might
give me a good sense of what the language was like.  (Maybe this is
wrong; I don't know.)  Perhaps someone here would oblige me by sending
me a program to do this, or some other simple, generic task.

Thanks.
```

```
Make the state an alternate key (perhaps this is what you meant)
but you may also have to cater for duplicate town names, for
example London.

While the problem as stated, with a handful of towns, is 
easy for Perl or AWK, or many other languages, where it
becomes a problem that Cobol would be used to solve is when
it is necessary to do the task with every town, city and
district in the US.  And then have a few hundred users
able to access this in whatever way is necessary.

While it may be possible for Perl to read every name into
and array and arrange it in a useful way it is then
within the memory of just that one task, every user
who wanted to know the state for a town would have to
repeat handling the whole file, this assuming that
the machine has enough memory and CPU time to do this.

In education environments all tasks are simplistic
because the projects are to be completed within a
reasonable time, and small data sets make the tasks
easy to grasp.  Unfortunately some tasks get much
more complex as the data size grows, and solutions
change.

I recall one discussion some years ago.  There was a
hypothetical requirement for dynamically building
input shortcuts for data entry.  The C programmer
designed a mechanism that relied on building
a binary tree in memory as the user entered, say,
customer numbers so that it could predict that
required.

Absolutely useless for any real world application
because it had not occurred to him that several
people (or hundreds) may be doing the same task
and each building their own completion table in
memory would make zero savings of keying.
```
