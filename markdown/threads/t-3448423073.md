Try repositioning, in your source, the parameter in question into the
first 64K of your data division. You can`t use call by number routines
with parameters outside the first 64K.

›
› I also do not have any documentation on function 26, the code was
› inherited from previous programs. The manual does not mention function

Functions 26 hasn`t been in the docs for a long time if ever. I looked
back as far as COBOL/2 in 1990! It could be that it isn`t supported in
the static linked library (LCOBOL.LIB) though I doubt that we`d remove
it if it was published somewhere/when. Maybe you or the previous
programmer got it from an old issue of our "Compilations" magazine or
something.

›
› With the Dynamic Linked Run time, or using INT's, it works fine.

You won`t hit the 64K limit problem using the normal runtime. Without
looking, I`m sure we use that call for the character interface "D"
program that handles directory listings but that runs as a .GNT anyway.

That function is an old one that is not actively supported any longer and one
that only works under DOS (it will fail on Windows or OS/2) and is only
supported in COBLIB and the OSX, not the LCOBOL library. For LCOBOL, about
the only way to do this in DOS is to do a shell out to DOS (X'91' function 35
call, which is supported in LCOBOL) and do a "DIR ï¿½ > filename.dat" command,
which would do a directory of your root directory and pipe it to a file,
which you can, then, read in your program and interrogate the last line which will
give you the amount of disk space available.
