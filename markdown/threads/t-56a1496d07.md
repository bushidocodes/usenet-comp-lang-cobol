On IBM mainframes, binary arithmetic is much faster than packed
decimal arithmetic, and no conversion from binary to packed is
necessary as long as all fields in the COBOL statement are binary.
However, the conversion of a numeric field for display/print purposes
is much faster for packed fields than binary because the binary field
must first be converted to packed to be displayed/printed. That is
why counters, subscripts, and offsets are usually in binary. Indexes
are also in binary (by default).

Since I haven't been acitively using COBOL for the last two years
(instead using Smalltalk and C) I had to check this out to verify my
old facts. I checked using ADDs and COMPUTEs and using same modes and
mixed modes. I also tested VS COBOL and VS COBOL II. I got essentially
the same results in both languages (using PMAP in VS COBOL and LIST in
VS COBOL II).

The binary (COMP) fields do not get converted to packed decimal
(COMP-3) unless the result is targeted for a packed field. For
example, in the statement "ADD 1 TO COUNT" where COUNT is defined as
binary (COMP), no conversion to packed decimal occurs. Binary
halfwords and binary fullwords are handled directly by COBOL (via load
and store instructions); however, there are no binary doubleword
machine instructions except for floating-point, which (I don't think)
COBOL handles anymore (the old COMP-1 and COMP-2).

There is a COBOL compiler option that can nullify this and cause
binary fields to always be converted to packed decimal for any
computation. This is the "TRUNC" option in VS COBOL. (I think it is
some variation of "TRUNC" in COBOL II). The "TRUNC" option directs the
compiler to truncate numeric fields based upon their PICTURE clause
(enforcing the specified length). This requires a binary to packed
(CVD) conversion to enforce the specified number of digits for binary
fields. The system programmers will usually set the compiler default
to "NOTRUNC", but not always. This could be the reason for our
discrepancy.

I see what you mean by the ZAP in VS COBOL II. I was not aware of
that. My experience is that a newer compiler does not necessarily mean
a more effecient compiler.
