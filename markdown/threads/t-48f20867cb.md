AFAIK you should declare the cursor within PROCEDURE DIVISION. Our company's
standards even point out that the declaration of cursor should be the first
part AFTER the PROCEDURE DIVISION statements. But if your DB2 precompiler
has no problems with CURSOR within WS, it should also be okay. Maybe just 
worth a try...

If you have XPEDITER available, you could test it interactively.

> not able to fetch it as always it is returning SQL +100, though I am

Hm, the more I think about it, the less ideas.

- you are not pointing to the correct database
- database is temporarily not availabe 
  (inactive... hm that should result in 04E...)
- you did not code an OPEN CURSOR before FETCH

Best guess:

- declare host variables in WS
- declare cursor in PD
- check sql codes of DECLARE, OPEN and FETCH (in this sequence)

Questions:

Is the environment pure DB2 or DB2 and IMS?
Any ON SQLERROR statements included in compile list?

The fact the the SQLcode is NO_DATA tells you the query is being executed
'as requested.'  If something else was wrong, you'd get SQL_ERROR (-1).

As someone else said, see if you can run your query through SPUFI or QMF or
something; then check your 'manufactured' SQL statement to see if it
matches.

Hmm, I see this...

>I am passing values into the variables which are used as host variables in
the declare cursor pgm before I fetch it.

You are not trying to use a CURSOR to vary the parameters of the SELECT, are
you? It doesn't work that way. The parameters for the query and associated
result set are fixed when you open the cursor.If you are trying to use
parameters, you need to look at using PREPARE.
