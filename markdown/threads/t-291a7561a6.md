I have not been happy with the performance of embedded SQL for some time
now.

Generally, ISAM performs much better, although the differences depend on the
application, of course.

Databases provide many other benefits and there is no doubt that we need to
move that way, but it is a bit disappointing if performance suffers as a
result.

Usually when clients are concerned about performance we can tweak the RDB
and improve it that way, or review the sequence of calls being generated by
Transformation and re-organize for better efficiency, but there is still the
inescapable fact that, generally, ESQL is not as good as ISAM when it comes
to performance.

Last year we had a client who encountered a limitation in the number of
connections that can be made through ESQL using Fujitsu COBOL. It would
require the entire application to be significantly restructured and this was
not an attractive option.

This led me to explore the possibility of using Language Integrated Query
(LINQ) instead of ESQL.

There was one slight problem with that: LINQ is not available to COBOL.

However, our general Migration solution automatically refactors all of the
COBOL indexed file IO in the Legacy application into a separate Data Access
Layer and the DAL uses generated (COBOL) Classes that use ESQL to do all of
the physical IO.

What we need is to be able to replace the physical IO in the DAL objects
with LINQ.

Before even thinking about the interfacing problems, experiments had to be
done with LINQ to ensure that it didn't suffer from the same connection
limitations that ESQL does.

I started these experiements in November last year and found to my
astonishment that not only were there no limitations, but the perfomance was
on average 3 times better than ESQL and sometimes up to 6 times better.

LINQ is the way to go.

The LINQ access has to be in C# but it must be callable from COBOL.
Initially it will be coded manually and refined iteratively until I'm
satisfied we have the best possible LINQ access. The solution is then
templated and the Toolset will generate it for any specific tableset on any
specific RDB, in much the same way that the Toolset currently generates a
COBOL DAL object.

For the last 3 months I have been working intensively on a Transport Layer
that enables the COBOL record layout seen by the legacy code to be
transported to/from the legacy program, through the DAL, and back. It has
meant that the COBOL view, the C# view, and the RDB view of data all had to
be accommodated, and it was really interesting to write. The Transport layer
operates within the DAL and permits a seamless interface between COBOL, C#,
and the RDB.

(It has to be a generic solution that will work for ANY COBOL record layout,
and it must faithfully implement REDEFINES and OCCURS back in the COBOL
record buffer, so the code has occupied me quite intensively, but I now have
it working. It uses a facility in C# called Reflection to identify every
single element of data without me having to specifically reference them. A
very powerful facility made possible by the way the .NET framework stores
metadata.)

So now we are into the nitty gritty of writing the LINQ access. It has to be
dynamic, as the predicates for access are unknown until runtime. This has
been addressed by using special LINQ extension classes and it all works very
well.

In some tests I ran yesterday which involved skip sequential access to a
normalized RDB tableset with a base table and 4 attached tables, the LINQ
code was 5 times faster than embedded SQL.(ESQL returned 4 COBOL records per
second; LINQ returned 20...).

PRIMA will announce this product around May this year.

Meanwhile, if any of you are having problems with ESQL performance, contact
me... :-)

Pete.

"I used to write COBOL...now I can do anything."
