Any suggestions?

I have two data types in COBOL

01  n1                    pic S9(8)V9(6).
01  n2                    usage comp-2.

The problem that I'm seeing happen is when I move data from n1 to n2 the
value is physically changed and yet COBOL does not recognize it.  I am
then passing n2 to a C program - an ODBC interface - which then inserts
the value into a table and truncates it.

Example:

move -8.36 to n1
(n1 now contains -8.36)
move n1 to n2
(n2 now contains -8.3599999....)
 n1 = n2  is true

-8.35999999.....]

n2 is passed as a parameter to a double in C which then inserts it into
an Access table
defined at 2 decimals of precision, -8.35.

Is there a MF COBOL data type comparable to C's float?  Will I have the
same problem with this type too?

Does anybody have any suggestions on how I might improve this
precision?  Or shall I just chalk this one up to the quirkiness of COBOL
and move on?

but the bottom line is that YOU CANNOT REPRESENT
VALUES EXACTLY WITH FLOATING POINT NUMBERS.

What you have to realize is that the fractions that are repeating
decimals are base dependant.  1/5, for example, is a repeating
decimal in binary.  As soon as you go floating point, you are going
to have to program for that.  In fact, Cobol is one of the *ONLY*
languages that tends to stick to base ten, just to avoid such
problems.  Your problems started when you started to think in C.

If you are going to do money stuff, then get rid of this idea that
floating point is useful. It creates untold problems.  Work with
either integers or stick to Cobol standard data.

This is perfectly 'normal' for floating-point and C.  It seems
that it is _impossible_ to represent -8.36 in a double, the
closest it can get is -8.35999999999.  Given 64 bits it is
only possible to represent 2^64 different values.  However,
the dynamic range of a floating point number may be 2^-256
to 2^256 (or so) with is hugely larger.  This means that
only certain discrete points on the range can be represented
and -8.36 is not one of the points.

> 
> n2 is passed as a parameter to a double in C which then inserts it into
> an Access table
> defined at 2 decimals of precision, -8.35.

It seems that it is truncating and not rounding.  Perhaps
you can use this information to cause your own rounding
to apply.  Add or subtract 0.005 to the number before
passing it and then the truncation will give the value
you want (probably).

> 
> Is there a MF COBOL data type comparable to C's float?  Will I have the
> same problem with this type too?

COMP-2 _is_ floating point, double in fact.  COMP-1 is a C 'float'
and has the same problems.

> 
> Does anybody have any suggestions on how I might improve this
> precision?  Or shall I just chalk this one up to the quirkiness of COBOL
> and move on?

It has nothing to do with 'Cobol quirkiness' and everything to
do with the quirkiness of floating-point numbers.  This is
why Cobol does not normally use floating point and why C is
a poor choice for a business language.
