I have a two-part question about word versus byte storage mode for USAGE IS
COMP[UTATIONAL] COBOL data.

With byte storage,  data use as many bytes are required to hold the value;
with word storage,  COMP data is always rounded up to use either 2,4 or 8
bytes of storage. For example, given:

 05  FOO    PICTURE IS S(06)V9(04) USAGE COMP.


With byte storage, 5 fives are used; with word storage, 8 bytes are used.

Question, Part I:

Does ANSI or any other organization, or your experience, tell you that the
excess bytes of storage are initialized to x'00' if positive or 'xFF' is
negative?

That is, if I need to translate the data, do I interrogate the sign of the
value in the 8th byte, or in the 5th byte (if bit 7, the high bit, is on and
the data are signed, it's negative)?

I ran some tests on an IBM mainframe (COBOL V2 r4) and IBM does initialize
these bytes as above; but I don't know if that is a COBOL standard or "just
what IBM does" or even "just what IBM does today in this release."

Question, Part II:

On the IBM mainframe I could find no way to force byte storage for COMP. On
my Microfocus Personal COBOL (PC) (the cheap version without a real
compiler) I could find no way to force word storage. (Not that I necessarily
wanted to do either).

What, in your experience, is the "default" or "my way or no way" method of
COMP storage by (what, hardware?  Operating system?  Compiler? )

Thanks,

Michael Mattias
Tal Systems
Racine WI USA
michael.mattias@gte.net

Easy (but unfortunate) part of the answer:
  USAGE COMPUTATIONAL is TOTALLY non-Standard. It can (and does) mean
different things for different compilers on different platforms. (On some it
means binary, on some it is display, and on some it is packed-decimal.)

If you switch to USAGE BINARY is part of the Standard - but what it means is
ALMOST as implementor-specific.   (It is always binary - but storage amounts
and byte-order ARE specifically up to the vendor).

IBM mainframe explicitly does NOT support what Micro Focus calls
"byte-storage".  Neither does it support "little-endian" storage.

The ANSI Standard is QUITE explicit that when you don't put a VALUE clause
on a numeric (or any other field) that "RESULTS ARE UNPREDICTABLE".  With
IBM Mainframe COBOL (IBM COBOL for this-and-that) the run-time option
STORAGE(00) tells the compiler to set it up as binary zeroes - but that
isn't part of the COBOL language. When you  say COBOL 2.4 - I think you may
be using OS/VS COBOL and that is an unsupported IBM product - and one of the
MAJOR migration inhibitors for sites getting to a supported version is this
very "bug" in initialization.  Micro Focus has its own compiler option to do
this too.  (I can't remember which it is right now - but someone should be
able to provide it for you.)  HOWEVER, this is a dangerous thing to rely on.
Use the INITIALIZE verb, a VALUE clause, or a MOVE statement to get initial
values into fields that you want to "guarantee" a value for.

I hope this answers most of your questions - unfortunately not with "happy"
replies for what you want.

It does vary by the TRUNC compiler option. If you compile with

PIC clause (in your sample if the number is greater than or equal to

option if your COBOL program calls or is called by a PL/I or C/C++
program since these languages always use the full binary value.

Michael,
  I don't know if this showed up in any of the earlier posts - but you also
have to worry about "packed-decimal" data.  This comes up every once in a
while in comp.lang.cobol.  The problem is with "even number of digits" in a
packed-decimal UNsigned field.  Given

   05 Whatever  Pic 9(06) Packed-Decimal (or commonly Comp-3 too)

The question is whether this takes 3 or 4 bytes of storage.  In some (I
think the majority) of implementations, there is a nibble reserved for the
NON-sign, so it takes 4-bytes of storage.  While a few implementations
"believe" the PIC and only allocate 3 bytes of storage.)

In fact, this also relates to a general conversion issue.  It some
implementations, the sign-nibble value in both USAGE DISPLAY and
PACKED-DECIMAL is the same for positive and un-signed data - while in others
there is one nibble value that indicates "unsigned" and another that
indicates "positive".  Whether or not there is a separate storage nibble for
the sign in unsigned items, is independent - I think - of which vendors use
the same value in DISPLAY items for positive and unsigned.

Do you have a copy of the '85 COBOL Standard?  You might want to look thru
the hardware dependent and implementor-defined lists for which USAGEs are
there.  Pretty much, what it boils down to is that you need to use USAGE
DISPLAY *and* SIGN IS SEPARATE LEADING - if you want to "guarantee" that you
know what a COBOL source really means.  Otherwise, you MUST know the
"vendor" to interpret the meaning (storage and permissible values) in a
numeric data item.
