I can tell you one thing about it.  When IBM "dropped support" for it (going
from OS/VS COBOL to VS COBOL II - about '83), "we" at GUIDE tried to find
ANYONE who was actually using it.  It turned out that "the phone company"
(before divestiture) did have one application that was using it.  (On IBM
mainframes, it required TCAM rather than VTAM - hard to find even in the
early '80s).  Several of us at GUIDE were from various "baby bells" - and we
discovered that this application had been "happily running" for years - with
NO ONE at the other end actually using the "output".

Other bits of information:

-  Obviously, it is "optional" in all Standards
-  It is OBSOLETE in the draft of the next Standard
_  Micro Focus (for example) supports (supported) its syntax, but was never
able to "certify" (FIPS) conformance to the Standard - but the test (but not
the Standard) required support for "dumb terminals"
- Last time I asked (several years ago), the only ANSI member who still
certified their compiler for it was Unisys - but I can't remember which
"half"

RM/COBOL provides a 'C' stub program that interfaces to the communication
section. I had a client who migrated from an NCR ITX machine to RM/COBOL on SCO
Unix. The communication section was the backbone of their application - don't
ask me why. They had programs accepting transactions from user terminals
(currency exchanges) which sent the transactions thru the MCS to a background
program which updated all their files. Kind of a spooler. I implemented a basic
MCS system using Unix IPC calls. Was a great opportunity to learn C. Later I
tried to augment the system to do connections over TCP/IP, but the COBOL syntax
is too limited (IMHO); it's easier to use CALLs.
