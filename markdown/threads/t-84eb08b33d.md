Others may be able to provide a solution to your problem as stated, but in
the "old days" when I did CICS programming, it was not possible to
DYNAMICALLY allocate members of PDS's.  I can think of two circumventions -
but both of these assume that the members of the PDS are "relatively static"
and don't change while the CICS region is up:

1) Option 1 - if there are a limited number of PDS members that the user
might select - then define EACH member as a different TD file - to your
region.  Then when the user selects what they want - you select the
appropriate file.

2) Option 2 - If the data never changes while the region is up, consider
using IDCAMS to REPRO each member to a VSAM file (either ESDS - or KSDS -
each would have advantages) and then access them via "normal CICS I/O" from
your transaction.

  ***

Where you said that you are not a CICS specialist, have you contacted one in
your company/shop?  It sounds to me like there are some "design decisions"
for which you could get some help from an experienced ON-SITE CICS guru.  If
your shop doesn't have (much) CICS expertise - I am not positive that the
type of application that you are considering is a "good match" for your
site.

What you did will work in IMS regions because it is the
processing region that will absorb the "MVS WAIT(s)" with no real
impact to any other IMS region or IMSCTL region.

A few others have mentioned this -- you will incur "MVS WAIT"
while using your SVC99, OPEN/CLOSE, and READ/WRITE|GET/PUT.  That
MVS WAIT will stop your whole Region/Partition or Address space
(what ever term you want to use).  The only way that I know to
handle this is to write a few ALC routines such that you have an
ATTACH and then you use some agreed upon method to get data back
and forth between the two tasks.  The attached task can do SVC99,
OPEN/CLOSE, EXCP or whatever.  It will probably need to be
attached early on in the CICS startup so that it can obtain the
work/data areas (<16MB GETMAINS) it needs.

You can write or create a service address space and use some type
of cross-memory access so that the service address space does all
the I/O on your CICS region's behalf.  UGLY if you do not know
how to do this.  PC routines would be of help.  This is how TIRS
and IBM Prolog for S/370 do it for CICS (and TSO and IMS and...).

Using an SVC to do this is bad news.  Real bad news.  You will
incur an MVS "dispatch wait" while the SVC is setup to be
dispatched (SVRB).  Then, you will have to obtain and release
locks and all kinds of wonderfulness to pull of the I/O while in
"SVC" state (not technically accurate, but, if you are a sysprog
how would you explain this headache?).  Then, you will have to
change the key of storage for the I/O areas that you got to that
of the "USER-KEY" before "giving" the block to the task (TCB/RB)
that is waiting for it (how else would you ensure MVS
integrity?).

There is another way to do this.  You can CLOSE/Disable a file
within CICS that is defined as sequential.  You can "spawn"
another task to go and get the PDS entry and copy it to the DSN
defined to CICS.  You can then Enable/OPEN the file again.  This
will require manual intervention so that you can schedule how you
do this.

Since I've been gone from CICS for years (on the applications
side), I don't know how easily this can be done in your shop. 
But this is the kind of screwy code that use to get assigned to
me.

Okay,

here we go.  It is possible.

You already have routines that alloc and open (and I suppose also CLOSE) the PDS.  These
routines might be used further on.  Of course, you will have to serialize the usage of
these programs (using EXEC CICS ENQ/DEQ), because otherwise multiple CICS tasks under the
same MVS task might try to open the same DDNAME - probably not a good idea.

But - I am wearing my System Programmer's hat today, and if I catch you doing it you will
get a fair trial, and then will be taken out and shot.  Period.  Perhaps we can skip the
trial, though.

Why? when you use your routines to read an existing member, every I/O is a synchronous
I/O, i.e. it makes the whole CICS region wait until the I/O is complete.  Same is true,
BTW, for your SVC 99 (DYNALLOC) and SVC 19 (OPEN).  Bad for all the other CICS tasks/users
- they all wait for the I/Os


What if open fails (Open 213-18 - member not found comes to mind).  If this happens under
CICS, the whole CICS region might goe down the drain.....


Thus, if you want to do it in COBOL, use a different approach:

Define an extra partition transient data destination for this purpose, let's say PDSM.

MYPDSM DFHDCT TYPE=SDSCI,DSCNAME=MYPDSM,.....

PDSM DFHDCT TYPE=EXTRA,DSCNAME=MYPDSM,DESTID=PDSM,...


(The parameters are quoted from memory, and most definitely not correct.  Go to your
sysprog, he will help you (possibly))


Define the CICS transaction you are going to use with a lower priority than the "regular"
transactions within CICS

Then have your DYNALLOC routine allocate the PDS(MEMBER) to DDNAME MYPDSM



In your CICS program you can now do EXEC CICS READQ TD Q('PDSM') INTO(MY-RECORD)


This will still use BSAM to read the queue, but will at least issue a dispatchable wait
call to allow other CICS processing with higher priority (i.e. all the others) to get the
CPU before you get it

Hi,

Probably the best way to do this in the lastest CICS is to use a 
DOCTEMPLATE.

You define a DOCTEMPLATE with the DDname and member. (see Resource 
Definition Online)

Then you code the EXEC CICS api thus:

EXEC CICS DOCUMENT CREATE DOCTOKEN(token) 
TEMPLATE(template_defined_above)

EXEC CICS DOCUMENT RETRIEVE DOCTOKEN(token) INTO(area) LENGTH(length) 
MAXLENGTH(max_length_of_area) DATAONLY

area will now contain the contents of the member defined by the 
DOCTEMPLATE.
