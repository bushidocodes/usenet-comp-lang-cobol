The nature of the MVCL is that it will set a condition code (NOT an abend, just
a condition code that can be tested with a BC or BCR) and does not do any
moving of data if the sending area and receiving area overlap.

This bug can still bite programmers in CICS if they are dealing with varying
lengths of DFHCOMMAREA and always attempt to move the largest one.

It is impossible for the operating system to anticipate this because we are
talking of a move instruction (MVCL = MoVe Character Long), not something that
the operating system will intercept, and the condition set is just one of the
routine condition codes that many arithmatic functions normally set.

In the DOS/VS COBOL manual, it stated that if one is moving more than 768 bytes
(what three MVCs can move), it would generate a MVCL unless operands
overlapped. (Overlapping operands would cause a subroutine call to be
generated.)
