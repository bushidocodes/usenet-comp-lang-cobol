Jeff,
  I think that you may be getting into another "religious war" question
here.  I think that there are definitely PROs and CONs for both sides - and
there certainly are issues related to "what platform are you on - and what
products are available to you".

For example,  on an IBM mainframe, if you are using OS/VS COBOL, then an
"external Sort" will almost always be more efficient (run faster) than an
internal sort.  On the other hand, if you have "upgraded" to VS COBOL II or
IBM COBOL  for this-and-that, (and use the FASTSRT compiler option), then an
internal sort may well be as fast (I don't think it will ever be faster)
than an external SORT.  Issues of SyncSort vs DFSort also USED TO impact
this - but I think do less so today.

On the PC (or UNIX?) there are less "portable" (universally available)
"external sort" tools (that are efficient).  Therefore, doing an "internal
sort" may well increase the portability of the code you want - and possibly
even the efficiency - Again, depending on the compiler and tools available.

As far as "maintainability" goes, this is where the "most adamant religious
wars" come in.  Depending on what you (your shop and/or those around you -
and certainly those who will be maintaining your application go) know/are
used to - different "maintainability" will occur.

I've heard this argument a number of times, but am still convinced that
it's probably not true.  Internal sorts are generally more efficient (even
with OS/VS COBOL) because the sorted data does not have to be written to
disk then read again.  

In most circumstances, SORT utilities are more efficient at reading data
than COBOL because they use EXCP as opposed to QSAM.  But with an external
SORT, OS/VS COBOL still has to use QSAM to read the sorted records, so I
don't see what you gain.  

Compared to an internal SORT, an external SORT step adds the following
overhead:

1. another job step (the overhead is significant, even if it doesn't
appear on the job's CPU and I/O usage stats)

2. allocate a temp disk data set (SORTOUT)

3. write temp data set (SORTOUT)

4. delete temp data set when you're finished

Of course, if the input file is poorly blocked and SORTOUT is optimally
blocked, then an external SORT may actually be more efficient than an
internal one, but I would consider this the exception, not the rule.

Note that the SORT verb introduces some CPU overhead because of the path
length incurred at each call to the exits, but I don't think it's nowhere
near the I/O overhead associated with an external sort step.

One of the "overheads" that an internal sort (in an ANSI/ISO Standard
conforming COBOL program) has is that it needs to go thru "File Status"
and/or Declaratives processing for both the input and output files (assuming
that you are using USING/GIVING - rather than INPUT/OUTPUT procedures).
This is why with the "newer" (post-OS/VS COBOL) compilers, IBM documents
that you must use NOFASTSRT to be Standard conforming.  This (pretty
definitely?) adds overhead.

I think (but am not positive of this), that OS/VS COBOL - and (maybe even
the newer IBM mainframe COBOLs) are documented as almost "ignoring" blocking
and buffering when doing internal sorts.  Considering that QSAM doesn't work
(?) this way, I don't know how this could happen - but that is what I have
heard.

Finally, I do know (for certain) that internal sorts (when you use NOFASTSRT
or OS/VS COBOL) do not use any of the "faster" sort algorithms that can be
selected with DFSort or SyncSort.

Having said all of this, I am not positive of WHY I have always heard about
this "efficiency" issue.  I, most certainly, never tested in with a
multi-million record file.  But as even IBM documented it in their
Programmer Guides, I guess that I have just always believed it.  (So,

Back when we had only 32K partions on DOS/360, 160K regions on MVT or
less than 500K real memory on MVS/370 (or XA) for a working set, the
invoking program took a significant portion of the memory that could be
allotted to the sort and program combined.  Now that we are in an era
where the newer middle of the line PC's come with 64 megabytes standard
and mainframes have hundreds or thousands of megabytes of main memory,
the invoking program can the sort and the sort in turn will still have
enough memory to sort sufficient portions of the file in memory. 
Basically it is a question of how many times the internal sort will have
to write the file being sorted to a work area.  If invoking the sort
directly from the command system, JCL, etc. reduces the number of passes
because more memory is made available than would be made available to
sort from a calling program, then the external sort plus a following
program will be more efficient.

Back in the early days of VS-COBOL-II, I ran a number of tests
involving multi-million record sorts to validate the FASTSRT option
with DFSORT and SYNCSORT. A COBOL II program with the FASTSRT option
with a  using/giving SORT benchmarked very close to a JCL sort.
Statistically it was close to a dead heat. 

The one combination that all the programming 'gurus'  wanted to use
was alternate indexes instead of sorts.  In any file of significant
size it was almost always possible to read a QSAM or VSAM file
sequentially feeding an internal sort and process the output faster
than you could process the same file through the AIX with no sort.

When DB2 started taking off, IBM's programming guidelines advised that
it was more efficient to work with the DBA to set up the SQL
statements to retrieve the data in sequence or using the ORDER BY to
let DB2 sort the data and present a sorted table to the program.  The
idea being that DB2 could manage the sort more efficiently.
Ironically it was still often faster to do a straight SELECT to read
the unordered table and release that into an internal sort although
the variability was much greater.

There are a couple of alternative techniques that will work well in an
environment where external sorts are preferred.  All IBM compatible
SORT programs  has a number of standard exits that can invoke
subroutines written in COBOL.  

The most common (and a couple of my favorites) were the E15/E35
Input/Output exits.  

The E15 exit is presented each record as it's read. You can then
either drop it, reformat it or insert additional records as needed.

This was a great feature in reading journaling files with multiple
transactions per record. We could read in one record buy the sort,
bust it apart and output 10-20 real records to the sort in a program
with no file declarations and no file processing logic. It's all
handled by the SORT

On the output side, the E35 exit processed each record as it was
returned from the sort and could either write a report and drop the
record or perform additional processing before it was written to the
sort output file.  Again with no file declarations or file processing
logic.  

The only thing needed to invoke these was an additional control card
in the SORT. In a lot of cases this gave us the best of both worlds.
This was especially important before COBOL II FASTSRT options were
available.

However, there are, and will probably continue to be for the foreseeable
future (that's about 24 hours for most people...<g>) applications that do
need permanent storage and it makes sense to hold the data on this permanent
storage in a form where it can be readily accessed and cross referenced.
(difficult with a single table in memory).

When disks were first invented many of them had "head-per-track" and it was
really like a giant dustbin lid with read/write heads strung across it.
There was no way to remove the disk so any data on it had to be sequentially
loaded (usually from tape, but sometimes from cards), processed, then saved
sequentially back to tape again (or punched into a new output deck).

You may wonder why we bothered, but these devices allowed random access
which meant we didn't have to sort. ("SORT" was a four letter word in those
days...hey, it STILL is (I just counted...<G>)). So we designed our systems
using report chains in the records, loaded them to disk, processed, ran off
the reports which were chained in the correct sequences, then wrote
everything back before loading the next application.

(You wouldn't believe what a nightmare it was keeping chains in
sequence...if you lost one link the entire sequence had to be followed
through to the drop point and then EVERY record had to be checked to see
which was next in that particular sequence...Some systems had up to 20
reports chained through the records in this way....It WAS a good grounding
in the use of COBOL Relative access methods as EVERYTHING had a disk
address, and we spent many hours devising faster hashing algorithms to
convert alpha numeric data to a disk address. Ah, happy days!)  But I
digress...

I'm sure you can see that your suggested solution is very similar in
principle to what I've outlined above... That is not knocking it. It COULD
work. Especially if you had very fast methods of getting stuff in and out of
memory.

There is a technology called "Content Addressable File Store" which is very
similar to what you describe but its day is yet to come. In this technology
EVERYTHING is stored in "Memory".

It is a future technology (yes, I know ICL built one but it is/was still
disk based...) because it will not come into its own until memory is so
vast, so cheap, and so fast, that we can afford to do it.

In CAFS, records and data do not have addresses (well that's not strictly
true, but it is totally different to the addresses on your PC or mainframe
disks) Remember, I am talking about a device which only exists on paper at
the moment; when it arrives, details may have changed).  Instead, the data
you require is requested by its CONTENT ..."Get me all the Smiths,
please..." and a "giant memory lookup" ensues.

I could speculate on the Access methods which will be used when CAFS
arrives, but I'm in enough hot water already in this thread, so I won't tell
you that some people I know are already writing a RDB based on the Codd/Date
Relational Model, to be accessed in standard SQL, for implementation on
CAFS....So far I know of nobody writing ISAM for it...<g>
