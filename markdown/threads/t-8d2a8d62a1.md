Ruth,
You might be able to replace the MOVE with

COMPUTE B = A ON SIZE ERROR do something...

but I'm not sure whether or not there would be any significant added
overhead, compared to testing the value in A before doing the MOVE to B
as Marksarn suggests. Actually, I would suggest compiling the program
both ways with the LIST option and see which method generates less
code...it *may* actually be more efficient to do the COMPUTE in this
instance (just a guess - no guarantee expressed or implied!). If you do,
please let us know what you find...

Also, take note that the actual maximum value that B can hold in your
example is different depending on the TRUNC() compiler option. With
TRUNC(STD), the maximum is determined by the PICture size; with
TRUNC(BIN) (or TRUNC(OPT)?), the maximum is determined by the size of
the storage area occupied by the data item based on the PICture size and
USAGE.

Renegade gave you the correct answer of a sort. The compiler *will* give
you a warning level message indicating the possibility of overflow in each
of the programs where it can occur. This is a case of where you actually
should check those warning messages, which very few people do (or are given
the time to do). Your problem really seems to be that your management does
not want to deal with the larger problem of fixing where your predecessors
didn't follow standards, and they didn't enforce them (for which reason
they deserve what they get...)

In that case you can try the NOTRUNC option. This tells the compiler to
treat the PIC S9(4) as a halfword binary, without generating code to
restrict it to +9999. This will allow you to put in your IF B > 10000 and
get what you expect. I agree with the earlier posters that this is not a
good approach to programming (in fact, it's really lousy). Someday someone
will set a 'use TRUNC' standard, and all these programs will break again.
But if you don't have the option to fix the real problem, that may help you
out of a bind.

The compiler will give you a warning that truncation might occur, but
that is it. If you really want to check it, do this....

MOVE ZERO TO B.
ADD A TO B
ON SIZE ERROR
DISPLAY 'OOOPS, A IS OVER 9999 AND B CANNOT HANDLE IT'.

On size error executes when there is an overflow in the receiving
field. In this case, a message will display, but the program will
continue to run (with a value of zero in B). Hope this helps,
