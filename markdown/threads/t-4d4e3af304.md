You have to be aware of the way that the ASCII codes work.  Each line is
terminated by a <CR,LF> in your output.  The page mark (^L) that advances
the page is just a character that fits into this. It sounds as if you are
getting the page character in the wrong place. Looking at the output with a
decent editor that can show you the result in hex or octal should show you
exactly what is happening.

There are a couple of ways that you can fiddle around to fix it. One is to
print a blank line at the end of the page, so that you know you are
positioned at the left column.  You can also printing that last line BEFORE
advancing PAGE, rather than AFTER.  Similar stunts can be done with the
header.

just because of a single dumb design decision by an IBM engineer that

Is it not true that you were taught to use AFTER ADVANCING logic? :-)
I was not taught either way.  Around 1970 I spent two to three years
seriously researching report writing and printing issues.  During that
time I wrote several hundred report programs.  Among other issues, one
thing became clear: that AFTER logic was less efficient than BEFORE
logic for several reasons.  One is that virtually all line printers
support single commands to print, then advance, with the early 1403's
being a rare anomaly (that IBM corrected, please note :).  Second,
AFTER logic leaves the print position on top of the previously printed
line, making that line, in a sense, 'incomplete', which violates the
rule of least astonishment.  Third, there are far more situations where
you want to place extra space after the data you are printing than
otherwise.  For example, why test in many places in your program if
you just printed a header of some kind which needs extra space?  Much
better to print the extra space at the one place associated with the
extra space than the many other places not associated with the extra
space.  Like I said, this is not a conclusion I came to casually, but
one I reached after a great deal of research.  :-)

Here is an example, using a 4 line header as in your book.  As in your
example, the initial 99 in LINE-COUNT forces the first page heading,
eliminating a perform of page heading in initialization.  But it also
signals the page heading routine that the first page is being printed,
in order to avoid printing a blank page at the front of the report.  I
prefer to de-couple page number from this this, because you sometimes
need to reset the page number after a control break, and still skip to
a new page.  I also prefer to make item MAX-LINES a constant, when the
compiler supports that, which you probably do, too. :-)

77  PAGE-COUNT         PIC  9(04) VALUE 0.
77  LINE-COUNT         PIC  9(02) VALUE 99.
77  MAX-LINES          PIC  9(02) VALUE 60.

PRINT-DETAIL-LINE.
    IF (LINE-COUNT > MAX-LINES)
        PERFORM PRINT-PAGE-HEADING
    END-IF.
    <move detail line to print record>
    WRITE PRINT-RECORD BEFORE 1.
    MOVE SPACES TO PRINT-RECORD.
    ADD 1 TO LINE-COUNT.

PRINT-PAGE-HEADING.
    IF (LINE-COUNT < 99)
        WRITE PRINT-RECORD BEFORE PAGE
    END-IF.
    ADD 1 TO PAGE-COUNT.
    MOVE PAGE-COUNT TO PH1-PAGE-COUNT.
    WRITE PRINT-RECORD FROM PAGE-HEADER-1 BEFORE 1.
    WRITE PRINT-RECORD FROM PAGE-HEADER-2 BEFORE 1.
    WRITE PRINT-RECORD FROM PAGE-HEADER-3 BEFORE 2.
    WRITE PRINT-RECORD FROM PAGE-HEADER-4 BEFORE 2.
    MOVE SPACES TO PRINT-RECORD.
    MOVE 3 TO LINE-COUNT.

I have used the control break method in your book, but prefer another
method that I feel is cleaner, and more flexible.  Perhaps we can
discuss that in another thread. :-)

The real problem with all this is defaults. Consider:
    write print-line.
    ? where am I?

Presumably, I am at the start of a new line, and the preceding line ends
with a <CR,LF>. What I really want is:

    move header to print-line.
    write print-line
        (after page AND before 1).

What I usually code is:

    move header to print-line.
    write print-line before page.
    move space to print-line.
    write print-line.

That will generally get things back into synch.
