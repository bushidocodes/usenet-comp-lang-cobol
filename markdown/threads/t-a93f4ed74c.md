As Bill mentioned, most vendors provide sample programs in their 
documentation.

However, these are often not that good and some environments use different 
access methodologies.

For example, by default, Fujitsu COBOL will use an ODBC driver, configured 
from an external file. You CAN access using ADO or DAO (DAO is especially 
good for MS Access DBs) but there is nothing in their documentation to show 
you how. PowerCOBOL provides an ADO control, but again, explanation and 
samples are sparse.

What we need is a shared resource where the community can publish samples 
using whatever flavour of COBOL they have.

I am currently revamping the cobdata website with a view to encouraging 
this, and I'm reviewing and modernising a lot of documents I have lying 
about, some of which deal with accessing RDBs in COBOL, specifically, MS 
ACCESS from Fujitsu.

If you understand OOP, then it is very simple to just use an Automation 
(COM) server to connect to whatever you want in MS Office: ACCESS, Excel, 
Word, PowerPoint, and so on. Without this knowledge it is a more difficult 
road, although it is quite passable.  We saw hundreds of lines of COBOL code 
here recently to enable production of  a .PDF from a web page; in .NET, 
using Automation, it is 3 lines in C# and about 10 in COBOL...(both running 
server side). It baffles me why people seem to prefer complex solutions.

In the meantime, have a look at the thread: "Fujitsu NetCOBOL 8" in this 
forum.

Pete.

> Is there any tutorial/guide available for accessing MS-Access from Cobol? 
> Everything is on Windows Vista.
> Thanks for your help.

If you can find any documentation on using the OLE provider via ADO, that 
will work. If you can find any documentation on using ODBC, that will work, 
too.

Both OLE via ADO and ODBC are "specific-DBMS-product agnostic". (well I 
"suppose" you could try the direct OLE interface, but that is really hairy, 
and with ADO available it would be IMO silly to not use it).

FWIW, with both OLE and ODBC interfaces, if you can find something which 
manipulates MS-Excel spreadsheets, other than the connection string the code 
will be the same,although the way tables are named in SQL statements is 
kinda funny with excel (using Jet/Access, you refer to table names as, well, 
the name of the table; for Excel spreadsheets the "table" name is expressed 
as "[worksheet_name]"

I do not believe there is an ESQL compiler  ( "EXEC SQL ... END-EXEC ") 
specifically for the Jet/Access DBMS, although there may well be one for 
OLE/ADO; and I am positive there is no direct "client interface API" you can 
call.

2. A Class used for accessing the above. Note TableDescrips inherits
from SQlConnect, which in turn inherits from SQLErrors.

$set sql (dbman=ODBC)
$set ooctrl (+W)
  *>-----------------tabledescrips.cbl----------------------------

  Class-id.          DescriptionsTable
                     data is protected
                     inherits from SqlConnection.

  Class-control.     CharacterArray       is class "chararry"
                     DateAndTime          is class "datetime"
                     DescriptionsTable    is class "tabledescrips"
                     SqlConnection        is class "sqlconnect"
                     .
  *>---------------------------------------------------------------
  *>FACTORY.
  *>END FACTORY.
  *>---------------------------------------------------------------
  OBJECT.
  *>--------------------------------------------------------------

  *> Using the appropriate table in MS Access, the SQL Statements
  *> are generated in the N/E IDE - OpenESQL Assistant module - and
  *> are then copy/pasted into the source below

  WORKING-STORAGE SECTION.
  EXEC SQL INCLUDE SQLCA END-EXEC

  copy "Descriptions.cpy".
  copy "\copylib\sqlerror.cpy" replacing ==(tag)== by ==ws==.

  01 ws-dbID               pic 9 value 1. *> General
  78 RecordLength          value 82.
  01 ws-RecordLength       pic x(4) comp-5 value 82.
  01 DBFlag                pic 9 value 2.
     88 SingleDB           value 1.
     88 MultiDB            value 2.

  *>--------------------------------------------------------------
  Method-id. "begin".
  *>--------------------------------------------------------------
  Linkage section.
  01 lnk-Parent                      object reference.

  Procedure Division using lnk-Parent.
     invoke super "setParent" using lnk-Parent
  *> compute ws-RecordLength = function length (DCLTable)

  End Method "begin".
  *>--------------------------------------------------------------
  Method-id. "checkForRectype".    *> SELECT DISTINCT
  *>--------------------------------------------------------------
  Linkage section.
  01 lnk-rectype                      pic x(02).
  copy "\copylib\sqlResult.cpy" replacing ==(tag)== by ==01 lnk==.

  Procedure Division using lnk-rectype
                    returning lnk-SqlResult.

   set ResultOK to true

   if   MultiDB
        invoke super "setCurrentConnection"
        using ws-dbID returning lnk-SqlResult

        if TableError
           EXIT METHOD
        end-if
   end-if

   move lnk-Rectype to A-DesID

       EXEC SQL
        SELECT DISTINCT
               `A`.`DesID`
        INTO
             :A-DesID:A-DesID-NULL
          FROM `Descriptions` A
         WHERE ( `A`.`DesID` = A-DesID )
       END-EXEC

   Evaluate true
       when SQLSTATE = "00000"
            continue
       when SQLSTATE = "02000"              *> No more rows
            set RecNotFound to true
       when other
            invoke self "PrepareErrorMessage"
            set TableError to true
   End-Evaluate

  End Method "checkForRectype".
  *>--------------------------------------------------------------
  Method-id. "deleteRow".     *> DELETE
  *>--------------------------------------------------------------
  Linkage section.
  01 lnk-Key.
     05 lnk-DesID                pic x(02).      *> = Record Type
     05 lnk-DesCode              pic x(04).
  copy "\copylib\sqlResult.cpy" replacing ==(tag)== by ==01 lnk==.

  Procedure Division using     lnk-Key
                     returning lnk-SqlResult.

   set ResultOK to true

   if   MultiDB
        invoke super "setCurrentConnection"
        using ws-dbID returning lnk-SqlResult

        if TableError
           EXIT METHOD
        end-if
   end-if

     move lnk-DesId   to A-DesID
     move lnk-DesCode to A-DesCode

       EXEC SQL
        DELETE FROM `Descriptions`
         WHERE ( `DesID` = :A-DesID )
          AND  ( `DesCode` = :A-DesCode )
       END-EXEC

     Evaluate true
        when SQLSTATE = "00000"
             EXEC SQL COMMIT END-EXEC
             set ResultOK to true
        when SQLSTATE = "02000"           *> No more rows
             set RecNotFound to true
        when other
             invoke self "PrepareErrorMessage"
             set TableError to true
    End-evaluate

  End Method "deleteRow".

The consideration that needs to be examined here is how important efficiency 
is to the end result.

While I wouldn't agree that either of the solutions offered is "ugly", 
neither of them is particularly "elegant" either.

ODBC is flexisble and allows connection to any database that provides an 
ODBC driver, sometimes without requiring any change to the COBOL code, BUT 
it is NOT efficient (compared to other options).

Cursors in ODBC hold the DB connection for as long as the cursor is open. 
This can have a very serious effect on throughput where multiple databases 
are being accessed from multiple terminals. New connections may have to wait 
to obtain a connection and this can be a "bottleneck"  which degrades 
overall system throughput. This factor alone is a major one in the 
deprecation of Embedded SQL (ESQL). (Many programmers see ESQL as being 
obsolete, replacing it with multiset constructs and triggered stored 
procedures...There are some very compelling arguments for doing this 
anyway...)

Automation (OLE originally, now COM) provides platform independent access 
and (for MS ACCESS in particular, but not exclusively) enables the use of 
DAO or ADO, both of which are many times more efficient than ODBC. ADO can 
be used with a number of databases besides MS-ACCESS and it runs a single 
query which returns a resultset, then disconnects. Connections are held for 
milliseconds rather than minutes and the resultset is processed in memory. 
DAO, although deprecated by Microsoft in favour of ADO, is a "no frills" 
package which runs very efficiently with ACCESS. You can run ADO and DAO 
from COBOL but you will need to use COM/OLE to establish easy communication 
with these subsystems. (COM is not the ONLY way but it is probably the best 
way for COBOL...)

I tried using ADO with dynamic SQL from COBOL and found that even with the 
dynamic PREPAREs it was STILL considerably faster than ESQL across ODBC.

These days I use LINQ with C# to access databases and I have no intention of 
going back to ESQL and ODBC; it is simply a different world.

The point of my post is that before we label anything as "ugly" we should 
look at the context it is intended for. Many things that we think are 
"cool", in a broader context can be seen to be not so...

I recall being taught - back in the Oldene Dayse, when DB2 releases still 
had a '1' to the left of the decimal point/period/full stop - that cursors 
had one major advantage and one major disadvatage.

The advantage was that they could be used to individually address rows 
which, due to database design (or lack thereof), would be multiply 
returned by a SELECT; this was a Good Thing because it allowed companies 
to skimp on training and design by telling their programmers 'It is just 
like accessing a VSAM file by a START KEY NOT < and then a READ NEXT'.

(Never mind that this attitude discouraged folks from using a database as 
a database... it Saved Money and Kept Things As 'We' Knew them, things 
that are *very* important in some organisations)

The disadvantage was called 'cursor drag'... that the opening of the 
cursor created, in effect, a snapshot of the subset of the rows which 
reflected the cursor's definition... and it takes DASD and CPU to first 
create these, then address them by FETCHes and UPDATEs and the like, 
create the transaction rollback entries in case something crashes and 
then, finally, CLOSE the cursor, apply the changes (if any), empty the 
rollback logs and COMMIT the whole mess.

In short... when in doubt, blame it on 'cursor drag'; when enough of that 
happens a committee/task group will be formed to look into the matter and 
maybe even make a few recommendations about redesign... which, of course, 
will be accepted with their usual delight.
