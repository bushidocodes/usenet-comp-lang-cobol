```
IMHO ...

Any program that is expecting (requires) initial values (rather than "last used
state") for all Working-Storage items should have the "IS INITIAL" phrase in the
Program-ID paragraph.  (Those still using '74 Standard compilers need not reply
<G>)

Although each "instance" of a program will get Local-Storage in "initial state"
when activated for the first time, they WILL get Local-Storage in "last used
state" when that instance is "popped" back on the stack (in the case of
recursive programs) - if the compiler conforms to the ISO 2002 Standard.

If a program expects (requires) initial values for some data items and last-used
state for others, then they should NOT use "IS INITIAL" - but should use the
INITIALIZE statement (or equivalent) at the beginning of the application logic.

  ***

Obviously there are application specific (and/or compiler specific) reasons for
using other approaches, but "in general" it would seem that above should provide
"safe" programming techniques for programs that may or may NOT be called by
"unsafe" calling (activating) programs.
```

Conversely, every reentrant program should say IS RECURSIVE, because the
Standard says:

"The program cannot be activated while it is active unless RECURSIVE is
specified."

Examples of reentrant programs are DB2 Stored Procedures and plain-vanilla CICS
programs, which load but one copy of the Procedure Division and permit parallel
execution of multiple transactions and queries. For years, IBM has been giving
each invocation its own initialized Working-Storage, which, in effect, turns it
into Local-Storage. It does the same for programs CALLed by those main programs.
Neither are required to use the words RECURSIVE, INITIAL nor Local-Storage.
Second-level called programs might choose to use INITIAL to be re-initialized
for each call within a 'thread', but the first one is free.

When IBM provides a 2002-compliant compiler, will every CICS program need to be
changed? 

Micro Focus supports reentrant (non-recursive) programs via two compiler
options, one of which works just like CICS and DB2. When they get to 2002
compliance, must those programs be changed to RECURSIVE, even though they are
not?

The answer to both rhetorical questions is no. The deviations will be 'vendor
extensions'. I cannot think of any extensions or changes required in ordinary
'85 Cobol programs (unless they use pointers). Many/most are already
2002-compliant.

IBM mainframers looking at a large percentage of their Cobol programs
non-compliant might throw up their hands and lose interest in the 2002 Standard.
All because of the 'high cost' of adding the words RECURSIVE and INITIAL to
every CICS program and DB2 stored procedure. 

The problem could have been avoided with a definition of 'run unit' that makes
it clear a run unit begins with a  transaction or query rather than the loading
of a program. In other words, one that's thread-oriented rather than
process-oriented.

Back in the 60s code was typically 'self-modifying' in that the code
segment would be directly changed during program execution, for
example by an ALTER or by a PERFORM changing code at the EXIT to cause
a return.

ie the code segment was not 'read only', this segment could not be
shared or re-entered in any way.

By having a separate PERFORM stack and redirected ALTERs the code
could be generated so it was not written to and thus could be shared
by separate tasks. This has been standard in modern machines for
decades and even the lowly 80286 could protect code segments from
being written to.

> "The RENT option  indicates to the linker  that the object module  is
> reenterable and can  be used by more than one task at a time.  This type of
…[3 more quoted lines elided]…
> reentrant."

Exactly.  It can be used by more than one task, ie by more than one
run-unit, at the same time.  This is only useful at the operating
system level and merely saves memory space and reload time.  ie it is
a re-entrant _code_segment_ merely by virtue of not writing to itself,
not a re-entrant _program_ or _library_.

MicroFocus Level II Cobol runtime on Concurrent-CP/M-86 used that 20
years ago so that we could run 4 or 5 users on a 1 megabyte machine
(8086).

Within a single run-unit a program is reentrant if the _same_ task can
enter it when it is still active.  For example, within one run-unit,
program A calls B which calls A-dash which is an ENTRY within program
A.  Or similar with messagepeek and callbacks.

This re-entrancy within the run-unit and requires Local Storage or
equivalent, but would share the one Working-Storage.  It _also_
requires that the conditions of non-modified code be met and that
there be a separate PERFORM stack and ALTER table (or no ALTER).

Recursive is self-reentrant in that it directly or indirectly
re-enters itself (whereas A -> B -> A-dash or even A -> A-dash is
re-entrant but not recursive).

In the case of DLLs or shared objects (.so) these may be activated by
different run-units.  A simple DLL may only have one data area and
this would be used regardless of which run-unit calls it.  This maybe
where RW got his 'idea' that another program could get the data from
another program.  In some cases this is actually the required
behaviour, though this is usually serial reusability and not
reentrancy.
