I have no experience with the compiler you're using, but in most
versions of Cobol, the record buffer (the storage decalared under the
FD) moves arround with each I/O to point to the current record. After a
READ which produces an AT END condition, there is no current record, so
the record buffer may not point to valid storage at all. When you MOVE
HIGH-VALUES TO S-CUST-CODE you may be corrupting part of the system and
causing a lock up.

Solution: Your end of file switch can't be in the record buffer. Use a
separate one byte switch declared in WORKING-STORAGE, or move the entire
record description to WORKING-STORAGE and use READ INTO .

RandallBart,

Your point is well taken but it is unlikely that it will take care
of the problem.

With BLOCK CONTAINS and multiple buffers, the record area
may (with some systems) move with each I/O operation.
However, I have yet to find a MicroFocus compiler that treats
BLOCK CONTAINS as, other than, documentation. With Micro
Focus, the record area doesn't move. Each read request will
cause a move from a buffer (512 or 4096 bytes) into the record
area. Only a single buffer is allocated, at open, for each file.
(For indexed files, separate buffers are allocated for data
and index.)

Angela Corner,

For portability with other COBOL systems,

1. If processing an input record from the record area, use a
switch (or flag) to indicate end of file.
2. If processing an input record from working-storage, it is
permissible to move a value to the record to indicate end
of file.

Like RandallBart said.

It seems to me that people are talking at cross-purposes in this thread and
that probably they mostly agree. (In this NG we NEVER completely agree.)

1) If you are a student, then you should write your programs to the
specifications your teacher gives you. (This is a corollary of the
traditional, "contractors code what you are paid for" and "employees code
what your boss tells you to.")

2) If your teacher is providing you with "bad data" for a program, this is a
GOOD teaching technique. You should learn to ALWAYS code for bad input.

3) In the "real world", if your program is having problems (and you are
getting called at 3am) because of bad data, your highest priority is to
enhance your program to handle bad data. (What you do with it will be
determined by the needs of the application. Some times you will need to
"crash the system" with a "user-friendly" message because nothing can go on
from there. More often you will do error reporting - but some times you
will be able to "fix the data" in your program, report on it, and continue
processing.)

4) *AFTER* you have gotten your real-world program to handle bad data, then
it is important that you find out and FIX why the program is getting bad
data. If another program is passing you the bad data, you need to track it
down and get if fixed. If a data-entry system is allowing bad data, then it
needs to get enhanced edit-checks. HOWEVER, even after the source is "found
and fixed" you still MUST make certain that your program always can detect
and handle "bad data" in an application specific way. Simply, letting the
operating system (which may change from release to release) die when you get
unexpected input is never considered a GOOD programming technique.

HOWEVER, unless your instructor really has never written a real program
(which is possible, but I can hope not true), I think that if you approached
the instructor and asked "Is there ever an application design which makes it
correct to "correct" the data?" Then, I bet your teacher will admit there
are such times in the real world. Let me repeat several techniques that are
each used based on application design requirements when you get bad data:
(There probably are more than this, but this will give you a good start -
that you could show your instructor and ask about.)

1) If some bad data is for a truly critical piece of information, then you
create an informative message (report, console, somewhere it will be seen)
with information to identify where the data came from, what is wrong with
it - and then you "gracefully" end the program.

2) If this is bad data in a field that fairly frequently gets incorrect
input (often in transactions requiring manual input), then you probably can
just create an error report (or transaction) and go on to the next record
(leaving a clear audit trail of what you skipped and why)

3) Let's say you are creating the actual online transaction where real
people enter data manually. If they enter bad data, it is often possible to
make a guess as to what they really mean. In such cases, you should go back
to the user with some type of message about "you entered "ABC", but this is
not valid - you probably mean "abc" - if so press enter to accept this,
otherwise, please re-enter the value for field XYZ"

4) Sometimes you really can guess what was meant by bad data. A classic
example of this is when a numeric field is left justified instead of right
justified. For example a 4 digit field (which accepts blank padding) had
"123 " instead of " 123". If your REAL WORLD users tell you that this often
happens - even though it isn't supposed to and that this is OK, then you may
well FIX THE DATA to make it " 123" - write a report indicating that you did
this (and for which record). Again, depending on the actual application
demands, it is possible that you will need to fix the data in the input file
before it goes to the next program in the application. In other cases, you
may just fix the data in your temporary storage and keep the bad data in the
record. THIS DEPENDS ENTIRELY on the application - what is needed and what
is expected.

If you have an instructor who tells you that this type of thing will never
happen (and be what is best for the application), then YES, your instructor
is lying to you! If on the other hand, your instructor says that your first

defined way and then get whoever or whatever has created the bad data to fix
it, then the instructor is correct.
