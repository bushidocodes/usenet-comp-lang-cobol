```
I suspect Bill will want to comment on this one. :-)  Yes, I know it's
way too late (by about 20 years), but I had to get it off my chest. :-)

IIRC, in COBOL 74 and earlier, any explicit REDEFINES other than
at the 01 RECORD level was required to be the same size as the item
being redefined, but COBOL 85 relaxed that and permitted an explicit
REDEFINES item to be smaller (not larger) than the item it redefines.
According to the latest working document I have (Dec 2001), the 2002
standard continues this.

I submit that this is not the way it should have been handled, for a very
good reason. Consider this pattern that is one of the most frequent uses
of REFEFINES:

            03    PRIME-VALUES.
                    05    FILLER        PIC 9(04)    VALUE  0002.
                    05    FILLER        PIC 9(04)    VALUE  0003.
                    05    FILLER        PIC 9(04)    VALUE  0005.
                    05    FILLER        PIC 9(04)    VALUE  0007.
                            ...
                    05    FILLER        PIC 9(04)    VALUE  9967.
                    05    FILLER        PIC 9(04)    VALUE  9973.
            03    PRIMES                   REDEFINES PRIME-VALUES
                                                     PIC  9(04)    OCCURS 1229 TIMES.

In this particular example, there are exactly 1229 primes below 10,000.
If the number of values is correct, the compiler will flag any OCCURS
greater than 1229, but not a lesser value. Why should a compiler permit
such an obvious error? I believe there should be syntax for REDEFINES
that would require the sizes to be equal. Actually, it would have been
better had COBOL 85 made this the default and provided a syntax to
relax the constraint.

Yes, it's beating a dead horse, but what do you gurus think?

P.S.- If you would like a simple prime number list generator, you can
download PRIMES.ZIP from the URL below.
```

```
You're right - it does lack that construct.  However, there is another 
way to do it...

03  array-values   Pic X(12) Value "000200030005".
03  array1 REDEFINES array-values PIC 9(04) OCCURS 3 TIMES.

This is a little less verbose, and works with strings up to 160 
characters.  After that, you're better off either using the above 
construct (which, if I were doing them, I'd leave off the names 
entirely, which is the same as coding FILLER), or writing procedure code 
to fill the table when the program starts.

One of the more common constructs you'll see is something like this...

01  Month-Area.
     03  Pic X(36) Value "JanFebMarAprMayJunJulAugSepOctNovDec".
01  Month-Table  Redefines Month-Area.
     03  Months  Occurs 12 Times  Pic X(03).

Then, Months (1) is "Jan", Months (2) is "Feb", etc.  It's virtually the 
same thing as saying

01  Months[12] Pic X(03) Values 
{"Jan","Feb","Mar","Apr","May","Jun","Jul","Aug","Sep","Oct","Nov","Dec"}.

As to the "why", that's before my time - but, this is "the way it is"...  :)
```

```
COBOL 2002 has the ability to "initialize" individual elements of a table, e.g.

  03  EachElem pic 99 Occurs 3 times Value 1,3,27.

range of elements to a value.

Check with your "vendor of choice" of (planned) availability of this 2002
Standard feature.
```

```
In both cases, the called (or invoked) subroutine (or method or function) gets
access ONLY to a storage of the "size" of the passed parameter.
```

```
Still true, to a degree.

For Unisys MCP/AS COBOL74, fixed-point 77-level numerics of any USAGE are
regularly coerced into on-stack USAGE BINARY items, unless the compiler
figures out a reason to disallow this optimization.  Moreover, elementary
01-level numerics are first optimized into 77-level items if the compiler
doesn't see a reason to prohibit the optimization.  And in both cases, if
the item is used as a source but not a destination and in addition has a
VALUE clause, it is turned into a compile-time constant; if it is never
referenced, it can be optimized out of existence.

MCP/AS COBOL85 treats 77-level BINARY, BINARY EXTENDED, BINARY TRUNCATED,
REAL and DOUBLE items as on-stack items, but does not second-guess the
programmer's USAGE specification for COMP and DISPLAY numerics at Level 77,
and does not treat 01-level elementary items of any USAGE into 77's.  This
is in part because COBOL85 strives to reduce its dependence on the D2 stack,
"pooling" most working-storage.
```

```
Several things seem to be going on in the mainframe shops. One is that there
are very few programmers on site relative to demands nowadays, so bit fiddling
for any purpose is a fading art.  Boundary alignment can make a big difference,
as can comprehension of the machines internal cache line allignments (greatly
model dependent in any case).  But only if your own application code is the
issue.

In the mainframe shops the DB2, IMS, IDMS, CICS, and VSAM for that matter
actually determine the CPU piggery of your application (no matter how those
mips are accounted). This is independent of the I/O time itself.  SQL in
particular sells iron because it is unbelievable inefficient if there are no
direct indexes to which to bind.  But also things like IMS and CICS are real
Santa Clause packages that offer a little for everyone, and it takes eons of
CPU time to get past all the what-ifs that ain't-if.

The unit of work, resource contention management facilities, and synchpoint
processing capabilities of modern packages chew mightily into the CPU as does
most notably the every constant march of the more robust OSes. There is little
you can do with your own code optimizations in this context.

The best bet for performance is to design your program and the databases in
such a way as to minimize the need for services.

Boundary alignment and data conversion in an application program still mean a
lot but just in an ever  decreasing portion of the whole situation presented to
the CPU.
```

```
DIVISION and suppress generation of code to initialize these data items to
their VALUE clauses. When OPT(FULL)is in effect, all unreferenced 77-level

the data item was discarded.

items. Two common ways this has been done in the past are:

  a.. A technique sometimes used in OS/VS COBOL programs is to place an
unreferenced table after a referenced table and use out-of-range subscripts
on the first table to access the second table. To see if your programs have
```
