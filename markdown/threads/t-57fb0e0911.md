The date functions do not abend, but they only deliver 2 digit years in
programs built with VS COBOL II (CMPR2 - running as a 74 compiler), and
OS/VS COBOL (can be set to 74 standard).

Note that VS COBOL II can also be set up as an 85 standard product -
NOCMPR2, same results in that mode.

Ditto for OS/VS COBOL, it can be set to 68 standard (sort of), but all of
it's date related syntax (note that I did not say functions, since
FUNCTION is a 1985 + addendum facility, available only in COBOL/370 and
COBOL for MVS and VM).

One escape.... (Best approach is to migrate code to current compilers) -
use a package called The Bridge to the Next Century. This package
provides functional equivalence for the 5 date related functions that are
not present in VS COBOL II and OS/VS COBOL (not to mention any other 68,
74 or 85 conforming compiler).

Details on the Bridge furnished via e-mail if interested.

Rex Widmer
Builder of software archeology tools and other strange programs to help
survive in a legacy based world.

The REAL problem is the when you use the ACCEPT from DATE
the value that is returned is only the last 2 positions on the year.

there is another intrinstic function that will return the entire 4 positions
of the year.

MOST programs already have code in them to handle the 2 digit problem
and expand it to 4 digits using some type of compare. For example
if year less than 80, the date becomes 20XX, else it is 19XX

For space reasons, older systems stored the date in files in the format
yymmdd, so when sorting a date of 000101 (jan 01, 2000) will sort
before 991231 (dec 31, 1999)

Many industries have already have adusted the programs MANY years ago.
Examples. Banking (30 year mortgages, long trem loans), Long term leasing,


The problem is NOT a cobol problem, but rather how the date is stored on
