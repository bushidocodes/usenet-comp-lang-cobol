Did you also want it to be cheap ?

Microfocus used to (and may still do) include a 'Compatibility Guide'
with their compiler that had a section on DG Interactive Cobol
indicating the use of the DG compiler directive and the REFORM5 and
CONVERT5 utilities to process the source and convert data files.

This will fulfill points 1 and 3.  The resulting programs will be just
like they are on the DG (presumably).  If they were to be implemented
on a Linux server and run on the client machines using, say, putty, as
the client then they could be easily accessed from a local network or
from anywhere on the internet using putty (or telnet) running ssh
(Secure Shell with passworded login) protocol on any client machine,
even over slow internet connections.

If you want point 2 (GUI) then point 3 goes out the window, as it
were.  Extensive re-writing is probable.  If you were to re-write to
Windows GUI then for point 2 (Internet access) you would probably be
stuck with TSE and CALs and possibly VPNs, probably requiring
broadband.

If you rewrote to Flexus SP/2 for the GUI (www.flexus.com) then the
same programs (there is just a few lines of code required) could use
the SP/2 Thin Client system for remote internet access, even with slow
dial up.

That was assuming that the 'internet capabilities' was to cater for
running the same programs, or a subset, over the internet as if they
were running locally.

If your 'internet access' was to create a web site that allowed, say,
online ordering, then there are many ways to do that, and most Cobols
will do it in some way, but it would be a different system, even if it
were sharing the live data files.
