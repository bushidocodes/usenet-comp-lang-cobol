Ok.  then it sounds like this is what you could do.  First, you want to
write a generic little program that will simply go through all the files in
a wildcard specification.  That will be compiler specific, but most
compilers have a subroutine library that will do the job. To test it, you
can just have it display the filenames.

Using that as a mainline, convert all existing code from being the mainline
to being a subroutine that accepts the filename as subroutine arguements.

Pretty well everything else can be done in batch files.  You might also want
to check the batch loop command that Jerry suggested.  I am far from a DOS
batch expert.

Mike,

This really doesn't need a COBOL solution.

You can run the existing program which you know works OK, using the single
DOS command suggested by Doug Miller.

If the errors are the only output from the program, you can append them by
extending Doug's very elegant solution as follows:

C:\> for %f in (ST125*.*) do myprog.exe %f >> errfile.txt

This command will present each member of the recordset of filenames which
match the wildcard pattern "ST125*.*" to your program (myprog.exe). The
output will be APPENDED to the "errfile.txt" file, so, as each iteration of
the myprog.exe runs, its output will be placed on a single error file
(errfile.txt).

Alternatively, you could send the errors direct to the printer with:

C:\> for %f in (ST125*.*) do myprog.exe %f > LPT1:

(Note that in this case there is no need to append unless you don't want a
page throw and headings at the start of each report).

You will lose nothing by trying these commands.

If the program has several outputs then the error file solution is not
viable and you would need to amend the program to OPEN APPEND on the text
file.

I would strongly recommend that you do NOT change the program code if you
can run it using the above "JCL".

Mike:

To achieve what you want try this.

1.  Continue to use the DOS window to prompt the user for input of file
names.  The shell will parse the command line and produce a list of file
names that match the wild cards.

2.  The key is to redirect the output list of matching file names to a file.

3.  You then open this file and read the names into a table defined with an
OCCURS DEPENDING ON NUMBER-OF-FILES  clause.  Increment NUMBER-OF-FILES for
each read until EOF.

4.  You then MOVE the names to working storage with this structure.

01 AF-ALL-FILES.
    05 FILLER           PIC X(7)
          VALUE "SELECT ".
    05 AF-FILE.
        10 AF-FILE-FIRST    PIC X(5)
              VALUE  "FILE-".
        10 AF-FILE-NUMBER OCCURS n TO p
                TIMES DEPENDING ON
                NUMBER-OF-FILES
                INDEXED BY WS-INDEX.
                15 FILLER    PIC X.

01 AS-ASSIGNMENT            PIC X(17)
          VALUE "    ASSIGN TO 'C:\".
            (or whatever directory)
    05 AF-INPUT-FILE-NAME        PIC X(12).
            (names from the file of file names go here)
    05 FILLER                      PIC X(2)
          VALUE "'."

01 OR-ORGANIZATION    PIC X(32)  VALUE
       "ORGANIZATION IS LINE SEQUENTIAL."
        (or whatever organization you need)

5.  Write the three lines to a file until NUMBER-OF-FILES is reached.  These
will become your FILE-CONTROL statements.

6.  Your next step is to CALL or LINK to a program that will COPY  this file
into it's FILE-CONTROL section.  Do you know how to set up and use
libraries?  It looks like this...

ENVIRONMENT DIVISION.
INPUT-OUTPUT SECTION.
FILE-CONTROL.

COPY INFILES.DOC. (or whatever name)

SELECT ERROR-FILE
ASSIGN TO etc.

7. Remember to keep NUMBER-OF-FILES in scope for the called program.  Your
OPEN and CLOSE statements can be done in a loop using NUMBER-OF-FILES as the
control and varying an index for the file number like this.

200-OPEN-FILES.
    PERFORM VARYING WS-INDEX-A FROM
        1 BY 1 UNTIL WS-INDEX-A >
            NUMBER-OF-FILES.
        OPEN INPUT FILE-(WS-INDEX-A).
    END-PERFORM.

8.  The called program can then use an index for its READS and process one
file at a time until EOF.  You will wind up with one ERROR-REPORT that can
easily have the file-name and record sequence number on it for each error
record.

The advantages of this approach are that you avoid your wildcard problem and
you can use as many files as you like without knowing the names beforehand.

The disadvantage is that it's very awkward.

I hope this helps.
