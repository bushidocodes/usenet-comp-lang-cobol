The $SNDJBC system service has thirty-three different function codes. I
can't give an example until you are a bit more specific about what you
want to do.

For those readers who are unfamiliar with the DEC VMS operating system,
it takes an unusual approach to portability. Most of the time, one has a
CoBOL program, or a ForTran program, or a Pascal program, or a C program,
or what have you, and ports it from one operating system to another. If
one needs a feature not provided by the language, one calls an
assembly-language subroutine. One doesn't usually talk about "porting"
ForTran to CoBOL on the same platform; that would normally be called
"rewriting".

However, every language implemented by DEC under VMS, including CoBOL,
has been enhanced with whatever it needs to be able to call a system
service directly. No assembly-language subroutines needed. This means
one can write a device driver or an interrupt (AST) routine in CoBOL!

Notice that once a program uses system calls, it is much more like
routines written in other languages for the same platform than it is like
routines written in the same language for other platforms. For example,
a CoBOL routine that uses $SNDJBC will be very similar to a C routine
that uses $SNDJBC, and neither will be portable to any other platform,
which of course is why DEC did it this way.

Most VMS system services require an itmlst. An itmlst is an array of
12-byte items terminated with a 4-byte zero. Each item contains an item
code that specifies a piece of information and buffer controls that
specifies where the piece of information is read from or written to.

Start an itmlst with an 01-level to obtain 2-byte alignment and to create
the symbol that will be passed by reference to the operating system:

01 WS-SNDJBC-ITMLST.

The first piece of information in an item is the size of the buffer. As
you can imagine, very bad things happen if you pass the wrong buffer size
to VMS! The best way to insure that the correct value is passed is to
use a compile-time constant:

REPLACE ==ITM-BUFSIZ== BY ==128==.

Pass the length in the item something like this:

05 FILLER PIC S9(4) COMP VALUE ITM-BUFSIZ.

Later on in WORKING-STORAGE define the buffer something like this:

01 WS-ITM-BUF PIC X(ITM-BUFSIZE).

Pass the item code in the item something like this:

05 FILLER PIC S9(4) COMP VALUE EXTERNAL SJC$_FILE_SPECIFICATION.

Pass the buffer address in the item something like this:

05 FILLER POINTER VALUE REFERENCE WS-ITM-BUF.

Lastly, you need the address in which the system will return the length:

05 FILLER POINTER VALUE REFERENCE WS-ITM-LEN.

Define the length elsewhere in WORKING-STORAGE something like this:

05 WS-ITM-LEN PIC S9(4) COMP.

Eventually, you will probably want to define the item length as part of
a descriptor, but let's stick to the simple case for now.

You can now start another 12-byte item, or terminate the itmlst:

05 FILLER PIC S9(9) COMP VALUE ZERO.

Define a function code elsewhere in WORKING-STORAGE something like this:

05 SJC$_ADD_FILE PIC S9(4) COMP VALUE EXTERNAL SJC$_ADD_FILE.

Define an I/O Status Block something like this:

01 WS-SNDJBC-IOSB.
05 FILLER PIC S9(4) COMP VALUE ZERO.
05 WS-IOSB-COND-VAL PIC S9(4) COMP VALUE EXTERN SS$_NORMAL.
05 FILLER PIC S9(9) COMP VALUE ZERO.

In the PROCEDURE DIVISION it looks something like this:

CALL "SYS$SNDJBC"
USING OMITTED
BY REFERENCE SJC$_ADD_FILE
OMITTED
BY REFERENCE WS-SNDJBC-ITMLST
BY REFERENCE WS-SNDJBC-IOSB
OMITTED
OMITTED
GIVING WS-VMS-COND-VAL.
