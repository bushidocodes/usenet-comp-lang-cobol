Judson McClendon writes:
 
> What I have stated repeatedly is that one should be aware of
> the *kind* of assembly code that would need to be generated for
> the HLL code you are writing.
 
Yes, you have said many things repeatedly.  I simply don't agree
with you.  It is important to make the _Cobol_ efficient. It is
unnecessary to wory about the grubby details of whatever CPU
underlies the code.  In many cases there is _no_ assembly code
generated.  Many compilers output byte codes for a virtual
machine that is not specified in any detail.  This may run
unchanged on anything from an 8080 to a 64bit Alpha.
 
Any particular source may run directly to the CPU or via a
called general purpose routine.
 
> It doesn't even need to be for your own CPU.  A move is a move,
> a multiply is a multiply,
 
Wrong.  A MOVE in Cobol may generate a simple sequence of
instructions, _or_ it may generate a complex sequence, _or_
it may generate a call to a routine to do some far more
complex manipulation.  The generated sequence may depend on
the target machine and the type of the variables.  For
example some processors can generate simple moves for
particular allignments while others cannot deal with those
alignments directly and need to generate far more code.
 
 
> and you can assume that the CPU can do them efficiently.
 
If you think that then I suggest that your code may be
horribly inefficient on some machines.
 
> Assembly languages tend to be more similar than HLLs
 
Wrong.  You should get out more.
 
Some assemblers look like high-level language.  I have seen
assembler where every instruction specified the next instruction
address (the processor did not update the PC to the next address
for some very good reason, it was already too late to do that one
next), assembler where a branch had to be coded two instructions
before it was required (due to run-on), assembler where every
instruction is conditional.
 
 
> But if you program in a HLL and don't have a clue what it takes
> for the CPU to execute the various commands, then you are
> programming blind, as far as efficiency goes.
 
I doubt that you would have a clue how modern CPUs execute
code with parallel sequences, speculative execution,
run on, loop unrolling, result elimination, ...
 
I suspect that you may know very well a 30 year old design
and base everything on this.
 
> And looking at the software being turned out these days,
> that happens quite a lot.
 
It is entirely possible to write efficient _Cobol_ code without
worrying too much about the underlying CPU.
 
For example, worrying about whether to use USAGE INDEX or
subscripts with COMP, COMP-3, or COMP-5 is _pointless_ if you are
coding a bubble sort of a table when a better algorithm is more
appropriate, or when the sort could be avoided altogether with a
better program.
 
Of course, on your other 'point' that efficicy in results means
efficiency in programmer time is also, in general, completely
wrong.
 
For example, most programmers should be able to get a bubble sort
working first time up.  They should be able to get the boundry
conditions right very quickly and make it solid code in a
reasonable time with little need to use a debugger.
 
However, if, say, a quicksort were to be more efficient, then
I doubt they would get it working correctly in every case in
anywhere near the same time.
 
In the other direction a SORT on a table is currently an MF
extension (will it be a Cobol9x feature?) but any programmer
can make this work in a few minutes.
 
Given SORT table is available would you suggest (based on your
ideas of 'efficiency') that programmer use?:
 
       a) SORT table
       b) Bubble sorting and argue about COMP-? or INDEX
       c) Quick Sort        "        "      "       "
 
and which do you think would use least programmer's time
and be least liable to a call at 2am ?
 
Myself, I would probably design the system in the first place so
the data for the table can be read already in order (with eg
an alternate key) as this removes all code and programmer
ineffiencies.  It would also, most likely, eliminate
problems of data exceeding table size.
 
That is the way that _I_ achieve efficiency.
 
 
> You only write code once, but it may execute billions of times.
> CPU power is cheap, and you don't have to hoard it. But there
…[7 more quoted lines elided]…
> mind boggling.
 
Feeling better now ?  or would you like a little lie down ?
