Your objective is probably easily achievable, but the coding strategy your
considering will not work with COBOL. Thesyntax of the language is highly
influenced by its extemely early genesis in DP history.  

COBOL syntax is essentially a compiled syntax. The details of the
WORKING-STORAGE section, for example, become entirely hardened as of
compile-time.  Even the ability to manipulate at run-time the location of
information mapped with working storage layout via an ADDRESS OF phrase is a
comparatively new language feature.

The coding technique you envision is more like BASIC or even LISP, where you
can interpret and re-interpret (or EVALuate and re-evaluate) source code.

COBOL is rather more rigid syntactically, even under the mediation of modern
interpreters or incremental compilers. A PIC cluase is essentially fixed. And
by execution time is actually gone and only its effects remain.

You may do better to consider one of two basic paths. Both use a PROCEDURE
DIVISION verb to respond to dynamic data conditions.

The first approach is to UNSTRING the input data area into one or more
subfields. This requires you to discover reliable delimitters within the input
data. Data which has been so input usually requires fixups after you isolate
the pieces (leading zeros, left/right justification within the targeted
subfields).  

From the subfield(s) you can do simple moves to the output, or construct an
outbound image with STRING.  It is not at all necessary to use STRING to
output, just because UNSTRING was used on part or all of your input operation.

(Be warned that you must init target fields for both UNSTRING and STRING verbs,
these are not MOVEs and will not blank fill!)

The second strategy is to INSPECT your input field TALLYING various LEADING
classes of text type data.
With the tallies you can do exact moves with reference modification.

This can be tough for beginners. But it is achievable for you if your input is
not tremendously complex.

Anyway, the layout of things in the DATA DIVISION gets set something like
concrete.  It is not unusual for folks to use numerous REDEFINES of the same
area to compensate for this.  But if your input is highly variable, you need to
scan it directly yourself with INSPECT, or more automatically with UNSTRING ...
DELIMITTED BY.

Keep posting and let us know how you do.

The next Standard does provide a method of "dynamically" changing what file
an FD points to.  (As mentioned in another post in this thread, this is
already a common feature with PC compilers - but is not allowed with IBM
mainframe compilers)  The new syntax is INcompatible with all existing
implementations (that I know of).  It has

   Select ABC USING xyz ....

where XYZ is a (working-storage) data item including the "physical file
name".  For IBM mainframers, I think this will still end up being the DDName
and not the DSName - but that is up to IBM to decide  upon.
