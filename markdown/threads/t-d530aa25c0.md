### System freeze in MF 4 under Unix

```

Can anyone help?

House: MF COBOL 3.1, DG/UX Unix SYR4.0, Motorola platform (AViiON)
Client: MF COBOL 4, DG/UX Unix SYR4.2, Intel platform (RAID CLARiiON)

We've been developing stock control applications which have worked well until
the client upgraded this weekend to MF 4 and DG/UX 4.2 (previously ran
identical versions of both). We have today encountered serious system
"freezes", where several programs which hitherto ran without incident are now
hogging CPU time or hanging. The first two days (Mon/Tue) after the upgrade
have been fine - it's only today that things have gone awry.

Example 1: A program which scans through two large-ish (850Mb/550Mb) files,
writing to a temp file and then generating a report from the temp file, used to
run OK; it now seems to hog CPU time to the detriment of _all_ other users.
Today it showed 42:08 of CPU time (with a very noticeable drop in system
performance), where other users were getting one or two seconds.

Example 2: A program which scans through a small-ish file (50Mb) and then CALLs
a print program hangs just at the point of the CALL. Admittedly, there is no
exception clause after the CALL, but the program has run OK up until today.

Example 3: A simple file enquiry program (all files opened input only) also
began to freeze - this was reported in the late afternoon today by a number of
users within minutes of each other, long after we'd killed the PIDs of the
"big-scan" program.

Our Technical Support boffins haven't adjusted any of the kernel parameters in
the last two days and there seems to be no record locking conflict (although
I'm not 100% convinced).

Is it possible that the version differences (in MF and/or DG/UX) are causing
the problems (we're compiling in-house and then sending the compiled software
via modem to the client's site)? Does anyone else have experience of this?
All answers on a postcard please (or a newsgroup or e-mail).

First valuable answer wins a pint of warm, flat UK beer! (No smart replies
about incentives, please...)

Thanks for your help, guys ('n' gals).

John
```
