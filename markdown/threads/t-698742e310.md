```
I've got a few questions for the COBOL experienced
readers  regarding the up-coming standard and
fixed-form vs. free-form formats.

First, the standard indicates a rather straight-forward
logical conversion of fixed-form input to free-form
at compile time.  But - such conversion would appear
to ignore the area A, B, C rules...  (i.e. IDENTIFICATION
DIVISION would no longer be required to be in area A.)
This would seem to force a conforming implementation
to no longer diagnose statements not correctly begun
in area A, since all FIXED format input is converted
to FREE format.  Am I reading that correctly?

Second, in section 7.2.15.2 there is the statement:

   2) The default reference format of a compilation-group is fixed form.

which would seem to indicate that the first line of any
input file is in fixed form, and thus, using free-form would
require a SOURCE FORMAT IS FREE.  statement, which
would begin anywhere after column 7 (as the required conversion
from FIXED to FREE basically ignores columns 1 through 7.)

  However, paragraph #4 later indicates:

   4) The first SOURCE FORMAT directive in source text or
        library text may be in either fixed form or free form.

Thus, does the compiler have to "magically"  handle the sequence
columns when looking for any SOURCE FORMAT statement?
That is, it appears that the SOURCE FORMAT statement
can occur in column 1, say, even if the current format is FIXED.
And - if that's the case; wouldn't this  be the same as:

  4) The first SOURCE FORMAT directive in source text
       or library text may be in free form, regardless of the
       current format.

Just for example...  If we use the following first few lines of
input - this would seem to be valid input...

123456*8901234567890
             IDENTIFICATION DIVISION.
 SOURCE FORMAT IS FREE.
     *>  The remainder of the source is in FREE format.
     *>   ...


Does anyone have a clear enough understanding of how this
should be handled?   In this example, the compiler is expected
to examine each input line's sequence number area for the
beginning of a SOURCE FORMAT statement...

And if that's the case - why restrict it to the first SOURCE FORMAT
statement?

My conumdrum comes from the implication in the logical
conversion from FIXED to FREE format that the compiler
ignores columns 1 through 7; except, apparently, in the
special case of a SOURCE FORMAT statement.... it's this
special case which is so troublesome.

Opinions?

    - Thanks -
   - Dave Rivers -
```

```
A few answers (I hope) - to both asked and unasked questions:

1) The revision EXPLICITLY gets rid of the old A-/B-margin restriction for
BOTH fixed-form and free-form reference format.

2) There are methods of indicating "continued" lines, debugging lines,
comments, etc that will work in BOTH fixed-form and reference format.  This
means that it is entirely POSSIBLE (not required) to produce code that will
compile when EITHER fixed-form or free-form reference format is in effect.
(Thus you can have copy members that can be used by both).  To do so, follow
these rules:

  a) only include code in column 8 thru the R-margin (whatever that is in
your implementation - usually between column 72 and 73 or after column 80).
Leave columns 1 thru 7 blank.

  b) never try to continue a COBOL word across lines, i.e. don't try and do a
         MO
   -          VE   abc to X
   -                                 YZ.

  c) If you want something to be a comment line, code it with
        *>
      as the first characters in the line (but still column 8 or latter)

  d) If you want something to be a debugging line, code it with
          >>D
     as the first characters in the line (but still column 8 or later)

  e) Use the
         >>PAGE
    directive instead of a "/" in column 7

  f) If you want to continue an alphanumeric literal, use the new style
continuation, e.g.
       Value "ABC"-
                 "XYZ"   *> means the same as VALUE "ABCXYZ"

If you follow all of these rules, then your source code will work when either
free-form or fixed-form reference format is in effect.

3) Yes, it is true that the source code line in a program is not a
source-format directive, then you "default" to fixed-form reference format
(in order to be compatible with the existing - '85 - Standard).  However,
there is nothing to say that an implementor cannot use an "external" method
to establish a different format.  (For example, if they already have a way of
specifying compiler options/directives, this can be used for this function
too.)

    ***

Please let me know, if there are any other questions on this that I can
(hopefully) help with.
```

```
To the best of my knowledge the interaction of
   "*>" and ">>"
with XML has not yet been considered by the Standards committee.  I do
believe, however, that Micro Focus (aka MERANT) already has support for both
character strings *and* XML.

I cannot think of any place that a COBOL XML program with have "problems"
with confusing these.  I would think (but this may show my ignorance) that
the XML "symbols" would ONLY appear in a COBOL program - in an alphanumeric
literal.  Such literals are required to support ALL characters and
combinations of characters.  (Actually this is really only true of fixed-form
reference format.  The new free-form reference format allows an implementor
to restrict "special" characters from non-hex-alphanumeric literals.)

Therefore, I don't see that there could ever be ANY set of characters that
couldn't be confused from "meaningful" characters when outside a literal and
a string inside a literal.

Obviously, ">" is already a part of the COBOL language so there is nothing
that can/should be done that would in any way impact its use in COBOL
programs.  Furthermore, it is already used in the "double glyph"  ">=".   And
the JOD (and possibly/probably the Standard after this one) includes "<>" to
mean "not equal to".  (Again, a number of vendors already include this as an
extension.)

As far as the actual characters chosen, I am *so* used to "*>" for inline
comments, that it is hard for me to picture other possibilities. (Micro Focus
has supported these for about a decade now.)  The only "technical" issue that
was discussed at J4 (and WG4) that I still have some "quandary" about is
whether there should be a "terminator" character string so you could have
"inline comments" in the middle of a COBOL line, for example

  Move ABC *> sending field <* to XYZ

where "sending field" is a comment and "to XYZ" is back to COBOL source.
However, I don't think that this "enhancement" is worth adding at this
stage - and it would mean that you would have to "parse" the comment to find
its terminator.

As far as ">>" goes, I was a little happier when an earlier draft required
you to code
    >>DIR
at the beginning of directives (rather than having the "DIR" be optional).
But that led to some really UGLY code for
    >>EVALUATE and >>IF
not to mention
   >>D (debugging indicator)

Bottom-line:
  If you can come up with some examples for J4 or WG4 to think about where
problems may occur with likely XML code, then please let them know.
Otherwise, I don't see this as a serious "open" issue - but that's my opinion
only.

NOTE:
   When OO was just starting, we spent almost a year changing the "inline OO
indicator" each meeting.  We ended up with "::" (double colons) but even that
has some problems that have only surfaced in the last year or so.  (The
problems have to do with the fact that the single colon is *ALWAYS* a
text-separator in COPY/REPLACING.) These types of "aesthetic" issues, really
drive the Standard committee into more "confusion" and circular reasoning
than usual.
```

```
Although I am not "thrilled" by this rule, there is a (semi-)good reason for
it.  The problem arises when the "CR" and "LF" (carriage-return and
line-feed) characters MIGHT appear in an alphanumeric literal.  In fixed-form
reference format, these can "happily" occur in alphanumeric literals -
because the literal *must* continue to the R-margin.  HOWEVER, when you start
having free-form reference format, it is a "tad difficult" to figure out if a
CR/LF (or whatever your "line sequential type file system uses") that appears
somewhere after an opening quotation mark is coded is

   A) a real line-feed (and you have a coding error with no ending quotation
mark)
   B) "data" to be included in the alphanumeric literal.

FYI,
  The exact wording from CD 1.7 are,

"The implementor shall specify any control characters that terminate a
free-form line, and whether such control characters may be specified in
comments and the content of alphanumeric and national literals."

(although I have a VAGUE feeling that those words may be under revision "even
as we speak")
```
