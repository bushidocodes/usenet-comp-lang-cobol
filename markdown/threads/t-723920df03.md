```
I just thought that as a follow-on to another current (and other well past) 
threads, I would start one not on what IS good COBOL, but how one can/should 
judge if a COBOL program contains "good COBOL".  The following are just some of 
what I would use (and not necessarily in order of importance):

1) Does the code actually accomplish what it was designed to do?

2) Does the compiled code run within the performance requirements set out for 
it.  (Better MAY be nice - but not always if other things suffer for performance 
reasons)?

3) Is the code "easily" maintainable and extendable?  (I know that IT *may* be 
moving to components that are replaced not enhanced, but that is not the USUAL 
environment for COBOL programs)

4) If there is any REASONABLE chance that the code will soon (or medium-soon) be 
ported to other environments (OS or compilers) does the code use "standard" 
techniques that are generally available (whether they are in a current standard 
or not)?

5) If those "behind" the program assure the programmer that it will NOT be 
ported, then does the program take appropriate advantage of those features that 
are available but specific to the environment where it is intended to run?

6) (Personal preference, but probably NOT universally expected)  Is there 
adequate documentation of what and how the program does what it does?  Are there 
adequate unit and regression tests available to insure that what works today 
STAYS working when changes are made to it.

7) Does the program use facilities of the language that are either commonly used 
or easily picked up by those who may have to maintain or enhance it?

8) Is the program internally consistent on "style"?  (e.g. No MIXTURES of 
PERFORMs and PERFORM THRUs; either all paragraphs numbered or none, etc)

9) If intended for a "changing part" of the business, it the program/application 
written in a manner that is likely to allow for easy change and enhancement?

10) If there ARE "shop standards" that are expected to be followed (whether they 
actively enforced or not), does the code follow them?  (If there are standards 
that are regularly "ignored" are they at least ignored so that "better" 
approaches can be used?)

11) If those requesting the program say (claim?) that it is a "one time" program 
needed immediately, is it done as quickly and simply as possible and NOT 
complicated for non-essential "beauty" or "style" issues.  (Personally, I never 
believed a manager who claimed that a program would be a "one-time" program, but 
I also didn't spend TOO much time making it so that it would be able to last 
forever)
```
