I have been doing this for just over 2 years now and it is a major part of
my ongoing work.

I wouldn't now develop systems any other way than by connecting components.

You will need a compiler that supports OO COBOL and, as this is still in its
infancy, it is worth getting to grips with Object Orientation concepts (if
you haven't already) so that you can evaluate what is going to be most
useful for you.

As you are in a MF environment you will probably progress to Merant's Net
Express which does support COM and OO COBOL.

However, in my opinion, the best environment for developing components in
COBOL is Fujitsu PowerCOBOL which caters SPECIFICALLY for building ActiveX
components and is ideal for an entry level to the exciting world of
components. PowerCOBOL is intrinsically Object Oriented and fits easily and
naturally into the component world.

If you don't like the sound of building components using OO COBOL, you CAN
put a COM "wrapper" around structured code (at least, you can with Fujitsu;
I'm not sure what NE provides in this area).

While this may make the purists shudder, it is a very effective way to
"leverage" your existing structured modular COBOL into "pluggable"
components which can be dropped into Windows Forms and Web Pages.

Good luck with your endeavours. Mail me privately if you need help or want
to clarify anything.

Pete.

I've just started developing COM using NetExpress, firstly taking some of
their supplied examples then building on it, depending on what you want to
do in terms of your COM design it can be hard in getting examples of exactly
what you want to do.

One thing that perhaps I'm not keen on is the editing of the idl files, each
dll has several idl file that get compiled to make up the type library,
adding a method in NetExpress updates these idl files but when you need to
remove a method or property you have to delete it from the .idl files and
the header file (.h).

The other thing regarding the idl files are if you create a method that
perhaps has one in parameter, "dothis(theparameter) returning hResult, in VB
it comes out like "dothis(p0 as string) As string" so you'll have to edit
the idl files to provide a more meaning full explanation and some help.

Otherwise help is at hand via newsgroups like this or Merant's Answerlabs
time permitting.
