Finally, folks are starting to recognize the need to use Language
Environment as the run time support for their "old COBOL" programs.
While it is still unclear (at least to me) exactly when programs
compiled with OS/VS COBOL and / or VS COBOL II, and linked with the old
run time libraries will stop running, it seems that recent announcement
for CICS and DB2 are finally getting companies' attention.

Programs compiled with the old compilers are supported in execution by
IBM if Language Environment is the run time in use.

Literally thousands of load modules that exist and still execute need to
be migrated to LE support.  The basic method is to run the Program
Binder (Linkage Editor) to replace library routines from the old run
times with equivalent library routines from the LE run time.  We are
ready to do this, from a mechanical standpoint.

What we wonder about is what will happen when we try to execute those
new modules in place of the old ones.  What has been your experience?
Has (almost) everything run first try?  Did you have to go one program
at a time?  What kinds of problems did you encounter?  Do you have
advice for those who are heading down this path?

Thanks,
Colin Campbell

We ran into 2 main problems.  The first with CICS programs because the maps
were not cleared by the program before display and we had garbage in them. 
There is a LE setting to handle this but I don't remember it off hand (we did
the LE conversion as part of our Y2K chaos).

The 2nd was U4035 errors (I remember 4035 - it could be 4038 - or even both).
Anyway what we found was if you define a file in a SELECT and do not have it in
your JCL you get one of the 2 errors above.  The other error is when the length
of the file in the DCB does not equal the length of the 01 level in the
program.  The SELECT error was prevelent with output files defined in programs
but not in JCL, it's the open that triggers is - you don't have to write
anything to it.

As part of this whole process we also tried to get everything AMODE=31 but not
everything cooperated.  We still have a few programs at 24.  One thing that I
did find out when working on getting some of these uncooperative messes changed
was that if you have the DD as REPORT the compile isn't happy unless you have
it as UT-S-REPORT.  Leave off the UT-S and it won't compile.  This could be
caused more by the newer version of Cobol we are using than LE, but if you are
changing compilers you should make note.  Which reminds me of one other thing -
again more to version of compiler than LE.  BMS maps used to assemble with the
attribute byte receiving a suffix of 'F' as well as 'A' (I think one was a
redefine of the other).  Programmers past used to redefine their maps using the
'F' suffix for the attribute byte.  A lot of the maps use the abreviation of
SEL for the selection field on the map and SELF is not allowed by our Cobol
compiler.  Changing to SELA works just fine.

Other than above it wasn't too painful.  Took about a week after the whole
thing was in production to get the rest of the bugs out that weren't obvious at
the time of conversion.

Thanks for your update.  At least one of the shops I have to work in is a CICS
shop, and I have almost no experience in that environment, so this will help.  I
think that's the STROAGE option, and it can be set just to apply to the CICS
regions, if I recall correctly.

With regard to your problem with SEL concatenated to F (SELF), you can specify an
option WORD with a value of NOOO (no object oriented) to make COBOL ignore the
reserved words for Object-Oriented COBOL, including SELF, SUPER, METHOD, etc.

There is also a WORD(CICS) that will ignore reserved words that are not supposed to
be used with CICS programs, like FD, SD, SORT, etc.

You can't use both of these special table together, though.  It is possible to make
custom reserved word tables for your installation, and if your systems programmer
did so, you could both keep out the invalid CICS words and the OO words.

Best of luck!
Colin
