```
I read your original message again, and would like to make a few comments.
These are personal, and I expect you will get a lot of other advice both pro
and con on what I say, so keep that in mind.

First of all, I think it  a bad idea to try to put that logic into the main
loop.  It is not "wrong", but to my mind, poor style.

Style is something that every programmer should work on constantly.  There
is going to always be a conflict between the personal style that you evolve,
and the style that your employer likes, but style is still crucially
important. The reason for that is habit.  As a professional programmer, you
are going to write millions of lines of code in your life.  If you adopt a
good personal style, it will tend to be good code.  If you whack it out,
then try to clean it up to "hand in" after the fact, most of it will be
crap.

Cobol, as a language, was developed specifically to allow a programmer to
develop a self documenting style.  The vast bulk of business code is
written, then changed over and over again to reflect changes in the real
world. The crucial thing to understanding Cobol is that it was developed so
that easily READ code could be written.  Note that.  Not so that code would
be easy to write, but so that code would be easy to READ.  The Cobol style
that you evolve should reflect that.

IMO, the most important tool that you have for readable code is the
paragraph name.  Each paragraph should not only be reasonably short, but
reflect a specific task, and the name of that paragraph should tell you
exactly what to expect that code to do.  Each paragraph should fit on a
screen, ideally, so that you can scan it easily.

Cobol has the ability to write straight line code, like any other language.
You can nest performs and try to document them with indentation, C-style, to
your hearts content.  However, doing so adds almost nothing to efficiency,
and defeats the purpose of the language to a very large extent.  The
overhead of a perform is negligible in 99.9% of all cases.  The only time it
will make much difference is in that rare (for business code)  case of a
highly computationally bound program doing weird mathematical computations
millions of times ... things like matrix multiplication.

Having said all that, I would write the code you want something along the
following lines.  Note that the code is simple and easy to follow.

01 END-FLAG                 PICTURE X VALUE "N".
       88 END-OF-FILE       VALUE "Y".
***
MAINLINE.
       PERFORM START-UP.
       PERFORM MAIN-LOOP UNTIL END-OF-FILE.
       PERFORM CLOSE-OUT.
       STOP RUN.

START-UP.
       OPEN INPUT FILENAME.
       READ FILENAME AT END
           PERFORM EMPTY-FILE-ERROR-MESSAGE
           STOP RUN.
       MOVE RECORD-CODE TO CURRENT-RECORD-CODE.

MAIN-LOOP.
       IF RECORD-CODE IS NOT EQUAL TO CURRENT-RECORD-CODE
           PERFORM SUBTOTAL-ROUTINE.
       PERFORM PRINT-THE-RECORD-ROUTINE.
       READ FILENAME AT END
           MOVE "Y" TO END-FLAG.
******* MANY PEOPLE WOULD USE "SET" HERE

CLOSE-OUT.
       PERFORM SUBTOTAL-ROUTINE.
       CLOSE FILENAME.
```

```
Its workings are kind of counter-intuitive. Intuitively, we understand sorting
things starting with the most significant digit and working things down to the
least significant. LSD-sort, by its own name, is a least-significant-digit sort,
that is, is starts from the least significant digit and works its way from right
to left until the file is in order. It is of paramount importance here that the
method of sorting the keys based on their n-th digit from the right is _stable_,
that is, the relative order of the records remains intact (quicksort, for one,
does _not_ possess this property). Usually, key-indexed sort is used, but there
is also a way of using a linked structure to do this, thereby avoiding moving
records at all. As an illustration, consider this example, where the numbers are
sorted in the stable manner based first on the rightmost (LS) digit, then the
second, and finally, the first (the leftmost column represents an unsorted
original file):

975 142 203 028
142 203 028 142
028 384 142 197
384 975 975 203
197 976 976 384
589 197 384 589
203 028 589 975
976 589 197 976

Note that the sorting time is proportional to the number of digits and keys, that
is, O(N). The only other sort  capable of achieving this is the proxmap (address
computation) sort. Furthermore, it is not necessary to sort based on just _one_
LS digit. Having enough memory, it is easy to use 5-6 rightmost digits at a time
(i.e. 100000 ot 1000000 radix) to do LSD sorting. The file above can be actually
sorted in a single pass by keying its keys into a 1000-element table and
maintaining the counts of each key in the bucket whose index equals the key.
```
