```
2000-08-11 12:45:00 MEST

Anyone of you who ever had the experience to
peek into the .../include/ directory of a Unix
System in order to find out about some details
of the interfaces knows the hierarchy of
nested #ifdef ... #endif intended for adaption
of the header-files to different environments.
Whereas a C-Compiler digests that quite easily,
the human reader has some difficulties to find
out what the code actually means.

I have seen header-files with hunderd of lines
for the declaration of just a few functions or
constants only. In some cases, one finds out,
after hard thinking, that a header file is
only a redirection to another header-file.

And now, OO-Cobol will have the
conditional compilation feature as well.

I am not arguing against conditional
compilation - but what can be done to
prevent the writing of such unreadable
code as I have described?

Conditional compilation *could* be wisely used
- just like alkohol - but I am afraid that we
will see deeply ">> if .. >> end-if" bracketed
code which will be a nightmare to read.

Is it wise that a compiler issues a warning
if to deeply nested conditional compilation
is encountered?

Herwig
```

Herwig,
  You might want to check with those who are used to such facilities.  I know
that those who have used Micro Focus conditional compilation have done so
QUITE HAPPILY for years - including VERY deeply nested IFs.

Conditional compilation (and compiler variables) also begin to include SOME
of the facilities of the PL/I precompiler (macro facility).

I think that you will find (especially with both >>IF and >>EVALUATE) that
people will want to use these as much as they do such constructs in "normal
code".

There is one feature that is NOT in the draft Standard, that I think most
vendors will find that they need to "add" as an extension. (Again, it is
already in the Micro Focus compiler) and that is a >>DISPLAY statement.  This
can write either a literal or the value of a compiler variable to the
compile-time output.  This is EXTREMELY useful in debugging conditional
compilation problems.

I was quite disappointed that J4 "rejected" adding it to the draft Standard.

I agree to that - and of course I have that in the debugging
code - that is the part which will never find it's way to the
customer. (That is one of the privileges of the implementor!)

But one thing has to be considered: as for now, compiler
variables can be numeric, can have string literal values,
and if I interpret the Draft correctly, can also have
Boolean literal values. That is a complete type concept
which is quite different from the type concept of
Cobol proper (Pic clause and everything). That means, we
have two data type concepts in the same language!

I am of course not sugessting that compiler variables
should have the full spectrum of Cobol data types - that
would be utter nonsense. But as it is now, it is two
different data type concepts, and the standard will be
criticized for that.

For instance, in chapter 7, where directives are described,
the reader is refered to chapter 8 (or so) in order to learn
about arithmetic expressions. However, because of the
different data type concepts, the arithmetic expressions in Cobol
proper and in the directives cannot be quite the same. There must be
subtle differences.

Well, on a second thought, also C has two data type concepts
that way, although the "#define ..." is a very primitive
one.

Herwig
