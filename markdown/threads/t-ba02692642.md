I am considering a new venture in the coming year and would like to know if
anybody is using, or is capable of using, OO COBOL to make ActiveX (COM)
components. I am also interested in people who have .asp knowledge with
VBScript or J(ava)Script.(Especially if you have built components for web

Components that can run as server side code (contain no user interface) are
of particular interest, but so are some that have windows msgbox interface.

also be of interest.

I am particularly  interested to trade components and know how, and possibly

Pete,

While developing our software product line, my company has developed a
huge number of COBOL DLLs that use the standard windows API
convention. These DLLS are accessed by COBOL, VB, JAVA and ASP modules
today. Our thoughts were to turn them into serverside com objects at
some point in the future. We just haven't had the incentive to do it
yet. One thought I had was to develop a server side and client side
com wrapper that would trap all client side API calls, do the
appropriate call to the server side com wrapper, which would then call
the appropriate DLL entry point and then vice-versa. How difficult do
you think this would be? Can you suggest an alternative approach
besides rewriting all the DLLs?

We have DLLs that do UPS, FEDEX, Airborne and LTL carrier shipping. We
have DLLs that talk to scales and barcode printers. we have DLLs that
completely support web based sales orders, including credit cards and
electronic checks. All you have to build is the UI. I have many more
that I could list.

Could what we've done possibly fit what you are looking for?

John

I am definitely interested in learning this stuff and I think I am capable.
 Downside is I do not have anything concrete to contribute at this point. 
I do have some books on these topics but most of them have not been read
yet.

I have been concentrating  on learning OO concepts and I feel comfortable
with most of them them but I have not used them enough to feel confortable
with OO design principles.  I can still get thrown off by some comments or
new ideas in a new language.  For instance the other day someone mentioned
that OO prog/design is bottom up as opposed to top-down.  I am not certain
if this is true or not.  It seems to me that for the most part OO
technology is a "forced" repackaging of data and functions. Various
languages provide varying support for OO resulting in in different program
organizations. The key concept seems to be abstract data types/user defined
data types. Also depending on what you are reading there is usually a
"laundary list" of concepts inculed in OO: abstraction, encapuslation,
overloading, polymorphism, etc depending upon what the various languages
support.

If a traditional structured program is viewed via a hierarchical structure
chart then an OO program seems pretty similiar except you find objects at
many of the nodes instead of functional paragraphs. The functions within
each object still seem pretty procedural to me and the high level logic
that gets everything started is often procedural code hiding under a
generic object name ending in "Manager". I guess what I am trying to say is
it still seems pretty procedural and not as revolutionary as is often
stated.  I post this at the risk of seeming a little naive about objects
and I admit I am lacking in practical experience.

Sorry, I think I lost my focus/purpose for this post a little. 

Despite all of this I agree that the advantage of OO seems to be in
leveraging components and I want to learn more about them.  I am leery of
M$ only solutions while admitting they are worth learning about.

The amount of time I can devote is unclear at this point as I am being
absorbed into the new Dept. of Homeland Security and whatever new demands
that will bring is unknown at this point.  I seem to learn better when I
have a specfic project or see a pressing need coming soon in the future.
This helps concentrate by focus on what I need to learn.  I think this
comes not from being lazy but from learning many things that then were not
used.  If I don't use it I seem to lose it. In my current job I don't see
myself being called on to use this technology anytime soon, but I want to
be ready when the time comes. I should say if it comes because ther is
distinct possibility I may retire before that time comes.
