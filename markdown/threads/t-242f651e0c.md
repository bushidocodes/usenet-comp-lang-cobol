I was having some discussions with some very competent and experienced 
(mostly COBOL...) programmers in Australia.

During the course of these discussions I was asked if there were any 
"universal rules" of programming I would particularly endorse, and I 
responded instantly: "Small is beautiful."

There was some surprise and I realised that I had just responded 
instinctively, based on what I actually believe.

It is based on some things I learned many years ago (from people much wiser 
than me, who have now departed to the big computer room in the sky...) and 
have applied ever since.

Irrespective of  Programming Language, there are three main factors that 
determine performance on computer systems:

(Oddly enough, this has been true since the centralised mainframes of the 
1960s and is still true today even for networked PCs. The fact is that any 
machine which is subject to the Von Neumann sequential processing model 
will be subject to these criteria.)

1. Queue wait time.

(How long your executable or .DLL is waiting in a system queue before it is 
loaded. No matter how good your code is, it is worthless until it is in 
memory...)

2. Load time.

(The longer your code takes to load, the less efficient it will be perceived 
to be. If there are no available resources and your code requires a fair bit 
of memory, it may wait longer than you would like, to be loaded. See 1. 
above... If your program is "large" it will take considerable time to load 
(especially if it has to be paged on load)). If it gets paged out, the whole 
process has to be repeated by the OS... it could happen many (thousands) of 
times...

3. Capture time.

(Once you are in and executing, how long can you effectively hold the CPU? 
Programs that are I/O bound will make small demands on system resources 
because the CPU is normally released until the I/O completes, and "smart" 
OSes will use these cycles to process other tasks. Other architecures may 
have separate I/O processors or "smart" channels (like Block Mux) that can 
take the load off the CPU. Either way, generally you don't want to be a CPU 
cycle hogger - the OS and schedulers will penalize you if you are.)

Obviously, smaller (modular) code meets all of these criteria better than 
huge monolithic programs.  Many of us learned that (particularly for online 
systems, where performance is still important) , writing small modularized 
codeblocks  made the code easier to manage and load and performance was 
generally improved by keeping things tight and small.

The advent of Object Oriented Programming reinforced these lessons. Now code 
could be encapsulated into a series of small functions which loaded quickly, 
executed quickly, and could be easily shoved around and relocated by the OS. 
Not only that, but these functions could be instanced for any number of 
different processes as and when required, without huge overheads.

So, the bottom line is that "small is beautiful".

But I realised as I had this discussion that it doesn't stop there.

If you use a language which takes less code to write (is therefore more 
powerful) , there are savings to be had not only in development time, but 
also in maintenance time.

I posted some code here in COBOL for HeyBub's Soundex routine. It was around 
300 lines.

Here's the same solution in C#:

    public static string Soundex(string word, int length)
    {
        // Value to return
        string value = "";
        // Size of the word to process
        int size = word.Length;
        // Make sure the word is at least two characters in length
        if (size > 1)
        {
            // Convert the word to all uppercase
            word = word.ToUpper();
            // Convert the word to character array for faster processing
            char[] chars = word.ToCharArray();
            // Buffer to build up with character codes
            StringBuilder buffer = new StringBuilder();
            buffer.Length = 0;
            // The current and previous character codes
            int prevCode = 0;
            int currCode = 0;
            // Append the first character to the buffer
            buffer.Append(chars[0]);
            // Loop through all the characters and convert them to the 
proper character code
            for (int i = 1; i < size; i++)
            {
                switch (chars[i])
                {
                    case 'A':
                        currCode = 0;
                        break;
                    case 'E':
                        currCode = 0;
                        break;
                    case 'I':
                        currCode = 0;
                        break;
                    case 'O':
                        currCode = 0;
                        break;
                    case 'U':
                        currCode = 0;
                        break;
                    case 'H':
                        currCode = 0;
                        break;
                    case 'W':
                        currCode = 0;
                        break;
                    case 'Y':
                        currCode = 0;
                        break;
                    case 'B':
                        currCode = 1;
                        break;
                    case 'F':
                        currCode = 1;
                        break;
                    case 'P':
                        currCode = 1;
                        break;
                    case 'V':
                        currCode = 1;
                        break;
                    case 'C':
                        currCode = 2;
                        break;
                    case 'G':
                        currCode = 2;
                        break;
                    case 'J':
                        currCode = 2;
                        break;
                    case 'K':
                        currCode = 2;
                        break;
                    case 'Q':
                        currCode = 2;
                        break;
                    case 'S':
                        currCode = 2;
                        break;
                    case 'X':
                        currCode = 2;
                        break;
                    case 'Z':
                        currCode = 2;
                        break;
                    case 'D':
                        currCode = 3;
                        break;
                    case 'T':
                        currCode = 3;
                        break;
                    case 'L':
                        currCode = 4;
                        break;
                    case 'M':
                        currCode = 5;
                        break;
                    case 'N':
                        currCode = 5;
                        break;
                    case 'R':
                        currCode = 6;
                        break;
        // Check to see if the current code is the same as the last one
                if (currCode != prevCode)
                {
                    // Check to see if the current code is 0 (a vowel); do 
not process vowels
                    if (currCode != 0)
                        buffer.Append(currCode);
                }
                // Set the new previous character code
                prevCode = currCode;
                // If the buffer size meets the length limit, then exit the 
loop
                if (buffer.Length == length)
                    break;
                // Pad the buffer, if required
                size = buffer.Length;
                if (size < length)
                    buffer.Append('0', (length - size));
                // Set the value to return
                value = buffer.ToString();
            }
            // Return the value
            return value;
        }
    }

Note that it is fully commented and it has not deliberately been made 
shorter.

No attempt was made to make either the COBOL or the C# smaller for this 
exercise, but the C# is just 159 lines; around HALF of the COBOL.

I'm not really interested in discussing the elegance or otherwise of the 
actual code; my point is that less code costs less. It costs less to develop 
and it costs less to maintain. It is easier to search through less source 
code when you need to "fix" it than it is to search through thousands of 
lines.

Both at run time and at development time, small is beautiful.

If you remember nothing else I ever posted here, remember that... :-)

Pete.

>                 switch (chars[i])

                 EVALUATE char(i:1)
                 WHEN "A"     currCode = "0"
                 WHEN "E"     currCode = "0"
                 WHEN "I"     currCode = "0"

relationships.

Even better, easier to understand, and smaller is:

                 EVALUATE char(i:1)
                 WHEN "A"
                 WHEN "E"
                 WHEN "I"
                 WHEN "O"
                 WHEN "U"     currCode = "0"
                 ...


While the switch does unroll the loop compared to the original it is
still a poor performer because of the number of comparisons. If the
switch statement only allows bytes to be compared (as does Pascal and
C) the compilers can create a branch table which is considerably
faster that a string of comparisons. I don't know it the C# switch is
limited to this, but the COBOL EVALUATE is _much_ more powerful.

If it is speed and compact code that is required then the best is in

Soundex: procedure (pword) returns (character(100) varying);
   declare pword character (*) varying, value character
(length(pword)) varying;
   declare word character (length(pword));
   declare size fixed binary, (prevCode, currCode) character (1);
   declare i fixed binary;

        word = pword;

        /* Value to return */
        value = '';

        /* Make sure the word is at least two characters in length */
        if length(word) > 1 then
           do;
            /* Convert the word to all uppercase */
            word = uppercase(word);

            /* The current and previous character codes */
            prevCode, currCode = '0';

            /* Place the first character in the buffer */
            value = substr(word, 1, 1);

            /* Loop through all the characters and convert them to the
proper character code */
            do i = 2 to length(word);
               select (substr(word, i, 1)) ;
                    when ('A', 'E', 'I', 'O', 'U', 'H', 'W', 'Y')
currCode = '0';
                    when ('B', 'F', 'P', 'V')
currCode = '1';
                    when ('C', 'G', 'J', 'K', 'Q', 'S', 'X', 'Z')
currCode = '2';
                    when ('D', 'T')
currCode = '3';
                    when ('L')
currCode = '4';
                    when ('M', 'N')
currCode = '5';
                    when ('R')
currCode = '6';
                    otherwise return ('0');
               end;
               /* Check to see if the current code is the same as the
last one */
               if currCode ^= prevCode then
                  do;
                   /* Check to see if the current code is 0 (a vowel);
do not process vowels */
                   if currCode ^= '0' then value = value || currcode;
                  end;
               /* Set the new previous character code */
               prevCode = currCode;
               /* If the buffer size meets the length limit, then exit
the loop */
               if length(value) = maxlength(value) then leave;
            end; /* of do i = ... */

            /* Pad the buffer, if required */
            value = left(value, maxlength(value), '0');
        end; /* if */
        return (value);

    end Soundex;
