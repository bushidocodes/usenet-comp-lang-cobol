In the CICS regions I program for, there are dummy terminals defined
for specific tasks.  This allows us to make certain long-running tasks
single-threaded, such as an asynchronous file-writer, for example.  In
older releases of CICS, the dummy terminals seemed to require special
DD statements in the startup deck for readers and punches.

Starting transaction which already has a terminal (including a dummy
terminal) will queue up a message to that task.  Most of the started
tasks I work with have no user interface screen.  We have some tasks
that used to have VTAM/SNA network interfaces, but have been heavily
modified to use TCP/IP.  They still have dummy terminals.

You can achieve the same function by queuing a message to a DCT that is

ensure that only one transaction is run against the queue at a time.

Sounds great, in theory, but probably wouldn't work as well as you'd
think.  Sure, you could have it do what you said -- communicate via
HTTP.  But not all clients speak HTTP.  In fact, not all of the "TCP/IP
terminals" are clients.  Here are the TCP/IP devices I need to
communcate with and their attributes:

Racal Host Security Module.  Functions as a server.  Messages are
prefixed by a two-byte binary (network byte order) length header. 
Messages are in EBCDIC or ASCII depending on how you've chosen to
configure it.  Also has a UDP option.

VisaNet USMG.  Functions as a server.  Messages are prefixed by a
four-byte header.  The first two bytes are the message length in binary
(network byte order).  Bytes 3 and 4 are ignored.  Messages are in
EBCDIC.

MasterCard.  Functions as a server.  Messages are prefixed by a two-byte
binary (network byte order) length header.  Messages are in EBCDIC.

NCR ATM.  Functions as a client.  Messages are prefixed by a four-byte
ASCII numeric header.  Messages are in ASCII.  Header can be configured
to be three-byte ASCII numeric.  Header can be configured to be two-byte
binary in network byte order.  Header can be configured to be two-byte
binary in Intel byte order.

VisaNet connection sends a 'heart beat' message after 90 seconds of
inactivity.  This message is basically just a header with the length set
to zero.  MasterCard also supports this as an option.  They also both
support the host sending these messages at a smaller interval.  The
others do not support this option.

All four device types use 'always on' connections.  At CICS startup I
will connect to all server-type devices.  I also start up a 'listener'
socket to allow connections from the client-type devices.  Each
connection stays established the entire time CICS is up (barring
problems, of course!).

My point here is that there's no way IBM could possibly anticipate all
of these different 'protocols', no matter how similar they may be.  The
"TCP/IP Communcations Manager" that I am writing handles all of them. 
Each "terminal" has its options configured in a VSAM file.  But
certainly we may get yet another 'device' type with some protocol
specifications that I did not anticipate, so at that point I would have
to add more features to my program.

As much as I wish all of this had been done for me already I can
certainly see why it was not.
