At least on IBM mainframe applications the is really no reason to code
OCCURS DEPENDING ON in any working storage table. Space will have to
be allocated off of a BL cell for the maximum possible storage size of
the table no matter what. Remember that when you define a variable
table, by definition the space is dynamic and has to becapable of
holding the maximum number of entries regardless of what's actually
used.

Where I used to work, ODO's were used a lot in files to 'save' space
with repeated data.  Even though it usually had only minimal effect
due to poor knowledge by the developers about how their data would
actually look in the fiels after a couple of years of use.  (a
favorite example of mine  was two years of payment history with a
average number of entries of about 18 out of a max of 24 and a
standard deviation of about 2). This kind of file also caused cpu
usage by the applications to skyrocket if the data was referenced in
the buffer because of the offset recalculations on every reference of
data elements. It was very difficult to pursuade programmers to move
the data to working storage definitions that did not have the ODO's in
the tables.  When we did, the resource reductions in some programs was
spectacular. 

I would be dubious of any significant savings when using SEARCH ALL.
This format of the SEARCH verb uses a binary search rather than
sequential and except in extreme cases the number of probes into the
table to find an element should not be that great.
