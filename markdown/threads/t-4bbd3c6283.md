I was speaking to a colleague this morning who has been told to teach the
following style:

1) Start off the working-storage section with a string like
"WORKING-STORAGE STARTS HERE".

2) Use READ THISFILE INTO WS_RECORD rather than just READ THISFILE.

She mentioned that this had something to do with searching through dumps
for the values inside the records.

it ever was, of course), today.

This sort of "handle" was/is used very commonly for just the cited reason
-- it makes it easier to locate areas if you are still unlucky enough to
have to read dumps of failing programs.

More modern environments obviate the necessity for stuff like this. I
haven't looked at a printed dump in at least 10 years. Our IBM mainframe
environment includes ABEND-AID which does the dump processing for me and
lets me look at working storage tagged with the actual variable names.
Also, the use of an interactive debugger, such as Xpediter, further
reduces the need to resort to these habits.

Nonetheless, old habits die hard and I still bring in contractors who
insert these "handles" into the code they write for me.

› 2) Use READ THISFILE INTO WS_RECORD rather than just READ THISFILE.
›

I have not heard dumps mentioned as a reason here, but I suppose it might
be easier to find the record in a working-storage area rather than trying
to find the file buffer itself. When I first started programming (long,
long ago -- don't ask!), rules like this were commonly preached because
various actions would cause the actual record area to be invalidated. For
example, on one of the first machines I programmed for, if you MOVEd
values to the record area, and then did a WRITE RECORD-AREA, the record
would be written and the contents of the record area "wiped out". Thus,
you couldn't simply make a few minor changes and write the same record. As
a result, we were always taught to build the record in working-storage and
then do a WRITE RECORD-AREA FROM WS-RECORD, so that you wouldn't get
caught by the invalidation of RECORD-AREA. This kind of problem doesn't
seem to exist for the machines I currently work on, but I still habitually
code the "old-fashioned" way.

Bottom line is that you need to know the machine for which you are writing
code *and* you need to program defensively to a certain extent, because
machines change and rules change over time.

The programming dinosaur here tells us that back in the 360/370 days
the file buffer area was located in the system's address space and
therefore inaccessable to the user. By using the READ INTO you
could get the data into user space and see it with a debugger.
And even if you could look into system space, the file
buffer area was malloc()'d & free()'d as needed and therefore
was never in the same place twice. By using WORKING-STORAGE the
data stayed around after the WRITE.

The first technique is used to help locate where the working storage begins in
a dump. This is used at a lot of facilities where they do not have a way to
detail what has happened to cause the dump. Where I work we use Abend-aid which
gives a a hex listing for each 01 level in working storage. Abend-aid can also
be set up to display all of the elements in the working storage section. The
second technique is used to help locating where an abend has occured in a
program. Abend-aid also gives a hex listing of each file buffer as it appeared
prior to the abend. If you don't have a product like this it usually a good
idea to use these technique just to help debug a program that has abended.

They have indeed Robert, but, like everything else, they are good for
some tasks only. A (perhaps extreme) instance: a program I wrote ended up

It's nice to have tools like symbolic dumps handy in the toolbox. It's

comment applies to the mainframe-versus-the-world slanging match too.

Sorry, Robert, but there are times when such things are needed. Once
an application is thought to be debugged, it is put into regular
production, with no dynamic debugger in sight and no ace programmer
in control to jump on problems. Obviously, nothing is completely
debugged. If it bombs, nobody is sitting there with his handy-dandy
debugger to find out what is going on. It has to get out of the
system because it could tie up resources for days until somebody came
around to check. When this happens at Tandem, a special snapshot of
stuff called a saveabend can be created. Later, the programmer can
invoke the interactive debugger referencing this file and it is just
like you were there when the problem happened. You use the normal
stuff to figure out what happened. I guess this really isn't a dump
exactly, but if we didn't have this and required every application to
be running with a programmer sitting there waiting for the job to
bomb, there would be a few complaints.
