The proposal to handle the multi-byte character sets, especially
ISO-10646 has one major problem - ISO-10646 does not have a collating
sequence. The reasons are multifold and the following comments are based
on my understanding of the Unicode manual and various presentations at
SHARE, an IBM mainframe user group. The first is that in some cases two
physical characters can combine to make one logical character such as the
accented characters in many of the European languages (one of the two
characters is non-spacing). A second is that the accented characters do
not have the same position in all of the languages in which they appear.
Another is that there are different orderings based on usage even in
English. The telephone directory order is different from the dictionary
order. I heard a presentation on the way you have to break down a word in
French to determine where it sorts and the alogrithm could not have been
handled by a collating sequence.

The challenge to the industry (much of which is still in "the good old
days" of upper case English letters only) is to come up with ways of
handling text. Under what circumstances do we want to say CLARK is equal
to Clark? I suspect we are going to need something like SORT SORTWK
ASCENDING KEY FIELD-1 Using Algorithm for French-dictionary-order or
German-name-order. I would suggest that the appropriate ISO groups (WG20
would be one if I read the Multiple Octet Character Set proposal correctly)
and the internationalization groups within the various companies (for
example, the IBM group that is, or at least was, based in Toronto) be
consulted.

I believe that for text processing which includes names, the days of being
able to depend on the collating sequence of a character set for either
comparisons or sorting are fast coming to an end. Even today I suspect
that saying "ABLE" is greater than "Acid" (if the comparision is in
EBCDIC) does not agree with what most people would expect. I would assume
that the developers and improvers of the various procedural languages are
facing similar problems and joint efforts may prove useful. The problem
is difficult. Defining the language to handle the complexity of the
situation in a clear manner will be difficult, probably more difficult
than implementing the language once it is defined. I recommend a process
of co-ordinated development of extensions by the vendors to see what works,
preferably using pre-processors to generate current COBOL so that programs
are not completely obsoleted when the agreed upon method is found.

These comments are also posted to comp.lang.cobol

Theres no need for preprocessors. X/Open have been working on these sort
of problems so that you can select a language, territory, etc. with an
environment variable and the operating system provides the string
comparison, etc. Micro Focus COBOL has a compiler directive (Ugh, I know)
that causes all comparisons, upper/lower, etc in COBOL to be carried out
with the language sensitive operating system support rather than the
standard ones. So you just put all your language sensitive stuff into
a separately compilable module and set the directive. There is no new
syntax, just use the same old IF, SORT and indexed files, etc and they
will all use the collating sequence the user selects at run time.

There is similar but more rudimentary support under Windows but the COBOL
run time adapts to it to make the functionality portable. Well, portable
after a fashion, but then the collating sequences the user chooses under
the X/Open scheme can be pretty variable as well.

As far as I know, only 8-bit chars are supported at present tho'.

For the reasons you point out, no one has ever used the collating sequence for
processing names. Take directory work, for example:
MAC ADAM, CLARA
MACADAM, CLARK
MACADAM, CLARICE
MAC ADAM, CLAUDE

How did we print directories like this, in 1960, in CoBOL, when only upper-case
letters were available? Well, the sort key field is not the same as the print
field. That's also how you get numbers to sort as if they were spelled out:
7 BROADWAY [SEVEN BROADWAY ]
SEVEN HILLS [SEVEN HILLS ]
7 SEAS [SEVEN SEAS ]
SEVENTH-DAY ADVENTIST [SEVENTH-DAY ADVENTIST]
76 MAIN [SEVENTY-SIX MAIN ]

These capabilities have never been provided by sorting the print field itself,
and there's no reason to do it that way. You will always need a separate sort
key no matter how the collating sequence may be enhanced. No one in his right
mind would compare "ABLE" to "Acid" without normalizing first according to the
application.

A separate sort key is indispensable to dictionary work, (which first became
possible when the Friden Flexowriter came out not just with lower-case letters
but with all the diacriticals):
la-nate [LANATE ]
Lan-cas-tri-an [LANCASTRIAN]
lance [LANCE ]
Lan-ce-lot [LANCELOT ]

Every dictionary ever compiled uses different alphabetization rules. It's not
reasonable for the language to limit you to one particular "French dictionary
order". The alphabetization rules belong in a subroutine that generates the
sort key from the print field, not in the language definition. (I'd have
included an example of the French sort you mention if I could have entered it
from this keyboard.)

The COBOL standard in this case is ahead of what the industry is doing. I
work on a Unisys A Series machine. The A Series COBOL74 compiler was
extended to use Unisys specific syntax to handle different characters. The A
Series programs are able to work in a different character set than the
default "english uppercase". The program could sort (in logical or
equivalent sequence), read a file, access a database or Indexed files (in
logical or equivalent sequence).

In COBOL85, Unisys will be using the "National Character Handling" feature
of COBOL85 to implement the different character sets.

In all cases you can define how the comparisons will be treated. In your
example "Clark" can be defined to be equivalent to "CLARK", or that "HOTEL"
is equivalent to "H魹ｽtel" (circumflex on the o).

All these points are already available in the COBOL "National Character
Handling" standard.
