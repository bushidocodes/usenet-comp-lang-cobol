were a formal description as a starting point, that is,
a formal description Cobol 85. But I would not attribute
the word "formal" to the Cobol 85 describing document.

the actual needs of programming life.

However, it was the same problem in the old days when I
was in an Ada-83 project. Endless disscussions about
the meaning of the evolving LRM. Many surprises when
implementing it - always new surprises lurking behind
innocent formulations in the manual. Ada-83 was an
entirely new language, and in spite of that, the language
description did not meet my preferences for precision.

In older yet days, I was in a government project where
I was investigating existing methodologies of
Requirement-Engineering. That's an important job
because there are indeed software-projects being launched
without anyone having a clue what the final product shall
accomplish. Then I was changing to compiler-writing, and
I thought: No problems anymore. The requirements for
a compiler is mostly a description of the language
to be compiled. There can't be any difficulties. That's
what I really thought.

Oops.

"9.3.4. Methods

The procedural code in an object is placed in methods. Each method has
its own method-name and its own data division and procedure division.
When a method is invoked, the procedural code it contains is executed. A
method is invoked by specifying an identifier that references the object
and the name of the method. A method may specify parameters and a
returning item".

I am working solely in OO. If I read the above for the first time, it is
as clear as mud. Initially I was 'duped' by the last four words - saying
to myself, "Well what do you do when you want to return......".
It needs expansion, plus EXAMPLES. You can only return ONE LEVEL 01 item
but that may also be split down to Level 05s etc to return one or more
data items or object references.

points, note that Chapter 9, containing that method reference is listed
under :-

	9. I-O, objects and user-defined functions

aspect of the Standards.

page, possibly two-thirds and with it I would emphasize the point by

Well, ...
  IMHO *any* substantive (and controversial) change at this time will be
looked at QUITE skeptically by the Standards committees. (and your thoughts
are both) However, please feel free to send a note/paper if you think there
is a "real" problem (not just a "why bother" issue).

OVERRIDE is critical, in my opinion.  It allows you to CHANGE the value of a
compiler variable during a compilation.  "Normally" it would occur within the
scope of an >>IF or >>EVALUATE statements. (BTW, "override" functionality
does NOT work if it is the default, unless there is some other syntax
introduced that works the opposite way.)

As far as "parameters" goes, this is another "feature" that I used heavily
with the Micro Focus "design".  In that environment, you could define a
"constant" in three ways:

 - 78 level (in source code)
 - CONSTANT directive (command to compiler)
 - $CONSTANT (half way between the previous two)

I think that the draft would really be hampered if all 3 variations were not
"possible" (albeit via new syntax) to those wishing to use them.

In the old days when I worked in an Ada (83) compiler project,
I had the idea of "Ada-lite". Interesting enough, it did not
catch on, for whatever cause.

In the case of Ada, some subsets could have been defined
very easily. For instance, you cold take "tasking" away and
still have a genuine subset. For other things to be
excluded, the situation was not that simple. Generics, for
instance, leave their trace in the simplest "Hello world"
program, so there was no subsetting possible there.

Now there might be one reason why the idea did not catch on:
Ada was entirely new. No bunch of old programmers who were
brought up in Ada. No one to ask how and which things were
useful in everyday work.

This of course is different with Cobol where we have the bunch
of old programmers - some of them even, which is essential,
familiar with many plattforms. The expertise of defining a
useful hierarchy of language-subsets is there.

However, that is additional work. Look at the CD 1.8 - it
approaches in length the bible, and the time of completing
the draft has a tendency to move forward in time. All the
thinking about making the language free of contradictions
etc had to be repeated for every language subset. Who shall
do all that work?

Maybe that is (one of ) the reason(s) that there never will be
a subset to any new version of Cobol.
