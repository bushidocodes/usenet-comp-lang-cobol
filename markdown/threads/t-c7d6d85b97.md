```
First of all, let me state that I have already solved this problem  This
said, let you show you what my problem was.  Here is a copybook known
as CVCMUCHR:

01  CHR-USER-RECORD.
    05  CHR-FIXED-DATA.
        10  CHR-MISC-DATA            PIC X(1298).
        10  CHR-DEPENDING            PIC 9(3) COMP-3.
    05  CHR-VARIABLE-DATA            OCCURS 10
                                     DEPENDING ON CHR-DEPENDING.
        10  CHR-ACCOUNT              PIC 9(18).
        10  CHR-ACCOUNT-TYPE         PIC X(3).
        10  CHR-MORE-DATA            PIC X(17).

(CHR-MISC-DATA and CHR-MORE-DATA aren't really the data names, but I
didn't want to include all of the various fields.)

Now here's the thing.  These records are kept in a variable length file
call CHF (cardholder file).  The records vary in size from 1300 bytes
(for when there are no accounts attatched) to 1680 bytes (when there are
10 accounts attached).  Up until now, all of the programs that access this
file are written in DOS/VS COBOL.

I wrote a VS COBOL II program to read this file from a tape, forward some
dates by a certain number of days, and write the resulting record out to
a VSAM file.

The problem was that in some cases I would get an error status code of
'44'.  According the the COBOL II manual this means that I was trying to
write a record that was either smaller than the smallest defined record
for my FD, or larger than the largest.

What happened is in COBOL II, if you use "OCCURS ## TIMES DEPENDING ON .."
rather than "OCCURS #1 TO #2 TIMES DEPENDING ON.." it defaults the lower
end to *1*, when what I needed it to be was ZERO.  So what I had to
do when copying the copybook in was say "COPY CVCMUCHR REPLACING
==10 TIMES== BY ==0 TO 10 TIMES==."

So now it works.  Wierd thing is, I looked in the DOS/VS manual and it
appears to me to not even *allow* you to say "OCCURS ## TIMES DEPENDING
ON .."  But apparently it *does* allow that, and it treats it like
"OCCURS 0 TO ## TIMES DEPENDING ON ..", while COBOL II treats it like
"OCCURS 1 TO ## TIMES DEPENDING ON .."  How irritating!  In fact, the
DOS/VS COBOL manual we have not only says you have to use the "TO"
format, but that the low boundry must be a POSITIVE number!  (In other
words, in cannot be zero (or a negative number)).  Again, the manual
seems to be incorrect.

In any case, I will probably change the copybook to say "0 TO 10 TIMES".
That is, assuming DOS/VS will allow this!  Yikes.

Any comments?  I wasted a few hours today figuring this out, and a few
more hours complaining about it!  :-)
```

```
Frank,

    You can specify an unsigned integer (defined in WORKING-STORAGE) and use
a COBOL2 FD option, that basically reads 'RECORD IS VARYING IN SIZE FROM
xxxx to xxxx DEPENDING ON unsigned integer'. If you use a signed integer,
you'll get a COMPILE Error. For whatever reason, Big-Iron COBOL generates an
ALR (Add Logical Register) instruction (instead of a regular Add Register)
in the OPEN and CLOSE Macro expansion (check the PMAP) when calculating or
setting a record length. ALR implies that both registers contain unsigned
values (High-Order bit turned-off). Therefore, a signed integer throws a
real monkey wrench into the mix (It COULD be negative. High-Order bit
turned-on). The silence of this subtlety has been deafening in the COBOL
community. Go figure.

    Anyhow so much for my ranting. After you've read a record, the unsigned
integer is set to the actual length of the variable length record. Before
you write a record, set the unsigned integer to the desired record length.
I'm not sure if this will solve your dilemma, but it may lay some ground
rules for future designs and considerations.
```
