I do not know if this is related or not, but I have found two
bugs (the first is a fault rather than a bug) that have caused
me untold problems. The first is that Fujitsu *never* clean up
after themselves.  If you look in the TMP area that they
set in the autoexec, you will find a couple files for every compile
you have ever done.  I recently erased 450 megabytes out of
there.  Ignore it, and everything will eventually crash, as they
default to drive C.

The other is a bit more insidious.  First, they always initialize
data to nulls.  Now that should not really matter if the Cobol is
well written, and the standard DOES say that data need not
be initialized. (Though why nulls? Why not space?).  However,
internally, the code follows C conventions ... IE-a string ends
in a null.  So data moves of PICTURE X(50), if there is a null
in position 5, really only moves 4 bytes of data.  This does not
seem to happen when moving Cobol data to Cobol data, but it
*does* happen when moving data internally.  For example,
data between a Cobol program and an object, even if the
object is another Cobol program.  So be very careful with
any data that is used internally(like file id's, and printer specification
strings).
