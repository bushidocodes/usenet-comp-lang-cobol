```
Yesterday I was looking at some COBOL code (not mine) which did not 
transform correctly.

Transformation is a process, part of refactoring and layering legacy COBOL, 
that we do where the COBOL codebase gets scanned and all references to 
indexed file IO are replaced with invokes of objects in a Data Access Layer 
(DAL). (The effect is that the application program now runs against a 
Relational Database instead of against Indexed files, but the processing is 
exactly the same as always.) We use a tool that is part of the PRIMA Toolset 
to do this: "Transformation". (The tool's actual name is MOSTAMD - 
Maintenance Object Server Template Amendments)

It is usually pretty straightforward; the STARTs, READs, WRITEs, DELETES and 
REWRITEs are replaced by equivalent method invokes in the appropriate object 
and the OPENs and CLOSES (of Indexed files) are simply commented out. 
Another phase of MOSTAMD does all the "Housekeeping", adding thread 
management, object references, DAL object program IDs, interface buffers in 
WORKING-STORAGE and all the stuff you need to empower procedural COBOL into 
becoming capable of using objects. We call the new code "Werewolf" code to 
identify it from the Legacy, and because it has undergone a transformation 
which makes it much more powerful... :-)

All of this is done completely automatically by the Toolset and requires no 
programmer involvement, although it is good to have a legacy programmer 
running it, in case there are system implications or some of the Transformed 
code does not compile. (This is a VERY rare event; in a recent case where a 
client Transformed  more than 800 Legacy PR1ME COBOL programs, there were 6 
that required intervention to get them working.)

Anyway, the results from the Werewolf code did NOT match the results from 
the Legacy and this caused me some concern, partly because it is an unusual 
event and partly because it might indicate a problem with MOSTAMD.

When I checked the offending Legacy program I found that all the READs were 
unscoped.

In other words, there were NO INVALID KEY clauses, NO END-READ, clauses and 
NO full stops to indicate the end of the read statement.

Something like this:

READ INFILE
ADD 1 TO IN-COUNT
IF INFILE-STATUS NOT = "00"
     ... go and do something about closing down and cleaning up.

I realised that the author had chosen to use FILE-STATUS and so didn't need 
INVALID KEY or AT END.

Perfectly valid COBOL.

MOSTAMD had been unable to detect the end of the READ scope and had 
therefore commented out some statements that it should not have. It was kind 
of guessing where the READ scope ended and it got it wrong.

I had two options:

1. Enhance the reserved word checking in MOSTAMD so it automatically 
detected scope ends for various verbs. (Not attractive. MOSTAMD does some 
complex processing and it is not as straightforward to do this as you might 
think. Other verbs, including IO verbs, could be nested in the INVALID 
KEY/AT END branches; MOSTAMD has to recursively process until it finds the 
end of the scope. INVALID KEY and AT END are already detected if present, 
but if they are missing it really needs a scope delimiter.)

2. Change the legacy source to have either a full stop or a scope delimiter 
after each IO statement. (Not attractive. One of the guiding principles of 
Transformation is to make MINIMUM changes to existing code and certainly not 
changes that require a Human to get involved...)

For this exercise we'll change the Legacy source with a Global edit (using 
some Regular Expressions) to put a scope delimiter on the IO verbs (option 2 
above, as the lesser of two evils...), but the whole business made me wonder 
if there are many COBOL guys who prefer using FILE-STATUS...

READ INFILE END-READ
ADD 1 TO IN-COUNT
IF INFILE-STATUS NOT = "00"

I would have thought that since NOT AT END/INVALID KEY has been available it 
kind of covers all bases and the only reason you might check a FILE-STAUS 
would be as part of determining what went wrong and reporting details in an 
abend routine.

I note in passing that Transformation analyses the SQLCODE returned by the 
DAL object and translates it to a FILE-STATUS so that legacy which uses 
FILE-STATUS continues to run properly. It also returns "0x" statuses which 
are just warnings with data transfer occurring successfully.

It also seems to me that whether you use FILE-STATUS or not, scoping verbs 
is a good idea.

Any thoughts or comments appreciated.

Pete.
-- 
"I used to write COBOL...now I can do anything."
```

```
Thanks for the note, Doc.

I guess my real concern is that some COBOL verbs don't actually REQUIRE a 
scope terminator; it is up to the programmer.

Obviously where IF is involved there is incentive to use scope terminators, 
but the same is true of implicit conditions like INVALID KEY etc. If these 
conditions are not coded there is less incentive to properly terminate the 
scope of certain verbs. Hence a "danger" in replacing them with File Status 
checking and the topic of this thread.

I wouldn't go so far as to say I think using File Status is not good 
practice (although I never use it myself), but I think that if someone 
chooses to use it they should still be aware that READ verbs have a scope. 
In fact, I would urge modern programmers to ALWAYS use scope terminators 
(and I do this myself).

Other opinions will probably vary and I'm interested to hear some of them.
```

```
READ verbs (and ADD, etc.) only have a scope when they have a
conditional quality.   This idiocy started with the 1985 standard
where they made the IMNHO decision that the conditional verb was the
READ or ADD rather than the SIZE ERROR or END clause.  

In the following which ADD is terminated?

ADD A to B
  ON SIZE ERROR
    ADD 1 TO B-OVERFLOW-COUNTER
   END-ADD

It is the first ADD that is terminated since the second has no
conditions.

If exclusively using STATUS-KEY fields, terminators are not relevant
to the I-O verbs related to files with STATUS-KEY.
```

```
 Accept  ABC

is invalid in '85 Standard COBOL, but is VALID for those compilers with a
```

```
You should note that after a WRITE or REWRITE the file buffer becomes
undefined.

    REWRITE xrec
    IF ( file-status = "23" )
        WRITE xrec
    END-IF

_May_ cause a corruption (unless FROM is used) because xrec may no
longer contain the record that you tried to REWRITE.

I learnt this the hard way on ICL 1900 40 years ago. The file system
for ISAM files had overflow areas. It held the records in a nominal
sequence withing the data 'buckets'. If a bucket already had records
'H' and 'L' and one write record 'J' it had to go physically in that
bucket between the two (by shifting 'L'). If there was not room then
it would put a tag between the two and the record into first level
overflow. The tag pointed to where the record really was.

Now we move to the tricky part. It write the record to 1st level goes
back to the bucket to insert the tag, but there is not enough room for
the tag. It can move an existing record to 1st level and then can
write two tags. Where does it keep the displaced record while moving:
- in the program's record buffer.

If it then finds there is not enough room in 1st level it can use 2nd
level overflow. If there is no room there the WRITE fails with a file
status.

Actually the record being written _is_ in the file and the one now in
the record buffer is a completely different record that may have been
in the file for years.

It is unlikely that this would ever be needed by a modern system, but
the record area may be used for data compression, or other purposes
prior to a physical WRITE or REWRITE.

In my systems it is usual to do a READ WITH LOCK so it is always known
whether a WRITE or REWRITE is required.
```

```
There is one other thing that I don't know if your source handler knows 
about (yet).

In the '85 Standard the "positive" conditional phrases had to be first, i.e.
    Read ...
        at end imperative-1
       not at end imperative-2
    end-read

With the '02 Standard, the order became optional, i.e. the NOT at end could 
come first.  I know that Micro Focus and Elastic COBOL and OpenCOBOL  have 
implemented this.  I know that IBM has NOT implemented this yet.  I don't 
know about Fujitsu.

I don't know if this would impact your parser/transformer or not, but it 
might.  I still see MOST code with the "positive" check first - even with 
compilers that can accept the alternative.
```
