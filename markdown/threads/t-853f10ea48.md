```
This is a follow-up posting on the TRUNC(BIN), TRUNC(OPT) issue. The reason
this is being cross-posted is that the issues involved affect both groups.
Systems programming groups (sys-admins in UNIX terms, I think) are concerned
with performance and the setting of the various compiler option defaults.
COBOL programmers would normally want to pick options that will cause
efficient code to be generated if they follow good practice. The specific
issue raised here becomes even more interesting on platforms that do not have
a native decimal arithmetic. Most fields used in arithmetically in an
application other than subscripts are probably carried in packed decimal on
IBM mainframes because the arithmetic operations match people's expectations
for the type of application. For the most part, the users of COBOL programs
would find the term accurate to the fourth decimal place more meaningful than
accurate to 9 significant digits. The results of division also differ in
binary and decimal. The major use for binary fields in this environment is
for subscripts so the normal truncation rules are not needed. If a field for
a subscript overflowed or was set to a value larger than indicated by the
picture, it is probably indicative of a worse problem. In the mainframe
environment most other uses of binary fields are such that the full word (or
half word) is desired and the COBOL description is viewed as describing that
word. It may well be that COBOL needs syntax added so that the programmer can
distinguish between fields which should follow standard COBOL truncation and
those for which it is a waste of resource. However the discussion here is on
the effect of compiler options. On mainframes, most of the systems
programmers have told people to carry subscripts in binary. However if the
COBOL compiler option TRUNC(BIN) is chosen this is counter productive because
the subscript is converted to decimal for all arithmetic and back to binary.
The compiler truncation option doesn't affect the handling of packed decimal
fields so I have been trying to determine the circumstances under which
TRUNC(BIN) would be needed in a properly coded program.

The latest attempt was using MOVE statements between binary fields of varying
sizes - S9(5) COMP, S9(8) COMP and S9(9) COMP. All of these fields take a
full 32 bit word. The results of the experiment showed that the compiler
generated code to move the full word (normally a MVC of 4 bytes) regardless of
the description. Thus I am left with the question of what conditions require
TRUNC(BIN). If these conditions are CICS related, it could be dangerous to
change the default compiler option to TRUNC(OPT), especially with application
vendor supplied code. If the conditions occur only if poor coding practices
are followed, then it may be possible to scan for and eliminate the problem
code. The COBOL programmer's guide is ambiguous about CICS requirements in
this regard and some of the COBOL code involved comes from a CICS pre-
processor developed by a different group within IBM. Any help would be
appreciated. If there are IBM developers following this group, input would be
appreciated.
```
