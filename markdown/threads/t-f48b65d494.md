Hello -

Any thoughts, ideas or comments will be appreciated.

I am in process of migrating a legacy application written in
COBOL/CICS/VSAM to Client/Server using PowerBuilder and Oracle. The batch
system will be migrated to MicroFocus COBOL. There are about 300 screens,
100 VSAM files accessed by the On-line system, 200 on-line transactions,
300 batch modules. I would appreciate feedback from any one who has
performed migrations of this size and what type of methodologies,
techniques, tools were utilized and the time it took to complete the
migration project.

I have done jobs which are comparable but maybe not as big. You must know
that to adapt and clean up the legacy data (my specialty) so that the new
system can accept it is a major task which will continue far longer than
you anticipate. You need to convert the data repeatedly, on many trial
runs, to ensure you're not losing any, that you've got it right, and that
the new system works with real-world data! You will discover many special
cases in the conversion data, and many old invalid records that have been
lurking in the old database, and probably some requirements errors in the
new design. Get real data into the new system as soon as possible because
test data generally represents wishful thinking.

The data-conversion flow should be highly automated so that it will be
repeatable on each attempt (with only improvements, not regressions), so
on "the big day" when you are to move to the new application, you will
have near-perfect data overnight (or in the minimum time required) without
interrupting business. As just one member of the data-conversion team,
I've acted as 'integrator' of this process, and my conclusion is that you
need someone doing just that job for the duration of the project. This is
a 'personal-ownership' type of assignment, because the process changes too
constantly to allow good documentation and cross training.

You should have a very fast and reliable link for file transfers between
the legacy system and the new environment, especially if there is any
physical distance.

If you're moving to a powerful Unix host, I recommend that you move the
data from the legacy system in as simple a manner as possible (in my
latest project we have used 4GL to extract some database information; but
most non-binary legacy files are just being moved to Unix as text files).
Unix is a good environment for the "ad hoc" procedures which will
transform the data. We have used Micro Focus COBOL under Unix for most of
the data conversion, with a lot of Korn shell scripts and a few Perl
programs.

You will need to debug a few hundred thousand lines of programs and
scripts which will be months in development, and will be thrown away after
the old system is abandoned. But doing this in the "new" environment will
add valuable experience with that environment, and may let you use parts
of the new system in the conversion process. It will especially give you
quick turnaround when you find that you had the correct legacy data but
need to change the conversion process -- you don't need to go back to the
other machine and repeat the transfer of data.

You need LOTS of disk space, to keep several copies of the conversion data
in process -- one copy downloaded from the legacy system; one copy loaded
into the conversion-development database; another copy of the database for
system testing; and some duplicated files so individual conversion
processes can be debugged without disturbing the global process.

Conversion processes must run quickly, because the business probably can't
stop for several days to permit conversion to take place. You should
probably use indexed files rather than relational databases. RDBs can be
very slow when processing EVERY relation in the system -- this is not like
normal production processing.

Plan on a few periods of true parallel testing. It's expensive, but it's
the only way to know that your new system meets requirements, and that the
users will accept what they're getting (people don't read specifications
very well in advance).

Victor's comments make really great sense for the conversion.

May I just make a point on a side issue.

If you have developers who like to create little utilities to do things,
make sure that someone who knows lots of Unix (if that is your new
platform) utilities reviews the requirements first. Don't let anyone
with limited knowledge of the new platform add utilities to the system
without first checking with someone who knows what the platform already
provides.

I had a case just today when someone told me he had ported from
Concurrent to AIX a utility he wrote to compare two files side by side.
He was surprised when I told him of sdiff which does just that and
more. He could have put his time to much better use.


Joan.
