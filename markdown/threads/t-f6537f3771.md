```
When you make a CICS application cursor sensitive your are "hard coding" the screen position in your program. Hard coding may be a little harsh. You could make your map source library available to your on-line program so that you could find out where a specific field is located on the screen, but this would be a "sorta kinda" complex programming task. I think that the amount of work involved to automate the BMS map change / program change task when it comes to cursor sensitive applications will never pay off using the current CICS architecture.

When developing cursor sensitive CICS BMS screens I always pray that the next developer / maintainer to touch the code will perform the required scans to determine what ripple effects his screen change will have on cursor sensitive applications. Since there may or may not be a God, I always code comments in the BMS screen source making note that THIS SCREEN IS CURSOR SENSITIVE.
```

```
I think I know of a solution to your cursor selection question.
There is an option that can be set in the DFHMSD macro of the map
(this macro is at the top of the map definition). This option is
CURSLOC=YES. This option will allow the 'A' map attribute to
indicate the cursor is in the map field. In order to check the 'A'
attribute you need to move the 'A' attribute to DFHBMFLG. DFHBMFLG is
a field in the DFHBMSCA copybook. There are two 88 level variables
under DFHBMFLG. They are DFHCURSR and DFHERASE. You can check the
DFHCURSR flag to see if the cursor is in the particular map field.
Example:
MOVE MFIELDA TO DFHBMFLG (MFIELDA is an example map field)
IF (DFHCURSR)
THEN
SET WS-CURSOR-ON-MAP-FIELD TO TRUE
END-IF

You will need to change any references to DFHBMEOF to DFHERASE. This
is because the cursor indicator and the Erase EOF indicator are
sharing the "A" attribute.

Also, MAPFAIL does not occur when receiving the map and nothing has
been typed on the screen. This may require some minor changes in
processing.

I hope this helps.
```
