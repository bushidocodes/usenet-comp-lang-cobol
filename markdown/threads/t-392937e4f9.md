It is very bad practice, in my opinion, to attempt to "de-edit" an edited
field by manipulating it. I don't do it and most of the versions of COBOL I
grew up with don't allow it. (Of course, nowadays the Language has become
bastardized in response to users who never really understood how to use it,
and an over-eagerness on the part of the Standards Authority to appease
these users... we are still seeing this, but that's another story...).

Once you move something to an edited picture, that's it... Leave it alone!
These edits are intended for output, not for manipulation.

OK, let me start by saying that this is purely a personal opinion and I
don't think that people who disagree are "wrong". I can only tell you why
<I> don't move edited fields to numeric fields ("de-edit" a field).

I see a strong difference between numeric fields and fields that contain
anything OTHER than a number. This goes back to my background in Assembler
on various platforms (particularly with IBM mainframes which, right from the
time Gene Amdahl designed the famous System 360, included checking
"circuitry" (firmware, actually)  to detect invalid numbers.)

These systems were what you might call (certainly by today's standards)
"unforgiving". If the "data check" failed, the program was wiped, with a
dump. Programmers came to dread the famous "S0C7" which usually meant you
had "packed a blank"...(moved blanks or some other non-numeric character(s)
to a field which was supposed to contain a packed decimal operand.)

To get numeric data displayed with decimal points and signs in it we used
MASK instructions and always ensured that this was the FINAL thing that was
done on that field. So the association with "editing" and "OUTPUT" began to
get established in our (well, certainly in mine, anyway <G>) brains. Very
often, when this simple principle wasn't followed, the effect was disastrous
and sometimes it could require tedious poring over hexadecimal output to
locate and correct what had happened.

With the advent of COBOL (after we got over our initial resistance to this
"system resource devourer"...<G>) we came to realise that typing and
automatic "casting" was a feature of the language and all the editing was
done by masks in the PICTURE. However, COBOL made no distinction between
"numeric edited" and "alphanumeric edited" fields. (So you couldn't just
assume that an edited field must be numeric...). We very soon learned the
rules and found that if you moved numeric fields to numeric edited fields
everything worked fine. If you got cheeky and redefined an edited field with
the idea of using it again, all Hell often broke loose... The "rule" that
editing should be the final operation on a number was thus reinforced.

This brings us to the present, where as you correctly point out, systems are
required to deal with edited input from spreadsheets, databases, and
reports.

I handle this by NOT thinking of these fields as "numeric edited" or even as
numbers, but as STRINGS. For me, if it contains ANYTHING other than a
number, then it is a STRING. The language has been extended to permit the
moving of edited numeric data to numeric fields and decimal alignment is
handled for the COBOL programmer (the old "dot goes on the V", trick). We
are currently examining intrinsic functions like NUMVAL and TEST-NUMVAL.

Not for me.

I do a fair bit of development for Windows in PowerCOBOL. Text fields on
screens can contain numbers or strings. But PowerCOBOL ALWAYS treats them as
a string. (This is a different mindset to the 3270 days where you could set
numeric attributes on the screen and be assured that only a number could be
input... I'm not sure whether PowerCOBOL supports numeric only input or not,
to be honest, but I have become accustomed to dealing with ALL input as
STRING.)

So, I spent a few days and decided to put this "problem" to bed forever. The
result is the component I described elsewhere, "String2Num".

If I have to use numeric input which may be edited, I just give it to this
component and it gives me everything I could possibly want, in every
possible format, or it tells me the input is not valid numeric.

It is exactly the same principle as having a callable subroutine which looks
at a string and does the biz...

As a result, I won't need NUMVAL or TEST-NUMVAL because both these functions
are combined in the component.

Now, if you are required to process data from reports (how sad is THAT?!
Sounds like there is room for some design improvements on your site...<G>),
I can see that using a de-editing move can be of some benefit. I still
believe it is error-prone, but needs must when the Devil drives...<G>.

BOTTOM LINE:

1. If you use the capability to "de-edit" fields which is now available in
COBOL, EXPECT to have problems. If you don't, then it will be a pleasant
bonus, won't it? <G>

2. Use NUMVAL to check input strings (if your compiler supports intrinsic
functions) or else do your own as I did.
(The fact that I wrote "String2Num" is not an implied criticism of NUMVAL. I
wanted a component that could run in different environments (not just COBOL)
and I wanted to extend the functionality.
In theory, (I haven't done it yet) I can embed this component on a web page,
for example, and check input to a HTML form, without using CGI script. (The
Applet code can use the component's Methods and Properties).

In the final analysis we all use what works for us. I don't see a "DO" or
