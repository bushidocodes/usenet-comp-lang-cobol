What this means is that they are dropping "support" (service) for IBM COBOL
for OS/390 & VM V2R2 - and REPLACING it with "IBM Enterprise COBOL for
OS/390 & z/OS" (for the "MVS" world - the VM and VSE sides of the mainframe
have *other* issues/releases).

What is particularly interesting is that IBM *seems* to be "pushing forward"
on trying to get all their COBOL customers on "current" compilers.  Some of
the differences between the "newest" compiler (available since last year)
and the one for which they will drop support at the end of 2004 (hardly
"instantaneous") are:

1) "old" compiler used SOM for their OO support; new is Java based and
*only* works in their Unix "mainframe" environment (Unix System Services)

2) "old" compiler still supported much of the '74 Standard (where it
differed from the '85) via the CMPR2 option; new compiler no longer supports
this.

3) *BOTH* compilers require "LE" (Language Environment) for the run-time
(so similar to .NET in "design" that you wouldn't believe that M$ wasn't
following IBM).  However, the new compiler requires a "pretty current"
version of LE (V2R10 - or higher).

  ***

NOTE WELL:
  There are *almost* NO "syntax" enhancements in Enterprise COBOL to bring
in "new features" from the draft COBOL Standard. HOWEVER, its OO support is
a "little" closer - but as it REQUIRES Java and USS, I don't see it as
"particularly" strategic for IBM mainframe OO use.
```

I do want to expand (a bit) on my original note.

A) The actual *implementation* of .NET and LE is *totally* different (both
from the "internals" and the programming interface).  I fully
accept/understand this.  My point is/was that the "intent/design" is VERY
similar.

B) As I did indicate in my original note, I understand that LE uses "APIs"
for accessing OS services while .NET uses OO facilities.  In my PERSONAL
opinion, that is not necessarily a "good" thing as it makes it difficult to
take existing programs and "move them" into the new environment.   It
certainly means that existing ANSI/ISO ('85) Standard COBOL programs canNOT
take advantage of this environment (nor can "new" .NET programs be
considered portable to OTHER existing ANSI/ISO conforming compilers - as
opposed to LE where this IS possible).  It also (as I understand it) means
that even C programs cannot be "enabled" for .NET; I am not even certain
whether C++ (as opposed to C#) programs can use the new environment.  I do
*not* consider it "desirable" to FORCE application programs into proprietary
*and* non-portable source code to take advantage of "enhanced" environments.
In actual fact, this WAS one of the problems with the original LE, that to
TAKE ADVANTAGE of the new features, you needed to "upgrade" your compiler.
(Although at least with LE the "new" source code was still portable to
non-IBM compilers).

C) I do NOT think that forcing programs into an OO paradigm is a good
thing - any more than I think that FORCING programs into a GUI interface is
a "good thing".  Often it is an EXCELLENT choice for new applications (or
even upgraded old ones).  However, that is something that the individual
"shop" (programmer? analyst?) should decide *not* the OS vendor.

D) "strong typing" is a "nice thing" in MANY cases, but definitely NOT in
all cases.  It is one of the things that is GOOD about COBOL (IMHO) is that
it allows the PROGRAMMER to decide this type of thing.  I *do* think that
the draft COBOL Standard is a GREAT improvement in that it SIGNIFICANTLY
enhances the ability for portable COBOL source code to use "strong typing".
Furthermore, it provide EXCELLENT run-time (as well as compile/bind time)
support for "detecting" type violations.   However, no (future conforming)
COBOL compiler will be able to FORCE strong typing.  If .net does this, then
(again) it will be forcing non-ANSI/ISO conforming source code.

E) I fully understand/accept that what was "good" and "expected" for a
"common run-time" on an IBM mainframe in the early '90s is *not* the same
thing as what is "desirable" in a Microsoft environment in 2002.  I don't
expect .net to provide "built-in" support for CICS, IMS, or even VSAM -
anymore than I expect LE to "expect" that all users use GUI interfaces and
have OO classes available to them.

   ***

Bottom-Line:
  .NET and LE applications/programs will look TOTALLY different.  Certainly
the capabilities of the new "environments" are (and probably will always be)
quite different.  Even the "paradigms" and "development environments" look
different.
  HOWEVER, I still maintain that what the two are "trying to accomplish" FOR
THEIR OWN operating systems is BASICALLY the same thing.
```
