YOU MIGHT BE IN A BAD COBOL PROGRAM IF...

* WORKING STORAGE is peppered with literals and variables for the sole purpose
of making dump reading easier.

* Production programs contain more commented out DISPLAY statements (solely for
debugging purposes) than there are lines of executable code.

* The comment blocks still reflect another program that was used as a starting
point.

* More than one person's name appears in the AUTHOR field ("bonus" points if it
includes Arthur Andersen, KPMG, etc.).

* An ELSE statement DISPLAYs something along the lines of "IMPOSSIBLE TO GET
HERE" or "NOW WE ARE IN TROUBLE".

* More than ten different names appear in a change log.

* There are more than 3,000 lines of code.

* The JCL to run it is more than 250 lines.

* IF statements are indented more than 20 columns.

* Linear searches are used on tables with thousands of entries.

* One logical table is split into many physical tables due to compiler
constraints ("bonus" points if the program only references the first tables and
lets subscripts exceed bounds to index into the subsequent tables)

* There are more than 50 CALL statements.

* There are more than 100 NEXT SENTENCEs.

* There are more than 2 GETMAINs (CICS).

* There are more than 3 ENTRY statements.

* The LINKAGE SECTION contains more than 500 lines.

* WORKING STORAGE VALUEs contains undisplayable characters.

* INDEXes for a table with three dimension are something like RUB, DUB, BUB.

* GO TO paragraphs are somthing like HELL, THE-STORE, etc.

* The JCL has comments for acceptable ABENDs.

* DATE-WRITTEN is from the 1970's or earlier.

* ALTER, ON FIRST, or GO TO DEPENDING ON are used liberally.

Please reply to the Newsgroup with your own personal "favorites" that make your
hair stand on end when you have to perform maintenance!

Continuing...

The program has been 'structured' to include performed paragraphs which are
a single line, which performs another paragraph, which contains a single
line, which...

The comments include production abend recovery procedures which instruct
you to insert code to bypass 'bad transactions' by key, recompile, and
restart. The severity of this clue is directly related to the ratio of the
referenced 'IF...ELSE IF...ELSE IF...' clause to the total length of the
Procedure Division.

The program references copybooks with over 20,000 lines.

All the variable names are of the form A0, A1, A2, ... Z7, Z8, Z9. Bonus
points for paragraph names. Extra bonus points if I0 through N9 are
reserved for integer variables.

The program error recovery procedure is 'DIVIDE +1 BY +0 GIVING TALLY.'
Bonus points on hardware on which a divide by zero is not an error.

All the lines in the Procedure Division have a 'D' in column seven.

The report output file is named SYS0007.

A batch program includes the statement 'CALL USING xxxx', where xxxx is a
paragraph or a section name.

The worst single program I've ever maintained, lessee.... the compiled
source listing was over 125,000 lines, not including error messages and
cross reference listing.

It was an entire health insurance batch claims system written as ONE
program.

Up to a dozen programmers, analysts, and team leaders met weekly to hash
out the next release, which was, oh, about two per month. Teams were
client specific, and the program grew by cloning a section of code then
customizing it for a client.

The really interesting part was when the literal pool got filled up, or
the maximum number of 88 levels was reached, or the limit on working
storage capacity, or number of paragraphs. ALL these things happened
with this program on a regular basis. Their solution was to carve out
some paragraphs from the main program and build a subprogram to be
called from the main. These subprograms had to carry most of the same
working storage as the main program, so they were pretty hefty in their
own right, up to 60,000 lines.

There were 6 or 8 unit test versions floating around at any one time.
It had its
own jobclass for compiles because that took over an hour of clock time
and the normal compile classes would SYS322.

I don't know James Walker. I don't think this particular bit of
stupidity occurred here. Of course there are lots of companies that
work with credit cards.

When we began looking at Y2K there were two assembler date routines in
use, and one COBOL date routine I wrote for a specific application.
Both the assembler date routines had pretty bad leap year bugs, but one
of them only had the leap year bugs for years less than or equal to 1968
(and nobody noticed them in production).

One of the assembler date routines had failed in a pretty nasty way in
1992. I was on the beeper and got stuck with the problem. The object
module was an alias and I could not locate the source code for a couple
of hours. The program was being called to add 1 to a date and generate
a new date in MMDDYY format. The input date was converted to YYDDD and
then into days since 01/01/1900 (obviously this routine would not work
too well after 12/31/99). The absolute day count was incremented and
then converted back to YYDDD and then to MMDDYY. But the conversion
from absolute days to YYDDD didn't work in leap years and got one day
off. The calling program was adding 1 to a Sunday date and getting a
Tuesday back. Since it needed to run on a Monday, it terminated
abnormally.

The COBOL date routine handles dates from 1600 to 3999 AD and has been
tested with all dates in that range.
