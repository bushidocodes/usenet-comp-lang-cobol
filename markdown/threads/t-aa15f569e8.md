What it is warning about is that the function requires either a numeric
string (literal) or an alphanumeric data item. As a group item has no
picture, the compiler cannot be absolutely sure how the user intends to use
it. However, COBOL convention since time immemorial has ordained that group
items are treated as alphanumeric (except for Thane...<g>), so the compiler
writer has taken it as such, and allowed it to be passed to the function.
That's why it is a WARNING. Just in case he made an assumption that was not
what was intended.

(Having worked on some COBOL compilers in my younger days, I can assure you
that it is quite a task interpreting what users really mean when they write
COBOL. I am also of the school which says: When in doubt, communicate! A
Warning message ensures that the user's attention is drawn to the area in
question...(It certainly got your attention...and mine.)

There are programmers who don't feel comfortable about seeing ANY
diagnostics come out of a compilation. If you are in this category and the
message bothers you, define the field with a picture (to suppress the
diagnostic) then redefine it for the breakdown. (Yes, I know it's inelegant,
but I'M not the one who's bothered by Warnings...).

My 'guess', is the compiler is warning about the fact that you are using a
group and the problem, the whole group, although alphanumeric (even although
it contains individual numeric items), as a whole may not contain a numeric
value which makes sence.

For example:

01 ALPHANUM-GROUP.
   03 MY-AGE PIC 99 VALUE 21.
   03 MY-IQ  PIC 99 VALUE 21.
   03 MY-DOOR-NUM PIC 99 VALUE 21.

COMPUTE NUMFIELD = FUNCTION NUMVAL (ALPHANUM-GROUP)
the value NumField will contain a numeric number which has no meaning,
perhaps.

What does the compiler say about:
01 ALPHANUM-GROUP.
    03 MY-AGE PIC 99 VALUE 21.
COMPUTE NUMFIELD = FUNCTION NUMVAL (ALPHANUM-GROUP)

Given that another example in this thread says the warning goes away when
the group is redefined and used in the COMPUTE statement instead, my guess
the compiler is making a distinction between an alphanumeric item which is a
group and an alphanumeric item which is elementary.

Up till now, I hadn't thought it worth making public - but because this
thread has continued, I thought I would post the message that I sent to
Fujitsu Support related to the warning message when a group item was used as
the argument to an Intrinsic Function requiring an "alphanumeric argument",

"... in researching the V4.0 documentation, I did find a couple of
"problems" in your documentation - that may relate to this problem.

1) In the LRM, Page 749,
  The item called "2" starts,
"When the argument type is alphabetic,"

I think that this is just a "typo" - as item 1 talks about alphabetic
arguments - and I think that this item is SUPPOSED to handle "alphanumeric"
items.

2) In the same bulleted item, you state (in part),
"the argument must be an alphabetic data item, ..."

This is CLOSE but not exactly the same as what the Intrinsic Function
Amendment says (See page A-28) - which is,
"A data item of *CLASS* alphanumeric"

This is important because there are both "class" and "category" of
alphanumeric.  Given what your LRM states, it is VAGUELY possible that group
items might not be allowed as alphanumeric arguments.  However, with the
correct (ANSI/ISO) wording and if you look at page IV-15 or the "base"
ANS'85 Standard, you will see that it is QUITE explicit that group items are
"data items of *CLASS* alphanumeric" - and those MUST be accepted as an
alphanumeric argument to functions which require such.

  ***

Bottom-Line (unless already done in V4.2)

 - LRM needs both typo and wording fixed
 - "Warning" message should either be removed or reworded"
