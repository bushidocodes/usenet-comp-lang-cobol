This is one of those - it depends on the environment in which you work ing
(often Mainframe vs PC).

- In most (all?) mainframe environemnts, "file structures" (even for
"relative" and "indexed" files) are part of the "operating system" - and
have NOTHING to do with the language "accessing them".  This is true
REGARDLESS of whether the mainframe hardware/OS vendor is or is not the same
vendor as the compiler vendor.  (Compare IBM mainframe where SAS/C uses
*IBM* VSAM files as the "indexed file" system.)

- In most (all?) "workstation" environments, only "line sequential" file
systems are part of the operating system.  This means there MAY be vendors
of file systems (compare C-ISAM and BTRIEVE) and these vendors do NOT create
compilers.  OR a compiler vendor may create their OWN file systems (compare
MERANT vs AcuCOBOL vs RM variable length sequential files - much less
indexed file systems).  There may or may NOT be "external interfaces" that
other languages can use to access these files - that depends TOTALLY on the
compiler vendor's decision.

Well, first off, COBOL is just a programming language.  It does not have a
native file format.  The end of file marking should be consistent with the
system the file was created on.

Your assembler should support an EODADD (end of file/data) routine that will
branch for you at the end of the file.  You will find information on that
within the description of the DCB.

> Generally the language doesn't dictate the physical
format/construction of the file, the operating system does (i.e. the
access method). I'm familliar with IBM mainframes (OS390/MVS, etc.).
In those environments an end of file marker indicates the EOF. This
marker is generated when the file is OPENed for OUTPUT or I-O and
subsequently CLOSEd. Generally, the application pgm does the CLOSE, but
if it abends the OS closes the file. If the OS fails, no EOF marker is
generated.
