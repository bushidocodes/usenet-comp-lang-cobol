Has anyone found a way to get ACCESS 2000 to return a date as a 64 bit
floating point field (the way it is stored on the database)?

In ACCESS 97 you could define a field as DATE and use comp-2 in your COBOL
program to read and write it through ODBC. It recognised the standard VB
date format.

ACCESS 2000 returns a numeric character string that can never be matched to
a comp-2 field. It will WRITE a 64 bit comp-2 field correctly but it won't
read it back in the same format.

I believe there may be an option somewhere in the bowels of ACCESS 2000 that
will change this behaviour.

I can program round it, but it is clumsy and makes COBOL look stupid...<G>.

Any ideas?

Pete.

Thanks Lee.

I already searched and found the functions that will do it, but it is
unwieldy and a pain.

No wonder people don't use DATE functions on ACCESS DBs...<G> (For most
stuff previously I have used CCYYMMDD and it works fine...the problem is
that you can't use such a field for date arithmetic in SQL)

For instance I could code under ACCESS 97:
...
       WHERE    (TCPIPAddress = :HV-TCPIPAddress) AND
                          (Timestampz = :HV-Timestampz)
...
Where "Timestampz" is defined on the database as type DATE and in the Host
Variables as type COMP-2.

Now (under ACCESS 2000) I have to code:

...
          WHERE (TCPIPAddress = :HV-TCPIPAddress) AND
                (DATEVALUE(Timestampz) = DATESERIAL(:getYEAR, :getMONTH,
:getDAY)) AND
                (TIMEVALUE(Timestampz) = TIMESERIAL(:getHOURS, :getMINS,
:getSECS))
...

I'm sure you're right and it is the new ODBC driver that simply doesn't
implement the VB Date format (64 bit float). But it DOES allow an insert
with this field set as comp-2! The problem arises when you need to access
the row and set search criteria (as above). If you have a comp-2 field in
the HV you get an error and no data returned. This is because it  actually
returns an 18 digit string and it won't convert this to floating point. So I
have to set an 18 digit number to receive this field, then break that down
into Year, Month, Day, Hour, Minute, and Second... good fun...

My advice to anyone looking at this would be to NOT design systems with DATE
type fields used as keys...

In fact I am seriously re-considering going back to CHAR (CCYYMMDD) and just
copping the hit on date arithmetic...

Pete.
