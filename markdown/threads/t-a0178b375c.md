```
I'm trying to use Informix with MF Cobol 3.1. To do this I need to
build a runtime, standard practice for mixing MF Cobol with such
tools. There is a 'cob' command which builds a runtime, linking-in
various libraries if required.

My problem is that the 'makerun' command that Informix supplies does
not manage to link-in the Informix library routines. The command that
it runs is:

cob -xe "" -dADIS -dmFFH /somepath/somefile.o -o newrun

where 'somefile.o' is a library of Informix routines.

The 'newrun' that this command builds is no larger than the standard
MF rts. Has anyone seen this problem with the 'cob' command before?
The Informix tech support tried to help, but since it worked on their
system, with the same OS, Informix and MF-Cobols, they had to give up.

My current solution is a C program that explicitly calls every function
that I could dig out of the 'somefile.o' file; I add this to the 'cob'
command, and it then links-in most or all of the 'somefile.o' routines.
```

```
This might be due to a "clever" linker (most linkers will include all of an
object if there is one symbol within it that is required, and will include all
objects explicitly specified on the command line - however some linkers will
only include those functions and data that are actually required (effectively
"splitting up" the object), and will ignore objects on the command line if they
are not pulled in because of a symbol reference).
You don't say what OS you are running on (and it's strange that it can't
be reproduced by Informix on the same OS ....) but I know that the AIX linker
is one of the more "clever" ones.

› My current solution is a C program that explicitly calls every function
› that I could dig out of the 'somefile.o' file; I add this to the 'cob'
› command, and it then links-in most or all of the 'somefile.o' routines.

That will do it. You can find out which routines are in somefile.o using the
'nm' command and looking for the external (or non-static) text symbols.
Alternatively, (to save you having to link in a separate program), try adding
the symbols to the cob command line using the "-I" switch, which forces a
symbol to be included in the executable.
```
