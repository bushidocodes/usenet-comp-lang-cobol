We've just begun using Micro Focus Cobol on Unix. When using Micro
Focus Cobol to write plain old sequential files, you can choose line
sequential or record sequential organization. Line seq is too slow,
we've found. Record seq, though, forces you to add a hex '0a' (LF) at
the end of the record. Without it, Unix commands like cat see the file
as one big record.

Our programmers (mostly ex-mainframers) aren't in the habit of being
responsible for end-of-record characters, and don't like it. I have to
agree; it seems like something the compiler ought to do for us. Does
anyone have any thoughts or workarounds?
```

Line sequential output files strip trailing blanks off the output
records, so if your average line is dramatically shorter than the
record length, you could be doing a lot a blank scanning that would
help explain your slowness.

To massage the output file from sequential to UNIX text format, try
using a process outside of the COBOL system. For example, create
a shell script named foldit:

#!/bin/sh
( cat ; echo ) | fold -w $1 >$2

then invoke it via the DD_ environment variable for the sequential
file; e.g., given a SELECT XYZ-FILE ASSIGN "XYZ" with a record
length of 234:

DD_XYZ='>foldit 234 xyz.text'; export DD_XYZ

Note this approach does not strip trailing blanks from the records.
This version of foldit would do that:

#!/bin/sh
( cat ; echo ) | fold -w $1 | sed -e 's/ *$//' >$2

Note there are two blanks in the sed argument.

Your flavors of these commands may vary requiring some changes.

This will keep your "programmers" from having to deal with their
programming environment.
```
