Does anyone have experience of printing from NetExpress 3.1 over a
WINDOWS network where it is necessary to ensure that all data to be
printed has been printed. We have two situations: to print a test page
to allow the user to line-up (on continuous paper, for pre-printed
multi-part paper) and to ensure that all output has been printed so
that the user can confirm that everything has been printed correctly.

The second ought to be straightforward by closing the printer but I
have a suspicion that this may not occur until the program is exited.
But the first is the real problem because here we obviously don't want
to cause the printer to carry out any initialisations in case these
cause any extra lines to be printed. We are converting the code to use
the PC_PRINTER... routines but this doesn't seem to be the issue.

And of course this is all aggravated by the WINDOWS network; I suspect
that the real problem lies here and this is an area that I've not a
lot of direct experience of. We'd like to keep using the standard
COBOL routines to keep some platform independence so I hope the
solution isn't to use WIN32 API ...

Any comments gratefully received.

graham

Graham,

I can't offer a foolproof solution because I only print to a desktop
printer. However, as with all accesses to COBOL files where I ALWAYS check
file-status, I do exactly the same, check printer-status for ALL PC_PRINT
instructions.

If you wanted to reduce the 'traffic' by restricting the number of printer
status checks,
you could at least check the printer status as you page change. I would be
attempting to check the printer status as being 'clean' before I used
PC-PRINTER_CLOSE.
(I'm musing here - but the printer dialog that you get, I wonder might it
be possible to capture the number of pages being generated - and check
that the same number have been printed - nothing concrete, just a
thought....)

"The second ought to be straightforward by closing the printer but I
have a suspicion that this may not occur until the program is exited ".

Not clear what you mean by that - there's no necessity to wait until Exit
Program.
The printer is closed when you issue the PC_PRINTER_CLOSE. Assuming you
had a second report to the same printer, then Open, print lines and
PC_PRINTER-CLOSE again.

Wisdom suggests avoid the Win32 API routines - Ken Mullins down in Atlanta
is 'big' on Win32 APIs for Dialoging but his preference is PC_PRINT for
reports.

I've got something written in OO for PC_PRINT, which could be adapted to
non-OO - if you would like a copy, email me privately.

I know nothing of the product - but folks enthuse about Crystal Reports
?????
