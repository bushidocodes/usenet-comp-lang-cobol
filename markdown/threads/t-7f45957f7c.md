If you will only ever want to compile in an ANSI 85 environment:

Use packed-decimal as that is what is in the ANSI 85 standard (together
with binary and computational). Before ANSI 85 there was no standard
way of specifying this, but many COBOL vendors (perhaps following the
lead of IBM?) provided computational-3 with comp-3 as a synonym. This
continues to be a supported extension to the standard COBOL language
because it had such widespread use so you would would be perfectly safe
using it with MF COBOL and probably with most other COBOLs. In MF COBOL,
and probably all others that support it, the run time effects of
packed-decimal and comp-3 are the same.

If you may possibly need to compile in a COBOL environment earlier
than ANSI 85 then use comp-3 or computational-3 as packed-decimal
wouldn't compile.

If it was an MF book that suggested computational-3 was ANSI 85 can
you email me the reference please so I can get it fixed. Thanks.

They are equivalent in the IBM range of compilers. COMP-n is an extension to
ANSI, and on most compilers there would be no difference. NCR used to have
a machine which had hardware unsigned packed decimal, which would not be
IBM's COMP-3.

COMP-n is implementation-dependant. PACKED-DECIMAL is packed decimal.
