It's the "01 00 10 00 EC 10" that's goofy. I think you're screwed.

There IS another clue: "RECORDING MODE IS V(arible)." Most (many) compilers
put control information at the start of each block/record when
variable-length records are involved. These extra bytes tell the program the
size of the following block/record and the initial seemingly-bizarre string
may very well be this control information. If so, then the 1st eight bytes
may be the block/record control information and your 13-digit NO1 variable
is 45,558,001,101+. A few more bytes of the record should confirm or
confound this hypothesis.

By the way, it's "Packed," not "Backed."

Hi matthias, comp-3 usually depends un compiler used, in ANSI-85 it means
"packed-decimal storage" and "two digits are stored in each byte. An
additional half byte is allocated for the sign, even if the value is
unsigned.  The sign is placed in the rightmost position, and its value is
0x0D for negative; all other values are treated as positive (...).  The size
of an item (including one for the implied sign) is divided by two to arrive
at its actual size (rounding fractions up)" (from Acucobol manual). One way
is to find a utility (usually included with cobol runtimes) capable to
convert file to ascii text.

*****IMPORTANT:******
Recording mode V is for variable length records.  These records have control
fields that can precede the record or segment of a record.  They require special
processing.  It is also possible that something like the file format could have
been changed since the program was written.

The best way to transfer data from the mainframe to the PC is to copy the
contents of the file to a temporary file in the file transfer area that has no
packed data and then ftp the file to the PC.  There are some mainframe products
like file-aid and DITTO that could print out or let you view the data to see
what is there.  A report writter could print out a few lines to let you see what
the data looks like before you download it.

COBOL isnt that complicated.  To unpack a field you just have to have a variable
like "05  NEW-NBR-1     PIC 9(13)" and then you move the packed data to the new
decimal data field.  It will then take up more space.  Of course, if the data is
bad (not numeric), when you move it the program would crash, unless you tested
it for numeric.  That is just a little harder to do.

It is possible that when the author of the program wrote it, there was more than
one program run to achieve the process.  Sometimes we use 2 jobs to achieve
something and run the first job and then the second job.  It is also possible
that the first record is a Header Record, that is not meant to be used.

Might be nice to know what type of file it came from, how you are downloading
it, what compiler you are using, what type of mainframe, what Operating Sytem on
the mainframe(Fujitsu, Honeywell, IBM, OS2, S390, VM, VSE/ESA, etc.).  If you
had the source code for the program you might be able to hire a consultant to
help you.
