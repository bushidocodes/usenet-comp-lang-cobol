Hello Trill,

I think there are two possible ways of interpreting your requirement

a)  A technical id for each set of original ids with the same value,
where each original id could possibly have as many different technical
ids as it has values if those values also occur in other original ids.

If this is the case, the value is the technical id, but assuming you
only want it for individual ids when that value is duplicated across
several original ids, you must sort the file into the table by value
and work through it adding an indicator field to mark up those
elements where the same value is duplicated.  If you use an output
procedure with your sort, the processing of duplicate values can be
done in that output procedure.  The duplicates indicator then
indicates whether or not the value is relevant as a technical id.

b)  A technical id that indicates whether each original id has any
values that are shared with those of any other original ids, in other
words a duplicates indicator.

If this is the case, the same solution can be used, except that the
duplicates indicator is the technical id.

Good luck, Robert

What you want (on OS/390) is to define your file as a VSAM KSDS file with
alternate record keys (with duplicates).  If you are not familiar with the
IDCAMS control cards to "DEFINE" such a file, contact someone "at your shop"
who can help you with this. You must have one UNIQUE key (as the primary
key) - probably, this would be your "customer number" or something like
this.

As far as the COBOL goes, your SELECT/ASSIGN will look SOMETHING like:

Select CustFile Assign to CUSTDDNAME
  Organization Indexed
  Access Dynamic      *>  this gives you the greatest flexibility - other
options are possible
    Record Key Cust-Num
    Alternate Record Key Cust-Type with Duplicates
    Alternate Record Key Cust-Value with Duplicates
   File Status   Cust-FS  Cust-FS2.

Then in your procedure dividision, you may use ANY of the keys (primary or
alternate) and "read next" thru the items that have the same (or greater)
values.

If any of this isn't clear, you might want to check out:


http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3pg10/1.10.3.3

and for "defining" your file,


http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3pg10/1.10.6.2


P.S.  *ALWAYS* remember to check your file status codes after every I/O
operation - including OPENS, READS, and CLOSES.

If you have access to a relational database, you might consider creating 
tables that mimic this file and dumping data to it.  Then, you can do 
things like

SELECT * FROM Customers
   WHERE Customer_ID IN
     (SELECT Customer_ID FROM Customers
       WHERE ZIP_Code LIKE '374%')

which will give you a "group" of people whose ZIP code begin with 374. 
If you know the value you're looking for, and the field it's in, this is 
a quick way to "group" customers together without actually having to 
assign an ID to a group.  This also allows customers to belong to more 
than one "group", depending on how it's defined.

(STARTing the ISAM file, as was suggested earlier in the thread, is 
definitely the way to go if you can't port the file to a table.  The 
advantage to the table would be when you want to combine conditions (ex. 
all males in the 374 ZIP code area).  You could define alternate keys 
for the ISAM file, but the SQL will process without having to change the 
table definition.  It may not be the fastest, if you're not hitting at 
least one primarily indexed field, but it will drop in very quickly.)
