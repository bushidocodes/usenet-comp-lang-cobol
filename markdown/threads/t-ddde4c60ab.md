COBOL syntax philosophy does not allow a null statement, so you need 'place
marker' statements like NEXT SENTENCE and EXIT, to satisfy that
requirement. For this reason, in ANSI COBOL 85 you can't use NEXT SENTENCE
in an IF statement with an END-IF; the NEXT SENTENCE would then be an
actual branch, not a place marker. Consider this:

IF (...)
MOVE A TO B
ELSE
ADD C TO D.

After generating the code for 'MOVE A TO B', the compiler will generate a
branch to the code following the period, and before COBOL 85 and END-IF,
that was always the same as NEXT SENTENCE would have done here:

IF (...)
NEXT SENTENCE
ELSE
ADD C TO D.

In other words, NEXT SENTENCE does nothing but serve as a 'place marker' to
avoid having a null, or empty, statement. No extra code is generated for
the NEXT SENTENCE statement. But with END-IF, you can have sentences like
this:

IF (...)
IF (...)
MOVE A TO B
ELSE
ADD C TO D
END-IF
MOVE F TO G.

In this case, replacing 'MOVE A TO B' with 'NEXT SENTENCE' would have a
completely different meaning. It would require a branch to the code
following the period, NOT a branch to the code after the END-IF, which is
automatically generated by the compiler. Decidedly NOT a simple place
marker. That's why in ANSI COBOL 85 you can't use NEXT SENTENCE with
END-IF.

And I believe that is the reason why MF COBOL 3.2 didn't give an error,
even though it is a COBOL 85 compiler. I'm not saying the compiler
SHOULDN'T have given an error, for the example is clearly a syntax error;
not because it precedes another statement, but because it should be the
ONLY statement. I'm just trying to give a reasonable guess for why it
didn't. I believe the explanation is simply that the compiler knows that
NEXT SENTENCE doesn't generate code, and then ignores it, failing to verify
that it is not the only statement between IF and ELSE.

Oh the times, I've discovered "compiler bugs" in COBOL....

NEXT SENTENCE. This phrase is used to force execution to the next
sentence structure. IF/ELSE is a sentence structure. In COBOL II,
it may or may not be, depending on how the period is used. If, for
example, you are writing:

IF
NEXT SENTENCE
ELSE

END-IF <--- note lack of period

PERFORM some-paragraph. <--- note period

PERFORM

END-PERFORM. <--- paragraph ends

Guess where your logic picks up? The inline PERFORM, not the
"some-paragraph" reference. Did ANSI get a little "picky" about
what is and is not a "sentence"?

NEXT SENTENCE is not compiled as a NO-OP; it is a branch, or loop-
terminator, which means it is assembled into a BRANCH to some
forward reference.

===================================================
CONTINUE. This is a no-op. You can code it and follow
with additional statements. In the above example, substituting
CONTINUE will yield the expected results.

===================================================

EXIT. This may appear to be a "no-op" as used in an "empty
paragraph", but again, it is not. It is a "return-from-call"
instruction. Use it in place of "NEXT SENTENCE" above and
you'll really see 'unexpected results'.

If you should happen to ever see a COBOL program using
SECTION's, and GOTO's, you will note that the EXIT is a
safeguard against "fall-through" logic (ie: you perform
a GOTO to paragraph label, the paragraph executes, hits
the next paragraph label and continues to execute, execute
until it either hits the bottom of the program, an EXIT, an
EXIT PROGRAM, STOP, STOP RUN, GOBACK,
or a CALL "ILBST???" to force a closure of files
and exit of the program.

Quoting from the IBM COBOL II Reference Manual at
http://ppdbooks.pok.ibm.com:80/cgi-bin/bookmgr/bookmgr.cmd/BOOKS/IGYL1101/3.14:

When control reaches an EXIT paragraph that is not the end of
a range of procedures governed by an active PERFORM or USE
statement, control passes through the EXIT statement to the
first statement of the next paragraph.

That is, an EXIT statement may or may not be a no-op, depending
on how control reaches it. It is *not* an unconditional
return-from-call instruction. In the context of a PERFORM THRU,
the only EXIT which matters is the one which is the target of the
THRU clause. All other EXITs in that context are no-ops, though
they may be effective in other contexts.

That's one reason I don't like relying on EXIT paragraphs. You
can't tell what an EXIT does just by looking at it. You have to
know, or guess, which PERFORMS are active at runtime.

Granted, few people will code logic to fall through an EXIT -- at
least not deliberately -- but I couldn't pass up an opportunity to
be pedantic.
