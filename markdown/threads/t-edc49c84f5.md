I've no doubt Pete Dashwood will tell you this is NOT the way to do it
with Fujitsu, but here's pseudo code as I do it in Net Express.
(Substitute your appropriate GUI instructions).

Firstly the full screen :-

        <------------- record layout ------------>    <-------- Listbox
---------->

        A/C #
.....                                                        Smith, Bill
.....

Name....                                                           Smith,
Gracie

Address                                                            Smith,
Harry

etc.............
Smith, Herman

1. Display empty screen. If you already have a list, display in the
Listbox.
2. User has two choices :-
    2a. Select from Listbox - if user has selected an element from your
Listbox
            use the key (A/C #) to display record contents in the Record
area.
    2b. Other option - user is entering a new record so puts in new A/c #
-
          if blank - just re-display empty Dialog. If existing key,
display record from
         your file - suggest you also 'make sure' it shows in your
Listbox
3. No existing record - prompt user to fill in fields. Then this goes to
Step 4.
4. 'New" or existing - accept changes or even 'Deletes'
5. OK or Cancel :-
    5a. OK         if RecordChanged

                            if Record Found
                                write
                            else rewrite

        Make sure changes/additions/deletions are reflected in your
Listbox

6. Go back to # 1 until user signals EXIT.

I have no idea of your F/J coding - but at least above is a fairly
generalized model.. And for reusability you can repeat it over and over
again.

If possible go for three tier system design - i.e. separate
programs/classes :-

(a)Business Logic - EditCustomers
(b) DBI (Data Base Interface) your file/DB handling, and
(c) UI (User Interface) - your dialog(s)

Couple of notes :

When I say display empty screen I mean 'empty record layout'. You should
're-show' what exists in your listbox.

Listbox design - I prefer to keep it small - just what is pertinent to
guide the user, i.e. the name and A/c #. Selecting from Listbox gets me
the PrimeKey(A/C # ) which I then use to access full record details from
file/DB.

There is an example of that in the "Samples" folder of your Power COBOL
installation.
In order to get 2 modeless dialogs that comunicate with each other, you have
to
1)Develop A.dll (your List) in a separate project as a OO module; make shure
you code some event to notify the other form that a record key was selected.
2)Develop your "Details" form (B.EXE or B.DLL) and include A.DLL in it as an
ActiveX control, unchecking the "Visible" property of the control. At the
"Opened" procedure of B.DLL code an 'invoke ??? "Activate" returning
ReturnValue' and voil� the first module pops in a separate window.
3)The next thing is to code a method in B.DLL to populate the "Details"
form. That method shoul be invoked from the event generated by A.DLL when a
customer is selected.
I'll leave to you the details of figuring out how to detect that the user
select an item in A.DLL. If you use "Editable Label", then the control fires
an event when the user presses "enter" over an item of the ListView.
Feel free to email me if there is any trouble. I have done this a lot a
