Hi Martin,
OLE requires all properties to be accessed via methods.

To get any properties, you must invoke the method on the appropriate 
object, prefixing the method name with a "get" (and "set" to set the 
property value). 
If the property is another Word object, the COBOL type needs to be an 
object reference.
For example:
01 theSelectionObject object reference.
..
invoke wordApp "getSelection" returning theSelectionObject
*> Change the text in the selection
invoke theSelectionObject "setText" using z"New text"
*> We have finished with the selection object
invoke theSelectionObject "finalize" returning theSelectionObject

COBOL can't concatentate method calls as VB can (yet...), but the 
underlying number of method invocations to Word (including what the 
finalize does) is the same. You just have to be a little more explicit.

Other differences are that VB syntax "hides" the default method sometimes, 
so when you see syntax such as Selection.Paragraphs(1).Range.Select, this 
is read as:
01 theParagraphsCollection object reference.
01 theParagraphObject object reference.
01 theRangeObject object reference.
..
invoke theSelectionObject "getParagraphs" returning 
theParagraphsCollection
*> (1) means invoke the default "Item" method with the item required
invoke theParagraphsCollection "Item" using by value 1 returning 
theParagraphObject
*> Get the range object
invoke theParagraphObject "getRange" returning the theRangeObject
*> Select the range
invoke theRangeObject "Select"
*> Finished with the objects
invoke theRangeObject "finalize" returning theRangeObject
invoke theParagraphObject "finalize" returning theParagraphObject
invoke theParagraphsCollection "finalize" returning 
theParagraphsCollection.

Again, COBOL is doing the same as the VB code above, but more explicitly.

Steve.
