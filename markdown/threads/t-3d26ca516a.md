To my knowledge...

Your question cannot be answered because a *.DAT file does not mean
anything specific.
It only means that this is a file containing data but it does not tell you
anything about the structure or format.

These are your options:

1. Contact the vendor/owner of this file and request information about the
internal structure.
If you indeed can contact the vendor/owner then you might as well ask for
available conversion routines (e.g. owners.DAT-to-csv, owners.DAt-to-txt,
owners.DAT-to-whatever...).

Note: In the discussion following it is assumed that the *.DAT file is an
ASCII file!
For argument sake I assume you are working in a MS windows environment,

2. Open the file using the windows TEXT editor (note: The windows TEXT
editor is, and MUST be, a plain ASCII editor).
You can also use any other compatible plain ASCII text editor.
Study the contents and try to 'guess' the internal structure.

Option 2 might not be possible when the file is very large!

3. Alternatively, write a quick and dirty program (simply done by some
(copy) utility, alternatively a REXX, COBOL or other language program can
be used) with a fixed record definition length (e.g. 100 bytes) and read
the first 200 records or so and write these records to an ordinary text
file.
(writing to ordinary text file will also(!) copy the non-displayable
characters).
The study the contents again as described before.

In all cases you must study and know the (field) format(s) and (record)
structure.

Once you know this a conversion program will be relatively easy
(regardless whether you use COBOL or some other super solution).

There are some 'things' that might still be an issue, like:

1. CR/LF  (Carriage Return / Line Feed).
2. binary "null" being used as end-marker.
3. EBCDIC (flavour) versus ASCII (flavour).
4. User (or vendor) defined (or dependent) formats.
5. ...other non-standard definitions (e.g. country codes, time-date
formats, etcetera...)

As far as I know there is no easy way out.
..though the (probably) smartest thing to do is contacting the owner of
the file or the vendor of the product producing this file!


Good luck!


Regards, Wim Ahlers.

Davide,
What COBOL are you using, and on what platform (as other have asked!)?

If this is a MicroFocus COBOL situation on a MS Windows platform, do you 
have what used to be called (and maybe still is) the Workbench?  If so, 
there is a Hex Editor that will allow you to examine the file for the 
presence or absence of CRLF.

I haven't used MicroFocus COBOL for some time, but there are ways to 
read files which do not conform to the PC's way of delimiting records 
(generally, the CRLF pair).

In IBM VisualAge COBOL, I believe standard PC files are denoted as 
ORGANIZATION LINE SEQUENTIAL, while fixed length files without the CRLF 
are processed as SEQUENTIAL.

I don't have experience with other COBOL's on the PC platform, so this 
is the limit of my suggestions.  Hope this helps.
