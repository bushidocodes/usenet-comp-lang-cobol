In the case of INSPECT CONVERTING look at the generated code. If you used
INSPECT FIELD-A CONVERTING FROM "abcdefghijklmnopqrstuvwxyz" TO
"ABCDEFGHIJKLMNOP" and on an IBM 360/390 class system all you got was a
translate instruction, then INSPECT CONVERTING was far better
performance-wise. If you got code that built a translate table it would be
more expensive and if you had used INSPECT FIELD-A CONVERTING FROM
DATA-NAME-1 TO DATA-NAME-2, you might have gotten a real pig. On other
platforms I would have to see the code generation for the alternatives but
the 26 separate compare replaces in a loop (the way the INSPECT REPLACING
would work) doesn't seem to be too efficient and would not be a joy to
maintain. It would be interesting to see what the code generation is for
various PC compilers with maximum optimization turned on. If this construct
were used against a number of your input fields (or other noticeable usage in
the program) this could affect performance. My general rule of thumb is find
simple general rules that give decent performance while leaving a
maintainable program. Then tweak the special programs or high use
sub-programs.
