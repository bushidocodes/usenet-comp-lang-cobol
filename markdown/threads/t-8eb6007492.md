### What's good and what's bad about CoBOL's directions

- **From:** Howard Brazee <howard@brazee.net>

```
I want to start off by saying that changing a language to fit all needs
never was a good idea, and is a worse idea now when we have the ability
to interact with other languages which specialize in different tasks.

One strength of CoBOL is that it is English Language based.   Status
checking should be in English.  Checking status codes is not CoBOL and
should have been replaced decades ago.

There should be a standard way to check to see if function calls worked
correctly though.

Making the Report Writer standard will just make compilers more
expensive.  Instead, CoBOL should have found a better way to interface
with specialized report writing programs such as Crystal Reports.

Calling API's in their native format is ugly and anti-CoBOL.  I'm not
sure what should be done here.  Certainly it should be easier to create
pseudo or true conversational CoBOL using other languages for the I-O.

Bloat by committee will add desired features - but at what cost?
```

In April of 2000, I had the privilege to be able to attend a talk by
Jeff Freedman of Computer Associates about the upcoming COBOL
standard.  He works for CA supporting their CA-Reala COBOL compiler,
and also happens to be a member of the international COBOL standards
committee.

What I learned from this short session is that developing new COBOL
international standards is really an incredible feat.   He covered in
detail some issues that ran into some bizarre roadblocks; i.e. Japan
won't sign the standard unless they get something in that is key to
their culture, or Norway won't sign the standard unless they get a
proposal removed from the standard.

How many other languages in the world even have a truly accepted
"international" standard that has been officially signed off by all of
the countries?

Report Writer sidebar:
==================
I only use COBOL on a mainframe system.  We have a lot of legacy COBOL
report writer software we must support, and I have never heard of a
mainframe product that could do what Crystal Reports does.  I think
this just emphasizes that every standard decision must be considered
for all platform requirements as well as for all international
requirements.

Pete

I'm really surprised no one picked up on this topic. So taking some of the
points you raise Howard :-

1. Yes, we should be communicating with other languages that specialize.
For instance at the current time, one person on J4 has said, 'while not
committed, they are keeping their options open on COBOL GUIs". My response,
"Forget it - we are too bloody late; ten years ago maybe". Besides which if
you've done a switcheroo to SP2 or Java or ......, why would you drop you
existing tools in favour of a COBOL GUI toolset - and in all probability it
would be a balls-up designed by a committee.

2. Checking status codes in 'English' - not sure if this is what you are
after; check out the FCD on Exception Conditions, file status included,
they do have 'English' names; Bill if wrong, correct me.

3. Not sure about your function calls - may also be in that Exception
Condition list ????

4. Report Writer - Bingo ! I've just told five J4 members exactly the same
thing.
I didn't mention Crystal - but that's the obvious choice isn't it.

5. APIs - I guess you are not referring to GUIs as you specifically mention
I/O. On the thread 'COBOL files v DB' - that's the other question I could
have asked - "Given that the COBOL file system was dropped completely, what
would you choose as an alternative ?".

6. Bloat - I think many of us are agreed on this, trouble is, nobody is
S-C-R-E-A-M-I-N-G- Bloody Blue Murder at  J4 !

I can't remember if that's how I started the thread "COBOL files v DB", but
it did start with me musing elsewhere :-

    "If you had a stripped-down COBOL, excluding things like Screen
Section, (sorry Judson <G>), R/W and COBOL file system......" which got the
reaction 'COBOL files are the language's major asset'.

I started out as a COBOL purist, dreaming/hoping the one language could do
it all for me in one comfortable and known coding environment. I agree with
your observations - we have a strong BUSINESS oriented language,  make it
gel with other languages or tools which do a better job with the ancillary
stuff covering UI and DBI. We would hardly consider COBOL as the language
to use for voice recognition - would we.

Meanwhile - we are still going to get the camel from the committee instead
of the horse we wanted.
