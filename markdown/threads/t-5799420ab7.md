By how much is your running total off? My experience has been that
running totals go off because of failures to deal with the first or last
records read (most often the last one); check your WRAP-UP rtn and see.
If there are only 15 records you might want to change the data so that
each one contains an unique number for totalling (101,102,103... 115);
it is easier to find which one is being dropped than if you use the
usual 100, 200, 100 that most folks enter.

As for finding the highest number in the table try comparing each value
to a save-area containing the highest previous value (on the first read
that would be the first value, of course); if the incoming value is
greater then just save and use *that* one for the next compare.

Bona Fortuna!

A couple of months ago I looked into intrinsic functions in COBOL for
MVS and VM using IBM's bookmangler.

Intrinsic functions are classified by the type of argument returned.
Numeric intrinsic functions can only be used in COMPUTE statements. The
CURRENT-DATE and WHEN-COMPILED intrinsic functions are not numeric
functions. They return character strings (actually structured records)
and therefore you can MOVE them.

I don't know if this is contrary to the standard, but that's the way IBM
implemented it. My impression is that IBM uses fewer proprietary
extensions now than they did with COBOL-74.
