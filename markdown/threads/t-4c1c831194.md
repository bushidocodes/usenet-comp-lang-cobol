While I agree that any of those controls *could* be used for
navigational purposes, that doesn't necessarily make them appropriate
candidates for consideration.

I think that the vast number of unique control types in a GUI

user interface design.  I believe that if consideration is given to
the primary purpose of the control, the direction on how the user

GUI doesn't have to mean dancing icons and fancy controls.  Those
things are great if they contribute to the navigation, usability or
marketability of a program, but too often people think that the screen
must dance if it is to be considered "GUI."  This is a false
assumption  One of the primary objectives of good user interface
design is ease of navigation and ease of data entry/inquiry.  Quite
often the designed will "overdesign" the GUI and actually detract from
the usability and productivity of the end user.  There's *nothing*
wrong with a simple GUI screen if it serves the desired purpose well.

having to know C.

...but unless you are directly invoking functions from the operating
system, you ARE going through an add-on module...it's called a
runtime.  The WinApi calls are not being invoked directly by your
source program...they are being interpreted by an intermediate
executable or dynamic link library which, in turn, is invoking the
operating system API.

Native is direct.  Anything else is interpretive.

>Borland Delphi has it's wrappers of the WinAPI, how would you consider that?

Not native.  Native, in my humble opinion, is direct use of the low
level routines as provided by the operating system itself.  Borland's
wrappers are a set of intermediate code which is, in turn, invoking
the API.

is native.  Anything else is interpretive.

Behind much of this "character to GUI" thread is a change in paradigm (more
for the user than programmer) shift that I don't think has been discussed
much.  It was a "biggy" in the days that IBM introduced their (now dead and
little lamented) "CUA" (Common User Access) methodology.  However, I think
it does still impact such conversions.

Mostly (not universally) Character Mode user interfaces are "action-object"
systems.

Mostly (not universally) GUI user interfaces are "object-action" oriented.

By this, I mean that the question is whether you decide what you are going
to do first or what you are going to do it to.  In the old IBM mainframe
ISPF (or Xedit) paradigms, you always used a "menu" to decide what you
wanted to do (Edit, Browse, Add, Change, etc) first - and then you decided
what object (PDS, member, program, etc) you wanted to do it to.

In "modern" GUI systems, it is most likely that you will find the "thing"
that you want to act upon.  Then you will be presented a (customized) list
of what actions are appropriate for this "thing." (If it is a a  program,
you can run it - while if it is a document you may want to edit it.).

When converting a Character interface to a GUI one - doing your "homework"
(as mentioned in another part of this thread) often concentrates on this
"paradigm" shift.  How best to implement it - and how best to *RE*-train
your users.

componants together to create a whole.

window approach that simplified things (but by no means made this 
simple).
