```
About a month ago I threatened to write up a "White Paper" on converting
COBOL data to other formats, and deal with the question of, "What do I do
about EBCDIC-ASCII conversions of COBOL data?"

Well, I started on the paper and some software I am going to throw in (this
will all be freeware), but before I get too deep into it I wanted to get
some opinions.

What I am looking at doing is dividing the paper into four main sections:

I Overview - EBCDIC, ASCII, Cobol and IEEE datatypes; why you cannot just
"download with conversion" .
II Moving COBOL data  from an EBCDIC machine to an ASCII machine, creating
standard IEEE data types.
III Moving COBOL data  from an EBCDIC machine to an ASCII machine, creating
COBOL data types.
IV.Moving COBOL data from an ASCII machine to an ASCII machine, creating
standard IEEE datatypes.

OK, does this look complete enough? Other ideas?

It is my intention to get this into a form such that it can serve as a
reference for all those people who ask, "How do I get data from my mainframe
so I can actually do something with it on my PC/Unix box in
COBOL/C/C++/BASIC/whatever?"

On the software side, someone recently posted some COBOL source using a
substitution table to go EBCDIC to ASCII. Would whomever that was please
contact me? I'd like to include it and want to make sure you get credit and
I don't go to jail for violating any copyrights.

For those questioners not having access to a COBOL compiler on the
destination box or who cannot get the COBOL code written with USAGE DISPLAY
SIGN SEPARATE on the mainframe, I am thinking about writing a DLL to handle
conversions, something like:

DECLARE SUB EBCDIC2ASCII ALIAS "Foo" LIB "Mike'sFreeLib.dll" (BufferIn AS
STRING, BufferOut AS STRING, Parameters as something) EXPORT

(I'll play with the exact parameters as I go along).

I want to create this in the form of a 32-bit DLL until I get the 16-bit DLL
compiler.
(That's right, I currently have a 32-bit DLL compiler but not a 16-bit).
Will this be useful? Is the 16-bit needed? Do I need an MS-DOS version,
too? Comments?

Lastly, for both the text and the software, I am looking for a couple of
volunteers to proof-read, edit and test. Anyone who participates gets a
credit.(No cash, no merchandise; but your Mom and Dad will be proud of you).

Please post suggestions and comments here; and even though it's bad
'netiquette' to ask for e-mail, I'd appreciate volunteers for testing and
editing writing directly to me.

Regards,
Michael Mattias
Tal Systems
Racine WI USA
michael.mattias@gte.net
```

```
As Bill Klein pointed out in another reply to this post, there are many
different EBCDIC code pages in the IBM world. Every non-english language
has its own code page. OK, the letters A-Z and a-z, the numbers 0-9 and
some special signs have the same hex value in all code pages, but more
than a hundred signs are used very differently. Same for the ASCII code
page on the workstation.
So, if you write a ASCII2EBCDIC converter, you have to know both
codepages to translate the character strings correctly. Windows can give
you the current code page on the current PC, but you can only guess what
is the appropriate EBCDIC code page. You could pass the EBCDIC code page
as a parameter to your converter program and/or you could look for the
EBCDIC code page of the same language as the current ASCII code page.

I searched the IBM documentations for a complete set of translation
tables, but I found only Redbook with some samples. But I think that you
could transform most data with these few sample tables.

You can download the Redbook from:
http://www.redbooks.ibm.com/pubs/pdfs/redbooks/sg244846.pdf
It's the document number SG24-4846-00 with the title 'Multi-Language
Solutions for Client/Server Database Environment'. Opend the file with
Acrobat Reader and refer to the Appendix C: Sample code pages.
AFAIK, Redbooks are not restricted, you may use the samples without
breaking the laws!
```
