### DMSII FIND's

- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<3edbc12f.30589625@news.norlight.net>`

```
When coding a DMSII selection statement for an A-SERIES/Clearpath
environment some Cobol developers are using FIND FIRST rather than
FIND when dealing with unique sets (no duplicates allowed). 
Example:

CUSTOMER DATA SET (CUST-ID ALPHA(10); CUST-NAME ALPHA (30););

UNIQUE-KEY SET OF CUSTOMER KEY IS CUST-ID NO DUPLICATES;

DUPLICATE-KEY SET OF CUSTOMER KEY IS CUST-NAME DUPLICATES;

FIND FIRST UNIQUE-KEY AT CUST-ID = <variable Id>
    ON EXCEPTION...

FIND UNIQUE-KEY AT CUST-ID = <variable Id>
    ON EXCEPTION...

A newbee (and most oldbees) could assume the set UNIQUE-KEY contained
multiple key fields or duplicates. Besides being misleading when
reviewing the code, is anyone aware of any performance implications
using FIND FIRST versus a simple FIND?

I've written a test program to get a rough idea of the difference in
processor times, and the FIND FIRST seems to require more processor
time.  However, due to the system mix at the time, these results could
be erroneous.

An OldBee
```
