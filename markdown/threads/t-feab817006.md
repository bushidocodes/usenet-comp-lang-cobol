I just had the same problem. My solution was to put a dummy record into the
file, then run the cobol program. I put a key value of zeroes into the file
and then the cobol II program was able to run successfully. I added code in
the program to delete this dummy record.
I've included the syncsort JCL I used for the dummy records:

//DUMMYREC EXEC PGM=SYNCSORT,COND=(0,NE)
//SYSOUT  DD SYSOUT=*
//SORTIN  DD *
000000  0000000 00
/*
//SORTOUT  DD DSN=THE.VSAM.FILE,
//         AMP=('BUFNI=3','BUFND=4'),
//         DISP=SHR
//SYSIN   DD *
  SORT FIELDS=COPY
/*

If there is a better solution, please let me know.

some new linking requirements) that are not changed in any way and which do
not attempt to interface with any of the new things WILL continue to run "as
they did in the past" - ASSUMING (and it is a big assumption) that they
didn't use any "undocumented" features (that used to work but were never
documented as working).  Therefore, if you really do have existing OS/VS
COBOL programs that you don't want to change and don't want to enhance and
don't want to work with new stuff, you shouldn't have any problems.  (Again,
assuming they were originally coded to correctly handle Y2K issues.)

I didn't see the original question, but if you attempt to open an empty
file as input using COBOL II or later you will get the '35' status.  That
is, unless you select the file as OPTIONAL (ie "SELECT OPTIONAL FILE-NAME
ASSIGN TO DDNAME, etc.)  If you do this it should be OK.  I can't remember
if it returns '00' or '05', but either way it's OK.

> I have never seen a VSAM relative file format used in a production
> environment.  What are you doing with it.

I obviously can't answer for him, but I can give you a useful example for
a relative file.  We purchased an ATM system package.  It's up 24 hours a day,
more or less.  Batch processing must be done while CICS is up.  Card traffic
is written to a relative record file that holds about 10 days worth of 
transactions batch processing starts at where it left off on the previous
day and processes the RRDS file until it reaches a record that is from an
earlier time.  Basically the RRDS is a circular file.  When it needs to write
to the next record after reach the end of the file it simply starts over at
the beginning of the file.

Anyway, it's a pretty clever way of doing it.  All of the maintenance log
files work the same way.

(There's a little more to it than what I've explained, but that's the basics
of it.  We even stole the idea for our inhouse written telephone transfer
processing file.)

Just for the non-IBM people who are reading this, please do not confuse ISAM
with isam!  (rather easy to do).  "ISAM" was a proprietary file system

saying that using indexed file systems are obsolete - only that this
specific file system is.

facitlies.)

The RRDS file format is essentially a bunch of record slots accessible
by record number.  They were included to provide a replacement for the
random access or relative file methods supported in pre-VSAM days. 

I've worked on a few of these. They were intended for high performance
random access applications where sequential processing was not an
issue. Access was strictly by relative record number or by a
hardware-dependent cylinder/head CCHHRR algorythm. Essentially IBM
implemented the file structure and then pretty much ignored it as user
implemented. No utilities, no backup utilities, no record locking for
concurrent access. That all had to be handled by the developers.

One of the places I remember this being used a lot was for transaction
files that could be read and updated concurrently by CICS and batch
programs. One warehousing system I worked on used a transaction file
where everything received and shipped was recorded in it. Record one
on the file was used as a control record that was constantly updated
by each CICS transaction that updated the file with the number of the
last record written. When batch processing would start, we'd read that
record and use that to set our end-of file point  Then we'd look for
records that had not been previously extracted, write them out to
another file and update them as extracted. At the end-of-file record
number from the control record we'd stop reading that file and start
processing the extracted records.  This proved to be completely
bulletproof as the batch and online never processed the same records
at the same time.

When VSAM was developed IBM originally marketed it as a replacement
for all the older access methods. They provided sequentail, indexed
and relative file access methods.. The RRDS file type was made
available as an alternative for the non-hardware-dependent
implementations only. I know of a few people that converted the CCHHR
files to VSAM by using the CCHHRR into a key and rewriting the
applications using indexed files, but this was very uncommon. We never
found it nearly as clean or easy as converting ISAM applications to
VSAM. Concurrent access and record locking was still user
responsibility, and there were never any major performance or
reliability justifications to make it worth while.
