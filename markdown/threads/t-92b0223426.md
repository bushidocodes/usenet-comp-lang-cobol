Some very quick responses:

1. Having a few remote locations (mobile or otherwise) does NOT mean you
require separate databases for each one. I have designed and installed a
database which was (and is) accessed from up to 1500 remote locations
simultaneously. One (very large and complex) database. No problem.

2. You may be confusing the functions of data entry and real time
transaction processing. Data entry doesn't need the full DB on line;
transaction processing does.

If transaction processing is what is required, then remote mobile links to
the database (wireless Internet or WAP protocols are just 2 options) must be
established. If you do this, the RDB software automatically takes care of
conflicting updates.

I believe that most of these remote teams are sensing corrosion and entering
data? That doesn't require either multiple databases or direct transaction
processing.

Specifically dealing with "DBAs" ....

A couple of years ago a wrote an application for a major insurance company
(I don't want to give you the name, but while I was there I was in "good
hands") . The application was to run on nine MVS mainframes with DB/2 as the
data access; all code was to be COBOL/ESQL.

I designed the database on paper, essentially creating a "data dictionary."
I included the proposed names of tables and columns, and datatypes; I added
a list of approximate number of rows to be stored per table, "how" the data
would be accessed for insert, delete, select etc, and a high-level narrative
overview of the application.

I then took the whole schmear to one of the DBAs (they had about six
full-time DBAs to support applications programmers).

After a couple of days, the DBA came back to me. He gave me 'final' table
names and column names (consistent with shop standards), changed a couple of
the datatypes I had used for better efficiency, told me about some indexes
he had created so my cursor accesses would perform better; and mentioned
some stuff about 'tablespaces' about which I had no clue (still don't). He
also said he had no idea "why" we were doing this application, but that was
really not important to him.

This still stands out in my mind as the best example of cooperation between
applications-types and systems-types I've encountered: The DBA did what he
did well, the application guy did what he did well. Wish I could enjoy this
more often.
