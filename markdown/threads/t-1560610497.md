Assuming IBM COBOL II or higher (ANSI85):

SET PTR1 TO ADDRESS OF PARAM1
SET PTR2 TO ADDRESS OF PARAM2
EXEC CICS LINK   
    PROGRAM('PROGRAM2')
    COMMAREA(COMMAREA-FOR-PROGRAM2)
    RESP(MY-EIB-RESPONSE)
END-EXEC

It's probably easier for PARAM1 and PARAM2 to reside in the calling
program's working-storage, but you should be okay with PARAM1 and PARAM2
in the calling program's linkage section, provided you've already
established addressability to them.

In the called (or linked-to) program PARAM1 and PARAM2 must be defined
in your linkage section.  They can have different names from the calling
program, of course.  You would then establish addressability to them as
follows:

SET ADDRESS OF PARAM1 TO PTR1
SET ADDRESS OF PARAM2 TO PTR2

If you don't have COBOL II or higher, then you have to do service
reloads, and I'm forgetting how to do that.  If you need that, then you
must be very careful about the order of level 01 items in your linkage
section, your BLL cell pointers must reside in linkage, and you will
need multiple pointers for any 01 area that exceeds 4K bytes (assuming
you're talking about CICS in a system 370 environment - I wouldn't know
about other platforms).  For example, with OS/VS COBOL you would need
SIX BLL cell pointers for PARAM1, since it's 22k bytes long.  Since your
brief example includes USAGE POINTER I'm assuming COBOL II or higher. 
It's much easier.

Passing around large COMMAREA's can create performance problems, but
perhaps that was more of a problem in older releases of CICS.  It =is=
more efficient to pass addresses.
Lots of ways to do that.  You can stuff them into your TWA and do EXEC
CICS ADDRESS TWA(MY-TWA-POINTER) END-EXEC.  Then establish
addressability to your TWA and pull out all the pointers you need.  You
could certainly pass addresses through temp storage queues, but then all
programs would have to use the same hard-coded queue name as a key.

I've done a lot of CICS programming with conversational tasks and no BMS
maps.  We generally pass dozens of pointers through the TWA.

I hope that helps.

OK, now to the POINTER usage in IBM COBOL: In the calling main program,
you have to define the parameters in the WORKING-STORAGE (comparable to
static or automatic in C language) since LINKAGE definitions don't
define storage (like typedef strukt in C).
Now, the problem is, that you cannot SET a pointer to the ADDRESS OF a
WORKIN-STORAGE field, only to a LINKAGE field. So, you need another
submodule which gets the WS field (in its LINKAGE SECTION) and can SET a
pointer to the ADDRESS OF that field. This little submodule can be a
nested program in the main-module.
Let's look at a sample:

IDENTIFICATION DIVISION.
PROGRAM-ID.  main-prog.
...
WORKING-STORAGE SECTION.
01 WS-PARAM-1 PIC X(22000).
01 PTR1 USAGE IS POINTER.
...
    CALL 'ADDR' USING WS-PARAM-1, PTR1
...
IDENTIFICATION DIVISION.
PROGRAM-ID.  ADDR.
DATA DIVISON.
LINKAGE SECTION.
01 PARAM-1 PIC X.
01 PARAM-2 USAGE IS POINTER.
PROCEDURE DIVISION USING PARAM-1, PARAM-2.
    SET PARAM-2 TO ADDRESS OF PARAM-1
GOBACK.
END PROGRAM ADDR.
END PROGRAM main-prog.

Now, you can MOVE your pointer to whatever area you want and CALL or
EXEC CICS LINK your module with that area.

And now to the sub-module. This one has to declare both the pointer it
gets from the main program and the real data in its LINKAGE SECTION. It
gets the pointer via the USING statement of its PROCEDURE DIVISION and
the has to SET ADDRESS OF the real data to the pointer:

IDENTIFICATION DIVISION.
PROGRAM-ID.  sub-module.
...
LINKAGE SECTION.
01 DFHCOMMAREA.
   05 PTR1 USAGE IS POINTER.
   ....
01 PARAM-1 PIC X(22000).
PROCEDURE DIVISION USING DFHEIBLK, DFHCOMMAREA.
    SET ADDRESS OF PARAM-1 TO PTR1 OF DFHCOMMAREA.
....
