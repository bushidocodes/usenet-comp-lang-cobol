In 'COBOLspeak' array=table.  Here is a post I recently made on the
subject.  Perhaps you will find it helpful.

Think of a table (any item which contains an OCCURS clause) to be several
copies of that item, all with the same name.  Because they all have the
same name, you must have another way to differentiate between them.  This
is done by appending a 'subscript' (or 'index', see below) to references
to the item.  The term 'subscript' is taken from math, where variables of
the same name but different incarnations are indicated by subscripts.

   a    = (a  + (n / a )) / 2
    i+1     i         i

A subscript is always an integer that indicates which one of the items
of the same name are being referenced.  Since computer languages (with
the notable exception of APL) are usually written using plain text,
true subscript fonts are not available, so COBOL uses parenthesis (C
and Pascal use square brackets) to designate the subscript.  You can
think of the table as a row of boxes, each with the same name, and the
subscript designates which of the boxes is being referenced.  In this
example:

   03  MY-ITEM   OCCURS 20 TIMES ...

   MOVE MY-ITEM(subscript) TO ...

The value of (subscript) determines which of the 20 MY-ITEM elements is
being referenced.  'subscript' can be a numeric literal, a numeric data
item, a numeric data item +/- a numeric literal, or an index, or an
index +/- a numeric literal.  It must always evaluate to an integer
between 1 and the limit in the OCCURS clause (20 in this example).
Indexes are a special type of subscript.  Indexes are associated with,
and must be used only with, a particular table, and are defined by the
INDEXED BY clause on that table:

   03  MY-ITEM   OCCURS 20 TIMES INDEXED BY MY-INDEX ...

This creates an index named MY-INDEX associated with the table MY-ITEM.
The specific internal format of indexes is left to the implementor, but
the general idea is that an index can be in a form other than an integer
representing the 'occurrence number' of the entry in the table.  It is
essentially a construct for permitting a form which can be manipulated
more efficiently by the CPU.  The details vary by implementation, and
are not important here.  There are a few differences in the way indexes
are used, as opposed to subscripts.  One is that you must use the SET
verb to change the contents of an index.  Instead of

   MOVE 1 TO MY-INDEX
   ADD 1 TO MY-INDEX
   SUBTRACT 2 FROM MY-INDEX

you would use, respectively

   SET MY-INDEX TO 1
   SET MY-INDEX UP BY 1
   SET MY-INDEX DOWN BY 2

You can also set and change the value of an index by using the VARYING
clause of the PERFORM or SEARCH statements.

   PERFORM ... VARYING MY-INDEX FROM 1 BY 1 UNTIL MY-INDEX > 20

If you declare a table with an index you can also use the SEARCH verb
to search the table.

   SET MY-INDEX TO 1.
   SEARCH MY-ITEM
      AT END ...
      WHEN MY-ITEM(MY-INDEX) = ...

If your table is completely populated and the values are in ascending or
descending order, you can use the SEARCH ALL syntax (see ASCENDING and
DESCENDING clauses of OCCURS), which may do a non-linear search (such as
binary search) of the table, resulting in much faster execution.  You
also do not have to set the initial value for the index before executing
SEARCH ALL.

Note that when you reference an individual table element, you must use
a subscript or index to indicate which element you refer to.  But when
you reference the table as a whole, such as in the SEARCH statement,
you do not use a subscript or index.

The power and usefulness of tables and subscripting is best understood
in the context of loops.  It is the use of these two concepts together
which permits writing one set of code which can operate on a number of
different elements.  Consider a program that tested some sort of code
to be valid, and there were 500 different codes.  Without the use of
subscripts, you would probably need to individually code 500 IF tests,
or 500 EVALUATE ... WHEN clauses.  What if there were 5000 different
codes?  What about 50,000?  With a table, you can load all the values
into the table, either through VALUE clauses in Working Storage if the
table is static, or from a data file.  Then just use a short SEARCH
statement or loop to test the codes against the values in the table.

If your table has subordinate entries like this

   03  MY-ITEM   OCCURS 20 TIMES INDEXED BY MY-INDEX ...
       05  MY-SUB-ITEM-1 ...
       05  MY-SUB-ITEM-2 ...

Then each of the subordinate items MY-SUB-ITEM-1 and MY-SUB-ITEM-2 are
referenced using subscripts or indexes in exactly the same way as the
item MY-ITEM.  The difference is that you may not reference any of the
subordinate items as the object of the SEARCH verb.  You can reference
subordinate items in the WHEN clauses of a SEARCH.

Two dimensional tables merely carry subscripting one step farther.

   03  MY-ITEM   OCCURS 20 TIMES ...
       05  MY-ITEM-A
       05  MY-ITEM-B  OCCURS 10 TIMES ...

Each of the 20 copies of MY-ITEM contains one MY-ITEM-A and 10 copies
of MY-ITEM-B.  There are 200 total copies of MY-ITEM-B in the entire
structure.  A reference to MY-ITEM or MY-ITEM-A is resolved by the use
of one subscript.  But if you reference MY-ITEM-B, you must use two
subscripts, the first to designate which of the 20 MY-ITEMS, and a
second to designate which of the MY-ITEM-B entries is intended within
that MY-ITEM.  Both subscripts are enclosed in the same parenthesis,
optionally separated by a comma.

   MY-ITEM-B(14, 6)
or
   MY-ITEM-B(14 6)

This references the 6th MY-ITEM-B, inside the 14th MY-ITEM.  You can
think of this as a grid of boxes, with the first subscript indicating
the row and the second subscript indicating the column.

Additional levels of subscripting merely carry this concept farther.

   03  MY-ITEM   OCCURS 20 TIMES ...
       05  MY-ITEM-A ...
       05  MY-ITEM-B  OCCURS 10 TIMES ...
           05  MY-ITEM-B1  OCCURS 5 TIMES ...
           05  MY-ITEM-B2 ...

   MY-ITEM-B1(14, 6, 3)

References 3rd MY-ITEM-B1, in 6th MY-ITEM-B, in 14th MY-ITEM.  You can
think of this as a 3D rectangle of boxes, with the first subscript
indicating the row, the second subscript indicating the column, and the
third subscript representing the level.  There are 20 * 10 * 5 = 1000
total copies of MY-ITEM-B1 in the structure, and 20 * 10 = 200 copies
of MY-ITEM-B2.

Two dimensional tables are used occasionally, three dimensional tables
are pretty rare, and you almost never see four or more dimensions.  I
won't even attempt to describe a method for visualizing a table with
more than three dimensions. ;-)  Thane Hubbell describes an interesting
way in chapter 12 of his book "Teach Yourself COBOL in 24 Hours", from
Sams.

there are 100 items to search and an extra, dummy item at the top of the
table (a must for efficiency to eliminate the extra comparison at the bottom
of the loop needed with VARYING),

   MOVE 0 TO SUB
   MOVE KEY TO TABLE(101)             
   PERFORM WITH TEST AFTER UNTIL KEY = TABLE(SUB)
      ADD 1 TO SUB   
   END-PERFORM
   IF SUB < 101 MOVE 'Y' TO FOUND                      

SEARCH ALL is no big deal, either. How many lines does it take to code
binary search? 

   MOVE   1 TO LO
   MOVE 100 TO HI
   PERFORM WITH TEST AFTER UNTIL LO > HI OR FOUND 
      COMPUTE MID = (LO + HI) * .5
      IF KEY < TABLE(MID) COMPUTE HI = MID - 1
      ELSE
      IF KEY > TABLE(MID) COMPUTE LO = MID + 1    
      ELSE 
         MOVE MID TO SUB-FOUND
         MOVE 'Y' TO FOUND
      END-IF
   END-PERFORM 

Better still, subscripts offer other possibilities. If the keys are
uniformly distributed, this can be transformed into interpolation search by
computing an interpolation point instead of the midpoint (runs in O(loglogN)
time instead of O(logN) for binary). Or how about organizing a key-indexed
(if the keys fall in a limited integer range) or hash table (any keys)
instead of a sorted array? Then neither sorting nor deduping is needed, plus
any search is done in at most 2 comparisons, hit or miss, regardless of the
number of keys. O(1) time, that is.

Judson,

By asking "How many lines of code...?" I did not mean the challenge of
coding linear search using as few lines as possible, just wanted to say "not
many". Your code excerpt, as Pete Withey has noted, will loop in the case of
unsuccessful search. That is why Pete added OR SUB > 100 condition:

PERFORM VARYING SUB FROM 1 BY 1 UNTIL KEY = TABLE(SUB) OR SUB > 100
END-PERFORM

But this is exactly what most programmers think is the right way to do a
serial search, whilst, in fact, it is inefficient. Note that this code makes
2 comparisons at every iteration of the loop. The code I gave (see above)
makes just 1 comparison. That's the entire reason for allocating a dummy
101th bucket and moving KEY there beforehand: It serves as a sentinel
allowing to eliminate the check SUB > 100 (or SUB = 100 if you code WITH
TEST AFTER). The subsequent logic is natural: If the loop stops at 101, the
search is unsuccessful, else it is a hit with SUB pointing to the node where
the right key resides.

Because of one less comparison, it will run, on the average, about 30
percent faster than the code you offered (with the change Pete has made), if
the number of items exceeds 5 in the case of a hit and 7 in the case of a
miss (tested; also see D. E. Knuth, The Art of Computer Programming, v.3.).

The above is a widely known general principle of speeding up inner loops of
high-performance algorithms: Instead of checking for a subscript still being
within the array bounds at each iteration (i.e. SUB > 100), place sentinels
next to the bounds and save the comparison. In sequential search, it is not
a very big deal since the algorithm is slow no matter whether it runs, 100
seconds or 70 seconds, say, - binary search will run 2 seconds, hashing will
do it in 0.5. But if you do something like SUB > 100 in the inner loop of
quicksort, expect its performance to plummet 2 or more
times.

There should be *no* difference between the two, at all, in this
example.  The COBOL 85 standard says on page VI-113:

  "If the TEST AFTER phrase is specified, the PERFORM statement
  functions as if the TEST BEFORE phrase were specified except that
  the condition is tested after the specified set of statements has
  been executed."

In other words, the PERFORM statement logic looks like this:
Before logic (see COBOL 85 standard page VI-114, Figure 1):

      <Set initial condition>
    Loop:
      <Execute TEST BEFORE Logic>
      <Execute specified statements>
      <Execute VARYING Logic>
      <Go to Loop>
    Exit:

After logic (COBOL 85 standard page VI-116, Figure 3):

      <Set initial condition>
    Loop:
      <Execute specified statements>
      <Execute TEST AFTER Logic>
      <Execute VARYING Logic>
      <Go to Loop>
    Exit:

For each of the above PERFORM statements this will result in:

  PERFORM VARYING SUB FROM 1 BY 1 UNTIL KEY = TABLE(SUB)
  END-PERFORM

      <MOVE 1 TO SUB>
    Loop:
      <IF KEY = TABLE(SUB) GO TO Exit>
      <Nothing>
      <ADD 1 TO SUB>
      <Go to Loop>
    Exit:

  PERFORM WITH TEST AFTER VARYING SUB FROM +1 BY +1
    UNTIL KEY = TABLE(SUB)
  END-PERFORM

      <MOVE 1 TO SUB>
    Loop:
      <Nothing>
      <IF KEY = TABLE(SUB) GO TO Exit>
      <ADD 1 TO SUB>
      <Go to Loop>
    Exit:

Since nither of these PERFORM constructs has any object statements,
it makes no difference if the end test logic occurs before or after
'Nothing'. :-)  There will be absolutely no difference in execution,
or result, of either syntax.  If your compiler gives you different
results, it is broken. ;-)

Of course, if you ADD 1 TO SUB inside the PERFORM, instead of using
VARYING, then there would be a difference.  Perhaps that is what
you were thinking of?

Well, you've been taught wrong. No _comparison-based_ method can do better
than binary search in the general case. In the case when keys are
distributed evenly throughout their range, interpolation search does much
better (you are correct in your estimates of loglogN), can be and is used to
improve performance. When you are looking a word up in a dictionary you do
not start in the middle searching for "COBOL". Instead, you intuitively
estimate how far the word is from the beginning and start searching there.
"C" is about 12 percent from the beginning of the alphabet, so you open the
book about 12 percent from the beginning. Likewise, searching for "SAS", you
would probably start closer to the end. This heuristic process can be
simulated using a simple proportion. That is, you would replace computing a
midpoint by computing an interpolation point leaving the rest of the logic
intact:

    MOVE   1 TO LO
    MOVE 100 TO HI
    PERFORM WITH TEST AFTER UNTIL LO > HI
                               OR FOUND
                               OR OUT-OF-RANGE
       COMPUTE INT-POINT = FUNCTION INT 
          (1+LO+(HI-LO)*(KEY-TABLE(LO)) /
          (TABLE(HI)-TABLE(LO)+1))
       IF INT-POINT > HI OR INT-POINT < LO 
          MOVE 'Y' TO OUT-OF-RANGE
       ELSE                                  
       IF KEY < TABLE(INT-POINT) COMPUTE HI = INT-POINT - 1
       ELSE
       IF KEY > TABLE(INT-POINT) COMPUTE LO = INT-POINT + 1
       ELSE
          MOVE INT-POINT TO SUB-FOUND
          MOVE 'Y' TO FOUND
       END-IF
    END-PERFORM       

+1 in the denominator is only to prevent zero-division without moving the
interpolation point far off target. If the keys are distributed uniformly in
a table with N=1,000,000 this code makes at most 5 iterations to either find
or reject the search key. Compare it with binary search (20 iterations).
Because a proportion is used to estimate the interpolation point, the
process can be badly fooled by grossly unevenly spread keys, so one should
exercise caution.  

>

Not at all. Any search based on address calculation (as opposed to
comparisons between keys) has this property. If a company has 50,000
employees each identified by a 5-digit non-negative integer ID, the entire
universe of possible keys is covered by the 0:999999 range. Suppose you have
10,000 IDs as keys to look up. You can sort them, load a table, and do
binary or interpolation search. However, you can also create a table with
100,000  entries and load it moving the satellite information accompanying
each key to the bucket whose number is equal to the value of the key:

IF TABLE (KEY) > SPACE MOVE SAT-INFO TO TABLE (KEY)

After that, searching is trivial: 

IF TABLE (SEARCH-KEY) > SPACE 
   MOVE 'Y' TO FOUND
   MOVE TABLE (SEARCH-KEY) TO SAT-INFO-OUT
END-IF
  
As the entire search amounts to a single array reference, the search time is
the same, no matter how many keys have actually been loaded. Note also that
the incoming lookup keys do not require any sorting, and unduplication
occurs automatically with SAT-INFO corresponding to the first instance of a
repeating key retained. If you want the last instance then load the
key-indexed table simply as

MOVE SAT-INFO TO TABLE (KEY)

I hope this illustrates the idea. Note that no comparisons between the
search key and the lookup keys are made. Of course, it works great only if
the keys are integers falling into a limited range, as in the situation with
IDs above. Hashing expands the idea onto keys of any nature and any range by
allowing short serial searches in the final stage of the algorithm. But the
fundamental properties remain intact: no sorting and/or deduping is needed,
and the operations INSERT, SEARCH, DELETE, RETRIEVE, UPDATE all occur in
O(1) time.

The PROCLIB statement is NOT a standard MVS JCL statement, but a feature
of some optional vendor program products which is not available in most
MVS installations.  The JCLLIB statement was introduced around 1991 in
MVS/ESA 4.1 and is what anyone can use to test JCL procedures.  Here is an
example:

  //JOHNDOE1 JOB (acct#),'John DOE', CLASS=A,MSGCLASS=X
  // JCLLIB ORDER=(JOHNDOE.PROCLIB,JANEDOE.PROCLIB)
  //  EXEC PROC=MYPROC2,DATE=19990808
