```
Having just gotten access to this news group, please forgive me if this
subject has been aired before but is there any real value to level 77
identifiers? I suspect they may speed up compilers and they may possibly
speed up access at run-time, but it seems to me that the overwhelming
disadvantage is the need to initialize each of them separately versus a
generalized
01 WS-MISC.
05 A-CHAR-VAR PIC X(23).
05 A-NUMERIC-VAR PIC 9(4) BINARY
etc etc etc.

We only moved to Cobol85 about a year ago and one of the things I'm
trying to promote within my co. is to use the above structure and then
INITIALIZE WS-MISC takes care of all the identifiers, character and
numeric. (One of the platforms our product runs on is the AS/400 where
IBM in its infinite wisdom decided to set numeric variables to spaces if
not otherwise initialized. That unfortunately leads to the dreaded
MCH1202 stop (numeric field has non-numeric value) and explains the
reason why we need to initialize most Working-Storage items at the
beginning of the program.) Most of the 05 level identifiers could be
level 77s but then we'd have to initialize each of the numeric ones
separately which doesn't seem worth the effort ---- unless someone can
really convince me that 77s are worthwhile. Thanks.

from David Silber Email: dav··.@hpd··o.uk
```

```
TTBOMK, the only architecture where 77 is different from 01 is Unisys A
Series. In A Series Cobol85 or Cobol74, a level 77 numeric item is
stored "on-stack", while everything else is stored "off-stack".
On-stack data is accessed more efficiently than off-stack data. Thus
changing 77 fields to 01 or subfields thereof is not "good programming
practice" on A Series.
```

```
David Silber wrote:
›
› is there any real value to level 77 identifiers? I suspect they may
› speed up compilers and they may possibly speed up access at run-time,

Records are an important part of the CoBOL language. The order of
variables within a record is significant. Level 77 is there so you can
tell your reader that a variable is not part of a record, and that the
order of its definition is not significant.

This simple concept got lost in implementation details. Some early
compilers aligned 77-levels on an 8-byte boundary, a penalty so severe
that few used them, and old habits die hard. On the other hand, there's
at least one midrange systems, not bound by mainframe tradition, or even
aware of it, that puts 77-levels on the stack for better access at run
time.

Personally, I put all variables that would otherwise be 77-levels into
one 01 FILLER so that I can align them. I use 77-levels only in
nonportable code for one specific platform.


› but it seems to me that the overwhelming disadvantage is the need to
› initialize each of them separately versus a generalized
…[9 more quoted lines elided]…
› numeric.

Never spend run-time instructions in the PROCEDURE DIVISION on something
you can do at compile time in the DATA DIVISION with the VALUE IS clause.


› (One of the platforms our product runs on is the AS/400 where IBM in
› its infinite wisdom decided to set numeric variables to spaces if not
…[5 more quoted lines elided]…
› doesn't seem worth the effort

There's no question that you should initialize all variables except
temps, but the VALUE clause is a better way to do this than the
INITIALIZE verb. Since the VALUE clause applies equally to 77-levels and
05-levels, it does not help in deciding between them. Alignment is the
factor that rules out 77-levels for me.

The INITIALIZE verb is like MOVE CORRESPONDING in that it saves typing
but not work. When you want to save work, group variables that must be
re-initialized by datatype, e.g.:

05 WS-TRANSACTION-AMTS.
10 WS-FOO_AMT PIC S9(16)V9(2) BINARY.
10 WS-BAR_AMT PIC S9(16)V9(2) BINARY.
etc etc etc.

Then MOVE LOW-VALUES TO WS-TRANSACTION-AMTS. This is significantly less
work than INITIALIZE WS-TRANSACTION-AMTS unless you have a better
compiler than the one on which I just tested this (VAX CoBOL).

When you have an area of mixed datatypes to re-initialize, copy the data
definition in twice, like this:

05 WS-AREA.
COPY "AREA.LIB".

05 WS-AREA-INIT.
COPY "AREA.LIB".

Then at the start of each transaction MOVE WS-AREA-INIT TO WS-AREA. This
is an order of magnitude (or more) faster than INITIALIZE WS-AREA, and
yet it has a smaller working set.
```
