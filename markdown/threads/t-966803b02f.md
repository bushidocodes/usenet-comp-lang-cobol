Well, I assume you are using mainframe IBM cobol; in this case, you
probably need to turn on runtime options MIGR or something like that;
there are about 3 options that need to be activated to allow older cobol
programs to call cobol II programs and vice-versa (without recompiling)
(and, presumably, other languages such as assembler, c, and pl/i). I
suggest you look in the programmer's guide to differences between cobol
and cobol II--especially the runtime options. I can look them up at
work tomorrow if you can't find them yourself. Be aware, however, that
cobol II support ends early this year and cobol LE is the only supported
language (although it's really not different except in the runtime
portion). You might consider going directly to cobol LE.

will make a difference.

You can just do an EXEC CICS ADDRESS command or you can pass it as a
parameter


Call asm-prog using
dfheiblk
dfhcommarea
parm-1
...
parm-n

and

Call cob-Pgm,((eibreg),(commreg),(parm1),...,(parmn))

*and don't forget to load the cobol runtime if this is the first
*program in the chain....

or you can do

dfheient eib=xx,comm=xx,stg=xx
...
exec cics address eib(my_eib_reg)

or

procedure division using parm-1 parm-2

exec cics address eibreg(dfheiblk) end-exec


or any combination of these so long as you can keep it straight, I would
suggest going
with whatever is cloest to your current parm passing method to minimize
headaches...

Last comment, the translator is rather dumb about inserting entry and
exit code, I always find it
easier to code that myself and use the options

*Asm xopts(noepilog, noprolog)

or

process cbl???xopts(nolinkage)

this keeps if from adding or removing items in your parm list and
generally makes life easier...
