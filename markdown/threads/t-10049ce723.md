foxgrove@shaw.ca wrote:

> items defined as "PIC 9(02) COMP" are shown as occupying two bytes

Yes, that's because, for the majority of COBOL users, comp implies binary, 
and binary is implemented as a 16 bit, 32 bit,  or 64 bit number, depending 
on the number of digits defined in the picture, and the compiler options in 
force..

Certainly, this isn't always the case, but this is a free tool which both 
Robert and myself wanted to keep simple.
>
> we use "NUMERIC SIGN IS TRAILING SEPARATE" in SPECIAL-NAMES
> a useful enhancement would be to have a stored option for separate
> signs

I've never come across "NUMERIC SIGN" (it sounds redundant to me... :-)) 
but, yes, the tool does not currently cater for separate signs unless the 
clause is applied at the picture level. (I'm not sure it does even then, but 
I could understand Robert not catering for it, if he didn't...)

I've never personally used this construct, although I can see cases where it 
would be useful. (data transfer across platforms springs to mind...)

COBOL is a very rich language with many variations in it.

I've recently come across a site where they use INVALID KEY and NOT INVALID 
KEY  frequently, with both clauses applied to the SAME READ...

The same place uses READ PREVIOUS without a preceding START REVERSED, 
expecting the Key to default to the prime key. It works. I have never seen 
these constructs used, in 40 years of looking at COBOL and had a great time 
devising and generating equivalent SQL for them... :-)

The point is that even if we had been building this tool as a commercial 
venture (and the main purposes of it were: to show use of nested procedural 
COBOL, to provide a useful COBOL parser, and to show how standard procedural 
COBOL can be wrapped as COM and run in .NET), were not under the time 
constraints we set, and had thoroughly researched the requirements, we would 
STILL probably have missed some possibilities... (The difference is that if 
you had paid for it, we'd be fixing it right now...:-)  That doesn't mean it 
WON'T be fixed or enhanced; just that both Robert and myself have other 
priorities at the moment. I'll be looking at a complete overhaul of the 
cobdata site (in fact, it is under way currently and I'm working on it on 
rainy days when I can't get to the beach...I do a large amount of design and 
problem solving sitting on some of our incredible Pacific beaches. :-), and 
I'll try to incorporate some fixes for known problems before it is 
re-released.) As we are coming to the end of the Summer here, rainy days are 
increasing so I'm hoping to have the site re-launched within a few weeks, 
before I leave for Europe.

Again, thanks for the feedback. Without that there is NO chance of it being 
changed...

Pete.

found the U.K. and New Zealand pretty much on the same page, Germany was 
similar with some minor differences, Spain was "interesting" but I never saw 
any START REVERSED or NOT INVALID KEY statements, and the USA and Italy I 
only worked on one site so it isn't a fair sample. I'd love to see what the 
Chinese are writing :-). Certainly, work being outsourced to India and Asia 
will probably have fairly rigid standards attached to it, but it would still 
be interesting to see if these diverse cultures take the same approaches.

It nearly blew my Migration Toolset out of the water and I had to burn some 
midnight oil to get it accommodated. In the end I built a stack - LIFO 
list -  (special type of collection in C# which comes with POP, PUSH, and 
PEEK methods predefined... very handy) so that any nested instances of 
(NOT) INVALID KEY/AT END  could be associated with the verbs they were 
connected with... this sort of thing...:

(Application code, being processed by a tool written in C# which translates 
COBOL ISAM access into invokes of objects in a Data Access Layer, that carry 
out the equivalent functions against a Relational DB. (The idea is to 
automate the conversion of existing code  from ISAM to RDB, but to ensure 
that exactly the same results are obtained off the new DB as were obtained 
from the ISAM files.) Obviously the new COBOL being generated must handle 
INVALID KEY/AT END conditions, even though these constructs cannot be used 
in the new environment.)


READ FILENAME
     NOT INVALID KEY
              CONTINUE
     INVALID KEY
               START KEY >= "LITERAL"
