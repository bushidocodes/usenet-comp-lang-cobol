An "index name" is a data item declared in the "indexed by" part of an
"occurs" definition. For instance:

05 A-GROUP-ITEM OCCURS 25 INDEXED BY AGI.

In this case, the "AGI" is an "index name".

An "index-data-item" is an elementary data item declared separately from
an "occurs" definition as a stand-alone field much like any other field,
except (in my case with IBM MVS, COBOL II) you don't give the field a
PICTURE clause because there is only one way to store an index data
item. For instance:

05 A-INDEX-ITEM USAGE IS INDEX.

The "A-INDEX-ITEM" is an "index-data-item".

The field AGI (an "index name") WOULD be allowed in a PERFORM XXXX
VARYING statement, where the field "A-INDEX-ITEM" WOULD **NOT** be
allowed.

The reason lies in the difference between the two fields. AGI is an
index associated with a data item that OCCURS, but more importantly, is
of a specific size (calculated by the compiler).

Since this is so, the COBOL compiler can use this size to figure out how
to change the value of the index when you wish the value to change from
"1" to "2", for instance. Changing the value of the index takes some
calculation, because the index does not store directly the "1" or the
"2", but it stores an "offset from the beginning of the table" that
corresponds to this occurrence number. For occurrence "1", this offset
is zero (since the first occurrence starts immediately at the beginning
of the table). For occurrence "2", the offset is the length of one
occurrence of the data item that OCCURS, which is why it is important to
know that size (or, for the compiler to know the size).

A-INDEX-ITEM, an "index-data-item", however is **NOT** associated with
any particular table, or data item that OCCURS. It can be used to hold
the value of an index (the offset value, **NOT** the occurrence number),
and that is about all. It CANNOT be used in the PERFORM VARYING
statement, since it is not associated with an OCCURS, and therefore has
no known length value to use when varying the value.

On the other hand, I expect that the following would work, but why would
you bother?

SET AGI TO 5.
SET A-INDEX-ITEM TO AGI.
MOVE A-GROUP-ITEM (A-INDEX-ITEM) TO B-PRINT-ITEM.

When I have needed to hold the value of an index, I have simply defined
multiple indexes for the table, up to a dozen or so for a particularly
complicated situation I remember.

Your example is exactly what you can do (of course the comma is optional as
always). This is handy and does work well. The only problem that I have
ever seen with it, is when someone forgets that when you set one of the
indexes, the other remains unchanged - so they try to use the wrong one.
For example,

Code that won't work as intended (probably)

05 aTable occurs 10 times indexed by ind1 ind2.
10 Elem Pic X.
...
Perform xyz varying ind1 from 1 by 1 until some-condition.
Move Elem (ind2) to another-field

Code that is probably what the programmer meant.

05 aTable occurs 10 times indexed by ind1 ind2.
10 Elem Pic X.
...
Perform xyz varying ind1 from 1 by 1 until some-condition.
* Either option 1
Move Elem (ind2) to another-field
* or option 2
Set Ind2 to Ind1
Move Elem (ind2) to another-field

OBVIOUSLY, the very first code MIGHT be what the programmer really meant -
if they intended on using the value of the index-name from before the
PERFORM VARYING - but I am saying what is "right and wrong" based on
assuming the application logic intended to use the index value determined by
"some-condition".
