The general COBOL using community probably uses little or no HFP
(COMP-1 and COMP-2) which I suspect was there primarily for
compatibility with FORTRAN.  The need for BFP (defined by the 2002
USAGES) again would probably be of little interest to the average CICS
or batch COBOL shop.  However, for anything that in new world (C/C++,
Java, etc.) not having BFP poses a potential interoperability problem
that is only partially fixed by the conversion kludge when calling
Java.  This is because the kludge does nothing for data that is
accessed directly by both COBOL and other languages.  Since I have had
nothing to do with mainframe related web page serving or handling of
graphics (catalogs, retail order sites, etc.), I don't know how much
floating point means in those environments.  Does SAP use floating
point?  Do other business packages use binary floating point?  It is
kind of a sleeper issue that would probably cause the MEGO (My Eyes
Glaze Over) effect on most CFOs and CEOs and maybe some CIOs.  If Pete
is right about the increasing irrelevance of COBOL and I can make an
excellent case for that view, then ignoring BFP makes good sense.  If
in the mainframe arena COBOL still has a future, then given the
investment IBM has made in z series BFP floating point performance
somebody hasn't gotten the IBM strategic message.  The need is not BFP
per se.  The need is to handle all data created by other languages and
packages where COBOL will be using that data and to pass data to
programs and packages in those languages.  If IBM doesn't use the 2xxx
COBOL Standard usages for DFP someone should get serious flack.

Yes and no...

SAP these days is "tailored" through an API which uses a language called 
ABAP. This is an OO language which supports Classes and Objects so data 
types can be defined in it. It uses a Dictionary to define basic types but 
you can also define data types "on-the-fly" or say you want them based on a 
previously defined object. Numeric types can specify how many decimals are 
required, so there is no real need for floating point, but you COULD define 
a floating point type if you wanted to.

Generally, the "non-COBOL" languages differentiate between "real" and 
"integer" numbers, using floating point for real, but you can define fixed 
point decimal numeric fields as well.
