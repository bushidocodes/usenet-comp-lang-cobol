```
In a COBOL/BATCH program (Compiled using the DYNAM option), we want to
establish addressability to a Static/External ASSEMBLER Table (CSECT),
using a PROCEDURE-POINTER, which was Assembled as AMODE(31)
RMODE(ANY).

Our question is whether the RETURN-CODE Special-Register is set to a
non-zero value AND/OR the PROCEDURE-POINTER itself is set to NULL if
the External Table address could not be resolved?

In reading about PROCEDURE-POINTER, its usage that we are attempting
resembles that of an MVS "LOAD" Macro, although we're unaware whether
there an underlying differences.

Example code -

WORKING-STORAGE SECTION.
03 WS-PP PROCEDURE-POINTER.
03 WS-PP-X REDEFINES WS-PP PIC X(08).
03 WS-TABLE PIC X(08) VALUE 'EXTERNAL'.
03 WS-PTR POINTER.
03 WS-PTR-X REDEFINES WS-PTR PIC X(04).
LINKAGE SECTION.
01 LS-TABLE-REC.
   03  LS-TABLE-NBR-ENTRIES PIC 9(09) BINARY.
   03  LS-TABLE-TBL OCCURS 1 TO 1024 TIMES
                    DEPENDING ON LS-TABLE-NBR-ENTRIES
                    INDEXED BY LS-TABLE-IDX,
                               LS-TABLE-IDX-MAX.
       05  LS-TABLE-ENTRY PIC X(12).
PROCEDURE DIVISION.
SET WS-PP TO NULL.
MOVE ZERO TO RETURN-CODE.
SET WS-PP TO ENTRY WS-TABLE.
MOVE WS-PP-X TO WS-PTR-X.
SET ADDRESS OF LS-TABLE-REC TO WS-PTR.
SET LS-TABLE-IDX-MAX TO LS-TABLE-NBR-ENTRIES.

Thanks in Advance....

Bill
```

```
Cool!  I think you may have actually found a use for IBM's
implementation of procedure pointer.

I cannot say for certain if this is proper and documented behavior,
but I know on the 2.1 and 2.2 versions of Cobol for OS/390 this will
work.  The procedure pointer is the address of the entry point
followed by 4 bytes of nulls (or is it 4 bytes of nulls followed by
the address).

Thanks for the great idea, I can now throw away my LOAD macro
assembler module.
```

```
RETURN-CODE, a NULL value in PROCEDURE-POINTER or both. I'm also going
```
