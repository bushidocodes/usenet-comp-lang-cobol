       01  MY-TABLE.
           03  MT-LEVEL-1         OCCURS 5 TIMES.
               05  MT-LEVEL-2         OCCURS 10 TIMES.
                   07  MT-ITEM            PIC  9(04).

           MOVE 0 TO W-COUNT.
           PERFORM VARYING W-SUB1 FROM 1 BY 1 UNTIL W-SUB1 > 5

               ADD MT-ITEM(W-SUB1, W-SUB2) TO W-COUNT.

Point of information... as I recall it, a nested IF has different levels
of logical dependency; the simples cases:

IF condition-1
    IF condition-2

... are the equivalent of IF condition-1 AND condition-2.

In an IF-ELSE structure all conditionals are at the same level of logical
dependence.

(note: this point annoys me when it comes to matters of indentation; I was
taught that indentation should be correlated to this.  Given this
teaching:

IF condition-1
ELSE
    IF condition-2

... or ...

IF condition-1
    ELSE
    IF condition-2

are both misleading and *wrong*; the only Good, Decent and True way to
code is:

IF condition-1
ELSE
IF condition-2

... maintaining, in the indenting, the appropriate level of dependence and
evaluation.)

From some of the answers, it appears my question was a bit simplistic,
in that the ways system design and programming are approached may be
very different.

When I do system design, I visualize the system in a sort of symbolic
block form.  I can't quite describe it, but I see how everything will
relate together and function.  I don't visualize code here, except in
areas where I want to think through how I will handle some specific
problem.  And then it is something like the 'private pseudo code' that
Donald described.  This is done as I conduct the user interviews, and
is usually complete by the time interviews are complete.  The rest of
system design is mostly documenting the process in detail, which I do

I've been pondering this one all week.  I don't think of it as visual
exactly, but I suppose that's as close as the English language can come to
describing it.

Visual analogies are common in data processing.  I recall a customer
support rep was mystified when I said our Saab terminals were TC
look-alike.  She had seen both of them, and they didn't look at all
similar.  Also on Burroughs Medium Systems, moving head disk drives with
100 byte sectors were described as head-per-track look-alike.  In both
cases, "look-alike" means the data format is the same.  Some IBM virtual
storage systems use a tranlation look aside buffer, pipelining involves
instruction look ahead, some compilers use peep hole optimization, and in
Cobol we scan with INSPECT and SEARCH to lookup data.  If we say that the
computer sees data and code (as opposed to hearing, feeling, tasting, or
smelling), then I see it in that sense.

To me the real question is whether you see it statically or in motion.
When I worked for Burroughs/Unisys I was surprised at how many otherwise
good programmers didn't have "the multi-threaded mind set".  The KEYEDIOII
library suffered for several months being maintained by a programmer
without the multi-threaded mind set.  

For the most part you didn't need to think multi-threaded to work on the
compilers (a compiler is batch application) but there was an exception.
On pre-beta (pre-ASD) A Series systems, we had this unholy process called
stack searching.  When an area of memory was paged out to disk, all stacks
with visability to that area had to be searched for copy data descriptors,
so they could be marked absent.  Stack searching also occured when an area
was resized; all the unindexed copy descriptors needed to be found to have
their size updated.

Descriptors have four frame sizes:  4-bit, 8-bit, 48-bit, and 96-bit.
Actually we were still supporting the B6900 which had 3-bit and 6-bit as
well.  The hardware had no atomic instruction to change the frame size in
a descriptor; the compilers generated multiple instructions to extract the
bits, modify them, and then put them back.  If a resize occured in
between, the copy descriptor now had the wrong size.  If this was too
short, your program might abort accessing data that was really there, but
worse, if you size was now too long you could clobber data which belonged
to another program.  The Algol compiler rigorously did its descriptor
updates in an order which would never result in too long a length, so I
conclude it was written by someone who understood this, but the other
compilers were sloppy until I rediscovered this bit of wisdom 15 years
later.
