Alain,

I'm not familiar with the library you are using but the thought strikes me 
that instead of using C type typedefs it might be better to use object 
references. Can you instantiate the library and then invoke the methods as 
methods of an object?

The other thing I noticed is some confusion over "by value" and "by 
reference"... if you specify one of these it remains in effect until you 
specify the other, and the default is "by reference", for COBOL. So, try 
changing...

call "HPDF_Page_Rectangle" using by value mypage, 50, 50,
mywidth1, myheight1

..to be..

call "HPDF_Page_Rectangle"
      using
             by value
                  mypage, 50, 50
             by reference
                  mywidth1, myheight1
       ON EXCEPTION
              ...whatever...
end-call

You might need to address this in other places as well; I haven't fine 
combed the code you posted.

Let us know what happens :-)

I create PDF files for invoices, statements, etc to be emailed but I
wouldn't do it by hard coding it into the program. I do it via a
postscript template and then run ps2pdf.

To draw the template I use tgif (http://bourbon.usc.edu/tgif/
index.html) but several other drawing programs could be used as long
as they output text fields contiguously. To mark the text merge points
I create text fields as: <!TAG> where TAG is the name that I will use
to merge. Several variations are used such as <!TAG---------> to fill
out to the length required, the '-' are ignored, and for multiline
fields:

<!TAG---------->
<!>
<!>

to the number of lines.

I do the merge using a Python program (There was a C program but the
Python one is much faster). The merge data is written out by the Cobol
program (or other) in a text format that I had used since the late 80s
for a commercial DOS package called FormFill which did the merge on a
proprietary graphics format (the design program ran on Windows 2).

This approach has several advantages (to me):
The Postscript can be printed directly via CUPS to any printer.
Or it can be converted to a PDF for emailing.
The format can be quickly and easily changed without changing the
program.
Several different variations can be available just by using a
different template name.

I think that you miss the point. Creating the postscript files or PDF
files by a series of CALLs to a library is _NOT_ 'the same way'.
Templating completely separates the production of the data by the
Cobol program from the presentation. The postscript is created by a
screen design process that even the users can do. The Cobol merely
outputs the data in a simple text file. The merge utility neither
knows nor cares what the data is or what the output is, it just merges
according to simple rules.

The final output could be postscript (or pdf via ps2pdf) or HTML or
XML or simple text and this can be changed completely without changing
the Cobol program at all.
