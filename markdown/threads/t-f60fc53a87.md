long-running program. What I'd like is is one of those ever-extending

verb.  This, I find, is one of the most frequently overlooked features of

I've just finished testing a replacement for my long runnin job. The

The problem was unsing an HDAM DLI DB. I need to read a root, read a child,
use the thild to get the key value of an XREF'd root and then read that
root.
I process what I've read, then read the next input in key sequence order,
get

My replacement process builds a DB2 table. I extract all roots and all

of the type that contain the xref key value. Then I use synchsort,

30 minutes - not in the critical path.

information I
need in a single read.

P.S. For the DLI read, I tried using 2 PCBs, I tried 2 pass processing and I

now code

tape mounts!

This was very interesting Rufio.

Your solution is smart and efficient.

However, I believe if your original IMS DB had been physically organised as
HISAM you would've got much better performance from it.

The problem with sequential access to HDAM roots is that they are accessed
by hashing, which is extremely efficient for random access but not so good
when you are accessing in key sequence.

By changing the organisation to HISAM you have a very good chance that
sequential records will be physically buffered together and you can still
access them randomly.

Poor old IMS suffers from being perceived as "outmoded" technology (a bit
like COBOL really...<G>) but for many years (up until 1983 actually) we used
it as the "only game in town" and found it very useful. By fiddling with
buffer sizes for indexes and data I believe the performance you describe
could have been improved many times. However, I don't blame you for going
RDB, and the solution you describe is excellent.

In my opinion, IMS-DC is still an infinitely superior TP Monitor to CICS,
but again, it has been tarred with the same brush as DL/1 IMS, "outmoded".

We need to remember that just because something is old, does not mean it
isn't any good... <G>

Lao Lao Lao,

I'm an oldtimer and thank God and the Standards Committee for giving us
reference modification.

Back to the problem at hand. I like Rufio's solution; it shows that PRODUCTIVE
work is being done. But I think all "long running" programs should be
performance reviewed (preferably before it goes into production). A case in
point: a few years ago I was asked to write a pgm to print invoices for a
telecom company. The specs ran something like this:

Prepare invoices for printig. Each invoice can contain up to 4 page types, each
type can contain a varying # of pages. All pages must be numbered. Each page
type is contained in its own dataset, sorted by acct#.

Read all invoice pages for an account and write to a temp dataset. Read the temp
d/s, number the pages, and write the invoice set to a final d/s.

OK, the pgm processed 20,000 accounts and created 3 million print lines and ran
3+ hours. It did other things of course, but this is the area that caused the
problem.

I mentioned to the PM that wrote the specs that opens/closes of the temp d/s is
probably eating up that time. I noted that the vast majority of the invoices
were 8 pages or less (one was 2500 pages) and suggested that an in-memory table
of 10 pages could be used to reduce the opens/closes. She didn't think it was
worth the effort. Anyway, I did it on my own time.

The results were AMAZING. I thought I could reduce the run time by an hour or
so: the job ran in 10 MIN!!!!! Who would have thunk? 

The moral of this little tale is: spend some time looking at why a job runs x
hours. Sometimes the answer is that's the best that can be done. But sometimes
the answer is 10 minutes.

Jack

Listen people, anyone who has programmed more than a year *ought* to
know the following:

1. Processing an entire file using random I/O is *MUCH* slower than
   using sequential I/O, simply because the I/O subsystem anticipates
   sequential reads and attempts to pre-fill the file's input buffers.
   Reading a sequential file from beginning to end is *many times*
   faster than doing the same to a random file.

2. Creating/deleting records in an indexed file structure is *very*
   expensive, relative to reading/writing a sequential record,
   because the index tables must be searched and updated for each
   add or delete, in addition to doing the actual add/delete, as well
   as other internal I/O.  And the index tables, as well as the data
   file, are all random access.

3. File open and close operation are *expensive*, for several reasons.
   An open requires at least a partial directory search, which can
   involve many directory I/O's (and remember, the directory is a
   shared resource with heavy activity) which may cause operating
   system overlays as well.  An open output also requires creation of
   a new directory entry, which could be several I/O's.  Both open and
   close operations require activity in whatever area is used to track
   use of shared resources, which can cause additional I/O's.

4. Memory access is millions of times faster than disk I/O.

5. Operating systems calls are (at best) hundreds of times slower
   than in-memory processes.

If you doubt the above statements, simply write a few test programs
and you will see for yourself.  It boggles my mind that there are
experienced programmers out there who have never bothered to find out

going on inside that box.  If you aren't willing to do that, don't

easier.
