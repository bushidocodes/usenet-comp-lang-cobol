It's not a SYSIN statement, it's not a JCL-Statement, it's just data in
this case. You could code //SYSIN DD DSN=.... as well and you will
understand that in this case, the JES interpreter is not allowed to
change your data, just because there is a '&' character in it.
Instream data is treated in exactly the same way as external data, so
you canot use symbolic parameters.
You could write your own little program which receives the name as
parameter and writes the 'DEL ...' statements on a file. Take this file
and use it as input to your delete-step:

//S1       EXEC PGM=yourpgm,PARM=&RDATE
//MYFILE   DD DISP=(NEW,PASS),SPACE=(TRK,1),
//            DCB=(RECFM=F,LRECL=80)
//*
//STEP10   EXEC PGM=AASCR,PARM=NOCOND
//SYSPRINT DD SYSOUT=*
//SYSIN    DD DSN=*.S1.MYFILE,DISP=OLD

You can now put data cards inside a PROC.  There is a religious
war about its merits.

JCL has some new commands, SET to set a variable and IF THEN ELSE
ENDIF to conditionally execute job steps. The IF is not real
flexible (most notably, if can't compare the value of a symbol to
a constant or another symbol value), but you can now test for
ABENDs, the ABEND code, and if a job step ran.  Some system
symbols are also available such as the RACF userid, which is nice
for generating data set names.

The precedence of setting a symbol value is a little difficult to
express, understanding that it can be changed by a SET statement
inside or outside a PROC, and when the value on an EXEC statement,
or a PROC statement is not overridden.  Nested procedures are the
complicating factor.

F.Y.I., the syntax of the SET command is:

SET parameter=value,[parameter=value,.....]

parameter = The name of a symbolic parameter you want to assign a
value to.
value = The value you want assigned.

The advantage of the SET statement is that you don't have to use it in
conjunction with procedures at all.  For example, suppose you have a
job that you frequently use with only minor variations.  If you use a
SET statement at the beginning of the job to set a symbolic parameter
that's used throughout the job, you can use a text editor to change
the SET statement each time you submit the job.  There's one
peculiarity of the SET statement you need to be aware of:  It's not
subject to the conditional processing imposed by IF, ELSE, or ENDIF
statements.  If you enclose a SET statement within an IF/ELSE/ENDIF
construct, the SET statement will be executed regardless of the
results of the IF statement's condition tests.
