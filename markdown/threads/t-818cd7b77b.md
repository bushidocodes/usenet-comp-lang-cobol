I have run into this "issue" more with those using MF for development than 
for those migrating off the mainframe and to Unix/Linux/Windows.


Having worked on "both sides" (customer and MF employee) and having worked 
with other "off-loading" of mainframe development, the "usual" problem is 
that mainframe developers are used to their existing (mainframe) testing and 
development environment. They also expect there to be "systems programmers" 
who maintain and set up the environment.


Duplicating a complex mainframe environment on each PC or even on a network 
takes extensive work. (This is not just for Micro focus, but for all 
workstation development platforms). There are things like (and this is NOT 
an exhaustive list)


1) JCL, Utilities, Source control systems, etc

2) IMS, CICS, and DB2 "control" settings and facilities (test regions, etc)

3) Test data (and all related files)


I have seen a number of "development off-loading" projects fail, because it 
was expected that every developer needed to "set up" their entire 
development environment - as opposed to "status quo" development (on the 
mainframe) where everything is already set up and the tools are what the 
programmer was "used to".


When it comes to actual migration of (production) applications from the 
mainframe to the "workstation" or "network", this is USUALLY (but not 
always) less of an issue. Usually appropriate resources (knowledgeable) 
people are either allocated or hired to figure out what needs to be done to 
"move" the production environment from the mainframe to the 
workstation/network. This is viewed as a "one-time" project and WHEN/IF the 
resources are allocated, then the migration often (usually) succeeds. 
However, even here, if mid-/upper-management has been "sold" that this is a 
transparent migration and that the vendor provides all the tools to make 
this "easy", then I have seen this fail as well.


This may or may not be what Alistair was referring to, but it does express 
when I have seen such project fail or flounder.

Depending on the requirements for a test environment, there is nothing
better as the same platform with the same software versions.

On the other hand, there are situations where a system coming close is
good enough. I can recall a PC based environment we had for developing
online applications for Unisys mainframes and this worked well enough
to get application development up to a point where it could be copied
to the maninframe, compiled and did work. Proper testing had however
still to be done within the target environment, no doubt about that.

I have never experienced myself a mainframe migration project to fail,
though I know of quite some that did.

So it is all a question of how this is approached and what technology
is used.
