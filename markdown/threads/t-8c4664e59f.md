This subject takes some research on my part.  I know what I want
to do, but I can't produce correct code off the top of my head for
Report Writer and XML.  So I going to do these posts in parts.

Part 1 -- Theory of Documents

A document has pieces.  Each piece is delimited and named (i.e.
tagged).

Using HTML like syntax (crossed with IBM DCF aka Script son of
Un*x roff), here is an example.

<DOC NAME=asdf>
<TITLE>Document Title</TITLE>
<H1>Outermost heading</H1>
<P>A paragraph</P>
<LIST BULLETED>
<LI>Item 1
<LI>Item 2
<LIST NUMBERED>
<LI>Subitem 2a
</LIST>
</LIST>
</DOC>

For a report, you might see
 
<REPORT>
<H1>Report Head
<H2>Group Head
<LINES>
detail line 1
detail line 2
<T2>
Group totals line 1
Group totals line 2
<LINES>
detail line 3
<T2>
Group totals line 1
Group totals line 2
<T1>
Report totals
</REPORT>


Now when you print on a page, if you know the page dimensions as
well as the dimensions of each line, then you are in.  With
standard height lines, you count lines just like report writer has
been doing since day 0.

The next advance is when you know the dimensions, say in points or
% of line heights or something else you can turn into inches.  The
layout is similar to counting lines, but you have factions of a
line (i.e. inches or cm).  But there is not a big problem figuring
if a group will fit on the remain page, or if a detail line will
be widowed or orphaned.

You can also replace the <tags> with escape sequences for a
printer, if you know something about the hardware.  I'd expect a
COBOL compiler for common use on a platform to know a little bit
about HP PCL printers, IBM 3800 printers, AFP printers, TTYs, what
ever matches the platform where the compiler is going to be
sold/used a lot.

But what if you don't know escape sequences?  Enter XML. 

Then you turn the job over to an external program.  I'll use a WWW
browser as an example.

If the tags with the correct HTML ones (<HTML> <HEAD> </HEAD>
<BODY> <CENTER> <TABLE> <SMALL> <P> </BODY> and a host of others)
then the output could be sent to a file, and that file loaded into
the browser.  The browser PRINT button could give hard copy.

Note that the browser doesn't have to insert page breaks.  WWW
browsers don't when displaying on the screen, and can when
printing.  When an external program is processing the COBOL
program output, some cooperation is needed so that report writer
does not insert page breaks if the typesetter does not want them
(e.g. screen display) or will decide itself where to put them. 
For example, it can format a report writer group to see how big it
is, then decide based on the current paper in use if it will fit
on this page or the next.  

The chief objection to WWW Browsers is that they lay out a page as
the choose to be viewed on a screen in a variable sized window. 
Most people would like to print to a fixed size piece of paper and
put print lines exactly where they want (fill in a form).

To overcome this, XML is being deployed.  GML (IBM DCF/Bookmaster)
HTML, XML, and others are defined by a very complex standard
called SGML.  SGML in all it's glory and complexity is used in
some type setting applications as well as others (indexing,
viewing, audio readers).  Being too complex to implement with good
performance and easy, SGML has been used to define XML.  XML is
itself a way to define tags, and how they are to be processed. 
That processing can include pixel resolution, type styles, style
sheets.  Almost anything you can ask a type setting program to do
can be marked up in XML.

Filling out forms is trivial IF you have a set of tags defined
which understand where things are on the form.  For last printing
subsystem I did, I caused the programmers to tag everything
including return addresses and zip codes.  The printing system
counted "to" addresses to see how many copies, printed BAR codes
for mailing from the ZIP codes, and even sorted by ZIP code for
the cheaper mailing rates.  Programmers first ask "why can't I
just tag an address as a bunch of lines (with line breaks)".  I
explained that the tag was not formatting directions (don't flow
these lines) but usage information (this is mailing ZIP info).

So now we face the problem of everyone and his sibling defining
tags and their implementation.  That's where the COBOL report
writer standard will have to exert some controls like "The report
writer expects the tag definition to include/not include these
actions".

For example, a tag that report writer will insert to delimit a
group could be required to maintain vertical alignment of the
enclosed lines so that a font with variable width characters can
be used, and duplicate information on lines within a group can be
blanked out on lines after the first.  (There is better way to
handle this problem -- see tables).

Next I want to conjure some report writer syntax and maybe a set
of tags with known dimensions in another post.  Feel free to type
yourself!
