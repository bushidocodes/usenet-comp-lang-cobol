As you can see, not only did allocation occur but a mount request showed
up because there was no request for defered mounting. Note that ,,defer
is a request for defered mount (at open time instead of allocation
time). ,,defer is *NOT* a request for deferred allocation. That is
called dynamic allocation and there is no provision in JCL for dynamic
allocation.

So, for a system with no online tape drives, the IEFBR14 technique will
always fail in allocation, but it will probably uncatalog the data set
anyway once the operator replies to the allocation recovery message. Any
steps after it are certainly at risk of not being executed.

The IEFBR14 technique will also result in allocation delays for other
jobs allocating the same device type. This is because allocation
serializes on the device type.

I use IEFBR14 sometimes myself. But it can cause other problems. IDCAMS,
as bloated as it is for this situation, avoids the allocation of the
tape drive.

There are several reasons the iefbr14 is preferred.

1) It uses less memory
2) It doesn't open the tape dataset
3) If there is a problem, the job aborts
4) IDCAMS return codes don't trigger abends unless the
installation has special exits
5) SET LASTCC = 0, means that you're ignoring possible
problems other than "dataset not cataloged"
6) UNIT=(,,DEFER) is the way consultants befriend operators
and gain influence in the queue
7) UNIT=(,,DEFER) is a good JCL-programming practice because
in large JCL streams, the system does not mount ALL
the tapes required at job start time should your job
read/write several differently named tape datasets.
8) UNIT=(,,DEFER) is less typing than IDCAMS utility
invocation.
