Shaun,
I know that you are used to using compilers (PC particularly) that use
line sequential format for compiler source code. As I said (internally)
when I worked for Micro Focus, in my opinion this is "convenient" but is NOT
ANSI conforming. I "real ANSI conforming compiler" should use all the
character positions up to the R-margin (column 72 or 80 for most
compilers) - even if they are spaces. In which case, it remains true that
an asterisk in column 7 would be more efficient than 80 (72) blank
characters.

ON THE OTHER HAND,
A) most PC (and Unix) compilers DO use "line sequential" files for source
code - and in such cases, have the "quick CRLF" is faster than going to
column 7.
B) The speed of a compiler is rarely (but not never) a deciding factor in
how to code. Even on the slowest PCs today, compiling a 10,000 to 50,000
line program should not be prohibitive - compared to having a WELL
documented set of source code. What does matter is speed of execution and
blank line vs comment line shouldn't make any difference to that. (Well if
you are running under a full-screen debugger, it might have some impact, but
even there I think it would be minimal.)

What I learned from an assembler goeroe about the execution time needed for
charecter string processing:

The time to fetch the instruction to the CPU registers
+
The time to decode it (i.e. see what it is)
+
The time to address the involved memory location(s)
adds up to some fixed amount which is some levels of magnitude bigger than
the amount of time needed to process one character. (Ten or even hundred
times the time for one char). In other words, the mere execution of an MVC is
the cost, not the length of the strings ...

... unless of course the strings are very long. He showed me some assembler
test with which he proved that there is no significant time difference
between an MVC of 5 characters or one of 50 characters. Above hundreds there
will be noticable differences of course. (Above 256 bytes a multiple of MVC's
is used or the once new MVCL is used)
