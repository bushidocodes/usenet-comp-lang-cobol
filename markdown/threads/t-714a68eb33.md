Anyone attempted using Microfocus COBOL (v3.2) and sockets under Unix?
I need to develop a server program using MF COBOL to get at MF data
files via socket from other environment/language connections (eg,
Java, C, etc).  We have a huge legacy system with gigabytes of MF
COBOL data, and no plans of upgrading to NetExpress.

I realize you can mix C & COBOL under MF, but coming up short on how
to communicate the results of a MF COBOL file operation back to a
driving C socket wrapper (ie, C calling COBOL program).  For example,
if I have a C program that calls COBOL to read a record, how do I pass
the record data _back_ to the C program so it can pass it back across
the socket connection?  Looks like all I can return is a status value.
Would using env variables be an option?

Any help/insight would be greatly appreciated.
```

In the C program declare a buff for the data
and pass the address of that buffer to the cobol 
program as a call parameter.

rc  int;
c_buffer char[200];

rc = cobolprogramname(&c_buffer[0])
if (rc) do the stuff with the file data
else you are finished

The COBOL program has to have a linkage section declaring
the parameter and it places the record read in the buffer

working-storage.

01  file-is-open pic 9 value 0.
  88  file-open value 1.

linkage section

01  data-buffer  pic x(200).

procedure division using data-buffer.

    if not file-open
        open file
        move 1 to file-is-open
    end-if.

001-loop.

    read file
       at end go to 002-finish
   end-read.

   move record-area to data-buffer
  exit program returning 1.

002-finished.

    move 0 to file-isopen
    close file
    exit program returning 0.

The program returns a 1 for the return code as long
as file data is returned and a 0 if the file is at end.

Check the documentation to make sure the RETURN-CODE
register is the same as the C int data type.

If you working with ISAM files and you want to obtain
specific records with a key value you will have to
pass the key through another parameter and use 
that for the read, and come up with some return
code convention to indicate a "not found" on the
read.
```
