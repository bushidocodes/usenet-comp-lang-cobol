Actually, all COBOL and AFAIK any language product will state that the
contents of a file buffer are "undefined" if the last READ was not a valid
read; and a READ which results in the AT END condition is not a valid read.

the last valid read. Next time you run the program, though, that may not be
true.

"9.1.2 Record area
The record area is a storage area associated with a file in which logical
records from that file are made accessible to a runtime element. The record area
is made accessible to the runtime element at the completion of a successfully
executed OPEN statement. For files open in the input mode, the logical record is
available in the record area after execution of a successful read. For files

When the AT END is followed it is an 'unsuccessful READ', ie the File
status will not indicate 'success' with a '0' in the first position,
in fact it will have '1'.

> ... the
> logical record is available after execution of a successful read until the
> execution of either a read or successful rewrite."
 
> I believe that the record area is available *and* defined after an AT END
> condition (assuming there was a preceeding successful READ) 

The record _area_ is available until after a successful CLOSE.  The
content of the area (ie the logical record) is only defined after a
successful READ and is undefined after a successful WRITE or REWRITE
or after an unsuccessful operation.  A READ that takes the AT END
branch is unsuccessful.

> - but becomes unavailable during the CLOSE. 

Unavailability is irrelevant and was not part of the problem. You are
confused. When the standard states "the logical record is available"
it is saying that the data items are defined and is not referring to
the availablity of the buffer.

I do know that some IBMers think that the buffer becomes unavailable
after an AT END, but that was a fault in their system many years ago,
not an issue with Cobol.

If your compiler lets that through, they're being generous.  Generally 
speaking, you read a file, and write a record.

Try placing that display within a "Not at End" clause...

Read myfile
   at End
     Move 1 to EOF
   Not at End
     Display myfile-Rec
End-Read.

Of course, you could do more than one thing within each of these clauses 
- that's the purpose the End-Read serves.  It's not required (unless 
you're doing this inside some sort of conditional statement, such as a 
"When" or an "If" statement, or an inline perform, which I'll show 
below), but it does tend to block the code so it guides the eyes nicely. 
  (That last bit is my opinion...)

Anyway, about the inline perform - it, combined with the "not at end" 
clause, will allow you to do all this in one paragraph.

Pgm.
     Open Input myfile
     Perform Until EOF = 1
         Read myfile
           at End
             Move 1 to EOF
           Not at End
             Display myfile-rec
         End-Read
     End-Perform
     Close myfile
     Stop Run.

Hope that helps - COBOL is a pretty easy language to pick up, and there 
are some really great things it can do.  It's also flexible - there are 
several ways to code what you've tried to do in this simple program.  :)
