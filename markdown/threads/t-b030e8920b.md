It sounds to me like you have a process/program that opened the file
for output then closed the file.  Opening the file for output in a non-
programmer since either creates a file that does not exist or, empties
out an existing file.

The fact that the file size is greater than zero is due to the way
RM/COBOL handles indexed files.  RM/COBOL stores index tables in the
file along with your data.  The 9600 bytes that you see as the file
size is primarily this index table.

My recommendation would be to write down every step you are doing,
(including key strokes) and send them to the developer so they can
trace through to find the problem.

Hi Eugene.

Of course if some program made an OPEN OUTPUT to this file that's the
natural outcome.
But after a lot of years using RMCobol I've suffered, very scarcely,
this same problem without any OPEN OUTPUT to put the blame on. After a
lot of testing I found that the problem might come from the 'SELECT
OPTIONAL' clause that creates a file if it doesn't exist. Under some
circumstances -data corruption on the file header- RunCobol thinks that
the file does not exist and opens it in output mode thus recreating the
file and index structure.

Hope it helps you.
