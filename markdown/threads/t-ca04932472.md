wherein the input file is to be split at intervals and this small
subset is to be written to a workfile. I need to close this workfile
as it would be handled by a C++ tool. Depending upon the return code
(in special register RETURN-CODE) I reopen the workfile in COBOL and
further process it. On completion of this cycle the workfile would be

Hi Kedar,

VIO is fine for this, it will give you a performace improvement over real
file I/O, as you have observed.  If VIO is used extensively at an
installation, the system programmer might need to extend the page space.

You might benefit even more if you keep it in work storage (avoiding
simulated I/O overhead, especially open/close).  If ordinary working-storage
is not enough and you are running with Language Environment, you may use the
callable service Get/Free Heap Storage (CEEGTST/CEEFRST).  This may also
require page space.

Regards,
Gunnar.

```
> Gunnar, are you suggesting that the call to CEEGTST is to be used for
> allocating every record so as to form a Linked list of records?? I am
> a bit confused. Could you kindly elucidate?

Not necessarily a linked list of single records.  You may define an ODO
structure of a reasonable size, and allocate a number of them with CEEGTST
as necessary, chaining them together.  This will minimize overhead and
maximize flexibility.  You may keep the allocated structures for reuse for
the next group of records to further minimize CEEFRST/CEEGTST overhead.

The size of each structure does not have to be fixed, maybe a progressive
growth is good (the ODO definition should accomodate the maximum).  If the
job takes a considerable time, you may lessen the pressure on page space by
freeing an excessive allocation after use (maybe the need for large groups
of records are infrequent)

This technique requires a bit more programming, but manipulating chained
lists and pointer variables is cheep.  I have used this technique in online
programming (CICS) in a case where the need for space was unpredictable and
varied much, it was cheeper than Temporary Storage.

As an alternative, using VIO requires minimum programming, and will
automatically adapt to the needed space as the job progress.  Use large
blocksize on the VIO file for minimum overhead.

regards,
Gunnar.

Just so there is NO confusion, with IBM mainframe COBOL (and *probably* all
other COBOLs) the amount of storage that is allocated for an ODO is *always*
the maximum size (maximum occurrence).  Therefore, if you code

     Occurs 1 to 9999 times
           depending on NumField

and NumField *never* actually gets larger than 10, you will STILL get enough
storage allocated (in your WS) for 9999 occurrences.  (rules are a little
different in Linkage Section and File Section)

Besides going to CEEGTST, it is, however, also possible to do dynamic CALLs
to COBOL subprograms (and cancel them after use) and define/allocate space
in them.  The storage allocated in such programs would ONLY be "used"
between the first CALL and the CANCEL statement.  For example, you could
have

  Call  "Get-1000-Occurrences-1"

and have that subroutine allocate 1000-occurrence for your table - returning
the address of the table - which could be in the main program's linkage
section.  Then *if* you need more than 1000 occurrences, you could go ahead
and

  Call "Get-1000-Occurrences-2"

which would allocate another 1000 occurrences.

Once, you were "done" with this set of records, you would cancel all the
subprograms - and the storage would be freed.
```
