Sorry, I didn't realise you were waiting a response. Your tape-based

scenarion which is as follows:-
Step10. A report generator program prepares a report dataset, which must
be converted to ASCII as the Cd Authoring software does a

Step20. The CD-Authoring software points at the ASCII-converted dataset

Step30. The ISO9660 CD-ROM image created in step20 is written to a

650Mb) it is impracticable to write the data out to tape, then read it
back in. The time delay between the steps would be too much, and would
cause too many tape mounts. (We produce around 100 CD-ROMs each night,
each containing around 20 ASCII-converted report datasets). Also the CD
authoring s/w uses (i beleive) some of the VTOC info about the Step10
report dataset when laying-out the Cd-ROM image.

Interesting comment at end of last posting about different perceptions
about which is mass-storage. Our operations group suggest that DASD
storage is (all costs in UK Pounds) 0.14 per Mb, but around 1.4 pounds
for Tape/Cart storage if the tape/cart is written / re-read. (Cheaper
if written and never read - e.g. disaster backup). These costs
obviously include overheads for physical media storage, human/robot
handling costs etc.

I still say the best solution is an assembler conversion program.
The XLATE macro calls an MVS system SVC to do the conversion. It
is simple, fast, effective, accurate and device independent. Change
to "TO=E" and flip the DCB's to go ascii to ebcdic. Code below:

TITLE 'CONVERT FROM EBCDIC TO ASCII'
EB2AS CSECT
*
USING *,3
STM 14,12,12(13)
LR 3,15
ST 13,SAVEAREA+4
LA 12,SAVEAREA
ST 12,8(0,13)
LR 13,12
OPEN (EFILE,(INPUT),AFILE,(OUTPUT))
LH 4,EFILE+82 GET LENGTH OF INPUT RECORD
READ EQU *
GET EFILE,WORKAREA
XLATE WORKAREA,(4),TO=A
PUT AFILE,WORKAREA
B READ
ENDEFILE CLOSE (EFILE,,AFILE)
GOBACK L 13,SAVEAREA+4
LM 14,12,12(13)
LA 15,0
BR 14
*
SAVEAREA DC 18F'0'
*
AFILE DCB DDNAME=AFILE,MACRF=PM,DSORG=PS
*
EFILE DCB DDNAME=EFILE,MACRF=GM,DSORG=PS,EODAD=ENDEFILE
*
WORKAREA DS CL32760
*
END EB2AS
```

Here's a small assembler program I use to perform this conversion.
It uses the MVS XLATE macro which calls a system SVC to do
the conversion. If you change "TO=A" to "TO=E" and flip the
DCB's you can go ascii to ebcdic. There's no special JCL optcd
parameters or anything. It works great.

TITLE 'CONVERT FROM EBCDIC TO ASCII'
EB2AS CSECT
*
USING *,3
STM 14,12,12(13)
LR 3,15
ST 13,SAVEAREA+4
LA 12,SAVEAREA
ST 12,8(0,13)
LR 13,12
OPEN (EFILE,(INPUT),AFILE,(OUTPUT))
LH 4,EFILE+82 GET LENGTH OF INPUT RECORD
READ EQU *
GET EFILE,WORKAREA
XLATE WORKAREA,(4),TO=A
PUT AFILE,WORKAREA
B READ
ENDEFILE CLOSE (EFILE,,AFILE)
GOBACK L 13,SAVEAREA+4
LM 14,12,12(13)
LA 15,0
BR 14
*
SAVEAREA DC 18F'0'
*
AFILE DCB DDNAME=AFILE,MACRF=PM,DSORG=PS
*
EFILE DCB DDNAME=EFILE,MACRF=GM,DSORG=PS,EODAD=ENDEFILE
*
WORKAREA DS CL32760
*
END EB2AS
```
