```
It is my opinion that DYNAM should be used whenever possible;
1) Your LOADLIB member will be smaller.
2) So that you can upgrade application subroutines and have them take
effect without having to re-compile every application that calls them.

There are cases where DYNAM is not the right thing to do:
1) I believe IBM still says CICS programs are to be NODYNAM.  We follow
this rule, and if an application subroutine is involved, we add it to
the PCT, and have the calling application call it via a working storage
variable to cause it to be called dynamically.
2) We use NODYNAM in all application subroutines so they may be used
from other environments. (e.g. Easytrieve Plus)
```

```
Philosophically as well as practically, I've always prefered the Dynam
approach. Otherwise, whenever a subroutine is changed, all the programs
that call it have to be relinked, which usually means they have to be
recompiled because they weren't saved in unlinked format anywhere because
Cobol shops (unlike PL/1 shops) don't usually think of that.

CICS automatically inserts a compile option card with NoDynam, I'm not sure
what other subsystems and preprocessors do; anyway in cases where NoDynam
is needed, it should be used. I'd rather see it handled as an exception
(even a blank automatic one for e.g. CICS) and Dynam be the default.
```

```
The compiler option DYNAM causes every CALLed program to be loaded at runtime.
The programmer cannot make difference between modules or stubs that have to be
linked to the application and modules that can/must be called dynamically.

CICS- and DB2-stubs MUST be linked to the application, it will abend if you
do not. So, if your programmer have to deal with both batch and CICS or DB2,
you'd better leave the compiler default to NODYNAM. Then they know that they
allways call 'string' for static and call variable for dynamic calls.

There are other aspects you have to consider when dealing with dynamic calls:
From the point of view of QA, linking is much better: WYTIWYHP, what you test
is what you have in production. If you copy a dynamically called module in
production, EVERY application that calls that specific module gets the new
version, even they NEVER did a test with this new version. From the point of
view of the person who does the production upgrade, DYNAM is preferred,
because he never has to find out in what programs a specific module is linked
to. From the point of view of the sysprog staff, either option is equal.
DYNAM causes more directory blocks in loadlibraries, but less space
allocation and the load time is not a big problem if you use VLF/LLA or a
similar utility.
```

```
Daniel's comments are "right on" - but I thought that I would add three more
things into the "mix" for making the DYNAM/NODYNAM decision

1) If you use the NODYNAM compiler option, you can still use the CALL
identifier format (rather than CALL 'literal) format to force a specific
CALL to be made "dynamically".

2) As IBM documents, the NODYNAM compiler option is *not* ANSI/ISO
conforming.  The reason that this is true (which is more important than just
whether your program is conforming) is that when you use the NODYNAM
compiler option, the CANCEL verb ends up as a "no-op".  This applies
particularly to freeing up memory - but also (in some? most? all?) cases to
getting a program in "last-used" versus "initial" state after a CANCEL.
This can be an application design issue.

3) If you still have some pre-31-bit addressing programs (AMODE or RMODE =
24), you most definitely want to call them dynamically - or otherwise, your
entire applications is "put below the line".  Obviously, you also want to
use the DATA(24) compiler option if you are trying to PASS data to a 24-bit
program, but this won't help unless you call it dynamically.
```

```
In most shops I've worked at, the standard was to use COBOL code to
decide that a call was dynamic.  Instead of calling a constant, it calls
a variable.

Unless compilers "outsmart" coders, this guarantees a dynamic load no
matter how the program is compiled.  But I don't know of any way to
guarantee a compile-time load.

Sometimes the migration process uses a slightly different set of compile
options as was used earlier.  And sometimes a particular program needs
to be different from the company standard.
```
