solve your actual assignment, but if your class has already had information
on "REDEFINES" - I think that will solve most of the problems that you
actually mentioned.  If you haven't gotten to them in your course already -
then you can look them up in your reference material - or you can use
SEPARATE fields for your numeric and non-numeric "values" and move
information between them.

The reason that I started out this note the way that I did was that it
(normally?  often?) isn't a good design decision to use the same field for
two "very different" things.  There may be times that this is the best
approach - or there may be reasons why you HAVE to do this, but if possible,
I would recommend against doing it - unless absolutely necessary.  It seems
to me, that you want the same fields to be used to do two different things -
and this OFTEN gets real messy as the application grows in complexity.

Here's the problem. Your definition of 'Reference Number' is X(5).
We assume your file looks like:
00001
00003
01234
77661
etc
with possible trailer records of
-00001 or
b-999     (b=blank)

the problem is that any concievable 'negative reference number' used
as a trailer would, because of the collating sequence used by the
compiler, come before live reference numbers in the file. If the file
were 'sorted,' then the trailer record (as defined) would be at the
beginning of the file.

There are three fixes:
1. Assume this is a 'trick' question and produce a file containing
only a trailer record.
2. Use a 'sentinel' record with a value guaranteed to never be
confused with data. In your case, HIGH-VALUES.
3. Let the operating system detect the end of the file - there are
codes the operating system returns when something happens (they vary
by operating system) (00=success, 10=end of file reached on a read
operation, etc.).

Since you probably don't want to do #1, and you can't control #2 (this
is common - often programmers have to deal with data generated by
fools ("It's mostly sorted")), I'd opt for #3.

JerryP

P.S.
Your problem becomes trivial (from a programming perspective) if you
can use the SORT verb or indexed files.

P.P.S.
This is not a COBOL problem; it's a logic problem. How do you merge
two files, not only by key, but by content. That is, when there are
duplicate keys. It is an enhancement of the problem of merging two
files with no duplicate keys. If you like this, wait 'til you see the
next step.
