```
John,

Create two test files, both with a very small number of records, just
enough to go through each situation once, then try it again., pointing
at these two test files.

You mentioned that you were getting File Status 23s, however you did
not mention which paragraph was getting them (unless they were all
getting the same error, which would send me to look at my source files
and compare the key fields in both to see if any of the numbers ever
match).

I also notice that there is variable error checking.  What do I mean?
Well, this program assumes that every transaction is valid and
correct.  Even though the transaction file may have been edited for
simple data validity before now, this does not mean that keys in the
transaction file are actually valid keys.
For instance:
  If the tran file says ADD, and there is already a record with that
tran key in the master file, there will be an error.
  If the tran file says CHG, and there is no record with that tran key
in the master file, there will be an error.
  If the tran file says DEL, and there is no record with that tran key
in the master file, there will be an error.

In 300-ADD-TRANS-REC, I can't see any code guarding against this.
There is code to detect the error that might occur, but none to stop
it in the first place.

In 301-DEL-TRANS-REC, there is a way to handle the lack of a key in
the master file.

In 0303-CHG-TRAN-REC and the modules it performs, I can't see any code
guarding against this.

Whenever you:
  ADD, you must first READ the master file to determine whether or not
that key already exists in the master file, and if it doesn't, you
would write that trans-rec to a trans-rec-error-file, so that you or
your client can review it and decide what to do with the invalid
record in question.

  Etc. for CHG.


An editing program can determine whether the tran keys are within a
valid range, but not their true validity viz a viz key existence in
the master file.
```
