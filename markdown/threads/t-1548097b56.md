```
OK...this is a weird one...

I am running a COBOL database application on an MS-DOS runtime executing on
Win 95/98 clients from a NT 4.0 server.  This app was migrated from a
Netware 3.12 server.  After migration to NT it was found that the app was
running very slowly.  A standard export routine would take up to 10 minutes
to run.  Quite by accident it was discovered that if you ran the application
windowed instead of full screen, the processing time decreased
dramatically..1 minute 30 seconds.  As I was trying to troubleshoot this, I
was toggling between full screen DOS and windowed by hitting Alt+Enter.
That's when I discovered that as soon as I pressed the Alt key, the export
process would take off like a rocket, even in full screen mode.  I tested
several keys and no matter which key on the keyboard is pressed & held down,
the processing takes off.  If you release the key, it slows back down.
While holding a key down throughout the export process in full screen, it
processed in 30 seconds!   I'm assuming that this is some sort of interrupt
issue, but am at a loss to fully explain it or correct it.  Any help that
anyone could give be would be greatly appreciated.

Thanks,
Brian Butterfield
```

```
Windows NT attemptes "pre-emptive" multi-tasking.  That is, it gives
your application some CPU then moves on.  Pressing SHIFT or any key,
tells Windows your application needs attention - and it gets it.
That's why it's faster.  You can try fiddling with the idle settings
for the dos prompt to speed it up.
```

```
    This particular application probably does a lot of whatever it is
that windows looks for when trying to decide when an application
is wasting time, and when it is working.  In this case, when it is
running slow, windows has decided that (with current properties)
it is wasting time.

    Holding a shift key down probably changes the internal code path,
causing windows to react differently.

    Many DOS editors will cause cpu% to stay at 100% all of the time,
even when nothing is happening.  This is an aspect of the same thing.
```

```
It is not the Keystroke but rather the following:

When it runs full screen (but you don't have that screen visible)
it is running as a 'background task' so that WIndows can
provide quick response to the 'active' window.

One you 'window' the app, it gets a bigger time slice.
Once you switch to it, or if you only have it and Desktop running,
once you press 'ALT', the system knows where you are going,
so then it boosts the priority of the App.
```
