```
We are having a debate over the proper use of prefixes when naming certain
Working-Storage items.	In the past, we have used short, commonly used
prefixes such as:

PRT-  (or PRT2- in the case of two print files) for print items
SWK- (or SWKx-) for sort-work items
DWK- (or DWKx-) for disk work items

There are some who feel that we should use (and mandate as a standard) the
full 7 or 8 character file name specified in the select statement.  I have
always assumed that the sort prefix was the most common, but maybe I'm wrong.

How do you assign prefixes in your shop?

Thanks.
```

```
I don't try to make all my prefixes the same size, I don't see any
benefit in doing so, I think clarity is more important.  I use 'WS-'
for all 77 level items, and only 77 level entries (or 01 level items
with a picture, when used like a 77).  For all other data names, I
take the name of the 01 level item and make an acronym from the name
and use it on all data items subordinate to that 01.  For file names
I usually use something like 'PRINT-FILE' or 'WORK-FILE' and name
the records 'PRINT-RECORD' and 'WORK-RECORD'.  The data items in the
records would have prefixes of 'PR-' and 'WR-'.  My prefixes are
usually two to five characters, whatever it takes for clarity and
uniqueness.  Larger programs often require longer prefixes to make
them both unique and clear.

I go to great lengths to make the remainder of the data name after
the prefix to be consistent throughout the whole system.  When I am
designing a new system, I create both a data name dictionary and an
abbreviation dictionary, so new data names can easily be created
using consistent abbreviations.  I hate to see 'number' abbreviated
'NO', 'NBR', 'NUM' and 'NR' all in the same system.  Chaos!  one of
the marks of excellence is consistency, consistency, consistency.
```

```
in the systems I inherited. I decided to use the consistent form, "id".

works at a company, which is identified by its "company-id"; is
assigned to a department, which is identified by its "department-id";
and works on jobs, which are identified by their "job-id"; etc.

contained letters, which is inconsistent with using number as part
of the variable name.
```

```
It does appear that most shops prefer unique datanames everywhere, but I
prefer Paul's (paulr@bix.com) approach: Define the data fields in a
given record type one time and use those names consistently whenever a
record reoccurs in the file section, working storage, etc. In other
words, take full advantage of COBOL's data qualification feature.

For each record type, I set up a copybook containing the data elements
in that record (but not the 01 level). Then I invoke it following a
hard-coded 01 level wherever that record occurs. 

All of this makes maintenance much easier, particularly in a large
system. It also makes our COBOL records a step closer to the records of
Pascal, the structs of C, and, yes, the classes of C++.
```
