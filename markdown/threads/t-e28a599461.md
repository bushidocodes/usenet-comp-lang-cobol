Carlos,

I think what you REALLY want (it isn't what you asked about) is to be able
to move a numeric value to a single byte and have it automatically
translated to the equivalent ASCII value.

You can do that with the following (although it is tedious...):


 01  char-num  pic 999 binary.
 01  char-x   redefines char-num.
     12 filler      pic x.
     12 target    pic x.

 PROCEDURE       DIVISION.
     move 059 to char-num

If you now check "target" it contains ";" which I believe is the desired
result.

I would've thought the same effect could be obtained by  using:

     move function CHAR(59) to target

...but it gives ":" in target instead of ";".

I have no idea why this is and I'm too busy to investigate it at the moment.
It may be a bug or may be correct behaviour.

Either way, you can get the effect you want with the definitions above.

Pete.

It's correct.  Function Char(1) will give you the character at position 
0 of the character set.  So, if 59 is ";", Function Char(60) should give 
it to you.

It's weird, I agree, but I was told that was intended behavior.  There's 
a 5-character control sequence that one can use in the Unisys 2200 
environment to signify the start of entry (SOE).  However, the SOE is in 
the character set (character 30).  So, I was using Function Char(30) and 
wondering why it wouldn't work.  I contacted someone at Unisys, and they 
said that you need to add 1 to that number, and when I changed it to 31, 
it worked like a charm...  :)

Fujitsu makes a COMP-* fields more than one byte depending on compiler
options.  The defult is BINARY(WORD) giving 2 bytes for a PIC 9 COMP.
You could set BINARY(BYTE) or redefine the above to:

    01  Linha-All.
        03  L-x         PIC 9(4) COMP-5.
    01  Linha-REDF REDEFINES Linha-All.
        03  Linha-X     PIC X.
        03  FILLER      PIC X.

The characters you actually get are dependent on the code page used. 
Perhaps you were running in DOS with a particular code set to get the
1/2 character.

For specific characters it may be better to put it into the source
code literal using say: ALT + 1 + 7 + 1 (where numbers are on numeric
keypad while ALT held down) then you will see what you get (if text
editor is in Windows too).

If you are using PIC 99 COMP-? then Fujitsu will not allow you to put
in a value greater than 99 because it will truncate the value - unless
TRUNC option is set correctly, and you also add rather than move.

Best is to use PIC 9(4) COMP-5 (little-endian) and redefine as PIC X
and FILLER PIC X.

But you cannot get 'nothing'.  You must get 'something' in the range
x'00' to x'FF' in the character position.  Note that with 171 on a
little-endian COMP-? the value will be in the first byte and in a
big-endian COMP-? the value will be in the 2nd byte.  COMP-5 and
COMP-X should be little-endian.  COMP is probably big-endian.
