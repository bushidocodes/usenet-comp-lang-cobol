I don't really know why you think there should be any stack entries in the

if it doesn't work (due to stack entries or whatever), it is a bug

it is not a GOTO, this would be a perfect example of someone who understood
the form, but not the substance, of the objections to the goto :-)

There are many different ways to implement PERFORM.  Some use stack 
entries and some do not (they use containers, one for each exit - 
when an exit is activated the return is stuck into the container).  I 
have seen people count on the behavior of this type of implementation 
by judicious use of GO TOs.  When they port the program it fails.  I 
think most modern implementations use some sort of stacking 
mechanism, and the GO TO out of a PERFORM can wreak havoc if the exit 
is never taken.

>Suppose: 
>            set cond2 to true
…[6 more quoted lines elided]…
>            perform a-section

This is an illegal statement.  NEXT SENTENCE is not allowed within an 
IF ... END-IF (there must be an END-IF when it is within an in-line 
PERFORM).
>
>I would not be very surprised to find that the return from a-section
picked up
>the address for end-perform. I suppose my question is really 'does ANSI85
>define a semantics for in-line performs that precludes this possibility
(it's
>obviously a bug) or is it down to the compiler writers?'
>
The standard more or less states that any GO TO out of this has no 
effect on any PERFORM.  All implementations that I am familiar with 
(and I've written a few myself) do in-line PERFORMs in a completely 
different way than the others.  There is no need at all to do any 
stacking and whatnot.

>> FOr my taste this is MUCH WORSE than a goto, because it is far less
clear
>> where it is going, and code should not be this sensitive to the
presence
>> of periods.

That is why it is not legal.  I will admit that through a flaw in the 
standard there is a way to get a NEXT SENTENCE in there, but I won't 
tell anyone how do to it.

>Agree entirely. Sometimes I have used NEXT SENTENCE rather than GO TO
<label>
>because I fear people will come along after me and see the presence of
the
>label as authorising GO TOs to it, but at least the NEXT SENTENCE (in
this
>context) can't be used for anything but its original purpose and so won't
>contribute to the code getting any worse. In fact I've seen this
situation.

We are making NEXT SENTENCE an archaic element in the next standard.
All this really means is that it is bad form to use it.

That's not good enough.

   PERFORM UNTIL SOME-CONDITION
        move 'a' to foo
        SET SOME-CONDITION TO TRUE
        move 'b' to bar
   END-PERFORM


What if I need to get out of the inline perform before 'b' is moved to bar? 
That's what the "leave-perform" verb would do - like so:

   PERFORM UNTIL SOME-CONDITION
        move 'a' to foo
        leave-perform
        move 'b' to bar
   END-PERFORM
