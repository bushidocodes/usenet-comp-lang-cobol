Boy - are *you* going to have fun! <grin>
There are several considerations, the biggest one of which you have correctly identified as
being the ASCII/EDCDIC conversion. This may make some of your comparisons fail, or
more accurately, make a wrong selection.

More subtle issues you will encounter are also, since you are coming from a Vax/VMS platform,
not really going to present much of a problem. For example, the AS/400 does all it's arithmetic
in Packed Decimal format, just like a mainframe. It is *much* more efficient to make your
variables "packed" formats rather than binary. I do not *think* the VMS system supports packed
decimal, or if it does, I do not think many VMS programmers use it. I could be wrong there,
but I don't think you will have the issues of converting packed decimal information from
ASCII to EBCDIC. That *is* a real pain - I assure you of that!

Screens are going to be fun too, if you are using VT100/ANSI/etc. screens, you will need to translate them to AS/400 'green screen'
mode. The screen formats are stored separately on the
AS/400, and data is passed back and forth in some extremely sophisticated ways - but those same ways can be difficult to get an
understanding of, depending upon your experience base. If you, for example, understand BMS on a mainframe, it makes a lot of sense.
If you are used to say, UNIX termcap style entries, it is insanely difficult to make the conceptual jump.

File storage on an AS/400 is also different. First off, disks are called DASD - for historical reasons. Again, that is similar to
the mainframe, but it is not so clear to a VMS or UNIX person.
"Libraries" are roughly equivalent to directories, they can contain programs, files, and some other exotic objects. But 'files'
alone are pretty exotic - there are two basic kinds of files and a smattering of special purpose files out there.

Physical files are equivalent to a PDS (Partitioned Data Set) on a mainframe, and as far as I know, do not have an analog under VMS
or UNIX. (Nor under Windows to be honest.) The source code for your programs would be stored as 'members' of a file - essentially
you might think of them like 'blob' records in a database table, but they are not really that at all. :)  If you think of a physical
file as being a subdirectory of a library, but one that can only contain a *single format of file* - you come close to the idea. For
example, COBOL source programs are stored as 92 byte records in a physical file. (By convention, the file is called QCBLSRC or
QCBLLESRC, depending upon what compiler you use. This is convention only, not a standard or requirement.)
In the same library, you would probably have another physical file called QDDSRC, in which you would store the source code for
database and screen 'formats.'

To create a database file for example, the very basic structure is to write a DDS source file with a 'type' of PF (physical file),
and in the source you define the record format and keys. To actually create the database file, you use a command to 'compile' the
DDS source - the result being the real physical file in which your data will be stored.

If that sounds more like ISAM than Database to you, in a way it is. But the physical files are really database tables. :)  In fact,
DB/2 on the mainframe does something similar - at the core of DB/2, you will find a set of VSAM files! <grin>

All that is to say, it is a lot of fun to work on the AS/400, and I think you will grow to love the platform. In many ways, it is a
poor man's mainframe. I say that from the point of view of having a old P/390, RS/6000, and AS/400e here at the house, and an up to
date modern fully powered z800 Mainframe complete with Shark at work. Of all the machines, I admit to loving the mainframe world the
most, but the AS/400 is purely wonderful too.

Indeed, should you need consulting help with the AS/400, let me know. I'd be glad to help out a bit in my spare time - it really
sounds like you have a very fun project ahead of you!

-Paul
