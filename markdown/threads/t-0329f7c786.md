How does the 16 bit MF 3.4 Compiler allocate memory segments for a PC
executable?
I work on a system that currently exists as a large executable MS Windows
application.  This is generated by statically linking 40 or so .OBJ's
generated by the 16 Bit MF 3.4 Compiler in LARGE memory model.   Recently I
tried linking a new .OBJ into this .EXE but the LINK failed due to the
number of segments in the .EXE now exceeding 255.
So I backtracked recompiled/linked sucessfully and examined the executable:

.EXE size (bytes)  237110
CODE Segments 39
DATA Segments 213

Total Segments  252

The program makes extensive use of Working Storage, Linkage Section, Local
Storage and EXTERNAL data items.
Pertinent Compiler Directives used:
target(286)
64kSECT
NOSEG

Given that a segment max. size is 64k I'm interested to know: How/Why are so
many segments are used?

Though you didn't ask, you can increase the maximum number of
segments by using the '/SEG:n' linker option; where 'n' may
be up to 16384.

Studying the MAP output of the linker may help you to understand
some of the "why" for the extra data segments. Use the '/MAP'
option to create the file.

Think of each data segment as a "named data space", that is,
each set of data definitions has its own separatey named data
space. The WORKING-STORAGE for each program is a data
space, each EXTERNAL is a data space, etc. Except that
LINKAGE SECTIONs and LOCAL-STORAGE SECTIONs do
not have their own separate data space. Furthermore, many of
the data segments are used by the COBOL run time.

IMO, the only good reason to change the design of an application
is if the design is defective -- a limit in the OS is not a good reason.

If the PACKCODE and PACKDATA linker options are not
sufficient. You could possibly use the RTE. Basically, you compile
all the programs to .GNT. Place all the .GNT files into library (.LBR)
files and either include the libraries in the configuration (.CFG) file
or CALL the libraries from your main program. If you do not, or can
not, use XM, you may need to add some CANCEL statements.

This is the technique I use. Although I do not run the system as
a Windows application, for demonstration purposes, I did put it
together as a non-GUI Windows application. Even with 60
programs and EXTERNAL data, there were no problem with
segment limits.

If you must use static linking with object (.OBJ) files. It is possible
to reduce the number of object files (and, therefore, segments)
without changing the design.

If the following suggestions make you uncomfortable, do not
use them. They all involve single-level nested programs.

1. If prog-D calls prog-E and prog-F and neither prog-E nor
prog-F have any COPY statements and are not CALLed by
other programs; you could copy prog-E and prog-F into
prog-D as nested programs. This will save at least two
segments per contained program. Source code changes
to prog-D would consist of two COPY statements and three
END PROGRAM statements. Compile prog-D with the
ANS85 and NESTCALL directives.

2. If prog-D calls prog-E and prog-E calls prog-F and neither
prog-E nor prog-F have any COPY statements and are not
CALLed by other programs; you could copy both into prog-D,
as above, except the prog-F will need the IS COMMON clause
after the program name.

3. If prog-D, prog-E, and prog-F share access to an EXTERNAL
data item, that EXTERNAL data item could be placed into a
program's WORKING-STORAGE as a GLOBAL data item. The
PROCEDURE DIVISION would consist of ENTRY statements
with the same names and parameters as prog-D, etc. Each
ENTRY would contain a CALL to the actual program which
would be a nested program as above. After each CALL
would be an EXIT PROGRAM. For example,

    ENTRY "prog-D" USING D.
    CALL "prog-D" USING D.
    EXIT PROGRAM.

Finally, COPY the programs as before. The same restriction
against nested COPY remains. This works because the
precedence for resolving program names is to look first at the
names for contained programs. Since "prog-D" is a contained
program, it will be called instead of creating a recursive call.
The savings is at least two segments per program, after the
first, plus the segment for the EXTERNAL data item.

Demo code for item 3; tested with MF WB 3.2.50:
-------------------
      $set ans85 nestcall
       program-id. prog-A.
       data division.
       working-storage section.
       01 A pic x value "A".
       01 B pic x value "B".
       procedure division.
           call "prog-DEF"   *> required for dynamic linking
           call "prog-D" using A
           call "prog-E" using B
           call "prog-F"
           stop run.
       end program prog-A.

       program-id. prog-DEF.
       data division.
       working-storage section.
       01 was-EXTERNAL GLOBAL.
         05 we-A pic x.
         05 we-B pic x.
       linkage section.
       01 D pic x.
       01 E pic x.
       procedure division.
           exit program.

       entry-for-D.
           entry "prog-D" using D
           call "prog-D" using D
           exit program.

       entry-for-E.
           entry "prog-E" using E
           call "prog-E" using E
           exit program.

       entry-for-F.
           entry "prog-F"
           call "prog-F"
           exit program.

       program-id. prog-D. *> simulated copy
       data division.
       linkage section.
       01 D pic x.
       procedure division using D.
           move D to we-A
           exit program.
       end program prog-D.

       program-id. prog-E. *> simulated copy
       data division.
       linkage section.
       01 E pic x.
       procedure division using E.
           move E to we-B
           exit program.
       end program prog-E.

       program-id. prog-F. *> simulated copy
       data division.
       procedure division.
           display we-A
           display we-B
           exit program.
       end program prog-F.

       end program prog-DEF.
-------------------
In summary, use the RTE if it is available. Use the others if you
are comfortable with them. None of these suggestions requires
any significant change in the design of the application.

I hope these suggestions are not too weird!
