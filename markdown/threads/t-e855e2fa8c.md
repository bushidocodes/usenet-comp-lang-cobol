```
Initialize does not nor has it ever referenced a VALUE clause. A VALUE
clause does not make a field in COBOL a CONSTANT never changing value.
ITIALIZE only looks at the type of variable, NUMERIC or ALPHANUMERIC, to
determine which values to move to the referenced fields. NUMERIC gets zeros,
and sometimes LOW-VALUES depending on the USAGE clause. ALPHANUMERIC fields
get SPACES or hex '40'. What a field contains is irrelevant to the
initialize clause. If you look into the assembler/machine code generated by
an INITIALIZE clause it would be the same as several separate moves moving
the appropriate ZEROS or SPACES to the initialized fields.

INITIALIZE has been frowned-upon by most programmers I have worked with. It
is less efficient than doing the moves manually. In one case I have seen a
record containing several thousand bytes initialized by saving off the
initial state filled in by value clauses. Then the initial state is simply
moved off to a large filler working storage field. Then moved back when the
record needs to be initialized. Seems crazy, but the CPU usage for the later
operation as opposed to an initialize statement were extremely significant.
Considering such a  thing may be performed several million times for a large
database, it would be impractical to use an initialize statement.
```

```
I have been off c.l.c most of the last week with flu - and thought I would
"summarize" a few things about INITIALIZE (from MY perspective):

1) Almost (but probably not all) INITIALIZE statements *can* be coded as
"MOVE" statements - with better run-time efficiency.  HOWEVER, depending on
the application design and record lay-outs, INITIALIZE *may* be a more
efficient "self-documenting" and easier to maintain structure.

2) INITIALIZE *defaults* to moving spaces to named alphanumeric items and
zeroes to named numeric items.  It *never* (currently - see below) changes
filler fields or uses the VALUE clause.

3) The current INITIALIZE statement fully supports using OTHER than spaces
or zeros for initialization - but this is not the default (and requires the
use of a REPLACING phrase).  This allows using separate values for each type
(category) of elementary item to be initialized.

4) In the next Standard (and I think some vendors have started adding the
syntax as a current extension) there WILL be ways to:
   -  INITIALIZE filler fields (when requested by a key word)
   - INITIALIZE using VALUE clause (again with an explicit keyword)

Bottom-Line:
  INITIALIZE is a powerful verb (and getting more so in the future).  It
probably is NOT your best choice for repeatedly done object-code sensitive
application logic - but does provide a significant advantage for maintenance
and self-documenting code.
```

```
Not necessarily.  I've seen linkage overlap either working-storage or
procedure -- or someone elses pgm -- due to programmer error.  All you have
to do is define your linkage section larger than the area being passed to it
by the calling pgm.

PGM-A.
WORKING-STORAGE.
01 PASSED-TO-PGM-B  PIC X(50).
PROCEDURE.
CALL PGM-B USING AREA-TO-BE-PASSED.

PGM-B.
LINKAGE.
01 PASSED-FROM-PGM-A PIC X(1024).

or the same principle in CICS.

WORKING-STORAGE.
01 WS-COMMAREA  PIC X(50).
01 WS-COMMLEN   PIC S9(8) COMP.
LINKAGE SECTION.
01 DFHCOMMAREA  PIC X(1024).
PROCEDURE.
MOVE LENGTH OF WS-COMMAREA TO WS-COMMLEN
EXEC CICS RETURN TRANSID(PGMA)
            COMMAREA(WS-COMMAREA)
            LENGTH (WS-COMMLEN)
END-EXEC.


In either example, anything beyond the first 50 bytes of linkage is 'no-mans
land.'  The complier has no way of knowing the calling program is passing a
pointer to a storage block that is smaller than what was defined by the
receiving program. It could be just about anything, and if you move anything
into it (past the first 50 bytes, in this example), you are almost guarenteed
to crash something -- and it could be just about anything.  I discovered this
years ago when I had to track down why a CICS program I was working on was
crashing in a random manner.
```
