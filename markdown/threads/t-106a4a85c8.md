NUMVAL, NUMVAL-C,  and TEST-NUMVAL are good, but they are limited.

Specifically, they don't provide a choice of numeric formats, assuming that 
the target is the only format you want, they don't validate but expect a 
valid numeric string,  and they don't tell you how many digits each side of 
the point.

I believe TEST-NUMVAL is part of the 2002 standard and, as such, has never 
been implemented as far as I know....

I use a component called String2Num for all of these reasons.

The component can not only validate, but it can notify the User as well, if 
required, and give examples of correct usage. It can also run on a web page 
or the desktop. (I don't think function NUMVAL can be invoked from 
JavaScript, but you could probably use it if you were writing .NetCOBOL for 
.NET and using it to script a web page).

You can use UNSTRING to split the string into parts delimited by '.'
or space. Each of the resulting parts can be INSPECTed to get their
length and then each can be tested using reference notation with an IF
NUMERIC.

     given a non-space alleged numeric in csv_string
     code may be approximately:

     move all zero   to resulting-parts
     move 1          to resulting-negative
     move 1          to csv-pointer
     inspect csv-string
         tallying csv-pointer for leading spaces

     move zero      to part-subscript
     perform
        with test after
        until part-length = zero
          and part-end = space

        unstring csv-string
            delimited by "." or "-" or "+" or space
            into part-string
                 delimiter in part-end
                 count in part-length
            with pointer csv-pointer

       if ( part-length not = zero )
          if ( part-string(1:part-length) not numeric )
              error - not numeric
           else
              move function NUMVAL(part-string(1:part-length))
                 to resulting-number(part-subscript)
          end-if
       end-if

       if ( part-end = "." )
           add 1   to part-subscript
           if ( part-subscript > 2 )
               error - too many .
           end-if
       end-if
       if ( part-end = "-" )
           move -1  to resulting-negative
       end-if
       ....


You should add checks for part-length exceeding the limits that you
require.

Robert,

a few observations...

1. The code above (in my opinion...) is not a proper solution for a true 
numeric validation of screen input. It doesn't handle a number of 
conditions. (multiple or conflicting signs, embedded blanks,  CR/DR leading 
or trailing, with or without a blank, presence of a currency symbol) and it 
is not suitable for European formats.

2. It doesn't provide alternate numeric formats for output (this is not 
"wrong". but it is very useful to have)

3. It doesn't tell us how many digits are on each side of the point. (A 
requirement specified by the OP).

4. I have accepted your challenge and extended String2Num to handle 
scientific notation. It does not require recursion to move a decimal point 
and pad with zeros right or left. (However, I do expect the string tohave 
been normalized and to be in base 10; if it is coming from EXCEL, this will 
be the case, just as in your example above. I also expect the floating point 
format to contain a point.(It can be a comma or a period), and the string 
MAY be explicitly signed with a leading or trailing sign and/or CR/DR. There 
are a number of different floating point formats, as I'm sure you are 
aware.) I would normally have written this as a separate method, but I broke 
my own "rule" and added a section to the base code, because I expected it to 
be trivial... : - ) In the event, it was not... Nevertheless, it was quite 
relaxing and enjoyable to write COBOL again after a long absence. It still 
took about 3 times as much code as it would have in C#, mainly because I

 01  varchar typedef.
     05  string-length  binary pic s9(04).
     05  string.
           10  string-byte occurs 1 to 255 depending on string-length.

 function-id. TrimRight.
 working-storage section.
 01  ctr pic s9(4) comp.
 linkage section.
 01  cobol-string    pic x(255).
 01  varchar-string varchar.
 procedure division using cobol-string returning varchar-string.
    move zero to ctr
    inspect function REVERSE (cobol-string)
        tallying ctr
        for leading spaces
    compute ctr = 256 - ctr
    move ctr to string-length
    move cobol-string (1: ctr) to string
    goback
    .

 function-id. Trim.
 linkage section.
 01  cobol-string    pic x(255).
 01  varchar-string varchar.
 procedure division using cobol-string returning varchar-string.
   perform until cobol-string (1:1) NOT = space
        move cobol string (2: 254) to cobol-string
   end-perform
   move function Trim (cobol-string) to varchar-string
   goback
   .

Obviously, you could use indexing instead of left shifting in the second

ANY LENGTH is ideal for this, because it passes the parameter's length as well as its
address. We wouldn't have to pick an arbitrary input length. A single function could be
called with x(30), x(50) and x(255). 

ANY LENGTH is not a varchar. It can only be used in the linkage section of a called
program or method, indicating the length is determined at execution time. The calling
program must have a prototype so the compiler knows to pass the length. Example:

01  input-30  pic x(30).
01  input-255 pic x(255).

call 'foo' using input-30
call 'foo' using input-255

program-id. foo.
linkage section.
01  input-string pic x any length.
procedure division using input-string.
    display 'Length is ' length of input-string.

The free Fujitsu compiler does not support ANY LENGTH nor prototypes.  One can do the same
by passing the length as a separate parameter, as you did above.
