void function_name (void) {

    more code

parameter and result fields for a performed section.

every GO TO statement with MOVE <next state> to
STATE-NUMBER and have the entire process driven by
a state machine driver, in the form:

state-machine-driver.
    move 1 to state-number
    perform until state-number = 0
        evaluate state-number
          when 1
              perform state-1
          ...
          when X
            perform state-X
          when other
            perform post-state-machine-failure
        end-evaluate
    end-perform
    .

This is dangerous because it provides the illusion that
the paragraphs implementing the various states of the
state machine may be modified, independently of the
documentation. They must not!

Some programming "genius" might decide that a
new state may be performed from within an existing
state, thus avoiding changes to the state machine driver.
The "genius" might also decide to perform states directly.
The result of several such changes is "ST spaghetti";
that is, the simple control structure of the state machine
driver is no longer correct because some states are not
reachable from the driver and some states are entered
by two methods.

state transition diagram because "anyone can understand
the change, I used structured programming". As a result,
the state machine can no longer be understood from the
documentation.

Thus a state machine, that may be well-understood by
its documentation and easily coded and maintained from
that documentation, using GO TO; becomes a confusing
and difficult mess without the use of a single GO TO!
