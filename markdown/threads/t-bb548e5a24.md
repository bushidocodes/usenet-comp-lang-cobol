```
I assume that you know that IBM *strongly* recommends against using
TRUNC(BIN) as the default.  This is true for a NUMBER of reasons - but
certainly is true for ANY performance sensitive applications.

Have you looked at TRUNC(OPT) which gives you "similar" results for many
(most?) situations - but with SIGNIFICANTLY better performance?  With IBM
COBOL for OS/390 & VM V2R2, it is even possible to use TRUNC(OPT) as your
default and to get TRUNC(BIN) behavior for SPECIFIC fields that need it.
This is done by defining them as USAGE COMP-5  (see the "What's new"
sections in the current LRM and programming guides).

If (for some reason) you are REQUIRED to go to TRUNC(BIN) as the shop
default, then I would strongly suggest that you look at some of the examples
in the Programming Guide on when you will get different results from the
same statements.
```

```
IBM has been pretty conflicting on whether they do or do NOT
recommend/require TRUNC(BIN) for CICS applications.  At one time (and I
think is some places still) they recommend doing so.  In other places, the
recommend against it.

Clearly if you are using IBM COBOL of OS/390 & VM V2R2 and do copy in DFH
copybooks (or EXEC SQL INCLUDEs) and *want* to guarantee TRUNC(BIN) behavior
for such field, then you might consider adding

   replacing ==COMP-4== by ==COMP-5==
                 ==COMP==    by  == COMP-5==
          .

The point of the COMP-5 "enhancement" in V2R2 is to ALLOW (encourage?) use
of TRUNC(OPT) or (STD) as the application default and limit fields that
actually require "true binary" to COMP-5.
```
