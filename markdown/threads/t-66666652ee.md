```
It's not working very well. I'm usign Microfocus Object Cobol 4.0.
I can build DLLs with this, and have been able to access them
```

```
I suspect it is more likely a conflict with the Micro Focus runtime system
architecture.  Perhaps someone from Micro Focus will address this.

We have encountered a variety of problems with the MF runtime, depending
upon how you implemented your application.  We tested at least six different
combinations with our products, and found three that worked great and three
that failed in (usually subtle) ways.  Once you get it figured out, however,
it works fine.

But, again, I don't know about implementing a Windows DLL in Micro Focus
COBOL that could be called by someone else...

We do most of our development with CA-Realia.

We implemented our ScreenIO COBOL screen manager as a standalone Windows DLL
which is readily callable from any language, including at last count five
different COBOLs.  It would work just as well from C or VB, though I doubt
that anyone would want to do that.

We did it using CA-Realia II Workbench Version 3.0 (a 32-bit compiler).  It
was relatively easy, and it certainly made our life easier 'cause now we
only use one compiler for all development.
```

```
Andrew,

The same principle applies generally to COBOL DLLs as to the specific
example Norcom described below. A COBOL DLL is just an executable module so
it should be callable from any language that supports the same calling
mechanism. Micro Focus Object COBOL 4.0 actually uses Microsoft Visual C++
itself so there should certainly be no problems calling a COBOL DLL from
VC++ (in fact I know of many people who do exactly that). The default
calling convention for COBOL and the default for VC++ are in fact the same
so nothing special is needed in terms of entry points or function
declarations. That being the case you will probably need to provide a
little more information before I can help you get to the cause of the
problem.

1) Is the program created with VC++ actually a C++ program or a C program ?
If it is C++ you will need to define the COBOL as extern "C".

2) Do you reference the COBOL DLL by linking the C program with an import
library, or is it loaded via the LoadLibrary API ? If it is the latter,
what error is returned from GetLastError after the LoadLibrary fails ?

3) Have you checked that the COBOL run-time system (mfrts32.dll) can be
found ?
```
