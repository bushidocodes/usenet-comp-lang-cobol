In many COBOLs there is a FILE-STATUS entry that can be defined under the
SELECT clause.  The FILE-STATUSs I've seen have always been 2-byte display
fields.  If there are problems with any I/O done to the file, a non-zero
return code is placed in the FILE-STATUS field.  I don't specifically know
about Acucobol, but I would check that possibility first.

Hi Stephan,

OK, default behaviour for an Acucobol-GT program is to stop when it hits 
a file error for which no declarative is declared. I'm not going to go 
into declaratives here (it would take a while to explain, and it's 
Sunday night..... You can,  however,  change this default behaviour by 
using the configuration file. Add a line to the file (or create a new 
file) which reads:

ERRORS-OK       1

You can now use this file by using the '-c' runtime flag:

wrun32 -c myconfig myprogram

would run "myprogram" using the configuration file "myconfig". Now your 
program will carry on when it sees a file error, and you can check your 
file status. Note there are other ways to select a config file, this is 
the simplest.

I see that James Gavan has pointed out that you can't open a file
I-O if it doesn't exist. Again you can override this behaviour by adding 
the line:

IO-CREATES 1

to your config file.

Hope this helps!

Cheers

Nigel

Failing that all you can use, (and it isn't totally satisfactory), is the

Open a file and if the file-status <> "00" - then you know you have a problem.
Either it doesn't exist or it could be a crashed index for an ISAM. Does
AcuCobol have any extensions to the standard file-status code list that will

Just looked at your reply to Paul :-

Here is how I try to open the (not yet existing) file:

       oeffnenIN.
           Open I-O dateiName
           If NOT dateiFehlercode
              Perform fehler
           End-If

It wont work if it doesn't exist (you can't use OPEN I-O on a non-existent
file) - try this substituting what's necessary in your perform fehler :-

       oeffnenIN.
           Open I-O dateiName
           If NOT dateiFehlercode
                open OUTPUT DateiName

    *> Now you can *only* WRITE (new) records

    *> If you want to be able to WRITE new records and also be able to change
    *> records, (REWRITES), which you have already processed in previous
    *> WRITES, in this run,  then do the following :-

        open OUTPUT DateiName
        close DateiName
        open I-O DateiName

now you can READ PrimeKey, REWRITE and WRITE.

This shouldn't be necessary for a data file.  Open INPUT and check
status to see if it exists and is usable.  If no exist then open output
else open i-o.  However, using select OPTIONAL _should_ avoid the issue
entirely.
 
> Failing that all you can use, (and it isn't totally satisfactory), is the
> COBOL file-status.
> Open a file and if the file-status <> "00" - then you know you have a problem.

You should only check the first byte of the file-status for a successful
result.  The 2nd byte may not be zero when the operation is successful. 
For example: opening i-o a non-existant optional file may return '05' to
indicate that file was successfully created.

> It wont work if it doesn't exist (you can't use OPEN I-O on a non-existent
> file) - 

Yes you can (on many Cobols at least) that is what OPTIONAL is for.
