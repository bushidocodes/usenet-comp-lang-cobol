I think your English is fine--you stated your question well, and
made a specific follow-up.  Also you exhibited etiquette that's
often foregone in newsgroup forums.

As to your question, I can provide some speculation for a
restricted viewpoint--Cobol for windows development.  I'm
involved in a pc-based (formerly dos) application written in
Cobol.

Given that there's no expectation nor desire to add support for
platforms other than windows, there's only one reason we've
continued with Cobol--existing code base.  From what I've seen
of the community at large, I suspect this is a powerful, if not
solitary, motivation in many other places.

Although the history of computer languages has seen a lot of
variations, it's not as whimsical as it may seem.  Fortran and
Cobol where on the scene first, and are still in use.  C showed
up and gained a lot of momentum when it was expanded in the 70's
to embrace object-oriented programming.

More languages and variations of existing languages appeared,
but the next significant advance was driven by changes in
operating systems, rather than new ideas in languages.
Windows-style OSs and the event-driven approach they took have
spawned a number of integrated development environments.

IDEs prior to that were a convenience, but with the addition of
such heavy resource requirements, began to qualify as languages
in their own right.  Clipper, VB and now application
environments are become a viable choice.

It's a bit unfair to compare Cobol to C for many reasons, not
the least of which is that C, C++, Java, and C# are considered
to be part of the C family.  By doing so, it's been able to grow
while leaving standardized versions behind to canonize existing
code.  Cobol does not seem to be so aggressively included
towards expansion.

Even though Cobol showed up quite a bit before C, the
specification (and consequently use of) and object-oriented form
has been slow.  If this is a good or bad thing is a point of
dispute, but it nonetheless marks Cobol as sedentary.

I believe this is because two reasonable goals are in direct
conflict with each other:  (1) Refinement of the language, and
(2) support for an existing code base.  There are ways to manage
this, but it won't fly if the community sees no value in the
aforementioned refinement.  This is apparently the case.
