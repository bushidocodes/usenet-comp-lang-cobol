1. You should make sure the size of DAYS-DELINQUENT is large enough to
   hold well over 2000 * 365 = 730,000, say 6 digits, which should work
   until sometime in the 29th century.

2. The two characters in DD, MM and YY imply two digit fields.  So data
   names TODAY-YY and LAST-YY should be changed to TODAY-YYYY and
   LAST-YYYY or TODAY-CCYY and LAST-CCYY for clarity.  You can change
   all the fields to -YEAR, -MONTH, -DAY, but that often conflicts with
   a similar field set up for the same date in month, day, year format.

Of course, resizing the variables in the files is the "best" and most solid
way of fixing the problem, even though it doesn't address all the issues.  It
is also deceptively difficult to implement.  You have to convert the file and
EVERY program that touches it all at once.  This can turn out to be a LOT of
programs, JCL, CICS tables, files and databases that all have to be converted
at once.  This is why logic (code) is continually evolving, but data
structures live forever.  A lot of shops will go to extraordinary lengths to
avoid changing a data structure.

Compound these issues with the fact that any "all at once" conversion
such as this cannot be undone should something go wrong, unless all
transaction data, changes, etc., since the conversion are also
available.  This does not even address A/B testing and such.

The other thing that cannot be overlooked is history/archival data.
Should all such data be converted?  Should it be retained in both
forms?  Should programs be maintained in versions that process old
data and also in versions that process new data?  (I hope not!)  If
so, how do we deal with data that crosses the Y2K barrier (the
original problem all over again)?  How is data subjected to such a
conversion affected by record retention policies, legal requirements,
and the like?

None of this even begins to deal with files that are shared between
system, sites, platforms, customers, etc.  It should be easy to see
how such an apparently simple concept can rapidly spiral toward
complexity.

As a side note, relative to the "data structures live forever" idea,
many people would be surprised how many 80 column card-type records
still exist in production systems (even in some newer programs).
While many of these have apparently been replaced by user-friendly
data entry methods, it is often the case that, behind the scenes, the
data is written to the old 80 column card image and then fed to the
batch input/editing routine.
