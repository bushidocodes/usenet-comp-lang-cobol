```
Our shop uses the Panvalet library managment tool in a COBOL, OS/390, ISPF
environment.

The shop standards mysteriously don't allow the use of standard PDS for code
management, everything has to be in Panvalet libraires.

Some of our code maintenance has been outsourced, and the supplier has
returned code to us which uses COPY REPLACING which appears to be
incompatible with Panvalet. To use copybooks instead of coding COPY
, you code ++INCLUDE . This isn't as powerful
as copy, as for example, there is no "copy replacing" equivalent.

Does anyone with any Panvalet experience know of anyway of reproducing the
COPY REPLACING functionality within Panvalet, and in general being able to
use the COPY statement in place of ++INCLUDE. We do not want to have to
edit the code our outsourcing partner has sent back to us.
```

I think Thane and Jeff missed a point here. In the original message,
the writer stated that the shop did not allow PDS libraries. That
would include copylibs. In which case the Cobol Copy statement would
not work. The ++INCLUDE is a Panvalet statement that works like COPY
except that it includes (copies) the member from a Panvalet library
rather than a PDS. I've worked with Panvalet for many years, but
we've always had copylibs and used COPY. I don't recall there being a
Panvalet equivilent to COPY REPLACING.

The advantage of ++includes is that when you retrieve the source from
Panvalet into a PDS (for maintenance, debugging, etc), the includes are
expanded and you don't have to compile to see the 'copy code'. On the
downside there is no 'replacing' option. What about replicating the
includes with the data names you want ?? Somewhat dificult to track, as
I can't remember if Panvalet gives you an XREF utility.
