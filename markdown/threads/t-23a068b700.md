```
A "COMP-3" number in Cobol is a binary coded decimal integer. Basically,
each digit of your number is "packed" into 4 bits (nyble) of the COMP-3
number, with the last nyble becoming the sign. For example, converting
the number 1234 would give you 0x01234F (in hexidecimal). Hex 'F' or 'C'
are used for a positive sign, and 'D' is used for negative. Note the
extra leading 0 fill it out to a full byte. Your mainframer's will
probably expect a field of a certain size, so you may have to do further
pading. This should be pretty easy to do using perl's pack and unpack
functions with the 'H' format. Sorry I don't have any example code
handy.
By "julian packed" they probably mean something in the format YYDDD or
YYYYDDD. This will pack nicely into a 3 or 4 byte COMP-3.

>Additionally, for a separate file that will eventually be loaded into DB2 on
>the mainframe, I need to write out a normal "comp" number (not comp3),
>whatever that is.  Any help here would be appreciated to.

A "COMP" in Cobol is just a binary integer, like an int or long in C.
Again, pack and unpack will be your friends.

>I'm hoping that if I select binary format on the ftp to the mainframe that
>none of this data will get distorted or converted on the way up....

Shouldn't be a problem, but remember that if you use binary mode, all of
your text data will _not_ get translated from ASCII to EBCDIC. The m/f
probably expects EBCDIC for everything. Looks like you might be stuck
doing your own translation.
```

```
Hold it right there, please.  Translating packed/binary numbers across
platforms is an... intricate exercise, to say the least; some of the
comments you've received here might have indicated this.

Translating display numerics (X'F0' - 'X'F9' in EBCDIC, X'30' - X'39') is
child's play and it is likewise an elementary exercise to write a program
to 'comp down' fields; this leads to the sugesstion of:

AIX side:

Extract data.
Convert data to display numeric format with sign indicator, leading or
trailing.

FTP file, AIX ==> Amdahl

Amdahl side:

Convert data from display numeric w/ sign indicator to desired packing
method.
Program/utility to load data appropriately.

If, however, there is resistance - and there are both strong reasons and
traditions why there might be - on the mainframe side to structuring the
```

```
DD has it.

When transfering cross platform, transfer in text mode. The sign should be
separate also. Don't send anything BINARY (comp) or PACKED (comp-3).  It is
very surprising how many times folks will leave off the separate sign.

Generally transmit in ASCII, but that can be a hard decision if you are bring
it into major online software like IMS or CICS as a transaction.  

If you are transfering a file, there are actually ways to get the mainframe to
read an ASCII file and present it to your program as EBCDIC.  (You have to
convince a batch program it is reading an ASCII tape!). But it is no major
problem in COBOL to convert the thing to ASCII yourself.

In UNIX environments you do have to be concerned about the notion of
white-space. Generally, you do not want the mainframe to be dealing with tabs
or form feeds or vertical tabs or newlines.  So hidden in this task is the
_possible_ need to generate a fixed length file on the UNIX side.  If you make
this a line sequential file, somewhere you will have to expand the UNIX newline
to a CR/LF sequence for the mainframe.

But really life will be easier on the Amdahl if you similate a fixed length
file (which to run the risk of confusing terms, will necessitate a binary file
on the PC as opposed to a text file).

On your text data be very careful of long quoted strings with lots of spaces. 
They can look like spaces as you view your edit session for the Perl script,
but the editor can generate tabs as the script file is saved.  ((It is a good
idea to have a look at the generated files in hex, at either end of the
transmission the first few times through)).

The auto tabbing problem can also apply to numerics that might be expressed as
quoted data, as in
     "987654"
     "     321" 
     "       13"
Your editor can turn those spaces into tabs.
If you are expresing numerics without quotes, then you have to have some kind
of format expression to align the data to a fixed format.

If you are up to speed on tabs, and know how to control your editor you won't
need this suggestion: but sometimes you need to generate data like "###321" and
"ANY#WORD###", and at some later point CONVERT the special character to spaces.

It is not a bad idea ot put a little extra bagage on your records when
transfering across platforms. Put maybe "BEG" in bytes 1-2-3, and "END" in the
last three bytes.  And a header record and trailer record are very nice ideas. 
The trailer might could have a record count. Check every record for integrity. 
But I bet you have thought of that stuff.  A lot can happen when you do this
transfer and these little shepard can save you a lot of time because they make
problem obvious. You can eventually weed them out if transmission time or file
size is an issue.


For your information the PowerPC 604 and the mainframe (IBM or Plug Compatible)
are both Big Endian. (The PPC can go otherwise but not so in AIX).  But don't
transmit in binary.

Oh by the way, and again I bet you all have work this out already,  the PC is
an environment where shifting case is easy.  Generally, the mainframe does
better with upper case only.  Your translator may need a hook to clean that up
(which actually argues for source code  translation, rather than JCL
parameter). EBCDIC 
does have lower case, but the traditions are different in the mainframe world. 
(A tiny thing that can burn you for example, is that some of the 3270 hardware
and emulators will display lower case as upper case. So you will think you have
it all upper case as you try to diagnose it with a browser on the mainframe,
whereas the data may be mixed or all lower case).

So anyhow, transmit text and CONVERT on the mainframe. Transmit signs as
SEPARATE character.  Probably transmit a fixed record length file.  Beware of
tabulation endemic to PC and infectious to mainframe.  And determine if you
need all upper case characters.

Best Wishes,

Robert Rayhawk
RKRayhawk@aol.com
```
