Unisys MCP/AS COBOL(68) and COBOL74 both unconditionally set indices to the
actual offset of the data item being indexed, not to the subscript value.
For example
"JerryMouse" <nospam@bisusa.com> wrote in message
news:oKydndQWy-SfH8qjXTWcjA@giganews.com...

> Nah (assuming, of course, the implementor didn't do something really dumb
> like using display subscripts).

Unisys MCP/AS COBOL74 unconditionally stores an INDEXED BY index in an
on-stack integer variable, and the value stored in it reflects the offset
from the first occurrence in the table as well as the table stride.  By
default, the range of the index is checked on reference *by the hardware* to
ensure that it does not go outside the bounds of the RECORD, but it is not
so rigorous as to insist that it may not go outside the bounds of the TABLE.

Subscripts are stored wherever they're declared, in whatever format they're
declared, and always represent a table entry occurrence NUMBER in that
format, so when used as a subscript the value must undergo a multiply and an
add to convert its value into an offset to the desired data item.  Though
it's possible to get a subscript quite close in performance to an index by
declaring it 77  USAGE BINARY EXTENDED PICTURE S9(11) and by setting the
Compiler Control Option $OPTIMIZE to defeat range checking on subscripts,
you still have the distinction that the stored value in an INDEX is an
OFFSET to the beginning of the desired field, and the stored value in a
SUBSCRIPT is an OCCURRENCE NUMBER of the desired field, and however little
the multiply (of the stride) and the add (of the table's offset) might add,
unless the table offset is zero and the stride is one there is *some* more
cost associated with the subscript.

As I said before, it may be negligible in general, but in a critical path
that difference may not be.

Unisys MCP/AS COBOL85 includes some, but not all, of these differences (it
allocates memory differently from its older brethren).

I don't think that many people would be interested in an object code
comparison between access by an index and access by a subscript (of whatever
USAGE), but while the "best case" for subscripting is not all that much
worse than the indexing case, trust me, the "ordinary case" and "worst case"
for subscripting are indeed CONSIDERABLY less efficient than indexing.

    -Chuck Stevens

With everything else made as much alike as possible (usage chosen to provide
the most efficient possible numeric description for the subscript, bounds
checking defeated for both index and subscript), I consume something over
40% more CPU using a subscript than I do using an index.  And that's a
best-case scenario for the subscript, as I see it.

Note that for best results the subscript needs to be a good old 77-level
item.

Change the level number from 77 to 01 and leave everything else the same,
and the subscript performance penalty can increase to about 115% (over
double the CPU consumption).

Put the subscript in a record, located something other than an even multiple
of six characters from the beginning of the record, and it increases to
150%.

Turn that record item into USAGE COMP or DISPLAY, and the subscript penalty
goes to nearly triple (196%).

And I haven't even added range checking yet!

Deleting INDEXED BY clauses and 77-level numeric items from all COBOL
programs (of either current dialect) at a Unisys MCP/AS site because those
syntactic constructs are outdated relics with no defensible reason for
continued existence would border on sabotage, pure and simple.

Know your machine.  That doesn't mean you have to make use of every nook and
cranny of every extension to standard COBOL.  But it does mean it's a good
idea to know which standard features of the language are geared for
performance and which are not, in the particular implementation you're
dealing with.

    -Chuck Stevens

The original figure of 0.7 seconds for 100,000,000 references to an INDEX
was derived from a figure of 10.8 seconds for 100,000,000 MOVE statements
using a simple variable as a destination, and the comparison of that figure
to 11.5 seconds when the destination was indexed.  The base line, all the
overhead *except* for the actual cost of indexing/subscripting, is thus 10.8
seconds.

In increasing order of processor resource consumption:

Using an index without bounds checking on the reference:  0.7 seconds.
Using a 77-level BINARY item without bounds checking: 5.5 seconds.
Using an index with bounds checking on the reference:  10.6 seconds.
Using an 01-level BINARY item without bounds checking:  13.9 seconds.
Using a 77-level BINARY item with bounds ckecing:  14.6 seconds.
Using an 03-level BINARY (unaligned) item without bounds checking: 18.1
seconds.
Using an 01-level BINARY item with bounds checking: 22.3 seconds
Using an 03-level COMP item without bounds checking: 23.8 seconds.
Using an 03-level BINARY (unaligned) item with bounds checking: 25.4
seconds.
 Using an 03-level COMP item with bounds checking: 30.0 seconds.

Notice that bounds checking on an index costs nearly the same as a
13-character MOVE, and you can get nearly three such MOVEs for the price of
using a COMP item in a record as a subscript.   And once I factor out the
"baseline", it turns out that the least expensive subscripting is nearly
*eight times* the cost of indexing, taken in isolation.

Again, I agree, if your program's spending all its time waiting for
something-or-other (be it I/O complete or some response from the data base
system), it's not likely to matter, and barring any specific indication that
there is a performance problem with the program I wouldn't suggest a
programmer worry about it.  If, however, your machine is approaching 100% CP
utilization over significant periods of time and your company is about to
spend Quite A  Few Large upgrading it, it might be a good idea to figure out
what programs are soaking the processor and see if spending a few days
making performance improvements would help!

On every system, there are things that are efficient and things that are
not.  They differ from system to system.  On some systems there is no
efficiency difference between subscripts and indices.  On such systems,
perhaps the only real utility for INDEXED BY is in SEARCH.

Personally, I feel the fact that the standard allows the implementor to
define both the format and content of an index data item (see e.g. ISO/IEC
1989:2002 page 308, 13.16.36, OCCURS clause general rule 2, which is the
definition for "index"), is a specific and explicit invitation for
implementors to define them in ways that are most efficient on the
particular operating environment.  I would encourage people to use indices
where applicable on the grounds that the standard specification is such that
it is *very* difficult for subscripting to outperform indexing, all other
things being equal, and *very* easy for the implementor to have indexing
outperform subscripting.  I can't see an instance in which indexing is more
expensive than subscripting, I do see benefits for using indices, they're
still standard COBOL, and I see no signs whatever of them disappearing from
standard COBOL anytime soon.    The arguments I've heard against using them
do not make sense to me.

    -Chuck Stevens

I agree that $STATISTICS is useful, and it's a good guess that I might have
used this mechanism.  But there are some good reasons I didnt *in this
instance*.  .

The first, and probably the most important, is that $STATISTICS itself
imposes some significant overhead.  It is VERY useful for comparing the
performance of various parts of a program, and even with comparing programs
with one another, but the figures it produces can be misleading when taken
as an absolute, because of that overhead.    The COBOL74 manual even points
out that this option should not be used in production code because of that
overhead.

Second, the granularity of the ACCUMPROCTIME attribute (as retrieved from
what in ALGOL is available directly from the TIME function with a parameter
of 12) makes it difficult to draw conclusions from fine differences in the
values taken during program execution. The reason that's the case is, as far
as I'm concerned, that incrementing the value of this particular register
for each of the active tasks on the system is not the MCP's highest
priority, and a corollary to that is that I have occasionally seen cases in
which two retrievals of this register in the same program obtained the same
value, even when the calls were separated by significantly more than the
2.4-microsecond "ticks" in which the value is presented.

Thus, I would be very comfortable drawing *relative* conclusions from
$STATISTICS output, but I would personally want to have my object code
rather more "stripped down" before I would derive the more absolute numbers
I was looking for here.

In this instance, I wrote programs that were designed to minimize overhead
and maximize the impact that I was trying to measure.  I have found from
experience that when I'm testing a verb I get pretty reliable and
reproducible absolute performance numbers if I write a program that has
nothing more than 100 instantiations of the verb in a paragraph, minimal
initialization of data items, and a single PERFORM that paragraph a large
number of times.  Unless the verb does VERY little, 100 instantiations
within the paragraph usually turns out to be enough to mask the overhead of
the initialization and the PERFORM.

The "PT" portion of CANDE's end-of-task output has all the information I
need, reflecting as it does the "final" (and accurate) value of the
ACCUMPROCTIME attribute for such a program, and requiring as it does a
program that runs long enough that a single fractional digit is sufficiently
granular without being excessively precise.

    continue

    set an-index up by 1
end-perform

is on or off. All compilers do it this way because the standard requires them
to.

There are some confusing terms and even perhaps some conflicting definitions
here.

Most strictly stated, an "index" (as distinct from an item of "class index"
or "category index"!!) is the index-name referenced in the OCCURS ...
INDEXED BY index-name.  They are unique to the data declaration containing
them, and their value reflects the location of the specified item in an
implementor-defined way.  An "index data item" is a data item explicitly or
implicitly declared as USAGE INDEX (including index-valued functions).

What I thought we were talking about here is the former, not the latter.
Indeed, I don't see just a whole bunch of unique functionality associated
with USAGE INDEX data items offhand; when they are used to reference table
entries, they function as subscripts, not as indices.  SETting an index data
item to the value of a index converts the implementor-defined value in the
source to an occurrence number in the destination, but it does that even if
the destination's a plain old integer.  While I see a whole lot of value in
INDEXED BY, on the implementations I'm familiar with USAGE INDEX doesn't
offer anything over PIC S9(11) USAGE BINARY EXTENDED, save portability.

Unisys MCP/AS COBOL85 does not do a range check on SET index TO statements
(even with range checking enabled) for the very simple reason that the only
time the value of an index actually matters is when it's referenced.  Unisys
sees no particular reason to prevent you from doing something like
    SET INDEX-NAME TO 1.
    SET INDEX-NAME DOWN BY 47.
    SET INDEX-NAME UP BY <data item containing 47>.

The rule in COBOL85 (ANSI X3.23-1985 Page VI-128, 6.23.4 SET statement,
General Rule 2) that states that the value to which the index-name is SET
must fall within the range of the valid occurrence numbers for the
associated OCCURS clause is a "general rule" that does not specify a
penalty.  The results of violating such rules are undefined.

ISO/IEC 1989:2002, Pages 521, 14.8.35, SET statement, General Rule 2c
explicitly allows an index to be set outside the bounds of the table when
the destination is an index-name and the source is either a regular data
item or an index data item.  Arithmetic expression sources and SET UP/DOWN
BY statements have different rules in this regard, but the bottom line is
that the current standard explicitly allows a SET to cause an index-name to
be outside the bounds of the table.

ANSI X3.23-1974 indicates that the source for "SET index-name TO ..."  can
be an integer literal, an index data item, an index-name, or a numeric data
item (page III-11, 3.4 SET statement syntax rules 2 and 4).  ANSI X3.23-1985
says the same thing, as does ISO/IEC 1989:2002 in this respect.

> Nor can they be used to index a table.

Huh?  Index data items can be used to index tables, even when the table has
an index of its own!  An index data item contains an ordinal occurrence
number and is perfectly valid as a subscript as well as a container for
indices.

>The index variable contains an offset, not an ordinal.

If you've shifted gears to indices and away from index data items, some
implementations do it that way (and I would contend that's likely the most
efficient way to do it), but there's no requirement that they do so.  So
long as an index contains something that can be used to identify an element
of the table, in whatever format the implementor decides, and so long as the
implementor provides a mechanism to convert to and from ordinal occurrence
number to whatever form the implementor's decided on, then the
implementation meets the standard.  There is absolutely NO requirement that
an index contain an offset; I have run into implementations, in fact, in
which the contents of an index are indistinguishable from the corresponding
subscript.

Yes and no, or perhaps more appropriatly, No, then Yes.

ANSI X3.23-1974 and prior standards did not define the order of evaluation
of the components of a conditional expression, and did not define whether or
not components of that expression were evaluated once it was possible for
the truth value of the conditional expression to be determined.

ANSI X3.23-1985 (and subsequent standards) specifies the order of the
evaluation of the components (as left-to-right), and explicitly states that
evaluation ceases for a component or for the expression once its truth truth
value is determined (page VI-62, 6.3.4, Order of Evaluation of Conditions,
item 1).

The latter standard uses exactly this circumstance as a defense of this
"Substantive Change Potentially Affecting Existing Programs" (page XVII-58,
item 12, Justification, second paragraph):   "This change will allow a
program to safely test that a subscript is within range immediately before
using it as a subscript in the same statement; for example:  IF INDEX-A IS
LESS THAN 5 AND TABLE-A (INDEX-A) IS EQUAL TO 25.  "
