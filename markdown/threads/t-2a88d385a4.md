Are all of the fields packed, or just some of them?

If all of the fields are packed, then you don't have to do any EBCDIC
to ASCII conversion.

If some of the fields are packed, then as Kerry pointed out, it gets
complicated.

I would consider writing a C function -- call it e2a -- to convert n
bytes at address p from EBCDIC to ASCII.  You might be able to use one
of the tables here:

http://www.opensource.apple.com/source/OpenSSL/OpenSSL-16/openssl/crypto/ebcdic.c

Your COBOL program could do something like this:

1.  READ the input file
2.  CALL "e2a" using field1 LENGTH OF field1
     CALL "e2a" using field2 LENGTH OF field2
     etc for each DISPLAY field that must be converted
3.  WRITE the record to the output file
4.  READ the input file again

Coding this would be tedious, but you might be able to write a script
or a program that would read a list of field names and generate the
CALL statements.

Have fun.

The first question in converting files from ASCII to EBCDIC is whether you 
have both character and comutational data or not.  If your COBOL program 
only has "PIC X" type data, then the conversion is easy with or without 
COBOL.  If your file (as most do) has mixed PIC X and COMP, BINARY, 
PACED-DECIMAL (etc) fields, and especially if there are multiple types of 
records (with different lay-outs), then you need to do an "intellegent" 
conversion

Writing a COBOL program to do an "intellegent conversion" is very easy if 
you understand MOVE CORR and CODE-SET syntax and YES, you can easily do this 
in OpenCOBOL.
