As of yet, I don't see where anyone here has addressed what is at the
heart of your performance problem. I've done a number of Acu Vision
conversions to Oracle, Informix and others and, although I always
found Informix to provide faster performance, there's an inherent
problem with what you're trying to do.

Your time/resource intensive COBOL applications are designed to update
ISAM files on a record-by-record basis (excepting the data buffer).
The records are inquired upon or updated individually. This type of
activity is extremely slow via an SQL-driven relational database and,
to my knowledge you'll always take a huge performance hit for much of
your code. 

The way we got around this on a project we did for DreamWorks was to
rewrite the performance-critical applications in a manner that was
condusive to the relational model.

Well, you are both right, sort of.
If you mean big millions of record updates ran in a batch job, then
VSAM is a little more efficient, mostly because it has less keys and
indicies to keep in check. But there are ways to do that to a SQL
database that elimiante most of the difference in speed, and
of course, you get a lot better ways to manage data itegrity
on a database than in VSAM.

If you mean in an online environment, well, even a couple thousand
online users would be hard pressed to tax a system with as many updates
as a single big batch program. And since with modern technology,
you can partition the client and server parts to run the updates
in a very efficient manner on the host system, well, you don't really
see any major slowdowns.

In fact, with a well designed and managed database system, you see a lot
better
response time. SubSecond response time is not unusual. And it scales very
very well.

So actually, there are very few advantages to using VSAM or ISAM files over
using a database, and they are not really realted to speed in most cases.

-Paul

Not really Thane.

I'll use DB/2 as an example, since I know that one the best and it is my
favorite. :)

A developer can get a personal use copy of DB/2 free for the download, or
for a few bucks on CD. That's a real working version and the SDK for
essentially zero. Develop away...  (Actually, I just looked up the price, it
is about $50 if ordered though IBM Software, but that includes hardcopy
documentation. It is available free for download but I'm not sure of the
license restrictions on the download copy.)

DB/2  production copies are a bit more expensive, but your average NT
installation cost is about $500, with discount.
Of course, the price does scale up in to the 10's of thousands of dollars,
but that is true of every database as well.
SQL Server, Oracle, Sybase, Informix, and Progress have similar pricing
levels for production deployments.

Then you start adding on the intangibles, such as Java based access to the
database, which means that servlets are incredibly easy to write, and the
idea of managing the data storage as separate entity enables data
warehousing and highly optimized OLTP processing as well. VSAM just doesn't
give you those options, at least not anywhere near as easily. (Yes, I've got
a couple systems out there running web pages against VSAM style files- it is
much harder! :)

There are also developer programs out there that can get you really decent
developer software for really low
prices. VSAM seems to me to be a great solution for data recording (ESDS)
and such though.  I'm writing a little application now that just provides
access into the UCC database. (Actually, I am doing it for fun mostly, but I
plan on
porting it around to different platforms as an example of an ASP hosted
application - my new entrepe-nueral (pun intended) direction. ) I think I
will write it into a VSAM file this afternoon and try to do an objective
comparison of how difficult it is to do in VSAM vs DB/2.  It would be fun to
see how it behaves anyway under different file systems. I have IBM Cobol,
AcuCOBOL, and Fujitsu here.  IBM uses Btrieve or STL, AcuCOBOL uses Vision,
and I'm not sure what Fujitsu uses, but I bet none of them can read the
other's files. :)
