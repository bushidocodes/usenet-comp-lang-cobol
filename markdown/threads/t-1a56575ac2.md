```
To clarify things (a bit) from another thread,

1) The ISO 2002 Standard has a phrase "IS RECURSIVE" which may be added to the
Program-ID paragraph; when so coded the program has recursive attribute and may
be called recursively. This is supported in current Micro Focus and IBM (OS/390
& z/OS) compilers.

2) The ISO 2002 Standard has a Local-Storage Section which may be used by
recursive OR NON-RECURSIVE programs. This section is supported by both IBM and
Micro Focus. (When used by a "non-recursive" program it is similar - but not
quite the same as using WS for a program with the "IS INITIAL" attribute.)

3) Micro Focus has a documented extension, that WHEN a Local-Storage Section
appears in a program, that they "auto-magically" make the program have the
RECURSIVE attribute - regardless of what is coded in the Program-ID paragraph..
(This doesn't hurt - if you don't use it recursively)

4) IBM (mainframe) gets Local-Storage from "STACK" storage and Working-Storage
from HEAP storage.  (Different LE options apply to these)

5) In a recursive program, if you use Local-Storage with an ISO 2002 conforming
compiler it uses "push" and "pop" logic to get, save, and retrieve storage as
you go into and exit out of instances of the program.

   ***

To what extent other compilers have the RECURSIVE attribute, Local-Storage
Section, and/or use these in an ISO 2002 conforming manner, I cannot say.
```

```
Well "Yes" and 'No" - but note this is specifically LS used in OO :-

Example # 1 :
*>--------------------------------------------------
method-id. "displayNames".
*>--------------------------------------------------
Working-Storage Section.
01 Name1   pic x(xx) value "Hello Bill".
01 Name2  pic x(xx) value "Hello Robert".

*> Procedure Division - can be ignored here - no sending/receiving 
parameters.
display Name1
display Name2
End-method "displayNames".
*>----------------------------

Prior to J4 changing syntax rules above was acceptable and still is - a 
way of getting around LS not having values.
The reasoning for J4 disallowing Working-Storage in a method was that 
this could conflict with the same variable name if used in Global 
Working-Storage for the OO Instance. (you have a situation where, per 
instance, the method variables are not 'clean', i.e. in their initial 
state when a method is  invoked, and may be re-invoked many more 
times).  M/F's own rules still apply  if you use Variable A in both 
global W/S and a method; then the method Variable A overrides the global 
Variable A.  I can live with above, but I know somebody who still 
grumbles, grumbles about removing W/S, each time I correspond with him :-)

Example # 2 :
*>--------------------------------------------------
method-id. "displayNames2".
*>--------------------------------------------------
Local-Storage Section.
78  Name1   value "Hello Bill".
78  Name2   value "Hello Robert".
01 DisplayName    pic x(xx).

*> Procedure Division - can be ignored here - no sending/receiving 
parameters.
move Name1 to DisplayName
display DisplayName
move Name2 to DisplayName
display DisplayName
End-method "displayNames2".
*>----------------------------

Again of course, the nice shorthand Level 78 has been replaced by J4 by 
CONSTANT  - recent proposals have been made
by Chuck Stevens to accommodate the above with a CONSTANT structure. - 
due with COBOL 2008 - I await with bated breath !

In their usual pattern of backwards compatibility, Micro Focus will  no 
doubt support both the "old" and "new".

I've no idea what goes on at these J4 meetings - but sometimes I just 
can't help feeling that, "If M/F dreams it up, let's knock it down". 
Absolutely no evidence to support my suspicions - BUT  note in the case 
of disallowing W/S in a method the reasoning was legit - like Bill and 
some others, I was privy to non-J4 discussions.  The funny thing is we 
bitch about the '17' years to get to COBOL 2002 - but the 'Exclusion of 
W/S'  was handled with alacrity - it was up, running, discussed  and 
rubber-stamped by J4 before folks like myself had a chance to digest the 
ramifications of dealing with the above examples.
Who knows - maybe there is a case for taking 10 years to implement 
changes - seems like a reasonable period for everybody interested in a 
topic to give it some thought - the draft spec hopefully should be right 
at the end of a decade >:o               

Jimmy, Calgary AB
```
