```
In Oldene Tymes when prehistoric IBM mainframes stalked the corridors of 
Corporate, and spinning platters were starting to provide random access to 
data as well as sequential, there was an Operating System call "DOS".

This OS had logical names for certain devices.

For example, a distinction was made for reading "data" (from a device called 
SYSRDR) and  reading "control information" (from a device called SYSIPT). 
Sometimes two distinct card readers were used to accommodate these streams, 
but, hardware being extremely expensive, sometimes the same physical device 
had to serve both purposes and it was then logically assigned to SYSIN.

Seeing a post here that required to look at elements of the JCL in a running 
job reminded me that in older (simpler?) times, this was not a problem. You 
could read a jobstream just like you could read anything else (although the 
operator might have to reload the card deck if they were required to be read 
again...normally you read it and saved it somewhere.)

I was wondering if, in today's modern magical world it was still possible to 
open and read a JCL stream, from an application program? If it is, then 
surely there is no need for dropping intoAssembler or writing convoluted 
COBOL to address OS control blocks?

Any comments?

Pete.
```

```
I'm not certain how it works in VSE (was DOS then DOS/VS then ...) but
in z/OS and predecessors JCL can be read with a //ddname DD DATA card
but that would not be the JCL that is executing the program in
question but some JCL that the submitter wants read.  What the program
in this case is probably going to try to is find out where it is being
executed from.  This means chasing the //STEPLIB concatenation, if
any, then the //JOBLIB concatenation, if any, then the system list of
executable libraries.  This is not for the faint of heart.  The
overhead for doing so might be low or not so low because the directory
of each library pointed to by the DD statement would have to be read.
The program would have to be authorized to read the directories in
addition to being authorized to execute programs from them.  The task
is cute in assembler and may not be doable in higher level languages.
I suspect the job is equally interesting in Windows depending on PATH
statements etc.
```

```
Comment...I did that migrating batch cobol (and JCL) from legacy to 
Windows/NT

(OS)RUN batchfile (a .bat file, equivalent to a JCL file because both 
are 'native' command files)

(batchfile) SET MYPARM=%0.bat

(batchfile) RUN cobolprogram

(cobol) ACCEPT batchfilename FROM MYPARM (environment label/name)

then OPEN and READ it LINE SEQUENTIAL

Another method is

(batchfile) RUN cobolprogram %0

which passes the name of the executing, native batch (JCL) file name to 
the cobol program as a parameter, rather than fetching it from an 
environment variable.  With either method, the physical name of the 
jcl/batch file can change without maintenance to the code.  If you run 
cobol from a modern script language instead (not a native OS batch 
command language) then the script will have to provide equivalent 
capability as the native OS, as provided in command.com on Windows/NT. 
I don't know unix, but I assume it's various 'native' batch facilities 
can do it.
```
