Just struck me I've never used indexing to search conventional text
tables, (not OO Collections/Dictionaries - which can use both
approaches, plus some others). I use :-

	perform varying n from 1 by 1 until
	  condition met or n > MaxTable size
		if condition met
		   do something, if necessary, and then
		   Exit Perform
		End-if
	End-perform

Anybody care to comment upon the difference in performance between the 
above and search using index - or are they comparable ?

As just a tiny point (don't get upset and rant at me), it may
be better to have:

            until n > MaxTable size
               or condition met

because n may be used within the 'condition met' and this will
lead to a bound check error if it really is > MaxTable size.
Checking n first prevents this bound failure because Cobol
will short circuit the test when it sees 'TRUE or ..'.

That is, Cobol is not clever enough to know that the check for
n > MaxTable should be done before using n if it is done the
way you had written, but it is clever enough to not continue
and cause a bound failure if you give it the check first.

Hmmm...interesting.

I NEVER write my own and ALWAYS use SEARCH.

I used to write my own and had some quite sexy methods which were even more
efficient than the binary chop used by SEARCH ALL on a sorted table, given
that there were certain conditions in the data (e.g. a preponderance of a
given generic key, say, more entries start with "C" than start with "F").
Most of these advanced techniques were skewed variations on binary chop. (I
don't really want to go into this as it was all a long time ago and I have
no intention of encouraging others into what I now consider were the sins of
my youth  <G>, besides, in those days, performance REALLY mattered; now it
just matters a bit...)

I offer the following observations in support of my position:

1. There is no "index overhead" and I don't know what Thane's talking about.
(Ah, sorry Thane, I just re-read what you wrote again and you are talking
about a CODING overhead. That's the time it takes to write INDEXED BY,
right?...(You should be ashamed of yourself...and you an example to new
COBOL programmers...!!<grin>).The overhead of conversion is less than it is
for a subscript, as the compiler does not have to multiply the subscript out
in order to obtain an offset into the table; the index is already held as an
offset so there is one less step in evaluating the address.  Where the index
is being manipulated, the compiler must obviously do arithmetic whether a
subscript or an index is used, but the index, being more directly under
compiler control will be in a form for optimised arithmetic; the subscript
(being defined by the programmer), may not be.

2. Defining your table with INDEXED BY and using SEARCH means that if the
table is extended, (you add new entries to it) you do NOT have to make ANY
change to the Procedure Division code. If you use PERFORM VARYING you must
go through and update the termination condition for every PERFORM which
looks it up, because you have now increased the OCCURS clause in your table
definition.

3. COBOL is a self documenting, easily maintainable language. Consider the
two following examples of table lookup:

PERFORM ...VARYING ...FROM ..BY ...
UNTIL
(Then check the termination condition AGAIN to decide if you found it or
not!)

as in the example given, which by the way, I have no issue with...

>>
>> perform varying n from 1 by 1 until
…[5 more quoted lines elided]…
>> End-perform

OR

SEARCH  table
      AT END perform it-wasnt-found
WHEN table-entry = thing-im-looking-for
       perform i-found-it
END-SEARCH

IMHO, there is no comparison in terms of readability and ease of
maintenance, so I subscribe to the latter.

You may object in terms of "performance" but that's a smokescreen, because
the difference in performance (if any) will be so slight as to not matter.
On today's hardware platforms you are talking maybe a few milliseconds. (I
haven't done comprehensive tests on different platforms with thousands of
entries, and I have no intention of doing so...I know enough about computer
architecture to know that this is acceptable.)

Compare it with the time taken by a maintenance programmer to find that the
OR condition in your PERFORM varying termination could never be fulfilled,
because there was some dodgy input, or somebody didn't extend the
termination condition (in just one of the six places where the table is
looked up), after the table had new entries put in it (well, it compiled OK,
didn't it?) at 3:00am, having being called from a warm bed...(I was that
soldier. That was around the time I started using SEARCH...)

SUMMARY:

Use whatever you feel comfortable with, and whatever meets your personal
criteria.

Don't write off SEARCH, it is valuable for the reasons outlined above. (I've
written 1 or 2 COBOL programs in the last 30 years, and all the table
lookups I write nowadays, use SEARCH)

Pete.

Given part of this thread, I wanted to make certain that anyone reading it
who doesn't already know about the EXIT PERFORM enhancements in the draft of
the next Standard understands that there are TWO variations (one with and
one without the word "CYCLE"), the definitions are:

"8) The execution of an EXIT PERFORM statement without the CYCLE phrase
shall cause control to be passed to an implicit CONTINUE statement
immediately following the next END-PERFORM phrase that matches the most
closely preceding, and as yet unterminated, inline PERFORM statement.

9) The execution of an EXIT PERFORM statement with the CYCLE phrase shall
cause control to be passed to an implicit CONTINUE statement immediately
preceding the next END-PERFORM phrase that matches the most closely
preceding, and as yet unterminated, inline PERFORM statement."

The "real word" difference is that you should use the "CYCLE" key word if
you DO want the "UNTIL" to be retested - and don't use it if you want an
"unconditional" escape from the loop.

side is that you have to handle the table load procedures and ensure
that everything is ordered by your search key.  Any time the table
exceeds previous design limits you are forced to recode the table limits

as you no longer have to have the entire table in memory.  The only overhead is
from the program-to-program call and the DYNAMIC/INDEXED read process.
