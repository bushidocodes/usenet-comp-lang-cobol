If there are those who have actually used COBOL object compilers, comments on
this would be great from the focus perspective of COBOL. Generally I think that
the answer is that there is no such limitation as the poster askes about..

You can get an object's features by inheritance or by aggregation. You do have
to be an object of class x to inherit that classes objects, but you can just
instantiate any object within your object.

You can 'use' code and data in these two fashions in object oriented
programming. 

The term code-reuse is fairly tired and worn out, but it relaly just means if
we plan ahead and design programs with good abstractions we can probably use
them in a number of places during the initial system deployment, and perhaps
use them again (invoke them as inherited features or aggregated features), in
new programs down the line in subsequent years.

Code re-use is one of the religions of modern data processing. It is not an
intrinsically OOD concept, it is more general.

The idea of the OOP tools is that they encourage analysts to plan ahead
somewhat differently than older tools do. The discussion can be about whether
that is true, and if so is it an improve manner of thinking.

Code re-use is hard. There are many factors that move against it, including
non-technical political issues in major corporations.

From the stand point of the programmer, inheritance is one way to get
capabilities of code that was previously written. Aggregation is another.
Importantly, as you seem to suggest in one of your prior posts, the flat
topology of APIs also is a way to get to previously written code.

That is, COBOL is an excellent model of how things will not be flatlander file
system OR object based systems. There will be a mix.  We will have flat files,
object files, and EXEC escaped APIs harnessed. It just keeps getting bigger, 
better AND badder.

We will have a mixture of object oriented programs and classically called user
code and
ever more APIs that become successful as part of middle ware products or OS
services.  

Inheritance is only part of the picture. Code re-use is a general goal to save
money in the long run by not re-writing code.  There is some religious belief
that if you design more abstractly the code is not only more likely to be
re-usable, but might even be more correct upon the initial system install. Even
if you believe in the virtue of
early abstraction in a project, it remains open to discussion as to how to use
the methodologies of OOD per se to make a real difference in the professional
work.


Kellie, do you use object orient technology at this time?

This generalizes beyond the simplistic bank example where we may want a
customer, plus an account and sometimes two accounts if we are going to do a
transfer fiduciary.

In the general ledger we have not only two sided transactions all the time in
double entry book keeping, but there is really a list on each side.

Asset increase:
  add to equipment  18,000
  add to cash              2,000

Liability increase:
   add to loans we gotta pay  20,000

A single object, the balancedEntry might be abstracted.  This kind of thing
_might_ have object thinking applied to it.

The resulting journal could still be written as a flatlander sequential file
(where we are in append mode), or a SQL file. The incoming transactions,
likewise could be flat file technology; or online one at a time transactions
acquired by APIs in major software like CICS, IMS, or a socket kind of thing
(via an API off the shelf).  We can still apply object thinking to the core
process.. It is a choice.

If object thinking improves the design, lets use it. If the available OOP tools
help generate code economically, then lets use that.

Examples abound, really!  The only thing that makes them hard to see is
resistance. When we move from them to us, it gets easier.

The objectives are strictly practical. Save money! by reducing development cost
or increasing reliability of deployed systems.

We can abstract the idea of account. We can abstract the idea of working
several accounts at once (in a fiduciary sense) in a combined transaction (a
transfer, or a multi element 'double entry' journal action). We use the object
think if it helps clarify.  The goal in the begining of design is to figure it
all out as best we can before we code. OOD does help, according to some
practitioners.

Once we have design, some people believe that the automation of the data

their conventions might lead them to try to instatiate that notion in our minds
