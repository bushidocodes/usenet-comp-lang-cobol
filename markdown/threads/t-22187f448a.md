Bernardi Gian Claudio asked:
: A program can corrupt an index file?
: If yes, how this can happen?

As a previous answer pointed out, the file handler might have some system
problem which would corrupt the index. But it is also very easy to bypass
the file handler, by using it incorrectly, and this can destroy your
index.

To avoid corruption, the program must always "select" the file with
"organization is indexed". And the name of the file in the "assign"
clause must be correct: If you open the index portion of the file
as a separate file, then you can corrupt the index.

If you open the file as a sequential file, or a relative file, you may be
referencing only the data portion, or only the index. These are two
separate files which are accessible to the operating system individually.

Micro Focus uses an 'assign' giving the name of the data portion of the
file, and it adds ".idx" to this name implicitly to open the index. So if
your 'assign' gives the name 'abc' for the file, then you will also create
or read the index file, 'abc.idx'. You should not open 'abc.idx'
explicitly. (Some other COBOLs use other structures and conventions, so
this rule may not apply in Accu-Cobol for example.)

I sometimes read the data portion of a Micro Focus indexed file using a
sequential organization, and it happens that this works OK, though the
order of the records is indeterminate. But if I were to write into this
file, especially if I were to change a key field, then I would destroy its
integrity.

Victor Odhner, Phoenix AZ http://www.primenet.com/~vodhner

Having worked with MF cobol for 12 years, I can answer this question.

No bugs. When you perform a WRITE, the data portion of the file will be
written but the index portion may be buffered. Upon a CLOSE, all data is
written to the files. If the program is terminated prematurlly prior to a
CLOSE of the file, the index will be banged. By using the WRITETHRU
directive in your COBOL.DIR file, the index data will be written upon a
WRITE. Performance will suffer, but only depending on the complexity of
the index for that file.
