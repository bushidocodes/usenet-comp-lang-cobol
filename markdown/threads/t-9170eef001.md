I resisted the OO urge for far too long.  My coding style is
structured and modular - but is that enough?  I've taken to giving my
existing systems an OO Analysis - what could I have done if this was
OO?  How would this have worked better (or worse!) with OO?  What I've
found is that I could have saved myself a ton of work with the reuse
OO gives me.  Don't get me wrong - I reuse code all the time.  I have
standard paragraphs that perform functions that I can use as copy
members or pasted into the code.  Things that were  of medium
complexity become simple with OO, and things that were simple become
trivial.

For a GOOD explanation of the benefits of OO, in plain language, using
common COBOL practices as examples, read Chapter 27 of COBOL
Unleashed.  It's a very good, plain language explanation.

Let me give you a real world example I recently found.  I am
converting a billing system from flat files to database.  One feature
of this system is the ability to "sponsor" another company and pay
their charges.  When the invoice is generated for teh sponsoring
company, the customer information from the sponsoree (name, address
etc) is printed on the invoice.  In the original version there is a
separate flat file that has just that information in it, keyed by the
customer number.  So a separate read is performed and the information
is moved to the invoice.  The "master" customer file record, of the
sponsor, remains untouched.  Going to the database, we don't have the
separate tables.  So now, we populate the record from one entry, then
save this data to a temp area, read again for the sponsoree, then move
the address info, then move our saved sponsor fields back to the
active fields.  An alternative is to write a second subprogram doing a
select on the database to get the sponsoree information we need. 

If this program were OO, I could have separate "customer" objects and
reference the one I want, having multiple copies of it, as objects,
active at one time.  

Only a single access program is needed, and no swapping of fields or
any of the other gyrations are necessary.

But, If you are backing your objects with a relational database,
you have to have some way of instantiating them with the data from the
database.
This takes extra work, but is probably worthwhile if you can abstract
the DB read.

It becomes a bit more of a problem when you want to write or update data
in an object.
Now, when you instantiate, you have to get the DB data for update and
ensure you obtain locks on the necessary rows to prevent others from
accessing the data.

This can be done, probably by a method that reloads the objects
variables locking the DB rows as it goes.

Now once you have done your updates, you have to COMMIT the changes.
After a bit of thought, you find that you need a whole CLASS to manage
the DB access.
Doing such things as remembering which objects are instantiated from DB
data, which objects have been converted from memory access to DB via a
"read" method, which objects have hold locks via a "read-for-update"
method, which objects have had data updated, which objects belong to a
transaction, how to commit multiple objects based on a transaction, ....

The list goes on and on and on.
However, If you spend the time to add the classes for DB access, or buy
an equivalent set of classes, you will find that OO DB access is
relatively easy.

OK for the uninitiated.

McConnell points out that modular is about grouping functionality
together.  When you group functionality together you often need to
perform multiple external calls to the 'module'.  This is achieved a
couple of ways:

1.  have one call with a flag that switches the function.  This is
typically, open, read, write and close type functions.  This is
coupling and considered bad.

2.  Have multiple calls that reference a common batch of data.  Sounds
like an object doesn't it.

As far as I can work out there is no other method of doing this in
Cobol.  In C you can have data that goes across a number of modules
but is protected from modification by other programs (Static globals
in a source module).

Before I read up on this I considered a module where a call ends up. 
I realized after reading this that it was a bit broader than this, it
covers encapsulation as well.

2) OO Concepts

To my mind there are two OOs � one is the additions to COBOL to make OO
COBOL and the other is the general concepts.  OO is often described as
a way o thinking,

As a transition, you might try using OO techniques with classic COBOL
code.

For example:

It is normal to use a code in transaction files to identify the type of
transaction.  Suppose that instead of a transaction code you put in the
name of a module that processes that type of transaction.  This gives
you encapsulation of data and code.

If you use standard extensions of the module name as entry points to
perform different functions, you have effectively created a class and
methods.  For example, a module name + ED becomes the entry point to
edit the transaction, + AP to apply it to master file etc.

Once you understand OO concepts, then making use of them is no more
difficult then using to use structured code or modular programming.

Breaking the job into two steps � first learning and applying OO
concepts then learning to use OO COBOL -should make the transition much
easier.
