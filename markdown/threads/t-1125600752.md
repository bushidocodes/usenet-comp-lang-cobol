 MOVE ZERO TO INT-OUT.

    SUBTRACT 1601 FROM DATE-YEAR GIVING THIS-YEAR.
    COMPUTE LEAP-ADJUST = THIS-YEAR / 100 - THIS-YEAR / 400.
    COMPUTE LEAP-DAYS = THIS-YEAR / 4 - LEAP-ADJUST.
    COMPUTE INT-OUT = THIS-YEAR * 365 + LEAP-DAYS.
******* add in # of days in preceeding months
           ADD DAYS-TO-MONTH( DATE-MONTH ) TO INT-OUT
    MOVE DATE-YEAR TO LEAP-YEAR-TEST.

******* add in day within month
    ADD DATE-DAY TO INT-OUT.
       A000-EXIT.
           EXIT.

The algorithm is fine. It's just not very clear.

First, THIS-YEAR means the end of the previous year.

Second, replace LEAP-ADJUST in the second COMPUTE,
with the right -side of the equation from the first COMPUTE
(be sure to change the sign), the algorithm is:

1. Total all years divisible by 4 as leap years.
2. Subtract all years divisible by 100 as NOT leap years.
3. Add all years divisible by 400 as leap years.

The result is the number of leap days between 31 Dec 1600
and 31 Dec of the prior year.

The problem seems to be that the first COMPUTE is
rounding instead of truncating the intermediate results.

When I ran the code using MF 3.2, I got the same error.
After adding $SET ARITHMETIC"VSC2" the results were
correct. The default for MF 3.2 is to round. The directive
forces truncation of intermediate results.

Thane, you are too kind.

The very first COBOL program I ever wrote professionally, back in 1979,
was a routine to determine day-of-week for a date.  I believe it used
Zeller's congruence, and I didn't understand why it worked.  I had to
translate it from an old DOS/VS 360 assembler program in ANSI-74 COBOL
for an IBM Series/1 minicomputer.  I noticed that it needed a 4-digit
year in order to determine the day of week.  Even in 1979 we knew there
would be some issues with the year 2000.

I am no mathematician, but I can (usually) do arithmetic correctly.  The
problem of determining the number of days between two dates becomes
difficult when you cross a calendar year boundary or a century
boundary.  So I eventually developed my own routine.  It worked, but I
doubt that it is as efficient as a more elegent mathematical routine. 
The only advantage my routine had was that I understood how it worked.  

Basically, the way I determined the number of days between two dates was
to convert each date into an absolute number of days from a base date. 
I used January 1st, 0001 as the base date, assuming the current
gregorian calendar rules.  Then it's easy to determine the number of
days between two dates.  It's also easy to get the day of week.  Just
convert to an absolute number of days from the base date, divide by 7,
and convert the remainder.  January 1, 0001 would have been a Monday,
and day 000001 of the absolute day count.  A 400-year cycle has exactly
146,097 days and is evenly divisible by 7.  So January 1, 2001,
(absolute day count 730,486) is also a Monday (730,486 mod 7 = 1, or
Monday).   

Converting a yyyy-mm-dd date into an absolute day count is not too bad. 
Converting an absolute day count back into a yyyy-ddd date or a
yyyy-mm-dd date was considerably harder!

If you have COBOL intrinsic functions, you can save yourself a lot of
work by using them instead.
