The line I quoted almost demanded my reply, which was basically my attempt
at being funny.  But...

C++'s limitations are mainly in the area of costly maintenance.  Most any
characteristic of a language has both advantages and disadvantages (as most
any choices in life).  Spreading code around, hiding methods, etc. are
supposed to eliminate bugs, but sometimes they just make them hard to find,
especially when you change some code which is widely used (C++ programmers
preach the virtues of reuse, I also see advantages in reuse).  As with
COBOL, C++ requires standards, in this case to make sure programmers aren't
too terse, and put appropriate code where we can find them.

We can do more with C++, but we can do even more with machine language.  So
maybe "limitations" is a poor word to use.  It isn't that the language is
limited, it is that effective business use of the language is not always
optimal.  It should be primarily a business decision which tools a company
should choose.

It is no better or worse than a subroutine, in that respect.  So the answer
is yes, every type of use should be retested.  However, there is one thing
to keep in mind.  The objects that contain business logic are quite
different than the ones that have only to do with screen display (for
example).

A Cobol object can be looked at as a set of subroutines with several
advantages.  First of all, the parameters are rigorously checked. You
cannot, for example, invoke a method if the calling parameters are not
*exactly* matched to the called parameters. That is done at compile time.
Secondly, you can have many "instances" of the routine.  Picture a
subroutine with it's own data.

You know that the data is internally available on successive calls ...
while local to the subroutine, it is not going to be in the same state on
the second call as on the first unless declared as INITIAL.  That is
standard Cobol, and quite different than most languages.

That is roughly analogous to a single instance of an object, and, in fact,
you can use OOP coding in exactly that way ... create a single instance of
the "object" at the start of your program, then destroy it at the end.  The
methods then replace the subroutine entry points, so you have the equivalent
of a subroutine with multiple entry points and persistent data for the
duration of the run. You gain the advantage of far more rigourous parameter
checking than with subroutine calls. If you do nothing more than replace
your subroutines with single instances of OOP routines, you will make a
system far more solid, and eliminate many potential bugs.

The next step is to simply duplicate the subroutine as many times as you
want it, so that each of the subroutines has it's own data. Picture a
subroutine that manages a screen.  I now need two screens managed by the
same Cobol program, so I make two copies of my subroutine and change the
program-id on the second
from "SCREEN" to "SCREEN1". I then call them using different initialization
routines, and do my IO to one or the other.

That is an object with two instances. Each inherits the original code of the
prototype, plus the "value" statements of the prototype, but once created,
each behaves differently, and each has independent data.

Try to maintain that code.  Change the subroutine "SCREEN", and you have to
also change the subroutine "SCREEN1".  With OOP's, you do not.  The
instances are created at the object level, rather than at the source level.


So debugging and testing is halved.

Now, take our program, and design it so that it can handle ten open windows
at the same time.  I can do that with ten subroutines called WINDOW1 through
WINDOW10, or I can write it as OOPS and at run time create ten instances, or
I could do it with one subroutine having a table of ten cursor positions,
etc., and pass the window number as a parameter on each call.  Which of the
three is going to be easier to debug and maintain?

There are ways around the above, of course.  You can get into copy libraries
to ensure that different programs/subroutines are co-ordinated. However, if
you fail to compile one module that uses the copy, you are screwed.   If you
take that copy statement, and hard code it into an object, then each program
that uses it can create an instance, and use the data in the exact same way.
Nothing needs to be recompiled, and every program starts to use the new copy
at the object level. The added data names are automatically initialized in
one place ... the object. One of the best possible examples of OOPs would be
for the replacement of working storage copy modules used throughout the
system.

That only covers methods and attributes.  When you get to inheritance, it
gets even better, and the payoff is even greater. Used wisely, OOP's has the
potential to *greatly* reduce maintenance and testing.  The real problem is
that people have been conned into thinking of it as a GUI tool, instead of a
Cobol tool.

You will see from the above examples that simple linked lists suffer from
some problems.  Most importantly, they use a variety of different pointers,
so that you need specific code to add and remove nodes in different
circumstances.  The various end-conditions make this code quite complex.
Further, you cannot scan the nodes backwards directly; there is only a
pointer to the next node.  To find the preceding node, you must scan from
the start of the list.  This makes simple operations such as removing a node
from the middle of the list very expensive.
Doubly-linked lists solve these problems, at the cost of slightly more
storage space.  A node in a doubly-linked list has two pointers; one
forwards and one backwards:

   to previous node <---- ? data ? ----> to next node
                       ??????????????

pointers, but also nodes (there is no list tail).  This has the effect of

    02  NEXT-PTR                PIC S9(3)  COMP.

    PERFORM ATTACH-NODE-TO-LIST
      VARYING NODE-NBR FROM FIRST-NODE BY 1
        UNTIL NODE-NBR > LAST-NODE
    .

ATTACH-NODE-TO-LIST.
    MOVE NODE-PREV (LIST-PTR) TO PREV-PTR
    MOVE LIST-PTR TO NODE-NEXT (NODE-NBR)
    MOVE NODE-NBR TO NODE-NEXT (PREV-PTR)
    MOVE PREV-PTR TO NODE-PREV (NODE-NBR)
    MOVE NODE-NBR TO NODE-PREV (LIST-PTR)
    .

Note that the paragraph ATTACH-NODE-TO-LIST is quite general, and will work
with either list, whether it is empty or not.  This generality is one of the
best reasons for using doubly-linked lists.
b. Adding data to a Doubly-linked List
An empty doubly-linked list looks like this:
    prev      next
    ??????????????
.----- ? main ?  ?<--.
`-->?  ? list ? -----'
    ??????????????

When one node is added, the doubly-linked list looks like this:
 .---------------------------------------.
 :  ??????????????      ??????????????   :
 `---- ? main ?  ?<------- ? node ?  ?<--'
.-->?  ? list ? ------->?  ?  A   ? ----.
:   ??????????????      ??????????????  :
`---------------------------------------'

When more nodes are added to the doubly-linked list, the list looks like
this:
 .----------------------------------------- - - -----------------.
 :  ??????????????      ??????????????          ??????????????   :
 `---- ? main ?  ?<------- ? node ?  ?<-- - - ---- ? node ?  ?<--'
.-->?  ? list ? ------->?  ?  A   ? ---- - - -->?  ?  Z   ? ----.
:   ??????????????      ??????????????          ??????????????  :
`------------------------------------- - - ---------------------'

Before you can attach a node to the list head, you must get a node from the
free pool.  To do this, take the node at NODE-NEXT (FREE-POOL) and remove it
from the free pool.  If there were no nodes left in the free pool, this will
have no effect (try it with pencil and paper).
GET-NODE-FROM-FREE-POOL.
    MOVE NODE-NEXT (FREE-POOL) TO NODE-NBR
    PERFORM REMOVE-NODE-FROM-LIST
    .

REMOVE-NODE-FROM-LIST.
    MOVE NODE-PREV (NODE-NBR) TO PREV-PTR
    MOVE NODE-NEXT (NODE-NBR) TO NEXT-PTR
    MOVE NEXT-PTR TO NODE-NEXT (PREV-PTR)
    MOVE PREV-PTR TO NODE-PREV (NEXT-PTR)
    .

After asking for a node from the free-pool, you should test that NODE-NBR is
not equal to FREE-POOL:
STORE-LINKED-LIST-RECORD.
    PERFORM GET-NODE-FROM-FREE-POOL
    IF NODE-NBR NOT = FREE-POOL
        MOVE LIST-HEAD TO LIST-PTR
        PERFORM ATTACH-NODE-TO-LIST
        MOVE THE-ITEM TO NODE-ITEM (NODE-NBR)
    .

It is also easy to add nodes into the middle of a doubly-linked list - if
you study the code for ATTACH-NODE-TO-LIST above, you will see that the
following code adds a node after any NODE-NBR:
    MOVE NODE-NEXT (NODE-NB) TO LIST-PTR
    PERFORM ATTACH-NODE-TO-LIST

c. Removing data from a Doubly-linked List
The general nature of doubly-linked lists makes it easy to remove any node
in any list.  If the node is at NODE-NBR, the following code removes it from
the list and returns it to the free pool:
    PERFORM REMOVE-NODE-FROM-LIST
    MOVE FREE-POOL TO LIST-PTR
    PERFORM ATTACH-NODE-TO-LIST

As well as adding or removing individual nodes, you can perform much more
complex operations on doubly-linked lists.  For example:
 ? To move a node from one place in the list to another, remove it and then
re-attach it where desired.
 ? To exchange two nodes, remove, then re-attach them both.
 ? To remove a sub-list of nodes (i.e. an entire chunk of the list), cut the
pointers at either end, and re-attach where desired.
 ? To exchange two sub-lists, cut both sets of pointers, and re-attach.
Clearly, you will need more than NEXT-PTR and PREV-PTR to perform such
operations.  Note however that at all times, we move and change pointers
rather than data.  This can be very efficient compared to a data structure
where large amounts of data are moved around.
d. Processing a Doubly-linked List
To process all nodes in a doubly-linked list, start with NODE-NEXT
(LIST-HEAD) and process each node in turn.  Note, that this loop will work
correctly even if the list is empty:
PROCESS-LINKED-LIST.
    MOVE NODE-NEXT (LIST-HEAD) TO NODE-NBR
    PERFORM DISPLAY-LINKED-LIST-ITEM
      UNTIL NODE-NBR = LIST-HEAD
    .

PROCESS-LINKED-LIST-NODE.
    MOVE NODE-ITEM (NODE-NBR) TO THE-ITEM
    ....
    MOVE NODE-NEXT (NODE-NBR) TO NODE-NBR
    .

e. Doubly-linked Lists Revisited
Picture the middle of a list:
        K          L          M
     ???????    ???????    ???????              ???????
     ?  L  ?    ?  M  ?    ?  N  ?              ?Next ?
---->???????--->???????--->???????-------->     ???????
     ?  J  ?    ?  K  ?    ?  L  ?              ?Prev.?
     ???????    ???????    ???????              ???????

The node L has a left node K and a right node M as neighbors.  The direction
of the list is determined by saying that M is the next node and K is the
previous node with respect to L.
Remove L by exchanging L,M and K,L as shown:
        K                  L                 M
     ???????    Swap    ???????           ???????
     ?  M  ?<---------->?  L  ?           ?  N  ?
     ???????            ???????    Swap   ???????
     ?  J  ?            ?  L  ?<--------->?  K  ?
     ???????            ???????           ???????

Now K and M are linked and L is left out in the cold.  Since the swap is
symmetric, we can insert L between K and M by performing exactly the same
swaps:
        K                  L                 M
     ???????    Swap    ???????           ???????
     ?  L  ?<---------->?  M  ?           ?  N  ?
     ???????            ???????    Swap   ???????
     ?  J  ?            ?  K  ?<--------->?  L  ?
     ???????            ???????           ???????

We are now back to where we started.  Thus removing and inserting a node are
actually the same operation:
 swap(next[left] , next[node])
 swap(prev[right], prev[node])

For this to work it is only required that nodes be initialized to point to
themselves.
How do we determine what is left and what is right?  For removal that is
straight forward:
 Remove: left = prev[node]
    right= next[node]

For insertion, it is even simpler because left and right are two adjacent
nodes between which we want the new node to go.  The only complication is
that we usually think of attaching the new node to a given one of the nodes
in the list.  There are now two possibilities for the other node: the
previous one or the next one.  If the other node is the next one, the new
node is inserted after the given node.  If the other node is the previous
node, the new one is insert before the given one.  Thus:
 Insert after: left = given node
    right= next[given node]

 Insert before: left = prev[given node]
    right= given node

With the following data definitions:
01  LIST-POINTERS.
    02 NODE-PTR                 PIC S9(3)  COMP.
    02 GIVEN-PTR                PIC S9(3)  COMP.
    02 LIST-PTR                 PIC S9(3)  COMP.
    02 LEFT-PTR                 PIC S9(3)  COMP.
    02 RIGHT-PTR                PIC S9(3)  COMP.
    02 SWAP-PTR1                PIC S9(3)  COMP.
    02 SWAP-PTR2                PIC S9(3)  COMP.

01  LIST-NODES.
    02  LIST-NODE                          OCCURS 100 TIMES.
        03  NODE-PREV           PIC S9(3)  COMP.
        03  NODE-NEXT           PIC S9(3)  COMP .
        03  NODE-ITEM           PIC X(30).


the code now becomes absolutely obvious:
SWAP-PTRS.
    PERFORM SWAP-NEXT-PTRS
    PERFORM SWAP-PREV-PTRS
    .

SWAP-NEXT-PTRS.
    MOVE NODE-NEXT (LEFT-PTR) TO SWAP-PTR1
    MOVE NODE-NEXT (NODE-PTR) TO SWAP-PTR2
    MOVE SWAP-PTR1 TO NODE-NEXT (NODE-PTR)
    MOVE SWAP-PTR2 TO NODE-NEXT (LEFT-PTR)
    .

SWAP-PREV-PTRS.
    MOVE NODE-PREV (RIGHT-PTR) TO SWAP-PTR1
    MOVE NODE-PREV (NODE-PTR)  TO SWAP-PTR2
    MOVE SWAP-PTR1 TO NODE-PREV (NODE-PTR)
    MOVE SWAP-PTR2 TO NODE-PREV (RIGHT-PTR)
    .

REMOVE-NODE.
    MOVE NODE-PREV (NODE-PTR) TO LEFT-PTR
    MOVE NODE-NEXT (NODE-PTR) TO RIGHT-PTR
    PERFORM SWAP-PTRS
    .

INSERT-AFTER-GIVEN.
    MOVE GIVEN-PTR TO LEFT-PTR
    MOVE NODE-NEXT (GIVEN-PTR) TO RIGHT-PTR
    PERFORM SWAP-PTRS
    .

INSERT-BEFORE-GIVEN.
    MOVE GIVEN-PTR TO RIGHT-PTR
    MOVE NODE-PREV (GIVEN-PTR) TO LEFT-PTR
    PERFORM SWAP-PTRS
    .

ATTACH-TO-LIST-AT-END.
    MOVE LIST-PTR TO GIVEN-PTR
    PERFORM INSERT-BEFORE-GIVEN
    .

ATTACH-TO-LIST-IN-FRONT.
    MOVE LIST-PTR TO GIVEN-PTR
    PERFORM INSERT-AFTER-GIVEN
    .

INIT-NODES.
    PERFORM INIT-NODE
      VARYING NODE-PTR FROM 1 BY 1
        UNTIL NODE-PTR > MAX-NODE
    .

INIT-NODE.
    MOVE NODE-PTR TO NODE-NEXT (NODE-PTR)
                     NODE-PREV (NODE-PTR)
    .

We have special procedures for INSERT and ATTACH because it is easy to get
it wrong: to attach at the end, you must insert before!
The list-head is just one of the nodes in the circular list.  It is only us
that attach special significance to it.
The advantage of initializing nodes to point to themselves is not only that
the remove/insert code becomes identical, but also that there is no garbage
floating around.  You can f. ex. tell if a node is currently in a list by
checking if NODE-PTR = NODE-NEXT (NODE-PTR).
The swap code can be made one line shorter by having only one temporary
pointer.  On most machines you do not gain anything because moving from one
subscripted variable to another subscripted variable involves storing the
value internally anyway as the registers are needed for the subscripting.
The above code is, of course, slightly less efficient than one that does
remove/insert as separate codes because it cannot take advantage of removal
of redundant code.  The remove/insert code is never time critical anyway.
For really die-hard speed freaks it is a purely mechanical procedure to move
the SWAP-PTRS paragraphs in-line and strike out redundant code.  This
process maintains the programmer's understanding of what is happening.
