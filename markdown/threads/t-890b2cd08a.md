```
Barticus@att.spam.net (Randall Bart) writes:
> Now,  Unisys says that it is incorrect to interpret those Zs as digit
> positions, they must therefore be insertion characters.  Therefore a check
> printing program (which was recompiled on Y2K implementation weekend)
> started printing amounts like *****23.ZZ.

This is clearly a priority A FTR, and I hope you've already
submitted it.

Also, this kind of sudden change violates Burroughs/Unisys'
long-standing committment on the A-Series to give two
releases of warnings for deimplementations. This is clearly
a deimplementation (of the interpretation of Z after
decimal point as 9). You should also make this clear in
your FTR.

If Unisys does not correct this promptly, please bring up
the issue with the UNITE A-Series Working Group officers
for referral to the UNITE Resolutions process. (I'm no
longer part of that, so you'll need to go through the
channel, which is now the working group officers.)

> This behavior is non-ANSI.  Any action other than a syntax error is a
> Unisys extension.  Changing the behavior to comply with ANSI is an
> acceptable change.  Changing bahavior of non-ANSI syntax to comply with
> Unisys standards is potentially acceptable.

Any change without adequate warning is unacceptable.

I think both Unisys interpretations are confusing. I prefer
that Unisys change it (WITH WARNING) to a syntax error.
There's an adequate substitute, it's non-standard, and it's
easy to convert -- even automatically.

Edward Reid
```
