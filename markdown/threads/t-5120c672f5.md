In support of the new PowerCOBOL software being released in May
(http://primacomputing.co.nz/primametro/pwrCOBmigration.aspx) I have been
looking to strengthen our download security and credibility.

This has involved the purchase of a code-signing certificate from a valid CA
(DigiCert).

It has been very interesting (but tiresome) acquiring the certificate (they
necessarily run very extensive checks to see that you are who you say you
are), installing the certificate, and then persuading Visual Studio and INNO
install (just 2 of the possible packages used to sign code) that the
certificate should be integrated in the build/compile process so that the
executables produced are code-signed.

The result is that you can download PRIMA software, extra secure in the
knowledge it is the right code, as produced by us, and not something that
has been substituted by a hacker... And Windows now agrees that the
Publisher is PRIMA, rather than "Unknown", which helps to inspire confidence
in people who are new to our products and services.

Before you all race off to try downloading stuff like COBDATA
(http://primacomputing.co.nz/primametro/newCOBData.aspx) please be aware
that I have not yet rolled code signing to EVERYTHING, (so far, it has taken
me 4 days to provide it for the PCOB2NET trials...) so it may be a little
while before all the PRIMA downloads are code-signed. For some years now,
PRIMA has been using Internet download as the primary delivery system for
our software and we are committed to using it in the future; for this reason
code-signing makes sense...

All well and good.

Except that today I tried to download the code-signed version of the new
PCOB2NET trial (see link above) to an XP machine.

It didn't work.

It USED to work for everything from XP - Win 10, but once it was
code-signed, XP spits the pacifier...

After some investigation and really tedious analysis (I guess I'm getting
old... :-)) I determined the cause of the problem (it has to do with the
type of hashing algorithm used to do the signing.)

I have built a specifc XP download that does work (I forced it to use the
old SHA1 algorithm which is now deprecated by MS), but it isn't entirely
satisfactory because it still reports "Publisher Unknown" when it actually
knows very well that the publisher is PRIMA...(In effect, I might as well
not have bothered to code-sign it at all...)

I don't want to invest more time in this, because I have other, more
pertinent things to do, but I started wondering how problematic it is likely
to be.

How many people (or what percentage of a prospective user base, if you
like...) are still using XP for production machines?

If you needed to modernize all your PowerCOBOL systems to .NET, would you
still expect to run them on XP, or would it be reasonable to expect to put
them on Win 7+ platforms?

I'll be interested (and thankful) for any comments you care to make.

Pete.
"I used to write COBOL...now I can do anything."

reassurance for both them and us, but it doesn't affect the execution of the
code.

› I do not think that the drivers held by MS will some how cut it.

What are you migrating?

Certainly very old hardware drivers may not run on newer hardware but that
is where a Virtual Machine can be a lifesaver.

Strongly recommend you have a look at Oracle Virtual Box.

It is free and it will run on your new hardware. You can set up a machine
with the same OS as your old one (XP for instance...) and your old drivers
will run happily in the VM.

You can run on your new machine and swith to the VM with a mouse click; it
is really well implemented.

If you are expecting to drive old, unsupported, hardware from your new
laptop, it is probably a forlorn hope, although you could certainly try it
and see if compatibility mode is any help...

You'll probably find you need to replace the offending hardware and the new
drivers that come with the new hardware will probably run on the new laptop.
