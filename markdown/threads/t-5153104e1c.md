Hi. It is safe to use these calls providing that the longjmp() does not
collapse the stack past a COBOL program.

(ie, if you have a C function 'A' which calls a COBOL program 'B' which calls
a C function 'C' which calls another C function 'D', then it's OK to setjmp()
in 'C' and longjmp() in 'D', but you can't setjmp() in 'A' and longjmp() from
'C' or 'D'. I hope that makes sense! :)).

The problem is that the COBOL RTS needs to know that the COBOL program has
become inactive (for example, if you never EXIT PROGRAM from the COBOL then
the RTS thinks it's still active and will give a recursion error when the
program is reentered (if it's not a recursive COBOL program!)). There may be
other reasons also (for example, the RTS may maintain a "COBOL stack" alongside
the system stack - especially when running .int code).

To get around this, recode your C functions to use the cobsavenv() and
coblongjmp() calls (they are fully documented), as then the RTS has knowlege
of what you are doing and can ensure the COBOL environment is kept sane. You
don't mention what OS or version of COBOL you are using, but these calls exist
in all Unix products and, I believe, in all 32-bit PC products.
