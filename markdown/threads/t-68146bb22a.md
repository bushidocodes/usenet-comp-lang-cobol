It is not clear to me that the 'method [for ascertaining the
numerical value of] an arithmetic expression' can be done
in any other than the specified order.

I was aware of 8.8.1.2 when I posted; but see a relation
between 'order of execution' and 'method of evaluating'.

Given, for example,
    COMPUTE F = A / B * C

What numerical value 'A / B' evaluates to is
implementor-defined; but that 'A / B' must be evaluated
before the effect of '* C' is 'order of execution.'

Now consider 8.8.1, below. 'A' is an arithmetic expression,
as is 'B' and 'C'. Also, the implicit '(A / B)', as required by
'order of execution', is an arithmetic expression, as is the
implicit (temp * C). Finally 'A / B * C' is an arithmetic
expression. To which 'arithmetic expression' does 8.8.1.2
refer? My understanding of 8.8.1 and 8.8.1.1 is that 8.8.1.2
applies to each arithmetic expression, in the order of execution,
and not merely to the complete arithmetic expression.

FDIS, page 120, 8.8.1 Arithmetic expressions
"An arithmetic expression may be an identifier referencing a
numeric data item, a numeric literal, the figurative constant
ZERO (ZEROS, ZEROES), such identifiers, figurative
constants, and literals separated by arithmetic operators,
two arithmetic expressions separated by an arithmetic
operator, or an arithmetic expression enclosed in parentheses.
Any arithmetic expression may be preceded by a unary
operator. The permissible combinations of identifiers, numeric
literals, arithmetic operators, and parentheses are given in
table 4, Combinations of symbols in arithmetic expressions.

"Evaluation rules for arithmetic expressions depend on
whether the mode of arithmetic in effect is native or standard."
