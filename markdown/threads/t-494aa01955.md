Matt,

this looks like it might be an Oracle-RTD application?

If it is, you can generate the Web Services in Java and set a listener to 
cause a callback on your COBOL code. (In effect, it is like raising an 
interrupt when data is posted by the service.)

If I am way off the beam and this is not your environment let me start 
again... :-)

What you are REALLY asking is: How can I collect data from a Web Service and 
give it to my COBOL application (which is not necessarily running on the web 
server - however it must be ACCESSIBLE from the web server)?

There are 2 possibilities, and the programming language really doesn't 
matter... you should use whatever you are most comfortable with and can do 
the job. If it were me, I'd use C# because (a) I already have a fair bit of 
experience at calling COBOL from C# and vice versa and (b) I enjoy doing 
tricky things in C#... YMMV.

1. You can use a mailbox approach with the web services posting data and the 
COBOL app. periodically checking the "mailbox". It doesn't have to be a real 
mailbox; could be a table on a shared DB, or even a file on the server. 
(This is kind of like "polling" and is for wimps and girls...)

2. You can trigger an event directly from the web service. (This is kind of 
like "interrupt driven" and is what real men do...)

I would take approach 2 every time.

However you need a proxy for your COBOL app. and you need some remoting and 
...Hang on a minute!

Proxy? Remote Procedure calls?

This is a job for.... ta! da!   http://www.xml-rpc.net/

...  or somethng like it... :-)

It is not difficult to do CGI in COBOL. I developed some systems using
this back in the 90s and some still run. It even does AJAX (which is
just another interaction on the server side).

Most of my stuff these days is in Python on the server side.

You will probably need to use Javascript on the client side. I use
jQuery which makes it easy to do browser based UIs without worrying
about which browser is being used.

The major issue is that you should be using templates to generate the
web pages. This makes it simple to change and tune the pages without
(usually) having to change the server code.

CGI, templating and other web based stuff is not difficult in COBOL
and does give the ability to interact directly with the existing
system including record locking.

I posted some of my COBOL code on Google groups:
