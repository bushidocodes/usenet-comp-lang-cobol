### The REPLACE stmt is invaluable

- **From:** "cwes..." <ua-author-13502721@usenetarchives.gap>
- **Date:** 1995-03-15T21:25:01+00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<1995Mar15.212546.11036@giant>`

```
In article <3ibsec$c.··.@ela··d.edu>, Vadim Maslov wrote:
› As to REPLACE statement, sane people have not been noticed using it, and I
› agree with COBOL-85 that it simply should be prohibited.

If sane people have not been noticed using the REPLACE statement it can only be
because the noticer in question is not looking.

The REPLACE statement is the only way there is to ensure that when you change
something, you automatically change it everywhere that needs changing. It is
invaluable alike in legacy systems and in systems that will be legacy systems
themselves someday.

For a simple example of what the REPLACE statement does for you, consider the
following fragments unearthed from a 10000-line paleolithic stratum:

01 WS-WORK-AREA.
05 WS-WORK-AREA-X PIC X(01) OCCURS 80
INDEXED BY WS-WORK-AREA-INDEX.

01 WS-WORK-AREA-SAVE PIC X(80).

77 WS-WORK-AREA-SIZE PIC S9(04) COMP VALUE +0080.

MOVE WS-WORK-AREA TO WS-WORK-AREA-SAVE.

SET WS-WORK-AREA-INDEX TO 1.
PERFORM 2560-PROCESS-CHAR
THRU 2560-PROCESS-CHAR-EXIT
WS-WORK-AREA-SIZE TIMES.

The unknown author was reasonably conscientious, as shown by the fact that the
repeat-count in the PERFORM is not hard coded. Nevertheless, when the size of
WS-WORK-AREA changes, the code must manually be changed in several places. At
that time, there was no better way. Our author has valiantly coded numerous
comments in the pious hope that the maintainer will read them; but the payoff
for this effort is not guaranteed.

Here are some fragments from the same species, found in a 12500-line neolithic
stratum:

* MANIFEST CONSTANTS

REPLACE
==WS-WORK-AREA-SIZE== BY ==132==
.

01 WS-WORK-AREA.
05 WS-WORK-AREA-X PIC X(01) OCCURS WS-WORK-AREA-SIZE
INDEXED BY WS-WORK-AREA-INDEX.

01 WS-WORK-AREA-SAVE PIC X(WS-WORK-AREA-SIZE).

MOVE WS-WORK-AREA TO WS-WORK-AREA-SAVE.

SET WS-WORK-AREA-INDEX TO 1.
PERFORM 2568-PROCESS-CHAR
THRU 2568-PROCESS-CHAR-EXIT
WS-WORK-AREA-SIZE TIMES.

When the X-terminals are finally installed and the work area size changes, just
change 132 to 138, and all the rest of the changes are made automatically.
There is no need for voluminous comments to the maintainer about where else the
change must be made. Note also that WS-WORK-AREA-SIZE will be in the
cross-reference, whereas "132" would not. As a final bonus, the code that uses
REPLACE is more efficient than the code that uses the hard-coded "80"s.

This is just the beginning. There's lots more that the REPLACE statement can do
for you. Sane people use the REPLACE statement in all new development, and in
their "spare time" they use it to simplify the maintenance of their legacy
systems.
```
