In this case, I would open Old-Master-File, Stock-Price-File,
New-Master-File and Print-File at the same time, and close them after the
last record was written. My decission is based upon the fact that you open
the indexed file for input and the new master and printfile for output. If
something goes wrong, you will have to rerun the job, and then the 2 output
files will be recreated.

When you have a match, you can move the stock no from the transaction file
to the record key and read the Stock-Price-File.

That brings me to the select-statement for the Stock-Price-File. I would
prefer something like this;
SELECT STOCK-PRICE-FILE  ASSIGN TO "PRICE.DAT"
                                            ORGANIZATION IS INDEXED
                                            ACCESS IS DYNAMIC
                                            RECORD KEY IS STOCK-NO
                                            FILE STATUS IS
STOCK-FILE-STATUS.

You will have to define Stock-File-Status and if your compiler is MF (bear
with me I cannot remember it) File Status looks like this:
01 STOCK-FILE-STATUS.
     05 STOCK-KEY-1                    PIC X.
     05 STOCK-KEY-2                    PIC X.
     05 STOCK-KEY-NUM   REDEFINES STOCK-KEY-2    PIC 99 COMP-X.

Well in short. If you read the stock file and Stock-File-Status = '00'
everything is fine, and you found a price. If not, you can handle the
situation according to the file status. That is much better than the
simplified 'invalid key'.

The way to read? move something to Stock-No (the record key) and say
READ STOCK-PRICE-FILE
END-READ
