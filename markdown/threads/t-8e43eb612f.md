The error is quite reasonable.  You have described it as 
numeric-edited, which would probably give strange answers according to 
standard COBOL.

The standard is a bit hazy in this area.  It says the data is just 
plunked into the receiver (sort of).  It also says any conversion 
necessary between the outside and the data item is defined by the 
implementor.  I assume MF edits the data assuming this is 
"conversion".  Many other implementations just plunk it in, so you 
would not get editing.  To have something portable you should accept 
it into an alphanumeric item, and then move that to the receiver using 
the NUMVAL function.  Like

     ACCEPT entry-field-an FROM CONSOLE
     MOVE FUNCTION NUMVAL (entry-field-an) TO entry-field  
```

I think that you will find that ANY (and all) ANSI conforming compilers will
accept PIC -ZZ. However, it is NOT the same thing as PIC ---.  If you
move -8 to a PIC -ZZ field, then you will get a hyphen followed by a space
followed by an 8.  While if you move it to a Pic --- field, you will get a
space followed by a hyphen followed by an 8.  The latter has a "floating"
hyphen (negative sign - called floating insertion) while the former has a
non-floating hyphen (negative sign - called fixed insertion).

If you find a compiler that CLAIMS to be ANSI (or ISO) conforming and it
does not accept a "-ZZ" picture clause, then report it to the vendor as
non-conforming.  (On the other hand, if you have an application where you
really do NOT want the negative sign to "float" - you probably have a fairly
unusual application in the first place.)

P.S.  I have verified this in the '85 Standard.  I haven't looked it up in
the '74 or '68 Standards, but I don't think that this is anything "new".
```
