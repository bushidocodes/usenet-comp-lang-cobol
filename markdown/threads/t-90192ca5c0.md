*ALL* of the CICS emulators are expensive.  Micro Focus has one that goes 
included with their MFEE product (but not their NetExpress product).  See:
   http://microfocus.com/products/mfeee/

Unikix is a common Unix emulator.  See:
  http://www.clerity.com/products/unikix/

Alchemy (formerly Fujitsu) has NeoKicks. see:
   http://www.netcobol.com/products/windows/neokicks.html

* * * *

Just having the translator won't get you very far at all. It creates CALLs to 
specific "CICS" (DFF...) routines passing hex values that tell the CICS 
environment what to do.  If you simply want to COMPILE your programs and don't 
expect to run them, then there is a solution in Micro Focus (but not OpenCOBOL). 
If you compile with the MF directive, then the "format 2, EXEC statement" is 
compiled and executed.  This creates a CALL statement passing all the data that 
follows the name of the called module and before the END-EXEC statement.

This is a relatively USELESS feature, UNLESS you are simply trying to compile 
source code.

NOTE:
  The new zCOBOL project is providing free support for (IBM Enterprise COBOL 
compatible) COBOL and EXEC CICS.  However, it is still a "work in progress" and 
my not provide you with the level of COBOL support you are looking for.  You 
might want to check it out at:
  http://www.zcobol.org/
```

JC wrote:

Well, that can be done easily with any of a number of scripting languages.

Of course, if you want the called programs to do something like what

> I am using Microfocus NetExpress and OpenCobol compiler for windows. I
> don't know whether only translator will help me compiling my Cobol-CICS

This is an excellent plan, as long as you realize the "next step" is
about 99.9% of the total effort, and "later" is likely measured in years.

> Could you please name some CICS emulators for Windows?

Well, there's Micro Focus Net Express / Micro Focus Server, with
Mainframe Transaction Option.

There's also the former Fujitsu product (which they spun off into a
separate company, if memory serves) that transforms CICS COBOL
applications into .NET apps.

> I can see IBM TxSeries. But it's costly.

Yes. That's because emulating CICS requires quite a lot of work.

CICS is a transaction monitor and application server. It provides a
large range of application features and APIs. The CICS operating
environment is not much like any of the environments native to
Windows. Transaction monitoring; keyed, managed file I/O; mainframe
terminal and printer I/O; screen maps; resource definitions...

It might be possible to adapt a handful of CICS applications that use
a very limited subset of CICS macros and APIs to use native Windows or
.NET facilities in a relatively short timeframe. But for the typical
application it'd be a major project.
```
