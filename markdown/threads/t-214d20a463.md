Well, if you are using the Cobol 85 standard compiler you can now use a class
test to make sure those fields contain numbers. In fact there are two ways to
check.

1: In the special names declare a class name called num which contains 0 thru 9.
In the working storage or record area make sure that the field is a PIC X field.
In the procedure division use the "IF field-name IS num THEN". This checks to
make sure that each character of "filed-name" is in the class name "num". Now if
you want to convert spaces to zeros you can do this by using the "INSPECT
field-name CONVERTING " " TO "0" ". Pretty neat huh, but it only works with a
COBOL 85 compiler.

2: Use the ANSI function NUMVAL to convert the number for you. The NUMVAL
function will take a number with spaces, decimal points, plus or minus signs and
convert it to the correct numerical representation. EX: COMPUTE num-field =
FUNCTION NUMVAL (field-name). Try it, it works really great!

Both AcuCOBOL and MicroFocus COBOL have the extension of level 78 to be
a named constant. E.g.:

78 QLM-MAX VALUE IS 60

01 QLM-TABLE.
03 QLM-ENTRY OCCURS QLM-MAX TIMES.
05 QLM-ITEM-1 ...

A named constant (78-level dataname) may appear in Working Storage or in
the File section. In the File section, it must appear before or between
FD's (i.e., it may not appear between the FD and its 01-level record).

Named constants may also have alphanumeric values:

78 PROG-NAME VALUE IS "J18306".

01 PROGRAM-NAME VALUE PROG-NAME PIC X(06).

It is my understanding that the clause "CONSTANT" will be introduced in
the next COBOL standard, and that it may apply to any elementary level
item (i.e., one with a PICTURE), e.g.:

01 GROUP-NAME.
03 ELEMENTARY-NAME CONSTANT "J18306" PIC X(06).

The big disadvantage of this approach is the constant cannot be used in
an OCCURS clause. I don't know if the standard even attempts to address
this latter capability.

I do not prefer the new standard's approach over the 78-level approach,
but standards are standards even if they don't do things as well as some
extant compilers do them.

Some comments.

VALIDATE is for things like screen input where the data has not
been checked and is not presumed good. There is also a need for the
exception handler which would be used for records which should be
good but are not. Rather than have a "victim" program abend after
500,000 records or have noticeable overhead for all the checking to
take care of a bug earlier in the system, this exception handler would
allow the programmer to take action such as ignoring the record with
the offending field and writing it out on an error log.

The USAGE CONSTANT which would be on level 77's and level 01's is a
way to tell a compiler that the following data can be a part of the
read only section of the executable program. In a transaction
processing environment there may be multiple copies of the working
area of the program but just one read-only part consisting
of machine instructions and literals. The USAGE CONSTANT
would explictly tell the compiler that the data area is
just a named literal. At least with IBM 370 compilers there is an
implicit INITIALIZE of all fields with VALUE clauses at the
beginning of the program and if a compiler is to support re-entrancy
this is necessary. USAGE CONSTANT is a way to cause less code to be
generated, not more. All checks would be done at compile time.
