The MOVE will cause a S0C7 not the class test.  Movement of invalid numeric data to a numeric field causes the S0C7.

:>Also asking if a PIC X field is numeric will give a false answer IF the field 
:>is signed because a sign always is alpha in a PIC X field so doing the numeric 
:>check on ARE-TEST won't work on a signed field or if DATO-NUM was packed.

> Why do you consider that to be false?

> It is a valid number.

Lets look at the data representation in EBCIDIC.

If  the number being looked at is +12345 then the EBCIDIC representation is in hex x'F1F2F3F4C5'.  Should the number be negative then the representation in hex is x'F1F2F3F4D5'.  In EBCIDIC the x'C5' equates to the letter E and the x'D5' equates to the letter N.  

If you put a display into the program to display AREA-TEST you would see 1234E or 1234N.  This same appearace is what is seen with a numeric check on AREA-TEST (the PIC X field).  Checking AREA-TEST-NUM would result in a valid numeric check assuming the data was valid.  If it wasn't valid the test would fail also if you haven't received the S0C7 in the previous MOVE.

I hope that clarifies the confusion.

Any such situation would be a bug in the compiler.

When used, NOT and the next key word define the class test to be executed

NUMERIC 
Identifier consists entirely of the characters 0 through 9, with or
without an operational sign. 


If its PICTURE does not contain an operational sign, the identifier being
tested is determined to be numeric only if the contents are numeric and an
operational sign is not present. 


If its PICTURE does contain an operational sign, the identifier being
