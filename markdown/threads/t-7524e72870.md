All in all, given the rigid format of ISBNs - not elegant but probably the
following is the simplest even though it looks wordy. A slight variation on
Pete's theme :-

*>-------------------------------------------

Program-id.  CheckISBN.

Working-storage section.

 01 n                       pic 9(03).
 01 input-ISBN       pic x(13).
 01 input-2 redefines input-ISBN.
    05 ISBN1          pic 9(01).
    05 Hyphen1       pic x.
    05 ISBN2          pic 9(03).
    05 Hyphen2       pic x.
    05 ISBN3          pic 9(05).
    05 Hyphen3       pic x.
    05 CheckDigit    pic 9(01).

 01 T1.
    05  filler pic x(13) value '1-234-56789-0'.
    05  filler pic x(13) value '1 234 56789 0'.
    05  filler pic x(13) value '1234567890'.
 01 T2 redefines T1.
    05 InputField occurs 3 pic x(13).

 procedure division.

 perform varying n from 1 by 1 until n > 3
     move InputField(n) to input-ISBN

     if (Hyphen1 = "-" )   and
        (Hyphen2 = "-" )   and
        (Hyphen2 = "-" )   and
        (ISBN1 is numeric) and
        (ISBN2 is numeric) and
        (ISBN3 is numeric) and
        (CheckDigit is numeric)
        display
        "OK so far - now we use Mod 11"
        perform VALIDATE-CHECK-DIGIT

     else display
         "Invalid ISBN - Enter as n-nnn-nnnnn-n"
     end-if

 End-perform
 .
 STOP RUN.

 VALIDATE-CHECK-DIGIT.

*> Left to the reader to figure out from Jerry's message !

Let me try and explain in plain English, how I arrived at this solution and
why it works...

The input string has a fixed format that REQUIRES hyphens in certain
positions. Therefore it must also require EXACTLY the right number of
characters between the hyphens. (You can look at the numbers or look at the
hyphens; they are the shadow of each other...). AND these characters MUST be
NUMERIC.

UNSTRING looks at a character string and breaks it up into defined fields,
using a pointer or a delimiter character. (Exactly the opposite of what
STRING does...). If you use a delimiter, it is discarded (you need to use a
pointer if you want it stored, or you can specify WITH DELIMITER)

One way (it is a bit lateral thinking) to determine that the hyphens are
always in the right place is to use them as delimiters in an UNSTRING, and
then make sure that we always have exactly as many characters as are
required by the format, in the unstrung fields.

This kills two birds with one stone. If a hyphen is misplaced, the correct
number of characters will not be placed in the field currently receiving
unstrung characters. It can only be too short or too long. If it is too
short, it will be padded with spaces. If it is too long it will be stored
OK, but the next hyphen will now be misplaced. The final field MUST be too
short as characters have been "used up" by the fields that were too long. So
either way, if ANY of the unstrung fields are of incorrect length, there
will be a space in at least one of them. If there is a space in one of them,
when we test the group for NUMERIC, it will fail. ONLY if EXACTLY the right
number of characters was unstrung into into each field defined for the
UNSTRING, will we have NO spaces in the fields. (i.e. the hyphens must have
been in the right place.)

So, using hyphen as the delimiter ensures that hyphens were present and that
they were in the right place. If you used anything else as a delimiter, you
would simply have a field too long and the succeeding fields would be empty
and therefore not numeric. (See above, unstrung fields are either exactly
the right length (correct number of characters between hyphens) or too long,
or too short, and in either of these cases one or more of the unstrung
fields will have one or more spaces in it.)

So UNSTRING has done a pretty good job of validating that hyphens were used
as delimiters and that the fields are of the correct length. But it doesn't
guarantee they are numeric. By unstringing into fields under a group level,
we can test the group for numeric and our problem is solved. If everything
unstrung correctly (and it can only do that if the hyphens are in the right
place) the fields under the group level will be full of characters. If even
ONE of these is NOT Numeric (it could be a space caused by a misplaced
hyphen (see above), or it could be a non numeric character introduced
erroneously into one of the "numbers", the check will fail. You tested it
and saw that this is so.

As a pedantic fuddy-duddy, it's "ISBN" not "ISBN number."

The ISBN is composed of 4 sets of characters, separated by three space or
dashes (preferred).

The first set is the country identifier and can be from 1 to 5 digits long
in the following fashion:
0-7
80-94
950-997
9980-9989
99900-99999

For example, 0, 1 = U.S. & U.K., 9974 = Uruguay, 99913 = Andorra, 965 =
Israel

The second set is the publisher identifier and can be from two to seven
digits long in the following fashion:
00-19
200-699
7000-8499
85000-89999
900000-949999
9500000-9999999

It will be seen that while there can be a publisher with the prefix 699,
there cannot be a publisher with the prefix 6991. Therefore, even with the
dashes omitted, one can easily tell the ISBN 0699102340 is published by
publisher "699."

The third set is the unique serial number assigned by the publisher for the
particular work.

The last set is the check-digit. It can be a digit from 0 through 9 or the
letter "X" computed in the following fashion:

Multiply each of the 1st 9 digits of the ISBN by the following corresponding
factors: 10, 9, 8, 7, 6, 5, 4, 3, 2. Sum each of these multiplications. This
sum, modulo 11, is the check-digit (if the difference is 10, the check-digit
is "X"). For example using the ISBN above
(0x10) + (6x9) + (9x8) + (9x7) + (1x6) + (0x5) + (2x4) + (3x3) + (4x2) =
0 + 54 + 72 + 63 + 6 + 0 + 8 + 9 + 8 = 220
220 Mod 11 = 0, which matches the last digit.

Point of all this is, the dashes are for human consumption. In a proper
world, you should remove all the dashes, squeeze the result to ten
characters and search, sort, compare, update, and all that stuff on the ten
character compacted ISBN (making sure the ISBN you have is correct based on
the check-digit computation). Then, when displaying for a human, put the
dashes back in the right place according to the schema above. We get the
dashes in the right place here with only a few hundred lines of code.

More information at:
http://www.isbn.org/standards/home/isbn/international/html/usm4.htm

also)) and vice-versa.
