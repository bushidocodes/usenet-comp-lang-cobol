I presume here that you really mean "9A", ie byte 1 = "9" and byte 2
has a decimal value of 065. This is the response for file locked.
You would seem to have LOCK MODE EXCLUSIVE either explicitly or
implicitly, and thus single user access to files which are then
used in turn by each user as required.

Alternately, on Netware files are implicitly single-user until
the 'sharable' flag is set on them. This is a pain (to me)
because I want to create files and have them sharable but it is
necessary to have the flag set.

› Kollisions between different users didn't happen. Now however,
› he saves to files on an NT Server. This server handles them in
…[5 more quoted lines elided]…
› the problem on his own but doesn't have an NT-Server...

My best guess is that on Netware the files were created without
the sharable flag. The Cobol has FILESHARE directive or LOCK MODE
AUTOMATIC/MANUAL but Netware forces exclusive access because the
sharable flag is not set.

WinNT work 'properly' by creating files that are sharable and thus
the program does not give a '9A' error status.

Solution: add LOCK MODE EXCLUSIVE to SELECT for those files that
need exclusive access.

Better still: make it work with file sharing and record locking.


Hope this helps
