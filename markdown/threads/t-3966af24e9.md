```
> I just want to write over top of the old record.

Do a READ WITH LOCK, if it does not exist then WRITE else REWRITE

If it is locked (by someone else) then wait for the lock to clear or
return with error to the user.
```

Belt and Braces - you could add file-status checks to the above. (I would).

fields with Alternate Keys built from Split Keys.

I had a program in which I stored information temporarily in a keyed 
vsam file and I used a process that did a random read for the record. 
If the record was found then i would rewrite, if it was not found, then 
I would write the record.

There is probably a way to attempt a write and look at the error code 
and decide if you have to do the rewrite.  Either way you need to learn 
to check the FILE STATUS to know if your write was successful.

When you set up the files you need the statement:

FILE STATUS is CHK or something like that.  CHK has to already exist in 
the working-storage section.  Your manual will probably have a list of 
what to look for in the FILE STATUS result field.  Your Variable could 
have 88 condition values so you can say things like
read blah-blah

01 check   pic xx
    88 no-record-found value ??
    88 record-found    value ??
.
.
.
   read blah-blah
   if no-record-found
       perform 200-write-routine
   else if record-found
       perform 300-rewrite-routine
   else
       perform 400-file-error-routine.

If you have a manual you might find some examples.  Some manuals are 
absolutely useless and others show you how set up some routines to 
process the file status.

We have some copybooks at work with these kinds of things in them, so I 
dont ever try to memorize this stuff.  My old manuals are presently 
holding up one end of the couch.  I dont konw if the syntax is right or 
  not, but the concept is valid.  Someone else may have some idea they 
think is better, but this will work.  There could be a faster way.

You have to open I-O and use an access method that supports rewrite.
