9/41 means corrupt (or some sort on inconsistent) index file. I presume 
this is an Indexed file.

By implication the index contains some reference to a record in the data 
file which cannot be found...COBOL 3.2 comes with a utility called REBUILD 
which rebuilds the index file from the plain data file and you can use 
this to 'correct' the file.

From experience common causes of 9/41s are premature program aborts or 
powering-off without exiting from the program (hence not actually CLOSEing 
the file and thus not guaranteeing that the file buffers were all written 
back). When thus has happened you may have lost data so although REBUILD 
will result in a usable file it is far better to ensure that the files do 
not become corrupt in the first place; and consider taking regular backups 
etc. to reduce any consequential losses if it does happen.

We think there is a limit on the number of deleted records that MF allows.
We had a similar problem on a large file.  We think the limit is c 64k which
is paltry nowadays.

To test this out, one of our guys wrote a file with 70000 records, and then
deleted 68000.  After this fhinfo showed the file had 67536 records.  We
reorganised the file and it then correctly showed 2000 records.  (BTW I've
just noticed that the wrong count is 65536 larger than the correct count.
This is 1 more than 64k.  Interesting but probably a red herring).

It seems the more you delete the more the indices get screwed up.  Sometimes
you can reorg and sometimes you have to rebuild the indices.

We reported this to Merant and they suggested using IDXFormat4 but that
didn't make any difference.  I don't think we've heard any more.

We're still trying to work out how to handle this.

BTW this was on HP Unix.

Regards
Rick
