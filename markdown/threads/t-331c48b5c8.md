Yo Magnus.

I had a similar problem some time back using MF Cobol om a
Lantastic network. The problem appeared to be due to disc buffering.

As the index file generally reads (reasonably) contigous records, much
of the file i-o happens in the buffer and may not make it onto the disc
for some time, whereas the data records tend to be not only larger (thus
exhausting buffer space faster) but also more scattered throughout the
file, therefore causing the buffer to be flushed out to disc and
refreshed much more frequently.

This could be the cause of your index corruption and the fact that you
can rebuild from stored data.

One possible solution could be to reduce the size of your cache. Another
would be to periodically close & re-open the data file. Either way will
slow you down but I don't see much alternative. Whichever, I don't think
this is so much an MF problem as a caching problem.

Good Luck

Fred Neale

Last year there was an extended discussion in the MictoFocus Compuserve
fourm about damaged ISAM files. How they occur, how to avoid, how to
fix and what responsibility MF should assume.

If I remember...

Corrupted index files occur most frequently on MS-DOS based networked
systems. Most occurances could be traced to users dropping thier
connection to the ISAM file without closing the file; usually by simply
turning off the user computer! There was a consistant set of failures
where this was not happening. Most of these could be solved by
adjusting network settings.

One of the simplest fixes was foe Win95 peer-to-peer: On the computer
where the ISAM files reside change change the "Typical role of this
machine" to "network server". (control panel, system, performance, file
system).

Once a file's inded was corrupted you are forced to create a new index.
You can go to a recent backup if you have one. You can try the MF
program REBUILD. What we have done is write a program that reads the
data portion of the damaged file (one byte at a time) find all valid
records, and write a new ISAM file.

The MF suggested solution was not to use the internal ISAM handler that
comes with out old compiler. Many users migrated mission critical
applications to UNIX using C-ISAM.

There was endless discussion about why MF should /should not fix a
problem that is being caused by the OS.

Charlie Goodman

Well, I can see this being SOMEWHAT a MF COBOL problem. When I design PC
based systems that run on a LAN, I make sure the files are not open for I-O
except in the few seconds during the time they are being updated. This
works great in a CA-Realia environment, and leads to a very stable system
with little or no file corruptions. However, MicroFocus COBOL is SO SLOW
opening and closing files on a LAN (or locally for that matter) that this
approach is not feasable. It makes the application terminally slow. The
only way to achieve acceptable performance is to do what this shop did
before I got here - open all the files up front and close them at the end.
We have many, many corruptions because of this, but using Micro Focus
COBOL, we apparently have no other choice.
