It might be an IBM thing - I've got the IBM OS/VS COBOL manual, and it lists 
all three (EXIT PROGRAM, STOP RUN, and GOBACK).

The manual says the EXIT PROGRAM statement is "Nonoperational" (!?) if placed 
in a "Main Program", one which is directly started by the operating system (as 
opposed to being started by being CALLed by another COBOL program).

The two we have always used are STOP RUN and GOBACK.  Both will end your 
program and return to the operating system, if the program is a "Main Program". 
If your program is a "Subprogram", started with a CALL from another COBOL 
program, a GOBACK will exit your program and return to the CALLing program, 
where a STOP RUN will **completely end this program and the entire set of 
programs that are running, returning you to the operating system**!!

This probably isn't what you want in a Subprogram.  :-)

This means that GOBACK works to end your program and return to either the 
operating system or to the CALLing program, whether your program is being run 
as a Main Program or as a Subprogram.  

So, GOBACK works in *every* instance to end this program, where STOP RUN only 
gives you what you want some of the time.

Q) Don't you know know whether a program is a Main Program or a Subprogram?

A) All of them can be Subprograms (at least on IBM mainframe -- I haven't tried 
this on other platforms).  You can take COBOL programs you wrote, intending to 
run them as Main Programs, and write a supervising COBOL program that runs (by 
CALLing) them in the appropriate sequence.  Without having to recompile them or 
change them in any way.  You do have to plan ahead, so that the file 
identifiers work together, so that files written are then read by the correct 
program.  On our mainframe, we specify the same "DD" name in the "SELECT 
file-name, ASSIGN TO ddname" statement.

I haven't run across many real applications for this, but one was back in the 
olden days when we still wrote bunches of report programs for systems (now we 
just give them access to the file and SAS or RAMIS reporting programs!).  Many 
of the report programs for this system reported the same data, just in 
different sequences, with different control breaks and sub-totals, and so on. 
Naturally, the same format wouldn't work, so we had 12-15 report programs that 
the customer could request 'nn' copies of any of the reports at any time.  We 
wrote a control program that ran the report programs.  It did this based on a 
control file that consisted of the report program name and the number of copies 
of the report desired.  This way, instead of having to have a variety of job 
setups for the different reports, we had one job setup that you fed control 
cards to.

I hoped to stay out of this, but there is so much misinformation 
going around that I thought I ought to state the facts.  First, as 
has been stated many times, GOBACK is not in the standard and is not 
likely to be added to the standard (it has never been seriously 
proposed to the standards groups, and most of the members feel it is 
rather grotesque).  It originated as an IBM extension a long time 
ago, and exists in lots of compilers (not mine, however).  According 
to the standard, EXIT PROGRAM is a no-op if it is specified in a main 
program.  Somebody said their compiler did not accept EXIT PROGRAM, 
so that compiler obviously does not conform to the standard.  STOP 
RUN stops the run unit or whatever you want to call it.  
Unfortunately, the simple expedient of EXIT PROGRAM immediately 
followed by STOP RUN is an illegal combination according to the 
standard.  The standard says that EXIT PROGRAM must be the last thing 
in a sentence (most compilers don't enforce this rule).  This means 
you have to put a period after the EXIT PROGRAM in the example above, 
which precludes it from being used in a delimited scope statement (IF 
... END-IF for example).  In the next standard, the period will not 
be required, so there is probably no reason to add GOBACK other than 
it avoids three additional words.

As far as I know, GOBACK is a workaround for the fact that EXIT
PROGRAM does not work in main programs.  You need to use STOP RUN
instead, and obviously someone complained that this was a pain.
Use GOBACK and your program returns to the caller, be it another
COBOL program, or the system.

There is, however, another way to do this, which is to structure
your program so that the EXIT PROGRAM paragraph is followed by a
paragraph containing STOP RUN.  If the EXIT PROGRAM 'fails', ie.
your program was run as a main, the STOP RUN will hit.  This is
more portable than using GOBACK, since the same problem will
hit you on other systems, where STOP RUN is also mandatory.  I
think this is the case for Micro-Focus (Unix).
