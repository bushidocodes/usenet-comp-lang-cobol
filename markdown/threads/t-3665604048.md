Every 3GL (COBOL , BASIC, FORTRAN, C) compiler I've ever seen has "automatic
allocation" based on the programmer defining datanames  (e.g.,
"WORKING-STORAGE SECTION") or or using certain intrinsic verbs and
functions. "How" it's done depends on the brand of compiler and the
operating system on which the application is to run.

Memory allocation is a lot like any other language element....in COBOL for
instance, 'ADD X Y GIVING Z'  means the same thing regardless of operating
system although the actual mechnism used is very much a function the
compiler brand and the target operating system.

But all quality compilers offer the ability to "tap in" to the (dynamic)
memory-allocation functions offered by the target operating system.

'C'  for example offers the 'malloc'  intrisinc to support dynamic memory
allocation.. but the actual mechanism used by a C compiler to implement
'malloc' depends on... you guessed it, the compiler publisher and the target
operating system.

Soooo... if you want to do dynamic memory allocation, you need to know your
operating system and your language compiler's mechanism for tapping into it.
But of course, one reason compilers exist at all is allow developers to
create applications WITHOUT the need to learn all this... that is, so
applications guys can be "applications guys" rather than "systems guys."

of grief when I had to do the Y2K conversion:  I mostly had to change

As others said, the difference is static allocation at compile time
vs. dynamic allocation at execution time. Classic C/Pascal did it
statically, same as Cobol, but provided high-visibility functions
(malloc) that allowed the programmer to control dynamic allocation. 85
Cobol implementations offered similar functions as extensions, but
they weren't used as frequently. The demos I posted here -- sorts and
trees -- used dynamic allocation extensively. 

OO languages such as C++ and OO Cobol make dynamic allocation
intrinsic to the language rather than a procedural call. They
automatically manage pointers to dynamically allocated objects rather
than depending on the programmer to do it. In the old days,
programmers found ways to screw up and step on their pointers, which
gave pointers a bad name. OO supposedly (not really) protects the
pointers from programmer error.

COBOL seems to have been pandering to the LCD until we reach a point where
programmers can't even distinguish between arithmetic and data

> Something I did pick-up on that you wrote, which I have never used -
> Invoke and End-Invoke - did that subtle scope-terminator tend to resolve
> problems with COM ?
>

I wish I could find the case where it did solve a problem that I had a
couple of years back, but I can't. It was subtle and resulted in the
parameter list to the object being considered incorrectly.

This led me to draw the following conclusions (all are totally one man's
opinion, but it works for me. I have very few problems getting COBOL to
work...)

1. Scope delimiters in general are a 'GOOD THING' (used in the sense of
'1066 and All That') in COBOL, no less in OO COBOL.
2. Passing parameters to objects is not the best way to keep a stable
interface that does not require maintenance. The fewer parameters in the
list, the better. If I am writing OO Classes and objects for my own use, I
minimise the parameters to be passed. Instead, I set what would have been
parameters, as properties. So, for me, using objects involves:

1. set the properties.
2. invoke the method.
3. check or get the properties back into the current process.

At the time I build the objects/classes I try to consider ALL possible
properties (key data fields) that might EVER be associated with this piece
of code.  (This is a useful part of the analysis and is fun to do...)

Example:

Xmas tree object.

Properties:

maxHeight
maxWeight
typeOfTree
ageOfTree
decorations (object reference to collection)

Methods

BringTreeHome
Decorate
Relocate
Recycle

(The above is not as comprehensive as I would do if it were for real, but I
know you like concrete examples (ever see a concrete Xmas tree <G>?) so it
will serve...)

Take it the tree has been created with 'New' or 'CreateObject' and we have
an object reference to it in 'XmasTree1'

Now, when I come to write the 'BringTreeHome' method, the height and weight
of the tree might be relevant (to see if it can fit in the car, for
instance)

instead of :   invoke XmasTree1 "BringTreeHome" USING maxHeight, maxWeight
end-invoke

I would use;

move weight to maxWeight of XmasTree1
move height  to maxHeight of  XmasTree1
invoke XmasTree1 "BringTreeHome" end-invoke

There could be a number of code modules that use the XmasTree object and
invoke the "BringTreeHome" method.

(The BringTreeHome method accesses details on the luggage capacity of the
car, checks height and weight of tree, checks if other properties are set,
initialises them if they are not. If the typeOfTree and ageOfTree properties
are set, it makes sure the tree meets these; if they are not, it sets them
after it has determined whether the tree fits in the car.  In other words,
the BringTreeHome method is written to be as "broad" as possible and to
consider all the factors that are likely to be important in bringing the
tree home. It is encapsulated and we really don't want to have to maintain
it, so we ensure it does what it does...If we later use a different car, the
car database will be updated and the method will still do exactly what it
always did, without amendment.)

Later, it becomes clear that under some circumstances, the tree we bring
home must be a real tree, in other cases it needn't be.

If the type of tree is added to the parameter list, the interface is changed
and every program that uses it must be recompiled.

I simply set the typeOfTree property in the modules that need it...

move weight to maxWeight of XmasTree1
move height  to maxHeight of  XmasTree1
if some-criteria-TRUE
   move 'real' to typeOfTree of XmasTree1
end-if
invoke XmasTree1 "BringTreeHome" end-invoke

The interface doesn't change, coding impact is minimised and so is
regression testing. Furthermore, the behaviour of the object has been
changed selectively, without violating encapsulation or changing the code in
the object/class.

The bottom line is that when thinking about Xmas (as a system), the process
of bringing home a tree is covered. This means that system design can be
thought about at a more abstract level without worrying about the details.

I find brainstorming objects (components, for me) as part of OO system
design to be amusing and entertaining. The above object could be used as a
"building block" for an overall "Hey, it's Xmas" system or a completely
different "Hey, let's have a pagan festival to clebrate the equinox" system,
or a "Let's practice making pine trees look pretty" system. The beauty of OO
is that objects are exactly that, objects.

      In-line Method Invocation

You can invoke a method in-line, as long as it has a returning 
parameter, by coding the object reference followed by the method and 
separated by two colons in place of an operand in a statement, for example:

move anAccount::"getBalance" to current-balance

This is equivalent to the following INVOKE statement:

invoke anAccount "getBalance" returning current-balance

      Invocation Using Object Properties

You can invoke get property and set property methods using object 
properties. Get and set property methods are methods that are either 
generated automatically by the Compiler in response to object property 
syntax in data definitions, or coded by you using special syntax in the 
Method-ID paragraph. To invoke a get or set property method, you use a 
statement such as:

move balance of anAccount to current-balance

This statement invokes the get property method balance of the account 
object referred to by the object reference anAccount.

For more details see the section /Get and Set Property Methods/ 
<http://supportline.microfocus.com/supportline/documentation/books/nx40/opmeth.htm#u040> 
in the chapter /Methods/.


    Conformance

Conformance is a quality that applies to interfaces. The term interface 
is used in two ways in OO COBOL:

    * Every object, whether factory or instance, is said to have an
      interface consisting of the data and methods that have been
      defined for the object in its class source element.
    * An interface can also be a collection of method prototypes,
      specifying some behavior.

Part of an object's interface (in the first sense) can be a set of 
methods that it provides because it implements an interface (in the 
second sense). Interfaces in the second sense are explained in more 
detail in the chapter /Interfaces/ 
<http://supportline.microfocus.com/supportline/documentation/books/nx40/opinte.htm>.

It is essential in OO programming that, when your program is executed, 
each object conforms to the interface that you defined for it. To help 
to ensure this, the Compiler provides conformance checking. In other 
words, as far as possible it ensures that your objects will conform at 
run-time to the interfaces you have described for them. If any 
conformance checks fail, the Compiler displays error messages. The more 
rigorously you use the OO COBOL syntax, the more thoroughly your program 
will be conformance checked.

Conformance checking relies on the existence of an external repository 
file, containing details of all the interfaces. The use of this file is 
controlled by the REPOSITORY Compiler directive, as follows:

    * To request that your program is conformance checked, compile it
      with the setting REPOSITORY (CHECKING ON)
    * To make sure information about the object interfaces in your
      program is added to the external repository, so that other
      programs can be conformance checked against it, compile with the
      setting REPOSITORY (UPDATE ON).
    * If you want both checking and update, compile with the setting
      REPOSITORY (UPDATE ON CHECKING ON).

Here are some examples of the kinds of checks included:

    * You invoke a method of an object of class A using a typed object
      reference. The Compiler checks that the method exists in the class
      and that all the using and returning parameters in the INVOKE
      statement match the parameters defined for the method in the class.
    * You define class B as inheriting from class A. Class A's factory
      object has two methods, and so does class B's. The Compiler checks
      that the methods in class B's factory object have the same names
      and same using and receiving data definitions as the methods in
      class A. (The methods in class B override the methods in class A.)
    * You define classes B and C as inheriting from class A, and you set
      a typed object reference for class B to a typed object reference
      for class A; the Compiler rejects this, because at compile time
      the typed object reference to class A might actually point to an
      object of class C. (You can get round this restriction by using an
      object view, which delays the checking until run time. For more
      information see the section /Object Views/
      <http://supportline.microfocus.com/supportline/documentation/books/nx40/opusob.htm#u005>.)


The Compiler uses the information in the external repository to perform 
conformance checking.

I re-read my previous posts and I think I reacted  bit too strongly. I don't
want to give the wrong impression here. Properties of objects are best
manipulated by the use of GET and SET methods (and I do that... I said so in
my post). But whether you use GET and SET or you don't, the fact that the
properties CAN be manipulated is what is important. If you use tongs to pass
the salad or you simply grab a handful of it, the result on your plate is
the same... <G>.

OO COBOL does not specifically provide for PUBLIC and PRIVATE properties,
but it does do so implicitly. (.NET OO COBOL allows methods to have these
attributes, and I think it may also allow properties to be described
specifically as such, but I'm not sure about that...)

The real point here is that you can pass parameters to methods, OR you can
manipulate the properties (directly in early bound code or using GET and SET
in both early and late bound OO COBOL).

I have found that there are certain advantages in minimising the number of
parameters passed to methods. (Ideally, zero...). The alternative of
manipulating object properties to change behaviour, is more flexible and
much lower maintenance.

This then has some beneficial effects on the design process. Objects
(classes, really) can be designed to include all the attributes that are
part of that object and all the methods that are likely to be applied to it,
rather than just considering the data and the process steps for the current
goal, as is usual in non-OO design. (Then we go back later and add more data
elements and more process steps and grizzle about what a pain "maintenance"
is...).

For me, considering what an object MIGHT need to do, is a fun exercise that
requires imagination and the ability to build scenarios. I use "paper
models" of objects before I ever start coding them. When the component
design is complete, I then go back to the problem at hand that originally
sparked the need to design the object, and make sure that the designed
component meets this need.(I have never had a case yet where the original
requirement was overlooked in designing an object, but there could always be
a first time...)  Having regained the focus for the problem solution, I then
move to the next component.This design process iterates until I am ready to
glue my objects together into the final solution.

I never think about maintaining these objects. If there was a flaw in the
design, and a future requirement comes up that pertains to that object and
was not considered in the original design of it, I scrap the whole object
and repeat the design process. (this is partly to reinforce where I went
wrong and partly as a kind of "punishment " for not doing it right the first
time... <G>) In just over five years, I have had to do this twice... If a
future requirement comes up that is related to that object but introduces
new properties and possibilities I would create a new "co-operation" object
that co-operates with the existing one.

For me, the essence of design is to make it simple, with non-complex but
comprehensive building blocks for applications.

The XmasTree example is a contrived one, but it serves to illustrate the
concept. There might be uses for Xmas trees that have nothing to do with
Xmas. The object should handle them. It manages this by considering the Xmas
Tree as an object and noting its physical characteristics  (deriving the
properties), then what you can physically do with a Xmas tree (deriving the
Methods).

Obviously, this is a very different approach from that which we
traditionally employ in application design. However, the two are not
completely mutually exclusive. The traditional methods are good for getting
an overall design. Things like the flow of documents and what happens to
them. I still use data flow analysis to derive all the data elements and
their source and disposition, for complex in-house processes. The overall
system flow becomes the basis for the top down Object design in large
systems.

In smaller systems it is sometimes enough to simply design and build the
objects.

Very often the same objects keep cropping up in different circumstances.

For example, I have a single component that handles anything to do with
people. It can interface to databases and it has properties like their
addresses, date of birth, phone number, hair and eye colour, distinguishing
marks, skills (can vary by DB attachment), etc. (there are around 60
properties to this component, some of which are simply DB attachments that
can be varied. NONE of them  (except the keys - Lastname and ID) are
mandatory. You can add a person with just a surname (the object assigns the
ID) and you can then update the information later. Duplicates are detected
and choices offered. Soundex is encoded automatically when a new person is
added.

This component currently is used in situations where the "people" are:
1. Clients.
2. Members.
3. Patients.
4. Prospects
5. Staff.
6. Subscribers.

All different application systems, all using exactly the same component to
deal with people. In the old days, it would have meant 6 different database
designs with 6 different sets of attributes, hard coded processes to add,
change, delete, search, retrieve, and configure the "people processing".

Nowadays when I am considering a new application, I don't even have to think
about the people processing. I can take it as read. All I have to do is
check that the component can handle the requirements of the new system... it
has never failed to, and I have never amended it since it was first written.

It can run on a Web Page or on a desktop, without amendment. (You need to
install the fujitsu runtime on your web server or point to it via a virtual
directory in the web server. Obviously, with .NET Cobol using managed code,
this requirement is removed.)

There are advantages in OO design and development that are just HUGE when
considered against traditional methods.

It is a tragedy that mainstream COBOL shops have failed to embrace them.

(this was in the days when Source and COBOL were still King). I worked on
one site in North London were it was "mandated". I refused to use it for a
particular job that required something more than batch processing and was
told my job was on the line. (I was a contractor). I said fine, and started
packing, (It was in the days when COBOL contracts where abundant), the
senior manager heard about it and, after hearing both sides gave me
dispensation for the specific programs I was writing. The manager who had
enforced it on me vowed to see me off the premises. (A year later they had a
party when he left and I signed his leaving card with the comment "I'm still
here..." Small victories are often the sweetest, but I digress...

One of the major criticisms about JSP was that it generated TERRIBLE COBOL.
It was full of GO TOs (very anti-structured for a product that was
Evangelical about structured programming...) and even used ALTER. (I liked
it actually, but I never let on...and it was the concept I liked not the
product...)

So, anyone who used JSP could quite legitimately say (after looking at the
generated code) : "Wow! This COBOL really sucks as a language. I could do
things in RPG much better than this."

In case I've lost you, the point was that your friend's experience of Java
was as a COBOL programmer viewing Java that was generated from COBOL. This
is no way to form an informed opinion about Java or anything else.

Warren, the fact that Java is good, in no way dimiinishes COBOL, and it is
silly to have a contest. They are both good for certain tasks.

Java is more successful because the proponents of it are more dynamic in
support of it (you don't find "infinite loop" threads in the Java forums
<G>), it is virtually free (as opposed to several thousand dollars for a
commercial COBOL compiler), has proliferated because it is pretty easy to
learn (provided you don't approach it using "COBOL think"), implements OO
(in my opinion, it does this even better than C++, but I don't profess to in
depth knowledge of C++; this opinion is based on admittedly superficial
reading and I will change it if evidence to the contrary becomes available),
and is virtually platform independent. (Quite a lucrative market is emerging
in writing JAVA games for WAP and 3G phones... I might even have a go at
this next year <G>)

So, irrespective of your friend's opinion, Warren, Java is successful and
looks like being with us for some time to come.

Procedural programming is on the way out (there are sound reasons for this)
and COBOL is a procedural language. A fantastic effort was made to provide
it with OO facilities, but these were not picked up by the COBOL community
at large, and it has,  largely, missed the boat.

Even OO languages like Java do not have unlimited lifespan, but it will be
longer than the languages which implement the procedural paradigm.

I have to be honest here: the main reason I am still using COBOL is because
it  (well, I should really only speak for the Fujitsu implementation of it,
as that is the only OO COBOL I have ever used) implements OO so well. (The
other reason is because I am attached to it after making my living from it
for 40 years...)
