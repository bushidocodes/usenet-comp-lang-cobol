```
Here's a question for all you fossils out there: path calls with sequential
processing of an IMS hiearchial database.

I've an IMS database named BILLS with the following segments, layed out in
hiearchial order:
A001
  A002
    A004
      A011

Each segment can have multiple occurances.
Each segment has one field as a unique key. The name of the key field
in segment A011 is A011-REFDES .

Suppose I wish to sequentially process the database for every occurance of
every A011 segment with a key value of 'RRR' and *only* those occurances. This
I could easily do with coding an qualified SSA in my DLI call:

	CALL 'CBLTDLI' USING DLI-GN
	                     BILLS-PCB
	                     A011-IO
	                     SSA-A011.

	            where SSA-A011 is declared as 'A011    (A011-REFDES= RRR)'
                    and RRR is the value of the key field I want.

BUT suppose I also want to grab information from other fields in the segments
above A011 in the hiearchy.  I could do this by making the DLI call a "path call":
	
	CALL 'CBLTDLI' USING DLI-GN
	                     BILLS-PCB
	                     A011-IO
	                     SSA-A001-P
	                     SSA-A002-P 
	                     SSA-A004-P 
	                     SSA-A011.

My problem is I can't figure out how to code an SSA to use the D command code
(for path call)  for sequential processing  i.e. withOUT coding a search argument
in the SSA.

The path-call SSAs examples I've seen all look like this:
	'A001    *D(A001-ITEMI1= ZZZ)'
	'A002    *D(A002-ITEMI2= AAA)'
	'A004    *D(A001-ITEMI4= BBB)'
  that is, they're set up for GU calls, not sequential GN calls. 

So my criteria are:

-sequential stepping through database, processing only lowest-segment occurances with a 
 specific key-field value
-load IO areas of higher-level segment occurances at the same time.

Can this be done? Anybody have an idea on how to set this up? Examples?



Please reply to the following address after removing "antispam" and
reversing "tom" in the following address:
antip23spam610@email.tom.com

or simply reply to this article in the ng, I'll see it.


Thanks,

Mike Striefel
Motorola SSTG
```
