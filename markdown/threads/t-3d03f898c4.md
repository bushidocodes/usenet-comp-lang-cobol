I don't know what environment this was taking place in but in CICS at least we do a full screen edit and don't lock the record until the update is to take place, to do otherwise would cause the file to lock (sorta).

Anyway I read the file, edit the fields on the map which in many cases have field dependent edits, then read the record again for update which locks it, rewrite the record with appropriate updates (which unlocks it) and go on my way.

While I don't have a single user environment it is not likely that two users would want to update the same record as each has their own little slice of the file that they are responsible for.  There are maybe 20 users doing updates (not all at the same time) and maybe 100 other users/programs doing inquiry.  I can't afford to lock a record for the time it takes to edit and update even if it is nano seconds as it has a tendency to cause weird things to happen as these requests pile up on each other as others jostle in line waiting their turn.

With that kind of attitude you shouldn't be programming multi-user systems,
IMHO.

1. How can you be sure that someone hadn't changed the record after you
read it the first time and before you read it again for updating?  (Yes,
you could compare the contents and retry if there was a difference, but the
user wouldn't be too happy having to re-do his job.)

2. If you really could garantee that two users would never try to update
the same record at the same time, how could a lock on the record level
cause the file to lock?

When updating files in systems like CICS, you should take advantage of the
builtin locking mechanisms.  If you need to lock multiple resources (very
common), just be sure (by program design) that each user lock them in the
same sequence, and dead-locks will be avoided.

While you are correct in principle, you are not correct in a real life situation.

Usually, the business transaction will be done using a pseudo-conversational design, which
means

Task 1 will display the current state of the record, and end, thus releasing all locks it
might have

Task 2 will read the user's input, validate it, and then try to do the updates to the
file, after reading the records with the UPDATE option, thus locking it.

If you haven't designed your own, private locking mechanism, there is no way to make sure
that the record is still the same as before, and CICS does not provide any solution for
this

I missed the beginning of this thread so I don't know what the original problem
was, but It may be helpful to point out that that when updating a VSAM file
using CICS, VSAM will lock not only the record but also the Control Interval
(CI) containing that record, until the update is committed.

The theory is that CICS is using VSAM Locally Shared Resource (LSR) buffer
pools (assuming that's the way it's been set up) to hold a copy of the CI in
storage. It means read requests can be satisfied much quicker than the time
taken for a 'real' I/O  but there must be some way of ensuring the in-storage
records agree with the ones on disk if an update takes place. A not-uncommon
problem in my experience occurs when two tasks both deadlock on different
records in the same CI.

This applies to CICS v4 and below. I believe the situation is different in
Transaction Server.
