```
    Following is a short example that should begin an outer transaction,
write some value into table, then start an inner transaction, again write a
row into table, commit inner transaction and  then rollback outer
transaction. As a result none of the two inserts should have written values
into table. What I am getting instead is that both rows are written, as if
rollback didn't exist.
    If I examine value of @@trancount variable that should show number of
levels of nested transactions I see that on the beginning of a program it is
already 1 and none of the subsequent 'exec sql begin transaction end-exec'
lines raises its value. But when first 'commit' is executed @@trancount is
lowered to 0 which means that no rollback can occur.
SampleTable is defined as
    SampleTable(numericValue int not null, stringValue char(50) not null)

      $set sql(dbman=odbc)
       working-storage section.
       exec sql begin declare section end-exec
       01  Numeric-Value           pic s9(05)  comp-5.
       01  String-value            pic x(50).
      *01  trancount   pic s9(09)  comp-5.
       exec sql end declare section end-exec

       procedure division.

           exec sql
               connect to 'SomeDatabase' user 'user'
           end-exec

      *    Check the value of @@trancount here with
      *    exec sql select @@trancount into :trancount end-exec
      *    Notice that it is already 1 althouth no transaction has been
      *    started yet

           exec sql
               begin transaction outer_transaction
           end-exec

      *    @@trancount still = 1

           exec sql
               insert into sampleTable
                   (numericValue, stringValue)
                   values
                   (1, 'First String')
           end-exec

           exec sql
               begin transaction inner_transaction
           end-exec

      *    @@trancount = 1

           exec sql
               insert into sampleTable
                   (NumericValue, StringValue)
                   values
                   (2, 'Second string')
           end-exec
           exec sql
               commit transaction inner_transaction
           end-exec

      *    After this point no rollback can occur since
      *    @@trancount now = 0

           exec sql
               rollback transaction outer_transaction
           end-exec

           @@trancount = 0

           exec sql
               disconnect current
           end-exec

           stop run
           .
```

```
Following is response from our ODBC/SQL Server support developer:

ODBC does not support nested transactions. OpenESQL accepts Toolkit syntax
but doesn't provide all the functionality that the Toolkit did because
it's using ODBC - it can't. It does provide compatibility with
non-nested transactions using Toolkit syntax, but that's as far as it goes.
```
