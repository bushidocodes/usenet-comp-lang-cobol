```
As has been discussed in this newsgroup on several occasions, the next
ANSI/ISO Standard will support (in most cases) a READ PREVIOUS statement
(similar to the current READ NEXT statement).  I know that several existing
implementations already support this (as an extension).  There is a current
J4 paper that is proposing a change (correction?) to the draft.  Can those
who have access to compilers that support this already (and who actually use
this feature), look at the following paper and tell me if you agree with what
it is doing (or not)?  I *think* it is reasonable, but I don't know if it is
or is not the way existing implementations really work.

Please check out:
  http://people.ne.mediaone.net/doncobol/99-0498.doc

and send me and/or J4 via
  doncobol <at> mediaone.net

if you have a comment indicating that there is a problem with this proposal.
```

```
I'm about to find out.  With Realia need to do a start before a read
"prior" (4.2) I found that to get the real prior record, took two read
priors (the way the draft is written <= current file pointer) where
Don want's to change it to <.  This will break some current programs
that do a read prior.

However, MicroFocus (Hence Don's bias because of experience) does NOT
suffer this "problem" when using read previous.  It behaves as he has
proposed.

I don't care either way, just so long as it's clear that if you want
to do a START and get the = record you must do a read next.  If you
then want to go backward, you do a read previous (prior) - and it
works the same in both compilers.

Now to say it clearly:

Start - then Read Prior in Realia returns a different record than
Start and Read Previous with MicroFocus.

However, BOTH behave the same if one does a 
Start, Read Next, Read (prior) Previous.

I do not know how Fujitsu behaves - but I'm converting some Realia
code to Fujitsu.  However, I started using the convention of Start,
Read Next, Read Prior (previous) some time ago and I expect the
programs to behave the same even of they don't treat the backward
reads immediately after a start or open the same.

PS - I find the implications of action after a START more important
than the implications of the action after an open.
```

```
Having used read previous in RM and ACU I agree with the proposal.  In

set via open *or* start. As proposed: previous = less than, next =
greater than or equal to.
```
