```
It occurs to me that you might think I'm arguing for the
adoption of OOD for all development.  This is not the case -- 
there are many situations where a "flat" design fits the
requirements well.  It entirely depends upon the project, but
generally, the benefits of OOD/OOP exceed the costs as the size
of the project increases.  When you have a lot of logic to
capture, encapsulation, inheritance and even componentization
can help to keep your algorithms concise.

A good example that comes to mind, is in handling Explanations
of Benefits (EOBs) provided by insurance carriers.  There are
hundreds (probably thousands) of formats that contain a
consistent subset of information that is of interest to a
medical claims application.

You can write all of these parsing/interpretation routines in a
piece of flat code, but it soon will become a behemoth.  Also,
the superposition of code can make these large "case statement"
routines difficult to analyze.

If you write it with OOD/OOP, you can devise a base class that
exposes the critical information you need, but leave the
details of that extraction to various subclasses.  The added
weight of that infrastructure will, at first, consume the
savings brought by inheritance.  As you add more carriers to
the mix, however, the design will help to keep your development
focussed and concise.

This may sound like an abstraction, but it's not -- I've done
this and am convinced it's a good idea.
```

```
In some implementations, perhaps, but not in all.  

In implementations like Unisys MCP/AS (ex-Burroughs-large-System,
ex-Unisys-A-Series) that have hardware memory protection (and have had
for forty years and more) even if subscript and index range checking
is turned off you will still get "bitten" on any attempt to write
beyond the end of the allocated memory space, or on any attempt to
overwrite anything in memory that hasn't been predefined as data
(whether the particular program "owns" it or not).

What a subscript bounds check doesn't catch a fatal memory protection
interrupt of some sort will.

True, some of our compilers "pool" data for a given program instead of
allocating separate memory spaces for each 01-level record, so it's
possible for a program to reference outside of a particular *record*,
but getting beyond the data spaces as they were originally allocated
for the COBOL run unit is well-nigh impossible.  Code, descriptors,
program control words and the like require special machine
instructions not generated by the COBOL compilers for manipulation and
are thereby protected by the hardware as well.
```
