That would be the case if you could only sort a file in Cobol. But it is
possible to use input procedures and output procedures, and that makes
the sorting procedure in Cobol far more flexible. 

As an example you can write a input procedure that checks if field-a
contains X, field-b = 100 to 200, and field-c at the same time does not
contain Z. Only if these conditions are met, field-d is multiplied by 4,
and the record is released to sort.
 
You can't do something like this in a 'normal' Operating System sort
procedure, other than by means of temporary files.

Hi, Mark.

In-line COBOL SORT is just great, and I see very few reasons why not to use
it.
COBOL SORT, however, never will remove the need for JCL utility sort, like
the JCL sort never will make COBOL SORT redundant. They both are
tools/components of great value to simplify system implementation.
When using COBOL SORT, I certainly will recommend the
INPUT-PROCEDURE/OUPUT-PROCEDURE option, in order to gain a nice sructure and
flexibility.

The reason why you don't find it in many IBM shops is pure historical: In
earlier days  COBOL didn't support the sort, and when it came along, there
could be tremendous overhead. Before that (in the punched-card era)  it was
natural to hand card decks over to the 084 sorting machine before program
processing on the 1401 and early IBM/360 systems. Then came JCL, and with it
the sort utilities. And by tradition, the "file-sorting-by-utility"-paradigm
survived.

The main reason why many IBM-shops do not utilise COBOL SORT, is lack of
competence.
That is, they never got a programmer to demonstrate that it works excellent,
because programmers also may  to stick to the "old and safe".

Today there is no overhead using COBOL SORT. It is exactly the same thing as
Utility Sort.
You should, however, consider the SELECT statement for the SD file (this is
the sort work file) by assigning more physical files  if input is big and
presequence is random. This resembles the common  JCL practice to establish
different predefined sort procedures in the JCL procedure library for
different file number and sizes.

The statement "more difficult system maintenance with COBOL SORT" is
bullshit: Any time the sort sequence should be altered, the corresponding
COBOL program also should be altered, and any time file structure alters,
both COBOL and any utility should be altered.
So, when getting down to maintenance, it's obvious that maintaining one
entity occurrence in one language is more easy than
maintaining both the COBOL and the JCL occurencies.

After having said this, though, I must comment on the advantages using
JCL-sort: If you study the DFSORT manual, you will find really easy ways to
handle tasks like inclusion/omit/reformatting of records  before/after
releasing to sort, removal of duplicates, totaling etc. In fact, the utility
could be utilized much more extensively than we normally see, in order to
eliminate many of the numerous  trivial and more or less "redundant"
"read-add-report" COBOL programs that float around.

Good luck
Bjorn

I never have got deep into analyzing the decision factors for choosing
either JCL- or inline COBOL SORT. So may be this is the time to give the
topic some intuitive thoughts.
(Before I proceed, let me just remind you that the SORT discussion also
applies to the  MERGE facilities available both as JCL utility and inline
COBOL).

One big difference is intepreting vs compilation. When using COBOL, you have
to code all the nessecary COBOL code and compile in addition to JCL
preparation. Utility sort inteprets the sort statements, and is ready to
use. Therefore, never use COBOL unless there is more to the function than
just sorting.

File and data shuffling: Very often you will see jobs with a huge mix of
utility sorts and very tiny  COBOL programs (performing record reformatting,
some kind of input filtering, simple addition, building sum files for
different control levels of the input sequence etc) These jobs, or parts of
them, often can be replaced by a few (maybe only one) utility sort step. So,
if the main function for a job is to produce a reformatted/condensed/summed
file and no tough logic shall be inposed on the data, I certainly would
avoid COBOL (or any programming language)

Function consistency: Breaking main business function down to series of job
steps (programs) can reduce the understanding of what is going on, it can be
unsecure to run and recover, and it can be a burden to maintain. Framing the
function into fewer  programs can enhance maintainance, security and
turn-around.

When (multiple) sequential processing AND complex logic (more than what
utility sort can offer) is involved, it is the time for considering use of
internal sort. There are lots of examples: Preprocessors, compilers
(wouldn't use COBOL, though), production of  cross-reference listings,
assembly and processing of data elements from multiple aind different
pre-sorted registers etc etc.

Table handling: Complex internal table processing can be eliminated by
combining internal Sort and ordinary matching/break-level logic.

> What do factors do you consider when deciding to use a JCL sort vs.
> internal cobol sorts?
>

#1.  Can what I want to be done be done with an external sort?  If not,
then it is internal.

#2.  I find KISS means more small steps.  This means external 90% of the
time.

#3.  Look at your environment - how difficult it is to migrate JCL vs
COBOL, how much testing will be involved, etc.

#4.  Do you have variable needs?  I have seen programs with variable
internal sorts.  On the other hand, I have seen 4 different external
sorts (with extract parameters) feeding the same COBOL program 4
different times for 4 different customers.

The INTERNAL SORT Issue went thru several phases.

At first the machines were too small to use Internal Sorts.
It wasn't unusual to find IBM360s with 64K, for operating system, and
programs.
256K was considered a hefty machine.
You also had to code SORT Exits. If I remember correctly, the INTERNAL SORT
didn't arrive until after the IBM370 came out.

As machines got larger, I/O was a bottle neck. It was more efficient to use
an INTERNAL
SORT to reduce the I/O overhead. The Sort Utility did not support
reformatting (extracting), while the internal sort did. The ability to
eliminate writing Sort Output, and then read it was considered to be a big
plus.

Now we have larger, and faster machines, and a more functional Sort Utility.
I/O is not a problem.
Now the trend is to not use the Internal Sort because it is cheaper to code
Sort Utility statements, than it is to code the equivalent Internal Sort.
