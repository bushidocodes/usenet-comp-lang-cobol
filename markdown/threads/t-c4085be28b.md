I have a file (this is on MVS) that contains a numeric field S9(9) SIGN
LEADING SEPARATE.
I want to sort this file. But, in the output I want to reformat this file to
get a S9(9) COMP-3 field.
We use SYNCSORT and it does Packed -to-Edited conversion but not the other
way around (at least my manual doesn't say how).

I check IEBGENR and found that it has some syntax to convert fields. I tried
the "ZD" (zoned-to-decimal??) option. It does not convert the signed field
properly.

Any utility I can use? It must be something that is always available on the
mainframe, because this will be part of a product that we market.

Ofcourse, I can always write a trivial COBOL program, but have my reasons
for trying to avoid that route if possible.

Any help will be much appreciated,

- Darius Cooper
```

In article <79it35$dh5$1@news.megsinet.net>,
"Darius Cooper" <Darius_cooper@Bigfoot.com> wrote:
>
> I have a file (this is on MVS) that contains a numeric field S9(9) SIGN
> LEADING SEPARATE.

That's ten bytes, and each comparison takes several instructions.


> I want to sort this file.  But, in the output I want to reformat this
> file to get a S9(9) COMP-3 field.

That's five bytes, and each comparison takes only one instruction. 

Therefore, reformat in the input, before releasing the record to the
sort, not in the output!


> We use SYNCSORT and it does Packed-to-Edited conversion but not the
> other way around (at least my manual doesn't say how).

You're looking at the output side.  If it exists at all, it's on the
input side.


> I check IEBGENR and found that it has some syntax to convert fields.  I
> tried the "ZD" (zoned-to-decimal??) option.  It does not convert the
> signed field properly.

That's correct.  SIGN LEADING SEPARATE is not the same as ZONED DECIMAL,
which has the sign overpunched on the rightmost digit.


> Any utility I can use?  It must be something that is always available
> on the mainframe, because this will be part of a product that we market.
>
> Of course, I can always write a trivial COBOL program, but have my
> reasons for trying to avoid that route if possible.

Too bad, because that's the way to go.

When you use a reformat utility, you must manually analyze your copy
books to prepare the reformat control cards, and you must somehow ensure
that you modify the control cards every time you change the copy books. 
When you use CoBOL, the compiler analyzes the copy books for you, and all
you have to do is recompile when they change. 

When you use a reformat utility, you must make two passes over the entire
input file: one for the reformating and one for the sort.  When you use
CoBOL, you make only one pass over the input file and reformat in memory
on the fly before you release the record to SYNCSORT.
```
