Good news and bad news -

Good news - The current IBM COBOL compiler *definitely* allows for input
with and without a decimal point.  What results were you getting when using
input without it?  This may be a "product error" in your service level and
you should contact IBM support if you are still having problems with this.

Bad news - Any ANSI conforming COBOL compiler (which includes the IBM
compiler) is allowed to "choke" on invalid input.  This includes fields that
have alphabet characters in it. Therefore, you need to "pre-check" your
input data before using the NUMVAL intrinsic function.

The exact rules (what you need to check for) for input to the NUMVAL
intrinsic function are found at:
 http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igylr205/7.1.30?

You need to look at the syntax diagram and figure out exactly what
combination of sign, space, digit, and decimal points are allowed.

P.S. If you are a member of SHARE or work with your shop's "system
programmer" who knows how to submit a "REQUEST" to IBM, you might want to
ask for "early" support of the TEST-NUMVAL intrinsic function which is
currently scheduled to be included in the next COBOL Standard.  This will do
the "checking" for you.

Fascinating!

I found this very interesting and quite an ingenious solution to the problem
of testing if a field is numeric.

However, since COBOL has provided a facility to test the class of a field
(NUMERIC, ALPHABETIC, and ALPHANUMERIC) since COBOL '59, I have to ask why a
COBOL programmer would not simply say: "IF NUMERIC..."

Then memory jogged and I remembered that on IBM mainframes, external decimal
fields could have the sign as an alphabetic character (a zoned overpunch
from unit record days).  The COBOL Class test (in IBM environments)
recognises this and allows certain non-numeric characters to be interpreted
as signs. So if an alphabetic character just happened to appear in the input
by accident in the right place,  the field would still be considered
numeric...

We used to get around this by stating:

IF NUMERIC AND POSITIVE...

I believe the example Peter posted is designed to avoid the same problem. It
just involves more work and is less "COBOL like".

Bill recently requested code for testing NUMVAL input and this should take
care of just about anything including +, -, decimals etc.

As I wrote a component in Fujitsu PowerCOBOL to do this nearly a year ago, I
didn't respond to Bill's request. It is unlikely that I will use NUMVAL as
it is easier to simply invoke "String2num", which does the combined
functions of the proposed TEST-NUMVAL and NUMVAL itself.

You pass the component a string when you invoke its "checknum" Method, and
it sets a Boolean indicating whether the string is numeric or not. If it IS
numeric, it also sets around 8 different object properties which give you
the number in every conceivable representation, including PACKED, ZONED,
FLOATING POINT, BINARY, and with the integer and the decimal parts as
separate fields. Other properties tell you how many characters the field is,
and how many of these are after the decimal point.

The component has other methods which can notify the User via a GUI message
box that the input is invalid, and give some clues as to what is required.

I have it packaged as an ActiveX control, and simply drag and drop its icon
onto any forms I am developing which require the input of numeric data. I
haven't tried using it on a Web Page yet, but I expect it to work just like
any other component or Java Bean.

This covers the whole story on Numeric Validation as far as I am concerned.

It also demonstrates what I have been saying for some time now about the
advent of components. You focus on a problem, solve it ONCE, and that's
it... You never need to worry about it again.

Object Oriented COBOL facilitates this, and that's the main reason I use it.

I have a couple of similar components that do EVERYTHING with dates
(including validation and computation). One of them is used for inputting
dates. It offers a choice between keyboard or mouse whenever the user moves
to a DATE input field, then remembers what was chosen. Any subsequent fields
will pop up a calendar, so dates can be entered by point-and-click if that
was the user's choice, or it allows DDMM to be keyed, (with a default to the
current year that can be overridden) if quick keying is desired. All the
problems regarding different date formats are taken care of. Dates are
displayed on screen as DD-MMM-YYYY (in the user's system  selected language)
but are available internally in any of 7 different formats, including the 64
bit floating point format used by Microsoft.

This trend of isolating and encapsulating functionality so you don't need to
worry about it any more is like providing extensions to the Language. I
don't think in terms of NUMVAL, or numeric validation, or what format I need
to have a number in; I simply think "String2num" and know that all my
requirements are met, every time.

Furthermore I know that ONE copy of this code will be used across my entire
system (it is a registered component) and if I ever need to extend it, I can
do so in ONE place, with ONE set of source code.

It goes way beyond User Functions.

If anyone would like more information and/or a copy of the components
described, mail me privately. (For non-Fujitsu Users, it is a fairly large
download, as the Fujitsu RTS has to be included...it still works.)

Pete.

Of course NUMVAL has been around since '89 - but there are a couple of
"nice" things that it does that Pete's routine may (or may not) do - but
NumVal does it in a "portable" way.

A) it allows any number of spaces at the beginning or end of the field.
This is "handy" when dealing with user input that hasn't already been
"normalized".

B) It handles + - CR and DB - with + and - allowed at either end of the
input (again handy if you have "unreliable" user input that you haven't
already normalized)

C) it "justifies" the decimal point for you.  You don't need to know how
many (if any) digits left/right of the decimal point their are in the input.

  ***

Having said all of this - the ORIGINAL "draft" Intrinsic Function module had
an (awful) solution to invalid input.  ANSI/ISO "removed" this and promised
a "quick" solution to the problem.  It should NOW be available in the
Standard - by the end of 2002 <G> or sigh
