3- The assumption behind the design of Cobol was that non-programmer
managers would be able to read the Cobol code, and to validate it.
Ie. to UNDERSTAND it, and to be able to decide whether the code
implemented their business rules or not. The great lesson of Cobol,
is that non-programmer, whatever their job and their level, are
incapable of such a feat. Unfortunately this lession is not learned,
and this error is repeatitively repeated all the time, even 70 years
after the invention of Cobol. (eg. Cucumber assumes that
non-programmer users will be able to specify formal use cases
(ie. tests)).

4- Cobol is designed to be used by non-programmer managers. Python is
designed to be used by non-programmer newbies. PHP is designed to be
used by non-programmer web designers. etc. Most programming
languages are designed for non-programmers or half-programmers. But
in the end, non-programmers are not programmers, and newbie
programmers or half-programmers don't stay that long. Unfortunately,
professional programmers are left with maintaining all those legacy
code bases written in those bullshit programming languages.

If you plan to become a professional programmer, learn a real
industrial-strengh programming language, with a standard, and that don't
get in your way (for example, Common Lisp, Eiffel, Ada, etc).
