I have no Casegen experience and have only started to play around with
Fujitsu COBOL, but I can shed a bit of light on the VB stuff.

The "form" is the window or panel on which the other controls will
reside. Controls are essentially everything else - labels (for
displaying non-changable information), input or combo boxes (for
displaying changable information or accepting input), radio buttons
(for selecting one and only one of a group of choices), flag boxes for
on/off type selections, and many more that aren't used as much.

You probably need to do the Casegen part first to build the form (or
forms) and put the controls on that you need. That will probably
create copy books for inclusion in your COBOL program.

In the VB world, you "paint" the forms and controls and then hang
BASIC code "behind" the forms and controls. So I would assume the
Casegen/COBOL system would be similar, except the COBOL is completely
separate from the forms. I took a VB course last year because I
thought I wanted to do some stuff at home in it. I was good enough
that by the end of the semester the instructor offered me a job. But,
I really didn't like thinking in that strange of a paradigm and have
yet to do anything at home with it. It was just too PC-weenie for an
old batch oriented COBOL guy.

Good luck,

Jay Moseley, CCP
http://jaymoseley.jumpcomputers.com/

I'm pretty much in the same boat as you are.  20+ years of mainframe
batch processing with big programs handling all file handling logic, 
input editing , file updating, and report creation.
It seems that in these PC/GUI programs, it works best to have the 
programs broken into User Interface(screens/forms), Business Logic
(What to do with the responses from the user), I/O Logic(file handling).
As for the CaseGen thing, I did not get very far myself.  Didn't have much 
time to exercise the full capabilities, the way that I have with using Fujitsu
for developing/testing mainframe programs.

I have found that the concept is pretty consistent across VB/Delphi/CaseGen et
al
that you start with a form/window upon which you drop all the controls
(buttons,checkbox,labels,textbox...) and then code little mini programs that
run for each 'event' that you want to respond to.  The fact that you can tie a
major batch process to a button-click is just another 'event' for the program 
to respond to.

This is the closest approximation that I have been able to come up with
in order tohelp me get going with any of this.  I'd be curious to see what 
you are able to come up with using CaseGen.  Is the EXE distributable?
How much bagage (support DLLs and Runtimes) have to be sent along 
with the program.

I've been toying with a menuing idea where the menu items are somewhat
customizable.  Basically read a control file and place the items on the main
window as clickable items linked to an associated sub-program also provided
in the control file.
