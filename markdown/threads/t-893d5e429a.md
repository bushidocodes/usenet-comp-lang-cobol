```
Background - Report Writer is an optional module in the current
ANSI, ISO, and FIPS standards. Therefore, an implementor does
not need to provide support for it - while still claiming to be
"HIGH" subset. In the early/mid-80's when IBM was first
delivering VS COBOL II, there beta-sites and user-advisory-sites
all indicated that they didn't have any RW code. Since then,
both IBM and most sites have discovered otherwise.

If all goes as planned, Report Writer will be REQUIRED in the
next COBOL Standard.

Although not well advertized, IBM *does* fully support Report
Writer with both VS COBOL II and COBOL for MVS & VM (renamed
version of COBOL/370). To get support for this, you need to
contact your IBM rep and ask them about IBM product 5798-DYR.
(This product can be used as a conversion tool - but produces
pretty "ugly" code that way. However, when run as an integrated
preprocessor it "looks like" it still support several dialects of
native COBOL report writer code.)
```

```
This doesn't surprise me.

At one place where I worked, every few years management would get all hot
and excited about the latest program generating tool to let programmers
write their (COBOL) programs in a fraction of the time.

Every one of them invented their own macros to support "quick and easy"
generation of reports. I'd point out that COBOL already had a report
writer feature that was more powerful, easier to use, and supported by the
vendor. The programmers would say "It does? Well, I never heard of it, so
it must be really complicated. With I don't have to
learn any arcane features of COBOL." I couldn't even get them to look in
the manual! If they weren't already using it, it must be no good.

I don't think anybody there ever used the Report Writer feature. (Except
me, but I only used it once or twice. Most of my programming was in
Algol.)
```
