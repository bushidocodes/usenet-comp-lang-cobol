I've been messing with learning OO COBOL for a little while now, and
there are some interesting possibilities.  It's more that you can
formalize a lot of things that you used to have to make up depending on
your platform.  For example, I code on the HP 3000 primarily, and HP
while having a pretty skimpy COBOL implementation, does have a very
nifty macro facility that expands the code inline during the compile. 
You can pass parameters and all sorts of things.  I've been using these
as make shift objects for years.

Say you have a bank finance system and you want to deposit money to an
account.  You could create an object such that you could almost say
"deposit money to account" where "deposit" is an object.  Then inside
that object you would standardize all the code that has to do with
depositing the money.

You can do this with sub-programs and paragraphs, but this takes it a
step further.  OO also makes it easier to deal with Windows officially. 
There is a lot of other stuff, but that should give you a basic
understanding.

Excellent questions.  Origins of OO came from what I regard as somewhat arcane
distinctions being made between what were being referred to as "Semantic
Models" and  "Object Models".  Roger King, in a lovely,(and mericifully brief),
 article published in the late 1980s titled "My Cat Is Object-Oriented",  does
the best 6 page summary of what OO was originally striving to be.  If you can
find it, read it.

There is one major notion underlying OO.   This is that the methods for
modifying and accessing data must be imbedded in the data type definition; ie
you may not define data without also defining how you will use it.  From this
notion spring subsidiary rules,  such as that OO is more concerned with 
describing behavior than structure, and that you may not really make
distinctions between properties and methods.  And then there is inheritance...

It is an interesting game, and you can argue over just where to "bound" an
"object" at least as long as you can take to argue the scope of a module.     

My take is that what are now commonly referred to as OO techniques are pretty
well unrelated to the original OO concepts. They are more about how to program
'C' and 'C' derivatives under UNIX( or under UNIX-derived operating systems).

As to OO and COBOL, I would say that the original, classic COBOL was and always
has been OO in the sense of imbedding methods for handling data as part of the
data definition.  The MOVE  statement, when coupled with the COBOL rules for
data definition,  are, to my mind, what that sort of OO is all about.   If C++
had MOVE and decent native data typing, you could dump 50% of the code in C++
programs I've seen.  If C++  had PERFORM, you could dump another 
30%.  Of course, then C++ would look a lot like COBOL, and we can't have that,
now can we?

As to who needs what is now the common definition of OO, unfortunately we all
do.  Right now, it is where the money is.

	    now play around with the data for string from n

different shapes.

OO and Flexus have nothing to do with each other.  One common
misconception is that OO and GUI go together, which is far from the
truth.

OO is a formalized method of code re-use and exchange.  With MicroSoft
formalizing their own Object Interface with COM (Common Object Model),
one can interface with a variety of objects using OO.  

Now, one does not have to have an OO COBOL program to use these
outside objects, but it helps the understanding.

Up to this point, I have avoided learning OO COBOL because there
really is no common implementation of it yet.  As the standard gets
closer to solidity, I plan to get heavily into it.

The benefits and drawbacks of OO cause much debate.  The ideas behind
OO are far from "NEW".  I actually think I finally understand it,
after many years.  Getting over the Vocabulary is one thing you must
do.  

A few years ago, I radically changed my coding style to become more
structured and modular.  Having done that, the next logical step for
me is to embrace OO.  The question is, is OO valid?  Is it a sound
development strategy.  I am leaning toward the side of the arguement
that it is.  It is probably worth the time investment necessary to
grasp and embrace it.  I don't think it's of much use in a conversoin
scheme, but I do thing it has great validity for new development.  

I've watched several talented and wise programmers in the group, come
to understand and endorse OO programming.   That's a good
recomendation to me.
