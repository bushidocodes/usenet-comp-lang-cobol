```
A year ago, some of you guys asked about getting around the
requirement of using the Fujitsu's installation wrapper utility.

I was wondering if anyone has had any success yet creating an install
using anything other than the installation wrapper provided by
Fujitsu.  I need to deploy the runtime dependent files for Fujitsu
Cobol v6, Powercobol & Powerform in the same directory as my
application (in lieu of the default C:\Program Files\Common
Files\Fujitsu\COBOL\).  One of my big reasons for doing this is to get
rid of that troublesome PATH setting to AUTOEXEC.BAT.

I used a program called InstallWatch Pro v2.5c to identify the large
number of registry entries and dependent files at the point of the
runtime installations.  I copied the dependent files to my application
directory and created a .REG file that included all of the added
registry items and correct path locations.

After working on this for a couple of days, I'm still getting the
error 'RUN-TIME SYSTEM NOT INSTALLED PROPERLY.'

Any input would be greatly appreciated.
Thanks,
Kate
```

```
I ran into the same problem with the PATH, and came up with a somewhat
dirty but completely effective solution using a shareware utility named
"REGSRCH" (Registry search & replace).

After installing the app I run REGSRCH and replace that outrageously
long Common Files path with "C:\Fujitsu", then move all the files there
from Fujitsu's program files subfolder and edit the PATH in
Autoexec.bat.  
REGSRCH lets you save script files, so once I got the procedure working
properly I just saved the necessary script and utility files in the
Extra's folder with the Fuji installer.

Only problem is that the install becomes too complicated for any user,
no matter how well you document it.  The Regsrch utility is well worth
the $20 registration fee.

Personally I feel that forcing those incredibly long pathnames into the
autoexec is a dirty trick and poor programming on the part of software
vendors.  It becomes an even greater problem when there are several apps
installed that do the same thing - PC Anywhere and Pervasive SQL to name
a few.

You can (or should) also modify the config.sys to increase the
environment size, and someone else here once stated that a longer path
can be accomplished by specifying the PATH there also, though I've not
tried that myself.
```
