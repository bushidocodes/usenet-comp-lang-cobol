IDMS is not a relational DBMS. It is what I believe is called a network
DBMS. It was developed by a company that was called Cullinane, then
Cullinet, and is now owned and supported by Computer Associates. (what
isn't?) Current releases of the software are Y2K compliant.

IMHO IDMS is considerably more programmer-friendly than IMS (or for that
matter, anything else developed by IBM). That is not meant to imply that
it is simple to learn, just that ordinary coding is less tedious.

I haven't looked for many books on IDMS recently but there should be some
out there. It may not be popular currently for new development, but there
are many large companies that have legacy IDMS applications. I have spent
most of the last 16 years working in IDMS shops, so obviously there is work
out there.


Tom McFarland

›
› IDMS is not a relational DBMS. It is what I believe is called a
› network DBMS.

Although I have my doubts as to whether the classifications 'relational'
and 'network' really cut the mustard.

In a relational DB the individual tables are relational but the
relationships between tables are effectively a network of foreign keys
ie you can have a chain of foreign keys that leads back to the first
table.
And in a network DB if you can read all records of a particular type in
any kind of sequence, eg an area sweep or indexed read in IDMS, then
that's the same as an SQL select, the only difference being getting all
columns for each row.

Jim Van Sickle wrote:
›
› IMHO, IDMS would better be described as 'hierarchical' as it is based on
› one-to-many set relations.

Can't let this pass! A relational table is also based on one-to-many set
relations - one column, many rows. It depends on how you look at it.
Between two record-types (eg Dept & Empl) there can be a set 'Employs'
where each Dept record owns sets of Empl records, also a set 'Manages'
where some Empl records own other Empl records (also a set 'Managed-by'
if some employees have more than one manager), and also a set
'Worked-for' where each Empl records own Dept records (in a sequence
showing their work history).

No hierarchical DBMS could do this, even a relational one would have
problems since the columns Dept-Id & Empl-Id are being used as both
primary & foreign keys.

Also the circularity of the set definitions can make it impossible to
add records when referential integrity has to apply after each update;
IDMS can get round this by making one of the set connections optional,
which means the pointers for that set don't get populated automatically
(they can be later in the code by an explicit 'Connect to set' command),
whereas DB2 can apply referential constraints only at data level so the
sets aren't nearly so independent.

IDMS has several other data structures/access modes besides sets
(=linked lists, essentially, in both sorted & 'as stored' styles):
hashed keys, binary-tree indexes, physically sequential, & LIFO/FIFO
queues, so saying it's like DL1 doesn't really fit. Certainly the
classic complaint about hierarchical DBMSes - that you have to keep
recompiling everything after every schema change - doesn't apply to
IDMS; hashed keys & system-owned indexes mean that there's more than one
start point to the structure.

The biggest weakness of IDMS is that it does not use SQL which has become
the standard database manipulation language for the major relational
databases such as DB2, Oracle etc. A COBOL programmer with a knowledge of
SQL should find it reasonably easy to write programs for disparate
database platforms provided they are SQL compliant. If you want to
maximize your employment prospects and future earnings stick with SQL. Non
SQL databases are doomed to extinction.

>From a technical point of view splitting hairs over whether IDMS/R is
relational is immaterial as in my experience practical use of referential
integrity in the DB2 community industry is pretty rare. I have never once
had an RI related SQL error. Having worked with both IDMS and DB2 I can
say that DB2 wins out mainly because the programmer does not have to
physically navigate the database to locate records. This allows the
programmer to concentrate on the logic of the program and results in a
cleaner more structured program design. Losing currency on the database
is a major headache in IDMS.
