Be careful here... are you talking about a combo box or a ListBox? Both of
them can have drop lists. If you simply want a drop down list that the user
can click on, that would be a ListBox; if the user can enter a string and
see the list scroll to the point matching the entry string, that would be a
combo box (Windows terminology, not Vendor or language specific).

I think you are talking a combo box but cannot be sure from the description.

It appears that if the user enters a string which is not in the Collection
the droplist has been loaded from, the new entry is added to the droplist
AND the Collection???

That would be specific to MF. There should be some attributes or Methods you
can use to stop this. Can you look at the combo control's attributes with an
Object Browser?

For example, one of the combo box controls I use has an attribute called
"IsQuery". If this is TRUE, only data which matches the string being entered
by the user will be displayed in the droplist. Setting it FALSE allows the
User to input data which does not currently exist. However, this data would
not be automatically added to the source Collection/File, neither would it
be added to the droplist... What happened to it would depend on what Methods
or code I invoked on the "Return" or "SelChange" events...

This may be true but I must point out that Net Express 3.1 from Micro Focus
*does* include this capability as does Server Express 2.0.10 and later.

Micro Focus provide a Java import that includes various classes and methods,
particularly cobCall class. The class wizard in NetExpress helps you to wrap
existing cobol code in Java for use in distributed applications. An example of
where it's useful to do this is where a Java Swing based GUI interface that
works across an intranet or the internet. The user interface works uses the
standard Java techniques for calling processes on the server which could be
cobol from a legacy system. Many micro Focus customers have chosen this route
for modernise and web enable existing systems.

Some people prefer this approach to using HTML and CGI because of the extra
control and interaction you can have between the aplication code and the user
interface that you can get with Java.

Also, I agree that using CGI with COBOL is not complicated. Net Express make
this surprisingly easy.

Of course, PLI uses the mathematical rules, i.e. 25+1/3 = 25 + (1/3).

The interesting thing, though, (and that is what Glen probably had in
mind) is that you won't get what you expect:

PUT SKIP LIST(25+1/3) will not display 25.333333333 or something
similar, but will do one of the following

- ABEND (S0CA)
- displays 5.33333333333333

If the default FIXEDOVERFLOW condition processing is active at the
time the instruction is executed, you get the abend, else you get the
(truncated) result.

BTW, it gets even more interesting if you then try

PUT SKIP LIST(25+001/3);
