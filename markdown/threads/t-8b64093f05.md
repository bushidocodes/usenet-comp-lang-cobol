OBJECT ORIENTATION 101
----------------------

For those who may not comprehend or are uncomfortable with OO.

It's probably only a minority of COBOL users who will be using more than 
one language. However, take the EU (European Union). Their documentation 
is translated into something like 39 languages. Using this DateAndTime 
technique I'm describing, then they could have 39 INSTANCES triggered by 
the sub-class names.

That doesn't mean that 39 copies of the Class DateAndTime are resident 
in memory; only one is used with an area for Global Working-Storage 
variables per INSTANCE. The DateAndTime methods are then drawn into 
memory as required per individual instance. Any variables coded in a 
method are lost on exiting, (EXIT METHOD), i.e. each method starts in an 
initial state.

All classes inherit from somewhere and through an hierarchial chain,
in Micro Focus finishing up at Class BASE; for Fujitsu that's FJBASE, I
think.

In this project DateAndTime, (the parent class for any DatesEnglish, 
DatesPortuguese etc...), inherits from Base. As you can see
for DatesPortuguese above it inherits from DateAndTime.cbl.

You never use : invoke DateAndTime "new" returning os-DateAndTime

Well you can use it - but your program will go belly-up if you try to
use methods that need to invoke day or month names !

Instead you invoke the sub-classes you need :-

invoke DatesFrancais "new" returning os-DatesFrancais
invoke DatesItalia   "new" returning os-DatesItalia

Both the above are separate Instances with their own day/month name
tables and they inherit (can use), all the methods in DateAndTime. 
Within DateAndTime source I have a method "FormatAnyDateorTime", which 
when necessary using input parameters will access the methods in 
DatesPortugues above.

Normally we think in terms of invoking upwards, but that method 
"FormatAnyDateorTime" contains invoke SELF (meaning this particular 
Instance) "GetMonthName". Regardless of how many Levels in the 
hierarchy, if I code :-

(1)	invoke super "GetMonthName"

then it is only going to search UPWARDS to the class immediatly above 
it, and any classes above that until it comes to Class "Base" and then I 
will get back an error "datesportuguese doesn't understand method 
'getmonthname'"; the program then aborts.

(2)	invoke SELF "GetMonthName"

implies that the class is to search within its own Instance. In such 
cases the class starts at the BOTTOM of the hierrarchial chain. So in 
this example, no searching occurs in the text of DateAndTime; instead it 
starts directly at the BOTTOM which is the source for DatesPortuguese.

Now having written # 2 it got me thinking, and I've never actually tried 
it. It's a lousy example but let's say DateAndTime inherited from 
DatesPortuguese, and again we use the same (2) Invoke SELF - in theory
DateAndTime will look within itself for the method and not finding it 
*should*, (I think), automatically search upwards to DatesPortuguese 
without the necessity of using the word SUPER - How say you PECD ?

However you should appreciate the difference between SUPER and SELF; 
when designing your own applications and knowing your hierarchial chain, 
the use of SUPER is quicker/more effective, when the method is up ABOVE.

I do quite a lot of dynamic method invocation where a Method name is

To try and explain:

Instead of...

Class1 -establishes connection to DB or opens/initialises file

Class2 inherits Class1's open and connect methods  - has an insert method

Class3 inherits Class1's open and connect methods -  has an update method

Class4 inherits Class1's open and connect methods -  has a delete method

fetches records/table rows

(Yes it is a contrived example, but the idea of inheritance is to have 
common processes nearest the root...)

The problem with this is that each Class must be instantiated before it can 
be used, and each Method invoke is hard coded.

 This gets tedious.

I find it easier to simply say...

Namespace  dealWithExternalData
        Class DataProcessing
                  Method1 - -establishes connection to DB or 
opens/initialises file (parameterized file/db name)
                  Method2  - inserts data (parameterized by record/table 
type) against the currently open DB/file (represented by  this
                                     instance of  Class "DataProcessing")
                  Method3  - changes data (parameterized by record/table 
type) against the currently open DB/file (represented by  this
                                     instance of  Class "DataProcessing")
                  Method4  - deletes data (parameterized by record/table 
type) against the currently open DB/file (represented by  this
                                     instance of  Class "DataProcessing")
                  Method5  - fetches data (parameterized by record/table 
type) against the currently open DB/file (represented by  this
                                     instance of  Class "DataProcessing")
                  Method6  - closes current file/DB connection (represented 
by  this instance of  Class "DataProcessing")

If I specify this Class as a COM component, I can use it with C# or other 
languages besides COBOL and I can run it in the .NET environment.

Now I can instantiate an instance of ONE Class and dynamically invoke 
whatever methods I want by parameter... like this:
                  invoke COM "CREATE-OBJECT"
                              using "DataProcessing.DataProcessing.1"
                              returning objDataProcessing
                  end-invoke
                  move "Method1" to required-method
                  move "MyDB" to archive-name
                  invoke objDataProcessing required-method
                             using archive-name
                             returning SQLSTATE (or File-status or whatever)
                 end-invoke
                 (error checking... then...)
                  move "Method2" to required-method
                  invoke objDataProcessing required-method
                             using tablename
                                      record-buffer
                             returning SQLSTATE (or File-status or whatever)
                 end-invoke
                 (and so on)

I can put this "frame" in a called module (or PERFORMed section...) and pass 
the parameters to that wrapper  and it will process whatever DB/file I tell 
it to. In fact, I can even pass the method I want as a parameter. It is 
completely flexible functionality that just encapsulates the problem of 
managing the file/DB. In effect, the dynamic invocation of the methods has 
replaced the need for inheritance. (If the thing you were going to inherit 
is part of the same object, you don't need to inherit it...)
(The Data Access Layer objects of the Toolset work on these principles but 
are much more complex, to allow multi-threading and reference and filtration 
of specific fields or columns, as well as sequential access via SQL cursors, 
with state being saved across invocations.)

Anyway....getting back to inheritance and SUPER...the thing that trumps all 
supposition is empirical data.

Why not just try it and see? :-)
