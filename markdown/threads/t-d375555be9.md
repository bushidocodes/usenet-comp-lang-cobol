```
Simple. Read into the edit picture that was used to create the
printable file, move to numeric, add to counter. Cobol will de-edit
the number on MOVE, not ADD. For example:

01  print-record.
     05  edited-amount  pic ++++++++++.99

01  normalized-amount pic s9(10)v9(02)

01  amount-accum       pic s9(10)v9(02).

read input-file into print-record
move edited-amount to normalized-amount
add normalized-amount to amount-accum.
```

```
> What? On what compiler does this work?

Any compiler whose Nucleus module conforms to ANSI X3.23-1985 Level 2.  See
Page VI-104, 6.19.4, MOVE statement, General rule 4b, last sentence:  "When
the sending operand is numeric edited, de-editing is implied to establish
the operand's unedited numeric value, which may be signed; then the unedited
numeric value is moved to the receiving field."   Also see the "Summary of
Elements in the Nucleus Module" on Page I-17 as well as "Substantive Changes
(Not Affecting)" Item 78 on Page XVII-47.
```

```
MicroFocus has allowed de-editing since earliest CIS Cobol.  ANS'85
introduced aligned de-editing which requires that the characters
exactly match the picture clause.

MF default behaviour is now DE-EDIT"2" which is ANS'85, DE-EDIT"1"
gives earlier behaviour which ignores picture, eg:

     01  In-Value             PIC X(10).
     01  In-Edited  REDEFINES In-Value PIC -(6)9.99.
     01  In-Value             PIC S9(7)V99.

         ACCEPT In-Value AT llcc
         MOVE In-Edited   TO In-Value

     Key in 99.1   gives result of 00000991
```
