Assumption: Each 'Record' in your file is terminated by
something the program (internally) recognizes as a
delimiter (usually a carriage-return and/or line-feed).
Otherwise, you have streaming input, like a teletype
machine (which COBOL is loathe to handle) OR you
have variable length records.

Rule: When an input record is shorter than the maximum
defined record length, the program fills the remainder of
record definition with spaces.

Assuming live data cannot
be spaces, you could loop through all five (or more)
occurrences looking for a data element of space and,
having found it, reduce the index by one to get the last
"live" data item.

Hi,
If your input file is line sequential, you should make some minor changes to
your program.

You will never know in advance, how many bytes you get (the length is the
number bytes from start to x'0D' x'0A' ).

Keep things simple. Read first and decide what to do with the data after
that. Her is my suggestion:

FD  INPUT-FILE.
01  INPUT-RECORD                   PIC X(255).

In Working-Storage:

01  Ws-Input.
       05  Filler                               Pic x(15).
       05  Data-I-Might-Need.
             07 Need-element          Occurs 80 times.
                    09 IR-Number         Pic X(2).
                    09 Filler                   Pic X.

The idea behind this is that records read can be examined (validated) as you
like.
Some compilers will not initialize Input-Record. If that should cause
problems,  move all spaces to Input-Record before read.

After read, move Input-Record to WS-Input and decide what to do.

The record size should be adjusted according to the actual max. record size.

Later on, you can try read into WS-input (Some Cobol-dialects will not allow
you to define fillers in the Input-Record).

Regards
Ib

 If the file is line sequential, which would be the norm for this type of
data, then Cobol will fill the rest of the line with space. In a case like
that, you can simply read the lines and check to see if you got data or
spaces.

You may be running into a problem because of your record being too short. If
your description is longer than the physical record, then Cobol will blank
fill on a line sequential file. If it is too short, however, the unread
portion of the record can carry over into the next read.  That may be your
problem.  I would try sticking a filler at the end.

The code, compared to the "XX"s that you drew, is also wrong. You have no
space between the x(2) fields.  IE:

    05  WHAT-I-NEED OCCURS 5 TIMES.
        10  IR-NUMBER          PIC X(2).
         10 filler                        picture x.
    05 maybe-a-filler            picture x(a bunch).

   The "Indexed by" clause just obscures the issue, and is not needed.  You
can reference the data with a simple subscript.
