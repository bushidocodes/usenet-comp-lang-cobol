### Passing an open file handle

- **From:** GregL <gregl@csi.com>

```
Hi,

I have a MFCobol program that prints a report.  I need to be able to
call a sub-program to print additional information on some occasions. 
When I try to open the printer in the sub-program I get a 'file already
opened' message.  If I close the printer in the main program before
calling the sub-program, the printer ejects the page and the sub-program
prints on a new page.

So I'm thinking I need to be able to pass a handle (or something) of the
open printer in the main program and let the sub-program use it for
printing.  I can put the file name in the call 'name' using fd-printer,
but I can't figure out how to define the file in the linkage section of
the sub-program.

Has anybody done this?  Is there a better way to skin this cat?

Greg L.
```

Others have quite properly suggested making the file EXTERNAL.  There
is also a useful alternative: confine all the IO to a single subprogram
which can be called from whatever module needs to access the file.
Instead of coding WRITE statements, code CALL statements, passing
the records to be written.  You also need a separate entry point, or
a separate action code of some kind, to trigger a CLOSE; the
subprogram has no other way of knowing when you're finished.

This approach has several advantages:


1. It works in older dialects that do not support EXTERNAL.

2. The FD is confined to a single place.  If you need to change the
FD you can change it in a single place.  If your flat file turns
into an indexed file, or a database, or a network socket, you can
confine the changes to a single place.

3. It enables you to encapsulate error handling and any tricky or
tedious processing that you'd rather not code over and over.  For 
example, if you're writing to a log file, the subprogram can add a
timestamp and/or sequence number to each record.


Circumstances alter cases.  In the case originally cited, where the 
program was simply writing a report, a separate access module like this 
would probably be overkill.  In other cases an access module makes
better sense, especially when item 3 (above) applies.

What you require is the EXTERNAL clause on the FD (look it
up).

This should be specified on both the main and the sub program
FDs and it makes the two specifications into one file so
that either program can operate it.  Open and close it in
the main program only, the sub-program can then just
write to it without any further requirements.
