SQL is an Ansi specification, use the IBM manuals online to work out
the various syntax details.  The basic problem with SQL is that there
are subtle variations in the way things actually work with the access
of data.  This means it is not as portable as originally hoped.

http://www.s390.ibm.com/bookmgr-cgi/bookmgr.cmd/Library

The connection to the database is generally externalised from the
program.  For example, the program is started in a database 'shell' or
there are non SQL statements that connect you to the database.

DB2 for example links the database via a 'plan' that is defined when
you start the program.  This plan allows the one program to run in
production and test without change by referencing the tables.

The other method is ODBC (microsloth) which connects to the database
and you can pass the SQL text to it.  Multiple vendors provide the
same API so that your application can hook up to many vendors. 
Gotcha's in this is that the SQL is subtly different (again).

Other platforms provide there own interface, MYSQL freeware on linux
and other platforms work this way.  You pass the SQL text in and it
returns the data by a series of calls.

I've found this thread quite interesting.

I've just helped write the SQL implementation for RM/COBOL and talk of
'standards' is an interesting (and amusing!) topic to me in many ways.

I am a firm believer in the concept and benefits of COBOL.  I write this
response therefore not as a representetive of Liant - but as someone
interested in COBOL as we all are.

I hate the 'traditional' way of COBOL EXEC SQL and preprocessors.  Why
should you have to preprocess source code?  To compile and then compile
again sounds rediculous to my simple way of thinking (obviously waiting for
the replies on the use of simple!).  I stepped back from any preconcieved
ideas of how SQL 'should' be used with COBOL and hopefully those guys back
40 years ago would be proud of us!  We've taken a look at how new 'SQL only'
languages use SQL and implemented it to COBOL ANSI-85 standards.  Obviously
there is some glue in there at execution time but any COBOL compiler could
compile the code without the need for any 'special' C or SQL datatypes and
the glue can be adapted for portability (to different databases by Liant
and, for the concept back 40 years ago, other COBOLs).

As previous postings have said, there are many different types of standards
when combining COBOL and SQL.  The talk of different vendors wanting to
change things to suit commercial results I don't think can be attached to
the COBOL industry as .  The COBOL syntax is only 1 of them (I think I'm
being impartial here).  The SQL syntax standards vary seemingly by the
vendor, package, major version and minor version (and day of the week).
They do though keep to a reasonable consistency and SQL is dynamic, hence
the need for a marriage with COBOL and also some allowance to vary.

What I believe is the real nightmare for SQL though is the way for 1 product
to talk to another so that the SQL can be passed to it and results gained.
Please do not tell anybody, but in this case I think we should thank
Microsoft for 'imposing' the ODBC standard.  Without this standard there is
a miriad of APIs which results in the impractical portability of SQL when
changing platforms and software.  If we take Oracle as an example, they
champion their OCI API and this has many incarnations dependent on version.

I havn't really helped the original question here.  I suppose that EXEC SQL
is a commonly used and documented way of using SQL from COBOL but there are
other ways of doing it to COBOL standards.

Putting my Liant hat back on for a second: Ken, why is RM/COBOL not on your
WEB site?

Driving an SQL database this way is BAD NEWS!  I worked on McCormick &
Dodge GL3 DB2 and it was 50% slower than VSAM because it used a direct
VSAM drop in like that described.

The benefit of SQL is that you can join multiple tables and return a
single record set.  It simplifies you program quite a bit.  With power
comes responsibility, you really have to watch for no matches and code
you SQL appropriately.

The second benefit is optimization:  If you code access for each table
in turn then you are forcing a path or method of extraction.  If you
code joining SQL the database can work out the optimal path based on
statistics it accumulates (called runstats in DB2).  These statistics
sometime provide a surprising access path.  It also stops reading of
data pages when you can link indexes.
