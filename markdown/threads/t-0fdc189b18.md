```
The major (and in my opinion only "common") reason for allowing qualified
data-names is for the case that there is an "application need" to use
multiple records with the same layout in multiple separate programs and/or
places within the same program.

It is (IMHO) important to "maintenance" of an entire application that there
be a "single record layout" for use in all programs and all sections and
places within each program.  Furthermore, the "maintenance" of this record
layout should (again IMHO) be the responsibility of a single "group"
(possibly with a "data-dictionary" or similar "repository").  This record
layout should be available to all programs and applications via a common
copylib member.  ("COPYLIB" is an IBM mainframe term - but from the
reference to "ISPF" in the original note, I *assume* you are in an IBM
mainframe environment.)

If there is a single copy member that describes the "record description"
(including any REDEFINES) then it may be maintained in a "single place" and
all applications will AUTOMATICALLY will get any/all changes to the record's
description.

It is possible that if all applications in all places ONLY reference a
record description once, then you don't need qualification.  However, if
even one of your programs require any of the following this doesn't work:
 - The same record is referenced in multiple sections (e.g. Linkage and
File)
 - There are multiple files with the same layout (e.g. input and output - or
master and transaction)
 - There are application needs to maintain copies of records for
simultaneous processing (e.g. "save" record and "current" record)

Once you accept that there are ANY reasons to have multiple copies of the
same record layout in the same program, there are two possible "solutions"
to this.  (The following assumes an '85 Standard compiler)

 1) A single copymember that uses a "tricky" prefix method - that will be
modified by ALL programs via COPY/REPLACING, e.g
   01 :TAG:-File.
      05  :TAG:-Elem1   Pic X.

2) A single copymember that does NOT include an 01-level and is always
copied in under an 01-level name and is referenced via qualification.

   ***

There are proponents (and opponents) of both of these methods.  Either one
is "easy to maintain" in shops that use it "exclusively" and both are hard
to maintain in cases where some/many programmers are not used to them  The
first method *requires* that every COPY of the record description use
COPY/REPLACING (the copymember is not valid COBOL source code). The second
method leads to (slightly) longer source statements (but also allows for
CORRESPONDING option for some statements - which is again good or bad
depending on how it is used).

Note: The one method that I personally find "bad" is where a shop stores
MULTIPLE copies of the same record layout (one for Linkage Section, one for
file section, one for input files, one for output, etc).  I know of shops
where this is "common" - but I find that it is the most error-prone (people
forget to keep all the copies "in sync").

Note2:  This entire thread relates to a shop where there are multiple
(many?) programmers and possibly programming groups.  As some readers of
comp.lang.cobol are in single (or small) programming environments, this is
really totally a matter of "personal preference".  Whatever technique a
single programmer is used to and expects in his/her own code is the "best"
method as long as they use their own consistent style.

Note3: The current ('85) Standard does not allow VALUE clauses in Linkage
section.  This *could* lead to a problem with using a single copylib member
for all references to a record layout.  HOWEVER, IBM (the environment of
this thread) does allow it already (and, furthermore, the next Standard will
allow this specifically so that programmers MAY have common copymember in
all sections).
```

```
see qualification as a complete waste of typing time and code space

then begin to justify, just about anything. :-)

Each data name in a given system should be spelled identically every
time it is used, except for a prefix (*NOT* suffix!), which is unique
to each 01 record in which it appears.  I find that 3 to 5 character
prefixes are generally best.  Two character prefixes are enough to be
unique, but usually not enough to be mnemonic, which is important for
clarity.  Making an acronym prefix from the 01 record name is often
convenient (e.g. 01 Date-Work-Area -> DWA-xxx).

Every abbreviated word should be abbreviated exactly the same in every
use of the word in data and procedure names, throughout a system.  It
is usually necessary to keep an abbreviations dictionary, with one
person responsible for maintaining it.  For example, having 'NUMBER'
abbreviated 'NO', 'NR', 'NBR', etc. in various places is nothing but
confusion.

One of the primary signs of excellence is consistency, consistency,
consistency.  Most programmers think they produce 'good code'.  You
have to know what their associates and clients think to tell the real
pros from the heard.  If you're really good, other programmers like
to maintain your code, and will often tell you so, especially when
they work on one of your programs after spending weeks modifying
someone else's crummy code. :-)
```

```
I agree.  I can also add another use of MOVE CORRESPONDING.
Suppose you have an input file that has up to 500 different record
layouts.  And suppose the customer enters the records they want using a
text editor.  The record type is the first 3 characters of the record.
Each record has from one up to 45 parameters the customer may enter.
Each parameter is separated by a single space.  Parameters vary in size.

With all of that said, now you can easily see that all of the
parameters do not fall outside of their "boxes".  In other words, the
spaces between them are indeed all spaces.

At the beginning of each record's routine you simply put

MOVE CORRESPONDING RECORD-IS-TYPE-ABC TO WS-WHITE-SPACE.

Here is what WS-WHITE-SPACE looks like:

01  WS-WHITE-SPACE.
    02 WS-WHITE-SPACE-1             PIC X(01).
    02 WS-WHITE-SPACE-2             PIC X(01).

The RECORD-IS-TYPE-... records all have WS-WHITE-SPACE-1, WS-WHITE-
SPACE-2, etc. defined as the one byte spaces between the
parameters.

Thus you can check for spaces between the parameters with just two
sentences.

MOVE CORRESPONDING RECORD-IS-TYPE-ABC TO WS-WHITE-SPACE.
IF WS-WHITE-SPACE IS NOT EQUAL TO SPACE
    234-REPORT-USER-ERROR THRU 234-EXIT
END-IF.

Life is grand in COBOL.
```

```
Yes, but the appropriate response is to reply to the client *in
writing* with the cons (and pros, if any) of the request, along
with an estimated cost to make the requested changes.  Then let
the client decide if it's worth it.  If you are strongly against
the client's request, require them to put the request in writing
before proceeding.  If anyone ever complains about the code, you
have the written request and your written objections.  If you
feel the request makes the code so bad you don't want your
company associated with it, explain that you will be happy to
complete the project as originally specified, or you will assist
(at fee) to transfer the project to someone willing to code it
their way.

Don't ever hold the client over a barrel, or stand in the way of
them doing what they want to do, other than something illegal or
unethical.  Just turn it around and think how you would feel if
you were the client and paying for the project out of your own
pocket.  If you treat clients as you would want to be treated,
you will always prosper in the long run.
```
