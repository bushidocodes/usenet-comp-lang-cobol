This is another topic where I keep "hitting my head against a wall" when I
try to get J4 (or WG4) to fix a problem in the draft Standard.  (In fact,
this may even be a problem in the '85 Standard.)  J4 (fairly) consistently
tells me there is no problem, but I just can't find what they do in the
actual words in the Standard.  If you are interested in this topic and want
to express an opinion that there *IS* a problem, please send your comments
to the J4 chair at:
    doncobol <at> mediaone.net

If you read the draft Standard and agree with J4 that I am "crazy," I don't
think you need to do anything. (You may or may want to send me public or
private mail telling me how stupid I am and why.)

The bottom-line problem (IMHO) is that the draft Standard does not allow
there to be any "data" (explicitly called "characters" in the draft
Standard) between the R-margin (after column 72 in IBM mainframe compilers -
and many other compilers) and the physical end-of-line (however, the
specific implementation defines this) - when you use "fixed format reference
format" (which is what we are now calling the "old-style" source lines).

The following text (from CD 1.5) is what makes me think this:

"Margin R is immediately to the right of the rightmost character position of
a line. The rightmost character position of a line is a fixed number defined
by the implementor."

The following is the latest reply that was drafted by a J4 member to my
concerns. (This has *not* yet been processed by all of J4 - but I do think
reflects a common if not universal opinion at J4.)

"3. Item 14 � what to do to allow (and then logically convert) characters
between the R-margin and the end of the physical line in fixed form
reference format.

Analysis:   No action is needed.  The standard does not define whether there
are or are not characters between the R-margin and the end of the line in
fixed-form.  J4 has rejected the notion of adding anything about this.
Logical conversion looks only at indicators and the program-text-area of
fixed-form.  It does not look at sequence numbers and it does not look
beyond margin R."

***

FYI - if you want to see either of the FULL documents that I am quoting
from, the draft Standard can be viewed at:
  http://people.ne.mediaone.net/doncobol/99-0273.doc

The draft reply to my concern can be viewed at:
  http://people.ne.mediaone.net/doncobol/99-0273.doc

***

P.S.  I don't know how common it is in IBM mainframes to STILL include data
(often program-name or revision information) in columns 73-80 - but I know
that it sure used to be.  I also don't know if this is allowed with other
compilers, so this may or may not be an issue for other programmers too.

In using AccuCobol (a RM variety), we can specify that a line be Included or
eXcluded depending on command line arguments when the compiler is invoked.
If the condition is met, commented code can become live.  This done to keep
a special "flavor" of a program for a specific client or other command line
options a certain client may want.
For example (deleting a number of chars to prevent wraparound):
    DISPLAY MY-FIELD LINE 5 POSITION 25 ERASE EOL
!FLAG
*  DISPLAY MY-FIELD LINE 5 POSITION 25 ERASE TO END OF LINE   FLAG

When we specify (-Si FLAG -Sx !FLAG) on the command line only the 2nd line
is seen by the compiler.  The absence of those arguments or a different
value for them would cause the first line to be read as expected.  Not
exactly my favorite feature, but this company was using it before I got

Cobol-74 said that the program text was in columns 8 thru 72.  Positions
73 thru whatever were comments.

Cobol-85 replaced column 72 with margin R.  The implementor was free to
set margin R between 72 and 73 and have it function just as in Cobol-74,
but the implementor could set margin R elsewhere.  This made any program
non portable if it contained continued literals, or if it made use of
positions which were within margin R on one compiler, but past it on
another compiler.  There were objections for exactly that reason, but the
committee let it through.  I thoguht there was a rule stating that margin
R couldn't be set to the left of column 72, but there's no such rule in
the draft.

As I understand the intent of the current draft is to handle fixed format
source exactly like Cobol-85.  This is the wording at p 36, 6.1.1 that has
Bill worried:

"Margin R is immediately to the right of the rightmost character position
of a line. The rightmost character position of
a line is a fixed number defined by the implementor."

The second sentence makes it clear that margin R is fixed by the
implementor.  Bill is concerned that the first sentence prohibits comments
in character positions past margin R.  After due consideration, I'm not
worried.  I don't think any implementor who allowed commnets past margin R
will now ban them.  This is a general rule, and according to the decoder
ring an implementor may relax a general rule without flagging as an error.

Both "source line" and "reference format line" are called "lines"
because they have a "linear" relationship.

A source line is an undifferentiated arrangement of characters such
that we may describe the character at any character position within
that source line. (As viewed in a COBOL text editor, the source line
may be differentiated; but, as input to a COBOL compiler, it is not.)

A reference format line is an arrangement of fields identified by
their relative character positions such that we may attribute meaning
to values that are contained within those fields.

By mapping the reference format line to the source line we may
then provide meaning to the characters that exist at each position
in the source line.

So, a source line is similar to a reference format line because of
the linear relationship found in each; but they are different because
one has values and the other has meaning. They are related because
the mapping of one to the other describes, in this case, a COBOL
program in fixed-form.

When an implementor establishes a value for Margin R, that
implementor is describing the minimum length of a source line
that may be mapped to the reference format line. Therefore,
the description of Margin R does not impose any upper limit
on the length of the source line. What it does provide is that,
should the source line be longer than the line described by the
reference format, the additional characters shall have no meaning
to the COBOL compiler.

What should an implementor do with any additional characters?
Since the standard doesn't address that issue, it is up to the
implementor.
