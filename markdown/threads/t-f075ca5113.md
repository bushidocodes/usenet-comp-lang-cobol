```
The rules for the assignment of data names in Cobol are as follows:

1.  Names may range from 1 to 31 characters in length;
2.  No spaces (blanks) may appear within a name;
3.  Names may be formed from the alphabet, numerals, and the hyphen.
No special characters may appear in a name except the hyphen.  (I
believe some manufactureres of Cobol for the PC do allow underscore);
4.  Although the hyphen may appear in a name, no name may begin or end
with a hyphen;
5.  The procedure name may consist entirely of numerals, but all other
names must have at least one alphabetic character;
```

```
Most of the other comments in this thread have answered your question.
HOWEVER, from the fact that your data name ("any_name") has an underscore in
it (rather than a hyphen), I would suspect that this is one of the PC
compilers - probably one with LOTS of extensions.

If it is one that I am most familiar with, it DOES allow your source line to
start in column 1 (not just 7 or 8) - even when you are running with "fixed
format" source code.  You may ALSO run into code written in free-format
source code - in which a WHOLE different set of rules apply. (also look out
for lines beginning "$SET")

FYI, COBOL has ALWAYS allowed copybooks to start with any level number (i.e.
01 *or* 05 or 16, etc).  If, however, you are dealing with a compiler that
handles the (very) OLD '68 Standard COBOL COPY statement, there are very
special rules for how copybooks starting with 01-levels are processed.
(Look at a Micro Focus documentation under "OLDCOPY" or IBM's OS/VS COBOL
under LANGLVL(1) for documentation on the impact of this.)

NOTE WELL:
  Not all copybooks have valid source code UNTIL after they are "copied in"
to a program.  The classic example (VERY valid in '85 Standard COBOL and
allowed sort-of as an extension in some earlier ones) is:

Copybook1  has

 01  :tag:rec.
      05  :tag:field-1  Pic X.
      05  (tag2)field-2  Pic 9.

and a COPY statement of

   COPY Copybook1
      Replacing ==:tag:== by ==pref1-==
                        ==(tag2== by ==pref2-==
      .

In this case, the actual copybook has INVALID syntax, but once the
COPY/Replacing is applied, the syntax is valid.
```

```
The truly odd thing is that multiple text words (durring source text
manipulation) can turn into a single COBOL word.  I don't think any other
language has this "feature".  An interesting one is this

  COPY TEXT-NAME REPLACING THIS-THING BY ==X(3)==.

In this case X(3) is four text words, and will be four COBOL words if it
lands in the procedure division, but if it lands after PIC or PICTURE it's
only one COBOL word.  Conventional compiler tools which aren't written for
COBOL just can't deal with this.
```

```
> 2. I have some copybooks which begin:

> without giving it a record name?

COBOL-68 had some bizarre rules about copybooks, but these rules all went
away in COBOL-74.

> 3. Is it valid to have a record/segment/variable name to begin with numbers?
> For example:
>         05 10any_name PIC x.

Perfectly legal in COBOL-74 and beyond.  It was also allowed by most of
the COBOL-68 compilers I used, but the standard may have said otherwise.
I know there were COBOL-68 compilers that didn't allow it.

>  It would be confusing to
> parse"10A" because you won't know if it is a hex
>   number or a name.

That form isn't a hex number in COBOL.  It may have been a hex number in
some weird compiler, but I've never seen it.  All the compilers I have
ever used which had hex literals required the literal to be enclosed in
delimiters of some sort.  The current technique is X"10A", but in
Burroughs (Unisys) it was @10A@ and I think I saw something else.

>  4. I'm under the impression that the first 6 columns are reserved for
> program numbers and that the 7th column is reserved for a comment or
> continuation. I have some copybooks that begin the "01" line starting in the
> 7th column, and some which start it in the 1st or 2nd column. Is this okay?

This is non-standard, but you'll run into it.  Some compilers accept
anything other than "-", "*", "/", or "D" in column 7 as part of the
program text.  The curent draft has a ">>SOURCE FORMAT IS FREE" directive
which eliminates the sequence field and starts the program text in column
1.  Some compilers have implemented this, or a different free form option.
```
