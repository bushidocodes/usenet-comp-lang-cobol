I left my ASCII chart in my Sunday pajamas and I cannot remember what the
hexvals of the TAB or comma are... but I think you can make a COBOL
conversion utility (or even utility-routine in the main program) in a
rather cheesy manner...

define two files, say, INCONV and OUTCONV, each having a DATA RECORD of a
single character. As part of your 0100-HOUSEKEEPING (hee hee... showing
my age here) include something which resembles the following... oh, you'll
need to define a FILE STATUS for INCONV and 88 it for NO-MORE-INCONV, of
course:

OPEN INPUT INCONV OUTPUT OUTCONV.
READ INCONV INTO WK-CHAR
AT END PERFORM EMPTY-INFILE-RITUAL
SET NO-MORE-INCONV TO TRUE.
PERFORM UNTIL NO-MORE-INCONV
IF WK-CHAR = X'qq' <== value for TAB which I do not remember)
MOVE X'zz' TO WK-CHAR <== likewise hexval for comma)
END-IF
WRITE OUTCONV-REC FROM WK-CHAR
READ INCONV INTO WK-CHAR
AT END SET NO-MORE-INCONV TO TRUE
END-READ
END-PERFORM.

... and, obviously, OUTCONV has the same ASSIGNed name at the file that
you want to process in the rest of the program.

FD INCONV.

READ INCONV INTO WS-SINGLE-CHAR.
IF WS-SINGLE=CHAR = WS-TAB

Assuming there aren't any radical differences between MS COBOL and MF
COBOL, you can use the byte-stream I/O functions to work with the raw
data. Look at the
call-by-name section of the COBOL System Reference.

The functions CBL_OPEN_FILE, CBL_READ_FILE, and CBL_CLOSE_FILE should do
it.
Use x'0D0A' as a delimiter for the records. x'09' is the tab character.
CBL_READ_FILE can be called with a flag of 128 to get the size of the
file being read.

If the file is less than 64k in size the entire file can be read in with
a single read operation, otherwise you need to read in 64k blocks (or
smaller) and provide special handling for records that span a block
boundary (like backing up to the end of the last whole record and
reading the next block from that point).

Karl Wagner
