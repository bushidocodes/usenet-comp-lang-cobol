Networked systems generally, running on modern servers, have little need for
batch processing, so systems designed for the network architecture don't
need to include it.

There are very simple reasons for this. At one time, batch processing was
the ONLY way to process transactions. There was only ONE limited capacity
CPU to do the job. Mainframe system design evolved around the available
facilities so, of course, batch processing was a big part of it. If the US
Navy is still running systems based on an architecture from the 1960s, that
is a matter for them, but it doesn't mean that batch processing is
critically important to the rest of the world. (I suspect there will be
other areas of US Naval IT where they don't use batch processing; what does
it prove?)

As processing expanded to do on-line transaction processing with software
like CICS and IMS-DC it became apparent that not ALL of the processing to
complete a transaction could be done in real-time, so transactions were
written to transaction files for later "back-end" processing in BATCH
(usually overnight). That wasn't a bad solution but as hardware power
exploded and networks of processors became available it soon became possible
to do ALL of the processing for a given transaction immediately in real
time. It was when I realized that this was possible, that I first saw the
Holy Grail of having a real time model of the business that was always
reflecting reality and could be relied on for decision making information.
NO BATCH processing, NO synchronization. NO disparate systems all needing to
be congruenced by overnight processing.

A central repository, accessible to all , and reflecting the true state of
affairs at any moment. Something happens in the real world (someone buys
something, or manufactures something, or despatches goods for transfer to a
branch, or whatever...) and it is INSTANTLY reflected in the data model. The
power to do this is available through modern networks. Instead of running a
batch process with some date parameters to get a monthly summary of
whatever, you can call off a summary for ANY period and get it IMMEDIATELY.
(You may have a data warehousing department with some dedicated servers that
do it for you right now as a parallel task.)

Because the average volume of transactions through most businesses is not
outrageous and because the power of modern networks is considerably more
than the power of a 1960s mainframe, all of this is feasible. (The exception
at the moment would be industries where the volume of transactions IS
outrageous (maybe millions per second) and that needs specific approaches,
including batch processing. However, that may not always be the case. As
server power and network bandwith increases, the model I have outlined can
be scaled up and batch processing could truly be completely eliminated and
seen for the quaint anachronism it actually is.)

I have designed quite a number of systems in my career (and modified quite a
number more...) and I have not included batch processing in anything for
well over 20 years now. Don't need it.

But I have never said it is dead.
