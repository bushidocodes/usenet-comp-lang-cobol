```
Hi everyone,

We are looking at implementing the usage of XML in our systems and since
most of our legacy data and applications (written mostly in COBOL) are on
an OS/390 platform, we need to move data from it to an NT box (and
eventually to partners).

On the NT side we plan to use MSXML 3.0 from Microsoft since it�s the most
advanced in terms of stability and conformance with the W3C standards
(XML, DTD, XSL).  But on the OS/390 side the solutions seems not so
trivial.

We plan to use the XMLife vocabulary (developed by ACORD) and we have
these solutions in mind and I would like to know your comments about them
and your recommendations :

1. Create XML files directly from a COBOL application (it is simple and
easy, but it is difficult to ensure the validity of the XML generated
according to a DTD).
2. Use the XML4COBOL parser/generator product (from MAAS) (we don�t know
if it complies with the W3C DOM specifications and if it is easy to use).
3. Install JAVA on the OS/390 to have access to the myriad of XML tools
available on this platform (we don�t have any JAVA expertise, but it could
allow us to concentrate more on business logic than on the creation of XML
files).
4. Convert the data from the OS/390 to XML on the NT platform (we are
limiting the benefits of using XML, since we cannot send an XML file
directly from the OS/390 to our partners, we must convert them on NT box
before).

What do you think of these solutions ? Am I missing something ?  Which is
our best choice considering we have a considerable amount of data and
applications on OS/390 ?

Charles.
```

```
You may also want to consider MQ Series. IBM was saying late last year that forthcoming versions would able to help with XML-to-structure conversions. 

If it were me, I'd probably not want to chew up mainframe MIPS doing data conversion, all things being equal. As well, there are some security benefits to be had hiding your mainframe behind another machine.

The last time I looked at XML4COBOL it didn't support XMLSchema. I think their approach offers some performance advantages at the expense of flexibility. Since it builds an assembler module to match a specific DTD, I don't know what happens if somebody starts sending you XML where the DTD differs from one  you're prepared to handle. Of course, the vendor (MAAS?)  would be able to provide more accurate information than me. Your mileage may vary.
```

```
There are a whole host of issues, but I am happy the original poster
understands them!

Some additional thoughts follow.  XML only a standard for defiing a
framework.  You or your partners will be responsible for defining the
actual "standard" for what DTD's your particular application will
support.  What's interesting so far is that the same data, fed to two
different XML converters, even when the "end result" definiton (the
DTD's you define) for your XML are the same - the XML itself is
different.

My take on this is that the best way to make USE of XML is actually in
the DOM.  The Document Object Model and XML are closely linked.  Once
you "feed" the XML into the DOM - you manipulate the data directly in
the DOM doing whatever you want with it - and then you generate XML,
or actually anything you like - from the DOM. 

To get the full advantage of this - you need to have some Object
Orientation knowledge and make use of the OO features offered on the
various platforms involved.  

The ENTIRE beauty of XML is tied up in the DOM.  Remove the DOM and
you are left with just another way to describe data.
```
