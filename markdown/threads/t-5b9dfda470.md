```
I've been getting some private mail about refactoring COBOL in general and 
about refactoring data access in particular.

There seems to be growing interest on the part of many COBOL guys about 
using OO COBOL. (Where were they 20 years ago... ? :-))

(I suspect that some of these are not COBOL guys as such, but guys who have 
been tasked with handling legacy COBOL...)

The COBOL21 site  shows details of converting Robert Wagner's procedural 
code into OO COBOL and downloads of the source (and the actual tool 
executable) have been increasing. But this is possibly a bit too much for 
someone trying to get their head around OO COBOL as a beginner.

I'm also getting beaten up about the video set on COBOL21 not being 
complete... The videos dealing with Transformation and Werewolf COBOL have 
not been made (there is a lot of work in making and posting effective 
videos) and they have been pushed down a stack of priorities.

For myself, I want to get the word out about automatically having the data 
access in the legacy code refactored to object code, thus avoiding the 
paradigm clash between procedural and Object Oriented code when moving COBOL 
to .Net. (The DZone article on Cretaceous COBOL and Jurassic Java has been a 
big help in doing this and it is now approaching 10,000 hits.)

I'm looking at ways to combine all of these agendas and will post some 
simple examples of a standard legacy procedural program  before and after 
Transformation, with a link so you can download and run the before and after 
versions, and backed by a video run through of how the Werewolf code was 
automatically produced. (This is actually a lot of fun; seeing COBOL code 
get modified "before your eyes" and then comparing the before and after side 
by side is amusing and entertaining, if you are into COBOL...)

The legacy code runs against Indexed files, the Werewolf code runs against 
an RDB of your choice. The results should be identical...

If anyone here is interested in getting onto OO COBOL or is interested in 
refactoring and salvaging COBOL code, please visit the COBOL21 site and look 
at what is there on the COBOL Structure analysis tool.

If this leaves you bewildered or with more questions than answers, please 
drop me a note privately and register your interest in refactoring. (I won't 
SPAM you or add you to a sales list, but I will let you know as soon as the 
new material is available, and I'm always happy to help anyone who is 
struggling with OO COBOL... (been there myself :-)))

Pete.
-- 
"I used to write COBOL...now I can do anything."
```
