I wonder if anyone has a hunch about this.

My last project ran fine, producing printed reports.  Now we're adding an
enhancement module which utilizes VSAM files.  The original module is now the
calling program.

My JCL compiles both modules, links them, and runs the resulting load module. 
The original reports are nowhere to be seen!

When I comment out the calling statements, I get my original reports, so I know
I have a working load module.  When I allow one call to take place (which opens

returned, so I suppose my STOP RUN's were prematurely ending execution.

But why didn't the DISPLAY's associated with those error conditions show up?

Any ideas?

The next thing I'm trying in the morning is heading up my called mainline with
a DISPLAY of a function-code I passed from the calling module, followed up by a
GOBACK.  I'll at least be able to see if the modules are communicating and
DISPLAYS are being printed.

Nighty night.

I assume the DISPLAY statement(s) are in the called module included in the IF code
block which tests for fatal VSAM I/O and before the STOP RUN. It doesn't sound like
there were VSAM I/O problems considering the fact that commenting out the STOP RUN
in the called module solved the problem. I'm gonna venture to guess your STOP RUN
statement was outside the IF code block and therefore execution proceeded to the
STOP RUN as there were no VSAM I/O errors (ergo it could not have branched to the
DISPLAY ergo my conclusion that the STOP RUN is outside the code bolck).

May I suggest that you not code STOP RUN statements in a called module. Rather,
return a status code to the calling program. In your calling program, test the
return code from the called module following the call and branch accordingly.

later
