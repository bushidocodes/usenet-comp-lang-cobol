You're right, there is a NOT INVALID KEY. I use it very often. Like this:

READ custfile INTO custvariables KEY custfilekey
INVALID KEY
MOVE 'Customer not in file' TO report-custname
(or you could test for file-status or maybe PERFORM error-in-file)
NOT INVALID KEY
MOVE custname TO report-custname
(and/or whatever you want to do)
END-READ.

I know that the above can be done in many other way's too, but I like it this way.

That way in my FILE-CONTROL I use:

SELECT K790VSAM-FRS-FILE
ASSIGN TO K790VSAM
ORGANIZATION IS INDEXED
ACCESS IS RANDOM
RECORD KEY IS K790VSAM-PRIMARY-KEY

Then in my i/o paragraphs (one example):

802-READ-RECORD-K790VSAM.
*
MOVE 'K790VSAM' TO K010-FILE-NAME
READ K790VSAM-FRS-FILE
INTO K790-FRS-DATA
END-READ
IF NOT K010-SUCCESSFUL-COMPLETION
AND NOT K010-NO-RECORD-FOUND
THEN
DISPLAY 'ERROR IN 802-READ-RECORD-K790VSAM'
PERFORM 998-PROCESS-VSAM-ERROR
PERFORM 999-ABEND-PROGRAM-RUN
END-IF
.

Then the only copybooked paragraph I use in my prgrams a standard VSAM
error handling technique:

998-PROCESS-VSAM-ERROR.
*
*COPY T271K020.
SET K010-INDEX TO 1
SEARCH K010-FILE-STATUS-ENTRY
AT END
SET K010-INDEX TO K010-NUMBER-OF-ENTRIES
WHEN K010-FILE-STATUS-CODE (K010-INDEX) = K010-FILE-STATUS
CONTINUE
END-SEARCH
MOVE K010-FILE-STATUS-DESC (K010-INDEX) TO K010-MSG
DISPLAY K010-MSG-BLOCK
.

It ain't pretty but it works.

Start of discussion:

Use what suits best. Upon an I/O-statement
(OPEN/READ/WRITE/REWRITE/DELETE/CLOSE/START) several situatons can occur.
You can ignore these by omitting the FILE STATUS construct and statement
options like INVALID KEY / AT END. Some situations then are bad enough for
the run-time system to abend the program. If you want each situation
trapped, good, bad or ugly, then use FILE STATUS. No more abends, each and
all is up to the program(mer). Now for AT END/INVALID KEY: At end traps
exactly *one* thing: read reached end-of-file. This is also reported with
file status 10. INVALID KEY traps a *collection* of similar things. The
similarity is that it concerns something being invalid with the key. The
file statuses for this group of situations all start wit 3. For example 37
means "not found". Do not use INVALID KEY if you just and only want to test
"not found". In that case use an IF/EVALUATE on file status. This last two
sentences are probably what Len Lewandowski meant when he wrote:

› Nobody, NOBODY uses INVALID KEY. Just use file status. End of
› discussion.

The test on status could well be combined with invalid key/at end. Examples:

READ aFile [INTO aRec]
AT END
SET aFileAtEnd TO TRUE
NOT AT END
EVALUATE aFileStatus
WHEN 00
CONTINUE
WHEN xx-other value interested in
perform something
WHEN OTHER
perform user-abend-routine
END-EVALUATE
END-READ

MOVE someValue TO anIdxFileKey
READ anIdxFile
INVALID KEY
EVALUATE aFileStatus
WHEN 37
SET anIdxFileRecNotFound TO TRUE
WHEN 3x-other value interested in
perform something
WHEN OTHER
perform user-abend-routine
END-EVALUATE
NOT INVALID KEY
EVALUATE anIdxFileStatus
WHEN 00
CONTINUE
WHEN xx-other value interested in
perform something
WHEN OTHER
perform user-abend-routine
END-EVALUATE
