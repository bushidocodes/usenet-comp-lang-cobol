```

look at what a person like me, who has
written a small tutorial on file expansion,
cannot get a job.


*Y2K knowledge:
*By G Moore

*I basically understand file expansion for the year 2000 problem. Once
*the variable which has the decade is found, you find every occurance
*of where it is read or written to (usually by FD's or COPY (in
cobol)).
*These places you comment for later alteration.

*Example:

environment division.
select "firstfile.dai" assign to firstfile
organization is indexed
record key is decade
access mode is dynamic.

data division.
FD first-file.
01 date.
02 decade pic 99.
01 other-data pic xxxxx.

*add the following code for the first step to the correct divisions
*******************************************************************

*environment division
select "secondfile.dai" assign to secondfile
organization is indexed
record key is y2k-decade
alternate key is y2k-century-expansion
access mode is dynamic.
*notice the addition of the alternate key


*data division
FD second-file.
01 y2k-expansion.
02 y2k-century-expansion pic 99.
02 std-date.
03 y2k-decade pic 99.
02 std-other-data pic xxxxx.


*size of first file and additional century digits

*You move all the variables in the first file to the second file,
except
*when you come to a decade variable.
*Example:
working-storage section.
01 switch-eof pic 9.

procedure division.

open input first-file.
open output second-file.
move 0 to switch-eof
perform expand-file until switch-eof = 1

close first-file
close second-file
stop run
..

expand-file.
read first-file
at end
move 1 to switch-eof

end-read
if switch-eof not = 1
perform move-data
..

move data.
move date to std-date
move other-data to std-other-data
move 19 to y2k-century-expansion
..

*Then you copy the file description, except add a pre-fix all the
variables,
*as well as make certain that the file description keys are indexed
*on the decade as well as y2k-decade variable.

**check this
*now change the variable names within the second indexed file to the
*first variable names such as std-decade to decade
*this allows any code still within the machine with old variable
*names to remain untouched until a later time for recompilation.


*the next change

select "firstfile.dai" assign to firstfile
organization is indexed
record key is decade
access mode is dynamic.

FD first-file.
01 date.
02 decade pic 99.
01 other-data pic xxxxx.

*to:

select "secondfile.dai" assign to firstfile
organization is indexed
record key is decade
alternate key is century
access mode is dynamic
..
*notice the addition of the alternate key
*and firstfile as the appropriate file-read-source

FD first-file.
01 y2k-fix-step3.
02 date.
03 century pic 99.
03 decade pic 99.
02 other-data pic xxxxx.


*since there was no intermediate code between the date and the decade
*sub-heading, we can add the century right underneath the date
superheading.


*now, once your file is expanded, you would have to find all the
*places where the variables decade and date are used.
*suppose you had some code which read:


01 current-year pic 99.
01 total-years pic 99.

*procedure division
subtract current-year from decade giving total-years.

*you would want to do the following:

01 y2k-fix-current-year.
02 current-century pic 99.
02 current-year pic 99.


*move function-wa-date (or equivilant) to y2k-fix-current-year.

*change
subtract current-year from decade giving total-years.

*to
subtract y2k-fix-current-year from
date giving total-years.

*this assumes of course, that you dont have more than 99 years
*of interest

*then recompile


*This code only takes care of 5 lines of expansion,
*(the file descriptions, indeces, and variables)

*but it is almost 100% certain that old code is not altered
*until you want to,
*as well
*as makes sure that you use the smallest amount of input-
*output for remaking an entire database. The other IO
*is simply relegated to recompiling small chunks of source code,
*which
*would not slow down online transactions on a large
*scale.

*The key to file expansion, as far as mainframe code goes,
*is to make sure that you make the most changes possible
*during your downtime, or keep your downtime to a minimum.
*Also, its very handy if you can schedule, with the business,
*a time when they will not use the database.
*Since one can, within file expansion, practically schedule
*all the changes within the data, I think it is a better
*method.
*The drawback is that it requires rebuilding an entire database
*at once, and if some variables are missed, you would have to rebuild
*multiple times. However, the file expansion method almost
*ensures that old code will run the same way it did before 2000,
*as all variable names are the same until they are changed.

*The down-time required to rebuild a massive database after the
initial
*memory is used up becomes very expensive for businesses.

*One method to protect from such a large database, with fixed size
*records would be to store more of the indeces in memory, temporarily.


*Another method would be to have 2 FDs online
*at all times, and you search for calls to write to the
*first FD, mark it, and also write to the second FD.
*After all calls to write and rewrite are found,
*you would write a program to move one chunk of the database
*at a time to the new FD, via pre-traversal search.
*Next, you find all calls to read, and set them to the
*new FD. Next, you remove the calls to write to the
*old FD, then back it up via tape server.
*Then you remove it from the mainframe.

*During the transition period, your IO speed would
*be half what it was, because of the dual reading
*and writing, but you would have guaranteed that
*all data is transferred safely, and you could
*get on with your Y2K remediation. The problem
*with this approach is, in the beginning, there
*is no real progress that can be measured.

*The only real problem within Cobol programs with
*file expansion is referencing (ie those
*programs which use COLONS to edit values)
*and inter-lingual calls.
```
