You may not need or want to compress at all.
What you are describing is fairly common.
You may have too much data in a single record.

What you may want to do is split the logical file into 2 or more physical
files.
Name & address data takes up a lot of space, and for the most part it's
rarely needed.
So, if you place the name & address in a parallel file (same key as main
file) you reduce the amount of data you have to pass when you are looking
for conditions not in the name & address.
If you have data that is frequently accessed, you may want to place it in a
file with a small record length to speed up the processes that query that
data.
Once you find what you want, you can use random access to get the additional
data to complete the full "record".

You can also partition "active" records from historical records, depending
on the system requirements.

The more "dead" data that you remove from processing, the better your run
times will be.

(I just spoke to the director at a previous place of employment, and he told
me they implemented my suggestion to move the name & address to a separate
file, and processing times dropped. They also implemented my suggestion that
"active" records be separated from historical records with another drop in
processing times.)

The Fujitsu indexed file data record compression is run-length oriented,
which compresses 2 or more repeating spaces, ascii zero, and binary zero;
and 3 or more of any other repeating character.  In other words, it is
designed to do for you what you already appear to have done in your build
program.

The Fujitsu indexed file system is also block oriented, so it should be
possible to use regular ZIP on the indexed file to further reduce the file
size for distribution (say, on a CD-ROM, as a self-extracting .exe).  This
will effectively eliminate any slack bytes due to blocking, and will be able
to benefit from duplications across records.

You may find that any further compression results in _larger_ amounts
of data.

One way to do compression is to allocate fewer bits per character.
First of all find the frequency of each character in the data and then
allocate 2 bit to the most frequent, 3 bits to the next 2 most
frequent, 4 bits to the next 4 etc.  It sounds improbable but it
works. Some of the least frequent have more than 8 bits given to them,
but you won't have those.

The killer is that a table of the bit allocations must be stored along
with the data.  For a megabyte file or so this is insignificant, but
for 500 byte data it may be more than the compressed data itself.

Another way is to search in the data for repeating strings > 3
characters long and hold those once with a flag of 2 or 3 bytes. 
Again for a megabyte of text this gives good savings, but for 500
bytes will result in expansion, not compression.

What works for megabyte files does not fit small data.  Anyway you've
already compressed it as much as some mechanisms do.

For a lossy mechanism you could put each character into 6 bits -
convert lower case to upper to cut down to only a 64 character set. 
That only gets 25% additional reduction though.

One possibility, not mentioned, is to use shift encoding as was
used with Baudot. Three shift codes would allow for three
alphabets (upper-case, lower-case, and numbers and punctuation)
each of which may be encoded in five bits. This may approach
35% compaction. A similar method is shown in James Martin,
"Computer Data-Base Organization" in the chapter titled, "Data
Compaction".

To achieve a greater reduction Huffman encoding may be used.
This could reach the goal of 75%; but 50% may be more realistic.
The compaction ratio for Huffman encoding relies upon some
characters occurring far more frequently than others, such as may
be found with the leading zeros and trailing spaces that have
already be removed. This method has been mentioned, just not as
Huffman encoding.

Both shift encoding and Huffman encoding work well on character
data.

The Huffman encoding method is sufficiently generalized that
a program may be written that accepts, as samples, the data
to be compacted; then generates the program sources to
accomplish the actual encoding and decoding. The first phase,
collecting frequency counts for each character, may be used to
assess how much compaction may be achieved and thus
whether to proceed with writing the program generator.
