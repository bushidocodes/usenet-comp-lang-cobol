We are using Net Express on a Novell network.  We recently had some
users upgraded to use XP machines, so we are suddenly having problems
with disappearing mouses (mice?).

MF personnel then advised us to terminate-then-activate the mouse
after every call to X"91".  This actually does seem to work in some
cases, but it does not work in others.  We are continually scratching
our heads trying to get a handle on this problem, which we are willing
to bet is based mostly on our own collective ignorance.

I also have a general-but-related question:  Is it a better
programming practice to only activate the mouse just before each
ACCEPT statement and then deactivate it just after?  Or, is it better
to keep the mouse enabled most of the time and then disable it just
before calls to X"91"?  I am given to believe the former, but
whichever direction we go in will mean a lot of program changes to
deal with.  I would really rather do this right as opposed to doing it
twice.  Any help would be appreciated.

Thanks,

In Windows mouse messages are merged for mouse movement. Check out your
mouse message receiving side document for details. I always enable mouse. If
there is no message handler available, the message just is discarded by the
system. If you disable it, when your application get complex, the trouble
will come because you can't find where it is disabled. If a program works
sometimes, it normally the coding problem rather system problem. Make a
minimal program to test for system problem. If you fail, then your code ....

Hope it will help.

fields. My approach used to be something like the following - after all 
this time I've forgotten the *true* syntax :-

perform S10-NAME
perform S20-ADDRESS-LINE1
perform ....

S10-NAME.
accept s-name (assuming it goes into a field Cus-Name)
if Cus-Name = spaces
   do an error routine for blank name - with me it was move the literal

    then display error-message-text at Line 25. Using Logper.cbl above

    message for different types (Warning/Informative) and even blink/not
