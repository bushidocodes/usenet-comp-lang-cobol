```
I haven't seen any technical discussion of the draft standard here,
but this group seems like an appropriate place, since we don't have
a comp.std.cobol. So I'll be brave and post a question.

Is the following program legal, and is its output defined?

IDENTIFICATION DIVISION.
PROGRAM-ID. OVERLAP.
DATA DIVISION.
WORKING-STORAGE SECTION.
01 X PIC X(3) VALUE "ABC".
PROCEDURE DIVISION.
BEGIN.
MOVE X (1:2) TO X (2:2)
DISPLAY X
STOP RUN.
END PROGRAM OVERLAP.

The rules about overlapping operands are in clause 14.7.6:

"When a sending and a receiving item in any statement share a
part or all of their storage areas, yet are not defined by the
same data description entry, the result of the execution of such
a statement is undefined. In addition, the results are undefined
for some statements in which sending and receiving items are
defined by the same data description entry. These cases are
addressed in the general rules associated with those statements."

In this case, the sending and receiving items are defined by the same
data description entry. The general rules of the MOVE statement
(clause 14.11.24.3) don't seem to address overlapping operands, so
apparently my example is legal. But the behavior does not seem to be
well defined.

In the previous working draft, this problem was identified as a
known error (Q.2(32)). It is no longer listed as such, but I don't
see that it has been fixed. Am I missing something?

Walter Murray
```

```
This is an interesting question. In my opinion it is completely
defined and works because of general rule 2 of MOVE which says that
you move the sender to a temp and the temp to the receiver. Although
the example uses subscripting rather than reference modification, it
is couched in general terms. We have an unassigned work item to
clarify this further by stating explicitly that you do this with
reference modification. However, nobody has picked it up. I'll
probably do so at the next meeting (in two weeks) because this should
be clarified. So, I would state that the answer to your question is
that it is defined but not clearly and unambiguously. And, it will be
clearly and unambiguously defined in the next standard. That is, you
will get "AAB" in X.
```

```
I realize that it seems strange, but it is a can be a useful way to do
stuff. We would get a zillion flames if we made it illegal. The main
problem is that it was not really defined before so any change will
probably be incompatible with something. I will add your comment to
the proposal to see what the committee thinks about making it illegal.
We would probably make it obsolete this standard and delete it from
the next. Or, make it archaic and leave it in. However, the
committee direction seems to be to make it clearly defined.

It is even worse for overlaps that are not obvious. The prime example
is when a program passes the same item as two different parameters to
a called program and that one doesn't realize that they overlap. The
compiler can't tell so it has to assume everything in the linkage
section can overlap (because in the new standard and most existing
parameters what you pass does not have to be an 01 or 77). Ditto with
pointers - even worse. This situation should probably be made
undefined. There is no way to make it illegal because there is no way
for the compiler to figure it out.
```

```
I haven't done an overlapping move in some time, and I've never done
it with reference modification, but I believe that you would get "AAB"
if this program were executed on an IBM mainframe.

We used to use an overlapping move to initialize tables in COBOL
programs, and the old compiler used to generate a forbidding message
about special runtime routines being required (especially for larger
tables). Over the years probably thousands of mainframe programmers
have looked at the generated code to verify that it would work, and it
always did. In fact, I was taught that an overlapping move was the
best way to initialize a field in IBM assembler programs. The result
was always predictable because byte field moves were always left to
right (or low address to high address) in system 370. Naturally, that
might not be true on some other platform.

I'm not sure how the standards were resolved. Maybe Don Nelson will
explain.
```

```
I just had to jump in to say that table initialization is not required, if
programs use OCCURS...DEPENDING. This technique is useful for several reasons.

It allows the run time routines for MOVE, SEARCH, UNSTRING, and other verbs to
do a lot of your work for you. For example, a SEARCH will only search the
'active' entries, UNSTRING will only look at the current length of the string,
etc.

It supports 'rational' table loading techniques, which result in proper
detection of overflow while loading from an external file, proper sequencing of
entries as they are loaded (so that SEARCH ALL can be employed), etc.

Most closely related to this note, however, is that it is never necessary to
"MOVE HIGH-VALUE TO MY-TABLE" before loading, or a loop after loading to move
HIGH-VALUE to the rest of the items in the table.

In our experience, table handling errors lead to a large percentage of our
production abends, and they can, of course, take many forms. Table overflow
(and occasionally underflow) result in addressing of data as if it
were part of the table, destruction of data, and even destruction of program
instructions, since instructions usually follow the data areas (in the
compilers I've used).

An organization should take a good look at how it allows programmers to code
table handling, and enforce good techniques. It really pays off!
Colin Campbell
```
