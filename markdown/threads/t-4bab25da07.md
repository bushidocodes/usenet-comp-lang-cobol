Interesting idea, I support it.
There will have to be a "central" that's for sure. I can also see a problem
with programming conventions, like sections vs parapgrahps :-)

>If so, what kind of agreements should we make?  Doing business over
>the net is somewhat new.  Should we be talking an irrevocable letter of
>credit, standard PO?, what currency, etc.

Sounds difficult, I for one, would suggest that the "central" I spoke of
administers the pay. Anyone who wants a particular piece of work to be done,
pays all or a part of it as a deposit to the "central", the "central" in
turn distributes tasks, and pay for the work when done. I then consider the
"central" of being the neutral link here. I think it is necessary that there
is no pay until the work is actually done. In order to convict the
programmer that he/she will get paid, it should be a deposit of the whole or
parts of the amount though.

You know, the more I think of this, the more scary it becomes, there raises
so many if's... It will be very dependent of reliability and honesty. I am
afraid that there might be someone out there which won't. Consider another
option, Cobol is targeting businesses at heart. Is it likely that businesses
will let out there source to an anonymous force?

Please do not misunderstand me, I still love the idea, I am just afraid of
all the if's.

>What happens re tax?  <;<)>

I believe most countries has the rule that you are taxable to the country
were the actual work is performed, thus, if I were doing such a job and
living in New Zealand, I would pay tax there. I am however convinced that
this is a minor task, that the individual programmer can handle, they
probably know what to do in their respective country. Keeping the broking
inside US will be beneficial as internet dealing is free of tax there.

Cheesle

The project that I am thinking of is rather interesting, and could be made
public domain when done.  It also would be easy to sign-off, so is well
suited
to a "world-wide tender".

About nineteen years back, I wrote a program that allows you to define
screen sections in a data-base.  It is very similar to the MF "panel"
approach,
where each screen is defined as a fixed background and multiple overlays.

The overlays consist of Cobol pictures, each with a from and to dataname,
and 0-4 attributes.  The system allows you to paint the screens, and build
a screen database for each "system".  It then writes all the screen
sections.
Screen I/O is all accept/display using the names from the database. The
database is standard Isam, and the code includes a dump to a nice flat ASCII
file, as well as a reload database from a flat ASCII file.

About seventeen years back, I converted my library to PC's.  I used MF Cobol
running on an Alto's.  My first step was to convert my screen generator, and
use
it to generate all new screen sections.  At the same time, I re-wrote the
system
itself so that it generated all of its own screens.  So the screen system
screen I/O
is written by itself.

I have followed that basic conversion pattern through half a dozen migration
patterns.  The last was to Fujitsu, and I now have hundreds of programs
converted
to Fujitsu code using the Fujitsu screen section (ugly, but they work and
are
completely tested as standard Cobol).

The next step is to re-write "Screen" to produce SP2 code.  Testing and
acceptance
is simple.  It has to re-write itself successfully, and run as a GUI when
done.
Obviously, there will be no new features in the first new version.  All
screens
will be a mimic of an 80*24 character mode screen, and all data-editing will
be according to standard Cobol picture strings.  However that step will give
me
a basic conversion of all existing code, a solid example, and allow me to
proceed by either extending Screen, or starting to convert the existing code
using a more "GUI" oriented tool.

It also has the advantage of no re-training, and I can just change the
illustrations
in the documentation to upgrade the documentation.
