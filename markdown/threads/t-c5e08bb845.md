The problems lie on the PL/I side of things. What kind of data is it expecting?
If you are simply passing elementary data items (comp, comp-3, display, for
example) there should be no problems. But if you need to pass an array (table)
or structure (like a record layout), then PL/I normally expects to be passed a
descriptor block that describes the data and points to where it is. The
simplest approach is to compile the PL/I programs with the parms declared as
having the COBOL attribute. Of course, if the routine is being called by PL/I
programs, then you have some difficulties; two possible solutions: maintain a
call-from-COBOL version and a call-from-PL/I version, or establish alternate
entry points in a single module, one for being called from PL/I and one for
being called from COBOL.

My response to almost every question is 'it depends'. In your case,
the answer depends on the runtime environment (LE or not LE) and the
compiler versions you use.
Supposing a current COBOL compiler, LE and PL/I for MVS, you have to
consider:
- The parameter mapping (unaligned/aligned vs. SYNC, PTR vs. direct)
- The data type mapping (there are PL/I data types without direct
equivalent in COBOL)
- The OPTIONS statement of your PL/I programs

First of all, let's have a look to PL/I source-code. The 'pre-existing
PL/I procedures' might have OPTIONS(MAIN) or OPTIONS(COBOL) in their
PROC-Statement. If they have, the call works fine. Calling a MAIN
program causes LE to create a child enclave which is a performance
issue. The only acceptable OPTION is COBOL. If none of these OPTIONSs
are code in the PL/I program, you cannot call them from COBOL. Period.
If the routines contain a RETURNS statement in the PROC statement, you
cannot CALL them from COBOL since such programs are not procedures but
rather functions. And you will not be able to get the return value set
by PL/I.

PL/I procedures often accept pointers to the parameters rather than
the parameters itself. In a COBOL program, you have to build the
address of the argument and CALL the PL/I routine USING this pointer.
Since both the argument and the pointer reside in the WORKING-STORAGE
SECTION of your COBOL program, you need a subprogram to build this
pointer unless you use the most current COBOL compiler which allows
SET ADDRESS of WORKING-STORAGE items.
Even if you get a PL/I sample how to call the routine, you have to
look at the PL/I source of the routine. PL/I allows the following:
calling program: DCL 01 parameter, 05 ....
                 CALL pliname(parameter);
called routine:  pliname: PROC (ad_parm);
                 DCL ad_parm PTR;
                 DCL 01 parameter BASED(ad_parm), 05 ....
If you CALL this same routine from COBOL, the statement must look like
CALL routine USING ad-parm
where ad-parm is a POINTER to the parameter.
And, as I wrote above, the entry of the routine must be changed to
pliname: PROC (ad_parm) OPTIONS(COBOL);

If PL/I expects a structure (or a pointer to a structure), you need to
have a look to padding bytes. PL/I alignes by default while COBOL does
not SYNCHRONIZE by default. And SYNCH even works differently in some
cases than PL/I ALIGNED does. So, you have to look carefully at the
layout of the structure.

Another problem is the different meaning of the 'S' in the PIC clause.
A COBOL PIC S9(2) mapps to a PL/I PIC '9R'. A PL/I PIC 'S9' mapps to
COBOL PIC S9 SIGN IS LEADING SEPARATE.

You should read carefully the ILC guide. You can access the manual at:
http://publibz.boulder.ibm.com:80/cgi-bin/bookmgr_OS390/BOOKS/CEEA4100/11.0

I would probably write small PL/I interface program, one for each
routine. Every interface has 'OPTIONS(COBOL)' in its PROC statement. I
would then link the interfaces to the COBOL program. The interfaces
then can FETCH the PL/I routine. In this architecture, you can leave
the routines as they are (since they probably are called by PL/I
programs too).
