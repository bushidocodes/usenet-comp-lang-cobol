Tron Clark wrote:
>I would like an indepth understanding of the y2k issue. I hear it dates to
>the puncard days, but that's all I can find on the net. In programming
>language, just what, where, why and how is it that this is an ISSUE? ThkS

The basic problem is that to conserve space dates were commonly stored
with only two digits for the year, omitting the century.  For example,
today's date would be stored on file as 990109.  This causes problems
in generally two ways.

First, programs do a lot of comparing of data fields, including dates.
If a program compares 981201 (Dec. 1, 1998) with 990101 (Jan. 1, 1999),
the latter is clearly greater, or later.  As long as both dates are in
the same century, everything is okay.  But if a program compares 991201
(Dec. 1, 1999) with 000101 (Jan. 1, 2000) the latter is smaller, and
appears to be earlier, not later.  If those dates represented payment
due dates, or some such, it would cause the comparison to give a wrong
result, and the logic of the program would not work correctly.  The
result would depend on what the program is doing and how it is written.
It is easy to fix these problems by changing the year to four digits,
then the compare will be on 19991201 and 20000101, the latter now is
clearly greater and later.  A no-brainer fix.

The second way this can cause problems is in date calculations.  Every
program which calculates interest, or rental fees, or anything based on
elapsed time over days, will normally calculate elapsed time using some
sort of date calculation.  However the calculations are done, it should
be easy to see that when you use a two digit year in such calculations,
and the year changes from 99 to 00, it is going to look like you went
99 years *backwards*.  There is no doubt in my mind that someone,
somewhere, sometime in early 2000, is going to get a bill for negative
99 years and 11 months interest!  Correcting these type of calculations
is more subtle, because you cannot just plug a four digit year into a
calculation designed for a two digit year and expect it to work.  The
calculation must be examined and corrected or replaced with code which
does what the program needs to do at that point.  This is a thinking
fix, and the programmer must be very careful and thorough.  Another
common date calculation is to calculate the day of the week from the
date.  Again, replacing a two digit year with a four digit year does
not automatically make these calculations work; the algorithms are a
bit different.  The pattern of weekdays is different between the
20th century and the 21st century (Jan. 1, 1900 was a Monday and Jan.
1, 2000 will be a Saturday), and 1900 was not a leap year, but 2000
will be a leap year.  The rule is that years divisible by 4 are leap
years, unless they are divisible by 100, in which case they are not,
unless they are divisible by 400, in which case they are.  1900 is
divisible by 100 but not 400, so is not a leap year, but 2000 is also
divisible by 400, so it is a leap year.

Virtually all Y2K problem areas are variations on these two issues.

Tron,

Judson left one thing out of his excellent reply:  The third basic
problem is date storage.  The programming fixes he mentions address the
*logical* problems with two-digit years.  There is another layer of y2k
work needed to fix the data *files* containing two-digit years in record
keys.  Many data files use the date as part of the file key so the
natural order of a file is chronological.  (For example, the transaction
detail file on a credit card system will likely have the date/time as
part of the unique key to the record.)  Unless these data files are
re-created with expanded years, new records will be stored 'before' the
old ones, which may cause other types of program misbehaviors.

This is a variation of the problems Judson mentioned, but you can see
how the problem keeps getting bigger.
