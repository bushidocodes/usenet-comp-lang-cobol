```
Actually, ALL standard COBOL compilers accept redefines at the 01 level,
if the 01 level is under an FD, and the redefines is implicit.

When you are redefining for tables like this:

 01  TABLE-VALUES.
     03  FILLER     PIC  X(05) VALUE "?????".
     03  FILLER     PIC  X(05) VALUE "?????".
     03  FILLER     PIC  X(05) VALUE "?????".
     03  FILLER     PIC  X(05) VALUE "?????".
 01  TABLE-ELEMENTS REDEFINES TABLE-VALUES.
     03  TABLE-ELEMENT  OCCURS 5 TIMES
                        PIC  X(05).

you should *NOT* redefine at the 01 level, because the above
error (should only be 4 OCCURS) will pass with no syntax error.
Do the redefines at a subordinate level, like this:

 01  TABLE-STUFF.
     03  TABLE-VALUES.
         05  FILLER     PIC  X(05) VALUE "?????".
         05  FILLER     PIC  X(05) VALUE "?????".
         05  FILLER     PIC  X(05) VALUE "?????".
         05  FILLER     PIC  X(05) VALUE "?????".
     03  TABLE-ELEMENTS REDEFINES TABLE-VALUES.
         05  TABLE-ELEMENT  OCCURS 5 TIMES
                            PIC  X(05).

and the above error will get a syntax error.  The reason for this
is that there is no requirement that 01 level entries which are
redefined be the same size.  If your OCCURS or your PIC sizes in
the two sections do not match, the compiler will not tell you this.
But if you redefine at a subordinate level, the redefining entry
cannot be larger, and the compiler will flag the table if it is.
Unfortunately, COBOL 85 does allow the redefining entry to be
smaller.  But you at least get a one-way check, which is better
than no check at all.  I have seen many such program bugs get
overlooked when redefining at the 01 level for this purpose.
```

```
Ed, it is very simple.  When you define a data item with values intended
for use in a table, then redefine that data item in order to create a
table of OCCURS items, the tables should be the same size for three good
reasons.  If the OCCURS is too small (or too many values), the redefining
entry will be able to access unitialized storage.  If the OCCURS is too
great (or too few values) there will be values inaccessible using the
redefining entries.  If either of the table elements, the ones with the
values, or the redefining ones referencing the values, are accidentally
declared to be the wrong size, then you get an even worse situation,
for the references are then skewed in relation to the data.  I have seen
this sort of error repeated many, many times in many, many programs.  It
is unavoidable, unless you never create tables this way, or you are
perfect and make no coding errors.  On vary large and complex tables, it
is very likely to happen.

Please note that all of these errors would be caught if the compiler gave
a syntax error when the redefining entry was not the same size.  Using
COBOL 74 or earlier, you could place the REDEFINES clause on a level 02
through 49 and the compiler would do just that.  But beginning with COBOL
85, the restriction was relaxed so the redefining entry could be smaller.
This only gives you that safeguard in one direction: if the error makes
the redefining entry larger.  Since you are as likely to be in error one
way as another, we have lost some safeguard against this kind of error.
However, people who use redefines at the 01 level for this purpose, have
thrown out all possible help the compiler would give in this case anyway.
That's why I recommend not to do it.
```
