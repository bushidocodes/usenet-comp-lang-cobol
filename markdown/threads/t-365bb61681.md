Ben,

I suspect that the problem is that the RTS32.EXE that you have created has
been linked with the static run-time library. As that library exists
primarily in order to allow standalone modules it does not contain the
support for loading INT/GNT code, hence the error you saw. If you relink the
EXE using the shared run-time (which is the default) that should, hopefully,
solve the problem.

Ben,

Sorry for any confusion caused by my previous note, the static run-time
library is mfrts32s.lib, the shared run-time library is mfrts32.lib. So, you
are using the correct one.

The LINK statement you specified includes cbllds.lnk, which is the output
from running CBLNAMES.EXE. The command for that is not in your reply but it
is worth mentioning that you only need to run it and include its output in
the link step if COBOL objects are included in the EXE. However, that in
itself should not cause a problem unless the -M option is specified on the
command line to CBLNAMES. If you do that CBLNAMES creates its own main()
which could explain why nothing happened when you ran the EXE. So, what I
would suggest doing (which is what I tried, successfully) is :

link -out:rts32.exe -subsystem:windows rts32.obj mfrts32.lib...

or

cblnames -s rts32.obj

then link exactly as before.

If that still does not work I suspect you may have an installation problem
on your NT machine given that you saw an error message on Windows 95. The
error is displayed by mftxtwin.dll (or mfscreen.dll if mftxtwin cannot be
found). Check that they are both in the same directory as mfrts32.dll and
that the directory containing them is included in the PATH environment
variable.
