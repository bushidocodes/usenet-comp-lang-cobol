I was just reading the users guide for IBM COBOL for MVS.  It appears
that NEXT SENTENCE is replaced by CONTINUE.  No big deal.

But then I got to the section on object oriented COBOL.  Now in my
search for my masters, I have had 2 C++, 2 JAVA, and 2 theory of OO
courses.  The professors talk about "obsolete" databases such as
relational (I use IDMS).  So I am familiar with OO.

But I don't get it.  Who uses mainframe (or other) object oriented
Cobol, and why?

I am a big defender of structured cobol and hierarchical databases.  I
see problems with fully integrated environments (but like Integrated
data dictionaries).  I like the extract, sort, report means of giving
users what they want.  So I am biased.

Is Object Oriented Cobol a marketing gimmick? Why should someone use
it?  I don't think Cobol programmers can be trained to write good OO
Cobol code significantly quicker than it would take them to learn to
write good OO Java code.

This is not quite true. CONTINUE behaves like a no-op.
NEXT SENTENCE like a go to where the destination is the statement
or action following the next period.

Notice, in the following code, how NEXT SENTENCE may be used
with a period to implement "exit perform" and "exit paragraph".
CONTINUE cannot be used in this manner.

------------------
       program-id. nextsent.
       data division.
       01 x pic s9(4).
       01 char pic x.
       procedure division.
           display "start"
           perform varying x from 1 by 1 until x > 100
               if x > 50
                   next sentence   *> same result as exit perform
               end-if
           end-perform
           .
           display "end next sentence " x
           perform varying x from 1 by 1 until x > 100
               if x > 50
                   continue    *> does not exit
               end-if
           end-perform
           .
           display "end continue " x
           perform paragraph-test-next
           display "end paragrpah next sentence " x
           perform paragraph-test-continue
           display "end paragraph continue " x
           accept char
           stop run
           .

       paragraph-test-next.
           move 1 to x
           if x < 50
               move 50 to x
               next sentence   *> same result as exit paragraph
           end-if
           move 100 to x
           .

       paragraph-test-continue.
           move 1 to x
           if x < 50
               move 50 to x
               continue    *> not the same as exit paragraph
           end-if
           move 100 to x
           .

> NEXT SENTENCE like a go to where the destination is the statement

Actually in COBOL, NEXT SENTENCE cannot be used this way.
NEXT SENTENCE is part of the IF statement and the standard
specifies:

     "If the END-IF phrase is specified, the NEXT SENTENCE
     phrase must not be specified."

Without the END-IF a "." would be required to terminate the
IF and thus the construct would be uncodable.

Microfocus has an extension where NEXT SENTENCE is a stand-alone
statement and not just a phrase of the IF statement.  I have
no idea what the next standard will have, but if you are 
using extensions then you should stick to a meaningful one
such as MF's EXIT PERFORM which _will_ become standard.

The other problem with using NEXT SENTENCE in this way
is that the behaviour will chenge if the full stop is
removed.  This has all the disadvantages of a GO TO 
without any of the advantages.  At least with a GO TO the
destination is a label (paragraph name) and this can be
searched for to see if it is being used.  With this
misuse of NEXT SENTENCE there is nothing to signal that
the "." is actually being used as the target of a GO TO
disguised as NEXT SENTENCE.

In this respect the 85 standard is correct in bannishing
its use in this way.

A couple of bits of information:

A) The next Standard (as currently drafted) STILL prohibits a NEXT SENTENCE
before an END-IF (or END-SEARCH)

B) Micro Focus has an extension that allows NEXT SENTENCE as a "verb" so it
can be used within other verbs besides IF and SEARCH

C) The extension to allow a NEXT SENTENCE before an END-IF (or END-SEARCH)
is really quite common (and is much more common today than the EXIT
PARAGRAPH/SECTION) extension.  Therefore, using code (like the original
example in this thread) is reasonably portable - but is still NOT standard.

D) NEXT SENTENCE is really incredibly error prone wherever it is used and
(IMHO) should be avoided whenever possible.  Although the use of CONTINUE
cannot be automatically inserted to replace NEXT SENTENCE, I would strongly
suggest that no new code be done with NEXT SENTENCE and existing code be
examined for modification to use CONTINUE.  (I think that NEXT SENTENCE was
put in the "archaic" list in the draft Standard - but I may be wrong about
this - it was fairly controversial.)

B) If you have an ANS'74 conforming program with NEXT SENTENCE, it is true
that you can (but need not) replace any NEXT SENTENCE with CONTINUE.
HOWEVER, the two are by no means identical when coding new ANS'85 conforming
code.

NEXT SENTENCE *always* goes to the next period.

CONTINUE *always* acts as a "no-op" which lets the program flow continue
(which may or may not go to a period)

In ANS'74 (before END-verb scope terminators and NOT exception phrases, the
only place you could go after a NEXT SENTENCE in a conditional statement was
to a period.  However, with the new structures, it is possible to CONTINUE
on and not get to the period for several statements.

With regard to the CONTINUE - we use this heaviliy in our generated Cobol,
as well as EXIT to avoid using full stops in generated logic. Here is an
example...

A100-OUTPUT SECTION.
    MOVE A-VARIABLE        TO ANOTHER-VARIABLE
    PERFORM D100-CURSOR

    CONTINUE.
A100-OUTPUT-EXIT.
    EXIT.

B100-INPUT SECTION.
......

The CONTINUE is used when the logic "continues" and EXIT is used when the
logic "EXITS" if you understand! This stops the compiler objecting to the
missing sentances and means never having to code a fullstop (period) unless
you build your own sections (Casegen allows you to add your own code to
sections generated by the product). This may not work on all compilers - it
is designed for IBM mainframe and MicroFocus compilers.

Of course CONTINUE and EXIT do nothing procedurally, its just that the above
sections will be performed from elsewhere.

We also use -EXIT after every section as a common exit point for the
section. We do not generate or advise using a GO TO anywhere else! Notice
also that the TO in MOVE statements is always on column 40 (FROM in col 38).

mini-program.
