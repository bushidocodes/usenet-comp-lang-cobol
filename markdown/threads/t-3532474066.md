I've had the same experience with ViaValidate from Viasoft.  The problem
seems to be that native COBOL date verbs in IBM OS/VS COBOL and VS COBOL
II (and the IGZEDT4 callable routine) use SVC 11 (Supervisor Call) to
get system current date.  But the newer intrinsic function calls
actually use a PC instruction instead (Program Call).

The SVC instruction can be trapped by the date override tool, but the PC
instruction cannot be.  This problem makes it much harder to test the
intrinsic FUNCTION CURRENT-DATE.  Unless these date override tools can
be upgraded to intercept LE/MVS PC instructions, your only option is to
test on a machine or LPAR that had been IPL'ed with a future date.

I don't know if Hourglass has the same problem, but it sounds possible.

The system date can be retrieved in numerous ways externally via utilities or
programming languages. Most programming languages provide a reserved
word or function that returns the system date in a useable format (such as
COBOL's CURRENT-DATE). However, internally there are only three
ways to retrieve this date.

The first, and by far the most common, is via a SVC 11 call. Another less
common method, used mostly by the operating system level code, is the STCK
instruction that returns the number of microseconds since January 1, 1900.
Lastly, a special format of the TIME macro (LINKAGE=SYSTEM) which issues a PC
instruction can be used. This last retrieval method is rarely used since it
wasn't available until MVS ESA Version 4.1.

I want to thank all the people who corrected my misinformation.  I
leaped to an incorrect conclusion.

In early 1997 I was playing around with an assembler date program to get
current date with 4-digit year.  I'm much better at COBOL than
assembler, and I chose to use a TIME macro with LINKAGE=SYSTEM, because
it was easier to convert the results into zoned decimal numbers.  From
the assembly, I knew that LINKAGE=SYSTEM translated into a PC
instruction rather than an SVC 11.  And I found that ViaValidate would
NOT override the my assembler program, but WOULD override the date for a
plain old COBOL II program, or an assembler program with an SVC 11
instruction.

Several months later I finally had an opportunity to play with
COBOL/MVS.  I wrote a small program that used FUNCTION CURRENT-DATE as
well as the older ACCEPT DATE.  I found that ViaValidate could override
the ACCEPT date, but not FUNCTION CURRENT-DATE!  The same program, using
two different methods, got two very different "current" dates.

My mistake was leaping to the conclusion that FUNCTION CURRENT-DATE used
a PC instruction.  I don't know if the most current version of
ViaValidate can override a PC instruction or handle STCK.

Even though there may be some problems, you can still do some useful
unit-testing with date override simulators.  But you can only test your
applications this way, not your underlying operating systems software.

The following reply from one of my sources is quite similar to the one from
Arnold.  I think it sort-of explains the problem.

"I do not have access to a system with Tick Tock, but can make the
following comments....

The common time shifter routines insert themselves in the SVC 11 path in
the nucleus.  The products differ primarily in security, features (i.e.
does time freeze / move, does date march correctly with virtual time,
etc), selectivity, and so on.  By intercepting the SVC 11's they should
be able to handle "well behaved" programs which reference time and date
values thru the SVC 11 / TIME LINKAGE=SYSTEM.  OS/VS COBOL does all of it's
time reference thru module ILBODTE, which uses the SVC 11
mechanism.

VS COBOL II also uses the SVC 11 interface, in module IGZEDTE.

LE related functions, including the MOVE FUNCTION xxxx use the
SToreClocK hardware instruction to obtain the system date.  The SVC 11
interceptor cannot grab this - although it can grab TIME FORMAT=STCK...
a closely related MVS service.  Some of the time shifters have specific
patches to the LE module containing the STCK instruction, changing it to
a TIME FORMAT=STCK they can then deal correctly with the LE time/date
related functions.  Messy, but allows the LE code to avoid issuing
SVC's.  By doing this, the LE routines must also deal with the current
GMT offset and need to dig in the CVT to get this.

Bottom line if MOVE FUNCTION is not shifted, either the product is low
end (and does not support LE), or the (ECHHHHHHHH) modification to LE
has not been installed."

Gday Ken
I see from your email address that you're in Ozzie so, as the Tictoc
distributor for Oz & NZ, I guess I should answer.  Tictoc has no problem at
all with anything that uses SVC 11, TIME, STCKSYNC, LINKAGE=SYSTEM, CICS
ASKTIME, DB2 CURRENT-TIMESTAMP or anything else.  The one thing that shafts
every date simulator is programs that use the STCK instruction as this
doesn't generate an interrupt for the simulator to trap.  This is easily
solved by replacing the STCK with Tictoc's SVC which then can pass control
to Tictoc's simulation routines.  It is a simple zap, overlaying the first
two bytes of the STCK with the SVC instruction - you don't have to extend or
re-link anything.

We provide zaps for about half a dozen products that need this support
(including LE370, LE390, DB2 and some 3rd party stuff) as well as supplying
a macro that you can assemble into any assembler programs you have that use
STCK.  They are all documented in the installation manual, they are on the
install tape and we have them available for emailing if required.

So you're wrong to say that Tictoc doesn't support those verbs - it supports

DB2 is involved.  Where were you working when you tested Tictoc?  We never
