Avoiding GO TO is a religious belief and it is wrong. It is not rooted in
performance or other real considerations. GO TO is often the best way to
code in COBOL. You should not write C like a COBOL programmer but you should
also not write COBOL like a C programmer. People should code idiomatically
and not try to make everything look like Pascal, C, etc. And let's be
honest, by the time your program is compiled all those structured verbs are
really nothing more than piles of GO TOs, usually 4 times as many as you
would have generated if you simply coded GO TO in the first place because of
the extra loops you have to code to be able to say you avoided GO TO.

Blind obesance to Dijkstra's idiotic structured programming tenets will get
you nowhere in the real world except long wall clock times. Do you really
want to code according to the principles of a guy who never delivered
anything in his life but reams of paper and diatribes about things he wasn't
even involved in? He and his pals did more damage to commercial programming
than any other single thing I can think of, maybe with the exception of the
H1B program.

The alternative to GO TO in COBOL is PERFORM which is much more costly than
GO TO and most often contributes to poor program design by making everything
a loop. Yes, most commercial processing is a big loop, but it is not
necessarily concentric circles of loops. Structured programming assumes
everything is algorithmic and consists of concentric loops. This is simply
not a valid assumption for commercial COBOL code. Use the LIST option and
compare the object code from a small sample program you write in two
different ways, one using GO TO sensibly and one using PERFORM and avoiding
GO TO entirely. Consider how much overhead those extra 5 instructions will
make in a 5 or 10 million record batch run. Then consider how many more
loops you had to code to do things you should have handled with GO TO like
checking return codes from called routines, VSAM file status, etc. A good
program is a good program and branching is part of life. If you can't write
good COBOL with GO TO and PERFORM (very sparingly) then yes, you should
focus on academic arguments and FUD to cover up the fact you have no idea
what the compiler or OS are doing (see your opening question).

If being a mindless lemming or meme is considered a virtue where you work
and it wouldn't be the first place that was true, by all means avoid GO
TO. It is often much safer to do the wrong thing that is now a standard than
to do the right thing that isn't popular with the know-nothing generation.

On Apr 16, 7:36 pm, Nomen Nescio <nob...@dizum.com> wrote:
> justmainfra...@GMAIL.COM (Jake anderson) wrote:

> Avoiding GO TO is a religious belief and it is wrong. It is not rooted in

The problem with GO TO is not the GO TO statement at all, but the fact
that it requires a paragraph label as the target.

When examining a program written by others that contains GO TOs, the
GO TO statements are perfectly clear as to where the logic paths flow.
However, at a label it is not easy to determine all possible logic
flow. The label may be dropped through, it may be the target of a GO
TO, of a PERFORM or of a THRU. The _only_ way to resolve this is by
examining the _whole_ program and constructing a detailed flowchart.

In other languages the 'labels' are of different types. It is not
possible to 'goto' a procedure nor 'perform' a label that can be used
for a goto.

If the use of labels is restricted in COBOL programs so that labels
can have only one mechanism for getting to that place, and that this
can be easily assured, then later programmers will have a much easier
ability to determine logic flow, and can do this at a local level
without having to examine and plot the whole program.

If we are examining a program for the first time and I see:

     Calc-AR00.
        some calculations.
     Calc-AX00.
        some more calculations.

Then I need to determine the relationship between the two sets of
code, that above and that below the label. In the whole of the rest of
the program there may be any and all of:
     PERFORM Calc-AX00
     PERFORM something THRU Calc-AX00
     PERFORM something1 THRU something2 (where Calc-AX00 is between)
     GO TO Calc-AX00
     GO TO something (above Calc-AX00)

For example if I wanted to add some code related to that as a new
paragraph to be performed then I may, or may not, be able to add it
locally between the end of Calc-AR00 and the start of Calc-AX00 and
this would depend on what is in the _whole_ of the rest of the
program.

If the program had been written in a style that did not allow GO TO or
THRU or PERFORMed SECTIONs, and was entirely based on PERFORMed
paragraphs (or CALLs) then this can be assured by simple text
searches. The logic flow around that label can then be determined
unequivocally without needing to plot out the complete program.

'GOTO-less programming' is not about the GO TO, it is about isolating
and localising the logic into more easily understood procedures.

The point about eliminating GO TO, THRU and SECTIONs, is that a simple
text search, or grep, can confirm that one particular style has been
enforced and give assurance about the level of isolation.

Users of GO TO will never discover this, nor could they even imagine
how this is possible, or why.

Some will have an intermediate step of having, say, sections where GO
TO is allowed, but this aside from being unnecessary since ANS85,
removes the opportunity of having clean code.

> You should not write C like a COBOL programmer but you should
> also not write COBOL like a C programmer. People should code idiomatically
> and not try to make everything look like Pascal, C, etc.

That may well have been a reasonable statement in the 1970s when it
was not possible to code COBOL in a reasonable manner. But in the last
quarter century (since COBOL'85) it has been possible to move to more
modern and more productive idioms.

> And let's be

> the extra loops you have to code to be able to say you avoided GO TO.

Compiler technology has also moved on since the 1960s.
