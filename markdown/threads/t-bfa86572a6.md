Paul - as Pete said THERE IS NO LIST OF INVOKES FOR COBOL.

a INVOKE is basically calling a OBJECT and doing something with it -
being it getting a new instance of a object, or executing a METHOD or
setting/getting a PROPERTY.

those METHODs or PROPERTIES vary according to the object - e.g. Excel
will have its own list of methods and properties, Word will have
another list, and for example Farpoint Spread will have a completely
diff set also.


so the so called list of invokes you are looking for is NOT on COBOL,
but rather on the object(s) you are using.
This link will give you SOME of the Excel object model overview - all
info you need is there or around it - but as mentioned before by Pete,
you can also do that if you look at the object browser within Excel
itself.

http://msdn.microsoft.com/en-us/library/aa272268(office.11).aspx

How you adapt the object to be usable by COBOL is another story - in
COBOL you need to do a bit more work than on VB or other languages -
but only because of the way objects are defined in both languages.

But all objects and their methods/collections/events/properties can be
worked in COBOL - you just have to learn how to do them - and once and
for all get into your head that there is NO LIST OF INVOKES suported
by MF. - what you have is a list of things that the Object you are
dealing with has - and for that you look on the documentation of THAT
Object.

external to you program and they are external to the OO COBOL compiler.

the world, and those methods are not inherently known to the OO COBOL 
compiler.  You would be better served by learning to use an Object browser 
like the one built into Visual Studio.  That way you can search through and

no longer work in that world. Flowcharters and such already exist but the 
ones I've seen can't give a "bigger picture"; it takes a programmer to do 
that. COBOL was designed to be "self-documenting" and, when written well, it 
is.

The automatic documentation produced with languages like Java and C# is 
adequate, provided the actual comments get written, so again, it comes down 
to the programmer.

I think the actual volume of code has a lot to do with it. Small 
encapsulated pieces of code  are much easier to get your head around (with 
or without added documentation) than large monolithic programs with many 
thousands of lines of code. For example, today I was working in some C# I 
wrote 3 years ago. It simply isn't a problem. The toolset lets you set book 
marks and jump around the code to your heart's desire. You hit the 
equivalent of a performed routine in COBOL (it's another method being 
invoked) and you need to check exactly what it does... put the cursor on the 
line, click on the bookmark toolbar icon, then right click and select "find 
the definition" from the pop-up context. Instantly, you are at the routine. 
No scrolling through code or opening other windows, just two mouse clicks. 
Read through the method, amend it if you want to, then click the "go to 
bookmark" icon in the toolbar (there are 2;one forward and one backward). 
Instantly you are back where you started from on the very line that you 
jumped out of. You may never know where the method you went to is located in 
the code (although it is easy to mark if you want to you) and it doesn't 
matter. You can track through a logic path without getting sidetracked by 
having ot search for "out of line" code. Not only can you "find the 
definition" but once you are in that code you can "find all references to 
this code" just as easily. Visual Studio simply rocks and we should have had 
tools like this for COBOL decades ago.

I had some fairly complex changes to make to some stuff (extending some 
existing methods and adding some new ones) and I tried to assess how long it 
would have taken if I was doing it in COBOL. For a start the existing code 
would have been 20 to 30 thousand lines. In C# it is around 6, but you make 
the areas you are not working with invisible, and only open them up when you 
need to. (Another feature of VS 2008). I start off with all methods and 
global variable definitions hidden and just work through what I need. It is 
much quicker and less cluttered than having to have a full listing open all 
the time. (Searches can be configured to search hidden code as well as 
visible.)

It is a combination of much better tools, and much more powerful code 
(requiring much less writing and reading) that improves productivity with 
the new languages over COBOL, at least in my experience. There is a learning 
curve, and some of the concepts are new (plus there are new capabilities 
that simply aren't there in COBOL), and different from the way things are 
done in COBOL, but if you persevere you get round the curve and things get 
written and fixed much more quickly than they used to. It is simply easier 
to find and fix stuff. I enjoy working with C# now about as much as I 
enjoyed working with COBOL for many years, and I cannot deny that I produce 
much more working functionality (and often more complex functionality) in 
less time than I did with COBOL.

Having said all of that I still find myself thinking "EVALUATE" 
nostalgically,when I'm coding "SWITCH and CASE"... :-) Old habits die 
hard...:-)
