Mark,

Assuming that the 'internal sorts' are accomplished through the SORT verb
(as opposed to using an indexed file)...

Are you using the runcobol command's T option to increase the sort
tournament area to a large value?  If not, you might try that, rather than
accepting the default (256K).  The larger the sort tournament area, the
longer your initial strings will be during a large sort (or the more likely
that the sort will be entirely in memory for smaller sorts), thereby
decreasing the number of merge passes over the data.  If setting the
tournament area larger gives significant improvement, try using runtime
configuration (the C option; RUN-SORT configuration record) to increase both
the tournament area and the number of intermediate merge files used in a
large sort.  You can learn more about these options from the User's Guide.

You may add the T or C options to all of your 'runcobol' commands; no
recompilation is necessary.  The tournament area is not allocated until the
SORT verb is encountered, so it should have no adverse impact on nonSORTing
programs.  What _might_ be necessary is to tune your AIX user parameters
with regard to user memory per process.  You do not want the sort tournament
area to be paged to disk!  I am not a UNIX/AIX guru so I can't give specific
instructions...

Best regards,
Tom Morrison
Liant Software Corporation
