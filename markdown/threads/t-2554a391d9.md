```
You need to identify the platform you are running on for someone to
give you a complete answer.  Assuming you are running on a mainframe
and you're running IBM OS/390 Cobol, data type COMP means binary.
Therefore one binary digit is stored in each bit of the item except
the leftmost bit where the sign is stored.  So a PIC 9(9) is
equivelent to 4 bytes or a fullword.

If you remember your binary arithimetic, each bit is a power of two.
The rightmost bit is power 0, going to the left the next is power 1,
next is power of 2 etc.  So since PIC 9(9) is 4 bytes and there are 8
bits to a byte, (8 * 4) -1 = 31.  You need the -1 to account for the 0
power.
```

```
Still assuming Cobol for OS/390:

Although you can determine the sign by looking at the leftmost bit, this is
not strictly correct.  Negative binary numbers are stored in 2's complement
form, so that -1 becomes all 1 bits.

PIC 9(n) COMP - or the equivalent PIC 9(n) BINARY -  is stored in a halfword
(n=1-4), fullword (n=5-9) or doubleword (n>9) as a binary number, but COBOL
never the less defines the range in decimal digits.  The maximum positive
number that can be represented in binary in a fullword is X'7F1FFFFFF' or
2,147,483,647.  The maximum capacity defined by PIC 9(9) is 999,999,999.  If
you compile with TRUNC(STD) Cobol will generate code to truncate the result
by converting to decimal, selecting the last 9 digits, and converting back
to binary.  With TRUNC(OPT) you may still loose high-order digits.
TRUNC(BIN) was invented to support full untruncated binary representation.

By level 2.2 of Cobol for OS/390 a new usage term was introduced - COMP-5 -
which finally treats the binary numbers in the proper way - like BINARY with
TRUNC(BIN).

Decent languages defines binary number range in bits, not decimal digits.

Gunnar.
```

```
With IBM COBOL for OS/390 & VM V2R2 (but not earlier) COMP-5 does allow for
a "better" binary data type allocated by storage not PIC  (without needing
to modify TRUNC compiler options). It also provides for BETTER (still not
perfect) performance with TRUNC(BIN) and for COMP-5 data items.

In the draft of the next COBOL Standard, there are new USAGEs that provide
for "true" binary data types - without PICTURE clauses.  You might want to
do a SHARE requirement (or service REQUEST) to get these implemented by IBM
"sooner than later". (The draft also includes support for one-byte binary
fields, which have ALWAYS been a pain to handle in IBM COBOL.)

The draft standard also (finally) provides for "bit" data types in "standard
COBOL".  Although not needed by MOST "business applications" - this feature
will be nice for those who have "real world" needs for bit-twiddling.
Again, use the normal channels to tell IBM if this is important to you.

  ***

Unlike today's announcement by Micro Focus, IBM has made no public
announcement on when/if they will EVER implement the next ANSI/ISO COBOL
Standard.  If this *is* important to you, then you need to let IBM know.
Otherwise, you should let IBM know which (if any) of the specific items from
the draft you want them to implement in their future COBOL compilers.
```

```
I strongly recommend that you don't have numbers outside of what the CoBOL
picture addresses.  You cannot access it as a number without special
compiler options or tricks.

That said, I use IDMS, and it stores a database key as PIC S9(8) COMP SYNC.
that contains an 11 digit number.  So I created a called CoBOL program that
can convert it between this number and a format such as 04292500-005,
suitable for cutting and pasting into DBOL, or into JCL as a restart parm.
The program is based upon the following redefines:


    02  PASSED-DBCONV-X.
     03  FILLER                                  PIC X(4).

     03  PASSED-DBCONV-DBKEY    PIC S9(08) COMP.
..
01  HOLD-KEYS.
    05  HOLD-DBKEY        PIC S9(16) COMP.
    05  HOLD-DBKEY-A      REDEFINES HOLD-DBKEY PIC X(8).
...

MOVE PASSED-DBCONV-X TO HOLD-KEYS
IF HOLD-DBKEY > ZERO
    CONTINUE
ELSE
    MOVE ZERO        TO HOLD-DBKEY
END-IF.
DIVIDE  HOLD-DBKEY BY 256 GIVING PASSED-DBCONV-PAGE-N
          REMAINDER PASSED-DBCONV-POS-N.
```
