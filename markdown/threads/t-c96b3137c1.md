If you can be guaranteed that the LE-runtime will be there, then IN DEED,
using a call to CEE3ABD is "preferred" over calls to ILBOABN0.  However, if
you can't be sure whether LE will OR WILL NOT be there, then use ILBOABN0 -
which is there in ALL IBM mainframe COBOL run-times (including LE).

Now to answer your problem,  the name of the module has the letter "O" as the
4th character and the number "0" as the 8th character.  I think you coded
both as the letter.  (Hard to make obvious in quick NG notes - sorry for
misleading you)

Please NO - don't do this.

Getting a S0C7 is just as "unpredictable" as doing a divide by zero. (As
anyone who has EVER converted from OS/VS COBOL to COBOL for this-and-that can
tell you.)

Please do NOT assume that doing something that is explicitly UNDEFINED will
result (tomorrow or the day after) in getting an ABEND.  Yes, the Assembler
instruction MAY cause an system ABEND today, but there is NO guarantee that
it will with the next release, version, or even maintenance level of the
compiler.

Please DO use the documented (and supported) way of getting an ABEND.

On a related note - sometimes it's better to set a return code, close
files, and stop run. Then have Condition Code in a subsequent JCL step,
which executes some Utility ABEND-like program. This is not always
practical, of course.
The advantage is that your output files will contain the most recent
records so you might be able to use that info in figuring out the
glitch. Whereas on an abend, the files will likely be dispositioned
(NEW,CATLG,DELETE) so won't be there to peek at. True - Abendaid should
show the last records, as will the dump, but it's sometimes easier to
view the files.
In addition all your displays will show - I've noticed that sometimes
the SYSOUT buffer does not get flushed to the output JCL-etc listing
when an abend occurs (at least that's my guess at the technical reason).
This technique has been useful in situations when there was a program-
forced abend because some end-of-program control totals were not OK.
I.e., could put the output files thru some investigation using mini-
programs, SORTs, etc.
