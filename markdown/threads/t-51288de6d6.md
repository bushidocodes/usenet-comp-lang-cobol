OK, that's fair. You only have one insane person in your organization.

1. Easier to maintain? I don't know. In spite of bugs in COBOL programs, I
suggest it is easier to make a ghastly mistake in Java (and double-secret
ghastly mistakes are common in C). The COBOL language itself imposes a
syntactic and logical discipline that is sometimes lacking in other
languages, including Java.

2. Java programmers are not, in the main, business programmers. They don't
know an amortization table from a double declining-balance. So you have to
factor in the learning curve for financial business rules. The Java
programmers I've met (not many) tend to not be detail driven. While there
may be a greater labor pool for Java, the labor pool of Java programmers
willing to work on financial stuff is, well, small. Don't forget that credit
card transactions move at 2400 Baud over dial-up modems (well, 90% do
including 30,000 ATM machines). What Java programmer wants to work with
that? Yuk!

3. The keeper of the Java standard, Sun, is in, um, straits. You recall Sun
did a silly thing - they pissed on Micros~1's leg and Micros~1 has a long
memory. Further, Sun is finding it difficult to sell their medium-sized
hardware as Intel multi-processor chips invade Sun's market. The outlook,
then, for control of Java is not great.

4. One incentive for Java - write once, run anywhere - is not much of an
incentive for financial applications. The chances of a credit card
application running on an unsecured PC (or a refrigerator) is zero. And the
chance of Java not being able to run on ANY PC is, however, non-zero (see #3
vies-a-vies Micros~1).

5. I'm sure you've got a few Java-capable programmers; take one application
that no one cares if it gets screwed-up - say Master Card processing for
Bank of America or Just-In-Time inventory control - convert it to Java and
let it rip. Note the time, resources, etc. Then you'll have a specific case
for comparison. (Pay no attention to customer complaints - Philistines just
don't appreciate art.)

PS
Can Java even print a credit card monthly statement? Can it do it without
pop-ups?

PPS

There is inherent risk in being a "dedicated follower of fashion".

If the systems are performing well but the concern is future maintainability
and the diminishing supply of COBOL skill, then that is a different problem
from converting the systems. There are several solutions to it, none of
which necessarily involve converting the code.

If you see conversion as an opportunity to revamp system functionality and
move to a more modern approach, then you are on firmer ground. In this case
it would be sensible to isolate and encapsulate existing functionality into
sharable "building blocks" for the new systems. As the source language these
components are in is irrelevant (they won't be maintained; they just do what
they do, analogous to Java Beans), it is feasible for at least some of them
to stay in COBOL. (Later on, if you need to change the functionality you
could replace the old COBOL component with a Java Bean. It is better to do
"conversion" on a modular basis, when it is necessary, than to strive for
complete conversion of everything, when it isn't.) If you REALLY want to
convert everything to Java, look at PerCOBOL, which compiles COBOL code into
Java. Even though this is an excellent tool, it can't re-create your systems
as OO, and it would not be my choice for "converting" hundreds of thousands
of lines of COBOL into "procedural Java". That really buys you very little.

I would recommend that you keep your existing systems running while you do
some pilots. Take a single (non-critical) system and look at encapsulating
the existing COBOL code into components. There are tools available to help
you do this. Your Java shop will be Object Oriented and your existing
systems are not. That's why you need to give a lot of thought to what can be
salvaged from the present (and how). That is also why it would be disastrous
to simply convert your existing code to Java. You will end up with a system
that simply degrades Java and needs to be rewritten anyway.

In today's environment, COBOL components can run alongside components in
other languages without problem.This can extend the life of this
functionality until it NEEDS to be replaced. You don't say what your
environment is (Mainframe or client server); there are different tools for
these environments.

If you need detailed advice and help with both planning and implementing a
major conversion (consultancy), contact me privately.

I agree with this. Over the years, the size of my COBOL programs (compile units)
keeps getting smaller. Where others would write a single program, I'll write a
dozen or more callable programs. Whether they're nested or separately compiled
matters little. 

I just finished a project extracting data from reports. The difference from your
project is my reports were in thousands of unknown formats. On the front end, I
wrote a program for each level of deconstruction and reconstruction. The
document guy calls the line guy who calls the lexical guy ... Data bubbles
upward through the pipeline. 

document level -- the driver for everything below
  line level -- statistics
    lexical level -- all knowledge of vocabulary in one place
      word level -- a complex parser that handled ambiguities such as hyphens  
                         and words run together
        input level -- "read a record" with various line terminators and tab
                           expansions

During preanalysis, a number of programs handled configuration morphologies, one
program for each:

two up, like an old fashioned balance sheet
"two deep" i.e. right side of page below left side
all lines folded
description columns folded and hanging below primary line
rotated 90 degrees i.e. columns reading down rather than across
elided titles, sometimes multi-level, for example:
      HARDWARE
         Widget
             Metric
                left-handed thread     12
                right-handed thread   15
              SAE
         Be careful not to mistake category titles (e.g. HARDWARE) as 
         elided titles. Lexical analysis provided clues, as did statistics 
         (upper/lower case). 
         Sometimes the whole description is elided, for example:
         Widget             12   bin number 10
                                15   bin number 12
                                27

free format -- columns delimited by a single space
verbiage, legal and administrative stuff above and below report, 
          sometimes with numbers that coincide with report columns
          or supplementary tables which look like data
footnotes, written in a dozen notations, containing relevant information

Now that data for one report line is on a "line" (a linked list of words), the
analysis begins. 

Pattern analysis -- find the most frequently occurring formats, some reports use

      multiple formats e.g. one each for hardware, software and supplies
Header analysis -- column headings _may_ say what the column contains, 
       or may be misleading due to different tabbing standard (text vs. 
       spreadsheet)
Report typing -- figure out what kind of report this is, sometimes a header 
       states that, other times you 'just know' based on data being reported
Binding -- for each detail line, determine which format it resembles, then 
       bind each word to a column. If two words appear in a column of numbers, 
       there is a problem. 
Scaling -- figure out whether columns of numbers are in units or thousands, 
       sometimes the column header says, sometimes it's in a footnote, sometimes

       it's in text at the top ('all numbers in thousands except quantity'), 
       sometimes it just doesn't say, sometimes the header says but is wrong
Normalization -- if a column of numbers has some or all decimals or scientific 
       notation, figure out whether it contains percentages or numbers. 
       If numbers, transform them into whole numbers (without causing rounding
       errors).
Line classification -- get rid of subtotal lines which look like detail, 
       distinguished only by their description or lack thereof

Now that we have nice clean columns of numbers, Control procedures analyze the
result. Do columns of numbers add up to totals, which may be in administrative
heading rather than at the foot of the column, may be scaled differently, may be
slightly off due to premature rounding? Do lines crossfoot? Do numbers look
reasonable .. for that currency, report type, previous report (if any), external
reference (prices)?  If not, some lines may have been mangled. Change
assumptions, UNDO alterations and REITERATE the whole analysis. Keep trying
until you run out of things to try. 

Finally there are output programs, including several loggers and a program that
reports internals for debugging purposes. 

The application runs under a larger interactive system, thus it should finish
within 100 milliseconds. For extremely large reports with thousands of lines
(uncommon), one second is tolerable. 

The application ended up with 35 programs. Most were in the original design but
a few were patched on as experience grew. I *can* conceive writing such an
application as one monolithic program, but I know what would have happened. At
some point in its evolution it would have become too complex to comprehend.
Changes made to fix one problem would have broken something else. Reiterative
systems are especially vulnerable to that happening. Without high volume
regression testing, its success rate would have formed a plateau -- one step
backward for every step forward. With regression testing, it would have taken
increasingly longer to fix the mishaps than the original problems. That's what
happens to poorly structured designs.
