Though I appreciate your comments and understand how
ENTRY may be considered to be unstructured, the way in
which I am using ENTRY is structured and provides a clear
and precise definition (documentation) of the interface for
each ENTRY. Furthermore, it is much cleaner and more
understandable than the two other means that are available
to me (using standard COBOL or other extensions) and
that could be used to accomplish the same task.

I think this is a case where it is not so much how it appears
on a structure chart or other diagram but on how it is used
in practice.

My experience in working with ENTRY, during the past several
days has convinced me that it is an important and valuable
addition to the language. As a practical matter, whether it
makes it into the standard, or not, is irrelevent. No
implementation will make it on to my computer, if it does not
contain ENTRY.

Incidently, the particular program, I developed, that provided
the convincing evidence of the utility of ENTRY, contains
27 distinct data definitions in the LINKAGE SECTION and
38 ENTRY statements. It validates and converts many of
the common formats for dates and times. The first paragraph
in the PROCEDURE DIVISION does some initialization and
contains an EXIT PROGRAM as its last statement. This is
followed by 38 paragraphs, each containing an ENTRY as the
first statement in the paragraph. The last statement in each of
these paragraphs is EXIT PROGRAM. There are no GO TOs.
Following this is all the common procedures.

After careful consideration, I have determined that only one
other means would make the solution to this problem easier
or clearer. FUNCTION-ENTRY function-entry-name USING
data-name-1 ... RETURNING data-name-2. No implementation
offers this and I am not suggesting that FUNCTION-ENTRY
and ENTRY be (re)considered for the proposed standard.
Maybe next time!
