I am refering to a project as anything that PSTAFF gives a PRJ extension too
... what it defines as a "project".  Yes, you can have lots in one, but
often a complete system cannot be easily organized into one project, simply
because various modules cannot be started before others are finished.  For
example, you may have DLL's that will not compile until another DLL has been
compiled and linked correctly.  Since PSTAFF will not allow you to order the
compilation, putting both of them in the same project is a recipe for
disaster.

Example... Object A inherits Object B.  Pstaff has them listed as Object A
first.  I do a rebuild, and the library for DLL B gets deleted.  Object A
will no longer compile because it requires the repository of object B.
Object B will no longer compile because the build craps out at object A.

The nice way of handling that would be three projects.  Each object set up
as a PSTAFF project, then a third project that includes the two sub projects
in the correct order.  It cannot be done with PSTAFF, but the two projects
*can* be set up easily, and a batch file made that uses the make files of
the first two to create a master rebuild.

I have three systems I sell.  Each has about 300 programs/modules, but about
100 of those are common to all three projects.  If Pstaff was capable of
nesting, it would make life much easier.

Donald

KISS is something we all know about. At one time or another someone has said
"keep it simple..stupid".
Uniformity is a mark of simplicity.  Trouble with that is the people
deciding the uniform doen't wear it.
Case in point. On the Univac I, there were tapes, period. The OS was a
simple load the first block of a tape,
and it would contain any operational features needed. At our site, this
developed into a tape library with
project objects on a master tape with variables coded to allow a
pre-application process to substitute the
current values for the variables. Eash program on the library contained the
"first block" logic, and was enhanced to
provide the program or the operator to select and load the next program.

After several years of this kind of operation, the Univac people sold us a
"650" type machine for one of the plants.
Data was on a drum. We specified the same kind of system for this operation
to obtain uniformity. Can't be done was
the answer as the system was built to run from a card reader. Not enough
columns to put the procedures needed in
one card. We gave in, but insisted they develop a consitant method to
provide the consitnancy we desired.

A few years later, they came up with one, and acknowledged that it taught
them a lesson. Big deal. Product
planning was in it's childhood then. Often today's hardware support seems to
still be in it's childhood.

Otherwise, why would we need "implementor defined"?

Warren Simmons
