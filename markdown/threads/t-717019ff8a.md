```
Enough Y2K stuff, how about we shake up the committee with a few
suggestions?

Here's some of mine:

- COBOL II is too damn finicky about specifying the data record size
for variable length records. Just have it open and read the damn file
without a lot of fuss even if the specified size is a little off! Or
even a lot off! I worked on a routine this morning in which I wanted
to read a file, selecting only certain records. These happened to be
a lot less than the maximum length. So why should I have to reserve
the rest of the space when it'll never be used? We're converting a
lot of programs to COBOL II and this is the biggest holdup!

- A lot of vendors use Assembler modules to read files, so their copy
members include the length fields at the beginning. This makes them
useless in COBOL without some copy...replacing finagling. Have an
option in FD for something like "Data Record has Length Fields"

- Ever have to write 50-100 or more Move's? Put options to allow
Move Corresponding to work with different prefixes and to ignore
levels and just match elementary fields.

- Why can't "Accept" use any DD name instead of just SYSIN?

- Bring back "Exhibit Named" and "On"!

- I think every program should display its compile date/time when it
starts. Make this a compile option.
```

```
I used EXHIBIT NAMED in the old days. It was handy, but pretty
inefficient. In any event, I believe both EXHIBIT NAMED and ON (for
debugging) were IBM extensions, and therefore not standard COBOL. IBM

As Bill Lynch mentioned, this would not work very well for CICS or even
for batch subprograms. But most of the batch programs in my shop have
initialization code to do exactly this. It takes about 30-40 lines of
code, half of it data division, but it is easily copied from one source
file to another. The result looks something like this on the SYSOUT
console log:

SI339010-0000 ANALYZE TABLE SIZES FOR OVERFLOW
SI339010-0001 PROGRAM COMPILED 05/07/98 AT 17:39:28
SI339010-0002 PROGRAM EXECUTED 05/21/98 AT 03:01:22

It's very helpful for determining if the new version was actually
installed. Of course, now we're thinking about expanding those DISPLAYS
to use 4-digit years! That becomes much easier if you have the
intrinsic FUNCTION WHEN-COMPILED and FUNCTION CURRENT-DATE, which can
easily be edited into ISO 8601 standard format YYYY-MM-DD.

We have even had COBOL subprograms display their date of compilation,
but they have to be coded to only do that on the first call.
```

```
Almost none of the stuff that you are talking about has much at all to do
with *Standard* COBOL - but has a lot to do with how IBM has "chosen" to
implement COBOL. Do feel free to send any suggestions that you would like
for the future of COBOL to the Chair of J4 (the ANSI COBOL committee) at
d.··.@mic··s.com

Make certain that you make it clear whether you want the enhancement to be
in the next Standard (and you are willing to have the Standard delayed EVEN
longer than being official in 2000) or if you just want it to be considered
in some future COBOL Standard.

As for changing the way the IBM compiler works, there are two things that
you can do today to impact this:

1) If your company is a member of SHARE, then submit a "requirement" to get
what you want (and that will let you know how many other customers agree
with you - you may be surprised that something that you think is a "great
idea" seems awful to others).
2) Via your IBM support channel, submit a "REQUEST" - this requires IBM to
respond within 90 days as to whether your "enhancement" is something that
they might do - and if so whether they might do it in the long term or the
short term.

******

Just a personal comment,
It "bugs me" when people/users have a problem with a software product or
a Standard and they do NOT use the channels that are available to tell the
responsible person about it. Such input rarely gets an "instant" change,
but it is the only way to get what you want at SOME time.
```

```
Under cobol 2:
Only the Identifaication Division is required.
To nest programs:
The last line in each Program is End Program
Before the "END PROGRAM" for the outer most program,
insert one or more nested programs;
each begining with "IDENTIFICATION DIVISION"
and ending with "END PROGRAM".
Any of these programs can have one or more nested programs.
Typically each of these programs will have at least a DATA DIVISION
and PROCEDURE DIVISION.
Configuration Section may only exist in the outer most program.
Program Id may have the optional keyword "IS COMMON"
after the program name. This increases the scope of the program
up one level.
Files may be declared "GLOBAL" in the FD description, extending
scope to all contained programs including programs contained
within other contained programs.
Variables may be declared "GLOBAL" in the variable declaration, extending
scope to all contained programs including programs contained
within other contained programs. In addition all level 88 items under a
global are also GLOBAL and all higher level number items under a
group GLOBAL item are also GLOBAL.
Parameters may be passed to a contained program either thru
GLOBAL variables or thru parameters on the call statement and
defined in the contained program in its LINKAGE SECTION.

hope this helps
(I havent had a chance to use it yet, the place I am at is not
thrilled with using anything that is not already used in "production"
including inline PERFORM ..... END PERFORM
```
