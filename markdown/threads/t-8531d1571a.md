A friend of mine and I were just discussing these context sensitive
messages earlier this week. And you are correct, they are utterly
worthless unless you've had the problem before and have some idea about
it.

The new kids in development think that the xxxnnns or ssscccnns formats
are ancient and worthless (xxx being a three letter ID that matches the
component, nnn being a unique number of 3-4 digits and s being a
severity such as W/I/E/S. And then the sssccc being system and
component (seen in VM/CMS systems)).

Perhaps if more companies were aware of how difficult it is to diagnose
problems because the issuer of a message doesn't identify themselves,
they might force Micro$**t to do a better job with *all* their
software. But even better will be to force the OPEN world to do the
same thing! Try finding something like this in MAN pages!

Now more to your point, could it be that the way FJ goes about getting
to a certain folder it would appear to be a network read/write attempt
and so XP is complaining?

Otherwise, you may have to scan all the .exe and .dll files for the
majority of that message and then try to find why that particular
component is issuing it.

Regards,
Steve.T

Pete,

You may have stopped a necessary Crypkey service.  This is the service
running that manages the Fujitsu license.  You may need Fujitsu to
reset your key, revoke your license and reinstall.  It's possible JUST
a reinstall will do it as the Crypkey information should still be
there.  It may be that it's not because of the cleanup you performed.

Here's a link to a page offered by some other software about this
problem.  A Google search with your error message as the search
information is how I found this.
http://www.ceerelays.co.uk/PTW_FAQ/NetworkKey.htm

OK this is now solved.

There are a few lessons here so I'll summarise them as follows:

1. If you mess with the Registry, make sure you can restore it to the state
it was in before you messed with it.

(I know it's obvious, but I THOUGHT I could restore it, and later found
that the tool that was supposed to do that (WinXP Restore) didn't work...
 I used a tool called Registry First Aid. It is excellent and I decided to
register it. Before it runs it asks you if you want a snapshot saved and, of
course, I said "yes".

It detected that I was running Win XP pro and used the Operating System to
do the snapshot by creating a restore point.

The trouble is that this restore software is dangerous. It does incremental
restore points and if one of them can't be used for any reason, ALL of them
are useless. I had to get the facility working correctly before I could even
think about re-installing COBOL. Closing it down (suppressing restore
points) is supposed to delete all the restore points from a special system
folder called "System Volume Information". I decided to do this manually for
good measure, but couldn't because it is owned by the SYSTEM logon. I tried
booting into DOS but of course my file system is NTFS and, although I have
software that can see it, and copy it, it won't allow me to amend it. I
tried booting from a Linux CD but I didn't have enough knowledge of Linux
commands to delete the files, even though I could mount and see them.(It is
possible that it is limited to viewing NTFS only). So, finally I just had to
hope that windows had done it when I turned off the facility. (It had... but
I only found that out later.)

2. Don't defrag if you have Fujitsu COBOL installed.

I didn't actually DO a defrag as it wasn't required, but I believe just
opening this program MAY have caused my problem.


So, here's the problem and solution, summarised:

1. Ran Registry First Aid (RFA) and snapshotted a system restore point.
2. Checked for defrag; didn't need it.
3. Found COBOL development failed to work.(ALL programs...NetCOBOL,
PowerCOBOL, utilities, even the registration and registration
checker...nada, zilch.)
4. Tried restoring system from restore point in 1 above. Failed.
5. Tried the "Repair" facility of COBOL. It ran successfully but didn't
solve the problem.
6. Had a look at advice received in CLC, especially Thane's post. Looked
like a problem with the site key (which I had on file) but there was no way
I could get to enter it as registration utility wouldn't run.
7. Reluctantly decided to uninstall COBOL. Went to Control Panel> Add/Remove
programs... It failed. Could not uninstall.
8. Tried using uninstall from the original install disk. It grizzled about
some dlls it couldn't delete but it worked.
9. Re-installed from scratch. When it got to registration, tried using the
previous site code. (It was being re-installed on the same machine.) Failed
to validate.  Decided to ask Casper (the Fujitsu registration server which
is available 24/7) for another key. Casper replied within minutes (Good
stuff, Fujitsu!) saying he had no more keys and needed human intervention to
have more keys added to his database for this registration.
10. Paused for thought. Rebooted.
11. COBOL came back and said it was installed and registered OK! I did a
check registration and it said I was fine.

Everything currently works. IF it fails in 30 days, I'll get onto Fujitsu
and ask for another registration key.

I have a bad taste about this because I STILL don't know what the
undocumented message actually means, or what caused it, but at least my
COBOL development environment is back...

Hope this experience is of use to others.
