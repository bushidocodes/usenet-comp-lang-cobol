```
I am having some real problem with OO stuff (Fujitsu)
and am wondering if anybody can help me figure out
what to do.

I want to create an OO object in PowerCobol, and place
it in a DLL.  That object should have a method that will
wait for some user action, massage some data, and return
a file key to the invoking program.

The problem is that there appears to be no way that I
can get the DLL to wait for user input.

I create the DLL with the method fine.  I invoke it.
the form opens and starts running, and I am back into
the invoking program.  However, the user has not yet
done a thing, the form is running asynch.

That is fine, theoretically I can simply sit in a poll
loop invoking a second method that checks for a status
bit set by the user actions in the DLL.

However.  The "sleepex" idle routine inactivates both
the main program *and* the DLL, so if I do any kind
of idle, then the program hangs. If I do not do an idle,
then I am in a really tight infinite loop.  Either way
the program never gets to the user action stage.

If I open a screen section, and ACCEPT data from the
keyboard, then the form activates nicely, and works fine,
as long as I keep the console in a keyboard wait state.

I can also open the DLL form as a completely different
thread, and wait for the thread to end by polling it's
PID.  However, that is as ugly as sin, as it leaves me
with no data communication between the DLL and the main
program except through a disk file or a named pipe.

What I really need is access to the idle that is used internally
in the screen section. If any of you OOP gurus out there
know how to handle this, I would appreciate it.
```
