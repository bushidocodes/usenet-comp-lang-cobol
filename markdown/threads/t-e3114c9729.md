It's likely to be platform independent, but I would expect writes to be
slower and reads to be basically unaffected, as John Langdon suggested.

Note that changing the content of keys via REWRITE can cause significant
problems.  I wouldn't expect the ISAM implementation to be unreliable, but I
would comment that the user could cause some rather nasty and perhaps
unexpected side-effects in the alternate key structures when changing the
keys is permitted, and the more alternate keys there are, the more likely
this is to happen.  This isn't the fault of ISAM, it's the fault of the
USER -- if you're going to change *any* key, it's safer to save off the
contents of the original record, DELETE the current record, make the
changes, and then add the record back to the file.

Even if you do use REWRITE, it doesn't allow you to change the value of the
*prime* record key, so that structure should be unaffected even if the
alternates get messed up.  A simple copy program or perhaps even a vendor
utility can be used to reconstruct the file with rebuilt alternate key
structures.

(Note that you can use the READ, DELETE, WRITE technique to effect a safe
change to the *primary* key.)

    -Chuck Stevens

Chuck, I find this bizarre advice, though I presume you are speaking from 
hard-earned experience.

Certainly your suggestion to DELETE and then WRITE has much more significant 
performance issues (since you are making two changes to *all* the indices 
regardless of before and after values), and you are removing the possibility 
of a recovery (i.e. permanent data loss) if there is a system failure 
between the DELETE and WRITE.

Modifying the value of an alternate key within a record indeed causes a bit 
of 'stirring' of that index tree as the file system has to remove the old 
value and insert the new value, but any modern indexed file system can do 
this and do it well.  Likewise, any modern file system should be able to 
offer the ability of recovering the original value (or completing the 
REWRITE) should a system failure occur during the REWRITE.  (Note however 
that such recoverablitiy comes at the price of performance measured in 
time.)

I would rate the benefit of an additional alternate key rather high, 
especially if it obviates the need for a sort during normal operations.  I 
am unaware of what unexpected side effect you are talking about.

Generally, you're right about speed and they _might_  be about
reliability, depending on the file system.

Set up a test file with 10 keys and an identical one with
user-maintained alternate indexes in 10 separate files. Do a few
thousand random reads on the primary key and one of the alternates.
Also do start followed by 20 read nexts, a common usage pattern for
browse screens and updates. Then do a few thousand insert and
read/update, remembering to update user-maintained indexes.

The biggest reliability issues are recovery, backup and import/export.
Pretend an index has been corrupted. Does the index recovery or
rebuild tool recover all alternates? Does a restore from backup? When
you export/transmit/import, are all keys intact?

I've run into so many reliability issues over the years that I avoid
alternate indexes entirely, preferring to maintain them myself. But I
have seen file systems where they work flawlessly.
