```
We have an old COBOL system that indexes records by a client code. The
client code is user entered Eg. John Smith = SMITH J. It has become
convention to add a numeric suffix to the code to make it unique.

Some customers have SMITH J, SMITH J1 etc
Others have SMITH J01, SMITH J02 etc
Others have SMITH J001, SMITH J099 etc

Does anyone have code to automate this that they're prepared to share
(for free)? I appreciate it's not a huge coding problem but there are
some peculiarities which have to be overcome e.g. file read sequence
SMITH J1, SMITH J10, SMITH J2.
```

```
When you add another SMITHJ to the file, you have to read through *all* 
previous SMITHJs to be sure you are able to prepare a unique suffix -- which 
is why you should assign sequentially incremented client numbers. Keep the 
next sequence number in a separate file; read that file once to get the 
number; increment it; and write it back to the file.
```

```
Doug,

Nothing wrong with what you are suggesting, but what Mark is after, as
he says subsequently, is a mnemonic code that users can remember. From a
design concept we are beyond the bean-counter days when everything was
expressed as numeric keys - that goes back to the 80-column punch-card
and manipulating columns of numbers to obtain different accounting or
statistical results.

From knowledgeable end-users there is now a demand for 'alphas' to
remember codes without referencing hard-copy lists. You can achieve this
with listboxes and dropdown lists, but in a heavily intensive data entry
system, such as plugging invoices into Accounts Payable, it has to be
less tedious. (I know that my son and other bean-counters in his company
had heated arguments with the systems designers before hitting on their
particular solution).

Can I offer a solution - No. It depends upon the company/type of
business. Other alternatives would be to add part of the Street/Road,
City etc - but that depends upon whether these codes are 'localized' to
a one city operation, or is this provincial/state, or national.

I can tell you one good solution that did work effectively - and we are
back to numerics - when VAT(Value Added Tax) was introduced into UK
about '73, the government issued a Mod 97 check-digited VAT # to each
business - and this had to be printed on EVERY invoice. Bingo ! A lot of
digits, ( 11, I think (?)) - but a unique way to identify a
supplier/vendor. 

And here's the real nice part - using Product X(?) which was a PDP 11 -
we shoved A/P data entry on to the bean-counters to stop their bitching
about delays in data entry - worked real good with our mainframe edits
reading, "batch accepted", "batch accepted", etc.............
```

```
MOVE 65535 TO numb-key
LOCK a control file
START file KEY <= key-fields (text+numb)
READ file previous
IF text-key = "SMITH" THEN
  IF numb-key < 65535
    MOVE my-data TO my-record
    ADD 1 TO numb-key
    WRITE my-record
    UNLOCK control-file
  else
    UNLOCK control-file
    DISPLAY "To many SMITH'S"
  end
else
  MOVE my-data TO my-record
  MOVE 1 TO numb-key
  WRITE my-record
  UNLOCK control-file
```
