How about just getting a source license?  I have a feeling any reverse
engineering would be a violation of the license agreement.

The file structures for RW products used to be quite simple -- all fixed
length with very few items stored as COMP (binary, packed or other) data
types -- if you could figure out the record length (the minimum info you'd
have to have to read the ISAM files), you could almost work out the layout
details by visual inspection of the data.  However, I'm quite certain this
would no longer be the case.

Short of getting the source, I don't have any real suggestion for you..

It would be extremely difficult to reverse engineer from the data files in
any sort of automated fashion. While it is possible to extract the position
and length of the indices, it is not (usually) possible to determine the
length and position of any other fields other than through intimate
knowledge of the data that is being stored. For all fields, including
indices, the name of the field is not (usually) stored in the data file
itself, nor is the scale of a numeric items (where the decimal point goes).

If you have some knowledge of the data you could reconstruct the copybook
using Micro Focus Data File Editor and build up your own structure using the
associated structure editor.

Since you don't have the source code I will make the assumption that you got
this software from some vendor rather than developing it in-house (assuming
that you lost all your source code would be insulting). If so, you may want
to explain to the vendor what it is you are trying to do and ask them about
supplying the copybooks for the data files. Even if they ask for money it's
probably cheaper and less error-prone than trying to reverse engineer the
data files.

-Karl

I'se jes' a COBOL-codin' fool, but... I cannot see how anything useful can
be generated from files and indices any other way.

Source code (of which an FD is a part) will tell you how a record is
broken up into FIRST-NAME, MIDDLE-INITIAL, LAST-NAME, HOME-PHONE, SSN and
the like; if all one has is a file then all one knows that positions
(mm) through (nn) have alphabetic data (usually), positions (oo) through
(pp) have numeric data (sometimes)... and this gets *really* nasty if one
has multiple record types where the same positions hold different kinds of
data.

In short... the files contain data, the code can contain information about
the data.

Hi, just read that Crystal Reports v7 has a built-in Document Import Tool.
This means that it can import and convert reports from a legacy DOS
accounting app like Real World (RW) Classic v8.4  It converts the
print-to-file to a MS Access compatible ODBC file. One can save the report
layout conversion file so that a particular report can be converted over and
over again. Then Crystal Reports can select and group desired data, and
other data files, to your heart's content.

Anybody use this facility? Is this enhanced in Crystal Reports 2000?

This workaround would be a way to convert accounting data without having the
proprietary RW Cobol source FD File Definition statements.

There is a chapter "Importing Legacy Reports with the Document Import Tool,"
p 364-82 in George Peck's "Crystal Reports 7: The Complete Reference,"
Osborne, 99, ISBN 0-07-211978-0, 771ppgs + CD, $50 list. It appears to be a
low end, low-cost solution compared to MonarchIS.com Transform ODBC v3
($400), DataJunction.com $400, Transoft U/SQL $1K++, Liant Relativity $1K+,
and RW / IQ Reports ($400+).

Real World Classic v8 has an MF Cobol ISAM DB, so Crystal Reports is
incompatible because it queries relational ODBC DBs. Real World only offers
the legacy IQ SW Reports and Access product along with limited FD source. RW
also offers an OEM Transoft U/SQL, but I don't know of anybody who has tried
using it.
