Report on May 22-26, 1995 ANSI X3J4 COBOL Committee meeting, Tokyo,
Japan

I did not attend this meeting, so the information is from the draft
minutes. Only four member organizations attended (just made quorum).

As usual, most of this meeting was spent discussing proposals for the
next revision and working on editorial papers concerning the base
document.

The major topics/proposals we discussed were:

1. The Committee had been given the direction to prepare an amendment
for multi-octet characters. A ballot was taken at ISO to remove this
requirement because the new standard will contain MOCS. The ballot
passed.

2. The revised proposal on user-defined types (allows strict typing and
allows a data definition to use another data definition - somewhat like
a source COPY) was passed at the last meeting. It was revised with
minor corrections suggested at that meeting, the corrections were
reviewed, and it will be added to the base document.

3. The revised proposal on allowing text-name to be a literal for COPY
was discussed and sent back to me for rewrite. This has already passed,
but needs one more review.

4. The revised proposal on ACCEPT/DISPLAY screen handling (uses a
screen section) was reviewed (it passed at the last meeting) and will be
added to the base document.

5. The revised proposal on modifying report writer (adds a few new
features and makes it easier to understand and use) was reviewed. Some
minor changes were suggested and it will be added to the base document.

6. The revised proposal on dynamic collating sequences (allows the
user to change collating sequences with the SET statement and will tie
into internationalization to automatically use locale information) was
discussed, approved, and will be added to the base document.

7. A proposal on CODE-SET and MOCS I-O was discussed. This provides a
means of mapping external data into the form described by a record
description (a USE WHEN clause identifies the proper record
description). The basic concepts were agreed upon and the proposal will
be rewritten.

8. A proposal changing the PROPAGATE directive (part of exception
handling) was discussed, approved, and will be added to the base
document.

9. Comments on proposals from ISO were discussed. It was decided to
remove display usage from boolean items. They can be bits only. A vote
(2-1-1 - hardly conclusive) said to allow comparisons and moves between
boolean and numeric items with an implied conversion rather than an
explicit function for conversion as was previously proposed. It was
agreed that if all the elementary items in a group were boolean the
whole group was boolean. It was agreed that an XOR operator be added to
normal conditions (AND and OR).

10. Some public review comments were discussed. One suggested not
adding OO. That idea was rejected. If you don't want to use it you
don't have to. The others were mainly editorial and will be added to
the base document.

11. A proposal on clarifying the action of I-O Status 44 and format 3 of
the RECORD clause passed and will be added to the base document.

12. A proposal on resolving the ambiguity of a function with optional
parameters being used as a function argument (like FUNCTION MAX
(FUNCTION RANDOM (123) - is this one or two arguments - that is, is the
(123) an argument to RANDOM or is it an argument to MAX) was discussed
and sent back for rewrite.

13. A correction to EXIT METHOD RAISING (OO and exception processing)
was discussed, passed, and will be added to the base document.

14. The revised proposal on zero-length data items (clarifies what
happens with the in comparisons and the like) was modified and sent back
for rewrite. The Committee wondered if some systems do not allow you to
read and write zero-length records. A survey will be taken on this.

15. The revised proposal on VALIDATE (validates data items based on
criteria specified in the data description entries for the items) was
discussed, passed, and will be added to the based document.

16. The proposal on true binary was discussed. It was decided to not
have a PICTURE, and have USAGE BINARY followed by SMALL or SHORT or LONG
or DOUBLE. The default is signed with an optional NO SIGN. There were
several questions about size error and all. More input is being sought
before the proposal is rewritten.

17. The proposal on concatenated keys was discussed and approved. It
allows various non-contiguous fields in a record to be concatenated to
create one record key. It will be amended and reviewed before
incorporation into the base document.

18. Some minor issues are: a WWW home page will be set up for X3J4
under the ANSI home page; the tests for the corrections amendment (from
NIST) will be available in September with conformance required by July
1996; some interpretations were discussed and passed (very minor); and a
new version of the base document (including all of the above) will be
available a few days before the next meeting.

The next meeting will be in Chicago on July 10-14 1995, with an editing
ad-hoc meeting on the 9th. The main topic will be the public review
comments.

The primary reason is for interfacing to non-COBOL stuff. The secondary
reason is for efficiency. The PIC makes no sense at all for true binary
numbers and is quite confusing. If you say PIC 9(4) and get a 16-bit
item
does this mean the maximum value is 65,535 or 9999? Many implementations
use the PIC for COMP items and have always had this confusion. What we
did
at Tandem is not use PIC and say USAGE NATIVE-n, where n is 2, 4, or 8
(the
number of bytes in the number).

There are some slight problems with true binary and we are trying to
resolve them. Are the numbers big-endian or little-endian? Are they
ones-compliment or twos-compliment? Are the sizes 16 bits, 32 bits, and
64
bits or something else? It appears that we will go with the native
machine
representation so there will be a loss of portability. Another problem
is
that the maximum number of digits in an unsigned 64-bit number is 20.
COBOL allows only 18, so you couldn't display the number.

machines.

Why not make some kind of generic USAGE that would use the base number type
on the machine. On the Unisys A Series, for example, you can declare
binaries, reals and doubles without some strange PIC clause. COBOL already
has a rich selection of USAGE's to choose from. Do we need another?
The A Series compiler allows the use of "REAL" and "DOUBLE", but also of
"BINARY" and BINARY EXTENDED". Here is an example:

77 i PIC 9(4) BINARY.
% This makes the binary number conform to COBOL rules
% so the maximum value is "9999". The item takes one A Series word
%(48 bits or 6 bytes).
77 j PIC 9(3) BINARY EXTENDED.
% This declaration is an Unisys extension to the COBOL standard. This
% declaration allows the binary number to use the FULL capability of the
% Binary numbers on A Series. That is 11 digits.
77 t PIC 9(15)V9(4) BINARY.
% This declaration will allocate two (2) words (96 bits). The developer
% DOES NOT have to choose to get double precesion binaries!

77 r REAL. % This is exactly what is written! A real number that has a
% floating decimal point. Maximum fixed precesion is 11 digits.

77 d DOUBLE. % Same as a above but maximum fixed precesion is 22 digits.

The COMPILER decides what memory allocation to use for declarations with the
same number usage declaration. The size of the number item determines when
to use one word or two words. When I declare "PIC 9(15)V9(4) BINARY" I am
getting an double integer. When I declare "DOUBLE", I am getting a double
precesion floating point number.

Simple, Clear and understandable. Do we need more?
