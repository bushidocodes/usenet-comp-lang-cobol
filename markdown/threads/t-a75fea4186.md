```
The difference is that one has "defined" results and one doesn't.  The
"record area" (often known as the "file buffer") is UNDEFINED before an OPEN
or after a CLOSE.  Therefore, the MOVE of spaces to the 01-level under the
FD does NOT necessarily move anything to the actual area from which data is
written. The error (that you are probably getting) relates to trying to move
SPACEs to some "undefined" (unallocated?) place in storage.  You only
OFFICIALLY get that area when you do the OPEN.

P.S.  There are some OBSCURE exceptions to this rule - having to do with the
use of the SAME AREA phrase - and some "possible" differences if the file is
defined with the EXTERNAL Clause, but in GENERAL, it is simply a "results
are unpredictable" to ever "reference" the record area before an OPEN or
after a CLOSE.
```

```
Just as a point of interest:  Unisys MCP/AS COBOL compilers typically
describe their data spaces using "untouched" data descriptors.  The act of
referencing such an "untouched" space results in a hardware interrupt,
fielded and recognized by the system software ("initial presencebit
interrupt") , which in turn allocates the memory space, adjusts the
descriptor to point to it, and then returns control to the program.  Thus,
record areas (and even data base structures) may be used in MCP/AS COBOL
programs with considerable impunity regardless of whether the files (or data
bases) in question have been opened or not.
```

```
Before you get too drunk, or when you are sobre again, and je sais ne rien about

    MOVE SPACES TO GOMER-REC

You *think* you've moved spaces to the record, when in truth nothing relating to
the file really happens until you Open Output, establishing a handle to the file
with the ability to return a file status code, and it's at that stage that the
record area becomes significant. (Go back to earlier usage of COBOL, where to
conserve memory, you could use the syntax SAME FILE or SAME RECORD AREA, for
several files/records - your moving spaces would be irrelevant).

Consider the following for an ISAM opened I/O  :-

    move MyKey to Gomer-PrimeKey
    Read Gomer key is Gomer-Primeky
      invalid key set RecordNotFound to true
      not invalid key set RecordFound to true
    End-read

    if    RecordFound
          start displaying record or whatever
           you could be changing fields and if so set RecordChanged to true and
            eventually do a REWRITE

    else initialize Gomer-Rec    <----- It's HERE your spaces are significant
           move MyKey to PrimeKey
           and move other stuff to Gomer-Rec
           now WRITE Gomer-Rec
    End-if

I'm no expert, but above would be my logical take on the situation you describe.

Parallel your query with SQL -

    move spaces to my SQL fields
    now CONNECT  to the Database or 'invoke' a particular table
    - that move spaces isn't worth tiddly-squat. - the Sql fields are only
    significant when you get into the Table
```

```
Jimmy,
  In your note, you don't show when you do the OPENs and CLOSEs.  I think
(but am not certain) that you agree that it is a "no-no" to "touch" the
record-area before an OPEN or after a CLOSE.  As this is "explicitly
undefined", the behavior DEFINITELY does vary from compiler-to-compiler,
release-to-release, and OS to OS.

As far as the READ INTO/WRITE FROM, this is (yet another) religious battle.
As long as you don't "touch" the record area (defined under the FD) before
an OPEN or after a CLOSE, you are OK, either way.

There are some "historical" (and/or hysterical) reasons that IBM mainframe
people "tend" (not always - not all programmers - not all shops) to use READ
INTO and WRITE FROM (for *all* file types).

A) At one time, IBM allowed you to access the "buffer" after a CLOSE.  Lots
of programs used the "move high-values to record-under-fd" approach to
indicate EOF.  Then IBM *stopped* allowing references to this area and all
those programs DIED when programs checked the fields for high-values AFTER
the file was closed.
  Once burnt, then NEVER let anything be "moved to" or to be "tested from"
those fields again.

B) Given some (primitive) debugging tools, it was MUCH easier to "read" the
data in WS copies of records (when an ABEND occurred) than it was to find
the FD (which might have "lots" of records "buffered" in them - at time of
"dump").

C) Many mainframe COBOL programmers came from master-file/transaction-file
"update" environments where you (often? usually? almost always?) needed to
"keep" the last record to compare it with the next record.  Therefore, you
ended up "moving" it to WS anyway, so why not START with a READ INTO - and
then do your WRITE FROM when you had finished doing all the "updates" in WS.

  ***

I believe that many (most?) IBM mainframe COBOL people still think it is
"good" programming to always do READ INTO and WRITE FROM - even if they
don't know WHY they need to do this.
```

```
Since I am fairly certain IBM COBOL for this and that and now just this
will give you a S0C4 protection exception trapped by the LE routines for
MOVE SPACES TO UNOPENED-FILE-REC, I think that Fujitsu is taking the
correct action.  Since COBOL II kept addressability to the buffer after
an AT END condition (addressability was dropped in earlier COBOL's at
END OF FILE rather than CLOSE) and the LE dump (or earlier Abend-aid
dumps) give you the record that was available at time of abend, most of
the necessity for READ INTO/WRITE FROM has gone away.  The one area that
remains on many systems is if you have arithmetic data that must be
aligned.  If the record isn't a multiple of the alignment factor (word,
half word, etc.), the arithmetic fields in the second and many of the
subsequent records will not be aligned properly if the records are
buffered.
```

```
Depends on the operating system. In IBM Mainframe DOS (used to be,
anyway), you could access the FD record area before the open.  In
OS/VS2 and succeeding releases, if a file is opened for INPUT, you
cannot access the FD record area UNTIL THE FIRST READ, because the
buffers are not allocated until a record is actually read. In VS1 and
earlier releases, you could access after the OPEN even on OPEN INPUT. 
As someone (Bill Klein) said, buffer access is an operating system (or
I/O subsystem) consideration, not a compiler feature.

---------
Regarding the religious issue, I very rarely use READ INTO or WRITE
FROM. For very large files, that can be a huge amount of overhead. And
I usually find it completely unnecessary.
Similarly, in CICS, I create my COMMAREA with a GETMAIN, and always
work with it in LINKAGE, not WORKING-STORAGE.  Similarly with maps and
other data acquired from I/O.
My experience with people who always use WORKING-STORAGE is that most
of them do not really understand buffers and/or LINKAGE adequately; or
they learned from such people, and prefer the style with which they
were trained.   For some, I suppose, in non-MVS environments, there
may also be error-message or debugging considerations.
In MVS, WORKING-STORAGE for I/O is inefficient.
```

```
The draft COBOL standard, and I think all previous standards, specify
that the record area is not available until a successful OPEN
statement has been executed.  The exception is when another OPEN
statement has already opened a file and the SAME clause of the
I-O-CONTROL paragraph has been specified such that both files use the
same record areaa.  Similarly, the record area is no longer available
after a CLOSE statement, unless another file is still open for which
the SAME clause rules also apply.

The benefits of not allowing such access is that it allows direct
access to the file buffers for reading and writing, i.e. in IBM
mainframe assembler terms one can then use locate mode for the I-O
instead of using GET and PUT mode.  This is arguably more efficient in
the general case, preventing the program using what would otherwise be
an implicit form of working-storage for the record areas even when
still in the FD definition and to which the object code would have had
to move the record from the real file buffer.

Whether or not I use READ INTO and WRITE FROM depends on the
circumstances, I tend to use them when doing traditional sequential
updates.

Another point to watch is that when writing or updating variable
length files, the use of working-storage for the record area can be
beneficial.  This is because changing the length of a record when
updating a file in situ necessitates a DELETE and WRITE instead of a
REWRITE.  And, because when using WRITE FROM, the I-O subsystem can
wait until the record is written to determine whether to write the
block before the current record or after it, depending on whether the
size of the current record would cause the file buffer to overflow,
which would not be known until the length is specified.  This allows
the I-O subsystem to prevent the creation of unneccessarily short
blocks for the file.

For those unaware of the idea of record blocking in files, on
mainframes (well the IBM ones that I know about) records are grouped
in blocks as physical rather than logical records to enable tape and
disk storage to be used more efficiently by reducing the the number of
inter-block gaps that are necessary to separate physical records.
```
