```
Iv'e been thinking (strange as that may seem!) about why OO seems to jar 
so much with many of us who think that it is a case of the "Emperor's 
new Clothes".

There appear to be some fundamental problems with OO that have not been 
addressed.

The problem of OVERLOADED OPERATORS.
Now while this is not a requirement for a OO system, overloaded 
operators are frequently associated with OO designs. The problem is that 
once you do this you neccessarily obscure the definition of the 
operator. This makes it difficult to determine extactly what operation 
is to be done.

The problem of TOO MANY METHODS.
In OO design, there is a great temptation to write a method for 
everything. In some cases (true OO) you are forced to write a method to 
update and read back every variable in an object to make them available 
to the invokers of the object. This is plainly rediculous. Who needs to 
write a MOVE for every variable in a object. One ends up writing a new 
programming language - one that is tailored to the needs of the problem 
being solved.
Now, I don't know about you, but I find that this precludes the ability 
to quickly find problems as you first have to re-learn the proprietary 
problem language and then dig into that to find the problem.
When writing new code, it becomes a task of scanning the existing code 
to find a method that is close and then to re-factor that object 
overloading existing methods and creating new methods to perform the 
task.
It is often said that OO is to programming what IC's are to circuit 
design. But that is false. In ciruit design, the IC's function is fixed 
and so are the interfaces. It is the combination of IC's and their 
arrangement that is designed. You don't find the average circuit 
designer overloading the pin functions with new meaning.
In OO the function (methods) and Interfaces of the IC's (Objects) are 
subject to change. I would venture to suggest that the closest thing to 
IC's in the IT industry is the Cobol Procedure Division statement 
definitions. It is the lack of new statements tailored to today's needs 
that have forced the usage of this unsatisfactory OO paradigm.

The COMPILER DEVELOPERs Slackness.
Much easier to provide a generic CALL interface or some cobbled up 
INVOKE objects that to try to design language elements that would 
standardise the way programmers interact with the new interfaces.
Just why are there no statements to help build and maintain server side 
WEB pages. Why are there no statements to allow TCP/IP communication. 
Why in an age of online transactions are there no statements to create, 
manipulate and communicate to other processes. Why do we have to rely on 
various server manufacturers to provide interfaces. Why doesn't the 
langage mandate and standardise them. How else can you take a program 
from system a and re-compile and run it on system b.

The FAILURE OF CONCEPT.
OO assumes that all DATA has associated methods. But this is plainly 
absurd. When you have trillions of pieces of data, associating them all 
with a set of methods becomes unmanagable. What to you do when you need 
to add another method to them all, change the name of a method, etc.
One of the most important things learnt in the online transaction game 
is to jealously SEPERATE your data from the programs and never ever 
trust the data. Any given program on a given day may process trillions 
of bits of data. You don't need trillions of copies of the program to do 
this - just one. And whilst OO systems can have one copy of a method, 
they all have to associate that copy with each of the trillions of bits 
of data.

Programs are a scarce resource. Data is overwhelmingly changable, 
wherease the programs do not need to be. A small subset of programming 
function may be able to cope with vast amounts of data and data 
variablilty. OO has got it exaclty BACKWARDS.
```

```
This is a simple confusion between overRIDING and overLOADING. I'm not going
to pedantically correct the terminology in what you wrote, as we both know
what you meant, but please accept that I may not respond with exactly the
same terminology you used.

Class A instantiates object AA, which uses the BLAH Method of Class A and
this Method requires object references to objects AB and AC.

Class B is a subclass of Class A (therefore inherits all of its Methods) but
OVERRIDES (not overloads) the BLAH Method.

The Object Method BLAH, defined in Class B, takes the same parameters as the
original Method BLAH in Class A, but behaves in a totally DIFFERENT manner.

This is all perfectly legitimate Object Oriented technology, irrespective of
language. I could post code samples for COBOL and Java but as you simply
snip them I haven't bothered...<G>

Your concern (If I read you correctly) is that there are now TWO methods
called BLAH which both have the same interface but behave differently. How
do you tell them apart?

The answer is quite simple. You don't.

You look at where the Method is invoked from and that will tell you which
one you're dealing with. It is no longer possible to invoke Method BLAH of
Class A from within an object instantiated from Class B, UNLESS you qualify
that you want this Method by specifiying SUPER. The default is to the
overridden Method in Class B. Here are some examples which should help to
clarify ...I have chosen Java because it is quicker to write than OO COBOL.

From within object BB (instantiated from Class B):

1. this.BLAH (AB AC)   Runs the overridden Method BLAH in Class B.

2. super.BLAH (AB AC) Runs the original Method BLAH in Class A.

3. BLAH (AB AC) Runs the overridden Method BLAH in Class B. (the default).

So there really is no problem with modern OO languages where Methods are
either default or self, or they are qualified to super.
```

```
Just so it is clear, there are GET and SET methods (both implicit and
explicit) for properties, but you do NOT need to use an method invocation to
actually "use" them.  There is a "new" format of identifiers that allows you
to implicitly USE these methods.

For example, if "prop-1" is an object property (defined in class "classj-2")

in the repository, you can say either:

 Move "ABC" to prop-1 of class-2
      or
  Move prop-1 of class-2 to some-data-item

(you can also use "object references" where this example uses the
class-name)

This is called an "object property" and is defined on page 99 of the FCD.
```
