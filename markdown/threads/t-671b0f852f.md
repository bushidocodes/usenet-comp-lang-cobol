Linkage section is used for two types of programs (mostly?)

1) (and this is the only STANDARD way) is to get data from a calling program
into called subprogram. Check your books under USING and passing data from
program to program and they should discuss this.

2) For getting addressability to a field for which you already have the
address and the lay-out - but do not currently have addressability to.  If
your compiler supports the
     SET ADDRESS OF ...
           or
     USAGE POINTER
          or
      SET xxxx to ADDRESS OF ...
statements, then it probably supports this EXTENSION. This may well not be
in your books - but should be in your "programming guide" with your compiler
IF it supports it.

aha, the famous this function does not work with indirection dilemma. You
have to remember that linkage items may or may not be really "there". In
most cases, what actually exists in the subroutine is a pointer to the real
data which is still back in the calling program.  There are various methods
used by the compilers, and you can change methods (the call by statements),
but in fact any operation done on linkage items is different than stuff done
to regular working storage data.

For example, DISPLAY linkage-section-data-name will often not work
with a given compiler.  The solution is to move the data to a working
storage data name before using it.  Another common ones that causes
problems is filenames, and records from files.  ( call "abc" using
this-is-an-fd-01-record)

easiest way to understand it (for me at least) is to think of it as shared
memory.  Calling program puts something into a field or fields (or not),
calls a program which (supposedly) looks at them and maybe does something,
passes control back to the calling program, which now has (possibly)
something new to play with.  Be careful with your fields; things can get
very interesting if they do not match.  Always practice safe computing.

calling program:
WORKING-STORAGE SECTION.
01  SWITCHES-N-SHIT.
...
01    TRN-CALL-AREA.
    03  TRN-WORK-FIELD         PIC X(64).
    03  TRN-LENGTH             PIC 9(09) COMP SYNC.
...
PROCEDURE DIVISION.                       
...
    MOVE MY-CITY               TO TRN-WORK-FIELD
    MOVE 20                    TO TRN-LENGTH
    CALL 'XXXTRN'           USING TRN-WORK-FIELD
                                  TRN-LENGTH
    MOVE TRN-WORK-FIELD (1:20) TO MY-CITY


called program:
*
**linkage section
*
LINKAGE SECTION.
01  CALLED-FIELD.
    03  CONVERT-BYTE         OCCURS 64
                            INDEXED CONV-INX
                                PIC X(01).

01  CALLED-LENGTH               PIC 9(09)  COMP SYNC.
/
*
** procedure division
*
PROCEDURE DIVISION USING CALLED-FIELD
                         CALLED-LENGTH.
MAIN-PROC
    .
*
** guard against overflow; if 0 we'll pretend we didn't notice
*
    IF CALLED-LENGTH > 64
       PERFORM YOURE_A_WANKER
       EXIT PROGRAM
    END-IF
*
** compress out multiple spaces, and left justify
*
    PERFORM COMPRESS-OUT-SPACES
etc.

Study the manual and experiment; there are alot of things you can do with
this feature.  You wouldn't want to write a game or figure out the
trajectory of a missile using COBOL - but you could if you have a good
run-time library.
