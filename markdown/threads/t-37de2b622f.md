We have tested some functionalities of the new version of MF Cobol. But HP
write us :

The only documentation we have on "file striping" is contained in the
$COBDIR/docs/callfh.1 file.  This file says:

The striping option is enabled for a particular file via the Callable File
Handler's configuration file which, by default, is called extfh.cfg. See
your "Programmer's Guide to File Handling" for full details on the Callable
File Handler configuration file.

The mFLFS file handler will look for extfh.cfg in the current directory by
default.  Page 13-40 of the "Programmer's Guide to File Handling" manual
says you can use the EXTFH environment variable to direct COBOL to look at a
different file.  For example, if all of your striped files are in /data, you
can set EXTFH to /data/extfh.cfg for all users that need access to the
striped files.  Please see call A5033968 for another example.
export EXTFH=/opt/cobol/cobdir/extfh.cfg

The "file striping" or mFLFS file handler is an "early release" feature in
B.12.26 and B.13.26.  Therefore, it is not supported in a production
environment.  The new feature will be supported in the next major release of
MFCOBOL, which is tentatively scheduled for release on 10.20 and 11.0 by
second quarter of next year.  Other companies are not using this feature in
production because it is not supported and because the file recovery tool,
rebuild, has not been enhanced to handle striped files; ie, currently, there
is no way to rebuild a corrupt striped file. We encourage your customer to
test out file striping in a development environment, and submit SRs for any
defects, but we do not recommend using striped files in production at this
time.  Please see call A5203547 for a list of alternatives your customer can
consider.

1. Split the file into several smaller files, or remove some records so the
data file is less than 2GB.
2. Write a C subroutine that accesses the 2GB file.

Then if you have better ideas...

On 30 Sep 1998 22:51:30 GMT, prata <prata@mail.teleweb.pt> wrote:
>My question is: Anyone knows if its possible to use an software that
>permits to develop on Win95 without the emulators? i.e. can i write
>programs on win95 environment with an automatic update of unix system?

If your Unix admin is willing to install Samba on the Unix system, this would
allow you to see the Unix box in Win95's network neighborhood. You could then
map the share containing your code to a drive letter.  You'd then be able to
edit those files on the Win95 box and your changes would be seen by anyone on
the Unix box as soon as you saved the file. Since the Acucobol objects are
portable, any compiles you do under Win95 would also be useable on the Unix
box.

The downside to this approach is that all your edits are "live".  If you want
to do development locally and then send the changes in once they're complete,
I'd suggest evaluating a source management tool such as CVS.


----

At our site, we ended up writing a set of routines which would allow one
"file" to be split across multiple files. Our routines used some a lookup
file with a list of directory names. Basically, the OPEN, READ, WRITE will
need to be done in a special way. The OPEN opens the "first sub-file". The
READ opens the second "sub-file" when it encounters an EOF, etc.

When AIX began supporting files greater than 2GB, MF-COBOL still didn't.
It's good to see (from another message in this thread) that they're
beginning to do so. However, we also found that Syncsort doesn't yet support
files greater than 2GB: so check the other utilities that you're using, too.

- Darius Cooper
