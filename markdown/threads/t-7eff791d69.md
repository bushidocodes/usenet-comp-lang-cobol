```
I think you want to have information on ABEND codes since this is
something like a synonym to completition in the MVS environment.
Unfortunately, there is not a place you can visit, but rather several
places. Firstly, you have to decide whether it is a 'system' or a 'user'
code.

System codes are some kind of hardware interrupts like 'zerodivide' or
'operation exception'. They are hex, have three digits, and mostly, they
have a 'S' as a prefix: S0C7, S013 and so on. System Codes are
accompanied by some diagnostic messages in the joblog.
You look first at the 'MVS System Codes' manual (chapter '2.0 System
Completion Codes') and the you have a look a one of the five 'System
Messages' manuals for the meaning of the accompaning messages. All these
manuals ar online at:
http://www.s390.ibm.com:80/bookmgr-cgi/bookmgr.cmd/Shelves/EZ2MO700

User abend codes are interrupts set by a user program or a subsystem
like the Language Environment or the DBMS. They are numeric, have four
digits, and mostly, they have a 'U' as prefix: U4093. For the
diagnostics, you have first to search for the program which caused the
user abend code and then look for the manuals for this program or
subsystem.
Language Environment codes are in the range of 4001 to 4099, but some
IMS codes are in the same range. User programs may use the same codes
too! The 'OS/390 V2R7.0 Lang Env for OS/390 & VM Debug & Msgs' manual is
at the same place where you find the system codes (see above). If you
still run the 'old' VS COBOL II runtime, look at:
http://www.s390.ibm.com:80/bookmgr-cgi/bookmgr.cmd/Shelves/IGYSH007
Refer to the 'VS COBOL II R4 Appl. Programming Debugging' manual.

You might also need the IDMS manuals for their codes! Consult CA, their
homepage is http://www.cai.com. AFAIK, there is no manual online, but
they can send you a printed manual or a tape about IDMS. They also have
a problems/solutions database which can be accessed by registred
customers only (ask your system programmer for your customer-number and
subscribe at: http://webtrack.cai.com)

If you have a specific problem, you may ask the newsgoup comp.lang.cobol
or, maybe better, bit.listserv.ibm-main.
```

```
  We have frequently found both application programmers and operators
alike thorougly confused about the distinction between completion codes
and ABEND codes.  They are definitely not synonymous, and because of
their different effect on JCL processing, there are times when it is
critical to understand that distinction.
  A step completion code is synonymous with the "return code" from the
program invoked by the EXEC JCL statement, and has a value ranging from
0 to 4095 (it is the value returned in Reg 15 by the main program).  
Nonzero values are typically used to signal that some kind of problem
was detected, but applications can choose to use other conventions, and
there are cases where nonzero values are used merely to control
conditional execution of later job steps.  A job step has a completion
code if and only if the program invoked by the job step runs to a normal
completion from the operating systems viewpoint.  Having a "normal"
completion does not necessarily indicate that no problems were
encountered, only that nothing serious enough was detected by the
operating system to warrant aborting execution of the job step and any
following job steps.  As far as MVS is concerned, a job step ending with
a completion code has run successfully.  It is up to the user, the
person coding the JCL, or automatic job schedulers to decide is some
completion codes should be interpreted as a failures.
  The other possibility is that a job step may ABEND.  In this case
there is no step completion code, only an ABEND code.  Most frequently
this is the result of some error or inconsistency in a request made to
the operating system by the application program, in which case the
result is a System ABEND code (indicated by a leading "S" in the ABEND
code reported in the job log).  A few of the possible ABEND conditions
are related to error interrupts detected by hardware, but most are
generated by MVS detecting something invalid in an operating system
service request made by the application program.  It is also possible
for the application program to force an ABEND by making an explicit
ABEND request to the Operating System, in which case the ABEND code will
having a leading "U" indicating "user-requested ABEND" and some
identifying numeric value in the range 0 - 4095, as requested by the
application.  A job step ABEND should always be interpreted as a problem
that should be fixed (an application could, I suppose, use User ABENDs
for conditions that could be tolerated, but this would be an abuse of
the intended semantics of an ABEND).  
  If a condition is encountered causing the ABEND of a job step, all
further execution of application code for that job step is immediately
terminated by MVS, the ABEND-dispostion rather than normal-dispostion
for all job step datasets is taken, and execution of any normal
following job steps is bypassed.  The alternative is that the job step
generates a completion code, in which case normal-dispostion processing
is used for all job step datasets and further job steps are eligible for
execution. 
  System ABEND codes are fixed by MVS and documented under System
Codes.  Return codes and User ABEND codes are up to the whim of the
application.  If they are generated by some IBM-written application
code, like CICS, LE runtime library, etc., they should be documented in
the appropriate message or diagnostic manuals.  If in installation
application code, they should be documented in appropriate local
documentation.
```
