Me too. Frankly, performance has never been a big deal for me anyway,
and I'm sure that there are lots of cases where COBOL is at least as
fast as C, and also requires less debugging and produces safer code.

But this morning I saw the other side of the story:

I have been working with a simple compression routine which encodes up to
91 spaces or zeroes into a two-byte sequence, and any other repeating
character (which rarely happens in my data) to three bytes. This is for a
cross-platform transfer of data, so I used a very simple minded approach
and implemented it in nice portable COBOL.

It is pretty effective, compressing 2.6 MB into 630 k bytes.

For small files, everything was fine. But on the biggie, which is 223 MB,
the COBOL decompression was killing me -- the thing ran all day and wasn't
near completion! I was forced to do decompression in C, which can be
called from COBOL. I just finished it this morning, and it's lightning-fast.

The time for the 2.6 MB decompression in COBOL was:
real 0m37.77s
user 0m34.98s
sys 0m1.36s

The same algorithm implemented as a C program did it in these times:
real 0m1.63s
user 0m1.22s
sys 0m0.31s

That's a compelling difference in C's favor, for this one operation. So,
while processing times in COBOL have never hurt me in the past, it's clear
there are *some* things that are going to be more efficient in C, if
performance is an issue. (I have to try calling this from COBOL now, but
I'm sure that won't change the basic picture.)

It's probably best to discover these problems as they emerge and move some
functions into C, rather than building whole systems in C in fear of
losing a few seconds here and there.

In COBOL's defense, remember that I'm a C programmer at heart even though
I choose COBOL most of the time! A 'real' COBOL programmer might have
found a better way to code this character-at-a-time stuff. And since I
implement most things much faster in COBOL (and never have trouble with
bad pointers and segmentation faults), it will remain my choice a lot of
the time.

begin int i; for (i=0;i<255;i++); end
PERFORM 255 TIMES CONTINUE END-PERFORM.
A: Neither, it depends on the specific compiler you use.

It's the same hoary old chestnut: language versus compiler.
It is the implementation that is fast, *not* the language!

When you have described your solution to a compiler, it depends
on the skill of the author(s) of that compiler to discover the
original problem and generate efficient code.

C programmers will probably describe the solution in terms which
suit a C compiler, eg ADD 1 TO SUB. COBOL compilers usually
expect a higher level solution like UNSTRING. Blame the
author(s) of your specific compiler, not the language itself.
