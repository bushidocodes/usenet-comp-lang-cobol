A search of the USENET newsgroups for the word COBOL returns a handful, but 
most of them are not very active.

The largest group would be the IBM listsrv group which is showing over 
12,000 posts on my system. (You will be required to subscribe to this group 
and it is NOT a COBOL group as such, although IBM COBOL does get discussed 
there.)

Here in comp.lang.cobol specific questions about COBOL usually do get 
answered  but there is not much non-specific discussion on COBOL,. Maybe 
because the group has been running for years and the regulars here have said 
most of what they want to say about COBOL. :-) The language has not had any 
major innovations for quite some time so there hasn't been that much to 
discuss.

There is no problem with sharing techniques or information, but you will 
need to ask for what you want or start a topic on what you are interested 
in.

There is a COBOL forum dealing with specific problems in specific COBOLs at 
the COBOL User Group (COBUG), and Micro Focus run a similar forum in support 
of their COBOL products.

If you are interested in looking at modern Object Oriented COBOL or seeing 
how to lever standard procedural COBOL into OO COBOL, plus a number of free 
tools and examples, have a look at the COBOL21 site: 
http://primacomputing.co.nz/cobol21
(You can comment here on anything you see there and there is a direct link 
from that site back to this forum on most pages of it.)

As for programming in COBOL in 1967.... Snap! so did I...:-)

Pete.

I looked at the presentations you provided the links for and the one
about the performance was interesting for me.

When I started programming in the early 70's, this was on a Unisys
9400 system using the assembler language. Performance was a very
important requirement and the CPU usage and I/O performance was as
important as the correct functioning.

Later on, when COBOL was the main language I used, performance
considerations had still their place in how I implemented code. In the
mid 90's we had a larger migration project from a Unisys system 80 to
a decent Unix system and I was running a batch program for an inital
test. While the program took one hour on the mainframe, it completed
in a bit less than a minute on the Unix system. This was when I
decided that performance  is not going to have the same importance for
me as it used to have.

I'm not saying, that performance it is not important, but the focus
had changed for me. These days however, mainframe usage is charged by
CPU usage in some licensing models, so it might change again.
