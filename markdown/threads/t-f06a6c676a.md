› Happy Easter:

› Does the START verb point to the first record of an indexed file
› matching the record key of the index and input?

Not necessarily. If a record matching the key, or part key, does not
exist, the record pointer will be *before* a record with the nearest key
value, depending on whether you coded EQUAL, GREATER THAN or LESS THAN
in your START statment.

›
› What is the function of the >, +, <, etc. parameter in a START command?

Positions record pointer before a record whose key satisfies the
value of the condition.
Don't know about the '+'; never seen that before.
›
› How do I update an indexed file that has only one key and may have none
› or several records with the same key(ie. different records concerning
› the same person). Do I use START and READ NEXT RECORD?

Use the REWRITE verb *after* you've successfully executed a read, for
which you will need START and READ NEXT. Use file-status to check
successful execution.

E.G.

START file-name KEY EQUAL to file-key

If File-status = '00'
READ file-name NEXT RECORD

Perform until not file-status = '00'
or file-key-value changes

REWRITE record-name

If file-status = '00'
READ file-name NEXT RECORD
End-if
End-perform
Endif

Jamie wrote:
›› 
› Does the START verb point to the first record of an indexed file
› matching the record key of the index and input?
 
› It does do that.
 
› What is the function of the >, +, <, etc. parameter in a START command?

There are no parameters in a START verb. Do you mean the relational
operator like GREATER THAN (or >) and such? There is no plus. When
you say "START some-file KEY > a-key" you are telling the system to
find the first record where the key is greater than the key you
mention. If you say =, it finds the one equal (of the key is an
alternate and is duplicated it is the first of the set of duplicates).

› How do I update an indexed file that has only one key and may have none
› or several records with the same key(ie. different records concerning
› the same person). Do I use START and READ NEXT RECORD?
›
You would probably do that. It all depends on whether or not you are
searching it or reading through the whole thing. One has to know what
you are really doing. Note that if there is only one key you have a
slight problem. The primary key must be unique. You would probably
need alternate keys for multiple records or some sort of trick of
extending the primary and then using partial key access.

All of this is a bit tricky. I suggest you study the manual a bit
about these before trying what you are attempting.
