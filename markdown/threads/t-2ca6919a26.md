```
Our class did a sub-program project and almost everyone used CALLs to an ENTRY
point in the sub-module.  They drove themselves crazy.

I liked the model in our book that passed a function code.  It was a lot easier
to keep track of what I was doing.

Right before each place in your main program where you want to branch to the
sub-program, set FUNCTION-CODE (in your main working storage) to a letter
representing the routine you will want to do, i.e. 'I' for init-routine, 'P'
for print report, 'C' for close files, etc.  Pass that code in your USING
clause.

In the sub-module, you will have a mainline that looks something like.

IF FUNCTION-CODE = 'I'
   PERFORM 000-INIT
ELSE IF FUNCTION CODE = 'P'
   PERFORM 100-PRINT
ELSE IF FUNCTION CODE = 'C'
   PERFORM 200-CLOSE.
GOBACK.

If you are using COBOL II, you can use an EVALUATE statement to code the above
logic.

Those of us who used function code logic in our sub-module had much less
difficulty in debugging.

RoadRaat
```

```
With static linking, multiple entry points were "OK". They aren't as common
now though, as dynamic linking proves far easier to maintain.

BUT, the rationale was - and it applied only to data-orientated programming
methodologies like JSP - a sub module dealing with a single output file
would have 3 entry points.
Entry point 1 would be the init process, open file etc.
Entry point 2 would be the detail record processing
Entry point 3 would be end processing of trailer etc.

But as problem-orientated logic still dominates, this really shouldn't
apply.
people fail to design their logic correctly and have to use a single
sub-module
for disparate processes.

Just remember to use REUS when linking programs, and there is no discernible
performance problem - that I have seen.

- I once supported a system that was a highly complex reporting tool. It
dynamically
coded COBOL & then compiled & linked it. When making changes to the skeleton
code, you had to statically link 11 levels of module. If you got just 1
module out of
sequence you were hosed, and probably took a whole day to start again. This
may
have influenced the decision to axe the system!
```

```
Since your request lacks any kind of specifics here is a very simple
example. This demonstrates simple operations performed on 3 different
variables. The linkage section indicates parameters passed to a subprogram.
Passed paramteres are indicated also in the USING CLAUSE which indicates all
01 level parameters defined in LINKAGE SECTION which are to be used by that
ENTRY POINT. ENTRY POINTS other than the default, which is used in this
example, should never be used by a beginner, amature or expert as they add
extreme complication to tracking errors in programs/subprograms. NOTICE the
field definition of PARM-2 is defined differently in the subprogram than the
calling program. This shows that a redefinition of the linked data can occur
as long as the redefined data is the same byte size of the pased data.
Certain compilers allow for inconsistent definitions, but that should never
be used. This is an excellent place to use a COPYBOOK with a single 01
containing all sub parameters to be used.

NOTE: the code which is actually here is correct. Missing DIVISIONS and
SECTIONS may be needed. I did not run this example through a compiler.


IDENTIFICATION DIVISION.
PROGRAM-ID. MAINPRG.
DATA DIVISION.
WORKING-STORAGE SECTION.
01 PARM-1 PIC X(08) VALUE '11111111'.
01 PARM-2 PIC X(08) VALUE '22222222'.
01 PARM-3 PIC 9(08) VALUE '33333333'.
PROCEDURE DIVISION.
0000-MAIN-PROCESSING.
    DISPLAY 'OUTGOING PARMS:'
    DISPLAY 'PARM-1 ' PARM-1
    DISPLAY 'PARM-2 ' PARM-2
    DISPLAY 'PARM-3 ' PARM-3
    CALL 'SUBPGM1' USING PARM-1 PARM-2 PARM-3.
    DISPLAY 'RETURNED PARMS:'
    DISPLAY 'PARM-1 ' PARM-1
    DISPLAY 'PARM-2 ' PARM-2
    DISPLAY 'PARM-3 ' PARM-3
    GOBACK.
    STOP-RUN.
END-PROGRAM.

IDENTIFICATION DIVISION.
PROGRAM-ID. SUBPGM1.
DATA DIVISION.
LINKAGE SECTION.
01 PARM-1 PIC X(08).
01 PARM-2 PIC 9(08).
01 PARM-3 PIC 9(08).
PROCEDURE DIVISION USING PARM-1, PARM-2, PARM-3.
0000-MAIN-PROCESSING.
    DISPLAY 'INCOMING PARMS:'
    DISPLAY 'PARM-1 ' PARM-1
    DISPLAY 'PARM-2 ' PARM-2
    DISPLAY 'PARM-3 ' PARM-3
    MOVE '00000000' TO PARM-1
    ADD 11111111 TO PARM-2
    ADD 22222222 TO PARM-3
GOBACK.
STOP-RUN.
END-PROGRAM.
```

```
Just because it is "easy to miss" for new programmers, the match between the
calling and called programs is by "order" and not by name or anything else.
For example, if the CALLING program uses,

   Call "ABC" using Parm1 Parm2 Parm3

Then it is the ORDER of the parameters in the PROCEDURE DIVISION USING
statement (and not in Linkage Section or by name) that does the matching.
Therefore, if the called subprogram (ABC) had the following:

Linkage Section.
01  FieldC  Pic X.
01  FieldB  Pic X.
01  FieldA  Pic X.
Procedure Division Using FieldA FieldC FieldB.

Then Parm1 in the calling program would be matched to FieldA in the called
program.
          Parm2 in the calling program would be matched to FieldC in the
called program
          Parm3 in the calling program would be matched to FieldB in the
called program.

NOTE: it is a "nice" (recommended) coding habit to get use to, to put your
fields in the same order in Linkage Section as they appear in the USING
statement.  Using the same name, however, is not nearly as common. (For
example, some programmers regularly prefix fields in Working-Storage with
"WS-" while they use "LS-" for Linkage Section items.)

It is - the PROGRAMMERS responsibility to insure that the appropriate fields
are defined identically in the CALLING and CALLED programs.  (There are times
that you want to do "tricky" things where they don't match - but that
shouldn't be the case in a "beginning" assignment).

It is also important to understand that BY DEFAULT the parameters are passed
"BY REFERENCE" which means the subprogram actually addresses the storage in
the calling program.  You can also specify BY CONTENT which means that the
subprogram only gets to "play with" a COPY of the data in the original
program (and any change that occurs in the called program does NOT get
reflected back to the calling program).  Your compiler MAY support other ways
of "passing data" - but again, these shouldn't be needed in a "beginning
assignment".

Finally, you should know that any data in the CALLED (subprogram) is (by
default) found in "last used state" when you call the same program for a 2nd
thru Nth time.  If you do NOT want that, you can override it by using the
"INITIAL" attribute on the PROGRAM-ID paragraph of your called subprogram OR
by using the CANCEL verb in the calling program.  Probably you will NOT need
to do this in a "beginning assignment" - but you do need to know what happens
to Working-Storage data in that subprogram - if you call it more than once.

P.S.  If this assignment goes so well (is so easy) you may also want to look
in your reference material for the EXTERNAL attribute for "extra credit".
But this is not the "normal" way of passing data and should only be used in
cases where it is either required or is "commonly used" as a shop standard.
```
