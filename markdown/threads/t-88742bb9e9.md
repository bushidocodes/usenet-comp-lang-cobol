Peter this invoke/call difference is getting my goat. Let me see if I can
compare the two in a way that means something to the average Cobol
programmer, and let you comment on it.

If we took a standard Cobol subroutine, and specified it as intitial, then
added several entry points to it, we would have an "object(in general
terms)" with several methods (also in general terms).

The key to understanding it in OOP terms is that the first "call", in Cobol,
does the oop equivalent of creating the "instance" automatically.
Similarly, the first "cancel" does the equivalent of destroying that
instance. You CANNOT have more than one instance of the routine, unless you
make a second copy of the source, then give it a new subroutine name.

So if we wanted to do OOP-like things, with only a single instance of a
specific object, we could call the subroutine, thereby creating it and
intializing it at the same time.  We could then call it several more times,
and cobol being what it is, would be using various procedural methods on the
data contained in the subroutine  Then we cancel the subroutine, and
everything in it is destroyed.

The prime difference with OOP is that we are not confined to a single
instance of that routine, and a single instance of data at the same time.
We create and destroy MULTIPLE instances of the same rountine.  EACH remains
in effect until the object is destroyed.

Not sure if that will make sense to people or not, but it seems to me one
way of looking at it that demonstrates the difference.

Donald

I think the problem arises because we try to look at something via the
mechanics of it.

To understand the difference between CALLed subroutines and INVOKED Object
Methods it is necessary to look BEYOND what the computer does when it
encounters these verbs.(In actuality, it may do exactly the same processing,
except that one is pointer based and the other isn't...This is why trying to
describe it in terms of the bottom level is confusing and, ultimately,
fruitless.)

To recognise why INVOKE is different to CALL you must recognise why an
Object Method is different from a subroutine or module. That involves a
conceptual understanding.

I posted something on this in response to a question from JerryMouse. In
case you missed it, and, given that it isn't too long, and is apropos this
topic, I've reproduced it here.

================== start quote ========================
>
> OK, I'll byte. Just how does a Method differ from a Subroutine, aside from
> "subroutine" having more vowels?
>
Jerry,

there are serious differences between subroutines, or even called modules,
and the Methods of an Object.

Without getting into the whole scholarly pedantics of it (which I find about
as entertaining as watching paint dry), the simple answers, based on
empirical experience, are these:

1. A subroutine requires some controlling logic to make it happen. The
subroutine is often part of the program which contains this controlling
logic. A Method is free standing encapsulated code. It doesn't require
controlling logic to make it happen.(see next point)

2. A called module is closer to a Method but still not the same. A called
module is activated by being called; a Method is activated by an event. (The
event can be the receipt of a message telling it to wake up, or it can be a
real time event like the click of a button.) This is where we get to the
subtle difference between CALL and INVOKE.

INVOKE LOOKS the same as CALL (see my response to Kellie elsewhere) but it
is conceptually causing a message to activate the Method to be generated.
(The mechanics may be identical to those used with CALL, but the concepts
are not, and this is important.). INVOKE is an OO COBOL Programmer's way of
raising an event that the Object will respond to, using a specific Method.
The system can INVOKE the Method by raising other events. This is,
obviously, not the same as handling called modules.

3, The Methods of an Object, considered collectively, represent the
"behaviour" of the Object. Called modules are not part of an entity like an
Object so they cannot be considered as "behaviour".

You can argue that this isn't important (I thought so, when I first
encountered it...) but as time goes by and I have gained a lot more
experience of using Objects, I have come to the realization that concepts
can be just as important as mechanics, particularly when you need to
conceptualize the design of a system. The behaviour of an Object (which, in
my case, will become a component) is significant when deciding the
interfaces it will require.

Part of the problem here is that there are many different definitions and
usages  within the OO and Component worlds. In a way this is good because it
means that each user can define the terms for his context and, as long as it
is clear how he intends to use the terminology, no harm is done. There is an
excellent paper on Component Based Design by Kirby McInnis that covers this:

http://www.cbd-hq.com/articles/1999/990715_cbdandreuse.asp

Quoting from the paper...

"Although most of the academic and industrial realm understands some
definition of a component, there tends to be some variance in what each
individual's definition of a component is. It is not always necessary that
everyone agrees on a definition, but it is necessary to understand the
author's use of the words."

======================= end quote ====================

Summarising:

1. Object Methods represent  the behaviour of an Object and are (arguably)
more "coherent" than Called modules.

2. Object Methods are independent, encapsulated, functionality and do not
require "control logic" to activate them, the way a called module or
subroutine does.

3. Object Methods are designed to be event driven. (this is why Objects are
taking the programming world by storm; the networks require event driven
code.)

3. INVOKE  (conceptually) raises an event against an Object that is
represented by a pointer.(The pointer is created when the first instance of
the Object is created (instantiated)). There can be many pointers
representing many Objects simultaneously executing.

4. CALL activates a single instance of a module.

I really don't know whether the above helps or simply muddies the water
further.

Although they may LOOK the same, an albino carrot is NOT a parsnip.

Neither is a called module or sub-routine an Object Method.

Pete.

and methodologies. So an object is a gathering of the design ideas for related
data and programs into a single named item. When you want to mess with that
gathered data, only its methods will be allowed to do the work.  So the data in
an object is definitely distinct from a flat lander 'record' (since anyone can
get to a record/struct, and not so an object's data). Encapsulation is both the
gathering of the data and the functionality as well as the protection of the
data (a 'wall' the original poster would say, is placed around an objects
data).

That wall makes the data inside an object much different from a record or
struct in working storage or linkage. Most traditional data in COBOL is shared
data, which makes it very vulnerable. Even a COBOL mainline Linkage back up
into the executive software's data makes the executive software vulnerable (you
can crash CICS and IMS from your COBOL program).  Data within Objects is
distributed data (you have to go through the methods to get the values). 
Static data of a Class allows the new technologies to revert some specific
fields back to common shared access. 

These notions are not just fancy, they are highly valuable distinctions.
Nothing in _traditional_ COBOL compares to polymorphism, except jump tables
like WHENEVER error handlers in DB2 and CICS, which many shops discourage.
And as far as I know nothing  is present in the traditional (older) COBOL
compilers
to compare to operator overloading  It is difficult to even thing of such
things in COBOLese.

But in the end you software is better if you plan it out in some detail before
you code, and OOP is just one collection of concepts that can help you do that.

OO requires that you have confidence that your invoked objects work correctly.


In mainframe shops, it is not uncommon at all to require full user testing for
any change.  If a program is called by 100 other programs, often the choice is
"it's too expensive to test this - let's not fix it".    Part of implementing an
OO environment is persuading the users that they need different testing
standards.

have seen a similar problem with a standard that says "when you hit an abort
condition, call this program that does our standard abort".   The trouble with
this, is that the dump is of the called abort program - which is pretty
worthless.

the same thing.   But at the level where this is useful, it can make testing
more difficult - how do you make sure you test each and every way it can get
triggered?   Often you can't.

OO isn't all upside.  Like any choice, there are costs as well.

Who know why they chose the terminologies they did - but, they are 
actually pretty meaningful.  Polymorphism is more than multi-form 
displayed text.  Encapsulation is packing code, but it's much more than 
that - when you take into account public methods versus private methods, 
properties, etc.  It's not just the code that's packaged, it's data too.

Invoke is a bit different than Call.  A call executes a predefined 
subroutine, whether it's statically linked or dynamically linked. 
Invoking a method isn't the same as calling a class - it's telling that 
particular -instance- of an object that you want it to do something.

Subtle, yes - and, if you're trying to learn OO, comparing the OO 
terminology to the structured terminology is often a great way to take 
some of the mystery away from OO.  However, there are subtle differences 
that make OO different, and these terms help emphasize that.

Personally, I'm more a fan of component-based software engineering 
(CBSE).  With OO, you generally have lots of objects, and if one needs 
to be changed, you change the object.  In the course of changing this 
object, it's quite easy to break something else.  CBSE applies the same 
processes as most other engineering disciplines, such as civil 
engineering.  When you build a bridge, you don't go design, develop, and 
stress test a great big concrete arch - you buy bricks, which do very 
little in and of themselves, but do what they do very well.  You'll also 
buy the mortar to hold them together, and the rebar to support it.

In CBSE, the bricks are your components, and the mortar is a small, 
structured program that ties the components together.  It uses the 
object model from OO (while removing a lot of the complex portions of 
it), and uses structured programming for it's strength, which is a 
top-down (or bottom-up) design.

Of course, this is biased by my life experiences and reading I've done 
over the years...  But, I have no financial stake in whether you use OO 
or CBSE.  :)
