Correct. Passing by reference in Cobol works just like passing
by reference in C++. The syntax refers to the parameter by name as
the object itself, but what is actually passed is a pointer. In
the subprogram, the compiler implicitly dereferences the pointer
wherever the source code mentions the parameter.

Note that passing by reference is the default. In my own experience
I've never passed anything by content in Cobol, nor felt tempted to.
It would probably be useful sometimes, but it never occurs to me
to do it.

›› by value mean it copies the values of the variables onto the stack?
› Since I am not sure if "by value" is legal, I don't know what would
› happen.  But I can tell you there is no stack in IBM System 360/370/390

We must regard the 'stack' conceptually. C *behaves* as if there is
a stack even if there isn't one in the hardware -- for example, when
C is implemented in the 360/370/390 architecture.

With Cobol the concept of a stack is much less useful, even if a
particular implementation actually uses a hardware stack. Each
program or subprogram only gets a single copy of its WORKING-STORAGE.
That's why traditional Cobol doesn't support true recursion (yes,
I know there are dialects that do support it).

Probably the simplest way to understand passing by content is to
imagine that the compiler invisibly allocates an unnamed chunk of
WORKING-STORAGE into which it copies the parameter which is to be
passed by content. Then it passes the copy (physically it passes
a pointer to the copy).

Of course a particular compiler could do it in some other fashion,
as long as the result is the same.

Arnold raised a number of good syntactical points, which I have
mostly snipped in order to concentrate on the comparison to C.
Here are several points to ponder.

(In all cases I am describing IBM Cobol II. Other dialects handle
pointers differently. In fact I suspect that the BY VALUE syntax
in the original post comes from some other dialect.)

--------------
1. A Cobol pointer is untyped. It is like a void * in C. It is
the programmer's responsibility to dereference a pointer into an
object of the appropriate type.

2. Cobol has no function prototypes. It is the programmer's
responsibility to pass the right number and type of parameters.

3. The called program neither knows nor cares whether a parameter
was passed by reference or by content. Either way, the called
program physically receives a pointer, which it implicitly
dereferences as needed.

Note that the source code need not say anything about pointers.
BAR may be declared, for example, as a PIC X(50), both in the
calling program and in the LINKAGE SECTION of the called program.
Every 01-level entry in the LINKAGE SECTION is really a pointer
in disguise.

4. CALL FOO USING BAR or CALL FOO USING BY REFERENCE BAR physically
passes a pointer to BAR. Through that pointer, the subprogram can
change the contents of BAR.

BAR may itself be a pointer. In that case the calling program
physically passes a double pointer, and the called program
implicitly dereferences it into a single pointer.

5. CALL FOO USING BY CONTENT BAR makes an unnamed copy of BAR and
physically passes a pointer to the copy. The subprogram can change
the contents of the copy, but it cannot change the contents of
the original BAR. (Here I am ignoring possible perversities such
as BAR being a pointer to itself.)

This kind of parameter-passing behaves like ordinary parameter-
passing in C (leaving aside the perplexities of array parameters).

6. CALL FOO USING BY CONTENT ADDRESS OF BAR is trickier. It
physically passes a pointer to an unnamed pointer to BAR. Got that?
It is equivalent to:

SET BAR-POINTER TO ADDRESS OF BAR.
CALL FOO USING BAR-POINTER.

At least I think that's what it does. I've never used this kind
of syntax myself.

Either way, BAR must reside in the LINKAGE SECTION, not in
WORKING-STORAGE, as Arnold has pointed out.
--------------

My Web pages contain a further discussion of pointers in Cobol,
including a comparison to pointers in C.

There is no way to dereference a pointer in the CoBOL language, since
there are no pointers in the CoBOL language. What you have to do is pass
the pointer through the LINKAGE SECTION, something like this:

call "mover"
using by value ptr-to-x
by reference y.

where mover is:

linkage section.
01 ls-src pic x(3).
01 ls-dst pic x(3).
procedure division using ls-src, ls_dst.
0000-main.
move ls-src to ls-dst.
exit program.


I repeat, pointers hardly enter into CoBOL at all, except when you are
interfacing with something else. If you think you need a pointer in
CoBOL, you are probably trying to translate a C idiom literally, when
what you should be doing is using the CoBOL idiom that does the same
thing.

Language extensions vary from vendor to vendor. However, perhaps the
following will help:

05 val pic s9(9) comp. int val;
05 ptr-to-val pointer. int *pval=&val

On the CoBOL side, we are already in trouble. Naming a variable
PTR-TO-VAL does not make it a pointer to VAL. Most platforms do not have
USAGE IS POINTER in the first place, and those that do have different
methods of using pointers. For example, in VAX CoBOL you could say:

05 ptr-to-val pointer value reference val.
05 ptr-to-ptr pointer value reference ptr-to-val.

In IBM AIX or OS/2 CoBOL you can't do that, but you could say:

set ptr-to-val to address of val.
set ptr-to-ptr to address of ptr-to-val.

This isn't as good as VAX CoBOL because it occurs at run time instead of
at compile time, but it's a piece of cake compared to IBM MVS or VM
CoBOL, where you can't take the address of of anything in WORKING-STORAGE
without passing it through the LINKAGE SECTION first.

I would say that the only method that is even vaguely portable is to use
the LINKAGE SECTION, and typically when you use the LINKAGE SECTION you
discover that you don't actually need pointers at all. But back to the
question:

call "a" using by value val a(val), a(*pval), a(**ppval)

call "a" using by reference val a(&val), a(pval), a(*ppval)
call "a" using by value ptr-to-val a(&val), a(pval), a(*ppval)

call "a" using by reference ptr-to-val a(&pval), a(ppval)
call "a" using by value ptr-to-ptr a(&pval), a(ppval)

call "a" using by reference ptr-to-ptr a(&ppval)

Note first that BY REFERENCE is the default, then remember that BY VALUE
is not standard and is not available on many platforms.


› [M]y basic question[s] are, does: by reference mean it [can alter] the
› variables [passed via] a function call?

The basic method of passing a variable from a calling CoBOL program to a
called CoBOL program is BY REFERENCE. This means that the address of the
variable in the calling program is passed to the called program. The
called program might use that address to change the value of the variable
in the called program, or it might not.


› by value mean it copies the values of the variables onto the stack?

There is no stack in CoBOL. Even the hardware on which CoBOL usually
runs doesn't have a stack. There is also no concept of BY VALUE in
CoBOL. Where it is provided at all, it is generally provided so that you
can call other languages from CoBOL. (Exception: MF CoBOL called
programs can receive parameters of certain limited types BY VALUE).

Look at it this way. If the calling program passes BY VALUE, the called
program receives a piece of data that is the same size as an address, but
it is a value, not an address. Obviously, the called program must
therefore know whether a parameter is being passed BY REFERENCE or BY
VALUE.


› by content mean it points to the variable given to determine the local
› variables in some_function3? [sic]

One more time, a CoBOL called program receives all parameters BY
REFERENCE. If any of those variables were passed BY REFERENCE, then the
called program has the ability to change the value of these variables in
the calling program.

BY CONTENT was introduced so that the calling program can protect itself
against the called program. When you pass a variable BY CONTENT, the
compiler makes a hidden copy of the variable and passes the address of
the copy to the called program. The called program can change the copy
to its heart's content without altering the original variable.

In C you could do it manually like this:

extern void a(int*);
int val, tmp;

tmp = val;
a(&tmp);

Note also that there is no way for the called program to tell whether a
parameter was passed to it BY REFERENCE or BY CONTENT.


› [A]lso, does call using returning into X mean that X is a pointer to a
› variable where it returns the value from the called program?

Return values are also nonstandard, but you can think of

call "a" returning x

as shorthand for

call "a"
move the-value-that-was-returned to x

according to the ordinary CoBOL MOVE rules. Pointers don't enter into it
at all. In fact, pointers hardly enter into CoBOL at all, except when
you are interfacing with something else. If you think you need a pointer
in CoBOL, you are probably trying to translate a C idiom literally, when
what you should be doing is using the CoBOL idiom that does the same
thing.
