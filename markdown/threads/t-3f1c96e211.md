```
Greetings,

We have a situation where we have a "listener" program that has been written
in COBOL (actually Microfocus on HP-UX).  When this program receives a
message, it dynamically calls a COBOL subprogram to process it.  Does anyone
know of a way we can trap that subprogram taking an extraordinarily long
time ?  In C it would be easy by setting a timer, and then trapping the
Alarm clock signal, and I thought about interfacing a C routine with the
COBOL.  However I'm not sure if this would solve my problem.

Your thoughts would be very much appreciated.

Steve Baldwin
```

```
Assuming that you want to catch this situation BEFORE you get back from the
CALLed subprogram *and* assuming that you only want your solution to work
with Micro Focus COBOL, then contact your MF rep and ask them about the
"CALLMAP" compiler directive.  This is a VERY LITTLE USED (and almost
undocumented) interface that lets you "capture" each CALL statement before
it gets to the CALLed subprogram.

Note: I am not positive this got ported from the 16-bit to the 32-bit
products - so you may want to ask MF about that too.
```

```
An alternative option is to avoid the use of a call altogether. This would
avoid control being parsed to the called module, and thus there is no
dependency on how long the called module takes. Your main app can then carry
on and the subrountine catches up eventually. And yes you have to be very
careful in real time situations.
Essentially the parameters are parsed as records in a file. You declare the
file as type PORT. Physically it is a stack file locating in memory, thus
volatile. But it means that there is never an AT END condition and you're
not forced to test for it. The main program just does the equivalent of the
CALL by writing a record to the stack (with all parameters as fields). The
subroutnine periodically reads the file and if there is a record it performs
its process on the record, checks again if there's a record and if so
repeats until such time as there are no rercords. At which time it goes to
sleep using a WAIT of some sort you feel appropriate.
In order to get the subroutine to actually end, you have to parse a special
record to it through the port that it understands as time to STOP RUN.

Hope this might give you an idea or two,
Grant McNamara
```
