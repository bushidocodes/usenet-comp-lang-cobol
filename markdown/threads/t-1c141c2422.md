John,

My OPINIONS related to the languages you listed are as follows:

COBOL: Learn it to make money.

C++: Learning it will make you a better programmer. This is because
it uses an Object Oriented Methodology for solving problems. This
methodology can be applied to any problem, so it will help your other
programming as well. This language will also help you make money.
Find a good book that starts with C, and then goes deep into C++

WINDOWS: You didn't mention this, but it is important to learn how
windows works. There are a lot of design issues in regards
programming for windows. I would say that at least 50% of windows
programming is the interface to the user. You can gain this knowledge
simply from using windows on a regular basis, and observing how GOOD
applications are written. Also look for apps that are difficult to
use and understand, and learn from these what NOT to do. Again, this
is related to making $$$$

Visual Basic: Learn it only if you need to write programs for windows
without having time to spend learning Visual C++. Visual basic is
easy to learn and use, but it's powers are llimited. The time you
spend learning Visual Basic would pay you much greater dividends if it
was applied to learning Visual C++.

Hope this helps ya,

Paul

Hi John,

This is not a strange question. The other answers raise good points.
Yes, the more languages you work in, as opposed to know, the more they
will interfere with each other. For example, I know COBOL and SAS. I
can usually write COBOL code without syntax problems, because I use
COBOL 85% of the time. I cannot write SAS without the reference
manual. I know the language, and I know what I want to do, but I
can't remember the syntax. But I only use SAS 5 or 6 times a year.

It depends on what you want. I have learned several languages
(FORTRAN was my first) and they all have been useful to me. But you
are correct that the languages can get mixed up in your mind. Good
luck.

My viewpoint is a bit different that that of you other responders (almost
all of whom had valid points that I don't dispute).

I think that the ancilliary issues (utilities, calling conventions,
debuggers, User interface, etc.) of the different environments *will*
eventually clutter your mind. But even more important, I think the learning
curve doesn't begin to flatten until after at least two years of language use.

IMHO, it's harder to find someone who knows their stuff in one language, than it
is to find so-so programmers with multi-language skills.

That said, I'll contradict myself and recommend learning an assembler
language to become a better high-level language programmer. But I could
understand substituting C as a near-substitute for assembler.
