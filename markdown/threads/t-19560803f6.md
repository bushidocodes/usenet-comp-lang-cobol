On Thu, 15 May 2008 04:47:03 -0700 (PDT), amir <ahsharif@gmail.com> wrote:

>> Don't overlook data type POINTER and dynamic allocation.
>Can you comment more?

There was no Cobol syntax for memory allocation before the 2002 standard, but it is easy
to abstract. If you have access to a C library, the abstraction is already done, just call
malloc(). There IS Cobol support for type POINTER, which is required to address the
allocated memory. You describe the external structure in linkage section, say SET ADDRESS
OF (linkage structure) TO (pointer). 

It is especially useful for linked lists and trees, which have no practical limit on their
maximum size. Contrast with arrays (defined with OCCURS), which have a arbitrary maximum. 

Here's a more mundane example. Suppose you want a function to use a default value when a
parameter is omitted. 

working-storage section.
01  default-value value 'ALL' pic x(99).

linkage section.
01  a-parameter                    pic x(99).

procedue division using a-parameter.
    if  address of a-parameter = null
        set address of a-parameter to address of default-value
    end-if

>> Try TYPEDEF instead of COPY. Try using TYPEDEF to define a date type.
>
>I will use it.

Bill Klein was correct when he said typedef is not supported by Enterprise Cobol. 

>> Try calling C library functions.
>I will try it.

C functions are especially useful for calls to the operating system API, because they are
OS-independent and  Cobol compiler-independent.  For example, getenv() reads an
environment variable the same on z/OS, Unix and Windows, the same on Micro Focus and
Enterprise Cobol. Another useful one is abort(), to abnormally terminate a batch program.
It surely is cleaner than dividing by zero, which is how mainframers once did it.

>> Avoid REDEFINES; it's a sign of amateurish programming.
>That's right. But, what about my problem with initializing some of
…[4 more quoted lines elided]…
>What is your suggestion for this problem?

Data belongs in databases or files, not in programs.

Assuming you're using a database, there will be two types of tables: reference tables,
which are read-only, and normal data tables. Load all configuration values from a
reference table. 

Database is better than flat file because it is one less environmental thing that must be
synchronized. It reduces the probability of using client A's configuration with client B's
database. To facilitate testing configurations, use a synonym as a layer of indirection. 

Security, such as an encryption key, belongs in a separate layer, not in application
programs. If you MUST read an encryption key, don't leave it in memory in the clear,
obfuscate it by xoring an unpredictable random number.
