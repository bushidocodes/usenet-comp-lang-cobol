```
If you are calling your own functions (entrypoints) from your window
procedure, YOU define the number, type, byref/byvalue and calling
convention. You'd select BY VALUE or BY REFERENCE based on your application
requirements.

If you are calling other Windows API functions, those calls are all
documented in the Windows API documentation. If you don't have your own API
reference, you can use the on-line reference starting at
http://msdn.microsoft.com/library/default.asp
```

```
I assume you have a group item for a paintstruct somewhere.. this is it in
```

```
Kellie, all your parameters appear to be "handles" (32 bit numeric
values) so they should be passed "by value".

COBOL record definitions (structures) can be passed "by reference". An
alternative would be to define a " usage pointer" item (a 32 bit
numeric value) that you would initialize with a "set" statement and
pass "by value".

Character strings (usually null terminated) would also be passed "by
reference" or the pointer alternative described above.

When you use "by reference" the compiler is creating a pointer item for
you , initializing the item and passing the pointer value "by value".
```

```
Passing and Receiving parameters
If you pass a 32 bit numeric by value, then it must be received by
value.
If you pass a 32 bit numeric by reference, then it must be received by
reference.
Depending on the complexity involved it is simplest to pass a numeric
value by value, and receive an additional value in return.
.... call 'EnrtyPoint' using by value aNumber returning anotherNumber
This way you have retained the original number and received an
additional number to be processed.
Handing values off to another program by value ensures that the other
program cannot corrupt your copy of the data.
Handing data off to another program by reference opens access to your
memory to the other program. This is usually done with some kind of
structure where you have assembled data for the other program to
process. Included should be some kind of indicator that the other
program can use to identify problems that exist with the data you
provided. Since you assembled the data you can recover the original
values and reverify its accuracy.
```

```
Note that there is only _one_ copy of each linkage section item when
using entry.  If you CALL .. using by reference lnk-hwnd and have this
as the entry using parameter then the original value changes and
becomes a pointer to its original value.

If this doesn't wind up with a complete corruption then returning back
to the call under WM-PAINT and using the resulting lnk-hWnd there will
certainly be wrong.

When you call an external program with its own linkage section the
items there are entirely distinct from the callers.  When you are
calling an entry there is no separate set of linkage items.  You can of
course set up another set yourself so that the ones used in the
callback are not corrupted by the call.  Or alternately just do a CALL
with no USING and the code in the ENTRY can just use the parameters in
linkage because they have been set up by the callback.
```
