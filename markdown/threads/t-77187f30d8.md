Hello everybody,

I am very much impressed by the current trend for open source software
led by Linux operating system as well as a multitude of application
software, commercialized with the source code at an unbeatable price.

I have a general accounging software, written in COBOL. During the
last ten years, I have sold, an average of three or for copies per
year at a price of 1200.-- Swiss Francs. It has always remained a
marginal activity in my business since I have realized the major part
of my revenue in custom-tailored business applications and other related
computer services and consultancy.

Now I am at a crossread, ready to take a decision as to what to do with
my accounting software. Open the source code to the entire planet and
sell it at an unbeatable price of say, 50 U.S. dollars, just enough to
cover the costs of recording it on a CD and the shipping costs, leaving
me a small margin of profit? Or to continue to content myself of the
modest sales of two or three copies per year, hoping that one day I
shall have, by some miracle, enough funds to invest for promoting
and marketing it, as dictated by business rules, in an already crowded
competition environment.

So, I will appreciate very much your opinion on this subject.

Here is a a brief historics and description of my accounting software.

It was initially developed on a Tany Model III, more than 15 years ago,
before the entry of IBM on the PC market, using the interpreted BASIC,
the only computer programming language then available on this platform.
After the entry of IBM in the PC market, I have acquired an IBM PC AT 
and rewritten it in COBOL, first using the Microsoft COBOL compiler and
later, using the Microfocus COBOL compiler. I have, afterwards, moved it
from MS-DOS platform to MS Windows 3.1, 3.11, Windows 95, Windows 98 and
Windows NT 4.00. I have also rendered it a multiuser, multi-currency
software running under NOVEL network operating system.

It is a general accounting software, which has proved to be solid like
rock during its uitilization of more than ten years by my few but
happy clients. It is simple to use, ideal for small businesses as
well as single-person businesses. The source code is split to a dozen
of modules, each one corresponding to one or more relevant accounting
functions. Altogether, that makes about thirty five thousand lines of
source code. The program is is written, using a structured programming
method, having some similarities to the method of Jackson, but having
after all my own personal method of structured programming. In the
entire source code, there does not exist a single GO TO instruction.
I have given priority to maintainablility of the source code rather
than to optimization for program speed. By carefully choosing the
constant names, the data names and the paragraph names, and by making
them as long as required, upto 30 characters, if necessary, I have
auto-documented it. I have largely used the concept of top-down
programming technique and devided the code to small logical blocks,
placed within the paragraphs. Despite all this, astonishingly, I have
had no speed penalty. My program executes at an honorable speed even
with old slow PCs. The whole source code is written, using only
English language for constant names, data names and paragraph names
except for the user interface elements since it was written to address
the needs of French-speaking clients. However, it would be very
easy to convert the user interface to English or to any other language.

So, please give me your opinion. 

Many thanks in advance.

E. DEMIRTAS
```

Here are some issues that you might want to think about.

1) Intention and Motivation What is your intention in this?  Is it just to
put out another software product that happens to include the source code or
is it to give free software to the world like Linux?

What is your motivation?  Linus Torvald had a passionate belief that software
should be free.  This belief not only motivated him; it attracted many other
people to contribute to his project.  If you are only looking for the best way
for you to make some money, that is an entirely different matter.

2) Copyright, Copyleft or Public Domain
The rights to Linux are controlled by a copyleft.  Any one who modifies Linus
code is required to make their modifications public.

I don�t think that this is practical for business applications.  Specialising
an application for a specific business usually means including their business
policies and procedures in the programs.  This is all information that the
business would not want to make public.

If you were just thinking about including source code as part of the product
that you sell, then a normal copyright would work.  Otherwise, I think
putting the source code into the public domain is the only practical option. 
This implies that other people could take your source code, make their own
private modifications and sell the result as a commercial product.  Could you
live with this?

3) Role
Linux is not just some software.  It is an ongoing development project.  If
Linux is your model, what sort of role do you want for yourself - project
manager, contributor, exploiter?

4) Tools
My choice of tools would be the Fujitsu v3 COBOL.  If is a full scale ANSI 84
COBOL development environment that is free for the downloading
(www.adtools.com) and does not require any runtimes.

Good points.  I've been thinking about this for a while and I have
some additional thoughts.

I've been part of "open source"  for a while.  My first encounter was
with the BBS software WWIV.  Registered users were given the source
code, and they could modify or "Mod" it in order to customize their
BBS.  Other sysops published their Mods and allowed others to install
them.  Distribution of the source code was prohibited - and while
pirate copies existed, for the most part this was successful.
Periodically the author would release new versions of the entire code
base - and the whole customization cycle would begin again.  

Toward the end (and it's still actually going, but the dial up BBS is
dead) there were professional modification services available for
those who did not want to do the modificaitons themselves.

This is not PURE open source, as it is currently considered, but it is
something similar.  

Now, the PROBLEM  (until recently) with COBOL projects was the sheer
COST of the compiler.  There are few hobbiest COBOL programmers - most
are professionals working as COBOL programmers.  With Linux, WWIV and
others, you have a lot of part timers and hobbiests.  The compiler is
free and easy to get.  

With Fujitsu v3 being free (as long as it is available) it makes a
fair platform.  Fair only because 4.2 is so much better than v3 (not
that v3 is bad). 

To be truly popular, an open source COBOL project would need to be
cross system compatible - at least in the root code base.  For an
Operating system  this is no big deal, but for an application this is
a major headache.  Designed properly perhaps not to large a headache.
I would isolate the user interface and File I/O from the rest of the
system.  

Those are just some preliminary thoughts.  I've got more running
around in my head, but I can't get them out right now.
```
