Oh, OK, I'll take it.

COBOL/LE is a misnomer or shorthand (depending on your point of view) for the
latest COBOL compiler from IBM. This compiler needs to have Language
Environment available in order to run.  LE is included with OS/390 but optional
with MVS.

The compiler has already gone through three official names:

COBOL/370 - Version 1, Release 1
COBOL for MVS & VM - Version 1, Release 2
COBOL for OS/390 & VM - Version 2, Release 1.

There ya' go.

Without a lot of pre-amble I'm assuming you mean IBM m/f.
IBM COBOL ( like other semi-prorietary brands ) has many flavours.
I suppose you are referring to either :
COBOL for MVS or COBOL for OS/390.
Once upon a time the execution environment and compilers had the same
'name'
vis-a-vis COBOL II and arrived on the same tape and used the same
libraries.
Nowadays you have a compiler ( as above ) and an 'execution environemnt'
which 
contains the environmental stubs for your code to talk to the OS.
This collective run-time environment is called 'Language Environment' or
LE.
LE may contain support for at-least and/or the following
PLI
Assembler 
COBOL ( at least 4 flavours )
C

However most 'old timers' would ( mistakenly) speak of COBOL/LE.
What they mean is COBOL (II / for MVS / for OS390 ...) + Language
Environment.

If I may add to the excellent response from
"Ian Reid" <reid.ian@pmintl.ch>


On the IBM mainframe, LE was optional with previous compilers. With the one
that will take you successfully into the new millenium, with vendor support, LE
is mandatory. That is OS/390 has LE, here it is. TA-DA!

Actually, a lot of it can be parametrically defused. But the point is that
system programming can make manifold decisions about runtime options, with
parameters THEY will own.  Presto, they show up one day in an application's
executable environment, either when you convert, or later when the system's
folks get around to studying for opportunities to 'improve things'.

But, still, LE is largely transparent.  It is a set of opportunities to exploit
code that can be called from all of the languages the previous poster
mentioned.  The point is to get exactly identical behavior across all
application systems, regardless of the tools used to develop them.  The
internet, it seems, is hastening this phenomenon known as 'convergence'. An
important fact about convergence is that the things converging come from very
different programming disciplines, and there are some incompatibilities.

On PCs, for example, you will find versions of Java that do not handle floating
point numbers according to the IEEE standards that are rigorously followed by
chip manufacturers. If such variance gets to the big bucks stored on big iron,
there could be a few complaints.

So what we need for all of this is a foundation. It is being deployed
gradually.  Besides the new date FUNCTIONS that have already become known to
many doing remediation work for Y2k at this time, only a few really basic
things like 'average' are offered as common routines via the COBOL hooks. 
There is more to come.

Each time new bundles of functionality are announced in LE, expect a
constellation of related IBM products which exploit them.  Market share is most
heavily influenced by who is there first.

We can assume, and this is a guess, but seems very likely, that the object
repository will be accessed through LE routines. And though we will have lots
of contenders, it is probable (again a guess) that all TCP/IP functionality
will be accessible through LE routines.

I would expect that LE will deliver the mainframe version of foundation
classes. And when it arrives, the totality of all the training so far
accomplished in COBOL shops will pale to insignificance, as we try to find the
staff to cope with the explosion of detail that will accompany it.

To accomplish convergence, we need to do more than hook programs together. We
will need to disassemble the barriers between the cultures that the different
language disciplines represent.  I think that it is possible that just after
Y2K it will become somewhat difficult to say in a few words what the trend in
business shops is; becasue we will be going in different directions at the same
time.

Today, if my experience is telling, the new technology people are just the ones
supporting the network of PCs for the big iron programmers. And the COBOL
people are just 'those people' who have that faceless legacy code, which causes
so much traffic across the company net while busy people are trying to do real
development for the internet and the company intranet.

If it was just technology, it might merge readily. But the COBOL stuff has an
inertia that is measured by how many people are NOT available to modify it. So
for a long time, LE will simply look like a subset of the routines that service
the COBOL modules.

In most places gradually, in some places suddenly, there will be important
major projects to convert to new technology, possibly only exploiting common
routines, possibly revamping and interfacing intimately with other non-COBOL
applications. The growing list of common functions available in the Language
Environment offering will then be much more relevant.

My prognostication, for what it is worth, is that on big iron the pathway to
the net is paved with bricks backed in the LE kiln.  But if your app is not
headed to the net (and many COBOL apps are not), LE is just a subset of the
routines that look like the overall set of COBOL language service modules. 

In this regard, Y2K is an opportunity that the vendor exploited to get OS/390
and LE in place for the convergence that follows. There will be a remarkable
uniformity of operation system environments as of 1/1/2000. Not a bad trick,
all things considered. 

Best Wishes,
