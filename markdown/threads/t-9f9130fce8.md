I am trying to write a utility program that will read a file, and
split it into several others.

The input file can have any record length (80-683 bytes). Based on
the first character, it will be split into one of three output files.

do I write the correct length record when I do know it?

UNIX machines. It is the basis of a series of FTP's.

I assume you are not referring to variable-length records but to
fixed-length records, whose length depends on which file you read.

I can address the MVS side but I've never done COBOL under UNIX, so
there I must remain silent. However I very much doubt that the same
code will work in both regimes.

For reading the file: in the FILE SECTION, code RECORDING MODE as U.
Code the record size as the maximum size of a block, which I think
is 32,768. The program will read an entire block of records at a
time, in a raw format which includes special length bytes from which
you can determine the record length and the number of records in the
block. I did this once, so I know it's possible, but I don't
remember the details of the format. If you can't find a good
reference, do some experimenting.

For writing the file: you might be able to get away with the same
trick in reverse -- I never tried it. Otherwise I suppose you could
code a long series of SELECT/ASSIGNs and FDs, one for each possible
record length, and only open one of them. ASSIGN them all to the
same external name.

If that doesn't work (there may be a limit on the number of files
you can code for), you could use one program to identify the record
length (using RECORD FORMAT of U) and set the return code
accordingly. Then in the JCL, code a long series of job steps, one
for each record length. Use condition codes to execute only the
one you want.

Another thought: use the first program to write JCL to an internal
reader. Then execute the generated JCL.

All of the above is really ugly stuff. COBOL is simply not an
appropriate language for this kind of task.

At the risk of touching off another language war, let me suggest that
this task would be almost trivial for C. In C370 you can specify DCB
information in the fopen() call. Therefore you can pick your record
length at runtime. Assembler would work too, if that's easier for
you. Good luck.

MVS. I cannot tell you which utility would be best on UNIX.

It is possible to modify your COBOL DCB before opening as one possible
approach, or to read data from other control blocks (JFCB, etc). However,

specific, as what control blocks are used, and how they are formatted, is
*very* OS specific.

consider it, as this sort of thing would be a snap in that. Also in SAS,
if you happen to have it on all of your target platforms (less likely, much
more expensive).

As I said... "I don't know anything about UNIX". My reply was
specific to MVS.

The FD I quoted *WILL* work for RECFM=FB files on MVS. Test it
yourself. It *WILL NOT* work for RECFM=VB because it specifically
states RECORDING MODE F. A RDW is not required for this FD. It is
not necessary to specify undefined or unformatted record formats.
Another example:

FD MY-FILE,
LABEL RECORDS ARE STANDARD,
RECORDING MODE IS F,
BLOCK CONTAINS 0 RECORDS,
RECORD CONTAINS 0 CHARACTERS.

01 MY-INPUT-RECORD PIC X(32760).

Cobol on MVS will open and read ANY fixed length RECFM=FB file with
this single FD. If the actual file has an LRECL of 200 for example,
cobol will treat it as if you had coded pic x(200). You simply put
the max in the FD. It is up to the user of course to determine,
usually from data, what to do with the record once it has been read.
Usually move to one or another copybook depending on a record code
and so on.
