Dan Hite wrote:

› I recently changed employers and my new shop insists on the use of Perform

Well, when in doubt fall back on the Timesheet Reason: if the person who
signs your timesheet says to do something then it is a *fine* reason!

› The only use I can see would be when one wants to be able to break out of a
› paragraph's logic at varying points without making the whole paragraph a
› group of nested Ifs.

That's what I've used it for (see further down here for more details);
if a record fails an edit at the top of a paragraph then blammo, GO TO
PARA-EX, end of story, finito, Benito.

>When I ask what the point of the standard I am told
› that it is necessary for structured programming (baloney) or that there is
› some advantage to it in the final compiled load module.

The bit about load module efficiency is addressed in the SECTION vs
PARAGRAPH thread... as for 'structuring'... well, there's stucturing
and... structuring. I prefer the modified Yourdon structure which
allows a GO TO under three conditions:

1) To the label of the paragraph containing the GO TO, e.g.

0100-READ-LOOP.
READ INFILE (etc. etc.)
IF NOT VALID-REC
GO TO 0100-READ-LOOP.
(personal note: I try to avoid this)

2) To the EXIT-paragraph of paragraph containin the GO TO, e.g.

0100-READ-LOOP.
READ INFILE
AT END MOVE 'Y' TO EOF-SW
GO TO 0100-EXIT.
(personal note: I use this'un, all the doo-dah day... and yes, I have
had the discussion about avoiding this by coding the READ at the bottom
of the paragraph already)

3) To the ABEND paragraph to blow the program up, e.g.
(note: this code assumes a file-status item with appropriate 88's for
GOOD-IO)

OPEN INPUT INFILE.
IF NOT GOOD-IO
MOVE FILSTAT TO WK-ERR-FILSTAT
MOVE WK-HSKPNG-PARA-LIT TO WK-ERR-PARA
GO TO 9999-ABEND-KERFLOOIE-MACHEN-PARA.

(personal note: I use this'un all the time, too)

Other schools of structure have other rules, of course... Modula-2, I
believe, was the GOTO-less language, and went nowhere.

This is *all* personal opinion and habit, of course, and the usual
caveats apply (your mileage may vary, professional programmer on a
closed track, kids, don't try this at home, void where prohibited,
sorry, Tennessee!)

Dear Dan,
The main reason shops do it this way or that way is for standards. In
COBOL 68, it actually stop problems caused by inattentive coding. COBOL
74 removed most of the need, technically.
Side note, I was in charge of a programming team in the military on
non-standard software(ie we developed and maintained the software
throughout it life cycle). Experienced programmers from multiple
services specifically the Air Force did not use the through -- exit. We
had a raging debate and a lot of hurt feelings(you know how some
programmers can be anal about "The right way to do it"). After playing
both ways, the through--exit was more effective. Less time was needed to
debug and it actually helped keep down the cost of maintaining.
A lot of our systems were for time critical information and a lot of
interactive programs(we even had to build our own screen painting and
retrieval programs) were using living data(tracking aircraft movement,
troop movement, daily food stores and equipment availabibly. There was
little if any time for extensive testing, so the code had to be tight to
be begin with. It did help, but I heard about for 5 years.
I hope this explains some.


Dan

PS Some old farts still do that way because they don't know any better.
