### TRUNC(OPT) vs TRUNC(BIN) and IBM arithmetic code generation

The logical differences for the various TRUNC options only have meaning
when at least one of the fields involved is BINARY. All other truncation
is according to PICTURE. TRUNC(OPT) truncates to the word size if all
operand and results are either literals or BINARY fields and to the
PICTURE if there are any DISPLAY or PACKED-DECIMAL operands. TRUNC(BIN)
truncates to word size for binary fields in all cases. TRUNC(STD)
truncates according to picture to comply with the COBOL standard.

1. A move of a field defined as S9(10) DISPLAY containing the value
2,000,000,000 to a field defined as S9(9) BINARY will result in the
value of zero if the compile options are either TRUNC(OPT) or TRUNC(STD)
and 2,000,000,000 if the option is TRUNC(BIN).
2. A move of a field defined as S9(10) BINARY containing the value
2,000,000,000 to a field defined as S9(9) BINARY will result in the
the receiving field containing the value 2,000,000,000 if the compile
option is either TRUNC(OPT) or TRUNC(BIN) and zero if the option is
TRUNC(STD).
3. An add of 3 to a field defined as S9(9) BINARY containing the value
999,999,999 will result in 1,000,000,002 if the compile option is
TRUNC(OPT) or TRUNC(BIN) and 000,000,002 if the compile option is
TRUNC(STD).
4. A move of a field described as S9(9) BINARY containing the value
2,000,000,002 to a field described as S9(10) DISPLAY will result in
the receiving field containing 2,000,000,002 if the complile option is
TRUNC(BIN) and 000,000,002 if the compile option is TRUNC(OPT).
5. The same move as in 4 except with the receiving field having a usage
of BINARY results in the receving field containing 2,000,000,002 for
both TRUNC(OPT) and TRUNC(BIN).

The general conclusion is that if you are dealing with fields where the
contents can exceed the picture size and the fields are never used with
fields described with a USAGE DISPLAY or PACKED-DECIMAL and are never
used in a DISPLAY statement, then TRUNC(OPT) is safe. It is also safe
if in fact the contents can never exceed the PICTURE size.

If the field described as BINARY must be moved to a field described as
usage DISPLAY or USAGE PACKED-DECIMAL either TRUNC(BIN) must be used as
the compile option or the binary field must be moved to an intermediate
field described as S9(10) BINARY.

BINARY arithmetic code generation is better with TRUNC(OPT) if the BINARY
field is described as S9(8) rather than S9(9). I think that the code
generator is a generalized routine that handles overflow correctly for ADD
A, B, C, D GIVING E (or ADD A, B, C TO D) where A + B + C is greater than
2,147,483,647 and D is negative (or some variation thereof). The special
case of only two operands where there is only final overflow rather than
intermediate overflow with a final result within the limits is not
handles separately.

The current code works and the question of whether we as agents of our
employers or contracting companies would recommend risking changing the
code generation in the lead-up to the year 2000 is open to question. The
optimizations done to move performed paragraphs in line and others that
are extremely clever may have done far more to improve performance than
anything that can be done with the inefficiencies in arthimetic code
generation. On the other hand I see no reason why the same code generator
can't be used for pure binary arithmetic for both TRUNC(OPT) and TRUNC(BIN).
