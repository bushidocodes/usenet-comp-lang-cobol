Hello all.

Here's a (hopefully) simple question that will surely divulge my lack
of COBOL expertise.

I have a file opened with ACCESS IS DYNAMIC. Let's call this file
CHILD. File PARENT has a similar key structure, with the CHILD file
containing (possibly) multiple records matching the PARENT key.

I set the CHILD key to the PARENT key, and search with READ CHILD KEY
IS child-key. So far, so good - I find the record, if it exists, and
bail if it doesn't. Now I want to read sequentially until there are no
more matching child records. This is where the trouble starts.

The code looks like this:

MOVE "N" TO last-child
MOVE parent-key TO child-key
PERFORM WHILE last-child = "N"
READ CHILD INTO child-rec KEY IS child-key
INVALID KEY
[imperative statements A]
MOVE "Y" TO last-child
NOT INVALID KEY
[imperative statements B]
READ CHILD NEXT INTO child-key
AT END
[imperative statements C]
MOVE "Y" TO last-child
NOT AT END
IF child-key NOT= parent-key
MOVE "Y" TO last-child
ELSE
[imperative statements D]
END-IF
END-READ
END-READ
END-PERFORM

When I run this, all goes well as long as there's only one child
record. As soon as the routine hits multiple child records, it goes
into an infinite loop over statements in [D].

Any idea what might be causing the infinite loop behavior? And is
there a better way to construct this routine? My reference text is
weak when it comes to explaining dynamic file reading - less than one
page that basically says "Read like RANDOM, or read like SEQUENTIAL,
and here's how to code 'em." Nothing more.

Thanks -
- Paul

Paul,
The first READ in your loop is a random read, it always resets the file
pointer to the first record that matches the parent's key, the READ child
NEXT gets the second child record (if there is one ) and the random read
sends you back to the first. If you want to use dynamic file access
correctly do :-

MOVE parent-key to child-key
START child-file key not < child-key
invalid key
perform not-found-routine

then you can read next until you hit an invalid key. The START sets the
file pointer but it does not retrieve any records, you must do a read for
that.

Hi Paul,
You can dispense with the INVALID KEY and last-child flag by
using a "FILE STATUS IS file-status" clause on the SELECT CHILD
statement.

In Working-Storage, you then define file-status PIC XX. I
usually give it an 88 level name of file-status-00 as well.

The READ statements can then be just READs, with no AT END or INVALID
KEY phrases to worry about.

Your code then becomes:

START CHILD KEY GREATER THAN OR EQUAL Child-Key

IF file-status-00
READ CHILD NEXT RECORD [into storage]

PERFORM UNTIL NOT file-status-00
OR NOT child-key = parent-key

process record

READ CHILD NEXT RECORD [into storage]
END-PERFORM
END-IF

This is how I program VSAM file access routines.

Hope this helps,
Steven.

I would establish a FILE STATUS clause in the SELECT statement to tell
the program where the VSAM return codes are at:

FILE-CONTORL.
SELECT PARENT-FILE
ASSIGN TO PARENT.
SELECT CHILD-FILE
ASSIGN TO CHILD
ORGANIZATION IS INDEXED
ACCESS IS SEQUENTIAL
RECORD KEY IS CHILD-KEY
FILE STATUS IS VSAM-STATUS.

There, some of the 88-level conditionals are:

WORKING-STORAGE SECTION.
01 VSAM-STATUS PIC X(02)
88 VSAM-SUCCESS VALUE "00".
88 VSAM-FILE-ALREADY-OPEN VALUE "97".
88 VSAM-EOF VALUE "10".
88 VSAM-INVALID-KEY VALUE "20".
` 88 VSAM-NO-RECORD VALUE "23".

Then using PARENT as the driver file I do:

PROCEDURE DIVISION.
000-START-PROGRAM-RUN.
OPEN INPUT PARENT-FILE
OPEN INPUT CHILD-FILE
IF NOT VSAM-SUCCESS
AND NOT VSAM-FILE-ALREADY-OPEN
THEN
--- process program abort routine ---
END-IF
MOVE 'N' TO WS-END-OF-FILE-SWITCH
PERFORM 800-READ-PARENT-RECORD
PERFORM UNTIL WS-END-OF-FILE
PERFORM 200-GET-CHILD-DATA
PERFORM 800-READ-PARENT-RECORD
END-PERFORM
CLOSE INPUT PARENT-FILE
INPUT CHILD-FILE
STOP RUN
.
200-GET-CHILD-DATA.
MOVE SPACES TO CHILD-DATA
MOVE PARENT-KEY TO CHILD-KEY
MOVE CHILD-DATA TO CHILD-RECORD
PERFORM 801-START-CHILD-FILE
IF VSAM-SUCCESS
PERFORM 802-READNEXT-CHILD-RECORD
END-IF
PERFORM UNTIL NOT VSAM-SUCCESS
OR PARENT-KEY IS NOT EQUAL TO CHILD-KEY
--- process the child record ---
PERFORM 802-READNEXT-CHILD-RECORD
END-PERFORM
IF VSAM-NO-RECORD
THEN
--- process no child record found ---
END-IF
.
800-READ-PARENT-RECORD.
READ PARENT-FILE INTO PARENT-DATA
AT END SET WS-END-OF-FILE TO TRUE
END-READ
.
801-START-CHILD-FILE.
START CHILD-FILE
KEY EQUAL CHILD-KEY
END-START
IF NOT VSAM-SUCCESS
AND NOT VSAM-NO-RECORD
THEN
--- process program abort routine ---
END-IF
.
802-READNEXT-CHILD-RECORD.
READ CHILD-FILE NEXT
INTO CHILD-DATA
END-READ
IF NOT VSAM-SUCCESS
AND NOT VSAM-EOF
THEN
--- process program abort routine ---
END-IF
.

The remainder is left as an excersize.

The above is a little lengthy, but usally skip sequential record
processing, which is what CHILD is, needs to be shown in context.
Important to note is the need to follow almost every VSAM action with
checking the FILE STATUS to determine what to do next. The
conditionals in the PERFORM UNTIL loop are checked before executing
each increment of the loop, hence the need to set up CHILD as another
driver file within the context of the PARENT driver file processing.

It works for me,

Boyce G. Williams, Jr.
