```
Does anyone know (or agree with me that this would be desirable)
whether or not the new standard will have some way of
limiting the scope of the evaluate verb when using
evaluate true?

Ie

77 SW1           pic x.
  88 AA  VALUE "A".
  88 BB   VALUE "B".

77 SW2            PIC X.
  88 CC VALUE "C".
  88 DD VALUE "D".


MOVE "A" TO SW1.
MOVE "C" TO SW2.

EVALUATE TRUE
  WHEN AA  DISPLAY "A"
  WHEN BB  DISPLAY "B"
  WHEN CC DISPLAY "C"
  WHEN DD DISPLAY "D"
END-EVALUATE.

        I would like for the compiler to at least complain about
the above evaluate.  Since this is currently legal, it would
need some new syntax, or a compiler option.
```

EVALUATE TRUE will take the first matching case if there are multiple
TRUE conditions.  

The example you created will display "A".  If you want it to display "A"
and then display "C", you could code something like this: 

EVALUATE TRUE
  WHEN AA DISPLAY "A"
  WHEN BB DISPLAY "B"
END-EVALUATE

EVALUATE TRUE
  WHEN CC DISPLAY "C"
  WHEN DD DISPLAY "D"
END-EVALUATE
    

What would you like the compiler to do instead?

What might be desirable is if a variable can be restrained in the values
it can hold in some way and setting and testing of this variable can be
checked at compile time.  For example:

    01  My-flag          PIC X VALUE "N"  VALID "Y" "N".

Then 'MOVE "X" TO My-Flag' or 'IF My-Flag = "X"' should be considered as
compile time errors.

Of course I have no idea how a compiler could ever determine this, while
it may appear trivial in the above examples it may be that another
variable is moved to My-Flag and the compiler could never know what
value it will hold at run-time.

Given the compiler could do this it will still not error the EVALUATE
TRUE example but it might be able to check:

      EVALUATE SW1
      WHEN "A" ..
      WHEN "B"
      WHEN "C"      <-- invalid value for SW1

I doubt it will ever be a feasible proposition.  I certainly wouldn't
rely on this check fixing any _programmer_ errors (as above).
