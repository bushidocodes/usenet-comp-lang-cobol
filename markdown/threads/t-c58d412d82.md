Generally not good practice this, it is best to point your FD to one
file, open it, process it close it. This process can be repeated over
and over, changing the name of the file you are pointing to each time
you enter the loop. In this way one FD can be used for many files.

Errors can be blocked by using the DECLARATIVE section within your
program, you will still get the errors but it won't crash the program.

You will then have to handle all the errors yourself.

OK, you're not using COBOL file handling syntax. So am I correct in
assuming that the "FD" you're referring to is actually a "FILE *" or
an "int" or equivalent returned from the C program ? This might explain
some of the confusion, as "FD" is part of the COBOL syntax for defining
file handles.

The C routines, if they offer an API to control these non-COBOL file,
should offer a "CLOSE" entry point.
So just call it to close the file when you've done reading it.
If you "lose" the fd (by assigning a new value to it when you use it
to store an fd for a newly opened file), then you're correct - you
will have a file handle leak in your program. The reason they will
close when running as a seperate executable is because Unix will close
all open file handles for you when the process terminates (BUT NOTE:
if the C functions do any caching of written data (FILE * streams,
for instance, do), then the OS may close the fd's behind their back,
and you will not have all your data written to disk! - *always*
explicitly CLOSE files you have OPENed, if at all possible).

› My Idea was to close the FD that is in the parameter transfered to
› the C routine "just in case" before filling it with new FD.

That's sound, except for the "just in case" bit. Say that it has
already been closed since you OPENed it, and re-used by a subsequent
OPEN (by another module or whatever) - you're now closing a file handle
which doesn't belong to you.
For each OPEN, explicitly perform a single CLOSE on that FD, and you
should be OK.

Far better would be to use COBOL syntax for accessing your files, as
the COBOL Runtime System is more forgiving than the C libraries.
I assume there is some other reason why you can't do that, though.

Cheers,
Kev.
