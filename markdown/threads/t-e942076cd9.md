```
First, I should mention that the mainframe I am referring to is VSE, not
MVS;OS/390;z/OS.

I am doing TCP/IP programming.  In TCP/IP a port is an unsigned halfword
value.  In other words, it can contain values from x'0000' (0) to
x'FFFF' (65535).

A halfword has to be defined in COBOL as PIC 9(4) BINARY (or PIC 9(4)
COMP).

Now, in order to allow values greater than 9999 to be moved to this
field I am using the TRUNC(BIN) option.  This is all well and good.  I
can do
01  PORT          PIC 9(4) BINARY.

MOVE 10000 TO PORT.
or
MOVE 65535 TO PORT.

My issue is what happens when I do the following:

01  PORT-HALFWORD  PIC 9(4) BINARY.
01  PORT-DISPLAY   PIC 9(5) DISPLAY.

MOVE 65535 TO PORT-HALFWORD.
MOVE PORT-HALFWORD TO PORT-DISPLAY.
DISPLAY PORT-DISPLAY.

If I use a value greater than 32767 (well, 32768 sort of) I am not
getting my desired results.  In this case PORT-DISPLAY would end up as
00001.

Now, I know exactly why this is.  For whatever reason it's treating
PORT-HALFWORD as signed, even though I have defined it as unsigned.  
x'FFFF' is 65535 when treated as unsigned, but 00001- when treated as
signed.  Since PORT-DISPLAY has no sign field it's moving the value of
0001-, and then of course not displaying the sign.

Now, don't get me wrong, I know how to get around this problem.  I can
do the following:

01  PORT-HALFWORD-X.
    05  PORT-HALFWORD       PIC 9(4) BINARY.
01  FULLWORD                PIC 9(9) COMP SYNC.                      
01  FULLWORD-X              REDEFINES FULLWORD.                      
    05  HALFWORD-HIGH       PIC X(2).                                
    05  HALFWORD-LOW        PIC 9(4) COMP SYNC.                      
    05  HALFWORD-X          REDEFINES HALFWORD-LOW.                  
        10  HW-BYTE1        PIC X.                                   
        10  HW-BYTE2        PIC X.                                   

MOVE 65535 TO PORT-HALFWORD
MOVE PORT-HALFWORD-X TO HALFWORD-X
MOVE FULLWORD TO PORT-DISPLAY
DISPLAY PORT-DISPLAY

But it's simply annoying to have the extra working-storage and extra
move!  Anyone have an better ideas.  Note that the COMP-5 option that I
believe is available for COBOL for OS/390 is *not* available for COBOL
for VSE/ESA.

It's interesting to note (for you VSE people) that TCP/IP for VSE has
this same issue when you use the "QUERY CONN" command.  For ports
greater than 32767 they are listed incorrectly because (I'm guessing)
the code is treating the halfword as signed instead of unsigned.  And
it's not even COBOL!  :-)
```

```
There is a statement at:
   http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igy3mg00/4.6.1

which says,

 "TRUNC
    In releases of COBOL for OS/390 & VM prior to Version 2 Release 2,
unsigned binary data items  with TRUNC(BIN) were correctly supported only
when the binary value contained at most 15 bits  for halfwords, 31 bits for
fullwords, or 63 bits for doublewords. In other words, the sign bit was not
used as part of the numeric value when the   data item was unsigned. With
Enterprise COBOL and  COBOL for OS/390 & VM Version 2 Release 2, all 16 bits
of a halfword, all 32 bits of a fullword,   and all 64 bits of a doubleword
can be used as part of the numeric value of an unsigned COMP-5  data item or
an unsigned binary data item with  TRUNC(BIN).

For example, in a program compiled with TRUNC(BIN), a data item declared
like this

                              01 X pic 9(2) binary.

correctly supported binary values from 0 through only 32767 in prior
releases, but with Version 2  Release 2 now supports values of 0 through
65535.

This support necessarily yields different  arithmetic results than were
obtained with the prior releases, if these very large unsigned binary values
were inadvertently used. "

    ***

As I recall the COBOL for VSE hasn't "yet" reached the "technical" level of
IBM COBOL for OS/390 & VM V2R2 - so my guess is that your compiler still has
this "bug" (which IBM *might* call "working as MIS-designed").  The only
"solution" would be to contact IBM to see when/if they hope to get the VSE
compiler to have this "fix".
```

```
        BTW, the larget number port-number you can have (and is documented
in the IBM TCP/IP Documentation) is 32767. This holds true even in an
ASSEMBLER TCP/IP Interface with an "H" directive (aligned or COMP SYNC) or
an "XL2" directive (unaligned or COMP) defined for the port-number. No way
around it. The API parmlist defines an unsigned halfword-binary, aligned or
unaligned. We ran across this during our VAP TCP/IP Interface developement.

        Just make sure they're not assigning a port-number in the VAP with a
value exceeding 32767 and you'll be OK.
```
