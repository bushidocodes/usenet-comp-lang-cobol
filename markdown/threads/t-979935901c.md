```
Hi !

I have developed an application using the latest version of MF cobol,
32-bit.
The application is running on novell and Windows NT servers.
The application consist of 80 programs, some of them with a lot of
file-handling. My data-files is created with "lock mode is automatic", and
the programs are compiled with the directives "nodetectlock" and "retrylock".
This work fine (almost)
but: Some of my programs consist of a lot of open, close, read and write.
Very often, I read a file with open input, and if the user want to change
something, i close the file and open again with open I-O and then read the
record again.

Would it be better if I:
-define "lock mode is manual"
-open the file I-O early in the program
-read the file without lock (read file with no lock)
-if the user want to do changes, read the record with lock


Is file open/close time-expensive, running over a network ?
Thanks for any comments !!!!


Greeting !
Arild Andersen
Trondheim, Norway
```

```
Repeatedly openning and closing the file is expensive. I suggest using manual locking and only lock the record when it is read for update.
```

```
I routinely write "utility" programs that read a large file (500,000
records) sequentially and change some of the records, but not most. On
these I generally read the file sequentially unlocked and when I find a
record that needs changing, I lock it, change it, and rewrite it. This
appears to be the most efficient way.
```
