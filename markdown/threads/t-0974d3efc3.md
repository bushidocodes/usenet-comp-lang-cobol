I'm testing a plain data entry CICS program and once the fields are edited I
have to move them into an output file.  The input consists of several numeric
fields one of which is quantity and another price.  Quantity is defined on the
screen as a PIC X(9) field and price as PIC X(10).   After all detail records
are processed I have to write a total record to the file which is the total of
all the (quantity * price)s.

The quantity field ends up on the output file as a PIC S9(10)V9999 and the
price as PIC S9(14)V9999 (no - I did not define these fields and yes they are
carved in stone).

I use the function NUMVAL to move the screen fields to the output record
fields.  The price converts just fine - but the quantity does not.

Straight from my Xpediter screen (it is a bit contorted from cut/paste but I
think you can see what I mean):

                        
03 ED810O-300-QTY-INVD 
    .000000004000{      
    0FFFFFFFFFFFFC      
    80000000040000      
    1...*...10....      


What's with the x'08' in the first position?  

Here are the other fields - again cut/paste from Xpediter:

LV ----- COBOL DATANAME ---------- -- ATTRIBUTES -- ----+---10---
02 WS-TOT-AMT                      S9(8)V99         +0000005336  
02 WS-TOT-AMT2                     S9(8)V99         +2800005336  

The first total was obtained using the 9 position working storage field for the
quantity, the 2nd using the output file field in the compute statement.  The
'28' is actually a truncation - the number is more like 16040...28 :). 

LV ----- COBOL DATANAME ---------- -- ATTRIBUTES -- ----+---10----+---2
03 ED810O-300-QTY-INVD             S9(10)V9(04)     .000000004000{     
03 ED810O-300-UM                   X(02)            EA                 
03 ED810O-300-UNIT-PRICE           S9(14)V9(04)     +000000000000133410

You can see the price field here and the quantity field not in hex.  I also
noticed that the quantity field has the sign ( { ) at the end while the price
has +.

What makes this even more interesting is that when looking at the field in the
output record that position is x'F0'.  And when I use it in the multiplication
I do not get an ASRA (S0C7) but a wild total amount.

If I use a PIC S9(9)  field as the target field for the function the conversion
is just fine.

Any ideas?  Reasons?

Eileen

If worse comes to worst and you can't figure out/get the numeric bits into
the field, you could redefine the quantity field:

 03 ED810O-300-QTY-INVD       PIC S9(10)V9(4).
 03 FILLER REDEFINES ED810O-300-QTY-INVD
     05 ED810O-300-QTY-INVD-WHOLE  PIC 9(10).
     05 ED810O-300-QTY-INV-FRAC       PIC SV9(04).

Then you can UNSTRING the text entry delimited by decimal point (or use
function numval on a reference-modifed portion of the input). You'll have to
adjust the sign to make sure the sign overpunch ends up in the "frac"
portion but not in the "whole" portion.

Crummy way to code, yes, but when you gotta go, you gotta go.

(Good thing the output is USAGE DISPLAY. This gets real ugly when it's
BINARY or PACKED-DECIMAL).

A hex 08 is a screen attribute byte.  It indicates a field is
unprotected, numeric with high intensity on.  It would seem then that
your definition of the input field or the field you moved to
ED8100-300-ATY-INVD is off by 1 byte.  Another disturbing (at least to
me) fact is that if this indeed the screen field and it's attributes,
the attribute for a data entry field that was entered should have the
MDT on.  So the attribute for this field should be a hex 09.  

We are talking CICS and BMS maps here are we not?
