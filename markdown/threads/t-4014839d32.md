Hi folks. I've done a bit of snooping inside the 650 compiler package
and I really have doubts about where it comes from. If you look at the
first program that runs, say using DOS debug, you'll find that it's
written in a different language from the rest of the package, probably
Turbo C. The sole function of this program is to do an EXEC call which
loads the first overlay into memory and passes it command line
parameters which come from the "Author"s screens. In fact, it is
possible to rename the first OVL which runs to a .COM file and invoke it
just as you would early Microsoft/IBM/MicroFocus products:

cobol source,object,list,map (I think).

Indeed the overlays and libraries are remarkably similar to such early
products, except that the two library files usually found have been
concatenated into one library. In addition, cobol 650 has the same bugs
reported in IBM cobol version 1.00:

- All of this nonsense about needing to include a program called DPATH
or using DOS subst commands arises because the program uses MS-DOS 1.0
type file control block (FCB) system calls. Replacing the appropriate
drive specifier byte in the FCB which is 1 (drive A) with 0, current
drive, permanently fixes the spurious requests for files in drive A.

In addition, library files expected by the linker can be renamed.
Usually a linker from DOS 2.0 and up is smart enough to look for or ask
for an alternate library file.

Lastly, there is a nasty but well documented bug involving printing out
a COMP number 32768 which arises because the binary to decimal prinout
routine makes an unwarranted assumption about what sign extending does
on the 8088. (It does *not* clear the high part of a register if the
number involved is *negative*). Anyway, this very bug is present in IBM
PC Cobol 1.00, and at the same location in its LIB file!

Going one step beyond and comparing files from Cobol 650 with files from
a copy of IBM PC Cobol 1.00, there are files in which the only
differences between the two sets are text strings such as error
messsages and such.

One substantial way that this program does differ from the old PC
product is that it does implement SORT and MERGE, so there may well be
original work here, but if you look at the whole package, the simplest
explanation is that someone wrote a loader in another language and used
a hex program editor to change key text strings. If this is original
work, it is hard to see how it was compiled byte for byte so close to a
commercial product!

If you are determined to use this, Microsoft's 16 bit linker is freely
available on the net. If you really are desparate enough to use this
program, please e-mail me and I'll send you my patch files.

Good luck
