Report on March 27-31, 1995 ANSI X3J4 COBOL Committee meeting

This is a bit late - I apologize.

As usual, most of this meeting was spent discussing proposals for
the next revision and working on editorial papers concerning the
base document. There was a one-day editorial ad-hoc the day
before the regular meeting.

The major topics/proposals we discussed were:

1. The proposal on dynamic collating sequences (allows the user
to change collating sequences with the SET statement and will tie
into internationalization to automatically use locale information)
was discussed and will be rewritten. Almost all of the changes
were minor. We decided that when you set a program's collating
sequence to a new one the default comes back when the program is
set to its initial state.

2. The proposal on user-defined types (allows strict typing and
allows a data definition to use another data definition - somewhat
like a source COPY) was discussed and was passed. It will be
rewritten before it is added to the base document. Most of the
changes were minor.

3. The proposal on allowing text-name to be a literal for COPY
was discussed and sent back to me for rewrite. It has already
passed by letter ballot, but needs another review before it can be
added. It was changed to allow a literal for the library-name as
well.

4. The proposal on zero-length data items (clarifies what happens
with the in comparisons and the like) was modified and sent back
for rewrite.

5. The proposal on VALIDATE (validates data items based on
criteria specified in the data description entries for the items)
was discussed at length (over a day's worth). There are some
problems with global and external that have to be resolved. The
proposal was sent back for rewrite.

6. The proposal to allow partial expressions in EVALUATE (as in
EVALUATE x1 WHEN > 10) was passed and will be added to the base
document.

7. The proposal on an accumulative REPLACE statement (doesn't
cancel any previous one) passed and will be added to the base
document.

8. The proposal on conditional compilation and using directives
to define constants was discussed but not completed. There was
some controversy over having more than one way to define
constants. I have a proposal that defines them in the program and
the subject proposal uses directives. We decided to do them both
ways.

9. The proposal on ACCEPT/DISPLAY screen handling (uses a screen
section) was discussed for almost a day, changed a lot, and passed
(7-2 (IBM, UNISYS)). It will be revised, reviewed, and then added
to the base document.

10. The proposal on revising report writer (adds a few new
features and makes it easier to understand and use) was discussed.
It had previously passed by letter ballot (10-2 (CA, IBM)). CA
changed their vote to positive. The proposal will be rewritten
and added to the base document. There will be a quick review of
it at the next meeting.

11. My proposal on defining constants (as in 01 abc CONSTANT
"abc" and for group items) was discussed and sent back for
rewrite. It was decided to not add a constant section.

12. My proposal on recursive programs (adds static and automatic
data and so on) was discussed and sent back for rewrite.

13. A paper on in-line exception processing (ties in with my
exception processing proposal in which all is done in a
declarative) was discussed. There was no agreement on how this
should be done. I was requested to write a paper showing
alternatives and pros and cons.

14. Some minor issues are: a postscript version of the draft
standard will be made available on the internet for ftp access;
the draft standard that is out for review was handed out; and we
are having problems creating the document because we are hitting
Word Perfect for DOS limits - we have no solution to this right
now.

The next meeting was in Tokyo, Japan, on May 22-26 1995. The main
topics were a review of the passed proposals and the VALIDATE
proposal. I will report on this when I get the minutes (I did not
attend the meeting).

While I want to review your posting on the X3J4 meeting before
commenting, I would be interested in the reason for not bringing
back the CONSTANT SECTION (it was a part of COBOL 61 or 63). I
would think that it is the simplest way to tell the compiler that
the constants can be a part of the read-only portion of the code
and that storage does not have to be obtained for them when
generating re-entrant programs. Is there another or better way? Are
modern compilers smart enough to do this if an entire level 01 area is
used read-only? Do they do this for level 77 constants? While the
return of the CONSTANT SECTION is not a major item, it is something
that I would have thought could improve the performance of COBOL
routines in transaction processing systems such as CICS.

I think the reason that the Constant Section was dropped from the
first standard (COBOL '68) was that there were no rules about what
could be done with such items. For example, they could easily be
receiving items (there was no rule that prohibited this). Rather than
putting in such a rule (and invalidating some existing programs), the
ANSI committee decided to drop it. I guess they figured a compiler
could be smart enough to determine if something was a constant or not
(and some do to this day). I wasn't involved in standards in those
days (not until 1971 - I was working on COBOL compilers, though) so
I'm not sure what the exact reasons were.

In my proposal I did suggest bringing the Constant Section back with a
rule that anything in it could not be a receiving item. However, most
of the committee did not like the idea (4 for, 0 against, 4 abstaining
- I was a wimp and abstained). They felt that my other method of
defining constants, as in "01 a-constant IS CONSTANT [VALUE IS]
1234.56" was all that was needed. A constant could be defined
anywhere in the Data Division (File, Working-Storage, Linkage, Report,
Communication). Also, it would be easy to copy constants in with
records and whatnot rather than having to copy something into one
section and the related constants into another section. One part that
was lost was the ability to define a record that is a constant. That
would have been allowed in the Constant Section. You could copy the
record into the Constant Section, where it would remain constant, and
then into Working-Storage or whatever for the one to work on.
However, INITIALIZE TO VALUE accomplishes about the same thing. What
I have decided to do for this is to allow you to define a constant
record in Working-Storage with the first line like this "01
a-constant-record IS CONSTANT." and then subordinate stuff in which
every elementary must have a VALUE. You could use COPY to bring in
the constant and the variable by using REPLACING to get rid of IS
CONSTANT. Any comments on this would be appreciated.

I hope everyone realizes that the REPLACE statement (part of COBOL 85)
can easily be used to define constants. As in:

REPLACE ==a-constant== BY ==1234.56==
==another-one== BY =="some text"==
.

Lots of people do not like REPLACE because a new one wipes out the
previous one. However, this is easy to get around. See my book.
