I have experienced similar problems testing CICS (Transaction Server)
applications.  Fortunately, I have CEMT authority in my test CICS
regions.  Logon on to the CICS file-owning region, clear the screen,
and enter the following command:

CEMT SET FILE(fctname) CLO ENA

That will close the file (forcing the buffers to flush to disk) and
leave the file ENABLED.  Even if you forget to re-open the file, the
next I/O request from any task will automatically open the file.  

If you do not have CEMT authority, you could add code in your test
application program to do the same thing.  You may want to verify the
syntax against the System programmers guide (NOT the application
programming guide), but I believe the command would like something
like this:

EXEC CICS SET
    FILE (fctname)
    CLOSED
    ENABLED
    NOHANDLE
END-EXEC.

Some shops may attempt to restrict the use of SP (System Programmer)
CICS commands in the pre-processor translator step.  In that case you
may be out of luck.  If you know how to override the compile JCL, you
should be able to force the SP option on the translator step.  I'm not
at work right now, but if anyone is interested I can scrounge up an
example of the compile JCL and the actual EXEC CICS command.

I hope that helps!

In my shop, applications developers have security access to some CEMT
commands in test CICS regions, and only to CEMT inquiry functions in
production CICS regions.

Obviously, you would not want to flush VSAM buffers to disk too often
in production, because that could cause response time performance
problems, or other problems with the production application.

I am no expert on RACF, ACF2, and other security systems (we use
ACF2), but I suppose it is possible to write security rules that would
allow a CICS program to update a VSAM while denying the same program
the authority to open/close the file.  The FCT (File Control Table)
entry in the CSD file could also be coded to prevent open/close on a
specific VSAM file.  The fact that a certain set of EXEC CICS commands
requires the SP parameter on the translator step seems like a very
poor security feature to me.  

But I should have been more clear in my original post that these
suggestions were only made to resolve a problem in a TEST CICS
region.  You would never want to do this kind of thing in a production
CICS region unless you had a very specific need.  The only example I
can think of is a SHAREOPTIONS(3 3) file we use to pass messages
between batch and CICS.  In one of my applications we have such a file
and it is open for update to both CICS and batch at the same time.  We
also have application programs that normally open and close VSAM
files, and they use the EXEC CICS SET command for that.

The original poster, Mary Z., wanted to know if there was any way to
flush an ESDS VSAM file to disk, so that their test results could be
verified.  I know of two ways to do that.  There may be other ways I
don't know about.  Setting the file to CLOSED and ENABLED (using CEMT)
should work without disrupting other testing.  That's what I would do
if it were my Test CICS region.
