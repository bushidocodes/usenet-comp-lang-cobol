```
The undertaking which began as a glimmer of an idea around 5 years ago, has
now come to fruition and there is working code available to migrate
PowerCOBOL projects into other environments.

Initially, we have targeted the Windows .Net environment, but it is
comparatively easy to target Silverlight (XAML), and other, non-Windows, GUI
environments.

The goals for this project were probably over-ambitious but as follows:

1. We need to AUTOMATICALLY reproduce the PowerCOBOL Screen layout and the
controls on it, in a more modern environment.

This has been effected by writing code to analyse the PowerCOBOL
screen, create an XML description of it and then use that XML to generate
designer code for the new environment. Currently, it generates Windows Forms
Designer code (in C#) for Visual Studio, so the PowerCOBOL screen becomes a
Windows Form, but it could just as easily be WPF or something else.

2. We need to salvage all the PowerCOBOL COBOL that is embodied in the
scriptlets and event processing for the PowerCOBOL screen.

This has been effected by writing code to analyse the PowerCOBOL
project and get all of the scriptlets and procedures defined in it. These
then get generated into a template which creates a new NetCOBOL COM
component, and that component can be used by the new Windows Form as
"code-behind". The old "scriptlets" now become "methods" of that
code-behind. There are 2 main problems here:
1. Getting the code-behind connected to the controls on the new Form so they
can be addressed.
2. The code contains PowerCOBOL references to controls and attributes of
controls which existed on the original PowerCOBOL screen. That functionality
has to be duplicated in the new environment and there is a huge variety of
possible controls and attributes...(Including third party and custom
controls which are not even defined to PowerCOBOL...)
Both of these problems have been resolved (at least for the basic controls
and attributes), but I am still considering alternative approaches to
resolve them better...
I think we need an extendable "GUI Support" Class, written in COBOL, that
people can write their own COBOL into so they can support custom controls
etc. The GUI Support Class would be called from the generated code-behind
when it encountered things it couldn't deal with.

Of course, there is a lot of scriptlet code that has nothing to do with
manipulating the GUI interface; it is concerned with access to databases and
flat files and checking business rules etc. All of this gets passed through
to the new environment and runs without problem. It is "salvaged" and can be
maintained as NetCOBOL. (Once the new Form and the code-behind have been
generated, there is no further dependence on PowerCOBOL...)

3. The new Form must implement the same functionality that the old
PowerCOBOL screen did.

As you can imagine, a HUGE amount of thought and effort has gone into the
design and build of the new PowerCOBOL Migration Tool (PCOB2NET).

>From a COBOL POV the code (both hand coded and generated) has been
"interesting"... It runs the gamut of accessing the Windows API in COBOL,
through to passing Object references to COBOL COM components around in
memory...

I have tried to make the solution maintainable by COBOL people who have no
experience of OO or COM, but if you have that experience, it probably helps.

You can see the tool running and the COBOL that is produced, at:
http://primacomputing.co.nz/PRIMAMetro/pwrCOBMigration.aspx

Click the DEMO buttons.

I'm interested in all thoughts, comments, ideas, and suggestions. Mail me
privately, or post here.

Thanks,

Pete.


"I used to write COBOL...now I can do anything."
```
