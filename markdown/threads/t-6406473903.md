```
Is there a built in COBOL command/ function to read the file list for
a particular directory. I need to scan a directory to see if any files
of a certain spec (*.output or whatever) are there.

I haven't managed to find a COBOL command to do this so I've resorted
to using the UNIX library but the results I'm getting back are weird.

I'm calling OPENDIR, READDIR, CLOSEDIR and using MEMCPY to move the
results into a COBOL structure. I reckon my problem is with the
definition of USHORTS, ULONGS and INO's etc.

I have them defined as COMP-5 and have varied the number of characters
allocated eg PIC 9(2) COMP-5 or PIC9(4) COMP-5 but still no joy.
The numbers coming back just don't sit correctly in the destination
structure and the file names themselves are just garbage or part of
the directory list with garbage.

The system I'm running on is an IBM RS6000.

Any help would be greatly appreciated, thanks in advance.

K.N.Fraser Travel Technology Systems Ltd (UK).

K.··.@wit··o.uk
```

```
It's OK, I've got it sussed. The PIC 9(4) COMP-5 should have been
defined as PIC X(4) COMP-5 to force COBOL to allocate 4 bytes and not
to deal with 4 digit numbers ( which it hold in 2 bytes).

I've attached a copy of the final routine for those interested.
Thanks to Michael Potter for the pointer modification, it cut out a
couple of calls to MEMCPY and makes the whole thing a lot neater.

Ken.
```

```
Ken,

You should not need memcpy(). Convert the header files to copy books with
h2cpy, you will then see how the data items are defined because your
sys/types.h will be converted. You probably have worked out how to handle
the pointers or you would not have got memcpy to do anything.

E-mail if you want more help. I am currently using AIX 3.2.5 & Cobol 3.2.
```

```
To get the correct structures, try finding their .h files somewhere in
/usr/include and then run the h2cpy utility on them to get the
equivalent COBOL copyfile. There is an old version and a new version of
h2cpy around, the new one is the one you want as it outputs COBOL type
definitions, call prototypes, etc. which makes things much easier. Its
been around for a while and can be distinguished by running it without
parameters when it will list its options.
```
