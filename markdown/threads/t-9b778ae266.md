It's the number of blocks of data on the tape. There is no way from
the header/trailer information to figure out the number of bytes from
this information for a VB tape.

If you can figure average record length then (BLOCKSIZE/ARL) * BLKCNT
should be an approximation, assuming you have a good fit of record to
block size. Of course, you should be using 32k blocks anyway.

We don't use TMS anymore, but as I recall BLKCNT was simply the number
of blocks in a file. If the file was FB (Fixed Block) you could easily
determine filesize in bytes by multiplying the BLKCNT by the BLKSIZE
(ignoring the likelihood of the last block being short).

Since your file is VB (Variable length records, variable block size), it
would be a lot harder to determine filesize in bytes. You could
approximate it by multiplying BLKCNT by BLKSIZE, but it would probably
be a little overstated.

You will likely see serious problems with tape read/write integrity in
an installation that tolerates tapes with enough bad spots on a tape or
cartridge to have a significant impact on tape capacity. Since you are
talking about 36 track, 38K bpi, you must mean 3490E or compatible
drives. This technology always writes using hardware data compression
(occurs within the tape subsystem itself). If you write data that is
already compressed, hardware compression achieves little extra benefit,
and from the software's viewpoint the tape capacity of a standard
cartridge approximates its physical capacity, which is in the
neighborhood of 400 MB. On the other hand, uncompressed data will
typically compress with a compression ratio between 2 and 3, so while
the physical capacity of the media is unchanged, the software would see
an apparent capacity of 800 to 1200 MB. So the capacity of a cartridge
is highly data-dependent; but, if you were to write the same data twice
you would achieve near repeatability.

The only way to reduce file size is to write less data. The only way to
reduce block count is by writing less data or larger blocks (increase
the BLKSIZE - with system-determined blocksize active BLKSIZE already
defaults to the max 32K size), provided your target system doesn't have
lower limits on max block size. The impact of smaller block sizes is
less signficant on 3490E drives because the hardware compression also
improves media efficiency by reblocking the compressed data into "super
blocks" in a manner transparent to the software applications. Although
each block contains a 4-byte control header (so more blocks does mean
slightly more physical data on the tape), any program that is properly
processing the logical records on the tape will see exactly the same
number of data bytes in logical records regardless of the block size and
block count.

For a variable blocked file, BLKSIZE specifies the max allowed block
size. The total file size in bytes can be approximated by
blksize*blkcount, although this could be an overestimate by as much as
avgrecordsize*blkcount , since there may be insufficient space at the
end of a typical block to include the next record (also the last block
is probably short). The relationship totalsize = nrrecords *
avgrecordsize should also be satisfied, within the limits of accuracy
used for estimating the totalsize.

I concur with your interpretation. Although we don't use TMS, The Txxxxx
counts and Pxxxxx counts appear to be temporary errors and permanent
errors, with xRxxxx and xWxxxx being READ & Write errors, respectively.
In all likelihood the physical tape used for volume 2 should be removed
from service and trashed. Unless there is a problem with the physical
drive (which seems unlikely considering vols 1 & 3 looked fairly
normal), any time you get permanent errors on a cartridge tape media, it
should be tracked, as it is likely just a matter of time and bad luck
before this media will give you an unrecoverable permanent read error.
Our tape librarian gets a daily report based on SMF (type 21?) tape
volume statistics records that track temp & permanent errors for each
mount of a volser. Any cartridge that gets a permanent error during a
mount is physically marked with a special sticker. Three strikes and
the cartridge is trashed, although the tape librarian can use his own
judgement and trash it earlier if the temp error count seems
unreasonably high or if multiple permanent errors occur and he knows the
physical drive is performing well on other tapes.

If your installation is not tracking tape errors and replacing marginal
tapes on a regular basis, it should be. It is usually much cheaper in
the long run to trash such tapes than to wait until critical data is
lost and must be regenerated. The remarks I made on normal expectations
of cartridge capacity presumed such media policies were common --
perhaps they are not.
