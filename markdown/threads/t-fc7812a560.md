Then your file is designed in a manner for which I would have  "minimum high
regard." (phrase courtesy Secy. Don Rumsfeld)

If you want to search by name, the name should be either a primary or
secondary key.

Unfortunately, you are not even searching on "name" as a key: you are
searching "lots of text" for "contains."  (e.g., you will not find "mouse"
with a START/READ NEXT on an alternate key). You are essentially reduced to
this:

PERFORM UNTIL String-found or EOF-file
  READ TheFile
   AT END
     SET EOF-FILE TO TRUE
   NOT AT  END
    MOVE 0 TO TALLY
    INSPECT File-Record TALLYING TALLY for SEARCH-STRING
   IF TALLY GREATER ZERO
      SET STRING-FOUND TO TRUE
   END -IF
  END-READ
END-PERFORM

Now, if the file is not going to change during the program run, you can do a
lot of cool stuff by reading the file once at the beginning of the program
run and storing the names in a table or creating an indexed file with
keys=possible search parameters.

Ain't no control on the planet gonna make this job any easier. But if you
have to load all the data to a control anyway (e.g., to facilitate a lookup
by a real key), then you have some options.

Depending on application needs, you might want to reconsider your file
design, your control usage, your search parameters,etc.  FWIW, how many
records are in this file to be searched, and does it change during the
program run? Does it even matter if it changes during the program run?

Spend some time thinking before you spend it coding.

OK. Given the above file (or something close), if you have to look up one
name, read the file sequentially until you find the desired name.

If you have to find many names, first read the file and create a new ISAM
file like:

01 TEMP-FILE.
   02  TEMP-KEY
      05  TEMP-NAME    PIC X(40).
      05  TEMP-RECNUM  PIC 9(6).

Then go after the required names in the TEMP file using the TEMP-RECNUM to
get the key of the original file. I'm assuming here that the original file
contains much more data than a customer number and a name.

Trying to load the entire file into memory may ultimately fail when the
database becomes too large (think Reader's Digest subscription list or calls
to DELL tech support).
