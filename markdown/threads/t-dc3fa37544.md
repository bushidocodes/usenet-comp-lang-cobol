```
The one AND ONLY way to get the current date with a century in (batch or
IMS) VS COBOL II is to call an IBM supplied sub-routine called IGZEDT4.  To
do so, you need to make certain that current maintenance has been applied to
your VS COBOL II software. (Check with your systems people that they have
applied the fix for APAR PN76666.)

If you are running under CICS, then do NOT call IGZEDT4, but instead use the
CICS ASKTIME function (or use the EIBDATE field - but be careful - as it has
a funny way to handle the century).

In the unusual case that you are using the VS COBOL II compiler, but are
running under the LE (Language Environment) run-time (and your programs are
compiled with the RES compiler option) then you can make dynamic calls (use
the CALL identifier format) and call the LE date/time routines.  (This
should work under both batch and CICS)
```

```
I've seen lots of good advice, so won't repeat it. However, just a
reminder ---  IBM has stated that COBOL II is not and never will be
Y2K-compliant... so, my recommendation is to alert your management
that you're saddling a dead horse.  COBOL/MVS has many powerful
features for Y2K projects, none of which are in COBOL II. If you must
spend the effort to convert a system for Y2K compliance, might as well
get the current compiler.  Also, support for cobol ii expires
somewhere around march, 2000 (or is it march, 00?   <g>)
```
