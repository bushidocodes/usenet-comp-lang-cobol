I don't know if this is standard ANSI 85 cobol, but it works in MF, Acu, &
Icobol: When I need to read a variable length ASCII file I use KEYBOARD
in the select statement. eg

select bigfile assign to KEYBOARD bigfile-id
organization is line sequential
file status is FS.

A read on a file selected this way fills the bigfile-record up to the next
set of page control characters. One gotcha: Before doing a read on the
file I always do a

move spaces to bigfile-record.

Otherwise when I read a short record not all of the previous data in
bigfile-record remains in place.

Hope this helps.
Sal

This is certainly a trip back to yesteryear but I'll try and remember some of this stuff!

BTW, my background with COBOL is all IBM mainframe, so I don't know how applicable this is, but
hopefully it will be of some help.

A variable-length file has a short "word" attached at the front which defines the length of the record. On an
IBM mainframe it is called an RDW (Record Descriptor Word) and is 4 bytes in length. The first 2 bytes
are the length, and second two bytes are unused. The only way to access these bytes is to actually point
to a location *before* each record's beginning-of-data.
The only way to do this is to move each input record into a "table" (of one-byte "entries" so you can
access each record-byte individually), and then move "backwards" before the beginning of the "table"
(which is really one of your input records) by the number of bytes which make up your word length (on the
mf, this would be 4 bytes).


OCCURS is the only way to tell COBOL that you are defining a table. If you use OCCURS DEPENDING
ON it is a variable length table, otherwise it is fixed-length.

You can either INDEX this table or subscript it. If you INDEX you need to use the INDEXED BY clause with
the OCCURS. An INDEX is an actual pointer holding a binary number address, and this tends to make it
faster. It also allows for the use of the SET verb, for moving the INDEX (which is the table-entry pointer)
forward or backward in the table. Indexes are defined by the compiler, so you don't have to do this
yourself in the data division.
If you subscript you don't add anything to the OCCURS, but you have to have a numeric field defined
which you then use as an entry-occurrence-number (such as 1, or 2, or 254, or any integer which
corresponds to a table occurrence).
Subscripting can tend to be slower due to the data conversions, but some people prefer it for the strange
reason of it being easier to pick out of an abend dump. You can't use the SET verb with subscripting, and
you have to define the subscript field yourself.

The SET verb allows positioning in the table. In your case you SET an INDEX x DOWN by whatever
number is appropriate for your architecture (in this case the word-length) to move backwards. Then you
can read the length from the RDW. I think that when WRITEing output records you have to build in this
word length and set the length appropriately before issuing the WRITE, but I can't remember. It may be
that you have to so the same SET DOWN trick that you use on the READ.

Actually I'd be curious about your results and experiences here; you never know, I may need to
understand this process with MicroFocus myself one of these days...
