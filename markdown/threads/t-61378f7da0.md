```
Hello All,

I have an application that uses several ISAM files.  The application
is distributed on a couple of hundred customer machines and has been
in production for about 8 months.  

Changes are being made to the app that requires I add a couple fields
to a couple of the ISAM files.  Because of this, I will need to
convert the files to the new format.  The app is written in MF 4.0.32
WorkBench.  My first thought was to use their CBL_CHECK_FILE_EXIST
routine which says it returns the date the file was created.  Not
true.  It returns the date/time the file was last updated, not
necessarily the date created.  Since I distribute a null ISAM file
with the app, I know the date the null file was created.  If I attempt
to open the file in the app using the new FD, I get a FileStatus error
#39.  I can use this to determine when I need to rebuild the file for
this particular upgrade, but if I have changes in the future, I may
have a couple different formats out there and I won't be able to tell
which format the current file is using.  The change I'm putting in
doesn't have to affect all users.

I'd like to know if anyone has a better way of knowing which
version/format a particular ISAM file is in or a definitive way of
getting the file creation date.  

Any ideas would be greatly appreciated.

Thanks,
Doug Blaze
```

```
I have a system that has had a couple of updates of file structure.
You need to have a program, or a series of tiny programs, that
determine the structure and rebuild appropriately.  So, when
the program starts it (for example) calls a program that just
tries to open the file (or files) in the latest format.  If
this fails it passes it to a program that tries to open the 
previous format, failure (ie 39 status) goes to the even older.
When one of these finds it then it can reformat to the latest
format.

As the latest format is opened first there is no significant
overhead at the start of running the system.

You could also have a control file that maintains file versions
in a record (or records).  Just check this at start of run and
apply rebuilds appropriately.
```

```
The description of File Formats has a layout of the variable
length header used for MF files. This header is the first 128 bytes
of the data file. According to my manuals for WB 3.2, the file
creation date and time is at offset 8 with length 14 within that
header. The following program accesses that date.

       program-id. ism-date.
       environment division.
       input-output section.
       file-control.
           select ism-file
               assign to "ism.dat"
               organization is record sequential.
       data division.
       file section.
       fd ism-file.
       01 ism-record pic x(128).
       working-storage section.
       01 file-creation-date pic x(14).
       procedure division.
           open input ism-file
           read ism-file
           move ism-record (9:14) to file-creation-date
           close ism-file
           display file-creation-date
           stop run
           .

HTH
------------------
Rick Smith
```

```
Of course the creation date may not be especially useful.  If
a REBUILD was done to copy the data file as well as rebuild
the index then it will have the date of the rebuild and not
the original creation date of the initial file.

However, if the header contains the record size that is 
probably exactly what is required.  It seems to me though
that he is better having a number of FDs each being one
of the older formats.  When one open cleanly this file is
copied to the latest format and the files are then renamed
with the original being sent to a backup directory.

This is then independant of variations in manufacturers 
formats.  I still use LevelII format is some systems, while
it is slightly slower than 'current' formats it is a lot
easier to debug out in the field.
```
