You have to be careful with this. The version in an INT/GNT or LBR is the 
version of the compiler that compiled it. A compiler vendor doesn't change 
the compiler that builds the product unless there is a very good reason as it 
introduces a variable into the final product. ie. is any bug in a particular 
release a problem with the original program or the compiler that compiled it.

COBLIB.DLE/DLL isn't a bad place to look for the real version though.

Exactly, Mr. Biondi wanted to know the version of the compiler that produced
the application. So you look in the INTs, GNTs, and LBRs that were shipped
as part of the application.

> A compiler vendor doesn't change
>the compiler that builds the product unless there is a very good reason as
it
>introduces a variable into the final product. ie. is any bug in a
particular
>release a problem with the original program or the compiler that compiled
it.

With Micro Focus, the version (of the development system) used to produce
the "current" release (development system) is always a few builds behind.
For example, Micro Focus used 3.2.16 to build the 3.2.24 that I use. The
"very good reason" is that each release contains "bug" fixes and other
improvements.

>
>COBLIB.DLE/DLL isn't a bad place to look for the real version though.


But COBLIB.DLE doesn't give its version unless you run it, as in
RUN COBLIB.DLE. It is not likely that the client would have the RUN
command. Therefore, the next best place is the INTs and GNTs.
