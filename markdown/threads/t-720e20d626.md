It appears that you want to get the binary value 9 at the end of each
record?
If so try

01 REC-BUFFER.
03 REC-DATA PIC X(nn).
03 REC-TERMINATOR PIC 9 COMP.

MOVE 9 TO REC-TERMINATOR.
Move your record data to REC-DATA and then WRITE REC-BUFFER.

You could use a VALUE clause depending on where REC-BUFFER was
declared.

I had a similar problem with a vertical bar used to separate columns
on printout.

In my case, the program is written on a PC and then transmitted to an
IBM System 36 for compilation. The mechanism which translated the
ASCII to EBCDIC couldn't handle the |.

My fix:

01 WS-BAR-C PIC 9(04) COMP-4 VALUE 0079.
01 FILLER REDEFINES WS-BAR-C.
03 FILLER PIC X(01).
03 WS-BAR PIC X(01).

WS-BAR is the correct character, and no process changes it. Keep
in mind that the above is an EBCDIC value in decimal, and that
the COMP-4 on your platform may not be the same.

Kindrick

I'm not sure about Mocrosoft COBOL 2.20, but it is a repackaged
MicroFocus compiler. Using MF COBOL there is a runtime switch that
affects the processing of TAB characters. If set the switch causes tabs
to be expanded before the input record is put in the record area when
executing a READ statement. There is an other switch that tells the
runtime system to insert a x"00" byte before every low-order character,
when writing records. On reading such files the x"00" bytes are
removed.

Check your manual for the N switch. If you can't find the manual try
setting an environment variable "SET COBSW=-N"

Perhaps someone else that has the manuals may be able to help.

Charlie Goodman
Winnipeg
