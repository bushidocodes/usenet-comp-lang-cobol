Michael tried to make this point, but I'll try to explain in my words...

I believe, though I could be wrong, that the .NET framework contains the
runtime assemblies for all Microsoft .NET languages, such as C#, VB and
C++/CLI.

There are many other languages that compile on .NET.  MF Cobol is one. 
Others are Boo and Nemerle (which I believe are .NET only) as well as .NET
versions of many other languages (IronRuby, IronPython and many others).

I haven't verified this, but my guess is if you downloaded, say, a .EXE that
was created by the Boo compiler you would not be able to execute it without
also downloading the "Boo .NET runtime" (or whatever they might call it). 
This is because there will be classes that are specific to Boo, and supplied
with the "Boo runtime" (Boo.Lang.Runtime) rather than with .NET itself. 
While the Boo runtime classes may in fact just use regular .NET classes
under the covers (and most likely they do), the Boo runtime is still "in the
middle", you might say.

Hopefully I'm not too far off in stating all of this!  :-)

Neither here nor there, but it might be interesting for you do use the .NET
Reflector product to "disassemble" an EXE created by MF Cobol (for .NET). 
Look at it in C# (or VB or even ILASM if you're a masochist) and you can see
what classes etc. are being used.
