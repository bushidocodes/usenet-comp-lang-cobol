```
Exactly right.

However you define your stuff in one module (EXE or DLL) your stuff
must be defined in exactly the same way (same variable names,
everything) in the other modules. Each definition should be an '01'
level and include the word EXTERNAL. For example:

01  MYSTUFF EXTERNAL.
   02  MYSTUFF-TABLE OCCURS 50.
      05  MYSTUFF-STATE      PIC X(2).
      05  MYSTUFF-NAME       PIC X(30).

If you intend to use MYSTUFF in multiple procedures within a form, in
addition to EXTERNAL you should include GLOBAL. So, think of EXTERNAL
as a multi-module directive for GLOBAL.
```

```
First, you should define your tables in EVERY form that is going to use
them, EXACTLY the same way (a nice aproach is to write them as separate
modules and #INCLUDE them in the apropriate WORKING-STORAGE of those forms).
As you didn't state the version of PowerCobol, beware that with version 5,
you'll need to specify a Cobol85 compiled module instead of a OO-Cobol
module, or the EXTERNAL clause will give you an error when compiling. Two
more considerations:
a) Note the #INCLUDE statement instead of COPY, as the later is targeted for
the compiler and the first for the PreProcessor of PowerCobol.
b) Besides EXTERNAL, the data items must be defined as GLOBAL as in:
01        MyTable IS GLOBAL EXTERNAL.
    05    MyElement    OCCURS ...(whatever)
     10    FirstField        PIC ...
     10    SeconfField    PIC ...
```
