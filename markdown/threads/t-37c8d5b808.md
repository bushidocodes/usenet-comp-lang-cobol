I just thought I would start a new thread on what (to me) is one of the most
significant motivations in the addition of BOTH "OO" and "C++" (and C) type
features into the next COBOL Standard.  (I have said this before - but do
think it relates to a number of the threads that are going on currently in
CLC.)

The GOAL of the next revision is to make as MANY of those "facilities" that
are outside the control of COBOL *available* to COBOL applications - that
WANT to interface with them.

For example, as has been pointed out many (possibly most) of the OO concepts
such as "encapsulation" and "re-use" CAN be obtained in "traditional" COBOL
multi-module structured applications.  Therefore, if you are enhancing an
existing COBOL only application (or developing a new one), then it is true
that you probably can stick with the existing (old) technology - and still
have most of these benefits.  *IF* however, you are developing a new
application that will interface with:
   - "pre-packaged" object libraries
   - non-COBOL application programs
   - most ANSI/ISO or other non-standard APIs (including but not limited to
de facto standards like MS Windows)

Then, it is IMPERATIVE that your COBOL language be able to use the "building
blocks" that such languages/facilities require.  Now, I have heard that at
least some readers of this NG like "single language" applications. (Such
people can continue as they have in the past.)  HOWEVER, for those of us who
want to interface with new "tools," packages, languages, etc - it is
imperative that the COBOL language provide (via syntax) the tools to speak to
them and to understand what they are saying back to us.

Much of the "C-ification" in the draft Standard is intended to simplify (and
in some cases make possible) inter-language communication.  If you follow
most language "bindings" that are defined today, they are all written
assuming that you have the language facilities that C has. There is ALSO
"user demand" for expanding COBOL away from being limited to a "business
oriented" language.  There is NOTHING in the new Standard that says that
programmers MUST use these new features, but for those who DO want to do
"bit-twiddling" (for example) the facility has been added.

There is no doubt that some future programmers will write "bad" and/or
"inappropriate" code using the new features.  On the other hand, there have
been some programmers doing that already - so I just don't see that as a
reason for not providing the facilities to those who want to use them.

NOTE:
  There are also some new facilities that are intended to "extend" the
traditional areas of COBOL business programming.  These include (among many
others)
   - enhanced report writer
   - validate facility
   - file sharing/record locking
   - "common exception handling" (to enhance 24x7 applications - among
others)
