have a built-in way to detect file presence.  After all, the concept of
determining a file's existence is not platform specific, and you just have
to do that from time to time.  I don't understand why it's often left out.)

I also don't use the mouse in charcacter based apps.  If you're going to
use the mouse, you might as well use GUI in the first place. :-)  I have
found the Micro Focus Screen Section facilities to be sufficient for
anything I have needed to do (though sometimes barely :), in a straight
character interface.  Unfortunately, there are features there that are not
in the new standard.  And, of course, it doesn't offer all the features that
SP2 does.

My feeling is that a language such as COBOL is not complete if it does
not provide for a user interface, both GUI and character based.  If you
can't write at least a simple application using the language alone, it is
just not a complete language.  I have written a lot (and I mean *a lot*)
of 'utility' programs over the years, many of them in COBOL.  Many of
them input and output only a couple of fields to the screen as their sole
function.  Having the Screen Section makes this easer and cleaner in
COBOL than in most other languages.  The *whole* application is
visible, right there in one printable source file.  Removing the Screen
Section would eliminate this whole class of application from COBOL's
preview.  My gosh, if we're adding OO, a huge monster which *has not
been proven* to be a true improvement, by *objective* industry wide
measure, such as development and maintenance cost and speed, and
especially not in learning curve, no matter what advocates claim, on
which the COBOL community is still deeply divided, and which even
advocated admit, the vast majority of COBOL programmers will never
use anyway, why would we want to *narrow* COBOL's effectiveness
by removing something that can be demonstrated to be useful?  With
Screen Section you can, for example, write a quick a program to test
a new subroutine in five minutes.  What other way is there to do this
as quickly and easily?  A much more sensible and practical approach
(IMO), would be to *expand* the Screen Section to provide GUI
capability as well.  Sure, it would never offer the features (GUI or
character) provided by third party products such as SP2.  But simple,
ubiquitously available tools have their purpose too, as I've been trying
to point out.  Remember my earlier posts about "planting COBOL
seed?"  Why in the world would anyone buy a COBOL compiler for
hobbyist use for even $49, if they could not write just a simple program
that inputs and displays some data without buying another product,
that certainly *is not* going to cost $49?  Don't just think about your
main bread-and-butter applications right now.  Think broadly.  Think
long term.  :-)
