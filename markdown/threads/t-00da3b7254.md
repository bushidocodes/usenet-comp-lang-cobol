A brief warning to some who read this thread and think chopping CANCELs might
be a viable approach in all such poor performance situations.

Quite often the CANCEL is a reflection of a design decision that is intended to
refresh a number of variables in the called routine to their original values
specified in the VALUE clauses when the subroutine was compiled.

((In modern COBOL this can actually be achieved with the INITIAL keyword on the
PROGRAM-ID, which triggers data refresh without a program reload. But you need
access to the source of the submodule for this solution.))

It is especially true that when dealing with sourceless modules, an
experimenter will have no idea if it is safe to remove the CANCEL. Regrettably,
the removal may look safe in test environments (especially if small data files
are used in test), and produce harmful effects in production (where volume and
data variety is more vast).

The comments about performance improvements are right on, and the well phrased
indication of the human frustration in the cited episode is very reflective of
how costly this mysterious COBOL feature can be  But chopping CANCELs in the
absence of source code is a roulette game with a dangerous implement.

Incidentally, speaking of design considerations, it is not rare that a project
leaders decision to deploy CANCELs in drivers involves poorly designed
programs, and in some cases imperfect understanding of the programming
environment on the part of the original coders. 

So in cases where you do have access to the source, and do have the resources
and time (certainly scarce things these days), it is often better to get into
the submodule(s) and rectify their initialization logic, than to use the
brutish CANCELs.

The central point is that good initialization logic in the procedure division,
or mechanisms like INITIAL will reset all or minimum necessary subcomponents of
the data division, whereas CANCEL-and-call-again sequences from the driver will
reload the submodules executable code image repeatedly.  In most cases the code
reload involves disk access which is orders of magnitude slower than the data
memory refresh accesses.

It is also worth noting that the traffic that CANCEL-reload sequences place on
I/O devices and channels has major implications for other tasks in a large
computing environment. (Especially when the phenomenon is performed from within
loops as in the cited case). When an entire application is dominated by the
CANCEL solution, other concurrent apps can run sluggish indeed as their I/Os
wait for the thrashing to ease up.   Other concurrent apps definitely run
faster after the offending app is revised.

The CANCEL solution can be used for many reasons. It is common for it to be
favored by batch project team leaders and project managers who had their
technical hayday in online environments wherein things are dynamic for
operational reasons, and to whom static components seem irksome and even
malfunctioning.

At any rate, it is not utterly and entirely unreasonable to actually place an
unCANCELed sourceless submodule into production. It is just that you should not
bet the farm on it. And as a previous poster has effectively indicated,
removing the CANCELs (blindly or with open submodule code) will definitely
improve performance.

Just goes to show - poor design/coding practice used up front. I remember
one of the things in Elements of Programming Style by K&P - always
initialize constants at compile time if possible, but especially always
initialize variables at run time (except maybe for first-time flags and
even then it's better to not design your program this way in the first
place if at all possible). Ideally, all routines should be serially
reuseable at the very least; reentrant is even better.
