A "hashing algorithm" might do what you want.

 From Joel C. Ewing in bit.listserv.ibm-mail on March 16, 2001 (you 
can find it with google advanced newsgroup search):

"A trivial hashing function that works adequately in many cases is to 
take your data entry "key" field 4-bytes at a time, XOR these 
together, take absolute value, and then mod N' (i.e., remainder of 
division by N' - Constraining N' to a prime tends to improve the 
dispersion) and then use this..."

Of course, XOR'ing binary fields in COBOL might be a little bit 
difficult to do without a library routine or assembler subprogram.

If your function is truly trivial, you might just break up your "key" 
field into two-byte subfields, coerce each one to be a PIC S9(4) COMP 
field (I'm assuming IBM mainframe COBOL here), and simply add each 
number together, truncating the results.  The final two-byte binary 
number cannot possibly be decrypted back to the original value, and 
will give you approximately 32,767 semi-random results.

Perhaps someone else can suggest a simpler or more secure hashing 
algorithm.

I hope that helps.

Whether it can be decrypted or not depends on the tenacity of the 
hacker.  Typical hashing algorithms, such as MD5, need more than 15 
characters.

However, check out "ord" intrinsic function - it gives you the character 
code of the specified character.  Once you have a number, you can 
(within reason) do some sort of bit-twisting with it, then use the 
"char" function to make this new number into a character.  Keep in mind 
that with ASCII, your character code doesn't need to be higher than 256.

The output of one-way functions like MD5 can never be "decrypted",
because they lose information - that's what makes them one-way.
(Also, they can't be "decrypted" by definition, since one-way
functions aren't ciphers and don't encrypt anything.)

The only way to recover the preimage of a one-way function from its
image is with additional information.  For example, with a crypto-
graphic hash that uses the MD construction (ie a block compression
function created by running a fixed-size bit-mixing function in some
simple chaining mode, with zero-padding for short blocks), then *if*
you know that the preimage is a single byte, you can brute-force it.
But that knowledge constitutes extra information.  Given only an
arbitrary hash value, there's no way to distinguish any of the
infinite number of possible preimages.  Even if the preimage is known
to be, say, exactly M bits long, M>N (where N is the output size of
the hash), there are ideally 2**(M-N) candidate preimages.

That said, if the preimage is short, and there's any reasonable
decision procedure for validating it, it can always be reconstructed
by brute force.  Choice of hashing algorithm makes no difference; an
attacker will simply ignore the hash value and brute-force the
original.  It's just a matter of the available computing resources.

We need a better set of requirements from the OP to propose a
suitable solution.  Perhaps simply zeroing out all the sensitive
information isn't suitable, for example, because the distribution of
values has to be preserved.
