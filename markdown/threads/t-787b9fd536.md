I wouldn't read all the invoice lines in and then print. Just read the
invoice file one rec at a time, and do a control break on the customer
number, printing the heading portion for the new customer (and totals from
the last customer). If the customer has more than 13 items, do another
control break, and print the heading for page 2 of the invoice. As soon as
you establish that there will only be up to 13 lines per customer, the
users will ask for more. Think dynamic, and don't impose hard-coded limits
on your applications.

Here's one approach: a table of the detail lines.

01 LINE-TABLE.
05 LINE-1 PIC X(132).
05 LINE-2 PIC X(132).
...
05 LINE-13 PIC X(132).

Build the PRINT-LINEs as you read the input.
When it's time to print, perform a print paragraph with 13 WRITE
statements in it.

A related and perhaps more elegant solution is to use a table with 13
occurrences with an index.
01 LINE-TABLE.
03 LINE-X OCCURS 13 INDEXED BY PRINT-IDX.
05 PRINT-LINE-CUSTOMER PIC X(30).
05 PRINT-LINE-AMOUNT PIC $$$,$$$,$$9.99.
...

Also build the print lines as you read.
Then you perform PRINT-PARA VARYING PRINT-IDX FROM 1 TO 13, with a
statement like
WRITE PRINT-FILE FROM LINE-X(PRINT-IDX) inside the PARA.

Or MOVE LINE-X(PRINT-IDX) TO FD-PRINT-LINE and then
the approp WRITE FD-PRINT-LINE AFTER ADVANCING, etc. statement if
you're going directly to print instead of to a flat file.

I'm not sure what you mean by "all at one time."

Most people want an itemized bill, not a lump-sum bill (unless
you are writing for a hospital).

Typically, you save off all the invoice-wide information at the
beginning--Customer Number, Name, Address, Vendor Name,
Address, etc. You print all the junk which goes at the top, print
the first item, and add to totals accumulators.

Then you read and print each following item until the Customer
Number changes.

Depending on circumstances, you may need to keep count of
how many lines you have printed so you will know how many
blank lines to skip to get to the bottom of the invoice.

When the Customer Number changes, you print the totals, clear
the totals, and add at the bottom any more needed junk (which
you saved at the beginning); then you save off the invoice-wide
info again, print a new invoice header, print the first item.
Et cetera.

If you are a good programmer you will also want to provide for
the instance where an invoice has too many items to go on one page;
the instance where an item description takes more than one
line to write; the instance where the Customer Number in the
input file is not a valid customer number; the instance where the
program runs fine but the printer screws up the printout.
Et cetera.
