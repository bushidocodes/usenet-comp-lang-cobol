If I understand your question correctly:

1) If data in an outermost program is defined WITHOUT the GLOBAL attribute,
then it is not "automatically" visible to a nested program.

2) Data without the GLOBAL attribute may be passed (by reference or by
content) to a nested program - just as  it is to a separately compiled
program.

3) Data WITH the GLOBAL attribute is automatically visible to all nested
(within that program) program.

    ***

P.S.  I am not familiar with the term "common-storage section".  Is this a
(Unisys?) extension?  If so, I have no PERSONAL knowledge of how it is
handled or what its "scope" is.  (Possibly Chuck or another Unisys person
will help out.)

P.P.S.  And before someone asks, EXTERNAL is totally separate from GLOBAL.
A data item, may be Global, External, neither, or both.

It's not standard, and it's not in any MCP/AS dialect of COBOL. that I know
of.  Unisys MCP/AS COBOL85 does have the $COMMON Compiler Control Option,
which turns all areas under its control into COMMON data items.  This is

intended for (and functional, so far as I know, only in) the bound
subprogram environment.   Is that perhaps what was meant?

> That would be a shame if it's an extension - but maybe a nice suggestion

One of the reasons it took nine years to produce the 2002 standard (dated
from the publication of the Corrections Amendment in 1993) (and seventeen
years from the publication of the base standard itself) was that so many new
"gee, wouldn't it be great if ..." pet features kept creeping into the
draft, along with all the "I won't vote to adopt this standard unless it has
..." pet features.

Both committees recognize the danger of "feature creep".   In order for the
first draft of the next standard to be in shape for beginning the
international ballot process by December 2004, which is pretty much a
requirement for getting the standard out in 2008, WG4 has asked that the new
feature content of that draft be limited to that list that was established
at their June meeting, and J4 wishes to conform to that request.  That
feature content "wish list" from WG4 for the 2008 standard has been
published (see document 03-0175 on the J4 web site,
http://www.cobolportal.com/j4/ , resolutions 2 and 3) , and revisions will
take convincing the National Standards Bodies of several countries of the
need for these features in this standard.  J4 continues to maintain a "wish
list" for upcoming revisions to the standard, but at this point anything
that goes on that list will be considered for  the revision *after* the one
currently under development.

I'm not sure what "COMMON-STORAGE" provides that the GLOBAL in the data
declaration entry, or for that matter $GLOBAL in the Unisys MCP/AS extension
to ANSI-85 COBOL, does not.

OK, Thanks for that. I am now better informed <G>.

Have you considered a separate layer for RDB access?

Your general "facilities" (like Date) would simply call out, passing a
single communication block  with a data  and control buffer (the same
concepts as the one in the Main program) but the DB connection and sharing
would be in a module that ONLY deals with DB.(Communication is through the
shared buffer, by reference).

There are a number of benefits to using an n-tiered structure like this.

It means you can have somebody writing the DB access stuff (maybe who's had
a fair bit of DB experience (?)) while someone else is writing the Business
logic.

Separating DB access from Business logic means that both of them can focus
on what they specifically need to achieve and changes to one are less likely
to impact the other.

Suppose you move to a different RDB system? (unlikely in your environment,
but not impossible...). N-tier design really facilitates the move.

Although embedded SQL for data access is probably OK in the Business layer,
when it comes to updating, n-tier is much better. Given the use of common
store by Unisys and access to the sub-schema I would definitely move this to
a "specialist" module.

I haven't tested any of this but I'll tell you what I think should happen.
Data in the common-storage section is defined to exist once for the entire
run-unit, that would include nested subprograms. However, the nested subs
have no way of knowing the structure of common-storage without either a
local definition (i.e. including the subschema section) or getting something
with GLOBAL on it. I'm not sure how you could get that on the DMS
definitions. It seems that the last time I tried a REPLACE on DMS data it
did not work. There is always the WORKING or LINKAGE options but they
typically have their own downsides.

Having read a couple of your other posts, I would try to keep the RDMCA and
program specific entities out of common-storage. Let the DMS stuff live
there by itself. I've seen too much grief doing it any other way. The RDMCA
definition (and whatever else you need) are easily accommodated by putting
the code in a proc and including GLOBAL in its definition.

You commented that you liked the idea of the common-storage section. Having
used it (and seen it misused) for a long time, I prefer using EXTERNAL
instead. Gives the same effect as common-storage (particularly when an item
is declared EXTERNAL and GLOBAL) but has the added benefit of being able to
define multiple "common-storage" sections.

Should work fine. Just remember the difference in visibility that the two
different clauses produce. GLOBAL only affects the program(s) in which it
appears. EXTERNAL makes item available to the whole run-unit. Whether you
only need one or both of these clauses I'm not sure. I don't think you have
the same issues with declaring RDMCA that you do with DMCA, i.e. RDMCA can
be local to any program. The same fields do not have to be used for all
access because it only contains status info. With that being the case, a
simple proc with neither GLOBAL nor EXTERNAL may be sufficient. It becomes a
matter of how you want to handle that data.

I will warn you there a few other subtleties in a multi-LDM program (DMS and
RDMS here). One that I particularly like is if a SQL statement forces a
thread rollback, UDS will also automatically rollback the DMS updates
(reasonable) as well as depart you from the schema so the next DMS command
you execute will get a E$21 (not imparted). Of course, if you have
bullet-proof error handling, not a problem<g>.
