While your statement is - rightly - careful to qualify you were testing for
the amount of 'interpreted'  vs. 'machine' code in the two compilers'
outputs, I am generally quite chary about trying to compare compilers for
*performance* when the two compilers' use different source languages. As
building the software which runs these comparisons is subjective, based on
the skill of the author, you never end up comparing apples for apples in
these cases.

That is, I hope no one infers that one source language may be generally
considered 'superior' to another for performance, generated code size or
other measure, as there is no such thing as a fair test.

Comparing "brand-X COBOL" to "brand-Y COBOL"  or "brand-X C" to "brand-Y C",
otoh, is fair.

I agree, Michael.  Certainly, languages should not be judged on such a
narrow example.  But my purpose was to see how efficient the MF
COBOL generated code was, using a known efficient C as a benchmark.

What I did was to code a very CPU intensive algorithm, the Sieve of
Erasthones (prime number generator, for those not familiar), in both
languages, using logic as close to identical as possible.  The MF
COBOL program was somewhat slower, but by less than one order
of magnitude (around 2:1 to 4:1, IIRC).  Knowing the C compiler as
well as I do, for such a compute intensive algorithm, it seems a bit
unlikely that a MF Virtual Machine would be quite that efficient.  If
it is, I applaud them.

If anyone is interested, I'm looking for the COBOL program.  Haven't
found it yet, but it may be on a backup.  Can't understand where it went,
I'm a packrat for such things, and keep them pretty well organized.  You
can download the C program (primes.zip) from my website, though
that program is dressed up a bit from the benchmark version.  It can
generate every prime number < 999,999,999 in a few seconds.
