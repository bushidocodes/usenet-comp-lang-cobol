To all.

A while back I posted a question to the group regarding a strange problem I
was having using the Fujitsu PowerCobol 6.1 "CBL_CHECK_FILE_EXIST" on a
Windows NT 4.0 server. I never got a chance to respond back to the group
regarding the status of this issue, due to way too much work and so little
posting time.  As it turns out, with many thanks to Thane for his
"Shellexecute" routine that he sent me, the problem seems to be resolved.
But I want to let everyone know that it was NOT a problem with the "file
exist" routine, nor with the Fujitsu code, per se.  Nor did his routine
provide an immediate fix.  However......

I was using the DDE control to execute the shell routine to update our
database and execute a FTP to another server.  When I changed the routine to
use Thane's "Shellexecute" code (which gives a little better diagnostic
return codes), I didn't get much different results but  I discovered
something new about the NT "cmd.exe" shell (read DOS prompt for Windows
95/98 users).  When I ran Thane's code on my Windows 98 development machine,
the routine worked every time, even though it didn't always work with the
DDE routine.  So the "Shellexecute" routine worked more consistently.  But
when I ran it in the NT server, it failed about as much as it did with the
DDE code.  The return code always seemed to contain the same value on every
shell out to the database update routine.  I communicated this back to Thane
and he said to not put too much into the return code and pointed me to an
article on the MSDN site.  After reading it I agreed with him, but it got me
to thinking that something just doesn't seem right here, and after doing
some web research, I found that the NT "cmd.exe" shell actually has the
capability of using buffers and the default is 4.  So just for grins, I
changed the buffer to 1 and the problem seemed to disappear with the
database update portion. Don't know why, but it just seemed to work.

That left the FTP part still not working.  I was using a "freeware" FTP
scripter named FTPBATCH, which always worked fine on my 98 machine (a slower
machine than the NT server) but on the NT server, I noticed that the routine
would take about 45 seconds to go through the FTP login, password,
parameters and the put file routine, and it would never close out the "cmd"
window nor would it complete the FTP.  So I changed the Fujitsu COBOL
program to actually write out a FTP script file, and using Thane's
"Shellexecute" code, actually called the Windows NT FTP executable with
the -s:<script name> command line parameter and it works perfectly every
time.

My many thanks to Thane for all his help in this matter, and hopefully I can
return the favor some day.  Now I know why I'm basically a mainframe guy and
have so much gray hair!!!

Denny
