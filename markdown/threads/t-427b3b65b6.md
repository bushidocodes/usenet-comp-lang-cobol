You are not clear with your question.

"file is in use"

Is this 'file' a shared indexed file, or a line sequential file or
some other ?

Is that:
 a (non Power-Cobol) program?

Or:
 a (non-Power) Cobol program?

In the case of a shared indexed or relative file you probably can't
detect whether a file is being used by another Fujitsu Cobol program,
whether that is Power Cobol or not. You also probably can't tell
whether another program, in any language, is using it via the file
access callable routines.

In the case of non-shared files such as LINE SEQUENTIAL you will most
likely fail to open it but will not be able to determine whether the
other program is Cobol, Power Cobol or some other.

You also don't indicate the purpose of the check. If is it because you
don't want to use it while another program is using it then try
opening it exclusive. It will fail if another is using it.

Instead of clock, use  change notification objects to trgger your 
processing.  This flag:

      FILE_NOTIFY_CHANGE_LAST_WRITE Any change to the last write-time of 
files in the watched directory or subtree causes a change notification wait 
operation to return. The operating system detects a change to the last 
write-time only when the file is written to the disk. For operating systems 
that use extensive caching, detection occurs only when the cache is 
sufficiently flushed.


... should handle it.

When I've used these before, adding  new file to a folder results in 
multiple notifications, the trick is to use this one, since the others may 
occur before the file is completely copied or written.

Of course, you could always use your size/timestamp technique  AFTER you 
have received this notification...

  WaitForSingleObject
  GetSize/Timestamp of 'file of interest'
  DO
      WAIT some reasonable amount of time
     Check size/TimeStamp again. If no change it must be 'done'
END DO
