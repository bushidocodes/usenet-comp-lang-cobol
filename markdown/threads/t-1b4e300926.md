This is true. I used the HP PCL escape sequences to control the
font and positioning of text via COBOL under DOS for many years
and it worked well.

Unfortunately, Windows screws everything up when you try to do
the same thing.

Windows does not appear to like the escape character (027) and
seems to remove it or something. (Having the 027 as the first
character in a print line also seems not to work. I had to
put leading spaces before the 027 to get it to work.)

Just yesterday, I found that an ASCII 028 in a text string
cannot be moved from one field to another. The 028 apparently
terminates the move.

So, it seems that Windows just doesn't want to cooperate if you
use those other ASCII characters.
