Thanks for looking at the source code.  You are corectly identifiying WHY I 
created these tests - and I am (sort-of) glad that all the tests are coming 
out the same.

Yes, Virginia (to borrow from another thread),
  the LOCAL item is being ignored in teh nested program.

I have been working with PL22.4 (previously J4) on a question that 
originated with Report Writer - Source and Sum clauses and 
Accept/Display/Screen Section - FROM, TO, and USING clauses.  It turns out 
that this is a general GLOBAL/local issue.

What it appears that the Standard requires ('85 and '02) and that 
implementors all did this correctly, is that

- When there is an explicit reference to a data item (normally in the 
Procedure Division, but this ALSO can be in a data or environment division 
statement, such as Occurs Dependin on,  FROM, SUM,  CURSOR IS, RELATIVE KEY 
is, etc0, that the rules for name "scoping" occur, i.e. you look for a local 
name that matches and if you don't find one you go up the chain of 
containing program to find a GLOBAL item that matches.

HOWEVER, when there is an implicit reference to an item, such as a reference 
to a group item with the GLOBAL attribute and taht group contains an ODO, 
then it is the "originally identified in the original data division" item 
that is used, and NOT any local item with the same name.

This is (or may be) a "little surprising" but it turns out there are rules 
to support this and (happily) all compilers tested seem to work this way.

In some (not all) cases, there are rules that state that if the "group-type" 
item is GLOBAL that the "referenced" item must also be global.  (This is 
true for Occurs Depending ON), but in some cases such rules do not exist and 
the implicitly referenced item may be used in an "implicit reference" (in a 
contained program) but the same item may not be referenced in an explicit 
statement in that same nested program.  This is "weird but true"
