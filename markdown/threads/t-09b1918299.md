I can't add much more information than Rex Widmer provided on Panvalet,
except that it was supported in the IBM DOS/VS and VSE environments. I first
encountered it back in 1979! And the first CICS full-screen editor I ever
used was PANE for Panvalet Online.

Even though we now use Endevor for most production source control management
under MVS/ESA, we are still using Panvalet for many thousands of test source
files.

One additional feature of Panvalet which we found quite handy was the ability
to store members with 10 character names. Normal IBM libraries only allow
8-character member names.

One problem we had with Panvalet was the liberal use of ++INCLUDE by
developers (like COPY but allowed nesting). Most developers don't seem to
understand how ++INCLUDE really works. The typical compile JCL expanded the
source code into the reader queue using ++INCLUDE. Any nested ++INCLUDE's
were also expanded at submit time. But all members INCLUDE'd had to be in
the same Panvalet library. This tended to encourage programmers to "forget"
to promote copybooks or included members into production libraries.

Needless to say, it's not good discipline to have production code running
with test copybooks. This created problems when trying to find and analyze
code for our Year 2000 project.

Panvalet is a very long lived program product that runs (or ran) on IBM
mainframe computers using the PCP, MFT, MFT-II, MVT, SVS, VS-2, MVS,
MVS-XA and MVS-ESA operating systems. (might have been a DOS / VSE version
as well) It has been marketed since the late 60's / early 70's first by
it's originator - Pansophic Systems, Inc, and currently by Computer
Associates.

The system provides a method of source storage, control, and retreival for
card-image type data. It also provided for a language independant COPY
facility - where a source member could reference another source member,
etc to some level of nesting.

This system sold many thousands of copies (as did it's original competitor
- Librarian then by ADR, now also by CA) to most mainframe shops.

Both of these systems allowed many thousands of source modules to be
stored in a single disk dataset. Key features were disk dataset self
compressed (unlike PDS), multiple concurrent updates were handled well,
source code was stored with simple compression (i.e. cobol keywords,
repeating blanks).

An ISPF editor interface was also sold and commonly used.

Source programs could be secured (production status) from update,
interfaces to common security systems (ACF, RACF etc) were provided for
additional security.

This product set is a programming tool, learning the tool is quick,
learning how it is deployed and utilized in a given installation's
environment is usually the bigger task.

Rex Widmer
Builder of software archeology tools and other strange programs to help
survive in a legacy based world.
