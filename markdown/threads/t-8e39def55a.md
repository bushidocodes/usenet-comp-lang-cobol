Here's why. I'm using a Cyrix/IBM 6x86 P200 MMX with 32 meg of 60 ns memory,

fonts and background assists.

Moore was looking at a tiny sliver of one of the technologies that supports
computing... that as densities increased in a linear dimension on a solid state
substrate, that circuit densities increased as a square of that because
substrates are two dimensional. He observed that the linear increase was enough
to result in a doubling every 18 months, roughly.

Being pretty much out of it as far as computers go, he couldn't anticipate how
the industry would use his efforts. Now we have 3D spreadsheets that can link
out to other spreadsheets (a 4th D), proportional colored fonts (a 5th D),
hidden assists (a 6th D), simultaneous updates of graphics (a 7th D)...

So each time the solidstate engineers came up with a doubling of power, it had

First off, I'm enjoying the humor being shared here on the posts...
entertaining (to my warped senses, no doubt....)
BUT... I think it's time for a checkpoint on where we're going....

1- if we're saying hardware power precedes software planned usage -
then my comment is that this was once true. the ibm 360 had an
instruction set that was not considered for application languages and
has been a source of constant problems since.... e.g., the 0C7 abend
was never needed -- a result of hardware over software. in an earlier
life as a burroughs systems analyst, the hardware and software people
worked together so customer didn't see the difference (I'm talking mid
1960's here...) in summary, items such as 0C7 abends are due to a
flaw in hardware/software coordination....(my opinion) the error it
indicates never required a program termination...
2- if we're saying software bloat precedes hardware growth, let's
remember the growth of IMS and DB2 -- capable of devouring mainframes
in the mid 1970's (to mid 1980's...) because they were capable of
application issues that overpowered mainframe power at the time. we
can also look at microsoft word -- I registered my copy in 1985 -- and
it wasn't a popular wordprocessor at the time because it took time to
create its beautiful pages of text -- but it helped drive expansion of
computer power....
3- the ibm term 'vscr' (for Virtual Storage Constraint Relief) emerged
because MVS with 16 meg of memory was being used by ibm software (mvs,
ims, cics, etc....) and little of the memory was available for
application software. migrating to cobol II was significant in that
it allowed applications to use otherwise unavailable memory. this was
a case where software overwhelmed hardware and changes to both
hardware and software benefitted all of us.

I'm reminded of a story about two military manufacturers.... one made
tanks, one made guns. the tank manufacturer tried to make tanks that
repelled the shells from the guns - the gun manufacturer tried to make
shells to penetrate the tanks... by working together, both improved
their products.... such it is with hardware and software..... when
intel introduced MMX features, it allowed software designers to use an
expanded instruction set. this is an iterative process.... has been
and always will be, especially if hardware and software people talk to
each other...

I was pleased to see I'm not the only one who kept a copy of Wordstar
3.3 - a great wordprocessor. Of course, it had no spell checker, no
options for different fonts, no thesaurus, no option to include
graphics, no options for columns of text, no ability to include
documents such as spreadsheets from other software, no option for
outline mode to document mode and vice-versa.... but I agree -- it was
great for capturing text -- it's preview mode that scrolled screens
automatically while the author reread it is still something I haven't
seen offered... was a nice feature.... AND... the ability to
customize every facet of the program, including how quickly the
program responded, is a feature we'll never see again.

In summary, I believe that software (i.e., the needs and desires of
the customer) will always drive hardware. that's no crime. if a
hardware feature can be visualised, it's use will be programmed even
when not available; if a hardware feature was not visualised (such as
satellite mapping from gulf war...) it will be added to customer base
as soon as practical.

thanks for listening....
