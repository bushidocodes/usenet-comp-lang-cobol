I'll give you ea break:

1st - you need to know how S999V99 is stored in internal memory

2nd - you need to ensure that a conforming bit pattern is entered on the
screen to ACCEPT it unconverted into the variable
OR
2nd - you need to accept another format (f.e. a more or less user friendly
format that allows for signs and thouands separators (i.e. commas) anf
decimal points (dot) a.s.o) and convert that to S999V99. This needs checks
for valid data of course.

Why step 1?
Because to understand the need for choosing the 2nd 2nd option mentioned
above.

Why choose the convert step and not the direct step as 2nd?
Because that is really user friendly and as such the only ACCEPTable way to
design an interface. And because that method is the one that checks for
errors.

On most platforms the sign of a S9(n) field is stored along with one of the
digits in one machine position (byte). The resulting bit pattern of that
combined digit and sign should be entered when it is unconvertedly moved
(ACCEPTed) into the variable and thus hope that the wanted value for that
digit plus sign as a one-character bit pattern is available on your
keyboard. In EBCDIC for example the characters A thru I are availabel for
+1 thru +9 and J thru R for their negative counterparts. A 0 (zero digit)
with + of - sign is then a { or } respectively. Up till now I never saw a
user document that correctly, understandably and user friendly explained
these matters to a non-computer-technics-trained end-user.

So go for option 2 and desig some program to ACCEPT anPICxxxxx field long
enough to contain a lot of digits and character dots, commas and +/-
characters. Let that program allow and check for a valid human readable
numeric value give some nice error texts when not valid and convert to
internally stored numeric when right. That kind of code has been made many
times by others, but is a great exercise for you.

Perhaps if you ask him kindly, comp.lang.cobol contributor Judson McClendon
will eventually give you such code. I believe he has some reusable stuff
for that, if I recall my memories well.

Good programming!

The Frog

P.S: COMMON-STORAGE? That's non-standard. What compiler do you use?

Simon,
  I really do NOT mean to discourage you from using this newsgroup. You have
shown what you coded and explained your confusion.  What I would suggest is
that you check DejaNews.  This very question is asked (and answered) about
once a month.  I think the latest answer was about a week ago.

Bottom-Line:

   ACCEPT input and DISPLAY output fields are almost alphanumeric or
possibly numeric-edited.  They are almost never "numeric"

  You need to "evaluate" the field that comes in to determine what sign
information is included and what space padding exists at the beginning and
end of the input.  (Periods are always "explicit"  PIC "." and not "implicit
PIC "v".

Do check your input for the possibility of non-numeric data.
