It's my recollection that STRING had some problems in OS/VS COBOL under
MVS.  The target field was not space filled when it was larger than the
sending fields.  This may not have been a problem with STRING, it may
have been a problem with programmers understanding how STRING worked. 
So many installations simply banned it because it created more problems
than it solved. 

01  RESULT        pic x(10).

STRING '123' DELIMITED BY SIZE
       '456' DELIMITED BY SIZE
       INTO RESULT.

RESULT then contained '123456' followed by 4 undetermined characters
(whatever happenned to be in those storage locations), NOT spaces as
many people might think.  STRING is not a substitute for MOVE, and the
proper breakdown of group items into elementary items in
working-storage.

Bob

03  NAME-ADDR-LINE-1            PIC X(36).

If I used MOVE for city, state and zip, it would look like this:

Opp                 AL  352153543      
Birmingham          AL  352150000    (sometimes the last 4 digits come

I use STRING so it will look like this:

Opp, AL  35215-3543      
Birmingham, AL  35215

I don't UNSTRING this into a detail print line, I STRING it into a

detail print line.  And yes, all this for a variable length City name

In article <36464CFC.3FECF140@att.net>, William Lynch <wblynch@att.net> writes:

>I don't know when it became legal to use in

In VSE/ESA, STRING and UNSTRING became legal with COBOL for VSE, which uses the
Language Environment. It became legal because the Language Environment routines
for CICS use a different way of allocating memory that is CICS-friendly.

The STRING and UNSTRING verbs were illegal in CICS for FCOBOL (DOS/VS COBOL)
and COBOL II because the runtime libraries did GETVIS (the MVS equivalent would
be GETMAIN, not the EXEC CICS GETMAIN, but an operating system GETMAIN) to
acquire storage for a work area and, since the program never ended, the storage
would be held on in case it was needed later. But during a program compression,
the program (and its pointer to the GETVISed memory) would be thrown out, and
the next time the program was executed, it did not have any pointers to that
GETVISed area, so under even moderate program compression rate, GETVIS storage
would eventually be eaten up.

One of our vendors did this and the way we eliminated the memory creap was by
making the faulty programs resident.

Mark A. Young

Yes, COBOL for VSE solves the problem. More precisely, the Language Environment
(which COBOL for VSE uses) solves the problem by using memory management
routines that are CICS-friendly. Actually, two separate sublibraries are used
for LE (SCEEBASE for common and VSE-specific routines; SCEECICS for
CICS-specific routines), which is a change from previous COBOL libraries where
one library was used for both batch and CICS and thus library routines that
needed memory would do so in a VSE-compatible but CICS-hostile manner.

Also, it apprears that not all STRING nor UNSTRING routines are equal; some
constructs require the use of routines that end up needing to acquire temporary
memory for their work, and some don't. Sorry, I don't have the details, but I
recall seeing it mentioned on the VSE-L mailing list.
