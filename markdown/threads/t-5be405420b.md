```
IF the reason for splitting the file is media capacity, DONT split the file.
Write output until the output media fills, then switch to new media. This
technique has two advantages:

1. For some reason records may be deleted in the future shrinking the output
to fit on only one tape (disk, diskette, etc.).

2. You'll be set when the file expands to fill THREE output devices.
```

```
Take a look at the DFSORT/ICETOOL job in the last post of this thread:

http://www.mainframeforum.com/t625752.html

It shows how to split an input file as evenly as possible into four 
output files with contiguous records (e.g. 1-100, 101-200, etc)using the 
SPLITBY=n parameter of OUTFIL.  You could do the same kind of thing for 
two parts.  It does requires a COPY pass to get the count of the 
records, but DFSORT copy is very fast.

By the way, if you can determine the total number of records without 
reading them, you could just as easily set up a DFSORT job to split the 
files (using SPLITBY=n or STARTREC/ENDREC) as an IDCAMS job.
```

```
There is a way go obtain the file size on a file that permits random reads
and has no deleted records. Use 'binary search' type logic, starting with
a value greater than the possible last record. Consider successful reads
as a 'low' and invalid key reads as 'high'. When you get to the 'not found'
point of the search, you are at the last record or last record +1, depending
on whether the last read was 'low' or 'high', respectively.

The number of reads should be int(log2(first-read-nbr)) +1.
```
