```
I am wrapping up the development and testing of a new batch job I just 
wrote.  About 15,000 lines of code.  After finishing the job, I realized
that most of the code I left uncommented.

Now, I usually write a moderate amount of comments in my programs, but I
simply don't have time to go back and spend time writing comments and doing
library migrations.  I feel, IMHO, that my time is better spent doing
additional testing.

Here is basically what I did:
1) All programs have at least 2 or 3 sentences in the beginning of the
program describing overall functionality.

2) I spent enough time coming up with good paragraph and variable naming.

3) Tricky sections, like when I merge 2 two dimensional tables, I write
inline comments in the code, like this:

***--->>> INIT TABLE.
    XHGUIGHREIGUERHGURIG.
    GJIOGRJGOIREGJEROGPJGE.

***--->>> READ EACH RECORD IN THE CURRENT BLOCK.
    IOJGOIERGJROGREJGOGOPE
    JIOGJOIGJEROIG ( GJOIGRJEGOIRJG  )
        GJERIOGREJGOEIGJEROIGREJGOG TO JIGJREIGJRGERIG.

***--->>> AND CHECK TO SEE IF IT IS ELIGIBLE.
        IF GREGJREOIGG = GREJGIOREGJROIGRJO
            JGIROEJGOEIRGJOGJ
         ELSE
             OIGRJGOIRJGOIERG
        END-IF

***--->>> IF IT WAS ELIGIBLE, BUILD THE KEY ON THE NEW TABLE.
        JOIRJGORGJOGERJO
            JIOGJROGIREJGOEIRJ
            JOIJOIJOGIREJOIGERJG

***--->>> AND ROLL THE DATA OVER FROM THE CORRESPONDING
***--->>> WORK TABLE.
            GRGRGGREKGPOGK
            GREGKERPOGREKGPO TO GRGRGREGER (GRGG , GRERGR )
        END-IF

***--->>> KEEP GOING UNTIL OUT OF RECORDS IN BLOCK.
    END-OIPOGRKGJPOGKER.


My point is, that for the most part, I kept comments to a minimum, only to
describe tough sections.  I think there are two main problems with comments:

1) They become out of date.
2) They become inaccurate.

I would like to know what your opinions on this are.
How many comments do you normally write?  What are your feelings?  Remember,
in the business world, time is money, and I don't feel like spending more
than my usual 9-10 hours at work every day.

Should I go back and comment the code?  Isn't testing the consumer's product
a better idea?

Jeff
```

```
Comments are falling out of use because few people use them intelligently.

In COBOL, comments should NEVER tell you WHAT the code is doing, it should tell
you WHY the code is doing it.

Here is an example of a BAD Cobol comment:

* Reject all type "Q" transactions and set
* message flag "F" to true.

Here is an example of a GOOD comment:

* A type "Q" item is now blocked
* because that service is no longer sold
* to PDQ customers.  We continue with 
* the transaction to process item types "P"
* and "D"; 
* the "F" flag gets set to trigger a
* "Sorry, Service Q NLA" message on the
* customer reply./NAM 06/17/00

Ideally, I like to paste in the Requirements/Func Specs and then code between
them.
Of course, often the Specs aren't logical enough to do that!

In Assembler or TAL or "C" I also like to see comments that say things like:

! I'm doing a bit shift here instead of 
! division

Because there one operation can be used for different purposes.
```

```
Hi,

I don't think you can really have 'bad' comments, in one sense, the second
set of 'good' comment could be out of date in 2 years (and with COBOL, 2
years is nothing!).

I would prefer comments like;

* Build the Queue table based on the input from the previous section.
* This new table holds values for History Processing based on XYZ standards

And then go for it (Obviously the above is whacked up in 4 seconds) But one
thing to avoid is HUMOUR! I personally have seen COBOL programs that have
things like

* This bit will never be called. If it is, I will buy beer and Pizzas for
the team..

Funny at the time, depressing at 4a.m.

So, I think you should write your comments FUNCTION specific and not
CUSTOMER/ENVIRONMENT/POLITICS specific - if the function remains the same,
it describes the function. If the customer changes or the envionment changes
or the politics change then your code is wrong.

But, what do I know, I've only been doing COBOL for 16 years now   8-)

Andy
```
