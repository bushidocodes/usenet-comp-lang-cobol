```
I encountered a curious situation tonight and wondered if any other
PowerCOBOL users have come across it or can shed some light on how they got
round it.

I don't normally use debugging lines in COBOL ('D' in the 'indicator area'
[column 7]) but I'm writing components that have to run on a Web Server and
this means that the standard debugging tools (similar to Animator) can't
work. The reasons for this are too lengthy to go into but I have been unable
to successfully get COBOL to go into debug when it is running on IIS.

It isn't a problem because I can write and debug the component before I put
it on the server but this is not 100% as the changed environment COULD cause
a problem. On the server there could be multiple instances and multiple
threads and it might be useful to have some "thread specific" debugging...
(So far, this hasn't been a problem, but I always expect the worst..<G>).
Sometimes I don't want to do a separate development, so I write something
and install it as an ActiveX (COM+) component on the server. Very
occasionally, it doesn't function exactly as expected so some debugging is
called for...

I found (almost by chance) that there is a Fujitsu option to display upon
SYSERR. Perfect! I can out the contents of a field when I need to, and the
server writes it to a log in
WINDOWS\system32\whateverIhavecalledthelogfile.txt. This is adequate to my
needs as I'm not doing extensive debugging.

So I slipped a few of these displays into the code and found the problems
(mainly caused by confusion on my part between local and factory storage
under the covers of Power COBOL.) Good stuff.

Then I thought it is kind of a pity to take out the displays in case I need
them again in the future. Aha! Why not make them "Debugging lines" and just
"switch 'em on" with a single re-compile if they are ever needed? (Ideally,
I'd rather have this controlled by a run time switch, but I can live with a
single compile if it is needed.)

So, off I go sticking Ds in col 7... It compiles without error and I think,
OK let's set up the WITH DEBUGGING MODE clause in the source-computer
paragraph of the Configuration Section.

OOps! PowerCOBOL does NOT allow a CONFIGURATION SECTION in a component!

Yet it DOES allow "D" lines. (You just can't turn them on or off...<G> There
is probably some inscrutable Zen type wisdom behind this, and no doubt some
little Jap programmer is reading this and laughing himself silly at my
discomfiture. Hope he chokes on his saki...<G>)

BOTTOM LINE: Has anybody found a simple way to get round this problem or am
I just SOL as far as debugging lines are concerned in PowerCOBOL?

All help appreciated.

Pete.
```

It is in the external connection information used by the component. Fujitsu
call this the "environment file". It is always called: COBOL85.CBR

The contents on this occasion are as follows:

@EnvSetWindow=UNUSE
@MessOutFile=WBRlog01.txt
@WinCloseMSG=OFF
@ODBC_Inf=odbc.inf

The "MessOutFile" environment variable controls the SYSERR output. Although
the COBOL85.CBR file is in the directory on the server where the Component
is activated, the Component itself is registered in a directory that is not
available to the Web Server. (Component Services under IIS allows it to be
invoked using Remote Procedure Calling (which is exactly what I want, as
eventually this component will be invoked from a number of remote
locations). I think that's why it opts for the  WINDOWS/system32 directory
to write the log. (The component is not registered in this directory,
although there is no reason why it couldn't be...))

The ODBC_inf EV points to DB connection data and works very well.
