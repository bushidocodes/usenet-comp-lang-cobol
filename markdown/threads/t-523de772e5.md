Tom,

In an attempt to answer some of your questions:

0.1 You didn't say exactly which COBOL type definitions you tried. For example,
if you can describe an integer field in COBOL with or without the sign,
e.g.,
PIC S9(08) USAGE BINARY.
PIC 9(08) USAGE BINARY.

Did you try the definition without the sign?

0.2 VA C++ favors the stack orientation common to PC platform environments:
LIFO. VA COBOL, on the otherhand, tries to maintain compatibility with the
older-and still current-mainframe environment which doesn't use a stack.
(Note: since the 3090-150, IBM's mainframes could support a stack for
passing parameters, but I know of no compiler that generates code to use
this feature.)

2. The current version is 1.1. A fixpak was released in February and is
available by CD-ROM or FTP. A "professional" version of VA COBOL is supposed
to be introduced soon, it may feature even more compiler fixes/enhancements.

3. A version supporting W'95 and W/NT is also expected very soon.

4. Compare the instruction set of C/C++ to that of COBOL. Make sure you're
comparing/contrasting like items. Beyond that, VA COBOL is still in its
infancy. VA C/C++, on the other hand, is now a version 3.0 product, and
along the way its compiler has been improved to produce more efficient
code. The same thing is bound to happen for VA COBOL.

5. In our tests, on Pentium platforms at 90 or 133 mhz, we've experienced no
performance problems attributeable to the code generated by VA COBOL.
Remember, time and technology cure these types of problems.

6. Micro Focus, CA-Realia, AccuCOBOL, RM-COBOL,... There are a host of
other compilers out there. Not all platforms are supported, however.
