Not an immediate answer to your problem, but it may give you some ideas.

When using RM/COBOL I had a standard Declaratives Section checking for
file-status errors, checking just like you are, and errors made the program go
belly-up. (Lousy I know, but that's how I started). Perhaps there's a dab hand
here knowledgeable about Declaratives who can illustrate an extension checking
for things other than file-status.

Using NetExpress, solely OO classes - I ran into a different problem. Separate
classes for files, I can signal back to the Business Logic that a file error has
occurred and take remedial action as appropriate.

Now rather like you are finding with your structured programs - just telling
something an error occurred isn't sufficient - you need a mechanism to break out
of the program, (i.e. 'neatly' return to your Main Menu without an ABORT).. As
an example, in Class X,  Method A finds an error and I want to exit the program.
It wont do that. It will happily trundle on attempting to invoke Methods B, C, D
etc..giving some weirdo results until it goes belly-up. As a parallel this is
your structured Program X performing Para A, (finding an error) and then still
attempting to perform Paras. B, C, D etc...

This is by no means the only solution, but this is how I do it in OO, (which you
could adapt to structured ) :-

Working-storage seciton.

01 ws-ErrorCode                     pic x(4) comp-5.
   88 ContinueProgram               value 0.
   88 CancelProgram                 value 99.
*>------------------------------------------------------------
Method-id. "begin".
*>------------------------------------------------------------
Linkage section.
01 ....
Procedure Division using .....
 .... initial stuff......
set ContinueProgram to true
invoke self "get-default-info"

if ContinueProgram
   invoke self "X10-open-facility-files"

  if ContinueProgram
     invoke self "X20-MaterialCollection"

    if ContinueProgram
       invoke self "X30-open-descriptions"

     if ContinueProgram
        invoke self "X40-PipeSizeCollection"

      if ContinueProgram
         invoke etc.......

          if ContinueProgram
             invoke self "exit-program"
             .                                           <--- The All important
'period' goes here
End Method "begin".
*>------------------------------------------------------------
Method-id. "error-exit".
*>-------------------------------------------------------------
Procedure Division.
  set CancelProgram to true
  invoke self "exit-program"
End Method "error-exit".
*>-------------------------------------------------------------
Method-id. "exit-program".
*>---------------------------------------------------------------
Procedure division.
   invoke os-Dialog "hide"
   invoke self "close-files"
   invoke self "finalize-objects"
End Method "exit-program".
*>---------------------------------------------------------------

and ..... a method which might either cause or receive the result of an error :-

       *>-------------------------------------------------------------
       Method-id. "item-and-point-classes".
       *>-------------------------------------------------------------
       Local-storage section.
       01 ls-result                         pic x(4) comp-5.
       Procedure Division.
        invoke ItemNames "new" returning os-ItemNames
        invoke os-ItemNames "begin"
               using os-Desktop os-DescripsDBI returning ls-result

         if   ls-result = zeroes
              invoke Points "new" returning os-PointNames
              invoke os-PointNames "begin"
                     using os-Desktop os-DescripsDBI returning ls-result
         End-if

         if ls-result <> zeroes                  *> somewhere prior I have
already issued
            invoke self "error-exit"            *> a windows messagebox for the
error
            EXIT METHOD
         End-if

       *> There could be some more code here but it gets bypassed by the
        *> EXIT METHOD above, if appropriate.

       End Method "item-and-point-classes".
       *>--------------------------------------------------------------

AND........ before Bill jumps in - the new COBOL 200X  has a list (approx 122)
of Exception Conditions, functioning much like the Declarative/file status
checking.

HTH.

Presumably, the user gets some form of output from the job, yes?
Instead code your COBOL program to produce a meaningful message telling
them exactly what the problem was and the steps required to remedy
the error. That error message could easily appear on the output

happier to find out that his output was in error rather than wondering
where the hell it was.  Expecting the user to check his email to see

Some of us automate processes for the users.  Instead of producing a
schedule for the user to run certain programs asking then to start the
program, enter parameters etc and inspect the result, we build a
schedule for the _computer_.  The computer runs the program, if all goes
well then the user need not know that his data has been updated,
however, if there is an exception the user needs to be involved.

For example orders may be arriving by ftp in EDI or XML format.  These
can be processed through to printing a picking slip in the warehouse
without involving the sales dept - unless an error occurs. (I do this).

For example in a shipping system, a process may be asking the port
company's web server for status of a number of containers to update the
shipping system and send EMails to carriers as they become available on
the wharf and cleared by customs and MAF.

Now this _could_ be done (and was being) by having a user connect to the
internet and bring up the enquiry form, key in the container number, see
the status and key this into the shipping system.  But now they only
need to be involved if there is an exception.

Do your clients have to do everything manually ?

But I suppose some are happier with a manual telephone patch panel than
an automatic exchange.
