This is just my usual plug to IBM Mainframe sites (well at least OS/390 or
MVS ones) that you really, REALLY, need to look at the
    FASTSRT
compiler option.

If you use this compiler option (available since about 1985 - with all
releases of VS COBOL II and COBOL for this-and-that), you will find (in the
VAST majority of cases - but not quite universally) that "internal sorts"
perform as well as "external" ones - AND are more "maintainable" in many
(most?) shops.

If you aren't familiar with this option - or you shop still has a "standard"
of using SyncSort or DFSort via External Sorts - please look into this
facility.

Most mainframe shops have the FSTSRT option turned on. The major reason MOST
COBOL programs are disqualified from using this option is that their internal
sorts do not use USING...GIVING, but have an input procedure  and/or an output
procedure. Even if the program starts out as a qualified candidate, maintenance
down the road can disqualify it for use of FSTSRT. Suddenly, the program that
ran in 20 minutes, now takes 90 minutes.

Both DFSORT an SYNCSORT  can perform reformatting records, including/omitting
records, summing fields, dropping duplicates, and much more. Things generally
thought of  as requiring a COBOL program to do.

Maintainability is in the eye of the beholder. The complexity of programs with
internal sorts are greater by  orders of magnitude. Too many times, programmers
write COBOL programs  which do nothing more than reformat a record because they
lack the skill set of knowing how to code an external sort. The result is that
you end up with another program and load module that must be maintained.

Regards,
John
