Hi Keith,
Firstly, OLE doesn't handle structures very well (the standard COM 
marshaller would need to know the structure of the data in order to do 
things like AddRef interfaces etc.) and is far better with basic data 
types.

I don't think VB 5 allows passing user-defined data structures (well, I 
couldn't get it to work, but then the errors VB gives completely baffle 
me at the best of times!). 

VB 6 uses the new COM VT_RECORD type (which were introduced in NT 4.0 SP4, 
IIRC, and might not be supported over DCOM). We don't yet support this 
easily in Object COBOL but it is possible to access the data with work.[1]

I'd recommend either passing the data items as separate parameters.
Apart from anything else, this gives COM less work to do, so should give 
better performance. Plus you only have to pass the data members that 
actually matter to the method.

You could also consider creating SafeArrays (VB arrays) of types (maybe 
even Variant types) - these will be received on the COBOL side as 
"olesafea" objects, with a set of methods for accessing the data.

[1] If you *really* want to know how it might be done (and it involves 
some deep COM/API programming), send me an e-mail. I'm assuming Net 
Express 2.0 or above, here.

Steve.
