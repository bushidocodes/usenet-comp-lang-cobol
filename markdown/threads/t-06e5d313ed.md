You _cannot_ match files with DFSORT or SyncSort. The MERGE statement is
intended to _interleave_ the files, not match them. Same in COBOL. If you
need a lot of speed and do have Easytrieve, use it. It does what you need
rapidly. As far as ruling SAS out is concerned, it does not match flat files
as fast as EZ, yet I use it on a regular basis (same MVS) to process
multiple files and DB2 tables in excess of 200 million records without
experiencing performance problems I can't deal with. On our OS/390 machine
(far from being as contemporary, fast, and uncongested as I wish), it takes
SAS about 60 CPU seconds to read, process, and output a 34 million record
flat file with LRECL about 100 bytes. Therefore, it takes a SAS programmer
to write your program (including JCL and comments) run it, and produce the
output faster than writing a post about how to do it. Assuming that 1) your
files are presorted the first 15 bytes, 2) both are 100 bytes long and
associated with DDs A and B, 3) the output file, whose DD is called C,
should contain the key and the non-key parts of the input files (185 bytes
total), here it goes:

// EXEC SAS,OPTIONS='ERRORABEND'
//SYSIN DD *,DLM='$$'

*** Refer to the input - can do same in JCL;

FILENAME A 'name of file A' DISP=SHR;
FILENAME B 'name of file B' DISP=SHR;

/*----------------------------------------------------*/
/* Note: Using views against presorted files as       */
/* opposed to physical SAS data sets is critical here */ 
/* since it avoids IO otherwise required to write and */
/* read intermediate results. Same with DB2 tables.   */   
/*----------------------------------------------------*/

DATA A / VIEW=A;
   INFILE A;
   INPUT @1 KEY $CHAR15. @16 REST_A $CHAR85.;
RUN;
DATA B / VIEW=B;
   INFILE B;
   INPUT @1 KEY $CHAR15. @16 REST_B $CHAR85.;
RUN;

*** Allocate output: don''t forget NONCAT2;

FILENAME C 'name of file C' DISP=(MOD,DELETE);
FILENAME C CLEAR;
FILENAME C 'name of file C'
   DISP=NEW LRECL=185 RECFM=FB UNIT=3390 
   BLKSIZE=0 SPACE=(CYL,(100,10),RLSE); 
    
*** Unmatch the files ;  

DATA _NULL_;
   MERGE A(IN=A) B(IN=B);
   BY KEY;
   IF A AND B THEN DELETE;
   FILE C;
   PUT KEY REST_A REST_B;
RUN; 

*** Done! ;
$$

care of LRECL's, and RECFM's, copying it from the input. When the file's LRECL

//S1 EXEC PGM=ICETOOL
//TOOLMSG  DD SYSOUT=*
//INPUT   DD DSN=FILE1...
//               DD DSN=FILE2...
//OUTPUT DD DSN=FILE3,DISP=(,CATLG),SPACE=(...),UNIT=...
//TOOLIN  DD  *
SELECT FROM(INPUT) TO(OUTPUT) ON(1,15,CH)  NODUPS

John,

Either you misunderstand the task, or it is formulated incorrectly. The
original poster asked how to _match_ the files and output mismatches. In
other words, one has 2 files:

File A:        File B:
------         ------
KEY REST_A     KEY REST_B 
--- ------     ----------
1   AAA        1   SSS
1   BBB        2   TTT
3   CCC        3   UUU
4   DDD        4   VVV
5   EEE        5   WWW
6   FFF        5   XXX
6   GGG        7   YYY 
7   HHH        8   ZZZ
----------------------
and is required to produce the following:

File C:
------
KEY REST_A REST_B
-----------------
2          TTT
6   FFF
6   GGG
8          ZZZ  	
-----------------

That's what _matching_ means. Note that the satellites are not required to
be of equal length or even the same type. To accomplish this, the input
files must be read concurrently, i.e. treated as different streams, which by
no means can be achieved by concatenating them in the input. ICETOOL treats
input as a single stream, no matter how many concatenated files FROM is
supposed to swallow, and has no way of knowing which key comes from which
file. As a result, your ICETOOL job (assuming that the satellites have
identical attributes) will produce

File C:
------
KEY REST_C
----------
1   AAA
2   TTT
3   CCC
4   DDD
5   EEE
6   FFF
7   HHH
8   ZZZ      
----------

and will certainly contain keys _common_ to both files, that is, fail the
task. So, ICETOOL, no pun intended, is not the tool for the job. As opposed
to that, EZ and SAS provide automated file-matching capabilities. As far as
performance is concerned, IBM have a very long way to go if they want their
DFSORT to catch up with SyncSort. Personally, I don't believe it ever will.

Yes, you can use DFSORT's ICETOOL to do this and since it will use
DFSORT to do the sort, the performance should be fine.  Here's an
example of the kind of ICETOOL job you'd use:

//DOIT JOB ...
//NODUPS EXEC PGM=ICETOOL
//TOOLMSG DD SYSOUT=*
//DFSMSG DD SYSOUT=*
//IN DD DSN=... input file 1
//   DD DSN=... input file 2
//OUT DD DSN=... output file
//TOOLIN DD *
  SELECT FROM(IN) TO(OUT) ON(p,m,f) NODUPS
