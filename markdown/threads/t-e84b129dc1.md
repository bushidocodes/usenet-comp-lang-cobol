How about an interface to XML?

XML is a HTML which can do type setting if needed.  Let the Report
Writer section generate lines and breaks.  Let it also generate
tags to go with the headings and details.  Provide a standard set
of tag names defined to work in a line printer environment.  Let
the compiler implementor deliver those tags, integrated into some
serial output scheme going directly to a dumb printer.

The report writer section would have all the stuff to count lines,
etc. as it does now.  Likewise, it does grouping and breaks.  Now
the extensions need to support two modes.

First, the compiler (or run time) needs the find out some basic
measures like how much vertical space the heading and detail line
will take, how big the page is, and them it can do all the
calculations for vertical layout.  Same for horizontal.  This
information can be derived from the name of the tags associated
with each part of the report.  For example, a heading line can
have one of H1 H2 H3 for the tag.  H1 can be defined by the
standard as 250% size of basic line, H2 200%, etc.  Another set of
tags for the basic line can say 10 point, 12 point, 16 cpi, etc. 
Again, the standard defines the a set of tags and dimensions for
each.

The second mode is type set mode when the tags are imbedded in the
data stream with the print characters.  Run time also needs to
know that it should suppress some breaks (say page breaks) because
the type setter will determine where they fall.  The type setter
will see tags around the heading lines, the data lines, etc. 
These tags will be chosen by the programmer written in extensions
"TAGS are <ASD> <ASD>"  to report writer syntax.  Understand that
typeset mode will require 1) coordination between the programmer
and type setter over the names/definitions of the tags, and 1) a
separate typeset program outside the compiler.  This program
should be XML compliant (to a XML subset defined in the COBOL
standard.  I add this last part because XML is so darn flexible
that the COBOL program won't be able to use every XML feature due
to COBOL report write definition limitations)   

For Typeset mode and an enhanced printer, the compiler generated
output delivers tags as well as the data in a stream which can
them be feed to a print/browse program.  In a windows environment,
this would be IE5 (or the underlying pieces of it).  In a MVS
environment, this could be AFP API calls and formdefs/pagedefs.

For basic mode, I would expect some environment related support in
particular compilers.  For example, an IBM MVS environment
compiler would probably provide generated code/runtime code that
would call the AFP APIs based on the tags.  A PC oriented compiler
product would probably provide I/O routines that use the tags to
generate some PCL (HP printer control language) so the program
produced a data stream that can go straight to the printer.  

The most primitive compiler i/o would need to use some common
white space characters (LF, CR, HT, BS) to produce something like
an old TTY would print, or carriage control and TRC like the IBM
3800 uses.

The problem with a solution depending on the use of Crystal Reports or a
browser is that they are not communicating. That's what I like with
Formprint. You load a report on the demand from the user, and fill it with
data directly from the Cobol application, you do not go via disk files and
continue to run without having any idea of what actually happened. Besides,
nor CR or a browser solution gives you field by field control. Also, why XML
in such a case? The XML is only good for the browser, given a solution that
identifies the field you would get less overhead.

Pseudo code for how I think it should be:
    Call LoadDLL
    Call LoadForm form_name
    Call PreparePage
    Perform until finished
        Call PutData "MyFieldIdentifier" "ThisIsMyData"
    Call ShowData
    If not Successful
        RollBack
    Call UnloadDLL

All logic related to the form should be stored in the form description file
and used by the form application with the essential difference from Crystal
Reports and Browser that there is a real time dialog field by field between
the print engine and the Cobol application. Simply said, I would equal the:
    Call PutData field_identifier data_item
with:
    WRITE printer_record FROM data_item

<T2>

Report totals
</REPORT>


Now when you print on a page, if you know the page dimensions as
well as the dimensions of each line, then you are in.  With
standard height lines, you count lines just like report writer has
been doing since day 0.

The next advance is when you know the dimensions, say in points or
% of line heights or something else you can turn into inches.  The
layout is similar to counting lines, but you have factions of a
line (i.e. inches or cm).  But there is not a big problem figuring
if a group will fit on the remain page, or if a detail line will
be widowed or orphaned.

You can also replace the <tags> with escape sequences for a

COBOL compiler for common use on a platform to know a little bit
about HP PCL printers, IBM 3800 printers, AFP printers, TTYs, what

sold/used a lot.

then the output could be sent to a file, and that file loaded into
the browser.  The browser PRINT button could give hard copy.

Note that the browser doesn't have to insert page breaks.  WWW
browsers don't when displaying on the screen, and can when
printing.  When an external program is processing the COBOL
program output, some cooperation is needed so that report writer
does not insert page breaks if the typesetter does not want them
(e.g. screen display) or will decide itself where to put them. 
For example, it can format a report writer group to see how big it
is, then decide based on the current paper in use if it will fit
on this page or the next.  

The chief objection to WWW Browsers is that they lay out a page as
the choose to be viewed on a screen in a variable sized window. 
Most people would like to print to a fixed size piece of paper and
put print lines exactly where they want (fill in a form).

To overcome this, XML is being deployed.  GML (IBM DCF/Bookmaster)
HTML, XML, and others are defined by a very complex standard
called SGML.  SGML in all it's glory and complexity is used in
some type setting applications as well as others (indexing,
viewing, audio readers).  Being too complex to implement with good
performance and easy, SGML has been used to define XML.  XML is
itself a way to define tags, and how they are to be processed. 
That processing can include pixel resolution, type styles, style
sheets.  Almost anything you can ask a type setting program to do
can be marked up in XML.

Filling out forms is trivial IF you have a set of tags defined
which understand where things are on the form.  For last printing
subsystem I did, I caused the programmers to tag everything
including return addresses and zip codes.  The printing system
counted "to" addresses to see how many copies, printed BAR codes
for mailing from the ZIP codes, and even sorted by ZIP code for
the cheaper mailing rates.  Programmers first ask "why can't I
just tag an address as a bunch of lines (with line breaks)".  I
explained that the tag was not formatting directions (don't flow
these lines) but usage information (this is mailing ZIP info).

tags and their implementation.  That's where the COBOL report
writer standard will have to exert some controls like "The report
writer expects the tag definition to include/not include these
actions".

group could be required to maintain vertical alignment of the
enclosed lines so that a font with variable width characters can
be used, and duplicate information on lines within a group can be
blanked out on lines after the first.  (There is better way to
handle this problem -- see tables).
