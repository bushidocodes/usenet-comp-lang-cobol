Just to highlight a little-known IBM software that can quickly open up
the mainframe as a server in a client/server world and with it, COBOL as
a key server language for large scale enterprise client/server systems.

The IBM software is called CICS Client V2.

If your organisation already has a mainframe and you have a need to
create client-server applications that will integrate with the large
number of mission-critical legacy applications that you already have;
ensure that your client-server architecture can accomodate a large suite
of highly inter-connected systems; with many systems having very large
volumes of data and with 100's to 1000's of users to support in an
online-immediate update type of environment, then this product might be
what you are looking for.

(I'm not from IBM - just a user who have used this product and find it
very good and would like more people to be using it so that there'll be
a stronger support from IBM and third parties for this product - there's
lots of potential to push the product further)

CICS Client V2 is an IBM product. It sits on the PC and allows PC
programs to carry out an EXEC CICS LINK to a mainframe CICS program (ie.
a subroutine call to a CICS subroutine; the PC playing the part of the
main program). Data is passed between mainframe and PC by means of the
COMMAREA (ie. CICS subroutine parameters).

Basically, it's an RPC(ie. Remote Procedure Call) method of
client-server but much easier than the standard DCE RPC (no need for
such complicated things as stubs and such).

The other thing is that this is basically a 3-tier method of
client-server with DBMS, CICS program and the PC program as the 3
logical tiers. Being 3-tier and thin client, there are few performance
and security problems. As I mentioned previously, it should have as good
a performance as normal CICS applications (ie. able to support 100's to
1000's of users) and security, likewise.

For those who know CICS programming, it is quite straight forward on the
mainframe end (just write normal CICS subroutine-type programs). The
CICS programs can be used to read and update VSAM, DB2, IMS and other
CICS-supported data. (You can even use CICS to trigger off batch jobs).

At the PC end, CICS Client support different PC operating systems and
different PC programming tools. For operating systems, there are
versions of CICS Client for Win 3.x, Win95, WinNT, OS/2 and Mac. For PC
tools, there is support for any of the PC COBOL programming tools,
Visual Basic, Visual Basic for Applications (ie. the macro language for
MS applications such as Excel), C, C++, VisualAge. The support for VB
and VBA is thru an OLE inteface so basically any PC tool that can call
OLE interfaces can work with it (this means almost all the popular
tools like Delphi, Powerbuilder, of course, VB and even applications
like Excel, Word).

The programming in VB to call CICS is just thru something like
this:

CONNECT.LINK flow, "GLPGM1", buffer, uow

where
- CONNECT.LINK is the command to execute an EXEC CICS LINK to a CICS
program,
- "GLPGM1" (or whichever you choose) is the name of the CICS program
being called,
- buffer is the COMMAREA being passed between Mainframe and PC
- flow and uow are parameters that don't really need to be used for
normal applications

You can then use LSET command to move the COMMAREA into a VB record-type
variable to break it up into its constituent variables for use in the VB
program.

(For PC COBOL applications, instead of CONNECT.LINK you just call a
Cobol subroutine provided by CICS Client that does the same function.)


The good news is CICS Client is free of charge but provided you upgrade
to the latest version of CICS (now know as something like Transaction
Server for OS/390). However, even if you buy it, it's quite cheap at
about US$100 per copy.

The bad news is that you need additional communication h/w and s/w to
support it -- basically to allow SNA to talk to PC network protocols
such as TCP/IP and Netware's IPX/SPX. This can be done thru a variety of
methods such as using MS SNA Server or Novell SAA Server or CICS/NT or
CICS/2, CICS/6000. The gateways should be quite efficient (able to
support many clients; for the SNA and SAA servers, should be able to go
up to 100+ concurrent active users per gateway - ie. users actually
doing a CICS link at the same time). So there should not be much
gateways required and it should not be too expensive to get the
necessary equipment.

(For those who already have TCP/IP for MVS, sorry, bad news - no direct
TCP/IP-for-MVS support for CICS Client. You still need one of the above
gateways).

One word of warning in using CICS Client. The documentation is not too
fantastic and there are still some frustrating missing pieces. However,
workarounds are available if you work on it or ask around.

Anyone interested in further details are welcome to email to me.

Best regards.
