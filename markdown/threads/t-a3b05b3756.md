If you use Nested Programs, they are compiled from a single source
unit into a single run-time unit.

If you use a static call (using the name of the called routine as a
literal directly in the calling program), they are compiled
separately, but link-edited into a single run-time unit.
EXAMPLE: 
            CALL 'SUBPROG1' USING parameter list.

If you use a dynamic call (using a field name containing the name of
the called program, or, in some operating systems, a Link-editor
parameter), they are compiled separately, and the called program must
be available at run time.
EXAMPLE:
       01  CALLED-PROGRAM            PIC X(08).
           -
           -
           MOVE 'SUBPROG1' TO CALLED-PROGRAM.
           CALL CALLED-PROGRAM USING parameter list.
    

The subprogram does not care if it is called statically or
dynamically.

Stephen J Spiro

FYI - the "following" information is absolutely correct in an IBM mainframe
environment - it is NOT a portable definition as the COBOL ANSI/ISO
Standards do *not* have the concept of "static" and "dynamic" CALLs.  In
fact, if you read the "fine print" in the IBM documentation, they do
EXPLICITLY state that when you are using static CALLs, that you are running
in *NON* ANSI conforming mode.

(The reason that using static CALLs is not ANSI conforming is:
  A) ignore the CANCEL statement
  B) may or may not correctly put subprograms into initial state when the
ANSI/ISO Standard requires it)

P.S.  You can tell that the sample code below is NOT ANSI conforming (for
the '85 Standard) as it uses "single quotes' rather than "double quotes".
The draft of the next COBOL Standard *does* (finally) allow for the use of
single quotes (aka apostrophes) in alphanumeric literals - as many (but not
all) implementors have allowed as an extension for YEARS.

There is no static or dyamic CALL in the COBOL standard.
The question of static versus dynamic refers to linkage, not
the CALL.

A COBOL "CALL literal" is static if, and only if, the called
program has been statically linked to the calling program,
otherwise linkage is dynamic.

Earlier posts referred to AcuCOBOL. Stephen J. Spiro
provided information that was relevant to IBM. Bill provided
a correction and additional information relevant to IBM.
Thane quoted Bill, out of context, and stated the comment
did not apply to Realia.

The ANSI/ISO Standards *do*  support both CALL identifier and CALL "literal"
(not 'literal' - with single quote).

Many (but not all) workstation (and mainframe) COBOL's currently support
