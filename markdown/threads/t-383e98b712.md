```
Sometimes the fact that Embedded SQL can't support DDL for RDBMS can be a 
problem.

If you want to create dynamic tables from COBOL you can't because there is 
no support for CREATE TABLE, for example.

When faced with this problem some years back I got round it by painstakingly 
analysing the Data Access Objects (DAO) and ActiveX Data
Objects (ADO) Object Models and wrking out how to implement them in COBOL. 
It was laborious  but I got what I needed. These days I could do it much 
more easily because I have learned more... :-)

As a result of this exercise, I have some OO COBOL components that implement 
DAO to create tables dynamically.

I'm currently working to implement support for Oracle DB into the Migration 
Toolset. As the Toolset already supports Access, Access 2007, PostgreSQL, 
MySQL, SQL Lite,  and SQL Server, I expected this to be fairly routine...

But Oracle is "something else..." :-)

I won't go into a rant here about what I really dislike in Oracle or why I 
would never use it personally, but I'm wondering if anyone has ever tried 
accessing Oracle through DAO?

I have it working and I eventually managed to get the right connection 
strings worked out, but there is ONE major problem. If I ask it how many 
tables it has, it lies. I invoke the get COUNT property on the TABDEFS 
collection and it tells me I have 5120 tables on a database instance that 
actually has about 40... As this number is used subsequently to limit 
iterative processing it isn't very long before the component throws an 
addressing exception because a COBOL table which stores table names has 
overflowed.

I realize this is kind of Twilight Zone for most COBOL guys, but if anyone 
is using Oracle and has an idea how I can obtain the number of tables 
attached to a given UserID (WITHOUT actually enumerating them; there has to 
be a property somewhere that is accessible in SQL...) please throw me a 
bone... :-)

I can probably get round it if I use .Net stuff (OLEDB provider), but I 
don't want to have to rewrite a COBOL component that is working fine in 
every other respect and with every other RDBMS, (especially as it is a DDL 
generator and does some complex processing.)

Pete.
-- 
"I used to write COBOL...now I can do anything."
```
