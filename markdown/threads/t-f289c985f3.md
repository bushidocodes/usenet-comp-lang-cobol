Hi,

I have a Problem with Files that were created on a NCR Unix machine under a
COBOL Version I don't know:

I have to convert these files to be used by MF COBOL -- at least to read them
in and output them in a form required by our Software...

Unfortunately these Files make use of COMP-6 fields. I only have the records
description from the files, no access to source code or so!

The MF COBOL Version I have is the COBOL/2 Workbench 2.5

How might I access these files from MF COBOL?? Will I have to make use of the
convert3 Utility, first? Or might I read the files in directly?

The files are Index-Sequential Files. The first one that I'm currently working
on has a record length of 512 Byte. The first record contains what seems to be

When submitting a "base" Program I wrote that includes select and FD for the

I already tried to declare the file as "sequential", gave the RM "ANSI"

field that contains "10" (looked at with the hexedit) and which is declared as
"PIC 99 COMP-6." will be displayed as "016". This works for fields that
contain values less like "01" or "00" -- but not for fields that contain
greater values).

Does anybody have a hot-hint how I might solve these problems and access the
data as soon as possible (we are a bit under stress, currently, because of the
file conversion). As mentioned: I do not have to make any "use" of the old
files, as the customer uses a completely new Program -- I "only" have to read
them in once and then, on-the-fly, write out the new records as required by
the new Software.

Many thanx in advance,

Ralf,

COMP-6 is not defined by the COBOL standard - it is compiler
specific and thus will depend on which compiler the system was
compiled with.

>From your information, it looks as though MF is treating COMP-6 as
standard COMP (or COMP-4/5).

I think that your COMP-6 *MAY* be an unsigned COMP-3, which you
can possibly recover by MOVEs into REDFINED work fields to extract
the value. You will need to know the values of the fields for a
couple of sample records and look at a hexdump to check this out.

Data conversions are fun!

Good luck,

Brian
