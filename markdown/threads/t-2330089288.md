Xanteus,

I can't comment on your particular environment - but I do use GUIs in NetExpress
(desktop version if you like). ...."pains or glory stories....." - pain, but not too
much glory, although I'm getting there. Three options (1) call APIs (2) GUI classes
or (3) Dialog System. I use # 2 and certainly initial pain  - if you are absolutely
new to windowing. Additionally, NetExpress now includes links to Java, but I haven't
used them.

I'm sure you can probably use all three, (four), or if you want to use 'vanilla
flavour' let Flexus SP2 or Norcom Screen IO handle the 'funny stuff' leaving you
with traditional structured programs. They both are geared to COBOL.

Just to throw it into the mix, Fujitsu POWERCOBOL offers a very interesting GUI
tool, generating (as the equivalent of int code), the necessaries for GUI-ing and
methods.

A factor to remember - GUIs are not and probably never will be part of the COBOL
standards, so if it is built into NetExpress or Fujitsu, it is NON-PORTABLE. That's
not the case if you are using SP2/ScreenIO or are invoking VB, Java etc. - they
represent stand alones. There are COBOL developers successfully using the
tools/languages I've mentioned.

Why I really responded to this was I picked up on Jeff's reference to a Component
Generator - this is the type of thing, as I understand it, that POWERCOBOL does.
It makes sense to pursue, probably outside of COBOL, a Component Generator.

When the person describing the concept to me, said in effect, ".... the Generator
lets you drag and drop controls from a toolbar, or a reference to a database, gets
the associated properties, plus the associated method procedure for each control.
etc......".   Then my ears pricked up when he continued,  "...... and you can do
this in 6 seconds as opposed to some 20 minutes in COBOL or C ......."

So before you jump - certainly pursue this Component Generator concept - important -
it can be language independent.


Jimmy, Calgary AB
