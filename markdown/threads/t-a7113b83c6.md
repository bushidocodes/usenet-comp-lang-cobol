```
When I was typing that last example I posted, I realized that almost 
none of the code snippets I see here have working-storage variables 
preceded by WS-.  When I first took COBOL, our instructor said it was a 
good practice, and then our shop here has style rules enforcing the same.

In our shop, we have database records with 01-level entries (i.e., 
Eqpp-158 is a piece of equipment), and all the items under it have the 
record name (minus the code) as a prefix (i.e., Eqpp-Equip-Part-Nbr). 
The reason given is that you will know, based on the name of the 
variable, where it is defined.

(We have a similar standard for copy procs - each proc is prefaces with 
W for working-storage, P for procedure division, etc., and a number. 
W338-Unit-ID is defined in W338-Online-UCOB-WSA, and P338-Traffic is 
defined in P338-Online-UCOB-Proc.)

A couple of questions, then (one factual, one opinion-based).  Is this a 
widespread practice?  And, is it a good practice?  (My mind's made up 
about the second one, but I'm interested in other opinions as well... ;> )
```

```
Glad to:

� Write a procedure like a story -- with a beginning, middle and end. In the
beginning you clear counters, acquire memory, construct objects, open files,
etc. In the middle, you loop through whatever you're doing. In the end you print
out totals, release memory, destroy objects, close files, etc. Initialization
means 'done at the beginning'. Programs that 'clean up' at the end of a
procedure rather than the beginning are poorly designed.
As a formatting convention,  leave a blank line between the beginning, middle
and end.

� When designing, picture a telescope. The middle of a procedure calls another
procedure with a beginning, middle and end. 
a.	open .	perform until .	call 'b' 				close
b.		open 	perform until  call 'c' 		close
c. 			move zero .. perform d until ..
d.				do something
. 
� Design from the top down; program from the bottom up. When you are writing a
program, part of the time you will be designing and part of the time you will be
programming. Learn to tell the difference so you can employ the correct mental
process. Mediocre programmers do everything bottom-up.

� Use the company as a design model for your program or system. High level code
should make high level decisions; low level code should do the grunt work and
refer exceptions to higher levels for handling. If the first code in your
program opens files, you have the model upside-down, you have a file clerk
running the company. If low level code can abort the run due to an exception,
that's like giving a file clerk authority to declare the company bankrupt. 
For real-time applications, use the network model, in which modules pass
messages to each other in response to events. Think of each module as a company
department, within which there can still be a hierarchy. 

� Write function-like programs of reasonable size and simplicity. Glue them
together to form 'programs'. A reasonable size is 25-100 lines of procedure.
File IO is an especially good candidate for a callable program. 

� Use descriptive data and procedure names that anyone can understand, not
abbreviations or jargon. Run a spell checker on your program. A program is
written once; it is read hundreds of times. If the purpose of some code or data
is not obvious, explain it with a comment written in complete sentences which
avoid jargon.

� Pass data between programs in memory, not by writing it to a temporary file.
Files are for transport through time, not between programs. Top down design
makes this easy; bottom up design creates temporary files.

� Handle errors. Do not ignore return codes. Do not say you'll get around to it
later.

� Check limits on arrays, pointers for null, loops for progress. This is called
Defensive Programming. Your programs should never abend nor go into an endless
loop. 

� Write one statement per line. Indent a consistent number of spaces under data
structures, if, evaluate and perform.

� Use the free-form option of the compiler and write in lower case. 

� Write code in a consistent style. This  facilitates searching source code. For
example, when you want to search 25 source modules to find every place where foo
is modified, you can search on 'to foo'. That search will not find 'to (2
spaces) foo',  'compute foo = ' or 'giving foo'.  If your style is inconsistent,
you will have to search on 'foo', then waste time skipping references which do
not modify it.

- Do not duplicate (cut and paste) the same procedure in more than one place. If
you feel the need to do so, your design is poorly structured. Redesign so the
code appears once. If that is not feasible, due to scope of project, at least
make it a callable program. 

� Use lists rather than tables. Lists use only the memory required yet are
nearly unlimited in size. Tables wastefully allocate the worst case, then are
limited to that size. Moreover, stepping through a list as fast as stepping
through an array because the compiler must do a multiplication on every
subscripted reference. Never store an array in a file. Files should be designed
with one record per occurrence, following the standard design model for
databases. 

� Use explicit parentheses when a statement contains both (AND and OR) or ( (+
or -) and (* or /)). The compiler has 'precedence rules' to disambiguate
statements lacking parentheses. Don't learn them, lest you be tempted to use
them.  When computing percentages, multiply the numerator by 100 before dividing
to avoid loss of low-order digits. 
 
� Do not use redefines when a structure name will accomplish the same thing,
which is true most of the time.  For example: 

05  number-x.						
	10  number-9			pic  9(04).

rather than

05  number-x				pic  x(04).
05  number-9 redefines number-x	 	pic  9(04).

------- comment ------

Most of these standards address structure rather than grammar. I have contracted
to shops where programming standards were nearly all about grammar. They were
followed rigorously. But the structure of those programs ranged from poor to
horrid. Grammar controls do not produce quality programs. I dislike them for two
reasons:

1.  They give the illusion of quality control without the substance. 

2.  They stifle growth. We should be empowering programmers, not disabling them.

How to achieve reasonably consistent style across a shop? During the 25 years I
managed programming shops, I made it very clear that nobody 'owned' a program.
If you write a program today and it needs maintenance three months hence, the
task will NOT be assigned to you. You will be working on someone else's code.
Code is all team property; we're all team players.

In that environment, a consensus style is developed by the programmers
themselves, out of self-interest. They'd be challenged if programs looked
radically different from each other .. in structure OR grammar. When someone
wants to depart from the norm, they discuss it with each other informally. Not
at a conference table where egos are at risk. They each have standing to voice
an opinion because they will be maintaining it next month. 

I, as a team member, voice my opinion, which carries a 'little' more weight, but
is not decisive. In the rare event of deadlock ... that's where leadership comes
into play.

In that environment, style can grow in a evolutionary fashion. When grammar is
formalized, it grows (if at all) much more slowly .. too slowly to keep up with
changing technology and BUSINESS NEEDS that its champions claim to be concerned
about.
```

```
No.  The problem with TIP (well, it's not really a problem, more of a 
feature) is that they have files with "absolute" elements in them that 
are assigned to TIP exclusively.  I say it's a feature because what this 
allows is the TIP subsystem to load these very fast.

(A bit about modes - "Basic mode" absolutes, for the most part, are 
actually loaded to a TIP file in "Fast-Load Format".  (Basic mode 
encompasses the compilers that preceded the Universal Compiling System 
(UCS) family of compilers.)  "Extended mode" programs are compiled to 
"object modules", which then are linked to "Zero-Overhead Object 
Modules" (ZOOMs).  The difference between "object modules" and 
"relocatables" (the intermediate set in pre-UCS) is that the 
intermediate object module is executable (although most of the time it 
doesn't work).  The Link processor resolves all the external references 
(calls) and creates the ZOOM.  You can tell it to not resolve particular 
calls (which is what we would do), and these references are resolved at 
runtime.)

Now, that leads to what the problem is - once the object module is 
loaded into the TIP file, it cannot resolve to these other libraries. 
Our background programs could do it (and we may go to that soon, so we 
only have to re-release the onlines if we have a major subprogram change).
```
