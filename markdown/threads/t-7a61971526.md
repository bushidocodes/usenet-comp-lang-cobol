With an extension of "idx" it's likely you are talking about Micro Focus
COBOL (???). Yes, the 'idx' file is an index of the records in the file
you have without a suffix

    - Customers       = data file
    - Customers.idx = index to Customers' data

Records are written to the first file, Sequentially, 'adding to the tail'.
To get key order, that's where the index file comes in.

You will need to look at his source programs to find the format for the
data file. Look at the top of the programs for file definitions - he may
have used :-

    copy "customer.cpy" - in which case the copy file will give you the
data format.

You don't say how you want to re-use it. In something other than COBOL ?
If so you will need to write COBOL programs to create CSV files which you
can read into another language or a Database. If your installation doesn't
know COBOL, then you are going to need to sub-contract to get those COBOL
CSVs written.

It certainly sounds like these are MicroFocus files.  If you do not have
access to the source files (or the file definition libraries) you can
possibly figure out the layout of these data files yourself.  It is
difficult but not impossible.

Dismiss the idx files.  All of the data is in the files without
extensions.  Each data file has a 128 byte header. After you've stripped
off the header, the data records themselves are PROBABLY in fixed-length
record format.  Most programmers will choose fixed-length if it is at
all possible due to ease of use and having an eye on future conversions
(which happens quite often).

Assuming fixed length, use a good file viewer (I rely heavily on Vern
Buerg's LIST.COM) and try various values for the record length until you
can see obvious repetitive columns of like data.  Generally you can
recognize fields such as dates, names & addresses, etc.  Some may be
stored as binary and will appear as extended character symbols.  These
will require further analysis and conversion later on.

Each record begins with a two-byte header. Use this record header to
identify the start of each record.  If the file uses variable-length
records the job becomes much more difficult.  In that case, the record
header is four bytes, with the second word (bytes 3+4) being the binary
value of each individual record's length, and remember the bytes may be
swapped (little-endian or whatever).

It is also possible that the data file is compressed, which would make
deciphering the format nearly impossible.  Fortunately, MicroFocus
introduced record compression around the same time in history that
larger disk drives became available, and again with an eye on future
conversion many programmers opted to NOT use such compression.

If you still have the original compiler available, there should be a
rebuild utility laying around.  It is named REBUILD.EXE and can be used
to identify the file layout using rebuild/i, and also can be used to
convert the data to flat files for import.  If you do not have Rebuild
available, some folks here in this group might be willing to provide you
with different versions until you find one the same rev as your data
files.  If you know when the original programs were first written that
would help also.

Good Luck!
-Rob
