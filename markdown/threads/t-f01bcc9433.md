(Please note the spelling of "Boolean"; it is named in honour of George
Boole, a 19th Century English mathematician who postulated an algebra based
on two values, long before computers as we know them were ever thought of,
and who happens to be a hero of mine...<G>)

I am assuming that you want to have say, 20, variables which can all
represent two states (could be binary bits, but not necessarily). You wish
to return a TRUE or FALSE indication if ALL of these variables are in the
TRUE state.
If this assumption is incorrect then you really will need to post your
requirement in simple, clear, English (American or otherwise...<G>)

Binyamin asked why you would want to do this and I endorse his sentiment...
I guess the answer is "Because it is an exercise and I need to do it". I can
see little practical application.

Thane mentioned the use of EVALUATE...ALSO. This is a good solution but
becomes unwieldy as the number of variables increases.

The following solution allows up to 128 variables. (It can easily be
extended if you need more...)

To accommodate COBOL the variables are passed as a string of bytes. Each
byte can be x'F1' representing TRUE, or x'F0' representing FALSE. Unused
bytes will be x'00'.  (You can arbitrarily set these values to anything you
like as long as you maintain consistency).It would also be possible to pass
a list of individual variables, but this would be tedious and, as each
variable can only represent 1 of two values, it's a bit unnecessary. The
solution could be easily modified to accommodate a string of binary values
if the platform you are running on has bit operations available to COBOL.
What is here will run on any platform supporting COBOL 85.

WORKING-STORAGE SECTION.

01   finished-flag  pic x value space.
       88 finished  value '1'.
       88 not-finished value '0'.


LINKAGE SECTION.
01   in-variables pic x occurs 128
                                  indexed by in-x1.
01   return-value  pic x.
       88 return-TRUE   value '1'.
       88 return-FALSE value '0'.


PROCEDURE DIVISION using in-variables, return-value.
...
       set not-finished to TRUE
       set return-TRUE to TRUE
       perform varying in-x1
                                from 1
                                   by  1
                                 until  in-x1 > 128 or finished
            evaluate in-variable (in-x1)
                 when x'F0'
                      set return-FALSE to TRUE
                      set finished to TRUE
                 when x'00'
                      set finished to TRUE
            end-evaluate
       end-perform
       .
return-to-caller.
       goback.

There are a number of enhancements that could be made to the above (it could
be modified to return the position of every variable which was not TRUE, for
example) but I think you will catch the drift.

The literal 128 should really be a variable and some versions of COBOL allow
it to be set at compile time so that any number of input 'variables' can be
accommodated easily with a single recompile and minimum amendment to the
code. The stuff above could be edited to replace '128' with anything you
like.

Similarly, the input string could be replaced with a binary string in COBOL
that supports it. It would then not be possible to recognise 'unused'
variables... COBOL cannot pass dynamic variables (the length of the passed
string cannot be varied wthout re-compiling).

Hope this helps and good luck with your studies.

Pete.

The danger with using "all '1'" Jack, is that it presumes the collating
sequence on the target platform will recognise x'F1' as a numeric 1.  We
could just as easily use x'FF' and test for high-values... The only way to
be certain is to recognise the three possible states (TRUE, FALSE, NOT
INTERESTED) with a specific hex value. That way we don't run foul of
different architectures and collating sequences.

effort. It's a bit like defensive driving; you start off doing it
