I've used Python some. I started my learning curve with the first
edition of _Introducing Python_ by Bill Lubanovic. A link to the as of
now "raw & unedited" second edition is here:

http://shop.oreilly.com/product/0636920252528.do

Having had more experience with Perl (and other languages, including
COBOL), I have to say that when I write Python, I miss being able to
declare variables, and I find a few things less than intuitive.

On the other hand, Python has some interesting and useful features,
and it's all the rage in a lot of places. You won't get invited to
many parties if all you can talk about is Perl or COBOL. C# might be a
different story; all I can do is tell people I've heard of it, which
makes for some short conversations.

C++ would be good if the people you want to party with wear capes and
only leave home at night. Just sayin'.

I would say that you should definitely learn Python. I think a lot of
people would be interested in reading your reaction to it.

Of course it's worth learning, Peter. :-) If only to be able to keep up with FLI COBOL.

At the moment, GnuCOBOL has a branch with Foreign Language Intrinsic support.

*> zen.cob
*> Tectonics: cobc -xj zen.cob
identification division.
program-id. zen.
procedure division.
move function python("import this") to tally
goback.
end program zen.

Which displays a Python easter-egg when running this COBOL sample

prompt$ cobc -xj zen.cob

Outputting:

The Zen of Python, by Tim Peters

Beautiful is better than ugly.
Explicit is better than implicit.
Simple is better than complex.
Complex is better than complicated.
Flat is better than nested.
Sparse is better than dense.
Readability counts.
Special cases aren't special enough to break the rules.
Although practicality beats purity.
Errors should never pass silently.
Unless explicitly silenced.
In the face of ambiguity, refuse the temptation to guess.
There should be one-- and preferably only one --obvious way to do it.
Although that way may not be obvious at first unless you're Dutch.
Now is better than never.
Although never is often better than *right* now.
If the implementation is hard to explain, it's a bad idea.
If the implementation is easy to explain, it may be a good idea.
Namespaces are one honking great idea -- let's do more of those!

So yeah, go ahead, learn some Python, then use the skills directly from COBOL.

By the by, currently FLI COBOL supports FUNCTION PYTHON, FUNCTION TCL, FUNCTION REXX, FUNCTION LUA, for scripting, and FUNCTION JVM for evaluation of methods in pre-compiled Java Virtual Machine class files (which works with Groovy, Scala, Java, et al).

If you build the GnuCOBOL compiler with two or more of the language extensions, you get to call it Super FLI COBOL.

Have good, make well
