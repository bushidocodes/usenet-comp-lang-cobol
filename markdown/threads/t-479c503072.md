```
The COBOL 2002 standard states the following:

"14.5.10 Normal run unit termination

When normal run unit termination occurs, the runtime system performs the following:

1) An implicit CLOSE statement without any phrases is executed for each file that is in the open mode. These implicit close statements shall be executed for all open files in the run unit, even when an error occurs during the execution of one or more of the CLOSE statements. Any declaratives associated with these files are not executed."

Given this, what are your opinions on a 'shop recommendation' that precludes, in general, the use of explicit CLOSE statements for all files?

The reason I ask is because sometimes a program is exited early because of an error, and at that time it may be that some files had actually not yet been opened, and thus a close of said file technically "fails". So sometimes there's code like:

OPEN-FILES.
OPEN INPUT MY-FILE.
SET MY-FILE-OPEN TO TRUE.

CLOSE-FILES.
IF MY-FILE-OPEN
CLOSE MY-FILE
END-IF.

And so on...

This is necessary because if no FILE STATUS is specified (and I don't like to for plain sequential files) then you actually (under Enterprise COBOL, anyway) get a run-time error, i.e.:

IGZ0020S A logic error occurred. Neither FILE STATUS nor a declarative was specified for file FILED in program FILES at relative location X'0366'. The status code was 42.

...status code 42, of course, meaning "A CLOSE or UNLOCK statement is attempted for a file connector that is not in an open mode."
Ugh!

All of the nonsense is avoided if the CLOSE of MY-FILE is simply omitted. Let it be done implicitly!

Now obviously there are some cases where you want to do an explicit close. But it seems to me as standard practice it might be better to let it be done implicitly.

Thoughts?
```

```
It's a very bad idea.

While run unit termination will close open files, there are many cases where
you may wish to have control over the state of your file and explicitly
exercise that control.

What if you need to re-pass the file under certain conditions?

What if you are in a stateless environment (like the Web...) where, if you
keep on opening files without closing them, you may blow CGI code or other
system resources?

What if you are in a Networked environment where closing the file is the
only way to resolve deadlock?

What if you have redefined the file in order to do a lookahead read, and
then abandon the process based on the lookahead?

But mostly, if you are processing a file, take responsibility for it.

That means you open it, process it, and close it. If your process terminates
prematurely, the system will close it anyway; if you NEED to terminate
prematurely call an abort routine that specifically closes any files you
opened. (Use a flag on each file as you showed. This is a very valid way of
using files in a dynamic environment. For myself, I ALWAYS use shared files
in this way. If different object instances are sharing the file I put the
flag into the factory...if it is not shared and there is only one instance,
I put the flag into global storage so that different threads can be aware of
its status. If I'm not running multiple threads I STILL have a status flag
available in global storage because I MIGHT want to multithread the process
sometime... In all cases I explicity CLOSE an OPEN file after testing the
flag. This could be on termination or at any other time that suits my
purpose. Using explicit OPEN/CLOSE with a flag indicating status makes the
process much more flexible.)

And finally, grant programmers some intelligence. DON'T make any more rules
and recommendations than are absolutely necessary. (This one is far from
necessary; using an explicit CLOSE let's everyone know where they stand;
relying on an implicit system CLOSE is just laziness or lack of
responsibility, and could be downright confusing under some circumstances.)

I would object strenuously to such a suggestion being accepted as "approved
practice" (and even if it was recommended, I'd ignore that
recommendation...)

Pete.
"I used to write COBOL...now I can do anything."
```
