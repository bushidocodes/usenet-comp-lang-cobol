```
Overview:
Whilst developing a new Pension Quotes system, we have been asked to
improve the robustness of the existing system.
An immediate problem with the latter is that due to the large amount
of working storage required, system abends are frequently encountered.
(It is written in OS/VS COBOL and therefore is constained by 24-Bit
addressing)
 
Possible Solution
For the given IMS/DC Transaction, define a SPA scratch area.
The working Pension Quote array can be logically broken down into
various areas.
Due to the limitation of the remaining storage "window" of the OS/VS
COBOL programs to the SPA, "paging" would have to be used to Write and
Read the array.
That is, Build Sect-1, Read, free storage, Build-Sect-2, Read, Free
storage, etc.
   
Requested Advice:-
1) Is it possible to define a SPA scratch area that is greater than
32768 bytes?
   For example, 1Mbyte.

2) Is it possible to associate a Transaction with multiple SPA areas?
   For example, four
```

```
Alex,
Aren't you considering upgrading to a supported run time (Language
Environment) and COBOL compiler?  There is no reason to be constrained by
24-bit addressing limits.  It is often surprisingly easy to migrate a
COBOL program upwards, even to the latest COBOL compiler (Enterprise COBOL
for z/OS and OS/390 V3R2, if you weren't aware).

You aren't still using the version of the operating system and IMS for
which the application was originally written, are you?  Then why haven't
you kept the application tools current, too?  (This is not a stab at you
or your company, but it is one of the most overlooked aspects of system
maintenance.  My company has customers with tens of thousands of COBOL
load modules that are still OS/VS COBOL and VS COBOL II, etc., etc.)
```
