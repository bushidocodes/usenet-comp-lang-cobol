He means the response code field availble on CICS 1.7 and above.
The field is coded thus:

WORKING-STORAGE SECTION.
01 WS-CICS-RESPONSE PIC S9(8) COMP VALUE +0.
PROCEDURE DIVISION.
EXEC CICS fuction
NOHANDLE
RESP (WS-CICS-RESPONSE)
END-EXEC.
IF WS-CICS-RESPONSE IS EQUAL TO DFHRESP (NORMAL)
THEN
do something
END-IF.

The CICS translator changes the RESP(NORMAL) to a numeric value for
use in comparsion.

You can code the numeric value for DFHRESP conditionals as an 88-level
item as follows:

WORKING-STORAGE SECTION.
01 WS-CICS-RESPONSE PIC S9(8) COMP VALUE +0.
88 WS-CICS-NORMAL VALUE +0.
88 WS-CICS-NOTFND VALUE +13.
PROCEDURE DIVISION.
EXEC CICS fuction
NOHANDLE
RESP (WS-CICS-RESPONSE)
END-EXEC.
IF WS-CICS-NORMAL
THEN
do something
ELSE
IF WS-CICS-NOTNFD
THEN
do something else
END-IF
END-IF.

This usally fake most front-end COBOL editors not set up for RESP
processing. Hope this helps.

Boyce Williams.

If IBM changed their CICS exception codes, more than 88-level
programmers would be upset as IBM would violate their "backward
compatible" rule. Has IBM ever changed their CICS exception codes in
the past? Do they expect to do so in the future?

A way of dealing with the 88-level concerns is to store the response
code field with its 88-level conditionals in a copybook. whenever
the exception codes are changed in a future CICS release, one can
simply change the affected codes in the copybook before recompiling
the programs to the new release. Of course IBM can make it simpler
by suppling the response code field and 88-level conditionals the
same way they supply DFHAID and DFHBMSCA (hint, hint).
