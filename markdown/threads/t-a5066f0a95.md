```
Even the 'experts' get it wrong then.  The two tests should
be reversed because Cobol will test the condition from
left to right.  When T-Num is zero it will give a bound
error in S-B before checking for zero.

It _will_ work OK with:

     UNTIL T-Num = ZERO
        OR S-B(T-Num:1) NOT = SPACES

that it does not need to continue evaluation:
```

```
>how can i determine the string length of a variable in Cobol ?
>The variable i'm using contains either a 9 or 20 digit code,
>how can i determine this ?


In general, to determine the length of any string, you can do this:

       IDENTIFICATION DIVISION.
       ENVIRONMENT DIVISION.
       CONFIGURATION SECTION.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  STRING-WORK-AREA.
           03  SW-SIZE                     PIC  9(02) COMP.
           03  SW-STRING                   PIC  X(20).
       PROCEDURE DIVISION.
       DO-IT-TO-IT.
           DISPLAY "ENTER STRING".
           ACCEPT SW-STRING.

           IF (SW-STRING = SPACES)
               MOVE 0 TO SW-SIZE
           ELSE
               MOVE FUNCTION LENGTH(SW-STRING) TO SW-SIZE
               PERFORM UNTIL SW-STRING(SW-SIZE:1) NOT = SPACE
                   SUBTRACT 1 FROM SW-SIZE
               END-PERFORM
           END-IF.

           DISPLAY SW-SIZE " CHARACTERS".


More specifically, if you *know* that your code is always 9 or 20
characters long, you can do this:

       01  STRING-WORK-AREA.
           03  SW-SIZE                     PIC  9(02) COMP.
           03  SW-STRING.
               05      FILLER              PIC  X(09).
               05  SW-STRING-20            PIC  X(11).

           IF (SW-STRING = SPACES)
               MOVE  0 TO SW-SIZE
           ELSE IF (SW-STRING-20 = SPACES)
               MOVE  9 TO SW-SIZE
           ELSE
               MOVE 20 TO SW-SIZE.
```

```
Left or right justified? ;-)

look up 'length of', 'Inspect ... tallying', 'function reverse' or use a 
perform loop counting characters from the start or end depending on 
justification rules.

eg. 
    >* left justified code
    move 0 to count-var
    perform until digit-code(count-var:1) = space or count-var > 10
        add 1 to count-var
    end-perform

You could also just try looking at the 10th digit using reference modification. 
If it's a space, it's 9 digits.

eg. if digit-code(10:1) = space
        display "9 digits"
    else
        display "20 digits"
    end-if

of course I didn't cater for it all being spaces but you can add that.
```

```
Technically, MAX20(0:1) is not legal, which I think you already know. 
The relation condition PTR = 0 (which should be the sign condition PTR IS
ZERO) is called a "guard", because it is used to guard against the
evaluation of an illegal expression.  However, in CoBOL, conditionals are
evaluated from left to right, so in a guarded loop the guard must come
before the expression:

    UNTIL  Ptr IS ZERO  OR  Max20(Ptr:1) NOT= SPACE

When the sign condition PTR IS ZERO is satisfied, the other half of the
OR is not evaluated.  This is called "short-circuit" evaluation: the
compiler bails out of the evaluation as soon as the conditional is either
true or false.

This simple problem can be used to illustrate a more general technique,
the sentinel loop.  Consider:

    01 FILLER.
       02 FILLER PIC X(01) VALUE LOW-VALUE.
       02 Max20  PIC X(20).

This guarantees the existence of a nonblank subscript zero, index zero,
or reference mod zero:

    PERFORM
    VARYING Ptr FROM 20 BY -1
      UNTIL Max20(Ptr:1) NOT= SPACE
            CONTINUE
    END-PERFORM

It is superficially similar to what you wrote, but notice that a guarded
loop must evaluate two conditionals per iteration at run time, while a
sentinel loop evaluates only one.
```

```
One common kind of loop executes two conditionals per iteration at run
time, one for the terminating condition and one to protect the loop from
running off into nowhere.

The idea of a sentinel loop is to arrange the data so that only one
conditional per iteration need be executed at run time.  In other words,
the terminating conditional is arranged so that it also prevents the loop
from running off into nowhere.

A previous poster was looking for the last non-space in MAX20.  The
terminating conditional is easy.  But what if MAX20 is all spaces?  A
simple loop would then fall off the beginning of MAX20.  The poster
therefore used the common dual-conditional loop in which the second
conditional protects against PTR passing zero.

However, if you arrange it so that the byte immediately preceding
MAX20(1:1) is guaranteed not to be a space, then you don't need the
second conditional at all!  The loop will automatically stop when PTR is
zero without your having to test it at run time.

If the FILLER were SPACE, we would be right back where we started.  If
PTR ran off the beginning of MAX20, it would not be stopped by the
FILLER, and it would then run off the beginning of WORKAREA.
```
