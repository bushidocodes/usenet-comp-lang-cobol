I wanted to start a new thread that is a follow-up on the OO Exception
Handling and GoBack discussions.

It seems to me that there is some confusion about MIXING "procedural" and
"OO" COBOL source code.

The draft Standard FULLY allows an INVOKED method to CALL a "procedural"
subprogram *and* vice versa

Every implementation that I know of (so far) also allows this.

THEREFORE, when you are in a "subprogram" - you don't know if you got there
from a method or another "procedural program".  Similarly, when you are in a
"method" - you don't know if you got there from another "procedural program"
or method.

This is why the draft standard introduces the concept of "activating
entity".

Please, when you are talking about either GoBack or "exception handling",
remember that you don't know "how you got here".  Any "solution" you talk
about MUST take this into consideration.

I can't disagree with what you have written, but back to my three types of COBOL
'applications' :-

- Procedural - no OO
- Procedural - invoke OO support classes as necessary
- OO Classes ONLY

You start 'shoving' procedural calls into OO classes will *certainly* offend
Rick's pure OO. Dammit it even offends ME <G>. I think you are going to finish
up with a real mish-mash. Design wise I think you should be making a firm
decision  between using #2 or #3 above.

#2 presents a 'reasonable' approach, and is probably the route most COBOLers
will go, IF they switch to OO at some time. Choice #3 presents a different
conceptual approach - with a tremendous amount of planning of the support
hierarchies up front before you get into your business classes, (Edit
files/transactions/printing etc.).

If you look at Michael Todd's message 'Migrating IBM programs....", some 18,000
- phew ! - could possibly be a candidate for migrating to OO - IF there were the
IBM tools to do it. With something that size, are they splitting business
'units' down into 'sub-programs' for convenience or is it in large part calling
generic sub-routines.
If  'generics' - a good candidate for thinking at the hierarchical level in OO
support classes, plus as you'll recall from our INVOKING UP/DOWN - the ability
to 'attack' (invoke a method) either 'upwards' from the lowest level of the
hierarchy or 'downwards' from the prime class.

Another interesting point - relating to #3 (totally OO) - your comment about
procedure pointers :-

"If you set your pointer item to an entry address in a dynamically called
load module and your program subsequently cancels that dynamically called
module, then your pointer item becomes undefined. Reference to it thereafter
is not reliable."

That problem just doesn't exist in OO. You determine at the design stage that
the invoked class will :-

- contain only Factory methods - invoke something/return something - finito -
the class is no longer 'active' after the invoke

- create an Object instance :-

        1 - do something, store something in global W/S,  return something
        2 - do something, get the stored something in W/S from #1 and return
                    something
        3 - after #'s 1 and 2 the class instance is still 'active' until
explicitly 'destroyed'

I'm not trying to suggest that switching those 18,000 programs to OO is a piece
of cake. Procedural or OO there is still a LOT of work. And in a transitional
stage you may well still be calling procedural programs. However, in the long
term,  I think they will have lost the overall benefits of OO constructs, if
those calls are not converted to OO invokes.

Now the interesting thing is where you point out :-

"THEREFORE, when you are in a "subprogram" - you don't know if you got there
from a method or another "procedural program".  Similarly, when you are in a
"method" - you don't know if you got there from another "procedural program"
or method."

Do you realize you're making my case for the suggested syntax  EXIT CLASS ?
Procedural source - GOBACK/EXIT PROGRAM and for OO source (#3) - EXIT CLASS <G>

Trust me - any solution I'm thinking of, applies to Option # 3 above.

Oh, I just have to throw in an oar here.... <grin>

It doesn't matter one bit whether or not you are in a PROCEDURE or a CLASS when a GOBACK is invoked, since BOTH types of software
modules are handled with the equivalent of a BALR (i.e. subroutine) call. At least on the mainframe. Now on something like a SPARC,
which is MUCH more of a stack based machine than a 370 based mainframe, it may be a different story, but not if the compiler
writers are really smart.

Much as it shows my age, there are certain things that are just done the same way as they have been done for decades, simply because
they work. And BALR and it's descendants work well. You can't get a whole lot more efficient than the way they operate, and almost
every application out there uses the "standard" parameter storage and R1 parameter passing conventions. They work very well.

When you get down to nuts and bolts implementation, it doesn't matter much if you are in Java or in COBOL or in almost any other
language; the method invocations or subroutine calls or LINKS or PERFORMs or whatever mechanism you choose to implement for software
modules will operate like that.

The above was a 'loose' comparison for convenience. Richard you are correct to a degree, and picking up on something Donald wrote earlier,
methods are in fact 'mini programs', and as per the current state of the Standard, function as mini programs with the exception of File
definitions, Declaratives and Working Storage. Haven't looked, but logically Report Writer and Screen Section would also be excluded, needing
to be 'globally' accessed within a particular class from specific methods.

Methods aren't automatically 'nested' programs, but *could* be :-

- Not nested - invoke CharacterArray using this string returning thisDisplayObject.
(In and Out from the Class CharacterArray)

- Nested - invoke Class1 "begin", invoke self "Method2" or invoke Class2 "begin",
etc.

It would be daft to do it but you could take a full-blown Procedural program and immediately prior to MAINLINE, insert :-

*>------
Factory.
*>------
Method-id. "mainline".
*>----------------
Procedure Division.
MAINLINE.
perform Para1
perform Para2 etc....

End Method "mainlihne".
*>------
END FACTORY

END CLASS XYZ.

>
> > It's this damned 'fuzzy' exiting a class QUICKLY (not aborting) that's the problem.
…[5 more quoted lines elided]…
> nonsense because the class is not called, the methods are.

Semantics - classes are called as evidenced by referencing objects in the Animator :-

CLASS-ID.        MaterialsEdit  inherits from Base.
                           (8D 01 00 00 Class MaterialsEdit)
                            (04 00 00 00 Class Base)

FACTORY
*>-------
Procedure Division using lnk-App.
 invoke super   "new" returning ls-self
 invoke ls-self "begin" using lnk-App
 ls-self (92 01 00 00 instance of the class MaterialsEdit - but this reference is lost on exiting this method)
End Method "new".
*>-----------------
END FACTORY.
*>-----------
OBJECT.
*>----------------------------------------------
Method-id. "begin".
*>----------------------------------------------
Linkage Section.
01 lnk-App                  object reference.
Procedure Division using lnk-App.

 set os-App to lnk-App
 invoke self "getHandles"
        self =  (92 01 00 00 instance of the class MaterialsEdit)
 invoke self "createDialog"
 set ContinueProgram to true
etc.....

Exactly how the compiler/runtime handles this, I have no idea. But the start point, whether using methods solely in FACTORY or using "new" to
create an Object Instance,  is the Class.

The Animator shows no object references for Method names. A guess - using dictionary techniques, methods are qualified from a dictionary for
this class - which may mean 'extracting' the information from the compiled source residing on disk, or if already activated and cached, based
on previous usage, a 'pointer' to the start of the method detail held in memory.

*Knowing* the class identity, (Factory or Instance) - hence my tentative suggestion of  'EXIT CLASS'.
