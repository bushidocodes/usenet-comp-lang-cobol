Report on May 20-24 ISO/IEC JTC1/SC22/WG4 COBOL working group in
Zaandam, The Netherlands, and June 22-28, 1995 ANSI X3J4 COBOL
Committee meeting, Post Falls, Idaho, USA. When a vote is shown, the
numbers are for, against, and abstain (f-a-a).

The following directions resulted from votes at the WG4 meeting: the
composite language skeleton should not be deleted; allow redefinition
of a constant defined with >>CONSTANT; do not restrict such constants
to conditional compilation; do not remove EXIT PARAGRAPH; change the
method of resolving functions with optional arguments (see 1 below);
rather than specifying AUTOMATIC for data allocated when a program is
called use a LOCAL-STORAGE SECTION; restrict boolean conditions to
items/literals of length 1; use statements to get and free storage
rather than functions; an attempt to delete the ACCEPT/DISPLAY screen
handling was defeated; remove the exception recovery method added at
X3J4 meeting 199 (February) and add something else (what else is not
known); it was decided to add an intrinsic function for date
formatting as an important item; and a proposal to allow numeric
functions in other than arithmetic expressions was moved from
desirable to important. The next meeting will be in the San Diego,
California, USA, in March 1997.

The primary topics at the X3J4 meeting were: discussions on the
approval process; internationalization issues (locale and such); .
The main thrust was to get the document ready for the CD (committee
draft) review at ISO.

The major topics/proposals we discussed were:

1. A proposal to change the method of resolving ambiguity of
functions with optional arguments (like RANDOM) was passed and will be
added to the base document. This reverses our previous direction.
Now you would use "(FUNCTION RANDOM)" rather than "FUNCTION RANDOM
()". An example would be "FUNCTION MAX ((FUNCTION RANDOM) (- a + b)
c)". This reversal was a result of discussions at the WG4 meeting.

2. A proposal to limit boolean conditions to one boolean character
was passed and will be added. Previously, you could say "IF x" where
"x" was "PIC 1(4)". Now it must be "PIX 1".

3. The proposal to provide dynamic cultural conventions (locale),
based on the Posix specifications (ISO/IEC 9945-2) was passed and will
be added. It was sent back for rewrite at the last meeting. The
proposal allows dynamic choosing of attributes like collating
sequences, currency signs, alphabets, etc. The SET statement is used
(like SET LOCALE LC-COLLATE ...).

4. A proposal for more cultural convention set processing that ties
in with the item 3 above was discussed. It provides sorting by
ISO-14651, an alphabet-name LOCALE for using local sets, a method of
specifying use of cultural convention sets for characters, currency
sign, and number format (DECIMAL-POINT IS COMMA) at runtime. These
can be changed dynamically by the locale methods provided by the
proposal in item 1 above. Also, it provides some functions for
getting various forms of dates and times and doing comparisons.

5. Several proposals that make lots of changes (mostly editorial and
to improve clarity) to the OO facility were passed and will be added
to the base document.

6. A proposal that replaces the OO concepts with a new section passed
and will be added to the base document. It is better than the old
concepts and more clearly explains how OO COBOL works.


--7. A proposal that adds a standard class library was passed and
will be added to the base document. It adds 35 classes somewhat based
on Smalltalk and Eiffel. It will probably be expanded before the
final standard because none of them are business-oriented (they are
computer science type things and not all that useful for COBOL
programmers).

8. The proposal on pointers (both program and data) that was sent
back for rewrite at the last meeting was discussed at length, passed,
and will be added to the base document. It was decided that an
ADDRESS OF should be added to CALL to allow passing pointers to data
and programs. A proposal for this will be developed separately.

9. The proposal on file sharing and record locking (revision 4) was
discussed, passed and will be added to the base document.

10. A proposal that originally provided functions to get memory
(GET-STORAGE) and return it (FREE-STORAGE) was rewritten to use an
ALLOCATE and FREE verb instead of functions. It was passed and will
be added to the base document.

11. Lots of editorial and minor change papers were passed and will be
added to the base document. Nothing very important was done in these.

The next X3J4 meeting will be in Minneapolis, Minnesota, USA, on
August 5-9 1996. The main topic will be making sure the base document
is ready for public review.
