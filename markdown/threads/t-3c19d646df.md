Jayne,

    Fields accessed via Reference Modification are always treated as EBCDIC
DISPLAY. When adding a value to a DISPLAY-NUMERIC field, the compiler
normally generates a PACK, MVN (Sign-Nibble), AP, UNPK and OI (Zone-Nibble).

    If you're unsure, use the LIST,NOOFFSET Compile Parameters and verify
the generated ASSEMBLER instructions.

    My conclusion would be that you'd get an 'E' Level Error on the Compile.
But why would you want to do this in the first place? I can't see the
benefit nor reason.

Several others have told you that what you propose is (as you suspect)
illegal.  It would, however be possible (but pretty convoluted) to get what
you want in a single step - as long as you are working with an unsigned
DISPLAY numeric field.  I think (but haven't tried to compile this) that you
would need to have the following code:

 Given:

 10   FIELD-A                      PIC  9(09).

then a procedure division statement of.

 Compute Field-A = ((Function NumVal (Field-A (1:2)) * (10 ** 7)) +
                                 (Function NumVal (Field-A (4:))  +
                                 (Function NumVal (Field-A (3:1)) + 1
  End-Compute

What did you want to have happen when the 3rd digit was 9? If you want it to
go back to 0, then you would need to change the last  line of the compute
statement to include a MOD statement (I think).

Obviously, it would be a million time easier (and clearer) just to code

  10  Field-A   Pic 9(09).
  10 redefines Field-A
      15           Pic X(02).
      15  Add-Byte   Pic 9.
      15           Pic  X(06).

Add 1 to Add-Byte.

Reference modification defines a subset of a data item by specifying its
leftmost character and its length.  Only data items whose USAGE is
DISPLAY may be reference-modified.  The result of reference modification
acts like an elementary data item of the same class and category as the
reference-modified data item.  So far, it would seem that you can do
this, but there's a joker: for reasons that are unclear to me, if the
reference-modified data item is numeric, numeric-edited, or
alphanumeric-edited, the result of reference modification is
alphanumeric.

So, FIELD-A is class and category numeric, and you can add one to it, but
FIELD-A(3:1) is class and category alphanumeric, and you can't add one to
it.

Sometimes the error message from the compiler can help clarify things:

    $ copy sys$input demo.cob
    IDENTIFICATION DIVISION.
    PROGRAM-ID.  DEMO.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  FILLER.
        10  FIELD-A  PIC 9(09).
    PROCEDURE DIVISION.
    0000-MAIN.
        ADD + 1 TO FIELD-A(3:1) .
    9999-EXIT.
        STOP RUN.
    ^Z
    $ cobol demo.cob
        9             ADD + 1 TO FIELD-A(3:1) .
                          1                   2
    %COBOL-F-ERROR  326, (1) Missing literal or data-name

    %COBOL-F-ENDNOOBJ, DEMO.COB;1 completed with 2 diagnostics - object deleted
    $

Now, here's the original CoBOL thought pattern:

    10  FIELD-A            PIC  9(09).
    10  FILLER  REDEFINES  FIELD-A.
        15  FIELD-A-1      PIC  9(01).
        15  FIELD-A-2      PIC  9(01).
        15  FIELD-A-3      PIC  9(01).
        15  FIELD-A-4      PIC  9(01).
        15  FIELD-A-5      PIC  9(01).
        15  FIELD-A-6      PIC  9(01).
        15  FIELD-A-7      PIC  9(01).
        15  FIELD-A-8      PIC  9(01).
        15  FIELD-A-9      PIC  9(01).

    ADD 1 TO FIELD-A-3.

Notice that there's no subscripting, indexing, or reference mod at all. 
In a well-constructed DATA DIVISION, the reader can see at a glance
everything that you plan to do later on in the PROCEDURE DIVISION.  I
started with the simplest case because the example you gave contains only
constants and needs nothing more.  If you need more, the next step might
be:

    10  FIELD-A            PIC  9(09).
    10  FILLER  REDEFINES  FIELD-A.
        15  FIELD-A-9      PIC  9(01)  OCCURS 9  INDEXED BY FIELD-A-9-NDX.

    PERFORM VARYING FIELD-A-9-NDX ...

            ADD 1 TO FIELD-A-9(FIELD-A-9-NDX) ...

and so on.

You can pull out subfields of any size in plain old CoBOL without
reference modification:

    10  FIELD-A            PIC  9(09).
    10  FILLER  REDEFINES  FIELD-A.
        15  FIELD-A-123    PIC  9(03).
        15  FIELD-A-456    PIC  9(03).
        15  FIELD-A-789    PIC  9(03).

    ADD 1 TO FIELD-A-123.

and, of course,

    10  FIELD-A            PIC  9(09).
    10  FILLER  REDEFINES  FIELD-A.
        15  FIELD-A-9-3    PIC  9(03)  OCCURS 3.

    ADD 1 TO FIELD-A-9-3(1).

Reference modification is a useful feature for processing variable
fields, but you should not use it merely to save typing fixed fields in
the DATA DIVISION, because then you lose the single most valuable
self-documenting feature of CoBOL.
