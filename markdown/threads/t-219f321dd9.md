```
Mark,

You'll probably have to do the conversion yourself.
Create a table with 256 entries, containing for each value its ASCII
equivalent, e.g. entry 64 (x'40' in EBCDIC = space) contains 32.
Read a record and convert all character fields using this table in ASCII.
The packed field conversion depends on the format your compiler uses, it is
even possible that you don't have to change anything here.

Hope this helps
```

```
You need to do the "conversion yourself" BUT the good news is that the
conversion is relatively easy (and does NOT require a HEX TABLE).  Fujitsu
supports both EBCDIC and ASCII (in the same program) as values pointed to by
the CODE-SET clause. Check that out and it may well do what you want.  The
"problem" is that you may (or may not - depending on how Fujitsu has
implemented this) need to have all your numeric fields as USAGE DISPLAY with
SIGN SEPARATE when using the CODE-SET clause.  If that is required by
Fujitsu, you have a three step job.

1) Get the file with everything REMAINING in EBCDIC (download in "binary")
2) Convert the file (using normal MOVEs) from having Packed-Decimal (and
Binary) item - but keeping it in EBCDIC
3) Convert the file (using CODE-SET) from EBCDIC to ASCII
```

```
What about the above would you expect as 'working' ?  The CODE-SET is

It is easy to convert EDCDIC to ASCII.  You set up a table of 256
entries with the equivalent ASCII characters in it, for each EDCDIC
character get the binary value and use this as a subscript into the
table to pick up the appropriate ASCII character.
```
