Well, I don't want to start switching between using a Browser and using a 
mailer, so for me the obvious answer is to simply write a VBA (Visual Basic 
for Applications) macro that can run in Outlook Express. As I use Word for 
my email editor and both Word and OE are available as COM components (via 
Automation) it is really easy to build a macro and install it. I'll design a 
simple form and add it to the tool bar in OE. When I want to reply to 
something that hasn't got chevrons on it, I can click the toolbar and the 
macro runs. It will pop up the form, which collects information like how 
many chevrons I want inserted and what column of the lines in the message I 
want them inserted at (both these values will default to "1"), click EXECUTE 
button and the macro will insert the required number of chevrons, in the 
required place, on any part of the message that has been selected with the 
mouse. I'll probably add a button that selects the whole document, although 
the standard Crtl/A would do it too.

I considered writing a C# program to do it (also pretty trivial)  but VBA is 
already present and designed to extend Office applications in cases like 
this.

The Automation interface sees the current document as an object, so all of 
the methods and properties that can be applied to Word documents can be 
activated through the VBA macro. This means updates can be applied in situ, 
(there is no need to "Read" the lines of the document and write them to a 
work file, then sequentially process the copy), the document can be saved 
before editing it, just as if I had pressed the SAVE button or used the SAVE 
menu option. In fact, the macro can do anything that a user in the Edit 
session could do... changing fonts and formats, inserting text and other 
objects, searching and replacing, and so on, all using the standard methods 
provided by Word.

It is a bit of a brute force solution but it should be quick (I expect to 
spend an hour on it tomorrow) and it will look quite elegant and be simple 
to use.

Messages quoted incorrectly (not quoted) by OE when you respond to them ALL have a header
line reading:

Content-Transfer-Encoding: quoted-printable

Messages quoted correctly do not have a Content-Transfer-Encoding line or have one reading
7bit or 8bit, which is the same as not having one.

quoted-printable is defined in RFC2045 http://www.faqs.org/rfcs/rfc2045.html
It is a variant of base64 intended to transport messages containing mostly 7bit US-ASCII
(Latin alphabet), but capable of containing non-ASCII.

The header causing the problem comes mostly from Google Groups and OE configured to send
Quoted Printable. 

I was able to duplicate the problem with my OE. 

Deselecting "Format messages using the format in which they were sent" does not fix the
problem. It does do something. When the box is checked and I specify output format is
HTML, it disregards HTML and responds in plain text, but with incorrect (no) quoting. When
the box is unchecked, it responds in HTML with correct HTML quoting (a vertical bar). I
posted an HTML reply upthread. It came out as multipart with the HTML version in an
attachment. That's not a usable solution. 

OE-QuoteFix doesn't fix the problem.

A Microsoft Most Valued Professional says:

designed that way. I don't know for sure, but I have heard that a reply to someone using
Google Groups will sometimes not insert the carats even if the OP used MIME / None.
