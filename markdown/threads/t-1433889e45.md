Back in the 1970's and 1980's I was taught to use this method because of
the code size savings.  We were renting computer time from another
outfit, and the size of your program could be directly connected to the
cost of running your program.

Today it makes very little sense, except maybe if it makes your
procedure division code read more like english. (e.g. UNTIL
W05-PAYMENT-TABLE-SUBSCRIPT GREATER THAN W07-PAYMENT-TABLE-SIZE)

My shop follows the following working storage organization scheme so
that everything belongs somewhere, instead of just coded
helter-skelter.  Over half of our COBOL is now coded this way, and it
adds a certain comfort level when performing maintenance:
   ****------------------------------------------------------------*
   **** STANDARD WORKING STORAGE AREAS:                             
   ****       W01- ACCUMULATORS AND COUNTERS                       
   ****       W02- SWITCHES                                        
   ****       W03- CONSTANTS AND LITERALS                          
   ****       W04- DATE FIELDS                                     
   ****       W05- SUBSCRIPTS                                      
   ****       W06- KEYS                                            
   ****       W07- TABLES                                          
   ****       W08- MESSAGES                                        
   ****       W09- MISCELLANEOUS                                   
   ****       W??- RECORD AREAS                                    
   ****          - CICS MAPS                                       
   ****          - DB2 SCHEMAS                                     
   ****       L01- LINKAGE SECTION                                 
   ****       R01- REPORT WRITER SECTION                           
   ****------------------------------------------------------------*
