```
Does anyone know the correct way to test for the End of File condition
in indexed files? My compiler (MFCOBOL) objects to this way:

	PERFORM UNTIL END-OF-FILE
		READ MY-FILE
			AT END MOVE "E" TO FILE-FLAG
			NOT AT END PERFORM DISPLAY-FILE
		END-READ
	END-PERFORM.

FILE-FLAG is defined in the w-s section like this:

01  FILE-FLAG	PIC X.
    88 END-OF-FILE VALUE "E".

This code is acceptable for sequential files but not indexed files.  For
indexed files, it generates the following errors:

After the "AT END" line: Exception phrase inappropriate
After the "NOT AT END" line: A "NOT" phrase did not have a matching verb
and was discarded
After the "END-READ" line: A scope-delimiter did not have a matching
verb and was discarded

The nature of the errors would lead one to look for a syntax error, but
like I said, the compiler accepts this exact same code for sequential
files.

I would greatly appreciate any suggestions.

Thanks.

Don
```

Your sequential file is specified as ACCESS SEQUENTIAL while
your Indexed file is specified to be ACCESS RANDOM or
ACCESS DYNAMIC.  In these access modes a READ is deemed to
be a random read so AT END is inappropriate and INVALID
KEY is appropriate.

Change ACCESS MODE to SEQUENTIAL or to DYNAMIC and add NEXT
RECORD to the read to indicate sequential reading.

Viele, if the access mode is SEQUENTIAL, you can check for an
AT END condition after a READ filename.

If the access mode is DYNAMIC, you can check for an AT END
condition after a READ filename NEXT [RECORD].  The current
record pointer is sitting on the first record (in primary key sequence)
of the file when opened in this mode.  A READ filename NEXT will
retrieve the first record, and subsuquent READ filename NEXT 
I-O will sequentially process the file until the AT END condition is found.

The INVALID KEY clause will not work with either example above.

INVALID KEY would only work with access mode DYNAMIC and
a READ filename.  INVALID KEY would be invoked if the record
matching the key field in the FD area was not present in the file.
(Invalid key also applies to the WRITE and REWRITE verbs, too.)

Most compiler manuals have a nice little matrix showing ACCESS MODES,
OPEN MODES, and ALLOWABLE FILE I-O depending on the access
and open modes that were going to be used in processing.

JC Jones,

If you're just going to process the file sequentially,

SELECT ...
    FILE STATUS IS CUST-FILE-STATUS

    88 END-OF-CUSTFILE         VALUE IS "10"

PERFORM UNTIL END-OF-CUSTFILE

    READ CUST-FILE NEXT RECORD
END-PERFORM.
