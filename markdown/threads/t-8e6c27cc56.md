Notes/Question:

1) Working-storage in an object instance definition is "unique" to each
object instance.  HOWEVER, working-storage in the methods within an object
instance definition is static for the entire run-unit.  This means that WS
items defined within a method are SHARED across all instances of object
instances of that class. (This may be "counter-intuitive" if not just plain
distasteful to OO purists - as it certainly impacts data encapsulation.)

2) Files defined in an Object Instance definition are speechify to that
instance - sort-of.  Actually, they are required to use EXTERNAL, dynamic
file assignment, or file sharing to handle the fact that the file connector
in each object instance is distinct from the "similar looking" file connector
in another object instance.  ON THE OTHER HAND, files defined within a method
are shared among all object instances. Therefore, each object instance
("auto-magically") references the exact same file connector.

3) As far as I know, none (few?) of the existing OO implementations match
this definition completely and exactly.  (A Siemens person told me theirs
did, but he mentioned a few things that are "missing" from their current
implementation.)

What Bill has posted is correct.  Now let me re-post something I sent
around to J4 et all.  At first I was stunned by this, thinking it was
really really really bad.  But now I think it's not GOOD, but
understandable and OK.  The upshot of all this is that when
considering OO COBOL you have to realize that the "method" portion of
an object is always resident and the "data" portion is what is
allocated and released when objects are created and destroyed.  My
summary:


In OO COBOL - Methods are in memory and persist to the end of
the run unit.  Methods are not destroyed when an object reference
is set to null.  Creating a new object, simply creates a new area of
object Working-Storage and allows methods to work on this
storage -- the nature of which is an "object" instance.


Now, concerning method Working-Storage:  This is associated with
the METHOD, and is in memory with the method.  It persists to the
end of the run unit.  It is in initial state upon the first invocation
of
the method, but thereafter is in last used until the end of the run
unit.  Destroying objects, creating new ones, etc has no impact on
method WS.  If one wants "Fresh and distinct" storage for a
method invocation, one must use Local-Storage.

Why this is good - Constants associated with a method, used for
whatever purpose, have a place to go OUTSIDE the object data.
Meaning a million instances of the object don't create a million
unnecessary data values.

Why it is bad.  It's not well understood.  That's the only reason it's
bad.  I think that if it can be clarified somehow, it would be a good
thing.  I do not propose changing the draft standard "model" for
OO.  I can see some very good reasons for having it this way.  It
CAN be abused.  We may revisit this in the future.  However, I
think that if OO COBOL programmers understand the way methods
work, everything will be fine.


------
More - Fujitsu Specific.  Local-Storage is an extension brought on
mainly by the recursion requirement.  I have not been able to test
Fujitsu with Local Storage as they do not yet support recursion
(outside OO) and so do not support Local-Storage. 

I am going to write a test program to see if METHOD working-storage
persists across all object instances - and I hope it does actually.
One should NOT use Working-Storage in a method (under any
implementation) if it EVER changes values -- that data, if single use
during the method invocation -- should be local-storage, or for those
who don't yet have it, working-storage in the object.

I could not wait.  My tests with Fujitsu COBOL yielded the following
behavior.

Working-Storage at every level except the method level behaves as the
standard defines and as we suspect.

HOWEVER - any working-storage defined at the METHOD LEVEL, behaves
like LOCAL-STORAGE, it is in it's first used state with each execution
of the method.  This means that if you DEPEND on this behavior with
Fujitsu COBOL OO, when the standards conforming compiler, offering
support for Local-Storage, is relased, your code may be broken.  It
might be a good idea NOW, to code a *LOCAL-STORAGE SECTION, (commented
out) in each method with WS so that later, to get the same behavior,
you can uncomment the Local-Storage, and comment out the
Working-Storage.
