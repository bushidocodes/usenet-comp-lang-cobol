Hi Gug,

You should try to solve the problem in a top down fashion. For example:

perform read header rec
if input-file-eof 
   display error msg1
   stop run
if not a header rec
   display error msg2
   stop run
end-if
perform process header
if input-file-eof 
   display error msg3
   stop run
end-if

perform read 1st detail rec
perform process detail-recs
     until input-file-eof
perform end it
stop run
.

Now in your process detail recs routine
you move the data from the previous read
to your table; bump the index; read the next detail rec.

Sometimes the performed routines are so trivial, you may want to code them in
line. But it helps you think at the "macro" (as opposed to "micro") level and
not get too bogged down with the details. It's like peeling an onion, one level
at a time. Sometimes it'll even make you cry :-).

HTH, Jack.

You need to ensure that the data file actually is like the way that you
have told the program that it is.

If you have defined the file as ORGANISATION SEQUENTIAL then it may be
expecting that each record is a 24 byte chunk of data and thus will read
the 2nd record staring at byte 25.  If the data file is actually just a
text file created with, say, notepad, then the data will not line up
with how you have told the compiler to expect it.

File (if done by notepad '\n' = carriage return line feed line
terminator (CR/LF)):
koch.IND\n7115554444505000444\n7115564444405000001\n 

Record layout for SEQUENTIAL specifies:
<-record number 1------><-record number 2------><-re .. etc

If you have defined the file as LINE SEQUENTIAL then each record is 'up
to the next CR/LF'.

As you have not indicated the SELECT statement or given us enough detail
on what the actual data file bytes are I can only guess that you need to
make the ORGANISATION LINE SEQUENTIAL.
