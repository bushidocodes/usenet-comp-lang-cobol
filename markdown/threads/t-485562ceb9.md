```
This would be the technique I would use.  But if Alex M. is not
familiar with the mainframe environment, there are a number of other
problems to resolve.

First, all mainframe files (not just VSAM files) are structured, so
every VSAM file will need a unique IDCAMS REPRO job or step, because
every mainframe flat file could have a unique RECFM, LRECL, and
BLKSIZE.

Next, most VSAM files I have seen will have a mixture of EBCDIC
character fields, IBM packed-decimal numeric fields, and even a few
pure binary numeric fields (big-endian, in lengths of 2, 4, or 8
bytes).  A flat file can easily be downloaded to a PC in binary
format, but translating the data on the PC will be problematic.  Some
VSAM files will likely be structured with multiple record layouts per
file, so you cannot rely on the same data type in the same column
position or relative displacement across all records. 

You could write a COBOL program to convert any single file to pure
EDCDIC character data, but you would need a different (though similar)
program for each file to be unloaded.  A flat file with only character
data could be automatically converted to ASCII by downloading it as
TEXT.

You should be able to learn how to clone such a COBOL program for a
new record layout in less than one eight hour day.  
```

```
I think you should pull out your VSAM manual and do REPRO the file.

Something like this:
//REPRO      EXEC PGM=IDCAMS
//SYSIN      DD *
    REPRO -
      INFILE(VSAM1) -
      OUTFILE(VSAM2)
//VSAM1      DD DSN=BRSV.TEST.ONLINE.DATA,DISP=SHR
//VSAM2      DD DSN=UMS.D44201.VSAM,UNIT=DEVEDISK,
//            DISP=(NEW,CATLG,DELETE),SPACE=(CYL,(10,10),RLSE),
//             DCB=(RECFM=FB,LRECL=160)
```
