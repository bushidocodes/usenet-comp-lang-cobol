```
I'm curious how others structure their COBOL EVALUATE statements.  Here are
the most common ways I see

Most common:

EVALUATE HPR-TELE-OFFSET-NBR                 
    WHEN 1                                   
        MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
    WHEN 2                                   
        MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
    WHEN OTHER                               
        MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT     
END-EVALUATE                                 

In this case, since each condition has only a single statement I've also
seen:

EVALUATE HPR-TELE-OFFSET-NBR                 
    WHEN 1  MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
    WHEN 2  MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
    WHEN OTHER  MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
END-EVALUATE

While I can see the latter being somewhat elegant, it then leads to things
like this:

EVALUATE HPR-TELE-OFFSET-NBR                 
    WHEN 1  MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
            SET VISA-TRAN TO TRUE
    WHEN 2  MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
            SET MASTERCARD-TRAN TO TRUE
    WHEN OTHER  MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
                SET TRANSFER-TRAN TO TRUE
END-EVALUATE

Which then needs to the even worse:

EVALUATE HPR-TELE-OFFSET-NBR                 
    WHEN 1  MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
            IF TRAN-AMOUNT > 100000.00
                SET LARGE-VISA-TRAN TO TRUE
            ELSE
                SET VISA-TRAN TO TRUE
            END-IF
    WHEN 2  MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
            IF TRAN-AMOUNT > 100000.00
                SET LARGE-MASTERCARD-TRAN TO TRUE
            ELSE
                SET MASTERCARD-TRAN TO TRUE
            END-IF
    WHEN OTHER  MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
                IF TRAN-AMOUNT > 100000.00
                    SET LARGE-TRANSFER-TRAN TO TRUE
                ELSE
                    SET TRANSFER-TRAN TO TRUE
                END-IF
END-EVALUATE

I have two specific complaints about the above
1) It fouls up the "condition level" indentation.  To me, each level should
be indented by a common amount.  I prefer four, but I'm OK with three or
even two.
2) It makes you run out of room on the right much sooner!

Here's the above with the most common usage I see at our shop:
EVALUATE HPR-TELE-OFFSET-NBR                 
    WHEN 1
        MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
        IF TRAN-AMOUNT > 100000.00
            SET LARGE-VISA-TRAN TO TRUE
        ELSE
            SET VISA-TRAN TO TRUE
        END-IF
    WHEN 2
        MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
        IF TRAN-AMOUNT > 100000.00
            SET LARGE-MASTERCARD-TRAN TO TRUE
        ELSE
            SET MASTERCARD-TRAN TO TRUE
        END-IF
    WHEN OTHER
        MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
        IF TRAN-AMOUNT > 100000.00
            SET LARGE-MASTERCARD-TRAN TO TRUE
        ELSE
            SET MASTERCARD-TRAN TO TRUE
        END-IF
END-EVALUATE

At first glance the above appears to meet my criteria of each "condition
level" being indented by a common amount (four).

But in fact that is not the case.  Prior to the EVALUATE statement this may
have looked like this:
IF HPR-TELE-OFFSET-NBR = 1
    MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-VISA-TRAN TO TRUE
    ELSE
        SET VISA-TRAN TO TRUE
ELSE
IF HPR-TELE-OFFSET-NBR = 2
    MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-MASTERCARD-TRAN TO TRUE
    ELSE
        SET MASTERCARD-TRAN TO TRUE
ELSE
    MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-MASTERCARD-TRAN TO TRUE
    ELSE
        SET MASTERCARD-TRAN TO TRUE.

You'll note that the first MOVE is only indented four from where the IF
statement is, whereas with the above EVALAUTE it's indented 8.  This is
because technically, the WHEN is just a continuation of the EVALUATE
statement; it's not a statement on it's own.  It doesn't introduce a new
'condition level'.

Therefore, after years of thought (literally!) I've come to the following
preference:

EVALUATE HPR-TELE-OFFSET-NBR                 
WHEN 1
    MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-VISA-TRAN TO TRUE
    ELSE
        SET VISA-TRAN TO TRUE
    END-IF
WHEN 2
    MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-MASTERCARD-TRAN TO TRUE
    ELSE
        SET MASTERCARD-TRAN TO TRUE
    END-IF
WHEN OTHER
    MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-MASTERCARD-TRAN TO TRUE
    ELSE
        SET MASTERCARD-TRAN TO TRUE
    END-IF
END-EVALUATE

Originally when I started using a lot of EVALUATE statements I would indent
the WHEN clauses by 2, and then the procedural statements by two more, as
follows:
EVALUATE HPR-TELE-OFFSET-NBR                 
  WHEN 1
    MOVE 'VISA' TO JRMACHA-BRCH-ACCT (1:)
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-VISA-TRAN TO TRUE
    ELSE
        SET VISA-TRAN TO TRUE
    END-IF
  WHEN 2
    MOVE 'MASTRCARD' TO JRMACHA-BRCH-ACCT (1:)
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-MASTERCARD-TRAN TO TRUE
    ELSE
        SET MASTERCARD-TRAN TO TRUE
    END-IF
  WHEN OTHER
    MOVE HPR-TELE-OFFSET-NBR TO JRMACHA-BRCH-ACCT
    IF TRAN-AMOUNT > 100000.00
        SET LARGE-MASTERCARD-TRAN TO TRUE
    ELSE
        SET MASTERCARD-TRAN TO TRUE
    END-IF
END-EVALUATE

But in the end I came to the conclusion that no extra indentation of the
WHEN makes the most sense.

I will mention, because I know someone will bring it up, that there is no
"shop standard" in place.  Everyone is free to code EVALUATE as they see
fit.  Of course if there was a shop standard, and it wasn't my preferred
method, I would be quite irritated! :-)

Frank
```

```
The alternating of WHEN and IF is something I try to do so that things are 
easier to follow; it breaks things up into smaller chunks.  Yes, I know 
that the WHEN is of equal hierarchy to the EVALUATE... but EVALUATE also 
takes TRUE and FALSE, eg

EVALUATE TRUE
    WHEN FIELD1 = 'VALUE1'
        PERFORM A-BUNCHA-STUFF
    WHEN FIELD1 = 'VALUE2
        IF TODAY-IS-MONDAY
            PERFORM ANOTHER-BUNCHA-STUFF
        ELSE
            PERFORM IN-A-MEDICINE-SHOW
        END-IF
    WHEN ANOTHER-CONDITION
...
END-EVALUATE

... and the over-riding sense to me of 
EVALUATE TRUE

... equates to
IF THIS-IS-SO

... and what follows gets indented... and since I'd *really* dislike 
having two separate styles to follow, one for

EVALUATE FIELD1

... and another for ...

EVALUATE TRUE

... then I do my best to code the same way for both.
```

```
I may be in agreement; but, for me, I may need to go
beyond changing for this one statement.

After minutes of thought (and hours of further contemplation),
it occurred to me that the pattern for EVALUATE, above,
is similar to the following pattern for IF. [THEN was added
in COBOL 85.]

IF condition
THEN
    statements
ELSE
    statements
END-IF

And the following for READ

READ ...
AT END
    statements
NOT AT END
    statements
END-READ

or

READ ...
INVALID KEY
    statements
NOT INVALID KEY
    statements
END-READ

And, similarly, for all other conditional statements.

I must admit to the occasional feeling that indentation when IF
has multi-line conditions is a bit awkward and, similarly, when
indenting a WHEN by two spaces; but not so much with the
other conditional statements. Using THEN with IF and removing
the indentation of WHEN with EVALUATE, seem to be
reasonable steps toward eliminating that feeling.

However, change often means exchanging one feeling for
another; in this case, exchanging the feeling of awkwardness for
one of unease as a result of doing something unfamiliar.

In any case, a bit of experimentation is in order.
```

```
I agree on the use of "WHEN OTHER".  I often have code that does error
processing such as
WHEN OTHER
    MOVE "ERROR!  Unexpected Result!  We should not be here!" TO
ERROR-MESSAGE
    GO TO ERROR-HANDLING

One thing I've seen that causes problems is...

EVALUATE TRUE
WHEN TIME-SAVINGS
    PERFORM PROCESS-TIME-SAVINGS
WHEN MONEY-MARKET-SAVINGS
    PERFORM PROCESS-MM-SAVINGS
WHEN PREMIER-SAVINGS
    PERFORM PROCESS-PREMIER-SAVINGS
WHEN OTHER
    PERFORM PROCESS-REGULAR-SAVINGS
END-EVALUATE

The problem comes in when a new account type is added and someone neglects
to add processing to this program.  So it defaults to processing as
REGULAR-SAVINGS.  Quite likely not the desired result.  Rather, it should
be:

EVALUATE TRUE
WHEN REGULAR-SAVINGS
    PERFORM PROCESS-REGULAR-SAVINGS
WHEN TIME-SAVINGS
    PERFORM PROCESS-TIME-SAVINGS
WHEN MONEY-MARKET-SAVINGS
    PERFORM PROCESS-MM-SAVINGS
WHEN PREMIER-SAVINGS
    PERFORM PROCESS-PREMIER-SAVINGS
WHEN OTHER
*   terminate with appropriate error message    
END-EVALUATE

Of course this is not a hard and fast rule.  There are often cases where you
*do* want all unchecked conditions to perform default logic.  But in both
cases you do have the WHEN OTHER.

(Now I can just see someone doing the following...)
EVALUATE condition
WHEN TRUE
    statements
WHEN FALSE
    statements
WHEN OTHER
    statements
END-EVALUATE
```
