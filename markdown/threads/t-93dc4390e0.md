Patrick Herring wrote:

> Asbjï¿½rn Danielsen wrote:
> >
…[9 more quoted lines elided]…
> the existence of a foreign key relation. Same thing?

No, not really. The set is accessed explicit (physical) when used in an
application using network DBMS. In a relational DBMS the foreign key is
only defined when the table is defined and is not used by the application
explicit, though the programmer should ofcourse know about it. As a
consequence the "foreign key" in network DBMSs are logical and physical to
both application and database, in relational DBMSs the foreign key is
logical to the application and logical/physical to the database.

> > > 'Abstracted' doesn't mean 'hide entirely'! In fact I see relational
> > > DBMSes as basically being cut-down networked DBMSes with a 4GL
…[16 more quoted lines elided]…
>

Agreed, the implementation is something quite different. E.g. Chris Date
claims that none of the RDBMS vendors have implemented the relational model
as defined by Codd. Take a look at "The Third Manifesto" by Darwen & Date,
available at
http://cdr.cs.uiuc.edu/cgi-bin/cred/security/credentials/record/95-3/papers/manifesto.ps,
or Date being interviewed by DBMS Magazine in October '94 at
http://www.dbmsmag.com/int9410.html, last section "What is the greatest
misconception about relational technology?".

> The theory was always that SQL would make it possible for non-techies to

> DP department all the time. The last time I worked with DB2 we found our
> user couldn't get his head round the concept of a file so he couldn't
> even enter his SQL statements. So we did it for him...

:-)No, I think your user represent the common user. Take a person, not
familiar or understanding the relational model, but able to write
syntacticly correct SQL-statements, and in a matter of mininutes he will be
able to create a query (using two or more tables) resulting in a cartesian
product. At least this is my experience :-))

> ...
> So whence came the view from your page, that programmers of networked DBs
> need to know more about the physical representation of the logical
> structure than their counterparts in the relational world?

I haven't said much about the logical structure of a network DBMS. In the
essay I merly present the mechanism of navigation in a network DBMS. The
only thing I mention is that whenever the structure of the database is
changed (meaning e.g. new sets added), the application need to be changed.
For relational DBMSs I say:

"The relational approach separates the program from the physical
implementation of the database, making the program less sensitive to
changes of the physical representation of the data by unifying data and
metadata in the database. This makes the development of programs more
effective and less dependent on changes in the physical representation of
data ".

Because metadata is not "programmed" in both application and database using
the relational model, the code needed, in the program, to be able to access
the data in the database and map them into memory of the application, is
bigger (not necessarily more complex) using a networked DBMS versus a
relational DBMS.
E.g. If you in a network DBMS access a record, and on this record you are
using a set to access a number of other records, you would end up with
something two or three explicit calls to the database and perform the calls
in some kind of controlled loop. In the relational model you would write
one explicit call, a query, and the DBMS would execute it (hiding how the
actual accesses to the data are performed). To put it in one sentence; It
is easier/shorter (for most people, including application developers) to
tell what they want, than expressing the algorithm to be used to achieve
what they want.

> And that network DB designs are more likely than relational ones to
> evolve themselves into an un-upgradeable corner because the interface
> navigates the DB structure explicitly by named connections rather than
> implicitly by foreign key connections? I still don't get it.

Well, to be honest, neither do I. I have never expressed such a view
neither verbally nor in any written form (e.g. the "Essay"). Any database
using any model may get into problems if the model is "wrong" (the only
problem is to find out what "wrong" is. AFAIK no metric exist to detect a
right or wrong data model, only "rules" on how things should be done to get
a "good" model). The most common reason for a model to become "wrong" are
changing requirements. If some DBMS-concept should be able to handle such
changing requirements well, I would suspect it to be network DBMS and
OODBMSs. They would prbably handle such changes better than RDBMSs and
hierarchical DBMSs, due to the fact that their data-models are able to
express relations which the others can't.

If you find any expressions in my Essay suggesting/implying that the
networked data-model are more likely to evolve into an un-upgradeable
corner or that the network data-model is less flexible than some other,
please let me know, because it have never been intended (if it exist in the
Essay, which I find hard to believe)

I'd like to add first that I was thinking of George Dodd at GM.

Second, if you think about the bill of material problem, you will see that
you deal
with parts, assemblies, and many different attributes.    It's common for a
part to be an assembly of a group of parts.  Assemblies are often parts.
Redundant perhaps. Inventory is one thing if you just keep track of each
lowest level part. If however you need to be able to identify an assembly by
it's serial number, date of production, color, power requirements, etc.,
then you find your self with a more complicated problem. Especially if some
parts are built in house, and some are purchased or manufactured elsewhere.
Production planning, and assembly control becomes very difficult. So much so
that many major applications are built on DB2, IDMS, etc.  I know first hand
that these kinds of applications put a major emphasis on pre-programming
design efforts at the expense of major changes and that means the user has
to know what he/she wants. We all know how hard that is on the simplest
application.  New requirements occur often that have not been planned for by
the system design.
If you also consider that a large file is different things to different
people/applications.
Reading a disk file in an indexed sequential basis is far from useful in
some cases where in small volume systems, the problem might be solved now
days by an internal table from what I read here.    Random stands a better
chance of providing better
access times for large fast response time applications.  Data Base tools use
this
a lot.  They also physically organize parts of files to add to the speed of
access.
And yes, there are lots of data management problems that result.  Why do you
think we have logging of transactions, recover and restart, redundant
recording, etc.?
Some applications are not unlike message switching or network services.
Often the
users are located over a wide area of the universe.

This kind of data base is like a virtual map of the data, and the programs
(and programmers) have to know how to follow a map based upon the
application NOT
programming techniques.  Normal batch processing is very simple by
comparison.
