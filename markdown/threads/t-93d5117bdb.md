```
Since the key is the entire item, you have to be careful what is actually
in the low order position.  Consider the following.

01 this-is-the-key.
    02 this-is-high-order        picture 9.
    02 this-is-low-order          picture 9.

Lets say that you have a key on file that is actually "55".  That is,
the both the low order and the high order are "5".  If you move a five
into this-is-high-order, and read with "key greater than", you might get
that record, and you might not.  If this-is-low-order happens to be a "6"
you will not.  If it happens to be a "0", then you will get the record,
(provided that the record "50" through "54" do not exist; if one does,
you will get the first).  If you read with a "key equal to" then you will
not;  equal to means just that -- there is no "almost equal to".

Certainly you do NOT have to initialize the entire key.  However you
do have to be aware of the ramifications. If your intention is to get
all records that contain the high order portion, then you should move
low-values to the low order, read with key greater than, then read
sequentially until you hit the next value in the high order portion. If you
want an exact match, then you need the entire key.
```

```
I think your message may be more confusing than clarifying.
There is no such thing as "read with key greater than",
you are probably using this as a shortcut for:

           MOVE "5"     TO High-Order
           MOVE LOW-VALUE  TO Low-Order
           START file
               KEY > file-key
           READ file
               NEXT RECORD

To clarify: To read an exact record by primary key (RECORD KEY)
the key must be set exactly and then a random READ (ie ACCESS
MODE RANDOM OR DYNAMIC) will get the record that matches the
key specifiied.

If it is required to read by partial match then the ACCESS
MODE must be DYNAMIC or SEQUENTIAL, the partial key set in
the record area and a START done to set that point in the
file.  A READ NEXT RECORD will then read whatever follows
in the file.
```
