```
I have not really used EXTERNAL before.  I coded the following:

     FD  STDIN-FILE EXTERNAL
           LABEL RECORDS STANDARD
           RECORDING MODE IS F
           BLOCK CONTAINS 0 RECORDS
           DATA RECORD IS STDIN-REC.
       01  STDIN-REC.
           05  STDIN-CHAR          PIC X     OCCURS 80 TIMES
                                             INDEXED BY STDIN-INDEX.

I was suprised to find STDIN-INDEX does not have the EXTERNAL attribute.  

The only way I can think of to do what I want is to code:

     FD  STDIN-FILE EXTERNAL
           LABEL RECORDS STANDARD
           RECORDING MODE IS F
           BLOCK CONTAINS 0 RECORDS
           DATA RECORD IS STDIN-RECORD.
       01  STDIN-RECORD         PIC X(80).

WORKING-STORAGESECTION.
       01  EXTERNAL-DATA IS EXTERNAL.
             03  STDIN-REC
                  05  STDIN-CHAR          PIC X     OCCURS 80 TIMES
                                                      INDEXED BY
STDIN-INDEX.

and then do READ STDIN-FILE INTO STDIN-REC

I would prefer to not have to do a READ ... INTO  does anyone have any
alternative suggestions.

Before you say "don't use EXTERNAL" or "why are you using "EXTERNAL" let me
explain.  I am reading  a book on compilers which has a simple one pass
compiler program written in C and I am trying to do the equivalent program
in Cobol.  At first I was going to use nested programs and GLOBAL but three
of the CALLed programs have to be RECURSIVE and nested programs cannot be
RECURSIVE.  That seems like a silly restriction.  Of course EXTERNAL data
items cannot have value clauses and that also seems silly.  This project is
for self-traing/learning only.

I want to share the file among the programs and I want to simulate C's one
character at a time I/O.  Any program which reads STDIN will PERFORM a
paragraph, 1100-GETCHAR, which will be the same in each program and will
use STDIN-INDEX which must be the same field in each program and hence
EXTERNAL.  Thanks.
```

```
It works that way because that's the way EXTERNAL works: It only shares the
file buffer. Datanames are different.

Usual way to handle this is with the same copylib in each program...

FILE SECTION.
    COPY  'FD_FOR_FILE_TO_BE_SHARED' .

... or by using READ INTO in each Program


In your case..
>I want to share the file among the programs and I want to simulate C's one
>character at a time I/O.  Any program which reads STDIN will PERFORM a
>paragraph, 1100-GETCHAR, which will be the same in each program and will
>use STDIN-INDEX which must be the same field in each program and hence
>EXTERNAL.

Instead of the same 1100-GETCHAR paragraph in each program, you could use a
copylib for a complete nested program or a CALL to a completely separate
module which accepts the buffer and the number of characters valid as
parameters....although this is bass-ackwards from what I've seen and done...
the separate module would normally be a common I-O module which would 'get a
record from file' on command.
```

```
      FD  STDIN-FILE EXTERNAL
            LABEL RECORDS STANDARD

          ...
   Working-Storage Section.
   01  Saved-Index   Usage Index   External.
        ...
   Read ....
   Set Saved-Index to STDIN-INDEX
   Call ...


An "index" is not truly a data-item (although some/most implementations may
treat it as such). Therefore, if you want to "externalize" an index, you
need to use an INDEX-DATA-(ITEM.
```

```
Even if every program  had the file SELECT and FD statements, I believe
each program would have its own local index item pointing to the next
character to be read.  Updating and index in one program would not change
the indexes in the other programs. I have not yet reached the testing stage
as an avalanche of interruptions of various types has distracted me
temporarily.  

As to good practice: Many people whose religious affiliation is "orthodox
software engineering" claim that globally accessible data is bad i.e. a
mortal sin. It violates the principle of encapsulation and information
hiding. Data should be passed by parameters and never by reference.  The
same people will tell you that the only good PERFORM statement is an inline
PERFORM.  All other PERFORM should be replaced by CALL statements.

I personally trust myself to only modify data appropriately. I find global
data can greatly reduce parameter passing. I try to avoid GOTO but if I can
simplify higly complicated nested IF I will occassionally GOTO an exit
paragraph.  Also when simulating recursion some people use a GOTO to remove
tail recursion. I don't mind other than inline PERFORM but writting these
programs using CALL has reminded me of the benefits of this choice for
providing Cobol equivalents of C library routines. The larger the program
that you write the more you will benefit from the othodox approach.
```
