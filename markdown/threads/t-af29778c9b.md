    EXIT

  Move low-values       to Child-Stack-Area

      End-Perform

As far a I am aware recursive performs are not currently permitted,
according to the draft new standard, the behaviour is unpredictable,
presumably because the exit, as translated by the compiler, of the
perform points to the place that it is supposed to return to, which in
your case will have been altered by the second perform of the same
paragraph.

You don't have to write a recursive subroutine, the equivalent to
recursion can be achieved by rearranging the way the procedure is
written and using PERFORM UNTIL a specific suitably devised condition.

Additionally, your code never sets children-done to true.

Hope this helps,

Robert

I managed to use Bill's suggestion and altered it as so:

    - The initial invokation was done using Perform X thru X-Exit (an
actual use for that twisted construct -- whoda thunk it).

    - A special check for Children=-Done (1) did a "Go To X-Exit".

Set CS-coll to root-tree
Move 1 to CS-Ndx
Set Not-Done (CS-Ndx) to true 
                                                  
Perform Iterate-CS-Nodes thru Iterate-CS-Nodes-Exit

Iterate-CS-Nodes.                                               
    Perform First-CS-Child                                      
    Perform until CS-Done (CS-Ndx)                              
      Perform Handle-CS-Child                                   
                                                                
      Add 1 to CS-Ndx                                           
      Set CS-Coll (CS-Ndx) to CS-Coll-Kids              
      Set CS-Not-Done (CS-Ndx) to true                          
      Perform Iterate-CS-Nodes
      Subtract 1 from CS-Ndx                                  
      If CS-Ndx = 0                                             
        Go To Iterate-CS-Nodes-Exit                             
      End-If                                                    
      Perform Next-CS-Child 
    End-Perform             
    Exit.                   
Iterate-CS-Nodes-Exit.      
    Exit.                   

As a side comment, I use "Exit" only to keep the SQL, DLI and CICS
tranlators from screwing up my programs.  If you use the following:


Para-Name-One.
    ..statements..
    Exec XXXX
       somecommand
    End-Exec
    .

The IBM translators consume the terminal period and read it as
"End-Exec.".  This results in code that falls through to the next
paragraph and really bothers me.  I'm very much a single '.' per
paragraph sort of programmer

The easy fix is to make it "Exit." instead of just ".", though
"Continue." would work as well.

Name-1.

Name-1 Move Name-2 to Name-3 Name-4 Name-5 Continue

Just so the actual "rule" is understood.  *ALL* statements that have a
conditional form have a scope termintor which can make them imperativer
statements.  (This is not a rule that relates to whether or not they are
"flow-of-control" statements but whether they are conditional or not.)

This ties in with the rule (stated elsewhere) that it is NONCONFORMING
(possibly illegal depending on compiler) to nest a conditional statement
within any other statement unless:

A) it is followed immediately by a period
    or
B) it is nested within either "half" of an IF...THEN/ELSE statement.

    ***

It should also be "noted" that every statement that has both an imperative
and a conditional format *ALLOWS* a scope termintor for the imperative
format.  Therefore, the following is non-conforming source code:

Add +1 to Num1
 On Size
   Read ABC
        At End
             Read XYZ
     end-read
  Not on Size
      Display "Other"
 End-ADD

this is because the "single" end-read goes with the most recent READ
statement (the imperative format) leaving the 1st READ as conditional -
which is nonconforming in the ON SIZE phrase.  This will get even more
"confusing" with implementations of the 2002 Standard where the "ON SIZE
ERROR" phrase and "NOT ON SIZE ERROR" phrase (and all other positive and
negative conditional phrases) may be coded in either order.
