I've been testing different DAL objects to see if there is any real
difference between using Embedded SQL (ESQL) from COBOL or using LINQ
(Language Integrated Query).

I'm satisfied that LINQ is superior in every area (tested for sequential,
skip-sequential, random and update functions), so we will standardize on
using LINQ for the Data Acess Layer as from April.

Fujitsu COBOL users will know that database access from COBOL requires the
use of ODBC and ESQL. (there is a way to do it through ADO also, but this is
more obtuse and most users don't go that route. PowerCOBOL provides a
control that supports ADO and that makes things considerably easier but ESQL
is still generally used.)

In addition to Defining, OPENing, FETCHing and CLOSEing SQL cursors, an
external support file is required with Fujitsu COBOL to define the Database
and various parameters used by ODBC.

There are limitations on the number of connections you can make through ODBC
and for at least one of our clients, this made Fujitsu and ODBC non-viable.
(That was what initially made me look for another solution and led us to
LINQ...) The client was very happy with Fujitsu (it is an excellent COBOL
implementation) but once they moved from ISAM to RDB, they hit the ODBC
connection limitation wall. They are now using LINQ DAL from their Fujitsu
COBOL applications with no change to their applications and no connection
limitations...

If LINQ is used none of the above limitations apply. No ODBC, no SQL
cursors, no external ODBC.inf files...

The LINQ DAL objects connect through a standard connection string which
defines a "data context". You CAN still use ODBC but you also have the
options for OLEDB, ADO and .NET data providers.

In general, we found that LINQ is at least 3 times faster than hand-written
ESQL and in a "best case" it was more than 5 times faster...

Our LINQ DAL objects use exactly the same interface to COBOL applications as
our ESQL DAL objects, so it comes down to just swapping a set of .DLLs to
make the upgrade. The Tools generate ESQL DAL in COBOL, or LINQ in C#, but
the source language of the compiled objects is irrelevant and both sets work
perfectly from COBOL. (or any other language that supports the Component
Object Model...(OO COBOL, Java [brokered through CORBA], VB.Net, C#))

LINQ of course generates SQL at run time to access a Relational DB, but this
is optimized and deferred with an infrastructure that just isn't available
without LINQ.

If you are using RDBs from COBOL, I'd strongly recommend you to take a look
at LINQ.

(Apart from all the other arguments, the actual syntax of LINQ queries is
more logical and (arguably) much easier than SQL, once you get used to it.
The same syntax can be used to query ANYTHING that is queryable: XML, RDB,
Hierarchic DB, Arrays, collections, and lists in memory, and Entity sets
with the Entity Framework.)

As I recall it - and my memory is, admittedly, porous - skip-sequential
processing is done with indexed datasets.

START filename KEY key-name NOT LESS THAN dataname
INVALID KEY imperative.

READ filename NEXT
AT END imperative.

The START positions the LRP (Logical Record Pointer) and the READ NEXT
gets the first record with a key value equal to or greater than the
supplied value...

... or end-of-file... or something like that. Since the LRP is placed
independently of previous processing the program can skip all preceding
records and then process sequentially.

Because LINQ does ONE access and gets a "result set" it is then over to the

So I basically had a START equivalent (that returned a set of data which it
did NOT return to the application, just as START does NOT return a record)
and then a READ NEXT/ READ PREV equivalent that maintained a pointer into
that collection, which could move in either direction. Designing it all and
getting it working was fun, not to mention optimizing caches, and handling
boundary conditions, but then I had to devise code that could generalize it
for any Key and generate a general version of the solution I had just
written manually in C#. The first cut used Reflection to determine data
types at run time and allowed one set of code to process ANY table data. But
it was horrendously slow. So I replaced the Reflection with hard coded
offsets and lengths generated into it for the specifc columns in a
particular table set. Now it just flies.

And, as a spin-off, we got a "COBOL profile" (a kind of data dictionary
describing every single element defined in COBOL, for the entire data
resource...offset in the COBOL FD buffer, length, data type, OCCURS at up to
3 levels, and whether it is a key field or not...) that has been useful for
a number of other things. (This was generated from the metadata we get when
we run the Migration Toolset to analyze the COBOL FDs and SELECTs in order
to design and build a proper database.)

DAL kept me off the streets for quite a while, but it is very satisfying to
see people using it now. :-)

Thanks for your kind words, Tom.

Your assumption is pretty correct but I'm not sure it was IBM who first
coined it.

"Skip Sequential" processing is not a term used by any particular
vendor/platform. It describes a form of access that just happens to be
particularly appropriate for indexed file access (and databases...).

The term indicates a processing sequence where you start at a random point
and then read forwards or backwards from that point in the key sequence of
the current Key of Reference. (It can be primary or alternate) The
sequence is terminated when the next random start point is issued, on the
same key of reference. The new random point does not have to FOLLOW or
PRECEDE the records you just accessed; it could be one of those records.
Unlike normal sequential processing, where records are returned in ascending
or descending sequence until EOF or some criterion is met which marks the
end of the process, skip-sequential processing can "move" to any randomly
accessible point in the sequence and return records from that point in
ascending or descending sequence and it can do this as often as you like,
moving "up" and "down" the file as often as you like.

In simple terms, it is a combination of random and sequential processing.
