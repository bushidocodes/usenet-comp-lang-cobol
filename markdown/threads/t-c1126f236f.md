Why? Let it abend so it'll be obvious during testing that it's bad. The
symbolic dump that results on most systems (if the right compile/etc.
options are chosen) should help pinpoint the error quickly. If the
program's poorly written and inadequately tested, then it shouldn't be in
production anyway, but if it is, then the same consideration applies if it
happens in production - the program is invalid and further processing
cannot be trusted, so it should abend and produce a symbolic dump which a
programmer can use to quickly fix the problem.

Obviously this is no substitute for properly written code in the first
place. But IMHO it's the most logical way of handling the impossible. It
would of course be adviseable to have some sort of error-trapping mechanism
in place so that reasonable cleanup can be done, resources released,
subsequent steps not done, doubt cast on results, etc. but this should be
the default anyway after an abend, and whatever recovery mechanism is used,
the abend messages and symbolic dump should still be produced to help
resolve the problem.

Let me give you a real world example.  I have several programs that
are quite complex.  Some of the data needed by the program is located
in an Oracle database - while other information is located in local
disk files.  We are moving more and more information to Oracle.  I
have a subprogram that accesses the data in the different tables - one
program per table presently.  The linkage includes several items.  The
items to be returned are also there.  This program can be called
dynamically - which means that the calling program need not be
recompiled.  However, for various reasons some of the calls are
static.  When a new field is added the copy book for the linkage is
changed, and the programs are recompiled.  As more programs make use
of this method and the same sub programs, the task becomes more prone
to human error.  The subprogram must be run through the pre-compiler,
then compiled, before the main program.  Dynamic calling programs must
be recompiled to obtain the new copybook.

To prevent errors in this order, I built in a failure mechanism.  This
works by using the FUNCTION LENGTH to get the lenght of the linkage
section (a single 01 level) and places this length in a field in the
linkage section.  The called program, on first execution, also does a
Funciton Length on the linkage section and compares this lenght ot the
passed lenght.  If different, it does not execute!  (Causing a
failure, not unlike that which would happen if the database were
down).

This is brittle - and has actually saved me many times.  I always test
the production compiles before moving into produciton - none of these
has failed in production.  This will also help other programmers here
using my routines to ensure that if a change is made, it is made
everythwere it needs to me made, in the right order.  The programs

can be used as values in a condition or expression, as for example IF 
FUNCTION SINE (A) > 0 MOVE B TO C. In C-type languages this makes

expressions.

Given the form [from CD 1.2]

PERFORM procedure-name-1
    VARYING identifier-2 FROM literal-1 BY identifier-4
    UNTIL condition-1

will be evaluated each time the content of identifier-4 is used in an
augmenting operation.

Sample ANS 85 code is included, below. This code demonstrates
using variable augment values. The performs of p1 and p2 use
subscripts; the perform of p3 does not.

Although it is clear, from my interpretation of CD 1.2, that the
performs of p1 and p2 are correct, it is not clear that the perform of
p3 is correct. That is, there are no statements in CD 1.2 which require
that the changed content of identifier-4 be used unless identifer-4 is
subscripted. (I expected the perform of p3 to be an infinite loop! It was
not.)

subscripted, the subscript will be evaluated each time the content
of identifer-2 is referenced for setting or augmenting. This is not
shown.

       identification division.
       program-id. test44.
       data division.
       working-storage section.
       01 tabl value "02010102010102010101".
         05 table-element pic 99 occurs 10.
       01 int pic 9(4).
       01 va1-sub pic 99 value 0.
       01 variable-augment-1 value "01020304050607080910".
         05 va1 pic 99 occurs 10.
       01 variable-augment-2.
         05 va2 pic 99 value 0.

       procedure division.
       begin.

             until int > 10

           perform p2

             until int > 30

           perform p3

Something that I've been doing for several years now is to dynamically
calculate a table hi-bound at the start of the program. EG:

01  WS-TABLE-REC.
    03  WS-TABLE-ENTRY OCCURS 100 TIMES
                       PIC X(20).
01  WS-VARIABLES.
    03  WS-TABLE-MAX   PIC S9(08) BINARY.

COMPUTE WS-TABLE-MAX = (LENGTH OF WS-TABLE / LENGTH OF WS-TABLE-ENTRY (1)).

Always use WS-TABLE-MAX as your maximum subscript range. Therefore, no matter
what the occurs is (the table could be in a CopyBook) you'll always be safe.

I'm not sure if this can be used with the LENGTH FUNCTION in ANSI '85, but you
could probably get around any difficulties by loading up some fullwords and
then performing the calculation.
