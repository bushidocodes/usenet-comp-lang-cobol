What we did was to find a point in time that matched the calendar backwards
in terms of equal day, week number and leap year. This was: 28 years ago,
ie: 1971 from today.

It is a sligthly different approach than the classic sliding window, but for
a short term solution it is good because of low source impact and
efficiency.

Considering that the problem is to be in a period where you would have dates
on both sides of a century we found that all logic, computation (to some
extent) and sorting would work painlessly if all were in the same side.

So, we changed all years fields in the data files to be current date - 28.
Like: 1999 - 28 = 1971.

The only problem with this, is if you have years that are older than 1928.
Luckily, we didn't have.

Benefit:
    - Fast implementation
    - Logic unchanged
    - Sort functions as before

Wouldn't this look stupid on screen? Like Today: 7-Oct-1971?

Yes, you are right. So what used to be:

    ACCEPT DATE-VAR

Were changed to:

    CALL "MODDATE"    USING DATE-VAR    "+"    5.
    ACCEPT DATE-VAR
    CALL "MODDATE"    USING DATE-VAR    "-"    5.

    What we do here is that we take the var DATE-VAR that holds a date:
120571, states that the year
    starts in position 5, and use the "+" to indicate that we should add.
DATE-VAR will then be:
    120599. After the ACCEPT we reverse the process.

    This applied to all "external" sources, like printer, files, system and
user input.

Yes, it is not infinite, so, one time in 2020 we will have to start figuring
out something smart :-)

Cheesle

The simple answer is that you can't "automatically" remove "date-windowing"
just because 1/1/00 is past.  For example - think of a state's driver license
programs - where a person MUST be 16 to get a license.  For those
applications, "00" "01" etc (as birthdates) will still remain 1900 and 1901
until (at least 2016 - or maybe 2015 - depending on the application logic).

There are really four types of date-windowing that I know of (there may be
more):

1) Those that "window" around a fixed date, for example "for today's date -
if the year in in the range 90 - 99 - it is ALWAYS  1900, otherwise it is
2000)

2) Those that have a "relative" or "sliding" window focus.  In other words,
if the two-digit year is 3 years or less before today's year, then it is
1900 - but otherwise it is 2000.

3) Those where there is some "control card" or other run-time parameter that
tells them what year to use as the switch over. (If I run the program today,
I use "99" but if I ran it last year, I would have feed it "98").

4) IBM has actually introduced "native COBOL langue" (via extensions to the
PICTURE/USAGE clauses) to identify something as a date - and they "slide"
when doing all sorts of operations (Moves, arithmetic - even SORT) depending
on a run-time (or maybe compile-time?) option.
