```
I have been heavily involved with both PowerCobol and
Fujitsu Standard Cobol 4 over the last few months, and am
interested in hearing how others (if any) are using these
new tools.

One very interesting technique that I have just figured out
is to include an entire heritage program, complete with
screen section, in a PowerCobol program as a custom
method, then invoke it from other programs.  The only
caveat is that the program must be an infinite loop, which
took a bit of getting use to.

Typical code is a master file update that accepts a customer
number then goes into an add, change, delete routine depending
on what was entered.  The main loop accepts the customer
number on the background screen, and never exits from
that loop.  To exit, the user *must* click the upper-right-hand
close window box ... anything else gets into trouble.  In effect,
the entire old program, complete with screen, becomes an
icon on the PowerCobol Toolbar with single entry point
and an exit under control of the GUI.

Once that has been done, tested, and working, the program
can be devolved into several subroutines internally, and additional
methods added that call those subroutines.  The end result is
a "customer handling tool" that can look up customers, add
delete, and change customers, etc., for other programs. The
entire process can be done regardless of whether you choose
to use the GUI or not.  It can be left as character mode only
and still use the complete OOPS methodology without a GUI.
Personally, I have been using the GUI to get rid of single
keystroke handling.  It is kind of disconcerting to be running
a character mode screen, and have a typical "OK" box jump
up with a message, but it is really kind of neat how it works.

I am having lots of problems trying to come up with uniform
naming conventions, librarying methods, etc., but it is starting
to fall into place, even if my disk looks like a dog's breakfast.

I would be interested in hearing how other are doing.
```
