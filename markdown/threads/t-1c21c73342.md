Greetings,

I'm a manager in a software company which specializes in custom development
tools. For instance, we supply Nothern Telecom with migration tools for
translation of legacy code written in their proprietary languages to ANSI C/C++
and compilers to allow such code to be used on new platforms. Our component
technology allows us to reuse already implemented back-ends, which currently
include Intel x86, Motorola m68k, PowerPC, and C/C++ in new products.

Right now, we are evaluating several directions for our company development,
one of them being creation of migration tools and possibly compilers for widely
used languages, such as Cobol. So my questions are:

1. Would you be interested in a Cobol->C or Cobol->C++ translator, provided
that it produces readable and maintainable code, enabling you to migrate your
existing code to C/C++?

2. Would you be interested in a Cobol complier for one of the mentioned
platforms?

3. Would you be interested in other Cobol-related tools, such as a static
analyzer?

One more thing: we are located in Russia, so we are still a bit cheaper than US
and European companies. :)

With best regards,

If you could actually take an existing unstructured or poorly
structured Cobol program and could translate it into an object
orientated design in 'proper' C++ then I would be impressed, but

a C++ compiler.

In what way is any C or C++ code 'readable and maintainable' ?
In many cases what is clear and well defined in Cobol will have
to be translated to C function calls incorporating additional
parametic data, or to C++ methods. Unless these have <>
effects to the original Cobol then the program may not be useful.

While I can quite understand why it is necessary to translate
'proprietry languages' to something better supported, I fail
to see <> advantage of moving Cobol programs to ancient C
or not-completed C++. Now if it was Java that was the target ...

I suspect that some of the logic that leads to Cobol to C conversion
projects is:
Our C programmers spend 20% (or so) on maintenance
Our Cobol programmers spend 80% on maintenance

Therefore converting Cobol to C will save 60%

If you don't know why this is flawed you should find out.
(I have actually seen managers use this argument).
