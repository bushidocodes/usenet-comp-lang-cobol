You might consider the following a bit 'out in left field', but it has
possibilities. I don't know Server Express but assuming your Version has
OO then it's likely to be very compatible with Net Express. For further
information, post your query on Micro Focus Answer Exchange under "Server
Express' - one of the M/F tecchies could jump in and help you.

You could package your COBOL routines as a series of classes, each class
representing a 'like-minded' set of routines. Other than taking a first
stab at the OO syntax you don't have to get into the gobbledy-gook of
using objects, (if that's a possible fear/reticence you have).  The
method-names of course take you directly to the routine as opposed to
using entry-points.

The only objects necessary to call text based routines (in classes) is,
assuming you had a class OurDates :-

    Working-Storage Section.
    01 os-OurDates                    object-reference value null.

    Method :-
    invoke OurDates "new" returning os-OurDates

Now you have a pointer/object-reference/handle to the routines (methods)
in that COBOL class.

Check out the M/F Examples on-line. There are several dealing with inter
language operability. It seems to me that turing your routines into OO
classes will make it easier to communicate between C and COBOL. (C can
invoke COBOL and COBOL can invoke C).

If any of the above has merit and you want further clarification of what
I'm suggesting, you can e-mail me privately. (But I'll qualify that -
don't look to me for the inter language communication bit - I haven't yet
done it, because as yet I haven't had the need <G>).

You must call cobinit() to get the COBOL run-time started, and you should call
cobexit() or cobtidy() to shut it down. It would be faster and cleaner to do
this once in your application rather than in every wrapper, although doing in
the wrapper will work too. 

When you compile C with cob, it generates that code without being explicitly
told to do so.

We have build something like that. Included is a cobol program which is the
main entry point
for your C library. Also part of a makefile how to build.

The tabs are removed

$Set Ans85
Identification Division.
Program-id. cobmain.
*================================================================
* Purpose : Startup cobol Runtime System
*
*================================================================
/
Environment Division.
Configuration Section.
Source-computer. Goofy.
Object-computer. Goofy.
Special-names.
Decimal-point is comma.
/
Data Division.
Working-storage Section.
01 cmd-program pic x(80).
01 cmd-line pic x(255).
01 ret-code pic S9(08) comp-5.
/
Linkage Section.
/
Procedure Division.
A1-00 Section.
A1-10.
Call cmd-program.
Move return-code to ret-code.
A1-Ex.
Stop run.

Here is a part of the makefile I used to build an rts with linked c-routines

my_program:    < extend withe dependencies needed >
                        $(COB) $(LDFLAGS) -xe cobmain -o $@ < extend with
libraries and objects needed >

Note :

1. $(COB) is de cobol compiler and $(LDFLAGS) = -g, $@=my_program
2. cobmain is the cobol program show above

Usage :

my_program <name of cobol program> ( We use *.gnt for the cobol files )

hope this gives you a hint
