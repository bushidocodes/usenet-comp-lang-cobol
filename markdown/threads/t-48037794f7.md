A GO TO structure can often work well when your code has one step

change.  Code gets inserted here and there, small paragraphs become
large paragraphs.  Exit paragraphs are cut and paste copied (hopefully
with some name changes but I have debugged where they weren't).

small modules.  That's why we so often see the following:

PROCEDURE DIVISION.
0000-MAIN.
    PERFORM 0010-INITIALIZE.
    PERFORM 0100-GET-FIRST.
    PERFORM 1000-READ-LOOP until SW-FILE1-EOF.

    PERFORM 9100-CLOSE-FILES.
    GOBACK.

Everything except the 1000-READ-LOOP can be moved into 0000-MAIN, but
this is very easy to read, and was very easy to write.   This logic gets
copied in each paragraph - we know that each paragraph is basically a
function which only gets run when called by a PERFORM (or you can use
subprograms if you like).  You can add new functions as you like,
without worrying that they will accidentally be called by dropping
down.  And the code is easy to read, easy to debug, and easy to add new
features to.

As soon as you start having labels mean more than one thing, you add
confusion for the maintenance programmer.  When the order of paragraphs
is important, they no longer are stand-alone functions which can be
compartmentalized and debugged in isolation.

One thing which some people use as gospel which I ignore is the mandate
to only read a file at one location in the program.

I find it easy to maintain and easy to understand the following
structure.

   READ MY-FILE
      AT END SET SW-MY-FILE-EOF TO TRUE.
   PERFORM 1000-MAIN-LOOP UNTIL SW-MY-FILE-EOF.

1000-MAIN-LOOP.
   Process data...
   READ MY-FILE
      AT END SET SW-MY-FILE-EOF TO TRUE.

9999-NEXT-PARAGRAPH.

(Replace the read with a perform to make all of your desired status
checks - two performs still get rid of the "at one place" advantage
which advocates like).

The "proper" way of getting rid of the two reads is to put a read at the
start of the loop, but then you have to branch around the not eof
logic.  The concept of priming the loop and then checking at the end
seems natural and puts all checking OUTSIDE of processing logic.

I understand advantages of alternative ways of doing this, but still
prefer this method.  It is a trade off.
