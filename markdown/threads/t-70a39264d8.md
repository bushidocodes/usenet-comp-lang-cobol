Well, I will tell you how it's done at MasterCard, but remember that
paragraph numbering is a religious issue. There are plenty of different
ways to do it, and you preference may be different. Some people advocate
not using paragraph numbers at all, because it eats up character
positions which could be used to make a more meaningful paragraph name.

That being said, here's how MasterCard does it, and I can work
comfortably with these guidelines. Paragraphs are numbered with 4-digit
identifiers. Five-digit groups are allowed, but you should consistently
use either four or five throughout the program. The paragraph number is
immediately followed by an action verb, which is followed by a direct
object. The first paragraph is always named 0000-MAINLINE (or
00000-MAINLINE if you are using five-digit paragraph numbers).

Assuming four-digit paragraph numbers, the 0000- paragraph can only
perform paragraphs 1000-, 2000-, 2000- et cetera. The exception is
paragraphs numbered 8xxx- or 9xxx-, which will be explained a little
later. Paragraph 1000- would only be able to perform paragraphs numbered
1100-, 1200-, 1300- et cetera. A paragraph can only be referenced by ONE
perform statement in the program, and of course GO TO and ALTER are
prohibited.

Paragraphs which need to be performed from multiple places in the
program must be numbered either 8xxx- or 9xxx-. 9xxx- paragraphs are for
abnormal terminations. Also, there is a rule that no perform statement
can call a paragraph with a number lower than the current paragraph's
number. This is intended to enforce a top-down heirarchy.

Therefore, a list of paragraph names in a program might look something
like this:

0000-MAINLINE.
PERFORM 1000-INITIALIZE-THE-PROGRAM.
PERFORM 2000-PROCESS-1-BILLING-RECORD
UNTIL 88-100-ALL-RECORDS-PROCESSED.
PERFORM 3000-TERMINATE-THE-PROGRAM.
GOBACK.

1000-INITIALIZE-THE-PROGRAM.
PERFORM 1100-VALIDATE-CONTROL-CARD.
PERFORM 1200-OPEN-THE-FILES.
PERFORM 8000-GET-NEXT-BILLING-RECORD.

1100-VALIDATE-CONTROL-CARD.
PERFORM 1110-READ-CONTROL-CARD.
PERFORM 1120-EDIT-CONTROL-CARD.

1110-READ-CONTROL-CARD.
PERFORM 9000-ABEND-THE-PROGRAM. *> IF A READ ERROR OCCURRED

1120-EDIT-CONTROL-CARD.
PERFORM 9000-ABEND-THE-PROGRAM *> IF AN EDIT ERROR OCCURRED

1200-OPEN-THE-FILES.
PERFORM 9000-ABEND-THE-PROGRAM *> IF AN OPEN ERROR OCCURRED

2000-PROCESS-1-BILLING-RECORD.
PERFORM 2100-CALCULATE-BILLABLE-AMT.
PERFORM 2200-PRINT-BILLING-REPORT.
PERFORM 8000-GET-NEXT-BILLING-RECORD.

2200-PRINT-BILLING-REPORT.
PERFORM 2210-PRINT-REPORT-HEADER. *> WHEN IT'S NEEDED

3000-TERMINATE-THE-PROGRAM.
PERFORM 3100-CLOSE-THE-FILES.
PERFORM 3200-DISPLAY-FINAL-MESSAGES.

8000-GET-NEXT-BILLING-RECORD.
PERFORM 9000-ABEND-THE-PROGRAM. *> IF A READ ERROR OCCURRED

9000-ABEND-THE-PROGRAM.

Obviously, a lot has been eliminated, in order to just show you the
paragraph naming scheme. Also, there are other shop standards for
paragraph names which are intended to make the program easier to read.
The example is for a batch program. A CICS program would still follow
this basic numbering scheme, but would do different things.

I am personally comfortable with coding a program this way, but
reasonable people will disagree about the best way to number paragraphs,
or even whether to number them at all. These standards are fairly easy
to read in the cross-reference listing. You should consider a number of
different numbering schemes and then pick one that seems to work best
for you or for your shop.

I hope this helps!

Although the subject can be greatly elaborated, the basic rules are:
- use 3- or 4- digit prefixes
- each paragraph has a UNIQUE prefix
- prefixes are in ascending order

Generally, sections (or major control structures, since sections are
becoming passe) cause a jump in numbering, so:

000-MAINLINE.
...
PERFORM 100-INITIALIZE
PERFORM 200-PROCESS
PERFORM 900-SHUTDOWN.

etc.

The reasons behind all this are: to add a bit of clarity to the design
phase; and to enable the miserable maintenance programmer to navigate
the program quickly and intuitively.

The one I use since college is Welburn's system. In is book "Advanced
Structured COBOL" page 12, he uses a structured three-digit system as
follows:

000: Mainline
100-199: Initialization of variable fields
200-699: General processing
700-799: End-of-run totals, statistics, etc.
800-849: Input (READ, ACCEPT)
850-869: General nonreport output (WRITE, REWRITE, DELETE, DISPLAY)
870-879: Report headings
880-889: Report top-line output (page skipping)
890-899: Report-line output (line spacing)
900-999: Subprogram communication (CALL) and abnormal program
termination

Mind you, this is designed for batch programming. With a little
thought, it can adapt to interactive environments as well.

I use a method that really works for me. By looking at the paragraph
number, I can tell where it is in the program listing, as well as
which paragraph calls it. Paragraphs called from multiple places in
the program are exceptions, so I use a different number for those.

First, I use a strictly structured approach--Control paragraph first,
which calls sub-paragraphs below. Sub-paragraphs may be control
paragraphs which call others below that.

The main control paragraph is 0000-main. It calls 1000-setup,
2000-big-process, and 3000-completion.

2000-big-process calls 2100-, 2200-, etc. 2200- calls 2210-, 2220-,
etc.

Common routines called from various places are placed below their
calling group. Example: 2701- and 2702- might be common routines
called from various 2xxx- paragraphs.

If paragraphs are nested deeper than four levels, I start another
group of numbers: 2212- might call 2212-100, which might 2212-110 and
2212-120.

All the paragraphs fall into numeric sequence in the program, and you
end up with a tree structure that is really easy to follow.

I have written 10k line programs, and can pick one up any time,
randomly pick a spot in the Procedure Division, and know what the
structure is of that particular part of the program. Saves me lots of
time, and doesn't require much time when writing a new program.

Only one problem-- nobody else seems to want to use the method, either
because they don't buy the structured programming concept, or can't
"see" the value of the parent-child-grandchild-... relationships.
