```
Victor,

if you used COBOL II or COBOL for OS/390 and VM, and used the compile time option RENT,
then your COBOL program is reentrant.

If you want to know if any old COBOL module you found in a library is reentrant - nothing
I can remember.  But, of course, you can always let the hardware figure it out.  If it is
a batch program, coax your sysprog to place it into LPA (Unprobable, though, that you will
succeed).  For a CICS program, link the program with the RENT attribute, specify
RENTPGM=PROTECT in the SIT of your test CICS, and off you go.  If the program is not
reentrant, it will abend on you
```

```
Given the IBM mainframe nature - and release of your product, then the answer
is (as others have stated),

COMPILE and LINK-EDIT with  the RENT compiler and link-edit options and your
code is "guaranteed" to be RENT.

If you want to check the link-edit attribute of an existing module, you can
use a number of tools (already mentioned in the thread).  HOWEVER, as also
stated elsewhere, you can link-edit a module as RENT even if the code is not
correctly coded to be rent (e.g. an OS/VS COBOL module or an IBM COBOL for
OS/390 module compiled with NORENT). Therefore, the link-edit attribute only
tells what the linkage editor (or binder) was told - not how the code
actually exists.

Does this help?
```

```
A (very old) OS/VS COBOL program is not reentrant. A newer one is
reentrant if:
- It is compiled with the option RENT
- It is linked with the option RENT
- It only calls subprograms that are reentrant
You might want to analize your COBOL loadmodules using FILE321 from CBT
tape:
http://www.cbttape.org/cbtdowns.htm
1. Download the file
2. unzip it
3. transfer it to the OS/390 mainframe (binary!)
4. TSO RECEIVE INDATASET(xxxxxxx) where xxxxxxx is the name of the
transferred file.
5. press ENTER
6. Assemble and link the program
```
