A new COBOL subroutine (batch) is written and accesses MVS DB/2 tables. 
The calling program has no SQL and calls the subroutine dynamically.  The
subroutine compiles and binds ok, but now the calling program must be
executed via IKJEFT01 with the RUN PROGRAM/PLAN statement.  If the calling
program is compiled as a DB/2 program, the compile issues errors because
there are no SQL statements and no package is created.  So the question is:
how can the main program be compiled to create a DBRM to bind?  A simple
SQL statement can be added, but that seems hokey.
Suggestions?

Doug McKibbin

Doug, yes, adding an SQL stmt to get your compile through is hokey.
Precompiling a program with no SQL stmts is only a level 4 (a warning). But
binding an empty dbrm into a package is an error. If your bind step was at
the end of your compile jcl then it wouldn't matter. This is the cheap and
cheerful approach.

However, you could just have separate jcl for compiling a program with no SQL
in it. Then you wouldn't be doing a bind which fails.

At run time, it is no big deal if your main (calling program), say MAIN has no
SQL in it and the called module, say SUB, has SQL.

You just BIND the package for SUB and do a once off BIND PLAN (MAIN) PKLIST
(<collid>.SUB).

Then RUN PLAN (MAIN) PROGRAM (MAIN)

Note that I have called the plan MAIN just because that is the name of the
loadmodule we're running, which distinguishes this execution of SUB from
executions of other loadmodules which may call SUB. This is useful for
performance monitoring, when looking at accounting traces from DB2PM etc.

Hope this is of some help
Stephen Goss
