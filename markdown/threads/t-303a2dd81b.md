Would someone briefly describe the differences between a file defined as
Record Sequential Line Advancing and a file defined as Line Sequential.

The platforms are all Unix: AIX, HP, Alpha/OSF1, Solaris.

The major difference that I have observed is that one can get imbedded
tabs in the disk version of the LS file. From the COBOL program view of the
world, this is tranparent. However, the semantics of some of the I/O
instructions are annoyingly different. In particular, REWRITE fails if
the disk version of the rewritten record is larger than then disk version
of the original record. RSLA does not have any such restriction.

Before I rediscover more LS trivia, I thought I'd ask the group at large
to see if any Micro-Focus COBOL users (or Micro-Focus themselves) have
documented the differences.

Jim
```

Its documented in the COBOL System Reference, Appendix "File Formats"
in the V3.2 docs I have.

In essence these are two types of file, one destined for the printer
and the other "compatible with the system editor" (ie. format of a
source file). In a printer file you can have write BEFORE/AFTER x
where x is a number of lines (with 0 for overprinting), page throws,
vertical tabs, etc. If you don't have a BEFORE/AFTER on a WRITE
ANSI says it should behave as if AFTER 1 were specified. All this
stuff is o/s independant and the resulting file will be the same for
say UNIX and DOS.

Line sequential files have a record terminator (which is o/s dependant)
ie. an implicit BEFORE 1 compared to printer files and so going against
the ANSI spec for print files. Perhaps more significant in practice
is the fact that if you use an AFTER 1 followed by a BEFORE 1 in a
line sequential file you will get a record terminator, two data records
and another terminator. When you read this back you will just see
and empty record followed by a double length one. Ugh!

With the printer file, each record is terminated with a carriage
return with line feed or other control characters added as appropriate
so you can recover each record when reading the file back.

The two types of file can be very similar in simple cases, but the
differences can come back and bite if you mix them up.

For both types of file, changing the length of a record with a REWRITE
doesn't sound like a very safe thing to do.
```
