```
Your override may be generating a compile return code of 4 and the link
edit step may execute only when the return code is zero.  Things go down
hill from there.  The FASTSRT option only takes effect for files in
USING or GIVING statements.  There should not be a status code
associated with those files since the SORT is doing the I-O, not code
generated by the compiler nor calls generated by the compiler.  If you
want to do status code checking then an INPUT PROCEDURE should replace
the USING statement (if any) and an OUTPUT PROCEDURE should replace the
GIVING statement (if any).  In my previous shops the FASTSRT option has
been the default.  If politically feasible you might recommend that the
appropriate group change the default compile option and possibly also
accept a compile return code of 4.  You also may not have other options
such as OPTIMIZE turned on because of the return code problem.  I think
a PERFORM of a paragraph with an always executed STOP RUN statement may
cause a return code of 4.  I know a warning message is generated.
```

```
excessive disk IO

    The optional sort is a more traditional sort, and uses much
less disk.
```

```
I am just starting to catch up on my reading of C.L.C, but in case no one has
mentioned it, there is one VERY OBSCURE problem with always using FASTSRT -
when applicable.

If your program (as unlikely as this sounds) has a file DECLARATIVE for a
file used as either the USING or GIVING file in a SORT (or MERGE) *and* if
you actually expect to go to that declarative if there is a problem with the
file, then FASTSRT stops this from happening.  If you use NOFASTSRT, you will
go to the DECLARATIVE, otherwise you won't.

I *strongly* recommend that all shops set FASTSRT as the shop default.
```

```
You wrote: "I am surprised the compiler & sort utility
don't just pick the most efficient sort option."  Well,
the sort utility has no choice in the matter when NO-
FASTSRT is in effect: the COBOL program passes
the address of an "E35 user exit" for output record
handling -- so that it is COBOL that actually does the
file writing.  When FASTSRT is in effect the SORT
utility does the creation -- usually much more quickly.
The only reason that I can think of for not using
FASTSRT is that standard COBOL file status codes
would be generated -- that is, if one were coding in
a defensive posture, anticipating problems, and want-
ed to process them within a COBOL context.
```

```
Howard,

The benefit of "FASTSRT" is that all SORT I/O is performed External to
the COBOL program, whereas, INPUT PROCEDURE and OUTPUT PROCEDURE use
COBOL I/O, which is somewhat slower.

Also, "FASTSRT" does not always need to use a USING/GIVING. You can
have an INPUT PROCEDURE/GIVING or a USING/OUTPUT PROCEDURE and still
realize savings and efficiency gains.
```
