Noticed this bug/feature in tiny-cobol 0.57 yesterday when 
trying to recompile a cobol program written about 10-15 
years ago.

It seems that in the source code the C language type 'char' is 
being promoted to an 'int'. (Or something like that. Anyway, 
what should be ONE ascii byte is being interpreted to be TWO
ascii bytes. Even if we assume that we're trying to include
the ability in Tiny-Cobol to use unicode, it is not performing
consistently at all.

I have condensed the  original cobol program  code  into the
code below in order to demonstrate the incorrect behaviour.

This is actually off-topic for this newsgroup and I haven't
had a chance to look at the Tiny-Cobol source code myself yet,
but I was wondering if any of you guys had seen this before
and had a fix for it.

jvstrangio@ozemail.com.au (Jack Strangio) writes:

> 000011	 01  INPUT-STRING.
> 000012	     02  CHAR        PIC X(1), OCCURS 8 TIMES.


As David Essex and Bill Klein emailed me, it was the
comma at the end of the "PIC X(1), " which seems to have caused
the problem. They stated that the line should have read

 000012             02  CHAR        PIC X(1) OCCURS 8 TIMES.


As I wrote to David, my recollection was that the comma was more
or less optional.

I used to put it in because it seemed  "tidier" to me. After the
emails, I rummaged through my bookshelf and hauled out the software
manual. I found that my memory actually had been correct regarding the
comma, but that the syntax sequence within the line was wrong. It should
have read something like this:

 000012             02  CHAR, OCCURS 8 TIMES	PIC X.


Ah, well......


Thanks, guys. 
(Perhaps I should apologise to the Tiny-Cobol folks. :-)    )

Jack
