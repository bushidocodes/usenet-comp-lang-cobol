Step 1 - READ the IBM migration guide that explains all of this in detail.

Step 2 - Do *not* recompile all (or even most) of your VS COBOL II programs.
There is absolutely no reason to do so - as IBM *fully* supports VS COBOL II
object code when used (including link-edited) with the LE run-time libraries

Step 3 - If you are not already using the LE run-time libraries for your VS
COBOL II modules, then plan this as your FIRST migration step - and plan on
LOTS of testing and tuning time.  (The change in compilers has minimal
impact - the change in run-times is SIGNIFICANT - with lots of "unplanned"
side-effects.)

I must be missing something here.  There is ABSOLUTELY no rule that you can
mix (and match) any number of dynamic and static CALLs between VS COBOL II
and COBOL for this-and-that compiled routines.

There *is* a rule that says that you cannot mix NORES VS COBOL II programs
with COBOL for this-and-that routines (where NORES is no longer available).
(Actually, there are some fairly complex "re-linking" and run-time options
that allow you to get away with this - but those rules are truly obscure.)

Therefore, if you have
   Module A (compiled with EITHER VS COBOL II (compiled with RES) or COBOL
for this-and-that) that calls Module B (this call may be either dynamic or
static),
  Then Module B (compiled with COBOL for this-and-that) makes EITHER a
dynamic or static call to:
  Module C (compiled with VS COBOL II - with the RES compiler option),

Then this absolutely, positively should NOT ABEND (or cause other problems).

P.S.  There is one other "potential" problem - but this has been true since
VS COBOL II first came out.  If you are passing data via a CALL USING
statements to a 24-bit (AMODE/RMODE(24)) module, then you need to compile the
calling program with the DATA(24) compiler option.  However, changing from a
static to dynamic CALL wouldn't solve this problem (if you had it), it would
actually make it worse.  My *best* guess is that you have some VS COBOL II
programs compiled with NORES.  This will cause you MANY problems as you move
to an LE and COBOL for this-and-that environment.  Check out the discussions
on this topic in the migration guides.

> Can anyone list the basic differences between OS/390 and VS COBOL II
> compilers?

COBOL for OS/390 & VM includes intrinsic functions and OO support not
available in VS COBOL II.  Otherwise, VS COBOL II syntax is completely
compatible with COBOL for OS/390 & VM, including PROCESS/CBL compiler
directives (except for NORES, which is no longer available).  But the
biggest difference is COBOL for OS/390 & VM requires LE/MVS.

The LE/MVS libraries include all the ILB* and IGZ* runtime modules to
support OS/VS COBOL and VS COBOL II, although some of those modules are
different from the originals.  When you compile with COBOL for OS/390 &
VM, you will be calling CEE* runtime modules in the LE/MVS library. 
LE/MVS runs predominantly above the line, so 31-bit issues will become
important.  

> My IS shop is switching over to OS/390 compiler in 3rd Quarter of this
> year.
> <snip>
> 
> Are there any other potential problems that we should be aware of???

My shop recently began converting to COBOL for MVS & VM.  As recently as
1998, we were converting OS/VS COBOL to VS COBOL II.  We elected to
convert gradually.  As programs are recompiled for normal maintenance
they are compiled with the newest compiler.  Our system programmers made
the Endevor batch COBOL defaults include DATA(31),RMODE(ANY), and
DYNAM.  These can be overridden using PROCESS/CBL compiler directives. 
Since our compiles and object modules are managed by Endevor, no other
way is allowed to change compile parms.  The SP's would prefer that our
batch COBOL run above the 16-meg line.

Recently, we have experienced some problems with older assembler
subprograms that were last assembled before MVS/XA and know nothing
about memory above the line.  Our system programmers tell us that the
older assembler programs must be modified to support 31-bit addresses in
the save area chain to be compatible with LE/MVS, even if they remain
AMODE(24),RMODE(24).

Converting older assembler programs from AMODE(24) to AMODE(31) can be
tricky.
