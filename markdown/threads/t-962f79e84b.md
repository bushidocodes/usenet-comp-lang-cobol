Pardon me Erik, but as a programmer who has written several million

Not that I am against source formatters, I just think your reason for
wanting one is flawed.  To be done well, programming requires as much
meticulous precision and attention to detail as any human activity
there is.  In over 30 years programming, I have never seen a  single
programmer who wrote sloppy code that was anywhere near 'good'.  One
of the hallmarks of the very best programmers is that their code is so
consistent that it looks like it came from a machine.  Formatting is
almost as important to them as code correctness, because they know,
even if instinctively, that the discipline and practice which leads to
the one leads also to the other.  They are inseparable.  The reason is
very simple, when you understand human nature.  For the same reason
that you never see a person who is meticulously honest yet dishonest at
the same, you never see a person who is both meticulously accurate and
sloppy at the same time.  If you want to be a 'good' programmer, you
will devote as much attention to every little detail of the process as
you are able to muster.  It is simply part of the discipline and mindset
which prepares a naturally error prone human being to work with high
precision and thoroughness. :-)

The first type of copying is selecting a program similar to the one
I want to write and modifying it.  How much of the code gets modified
depends on the programs, of course.  In general, when starting a new
system you may not get much benefit, because you are using programs
written for other systems.  As more and more programs are written for
that system, you can find more and closer programs to use.  These
are all programs I wrote originally.

The second type is copying a piece of code from another program or
another part of the program I'm writing.

The third type of copying is not code at all, but copying data names
or parts of data names from database definition files or other parts
of a program to construct data definitions or instructions, often
series of moves.  This is as effective for precision as it is for
efficiency.

input data validation to be done in online programs by performing
dictionary edit routines.  Many large online programs I write have
sections of code that contain many (perhaps 10-50) sets like this:

      MOVE "data-name" TO hold-name.
      MOVE data-value  TO hold-value.
      PERFORM dictionary-edit
      IF  error
          ...

I copy the field names from the program or database definition to
create an ASCII file of the field names and a few parameters, and
run a BASIC program I wrote which generates that code.  Virtually
guaranteed to be bug-free.

I'm sure virtually all COBOL programmers use at least some of these methods.  The result is a very significant increase in
productivity,
