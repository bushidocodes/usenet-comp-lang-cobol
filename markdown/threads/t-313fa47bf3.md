I don't expect a text editor to do error parsing. I use a single-keystroke or
mouseclick function that will a) save the source file b) shell to the compiler
and c) capture compile errors into a memory 'document'. That allows me to
correct errors without leaving the editor. 

Any GOOD editor will let you bind keys to functions to simulate Brief. A
SUPERIOR editor lets you do a 'hot grep', so you can see all occurrances of a
variable both in and out of context. That's faster than Next, Next, Next.

When I'm not using my own editor, I use Ultra-Edit. It lets me edit files on
another machine by automatically FTPing them.

I am guessing from Stephen reference to Brief and also his mention
of two very famous brief clones, that by "error parsing" he means 
the ability to:

    a) Captured the error output as you describe above

    b) Have the editor parse the lines of output so that if the user 
       clicks on a line of error the editor will move to the correct 
       file at the correct line number.

    c) Have the ability to move to the next or previous error with 
       a single key stroke without having to go back to the original
       output window.

These are the "error parsing" features found in the original Brief 
editor and once you are trained to work the "Brief way" is very 
hard work any other way.
