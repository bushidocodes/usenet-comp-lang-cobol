I've read the comments here wondering whether OO on OS/390 is viable
and/or recommended.  I've also been having private discussions with

During my research into this subject, I learned something about
collection classes.  Collection classes are a set of routines designed
to let you manage multiple instances of objects, such as "customers"
or other things.  This is a very useful thing, but something that is
decidedly not standard.  NetExpress has them, and Fujitsu has them -
with very different implementation.  With the next Fujtisu release,
the Class Collection Library will only be available with the
"Enterprise Edition" of Fujitsu COBOL.  It's really the only extra
feature of the Enterprise Edition that I really desire.  Of course,
that's the idea. I'm going to have a hard time coughing up $3000.00 to
get a class collection library, however.  So, I'll stick with my
"Standard" version.  This lead me to the realization that Jimmy is
right.  OO really does help with sharing code and re-use.  It helps to
the point where there can "easily" be created an Open Source set of
common class libraries.  We can all share them.  

Realizing this, I decided to send Jimmy my code to see if it will
compile under Merant's Net Express.  I didn't use any features of
Fujitsu that were not in the 1.8 draft standard.  I should note that
Fujitsu is very close to the standard with their 5.0 (beta) product.
The lack only a few minor modifyers for Property behavior and the
whole "INTERFACE-ID" and items related to that.  

At the same time I sent the code off to Jimmy, I uploaded it to our
OS/390 2.7 box, running COBOL for OS/390 and VM version 2.1.  

I don't yet know wich version of NetExpress Jimmy was running, but I
was under the impression it was the latest - I would really like to
hear from some Merant people on the issues I am about to discuss.  

NE does not understand "Properties".  Properties are special (declared
as properties) elementary items in Working-Storage of an object.  They
can be retrieved form the object using a "GET" method, and set to
values in objects uisng a "SET" method.  This means that data items in
an object can be manipulated with these methods, and need not be
passed to the class using parms when a method is invoked.  Now, this
is REALLY attractive to me.  It's true encapsulation, and it's how
ActiveX controls work.  You set properties, and invoke methods with
ActiveX -- it's REALLY OO sharing.  At any rate, when using Parms, you
actually have 2 copies of the data - the one you pass when invoking a
method and the one the object has under it's control.  The whole tenet
of OO is that the data is protected within the object - only methods
of the object can change the data.  So, I think you see why I really
like them.  You can see their use in my attached programs.

Well, the version of NE that Mr Gavan has doesn't know what a property
is.  In addition, it requires the Factory to be defined for a class,
even if nothing is in it -- since I don't use the Factory in my
example (Although Donald Tees did explain how I could have done
EVERYTHING this program does in a Factory, avoiding the necessity of
creating the object first with "new") I didn't include the Factory in
my code.  NE choked on this, and on the OBJECT paragraph and on
Properties.  

I had higher hopes for OS/390.  But alas, the manuals don't mention
Property, Object paragraph or even the FACTORY!  

OO on OS/390 is a small subset of the total specification in the draft
standard - and that is the primary reason in my mind that it's not
quite ready for prime time.  

The vast differences in these 3 implementations (Does anyone know of
any other OO COBOL implementations?) makes it virtually impossible for
those of us experimenting with OO to share code.  The sooner the
standard is approved - and implemented - the better.

In addition, Merant seems to have some pecular dependancies in the
invoking program that don't seem to exist in OS/390 or Fujitsu.  Jimmy
keeps telling me about a "Trigger" program - that's what I thought my
mainline was!

Anyway, this is kind of just a status report.  What follows is my
code.  This class that I have started is designed ot take a decimal

EXTREMELY efficient algorithm for this process, not the normal "Brute

hit.  This isn't my "time tracking" project, but the project I am
using this for is kind of special, and is taking priority.

The algorithm I used is described in full here:

I wanted to explain a bit more about Properties.  The Moves above are
the same as saying:

Move 3.14159 to Parm-Area
Invoke Fraction-Object "Set" using Parm-Area

It's just an inline shortcut.

The move for the Display later is the same as:

Invoke Fraction-Object "Get" returning whatever.

The use of the qualification of the object causes an implicit get or
set method to be executed.

I wanted to say one quick think about the OO implementation in IBM COBOL for
OS/390 & VM.  It was a "reasonable" subset of the OO definition in the draft
Standard - circa 1997/8.  The draft has moved DRAMATICALLY since then - and
IBM's implementation has NOT.  It is my understanding (totally UNOFFICIALLY)
that IBM is pretty well committed to NOT changing their IBM syntax UNTIL the
Standard is either "approved" or at least an approved "final committee draft"
(after which no "technical changes" are supposed to occur).

the OO syntax that they originally introduced.  This means that they are both
CLOSER to the current draft (and which ever vendor's compiler is "most
recent" is most similar to the MOVING draft).
