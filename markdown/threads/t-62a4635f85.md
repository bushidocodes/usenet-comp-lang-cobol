Gerd Nachtsheim has posted articulate expressions of interest in comparisons of
COBOL and Java.

In truth it is just such expressions that are the kernel that start a real
techincal effort, but I fear you are one of the very first. And it just
reflects back to you.

I would say that there are three kinds of comparisons you could do. 

First you could hunt like Diagonese for the folks who actually know how to use
the modern object oriented capabilities in modern COBOL. Then conceive of a
competition and code it in compiled COBOL and JITted Java. You have to make
certain that such a comparison really does involve major class structuring and
exercisies all relevant technical issues like the speed of comparable instance
methods and the speed of class methods. You must make certain to layout a plan
that fully engages polymorphism and also method overriding to see if the whole
addressing scheme for dereferencing the correct method pointer has comparable
performance.  No small plan I would think. But if the new COBOLese is for real
then you ought to be able to get some of the techno wizardry compared.


Second you could conceive of a compare of somekind of supposedly 'equivalent'
algorithms and deploy them with Java and with  flat COBOL using no new OOP
syntax. This will get you deep into the argument about what is comparable. (see
next note).

The third kind of comparison would be to just take the supposedly 'equivalent'
algolrithms and deploy them with new COBOL using OOP syntaxt and compare to a
deployment with flat COBOL using no new OOP syntax.  You will be deep and
headfirst in the morass at this stage.


There is a need to establish terms. New COBOL has Object Oriented Programming
syntax features. New COBOL still has all of the old stuff, too. You can use
COBOL either the old way, the new way, and very very importantly you can mix
(as of right now interoperability is the most important business aspect of new
COBOL: that applies to interoperability with 
C_yada_yada and Java, as well as flat COBOL interoperating with new COBOL).

If you are honest about the total set of comparands here, you must include new
COBOL compared to old COBOL..

The real issues are Object Oriented Design or not Object Oriented Design, and
to use Object Orient Programming tools (such as new COBOL, _or_ Java) or to not
use them.

The comparisons you call for are legitimate, but the expression Java vs COBOL
is not fully encompassing of the challenge at hand.

In my opinion you will find all object oriented deployment slower than
flatlander systems. A person could be way wrong about that. Honestly I have
oftern been surprsed, for example, at how well COBOL itself stands up in
comparisons on identical platforms to other compiled code.

But my hunch, and it would be interesting to see it really seriously disproved,
is that OOP tools are not here because they make machines efficient.  This is a
stuck record, not because I like to bore people, but truthfully because I want
to help folks.  OOP tools are here because they make people efficient. In
particualr OOP tools make data processing professionals efficient - that
contention is also a worthy subject of professional metrics. But not to
displace Gerd's extremely good idea.

We should compare! The comparands are not just simply COBOL and Java; the
comparands are more like: flat COBOL, OO COBOL, Java, flat C, C++/C#. The
comparisons will not be easy. They are worthy of discussion.

But the real point is to build _down_ the walls of division. Meaningful
comparisons will have to move towards measuring interoperable environments as
well. These issue are major. But comparison work is most useful if it help
guide folks on how to use these wonderful tools _together_.

Interesting you would say this. In 1976/77 while I was in 
college, it was decided to do a benchmark of COBOL, PL/1, 
FORTRAN IV, ALC and RPGII (using a S/360-30 running DOS). It was 
agreed on that all three would be used to solve the infamous 
Indian Problem that was given to each class (Assume the Indians 
who sold NY to the settlers had put the $12.00 in the bank for x 
years at y interest rate compounded annually....)

It was found that the problem was solved by RPGII faster than 
any of the others could do it! It took fewer statements to do 
than did FORTRAN IV! The longest compile was PL/1. ALC took more 
statements to do it than did COBOL! Note that each program was 
written by someone who loved that particular language and really 
knew its ins and outs.

Basically, the RPGII program compiled, linked and ran in the 
time it took the ALC program to assemble! The ALC program ran in 
the least time. The FORTRAN program compiled, linked and 
executed faster than the COBOL compile. PL/1 was cancelled 
before it finished compiling as that took longer than the COBOL 
program did to compile, link and run!

What was measured in all cases was the amount of time it took to 
write the program (assumed the program was written correctly the 
first time as all debugging and syntax error fixes were 
ignored). Then the compile/assembly time was measured, and 
finally the run times were compared.

Remember, that in those days a programmer's time was much 
cheaper than machine time, which is why those measurements were 
done in that way.

So back to the JAVA v. COBOL benchmark idea: Defining a set 
problem and looking to see how long it takes to develop the 
logic in one language over another is an interesting item.

The next most interesting item is seeing which language produces 
the fastest results.

Now in this last benchmark (and this will show my ignorance), 
does the JAVA JIT run on the server or the client? Does it have 
to be shipped to the client to run? Will this time have to be 
taken into account when comparing the COBOL runtime results? 
That is, will there be a network latency that will be realized 
by JAVA that won't be realized by COBOL (or vice versa)?

A true wholistic comparison may show the one environment to be 
much better than the other. It was comparisons like this that 
resulted in shops deciding to NOT use PL/1 or FORTRAN in years 
gone by (RPG for some reason was not liked or used by many 
programmers because of its fixed logic, fixed format 
requirements), but to use COBOL.
