Anyone come across this one before?
This is happening in a test online system I am setting up. The IMS
databases, DBD source, PSB source and COBOL code have all been copied from
production.
Needless to say the production system is working fine but the test system
is having problems.

CALL 'CBLTDLI' USING GU
MTE-PCB-1
MTE-SSA1
MTE-SHOP-IO
MTE-SSA1
MTE-SSA6

Before the call MTE-SSA1 looks like this
TDSMTERT*--(TDEMTEKY= DUPX025029)TDEMTEKY<=.........) - the dots are high

gets ignored.

...TDEMTERT*--(TDEMTEKY= DUPX025029)TDEMTEKY<=........

MTE-SSA1 has been moved 3 bytes to the right and truncated at the end.
This causes subsequent calls to abend with IMS errors, because the fields
in the SSA are now out of whack. MTE-SSA1 references a root segment which
contains a twin backward pointer, which may or may not be causing the
problem.
I've run DFSDDLTO against the database and the output looks fine.

This is a little bit of a gotcha as well.
The copybook for MTE-SHOP-ID is 2401 bytes long but the segment in the IMS
database is 2500 bytes long. So it was just a matter of adding the extra
99 bytes to the end of the copybook and recompiling all of the sub modules
and the main module.
Although production source and test source match, the inference is that the
program is working in production because someone had compiled it using
their own copybook source, which was correct rather than the production
source which is wrong.

Check your IMS call.

The format should be:

CALL 'CBLTDLI' USING
function-code,
pcb,
i-o-area,
ssa1, ssa2, etc.

The way your call is setup, IMS will think that MTE-SSA1 is your IO area
and that is where
it will return the first segment (trashing MTE-SSA-1's contents).
IMS would think that MTE-SHOP-IO is an SSA.
MTE-SHOP-IO is probably spaces or LV before the call, so IMS will think
you doing an
"unqualified SSA"-type call.
