2. Procedures :-
- open files
- read input records
- validate your incoming records to see if they are 'good' records
- write 'good' records to the OutFile
- what are you supposed to do with the bad records ?
- now close files
-  sort your 'good' OutFile.

Note that you can mix USING...GIVING with RELEASE..RETURN.. so you could
have something like..

SORT infile  ON ASCENDING KEY keyname
  INPUT PROCEDURE IS  procedure-name
  GIVING  outfile

Me, I'd just use INPUT and OUTPUT procedures with something like...

SORT infile ON ASCENDING KEY keyspecs
 INPUT PROCEDURE   IS input-Procedure-name
 OUTPUT PROCEDURE is output-procedure-name

INPUT-PROCEDURE-NAME.
   (get record from input, qualify for good-bad)
   IF Good-record
      RELEASE to sort
  ELSE
     WRITE BadFile-Record
  END-IF
 ...

OUTPUT-PROCEDURE-NAME

RETURN good-record
  you will get good records in sorted order here.. and you can do whatever
you want with them


This does not provide the bad-records in sorted order (left as an exercise),
but USING..GIVING looks like a good way to handle those.


MCM

In the mainframe environment i've seen this a lot: smaller programs, many
job steps. My personal preference is for fewer job steps, which of course
means more complex programs.

When I first started working in mainframe environments (1994-95) I was
somewhat overwhelmed by JCL, so maybe that was a natural reaction, but even
as I got proficient with JCL, I never changed. I still like fewer job steps.

As I understand it, the main reason for the predominance of the 'many small
steps' method is to facilitate restart .. no sense restarting a multi-hour
batch job from the beginning if the first two hours' work were already
completed successfully. But between better hardware (fewer errors and faster
speeds) and better software (we learned new techniques to speed up
processing), this consideration recedes. Today the reason is likely twofold:
first, ease of program maintenance by contractors and others less familiar
with existing systems, and that old standby, inertia ("we've always done it
that way.").

You are overlooking arguably the most important factor.  Job scheduling.

If you have one job with many steps which requires multiple databases
to be open in exclusive update mode, then such a job can play havoc
with the remainder of the production batch schedule, as there will normally be
many other jobs which could run concurrently with a better overall usage plan.

Many other batch jobs which could be run (if the behemoth were broken up)
have to sit in the queue until Godzilla completes successfully, especially if
earlier steps and later steps both demand exclusive access to widely used databases.

Another important factor is that smaller existing production jobs can be more easily
integrated into special runs, such as at month-end, fiscal year-end, calendar year-end
and also during conversion projects, when you might only wish to run certain steps.

       01  sorted-account-record.
