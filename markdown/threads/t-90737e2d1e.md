```
I am running Microfocus Advanced Workbench under Windows NT 4.0

I am trying to find a way of  calling native DOS functions from within my COBOL
program.  I have been successful in using the Microfocus Library Call-Routines,
but some functions are not available.  For example, mapping a network drive,
doing a "dir" command and piping the results to a file, or changing the file
attributes(attrib).  

Does anyone have an example of  (or could point me in the direction to where I
may find information about) how to do this.  Can this be accomplished using the
WINAPI?  Could I create a "C" routine that I could call from my COBOL program
that would carry out these functions?  Can I write a COBOL routine that would
call a WINAPI (call-convention ??) to process the commands?  At the very least
I would like to suppress the DOS prompt from appearing.

Currently I am using the DISPLAY UPON COMMAND- LINE verbiage to accomplish the
task.  However, this works best if you are in a 16-bit environment; because if
you use this under NT 4.0, you get a DOS prompt box, or if you have multiple
DOS commands, you get a cascade of DOS prompt boxes. The prompts disappear
after they complete, but it's not esthetically pleasing.

Any comments or suggestions are welcome.

Thanks!
```

```
You can also call WinExec, passing it the full path to COMMAND.COM, and
providing the /C command tail, which will fire up COMMAND.COM and tell it to
run the command appended to the /C.  One may venture that this is a more
"standard" method than using a Micro Focus call-by-number routine...

For example, to use a DOS command to format a floppy:

STRING
    'COMMAND.COM /CFORMAT A:'
    LOW-VALUE DELIMITED SIZE
        INTO WIN-COMMAND

CALL WINAPI 'WinExec' USING BY REFERENCE WIN-COMMAND
        BY CONTENT CMD-SHOW
        GIVING HINSTANCE.

CMD-SHOW specifies how the command window is to be run, e.g., minimized,
maximized, hidden, etc., and HINSTANCE is the instance handle of the spawned
process, which runs asynchronously.

FWIW, in our ScreenIO package, we're gradually bundling up and exporting
useful Windows APIs in a COBOL-friendly sort of way.  You just call our
entry point (in this example, it would be SCR_WINEXEC), passing the
arguments in the standard COBOL form, and we take care of rendering them to
the BY VALUE, BY REFERENCE form needed for the native C form call used by
Wndows.  This is much easier than wading through the C examples found in the
Windows SDK and trying to figure out how to translate them to COBOL

So, the call would be:

CALL 'SCR_WINEXEC' USING WIN-COMMAND
        CMD-SHOW
        HINSTANCE.
```

```
You can also make normal WIN32 API calls from COBOL.  It is just a matter of
using a calling-convention.  I believe the appropriate one is 66 like:

            call-convention 66 is WINAPI.

Then you can make most any WIN32 call that you want like:

            call WINAPI "SetForegroundWindow" using
                                              by value anId
                                    returning anOldId

The only trick is translating the 'C' data types to cobol.  Email me if you
decide to choose this route and I can send you a list of the translated data
types or you can use the MicroFocus H2CPY command on a 'C' .h file and it will
give you the most commonly used ones.
```
