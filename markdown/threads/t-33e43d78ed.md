software as PROGRESS provides?

Progress is VERY widespread, and in fact, is a very capable development
system. It is very much like COBOL, and is portable over a large
number of systems (two or three hundred I think.)

It is, however, very tightly linked to the Progress Database Engine,
and though it can be used to access Oracle, it is surprising to see
anyone actually doing that.

Check out www.progress.com. I think they have a limited evaluation
set they will send you for $100. It is a lot of fun to use in some ways,
and it is very fault tolerant. The favorite Progress trick used to be
loading it on a small machine and pulling the plug while it was updating
a table. The system invariably recovered and rolled back the failed
action.

-Paul

A Progress success story I heard, take it for what it's worth.

Someone developed a system that ran on a small box. The small box was located
in a closet somewhere. After a couple of years, a couple of users came
screaming to systems that their system was down, and were told that the
system was up and running fine. Systems had forgotten about the Progress Box.
An electrician had shut off the power to the box, so they turned it back on,
everything recovered and was fine.

The reason, I gather, that Progress users resist going to Oracle is that the
Progress Database is as close to a no administration database as you can get.
The overhead of administering an Oracle Database is something that many users
just don't want.

A "screen painter" is a programming tool that allows you to visually
and interactively design a screen image this is typically accomplished
in the following manner:

Various GUI controls can be selected from an icon toolbar or from menu
options and "dropped" in a screen format area.

Behavioral properties may be assigned to the screen controls/screen
fields an an interactive manner.

Screen controls/screen fields may be selected and moved around with
the mouse or keyboard keys in a fashion that will allow the programmer
to visually and quickly see the results of his/her screen design.

The screen definition may be tested in a "test mode" so that the
programmer ensures that screen behavior has been established to
properly meet the requirements of the project design.

The programmer can generate the COBOL code necessary to display the
screen and control the screen from the COBOL program.

I attended a Progress Programming class in January to prepare for a MFG/PRO
non-oracle implementation. Keep in mind that I haven't actually used Progress
since that training class.

I'd liken progress syntax more like Mantis, than with COBOL. However if you
took SQL and replaced all the statements with a COBOL-like syntax, yes, you
would come up with something close to progress, but with a whole lot more.

I wouldn't say that Progress is paragraph oriented. Progress is heavily block
structured though. This is one of Progress' greatest strengths. The way they
use it is to provide a LOT of default behavior. Rather than have to specify
every single parameter, defaults are used that you can override if you want
to. This makes progress very quick to develop in, since you can incrementally
refine applications until they are *Just Perfect* or *Good Enough* for the
task at hand.

MFG/PRO is not the only application written in Progress. I'm aware of another
Manufacturing Package SIIMIX ( don't know the spelling ), some time and
attendance packages, bar coding ( TEMPEST, I think ) and more.

Be aware though that MFG/PRO, from what I understand, is moving away from
Progress. Their current GUI version does not use the Progress GUI, but
something else. I had heard that MFG/PRO was moving to FORTE, but I'm not
sure how true that is.
