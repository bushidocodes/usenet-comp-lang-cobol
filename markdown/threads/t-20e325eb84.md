```
Just because it might have gotten lost in all the MIS-information that has
been posted about periods, I did think that I might restate something that I
have posted before.

In my PERSONAL opinion, a style that uses a single period (best if on a line
by itself) per paragraph *is* the easiest style to maintain in an error-free
manner.  I see nothing "religious" about this - and it is best implemented
in NEW development or a "major" rewrite - not in the normal maintenance of a
program that currently has lots of periods within paragraphs (or sections).

The reasons that I would give for my preference are:

1) A period is an EASY part of code to "miss".  Sometimes insertions move
them past the R-margin and programmers don't see them.  Sometimes, they are
just hard to see on the screen or printed compiler listing. (Note: I am
visually impaired, so this is a real concern for me.)

2) A period in the middle of a paragraph (either inserted or deleted by
maintenance) *may* change logic flow in any paragraph with a NEXT SENTENCE -
and I believe in the use of the NEXT SENTENCE phrase  - except in compilers
that already support (as the 2002 Standard does) the EXIT
PERFORM/PARAGRAPH/SECTION features.  If you have a compiler with this
feature (and don't think you will EVER port your source code to one without
it) then I personally do NOT like NEXT SENTENCE as all its "valid"
functionality can be achieved by other and "more commonly understood"
features.

3) Avoiding paragraph internal periods, DOES encourage the programmer to use
scope terminators.  Again, I personally find this to be a "good" thing - as
it encourages the programmer to design the most "bullet-proof" code by
making him/her think about all the  conditional phrases available to various
statements.  It also allows for "easy" insertion of conditional phrases and
nested statements - during the maintenance of existing code. Again, in an
environment where the new EXIT facilities are available - and even the 2002
Standard feature of being able to "reverse" NOT and "positive" conditional
phrases, this MAY allow a programmer to both "optimize" their code by
placing the most common conditions FIRST as well as avoiding repeated tests
of "skip-the-rest-of this-paragraph's-logic" switches.

   ***

There have been reports (usually contradicted - so I don't know the truth)
about vendor products that insert periods within the Procedure division via
their "pre-compilers".   Obviously, in such an environment, it becomes
difficult if not impossible to get rid of "internal" periods.  On the other
hand, I did want to get back to the question of why (at least one person)
does RECOMMEND *not* require - that periods not be used except at the end of
a paragraph.
```

```
(the following, for the sake of convenience, assumes that 'thoughts' can 
be combined to form 'ideas' and that 'ideas' can be combined to form 
'concepts')

I consider the use of periods in COBOL to be analagous to the use of
periods in English, in that they should be used to define complete
sentences which do their best to relay a single, complete 'idea' even 
though this 'idea' might be composed of many simpler 'thoughts' ; much
as a SYNCPOINT in CICS or a COMMIT in DB2 should be implemented after each 
'logical unit of work' (as difficult as such a thing might be to define) 
so should a period be used in COBOL to signify, likewise, a logical 
completion.

Now... notice that in the preceding paragraph I used one period.  (notice 
that in the preceding sentence describing the preceding paragraph I used 
one period; I wonder if someone who responds to postings line-by-line will 
miss this and respond 'See... you used only one period per paragraph, nyah 
nyah nyah!')  In that what we deal with here is a language - albeit a 
computer one - there will be, obviously, differences in theories about 
what constitutes appropriate constructions to best suit both the 
limitations of the language and the purposes to which it will be applied.  
Consider English as an analogue (not *too* inappropriate as COBOL was 
designed to be 'English-like'); the English of a loan-contract is not the 
English of a VCR instruction manual is not the English of a 
romance-novel... and yet the goal of each, it can be argued, is the same 
in that each can be said to be intended to transmit ideas and concepts.

The use of periods in an address-label-printing program and a online 
inquiry program for long-term insurance disability claims might just 
possibly not be the same... but the goals of the use - to direct program 
flow and increase readability/maintainability - is the same.

It seems that this first goal of 'directing program flow' is that is being 
argued here.  If this is the case then the argument is a variant of 'GO TO 
or no-GO TO' and it seems that it will be as immediately and readily 
resolved.  It might be a sign of my education only but I was taught simple 
things... every paragraph has an -EXIT paragraph associated with it, all 
PERFORMs are done THRU this -EXIT, a GO TO is allowed only if it directs 
flow to the -EXIT or the paragraph-label above or the -ABEND paragraph and 
that a NEXT SENTENCE transfers execution of the program to the statement 
which follows.  These rules were not taught as an end-all and be-all of 
style, they were taught because most coders were thought to be able to 
implement and understand them... and 75% of your shop's budget goes for 
maintenance.  Development, of course, should be done in the 'shop's style' 
so that all code has a similar 'smell'... *not* 'identical', 'similar'; 
this allows for the introduction of different styles and techniques... 
little things, like scope-delimiters and EVALUATEs.

There... I have now voiced the opinions of All Reasonable Folks and now 
everyone can consider this debate concluded... oh, and I am the King of 
England, as well.
```

```
This might be off-topic or irrelevant... but I regularly code in at least 4
languages (COBOL, Java, Delphi, VB).  I've also been looking at C# lately.

If you don't end each complete statement with an explicit terminator, you're
bucking the norm of every other language out there.  It might not really be
a big deal to someone who never touches anything but COBOL.  But it would
annoy the heck out of me to spend a day working in Delphi, and then jump
into some COBOL that had "implied" terminators.

I also agree with the guy who mentioned that he didn't like adding new code
unless he knew it was at a spot where the logic was explicitily terminated.
I guess in a new project you could compartmentalize the code more (as in OO
coding), so back-tracing logic might not be an issue.

I agree that periods are sometimes hard to see.  And I think that is why
you'll see that most current languages use something more visible... ie: a
semicolon.  A missing period in a book, at worst is going to cause a run-on
sentence.  In a program it could cause critical failure.  I know COBOL is
supposed to be "english" like, but the period choice for a terminator was
just short-sighted IMO.
```

```
code will be executed as a "main" or "sub" program.  GOBACK works great for
this.  If, on the other hand, I *must* decide that a program will be used as
a MAIN - then I might as well use STOP RUN.
```

```
In case it's of interest:  Unisys task parameters are at Lex Level 2,
library parameters at Lex Level 3.  A program that can be executed as either
has to copy the LL=3 parameters up to LL=2 (so they're visible to the likes
of SORT procedures along with the rest of the Data Division, as well as to
the "body" of the Procedure Division, both of which run by default at LL=3).
"Uplevel" pointers are dangerous (less so since the introduction of ASD's),
so COBOL85 avoids all the danger spots inherent in uplevel pointers by
requiring $LIBRARYPROG for programs intended as "libraries" (in MCP's sense
of the term); those that aren't libraries are executable as tasks.  There
are some ways you can seriously shoot yourself in the foot with COBOL74 that
the distinction between a library and a task disallows with COBOL85.  Nearly
twenty years ago now we had one of these foot-shots that took us nine months
to find!

Point of all this being:  There may be a valid, and appropriate,
architectural reason in the underlying Fujitsu operating environment that
makes this requirement ultimately, and perhaps even vastly, more attractive
than the alternatives in that context.
```

```
I have no interest in playing
my-compiler-is-better-than-your-compiler.  The original start of this
thread was discussing the BEHAVIOR of STOP RUN as opposed to the
functionally identical BEHAVIOR of EXIT PROGRAM (non-mainframe) /
GOBACK (mainframe).  There is no need to get pedantic about it.  I'll
try and remember to use "EXIT PROGRAM" when I mean "GOBACK" just to
make things clearer.

The behavior of STOP RUN, as I understand it, is to terminate the
currently running process.

The behavior of EXIT PROGRAM, as I understand it, is to return control
to the program that originally invoked the EXITing program.

The difference is subtle, but important.  There is a certain symmetry
to most programming -- files are opened and closed; memory is
allocated and freed;  processes are created and terminated.

When I say process, please include the other possible terms like RUN
UNIT, TASK or THREAD, I'm trying to be OS non-specific.  Process
creation takes many forms -- command shells can create a process in
response to user input; existing processes can 'fork' or 'attach'
another process to perform some work in an asynchronous fashion.  All
operating systems offer some facility to create a new instance of a
process, for which I will use the generic term "CreateProcess".  In
all cases, it just means taking the necessary steps to build an
environment, load the program text, allocate resources and begin
execution of said program.

When thinking of the "TerminateProcess" event there are several things
we know need to happen.  Buffers need to be flushed; database and
network connections dropped; files closed; memory released; resources
unlocked; etc.

In most cases, Cobol programs will never perform the "CreateProcess"
actions for themselves.  That is usually performed by the invoker --
operating system or command shell.  As such, allowing that invoker to
perform the "TerminateProcess" event is nicely symmetrical.  There is
no other language and/or environment that performs the
"TerminateProcess" as a normal case -- it is usually the exceptional
case (C's abort(), S/390's ABEND service, etc)

To make this the normal case violates the principal of a single
entry/single exit.  If program A calls program B which calls program C
which performs a STOP RUN, you have just introduced, at runtime, a new
exit for programs A and B.  It is possible, as you note, to make sure
that the STOP RUN always occurs in program A, and thus does not
violate this principle, but it is an unnecessary juggling act.

But lets get back to my original statement -- Cobol dates back to the
days when the 'operating system' was a small deck that served as a
loader, computers were single task batch systems.  Such was life in
the early 60's (or so I'm told -- I'm just a wet behind the ears PFCSK
that started this gig in the late 70s).  An operator would place an
object deck in the hopper of a device called a card reader, and press
the "read" button -- this is analogous to the "CreateProcess" event
described above.  Thus, it fits that early Cobol would end its
execution with a "TerminateProcess" style action to  return the
machine to the "I'm waiting for input, Dave" state.

>"Not knowing" is not a _useful_ abstraction.  Relying on certain
'global' data is useful.

The useful abstraction is not knowing if you are going drive a stake
through the virtual heart of your process, or if you are just
gracefully handing control back to program that invoked you.  That is
the cool of "EXIT PROGRAM".

I'm not sure what global data serves in discussing the behaviors of
these two constructs.  You can certainly simulate STOP RUNS with API
calls if flag-x is set, or simulate EXIT PROGRAM if flag-y is set, but
the entire thing is unnecessary, the EXIT PROGRAM/GOBACK is smart
enough to keep track of things for itself, so you don't have to.  And
isn't that really the point of a high level language?
```

```
An EXIT PROGRAM in a "main" program is defined (in the '85 Standard) as

control on to the next (logical) statement in the SAME program.  On the

run-unit.  If this is NOT what is intended, then you are "in trouble".
```
