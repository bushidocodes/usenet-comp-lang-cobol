CICS (Customer Information Control System) is an application server program
of a type known as teleprocessing monitors. In effect, it provides a
different environment to an application program than the base opertaing
system, adding special functionality for interfacing with terminals,
printers, etc. It also does a it's own dispatch scheduling, record level
file sharing, memory management, terminal virtualization, security and a
whole host of other things. It does so in a way which is consistent across
a number of platforms, regardless of operating system, and virtualizes a
whole load of operating system like functions. As others have said in
earlier posts, it is kindof like Windows in that respect. Some
documentation is even starting to refer to the interface between the CICS
program and applications running in it as the CICS API.

On a typical system a CICS image runs in a single virtual address space and
functions as a single task. A terminal interface (typically VTAM-Virtual
Tellecommunications Access Method) allows a terminal or printer to
establish a session with the CICS region (think socket, and you won't be
too far off). CICS will allocate some virtual storage to the terminal, and
either wait for a request, or dispatch an initial 'Transaction'. Once a
transaction is started CICS will look up the program corresponding to the
transaction ID, load it (if a copy is not already in storage), allocate an
instance of whatever local storage the program has requested, fill in the
appropriate linkage to the particular terminal (socket), and dispatch the
program.

Programs can communicate with the terminal (socket), files, other programs,
other servers (database, other CICS regions, etc.), and generally do what
programs do. They can be written in a variety of languages, although COBOL
is currently the most widely used. The application programs run as called
subprograms of the CICS monitor, and make requests for services (think API
calls) by loading a function block, and then calling an entry point to a
routine whihc is part of the CICS program. In typical coding the
application program will execute some function, send a mesage to the
terminal, and then exit itself (return control to CICS). Once the terminal
operator responds, CICS receives the notice of the response, and then
dispatches whatever program is designated to process the response. This is
referred to as 'pseudo-conversational' programming. It allows CICS to be
dispatching other programs for other terminals while the first program is
waiting for a terminal response. It also allows programs to be very
modularized (if the programmer so chooses), as each program can designate
iteself, or another program, to resume after the terminal user response.

This, of course, is not an exhaustive description, but it should give you a
flavor for what's going on.
