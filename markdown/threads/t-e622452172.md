I do NOT think that you are going to be able to do what you want. The "new"
file status codes are NOT an IBM invention; they are defined by the ANSI and
ISO Standards. When you compile with CMPR2, you are explicitly stating that
you don't want (don't need or care about getting) *ANY* of the new ANSI'85
features. Once you go to NOCMPR2, you get ALL of the new stuff - including
all the new status codes. (By the way, really get to know and understand
FS=39 when/if you go to NOCMPR2. It will probably be your biggest migration
issue.)

Bottom-line (as far as IBM's mainframe compilers go)

To get any new ANSI'85 stuff (including intrinsic functions, nested
programs, etc) you will need to use NOCMPR2 and get ALL the ANSI'85 stuff -
including new file status codes.

To get the old file status codes, use CMPR2 (not available on IBM's
Workstation compilers) but also lose most of the new features of ANSI'85.

Sorry about that, but that is the way it is designed.

These status code changes were introduced with COBOL II Rel 2 many years
ago, not with COBOL for MVS. IBM got some "remarks" from our
installation, not so much because of the changes themselves, but because
some subtle changes like these were considered "too insignificant" to
warrant listing in the change/migration notes in the manuals (they were
buried in the manuals themselves, but we didn't find some of them until
after we had been bitten). We had a "major" migration effort going from
old COBOL/VS to COBOL II Rel 1, and then a minor migration effort going
from COBOL II Rel 1 to COBOL II Rel 2. We have so far not encountered
any language incompatabilities going from COBOL II to COBOL for MVS.

We never even considered trying to run for an extended time using old
code with COMPR2. One of the things you learn to expect in this
business is change, and you also learn that old standards kept around
for transitional compatability are never supported indefinitely. It's
always much better to convert ASAP so you can do it on your own schedule
rather than delay until you're forced to do it on an emergency basis for
other reasons. It is never a wise long-term strategy to tolerate a too
large deviation from "currency" in hardware, software, or coding
standards, as it will eventually cause much pain.
