Hi to all,

I need help in the following problems:

1) I have a program that reads an indexed file. The last record in
   the file has zero values in the key. When this record is read
   the program should close the file and exit, however, this is not
   happening:

   if mobile-num = zeros
   then
      write file-name invalid key display "error with key"
      display "last record read"
      go to aa-close-files
   end-if.

It appears that the invalid key clause seems to be grouping all the
statements below it. This means that the section aa-close-files and
the display "last record read" statement are not processed. Control
returns after the end-if since no invalid key problem occurs. Why
is this happening?

2) I have an indexed file with the key:

   05 mobile-no pic x(10).
   05 end-date  pic 9(6) COMP-3.
   05 end-time  pic 9(6) COMP-3.

I want to read each record and then write it with a new valued
for end-date and end-time in the same file. For example, first record
read and put end-date = '990506', end-time = '170501', second record
read and put end-date = '990506', end-time = '170502' etc...

Since the key is being updated, I read the record, saved it, then
deleted it and then wrote it again with the new values.

When I ran the program for the whole file, it updated the first and
second record but then looped continuously between the first and second
record.
I know this is a problem with the indexes but why is it happening?
Should I just write to another file rather than trying to update the
records in the same file by reading a record and then deleting and then
writing it again.

3) I have some old programs that seem to compile without using
   the invalid key clause (note that the compilation is done through
   utility programs which I do not have access to).
   However, any new programs I have written do not compile without
   the invalid key clause. Is this due a compile flag I am missing?

I'm very familiar with the VMS compiler.
There are two posibilities in this case:

1. You do not have a declaratives section ("use after expection") for
the file involved. In this case the VMS compiler requires that you use
the "invalid key" phrases.

2. There is a declaratives section for the file involved. Now you have a
choice of using the "Invalid key" phrase, but it is no longer required.

It is my guess that the old programs have a declaratives section, and
the new programs don't.

With regard to indexed files on VMS the following:

1. Normally it is not possible to change the primary key, however the
file can be created in such a way that changing the primary key is
possible.

2. The records in a indexed file are physically in the same sequence as
the primary key.

3. Inserting records in a indexed file is very well possible without any
problems for performance etc. IF you reserve enough free space when
populating the file for the first time.

4. As with all other VMS languages, Cobol uses the Record Management
System (RMS file system) of VMS for its file access.

5. Creating and populating an indexed file by means of a Cobol program
is not a very good practice. The (easiest) way to do it is like this:

5.1. Let your Cobol program open a small (!) indexed file for output.
5.2. Use Analyse /RMS/FDL to create a FDL file description where you can
find all information about the keys etc.
5.3. Use Edit/FDL to make a new FDL file where you can use all the
desired parameters for free space, number of records etc. (Using global
buffers will speed up your program dramatically).
5.4. If you have data to fill the file for the first time, let a Cobol
program create a sequential (!) file with the same record layout as the
indexed file.
5.5. Use Convert/FDL to convert the sequential file to a the indexed
file.
5.6. If you already have a indexed file, you can use the same
Convert/FDL command to make the new indexed file. 

6. Read the RMS manuals !! If you want to use indexed files on VMS, you
can't do it without knowledge of RMS. Trying to do so will almost always
result in extremely poor performing Cobol programs if the files have a
size of any importance. In fact with larger files with several keys it
may even be impossible to populate a file, or to get any reasonable
performance. 
  
If you want to read a indexed file sequentially on the primary key, and
at the same time be able to change (=delete + write or rewrite if the
file has been created the right way) the primary key, you can use two
access streams to the file. The first reads the file sequentially, the
other in random mode.

It works like this:

Loop.
	read my-file-seq at end etc.
	move file-rec-seq to file-rec-random
	read my-file-random
	delete my-file-random
	(change key values in file-rec-random)
	write file-rec-random
	go to loop.

If you need more info on Cobol on VMS please let me know.
