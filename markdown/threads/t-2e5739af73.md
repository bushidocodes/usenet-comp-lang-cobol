As far as I can understand from your writings you're actually asking for an
indicator telling you that you have now read the last record of a file. OK.
That doesn't exist, and I'm not sure I understand why you want it. the EOF
(end-of-file) quite naturally occurs when you are exactly that : at the end
of the file. There are no more records to be read. Sorry, I (the file
io-system) cannot do what you have just asked me to do (read another
record). or graphically : rec rec rec rec rec rec ! EOF.
As some other people suggest : use read into to keep your record in
working-storage. Then you can always use it after getting EOF.

Another reason. Normally you do not want to do special processing when you
read the last record of a file. Only after having done the normal
processing and you try for the next record - you get EOF and do your
special EOF-processing.

Ben

When an end of file is seen, the file position indicator is set to
indicate that there are no more records in the file. It is defined.
However, any other READ NEXT must return an error because to do
otherwise would lead to endless loops reading who knows what
(something past the last record?).

› I can't read previous record and expect the record before the last to be the current
› if I have recieved EOF the last record will be read again - this is a problem that can be
› overcome with duplication of record in working storage - thats the way we all do it
› but does it really need to be this way?

The standard does not allow you to read backwards. If you are doing
so, you are using an extension. In the proposed standard we debated
turning around automatically and decided that it would be a bug
breeder. You have to say START < a value (like HIGH-VALUES) that is
past the end.

› You say you don't normally need any special processing at EOF? Well when yo browse
› through a file and the user does it via a lookup - he/she expects it possible to
› get previous record if he reach the last...

Obviously, the user has to switch from a READ NEXT mode to a READ
PREVIOUS mode. That is special action. The extra START (without
saving anything) makes it explicit. To just start reading backwards
with READ NEXT when you hit the end would cause all kinds of problems
with existing programs.

› In processing where you produce reports in background perhaps not - but defently in
› online situations.
