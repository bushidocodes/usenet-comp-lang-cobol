It won't hurt, but you can take a severe performance hit and
inefficiency is frowned on by puritans (I'm one). Depending on the
size of the file, the operating system, and speed of the machine,
several seconds may be required just to open or close a file. The more
common way is:

OPEN TRANSACTION-FILE.
OPEN MASTER-FILE.
PERFORM UPDATING-STUFF.
CLOSE TRANSACTION-FILE.
CLOSE MASTER-FILE.

The only 'error' I see in the indexed update is that you don't have to
open/close the indexed file each time.  Open it once in the initialization and
close it at the end of the program.

HIGH-VALUES checking isn't needed in indexed updates.  It is usually only used
in the sequential updates that you have already coded.

As for the compiler error I'm a mainframe programmer so it doesn't make sense
to me.  I don't see a SORTFILE in a SELECT statement and you didn't post the
FDs so I can't make an educated guess.  Have to wait on the PC bodies on this
one.

The only other thing you should have is a FILE-STATUS on the indexed file. 
This should be checked to see if the file opened correctly and be checked to
see if the DELETE worked properly, and last after the CLOSE.  I don't know if
you have any DECLARATIVES in your program for this or not so I'm giving my
preference as to technique.  I think DECLARATIVES still needs a FILE-STATUS in
the SELECT though (which may be giving you your compiler error??? - just a wild
guess).

Good post :)  Come back again.
