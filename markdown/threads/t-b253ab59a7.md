If you don't understand MS Windows programming then you would be
well advised to not 'fiddle' with the code.

Windows programs are event driven. At the end of the initialisation
code (the main program) there will be an almost infinite loop of
'GetMessage'/ 'DispatchMessaage', possibly with a few 'Translate..'
thrown in depending on what accelerators and modeless dialog boxes
are available.

During the initialisation it would have created one or more Windows
each of which will have a callback routine.

Callback routines will be activated at any time by user interaction.
With a 'normal' program it is possible to follow the execution path
in a logical sequence. With Windows programs code will be executed
in response to mouse moves, button clicks, menu activates, etc ...

Get Charles Petzold's 'Programming Windows'. This will explain all
the detail you will need (probably), though it is a little brief in
places. Supplement this with the MS Windows SDK manuals and
MF guide on Windows.

It took me a few months to get competent with it.

Oh, and you have to get used to LOCAL-STORAGE SECTION too.

It is _very_ easy to get the system to go belly-up by using an
incorrect value in a call. Or worse the system doesn't give a
problem at the time of the wrong call then goes belly-up at some
later time.

Try and use the Win3.1x debugging kernal that comes with the MS
Windows SDK. This will tell you when invalid calls are made.
