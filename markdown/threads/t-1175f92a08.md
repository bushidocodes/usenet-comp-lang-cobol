Hello Joern

It is usually a good idea to indicate the operating system, hardware
and compiler that you are using, so that responders have more idea
what is good advice in your specific circumstances.  The advice that I
shall provide is appropriate for the IBM mainframe MVS COBOL compilers
that I have worked on.

I haven't gone into the fine detail of the logic and aim of the code,
but the following are some (what I think are) useful tips, mostly
taught to me by my first COBOL teacher, while I was learning on the
job over 30 years ago.

Try to remove as much as possible from the innermost loops, since
these are the most intensively executed and will show the greatest
benefit from any efficiency improvements.  In this case I can't see
any improvements of this nature that you can make to your code, but
you may know better.

Try to make sure that indexes and subscripts are in a binary format,
they are in this case, and that they are aligned on half-word,
full-word or double-word boundaries according to the sizes of the
field being used, in this case full word alignment would be
appropriate.  This is much less important than it used to be, but I
think it is still relevant.  This would mean that each table element
would need some filler bytes to force the appropriate alignment.  You
can obtain filler bytes either explicitly by specifying them directly
yourself, or by using the SYNCHRONIZED clause on the data definitions.
 I prefer to specify them explicitly, but that is my own preference. 
On older compilers you had to specify SYNCHRONIZED to avoid
inefficiency.  I suggest you refer to the programmers guide for your
compiler to see what that recommends.

You can use USAGE INDEX to define working-storage variables as index
data items for comparing with table indexes, there isn't much
inefficiency in using ordinary binary data items, but it avoids having
to make the conversions from index to subscript.  Indexes hold values
as displacements which have to be converted to equivalent subscript
values by multiplication by the size of the table element.  Index data
items have the same internal format as indexes so don't need this
multiplication step.

Make sure all fields being used as subscripts are signed, this makes
for faster comparisons by avoiding unnecessary checks and conversions.
 But if you convert the field to be compared with the indexes to index
data items, this won't be necessary.  Using signed numbers is
generally more efficient in all forms of arithmetic.

Try to avoid multiple resolutions of the same index value for
different data items in the same table element or multiple tests of
the same data item.  Resolving indexes and subscripts can involve
quite a high overhead, so should be minimised.  This can be done by
having a working-storage copy of a table element to which each table
element being processed is moved for further testing/processing and if
necessary moved back again into the table after any changes have been
made (you aren't making any changes to the table elements in this
example, so you don't need to do the reinstatements).  Obviously if
only one data item in a table item is being tested/processed this
isn't necessary, but your examples show cases of two and four being
tested.  Another way is to meke two or more associated items
subordinate to a group item and move that instead of doing two or more
independently subscripted separate moves.

Bear in mind that it is often more desirable to have easily readable
code than the fastest possible, it's usually faster to debug and amend
easily readable code, which is often more likely to be logically
correct.  Which is more important depends on your particular
circumstances.  However, it is not always the case that efficient code
is less easy to understand, so it is worth knowing what's efficient
and where and when to use it to best advantage.

Best of luck

Robert

Looking at this code, a few things jump out at me.  The table you are
processing is huge.  I would look at it and consider whether there's
another way to get around having such a big table in memory.

Furthermore, when you consider loops, they can and will be the major
amount of processing.  Also, remember any small coding problem will be
magnified 100 fold so it pays to optimize it.  This means simplify
simplify simplify...the fewest instructions to do the job...

The first thought that comes out other than the table size and all the
paging to virtual memory the system must be doing to handle that is
data conversions.  Make sure every move and comparison in that loop
are with like variables.  Comparing unlike variables (say PIC S9(4)
binary and PIC S9(3) COMP-3) require type conversions to do that,
which is extra cpu time, since it will have to do that each time
through your loop.  One example has already been pointed out to you
(the comparisons of abg-ind and abg-end) and I would point out the
rest if there are any but you don't post variable definitions for all
of the variables you use.  For ABG-END make sure it is an INDEX type,
as an example...

The second thought I had, which it seems you do do is make sure all
table access is indexed.  I remember hearing a case in school with TWA
where they had a program run 24+ hrs and then cut it down to 4 hours
simply by coding for indexed access.

The third thought I had is to truly think through your logic and try
to decouple some of it from the loops - if you can do that that will
help tremendously on your speed.  Basically anything you can get out
of those perform loops and into an iterative cycle will help you.  I'm
looking at your code for that and will probably post something if I
can think of it.

and make it hard so you can see what is going on in your code.  I
would suggest trying to take some of the inline performs and make them
into separate paragraphs, so your logic becomes clearer to you.  If
you do that you will see that the following is getting performed as
part of a loop.

PERFORM UNTIL BLA BLA
IF WS-FORMAT = 'X'
   IF  WS-VFELD >= TB-ABG-ABGRVON (ABG-IND)
       AND WS-VFELD <= TB-ABG-ABGRBIS (ABG-IND)
            SET PS-TREFFER-JA TO TRUE
   END-IF
ELSE
   IF  WS-VFELD-N >= TB-ABG-ABGRVON-N(ABG-IND)
       AND WS-VFELD-N <= TB-ABG-ABGRBIS-N(ABG-IND)
          SET PS-TREFFER-JA TO TRUE
   END-IF
END-IF
SET  ABG-IND UP BY 1
END-PERFORM.

What I mean by decoupling the code is that we can rewrite things more
efficiently to move logic OUT of the perform loop.  This logic is hard
to see, but becomes clear once you set things out as separate
paragraphs.  Compare the code up top with this code...

IF WS-FORMAT = 'X'
   PERFORM UNTIL BLA BLA
      IF  WS-VFELD >= TB-ABG-ABGRVON (ABG-IND)
           AND WS-VFELD <= TB-ABG-ABGRBIS (ABG-IND)
                SET PS-TREFFER-JA TO TRUE
      END-IF
      SET  ABG-IND UP BY 1
   END-PERFORM
ELSE
   PERFORM UNTIL BLA BLA
     IF  WS-VFELD-N >= TB-ABG-ABGRVON-N(ABG-IND)
         AND WS-VFELD-N <= TB-ABG-ABGRBIS-N(ABG-IND)
            SET PS-TREFFER-JA TO TRUE
     END-IF
     SET  ABG-IND UP BY 1
   END-PERFORM
END-IF.

What is important to remember is that more lines of code does not
always equal more processing time.  This is a big fallacy that most
coders have, and becomes apparent here.  While yes the same logic is
being expressed in more lines of code, the code I posted will be more
efficient.  Why?  Look at the first code.  The "IF WS-FORMAT = 'X'"
question is being asked EVERY time the loop is entered.  In the
rewritten code, the question is only asked ONCE before the loop is
even entered.  In rewriting this code for your application we reduced
the number of comparisons by 399999.  I noticed as well that other
code that you posted may be decoupled as well.
