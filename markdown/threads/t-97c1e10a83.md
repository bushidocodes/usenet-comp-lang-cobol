Open Implementation

I recently ran across a new (to me) concept that might interest the
group.

The standard concept of modular programming says that the calling
program does not know or care about how the called module does its
job.  Open implementation says that there are cases where the strategy
used by a called module is important.  Where the strategy used by the
called module is inappropriate, the calling program may create its own
version of the module code using a more appropriate strategy or
otherwise fudge the process to achieve better results.  OI calls for a
mechanism to let the calling program set the strategy used by a module
or provide information to the called module so it can select an optimum
strategy.


A COBOL Example

Suppose you have a list of codes and related data in a sequential
file.  Various programs need to retrieve the data associated with a
specific code.  This is a natural situation for creating a module to
load the data from the file into a working storage table and retrieve
the data for a code.

There are two strategies that the module can use.

1. Use the file data in an unsorted state and use a sequential lookup
(SEARCH).
2. Sort the file data after it is loaded and use a binary search
(SEARCH ALL).

Sorting the file data adds an overhead cost to the module, but the
binary search is more efficient, especially if there are many values in
the table.  The optimum strategy depends upon two factors - the number
of values in the table and the number of times the module is called.
If there are only a small number of values in the table, the cost of
the two types of searches may be equivalent.  If the module is only
called a small number of times, there may be insufficient gains from
the more efficient search to offset the cost of sorting the data.

The OI solution is to have both strategies in the module and let it
choose the best one for the circumstances.


Objections

I can think of a few objections to the OI concept.

1. It is unnecessary.  Given the power of modern computers, it makes no
difference whether the called modules are efficient or not.

However, COBOL systems frequently deal with very large files.  When you
start processing millions of records, small improvements in performance
can still count.

2. It is too costly.  The extra code will cost more than the benefits
gained.  It would be more cost efficient to spend the money on hardware
than on programming.

However, the cases where OI is appropriate can occur frequently within
an application or company.  If you write off the cost over many
instances, then the programming can be justified.

3. Good application specifications should allow a software designer to
choose the best strategy from the beginning.  There should be no need
for elaborate programming.

Unfortunately, specs are not always trustworthy.  (Let�s face it. They
are never trustworthy.)  Even in legacy systems where the use of a
module is well known and understood, business rules can change
overnight.  Its all too frequent that we get a message from the gods
like �Now that we have completed the web site that allows our clients
to enter orders directly, management has decided to process orders
every half hour rather than in one overnight batch.  We expect that
each transaction batch will have no more than 10% of the previous daily
batch and most will have less than 1%.  Of course, the rapid and
efficient processing of orders is now more important than ever to meet
the expectations of our web customers.�


Code

I have written an example to test out the OI solution.  The code is in
a separate message.  The code consists of a program (openimp) to test
the called module and a called module (tblookup) to look up values in
the data.


Documentation

The tblookup module accepts three values:
- a one-character code that tells the module which function to
perform.  The codes are - "C" to configure itself for an optimum
strategy and "F" to find and return the data associated with a code
from the table.
- parameter1 - for the configure function, this is the expected number
of calls to the module as a COMP-3 S9(7) value.  For the find function,
this is the code to find in the table.
- parameter2 - for the configure function, this value is not used.  For
the find function, this area is filled with the data found in the
table.  If tblookup cannot find an entry for a code, it fills this area
with hex 00 (LOW-VALUES).

Note that tblookup is written to make it as easy as possible to clone
the module for tables with different code and data sizes.

To choose the optimum strategy, tblookup estimates the cost of using
each strategy and chooses the one with the lowest cost.  The cost of a
sequential search is number-of-calls * file-size * factor1, where
factor1 is the cost of doing a sequential search per table item.  The
cost of the binary search is the cost of sorting the data (file-size *
factor2) plus the cost of doing the search (number-of-calls * file-size
* factor3).

I picked the values that for the factors in the code from thin air and
they no relationship to reality.  The factors themselves will vary with
the environment - computer, operating system, compiler used, etc.
However, it should be possible to determine them experimentally.

The openimp program
- creates a test file of values
- calls tblookup to configure the module
- calls tblookup to find a value that is in the table
- calls tblookup to find a value that is not in the table.

The test file is written in a reasonably random order to test out the
sorting procedure.

Here is the source code described in the previous message.

       IDENTIFICATION DIVISION.
       PROGRAM-ID. tblookup.
      * Generic template for a module to lookup and return data from
      * a sequential file.

      * To change the maximum size of the table
      * - change the value in table-max
      * - change the OCCURS value for table-item
      * To change the length of the key
      * - change the table-key definition
      * - change the value in table-key-length
      * To change the length of the data
      * - change the table-data definition
      * - change the value in table-data-length

       ENVIRONMENT DIVISION.
       INPUT-OUTPUT SECTION.
       FILE-CONTROL.
           SELECT tabrc-file ASSIGN TO tabfile
             ACCESS MODE IS SEQUENTIAL
             ORGANIZATION IS LINE SEQUENTIAL.

       DATA DIVISION.
       FILE SECTION.
       FD  tabrc-file.
       01  tabrc-record                                    PIC X(1000).

       WORKING-STORAGE SECTION.
       01  work-fields.
           05 ws-when-compiled         VALUE SPACE        PIC X(8)BX(8).
           05 item-nbr                 VALUE ZERO   COMP   PIC S9(5).
           05 swap-nbr                 VALUE ZERO   COMP   PIC S9(5).
           05 jump-size                VALUE ZERO   COMP   PIC S9(5).
           05 upper-limit              VALUE ZERO   COMP   PIC S9(5).
           05 no-of-calls-x.
             10 no-of-calls            VALUE ZERO   COMP-3 PIC S9(5).
           05 sequential-search-time   VALUE ZERO   COMP-2.
           05 binary-search-time       VALUE ZERO   COMP-2.
       01  switches.
           05 record-swapped-sw        VALUE 'Y'           PIC X.
             88 record-swapped         VALUE 'Y'.
           05 first-time-sw            VALUE 'Y'           PIC X.
             88 first-time             VALUE 'Y'.
           05 end-of-file-sw           VALUE SPACE         PIC X.
             88 end-of-file            VALUE 'Y'.
           05 strategy-sw              VALUE 'S'           PIC X.
             88 strategy-sequential    VALUE 'S'.
             88 strategy-binary        VALUE 'B'.
       01  table-area.
           05 table-max                VALUE 1000   COMP   PIC S9(4).
           05 table-size               VALUE ZERO   COMP   PIC S9(4).
           05 table-key-length         VALUE 2      COMP   PIC S9(4).
           05 table-data-length        VALUE 10     COMP   PIC S9(4).
           05 table-item OCCURS 0 TO 1000
                         DEPENDING ON table-size
                         ASCENDING KEY table-key
                         INDEXED BY tabidx.
             10 table-key                                  PIC XX.
             10 table-data                                 PIC X(10).

       LINKAGE SECTION.
       01  link-method                                     PIC X.
           88 link-method-configure    VALUE 'C'.
           88 link-method-find         VALUE 'F'.
       01  link-parm1                                      PIC X(1000).
       01  link-parm2                                      PIC X(1000).

       PROCEDURE DIVISION USING link-method
                                link-parm1
                                link-parm2.
      *-----------------------------------------------------------------
      * Main program logic
      *-----------------------------------------------------------------
           EVALUATE TRUE
           WHEN link-method-configure
             MOVE link-parm1(1:LENGTH OF no-of-calls)
               TO no-of-calls-x
           WHEN NOT link-method-find
             DISPLAY 'tblookup-001: Invalid method code ' link-method
             STOP RUN
           WHEN first-time
             INITIALIZE first-time-sw
             PERFORM 0100-initialization
             PERFORM 2000-find-item
           WHEN OTHER
             PERFORM 2000-find-item
           END-EVALUATE

           GOBACK
           .
      *-----------------------------------------------------------------
      * Prepare the program for processing
      *-----------------------------------------------------------------
       0100-initialization.
      * Display the WHEN-COMPILED date as the program version.
           MOVE WHEN-COMPILED              TO ws-when-compiled
           DISPLAY 'tblookup-002: Version ' ws-when-compiled
      * Load the table with data from the file
           PERFORM 7000-load-table
      * Determine the best strategy
           PERFORM 1000-determine-strategy
      * Sort the table if using a binary search
           IF strategy-binary
             PERFORM 8000-sort-table
           END-IF
           .
      *-----------------------------------------------------------------
      * Determine the optimum strategy from the configuration parameters
      *-----------------------------------------------------------------
       1000-determine-strategy.
           COMPUTE sequential-search-time = no-of-calls
                                          * table-size
                                          * .001
           COMPUTE binary-search-time = (table-size * .002)
                                      + (no-of-calls
                                       * table-size
                                       * .0005)
           IF sequential-search-time > binary-search-time
             SET strategy-binary           TO TRUE
           ELSE
             SET strategy-sequential       TO TRUE
           END-IF
           .
      *-----------------------------------------------------------------
      * Find a matching item in the table and return its data
      *-----------------------------------------------------------------
       2000-find-item.
           SET tabidx                      TO 1
           IF strategy-binary
             PERFORM 2100-binary-search
           ELSE
             PERFORM 2200-sequential-search
           END-IF
           .
      *-----------------------------------------------------------------
      * Find a matching item in the table using a binary search
      *-----------------------------------------------------------------
       2100-binary-search.
           SEARCH ALL table-item
           AT END
             MOVE LOW-VALUE
               TO link-parm2(1:table-data-length)
           WHEN link-parm1(1:table-key-length) = table-key(tabidx)
             MOVE table-data(tabidx)
               TO link-parm2(1:table-data-length)
           END-SEARCH
           .
      *-----------------------------------------------------------------
      * Find a matching item in the table using a sequential search
      *-----------------------------------------------------------------
       2200-sequential-search.

           SEARCH table-item
             VARYING tabidx
           AT END
             MOVE LOW-VALUE
               TO link-parm2(1:table-data-length)
           WHEN link-parm1(1:table-key-length) = table-key(tabidx)
             MOVE table-data(tabidx)
               TO link-parm2(1:table-data-length)
           END-SEARCH
           .
      *-----------------------------------------------------------------
      * Load data from the file into the working storage table
      *-----------------------------------------------------------------
       7000-load-table.
           OPEN INPUT tabrc-file

           PERFORM
             UNTIL end-of-file
             READ tabrc-file
             AT END
               SET end-of-file             TO TRUE
             NOT AT END
               ADD  1                      TO table-size
               IF table-size >= table-max
                 DISPLAY 'tblookup-003: table overflow'
                 STOP RUN
               END-IF
               SET  tabidx                 TO table-size
               MOVE tabrc-record           TO table-item(tabidx)
             END-READ
           END-PERFORM

           CLOSE tabrc-file
           .
      *-----------------------------------------------------------------
      * Sort the table
      * Logic for a COMB Sort taken from
      * www.etk.com/papers/sorting/sorting.htm
      *-----------------------------------------------------------------
       8000-sort-table.
           MOVE table-size                 TO jump-size
      * use the next free item in the table as a swap area
           ADD  1                          TO table-size
           PERFORM 8100-sort-cycle
             UNTIL NOT record-swapped
               AND jump-size <= 1
      * reset the table to its correct size
           SUBTRACT 1                    FROM table-size
           .
      *-----------------------------------------------------------------
      * Process one cycle of the sorting procedure
      *-----------------------------------------------------------------
       8100-sort-cycle.
           COMPUTE jump-size = (10 * jump-size + 3) / 13
           COMPUTE upper-limit = table-size - jump-size - 1
           INITIALIZE record-swapped-sw
           PERFORM 8200-swap-items
             VARYING item-nbr FROM 1 BY 1
               UNTIL item-nbr > upper-limit
           .
      *-----------------------------------------------------------------
      * Compare and, if necessary, swap two table items
      *-----------------------------------------------------------------
       8200-swap-items.
           COMPUTE swap-nbr = item-nbr + jump-size
           IF table-key(item-nbr) > table-key(swap-nbr)
             MOVE table-item(item-nbr)     TO table-item(table-size)
             MOVE table-item(swap-nbr)     TO table-item(item-nbr)
             MOVE table-item(table-size)   TO table-item(swap-nbr)
             SET  record-swapped           TO TRUE
           END-IF
           .
       END PROGRAM tblookup.

       IDENTIFICATION DIVISION.
       PROGRAM-ID.  openimp.

      * Test the tblookup module

       ENVIRONMENT DIVISION.
       INPUT-OUTPUT SECTION.
       FILE-CONTROL.
           SELECT tabrc-file ASSIGN TO tabfile
             ACCESS MODE IS SEQUENTIAL
             ORGANIZATION IS LINE SEQUENTIAL.

       DATA DIVISION.
       FILE SECTION.
       FD  tabrc-file.
       01  tabrc-record                                    PIC X(1000).

       WORKING-STORAGE SECTION.
       01  work-fields.
           05 sub1  OCCURS 2           VALUE 1      COMP   PIC S9(4).
           05 sub2                     VALUE ZERO   COMP   PIC S9(4).
       01  switches.
           05 count-overflow-sw        VALUE 'Y'           PIC X.
             88 count-overflow         VALUE 'Y'.
       01  fixed-values.
           05 codes
           VALUE '1QAZ2WSX3EDC4RFV5TGB6YHN7UJM8IK9OL0P'    PIC X(36).
           05 file-size                VALUE 0100   COMP-3 PIC S9(5).
           05 no-of-calls              VALUE 1000   COMP-3 PIC S9(5).
       01  table-record.
           05 table-key                VALUE SPACE         PIC XX.
           05 table-data.
             10                        VALUE 'REC# '       PIC X(5).
             10 table-rec-no           VALUE ZERO          PIC 9(5).

       PROCEDURE DIVISION.
      *-----------------------------------------------------------------
      * Main program logic
      *-----------------------------------------------------------------
      * Create the table file
           PERFORM 1000-create-file
      * CALL tblookup to set strategy
           CALL 'tblookup' USING BY CONTENT 'C'
                                            no-of-calls
                                            ' '
      * CALL tblookup to find last value in table
           CALL 'tblookup' USING BY CONTENT   'F'
                                              table-key
                                 BY REFERENCE table-data
      * CALL tblookup to find a value not in the table
           MOVE ALL ':'                    TO table-key
           CALL 'tblookup' USING BY CONTENT   'F'
                                              table-key
                                 BY REFERENCE table-data

           GOBACK
           .
      *-----------------------------------------------------------------
      * Create a test table file
      *-----------------------------------------------------------------
       1000-create-file.
           OPEN OUTPUT tabrc-file

           PERFORM
             file-size TIMES
             PERFORM 1100-create-key
             ADD  1                        TO table-rec-no
             WRITE tabrc-record          FROM table-record
           END-PERFORM

           CLOSE tabrc-file
           .
      *-----------------------------------------------------------------
      * Create a record key with a random but unique value
      *-----------------------------------------------------------------
       1100-create-key.
           SET  count-overflow             TO TRUE
           PERFORM
             VARYING sub2 FROM 1 BY 1
             UNTIL sub2 > LENGTH OF table-key
             IF count-overflow
               ADD 1                       TO sub1(sub2)
               IF sub1(sub2) <= LENGTH OF codes
                 INITIALIZE count-overflow-sw
               ELSE
                 MOVE 1                    TO sub1(sub2)
               END-IF
             END-IF
             MOVE codes(sub1(sub2):1)      TO table-key(sub2:1)
           END-PERFORM
           .
       END PROGRAM openimp.
