Is it possible to dynamically generate files from within COBOL without

allocated would not already be allocated on the Mainframe.

processed

Almost all PC compilers have a method of "allocating" new files (with file
names not known at compile-time) - this has NOT ever been a part of IBM
mainframe compilers.

In the draft Standard, there is a new format of the Select/Assign clause
(that has
   Select file-name USING variable-name

which is "similar" (but not quite identical) to how many/most PC
implementations work today.

The IBM mainframe implementations (old and new) are quite different.

This is an area of "Standard COBOL" that is (rightly?) left almost entirely
up to the implementor as what a "physical file name" means is so different
on different operating systems.

It sure is possible to dynamically allocate either new or existing files in
Cobol, using the BPXWDYN  IBM-supplied subroutine (which we also use in
assembler and PL1 programs).  Just set up a table of file allocation
parameters, then go through the table and call the subroutine for each file
to be allocated, similar to this:

Dynamic Allocation in High-Level Languages IBM has made available (as part
of Unix Services for OS/390) a generalized file allocation/deallocation
routine that makes it easy to allocate files in high-level languages, such
as Cobol or PL/1. This subroutine can also be called by Assembler programs
and REXX execs.  Called BPXWDYN, the subroutine is passed a single, varying
length character string of up to 100 bytes in length, containing the ALLOC
or FREE command and its operands, coded exactly as you would enter the
corresponding TSO command.   For the complete syntax rules, see the TSO HELP
section on the ALLOC command.   The subroutine can be called to allocate
input, output, or SYSOUT files. Since this is an IBM-supplied and supported
subroutine, it can be used to replace various user-written subroutines. When
using this subroutine, no JCL DD statements are required when running the
program. If any error conditions occur, they will be listed in the job lob
if the MSG(WTP) operand is included in the parameter string, as shown in the
following example. This example is of a Cobol program calling the subroutine
to allocate, open, read, print, close, and free a member of a PDS. For
simplicity's sake, the varying length parameter is always set to 100 bytes,
the maximum allowed.

       CBL NOLIST,NOSEQ
       IDENTIFICATION DIVISION.
       PROGRAM-ID. COBDYNAL.
......*===============================================================*
......* This program dynamically allocates and deallocates a file
......* using the BPXWDYN IBM subroutine. No DD statements for the
......* file sould appear in the JCL:
......*===============================================================*
       ENVIRONMENT DIVISION.
       CONFIGURATION SECTION.
       INPUT-OUTPUT SECTION.
       FILE-CONTROL.
       SELECT FILEIN ASSIGN TO FILEIN.
       DATA DIVISION.
       FILE SECTION.
       FD FILEIN
       RECORD CONTAINS 80 CHARACTERS
       RECORDING MODE IS F
       LABEL RECORDS ARE STANDARD,
       BLOCK CONTAINS 0 CHARACTERS
       DATA RECORD IS FILEIN-REC.
       01 FILEIN-REC.
          05 FILLER PIC X(80).

       WORKING-STORAGE SECTION.
       77 INPUT-REC PIC X(80).
       77 IN-COUNT PIC 9(9) BINARY VALUE 0.
       77 SUB-PGM PIC X(8) VALUE 'BPXWDYN '.
       77 ALLOC-CMD PIC X(100) VALUE
            ' ALLOC DD(FILEIN) DSN(P2RSY.TSO.DATA(COMPILE)) CYL(2,1)
MSG(WTP)  '.

       01 COMMAND-STRING.
          05 C-S-LEN PIC S9(4) BINARY VALUE +100.
          05 C-S-DATA.
             07 FILLER PIC X(100).

       PROCEDURE DIVISION.
       INITIAL-STUFF.
......*===============================================================*
......* MOVE THE "ALLOC" STATEMENT TO COMMAND, THEN CALL SUBROUTINE:
......*===============================================================*
       MOVE ALLOC-CMD TO C-S-DATA.
       CALL SUB-PGM USING COMMAND-STRING.
......*===============================================================*
......* OPEN THE FILE JUST ALLOCATED:
......*===============================================================*
       OPEN INPUT FILEIN.

...... READER.
......*===============================================================*
......* READ AND PRINT ALL THE RECORDS IN THE INPUT FILE:
......*===============================================================*
       READ FILEIN INTO INPUT-REC,
       AT END GO TO E-O-J.
       ADD +1 TO IN-COUNT.
       DISPLAY FILEIN-REC.
       GO TO READER.

...... E-O-J.
......*===============================================================*
......* AT END-OF-FILE, CLOSE INPUT FILE, MOVE "FREE" TO COMMAND:
......*===============================================================*
       CLOSE FILEIN.
       MOVE 'FREE DD(FILEIN)' TO C-S-DATA.
       CALL SUB-PGM USING COMMAND-STRING.
       STOP RUN.
//*********************************************************************

system software. It functions with all MVS version of COBOL. The following
is the introduction to the documentation attached.

DYNAM provides an interface between a high level language such as COBOL or
PLI, and SVC 99, the system dynamic allocation  routine.  It allows the user
to acquire system resources as the need arises and free them when they are
no longer required.  DYNAM takes simple parameters consisting of character
strings and translates them into the text units required by SVC 99.  In
addition, a special information retrieval function can be called to provide
all the information available about a particular allocation.  This
information is returned in a fixed format data structure.  For more
