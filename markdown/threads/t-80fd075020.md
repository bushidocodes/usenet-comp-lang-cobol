EXEC SQL SELECT  *
         INTO :DCLVar

         WHERE  Table_Var_1   =

CopyBook.  (  With the Statement "COPY CopyBook." in the LINKAGE SECTION
)

While Compiling the program, the PreCompiler throws up an error  telling
the vairable is UNDEFINED or UNUSABLE Host Variable.  There is no
problem if I use a locally defined variable in place of CopyBook
Variable.  Looks like the PreCompiler ignores the CopyBook variables.

Just copy your definitions via the DB2 Preprocessor:

The Library in which your CopyBook resides has to be defined onto the
Preprocessor via '//SYSLIB DD DSN=your.copybook.libaray,DISP=SHR'.

when compiling C and PL/I. That's life!

In several (some obscure) places, IBM documents that you cannot use COPY
statements for *either* EXEC CICS or EXEC SQL type programs that need to
reference certain types of COBOL/SQL or COBOL/CICS statements.

If you think about this, it sort-of makes sense.  For IBM (where
precompilers and/or translators are used), the "step" before the compile has
to handle all EXEC CICS statements or EXEC SQL statements - but it is the
step with the compiler that handles COPY statements.  Therefore, if the COPY
statement needs to be resolved to handle the EXEC statement's operands -
this causes a problem.

The EXEC SQL INCLUDE solution solves this for SQL.  For CICS, the only way
to handle this is to "translate" your CICS code and put the translator
OUTPUT into your COPY library.  This doesn't always work (for example if you
have an EXEC CICS HANDLE statement - don't try translating that in a
separate job from the rest of your program).

For other environments (than IBM mainframe), where the EXEC SQL and EXEC
CICS statements are handled in the same "step" as the compile, this is not a
problem.
