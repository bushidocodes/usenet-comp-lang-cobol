Hi Duncan,

We use Acu4GL with Sybase and had heaps of performance problems until
release 4.3.1.

I have heard along the grape vine (second hand news so beware!!!) that the
Acu4GL MS SQL version is still very slow.

My initial tests seem to indicate that using Acu4GL for Sybase is between 2
to 3 times slower than straight vision files.

This was after some small modifications to code to enforce a "where"
constraint with the selects that Acu4GL automatically generate.

We haven't put this release into a production environment yet so I can't
tell you it was a success but we are confident it won't be more than 3 times
slower than vision.

Acu4GL  can be setup to allow some files to remain in vision and some to be
point to your database.  This is the approach we will be using with index /
code type files being left in Vision and the actual data files to be put
into Sybase. This will help with performance a little, and also gives us /
and our clients a backup plan for higher performance systems ... with the
draw back it isn't in there DB (but might be uploaded daily or weekly for
example).

Not having looked at all the options others have suggested here, but I'd be
pretty surprised if any can improve significantly on what we have achieved
here with Acu4GL and Sybase without major rewrites.  If your looking for
performance from a DB anywhere near vision without re-writing I think you
will be dissappointed.

I would suggest you contact Acu support and see if they can give you some
pointers as to how to improve your performance.
Mentis support us here in Oz and have been a great help sorting out our
Sybase performance problems and numerous others.

Another thing to consider is that DB's (doesn't matter which one) will never
achieve the performance of ISAM files.  Maybe your expections are to high?
I'd be interested if anyone could give actual examples of where moving to a
DB increased performance ... any takers?

Good luck,

Ray Smith
