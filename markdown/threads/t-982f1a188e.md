Hoooo boy!! Is there ever...

01 TMyRecord.
05 a PIC S9(05) COMP-3.
05 b PIC S9(05) COMP-3.

Unlike Pascal, there are no 'type definitions' per se. You simply define the
storage area directly. So, instead of setting up a type definition, and then
applying it to multiple data items later (as in Pascal or VB), you lay down the
structure in whatever section of the DATA DIVISION it's needed.

The neat part about COBOL is that you can turn around and redefine what you've
just laid down, as something entirely different, and still be addressing the
same place in RAM (can't do that in VB, and, I don't think you can do it in
Pascal either).

For example...

01 TMyRecord.
05 a PIC S9(05) COMP-3.
05 b PIC S9(05) COMP-3.
01 TYourRecord REDEFINES TMyRecord.
05 q PIC XX.
05 FILLER PIC XX.
05 r PIC XX.
01 TSomeonesRecord
REDEFINES TMyRecord.
05 AByte OCCURS 6 TIMES
PIC 9.

Each record (TMyRecord, TYourRecord, and TSomeonesRecord) defines what the same
six bytes of storage look like.

In typical Cobol I don't know of anything very similar to a
typedef, though some dialect somewhere may have a special feature.
(I'm assuming that by 'typedef' you mean something like a typedef
in C. Your example looks more like Pascal or PL/I; I don't know
those languages well enough to draw analogies in public, so I will
compare Cobol to C.)

You *can* define something like a C struct (a 'group item' in
Cobolspeak) which contains members (called 'elementary items').
Your example might look like this:

01 TMTYPE-RECORD.
05 TMTYPE-A PIC S9(04) COMP.
05 TMTYPE-B PIC S9(04) COMP.

A group item may also contain other group items, in a hierarchical
arrangement.

You can reference a group item in an assignment statement, or pass
it as a parameter:

MOVE SPACES TO FOO-RECORD.
MOVE CURR-SORT-KEY TO PREV-SORT-KEY.
CALL 'BV42921A' USING FOO-RECORD.

However, the group item is just a collection of fields. It is not
a user-defined type in any very useful sense.

To me, the notion of a user-defined type implies at least two
things, and neither is true in Cobol:

1. Having defined a type, you should be able to declare one or more
objects of that type without repeating the details of their
internal structure.

In Cobol, if you have two objects of the same type, you have to
explicitly specify all the members of each one. Corresponding
members of the two structs may have different names (MOVE
CURR-FOO-KEY TO PREV-FOO-KEY), or they may be distinguished by
qualification (MOVE FOO-KEY OF CURR-FOO TO FOO-KEY OF PREV-FOO).

2. The compiler should enforce type safety, so that (for example)
it doesn't let you assign one object to another unless they are of
compatible types.

Cobol is less strongly typed than C. Among group items it is
completely untyped. You can MOVE FOO TO BAR even if FOO and BAR
have completely different and incompatible structures. You can
pass anything to a subprogram; at compile time the parameters need
not match what the subprogram expects, either in number or in type.
Someone may come with an obscure exception, but in general there is
no type safety for group items.

In a word, the answer is no. Cobol supports aggregates of data but
does not give these aggregates any special status worthy of the
name of typedef.

Clinton,

In cobol you don't necessarily need to create typedefs. In cobol you
can assign any file name to a file in the "file-control" section, then you
can use the file section to define any fields, lengths, types, etc., that
you want the file to contain. (variable or fixed, of course) You can then
access this file name via read, write, etc. any which way you would like.

In cobol you don't assign variables to be of type TMyType (or any other
record type) like you can do with the typedef declaration you listed,
however like all different languages, you have different ways of reaching
the same end result. :)
(Keep in mind that I am not referring to the object oriented cobol which
is now available).

(together with a C to COBOL converter of headers). For example, the
following .h file:

typedef struct {

char L_PWX53_INL_ACC_NO[11];
double L_PWX53_PYT_AM;
char L_PWX53_RTN_ID[6];
long L_PWX53_RTN_CD;

} FORTWX53 ;

typedef struct {

char L_PWX53_INL_ACC_NO[10];
double L_PWX53_PYT_AM;

char L_PWX53_RTN_ID[5];
long L_PWX53_RTN_CD;

} LINKWX53 ;

definitions can be used (using USAGE ):
