How's this:

MOVE 1 TO STRING-PTR.
PERFORM VARYING INDX FROM 1 BY 1 UNTIL INDX > 10
    UNSTRING MYRECORD DELIMITED BY ',' INTO FIELD(INDX)
            WITH POINTER STRING-PTR
   END-PERFORM.

I'd like to add my two cents worth...

I finished very sophisticated program about nine months ago on the mainframe
which required several routines that would unstring different areas into output
fields as you suggested in the original message.

I wanted the routines to be open ended in that I didn't want to add more
parameters every time I upgraded the program with more features which resulted
in more output fields.  So I wrote the routines to loop until it found the end
of the input area automatically.  It had to find the end either by first
finding and transforming the first characters (that would never be possible in
the input area normally) to another set of characters.  If you can trust blanks
then use blanks in the test of the loop.  Any way, make it know when to stop
examining the input area for more fields.

The output fields would be subscripted and therefore "open ended".  These
routines were mostly self supporting.  I might have to code a static count for
a limit or possible enlarge the definition of the maximum possible output
fields (increase the occurs clause) but mostly it was self managing.

Believe me, the work in getting it to work paid off in the long run.  I got
very smart about something I was not that familiar with (and a little scared of
at first) and a routine/program I could be proud of.  That was a very
satisfying time in the recent past.

I don't have the logic in front of me, and therefore I can't tell you if there
isn't a little more code in front of the suggestion, but you'll have to
experiment and determine the possibilities.  You'll be the better for it in the
long run.

Roger.

this newsgroup, it appears that loops with reference modification are
