I think either I am not understanding your question, or that you are not 
understanding static vs. dynamic calls.

If you call a program dynamically, using either the DYNAM option and 
CALL 'literal' or CALL data-name (regardless of the DYNAM option 
setting), the calling and called programs are separately link-edited.

If you have a number of dynamically called subroutines, you could link 
all of them together, but I see no advantage to doing so.  Each of the 
dynamically called subroutines can be separately stored in the load 
module library, and each subroutine can be changed as needed without 
recompiling or relinking any other programs in the run unit.

If you link all of the dynamically called subroutines together into a 
single load module, you will need an alias for each routine, in order to 
  find that routine at execution time, and if any one of the subroutines 
is changed, the entire group will have to be relinkedited (and the 
aliases all replaced).  These programs probably have no relationship to 
each other, except that they are called by the driver.  So why try to 
put them together?

If you make dynamic calls, you should not try to put the subroutines 
into the driver's load module.  (This was one possible interpretation of 
your original question.)

Final note:
You should make every effort to get off of VS COBOL II, and onto a 
modern, supported COBOL compiler.

It is interesting that some of the IBM COBOL team have said they believe 
nearly all of their customers are using only supported COBOL compilers, 
but there is daily evidence that many IBM customers are using one or 
more of the no longer supported compilers.  I am working with two 
different large companies, one of which has both OS/VS COBOL and VS 
COBOL II with a little "new COBOL", and the other is using OS/VS COBOL 
with LANGLVL(1) - 1968 ANSI COBOL syntax - and has never even licensed 
VS COBOL II.  Both have a "new COBOL" compiler, but the applications 
staff do not use the new version.  The shops both have yet to convert to 
Language Environment run time.  This means their applications (the only 
reason they have a mainframe computer and OS/390 or z/OS) are completely 
unsupported by IBM!
