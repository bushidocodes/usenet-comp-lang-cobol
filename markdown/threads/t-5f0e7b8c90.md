There is an IDMS newsgroup, but it gets very little traffic. 

The difference between accessing records in local or CV modes is in the
JCL:

I commented out the following card to run a job in local mode
//*SYSCTL     DD DSN=IDMS50..SYSCTL,DISP=SHR

Of course you lose journaling when you do this - but for retrieval only
that's no problem.  CV mode keeps track of all of your changes and does
a rollback after an abort or when you tell it to roll back.

Programmers where I have worked don't have the ability to run CV mode,
nor local mode update against production.  Usually we could run local
mode retrieval.

With fast machines of today, it is important to do commits often to
avoid filling up the journal files.  Trouble is is you can't then roll
back to the beginning of the job!!

It can be done with some other routines.  We use something called RSAM
Restartable Sequential Access Method.  I think it's homegrown.  At least
some of the COBOL modules are.

The logic says that I call RSAM to do all my I/O.  It also keeps track of
the database and file positions.  I do opens, closes, writes, reads,
commits, etc using calls to a module.

What happens is let's pretend that my program uses RSAM and bombs out in the
middle of execution (like CV goes down for backup or whatever, or the job
times out, or there is contention, or it's just plain killed by accident).
RSAM holds the files open and also keeps a record of my w-s image on the
database.  When I restart my program, it calls RSAM to see if it's being
restarted.  If so, RSAM reloads the program's w-s and processing continues
as if it had never stopped.

The beauty of this is that we can commit large jobs often keeping the
database running smooth.  We call RSAM to do a commit that's when it copys
the w-s area to the database and releases all the committed file records.
All JCL DD statements are set up to (MOD,CATLG,CATLG) any output files.

Pretty slick stuff I think.  Most of you will probably yawn.  I'm sure it
has limitations but it works well for a database sweep or update from/to
flat files and it's nice to have for programs that spawn many other
processes and are time critical.

Jeff
