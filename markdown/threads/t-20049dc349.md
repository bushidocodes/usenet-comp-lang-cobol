```
I don't program under Windows, but I do use select.  What is your
question exactly?  I can tell you that if the peer has closed his end of
the socket the select will return with a read bit set (assuming you set
it when you called select).  When you do the read it will return with 0
which means the peer has disconnected.  In other words it will return as
soon as TCP receives the FIN segment.
```

```
This is only a guess, since I don't actually use it, but based on your
example...

01  TIMEOUT.
    05  TO-SEC		PIC S9(4) BINARY.
    05  TO-USEC		PIC S9(4) BINARY.

01  BITMAPS.
    05  RSOCKS		PIC X(4).
    05  WSOCKS		PIC X(4).
    05  ESOCKS		PIC X(4).

ADD 1 TO MYSOCK GIVING MAXSOCK
MOVE LOW-VALUES TO RSOCKS
MOVE LOW-VALUES TO WSOCKS
MOVE LOW-VALUES TO ESOCKS
CALL 'SETBIT' RSOCKS MYSOCK
MOVE 60 TO TO-SEC
MOVE 0 TO TO-USEC
CALL WIN32API 'select' using
    BY VALUE MAXSOCK
    BY REFERENCE RSOCKS
    BY REFERENCE WSOCKS
    BY REFERENCE ESOCKS
    BY REFERENCE TIMEOUT
    RETURNING WS-RETURN-CODE
END-CALL

MAXSOCK has to be set to 1 more than the highest socket value you want
to check for.  (Actually, I heard in Winsock this is ignored, but better
safe than sorry.)

RSOCKS, WSOCKS and ESOCKS are all 'bit maps' with bits set for each
socket you want to check for 'reading', 'writing' or 'exception'.  In
this example we're just checking one socket and we're checking it for
reading.  TIMEOUT is a structure with the first field being seconds and
the second being milliseconds.  If TIMEOUT has a NULL address it will
block indefinately, or until a socket is available for R/W/E (whichever
you've chosen).  (The two 05 levels may be fullwords (PIC S9(8) BINARY)
instead of halfwords.  I forget which!)

The return value is the number of selected sockets actually available. 
If ir returns zero then the select has timed out with no activity.

'SETBIT' would have to be a subroutine to set the specified bit in the
bitmap.  In other words, if MYSOCK had a value of 1 it would set RSOCKS
to b'00000010000000000000000000000000' (x'02000000').  I don't know if
your COBOL has a built in subroutine to do this, but I wouldn't be
surprised if it did.

Your 'bitmaps' must be long enough to hold all the bits you're setting. 
In other words, if MYSOCK is more than 31 you better have more than a
PIC X(4) or you'll be writing over later working-storage.

Once the select returns with your bit set you should be able to do a
read/recv and get some data without it blocking.

There's more to it than this, but I hope this gets you started.  Check
out this webpage I just found:
http://www.lowtek.com/sockets/select.html

Unfortunately the examples are in C, but it still looks useful.
```
