      RECORD CONTAINS 1 TO 100 CHARACTERS.
01  ONE-BYTE-REC     PIC X.
01  FIFTY-BYTE-REC   PIC X(50).
01  ONE-HUNDRED-BYTE-REC PIC X(100).

01  WORK-AREAS.
     05  WS-WORK                  PIC X(50) VALUE '1........10.........20........30........40........5'.

...
MOVE SPACES TO ONE-HUNDRED-BYTE-REC.
MOVE '1' TO ONE-BYTE-REC.
WRITE ONE-BYTE-REC.
MOVE WS-WORK  TO FIFTY-BYTE-REC.
WRITE FIFTY-BYTE-REC.
WRITE ONE-BYTE-REC.
STRING FIFTY-BYTE-REC DELIMITED BY SIZE,
             FIFTY-BYTE-REC DELIMITED BY SIZE
   INTO WS-100-BYTE-WORK.
WRITE ONE-HUNDRED-BYTE-REC FROM WS-100-BYTE-WORK.
...
STOP RUN.
Your output will be

1

I don't think that is possible. In your record area you have to reserve
space for the maximum record size, and somehow you have to tell your
filesystem that only a part of that space is to be written to the file.
For that purpose you have to use the "depending on" phrase.

The only other way I can think of (don't know if it works !!) is to make
several record entries with different sizes, like this:

FD	my_file.
01	record_1
	05 field_a	pic x(20).
01	record_2
	05 filed_b	pic x(100).

You could give it a try.

Regards,

Dirk

```
I also joined this conversation "late" - but in ANY (every) ANSI'85
conforming compiler, you can specify

FD file-name
      Record varying in size from n to m
           depending on ws-data-item
    .
(where "n" and "m" are the smallest and largest record sizes - excluding any
"internal" RDW or similar control word).

If you then move the "correct" size to ws-data-item (defined in
working-storage) - then the record that is written will be the "desired"
size.

NOTE WELL: To ensure that this works, you MUST use "RECORD VARYING IN SIZE"
and *not* "RECORD CONTAINS".  The latter may (is permitted) to produce
variable size records - but is not guaranteed to.
```
