1) You must define the same FD in both the "parent" (calling) and "child"
(called) programs. Each needs the EXTERNAL attribute. (Select/Assign must
also exist in each.)

2) You do *nothing* in the Linkage Section or CALL statement.

3) Once you have done this, you just put whatever I/O statements you want in
whichever program you want - nothing more complicated that this.
(Obviously, you logic needs to make certain that you have done a successful
OPEN before you do any other I/O - but it doesn't need to be in the same
program.)

I saw a reply to this that explained the syntax of how to use external on
the fd. If your question is how does this work, think about it this way.
When you open a file a file handle is returned from the O/S. When you use
the external clause that file handle, accessed by the file description in
one, two, or many programs now points to a global variable that holds the
file handle. It is just like defining any other external variables in
programs, it just ensures that myvar pic x is external points to the same
memory address as myvar pic x is external in another program.

› how do I do it?


An easy way is to do all the file i/o in a THIRD program.
In this third program you have the select statement and the FD, etc.
Then you pass the program TWO parameters: the second should
be the record, the first is small control block with the filename, an
operation code, (open, read, write, etc), a file-status returned,
possibly a position indicator if you do START (i.e. Next, Greater Than,
etc).

Now, your two original programs CALL the third one and you have
a portable solution. It is even more portable than before, because
you have hidden all the File details in the third program.

AFAIK on most COBOL platform, all FD entries are EXTERNAL and therefore
you (should)/could place all code handling a certain file in a
sub-module, called from one or several programs in your run-unit...
The module should of course be "state-aware" (using WS-switches)
regarding it the file is open or not..

At our shop (MVS/NT/OS2) we use this approach for most of the
applications accessing files..., a standard copy-area (interface) for
the data to be read/written is passed to a I-O module for a certain
file/files...
