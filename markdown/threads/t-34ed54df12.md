The point about using a computer for a straightforward conversion is that it
can do it quicker, cheaper & more reliably.

to develop in Cobol, but to supply a VB version with GUI enhancements & SQL
access to a potentially new market.

developments in a language that they understand.

I'm with Mr. Moore on this.

I am a longtime (and some might say, "outspoken") (I accept) critic of
'verb-for-verb' conversions. In the case cited - two common source
languages, lots of available talent - my main criticism of 'mindless'
conversions is especially appropos.

If the target language does not offer better ways to do things, then why
convert at all? And I can see no way in hell an automated process can
include the kind of logic which goes something like, " in COBOL, you use
global variables and fixed-size tables because that's all that's available.
VB permits locally scoped variables and dynamically sized tables, which are
(at least arguably) a marked advantage over COBOL. So we will convert the
globals and fixed-size tables to local data and dynamic tables."

There are dozens more examples of "better ways" to do certain things; and
here we are dealing with two very similar source languages. Well written
COBOL looks a lot like well-written BASIC. (Poorly-written COBOL looks like,
um, poorly-written COBOL).

Maybe there is a case for "move to VB for future maintenance needs," but it
sure seems to me that version 1.0 of anything - on which significant
maintenance is anticipated - should not be some contrived, bloated and
inefficient amalgam of some hard-wired verb-swapping.

It would be silly of me to argue the relative merits of modern Cobol
toolsets & VB in a Cobol newsgroup. I suspect that they'll both be all right
when they're finished.

The reality for users of many Cobol systems is still character-based screens
and critical company information locked away in legacy data files. There are
other solutions, but I obviously feel that from the users' perspective the
converted product would offer significant improvements.

The resultant package would be version n + 1. A few, choice, inexpensive
components could then add the sort of modern functionality that users now
expect.

The opportunities for improving the structure of code whist converting are
similar to those available without conversion. A more powerful conversion
can be obtained by replacing standard code elements, copy files & library
routines as logical entities rather than at the individual verb level, but
this requires a little more 'mindless' effort.

As an experienced COBOL programmer I decided a few years ago to enhance my
skillset to include VB. The straight-forward way to learn VB seemed to be to
take some existing COBOL programs & translate them into VB. That way I wouldn't
need to re-develop the logic, etc. Basically, this was the 'verb-for-verb'
translation you would most likely get from a conversion program. (BTW-this
worked as a learning tool. In some respects I would have learned more by taking
a VB class, but after seeing how 'programmers' who've taken VB classes write VB
code I think I was better off avoiding them. No discipline. It appears that the
classes focus on glitz, not on programming.)

Anyway, the result was basically character-based screens, just like you're
complaining about in COBOL. I also tried switching from COBOL indexed (VSAM)
files to VB files & it didn't work at all-performance was much slower. So I
ended up converting the files to a database, then realized that I could just as
easily modify the COBOL programs to use the database as rewrite them into VB, so
the end result was nothing more than a screen 're-design' and that didn't really
improve things any.

The reality is that a conversion to VB requires rewriting the programs to
achieve any benefits, and that requires people.
