Clark,
    Just a "personal" guess on this.  For MANY places where COBOL already
supports 64-bit addressing, there is little if any difference between
procedural locations and dataset locations.   I know that this is the MAJOR
motive behind 64-bit COBOL in "Unix-like" environments.  The COBOL programs
are "required" to address "large data files" via addresses (and hence 31- or
32-bit addresses limit the size of DATA FILES in such environments).

As far as IBM's OS/400 / i-series, I believe (but won't swear to it) that
they have large "pointers" and addresses in order to support "cross-system"
(separate processors) access.

Neither of these are really "relevant" in a z/OS (or similar) IBM mainframe
environment.

As far as current or future z/OS (or z/VM) "need" for 64-bits, my guess is:

 A) "RMODE 64" (if available for anyone) won't really be a need for COBOL
ever
 B) "AMODE 64" *might* be needed (or at least "useful") for applications
that
    - currently use and create temporary VSAM files (that only last for the
length of a single program - not even a jobstream)
   - programs that need to share data between COBOL and some other language
that DOES support it (e.g. C/C++)

Actually, Data in Virtual is a better approach.
This lets you map the full address range of n TeraBytes into a given
virtual storage range.

The benefit is that the system does not have to provide page space for
your address space as you can use Linear datasets to do it for you.

The additional benefit is that page swaps to your private linear
dataset are done by the system using system optimised IO.

DO NOT TRY using Main Storage as a replacement for IO until you have
talked it over with your systems programmers. You might find that the
system workload manager thinks you are an abnormal user of system
resources and blithly swaps you out for the duration.

If this is a purely Batch process and you are experiencing I/O problems,
wouldn't the first line of attack be to try and improve the VSAM
performance?  Talk to your DBAs and Systems people before you re-design this
application.

I don't know whether your system is supporting on-line access concurrently
with the batch (most mainframes are) but it is worth considering the
fundamentals:

1. On-line performance depends on 3 factors: load time, queue wait time, and
capture time. Without a great discourse on this, what it means is that
"SMALL is beautiful" and memory MUST be available for on-line modules to be
loaded. (The more memory you have, the less they need to be swapped.)

2. Batch processes should utilise as much memory as possible (which is
exactly what you are proposing) PROVIDED they do not do so at the expense of
the on-line system. So, if this is an overnight Batch run, no problem. ("BIG
is beautiful", for Batch)

But hang on a minute... Even if your process is overnight Batch and there is
no danger of it degrading other key processes, won't it require significant
changes to convert what are currently I/O areas into (VERY LARGE) tables?
Then you must provide a mechanism for loading these tables (and storing
them), change all existing access to be subscripted/indexed, and regression
test.

All of this can be done of course, but what I am saying is don't undertake
to do it overnight...<G>

If the problem is really that the VSAM random access is slowing everything
down, you should certainly discuss it with your DBA (he/she might be glad of
something interesting to do for a change...<G>). If this is the case, and
everyone agrees that the I/O has been optimised as far as it can, then you
could try progressively replacing the most frequent random accesses. (In
other words, I would approach this on a phased basis.)

Before you decide to go and use the entire address space, you MUST discuss
it with your system people. There are likely to be repercussions for JCL
(Job Classes), system paging, load balancing, scheduling, etc. etc...

Yes, your job will be allocated a Virtual space as big as you desire, but
that Virtual space still has to be mapped to real memory. It would be a pity
if you simply exchange an I/O problem for a thrashing one...

Talk to DBA, talk to systems, and Good Luck!

What I found out is this:
1. The VSAM files are ALL defined with one or more Alternate Indexes.

a. Defining the Alternate Indexes with UNIQUEKEY (no duplicate alternate
Index Keys)
b. Updating in Primary Key, Secondary Key sequence in Sequential, rather
than Random mode.

support from the Systems area. It would only take me about 6 months to fill
