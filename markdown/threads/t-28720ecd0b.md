At work, our philosophy is pretty much "let the program do what it can, report
what it can't, but abend only in the most dire situation." Bad data should
never cause an abend. However, recognizing the wrong input file or having the
master data base unavailable could be valid reasons for abending the job
(batch) or rejecting data (CICS) _after_ an appropriate message is displayed
somewhere where Operations (batch) or the terminal user (CICS) can see what is
clearly wrong.

But in no case should bad raw (unedited) data cause any program to abend.
(Well, almost no. If this is a critical control date, it may be worth causing
an abend after printing an appropriate message to let the operator know that
something has to be fixed before other jobs can be run.)

There is a grey area, though: computer-generated data received from another
site that should be perfectly good is sometimes handled without editing
routines. However, I think a certain amount of data validation should still go
on because we have seen abends caused downstream when an upstream editing
routine could have caught the problems.

And in absolutely no case should a programmer code an abend without producing
an informative message first.
