"Cobol Files" aren't in a format that can be read by an outside tool and
converted; they're just data.  The description of the data is usually
contained in a copybook, similar to an include header file that you may be
familiar with from the Unix/C environment.  Translating these files to ASCII
usually involves finding a Cobol programmer to build a set of conversion
programs by hand.  There are many issues to consider, such as EBCDIC/ASCII
conversion, and native numeric types ( COMP, packed decimal, signed zoned
decimal, etc).  There are plenty of people in this newsgroup who would gladly
provide this service to you for a fee. :}

Most of the UNIX COBOL compilers "default" to creating ASCII files.
However, if you are in an environment that has either non-ASCII data (EBCDIC
for example) or *some* fields that are Binary or Packed-Decimal. There there
is an easy and Standard way to convert these to "ASCII".

Simply define your input file however it was created and then make an output
file specifying:

   Code-Set - pointing to an alphabet defined in Special-Names as
"Standard-1"
   Define all numeric fields as Usage Display - Sign is Separate

Then move your individual input fields to your output file - and write it!

Unix is using ASCII as its default standard. In fact EBCDIC is not supported
unless it is supported by the application you are running.

If you actually have an EBCDIC file on a UNIX system, I would imagine that
it actually originated from one of two possible sources.

1) From a Mainframe and your emulation software allowed you to download it
without converting it (for whatever reason I don't know?)

2) You are using MicroFocus COBOL and you've got the EBCDIC standard set as
the default. If this is the case microfocus has an EBCDIC to ASCII converter
(I believe in their "samples" directory, or a utility).  Or you can write
your own....
