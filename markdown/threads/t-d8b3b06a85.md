```
(As a follow-on to one or more other threads).

Once "your" implementation supports the new EXIT PARAGRAPH option, I would
(personally) recommend the following coding rules.  I believe that a
(relatively) easy source-code-checker could verify that these rules were
followed:

1) *always* use PARAGRAPHS, never SECTIONS (2nd choice is to always use
SECTIONS and ONLY code one paragraph-header at the top of each and never
reference it).

2) Never code a PERFORM ... THRU statement (i.e. never code the THRU phrase)

3) Never code a GO TO statement

4) Only code one STOP RUN, EXIT PROGRAM (method, whatever) *or* GoBack
statement in your program (that means only ONE of those, not one of each)

  ***

If you followed these 3 rules, I believe you would be FORCED into structured
programming (and IMHO) easily maintainable source code.

 ***

Having said this, let me say that I completely "acknowledge" two problems
with this:

A) If you PERFORM a paragraph with a STOP RUN, (EXIT PROGRAM), GoBack -
then you have a "hidden" exit.  This violates at least SOME rules for
structured programming.  This can be avoided by coding your STOP RUN
(whatever) in your "mainline" code - but that often "hides"
end-of-processing logic.

B) These rules have NOTHING to do with maintaining existing source code.  I
(personally) guess that there is SIGNIFICANTLY more COBOL "maintenance" than
"new development" going on.  Although I think that "moving toward" these
rules will *eventually* help with maintaining existing code, I strongly
doubt that the effort to change existing code to these rules can be
cost-justified.

P.S.  I have MIXED feelings about the new "RESUME" statement.  It seems to
be a GO TO in some ways, but also could be viewed as changing a DECLARATIVE
from an (implicit) GO TO - to an implicit PERFORM.  My "best guess" is that
every (new) DECLARATIVE should end EITHER with a RESUME or a (Perform of) a
STOP RUN/GoBack.
```

Using PERFORM ABORT-ROUTINE can be preferable to using GO TO
ABORT-ROUTINE, because, when testing, ABORT-ROUTINE can be replaced by
a dummy procedure to display messages instead, so multiple errors can
be checked in the same run more easily, cutting down on the testing
time and also allowing secondary defences to be checked.

When using SECTIONs, which has been (and at present still is) my
personal preference, I don't think I have ever encountered compilers
that required a section to contain any paragraphs at all, and when I
have used paragraphs in sections more recently it has usually been to
mark the exit, except where in-house standards also required them at
the beginning.  The current version of the draft standard doesn't
require sections to contain paragraphs either.
