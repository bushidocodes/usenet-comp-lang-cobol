If you already know Cobol then I world recommend Mike Murach's CICS book.
(www.murach.com) I taught myself CICS by reading this book and it is very
good for the basics.

Maps/screens are defined using BMS - basic mapping services - i.e.
assembler language macros.  A physical map (load module) and a Cobol copy
book for the screen fields are produced by assembly of the screen.  Cursor
positioning is controlled via attribute bytes.  You can set initial values
in the map and can override them in the program.

Within the Cobol program  you  have EXEC CICS cics_command END-EXEC.  These
get translated into the appropriate CALL statements. CICS provides a lot of
different services and in particular services for accessing files.  If you
use a dbms already then that will help reduce the learning curve.  Many
programs simply SEND/RECEIVE maps, handle PF keys, edit screen field data,
and add, delete, or update files/databases.  You have to worry about things
like:

    Is this the first time thru?
    Has the transaction code changed?
    Has the key changed?
    Do I transfer to another program?
    In a pseudo-conversational program how does the program  remember
important data (commarea or     temporary storage)?
    What exceptional conditions do I need to handle for this cics command? 
etc.

Like I said the book gives you the basic coding techniques and example
programs of the most common type of transactions.

One one the "hard" things about CICS is that it uses certain tables to
define transaction ids and program names etc.  The  "hard" part is that
usually a system programmmer is involved in setting these things up and
this can be block to experimenting with practice programs. Some places make
this easier for the programmer to do this himself. You will have to check
out your local environment.  Having a sys prog friend may be helpful.

CICS is  the online system for mainframe machines like that of IBM,
AMDAHL...etc. It's just a system where the users
can enter data, enquire, update files, etc. Just like any Windows data entry
screens, except that you are
presented usually with a 3270 terminal - a dumb terminal (Usually used by
dumb users too!). A terminal with an 80X24 dimension.
CICS is an ordinary COBOL program.If you see a compilation of CICS, u can
see that it'll comment out the EXEC CICS ..
statements and create the COBOL (batch) equivalent and compiled just like
any other batch programs (with CICS libs linked to it of course).

So here it is, I've been programming CICS, batch COBOL, ASM-H for the past 6
years and have written some
pretty nice programs myself. This is what I learned first. Try looking in
ibm.com too.

1. Learn only how to write pseudo-conversational CICS programs. It saves
resources and your boss will like you.
2. Learn how files are accessed thru CICS, basically they are analogous to
batch programs BUT to check or handle
    file status, you use the CICS return code. So, learn a lot about
handles.
3. Next, learn how to create maps/screen maps thru BMS.
4. Learn how to use COMMAREA - this is usually LINKAGE SECTION in batch
4. Try to dissect the EIB resources
5. Since most CICS programs deals with data entry, and file browsing, it is
imperative that you learn paging.
6. Learn about queues - TSQs
7. Learn the above first, all of them. Then try the TWA, CWA stuff
