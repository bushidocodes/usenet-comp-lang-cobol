First of all, are all the records in the file the same length?
Probably not.
It is difficult in COBOL to have a file with records of different
length unless each record is proceeded by a LENGTH-ITEM.

Once you have gotten past that,
How do you know 'WHAT IS THE SEPERATOR?'?

If you can rely upon it always being some value, or always
being the FIRST n CHARACTERS,
then you set up a list of items of the maximum size each
all PIC X.

Then you do an 'UNSTRING DATA-ITEM INTO
item-1 DELIMITED BY WS-DELIM
item-2 DELIMITED BY WS-DELIM,
item-3 DELIMITED BY WS-DELIM'

This will split the DATA-ITEM into each of the item-n items
from LEFT to RIGHT, skipping whatever you have put into
WS-DELIM.

WS-DELIM can be a single character or several bytes!

    move space to field1, field2, field3, field4, ...
    unstring
        data-input-record delimited by "!"
    into
        field1, field3, field3, field4 ...

*******Gives you data right justified in field1, etc.

You still have to parse the resultant fields to make sure things are aligned
as you want.  The results of the above will be left justified and space
filled, so you will have to unstring the unknown lengthed numerics delimited
by space into a picture 9(n) numeric field, or use the function numval.  The
function will give you more reliable results if you have it.  That would
also take care of "."'s that are imbedded if there are any.  The example you
show has only integers, or data with an implied decimal place.  If there is
an implied decimal place, though, the data would have to be of a known
format/length.  If you are talking real free format, then numval makes it
easy.

It is what comes after the UNSTRING.

Donald Tees suggested FUNCTION NUMVAL, which is not 85 standard, but rather
an add-on to that standard which might not be available on a mainframe
installation.  My company's product, RM/COBOL, implementing the 1985
standard MOVE statement, has a "deedit" feature when moving from a numeric
edited item to a numeric item.  Now the "deedit" is required by the standard
(Section VI, Paragraph 6.19.4 (4)), but the committee did not provide
semantic rules for de-editing.  I would suggest that each vendor's
implementation is therefore proprietary to some extent, and the response
would need to be tailored to the particular vendor.

In private e-mail, the compiler was identified as Fujitsu v3 on a PC.  I
replied with the following suggestion:
=== quote ===
Then Donald Tees' suggestion is the best one:  UNSTRING the data delimited
by whatever the delimiter is, and where you wish to retrieve numeric values,
use the NUMVAL function on the result of the UNSTRING.

Your example was:   ST!130!00032001"

I presume that the first 'chunk', i.e. "ST" indicates the format of the
remainder of the item.  I would probably set up something like this:

01  EDI-ITEM pic x(100).
01  EDI-CURSOR  pic 9(4).
01  EDI-ST-FIELD1  pic x(30).
01  EDI-ST-FIELD2  pic x(30).
01  EDI-GPA-FIELD1  pic x(30).
01  EDI-GPA-FIELD2  pic x(30).
...
move 1 to EDI-CURSOR.
unstring EDI-ITEM into EDI-INDICATOR delimited by "!"
    pointer EDI-CURSOR
    on overflow <error action>
end-unstring
evaluate EDI-INDICATOR
when "ST"
    unstring EDI-ITEM into EDI-ST-FIELD1 delimited by "!" or quote
                                     EDI-ST-FIELD2
        pointer EDI-CURSOR
        on overflow <error action>
    end-unstring
    <rest of ST processing e.g.>

when "GPA"
    unstring EDI-ITEM into EDI-GPA-FIELD1 delimited by "!" or quote
                                     EDI-GPA-FIELD2
        pointer EDI-CURSOR
        on overflow <error action>
    end-unstring
    <rest of GPA processing>
...repeat 'when' clauses for the various item types...
when other
    <error action for unexpected item type>
end-evaluate

Assuming you have to deal with each item type differently, this technique
works well.  The variable EDI-CURSOR retains the information about how many
characters have been examined in EDI-ITEM so the subsequnt UNSTRING(s) know
at which character position to start.

The UNSTRING verb also has a COUNT IN phrase that allows you to know how
many characters of the sending item (EDI-ITEM) were used in the move.  There
is also a DELIMITER IN phrase that will store the delimiter that terminated
the field.  Both of these phrases are most useful when you use a separate
UNSTRING verb for each field you are separating out of the sending item.

The bottom line is:  UNSTRING is a quite powerful tool for doing the job you
appear to have before you.

=== unquote ===

I process EDI files, such as EDIFACT standard BAPLIEs and such using a
SEQUENTIAL file with a 256 character fixed size.  I read blocks into
memory and tile these into working-storage 512 byte buffer.  This caters
for non-standard or no line terminator.  BAPLIEs and such are in
'segments' with an apostrophee (or other nominated character) as the
terminator. There may or may not be CR and/or LF.  Everything after an '
(or nominated) up to the next segment start is to be ignored. LINE
SEQUENTIAL is of little use.

The only problem is when the file is not an exact multiple of 256
characters and an exception occurs on the last block.  At this point the
last block has been read up to actual end of file.  The EDI file has a
segment marking end of data (UNZ) and so physical end of file is
unrequired.

This all works in ANSI 74 without any extensions.
