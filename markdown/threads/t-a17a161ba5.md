```
I am a long-time mainframe assembler programmer and I'm looking
for a way to simulate something that is very easy to do in assembler, but
cannot seem to do in COBOL.
I can create a table that contains values that I will
match against and address constants (like a pointer) of routines that I
can PERFORM when I find a match against the value. I simply load the
value of the address constant and use that as an address to branch to.
I cannot seem to find an equivalent in COBOL. The only solution
I can come up with, that does not really solve the problem it just goes
around it, is to use a very (very, very) long EVALUATE statement. This
defeats the purpose of using a table.
I hope someone can help. Remember, I must be able to do the
equivalent of a PERFORM, not the ON variable GO-TO process. I do my best
to avoid long EVALUATES (several pages) and the GO-TO in all forms, so
don't bother suggesting it.
```

```
I don't believe there is any way to do this in COBOL. The target of a
PERFORM verb cannot be a variable, it's fixed at compile time.

I don't know if there is something in the 97 standard that will cover
this (Don Nelson?).

If the routines you wanted to perform were all dynamically callable
sub-programs, you could table the subprogram names up, and move a
subprogram name into a work variable, and call it.

MOVE BRANCH-TABLE (MY-INDEX) TO SOME-PROGRAM.
CALL SOME-PROGRAM USING GENERIC-PARMLIST.

If your compiler supports nested programs you could make all the
subroutines nested programs, table up their names and do something
similar to the above.
```

```
I will, just in case you are stuck, and in case anyone else is
interested: You can sort-of simulate a PERFORM depending on, using GO TO
DEPENDING ON by using something like:

SEC SECTION.
GO TO P1, P2, ... DEPENDING ON N.

P1.

GO TO SEC-X.

P2.

GO TO SEC-X.

...

SEC-X.
EXIT
.

Now, if you PERFORM SEC you have the equivalent to PERFORM DEPENDING ON
with the limitation that all of the things you might want to perform
have to be paragraphs inside the section.
```

```
I am a COBOL and assembler programmer, too. I understand what you are
trying to do, but COBOL doesn't allow using a table of address pointers to
subroutines in a PERFORM statement. You need to let the compiler and linker
do all the address manipulation for you.

If COBOL _did_ support what you wanted, it would probably be a

PERFORM a b c ... DEPENDING ON index

but unfortunately, there is no such animal.

One way to do what you want is to code the performed routines as individual
programs, then use a dynamic CALL statement using an indexed program name,
i.e.,

CALL sub-program(routine-index)

You would still not be using program addresses directly, but it probably
would simplify the logic of the calling program.
```

```
This reply is in reply to all of the others that gave various
suggestions (and one requested that I reply). In COBOL 98 (or
whatever it will be - definitely not 97), program pointers exist but
not procedure programs. That is, you can set such a pointer to a
program-name that is visible to this program. You could write a bunch
of very small contained programs - if all the data is global the
following is all you need:

PROGRAM-ID. whatever.
PROCEDURE DIVISION.
a-name.
...
END PROGRAM whatever.

Note that no identification division header is needed. We were also
supposed to get rid of the requirement for a paragraph-name, but that
proposal has not yet passed and probably will not make it.

So, the answer might be to use program pointers. I might mention that
we did not add procedure pointers (pointers to paragraphs and the
like) because they are fraught with problems. Like what happens when
you pass one to another program, add to one, etc.

By the way, GO TO ... DEPENDING ON has never been removed from the
standard. It is the same as it always was, and requires that the
depending on data item be numeric as it always has - any compiler that
allowed alphanumeric characters in the item was in error.
```
