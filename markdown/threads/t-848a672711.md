Doesn't look varying to me.

It IS possible that the record is terminated by a CR/LF representing
blank-filling.

It is also possible that the LTH-RECORD-TYPE provides a clue to the record
length ("A" implies "x" bytes, "B" implies "y" bytes, etc.), but this is a
really-weird way to do things.

In COBOL, as with other languages, text files have a standard delimiter (CR
or CR/LF). Absent that, there must be some explicit or implicit declaration
of the record length. Sometimes this declaration is explicit in that the
record length is carried as a data element within each record. Sometimes
this length declaration is implicit in that the record is presumed in
advance (by the program) to be a certain length.

This latter method SEEMS to be the case here.

Then, too, programmers can get tricky:
1. The first record (H) in the file is always 37 bytes.
2. This is followed by five 120 byte records (A).
3. Each of these is followed by 2 60-byte records (B) UNLESS the first
record has a "7" in byte 12. If "H" contains a "7" in byte 12, then each of
the "A" records is followed by 1 1200 byte record (C).

You see this kind of code written in the '80s by programmers having LSD
flashbacks from their '60s and '70s mis-spent youth.

There are several ways to do this:

1. Use the program to generate a single record file and a two record file
with special first char and last char. Then use a hex editor, you will find
the structure of the file. The file maybe in one of the structures: records
are separated by a special delimiter CR/LF, or a length field in first
several bytes. If you are lucky, you can find the structure right away. The
worst is the file format is system (compiler or OS) dependent.

2. A good way is write a COBOL program in the same system and write it out
as pure TEXT file with delimiter you know. This is a safe way, but you have
to do some Cobol programming.

By the way, if you want to do Cobol source code to Java conversion, check
http://www.corporola.com.

Good luck.


Thomas Li
