One of the big problems with performance testing, is making
sure of that which is being tested. We had a similar issues in
a thread titled "Right Justify Routine," during September 1998.
The results I posted were:

"The differential for Function Reverse in MF 3.2 comes to about 70:1
when following the requirement for the use of a "temporary elementary
data item." When treating it "as if" it used a "temporary elementary
data item," the ratio is about 106:1 when the argument and receiving
operands are different and 125:1 when the argument and receiving
operands are the same."

The question then was "Why is there such a big difference in
performance?" I was unable to explain all the difference, at that
time. Further analysis showed a nearly 3:1 difference between
static and dynamic linking. The balance appears to have been
related to dynamic allocation of the required temporary variable,
the overhead of the (compiler generated) call, checking for the
selection of character set (single-byte versus multi-byte), and
checking for crossing segment boundaries on a 16-bit
implementation.

The conclusion was that, if one is uses Function Reverse in
a loop, performance will be reduced. Furthermore, the tighter
the loop, the greater the loss of performance.

This is the same issue I raised concerning the use of OO.
If the general process model for an application is record
at a time processing, the use of OO to implement that model
will cause a loss of performance due to memory allocation
overhead. The amount of loss will depend upon the duration
of the object. (How tight the loop is.)

If one happens to use Function Reverse in a tight loop, one
could reverse the loss of performance by replacing the
function with tailored code. The same is not true of OO. This
is because the decision to use OO and the design of the class
is, somewhat, independent of the applications where it will be
used. Though this provides more flexibility for creating new
applications by "reusing components," it removes the ability
to optimize loops to reverse performance loss.

Just as the testing, I did, proved nothing about the
desireablility of using, or not using, Function Reverse; no
simple performance test will demonstrate whether or not
to use OO. This is because the tests do not reflect reality.

What would be useful for clarifying the performance issues
is for a testing organization to create a few identical
applications using modular and OO techniques. If these
applications can be shown to simulate "real" applications,
then the results should show "real" relative performance.
(No, I am not volunteering!)

At best, I have only seen *sketchy* evidence that the use
of OO *may* have been *partly* responsible for some
publicized failures of new C/S applications that were
intended to replace mainframe applications. Even this
"evidence" did not clearly blame performance as the
reason for the failure.
------------------
Rick Smith

Academia may not be a good starting point; "After all, what
do Computer Science professors know about COBOL.
They don't teach it!"

Assuming that a test is constructed and results published;
many will not accept the results. Some will question whether
any such test adequately reflects the needs of *their* business
(or business in general); whether computer performance is
more important than productivity; whether the sample
organization model and the OO and modular designs
compared are correct; etc.

Furthermore, it may not matter if (the proverbial) *you* like OO,
if *you* feel it increases *your* productivity, and if *you* are
willing to sacrifice a little performance. There will always be
a Shirley or Bob with a story about how *your* new program is
too slow and is making it impossible for them to do their job.
It is difficult to be proud of your work when your customers are
complaining.

I think the best that can be accomplished, in the near term, is
to identify where and why using OO might affect performance.
This allows individuals to determine whether to construct
their own tests using real problems for comparison. Perhaps
some will share their results.

In the long term, we will see the results in stories and job
postings.

"batch" type applications (but certainly CAN be used in real-time ones as
well).

4)  In general, J4 (and WG4) have removed ALL "optional" facitilites from
the next Standard - along with removing the concept of "subsets".  The
theory is (or so it goes) that a feature is either in the Standard or it

The only time I ever did a head-to-head comparison
of runtime performance in COBOL versus C, C lost.
What I wanted to accomplish, at that time, was to
find the fastest language for implementing a particular
algorithm.

The C compiler was MS C 6.0. The COBOL was
MF COBOL/2. It was about 1990 or 1991.

I did a direct (naive) translation of the COBOL program
into C. The C program was about half as fast.

I then modified the C program to adjust for the standard
runtime features of MF COBOL. This included buffering
of I/O for sequential reads and writes, reducing the frequency
of displays to the console to compensate for the speed
of direct versus indirect screen output, etc.

Then I used available C language features to speed
other operations. This mostly involved using pointers
instead of subscripts for array (table) operations. At this
point, the C program was slightly faster than the COBOL.

I then looked at the assembler listing for both languages.
I found that I could optimize a critical COBOL in-line perform
by replacing it with loops using GO TO. This made the
COBOL program about 30 per cent faster. At that time,
I could find no additional changes, I could make, to the
C program, without resorting to assembler.

Much later I discovered a few changes that would have
increased the performance of the C porgram. I also
discovered additional changes I could make to the
COBOL program. I did not implement those changes
because I was more interested in modifying the
alogorithm.

I am at a loss to explain your experience with COBOL
performance being much worse than C. Obviously, my
experience is that compiled COBOL and compiled C
should have comparable performance for the same problem
using identical alogoritms.

As a practical matter, I think a 4:1 difference for COBOL
versus C would be tolerable with a version 1.0; if the
difference is primarily in internal performance that can be
optimized later. I think the target should be within 2:1 for
a version 2.0.

My rationale for these numbers is that, initially, it is better
to have something that works. Once it works, make it
run faster. Also, for business programming, a programmer
who knows both COBOL and C will not choose C based
solely upon performance. However, given multiple COBOL

selection process; that is why performance must improve
in a later release.

There are 2 issues John.
1) How fast (and accurately) you can develop the program
2) How fast will the resulting program run.

With regard to the first issue (and excluding ALL external influences
such as portability, compilers available, shop standards, etc) only you
know where your proficiency, and efficiency, lies.  Being bi/multi-
lingual can be advantageous but with programming, like any language, I
suspect you will find 1 language that you consider native or prime.

With regard to the second issue (and again excluding the external
influences) inside every computer is a "clock" that "beats" at a very
precise rate.  Each "beat" is a "cycle". The "speed" of the clock
determines the number of "cycles" available during a given time
segment.  For example, a 233Mhz system will provide 233 million cycles
during a 1 second time segment.  Each "instruction"/"action" which the
computer performs (at the lowest level) requires 'n' cycles to complete.
'n' is fixed for any given "instruction" performed by a given processor
(but NOT, necessarily, the same value for the same instruction
performed on different processors.  For example, 80286 vs Pentium would
yield different 'n' values for the same instruction).  That said; the
'speed' of the compile will be determined by the overall efficiency of
the compiler with regard to its usage of cycles.  Consider, it will
take more cycles to load a long value into a register (at the low
level) than to load an integer into that register.  Obviously, a
compiler which always deals with longs when integers would do the job
will utilize more cycles than one which has differentiated and
optimized the use of the cycles.  The same type of scenario will hold
true with the runtime (interpretive or directly executable) application
programs which have been generated.  That is; the more "efficient" the
use of cycles the "faster" the program will be.  Remember, the number
of cycles available during a given time segment is static. So too, are
the number of cycles each low level instruction will require.  So too,
given a given compiler (COBOL, C, C++, JAVA...), are the number of low
level instructions which will be generated from a given higher level
instruction (  if IsDate(txtMainNm.Text) then ... )when either compiled
or interpreted.  [It should be apparent that compiled would be faster
because interpretation will require the additional cycles to do the
interpretation].

That said: the bottom line is - the best language to use is the one
which provides the optimum results for the task at hand.  In some cases
the task at hand will be "I need report x in 2 hours" and in other
cases the task at hand will be "you have got to make that x run
faster".  Having some 'sense' of advantages/disadvantages of various
languages both in terms of ease of use and in terms of how a language
"works" is very important and will be acquired through your own
experience. From your comment/question I suspect you already knew that
but were just hoping it would be easier!  It isn't.  After all; only
experience will tell you that the same language, from the same vendor,
used to develop the same application with the same code, will yield
different results (with regard to "speed"/"efficiency") even on the
same hardware platform dependent upon such factors as configuration and
the sequence of activity which came before.  [And I don't even want to
get into various methodologies of writing specific applications].

Good luck!!  Have fun!! (because if you're not you're in the wrong
business)
