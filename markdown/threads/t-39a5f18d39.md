First i don't know where you get the 'aa parameter can not be a table or
coulmn'.
That is exactly what Dynamic SQL is designed for.

You just don' thave a good example to go by:

          Parameter markers:  Although a statement string cannot include
references to host variables, it
may include parameter markers. These can be replaced by the values of host
variables when the
prepared statement is executed.  A parameter marker is a question mark (?)
that is used where a host
variable could be used if the statement string were a static SQL statement.
For an explanation of how
parameter markers are replaced by values, see "OPEN" in topic 6.28 and
"EXECUTE" in topic 6.20.

          Rules for parameter markers:

             Parameter markers must not be used:

              -   Anywhere in a select list (SELECT A+? is invalid)

              -   As an operand of the concatenation operator

              -   As both operands of a single arithmetic or comparison
operator (WHERE ? = ? is invalid)

              -   As an operand of a unary minus

              -   As an operand in a datetime arithmetic expression

              -   In the ESCAPE clause of the LIKE predicate.


             At least one of the operands of the BETWEEN or IN predicates
must not be a parameter marker.

             An argument of a scalar function cannot be specified solely as
a parameter marker.  For  example,   VALUE (COL1, COL2, ?) is not valid.
             If a scalar function is used in other than a SELECT list, and
it has an argument that can be
              specified as an arithmetic expression, a parameter marker can
be included in that expression,
              provided that it is the operand of an arithmetic operator and
that the other operand is a number.



Here's an example:
01  MY-SQL.
    05  Length-sql-string       PIC S9(4) COMP.
    05  filler                             PIC X(I w'm not counting) VALUE
        "SELECT * FROM EMPLOYEE WHERE  (:? = 0 OR ? = ? ) AND ".
    05  filler                              PIC X(still not counting) VALUE
        "   (?  = ' ' OR ? = ?) ".
01  WS-ITEMS.
    05  WS-PARM1                PIC X(07) VALUE 'SALARY'.
    05  WS-PARM2                PIC X(04) VALUE 'ZERO'.
    05  WS-PARM3                PIC X(07) VALUE 'DEPT'.
    05  WS-PARM4                PIC X(01) VALUE ' '.
    05  WS-PARM5                PIC S9(9) COMP VALUE +0.
    05  WS-PARM6                PIC X(05) VALUE 'SALES'.

MOVE 101 TO Length-SQL-String.
MOVE 50000 TO WS-PARM5.
EXEC SQL     PREPARE SQLSTMT FROM :MY-SQL END-EXEC.
EXEC SQL     EXECUTE SQLSTMT USING :WS-PARM1, :WS-PARM1,  :WS-PARM5,
                            WS-PARM3, :WS-PARM3, :WS-PARM6
END-EXEC.

NO it is NOT.
That example is evaluated at COMPILE/BIND time.  DB2 determines at that
time which Cols / indexes to use.  The only thing not determined is the
values
that will be compared for.  The DB2 optimizer used at compile/Bind time
looks only at what IS, at that time.  Indexes add, changed, etc are NOT
used at run-time.   Your example

The example I provided allows ANY Column to be used!
It also requires a run-time BIND and will therefore get the LATEST
RUNSTATS from the system.  This means changes to indexes,
plans, etc are picked up.

You could change:
     05  WS-PARM1                PIC X(07) VALUE 'Years-School'.
     05  WS-PARM2                PIC X(04) VALUE '0012'.
     05  WS-PARM3                PIC X(07) VALUE 'Awards'.
     05  WS-PARM4                PIC X(01) VALUE ' 3'.
     05  WS-PARM5                PIC S9(9) COMP VALUE +0.
     05  WS-PARM6                PIC X(05) VALUE ''.
and EXECUTE THE PREPARE again
and get a totally new statement.
