OK, let's try...

1. The linkage between COBOL and C is OK. Depending on the Compiler
level of both compilers, you could even omit the #pragma linkage in the
C Code.

2. There is no error in the calling COBOL program. Since you do nothing
else than call the C-module, you cannot have a mistake here.

3. Ergo: there must be an error in the C-module, I'd guess a storage
overlay.

It is not so seldom that a program containing some errors runs OK in one
environment (batch), but not in another (CICS). You can even see a
problem in a program that 'runs OK since years and has never been
changed' if you have e change in the environment (COBOL runtime to LE or
even another level of LE runtime). The error is mostly in the user
program and the real error was that it did run for some time instead of
abending.

This kind of error is very hard to find. You could try:

- have a code inspection of your program by an experienced programmer
- debug your code using a tool like Xpediter or Debug Tool
- comment part of your code until the cropped part runs OK, then
uncomment half of the incative code until you find the error
- call for IBM support which may cost some $, but might be cheaper than
search many day without finding the error

Good look!

The main difference between batch and CICS (from a technical view) is
that under CICS, programs are used reentrant. If you have static storage
in your C-Module, you have to set the compiler-option 'RENT' both in the
C and the COBOL compile. If it is not the current C/C++ for OS/390
compiler, but rather a C/370 compiler, you should omit static storage.

You could also try a dynamic call from COBOL to C:


WORKING-STORAGE                SECTION.
01 BCOENDEC PIC X(8) VALUE 'BCOENDEC'.
....

PROCEDURE                      DIVISION.
....
CALL BCOENDEC            USING
                 WS-RC WS-TYPE WS-KEY WS-OUTDATA WS-INLEN WS-INDATA

If you do it like that, you have to define BCOENDEC to CICS (Language
Le370). So you give CICS the possibility to initiate the C-program. In
you shop, there might be a restriction which forbids dynamic calls in
CICS. If so, tell the CICS-peaople that this restriction is outdated and
that IBM recommends dynamic calls rather than EXEC CICS LINK.

There might be other problems if CICS is not configured correctly for
using Language Environment like mixing of old and new runtime in STEPLIB
and DFHRPL. You could also try to define the COBOL programs as Language
Le370 instead of COBOL.

Really hard to find the bug from here... All I can say is: It should
work ;-(
