```
Chapter 1

What is Multi-Threading?
========================

Good question. When we refer to multi-threading we can be referring to a
whole
variety of things. There are two major topics, though, multi-threading
from an
operating system perspective and multi-threading from an application
perspective, the following section will discuss the former, while the rest
of
this document will deal with the latter.

Multi-Threading and the Operating System
----------------------------------------

First and foremost, we are referring to a facility provided by the
operating
system which allows an application to create threads of execution within a
process (a single running application). But what is a thread? Its best to
step
back and take a look at what a process is and how the operating system
handles
multiple processes running at the same time.

Most modern operating systems allow many different users to be running
programs (processes) at, what appears to be, the same time. If the
operating
system is hosted by a hardware system with a single Central Processing
Unit
(CPU) then these programs cannot, obviously, be running at exactly the
same
time - it just appears that way to you. The operating system is
responsible
for allocating the CPU and other system resources to the programs that
need
them and (preemptively) switching these resources from one running program
to
another 'every now and then'. This 'every now and then' is what gives the
appearance that the programs/processes are running at the same time.

In a very similar way, many modern operating systems allow multiple
threads to
exist within each of these individual processes. Again, the operating
system
handles the allocation of the CPU and various resources to individual
threads
and switches these allocations from one running thread to another 'every
now
and then'. Again, this gives the appearance that the threads are running
at
the same time.

So processes and threads get similar treatment from the operating system.
What
distinguishes them from each other? The major thing is the nature of the
resources shared between processes running within the Operating System and
those shared between threads running within a single process. All
processes
have distinct address spaces, open files and other system resources - this
allows each process to run with minimal or no effect on other processes
running at the same time. So a running copy of the program Solitaire will
have
no effect on a word processor or a data-base server running on the same
machine. This separation of resources has a definite run-time cost -
firstly,
each process consumes system memory for its address space and, secondly,
the
operating system must update a lot of internal information whenever it
switches execution context from one process to another.

On the other hand, because threads run within a single process, all of the
process' threads share the same data and code address space, the same open
files and most other resources. Each thread does have its own register set
and
stack space. So, overall, creation of a thread does not require extensive
system memory and when the operating system finds it has to switch
execution
context from one thread to another within the same process, about the only
thing the operating system needs to do is switch CPU ownership and
register
sets. This is the reason that threads are often called 'light-weight
processes'.

Multi-Threading and the Application
-----------------------------------

A multi-threaded application is an application whose architecture takes
advantage of the multi-threading provided by the operating system.
Usually,
these applications assigns specific jobs to individual threads within the
process and the threads communicate, through various means, to synchronize
their actions. For example, a GUI data processing application might be
architected such that the User Interface is completely handled by a single
thread, while the actual work of the application is performed by another
thread. This architecture would allow the complete separation of the
business
logic from the user interface logic within the application.

Another interesting use for multi-threading within an application is in
the
server side of a client-server application. The server could be designed
so
that for each service request submitted to a main
controller/communications
thread, a separate service thread is created to process that request and
communicate the results back to the controller. The controller would then
communicate these results back to the client. The main benefit here is
that
the server is not tied up acting on one client's request to the extent
that it
cannot respond to other client's requests quickly.

Of course, client-server applications that have immediate (or very fast)
server response have been written without multi-threading for years. This
is
usually accomplished by a main controller/communications process creating
separate service processes for each client request. So why does an
application
like this need multi-threading? The main answer is: Efficient use of
system
resources. As was discussed before, the creation of a thread requires very
little operating system resources (memory, context swap overhead and so
on)
and the sharing of files and other resources is very easy. The creation of
a
process requires more system resources and it tends to be much more
difficult
to communicate and share files between processes. So, multi-threading in
applications such as this allows you to make the best use out of your
existing, limited, hardware resources and also allows simple resource
sharing.

There is a down-side to all this though. Each thread in a multi-threaded
application must be (painfully) aware of the resources that it might be
sharing with other threads in the same process. The programmer must
remember
that threads execute at the same time and that two threads writing to the
same
data item, without any form of execution synchronization, will likely
result
in the corruption of the data item.

For example, consider the following code fragments and their possible
order of
execution:

Working-Storage Section.
01 group-item.
05 field-1 pic x.
05 field-2 pic x.

*> Thread 1 execution *> Thread 2 execution

MOVE 'A' TO field-1 *>1st MOVE 'C' TO field-2 *>2nd

MOVE 'B' TO field-2 *>3rd MOVE 'D' TO field-1 *>4th

DISPLAY group-item *>5th DISPLAY group-item *>6th

In this case, neither thread will display their 'expected' results. While
Thread 1 expected to display 'AB' and Thread 2 expected to display 'DC',
both
threads will display 'DB'. So neither thread achieved their intended
results.
This is where the best part of multi-threaded programming comes into play.
That is synchronization of execution between threads and
avoiding/resolving
data contention between threads.

Chapter 2

Synchronizing Execution and Resolving Contention
================================================

Uncontrolled multi-threading within an application can cause unpredictable
end
results. There are a variety of ways to synchronize execution and resolve
data
contention between threads that help achieve predictable application
results.
The methods you use rely on the characteristics of your application's data
access and the requirements for true multi-threading within the various
application modules (the individual COBOL programs).

The following sections outline some of the techniques that can be used
within
your application to achieve predictable results. These techniques include
assigning each program and data item one of the various multi-threading
attributes and using the various synchronization primitives.

Use of Program Attributes
-------------------------

There are three main program attributes that affect multi-threading within
a
COBOL program. These are attributes are assigned by including or omitting
one
of several compiler directives on program compilation. The directives used
affect the allocation of the program's system work areas and can cause
automatic system locking of the compiled program.

By default, the first program attribute provides no provision for
multi-threading and is achieved by specifying no special compiler
directives.
This option causes system work areas to be allocated statically and so
they
are subject to contention. This approach has several advantages, call
speed
and stack usage are among them, but it is up to the user application to
make
absolutely sure that only one thread executes within a non-multithreading
program at a time. This can be accomplished through implicit program logic
in
any calling program, for example, when the application, by design, has
only
one thread calling a non-multi-threaded program. Alternatively, one of the
synchronization primitives (such as a Mutex) can be locked in the calling
program just before the called program is entered and then unlocked when
the
called program has returned.

The second attribute is assigned by specifying the SERIAL compiler
directive.
This directive causes system work areas to be allocated statically but
also
causes the program to be locked on entry and unlocked on exit. This ends
up
allowing only one thread to execute the program at a given time and so
eliminates any contention on system or user work areas. No other explicit
application logic is required. This technique has the advantage of
allowing
traditional COBOL programs to be included in a multi-threaded application
without any source changes. One of its major disadvantages, though, is the
limiting of the level of multi-threading within the application. Ideally,
a
multi-threaded application should allow free execution of as much code as
possible to allow maximum performance.

Another major disadvantage to the SERIAL attribute is the run-time cost of
locking and unlocking the program - this can significantly affect call
speed
and overall application performance. For this reason, as few modules as
possible should be compiled with this directive and heavy reliance should
be
placed on specifying no multi-threading attribute for called programs. For
example, if program A must be compiled with the SERIAL attribute and
program A
is the only program in the application that calls program B and C, then by
application design, program B and C are already serialized and can be
compiled
with no multi-threading attribute (as discussed above).

The final attribute is given by the REENTRANT compiler directive. This
directive causes all system work areas to be allocated dynamically on the
stack (eliminating thread contention on these areas) and allows multiple
threads to enter the program at the same time. This is the directive that
the
programmer would want to use for most (if not all) of the modules in a
multi-threaded application. The only problem with reentrant programs is
that
the program itself must resolve all possible data contention on its
WORKING-STORAGE and FILE SECTION. One or more of the other techniques for
resolving data contention must be used to accomplish this.

Use of Data Attributes
----------------------

The attributes of a data item determines if there is possibility for
contention on that data item amongst the threads that access it. There is
never a possibility for thread contention on data items defined in the
LOCAL-STORAGE or THREAD-LOCAL-STORAGE SECTIONs or on data items defined
with
the THREAD-LOCAL data attribute. Any other data items defined within the
program are shared between threads and so there is possibility for
contention.
This includes WORKING-STORAGE items, FILE SECTION items, some compiler
special
registers (except the RETURN-CODE register) and other various implicitly
defined data items such as file status fields.

Data defined in the LOCAL-STORAGE section is traditionally used in
recursive
COBOL programs, for every recursion of the program a new instance of this
data
is allocated on the stack. Since each thread in a multi-threaded
application
has its own stack, this attribute is ideal for defining contention-free
temporary and work items within reentrant programs. By design, a reentrant
COBOL program also allows recursion within a single thread.

There are two disadvantages to using data defined in the LOCAL-STORAGE
SECTION. Firstly, it can't be initialized with value clauses (any value
clause
is treated as documentation only) and, secondly, it disappears when the
reentrant/recursive program exits. After an exit, on the next entry into
the
program, the LOCAL-STORAGE field values will have undefined values. So if
a
thread requires the program to preserve state across calls, some other
mechanism must be used.

The THREAD-LOCAL attribute and the THREAD-LOCAL-STORAGE section resolve
these
problems. THREAD-LOCAL data is unique to each thread, is persistent across
calls and can be initialized with value clauses. A data item defined as
THREAD-LOCAL can be viewed as a thread-specific WORKING-STORAGE data item.
This kind of data is very useful for resolving contention problems in most
reentrant programs. In many cases, a non-file-handling program can be made
completely reentrant by simply changing the WORKING-STORAGE section header
with a THREAD-LOCAL-STORAGE section header. The programmer can fine-tune
data
allocation by defining all read-only constants in WORKING-STORAGE and all
read-write data items as THREAD-LOCAL.

There are some disadvantages to the use of THREAD-LOCAL storage though.
Firstly, it does not allow easy communication of data values between
threads,
WORKING-STORAGE with appropriate thread synchronization should be used for
this. Secondly, its use can add quite a bit of an overhead to program
calls -
if it is used in a high-traffic module, then overall application
performance
can be seriously affected. Finally, it is not possible to have EXTERNAL
items
defined as THREAD-LOCAL. Currently, if thread-local data must be shared
between application modules, the CBL_TSTORE_ API must be used (more
details on
this in section Creating thread-local data on page 1).

Of course, sometimes threads require more than just private data. Most
multi-threaded applications communicate and coordinate thread execution
through shared data that is accessed by each thread under a strict
protocol.
In COBOL this is accomplished through the use of WORKING-STORAGE or FILE
SECTION data in conjunction with the use of various synchronization
primitives
that resolve any possibility for destructive contention on the data.

Use of Synchronization Primitives
---------------------------------

Execution synchronization of threads accessing shared data is critical for
predictable results in a multi-threaded application. Understanding the
nature
of the various data accesses between threads is the first step in
determining
what synchronization primitives and regimes should be used. Once the data
access has been characterized, it is a simple matter to follow a
synchronization regime for data items among all threads. The following
sections outline common data sharing problems and their solutions.

Using a Mutex
-------------

The simplest form of data sharing problem is when multiple threads require
mutually exclusive access to shared data at some point during their
processing. The area of code that accesses this shared data is called a
critical section and these critical sections can be protected by the use
of a
Mutex which is logically associated with the shared data items (the name
Mutex
comes from Mutual Exclusion). The Mutex associated with the shared data
items
is locked before a critical section is entered and unlocked when that
critical
section is finished.

It is vital that all threads lock the protecting Mutex before accessing
any of
the data that is being protected. If even one thread fails to follow this
regime, then unpredictable application results could ensue.

The following code illustrates the protection of two critical sections
that
access a table, adding to it or counting items in it. The working storage
data
items table-xxx are protected by table-mutex.

Working-Storage Section.
78 table-length value 20.
01 table-mutex usage mutex-pointer.
01 table-current pic x(4) comp-x value 0.
01 table-1.
05 table-item-1 pic x(10) occurs 20.

Local-Storage Section.
01 table-count pic x(4) comp-x.
01 table-i pic x(4) comp-x.

*> Initialization code executed while in single threaded mode
move 0 to table-current
open table-mutex

*> Add an item to table-1, this is a critical section
set table-mutex to on
if table-current < table-length
add 1 to table-current
move 'filled in' to table-item-1(table-current)
end-if
set table-mutex to off

*> Count items in table-1, this is a critical section
set table-mutex to on
move 0 to table-count
perform varying table-i from 1 by 1 until table-i >
table-current
if table-item-1(table-i) = 'filled in'
add 1 to table-count
end-if
end-perform
set table-mutex to off

It is worth noting that, in the last example, the Mutex was definitely
required to prevent a thread from adding to the table and another thread
from
looking at the table at the same time. The only problem is that in using a
Mutex we are severely limiting the level of multi-threading within the
application. Ideally, in order to maximize multi-threading in applications
whose data access is similar to this, multiple threads would be able to
count
items in the table simaltaneously, but no counters or other adders would
be
allowed access to critical sections while one thread is already adding to
the
table.

This illustrates a case of the Readers and Writers synchronization
problem,
which has been written about extensively in operating system text books. A
solution to this problem would be given by a synchronization primitive
that
allows multiple threads to be active in read-only critical sections but
prevents any other access when a thread is active in a writer critical
section.

Using a Monitor
---------------

Monitors provide a clean solution to the Readers and Writers Problem.
Basically, they are used by having the critical section 'declaring' what
type
of data access the critical section will be performing on the protected
data
i.e. reading or writing (or browsing, but we'll get into that later). The
following code is an example of a monitor being used to allow multiple
counters on the table but disallowing any adder when a counter is active
and
allowing one adder but disallowing any counters (readers) when that adder
is
active.

Working-Storage Section.
78 table-length value 20.
01 table-monitor usage monitor-pointer.
01 table-current pic x(4) comp-x value 0.
01 table-1.
05 table-item-1 pic x(10) occurs table-length.

Local-Storage Section.
01 table-count pic x(4) comp-x.
01 table-i pic x(4) comp-x.

*> Initialization code, executed while in single threaded mode
move 0 to table-current
open table-monitor

*> Add an item to table-1, this is a writer critical section
set table-monitor to writing
if table-current < table-length
add 1 to table-current
move 'filled in' to table-item-1(table-current)
end-if
set table-monitor to not writing

*> Count items in table-1, this is a reader critical section
set table-monitor to reading
move 0 to table-count
perform varying table-i from 1 by 1 until table-i >
table-current
if table-item-1(table-i) = 'filled in'
add 1 to table-count
end-if
end-perform
set table-monitor to not reading

There is a fairly straightforward extension to the classical monitor
synchronization facility that can be very useful in real world
applications.
This is the provision of a 'browser' critical section. A browser is
basically
a reader that, after an initial scan to see if conditions are right, may
(or
may not) want to write to the protected data items. While a browser is
active,
any number of readers are allowed in their critical sections but no other
browsers or writers are allowed. If the browser thread determines that it
needs to write to the protected data, it requests a conversion of the
browse
lock to a write lock. The conversion process waits until all readers are
finished (not allowing any more readers or writers) and then allows the
browser to proceed as a writer with exclusive access to the protected data
items (whose state is guaranteed to be the same as it was when the browser
was
just looking at the protected data). The following is a very simple
example of
a browser critical section:

Working-Storage Section.
01 data-monitor usage monitor-pointer.
01 data-value pic x(4) comp-x value 0.

*> Initialization code, executed while in single threaded mode
open data-monitor

*> Add an item to table-1, this is a browser critical section
set data-monitor to browsing
if data-value < 20
set data-monitor to writing
converting from browsing
add 5 to data-value
set data-moniotr to not writing
else
set data-moniotr to not browsing
end-if

Note that we do NOT recommend that you use a browse lock for such a simple
check. Usually, a browser would be used only if a significant amount of
work
has to be done to determine if a write lock is actually required and you
want
to maximize multi-threading throughout your application. There are various
other monitor conversions available to help the program maximize the level
of
multi-threading in his application - please see the API reference in the
appendices to this document.

Using a Semaphore
-----------------

One of the meanings of semaphore in real life terms is a gate that lowers
to
prevent passage over a level-crossing and raises to allow passage through
that
same crossing. In software terms it is a synchronization object that, in
simple usages, acts as a gate which lowers to prevent passage through code
and
raises to allow passage (of another thread) through that code. This
semantic
is very much like a Mutex. In fact a Semaphore can be used instead of a
Mutex.

The following code illustrates this:

Working-Storage Section.
01 data-semaphore usage semaphore-pointer.
01 data-value pic x(4) comp-x value 0.

*> Initialization code executed while in single threaded mode
open data-semaphore
set data-semaphore up by 1 *> Initialize to raised

*> Add change data-value, this is a critical section
set data-semaphore down by 1
add 1 to data-value
set data-semaphore up by 1 *> Allow other thread to pass
semaphore

Note that just after the OPEN verb, the Semaphore is raised by 1. This
allows
the first acquire to succeed but any following acquires to be blocked
until
the Semaphore is released again.

So why provide both Semaphores and Mutexes? Well, while a Semaphore is
less
efficient than a Mutex, it is more flexible: one thread can simply release
(or
raise) it while any other thread can then acquire (or lower) it. Contrast
this
with a Mutex which must always be acquired before it can be released and
these
operations must happen within the same thread. So Semaphores provide a way
of
signaling from one thread to another, much like another kind of real world
Semaphore, the flag Semaphore.

The following code sample uses two different Semaphores to establish
handshaking between two threads. This handshaking allows one thread to
signal
the production of a new data value and the other thread to signal the
corresponding consumption of that data value:

Working-Storage Section.
01 produced-semaphore usage semaphore-pointer.
01 data-value pic x(4) comp-x value 0.
01 consumed-semaphore usage semaphore-pointer.

*> Initialization code executed while in single threaded mode
open produced-semaphore
open consumed-semaphore
set consumed-semaphore up by 1

*> This code is executed once to produce a data value
set consumed-semaphore down by 1
add 10 to data-value
set produced-semaphore up by 1 *> Signal data value
changed

*> Another thread, waiting for the data-value to change,
*> executes this code once as well.
Set produced-semaphore down by 1
display data-value
set consumed-semaphore up by 1 *> Signal data value used

The above code illustrates another common synchronization problem - the
Producer-Consumer Problem. The simplest Producer-Consumer Problem asks,
'Given
that there is one producer of data and one consumer of that data, how do
you
synchronize execution between the producer and consumer such that when the
consumer is awake he always has data to operate on?'. Semaphores, as used
above, solve this problem.

Further, these Semaphores also count the number of releases/raises that
are
outstanding on a Semaphore and will allow that many acquires to pass
unblocked. This kind of counting Semaphore allows the producer to produce
multiple data values (presumably in an array) before blocking to wait for
the
consumer to catch up.

The following code illustrates a simple example of a producer/consumer
pair
where the producer is allowed to create data values until the data table
cannot handle any more - at which point he blocks until some values are
consumed:

Working-Storage Section.
78 table-size value 20.
01 produced-semaphore usage semaphore-pointer.
01 filler.
05 table-data-value pic x(4) comp-x occurs table-size times
value 0.
01 consumed-semaphore usage semaphore-pointer.

Local-Storage Section.
01 table-i pic x(4) comp-x.

*> Initialization code executed while in single threaded mode
open produced-semaphore
open consumed-semaphore
set consumed-semaphore up by table-size *> start raised 20
times

*> Producer thread
move 1 to table-i
perform until 1 = 0
set consumed-semaphore down by 1
add table-i to table-data-value(table-i)
set produced-semaphore up by 1
add 1 to table-i
if table-i > table-size
move 1 to table-i
end-if
end-perform.

*> Consumer thread
move 1 to table-i
perform until 1 = 0
set produced-semaphore down by 1
display 'Current produced value is'
table-data-value(table-i)
set consumed-semaphore up by 1
add 1 to table-i
if table-i > table-size
move 1 to table-i
end-if
end-perform.

Using an Event
--------------

An event is similar to a Semaphore in that it allows one thread to signal
another that something has happened that requires attention. For several
reasons it is more flexible and slightly more complex. One reason is that
an
event, once posted, must be explicitly cleared.

The following is an example of the Producer-Consumer code using Event
synchronization instead of Semaphore synchronization:

Working-Storage Section.
01 produced-event usage event-pointer.
01 data-value pic x(4) comp-x value 0.
01 consumed-event usage event-pointer.

*> Initialization code executed while in single threaded mode
open produced-event
open consumed-event
set consumed-event to on *> Initialize as 'posted'

*> protocol for the producer side...
wait for consumed-event
set consumed-event to false *> Clear event
add 10 to data-value
*> Signal that data value has changed
set produced-event to true *> Post event

*> protocol for the consumer side, wait for the data-value to
*> change
wait for produced-event
set produced-event to false *> Clear event
display data-value
*> Signal other thread that he can proceed we got the
data-value
set consumed-event to true *> Post event

If there are only two threads (the producer and consumer threads)
executing
the above code, then everything will work fine. If another thread comes in
and
starts executing as a producer or consumer, interesting (i.e. unwanted)
things
will happen. This is because an event once posted will wake up all threads
waiting on that event, unlike a Semaphore which allows only one thread to
pass
after a release/raise. After the event has been posted and the waiting
threads
woken, it is necessary for each of the woken threads to determine if it
should
take some form of action on the event (including clearing the event).

The last point can make events difficult to work with when there are
multiple
waiters but it also allows the building of custom synchronization objects
for
special needs.

Chapter 3

The Micro Focus COBOL RTS
=========================

The previous chapters have attempted to illustrate the difficulties and
costs
(on a per-thread basis) of writing a multi-threaded application. Most of
this
difficulty and cost comes from the need to synchronize the various
threads'
execution and resolve data contention issues. While, the flexibility and
power
of a multi-threaded application far outweighs these costs, if you do not
need
to create a multi-threaded application, you can program traditionally and
should not incur any extra run-time costs in your application. There are
some
specific steps necessary to achieve this end though, as detailed below.

The multi-threaded Run-Time System which supports Micro Focus COBOL
applications must be able to handle both multi-threaded and
single-threaded
applications. As a result, the multi-threaded RTS incurs many of the
synchronization costs that a standard multi-threaded COBOL application
would.
Further, most of these costs are incurred regardless of the kind of
application being run, either multi-threaded or single-threaded. Because
of
this, and in order to allow single-threaded applications to run at maximum
speed, we have provided two separate run-time systems, a multi-threaded
RTS,
which supports all applications, and a single-threaded RTS which supports
only
single-threaded applications.

You may chose between these run-time systems at application run time by
the
trigger name or at CBLLINK time by the use of an option, as detailed in
the
following tables.

The following table details the run trigger names and the various COBOL
RTS
DLLs that will be used when the application is run. Note that the listing
of

Chapter 4

Writing a Multi-threaded Application
====================================

Reentrant COBOL Programs
------------------------

The first thing to consider when writing a multi-threaded application is
how
each module should operate. Should the module be compiled as a standard
COBOL
program, should it be compiled as a SERIAL program or should it be
compiled as
a REENTRANT module? In the section Use of Program Attributes on page 1 we
gave
an overview of the usages of standard COBOL compilation, SERIAL
compilation
and REENTRANT compilation. Out of the three, reentrant programs need more
attention (in addition to the cautions of protecting user-defined static
data
such as WORKING-STORAGE).

First, not every program can be compiled as reentrant. Use of some COBOL
features will preclude reentrant compilation, but, thankfully, most of
these
COBOL features are obsolete or archaic in ANSI Standard COBOL and their
use
should be discontinued. Programs that use the following features cannot be
compiled with the REENTRANT directive:

o the ALTER statement
o the ON statement
o the COBOL DEBUG facility
o PROCEDURE DIVISION overlays - note this can be overcome by using
either
the NOSEG or NOOVL directives
o the IS INITIAL attribute in the PROGRAM-ID
o the STICKY-LINKAGE compiler directive
o the STICKY-PERFORM compiler directive

The following restrictions and notes on the use of reentrant programs and
multi-threading within those programs also apply:

o Most file handling verbs will need explicit user protection with
one of
the execution synchronization primitives discussed earlier.
Internally,
the file handler is thread-safe, however within the user program
file
records, buffers and status fields are set without the benefit of
locking - this can lead to contention and resulting corruption of
these
fields if no user measures are taken.
o Execution of any file operation is serialized within the
file-handler
with a global file-handling Mutex. This means that only one thread
may
access any file at a given time. This could lead to problems if a
read
from a file is delayed for some reason (e.g. because of a record
lock),
in this case no other thread will be able to do any file handling
operations until the delayed read is finished.
o Execution of any sort operations are serialized within the sort
handler
with the global file-handling Mutex. While a sort is on-going no
other
thread will be able to perform other sort or file handling
operations.
Execution of the INPUT and OUTPUT procedures occur in the thread
that
started the sort operation. File handling within these procedures
will
execute normally.
o Execution of the DISPLAY verb will require user protection with one
of
the synchronization primitives discussed earlier. Internally, the
console display handler is thread-safe, however each complex
display is
broken down into multiple calls to the display handler - this can
lead
to interleaved displays of data items. In general, it is best to
assign
only one thread to console I/O within a multi-threaded application.
o Nested programs may not be the starting point of a newly created
thread. Only ENTRY points, outermost programs in a compilation unit
and
other-language external entry points may be the starting point for
thread creation.
o Execution of the ACCEPT verb is serialized within the accept
handler.
This means that all other threads are prevented from doing any
ACCEPT
or DISPLAY operations while the original ACCEPT is outstanding.

The following performance and resource penalties are incurred in reentrant
COBOL programs within a multi-threaded application:

o Use of the THREAD-LOCAL-STORAGE section, on some platforms, can
incur
significant program entry costs. If possible, use LOCAL-STORAGE
instead
for thread-local work variables.
o A reentrant program will consume more stack space than either a
standard
or a serialized COBOL program. This is because system work-areas
must
be allocated on the stack, instead of statically.

Initialization of applications
------------------------------

The initialization of multi-threaded applications needs a bit of care.
First,
as good programming practice, the application should determine if it is
running with an RTS that supports multi-threading. This is easily done
with
the following code:

Working-Storage Section.
01 thread-id usage pointer.

*> Initialization code executed while in single threaded mode to
check
*> if the RTS supports multi-threading or the CBL_THREAD_ API.
call 'CBL_THREAD_SELF' using thread-id
on exception
*> No cbl_thread api support
end-call
if return-code == 1008
*> Running in a single-threaded rts
end-if

Note that this last code sample makes use of a CBL_ API call for it's
function. While many of the facilities provided by the RTS for
multi-threaded
programming are accessable through Micro Focus COBOL syntax, all can be
coded
directly through the use of general API calls. Further, to make use of
some of
the more advanced multi-threaded programming functionality, API calls must
be
used directly.

After it is determined that the RTS supports multi-threading, all
synchronization objects should be be initialized using the appropriate
OPEN
syntax. It is easiest to do this as part of the main program, before any
other
threads are created within the application. However, if by application
design
or because of modularity or mixed-language considerations, this is not
possible, we have provided a single pre-initialized Mutex per program.
This
Mutex is accessed using CBL_THREAD_PROG_LOCK and CBL_THREAD_PROG_UNLOCK
API
calls. Using these calls, it is possible to ensure that any handles for
program-local synchronization primitives are initialized once (and only
once)
during the execution of the application. The following code illustrates a
sample use of these API calls for program initialization while running in
multi-threaded mode:

Working-Storage Section.
01 first-flag pic x comp-x value 1.
88 first-time value 1.
88 not-first-time value 0.

*> Initialization code executed while in multi-threaded mode
*> Ensures that program local data is initialized properly.

if first-time then
call 'CBL_THREAD_PROG_LOCK'
if first-time then
*> Initialize program local data and synchronization
objects
...
set not-first-time to true
end-if
call 'CBL_THREAD_PROG_UNLOCK'
end-if

Note the double checking of the first-time level 88. This is a good
optimization technique in multi-threaded applications. The idea of the
optimization is to avoid the overhead of locking a Mutex if we know that
our
intended action has already been performed. In this case, if multiple
threads
enter the program before it is properly initialized, they will all find
that
first-time is true and so issue a call to CBL_THREAD_PROG_LOCK, but only
one
will acquire the lock while the first-time is still true. This is the
thread
that will do the initialization.

After the initialization is completed, and while the initializing thread
still
owns the lock, the first-time flag is set to false. All successive threads
acquiring the lock will see that the initialization has already been done,
as
first-time is now false and unlock the program immediately. Any following
threads that enter after initialization will also see the first-time flag
is
false and will not attempt to acquire the program lock (and so reduce
program
entry overhead).

Note that in order for this optimization to work properly on all
platforms,
the first time in flag should as simple as possible (i.e. a single byte
data
item).

Manipulating Threads
--------------------

The multi-threaded RTS supports threads created by the RTS itself, via the
START verb or the CBL_THREAD_ API, and also other-language threads created
directly by the operating system. Since COBOL syntax and the CBL_THREAD_
API
are portable, these are the preferred methods of thread manipulation.
Further,
the CBL_THREAD_ API can also be called from other language modules,
allowing
any multi-threaded application to take full advantage of the advanced
facilities provided by the RTS.

Thread handles
--------------

In general, threads are identified to the RTS with thread handles. Thread
handles are provided to creating threads as return values from the START
verb
or the CBL_THREAD_CREATE API call or to an individual thread by the return
value from CBL_THREAD_SELF API call. This handle uniquely identifies the
thread to the various multi-threading COBOL verbs and the CBL_THREAD_ API
calls and, with reasonable care, can be used by any of the active threads
in
the run-unit not just the creator of a thread. It is important to remember
that the lifetime of a thread and that of its thread handle might not be
the
same. Depending on how a thread was created and what operations have been
done
on it, when a thread terminates, its handle could still be valid. In order
to
allow the system to recover the resources associated with the handle, it
is up
to the user to explicitly detach the handle for a thread.

A handle can become detached in one of several ways. When a thread is
created
with the START verb or the CBL_THREAD_CREATE API call, its handle is, by
default, created as detached. When an other-language thread becomes known
to
the RTS, its handle is automatically and always created as detached. A
non-detached handle can be detached with a call to CBL_THREAD_DETACH. In
any
case, a detached handle must always be used with care, as when the thread
terminates the handle immediately becomes invalid. If a thread with a
non-detached handle has already terminated, then a call to
CBL_THREAD_DETACH
with that thread handle will immediately cause the handle to become
invalid.

A non-detached handle is useful for retrieval of return values (via the
WAIT
verb or the CBL_THREAD_WAIT) API call and for inspection of thread
identification data (via the CBL_THREAD_IDDATA_GET API call) after a
thread
has potentially terminated. The START verb provides a way to identify a
newly
created thread by returning a non-detached thread handle and the
CBL_THREAD_CREATE API call provides a flag to indicate that the thread
should
be created and a non-detached handle should be returned.

Creating and terminating a thread
---------------------------------

Threads are created by the START verb, the CBL_THREAD_CREATE API call or
with
the specific operating system call for thread creation. Threads created by
the
first two of these methods are called COBOL Threads, threads created
directly
by the specific operating system facility are called Other Language
Threads.
Again, as the creation and manipulation of COBOL Threads is portable and
preferred, we will only deal with them in the following examples.

The following complete application illustrates simple thread creation:

$set reentrant
Data Division.
Working-Storage Section.
01 thread-handle usage thread-pointer.
01 thread-return usage pointer.

Linkage Section.
01 thread-parm picture x(32).
01 thread-return-record picture x(32).

Procedure Division.
Start 'CREATED' using 'This is a 32 character parameter'
identified by thread-handle
on exception
stop run
end-start
wait for thread-handle returning into thread-return
set address of thread-return-record to thread-return
display thread-return-record

stop run.

Entry "CREATED" using thread-parm.
display thread-parm
stop run returning address of thread-parm.

This application simply creates a thread with the starting point
"CREATED".
The "CREATED" thread displays its parameter and returns the address of
that
parameter for use in the parent thread. Note that:

o The starting point for a thread must be either a non-nested
PROGRAM-ID,
a COBOL ENTRY point or an other-language (e.g. C) externally
visible
routine. It must not be a nested program, a section or paragraph
name.
o The starting point name is specified as a text string, the overhead
in
using this text string to find the entry point is equivalent to
resolving the verb CALL identifier. It is possible to avoid this
overhead by using a procedure pointer as the object of the START
verb.
o The starting point for every created thread is provided with only
one
parameter. If the BY CONTENT phrase is used in passing this
parameter a
copy of it is made by the system before the thread is created,
leaving
the calling thread free to modify the original parameter upon
return
from the START verb.
o Creation of a thread with a non-detached handle (via the IDENTIFIED
BY
clause) allows the return value to be obtained later by the WAIT
verb.
Upon completion of the WAIT verb the specified thread handle
becomes
invalid and all resources associated with the subject thread are
released.
o The STOP RUN RETURNING in the "CREATED" thread does not end the
run-unit, instead it simply provides a return value and terminates
the
thread. It is equivalent to:

call 'CBL_THREAD_EXIT' using by value address of thread-parm.

o STOP RUN in an Other Language Thread threads or the main thread of
a
Run-unit will wait for all active COBOL Threads to finish and then
terminate the run-unit.
o Return values from threads are always pointers. This allows you to
return both simple and complex data structures.

As was mentioned above, a thread can terminate itself with STOP RUN or a
call
to CBL_THREAD_EXIT. A thread also terminates normally when the starting
point
program exits via EXIT PROGRAM or GOBACK. It is also sometimes useful for
another thread to cause a COBOL Thread to be terminated. The
CBL_THREAD_KILL
API call achieves this.

Canceling a thread
------------------

Threads are canceled with the CBL_THREAD_KILL API call. This call causes
immediate, abnormal termination of a COBOL Thread thread but, in general,
it
should not be used as part of general application thread control. The main
reason for this is that some user and system synchronization resources
will
not be properly unlocked and/or freed upon thread termination. This can
affect
the synchronization within the user application and the RTS, very likely
leading to serious problems with the application run.

CBL_THREAD_KILL can reasonably be used within a critical error handler in
the
main thread. In this error handler, thread handles can be acquired by way
of
the CBL_THREAD_LIST_ API, all threads can be canceled and the application
can
then exit via STOP RUN. This is less dangerous than the random usage of
CBL_THREAD_KILL as it minimizes the chance of needing a locked
synchronization
resource, but there is still a possibility of file corruption or deadlock
during run-unit termination.

In any case, most applications should avoid using CBL_THREAD_KILL. This
can be
achieved by creating thread identification data (through
CBL_THREAD_IDDATA_ALLOC) that has a terminate flag in it. That data can
then
be polled in each thread at a level where no locks are held. If the
terminate
flag is set, then the polling thread can exit gracefully. See the section
Identifying a thread on page 1 for an example of using the
CBL_THREAD_IDDATA_
API.

Suspending a thread
-------------------

In many multi-threaded applications it is not uncommon for a thread to
'run
out of work' but not want to terminate just yet. In a client-server
architecture, for example, the main service thread could be polling an
input
queue for requests and if it finds none then it makes sense to allow
another
process or thread to have the CPU before polling the input queue again.
This
is simply done by calling CBL_THREAD_YIELD which yields the processor to
some
other thread in the application (or some other thread in another process,
depending on the operating system).

Another possibility is that a thread needs to yield the CPU indefinitely
and
obtain it only when some sort of event has definitely happened.
CBL_THREAD_SUSPEND allows suspension of the calling thread until some
other
thread does a CBL_THREAD_RESUME using the thread handle for the suspended
thread. A thread can call CBL_THREAD_RESUME one or more times before the
targeted thread CBL_THREAD_SUSPENDs itself and, in this case, the call to
CBL_THREAD_SUSPEND will return to the calling thread immediately and not
give
up ownership of the CPU. From the discussions on synchronization objects,
you
will see that this semantic is very similar to that of a counting
Semaphore,
in fact the Producer-Consumer problem can be solved using only the
CBL_THREAD_SUSPEND and CBL_THREAD_RESUME API calls.

Identifying a thread
--------------------

It is sometimes useful for a thread to distinguish itself from any other
created threads in an application. For example, if two threads in a four
thread application establish a producer-consumer relationship, it can be
useful for the two cooperating threads to find out what each others thread
handles are. Once these handles are obtained, all synchronization can be
done
using only the CBL_THREAD_SUSPEND and CBL_THREAD_RESUME calls. If each
thread
in the application creates a name for itself (and the name relates its
functionality) and associates the name with its thread handle, then the
cooperating threads can scan the thread name list and, so, find each
other's
handle.

Another possible use for associating globally accessible data to each
thread
and its handle is to hold a termination flag, getting around the possible
need
to use CBL_THREAD_KILL. Each thread in the application can poll its
termination flag to check for a termination request. The terminating
thread
can then make sure no locks are held and no synchronization actions are
required before terminating normally.

Globally accessible data for a thread is associated with the thread handle
by
the CBL_THREAD_IDDATA_ALLOC call executed within the thread. This data is
retrieved by the CBL_THREAD_IDDATA_GET call, if the thread handle is
already
known, or by the CBL_THREAD_LIST_ API calls if the thread handle is not
yet
known. These mechanisms are illustrated in the following sample
application:

**************************** MAINPROG.CBL
****************************

This rather lengthy example doesn't actually do anything but establish
handshaking for thread and application termination. Before going into any
discussion on it, it is worthwhile to note that this kind of handshaking
could
have been accomplished more easily by passing the main thread's handle
into
the child as a parameter. This would have avoided the need to rely on
identification data or to step through the thread list.

First notice the two different ways of creating thread identification
data.
The first way, in the parent thread, creates the data uninitialized,
retrieves
the iddata pointer and then initializes the area of memory that was
allocated.
The second, in the child, provides initialization data to eliminate any
possible execution window between iddata allocation and iddata
initialization
by the application. The method used in your application will depend on
what
level of contention you expect on identification data.

Also, notice the loop in the parent that waits for the child to create its
thread identification data and the loop in the child that waits for the
parent
to flag termination. The call to CBL_THREAD_YIELD prevents these loops
from
being hard busy waits, but would have been better coded using an event or
condition synchronization object or using CBL_THREAD_SUSPEND and
CBL_THREAD_RESUME. See the section Common Programming Pitfalls for more
discussion on busy waiting.

Finally, notice the use of the CBL_THREAD_LIST_ API. This API allows a
thread
to step through all threads known to the RTS, obtaining the thread handle,
thread state and identification data pointer. In this example we relied
only
on the handle and identifiction data pointer but the state information can
also be useful, as it lets the calling thread know if the subject thread
is
detached, suspended or an other-language thread.

It is important to realize that whenever the CBL_THREAD_LIST API is used,
the
using thread has limitations on further CBL_THREAD_ calls and all other
threads are completely locked out from using any of these calls as well as
quite a few of the other RTS calls. For this reason, it makes sense to
keep
the amount of code executed while stepping the list to a minimum and
certainly
avoiding file or user I/O while the list is locked. These restrictions are
removed only after the stepping is terminated by a call to
CBL_THREAD_LIST_END.

Other Language Threads
----------------------

In a mixed language environment, threads created directly by operating
system
facilities and not by the START verb or the CBL_THREAD_CREATE API call
have
several restrictions and requirements on them:

o They must be terminated by the approved operating system method of
thread termination and not by any function in the CBL_THREAD_ API.
o Their handles are always created detached. This means that they
cannot
be waited on by the WAIT verb or the CBL_THREAD_WAIT API call and
no
return value is available from them.
o They must issue the call cobthreadtidy() or cobtidy() when the
thread
or the application, respectively, no longer needs COBOL RTS
services.
This allows the RTS to free memory and clean up thread state
information.
o cobtidy(), cobexit() or the COBOL STOP RUN statement will cause the
calling thread to be treated as the 'main' thread. As a result, the
RTS
will wait for all COBOL threads to terminate before any RTS cleanup
processing is completed.

Note that all multi-threading functions supported by the RTS will detect
Other
Language Thread handles and report invalid usages of them as. It is,
however,
entirely up to the application to ensure that RTS resources are freed for
any
thread that has executed any COBOL or RTS code in the course of it's
existence
(as detailed above).

Creating thread-local data
--------------------------

We have already discussed some of the ways of creating and using data in
multi-threaded applications in the section Use of Data Attributes on page
1.
These methods all created static data for direct reference by the using
program. It is however entirely reasonable for the programmer to require
dynamic data whose lifetime is that of the creating thread or selective or
external thread local data. These facilities are provided by the
CBL_ALLOC_THREAD_MEM and the CBL_TSTORE_ APIs, respectively.

First, CBL_ALLOC_THREAD_MEM will allocate dynamic memory that is
automatically
freed when the creating thread is canceled. This call is ideal for
creating
thread local linked lists or other thread specific operational structures.
By
default the memory allocated is also attached to the allocating program,
so if
the program is canceled then the memory is automatically freed (this
default
semantic can feature can be disabled). Note that the dynamic memory
pointer
returned by CBL_ALLOC_THREAD_MEM should probably be stored in a
THREAD-LOCAL-STORAGE section data item to protect its modification by
other
threads.

Second, the CBL_TSTORE_ API allows the programmer to create a handle for
thread local storage and in any thread get a pointer to the dynamic thread
local data through this handle. When the handle is created by
CBL_TSTORE_CREATE, you provide the size of the thread local data area that
will be eventually created for each thread. The data area is actually
allocated only in threads that call CBL_TSTORE_GET using the returned
handle.
All data allocated to a thread is freed when the thread ends or,
optionally,
when the handle creating program is canceled. It is possible to free all
data
associated with a handle and the handle itself by using the
CBL_TSTORE_CLOSE
call.

A CBL_TSTORE_ handle is usually stored in a WORKING-STORAGE data item so
that
it is accessible to all threads requiring it. This data item can also be
EXTERNAL, so multiple programs can access the handle and, within the same
thread, have access to the same thread local data area. The pointer to the
thread local data returned by CBL_TSTORE_GET should be stored in a
thread-safe
data item, i.e. one defined either in THREAD-LOCAL-STORAGE or
LOCAL-STORAGE.

The following code fragment shows the use of the CBL_TSTORE_ API calls:

Working-Storage Section.

zero *> Associated with creating

*> If we need it then get the thread local storage area -

Common Programming Pitfalls
---------------------------

It is worthwhile to discuss a few of the common problems a multi-threaded
application can encounter while trying to syncronize thread execution.
These
are outlined in the following sections.

Busy Wait
---------

The first of these is the use of a busy wait for synchronization, as
illustrated in the following code fragment:

*> Code executed by one thread
perform until done = 1
*> Do nothing - we are waiting for the following thread to
finish
end-perform
*> Code executed by the working thread
.... a lot of time-consuming work is done here ...
move 1 to done

While this might seem reasonable at first glance, the waiting loop is
actually
consuming CPU resource without doing any meaningful work (ie. the CPU is
busy
while just waiting). It would be much better if a CBL_THREAD_YIELD was
coded
in the loop to yield the CPU to the working thread so it will finish
faster.
It would be even better to use one of the synchronization objects, such as
an
event or a condition, to signal completion of the work to the waiting
thread.
Since, on some operating systems, these objects can be implemented so that
the
waiting thread is never awake until the moment when the work is finished
and
the event/condition is signaled, this would maximize CPU utilization.

Deadlock
--------

A deadlock condition is one in which two (or more) threads are competing
for
resources that the other has locked. Avoiding or correcting this situation
is
not quite as straight forward as dealing with busy waiting. The following
code
fragments, executed in two separate threads, illustrate a potential
deadlock
situation:

*> Thread 1 execution *> Thread 2 execution
set mutex-1 to on
set mutex-2 to on
set mutex-1 to on
set mutex-2 to on

Consider the case where thread 1 acquires the lock on mutex-1 and then
gets
suspended, at which point thread 2 acquires a lock on mutex-2. It now does
not
matter who will execute next, as neither thread will be able to get their
required lock (i.e. thread 1 can't get a lock on mutex-2, as thread-2
already
owns it and similarly for thread 2 and mutex-1). The application has not
provided a way to release the sequence of locks, so these threads will
wait
indefinitely on a condition that will never happen. There are two methods
an
application can use to deal with this situation.

To avoid deadlock in the first place, the application can impose on itself
a
lock hierarchy. In other words, if the code needs more than one lock
imposed,
then the order that those locks are obtained is always the same. In this
case,
we could solve the deadlock potential by making sure that if thread 2
needed
locks on mutex-1 and mutex-2 at the same time, then it always acquired
mutex-1
before attempting to acquire mutex-2.

To correct a potential deadlock situation, each thread must be aware of
all
locks outstanding at the point another lock is required. When the next in
a
sequence of locks is attempted, then the CBL_MUTEX_ACQUIRE API call must
be
used and it's flag parameter should be set to h'01'. This will allow the
call
to return immediately with a non-zero return code if the lock cannot be
established. Once the lock attempt has failed, it is up to the requesting
thread to unlock all other locks held to that point and yield, allowing
any
other thread to complete its transaction. After the yielding, the original
thread can attempt to reestablish all locks that were released and attempt
to
establish the lock that originally failed.

While the deadlock correction technique might (or might not!) sound
simple, it
can be complex to implement properly, as it usually involves partial
'transaction' roll-backs. Also, in the worst case, it can lead to two or
more
threads repeatedly attempting to establish a lock hierarchy but failing
and
having to bail out each time. This can be almost as bad as a deadlock
situation (but it is extremely unlikely that this would happen
indefinitely).
In general, we would suggest that you try your best to use avoidance,
rather
than attempt to implement correction properly.

Single Thread Deadlock and Writer Starvation
--------------------------------------------

When using monitors to syncronize data access it is possible, via the
CBL_MONITOR_OPEN API call, to specify what method is used to schedule the
granting of read and write access of the monitor. By default, readers and
writers will have interleaved priority. This means that when a write
request
is submitted, all following read requests will block until that write
request
is granted and subsequently released. If a further write or browse request
is
submitted while those following readers are waiting, then the waiting
readers
are allowed access first before that writer is granted access. This is a
very
fair scheduling for readers, writers and browsers but, when monitor read
locks
are nested in a single thread, there is a possibility of a single thread
deadlock condition arising.

Consider the following code:

*> Thread 1 execution *> Thread 2 execution
set monitor-1 to reading
set monitor-1 to writing
... other code + calls ...
set monitor-1 to reading

When thread 1 acquires a read lock on monitor-1, all further writers will
be
blocked until thread 1 releases its lock. Specifically, thread 2 will be
unable to acquire his write lock and, from our scheduling of readers and
writers, thread 2's request will stop any further read locks from being
granted. Now, if thread 1 requests another (nested) read lock then it will
block until thread 2's write request is granted but, of course, the write
request cannot be granted because thread 1 still has a read lock
outstanding.
This is single thread deadlock.

There are a few ways of avoiding this condition. First, if possible, the
programmer can avoid nesting of monitor read locks, but this is sometimes
difficult to arrange in some modular applications. Alternatively, if a
nested
lock request is expected, a write or browse lock can be obtained at the
highest level. This approach will tend to reduce the level of
multi-threading
through the application, but it is definitely preferable to a deadlock

condition being encountered. As final option, the programmer can change
the
scheduling priority of the monitor in question to readers priority by
specifying the appropriate flag on the CBL_MONITOR_OPEN call.

A readers priority monitor gives absolute priority to read requests on a
monitor. All write requests must wait until no readers are active on the
monitor before being granted access. New readers are not blocked from
access
and will continue to prevent outstanding write requests from being
fulfilled.
If new read requests are continually being received on a monitor while at
least one has already been granted, then the writer will never get access
to
the resource being requested. This condition is called writer starvation
and
can reasonably occur in high read traffic multi-threaded modules. The only
way
to avoid writer starvation is to use the interleaved scheduling priority
for
monitors and make sure you avoid single thread deadlock.

Chapter 5

Optimizations & Programming Tips
================================

Throughout this document there are various hints on general coding and
optimizations that allow a multi-threaded application to run as fast and
as
flexibly as possible. It is worthwhile recapping them in point form here:

o Always code a check to see the kind of RTS the application is
running
under (i.e. single-threaded or multi-threaded).
o Use the NOREENTRANT and NOSERIAL program attributes (the default)
wherever possible. This minimizes stack space and call entry
overhead
but it is up to the caller to ensure this program is only active in
one
thread at a time.
o The use of the REENTRANT program attribute will increase stack
space
usage but will not significantly affect the call speed of the
program.
o The use of the SERIAL program attribute will significantly affect
call
speed but will not affect stack space requirements of the program.
o Use the LOCAL-STORAGE section to hold thread work variables. While
this
will increase stack space requirements, the allocation of data is
very
fast.
o Use of the THREAD-LOCAL-STORAGE section will significantly affect
call
speed.
o Replacing the WORKING-STORAGE SECTION with the THREAD-LOCAL-STORAGE
SECTION will easily allow some applications to be made entirely
thread-safe.
o Use Mutexes for simple locking. These are the fastest of the
synchronization structures.
o Use Monitors to provide maximum multi-threading in Readers/Writers
based problems.
o If possible, initialize all synchronization data items while in
single
threaded execution. If this is not possible, then use a first time
flag
which is checked once before a PROG_LOCK and checked again after
doing
obtaining the lock.
o Remember the difference between a thread and a thread handle.
Create a
non-detached thread only if necessary.
o Use CBL_THREAD_KILL only if absolutely necessary and if you do use
it,
terminate the run-unit as soon as possible afterwards. Use
termination
polling instead.
o Use CBL_THREAD_SUSPEND in preference to CBL_THREAD_YIELD. Avoid
busy
waits.
o Keep the amount of work done between CBL_THREAD_LIST_START and
CBL_THREAD_LIST_END calls to a minimum.
o In certain circumstances, the application can conserve per thread
memory by using the CBL_TSTORE_ API in preference to the
THREAD-LOCAL-STORAGE section.
o Try to avoid deadlock rather than attempting correction once a
situation has been found.
o Avoid nesting read locks on monitors and, if you must, then be
aware of
the possibility of single thread deadlock.

Chapter 6

Compiler Directives
===================

There are new compiler directives that can affect the performance of your
multi-threaded and single threaded application.

[no]paramcountcheck controls checking of number of entry point
parameters.

If paramcountcheck is specified, generated code will only take the number
of
parameters passed by the caller. The address of all other linkage section
items will be set to NULL. paramcountcheck should should be used if a
program
could be called with a variable number of parameters.

If noparamcountcheck is specified, then generated code takes the number of
parameters expected by the entry point, irrespective of how many are
actually
passed. In this case the syntax IF ADDRESS OF = NULL will
not
work.

[no]fastcalls controls the behaviour of a called program.

If fastcalls is specified, the program does not set up information to
allow it
to determine whether it is a main program, the RTS will always assume it
is
not and EXIT PROGRAM will always exit to the caller Also, programs do not
set
up information to allow further called programs to determine whether they
are
main programs. Thus, any program called by a program compiled with
fastcalls
should itself be compiled with fastcalls.

[no]reentrant controls reentrancy of a program

If reentrant is specified, many program areas will be allocated
dynamically,
so that it is safe to have multiple copies of the program running.

[no]serial controls serialization of a program.

If serial is specified, the Run Time System will determine whether the
program
is active in a thread and prevent any other thread entering the active
program. Note that the code to determine this takes time and is called on
every program entry. Programs should only be compiled with serial if they
are
going to run in a multi-threaded environment.

[no]fixopt controls placing of certain control areas.

If fixopt is specified, the frame pointer will point to the static data
area,
allowing data addressing without fixups. This directive may be necessary
on
large GNT or COFF programs which exceed the COFF limit of 65536
relocations.
If set, fastlink behaviour is not available and stack tracing under an
object
code debugger will also not work.

[no]fastlink controls handling of parameters.

If fastlink is specified, and the parameters conform to the specifications
given below, parameters are not unloaded from the stack, but manipulated
in
place. The requirements are:

1. The parameters must be in the same order on every entry point
2. This order does not have to be the same order as the order in which
they
appear as 01 items in the linkage section
3. The program must be compiled with nofixopt and notypecheck

The following directive settings are made by default:

PARAMCOUNTCHECK NOFASTCALLS NOREENTRANT NOSERIAL FIXOPT (FASTLINK)

Note that the fastlink will not work, because typecheck and fixopt are
set.

For maximum speed, but with potential semantic changes:

NOTYPECHECK FASTCALLS NOREENTRANT NOSERIAL NOFIXOPT FASTLINK
```
