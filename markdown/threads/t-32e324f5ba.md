```
Hi all,

Has anyone come accross the following situation?

We have ported an old character application written in Microsoft Cobol 2 to
Net Express 3.1 (same character interface) and installed it on a Windows NT
4 (sp5) server with Windows 98 SE clients.

Every data file was regenerated as MF"4" type with Net Express.

The problem we face is a quite strange one, as sometimes some of the records
that are being written on a file, are not finally stored to it! The flow of
the suspect part of the application is :

write a record to a file
print an invoice according to this record

Then in the end of the day we are trying to print the total invoices of the
day from the same file and some of them do not appear in the final list,
although they were printed in the first place ! This does not happen all the
time of course, but once or twice a week (which are enough to loose a lot of
money).

It seems that the server caches the new records before writing them to the
disk, and sometimes this write never happens. I have tried the Writethrough
complier directive but we did not see much improvement. Will Fileshare make
a difference or do you have anything else in mind?

Thank in advance for your help.

Panos Zotos
Athens, Greece
```

```
This might be one explanation:

Be sure that you close your files.  Part of the "close" operation is to
finish copying any data in the buffer to the file.  

The size of the buffer is equal to the "allocation unit" reported by the
"chkdsk" command.  Each write operation actually writes to the
buffer.    When the buffer is full, data is copied to the disk.

By shutting down a program - without executing a "close" - you will not
have copied the last set of records to the disk.

Floyd Johnson
```

```
If the application terminates through a STOP RUN,
all open files are closed, implicitly. Open files in
a subprogram are closed, implicity, by CANCEL,
as well. See STOP RUN and CANCEL in your LRM.

While it is a good idea to close files, it is not required
by the standard.
```

```
Are you using Fileshare already?  If so, try stopping and
restarting Fileshare.  The records might not really be lost.

We had a similar problem, but we noticed that those
missing records could be accessed by a seldom-used
alternate key.  Fileshare somehow had marked the primary
key on those records so they would not be found, even though
we never read them with lock.  Restarting the Fileshare
process released the records.
```
