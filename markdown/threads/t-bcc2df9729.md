Ahhhhhhh, Mr Klein... with all due respect I would say that this has been
*most* relevant as an example of thorough pre-design system analysis.

Every system, of course, has to be designed in order to take account of
various conditions... 'Hey, if it is Tuesday and raining in Bangalore then
the tax-rate tables change; don't forget to code for that!'... and,
likewise, there are things that one does *not* need to take account of...
'What do you mean, 'commission rates change for the Small Animals' Meat
Festival'?  That practise was stopped in my grandfather's time!'

*Any* code involves overhead, from the system and the programmers... so
one should know one's conditions before writing it.  It was here suggested
that the condition of marital status be expanded to allow for C(ommon
Law)... on the surface a Good Idea ('Nice thinking, Catherwood... put
yourself down for a bonus!') but upon further examination it is shown to
be not only of no positive value (there have been shown no benefits
accrued or denied common-law status present in contractual married status)
but in that the condition requires overhead - both programmer and system -
there is, one might readily conclude, *negative* value, or 'detriment'.

I would say that demonstrating this is *very* valuable, Mr Klein... of
course, until very recently I, myself, was still coding exception-routines
for the Small Animals' Meat Festival.
