Whether this is practical or benficial depends a lot on your
environment.
We are large CICS & batch, COBOL II & later, some Assembler, and large
DB2.
Several years ago we encouraged edit,
compile, and testing on workstations to reduce
mainframe load during a period when a mainframe upgrade was not
feasible.
While we achieved our short-term goal, when mainframe resources became
less contrained, things started migrating back to the mainframe to where
I would now estimate over 80% of this activity is now on the mainframe.
In our present environment, programmers seem to feel they are more
productive
when they don't have to spend additional effort to recreate a complex
testing environment on the workstation when such an environment already
exists on the mainframe. Contributing factors in our environment:
(1) with rare exceptions, applications don't tend to be self-contained.
Interactions with other applications and support routines from other
development areas make maintaining a work station environment that is
completely in sync with current production environment a challenge.
(2) test data typically has to be derived from large databases and
files on the mainframe, tape files in some cases. The process of data
subsetting must be done on the mainframe. For non-DB2 data, getting
this
test data accessible to a workstation envolves extra work and person
time;
(3)Despite tools that are close in functionality (SPF, MF COBOL, XDB and
later DB2/2) there are always differences in behavior between mainframe
and WS environments that are sometimes significant. You will find
subtle
differences between COBOL II and MF COBOL where the COBOL language is
implementation dependent, and code is sometimes either intentionally or
accidentally dependent on these differences. When we were using XDB,
there was never a week without the XDB server crashing several times.
There were even subtle differences in DB2 behavior that programmers had
to
work around. We always had a list of outstanding MF COBOL or XDB bugs
outstanding, and unlike the mainframe, you don't get PTF's--the fix is
in a later release at least 6 months down the road.

In our environment, many application development areas simply found
that for workstation development of mainframe applications,
the benefits of a somewhat nicer user interface for testing simply
wasn't
worth the extra hassle of resolving problems. in two environments.
Your milage may vary, especially depending on what kind of debugging
facilites are available on the mainframe side.

Our organization is not allowed to endorse products, so this is not such
a response. This is just an informational answer on some things that I
have seen, here and other places.

There's a suite of products called COMPOSER, ARRANGER, et al., formerly
known as IEF. This was developed by Texas Instruments, but now is owned
by Sterling Technologies. It features code development on PCs, with
options for testing completely on the PC or only on the mainframe or
both. It has its own code syntax and there is a learning curve. However,
most lines of IEF code generate a number of COBOL (or C++) statements,
so there is a productivity gain once you get the hang of it.
It can do batch or online, any famous relational Database (not sure
about heirarchical), and IMS DC and CICS (probably more). It also has
web development components for the GUI interface (C++, not yet Java, I
think).

About 6 years ago, I also used a similar product provided by Oracle. It
was called ORACLE CASE then, but I would imagine it has a new name by
now.

Both of these products utilize a CASE design paradigm to some degree of
rigor. However, that doesn't get in the way too much. I.e., when specs
change, you can modify program code without going back to the
analysis/design portions of the product and losing a bunch of your
previously-written program code. I think there's some retro-design
facility too, where your code can be shoved backwards to alter the
design, but I've never used that.
Both offer RAD development and prototyping, and mouse-driven screen
painters.

I don't imagine the cost of either is pocket change, but the
productivity gain can be significant. One nice thing is that the code
almost always compiles the first time. The only common exception is when
the database on the mainframe has not been accurately portrayed to the
PC code generator. This doesn't mean you can't code a mistake in how you
process the data, but you have a real hard time getting a syntax error.
