1. The presence of a "DAT" extension is meaningless - the programmer
could name the file - and its extension - anything he wished.
2. There does not exist a conversion routine to change an arbitrary COBOL
file
to anything known to God or man.
3. The file may contain virtually any sized records each containing a
variable number of fields each of which can be in a variety of renditions
(binary, ASCII, EBCDIC, graphics, etc.). Further, the file itself may be
arranged in a proprietary format (ISAM, etc.). And there does not exist
an external or internal clue as the the arrangement or format of any
of the above.
4. If all you have is the file, this is the only job you will every have.

Use the COBOL product that produced the .dat file to create a
display-readable .txt file, then use MS Access97's Import feature, and
viola, you have your data.

If you do not have the COBOL product that produced the file, then head
over to www.dejanews.com and do a PowerSearch on this forum on the
keywords COBOL, Conversion, and ODBC.  This topic has been mentioned
in the past, and a product capable of granting ODBC access to the
COBOL files produced by various products was mentioned.  I would tell
you the product name myself, but since it didn't interest me at the
time, I didn't remember it; and anyway.

Let me explain.  

This is like asking - "Can someone provide a
tool/utility/procedure/... to convert a RDBS data file to plain
ASCII-text".  Which one?  

Try to convert a Oracle database to ASCII, without using Oracle!
Try to convert a Microsoft Access database (.mdb) file, without using
Access!
There are numerous brands of COBOL on numerous systems - SOME use
proprietary, vendor invented, file strutures.  Others use the native
file structure of the platform involved.  

We need more information.  Companies such as LIANT provide ODBC
drivers to access many different type of COBOL data files.  There is a
company called "Data Junction" that makes a very good tool for
converting various data formats, take a look at that.  Tell us which
COBOL version, brand and platform and we'll try to help further.
