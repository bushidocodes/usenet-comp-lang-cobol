One of my regular uses is to convert date formats from MMDDYY to YYMMDD. If
you have a program which has a lot of these, it's worth the effort. (Of
course, the program need notbe 'nested', it could be 'stacked'


PROGRAM-ID. MAIN.
...
...
   CALL MDY2YMD USING (any dataname with correct PICTURE)

PROGRAM-ID. MDY2YMD.
DATA DIVISION.
WORKING STORAGE SECTION.
01  YYMMDD  PIC X(06)
LINKAGE SECTION.
01 MMDDYY  PIC X(06).

PROCEDURE DIVISION USING MMDDYY.
   STRING MMMDDYY(5:2) DELIMITED BY SIZE
                MMMDDYY(1:2) DELIMITED BY SIZE
                MMMDDYY(3:2) DELIMITED BY SIZE
     INTO    YYMMDD
     MOVE YYMMDD to MMDDYY
     GOBACK.
END PROGRAM MDY2YMD.
END PROGRAM MAIN.

I totally endorse what you have said here, Arnold, and the object code bears
out my own contribution (which argued that MOVE has to be more efficient).

However, the  OI instruction  immediately after the PACK (which forces a
positive sign)  could be eradicated by making the pictures of YYMMDD and
DDMMYY signed. (The COBOL compiler generates this to force a positive sign
because the picture is unsigned...)

It's another example of how COBOL programmers can "help" to generate better
code by careful data definitions... in the old days we needed to; nowadays
you can argue that it makes so little difference it doesn't really matter...
('OR Immediate' <OI>, along with the other Boolean Operations, are amongst
the fastest executing instructions in the IBM mainframe's repertoire).

I remember once writing an entire program using ONLY 'OR', 'XOR', 'AND' and
'NOT' (the Boolean Operations) on an ICL 1900 in PLAN ('Program Language
1900', the "Assembler" for this machine range), just for a bet... (and to
demonstrate that computers really ONLY 'understand' binary operations when
all is said and done... <G>) This program read input data and actually
displayed processed data, all using ONLY the PLAN mnemonics mentioned...(NO
ACCEPT or DISPLAY or READ or WRITE...or Assembler equivalents) However it
DID modify its own instructions in a way which would make a COBOL ALTER look
like the epitome of good programming practice...<G>

Logical bitwise operations are blazingly fast on most platforms (once you
have the necessary registers loaded...(storage to storage gets a bit
slower...)) and you can do some quite 'interesting' things with them.

They are governed by the rules of Boolean Algebra and Propositional Calculus
(a couple of things very dear to my heart, since I was a very junior
programmer) which enforce an optimized and simple binary logic system, and
underlie the power of computer systems in general.

For those who MIGHT be interested (and I accept that the majority here won't
be...) here's something to whet your appetite....

Consider the case where two data fields must be "swapped" WITHOUT the use of
an intermediate area.

Impossible in COBOL? maybe... but using Boolean Operations, effortless...

field A:  1011   0011
field B:  0010   1100

(arbitrary values, just for fun...)

Using the Boolean Operation 'Exclusive OR' (this means: "one or the other,
but NOT both..."), we can obtain the following:(Compare fields bit against
bit, from right to left...)

1. XOR A:B (Exclusive OR the 2 fields and leave the result in field B...)

field A =  1011  0011
field B=   1001  1111

2. XOR B:A (Exclusive OR the 2 fields and leave the result in field A...)

field A =  0010  1100
field B=   1001  1111

3. XOR A:B (Exclusive OR the 2 fields and leave the result in field B...)

field A =  0010  1100
field B=   1011  0011

The original fields have miraculously swapped values without moving to
intermediate storage...it took 3 (extremely fast) machine instructions...

I leave it to the reader's imagination to think of circumstances where such
a facility could be useful...

Could we emulate this in COBOL?

A better question is: "Why would we want to?" (The 3 Machine instructions
are hardly "self-documenting" and, in COBOL, a bit of extra storage is never
a problem...)

(the answer is that it CAN be simulated in COBOL, but you'll have to write
to me or post here if you want to see how...<G>)

A much better use of this Mathematical Tool is in optimizing complex logic
(you may remember I posted a solution here, which was derived using the Laws
of Boolean Simplification, to Yukon Mama's COBOL "IF" logic maze. To be
fair, the "logic" which Eileen "inherited" was so awful that even such a
tool as this was not able to improve it much...)

The point I'm trying to make (I took the long way round as a bit of
self-indulgence) is that you can argue all you like about optimizing code in
COBOL... it is really irrelevant. If efficiency is your primary goal in
computer programming, don't use COBOL.

Now, before you all start putting on your white hoods and sharpening stakes,
I am NOT advocating that INEFFICIENT programs in COBOL are OK...all I am
saying is that intense arguments about it serve little purpose; efficiency
will NEVER be the reason that people use COBOL. When this design goal is
balanced against others, we can achieve an "acceptable" level of efficiency
with COBOL, and that is good enough for most people. But let's not delude
ourselves into thinking that ANY solution in COBOL can be the MOST efficient
possible....(It CAN be the "most efficient in COBOL" but is it really worth
fighting about?)

Pete.
