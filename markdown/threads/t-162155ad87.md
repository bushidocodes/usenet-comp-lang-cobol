```
to: CLC
 I just thought that I would start a new thread about the 2nd byte/character of 
file status values.

IBM has a "long-stranding" (and documented) extension, that allows one to code a 
file status field as
  Pic 99
rather than
  Pic XX.

See:
      http://publibfp.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/IGY3LR40/APPENDIX1.1
which includes

"A numeric data item of usage DISPLAY or NATIONAL in the FILE STATUS clause.
 [Standard COBOL 85 requires an alphanumeric file status data item.] "

When (if) IBM actually implements the '02 Standard feature of "user-defined 
SUCCESSFUL" file status values, this will become a real problem.  This certainly 
will/would be a problem if IBM ever provides a successful (rather than 
unsuccessful) file status value for what is currently handled by the value "97".

This IBM extension probably ALSO seems "odd" (at best) for those used to the 
Micro Focus (for example) feature where the second byte of a "9x" file status 
value contains a binary numeric value (from 0 thru 255 - I think).  This can be 
checked against something like X"AB" - or the more (user-friendly - but 
non-Standard" H"123".

The bottom-line (from my perspective) is that IBM COBOL programmers SHOULD be 
using "XX" and not "99" for file status fields NOW.  Whether IBM actually ever 
provides non-numeric 2nd byte/character values is certainly a "question", but 
you can't be hurt by starting to use the *STANDARD* definition now.
```
