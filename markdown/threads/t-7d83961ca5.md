```
It is always BEST to initialize (initialise) them in your program.  This can
be done via the INITIALIZE verb (if you do this several times in your
program) or via the VALUE clause if you only need it done once - at the
beginning.

You do NOT need to initialize fields that are first used as "receiving"
fields - but I have found it (usually) a good practice to do so - just in
case some maintenance comes along that expects them to be "tested" before
receiving anything.

Many (most?) compilers do have methods of setting up "defaults" for
initialization - but these differ from compiler to compiler.  For example,
look at the Micro Focus DEFAULT-BYTE directive or the IBM mainframe WSCLEAR
and STORAGE options.  However, as implied earlier, I wouldn't rely on these
as a "good programming technique".
```

Ephatically yes. Initialize constants using the VALUE clause, initialize
variables in the program logic (i.e. PROCEDURE DIVISION). All variables
used in COMPUTE statements should be initialized, keep in mind that the
receiving variable in a COMPUTE statement is initialized by the value
returned by the COMPUTE, its the other variables in the COMPUTE statement
that you really have to worry about so make sure they are properly
initialized.

It is good programming practise to initialize your variables and believe me
its a real progarmming time saver (in the long run).

have a valid default value.  Just what this default is depends on contents
of data name and size of data name.  The default for a one byte data name
would be 32.  a 2 byte item defaults to 8224 (assuming ascii spaces).

    Non binary data names normally default to non numeric.

    So far as initializing fillers, I suggest a move spaces before the
initialize.  A move spaces is trvial in code size and time of execution.

initialize them in some manner, they will not trigger a run time error as a

binary numeric items, but might cause problems elsewhere, as with pic X
items that seem to be blank, but are not = spaces.
