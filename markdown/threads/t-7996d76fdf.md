There are actually quite a few COBOL screen handlers which utilize a
CALL USING statement. Micro Focus DISPLAY/ACCEPT doesn't really
support the ability to display GUI screens (which is what I assume is
your objective) in Windows.

You could use your existing copy of Micro Focus COBOL (assuming that
it is version 2.5 or higher) with COBOL sp2 and display your screens
in native Microsoft Windows. But COBOL sp2 also uses CALL USING
statements. Access http://www.flexus.com and download an evaluation
version for Micro Focus. I suggest that you try to solve the problem
using COBOL before you embark on a project to re-write (or even worse,
automatically convert) to VB.

› 2."writing a windows program" means to care windows standarts. is
› everything to port only screens to windows. what about file systems. CALL
› "SQL" USING XZ for file I-O . WHAT ELSE, it is cobol or "CALL language"

CALL USING is perfectly acceptable under the ANSI 85 standard. In
fact, if I'm not mistaken, it was also in the ANSI 68 and ANSI 74
standard. Why do you think that CALL USING isn't valid COBOL syntax?

› 3. other like this reasons ...
› it must be very easier than this, must seem very like COBOL, and also it
› must be port same dos sources to windows with emulating screens.

That's asking a lot. Was Microsoft Quick Basic DOS screen handling
source code emulated in Visual Basic? To be honest, I realy don't
know the answer to that question, but I doubt it.
