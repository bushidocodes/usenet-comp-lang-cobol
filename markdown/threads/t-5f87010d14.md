If by your last paragraph you mean write records back to the data-file -
no way Jose. That's the cop-out MicroFocus use in their demo telephoone 
application. Read in a sequential file, change entries in the ListBox
and then write a new output file when the program is exited. That just
isn't the real world; and suppose a user had spent three hours changing
records in the Listbox and the machine goes belly-up - he/she's gonna
really love ya.

My data is fairly compact related to one oil plant, but I would have to
speculate that if I were in a 'large' environment, huge customer file
and multi-users - then I would probably consider a timer to close, then
re-open the data file based on user activity or how many records had
been updated - at least that way a whole day's entries aren't lost.

As I said my data is compact per gas plant - whereas the oil companies,
God bless 'em - they think real big, (call it mainframe mentality if you
wish), shoving all data into one huge DB - and boy do they have fun when
the DB crashes. 

To be fair, and as background to what I'm doing - I'm creating OO
collections of variable length, I can set ordered or sorted
when designing the Listbox in M/F Dialog Editor (which is a spin-off of
M/S Dialog Editor, plus 4 additional controls). The Dialog Editor is 
set to sorted by default (property, if you like), but I can override it. 

In terms of updating the Listbox it is as follows, bearing in mind I
have a pointer(index) to where a current record appears in the list and
I have the PrimeKey for the datafile  :-

	*> Extracts from my Class : ClientsDBI 

	*> NOTE : I've already updated the Client Datafile before 	
	*> triggering these updates to the collection - if the file
	*> update gives me a file-status error, not much point in
	*> attempting to update the collection - is there <G>

	OBJECT-STORAGE SECTION.
	01 os-Table.
           05 os-Collection occurs 2		object reference.

	   *> Collection 1 = RecType 1 - Clients
	   *> Collection 2 = RecType 2 - Facilities (for this Client)
       *>------------------------------------------------------------
       Method-id. "add-to-collection".
       *>-------------------------------------------------------------
       Local-storage section.
       01 ls-string                        object reference.
       Linkage section.
       01 lnk-record.
          copy "clients2.cpy" replacing ==(tag)== by ==lnk==.
       01 lnk-Collection                object reference.

       Procedure Division using     lnk-record
                          returning lnk-Collection.

          move lnk-record to CLIENT-record-contents
          move CLIENT-recordtype to ws-CurrentRectype
          invoke self "field-positions"
          invoke self "make-list-record" returning ls-string
          invoke os-Collection(ws-CurrentRectype) "add"
                 using ls-string returning ls-string
          set lnk-Collection to os-Collection(ws-CurrentRectype)
       End Method "add-to-collection".
       *>-------------------------------------------------------------
       Method-id. "change-collection".
       *>-------------------------------------------------------------
       Local-storage section.
       01 ls-Index1                         pic x(4) comp-5.
       01 ls-Index2                         pic x(4) comp-5.
       01 ls-string1                        object reference.
       01 ls-string2                        object reference.
       Linkage section.
       01 lnk-index                         pic x(4) comp-5.
       01 lnk-record.
          copy "clients2.cpy" replacing ==(tag)== by ==lnk==.
       01 lnk-Collection                    object reference.
       Procedure Division using     lnk-index
                                    lnk-record
                          returning lnk-Collection.

         move lnk-record to CLIENT-record-contents
         move CLIENT-recordtype to ws-CurrentRectype
         invoke self "field-positions"
         move lnk-index to ls-Index1 ls-Index2
         invoke os-Collection(ws-CurrentRectype) "at"
                using ls-index1 returning ls-string1
         invoke self "make-list-record" returning ls-string2
         invoke os-Collection(ws-CurrentRectype) "replaceAll"
                using ls-string1 ls-string2 ls-index1 ls-index2
         set lnk-Collection to os-Collection(ws-CurrentRectype)
       End Method "change-collection".
       *>-------------------------------------------------------------
       Method-id. "delete-from-collection".
       *>------------------------------------------------------------
       Local-storage section.
       01 ls-string                         object reference.
       Linkage section.
       01 lnk-index                         pic x(4) comp-5.
       01 lnk-rectype                       pic 9(01).
       01 lnk-Collection                    object reference.

       Procedure Division using     lnk-index
                                    lnk-rectype
                          returning lnk-Collection.

         move lnk-rectype to ws-CurrentRectype
         invoke os-Collection(ws-CurrentRectype)
                "removeAtIndex" using lnk-index returning ls-string
         set lnk-Collection to os-Collection(ws-CurrentRectype)
       End Method "delete-from-collection".
       *>------------------------------------------------------------

Having updated the os-Collection in the DBI(Data Base Interface), via
linkage, it is passed back to the Edit program which tells the Dialog to
re-display the ListBox using the current object reference for the
collection.

Now there is a kicker to all this, even with OO - what is an acceptable
size for the collection, based on the machine you are using. That is why
I started a new thread 'Scrolling records', but it got little response;
Michael Mattias suggested using a relative file as a workfile and
SkippyB mentioned that in a mainframe environment he set a table-size to
22 records which was what could be displayed on a terminal at one time.
I know I'm pretty safe with the volumes I'm handling, but this requires
serious consideration in large data systems.

To an extent it is comparing apples with oranges, but what is being done
in an OO design can be emulated in a text environment by setting a
maximum table size and using OCCURS DEPENDING; say table-size was a max
of 100 then you would need to store First and Last Keys in the table to
be able to READ NEXT or READ PREVIOUS as the user attempts to move
forwards/backwards through the list. NetExpress provides a Table Sort
(not sure if other compilers do) so a 'changed' text list for a non-OO
application could be re-sorted before re-displaying in the Listbox. As
an alternative, record pointers within Fujitsu probably provide the same
ability to re-sequence. (?)

Hope this clarifies the points I was making - with a qualifier - depends
upon the concepts you use for designing the system, rather than any
constraints from the compiler.
