This issue has certainly been discussed before, eventually I managed
to convince the other party that you couldn't just read the data file
records using, say, a text editor, and expect this to be a wholesome
set of records because, as you point out, 'deleted' records still
exist with just the terminator set to x0D00.

The end of the file may also contain junk which may appear to be data
records depending what was left in the block buffer.

The default for Unix/Linux system is/was C-ISAM style MF Level II
format rather than the C2 default on some other systems.  Neither
'null fill' deleted records.

I don't know of any reason why it would be useful to null fill the
records if the files are being used properly. If you are using
sequential organization or other languages to access the data it is
essential they check the terminator.

If you need to have the file used by a system that can't check the
terminator then open the file with sequential organization and a
record two bytes longer and for each read that results in a record not
having x0D0A then null fill and rewrite. You may also get a 9/139 or
something at the end which is 'partial record read'.
