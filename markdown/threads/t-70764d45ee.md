```
I had an issue surrounding the following COBOL statement.  We use LE
COBOL on z/OS.
IF PCT> 0

When the value of PCT, which is defined as PIC 9.99, was 0.00, the
0.00 was greater than 0. This surprised me; I thought a numeric 0.00
should be the same as a numeric 0.

I tried to change the offending statement to
IF PCT> 0.00.
COBOL didn't like that. I got a compile error.

My next approach was to define a constant like so:
05 C-ZERO PIC 9.99 VALUE 0.
COBOL hated that. Another compile error.

My next approach was to define the constant like so:
05 C-ZERO PIC 9.99 VALUE ZERO.
That compiled OK, but the value of C-ZERO was 0000, and I was looking
for 0.00.

The final attempt worked like a charm:
05 W-ZERO PIC 9.99.
MOVE 0 TO W-ZERO
IF PCT > W-ZERO

All of these findings surprised me.  I was disappointed that I had to
define a variable and initialize it to zero in the procedure division.
```

It is. 

But you're not comparing a numeric 0.00 to a numeric 0. 

You're comparing a numeric-*edited* 0.00 to a numeric 0, which is an 
alphanumeric comparison. And in an alphanumeric comparison involving operands 
of unequal length, the shorter operand is padded on the right with spaces to 
the length of the longer one.

In a nutshell, you're comparing "0.00" to "0   " with results exactly as 
should be expected.

>I tried to change the offending statement to
>IF PCT> 0.00.

I'm pretty sure that numeric-edited IS alphanumeric, with the added 
benefit of being able to move a number into it and have the formatting 
occur automatically.  :)  While you're only using a decimal point in 
this particular case (which some would consider an acceptable character 
for a numeric expression), editing can insert all different kinds of 
characters - the one I've used it for most often is spaces.  There's 
pretty much no way that spaces can be numeric (unless you count their 
ASCII character code).

I concur with Pete Dashwood's earlier suggestion to look up the 
definitions in your manual - while you may know COBOL backwards and 
forwards, the manual will often have information on that particular 
implementation that can be useful.  (I know that the Unisys 2200 COBOL 
compiler was like that.)  Judson McClendon's suggestion of FUNCTION 
NUMVAL is also a good one - though, keep in mind that intrinsic 
functions (in COBOL 85) don't have the most robust of error handling 
mechanisms.

```
Where you started out saying you were on z/OS (and I assume using Enterprise 
COBOL), I thought I would give you a few online references - but first, let me 
confirm that the BEST solution for you is probably

 If Function NumVal (PCT) > 0    (or Zero - you'll get the same results).

This ASSUMES that you have already "validated" that the field actually matches 
its picture, i.e. that you don't have all spaces or other non-numerics in it.

Now, for online references:

      http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3lr31/5.3.12.3.4
"If the usage of the item is DISPLAY, any associated VALUE clause must  specify 
an alphanumeric literal or a figurative constant. The value is  assigned without 
editing."

(This explains why you couldn't have
   Pic 9.99 Value 0
It also explains why
   Pic 9.99 Value Zero
ends up with "0000"

    ***
"Table 24. Comparisons involving data items and literals"
at
    http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3lr31/6.1.6.5

shows how (USAGE DISPLAY) numeric-edited items (category) are treated as 
alphanumeric (class) in comparisons

This goes along with the table at:
  http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3lr31/5.1.6.6

Up until "relatively recently" (later releases of Enterprise COBOL - and the '02 
ANSI/ISO Standard), numeric-edited items were always alphanumeric.  Now, 
(current Enterprise COBOL and ANSI/ISO '02 Standard), they are CLASS 
alphanumeric if they are USAGE DISPLAY and national if USAGE NATIONAL.  (this 
can be implicit or explicit depending on other factors).

   ***

One other thing to mention, is that for those who suggested using

  If PCT > "0.00"

this has some POSITIONAL problems (as does the NUMVAL solution) if you have bad 
data (non-numerics or other issues) with the field.  For example, if PCT has a 
value of "00.0" this would cause an alphanumeric comparison between
  "0.00" and
  "00.0"

The 2nd would be LARGER (EBCDIC collation sequence) if you use the
  > "0.00"

Other bad data may cause other unpredictable or undesirable results.

Bottom-Line,
  You really need to validate your data before doing the comparison (or know it 
is "good" already)
```
