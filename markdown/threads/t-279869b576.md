```
Hi there,

I am having a problem converting this monstrous query into cobol code as it
takes 14 hrs to run this QMF Proc. Could any one help me out pls ?? It would
be greatly appreciated. Thanx in advance for this help.

Here is the query  !!

SELECT FULL_TERM, A.REM_MONTHS, COUNT(*), SUM(A.BILL_BALANCE)
     , END_LMTH_CHAR
FROM CSTMT A, LCUST B, ACCPLN C
   , CSTMT D
   , MDATE
WHERE D.CUST_NBR      = A.CUST_NBR
  AND D.ACCT_NBR      = A.ACCT_NBR
  AND A.STMT_DATE     >= (SELECT STRT_2MTH
                           FROM MDATE )
  AND A.STMT_DATE     <  (SELECT STRT_LMTH
                            FROM MDATE )
  AND A.BILL_BALANCE  > 0
  AND A.REM_MONTHS    > 0
  AND D.STMT_DATE     >= (SELECT STRT_LMTH
                            FROM MDATE )
  AND D.STMT_DATE     <= (SELECT END_LMTH
                            FROM MDATE )
  AND D.BILL_BALANCE  < 0.01
  AND D.CREDITS       < 0

  AND A.CUST_NBR       = B.CUST_NBR
  AND A.ACCT_NBR       = B.ACCT
  AND B.ACCT_TYPE      = C.ACCT_TYPE
  AND B.REVL_PLAN_NBR  = C.PAY_PLAN
  AND B.ORG_ID         = '600555'
  AND B.ACCT_TYPE      = '30'
GROUP BY FULL_TERM, A.REM_MONTHS
       , END_LMTH_CHAR
ORDER BY 1,2


Jagster
```

You could plug the statement into a Cobol program, but it wouldn't run any
faster. You'll have to turn it into a cursor and add FETCH .. INTO. Conversely,
if the solution turns out to be a temp table, you can do that with QMF as easily
as with Cobol. 

I would try these fixes:

Delete the sub-selects on MDATE. Move the three dates to the top and do a simple
compare. In the Cobol version, read them with a separate SELECT and store them
in host variables. 

Change COUNT(*) to COUNT(A.ACCT_NBR). You are wastefully asking it to read all
fields just to get a count.

Run an EXPLAIN PLAN to find out where it's spending all its time. Look for
missing indexes, especially on C.ACCT_TYPE. Make sure the optimizer is turned
on.

Try changing the self-join -- first 14 lines of WHERE -- into an INNER JOIN. The
optimizer should be smart enough to do that for you.

Try making the detail SELECT into a sub-select with DISTINCT on CUST_NBR. After
finding the first pair for a customer, it will stop looking for more.

AND A.STMT_DATE     <  :MDATE-STRT-LMTH
should be 
AND A.STMT_DATE  <= :MDATE-END-DATE

Doesn't CSTMT have a BEGINNING_BALANCE or PREVIOUS_BALANCE? Using that would
eliminate the self-join. To find out tell QMF SELECT * FROM CSTMT FETCH FIRST 1
ROWS ONLY. If that doesn't work, try SELECT * FROM CSTMT WHERE row_id = 1

Issuing a SELECT against LCUST and ACCPLN for every account could be slow,
depending on how many accounts are involved. It may be faster to let SQL read
them as you had in the original. 

Delete 3 from ORDER BY. A single account has only one value.

There is no need to sort your *many* account rows on FULL_TERM, etc. Setup a
COBOL table (array) keyed by  FULL_TERM, REM_MONTHS and  END_LMTH_CHAR,
containing a record counter and accumulator for BILL_BALANCE. For each account
returned by FETCH, look it up with the COBOL SEARCH verb, if not found add it to
the table, then add 1 and BILL_BALANCE. 

During the run, say every 10,000 accounts, you may want to sort the table
descending on count. That will put frequently used entries near the front so
sequential lookup runs faster. At the end, you may want to sort on the key for
presentation. IBM COBOL doesn't support table sorts. Assuming the table is small
(<200), a bubble sort is adequate.

01  accum-table.
     05  accum-limit value 1  binary  pic s9(4).
     05  accum-entry occurs 1 to 300 depending on accum-limit indexed x-accum.
         10  accum-key.
              15  accum-full-term     pic  x(xx).
              15  accum-rem-months pic  9(xx).
              15  accum-end-lmth-char pic x(xx).
         10  accum-count     binary  pic s9(9).
         10  accum-bill-balance binary pic s(11)v9(2).

 lookup-and-accum.
     move FULL-TERM to accum-full-term (1)
     move REM-MONTHS to accum-rem-months (1)
     move END-LMTH-CHAR to accum-end-lmth-char (1)
     set x-accum to 2
     search accum-entry
         at end
             add 1 to accum-limit
             if  accum-limit > 300
                 display 'table overflow ' accum-key (1) goback
             end-if
             initialize accum-entry (accum-limit)
             move accum-key (1) to accum-key (accum-limit)
             set x-accum to accum-limit
         when accum-key (x-accum) = accum-key (1)
             continue
     end-search
     add 1 to accum-count (x-accum)
     add BILL-BALANCE to accum-bill-balance (x-accum).
 sort-accums.
     move 1 to out-of-order
     perform until out-of-order = zero
         move zero to out-of-order
         perform varying x-accum from 2 by 1 until x-accum not < accum-limit
             if  accum-key (x-accum) > accum-key (x-accum + 1)
                 add 1 to out-of-order
                 move accum-entry (x-accum) to accum-entry (1)
                 move accum-entry (x-accum + 1) to accum-entry (x-accum)
                 move accum-entry (1) to accum-entry (x-accum + 1)
             end-if
         end-perform
     end-perform.

 IDENTIFICATION DIVISION.
 PROGRAM-ID. TESTSQL.
 ENVIRONMENT DIVISION.
 DATA DIVISION.
 WORKING-STORAGE SECTION.
     EXEC SQL BEGIN DECLARE SECTION END-EXEC.

 01 SQLSTATE    PIC X(5)         Value Spaces.
 01 SQLCODE     PIC S9(9) COMP-5 Value 0.
 01 SQLMSG      PIC X(255)       Value Spaces.
     EXEC SQL END DECLARE SECTION END-EXEC.
 PROCEDURE DIVISION.

     EXEC SQL
             SELECT FULL_TERM, A.REM_MONTHS, COUNT(*),
                    SUM(A.BILL_BALANCE),
              END_LMTH_CHAR
       FROM CSTMT A, LCUST B, ACCPLN C,  CSTMT D,  MDATE
       WHERE D.CUST_NBR = A.CUST_NBR
         AND D.ACCT_NBR = A.ACCT_NBR
         AND A.STMT_DATE >= (SELECT STRT_2MTH FROM MDATE )
         AND A.STMT_DATE 0
         AND A.REM_MONTHS > 0
         AND D.STMT_DATE >= (SELECT STRT_LMTH FROM MDATE )
         AND D.STMT_DATE <= (SELECT END_LMTH FROM MDATE )
         AND D.BILL_BALANCE < 0.01
         AND D.CREDITS < 0
         AND A.CUST_NBR = B.CUST_NBR
         AND A.ACCT_NBR = B.ACCT
         AND B.ACCT_TYPE = C.ACCT_TYPE
         AND B.REVL_PLAN_NBR = C.PAY_PLAN
         AND B.ORG_ID = '600555'
         AND B.ACCT_TYPE = '30'
       GROUP BY FULL_TERM, A.REM
       ORDER BY 1,2
     END-EXEC.
