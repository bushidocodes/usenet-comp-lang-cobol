This is all about editors and how to make them really smart by
understanding the fundamental nature of the language. 

I described the Small talk editor that can automatically pull out
logic into a new method.  Jimmy has requested a sample.  Here is some
code code from my parser.

/*****************************************************************
*         The input-output section  basically the heading and
*         comments...
******************************************************************
       F000-PROCESS-IN-OUT-SECTION.

           PERFORM U000-READ-TOKEN
           EVALUATE TRUE
      *      We have another section
             WHEN NOT INPUT-OUTPUT-SECTION
                 CONTINUE

             WHEN TOKEN-EJECT ( ACTIVE-TOKEN )
             WHEN TOKEN-SKIP1 ( ACTIVE-TOKEN )
             WHEN TOKEN-SKIP2 ( ACTIVE-TOKEN )
             WHEN TOKEN-SKIP3 ( ACTIVE-TOKEN )
                 CONTINUE

             WHEN TOKEN-COMMENT ( ACTIVE-TOKEN )
                 CONTINUE

             WHEN OTHER
                 DISPLAY 'Parsenv: in input-output '
                         TOKEN( ACTIVE-TOKEN ) ' : '
                         TOKEN-WORD (1:TOKEN-WORD-LEN)
                        ' is not something I understand'
                 ADD   1             TO TOKEN-ERR-CNT

           END-EVALUATE
***************************

I have decided that the error handling routine is better handled by a
separate method.  In the small talk editor I highlight the display and
the add 1 and press a move to new method button.

The editor requests a new name (say error-handling) and I press enter.
It inserts a method call in place of the old code and creates a new
method amending the interface definition (the prototype of each call
for the method for compile time check - read the new standard or a
good book).

The editor knows that token and active token are part of the object
(say).  No need to pass then in the method call.  The word length is
transitory (part of this method not part of the object storage) so
this would be passed to the new method (using clause) the interface
has the word length defined in the interface and the call statement
adds the word length as a parameter.

All this is achieved with very few keys strokes and is integrated
straight into the editor.  It was originally built outside the editor
but they found that very few people used it.  Once it was in the
editor then everyone used it.

Right.    A good "structure" editor can move language pieces without making
syntax mistakes,
because it knows the language syntax.   To go beyond that, to move code
without making
semantic mistakes, the editor must know the semantics of the language too,
as the Smalltalk
refactoring editor seems.

Most interesting changes to programs can be considered as changes that
either preserve semantics
(usually optimizations) or modify semantics (change of functionality).
You can carry this out
with editors if the changes are small.    If the changes are big or
pervasive, you really want a
tool.

One tool designed for massive changes, that knows COBOL syntax and some
degree of COBOL semantics,
is the DMS Reengineering Toolkit
(http://www.semdesigns.com/Products/DMS/DMSToolkit.html).

Jimmy,
Although undocumented (and therefore not something support are likely to 
want queries about :-), the class wizard works off a template file 
(class.wiz) in your \Base\Bin directory. Names generated by the wizard are 
specified between '%' markers, and lines beginning "*> OCWIZARD" allow the 
wizard more specific access to add code (so don't delete them!). 

You should be able to *reasonably* safely edit the template to your style.

Be warned, though, that these template files *are* likely to change 
a bit between product versions.
