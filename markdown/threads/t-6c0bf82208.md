Check your compiler options for NUMPROC(PFD) or NUMPROC(NOPFD). The PFD
flavor basically tells the compiler to use (for example) CLC (Compare
Logical Character) instructions instead of CP (Compare Pack) instructions
against Packed-Decimal data greater than 1-Byte. I've also seen CLI (Compare
Logical Immediate) instructions used instead of CP when the Packed-Decimal
data is only 1-Byte and PFD is active.

The generated instruction for your numeric check should be a TRT (Translate
and Test). You'll have to look at the PMAP (specify LIST,NOOFFSET for PMAP
generation along with your other compiler options). Also, if the numeric
checks are in fact TRT instructions, you should find that the Translate
Table address (operand 2 of the instruction) is the same for Variables 'A'
and 'C'. Variable 'B' should point to a different table address.

If someone has specified PFD, tell them that this is a risky option and that
they need to always use NOPFD just to be on the safe side. I use CLC's in
ASSEMBLER only if I KNOW without a doubt that I have LIKE sign-nibbles. Data
from other sources should always be handled by program's using NOPFD. That's
just my opinion. The savings is NOT worth the headache. Granted, CLC's use
about 30% of the clock-time needed by CP's, but you'll find this to be
almost negligible in the overall scheme of things.

HTH....

Happy Holidays,

WOB,
Atlanta

In IBM mainframe COBOL, a PIC 9(5) COMP-3 field would have an "F" sign. 
The value 12,345 would have an internal representation of X'12345F'.

There's no way in IBM mainframe COBOL to define an elementary three-byte
numeric field containing X'012345'.  That's BCD (Binary Coded Decimal)
and it's not a supported numeric type.  IBM packed decimal always has a
low-order sign nybble, X'C' for positive, X'D' for negative, and X'F'
for unsigned.

You can define a field as PIC X(3) VALUE X'012345'.  But you won't be
able to do arithmetic with it in the IBM world.

Hey Arnold,

03 WS-PACKED-X.
   05  WS-PACKED-V9 PIC S9(16)V9 PACKED-DECIMAL.
03 WS-BCD PIC X(08) VALUE X'1234567812345677'.

MOVE WS-BCD TO WS-PACKED-X (1:8).
MOVE X'0C' TO WS-PACKED-X (9:1).
ADD +1 TO WS-PACKED-V9.
MOVE WS-PACKED-X TO WS-BCD.

WS-BCD now contains X'1234567812345678'. Check your PMAP and see.

I don't know if other platforms can use this technique. I guess it's a matter
of trying it out.

Look for COMP-6 in several compilers.  (I know that MF COBOL supports it -
but I think that it is an emulation of someone else's system.)

COMP-6 supports a "truly" unsigned packed-decimal item (so you can fit 6
digits in 3 bytes).

FYI,
  The Standard says that even USAGE PACKED-DECIMAL is implementor defined
(to some extent) and this is exactly why.  Given
   05  Whatever Pic 9(06) Packed-Decimal
the Standard neither requires nor prohibits the use of a "sign-nibble"
