Bob,

I'm going to take a different tack here. I don't think you (any of us in
the commercial DP world) should worry very much about internal
efficiency, e.g., execution time, memory utilization, etc. The compiler
developers can, and do, address these issues and make improvements over
time. In a commercial DP environment the key issue (i.e., where the DP
budget money goes) is program maintenance. Subject to your
installation's standards, I strongly urge you to:

1. Write truly structured code, meaning no GO TOs, no THRUs, no ALTERs,
etc. Keep in mind that structured code is driven from the top down, not
from the bottom up. E.g.,

PERFORM 1234-LOOP VARYING SS-1 FROM +1 BY +1
UNTIL LOOP-IS-DONE
OR SS-1 > +10000

All paragraph 1234-LOOP does is what it needs to do with the data and
setting LOOP-IS-DONE (if that occurs). It does NOT increment SS-1, test
for +10000, or GO TO 1234-LOOP;

2. (Assuming your compiler supports them) use the scope delimiters, esp.
END-IF;

3. Resist adding functions to a program to the point where it becomes
huge - split it functionally. I did some work in the late 80's
restructuring Cobol software and the president of one of the companies
marketing Cobol analysis & restructuring tools told me that program size
is the single biggest factor in maintainability (and I agree);

4. Write GOOD comments as you write a program, i.e., if you sit with a
business analyst for 2 hours to get straight the rules for adding a
certain kind of financial instrument to the master file, make a big
comments box in the source and list all the rules, who you met with &
when. This is worth gold a year or two later when the rules get changed,
even if you are the programmer who makes the modifications.

This is a first pass off the top of my head - I'm sure there are more I
didn't think of at this time. Email if you have questions.

Regards & good luck,
Bill Lynch

While it is true that the machines and language keep getting faster, and
also true that maintenance and so forth are key elements of a program's
utility, simply dumping efficiency considerations is not really a good
idea. Some simple (real world, I swear!) examples:

1) A computer science pundit published an article in which he stated that
programmers should no longer sort files, or even think about sorting. All
modern production environments would use indexed databases, so sorting was
irrelevent, and counter productive. About a week after his article was
published we modified a program which was running against a large, modern
indexed database. We did not actually change the code, we simply put a
sort on the incoming transaction file to place the records in the physical
key sequence for the large account file they would be updating. The
execution time for the nightly run of that program (one of 180 in the
nightly cycle) dropped from an average of six hours, to 15 minutes (wall
clock time). This significantly helped us meet our four hour nightly batch
window.

2) A large human resources system on which I worked had a very structured
program in which, as part of evaluating the 401k contributions for an
employee, a module looked up, from an indexed file, the relevant factors
for the employee's investment options. The evaluation was in a small, very
structured routine, which made for very tightly cohesive, loosely coupled
code. The program also read the file (which had 36 records in it) about 25
million times per run, and ran for several hours (the file was on a heavily
loaded system device). By moving the read for the record outside of the
small routine, and checking whether the record was the last one read
(hence, still in the buffer), the number of reads dropped to 1 million, and
the execution time dropped to under an hour. After noting the small total
number of records in the file, we further altered the program to read the
entire file into a working storage table, and the execution time dropped to
under five minutes.

3) A piece of pseudo-code written by a senior consultant (later a partner)
of a big five consulting firm specified the following. For the online data
entry routine for new loans in a student loan guarantee system (the second
most often used screen in the system), the operator would enter the social
security number for the applicant. The system needed to check against a
list of current customers to see whether it should present a screen to
request borrower address information. The borrower address was held on a
master VSAM file with a primary key of (you guessed it) social security
number. The fully structured, highly modular pseudo-code would a) set a
'borrower-found-flag' to 'no'; b) move low values to the borrower key; c)
read the next greater borrower record; d) if the SSN matched, move 'yes' to
borrower-found-flag. The best part was e) the routine would continue until
end-of-file, whether it found a match or not, and then f) check whether the
flag was set to 'yes' or 'no'. (This code, BTW, won the
'All-time-dumbest-code-in-the-system' award for this $33,000,000 5-year
project). The routine worked perfectly in the test stages (against a
borrower file with 45 records). It did, however, cause some response time
problems (system wide) when put into production, against a borrower file of
7 million records. The consultant, when questioned about this, replied,
"Efficiency was not one of the contract requirements."

I fully agree that we should concentrate a great deal of effort on
maintainability, good structure, and clear documentation. You can go too
far the other way (like our senior analyst who advocates using GO TO
DEPENDING ON, because it uses fewer machine instructions than a PERFORM).
However, with the cost of machine time, user wait, and so forth, we have to
at least consider efficiency. As these examples show (and the millions of
others out there), a little thought can go a long ways. I don't think you
should sacrifice readable code or understandable algorithms, but a moderate
middle ground is the best approach.

Unfortunately, what is efficient depends not only on the platform but on
the compiler. A good place to start is the chapter "Optimizing your
CoBOL Program" or similar in the user's guide or programming guide for
your compiler.

In a typical CoBOL application, the efficiency of a program depends more
on the efficient use of I/O than on anything else, and that's a topic too
large to cover here. Find someone in your shop who knows how to do it
and learn it from him or her.

I would however like to put in a plug for the old VS locality of
reference, especially in transaction procession systems. Many people
reject recommendations for efficient code because they do not believe
there is sufficient payoff from reducing CPU cycles. But the reduction
of CPU cycles is only a side effect; the payoff comes from reducing
paging, which is after all a form of I/O.

Paging has a nonlinear effect on efficiency. The first page fault has a
drastic impact, and the second, and the third. Eventually you get to the
middle of the curve, where one more page fault more or less is hardly
noticeable. Then the curve starts rising steeply again as the system
begins to thrash.

The way to reduce page faults is to reduce your working set, the number
of pages you need in core at any one time. Keep your temporary data
together, on one page if possible. Keep your main line clear; handle the
normal case straight through. When there is an exceptional case, always
go out of the main line to handle it.

An average CoBOL program written without any thought to locality of
reference might take 100 page faults to handle an average transaction,
because the code is scattered all over the load module and touches every
page. Designed with locality of reference in mind from the beginning, it
might take 10 page faults to start up, and if the transaction is not
exceptional, it will never load those other 90 pages at all.

For example, typical CoBOL applications have lots of error messages, and
error messages take up lots of space. "INVALID TRANSID - PLEASE
RESUBMIT" takes up as much space in the load module as a dozen lines of
assembler. You can take almost any old CICS transaction and get a
measurable improvement in response time simply by segregating the error
messages and the error code so they don't get paged in unless there's an
error.

Unfortunately, the CoBOL II optimizer, not being intelligent, can produce
bad results as well as good, and it is up to the programer to use it
wisely.

There are actually quite a few well-designed CICS transactions out there
in old OS/VS CoBOL precisely because VS locality of reference was widely
taught for a few years after VS came in (it has since fallen into
desuetude).

What happens when you take one of these and convert it to CoBOL II? Why,
the optimizer takes all the exception code that the original author so
carefully isolated and inlines it again, turning the 10-page-fault
transaction back into a 100-page-fault transaction!

In order to prevent the CoBOL II optimizer from destroying your VS
locality of reference, you must at a minimum put the things you don't
want the compiler to inline in different segments, and put the things you
do want the compiler to inline in the same segment.

There are two broadly different approaches to achieving execution
efficiency, each one addressing different issues. For best effect, both
approaches should be applied, to the extent that they are applicable. One
approach encompasses the methods discussed below; that is, optimizing the
'mechanics' of execution. You could call this approach 'tactical'. The
other approach is to carefully choose the fundamental algorithmic methods
used in the application. You could call this approach 'strategic'. In
general, algorithmic, or strategic, improvements have the potential for
greater improvements, depending on the application, and should always be
applied first. For example, you can sort a random 1,000,000 element table
in memory faster using quicksort than bubble sort, regardless of how
efficient the bubble sort code. But for sorting 10 elements, the simplest
sort algorithm is likely the best. You first choose the best algorithm you
can, then write the code as efficiently as possible, in that order. Of
course, we all should agree that speed is not always as important as ease
of maintenance. Particularly to the maintenance programmer! :-)

Let's not forget about subscripting (vs indexing). When using a
subscript
to access an array, the number must be converted to the offset matching
the occurrence value contained in the subscript. In order to convert to
an offset, the occurrence must be binary, if it is not binary it must be
converted to binary from packed or display.

The sequence is:
Numeric -> Packed -> Binary -> offset value.

Of course you should always use an index register to access an array,
but, sometimes there are reasons you cannot. If you must use a
subscript
the most efficient internal representation is binary.

"It is the twelfth occurrence" stated Holmes in his usual unemotional
manner.

"You have only a few scraps of a program and a partial dump, yet you are
able to decide which occurrence of the table is being referenced by the
index?" I questioned.

"The TGT (task global table) was the major piece of evidence" he said.
"As you can see, it begins at a displacement of X'4ED0' into the
program. Also, the beginning of the fields known as 'index cells' is at
a displacement of X'5100' into the program", the great programmer
stated.

"I see the EPA (entry point address) of the program is X'000B5C68'", I
retorted. "If I add the displacement of the index cells to that, I
calculate that the index cells begin at core address X'000BAD68', but
located at that address is the full-word (four bytes) X'00000058', which
hardly suggests occurrence 12 to me".

"That is the usual sticky point for debuggers, but the translation from
the core value to the occurrence number is quite straightforward.
X'00000058' equals '88' in base 10, which suggests we are interested in
the occurrence that begins at the 88th byte past the table's first
byte", he advised.

"Since the occurrences are each 8 bytes long", he continued, "and the
first occurrence begins at the 0th byte past the beginning of the table,
and the second occurrence begins at the eighth byte past the beginning
of the table, it follows that the 12th occurrence begins at the 88th
byte past the beginning of the table."

"One could say", I offered, "the hexadecimal value found in the
full-word index cell is added to the address of the beginning of the
table to calculate the address of the occurrence being referenced."

"Quite - but it takes some translation to decide which occurrence that
really is", he continued. "A formula could be devised:
O = 1 + V/L, where O = Occurrence, V = value of the full-word in base
10, and L = the length of each occurrence."
"In our case, O = 1 + 88/8 = 12", he added.

"If there were more than one index in the program, I surmise that their
values would appear in successive full-words following the first index
cell's full-word, in order as they were defined", I stated somewhat
smugly.

"Elementary, my dear Watson".

Patrick Herring wrote:
› 
› In article <340··.@192··3.214>,
…[7 more quoted lines elided]…
› What's the connection with printing, just out of interest?

It is expensive to convert input from decimal to binary, cheap to do
arithmetic in binary, and expensive to convert back to decimal for
printing. The heavy-duty computational applications for which early
computers were designed did not require a lot of printing.

When computing costs came down to the point that ordinary business
applications were practical, the hardware designers came up with (the
ancestors of) packed decimal, which is halfway between decimal and
binary. Actually, it's about a fifth of the way to binary. So it's
cheap to convert input from decimal to packed decimal, expensive to do
arithmetic in packed decimal, and cheap to convert back to decimal for
printing.

The classic example of a computational application is linear programing.
You take a few thousand bytes of input, convert to binary, compute for 96
hours, convert back to decimal, and print a few thousand bytes of output.
In packed decimal, the job could never finish, since it would take longer
than the MTBF of the machine.

The classic example of a packed decimal application is the page number on
a report. The only thing you ever do to it is add one, convert to
decimal, and print. Packed decimal was invented for this.


› This rule implies that the compiler adds 1 to COMPs without packing,
› but packs all non-COMP-3 fields in a COMPUTE or other arith clause.

Well, no, it doesn't. The point is, how many conversions do you do
versus how many arithmetic operations.

You're going to be adding to a counter a lot, but you're hardly going to
print it at all, so you might as well make it binary and get the
advantage of cheap arithmetic.

The hardware cannot use a packed decimal field as an index, so if you
insist on doing so, the compiler must generate the expensive conversion
anyway, so you have spent expensive arithmetic without avoiding the
expensive conversion, so you might as well make it binary in the first
place and get the advantage of cheap arithmetic.

On the other hand, if you must print a number, you might as well make it
packed to get the advantage of cheap conversion. For example, the amount
of a check is going to be printed on your bank statement, but it is going
to be subtracted from your balance just once. No point in going all the
way to binary for one measly subtraction.
