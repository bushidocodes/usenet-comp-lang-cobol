Do the conversion on a PC, using Microsoft's ODBC driver for flat files. 

..Install an Oracle ODBC driver on the PC
..Export the MF-COBOL "database" (indexed file) to a flat file
..FTP the file to a PC (not essential, you could read it from the Unix box)
..Describe the flat file in a 'schema', following driver instructions
..Write SQL to load the Oracle database from the flat file

You can do it in one day, with no programming required.

A straight conversion of a "Flat" COBOL accessed file to a relational
database such as Oracle buys little - except the added expense of an
Oracle database license and user licenses for each connection.

Often such a conversion is undertaken under circumstances where the
actual business problem being solved is one of the following:

access to the data from systems written in other languages
Open access to data via the internet or another network
Necessity for multiple entities to update the data
Corporate edict

All but the last are fairly easy to do with COBOL itself.  There are
pitfalls with the conversion, not the least of which is the fact that
as soon as the data is stored in an open system such as an RDB,
garbage can appear more easily in the data unless strict controls are
placed within the table definitions and in the programs and systems
that will now update the data.

If the determination is made that such a conversion is necessary then
what is needed are two things.  First - someone that understands the
COBOL based system.  Quite frequently COBOL systems have business
rules coded in the programs.  For example - a field labeled Whazzis -
might have a value of 0, 1, 2, 3, or 99.  These are the only valid
values and the field's values only have meaning in the context of how
the program makes use of the field.  Additionally, data might be
organized such that a single header record is referenced by multiple
detail records, or perhaps records of different types, or perhaps a
record that is a varying length table of values.  Great understanding
of the COBOL program is necessary.

Second - a qualified, and talented DBA to normalize the COBOL data
into the relational database to help make the data held there
meaningful.  I can't overstate this necessity.  You can't approach an
RDB - especially Oracle - as a neophyte and expect good results.  For
each RDB in use - you should have a full time DBA to monitor and tune
the database on a continuing basis -- something you may not have been
used to with your COBOL flat file based system.

I don't want to marginalize or detract from the awesome power of a
correctly implemented RDB.  Nor should a conversion to one be
trivialized.
