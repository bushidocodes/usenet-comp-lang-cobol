› Did you try turning on "optimization" in the compiler directives under

Again, if you're going to switch optimisation on for one product, you
have to do it for all products to get more accurate results. The
following is information on how to get the Micro Focus optimised
version:

MF COBOL by default will generate code which will run on as many
systems as possible. This means that for 16-bit products, the 8086
is the default target chip, and for 32-bit products the 80386 is the
default target. This can be changed using the "TARGET(chip)" generator
directive, eg. TARGET(386), TARGET(PENTIUM).

As the program is doing a lot of indexing operations, the "NOBOUND"
directive will eliminate bound checking code (once you have debugged
your program, you shouldn't need it).

As the program is using COMPUTATIONAL items, you could switch on the
"COMP" directive.

In addition, if you're running under DOS, use the XM memory manager,
and try compiling with the "PROTMODE" generator directive.

In summary, as a minimum, I would suggest the following:

NOBOUND TARGET(appropriate-chip) COMP [PROTMODE if DOS, and use XM]

There are other optimisation directives, but they do not apply to the
program you posted (to do with speeding up CALL statements,
specifically).

Let us know how you get on.
Cheers, Kev.

---- BEGIN EXCERPT ---- In general, the simpler the statements, the greater
the speed difference between a native version and a non-native version of
some program (regardless of language).

A more comprehensive way to do benchmarking is to use a program that uses
statements in a more realistic fashion - i.e., one that comes close to doing
what real programs do. The best benchmark that I know of along these lines is
CSAMPLE - it times statements based on a sampling of a large number of real
(though now old) COBOL programs.

In general, you should see about an order of magnitude difference between
native and non-native, though this can vary widely depending on the exact
statements you are timing and the implementations one is comparing (again,
regardless of language).

Perhaps an examination of a particular case is useful.

Assume that we want to move a single character in memory with a statement
like:

MOVE A TO B

On an Intel machine, this would generally generate 2 instructions in native
code.

Under ACUCOBOL-GT, this generates one pseudo-instruction. It requires 19
instructions to decode the pseudo-instruction (this is pretty optimal, though
in 5.0 we got this down to 17 instructions - pretty much at the limit for the
architecture except that we take a 2 instruction hit because we test to see
if we are in debugging mode for each instruction). Once at the particular
opcode, it takes about 5 instructions or so to compute each address, and then
the 2 instructions to do the actual MOVE. Thus, the difference is 19 + 5 * 2
+ 2 = 31 instructions vs. 2 instructions - about a 15-to-1 difference.

Now lets take a more complex case, like adding two USAGE DISPLAY unsigned
numeric items together. We still have the same overhead of roughly 30
instructions to get to the appropriate opcode and determine the addresses.
Add another 5 instructions to get the length of the data items (lets assume
they are the same size). The code to do the addition is hard to estimate,
but it is at about 5 instructions per digit. Let's assume 5 digits for now.
Thus, the native version would execute 25 instructions while the non-native
versions would execute 60 instructions, a difference that is less than
3-to-1. If the numeric items are signed, the amount of work to do the
computation rises substantially and the difference between the two
implementations drops to (probably) less than 2-to-1.

All this really says is (a) determining real performance characteristics is
non-trivial - you have to time many things to get a good understanding, and
(b) given otherwise identical implementations, a native version will always
be at least somewhat faster than a non-native version.

Ultimately, all that matters is whether or not performance is adequate to the
task. In our 10-year history, we have only had product returned about 5 times
because non-native performance was not good enough.

And lastly, we have reworked quite a bit of the instruction set for version
5.0. Under Intel, our performance is about 75% faster on average in that
version. ---- END EXCERPT ----
