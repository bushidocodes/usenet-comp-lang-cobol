Sure you do. Here's how:

Suppose you come into $100k that you could use in your business netting a
20% return.

By just using the $100k, at the end of the year you have the original $100k
+ $20k profit. You'll pay income tax on the $20k.

Alternative:
Buy $100k worth of stock in which you (and others) have a great deal of
confidence. Borrow against the stock from your bank at, say, 6%. At the end
of a year, what have you got?

1. The original $100k
2. Increased value of the stock, say $10k
3. Any dividend, say $2k
4. Profit from the use of $100K, $20k
5. Minus interest, $6k
Result, $26k. But you'll only pay tax on $20k because the $6k interest is
deductible.

All in all, you're 30% better off (assuming everything goes OK).

any rela experience with using the "EXEC DLI" calls.  But I know there are a
few differences.  For one, you cannot mix CBLTDLI calls and EXEC DLI
commands in the same program.  An EXEC DLI does *not* translate to a "CALL
'CBLTDLI'".  I just did a translate/compile to get some information.  Just
like an EXEC CICS, it translates them to several MOVE statements followed by
"CALL 'DFHEI1' USING ...".  You also don't code the PCB status areas in to
the LINKAGE SECTION.  The translator seems to define one called DLZDIB, but
it's structure is a bit different than the one used for the CALL interface.

As for why you have to use the CICS pre-processor for a batch program with
EXEC DLI commands, it's because "EXEC DLI" is not valid COBOL.  The COBOL
compiler does not know what it means.  Only the CICS pre-processor knows. 
The CICS pre-processor translates the EXEC DLI commands in to COBOL CALL
statements as I described above.

That's about all I know.  I was never able to get a program to run properly
using EXEC DLI calls, so I gave up.
