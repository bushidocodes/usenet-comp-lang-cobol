Hopefully I can clarify some of the problems. After this you'll still be
confused - but in a higher level.

I took the liberty to copy/paste the RTS-Switches. I took them from
Workbench 4.x, because some of them changed when the 32-bit compiler was
introduced. This list contains the meaning of switches both in 16-bit and in
32-bit.

The CANCEL statement has no effect on OS/2.

In 16-bit systems only MF-cobol programs will be removed from memory. Mixed
language applications will depend on other compilers ability to clean up.

In 32-bit systems te cancel-statement will remove the module from memory,
and you will depend upon the 'foreign' compilers ability to handle open
files, pointers etc.etc.

As far as I can see, the CANCEL will free up memory without any switch
settings. Actually there was a bug in the earlier versions of WB 3.2 with
Dialog System 2.5.<39. The memory was not freed properly. One of my
customers had to reboot at 1 p.m. until the bug was fixed.

The switch, you remember could be the 'M'-switch - It frees memory (28K)
below 640K in old DOS-systems, but if you invoke the application with .BAT
files, you might have a problem.

Please, Bill. Look at the list, and see if one of the switches ring a bell,
or if a switch is missing (I shouldn't be surprised).

Ib

The switch is giving you command.com's transient area for your application.
That means, that command.com cannot 'remember' last command entered. You
need that when you return to the batch file to stop or execute next program.

Remember that this switch is only for good old DOS. If you need to maintain
programs on DOS, I can recommend that you use XM. Another thing about old
DOS systems if they run on old computers - the problem will be solved in a
couple of months - they will not pass Y2K.

If this didn't help you out, let me hear about it. I have this funny feeling
that your problem is something else - Very large WS- or Linkage-areas,
Directives or Local Storage.
