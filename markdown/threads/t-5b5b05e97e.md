I've never used anything with TAB delimiteds - so that is a separate
problem. The file will have been created as variable in Workbench and the
FCD block knows this. So ignoring the TABS for the moment, something like
the following should work :-

FILE SECTION.

FD  Data-File
    record varying from 1 to 536 characters
       depending on ws-RecordSize.
01  Data-record           pic x(536).

I'm assuming you are reading a Sequential. If above is ISAM with the
Primekey as the first ten characters, then you would have to specify :-

    record varying from 10 to 536 characters
       depending on ws-RecordSize.

otherwise the compiler is going to 'groan' at you.

*>------------------------------------------------------
Method-id. "readNextVariable".
*>------------------------------------------------------

*> Used for VARIABLE LENGTH records

Linkage section.
01 lnk-ReturnValues.
copy "sqlreslt.cpy" replacing ==(tag)== by ==05 lnk==.
05 lnk-RecordSize                    pic 9(04).
05 lnk-record.
   10 occurs 1 to MaxRecordSize  *> (your 536)
      depending on ws-RecordSize     pic x.

Procedure Division returning lnk-ReturnValues.

  initialize lnk-ReturnValues
  Read Data-File next record
    At End
      Set fileFinis to True
    Not at End
      Move  Data-record (1:ws-RecordSize)
      to    lnk-record  (1:ws-RecordSize)
      Move  ws-RecordSize to lnk-RecordSize
  End-Read

  if   ws-fileStatus = "00" or "10"
       continue

  else set FileError to true
       invoke super "showFileError" using ws-ErrorParams
  End-if

End Method "readNextVariable".
*>------------------------------------------------------

If you ANIMATE the readNext you should see the 'current record length' in
ws-RecordSize.

What you need to do is run your file thru a "conversion" routine (NOT in
Micro Focus COBOL) first to "change" all the tabs to some other character
(that never appears in your data).  Then you can read in the file and use
UNSTRING to get all the actual fields (and those that are optional).

UNFORTUNATELY, Micro Focus (all products - as far as I know) does NOT
provide any way of reading in a file with TABS and not "converting" them.
The "T" run-time switch impacts the WRITING of files with/without tabs, but
not the reading.  See the documentation which says,

"On input, tab characters are always expanded to spaces to the next
position, regardless of the setting of the T switch."

This is a little tricky, but you could do the following:

Declare the file to be binary sequential, and 1 byte long...

Read the file through a loop, moving information into the first field
until you get to the x'09' then switch fields (you could use a table
to move the data, incrementing as you hit the x'09' bytes.)

Also check for a x'0A' byte that declares end of record, and then
reset yourself to the first field for the next bytes, and move the
table data into the appropriate fields for the record just processed.

This will get around the problem of expanding the X'09' to spaces, and
the problem of a variable length record being read as fixed length
causing an incomplete record at the end of the file.

I have several file workarounds that I do somewhat like this because
of issues with Line Sequential files.  Unfortunately I need record
delimited formats for some of the files I transfer from UNIX back to a
PC or other RECORD oriented environment...

Just as others may have missed it ...

As far as Micro Focus goes:

1) For RECORD sequential files, tabs are NEVER expanded to spaces (input or
output)

2) For Line Sequential files with EXPANDTAB=OFF, tabs are not expanded on
READs.  With it ON (the default), they are expanded.

3) The T run-time option, determines whether tabs are or are not expanded on
WRITEs (rewrites) for LINE SEQUENTIAL files.
