I've been busy putting out a production fire (luckily it ain't my fault,
ain't even my part of the system.)

I'll have a bit more time to look into a few more options for the Work-Flow
component.

I originally proposed changes to the current method ,Option A, but Option B
was already on the table, and I was told to proceed on Option B which
required the Work-Flow component. (Luckily, I got the Option A algorithm
incorporated into Option B.)

Using current production data I modeled (produced listings showing the
Work-Flow) the Current system, Option A, and Option B. (I wasn't supposed to
model Option A, but it's only a listing, I'm rather hard-headed, and they
humor me from time to time.)

The Option A results showed a significant improvement over the current
system, and the decision was made to implement Option A ASAP, and phase in
Option B. In the end, Option A will be used for 70% of the work, and Option

I would strongly advise you to use PowerCOBOL Projects and set up a project
for each function of your system.

(Avoid making projects too big... usually 3 forms is the limit for mine. It
is much easier to have many small projects that different people can be
working on, than to have a few large ones that people are waiting on...)

You could certainly control the Project files with PowerGEM.

Remember that PowerCOBOL is different from NETCOBOL, in that interrupt
driven event code is stored within the project framework, rather than as
separate COBOL Source files in the traditional sense.

Over time I have found that it is useful to code event routines as CALLable
common routines, rather than propagating event code all over a form. Most of
my event code (except the really trivial instances) simply says CALL xxxx,
where xxxx is defined as a common routine in the factory for the form. The
advantage of this is that you can maintain event code centrally for the
form, can cut and paste it easily to other forms, or even use a traditional
NETCOBOL .DLL, develop it externally in the traditional way, and call it
from the form. Consider this particularly for SQL access because this is the
area that is most likely to need maintenance. Personally, I don't use the
ADO/DAO COM controls that you can plug into your PowerCOBOL forms,
preferring instead to write my own SQL and access my databases myself, using
ODBC as a separation layer. I use the databases purely as repositories and
let COBOL do all the intelligent stuff. This is obviously best managed from
external .DLLs, which can be developed/maintained separately from
PowerCOBOL. (I have a number of PowerCOBOL applications that do this,
especially where the function required is complex. It never ceases to amaze
me that there is absolutely no problem with mixing NETCOBOL and PowerCOBOL
in this way.)

BOTTOM LINE:

Spend time on DESIGN. Know what forms you are going to have and what
Business functions they will support. Where there is "tricky" functionality
consider putting it into an external NETCOBOL .DLL, that can be CALLed from
the form. Identify all these areas and know what forms will CALL these
functions. PowerCOBOL is a brilliant "Quickbuild" tool, but you can get it
horribly wrong if you simply let everyone indulge themselves with it...<G>

If you are going to be doing joint development in PowerCOBOL it would be
wise to lay down some agreed conventions (standards?) before you start (for
example, centralising event code, as noted above). There are many ways of
doing things in PowerCOBOL and, while most of them are excellent, it is
important to be consistent in a joint development environment.

PowerCobol (PC) project files contain everything and are not (really
searchable). You can GREP a PC Project file (PPJ) to find out whether a
string exists inside it, then start up the project and use the project's
search routine to find a bit of code.

You could also store your source in an ordinary text file and include the
name of the external file in the PPJ where it will be fetched at compile
time (think COPY), thereby making the file searchable by external routines.

We use GEM (a sort-of Source Safe) provided by FJ. Because PPJ formats are
unknown to GEM(!), GEM treats PPJ files as miscellaneous binary files - no
versioning, no roll-back, no deltas, etc.

In sum, PC stores its source files in PPJ format which is unknown to anybody
and cannot be manipulated by anything other than PC.
