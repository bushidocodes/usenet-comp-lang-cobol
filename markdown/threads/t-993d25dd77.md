```
We have numerous subroutines that are "CALLed" from numerous sets of
reusable code (and the reusable code is automatically "included" into
various programs by Panvalet).  While converting from OS/VS COBOL 2.4

subroutines needed separate versions for the two environments (I'm not
sure why, may LE).  This caused us to create and maintain two versions
of the each reusable code set that issues these CALLs.

Can you think of a technique the reusable code could employ to
recognize which version of COBOL it is operating in, so a single
version of the reusable code could then use an IF structure to CALL
the appropriate version of a subroutine?

Rick Jones
```

```
First and foremost, you really need to find out WHY someone thinks that you
need separate versions of these programs.  The only time that this MIGHT be
true is if you are doing something "illegal" in the OS/VS COBOL version -
that no longer works with the "current" supported environments.  (All OS/VS
COBOL and VS COBOL II run-times are no longer supported by IBM.  OS/VS COBOL
and VS COBOL II compiled programs using ONLY the LE run-time are still
supported)

Having said that, you ask about which environment you are running it.  You
do realize (I hope) that you are NOT asking which COBOL product is being
used, but which RUN-TIME (and/or link-edit) library is being used.  With
this information, it is "relatively" easy to determine if LE is or is not
available during the execution of a program.  All you need to do is make a
DYNAMIC call to *any* LE run-time module.  For example,

 01  Test-Program  Pic X(7)  Value "CEEGMT"
 01  Test-Parms.
    05  LILIAN                  PIC S9(9) BINARY.
    05  SECS                    COMP-2.
    05  FC                         Pic X(12).

    ...

    CALL Test-Program USING LILIAN , SECS , FC
           On Exception
                 **** > do your pre-LE run-time logic
           Not On Exception
                 **** >  do your LE run-time logic
    End-Call

NOTE WELL:  As this is a "dynamic" CALL, you must (of course) have compiled
your OS/VS COBOL (if that is the compiler you used) program as RES, not
NORES - but as the entire concept of "NORES" goes away with the currently
supported IBM COBOL compilers, I hope you are already at - or moving to
this.
```

```
So, if I am to return to a single set of reusable code, what I actually 
need is a COBOL code snippet that can recognize whether the current 
program is CICS or batch.

I thank the people that sent emails with solutions for my original
question (is the COBOL code is executing within OS COBOL or MVS COBOL).

Any suggestions about recognizing CICS vs. batch will be appreciated.
```

```
I have been working on 31-bit conversions quite a bit recently.  My
shop is preparing to upgrade from OS/390 release 2.8 to 2.10 and then
to Z/OS next year.  One of the issues the sysprogs have is that DSA
below the 16-meg line will be shrinking as more space is consumed for
operating system components.  Therefore, some large COBOL programs
that are RMODE(24) will no longer execute under z/OS.  The limit seems
to be about 8 megabytes, and these COBOL programs generally exceed it
by having large tables defined in working-storage.

Another issue the technical services gurus have identified is that old
AMODE(24),RMODE(24) assembler subprograms are not safe to run with LE
(Language Environment) because the savearea chain does not support
31-bit addresses.  I'm no assembler expert, but the thought occurs to
me that this is not a problem unless an LE-enabled COBOL program calls
assembler, which in turn calls a lower level LE-COBOL program.  Twenty
year old assembler programs are being modified to change their
initialization prolog, even when they will remain RMODE(24) due to
dependencies on OS services or control blocks.

Some of the problems I've run into with COBOL 31-bit conversions:

1.  Finding OS/VS COBOL programs that somehow weren't converted during
our Y2K project (there was a mandate that all programs would be
recompiled in VS COBOL II, and we are currently converting from COBOL
for MVS & VM version 1.2.2 to COBOL for OS/390 & VM version 2.4).  By
some lucky chance all COBOL II programs were compiled with RES &
NOCMPR2.  The OS/VS COBOL programs were usually compiled NORES.
2.  Assembler subprograms with missing source (not many of those,
thank god).
3.  Assembler subprograms that need their prolog code changed (some of
them owned by sysprogs).
4.  Assembler subprograms that must remain RMODE(24) due to ACB/DCB
restrictions.
5.  Programs last compiled with VS COBOL II and built as
AMODE(ANY),RMODE(24).  All COBOL programs must be rebuilt as
AMODE(31),RMODE(ANY).  Any exceptions must be documented and
justified.  

Some assembler programs are very tricky to convert to AMODE(31), at
least for me.       

If a batch program has only COBOL components it is pretty easy to
recompile it as AMODE(31),RMODE(ANY), and there is no reason NOT to
fix them.  Pretty soon running RMODE(24) applications will be a "bad
thing", like running 1401 emulation.
```

```
Well, as long as the option All31(OFF) is set, then MVS will automatically
'translate' 31bit to 24bit addresses in calls (provided the call is
DYNAMIC).

Most of those assembler programs can still be converted:

do one of the two:

put the actual I/O below the line and call that from the converted
31bit assembler

have the assembler pgm entered as 31bit, ask for enough below the line
memory to move the ACB/DCB to there AND to copy the code to
execute to below the line.  Then do a BRANCH and CHANGE Address
Mode to go to the 24bit, do the I/O, do the complementary Branch and
change address mode.
```

```
OK - if the issue is being able to call the SAME *assembler" program from
both AMODE(31) and AMODE(24) programs, life is probably even easier (again,
I think you can "forget" CICS vs non-CICS).

If all your programs do a DYNAMIC call (CALL identifier - not CALL
"literal"), then the COBOL run-time will happily do AMODE switching FOR you.

The only "real" issue might be if you are passing parameters to this
sub-program (which in SOME cases would want data "below" the line).  To make
certain that this ALWAYS happens, have every program that passes data to
this subroutine compiled with the DATA(24) option, and everything should
work "like a charm".

Having said that, there ARE real issues about have the same (COBOL at least)
subroutine available in batch and CICS.  These are discussed in the COBOL
programming guide - as I recall.  You might want to start your search at:

 http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igypg205/3.1.1.4?
```
