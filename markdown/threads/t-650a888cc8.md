```
No experience with the 9000, but I did work in a couple shops on the
3000's.  Their COBOL had a few quirks relating to the machine's word
size (I recall lots of agony over an even number of bytes).  

There was no implementation of the screen section and instead you made
calls to HP's "View/3000" subsystem, which itself had some nice
features, but still it was outside of COBOL and probably not easily
transported to other platforms.  Screens were built and invoked via
CALLS.  The folks from HP who formed Tandem carried this same madness
with them in the form of an entire other language called SCOBOL or
screen cobol.  

Also, I recall HP's ISAM utility also had a few quirks.  I think they
called it RSAM, or was it KSAM?  But that's not really unusual, as every
implementation has its oddities.  I was however impressed with the
processing power of that system.

As far as conversion to IBM, I can't help you there.
```

```
Hi, steve, I support all three, so here are my thoughts...

HP3000s run the MPE operating system.  'final' release is 7.0
HP recently anounced their support for HP3000 class minis will
terminate in 2006.  The 3000's database is turbo IMAGE -
a hierarchical databse.  ODBC connectivity to IMAGE is available
through ALLBASE.   

HP9000 run HPUX having unix file structure.  They typically use an
RDBMS, like Sybase with SQL.

The 3000's file structure is 3-tiered where unix is the more familiar
root tree structure.

The HP3000 is more of a mini computer than a server.  So its more like
an IBM mainframe in some respects.  Its job handling facilities are
solid.  Trying to use IMAGE in the client server world will demand
some thought.  For example, using ODBC SQL against a hierarchical
database structure requires understanding turbo IMAGE rather than just
relational databases. 

Your biggest challenge is utilities.  The 3000 has a set of third
party tools and editors very different from the 9000.
```

```
Whoo boy - don't bet on that. A closer fit to the HP3000 is probably
an AS/400, which has the added advantage of the toolsets and the file
system being more like a 3000 than any UNIX filesystem.  (And I love
UNIX, so don't get me wrong here...)

The COBOL will also be a closer match, though screen handling is
somewhat different as well; much more like a mainframe than the Micro
Focus based 3000 view system.

Database issues, well under UNIX, you are going to need to have
someone who really knows any RDBMS you use to maintain, tune, and
configure it. On the AS/400, this isn't an issue since DB/2 is a
built-in integral part of the system.

Finally, the development tools available under Linux, while very good,
are about as different as they can be from MPE/ix.  SDE and PDM under
an AS/400 will be a lot more comfortable.

Just my $0.02.
```
