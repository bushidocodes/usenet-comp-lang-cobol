By now, Moose, you have had a lot of answers. The closest one had a
reference to water cooled computers.  Even that reference was just a jab and
missed the mark.

1. Early computers did not come with an operating system as we know of
today.  For almost a decade, the programmer was responsible for keeping
track of the number of tapes in a file, the beginning and ending of a file,
block counts, and anything else needed. We in the Univac I and II crowd (who
had water cooled computers) had to write all of our I/O, including an
interface to the "run to run" locator on the drive that held the program
library.

When you started an application, the operator (and often the programmer
responsible for the application) would mount a program library tape, and a
tape carrying variables shared by the chain of runs that were in that
library.  The first program run was one that made a "working tape library"
for production. An example of the function of this run is that each "word"
of the library tape was
inspected for the value "TODAY_______" and replaced with a value
representing the date to be used for the production run.

2. Tape jockeys would find the grand father tape(s) for the first program in
a chain, and mount it on the drive the operat;ing instructions specified,
and then mount blanks everywhere else needed. Note: other input files might
be included.

3. Each program was responsible for checking a label record on the first
block on the tape to determine if it was the correct file, and if not rewind
the tape, and notify the operator to mount the correct tape.

4. Each program that created output tapes was responsible for labeling each
output correctly, and that included labels for the beginning and ending of
each tape. The ending label carried unique information that told the next
program if this was a final tape, or an intermediate tape.

5. When we got smart enough to provide recovery points, the
program would also write a memory image on the end of the tape so that the
program could be restarted anywhere in the process.

While I may have missed something, and my memory of it may be somewhat
skewed, that's the sum and substance of standard tape labels.  They became
the audit trail of a highly manual computer operation.  Twenty years later,
many sites had operating systems that did it all including staging tapes and
disk files.  COBOL reflects it's roots in these two statements.

I hope that computer operations still has some way of insuring their files
are what they think they are and ways to check this before removing a file
from a system for use elsewhere.

Anyway, reminds me of an interesting program we have in our shop.  Basically, it's
pretty simple.  The program reads an input tape which contains some records for the
month, then reads in today records and writes both to today's output tape.  At the
end of the month a report is produced based on the final tape, which contains an
entire months worth of records.  (At the beginning of the month the input tape is
ignored.)

Problem...On Tuesdays after a Monday holiday the operators would often load the Monday
tape as the input, rather than the correct Friday tape.  Since Monday was a holiday it
actually contained data from the Monday eight days ago.

So one of our (old time, 20+ years programming) programmers came up with a solution.
Each of the five (Monday thru Friday) tapes was given a label which contained the day of
the week.  Both the input and the output tape files were given *two* file control entries 
and *two* file section entries, one with LABEL RECORDS ARE STANDARD and one with LABEL 
RECORDS ARE OMITTED.  He would open the "label omitted" files as input and read in the
first records, which just happens to *be* the label.  He would check today's day of
the week, check for a holiday the previous day, and make sure the correct day of the
week tapes were loaded, both for the input and the output dates.  If they were wrong
he'd give an error and abort.  If they were correct he'd close both files and then
open the "standard labels" tapes as input and output respectively and proceed with
processing.

On the same note, since this terminology seems to have gone away in COBOL II and later
(or has it?), I wonder how I could accomplish this same thing if I convert this program.

The LABEL RECORDS clause in COBOL is obsolete not because labels have
disappeared but because they are managed via JCL (Job Control Language)
and their use is so ubiquitous that allowing their presence or absence
to be specified at the program level is no longer reasonable.  Standard
tape labels are essential both to automated tape management systems and
to security management systems designed to manage and protect data
residing on tape media.

At least in the case of IBM operating systems, all DASD datasets have
"labels" which define the external dataset name and characteristics of
the dataset.  Also all shops of any reasonable size use standard tape
labels (with the occasional exception of data interchange with some
other installations with conflicting standards).  Tape labels involve
volume header label records (uniquely identifies the physical volume and
hopefully agrees with any external label on the tape), dataset header
records and dataset trailer records, which contain the dataset name and
various dataset characteristics. There are also standard label
conventions for indicating multi-file tape volumes and multi-volume tape
files.  The JCL directing program execution establishes the relationship
between program "file" names, and an external dataset name and media
used to store the dataset.  Some kind of tape management system (TMS) is
invariably used to track which tapes are scratch vs those with current
datasets.  For output tapes, the TMS guarantees that the tape mounted
for output is really a scratch tape.  For input tapes, either the TMS,
the operating system, or both determine the physical tape(s) that should
contain the requested dataset, and the label records on the tape are
used to verify that the correct tape was actually mounted by the
operator and that the tape actually contains the requested dataset name.

If you really wanted to handle a weekday master-in/master-out job using
unique labels (i.e, unique dataset names) for each day of the week, you
would use unique JCL for each day of the week (with different dataset
names) to drive the program execution, with a special version to use
after a holiday.  If what you really want is to always use the previous
run's output as input and only run the job on nonholiday weekdays, then
much simpler solutions are available.  Most TMS's will allow you have
several active datasets with the same name, and there are techniques
that can be used to have an input request be satisfied by the most
recent version while an output request for the same name goes to an
available scratch tape.  In the MVS arena, you also have the choice of
using a Generation Dataset Group (GDG), where the JCL contains a fixed
dataset name, but the real dataset name is forced unique by the
operating system adding a unique sequentially-assigned part to the end
of the name (called a "goo-voo" number because they are values like
G0001V00, G0002V00, G0003V00, etc.).  GDG's have the advantage that JCL
need only specify "relative" generation numbers (like -1, 0 , +1), and
multiple generations other than the most current and next (new) can also
be easily referenced in JCL.
