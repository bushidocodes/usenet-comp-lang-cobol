Most COBOL indexed file implementations now support READ PREVIOUS.  However
this was not always the case and the standard does not include READ
PREVIOUS.

You most likely will find that the 9s complement date field(s) are used in
indexes (as you say, the product uses this to look up certain records).  By
this means one can read records in reverse chronological order using
standard COBOL.  Your lookup probably does a START followed by one or more
READ NEXT.

Smells like homework to me... but what the heck:  Inverting the sequence
for sequential reads. Consider the keys:

20000101
20000201
20000301
...
20001231

As written a series of sequential reads on that key will you ascending
date sequence.  If you want to reverse the sequence:

99999999 - 20000101 = 79999898
99999999 - 20000201 = 79999798
99999999 - 20000301 = 79999698
...
99999999 - 20001231 = 79998768

While it is impossible to know why anybody does anything without reading the
code, the main reason for 9's compliment is to allow negative numbers
without using a sign.  That could be so that things could be read in reverse
order, or it could be for some obscure arithmetic reason. Certainly BCD math
using 9's complement works as well as converting to binary one's compliment.
I have never used it in Cobol, but it is/was common in assembler.

In billing telephone calls, the last field in the rate table file key
was a 9's complement of the date that rate went into effect.  Once the
date the call was made was converted into 9's complement, READ ... NEXT
could find the correct rate.  This allowed new rates to be entered
before their effective date.
