No.  That will not work.  A sequential read requires that records use <CR
LF> as record terminators.  A relative file description may allow access,
but you will get a great deal of garbage along with the data.

You can always read a data file by defining it as a one byte record, and you
can do the same with the data portion of and Isam file, but you are then
going to have to filter out the deleted records, pointers, delimiters, etc.
that are used internally.  To do that, you need a fair idea of the way the
Isam data is physically stored.

The rebuild facility is by far the easier solution.

That depends entirely on the compiler.  Yes, a line sequential file is the
standard <CR LF> format.  However, assuming that an ordinary sequential file
and an Isam data file are compatible is a huge assumption, and will be wrong
99% of the time.

Many Isams are linked lists.  As well as possible internal pointer data,
there is also deleted data to account for.  There may be index blocks
intermingled with the data blocks.  There may, or may not be record
delimiters. The actual data blocks may be variable lengths.  About the only
way that you can be assured of reading it sequentially and accurately is one
byte at a time, analysing the data as you go.

If you happen to be on a specific platform, and you happen to have a
specific Isam from a specific vendor, and you specifically know the
internals of that Isam method, and you know the sequential definition for
the Cobol you are trying to do the recovery with matches, then sure.  Write
a program and read it sequentially.  In a situation where I knew no
specifics, and I did know there was an Isam recovery program, I still think
the using the recovery a better solution.

BTW, as another person has noted, the index need not match the file for a
rebuild.  You can take an old index for the file, from a previous backup,
and the MF rebuild will replace it with a new one.  The MF
index is only needed  to get the structural details for the file.

MF does format the data portion of a file sequentially, and I have done a
recovery using that method.  However, it is not a trivial exercise. There
are places where the data is realigned to physical blocks that are
exceptions.  You also have to check for deleted records.  If the file is
old, there could be hundreds of garbage records in the middle of it. It only
works easily immediately after a full rebuild, which sort of defeats the
purpose.
