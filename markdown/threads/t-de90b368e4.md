In Cobol you would, of course, use the Call statement.  When you code:

CALL PRGMA USING PARAM-1, PARAM-2.

The address of PARAM-1 and PARAM-2 are passed to the called (PRGMA)
program.  In the mainframe world, standard IBM register convention is
used.  That is, the address of the first executable statement in PRGMA
is put into register 15; the address of the next statement after the
CALL is put into register 14.  The address of an area that contains
the address(es) of the paramters is placed in register 1 and the
address of an 18 fullword save area is placed in register 13.  

You can place a return code in register 15 which can then be checked
by the Cobol program.  I have no idea how you do this in C.

There's lots of new features in COBOL that make the above just the bare bones. 

In the latest OS/390 COBOL compilers you can :

* pass arguments BY REFERENCE, BY CONTENT, and BY VALUE. You can also receive
them this way. This was done to facilitate interlanguage communication between
COBOL and C

* set a subroutine to return a value, like a C function

* set a COBOL main to receive a value when invoking a C function (or even a
COBOL subroutine set up properly)

* specify an argument as OMITTED, and test if a parameter has been omitted in a
subroutine

All of these influence the way your COBOL subroutine access the parameters and
returns a value.

Here's a little summary of how COBOL passes parameters in OS/390:

R1 points to a parameter area (the same area is used for all CALLs in the
program and does not clear this out between CALLs)

The parameter area contains
* a(value to be returned) - if RETURNING was specified on the CALL
* a mix of
    - addresses of items passed by reference or by content
    - fullword zeros for omitted arguments
    - copy of any data passed BY VALUE (only certain data types are eligible)
* the last word has its leftmost bit set on (the end_of_list flag) unless the
last argument is being passed BY VALUE

Then, of course, there's the possibilities of simply sharing common storage
using dynamically allocated memory.

Marjeta,

COBOL generally passes the address of arguments, which is equivalent
to passing a pointer in C.  You can (to a degree) control how
arguments are passed by using modifiers in your call statement, like
this:

CALL 'subroutine' USING 
          BY REFERENCE A
          BY CONTENT B
          BY VALUE 2 SIZE 4
          GIVING returned-value.

Where the "BY REFERENCE' modifier passes a pointer, BY CONTENT passes
the value of the data found in "B", and BY VALUE creates a temporary
variable and then passes the value "2" as a 4-byte binary value.

The arguments are generally passed on the stack in the order
determined by the requirements of the called subroutine.    Returned
values are obtained using GIVING.  

For example, CA-Realia has you add a prefix to the actual subroutine
name that specifies the exact format of the call, e.g,, 

CALL 'S_windowsapi' USING ...

Where the "S_" prefix specifies the compiler should use the Windows
system calling convention.  

Other compilers use different ways of accomplishing this end.

Most COBOL compilers have a section in the documentation that
discusses how to call other languages  

Expect to struggle with this a bit until you get it figured out...

Kevin

Hi, Bill,

Darn, I hate it when I'm wrong...  You're correct for the general
COBOL case.  I've been working with Windows APIs too much for the last
4 years. 

The original post wondered how to call "C" APIs from COBOL.

In CA-Realia COBOL, if you specify the Windows system calling
conventions, BY CONTENT passes the value on the stack.  So, if you
want to call the WinExec Windows API:

UINT WinExec(
  LPCSTR lpCmdLine,  // command line
  UINT uCmdShow      // window style
);

You note that the lpCmdLine is a pointer, and uCmdShow is an unsigned
integer.

The CA-Realia call is:

CALL 'S_WinExec'USING 
          BY REFERENCE COMMAND-LINE
          BY CONTENT SHOW-COMMAND
GIVING RESULT-CODE.

In this case, the BY CONTENT is indeed passing the value, not a
pointer to a temporary data item.

Which only goes to show that interfacing to software written in other
languages requires fairly intimate knowledge of the target's calling
conventions as well as how the COBOL vendor implemented their
third-party language interface.

Kevin

Well, Kevin you were pretty close.  The prior posters just wasted Marjeta's
time!

I can tell you that on IBM (MVS / VM / OS-390), Wang, Honeywell, and IBM
PC's ('C')
the default method is that for CALL 'xxx' using var1, var2, var3, what it
does is:

the compiler takes 4 address sized words contigously in memory and
does this:
word0                       word1                  word2
word3
address_word1    addrress_var1    address_var2        address_var3

and passes one address:  word0.

Now, some systems add an address of all 00000 at the end (word4),  (32bit
addressing systems)
some systems (24/31 bit addressing systems) set Bit 0 of word3 'on'. (Since
an address can't use
32 bits on a 24/31 bit addressing system, no need to waste a word!)


So generally, the subroutine takes the pointer in R1 / TCB#1, (or parm), and
dereferences that address
, then 'bumps' the address in R1 by (usually 4) whatever is an address sized
int, and repeats
until bit 0 is on, or word = all_zeroes (NULL = invalid address)

Actually, Bill, (and I could be wrong, but a careful reading of the language
ref leads me to think not)
I believe the method I described is the method ALWAYS used.
The only difference between By Reference / By Value / By Content  (And for
Ada
INOUT / IN / OUT) is  for non BY Reference, a COPY of the variable is made
and the
address of the COPY is used in the LIST, since in most Languages, a 'By
Value' or
'By Content' parm can be altered in the SubRoutine all you want, On Return
Linkage,
it is discarded.


On almost all Hardware, indirect addressing AKA Base+Displacement  IS
fastest.
All hardware has memory addressing and memory value moving restrictions,
and to overcome  these, the OS has limits and guides (see IBM's 4K paging
and its
impact on non-linear coding.).

In a world of Unlimited Stack size, the values could be moved back and
forth, but
it is and will always be more efficient pass the address of the data (even
when it is a
'BY VALUE' or 'BY CONTENT' copy), COPY the data to local storage, then work
on
the data and copy it back, than to try to store hundreds to thousands of
bytes on the Stack,
which after all, is only big enough to hold one complete set of registers.
