```
Yes we have looked at AcuODBC, but it only allows ODBC-compliant applications
(such as Access and Excel) to access vision files.  In other words, it's the
wrong direction.  We want to move away from vision index files.

The performance problems with Acu4GL were not network based.  We were running
the 'client' on the same unix server that the database was installed on. (our
application is purely host-based)

Basically, there seems to be something fundamentally wrong with the way Acu4GL
translates START and READ NEXT/PREVIOUS statements.  Most things we do in Cobol
involve starting an index, then reading until you find a record outside your
search scope.  Acu4GL would do this, but with several SELECT statements that
got more and more general until it was selecting the rest of the table just so
that we could read one record outside the original search scope!

One example we had was chronic.  One program started up and before doing
anything else, it attempted to read one record for the current year to check if
there were any records.  In Cobol this was just a START and one READ NEXT.
However, Acu4GL would select ALL the records for the year just so we could read
one!  For a full year's worth of data it took about 3 minutes for it to
complete that one SELECT!

But this has got to be because of the way that Acu4GL does it.  Our new GUI
development language can perform a similar 'cursor open' and 'cursor fetch' in
about 30ms!  And that's on exactly the same database on exactly the same unix
server.

Acu's own support staff told us: "It's a square peg for a round hole"!

Admittedly, we did test Acu4GL over three years ago, but there's been no word
from Acu about any drastic changes to Acu4GL that have increased performance.
```
