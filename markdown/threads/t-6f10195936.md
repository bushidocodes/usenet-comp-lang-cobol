Aloha all,

I was intrigued by Thane's inbreeding co-efficient problem, so I ended up
writing working routines in MicroFocus COBOL. I used a recursion algorithm to
solve the problem; Thane suggested posting the code in this NG for your
perusal. It's about 9K, zipped up, as an uuencoded part of this message.
I'll briefly describe the processing here and let you look at the code for
details. If you can't decompress the file, e-mail me and I'll send an
uncompressed version or a self-extracting one.

The basic design is straightforward: build a file containing all of the
ancestors and the path(s) to reach them for the father's side of the family
tree. This is done by the SIRETREE routine. Then a second subroutine,
DAMETREE, tests each ancestor on the mother's side for commonality - by
reading the file created from SIRETREE. If a no "hit" is found, the ancestor
is not common; if it is found, DAMETREE tests for "prior" common ancestors and
calculates the co-efficient for each unique path.

The files have extension .CBL and are:

CATPATH copy member for table of parms passed between subroutines
CATDAP copy record layout for a intermediate file
CATDB copy record layout for individual dog file
CATCONST constants and literals

CATIO simple I/O routine for CATDB file
CATCOEFF this is the main subroutine that is called to calc the co-eff
SIRETREE subroutine to build CATDAP records from father's family tree
DAMETREE subroutine to traverse mother's tree and calc co-eff
CATTEST This is a simple test program to build the CATDB file and test
CATCOEFF.

For those who don't know MF and/or recursion, I'll walk through the
"strip-down" version of SIRETREE:

WORKING-STORAGE SECTION.
..
* as MF implements it, recursive calls to routine have the same effect on
* W-S and FILE SECTION fields as regular calls - if you change a field, the *
value remains there for all calls. In effect, these are GLOBAL variables.

LOCAL-STORAGE SECTION.
* LOCAL-STORAGE are "local" variables. Each time SIRETREE is called *
recursively, a new copy of these fields is created. When a GOBACK
* is executed, the values that were present before the CALL are restored.

01 LCL-TABLE.
05 LCL-GEN PIC S9(4) BINARY.
05 LCL-PATH.
10 LCL-ANCESTOR OCCURS 20 TIMES PIC X(10).

01 LCL-CAT-RECORD.

... this is the record for the ancestor we are currently working
on.

LINKAGE SECTION.
01 LNK-TABLE.
... same layout as LCL-TABLE ...
01 LNK-CAT-ID

PROCEDURE DIVISION USING LNK-CAT-ID, LNK-TABLE.
* keep a copy of the current position in the tree.
MOVE LNK-TABLE TO LCL-TABLE

* add the ancestor to the path; on the initial call this is the father ID
ADD 1 TO LCL-GEN
MOVE LNK-CAT-ID TO LCL-ANCESTOR (LCL-GEN)

* use CATIO to read the ancestor's record for father and mother data
CALL "CATIO" USING LNK-CAT-ID
LCL-CAT-RECORD

* use a recursive call to process first the father and then the mother of
* LNK-CAT-ID
IF LCL-CATS-FATHER NOT = SPACES
CALL 'SIRETREE' USING LCL-CATS-FATHER LCL-TABLE
END-IF
IF LCL-CATS-MOTHER NOT = SPACES
CALL 'SIRETREE' USING LCL-CATS-MOTHER LCL-TABLE
END-IF
GOBACK.

When the CALL "SIRETREE" is executed the program "begins" again with the MOVE
after the PROCEDURE DIVISION. When the GOBACK is executed, the logic returns
to the appropiate END-IF following the CALL that was executed. This code will
travel up and down the family tree until every ancestor for the initial
LNK-CAT-ID has been processed. NOTE: this code will also loop forever as
written if cat A is incorrectly listed as a descendant of itself.

DAMETREE has almost identical logic. The recursion is simple once you "get
your mind around it". I spent the most time in the DAMETREE processing on the
"COMMON-ANCESTOR" logic. The code that discards "duplicate" paths, e.g.
B-E-H-E-C - the paths that Thane indicates in his examples as "invalid".

Enjoy - I know I did!
