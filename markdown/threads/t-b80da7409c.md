The size might depend on the platform / compiler.

For IBM MVS COBOL compilers (COBOL II and COBOL for MVS and VM), the field
you defined will reserve four bytes. For COMP data, if the PIC includes
1-4 '9's the data item takes two bytes; for 5-9 '9's the item takes four
bytes; for 10-18 '9's the item takes eight bytes. Also note that because
the field is an 01 level the compiler will ensure it begins on a
doubleword boundary (this would also be true for a level 77).

If you specify one to four digits, inclusive, in the PICTURE clause, you will
get a two-byte fixed-point binary field. Example: PIC S9(4) COMP.

If you specify five to nine digits, inclusive, in the PICTURE clause, you will
get a four-byte fixed-point binary field. Example: PIC S9(9) COMP.

If you specify ten to eighteen digits, inclusive, in the PICTURE clause, you
will get an eight-byte fixed-point binary field. Example: PIC S9(18) COMP.

An eight-byte fixed-point binary field can obviously hold more than eighteen
digits, but eighteen digits is the maximum size in the CoBOL standard for a
PICTURE clause. This is true of all USAGEs that have PICTURE clauses. For
example, the underlying representation of COMP-3 can have up to 31 digits, but
you can't use more than 18 of them from CoBOL.

PIC 9(05) USAGE IS COMP has five digits in the PICTURE clause so it is
four-byte unsigned fixed-point binary.

There are two common ways of using COMP fields. One way is to consciously use
fixed-point binary. The clauses PIC S9(4) COMP and PIC S9(9) COMP are pretty
widely-used conventions for showing that you really mean fixed-point binary.
For example, under this convention you would use PIC S9(4) COMP for values
-32768 to +32767, despite the fact that there are only four digits specified in
its PICTURE clause.

The other common way is to use COMP fields like any other field. In this case,
PIC S9(4) means just that. The compiler will generate extra code every time you
use the variable to insure that it never exceeds the specified four digits. You
generally have to request this option at compile time, for example the TRUNC
option on IBM mainframe CoBOL.

The use of a PICTURE clause other than S9(4) or S9(9) in a COMP field is either
a warning from the author to pay attention or an indication that the author
hasn't a clue.

Based on the name of the variable in your example, I would hazard a guess that
the date is of the form YYDDD, and the author was merely trying to document that
fact, without realizing that when you leave out the sign from a PICTURE clause
you are asking the compiler to generate extra code to strip the sign every time
you use the variable.

The best way to declare this variable is PIC S9(9) COMP, to tell the compiler
not to worry, and a comment, to tell the reader what date format is being used.

No, sorry, S9(4) COMP is two-byte fixed-point binary, and S9(5) COMP is
four-byte fixed-point binary. And it won't be aligned on either boundary unless
you explicitly code a SYNC clause or count the bytes above it.

Some people code S9(5) COMP because they know it takes five digits to *print*
-32768 to +32767 (or 0 to 65535). However, the standard was written to make
*assignment* safe. Four digits (-9999 to +9999) is the most that you can safely
fit into two bytes. It is syntactically valid to MOVE +99999 to an S9(5) COMP,
and if it were only two bytes long, the assignment wouldn't work. Try it on
your compiler and see.

Strictly speaking, the CoBOL language does not recognize the part of the
halfword range that lies below -9999 or above +9999, although we all know how to
fool the compiler into using it. This is similar to the fact that in S9(18)
COMP-3 there are physically nineteen digits, but the CoBOL language does not
recognize the nineteenth digit, and you have to fool the compiler into using it.

This is not a problem, nor is it a machine dependency, it is an explanation.

USAGE IS COMPUTATIONAL is CoBOL for fixed-point binary. The way it works is,
S9(1) COMP through S9(4) COMP take two bytes; S9(5) COMP through S9(9) COMP take
four bytes; and S9(10) COMP through S9(18) COMP take eight bytes.

Most people just memorize this and leave it at that, but sometimes people wonder
why S9(5) COMP is four bytes, not two bytes. So I tried to explain.

+99999 is allowed by the picture S9(5) COMP, but it won't fit in two bytes.
Therefore, CoBOL uses four bytes. If my explanation is not clear, I apologize,
but S9(5) COMP still takes four bytes regardless of whether my explanation is
clear or not.


›› Strictly speaking, the CoBOL language does not recognize the part of the
›› halfword range that lies below -9999 or above +9999, although we all know
…[10 more quoted lines elided]…
› worked.  Can you please clear up this point.  (machine dependency).

OK, let's review packed decimal:

PIC 9(1) COMP-3 VALUE 1 gives one byte: X'1F'
PIC S9(1) COMP-3 VALUE +1 gives one byte: X'1C'
PIC S9(1) COMP-3 VALUE -1 gives one byte: X'1D'


Easy enough, let's try some more:

PIC 9(3) COMP-3 VALUE 123 gives two bytes: X'123F'
PIC S9(3) COMP-3 VALUE +123 gives two bytes: X'123C'
PIC S9(3) COMP-3 VALUE -123 gives two bytes: X'123D'


But what happens when there is a half-byte left over?

PIC 9(2) COMP-3 VALUE 12 gives two bytes: X'012F'
PIC S9(2) COMP-3 VALUE +12 gives two bytes: X'012C'
PIC S9(2) COMP-3 VALUE -12 gives two bytes: X'012D'


Notice that each of these values has had a leading zero added, so that the
number will fit into an integral number of bytes (there is no such thing as a
half-byte filler). All COMP-3 fields physically have an *odd* number of digits,
even if the PICTURE clause specifies an even number of digits. The hidden digit
is not a filler; it is real, and it participates in arithmetic operations. PIC
S9(2) COMP-3 can physically take on values up to +999, but theoretically CoBOL
should not "see" digits not specified in the PICTURE clause (although you can
fool it).

If you specify S9(2) COMP-3, you physically get S9(3) COMP-3, and if you specify
S9(18) COMP-3, you physically get S9(19) COMP-3. The difference is, you can
specify S9(3) COMP-3 to legally access the extra digit, but you can not specify
S9(19) COMP-3 to legally access the extra digit because of the eighteen-digit
limitation.

I think the 'unsigned packed' to which the poster referred is what I have
heard called 'stripped packed'. In the stripped packed format, a regular
packed decimal (odd numbber of digits plus sign nibble) is shifted four
bits right, which drops the sign, and the resultingleftmost byte is also
chopped off. This leaves the rightmost n-1 digits in one fewer bytes.
E.g. +01234 packed as X'01234C', or X'01234F', becomes X'001234'. Then the
leftmost X'00' byte is dropped, giving X'1234' in two bytes.

I don't know of a system on which this is native. It was not on IBM, but
we did get tapes from other machines with this format, and had to write
routines to convert them (using a redefines, and a divide by 10). They
were used alot in banking tapes, to carry the date field in YYMMDD. I
would suppose someone thought it was a way to save some space on archives,
or some such thing, since our banking transactions had alot of dates in
them, and we had to archive alot of transactions.
