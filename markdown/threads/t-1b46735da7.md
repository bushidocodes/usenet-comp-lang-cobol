Usually Maven is used for Java builds. The key thing is the "convention 
over configuration" aspect. That is, as long as your project follows a 
certain layout, you can use a very simple default pom.xml for building, 
reporting and documentation. E.g. all your Java code has to be in the 
"src/main/java" directory, all tests must be in "src/test/java" etc.

This simplifys the build dramatically. Whereas every ANT build script is 
sligthly different, a pom.xml may be reused for a lot of builds.

The aspect I do not like is (and that is the reason for my posting), 
that Maven includes every library, which _may_ be nescessary. If. e.g. 
you need the org.apache.commons.lang.StringUtils class, you end up with 
an application including almost every org.apache.commons-library 
(commons.net, commons.logging, commons.io, etc). In the end, your 
application is twice as big as nescessary, including libraries you never 
heard of and you have no idea, if you really need them or not.

Nevertheless Maven is extremely powerfull and, as long as you project 
follows Maven's layout, easy to use. I know that IBM has a ANT/z for 
COBOL and PL/I builds so maybe a Maven/z is not far away.
