Hi there,

We have run into a little problem with COBOL conditional expression evaluation.
A developer while writing a program made a error and typed a minus sign instead
of an equal sign in an IF statement. The funny thing is that the IF statement
gave a result. Not the correct result, and no syntax error was generated. Here
is what the developer entered:

77 a pic 9(6).
77 b pic 9(6).

PROCEDURE DIVISION.
MOVE 1 TO a.
MOVE 3 TO b.
IF (a = 1) AND (b - 2) THEN
* What the developer wanted to enter is:
* if (a = 1) and (b = 2) THEN....
DISPLAY "yes"
ELSE
DISPLAY "no".
STOP RUN.

We took a look at the COBOL (both 74 and 85) manual and from we can interpret it
looks like the parens should evaluate the conditional expression first giving a
boolean result. Then evaluate the second conditional expression giving a boolean
result and then ANDing the two booleans. What it looks like is happening is that
the IF statement is getting extended to:
IF (a = 1) AND a = (b - 2) THEN ...

Is this possible? We thought the COBOL short cut for IF's only worked when there
are no parens. Could this be a bug? Can you all try it out on your compilers
please and post the results. Can someone explain what is happening?

The answer that our program gives after compiling with our compiler is
DISPLAY "yes".

Any help would be appreciated.

Thanks

Contraste Europe S.A.

› What it looks like is happening is that

›    IF (a = 1) AND a = (b - 2) THEN ...

› Is this possible? 

Yes, it must! Because (b - 2) is an expression and expressions are valid
items in relation conditions! The rule for "abbreviated combined relation
conditions" says that the object of a succeeding relation condition can
be omitted.

› We thought the COBOL short cut for IF's only worked when there
› are no parens. Could this be a bug?

What about the parentheses? (a = 1) is a simple relation condition and
therefore the parentheses take no effect. The same with (b - 2): It's a
simple expression with no hierarchical dependencies to the preceding
condition.

Therefore, I think the "bug" is ANSI-compatible...
