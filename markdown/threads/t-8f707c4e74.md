You should be able to run sdb/gdb and Animator concurrently...

Compile the COBOL programs to animatable int-code using cob -a.

Compile the C programs to debuggable objects using cc -g.

Build a new RTS, linking in the C objects. You may need to use the
cob -g flag to prevent the linker stripping debugging information
out of the final executable. You also need to add "-d "
to the command line for every COBOL entry-point called from the C
routines. The -U flag may work instead of many -d flags, but I'm
not sure that it is supported on all systems.

Now set COBSW=+A in your environment and run the new RTS using gdb.
The exact behaviour will depend upon the location of your main
program entry-point (ie. whether it is C or COBOL), but you should
find that Animator is started during execution of the int-code
programs compiled with the -a flag.

There should be a demo of this in the $COBDIR/demo, based on the
cmain.c and account.cbl programs.

Obviously, most vendors have their own methods of doing this. Acucobol,
for example, requires you to relink the runtime system, (which then allows
COBOL and C debugging, one with the Acucobol debugger, the other with your
native C debugger, dbx of gdb or sdb (shudder) or adb (truly shudder!) )
This also allows different argument-passing conventions. You can pass
arguments in as a structure (to get some idea of how COBOL understands
them, with length information, type (comp or display), number of digits to
the left and right of the decimal point, etc.), or as strings of bytes, or
as native C types.

Once this relinking is done, you call the C function just like a COBOL
subroutine.

The main point here is that it would help to know which implementation of
COBOL you are using. The answer to your original question depends on it.
