```
COBOL II. One of the particularly upsetting things to them is
that they can no longer use READY TRACE with COBOL II.

Is there some COBOL II equivalence to READY TRACE or some other
suggested way to get the same output as READY TRACE provides?
```

```
I'll never understand why they took READY TRACE (along with
CURRENT-DATE) out of VS Cobol II. But it's gone, so you just have
to live with it. The alternative; pain in the b.tt that it is; is to
use DISPLAY statements. There's all there fancy debuggers,
Expediter, Cobtest, Micro-Focus etc...., but they take a heck of
long time to set up. READY TRACE was so easy and convenient and
many times gave you just what you needed in a crash hot debugging
session at 3am. Probably the people that make these dumb decisions
never have to do that.
```

```
My replacement for the READY TRACE in COBOL 2 when not using COBTEST
involves modifying my programs as follows:

1) Add a SOURCE-COMPUTER WITH DEBUGGING MODE line in the
CONFIGURATION SECTION:

001400 ENVIRONMENT DIVISION.
001500*----------------------------------------------------------------*
001600 CONFIGURATION SECTION.
001700*----------------------------------------------------------------*
001800*SOURCE-COMPUTER. IBM-370 WITH DEBUGGING MODE.
001900 SOURCE-COMPUTER. IBM-370.
002000 OBJECT-COMPUTER. IBM-370.
002100*----------------------------------------------------------------*

I use two SOURCE-COMPUTER lines: one with "WITH DEBUGGING MODE" and
one without. For normal runs I put an astrisk in column 7 before the
SOURCE-COMPUTER WITH DEBUGGING MODE line and have the other active.
When I need to trace the program run, the astrisk is removed from the
SOURCE-COMPUTER WITH DEBUGGING MODE line and placed before the
SOURCE-COMPUTER stand-alone line.

2) Add a DECLARATIVES at the start of the PROCEDURE DIVISION as
follows:

044000 PROCEDURE DIVISION.
044100*----------------------------------------------------------------*
044200 DECLARATIVES.
044300*----------------------------------------------------------------*
044400 AA-DEBUG SECTION.
044500*---------------
044600 USE FOR DEBUGGING ALL PROCEDURES.
044700 DISPLAY DEBUG-ITEM.
044800*----------------------------------------------------------------*
044900 END DECLARATIVES.
045000*----------------------------------------------------------------*
045100 000-START-OF-PROGRAM.

This will display your paragraphs just like READY TRACE, but in column
form and not across the page as READY TRACE does (easy to read). You
can leave the DECLARATIVES in place and comment out the
SOURCE-COMPUTER WITH DEBUGGING MODE line and the whole thing is
treated as comments.

3) If you need to perform additional processing (like displaying your
data-name contents) only while DEBUGGING MODE is on you can place a
"D" in column 7:

060000 IF WS-NAME-FIRST(1:1) IS NOT GREATER THAN SPACES
060100 THEN
060200 MOVE WS-NAME-FIRST (2:16) TO SUBR-NAME-FIRST
060300 ELSE
060400 MOVE WS-NAME-FIRST TO SUBR-NAME-FIRST
060500 END-IF.
060510D DISPLAY 'FIRST NAME IS ' SUBR-NAME-FIRST.

When you comment out the SOURCE-COMPUTER WITH DEBUGGING MODE line the
"D" lines are treated as just more comments.

With DECLARATIVES you can get real fancy with conditionals, additional
processing, etc. Like checking DEBUG-ITEM for a certain paragraph to
set in motion other conditionals to start a trace (good for loops). A
book to recommend on the use of DECLARATIVES is "COBOL Debugging
Diagnostic Manual" by Eric Garrigue Vesely, ISBN 0-13-140187-4.
Another is, of course, Gary DeWard Brown's "Advanced ANSI COBOL With
Structured Programming" ISBN 0-471-54786-7. Page 324 gives some
additional considerations on the use of DEBUGGING MODE in mainframe
and PC envionments.

Hope this helps,
Boyce Williams
```
