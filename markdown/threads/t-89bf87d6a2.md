Yes, I can confirm that OO is part of the '02 Standard.  (There is NOTHING that 
is called or is a "97" Standard.

In the '02 Standard all of Object Orientation (as defined there) is required. 
There is possibility that some of this (particularly "factory objects") will be 
made optional in the nest Standard.

OpenCOBOL started with implementing the '85 Standard and many of the most common 
extensions to that.  It is gradually adding features from the '02 and draft 
revision. (While still "cleaning up" features of the '85 Standard and 
amendments, e.g. the ALL subscript for functions and various aspects of nested 
programming.)

The fact is that modern software will AUTOMATICALLY create the "collection" 
for you transparently (whether it is a dynaset or a result set from an SQL 
query, or something else.) There are times when you want to throw something 
into a collection and you may want it ordered or hashed, but these times are 
not that frequent and there are constructs in other languages that make it 
very easy to enumerate and process "collections" whether they are officially 
Collections or not.  I can use "foreach" in C# to treat a sequential file 
exactly as if it were a collection, for example.

I know you like code so here's a sample...

I have created a collection of  metadata lines (called TP6MOSTMTD) 
describing a COBOL record layout.. This is really a series of element 
descriptions and was created from a sequential file by reading the file and 
shoving each line into the collection. Each line has information about the 
dataname, level number, record it bolngs to, type, group or element, occurs, 
redefines, offset in the record, and length. (This last was derived by 
runnning newcobdata as a component, from C#. I originally had my own code 
doing it, but I liked Robert's better, so I simply embedded the component 
into my C# application...That's the point about component based 
applications; you can easily replace a component, with no knock on effects, 
because everything is encapsulated)

Now I want another collection that is just the datanames, enclosed in square 
brackets, and I want to ensure that the first entry is the primary key field 
(or fields), so, as I get the datanames I prefix them with a "0" for primary 
key, and a "1" for everything else. This list of datanames will be used to 
generate lists for SQL statements.

The code iterates over the metadata (MTD) collection and creates a String 
Array (which is really another kind of collection).

// build a collection of COBOL datanames from the MTD
string subsetline = "";
ArrayList subset = new ArrayList();
subset.Clear();
foreach (string metaline in TP6MOSTMTD)
{
    if (metaline.Substring(79, 1) == "1") // prime key
    {
        subsetline = "0[" + metaline.Substring(89, 35).Trim() + "]";
        subset.Add(subsetline);
    }
    else
    {
        if (metaline.Substring(0, 35) == RGTableName.PadRight(35))
        {
            subsetline = "1[" + metaline.Substring(89, 35).Trim() + "]";
            subset.Add(subsetline);
        }
    }
}
subset.Sort(); // should get prime key first
// do the fields...
foreach (string COBOLname in subset)
{
    if (COBOLname.Substring(0, 1) == "0")
    {

        line = "                    " + "(" + COBOLname.Substring(1, 
COBOLname.Length - 1) + ",";
        line = line.PadRight(120);
        sw = TP6writeCOBSource(sw, "WRITE", line);
    }
    else
    {
        line = "                    " + COBOLname.Substring(1, 
COBOLname.Length - 1) + ",";
        line = line.PadRight(120);
        sw = TP6writeCOBSource(sw, "WRITE", line);
    }
}

While this is not particularly elegant code, it does demonstrate how 
"natural" the processing of collections is in C#. Notice that collections 
have innate methods like "Sort" and "Clear" which does make life pretty 
easy. You can also retrieve individual items with an index or value, as you 
would expect.

In procedural COBOL I'm not sure how valuable the provision of Collection

I could well be speaking out of turn, but this is my take on the
potential for Object COBOL in future OpenCOBOL releases:

Currently the Intrinsic FUNCTION and other compiler internals for
prototyping fields is in C code sections deemed BEWARE; DO NOT TOUCH.
For implementation of User FUNCTION this prototyping code will need to
be exposed to higher levels of the code generation sequence.  That
exposure and lifting of the code from internal to developer accessible
would be a first step toward Object COBOL.  As far as I know; which
really isn't a whole lot.
