I must have gotten 50 emails recently on the "<> for not equal" issue.  What
a waste of brainpower!

There are far more important features needed in COBOL. As an example, there
are 3 competing proposals for an XML interface proposed to J4.  As it is, if
an XML proposal was already ready and approved it would not make it thru the
process to be approved until 2006.

Instead of complaining how long it takes to get a new COBOL Standard out,
how about contributing to the effort by actually considering and commenting
on the imporant features (and proposals) which are already on the list for
inclusion to the next standard!

J4 has a standing document (S-16) listing the proposed features, which
include features such as asynchronous processing, national character and
locale enhancements, date and time features, editing enhancements, file
processing enhancements, etc..   That document has recently been prioritized
in terms of feature importannce. If not already posted, maybe Bill Klein can
post a copy of that for you of you to see and consider.   If all you COBOL
experts really want to contribute in a meaningful way, spend time thinking
about those important features - not these nits.

There already is an acceptable way to do "not equal" - not so for many of
the proposed features, for which there are important business needs.  Put
all that great brainpower out there to use on important features for a
change.

Just my humble opinion.

Jeff Friedman

You're absolutely right, Charles.

The Component Object Model (COM) has just such a standard that allows any
language to map its data.

That's one reason why components can be used from any language.

Without going into the full details:

VT_I4    =  4 byte binary.  (COBOL  pic s9(9) binary)
VT_CY  =  real number with 14 digits and 4 decimals. (COBOL  pic
s9(14)v9(4))

The above are a couple of examples.

When using COM with COBOL, conversion to the COM types is accomplished by
the compiler (Fujitsu), but I still like to do it explicitily so as to be
sure.

Here is an actual code snippet that demonstrates this...

*----------------------------------------------------------------
 set-visible-properties section.
*
* Set the External Properties (Custom Properties) of the Method
* from the internal fields above.
*
* The custom properties are defined using the OLE (COM) standard
* so, instead of having COBOL pictures, they have a cross-platform
* format. For example, The REAL number we have built is defined
* as "VT_R8", the currency field is "VT_CY", and so on...
*
* These COM formats can be mapped to COBOL pictures and the compiler
* knows how to do this...(In other languages the same formats are
* mapped to whatever is suitable for that language).
*
* To be on the safe side I am going to ensure that the custom
* properties are built from EXACTLY the right kind of COBOL picture,
* even though this is not strictly necessary...
*
 svp000.
     move dec-len to VT-I2          *> VT-I2 is a binary two byte field
(Mainframe halfword)
                                                  *>  Here's the definition:
01  VT-I2    pic s9(5) comp.
*
*  All the quoted fields below are  custom PROPERTIES of POW-SELF.
*  POW-SELF is the current instance of the "Get-Number" method of the
*  String2Num Object.
*
     move VT-I2 to "intDecimal" of POW-SELF
     move int-len to VT-I2
     move VT-I2 to "intInteger" of POW-SELF
     move num-fp to "realNum" of POW-SELF
     move num-currency to "realCurrency" of POW-SELF
     move dbl-integer    to "dblInteger" of POW-SELF
     move dbl-decimal    to "dblDecimal" of POW-SELF
     move string-integer to "strInteger" of POW-SELF
     if dec-len NOT = zero
        move string-decimal to "strDecimal" of POW-SELF
     else
        move spaces to "strDecimal" of POW-SELF
     end-if
     if is-a-number
        move POW-TRUE to "ItsANumber" of POW-SELF
     else
        move POW-FALSE to "ItsANumber" of POW-SELF
     end-if
*     invoke POW-SELF "ShowStuff" using input-string
     .
 svp999.
     exit.

The COM (previously OLE) standard, allows cross platform data definition and
is used by component builders all over the World.

A vendor MIGHT be able to provide access to other "non-COBOL" Class
libraries under the covers, but I believe it is a pipe dream and not worth
the effort.

The answer is to use the best tools for the job and not try and stretch
COBOL into a "one language wonder", as some here seem to be intent on doing.

Although I can invoke the Java Class libraries through COBOL, I don't. If I
want Metal windows (for example...) I'll write them in Java.

Now, I accept that that is my preference and other people will want to do
other things, but the question that has to be asked is this:

"Should we even try and make COBOL all things to all people, at this stage
of the game?"

I believe the answer has to be "No".
(If we were going to do this we should have done it 20 years ago...)

We will succeed in making it nothing to anybody.

A better chance of success is to focus on the strengths of COBOL and ensure
that those are enhanced and rejuvenated.

That way COBOL can maintain its place as a very useful language for
commercial application development (as long as commercial applications need
to be "developed").
