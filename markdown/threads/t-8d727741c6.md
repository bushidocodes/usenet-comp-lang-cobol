I don't usually respond to insults (actually I don't usually receive any to respond to), 
but clearly you have never done any forensic work.  Having the layouts obviously makes 
it easier, but it is far from impossible without them.  Over the last 35 years I've had 
to do it many times in circumstances where the layouts just plain weren't available. You 
can't just tell the client "I'm sorry but you can't get there from here".

I well remember an early assignment (1969) where I was told "Our COBOL experts have told 
us it can't be done".  They meant that they didn't want the hassle of doing it, and 
should have said so. Their credibility suffered when I finished it.

You (or the client) usually have some indication of what is in the record and by 
examining a few records you can determine the layout - the parsers help you do that by 
providing various views.  

Of course, as you note, if you have the original program and can generate reports that 
also makes it much easier.  You can re-direct the reports to a print file and turn the 
parser loose on that.   This is often the only route when the system designer has 
deliberately obfuscated the data files to keep those quarterly usage fees coming in - as 
with one library package that I had to rescue the catalog data from.

Typical example (couple of weeks ago - client identifying data removed):
=================================================================================
First email to me:
I am attempting to import my data files from an old program named "****"
made in 1987.  I believe the process I should be using is a binary
defenition, however the data is not ouputting properly.  I have attached the
data files and a text version of the form it is entered in.  My hope is that
you may have some suggestions so I may export it to a standard dBase file
importable by Microsoft Excel.

My response:
Since ParseRat will extract a maximum of 99 fields from a binary file in one pass, it 
would take two passes to extract all of the fields.  The way that works would be to 
extract the first batch into a dBase table including a record number - then the second 
batch would be extracted into another also with the record number.  Then the two tables 
would be joined based on the record number into one resulting table. (This way records 
with ANY number of fields can be extracted).  You'd create a PRZ preference file for 
each batch.

Your records contain a combination of what today would likely go into different tables 
anyway, since they combine customer data, order data and item data.  A modern database 
would likely keep them in three tables so that:
1. You wouldn't need to keep entering the customer data for every order.
2. You wouldn't be limited to ten items per order.
2. You would be able to run reports by item type and customer.

Some other notes:
The text data will extract fine (as in the example I sent).
The numeric data will extract fine (the figures are include in binary fields as cents 
and I know they are right because the arithmetic works out properly).
The dates appear non-standard in format - they might be coded as a number of days after 
a datum date but I can't tell because I don't know what the values should be.

The above notes, together with the sample should allow your local database guru to do 
all that you need.

Second email to me:
Thank you VERY much for the information and quick response.  I am analyzing
the data now and adjust the settings of the .prz to read our other .dat
formats.  Thanks again!
=================================================================================
