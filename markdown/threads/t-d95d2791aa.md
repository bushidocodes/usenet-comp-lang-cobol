```
The folowing is related to OO, but probably applies equally to
Structured. Based on a tip Bill Klein gave me I have come up with a
generic class template for handling SEQUENTIAL and LINE SEQUENTIAL
files. The latter is used to read or write some 20 separate ascii
delimiteds

I would like to do the same - create a template for indexed (ISAM)
files. Even though my indexed files are fixed length I could still use
the "record varying clause" to 'fool' the compiler - however the problem
is the ISAM PrimeKey and Alternate Keys, which could be located anywhere
in the record; of course their positioning in the record is determined
the first time the file is created. Any ideas how to resolve the
Prime/Alternate Key problem ?????

The following is the File Class for Ascii-Delimited/Line-Sequential :-

Jimmy, Calgary AB
-----------------------------------------------------------------------
       *>-----------------lsqfile.cbl-----------------------------------

       Class-id.	  LineSequentialFile
			  inherits from FileMethods.
	..............

       *>---------------------------------------------------------------
       FACTORY................
       End FACTORY.
       *>------------------------------------------------------------
       OBJECT.
       *>------------------------------------------------------------
       INPUT-OUTPUT SECTION.
	    Select		 Data-File
	    assign		 Data-filename
	    organization	 line sequential
	    access		 sequential
	    file status 	 ws-filestatus.
	
	*> I would need references here to Prime/Alternate Keys

       *>---------------------------------------------------------------
       FILE SECTION.
       FD  Data-File
	   record varying from 1 to 1000 characters
	      depending on ws-record-size.

	*> Again a reference required for Prime/Alt Keys
	
       01  Data-record		 pic x(1000).
       *>---------------------------------------------------------------
       WORKING-STORAGE SECTION.
       copy "LinkErr.cpy" replacing ==(tag)== by ==ws==. 
       copy "wsresult.cpy".
       01 Data-filename 		     pic x(100).
       01 ws-record-size		     pic x(4) comp-5.
       *>------------------------------------------------------------
       OBJECT-STORAGE SECTION.
       01 os-record.
	     10  occurs 1 to 1000
		 depending on ws-record-size	pic x.
       *>----------------------------------------------------------
       Method-ID. "access-file".
       *>-----------------------------------------------------------
       Local-storage section.
       01 ls-result				pic x(4) comp-5.
       Linkage Section.
       copy "fileactn.cpy".
       01 lnk-result				pic x(4) comp-5.
       Procedure Division Using     file-action
			  returning lnk-result.
	  set file-result-ok to true
	  initialize ws-Error-Data

	  if   dont-initialize
	       continue

	  else initialize os-record(1:ws-record-size)
	  End-if

	  Evaluate TRUE

	    When read-next
		 Read Data-File next record
		   At End
		     Set file-finis to True
		   Not at End
		     Move Data-record  (1:ws-record-size)
			  to os-record (1:ws-record-size)
		 End-Read
	    When write-record
		 Move os-record     (1:ws-record-size)
		      to Data-record(1:ws-record-size)
		 Write Data-record
	    When close-file	    Close Data-File
	    When open-input-file    Open input Data-File
	    When open-output-file   Open output Data-file
	    When delete-file
		 call "CBL_DELETE_FILE" using	  Data-Filename
					returning ls-result
		  move "00" to ws-filestatus
	   When other
	     invoke super "invalid-action" using file-action
	     set file-error to true
	  End-evaluate

	  *> Code below is not moving a 'key' but the first 50
	  *> characters of the line-sequential record

	  if	  ws-filestatus <> "00"
		  move os-record(1:10)	to ws-error-key
		  move os-record(11:40) to ws-error-name
		  invoke super "check-file-status"
			 using	   ws-ErrorFields
				   file-action
				   ws-file-result
			 returning lnk-result

	  else	  move ws-file-result to lnk-result
	  End-if
       End Method "access-file".
       *>---------------------------------------------------------------
       Method-id. "initialize".
       *>---------------------------------------------------------------
       Linkage section.
       01 lnk-Parent			    object reference.
       copy "makefile.cpy" replacing ==(tag)== by ==lnk==.
       Procedure Division using     lnk-Parent
				    lnk-makefile-params.
	 initialize		  ws-ErrorFields
	 move lnk-filenum      to ws-filenum
	 move lnk-filename     to Data-Filename
	 move lnk-record-size  to ws-record-size
	 invoke super "set-window-title" using lnk-Parent
					       lnk-makefile-params
       End Method "initialize".
       *>---------------------------------------------------------------
       Method-id. "get-file-object".
       *>---------------------------------------------------------------
       *> Gets the data object after a READ NEXT
       Linkage section.
       01 lnk-record.
	  10  occurs 1 to 1000
	      depending on ws-record-size    pic x.
       Procedure division returning lnk-record.
	Move os-record(1:ws-record-size)
	  to lnk-record(1:ws-record-size)
       End Method "get-file-object".
       *>---------------------------------------------------------------
       Method-id. "set-file-object".
       *>---------------------------------------------------------------
       *> Sets the data object prior to a WRITE
       Linkage section.
       01 lnk-record.
	  10  occurs 1 to 1000
	      depending on ws-record-size	     pic x.
       Procedure division using lnk-record.
	Move lnk-record  (1:ws-record-size)
	     to os-record(1:ws-record-size)
       End Method "set-file-object".
       *>---------------------------------------------------------------
       End OBJECT.
       End CLASS LineSequentialFile.
       *>---------------------------------------------------------------
```

```
Jimmy,

it's time you dropped this archaic ISAM nonsense and got into real
databases..<G>

I did exactly what you're trying to do in the early 1970s (as disk drives
became readily available), with VSAM KSDS on system 360.

We started off by writing a general purpose VSAM file handler (incorporating
all the Methods you posted) which became a "template". This then had certain
symbols substituted into it , and could be used to quickly generate a
specific file handler by editing these symbols.

You ended up with a module which did everything.

However, we had one of these modules for every file.

Just as you are experiencing, the problem was "knowing" where the keys were
located.

Reference modification was not available at the time, so I wrote an
assembler routine which we called VARMOVE. It allowed you to reference
fields with an offset and a length. It lasted me for many years and later I
translated it into Intel Assembler so I could use it on the PC.

Nowadays, (we all have to move with the times, Jimmy...<G>) I have replaced
it with reference modification in Standard COBOL.

So, to solve your Key problem make sure that when the Method is invoked it
is passed the offset and length of the required key (or keys). Like this...

RECORD KEY IS os-record (offset:length)

The first time you write such a record, the necessary index(es) will be
created. You will need to recognise and differentiate between individual
files.

 (This does not preclude having a generic solution to handle them (you will
instantiate a different object for each file, with the appropriate file name
and key data as outlined above.)

And ensure that the correct offset and length for the key(s) is passed when
you invoke your generic solution to access them later.)

OO allows a much better solution than our original one of generating
individual modules (one for each file, and then compiling them) although the
effect is identical.
```
