```
The result is that FIELD04 will have a value of zero.

>What is the usefulnes of this clause?

'P' is called the 'scaling symbol'.  It is used to store values
which will never have digits within a certain range of the decimal
point (e.g. two in the above).  For example, if you were storing a
number which will always be in whole millions, you can save storage
space by using a PIC like this

   03  MILLIONS         PIC S9(4)P(6) COMP.

Only two bytes are used to store the above number (assuming COMP
is binary).  Other than the fact that you cannot store any values
in fractional millions there (and it takes less space), it acts in
all other respects as if it had been declared as

   03  MILLIONS         PIC S9(10) COMP.

One other use is that you can truncate to a certain range of digits
other than the decimal point by moving a value to a field with the
'P' character where you want to truncate the digits (e.g. millions
in the example above).  This is why in your example FIELD04 is set
to zero.

>The documentation says that the P's are replace by zeros.

That is not strictly correct in a general sense.  The more general
statement is to say that the 'P' characters are 'treated as if' they
were zeros in calculations, editing, compares, etc.  There's nothing
actually 'there' at all.  'P' is similar to the 'V' assumed decimal
point in that respect.
```

```
Good explanation, Jusdon.  For the benefit of the original poster my
old college text explains 'P' thusly:

The character P indicates the presence of an assumed zero.  The Ps are
used to position the assumed decimal point away from the actual
number.  For example, the actual value in storage is 15.  It would be
treated as 15000 if the PICTURE clause is 99PPP, or as .00015 if the
PICTURE clause is VPPP99.

The scaling position character P is not counted in the size of the
data item.
```
