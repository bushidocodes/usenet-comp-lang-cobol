```
What you need to do is install a hook process. A hook process can intercept
all messages on the system or for a thread, and prevent WM_CLOSE from
reaching your launched process. See the WinApi doc for the
SetWindowsHook[Ex] function for details.

If the launch you make in your "first" program is what triggers the start of
the "monitor this application and don't let it close until Windows
shutdown", then that's actually not too hard to write. It's easier if done
at this point because you have a process handle and thread id (or maybe a
thread handle, but that's good enough). Otherwise you have to hunt thru all
the open windows on the system, or hunt through the task list and get the
EXE names, in both cases hoping you can match a text string ( and hope no
one renamed it, etc). (Plus, task list enumeration specifics vary across
Win9x, NT & 2K).

But, if the second (launched) process is one you wrote, there are a lot
easier ways (and I can think of two off the top of my head) to do this
within that application. (And for the proper consideration, I'm sure there
are people available to show you how). (Hint, hint).
```

```
Well, no, it can't be quite that simple to use the message technique. But..
the logoff situation (a new situation not previously described) is covered
by one of the parameters of the WM_ENDSESSION message. The lparam tells you
the difference if Windows is ending or the user is logging off. (One source
I have from 1996 says this only works under Win 9x (not NT);  but the other
source (newer) says it works on all Win/32 systems.)

But if your spawned program does not have a Window procedure (e.g., your
sample
program), this method cannot work. So, it is not a matter of "any" child
program; rather, it is a child program with a window procedure.

As far as "eating" messages, you need to add a window procedure to your
program. That, or launch a program which has as its sole function creating a
window, and IT launches the "real" server and watches for important
messages.

But now you are getting into an area beyond my experience: trying to do
something different between "shutdown" and "logoff", and perhaps involving a
console program.

If your child program is that straightforward, you might try another
approach. To check for new records, you can use WaitForSingleObject and
Windows will tell you if the file has changed. Or, you can use an event an
pulse it when the first program needs the services of the second.

Given your launched program is only checking for new records in a file, I
don't understand why the program has to keep running; is what's  required
just that it be available when the first program runs?  In this case, have
 the first program create a mutex when it starts. If the create succeeds,
the second program is running and does not need to be launched, because...
you have the second program create the same named mutex object whenever it
starts. If the create there fails, you don't care!
```

```
My company's software package, a warehouse management system, runs on
COBOL because it must run on both Unix and Windows NT.  On Windows NT
it runs as a console app because that is the way character based apps
work on NT.  The codebase, except for some unusual places is the
essentially same, and the look and feel on both platforms is EXACTLY
the same.

Now, on Unix, there is no standard nice way of starting, stopping, and
monitoring servers.  So, what did we do?  We build our own Server
Monitor program into our package.  It starts, stops, and monitors all
servers that pertain to our package.  Our documentation outlines how
to use this Server Monitor.  Now, before this current release, the
Server Monitor was not supported on thr NT version of our package
because of the extreme differences between Unix and NT with regard to
this functionality.  However, it was required to become part of the
current release with the goal that it must work exactly the same as on
Unix as far as the way the user sees it is concerned.

Making the process different for NT, such as requiring additional
setup in NT's server control panel is not an option for me because of
the aforementioned requirements.

I appreciate the alternative suggestions, but they are not options. 
As far as I'm concerned, I'm almost there.  The server monitor works
to the point that servers start, stop, and run properly.  They just
die when the user logs out.  That was ONE SMALL hurdle I though I
could overcome, rather than re-engineering the way things work.

What we've decided to do in the interim is the following.

1. Require that the customer run Windows 2000 with Terminal Services.
2. Log onto the console.
3. Start a terminal services client session to itself under a unique
login name reserved for the person running Server Monitor.
4. Run Server Monitor in that session and then disconnect (but not log
off) that terminal client session until needed again.
5. Activate that session when returning to Server Monitor.

This is a kludge, but it works!
```
