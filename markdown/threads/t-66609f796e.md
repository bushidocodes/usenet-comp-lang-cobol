### MF Cobol's 32bit CCIIPX bad performance

- **From:** Scott Dickson <scottdickson@ibm.net>
- **Date:** 1998-10-07T00:00:00
- **Newsgroups:** comp.lang.cobol
- **Message-ID:** `<361B7F25.514F9DBC@ibm.net>`

```
We use Micro Focus Cobol's Fileshare NLM in our production environment.
The live system uses 16bit cobol (version 3.2.50 with v. 3.2.20 cciipx
modules).  We have been desperately trying to upgrade to 4.0.32 (32bit)
but CCIIX32 performance is 22 times slower on a Dell PowerEdge 4200
server with dual 333 PIIs and 256mb RAM.

10,000 reads with 16bit takes 25 seconds versus 9min 32sec using 32bit.

This is ludicrous and must be attributable to some setting or settings.
I am using the same COBOL.DIR for both, the same FHREDIR.CFG for both
and everything else is exactly the same.

MF tech support tells me they should perform about the same, but when
they ran my tests, they had the 32bit outperforming 16bit by a 4 to 1
ratio.

This problem has gone unresolved for 5 months now.  Any suggestions?

Anyone who has any experience with cciipx and fileshare NLM?

Scott
```

The only screen i-o is the display of the time before the test and the display
of the time after the test.  I have run into the poor text video performance
myself and know better.  Also, we only use .gnt code (in both production and
development) as we are the end user of the system.  I've animated the program
(I know performance is much worse and I don't use any times from animated
runs) and I have isolated the performance problem to the actual READ
statement.  I can actually time the interval from when I Step at the READ
until control comes back to me (for the 32bit cobol).  Under 16bit, it is
instantaneous, like it is for every other line of code.  MF is totally in
control at that point, so I can't programmatically affect the performance
(within the program).

According to MF development, there may be a problem with the CCIIX32.NLM on a
server with a second CPU.  They stated "There is a slim possibility that the
presence of the second CPU in the new server is affecting the threading
algorithms of the NOS, even with SMP disabled.  The newer CCI modules use a
different threading model to the older CCI modules".  At this point, we are
awaiting Lanalyzer software to trace the IPX traffic before, during, and after
the tests.
