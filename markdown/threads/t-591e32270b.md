Were evaluating Realia II Workbench, and I can't seem to figure out how to
determine the length of a variable in a c routine. For a simple example lets
use:
01 VAR-1 PIC X(10) VALUE "fdfdfd".
01 VAR-2 pic X(30) VALUE "AJHDHFDHGFGDFydtdtdtd".
. . .
CALL "C__UPPERCASE" USING VAR-1.
CALL "C__UPPERCASE" USING VAR-2.

Then the C code would be as follows:

void FAR UPPERCASE(char *String)

{
...
}

The question is how do I determine the length of variable that is sent to the
C subroutine (procedure/function)? For the first call, it would only be 10
bytes long. For the second call it would be 30 bytes long. If I can't
determine the length of a string then how could I have the C code make the
string ASCIIZ before I processed it.

There must be an easy solution to this. I can't believe that that there is
no way to do this. I have tried the MLI_GETMAXLEN routine, but it forces me
to make a statement like:

CALL "MLI_GETMAXLEN" USING VAR-1, X.
CALL "C__UPPERCASE" USING VAR-1, X.

But this would be pretty stupid to do for all my external calls.

Thanks in advance. ;=)

How about this (I'm assuming you have a LENGTH intrinsic) --

MOVE FUNCTION LENGTH (VAR-1) TO LEN-VAR-1.
CALL "C__UPPERCASE" USING BY VALUE LEN-VAR-1,
BY REFERENCE VAR-1.


void FAR UPPERCASE(int StringLength, char *String)

The general problem here being that variable lengths (in this case, anyway) are
determined at compile time and the (COBOL) code which declared the variable must
communicate its length to the other (C) code.

I do not understand how this code could have ever worked in the past: An X(10) field
with a value 'fdfdf' would have a value of 'fdfdfd space space space space' (COBOL likes
to space fill fields, i.e. MOVE and VALUE clauses in this example, but your C routine would
have been looking for a LOW-VALUE byte to terminate the string - and would
not have stopped at the end of each parameter buy would have continued until that
terminator was found.

I do not believe there are any solutions which require no source code changes. I see two
approaches (both require some work)

1. Change your procedural code, as has been suggested, to pass a length.

2. Change the VALUE clauses for the items to include the terminating low-value byte,
for example (using the supported, and soon to be approved ANSI feature of concatenated
expressions):
05 VAR1 PIC X(10) VALUE 'dfdfdf' & x'00'.
Note that you may need to get that low-value byte out of there later for most COBOL-like
processing (INSPECT REPLACING x'00' by SPACE). (I use the hex-literal extension as
LOW-VALUE might not have that value if a program collating sequence were in effect).

Or (maybe better) follow each field which you want to uppercase with a low-value field
(i.e. 05 FILLER PIC X VALUE x'00'). This may be a bit better as the value of your data
fields won't have the low-value byte within it. Your uppercase routine will have to
consider all the characters in the field in this case though.

P.S. Have you considered rewriting your uppercase routine in a "real" language. If you are
careful in your coding, our generated code is probably better than your C routine's. Maybe
something like the following:

WORKING-STORAGE SECTION.
05 cnt PIC S9(4) BINARY.
05 binary-field PIC S9(4) COMP-5.
05 filler redefines binary-field.
10 binary-field-lsb pic x.
10 filler pic x.
LINKAGE SECTION.
01 parm-len PIC S9(4) BINARY.
01 parm.
05 pic x OCCURS 1 TO 32767 DEPENDING ON parm-len.
PROCEDURE DIVISION.
PERFORM VARYING cnt FROM 1 BY 1 UNTIL cnt > parm-len
IF parm(cnt:1) is not less than 'a' and not greater than 'z'
move parm(cnt:1) to binary-field-lsb
subtract 32 from binary-field
move binary-field-lsb to parm(cnt:1)
END-IF
END-PERFORM.

For relatively short fields, this code might be quicker if the parm was moved to
a working-storage area, upper cased there and then moved back. References to
linkage section are indirect and involve extra segment register loads which are quite
inefficient compared to other processing. Try it and see.

Enjoy!
