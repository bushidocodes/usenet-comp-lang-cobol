You can't control what users will send you.  You need to consider context - treat a 
quote as a literal rather than a delimiter unless it is either
1. Adjacent to a comma
2. At the beginning of a record
3. At the end of a record

Resolve for the commas to split the fields - with a comma inside a quoted string 
(identified using the rules above) being considered a literal.

When you trim quotes, remember that there not be any for numeric fields.

If you don't mind translating from C, here is a routine that works for me:
----------------------8<--------------------------
 char linework[MAXLINE];
 char *letter;
 char *nextcomma;
 int fieldcount=0;
 strcpy(linework,InputLine);
 if(*(linework+strlen(linework)-1)==34){*(linework+strlen(linework)-1)=0;}
 letter=linework;
 while(*letter&&fieldcount<MAXDELIMFIELDS)
		{
		fieldcount++;
		if (*letter!=34)                      /* its not a quote */
			{
			nextcomma=strstr(letter,",");
			if(nextcomma>=letter)             /* it found one */
				{
				*nextcomma=0;
				strcpy(f[fieldcount],letter);
				letter=nextcomma+1;
				}
			else
				{
				strcpy(f[fieldcount],letter);  /* use the rest of the 
line */
				*letter=0;                     /* terminate loop */
				}
			}
		else
			{	/* it is a quote */
			char *nextquote;
			letter++;                          /* skip over it */
			nextquote=strstr(letter,"\",");    /* find the next */

			if(nextquote>=letter)             /* it found one */
				{
				*nextquote=0;
				strcpy(f[fieldcount],letter);
				letter=nextquote+1;         /*should be a comma*/
				if(*letter)letter++;        /* to next field   */
				}
			else
				{
				strcpy(f[fieldcount],letter);  /* use the rest of the 
line */
				*letter=0;				/* terminate 
loop */
				}
			}
		}

------------------------------------------8<------------------------

By no means definitive, but my particular approach on a PC. (I need CSVs to convert
old RM data files to MF data files, plus from M/F output CSVs going into oil/gas
company DBs. Additionally I need CSVs to create data sets as imports to MS Access
tables).

1 - Some propose, but I dislike, using STRING and UNSTRING

2 - I go with a template, fixed format :-

01 CSV-RECORD.
    05 CustomerNumber            pic x(05).
    05                                        pic x value ",".
    05 CustomerBalance            pic -999,999.99.
    05                                        pic x value ",".
    05 etc......

2a) It used to be that you designated text fields by surrounding them with quotes(")
as well as the commas. I think 'importing' systems are less stringent now and will
accept just commas - I'm just in the process of trying this out.

2b) Fixed format - easier to read taking the CSV into an editor that handles really
*long* lines, allowing you global replace etc.

2c) Make the CSV record a copyfile, then you can include it any relevant program

2d) Take advantage of the MOVE CORRESPONDING by keeping the field names generic,
(thanks Thane <G>)  :-

Input record :-

01 CUS-record.
    05 CustomerNumber            pic x(05).
    05 CustomerBalance            pic s9(06)v9(02) comp-3.
    05 etc.....

    MOVE CORR(ESPONDING) Cus-record to CSV-record

*BUT* carefully check out the syntax for MOVE CORRESPONDING so that you get it
right, for your particular record.

3) If you've dabbled with anything like MS Access, you'll know that it will quite
happily prompt you that you appear to be importing fixed format with the comma
separator.

4) Given that you take the above sort of standard approach, you write one program
with necessasry read/write logic, take copies of this program, re-identify the
Program ID and use copyfiles for both input and output.

Backtracking to the MOVE CORRESPONDING and copyfiles :-

FD IN-file
01 IN-record.
copy "cusrecord.cpy" replacing ==(tag)== by ==IN==.
FD OUT-file
01 OUT-record.
copy "cusCSVrec.cpy" replacing ==(tag)== by ==OUT==.

read IN-file next record etc......
move corresponding IN-record to OUT-record
write OUT-record

Obviously just switch the "INs" and "OUTs" around when you want the CSV to be the
input file.

(I can do it slightly 'neater' in OO - but oh dear, you are on a mainframe <G>).

Jimmy, Calgary AB

One trick is to not tamper with commas imbedded within quotes. We convert
non-imbedded commas to low-values first, then UNSTRING the result based on
the low-values tag. We also remove the quotes. Here's a snippet (hint: SW is
either 1 or 2):

000078     ENTRY 'DECSV' USING LINK-LEN LINK-RECORD.
000079
000080     MOVE 1 TO SW    *>Looking for quote 1 or 2
