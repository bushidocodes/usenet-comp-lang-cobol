In fact, the 2002 standard makes it slightly easier to write
conforming code when using sections. The 2002 standard does not
require a paragraph name following the section header; the 1985
standard did.

----- 1985 conforming

       paragraph-name-1.

           sentence.

procedure division without even one paragraph name. <g>

The 85 program will have more nesting of code within its paragraphs.

Having to go through more paragraphs, to a greater depth, to

bulky paragraphs trying to do far more than a paragraph body
should.

recommendation says nothing about why to use GO TO statements;
that is left to the programmer.

On Mar 26, 9:16 am, charles.levi...@gmail.com wrote:
> Saw these 2 standards in IBM's supplied list of code review recommendations
> "Avoid THRU phrases in PERFORM statements "
…[5 more quoted lines elided]…
> Any basis to making these recommendations? Does anyone adhere to these?

The first point to make is that consistency in the code is by far the
most important consideration. If you are working on a program that
does PERFORM section then continue to keep that style. The worst
programs are the ones that have a variety of styles. In general this
should apply to the whole system, and probably to the whole site. This
is why they have site standards.

If the program have a consistent style then it is much easier to
predict the effects of changing or adding code at particular points.

However, the _ONLY_ reason for using PERFORM section or PERFORM para
THRU para is to allow for additional paragraph labels and/or to cater
for using GO TO. This was usually required in pre-'85 COBOL. Anything
written with a modern style in recent (quarter century) standards has
no need of that.

In most modern languages there are different types of labels, the type
restricting what means can be used to access it. eg in C (which is
minimally 'modern') a procedure or function name can only be 'called'
and cannot be 'dropped into' or gotoed, a label can be used for a
goto, or an implied goto of a switch, but not in other ways.

In COBOL a label, whether section or paragraph, can be dropped
through, PERFORMed, or GOTOed. At those verbs it is obvious what will,
or should, occur but when modifying a program the changes are not done
at those points, they are done at the lower paragraph level.

The actual problems with various styles happens when a paragraph label
occurs in the code. The label may be commentary, ie having no
functional purpose (such as Peter's SECTION header prior to a method).
It may be the subject of PERFORMs or a PERFORM THRUs or GOTOs or all
of these. It may also be dropped through because some label above was
subject to PERFORM THRU or PERFORM section.

The _only_ way to fully resolve which of these, if any, apply is to
examine the _whole_ program.

Site standards may dictate that only specific forms of these
constructs may be used. Code reviews may be used to ensure these
standards. But if the code does not have a 100% guarantee that it
follows the standards, or there are no standards, then any occurrence
of THRU, GO TO, or SECTION could mean a complete analysis of the whole
program, including any copybooks, is required before the logic flow
around each paragraph label can be definitively determined.

When I moved to '85 compilers I adopted the style of only using
PERFORM paragraph (no thru or section). This meant that GO TO could no
longer be used and forced me to use 'proper' logic structuring and
shorter paragraphs. In some cases this led to deep indenting and
longer paragraphs but it was easy to extract a block and make this the
next paragraph, replacing it with a PERFORM of that new paragraph.
Because there were no PERFORMs of ranges of paragraphs there was no
need to site this relocated code away from that it was taken from, no
need to surround it with section headers or exit points, no need to
examine any of the rest of the program to see it it was within range
of some other control. It wasn't, each paragraph was separately
PERFORMed.

      Process_some_data.

          IF ...
              PERFORM UNTIL ...
                  IF ...
                      ...
                         ...
                  ...
                  IF ...
                      ...
                         ...
                  ...
              END-PERFORM
          END-IF
          .

Could become: _without_ having to examine the whole program:

      Process_some_data.

          IF ...
              PERFORM Process-some-data-record
                  UNTIL ...


          END-IF
          .

       Process-some-data-record

          IF ...
              ...
                 ...
                  ...
          ELSE
          IF ....
              ....
          END-IF
          .

The question then arises about how the program is guaranteed to follow
a specific style, which any changing of code must rely on. In the case
of the PERFORM paragraph style this can be checked with simple text
searches looking for SECTION, THRU, THROUGH, or GO. The complete
absence of these gives assurance that there is only one usage of any
paragraph label, and that is: it is PERFORMed and the range terminates
at the next label.
