As I stated in a much earlier note in this thread, it is entirely correct
for you to following your teacher's instructions when doing an assignment
(Just as you need to follow a user's requirements when creating application
logic in the "real world".)

HOWEVER, unless your instructor really has never written a real program
(which is possible, but I can hope not true), I think that if you approached
the instructor and asked "Is there ever an application design which makes it
correct to "correct" the data?" Then, I bet your teacher will admit there
are such times in the real world.  Let me repeat several techniques that are
each used based on application design requirements when you get bad data:
(There probably are more than this, but this will give you a good start -
that you could show your instructor and ask about.)

1) If some bad data is for a truly critical piece of information, then you
create an informative message (report, console, somewhere it will be seen)
with information to identify where the data came from, what is wrong with
it - and then you "gracefully" end the program.

2) If this is bad data in a field that fairly frequently gets incorrect
input (often in transactions requiring manual input), then you probably can
just create an error report (or transaction) and go on to the next record
(leaving a clear audit trail of what you skipped and why)

3) Let's say you are creating the actual online transaction where real
people enter data manually.  If they enter bad data, it is often possible to
make a guess as to what they really mean.  In such cases, you should go back
to the user with some type of message about "you entered "ABC", but this is
not valid - you probably mean "abc" - if so press enter to accept this,
otherwise, please re-enter the value for field XYZ"

4) Sometimes you really can guess what was meant by bad data.  A classic
example of this is when a numeric field is left justified instead of right
justified.  For example a 4 digit field (which accepts blank padding) had
"123 " instead of " 123".  If your REAL WORLD users tell you that this often
happens - even though it isn't supposed to and that this is OK, then you may
well FIX THE DATA to make it " 123" - write a report indicating that you did
this (and for which record).  Again, depending on the actual application
demands, it is possible that you will need to fix the data in the input file
before it goes to the next program in the application.  In other cases, you
may just fix the data in your temporary storage and keep the bad data in the
record.  THIS DEPENDS ENTIRELY on the application - what is needed and what
is expected.

  If you have an instructor who tells you that this type of thing will never
happen (and be what is best for the application), then YES, your instructor
is lying to you!  If on the other hand, your instructor says that your first
priority is to get your program to handle bad data in the application
defined way and then get whoever or whatever has created the bad data to fix
it, then the instructor is correct.

Nothing in the world will convince me that anyone who knows about any real
world computer application written since about 1965 would think that "crash
the system" is always the best way to handle bad data.  If you have an
instructor who really thinks that, please tell him/her that I am awfully
GLAD they are in the class room and not working at my Bank, Insurance
company, or even fast-food restaurant's data processing center.

Finally, if you are at a school that encourages to take EVERYTHING that any
instructor says as the ONLY way to think about it, then you in a military
organization (that never heard of Neurenburg and what following orders
means) and not at a school.  Use what your instructor tells you - but if you
don't understand WHY or how it fits in with the rest of the world (or even
why it seems to be different from what others are telling you) then you need
to ASK the teach to help you understand - and then you need to make up your
mind for yourself.

records that come from 1000 different sources.  These records have the
julian DAY, no year, in them that must be converted to a gregorian 
date for a database load.

I generate a year, based on the current year and the current julian 
date, and use the function integer-of-day and then function 
date-of-integer to get the gregorian date.  

I'm sure you are aware that the intrinsic functions are intolerant of 
bad data.  Talk about programming to crash!

them!

alpha gets a "boilerplate" date of 01-01-50 thrown in.  (Yes it's 2 
digit years, we don't need 4 digits to ID the record!).

Telling the people to fix their data is impossible.  It has to run, 
and there is a LEGAL requirement for this data, that it has to run 
NOW.  There are a lot of reasons to fix the data - and almost NONE for
blowing up (there are some though - ie NO data <G>).

Deadline always play an important role on how we react to emergency.

I've worked for a Telco that also receive millions of toll calls
records from other carriers telco, they do it in a more formal
way. The frontend validation jobs will validate the incoming
data and then divides the data into two group, the good one
upload to the validated DB and the bad one upload to the
recycle DB. Of course the routine exception reports are a must.

The design of the validation jobs won't abent even with bad
data is welcome by the programmers and the operators. They
don't need to support the abend in the middle of the night.
The batch window is 10pm - 7am.

Then the users are required to handle all those exception data
in the recyle DB thru a small online system. They can either add/
delete/modify the data records. And they have 24 hours to work
on that before the monthly billing jobs start.

When the monthly billing jobs start, it will rerun the same
frontend validation job again but will feed in the records from
the recycle DB. If it still gets rejected, sorry you miss the
bus, and need to be stayed at the recycle DB until next month
billing.

But the idea still the same. "Edit the data" but in a more 
traceable way.
