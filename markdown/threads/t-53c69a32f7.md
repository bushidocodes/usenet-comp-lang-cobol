```
"there is absolutely no reason to use GOTO"

This is *nearly* true in a language with rich control structures (although
many would say that the use of GOTO for implementing finite state machines
is a residual use), but in a language like COBOL with very weak control
structures, it is hard to avoid the use of GOGO except by kludges that
make the code less readable.

In particular, given that COBOL has no effective exception mechanism, one
often has to use a GOTO to handle exception conditions. After all if you
*really* don't like GOTO, then of course you should not permit STOP RUN
except at the end of the program, since its use anywhere else is of
course a GOTO. In practice the use of exceptional termination processing
ending with STOP RUN, and accessed by GOTO is often appropriate.

THe whole business of gotos in programming languages sure was an odd
phenomenon in the US. The US had standardized on Fortran and COBOL, both
of which required the extensive use of GOTO in earlier versions, and when
Dkikstra's letter appeared, it caused somewhat of a sensation (note that
one person in this group actually credited Djikstra with inventing the
idea of not using gotos). This of course is ludicrous, anyone writing
ALgol-60 (including all students taught programming throughout the 60's
in Europe) of *course* knew about restricting the use of gotos. It's
actually a completely obvious principle if you are working in a language
with a nice recursive nested control structure syntax like Algol and its
deriviatives.

Furthermore, not only did the US suddenly "discover" gotoless programming
overnight, but as is so often the fashion, became fanatic about it, and
decided that good = "no gotos", bad = "gotos". This attitude is of
course completely silly, but it is *amazing* how many otherwise reasonably
competent programmers fall into this trap. It is also interesting to see
how often it is the case that people don't really understand the principle
behind the idea. I have met many COBOL programmers who could not begin to
understand that STOP RUN is a GOTO that is a particularly distasteful 
version of GOTO< to be avoided where possible (but certainly not to be
avoided completely!)

I find it remarkable that this kind of discussion could be taking place
in 1994. I had thought that naive views of gotos had pretty much
disappeared and that peopel had generally settled into a more reasonable
view (basically sumamrized by the position that one should minimize
the use of gotos, but in certain circumstances, they are not only
acceptable but preferable to atlernatives).
```

As we all know, structured programming can be done in any language,
English and 86x assmebler included. So it can be done in COBOL too.
However, COBOL is rather inconvenient for this. Why?
Here's a short list of things that I discovered while writing
a COBOL parser:

1. Some statements are plain misleading. Especially EXIT statement.
Standard says it's equivlent to label in other languages.
So when you GO TO name-of-exit-paragraph, it does not mean at all
that you are going to exit the current paragraph/section.
Control can fall thru if PERFORM THRU that ends at some other later
paragraph was called.

2. There is no clear separation of call graph and control flow graph.
To find the logical end of function, you need to do a complete
semantical analysis of the whole program.

3. Control structures are sick. Doing the loop and procedure call
with the same PERFORM statement is hardly encouraging a structured
style.

Speaking of attempts to structurize COBOL, we should mention
Intersolv APS Structured COBOL. While it's a macro expansion
of regular COBOL and it does not create a new semantics
(that is, it's impossible to define an end to a function --
a point where control returns no matter what),
it is better when regular COBOL in encouraging structured style.

BTW, has anybody encounered a tool that will build a good
call graph - control flow graph of a COBOL program?
This should help a lot in discovering properties of a program
and in enhabcing its maintainability.

There is indeed an implicit GOTO between successive statements. 
Furthermore, you can obviously write an IF statement with a set
of equivalent GOTO's.

There are actually two things that people worry about when they worry
about GOTO's, and the IF example indicates the difference nicely.

First, using GOTO's you can construct wierd convoluted control structures
that are hard to understand, and violate principles of structured
programming (one such principle is the general rule that control
components should have a single entry and single exit. Gotos can for
example be used to generate programs that have irreducible flow graphs,
which is a bad thing for some optimziation techniques, and for much the
same fundamental reason, a bad thing for humans trying to understand.

Second, a quite different point, it is easier to read code that uses higher
level constructs than the GOTO. Take the example of an IF written out with
GOTOS. This does NOT violate the fundamental structuring rules, but we still
prefer the IF.

So the implicit GOTOS between procedural statements, if made explicit,
would not violate the first principle, but they surely would violate
the second principle.

A knee-jerk absolute anti-goto philosophy will of course guarantee not
violating either of these two principles. However, there is a third
principle: WRITE CLEAR CODE!!

THe acceptable uses of gotos, for those who do not have the knee jerk
reaction, typically involves violating the second principle, but not
the first. For example an exceptional exit from a loop that uses GOTO
still does not violate the fundamental structuring rules. 

The reason one may want to violate the second principle is simply to
accomodate the third (WRITE CLEAR CODE). Of course using a goto that
does NOT achieve this third goal is indeed gratuitous.

It is much rarer to want to violate the first (structuring) principle. The
only generally accepted case (generally accepted means lots of competent
programmers accept it, not all!) is the finite state machine case.

Many programmers find the style:

       Initial_State.
          Perform Read_Next_Data
          if Next_Data is zero go state_zero_found
          else go state_no_zero_found.

to be clearer than the equivalent formulation with a big case statement
(evaluate), and an explicit state variable.

Tastes differ on this, but clearly the judgment should be made on the
basis of which seems easier to read. The two formulations are identical.
The case version is of course more verbose. What seems quite wrong is to
base your preference between these two versions on a knee-jerk
anti-goto reaction.

I should say that the only time this style seems justified to me is
when you really do have a big complex multi-state finite machine, so
that there are lots and lots of state paragraphs like the above.

Clearly if you see just that Initial_State paragraph in isolation it looks
suspiciously like (and probably is) a bad case of mis-structuring. I trust
that what I am talking about is clear (if you don't know what a finite
state transition diagram is, then it probably won't be :-)

AN interesting anecdote. At one time IBM had an absolute rule that code
with more than x% gotos required high level management approval (a wonderful
bearocratic adaptation to the no gotos school).

Fran Allen had an expert compiler writer working for her, who wrote a
lexical analyzer. Since lexers typically DO have organized finite state
machines, often of some complexity, they are often a repository for the
kind of style I suggest above where the state is encoded in the program
counter, and state transitions are achieved by gotos. THe code of course
required high level management approval :-)

Another anecdote. Jean Ichbiah was looking at some of my Ada code that was
part of the Alsys Ada compiler (Jean was the original designer of Ada). He
was surprised to see some gotos in my code:

  JDI:  I am surprised by all the gotos
  RBKD: Finite state machine
  JDI:  Ah, fine!

Finally a little efficiency note. I still have not seen a compiler that
will take the big case statement and convert it into the efficient form,
which is too bad. Certainly the fundamental notion of encoding state into
the program counter is one of the really powerful notions in sequential
computers, it is too bad when people don't get to take advantage of it!
