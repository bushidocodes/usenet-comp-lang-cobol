SaaS and  Web Services and getting to grips with objects and layers. Slowly, 
certain legacy code gets refactored or re-structured to run as OO COBOL.

they haven't, they can take solace in the fact that they can still write 
COBOL and it compiles for .Net. Only, performance is crap and if they want 
it to be better they have to start understanding the Framework. That means 
getting to grips with Classes and Objects...)

After about a year the programmers are starting to realise that programming 
objects in COBOL is like wading through treacle. All the hype around the new 
cool Visual COBOL is starting to sound a bit thin. (Most of the "benefits" 
being touted for it come from the use of Visual Studio 2010 and are 
available with the free .Net languages, as well as with the expensive .Net 
COBOL)

Management are starting to see COBOL maintenance in the new environment as 
"expensive" (just like it was in the old one) and upgrades and licenses cost 
an arm and a leg. Then someone has the light bulb over their head light up.

They can get all the benfits of .Net WITHOUT all the costs of COBOL if they 
simply move off COBOL.

All it takes is a one time investment in some training. Programmers LOVE 
being trained in cool stuff at company expense and it improves morale no 
end. The old-timers all scream "Heresy!" and start turning widdershins to

learned about objects and layers at school and had to "unlearn" it to deal 
with COBOL.)

.Net in a language designed for doing that. Programmers find their value 
increases, (and, consequently, their job options...), Management find their 
maintenance costs go down (significantly...), everybody is happy (except 
Micro Focus.)

Visual COBOL fell off the knife edge just as ANY COBOL will . It isn't a

COBOL as it's main attraction. It is a recognition that the COBOL boat has 
sailed. However, during the next few years there is still a market for COBOL 
vendors. It depends how long the market takes to wake up.

I agree completely.

The features were added (and it was a superb job of software engineering) so 
that people would not HAVE to move off COBOL when they went to OO 
environments.

Sadly, the uptake was poor (most COBOLlers had no idea why you might want or 
need Classes and Objects) and by the time people started to think about 
using Objects, COBOL had been overtaken by languages designed for (and 
therefore better suited to) an Object environment. I used Fujitsu Object 
COBOL for nearly 10 years (mainly to write components that could run on the 
web or a desktop) and found it excellent.

Eventually I moved to Java and C# and the scales fell off my eyes. It is 
just SO much more productive and less verbose than using COBOL.

Currently, there are expensive compilers available for COBOL to compile CIL 
for .Net. I believe they are pretty much pointless. If you want that 
environment use a language intended for it. (C# and VB.Net are free...) 
Legacy COBOL can be easily wrapped using InterOP services and there is no 
need to perpetuate non-object code into an Object environment.

In a way, although I greatly admire the people who did it, it can be argued 
in retrospect that adding OO facilities to COBOL was a bad move.

Without it, people would have needed to explore other options which, as it 
turns out, are actually better.

In looking at another reply in this thread (the one that mentions
Veryant's isCOBOL Evolve) it crossed my mind that you may be
constrained in your choice of compiler by the file structures that it
will be required to work with. ISAM files vary in their content from
one supplier to the next; Fujitsu and Micro Focus are not compatible
(in my experience) ie you can't take a Fujitsu ISAM file and run a
Micro Focus program against it and vice versa. Also, if there is one
