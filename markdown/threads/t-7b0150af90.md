EVERYTHING YOU WANTED TO KNOW ABOUT COMP*

*But were too #%&%$ upset to ask


There are several ways that numeric data can be stored on IBM mainframes, and
the specific method of storage affects both program efficiency and storage
requirements. As a result, Cobol allows programmers to specify the way
numeric data is stored by means of the USAGE clause. This is not a related
to the nature of Cobol itself. It is strictly an artifact of the use of Cobol
on specific systems, especially IBM mainframes. Therefore several of the
usage options are not part of the ANSI-85 Cobol standard, but are unique to
Cobol compilers used on IBM or IBM compatible mainframes.

The format for the usage clause is
---------------------BINARY-----------------------------
|-USAGE---------| |-COMP------------------|
|-IS-| |-{--COMP-1--}----------|
…[12 more quoted lines elided]…
|-{--POINTER--}---------|

Each option carries with it specific advantages and disadvantages.

DISPLAY: The default usage, if no clause is specified. The data item is
stored in character form, 1 character for each 8-bit byte. It is the least
efficient method; the farthest away from the format in which the IBM mainframe
does arithmetic. However, since it is the only one that is displayable, it
has to be used for any numbers that need to be displayed on the screen or
printed out.

USAGE IS DISPLAY is valid for the following types of items:
魹ｽ Alphabetic PIC AAA.
魹ｽ Alphanumeric: PIC XXX.
魹ｽ Alphanumeric-edited PIC XXBXX/XX0X.
魹ｽ Numeric-edited PIC $ZZ9.99CR.
魹ｽ External floating-point

魹ｽ External decimal (numeric) PIC 99999. Sometimes referred to as zoned
decimal items. Each digit of the number is represented by a single byte
(composed of 8 bits). The 4 high-order bits of each byte are zone bits and
represent the sign of the item. The 4 low-order bits of each byte contain the
value of the digit. The maximum length of an external decimal item is 18
digits. The PIC character-string of an external decimal item can contain only
9s; the operational-sign, S; the assumed decimal point, V; and one or more Ps
(don't even ask what P stands for!. However, if you are ever consigned to the
7th level of hell魹ｽ it is an old and now neverused method to extract even more
storage. In essence it was an 'implied' zero. Thus PIC 9PPPP could store the
number 90000, yet take up only 1 byte (the 9).)

COMP魹ｽ: Used exclusively with numbers. It defines various methods of storing
and manipulating numbers. The PIC of a computational item can contain only:
9 one or more numeric character positions
S One operational sign
V One implied decimal point
P One or more decimal scaling positions.

There is a fair amount of confusion related to the nomenclature used to refer
to computational items. The representation of the COMPUTATIONAL phrase is
system-dependent and is normally assigned to representations that yield the
greatest efficiency when arithmetic operations are performed on that system.
The VS COBOL II compiler, uses the following terms.


COMP or COMPUTATIONAL
For the VS COBOL II compiler, the COMPUTATIONAL phrase is synonymous with
BINARY. See below for explanation.

These next two are very rarely used, since programming requiring such
precision is better handled by FORTRAN or C.
COMP-1 or COMPUTATIONAL-1
Specified for internal floating-point items (single precision). COMP-1 items
are 4 bytes long. The sign is contained in the first bit of the leftmost byte
and the exponent is contained in the remaining 7 bits. The last 3 bytes
contain the mantissa.

COMP-2 or COMPUTATIONAL-2
Specified for internal floating-point items (double precision). COMP-2 items
are 8 bytes long. The sign is contained in the first bit of the leftmost byte
and the remaining 7 bits contain the exponent. The remaining 7 bytes contain
the mantissa.

COMP-3 or COMPUTATIONAL-3 (internal decimal)
The most widely used storage format. For VS COBOL II, this is the equivalent
of PACKED-DECIMAL. See below.



COMP-4 or COMPUTATIONAL-4 (binary)
For VS COBOL II this is the equivalent of BINARY. COMPUTATIONAL phrase is
synonymous with BINARY.



COMP-3 or Packed Decimal: The most efficient manner of storing and
manipulating numbers for general purposes. Each digit is stored in only 4
bytes (not 8); thus numbers fit in just about half the space on disk, tape,
and memory. Each character position stores 2 digits, except for the trailing
character position, which is occupied by the low-order digit and the sign. A
COMP-3 field can contain any of the digits 0 through 9, plus a sign, and can
represent a value not exceeding 18 decimal digits.
Packed decimal is also the format that the IBM SYSTEM/390 and IBM ES-9000 use
for ordinary arithmetic. If the number is not in this format already, the
compiler has to convert it to packed decimal when you do arithmetic with it.

CODING FORMAT:
When a number is to be coded using a comp-3 or packed-decimal
usage, the PIC format RELFECTS THE SIZE OF THE UNPACKED NUMBER!! It is a
common mistake to give it the size that it eventually takes up in storage.
This however leads to an error. Thus if you have a 6 digit number (123456)
the correct pic format is PIC 9(6) COMP-3 NOT PIC 9(3) comp-3. Your record
size will reflect the smaller amount of bytes that will be taken up, however,
the record description will reflect the unpacked size of the number. Keep
this in mind when determining a record layout for a record that has comp-3
fields.

BINARY: Used for integers such as subscripts and counters. The System/390
will do arithmetic for integer numbers less than 10 digits in binary. Pure
binary fields are different from other numeric formats because they come in
definite increments of memory. The number you code within the parentheses for
a binary field establishes how large a decimal value you can store, but all
binary-stored values take up either a 2, 4, or 8 bytes of storage. Why not use
BINARY for all numbers? It can take up more storage and is still not as
efficient as packed-decimal. They cannot be used for numbers with implied
decimal points since they can not accurately represent numbers to the right of
the decimal point. However, for counters and subscripts, especially if less
the 9,999, it is the most efficient and fastest method.

CODING FORMAT:
Binary fields require alignment on something called a "word
boundary" (it refers to how memory is arranged on the mainframe) which occurs
every 4 bytes. 01 levels always start on a word boundary. At other levels
you should code SYNC to synchronize a binary field on a word boundary.

For example:
01 WS-SUB-1 PIC S9(4) BINARY VALUE +0

01 WS-COUNTS.
05 WS-COUNT-1 PIC S9(4) BINARY SYNC VALUE +0
05 WS-COUNT-2 PIC S9(4) BINARY SYNC VALUE +0
