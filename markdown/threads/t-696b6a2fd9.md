SEARCH checks each table entry until it finds a match - SEARCH stops on a
match.  SEARCH ALL hunts and pecks until it finds (or does not find) that
for which you are looking (SEARCH ALL requires the table to be sorted
first).

SEARCH ALL uses fewer (in some cases, FAR fewer) comparisons than SEARCH.
But until you get a table of several hundreds or thousands of entries,
you'll never see the difference.

> SEARCH ALL uses fewer (in some cases, FAR fewer) comparisons than SEARCH.

... umm ... no ... For most implementations, probably, but not a flat fact.

The standard provides enough restrictions on the table contents so that
doing, say, a binary search within those restrictions is arguably feasible,
but it does *not* require it.  All three standards ('74, '85, '02) use the
wording "a nonserial type of search operation may take place", and in
"standardese" the word "may" connotes permission to the implementor.  Had
the intent been that a nonserial search algorithm was *required* of the
implementor, the word "shall" would have appeared in this phrase.

I know of one implementation of SEARCH ALL that nonetheless does a serial
search, which *always* gives the same results as SEARCH for the same
selection criteria, whether the data is in monotonically-varying order by
the key or not.

In this instance, the implementation of SEARCH ALL once provided a binary
SEARCH, but enough users hated the behavior with out-of-order data that the
implementor had to defeat the feature.  A serial search does, after all,
meet the requirements of the standard.

(As to the user pressure on the implementor in this case, I'm reminded of
the complaints of a college roommate who hated his car:  every time he took
the engine more than 1,000 RPM over the tachometer's redline the fan belt
came off!)

A later implementation from this same implementor did a binary search for
SEARCH ALL (and received a few complaints); analogously, some auto
manufacturers have incorporated electronic engine-speed limiters in their
ignition systems.

    -Chuck Stevens

No, precisely because the issue is behavior of the WHEN condition regardless
of whether the table is in order or not; the next one in the table could be
*less* according to the key (contrary to the rules for SEARCH ALL).    And
if there are duplicates according to the key (also contrary to the rules for
SEARCH ALL),  this implementation always points to the first of those
duplicates.   This particular implementation provides defined and
predictable results for those cases that the standard describes the results
as undefined or unpredictable.  That, along with providing the results that
the standard defines as being neither undefined nor unpredictable given the
rules of the standard, is precisely what the users wanted -- *defined*
results with garbage data as well as with good data, instead of *undefined*
results with garbage data and good results *only* with good data.

For an unsuccessful SEARCH, the standard says the index value ends up
outside the range of permissible occurrence values; for an unsuccessful
SEARCH ALL, it says the index value is unpredictable ('74, '85) or undefined
('02);  in this instance the unpredictable/undefined value it ends up with
happens to be the same as for SEARCH.

> is there any ways to tell SEARCH to not increment idx (my index) further

I've seen the folks that show you implementing Occurs Depending On (ODO) 
structure, and that would work.  What I do is make sure that the table 
is initialized to spaces before you begin, and then add the lines in all 
caps. (They don't have to be - I just did it that way to set it off from 
your code.  I'm also assuming that "nom" = "name".)

-=-=-=-=-

    set contrat-false to true.
    set idx to 1.
    search ouvrier
      when num-contrat(idx) = rec-num-contrat in rec-recolte
         set contrat-true to true
      WHEN NOM (IDX) = SPACES
         CONTINUE
    end-search.
    if contrat-trouve = 1 then
        add rec-poids to recolte(idx), total-recolte,
           total-recolte-jour
    end-if.

-=-=-=-=-

This will cause the Search to exit (as it's satisfied a When condition), 
and "idx" will be set at the first "hole" in your 20-occurrence table. 
(Of course, for ODO, you'd just add one to "bs" and you'd have the first 
"hole" as well.)

Good question. The answer is yes, it looks at all 20 if the worker is not found.
There are two ways to stop this. 

.. 01 ouvrier occurs 1 to 20 depending on bs indexed by idx.

.. search ouvrier
           when idx > bs
               set contrat-false to true
           when num-contrat(idx) = rec-num-contrat in rec-recolte

>	sort ouvrier on descending key recolte.

Yes to both. Your program looks excellent for a neophyte. Two minor stylistic
suggestions:

Change:
     perform aff-recolte until idx = bs
To:
    perform aff-recolte varying idx from 1 by 1 until idx > bs
    (delete initialization and set up by 1)

Change contrat-trouve to filler and change the test after the search to 'if
contrat-true'. Using both data name and 88 is bad form. If your compiler
permits, say 
    88 contrat-trouve                   value 1 false 0.

    when idx > bs
               set contrat-trouve to false
