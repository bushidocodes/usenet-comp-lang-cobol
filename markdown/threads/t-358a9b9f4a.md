I'm pleased to report that these pages are now working in Firefox.

Iframes work the way I described in this thread. You DON'T need to define 
the frame (other than with the <iframe> tag), neither do you need to confine 
the content to HTML alone.

If you visit the site you will see plain text and diagrams all running 
through the iframe, being dynamically generated on the fly by server side 
code. There is no need to amend the content files or prime them in any way, 
neither is it necessary to define framesets or frames for an inline frame.

The problems I experienced were not due to "bad HTML" or "no HTML", they 
were caused by differences in interpretation of a standard by MicroSoft and 
others. I added some "Firefox friendly" Classes to my CSS and everything 
fell into place. No changes were made to code (other than some slight 
simplification I did in an unrelated area.)

The W3C definition which Richard quoted here is absolutely correct. It just 
doesn't apply to HTML alone. I believe they implied that when they said it 
covered OBJECTS. It does. But this demonstrates how a standard can be open 
to interpretation.

I have to say that Firefox is more demanding than IE and it is more of a 
pain to write for because it requires options in tags that IE will default. 
Nevertheless, I'm happy with the results and especially pleased that (now I 
know what it needs and what CSS options it enjoys) I can make sure what I do 
in future is more Firefox friendly.
