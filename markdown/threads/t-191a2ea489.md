I've used .EXEs with MF 16-bit Cobol for many years.  .INT is
intermediate code (byte coded) and can be executed using the run-time
system on any machine (in theory) regardless of processor type or OS, as
long as a run-time is available.

.GNT is 'generated' with much of the code converted to machine code for
the specific processor, but it still requires the run-time which may be
for any OS on that processor type.

.EXE makes the program only for Intel and DOS or Windows or OS/2
(depending on link options) but it doesn't require the run-time system
(cobrun), it may need a support module such as COBLIB.DLE or similar, or
this may be statically linked in.

The _content_ of an .EXE may be byte coded (equivalent to .INT), or
generated machine code and optimised (equivalent to .GNT).

The advantage of not using the run-time system (in MF 3.x and 4.x) is
that you don't need to deploy it.  .EXEs (with some additional
restrictions) may be distributed free.  This is not available in the
latest Net Express as I understand it.

The disadvantage of not using the run-time system is that you may need
to do some simple memory management, and some esoteric facilities may
not be available.

Two other advantages of using a runtime

1. The common (library) code need not be included in each program, saving
disk space. (Which today, of course, means nowhere near what it usedto
mean).

2. If a bug is discovered in a common routine, you fix it and change the
runtime, install it and you are done. There is no need to recompile/relink
every program in the system which uses that routine. (Or for that matter,
*identify* every program in the system which uses that routine).

"Runtime" here is not limited to the runtime support module(s) provided by
the compiler vendor: it can include common I-O or validation routines
developed by the user and used in multiple programs within the system.

"Michael Mattias" <michael.mattias@gte.net> wrote

> Two other advantages of using a runtime

In MicroFocus there are several 'runtime's.  

The "Run Time Environment" (cobrun) or "Application Server" will
execute .int and .gnt and provides portability.  This is what is
usually meant by 'runtime'. This provides memory management,
libraries, application level control and several other things that I
don't care about.  It is chargable per user.

The 'Static linked run time support' where the program is linked to
form one .EXE. This is 'free' in MF 2.x thru 4.1 (with certain
restrictions) and may be in older versions of NE as long as some rules
are followed.

The 'Shared run time support' where an application can be many .EXEs
(DOS) or .DLLs (OS/2 or Windows) that are CALLed and CANCELed. This is
'free' as above, but each user should check their actual licences and
documentation.

The Microsoft versions of MF Cobol did not have the "Run Time
Environment" and all programs had to be built static or shared, but
there were no runtime fees either.
 
> 1. The common (library) code need not be included in each program, saving
> disk space. (Which today, of course, means nowhere near what it usedto
> mean).

No. Using the shared support module achieves this.  In MF 2.x to 4.x
the support code can be in separate loadable files (eg COBLIB.DLE,
EXTFH.EXE, ADIS.EXE, SP2.EXE).  There is a small amount of code linked
in to each EXE/DLL but this is not significant.
 
> 2. If a bug is discovered in a common routine, you fix it and change the
> runtime, install it and you are done. There is no need to recompile/relink
> every program in the system which uses that routine. (Or for that matter,
> *identify* every program in the system which uses that routine).

No. Programs can be completely dynamically loadable from separately
CALLed and CANCELed .EXE or .DLL files exactly as if they were .int or
.gnt.

Of course you _can_ statically link everything into one large .EXE if
this is required.  Even then there is no need to recompile everything,
only those source s that have changed and then relink all the .obj.

> "Runtime" here is not limited to the runtime support module(s) provided by
> the compiler vendor: it can include common I-O or validation routines
> developed by the user and used in multiple programs within the system.

Yes.  Each separately compiled source code can become a .EXE or .DLL
that is loaded and discarded on CALL/CANCEL.  Common routines can be
replaced individually without needing to recompile/relink anything
else (unless there are interface changes).  Only the newly changed
files need to be redistributed.

When you use "dynamic" calling of programs the MF documentation seems 
to indicate that you cannot have more than a single program linked 
into an EXE or DLL. This is not true.

You can link as many objects as you want into a DLL. In the 
initialization logic of the program that will use the routines in the 
DLL you should use "SET xyz-dll-pointer to "xyz.dll" where 
xyz-dll-pointer is a procedure pointer. This will load the module 
names so that dynamic calls will locate the subroutines.

WARNING - if you CANCEL the first subroutine called from the DLL the 
DLL will be unloaded from memory and further calls to the subroutines 
in the DLL will fail.

You can also link as many objects as you want into an EXE

You can place multiple programs in a single source code file. See the
entries in your compiler's manual under:

PROGRAM-ID. PROG1.
..
END PROGRAM PROG1.
PROGRAM-ID. PROG2.
..
END PROGRAM PROG2.
PROGRAM-ID. PROG3.
..
END PROGRAM PROG3.

This will (should!) put all three programs into a single executable. You can
then CALL 'literal' any of the programs from within the resulting
executable.

You can also play with the nesting...

PROGRAM-ID. PROG1.
..
PROGRAM-ID. PROG2.
..
END PROGRAM PROG2.
PROGRAM-ID. PROG3.
..
END PROGRAM PROG3.
PROGRAM-ID. PROG1.
..
END PROGRAM PROG1.
PROGRAM-ID. PROG2.
..
END PROGRAM PROG2.
PROGRAM-ID. PROG3.
..
END PROGRAM PROG1.


When you nest programs like this, there are some limits on what can be
called from where, as well as some data-sharing considerations when GLOBAL
and EXTERNAL datanames are used.

Someone gave you the a sequence to to this with the Microfocus compiler;
with the IBM mainframe compiler the the linker is also used, except for the
most part the required linker commands are created by the compiler.

But using "multiple programs per load module" is not quite the same as
creating dynamically-called modules. I have used multiple programs per
source code module for forms-printing applications when the input file may
be one of two or three seperate versions. Instead of two separate programs,
one program reads the input and decides what the format is, then CALLs the
appropriate output routine using an EXTERNAL file connector. These
applications do not have any 'reuseability' requirements, in that the code
only does one thing and no other programs will ever need access to the code,
so for ease of maintenance I just put everything in one place.

That is, the exact 'architecture' you choose will be application-dependent.

Use an EXE with DLL's -- this is likely to be the most efficient footprint
you will have for executing your application. GNT's perform well (they are
processor code), but are not shareable -- i.e. if you have multiple
instances of A.GNT you will have duplicate copies of the code in memory.
