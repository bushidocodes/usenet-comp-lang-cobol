In GENERAL, I try and get people to IGNORE any similarity between GLOBAL and 
EXTERNAL and look at each individually.   I don't know if this helps (or not), 
but:

GLOBAL
  - applies (basically) only to the "visibility" of a user-defined word.  It 
makes that "name" available to ALL nested programs (nothing else) - but MAY be 
"overridden" by a local definition or by another GLOBAL definition at an 
intermediate level.

EXTERNAL
   Applies to the "data area" itself.  When an item is defined as EXTERNAL in 
any two programs, then it makes the single data area "shared" (like linkage 
section) between the two programs.  If you want to "compare" it something, think 
about the (old) SAME RECORD AREA concept.

The only "addition" to the above, is GLOBAL in a DECLARATIVE and that is such a 
mess (IMHO) that I would put it LOW in the list of things to think about.

  ***

Does this help?

Using the term 'program' may confuse rather than illuminate.  EXTERNAL
creates a shared data block between compile units within a single
run-unit.

GLOBAL makes the name visible to all programs nested in the program
declaring it within a single compile unit (unless, as you say, it is
overridden with a local name).

Data items may be both GLOBAL and EXTERNAL.

> If you want to "compare" it something, think
> about the (old) SAME RECORD AREA concept.

I am not sure that is a good analogy, that is more like a REDEFINES but
across FDs.
