```
Some of our customers have been experiencing mysterious file
corruptions with our application written with CA Realia Cobol.  The
problem have begun since we upgraded from DOS based version 4.102 of
Realia to 32-bit Windows version with the Workbench product (a variety
of version in the 6.0.XX range; 3.0.XX for Workbench).  A majority,
but not all, of the corruption problems seem to be with networks
running NT Server.
 We recently upgraded to Workbench 3.1 (3.1.63) with Realia version
6.0.93.  We recompiled & linked all programs and distributed them on a
test basis to a couple of customers.
 One customer in particular has experienced, both before and after the
3.1 version, massive file corruptions.  It begins with one workstation
getting 'invalid page fault' message(s).  Soon other workstations get
similar messages.  Before long, the NT server does not respond.  We
shut everything off, normally if possible - or just shut them off if
they're not responding.  When everything is brought back up, multiple
files are corrupt.  Based on what we believe people were doing at the
time, we have files that end up corrupt that were not even in use.
 Our application uses file locking.  Therefore it is very good about
holding files open only for the periods of time it needs them.

 Has anyone else had similar situations?
 Does anyone have any idea what could be happening?
 Does anyone know for absolutely, positively, 100% sure that using
Realia's Realcopy utility with the /rads swtitches will recover every
record within a corrupt file?
 Is there any way to 'normally' close files that are left hanging open
when an application or computer quits responding?
```

I vaguely recall something along these lines being dicussed here many moons
ago.  It seems that the problem then had to do with a bad mix of internal file
locking and Windows 'helping' with some new 'file sharing and locking
semantics'.
IIRC, the solution was to go into MyComputer|Performance|Advanced Settings ...
File System|Troubleshooting then click to check DISABLE new File Sharing and
locking semantics.

I don't know if this would be your solution, but it might be worth a try on a
couple of the 'problem children' to see if the problem gets any better.
