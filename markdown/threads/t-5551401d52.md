Howard,

There is a lot of semantical confusion with the word 'merge'. In COBOL and
external sort sense (SyncSort, DFSORT) 'merge' actually means 'interleave'.
That is, two ordered files with the keys like

1, 2, 3, 5, 7,   and 2, 4, 5, 6, 7, 8

will result in

1, 2, 2, 3, 4, 5, 5, 6, 7, 7, 8

unless unduplicated. OTOH, 'merge' is often used in the sense 'match merge'
meaning that the keys above, when combined with the request that only
matching keys are to be output, will result in

2, 5, 7.

MERGE in the interleaving, i.e. COBOL or SyncSort, sense can be quite useful.
It takes a lot less time and resources to interleave a number of already
sorted files than to sort an unsorted concatenation. Hence, whenever a very
large file cannot be sorted as a whole because of resource limitations, it
can always be split into fragments small enough to be sorted, and the
fragments can be sorted and interleaved. Likewise, if one has a number of
already sorted files and wishes to obtain a sorted copy of their
concatenation, concatenating them and sorting the whole is a most irrational
thing to do - MERGE will do it much faster and more efficiently.

Note that in SAS, the MERGE instruction _never_ means 'interleave', but  a
kind of 'horisontal' file combination, which can be a  match-merge in
particular. People working in mixed COBOL/SAS shops must be aware of this
circumstance.

Merge can be very handy for dealing with previously sorted data. The
advantage of using sort/merge facility is that you can simplify your
program as all the matching logic doesn't need to be procedurally
coded. You also gain the same file i/o efficiencies as you do with
sort input files and no intermediate sortwk files are needed. I've
benchmarked on large (multi-million record files) that they run
several times faster using MERGE that they do if I coded the match
logic in my program. 

There are also a  couple of useful  exits that allow records to be
reformatted, dropped or additional records generated  prior to the
merge that can be used in same manner as an input procedure if you
need this.

One place where this was very handy was when we got financial journal
files. We kept a utility deck around to merge them and it was a set
and forget operation. 

This was also a very easy way to create daily/weekly/monthly files for
archival purposes. 

For another example, It wasn't in COBOL, but I used merge a lot when I
was doing MVS performance work to bring together SMF, RMF, EREP and
other performance data from multiple systems for analysis.  This was
necessary because the volume of data could be truly huge. 

Since this data was stored in a timestamp sequence, it was easy to
write merges that had include/exclude control cards  to only bring in
certain records and pass them through for additional processing with
minimal programming.  I could include as many MERGEINxx DD statements
as I had systems to merge in that particular run (usually between two
and six systems)  and change it whenever the number of systems being
included changed. Obviously doing this in a COBOL program would
require logic changes to do this since files to be merged couldn't be
concatenated.
