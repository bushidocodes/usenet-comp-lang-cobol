```
One thing that it is important to remember is that there is NO GUARANTEE
that two 01-level items with the EXTERNAL attribute will actually be "next
to each other" in physical storage.  In IBM mainframes, this is almost
guaranteed NOT to work.  With some other environments, this will work
SOMETIMES.

If your program logic "requires" that you have a large table with the 1st
part "unchangeable" by any application - and the 2nd part changeable in
specific programs, then you might try having an EXTERNAL table with the
"constant" part and a 2nd table defined large enough to hold both the
constant and changeable parts.  Then as the 1st part of your application
code, copy the constant part to the beginning of the larger table.

As Richard has indicated in another post, you should make certain that each
element of the table is the SAME SIZE.  Otherwise, you have a totally
different programming task.
```

> Anyway I would like to know if there is a solution :)

Write the code to move from the table in a copy book EXTERNALPD.CBL and
change each program to PERFORM Get-EXTERNAL which does whatever is
necessary.

or: write a callable subroutine which loads and holds the field data and
can be called to return the field.  eg:

CALL "extfield" USING "G" W-X Wext-Field

where "G" is get and "S" is store and "L" is load or similar.

The issue here is not what you need to do _this_ time but what would be
needed _next_ time the size changes.  

Two tables and hard coding 20 and 200 into 125 programs means that this
needs to be redone to all those the next time.  

One table of 220 plus putting the limit value in that copy book, or
doing the access in a copybook too, means that the 125 programs only
will need recompiling.

Putting the whole thing in a separate subprogram means that only that
need change in the future.
