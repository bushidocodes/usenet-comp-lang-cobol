Had an odd question from a co-worker today.

First, we have two Z900 model 2064's.  We've been on OS/390 2.10 since
last summer, and expect to upgrade to z/OS 1.4 next summer.  Current
compiler is IBM COBOL for OS/390 & VM 2.2.  We still have production
programs executing that were last compiled with VS COBOL II and COBOL
for MVS & VM.  We're pretty sure that we do not have programs running
that were last compiled with OS/VS COBOL.  And we only run LE/MVS as
the runtime library.

The programmer said he had been told that our COBOL programs will
abend when we upgrade to z/OS 1.4, if they do not close VSAM and QSAM
files.  I can't remember if this has been discussed in
comp.lang.cobol, and I am wondering if this is true.

Apparently, this programmer NEVER closes VSAM or QSAM files in batch
programs, and trusts COBOL, or the runtime environment, or the
operating system to do it for him.  I always close all my files before
terminating a batch program, and I do not endorse his shortcut.

Does anyone know if this is true?  Will my co-worker be forced to fix
all his programs to do what he should have been doing anyway?

The technical issue is where are the DCB's/ACB's located, who controls the
storage, etc.

You will get abends if:

1. The DCB/ACB is allocated in dynamic storage  and
2. LE is aware of the storage but is not aware that there are DCB's there  and
3. LE explicitly frees the storage at termination  and
4. The COBOL runtime is not aware that the program is ending and files need be
closed

I doubt that this is true, but it is possible.

At any rate I would suggest explicit closing of files. This would help if
there is an abend after the file should have been closed because when an abend
causes the CLOSE there may be some lost writes.

Well, ....

Arnold, - as I recall, your site (for various performance reasons) may use
Assembler drivers or for various "functions".

In an all COBOL application, you will NOT see any ABENDs when you STOP RUN
(or GoBack from a main program) with OPEN files.  This is because ANSI
*requires* COBOL to work this way.  However, if you have used Assembler to
do various things, it *sometimes* possible to see such ABENDs.

If your application programmer doesn't believe this is the way COBOL works,
have them check out

 http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/igylr205/6.2.36

which says,

 "The STOP RUN statement closes all files defined in any of the programs
comprising the run unit."

P.S.  I agree with those who say that this is a BAD feature to rely on.
Explicitly closing a file is ALWAYS a good idea.

P.P.S.  There is a separate "question" of file status "97" when your program
OPENs a VSAM file that wasn't "properly" closed by the last application that
used it.

deliberately ABEND a batch COBOL program, I try to close any open VSAM
files before forcing a storage dump.  It might save a verify on the

Here's the whole story.  Another department in my company provides
financial software to banks in COBOL source format.  Our COBOL source
program does not close the files (although it WOULD if I had written
it), instead relying on STOP RUN to close files automatically.  

One customer upgraded to z/OS (I don't know which version) and got C03
ABENDS.  Naturally, they blame us.  Our applications programmers tell
our Sysprogs "it's always worked before, so it must be a problem with
z/OS".  I don't know which version of COBOL was used by the customer,
or whether the program was recompiled for z/OS, or what runtime
library they use.  But I do know that our source code calls
subprograms dynamically.  It's even possible that some files are
opened in a dynamically called subprogram.

Our of our sysprogs researched IBM APARs for this, and his advice to
the applications programmers is to change the COBOL program to
explicitly close the files.

Here's the text of the IBM APAR which was closed 11/21/1990, and last
updated 09/25/2002.  Note also that in some cases a file cannot be
closed, or a file can be closed but the last buffer may not be

programming, it is recommended that all APPLICATIONS that open

  ERROR DESCRIPTION:
  System C03 abends occur at task termination when a program opens
  a data set and DOESN'T CLOSE it before returning to the system.
  This can be verified with REG0 (REG 0) at the time of the CLOSE
  (SVC14) in the trace table.  If REG0 = E3C3F0C1, then the close
  was issued by IFG0TC0A and if there are no previous errors, the
  ERROR IS IN THE APPLICATION.
  See also II02095 for additional information.
  *************************************************************
  NOTE:  OW00185 fixes problem where ABENDC03 is issued with NO
         IEC999I message
  *************************************************************
  .
  As part of task termination, RTM will free user subpool storage
  that was still allocated when the program terminated.  The DCB
  for the open data set can reside in that storage.  When the data
  management resource manager IFG0TC0A detects the open data set,
  it program checks when it attempts to access the DCB.
  IFG0TC0A returns to RTM with return code 4, who then issues
  the ABENDC03.
  -NOTE--
  RTM does nothing with datasets at all.  At task termination, RTM
  calls a DFP module, IFG0TC0A.  IFG0TC0A runs the DEB chain for a
  terminating task, closing each dataset as it comes to it.
  If any CLOSEs don't work, IFG0TC0A gives RTM back a bad return
  and RTM responds by issuing the abendC03.
  When the ENTIRE PAGE occupied by the DCB(or ACB) has been free'd
  CLOSE can't work and the abendC03 is inevitable.  If, however
  there is anything else still getmained on the same page as the
  DCB and the area the DCB is in hasn't been reused, the residual
  information on that page would be sufficient to close the
  dataset and the abendc03 does not get issued..
  In all events (even when the abendC03 is issued) the dataset is
  "cleaned up."  The difference is, when an abendC03 (and its
  corresponding msgIEC999I) is issued, you may not get the last
  buffer written to the dataset.  The dataset treatment itself is
  not inconsistent, but the environment in which this treatment
  is occurring may not be quite the same.

I hate to say this, but the APAR does not correctly state
what the problem is and then goes on to say lots of stuff
that unless you work in "systems", probably won't make a lot
of sense.

The DEB chain is run during task termination (because of
"normal" termination or ABEND). This is done to ensure that
all files are closed.

When a DEB is found to be OPEN but the storage to which it
points (DCB, ACB, etc.) is NOT available, the only choice
here is the ABENDSC03.

This is not something new with z/OS. This may be the result
of something different about the LE runtime, or DFSMS, or
some other item that the COBOL program is using that causes
the DCB/ACB storage to go away. But it is not new with z/OS
because I used to force this type of thing to happen to test
the behaviors of certain recovery routines at one time (this
was at MVS/SP3.x). I've also seen this same thing happen in
MVS/SP1 (what you might call MVS/370).

Typically, I would get this by doing a GETMAIN for storage,
build a DCB in that storage and then "forget" to close a
file before doing the FREEMAIN. Task ended with ABENDSC03
regardless of anything else I did.

Your programmer(s) has implemented a bad design, COBOL
development has issued warnings about this behavior, and now
they are getting caught by this. This is similar to the
problems in the VSE world where one had access to the I/O
areas before OPEN and after CLOSE for release after release.
People were warned that this would no longer be a valid
operation as DOS moved to be closer to the behavior of VM
and MVS.

The COBOL compiler can't necessarily make you put a CLOSE in
your program(s) for a file as you can use a subroutine to do
some kind of action(s) on files defined within the calling
program (OPEN/CLOSE, READ, WRITE, DELETE, etc.).

I think you need to tell your people that it is now time to
pay the piper.
