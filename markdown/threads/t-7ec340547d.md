```
I had occasion recently to prepare an OO COBOL class as a COM component,
available to software that can implement that interface. The code would
be generated by a PRIMA tool for specific use as a Database handler (DAL
object), so it needed to be thoroughly debugged before investing into
the generation of it.

The component would be called from normal COBOL, OO COBOL, and
PowerCOBOL, or, indeed, any language that implements the COM interface,
but the immediate requirement was for it to be invoked from PowerCOBOL.
(A Fujitsu Native code compiler which allows a GUI interface with COBOL...)

Because the DAL object is not standalone it has to be invoked by
something, so I wrote a "harness" for it in Fujitsu NetCOBOL.

It was very satisfying to step through the harness code then step into
the DAL Object code when it was invoked, and this was really useful for
checking the state of passed fields, the logic paths of various methods,
and so on.

Then I tested it from PowerCOBOL, and, to my horror, it fell apart when
it was required to step from The PowerCOBOL scriptlet into the DAL object.

The source code was double spaced and the exercise was useless.

A colleague in Spain explained that the DAL code lines have to be within
80 characters so I reformatted it to comply, and tried again.

This time, on entering the DAL code, it was formatted correctly, and the
path of control could be traced easily by stepping through it, just like
the results from the Harness above.

BUT... I could not view the contents of any fields! (It wouldn't even
allow me to set a watch, although the facility to set it was there...)

This makes the debugging exercise pretty useless...

In the end I had to write an "AuditDump" method into it and put my own
displays into it... pretty ugly.

So, if anyone here is using PowerCOBOL and you have a scenario where you
want to debug into NetCOBOL, have you encountered this problem and how
did you solve it?

TIA,

Pete.
```
