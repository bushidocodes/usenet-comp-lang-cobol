```

 
> John points out that in his view, there is no perform
> section strategy.
 
> If section names were for overlay purposes, and section
> numbers were specified (long gone and ignored now I guess),
 
No, I still have some large programs overlayed under DOS,
actually under a Multiuser, multitasking OS that runs MS-DOS
type programs.
 
> then perform a section would include planning to have the
> available memory allocated,
 
The whole point of needing to overlay a program is that there
_isn't_ any memory available from which an allocation can
be made.  When the program is loaded a single overlay buffer or
area is created into which each overlay may be loaded in turn.
 
> and rolling out that area if needed,
 
This is code only, it does not need to be rolled out because
it should not have changed.
 
> then reading the section into ram for execution.
 
Yes, this is needed.  But the section must be loaded into
RAM at some point.  Unoverlayed, every section is loaded
when the program starts.  Overlayed, this is deferred until
the section is needed.  If you get the overlaying _right_
each section is only loaded once.
 
But you are mistaken in that the overlaying of sections is
related to performing of sections.  The section is loaded if a
paragraph within it is performed from outside.  In this case the
section is used as a container for that group of related
paragraphs that should be loaded together.
 
In particular it would be a mistake to have:
 
           PERFORM Process-Transaction
               UNTIL End-Of-File
           ....
 
       Process-Transaction SECTION 77.
       Start-Process.
 
           READ ...
           ......
 
As this would cause the overlay to be reloaded for every read.
In this case it is far better to have:
 
           PERFORM Process-All-Transactions
           ....
 
       Process-Transaction-Overlay SECTION 77.
       Process-All-Transactions.
 
           PERFORM Process-Transaction
               UNTIL End-Of-File
           .
 
       Process-Transaction.
 
           READ ....
 
If the program is carefully structured and assembled then the
overhead of overlaying need not be significant, or even
noticable.  It can be hugely more efficient than, say, dynamic
CALL and CANCEL.
 
But those with a fixed idea about a program should read and
be laid out in the source file may find many inefficiencies.
 
 
> Reducing sections to a method for simplifying perform through,
> etc. in my view has taken the wrong fork in the road. Perform,
> Perform through, and the new perform (immediate) seem easy
> enough to understand even for a slow learner like myself.
 
But why do you think that you need PERFORM .. THRU .. ?
```
