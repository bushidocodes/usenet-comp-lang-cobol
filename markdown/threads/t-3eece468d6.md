version, ANSI-74, ANSI-85 ANSI-89, and soon a new one. There is some

I have work with the IBM compilers and Microfocus compilers. If You keep

I think you are missing a fundamental step. Just because a vendor has a
COBOL85 compiler does NOT mean that they have limited themselves to ANSI
syntax. They all (or almost all) have extensions. The stuff that you
haven't been able to port from one ANSI compiler to another was all
extensions. For each of your compilers (and I know this is true for OS/400)
there is a compiler option that lets you flag when you are using extensions.
I think that if you tried compiling your program on one platform and got NO
extension flags, that you would find it would compile cleanly the first time
when you went to another Standard conforming compiler. (In most cases, it
would even run correctly the first time).

I am *not* saying that the use of extensions is "bad" - I am simply saying
that Standard COBOL is (almost) totally portable - but what you are coding
is using stuff that is not part of the Standard and very definitely does
change from vendor to vendor and platform to platform.

For example, both of your DISPLAY statements should get flagged as
non-Standard *if* you turn on your extension/standard flagging mechanism.

While I do agree that the use of extensions is not inherently
"bad".... I always caution programmers on the use of extensions
because they do indeed make the code less portable.

Many programmers who wrote Wang COBOL in the past are now forced to
either use emulation software or re-write a significant portion of
their code, because they used all the wonderful Wang COBOL extensions
which are not supported by any of the major COBOL compilers.

I suppose that I'll get beat up for voicing my opinion, but it is
indeed my opinion that the safest way to ensure that your code will be
portable across different compilers is to stick to the ANSI standard
as much as possible.

Our shop tried to ameliorate the portability problem by modularizing the code in
such a way that platform or compiler specific usage would be contained in a COPY
library or an object library. All changes would be made on one platform and
then the changes would be ported to the others. The only time the job became
difficult was when you had to change one of the platform/compiler specific
sections.

Perhaps an example would illustrate my point a little better. At one time we
had only one compiler that had an extension to get a long date
(YYYYMMDDHHMMSSTTHHMMD), all the others only had YYMMDD available. All programs
would have a WORKING-STORAGE entry something like:

COPY "WS>DateTime.cob".
With the COPY containing something like:
01 WS-DATE-TIME.
03 WS-DT-YEAR PIC 9(04).
03 WS-DT-MONTH PIC 9(02).
03 WS-DT-DAY PIC 9(02).
03 WS-DT-HOUR PIC 9(02).
03 WS-DT-MINUTE PIC 9(02).
03 WS-DT-SECONDS PIC 9(02).
03 WS-DT-HUNDRETHS PIC 9(02).
03 WS-DT-OFFSET-HOURS PIC 9(02).
03 WS-DT-OFFSET-MINUTES PIC 9(02).
03 WS-DT-OFFSET-DIRECTION PIC X(01).

Then in the PROCEDURE DIVISION you would have a statement like:
PERFORM 90100-GET-DATE THRU 901ZZ-GOT-DATE.
and a COPY library defining the 90100 paragraphs and containing the appropriate
code to return the values to the common WORKING-STORAGE items.

Anyway, as long as you are aware of the usage of extensions and other
non-standard constructs, you can simplify moving to other platforms by
segregating those items into more modular groupings.

B
