Hello all,
I'd like to get some idea of what naming standards other COBOL
programmers use in section names.

For about 12 years now, I've used the A-INITIALISE, B-PROCESS method,
where MAIN PERFORMed B-PROCESS which in turn PERFORMed BA-DO-THIS and
BB-DO-THAT, which performed BBA-WHATEVER. Common routines were
Xx-SOMETHING and I/O modules were Zx-function. I used this successfully
in PL/1 installations as well. It's main advantage for me was when the
program abended, in whatever section, the PERFORM path was immediately
obvious!

I've now been asked to put as EXIT! on the MAIN Section, not use STOP
RUN ("it causes problems"?) and this numbering method. A section
performed from MAIN is now Cnnn, which performs Ennn, which performs
Gnnn, etc. I/O routines are Xnnn and common is Unnn. As it is possible
for an Xnnn section to perform a Unnn section, the program is
effectively performing code "above it". I understood that was not
desirable.

Not surprisingly, I'm having a lot of trouble coping with this naming
standard.

Comments, commiserations, sympathy?

Thanks,
Steven.

I've seen/used a number of section/paragraph numbering schemes over the
years. Most of them fall into three general classifications, each with
associated advantages and disadvantages. I'd classify them as follows:

The first type numbers each subordinate module according to it's "owner"
with the mainline being either "000-MAIN" or just "MAIN". Thus, the three
subordinates of "000-MAIN" are "100-xxx", "200-xxx" and "300-xxx". This
scheme is basically what Steven describes below, except that he has been
using letters (B, BB, BBA), instead of numbers (200, 220, 221). There are
variations on this scheme that always use numbers of a certain length, or
that breakout "utility" modules into a separate numbered/lettered section
(as with Xx-SOMETHING that Steven notes). The advantages here are as
Steven notes, i.e. primarily tied to the logical placement of modules
within the code listing itself.. The disadvantages are that when modules
are inserted between levels the numbering is no longer in sync; and that
there is no tie back to any other documentation (i.e. design charts).

The second type numbers each subordinate module according to where it falls
on the graphical representation of the program design chart. In this
scheme, the top level is Annn-xxx, the second level is Bnnn-xxx, the third
Cnnn-xxx, etc. The "nnn" are assigned going across the rows of the
structure chart. Some variations on this are using A, C, E, etc as Steven
mentions (allowing for the addition of a level at some future point, and
the breakout of "utility" modules into a separate level (Xnnn is I/O as
Steven mentions). The advantages/disadvantages for this are exactly the
opposite as in the first type, namely that it is easy to tie modules back
to the design charts, but not as easy to locate the calling path or module
relationships from the code itself. This is OK, as long as the
documentation exists and is kept up-to-date.

The third type, which is not as common as the two above, assigns each
module a unique number according to the type of module it is. Thus, the
module which loads a table into working-storage is always given the same
number in every program, regardless of where it falls in the hierarchy.
Primary advantages of this type is that you do not have to rely on the
design at all for choice of module names, and moving common modules to
COPY libraries is greatly facilitated.

There are no clear winners here. Each type of numbering scheme has its own
distinct advantages and disadvantages. Any, of course, as with most
"standards", what you are used to always seems the best.

Finally, in regard to the other issues Steven mentions, the use of EXIT
PROGRAM vs. STOP RUN vs. GOBACK (for the IBMer's among us) is not that they
cause problems. Strict conformance to structured code dictates that any
such "exit" verb always be coded in the mainline module only. Otherwise,
if the STOP RUN were in the 300-TERMINATE module for instance, you are
"guilty" of performing a module which does return control to you. I
personally always code this verb in the mainline for that reason so I don't
have to worry about any possible implications of coding it elsewhere, but I
cannot think of an instances where it ever "caused problems". The actual
choice of EXIT PROGRAM vs. STOP RUN is actually dictated by whether the
program is a subroutine (must use EXIT PROGRAM), or a main program (must
use STOP RUN). GOBACK has the advantage in that it operates like a STOP
RUN if the program is a main program and like an EXIT PROGRAM if the
program is being called as a subroutine.

Also, performing paragraphs "above" you (Xx-xxx performing Ux-xxx) is not a
problem. The placement of modules within a source programs is of no
concern to the computer. The only possible issue (which is actually an
issue in any numbering scheme) is that when memory paging is a constraint
the "locality of reference" can play a role in system performance. But
since COBOL programmers have so little control of where code falls relative
to page boundries, it's not something that is usually of concern.

Hope this help.

› Using sections in the Procedure Division is an abomination.

Errm - little strong maybe? :-)

I have had to amend programs written without the use of sections. They
are a bloody pain : because you don't have the "get-out" option of a
branch to the end of a section, a small one-line amendment can lead to
putting most of a paragraph within an IF statement.

Say for example I had a paragraph that filled all 40 fields in a
record using straightforward MOVEs - 40 of them. Fine - easy to code
using either paragraphs or sections. However, my boss then tells me to
do a "SMALL" change to your program - only do the last 35 MOVEs if a
particular condition is true. Sounds easy, doesn't it? Not your way -
I would either have to stick 35 move statements in a monolithic IF
statement, or create a new paragraph containing only the 35 MOVEs, and
perform that if my condition was true. However, by putting the MOVEs
in a SECTION on their own, I would only have to put an "IF-THEN GO TO
END-OF-SECTION" at line 6, and I have done my amendment. This doesn't
lead to "spaghetti code" - the GO TOs are always to the same point in
the section - the end.

I can't think of a reason why you would want to straight-jacket
yourself by leaving out a very useful part of the COBOL language.

Having said that - if you want to do it your way, then fine - I
wouldn't dream of saying that your way was an "abomination" - there is
more than one way to arrive at good code, although it helps to start
with a good coder ;-)

cheers

I would add: check your compiler. If you're sorting and using a
COBOL-85 or beyond compiler, then you don't need to use SECTIONs at
all. If you're using a COBOL-74 or eariler model, then make every
paragraph a SECTION with comments early in the code suggesting
removing the offending word at the next compiler upgrade.

To quote James G. Janossy:

"SECTIONs are unusual and complicate program maintenance, so it is a
very good idea to abandon them now. SECTIONs physically couple
together all the paragraphs following the SECTION heading. If extra
code is added into or between these paragraphs, it can be executed
unintentionally.

"A SECTION is an awkward construct because it ends in code only when
the next SECTION starts. COBOL does not provide any actual command or
syntax to mark this point. You can code an EXIT paragraph at the end
of a SECTION but it is really just an empty paragraph. A SECTION-EXIT
paragraph usually fools people into thinking it marks the end of a
section, but it marks the end only if another SECTION starts
immediately afterward.

"SECTIONs are strange for another reason. There is no COBOL command
that lets you end the execution of a SECTION and get out of it. YOU
HAVE TO EXECUTE THE LAST INSTRUCTION IN A SECTION TO END ITS
EXECUTION. The inventors of the SECTION had no concept of modern
structured programming. They assumed that you would be at the bottom
of your section of code when your were finished executing it. This
was true of flowchart-developed code but is almost never true of
modern structured code that you implement with PERFORM verbs. In
modern programming this quirk of SECTION operation forces to code a GO
TO to send control to the bottom of the SECTION. Since nothing
inherently marks the bottom of a SEDCTION, you usually have to code an
empty paragraph (the EXIT paragraph) at the end of the SECTION just to
have a place to GO TO!"

(James G. Janossy, VS COBOL II: Highlights and Techniques (New York:
John Wiley & Sons, 1992), p 261 - 262.)

Hope this helps,
Boyce G. Williams, Jr.
