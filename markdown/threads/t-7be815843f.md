Thanks to all who offered assistance... especially Messrs Trembley and 
Klein.  The problem was one of 'sneaking in something technically more 
sophisticated than the rest of the code.'

Mr Klein posted some documentation about mixing static and dynamic calls;  
I knew this was not the case since the same CLIST generated compile JCL
for all programs and the output showed the compiler invocation had
PARM='SIZE(MAX),LIB,NUMPROC(MIG),OFF,MAP,XREF,OPT,FLAG(I,I), 
NOSEQ,TEST(NONE,SYM),NOADV' and thi would cause the compiler to default to 
the NODYNAM option.

Mr Trembley posted a problem from the Oldene Dayse about mixing static and 
dynamic calls... and again I knew this was not the case, same CLIST for 
all, compiled in deepest-call-to-driver sequence... and just to be sure I 
put literals in each program and browsed the load modules.  Program C had 
its literal, program B had literals for B and C... and program A had its 
own literal only.

This confused me... and a mistake showed me what was going on.  I brought 
up A in the ISPF editor and wanted to position myself at the PROCEDURE 
DIVISION; I do this by typing 'f proced 8'.  The screen showed no 
change... I had typo'd 'proced' to 'priced' and there was an error-message 
in the upper right-hand corner telling me 'No CHARS 'PRICED' found'.

That wasn't the helpful part... what was helpful was having to stare at 
the top of the code for an extra few seconds.  Instead of first line being 
the expected IDENTIFICATION DIVISION I saw, for the first time, that it 
was:

000100  CBL DYNAM,TEST(NONE),MAP,FASTSRT,LIST

For those who don't get the 'joke'... these are directives which will
override the PARM which was specified at the invocation of the compiler.  
The DYNAM is the fly in the ointment... even though the JCL showed that 
there was no overriding of the compiler's default of NODYNAM (calls 
are static) the compiler would read the line hown above... and set the 
option to DYNAM (calls are dynamic).

This is why I did not see the literals from B and C in the load module for 
A... and why A, when it called C directly, got an initialised version and 
not one which reflected having been called by B.

Compiler directives in the code... that's 'whodunnit'.

Thanks to all who e'er-so-generously donated their time, effort and 
expertise.

DD

Doc,

Here's a snippet of REXX code that I've used in a "pre-compiler" step to ward
off such iniquities:

/*
  Loop through the program to find the items that need to be validated
*/
          intECode = 0
          Do i = 1 to InFile.0
             /* Bypass all comments */
             If Substr(InFile.i,7,1) = '*' Then
                Iterate
             Else Do
                /* Standardize the input line */
                Call CleanUp_Line
                /* We came for the PROGRAM-ID field                   */
                If Pos('PROGRAM-ID',InFile.i) > 0 Then
                   Leave
                /* PROCESS/CBL must appear before Identification Div  */
                If Pos(' CBL ',InFile.i) > 0 Then Do
                   intECode = 12
                   Leave
                End
                If Pos(' PROCESS ',InFile.i) > 0 Then Do
                   intECode = 12
                   Leave
                End
             End
          End
          /* No need to check for any other errors at this point... */
          If intECode > 0 Then Do
             Say ' A PROCESS/CBL statement was detected in program',
                  SRC_Name,
                 'on line' i'.'
             Say ' PROCESS/CBL statements are not permitted in source',
                 'code!'
             Say ' All compile options must be entered on the compile',
                 'options panel.'
             Say ' '
             Exit intECode
          End

Perhaps you can use something similar in your CLIST.

Larry Kahm
