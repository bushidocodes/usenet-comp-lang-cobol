This confuses many people -- here's my answer:
- A RETURN terminates your program. If that is the highest level in the
transaction, the task is terminated
- A RETURN with TRANSID also coded will cause transaction to be reinvoked
when terminal enters more data (pseudo-conversational, here...)
- A LINK is similar to a COBOL CALL in that program invokes another module that
returns control to next sequential statement (NOTE: In COBOL II or
COBOL/370 you can convert LINKs to CALLS with no changes to invoked modules --
and performance is much better)
- An XCTL is a lateral transfer - the task remains alive but with a new
program (similar to a GO TO...). At this point, your module has given up
control of the task.
- A START will initiate a new task, but normally that occurs only after you
terminate current task (with RETURN and no TRANSID).

Of the above, only the LINK keeps your program intact and in control.
You will occasionally see GOBACK in a program but that was generally there
due to restrictions in prior versions of compiler. Also, EXIT PROGRAM and
STOP RUN have had several problems over the years and I don't recommend
using them even though some IBM manuals say it's ok.

Hope this helps you.

I'd like to amplify David's excellent response on a couple of points.

An "EXEC CICS XCTL ... END-EXEC" command could fail if the program to be
transferred to is not defined to CICS. This will leave you in the original
program. This is also a standard error that can be dealt with as a HANDLE
CONDITION or by checking the EIBRESP (or EIBRCODE).

It is perfectly possible to start an asynchronous transaction using the EXEC
CICS START command. In that case, two transactions are running. The
asynchronous transaction would have to be identified by a different
transaction-id and/or terminal-id from the current transaction. Most CICS
applications won't have a need to start asynchronous transactions.

In the past, if you actually executed a COBOL "STOP RUN", "EXIT PROGRAM", or
"GOBACK" statement, the entire CICS region could terminate as if you had
done a CEMT P SHUT,IMM. I believe this was fixed in IBM COBOL II. Some
versions of the COBOL compiler required some kind of program-terminating
statement or a compile diagnostic was generated. If you code EXEC CICS
RETURN immediately before any COBOL program-terminating statement, you
should not have any problems.

More or less correct, although it is a good habit to code the RESP
parameter on an XCTL statement (and most CICS calls). That way if the
program/transaction being transferred to is not available, you can abend
gracefully instead of the generic CICS abend error message. So in this
case, code is executed after an XCTL - but only if the module being
invoked is unavailable (for whatever reason).
