My question is related to OO COBOL, but the same must prevail with those
of you using C++, Java etc., where you have classes which are more than
just called sub-routines.

As with procedural COBOL, there are many instances where an OO  program
can go wrong, using COBOL files - an error on file-status, SQL tables -
SQLSTATE/SQLCODE <> zeroes, attempting to create droplists or listboxes
where the records required are missing - the list is not exhaustive, but
you get the idea.

With a procedural COBOL program, the above is handled fairly concisely -
hit an error condition and instruct the program to GO TO EXIT-PROGRAM or
PERFORM EXIT-PROGRAM.

I can't help but feel that I'm making a 'big meal' out of  this, but
this is how I'm currently handling it :-

method-id. "mainLine"
----------------------
do some housekeeping
set ContinueProgram to true
invoke self "openFiles"

if ContinueProgram
    invoke self "makeListboxCollection"

    if ContinueProgram
        invoke self "processRecords"

        if etc...
-------------------------------------
method-id. "openfiles".
-------------------------------------

*> not exactly how I do it, but conveys possible errors

Local-storage section.
copy "sqlreslt.cpy: replacing ==(tag)== by ==01 ls==.

invoke os-CustomerFile "openIO" returning ls-SqlResult

if ResultOK
   invoke os-InputTransactions "openInput" returning ls-SqlResult

    if ResultOK etc.....
    .

 if FileError ( OR TableError)
    invoke self  "abortProgram" - and the method "abortProgram" merely
does this :-
                    (set CancelProgram to true)
End-if
----------------------------------------------------------------------------------

By their very nature, methods in OO (parallel to a PERFORM PARAGRAPH  if
you like), start at the beginning of the method, proceed through the set
of code, possibly invoking further methods from within the same class or
another class, returning back to the 'active' method and continue
processing through to the end of the method, indicated by either EXIT
METHOD or END METHOD. (This is clearly apparent when viewed through the
debug Animator - going step-by-step, you see the program 'closing out'
on each method which has been invoked in a particular cycle. Same of
course applies to performing PARAGRAPHS).

It seems (at least to me), that the trick is to signal an error, which I
test with 'if ContinueProgram.....", to ensure that the 'active' method
doesn't continue to function as though everything is still kosher - if
you allow it to do that, inevitably the program/class logic will crash.
I can invoke the "abortProgram" to 'set CancelProgram to true' from many
methods, including those resulting from invokes to other classes." This
tends to result in numerous tests throughout many methods to ensure that
CancelProgram hasn't yet been triggered, and also to ensure that these
methods 'jump' to EXIT/END METHOD, rather than attempting further code..

Recall, procedural code, just tell the program to either go to or
perform EXIT-PROGRAM.

Does the question make sense to you - and if you are doing something
parallel with other OO languages - exactly how do you handle this ?

Qualifier to above : I'm am totally using OO classes. You can invoke
'utility' classes from a procedural program - and that gives you the
ability to do the standard EXIT-PROGRAM.

There are at least four styles of error handling, but I wouldn't say that
any one of them is wrong.  I do have my preference, though, backed up by a
rational argument.  I hope it is of use to you, but I'm unsure which of
these options are available to you in OO COBOL.

Option 1:  Error flags

This is the approach you're asking about.  I certainly do this (for exit
code, for example) but will use Inline errors if they're available to me.

Option 2:  Inline errors

This is a popular mechanism for libraries to use.  The snag for COBOL is
that it requires a functional form.  Just for illustration, please allow me
to make "openFiles" into functions that return True on success, False on
failure.

Your code would look something like this:

method-id. "mainLine"
----------------------
do some housekeeping

if invoke self "openFiles"
    if  invoke self "makeListboxCollection"
        if invoke self "processRecords"
            ...

        end-if

        invoke self "destroyListboxCollection"
    end-if
    invoke self "closeFiles"
end-if

This may only seem marginally different that your original code, but it has
an advantage.  The information you need is important at the instant you
return from your call.  By using a temporary variable (the return value of
the function,) you can tightly scope that information to the part of your
code that needs it.

When you, or someone else, comes back to the code, you don't have to seek
out the dependencies (who writes and who reads) upon your working-storage
flag, because that logic is now in the syntax.

Option 3:  Error handlers

If the language permits it, you can set an error handler.  On a macroscopic
scale, this is similar to establishing a debugger for your operating system
to run when there's a system fault.  In code it would look something like:
(Please forgive the syntax, I'm making it up as I go along:


method-id. "mainLine"
----------------------

    on error goto MainExit:

    do some housekeeping


    invoke self "openFiles"
    invoke self "makeListboxCollection"
    invoke self "processRecords"

MainExit:
    invoke self "destroyListboxCollection"
    invoke self "closeFiles"

This is simpler code, but in part because of some different assumptions
about the calls.  "closeFiles", for instance, will have to be smart enough
to not generate additional errors in attempting to close files that are
already closed.

I like this better than inline errors because it's a bit cleaner to read.
"openFiles" will generally not fail, so it's error-trapping is physically
(if not algorithmically) distanced from it.  For cases where success/failure
is an expectation rather than an exception, a inline errors can be combined
with this approach:

method-id. "mainLine"
----------------------

    on error goto MainExit:

    do some housekeeping


    if invoke self "checkForNewData"
        invoke self "openFiles"
        invoke self "makeListboxCollection"
        invoke self "processRecords"
    end-if

MainExit:
    invoke self "destroyListboxCollection"
    invoke self "closeFiles"


Option 4:  Exception blocks

This is my personally preference.  It's a good comprimise between the tight
localization of inline errors and the "as-needed" application of
error-handlers.  There are a couple of forms, but they basically look like:

    invoke self "openFiles"
    try
       ...
    finally
        invoke self "closeFiles"
    end-try

I've only included one call in this example.  After entering the try block,
the finally block will be executed in any event -- error or not.  After the
finally block is executed, any error will be re-raised for containing blocks
to handle.  This is a concise way to indicate that the files must be closed
if they've been opened.  You can rack up multiple blocks:

    invoke self "openFiles"
    try
        invoke self "makeListboxCollection"
        try
           ...
        finally
            invoke self "destroyListboxCollection"
        end-try
    finally
        invoke self "closeFiles"
    end-try

Or, you can use an except block.  Except blocks will be executed in the
event of an error, but the error will not be re-raised.  So, the
error-handling routine can look like:

method-id. "mainLine"
----------------------
    do some housekeeping

    try
        invoke self "openFiles"
        invoke self "makeListboxCollection"
        invoke self "processRecords"
        ...
    except
        // report error
    end-try

    invoke self "destroyListboxCollection"
    invoke self "closeFiles"

Try ... Finally/Except/Catch blocks are a fixture of OO languages, so I
would expect there to be an analog in OO COBOL.  Even if they cannot be
teamed with inline errors, they're a descriptive way of marking exceptional
blocks and their means of recovery.

________

I hope that this is of some use to you -- please don't hesitate to ask for
elaboration on any point.

I've missed an important point of your question.  What I've said still
holds, but it doesn't draw out any difference between procedural and
objective error-handling.  Fortunately, there isn't much.

The rules of thumb for error trapping/handling that I use:

(1)  Trap any error that you can handle.
(2)  Let those that you can't, pass up to "someone" who can.
(3)  Raise errors for situations you can't handle.

If you do this, your code will read like a description of its normal
operation, rather than a laundry list of exception conditions.

Your strategy concerning objects will be affected by the platform.  Some
systems have garbage collection, such that you do not have to explicitly
destroy objects.  If that's the case, you can haul off and use them like
your working storage.  All of the aforementioned "rules" apply.

If you don't have garbage collection, try ... finally blocks are
particularily useful.  (Sorry about the Pascal.)

// attempt to instantiate an object
// if an exception occurs, execution will jump
// to the next exception block (not shown.)
SomeObject := TSomeClass.Create;

// do what you need to with the object
try
  SomeObject.SomeMethod;

  if SomeObject.SomeFunction then
  begin

  end;

finally
  // if the object is created, it will always be
  // destroyed here
  SomeObject.Destroy;
end;

RASING AN EXCEPTION :
An object raises an exception when it has trapped an error. The objects in the
supplied class library define approximately ninety exception conditions
altogether. You can define your own exception conditions for objects you create.

ExceptionManager class provides most of the logic for actually handling
exceptions, including a default exception handler. The default exception handler
displays an error message and shuts down the application when an object raises
an exception, but an application can define its own exception handlers and
attach them to individual classes or objects to redefiine their behavior.

An object raises an exception by sending itself the "raiseException" message,
passing an error number as a parameter. The Base class implements the
"raiseException" method, so it is inherited by all objects.
We will use the Bank application introduced in the previous section as an
example of how to raise an exception. When an account object traps an error, it
responds by raising an exception. For example, attempting to open a high rate
account with an initial balance of less than $1000 is an error condition.

REGISTERING AN EXCEPTION HANDLER
In the previous session, you saw how to raise an exception. This session shows
you how you can register an exception handler for an object. An exception
handler is a method which gets the exception number and the object raising the
exception as parameters.
You can register an exception handler for a class or an instance object. If you
register an exception handler against a class, then it is invoked for exceptions
raised by all instances of the class as well as by the class itself.

Exception handler registration is a two step process:

Create a CallBack for the exception method.
 Register the CallBack against an object with the ExceptionHandler.

Next you will animate through the code which sets exception handlers for the
Account classes in the Bank application.

Animating exception registration code

At this point the BankApplication has finished its initalization and is ready to
run. In the next session, you will look at the exception handler method, and see

WRITING AN EXCEPTION HANDLER
This section of the tutorial shows you how to write an exception handler method
to deal with exceptions raised by an object. We will look at the exception
handler the Bank application uses for dealing with exceptions from the account
classes. In the case of a failure to open a High Rate Account, the exception

The "openAccount" method code in the HighRateAccount class then returns the null

Well ...

Speaking of what is in the draft of the next Standard (not what is in the
current one - or some existing implementations) and TRYING to make
distinctions that I *think* match what you are talking about.

The draft Standard has "several" types of things that appear where
traditionally one "just" had Program-Id paragraphs.  I think speaking a
little about these may help you understand what is in the draft.

1) Program-Id defines a "program".  These programs may be nested in each
other (with Global data or "local" data).  Such programs can have parameters
passed to them (as they always have) but they now can also "return" a value
to the accessing entity.  Programs *may* be CALLed by other programs or by
functions (see below) or by object methods (see below).  Another "new" thing
is that they may be recursive (if explicitly identified as such) - in which
case "local-storage" items are allocated for each instance - while
working-storage is available to every instance (and EXTERNAL data is
available to everything in the run-unit).

2) INTRINSIC functions are a list of "standard" functions that return
values.  These do lots of "numeric" things (Standard deviation, PI, etc) -
data manipulation (To/From Alphanumeric/National, To/From Upper-case) -
comparisons (MAX, MIN, ORD-MAX, etc) - system/environment things
(CURRENT-DATE, DATE-COMPILED, etc) and a variety of other things. Again,
these may be accessed by programs, user-defined functions, or object methods

3) User defined functions are very similar to "programs" - in fact, other
than a "new" Function-ID paragraph, there is almost no difference between a
user-defined function and a program with a RETURNING phrase.  The difference
is that *ALL* user-defined functions MUST have a RETURNING phrase. Again,
such functions may be accessed by other user-defined functions, by programs,
or by object methods.  Because these user-defined functions are ALWAYS
"pre-defined" (by the "user"), there is "complete" conformance checking at
compile time that parameters passed and returned to user-defined functions
match what they are supposed to be.  (Conformance checking via "program
prototypes" is now available, but not REQUIRED for programs)

4) Objects - as indicated elsewhere have both "FACTORY" (one per class) and
INSTANCE definitions (one definition per class, but the actual "code" exists
separately for each instance of an object of that class).  Within both
FACTORY definitions and INSTANCE definitions, there are "methods" which are
the actual procedural code.  These may include inline (via :: operator) or
out-of-line (via INVOKE) references to other object.  However, they may all
use CALLs to programs and references to both intrinsic and user-defined
functions. Like user-defined functions, "conformance" checking is done (or
can be done) at compile-time (when possible), at run-time, or be
"turned-off" by the programmer.

  ***

Bottom-Line:
  The new COBOL "world" is a "mix-and-match" one where you can use as many
(or as few) of these types of entities as you want.  Furthermore, there are
"definitions" which make COBOL "procedures" and "functions" and "objects"
available to non-COBOL entities and allow COBOL to access such entities in
other languages as well.

This is (obviously) somewhat of a "simplification" - but does it help?
 Looking thru the CONCEPTS annex in the draft Standard gives examples and
"over-view" explanations of much of this.

I read the posts in the thread, and Grinder has given a pretty comprehensive
solution. Unfortunately, OO COBOL does not support the preferred solution he
offered (yet), and his syntax is also invalid in OO COBOL. Nevertheless, his
ideas are spot on. It is my intention to look at implementing the Exception
Classes and Methods described below, in OO COBOL. For more on this, see
below.

As you have asked for comparison with how it is handled in C++, Java, etc.,
I'll expand a little on Grinder's excellent post and hope you can catch the
drift, Jimmy...

The following is based on my own experience which, in turn, is based on the
writings of John Zukowsky, who is (in my opinion, at least) one of the
World's best educators on Java. (Nearly everything I know about Java I
learned initially from his writings...)

Talking about the Basic Exception Handling model in Java he says:

"Java follows the basic C++ syntax for exception handling. First you TRY to
execute a block of statements. If an abnormal condition occurs, something
will THROW an exception that you CATCH with a handler. In addition, FINALLY,
there may be a block of statements that you always want executed, no matter
whether an exception occurred and no matter whether it is handled if it does
occur.

Throwing an exception is friendlier than terminating the program, because it
provides the programmer with the option of writing a handler to deal with
the exception."

So what we end up with is an event driven model. The main program invokes
some other Class. It raises an exception. Rather than coding a bunch of IFs
to test various possible error flags (which was one of the things I think
you are trying to avoid), you can simply let the "event" of raising the
exception, trigger the User written (or, in Java), System Provided handler
for it.

There are intimations of this in CICS where you can specify ON... EXCEPTION,
and SQL where you can specify (at least in DB2 you can...):

WHENEVER
  [NOT FOUND/ SQLERROR/ SQLWARNING]
  [CONTINUE/ GOTO host-label/ GO TO host-label]

         EXAMPLES

The following statements can be embedded in a COBOL program.

Example 1:  Go to the label HANDLER for any statement that produces an
error.

     EXEC SQL WHENEVER SQLERROR GOTO HANDLER END-EXEC.

Example 2:  Continue processing for any statement that produces a warning.

   EXEC SQL WHENEVER SQLWARNING CONTINUE END-EXEC.

Example 3: Go to the label ENDDATA for any statement that does not return.

     EXEC SQL WHENEVER NOT FOUND GO TO ENDDATA END-EXEC.

However the Object Oriented Languages take this much further, to the point
where you can write virtually ANYTHING as an event that can be caught by any
Method code that is interested in dealing with it.

Two Java Classes which may be of interest to you are:

1. SQLException

defined as:

public class java.sql.SQLException
extends java.lang.Exception
extends java.lang.Throwable
extends java.lang.Object

and

2. SQLWarning

defined as:

public class java.sql.SQLWarning
extends java.sql.SQLException
extends java.lang.Exception
extends java.lang.Throwable
extends java.lang.Object

Both of these Classes have their own Constructors and Methods, the second
one simply extending the first by adding 2 new Methods, as OO Classes are
wont to do...

Both of them are provided as Classes in the java.sql package. But suppose I
want to extend this even further? Say, for instance I want to know if an
INSERT failed.(This is a contrived example because the standard SQLException
can be made to return the SQLSTATE by invoking one of its methods, but it
will serve for the purpose of discussion...)
...
//InsertFailedException is a general exception for SQL INSERT problems...
//
class InsertFailedException extends SQLException{
 public  InsertFailedException () {
 }
 public InsertFailedException (String  whyitfailed) {
       super (whyitfailed);
 }
}

The Method that invokes this Class must specify that it THROWS
InsertFailedException. Let's say I have a Method in the Customer Class
called "insertNewCustomer"...

public String insertNewCustomer (...parms for this Method...)
          throws InsertFailedException, SQLException {
          ...includes code to set up the  INSERT statement and invoke a
stored procedure on the database to get a new
         Customer ID....
         ....
         if ((newID = getNewID ()) == null) {
             throw new InsertFailedException ('Could not obtain Customer
ID');
         }

(Note the difference between THROW and THROWS... a common source of error in
early Java programs).

Now I can insert a new customer and have the system automatically invoke my
INSERT Failure handler or my general SQL failure handler, without the need
for any flags or ifs...

try {
      insertNewCustomer = ...various stuff required for the Method, that we
are not concerned with here...
      }
      catch (InsertFailedException e){
        system.err.println (e);
      }
      catch (SQLException e) {
         system.err.println ('Database failure when creating Customer=' +
e);
      }

So, from this very brief exploration you can see that the OO paradigm allows
us to handle exceptions as events, allows us to extend existing exception
classes to tailor our own exception events, and allows us to write our own
exception handlers.

Now the question is: How could we implement all of this in COBOL
(specifically, OO COBOL)?

After looking carefully at the underlying Java Classes and how they are
built up from basics, I believe it IS possible. Certainly with PowerCOBOL it
is possible to define your own custom Events and Methods for an Object. (In
fact, Events are defined just like any other custom property.)

The TRY construct in Java kind of alerts the system to the fact that
Exceptions can be expected. I can think of no equivalent in COBOL at the
moment, but it may be possible to write a Class that provides the same
service.
(It might not even be necessary if the right Event processing can be
implemented...)

I would like to thoroughly investigate this and come up with some OO Classes
that could be invoked from OO COBOL to achieve the same effects as you see
above. Unfortunately, I simply don't have time at the moment. I have written
this response in the middle of the night...<G>

Nevertheless, I hope this gives some insight into how exceptions CAN be
handled in OO, and may stimulate you to do some more investigation of your
own, Jimmy.

Pete.
