Recommend "Systemantics - Why Systems Fail".

As it turns out, there are "laws," some of which are:

* A large system designed from scratch will not work and cannot be made to 
work. You have to start over with a smaller, working, system.

* Malfunction, or even total non-function, of a large system may not be 
detectable for a long time. If ever.

* Fail-safe systems often fail by failing to fail safe.

* Humans within a system do not do what persons with the same title do in 
the real world.

Just to be clear Bill:

You are saying that both of the above systems in the United States can NEVER 
be replaced and will process in COBOL for evermore?

Obviously, that is unlikely.

So how long then? 20 years? 30 years? 50 years?  What?

If you think about this you will realise that our point of difference is 
just about the time scale. (Unless you want to affirm the first question 
asked above.. which says NEVER...)

So, if we agree they will be replaced at some point, what are the factors 
that have a bearing on that?

Mostly time and resource, coupled with the fact that they are generally 
working fine so there is no reason to replace them.

However, I think there is an "evolutionary" solution that will see these 
systems gradually replaced, rather than an overnight migration. Little bits 
of them will be chipped away by new technology and eventually there will be 
very little left of the existing COBOL core. While it would be extremely 
difficult to fund and implement a complete replacement as a single project, 
it is not so hard to replace minor sub-systems, maybe on a local basis.

NO EXISTING SYSTEM is "doomed to failure" if the will and the means to do it 
are available.

1. The system exists.

2. Therefore the system was able to be programmed. (It has rules which can 
be presented in computer sensible form.)
    Even if these rules have changed over time it is still possible to 
identify what the rules are. (If it weren't, the current system could not 
run).

3. If the system was able to be programmed, the system is able to be 
re-programmed.

4. It is therefore possible to rebuild ANY existing system. (QED)

There are a couple of riders to the above:

1. If you approach it from the point of view of re-doing it IN COBOL, it may 
well be more trouble than it is worth.
2. You may need to use a completely different approach from that employed 
originally.
3. You can only be guaranteed to rebuild the functionality which is 
CURRENTLY available. If you want improvements, it MAY be possible.
4. The amount of effort and resource needed to do a rebuild may be huge, it 
may not be justifiable, but if it HAS to be done, it CAN be done.

As someone who makes a living salvaging COBOL, I have yet to see a system 
that cannot be salvaged. New and better tools to do this are appearing all 
the time (I'm writing some of them... :-)) Even in the last 10 years new 
approaches to the problem have evolved. Data driven Migration is an approach 
that PRIMA has pioneered and it is working very well. Although data 
structures and types within an organization can change over time, the 
fundamental elements that are required by that process don't change much.. 
(A payroll will need a pay rate, and someone to be paid, for example...) The 
existing system (especially if it is written in COBOL) conveniently 
identifies the elements needed. (PICTURE clauses...) Once a data analysis 
has been done (PRIMA has a tool to do this automatically) and the data is 
available in a modern repository, it is then possible to use it for both Old 
and New development. (Again, we transform the existing codebase 
automatically and make it access through a generated Data Access Layer 
comprised of objects.) Legacy processes as it always has, but now it gets 
its  data from the common repository.

New processes are developed separately and may or may not be in COBOL. 
(Whatever it is written in, it will be written as objects) Both Old and New 
technology use the SAME data repository. The Legacy code continues to update 
it and the new systems use it in new ways. No Batch synchronizing, everybody 
is singing off the same hymn sheet. The repository is accessible (through 
the DAL objects) from the desktop or the network and it is accessible by Old 
and New program code and even by ad hoc scripting. (Note that there is NO 
embedded SQL in the transformed legacy or in the new development. Separation 
between Business Logic and Data Access is maintained.)

The main difference is that the New is object oriented and the Old is not.

Useful routines in the legacy code can be refactored for use by the new code 
or they can be rewritten in new code. (Depends on priorities and resources 
available)

In this way, the NEW code gradually supersedes the OLD. There is no pressure 
to migrate, there is no Big Bang Migration, just the evolution of the system 
towards New technology.

No matter how large or convoluted the legacy may be, it keeps running until 
it is replaced or overtaken by events.

This is a different idea from sitting down and trying to rebuild a system in 
COBOL.
