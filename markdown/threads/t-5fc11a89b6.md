So did the Burroughs Medium System, and so (now) does the Unisys MCP series.

MCP COBOL compilers (COBOL(68), now gone; COBOL74, and for compatibility 
with both, COBOL85) originally took a different approach:  any character 
outside the strict ANSI syntax was treated as an insertion character.

The original idea was that it be applied only to embedded hyphens (for 
social security numbers) and perhaps colons (for times), but that's not the 
way the authors did it; they didn't include any limits on it.

The approach turned out, in my view, to be Not a Very Good Idea, because it 
meant that if that approach was taken "correctly", the only PICTURE syntax 
errors one could get were (1) it describes a numeric value with more digits 
than is permitted (18 for ANSI, 23 for our architecture); (2) the picture 
character-string has too many characters in it, or (3) it describes an 
alphanumeric data item bigger than the compiler permits.

This made for nightmares for both the compiler folks and for the end user --  
something as simple as "PICTURE 9(3V99" is valid in this context, and is a 
numeric-EDITED picture that allows only one digit before the decimal 
point -- putting the value 123.45 into it results in "3(345" ... You can 
imagine how bad it gets when data corruption only occurs with large values 
...

Let's just say I know the relevant areas of the Unisys MCP COBOL74 and 
COBOL85 implementations pretty well at this point, in both their "before" 
and "after" incarnations ...

    -Chuck Stevens

I forgot to reiterate that I *do* mind *abbreviated* combined relation 
conditions.  I believe the reasonable expectation of a native speaker of 
English is that an elision of a comparand or a comparison operator implies

a desired determination of a condition may not be proper to do in English, 
and vice versa.

them!
It's 40 years to late, most likely, it seems to me (and probably most?) that
the following:

1) if result = 'a' or 'b' and something-is-true
means
2) if (result = 'a' or result = 'b') and something-is-true

when in fact in COBOL it really means
3) if result = 'a' or result = 'b' and something-is-true
which of course means
4) if (result = 'a') or (result = 'b' and something-is-true)

Ah well...  I will most likely continue to use them and be bitten by them,
simply because I like to code as few characters as possible.  The use of
88-levels at least somewhat mitigates the problem, though.  eg
if good-result and something-is-true
where 88 good-result value 'a' 'b'.

'good result' if needed.

Anyway, no, I don't expect COBOL to change this at this stage.  Would break
too many programs, though I do find it hard to imagine someone using example
1 and really expecting it to bind like example 3.  Has anyone ever done this
intentionally?

Does anyone know of any other programming languages with abbreviated
conditions?  How do they handle them?
(It's ironic that COBOL has them, as COBOL is generally known as being more
verbose than most languages!)
