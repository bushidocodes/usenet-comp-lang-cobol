One can specify the space allocation at either the cluster level, or at the
component level (DATA, INDEX), but not both.

I generally prefer specifying space allocation at the component level because
it gives me better control on the allocation instead of relying on IDCAMS to
make its own assumptions.

Generally the index allocation doesn't have to be anywhere near as large as the
data allocation--I generally find a few cylinders of data need just a few
tracks for the index. However, I have used several cylinders for the index when
the data has occupied a thousand or more cylinders. Of course, the real answer
on allocation is "it depends" and I find the LISTCAT listings a real friend for
adjusting space allocations.

Mark A. Young

Unless you have an "unusual" case or really know what you are doing, I
wouldn't recommend ever giving explicit space for the INDEX component
(or giving a NAME parameter for the INDEX component either--let it
default).  Either specify space on the CLUSTER (which is used for the
DATA component) or specify SPACE for the DATA component.  Normally let
IDCAMS decide what is reasonable space for the INDEX.  When we get JCL
from vendors with INDEX space, it is invariably grossly overallocated. 
In your case above, the initial dataset DATA size requires only 10 index
records (1 record per CA, which in this case equals one DATA cylinder)! 
Depending on INDEX CISZ chosen by IDCAMS, a full cylinder probably has
space for at least 180 index records, maybe more, so 5 cylinders would
be considerable overkill.  I notice no DATA CISZ specification.  This
probably means a default of 32768, which may be good for sequential
batch, but questionable for either online applications or if majority of
accesses are random.

Allocating in records will result in a much smaller than optimum CA 
(Potentially a single track CA) resulting in poor read and insert 
performance - multiple CA splits.  You are better of (SPEED wise) to 
allocate at the cylinder level.

From the "Using Datasets" manual:

--- just above the heading liste below:

"If the control area is smaller than a cylinder, its size will be an 
integral multiple of tracks".

Under the heading "Advantages of a large Control Area Size"

"Control area size has significant performance implications.   
One-cylinder control areas have the following advantages:

There is a smaller probability of control area splits.

The index is more consolidated.  One index record addresses all the 
control intervals in a control area.  If the control area is large, 
fewer index records and index levels are required.  For sequential 
access, a large control area decreases the number of reads of index 
records.

(--- my comments now - this next one is the biggie!--)

There are fewer sequence set records.  The sequence set record for a 
control area is ALWAYS  (emphasis mine) read for you.  Fewer records 
means less time spent reading them.

........"

There's more but that's the gist of what I'm getting at.

The formula you gave involving alt-key-length is the formula for
calculating the RECSZ values for the DATA component of an alternate
index -- doesn't direcly relate to the space for the INDEX component of
either the base CLUSTER or the AIX.   The INDEX component always
requires one INDEX CI (Control Interval), recorded in the stats as one
INDEX record, per DATA CA (Control Area).  Since IDCAMS knows at
creation time what the CA size is and exactly how many CA's are in the
DATA component's primary and secondary allocations, it can compute the
exact space requirements for the INDEX component more easily than you
can.  The only case where it makes sense to explicitly give the INDEX
space is when you know the DATA component will eventually expand to many
secondary allocations and you want to allocate the INDEX at max required
size just to avoid fragmenting your volume with a large number of tiny
index secondary extents.

I personally like using RECORDS allocation myself because it is device
independent.  Several things to watch for:  
* Unless the entire dataset is small, you may need to experiment to be
sure your secondary allocation is large enough to exceed one cylinder. 
If both the primary and secondary allocation are larger than one
cylinder, then you will get a CA size of one cylinder and file
allocation in cylinder units, which tends to be optimum for large VSAM
datasets (it makes no difference here whether the original allocation is
in RECORDS, TRACKS, or CYLINDERS).
* RECORDS allocation presumes that maximum use is made of each CA and CI
in the DATA component.  If you define the CLUSTER with nonzero CI or CA
FREESPC parameters and want to be able to do an initial load of "n"
records into the CLUSTER, then the requested space in RECORDS needs to
be proportionally adjusted to a higher value
 n' = n /((1-(CIFREESPC/100))(1-(CAFREESPC/100))
* If you have adjusted the RECORDS parameter properly and the average
record length is correct, but the dataset still goes into secondary
allocation on initial loading with far fewer records, you may have one
of those rare cases with large keys that compress poorly, where the
default INDEX CISIZE chosen by IDCAMS may not be large enough to allow
use of all the CI's in a CA.  You may need to force a larger INDEX CISZ
by explicitly requesting a larger CISZ in the INDEX component.
* For RECORDS allocation of an AIX cluster, remember that the number of
records in this context is the number of unique alternate key values and
the record length is not the original base cluster record length but
given by your 5+altkeylength + basekeylen*nrdups formula, which is
usually much shorter than the base cluster record size.
