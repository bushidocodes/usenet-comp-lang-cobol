It's not really an IBM COBOL issue.

The 9(10).99 is an edit pattern.  The field is an alphanumeric field
at this point - you can't use it as anything other than the
destinatoin field in a calculation.  So what they are saying is
technically correct - with one exception.  It's not the printer that
adds the decimal point!!!  You can write it to a data file with the
picture clause9(10).99 and you WILL get the decimal point in the data.

However -- It's not a number anymore.

14 layers higher in this discussion, someone called 9(10).99
"alphanumeric" - and we all "know" <g> that this isn't true.  What it is, is
"numeric-edited" - which has different rules than either numeric or
alphanumeric (or alphanumeric-edited) fields.

It is perfectly true that "numeric-edited" fields can be used most places a
numeric field can be used AS A SENDING ITEM in ANSI'85 (but not earlier).
Furthermore, it can be used as an argument to the NumVal and NumVal-C
intrinsic functions in ANSI'89 (with the output of the function being usable
wherever an arithmetic expression is valid - and in the next Standard
wherever a numeric sending item can be used).  Therefore, (almost as the
name implies) it really is "half way between" a numeric item and an
alphanumeric-edited item.

Bottom-line:

A) of course the original post was just plain foolishness  (about IBM
mainframes and explicit decimal points)

B) (assuming ANSI'85)  any numeric-edited item can be used lots of places
that a numeric item (but not an alphanumeric or alphanumeric-edited item)
can be used - but certainly not all such places.
