Just curious as to what experienced programmers believe the definition is of
these two styles of programming. We believe them to be basically the same
thing, but there is some difference, from what we are told:

bullet-proof programming - a program that has been tested using real or live
input for test data by someone other than the programmer to diminish bias

defensive programming - a program that has been thoroughly tested and has

Drifting somewhat off topic, but this still has to do with code that was
_not_ bullet proof.

In the early 1980s, a fire broke out on a BART (no relation) train under
the San Francisco Bay.  The computer responded to the fire alarm by
stopping the train.  Everyone died.  The official post-mortem on the
incident concluded that the train should have continued on to the next
station so the the people could get off.  Whoever designed the program
hadn't thought it through.

Ten years later, a fire broke out in a freight car on a Chunnel train
under the English Channel.  As soon as the second fire senser was tripped,
the computer stopped the train!  This time, no one died, because the fire
was small and all the way at the back of the train far from the
passengers, but if the fire had been bigger and/or closer to the
passengers people would have been killed.  This time it wasn't just a
matter of not thinking it through, they hadn't even studied the history of
tunnel disasters.  There wasn't even a way to restart the train from the
control station until the fire was put out.  They had to send little
shuttles in from the other end to retrieve the passengers.

There arre applications where you just can't write brittle code.

Well, there's English grammar. "bullet-proof programming" cannot be a program;
nor can "defensive programming". These terms relate to a style of programming.

Alternatively, you can say "a bullet-proof program is a program that ..." and
so on.

In any event, in my mind, a program is bullet-proof if it cannot fail no matter
what data the user throws at it. This means that it checks data on entry an
ensures that it is correct format before using it; it also means that all kinds
of errors (e.g., hardware failure on a tape or disk drive) are trapped and
dealt with.

Very few programs are written with this kind of rigor (and that's appropriate:
it takes a lot longer to write and test such as program; the reality is that
for most programs there is not enough payback to write to this level; if you
have a problem, abend with a message, fix the problem, and rerun the step /
job).

The term "defensive programming" has about the same connotation. Perhaps the
distinction would be that program written in a defensive style will prevent bad
data but not trap hardware errors. I don't know that there is any definitive
take on this (at least I don't recall seeing any).

Unfortunately the definitions and testing are unrelated.  You can
program defensively and test attrociously :->.  For a definition
of defensive programming see Writing solid code by Steve Macguire
page 33. He describes what I will call bullet proofing.  You may
want to look up Xtreme programming (great name) which is about a
concept close to what you discuss.

Coding defensively is a matter of ensuring that you code does not
get stuck on unexpected results.  For example

perform varying i from 1 by 1 until i > 20

This code does not need the '>' EQUALS will do fine.  Code changes
later may affect the content of I and memory problems may mean
that 'i' could be corrupted sending the program into an infinite
loop.  Incorrect results are better (in some instances) than no
results at all.

Bullet proofing the same code would be

perform varying i from 1 by 1 until i > 20
D  if I > 20
D	   something has gone seriously wrong ring bells!!!
D  END-IF

Again this code may NEVER be tested!  Because the i varaiable
should never ever be greater than 20, the code under the if will
be redundant and in most cases removed by the compiler optimiser. 
It is surprisingly rare for a programmer to force the error to
test the error routines and the tester cant without instrumenting
the code which is not something they should do.

There is a great cost (about 3 times according to Steve McConnell)
to bullet proofing systems.  A program for inhouse must have a
single path that works sucessfully, if it does not work then train
the user to make it work.  A system that is bullet proof must have
no path that does not work, if it does have a broken path then fix
the program.  It is a big difference in coding style.

The main problem is that a particular program may be bullet proof
but the design is shot.  Quality is about the whole system. 
Analyse well, code well, test well and all will be well, stuff
anything up and it wont!.
