Does anybody know if the latest (or any) of the COBOL standards address the issue of
how COBOL applications function in a multi-threaded environment. (By thread, I mean a
OS schedulable task that executes in the same addressing space as other threads within a
process.)

If so what does it say about...

Two separate threads are started within a process, both can run concurrently and
asynchronously of each other. Suppose that both threads invoke the same COBOL
application module. Can one instance modify the working-storage contents of the
other instance?

My two cents would be no. The application should be have any different when two
threads execute the same module then when one thread executes it. Allowing them
to share working storage would be very chaotic.

Does the standard address a mechanism by two (or more) such thread can intentionally
share memory? What about a semaphore ,monitor, or critical section mechanism?

i'm not aware of any standards work wrt multi-threading. however, we (Micro Focus)
have implemented a fully multi-threaded compiler (including syntax extensions).
the first product that will be released on top of this compiler will be
Visual Object COBOL for Windows 95/NT (you can find more about this product at
http://www.mfltd.co.uk/win95.html).

it's certainly possible for two threads to access the same data (and other
shared resources) so synchronization (in your application code) is key. it is
also possible to "protect" yourself against this by marking the COBOL program
as "serial" (but then you lose the major benefits of a multi-threaded
architecture -- this mode is really only useful for letting an existing module
"play" in a multi-threaded environment).

there are several mechanisms in the compiler that allow you to synchronize
execution (supplied primitives for mutexes, monitors, semaphores, events)
and resolving data contention (for example, using data-items in a THREAD-LOCAL
section).

if there's enough interest in this topic, i could be persuaded to post a "white-
paper" to this forum that deals with this entire subject (it starts out "What is
multi-threading ? Good Question..." and goes from there into an in-depth
discussion of threading and COBOL). it was written by Barry Rosetti who is the
run-time architect primarily responsible for the compiler implementation. Barry
can be found at b.··.@mic··s.com

Actually, after thinking about this for a bit I came up with the
following. There is one standard feature that hardly anyone implemented
(it is optional) that can accomplish some of this sort of thing. It is
the message control system (MCS). If implemented reasonably, it allows
you to send a message to a program, which fires it up, and then
communicates with messages. Lots of programs can send messages to the
same program (the messages are queued). Basically, you use messages to
synchronize and all. Not everything you mentioned, but some of it. MCS
will be declared oboslete in the next standard, and will slowly fade
away. Most people think the queue structure and the whole thing is too
complex and out of date.

Terry,

I used to work on a COBOL compiler. The current active standard (ANSI COBOL-85) is
silent on the issue of threads but imposes semantics in that makes the use of threads
very tricky. I have been working on multi-threaded code for sometime in other languages
and would suggest that you do not try to uses threads anyway - it is just to hard to
nail the numerous race conditions. My suggestion would be to use the UNIX model of having
seperate programs with a shared memory segment. This makes life a lot easier while
giving almost all the benifits of threads.

I have a lot of experience of using multiple threads with COBOL (both 74 & 85).
Unfortunately the implementation is the vendors implementation (Unisys A
Series). The way it work is that the parameters that are passed to a process can
be modified by any of the processes. This requires you to protect the resource
every time that you would like to update it. In our implementation we use
something call a LOCK. The procedure that you use to update a shared data area
looks something like this:

77 my-lock LOCK.
:
:
PROCURE my-lock.
COMPUTE global-amount = global-amount + local-amt.
LIBERATE my-lock.
:
:

The compiler generates all the necessary operating system (MCP/AS) call to
implement this. If the lock my-lock is procure'ed by some other process then the
process trying to procure the lock will wait. When the process that has the lock
does the LIBERATE then the next process waiting for the lock will become active.
It is simple and elegant and works every time. The only problem being that the
developers are responsible for making sure the shared resource is protected
while being changed. This should not be a problem, since the programs will have
trouble providing consistent results.

The COBOL74 & 85 compilers implement both asynchronous and synchronous tasking
(threads). We call them processes and co-routines. All kinds of parameters can
be passed to this threads including files and 01's.

I can post an example program with all the extended syntax that allows you to do
this. Is anyone interested?
