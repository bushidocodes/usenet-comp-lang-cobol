Yep - that is the way to do it. :)

I would have advised you to download the Fujitsu compiler, which is an
outdated version but is an unrestricted Windows based compiler that is
perfect to learn on. (Well, it is if you know and remember that the SCREEN
Section is compiler specific and not found in every compiler, and ignore
their
wierd editor and IDE. <grin>)

Or pick up Thank Hubble's book (COBOL in 24 HOURS I think) or Mo Bundings
COBOL in 21 days book at your local Barnes and Noble. This is much less
expensive
than $400 and provides pretty much the same stuff.

After you get a little feel for the language, you (or your client) can
decide what Compiler
is most appropriate for the task(s) at hand and the development environment
available.
For example, if you are deploying on Windows and using just VSAM (indexed)
files,
then it is possible that even the free Fujitsu compiler might be
appropriate. If you are
on an AS/400, then only the AS/400 Language Environment compiler is an
appropriate
choice. (And everything you learned about screens is out the Windows --
welcome
to the world of workstation files. :)

The important thing is to learn what the language really provides - which is
a powerful lot
for business style processing. It isn't appropriate to write say, HTTPD
Servers in COBOL,
but believe it or not, people have done it. (It's a fact, the Wang VS Web
Server was originally
written in COBOL. I think the current source is BASIC though, since the
author did not
have COBOL-85 available,with nice things like POINTER variables and ADDRESS
OF...)

It is reasonable and practical, and often accomplished that a basic
accounting system can be
written from a standing start in two to four weeks though.

Basically, the best advice that I or anyone else can give you is to ask
questions, and keep asking them
until you understand the answers. And good luck. :)

-Paul

MMM- well, Micro Focus is a great compiler, but I do not know what is in the 'Personal Cobol' set. 
I do know however that the MicroFocus NetExpress 'University' package is really good. Nice debugger,
nice compiler, limits are very appropriate for a student package. Not a current version, but still very
very nice.  Especially the debugger - though more on that below.

On the other hand, I would probably send the $100 Personal Cobol back and simply download
the Fujitsu compiler. Just take 10 minutes to learn the command line utilities instead of the 
IDE.

a different world!

nuisance with runtime charges, but their compiler is fast, supported, and has switches that enable very
HIGH compatibility with R/M Cobol. You can also develop on any platform, targeting deployment on 
any other platform.  Given that developing under DOS is a painful and time wasting experience these
days, that would be a heck of a option to me.

Good points Jimmy - 

Usually with clients who want C programmers to write COBOL, they are just looking for two things:
(1) This is the solution you need to make it work and
(2) This is lowest cost I can find to make it work that way. 

You can make a lot of money dealing with one up situations and customers who think they know a lot 
more than they do, but it is not very satisfying. On the other paw, if you go in with (1) and (2), all the 
onus is on you to deliver - in a timely and appropriate manner. ;) 

Re: debuggers - they are available on the mainframe, but they are not all that common. Deskchecking 
is still very important. <grin> Since a lot of what I do winds up on the mainframe one way or another, 
I tend to discount debuggers to a large degree. When you have a really good debugger, like MF or 
Wang's - then they are a perfectly wonderful joy to use. But I learned not to depend on having them around. 
;)

Now to the question of Ada and safety-critical software, since you asked.

Ada, formerly a language focused on military environments, is now used
for commercial software such as avionics in which high reliability is a
key factor.   What makes Ada the language of choice?  The reasons are
multiple, but we will give only a few here.

1) Ada, the language, is designed to maximize the  amount of error detection
    in design, syntax, and semantics as early as possible in the development
    process.    This is more of an engineering issue than a programming concern.

2) The compilers for Ada, those used in safety-critical software, are required to

    pass a set of validation tests before they can be used in production.   The
    validation suite consists of a large number of programs that exercise the
    compiler to determine whether it conforms to the ISO language standard.

3) Ada is the only language with an Annex specifically targeted to the issues
    of software safety.   Annex H of the Ada Language Reference Manual
    defines specialized pragmas and rules that provide  restrictions, code
    review capabilities for ensuring a higher-level of integrity of the final
    product.

4) Ada programs are compiled to a specialized Run-time Environment that
    sits on top of the operating system or bare-board platform.  This RTE
    includes a set of specialized features for run-time checking of the
    program.   It is augmented by an error management capability at the
    source code level.

Programmers are always going to prefer using debuggers over desk 
checking when the debugger is fun to use. Though debuggers are very 
useful, I don't find this imbalance to be good, in any environment. 
Dijkstra's famous comment applies: testing can prove the presence of 
bugs, but not the absence.

Yeah, you'll hear lots of long-time Unisys customers talk about this 
phenomenon. Mikey Blumenthal made a big deal about combining the 
strengths and duplications of Burroughs and Sperry, but what got 
combined most effectively was their Sales Prevention Departments (SPD), 
which seem to have been equally effective in both companies. The

> I do have a micro-A series board here still, but I don't have any

The current equivalent of the MicroA is the LX100 -- a complete MCP 
system several times faster than a MicroA, running in a stock laptop 
(albeit a $5000 laptop plus I think a $5000 software license). It's 
only for development and demonstration purposes, no production use, and 
there's no support. A wonderful boon to small developers. Runs circles 
around an LX5120, the smallest production system.

But the SPD got their fingers in here too:  you can't upgrade the

pay-per-call basis. (These might change after a long meeting at UNITE

> CANDE isn't ISPF, but it is a good editor in itself. Did they ever 
> get the debugger integrated into it? 

Not into CANDE. CANDE is the low end editor, although a lot of people 
who don't like change still use it. Because it's basically a line mode 
editor (despite the page mode kludge), Unisys aren't trying to update 
it much. CANDE has been far surpassed by two other editors, 
SYSTEM/EDITOR (aka U ED) and Programmer's Workbench (aka NX/Edit).

SYSTEM/EDITOR was developed at one of the software plants around 1980. 
It was first for internal use and later released to the field. It's a 
true full screen editor, always showing a current image of the work 
file on the screen. It handles patch mode, integrated cross-reference 
files, macros, up to 256 lines and columns, dump reading, object code 
reading, and numerous other features. It's the editor that the software 
plants have used almost exclusively for over 20 years, and I'm sure you 
know what happens to software when the users are also the developers: 
it gets extended when there's a need and fixed when it breaks, with 
little or no red tape. TADS is fully integrated into EDITOR; you can 
display variable values, monitor and trap on variables, walk, and so on 
from within EDITOR. The big problem with EDITOR was always that it 
wasn't bundled and the SPD never figured out how to market it.

NX/Edit (renamed Programmer's Workbench, or PW, in the newspeak) is a 
Windows-based product which edits mainframe files. It maintains all the 
conventions of MCP file usage and program development while giving the 
user a Windows environment. It can run either in integrated mode (with 
the files on the mainframe and a server providing the access), or 
completely standalone with all the files on the PC and no connection to 
the mainframe, or any combination. It has most of the important 
features of EDITOR, including cross-reference integration -- and the 
cross-reference files can be downloaded to the PC for standalone 
operation. TADS is also integrated with PW. It's still a new product 
with a few raw edges, the biggest one being no macros or scripting, but 
most of these are being addressed. So far it's bundled, thus bypassing 
the SPD. This combined with the now-familiar Windows UI gives it a good 
chance of much greater acceptance than EDITOR has ever had.
