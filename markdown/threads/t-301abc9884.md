Hi all,

We are just starting out with a hybrid system:

HP-UX 10.01 Database Server DB2/HP
AIX 3.5 CICS Server
AIX 3.5 Terminal Server

All systems use MF Cobol:
HP 1.37
AIX 1.33

We have the source code on HP and NFS'd to the CICS Server.

The compilation plan is as follows:

Compile all .cbl to int on HP
Compile all .ccp (cics modules) to int on AIX
Generate gnt on HP and on AIX
Bind the bnd files generated when the int code is created to the various
databases as needed.

The problem now arises that the compiler on AIX cannot read the source
files through the NFS mount either for the first stage or for the
generation stage.

If the files are copied to a local file system on AIX they are of course
visible and usable.

Does anyone out there know a solution to this? It may be to do with the
exclusive lock required by MF Cobol on source files while compiling and
the NFS versions for both operating systems.

Joan Colley
ADC Consultants
Sydney Australia.

Hi all,

We are just starting out with a hybrid system:

HP-UX 10.01 Database Server DB2/HP

AIX 3.2.5 Terminal Server

All systems use MF Cobol:
HP 1.37 (7a.1a.11.10)
AIX 1.33 (7a.1a.11.02)

We have the source code on HP and NFS'd to the CICS Server.

The compilation plan is as follows:

Compile all .cbl to int on HP
Compile all .ccp (cics modules) to int on AIX
Generate gnt on HP and on AIX
Bind the bnd files generated when the int code is created to the various
databases as needed.

The problem now arises that the compiler on AIX cannot read the source
files through the NFS mount either for the first stage or for the
generation stage.

If the files are copied to a local file system on AIX they are of course
visible and usable.

If the files are on AIX and NFS'd to HP then the compilation is OK. But
to implement this solution would mean that the developers were editing
source files through NFS. I could not cope if packets were lost because
of an NFS failure. The developers will definitely be using the HP box
for their work.

Does anyone out there know a solution to this? It may be to do with the
exclusive lock required by MF Cobol on source files while compiling and
the NFS versions for both operating systems.

HP told me today that the problem has been reported before (August 1995)
in that with the same hybrid situation, that MF Cobol 3.1 worked but 3.2
did not.

Is there some way to get this working?

Joan Colley
ADC Consultants
Sydney Australia.

MicroFocus say that the problem is not their software.

Funny how everything else works: cicstran, vi, MF Editor.

IBM suggest an upgrade.

HP suggest running IPTRACE.

Well I have created an ugly solution. remsh the cicstran because it
works. remsh a little script which copies the .int to a home directory,
then runs cob -u and copies the resulting .gnt back to the nfs
filesystem. This means of course that the compile script must only work
on HP although I was trying to write one which could be invoked from
either system.
