I know there's no way around this but I'm just aggravated enough about
it to gripe on the newsgroup, especially since it USED to work with
the old OS/VS Cobol.

I'm using Cobol for MVS and VM on OS/390. I think it's Cobol version
2.2.1. I have an unknown quantity of RECFM=VB files with unknown
maximum LRECL's. The goal is to have a universal logging program to
record files coming from external company sources. I want to open the
files as input and read the records so I can log the record count
amoung other things.

Problem is, in order to open a 'VB' file the max record length in the
FD has to match the LRECL otherwise you get the dreaded file status 39
open error. :( This is just dumb in my opinion and I know I'm going to
get all kinds of contrary opinions from others why they think it's
great.  The old OS/VS Cobol used to let you put RECORDING MODE V and
have a record with PIC X(32756) and it would open any 'VB' file on a
single FD.  That's the way SHOULD work.  That's the point of having
variable length records isn't it? They're variable! Why make the
maximum fixed? Why should the program care? Pick it up dynamically
from the system.

The other thing that makes this especially annoying is that you CAN
do just that with fixed length RECFM=FB files. If you say RECORD
CONTAINS 0 CHARACTERS with a record of PIC X(32760), it will pick
up the LRECL from the system and open ANY 'FB' dataset on a single FD.

So you can have variable - fixed length records but you can't have
have variable - variable length records.

I going to have to write an assembler subroutine to count records just
because of this one dumb limititation in Cobol.

.. Okay end of rant, I feel better.

A few "bits" of information on this:

1) In some (not all) cases, switching from NOCMPR2 to CMPR2 will allow you
to avoid FS=39 (on IBM mainframe compilers) - but will also prohibit you
using many '85 Standard features.

2) The ANSI/ISO committee was *very* clear that it is up to each implementor
(not the Standard much less the committee) as to WHICH "fixed file
attributes" would be checked to give a FS=39.  Therefore, if you don't like
IBM's "choice" on this - blame IBM, not the Standards committee (although I
have seen IBM "in print" claim that it was the Standard that required this).

3) Record Mode is V; LRECL, DCB, etc ... are all IBM-things.   The POINT of
FS=39 was (is?) to make certain that when you create a file (using COBOL-x,
i.e. implementor X's Release Y.Z) with certain attributes and then go to
READ it in a program created by the same compiler, that you are actually
looking at the same file defined the same way.  The reason that the Standard
was "clarified" to make it clear that not all "fixed file attributes" needed
to be checked - was that not all files are even created by COBOL much less
the SAME COBOL.

4) The next Standard will allow "dynamic" changing of files associated with
an FD (already allowed in many implementations - but not IBM's).  HOWEVER,
it will not let you dynamically change the "attributes" associated with a
file.  If this is something that you think COBOL needs, then

   A) tell ANSI/ISO to add it in a future Standard
   B) tell your "vendor of choice" to add it soon as an extension (some
already do - but not IBM)

Actually you can use a single program, with 1 FD for both fixed and
variable record lengths. The problem is that you are are forcing COBOL to
assume that the file is VB, when actually you do not know the true DCB
attributes of the file. What you really need is

FD  INFILE01
        RECORD CONTAINS 0
        RECORDING MODE IS U.
 01  INFILE01-REC.
       05  INFILE01-REC-CHAR OCCURS 1 TO 32756.
             10  FILLER  PIC X.


The key here is RECORD CONTAINS 0 which forces attributes to be accepted
from either the execution JCL or the file itself  *and*  the RECORDING
MODE U clause which tells the compiler that the file might be fixed length
or variable length and that it might be blocked or unblocked. IOW - You
really don't know right now, but will tell the program at runtime.

ISPF lesson 1:  variable length records :-)

To check a variable length record use 'HEX ON'.  Where the hex
numbers run out so does your record.

ISPF lesson 2:  variable length records :-)

If you have a record padded at the back with spaces and you edit
it (and actually save it).  You delete trailing spaces.  The can
cause some interesting results when you end with a binary field
that happens to be X'40' or the next program uses a record code
and assumes the record is that long.  For example if your end with
PIC x(40) which has 'happy birthday' you drop the last 20 odd
bytes when you edit it.  The next program to run will see the next
record in sequence as part of the current record (whatever happens
to be in the buffer past the end of record)  the output can be
*very* interesting.
