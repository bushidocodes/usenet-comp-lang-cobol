```
Well, I don't understand your question. XML is XML. What would XML for COBOL
look like?

On the other hand, if you want to write XML documents _in_ a COBOL program,
it's just like any other output file: you put out records satisfying the XML
formmating rules. So you have to know your data. Initially put out a header
line (<?xml version="1.0" encoding="utf-8"?> then build elements as you get the
data. An element would consist of an start tag, the data piece you want, and an
end tag. Such a program would, of necessity, be a different program for each
set of differently-structured input data.

If you're on a mainframe using the current IBM Enterprise COBOL compiler, you
can even translate the EBCDIC records to UTF-8 before writing them out, since
XML is supposed to be coded in Unicode.

Of course if you're on Windows or UNIX and not using large character sets you
are already using ASCII which is effectively UTF-8.
```

```
> I want to know if there is a tool / sources available to write XML documents
> for cobol ?

Writing XML is rather easy, it is just a text file with the data items
with tags and attribute names surrounding it.  You could code the
formatting into the program or, much better, have a template file that
has all the tags and attribute names that are required and also has
markers where the data items are to be put.

For repeating groups you could have markers that indicate the start
and end of the group, and these could be nested, or simply divide the
template up into blocks and have the program repeat the blocks the
number of times that the data item occurs.

If you wrote a callable module (or class) that could be called with
sets of data items then this could be used to write out many different
XML documents using appropriate templates and data items.

Change the template and it will write HTML, or EDIFACT, or CSV, or
Postscript, or anything really.

-------- template order.tem ----------------------------
*PACKED
*LINES 0
*OUTPUT directory/filename
:HEAD
<?xml version="1.0" encoding="ISO-8859-1"?>
<despatch>
   <shipment>
     <order><!%ORDER%></order>
     <reference><!%REF%></reference>
     ...
     <lines count="<!%LINES%>">
:LINE
     <line number="<!%LINE%>">
         <product><!%PCODE%></product>
         <ordered><!%QORDERED%></ordered>
         <sent><!%QSENT%></sent>
         ....
     </line>
:TRAILER
     </lines>
  </shipment>
</despatch>

I would have an array as the interface to the template routine that
is, say:

      01 Template-interface.
         03  Template-name     PIC ..
         03  Template-function PIC ..
             (Open, print section, close, etc)
         03  Template-Response PIC ...
         03  Template-section  PIC X(8).
             (HEAD, LINE, TRAILER in this example)
         03  Data-I            PIC S9(4).
         03  Data-items OCCURS 1 TO 100 DEPENDING ON Data-I.
             05  Data-name          PIC X(12).
             05  Data-Value         PIC X(40).

       MOVE ZERO    TO Data-I
       MOVE "ORDEROUT"  TO Template-Name
       MOVE "HEAD"      TO Template-Section
       MOVE "S"         TO Template-Function
       
       ADD 1        TO Data-I     
       MOVE "ORDER"     TO Data-Name(Data-I)
       MOVE ON-Number   TO Data-Value(Data-I)
       ADD 1        TO Data-I
       MOVE "REF"       TO Data-Name(Data-I)
       MOVE ON-Ref      TO Data-Value(Data-I)
       ...
       CALL "TEMPLATE" USING Template-Interface

The template program then has to read all the template file into an
array of text lines.  Process '*' commands to set options, process ':'
section names into an array of section names recording the line
numbers, ignore comment lines.

For each section print do a lot of INSPECT and UNSTRING/STRING on each
line.
```
