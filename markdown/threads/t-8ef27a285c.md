```
There have been a few conflicting and confusing items about REDEFINES
recently and I thought I would put out a small (well small for me) statement
on the topic.  Unless stated otherwise, all the following reflects the rules
of the ANSI ('85) Standard.  Many compilers have extensions that allow
additional uses - and (hopefully) define what they mean.

1) It is absolutely legal to have a REDEFINES on an 01 level in every Data
Division section *except* the File Section and the Communication Section.

2) In ANS'85 a smaller item IS allowed to redefine a larger item - but
normally not vice versa. (This is an "enhancement" from the ANS'74 Standard
where they had to be identical.) A ;larger item may redefine a smaller
item - BUT if and only if the item being redefined is an 01-level without
the EXTERNAL attribute.

3) A parameter in a Procedure Division USING statement may be the object of
a REDEFINES - but may not be the subject of a REDEFINES, for example, given

    01   That Pic X.
    01  This Redefines That Pic 9.

Then "That" may be in a Procedure Division USING statement, but "This" may
not be.

4) When multiple items all redefine the same item, the Standard requires
that you use the 1st item as the object of EVERY redefines - not the most
recent one.

5) You cannot REDEFINE an item that is defined with an OCCURS clause - but
you may REDEFINE an item subordinate to one with an OCCURS clause
```

I know that is the common term for it, but I always think that "confuses"
the issue.  In my terminology.  01 level items do "implicitly redefine the
same storage" - but I would never call it an "implicit REDEFINES".  They
probably mean the same thing, but I hate using any reference to REDEFINES
for 01-levels in FDs (implicit or explicit) - because it was such a
nightmare when IBM (on the mainframe) "turned off" the ability to do an
explicit REDEFINES for an 01-level in the FD - and did this in the "middle"
of a release as simply "fixing a bug".

to happen.

What about dealing with addresses in COBOL?

The following sample is IBM mainframe only code, so sorry if most readers of
this NG cannot do such nasty things on ther own computers. In this piece of
code, I go 4 bytes to the left of a working storage field (F1). I know that I
could do that much easier, but I want to show the principle:

       CBL TRUNC(BIN)
       IDENTIFICATION DIVISION.
       PROGRAM-ID. MAINPROG.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01 PTR1 USAGE IS POINTER.
       01 COMP-PTR REDEFINES PTR1 PIC S9(9) COMP.
       01 F1   PIC X(100) VALUE ALL 'X'.
       LINKAGE SECTION.
       01 F2   PIC X(100).
       PROCEDURE DIVISION.
           CALL 'ADDR' USING PTR1, F1
           SUBTRACT 4 FROM COMP-PTR
           SET ADDRESS OF F2 TO PTR1.
           DISPLAY F1
           DISPLAY F2
           GOBACK.
       IDENTIFICATION DIVISION.
       PROGRAM-ID. ADDR.
       DATA DIVISION.
       LINKAGE SECTION.
       01  AD-ELEMENT                      POINTER.
       01  ELEMENT-FIRST-BYTE              PIC X(001).
       PROCEDURE DIVISION USING AD-ELEMENT ELEMENT-FIRST-BYTE.
           SET AD-ELEMENT TO ADDRESS OF ELEMENT-FIRST-BYTE
           GOBACK.
       END PROGRAM ADDR.
       END PROGRAM MAINPROG.
