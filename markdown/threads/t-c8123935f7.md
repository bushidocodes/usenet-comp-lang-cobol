Ahangar wrote:

> What is the difference between a Class and a Factory in the proposed 200x
> COBOL standards? Can someone please explain it with a short example?
> Thanks

There is no quick answer to this one. Actual OO syntax like "invoke this
method-name using xxxxx returning yyyyy" is no more difficult to comprehend
than "call thisprogram using xxx returning yyy".

Structuring classes, Factory methods and Object Section methods is the 'guts'
of what OO COBOL is all about. There are so many permutations that you can
make. For a very brief outline :-

    *----------------------
        Class-id.                        ThisClass inherits from Base.
        *--------------------
        FACTORY
        method1
        method2
        method etc...
        END FACTORY
        *---------
        OBJECT SECTION
        method10
        method11
        method....
        END OBJECT
    *----------------
    END ThisClass.
*-----------------------

Think of a Class as your source program - which is what it is. The above
would have a source name of "thisclass.cbl."  But classes are used to gather
together 'like' procedures to be performed on logical groups of objects :-

    Class - EditCustomers
    Class - CustomerFile
    Class - CustomerDialog

Each of the above classes is an OBJECT, that is, the runtime maintains an
object reference. Using an Animator I can see the object reference displayed
as an Hex value stored in memory. EditCustomers would invoke CustomerFile
(for file data, READ, WRITE, REWRITE) and  CustomerDalog (to both display
existing data and accept new or changed data). Much of this is achieved by
INHERITANCE :-

                    Base (the 'mother' of all Supers) and 'the buck stops
here'
                    ClassA inherits from Base  (Base is the Super for A)
                    ClassB inherits from ClassA (A is the Super for B)
                    ClassC inherits from ClassB etc.....

At the current time Fujitsu = FJBase, Hitachi = HB-Base and Micro Focus =
Base.

If I invoke ClassC "do-this-method" and that method-name appears in ClassA,
because of the Inheritance chain, it 'looks upward' to find the method in
Class A.
Note that this happens automatically without me having to do the equivalent
of non-OO :-

            call ClassA entry-point "do-this-paragraph" using xxx returning
yyy

You don't necessarily need all the pieces I show in the initial diagram -
depends upon what you want to do.

Non OO - call Myprogram using these parameters
OO         - invoke MyProgram "method-name" using these parameters

Both function in the same way, continuing until you signal "'EXIT" which for
PCs would take us back to the application's Main Menu.

Factory - the key here is a 'Factory' is used to MANUFACTURE (process) or
BUILD objects. Sometimes you need a "new" object, sometimes you don't :-

- Single User - wanting EditCustomers - we don't need to differentiate
between Factory and Object Section so we invoke, say from a Main Menu :-

    invoke EditCustomers "begin" using Desktop

Here all the methods we are gong to use can be in Factory.

Using Desktop gives us a reference for Windows to display any dialogs Think
of "begin" as the Main-Line in a non-OO program - the starting point. I could
just as easily have used the method name "main-line" as an alternative to
"begin".

- Multi User - here we have 10 people using EditCustomers - so now Factory as
a 'controller' comes into play :-

    for User1 invoke CustomersEdit "new" returning os-CustomerEdit1
    for User2 invoke CustomersEdit "new" returning os-CustomerEdit2
    for User3 etc....

You don't "actually' return 1, 2 and 3 etc  as I've shown, but a reference
equivalent of 1, 2 or 3 to each user's PC.

Class : CustomersEdit
*>-------------------
FACTORY
method-id. "new".
linkage section.
01 lnk-self                    object reference.
Procedure Division returning lnk-self.
  invoke super "new" returning lnk-self    *> Super = Base
End Method "new".
End Factory
Object.
Method-id "begin".
  invoke Method2
  invoke Method3
  invoke etc....
Method2.
Method3
End Object..
End Class CustomerEdit
*--------------------------

Where I have "...returning os-CustomerEdit1, os-CustomerEdit2....." above,
each has an OBJECT REFERENCE or HANDLE.

Having got that handle now each user has :-

    invoke os-CustomerEdit "begin" using the methods in the OBJECT SECTION

Factory contains the method "new" and this creates an INSTANCE of the
class(program) as it will be used by say User #9. This is the bit which is a
little tricky to put across. Those 3 classes I started out with - times 10
users equals 30 programs. Right ? Wrong !  Only 3 - one copy of each. Even
though we have 10 users the INSTANTIATION allows each to reference common
methods but for each User there is a separate copy for their specific data,
held in Global Working Storage..

If that seems weird, you have to bear in mind that all methods are
transitory. You invoke methods in different ways :-

    StartProgram -  invoke ProgramC "do-this" using xxx returning yyyy
    ProgramC - invoke self "do-this" using xxx returning yyyy
    ProgramC - invoke super (ProgramB) "do-this-one" using xxx returning yyyy

*                    here I'm telling C to go to Super (B) to action a
specific method

When you have finished with a method - "End method",  the local-storage
variables in that method are 'gone'; that's not the case if you move values
into Global Working Storage.
-----------------------------------------------------------------------------

It's one hellava topic to try and paraphrase. If you are intrigued by OO, go
to cobolreport.com, a reasonable number of articles in their archives and you
could order a copy of Will Price's book "Elements of OO".

To summarize - if you have a firm grasp of non-OO COBOL you are 90% of the
way there on coding for OO. Literally, you have to let your imagination fly
on all the twists and turns you can do in OO to give you very, very great
flexibility - and that comes about using the hackneyed word REUSABILITY -
where you are constantly searching where you can re-use code, by segregating
into invokable classes, rather than keep on "copy and paste', "copy and
paste'...
.

The factory is part of the class.  Every class has a single factory object.  You
can have multiple instance objects, but only a single factory object of a class
exists.  This is the "constructor" as it were.  The home of the "new" method.
Any data that you want to exist for all instances of a class should be defined
in the factory and accessed via factory methods.  

To reference a factory method or property you specify the class name for the
object reference.

Will Price tried it, and I've tried it - I don't think the ISO2000 Directive does
much for you, other than trying to get the 'future' syntax - but I gave up as I had
compiler errors.

Where I'm using FACTORY, I'm not setting any directives - just using the word
'FACTORY" instead of "CLASS-OBJECT". To recap :-

    Class-id.
    FACTORY or CLASS-OBJECT (It is Class-object that you see in all M/F
        examples)
    Factory Methods
    END FACTORY
    *>---------
    OBJECT
    Object Methods
    END OBJECT
    *>--------
    End Class

Something which you are indirectly familiar with - my file handling. Given I want 10
different files all using a 'common' Class - Line Sequential,. I invoke Line
Sequential "new" 10 times with the method "new" in Factory. The 'common' file
specification (occurs a to b depending on c) appears at the top. The file access
VERB methods appear only in the Object 'Section', after Factory.

Brief example :-

*>---------------------------------------------
Class-id.              LineSequentialFile inherits from FileErrors.
Class-control.      LineSequentialFile      is class "lseqfile"
                           FileErrors                   is class "filerror"
INPUT-OUTPUT SECTION.
     Select               Data-File
     assign               etc.......
FILE SECTION.
FD  Data-File
    record varying from 0 to 1500 characters
       depending on DataRecordSize.
01  Data-record pic x(1500).
WORKING-STORAGE SECTION.
01 DataFilename                       pic x(100).
01 DataRecordSize                   pic x(4) comp-5 value 0.
01 FileStatus                             pic x(02).
*>----------------------------------------------------
FACTORY.
*>----------------------------------------------------
Method-id. "new".
*>----------------------------------------------------
Linkage section.
01 lnk-self                object reference.
Procedure Division returning lnk-self.
   invoke super "new" returning lnk-self
End Method "new".
*>----------------------------------------------------
End FACTORY.
*>----------------------------------------------------
OBJECT.
*>----------------------------------------------------
OBJECT-STORAGE SECTION.
*> above becomes W-S Section in COBOL 2002
01 os-Parent                        object reference.
*> This is my parent Window so I can pass this to FileErrors for a message box

*> Following is one of the methods applicable to a Line Sequential
*>-------------------------------------------------------
Method-id. "ReadNext".
*>-------------------------------------------------------
*> "dbresult" - I'm using a common file for Level 88's for both COBOL files
*> and DB Tables (MS Access)

Linkage section.
01 lnk-values.
copy "dbresult.cpy" replacing ==(tag)== by ==05 lnk==.
05 lnk-record.
   10  occurs 1 to 1500
       depending on DataRecordSize     pic x.
Procedure Division returning lnk-values.
  set ResultOK to true
  Read Data-File
    At End
      Set FileFinis to True
    Not at End
      Move Data-record to lnk-record
  End-Read

  Evaluate true
    when FileFinis and FileStatus = "10"
         continue
    when FileStatus <> "00"
         set FileError to true
         invoke self "fileError"
  End-evaluate

End Method "ReadNext".
*>-------------------------------------------------------
End OBJECT.
End CLASS LineSequentialFile.
*>-------------------------------------------------------
