Writing a conversion program in Microfocus Cobol 4.0.38, target

Basically reading an input file, doing a bunch of processing

output file.  Approximately 4.2 million input records (~2.1 GB).

Trying to optimize performance.

business rules. (e.g. reject records with certain fields blank, or

Successively work my way up to the more complex conditions until

Approximately 2.8 million records are accepted for processing.

The program doesn't run too bad, but I would like it to run better.

Of the eight internal tables used for lookups, one of them is huge

So, if THAT wasn't slowing the program down, where's it spending it's

I may try going back a compiler version to 3.2.43 where we have a
"gntanlz.lbr".  Maybe that's what it wants.

Anyway, next I tried the "PROFILE" compiler directive which is
described thus:

"Includes code in your program to produce detailed performance

That worked.  I ran the program.  Expected a report file or something.
 Instead what I got was a tiny, 1600 byte file with an ".ipf"
extension.  Contents not readable by an editor, and cannot find any
trace of what to do with this file nor how to process it anywhere! 
(Not MF documentation, nor MF/Merant knowledgebase, etc.)  ***HELP!***
 Any suggestions on this?

Anyway, since I know you're probably wondering, back to basics. 
Program:

- Loads 8 internal tables for look-up's.  Range from 55 records to
64K.
  These are used to "translate" values from old system to new.
  Largest three are 5,000, 10,000, and 64,000.
  Standard serial searches, except for largest which is "search all".

- Reads ascii, fixed rec length input file.  (4.2 million input recs)

- Reads three other MF indexed files.  Two are a direct read, the
other is a "start" followed by "read nexts" until control-break.

- Does some fairly tightly coded parsing on two of the input fields on
each record.  (Phone number field and an electronic serial # field)

- No huge, ungainly, complicated nested conditions.  I try to keep the
"IF" statements fairly simple and straightword and of limited depths. 
Just a lot of 'em.

- Spits out new record containing converted values, plus a variable
number of additional, secondary records based on associated
information read from that third index file.

- Also writes a summary, a detail, and an additional index file.

- Loops back for next input record.  Repeat until done.

That's it!

In advance, thanks for any input you can provide!

Can't specifically offer any solutions, but if you use some obvious (to
you <G>) keywords against your Help Files, Micro Focus certainly addresses

The only conversion I have ever done is to convert RM/COBOL V2 data to
M/F. But volumes were an absolute flea bite, compared to what you are
talking about.

It might just be your approach. Example, back to when Noah was hewing wood
for the Ark. 1965 - Unigate (dairy company), we were using an ICT (ICL)
1500 with RCA's random access MCF(Magnetic Card File). Input was on paper
tape and data was stored on mag tape. We only used the MCF unit for "ISAM"
lookups - it was too *new" to trust it wouldn't lose our transaction data
!

Volumes, I've absolutely forgotten, but as a rough daily figure :-

- say 60 distribution depots - some 200 deliveries a day     = 12,000
- 150 van salesmen - about 50 deliveries per day                =   7,500

Two things we needed to do (1) ISAM lookup for Product Code and (2) lookup
for Customer A/c #.

The input mag tapes were sorted TWICE - Product sequence, then Customer
A/c # sequence.

(Later of course a sort had to put the data back into Customer A/c #
followed by Product #'s for the delivery document - then we printed
invoices).

Those two sorts offered the following advantages (1) Our Product File was
wrapped on one magnetic card that continuously spun around the read-write
head. So look-ups were ascending, and if IN-Product # = Previous-Product #
- use same info without a further ISAM search. Same goes for the Customer
file, although that was on some 20 cards, but nevertheless, each card was
pulled in ascending sequence. (Again if IN-A/c # (F.W. Woolworth) =
Previous-A/c # (F.W.Woolworth) - no search).

Archaic, compared to what you can do today. But given that you say your
Search Tables are sorted, might you benefit from the input going through a
couple of sorts, so that you access the tables strictly ascending - or
perhaps even use a pointer like an SQL Cursor ? Might not make the
slightest bit of difference, e.g. any sorts might eat up any time you were
trying to save - just mention it as a possible approach.

Now there's a thought - could you put those look-up tables in a DB - might
just give you the edge.

We'll stop here. I betch dollars-to-donuts virtually all your overhead is
spent in I-O. You would be AMAZED at how much work a PC can do in the time
it takes to fetch a single record.

So, then, speed up the I-O.

How?

Well, you already tried one trick: move all the files to the local machine
before processing then move them back when done. That way, each record moves
over the network only twice.

Someone else suggested splitting the files between several disks. That's a
good idea. Another is a second disk and a RAID controller. Another idea is
getting faster disks. Another is changing the cluster size on the disk you
have (512byte vs 4K clusters). Implement 'Dirty Writes' in the OS (if you're
not familiar with the term, check it out). If much of the work is
sequential, implement the "Read Ahead" feature of the disk drive or OS.

You might see a suggestion about not fetching a record if you already have
it; that's probably not necessary inasmuch as operating system sorts that
out for you - that is, if you want a record already in the OS's cache, the
OS won't do a physical read.

The EASY way to find out if fine-tuning the code will help is to run the
program on another machine that's twice as fast as the production machine.
Does that help 10% (the program is I-O bound) or does it help 50% (the
program is compute-bound).

Anyway, for that much data, my money is on I-O bound.
