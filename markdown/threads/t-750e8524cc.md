```
We have copybook describing a file/record layout with fairly std fields:
05  ABC123-A         PIC ....
05  ABC123-B         PIC ....
etc. There are a couple of hundred fields. We also have a very old program
that does a COPY of this copybook but it also needs 4 other concurrent
copies of this copybook. So because you can't replace the prefix part of a
data item with COPY REPLACING, we actually have 4 hard coded copies of this
copybook in the program. Needless to say, when we add a new field to the
copybook, we sometimes forget to change these 4 hard coded copies. The other
option of course is to make 4 new copybooks with different prefixes.

Has anyone discovered a way to 'fake out' COPY REPLACING so that you CAN
replace the prefix part? Our platforms are AS/400 and UNIX. Thanks.

David Silber
DavidS@hpd.co.uk
```

```
Two answers:

1)  Contact your "vendor of choice" and ask them WHEN they plan on supporting
the COPY REPLACING *leading* option from the draft of the next COBOL
Standard.  (This probably will not get you a useful/timely answer - but will
get your COBOL vendor at least looking at this.)

2) If you are willing to change your copybook (which it sounds like you might
be doing anyway), create a new member coded something like:

  05  (ABC123)-A         PIC ....
   05  (ABC123)-B         PIC ....

Then do a COPY ... REPLACING ==(ABC123)== by
                                                         ==A-Prefix-==.

and you should get what you want.  You will NOT however, be able to use this
same "copy member" in your existing code - or any new code that does NOT use
a REPLACING phrase.
```

```
I've seen this on UNIX and MVS. You can replace the prefix part with COPY
REPLACING.

Just have one copybook and in it have a dummy prefix. Something like this:

05  PRFX-A             PIC ...
05  PRFX-B             PIC ...

and so on.

Then have 4 copy replacings, like this:

COPY copybook REPLACING ==PRFX-== BY ==ABC123-==.

That's 2 equal signs on each side of PRFX- and ABC123-.
```

```
The "prefix" replacing only works if the prefix
part is not percieved as part of a standard COBOL
token. That's why the (ABC) as prefix worked as
that is in fact three seperate COBOL tokens.
These get REPLACED correctly. I'm not even sure
if this behaviour is supported by the standard,
but works in IBM compilers so it has become a de-
facto standard for other compilers as well.
```
