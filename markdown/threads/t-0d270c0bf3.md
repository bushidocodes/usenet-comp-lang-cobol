COMPUTE :

OK, you can express a problem as several lines of code which will work.
Up here in the Great White North we have GST (Goods and Services Tax), 
just like the EU's VAT.

You want to sell something for $100 but there's GST of 7% - therefore 
final price $107.

How much GST in $107 - EASY - $7. But how much GST in $253.77 ?
Pseudo code :

a)GSTNumber = 7/107
b)GSTAmount = SalePrice - (GSTNumber)
c)TicketPrice = SalePrice - GSTAmount

Above code does the trick, but prof/teacher wants you to come up with 
something more elegant, like :-

Compute TicketPrice Rounded = SalePrice - (SalePrice * 0.0654)

ON SIZE ERROR

Why is he asking you for this ? He's told you to allow for pic 9(09), 
"nine-slots" for a number. So if you and I add 1 to 999,999,999 we know 
the answer is 1,000,000,000. But what happens if you add 1 to 
999,999,999 contained in a pic 9(09) ? Hence the ON SIZE ERROR. (Not too 
many people use ON SIZE ERROR by the way, ensuring their fields are big 
enough to take the largest result - but your prof/teacher is quizzing you).

- I used three variables (fields)
- I used a PERFORM/END PERFORM varying n from 1 by 1 until n > 10
- One line of COMPUTE
- then I MOVED values from RIGHT TO LEFT
- I didn't bother with ON SIZE ERROR, just confirming my number 
increments were correct.

In your case, presumably, you enter a start number which is reasonably 
close to 999,999,999, so that you don't have too many iterations of the 
PERFORM. So something like :-

	COMPUTE this.............
		ON SIZE ERROR do this
		NOT ON SIZE ERROR
                  display result, (OR write result to a file record) 
                                                 	END-COMPUTE

In the same vein, here is a program that also solves the OPs problem in
a slightly obscure fashion. It uses a circular buffer to hold the data
and instead of moving data around, it moves pointers to the data. (If

       IDENTIFICATION DIVISION.
       PROGRAM-ID. FIB.
       ENVIRONMENT DIVISION.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01 PTRS.
         02 P PIC 9 VALUE 1.
         02 Q PIC 9 VALUE 2.
         02 R PIC 9 VALUE 3.
       01 VARS.
         02 N PIC 9(9) OCCURS 3 TIMES.
       01 FLAGS.
         02 OVF PIC X VALUE "N".
       PROCEDURE DIVISION.
       MAIN.
           MOVE 1 TO N(Q).
           MOVE 0 TO N(R).
           PERFORM SUB UNTIL OVF IS EQUAL TO "Y".
           STOP RUN.
       SUB.
           DISPLAY N(Q).
           COMPUTE N(P) = N(Q) + N(R)
               ON SIZE ERROR MOVE "Y" TO OVF.
           SUBTRACT 1 FROM P.
           SUBTRACT 1 FROM Q.
           SUBTRACT 1 FROM R.
           IF P IS EQUAL TO 0 MOVE 3 TO P.
           IF Q IS EQUAL TO 0 MOVE 3 TO Q.
           IF R IS EQUAL TO 0 MOVE 3 TO R.

Maybe it is not as efficient, but it has the conceptual advantage that
you are not shuffling data around.

To answer this (and the other post),

PERSONALLY (again style - but explaining my preferences),  I don't like
   ON SIZE ERROR statement-1 . (period) statement-2

or
   IF condition-1 statement-1.  (period) statement-2

within paragraphs UNLESS one wants to do the "statement-2" logic WHETHER OR NOT 
the first test passes.  As I "analyzed" the original code, although it would (as 
the 2nd post indicated) do the following statements, it seems to me that it 
either should not or would "gain nothing" from doing it, i.e.

When ON SIZE ERROR occurred, the OVF was set to "Y" so whatever happened later 
in the paragraph was never "reached" in a subsequent PERFORM (so no "useful" 
code was executed).

In the cases of the tests for "3" - I think the logic requires that one AND ONLY 
ONE of the items could be 3 during a single iteration of the paragraph.

  ***

That is why I would PREFER "nested" the tests so that one only hit logic that 
COULD be true during this iteration of the paragraph.

(I would have had no problem with nested IF's rather than EVALUATE, but it 
seemed to me that where all 3 tests were against the same value, this would be a 
"good" place for an EVALUATE test for "3").

preference"?
