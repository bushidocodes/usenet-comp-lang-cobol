```
This was one of those days I shouldn't have gotten out of bed.  Several things
happened before I got to work that should have given me a clue it wasn't going
to be pretty.

We have a client/server set of programs that because I finally figured out how
to do it I'm the expert in.  The batch side (client) does all the computations
and data manipulation and the CICS (server) side writes the records to the
on-line VSAM file.  This particular VSAM file is being converted from variable
length to fixed length (yeah! finally!) and my cubemate was given the task to
test the changes. 

The only thing changed in the programs was the length of the commarea to
accomodate the change in the length of the records (I know - famous last
words). She was merrily testing away when the whole thing blew up.  After
drilling down through the code of both programs (the return code received by
the batch program indicated the CICS program had failed but there was no dump
and we couldn't for the life of us find an abend code that made any sense) we
determined that part of the commarea contained spaces when there should have
been data in it.  The worst part was that particular area was smack dab in the
middle of the commarea.

Several hours later and many many displays later I finally decided to put in
one final display and we had it.  

The commarea is defined thus:

01  WS-COMMAREA.
      05  WS-COMM-TYPE   PIC X(6).
      05  WS-COMM-SEQ    PIC S999 COMP-3.
      05  WS-COMM-HEAD  PIC X(350)   ****
      05  WS-COMM-DETAIL OCCURS 4 TIMES PIC X(350) *****

*****  fields were originally PIC X(250) and PIC X(90) respecitvely.


In the procedure division I had this:

PERFORM 4 TIMES
     MOVE SPACES TO WS-COMM-DETAIL (SUB)
END-PERFORM

Well needless to say the subscript was never incremented and WS-COMM-DETAIL (0)
was spaced out.

The first time the CICS program is entered a file containing the invoice
numbers is read and the field in the Header is populated and the Header written
to the VSAM and then returned to the batch.  The next time the CICS program is
entered the Details are written out and only the first 7 positions of the
Header are used.  Since in the old version of the program WS-COMM-DETAIL was
only 90 positions long, the last 90 positions of the 250 were spaced and the
bug wasn't caught.

How long had this been going on?   About 18 months.

Pass the weed please.

Eileen
```

```
This actually relates to the optimization thread.

This is a good example of why subscript range checking should always be 
active. But this is expensive unless it's designed as part of the 
instruction set architecture, so that it can be done in parallel 
without delaying the CPU. Thus good design up front promotes not only 
efficiency but also correctness. It's a feature that cannot be added on 
later without incurring large performance costs.

Unisys (previously Burroughs) MCP systems are designed with subscript 
range checking always active. My experience with these systems is that 
it helps programmers and reduces costs considerably.
```

```
news:01HW.B8F80E4F0038DFAD0DAA47E0@news-east.usenetserver.com...

> Unisys (previously Burroughs) MCP systems are designed with subscript
> range checking always active. My experience with these systems is that

I agree with the sentiment, but the details are not quite right.

The MCP/AS hardware is designed such that it is impossible to access outside
the physical array that contains the logical record, that's true.  In the
example cited, the subscript would have to be outside of the bounds 0
through +4 to encounter this interrupt.  To get the "granularity" required
here requires explicit object code from the compiler.

By default, MCP/AS COBOL74 does not do "detail" range checking *on indices*
(relying on the "container limits" and the various hardware index-limits
interrupts).   It does do "detail" range checking on subscripts, and that
logic can be omitted by setting $OPTIMIZE.   If "detail" range checking is
not specified, SUB would have to be -1 or less (or +5 or greater) for the
hardware interrupt to occur.  If it is specified, the lower limit is +1.

MCP/AS COBOL85, on the other hand, does explicit range checking by default
for both indices and subscripts.  This is partly because its storage
allocation algorithms make reliance on the container boundaries pretty much
ineffective, and it's difficult to say exactly what the lower and upper
limits provided by the hardware without compiler assistance are.  The detail
range-check logic can be omitted (for both subscripts and indices) by $RESET
BOUNDS (INDEX).

    -Chuck Stevens
```

```
On Fri, 3 May 2002 11:18:08 -0400, Chuck Stevens wrote
> I agree with the sentiment, but the details are not quite right.

Ah, yes. Chuck is right of course. I forget that the implementation has 
changed.

With COBOL on the Burroughs B6700 -- circa 1970 -- there was indeed 
architectural support for combining address calculation with subscript 
checking in this kind of situation. This was 1968 COBOL of course -- at 
that time no one thought of distinguishing one version of the standard 
from another.

However, this architectural support proved rather inflexible and was 
gradually dropped over the years. The particular instruction still 
appears to be supported in the ISA, but as Chuck says, the COBOL 
compilers no longer generate it, at least not in this context.

So the current implementation proves that you can get excellent 
performance even WITH subscript range checking and WITHOUT 
architectural support.

Edward Reid
```
