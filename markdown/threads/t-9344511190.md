i understand p-code to mean "psuedo-code". doesn't help much does it? :-)

essentially, it's an instruction set that is machine independent. the code
is interpreted by a run-time system. the run-time system is providing a
"virtual machine" upon which the code can run.

personally, i wouldn't call this process "compilation" -- to me that means
you end up with an object file that contains the machines native instruction
set. hence "psuedo-code"...

I would :)

It depends on what this "virtual machine" looked like. In COBOL terms, if it
had an op code for "STRING" and one for "DISPLAY" etc (ie, one for each
COBOL statement) then I'd agree it's more of a translation than a compilation.

However, p-code tends not to be a direct representation of a given language,
so it would require some compilation techniques to produce (ie conversion of
high-level constructs into a (hopefully efficient) succession of primitive
ones).

› file that contains the machines native instruction set.

But what is that, other than p-code that a CPU happens to be able to execute
very quickly :) You could write a portable interpreter for 68000 m/c. Isn't
the 68000 instruction set then a form of p-code ? Now, where have I heard that
before ....
