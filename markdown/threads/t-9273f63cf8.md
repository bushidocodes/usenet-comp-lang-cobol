```
A period in COBOL is not like a statement delimiter in C.
The only place a period is required in the procedure division
is procedure division itself, paragraph names and a period must
be the last thing in a paragraph. You can do everything else without
any periods. However, if used a period functions as a hard
scope terminator. A complex series of nested if's can be
terminated with one period instead of a long series of
end-if. I've sometimes had long series of IF, IF, IF ELSE
IF ELSE ELSE IF etc. One period, splat ends the whole structure
without a bunch of END-IF END-IF END-IF etc. or a
whole bunch of braces as in C. Whether this is good or not
depends on who you ask. As you discovered a period will also
terminate the scope of an in-line PERFORM or other verb.
General rule, a period hard ends whatever is above it.
Don't put a period in the middle of something.
These are all functionally equivalent, that is
move, add, divide are subject to "if a = b"

if a = b
  move c to d y
  add f to g
  divide f by g giving x rounded remainder z.

if a = b
  move c to d y
  add f to g
  divide f by g giving x rounded remainder z
end-if

if a = b
  move c to d y
  add f to g
  divide f by g giving x rounded remainder z
end-if. <-
          this period is allowed but not required
          unless it is the end of the paragraph



As an aside, COBOL does not have to be in all upper-case if
you have anything newer than a 20 year old compiler. Also,
data-names (variables) are not case sensitive as they are
in C. Counter = countER = CoUnTeR, it is all refers to the
same variable.
```

```
If only you knew the debate that has occurred here over this issue... 
but I'll try to explain it for you in a nutshell.  :)

Craig Franck wrote:
> WROX says "a  period signifies the end of a paragraph
> (and as you will see later, can be the end of a sentence or
> of some statements);"

This is the key.  A period (full stop in UK and NZ) delimits a sentence, 
just as it does in English.  Prior to COBOL 85, you had to end a 
conditional statement (among other statements, such as file i/o) with a 
period.  However, COBOL 85 introduced "scope delimiters" which you can 
use to terminate a conditional (again, among other things).  This opened 
up some flexibility as to when you have to use periods.

There are two prevailing philosophies.  The first is similar to 
pre-COBOL 85, except that the scope delimiter is included.  The second 
is based on the rule that a paragraph must end in a period.  Using this 
style, you put a period and a space -before- each paragraph name (and 
one immediately after the name).  So, under concept one, you might have 
code like...

v (column 7)
  Procedure Division.

  Driver-Paragraph.
      Perform Hello.
      Perform 10 Times
          Perform Hello
      End-Perform.

  Stop-Run-Paragraph.
      Stop Run.

And with the other, you would have...

                    v note omitted period
  Procedure Division

  . Driver-Paragraph.
      Perform Hello
      Perform 10 Times
          Perform Hello
      End-Perform

  . Stop-Run-Paragraph.
      Stop Run

  .
  ^ note ending period

Personal preference - #2.  You can cut and paste code between 
paragraphs, add If statements, and all that stuff, without having to 
worry about whether you deleted periods or moved them to the proper line.
```

```
Ahh, what a can of worms this could open.  What follows is my opinion on 
periods.

A period is an absolute scope terminator, once used it stops everything.  
This caused some problems in previous versions of Cobol when people 
tried to add structure to their code.

So in 1985 the explicit scope terminators arrived to correct this 
problem.  They are quite similar to the end-curly-brace from C, and all 
scope initiating keywords have an implicit start-curly-brace.

For example, in the case of C, a "while (cond) {  statements  }" would 
be identical to the cobol "Perform cond statements End-Perform", with 
the implicit start-curley-brace between the cond and the statements.

To anyone who groks block structured languages this makes perfect sense.  
Each Keyword starts a new scope, and is terminated by the cooresponding 
End-Keyword.

If all of this sounds too simple, you are right.  What the 85 standard 
did not do is remove the nuclear bomb-like power of periods to terminate 
scope, so now you could nest keywords and still stop everything with a 
single period.  

If you had the statements in C or COBOL:

   if(cond1){        if cond1
      statements        statements
      if(cond2){        if cond2
         statements        statements
      }                 end-if
   }                 end-if

you would expect that the start-curley-braces, explicit or implicit, 
would have a 1 for 1 pairing with the end-curley-braces.  But because of 
the periods absolute scope destructive power, this is not true, you 
could have:

   if cond1
      statements
      if cond2
         statements .    <=====this period is now 2 "end-ifs"

What is more, anything above the first if is also terminated.  This is 
the real problem that I have with thier use.  As a practical matter, 
when they are used nesting beyond a single level is rarely used, and 
structure suffers because of it.

There are several other problems with their indiscriminant use like 
source maintenance errors and cut&paste scope errors.

There is good news in all of this though -- there are only two places 
you need to use a period in post-85 cobol.  The first is after a 
paragraph-name label, the second is at the end of the paragraph.

If you avoid periods in  all other places, and use explicit scope 
terminators (END-*) instead your code will not suffer from these 
problems.

My best advice is to let periods, as scope terminators, whither and die.

Joe Zitzelberger
```

```
Hi Craig,

If you want to avoid the confusion while learning the basics, my
advice is to abandon the periods except at end of pgraphs and after
pgraph heads and to terminate a COPY stmt (they're required). Use
scope delimiters where appropriate and if multiple END-IFs get
tedious, use a period in those situations if you want to. You DON'T
have to be "religious" about ant of this.

It's also a good idea to put the periods used in the P DIV on a
separate line. It's another way of avoiding copying a period into the
middle of stmt when "borrowing" code from another part of your pgm and
causing an error.

Here's a rewrite of your code (without the comments) using what I said
above:

       PROCEDURE DIVISION.
           PERFORM HELLO
           PERFORM UNTIL Counter = 10
              PERFORM HELLO
              ADD 1 TO Counter
           END-PERFORM
           .
       PROGRAM-DONE.
           STOP RUN
           .
       HELLO.
           DISPLAY "HELLO"
           .

Try it, you'll sleep better. :)

Regards, Jack.
```

```
<G>, well, it does, it just does not have anything to do with paragraphs.
The "goback" statement, the "exit method" statement, the "exit program"
statement, and the "stop run" statement can are all examples of explicit
return statements. There is also an implied return statement at the end of
every program ... but they all return from *programs/subprograms* rather
than paragraphs.

In fact, the paragraph name and the paragraph that go with it are NOT
subroutines in the generally accepted meaning of the word.  The paragraph
name is simply a label for all the code that follows it up to, but not
including, the next paragraph name.

If you simply list statements with interspersed paragraph names, the
statements will execute one after the other in an identical fashion to a
basic program with all lines numbered.  The paragraph names in and of
themselves are simply place markers.

The big difference between cobol and other languages is not the return
mechanism, but the calling methodology ... the perform statement.  If you
want to compare it to basic, think of an entirely new type of "call", where
you can specify both the start and the end of the subroutine in the call
statement.  For example "call line 300, treating line 556 as the return".
That would be the rough equivalent of the "perform thru" in Cobol.
```

```
It may be that you think of C's (for example) return as being 'go back
from whence you came', and thus should be in other languages.  This is
not strictly so.

A C void routine (function) need not have a return statement, the end
of the routine will cause the 'from whence you came'.

void testme()  {  puts("in testme"); }

The return statement indicates the _value_ that is to be returned, and
thus a non-void function must have a return statement.

As a Cobol PERFORMed paragraph cannot return a value there is no need
for an explicit 'return' statement.
```
