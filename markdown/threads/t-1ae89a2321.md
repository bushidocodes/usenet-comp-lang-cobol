In general, MF COBOL uses the ANSI character set. Windows does not.
Windows provides a pair of API functions to deal with this, CharToOem()
and OemToChar(), where Char = Windows and Oem = ANSI.

The call is pretty trivial

 SPECIAL-NAMES.
     CALL-CONVENTION 74 IS APIENTRY.

 CALL APIENTRY "OemToChar" USING
      BY REFERENCE my-source-var
      BY REFERENCE my-target-var

 my-source-var and my-target-var can be the same variable.

You have to find out whether the MF application prints extended ascii or
ansi. It does not help very much that your application is ANSI if your data
is ascii. Set up the printer driver to print to a file. Open that file in
notepad and edit. Wherever the result is correct give you the answer. If it
displays correct in edit, then it is ascii, if it is correct in notepad it
is ansi. For the latter case, there must be something wrong with your
printer driver. If it is in ascii, you will have to continue inspect replace
if not MF has a switch you may use that translates on the fly.

Acucobol which I use, has a runtime switch which does the translation for
me.
Like:
    SET TRANSLATE-TO-ANSI TO 1.
    WRITE PRN-REC FROM "This is not an ansi string".
    SET TRANSLATE-TO-ANSI TO 0.
    WRITE PRN-REC FROM "This is in ansi".

I have set the TRANSLATE-TO-ANSI in my program startup once and for all,
hence no problems. Ask MF if they have something similar. If not, you will
have to deal with it manually as you do. Or, you may write directly to the
printer bypassing the spooler. That demands that you have set up the printer
to correct character set though (which may be a tedious task).

Cheesle
