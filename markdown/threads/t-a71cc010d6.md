The Syncsort manual I have is old (and so is the version I'm using,
3.4, back from 1989), but since you've mentioned cylinders, I take it
you're on a mainframe?

If you are, it basically comes down to providing enough SORTWKnn
space, at least a little larger than the sorting file, and preferably
about double.  Your documentation should contain a recommendation
formula.  If it doesn't, post back (with your version and OS), and
I'll dig through mine and check.

Example, if I have a file with SPACE=(221,(200,100),RLSE),AVGREC=K
wherever the SORTIN file is being created;
then I usually try for SORTWK01 DD
UNIT=WORK,SPACE=(221,(400,200),RLSE),AVGRECK=K.

If you're experiencing a shortage of usable sort space at your
location, you might want to consider cutting down the amount.



BTW: While Syncsort (at least in 3.4) will allow you to use SORTWKn as
long as you're only doing SORTIN and SORTOUT, if you have multiple
output files using SORTOFnn, you must have SORTWKnn.  That is to say,
you have to have two digits after SORTWK, or Syncsort will be unable
to recognize the ddnames.  I don't know if this has been corrected in
later versions of Syncsort.

That would depend on the length and type of the chosen sort, include, and
omit fields which are used. It would further depend on the current file
size, output file(s) size(s) and the available space on the particular
devices which are available to your DASD class,job class, or whatever is
