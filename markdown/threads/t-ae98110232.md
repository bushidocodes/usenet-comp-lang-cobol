Now as to this, in order to read any record on a tape dataset (in MVS), you
must first read through every other record; so yes, if you had a 47 volume
tape dataset, and only wanted to add records to the end, the tape device
would have to read all the way through all 47 tapes to the last record
before whatever program you're using could write new records.  I'm not sure
if MAGSTAR or the other new virtual tape devices get by this, but on old
devices where your dataset is on actual physical tapes, the devices need to
go through every tape in a multi-volume dataset.

The DISP=MOD has no effect on that because the DST is comprised of
multiple VOL. SER numbers. With out identifying a specific volume you will
always get all of the tapes, operators can't guess which one to mount so
they start from 1 until all have been mounted. And if your shop has an
automated mounting device machines can't guess either.

>
>
> Given that I can't code a particular Volume (because on the first of the month
> I don't have one), any ideas how to get to last volume of a tape dataset
> without going thru all of the other volumes?

You might have an option if you set up a separate job for the first work day,
first day of the month. But you still will need to keep track of all of the volume

numbers to know which one to mount. That is not very nice or very easy to
accomplish. In fact it is very messy. I have seen something like this and it was
not pretty.

I will see what I can find on this but I don't think you really want to do
this, IMO.

I can't comment on the tape errors you received since you did not identify what
they were. As for as the job itself, you may want to create daily tapes, and have
a weekly merge job to combine these tapes,and a monthly job to combine the weekly
tapes. In general, I use SORT  with OPTION COPY for these kinds of jobs. It's much
faster, gives you record counts, and will abend on I/O errors. Also, check that
the JCL has BLKSIZE=0.
