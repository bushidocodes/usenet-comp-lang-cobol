That term comes from (where else?) the punched card days.  A digit is
represented by one hold in a given column in a card, but a field may be
negated by punching another hold in the 11 zone of the low end of a numeric
field.  If a positive value is needed, another punch in the 12 zone (very
top of the card) would be added.  The two punches in one column usually
translate (in punched card format) to alphabetic characters.

A + zero = {     positive 1-9 is A-I respectively    [uppercase "i" not a
digit]
neg zero = }    negative 1-9 is J-R respectively

If a field in cols 1-4 represents an unsigned value of 1234, that is exactly
what you would find in the first 4 columns.  A positive 1234 would be 123D
while -1234 would be
123M

This scheme makes much more sense in EBCDIC than in ASCII when you look at
the hex code for a digit (signed or not)--it is continued in ASCII for
compatibility with older machines, programs, etc.

HTH

The IBM punch card can contain all 256 possable byte values.  If
you still had a keypunch and card, the punch combinations for most
of these values would be entered by holding down the "overpunch"
key and then pressing several key to punch the individual rows
needed to make up the desired code.

Ascii uses byte values 0-127.  In EBCDIC, a zero is 250 or x'F0'. 
In ASCII, a zero is x'30'.  If you had a keypunch and needed to
punch an ASCII zero into a card, you would have to look up the
special code for x'30' and use the overpuch key to enter it.  This
is because there is no EBCDIC character assigned to x'30' which
appears on the keypunch keyboard.  (Or, you could find an ASCII
keypunch!).

If the data on the card is binary decimal data, then no conversion
is necessary.  x'30' is 48 decimal in both EBCDIC and ASCII.

If the data is character data (or decimal data) then you read the
data business as usual.  BUT as soon as you read it, you need to
translate the character field from ASCII code to EBCDIC.  INSPECT
REPLACING is the verb.

For MVS, if a record is all ASCII Characters, code OPTCD=Q on the
DD statement to have the system translate from ASCII to EBCDIC
during the read operation.

If you are sorting or comparing character fields that contain
ASCII data (i.e. have not been translated) the you need to adjust
the character set or COLLATING SEQUENCE in your COBOL program.

Overpunched Data:

In the olden days of punched cards, there were 12 holes in each column.
Ten of them were labeled 0 thru 9, the other two had various names: 12 and
11, 11 and 10, X and Y, or + and -.  Various combinations were used to
indicate alphabetic characters and special characters. 

The letters A thru I were the 12 (+) punch plus a digit 1 thru 9.  The
letters J thru R were the 11 (-) plus a digit 1 thru 9.  The letters S
thru Z were the 0 plus a digit 2 thru 9.

Digits were indicated by just punching the apropriate digit (0 thru 9).
Signed fields were indicated by overpunching the sign over the first or
last digit position.  On IBM 360 and its successors it's almost always the
last digit position.

So if the value +123 was put in a three byte field you end up with "12C".
The last position is a combination of a 3 and a +.  Here's a conversion
table:

Unsigned  Overpunched   Overpunched
 Digit       Plus          Minus

   0           {             }
   1           A             J
   2           B             K
   3           C             L
   4           D             M
   5           E             N
   6           F             O
   7           G             P
   8           H             Q
   9           I             R

These symbols would be generated in EBCDIC, but they survive translation
to ASCII, because these are all valid ASCII characters.

All the forgoing assumes that you are actually talking about overpunched
data.  If you are actually getting packed data, then you are screwed,
because it does _not_ survive EBCDIC to ASCII translation.
