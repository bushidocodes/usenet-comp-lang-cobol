Without your input file, it's difficult to know for sure.  I did see a 
few things, though, at which you may want to look.  (I'm assuming that 
it compiles and executes without erroring...  If not, that's a good 
place to start. :> )

> ******************************************************************

…[9 more quoted lines elided]…
>              PERFORM 201-IF.                                      

The compiler I use requires the use of End-Perform to terminate an 
inline perform like this (and you'd also need an End-Read to complete 
the Read clause).

>      PERFORM 165-CLOSELINE.                                       
>      PERFORM 170-TOTALLN.                                         
…[19 more quoted lines elided]…
> ****THE NEXT LLINE CREATES THE LINES ON THE REPORT           *****

This Write statement seems to be in Area A.  Again, the compiler I use 
would gripe severely at this, and probably would not include the line in 
the object module.  If this is the case, and all of your input records 
are hitting this error routine, that would explain why nothing is being 
written to your output file.

Best of luck - analyzing and debugging programs is often more 
complicated than writing them in the first place.  And, the problem with 
it is that you only get that skill through "experience" (so think of 
this as great preparation for the real world!).

(I'm also assuming that your code conforms to whatever style the class 
dictates - I personally find If statements easier to track if I indent 
all the statements under it, not just when there's another If.  For 
example...

      If WS-A = WS-B
          Move WS-C to WS-D
          Display "Isn't this Useless Code?!?!" upon Printer
*        This next If should always be true
          If WS-C = WS-D
              Move WS-D to WS-C
              Display "Still Going!" upon Printer
              If WS-E = WS-F
                  Display "...and so on" upon Printer
              Else
                  Display "E and F are not equal" upon Printer
              End-If
          End-If
      End-If

That way, the End-Ifs match up with the Ifs.)

     else
         perform 220-errrtn
     end-if.
     MOVE WS-D1 TO D-O
     IF GROSSPY-N IS NOT NUMERIC
         OR NEPY-N IS NOT NUMERIC
         continue
     else
         perform 220-errrtn
     end-if.
     MOVE WS-E1 TO E-O
     (???)

     MOVE WS-F1 TO F-O

     if error-found = 'N'
         PERFORM 155-PROCESS-RECORDS
     END-IF.

220-errrtn.
    move 'Y' to error-found.
    [...]

## Best practice is to check for all possible errors - not simply bailing at
the first one found. Otherwise, the user may correct the first error,
resubmit the job, only to have the job returned with another error on the
same data.


## You can improve the error-checking / reporting section by giving the user
a clue as to what the error might be. For example:

    IF STCD-N IS > 0
       OR STCD-N IS < 5
       MOVE 'Bad STCD-N parameter' TO ERROR-MESSAGE
    [...]

220-errrtn.
     Display ERROR-MESSAGE
     [...]
