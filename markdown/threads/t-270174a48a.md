```
I recently have seen code containing multiple definitions of the same name as in

01 FILLER
   02 item-a PIC X.
   02 item-a PIC X.

I wonder whether Cobol really allows this.
My original understanding of the ANSI Cobol 85 standard was no.
However, after seeing this fragment in a real piece of code I' not
sure any more (even after revisiting the standard).

The same code also contained the fragment

77 item-b PIC X.
   88 cond-b VALUE 'A'.
   88 cond-b VALUE 'B'.

Does the condition cond-b now actually mean 
  item-b == 'A' OR item-b == 'B'
?
```

```
The '85 Standard (unlike earlier Standards) explicitly allows you to have
non-unique and impossible to make unique via qualification data-names.
HOWEVER, this is only in the case where the items are never used (referenced)
in the procedure division.  This is "useful" <G> in the case of COPY
statements that bring in data-names that are never used.  Other than that,
they serve no real "purpose".
```

```
Sorry for being "pedantic" - but "filler" (whether specified or implied) is
NOT a data-name - but an alternative to specifying a data-name in a
data-definition.

HOWEVER, the rules (for multiple uses) are the same, i.e. it is legal to have
multiple FILLER items, but you cannot reference them anywhere.  (similarly,
as stated elsewhere in this thread) you can have multiple data-items with the
same name (including all levels of qualification) - as long as you never
"use" them.
```
