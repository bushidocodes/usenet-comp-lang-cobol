I thing the UNSTRING is the place to start.  Use the comma as your
delimiter, and allow significantly more receiving fields than the
number of fields in the CSV record.  You might also want to add a
comma to the end of the record before UNSTRINGing it.

Then look at the first and last character in each of the receiving
fields for the quote character.  (If you have any fields for which the
data may start or end with a quote, then you would need to also
examine the first three and last three characters for three
consecutive quotes.)  If one field starts with a quote, and the next
ends with one, then STRING the two together as one alpha item.

Note, also, that numbers that are displayed as comma-formatted in
Excel will be saved into the CSV file with the comma, the decimal
point, and surrounded by quotes.

Another strange occurrence is that text fields, if all numeric, will
be written to the CSV file as numeric fields (without quotes).

In a sample Excel file, i entered several values into one row, and
saved them into a CSV file.  These are the results:

Field   Format            Value    Displays As   CSV Result
-----   ---------------   ------   -----------   ----------
  A     General           1        1             1
  B     Numeric w/Comma   1234     1,234.00      "1,234.00"
  C     Text              ab,cd    ab,cd         "ab,cd"
  D     Text              abcd     abcd          "abcd"
  E     Text              1234     1234          1234
  F     Text              "abcd"   "abcd"        """abcd"""

Resulting CSV file record:

1,"1,234.00","ab,cd",abcd,1234,"""abcd"""

I hope you might find some of this to be useful.  Good luck.

Try something like this

There is a table of buffers to receive the decoded data

78  decode-buffer-number value 100.
01  buffer-table.
      02  decode-data occurs decode-buffer-number.
            03  decode-len  comp pic s9(9).
            03  decode-buf   pic x(200).

The input record is read into decode-record.

      * do delimited file conversion

           move 1 to i-ctr.

           perform varying idx from 1 by 1
               until idx > decode-buffer-number

               move space to wrk-buf1
               move space to decode-buf(idx)
               move zero to decode-len(idx)

               move 1 to ptr1
               move zero to ctr1
               if dosfile-record(i-ctr:1) = '"'
                   add 1 to i-ctr
                   unstring dosfile-record delimited by '",'
                       into wrk-buf1
                       count ctr1
                       pointer i-ctr
               else
                   unstring dosfile-record delimited by ","
                       into wrk-buf1
                       count ctr1
                       pointer i-ctr
               end-if
               move wrk-buf1(1:ctr1) to decode-buf(idx)
               move ctr1 to decode-len(idx)
           end-perform.

Here's the way we do it:
1. Check the record byte-by-byte looking for both double-quotes and
commas. Convert the commas enclosed within quotes to something else
you can find later (LOW-VALUES in the example below). (Pay attention
to FLIP - it's a nifty way to turn a switch on and off)

MOVE 1 TO FLIP.
PERFORM VARYING INDX FROM 1 BY 1 UNTIL INDX > RECORD-LENGTH
     IF MYREC(INDX:1) = ' " '
          COMPUTE FLIP = 3 - FLIP
     END-IF
     IF MYREC(INDX:1) = ','
          AND FLIP = 2
          MOVE LOW-VALUES TO MYREC(INDX:1)
     END-IF
     END-PERFORM

2. Having gotten rid of the interior commas, you can now unstring the
whole shebang into separate fields:

MOVE SPACE TO MYFIELDS.
MOVE 0 TO FIELD-INDX.
MOVE 1 TO MYREC-POINTER.
PERFORM UNTIL MYREC-POINTER > RECORD-LENGTH
     ADD 1 TO FIELD-INDX
     UNSTRING MYREC DELIMITED BY ',' INTO MYFIELD(FIELD-INDX)
          WITH POINTER MYREC-POINTER
     END-PERFORM.

3. Now reinsert the interior commas:
INSPECT MYFIELDS REPLACING ALL LOW-VALUES BY ','.

The trick is letting the double-quotes clue you as to the presence of
non-delimiting commas.

Some minor cleanup (like removing the double-quotes, justifying
fields, converting to numeric, etc.), and
you're done.

Comma delimited files are written wrong by more programs than not
Excel works and most others do not.

The standard is that the fields are either numeric (no quotes) or
string (quotes).  If you have quotes on a field then you read to the
next quote,  if this quote is doubled, quote quote, then it is a
single quote in data.  There is commas between fields.

I cannot think of how to do this except for scanning one character at
a time.

If you are parsing non-excel data then you must consider going
backwards and guess whether this comma is inside or outside a field. 
For some reason this works better but I cannot remember why.

If you do write a routine please consider giving it to the cobol
public domain.
