COBOL is real work.  For every programmer coding twirling icons under Java,
there's a COBOL programmer working on your paycheck or your car loan or your
tax return.

Pascal last week, C a few days ago, C++ yesterday, Java today, something new
tomorrow.  Why?  Because they build applications that don't matter as much.


Games that last a month, defrag programs that last until the next hard drive
model comes out, web pages that last ten minutes.

I built an accounting system in dBase II, upgraded to dBase III, ran it
under dbCompiler till they tanked, ported to Clipper.  They were bought by
Computer Associates and reduced to rubble, so I ported to FoxPro.  DOS was
dying, so I ported to FoxPro for Windows.  They were bought by Microsoft, so
I ported to Visual FoxPro to keep up.  I've upgraded to version 3, 5 and 6.
Now, with the advent of Visual Studio, I hear rumors that VFP's days may be
numbered.  Pisses me off.

The largest banking software package was written in COBOL in 1977 to run on
370s with ISAM.  It has a common file handler.  By changing the handler, the
system now runs on the fastest machines under VSAM, IMS and DB2, unheard of
when the system was written.  But, by changing 25 of 11,000 modules, it's
handling the new environments.  Many of the modules (interest calculation,
posting, etc.) haven't been changed in twenty years, because their functions
haven't changed.  Upgrades are made to handle new banking features or IRS
compliance, not to dazzle or impress.  It has green screens, but if you want
a nice windows front end, you can buy a screen scraper and build your own
with three kids with PCs and Java - just don't let them near the money.

Will that hold true of the latest 3D game?  Probably not.

You can write games in COBOL, you must remember to make all number 
variables BINARY to speed the computations.

Generally, games are not written in COBOL, because COBOL requires much 
more code than C or C++. This often gets in the way of the game builders 
creative processes.

Much easier to write:
     a := c;
than
      MOVE A TO C.

Other than this minor problem, I don't think there is anything inherent 
in the COBOL language that prevents games being written in COBOL.

Oh yes!
COBOL generally requires a runtime environment. This is usually specific 
to the COBOL compiler and is NEVER provided by the operating system as 
is the case for C. C's runtime environment is generally linked into the 
ending module. So there is usually no need to ship a runtime environment 
with the application.

IBM have got around this restriction by providing a language runtime 
environment for all High Level Languages on the Mainframe. This comes 
with the OS and all languages use it. COBOL, FORTRAN, PL1, C, C++.
