Just on a single report:
1. Add a column of data not already present:
   a. When the requested additional column won't fit on the page
(possible solutions: 2 lines per detail, change orientation of the
paper, 2 different reports, abbreviate contents of one of the fields,
maybe others).
   b. When the requested data are not available from the file being
read (such as number of transactions that made up the total),
necessitating access to a second file.
2. The existing report has the title on the upper right, page number
on the upper left. Switch these locations on even numbered pages so
that the report can be printed on both sides of the paper.
3. The existing report's sub-total line sometimes appears at the top
of a page. Fix the report so that all the details appear on the same
page with their sub-total. Bonus points for handling the case where
there are more than one page of details per sub-total.
4. Report has details and subtotals. New report has subtotals only.
Vice-versa. Optionally either.

One we've had to deal with lately.

A field has grown (from 5 bytes to 6). The record layout cannot be
changed, but 4 slack bytes exist in the record. Your mission is to use
one of the four spare bytes to augment the existing 5. Modify an
update program to access this new field (which now lives in two
non-contiguous locations). Extra credit if the field is the record's
key. Bonus points if the solution is generally invisible to the next
programmer.
```

Robaire wrote:
> 
> I'm glad that you clarified that - for a moment, I thought that COBOL
…[3 more quoted lines elided]…
> all prevalent in the real world?

Pointers can be used in CoBOL.  They are primarily used to reference
things outside of the program (they can be used to find a DD name in
JCL, they can be used by subprograms written in C).  I believe PC
programmers use them more frequently due to memory constraints.  Someone
here will chime in who uses them more frequently than I do.

Extracting substrings can be done using reference modification.  This is
done in two places - where the original variable is not available for 05
levels:

display              ' Started ' FUNCTION CURRENT-DATE (5:2)
                     '/' FUNCTION CURRENT-DATE (7:2)
                     '/' FUNCTION CURRENT-DATE (1:4)
                     ' ' FUNCTION CURRENT-DATE (09:2)
                     ':' FUNCTION CURRENT-DATE (11:2)
                     ':' FUNCTION CURRENT-DATE (13:2).

The other time is when we are walking through a string:

PERFORM VARYING MY-INDEX FROM 1 BY 1
          UNTIL MY-INDEX > STRING-LENGTH
             OR MY-STRING (MY-INDEX: MY-INDEX + 1) = '**'
END-PERFORM.
```

process electronic claims. The record format for these claims (NSF) changes
every couple of years, but generally by the addition or redefinition of a
small number of fields. We always end up with a bunch of conversion-type
stuff to do, and I find it makes future maintenance a lot easier when I use:

  MOVE field-1 of record-v3 to    field-1 of record-v4
  MOVE field-2 of record-v3 to    field-2 of record-v4
  STRING SYSTEM-CC DELIMITED BY SIZE
               date OF record-V3  DELIMITED BY SIZE
     INTO
               date of record-v4
  MOVE field-4 of record-v3  to field-4 of record-v4
  ...

weight in gold to the next poor schmuck (also, often moi) has to put in the
next round of changes.

>I would NOT, however, say that I would want to keep ALL variations of code

Usually, I find that keeping one (the most recent) group of changes
still in the production source is a good compromise.  I have
occasionally needed to re-insert some code that was removed several
generations back, and for that I can always copy-and-paste from a
previous source file, and insert a comment like "The following code
for validating xxx was restored from Version 3.26" or some such.  This
way, the revision history still makes sense, reinventing the wheel is
kept to a minimum, and the production source is kept relatively clean.

Obviously, the scales can tilt if you "know" that some change you are
making will later be reversed (several times?); or if a routine was
particularly complex to write and debug, in the first place.

Again, company coding standards may also drive this decision.
