Part of my problem is that I have moved from a totally unstructured 
if-it-feels-good-do-it environment to a highly structured environment, with
walkthroughs, turnover approval committees, and so forth.  
Assuming a pure COBOL environment, if I change a program, I should
then test the WFL, or JCL, that is to going to execute the changed program,
along with some other programs that I have not changed.  This is 
called, if I understand the terminology, "system testing."  
The "unit testing" is testing the stand-alone version of the COBOL program. 
When you say "module," I guess you mean some reasonably well-isolated
portion of the code.  
For example, I recently fixed a bug in a program that reads input in
16 possible formats and routes the output to one of 18 possible output
files.  In testing, I checked only the input format and the output file that
were involved.  Then I ran the jobstream, just to see that the jobstream
results were the same as the stand-alone results.  Some things I did
not do:  
1.  I did not test the other 15 formats (I don't know what they do). 
2.  I did not verify that the other programs in the job behaved normally 
    (I don't know how they're supposed to behave).
So your judgement would be:  
A.  I performed a Unit Test only 
B.  I performed a Unit Test and a System Test, but not a Regression Test
C.  Something else

Regression testing is saying : So you made a change and you want to test it.
But remember, it’s not sufficient to prove that your change does what you
intended it to do. You also have to prove that you didn’t undo or muck up
what was working before you made your change.

A regression test-case set will contain all the test cases  that proved the
functionality of the original code, plus all the test cases that prove the
functionality of the mods and  fixes applied to date.

That was my first impression, but I am backing away from that
view at present.  
My current impression is that in that the world of class-and-method
programming, the test of a class constitutes a Unit test.  One might
not feel a strong need to retest the parent class or the sibling classes
of said parent.  Regression testing would mean, I guess, that you
go ahead and test the parent class or classes and the relevant 
inheriting classes anyway, even though you don't see how they
could be affected.  
A system would comprise several "main" type classes, and the
the test of all the classes making up the system would comprise
a System test. 

In the typical COBOL program, a sort of undisciplined regression 
testing comes along with the unit testing, whether you want it 
or not.  Not too many people that I know of would attempt to test 
a paragraph without running the whole program.  
If the COBOL program were constructed unusually as a collection
of called subprograms, then the concept of Regresssion testing
would make sense.  Testing the subprogram would be the Unit
test, while testing the calling program and its other subprograms
would be the Regression test. 

As another poster pointed out, the idea of regression testing
is meaningless unless you have a predetermined set of tests 
with a predetermined expected outcome that can be run and 
verified even if you don't understand what all the other little 
doohickeys are doing.

Well disagree vis a vis OO. Consider Donald's emphasis that OO design is top
down. You start at the top. Not always easy because like it or not you have
to think lower levels to get what you want to achieve. But having a broad
spectrum of your needs you can then return to the top class for design,
because you have worked out the methods it requires to be invoked. Lets
assume the following model :-

Class A inherits from Base ( Base = Super
Class B inherits from Class A  (Class A = Super for Class B)
Class C etc....

You could go on and on, but it gets tricky if there are too many 'levels',
but that's not suggest it can't be done - just that we 'humans' tend to have
a hard time with a deep hierarchy. (But without checking it,  I guess Net
Express for Collections or GUIs probably drops down to Class F).

We've now got a Business Class 'Z' that needs access to the above structure.

Class A is written and compiled.

You know what you have included in Class A so Class B is written to invoke
the super's (Class A) methods as appropriate. Alternatively Class B has its
own methods which don't require messages to be passed up to Class A. Again
compile.

Class Z is written with the invokes against the above hierarchy - there are
three possibilities (1) invoke MethodB1 in Class B (2) Invoke MethodB2 in
Class B which in turn invokes super MethodA2 for additional 'support', or
(3) Invoke MethodA3 - ClassB is 'bypassed' because it doesn't contain the
invoked method.

It's not there in OO COBOL yet, (I think Thane said there is a variation in
current Fujitsu ?) - but the REPOSITORY SECTION allows each class being
compiled to "forward check" any immediate classes it will invoke - so Class
B will forward check against Class A and Class Z will forward check against
Class B. That's a major boon, because you ascertain whether or not named
methods are missing, including typos on the method names.

You are still stuck with passing/returning  incorrect/incomplete parameters
via Linkage, plus the bugaboo of specifying the wrong object, (back to the
'Pure OO' - you ONLY need to pass object references when relevant -
otherwise pass strings of pic 9's/x's).

Does the above work ? Having compiled Classes A, B and Z - you most
definitely want to test Z. But Z may only send messages to a portion of the
methods available. Testing Z may prove say 25% of the methods in A and B,
although ideally you would hope for a better percentage. But given the 25%,
you know that by using Z, 25% of A and B definitely works. One solution, if
practical,  might be to write a dummy Business Class X which invokes ALL the
methods in A and B

The Class X approach is a bit tedious and perhaps you can ensure complete
tests against A and B by actually doing the coding for Business Classes Z1,
Z2, Z3 etc., (at least the part of those classes that will invoke A or B),
covering all the remaining 75% of methods. That's not unreasonable, because
taking you top down approach you planned to include methods in A and B which
you already know will be required by Z, Z1, Z2, Z3 - even before you write a
line of code for the 'Zebras'.

From my own limited experience the above approach works. Major errors I find
occur in the Zebra classes - wrong parameters/wrong objects - and NOT in the
invoked A and B. Inevitably, (and here I'm thinking in the realm of GUIs),
what I want to invoke doesn't quite do it. (Don has found the same kind of
problems, which caused him to backtrack in his design).

No major overhaul is required, but there are two possibilities :-

(1) Perhaps you have to 'tweak' an invoked method with more parameters - in
effect you have created a new method.- and as a minimum - requires that you
recheck the Zebra programs for correct parameter usage. And this really
isn't to do with regression testing - it's more at the design stage when you
first start out - you've discovered a design flaw, and the likelihood is you
will discover it before you have written too many Zebras.

(2) Assume you have changed a value in MethodA1 - you have changed a
percentage discount calculation from 1.5% to 2.75% - the percentage being a
CONSTANT in the method. Only one test of any of the Zebra classes that
invokes that method need be checked. Once proven, it works for all Zebras.

Ross,
My understanding of Regression Testing comes from The Progeni Corporation's
3R's product, a pure regression testing tool.
.
It assumes that a working system exists before maintenance changes begin.
You run a test scenario, with a test database through and 3Rs captures all
the screen output, the report outputs and snapshots the database.  You save
this output.
Then after making changes, you run the same test scenario with the same test
database.
Again 3R's captures all the screens, reports and resulting database.
The package then compares the two copies of output and reports on
differneces.  The important feature of this package is it's ability to
discern and filter out "noise".  For example the report date on the top of a
report should not be reported as an error.  You can also "tell" the package
that what you changed was the length of an account number field.  It will
then filter out these inteneded changes and only report on any unintended
changes that occur.
