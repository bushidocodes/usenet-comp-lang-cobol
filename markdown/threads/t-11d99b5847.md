```
You should specify the file status as PIC XX not PIC 99.  In fact in a lot
of compilers, including AS400, fs2 does not have to be numeric.  In
microfucus cobol whene fs1 = 9 then fs2=b where b is a one byte binary
value.

In your case you have fs 9/018 which is described in the manual as 'Read
part record error: EOF before EOR or file open in wrong mode.' whatever that
may mean.  It sounds as though the file does not match the definition.

Regards
```

```
add
SET COBSW=-F
to autoexec.bat
stops illegal character in numeric field from being reported as error.

File status fields in Microfocus can be defined as follows
01 file-status   pic xx.
     88 file-success  value "00".
01 fs redefines file-status.
    03 fs-1   pic x.
         88 run-time-error value "9".
    03 fs-2   pic x.
    03 fs-2-bin redefines fs-2  pic x comp-x.

Then process as follows after any file operation
 if not file-success
   if run-time-error
       move fs-2-bin to w-pic-999
       display "Run time error " w-pic-999
   else
       display "File error: " file-status
   end-if
end-if.


Your file error would turn out to be Run-time error 018

<from NxtExpress V3 Help> - Probably, but not guaranteed to be, the same as
MF personal COBOL

"018 Read part record error: EOF before EOR or file open in wrong mode
Recoverable
A part record has been found at the end of a file. Consequently your
run-time system treats the data file as a record, and not finding a full
record, reports this error.
Resolution:
Ensure that the record size you give when you read from or WRITE to a file
is consistent."

You could try adding the clause "Organization line sequential" - otherwise
it may default to record sequential, which will read an exact number of
bytes per read, regardless of the position of newline characters.

Regards Tom
```
