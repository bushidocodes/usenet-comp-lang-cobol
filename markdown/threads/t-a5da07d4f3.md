In it's latest free form version, with "bulit-in functions", there is
almost no difference.  The data specs are just like working storage,
you can define constants, etc.  A bult-in function example: when you
are readng through a file, instead of using indicators out in some
specific column, to check for end of file, it is just;

Read Filename
If  %eof(filename)
  xxxxxxx
Endif

(Or you could use, not %eof).

For random reads, RPG uses 'Chian', but then you just check, "%found",
instead of "invalid key".

There are no more columns, no more indicators needed.  My work is
about 50% COBOL, 50% RPG and it's a breeze, although sometimes I type
"MOVE *BLANKS" in my COBOL program (instead of "SPACES") which gives
me a compile error.

Well, from an old RPG programmer that had to learn COBOL the
hard way (one week to read the book and have a working fix
for a cobol based payroll program or find a new job...), I
think I can answer this.

RPG was designed around the idea that one does I/O then
computes then does I/O. So, the general programming cycle
was as follows:

A) Read an INPUT file
B) Set control break level indicators on
C) If control break level indicators on, do control time
CALCULATIONS
D) IF control break level indicators on, do control time
OUTPUT
E) Interrogate input record to set general indicators
F) check to see if matching record indicators and other
special indicators need to be set
G) Move data from input record to fields
H) Detail time calculations
I) Detail/Exception OUTPUT
J) Go to A  

COBOL attempts to approximate this when one uses the REPORT
WRITER feature. However, it takes more code and time to set
up all of this in COBOL initially.

For RPG & RPGII, you have no control over OPEN and CLOSE of
your files. I believe this changed with the advent of
RPGIII.

currently on assignment at a client site and am only home on
weekends...
