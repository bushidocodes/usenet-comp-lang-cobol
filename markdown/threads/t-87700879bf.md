```
I hope I am able to express this clearly. Bearing in mind what I write
below, can somebody recommend a site or book that covers this topic,
preferably using pseudo code.

I'm using windowing and achieving what I'm setting out to do, but I
don't find it easy - my fault or Mr. Microsoft's ? Charles Petzold in
Programing Win 95/98 asks the question "Who's in charge ?", then answers
it with "Windows is".

The prevailing school of thought appears to be, keep your windowing
separate from your application logic, i.e., CustomerEdit program and
CustomerDialog program, (using the latter as a 'slave'), passing values
between the two and let CustomerEdit do all validity checking etc.
Behind this concept is, if you change your environment requiring some
new Dialog you have less to change in your Business program
(CustomerEdit).

This approach is fine when displaying a dialog, such as signing up for a
service on the Net; the dialog is in control until the user presses the
default <Enter> key and you return to your edit program which checks for
missing fields and responds accordingly.

This simplistic approach just doesn't work when designing a data-entry
intensive screen, where you require user to follow a specific flow,
(just as if you were designing using Screen Section). Further, follwing
your flow, certain droplists or radio buttons may need to be disabled,
depending upon previous entries user has made. With Windows the user can
set focus at random, and it takes the devil's own to write code to get
him/her back on course, how you really want the data entered.

I am aware that the various COBOL Windows toolkits offer various levels
of validity checking, and focusing to next field, but in the end you
have to get back to the business logic program. My concern with toolkits
is (a) they are platform specific and (b) they are, for the most part,
very compiler specific - we lose the universality of COBOL.
 
If you follow the two-program approach then in the case of OO you can
invoke CustomerEdit to return values. For structured, presumably you
could use entry-points to return values. Or are there those who say, to
hell with it, I'm going to keep my edit and dialog code in one program
for ease.

I really would like to know what approach others have taken.

Can I pose a provocative suggestion - perhaps Windows shouldn't be in
charge, with COBOL only accepting close events ? ( Bill go take a hike,
COBOL is in charge until I'm finished or you signal the user wants to
cancel/exit).

Jimmy, Calgary AB
```

Without wishing to get into the pros and cons of OO, an automatic garbage 
collector *is* specified in the new COBOL Standard.

In Java, it's vital, because objects are never explicitly discarded. This 
all means that the programmer rarely has to worry about memory usage, but 
can leave it to the language's run-time. It not only prevents memory 
leaks, but stops the (incorrect) freeing of memory multiple times.

Ideally, this will also be true for COBOL objects and object memory when 
the Standard is finalized in this area.

Just a thought (semi-new thread) about some of the recent posts.

Micro Focus (and Fujitsu and IBM - that I know of) have gone along a "path"
that resembles CLOSELY the draft of the next COBOL Standard.  AcuCOBOL uses
"native" syntax that WAS available before the 1st draft work on OO was
introduced.  AcuCOBOL "windowing syntax" (for which - by the way Micro Focus
provides an "emulator"  - or at least used to) does seem "more user
friendly" in many ways.  This approach WAS considered by J4 (then X3J4) and
WG4 when they were first looking at additions to the next COBOL Standard.

As I recall (and this was QUITE A WHILE AGO), the reasoning that J4 took was
that OO COBOL *should* be available for all types of classes and
applications - NOT just for "user-interfaces" (aka GUIs).  Although it would
be possible to provide BOTH in the next Standard, it was felt (as I recall
it) that there really was NO need to add the ACCEPT/DISPLAY enhancements for
GUIs - at the same time that OO was being introduced - if the "native OO
syntax" could include interfaces to the operating-system-specific GUI calls.

In a way this was a little contradictory - because this draft Standard
*does* introduce the Screen Section and "enhanced Accept/Display" for
character based applications.  However, this was taken (mostly) from the
X/Open specification that WAS already available on many platforms and with
many compilers (unlike the AcuCOBOL windowing syntax which was basically
only available from one vendor).

The bottom-line (from my PERSONAL perspective) is to

1) Use the AcuCOBOL syntax if you think that your application will ALWAYS
live in an AcuCOBOL world (many operating systems - but only one vendor's
compiler).

2) If you think that your COBOL application will ever need to run on
environments or with applications that AcuCOBOL doesn't support, then go
with:

  A) an Sp2 approach - if you think that Sp2 is available now (or in the
foreseeable future) on all the platforms and with all the compilers that you
are interested in.

 B) Stick to "isolated" OO syntax if you need to develop an application that
is guaranteed to run on *ANY* ANSI/ISO  compiler (when the next Standard
actually becomes official)
