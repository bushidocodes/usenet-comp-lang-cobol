Yes, I use EXTERNAL data. No, I have not had any maintenance
problems as a result of using EXTERNAL data.

I am in the process of updating a system to which I will be adding some
EXTERNAL working-storage data. The EXTERNAL data will consist of
values that do not change during the use of the application, i.e.,
configuration data and security information. These values are set at the
start-up of the application and remain the same through its termination.

I am also removing some EXTERNAL data and placing it into
programs as constants. In eight years, the data has never changed.
It was thought that it might vary among customers.

I have primarily used EXTERNAL data to hold program names. This
has allowed me to customize the system with far less concern for the
distribution of the application. Every customer receives the same
distribution. Customizations are sent separately. At start-up, the
application searches for a custom library. If a library is found, it calls
a program within the library to change the program names in the
EXTERNAL data. During execution, the core programs will call the
custom programs, rather than the standard programs. Again, these
values do not change during the execution of the application.

Since I have not experienced any problems, I am not certain when
the problems would out weigh the advantages. However, it seems
that using EXTERNAL data as configuration constants is probably
the least controversial use.

Let's clear up what is in the draft (and some of what is still being worked
on for the draft)

A) The word "function" will still be supported before BOTH intrinsic
functions and user-defined functions (hence no confusion with data names)

B) The word "function" will be optional before intrinsic functions IF AND
ONLY IF
  - there is a declaration in the REPOSITORY paragraph requesting it
  - the same COBOL word is NOT used as a user-defined word any where in the
compilation group

C) The word "function" will be optional before a user-defined option IF ANY
ONLY IF
  - the same COBOL word is NOT used as a user-defined word any where in the
compilation group
    and
 - either there is a function proto-type or a repository paragraph entry for
it (I am saying this based on my memory - there may actually be slightly
different rules - but they boil down to "declaring" which user-defined
functions you plan on using in your program)

D)  The original statement about "J4 eliminates the parentheses after
parameter-less functions" is quite confusing.  In fact J4 is *adding* the
OPTION of including parenthesis for parameter-less functions.  Today, it
would be illegal to say
    Move Function Current-Date () to ABC
In the next Standard this will be allowed (along with the upwardly
compatible syntax without any parenthesis)

The original example had a problem in that EITHER way it used either an
arithmetic expression (e.g. FieldA * FieldB) or a FUNCTION as a parameter to
the ADD statement - which is a "No-No" in all Standard variations of COBOL.
A better example of the problem is

Compute Out-Field = Function Max (Function Random (FieldA * FieldB))

The question is whether there are one or two parameters passed to the MAX
function.  The (existing) interpretation is that the parenthesis around
(FieldA * FieldB) *automatically* make it a parameter to the Random
function - and hence this is a single parameter passed to Max (which I think
can take an arithmetic expression as input - or otherwise my example is just
as bogus as the original one).

And of course it never really WAS ambiguous - it was just not what some
people wanted.  If you wanted the semantics of the new "empty parameter
list" feature, you can (already) in '85 Standard say

Compute Out-Field = Function Max ((Function Random) (FieldA * FieldB))

and force Function Max to get 2 parameters.  But in the next Standard, you
will also be able to say (as a synonym for this 2nd example and not the
first one)

Compute Out-Field = Function Max (Function Random() (FieldA * FieldB))

Which saves you EXACTLY *zero* keystrokes - but which seems more natural to
some  (I think primarily those who have worked with other languages)
