```
In message <<19980925221123.29328.00001652@ng124.aol.com>> jcj0347@aol.com  writes:
> +
> +1/1/99
> Why should this be any more special than 1/1/98 ?

Because a system may have had built into it that year '99'
(or indeed any specific year) is to be used for 'unknown'
dates and dealt with differently from other numbers.

There is a widely used accounting system here (not mine,
not Cobol) that used year 99 in this way.  On April 1
this year the new 'year end' date was input as end March
1999 and various problems appeared in the system.  For
example forward-dated invoices came back out of the 
system immediately.

It was easy enough to change year-end to 31/12/98
until a patch arrived, but date related problems are
not restricted just to 2000.

> +9/9/99
> Why would there be a problem with this? 9 Sept 1999 would *not* be
> stored as 9999, as many people seem to think, but as 990909. I see
> no reason to believe that anything will break on that day.

Because 9/9/99 may be treated in the system as an 'unknown'
date.  The program may require a valid date be entered but
in cases where the date is simply not known a special
date (or year) is used as a 'flag' to indicate that it
is, for example, not to be processed.

These only apply to systems where this trick was used. The
problem is knowing whether the system in use at _your_
site uses this trick, or did use it years ago in the
source code that you cannot find.
```
