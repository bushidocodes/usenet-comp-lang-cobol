In pseudoconversational design the COBOL working storage is only availbale to the current
task (i.e. dialogue step).  After the RETURN TRANSID command the working storage is
released by CICS.  The next transaction0task to run will get a new (initial) working
storage.

To pass data from one task to the next you can
	- use a COMMAREA (up to 32760 bytes) (usually recommended method)
	- use temporary storage (up to 32767 items [don't!!] of up to 32700 bytes ) 
	- use shared getmain (not recommendsd)

Yes, that's exactly what you would need to do - but keep in mind: Everyone that uses
conversational design will get a fair trial - and then will be taken out and shot!  Only
IBM can get away with it.  Using conversational design can effectivly lock out a large
part of the users from CICS.  

>-I have read somewhere that one can only use a CICS RETURN TRANSID option at
>one level below CICS, is that true ? For instance if I had a fairly large
>application thread with a few EXEC CICS LINK type commands, could I return
>to CICS and end of the preceding thread ???

This used to be true in earliere releases of CICS.  With CICS Version 4 though you can
issue the RETURN TRANSID option at any time.  BUT....this command would NOT return you to
CICS!!!  It would only return you to the next higher program in the link hierarchy.  The
TRANSID would just be remembered for the highest LINK level.  It will only take effect
when returning to CICS.  NOTE:  At any time before RETURNing to CICS you can check the
TRANSID via EXEC CICS ASSIGN NEXTTRANSID.
