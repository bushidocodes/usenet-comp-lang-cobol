There isn't "a problem with" COBOL or .DAT files.  The problem is
that you are trying to look at .DAT files as if they were created by some
sort of database program, such as Microsoft Access or Oracle.  COBOL
has a built-in indexed (keyed) file system, called 'Indexed Files'.  You
can think of it as something like a b-tree library used in a C program.
C has no such built in capability, so you should not think of COBOL as
"having a problem" because it offers capability not found in C.  If one
doesn't like it, one does not have to use it. ;-)

Specifically, .DAT and .IDX files are created by COBOL programs
compiled using Micro Focus COBOL of one version or another, and I
strongly suspect your files were created in this way.  From what you
wrote above, it appears that you have the format of the .DAT file
records.  The .IDX file contains the index data.

Here are a few possible solutions to your conversion problem:

1. If you have access to a Micro Focus compiler compatible with the
file, create a small COBOL program to read the file and write out
the data in 'Line Sequential' format, converting any COMP fields to
DISPLAY (printable ASCII), then compile and run the program.
If there are any signed or non-integer numeric fields in the file, you
should use 'numeric edited' pictures in the output, to make the data
easier to parse.  You can add comma delimiters, and enclose alpha
data in quotes too, if you like.  This is your 'best' option, in that it
is guaranteed to work, you can make your output data look any way
you like, and avoid having to convert non-integer BCD data into
floating point in a C program.  Decimal arithmetic is another facility
COBOL has that C doesn't, which is one reason why there is about
100 times more business software written in COBOL than in C
(10 million copies of MS Access is still only one C/C++ program).

2. If your file has *no* binary data (i.e. no COMP fields), you can use
the Micro Focus utility program REBUILD to write the file out into
'Line Sequential' (normal text) format.  REBUILD is considered part
of the Micro Focus COBOL runtime, and can be distributed with the
application software created using the compiler.  If you can legally
run the application, you can legally have and run REBUILD.  Check
with Micro Focus if you can't find it.

3. If your file contains any binary data, you can't use Line Sequential
format, but you can use REBUILD to convert the data to  'Sequential'
format.  Each record of a Sequential format file has a record length
field preceding each record.  Either get the record length format
from the Micro Focus documentation, or examine the file enough to
figure it out.  Then write a C program to extract the data into any
format you like.

It seems to me that you ought to be able to just read it directly.  

If the COBOL layout says
FD something...first file
01 RECORD-1-EMPLOYEE.
   05 FIELD1-EMP-NAME.
      10 FIELD1-A-NAME-FIRST                PIC X(10).
      10 FIELD1-B-NAME-LAST                 PIC X(10).
   05 FIELD2-EMP-NUMBER.
      10 FIELD2-A-EMP-NUMBER-FIRST   PIC 9(4).
      10 FIELD2-B-EMP-NUMBER-LAST    PIC 9(6).

FD something else...next file

then you may rest assured that Record 1 will always have
30 bytes.  That is to say, there are no INT's or VARCHAR's 
or stuff like that.  There are no field delimiters.  If your data has: 
JOE       BLOW      0000123456          SMITH     7986000000SAMU...etc.
Then 
       Firstname Lastname ID-prefix  ID-suffix
1st guy Joe Blow <no prefix> 123456
2nd guy <first name unknown> Smith 7986 <no suffix> 
3rd guy Samu...etc.

You just pull the first 10 characters into your First_Name variable, then
the next 10 characters into your Last_Name variable, etc. 

When I say "always" I am, of course, lying.  Some formats will have
one or two characters at the end of each record just to make
it easy for you.  I believe the Microfocus .DAT format will have
two characters marking the end of each record.  So the record
would actually be 32 characters long instead of the 30 indicated
by the file layout.  

Some formats will have spaces counted and summed, instead of
having every single space put into the data.  Some formats will
have numbers in a different order that what you might expect, and
you have to rearrange the digits. 
I don't think .DAT files do either of those.  If you look at the data,
you should be able to tell if anything unusual is going on. 

I don't think you need to look at the .IDX files.  Presumably, you
are going to move this data to a database.  Near the beginning of
the COBOL program, in the SELECT statement, it should tell you 
what the (Primary) Key is.  Most likely, you will want to keep this 
for the Primary Key in your database.  So, if the COBOL program
says KEY IS FIELD1-EMP-NAME, then you would designate
your Primary Key as First_Name + Last_Name. 

It doesn't seem all that hard to me, so I am probably missing
something obvious.

You could probably read the .DAT file with a C program, or in any
other language.  Simply add up the total lenght of the record, say
100.  Then write a program in whatever language you use to read 100
characters and display it, then read another 100 and display, etc.  If
you can see a pattern, maybe offset on each record, then simply change
the 100 to some other number.  By trial and error you may be able to
set the read so that the same data is always in the same position on
each record read.

Compressed numeric data may be a problem - but if you know the answer
you may be able to determine the conversion.  Or initially miss the
numerics out and try to extract just the text.

Compressed files are more of a problem, as the resultant records can
be variable length.  But even this is doable, if each record ends in a
recognizable character - simply read one byte at a time until you
recognize the character, process that record, then continue reading
...

It is doable.  I converted files from a Unix system, written in basic,
to text files - with no help from the originator of the data.  These
had variable length records, compression, and all sorts of problems.

Ask, or send a sample, if you need any help.

All the best.
Phil Hickling.
