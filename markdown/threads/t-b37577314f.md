Hello:
This is the third in a series of postings about the COBOL language.
In this posting, I will discuss various aspects of using COBOL
to process a sequential file. In this case, I will write
a program to read a line sequential file under MS-DOS and
generate a phone book.
The program source code is named PHONESEQ.CBL. I used the
following text input file:
---- cut here ------
CLINTON BILL 8189760666
DOVER BENJAMIN 4083337954
EDWARDS GERBILATE 5102227954
MASON CHRIS 4087427954
---- cut here ------
And I got the following report output in file report.txt:
---- cut here ------
PHONE NUMBER REPORT DATE 03/10/95 PAGE 1
FIRST NAME LAST NAME PHONE NUMBER


BILL CLINTON (818) 976-0666

BENJAMIN DOVER (408) 333-7954

GERBILATE EDWARDS (510) 222-7954

CHRIS MASON (408) 742-7954
---- cut here ------
Basically, the program reads the line sequential file
(Native MS-DOS text file delimited by carriage return/linefeed)
and outputs the report.
As we look thru the code:
The apropos parts begin with the FILE-CONTROL paragraph in the
ENVIRONMENT division. The select/assign statement tells the
system what the name of the file is and it's type. ASSIGN
is followed by the file name in quotes. ORGANIZATION tells
the type of the file, in this case line sequential.
The FILE STATUS is returned to our program in the field
WS-FILE-STATUS. We should query this field after each READ/WRITe
operation to check if they worked.
In the DATA DIVISION, we define the record layout of our files.
In this case the FD (File Definition) of INPUT-SEQ-FILE shows
that the record length is 80 characters. When we read the input
record, our system will pad the record with spaces.
Skipping on to the Procedure division, we see what I consider
to be a classic COBOL report program structure (although simplistic).
Our A0000-MAIN paragraph is our main module. We do
Beginning housekeeping (open files, accept date/time, display
startup message), Perform our processing loop, And then
perform the wrap up(close files, display run stats), finally
we STOP RUN (end program).
The housekeeping paragraph opens our files and does the
primeing read (first read on input file). Notice that
our loop will be terminated either on end-of-file condition,
or another file error. Be warned that READ AT END is only
executed at end of file. Other errors will not cause
the AT END clause to be executed. Thus we need to check
the file status to guard against such problems as disk full
or bad file.
Notice the processing loop performs the report process and then
attempts to read the next input record. The Perform loop in
A0000-MAIN will detect the result.
The Report process paragraph has a trick in it to cause page
eject on the first entry into it. The value clause of the
line count is primed with a value 1 greater than the max value.
This way the if statement will be true on the initial entry
into the paragraph.
Inside the headers paragraph, the line count is reset. The
line count is incremented on each write. In order to get
to top of form, the Write after advancing page option is used.
On some systems, you may have to embed a printer escape
sequence on the report record. Caveat Coder!
In one area I worked, I picked up the concept of using
names instead of "magic numbers" embedded in the code.
LI-LINES-PER-PAGE is more understandable than say, 55.
I used the brute force method to double space some of the
output. (writeing out a spaces detail record). Once again
there are ways of using printer dependant commands. For
example, In the old days, you would have carriage control
in Column 1. The value 1 in your record would cause a page
eject. Value 0 would cause double space. + would cause no
eject. Go figure...
The Move report fields paragraph may seem overly structure, but
it alway helps to modularize your code.
Some of the features in this program are COBOL85, and aren't
found in COBOL74. One feature I have grown to love is the
new READ statement extenstions. Now you can use the NOT AT END
phrase to increment your record counter. Before, you would have
to code an ugly File status if check before incrementing or
moveing the record data to make sure you got a record.
