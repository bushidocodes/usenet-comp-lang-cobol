Roland,

I would strongly discourage you from specifying a COBOL program or entry
point in the /ENTRY statement when creating a COBOL DLL. The entry, as has
been mentioned in a separate note in this thread, is used for DLL
initialisation and terminationn and the MF product actually contains code to
do that. If you create a DLL using CBLLINK you should notice (if you use the
/V option) that it specifies /ENTRY:LibMain@12. The code that gets used will
ensure that a DLL initialises and terminates cleanly and that any programs
in the DLL are properly CANCELed, including freeing memory etc. By
specifying your own entry you prevent that code from being used and so
CANCELs may not work and you may have memory problems resulting in 114
errors.

I know nothing about MF-Cobol, but I wanted
to clarify something. Linker /entry:roland tells
the linker that the entry point for the DLL,
i.e. the function to invoke when the DLL is
loaded with LoadLibrary, is called roland, and
has a signature equivalent to this C declaration:

long _stdcall DllMain (void  *hInst,
        long ul_reason_for_call, void * lpReserved)

Does MF-Cobol generate such an entry-point
function for you with the name the same as the
program-id?

If not, run DUMPBIN /exports (a VC utility; you'll
see it in the docs) on the MF-Cobol DLL and see
if you can find a function that may be the entry-point.
