```
Update

BINARY is a reserved word that I believe was introduced back in VS Cobol
II - Rel. 3

Our compiler settings for the most part are set by an OEM source mgt.
product - they can be adjusted at the module level.

One of the compiler settings being used is CMPR2. The BINARY description
will drive the compile error mentioned above using CMPR2. In order to
compile with BINARY, one must use compiler option NOCMPR2.

To be straigthforward, I'm not a Cobol programmer. I'm a mainframe
system programmer(CICS and MQ as of late). My coding is done in
assembler. One of our Cobol developers went about the business of coding
some MQ programs. He added SCSQCOBC(MQ cobol copybooks) to his IGYCRCTL
SYSLIB concatenation. Using the "been around forever" compile option
CMPR2, he encountered the IGYDS1089-S "Binary" Invalid errmsg.

I believe we have a ton of COB II style code. If I read the manual
correctly, NOCMPR2 invokes COBOL 85 vs the COBOL 74 used in CMPR2.
Fortunately, this NOCMPR2 will be used in "newly" developed MQ code and
not 20 yr. old legacy code that NOCMPR2 might impact.

Thanks for the interest.
```

```
It is true that when you specify CMPR2, that BINARY is *not* a reserved
word.  See:

http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igya1101/APPENDIX
1.4.2.1

If your application programmers (and/or your OEM source mgt product) isn't
aware of it, the latest releases of IBM's COBOL compilers don't even support
the CMPR2 compiler option.  You should have SOME type of plan on migrating
to NOCMPR2 - if you expect to maintain these applications for the
foreseeable future. (especially if you want to use compilers supported by
IBM).
```
