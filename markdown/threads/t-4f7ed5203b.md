```
Hi all,

A few days ago, I posted a question about COMP-3 fields showing incorrect
results. (This is MicroFocus COBOL on AIX). I have finally narrowed the
problem down to its essentials. [The compiler directive NOTRUNC is
required]

In a MOVE statement, if
1) the source field is S9(4) COMP; and,
2) the target field has *exactly* 5 decimal places (4 decimals and 6
decimals don't cause the bug); and,
3) the value in the source field is greater than 21474
then, the value is not moved correctly.

A sample program follows, if you'd like to try it on your MicroFocus COBOL
compiler.

I guess that is is more than pure coincidence that the max s9(9) value is
2147483647, which is approximately 21474 * (10^5)..... 5 decimals...??
Somthing happening during the conversion?

if anyone knows if there is a patch out to fix this bug, I'd appreciate
knowing about it.

Thanks,

- Darius Cooper

====================================================
IDENTIFICATION DIVISION.
PROGRAM-ID. TRY.

DATA DIVISION.
WORKING-STORAGE SECTION.

01 WS-DEC PIC S9(11)V9(5) VALUE ZERO.
01 WS-SMALL PIC S9(4) COMP.

PROCEDURE DIVISION.

MOVE +21474 TO WS-SMALL
MOVE WS-SMALL TO WS-DEC
DISPLAY 'WS-DEC IS (Displays Correctly) : ' WS-DEC

MOVE +21475 TO WS-SMALL
MOVE WS-SMALL TO WS-DEC
DISPLAY 'WS-DEC IS (Display is WRONG) : ' WS-DEC

STOP RUN.

====================================================
```
