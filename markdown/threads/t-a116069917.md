Defining record structures, in COBOL, is probably the worst use
for typedefs; though, undoubtedly, there are cases where it is
sensible to do so.

Using typedef to define the usage and picture for elementary
items would not conflict with replacing and would be, I think,
its best use.

For example, the same 'internal-date' definition may be used
in many places, including multiple 'copybooks'. The same is
true of typedefs for 'external-date' representations.

Defining an 'internal-date' type as packed-decimal pic 9(7)
and using that typedef for each elementary item of type
'internal-date' means that each of, say 50, 'copybooks' will
have the same definition for each 'internal-date' item. Again,
typedefs may be defined for each 'external-date' in, say 200
or more places. Subprograms may be used to convert
between internal and external respresentations.

Futhermore, should it become necessary to port the code to
a different computer, it is a very staightforward task to convert
the type to a more efficient representation by changing only
the typedef and subprograms.

Another possible use is aliases for the underlying type. For
practical reasons, it may be necessary to create separate
sub systems for different parts of an enterprise. The programs
in these sub systems may need to use different names and
different record layouts; but some of the data will have the same
underlying type. For example, 'employee-id', 'contractor-id', and
'trainee-id' may be the same 'person-id' type, with the same
operations available; but each sub system needs to use the
alias for clarity.

Typdefs may assist in code re-use. Any code that uses some
of the same typedefs that will be used in a new program, might
have procedures that can be used in that new program. This is
because the other programs are operating on the same
underlying data type regardless of the name given to the
realization of that type or the description given to the program.

Overall, a welcomed addition to the language.
------------------
Rick Smith

If you have a copybook that defines a structure that must occur in several
places in your program all you need do is start the copybook structure with
level number greater than 1.  The you can put your own 1 level in your
program and include the copybook.  Use qualified data references and you are
all set!

As far as copybook pre-processors go, I would recommend avoiding them unless
you are adding significant macro functionality, and even then I would try to
avoid processing copybooks.  Many analyzing compilers and documentation
tools will then see the copybook as inline code, hindering their usefulness
for analysis and documentation.
