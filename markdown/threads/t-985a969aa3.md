I don't think that is necessarily true.

From my understanding --- When you operate on a record that has an
alternate key that allows duplicates, and that record duplicates
another's alternate key - the successful file status will be "02".

If you do a "Start" - then "read" sequentialy - you seem to indicate
that the first, and subsuquent reads will return an "02" - up to, but
not including the "last" record with that alternate key, which will
return what?  "00"?   That doesn't seem right to me, so let's test it.

Here's the program:

 IDENTIFICATION DIVISION.
 PROGRAM-ID.  TESTC.
*
*  Test Duplicate alternate key file status values.
*
 ENVIRONMENT DIVISION.
 CONFIGURATION SECTION.
 SOURCE-COMPUTER.    IBM-PC.
 OBJECT-COMPUTER.    IBM-PC.
 INPUT-OUTPUT SECTION.
 FILE-CONTROL.
     SELECT OPTIONAL TEST-FILE ASSIGN TO "Test.Fil"
            ORGANIZATION IS INDEXED
            ACCESS IS DYNAMIC
            RECORD KEY IS PRIME-KEY
            ALTERNATE RECORD KEY IS ALT-KEY WITH DUPLICATES
            FILE STATUS IS TEST-STATUS.
 DATA DIVISION.
 FILE SECTION.
 FD  TEST-FILE.
 01  TEST-RECORD.
     03  PRIME-KEY PIC 999.
     03  ALT-KEY   PIC XXX.
     03  FILLER    PIC X.
 WORKING-STORAGE SECTION.
 01  TEST-STATUS PIC XX.
 01  CNTR PIC 9(3) VALUE ZEROS.
 PROCEDURE DIVISION.
 START-OF-PROGRAM.
     OPEN I-O TEST-FILE
     PERFORM VARYING CNTR FROM 1 BY 1
             UNTIL CNTR > 9
           MOVE CNTR TO PRIME-KEY
           EVALUATE CNTR
              WHEN 1
                MOVE "AAA" TO ALT-KEY
              WHEN 2 THRU 4
                MOVE "BBB" TO ALT-KEY
              WHEN 5 THRU 8
                MOVE "CCC" TO ALT-KEY
              WHEN 9
                MOVE "DDD" TO ALT-KEY
          END-EVALUATE
          DISPLAY "WRITING " TEST-RECORD (1:6)
          WRITE TEST-RECORD 
          DISPLAY "STATUS " TEST-STATUS
     END-PERFORM
* realia needs a close/open here
     CLOSE TEST-FILE
     OPEN I-O TEST-FILE
     INITIALIZE TEST-RECORD
     MOVE "AAA" TO ALT-KEY
     START TEST-FILE KEY NOT < ALT-KEY
     DISPLAY "Start AAA " TEST-STATUS
     READ TEST-FILE NEXT RECORD
     DISPLAY "ReadNext AAA " TEST-STATUS
     MOVE "BBB" TO ALT-KEY
     START TEST-FILE KEY NOT < ALT-KEY
     DISPLAY "Start BBB " TEST-STATUS
     PERFORM 4 TIMES
       READ TEST-FILE NEXT
       DISPLAY "ReadNext BBB " TEST-STATUS " " ALT-KEY
     END-PERFORM
     CLOSE TEST-FILE
     STOP RUN
     .

Here are the results:

WRITING 001AAA
STATUS 00
WRITING 002BBB
STATUS 00
WRITING 003BBB
STATUS 00
WRITING 004BBB
STATUS 00
WRITING 005CCC
STATUS 00
WRITING 006CCC
STATUS 00
WRITING 007CCC
STATUS 00
WRITING 008CCC
STATUS 00
WRITING 009DDD
STATUS 00
Start AAA 00
ReadNext AAA 00
Start BBB 00
ReadNext BBB 02 BBB
ReadNext BBB 02 BBB
ReadNext BBB 00 BBB
ReadNext BBB 02 CCC

Well looky here.  Walter is right!  I learned something new!

If someone had bet me I would have sworn that the third "ReadNext BBB"
display would hav returned a status of "02" also.  VERY interesting.

Also - my earlier assertion that "02" would be returned for the write
is equally incorrect.

There is a link option with Realia if you want the duplicate alternate key
file stati returned at the time of a WRITE for an indexed file with
alternate keys opened OUTPUT:  Link with XBLDSLOW.  By default we build
the alternate index entries at CLOSE time - this is dramatically faster as
we sort them and write them sequentially rather than doing the random
inserts at the time of each WRITE.  But of course, with this model we
cannot return the duplicate indication on WRITE.  Note that that module is
appropriately named:  load times for large files with alternate keys will
be slower.
