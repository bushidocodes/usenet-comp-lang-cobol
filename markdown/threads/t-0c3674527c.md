You can call any C function directly from COBOL, so there is no
reason you can't use the getpid() call. Obviously, you need to make
sure your parameter and return types are correct when porting (as you
cannot just include a system header file as you could if you were
coding in C - perhaps writing a small C wrapper function would give
you a more portable solution).

However, given your stated intent (building an output filename which
"should not overwrite old file") you must be very careful. The
allocation of process ID's is dependant on the OS you are running on.
Some operating systems will re-use low process id's as they become
available - it will not wait for the PID to "wrap". [This was certainly
the case for some versions of AIX, AFAIR]. This will greatly
increase the possibility of your file being overwritten.

A better solution would be to either use date/time information, as
suggested by someone else, or to use the system tmpnam() or
tempnam() functions which provide exactly this functionality. Again,
a small C wrapper around this function might give you the best/most
portable solution for use from COBOL.

Cheers, Kev.

I'm not as keen to recommend mktemp() as it's (a) not ANSI and (b) is
more likely to "use up" the available namespace - at least from reading
the docs I have, the variable part of the name returned is composed of
a single letter and a PID. If running on an OS where the PID reuse
problem is an issue, it depends on what constitutes "a letter" as to
how many unique filenames can be created, assuming the COBOL process
is repeatedly allocated the same PID.

I agree that it's better than just the PGMID/PID combination, though.
