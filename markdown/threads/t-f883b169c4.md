```
Well, for a static call, you don't get the big performance hit on the first
call. And even on subsequent dynamic calls, there is a hit for the lookup of
where the loaded module is. On a static call, the linkage is a few instructions
resulting to a direct branch to the target. If a routine is called millions of
times in a run, this can add up significantly.

Dynamic calls are definitely the way to go in most cases. But for high
performance applications, static calls can still make a big difference.
```

```
On May 24, 2000, I posted the following on comp.lang.cobol under the
title "Dynamic Call using procedure pointer":

*** original ***

I recently had a conversation with a programmer in another department,
and he tells me they are using dynamic calls with procedure pointers. 
This is using COBOL for MVS & VM, under 0S/390 release 2.8.

I've never seen this before, but apparently they also use EXTERNAL
working-storage items.  All programs that use exactly the same 01
EXTERNAL record definitions have them available in their working-storage
section without naming them in the call statement or procedure division
using clause.  The EXTERNAL 01 records cannot have VALUE clauses in
them, and they apparently reside in working-storage for both the caller
and callee.

The call is set up something like this:

01  SHARED-DATA-RECORD   EXTERNAL.
    05  WHATEVER-ELSE     ...

01  PTR-SUBPROG         USAGE IS PROCEDURE-POINTER.

SET PTR-SUBPROG TO ENTRY "PROGNAME"
CALL PTR-SUBPROG

My colleague tells me they've done extensive performance testing, and
this form of dynamic subprogram call with procedure pointers and
EXTERNAL data items is significantly faster than a standard dynamic call
(CALL MY-SUBPROGRAM USING PARM-1, PARM-2, et cetera).

Has anyone else used this technique?  Any comments on it? 

*** end quote ***

I had further conversations with my colleague, and the way he explained
it to me is that once you have obtained addressability to the
dynamically called subprogram using a procedure pointer, every
subsequent call to the same routine has no more overhead than a PERFORM
statement in your mainline.  A Standard dynamic call must check each
time to see if the contents of the program name variable have changed. 
He verified this by testing, proving to his satisfaction that there was
a measurable performance benefit to using procedure pointers for dynamic
calls (at least in an OS/390 environment).  In a large batch program
that may call the same routine ten million times, the performance gain
could be significant.  You can have all the benefits of dynamic calls
*and* the high performance normally associated with static calls.
```

```
Having worked at places that did both static and dynamic calls (we're talking
OS/390 here folks) I can state that static are more efficient because the load
module is right there with the other program - linked together.  Of course it
can be a maintenance nightmare because any change to the called program
requires a recompile of the calling programs and one can loose track of who is
calling whoml.

Back when we were running 256K mainframes things like this made a huge
difference, as did paging and so on.

Today the dynamic call is the call d'jour and when we upgraded to COBOL II then
to COBOL MVS we changed most of our calls in this shop to dynamic for what we
were told ease of maintenance and 'surprise' efficency!  Something to do with
LE and all that.  Yes I guess the sysprogs can tweak stuff to increase
efficency as well.

We have some packages that have static links and all that for I/O to IDMS
databases (which is the only IDMS we have - everything else is DB2 and VSAM)
and they are the worst system hogs we have.  The code in some of the programs
is enough to give one night terrors.
```
