The problem you are having is almost definatley this, Microfocus's definitive UNIX
system appears to be SCO. On SCO the process id's are defined as a short
under AIX the same construct defines the process id as an int an average process
id on sco is in the low thousands, under AIX on a big box with high numbers of
users they are in the hundreds of thousands, anyway they are bigger than a short

The problem is that MF use the process id as the key to locking records and files
if the process id is to big then puff in a bag of smoke goes the program, any program
not just compliers but any MF program including ones that don't use files and don't
ask me why I don't know.

There is however a way out, the problem is a can't quite remember the exact answer
but this should point you in the right direction anyway enough of my whiterings

There is a configuration option in the base settings file which specifies ,the type
of file/record locking to be used with a value in the range 1-4 set it to old style
record locking, I may be worng with the value I haven't got the manuals handy
the option is probably called file_lock_type and is designed to get round the
problem of NFS support for locking of cobol files
