```
In COBOL/for this-and-that (as with any ANSI '85 Standard compiler), it is
finally possible to get the length of a "variable length" record via
officially supported means.

In your FD code the

  Record varying in size depending on WS-item

phrase (rather than the older RECORD CONTAINS clause).

Then, after every READ, your WS-item will contain the length of the record
that was just read.  (You need to "update" it before a WRITE)
```

Ok, let's clear up two related issues - that may have been muddled by now.
Assume you have an FD with

FD MyFile
     Record Varying in Size from 10 to 100
       Depending on WS-Num1.
01  MyRec.
    05    Fixed-Part   Pic X(08).
    05    FD-Num2      Pic 9(02)
    05    Tabl.
             10 Each-Elem Occurs 1 to 90
                    Depending on FD-Num2.
                15  Elem    Pic X.

Later in Working-Storage you have
  05    WS-Num1  Pic  9(03).

****

OK,
  A few things to think about:

1) The item in the RECORD VARYING clause *must* be Working-Storage.

2) The item in the OCCURS clause can be in the FD or in Working-Storage.
(Traditionally, it has been in the file itself - but I could imagine a
"strange" application where it was in WS.  Writing such records would be
pretty easy, reading them would be quite a challenge)

3) The item in the OCCURS clause should be "updated" before you try to build
the record (which itself is before a WRITE)

4) The item in the RECORD VARYING clause doesn't need to be updated until
just before you do the WRITE statement (you can build the record without
updating it)

5) The item in the OCCURS clause needs to be "created" by the application
logic (before a WRITE) and you can treat it like a "normal" part of the
external record when you READ the record.

6) The item in the RECORD VARYING clause is sort-of a "magic" field in that
it gets updated by the system when you read a record (and need not actually
be stored anywhere in the file).

  ****

I hope this clears up some of the differences between these two related but
not identical functions of ANSI'85 Standard COBOL.
