NULLs are flagged using a NULL indicator field. You have to set the
indicator to -1, not the contents of the field. Declare a 2 byte
integer field and use it in the SQL, following the nullable field.

01 MY-NULLIND PIC S9(04) COMP-5. (maybe COMP-4?)

To select (forgive my lame SQL):
EXEC SQL
SELECT BOB
INTO :BOB
:MY-NULLIND
FROM MYTABLE
WHERE ...
END-EXEC.

Check MY-NULLIND to determine if the contents of BOB are valid. if
MY-NULLIND < 0 then the field contains no usable data.

To write a null it looks something like...
move -1 to MY-NULLIND
EXEC SQL
INSERT INTO MYTABLE
(BOB)
VALUES
(:BOB
:MY-NULLIND)
...

Beats me. I pretty much exhausted my knowledge of NULL indicators on
the last go around. About the only thing I can suggest is to make sure
you are resetting the NULL indicator on every insert. The behaviour you
describe is what I would expect if the indicator was >= 0. On an
INSERT, the contents of the host variable should be irrelevant as long
as the NULL indicator is set to -1. Conversely, the contents of the
host variable are unusable on a SELECT if the null indicator is less
than one. If the contents matter to your application, you should
explicitly initialize the host variable following a SELECT or FETCH if
the null indicator is < 1.

Hope this helps.
