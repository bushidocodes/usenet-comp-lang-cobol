Don't be so hard on Niklas, please. English isn't his first language
obviously.

ANSI COBOL does have 'constants'. eg. figurative constants such as ZERO,
SPACE, QUOTE etc.

Quite a few other implementations such as Micro Focus allow you to define your
own figurative constants that you can use instead of a value. I think this
was what Niklas was aiming at. A sort of #define from C equivalent.

In MF COBOL you define these by using level 78 items. eg.

working-storage section.
78 fifty value 50.
78 fred value "fred".

01 pic-9-item pic x(999).
01 pic-x-item pic x(20).

procedure division.

*> example of ANSI figurative constants
move ZERO to pic-9-item
move SPACES to pic-x-item

*> example of user defined figurative constants
move fifty to pic-9-item
move fred to pic-x-item
stop run
.

User defined figurative constants are extremely useful as you don't have to
remember strange values and they don't take up any extra working-storage
space. If I put all the figurative constants I use in graphical API
programming in as working-storage items, I'm sure the programs would be 2-5
times larger!

The answer is that the "COBOL constants" as described don't exist (yet
anyway) in standard COBOL but they are in as a language extension (no,
don't groan - you don't have to use them) in MF COBOL at least. As to
what they can accomplish, well they are most flexible when used
together with conditional compilation. Values can be set right at
the start of the program and then the names can be used in place of
a literal of the correct type thereafter, as in:

working-storage section.
78 o-system value "DOS".

$if o-system = "DOS"
78 path-name value "d:\mydir\".
78 filename-max value 100.
$else
78 path-name value "/user/data/mydir/".
78 filename-max value 1024.
$end

01 filename pic x(filename-max) value
path-name & "default.dat".

procedure division.
string path-name "myfile.dat" delimited by size
into filename
display "Filename (max size " filename-max "): " filename
.

Oh, the & is the concatenation operator that is proposed for the next
COBOL standard. Used sensibly constants can be a boon, but of course
used carelessly you could get a mess. Like in the above, I shouldn't
have used the literal "DOS" in the condition but another constant so any
typo would get thrown out.
