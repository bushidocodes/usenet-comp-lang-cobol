called program( both with CMPR2 and NOCMPR2 compiler options) and not
that amount as is defined in the DEPENDING ON field!!.

  Is this true??

 If it is true is there any way for me to define storage dynamically???

program to be able to run maximum number of concurrent tasks in the

 any help is greatly appreciated.

Instead of using an OCCURS DEPENDING ON, why not just use A-FIELD as
you maximum, setting it to the actual data-length of A-DATA.

Assuming that you can pass two parameters, A-FIELD and A-DATA (for
example), define A-DATA as the maximum allowable in LINKAGE, which is
16MB-1. However, use A-FIELD as the LENGTH (as in reference
modification).

All subsequent programs must pass and use A-FIELD as the maximum
length. If you don't address A-DATA beyond the value found in A-FIELD,
you won't get an S0C4. By defining A-DATA as a maximum of 16777215, you
should never have to change the picture clause at all. The key is to
ALWAYS address A-DATA using reference modification.

Remember, when data is passed in LINKAGE, it is merely a series of
fullword addresses off R1. In your case, it will consist of two
addresses, one for A-LENGTH and one for A-DATA.

Basically, you've built your own 'DEPENDING ON' this way. If you think
about it, when passing a parameter to a BATCH program from the JCL,
you're doing the same thing as well.

Just my .02 cents....

My comments are as follows:

1.LINKAGE occupies NO storage at all. (It corresponds to an Assembler
DSECT). In effect it simply allows the storage of the CALLING program to be
addressed by the CALLED program. (Unlike C, which copies the passed
parameters to the called program, unless you pass a pointer. How many C
programmers have perished on that particular rock...?)

2. OCCURS DEPENDING ON is best avoided...If you MUST use it, ensure you set
the length into the target field, BEFORE you do the MOVE...(on some
platforms the effect is devastating if you don't do this...)

3. The 3MB area needs to be defined in the WORKING-STORAGE of the CALLING
(Main) program and in the LINKAGE  of the CALLED program. This raises
another problem in that the compiler will probably grizzle about a field
that large... (I can think of several ways to get round this, but they are
all inelegant, so I'll leave you to solve it yourself if the need
arises...:-) )

4. The answer to your question "Is this true?" is "Yes". There is no
standard COBOL construct which allows dynamic memory allocation. Most
Mainframers call an Assembler routine which issues a "GETMAIN" and similar
for "FREEMAIN", or write this stuff in PL/1 which DOES allow dynamic memory
allocation. This is not a criticism of COBOL; just a statement of fact.
Most of the time there is very little need (In commercial data processing)
for dynamic allocation of memory. You have come up with a possible one.

I believe you can solve your problem by ensuring that the Main program has a
3MB area and that the program which actually shovels it down the comm line
(the one which makes the LU 6.2 calls) is called by this program and has
addressability to its storage (via LINKAGE). You can then ensure that the
address and length of the message buffer (as required by LU 6.2) is provided
from the linkage section of the called program. It is not so silly for COBOL
to require an area equal to the maximum possible size; the message buffer
would need to be this big anyway... Of course your real concern is that you
are not actually TRANSMITTING 3MB each time. Provided you set the length of
the LU 6.2 message buffer, you won't be.

Pete.

In you're running AMODE 31, RMODE ANY together with COBOL2 or greater,
a 3MB WS Area should not be a huge overhead. If you were running the
old OS/VS COBOL, then I could see your cause for alarm. In COBOL2 or
greater, WS is essentially Dynamically Allocated from EDSA. EDSA is
normally around 2GIG for most CICS/ESA regions nowadays. Not knowing
your program flow, if the Front-End program that contains the potential
3MB WS area could be defined as a maximum 3MB in LINKAGE (as an option)
and you issue a GETMAIN on the length passed over (IE. A-LENGTH), then
you could pass A-LENGTH and the ADDRESS of the GETMAINED area in the
commarea downstream. However, note that passing of addresses could be a
potential problem if you decide to run a SYSPLEX environment and one of
the downstream modules is accessed via DPL.

Your PPT and PCT parameters regarding the allocation of data, namely
DATALOCATION and TASKDATALOC (respectively) must be set to ANY and the
GETMAIN must use the FLENGTH (Fullword Length) parameter to ensure 31-
Bit Storage Allocation.

If you're LU6.2 conversations are CONVERSATIONAL, then it would be best
to issue a FREEMAIN before your next RECEIVE (I'm assuming CICS is the
Back-End). If the sessions are dynamically allocated via an ALLOCATE,
then it doesn't matter.

As far as hyperspace, this would require all addressing of this area in
what's called 'AR' (Access Register) Mode and it must be done in
ASSEMBLER. Although I have not personally called an 'AR' mode ASSEMBLER
Sub-Program from CICS (and I don't know if it's possible), there may be
issues regarding STORAGE PROTECTION that would need to be addressed by
your SYSPROG. In any case, COBOL could NOT address any of this
hyperspace storage.

HTH....
