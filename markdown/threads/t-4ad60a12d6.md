Actually it is a very debated subject.
Many well known authors love it, others express
their criticism.
In these days in Italy there is an international workshop
(see http://numa.sern.enel.ucalgary.ca/extreme), with
experts like Kent Beck, Rober C. Martin, Martin Fowler,
Ron Jeffries.

In "comp.object" there is an hot debate on the subject.
In particular (and this was the sense for the link to
Reeves' article) in connection with a particular aspect
of XP: code as design language.
Software development is a very iterative process, in which
you produce a lot of design documents (UML diagrams --
I'm speaking about OO development).
You have to do a lot of rework on such diagrams in order
to synchronize them with code.
So, XP experts say it is more efficient to have a language
and code structuring techniques, that let you express
your design decisions directly in code.

They simply have the need of expressing in code
what they express in UML diagrams by means of
rectangles and other graphic marks, saving in this way
a great amount of time.

As I told before, many agree with XP, many don't.
I'm not an XP expert, and so these are the only
informations I can give you for the moment.
I'm studying the subject.

<snip>

Surely.  However, as soon as you say the code is part of the design, or
better yet, that the design must be specified in such a way that code is
included, then I think you are getting closer to the mark.

There is probably not a single person that has been programming for any
length of time that has not received a specification containing "stuff" that
is not clearly enough specified to be coded. To my mind, that is an
incomplete specification.

However, as soon as you say that enough information must be specified that
the code can be written, then the next question is "Why not write a program
to do it?".  After all, if the specification is "Complete", then the process
could be automated.

It boils down to language.  The natural language usually used to write a
"specification" is simply not precise enough to completely specify
everything.

A similar situation exists with traditional engineering, and is overcome by
the use of blueprints.  A scale drawing of the end result, complete with
measurements and enough detail to actually build the unit.

The only way that the same level of detail can be given for a software
specification is by the use of a specialized language.  We have known that
for years.  Actual computer languages are designed to do just that.  They
describe a proccess in sufficient detail that a computer can write a program
to do it.
