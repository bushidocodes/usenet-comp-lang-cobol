```
Interestingly enough - there is nothing in the COBOL Standard that REQUIRES
that a COBOL run-time "release storage" when a CANCEL is done.  The only
thing that CANCEL is "guaranteed to do" is to place the sub-routine in
"initial state" when it is NEXT called.

It is true that many (not all) implementations DO "free storage" - but with
Micro Focus (for example) this is actually subject to a run-time switch.
With IBM mainframe COBOL, it is dependent upon the DYNAM compiler option.
```

```
Without a Cancel the program will remain in memory until the
STOP RUN.  It is quite common (in my systems) to have some
callable programs that, say, look after system control
functionality, or file access.  These are CALLed and never
CANCELed until end of run because they have files open or
hold data items and this would be lost if they were 
cancelled or otherwise removed.

Some run times are able to discard code segments when memory
gets tight because these can be reloaded from the file - the
code is not modified.  Data segments, however, must remain
resident (at least in virtual memory) because they must 
retain their 'last used' state unless they are cancelled.
```

```
As already mentioned by several other people, if you don't CANCEL, the
memory will not be free'ed. However, you may be able to get better memory
usage by changing how the application is built.

If you compile to int or gnt code, the program is loaded by the COBOL
runtime into data space - each user on the system will get their own unique
copy of the program.  This can rapidly drain system memory as you add users.

If you compile the modules to shared object (shared library on HP) or shared
gnt (.snt - an HP specific format, really just a shared library), then there
will be one copy of the program in memory for all users. This doesn't affect
memory usage for one user but is much more efficient system wide for
multiple users.

If you're currently using int or gnt format, I'd recomment giving shared
gnts or libraries a try.

NB: Later releases of the HP-UX product replaced gnt's with snt's (but
called them gnt's) - check your specific product docs to see what you have.

Final question - you mention that you get "out of memory" errors from the
OS.  Is this the COBOL runtime reporting the error or is it the OS reporting
it on the system console?
```
