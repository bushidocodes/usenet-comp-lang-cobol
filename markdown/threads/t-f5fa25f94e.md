```
Corresponding with Tony, I think we came up with an idea.  He
suggested a time clock application.  I have need to track my time on
multiply projects - as do others in my department.  I think a network
based Time Tracking application would be an interesting, and simple
enough project.  

Initial specs - Choose Employee, Chose Project, Clock In and Out.
Some allowance for adjustment of time based on late entry should be
provided.  Maint to Add/Change/Delete employees and projects should be
included, as should some reporting - up to and including export into a
CSV for inclusion in a spreadsheet.  It's getting bigger already.

So, OO GURUs - Given the above specs, (for lack of a better word) how
would you structure the classes and methods for an OO implementation.

For now, ignore the intracies of the UI.  I will take care of that
with SP2 - but the class that handles that should be isolated from the
rest of the application, allowing any interface to be employed, from
Screen Section to Native API calls.
```
