 We can get an 8 digit (Y2K- compliant) current-date in a COBOL/MVS program
using the CURRENT-DATE function. For example,MOVE FUNCTION
CURRENT-DATE(1:8) TO WS-CURRENT-DATE.  We can also get an 8 digit
current-date by calling LE program CEELOCT.CALL 'CEELOCT' USING
WS-LILIAN-DATE,    WS-OUTPUT-SECONDS,	  WS-CURRENT-DATE,   
FEEDBACK-CODE.	  After the call to CEELOCT, WS-CURRENT-DATE has a 17 byte
value,the first 8 bytes of which are the current date in the format
YYYYMMDD.  I've heard that the only way to get an 8 digit current-date in a
COBOLprogram is to convert to COBOL/MVS and use the CURRENT-DATE function. 
It appears that you can get a Y2K-compliant current-date in a OS/VS orCOBOL2
program by calling CEELOCT. I've checked the PMAPs ofCOBOL/MVS programs and
found that the compiler resolves the CURRENT-DATE function into a call to
CEELOCT, so it appears thatthere should be no difference whether you get a
current-date from theCURRENT-DATE function or by a native COBOL call to
CEELOCT.  Do you know of a reason not to use CEELOCT to get a Y2K-compliant
current-date in a OS/VS or COBOL2 program? I understand that we should
migrate older programs to COBOL/MVS, so I am not proposing that we do not
upgrade older programs to the current COBOL dialect. However,for legacy
programs in sunsetting systems that must be upgraded immediatelyto
Y2K-compliance, it seems that using CEELOCT to get the current-dateis an
alternative to a conversion to COBOL/MVS.

Good news and medium news:

1) Yes, you can call the CEE (LE) date/time functions from VS COBOL II
programs.  This was documented in an IBM APAR as "supported" - even though
most calls to LE (non-date routines) are NOT supported.  HOWEVER,
  - such calls must be DYNAMIC (either use the DYNAM compiler option or use
the CALL identifier rather than CALL literal format of the CALL statement -
both of these require the RES compiler option - if you aren't using it
already)
 - CALLs to LE routines from OS/VS COBOL programs are *not* supported - but
probably will work.

2) If you are willing to use a CALL statement from your VS COBOL II program
to get a date with a 4-digit year, look at calling IGZEDT4.  This CALL works
whether or NOT you have an LE runtime system available (but does require
that your VS COBOL II product has current maintenance applied)
