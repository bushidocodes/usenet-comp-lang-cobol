On discussing the use of PACKED Decimal...

<begin quote>
..."Standard" more or less numeric data types include USAGE BINARY and USAGE
PACKED-DECIMAL. These were introduced in the '85 Standard and provide for
portable source code - but NOT NECESSARILY PORTABLE STORAGE. Taking the most
simple example,

05 How-Many-Bytes Pic 9(04) Packed-Decimal.

may be stored as either 2 or 3 "bytes" (a concept that isn't in the current
Standard - but will be in the next one) of storage - depending on whether
your compiler "requires" a sign-nibble for even unsigned packed-decimal
items. The general rule is that all "meaningful" digits of a packed-decimal
item take 1 nibble (half of a byte) and the sign takes another. But when the
data definition does NOT include a sign (S), then some systems still require
the storage and some don't; some store an unsigned numeric field with the
same sign-nibble as a positive number and some don't. (Most common
examples - Most IBM-compatible PC's treat positive and unsigned  numeric
fields with a X'3' sign-nibble - while most IBM compatible mainframes treat
positive numeric fields with a X'C' while unsigned have a X'F' sign-nibble).
<end quote>

Now, I can see the dilemma here... It is trying to be all things to all
people and covers the bases very well  but I ask myself whether someone
consulting the COBOL FAQ really needs to know or care about this stuff. It
has been highly influenced by your own exposure to the forthcoming standard,
which I might remind you is NOT implemented yet. People accessing a FAQ are
not really interested in the THEORY. They want simple solutions to REAL
problems NOW.

Students coming to the FAQ are looking for some simple rules to use when
dealing with packed fields.

I teach the following:

"ALWAYS make packed fields an ODD number of  digits and ALWAYS specifiy a
sign."

It is a simple and effective rule, yet the "most simple example" above,
violates both of these considerations <G>.
(Anyone from an IBM background will understand that specifying UNSIGNED
packed fields in COBOL requires the compiler to generate extra code to force
the positive sign nibble, and specifying an EVEN number of digits forces the
allocation of an ODD number anyway, but I don't think there's any need to
burden  young minds with this level of detail unless they ask why.)

The "Wisdom of Dashie" then continues...<G>

"Don't use Packed unless you have to, for compatibility with incoming data.
It is an archaic and outmoded representation that was developed to save
space when memory and external storage were prohibitively expensive, and to
provide fixed point decimal working on an architecture that was designed for
binary. "

So, we end up with three simple things to remember:

1. Don't use Packed unless you HAVE to... (I am hopeful that we can
eventually remove ALL packed data from ALL systems if the rising generation
implement this...<G>)

2. If you do use it, put a SIGN in the picture.

3. If you do use it, have an ODD number of digits in the picture.

As this format was invented by IBM, and is implemented these days by other
vendors because of the necessity for IBM compatibility, it seems reasonable
to me to explain it in terms of IBM's implementation, THEN mention that some
other implementations may hold different sign nibbles etc. People who really
want that level of platform specific detail should RTFM or post to CLC, not
look for it in a FAQ. As for the pending standard deciding that storage
could be two or three bytes or whatever, I have two observations:

1. Do I care? (COBOL is supposed to shield me from this...)
2. Are we ever likely to see "The Word made Flesh"?

</teeth grind> <G>

In defence of packed decimals:

Remember that programmers sometime forget to initialise fields, or sometimes
move crud into a numeric field.

I would much rather be woken up in the middle of the night to fix a S0C7
data exception,
that try to figure out two weeks later why the accounts don't reconcile.

Binary fields do not provide such protection - that extra sleep is not a
good trade off in the long term.
Getting rid of packed fields is like shooting the messenger, instead of
fighting the war.
I would favour quality control over "quick and dirty" most of the time.

However, I don't say that no fields should be binary - it's a matter of
horses for courses.
For example, I use indexing rather than subscripts most of the time.

I doubt the suggestion that packed fields get converted to binary format for
arithmetic.
More likely the circuits are decimal adders.
Not quite as quick as a binary adder,
but a lot faster than coverting the whole number into binary and back again.

By the way some of IBMs midrange computers have packed decimals with no sign
nybble,
so it is a shame that their own mainframes don't have better support for it.

The suggested code for converting should have specified division by 10 not
100.

Strictly speaking the x'0F' sign nybble was incorrect for a signed field.
For full compatibility with all software, I suggest using x'0C' for signed
non-negative numbers.
The CPU may not care, but some software out here is fussy about this sort of
thing.

I would also suggest using DIVIDE rather than COMPUTE, to avoid the chance
of high precision intermediates,
which some compilers sometimes go overboard with.

which says (in part),

"When you use a data item mainly for arithmetic or as a subscript, code
USAGE BINARY on the data description entry for the item. The operations for
manipulating binary data are faster than those for manipulating decimal
data.
However, if a fixed-point arithmetic statement has intermediate results with
a large precision (number of significant digits), the compiler uses decimal
arithmetic anyway, after converting the operands to packed-decimal form. For
fixed-point arithmetic statements, the compiler normally uses binary
arithmetic for simple computations with binary operands if the precision is
eight digits or fewer. Above 18 digits, the compiler always uses decimal
arithmetic. With a precision of nine to 18 digits, the compiler uses either
form.

To produce the most efficient code for a BINARY data item, ensure that it
has:


  a.. A sign (an S in its PICTURE clause)

  b.. Eight or fewer digits


For a data item that is larger than eight digits or is used with DISPLAY
data items, use PACKED-DECIMAL. The code generated for PACKED-DECIMAL data
items can be as fast as that for BINARY data items in some cases, especially
if the statement is complicated or specifies rounding."

Thanks Bill, this is "interesting" but it misses the whole point...

The issue is not whether Packed or Binary is efficient on Mainframes.

Most of us who are not working on IBM mainframes don't care...<G>
(In fact, there are mainframe programmers who don't care and quite properly,
given the power of modern hardware.)

The issue is whether PACKED decimal has a place in the modern World.

IBM will argue that it does. (As they are still charging people an arm and a
leg for hardware that implements it, they WOULD say that...<G>)

I'm arguing that it doesn't.

I summarise my position as follows:

"PACKED format was originally implemented to save expensive resources and
provide fixed point decimal arithmetic on an architecture that was
intrinsically binary. It did this very well. At the time there was little
requirement to transfer data across platforms. Today, the resources it was
designed to save are no longer expensive, processor power has increased to
the point where an arithmetic conversion overhead (except in really
exceptional circumstances) is no problem, and the sharing of data between
platforms is becoming ESSENTIAL.

I therefore submit that this is a stumbling block to the development of open
systems, it provides little gain even on the architecture for which it was
originally intended (how many COMMERCIAL calculations involve numbers of
more than 18 (or even 8) digits?), and it is of arguable use in controlling
the "quality" of data being processed on these platforms.

For the above reasons, I suggest that IBM Mainframe programmers should avoid
using it, and new programmers should be advised to do likewise."

There are millions of lines of COBOL written on platforms that DON'T support
PACKED Decimal and those applications manage fine without it. Many of the
old BUNCH  (Burroughs, Univac, NCR, CDC, and Honeywell, also include ICL
(now Fujitsu) in there... ) systems had their COBOL converted to other
platforms and still continued to run fine without the need for Packed
Decimal...

I realise this position will not be popular (especially at IBM <G>) but I
hold it nonetheless...

The sooner this data format disappears from IT, the better for all
concerned.

Pete.
