```
I'm reading in a simple sequential file that contains a 1 byte binary
field representing a 2 digit year portion of a date. I'm trying,
unsuccessfully  to convert it to a DISPLAY format.  I define the pic
clause like this

  03   im-m-yy-bin    pic 99 comp.

I then move that to

01  work-member-year    pic 99.


This results in erroneous years in the output. According to the manuals
the binary format uses 2 bytes of storage. Examination of the input
records and the erroneous output indicated the program was taking the
two bytes at that location as the binary number.

 This is on an IBM Main frame (ES9000). The compiler reported in the
listings is IBM  VS COBOL II Release 4.0. If anyone has any idea if
there is a way around this I'd appreciate the help.
Thanks

John Roberts
puck@bestweb.net
```

This is, of course, almost the extreme of the Y2K fiasco,
representing the year in, not two, but ONE byte.

       02  im-m-yy-bin     PIC X.

01  NEW-YEAR            PIC S9(4) COMP VALUE 0.
01  FILLER REDEFINES NEW-YEAR.
     02  FILLER            PIC X.
     02  OLD-YEAR      PIC X.

MOVE im-m-yy-bin TO OLD-YEAR.
MOVE NEW-YEAR TO WHERE-I-WANT-IT.

Of course this doesn't work if the input year is negative.

are consistent with the definition.

There are some interesting code relying on the fact that code is the
right endian (non portable).  If you have a choice of compilers (the
one you have does not support functions) you may consider FUNCTION ORD
as the portable method.  This returns the numeric value of the

are attempting (possibly get better optimization).

John,

I'm assuming that the 1-byte binary-year value can be in the range of
X'00' through X'FF'. If, for example, the binary-year value is X'3F'
(decimal 63), try the following:

03 WS-PACKED-X.
   05 WS-PACKED PIC 9(01)V9 PACKED-DECIMAL VALUE ZERO.
03 WS-DISPLAY-X.
   03 WS-DISPLAY PIC 9(02).
03 WS-BINARY-YEAR PIC X(01).

MOVE WS-BINARY-YEAR TO WS-PACKED-X (1:1).
MOVE WS-PACKED TO WS-DISPLAY.

INSPECT WS-DISPLAY-X CONVERTING X'FAFBFCFDFEFF' TO 'ABCDEF'.

The answer in WS-DISPLAY-X (WS-DISPLAY) will be X'F3C6' (C'3F').

I've only used this technique on an IBM Mainframe.

OR, maybe I've misunderstood. Maybe all you need to do is:

03 WS-HWORD-X.
   05  WS-HWORD PIC S9(04) BINARY VALUE ZERO.

MOVE WS-BINARY-YEAR TO WS-HWORD-X (2:1).
MOVE WS-HWORD TO WS-DISPLAY.

WS-DISPLAY WILL NOW CONTAIN 63.

On IBM mainframes, a binary field's size depends on how many digits are
in the picture clause:
1-4 = 2 bytes
5-9 = 4 bytes
10-18 = 8 bytes

Try creating a  PIC XX field around your binary value, with your number
in the second character.

Then move that PIC XX value to:

05   BIGGER-ALPHA.
     10  BIGGER-FILLER   PIC X.
     10  FILLER          PIC X.
05   MY-NUMBER       REDEFINES BIGGER ALPHA   PIC 99  USAGE IS BINARY.

Then move LOW VALUES to BIGGER-FILLER.

Now MY-NUMBER will have your value on the right, with binary zeros to
the left.  It should work now, depending on how your sign is stored.

In my previous post, I defined WS-DISPLAY as PIC 9(02), whereas it
needs to be PIC 9(03). You could encounter high-order truncation when
the DECIMAL-value of the 1-byte binary field is greater than 99 and WS-
HWORD is moved to WS-DISPLAY. Granted, we are talking about a year-
value. But, it is better to be safe than sorry.

The original post indicates that this is an IBM Mainframe. You would
get a compiler error moving the 1-byte binary field to an aligned (or
unaligned) halfword. This would be PIC X(01) to PIC 9(04) BINARY SYNC,
which is illegal. Review several of the other replies, which redefines
the receiving halfword (IM-M-YY-BIN) workarea as two separate bytes.
