    ... The EXIT PARAGRAPH and EXIT SECTION statements provide
    a means of exiting a structured procedure without executing any of
    the following statements within the procedure."

The draft seems to be missing the description of what constitutes
a "structured procedure." I would suggest that it is reasonable for
the standard to define this term and to provide a translator option
that would allow for diagnostic messages should the translator
find violations of the rules supporting the definition.

The rules for implementing "structured procedure(s)" required by
EXIT SECTION could include:

1. Sections may only be PERFORMed.
2. PERFORM section-name may only refer to a single section.
3. PERFORM may not refer to a paragraph-name in a different
    section.
4. GO TO may not refer to a paragraph-name in a different
    section.

Under these rules, EXIT SECTION would work as expected; that is,
it would return control to the executable statement following the
PERFORM that referred to the section. Without these rules, EXIT
SECTION should not be permitted.

There is a perfectly worthless rule at p 437, 14.10.13.2 SR 11:

"The EXIT statement with the SECTION phrase shall be specified only in a
section."

IOW, EXIT SECTION can't be used in a program without sections.  If a
program has any sections, all statements are in sections.

P 438, 14.10.13.3 GR 12 is the relevant rule:

"If an EXIT statement with the SECTION phrase is executed, the exit
statement causes control to be passed to an implicit CONTINUE statement
immediately following the last statement in the last paragraph in the
section."

Clearly this is a branch a point outside paragraph D, the end of the
perform range is never reached, and the program falls into F.  Since this
condition can't be detected at compile time, I can think of two other ways
this could work:  Exit the perform, or raise an exception
(EC-FLOW-something).  If this exits the perform it must exit any perform
whose range ends between here and the end of this section, which
complicates the compiler's job immensely.  Raising the exception would
also be hard to implement.  This is an unfortunate case where we just
can't engineer a safety into the tool, and programmers will be able to
write stupid code.

The wording of GR12 clearly makes the "implicit CONTINUE statement" part
of paragraph E.  Thus an EXIT SECTION exits the perform range iff the end
of the perform range is:

1) The section where the EXIT SECTION occurs.
2) The last paragraph of the section where the EXIT SECTION occurs.

The problem is that what people really want is EXIT PERFORM, but J4 has
decided that EXIT PERFORM should only work for inline PERFORM.  Don't
worry, it will be in COBOL 21XX.

type looping errors disagree on whether they should allow, disallow, or
mandate going to an exit paragraph following the paragraph being

paragraphs be stand-alone.  The first paragraph ends with a program
terminator (GOBACK), and all other paragraphs are performed as stand
alone paragraphs.

have standards which use EXIT PARAGRAPH, will have thought through the
other bad programming practices and proscribed them as well.

For this style of programing to work properly, you need the function name
in the exit paragraph name as well as in the paragraph name.  The numbers
and the function names are independent; the numbers act as an index to
help you locate the paragraph and the function name tells you what it
does.  The function names should be long enough to be "grabbed" by the
editor's global replace.  You can move a routine by global replace on the
number, and you can clone a routine by global replace on the function
name.  To write 0800-EXIT instead of 0800-FOO-BAR-EXIT is like calling
your variables A1, A2, A3, etc., instead of using meaningful variable
names.  It may save you typing, but it does not make for a good program.
