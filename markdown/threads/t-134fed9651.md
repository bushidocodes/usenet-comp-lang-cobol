I often deal with date-formatting and I use to handle it in
a different way than you tried to do. I respect personal 
COBOL styles, so I was hesitant just to present my own 
approach to you. But I think you will have to change
more than one feature.

Let us see.
Output : pic 9999/99/99
Input  : user types "dd/mm/ccyy"     (c = century).

I don't know what type of COBOL compiler you use, but I
cannot find anything about a print-format with slashes,
so /-signs. So I don't know whether it will work.
And what do you intent to do with "freleve - dat"?
Do you want to print it or is it the format for
saving data on disk or tape?
If you want it printed with slashes, then it is okay to
try it this way. If you keep it on magnetic storage, you
might leave out the slashes.

Never string or unstring numerics, because only pic X is
fit for (un)stringing. That is for sure.

I would recommend you to handle it this way.

In your Working-Storage Section:
01  plg-insertion     (accepts date from screen)
    02  plg-dd        pic xx.
    02  filler        pic x.
    02  plg-mm        pic xx.
    02  filler        pic x.
    02  plg-ccyy      pic x(4).

In your File Section:
fd  releves.
01  releve.
    02  dat.
        03  dat-ccyy  pic x(4).
        03  filler    pic x value '/'.
        03  dat-mm    pic xx.
        03  filler    pic x value '/'.
        03  dat-dd    pic xx.

This is the interaction with the user as you described it:
- accept plg-insertion
- I enter 21/03/1999
- So in dat we have 2103/19/99 due to the pic 9999/99/99
Well, in plg-insertion we now got 21/03/1999! Due to the
picture I just described. Now we move it to releves in
such a way that it fits. You code in your procedure division:
    move plg-dd    to dat-dd
    move plg-mm    to dat-mm
    move plg-ccyy  to dat-ccyy

Assez simple, n'est-ce pas?

- Harry -
