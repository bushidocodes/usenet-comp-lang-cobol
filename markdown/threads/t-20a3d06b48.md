> or' or 'both'?

Both.

The code required to test each possible status code
and execute the necessary logic can be extensive.

You normally would not want all those statements
cluttering up your Read / Write statements.

So, in the interest of programming elegance,
you use the AT END or INVALID KEY clauses
to direct program execution out-of-line to your dedicated
status code test procedures, where you can wax complex
to your hearts content, by testing for each and every possible
status code and executing the appropriate response logic.
```

You may use both at one time if you want. You can even use AT END .. INVALID
KEY and DECLARATIVES at the same time. That is, if a FILE STATUS IS is in
effect, the explicit verbs work as well, and the order of what gets control
is documented somewhere.

The main advantage of FILE STATUS is there is no method other than
file-status or declaratives to check the success of any OPEN, or to check a
WRITE of a non-keyed file.

Just as a matter of style, I do not 'mix and match.'   For me it's just as
easy to set up a FILE STATUS to query AT END or INVALID KEY conditions...

FILE STATUS is INPUT-FILE-STATUS....

05  INPUT-FILE-STATUS    PIC X(02).
       88   INPUT-AT-END      VALUE  "10".   << or whatever those codes are
      88    INPUT-NOTFOUND VALUE "23".  << sequential
     88     INPUT-DUP-KEY    VALUE "22"   << Indexed or keyed
     88    INPUT-OK              VALUE "00".

      READ INPUT-FILE
      EVALUATE TRUE
         WHEN INPUT-OK
           yadda yadda
         WHEN INPUT-AT-END
          yabba dabba
         WHEN INPUT-NOTFOUND
           tum-de-dum


Works good for me.


MCM

The status code is a suppliment for additional information, as well as a 
way of interpeting abnormal conditions (File does not exist, for 
example).  You can get along without it for simple programs, but at the 
systems level, you need it if your code is going to correct error 
conditions.  A common usage, for example, is to create a new file on a 
status 35 (File does not exist).

While I seldom code for all types of status codes, there are a couple of 
instances where I do usually code it.  The three that come to mind off 
the top of my head are newly installed systems (where you want to create 
the file), malformed Isam files (where you want to do a rebuild), and 
sharing errors (where either file or record is locked, and you want to 
either wait with a timeout then retry, or give a message that the file 
is in use and stop nicely).

There is really no way to handle any of those situations without looking 
at the file status codes, particularly sharing of files.

On the other hand, there are lots of simple programs that can get along 
without any of the above.  I'd probably not bother, for example, if I 
was writing a program to dump an Isam file to a sequential for backup or 
for conversion to a larger file.  There are default error handling 
routines that will give an error and abort for the weird stuff in a case 
like that, and since the program is being run by a pro, they are 
qualified to read the message and take appropriate action. (like trying 
to dump a file when it is in use by another process).

It is, as you say, not black and white. It is more a matter of how 
sophistocated the program is in handling weird stuff.

Donald

Note that AT END / INVALID KEY will not be activated just because a
READ/WRITE is unsuccessful but only for certain conditions.  For
example if you do a START or READ on a file that is not open then you
will not get INVALID KEY branch but there will an unsucessful
file-status.

If you want to test for AT END using the file-status only then you
should check the first byte for '1' and not the two bytes for '10'. 
If you want to test for success then also only check first byte for
'0' as any '0x' file status indicates success, such as '02' on a read.
```
