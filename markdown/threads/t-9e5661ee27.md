expanded.  Other controls may be stripped too.

Form feed is a thing we send to printers to tell them to skip to the next page.

PC text file records (not just DOS) are terminated by cr/lf. Unix text file

When you read a text file as LINE SEQUENTIAL, you do not see cr nor lf. If it
contained form feeds, you _would_ see them. 

If you want to see the 'raw' file contents, do this:

 environment division.
 input-output section.
 file-control.
    select input-file assign to (whatever)
        organization is record sequential
        file status is file-status-code.

 data division.
 file section.
 fd  input-file.
 01  sector-record.
     05  sector-byte occurs 512 pic x.

1 - Pic S9(2) COMP can be a single byte if BINARY(BYTE) is utilized -

2 - If the file is opened and read as LINE SEQUENTIAL the Form Feed is
stripped from the input and that is likely the problem here.  Two
possible solutions:

Solution A - Unless you use a directive or Varying record lengths,
Fujitsu COBOL pads line sequential records so you might be able to
read them as sequential instead of line sequential.

Solution B - Pre-Process the file sequentially one byte at a time
converting the Form Feed character to another character such as a ~ or
a ^ then read it as a line sequential file (assuming it truly has
trailing spaces stripped) an search for the replacement character.
