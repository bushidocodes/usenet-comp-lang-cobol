Most of my COBOL programming has been in a mainframe environment. 
Call by reference seems to be the only way arguments are passed to
subprograms (except for EXTERNAL, but then there is no parameter
list).  Perhaps mainframe programmers aren't adventurous, because I've
never seen arguments passed BY CONTENT or BY VALUE.  They also don't
seem to use RETURNING.   

If you code something like this:

CALL MY-FAVORITE-SUBPROGRAM USING PARM-1, "USERNAME", PARM-3, 3.1415,
PARM-5.

...the compiler will flag each use of a literal as invalid in that
context (from COBOL for MVS & VM version 1.2.2) and the link edit step
will be bypassed.  If the compiler would allow me to do this, then the
subprogram could modify the contents of "USERNAME" or 3.1415 in the
calling program's literal pool.

A colleague of mine once spent a full day debugging someone else's
FORTRAN programm that had the disturbing feature of dividing 4 by 2
and getting an answer of 4.  He finally determined that a subprogram
had received the literal 2 as an argument and had changed it to a 1 in
the calling program.

William M. Klein wrote:
> 
> > for example: in Windows 16 bit API
…[5 more quoted lines elided]…
> >               by value MB-OK size 2

> What you "describe" is the entire difference between BY CONTENT and BY
> REFERENCE.  All of you BY REFERENCE "literal" values should be changed to BY
> CONTENT - if you know that the "called" program won't modify them.

No. Wrong. Using a by content would create a copy of the original and
pass this copy by reference.  One would only need to use by content if
it were possible that the routine _would_ change the parameter (because
then it will change the copy). If you _know_ (as in this case) that the
routine won't change the parameter then it is perfectly safe to call by
reference and save the copying.

Actually 'by content' is unlikely to be safe where the routine changes
the parameter as it could still overwrite a longer area (unless told the
length) and corrupt other items in the copy area.

> (Does your compiler actually ACCEPT by REFERENCE in these cases - or are you
> saying that it "should").

Yes MF accepts by reference for a literal.

> Again, BY VALUE is also a "different" issue than BY CONTENT/REFERENCE.

Correct. But the issue actually was the claim that one _cannot_ use a
literal in a call using.  By content is a perfectly valid and standard
way of doing so.  By value is X-Open, by reference literal is usable
with MF (and others).

The reason there was a prohibition against passing a subordinate group is
what is called "major-minor equivalence".  It is primarily involved with
machines that require computational fields to be aligned on word boundaries
or some sort of boundary.  If you pass a group the called program thinks
stuff is aligned correctly but it actually may not be.  This prohibition has
been removed in the proposed standard but it is still a bad idea to do it on
some systems.

Don Nelson
