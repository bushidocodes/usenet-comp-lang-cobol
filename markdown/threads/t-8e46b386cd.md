The following is opinion based on what you posted; it is free advice and 
should be treated as such. If you want details and references, examples, and 
consultancy, I charge for that.

1. Web Services are definitely the way to go.

2. I am just completing an exercise that uses a Web Service to do something 
very similar to what you describe. The only difference is that in my case 
they are web clients who are accessing the service over the Internet. In 
yours, it will be a mainframe and the service will run over your intranet.

I opted for a Web Service after discussing implications with people I 
respect and whose judgement I value. I wasn't keen on the idea but I have to 
say, I am really glad I opted for this. The more I have learned about it, 
the more persuaded I have become.

I'm now a Believer... Web Services rule!

3. The advantage of a Web Service is that it is totally platform independent 
and can run anywhere you need it. It is invokable from any language on any 
platform that can access the network, including COBOL. It is virtually 
"future proof"; as long as that service is required, it runs, and anything 
with access to the network can use it.

4. There are no hassles with updating client software; everything is in one 
place and doesn't need to be duplicated, as it is accessible from anywhere 
any time.

5. Look at Websphere to access a Web Service directly from your mainframe. 
However, that is not the ONLY way.

(In C# it is less than 20 lines of code to provide a web service; the 
service is really a "wrapper" to present a consistent interface. Most 
services use SOAP as the transport protocol and it is very effective.) 
Calling it from COBOL is a doddle; calling it from a CICS transaction is 
also trivial. You can also use mainframe COBOL to WRITE your web service, 
but this involves interfacing to the SOAP server and is a bit tiresome 
because the tools are not there.... (under VS 2005 this SOAP interface is 
completely generated for you; C# does the WSDL as well...).

A CAUTION: I'm not sure about SOAP services under z/OS; I know it all works 
under VSE. Bill could probably comment.

General considerations:

1. You need to protect your overnight batch window, so you don't want 
cleansing running at the end of the day.

2. But, presumably, the data should be cleansed before batch updating 
occurs.

Running a web service allows the various sources to be cleansed as they are 
received. It can be transactional or batch, and it can run as background or 
foreground with appropriate priorities. As data arrives, it is presented to 
the service, and then staged for overnight update.

You don't state volumes, but given they are not outrageous, there should be 
little or no discernible impact on the system if you do transactional 
cleansing.

The concept is to create a data "bathroom" that is available 24/7. Incoming 
data is routed to it, then clean data is applied to the back end updates 
overnight, or even online after cleansing. The rules for cleansing and the 
cross referencing, sorting, and parsing, are all part of the web service. 
The design of the bathroom should be to detect the type of incoming data and 
apply the correct rule sets to it. It is like a process machine on a 
production line.

From the mainframe programmer's POV it is simply passing data to a service, 
then doing something with it when it comes back, secure in the knowledge it 
is clean and valid.

Web Services 101:

1. A web service is a Class that runs on a server and exposes various 
methods to the world. (The "world" is the visible network.)

2. Each user of the service instantiates their own copy of it as is usual in 
OO environments. (Clients see it as an object).

3. The required parameters and interfaces to the service are described in 
WSDL (Web Services Definition Language). This is so akin to XML that you 
could be forgiven for not seeing any diference :-))

4. Once we know what the service receives and transmits (from the WSDL) we 
simply need to transport that data from and to our invoking program. This is 
accomplished by "wrapping" the parameters (and the response) in XML and 
transporting this "wrapped XML" across the network using a protocol called 
SOAP. (In its infancy, when I first started using it, this stood for "Simple 
Object Access Protocol" which is pretty self-explanatory and has a 
comforting word "simple" in it...:-) Unfortunately, SOAP has moved on and is 
no longer simple, so the acronym, while it still represents the protocol, is 
no longer official and has been "de-registered"... SOAP doesn't stand for 
anything any more, it just represents a protocol...)

Given the information above, you can see that in order to communicate with 
your web service you must create XML in your COBOL program, that wraps the 
data the service needs to process. You can accomplish this pretty easily by 
using a template for the message.

Good stuff... you are finding what is needed. Don't be intimidated by XML;

Seeing you're actually trying (so many who come here don't...:-)) I'll give

Here's some SOAP....

SOAP 1.1
The following is a sample SOAP 1.1 request. The PLACEHOLDERS shown need to 
be replaced with actual values.

POST /AVSWebService/AVSWebService.asmx HTTP/1.1
Host: primacomputing.co.nz
Content-Type: text/xml; charset=utf-8
Content-Length: LENGTH
SOAPAction: "http://primacomputing.co.nz/AVSWebService/ValidateNZaddress"

<?xml version="1.0" encoding="utf-8"?>
<soap:Envelope xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
xmlns:xsd="http://www.w3.org/2001/XMLSchema" 
xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <ValidateNZaddress xmlns="http://primacomputing.co.nz/AVSWebService/">
      <AVSInterfaceBlock>STRING</AVSInterfaceBlock>
    </ValidateNZaddress>
  </soap:Body>
</soap:Envelope>Ths is describing the interface to a web service called 
"AVSWebService" and invoking the method called "ValidateNZaddress". So this 
is what theCOBOL program needs to build in order to use the service.

Notice the tag: <AVSInterfaceBlock>... the word "STRING" is a placeholder 
and needs to be replaced with the
actual interface block (The LENGTH in Content-Length: would need to be set 
as well, if we were doing this from a Browser)

Build a text file (template)  using the second part of the above... like 
this...

<?xml version="1.0" encoding="utf-8"?>
<soap:Envelope xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
xmlns:xsd="http://www.w3.org/2001/XMLSchema" 
xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <ValidateNZaddress xmlns="http://primacomputing.co.nz/AVSWebService/">
      <AVSInterfaceBlock>STRING</AVSInterfaceBlock>
    </ValidateNZaddress>
  </soap:Body>
</soap:Envelope>

Read the file and write each line to an output .XML file. When you hit the 
STRING, replace it with the interface block you have previously constructed 
in your Working-Storage  and continue to EOF.

Close the output and make your SOAP call, referencing the XML you just 
created. Everything the system needs to know is in the XML (SOAP envelope). 
The service will be passed your interface block and the ValidateNZaddress 
method will be activated.

The results will arrive back as another XML file (or in-memory object).

Notice that the service can be located anywhere on Earth if you are using 
the internet (as we are here). You could actually invoke this service from 
Sydney and it would instantly return data to you. It is a PUBLIC WebService; 
anyone can use it.

(It won't, because I have not activated it yet...and I am still building the 
login for it.. But it has been tested thoroughly and works :-))

If you want to see the format of the returned XML just enter this link into 
your Browser:

http://primacomputing.co.nz/AVSWebService/AVSWebService.asmx

If you want to see the WSDL... :
http://primacomputing.co.nz/AVSWebService/AVSWebService.asmx?wsdl

Remember that, from the Client end you don't have to worry about the WSDL, 
it is only for defining services and VS 2005 generates it all anyway...
