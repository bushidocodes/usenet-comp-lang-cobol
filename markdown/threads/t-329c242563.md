```
Is there a way to transfer a MF COBOL file, with binary data and
variable length records, to MVS?

Our shop is migrating an MVS application to Unix (specifically HPUX),
using among other things MicroFocus COBOL.  One of our clients is now
getting a file from the MVS version.  It is a variable-blocked file
(recfm=vb) with both character and binary fields (i.e. COMP and COMP-3 
as well as PIC 9 and PIC X).

This client insists on receiving exactly the same file format from the
Unix-based system.  He refuses to make any changes to his programs, and
upper management has promised (without asking us grunts, of course)
that he won't have to.

We can handle the ASCII-EBCDIC issues, the use of COMP-3, and so 
forth.  However, we haven't found a way to ship such a file from Unix 
to MVS.

I've looked into both FTP and NDM.  So far as I can tell, either can 
produce variable-blocked files on the MVS end, but only if the Unix 
file is a text file, with records delimited by line feeds.  Since
our file must contain binary data, we can't rely on line feeds to
mark the record boundaries.

MicroFocus offers their own variable-length format where each record
is preceded by something like a record descriptor word to encode the
length.  However, this format is not identical to the corresponding
MVS format.

I wrote some C routines to emulate the MVS style of record blocking.
However, when I FTP the file to MVS in binary mode, the records are
are simply concatenated into big long records, with record descriptor 
words treated as data.  The intended file structure is lost.  I
haven't tried NDM yet, but the manual leads me to expect similar
results.

Of course, the sensible approach would be to send the file in an 
all-character format, and then convert it on the other end.  For 
political reasons, however, this approach is not an option unless we 
can show that it is absolutely unavoidable.

Our Plan B is to send an all-character version to our mainframe, 
reformat it, and then send the reformatted version to the client's 
mainframe.

Can anyone suggest a less ugly approach?

Michael C. Kasten	mck9@swbell.net
http://home.swbell.net/mck9/cobol/cobol.html
```

If you're using COBOL Workbench or Mainframe Express from Micro Focus,
then they both include very powerful data file conversion and transfer
tools.

For conversion use the Data File Convert tool - this will
"intelligently" convert the necessary data from ASCII/ANSI to EBCDIC.
The decision is controlled by the user, but there are tools to assist
you by browsing record layouts, indicating which fields are to be
converted, specifying rules to determine different record layout
within the same file etc. The convertor can also convert between
different file formats - e.g. Record Sequential to VSAM. This is in
both Workbench and Mainframe Express.

For transfer, MFXFER is provided with Workbench which will transfer
source and data files, optionally doing the codeset conversions as
part of the process. The Mainframe Access Drag & Drop tool (an option
to Workbench, but a standard part of Mainframe Express) provides the
same functionality and a lot more.

Contact your MF rep for more information or take a look at the website
: http://www.microfocus.com
