There is an eye-opener for me on that same page. COBOL has no public instance
variables.  Boing! So the grads will flow into the shop with not only the
recommendation that they wrap everything, but with OO COBOL the access from
outside of a class to ALL attributes in the object is through a wrapper.
(instance methods within can get to the attributes directly). So is that
standard and settled?

Well, I guess my next question is not that urgent then. But if we do get this
feature, then there is the fact that changing an attribute that is PROPERTY
immediately changes the interface
of the accessor and mutator, and would require a complete review of all users
of these fundamentals.  Whereas manually coded accessors could stay put,
conceivably, in some change situations.

In otherwords, we are assured by theorist that the payoff of wrapping
everything is that we can change the internal implementation, and not
necessarily have to modify any users of wrapping functions. But the
automatically generated get/set methods also are _automatically_regenerated_ if
we touch the PROPERTY field. So that might not be so transparent. 

But I am just learning this part, and am not clear on 'the repository'; so
can't see if this really matters.  I am kind of just exploring this one. I am
keen on the notion of automating the technical worker, and this mechanism
obviously relates.  But we need to know the pitfalls as well.

What COBOL lacks are variables that are LOCAL only to methods that
persist between method invocations.  Otherwise, COBOL has what you
think it lacks (at least standardwise).  To my knowledge, there are NO
implementations of the standard.
