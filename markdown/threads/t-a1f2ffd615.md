I agree with Danny.

But as to how long COBOL will be around, just think about it. There are
literally millions of COBOL programs out there; many are 20-30 years old
already. Most large companies still have tons COBOL of batch programs.
The time and cost to fix all of them for Y2K is already turning MIS and
finance managers gray. The cost of totally rewriting all that code would
be truly staggering. On top of that, COBOL is probably still the best
language in common use for writing many types of business programs. Most
'hot' languages don't offer the scaled integer data types or the kind of
data formatting COBOL does, and which businesses need. Once the Y2K
problem is past, there will be a huge backlog of new systems needed which
have been delayed to implement Y2K. And the 'bottom line' pressure in
business encourages an "If it ain't broke, don't fix it" attitude. Barring
some truly revolutionary new concepts in computers and/or compilers which
would make those old programs unnecessary, or allow replacing them easily,
my guess is that it will be a long, long time before all that COBOL code is
gone.

This brings to mind an image. Imagine, some decades from now the "Last
COBOL Programmer", and old geezer making one last code change to keep the
"Last COBOL Program" running just a few days more until it is replaced, and
he retires. Sigh. Brings a tear to your eye, doesn't it. ;-)
