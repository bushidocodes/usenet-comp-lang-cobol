Interesting view. I would.

As a matter of fact I have, I once were put in charge of an old payroll
system, which noboy knew very much about there then, except that it worked.

I realized that if I were to do serious work with this program. I would have
to get known to it. What is a more efficient way to get known to a program
than cleaning its code? Nothing.

Yes I got my codestyle, and I force any code I work with into that style
unless there are minor changes.
This is not only because I learn the code better, but also because cobol
code as any code, needs a cleanup once in a while. An example of the above
mentioned the payroll program; There used to be 6 programs related to salary
calculation, this number were reduced to 4. All of these programs were
significantly reduced in terms of code size. Actually, one of them as much
as 2/3! No, it was not all just comments. But leftover code, not to be
removed in case bla. bla. bla.
Needless to say, performance increased as well.
Today I may look up these programs, find what I look for, and understand it
right out of the box. I can ensure you that was not true for the original
version, which were considered a "black box" "not to be touched" back then.

of cleanup when I open a program, but there are times when code should be
left as alone as you can.  It you are asked, professionally, to take a
specific bug out of a program, then tinkering with other parts should be a
no no, without specific approval for that tampering.

There is a major difference between doing a small fix on a production
program, and doing a major overhaul. You have to use judgement to determine
which is which.

If you rewrote a program here because you didn't like the style, you would
probably be sacked.

It is even worse if you rewrite a common sub routine.

A rewritten program is not a maintained program.  It is a new program.  I
don't care how much regression testing is done,  system testers and QA
concentrate on the planned modifications not on the areas that they expect
to be unchanged.

I know this from bitter experience.  I once did some unplanned 'minor'
changes to make a screen seem 'more professional' and for efficiency.  These
changes worked perfectly through all the testing, system testing etc but
were a disaster when actually sent out to the client who requested the
original mod.

The testers did not concentrate on the area I'd changed because it was very
very complex to test fully (something I hadn't realised) and those functions
(features) hadn't changed for years.

Sometimes things are written 'badly' because its the correct thing to do.

Regards
Rick
