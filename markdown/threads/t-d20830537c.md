```
Your book is wrong. Passed parameters in the calling program may be at
any level; in the called subroutine they must be 01 level. There is no
restriction on group/elementary in either place.

In CALL BY REFERENCE (the most common), when passing to a subroutine,
what is actually passed is the address of the first byte of the passed
parameter. Whether the (logically) passed item is a single byte,
10,000 bytes, elementary item, group item, or a sack of nuts, is
irrelevant; The compiler doesn't care - the linker doesn't care. Only
the address of the beginning byte of the logical parameter is
furnished to the called program. What the called program does with
that address is up to it.

Here's one that does a body good:

05  MYBYTE  PIC X.
...
CALL 'MYSUB' USING MYBYTE.


(MYSUB)

LINKAGE SECTION.

01  MYBYTE  PIC X(10000).

PROCEDURE DIVISION USING MYBYTE.

MOVE HIGH-VALUES TO MYBYTE.
GOBACK.

This is an exaggeration, but much sweat is sometimes expended in
tracing bizarre symptoms to a mis-match in calling/called parameters.
But note that the above silliness IS possible.
```

```
Actually ...
  There is an obscure (and not enforced by many compilers) rule that says
that a passed parameter *must* be defined as an 01-level, a 77-level, or an
elementary item.  "intermediate" level group items may NOT be passed in an
ANSI'85 conforming compiler (unless they have an extension - which most
compilers do).  This "rule" will be removed in the next Standard.

For example, given:

 01 Group-1.
     05  Group-2.
          10 Elem1  Pic X.
          10 Elem2  Pic X.
     05 Elem3   Pic X.

ANSI rules allow you to pass Group-1, Elem1, Elem2, and Elem3 but *not*
Group-2.

(The historical reasons have to do with slack bytes and synchronousation -
but most compilers have not enforced this rule for YEARS)
```

```
Ok, the *actual" problem case (still not worried about by any compilers) is:

  01  Group1.
       05 Elem1  Pic X.
       05  Group2.
             10 Synced-Field  Pic S9 Binary Sync.
       05  Elem3  Pic X.

 Call "whatever" using Group2
        vs
  Call "whatever" using Synced-Field

In *some* implementation any required "slack" bytes to put synced-field on
an "appropriate" boundary are included in Group2 - while in other
implementations they are put BEFORE Group2.  (both are standard conforming)

The called program may or may not expect its "received" item to be on an
appropriate boundary - and "other" peculiar things can happen
(theoretically).  Anyway, this rule is being removed from the Standard - and
isn't even enforced by any existing compiler (that I know of - but there may
be some somewhere that does)

An even "uglier" example is:

01 GroupA.
    05 Tabl occurs 5 times.
          10   ElemA Pic S9(01) sync
          10   ElemB Pic X.

Call "whatever" using Tabl (2)
```
