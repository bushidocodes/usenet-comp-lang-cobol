```
As I think the original poster knew, you canNOT successfully use "standard"
(dumb) ASCII/EBCDIC conversion tools for moving COBOL-created (or known)
files from an EBCDIC to an ASCII environment - or vice versa.  The problem
is that you really ONLY want to convert individuals fields that are defined
as "character" (PIC X in COBOL terms) and (sometimes - but not always) the
sign "nibble" of display and packed numeric fields.  You do NOT want to
convert binary (or COMP) fields - unless you need to convert them from
big-endian to little-endian.

The bottom-line is that to move a file from an ASCII to EBCDIC environment
(or vice versa), you have two choices:

1) In the "original" environment, create your file defining ALL numeric
fields as USAGE DISPLAY with the SIGN IS SEPARATE option.  Then you can use
any "dumb" ASCII/EBCDIC conversion tool - and no data will be corrupted.

2) Use a product that is capable of reading the file-layout (field by field)
and do only the conversions that are required.  Such a product may be
required to "understand" (and detect) different record layouts.
```

```
>- convert downloaded binary Ebdcdic files in ascii files(and import in
Access
>or excel)


You may have problems with this one, if I read your posting correctly.
EBCIDIC to ASCII, to my understanding, is only valid for text files. You
state above that you are looking at "binary Ebdcdic". My interpretation of
that would be you have mixed text and binary data in your file. When you
convert the data with any simple utility it will convert the binary data
also, thus corrupting it.

You may be downloading your data incorrectly. To the best of my knowledge,
using FTP you have to set the download mode to BINARY to download the data
without EBCDIC to ASCII conversion. If your FTP session is in ASCII mode the
text as well as the binary data will be converted. Make sure you are in
BINARY mode then download the file again. I have worked on several different
IBM MVS mainframes and never had a problem, but they may use ASCII.
```
