```
Jason,

**** Begin Dissertation ****

        Could you substitute a CICS Dynamic CALL, with Program "A" calling
(instead of linking-to) program "B'? This would only make sense if "A" calls
"B" more than once during its invocation.

        What is the size of program "B"? Does it have a large WS allocation?
Do you rely upon the VALUE clauses to be in affect when you currently issue
the LINK-API ? Is this a required DPL (Distributed Program Link) ?

        When "A" Dynamically Calls "B", program "B" must return to program
"A" via a GOBACK and WS variables are left in their last-used state. If "A"
Links to "B", program "B"  must return to program "A" via a "CICS RETURN".
When a "GOBACK" is issued in a "CALL" scenario, the WS of program "B" is NOT
freed until task termination, whereas, a "CICS RETURN" (issued in a LINK-API
scenario) frees the storage acquired by "B". Please keep this non-freeing of
residual WS in mind as it could save you from the dreaded "SOS" situation.

        Note that if you issue a "CICS RETURN" from program "B" after being
Called by program "A", this will terminate the run-unit and possibly
terminate the entire task.

        If program "A" were CICS/Assembler, you could NOT Dynamically Call
Cobol program "B". In this case, you must use continue to use the LINK-API.
If everything is COBOL, then this is a moot point.

**** End Dissertation ****

        Sorry for the long-winded assessment but, Dynamically Called
programs have their place, but require special consideration before
implementing. In regards to your dilemma, I have NEVER issued a "GOBACK"
from a program that had been accessed via a LINK-API and never would
(especially if Linked-To from CICS/Assembler). The old "UNPREDICTABLE
RESULTS" warning comes to mind and this is what may be occurring in your
situation as it seems to be unpredictable.

        BTW, coding a GOBACK directly after the CICS RETURN is common
practice and can be left as such. In spite of what the IBM manuals indicate,
I think you can't go wrong in changing the GOBACK's to CICS RETURN's. I've
been coding CICS/Command level (Macro prior to that) since Version/Release
1.4 (1975?) and this is the way it has always worked. Grandpa says "If it
works, don't fix it", but in this case, you must fix it before going forward
;o)

Carry On,

Bill
```
