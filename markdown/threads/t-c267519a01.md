```
Actually what is passed is the address of the first byte of each argument.
It is up to the called program to define the lengths and format of the
passed data. There is no easy way for a called program to know, for sure,
the characteristics of what's actually passed. The called program takes it
on faith that it's getting handed the correct data.

Even though the called program can't determine what's been passed; the
calling program can.

SOLUTION I.

Try this (called program):

LINKAGE SECTION.
01  8BYTE-PARAM        PIC X(8).
01  6BYTE-PARAM        PIC X(6).
(other params)

PROCEDURE DIVISION.
    ENTRY 'TIME-6' USING 6BYTE-PARAM (other params).
    MOVE 6BYTE-PARAM to WS-6
    Perform MESS-WITH-SIX-BYTES
    GOBACK.

    ENTRY 'TIME-8' USING 8BYTE-PARAM (other params).
    MOVE 8BYTE-PARAM to WS-8.
    Perform MESS-WITH-EIGHT-BYTES
    GOBACK.

Then the calling program does one of the following:

    CALL 'TIME-6' USING MY6BYTES

or

    CALL 'TIME-8' USING MY8BYTES

depending on what the calling program has at hand.

SOLUTION II.

Don't pass any times. Let the subprogram access the clock and figure out
everything.

For example:

CALL 'START-CLOCK'
(time passes)
CALL 'END-CLOCK' USING DURATION.

(Subroutine "MYCLOCK" has two entry points (as above): "START-CLOCK" and
"END-CLOCK")

In sum:

I don't see how you can inspect a field and determine whether it contains
six bytes of valid data or eight. That is, suppose the passed field contains
a six-byte time, but the next two bytes in memory, accidentally, contain
random data that SEEM (to the subprogram) to be part of an eight-byte value?
No joy in that.

Let us know what you decide.
```

```
Nope - (or at least not with all compilers) BY CONTENT *still* passes the
address of a TEMPORARY copy of the passed parameter and there is no way to
determine the length of what was passed (as it still is only an address).
BY VALUE (where available) *might* work for some compilers - but even that
isn't guaranteed to do what is desired.

The bottom-line is that "standard COBOL" ('85 Standard) simply has NO way to
determine the length (in a subprogram) of the parameter passed to it. The
2002 Standard introduces the "ANY LENGTH" feature for Linkage Section items,
but I don't think that will give you what you want either.
```
