The animator is very easy to use.
One can set break points by putting ones cursor on the line in
question and issuing the proper series of keystrokes.
One can view variable values by placing ones cursor on the variable
name in question and issuing the proper series of keystrokes.

The proper series of keystrokes is easy to determine as there is
a menu at the bottom of the screen.

Here is the bad news:
The animator and the user's COBOL program run as the same process.
The animator uses the interpreted code to debug.
This leads to at least two problems:
1) Wild pointers in the user's COBOL program can affect the running
of the animator itself. It is very frustrating to have a memory
fault occur in the animator itself.
( I prefer to have the debugger only quit running when i issue the
quit command).
2) The additional memory use of the animator can mask or move
an error.
( I prefer to have errors reproduce with or without the debugger).
3) If one intends to run the statically linked version of the program
in production, one is not debugging what will run in production.
( I prefer to debug code in manner as close to production as possible ).
4) Lack of support for third party debugging tools.
mF does not include enough debugging information in their
object code to allow third party debuggers to run, Third party
tools such as purify also do not run.
( I prefer to have a choice in the debugging tools that i use).

background info:
Traditional UNIX debuggers run as separate processes from the debugged
program. The debugger has privilege to query into the memory space of
the program and start and stop it via signals dedicated to debugging.

I have pestered mF for years to support use of traditional UNIX debuggers.
I believe that they have not supported third party debuggers in the
past because of the number of different UNIX platforms. I know that
they have taken steps to supporting object code debuggers in their
current release on some platforms, but I have not recently tried to
use the object code based debugger.

Once mF supports object code debugger, I hope that they allow an
option on the link phase of the compile to not allow int code to run.
