Any *conforming* compiler ('85 or 2002 Standard) *must* already issue a
message for missing scope terminators except in one case.  (This message
needs to be either an error message or an "extension" flagging message -
depending on whether this compiler has certain extensions.)

The code:

 Read ABC
   At End
      If X = "Y"
          Display "this"
      Else
         Display "that"
   Not at End
      Display "something else"
End-Read

is NOT conforming source code and a compiler error must be issued (by
conforming compilers - when extension flagging is specified)

Similarly,

   Call  "ABC"
      On Exception
         Add 1 to Num-Field
               On Size Error
                     Display  "Too Big"
      .

needs to get a compiler message.

    ***

The *only* time that a conditional statement without a scope terminator may
be NESTED within another statement is in either side of an
    IF ... THEN
    ELSE
construct.

Therefore, what you are asking (in another thread) for is not flagging of
missing END statements but unmatched IF-THEN-ELSE-END-IF phrase.

Rather than such a (required to be informational - because the logic MIGHT
be correct) message, IBM's compiler shows "nesting" levels in their compiler
listings.  To me this seems like the "best of both worlds".  It doesn't
imply that something is an error (when it may or may not be) but DOES give
the programmer a "visible" clue in the listing of what IF's are matched with
what END-IF's.

For those of you not familiar with how this looks, see:

 http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3pg10/2.5.3.2

and notice the first column of single digits and how it is EASY to tell
which END-IF goes with which IF
```

I won't go into the LONG and DIFFICULT work and study that the Standards
committees did concerning the text,

"When statements are nested within other statements, a separator period
which terminates the sentence also implicitly terminates all nested
statements. Whenever any statement is contained within another statement,
the next phrase of the containing statement following the contained
statement terminates the scope of any unterminated contained statement".

However, suffice it to say that this was "studied" in depth and there were
corrections made to that section to make it clear that the ONLY time a
conditional statement may be nested within another statement is within an
IF/ELSE structure.  Therefore, the only conditional statement that can be
terminated by a period is an IF/ELSE *of* if it is the "outside" statement
before a period.

This means that

  Call "ABC"
     On Exception
           Display "the displays statement is imperative"
     .

Is conforming - while

  Call  "ABC"
     On Exception
             Add 1 to NumField
                On Size Error
                       Display  "You can't nest a conditional ADD statement
within a CALL ON EXCEPTION"
         .

is NOT conforming (and requires a compiler error - or an extension flag)

Similarly,

If A = B
    Display "This is imperative"
Else
   Add 1 to NumField
     On Size Error
          Display "nesting a conditional within an ELSE is valid"
      .

is conforming while

If A = B
    Display "This is imperative"
Else
   Add 1 to NumField
     On Size Error
          Call "ABC"
             On Exception
                  Display "You ma;y NOT nest a conditional within the ADD ON
SIZE ERROR"
    .

is NOT conforming (and requires a compiler message - error or extension
flagging) as there is a conditional nested within the ADD statement.

    ***

This is why I say that OTHER than with nested IF statements, there is never
a time that a conforming compiler won't issue a message when a scope
terminator is missing.

P.S.  If anyone tries this with any compiler - with extension flagging
turned on - and doesn't get the messages where I say they will, I would be
interested in which compiler (which claims to conform to the '85 Standard)
they are using.
```
