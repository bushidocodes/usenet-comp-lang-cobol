We have developed technology to permit synchronous bidirectional control and comunication with
DOS sessions from a Windows program. This allow things such as DOS programs, batch files etc.
to run in DOS but report progress to a Windows session and to be controlled from Windows. The
debugger works the same way, The user interface is in windows but the debuger runs in DOS.
Obviously this technology is very sensitive to the nature of the DOS Box and how it operates.
(OS/2 prohibits a windows task from starting DOS for example) and I presume that Win95 behaves
differently than Win31.

While this was true under OS/2 2.x. a WinExec() under WARP works, but
it's always asynchronous because OS/2, not Windows handles the actual
VDM. If you're running full-screen, WARP switches to the desktop to
open the DOS window. It does not, however, switch back when the
DOS process is finished.

KEDIT for Windows nicely demonstrates this, as you can execute a DOS
command from within KEDIT, i.e. "DOS CHKDSK".
