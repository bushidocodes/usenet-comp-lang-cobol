In "Standard" COBOL, there is NO way (currently) to use "real numbers" (i.e.
what I would call floating-point numbers). The only way to store and use
numbers is via format where very little decimal point is kept, stored, and
"reliable".

May (by now most) compilers DO have extensions that support floating-point
items (so you can pass them back and forth to C programs - among other
reasons). The storage and "reliability" of these items is very implementor
dependent and it is quite true that "15.00" may well be stored as
"14.999999". One of the more common ways of doing this (on PC and UNIX
COBOL products) is via a mechanism that is defined by IEEE.

In the draft of the next COBOL Standard, support is being added for
floating-point items. I won't get into the details but some times (well
defined times) the items will be stored with 31 significant digits of
"reliability" and in other cases (again well defined in the draft Standard)
it will be totally implementor defined as to what digits are reliable.

I hope this helps. If not, can you make your question a little clearer?

All datatypes in a language are either directly implemented by your
hardware or are built out of datatypes that are directly implemented by
your hardware. If you use the same datatype in C and in CoBOL, and run
the program on the same machine, the results will be the same.

There are no built-in real numbers in C or in CoBOL. The C datatypes
"float" and "double" are floating-point binary. If CoBOL on your
platform has floating-point binary datatypes (often COMP-1 and COMP-2),
you will find that the CoBOL compiler emits the same assembly-language
instructions as the C compiler for these datatypes.

The other common C datatypes, "char", "int", and "long" are fixed-point
binary. If CoBOL on your platform can match the datatype (often
variations of COMP), you will find that the CoBOL compiler emits the same
assembly-language instructions as the C compiler for these datatypes.

It happens that the hardware on which CoBOL was invented also had
fixed-point decimal datatypes, whereas the hardware on which C was
invented did not. Therefore, the CoBOL language has built-in fixed-point
decimal datatypes, and C does not.

When CoBOL is implemented on hardware that has fixed-point decimal
datatypes, it uses them. When CoBOL is implemented on hardware that does
not have fixed-point decimal datatypes, the implementor is responsible
for simulating them out of whatever hardware datatypes there are.

Of course you can simulate fixed-point decimal datatypes in C, but there
are two disadvantages: first, even though it is simple to do, you have to
do it, whereas with a built-in datatype the compiler writer has already
done the work (and the testing); and second, even when C is implemented
on hardware that has fixed-point decimal datatypes, C can't take
advantage of them. (Recall that the first port of C was to an IBM 360 at
Princeton in 1977.)

Now here's some good news. The odds are that your problem is not where
you think it is. Most people who have "precision problems" with
floating-point binary or floating-point decimal are actually looking for
fixed-point binary or fixed-point decimal. Currency is a fixed-point
number! Never store currency in a float or a double.

You will find that CoBOL is nice because fixed-point decimal is a
convenient feature, and CoBOL has many other convenient features, but you
probably don't _have_ to convert from C to CoBOL if you don't want to,
because C's fixed-point binary is perfectly serviceable up to
$21,474,836.47 (or more, if you're running 64-bit hardware).
