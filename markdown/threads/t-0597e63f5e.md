Its not a bad shot at it. Object Oriented people tend to think of it
as a group of data (attributes) and a set of functions (methods) which
operate on the data. The class is a object type (specifies a size for
attributes, etc. Think of it as a cookie cutter.) Objects are
"instances" of the class, typically dynamically allocated. Each of
them has their own "copy" of the attributes (and methods) of the
class.

Code re-use by inheritance is when you create a new class which has
all of the attributes and methods of the previous class, plus extends
it by adding new attributes and methods. You only have to specify the
new attributes and methods.

I don't know how to break it down any simpler. A good example of OO
stuff at work tends to be GUI stuff. C++ Builder (Borland) starts
with a basic class TObject, which is an item on the screen, and
derives (creates new versions with more specific features) all of the
screen components (TListBox, TComboBox, etc) from it.

Disclaimer: I don't know Object Cobol. They may have a different
idea about this whole thing. In fact, I tend to write stuff in C all
the time, even when I am supposed to doing C++, Milk Language?

(Oh -no- COBOL bashing)
What I dislike about the COBOL programming style, is that even if
someone is using COPY commands for code re-use, it doesn't enforce/
encourage good programming practice of specifying the inputs and
outputs of a section of code. I don't know about other variants of
COBOL, but I am stuck documenting and converting legacy UNISYS
COBOL and it drives me crazy.

My understanding of OO technology (very limited) is that once you define
a process for an object then all objects derived from that object will
inherit the process.

For example, suppose you fefine a print process that will dumpt the
contents of an object in hex. Then any derived process (derived is the
wrong terminology, but I don't know the correct terminology) will
inherit the ability to be printed in hex. Just send the print message to
the object and it will print itself. No coding required.

Later on you may define an object that needs to be printed in EBCDIC and
formatted. Just define a new process called print_pretty. Then you can
send the object a message to either "print" or "print_pretty".

Later, when you define another object, if you define it based on this
second object it will automatically inherit both print capabilities.

Jeff,

Jolly good topic for this news group!

1. A Transaction Process Monitor System is an application system
architecture, providing a specific set of functions for a multi-user
environment. Object Orientation is simply a coding methodology and
technique. A Transaction Process Monitor system may be written using
Object Orientation; where as, a Transaction Process Monitor system
cannot be used to produce an Object Oriented programming environment.

2. The terms "INVOKE" and "CALL". Not too long ago they were very much
synonomous, but not so true today. Now with the inclusion of Object
Orientation to COBOL in the current ANSI COBOL standards draft, INVOKE
may become a new command to "call" an object and have it perform an
operation (method) upon its data; the result potentially being a
subsequently addressable element. (Problem in concept is, we used to
invoke commands and call programs/subprograms, but soon we will invoke
the INVOKE command. With other language OO implementations, the
resulting objects are coded as if they were commands. Similar to some
languages' support of "function procedures", where the language supports
programatic level expansion of the language. This is a contradiction to
the COBOL standardization. Let's stick with INVOKE and keep the
language pure.)

Here's a real world example; as to the concept of INVOKE vs. CALL:

I CALL my wife and tell her "I going to be late for dinner. I'm playing
poker." I go on to tell her, "I have a royal flush and I'm about to
invoke a raise with a call." It INVOKEs a response. The call ends. I
return to the game, and make my CALL. Instead of a invoking a raise,
everybody folds and the game ends. I return home. We sit down to
dinner. The INVOKEd response is addressable. The call is history; both
the telephone CALL and the other one. The next morning the INVOKEd
response is still addressable. Hey guys, finalize and approve that
standard, will you, we need the rest of the approved standard syntax to
clear those invoked responses from memory. (Honest Honey, this is only
a hypothetical example;^)
