```
The company I'm working for has decided to stop using it's fourth
generation language (which will not be supported after 2004), and go
back to COBOL II.

Of course this means that all existing code has to be converted to
COBOL, and to do this they have bought a program from a company in
Norway.  With homemade pre- and post converters the program more or
less works.

We've run out of time to perfect the process, and have to show some
results, which means that a number of us are correcting the last 20
types of errors by hand.

Here's the rub - one of the things that the source code conversion
program didn't do well, is code indentation.  In the complex
if/evaluate/perform structures created by the conversion program, a
lot of the code has unnecessarily been pushed far to the right.

 The bosses don't care about the way the code looks, but reading this
junk is a nightmare. I've been unsuccessfully looking for a program
that could redo the indentation.

Anybody know of a (preferably free) program that could do the job??

- J.B.
```

Well, in part we don't have the time required.
In part the original conversion program was supposed to do the job,
and didn't.

It isn't as simple at you might suspect - you have to take all the
COBOL reserved names into consideration (in the version of COBOL you
are using), as well as text strings in the source that use
continuation marks, all the variations of the different COBOL and SQL
statements and a number of other factors.

I've been told that the reason that the conversion program made the
mess that it did, was that it did not consider the problem statement
by statement, but line by line, which must have been easier to code.

A couple of simple ideas to add to the collection of excellent responses you
already have. 

1) if any of the tools you are using was intended for multiple platforms or
infact you used it on a PC, there is atleast some chance that intermediate
files holding your 'converted/generated' code have tabs rather than spaces.
Tabs can interact with editors and filters in a way that makes some lines look
way over to the right (or in filtration actually instantiate big groups of
spaces to accomplish the 'tabbing'). 

Do you know if the indentation is actually spaces in all intermediate files, or
is there any chance that a tabs are  in there? If this is a tab problem you may
have easy solutions.

2) The other idea is to just write a simple fixer rather than a perfect fixer.
assuming no posted suggestion gets you to a pretty printer that can create the
proper output. 
Instead of writing something that understands all of COBOL or all of the
procedure division just think in terms of finding large sets of lines that are
