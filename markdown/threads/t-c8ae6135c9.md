Robert Wagner wrote:

> it's about culture. ...
> decimal arithmetic sounded like a Good Idea .. because
> that's what the customer was comfortable with.

No. Decimal arithmetic _is_ a good idea.  Or specifically scaled decimal 
arithmetic.  Trying to do business calculations in floating-point or in 
intergers gives _huge_ problems.

Given that most business applications read display format input (card 
packs, paper tape, input screens, web server forms) and produce display 
format output (printer, screens, html text) then catering for scaled 
decimal calculations is a good choice.  Otherwise a large amount of CPU 
time is spent converting between decimal and binary.

While the current performance measures are of _one_type_ of data, this 
leads to invalid conclusions about what is the 'best' to use for 
application data (ie other than loop variables and counters) because it 
takes no account of the neccesary conversions between internal and external 
formats.

As you also say, disk records should be display format.

On some systems this may result in best overall program performance being 
when internal calculations on appliocation data use display or packed 
formats.

The S/360 thru z/ARCH systems from IBM are interrupt driven. 
Therefore, an "MVS" system tends to be quite noisy and the tests 
can't be run in a deterministic fashion. VM and VSE environments 
are also noisy, but since I don't get to work with them that 
much, it is difficult for me to tell you how noisy they may be.

The only way to run tests to find out the true times for 
instructions is to get the machine in stand-alone mode and run an 
architectural testing program (that's rather what you are 
attempting to do).

There are a few other things you need to know:

Cache lines are 256 bytes long for the G6 chip sets (as I 
recall). And these machines have I & D banks (shades of the old 
U/1100 machines!). So one needs to check the difference in 
fetches where data is mixed in with the instructions. And, where 
one has "generated instructions on the fly" there is the 
possibility that there will be serious performance degradations 
(ask SAS about that one).

Since different machines implement different stages for the CPUs, 
it would help to know how many bytes can be processed in one 
microcode cycle and how many stages there are and what their 
functions are.

Armed with this, you can then remove the problems of "locality of 
reference", timer 'rupts, I/O 'rupts, etc. to find out the "raw" 
time of an instruction.

But in the real world, the way programs are written (or object 
generated), the type of alignment a series of instructions has 
(page, double-word, cache-line, etc.), and how busy the system is 
will have large effect on the results.

I do respect your attempts to get these timings, and this is why 
IBM doesn't publish this kind of thing any more (they use to with 
the S/360 machines -- Functional Characteristics manuals). And so 
MIPs has truly come to be Meaningless Indicators of Performance.
