As others have pointed out, without seeing the code we can only
guess. Here is my guess:

There are two main ways to read a file sequentially. One way is
to start with a "prime read" before entering the main loop, and to
do subsequent reads at the end of each iteration of the main loop.
The other way is to put your read in a single place, namely at the
top of the main loop, and bypass the rest of the loop when you reach
end-of-file, using either an IF or (shudder) a GO TO.

My guess is that you are combining these two approaches into a
mish-mash, as in this highly schematic example:

MAINLINE.
PERFORM BEGIN-JOB.
PERFORM PROCESS-RECORD
UNTIL END-OF-FILE.
*
BEGIN-JOB.
PERFORM READ-RECORD.
...
PROCESS-RECORD.
PERFORM READ-RECORD.
PERFORM PROCESS-RECORD.
*
READ-RECORD.
READ INFILE INTO INREC
AT END SET END-OF-FILE TO TRUE.
...

This code does a prime read, but then does the second read without
processing the first record. When it reaches end-of-file, it
nevertheless performs PROCESS-RECORD, thereby processing the last
record a second time.

There are two ways to fix such a program:

1. Eliminate the prime read, and don't perform PROCESS-RECORD when
you reach end-of-file.

2. Retain the prime read, but in the main loop, put the read at the
end instead of the beginning.

If your program doesn't fit my guess, post code and we'll try again.

Linda, I'd like to suggest a more modern(?), structured (I hope so )
approach to this ancient problem with COBOL. I'm assuming you're using a
modern (COBOL 85 or more) compiler which supports scope delimiters:

MAINLINE.
PERFORM program-initialization

PERFORM PROCESS-RECORD
UNTIL END-OF-FILE
GOBACK
.

PROCESS-RECORD.
READ INFILE INTO INREC
AT END SET END-OF-FILE TO TRUE
NOT AT END PERFORM record-processing-logic
END-READ
....
record-processing-logic.
...

"program-initialization" will OPEN INFILE (for input). If it cannot OPEN
the file it might have terminating logic, or it might set END-OF-FILE to
TRUE and not terminate (can you see what'll happen in the PERFORM in
this case?).

As someone else said, you can't READ the same record twice; however, you
can list (or otherwise process) it twice. Finally, as others have also
said, if you continue to have problems, post your code.
