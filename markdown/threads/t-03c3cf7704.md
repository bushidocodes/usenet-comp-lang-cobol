I can state with a fair degree of certainty that for Unisys MCP/AS dialects
of COBOL, so long as all other things are equal, the performance of the
COBOL statement "ADD A TO B" is absolutely identical to the performance of
"COMPUTE B = B + A" on that system.  Though the object code may differ
slightly between COBOL *dialects*, the object code is *the same* for these
two *statements* within a given dialect.

If you're trying to add exactly two variables together and store exactly one
sum into exactly one of them, either ADD A TO B or COMUTE B = B + A is OK.
But if you're trying to add three variables together and add the sum of
those three into each of four variables in turn *in a single statement*,
between ADD and COMPUTE the choice is clearly ADD.

The capabilities available to ADD are different from those available to
COMPUTE, according to the various COBOL standards. While
    ADD A, B, C   TO  D, E, F, G
is both legal and meaningful,
    COMPUTE D, E, F, G = D, E, F, G  +  A, B, C
 is neither.

To do something similar to ADD A, B, C to D, E, F, G using COMPUTE you end
up with something like
    COMPUTE temp = A + B + C
    COMPUTE D = D + temp
    COMPUTE E = E + temp
    COMPUTE F = F + temp
    COMPUTE G = G + temp
That's FIVE statements, not ONE, and requires an extra temporary to boot.

    -Chuck Stevens

"JerryMouse" <nospam@bisusa.com> wrote in message
news:P0qdnSxU_tJXMriiXTWJkw@giganews.com...

> I stand corrected - if the receiving field is floating-point, which makes

No, Jerry, the same issue obtains for standard COBOL when multiple
destinations have wildly disparate scale factors.  It might -- *might*  -- 
be reasonable to make such an optimization if the destinations all had *the
same* scale factors and precisions; it gets really risky otherwise.

Standard COBOL requires multiple-destination COMPUTE and
multiple-destination ADD to behave as if a temporary field contains the
(single) sum that is used for all destinations.

That isn't the case for single-destination ADD or single-destination
COMPUTE.  If the compiler is able to deduce for the single-destination case
that certain portions of the argument values are not reflected in the
destination, the implementor is not precluded from choosing to truncate
high-order digits in the arguments of an expression to ensure a more precise
result; the rules for multiple-destination variants of these statements
*require* the production of a single result applied to all the destinations.

A compiler that sacrifices speed for accuracy and precision in cases like
this is not necessarily a "toy compiler".   An implementor that chooses not
to "optimize" four COMPUTEs with the same expression into four different
destinations into a multiple-destination COMPUTE with a single temporary
expression value may not be doing so out of ignorance or laziness, but
rather out of a desire to give his customers the most precise value
available in the particular context of argument and destination
characteristics.  .

> For this compiler, true. Put parens around A+B+C and the optimization
> latches in.

*For this compiler*.  The rules for parentheses don't even *suggest* the
production of a common temporary in this case.

> Which illustrates the point about the programmer conforming his
> intentions to the actions of the compiler - not vice-versa.

It's a good idea for the programmer to know, first and foremost, which
approach is most likely to produce the *most accurate* answer, and why, for
a given compiler.  The *standard* requires the production of a single result
to be applicable to all destinations of a multiple-destination ADD or
COMPUTE; in the single-destination case the implementor is free to tailor
the expression evaluation process to match the characteristics of the
destination, because there's only one destination to contend with.  There
*may* be multiple-destination cases in which this is also possible, but it
cannot be said that it is possible for *all* multiple-destination cases, and
that's true regardless of whether any or all of the destinations are
floating-point.
