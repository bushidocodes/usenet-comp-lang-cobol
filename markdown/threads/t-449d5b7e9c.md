Jimmy:

sp2 supported the Motif interface on Linux for quite some time, but
very few people wanted it, therefore customer demand tends to dictate
which of our versions get attentiona dn which do not.

I believe that Micro Focus dropped support for their Motif version of
Dialog System years ago.  I don't think that Acu supports Motif with
their GUI extensions.  I'm also quite sure that none of the compiler
companies support Motif with their GUI solution.  Perhaps my friends
at the various compiler companies can verify this or refute my
statements if I am incorrect.

You probably don't want to write the low level X-Window routines
because:

A.  They are a pain in the rear.
B.  There are some GUI features in Windows which are not supported by
Motif...parent/child windows and combo boxes to name two of them.
This means if you wanted to maintain 1 set of code for both, you would
have to do a lot of extra programming to accomodate GUI screens which
have a "one for one" relationship in both Motif and Windows.

There is a possible solution for you.  I'm not sure if it is a good
solution or not, but there is a Linux "user supported" emulator called
WINE which allows Windows programs to run in Linux Motif.

If you go to the official WINE web site, you will see that there are
quite a few Windows applications which have been certified to run in
the WINE environment.

More information on the WINE emulator can be found at:

WINE Headquarters

http://www.winehq.com/

WINE FAQ

ftp://rtfm.mit.edu/pub/usenet-by-hierarchy/comp/os/linux/answers/windows-emulation/wine-faq

There is also a USENET Newsgroup for WINE at:

comp.emulators.ms-windows.wine

Hope this is helpful.

James,

I'm not sure if you're interested in writing native Java, or Cobol that
does GUI via a Java toolkit of somekind?

Java Runtimes are now coming out (especially from IBM, try
www.ibm.com/java) that feature JIT (just in time) compilers.  These compile
Java bytecode into native machine code at runtime, giving a (sometimes)
huge performance boost over the Sun's basic JRE.  Still, Java performance
is considerably slower than code compiled with a good optimized Cobol
compiler--at least with the Java apps that I've seen (take a look at DB2
6.1).  Whether that's because of JRE performance, or generally inefficient
code, I don't know.

I must say, however, that the portability of Java is fantastic.  DB2 6.1 is
a great example of this as well.  IBM has rolled out DB2 6.1 across several
platforms almost simultaneously, and I'm sure that the Java front end is
largely responsible for this possibility.
