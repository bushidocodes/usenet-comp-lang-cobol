When you set an intermediate field to an index, the value in the field will
be that of the ordinal occurance position in the table referenced.

If you set the index for table 1 to the index for table 2, the occurance
numbers after the set will be identical.

COBOL stores the index as a byte offest into the referenced table but any
external reference to the value is given the occurance number.

The preceding is true for IBM COBOLs (and I think some others).  HOWEVER,
there is no requirement in the COBOL Standard that it work this way.  It is
true that you can set an integer data-item to an index and then set another
index to that integer-data item to get the same ordinal number.  It is *not*
true, however, that indexes are always stored as offsets or that you will get
"usable" values if you set an index from one table to an index in another
table.  This is most likely to be a problem if the tables have different size
elements, but it MAY give you bad results - even if the elements are the same
size.
