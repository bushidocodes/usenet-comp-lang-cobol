to look at every byte in order to know whether to remove it. (Even if you
COULD use INSPECT or UNSTRING they will still generate byte by byte

(NO correspondence regarding style, flag values, use of sections, or any

Working-Storage Section.

01   input-string.
        12 is-char     pic x occurs 256
                                        indexed by is-x1.

01   output-string.
        12 os-char     pic x occurs 256
                                         indexed by os-x1.

*> if your compiler supports 78 levels you might like to set the length of
both the
*> input and output strings up as 78 and include these in the occurs. I'm
*> continuing on the assumption that you HAVEN'T got this facility...

01  binary-fields usage comp-5.
*> Manually amend these values to match your OCCURS...
*> They DON'T have to be the same...
       12 in-length   pic s9(4) value 256.
       12 out-length pic s9(4) value 256.

01 flags.
      12 finish-flag pic x value space.
            88 finished                  value '1'.
            88 NOT-finished         value '0'.

Procedure Division.
...
remove-spaces section.
*>
*> This code scans the input string and leaves it in the output string
*>  with all spaces removed from it, apart from trailing ones.
*>  On exit, the index os-x1 will point to the last character in the output
string
*>  BEFORE trailing spaces (if any).
*>
rs000.
        set NOT-finished to TRUE
        set os-x1 to 1
        move spaces to output-string
        perform
              varying is-x1
                   from 1
                      by 1
                    until  is-x1 > in-length
                                    OR
                             finished
                          if is-char (is-x1) NOT = space
                             move is-char (is-x1) to os-char (os-x1)
                             set os-x1 up by 1
                              if os-x1 > out-length
                                  set finished to TRUE
                              end-if
                          end-if
         end-perform
         set os-x1 down by 1
         .
rs999.
          exit.

IF the input string was VERY long (more than 16K) you MIGHT need to amend
the above code to detect the last non-blank character in it and set the
position of this character into the input string length. There is no easy
and efficient way  (I can think of) to do this on the PC, but the mainframe
has storage instructions which can do it efficiently. For most normal
purposes it shouldn't be necessary (and besides, PCs are only "toy" machines
anyway, right Mainframers <G>?)

Hope this helps,

Thane,

I completely endorse the separation of this code to a separate module or
Object Method, and I should have suggested such in my post.

Not sure that I like "SquishPete" as a name for it...you could be giving
certain people ideas...<G>

As to re-inserting removed spaces, as a string function, I contend that this
cannot be done without storing another string.

It is therefore not really viable.

Every string that you removed spaces from, would require another string to
save the information as to where the spaces go (in fact, it would probably
make better sense to provide an "undo" function and simply save the original
string in the Method which did the removal...) This would allow undo until
the next string had spaces removed from it...

Logically, when you remove the spaces from the string you are also removing
"information" and it is therefore not possible to re-construct the original
string, just using the information now contained in it.

Even if you went to all the trouble of saving the original strings on a
database you would still need to connect them to the "converted" string
somehow, yet without adding information to the converted string.

The only way I can see to do this would be to build a database of string
"pairs" with both the spaced and de-spaced strings connected together and
the de-spaced string used as a key....A HUGE amount of trouble for a dubious
benefit, hence my conclusion: not viable.

An interesting logical exercise though...<G>

Pete.

I really don't want to get into a PC versus Mainframe war (I've upset D.
Buck enough already...) because they both have their place. I haven't seen
recent performance (Million Instructions Per Second (MIPS) type data - which
is often misleading anyway) for the current mainframes.

It's important to realise that (despite what your local PC salesman tells
you) processor power is NOT the be all and end all...especially when it
comes to "data processing" (as opposed to "computing").

Mainframes excel at data processing because the channel architectures (like
Block MUX) are light years ahead of the the PC integrated bus architecture
(both platforms can utilise Direct Memory Access (DMA) buses, but actual
data transfer from peripherals on the mainframe is carried out by a
subsystem which the PC is yet to emulate.)

Irrespective of processor power, this gives the mainframe a decided edge
when it comes to shovelling data around.

However, the picture is different when it comes to actual Arithmetic Logic
Unit (ALU - the essential heart of the CPU) cycle time and instruction
execution. Both platforms are much closer  in this department.

30 years ago we thought a mainframe capable of around .3 MIPS (IBM 360-30...
it took 1.5 microseconds to access 1 byte in hand threaded memory, and you
needed at least two bytes before you could really do anything) was pretty
good stuff. I remember watching the CDC Cyber 70 when I was first introduced
to it, with awe. It was designed by the legendary Seymour Cray and he
eschewed the then fashionable integrated circuitry for good old fashioned
(Transistor-Transistor Logic (TTL)). Result: a machine that was capable of
nearly 4 MIPS, but ran so hot you could heat the building with it and it
required water cooling...
(part of this huge increase in performance was obtained by utilising a
different software architecture as well as hardware: the Cyber used a 64 bit
"word" with 6 bit characters; I can't remember what the spare four bits
did...<G>)

Today we have memory chips that can provide a byte in under 100 nanoseconds
(and have saved the eyesight of thousands of Koreans who used to "knit"
little ferrite doughnuts onto wire lattices with a microscope) so the actual
cycle time for the ALU has decreased dramatically.

I have seen claims for P4 of around 20 MIPS. I don't know whether this is
true or not, or who made the calculations, or on what basis. As I said
above, I have seen no recent claims for mainframe MIPS (maybe someone
reading this who is on a mainframe site could ask the engineers and post
it?)

While these kind of figures are interesting as a measure of the dramatic
increase in power in hardware over time, it would be very wrong to place
credence in them for determining throughput or comparing performance.

The bottom line is that modern processors (both PC and mainframe) are
unimaginably FAST ...!!!

(Of course, we manage to slow them down by writing crap code...<G>)
