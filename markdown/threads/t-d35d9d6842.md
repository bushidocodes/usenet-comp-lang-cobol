Have you thought of the folloing scenario. The string that is used for
the display takes up some space in the area where the strings are saved.
If you have a fault in your program where a part of that area is overwritten
the bug will not show when the area is overwritten contains the display string.
When not using the display string, something else will be overwritten.....therefor
a bug.
Without knowing the working storage or the bug in your report, that's the
best guess I can give you.
the way to check is simple, just add 01 xxx pic x(512) at the start and end
of your ws and see what happens (without any displays).

What is the nature of this bug? Knowing this might help think of

By COBOL 2 release 4, are you referring to OS/VS COBOL release 4? If so, I
don't think that release of the compiler is supported any more.

Have you tried using the 'COBOL II' or 'IBM COBOL/370' products? They
manage storage quite differently.

Have you tried turning off optimization?

Do you have working storage tables in your code? Check for table overflows.


What about called programs that you pass parameters to. Are those
parameters largre enough?

Does your SQL perform 'SELECT * INTO :WS-GROUP-ITEM' type of operations? If
so if WS-GROUP-ITEM is not large enough you could encounter storage
overlays. Try 'SELECT A,B,C INTO :WS-A, :WS-B, :WS-C' instead.

I haven't worked with release 4, so this may not apply...

Yes, the DISPLAY can impact addressing. I have occassionally run
across similar problems. I have solved them by inserting a paragraph
name at the appropriate place (perhaps the beginning of the PROCEDURE
DIVISION is sufficient.)

You can investigate further with an assembler translation of the
PROCEDURE DIVISION. Even if you aren't very good with assembler, you
can notice changes in the code generated. (I have used this technique
on other wierd, obscure bugs too.)

My recommendation - try inserting the dummy paragraph name & if it
works, forget it.

Celeste
