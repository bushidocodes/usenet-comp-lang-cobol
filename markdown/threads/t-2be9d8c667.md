shyam nivas writes ...

>Hi,
>             As we know on V2R10 version of OS/390 support for DLL
>call from COBOL is not available.

You are wrong. It is not the level of the operating system here, but the level
of your COBOL compiler. If you use COBOL for OS/390 & VM (2.1) or later, COBOL
programs can call DLLs.



>We also know that compiling a source
>code of C with DLL option makes it a DLL 

You are wrong. Compiling a C source with the DLL option tells the compiler to
make all dynamic calls DLL-style calls.


>and C++ code on OS/390 is
>always DLL.Based on these facts following are our doubts.
>              1. If C++ code on OS/390 is DLL then how is it
>executable?

Again, C++ code is just a program; it just happens that all its dynamic calls
use DLL linkages instead of traditional MVS dynamic linkages. Just run the
program.


>              2. we tried calling a C program dynamically from
>COBOL.The C program is using a DLL and compiled with DLL option(we
…[3 more quoted lines elided]…
>call is successful?

Again, the C program does not have to be compiled with the DLL option, as long
as you compile with option EXPORTALL or (Much Better) include a

       #pragma export(name)

statement for each function and variable you are exposing. Also, the C DLL
should include a 

      #pragma(function_name,FETCHABLE)

for the name of the C function itself. Be sure to compile with RENT and to bind
with RENT, DYNAM(DLL), AMODE=31, RMODE=ANY, and UPCASE(YES); also, point your
SYSLMOD DD statement to a PDSE and your SYSDEFSD DD statement to a member of a
PDS or PDSE. [Note: SYSLMOD can be a PDS in some cases, but PDSEs are supported
for all cases.]

(note, you will need to compile with DLL option if your C program in turn makes
dynamic calls to lower level functions).

Now, for your COBOL calling program, first be sure you are using a current
COBOL compiler.

Compile your calling program with these options: DLL, RENT, NOEXPORTALL,
NODYNAM, and possibly PGMNAME(LONGMIXED).

Your calls to DLL functions must be of the type "call identifier", such as:

          move 'dll_function_name' to pgm-name
          call pgm-name using arg1, arg2, ...

where "pgm-name" is defined as a character string in working-storage with
length long enough to contain the name of the function you are calling.

When you bind (link edit) your COBOL program, you need to include this
statement:

      IMPORT CODE,dll_member_name,'dll_function_name'

alternatively, you can supply a DD statement, say SYSDEFSD, that points to the
library used in your compile and bind of the C DLL, and then pass this
statement into the binder:

     INCLUDE SYSDEFSD(dll_name)

this will bring in the IMPORT statements generated at bind time by your bind of
the C DLL.

Hi, Shyam,

Hmm. It seems you've changed the problem slightly, but it should be solvable
anyway.

Just as a first comment, you no longer need the prelinker (at least if your
stuff goes to PDSEs) since the binder has been updated to do all the prelinker
stuff. Also, you will need to use binder option COMPAT(PM3) if that is not your
default.


1. In your DLL source, you must include

         #pragma linkage(badd,FETCHABLE)
         #pragma linkage(bmultiply,FETCHABLE)
         #pragma linkage(bdivide,FETCHABLE)

(I made an error in format in my earlier note); you do not need to compile this
with EXPORTALL, since you have #pragma exports for your variable and function
names, and you don't need the option DLL since this is a leaf routine and does
not do any calling itself.



2. In your C code, you will need 

          #pragma linkage(test,FETCHABLE)

Ahh, there it is: your C program cannot be a main!

A DLL is a collection of subroutines. In the LE world, you may only have one
main program in an enclave, and that role is played by your COBOL routine.

It looks like you have a C program to test your DLL and a COBOL program to test
your DLL; but your COBOL program cannot call your C test program: it's main
calling main.



3. Also, in your COBOL program you are calling DLLFINAL - you can't call a DLL:
you can only call functions in the DLL.



So the main thing seems to be you need to rethink your program structure.

Shyam,

Well, I took your code and uploaded it to a mainframe running OS/390 2.10, and
here is what I found:

You need to put each C DLL function in a separate source module (so, BADD,
BMULTI, and BDIVIDE each contained just that function).

Each of these has to have a #pragma linkage(funct_name,FETCHABLE). Compile and
link each module.

Then build your DLL using the program binder using this input:

   INCLUDE SYSLMOD(BADD,BMULTI,BDIVIDE)
   INCLUDE SYSDEFSD(BADD,BMULTI,BDIVIDE)
   NAME MDLL(R)

this bind step should have at least RENT, DYNAM(DLL), AMODE=31, and RMODE=ANY

Compile and bind COBOL1 as you have it, and in the link step include:

//LKED.SYSIN  DD  *
   INCLUDE SYSDEFSD(MDLL)

This works like a champ.

I could have sworn I've put multiple functions in a C source that I built into
a DLL and then invoked from a COBOL program, but I couldn't get it to work. I
have accessed exported variables from C DLLs from COBOL code (in a variety of
ways), and I have included multiple functions in a C source module that I built
into a DLL and was able to access all the functions from a C program.
