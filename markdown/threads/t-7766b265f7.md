Look at the names and the tasks. COBOL was designed to be the COmmon
Business-Oriented Language, just as a torque-wrench is designed to tighten
a bolt with a particular amount of rotating force.

One can use a torque-wrench as a tack-hammer, certainly... but the task
might be best accomplished with the tool designed for it.

If your compiler has the option to do so I'd suggest that you compile a
simple COBOL program and generate a listing of the Assembley code which is
the equivalent of the machine-language statements the compiler produces.
(the options I know of that do this are PMAP and LIST... but those are for
compilers IKFCBL00 and IGYCRCTL)

Now remember... Assembley language was developed to replace machine
language and the ASM statements have a one-to-one correspondance with the
machine language. Truly time-sensitive devices are still (I believe)
coded 'to the metal', in machine language, because it is quick to execute
and because the machine language can take advantage of certain features
unique to the hardware (machine) for which it is being written.

COBOL is a third-generation language, meaning that a statement can compile
to more than one machine language statement. That's overhead you don't
want in your pacemaker or manufacturing-robot.

What you'd like to do, I would say, is use a tool completely inappropriate
for the task you'd like to accomplish. With that in mind... everyone
needs a hobby and who's going to get hurt?

Enjoy your attempts and if anything interesting comes out of them feel
free to report on it here... and don't forget that some folks consider it
very interesting when a statement begins with 'I/You was/were wrong
because...'

There is nothing wrong with using COBOL for this except that the
language wasn't really designed for that type of activity. Common
Business Oriented Language was designed to be used in business,
finance, and administrative systems for companies and governments.

In order to do something like you want, COBOL would need an interface
(or I/O routine) to talk to the device and receive information from
it. Those don't exist for COBOL unless the device mimics something
else that COBOL does talk to. You could write your own I/O routine
and call it from COBOL but that seems like a lot of unnecessary work
when there are languages already in existence that can do what you
want.

Many years ago I worked for a company that specialized in scientific
computing and provided systems to oil refineries where valves were opened
and closed on the basis if feedback from pressure sensors. SOME of that code

A hospital in Adelaide Australia used COBOL to control laboratory equipment

quickly, but there were no problems with any of these systems.

In the network space there are many instances of COBOL reading and writing
COM ports to drive devices across the RS232 interface.

If COBOL is what you have and you have the time and energy to devote to
this, go for it.

As you pointed out, no lives will be endangered.

At the end of the day, it is a question of getting code to the right place
at the right time. If that code is not as absolutely optimum as it might be
there are times when that is important and times when it isn't.
