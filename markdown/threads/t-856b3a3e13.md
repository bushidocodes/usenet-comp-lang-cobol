OK, I don't use NetExpress. (Maybe you are startng to understand why ?<G>)

I believe that once the connection is established with ODBC, that will be
held for all SQL requests from that thread, IRRESPECTIVE of whether they are
issued by JAVA, C++, or COBOL, or anything else....

You are quite right to use whatever is best for the job, (whether it is Java
or COBOL) and you shouldn't have to worry about establising a separate
connection for JDBC. I don't believe you do, but if I'm wrong, then NE is
sadly lacking...

The COMMIT issued by your COBOL should apply to all actions against the
database which were issued from this thread. (ROLLBACK should function the
same way...).

The point is that the actions against the database should not care whether
they were issued by COBOL or anything else...

Try it and see how you get on... JDBC should recognise the established
connection. If it doesn't, then there is a problem with the way NE issues
these function calls.

JTA is not applicable to this problem (in fact, it muddies the water...).

Do NOT issue a JDBC CONNECT. (Assuming your COBOL CONNECT has returned a
successful completion code...)

Treat the Java code as if it were encapsulated within the COBOL code. I
believe this will work.

Let us know how you get on.

Pete.
