the date and the time)
- LENGTH OF constructs.

› What is wrong with this ELSE IF structure?

Nothing's wrong with it (which was of course your point). However,
EVALUATE offers at least a couple of advantages:

1. It makes it clear that all the tests concern the same variable,
or the same pair of variables if you use an ALSO clause, and so forth.

2. Suppose you don't want to end the case structure with a period.
For example, it may already be conditional on a previous IF, and you
may want to follow it with some other action no matter which branch
is chosen. With EVALUATE you can code:

EVALUATE TRAN-CODE
...blah blah blah
END-EVALUATE
PERFORM WRITE-SUMMARY-RECORD.

With ELSE IF your options are uglier. You can duplicate the PERFORM
in every branch of the case structure; you can code a long series of
END-IFs, and hope that you counted write; you can isolate the case
structure and the following PERFORM in a separate paragraph. This
last option might not be a bad idea, if the new paragraph forms a
nice modular unit, but you shouldn't have to create a new paragraph
solely for reasons of syntax.

Naturally EVALUATE can be abused. EVALUATE TRUE, in particular, can
be used as a disguised ELSE-IF.

Stylistically, the use of EVALUATE implies (to me at least) that the
various branches are mutually exclusive, so that you could code them in
any arbitrary order. If any rearrangement of the branches would lead
to a different result, then you shouldn't use EVALUATE TRUE. A
maintenance programmer might rearrange an EVALUATE where he would think
twice about rearranging an ELSE IF.
