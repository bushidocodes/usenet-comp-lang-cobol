Everything is behaving as it should.

Pic 9(4)v99 Value 1234.56.

when examined in memory will contain 123456.

The reason your example is truncated is because the compiler has to choose where
to align the implied decimal.  It uses the actual decimal from your VALUE clause
-- which is at the end -- So the result is just the same as if you had moved
123456 to this item -- it will contail 3456.

The various points made are valid.

Take both your fields :-
3  01    ws-num-field            pic 9999v99    value 123456.
                                            REAL VALUE = 3456.00.
4  01    ws-edited-field         pic 9(4).99.    value 123456.
                                            REAL VALUE = 3456.00.

That's a very basic thing about COBOL when MOVING,  the receiving field must
be big enough to take the number, plus the sending number must also contain a
decimal point/implied decimal point, when appropriate.(Also don't forget about
negative signs).

move 1,234,567,890      to num3     (pic 9(03).)          = 890
move 1,234,567,890.12 to num3d2 (pic 9(03)v9(02)  = 890.12

Go back and read your notes or manuals on MOVING numbers.

This is a subtle point - but the "MOVE" rules do not always apply to "VALUE"
clauses.  Reading about "MOVE" in relation to "VALUE" won't always lead to
getting the right answer.

Case in point:

01  Some-Edited-Field Pic 999,999.99  Value 12345678.

This gives you a field that contains "12345678  " and not
"123,456.78" as you would expect if you applied the things you read about"MOVE".

In order to get the value you expect you have to code:

01  Some-Edited-Field Pic 999,999.99 Value "123,456.78".

While in the present situation - reading about truncation as related to MOVE
gives the same result, there are actually separate rules for MOVE and VALUE.

Thane - my friend,
  Can we say "nonconforming source code" (aka extensions to the '85
Standard) ???

1) Page VI-48, SR(3) says that the value clause *must not* cause truncation.
Therefore,

      01 ws-num-field   pic 9999v99  123456.
          is NON-conforming (although I suspect many/most compilers accept
it and DO truncate)

2) Page VI-48 GR(1b) says that a numeric-edited field must have an
alphanumeric literal in its VALUE clause.  Therefore,
       01  Some-Edited-Field Pic 999,999.99  Value 12345678.
            is NON-conforming (and I think that only a LIMITED number of
compilers accept it)

        ***

In both cases, if your compiler does accept it, then they should provide a
DOCUMENTED extension telling what they do with the results - and results are
(at least theoretically) non-portable.  Also any ANSI/ISO/FIPS flagging
should detect the syntax "extension".

NOTE: The 2nd "item" has (correctly - IMHO) been changed to a syntax rule in
the FCD - so flagging will be required there too.  It took me a while to
confirm that it is correct in the FCD, because the '85 Standard talks about
the category of the data item while the FCD talks about its class.  However,
I think the "results" are the same/compatible.

Hi,

If you define a field 9(4)V99 you are defining a field with a maximum value
of 9999.99. Moving a higher value, in your case 123456 will chop off the
first two positions.
If you use value 1234.56 you ARE using a strictly numeric value and the data
in ws-num-field is strictly numeric (123456 and the compiler assuming the
last two positions to be decimals)..
The VALUE clause is the same as using a MOVE statement
