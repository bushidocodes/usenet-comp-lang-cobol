Bugger, I thought you might ask that.

These are difficult concepts to get across and it is actually hard to
determine which is which.  It is kind of like the OO shift in
paradigm, once you get it, you have, until then you have not.

Data coupling (bad):

	move 1 to perform-type
	perform something

something.
	if perform-type is 1
		...
	else

Generally when you see the above it is a warning that you are really
trying to do two different things in the one function.  The first
routine 'knows' something of the internals of the second.  This is
data coupling.

External coupling (bad, common in cobol):

perform abc.

In this instance you don't know what abc affects without inspecting
the code contained in abc.

call abc using xyz,   qrs.

We know that only xyz and qrs are modified.  If we are not interested
in those variables we need look no further.

Stamp coupling (bad, common in old-bol):

perform calculate using entire-output-record.

This was done in one program I maintained.  It turned out that it used
two fields and updated a third based upon a look up and calculation. 
Code should have been more explicit, I had to review a lot of code in
this routine to determine what actually was affected (several hundred
code lines).

Common coupling:

This is where you store general stuff into a bin with no common
features. You should group customer info so that you can initialize
all customer stuff in one hit.   Can't give you a sample when this
caused me problems.

Content coupling:

Entry points & go to, I simply ban these outright.  There are some
good uses but they are too easily abused.


Good coupling:

call mailcentre-lookup using postcode,  mail-centre.

Version 1 of this routine used a VSAM file on look up.
        2 had a DB2 look up
        3 had a sequential file loaded into memory on the first call.

It is loosely coupled because:

1.  It can (and is) used in many programs.
2.  The underlying implementation can be changed without modifying the
    calling program.


By the way,  the main benefit of OO is that far easier for cohesion
and coupling to be done correctly in the design of OO systems. 
Classes are generally concerned with one thing, actions are related
and singular, etc.

Do you want me to try and explain cohesion or has this mangled
explanation confused you enough?
