Your points well taken. Don't want to blow the horn, but IMHO opinion
NetExpress is a great product - but with serious shortcomings when it
comes to GUI design. So to answer your questions :-

- Yes the class libraries are proprietary - and that concerns me - if
each compiler vendor writes different sorts of classes, different
method-names to do the same thing - then we the developers are screwed -
we wont be able to move from compiler to compiler with reusable code. I
feel VERY STRONGLY that this is something the COBOL Standards should
address before COBOL-XXX is given the green light. 

- Dialog System only applies to NetExpress - no good even thinking of
using it with Fujitsu or Liant(Ryan McFarland) - similarly no point in
trying to use their 'window builders' with NetExpress. (There are
generic GUI builders out there like SP2 from Flexus - that's before Bob
Wolfe makes his pitch <G>)

- There are no books available on N/E class usage, in the context you
are asking - building windows - if there was such a tome I would be
reading it avidly. And this is the area where N/E has its shortcomings.
Don't get me wrong, they have totally embraced OO as a tool, whereas
other compilers, as a generalization, for the moment have assiduously
avoided the full OO issue - I think based on the failure of Visoc
(precursor to N/E) - the others are playing a wait-and-see game.

It's ironic they (Merant) build a great product, with a tremendous
amount of pizazz and a lot of OO technique built into the IDE that you
stare at each time you load their development package, but only give you
fairly basic examples. It has been a long learning-curve to get a handle
on just some of the GUI class information. (It is rather like buying
that glossy new auto with the crisp driver's manual - but you have to go
to one of the auto/book-store chains to get a manual giving you the full
McCoy - as yet for N/E GUI classes you can't go to that bookstore).

I haven't done any work yet designing a progress-bar, but over the
weekend, I'll hash something together showing how I build what I call an
InfoBox, (displays records instead of the progress-bar). How it is
constructed might help you to do a progress bar and the dialog you want
for general messaging.  If when you say 'general messaging' you mean the
standard Windows messagebox with 'OK, Retry, Cancel' - that's dead easy
and I'll include some examples. (When Simon Hart reads this, he may also
offer some help).

PS: You have to use the Progress-Bar class, as it isn't possible in 
Dialog System, but I would have thought you could build your general
messagebox from within Dialog System. (I don't use D/S).

Jimmy, Calgary AB

Bill,

Using a progress bar is good for an application when procesing lots of
data.
So I use this object quite often.  The example im about to give is a
progress loaded
into a basic dialog box which pops up with no caption to show the user that
data is processing.

Most recently i have added the progress to the status bar, then when needed
i have displayed it
then processed it as below, only problem with this is you have to code
where to put it,
but i will give a simple example here.

       copy "guibase.if".



       Class-Id.   ProgramProgress
                   data is private
                   inherits from Modal.

       Object Section.

       Class-Control.
           Modal is class "modal"
           ProgramProgress is class "progind"
           .

       Working-Storage Section.

       copy "logevnt.cpy".
      *--> Copy files
       copy "mfres.cpy".    *> Symbol nos to identify screen controls
       copy "p2cevent.cpy". *> 78s used to identify physical events

       78  wsNullTerminator        value x'00'.
       78  wsNewLine               value x'0D0A'.


       Object.

       Object-Storage Section.
       01 oProgressIND          usage object reference.

       method-id. "create".
       local-storage section.
       01 lsResourceId                 pic x(4) comp-5.
       01 lnkStep     pic 9(9) comp-5.
       procedure division.
           invoke super "create"

           move ID-PROGINDICATOR to lsResourceID
           invoke self "GetObjectFromId" using lsResourceID
                                       returning oProgressIND
           move zeros to lnkstep
           move 100 to lnkstep
           invoke oProgressIND "setStepSize"

           invoke oProgressIND "step"


       Exit Method.
       End Method "Create".
       

 This method is invoked from the calling class.

       method-id. "killProgress".

       procedure division.

           Invoke oProgressIND "hide"
           Invoke oProgressIND "finalize" returning oProgressIND
       exit method.
       end method "killProgress".

This method is invoked from the calling class
Note: you can set up the step value using method "SetStepSize"
once you have calculated how much data needs to be processed.

       method-id. "stepprog".

       local-storage section.
       01 lsResourceId                 pic x(4) comp-5.
       01 lnkStep     pic 9(9) comp-5.


       procedure division.


          invoke oProgressIND "step"

       exit method.
       end method "stepprog".

       Method-Id. "Show".
       working-storage section.

       78  ws-X         value 400 .
       78  ws-Y         value 200.

       Local-Storage Section.
       01 aNewProgDialogBox     object reference.
       01 ResourceId               pic x(04) comp-5.
       Linkage Section.

       01 lsX                      pic x(4) comp-5.
       01 lsY                      pic x(04) comp-5.


       Procedure Division using lsX, lsY.

           compute lsX = (lsX - ws-X) / 2
           compute lsY = (lsY - ws-Y) / 2

           invoke self "setxy" using lsX lsY

ShowNoFocus is used here so that the program
will continue to execute.

           invoke super "showNoFocus"

       Exit Method.
       End Method "Show".

       End Object.

       End Class ProgramProgress.

If you would like an example of the status bar then mail me(little more
complex)

Message Boxes are very common, this is nice all we have to do is invoke the
class
with a few parameters of course.

The example i will show is the most complex you will ever need.  It is
someone trying
to write to a network server when the server has gone down, so the
retry/cancel
msg class is used:.

Note: The reference MessageBox is class "msgbox"

             Perform Until ws-EOL = 'y'
             Move zeros to status-code
             call "CBL_CHECK_FILE_EXIST" using totalPath
                                               file-details
                                               returning status-code

             If status-code = 14625 or status-code = 14601
             *>cant read from device
                 invoke MessageBox "new" using self
                                           returning lsMessageBox
                 invoke lsMessageBox "setTitleZ" using wsHardWare-Fault
                 invoke lsMessageBox "RetryCancel"
                 string 'Cannot read from device - ' lnkserver-path
                      wsNewLine
                      'Please contact Administrator on Ext. 6561'
                      wsnullterminator
                      delimited by size
                      into wsTotalMSG
                 end-string

                 Invoke lsMessageBox "SetTypeCritical"
                 Invoke lsMessageBox "setUnmoveable"

A beep sound :)
                 Invoke Alarm "warning"

                 invoke lsMessageBox "setMessageZ"
                                        using wsTotalMSG
                 invoke lsMessageBox "show" returning lsId
                 invoke lsMessageBox "finalize" returning lsMessageBox

                 If  lsId = mb-return-retry
                   *>lets try to access the server again:.

                     Move 0 to lnk-HardWare-Code
                 Else
                   *>cancel was choosen
                     Move 'y' to ws-EOL
                     Move 1 to lnk-HardWare-Code

                 End-If

                 Move spaces to wsTotalMSG

             else

             Move 'y' to ws-EOL
             If status-code = zeros *>file already exists.

      *      If File-Size <> zeros *>file already exists
                 invoke MessageBox "new" using self
                                           returning lsMessageBox
                 invoke lsMessageBox "setTitleZ" using wsOverWrite
                 invoke lsMessageBox "yesno"
                 string 'Overwrite file? - '
                      totalPath
      *               delimited by x"00"
                      into wsTotalMSG
                 end-string


The below settypequestion puts the little question
bitmap into the msgbox

                 Invoke lsMessageBox "SetTypeQuestion"
                 invoke lsMessageBox "setMessageZ"
                                        using wsTotalMSG

                 Invoke Alarm "warning"
                 invoke lsMessageBox "show" returning lsId
                 invoke lsMessageBox "finalize" returning lsMessageBox
                 *>test outcome
                 If lsId = mb-return-yes
                    Move 0 to lnk-Filecode
                 Else
                    Move 1 to lnk-Filecode
                 end-if
                 Move 'y' to ws-EOL
             end-if
             end-if
             end-perform
