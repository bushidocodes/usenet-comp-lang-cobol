I noticed that you're creating a semicolon delimited file, possibly
suited to being read by a spreadsheet.    In that case the output
number of PIC 999V99 will be read in by the spreadsheet software as
PIC 99999.    Unless the program that reads your file knows where to
enter the decimal point, it won't.    Spreadsheets read in numbers
with real decimal points    PIC 999.99.

Also, when I write such programs, I am using TSO/SPF as my editor, and
the semicolon is a delimiter, so I use:
03  SEMICOLON                 PIC X         VALUE X'5E'. 


=========

Usually when someone wonders why their data using PIC 99V99 is wrong,
it is because their expectations were wrong.

The best way of determining what is actually happening is to look,
either using a debugger, or displays.

This is how you can see what is happening:

MOVE 450 TO EHIS-TEMPS
MOVE EHIS-TEMPS TO DISPLAY-1.
DISPLAY 'EHIS-TEMPS =' EHIS-TEMPS ', FORMATTED=' DISPLAY-1.

(this should display, depending on the picture of EHIS-TEMPS)
EHIS-TEMPS=4590, FORMATTED=       450.00

DIVIDE EHIS-TEMPS BY 100 GIVING WS-DIVQOTE.
MOVE WS-DIVQOTE TO DISPLAY-1.
DISPLAY 'WS-DIVQOTE=' WS-DIVQOTE ', FORMATTED=' DISPLAY-1

(this should display the following:)
WS-DIVQOTE=00045, FORMATTED=     .45

Do the same thing at
MOVE WS-DIVQOTE    TO TTXSEQ-QTE.
MOVE TTXSEQ-QTE     TO DISPLAY-1.
DISPLAY 'TTXSEQ-QTE=' TTXSEQ-QTE ', FORMATTED='  DISPLAY-1

which should display:
TTXSEQ-QTE=00045, FORMATTED=     .45

The problem is not the decimal place, but the display.  A "V" is an
*implied* decimal place.  That is, it tells the compile to use that
point for allignment with arithmetic, and moves.  However, there is
really nothing in the data ... only in the compile time symbol table for
that name.  So displaying a 9V9 field shows the exact same thing as
displaying a 99 field.  Two digits, no decimal point.

Move 1 to a pic 9v9 results in "10" in the field. Move 1 to a pic 99
field results in a "01" in the field.
