Without specifying the GOBACK, the compiler "used to" automatically generate
a STOP RUN (IIRC, it was in OS/VS COBOL?), which if reached by the program
logic, had the potential of terminating the region.

I don't believe it's necessary anymore. However, out of force of habit, it's
included by many seasoned CICS programmers.

FWIW, a side note. If a CICS/COBOL program is Dynamically Called and the
logic hits a CICS RETURN, the task is terminated (not a good thing) or you
wind up somewhere in "ZORGON" (a fictitious unknown area of the task/run
unit, sometimes referred to as "I have no friggin idea"). Instead, to
gracefully (and correctly) return to the Caller, the logic must hit a
GOBACK.

HTH....

Some compilers squawk bloody murder if you don't have either a STOP RUN or a
GOBACK statement in a program.  Putting a STOP RUN in a CICS application
would probably get you canned because (I think) it would terminate the
entire CICS partition if for some reason it accidentally got executed by
mistake.  A GOBACK probably would just get the application tossed for
violating something or might not cause a problem.
