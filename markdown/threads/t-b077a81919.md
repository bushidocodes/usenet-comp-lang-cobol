Writing graphical interfaces using native coding is a daunting,
tedious and time-consuming task. If time is important, you should
consider using a good graphical screen development package. I use
a very good product from Flexus called SpII. It is easy to use and is
designed with the Cobol programmer in mind.

Most Cobol's do not have built-in graphical screen handling, including
the screen section (I say most because I cannot be absolutely sure
that some version may be out there). For the most part, the
ACCEPT/DISPLAY semantics of these Cobol verbs were designed for
simple, text-based applications only.

Michael Dodas

If you use the 'QuickWin' mechanism then the program will become a
real Windows program but will present a screen appearance and
operation identical to the DOS version (give or take). This is
easy to do but is probably no advantage to running existing prog
in a DOS box with suitable PIF. In fact you can run several
DOS boxes but only one QuickWin version.

To make a true Windows program with multiple windows active, dialog
boxes, pulldown menus, etc you need to start by learning how
MS Windows works and then design code that will do the task. This
is unlikely to make much use of existing DOS code.

It is also possible to use something like Cobol SPII or MF Dialog
to give Windows programs with having to attack the Windows API.
