```
I thought that I should post a reply from a "usually reliable source" (at
IBM's mainframe COBOL development sight at STL) about the reports of being
able to use the DISPLAY verb in a CICS application (MVS or OS/390) when
using the LE run-time library,

"Bill,

   DISPLAY has never been supported under CICS by IBM.  It may 'work' in
some cases, but if it ever has a problem IBM will point to all of the
different manuals that explicitly say that DISPLAY is not supported under
CICS.  COBOL APG, CICS App Prog Guide, etc."

If anyone can find ANY "LE - or other" documentation indicating that it is
supported (whether it writes to the CEEMSG DDname or a TSQueue), please let
me know.
```

```
The COBOL DISPLAY allways writes to the LE/370 MSGFILE. Under CICS, this is
the transient data queue named 'CESE'.

All RPTSTG, RPTOPTS and LE-Messages are written to CESE, as well as the
CEEDUMP.

This is documented, for example in chapter '3.1.2.2  For CICS Applications' of
the 'COBOL for OS/390 & VM Version 2 Release 1 Compiler and Run-Time Migration
Guide', Document Number GC26-4764-04.

And - not IBM-supported - the DISPLAY output appears at the same location.
```

```
Let's be 100% clear, what 3.1.2.2 states is

"Under CICS, Language Environment output goes to a transient data queue
named CESE. Each record written to the file has a header that includes the
terminal id, the transaction id, date, and time. Figure 22 lists the types
of Language Environment output and location. "

So you are certainly correct that IBM documents the RPTxxx stuff and other
LE messages going to this location.  HOWEVER, this says NOTHING about
DISPLAY or other "application" information going anywhere (or being even
slighly supported).

I really, REALLY, hope that people are not using a documented as UNSUPPORTED
feature under CICS.

FYI - if you want to limit your application programmers to SUPPORTED
language, I suggest that you compile all CICS application programs with the

    WORD(CICS)

compiler option.  This will avoid any future problems with unsupported
interfaces and will help you insure that future IBM modifications don't
break your existing applications.
```
