Thanks for taking the time to respond to my question, sorry I took so long--too

The calling program performs a sequential read of a variable length file,
containing 3 types of records, received from another job. One of the record
types contains check data (and EFT, or direct deposit, data, but these
transactions receive transaction numbers from the same table--different row--in
the same manner as checks, so they are only mentioned here) which is processed
to produce an image file for printing.

The program does not know how many checks it will write in advance, and the
check numbers must be contiguous in 2 ways: within the run and spanning multiple
runs (process 1 writes numbers 1 - 900, process 2 writes checks 901 - 1000,
process 3 1001 - 1050, the next day process 1 writes checks 1051- 1800 etc.).

The calling program is designed to get the last check number used by any
previous process in its initialization paragraph by performing a call to a COBOL
subroutine that accesses a DB2 table with a field, last_trans_nmbr, that would
contain the last number used by the previous run (process 2 above would find
900).

We want the subroutine and the table to be used by more than one program, and we
don't want to insert a new row with each run, we just want to update the the row
for our system's checks (there are other rows for other systems and types of
payments, e.g. EFT) by updating last number used, by what program, and a
timestamp field.

So, in initialization the program calls the subroutine, gets the last number
used, adds +1 to it giving a working-storage field, ws-current-chk-nmbr.  Once
it has the last number used the calling program ends the call with GOBACK.  Then
it processes the input file and each time it writes a check it moves
ws-current-chk-nmbr to the check image and adds +1 to it for the next check.

After the file has been read, everything processed and all files closed, the
calling program subtracts +1 from ws-current-chk-nmbr giving ws-last-chk-nmbr.
It the moves ws-last-chk-nmbr (and 3 key field variables) to a record-include
with a format identical to the DB2 table DCLGEN and used in the Linkage-Section
of the called program for passing values, and calls the subroutine again to
update the the last_trans_nmbr column with the value of last-chk-nmbr.

In the subroutine I declared a cursor exactly as you suggested (...with hold for
select...for update of  last_trans_nmbr).  It is declared in the Data Division.

In the Procedure Division,

during the initial call the subroutine:

1. opens the cursor
2. fetches the last_trans_nmbr into a host variable (from the DCLGEN)
3. closes the cursor
4. moves the host variable field to the record-include field in the
Linkage-Section
5. ends with GOBACK

during the final call it

1.  opens the cursor
2.  updates the last_trans_nmbr
3.  closes the cursor
4.  ends with GOBACK

It seems like a nice enough design, but my thought is, and hence my original
question, that my initial GOBACK (or even close cursor) releases the table and
any lock I might have on a row.  This would allow other programs to access the
same table/row, get the last_trans_nmbr while my program is writing checks and
before it has updated last_trans_nmbr, and start their numbering processes using
the same beginning number.

Hopefully, I've explained well enough to adequately describe our problem.
Perhaps the problem is the design?

Thanks again for your time.

 Yes, the design is seriously flawed if concurrent processes are able to hit
the "control" table and pick up last_trans_numbr while a task is already
processing, exactly as you describe. You have attempted to prevent this by
locking the table row. (You are correct that you will lose this lock as soon
as you close the cursor, even if you specify WITH HOLD). The solution is not
to lock the table row, it is to re-think the design.

Supposing you had been successful in locking the row. The task attempting to
run would simply sit and wait...and wait....and wait...until the currently
executing task has completed and released the lock, or the database software
or the Operating System abends it. NOT a good solution. Although it would
achieve the desired result, the problem is that the held lock could affect
other rows (it depends on how your system is configured, normally a virtual
page is locked even if the programmer thinks it is "only" a row...)

However, from what you have described, only one program at any given moment
is "allowed" to use the number in the row  and other programs must not use
it until  it completes. It sounds as if you are batch processing these
blocks of checks (incrementing in WS is batch processing...) so why not
control it thru Job dependency? (One run can't start until another
completes). If that is not a viable option (any process can start at any
time and may need to use the row) then you could hold the lock by NOT
closing the cursor (GO BACK will not lose it, but closing the cursor will).
As mentioned above I really don't like this as a solution, but it is
feasible...with the possibility of the pending task being aborted...talk to
your systems people before you proced with this one.

To me, a better way would be to add another column to the row indicating
that it is in use. A Boolean flag would do it. You access the row, get the
number, set the "in use" flag to true and update, releasing all locks. Any
other task trying to access, sees the flag  and waits until  it is clear
(you would do a certain  (largeish) number of tries, interspersed with a 1
second wait, then notify a human... the row contains the name of the program
which has locked it , so you have no trouble controlling it. )

When the current processing completes, the program grabs the row, sets the
last used cheque number in the table,  clears the "in use" flag, updates and
exits. On its next access the pending task finds the flag is clear so away
it goes again, setting the flag and its own program ID into the table. This
involves minimum locks and is not too heavy an impact on the table
definition and the programs which access it.

Summarising how I see the options:

1. Ideally, drop the whole idea of programs doing blocks of cheques and and
let any program do any cheque, getting the highest current cheque number
from the output cheques table using the MAX function of SQL (as I described
last time). (This is a more transactional based approach and, although you
have indicated batched blocks of cheques are what is required, I would query
this with the Business and make sure it has to be that way. With this
approach there is no need for a control table at all. Of course, if the
volumes are large, then there is no reason NOT to use batch processing, but
let's accept that that is what it is, and not pretend it is going to run as
a transaction...)

2. Given that option 1 is non-viable and we must do batched blocks of
cheques:

2.1. Lock the table by NOT closing the cursor. EXIT PROGRAM should NOT lose
the lock as cursors are held within the database software, not the user
program, but check with your DBA (It's 15 years now since I was one...things
change <G>)

2.2. Use batch job dependency through JCL to control the processes.

2.3  Set an "in use" flag on the table row (this would be my personal
preference).

There are other options but I don't like any of them... Best is to review
with the Business the way cheques are produced and the volumes of them.
Maybe set up some batch runs overnight just specifically to produce batches
of cheques and then it can all be controlled by JCL.

Hope this helps,

Pete.
