No, not necessarily.  All other things being equal, I would contend a
"minimal" INPUT/OUTPUT procedure sort (that is, one that did nothing
substantive more than READ/RELEASE and RETURN/WRITE in the respective
procedures) would be less straightforward in coding, and would cause the

USING/GIVING sort.  I would vote for USING/GIVING in terms of performance
and overhead.   So, yes, you *could* write obscure, slow code, but I would
disagree with your contention that you *should*.

The only reason I have come up for having a CoBOL with both using and giving is
so that the program can pick the sort order.   Typically internal sorts are done
because they have processing that gets us the output better.

e.g.  Create a sorted file of all vendors who have each part needed for my
widget - sorted by average distance from our factory.

One way to make a sort efficient is to exclude unneeded records early.   Input
procedures sometimes do this more easily and clearly than sort parms - and
sometimes do this using criteria that cannot be done by sort parms without
multiple passes.

which?" The answer is of necessity the product of both values, so
you must ask the question in a way that involves the product of both
unknowns, then reverse the sign (truth). By including both unknowns,
you know there is an element of truth, and also that there is a lie too.
I.e. - the truth about a lie is equivalent to a lie about a truth, in a
binary solution set.
