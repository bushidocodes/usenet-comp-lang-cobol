Just thought I would share my personal opinions on these structures - starting a
new thread although much has already been stated that I agree with and others
that I do not.

FYI - for purposes of this note, I am *not* distinguishing between paragraphs
and sections.  It seems to me that the "style" discussions on that distinction
is quite distinct from what is being discussed recently.

"Traditional" (pre-85 Standard) COBOL used paragraphs as the one any only
*STANDARD* way to include "inline distinct procedures" (I'll ignore the ENTER -
not ENTRY - statement that probably was intended to do this - but I never used a
compiler that did much with it - so I can't speak from my experience to that
structure.)

During the development of the '85 Standard (and its implementation by a variety
of vendors), it was thought that "localization" of procedure *and* code was a
"good" thing for ease of understanding and (for some implementations)
performance.  Hence the creation and inclusion of NESTED PROGRAMS.  As such
programs could be "brought" in by a COPY statement and such procedures reached
by CALL IDENTIFIER (as well as CALL "literal").  This feature (at least in
theory) provided the following (theoretical?) advantages over PERFORM paragraph
logic:

 - data defined "near" where it was used (not a single "GLOBAL" cover-all
Working-Storage section)
 - localization of programming logic
 - ability to "pass data" to a routine within the same source "deck".

Personally, I haven't seen any "shops" or development projects that have WIDELY
implemented nested programs to "replace" PERFORMed paragraphs. My impression of
WHY they haven't "caught on" (widely) include (but are not limited to):

 - lack of (or at best delay in)  "good" debugging tools for them
 - hesitation to use COPY statements to bring in "large chunks" of procedural
code
 - fact that you still need to recompile the ENTIRE source program when a nested
program changes (unlike what you need to do for a (dynamically) called
separately compiled program)
 - Fact that most COBOL programmers don't see any harm in LARGE monolithic
programs (with HUGE generic Working-Storage sections)
 - General feeling that there is "little gain" for the required re-training and
change in paradigm for "large shops"

As opposed to both PERFORM statements and nested programs, ENTRY statements are
neither portable to all compilers nor to they actually work the same for all
compilers that accept them.

Furthermore,  I don't (personally) know of any compiler that documents support
for a CALL statement from within one program to an ENTRY statement in that SAME
program.  (I have only seen it as documented where one could use it to replace a
CALL statement to a SEPARATELY compiled program with an ENTRY statement).
Therefore, it seems (to me) not to do the same thing as either a PERFORM or a
NESTED PROGRAM, but rather to provide an "alias with a different entry point"
for a separately compiled program.  To me, this seems to provide NO functional
enhancement over simply having an EVALUATE statement at the beginning of a
program and from within it either doing a PERFORM or a CALL of a nested program.
In this way, one could be "guaranteed" of portable source code with identical
semantics on all conforming implementations.

NOTE:
  For those using CALL to ENTRY points in an IBM mainframe environment, there
are a variety of "gotchas" that some programmers are "hit by" because they are
relatively poorly documented.  A couple of my favorites are:

from:
  http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igy3pg20/4.1.6.3

"An intervening CANCEL for dynamic calls to the same module at differing entry
points. CANCEL causes the program to be invoked in initial state when it is
called at a new entry point. "

and of course there is always the IBM recommendation NOT to use "entry"
statements in a "structured application" as at:
  http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/igya1101/3.9.1

where they say,

  "It is possible to specify another entry point where execution  begins, using
the ENTRY label in the called program.  However, this is not  a recommended
practice in a structured program.)"

There is no RECURSIVE compiler option or "attribute". The compiler determines
you are doing recursion by seeing the program call itself.

-- quote --
Each call of a recursive routine is called an instance. Each instance needs a
different set of the data items used by the routine; so this COBOL system
enables you to have a Local-Storage Section. Every time a new instance of the
routine starts, a new, initialized copy of this section is created in memory;
this is deleted when the instance finishes. You get a run-time error message if
you try to use recursion in a program that does not contain a Local-Storage
Section." 
Program Development, chapter 3

>NOTE WELL:
>   Once you get into making the calls RECURSIVE - remember that you will
>(always/autmatically) get WS in "initial state" each time you do the call - and
>when you GoBack or EXIT program you will "pop" the state that you "pushed" when
>you did the call.

The above quotation says you get a Local-Storage Section for each instance, NOT
a new Working-Storage Section.

Someone is confusing RECURSIVE, there is no such option, with REENTRANT, which
comes in two flavors. 

REENTRANT(2) does exactly what you described. It is intended to painlessly make
any program thread-safe i.e. having two simultaneous callers. The manual says
you should change Working-Storage to Local-Storage or Thread-Local-Storage.
Experimentation shows that's not necessary; it works fine when left as
Working-Storage.

Local-Storage does not initialize with VALUEs. It doesn't issue any diagnostic;
it just doesn't do it. Thread-Local-Storage does initialize with VALUES, same as
Working-Storage. 

REENTRANT(1) makes Working-Storage 'shared' (one copy) and gives a distinct
Local-Storage for each instance of the program. This is the preferred mode for
reentrant programs, but the programmer must use synchronization measures to
simultaneous updates to Working-Storage.

-- quote --
If you specify REENTRANT"1", all compiler-generated temporary work areas are
allocated on a per-thread basis. All user data and FD file areas allocated in
the environment and data divisions are shared by all threads. It is the
responsibility of the programmer to ensure serialization to the program's data
by use of the CBL_ synchronization calls.

If you specify REENTRANT"2", all working-storage and file section data, as well
as system work areas, are allocated dynamically on the stack. This eliminates
thread contention on these areas, so that the program behaves safely if called
from multiple-threads concurrently. No program locking or unlocking is needed.
The disadvantage with this compiler directive setting is that there is no data
shared (other than any data defined as EXTERNAL) between the threads.

REENTRANT"2" is a quick and simple way of getting a program running in a
multi-threaded application, but you should aim to compile with REENTRANT"1".

A reentrant program must itself resolve all possible contention on data items in
its Working-Storage Section and File Section. One or more of the techniques for
resolving data contention must be used to accomplish this; see the next section
for details. 

Multi-threading, chapter 2

        ENTRY 'SOLVE' USING BY VALUE A B C BY REFERENCE X Y.

            IF A > B
      *     Use TEMP as a flag to indicate swapped or not.
                MOVE 1 TO TEMP
                CALL 'SWAP2' USING A B
            ELSE
                MOVE 0 TO TEMP
            END-IF

            EVALUATE A
                WHEN 0
                    DIVIDE C BY B GIVING D REMAINDER E
                    IF E = 0
                        MOVE 0 TO X
                        MOVE D TO Y
                        SET OK TO TRUE
                    ELSE
      *     No solution exists.
                        SET BAD TO TRUE
                        MOVE 0 TO X, Y
                    END-IF

                WHEN 1
                    MOVE C TO X
                    MOVE 0 TO Y
                    SET OK TO TRUE

                WHEN OTHER
      *     We must delve deeper to find a solution.
                    DIVIDE B BY A GIVING D REMAINDER F
                    DIVIDE C BY A GIVING E REMAINDER G

                    CALL 'SOLVE' USING BY VALUE A F G BY REFERENCE v Y

                    COMPUTE X = E - ( D * Y ) + v

            END-EVALUATE

            IF TEMP = 1
                CALL 'SWAP2' USING X Y
            END-IF

            EXIT PROGRAM.


      * Second level program to swap 2 variables using local temp variable.
        SWAPPER SECTION.
        ENTRY 'SWAP2' USING X Y.
            MOVE X    TO TEMP
            MOVE Y    TO X
            MOVE TEMP TO Y
            EXIT PROGRAM.

For code that I do, paragraphs or CALLs to separately compiled routines are
it. No ENTRY nor nested programs. For the record, I tend to like nested
programs for the basic reason of  "locality of reference" - not lumping
every WS field together in one place, and having clearly delineated
parameters.

So, why don't I use nested programs? First, our build process does not
support their use. We know how to fix it but it just isn't important. (Site
note - the majority of our source is still COBOL 74 and most of the
developers could care less about upgrading, even when it is simple.) Second,
if it was possible to utilize nested programs, there is the familiarity
factor for staff. Many of them only know COBOL 74 and have never dealt with
more structured languages so the concepts would have to be explained as well
as the syntax. (I know too many staff that find it preferable to duplicate
code rather than create a copybook or subroutine as it is. Why would they
embrace this?) Third, my own personal bias, is that although I like the
parameter passing, it causes me a little grief. Our general standard is that
subroutine parameters should be in a copybook to ensure that the caller and
called program have matching descriptions. I think this is a good thing.
With nested programs, do I now create copybooks for their parameters and
therefore have more source elements that need to be managed? Since the
source may all be contained in a single file, is keeping the parameter
descriptions consistent without a copybook going to be acceptable?

Perhaps this is a platform dependent issue, but what value is there in
placing a nested program into a copybook versus making it a separately
compiled program? I'm on Unisys 2200 Series hardware and all of the
executable programs are statically linked. I can't image that there is any
discernible difference in performance between the two methods.

Finally, ENTRY. I'm not even sure I'm thinking the same syntax as you on
this one. If I am, I would consider it something to stay away from in most
instances. As you mentioned, there are alternatives that I think should be
preferred. However, I have used it in a couple of special situations where
it was convenient and I felt that readability would actually be reduced by
employing a different approach.
