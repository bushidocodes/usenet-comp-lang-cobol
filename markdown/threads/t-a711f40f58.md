```
In our shop, we use parm files for such, and pass a symbolic or two so that  the
proc knows where to find the parm file.  Or you can override the proc DD with an
in-line data field.  We don't pass values in the Linkage division for main
programs unless we're just writing a one-time job such as a conversion job,
standards have us read the parm card.  Standards also say all production jobs
run procs.

You can also override the actual executable line - the JCL that you use to
compile probably does this.
```

```
Hi Jay,

In an earlier post I suggested that you put the parm info in the D/S pointed to
by SYSTIN (it s/b SYSTSIN). I'm assuming you're running a batch job under TSO.
Apparently you misunderstood my point. I'm including sample JCL taken from IBM's
DB2 Pgmmr's Guide:

//jobname JOB USER=MY DB2ID 
//GO EXEC PGM=IKJEFT01,DYNAMNBR=20 
//STEPLIB DD DSN=prefix.SDSNEXIT,DISP=SHR 
//        DD DSN=prefix.SDSNLOAD,DISP=SHR 
 
 
 
//SYSTSPRT DD SYSOUT=A 
//SYSTSIN DD * 
DSN SYSTEM (ssid) 
RUN PROG (SAMPPGM) - 
PLAN (SAMPLAN) - 
LIB (SAMPPROJ.SAMPLIB) - 
PARMS ('/D01 D02 D03') 
END 

Notice that the DSN stmt includes the parm passed to the DB2 pgm. Your pgm
should construct a LINKAGE SECT exactly as you would for a std COBOL pgm
expecting PARM data from the EXEC card in the JCL.

HTH, Jack.
```

```
This thread has gone off into a variety of directions.  The "real" answer
(requiring NO source-code programming changes - which was my understanding
of the original question) has to knowing how to work with PROCs.

Step 1 - "look at" the PROC that you are using.  (You can do this by running
a "dummy job" or by looking in a PROC library or a variety of other ways).
What you want to look for is whether the proc does or does not already have
any PARM= statements on the step to which you want to add your
"new/additional" parm.

Step 2a - If the existing PROC/step doesn't pass any parameters, then you
can "send it one" by using a PARM.procstepname= statement on your execution
of the proc.

Step 2b - if the existing PROC/step *does* pass any parameters, then you can
still use the PARM.procstepname= override on your execution JCL, but you
need to be careful about "keeping" any existing parameters by including them
in the override.

For more details, see

 http://publib.boulder.ibm.com/cgi-bin/bookmgr/BOOKS/iea2b600/16.7.3?


NOTE WELL: This "technique" is intended for existing (or even new) programs
that expect to get PARM information in the LINKAGE section of the "main"
program.  This feature is also subject to the CBLOPTS LE run-time option.
This is particularly important to understand if you are using any slashes in
a date field that is passed to your program.

For additional information on that option, see

 http://publibz.boulder.ibm.com/cgi-bin/bookmgr_OS390/BOOKS/ceea3100/2.1.9?


If you ARE designing (or even modifying) a program's source code, then it
certainly is possible/reasonable to code it so that you do NOT expect such
parms to be passed on the JCL PARM statement.  This note is based on my
understanding that your "GOAL" is to use existing COBOL source code - and an
existing catalogued PROC.
```
