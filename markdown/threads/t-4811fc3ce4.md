MVS is Multiple Virtual Storage, which is the name used for the last 10
years or so for IBM's most profitable operating system for their line of
mainframe computers using the system 360/370/390 instruction set.

MVS started as OS (operating system) in the mid to late 1960's, and went
thru several versions: OS/MFT (Multiprocessing with a Fixed number of
Tasks) and OS/MVT (Multiprocessing variable number of tasks) until they
renamed it MVS. Until recently the name was MVS/ESA or just ESA
(Enterprise System Architecture). I understand the product is being
renamed to OS/390, and is now being enhanced to run on the parallel
sysplex mainframes which support a variable number of CMOS system 390
engines. MVS/ESA supported at least 4 "engines" in the bipolar versions
of the system 390. So IBM is moving toward more parallelism in their
mainframes. OS/390 supports UNIX in an LPAR (Logical Partition), so UNIX
runs as a guest under MVS (sort of like running virtual DOS machines
under OS/2). I'm not well-versed in UNIX, but I know that MVS does not
support character stream files, and the UNIX file system is vastly
different from the IBM file system, so I wonder how resources are shared
in an MVS with UNIX environment.

JCL stands for Job Control Language. Since MVS has always been a batch
processing operating system (unlike UNIX or VM/CMS), you need Job Control
Language to run programs. JCL is a scripting language that structures
tasks into "jobs" with have "steps". Steps may be saved in stored
procedures. JCL is interpreted at runtime (it is not a compiled
language).

Job control statements are used to connect system resources (files,
devices) to the program executing in a step. This allows programs to be
re-used without recompiling them to connect them to different files,
devices, or device types. For example, a batch COBOL program can read or
write a sequential file, and the JCL is used to determine whether that
file is tape or disk. MVS allows multiple JCL jobs to run concurrently,
and they are managed in classes and prioritized. Output reports may be
spooled for later printing after a job has completed.

There are books on IBM JCL. Job steps can be coded to execute
conditionally, depending on the results of one or more previous steps.
Jobs are submitted to the system reader and may either begin executing
immediately, or wait until the scheduler releases them.

I hope this gives you a bit of background on the subject.
