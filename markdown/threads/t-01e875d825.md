```
I want to run a job that uses different datasets depending on a
parameter.  The name of the dataset will be of the form
"HLQ.parm.other qualifiers", where the only change in the Dataset name
will be the "parm".  The operator will be able to change it prior to
running the job.

My first question is where can I place this parameter?  Which leads me
to the next question, viz can the DSN be like a variable where the
name is resolved only when the job is executed?

I hope I have phrased the question in an understandable fashion.  To
give an example, I have a program that processes a payroll file.  On
different days I want to process payrolls for different locations but
I want to use the same job and supply the location name as a
parameter.
```

```
Charles,

There are a number of ways that symbolic parameters or some other type
of variable may be of use.

I know that some Batch Scheduling systems, such as OPC or whatever its
called these days, cater for variables and a word with your friendly
Operations team may assist in this area.

Beyond that JCL itself caters for Symbolic Parameters which are
basically variable with a name upto 7 characters long preceeded by a
single ampersand (&), so

// DSN=HLQ.&PARM..OTHER.QUALIFIERS

would be OK.

Assuming for a moment that catalogue procedures are being used, then if
the value of &PARM is more often one value rather than another, then a
default can be established on the PROC statement within the procedure.

Any default can be overridden at run time on the invoking EXEC statement
for the procedure without having to amend the procedure itself.

There are other examples of symbolic parameter use on my WEB site at:

www.kmsitltd.co.uk/jclex5s.html

I hope this is of use.

Regards - Terry
```

```
You can use the SET statement which will let you bypass the PROC.  The
syntax is fairly simple, you set a variable to a value in a "step" prior to
the one you want to execute your program in.  Then in that step you use the
```

```
If you read a parmlib member, you can put the DSN you are
after into a file:

//PARMDATA DD
UNIT=SYSDA,DSN=&&DSNDATA,SPACE=(TRK,1),DISP=(,PASS),
//            DCB=(LRECL=??,RECFM=FB,BLKSIZE=0)


Successive steps can read this and get the data set name
they need from the file and then do dynamic allocation for
the DSN. However, your shop may or may not have the level of
the COBOL compiler & Runtime that supports dynamic
allocation (which may mean you will have to go to an ALC
subroutine).

Otherwise, you are in the position of generating the JCL
from w/in the COBOL program of a one step JOB and have that
JOB write to the INTRDR.

Were you in a VSE shop you might get away with generating
the JCL on the fly...
```

```
Hi Ubie,

If entering the parm data into a dataset is acceptable to you, you can
write a CIBOL pgm that will construct job JCL and submit it to an
internal reader.

The general logic of the pgm is to read your parm data and use it to
construct a DD stmt that contains a DSN unique to that run. That and
the other JCL stmts would be written to a SYSOUT file with an INTRDR
clause. Closing that file triggers the submission of the JCL to a JES
reader and eventually executed.

Each time this pgm is executed the parm data supplied by you is used
to create the desired DSN.

Search the archives here using INTRDR as the argument, I'm sure you'll
find many hits to provide the coding details you'll need.

HTH, Jack.

P.S. Some may say this is frowned upon, but it's used every day in
CICS environments.
```
