So, you don't want to have your existing text mode screen within a GUI
screen you want your program to operate a GUI screen.

You can't do that with MF Cobol doing ACCEPT/DISPLAY or Screen SECTION.

(Actually you _can_ do that using ACCEPT/DISPLAY if you had ACCU COBOL,
but it still needs quite a bit of rewriting)

You would have to rewrite the program to use Dialog or SP/2 or Windows
API if you really want a proper GUI.  I have done these.  SP/2 is
relatively easy as you draw what the screens look like, ie where all
the buttons go, where the fields are, what the menu has.  The program
then sets up the field values that are to be displayed and CALLs "SP2"
USING Panel-Area ...  When the CALL returns there are values that tell
you what the user has done.  You can get a demo from flexus.com.

Dialog I have not used.  Windows API is complicated and needs some
months to be learnt.

You _can_ simulate a GUI using ACCEPT/DISPLAY.  The CURSOR IS .. allows
you to control the position of the text cursor and will tell you where
it was when the user returns.

The mouse can be used with ADIS as you can make mouse clicks return to
the program with the position.  You can display a 'sort of' button
using double and single line draw characters or just -|+ (you need to
use fixed font here):

+-------+
| START |
+=======+

You need to check for these after _every_ ACCEPT.

You can simulate a menu too, but it all rewrite and you would be much
better to just get SP/2 as you can use this on different Cobol systems,
different operating systems and even use it over the internet.

> I want to run my cobol program under the control of the
> graphical window, rather than the default
> console window that gets created by the cobol runtime system.

You can write your program to CALL Windows API routines to create
graphical windows, dialog boxes and controls and to have callback
routines to handle user interaction.  ie it can be a full GUI program
written in COBOL.  You can even use the drawing routines in the API to
draw graphs or pictures, or anything.

You could also use a GUI interface such as MF's Dialog or Flexus SP/2
to more easily create the user interface and handle the grubby MS
Windows (or other) API.  The program would still need to CALL the
interface handler.

If you want to find something that will take an existing program that
uses ACCEPT/DISPLAY and automatically transform it into a GUI program
with drop-down boxes and push buttons, then no, there  is unlikely to
be anything that would do that sensibly.

I've not done this (or anything) with Net Express.  But I have done this 
with other languages.  It requires knowledge of the "CreateProcess" (or 
maybe "ShellExecute") WinAPI call.  You are allowed to pass descriptors 
for the three standard file streams, sysin, sysout and syserr.

Write the program like any normal DOS application, with ACCEPT and 
DISPLAY.  Expect it to run under a CMD.com shell.  When you do your 
CreateProcess call, you might be able to get away with just calling your 
application.  But depending upon the Net Express runtime and all, you 
might have to invoke a CMD.com shell and tell that to call your 
application.  Your command string is going to look like one of these two:

   "myprog.exe arg1 arg2 ... argn"

   or 

   "cmd /k myprog.exe arg1 arg2 ... argn" (the /k flag keeps the cmd 
window hidden)

So set up your GUI application with whatever text window you want to 
have and populate it with the characters coming in from the programs 
sysout stream.  Send your syserr stream to the same place and save 
yourself 1/3 of the trouble.

Send input by writing to your sysin stream.
