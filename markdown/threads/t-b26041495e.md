```
I found your inquiry about GO TO less programming interesting. I offer the
following solution to your question (using your program variables):

PERFORM 010500-FIND-HELP.
IF HL-HELP-SENT-FLAG = 1
...


010500-FIND-HELP.
*
MOVE 0 TO HL-HELP-SENT-FLAG
SEND-TAB-COUNT.
*
MOVE "O101-ACTION" TO HL-DATA.NAME.
MOVE RECV-01-ACTION TO HL-DATA-VALUE.
PERFORM 002500-TEST-HELP.
IF HL-HELP-SENT-FLAG = 0
MOVE "O01XX-TEL-NBR" TO HL-DATA-NAME
MOVE RECV-01-TEL-NBR TO HL-DATA-VALUE
PERFORM 002500-TEST-HELP.
*
IF HL-HELP-SENT-FLAG = 0
MOVE "O1XX-QQQ-NBR" TO HL-DATA-NAME
MOVE RECV-01-TAG-NBR TO HL-DATA-VALUE
PERFORM 002500-TEST-HELP.
*
IF HL-HELP-SENT-FLAG = 0
MOVE "RSMR-NAME" TO HL-DATA-NAME
MOVE RECV-01-NAME TO HL-DATA-VALUE
PERFORM 002500-TEST-HELP.
*
IF HL-HELP-SENT-FLAG = 0
MOVE "RSMR-2NAME" TO HL-DATA-NAME
MOVE RECV-01-2NAME TO HL-DATA-VALUE
PERFORM 002500-TEST-HELP.
...
...
...
IF HL-HELP-SENT-FLAG = 0
MOVE "O1XX-NXT-SCR" TO HL-DATA-NAME
MOVE RECV-01-NXT-SCR TO HL-DATA-VALUE
PERFORM 002500-TEST-HELP.
*
*
*
002500-TEST-HELP.
etc.
etc.

This version of the program has much to offer: 1) simple - there are no
complex logic structures or "tricky" code; 2) easy to read and understand -
notice that there are no confusing nested IF statements; and 3) easy to
maintain - even when maintenance is being performed years after the
original code was generated and by a person who did not produce the
original code. Granted, you may well incur additional machine cycles but I
think that that is a small price to pay in view of the benefits gained.

For what it is worth, I was first exposed to Yourdon's structured
programming methodology in 1977 and was so impressed by it that I vowed to
follow its tenants religiously. Since then, I haven't written one program
which uses a GO TO or a PERFORM THRU statement. There hasn't been one
assignment to come along in all those years which required that I use these
constructs. But you have to be prepared to give the program logic some
serious thought and planning.

I hope that this is the type of response you were looking for and is of
some value.
```

```
Thanks for your input. Yes, this method does have certain good points, and
I have used it on occasion. But there are three reasons why I don't use
it; you can judge whether the reasons are sufficient for yourself.

First is the number of additional IF statements. For a 'typical' online
program of 30 screens, averaging 50 fields each, that's 1500 extra
conditional tests, just for the help test, plus another 1500 for the field
edits. An extra 3000 compares does not seem negligible to me, in memory or
execution time. It is true that you would eliminate the same number of GO
TO statements, but a simple IF consists of a compare and a branch, the
branch being equivalent to the GO TO, the net result being to add one
compare when replacing the GO TO with an IF.

Second, while this method would work okay for the help tests or field
edits, which are very similar, it would not work as well for the general
edits. The general edits for a given field, or set of fields, may take
several statements with complex logic. You wouldn't want to repeat those
for each following test, so you would need to set a flag which is tested
repeatedly as in your example below. To me, this is substituting
complexity for simplicity, to no gain.

But most importantly to me, these methods, such as you and others have
suggested, are just not simpler or more elegant than the GO TO exit. Like
you, I was strongly influenced by Yourdon. But Yourdon's criteria is that
each logical code block contain one entry point and one exit point, which
the PERFORM THRU exit with GO TO exit strictly satisfies. Almost all block
structured languages provide constructs for early exit of a code block.
The GO TO exit, to my mind, exactly fills this purpose, and is quite in
order. Do you really think one long 500 line IF statement is more clear?
I think not.

>From the original 'GOTO Considered Harmful', the GO TO statement has been
looked upon by many as bad news, no matter how it is used. In my opinion,
this is short sighted and does not convey the true picture. What Yourdon
saw, I believe, is the real truth behind use of the GO TO. I believe
Yourdon expressed that truth succinctly in the principle that a code block
must have only one entry point, and one exit point. Consider why GO TO is
considered harmful in the first place. I believe it is the 'unrestrained'
GO TO which is harmful, not the entire concept of GO TO, though I do agree
in minimal use of GO TO. Here is a paragraph from a computer science
lecture I found at URL:

http://www.cis.ksu.edu/~mcraig/200/history/S96Lecture4

Many years ago Edsger Dijkstra published a paper whose title has now
become
one of the famous titles in the computer field: "GOTO Considered
Harmful".
When used without discipline or constraint, GOTO's make source text
unreadable: people are unable to keep track of where a GOTO goes or how
various statements can be reached during execution of the program. We
cannot avoid the GOTO at the machine level, but we can insist on
languages
that will make it difficult, if not impossible, to use "bad" GOTO's. To
distinguish between "good" and "bad" GOTO's we need to discuss some of
the
art of program design.

There are basically two situations I have encountered where I would use GO
TO. One is in languages like COBOL74 and earlier, which do not provide
logical code blocks. The other is when there is a long sequence of code in
which there is need to terminate the sequence early, but also no logical
reason for splitting the code into separate routines. In these two
situations, carefully restricted use of GOTO provides, in my opinion, the
simplest, cleanest, most straightforward, and most efficient solution. All
the GOTOless methods suggested in this thread have suffered in one or more
of these areas, and I believe that all will inevitably do so. I believe
that completely eliminating GO TO in these situations is forced, and simply
provides no benefit, either in code clarity or efficiency, but actually
accomplishes the reverse.
```

```
I want to toss my $.02 into the maelstrom here. Basically, I agree with
Michael. I think the essential point it that the structure of a program
with GOTOs and PERFORM THRU's is almost bound to become corrupted over
time as maintenance occurs. I have no doubt that Judson (and many other
people, too) can write a straightforward program using these constructs
- the issue is what happens to the program as multiple programmers apply
their fixes/enhancements over time, some with *very* tight deadlines.

I was struck several years ago while using PATHVU to analyze Cobol
source (this was OS/VS/COBOL) by how much more there is to keep track of
(meaning entry and exit points) when using GOTO & PERFORM THRU than when
not. You may believe that you do this automatically but your brain has
to keep them all straight in the same manner that the source analyzer
has to.

Finally, I'm completely unconvinced by the argument of "greater
efficiency". I consulted with Candle Corp. (the OMEGAMON people) about
10 years ago. I learned there that this is generally a waste of time and
that the company paying the programmer doesn't see enough savings (in
machine cycles) in ten years to have the programmer spend an hour on it,
e.g., the extra IF statements. Consider how many cycles are saved in a
screen editing routine, whose usage is obviously limited by how soon the
terminal user can read the screen, make changes and press ENTER. Even
with multiple people entering data, it's nothing compared to the
internal CPU speeds. I worked for IBM in 1965 - 1970 as an SE. This was
the time that the S/360's were hitting customer sites and customers were
(mostly) converting from Autocoder to Cobol. I remember many bulletins
recommending using GOTOs instead of PERFORMs because the PERFORM used so
many more instructions. I *never* saw any evidence that this actually
saved anyone anything tangible, or if it simply reflected the prejudice
of people who'd been coding Autocoder (the 2nd generation's Assembler).

We've learned the hard way that the single most important issue for
commercial data processing (e.g., Cobol shops in banks, distribution,
manufacturing, etc.) is maintainability. Using GOTOs and PERFORM THRUs
tends, very strongly over time, to reduce maintainability.

Bill Lynch

PS: Keep in mind that what seems "elegant" or "messy" depends largely on
the programmer's point of view, i.e., what he/she is used to. It seemed
messy to me to code "MOVE A TO B" when I started coding Cobol in
1984, because I was used to coding "MVC B,A" in Assembler. I've worked
with many programmers over the years who, after spending some time
working with structured code, say they would *never* go back to using
GOTOs & PERFORM THRUs. I can't think of a simgle person who converted
from structured code to using GOTOs & PERFORM THRUs.
```

```
Actually, in this case, the situation is exactly reversed, for two reasons.
First, when you see the GO TO exit, you KNOW that none of the following
statements are executed, without having to look. But with your method, you
have to examine every single following statement, checking the IF
statements and scanning for extra periods to know that. And second, you
may have missed my earlier posts, but I scan all my programs with a checker
program to verify that each paragraph has exactly one exit, and that all GO
TOs are to the proper place. So control cannot go "flying off in
unexpected directions". As I pointed out, this exactly satisfies Yourdon's
'one entry, one exit' principle.
```
