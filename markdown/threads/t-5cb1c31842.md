Without seeing the code it is difficult to say, but...

It is pretty obvious that CANCEL is cancelling the whole run unit, rather 
than just the program specified.

Check that you have correctly specified the program to be CANCELled in the 
CANCEL statement. If this is dynamic, display the contents of the dataname 
before the CANCEL executes. If it is static, make sure you have the correct 
Program-ID or File name and that it is correctly quoted. It is possible that 
if CANCEL doesn't find what it is supposed to be cancelling, the results may 
be "unpredictable".

Check that Program B is not issuing a STOP RUN rather than an EXIT PROGRAM 
or GOBACK.

Check that any necessary compile options are in place to ensure that Program 
B is not compiled as a main program. If you exit from a main program, your 
run unit will be stopped.

It is very unlikely that you have a bug in CANCEL; far more likely there is 
a program(mer) error... :-)

Pete.

Chris,

I see you're using MF COBOL, so I'm not sure if this applies, but here 
goes:

In ACUCOBOL-GT, it's possible to code a DECLARATIVES section with a USE 
AT PROGRAM END clause. That section then executes whenever the program 
ends.

Now, if Program B had such a section, and it included a STOP RUN, then 
you'd get the result you're seeing. That's to say, the STOP RUN would 
execute (quite legitimately) when Program A executes the CANCEL.

That's the only way I can see (other than a runtime error) to get the 
result you're seeing.
