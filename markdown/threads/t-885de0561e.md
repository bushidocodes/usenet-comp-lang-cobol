Hello All,

I have a basic question about data 'retention' in CICS. I am in the
middle of a systems project at school where my team and I are
developing an online CICS system for a theoretical light industrial
machine shop.

On one of our screens, we'd would like to start a counter which
increments automatically when we enter any new data. This counter is
known only to the system and does not show up on screen. This number
in effect would be the primary key for a VSAM file which holds the
data from that particular entry.

However, from past experience I understand that data in CICS cannot be
held from screen to screen since it is re-entrant, i.e. when we call
up the transid again, whatever the counter was before would be
clobbered. One of our solutions would for the CICS proggie to 'read' a
file where the counter was written to in the previous screen. After
the data from an online screen was written on a file e.g. part number,
part name, etc., the program would increment the counter again, write
that number to a file, and the process of reading that file to receive
the next number would begin again. I was wondering if there were any
more elegant solutions to this? If our solution is doable, we'll
probably stick to it. I'm just curious to hear other options.

Thanks for yer time,


Rowell


PS. Yeah, you've heard it a million times before but, remove the X's
in the mail address if you want to reply via email. Otherwise I'll
read it in the group.

Like many other things, at all depends. If you stick the no. in transaction
storage, obviously it'll disappear when you RETURN. However, if you GETMAIN
SHARED storage, it won't be released when your transaction end. Problem,
where do you save the address of the SHARED storage?

If you save the no. in Temp Storage, hmmm, you can READQ & update it until
the cows come home, no problemo. Same if you save the no. in a file, READ &
REWRITE to your heart's content.

Don't forget to consider what happens if the system/CICS region crashes. If
you saved the no. in SHARED storage, it's gone, even if you solve the
problem of where to save the address of the SHARED area. If your Temp
Storage queue is recoverable, no problemo (remember Temp Storage queues in
MAIN are not recoverable, just AUX queues). I hope your files are
recoverable - so, again, this is also no problemo.

The systems I work on mostly use the technique of saving the no. in a MAIN
Temp Storage queue (for performance, plus it's a pretty small record) and
handle a region crash like the region startup. Meaning you READQ the TS
queue, get a QIDERR, and you know you have to:

1. browse the file to get the highest no. assigned;
2. park this no. in a TS queue.

Hope this is helpful,
Bill

You can save data between invocations of your program by defining a
COMMAREA to hold variables. You have to have initialization logic to
determine if you have received this COMMAREA. When you are ready to
release control, do EXEC CICS RETURN COMMAREA(your-commarea)
LENGTH(LENGTH OF your-commarea) END-EXEC. This is common technique for
saving data between displays to the screen.

› file where the counter was written to in the previous screen. After
› the data from an online screen was written on a file e.g. part number,
…[8 more quoted lines elided]…
› Rowell

There are two other solutions that I have used. In one application we
need to identify transactions with a unique number for audit purposes.
We store the counter in a global table (loaded as a program). The table
is an assembler program that consists of DC statements only, in order to
allocate storage. The in-core table program must be defined as
resident, so it doesn't get paged out. You have to do a little more
work to pass the address of the table around and map it into your
program. When we need a unique number, we add 1 to the counter and use
that as our audit control number.

An easier alternative would be to use a TEMP STORAGE queue. Assign a
Queue name (8 bytes). In your program initialization logic, do EXEC
CICS READQ TS QUEUE('NAME') END-EXEC. If the queue doesn't exist,
WRITEQ TS to create it. You will need to create the data area with a
counter initialized to zero. When you need a unique number, READQ TS,
increment the counter, WRITEQ TS (actually a rewrite), then use the new
number. Your TS queue should use main storage rather than auxiliary
storage and you should default to using item(1), rather than allowing
your TS queue to contain multiple records. This will work much faster
than a file, and it does not use much storage.

If you use this technique, the counter will be reinitialized every time
the CICS region is brought down and back up. You may also want to
Enqueue/Dequeue on the TS Queue to prevent two tasks from getting the
same number. This is actually very unlikely with a TS Queue or an
in-core table, but a very real risk when using a dataset.

I hope that helps!

Using a VSAM file with a control number in it is not a bad idea since the
enque issued by READ UPDATE prevents two terminals from assigning the same
number to a transaction and the sequentiality is maintained across
invocations of CICS...Data is generally buffered adequately so performance
is not a real problem, i.e. the data is written but would rarely be read
from the file (no SHR(4) of course) ... As an alternative, an easy
serializer is delivered via the ASKTIME command which gives an S9(15)
COMP-3 field know as absolute time ... the time in hundreds of seconds
since Jan 1 1900 ... there is an outside chance of duplications of this
number (depending on the speed of your system) so it may be advisable to
issue and enque to serialize access by all programs using this resource ...
This is a great audit trail stamp but is cumbersome for requesting access
by record number since it's a pretty long number.

Kevin's talking about the "...ASKTIME ABSTIME()..." command. I researched
this roughly 5 years ago and recall that the resulting 15-digit packed
decimal number is resolved to 1/300 of a millesecond. We tried to use the
ABSTIME to assign a unique REQID for internally-generated tasks, and it
worked great for about 2 years. Then we more than doubled our volume and
started having problems - duplicate REQIDs. When the 2nd task starts and
issues a RETRIEVE to find out what it was supposed to do, it gets the data
from the 1st task (with the same REQID). We simply had too much concurrency
on the big Amdahl (5990?), which has since been upgraded to a bigger IBM
system.

I rewrote the REQID generating logic. We're using the 2-char. project
prefix plus a 6-digit number (display) to make up the REQID. The no. is
kept in a record and rolls from 999999 to 000001 (we use REQID = zeros for
system-level control records). All the code to increment the no., update
the file, establish the TS queue, is in a dynamically-CALLed Cobol II
program. Not a single hiccup from it since we implemented it 2.5 years ago.
