Actually it's common to programs running under Windows REGARDLESS of source 
language if the program does not process messages; that is, if it's a 'batch 
type processing' program containing "tight loops" ..  eg. read a record, 
modify a record, write a record.

However, I'm not so sure I would call it a "problem" to fully utilize your 
computer hardware.  That said, programs with "tight loops" which eat CPU 
time tend to have a negative impact on all the OTHER processes running on 
the box.

The most common adjustment to these programs if the CPU utilization is 
deemed a problem is the one you have found: yielding the balance of your 
time slice with the Sleep function at logical points within the processsing, 
e.g., after processing each single record.  BTW, you don't show the value of 
SLEEP-TIME; if this value is zero control is yielded to other threads of the 
same priority (if not in a wait state); if non-zero, control is 
unconditionally yielded to other threads for *at least*  the value of 
SLEEP-TIME.

So, you may get better overall performance by setting 'some value' of 
SLEEP-TIME. Sorry, there is no magic formula or algorithm to use. If this 
application is widely distributed you might consider making this value a 
"user option" or something you read from an INI file to allow  tuning 
without recompilation.

Another solution is to modify the thread priority (see the SetThreadPriority 
function) to a lower priority; after all, this IS a 'batch' process, right? 
(but remember how 'Sleep' works vis-a-vis to whom and for how long control 
is yielded).

If the process is I-O intensive, Asynchorous I-O could be an avenue of 
approach as well, as Asynch I-O automatically involves yielding control to 
other threads of execution.

Pre-emptive multi-tasking such as exists under Windows/32  came well after 
COBOL was created, so it should not be a surprise to anyone that several - 
or many - "accepted COBOL programming techinques"  don't play well with 
certain operating systems.

I suspect that the version of MF was designed to be used with Windows
95/98. PC_WIN_YIELD and such was required because Windows worked using
a message queue. In win3.x there was a single message queue for all
processes and not yielding locked up every process. Win 95/98 was
supposed to fix this but all Win3.x programs ran in a single process,
and Win32 (ie Win95/98) programs should have run in their own process.

I find it interesting that it can use 100% while reading disk files.
In rational systems a request for a disk transfer (at the block level)
is handed off to the OS and the program uses no CPU time while the
disk transfer takes place, even if this is just drive cache to RAM
cache, and the OS gives this time to another program to use the CPU.

The implication of 100% is that the program is idling in a loop using
CPU time even when the next record it is wanting to process is sitting
on the disk drive and the drive heads are moving to the track (10ms)
and/or waiting for latency (6ms).

This is what I associate with MS-DOS programs in the 80s that sit in a
keyhit() loop using 100% CPU because 'what else could use it?'.
Fortunately I used multi-user/multi-tasking systems that detected this
behaviour and catered for it.

It may well be that the run-time is checking for keyhit() or similar.

What happens if every, say, 100 records, you display a message on the
console, perhaps the record key or a counter ?

If, however, the read-ahead is working perfectly and the 100%
represents actual processing load (rather than busy-wait) then
reducing the CPU load to 50% would double the run time.

The best solution may be to not use Windows.
