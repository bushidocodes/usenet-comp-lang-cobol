business-oriented specifics, for two reasons.  One, I'd probably get in

Here's kind of an 'outline' of the program.

Main program, contains all other programs:
  'CICSV2K'

Nested programs that can be called by the main program but not by each
other:
  'acctCardListByCust'
  'cardLimits'
  'cardTranHistory'

COMMON programs that can be called by all other programs within this
compilation unit:
  'NEXTCARDHOLDERRECORD'
  'TERMINAL-LOCATION'
  'OUTPUT-ROUTINE'
  'CICS-ERROR'
  'GET-ERROR-INFO'

Finally, there are some actual 'external' subroutines (source code not
posted):
  'CARDSYS'
  'CEESITST'
  'CVASJULD'
  'COB2XML'
  'SENDDOC'
  'ERRINFO'


Notes of note...

The three primary nested programs perform three primary functions that can
be requested by the caller (meaning the program linking (this is CICS) to
CICSV2K).  You may notice that none of the three have any parameters passed
in.  All of the parameters they use are declared as being GLOBAL.  Really,
there is only one 'parameter', as such, that being the COMMAREA (VIEWCOMM
copybook).

I could instead have passed the commarea as a parameter to each of the 3
primary nested programs, and I'm sure there are those who think that I
should have.  The main reason I did not is because I just didn't see the
point of having the same copybook in four different places.  It just bloats
the compile listing and simply did not seem like a justifiable thing to do.

To further justify myself, try looking at it as if it were an OO program. 
Think of all of the nested programs as being methods, with the main program
procedure division kind of being a 'driver' method.  (Though in real OO it
wouldn't really be needed.  You'd just invoke the three primary methods
directly.)  Anyway, going with this thinking I see the "global" working
storage as performing the same function as would "Object" storage (in OO
COBOL that would be the WORKING-STORAGE section defined in the OBJECT; not
as a parameter or local-storage to a method.)

Of course my justification falls apart a bit there because in true OO, as I
said before, one probably wouldn't have the 'driver' method at all.  But
lets imagine, for whatever reason, one did.  So, as I said, it would (I
think!) be quite acceptable to have the 'commarea' as instance (object)
fields, rather than passing them in linkage.

Now we get to where I'm a bit less justified, but I still did it for the
same reason.  I have some other fields defined as GLOBAL.  The CVCMUCHR
copybook (01  CHR-USER-RECORD), for example.  Again, I could have (should
have?) defined these in the local working-storage for each nested program. 
But I simply couldn't see the point.  All it would do is take up more
memory, more space on the compile output, etc., and I wouldn't (in this
case, at least) have gained any real advantage.

Anyway, the 'NEXTCARDHOLDERRECORD' is more of a traditional called nested
program, in that I pass two parameters.  But again, I'm not passing the
cardholder record itself, again using the GLOBAL record.

One of my biggest disappointments with this whole process is how I am
handling the call to 'OUTPUT-ROUTINE'.  Rather than simply calling it from
several places, I am instead performing a paragraph which does the call. 
I'll show it again:

 OUTPUT-ROUTINE.          
     CALL 'OUTPUT-ROUTINE'
     IF GLOBAL-ERROR      
         EXIT PROGRAM     
     END-IF               
     EXIT.                

Obviously, the reason I am doing this is because if the error-checking logic
that follows it.  But it really annoyed me that I had to duplicate logic in
order to not duplicate logic!  In other words, had I not insisted on doing
nested programs I could have simply had what is now the OUTPUT-ROUTINE
program as a paragraph that I could have performed from multiple places!  I
did something similar for the call to the external subroutine CARDSYS, for
function READCARDHOLDERREC.

In the end, would I do something like this again?  Hmm, I'm not sure.  Very
possibly not.  External subroutines are one thing, and are very useful
because the same logic can be called from many programs.  Nested programs
don't have the same benefit.  They can be called only from the programs in
which they are nested (or, if COMMON, from other nested programs within the
same compilation unit.)  There are only two real benefits I can see that
nested programs have:
1) Locality of reference.  Having a variable declared closer to where it's
being used.  Personally, I would like COBOL to have "paragraph local"
variables.  Especially for subscripts for use within loops.

2) Passing of 'like' but actually different fields.  In other words:

CALL SUBROUTINE USING FIELD-1, FIELD-2
CALL SUBROUTINE USING FIELD-3, FIELD-4

rather than
MOVE FIELD-1 TO A-PARA-DATA-IN
PERFORM A-PARAGRAPH
MOVE A-PARA-DATA-OUT TO FIELD-2
MOVE FIELD-3 TO A-PARA-DATA-IN
PERFORM A-PARAGRAPH
MOVE A-PARA-DATA-OUT TO FIELD-4


A-PARAGRAPH.
*  input being A-PARA-DATA-IN
*  output being A-PARA-DATA-OUT

But in the end, is really worth it?  Again I am not convinced.
What I am somewhat convinced of is that it appears that the whole idea of
nested programs was added to the COBOL 85 standard prior to anyone actually
implementing them and working with them.  (Much of the COBOL 2002 standard
appears to have been done this way as well.)  The idea is quite sound.  But
I'm not sold on the implementation.  One thing that may have made it work
better would have been if the TYPEDEF clause had existed in COBOL 85.  Had
this been the case I would have bad no hesitation in making the COMMAREA
structure a TYPE, and passing it around when necessary.

Anyway, enough rambling for now.  Comments, criticisms, and especially
suggestions on how to make this program better are all welcome, as long (as
they are reasonable!).

I've used them in a couple of places.  I really like the separation of 
data.  However, on the Unisys platform, the network database stuff isn't 
visible from them, which is kind of a downer.

The biggest program I did was an online program that did input 
validation for nearly every batch program in the system.  That way, if 
the input was bad, we wouldn't waste the cycles starting the program, 
only to have it reject.  Plus, it gave the users instant feedback, as 
opposed to the run getting queued, them going home for the day, then 
coming in the next morning to get their reject.

Anyway, each run has a TRansaction Identifier Code (TRIC).  There was a 
paragraph called "Program-Specific-Edits" that was basically one big 
evaluate...

     evaluate wsInputTric
       when "ABC"
          call "ABC-Edits"
       when "DEF"
          call "DEF-Edits"
       ...
     end-evaluate

For those that needed network database validation, I had to put a 
"perform ABC-DB-Edits" after the call, and in some cases, I had to pass 
parameters to the subroutines.  They modified GLOBAL variables, but 
their work variables were isolated.  The program ran pretty efficiently, 
plus it introduced several other people to nested subprograms, as people 
could check it out and add edits for their programs.
