```
Hi,

I've been wondering about this ever since I graduated from school.
When I first learned COBOL, I was taught to perform SECTIONs as paragraphs.
The reasons my professor gave us for coding this way were:
1) It was alot safer as it eliminated the use of GO TOs
2) It eliminated the need for a PERFORM THRU and was thus more
"effecient".
Example :
000-MAIN SECTION.
OPEN FILE........
PERFORM 100-INITIALIZE.
PERFORM 200-LOAD-TBL.
STOP RUN.
000-EXIT. EXIT.

With paragraphs coded as:
100-INITIALIZE SECTION.
MOVE.........
...................
...................
100-EXIT. EXIT.

200-LOAD-TBL SECTION.
MOVE.........
...................
...................
200-EXIT. EXIT.

Since that time (2 1/2 years ago) I've been with a company whos Company Standard
is to perform paragraphs with the THRU option as such:

PERFORM 100-INITIALIZE THRU 100-EXIT

With the paragraph coded as follows:
100-INITIALIZE.
MOVE.......
.................
.................
100-EXIT. EXIT.

I've read plenty of articles where both have been either praised, or described
as "never, ever, do this.....". Talking with some of my fellow programmers
solves
nothing as they were all exposed to nothing but the PERFORM with the THRU
option attached and swear by it. Can anyone offer some insight as to the pros &
cons of these? They both seem to eliminate fall through logic, but the coding
of the PERFORM THRU (IMO) seems a bit tedious an unneccesary.
```

```
There are rooms full of books analyzing & recommending top-down,
structured coding, so I really don't want to debate that point. As far
as writing structured code goes, Cobol II solved many problems with the
OS/VS Cobol implementation, e.g.:

1. unbalanced constructs, like
READ ...
at end
MOVE 'Y' TO END-OF-FILE-SWITCH

have been replaced with balanced constructs like
READ ...
at end
MOVE 'Y' TO END-OF-FILE-SWITCH
(or, even better)
SET END-OF-FILE TO TRUE
not at end
PERFORM 2000-RECORD-ROUTINE
END-READ

2. Cobol finally supports DO loops (called in-line PERFORMs), with
PERFORM VARYING index-variable
UNTIL END-OF-FILE
process statements...
END-PERFORM

3. Cobol has scope delimiters, e.g., IF ... END-IF, which makes nested
IFs both readable & doable (no pun intended). If you're in the middle of
IF ... and want to CALL something, no problem:

IF ...
CALL 'SUBROUTINE' USING PARM1 PARM2
END-CALL
ELSE
OTHER PROCESSING
END-IF

This makes for nicely structured, readable, code. I recommend that
people code END-IFs as though they were required, like END-EXECs for
CICS commands. If I could, I'd tailor the compiler to require them and
to flag PERFORM THRU and GO TOs as errors.

As far as CICS HANDLEs are concerned, I almost always code something
like the following:

EXEC CICS READ
FILE (.....)
RIDFLD(key info)
INTO (record area)
RESP (WS-RESP)
NOHANDLE
END-EXEC

IF WS-RESP = DFHRESP(NORMAL)
process the record
ELSE
PERFORM 9999-ERROR-ROUTINE
END-IF

This keeps the code nicely structured, provides proper error handling,
etc.

If you code simple PERFORMs and no EXITs, it makes it very easy to
rearrange your program, if the need develops, or even to split the
program into two, because it's become too big or because you want the
functionality available to multiple programs.

This is an applied perspective, not an academic one. I have no idea what
a CS professor might recommend.

Anyway, hope all this is helpful,
Bill Lynch
```

```
The PERFORM ...THRU was nice for paras with EXEC CICS commands and
HANDLEs in them. Because you could code the HANDLE to drop to the EXIT.
(Which is really just another paragraph with only one statement
(EXIT.)).
I.e. HANDLE CONDITION NOTFOUND (0100-EXIT).

Some people went goofy with them: PERFORM PARA-1 THRU PARA-9, where each
para had some real lines of code. It was easy to get lost in that
paradigm if something in the middle caused you not to want PARA-9 to
execute.

You could also use GOTO's (let's not get into that again). For example,
when processing an array, you could GOTO 0100-EXIT if ARRAY-MEMBER (IDX)
= HIGH-VALUES instead of coding IF ARRAY-NOT-DONE for each processing
statement in the para after the array was interrogated.

One common problem was cloning code and forgetting to replace 0xxx-EXIT
statement with 0yyy-EXIT. Then the GOTO EXIT would go to the wrong one
and you'd execute all kinds of strange paras on the way to fall out of
the bottom of your program.

Not using the EXIT strategy required a different logic. I.e., pre-read
the input file and put the next read at the end of the SECTION/PARA, and
perform the SECTION/PARA UNTIL EOF. That way you would drop out of the
SECTION when the input hit EOF - didn't need to dance around subsequent
record-handling logic.

For CICS, the no-EXIT strategy was tough for HANDLEs, but I've seen code
(which I forgot) that gets around that. The problem is that the HANDLE
is a GOTO and you can end up falling out the bottom of the program for a
SOC4.

As I remember (may be incorrect) the SECTIONS paradigm was needed for
the INPUT PROCEDURE and OUTPUT PROCEDURE for SORT (may not be true now).

Years ago I preferred PERFORM THRU EXIT (yes - I'm a closet GOTO EXIT
pervert) but it is no big deal to use the other techniques. Most of my
recent coding has been to simply perform PARA after PARA with embedded
'indented' PARAs - no mention of SECTIONS although it was defacto
similar to them. I.e., the 'A' PARAs were the inits, the 'B's were the
handling for situation B, the 'C's for situation C, the 'X's for DB2/SQL
statements, the 'Y's and 'Z's for error handling, etc.
And no GOTOs nor EXIT PARAs - pre-reads being the rule.

I don't have a clue about efficiency or any of that aspect.
```

```
Well, speaking as a compiler writer (more than a quarter century ago, mind
you) ..., not to mention being a former member of the Codasyl COBOL
Committee and ANSI X3J4.

The purpose of SECTION was to allow you to structure your program so that it
could be organized into overlays to conserve memory. A properly written
program was expected to *not* transfer control back and forth between
SECTIONs that were in different overlays. Ideally, there would be no
transfer of control back and forth between sections even if they were in the
same overlay.

There's absolutely NO difference in efficiency between PERFORM THRU and
PERFORM without the THRU option. Compilers are supposed to use an implict
THRU when that option is missing:
PERFORM 123_FOO
is equivalent to
PERFORM 123_FOO THRU 123_FOO

The THRU option just tells the compiler the paragraph or section at the end
of which the exit code is to be placed.

You can actually use the THRU option when referencing SECTIONs, and you can
also have more than just an EXIT statement in the ending paragraph of the
PERFORM range. Compilers are supposed to check that the exit code is
actually reachable. (At least, any of the compilers I messed around with
did that.) It IS, of course, good practice to have ONLY an EXIT statement
in the last paragraph of the PERFORM range.
```

```
Paragraphs can be grouped together in sections. When the section is
performed all the paragraph names within the section are performed.
This is more efficient than opening several perform macros during
execution to perform each paragraph(a bit like a working storage group
move instead of individual move statements). To my mind a section
contining no paragraph names is the same as a paragraph.

Some shops code paragraphs within sections for documentation purposes.
In large batch programs these can, along with display statements help
locate errors. These paragraphs are not performed. See example below.

You should never perform both sections and paragraphs in the same
program. A perform section will continue until it hits the next
section. In the example below if you mistakenly leave out the word
'section' after C-MAIN-PROCESS it becomes a paragraph an is considered
part of B-INITIALIZE.

Perfrom thru should be avoided because it is not consistant with
structured programming as per Jackson Structured Design and makes
maintainance more difficult.

It is still a good idea to delimit the end of a paragraph or section
with an exit paragraph name. This makes the program more readable and
you can then code the paragraph statements without any full stops except
for one just before the exit paragraph name(see example below).
Misplaced full stops cause silly program errors and their use should be
minimized.



A-CONTROL SECTION.
*******************

****************************************************************
*
* FUNCTION : MAIN CONTROL
*
* PERFORMS : B-INITIALIZE
* C-MAIN-PROCESS
* X-TERMINATE
*
* PERFORMED BY : NONE *

*-----------

DISPLAY '***************************************************'
DISPLAY '* PROGRAM FW621 COMPILED ON ' W010-WHEN-COMPILED
DISPLAY '***************************************************'

PERFORM B-INITIALIZE
PERFORM C-MAIN-PROCESS UNTIL W060-END-OF-FE37
PERFORM X-TERMINATE

DISPLAY '***************************************************'
DISPLAY ' PROGRAM FW621 - ENDED SUCCESSFULLY'
DISPLAY '***************************************************'

GOBACK

.
A999-EXIT.
EXIT.
EJECT
```

```
I'm reluctant to touch this thread, but there are two points I would like
to make.

First, I suspect that some of the proponents here of PERFORM ... THRU hold
a view similar to my own. I don't PERFORM paragraph 1 on past paragraphs
2, 3, 4 and 5 THRU paragraph 6, but construct each paragraph with its own
EXIT, which I PERFORM THRU, like this:

PERFORM 012340-DO-SOMETHING
THRU 012340-EXIT.

012340-DO-SOMETHING.
...
012340-EXIT.
EXIT.

I never, never, never GO TO outside the paragraph. The requirement of good
structured design is that each code block have exactly one entry and one
exit point, and this construct satisfies that. As Colin Lucas points out
above, other languages have constructs which allow premature termination of
a code block, and here a GO TO EXIT simply provides that. I have been in
the position of setting or suggesting programming standards for development
groups since the mid-70's, and I have seen this approach work successfully
for some 20 years in a number of MIS departments. Many of those MIS shops
adopted the standard because of how successfully it worked where I
implemented it. And I have yet to see one example of a programmer being
confused by this. I have certainly seen programmers confused by IF
statements nested too deeply.

Second, before inline PERFORM loops were available in COBOL 85, there was
no way to code a loop (other than in certain cases using a SEARCH, or
perhaps an INSPECT) without incurring the performance penalty of a
subroutine call, unless you used GO TO. Most of the time the clarity and
elegance of coding the loop in the PERFORM statement is worth it. But
there are times when speed is a high priority, and to achieve it, you had
to code your loops with GO TO. In these cases, my standard is that you can
GO TO the beginning of the paragraph you are in, but never outside it. My
COBOL programming is divided between PCs and mainframes. I use a COBOL 85
compiler on the PC, but am restricted to COBOL 74 on the mainframes. Code
reusability is also very important, so I usually code to COBOL 74 so I
don't need two versions of all my common routines. So, unless the code is
specific to the PC, I use GO TO for loops in those situations where
performance demands it. GO TO purists may fault me for it, but it works
well with no problems I can discern, and I intend to continue to practice
and teach this method without apology. :-)
```
