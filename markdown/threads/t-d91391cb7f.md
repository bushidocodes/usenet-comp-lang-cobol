The opening and closing of a GDG in a realtime context, as a user file, is
almost a non-sequitur. But CICS can get to almost anykind of a file if you can
find a DBA or systems programmer willing to support your grand plan. 

I have seen CICS write SYSOUTs! So QSAM should not be impossible. (although my
knowledge could be outdated in some newer CICS releases). 

Getting to the generation number is highly unlikely. So if you need to know it,
or want to 
specify it way over in your application, that is 
unlikely, IMHO.

A GDG dataset could be anything, and can land on nearly anykind of device, but
from the get-go this kind of thing is usually huge, and oriented to some very
long process that is the antithesis of an online transaction.  It is not rare
that a GDG is on tape, again not exactly speedy.  And a tape mount is much more
of an issue than the access method.

Frankly, it would be better to think of some alternatives, like just a regular
old file name. Whatever is producing a GDG file name could just as easily
generate a regular file name for you or you can copy the thing.

But if you want dynamic selection of data based upon the generation number, I
would say you need a different design.  The real issue here 
might be the possible management of 
dataset names inside you online app, ... generally CICS is more of a planned
environment, where the DBA control dataset name and dataset selection.

Separately, to offer as much positive response as possible, please note that a
GDG ultimately just creates a unique DSN that happens to have a goovoo node. It
is not rocket science. But if you want to be trafficking in goovoos inside a
CICS transaction in some literal form or as generation numbers then rethink
this. But for special cases the CICS JCL can specify the full DSN.

Best Wishes,
