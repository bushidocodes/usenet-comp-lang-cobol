What platform?  It depends.  If this is IBM mainframe OS/390 Cobol,
then COMP-3  is packed decimal format.  One byte holds two values
except the last byte which holds 1 value and the sign (assuming all
other options are standard).

For example, assuming you had the number +6880976..in display format
it would look like F6F8F8F0F9F7F6.  If you were to move this number to
cod, it would be converted to packed decimal and would look like this:

006880976F (it could also look like 006880976C if the compiler
generates a ZAP instruction to do the move or does an OI on the last
byte).  The size of a packed decimal field is the number of digits
divided by 2 plus 1.  You should always use an odd number of digits to
define your packed decimal fields.

One of the most common comp or oomp-3 formats is created using some technique to
"pack" numbers to save space.  normally a one digit number takes up the space
required for a single character.  When you pack a number, the space is devided
in half.  Half of the first byte is used for the sign.  Then after that each
byte is capable of holding 2 numeric digits.

pic 9(1) comp-3    = 1 Character
pic 9(3) comp-3    = 2 Characters
pic 9(5) comp-3    = 3 Characters

Normally if you are packing numbers you use an odd amount of digits to use the
entire byte. Half a byte by itself isnt worth much.  Many compilers can not
access just half a byte.  Some languages like Assembly may be able to access the
left or right side of the packed bytes.  Some compilers have different data
types like binary, character, zoned decimal(number), packed(number), boolean,
etc, and these are referred to as comp-? to indicate a non character-field.
How the compiler implements the packing may be different.
