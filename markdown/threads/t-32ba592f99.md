You can probably convert relatively painlessly.  It will really depend on
how the screen I/O was handled, but most Cobol code is pretty well the same
until you get into screen stuff.  Even there, there it tends to be similar
enough that the conversion can be mechanical.

> 2)Should I rather find learning material that focuses on Accucobol?
>

No, not neccessarily.  I would decide what I was going to use first.  It
would probably be a good idea to do your learning with the compiler you
intend on using for the project.

>3)What will I not be able to do with the free compiler from Fujitsu?
>

GUI.  You are going to have to come up with a screen interfacing mechanism
... if you go with Fujitsu, then your choices are probably PowerCobol (also
Fujitsu) or Flexus.  I use Flexus.  You should keep in mind though, that the
free Fujitsu Cobol compiler is several versions old.  Anything you buy is
going to be up-to-date, and may not work with software that old.  Whatever
Cobol you decide to use, I would suggest getting and up-to-date compiler.
Deciding to convert to a compiler already five or six years out of date is
no favour to your employer.

> 4)From an experience point of view, will I be able to become a complete
COBL
> programmer to create full and new GUI interface program from books, or
> should I rather find a school?

If you are going to actually write code for 8 hours a day, you will outpace
a typical school in about a week.  The only real way to learn is to write
lots of code.  Just be willing to do the first couple of modules over after
you finish the last few.

By 'dialects,' I mean slight differences between the languages. Some (very
few) parts of COBOL were not fixed by the standard and left as 'implementor
defined.' Other (even fewer) parts of COBOL are peculiar to a specific
vendor, the most common oddity being the hook from an internal name (such as
a file) to an external device (such as a physical printer). Compiler vendors
have different ways of implementing or defining the 'hook.' But that sort of
conversion is almost mechanical.

You can use PowerCOBOL version 3 which comes with the version 3 COBOL
compiler, to develop GUI applications. It is really very simple. You can
define GUI forms and drag and drop controls onto them. You then select the
events you are interested in (like "Click" for example) and write COBOL code
to handle the event. It is a little diferent from normal procedural code
because it is event driven, but you may be able to cut and paste the
existing code into the events. So, for example, when a certain button is
clicked, update the master file. You have all the existing code to do that.
All you need do is cut and paste it into the Click event of the "update"
button. PowerCOBOL is a superb product and saves many hours of work by using
prebuilt controls with their own events, Methods and Properties. Be aware
that version 3 is 16 bit and you may need to get a later copy (current is
version 6.1) to run under Win 32. (You can probably buy it from e-bay if you
search...)

The screen I/O has never quite solidified into a standard.  For example, to
clear a screen, it may be "DISPLAY SPACE" or it may be "DISPLAY (1, 1)
SPACE", or it mat be DISPLAY SPACE AT LINE 1 COLUMN 1, depending on the
compiler (and the era the program was written in). The stuff is always
similar enough that conversion is something that can be done with a good
text editing macro, but there will be differences.  The files could be
different as well, as Jerry said.
The differnces tend to be relatively trivial, but you should be able to get
the current programs working without any logic changes at all.  The changes
will be changes in syntax (like replacing all instances of "COLOUR with
COLOR", because Fujitsu did not recognize the former in version 3.

No it shouldn't be a "No" <G>.

Event code in PowerCOBOL is written in standard COBOL complete with all
divisions.

It is just that what may have been a single program as a "batch" solution,
may be "split" to a series of functions or modules (actually nested
subprograms, although this is transparent to the User) which will be
activated by events in the GUI interface.

It is still COBOL, unlike MicroFocus DIALOG, which has it's own language
(albeit a good one) to handle event code.

In a Windows environment, there could be literally millions of events
happening every minute (disk access, mouse movement, incoming network
traffic, etc.). Not only, but Windows is a pre-emptive multi-tasker (the
events are not necessarily processed sequentially). A GUI front end
continually monitors these events and sorts out the ones in which your
program is interested, tries to reconcile conflicts within your program to
assign priorities, and only then does your program get an opportunity to do
something.

for a moment, that other parts of your program can get control when you

automatically, that, in turns, needs information in yet another field. By
accessing field #3, the validation routine in turn triggers a spell-checker.
Finally, everything collapses back to you. Or not.

The whole thing is immensely complicated and quite beyond the range of human
comprehension. Fortunately, PowerCOBOL (et al) reduces your programming
effort to:

Perform Until DONE
   Call "GET"
   Call "MULL"
   Call "PUT"
End-Perform
LayBack.

With suitable macros, the above can be shortened to:

Ponder.
