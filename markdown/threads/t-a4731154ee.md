```
I am looking for some documentation or help in hand-decoding numbers stored
in COMP fields.  My background is not in COBOL, but I work for a credit
bureau that receives data from miscellaneous providers, much of it output
from COBOL applications.  I have to get this data into a SQL Server database
which is ultimately published on a private web site for our members.

I use Data Junction for initial review and decoding of new data formats.
This is a general-purpose data conversion product that understands COMP and
various COMP-n numeric formats.  Generally I get some kind of documentation
from the data provider.  Some providers are more willing than others to
answer questions or provide support; the provider I'm currently working with
is in the "not helpful" category and has basically thrown a 9-track tape and
a semi-up-to-date documentation sheet "over the wall" at me.  My job is to
get this data into our system without antagonizing the provider with
anything so presumtuous as a sensible question.

Here's the deal:

The documentation for the file layout shows three different definitions for
numeric fields.  They are:

PIC S9(7)V9(2) COMP.
PIC 9(9)       COMP.
PIC S9(4)      COMP.

My understanding of the above notations is that the actual stored number
consists of:

9 digits with 2 implied decimal places
One digit integer  (although why one would bother to compress a single digit
is beyond me)
Four digit integer

I'm given to understand that a COMP field can be anywhere from 1 to 8 bytes
wide on disk.  But is it possible to look at, say, an S9(7)V9(2) format and
via some formula, be able to state with confidence that this would always be
a 4-byte value on disk?  Or might a programmer elect to make a field wider
than necessary, perhaps for future expansion?

I have a series of perhaps 20 numeric fields with a mixture of the above
three formats and no matter how I try to decipher these fields I end up not
accounting for all the bytes in the record.  There are a couple of clearly
identifiable character fields following these numerics and I'm trying to
figure out if there's some undocumented padding being used, or if I've got
the widths of some or all of the numerics wrong.

It'd be helpful in my analysis if I could look at the hex bytes and
recognize what's actually going on.  So I also would like to be pointed
toward an online or book source for the algorithm used to encode and decode
various COMP field types.  Or, perhaps the algorithms are simple enough for
someone to post here.

Can anyone enlighten me on either of these topics?

--Bob Grommes
  Phoenix AZ
```

```
Unisys A Series COBOL74 and COBOL84 implement COMP[UTATIONAL] as packed
decimal, that is, four bits containing decimal digits 0 through 9 per "9" in
the PICTURE clause.

Unisys A Series COBOL (68), on the other hand, implemented PIC 9(n) COMP (as
well as COMP-1) as native binary integer.  Where 1 <= n <= 11, a
single-integer resulted; for 12 <= n <= 23, a double-integer.  A canonic
single-precision integer occupies a 48-bit word; the sign appears in bit 46
(zero-relative), and the mantissa occupies from bit 38 to the low-order end
of the word, which is elsewhere zero. A canonic double-precision integer
occupies two 48-bit words; the low-order word is  as described above, and
the high-order word contains 13 in the exponent (downward from bit 44 for 6
bits).
```

```
But I have an alterior motive for posting.  I hope you can clarify something
for me.  Once you know how wide a data item is then you will need to know the
endianness of the item.  And that is where the old compiler/machine defaults
come in and the modern compiler comp-4 and comp-5 options can relate.

But on VMS machines, hardware wise, was there not actually an intrisic integer
item that was neither big-endian nor little-endian.  I would call it
transverse-endian.  

This related to 4-byte integers. There are two theoretical possibilities. I
think the actual machine pattern was to transpose the pair of 16-bit halfs,
low-order first, but within the 16-bit half the bits were high-order first.
This can be done the other way around (and still be neither big-endian not
little-endian).  It only relates to 32 bit data items.  It results from a
16-bit architecture being extended to a 32-bit architecture.

This was hardware dependent, not OS dependent.  I do not know if it could still
be evident in the COBOL generated file you are dealing with.  

Hope you can comment on this.
```
