question included in the end of the post.)

I think that the most important question (and there are some others) that
you need to ask - is:

Which is more important -
   portability across languages (but all on OS/390 - or at least S/390)
      or
   portability across platforms (including IBM OS, such as OS/2 - but also
all Unix and other proprietary OS).

If you are interested in "cross-language - but always on S/390" portability,
then I would (probably) go with the LE-services.   This would, for example,
get around the problem of COBOL (according to ANSI/ISO) using a different
"Lillian" date than IBM does (which is controllable by a COBOL compiler
option, but ...).  Furthermore, there are LOTS and lots of LE callable
services that have no corresponding COBOL features.  Therefore, if your
programmers could "get used to" using services, their training/maintenance
should be (?) easier.

If, on the other hand, you think your application will EVER go to any other
platform (including IBM's own OS/2 - much less AIX or Windows - and who
knows about USS), then you should avoid the LE callable services like the
"plague".  I once (in a former life) wrote a medium complete set of
LE-callable service routine emulators for Micro Focus - and it certainly CAN
BE DONE ... but there is no indication that IBM ever intends to provide a
complete set across their own platforms - much less in "open source code"
for you to port to other systems.  (Note initially - in the days of SAA and
AD/Cycle - there was supposed to be a difference between CEE3xxxx routines
and CEExxxx routines - but this is a "promise" that never came to fruition.)

    ***

It should be noted that as far as "different code" - "same results",  I
think it is (relatively) fair to assume that GENERALLY, you will (should?)
get the same results from using Intrinsic Functions and "native language"
and "callable services" - because the compiler (on OS/390) usually uses the
callable service in the object code that it produces.  Back in the days of
VS COBOL II (as compared to OS/VS COBOL) one of the big "selling points" was
that DB2, Fortran, *and* COBOL were finally set to all get the same results
from the same arithmetic expressions.  I think that today, this is something
that is FAIRLY safe to assume in whichever COBOL solution you go with.

   ***

One other issue to think about is the fact that the draft of the next COBOL
Standard includes support for USER-WRITTEN functions (and access to
non-COBOL functions).  Although IBM doesn't support these yet (as an
extension) and may or may not introduce them in the short-term, there are
many other COBOL vendors that already do support them.  My guess (crystal
ball) is that you will see them in COBOL sooner rather than later.  This
would be another reason to "stay with" functions rather than callable
services - as it provides a "standard" coding technique that can be
used/extended in the future (where the callable services could even be put
in a user-written/platform-dependent function).
