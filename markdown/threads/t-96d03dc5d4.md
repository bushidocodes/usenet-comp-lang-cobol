Hi:

----Are these data entry files flat?

A data entry file would probably be a ESDS VSAM file which is non-keyed and is usually treated as a flat file.  Around here and in a lot of places it is copied to a true flat file (ie QSAM) before processing.  In some shops there is real time update to the VSAM or database from on-line.   It all depends on the system etc etc.  For the sake of this discussion I'm using the first example.

 and create a VSAM I'd have to write a program (COBOL of course :D) to
write the records to the file.  Since these may not be in key sequence
I have to make sure the size of the VSAM file when defined allowed for
all the nasty splits such an operation could produce as the file is
being written to.  I have no idea if an ISAM file on a PC has such
things as CI or CS splits but VSAM on mainframes do and they cause loss
of space for the file.  Anyway because of all these splits processing
is slowed down.


----I have always been convinced that IBM wrote deliberately
inefficient software in order to sell faster hardware/more disks,
etc. E.g., ISAM vis-a-vis CICS

ISAM and CICS are not synonomous - not even in the ball park


>
> Now - sorting the flat file into key sequence and using a utility to
load the file makes a nice clean file and takes about 20% of the time.
Of course once the file is sorted it may not even be necessary to
create a VSAM file because probably all you're doing is a report and
now that the file is in order by the control breaks the program for the
report is COBOL 101 (for most students - we've seen examples of others
here).

----It does seem that your data-entry files are flat?  Anyway,
for example, would your data-entry method for an Accounts Receivable
system create flat files, or, would the entered data be posted
DIRECTLY to the A/R file thus eliminating any sorting or stuff?

Around here  both methods used for A/R.  In one case there is direct update to the A/R files which here anyway are Oracle databases on a Unix system with the programs written I think in C++.   The mainframe captures input for the A/R system via CICS to an ESDS VSAM file and FTPs the information to the Unix system after the data is processed for errors and reporting.
