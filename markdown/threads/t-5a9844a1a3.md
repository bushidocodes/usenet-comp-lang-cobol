For whatever reason, my employer has decided to use MicroFocus Visual
Object Cobol to build our intranet.  (if you're not laughing, please
continue on...)

I've run into a snag with it (being a very early 1.x version of the
product) where it will not deal with CGI input sent via "GET" method, or
as attributes of a HTML link.  Due to cost constraints, we will not be
upgrading to the newer (v3.x) suite.

So... I need to build that functionality myself.

Having little to no docs (musta been a $9.95 special we bought in the
bargain bin), I've found no method of exposing the Windows environment
variables to the cobol program, so I'm trying to access them via the Win32
api.

The problem is that a properly structured call to "GetEnvironmentString"
which works well under Watcom V10.0 C compiler, results in an undefined
link when run in Visual Object Cobol.  (my feeling is that Watcom is
hiding the guts of the API from the programmer, but I havent had time to
find out myself).

So - my question is - does anyone have any example COBOL code which
accesses the Win95 environment strings, in Microfocus Visual Object Cobol?

The easiest way to deal with environment variables is using the X/Open
extensions provided by Micro Focus.

DISPLAY "my_name" UPON ENVIRONMENT-NAME
ACCEPT my-var FROM ENVIRONMENT-VALUE


As for the the call to GetEnvironmentString, I can't find a function by
that name in the Win32 API. You can try GetEnvironmentStrings or
GetEnvironmentVariable. The former returns a pointer to the environment
block. The latter is used to retrieve the value of an environment
variable. Here's the code for the latter:

       IDENTIFICATION DIVISION.
       PROGRAM-ID.  GETENV.
       ENVIRONMENT DIVISION.
       CONFIGURATION SECTION.
       Special-Names.
           call-convention 74 is APIentry
            .
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  szComSpec           pic x(80) value low-value.
       01  iComSpecLen         pic 9(9) comp-5 value 0.
       01  iError              pic 9(9) comp-5 value 0.
       01  iErrorDisp          pic 9(9)        value 0.

       PROCEDURE DIVISION.
       MAIN SECTION.
           Move FUNCTION LENGTH(szComSpec) to iComSpecLen
           Call APIentry "GetEnvironmentVariableA" using
               by reference     z"COMSPEC"
               by reference     szComSpec
               by value         iComSpecLen
               returning        iComSpecLen
           End-Call
           If iComSpecLen > 0
               Display szComSpec(1:iComSpecLen)
           Else
               Call APIentry "GetLastError"
                   returning       iError
               End-Call
               Move iError to iErrorDisp
               Display iErrorDisp
           End-If
           STOP RUN.


I put together a chapter on writing CGI applications (using MF COBOL
4.0) in "COBOL Unleashed" from SAMS publishing. It's easy, and not
necessarily a bad way to go, although you may want to consider using
ISAPI if you've got an NT web server.

Scott,

There are Win32 demos provided with the Visual Object COBOL product so I am
sure you will get some useful background information on calling Win32
functions by taking at look at those. However, here is some quick
information to get you started :

1) use call-convention 74 for Win32 API calls

2) Use the CASE compiler directive

3) If parameters include one or more strings, add 'A' to the call name.

eg


$set case
  special-names.
    call convention 74 is win32.
procedure division.
    call win32 "GetEnvironmentStringsA" using ...

Note however, that if all you want to do is get specific environment
variables you do not need to make direct API calls, there is syntax provided
for that.

DISPLAY "vartoget" UPON ENVIRONMENT-NAME
ACCEPT val FROM ENVIRONMENT-VALUE

I hope this helps.
