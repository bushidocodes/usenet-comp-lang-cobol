I am trying to write a COBOL subroutine that will take some input and return
some output.  The input has some manditory and some optional parameters. 
Depending on the optional input some of the output may or may not be
created. 

I want to make the input and output flexible enough that I could add an
input option and/or an output field without requiring recompiling *all*
CALLING programs; specifically those that do not care about the new fields.

The CALLED subroutine will always be dynamically linked.

What would be the best way to do this that would allow me to not recompile
each caller but would also allow be to not read from or write over the
calling programs working-storage?

As an example, take the following copybooks:

* GETACCTI
01  GET-ACCT-INPUT.
    05  GAI-ACCT-NBR        PIC 9(9).
    05  GAI-NAME-ADDR-SW    PIC X.
        88  GET-NAME-ADDR       VALUE 'Y'.

* GETACCTO
01  GET-ACCT-OUTPUT.
    05  GAO-CURR-BAL        PIC 9(11)V99.
    05  GAO-AVAIL-BAL       PIC 9(11)V99.
    05  GAO-NAME-ADDR.
         10  GAI-NA         PIC X(40) OCCURS 5 TIMES.

ID DIVISION.
PROGRAM-ID. CALLTEST.
DATA DIVISION.
WORKING-STORAGE SECTION.
01  GETACCT    PIC X(8) VALUE 'GETACCT'.
COPY GETACCTI.
COPY GETACCTO.
01  MORE-WORKING-STORAGE    PIC X(80).

PROCEDURE DIVISION.
    INITIALIZE GET-ACCT-INPUT
    MOVE 123456789 TO GAI-ACCT-NBR
    DISPLAY 'NAME/ADDR? ' UPON CONSOLE
    ACCEPT GAI-NAME-ADDR-SW FROM CONSOLE
    CALL GETACCT USING GET-ACCT-INPUT GET-ACCT-OUTPUT
    IF RETURN-CODE = ZERO
        DISPLAY GAO-CURR-BAL ' ' GAO-AVAIL-BAL
        IF GET-NAME-ADDR
            PERFORM VARYING SUB FROM 1 BY 1
              UNTIL SUB > 5
                IF GAO-NA (SUB) > SPACES
                    DISPLAY GAO-NA (SUB)
                END-IF
            END-PERFORM
        END-IF
    END-IF
    STOP RUN.

END PROGRAM CALLTEST.

ID DIVISION
PROGRAM-ID. GETACCT.
DATA DIVISION.
WORKING-STORAGE SECTION.
COPY DMDMSTR.
COPY SAVMSTR.
01  AVAIL-BAL           PIC 9(11)V99.
01  NAME-ADDR               VALUE SPACES.
    05  NA              PIC X(40) OCCURS 5 TIMES.

LINKAGE SECTION.
COPY GETACCTI.
COPY GETACCTO.

PROCEDURE DIVISION USING GET-ACCT-INPUT GET-ACCT-OUTPUT.
    INITIALIZE GET-ACCT-OUTPUT
    PERFORM GET-DMDMSTR
    IF GET-SUCCESSFUL
        MOVE DMSTR-CURR-BAL TO GAI-CURR-BAL
        PERFORM CALC-AVAIL-BAL
        MOVE AVAIL-BAL TO GAI-AVAIL-BAL
        IF GET-NAME-ADDR
            PERFORM GET-DMSTR-NAME-ADDR
        END-IF
        MOVE ZERO TO RETURN-CODE
    ELSE
        MOVE 12 TO RETURN-CODE
    END-IF
    EXIT PROGRAM.

END PROGRAM GETACCT.

Now imagine that I want to add a new output field that is based on the
setting of a new input field.  Like this:

* GETACCTI
01  GET-ACCT-INPUT.
    05  GAI-ACCT-NBR        PIC 9(9).
    05  GAI-NAME-ADDR-SW    PIC X.
        88  GET-NAME-ADDR       VALUE 'Y'.
    05  GAI-CASH-RSRV-SW    PIC X.
        88  GET-CASH-RSRV       VALUE 'Y'.

* GETACCTO
01  GET-ACCT-OUTPUT.
    05  GAO-CURR-BAL        PIC 9(11)V99.
    05  GAO-AVAIL-BAL       PIC 9(11)V99.
    05  GAO-NAME-ADDR.
         10  GAI-NA         PIC X(40) OCCURS 5 TIMES.
    05  GAO-RSRV-INFO.
        10  GAO-RSRV-LIMIT  PIC 9(7).
        10  GAO-RSRV-BAL    PIC 9(7)V99.


If I changed the two copybooks but did not recompile CALLTEST then obviously
my attempt to read GAI-CASH-RSRV-SW would get me junk and writting to
GAO-RSRV-INFO would over-write CALLTESTs MORE-WORKING-STORAGE field.

Obviously I could do something like adding two length fields as part of the
parameter list.  But isn't there something simpler?  I tried declaring
GET-ACCT-INPUT and GET-ACCT-OUTPUT as EXTERNAL and not passing them as
parameters, but when I changed GETACCTI and GETACCTO and recompiled GETACCT
but not CALLTEST I got a run-time error that external storage length did not
match.  Which, of course, makes sense.  I was just hoping it might be extra
smart about this, but really I don't see how it could be.

Note, I do not have any access to OO COBOL, so while that would probably
make this quite simple, I have to do with standard ANSI 85/89 COBOL
(specifically COBOL for VSE/ESA).

Thanks!

When you CALL a subroutine, you do not pass parameters, you pass addresses 
to the parameters in the calling program. That is, the CALLING program 
passes ONE address for each parameter in the calling list. In your scenario, 
you may augment the copy books at will and, if the uncompiled calling 
programs never bump into the new fields, the old programs need not be 
recompiled or re-linked.

Specifically:
Pgm A.
01  Input-Group.
  02  Op-code   pic x.
  02  Input-A pic xxx.
01  Output-Group.
  02  Output-1  pic xxx.



Pgm B
01  Input-Group
  02  Op-code  pic x.
  02  Input-A pic xxx.
  02  Input-B pic xxx.
  02  Input-C pic xxx.
01  Output-Group.
  02  Output-1  pic xxx.
  02  Output-2  pic xxx.

In both Pgm A and Pgm B is found the statement:

  CALL 'SUBPGM' USING Input-Group Output-Group

It is up to SUBPGM to make sure it gets stuff from Input-Group and puts 
stuff in Output-Group properly, that is, according to Op-Code. For example, 
assume Pgm A processes Op-Codes "A" "B" and "C," each of which take as input 
a single value (Input-A) and return a single value (Output-1). Pgm A *MUST* 
only pass Op-Codes A, B, and C.

In Pgm B, however, can pass Op-Code "D" which takes THREE inputs and returns 
TWO results.

In developing Pgm B, you need not be concerned with anything associated with 
Pgm A save for the Op-Code idea.

In MY opinion what you have now is two separate requests - get current 
balance and get paid interest.   As far as I am concerned you have 5+ 
options:

1. In your new module call one program to get the balance and another 
program to get the interest (NO recompile of old stuff)
2 .In your new module call one new program that gets interest (which in turn 
calls get balance for the balance portion)  (NO recompile of old stuff)

Now the advantages? You have very clearly defined what each function does - 
you do not need to read the code to know sometimes it does balance only, 
sometimes it does interest as well.  If for example you were calling the 
module - you'd have to look at both module each time to figure out which 
ones did what.

3. Make the parameters be "variable" and outputs "variable"
4. Recompile

If your program is truly common then all calling programs should get back 
accrued interest (none if there is none).  This means only a recompile - I 
find very few legitimate reasons why a recompile is a real issue...but a 
recompile makes the code very clear - I don't like explaining why I don't 
pass parameters, and what is the default anyway ? All of a sudden a calling 
program loses control of the called program.  By recompiling ANY program 
calling this module will get back both the balance and interest...and IT can 
determine whether the returns have value or meaning.

If you change the CALLED module, the fact that you don't change the CALLING 
module means that this will often NOT get retested with the new "default" 
parameters.  In a financial industry that's very risky - especially with a 
SOX (Sarbanes Oxley)  guy walking around.

5. Use static links - that way you will have the code appropriate to the 
link time
This is the worst as you would have to refer to the listing versus the 
source when reviewing.

There are 5 solutions here...you ruled out the last.  You started with the 
third.  I prefer the first and second...Only you know your environment.  The 
factors to consider are - how many recompiles are we talking? What is the 
quality of your test repeatability (the better the less intrusive a 
recompile but you should be testing anyway)? how often do you find the 
requirements for the "common module" to change ? How _many_ options will 
there be in that one module before you think it gets too much? How often is 
the called vs calling changed? How much employee turnover do you have? How 
good is the system documentation? Which is easier to figure out on short 
notice when you have a problem - the one with specific parms or the one that 
has "options"?

In short, 3 is acceptable to me, and in _your_ specific example might 
actually be the best solution but I wonder why you cannot just have a 
"rebuild all" function for your system.

Call GETACCT-INTERFACE  with INPUT OUTPUT where INPUT contains an ID "such

In program GETACCT-INTERFACE set the addressability of the INPUT field to

In program GETACCT-INTERFACE set the addressability of the OUTPUT field to 
the appropriate Output Structure and copy in the GET-ACCT-IMPLEMENTED
