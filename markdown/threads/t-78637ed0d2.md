I am afraid that this is one of those "unanswerable" questions.  It certainly
will depend upon your workload, hardware, etc.  You mention "how long it will
run" - I assume this means that you are interested in "clock time" and not
CPU time.

My personal suggestion would depend upon whether your file layouts (or actual
files) already exist or not.

If the files exist (or can be created in a test version), then I would
suggest that you write a "dummy program" that simply  opens all the files
then loops thru reads/writes of the appropriate files, and closes them.  Then
run this job (not impacting any product files OF COURSE) at the same time
that you plan on running your final program (and on a machine with its normal
work load) -
   and see what happens.

If you are really concerned with IO take the following simple rules:

a)  Do not open and close datasets, it is very expensive.

b)  If your output is sequential and you use a lot of it, use BUFNO. 
I use BUFNO =50 to good effect on some of my runs.

c)  If you are using VSAM clusters look up Batch LSR's,  on bad code I
reduced run time from 60+ minutes to 10.  One reasonable code I got
about a 30% reduction.

If you want to really track this I can recommend a product called
STROBE.  It is pretty pricey, if you need to make it sing then it is
your tool.

If you have simple monitoring requirements I have a program that works
over SMF dumps and pulls some basic information, basically for
charging algorithms. SMF can give you step timings for example. It is
surprising when user pays how many tune ups happen to code.

Time is generally based on 3 things.  How fast is the machine running?  How
fast is the program?  and how much data am I going to process?  More
importantly, how much work am I doing?

You can't control the speed of the machine really because there are a lot of
factors.  Unless you're also the IT manager, you won't magically get extra
disk space or a faster CPU.

You CAN control the speed, or efficiency of the program.

Expect that the amount of data can vary widely.

The time a batch program takes is probably something like:

    ( Amount of Data * Speed of Program ) - Other Factors

Other Factors being the speed of the computer, the current workload, amount
of work space available, etc.

What I'm getting at is the amount of time to do any of the 4 actions you
listed, one time, is unimportant.  It all depends on HOW MANY TIMES you
perform those actions.

I remember learning about Big O analysis in college.  The amount of time to
do any one instance of an action is unimportant.  A good program will
perform at O(n) or possibly faster I guess, where the time is equal to the
number of iterations.  An ok program will be at O(nlogn) or O(2n) or
something like that.  Bad programs get exponentially slower as more work is
done, and eventually grind to a halt.  Like O(n cubed).

The best way to estimate the time batch programs will run is put them in a
test environment, put some data in there, and run em.  Then, do the math to
figure out how much data you coud have and the amount of time the programs
will run.

Then, mulitply that by 2.
