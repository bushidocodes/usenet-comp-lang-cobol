```
embedded wild cards also.

Example:

User wants to see all part numbers that match his input:  312***59-*
(where '*' is any single character).

Part numbers from a sequential master file that match are:  312A2C59-1
```

```
Allowing for a single character wildcard is fairly simple...

SET ITS-A-MATCH TO TRUE.
PERFORM VARYING ST FROM 1 BY 1 UNTIL ST > 20
                                                                  OR
NOT-A-MATCH
   EVALUATE TRUE
       WHEN SELECT-FIELD (ST:1) = '*'
       WHEN SELECT-FIELD (ST:1) = PART-NUMBER (ST:1)
             CONTINUE
        WHEN OTHER
             SET NOT-A-MATCH TO TRUE
   END-EVALUATE
END-PERFORM.

That should get you on your way....

In your example, you actually allow the match against one or more
characters,
depending on whether it falls at the end or not.  While this works, I would
suggest
that you choose one or the other, or allow for another wildcard character.
IE-
'%' = single character wildcard, '*' = multiple character wildcard

Allowing for imbedded multi-character wildcard is a bit more difficult, but
still very
possible...
```

```
returns a flag that indicates match or fail.  If the product code allows
```

```
You could save more iterations with more "up front"
preparation. Consider the following:

       prepare-pattern.
           move 0 to first-group-length
           move 1 to second-group-start
           inspect user-pattern
               replacing all space by "*"
           unstring user-pattern
             delimited by all "*"
             into start-key
             count first-group-length
             pointer second-group-start
           end-unstring.

Assuming the part numbers do not have spaces, the
above uses UNSTRING to extract the leading characters,
provides a count (not really needed) of the number of
significant characters in the first group, and points to the
start of the second group.

       prepare-keys.
           inspect start-key replacing all space by low-value
           move start-key to end-key
           inspect end-key replacing all low-value by high-value.

The the above code sets the lowest and highest keys for
comparison with the record key. Any value greater or equal
to start-key and less or equal to end-key is selected for
matching.

       pattern-match.
           set patterns-match to true
           perform varying x
             from second-group-start by 1
             until x > 20 or patterns-do-not-match
               evaluate true
                 when user-pattern (x:1) = "*"
                 when user-pattern (x:1) = work-pattern (x:1)
                   continue
                 when other
                   set patterns-do-not-match to true
               end-evaluate
           end-perform.

Notice the in-line perform begins from second-group-start
saving comparison against all leading positions. If user-pattern
has only a leading group, second-group-start will be 21 causing
the perform to terminate with no comparisons. Otherwise, the
character-by-character comparison will begin with the second
group of significant characters.
------------------
Rick Smith
```

```
         unstring match-field delimited by match-delimiter (1:del-size)

            with pointer field-ptr
         end-unstring

otherwise, we have no match
         if check-delimiter (1:del-size) not = match-delimiter (1:del-size)

        end-if

   end-perform.

   if its-a-match

I haven't tested this, but it should work to allow your wildcard character
to be one or more characters.
Anything starting with '**' is a comment, not actual code.  Sorry if it is
hard to read....

This routine *should* allow you to use a pattern (select-field) that
includes '*' as wildcard (even with imbedded spaces) and assumes that you
don't want to match the trailing spaces.  It actually looks for the
non-wildcard portions in the match-field, ensuring that they are in the
match, in the same order as the pattern has them.  If your pattern ends in a
wildcard, and the routine has found everything else, it assumes a match.  If
it does not end in a wildcard, then it makes sure that the rest of
match-field is spaces, otherwise, it is not a match.

Just as a note, hex '5c' is the '*' on a mainframe.  I'm not sure about the
PC.
```
