```
We are converting most of our COBOL for OS/390 programs to use LOCAL-STORAGE
instead of WORKING-STORAGE.  In a static link of about 800 programs, we've
cut down our overall storage use by 6MB (about 80%) and cut CPU time by
about 20%.  This is because local storage is made available to other
programs on return from a call, rather than being retained as part of the
program's "last-used state", with the resultant serial search each time a
program is called.
My problem is in my condition handler routine, which formats an "intelligent
mini-dump", including control blocks and other areas that are specific to
our environment, as well as key COBOL areas.  I can find working storage
easily enough (start of working storage is a TGT+X'114', then add 4k until
you've exhausted the length at TGT+X'68'), but the BLK cells and local
storage itself are in the variable part of the DSA and the location (like
many other things) is not documented in any of the LE or COBOL for OS/390
manuals.  They are mapped in the program listing, of course, but that's no
help at run-time.
There is obviously a way to find the BLK cells, because they are displayed
in an AbendAid dump.  Does anyone know how to find them?
D de J
```

```
No. For local storage, results are unpredictable after a call to another
program.  Suppose program A calls program B and program C.  After the call
to program B, the contents of program B's local storage may still be valid,
though I'd say that was unlikely as the COBOL termination routine (IGZETRM)

No.  The maximum space used is the total used in the calling chain with the
largest stack, rounded up to the next allocation level.
> So what do they get from using local storage on the static call?  What
> accounts for the storage and CPU savings?  I thought the CALL/CANCEL did
> most of that...
Every time you call a re-entrant COBOL subroutine, the COBOL space manager
(IGZESPM) searches the storage chain index (serially) to see if storage has
already been allocated for this program.  If so, it sets the pointer to the
existing storage.  Otherwise it allocates a new area and adds it to the end
of the chain. Stack storage saves the address of the NAB (next available
byte) in each stack frame, so there's no search requirement.

WORKING storage, reply to this thread.  If you know the answer to the
```

```
When a CANCEL is done the whole subroutine's memory [may] become
available for use by other programs, though probably not when statically
linked.

Local-Storage (in my systems) in a sub-program is created anew for each
CALL and is released when that program does an EXIT PROGRAM, even when
static linked.  It may be created on the stack, or in a 'heap'. It is
vital that the only assumption made about local-storage is that it
_will_ be junk values when the sub-program is entered.   

In re-entrant programs (such as under Windows) each new entry creates
another local-storage (which is why it is stack based).
```
