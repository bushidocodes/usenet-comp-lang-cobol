I wrote a quick and dirty little program this morning to convert a SEQUENTIAL file to a LINE SEQUENTIAL file so that it would be trivial to upload to the mainframe (while still doing an ASCII->EDCIDIC conversion via FTP :) 

The program will NOT run under Visual Age COBOL! It consistently returns a File Status error of 39 (incompatible file). 

For the heck of it, I compiled the same source code with Microfocus NetExpress and under AcuCOBOL. 
Both of them worked, but even though the output file compares exactly using SUPERC from SPF, the files are more than 2 megabytes different in size!  Duh...

Is this just some weird Windows thing going on, or am I really loosing my mind? Opinions on the first speculation welcome... <grin> 

The input file is just a stream of data written in 416 byte records, nothing unusual about it that I know of. 
Code and data sample included below for your amusement. ;)  The filenames in the code are hardcoded because this is just a one time program... 

-Paul

--------

      **********************************************
      * CONVERT EXPORTED SEQUENTIAL FILE TO
      * 416 BYTE LINE SEQUENTIAL FILE
      **********************************************
       IDENTIFICATION DIVISION.
       PROGRAM-ID. CVT001.

       ENVIRONMENT DIVISION.
       INPUT-OUTPUT SECTION.
       FILE-CONTROL.

           SELECT INFILE
             ASSIGN            TO "PA.DAT"
             ORGANIZATION      IS SEQUENTIAL
             ACCESS MODE       IS SEQUENTIAL.

           SELECT OUTFILE
             ASSIGN            TO "PA1.IMP"
             ORGANIZATION      IS LINE SEQUENTIAL.

       DATA DIVISION.
       FILE SECTION.

       FD INFILE
           RECORD CONTAINS 416 CHARACTERS
           RECORDING MODE IS F.
       01 INFILE-RECORD         PIC X(416).

       FD OUTFILE.
       01 OUTFILE-RECORD        PIC X(416).

       WORKING-STORAGE SECTION.
       01  EOF                  PIC X(1)  VALUE 'N'.
       01  RECCOUNT             PIC 9(10) VALUE ZEROS.


       PROCEDURE DIVISION.
       START-PROGRAM.
           DISPLAY 'CVT001 PROGRAM IN PROGRESS'
           DISPLAY 'OPENING INPUT FILE...'
           OPEN I-O INFILE
           DISPLAY 'OPENING OUTPUT FILE....'
           OPEN OUTPUT OUTFILE
           DISPLAY 'LOADING FILE NOW...'
           MOVE ZEROS TO RECCOUNT
           READ INFILE
             AT END MOVE 'Y' TO EOF
             END-READ
           PERFORM UNTIL EOF = 'Y'
             ADD 1 TO RECCOUNT
             MOVE INFILE-RECORD TO OUTFILE-RECORD
             WRITE OUTFILE-RECORD
             READ INFILE
               AT END MOVE 'Y' TO EOF
               END-READ
             END-PERFORM
           DISPLAY 'LOADED ' RECCOUNT ' RECORDS.'
           DISPLAY 'CLOSING FILES'
           CLOSE INFILE
           CLOSE OUTFILE
           GOBACK.

The problem is in your assumption of the physical structure of SEQUENTIAL
file.  In VA COBOL, SEQUENTIAL files are support by the STL filesystem,
Btrieve, or workstation VSAM.  Typically, files in these filesystems will
contain header information about the file: organization, record length, etc.
Trying to open a file that wasn't created specifically by the filesystem
you're using will invariably lead to file status 39.  The exception is LINE
SEQUENTIAL, which is supported by the native byte-stream files of records
terminated by CRLF's on Windows.

If the file consists of 416 byte records, you don't need to even convert it,
do you, before you ftp it to the mainframe?  Just add the CFLF at the end of
each line.

Al...

In your SEQUENTIAL file the records are all physikally the same size
while LINE SEQUENTIAL replaces trailings spaces by a single record
delimiter.

With a logical record length of ten, the input

"foo bar   "
"foobar    "

becomes

"foo bar<"
"foobar<"

where "<" represents the record delimiter (x"0D0A" in Win
environments). In the example above your conversion saved 3 bytes (5
on Unix). If SUPERC supports binary diffing, try that...

Sven

There are also many other issues.  eg if the file contains control
characters (or BINARY or PACKED data) and these characters may be
prefixed with a null character on output (depending on vendor and
options) making the line sequential larger, and/or blocks of spaces may
be replaced by tab characters making this smaller.

OTOH the sequential file may cater for records of different sizes
(depending on vendor/options) and thus have a header block and record
length/pointer headers and thus the line sequential becomes smaller.

And then again some vendors may have options to transparently use
compressed data in files (not in LINE SEQUENTIAL).  I have reduced 6
MByte of data to a 1.4 Mbyte file using this.

Of course some of these may result in file comparisons failing unless
the same file techniques are used.
