yauyau <member23982@dbforums.com> wrote 

> I choose to use 'contained subprogram' because of localization and
> reusability. I want to show that only "CALCULATE" can CALL "ADD" and
> "MINUS", while "PRINT" cannot.
 
In this poarticualr case nested programs do not help re-usability -
they cannot be called from a different program. They _might_ help
localisation.  For example if 'PRINT' did want to call 'ADD', but a
quite different ADD that did something completely different then it
could have its own local 'ADD'.

If PRINT doesn't CALL "ADD" then why bother making it so that it
cannot ?

If CALCULATE is the only program that will ever call that 'ADD' then
why have it as a subroutine at all ?  Have it as a paragraph that is
PERFORMed, quicker, easier to write, and identical in practical terms
to what your code will achieve in 'reuability (performed from several
places) and locaization (exactly what private data will the subroutine
have?)'.

Having separately compiled and linked sub-routines (not nested) allows
them to be reused by another program without having to cut-and-paste
the code or put it is a COPY book, without having to recompile it
even, CALL it and it will load (in an appropriate environment).

There is often a conflict between 'localization and reusability' -
choose one. Still, they are good buzz words to chuck around.
