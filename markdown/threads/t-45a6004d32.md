```
As a follow-up on the "symbols" thread, I thought that I would post the
"problem" (for upward compatibility) in introducing new symbols as
COBOL-defined "words" (e.g. as alternatives to AND, NOT, and OR)

Consider the following statement in an '85 Standard compiler that has an
EXTENSION allowing non-COBOL characters in pseudo-text

 Replace  ==!== by ==Not==
               ==!-True== by ==Never-True==.

 01 Switch-Field  Pic X.
      88  !-True  Value "N".
      88 Not-True  Value "n".

     ...
If !-True and !  A > B
     ...

Now, let's consider how this source code is "processed" when "!" is and is
not defined explicitly as a COBOL word (if such a change were made in the
future).

When "!" is *not* a COBOL reserved word (i.e. a defined symbol for "not")
the source gets converted to:

01 Switch-Field  Pic X.
      88  Never-True  Value "N".
      88 Not-True  Value "n".

     ...
If Never-True and !  A > B

but when/if  "!" *were* a COBOL reserved word (i.e. a defined symbol for
"not") the source gets converted to:

01 Switch-Field  Pic X.
      88  Not-True  Value "N".
      88 Not-True  Value "n".

     ...
If Not-True and !  A > B

        ***

Bottom-line:
   Be careful, VERY CAREFUL, when you consider adding new "symbol reserved
words".  Although this won't (shouldn't?) impact code developed for
"strictly conforming" compilers that do NOT allow non-COBOL characters in
pseudo-text, it will be POTENTIALLY disastrous to those programs developed
for environments with such an extension.

NOTE:
  This is *not* a "hypothetical" situation that has never occurred.  The
introduction of ":" as a text-word in the '85 Standard most certainly *did*
impact a variety of existing applications.

P.S.  As someone who has worked closely with compiler developers, changes to
"text-words" or anything else that impact COPY/REPLACING or REPLACE
"processing" are always frightening.  In many compilers, this is some of the
most "complex" and "exception driven" code.  Especially when it comes to
some of the rules that apply differently to REPLACE and COPY/REPLACE (such
as the ability to use "identifiers" instead of pseudo-text), changes in the
definitions need to be incredibly carefully considered - before
(unintentionally) impacting existing code and processing.
```

```
REPLACE allows only pseudo-text to begin with, and though the use of "text"
(as distinct from "word") in COPY ... REPLACING is still supported, it's
marked as archaic in ISO/IEC 1989:2002.  I wouldn't recommend changing
*anything* beyond that which was necessary to allow some extra capabilities
in pseudo-text-1, and those would be very limited indeed.
```

```
I like that. Here's a subroutine for inserting:

PROCEDURE DIVISION USING
   LINK-STRING
   LINK-STRING-LEN
   LINK-INSERT
   LINK-INSERT-LEN
   LINK-LOC.


*>  Move "Tail" of receiving field to new location

   MOVE LINK-STRING-LEN TO J.
   PERFORM VARYING I FROM BEG BY -1 UNTIL
      I < LINK-LOC
      MOVE LINK-STRING(I:1) TO LINK-STRING(J:1)
      SUBTRACT 1 FROM J
      IF J < 1
         MOVE 1 TO J
         END-IF
      END-PERFORM.

*> Move insert-string to area opened up

MOVE LINK-INSERT(1:LINK-INSERT-LEN) TO
   LINK-STRING(LINK-LOC:LINK-INSERT-LEN).

      GOBACK.
```
