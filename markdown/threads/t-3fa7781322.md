Hi,

I have never done what you are attempting.

AcuCorp have a product called AcuServer which will do what you want
much better than mapped drives.

AcuServer is a server program that accepts COBOL IO requests from clients,
processes them on the server and returns the results to the program that
requested
the it.

This cuts the network IO to one send / receive per request.

There is obviously a cost involved in purchasing AcuSever and a little more
configuration (not much!) but the results will be greatly improved and scale
much better.

The great thing about all this is you can try it mapped drives and if it
doesn't
work very well you always have the option to use AcuServer.  I don't think
you even need to recompile your program - just setup a config file to say
you
want to use Acuserver (you will also need an AcuServer enabled runtime which
from memory I think comes with the AcuServer "server").

Ray Smith
(Please Note:  I am in no way connected to AcuCorp - just a happy user)

Dear Dennis.

Acucorp has many customers that access Vision file from a mapped drive using
a NFS product with no problems.
But, Acucorp does not support products in this environment as NFS use their
own record locking and file caching, which can result in problems with
accessing the data or even file corruption.

To prevent the above, Acucorp recomends that you do not use NFS products,
but instead use AcuServer that will allow full locking and caching abilities
from within the Acucorp products.

We have a few customers who do this.  There are some settings in the Samba
config that you need to play with to get file locking to work reliably and
to avoid file corruption.  If I recall correctly, we needed to set "oplocks
= false" and "blocking locks = false" in the smb.conf file.

You may also need to set "read raw = yes" and "write raw = yes" but try
without it first.  And make sure you have the latest Samba.
