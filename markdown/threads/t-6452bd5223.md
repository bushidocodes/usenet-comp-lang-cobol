```
If I understand correctly, you wish to statically link the runtime code
contained in a DLL into your application, instead of shipping the DLL with
your application.

The short answer is:  You can't do it.

The entry points in a DLL are generally called statically by your program.
Your program gains access to these entry points by being linked with an
import library derived from the DLL.  In the 16-bit world, you could build
an import library using the utility IMPLIB.EXE.  In the 32-bit world, the
LINKer creates an import library automatically when the DLL is built.

The import library merely provides a mechanism that will, at runtime, load
the DLL when you call one of the entry points exported by the DLL, and pass
control to the DLL.  That's why import libraries are so small; they're just
pass-through code.  The real meat resides in the DLL.

There are a number of attractive points to using the DLL approach:

- Only one copy of the DLL is loaded into memory; the code may be used by
many applications.  Each application gets its own copy of the DLL's data
segment, so there are no conflicts or reentrancy problems.

- From the software vendor's point of view, you can (generally) distribute
bug fixes and upgrades in DLLs and not require that the developer re-LINK
the application; all your have to do is to replace the old DLL with the new
one and, Voila!  you're running the latest code.  It's much less trouble for
all concerned.

- For our ScreenIO application, we only need to maintain one version of the
software (in DLL form) that may be used by any 32-bit compiler.
Compatibility issues are addressed using simple COBOL stubs (compiled using
the target compiler) that are placed in the import library distributed with
the runtime DLL.

Of course, there are downsides...

- Your installation software has to be careful not to clobber newer versions
of DLLs by installing older versions over the top of them.

- If your application terminates abnormally, and you're testing code
contained in a DLL, you may have to reboot to (metaphorically speaking) kick
a loaded DLL out of memory in order to replace it.

But I digress...  Here's the bottom line:

If the software vendor provided an object library containing the runtime
routines to you, and you linked your application with that library (instead
of the import library), you would be able to bundle the whole application
into an EXE without the need to ship runtime DLLs.  If the vendor doesn't
provide the runtime routines to you in this form, you're going to have to
ship the vendor's runtime DLL files.
```
