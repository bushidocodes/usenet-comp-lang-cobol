Agreed.  I haven't nominated the behemoth that I worked on earlier
this year for that reason.

My behemoth is really a couple dozen programs.  We just refer to it as
Report 57.  The begining program is a small "traffic cop" program,
that calls five 50,000 to +100,000 line programs, with about a dozen
smaller programs that are called from any of the calling five.  And
those dozen may call other programs.

The five biggest would only be anywhere from 3500 to 9500 lines of
code uncompiled, but 80 data copybooks and 20 procedure copybooks
cause each program to bloat very quickly.  (BTW, with 80 data
copybooks, 88 levels are a nightmare).

And it continues to get smaller, as procedure division copybook
routines get replaced with called subroutine programs.  But then
again, new code is always added to it.

It's still a real bitch to work with.

Moving to more modular structures has solved some problems, but it
created some new ones.  All calls are static.  You forget to recompile
a key subroutine, in the correct order, and unpredictable results
occur.  Sometimes it crashes, sometimes the data is wrong and you're
beating your head against the wall, trying to figure out why.

And BTW, I heard about the monster at FDR, when I contracted there,
but fortunately, I never worked on it.
