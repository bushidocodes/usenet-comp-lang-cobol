I've got a question for you OO folks out there.

Assume I have an object, and there are two "constructors" for it.  One 
takes no parameters (which creates a "new" instance), and one takes a 
single parameter, which represents the ID of this already-established 
object (whose properties are stored in a database under the key of this 
ID).

My question is this - is is apropos for the ID-receiving constructor to 
fill private variables representing the properties for this instance, or 
should the "get" methods actually each hit the database for the value 
based on the ID I gave when the object was instantiated (i.e., "getName" 
and "getAddress" would each result in a database query instead of 
returning the value of a private variable.)  I believed the former to be 
a bit better, but when I bounced my design off someone else, they said 
the latter was the way to go.

I know we've got a pretty good base of OO users here, so I thought I'd 
ask the question.  Thanks!

CustomerDBI and ChartDBI inherit from DBIMain. As you create instances

record-length to DBI Main - keeps a separate copy for each  List.
Also passed for each instance of DBIMain, is a reference to the COBOL 
filetype or DB Table.
Regardless of database type, DBIMain does the following :-

- "createcollection"
- "addToCollection"
- "changeCollection"
- "deletefromCollection"
-  "getKeyFromIndex"
-  "getIndexfromKey"
- plus others...........

So from Business Logic, when the Listbox selection event is returned, I 
invoke either os-CustomerDBI or os-ChartDBI with the method 
"getKeyFromIndex" - starting at the *bottom* of the inheritance 
hierarchy, this method completely bypasses the two sub-classes and takes 
action on the method in DBIMain.

It seems to me that you are describing two objects with different
behavior (in the second case).  The object created with no parameters
causes no database hit. The object created with the database key is
initialized in the first case, but linked to the database record in
the second case. 

It really depends, on what you want, but it seems to me to be "purer
OO" to define two classes with the same interface.  For example:

EmployeeRecord is the interface that describes how you manipulate
these objects. Maybe, for simplicity's sake, this is the abstract
parent class.  You have two subclasses, SimpleEmployeeRecord and
LinkedEmployeeRecord.  Access methods for SimpleEmployeeRecord set and
get object members.  It has a default constructor and one that loads
its data from the database once. Access methods for
LinkedEmployeeRecord do I/O on the database. Most of the code that
uses EmployeeRecord doesn't need to know which type you have.

You may not need both kinds of objects, but I think they really should
be different classes.
