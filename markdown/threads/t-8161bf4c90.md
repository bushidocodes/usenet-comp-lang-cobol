```
Just a follow-up on another thread.

There is at least one major "disadvantage" to using

  Compute A = B
    On size Error Display "B is too big to move to A"
  End-Compute

as compared to

If B > 999....99  *> whatever the PIC is of  A
   Display "B is too large"
Else
   Move B to A
End-IF

In pre-02 Standard COBOL the results of the COMPUTE statement are totally 
implementor defined.  A conforming implementation may truncate BEFORE 
testing for ON SIZE ERROR or do just about anything they want.  Therefore, 
the COMPUTE *probably* - but not necessarily will detect the "truncation" 
issue.

I am away from home now - so I don't have my copy of the '02 Standard.  I 
know that when STANDARD arithmetic is in effect, the '02 Standard CHANGED 
the rules for COMPUTE with a single identifier or literal as the sending 
operand - so that it was a "move" - but I can't remember if that also is 
true for NATIVE ARITHMETIC (as a substantive change).
```

```
There was an interpretation of exactly what "implementor defined" means for 
arithmetic expression in the '85 Standard.  The bottom-line is that the 
implementor may do ANYTHING they want.  (Produce random numbers, return the 
day of the week, WHATEVER).

In fact, a "widely known" implementation did fractional exponentiation by 
FIRST truncating to a decimal point and then calculating the results, e.g.

 Compute whatever = num-field  ** .5

was the same as

 Compute whatever = num-field ** 0

(talk about incorrect results !!!)

That compiler was conforming and even passed the NIST conformance tests.

Of course (eventually) they fixed their results.

***

As far as "more real world" cases go,  there are a number of implementation 
with an EXTENSION of "NOTRUNC" where binary item can be larger than the PIC 
would allow - but that fit in the same storage.  I know of at least two 
different ways in which

Compute Display-Numeric = True-Binary
    On Size Error

and

Compute True-Binary = Display-Numeric
    On Size Error

are implemented - and in some cases the "ON SIZE ERROR indicates that the 
value won't fit into the receiving field and in other cases where it won't 
fit into a data item that matches the PICTURE.

BOTTOM-LINE:
    COMPUTE results are NOT portable or predictable in the '85 Standard and 
are ONLY predictable in the '02 Standard with ARITHMETIC is STANDARD (and 
for some - not all - cases)
```

```
The bottom line on this is:

Use arithmetic when you need to do arithmetic.
Use MOVE when you need to do data manipulation.

Because of the English-like nature of COBOL these boundaries sometimes
become blurred. It is easy to lapse into "COBOL-think"  and believe you are
using the language effectively, when  in fact, you had lost sight of what
the problem was really about.

There is always added complexity when an arithmetic solution is chosen,
because there are considerations such as data conversion and decimal
alignment that get introduced into the equation. So, arithmetic solutions
generally require more care.

I don't think it is really that important, provided the solution works and
the code is clear.

language standards leaves me gasping <G>.
```

```
That passage says "the exact VALUE"; it doesn't say what format the
value is in. When standard arithmetic is being used, the format is
SIDI (per 8.8.1.3.1). When native arithmetic is being used, the
passage says the value must be stored exactly. 

There is a potential problem when the implementor is using binary
floating point for intermediates. In that case, digits right of
decimal that cannot be represented as the sum of powers of two will
have their value changed. The result can be:

01  a  pic 9.9999999.
compute a = 1.4
display a   *> 1.39999999

Conversely, if the FPU automatically rounds

compute a = 1.39999999
display a   *> 1.40000000

The passage you quoted is telling the implementor to use a more
precise intermediate for the special case of one operand under native
arithmetic. The exact value WILL be stored under standard arithmetic
(or an exception raised if it doesn't fit).
```
