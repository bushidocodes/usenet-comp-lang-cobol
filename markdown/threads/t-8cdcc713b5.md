```
Definitely not.  Dynamic tables were removed because they are not 
really necessary with the capability of allocating and freeing 
storage.  Also, they would be extremely inefficient in many cases and 
if pointers were used to point at parts of them they would lead to 
bugs that would be almost impossible to find.

The object data one has nothing to do with implementation but has 
everything to do with preventing another really big class of bugs.  
Also, it is of dubious value.
```

```
(Posted to the NG - CC'ed to Don and Judson)

To me, one example of where the current draft has been designed to "make
implementation easier" is something that Judson has commented on frequently.
The current direction is to make READ PREVIOUS "processor dependent" (which
really now means "optional").  Not only do most (but not all) existing
indexed file systems already support this functionality, but it is seems no
more "demanding" to REQUIRE all implementations to support this than the
original requirement (many Standards ago) to require all implementations to
support an indexed file system.  Yes, it might require some additional
"implementation work" - but I just can't see any good reason (for the
language or users' sakes) not to REQUIRE this feature.  For that matter,
almost all the new "processor dependent" features and certainly the fact
that this whole category is now explicitly defined to allow implementors to
decide whether or not to include a feature based on "market demand - or
anything else" is very much an "implementor-oriented" decision rather than
something to improve the language (or make life easier or better for
programmers) - IMHO.
```
