Yes, it is a bit bewildering at first but that's because we aren't used to
it (well, at least I certainly wasn't...<G>)

Basically, it comes down to dropping controls onto a form in a graphic
environment, deciding what events you want to handle from the controls you
have dropped, and writing the appropriate code in COBOL.

What is happening is that all the code you would normally have to write
yourself (the windows message loop, raising events, creating, controlling,
repainting, and destroying windows) is all abstracted into a standard
template which you never see and don't have to worry about. All that is
visible to you is the COBOL code you write to handle events, and any common
routines (actually, nested sub-programs, but again, that is transparent
too...).

It is a little hard at first for COBOL programmers, used to developing text
with an Editor, compiling and linking it, to suddenly find a Graphic
Development Environment where you can do just about anything by simply
clicking a mouse...

Obviously, we still write specific code with an Editor, but PowerCOBOL takes
care of all the underlying stuff and the compiling and linking (you simply
build the project) and, more importantly, it ensures that the only code you
ever see is what you wrote yourself. This means you can focus on your own
code and remain removed from the vagaries of the Windows environment.


>I plan to have the GSGuide re-printed and push thru the projects
>so that I might get a better feel for the GUI development side of things.
>I've noticed that the POW.... event coding tends to be a bit cryptic.

In the normal COBOL environment we get used to controlling everything and
doing things when WE want to...Open the files, read the data, close the
files, etc.

I believe the single most difficult adjustment for COBOL programmers making
the transition into GUI programming is that GUI is event driven. You can't
tell the user "Hang on a minute! Don't click the mouse yet! I'm not
ready...!"

Once you make this transition and become familiar and comfortable with event
driven programming it is fascinating and rewarding. PowerCOBOL eases it by
allowing event code to written in COBOL. So you can say "Well, if you decide
to click on this control, here's what I want to happen..." If and when the
user does click on it, your code will be activated. You don't have to worry
about the actual activation of it, it is done for you.

It is simply COBOL. Maybe the examples are not the best. If you  (or anyone
here)have problems, mail me the project and I'll give you the background and
try and help.

Ah! I understand your reasons... VB is familiar and comfortable when it
comes to GUI. (And it is also very useful as a "Quick Build" prototyping

heart...<G>)).

Believe me, there are huge similarities between using PowerCOBOL and
developing with VB in the VB/IDE.

It is very possible to develop the GUIs in COBOL (PowerCOBOL) AND call COBOL
97 .dlls from the same environment. There is a Fujitsu supplied module
called "POWEROPENFORM" but I have had problems getting it to work under
Version 5 on Win 98. Fujitsu assure me the problems are sorted in Version 6
which is due for release any time now.
