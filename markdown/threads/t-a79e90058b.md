I have a client who is having a problem with a legacy COBOL application.
Some of the client's data files have become corrupt and the application
fails to run with the notice: "Error 98 on SRACT."

The application is compiled under RM COBOL and the client has RM COBOL runtime
version 2.02.06 running under SCO Xenix 2.3.4GT on a 8M RAM 486DX266 box
with Adaptec 1542CF controller and 1G SCSI hard disk.

I have restored the applications's data files in /u/mom from a series of
daily backup tapes, starting with "yesterday's" and going back one day each
time the data is restored. All available backup tapes (five tapes rotated
on Monday through Friday) have been restored in turn, but each time the client
used the restored application, the "error 98" message is encountered after
four to six hours of data entry.

I investigated the system for hardware problems and can find none. I ran
"verify disk" from the Adaptec 1542's SCSI select utility and the disk
gets a clean bill of health. The client has Fox Pro 2.6 runtime with SBT
Accounting and another application running under Dataflex runtime.
Both of these applications are running without errors.

As a test, I set the system date to two months into the future (06/01/97), and
executed the application to the point of generating the "error 98" message.
I then used the find command with -atime at -1 to identify the following files
as being accessed:

› -rw-rw-rw-   1 mom      group    1216512 Mar 11 14:40 CMSTR.MST
› -rw-rw-rw-   1 mom      group    11378688 Mar 12 08:11 MAIL.MST
…[4 more quoted lines elided]…
› -rw-rw-rw-   1 mom      group       8192 Mar 26 13:25 USERFLAG.MST

I contacted Ken in tech support at Solid Data Systems in Lenexa KS (the
new name of the company that bought the company originally developing
the application for my client) and was told: "...I did not know anyone
was still using the application, I cleaned up our systems 2 years ago and
erased all source code for this application."

Ken informed me that error 98 refers to a corrupted index for the named
data file and suggested that I run the RM COBOL utility "recovery" on
SRACT.MST to see if it can repair the damage.

The following is a recreation of the start up screen for recovery on
SRACT.MST restored from backup tape and prior to running the application
to the point of receiving the error 98 message on SRACT.MST:

›                          Indexed File Recovery Utility
›                                Recovery  In-place
…[12 more quoted lines elided]…
› 

When I saw the 65536 for number of index blocks and the "*****" for number
of records I became worried. The 65536 (2^16) looks like RM COBOL has hit
a two-byte limitation to the number of index blocks available. When recovery
ran, it counted the index blocks up in odometer fashion and stopped at
65536 and reported 416471 records processed. When I used ls -ls SRACT.MST
to show the number of blocks for the data file it returns:

› 65794 -rw-rw-rw- 1 mom group 33554432 Mar 31 09:23 SRACT.MST

Running recovery SRACT.MST SRACT.ERR SRACT.RAW a second time results in
the message: File not marked for recovery. Continue (y/n)?" This file
looks to be recovered successfully.

On file SRMST.MST, attempted recovery results in the following messages:

›                          Indexed File Recovery Utility
›                                Recovery  In-place
…[21 more quoted lines elided]…
› 
 
› AND
 
›                            Indexed File Recovery Utility
›                            Recovery  In-place
…[40 more quoted lines elided]…
› See Utilities Supplement to rebuild index file.


When I attempt to execute the application after recovery on SRMST.MST aborts,
I get an error 98 message on SRMST.MST while the application is loading and
after the main login screen is negotiated, when I attempt to list the "Service
Request" records.


After all the above has been said, I have three questions:

1) Is the 65535 blocks reported for SRACT.MST truly a limit to version
2.02.06 of RM COBOL runtime, or are there start-up options for the runcobol
command to specify increased block size?

2) With the error message on SRMST.MST and the fact that the original source
code and file structure for this application are unavailable, how can I
repair SRMST.MST prior to running recovery (i.e. find the record position with
the problem, cut it out of the file with dd, and adjust the record and block
count to keep recovery happy)?

3) Is there a source on the 'net to get documentation on RM COBOL runtime
2.02.06?

I am posting this request to dis··.@xen··n.ca, alt.cobol, and
comp.lang.cobol if you can help and read this posting on the cobol groups
please respond by e-mail to smf··.@wor··t.net.

Have you checked the ulimit of the UNIX machine that you're running on?
It looks like the file is running into a 32MB barrier of some kind, and
ulimit is a prime suspect when this occurs. Try writing a program that
creates and writes a simple file sequentially and see what happens when
it gets to 32MB. I've attached a simple "C" program below my sig; to
compile it, just copy it into your UNIX directory and enter 'make test'.
It writes a file until it can't write any more, so you might want to
warn other users that you're about to do this.

Regarding recovery, I suggest that if you do find a 32MB problem, you
start over with one of your backups.
