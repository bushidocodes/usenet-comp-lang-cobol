Right.  This can cause DFSORT to issue an ICE025A message (RECORD COUNT OFF)
unless option NOCHECK is in effect.  Here's what the DFSORT Messages book says
about the situation for COBOL:

"If a COBOL-invoking program contains an output procedure, make sure the
RETURN statement is iterated until the AT END condition is executed.  This
occurs the next time the RETURN statement is executed AFTER the last record
was returned to you.

If the iteration of the RETURN is controlled by a PERFORM statement, the
PERFORM logic should be controlled by the execution of the AT END clause of
the RETURN statement."

You can access all of the DFSORT books online, including the Messages book,
from the DFSORT home page at URL:

One common problem in multiple SORTs within the same COBOL program is
that the SORT may be just too clever for it's own good.

I have seen in the past cases where SORT released work-area space at the
end of the input phase of SORT #1 in a multple sort application, and
then failed on the second SORT in the same program.  In the particular
case the sort product was SYNCSORT, and a communication with their
excellent tech support yeilded a $ORTPARM value which prevented this
release of work areas.  

Collisions in the use of SYSOUT by sort and COBOL / LE are also common
trouble.  Special register SORT-MESSAGE or $ORTPARM would allow the
application to direct SORT to use some other DD-name than SYSOUT for
it's messages.  Sharing a DD-name between multiple programs seems to me
to be playing with fire anyway.

It appeared that the program was checking for a logical end of file and a
physical end of file, since the logical end of file came first the return
statement was never executed for the last time to get to the Physical end of
file....here is the offending code...

b400-paragraph

 RETURN SORT-FILE
     AT END  MOVE  'YES'  TO  END-OF-SORT2
             GO TO  B400-EXIT.
..snippage

IF INP-REC-TYPE  =  ' '
   MOVE WS-INP-REC    TO  WS-TRAILER-REC
   MOVE TOT-RECS-OUT  TO  TRL-REC-CNT
   WRITE REBTCH-REC   FROM  WS-TRAILER-REC
   ADD  1             TO  TOT-RECS-OUT
   MOVE  'YES'  TO  END-OF-SORT2
   GO TO B400-EXIT.

..snippage

b400-exit

where the b400 was executed until end-of-sort2 was = "yes"

To resolve the problem all I had to do was comment out the move 'yes' in the
logical end of file check in order to force the return sort-file for the
last time.....

ICE025A IS an ERROR message.  The A means ERROR (I means information).	The
DFSORT Messages book explains what the A and I mean.  For ICE025A, the

back a return code of 16 to the calling program for any A message including
this one.  This all seems sufficient to tell you that there was an error, so
I don't understand why you say the DFSORT messages indicated the sort was
successful or that there were NO MESSAGES indicating a 16 code.

This is really a standards and good programming practices issue, then
and now.. 

The first COBOL I wrote code in was COBOL "F". The IBM COBOL
programmers guides from then to the present have always stated that
testing the sort-return-code is mandatory and continuing execution
without doing that is unpredictable. 

I think the original problem was that SORT was a called program from
COBOL  It's always the responsibility of the calling program to
determine that the called routine ended correctly. In most older
versions of COBOL, SORT was treated exactly this way. You can't say
IBM made it difficult, SORT only gives two return codes. 0 or 16. <G>

I remember working on a number of old programs where adding the
sort-return-code test made the program instantly fail. Checking old
production runs showed that the SORT  had been returning a return code
of 16 for years and no one had caught it. Often because I didn't have
time to rework that part of the code I had to comment out the check
and leave a note in the code for the next guy hoping he'd have time.

As a systems programmer I often thought that the COBOL and SORT
product teams didn't talk except under duress. Resolving any problem
involving the two problems was always a nightmare. 

Anyway, what really brings this to a head is that newer versions of
COBOL and SORT are much  more tightly integrated and newer versions do
propagate an error message.

Check your output procedure.  Almost every time I've seen this the
reason has been failure to activate the AT END condition on the RETURN
statement. I've seen cases where people have counted the records and
then looped the output procedure that many times, or where a
high-value record is sorted as an EOF indicator for other reasons and
is then used in the output procedure to trigger exit from the sort
routine.. 

Bottom line, if you have N records, you must do N+1 RETURN statements
so the AT END condition is activated. Try putting a counter after the
release and return statements and display them at the same time you
display the sort return code.
