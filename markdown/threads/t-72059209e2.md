quickly... and then a simple batch COBOL program to (what else?) print 
some reports.

Now... it would be easy, at the start of the run, to grab a timestamp and 
toss it off to a file so that (if I were working in COBOL) I could read 
this file and get rows WHERE DATE_TIME_STAMP BETWEEN :WS-YESTERDAYS-TS AND

Hello again,

I should have thought about your question more carefully, though what

Thinking about your initial timestamp selection question, why not just
select everything from midnight or some other fixed convenient and
consistent time when system load is low and working back one day,
year, etc for the various selects. Putting all the selects and cursors
in a COBOL program would allow you to enforce table locking to ensure
that all the selects are working with the same data, if that matters!

Syncsort when used internally as a COBOL program's sort still allows
you to use its independent control statements, or did a few years ago,
when I last used it.

DISTINCT is a phrase that can be used with SELECT statements to remove
duplicates.

COBOL is quite good at reformatting, but the SELECT statements can be
used to produce a common set of columns from different selects by
using literals instead of column names where appropriate.  Such
SELECTs could even be combined with UNION using the ALL phrase to
remove duplicates, but this might upset your sequencing requirements
as the ORDER BY would apply to them all in common.

Organising a COBOL program in a "structured" fashion can be quite
effective in handling even quite complex tasks and isn't necessarily
more complex than the JCL for invoking several different invocations
of SYNCSORT, FILEAID and the already required COBOL report program,
though your approach can provide greater flexibility, especially where
the requirement frequently changes.

While I don't know a way of doing what you request in exactly the way
that you wish, there is another possible approach, which is to have a
new small table created for your purpose and to add an additional step
to the end of the billing run to insert a row containing the current
timestamp every night.  Then your procedure could select the
timestamps from the last two rows of this new table to construct your
selection criteria.  This table would also provide an historical
record of the timestamps when each billing run completed, which might
be of use for other purposes.

As a possible "refinement" to make the selection criteria easier to
specify in your current SQL, the rows on this table could each contain
two timestamps, the first would be the timestamp on which the previous
billing run completed and the second would be that on which the
current billing run completed.  Each night, the last row on the table
would be updated to add the current timestamp in the second position
and to insert a new row with the current timestamp in the first
position, leaving the second position null in the new row, ready for
update the following night.
