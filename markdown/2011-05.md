As regards the conversion the user wants, and the ability to run both RM
and Fujitsu, if there is no surefire guarantee, then the quickest route
is the simple way, rather than 'trying' to see if it works and finding
that you have 'gotchas' that you didn't know about; particularly
important if you are converting large files.

My own experience, years ago, of conversion RM to MF : The application
was for oil/gas plants - two groups of files (a) Generic files
applicable to ALL Oil companies - some 4 ISAM files, descriptive
records, primarily text - these being used as required for each separate
location. (b) some 6 ISAM files applicable to a specific location,
(plant/facility). Volumes were not large, something like 100-150 records
in individual files. However from early eighties until 2000 the end-user
had accumulated 700 sets, (individual locations), some obsolescent but
the majority still being active. So that he wasn't having to guess which
sets hadn't been converted he converted them ALL.

As an example Y2K - bearing in mind the application started about 1980 -
Inspection dates were stored as YYMM (didn't need Day). The Y2K program
for each ISAM with dates was converted from YYMM to CCYYMM. He converted
all 700 sets in one day.

Approach - write a program per file. Use a fixed length record for a CSV
file: any RM pic s9(03)v9(02) comps, etc., were written to the CSV as
-999.99; then read the CSV into MF ISAM. I used COBOL CSVs (text) but
could have been written to a spreadsheet format so that you could
clearly see data in rows and columns. The latter allows for easy visual
editing - if required.

To clarify above, I may have a program 'Readings file', the external
filenames below are used as input/output :-

PetroCanada  - Waterton Plant - PEC022READGS
Amoco Canada - Slave Lake     - AMC170READGS

In your situation, if you are trying to use the RM and Fujitsu in
parallel for testing, and subsequently live runs, ensure that the
External filenames, (which your conversion program prompts for),
indicates RM or FJ. Taking the above filenames I would have :-

Input CSV - RM-PEC022READGS,  Output ISAM - FJ-PEC022READGS
Input CSV - RM-AMC170READGS,  Output ISAM - FJ-AMC170READGS

Above might look long-winded but given a CSV that you 'prove' per file
format, it is correct and you don't have to second-guess. And frankly,
giving the 'project' some initial design thoughts, I'd guess you will
resolve the problem quicker with the supposedly longer approach.

 > level of compatibility there, but sometimes it works and sometimes it
 > doesn't.


Recommended method for converting between compilers is (assuming you have
access to the source code and the old compiler):

Check that both compilers produce the same format for sequential files
(they should but you never know until testing).

Create a convert program for all (Indexed and Random) needed files to
sequential (or line sequential if no binary fields used) :


Create similar for new compiler but in the reverse eg, Sequential to
indexed.

This is the only sure way of converting between compilers.

Some compiler systems supply tools for doing this, e.g., MF, but NOT for
all of their products, but their support team should be able to supply a
tool providing you do not tell them you are dropping the compiler. :)

Your mileage may vary, but this method has always worked for me and that
includes changing computers (However in this case all files should be in
LS formats, and ALL forms of binary fields changed to display).
